Query Context vs Filter Context

ES Query DSL 的查詢分為兩種「上下文」，影響是否計分與是否 cache：

flowchart LR
    subgraph QC["📊 Query Context"]
      direction TB
      Q1["計算 _score (相關性)"]
      Q2["不 cache"]
      Q3["全文搜尋首選
(match, multi_match)"]
    end

    subgraph FC["✅ Filter Context"]
      direction TB
      F1["不計分
(constant_score=1.0)"]
      F2["✅ 結果 cache"]
      F3["精確比對首選
(term, range, exists)"]
    end

    Req["Query"] --> QC
    Req --> FC

    classDef qctx fill:#fed7aa,stroke:#ea580c,color:#0f172a
    classDef fctx fill:#dcfce7,stroke:#16a34a,color:#0f172a,stroke-width:2px
    class Q1,Q2,Q3 qctx
    class F1,F2,F3 fctx
圖 36-1：Query Context vs Filter Context 的差異
✅ 黃金法則
  • 需要計算「相關性分數」 → query context（用 must）
  • 純條件過濾（yes/no） → filter context（用 filter）
  • filter 結果會 cache，效能遠勝 query

最重要：match vs term

Query是否 analyzed用途
term❌（精確比對）keyword、numeric、boolean、ID
match✅（過 analyzer）text 全文搜尋
match_phrase✅連續詞組（"new york"）
multi_match✅跨多欄位 match
terms❌多個值的 IN 查詢

term：精確比對

{
  "query": {
    "term": {
      "category": "electronics"
    }
  }
}
⚠️ 常見陷阱：在 text 欄位用 term 
{ "term": { "title": "Hello World" } }   // ❌ 大概率搜不到
原因：title（text）已被 analyzer 切成 ["hello", "world"]，但 term 不分析 query，所以找不到 "Hello World" 完整字串。
正解：用 title.keyword（multi-field）或改用 match。

match：全文搜尋

{
  "query": {
    "match": {
      "description": "wireless bluetooth headphones"
    }
  }
}

// 進階：指定 operator 與 minimum_should_match
{
  "query": {
    "match": {
      "description": {
        "query":    "wireless bluetooth headphones",
        "operator": "and",                     // 預設 or
        "minimum_should_match": "75%",
        "fuzziness": "AUTO"                    // 容錯
      }
    }
  }
}

match_phrase：連續詞組

{
  "query": {
    "match_phrase": {
      "title": {
        "query": "new york city",
        "slop":  1                  // 允許 1 個 token 偏移
      }
    }
  }
}

multi_match：跨多欄位

{
  "query": {
    "multi_match": {
      "query":  "elasticsearch tutorial",
      "fields": [ "title^3", "description^1", "tags^2" ],   // ^N 為 boost
      "type":   "best_fields"     // 或 most_fields, cross_fields, phrase, phrase_prefix
    }
  }
}

multi_match 的 type 選擇

type行為適用
best_fields（預設）取分數最高的欄位單一概念分散在多欄位
most_fields所有欄位分數加總同義欄位（title.en + title.zh）
cross_fields把多欄位視為一個大欄位結構化資料（first_name + last_name）
phrase每欄位都用 match_phrase連續詞組
phrase_prefix類似 phrase + 最後 token 用 prefixautocomplete
bool_prefix類似 most_fields + 最後 prefixautocomplete 變體

完整範例

{
  "query": {
    "bool": {
      "must": [
        { "match": { "description": "wireless headphones" } }
      ],
      "should": [
        { "match": { "brand":   "sony" } },
        { "match": { "feature": "noise cancelling" } }
      ],
      "minimum_should_match": 1,
      "filter": [
        { "term":  { "category":  "electronics" } },
        { "term":  { "in_stock":  true } },
        { "range": { "price":     { "gte": 50, "lte": 500 } } }
      ],
      "must_not": [
        { "term": { "discontinued": true } }
      ]
    }
  }
}

語意規則速查

Clause必須匹配影響分數結果 cache
must✅✅❌
filter✅❌✅
should非必要（依 minimum_should_match）✅❌
must_not反向匹配❌✅（filter context）

Range / Exists / Prefix

range：範圍查詢

{
  "query": {
    "range": {
      "price": {
        "gte": 50,
        "lt":  500
      }
    }
  }
}

// Date 範圍
{
  "query": {
    "range": {
      "@timestamp": {
        "gte": "now-7d/d",            // 7 天前的日初
        "lt":  "now/d",
        "time_zone": "+08:00",
        "format": "yyyy-MM-dd"
      }
    }
  }
}

Date Math 速查：

  • now、now-1d、now-7d/d（rounded down to day）
  • now+1h、now/M（month start）
  • 單位：y、M、w、d、h、H、m、s、ms

exists：欄位存在

{ "query": { "exists": { "field": "email" } } }

// 找「沒有」某欄位
{
  "query": {
    "bool": {
      "must_not": [ { "exists": { "field": "email" } } ]
    }
  }
}

prefix / wildcard / regexp

// prefix：前綴（用於 keyword）
{ "query": { "prefix": { "user_id": "user_" } } }

// wildcard：通配（小心，慢）
{ "query": { "wildcard": { "filename": "*.log" } } }

// regexp：正則
{ "query": { "regexp": { "user_id": "user_[0-9]+" } } }
⚠️ wildcard / regexp 開頭通配很慢 *foo 需要掃所有 term，效能差。建議：
  • 用 wildcard 型別欄位（CH 26）
  • 反向 token：建立 reversed analyzer 把字串反過來索引，把 *ing 變成 gni*

ids：依 ID 取多筆

{
  "query": {
    "ids": {
      "values": [ "1", "2", "3" ]
    }
  }
}

等同於 _mget，但可在 search context 用（與其他條件組合）。

match_all / match_none

{ "query": { "match_all": { } } }    // 全部
{ "query": { "match_none": { } } }   // 空

與 Aggregation 同用

{
  "size": 0,                     // 不要回 hits，純 aggregation
  "query": {
    "bool": {
      "filter": [
        { "range": { "@timestamp": { "gte": "now-7d" } } }
      ]
    }
  },
  "aggs": {
    "by_category": {
      "terms": { "field": "category", "size": 10 },
      "aggs": {
        "avg_price": { "avg": { "field": "price" } }
      }
    }
  }
}

詳見 CH 38 aggregations。

Boost 與計分控制

欄位 boost（multi_match）

{
  "multi_match": {
    "query": "elasticsearch",
    "fields": [ "title^3", "description^1" ]    // title 加權 3 倍
  }
}

Query 層 boost

{
  "bool": {
    "should": [
      { "match": { "title":       { "query": "elasticsearch", "boost": 2.0 } } },
      { "match": { "description": "elasticsearch" } }
    ]
  }
}

constant_score：固定分數

{
  "query": {
    "constant_score": {
      "filter": { "term": { "category": "electronics" } },
      "boost":  1.5
    }
  }
}

function_score：自訂計分

{
  "query": {
    "function_score": {
      "query": { "match": { "description": "phone" } },
      "functions": [
        {
          "filter": { "match": { "brand": "apple" } },
          "weight": 2.0
        },
        {
          "field_value_factor": {
            "field":    "popularity",
            "modifier": "log1p",
            "factor":   1.2,
            "missing":  1
          }
        },
        {
          "gauss": {
            "release_date": {
              "origin": "now",
              "scale":  "30d",
              "decay":  0.5
            }
          }
        }
      ],
      "score_mode": "sum",          // 各 function 分數的合併方式
      "boost_mode": "multiply"       // function 與 query 分數的合併方式
    }
  }
}

Explain：為何這個分數？

# 看為什麼某文件得這個分
curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/products/_explain/1?pretty" -d '
{
  "query": { "match": { "description": "wireless" } }
}'

GET products/_search
{
  "explain": true,
  "query": { ... }
}

輸出包含 BM25 計算細節（tf、idf、boost、norm）。

BM25 計分簡介

9.x 預設用 BM25 計分（取代舊的 TF-IDF）。重點：

  • TF（term frequency）：詞在文件中出現次數，有飽和點（避免長文章靠重複壓垮短文章）
  • IDF（inverse document frequency）：詞在多少文件中出現，越少分數越高
  • Length norm：短文件比長文件更具競爭力（可關閉 norms: false）

自訂 similarity

{
  "settings": {
    "similarity": {
      "my_bm25": {
        "type": "BM25",
        "k1":   1.5,        // 預設 1.2
        "b":    0.5         // 預設 0.75（length normalization 強度）
      }
    }
  },
  "mappings": {
    "properties": {
      "title": {
        "type":       "text",
        "similarity": "my_bm25"
      }
    }
  }
}

dis_max：最佳欄位拉抬

對「相同概念在多個欄位」的場景，best_fields 即內部用 dis_max：

{
  "query": {
    "dis_max": {
      "queries": [
        { "match": { "title":       "brown fox" } },
        { "match": { "description": "brown fox" } }
      ],
      "tie_breaker": 0.3
    }
  }
}

分數取最高欄位 + 其他欄位 × tie_breaker。

boosting：負面 boost

把符合 negative 條件的文件分數降低（但不剔除）：

{
  "query": {
    "boosting": {
      "positive": { "match": { "description": "headphones" } },
      "negative": { "term":  { "brand": "no-name-brand" } },
      "negative_boost": 0.3
    }
  }
}

Nested 與 Parent-Child Query

Nested

{
  "query": {
    "nested": {
      "path": "comments",
      "query": {
        "bool": {
          "must": [
            { "match": { "comments.author": "alice" } },
            { "range": { "comments.stars":  { "gte": 4 } } }
          ]
        }
      },
      "score_mode": "avg",        // sum, max, min, none
      "inner_hits": { }           // 取出 nested hit 細節
    }
  }
}

has_parent / has_child（少用）

Join field（CH 26）對應的 query。但 join 會帶來顯著效能成本，盡量先考慮 denormalize。

Geo Queries

// 座標範圍
{
  "query": {
    "geo_bounding_box": {
      "location": {
        "top_left":     { "lat": 25.10, "lon": 121.50 },
        "bottom_right": { "lat": 25.00, "lon": 121.60 }
      }
    }
  }
}

// 距離
{
  "query": {
    "geo_distance": {
      "distance": "10km",
      "location": { "lat": 25.03, "lon": 121.56 }
    }
  }
}

// 多邊形
{
  "query": {
    "geo_polygon": {
      "location": {
        "points": [ ... ]
      }
    }
  }
}

Query Cache 與 Filter Cache

ES 自動 cache 以下：

  • Shard request cache（indices.requests.cache.size，預設 1% heap）
  • Query cache（filter context 的結果，indices.queries.cache.size，預設 10% heap）

看 cache 命中率：

curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_nodes/stats/indices?human&pretty" | jq '.nodes[].indices.query_cache'

Cache hit 過低代表 filter 條件變化太多，難以 cache（如 timestamp 用 now 而非 now/d）。

Query DSL 最佳實踐

  • 能用 filter 就用 filter，效能 + cache 雙贏
  • 固定條件用 now/d 而非 now，提升 cache 命中
  • 排序欄位確保有 doc_values（keyword/numeric/date 預設有，text 沒有）
  • 大量結果用 search_after / PIT（CH 37），不要 deep pagination
  • aggregation only 場景設 "size": 0
  • 明確指定 _source，省網路頻寬
  • 用 profile API 找慢查詢瓶頸
  • 避免 leading wildcard（*foo），改用 wildcard 欄位型別
  • 慎用 function_score，會大幅降低 cache 效益

本章重點回顧

  • Query Context 計分但不 cache；Filter Context 不計分但 cache
  • term 不 analyze 適合 keyword/numeric；match analyze 適合 text
  • bool query 是組合王者：must（計分）+ filter（純條件）+ should（加成）+ must_not（排除）
  • multi_match 的 type 選擇影響跨欄位計分行為
  • 9.x 預設 BM25 計分；可調 k1、b 參數
  • function_score 可加入欄位值、距離衰減等自訂邏輯
  • Geo / nested / parent-child 各有專用 query 語法
  • 用 explain API 與 profile API 診斷分數與效能
