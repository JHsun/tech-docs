Inference API 統一介面（9.x）

9.x 把 ML / 外部 LLM 統一在 Inference API 後面：

# 列出已建立的 inference endpoint
GET /_inference

# 建立 endpoint
PUT /_inference/<task_type>/<endpoint_id>

Hybrid Retriever for RAG

POST articles/_search
{
  "retriever": {
    "rrf": {
      "retrievers": [
        {
          "standard": {
            "query": {
              "match": { "title": "How does Elasticsearch work?" }
            }
          }
        },
        {
          "standard": {
            "query": {
              "semantic": {
                "field": "content",
                "query": "How does Elasticsearch work?"
              }
            }
          }
        }
      ],
      "rank_window_size": 50
    }
  },
  "size": 5,
  "_source": [ "title", "content.text", "metadata" ]
}

Rerank：第二階段精排

Hybrid 取 top-50，再用 rerank 模型挑出 top-5：

POST articles/_search
{
  "retriever": {
    "text_similarity_reranker": {
      "retriever": {
        "rrf": {
          "retrievers": [ ... ],
          "rank_window_size": 50
        }
      },
      "field":           "content",
      "inference_id":    "cohere-rerank",
      "inference_text":  "How does Elasticsearch work?",
      "rank_window_size": 10
    }
  }
}

常見堆疊：BM25 + 向量 → RRF top-50 → rerank → top-5 給 LLM。

完整 RAG 端到端

from elasticsearch import Elasticsearch

es = Elasticsearch("https://es:9200", api_key="...", ca_certs="ca.crt")

def rag_pipeline(question: str) -> str:
    # 1. Retrieval（hybrid + rerank）
    res = es.search(index="knowledge-base", body={
        "retriever": {
            "text_similarity_reranker": {
                "retriever": {
                    "rrf": {
                        "retrievers": [
                            {"standard": {"query": {"match": {"content": question}}}},
                            {"standard": {"query": {
                                "semantic": {"field": "content", "query": question}
                            }}}
                        ],
                        "rank_window_size": 50
                    }
                },
                "field":          "content",
                "inference_id":   "cohere-rerank",
                "inference_text": question,
                "rank_window_size": 5
            }
        },
        "size": 5,
        "_source": ["title", "content.text", "metadata.url"]
    })

    # 2. 組 context
    context_parts = []
    for hit in res["hits"]["hits"]:
        src = hit["_source"]
        context_parts.append(
            f"[{src.get('title', '')}] ({src['metadata']['url']})\n"
            f"{src['content']['text']}"
        )
    context = "\n\n---\n\n".join(context_parts)

    # 3. 透過 ES Inference API 呼叫 LLM
    res = es.options(request_timeout=120).perform_request(
        "POST", "/_inference/completion/claude-chat",
        body={
            "input": (
                f"Use ONLY the following context to answer the question. "
                f"If the answer is not in the context, say so.\n\n"
                f"Context:\n{context}\n\n"
                f"Question: {question}\n\n"
                f"Answer:"
            )
        }
    )
    return res.body["completion"][0]["result"]

好處：把 LLM call 也走 ES Inference API，所有 secret 集中管理，application 端不需處理 LLM API key。

Playground（9.x Kibana 內建）

Kibana → Search → Playground 提供互動式 RAG 開發介面：

  • 選擇 source index
  • 選 LLM connector（OpenAI / Anthropic / Bedrock 等）
  • 調 prompt template
  • 調 retrieval 參數（hybrid weights、rerank、top-K）
  • 輸出 ready-to-deploy code（Python / JavaScript）

對 PoC 與 prompt engineering 極快。

AI Assistant（Observability + Security）

9.x 整合進 Observability（CH 57）與 Security（CH 58）：

  • 自然語言查詢：「過去 1 小時哪個 service error 多？」
  • Auto-RCA：對 anomaly 提供根因建議
  • Custom knowledge sources：附自家 runbook、過往 incident
  • Workflow 整合：可觸發 connector action（CH 45）

本質上是預先 packaged 的 RAG，知識庫由 Elastic 維護 + 用戶可加自家文件。

Agent Builder（9.x 新）

Kibana → Stack Management → Agent Builder：把 RAG 升級為 agentic workflow（多步驟、工具調用）：

  • Agent 可調用 ES 查詢、外部 API、其他 agent
  • 支援 function calling（OpenAI/Anthropic/Bedrock 模型）
  • 典型場景：
    • SOC 自動分流：alert 進來 → agent 查 user 風險 → 查 host history → 建議 disposition
    • SRE 自動診斷：alert → agent 查 metrics → 查 log → 提供 RCA 與建議 action
    • 客服：query → 查 KB → 查 ticket history → 產生回覆

9.3+ Agent Builder 整合 Workflows for orchestration，能組合多 agent 與外部系統。

RAG 實戰最佳實踐

資料準備

  • chunking 策略匹配內容：FAQ 用 sentence；技術文件用 paragraph；程式碼用 function
  • chunk_overlap 約 10-20%，避免邊界資訊丟失
  • 保留 metadata（source、date、author、tags），便於 filter 與 attribution
  • 定期 re-index：原始文件變動就更新

檢索品質

  • 始終 hybrid（BM25 + 語義），純向量對 keyword（如型號）弱
  • RRF rank_window_size = top-K × 5-10
  • 大量文件加 rerank（Cohere v3.5 / OpenAI rerank）
  • Filter 在 retriever 內做（permission、date range、language）
  • 用 recall@K 評估：人工標註 ground truth，定期測試

生成品質

  • System prompt 明確：「Use ONLY the context」+「If not found, say so」
  • 給 source citation：要 LLM 標註用了哪個 chunk
  • top-3 ~ 5 chunks 通常足夠，太多反而稀釋注意力
  • chunk 大小 200-500 token 是甜蜜點
  • 監控 hallucination：人工抽樣 + automated 評估（RAGAS、TruLens）

安全與權限

  • Per-user filter：retrieval 時加 filter: { terms: { acl_users: ["alice"] } }
  • Document-level security (DLS)（CH 21）：role-based 過濾
  • 用 ES Inference API 走 LLM，不要 application 端持有 API key
  • Audit log：記錄誰問了什麼、查到什麼
  • Prompt injection 防護：sanitize user input、limit retrieval results size

成本控制

  • BBQ 量化（CH 40）：vector 儲存省 95%
  • DiskBBQ：記憶體只需 100 MB
  • 用 ELSER 取代外部 embedding：免 API 成本
  • cache LLM 回應：相同 query 不重複呼叫
  • 計算 token cost / query：監控避免突增

本章重點回顧

  • 9.x Inference API 統一介面，5 種 task type × 12+ provider
  • 支援 OpenAI / Anthropic / Bedrock / Cohere / Gemini / Jina / 本地 ELSER
  • semantic_text 欄位：自動 chunk + embed，最簡用法
  • Hybrid retriever (RRF) + rerank 是高品質 RAG 黃金組合
  • 把 LLM 也走 Inference API，secret 集中、application 簡化
  • Playground：Kibana 內建互動式 RAG 開發
  • AI Assistant：Observability / Security 內建 RAG-driven 助手
  • Agent Builder（9.x）：把 RAG 升級為 multi-step agentic workflow
  • 實戰：chunk 策略、hybrid + rerank、permission filter、cost 控制
