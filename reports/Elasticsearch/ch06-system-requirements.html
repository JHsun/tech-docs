硬體需求

CPU

Elasticsearch 的 CPU 用量受工作負載影響很大：

工作負載CPU 特性建議規格
純搜尋（low qps）突發、非持續高4-8 vCPU 起
大量索引寫入持續高，多執行緒8-16 vCPU
聚合、ES|QL、MLCPU 密集16-32 vCPU
kNN/向量搜尋SIMD 加速，建議新世代 CPU16+ vCPU、AVX2/AVX-512 支援

選擇較新世代的 CPU（Intel 第 4 代 Xeon Scalable / AMD EPYC Gen 4 / AWS Graviton 4 / Apple Silicon），對 BBQ（Better Binary Quantization）與向量運算有顯著加速。

記憶體（RAM）

記憶體是 Elasticsearch 最關鍵的資源。三大用途：

  • JVM Heap：用於 indexing buffers、cluster state、search request、aggregations
  • OS File System Cache：Lucene 倚賴 OS 把 segment 檔案快取到 RAM 中加速搜尋
  • Off-heap memory：Lucene 的某些結構（doc values、bitmaps）使用 mmap 記憶體
💡 黃金比例：Heap ≤ 50% RAM，且 Heap ≤ 31 GB
  • JVM heap 設為總 RAM 的 一半，另一半留給 OS file cache
  • Heap 不要超過 31 GB（壓縮指標 compressed oops 的限制；超過後反而效能下降）
  • 若機器有 128 GB RAM，heap 也只設 31 GB，剩下給 OS cache
  • 若需要更多 heap，請用更多節點而非單節點加 heap

儲存（Disk）

儲存類型適用 Tier說明
本地 NVMe SSDHot最佳寫入與搜尋效能。Hot tier 強烈建議
本地 SATA SSDHot / Warm夠用，但 IOPS 較低
本地 HDD（7200rpm+）Cold容量大但隨機 I/O 慢，僅適合舊資料
網路儲存（SAN/NAS/EBS gp3/io2）Hot 可，但需注意EBS io2 可達 64K IOPS；網路抖動會影響搜尋
NFS❌ 不建議會有 lock、stale handle、效能等問題
🚨 不要用 RAID 0 以外的本地 RAID Elasticsearch 已透過 replicas 提供叢集級的冗餘。本地 RAID 1/5/6 只會降低寫入效能而沒有實質幫助。直接用 RAID 0 或單一 SSD 即可。當然，雲端的 EBS / Persistent Disk 等服務本身已含底層冗餘，無此顧慮。

網路

  • 叢集內部：10 GbE 以上（生產級），1 GbE 僅適合小型開發
  • 所有節點之間延遲應 < 5 ms（同一 AZ 或機架）
  • 跨 AZ 部署：< 50 ms 可接受，CCS/CCR 跨 region 需特別調校
  • 啟用 jumbo frames（MTU 9000）可降低高吞吐時的 CPU 用量

Java / JVM 需求

✅ 強烈建議使用 Bundled JDK Elasticsearch 9.x 發行版內建一份 OpenJDK，由 Elastic 維護並隨版本更新。不要自己安裝 JDK 取代它，除非有強制要求（例如 FIPS 合規）。

Bundled JDK 位於 $ES_HOME/jdk。如果你必須使用自己的 JDK：

# 設定環境變數指向自有 JDK
export ES_JAVA_HOME=/path/to/your/jdk

# 啟動時 ES 會用這個 JDK 取代 bundled
sudo systemctl start elasticsearch

注意事項：

  • 必須使用 OpenJDK 衍生版本（不支援 GraalVM、Azul 等少數變種，但 Adoptium、Amazon Corretto 沒問題）
  • 建議使用最新 LTS 版本（目前為 JDK 21；JDK 17 仍受支援）
  • FIPS 模式必須使用 Oracle JDK 17（自 8.13+ 起）
🚨 永遠不要使用第三方 Java agent APM agent、profiler、tracer 等以 -javaagent:xxx.jar 形式附加的工具，可能造成 ES 啟動失敗、節點凍結甚至資料損毀。請用 Elastic 自家的 APM 整合（CH 60）。

Bootstrap Checks 全清單

Production mode 下，以下檢查必須全部通過。逐一說明：

1. Heap size check

JVM 啟動時的 -Xms（初始 heap）必須等於 -Xmx（最大 heap），避免動態 resize 造成 GC pause。

Elastic 9.x 預設使用自動 heap sizing（依節點記憶體與角色計算），無需手動設定。若要手動覆寫：

# /etc/elasticsearch/jvm.options.d/heap.options
-Xms16g
-Xmx16g

2. File descriptor check

每個 shard 會開啟多個 segment 檔案 + 網路連線。預設 OS 的 4096 / 65536 限制都太低。

需求：elasticsearch process 至少能開 65535 個 file descriptors。

檢查當前設定：

ulimit -n
# 或對特定 user 檢查
sudo -u elasticsearch bash -c 'ulimit -n'

調整方式（systemd 環境）：

# 編輯 systemd unit override
sudo systemctl edit elasticsearch

加入：

[Service]
LimitNOFILE=65535

傳統 SysV 環境，編輯 /etc/security/limits.conf：

elasticsearch  -  nofile  65535
elasticsearch  -  nproc   4096

3. Memory lock check（如啟用 mlockall）

啟用 bootstrap.memory_lock: true 後，JVM 會把 heap 鎖定在實體記憶體，避免被 swap 出去（swap 會嚴重影響搜尋延遲）。

# elasticsearch.yml
bootstrap.memory_lock: true

需要 OS 允許這個 process 使用 memlock：

# systemd 環境
sudo systemctl edit elasticsearch

[Service]
LimitMEMLOCK=infinity

驗證：

curl -X GET "https://localhost:9200/_nodes?filter_path=**.mlockall&pretty" \
  -u elastic:CHANGEME --cacert /path/to/http_ca.crt

4. Maximum map count check（Linux only）

Lucene 用 mmap 把 segment 檔案映射到記憶體。需要足夠的 memory map areas。

需求：vm.max_map_count ≥ 262144（建議 1048576）。

# 立即生效
sudo sysctl -w vm.max_map_count=262144

# 永久生效
echo 'vm.max_map_count=262144' | sudo tee -a /etc/sysctl.d/99-elasticsearch.conf
sudo sysctl --system

5. Max number of threads

需求：elasticsearch user 至少能建立 4096 個 threads。

ulimit -u
# 應 ≥ 4096

調整見上方 file descriptors 章節，加入 nproc 4096。

6. Max virtual memory

elasticsearch 必須能使用無限的 virtual memory（mmap 需要）。

ulimit -v
# 應為 unlimited

7. Discovery configuration

Production mode 必須設定以下其一：

  • discovery.seed_hosts
  • discovery.seed_providers
  • cluster.initial_master_nodes（僅首次 bootstrap）

8. Use server JVM

必須使用 server JVM（不是 client JVM）。Bundled JDK 預設即為 server JVM，這個檢查通常不會失敗。

9. JDK version

最低支援版本：見當前版本的 support matrix。9.x 預期需 JDK 21+。

10. -XX:OnError / -XX:OnOutOfMemoryError 限制

不可使用 -XX:OnError 或 -XX:OnOutOfMemoryError JVM 參數。請改用 Elasticsearch 的 jvm.options 內建 -XX:HeapDumpOnOutOfMemoryError。

11. Early-access build check

不可使用 JDK 的 early-access 預覽版（如 JDK 22-ea）。

12. JVM SecurityManager all-permission

不可使用授予 java.security.AllPermission 的 SecurityManager 政策（可能繞過 ES 自己的安全機制）。

完整 OS Tuning Script（Linux）

以下是一個一次性套用全部建議設定的 script，適用 RHEL/Ubuntu 等主流 Linux：

#!/bin/bash
# elasticsearch-os-tune.sh
set -e

# 1. 提高 vm.max_map_count
cat > /etc/sysctl.d/99-elasticsearch.conf < /etc/security/limits.d/99-elasticsearch.conf < /etc/systemd/system/elasticsearch.service.d/override.conf < /sys/kernel/mm/transparent_hugepage/enabled
echo never > /sys/kernel/mm/transparent_hugepage/defrag

cat > /etc/systemd/system/disable-thp.service < /sys/kernel/mm/transparent_hugepage/enabled && echo never > /sys/kernel/mm/transparent_hugepage/defrag"

[Install]
WantedBy=multi-user.target
EOF
systemctl enable disable-thp.service

echo "✅ OS tuning complete. Reboot recommended."

Swap 設定建議

三選一，按建議度排序：

  1. 完全關閉 swap（最佳）— swapoff -a + 移除 /etc/fstab 的 swap entry
  2. 啟用 bootstrap.memory_lock: true（mlockall）
  3. 降低 vm.swappiness 到 1（盡量避免使用 swap）

不要做：保留 default swappiness 60。在記憶體壓力下 OS 會把 ES heap 部分換到磁碟，下次 GC 時整個叢集會看似掛掉。

NTP / 時間同步

分散式系統對時間敏感（特別是 cluster state 的 versioning、ILM age 觸發、稽核日誌）。所有節點應啟用 NTP / chrony：

# RHEL/CentOS
sudo systemctl enable --now chronyd
chronyc sources

# Ubuntu/Debian
sudo systemctl enable --now systemd-timesyncd
timedatectl status

防火牆與 SELinux

防火牆 Port

節點間：9300/tcp（transport）
HTTP/REST：9200/tcp

# firewalld（RHEL）
sudo firewall-cmd --permanent --zone=internal --add-port=9300/tcp
sudo firewall-cmd --permanent --zone=internal --add-port=9200/tcp
sudo firewall-cmd --reload

# ufw（Ubuntu）
sudo ufw allow from 10.0.0.0/24 to any port 9300 proto tcp
sudo ufw allow from 10.0.0.0/24 to any port 9200 proto tcp

SELinux

一般情況 ES 與 SELinux 相容。若遇到 AVC denied，可調整：

# 檢查 audit log
sudo ausearch -m avc -ts recent | grep elasticsearch

# 暫時改為 permissive 排查
sudo setenforce 0

# 確認問題後撰寫專屬 policy
sudo audit2allow -a -M elasticsearch-custom
sudo semodule -i elasticsearch-custom.pp

預估規模工具

  • Elastic Cloud Sizer — 互動式估算工具，可用於 self-managed 規劃參考
  • 經驗公式：每日資料量 ÷ shard 平均大小（30 GB）= shards 數；每節點 shards 數 ÷ 20 = 所需 heap GB

本章重點回顧

  • 記憶體比 CPU 重要：heap ≤ 50% RAM，且 heap ≤ 31 GB
  • 儲存：Hot tier 用本地 NVMe SSD；不要用 NFS、不要用 RAID 1/5/6
  • 網路：節點間建議 10 GbE 以上、低延遲
  • 強烈建議使用內建 JDK，不要安裝第三方 Java agent
  • 當 network.host 不是 loopback 時，自動進入 Production Mode 並強制執行 bootstrap checks
  • 核心 OS 調整：vm.max_map_count ≥ 262144、file descriptors ≥ 65535、關閉 swap、關閉 THP
  • 所有節點都要時間同步（NTP / chrony）
  • 開放 9200（HTTP）與 9300（transport）port
