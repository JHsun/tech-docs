CH 35 · Failure Store 與資料品質保護

適合對象 🟡 開發者 🔴 維運者
閱讀時間 約 22 分鐘
前置知識 CH 29 data streams、CH 30 ILM
本章目標 啟用 failure store 捕獲被拒文件、ingest pipeline 錯誤處理、dead letter queue、資料品質監控

為什麼需要 Failure Store？

真實世界的資料攝取會失敗：

  • 欄位型別不對（mapping 期望 long，傳了 string）
  • TSDS time bounds 超出（CH 29，遲到資料）
  • Ingest pipeline 處理 exception（如 grok parse 失敗）
  • 欄位值超長（超過 ignore_above）
  • 違反 mapping strict mode

沒有 failure store 時，這些文件會被悄悄丟掉（client 收到 4xx 錯誤但通常 fire-and-forget）。Failure store 把它們存到一個隱藏的副 data stream，方便後續分析與重送。

圖 35-1：Failure Store 攔截被拒文件

啟用 Failure Store（9.1+）

新 data stream 啟用

PUT _index_template/logs-app-template
{
  "index_patterns": [ "logs-app*" ],
  "data_stream": {
    "failure_store": true       # 9.1+ 啟用
  },
  "priority": 200,
  "composed_of": [ "logs-app-mappings", "logs-app-settings" ]
}

既有 data stream 啟用

PUT _data_stream/logs-app/_options
{
  "failure_store": {
    "enabled": true
  }
}

Cluster 預設啟用

PUT _cluster/settings
{
  "persistent": {
    "data_streams.failure_store.enabled": "*"   # 對所有新 data stream 啟用
    # 或 "logs-*,metrics-*" 限定 pattern
  }
}

檢視 Failure Store 內容

Failure store 是 hidden data stream，名為 .fs-{原 stream}-{date}-{gen}。查詢用特殊 selector：

# 查詢主 data stream（一般搜尋）
GET logs-app/_search

# 查詢 failure store（用 ::failures selector）
GET logs-app::failures/_search
{
  "query": { "match_all": {} }
}

Failure document 結構

每筆失敗紀錄包含三大部分：

{
  "@timestamp": "2026-04-15T10:23:45.123Z",
  "document": {
    "id":     null,
    "index":  "logs-app",
    "source": {
      "@timestamp": "2026-04-15T10:23:00.000Z",
      "message":    "...",
      "user.age":   "twenty"            // ← 這是讓索引失敗的原始 doc
    }
  },
  "error": {
    "type":     "document_parsing_exception",
    "message":  "[user.age] of type [long] cannot be parsed",
    "stack_trace": "..."
  }
}

同時查主 + failure

GET logs-app::*/_search
{
  "query": { "term": { "_index": "logs-app::failures" } }
}

Failure Store 管理

獨立 retention 設定

PUT _data_stream/logs-app/_options
{
  "failure_store": {
    "enabled": true,
    "lifecycle": {
      "data_retention": "30d"     # 失敗資料保留 30 天
    }
  }
}

主 stream 與 failure store 可各有自己的 retention。Failure store 通常設較短（足夠分析失因即可）。

手動 rollover failure store

POST logs-app::failures/_rollover

關閉

PUT _data_stream/logs-app/_options
{
  "failure_store": {
    "enabled": false
  }
}

Ingest Pipeline 錯誤處理

Ingest pipeline 中的錯誤可細分處理：

每個 processor 的 on_failure

PUT _ingest/pipeline/log-pipeline
{
  "processors": [
    {
      "grok": {
        "field":    "message",
        "patterns": [ "%{TIMESTAMP_ISO8601:timestamp} %{LOGLEVEL:level} %{GREEDYDATA:msg}" ],
        "on_failure": [
          {
            "set": {
              "field": "log.parse_failed",
              "value": true
            }
          },
          {
            "set": {
              "field": "log.original",
              "value": "{{{message}}}"
            }
          }
        ]
      }
    },
    {
      "date": {
        "field":   "timestamp",
        "formats": [ "ISO8601" ]
      }
    }
  ],
  "on_failure": [
    {
      "set": {
        "field": "_index",
        "value": "failed-{{{_index}}}"      # 或路由到別的 index
      }
    },
    {
      "set": {
        "field": "ingest.failure_message",
        "value": "{{{ _ingest.on_failure_message }}}"
      }
    }
  ]
}

三層 on_failure 規則：

  1. Processor 層級：該 processor 失敗時執行
  2. Pipeline 層級（外層 on_failure）：任一 processor 沒被自身 on_failure 處理就執行
  3. 都不處理 → 文件被拒，進 failure store（如有啟用）

忽略失敗

{
  "grok": {
    "field": "message",
    "patterns": [ ... ],
    "ignore_failure": true       # 失敗就跳過此 processor，繼續下一個
  }
}

Dead Letter Queue Pattern

把失敗的文件路由到一個 DLQ data stream：

PUT _ingest/pipeline/main-with-dlq
{
  "processors": [
    { "grok": { ... } },
    { "date": { ... } }
  ],
  "on_failure": [
    {
      "set": {
        "field": "_index",
        "value": "dlq-logs-{{{_index}}}"
      }
    },
    {
      "set": {
        "field": "dlq.failed_at",
        "value": "{{{ _ingest.timestamp }}}"
      }
    },
    {
      "set": {
        "field": "dlq.failure_message",
        "value": "{{{ _ingest.on_failure_message }}}"
      }
    },
    {
      "set": {
        "field": "dlq.failed_processor",
        "value": "{{{ _ingest.on_failure_processor_type }}}"
      }
    }
  ]
}

讓失敗文件存進 dlq-logs-* data stream，便於修復後重送。

重送失敗文件

從 failure store 取出文件，修正後 reindex 回主 stream：

POST _reindex
{
  "source": {
    "index": "logs-app::failures",
    "query": {
      "term": { "error.type": "document_parsing_exception" }
    }
  },
  "dest": {
    "index": "logs-app",
    "op_type": "create"
  },
  "script": {
    "source": "
      // 從 failure doc 抽出原始 source
      def src = ctx._source.document.source;

      // 修正欄位（範例：把 user.age 從字串轉 long）
      if (src.user != null && src.user.age instanceof String) {
        try { src.user.age = Long.parseLong(src.user.age); }
        catch (Exception e) { src.user.age = null; }
      }

      ctx._source = src;
    "
  }
}

Logstash Dead Letter Queue（替代方案）

若用 Logstash 攝取，可啟用 Logstash 的 DLQ：

# logstash.yml
dead_letter_queue.enable: true
path.dead_letter_queue: /var/lib/logstash/dlq
dead_letter_queue.max_bytes: 1gb

失敗的事件會寫到本地 DLQ 檔案，可用 dead_letter_queue input plugin 重新處理：

input {
  dead_letter_queue {
    path           => "/var/lib/logstash/dlq"
    commit_offsets => true
    pipeline_id    => "main"
  }
}

filter {
  # 修正、補欄位
  mutate { ... }
}

output {
  elasticsearch { hosts => [ "..." ] }
}

資料品質監控

用 Watcher 監控失敗率

PUT _watcher/watch/dlq-rate-alert
{
  "trigger": { "schedule": { "interval": "5m" } },
  "input": {
    "search": {
      "request": {
        "indices": [ "logs-app::failures" ],
        "body": {
          "size": 0,
          "query": {
            "range": { "@timestamp": { "gte": "now-5m" } }
          }
        }
      }
    }
  },
  "condition": {
    "compare": {
      "ctx.payload.hits.total.value": { "gt": 1000 }
    }
  },
  "actions": {
    "notify": {
      "webhook": {
        "method": "POST",
        "url":    "https://hooks.slack.com/services/...",
        "body":   "DLQ rate高: {{ctx.payload.hits.total.value}} failures in 5min"
      }
    }
  }
}

Kibana 儀表板

建議在 Kibana 中建立資料品質 dashboard：

  • 每分鐘 ingest 成功 vs 失敗筆數（line chart）
  • 失敗類型分佈（pie：error.type）
  • 失敗最多的欄位（top values: error.message）
  • 主要失敗來源（terms: document.source.host.name）

最佳實踐

  • 所有 production data stream 啟用 failure store（9.1+）
  • Failure store retention 設較短（7-30 天足夠分析）
  • Ingest pipeline 一律設 on_failure，明確處理失敗情境
  • 關鍵業務資料用 DLQ pattern 路由失敗到專屬 stream
  • 建立資料品質儀表板，定期檢視失敗類型
  • 用 Watcher 設失敗率告警（突增是上游 schema 變更的訊號）
  • 失敗文件修正後 reindex 回主 stream，避免資料缺失
  • 對 mapping 採 strict + dynamic templates 雙保險：未預期欄位被擋下，並進入 failure store
  • 測試 mapping 變更前先在 staging 用真實 sample 跑，看會不會有大量失敗

本章重點回顧

  • Failure store（9.1+）自動捕獲被拒的文件，避免悄悄丟失
  • 查詢用 {stream}::failures selector，看 hidden data stream
  • 失敗文件結構：document.source 原始 doc + error.type/message 失因
  • 主 stream 與 failure store 可各有 retention
  • Ingest pipeline 三層 on_failure：processor 層、pipeline 層、failure store
  • DLQ pattern：路由失敗到專屬 stream，便於後續處理
  • Logstash 也有獨立 DLQ 機制（dead_letter_queue.enable）
  • 從 failure store reindex 回主 stream 是修復資料的標準流程
  • 用 Watcher + Kibana dashboard 持續監控資料品質
