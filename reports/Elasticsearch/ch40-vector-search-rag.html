Vector Search 與語義搜尋

傳統 BM25 全文搜尋只能匹配「字面詞彙」。Vector search 把文字轉成向量（embedding），用「向量間距離」找語義相近的文件——即使沒有共同詞彙。

Dense Vector 設定

PUT products
{
  "mappings": {
    "properties": {
      "title": { "type": "text" },
      "embedding": {
        "type":       "dense_vector",
        "dims":       384,
        "index":      true,
        "similarity": "cosine",
        "index_options": {
          "type": "int8_hnsw",
          "m":    16,
          "ef_construction": 100
        }
      }
    }
  }
}

Similarity 比較函式

Similarity說明適用
cosinecosine similarity（夾角）BERT、Sentence Transformers 大多用這個
dot_product內積（向量需 normalize）已 normalize 的向量，最快
l2_norm歐式距離（越小越近）原生 L2 訓練的模型
max_inner_product未 normalize 的內積某些 OpenAI ada-002 等

量化（Quantization）：節省記憶體

9.x 大力推進量化，BBQ（Better Binary Quantization）成為新建 dense_vector 的預設（9.1+）：

Type儲存節省精度適用
flat原始 float320%100%需最高精度
hnsw原始 float32 + HNSW 索引0%100%需最高精度
int8_hnsw1 byte/dim~75%~99%大多場景
int4_hnsw0.5 byte/dim~87.5%~95%大規模
bbq_hnsw ⭐1 bit/dim~95%~95%（rescore 後）9.1+ 預設，超大規模
bbq_disk ⭐從磁碟讀記憶體 ~99% 節省~95%9.2+ GA，記憶體吃緊

BBQ + rescore 範例

{
  "mappings": {
    "properties": {
      "embedding": {
        "type":       "dense_vector",
        "dims":       1024,
        "similarity": "cosine",
        "index_options": {
          "type": "bbq_hnsw"      // 9.1+ 大型索引推薦
        }
      }
    }
  }
}

查詢時可加 rescore 提升精度：

{
  "knn": {
    "field":         "embedding",
    "query_vector":  [...],
    "k":             10,
    "num_candidates": 100,
    "rescore_vector": {
      "oversample": 3.0    // 多取 3x 候選用 full precision rescore
    }
  }
}

DiskBBQ（9.2+）

把量化向量直接讀磁碟，sub-20ms latency 且記憶體只用 ~100 MB（不論 index 多大）：

{
  "index_options": {
    "type": "bbq_disk"
  }
}

取捨：p99 比 in-memory HNSW 略高（仍 sub-100ms），但成本低很多，適合大規模 RAG。

9.x bfloat16 storage

9.x 新增 bfloat16（16-bit）儲存格式，比 float32 省一半空間，精度損失極小：

{
  "embedding": {
    "type":         "dense_vector",
    "element_type": "bfloat16",
    "dims":         1024
  }
}

KNN Search Query

Approximate KNN（HNSW，最常用）

curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/products/_search?pretty" -d '
{
  "knn": {
    "field":         "embedding",
    "query_vector":  [0.12, 0.34, ...],
    "k":             10,
    "num_candidates": 100,
    "filter": {
      "bool": {
        "must": [
          { "term":  { "category": "electronics" } },
          { "range": { "price":    { "lte": 500 } } }
        ]
      }
    }
  },
  "_source": [ "title", "price" ]
}'

關鍵參數：

  • k：要返回幾個結果
  • num_candidates：每 shard 探索的候選數，影響 recall（建議 k × 5 ~ 10）
  • filter：pre-filter，先過濾再 KNN（更準）

Exact KNN（小資料集）

用 script_score 暴力比對所有文件：

{
  "query": {
    "script_score": {
      "query": { "match_all": {} },
      "script": {
        "source": "cosineSimilarity(params.q, 'embedding') + 1.0",
        "params": { "q": [0.12, 0.34, ...] }
      }
    }
  },
  "size": 10
}

只適合 < 10K 文件的小資料集（線性掃描）。

Filtered KNN：先過濾再向量搜

9.x 對 filtered vector search 大幅優化（vs OpenSearch 快 7x）：

{
  "knn": {
    "field":         "embedding",
    "query_vector":  [...],
    "k":             10,
    "num_candidates": 100,
    "filter": [
      { "term":  { "language": "zh-TW" } },
      { "range": { "@timestamp": { "gte": "now-30d" } } }
    ]
  }
}

ELSER：Elastic 自家稀疏向量模型

ELSER（Elastic Learned Sparse EncodeR）是 Elastic 預訓練的稀疏向量模型，不需 dense embedding model 就能做語義搜尋：

部署 ELSER

# 1. 下載 ELSER 模型（在 ML node 上）
POST /_ml/trained_models/.elser_model_2/_download

# 2. 部署
POST /_ml/trained_models/.elser_model_2/deployment/_start

# 3. 建立 inference endpoint
PUT /_inference/sparse_embedding/my-elser
{
  "service": "elasticsearch",
  "service_settings": {
    "model_id": ".elser_model_2",
    "num_allocations": 1,
    "num_threads":     1
  }
}

使用 sparse_vector 欄位

PUT articles
{
  "mappings": {
    "properties": {
      "content": { "type": "text" },
      "content_embedding": {
        "type": "sparse_vector"
      }
    }
  }
}

用 ingest pipeline 自動產生

PUT _ingest/pipeline/elser-ingest
{
  "processors": [
    {
      "inference": {
        "model_id":      "my-elser",
        "input_output": [
          {
            "input_field":  "content",
            "output_field": "content_embedding"
          }
        ]
      }
    }
  ]
}

# Index 時自動套用
POST articles/_doc?pipeline=elser-ingest
{
  "content": "Elasticsearch 是分散式搜尋引擎..."
}

查詢

{
  "query": {
    "sparse_vector": {
      "field":   "content_embedding",
      "inference_id": "my-elser",
      "query":   "How does distributed search work?"
    }
  }
}

semantic_text 欄位（最簡）

9.x 推出的高度自動化欄位，內含 chunk + embedding，不用手動寫 ingest pipeline：

PUT articles
{
  "mappings": {
    "properties": {
      "content": {
        "type":         "semantic_text",
        "inference_id": "my-elser"
      }
    }
  }
}

POST articles/_doc
{
  "content": "Elasticsearch 是..."
}

GET articles/_search
{
  "query": {
    "semantic": {
      "field": "content",
      "query": "How does distributed search work?"
    }
  }
}

系統自動：分塊（chunk）、產生 embedding、存到內部欄位、查詢時也自動 embed query。

Hybrid Search：融合 BM25 + Vector

單純 vector search 對「精確 keyword」場景仍弱於 BM25（如型號編號）。Hybrid 結合兩者優勢：

Reciprocal Rank Fusion (RRF)

9.x 內建 RRF 融合演算法：

{
  "retriever": {
    "rrf": {
      "retrievers": [
        {
          "standard": {
            "query": {
              "match": { "title": "wireless headphones" }
            }
          }
        },
        {
          "knn": {
            "field":         "embedding",
            "query_vector":  [...],
            "k":             50,
            "num_candidates": 100
          }
        }
      ],
      "rank_window_size": 50,
      "rank_constant":    20
    }
  }
}

RRF 公式：score(d) = Σ 1/(k + rank_i(d))，把多個排序器結果合併。

Retriever 語法（9.x 新）

9.x 推出 retriever API，把 query 與 KNN 抽象為「retriever」可組合：

{
  "retriever": {
    "rrf": {
      "retrievers": [
        { "standard": { "query": { "match": { "content": "..." } } } },
        { "knn": { ... } },
        {
          "rule": {
            "retriever": { "standard": { ... } },
            "ruleset_ids": [ "promotion-rules" ]
          }
        }
      ]
    }
  }
}

RAG 實作步驟

Step 1：建立 index 與 inference endpoint

# 用 ELSER 最簡單
PUT /_inference/sparse_embedding/elser-rag
{
  "service": "elasticsearch",
  "service_settings": {
    "model_id":         ".elser_model_2",
    "num_allocations":  2
  }
}

# 或用外部 OpenAI
PUT /_inference/text_embedding/openai-embeddings
{
  "service": "openai",
  "service_settings": {
    "api_key": "sk-...",
    "model_id": "text-embedding-3-small"
  }
}

# 用 semantic_text 自動處理
PUT knowledge-base
{
  "mappings": {
    "properties": {
      "content": {
        "type":         "semantic_text",
        "inference_id": "elser-rag"
      },
      "metadata": {
        "properties": {
          "source":  { "type": "keyword" },
          "url":     { "type": "keyword" },
          "updated": { "type": "date" }
        }
      }
    }
  }
}

Step 2：Ingest 文件（會自動 chunk + embed）

POST knowledge-base/_doc
{
  "content":  "Elasticsearch 是基於 Lucene 的分散式搜尋與分析引擎...",
  "metadata": {
    "source":  "official-docs",
    "url":     "https://elastic.co/...",
    "updated": "2026-04-15"
  }
}

Step 3：Retrieval（hybrid + RRF）

{
  "retriever": {
    "rrf": {
      "retrievers": [
        {
          "standard": {
            "query": {
              "match": { "content": "ES cluster discovery" }
            }
          }
        },
        {
          "standard": {
            "query": {
              "semantic": {
                "field": "content",
                "query": "How do nodes find each other in Elasticsearch?"
              }
            }
          }
        }
      ],
      "rank_window_size": 20
    }
  },
  "size": 5,
  "_source": [ "content.text", "metadata" ]
}

Step 4：組裝 Prompt + 呼叫 LLM

Application 端做（Python 範例）：

from elasticsearch import Elasticsearch
from anthropic import Anthropic

es = Elasticsearch(...)
claude = Anthropic(api_key="sk-ant-...")

def rag_answer(question: str) -> str:
    # 1. Retrieval
    res = es.search(index="knowledge-base", body={
        "retriever": { "rrf": {
            "retrievers": [
                {"standard": {"query": {"match": {"content": question}}}},
                {"standard": {"query": {"semantic": {"field": "content", "query": question}}}}
            ]
        }},
        "size": 5
    })

    # 2. 組 context
    context = "\n\n".join([
        f"[{i+1}] {hit['_source']['content']}" 
        for i, hit in enumerate(res['hits']['hits'])
    ])

    # 3. 呼叫 LLM
    response = claude.messages.create(
        model="claude-opus-4-7",
        max_tokens=1024,
        messages=[{
            "role": "user",
            "content": f"Use the context to answer:\n\n{context}\n\nQuestion: {question}"
        }]
    )
    return response.content[0].text

Chunking 策略

長文件必須分塊。9.x semantic_text 內建 chunking：

{
  "mappings": {
    "properties": {
      "content": {
        "type":         "semantic_text",
        "inference_id": "my-elser",
        "chunking_settings": {
          "strategy":      "sentence",   // 或 word
          "max_chunk_size": 250,
          "sentence_overlap": 1
        }
      }
    }
  }
}

策略選擇：

  • sentence：依句子切，較自然
  • word：依單字數切，固定大小
  • recursive（社群常見）：分層切，先大段再小段

效能調校

HNSW 參數

參數預設影響
m16每節點連接數，更高 → 更準也更耗記憶體
ef_construction100建構時的搜尋深度，更高 → 更準也更慢建構
num_candidates—查詢時候選數，與 k 比例越高越準

9.x GPU 加速（Tech Preview）

9.3+ 支援 NVIDIA cuVS GPU 加速 HNSW build，索引速度可達 12x：

# elasticsearch.yml（需 NVIDIA Ampere GPU + Enterprise license）
xpack.searchable.snapshot.shared_cache.size: "5%"
indices.knn.use_gpu: true

Vector Search 與 RAG 最佳實踐

  • 新建 dense_vector index 用 BBQ 或 int8_hnsw，省 75-95% 記憶體
  • BBQ + rescore：oversample 3x 用 full-precision 重排前 K 個
  • 大規模 + 記憶體吃緊用 bbq_disk（DiskBBQ，9.2+）
  • 用 semantic_text 欄位 取代手刻 chunk + embed pipeline
  • Hybrid search 用 RRF，比單純 vector 或 BM25 都好
  • Filter pre-pruning：在 KNN 中加 filter 先過濾再向量搜
  • num_candidates ≥ k × 5 確保 recall
  • 用 retriever API（9.x）統一 vector / BM25 / rule 等多源
  • RAG 中先 retrieve top-20，再 rerank（用 Cohere rerank API 或 LLM 再排序）
  • 檢索品質可用 recall @ K 評估（用 ground truth 集驗證）
  • Application 層做最後的 LLM call，ES 只負責 retrieval（職責單一）

本章重點回顧

  • Vector search 用 dense_vector 欄位 + KNN query 達成語義搜尋
  • 9.x 多種量化選項：int8 / int4 / bbq / bbq_disk / bfloat16，BBQ 是新預設
  • HNSW 是內建演算法；m、ef_construction、num_candidates 是調校點
  • ELSER 是 Elastic 自家稀疏向量模型，不需外部 embedding service
  • semantic_text 欄位（9.x）自動 chunk + embed，最簡單實用
  • Hybrid search 用 RRF（reciprocal rank fusion）融合 BM25 + vector
  • 9.x retriever API 是 query / KNN / rule 的統一抽象
  • RAG = Retrieval（ES）+ Generation（LLM），ES 負責找資料
  • 9.3 GPU-accelerated HNSW（cuVS）tech preview，索引快 12x
