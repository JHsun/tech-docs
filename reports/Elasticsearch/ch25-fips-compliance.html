CH 25 · FIPS 與合規

適合對象 🔴 維運者 ⚡ 進階
閱讀時間 約 25 分鐘
前置知識 CH 19 TLS、CH 20-24 安全章節
本章目標 FIPS 140-2/140-3 啟用、加密 at-rest、合規架構（HIPAA、SOC 2、PCI-DSS）、私有部署

什麼是 FIPS？

FIPS（Federal Information Processing Standards） 140-2 / 140-3 是美國 NIST 訂的密碼學模組驗證標準：

  • 規範允許的密碼演算法、金鑰長度、雜湊函式
  • 美國聯邦政府、軍方、醫療、金融常要求 FIPS-validated 的密碼模組
  • 140-2 是廣泛採用的版本；140-3 自 2022 起成為新標準（過渡期至 2026 年）
⚠️ FIPS mode 是嚴格的限制，不是更安全的「升級版」 啟用 FIPS 後某些功能會受限：
  • 禁用部分 cipher suites（如 ChaCha20）
  • 禁用 BCRYPT（必須改用 PBKDF2）
  • 限制 key derivation 與 random number generation
  • 不建議在沒有合規需求時啟用

FIPS Image（最簡單方式）

Elastic 提供 FIPS-ready Docker image，已預裝 Oracle JDK 與 Bouncy Castle：

docker pull docker.elastic.co/elasticsearch/elasticsearch-fips:9.3.4

用法與一般 image 相同，只是底層 Java 與 crypto provider 已換好。

手動設定 FIPS Mode

Step 1：安裝 Oracle JDK 17+

# 從 Oracle 下載並安裝（接受授權）
sudo rpm -ivh jdk-17_linux-x64_bin.rpm
# 或 tar.gz：
sudo tar -xzf jdk-17_linux-x64_bin.tar.gz -C /opt

Step 2：下載 Bouncy Castle FIPS jars

從 bouncycastle.org/fips-java 下載：

  • bc-fips-1.0.2.x.jar（Crypto provider）
  • bctls-fips-1.0.x.x.jar（TLS provider）

放入 Java security extensions 目錄：

# 放入 Oracle JDK 的 lib/ext（或自訂位置然後加 -classpath）
sudo cp bc-fips-1.0.2.5.jar /opt/jdk-17/lib/
sudo cp bctls-fips-1.0.18.jar /opt/jdk-17/lib/

Step 3：修改 java.security

編輯 $JAVA_HOME/conf/security/java.security，把 BC FIPS providers 設為前兩個：

security.provider.1=org.bouncycastle.jcajce.provider.BouncyCastleFipsProvider
security.provider.2=org.bouncycastle.jsse.provider.BouncyCastleJsseProvider fips:BCFIPS
security.provider.3=SUN
security.provider.4=SunJSSE
# ... 其餘標準 provider

Step 4：jvm.options 啟用 FIPS

# /etc/elasticsearch/jvm.options.d/fips.options
-Dorg.bouncycastle.fips.approved_only=true
-Djava.security.policy=/etc/elasticsearch/fips.policy

fips.policy 檔案內容：

grant {
    permission java.security.SecurityPermission "putProviderProperty.BCFIPS";
    permission java.security.SecurityPermission "putProviderProperty.BCJSSE";
};

Step 5：elasticsearch.yml 啟用 FIPS mode

xpack.security.fips_mode.enabled: true

# 必要：使用 PBKDF2 而非 BCRYPT
xpack.security.authc.password_hashing.algorithm: pbkdf2_stretch_1000000

# API key 也要 FIPS 相容
xpack.security.authc.api_key.hashing.algorithm: pbkdf2_stretch

# Cache hash 也用 PBKDF2
xpack.security.authc.realms.native.native1.cache.hash_algo: pbkdf2_stretch

Step 6：重設所有 user 密碼

啟用 FIPS 前以 BCRYPT 雜湊的密碼無法在 FIPS mode 下驗證。必須全部重設：

sudo bin/elasticsearch-reset-password -u elastic
sudo bin/elasticsearch-reset-password -u kibana_system
# ... 對所有 reserved + native user

FIPS Mode 限制

限制影響
Cipher：禁用 ChaCha20、3DES、RC4必須使用 AES-GCM、AES-CBC（with HMAC）
Hash：禁用 MD5、SHA-1（用於簽章）SAML / OIDC IdP 必須支援 SHA-256+
Password：BCRYPT 禁用必須用 PBKDF2_STRETCH
RSA key 必須 ≥ 2048 bits1024 bit 憑證需更新
EC key：限制曲線使用 P-256 / P-384 / P-521
TLS：限制 TLS 1.2+禁用 TLS 1.0 / 1.1（本來也應禁）
不能用 PKCS#12 keystore（部分情境）改用 BCFKS 格式

轉換 keystore 為 BCFKS

keytool -importkeystore \
  -srckeystore transport.p12 -srcstoretype PKCS12 \
  -destkeystore transport.bcfks -deststoretype BCFKS \
  -providerclass org.bouncycastle.jcajce.provider.BouncyCastleFipsProvider \
  -providerpath /opt/jdk-17/lib/bc-fips-1.0.2.5.jar

並在 yml 設：

xpack.security.transport.ssl.keystore.type: BCFKS
xpack.security.transport.ssl.keystore.path: certs/transport.bcfks

加密 At-Rest（靜態資料加密）

ES 本身不提供應用層的 at-rest 加密，必須在儲存層處理：

Linux dm-crypt / LUKS（自管硬碟）

# 建立加密 partition
sudo cryptsetup luksFormat /dev/sdb1
sudo cryptsetup luksOpen /dev/sdb1 es-data
sudo mkfs.ext4 /dev/mapper/es-data
sudo mount /dev/mapper/es-data /var/lib/elasticsearch

金鑰管理可整合 LUKS + TPM 或 Vault。

圖 25-2：合規場景的網路隔離架構

關鍵原則：

  • ES 節點僅在私有子網路，無 public IP
  • Kibana 透過 LB / 反向代理對外，限定 IP whitelist 或 VPN/Zero Trust
  • S3 / KMS 用 VPC Endpoint 走私網，不繞 Internet
  • 跨叢集（CCS/CCR）用 VPC peering 或 Transit Gateway / Cloud Interconnect

合規場景 1：HIPAA（醫療）

HIPAA 對 Elasticsearch 的要求

  • ☐ 加密 in-transit：TLS 1.2+ 全程
  • ☐ 加密 at-rest：硬碟 / EBS 加密 + KMS
  • ☐ Authentication：MFA（透過 SAML/OIDC IdP）
  • ☐ Authorization：least privilege RBAC，DLS 限制只看 patient 自己的記錄
  • ☐ Audit：所有 access 紀錄至少 6 年
  • ☐ Backup & DR：snapshot 加密 + 跨 region 複製
  • ☐ Vendor BAA：與 Elastic / 雲端 provider 簽 Business Associate Agreement
  • ☐ Network：私有部署，無 public endpoint

合規場景 2：SOC 2

  • ☐ Audit log 集中收集 + 1 年以上保留
  • ☐ Change management：所有 security setting 變更走 PR + 審核
  • ☐ Access review：每季 review user/role
  • ☐ MFA 必須對所有人類 user 啟用
  • ☐ Incident response plan：定期演練
  • ☐ Vendor 風險管理：列入 Elastic 與雲端供應商

合規場景 3：PCI-DSS（信用卡）

  • ☐ 不要在 ES 儲存信用卡完整 PAN（請 tokenize）
  • ☐ 若必須儲存，欄位需 encrypted（用 ingest pipeline 在寫入前加密）
  • ☐ FIPS-validated cryptography（建議啟用 FIPS mode）
  • ☐ Audit log 保留 ≥ 1 年（其中 3 個月可即時查詢）
  • ☐ 季度 vulnerability scan + 年度 penetration test
  • ☐ Strong cryptography：TLS 1.2+，AES-128+，禁用 SSLv3

合規場景 4：GDPR

  • ☐ Data minimization：只儲存必要 PII，用 ingest pipeline 過濾
  • ☐ Pseudonymization：將 user_id 雜湊化（用 fingerprint processor）
  • ☐ 「被遺忘權」：能依 user_id 刪除所有相關文件（用 delete by query）
  • ☐ 資料居住地：ES cluster 部署在歐洲 region
  • ☐ DPA（Data Processing Agreement）與雲端供應商簽

最終 Hardening Checklist

網路

  • ☐ ES 不暴露 public Internet（VPC private subnet only）
  • ☐ Security Group 僅允許必要 port（9200/9300）來自必要來源
  • ☐ Kibana 對外用 reverse proxy + WAF
  • ☐ 跨叢集走 VPC Endpoint / Peering

存取

  • ☐ elastic superuser 僅 emergency 使用，平時走 MFA SSO
  • ☐ 所有 service 用獨立 API key + 最小權限
  • ☐ 用 SAML/OIDC + IdP 強制 MFA
  • ☐ Role mapping 走 group，不直接綁 user

資料

  • ☐ TLS 1.2+ 全程（transport + HTTP）
  • ☐ 硬碟 / EBS 加密 + KMS
  • ☐ Snapshot 加密 + 跨 region 複製
  • ☐ 敏感欄位 FLS 限制存取
  • ☐ 多租戶用 DLS + Kibana spaces 區隔

稽核

  • ☐ Audit log 啟用 + 集中到獨立 audit cluster
  • ☐ Watcher 設定關鍵告警
  • ☐ Audit log 保留期符合合規
  • ☐ 紀錄 ILM、role、user 變更（security_config_change）

維運

  • ☐ JVM heap 100% memory_lock
  • ☐ Bootstrap checks 全通過
  • ☐ 跨 AZ replica（HA）
  • ☐ Snapshot 自動化 + 定期還原驗證
  • ☐ Stack Monitoring 啟用，警報已測試

本章重點回顧

  • FIPS 140-2/140-3 是密碼學模組的合規標準，不等於「更安全」
  • 啟用 FIPS 需 Oracle JDK + Bouncy Castle FIPS provider；最簡單用 elasticsearch-fips Docker image
  • FIPS 必須用 PBKDF2_STRETCH 取代 BCRYPT；切換前要重設所有 user 密碼
  • FIPS 限制 cipher、hash、key size；某些 keystore 格式需轉 BCFKS
  • At-rest 加密要在儲存層實作（dm-crypt、EBS 加密、KMS）
  • 合規架構通用要素：私有網路、TLS 全程、MFA、最小權限 RBAC、audit log 長期保留
  • HIPAA、PCI-DSS、SOC 2、GDPR 各有額外要求（保留期、加密強度、資料居住地）
