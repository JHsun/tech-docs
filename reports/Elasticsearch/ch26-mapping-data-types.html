CH 26 · Mapping 與資料型別

適合對象 🟢 初學者 🟡 開發者
閱讀時間 約 30 分鐘
前置知識 CH 02 核心概念
本章目標 認識 mapping、所有資料型別、dynamic mapping、multi-fields、runtime fields、mapping 最佳實踐

什麼是 Mapping？

Mapping 是 Elasticsearch 中定義文件結構與欄位型別的 schema，類似關聯式資料庫的 table schema。但 ES 的 mapping 還包含：

  • 欄位型別（text、keyword、long、date...）
  • 分析器（analyzer）配置
  • 是否要建立索引（index: true/false）
  • 是否儲存原始值（store）
  • 多重欄位（multi-fields）
  • 動態欄位規則（dynamic templates）

Dynamic vs Explicit Mapping

Dynamic Mapping（自動推斷）

插入文件時 ES 自動依值推斷型別：

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/auto-index/_doc/1" -d '
{
  "title":      "Hello World",       // → text + .keyword (multi-field)
  "view_count": 42,                   // → long
  "is_active":  true,                 // → boolean
  "created_at": "2026-04-15",         // → date
  "tags":       ["es", "mapping"]     // → text + .keyword
}'

查看 ES 推斷的 mapping：

curl -X GET -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/auto-index/_mapping?pretty"
⚠️ Dynamic mapping 的陷阱
  • 第一次出現「123」會被當 long；之後出現「3.14」會報錯（型別衝突）
  • 「2026-04-15」會推斷為 date；但「2026/04/15 14:30」可能變 text
  • 欄位數會無限增長（mapping explosion）
  • 生產環境 → 明確定義 mapping

Explicit Mapping（手動定義）

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/products" -d '
{
  "mappings": {
    "properties": {
      "name":        { "type": "text", "fields": { "keyword": { "type": "keyword" } } },
      "sku":         { "type": "keyword" },
      "price":       { "type": "scaled_float", "scaling_factor": 100 },
      "stock":       { "type": "integer" },
      "category":    { "type": "keyword" },
      "description": { "type": "text", "analyzer": "standard" },
      "tags":        { "type": "keyword" },
      "created_at":  { "type": "date" },
      "is_featured": { "type": "boolean" },
      "location":    { "type": "geo_point" },
      "metadata":    { "type": "object", "dynamic": "true" }
    }
  }
}'

常見資料型別速查

Text vs Keyword（最重要的區別）

項目textkeyword
用途全文搜尋（match query）精確比對、聚合、排序
是否分析✅ 經 analyzer 拆 token❌ 整字當一個 token
聚合 aggregation❌（除非 fielddata=true）✅
排序 sort❌✅
doc values❌✅
例title、description、article bodytag、id、status、enum value

Multi-fields：同欄位多種索引

同一個欄位可同時用 text 與 keyword 索引：

{
  "title": {
    "type": "text",                    // 主欄位用 text 全文搜
    "analyzer": "standard",
    "fields": {
      "keyword": { "type": "keyword", "ignore_above": 256 },  // .keyword 子欄位
      "english": { "type": "text", "analyzer": "english" }    // .english 子欄位
    }
  }
}

使用：

// 全文搜尋
{ "match": { "title": "elasticsearch" } }

// 聚合（用 keyword 子欄位）
{ "aggs": { "by_title": { "terms": { "field": "title.keyword" } } } }

// 英文 stemmer 搜尋
{ "match": { "title.english": "running" } }   // 會 match "run", "ran"

數值型別

型別範圍用途
long-2^63 ~ 2^63-1大整數、ID、count
integer-2^31 ~ 2^31-1一般整數
short-32768 ~ 32767小整數
byte-128 ~ 127極小整數
double64-bit 浮點科學計算
float32-bit 浮點較不精確但省空間
half_float16-bit 浮點低精度（如 ML feature）
scaled_float儲存為 long × scaling_factor金額（推薦！）
unsigned_long0 ~ 2^64-1無號大整數
💡 金額用 scaled_float 而非 float 
{ "price": { "type": "scaled_float", "scaling_factor": 100 } }
19.99 內部存成 1999（long）。
  • 避免 float 精度問題（19.99 + 0.01 ≠ 20.00）
  • 儲存與計算更快
  • 輸入輸出仍是 float 形式

Date 型別

{
  "created_at": {
    "type": "date",
    "format": "strict_date_optional_time||epoch_millis"
  }
}

支援格式：

  • 2026-04-15
  • 2026-04-15T10:23:45
  • 2026-04-15T10:23:45.123Z
  • epoch milliseconds：1735689600000
  • 自訂 format："format": "yyyy-MM-dd HH:mm:ss"

型別用途範例
booleantrue/false"is_active": true

Object vs Nested：陣列中的物件

問題場景

{
  "user": [
    { "first": "Alice", "last": "Lin" },
    { "first": "Bob",   "last": "Wang" }
  ]
}

用 object 型別索引時，陣列會被「攤平」：

user.first: ["Alice", "Bob"]
user.last:  ["Lin", "Wang"]

結果：搜尋 user.first=Alice AND user.last=Wang 會誤中（Alice Wang）。

用 nested 解決

{
  "mappings": {
    "properties": {
      "user": {
        "type": "nested",
        "properties": {
          "first": { "type": "keyword" },
          "last":  { "type": "keyword" }
        }
      }
    }
  }
}

每個 nested 物件被當獨立 sub-document索引，必須用 nested query：

{
  "query": {
    "nested": {
      "path": "user",
      "query": {
        "bool": {
          "must": [
            { "match": { "user.first": "Alice" } },
            { "match": { "user.last":  "Lin"   } }
          ]
        }
      }
    }
  }
}
⚠️ Nested 開銷 每個 nested object 變一個 sub-doc，shard 中的 doc 數會膨脹。索引/查詢都比 object 慢。
  • 只在「陣列中物件需獨立精準匹配」時才用
  • 限制 nested 數量：index.mapping.nested_objects.limit: 10000
  • 替代：denormalize 改成 parent-child 或 join 型別（但都有自己的開銷）

Dense Vector：語義搜尋與 RAG

9.x 大幅強化向量搜尋。基本 mapping：

{
  "mappings": {
    "properties": {
      "embedding": {
        "type": "dense_vector",
        "dims": 1024,
        "index": true,
        "similarity": "cosine",     // dot_product / l2_norm / max_inner_product
        "index_options": {
          "type": "int8_hnsw",      // 9.x 預設量化壓縮
          "m": 16,
          "ef_construction": 100
        }
      }
    }
  }
}

向量量化（節省 75% 空間）

9.x 支援多種量化等級：

類型儲存適用
flat / hnsw原始 float32需最高精度時
int8_hnsw（9.x 預設）1 byte/dim，~75% 節省大多數場景
int4_hnsw0.5 byte/dim，~87.5% 節省大規模、精度可接受
bbq_hnswBetter Binary Quantization，1 bit/dim極大規模 RAG（9.x 新）

詳見 CH 60 RAG 章節。

Dynamic Templates：動態欄位規則

當無法預知所有欄位（log fields、metric tags）時，用 dynamic template 套規則：

{
  "mappings": {
    "dynamic_templates": [
      {
        "strings_as_keyword": {
          "match_mapping_type": "string",
          "mapping": {
            "type": "keyword",
            "ignore_above": 256
          }
        }
      },
      {
        "ips_as_ip": {
          "match": "*_ip",
          "mapping": { "type": "ip" }
        }
      },
      {
        "histograms_for_metrics": {
          "match": "*_hist",
          "mapping": { "type": "histogram" }
        }
      },
      {
        "long_strings_as_text": {
          "match_mapping_type": "string",
          "match": "msg*",
          "mapping": { "type": "text" }
        }
      }
    ],
    "properties": {
      "@timestamp": { "type": "date" }
    }
  }
}

規則先後順序很重要：第一個 match 的會生效。

Runtime Fields：查詢時計算

不索引、查詢時動態計算的欄位（schema-on-read）：

{
  "mappings": {
    "runtime": {
      "day_of_week": {
        "type": "keyword",
        "script": {
          "source": "emit(doc['@timestamp'].value.dayOfWeekEnum.toString())"
        }
      },
      "full_name": {
        "type": "keyword",
        "script": {
          "source": "emit(doc['first'].value + ' ' + doc['last'].value)"
        }
      }
    },
    "properties": {
      "@timestamp": { "type": "date" },
      "first":      { "type": "keyword" },
      "last":       { "type": "keyword" }
    }
  }
}

Runtime field 用法跟一般欄位一樣，可在 query / agg 中參照：

{
  "aggs": {
    "by_day": { "terms": { "field": "day_of_week" } }
  }
}
✅ Runtime field 的優勢
  • 無需 reindex 即可加新欄位
  • 適合臨時分析、debug
  • 不佔索引空間
  • 可在搜尋請求中臨時定義
缺點：每次查詢都要跑 script，比已索引欄位慢得多。常用的還是要 reindex 為一般欄位。

在搜尋請求中動態加 runtime field

GET my-index/_search
{
  "runtime_mappings": {
    "price_with_tax": {
      "type": "double",
      "script": {
        "source": "emit(doc['price'].value * 1.05)"
      }
    }
  },
  "query": {
    "range": { "price_with_tax": { "lte": 100 } }
  },
  "fields": [ "price", "price_with_tax" ]
}

索引選項：可關閉的功能

很多欄位的索引/儲存可關閉以省空間：

{
  "mappings": {
    "properties": {
      "internal_id": {
        "type": "keyword",
        "doc_values": false,    // 關閉 doc values（不能聚合/排序）
        "index": false          // 不索引（不能查詢，只回 _source）
      },
      "log_message": {
        "type": "match_only_text",   // 比 text 更省（不存 positions）
        "norms": false               // 關閉欄位長度 normalization
      },
      "huge_string": {
        "type": "keyword",
        "ignore_above": 1024    // 超過 1024 字元跳過索引（仍在 _source）
      }
    }
  }
}

Mapping 修改規則

建立 index 後，只有少數修改是允許的：

修改允許？
新增欄位✅ 可
新增 multi-field 子欄位✅ 可
新增 dynamic template✅ 可
修改欄位型別❌ 不可（需 reindex）
改變 analyzer❌ 不可（部分例外，需 reindex）
刪除欄位❌ 只能在 _source 移除（需 reindex）
修改 ignore_above✅ 可

新增欄位範例

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/products/_mapping" -d '
{
  "properties": {
    "rating": { "type": "float" },
    "reviewer_count": { "type": "integer" }
  }
}'

Mapping Explosion 防護

欄位太多會：

  • cluster state 過大（影響所有節點）
  • 記憶體用量爆炸（每欄位都吃 heap）
  • 查詢/索引變慢

限制：

PUT my-index/_settings
{
  "index.mapping.total_fields.limit": 1500,         // 預設 1000
  "index.mapping.depth.limit": 20,                  // 預設 20
  "index.mapping.nested_fields.limit": 50,          // 預設 50
  "index.mapping.field_name_length.limit": 100      // 預設 Long.MAX_VALUE
}

緩解策略

  1. 用 flattened 欄位：對動態 metadata 整個當一個欄位
  2. 用 nested 替代 dynamic mapping：把 key/value 對轉為 list of {name, value}
  3. 限定 dynamic mapping 範圍：用 "dynamic": "strict" 拒絕新欄位
  4. 檢視 cluster state 大小：
# 看 cluster state 占多少
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_cluster/state?filter_path=metadata.indices&pretty" | wc -c

# 找最寬的 mapping
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_cat/indices?v&h=index,fields&s=fields:desc" | head -20

Dynamic Mapping 三種模式

{
  "mappings": {
    "dynamic": "strict",      // 拒絕未定義欄位（推薦給 schema 嚴謹場景）
    // "dynamic": "true",     // 預設：自動推斷新欄位
    // "dynamic": "false",    // 接受但不索引（仍在 _source）
    // "dynamic": "runtime",  // 推為 runtime field（不入索引）
    "properties": {
      "title": { "type": "text" }
    }
  }
}

實務建議：

  • 業務 entity（products、orders）："strict" + explicit mapping
  • Log / metrics（schema 多變）："runtime" 或 dynamic templates 控制
  • 探索性 PoC："true"（預設）

Meta 與 Field Aliases

_meta：給 mapping 加註解

{
  "mappings": {
    "_meta": {
      "version": "2.0.1",
      "owner": "team-platform",
      "schema_url": "https://docs.example.com/schemas/products.json"
    },
    "properties": { ... }
  }
}

不影響搜尋，純為 documentation。

Field Alias：欄位別名

{
  "mappings": {
    "properties": {
      "user_full_name": { "type": "keyword" },
      "user.name":      { "type": "alias", "path": "user_full_name" }
    }
  }
}

用途：欄位改名時兼容舊查詢；統一不同來源的欄位命名。

Mapping 最佳實踐

  • Production 一律用 explicit mapping，不要靠 dynamic
  • 先想用途再選型別：要全文搜→text；要聚合/sort→keyword；要範圍查→numeric/date
  • 金額用 scaled_float，避免 float 精度問題
  • 不需要的欄位關 doc_values 或 index，省空間
  • 盡量避免 nested，只有真正需要時才用
  • 用 dynamic templates 處理 log/metric 動態欄位
  • 限制 total_fields.limit，配合監控警報
  • 用 _meta 紀錄 schema version，便於演進
  • 大量布林狀態用 enum keyword（active/inactive）而非多個 boolean
  • 需要常變 schema 時用 runtime fields 過渡，再決定要不要 reindex

本章重點回顧

  • Mapping = ES 的 schema，定義型別、analyzer、索引選項
  • Dynamic mapping 適合 PoC；生產用 explicit mapping
  • text（全文搜）vs keyword（精確/聚合/排序）是最重要的區別
  • Multi-fields 讓同欄位同時支援多種索引方式（title.keyword、title.english）
  • 金額用 scaled_float；陣列中物件用 nested（小心開銷）
  • 9.x 向量搜尋用 dense_vector + 量化（int8/int4/bbq_hnsw）
  • Dynamic templates 處理動態欄位；flattened 防 mapping explosion
  • Runtime fields 提供 schema-on-read，無需 reindex 即可加欄位
  • Mapping 大部分修改都需要 reindex，所以一開始就要設計好
