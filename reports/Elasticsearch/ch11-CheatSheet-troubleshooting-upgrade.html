🐢 A. 場景：查詢突然變慢

情境
命令 / Code
說明
1. 用 profile 找慢點
GET /my-index/_search
{
  "profile": true,
  "query":   { ... 你的慢 query ... }
}
看哪個 shard / phase 最慢；找熱點欄位
2. 啟用 slow log
PUT /my-index/_settings
{
  "index.search.slowlog.threshold.query.warn":  "10s",
  "index.search.slowlog.threshold.query.info":  "5s",
  "index.search.slowlog.threshold.fetch.warn":  "1s",
  "index.indexing.slowlog.threshold.index.warn": "10s"
}
寫到 logs/<cluster>_index_search_slowlog.json
3. 找執行中慢查詢
GET /_tasks?actions=*search*&detailed=true
看 running_time_in_nanos；找候選 cancel
4. 取消慢查詢
POST /_tasks/<task_id>/_cancel
中斷正在跑的搜尋
5. 看 search rate / latency
GET /_nodes/stats/indices/search?human
query_time / query_total / fetch_time
常見原因
• 沒用 filter context（每次重算 score）
• wildcard / regex 對大欄位
• fielddata on text 欄位
• shard 太多（搜尋 fanout 大）
• 沒 cache friendly（用 now 而不是 now/d）
對應修法見 CS 04

🚫 B. 場景：寫入被拒絕（rejected）

情境
命令 / Code
說明
1. 看 thread_pool
GET /_cat/thread_pool/write,bulk?v&h=node_name,name,active,queue,rejected
rejected > 0 = 寫入被拒；queue 持續滿表瓶頸
2. 看 indexing pressure
GET /_nodes/stats/indexing_pressure?human
memory.limit_in_bytes 與當前用量；過高表 backpressure
3. 看 indexing rate
GET /_nodes/stats/indices/indexing?human
index_total / index_time
4. 調 refresh_interval
PUT /my-index/_settings
{
  "index.refresh_interval": "30s"
}
預設 1s；批次寫入時拉長到 30s 大幅提速
5. 暫關 replica（一次性大量寫）
PUT /my-index/_settings
{ "index.number_of_replicas": 0 }
# 寫完後再開回
PUT /my-index/_settings
{ "index.number_of_replicas": 1 }
只在 reindex / 大量初始化時用
常見原因
• Bulk size 太大 / 太小（5-15 MB 甜蜜點）
• shard 數過多
• disk IO 飽和
• segment merge 跟不上
對應修法見 CH 49

🔥 C. 場景：Heap 過高 / GC 頻繁

情境
命令 / Code
說明
1. 看 heap 用量
GET /_cat/nodes?v&h=name,heap.percent,ram.percent,heap.current&s=heap.percent:desc
> 75% 開始警告；> 85% 嚴重
2. 看 GC 頻率
GET /_nodes/stats/jvm?human&pretty
gc.collectors.young.collection_count / old.collection_time_in_millis
3. 看 fielddata 占用
GET /_cat/fielddata?v&s=size:desc
找吃 heap 的欄位；text 欄位開 fielddata 是常見原因
4. 看 circuit breaker
GET /_nodes/stats/breaker?human
parent / fielddata / request 是否觸發
5. 清 fielddata cache
POST /_cache/clear?fielddata=true
POST /my-index/_cache/clear
緊急釋放；但會影響 agg 效能
常見原因
• 大量 unique terms agg（cardinality 高的 keyword）
• 開 fielddata on text
• shard 過多
• segment 過多
• HEAP 設太小（建議：節點 RAM 50%、上限 31GB）
對應修法見 CH 49

⚡ D. 場景：Master 不穩 / 選舉頻繁

情境
命令 / Code
說明
1. 確認當前 master
GET /_cat/master?v
GET /_cluster/state/master_node?pretty
看當前 master ID + 變動史
2. 看 master 變動
# Filebeat / 直接 grep
grep "elected-as-master" /var/log/elasticsearch/*.log
grep "master_not_discovered" /var/log/elasticsearch/*.log
頻繁選舉是大警訊
3. 看 pending tasks
GET /_cluster/pending_tasks?pretty
pending 持續累積 = master 處理不完
4. 看 health
GET /_health_report
9.x 統一健康報告（含 master_is_stable）
常見原因
• Master 節點負擔太重（同時是 data role）
• 網路不穩 / 跨 DC 高延遲
• discovery.seed_hosts 不正確
• master 數 = 偶數（容易 split-brain）
• 太多 cluster state changes（很多動態 mapping / template）
解法：3 個 dedicated master node；cluster.routing.allocation.enable 用 persistent；減少 mapping 變更
⚠️ Production master 配置黃金規則
  • 3 個 dedicated master nodes（不兼任 data）
  • 不同 AZ / 機架
  • 規格：4 core / 8 GB RAM 即可（master 工作負擔輕）
  • cluster.initial_master_nodes 只在初次啟動設，之後移除

💾 E. 場景：磁碟快滿

情境
命令 / Code
說明
1. 看磁碟用量
GET /_cat/allocation?v&h=node,shards,disk.indices,disk.used,disk.avail,disk.total,disk.percent&s=disk.percent:desc
哪個節點最滿
2. 找最大 index
GET /_cat/indices?v&s=store.size:desc&bytes=gb
依 size 排序
3. 立即 rollover
POST /logs-app/_rollover
產新 backing index；舊的可進 ILM 走 cold/frozen
4. 解除 flood_stage 鎖
PUT /_all/_settings
{ "index.blocks.read_only_allow_delete": null }
清空間後執行；否則 index 持續無法寫
5. 立即刪除舊 index
DELETE /old-logs-2024.*
最後手段；先 snapshot
長期
• ILM (CS 03) 自動 rollover + delete
• Searchable snapshot 移到 frozen
• 加節點或 disk
見 CS 03 + CS 10

🚀 H. Rolling Upgrade SOP（14 步）

9.x 預設啟用安全，所有 API 都需 HTTPS + auth；範例假設你已配置好 client。

情境
命令 / Code
說明
1. Snapshot 整 cluster
PUT /_snapshot/s3_repo/pre-upgrade-{{now/d}}
{
  "indices":              "*",
  "include_global_state": true
}
必做！含 global state 才能還原 setting
2. 跑 deprecation 檢查
GET /_migration/deprecations?pretty
看所有 critical / warning；critical 必須先解
3. Kibana Upgrade Assistant
Kibana → Stack Management → Upgrade Assistant
視覺化 deprecation + 自動修復精靈
4. 暫停 ML（如有）
POST /_ml/set_upgrade_mode?enabled=true
不需手動停每 job
5. 暫停 ILM / SLM
POST /_ilm/stop
POST /_slm/stop
避免升級期間 ILM 動作出錯
6. 對單一節點：關 shard 分配
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "primaries"
  }
}
只允許 primary 分配；replica 等升完再開
7. Flush translog
POST /_flush
把 translog flush 到 segment；9.x 已不需 synced_flush
8. 停節點 → 升級 → 啟動
# Linux deb/rpm
sudo systemctl stop elasticsearch
sudo apt-get install --only-upgrade elasticsearch
# 或
sudo yum upgrade elasticsearch

# 升級 plugins
bin/elasticsearch-plugin list
bin/elasticsearch-plugin remove <plugin>
bin/elasticsearch-plugin install <plugin>

sudo systemctl start elasticsearch
每次只升一個節點！
9. 等節點 join
GET /_cat/nodes?v
# 確認新節點版本顯示新版
驗證 Java 版本、TLS、認證 OK
10. 恢復分配
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": null
  }
}
null 重設為預設
11. 等 GREEN
GET /_cluster/health?wait_for_status=green&timeout=60m
可能等很久（recovery shard）
12. 重複 6-11 給每節點
建議順序：data nodes → coord-only → master nodes（最後）
一次只升一個
13. 全部升完：恢復 ML / ILM / SLM
POST /_ml/set_upgrade_mode?enabled=false
POST /_ilm/start
POST /_slm/start
依序開回
14. 後續監控 24-48 小時
看 deprecation 警告、cluster.health、indexing rate、search latency
提早發現問題

📋 I. 節點升級順序（重要！）

順序
原因
備註
1. **Frozen / Cold tier**
影響面最小
可單獨升級
2. **Warm tier**
搜尋仍可進行
較少寫入
3. **Hot tier (data)**
Bulk 寫入暫時降速
主資料節點
4. **Ingest-only**
Pipeline 處理層
若有獨立 ingest node
5. **Coordinating-only**
API gateway 層
若有獨立 coord node
6. **Master nodes**
最後升級
絕對最後，避免選舉混亂

🛤️ J. 版本升級路徑

升級類型
路徑
備註
8.x → 9.x
必須先到 8.19.x
8.18 → 9.0 是例外（同時發布）；其他必須 8.19 → 9.x
7.x → 9.x
兩階段：7.17 → 8.19 → 9.x
不能跳級
同 major minor 升
直接 rolling upgrade
如 9.0 → 9.3，無需先到中間版本
同 minor patch 升
無 deprecation 風險
如 9.3.1 → 9.3.4，最簡單
⚠️ 9.x 升級重點（截至 2026/05）
  • 從 8.x 升 → 必須先到 8.19.x（最後 8.x minor，與 9.0 同時發布的橋接版）
  • 例外：8.18 → 9.0 直升 OK
  • 9.0 → 9.3 是 minor 升級，rolling 即可
  • 9.3+ 把 rollup 已正式移除（多年來 deprecated）
  • Java SecurityManager 完全棄用，改 Entitlements（影響自定 plugins）
  • 升級前 indices 須是 8.0+ 建立的；7.x 建立的須先 reindex

🎚️ L. Allocation 控制（升級必備）

場景
命令
說明
完全停止分配
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "none"
  }
}
所有 shard 停動；只用於緊急 freeze
只允許 primary（rolling restart 前）
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "primaries"
  }
}
replica 不重建，等節點回來
重設為預設
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": null
  }
}
恢復為 all
排除特定節點
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.exclude._name": "old-node-1"
  }
}
下線節點前先 drain
暫停 rebalance
PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.rebalance.enable": "none"
  }
}
不影響新 shard 分配，只停 rebalance

🔀 M. Reroute 緊急操作

詳見 CheatSheet 01 · I。重點：

  • POST /_cluster/reroute?retry_failed=true：重試 max_retries 達到的 shard
  • move：強制移動 shard（從擁擠 → 空閒節點）
  • allocate_empty_primary：放棄 primary（資料丟失！）
  • allocate_stale_primary：使用舊版 primary 副本（部分資料）

(function () {
  function init() {
    document.querySelectorAll('main pre').forEach(function (pre) {
      if (pre.parentElement.classList.contains('code-wrap')) return;
      const wrap = document.createElement('div');
      wrap.className = 'code-wrap';
      pre.parentNode.insertBefore(wrap, pre);
      wrap.appendChild(pre);
      const lang = pre.querySelector('code')?.className.match(/language-(\w+)/)?.[1];
      if (lang) {
        const tag = document.createElement('span');
        tag.className = 'code-lang';
        tag.textContent = lang;
        wrap.appendChild(tag);
      }
      const btn = document.createElement('button');
      btn.className = 'copy-btn';
      btn.type = 'button';
      btn.textContent = '📋 複製';
      btn.addEventListener('click', function () {
        const text = pre.innerText;
        if (navigator.clipboard?.writeText) {
          navigator.clipboard.writeText(text).then(success).catch(fallback);
        } else {
          fallback();
        }
        function success() {
          btn.textContent = '✓ 已複製';
          btn.classList.add('copied');
          setTimeout(function () { btn.textContent = '📋 複製'; btn.classList.remove('copied'); }, 1800);
        }
        function fallback() {
          const ta = document.createElement('textarea');
          ta.value = text;
          document.body.appendChild(ta);
          ta.select();
          try { document.execCommand('copy'); success(); } catch (_) {}
          document.body.removeChild(ta);
        }
      });
      wrap.appendChild(btn);
    });
  }
  if (document.readyState === 'loading') document.addEventListener('DOMContentLoaded', init);
  else init();
})();

(function () {
  function init() {
    const toc = document.querySelector('.toc ul');
    if (!toc) return;
    const headings = document.querySelectorAll('main h2, main h3');
    headings.forEach(function (h) {
      if (!h.id) {
        h.id = h.textContent.trim().toLowerCase()
          .replace(/[^\w\u4e00-\u9fff\s-]/g, '')
          .replace(/\s+/g, '-');
      }
      const li = document.createElement('li');
      const a = document.createElement('a');
      a.href = '#' + h.id;
      a.textContent = h.textContent;
      a.className = 'toc-' + h.tagName.toLowerCase();
      li.appendChild(a);
      toc.appendChild(li);
    });
  }
  if (document.readyState === 'loading') document.addEventListener('DOMContentLoaded', init);
  else init();
})();

      if (el.dataset.source) el.innerHTML = el.dataset.source;
      else el.dataset.source = el.innerHTML;
      el.removeAttribute('data-processed');
    }
    if (myCounter !== renderCounter) return;
    try {
      await mermaid.run({ querySelector: '.mermaid' });
    } catch (e) { console.warn('Mermaid render error', e); }
  }

  function init() {
    renderAll();
    window.addEventListener('jhsun:theme-changed', renderAll);
  }

  if (document.readyState === 'loading') document.addEventListener('DOMContentLoaded', init);
  else init();
})();
