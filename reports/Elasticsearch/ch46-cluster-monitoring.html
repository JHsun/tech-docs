_cat APIs：人類友善的速查

API用途
_cat/healthcluster 整體健康
_cat/nodes節點清單與基本資訊
_cat/indices所有 index 與大小
_cat/shards所有 shard 分配狀態
_cat/allocation每節點 disk 用量
_cat/recovery正在 recovery 的 shard
_cat/thread_poolthread pool 狀態
_cat/pending_tasks等待中的 cluster tasks
_cat/master當前 master 節點
_cat/snapshotssnapshot 清單
_cat/segmentsLucene segment 細節
_cat/templatesindex templates
_cat/aliasesaliases

通用語法

# 含 header
GET /_cat/health?v

# 指定欄位
GET /_cat/nodes?v&h=name,role,heap.percent,ram.percent,cpu,disk.used_percent

# 排序
GET /_cat/indices?v&s=store.size:desc

# JSON 格式
GET /_cat/health?format=json

# 列出可用欄位
GET /_cat/nodes?help

每日必看

# 1. cluster 健康
GET /_cat/health?v
# epoch      timestamp cluster      status node.total node.data shards pri relo init unassign pending_tasks ...
# 1735689600 10:00:00  production   green  9          6         500    250 0    0    0        0             ...

# 2. 節點狀態
GET /_cat/nodes?v&h=name,role,heap.percent,ram.percent,cpu,load_1m,disk.used_percent,master&s=name

GET /_cat/shards?v&h=index,shard,prirep,state,unassigned.reason,node | grep -v STARTED

GET /_cat/indices?v&s=store.size:desc&h=index,docs.count,store.size,pri.store.size | head -20

_cluster APIs（程式化使用）

# 整體健康（最常用）
GET /_cluster/health?level=indices&pretty

# 等待 yellow 以上才繼續
GET /_cluster/health?wait_for_status=yellow&timeout=30s

# 完整 cluster state（量大！）
GET /_cluster/state?filter_path=metadata.indices.*.settings.index.number_of_shards

# Pending tasks
GET /_cluster/pending_tasks

# Allocation explain
POST /_cluster/allocation/explain
{
  "index": "my-index",
  "shard": 0,
  "primary": false
}

# 重新分配 stuck 的 shard
POST /_cluster/reroute?retry_failed=true

_nodes/stats（細緻指標）

GET /_nodes/stats?human&pretty

# 篩選某類別
GET /_nodes/stats/jvm,os,fs,thread_pool?human&pretty

# 單節點
GET /_nodes/{node-id}/stats

部署 Stack Monitoring

Metricbeat 模式（推薦）

Elastic 官方推薦的方式（取代舊的 internal collection）。在每個被監控節點裝 Metricbeat：

# 在 ES 節點上裝
sudo apt-get install metricbeat

# 啟用 Elasticsearch xpack module
sudo metricbeat modules enable elasticsearch-xpack

# 編輯 /etc/metricbeat/modules.d/elasticsearch-xpack.yml
sudo tee /etc/metricbeat/modules.d/elasticsearch-xpack.yml << 'EOF'
- module: elasticsearch
  xpack.enabled: true
  period: 10s
  hosts: [ "https://localhost:9200" ]
  username: "remote_monitoring_user"
  password: "${ES_PWD}"
  ssl.certificate_authorities: [ "/etc/metricbeat/ca.crt" ]
EOF

# 設輸出目的地（monitoring cluster）
sudo tee -a /etc/metricbeat/metricbeat.yml << 'EOF'
output.elasticsearch:
  hosts: [ "https://monitoring-cluster.example.com:9200" ]
  username: "beats_writer"
  password: "${MON_PWD}"
  ssl.certificate_authorities: [ "/etc/metricbeat/ca.crt" ]
EOF

sudo systemctl enable --now metricbeat

Kibana 模式同理（kibana-xpack module），Logstash 用 logstash-xpack。

Elastic Agent 模式（9.x 推薦）

9.x 已將 Elastic Agent 列為主推。一個 agent 取代 Metricbeat / Filebeat / Auditbeat：

# 在 Fleet 中為節點 enroll
elastic-agent enroll \
  --url=https://fleet.example.com:8220 \
  --enrollment-token=...

elastic-agent install

在 Kibana → Fleet → Integrations 加入「Elasticsearch」integration，自動配置監控收集。

Kibana Stack Monitoring UI

連到 monitoring cluster 的 Kibana → Stack Monitoring，可看到：

  • Cluster overview：cluster status、node 數、shard 數、用戶數
  • Nodes：每節點 CPU、heap、disk、indexing/search rate
  • Indices：indexing rate、search rate、segment 數、storage
  • Logs：cluster log（需 Filebeat 裝 elasticsearch module）
  • Kibana / Logstash / Beats 視角

為何要獨立 Monitoring Cluster？

  • 循環依賴：production 掛了，自我監控也丟，無法 troubleshoot
  • 資源隔離：監控查詢不要佔用 production 資源
  • 合規：audit/監控資料可獨立存取控制
  • 規模小：3 個 small node 就夠監控數十個 production nodes

關鍵 Cluster Alerts（內建）

Stack Monitoring 啟用後 Kibana 自動建立預設 alert rules：

Alert觸發條件
Cluster health狀態變 yellow / red
License expiration授權即將過期
CPU usage節點 CPU 持續高
Disk usage節點磁碟用量超過 watermark
JVM MemoryHeap 持續 > 85%
Missing monitoring data沒收到某節點 metrics
Thread pool rejectionsqueue 持續滿
CCR read exceptionsCCR 失敗
Large shardsshard 大小超過 50 GB

可在 Stack Monitoring 頁面 enable / 客製。

Prometheus / Grafana 整合

若你已有 Prometheus 生態，可用：

  • elasticsearch_exporter（社群）：把 ES metrics 轉 Prometheus 格式
  • Metricbeat → Prometheus remote write
  • Grafana 內建 Elasticsearch datasource：直接從 ES 查 logs 與 metrics 畫圖
# elasticsearch_exporter 範例
docker run -d \
  -p 9114:9114 \
  --name es_exporter \
  quay.io/prometheuscommunity/elasticsearch-exporter \
  --es.uri=https://es.example.com:9200 \
  --es.all

Slow Log：找慢查詢/慢索引

# 啟用 search slow log
PUT my-index/_settings
{
  "index.search.slowlog.threshold.query.warn":  "10s",
  "index.search.slowlog.threshold.query.info":  "5s",
  "index.search.slowlog.threshold.query.debug": "2s",
  "index.search.slowlog.threshold.fetch.warn":  "1s"
}

# 啟用 indexing slow log
PUT my-index/_settings
{
  "index.indexing.slowlog.threshold.index.warn":  "10s",
  "index.indexing.slowlog.threshold.index.info":  "5s",
  "index.indexing.slowlog.threshold.index.debug": "2s"
}

Log 寫到 {cluster}_index_search_slowlog.json，可用 Filebeat 收集分析。

執行中的長任務

# 列所有 task
GET /_tasks?detailed=true&group_by=parents

# 找跑超過 5 秒的 search
GET /_tasks?actions=*search*&detailed=true

# 取消
POST /_tasks/{task_id}/_cancel

Audit 自我監控（CH 24）

把 audit log 也送到 monitoring cluster，用 Watcher 監控異常：

  • 多次認證失敗 → 可能 brute force
  • access_denied 暴增 → role 設定錯
  • 新 user / role 被建立 → 異常授權變更

Health API（9.x 增強）

GET /_health_report

9.x 新的 unified health API，回傳：

  • cluster status（master、shards、ILM、SLM、disk、repository_integrity、disk）
  • 每個 indicator 的 impact、diagnosis、user_action

取代舊的多個 health 來源檢查。

最佳實踐

  • 用獨立 monitoring cluster，避免循環依賴
  • 9.x 用 Elastic Agent（取代 Metricbeat 模式）
  • 關鍵 alert：cluster health、heap、disk watermark、unassigned shards、license
  • 啟用 slow log，threshold 從寬到嚴調整
  • 加 cluster._cat 到 dashboard，每日早會看一眼
  • 9.x 用 _health_report 取代分散的健康檢查
  • Audit log 收到 monitoring cluster，與業務分離
  • 整合 PagerDuty：critical alert（heap > 90%、red status）走 oncall
  • 定期 fire drill：模擬 node down 看告警是否觸發

本章重點回顧

  • 監控分層：host → JVM → ES → stack（Kibana/Logstash/Beats）
  • _cat APIs 給人看；_cluster / _nodes/stats 給程式
  • 關鍵指標：heap %、disk %、CPU、unassigned shards、thread pool rejected
  • Stack Monitoring 自動收集所有 stack 元件指標
  • 9.x 推薦用 Elastic Agent 取代 Metricbeat 收集
  • 獨立 monitoring cluster 是必備配置（避免循環依賴）
  • 內建 alert rules 可直接 enable，省自己寫
  • Slow log 是找慢查詢/索引的標準工具
  • 9.x _health_report API 統一健康檢查
