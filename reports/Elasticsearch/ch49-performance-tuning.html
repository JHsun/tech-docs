Step 1：OS Baseline（CH 06 複習）

# File descriptors
ulimit -n              # 應 ≥ 65535

# /etc/security/limits.conf
elasticsearch  soft  nofile  65535
elasticsearch  hard  nofile  65535
elasticsearch  soft  memlock unlimited
elasticsearch  hard  memlock unlimited

# vm.max_map_count（mmap 索引必需）
sysctl vm.max_map_count
# /etc/sysctl.conf
vm.max_map_count = 262144
vm.swappiness = 1

# 關閉 swap（強烈建議）
sudo swapoff -a
# /etc/fstab 註解掉所有 swap 行

# Transparent Huge Pages 關閉
echo 'never' | sudo tee /sys/kernel/mm/transparent_hugepage/enabled

Step 2：Heap 與 OS Cache 黃金法則

✅ 50/50 黃金法則
  • Heap = RAM 的 50%，最多 31 GB（避免壓縮指標效能損失）
  • 剩下 50% 給 OS file system cache（Lucene 大量靠這個）
  • 例：64 GB 主機 → heap 31 GB、剩 33 GB 給 cache
  • 例：128 GB 主機 → 仍是 heap 31 GB、剩 97 GB 給 cache（更好）
# jvm.options.d/heap.options
-Xms31g
-Xmx31g

9.x 預設 G1GC，無需特別調整。

Step 3：Shard 設計

Shard 大小

  • 10 GB - 50 GB 是甜蜜點
  • 過小：cluster state 壓力大、overhead 高
  • 過大：recovery 慢、shrink 慢、search latency 增

Shard 數量目標

primary shards 數 ≈ 資料節點數 × 1.5（最大不超過節點 × 3）

例：6 個 hot data 節點，logs index 預期 100 GB
  primary shards = 6 × 1.5 = 9 → 取 6 或 9
  每 primary 約 100/6 = 16 GB（理想）

整體 cluster shard 上限

9.x 預設限制：每個非 frozen data 節點 1000 個 shard。

# 看 cluster 總 shard 數
GET /_cluster/health
# 太多就要用 ILM shrink + 整併小 index

# 調整上限（不建議無限放大）
PUT /_cluster/settings
{
  "persistent": {
    "cluster.max_shards_per_node": 1500
  }
}

Step 4：Bulk 寫入優化

Bulk request 基本

curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/x-ndjson' \
  "https://localhost:9200/_bulk" --data-binary @bulk.ndjson

bulk.ndjson 格式：

{ "index": { "_index": "logs", "_id": "1" } }
{ "@timestamp": "...", "msg": "..." }
{ "index": { "_index": "logs" } }
{ "@timestamp": "...", "msg": "..." }

Bulk size 調校

  • 單 request 5-15 MB（不是文件數！）
  • 太小：每筆 overhead 多
  • 太大：超過 http.max_content_length 預設 100 MB；shard 上的 buffer 也吃緊
  • 調 client batch 抓量 + 觀察 indexing latency 找甜蜜點

Write-heavy index 設定

{
  "settings": {
    "number_of_replicas":   0,                  // 寫入期間關 replica，事後再開
    "refresh_interval":     "30s",              // 不需即時搜尋就拉長
    "translog.durability":  "async",            // 改 async（風險：crash 可能丟最多 5s）
    "translog.sync_interval": "5s",
    "translog.flush_threshold_size": "1gb",
    "merge.scheduler.max_thread_count": 1       // HDD 用 1；NVMe 可預設
  }
}

寫入完成後恢復：

PUT logs-load/_settings
{
  "number_of_replicas": 1,
  "refresh_interval":   "1s",
  "translog.durability": "request"
}

POST logs-load/_forcemerge?max_num_segments=1   # 大量寫完後 merge（warm phase 才做）

Client 端

  • 用多個 client thread並行送 bulk（不是單一 thread 大 batch）
  • 每 thread 約 5-10 MB / batch
  • 適度 client backoff：429 (Too Many Requests) 重試前 wait
  • 用 keep-alive HTTP connection、TLS session resumption 減少 handshake

Step 5：Search 優化

能用 filter context 一定用

// ❌ 全 must（每個都計分）
{
  "query": {
    "bool": {
      "must": [
        { "match": { "description": "wireless" } },
        { "term":  { "category":    "audio" } },
        { "range": { "price":       { "lte": 200 } } }
      ]
    }
  }
}

// ✅ 計分的進 must，過濾的進 filter（cache 友善）
{
  "query": {
    "bool": {
      "must":   [ { "match": { "description": "wireless" } } ],
      "filter": [
        { "term":  { "category": "audio" } },
        { "range": { "price":    { "lte": 200 } } }
      ]
    }
  }
}

Search 速度提升參數

{
  "track_total_hits":    false,     // 不算精確總數
  "size":                10,
  "_source":             [ "title", "price" ],   // 限定欄位
  "preference":          "_local",   // 優先本機 shard
  "request_cache":       true,
  "query": { ... }
}

Cache 友善寫法

// ❌ now（每秒不同）
"range": { "@timestamp": { "gte": "now-1d" } }

// ✅ now/d（按日對齊，cache 命中高）
"range": { "@timestamp": { "gte": "now-1d/d" } }

Index 階段禁用不必要欄位

{
  "mappings": {
    "properties": {
      "internal_id": {
        "type": "keyword",
        "doc_values": false,    // 不需 sort/agg
        "index":      false      // 不需查詢，只回 _source
      },
      "log_message": {
        "type":  "match_only_text",   // 比 text 省（不存 positions）
        "norms": false                  // 關 length normalization
      }
    },
    "_source": {
      "includes": [ "title", "msg" ],
      "excludes": [ "huge_blob" ]
    }
  }
}

Step 6：Aggregation 優化

  • "size": 0：不要 hits
  • terms agg 用 keyword（不要 text）
  • cardinality precision_threshold：不需極精確就調低
  • shard_size = size × 1.5：terms 精度提升
  • composite agg：大量 bucket 時分頁
  • 避免 nested agg 過深：每層 sub-agg 都倍增成本

Step 7：Index Sorting（提升某些查詢）

建立 index 時指定 segment 內部排序：

{
  "settings": {
    "index": {
      "sort.field":  [ "@timestamp", "service.name" ],
      "sort.order":  [ "desc", "asc" ]
    }
  }
}

好處：

  • 對 sort 一致的查詢，可提早終止（找到 top N 就停）
  • 對 range 過濾常見欄位也快
  • 同欄位範圍的 doc 在 segment 中聚集，提升壓縮率

缺點：

  • 索引時要排序，indexing 略慢

Step 8：硬體選擇

磁碟

類型適合
NVMe SSDHot tier 必選；single-digit ms latency
SATA SSDwarm tier 可
HDD（spinning）cold tier；不要在 hot 用
EBS gp3AWS warm/cold；好調 IOPS
EBS io2AWS hot；穩定高 IOPS（貴）
EBS gp2已過時（IOPS 隨容量變）

CPU 與 RAM

  • RAM：32-64 GB（每節點），128 GB 是甜蜜點以下
  • CPU：8-16 核（modern Xeon / EPYC），避開 NUMA 跨槽
  • 避免 burstable instance（AWS T 系列），CPU 限速會卡住

網路

  • 10 Gbps 以上
  • 跨 AZ 流量是實質成本（AWS $0.01/GB），CCR / replica recovery 計入
  • 低 latency：AZ 內 RTT < 1ms；跨 region 慎用 CCR

Cloud 特殊考量

AWS

  • EBS gp3 vs io2：hot 用 io2 穩；warm 用 gp3 經濟
  • placement group: cluster 提升節點間頻寬
  • Instance：r6id.2xlarge（NVMe instance store + 64 GB RAM）給 hot
  • EBS-optimized 必開
  • EFS / FSx 不適合當 ES data volume（latency 不穩）

GCP

  • Compute Engine + local SSD 給 hot tier
  • Persistent SSD 給 warm
  • 跨 zone 流量計費，跨 region 較貴

Azure

  • Premium SSD v2 給 hot tier（IOPS 可單獨買）
  • Lsv3 系列含 NVMe local disk

Rally：官方 Benchmark 工具

esrally 是 Elastic 自家 benchmark 工具，能跑「track」（資料集 + 操作）：

# 安裝
pip install esrally

# 對外部 cluster 跑 official track
esrally race \
  --target-hosts="https://es.example.com:9200" \
  --client-options="basic_auth_user:'rally',basic_auth_password:'...',use_ssl:true,verify_certs:true,ca_certs:'ca.crt'" \
  --track=geonames \
  --challenge=append-no-conflicts \
  --pipeline=benchmark-only

# 列出 official tracks
esrally list tracks

常用 tracks：

  • geonames：地理資料、簡單 query
  • http_logs：HTTP log，aggregation
  • nyc_taxis：大量寫入 + 範圍查詢
  • so：StackOverflow data，全文搜尋
  • elastic/logs：模擬 OTel logs 寫入
  • dense_vector：向量搜尋

診斷工具：Profile API + Slow Log

Profile API

{
  "profile": true,
  "query": { ... }
}

Kibana → Dev Tools → Search Profiler 提供視覺化（建議用）。

Slow Log（CH 46 補充）

PUT my-index/_settings
{
  "index.search.slowlog.threshold.query.warn": "5s",
  "index.search.slowlog.threshold.query.info": "2s",
  "index.indexing.slowlog.threshold.index.warn": "5s"
}

用 Filebeat / Elastic Agent 收集 slow log 到另一 index 分析趨勢。

Hot Threads API：找熱點

curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_nodes/hot_threads"

顯示哪些 thread 在跑、跑什麼方法。常用於 CPU 高時找熱點。

效能調校 Checklist

寫入慢

  1. bulk size 是否在 5-15 MB？
  2. refresh_interval 是否設 30s 以上？
  3. replica 是否暫時關閉？
  4. thread_pool.write.rejected 是否頻繁？磁碟 IO 是否瓶頸？
  5. JVM heap 是否吃緊（> 75%）？
  6. 有沒有 ingest pipeline 拖慢？看 _nodes/stats/ingest
  7. shard 數量是否合理？（不要 100 個 1 GB shard）

查詢慢

  1. 用 Profile API 找慢 sub-query / agg
  2. filter context 用了嗎？now/d 對齊？
  3. shard 大小是否在 10-50 GB？
  4. OS file cache 是否夠（free -h 看）？
  5. track_total_hits 是否設 false？
  6. terms agg 是否用 keyword 欄位？
  7. routing 用了嗎（多租戶）？
  8. 是否有 deep pagination（> 10000）？改用 PIT

Cluster 不穩

  1. master 節點是否獨立？
  2. cluster state 是否過大（mapping explosion）？
  3. pending_tasks 是否堆積？
  4. 有沒有 long GC pause（> 1s）？
  5. 跨 AZ replica 是否合理？

效能調校最佳實踐

  • 先量測再調校：用 Rally 與 Profile API，別瞎猜
  • 一次只調一個變數，便於歸因
  • 建立 baseline：每月跑一次相同 benchmark
  • 大改前先在 staging 驗證
  • OS / heap 設定一勞永逸：不要忘記基本
  • shard 設計重於微調：50% 效能來自合理 shard 大小與數量
  • 讀寫分離：高寫入時 hot tier 與 search 分開（用 routing 或 separate index）
  • 用 ILM tier：成本與效能的最佳平衡
  • 監控 thread pool rejection：是過載最早的訊號
  • 定期 review 慢查詢 top 10 與 大 mapping top 10

本章重點回顧

  • 調校八大層級：OS → JVM → Cluster → Index 設計 → Bulk → Search → Aggregation → Tier
  • Heap 50% RAM，最多 31 GB；剩 50% 給 OS file cache
  • Shard 大小 10-50 GB，數量 ≈ 節點 × 1.5
  • Bulk 寫入：單 request 5-15 MB；寫入期間關 replica + 拉長 refresh
  • Search：filter context、track_total_hits=false、now/d 對齊提升 cache
  • Aggregation：size=0、shard_size = size × 1.5、合適的 precision_threshold
  • Index sorting 對固定 sort 查詢有奇效
  • 硬體：Hot tier 用 NVMe SSD；雲端慎選 instance type
  • esrally 是官方 benchmark 工具
  • 診斷三劍客：Profile API、Slow Log、Hot Threads API
  • Thread pool rejection 是過載的最早訊號
