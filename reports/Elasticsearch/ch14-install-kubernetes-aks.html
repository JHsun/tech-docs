CH 14 · Azure AKS 部署

適合對象 🔴 維運者 ⚡ 進階
閱讀時間 約 28 分鐘
前置知識 CH 11 · ECK Operator、Azure 基礎
本章目標 在 AKS 上部署生產級 ES、整合 Azure Disk、AGIC、Entra ID Workload Identity、Azure Blob snapshot

Step 1：建立 AKS 叢集

RESOURCE_GROUP=elastic-prod-rg
LOCATION=eastasia
CLUSTER_NAME=elastic-prod-aks
SUBSCRIPTION_ID=$(az account show --query id -o tsv)

# 1. Resource Group
az group create --name $RESOURCE_GROUP --location $LOCATION

# 2. AKS 叢集（Availability Zones 模式 + Workload Identity）
az aks create \
  --resource-group $RESOURCE_GROUP \
  --name $CLUSTER_NAME \
  --location $LOCATION \
  --kubernetes-version 1.31 \
  --node-count 1 \
  --node-vm-size Standard_D4ds_v5 \
  --zones 1 2 3 \
  --enable-managed-identity \
  --enable-oidc-issuer \
  --enable-workload-identity \
  --enable-azure-rbac \
  --network-plugin azure \
  --network-dataplane cilium \
  --enable-cluster-autoscaler \
  --min-count 1 \
  --max-count 3

# 3. 連線
az aks get-credentials --resource-group $RESOURCE_GROUP --name $CLUSTER_NAME

加入 Elasticsearch 專屬 node pools

# Master pool（小、跨 3 zones）
az aks nodepool add \
  --resource-group $RESOURCE_GROUP \
  --cluster-name $CLUSTER_NAME \
  --name masterpool \
  --node-count 3 \
  --node-vm-size Standard_D4ds_v5 \
  --zones 1 2 3 \
  --node-taints "dedicated=elasticsearch-master:NoSchedule" \
  --labels role=elasticsearch-master \
  --node-osdisk-type Managed \
  --node-osdisk-size 100

# Hot data pool
az aks nodepool add \
  --resource-group $RESOURCE_GROUP \
  --cluster-name $CLUSTER_NAME \
  --name datahotpool \
  --node-count 3 \
  --node-vm-size Standard_E16ds_v5 \
  --zones 1 2 3 \
  --node-taints "dedicated=elasticsearch-data-hot:NoSchedule" \
  --labels role=elasticsearch-data-hot \
  --node-osdisk-type Managed \
  --node-osdisk-size 200 \
  --enable-cluster-autoscaler \
  --min-count 3 \
  --max-count 9

Azure VM size 選擇

VM SizevCPU / RAM適用
Standard_D4ds_v54 / 16 GBMaster nodes
Standard_E16ds_v516 / 128 GBHot data（推薦）
Standard_E16ads_v516 / 128 GBHot data（AMD，省 10%）
Standard_D32ds_v532 / 128 GBCPU 密集（kNN、聚合）
Standard_E16pds_v516 / 128 GBCobalt 100 (ARM)，新世代省成本
Standard_L16s_v316 / 128 GB + NVMe本地 NVMe（注意：ECK 不建議用，重啟資料消失）

Step 2：Azure Disk 與 StorageClass

Azure Managed Disk 類型

類型IOPS（最高）適用
Standard HDD500不適合 ES
Standard SSD6000Warm/Cold tier
Premium SSD v120000Hot tier 基本款
Premium SSD v280000✅ Hot tier 推薦（IOPS/throughput 獨立計價）
Ultra Disk400000極致 IO；昂貴
⚠️ 不要用 Azure Files Azure Files (SMB/NFS) 對 ES 而言效能與一致性都不適合。一律使用 Managed Disk。

StorageClass 設定（Premium SSD v2）

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: premium-ssd-v2-elastic
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: disk.csi.azure.com
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
reclaimPolicy: Retain
parameters:
  skuName: PremiumV2_LRS
  cachingMode: None       # PremiumV2 必須是 None
  fsType: xfs
  DiskIOPSReadWrite: "5000"
  DiskMBpsReadWrite: "200"

傳統 Premium SSD v1：

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: premium-ssd-elastic
provisioner: disk.csi.azure.com
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
parameters:
  skuName: Premium_LRS
  cachingMode: ReadOnly
  fsType: xfs

Step 4：部署 Elasticsearch

apiVersion: elasticsearch.k8s.elastic.co/v1

            - name: elasticsearch

                - name: ES_JAVA_OPTS

                limits:   { memory: 4Gi, cpu: 2 }
      volumeClaimTemplates:
        - metadata: { name: elasticsearch-data }
          spec:

            resources: { requests: { storage: 50Gi } }

    - name: data-hot

        metadata:
          labels:
            azure.workload.identity/use: "true"
        spec:
          serviceAccountName: elasticsearch-snapshot
          tolerations:
            - key: dedicated
              value: elasticsearch-data-hot
              effect: NoSchedule
          nodeSelector:
            role: elasticsearch-data-hot
          initContainers:
            - name: sysctl
              securityContext: { privileged: true, runAsUser: 0 }
              command: ['sh', '-c', 'sysctl -w vm.max_map_count=262144']
          containers:
            - name: elasticsearch
              env:
                - name: ES_JAVA_OPTS
                  value: "-Xms16g -Xmx16g"
              resources:
                requests: { memory: 32Gi, cpu: 4 }
                limits:   { memory: 32Gi, cpu: 8 }
      volumeClaimTemplates:
        - metadata: { name: elasticsearch-data }

Step 5：Microsoft Entra ID Workload Identity

類似 EKS 的 IRSA、GKE 的 Workload Identity，AKS 用 Microsoft Entra ID Workload Identity（前身 Pod Identity v2）讓 K8s SA 對應到 Entra ID 應用程式身份。

建立 User-Assigned Managed Identity

IDENTITY_NAME=elasticsearch-snapshot-id
KEYVAULT_NAME=elastic-prod-kv
STORAGE_ACCOUNT=elasticprodsnapshots

# 建立 Managed Identity
az identity create \
  --name $IDENTITY_NAME \
  --resource-group $RESOURCE_GROUP \
  --location $LOCATION

IDENTITY_CLIENT_ID=$(az identity show -g $RESOURCE_GROUP -n $IDENTITY_NAME --query clientId -o tsv)
IDENTITY_PRINCIPAL_ID=$(az identity show -g $RESOURCE_GROUP -n $IDENTITY_NAME --query principalId -o tsv)

建立 Storage Account 與授權

# 建立 Storage Account（v2）
az storage account create \
  --name $STORAGE_ACCOUNT \
  --resource-group $RESOURCE_GROUP \
  --location $LOCATION \
  --sku Standard_LRS \
  --kind StorageV2 \
  --min-tls-version TLS1_2 \
  --allow-blob-public-access false

# 建立 container
az storage container create \
  --account-name $STORAGE_ACCOUNT \
  --name snapshots \
  --auth-mode login

# 給 Managed Identity Storage Blob Data Contributor 角色
SCOPE=$(az storage account show -n $STORAGE_ACCOUNT -g $RESOURCE_GROUP --query id -o tsv)
az role assignment create \
  --assignee-object-id $IDENTITY_PRINCIPAL_ID \
  --assignee-principal-type ServicePrincipal \
  --role "Storage Blob Data Contributor" \
  --scope $SCOPE

建立 Federated Credential

OIDC_ISSUER=$(az aks show -n $CLUSTER_NAME -g $RESOURCE_GROUP --query "oidcIssuerProfile.issuerUrl" -o tsv)
KSA_NAMESPACE=elastic
KSA_NAME=elasticsearch-snapshot

az identity federated-credential create \
  --name "elastic-snapshot-fc" \
  --identity-name $IDENTITY_NAME \
  --resource-group $RESOURCE_GROUP \
  --issuer $OIDC_ISSUER \
  --subject "system:serviceaccount:${KSA_NAMESPACE}:${KSA_NAME}" \
  --audience api://AzureADTokenExchange

建立 K8s ServiceAccount

apiVersion: v1
kind: ServiceAccount
metadata:
  name: elasticsearch-snapshot
  namespace: elastic
  annotations:
    azure.workload.identity/client-id:    # 從上方步驟取得

Step 6：Azure Blob Snapshot Repository

註冊 repository

對於使用 Workload Identity 的 Azure，需在 elasticsearch-keystore 設定 client：

# 透過 ECK Secret 注入（不是手動 keystore）
kubectl create secret generic azure-storage-creds \
  --from-literal=azure.client.default.account=$STORAGE_ACCOUNT \
  -n elastic

修改 Elasticsearch CR 增加 secureSettings：

spec:
  secureSettings:
    - secretName: azure-storage-creds
  # ... 其他設定不變

註冊 repository：

PASSWORD=$(kubectl get secret production-es-elastic-user -n elastic \
  -o go-template='{{.data.elastic | base64decode}}')

kubectl exec -it production-es-data-hot-0 -n elastic -- \
  curl -k -u elastic:$PASSWORD -X PUT "https://localhost:9200/_snapshot/azure_repo" \
  -H 'Content-Type: application/json' \
  -d '{
    "type": "azure",
    "settings": {
      "container": "snapshots",
      "base_path": "production-cluster"
    }
  }'

kubectl exec -it production-es-data-hot-0 -n elastic -- \
  curl -k -u elastic:$PASSWORD -X POST "https://localhost:9200/_snapshot/azure_repo/_verify"

Step 7：Application Gateway Ingress Controller (AGIC)

AKS 推薦的 Ingress 是 AGIC，整合 Azure Application Gateway（含 WAF v2）。

啟用 AGIC add-on

APPGW_NAME=elastic-prod-appgw

# 建立 Application Gateway 子網路（如尚未有）
az network vnet subnet create \
  --name appgw-subnet \
  --resource-group $RESOURCE_GROUP \
  --vnet-name aks-vnet \
  --address-prefix 10.0.4.0/24

# 啟用 AGIC（會自動建立 App Gateway）
az aks enable-addons \
  --resource-group $RESOURCE_GROUP \
  --name $CLUSTER_NAME \
  --addons ingress-appgw \
  --appgw-name $APPGW_NAME \
  --appgw-subnet-id $(az network vnet subnet show -g $RESOURCE_GROUP --vnet-name aks-vnet -n appgw-subnet --query id -o tsv)

Availability Zones vs Availability Set

Azure 的高可用機制：

  • Availability Zones（推薦）：跨多個 datacenter，最多 99.99% SLA。AKS 建立時 --zones 1 2 3
  • Availability Set：同 datacenter 內分散到不同 fault domain，99.95% SLA。較舊機制
  • 對於 ES，必選 Availability Zones（zoneAwareness 直接對應）

費用優化

  • Reserved Instance：1/3 年保留可省 20-60%
  • Azure Hybrid Benefit：擁有 Windows Server 授權的話，Linux VM 也能使用此福利？實際只對 Windows VM；ES 用不上
  • Spot Node Pool：⚠️ 同 EKS/GKE，ES 資料節點不要用 spot
  • Cobalt 100 (ARM)：v5 系列的 p 變體（如 E16pds_v5）較 Intel/AMD 便宜 20%
  • Premium SSD v2：相同效能下比 v1 便宜，且 IOPS/throughput 獨立計價，適合 hot tier
  • Storage Account lifecycle：snapshot 30 天後轉 Cool tier，90 天 Archive

Entra ID 整合 Kibana 認證

用 OIDC 讓 Kibana 接受 Entra ID 登入（CH 21 詳述）：

# 在 Elasticsearch CR 加入
spec:
  config:

      order: 1
      rp.client_id:

      rp.redirect_uri: "https://kibana.example.com/api/security/oidc/callback"
      op.issuer: "https://login.microsoftonline.com/{TENANT_ID}/v2.0"
      op.authorization_endpoint: "https://login.microsoftonline.com/{TENANT_ID}/oauth2/v2.0/authorize"
      op.token_endpoint: "https://login.microsoftonline.com/{TENANT_ID}/oauth2/v2.0/token"
      op.jwkset_path: "https://login.microsoftonline.com/{TENANT_ID}/discovery/v2.0/keys"
      op.userinfo_endpoint: "https://graph.microsoft.com/oidc/userinfo"
      claims.principal: email

AKS 上常見問題

症狀處理
Premium SSD v2 跨 zone 無法 mountv2 只能在同 zone；StorageClass 設 WaitForFirstConsumer
Workload Identity 無效確認 cluster 啟用 --enable-oidc-issuer --enable-workload-identity，pod label 加 azure.workload.identity/use: "true"
AGIC 無法產生 LB rule檢查 App Gateway SKU 是 Standard_v2 或 WAF_v2
節點升級造成 pods evictnode pool --max-surge=10% 並設 PodDisruptionBudget
Azure Disk attach 慢（> 1 min）用 v5 系列 VM（vCPU/disk attach 比例更佳），避免 D2 系列小規格

本章重點回顧

  • AKS 建議：v5 系列 VM、Availability Zones（--zones 1 2 3）、啟用 OIDC + Workload Identity
  • Master 用 D4ds_v5；Hot data 用 E16ds_v5 或 ARM 變體 E16pds_v5
  • 儲存：Premium SSD v2 為主流選擇；極致 IO 用 Ultra Disk；不要用 Azure Files
  • 用 Microsoft Entra ID Workload Identity 取代 service principal secret
  • 用 AGIC（Application Gateway Ingress Controller）暴露 Kibana，含 WAF v2
  • Snapshot 寫到 Azure Blob Storage，配合 Storage Account lifecycle 自動 tiering
  • Entra ID OIDC 可整合 Kibana 認證
