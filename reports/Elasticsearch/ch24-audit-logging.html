CH 24 · 稽核日誌（Audit Logging）

適合對象 🔴 維運者 ⚡ 進階
閱讀時間 約 22 分鐘
前置知識 CH 19-23 安全章節
本章目標 啟用稽核日誌、event types、log filtering、解析 JSON 結構、合規場景應用

為何需要稽核日誌？

  • 合規需求：HIPAA、PCI-DSS、SOC 2、GDPR、ISO 27001 都要求紀錄誰存取了什麼資料
  • 異常偵測：監控失敗登入、權限提升、敏感資料存取
  • 事件回溯：發生資安事件時還原 timeline
  • 內部稽核：定期檢視管理員操作軌跡
⚠️ Audit logging 需 Gold 訂閱以上 Audit logging 是 Gold/Platinum 級功能。Basic 訂閱不可用。但 Basic 仍可用 logger.org.elasticsearch... 開高 log level 達到部分目的（不正式，不建議）。

啟用稽核日誌

在每個節點的 elasticsearch.yml：

xpack.security.audit.enabled: true

重啟節點後，$ES_PATH_CONF/../logs/{cluster_name}_audit.json 會開始產出 JSON Lines 格式的稽核事件。

調整哪些 event 紀錄

xpack.security.audit.enabled: true

# 加入額外 event（除預設外）
xpack.security.audit.logfile.events.include:
  - access_denied
  - authentication_failed
  - authentication_success    # 新增
  - connection_denied
  - tampered_request
  - run_as_denied
  - run_as_granted
  - security_config_change

# 排除某些 event（即使在 include 中）
xpack.security.audit.logfile.events.exclude:
  - access_granted             # 量太大，排除

# 是否紀錄 request body
xpack.security.audit.logfile.events.emit_request_body: false   # 預設 false
⚠️ 啟用 emit_request_body 的注意事項 紀錄 request body 可看到完整 query 內容，但：
  • 會大量增加 log 體積
  • 密碼、API key 等敏感字串會被記錄到 log（即使 ES 嘗試 redact，仍應視為高敏感）
  • 某些合規場景反而要求不要記錄含 PII 的 query body

Log Filtering Policies

限制 audit 範圍，避免無關事件刷爆 log：

xpack.security.audit.logfile.events.include: [ "access_granted", "access_denied" ]

# Policy 1：忽略某些 user
xpack.security.audit.logfile.events.ignore_filters.healthcheck:
  users: [ "monitoring_user", "kibana_system" ]
  actions: [ "*" ]

# Policy 2：忽略某些 realm
xpack.security.audit.logfile.events.ignore_filters.system_realms:
  realms: [ "_es_api_key", "_service_account" ]

# Policy 3：忽略特定 indices（system indices）
xpack.security.audit.logfile.events.ignore_filters.system_indices:
  indices: [ ".monitoring-*", ".kibana_*", ".security-*" ]

# Policy 4：忽略低風險 actions
xpack.security.audit.logfile.events.ignore_filters.read_only_actions:
  actions:
    - "indices:data/read/search"
    - "cluster:monitor/health"

支援的 filter 欄位：users、realms、roles、indices、actions。每個都可用 wildcard。

Tip：先看再過濾

建議流程：先全開觀察 1-2 天 → 找出量最大的 user / action → 用 ignore_filters 排除無意義事件 → 持續優化。

把 Audit Log 送進 ES 自己

用 Filebeat 收 audit JSON，倒入 ES indices，然後在 Kibana 查詢/視覺化：

# /etc/filebeat/filebeat.yml
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /var/log/elasticsearch/*_audit.json
    json.keys_under_root: true
    json.add_error_key: true
    fields:
      data_stream.dataset: elasticsearch.audit
      data_stream.namespace: production

processors:
  - add_host_metadata: ~
  - add_cloud_metadata: ~

output.elasticsearch:
  hosts: [ "https://audit-cluster.example.com:9200" ]
  api_key: "..."
  ssl.certificate_authorities: [ "/etc/filebeat/ca.crt" ]
💡 用獨立的 audit cluster 別把生產 ES 的 audit log 灌回同一 cluster：
  • 避免循環依賴（生產 cluster 掛了，audit log 也丟）
  • 合規需求：稽核資料應與業務資料分離
  • Audit cluster 可較小規模（只裝資安/維運少數 user）

用 Filebeat module（更省事）

sudo filebeat modules enable elasticsearch
# /etc/filebeat/modules.d/elasticsearch.yml 編輯：

- module: elasticsearch
  audit:
    enabled: true
    var.paths: [ "/var/log/elasticsearch/*_audit.json" ]

module 自帶 ingest pipeline，會幫你把 JSON 解析成 ECS 欄位。

日誌輪替

Audit log 用 log4j2 同樣機制輪替。預設配置（$ES_PATH_CONF/log4j2.properties）：

appender.audit_rolling.type = RollingFile
appender.audit_rolling.name = audit_rolling
appender.audit_rolling.fileName = ${sys:es.logs.base_path}${sys:file.separator}${sys:es.logs.cluster_name}_audit.json
appender.audit_rolling.layout.type = PatternLayout
appender.audit_rolling.layout.pattern = %m%n
appender.audit_rolling.filePattern = ${sys:es.logs.base_path}${sys:file.separator}${sys:es.logs.cluster_name}_audit-%d{yyyy-MM-dd}-%i.json.gz
appender.audit_rolling.policies.type = Policies
appender.audit_rolling.policies.time.type = TimeBasedTriggeringPolicy
appender.audit_rolling.policies.time.interval = 1
appender.audit_rolling.policies.time.modulate = true
appender.audit_rolling.policies.size.type = SizeBasedTriggeringPolicy
appender.audit_rolling.policies.size.size = 1GB
appender.audit_rolling.strategy.type = DefaultRolloverStrategy
appender.audit_rolling.strategy.fileIndex = nomax
appender.audit_rolling.strategy.action.type = Delete
appender.audit_rolling.strategy.action.basepath = ${sys:es.logs.base_path}
appender.audit_rolling.strategy.action.condition.type = IfFileName
appender.audit_rolling.strategy.action.condition.glob = ${sys:es.logs.cluster_name}_audit-*
appender.audit_rolling.strategy.action.condition.nested_condition.type = IfAccumulatedFileSize
appender.audit_rolling.strategy.action.condition.nested_condition.exceeds = 50GB

重點：

  • 每天輪替 + 單檔 1 GB 上限
  • 累積總大小 > 50 GB 時刪舊檔
  • 合規場景應改為依保留年限刪除（例如 PCI-DSS 要求 1 年）

合規場景與 audit 設定建議

HIPAA（醫療資料）

  • 啟用 access_granted、access_denied
  • 不要 emit_request_body（避免 PII 流出）
  • Audit 至少保留 6 年
  • Audit cluster 與業務 cluster 分離

PCI-DSS（信用卡）

  • 所有 admin 行為、所有對 cardholder data 的存取都要紀錄
  • Audit 至少保留 1 年（其中 3 個月可立即查詢）
  • Audit log 完整性保護（防止竄改）—— 寫入 immutable storage 或啟用 write-once

SOC 2

  • 監控管理者帳號活動（user_admin、role 變更）
  • Audit log 集中收集 + 至少 1 年
  • 定期檢視 access 異常

GDPR

  • 對個資的 access 都需可追蹤
  • 當 user 行使「被遺忘權」時，audit log 中的個人資訊也要刪除（用 logstash filter / ingest pipeline mask）

Audit Logging Checklist

  • ☐ 已 xpack.security.audit.enabled: true 並重啟所有節點
  • ☐ 確認 audit log 寫入並有 rotation
  • ☐ Filebeat 將 audit log 收集到獨立 audit cluster
  • ☐ 設定 ignore_filters 排除無意義流量（kibana_system 心跳、monitoring）
  • ☐ 是否 emit_request_body 由合規團隊決定
  • ☐ Watcher 設定關鍵告警（多次認證失敗、access_denied 暴增、role 異動）
  • ☐ 保留期符合合規需求（HIPAA 6 年、PCI-DSS 1 年、SOC 2 1 年...）
  • ☐ Audit log 寫入後不可竄改（write-once / immutable cold tier）
  • ☐ Kibana 中已建立 audit dashboard 提供日常查詢

本章重點回顧

  • Audit logging 為 Gold/Platinum 功能，啟用後在 {cluster}_audit.json 寫入 JSON Lines
  • 主要 event：認證（success/failed）、授權（granted/denied）、連線（granted/denied）、tampered、run_as
  • access_granted 量極大，預設啟用但通常需 ignore_filters 排除大宗
  • emit_request_body 預設 false，啟用前評估隱私與合規影響
  • Log filtering policies 可依 user/realm/role/index/action 排除事件
  • 用 Filebeat module 把 audit log 灌進獨立 audit cluster，與業務分離
  • 用 Watcher 監控異常（多次失敗登入、敏感 index 異常存取）
  • 合規場景的保留年限與 immutable 要求要在 ILM / 儲存層滿足
