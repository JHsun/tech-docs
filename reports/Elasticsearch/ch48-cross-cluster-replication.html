CH 48 · Cross-Cluster Replication (CCR)

前置知識 CH 04 discovery、CH 22 API keys、CH 47 snapshot

CCR 是什麼？

CCR（Cross-Cluster Replication）讓一個 cluster 的 index 即時複製到另一個 cluster 的 follower index，背後重播 leader 的寫入操作（透過 soft deletes 機制）。

CCR 的三大用途

用途架構說明
Disaster Recoveryuni-directional主中心掛了，切到備援中心查詢
Data Localityuni-directional × N把資料複製到多個地理區，使用者就近查詢
Centralized Reporting多 leader → 1 follower多個小 cluster 的資料匯到中央 cluster 做分析
Bi-directional互相 follow兩 cluster 都可寫不同 index，互相同步（複雜，慎用）
⚠️ CCR 需 Platinum 訂閱 Basic / Gold 訂閱不可用。

CCR vs Snapshot

項目CCRSnapshot
同步即時性近即時（秒級）排程（小時/天）
RPO秒排程間隔
RTO秒（切 alias 即可查）分鐘到小時（restore 時間）
儲存成本2x（兩 cluster 都要硬碟）1x + S3 費用
網路成本持續流量排程時段流量
授權PlatinumBasic 即可
適合RPO/RTO 嚴格、跨地理查詢一般備份、長期保存

實務上常同時用：CCR 處理近期資料的 DR，snapshot 處理長期保存。

前置條件：Soft Deletes

CCR 透過 leader 上保留的「soft delete history」讓 follower 重播。Index 必須啟用 soft deletes：

  • 7.0 之後新建的 index 預設啟用 ✅
  • 7.0 之前的 index 必須 reindex 才能用 CCR

新建時可調整 retention：

{
  "settings": {
    "index.soft_deletes.enabled":            true,
    "index.soft_deletes.retention_lease.period": "12h"
  }
}

設定 Remote Cluster 連線

9.x CCR 推薦走 dedicated remote cluster interface（port 9443）+ API Key 模式，比舊的 transport 9300 + cert 安全。

Step 1：Leader cluster 啟用 remote interface

# elasticsearch.yml on Leader nodes
remote_cluster_server.enabled: true
remote_cluster.port: 9443

# TLS（必設）
xpack.security.remote_cluster_server.ssl.enabled: true
xpack.security.remote_cluster_server.ssl.key:         /etc/elasticsearch/certs/remote.key
xpack.security.remote_cluster_server.ssl.certificate: /etc/elasticsearch/certs/remote.crt
xpack.security.remote_cluster_server.ssl.certificate_authorities: [ "/etc/elasticsearch/certs/ca.crt" ]

Step 2：Leader 產生 cross-cluster API key

curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://leader:9200/_security/cross_cluster/api_key" -d '
{
  "name": "follower-cluster-key",
  "access": {
    "replication": [
      { "names": [ "logs-*", "metrics-*" ] }
    ]
  }
}'

回應會給 encoded 欄位（base64）。立即抄下來。

Step 3：Follower cluster 註冊 remote cluster

在 Follower nodes 設 follow role：

# Follower 節點 yml
node.roles: [ "data_hot", "data_content", "remote_cluster_client" ]

API Key 存 keystore：

sudo /usr/share/elasticsearch/bin/elasticsearch-keystore add \
  cluster.remote.leader-prod.credentials
# 貼上 encoded API key

# Reload
curl -X POST -u elastic:xxx --cacert ca.crt \
  "https://follower:9200/_nodes/reload_secure_settings"

設定 remote cluster：

PUT /_cluster/settings
{
  "persistent": {
    "cluster": {
      "remote": {
        "leader-prod": {
          "mode": "proxy",
          "proxy_address":           "leader.example.com:9443",
          "server_name":             "leader.example.com",
          "skip_unavailable":        false
        }
      }
    }
  }
}

Step 4：驗證

GET /_remote/info?pretty

應看到 connected: true、num_nodes_connected > 0。

建立 Follower Index

# 在 Follower cluster 上執行
curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://follower:9200/logs-app-2026.04.15-000003/_ccr/follow?wait_for_active_shards=1" -d '
{
  "remote_cluster":  "leader-prod",
  "leader_index":    "logs-app-2026.04.15-000003",
  "max_read_request_operation_count":      5120,
  "max_outstanding_read_requests":         12,
  "max_read_request_size":                 "32mb"
}'

建立後 ES 會：

  1. 透過 remote recovery（snapshot infrastructure）複製目前所有 segments
  2. 切換到 follow 模式，持續拉新的 operations

查狀態：

GET /logs-app-2026.04.15-000003/_ccr/info
GET /logs-app-2026.04.15-000003/_ccr/stats

Auto-follow Patterns

對 data stream 與 rolling 索引，逐個建 follower 太麻煩。Auto-follow 自動把符合 pattern 的新 leader index 變成 follower：

PUT /_ccr/auto_follow/logs-pattern
{
  "remote_cluster":         "leader-prod",
  "leader_index_patterns":  [ "logs-*", "metrics-*" ],
  "leader_index_exclusion_patterns": [ "logs-debug-*" ],
  "follow_index_pattern":   "{{leader_index}}",       // 預設同名
  "max_read_request_operation_count": 5120
}

之後 leader 上每出現新 logs-* 或 metrics-* index，follower 會自動建立同名 follower。

管理

# 列出所有 auto-follow patterns
GET /_ccr/auto_follow

# 暫停（升級時用）
POST /_ccr/auto_follow/logs-pattern/pause

# 啟動
POST /_ccr/auto_follow/logs-pattern/resume

# 看執行狀況
GET /_ccr/stats

CCR 不複製的內容

CCR 只複製 user-generated 索引內容，不複製 cluster 層級設定：

  • ❌ Cluster state（settings）
  • ❌ ILM / SLM policies
  • ❌ Index templates
  • ❌ Ingest pipelines
  • ❌ Watcher / Alerting rules
  • ❌ Security（users、roles、API keys）
  • ❌ ML jobs
  • ❌ Searchable snapshot data（資料在 repo）
⚠️ Cluster 層級設定要分別維護 Follower cluster 必須獨立建立 templates、ILM、users 等。建議用 GitOps（Terraform / 同步 script）讓兩 cluster 保持一致。
也可定期用 snapshot 備份 system indices 然後 restore 到 follower 來確保同步。

Bi-directional Replication

兩 cluster 都可寫，互相同步。但同一 index 不可雙寫，否則會造成衝突。常見做法：

管理操作

暫停 / 繼續

# 暫停 follow（leader 暫時不需要保留歷史）
POST /logs-app-2026.04.15-000003/_ccr/pause_follow

# 繼續
POST /logs-app-2026.04.15-000003/_ccr/resume_follow

# 解除 follow（變回普通 read-only index）
POST /logs-app-2026.04.15-000003/_close
POST /logs-app-2026.04.15-000003/_ccr/unfollow
POST /logs-app-2026.04.15-000003/_open

Failover：把 follower 變獨立可寫

# 1. 暫停 replication
POST /logs-app/_ccr/pause_follow

# 2. close
POST /logs-app/_close

# 3. unfollow
POST /logs-app/_ccr/unfollow

# 4. open，現在可寫了
POST /logs-app/_open

CCR Stats 與監控

# 整體 CCR stats
GET /_ccr/stats

# 看 follower lag
curl -u elastic:xxx --cacert ca.crt \
  "https://follower:9200/_ccr/stats?pretty" | jq '.indices[] | {
    index: .index,
    fatal_exception: .fatal_exception,
    follower_global_checkpoint: .shards[0].follower_global_checkpoint,
    leader_global_checkpoint: .shards[0].leader_global_checkpoint,
    lag: (.shards[0].leader_global_checkpoint - .shards[0].follower_global_checkpoint)
  }'

GET /logs-app/_recovery?human&pretty&active_only=true

效能調校

參數用途
max_read_request_operation_count每次拉的 op 數（5120 預設）
max_read_request_size每次拉的 size（32mb 預設）
max_outstanding_read_requests並行 read request 數（12 預設）
max_write_request_operation_countfollower 端寫入 batch 大小
max_outstanding_write_requests並行寫 request 數
read_poll_timeoutpoll leader 的 timeout（1m 預設）

高頻寫入場景把 read/write request count 與 outstanding 都調大。

與 ILM 的互動

⚠️ ILM 不會自動傳到 follower Follower index 不應該被 ILM 管（除了 unfollow 後）。建議：
  • Follower cluster 上對應的 ILM policy 不要包含 rollover（leader 已 rollover，follower 跟著）
  • 用 auto-follow pattern 設 follow_index_pattern 自動處理 backing indices
  • 當 leader index 進入 cold/frozen，follower 也應對應移動（用獨立 ILM 控制 follower tier 配置）

常見問題

Follower lag 太大

  • 檢查 follower cluster 寫入能力（可能 disk IO 不夠）
  • 調大 max_outstanding_read_requests
  • 檢查網路延遲（跨 region 可能 50-200ms RTT）

Leader history insufficient

Follower 跟不上時，leader 的 soft delete history 可能已過期。錯誤訊息類似 operations not found。

解法：增加 leader 的 retention：

PUT /logs-app/_settings
{
  "index.soft_deletes.retention_lease.period": "24h"
}

或重建 follower（unfollow + 重新 follow）。

Auto-follow 沒觸發

  • 檢查 GET /_ccr/auto_follow 看 pattern 是否存在且未 paused
  • 檢查 GET /_ccr/stats 看是否有 auto_follow_stats.recent_auto_follow_errors
  • API key 權限是否有 replication 涵蓋新 index

CCR 最佳實踐

  • 9.x 用 API key + remote cluster interface 9443，取代舊 cert + 9300
  • Follower 節點要有 remote_cluster_client role
  • 用 auto-follow pattern 對 data stream 自動化
  • Templates / ILM / Ingest 用 GitOps 同步到 follower
  • 同時用 CCR + snapshot：CCR 做 RPO，snapshot 做長期保存
  • 監控 follower lag，> 10 分鐘設告警
  • 保護 leader history：retention_lease.period 設足夠覆蓋最差 lag 情境
  • 定期演練 failover（pause + close + unfollow + open + 切流量）
  • Bi-directional 慎用：必須業務上保證 index 不雙寫
  • 跨 region CCR：注意 egress 流量費用，可能比 snapshot 高

本章重點回顧

  • CCR 即時複製 leader index 到 follower（read-only），透過 soft deletes 重播
  • 需 Platinum 訂閱；7.0 後新建 index 自動啟用 soft deletes
  • 9.x 推薦走 remote cluster interface 9443 + API key 模式
  • Follower 節點需 remote_cluster_client role
  • 三大用途：DR、data locality、centralized reporting
  • 用 auto-follow pattern 自動 follow data stream 的新 backing indices
  • CCR 只複製 user data：cluster state、ILM、ingest、users 都要分別維護
  • Failover 流程：pause → close → unfollow → open
  • 監控 _ccr/stats 看 follower lag
  • CCR 與 snapshot 可並用：CCR 嚴格 RPO/RTO；snapshot 長期保存
