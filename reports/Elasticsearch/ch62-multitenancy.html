CH 62 · Multi-tenancy 設計模式

適合對象 🟡 開發者 ⚡ 進階
閱讀時間 約 28 分鐘
前置知識 CH 21 RBAC、CH 30 ILM、CH 36 query DSL
本章目標 五種 multi-tenancy 模式、權衡比較、何時選哪個、scale-out 策略

什麼是 Multi-tenancy？

多租戶（multi-tenant）系統：同一 ES 部署服務多個邏輯獨立的客戶 / 使用者組。常見場景：

  • SaaS 產品（每家公司一個 tenant）
  • 大型企業內部多 BU 共享平台
  • MSP 為多客戶提供托管 ES
  • 多 application 共享 cluster

核心挑戰：隔離 vs 成本 vs 規模。

模式 1：Cluster-per-tenant

每 tenant 一個獨立 ES cluster：

ES Cluster: customer-acme    →  ACME 公司用
ES Cluster: customer-globex  →  Globex 公司用
ES Cluster: customer-initech →  Initech 公司用

優點

  • 完全資料隔離（合規）
  • 單 tenant 故障不影響其他
  • 每 tenant 獨立調校（shard、tier、ILM）
  • 資料 sovereignty 容易（不同 region）

缺點

  • 成本爆增（最低 3 master + 2 data 節點）
  • 運維複雜（升級、監控乘 N）
  • 不適合超過數百個 tenants

自動化

用 ECK（CH 11）自動化：

apiVersion: elasticsearch.k8s.elastic.co/v1
kind: Elasticsearch
metadata:
  name: customer-{{ tenant_id }}
spec:
  version: 9.3.4
  nodeSets:
    - name: master
      count: 3
    - name: data
      count: 3

用 GitOps + tenant onboarding workflow（Argo Workflow / Tekton）自動建。

模式 2：Index-per-tenant

共享 cluster，每 tenant 一個 index（或 data stream）：

logs-tenant-acme
logs-tenant-globex
logs-tenant-initech

優點

  • 邏輯獨立 mapping / settings
  • 用 alias 對外統一接口
  • RBAC 簡單（role 限制 indices pattern）
  • tenant-level snapshot / delete 易

缺點

  • 大量 tenant → 大量 index → cluster state 膨脹（每 index ~10 KB）
  • 每 tenant 至少 1 shard，hard ceiling 約 數千 tenants
  • 跨 tenant 查詢慢（要 query 多個 index）

RBAC 範例

PUT _security/role/tenant-acme-user
{
  "indices": [
    {
      "names":      [ "logs-tenant-acme*", "metrics-tenant-acme*" ],
      "privileges": [ "read", "view_index_metadata" ]
    }
  ]
}

模式 3：Routing-per-tenant（推薦大量 tenants）

共享一個 index，用 routing 把同 tenant 資料放在同一 shard：

# 寫入時指定 routing
POST logs/_doc?routing=tenant-acme
{
  "tenant_id": "tenant-acme",
  "@timestamp": "...",
  "msg": "..."
}

# 查詢時指定 routing（只查到該 shard，跳過其他）
GET logs/_search?routing=tenant-acme
{
  "query": {
    "bool": {
      "filter": [ { "term": { "tenant_id": "tenant-acme" } } ]
    }
  }
}

優點

  • 查詢只命中 1 個 shard，極快
  • cluster state 只有少數 index
  • 支援數十萬 tenants
  • tenant 間資源共享（小 tenant 不浪費 shard）

缺點

  • 大 tenant 可能 hot shard（單 shard 過大）
  • Mapping 全 tenant 共用（不能 per-tenant 客製）
  • ILM 不能 per-tenant 客製
  • 必須用 filter 配 routing，否則 routing 同一 shard 的不同 tenant 資料會被一起查到

強制 routing

{
  "mappings": {
    "_routing": { "required": true },
    "properties": { ... }
  }
}

確保所有寫入必須帶 routing。

處理 hot tenant

  • 大 tenant 拆出獨立 index（hybrid 模式）
  • 用 partition_size 把大 tenant 散到多 shard：
{
  "settings": {
    "index.routing_partition_size": 4   // tenant 散到 4 個 shard
  }
}

單 tenant 查詢命中 4 shard 而非全部，仍比掃所有 shard 快很多。

模式 4：Filter-per-tenant

共享 index，每筆有 tenant_id 欄位，查詢時加 filter：

GET logs/_search
{
  "query": {
    "bool": {
      "filter": [ { "term": { "tenant_id": "acme" } } ]
    }
  }
}

優點

  • 最簡單實作
  • cluster state 最小
  • 不需 routing 設計

缺點

  • 每查詢命中所有 shard，效能不如 routing
  • 容易 bug（忘記加 filter → 跨 tenant 資料洩漏）
  • 授權容易做錯

用 RBAC 強制 filter

PUT _security/role/tenant-acme-user
{
  "indices": [
    {
      "names":      [ "logs" ],
      "privileges": [ "read" ],
      "query": {
        "term": { "tenant_id": "acme" }
      }
    }
  ]
}

使用此 role 的 user 自動帶 filter，application 層忘記也安全。

模式 5：共用 + DLS API Key（RAG 推薦）

結合多種：共用 index + 每 user 一個 API key（含 DLS query template）：

POST /_security/api_key
{
  "name": "alice-search-key",
  "role_descriptors": {
    "search": {
      "indices": [
        {
          "names":      [ "search-confluence" ],
          "privileges": [ "read" ],
          "query": {
            "template": {
              "params": {
                "user_id": "alice",
                "groups":  [ "engineering", "platform" ]
              },
              "source": {
                "bool": {
                  "should": [
                    {
                      "term": { "_allow_access_control": "{{user_id}}" }
                    },
                    {
                      "terms": { "_allow_access_control": "{{groups}}" }
                    }
                  ]
                }
              }
            }
          }
        }
      ]
    }
  }
}

Application 用 user-specific API key 查 ES，自動套權限 filter。

混合策略：大 tenant 用 index，小 tenant 用 routing

大客戶 (>10GB)：logs-{tenant}     # 獨立 index
小客戶 (<10GB)：logs-shared       # 共用，用 routing

優點：兼顧成本與隔離。Application 層判斷 tenant 大小，路由到對應 index。

Kibana Spaces 多租戶（CH 52 補充）

Kibana 端用 Spaces 隔離 dashboards / saved objects：

  • 每 tenant 一個 space
  • Role 限定可進的 space
  • Saved object 完全隔離

對於提供「客戶可登入看自己 dashboard」場景必備。

Multi-tenant 的 ILM

共用 ILM policy

對所有 tenant 用同一個 policy（最簡）：

PUT _ilm/policy/logs-policy
{
  "policy": {
    "phases": {
      "hot":    { ... },
      "warm":   { ... },
      "delete": { "min_age": "90d", "actions": { "delete": {} } }
    }
  }
}

Per-tenant tier 配置

客戶分級給不同保留期：

  • 免費 tenant：30 天
  • 付費 tenant：90 天
  • 企業 tenant：365 天 + cold tier

實作：每級用獨立 ILM + index template + alias 模式。

資源隔離

Search rate quota

Elasticsearch 9.x 提供（部分）：

  • Adaptive replica selection：自動分散 search 負載
  • Search rate limit：對某 user / API key 限制 QPS（外層 LB / API gateway 做）
  • Circuit breaker：防止單 query 拖垮 cluster

專用 tier

對 high-tier tenant 用 hot tier 與 dedicated coordinating node：

# 對 enterprise tenant 的 index 強制只進 dedicated 節點
PUT logs-tenant-enterprise/_settings
{
  "index.routing.allocation.require.tier": "premium"
}

節點端：

node.attr.tier: premium

跨 tenant 查詢（管理用）

SaaS 平台需聚合所有 tenant 統計：

# Admin role：可跨所有 tenant 查
GET logs-tenant-*/_search
{
  "size": 0,
  "aggs": {
    "by_tenant": { "terms": { "field": "tenant_id", "size": 1000 } }
  }
}

Admin role 不加 DLS query；只給內部 admin 用。

Tenant Onboarding Workflow

新 tenant 加入時的標準流程（自動化）：

  1. 建立 namespace（K8s）/ project（Cloud）
  2. 建立 ES index template + ILM policy
  3. 建立 Kibana space + dashboard 範本
  4. 建立 RBAC role + user / API key
  5. 產生 connector（CH 61）認證
  6. 初始化 sample data / 引導 user

用 Terraform / Ansible / 自動化 script 完成。

最佳實踐

  • 從業務需求選模式：合規嚴格 → cluster-per-tenant；大量小 tenant → routing
  • 能用 routing 不用 index-per-tenant，可擴展性好太多
  • 用 RBAC 強制 filter（DLS query template），不依賴 application 層自律
  • 大 tenant 拆獨立 index，hybrid 模式
  • Mapping schema 盡量穩定：跨 tenant 共用
  • Kibana Spaces 必用：dashboards 隔離
  • 監控 cluster.max_shards_per_node：index-per-tenant 易爆
  • 用 ECK / GitOps 自動化 cluster-per-tenant
  • Tenant onboarding 完全自動化，避免人工建
  • 大量 tenant 寫入用 bulk 同 routing key：減少 round-trip

本章重點回顧

  • 五種 multi-tenancy 模式：cluster-per-tenant、index-per-tenant、routing、filter、mix
  • Trade-off：隔離程度 vs 成本 vs 規模
  • Routing-per-tenant：大量 tenant 推薦，查詢只命中 1 shard
  • 用 routing_partition_size 處理 hot tenant
  • RBAC + DLS query template 在伺服器端強制 filter，避免 application 漏洞
  • Hybrid：大 tenant 獨立 index、小 tenant 共用 routing
  • Kibana Spaces 提供 dashboard 級隔離
  • Cluster-per-tenant 用 ECK 自動化（金融 / 醫療場景）
  • Tenant onboarding 必須完全自動化
