Step 1：建立 EKS Cluster

用 eksctl 是最簡單方式。先安裝：

brew install eksctl  # macOS
# Linux
curl --silent --location "https://github.com/eksctl-io/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" | tar xz -C /tmp
sudo mv /tmp/eksctl /usr/local/bin

建立 cluster config eks-cluster.yaml：

apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig

metadata:
  name: elastic-prod
  region: ap-northeast-1
  version: "1.31"

iam:
  withOIDC: true   # 啟用 IRSA 必要

addons:
  - name: vpc-cni
    version: latest
  - name: coredns
    version: latest
  - name: kube-proxy
    version: latest
  - name: aws-ebs-csi-driver
    version: latest

managedNodeGroups:
  - name: es-master
    instanceType: m6i.xlarge
    minSize: 3
    maxSize: 3
    desiredCapacity: 3
    availabilityZones: [ap-northeast-1a, ap-northeast-1c, ap-northeast-1d]
    volumeSize: 50
    volumeType: gp3
    labels:
      role: elasticsearch-master
    taints:
      - key: dedicated
        value: elasticsearch-master
        effect: NoSchedule

  - name: es-data-hot
    instanceType: r6id.2xlarge   # 含本地 NVMe SSD
    minSize: 3
    maxSize: 9
    desiredCapacity: 3
    availabilityZones: [ap-northeast-1a, ap-northeast-1c, ap-northeast-1d]
    volumeSize: 100
    volumeType: gp3
    labels:
      role: elasticsearch-hot
    taints:
      - key: dedicated
        value: elasticsearch-hot
        effect: NoSchedule

  - name: es-data-warm
    instanceType: r6i.xlarge
    minSize: 2
    maxSize: 6
    desiredCapacity: 2
    availabilityZones: [ap-northeast-1a, ap-northeast-1c]
    volumeSize: 100
    volumeType: gp3
    labels:
      role: elasticsearch-warm
    taints:
      - key: dedicated
        value: elasticsearch-warm
        effect: NoSchedule

eksctl create cluster -f eks-cluster.yaml
# 約 15-25 分鐘

aws eks update-kubeconfig --region ap-northeast-1 --name elastic-prod
kubectl get nodes

Step 2：EBS CSI Driver 與 StorageClass

建立 EKS 推薦的 gp3 StorageClass：

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: ebs-gp3
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  iops: "3000"
  throughput: "125"
  encrypted: "true"
  fsType: ext4
volumeBindingMode: WaitForFirstConsumer
reclaimPolicy: Retain
allowVolumeExpansion: true

kubectl patch storageclass gp2 \
  -p '{"metadata":{"annotations":{"storageclass.kubernetes.io/is-default-class":"false"}}}'
kubectl apply -f sc-gp3.yaml
kubectl get sc

EBS gp3 vs 本地 NVMe（instance store）

項目EBS gp3Instance Store (NVMe)
持久性✅ 機器掛掉資料保留❌ 機器停止即消失
IOPS3000 baseline數十萬 IOPS
吞吐量125-1000 MB/s3000+ MB/s
跨 AZ 移動不能（綁 AZ）不能（綁 instance）
適合master、warm、coldhot 高 QPS（搭配 ES replica）
💡 io2 Block Express（極致 IOPS） 對需要 64K+ IOPS 的工作負載可用 io2 Block Express，可達 256K IOPS。設定 type: io2, iops: "64000"。

kubectl create -f https://download.elastic.co/downloads/eck/3.3.2/crds.yaml

Step 4：安裝 AWS Load Balancer Controller

# 1. 建立 IAM Policy
curl -O https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/main/docs/install/iam_policy.json
aws iam create-policy \
  --policy-name AWSLoadBalancerControllerIAMPolicy \
  --policy-document file://iam_policy.json

# 2. 建立 IRSA service account
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
eksctl create iamserviceaccount \
  --cluster=elastic-prod \
  --namespace=kube-system \
  --name=aws-load-balancer-controller \
  --attach-policy-arn=arn:aws:iam::${ACCOUNT_ID}:policy/AWSLoadBalancerControllerIAMPolicy \
  --override-existing-serviceaccounts \
  --approve

# 3. Helm 安裝 controller
helm repo add eks https://aws.github.io/eks-charts
helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system \
  --set clusterName=elastic-prod \
  --set serviceAccount.create=false \
  --set serviceAccount.name=aws-load-balancer-controller \
  --set region=ap-northeast-1

Step 5：IRSA — 給 ES 存取 S3 的權限

IRSA（IAM Roles for Service Accounts）讓 pod 可直接 assume IAM role，無需 AWS credentials。比 access key 安全多了。

5.1 建立 IAM policy（snapshot 存取）

cat > es-s3-snapshot-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:ListBucket",
        "s3:GetBucketLocation",
        "s3:ListBucketVersions",
        "s3:ListBucketMultipartUploads"
      ],
      "Resource": "arn:aws:s3:::my-es-snapshots"
    },
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject",
        "s3:AbortMultipartUpload",
        "s3:ListMultipartUploadParts"
      ],
      "Resource": "arn:aws:s3:::my-es-snapshots/*"
    }
  ]
}
EOF

aws iam create-policy \
  --policy-name ESS3SnapshotPolicy \
  --policy-document file://es-s3-snapshot-policy.json

5.2 建立 ServiceAccount 並綁 IAM role

eksctl create iamserviceaccount \
  --cluster=elastic-prod \
  --namespace=elastic \
  --name=elasticsearch-s3 \
  --attach-policy-arn=arn:aws:iam::${ACCOUNT_ID}:policy/ESS3SnapshotPolicy \
  --approve

5.3 把 SA 綁到 Elasticsearch pods

在 Elasticsearch CR 的 podTemplate 中：

spec:
  nodeSets:
    - name: data-hot
      podTemplate:
        spec:
          serviceAccountName: elasticsearch-s3   # 綁 IRSA
          containers:
            - name: elasticsearch

Step 6：部署 Elasticsearch CR

# elasticsearch-eks.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: elastic
---
apiVersion: elasticsearch.k8s.elastic.co/v1
kind: Elasticsearch
metadata:
  name: production
  namespace: elastic
spec:
  version: 9.3.4

  http:
    tls:
      selfSignedCertificate:
        subjectAltNames:
          - dns: elasticsearch.example.com
    service:
      spec:
        type: ClusterIP

  nodeSets:
    # ── Master ──
    - name: master
      count: 3
      config:
        node.roles: ["master"]
        xpack.ml.enabled: false
      zoneAwareness:
        topologyKey: topology.kubernetes.io/zone
      podTemplate:
        spec:
          tolerations:
            - key: dedicated
              operator: Equal
              value: elasticsearch-master
              effect: NoSchedule
          nodeSelector:
            role: elasticsearch-master
          initContainers:
            - name: sysctl
              securityContext: { privileged: true, runAsUser: 0 }
              command: ['sh', '-c', 'sysctl -w vm.max_map_count=262144']
          containers:
            - name: elasticsearch
              env:
                - name: ES_JAVA_OPTS
                  value: "-Xms2g -Xmx2g"
              resources:
                requests: { memory: 4Gi, cpu: 1 }
                limits:   { memory: 4Gi, cpu: 2 }
      volumeClaimTemplates:
        - metadata: { name: elasticsearch-data }
          spec:
            accessModes: ["ReadWriteOnce"]
            resources: { requests: { storage: 50Gi } }
            storageClassName: ebs-gp3

    # ── Hot ──
    - name: data-hot
      count: 3

        node.roles: ["data_hot", "data_content", "ingest"]

        topologyKey: topology.kubernetes.io/zone

          tolerations:

            storageClassName: ebs-gp3

Step 7：用 ALB Ingress 對外暴露

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: elasticsearch
  namespace: elastic
  annotations:
    kubernetes.io/ingress.class: alb
    alb.ingress.kubernetes.io/scheme: internal
    alb.ingress.kubernetes.io/target-type: ip
    alb.ingress.kubernetes.io/backend-protocol: HTTPS
    alb.ingress.kubernetes.io/listen-ports: '[{"HTTPS":443}]'
    alb.ingress.kubernetes.io/certificate-arn: arn:aws:acm:ap-northeast-1:123456789012:certificate/xxxx
    alb.ingress.kubernetes.io/healthcheck-protocol: HTTPS
    alb.ingress.kubernetes.io/healthcheck-path: /
    alb.ingress.kubernetes.io/success-codes: "200,401"   # ES 401 也算健康
spec:
  rules:
    - host: elasticsearch.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: production-es-http
                port:
                  number: 9200

Step 8：建立 S3 snapshot repository

使用前面 IRSA 已賦予的權限：

# 建立 S3 bucket
aws s3 mb s3://my-es-snapshots --region ap-northeast-1
aws s3api put-bucket-versioning \
  --bucket my-es-snapshots \
  --versioning-configuration Status=Enabled

在 Elasticsearch 中註冊 repository：

PASSWORD=$(kubectl get secret production-es-elastic-user -n elastic \
  -o go-template='{{.data.elastic | base64decode}}')

kubectl exec -n elastic production-es-data-hot-0 -- \
  curl -k -u elastic:$PASSWORD -X PUT "https://localhost:9200/_snapshot/aws-s3" \
  -H 'Content-Type: application/json' -d '{
    "type": "s3",
    "settings": {
      "bucket": "my-es-snapshots",
      "region": "ap-northeast-1",
      "base_path": "production-cluster/"
    }
  }'

因為用的是 IRSA，無需設定 access key 與 secret key。

Step 9：節點自動擴容（Karpenter / CA）

傳統 Cluster Autoscaler（CA）

適合 managed node group 已固定的場景。eksctl 已內建支援。

Karpenter（推薦）

更現代、更彈性。Karpenter 會根據 pending pod 動態挑選最適 instance type，不限於預設的 node group：

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: elasticsearch-warm
spec:
  template:
    metadata:
      labels:
        role: elasticsearch-warm
    spec:
      taints:
        - key: dedicated
          value: elasticsearch-warm
          effect: NoSchedule
      requirements:
        - key: kubernetes.io/arch
          operator: In
          values: ["amd64"]
        - key: karpenter.k8s.aws/instance-category
          operator: In
          values: ["r"]
        - key: karpenter.k8s.aws/instance-cpu
          operator: In
          values: ["4", "8"]
        - key: topology.kubernetes.io/zone
          operator: In
          values: ["ap-northeast-1a", "ap-northeast-1c", "ap-northeast-1d"]
      nodeClassRef:
        name: default
        kind: EC2NodeClass
        group: karpenter.k8s.aws
  limits:
    cpu: 100
  disruption:
    consolidationPolicy: WhenUnderutilized
    expireAfter: 720h   # 30 天輪替

⚠️ 注意：跨 AZ 資料傳輸成本

AWS 跨 AZ 流量收費 ES 的 replica 同步、search query 廣播會跨 AZ 產生流量。在大型叢集上每月可能數千美元。
  • 用 shard allocation awareness 讓 query 優先在同 AZ 完成（自動由 zoneAwareness 設定）
  • 大量 ingest 集中到同一 AZ 的 hot tier，由 ILM rollover 後再分散
  • 啟用 VPC Endpoint for S3（免跨 AZ 費）

災難復原（DR）

EKS 上的 ES DR 策略：

  1. 跨 AZ HA：3 master + 跨 3 AZ 的 data nodes
  2. 自動 snapshot 到 S3（用 SLM，CH 47）
  3. 跨 region 複製：S3 cross-region replication；或 CCR（CH 48）到另一 region 的 EKS 叢集
  4. 恢復測試：每季從 snapshot 還原到 staging cluster 驗證

本章重點回顧

  • 用 eksctl 建立 EKS cluster，搭配 OIDC provider 啟用 IRSA
  • 必裝 add-ons：vpc-cni、coredns、kube-proxy、aws-ebs-csi-driver
  • StorageClass 用 gp3、WaitForFirstConsumer、Retain、allowVolumeExpansion=true
  • Hot tier 可選用 r6id 帶本地 NVMe，配合 ES replica 補資料安全
  • IRSA 取代 access key：給 ES pod 一個綁了 IAM role 的 ServiceAccount
  • 用 AWS Load Balancer Controller 建立 ALB Ingress
  • S3 bucket 啟用 versioning + lifecycle 控制 snapshot 成本
  • 用 zoneAwareness 自動處理跨 AZ allocation awareness，降低跨 AZ 流量
  • Karpenter 提供比 CA 更彈性的 autoscaling
