1. Self-managed（本書主軸）

本書 65 章涵蓋的主題。完全自己管：

  • OS 安裝、JVM 調校（CH 06-09、CH 18）
  • TLS / 認證 / RBAC（CH 19-25）
  • 升級、備份、CCR（CH 47-51）
  • 監控、調校（CH 46、CH 49）
  • 故障排除（CH 50）

適合：

  • Air-gapped 環境（CH 16, CH 63）
  • 絕對 cost / control 需求
  • 強合規（資料完全自管）
  • 已有資深 Elastic ops team

2. ECK：Elastic Cloud on Kubernetes

K8s Operator 自動化 cluster lifecycle（CH 11）：

apiVersion: elasticsearch.k8s.elastic.co/v1
kind: Elasticsearch
metadata:
  name: production
spec:
  version: 9.3.4
  nodeSets:
    - name: master
      count: 3
      podTemplate:
        spec:
          containers:
            - name: elasticsearch
              resources:
                requests: { memory: "4Gi", cpu: "1" }
                limits:   { memory: "4Gi", cpu: "2" }
    - name: data-hot
      count: 6
      podTemplate: { ... }

優勢：

  • declarative 配置（GitOps friendly）
  • K8s 原生擴展（HPA、PDB、Network Policy）
  • 滾動升級自動化
  • 多 cluster 在同一 K8s
  • 適用於 EKS / GKE / AKS / OpenShift / 自架 K8s（CH 12-14）

適合：

  • 已有 K8s 平台
  • 多 cluster 部署
  • 希望 GitOps 全面化

3. ECE：Elastic Cloud Enterprise

把 Cloud Hosted 的 SaaS 體驗帶到自家機房。Elastic 的 control plane 軟體：

  • UI / API 建立、調整、升級 cluster
  • Multi-tenancy（不同 deployment 隔離）
  • 內建 monitoring / proxy
  • 支援 hot/warm/cold/frozen tier 自動配置
  • 跑在自家 VM / K8s

適合：

  • 大型企業有大量 deployment
  • 內部 IT 提供 ES-as-a-Service
  • 合規不允許 cloud，但想要 SaaS 體驗

4. Elastic Cloud Hosted（SaaS）

Elastic 自家 SaaS，跑在 AWS / GCP / Azure：

  • UI 一鍵建立 deployment
  • 多 region、多 cloud 選
  • 內建 Kibana、Fleet、APM
  • 升級一鍵（minor / patch 自動）
  • Cross-cluster search / replication 內建
  • Snapshot 自動排程

計費：

  • 按 deployment 大小 + uptime（不是查詢量）
  • cloud infra cost 包在內
  • Standard / Gold / Platinum / Enterprise 訂閱層

適合：

  • 大多數 production deployment
  • 不想管 infrastructure
  • 需要混搭多家 cloud

5. Elastic Cloud Serverless

2024 年推出的全託管：完全沒有 cluster 概念，只看「project」：

  • 三種 project type：Search、Observability、Security
  • 無需選擇 instance size、shard 數
  • autoscaling、按用量計費
  • 儲存與計算分離（Search 與 ingest 各自 scale）

計費模式

  • Ingest：依寫入 GB 計
  • Search VCU：query 計算量
  • Storage：依 retain 量計
  • 無 baseline 月費，真正按用量

限制

  • 不可裝 plugin
  • 不可改 cluster setting（很多 setting 預設）
  • 沒有節點 / shard 概念給用戶看
  • 有 API 與功能差異（部分 self-managed feature 不支援）
  • 但新功能優先 serverless 推出（如 Managed OTLP，CH 64）

適合

  • 變動性大的 workload（行銷活動、季節性）
  • 新 application PoC
  • 不想煩惱 capacity planning
  • 需要最新功能
  • 輕量 RAG application

TCO（總成本）比較

1 TB 資料、5K events/sec 寫入、低查詢量的範例（粗估）：

模式月成本範圍（USD）包含內容
Self-managed (on-prem)$2K-5K硬體分攤 + Platinum license + 1 維運人力
Self-managed (AWS EC2)$3K-6KEC2 + EBS + license + 維運
ECK on EKS$3K-6KEKS + EBS + license + Platform team 分攤
Cloud Hosted$2K-4K全包，無維運成本
Serverless（用量大）$1.5K-5K純按量；變動大可能省也可能爆

注意：運維時間是隱性成本。Self-managed 看似硬體便宜但需要全職人力；Cloud / Serverless 表面貴但團隊能專注業務。

遷移路徑

Self-managed → Elastic Cloud

步驟（用 snapshot）：

  1. 在 Cloud 建立 deployment（同版本）
  2. 來源 cluster 註冊 S3 repository（Cloud 也要能存取）
  3. 來源 take snapshot
  4. Cloud cluster register 同 repository
  5. Restore
  6. 切應用 endpoint

另一種：用 reindex from remote。

Cloud → Serverless

  1. 確認 Serverless 支援的 feature 你都用得到
  2. 建立 Serverless project
  3. 用 reindex from remote 或 snapshot/restore（部分支援）
  4. 切 endpoint

Self-managed → ECK

  1. 建立 K8s cluster
  2. 用 ECK operator 部署 ES（同版本）
  3. 把 self-managed 設定（template、ILM、users）匯出
  4. Snapshot / restore 到 ECK cluster
  5. 切流量

混合部署

實務上常混搭：

  • Production：Cloud Hosted（穩定、有 SLA）
  • Dev / Staging：Serverless（按用量便宜）
  • Logs / Observability：Serverless（變動大，autoscale 友善）
  • 核心交易資料：Self-managed / ECK（合規、控制）
  • 跨 region DR：Cloud Hosted 跨 region

用 CCS / CCR 串聯（CH 48）。

部署選擇最佳實踐

  • 從業務需求倒推：合規、SLA、預算、團隊能力
  • 先試 Serverless 做 PoC：開發成本最低
  • 大規模 production 用 Cloud Hosted 或 ECE：cost / 維運平衡
  • 已有 K8s 平台優先 ECK：複用既有運維能力
  • Self-managed 只在合規 / 客製化必要時：運維成本不容低估
  • 定期 review：每年評估是否還是最佳模式
  • 用 IaC（Terraform / Helm / Ansible）抽象：未來換 backend 不痛
  • 備份永遠在身邊（CH 47）：不論模式都需要
  • 監控集中化：跨模式用統一 dashboard（CH 46）

本章重點回顧

  • 五種部署模式光譜：Self-managed → ECK → ECE → Cloud Hosted → Serverless
  • 運維負擔逐漸降低，控制與客製性也降低
  • Self-managed：絕對控制與合規，本書主軸
  • ECK：K8s 平台用戶首選
  • ECE：大型企業內部 SaaS
  • Cloud Hosted：多數 production 的甜蜜點
  • Serverless：變動 workload、PoC、新功能優先
  • TCO 包含運維時間，self-managed 表面便宜但人力成本高
  • 遷移路徑：snapshot/restore、reindex from remote、CCR 漸進切換
  • 實務常混搭：production 用 Cloud，logs / dev 用 Serverless，核心用 self-managed
  • 無論哪種模式，備份、監控、IaC 三件事不變
