從零開始：Cluster Formation

當你啟動一個全新的 Elasticsearch 節點時，它會經歷以下過程：

這個流程的核心在於 discovery（節點發現）與 election（master 選舉），兩者結合稱為 cluster formation。

Discovery（發現機制）

節點需要知道「叢集裡其他節點在哪」才能加入。Elasticsearch 不依賴 multicast（早期版本支援，已移除），而採用 seed-based discovery：

# elasticsearch.yml
discovery.seed_hosts:
  - master-1.example.com:9300
  - master-2.example.com:9300
  - master-3.example.com:9300

# 也可以省略 port，預設用 transport.profiles.default.port (9300)
discovery.seed_hosts:
  - master-1.example.com
  - master-2.example.com
  - master-3.example.com

節點啟動時會：

  1. 解析 discovery.seed_hosts 列表中的位址
  2. 透過 transport（9300）連線並交換節點資訊
  3. 取得整個叢集的節點清單
  4. 嘗試與 active master 建立連線
💡 只列 master-eligible 節點即可 discovery.seed_hosts 不需列出所有節點，只要列出足以聯繫到 master-eligible 節點的位址即可。data-only 節點可省略。

Seed Hosts Providers

除了直接列 IP/hostname，ES 也支援多種動態 seed provider：

Provider用途
settings (預設)從 elasticsearch.yml 直接讀取
file從 $ES_PATH_CONF/unicast_hosts.txt 讀取，可動態更新無需重啟
ec2透過 AWS API 自動發現有特定 tag 的 EC2 instances
gce透過 GCP API 自動發現 GCE instances
azure-classic透過 Azure API 自動發現

啟用 file-based provider：

discovery.seed_providers: file

建立 unicast_hosts.txt：

cat > /etc/elasticsearch/unicast_hosts.txt << EOF
master-1.example.com:9300
master-2.example.com:9300
master-3.example.com:9300
EOF

檔案有變更時 ES 會自動重新讀取，非常適合搭配 service discovery（Consul、Kubernetes、雲端 SD）。

Quorum（法定人數）

分散式系統最大的挑戰是避免 split-brain（腦裂）：當網路分割造成兩個子集合，雙方都各自選出 master 並各自寫入資料，事後無法合併。

Elasticsearch 透過法定人數（quorum）規則保證在任何時刻最多只有一個 master 能做出決策。Quorum 定義為：

圖 4-2：Quorum 容錯示意 — 為何 master-eligible 節點數要奇數
quorum = floor(N/2) + 1

其中 N = voting nodes 的數量

範例：
  N = 1 → quorum = 1   （單節點，無法容忍故障）
  N = 2 → quorum = 2   （任一掛掉就無法達標 — 不要這樣配）
  N = 3 → quorum = 2   （可容忍 1 個故障）⭐ 最小推薦
  N = 5 → quorum = 3   （可容忍 2 個故障）⭐ 大型叢集
  N = 7 → quorum = 4   （可容忍 3 個故障）
🚨 為什麼是奇數？ 4 個節點與 3 個節點的容錯能力相同（皆為 1 個故障），但 4 個節點的成本更高、且網路分割成 2-2 時兩邊都無法達標。所以master-eligible 節點數應為奇數（3、5、7）。

Voting Configuration

從 7.0 起，ES 改用動態 voting configuration 取代舊的 discovery.zen.minimum_master_nodes。Voting configuration 是當前用於 quorum 計算的節點集合，由 master 自動維護。

檢視當前 voting configuration：

curl -X GET "https://localhost:9200/_cluster/state/metadata?filter_path=metadata.cluster_coordination&pretty" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt

輸出範例：

{
  "metadata": {
    "cluster_coordination": {
      "term": 7,
      "last_committed_config": [
        "node-id-master-1",
        "node-id-master-2",
        "node-id-master-3"
      ],
      "last_accepted_config": [
        "node-id-master-1",
        "node-id-master-2",
        "node-id-master-3"
      ]
    }
  }
}

Voting Configuration 的自動維護

當你新增或移除 master-eligible 節點時，ES 會自動調整 voting configuration：

  • 新節點加入後，會自動加入 voting config
  • 節點優雅關閉（graceful shutdown）後，會自動從 voting config 移除
  • 但若節點突然消失（crash），預設不會自動移除（避免短暫網路問題誤判）

主動排除節點（Voting Exclusions）

若要永久移除一個 master-eligible 節點，應先把它加入排除列表，讓 voting config 自動縮小，再關閉節點：

# 排除特定節點（按 node name）
curl -X POST "https://localhost:9200/_cluster/voting_config_exclusions?node_names=master-3" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt

# 確認 voting config 已縮小（剩 master-1 和 master-2）
curl -X GET "https://localhost:9200/_cluster/state/metadata?filter_path=metadata.cluster_coordination" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt

# 然後可以安全關閉 master-3，操作完成後清除排除規則
curl -X DELETE "https://localhost:9200/_cluster/voting_config_exclusions" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt

Cluster Bootstrap（首次形成）

當你第一次啟動一個全新叢集時，必須告訴 ES 「初始的 master-eligible 節點是誰」，否則它無法決定第一個 voting configuration：

# elasticsearch.yml — 只在第一次啟動時設定
cluster.initial_master_nodes:
  - master-1
  - master-2
  - master-3
🚨 cluster.initial_master_nodes 只用一次 叢集成功 bootstrap 後，這個設定必須立刻從 elasticsearch.yml 中移除。原因：
  • 叢集已經把 voting configuration 寫入持久化儲存
  • 之後重啟單一節點時，若這個設定還在，可能會誤觸 bootstrap，建立一個新叢集（與原叢集分裂），造成 data loss
  • 新加入的節點不需要這個設定，它們會透過 discovery + join 流程自動加入

Auto-bootstrap 開發模式

如果 ES 偵測到是單機開發模式（綁定 localhost、且未設定 production 相關參數），它會自動 bootstrap 為單節點叢集，不需設定 cluster.initial_master_nodes。

判斷是否為 development mode 的條件：

  • network.host 為 loopback（127.0.0.1）或未設定（預設 _local_）
  • discovery.seed_hosts 與 cluster.initial_master_nodes 都未設定

一旦把 network.host 改為非 loopback（例如 0.0.0.0），ES 就進入 production mode，會啟用所有 bootstrap checks（CH 06 詳述），並要求明確設定 cluster formation 參數。

Master 選舉演算法

當叢集啟動或現任 master 失聯時，會觸發選舉。ES 使用基於 Raft 變體的演算法：

  1. 每個 master-eligible 節點計算自己看到的 voting nodes 集合
  2. 當節點偵測到沒有 master 時，發起 pre-vote 階段（避免無謂的 term 增加）
  3. 取得 pre-vote 多數同意後，提升自己的 term，發起 vote 請求
  4. 其他節點收到 vote 請求後，根據 term、cluster state 版本等規則決定是否投票
  5. 取得 quorum 票數的節點成為新 master
  6. 新 master 廣播 cluster state 並開始服務

觸發選舉的時機

  • 叢集首次形成（bootstrap）
  • 所有節點啟動時，發現沒有現任 master
  • 原 master 失聯（網路分割或機器掛掉）
  • 節點被踢出 cluster fault detection

選舉相關參數（一般無需修改）

# 選舉超時（預設 10s）
cluster.election.duration: 500ms
cluster.election.initial_timeout: 100ms
cluster.election.back_off_time: 100ms
cluster.election.max_timeout: 10s

# 發現超時
discovery.find_peers_interval: 1s
discovery.probe.connect_timeout: 30s
discovery.probe.handshake_timeout: 30s

Cluster State 發布（Publishing）

Master 對 cluster state 的變更採兩階段提交（two-phase commit）：

sequenceDiagram
    participant M as Master
    participant N1 as Node 1
    participant N2 as Node 2
    participant N3 as Node 3

    Note over M,N3: Phase 1：Publish（發布新 state）
    M->>+N1: 發送新 cluster state
    M->>+N2: 發送新 cluster state
    M->>+N3: 發送新 cluster state
    N1-->>-M: ACK（已收到）
    N2-->>-M: ACK（已收到）
    N3-->>-M: ACK（已收到）
    Note over M: 等待 quorum 個 ACK 才繼續

    Note over M,N3: Phase 2：Commit（正式套用）
    M->>N1: Commit signal
    M->>N2: Commit signal
    M->>N3: Commit signal
    Note over N1,N3: 各 node 正式套用新 state
圖 4-3：Cluster State 兩階段提交流程

關鍵時間參數：

# Master 等待 publish 階段 ACK 的最大時間（預設 30s）
cluster.publish.timeout: 30s

# Master 等待 commit 階段完成的最大時間（預設 30s）
cluster.publish.info_timeout: 10s

如果 publish/commit 超時，master 會降階並重新選舉。這通常是 master 過載 或網路問題 的徵兆。

Cluster State 效能調校

常見導致 cluster state 過大的原因：

  • 數量過多的 indices（例如 daily index × 數年）
  • 單一 mapping 上千個 field（mapping explosion）
  • 過多 ILM/SLM policies、ingest pipelines、role mappings

排查工具：

# 檢視 cluster state 大小
curl -X GET "https://localhost:9200/_cluster/state?filter_path=metadata.indices.*.settings.index.number_of_shards" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt | wc -c

# 計算 indices 數量
curl -X GET "https://localhost:9200/_cat/indices?v" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt | wc -l

處理對策：

  • 合併小 indices（透過 ILM rollover）
  • 用 Data Streams 取代 daily indices
  • 合理使用 dynamic mapping，必要時設 "dynamic": "strict"

Cluster Fault Detection（故障偵測）

節點之間透過定期 ping偵測對方是否存活：

Follower Checks（master 檢查 followers）

Master 定期向所有非 master 節點發送 ping，超時或連續失敗則認定該節點離開叢集。

# 每次檢查的間隔（預設 1s）
cluster.fault_detection.follower_check.interval: 1s

# 單次 ping 超時（預設 10s）
cluster.fault_detection.follower_check.timeout: 10s

# 連續失敗次數閾值（預設 3）
cluster.fault_detection.follower_check.retry_count: 3

預設下，節點需連續 3 次每次 10 秒 ping 失敗（共約 30 秒），才會被踢出。

Leader Checks（followers 檢查 master）

非 master 節點也會定期 ping master：

cluster.fault_detection.leader_check.interval: 1s
cluster.fault_detection.leader_check.timeout: 10s
cluster.fault_detection.leader_check.retry_count: 3

若連續失敗，followers 會發起新一輪 master 選舉。

⚠️ 不要把 timeout 調太短 在 GC pause、磁碟暫時 I/O 飆高、跨 AZ 網路抖動時，短暫 ping 失敗很正常。把 timeout 調到 1-2 秒可能造成頻繁的不必要選舉，反而拖累叢集。預設值在絕大多數場景表現良好。

常見問題排錯

問題 1：master not discovered

節點啟動後 log 持續出現：

master not discovered yet, this node has not previously joined a bootstrapped cluster

可能原因與檢查：

  1. discovery.seed_hosts 列表中的 hostname/IP 是否正確？網路是否通？
  2. 9300 port 是否被防火牆阻擋？
  3. cluster.name 在所有節點是否一致？
  4. 是否有設 cluster.initial_master_nodes？（首次啟動必須）
  5. TLS 憑證是否相互信任？（CH 19 詳述）

問題 2：node name 不在 initial_master_nodes 列表

cluster.initial_master_nodes did not contain any nodes that match this node

原因：cluster.initial_master_nodes 中列的名字必須與 node.name 完全相符。預設 node.name 為機器 hostname，可能與你預期不同。明確設定 node.name 即可解決。

問題 3：split-brain 已發生

若不慎讓兩組節點各自形成獨立叢集，補救方式是選一邊保留、另一邊重新加入：

  1. 停止「要被廢棄」那邊的所有節點
  2. 清除這些節點的 path.data 資料目錄
  3. 確認 elasticsearch.yml 的 cluster.name、discovery.seed_hosts 指向「要保留」那邊
  4. 移除 cluster.initial_master_nodes
  5. 啟動，節點會以新成員身份加入
⚠️ 資料無法自動合併 Split-brain 期間兩邊各自寫入的資料無法自動合併。若資料重要，先用 snapshot 備份雙方再決定保留哪邊。

本章重點回顧

  • 節點透過 discovery.seed_hosts 發現彼此，可使用 file/cloud SD 動態管理
  • Quorum = floor(N/2) + 1，master-eligible 節點數應為 奇數（3/5/7）
  • Voting Configuration 由 master 動態維護；移除節點前應先用 voting exclusions API
  • cluster.initial_master_nodes 只在首次 bootstrap 使用，之後立即移除
  • Master 選舉採 Raft 變體，由 quorum 票決定
  • Cluster state 變更採兩階段提交，過大會拖累 publish 速度
  • Fault detection 透過定期 ping 偵測，預設 ~30 秒判定節點離線
  • 常見問題多源自網路不通、cluster.name 不一致、initial_master_nodes 設定錯誤
