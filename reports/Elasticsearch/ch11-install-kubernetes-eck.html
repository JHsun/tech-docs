CH 11 · Kubernetes ECK Operator 通用安裝

本章目標 認識 ECK Operator、用 YAML manifest / Helm 兩種方式安裝、部署第一個 Elasticsearch 叢集、CRD 完整對照

什麼是 ECK？

Elastic Cloud on Kubernetes (ECK) 是 Elastic 官方的 Kubernetes Operator，將 Elastic Stack 完整納入 K8s 原生管理：

  • 用 CRD（Custom Resource Definition）描述想要的叢集狀態
  • Operator 持續調節（reconcile）K8s 資源以滿足 desired state
  • 自動處理：StatefulSet 建立、PVC 配置、TLS 憑證輪替、滾動升級、配置變更、health check
  • 支援 Elasticsearch、Kibana、APM Server、Beats、Elastic Agent、Logstash、Maps Server、AutoOps Agent、Package Registry

支援的 Kubernetes 版本

ECK 3.3.x（2026/04 最新）支援：

  • Kubernetes 1.31 以上
  • OpenShift 4.16 以上
  • EKS、GKE、AKS、Rancher、Tanzu、其他 conformant 發行版
  • Kind、Minikube、k3s（開發用）
💡 ECK 與 Stack 版本對應 ECK 3.0 起支援 Elastic Stack 9.x。ECK 2.16+ 仍可管 8.x。單一 Operator 可同時管理多個不同版本的 Stack 部署。

兩種安裝方式

方式適用
YAML manifest快速安裝、無 Helm 環境、CI/CD 直接 apply

方式 1：YAML manifest 安裝

Step 1：安裝 CRDs

ECK_VERSION=3.3.2

# 建立 CRDs（cluster-wide 資源，需 cluster-admin 權限）
kubectl create -f https://download.elastic.co/downloads/eck/${ECK_VERSION}/crds.yaml

此步驟會建立以下 CRD：

CRD用途
elasticsearches.elasticsearch.k8s.elastic.coElasticsearch 叢集
kibanas.kibana.k8s.elastic.coKibana 部署
apmservers.apm.k8s.elastic.coAPM Server
beats.beat.k8s.elastic.coBeats（Filebeat、Metricbeat...）
agents.agent.k8s.elastic.coElastic Agent
logstashes.logstash.k8s.elastic.coLogstash pipeline
elasticmapsservers.maps.k8s.elastic.coElastic Maps Server
autoopsagents.autoops.k8s.elastic.coAutoOps Agent
packageregistries.packageregistry.k8s.elastic.coElastic Package Registry
stackconfigpolicies.stackconfigpolicy.k8s.elastic.co跨多叢集配置政策

Step 2：安裝 Operator

kubectl apply -f https://download.elastic.co/downloads/eck/${ECK_VERSION}/operator.yaml

會建立：

  • namespace: elastic-system
  • ServiceAccount、ClusterRole、ClusterRoleBinding
  • StatefulSet：elastic-operator（單副本）
  • Service、ValidatingWebhookConfiguration

Step 3：驗證

kubectl get pods -n elastic-system
# NAME                 READY   STATUS    RESTARTS   AGE
# elastic-operator-0   1/1     Running   0          30s

kubectl get crd | grep elastic.co
# elasticsearches.elasticsearch.k8s.elastic.co   2026-...
# kibanas.kibana.k8s.elastic.co                  2026-...
# ...

# 檢視 Operator log
kubectl logs -n elastic-system -f sts/elastic-operator

方式 2：Helm chart 安裝

Step 1：新增 repo

helm repo add elastic https://helm.elastic.co
helm repo update

# 查看可用版本
helm search repo elastic/eck-operator -l

Step 2：Cluster-wide 安裝

helm install elastic-operator elastic/eck-operator \
  --namespace elastic-system \
  --create-namespace \
  --version 3.3.2

Step 3（替代）：限定 namespace 安裝

多租戶環境下，可讓 operator 只管特定 namespaces：

# 1. CRDs 仍須由 cluster admin 安裝（一次）
helm install elastic-operator-crds elastic/eck-operator-crds \
  --version 3.3.2

# 2. 限定 operator 只管 'team-a-elastic' 與 'team-b-elastic' namespaces
helm install elastic-operator elastic/eck-operator \
  --namespace elastic-system \
  --create-namespace \
  --set installCRDs=false \
  --set createClusterScopedResources=false \
  --set managedNamespaces='{team-a-elastic,team-b-elastic}' \
  --set webhook.enabled=false \
  --version 3.3.2

完整 values.yaml 範例

# eck-values.yaml
replicaCount: 1

helm install elastic-operator elastic/eck-operator \
  --namespace elastic-system \
  --create-namespace \
  -f eck-values.yaml \
  --version 3.3.2

部署第一個 Elasticsearch 叢集（Quickstart）

建立 quickstart.yaml：

apiVersion: elasticsearch.k8s.elastic.co/v1
kind: Elasticsearch
metadata:
  name: quickstart
  namespace: default
spec:
  version: 9.3.4
  nodeSets:
    - name: default
      count: 1
      config:
        # 在 K8s 中通常關閉 mmap（影響效能但避免 vm.max_map_count 設定問題）
        # 生產環境改為使用 init container 設定 sysctl（見後文）
        node.store.allow_mmap: false

kubectl apply -f quickstart.yaml

# 觀察狀態
kubectl get elasticsearch
# NAME         HEALTH   NODES   VERSION   PHASE   AGE
# quickstart   green    1       9.3.4     Ready   2m

kubectl get pods --selector='elasticsearch.k8s.elastic.co/cluster-name=quickstart'

取得 elastic 密碼與 CA 憑證

# 密碼存在 Secret 中
PASSWORD=$(kubectl get secret quickstart-es-elastic-user \
  -o go-template='{{.data.elastic | base64decode}}')
echo $PASSWORD

# 取得 CA 憑證
kubectl get secret quickstart-es-http-certs-public \
  -o go-template='{{index .data "ca.crt" | base64decode}}' > ca.crt

從外部 / 本機連線測試

# 用 port-forward 暫時打通
kubectl port-forward service/quickstart-es-http 9200

# 另一個 terminal
curl --cacert ca.crt -u "elastic:$PASSWORD" https://localhost:9200/

生產級 Elasticsearch 叢集（多節點集合）

以下是一個典型的 hot/warm 分層生產配置，3 master + 3 data_hot + 3 data_warm：

apiVersion: elasticsearch.k8s.elastic.co/v1
kind: Elasticsearch
metadata:
  name: production
  namespace: elastic
spec:
  version: 9.3.4

  # ─── HTTP Service 設定 ─────────────────
  http:
    tls:
      selfSignedCertificate:
        subjectAltNames:
          - dns: elasticsearch.example.com
    service:
      spec:
        type: ClusterIP   # 內部使用，外部用 Ingress 暴露

  # ─── Update Strategy ──────────────────
  updateStrategy:
    changeBudget:
      maxSurge: 1
      maxUnavailable: 1

  # ─── Pod Disruption Budget ────────────
  # ECK 3.x 預設自動產生，無需手動設

  # ─── Node Sets ────────────────────────
  nodeSets:

    # ── Master nodes ──
    - name: master
      count: 3
      config:
        node.roles: ["master"]
        xpack.ml.enabled: false
      # 在 cloud K8s 上推薦用 zoneAwareness（ECK 3.4+）
      zoneAwareness:
        topologyKey: topology.kubernetes.io/zone
      podTemplate:
        spec:
          # init container 設定 sysctl（替代 node.store.allow_mmap: false）
          initContainers:
            - name: sysctl
              securityContext:
                privileged: true
                runAsUser: 0
              command: ['sh', '-c', 'sysctl -w vm.max_map_count=262144']
          containers:
            - name: elasticsearch
              env:
                - name: ES_JAVA_OPTS
                  value: "-Xms2g -Xmx2g"
              resources:
                requests:
                  memory: 4Gi
                  cpu: 1
                limits:
                  memory: 4Gi
                  cpu: 2
      volumeClaimTemplates:
        - metadata:
            name: elasticsearch-data
          spec:
            accessModes: ["ReadWriteOnce"]
            resources:
              requests:
                storage: 20Gi
            storageClassName: fast-ssd

    # ── Hot Data nodes ──
    - name: data-hot
      count: 3
      config:
        node.roles: ["data_hot", "data_content", "ingest"]
      zoneAwareness:
        topologyKey: topology.kubernetes.io/zone
      podTemplate:
        spec:
          initContainers:
            - name: sysctl
              securityContext: { privileged: true, runAsUser: 0 }
              command: ['sh', '-c', 'sysctl -w vm.max_map_count=262144']
          containers:
            - name: elasticsearch
              env:
                - name: ES_JAVA_OPTS
                  value: "-Xms16g -Xmx16g"
              resources:
                requests:
                  memory: 32Gi
                  cpu: 4
                limits:
                  memory: 32Gi
                  cpu: 8
      volumeClaimTemplates:
        - metadata:
            name: elasticsearch-data
          spec:
            accessModes: ["ReadWriteOnce"]
            resources:
              requests:
                storage: 1Ti
            storageClassName: fast-ssd

    # ── Warm Data nodes ──
    - name: data-warm
      count: 3
      config:
        node.roles: ["data_warm"]
      zoneAwareness:
        topologyKey: topology.kubernetes.io/zone
      podTemplate:
        spec:
          initContainers:
            - name: sysctl
              securityContext: { privileged: true, runAsUser: 0 }
              command: ['sh', '-c', 'sysctl -w vm.max_map_count=262144']
          containers:
            - name: elasticsearch
              env:
                - name: ES_JAVA_OPTS
                  value: "-Xms8g -Xmx8g"
              resources:
                requests:
                  memory: 16Gi
                  cpu: 2
                limits:
                  memory: 16Gi
                  cpu: 4
      volumeClaimTemplates:
        - metadata:
            name: elasticsearch-data
          spec:
            accessModes: ["ReadWriteOnce"]
            resources:
              requests:
                storage: 5Ti
            storageClassName: standard-ssd

StorageClass 規劃

本機 K8s 通常需要先建立 StorageClass。範例（與雲端對應的具體 storage 類型在 CH 12-14）：

# local-path-storage.yaml（適用 k3s/Kind/開發用）
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: rancher.io/local-path
volumeBindingMode: WaitForFirstConsumer
reclaimPolicy: Retain
allowVolumeExpansion: true

關鍵屬性：

  • volumeBindingMode: WaitForFirstConsumer — 確保 PVC 與 Pod 在同一 zone
  • reclaimPolicy: Retain — 刪除 ES 時 PV 與資料保留（避免誤刪）
  • allowVolumeExpansion: true — 允許後續擴容（ECK 自動處理）

日常管理操作

觀察叢集狀態

# 列出叢集
kubectl get elasticsearch -A

# 詳細資訊
kubectl describe elasticsearch production -n elastic

# 即時 events
kubectl get events -n elastic --sort-by='.lastTimestamp'

水平擴容（增加節點）

# 把 data-hot 從 3 → 6
kubectl patch elasticsearch production -n elastic --type='json' \
  -p='[{"op": "replace", "path": "/spec/nodeSets/1/count", "value": 6}]'

儲存擴容（增加 PVC 大小）

修改 manifest 中的 storage 大小（如 1Ti → 2Ti），kubectl apply。需要 StorageClass 支援 allowVolumeExpansion。

Stack 版本升級

# 直接修改 spec.version
kubectl patch elasticsearch production -n elastic --type='merge' \
  -p='{"spec":{"version":"9.3.5"}}'

# 同步升 Kibana
kubectl patch kibana production -n elastic --type='merge' \
  -p='{"spec":{"version":"9.3.5"}}'

ECK 會自動執行滾動升級：每次只重啟一個 pod，等 cluster green 後再下一個。

使用內建 elastic 使用者

kubectl get secret production-es-elastic-user -n elastic \
  -o go-template='{{.data.elastic | base64decode}}'

透過 StackConfigPolicy 統一配置

ECK 3.x 引入的 CRD，可批量管理多叢集的 settings、ingest pipelines、ILM、role mappings：

apiVersion: stackconfigpolicy.k8s.elastic.co/v1alpha1
kind: StackConfigPolicy
metadata:
  name: my-policy
  namespace: elastic
spec:
  resourceSelector:
    matchLabels:
      tier: production
  elasticsearch:
    clusterSettings:
      indices.recovery.max_bytes_per_sec: "100mb"
    snapshotRepositories:
      backup-repo:
        type: s3
        settings:
          bucket: my-es-backups
    ilmPolicies:
      logs-retention:
        policy:
          phases:
            hot: { actions: { rollover: { max_age: "1d" } } }
            delete: { min_age: "30d", actions: { delete: {} } }

升級 ECK Operator 本身

# YAML 方式
kubectl replace -f https://download.elastic.co/downloads/eck/3.3.3/crds.yaml
kubectl apply -f https://download.elastic.co/downloads/eck/3.3.3/operator.yaml

# Helm 方式
helm upgrade elastic-operator elastic/eck-operator \
  -n elastic-system --version 3.3.3
⚠️ Operator 升級可能觸發 rolling restart 某些 ECK 版本升級會調整 pod template（hash 變化），導致所有 ES pods 重啟。生產環境升級前先閱讀該版本的 release notes，並用 eck.k8s.elastic.co/managed=false annotation 暫時排除部分叢集，分批 rolling。

暫時排除某叢集

# 排除單一叢集
kubectl annotate --overwrite elasticsearch production -n elastic \
  eck.k8s.elastic.co/managed=false

# 此時 ECK 不會碰它，可手動 kubectl 操作底層 STS

# 重新交給 ECK 管理
kubectl annotate elasticsearch production -n elastic \
  eck.k8s.elastic.co/managed-

解除安裝 ECK

🚨 刪 CRD 會刪除所有 Elasticsearch！ CRD 是 cluster-wide 資源，刪除會 cascade 刪除全 cluster 內所有 elasticsearch、kibana 等 CR，導致所有叢集被刪除。先備份資料！ 
# 1. 先刪除所有 CR（保留 PVC 視 reclaimPolicy 而定）
kubectl delete elasticsearch --all -A
kubectl delete kibana --all -A

# 2. 刪除 operator
kubectl delete -f https://download.elastic.co/downloads/eck/3.3.2/operator.yaml

# 3. 最後刪 CRDs（如果你確定不再用）
kubectl delete -f https://download.elastic.co/downloads/eck/3.3.2/crds.yaml

本章重點回顧

  • ECK Operator 用 K8s 原生方式管理 Elastic Stack，CRD 描述 desired state
  • 最新版 ECK 3.3.2，需 K8s 1.31+ 或 OpenShift 4.16+
  • 兩種安裝方式：YAML manifest 或 Helm chart；Helm 適合 GitOps
  • NodeSet 是核心抽象：每個 NodeSet 對應一個 StatefulSet
  • 用 node.roles 區分 master / data tiers / ingest 等角色
  • StorageClass 必須支援 WaitForFirstConsumer 與 allowVolumeExpansion
  • 用 init container 設定 vm.max_map_count（避免 node.store.allow_mmap: false 的效能損失）
  • ECK 3.4+ 引入 zoneAwareness，自動處理跨 zone 部署
  • 升級 Stack 版本只需改 spec.version，ECK 自動 rolling
