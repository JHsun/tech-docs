關鍵特性：

  • 增量備份：第二次起只存 delta，省空間與時間
  • 共享 segment：跨 snapshot 共用未變的 Lucene segments
  • 不影響線上：背景執行，但會有 IO 影響
  • 包含內容：indices、cluster state、ILM/SLM/template/users 等 system 設定

Repository 類型

Type說明用途

設定 S3 Repository

# 1. 把 access key / secret 存 keystore（每節點）
sudo /usr/share/elasticsearch/bin/elasticsearch-keystore add s3.client.default.access_key
sudo /usr/share/elasticsearch/bin/elasticsearch-keystore add s3.client.default.secret_key

# 重新 reload（不需重啟）
curl -X POST -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_nodes/reload_secure_settings"

# 2. 註冊 repository
curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_snapshot/primary?pretty" -d '
{
  "type": "s3",
  "settings": {
    "bucket":      "es-snapshots-production",
    "region":      "ap-northeast-1",
    "base_path":   "cluster1/",
    "compress":    true,
    "max_snapshot_bytes_per_sec":  "200mb",
    "max_restore_bytes_per_sec":   "400mb",
    "server_side_encryption":      true,
    "storage_class":               "standard_ia"
  }
}'

# 3. 驗證
curl -X POST -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_snapshot/primary/_verify"
✅ AWS IRSA / GCP Workload Identity / Azure Managed Identity 雲端 K8s 環境（CH 12-14）建議用原生身份取代 access key：
  • AWS：把 IAM role 透過 IRSA 綁到 ES pod 的 ServiceAccount
  • GCP：用 Workload Identity
  • Azure：用 Managed Identity
完全無需 access key，更安全且無 rotate 負擔。

設定 FS Repository（NFS）

# 1. 每個節點 elasticsearch.yml 設定 path.repo
path.repo:
  - "/mnt/nfs-snapshot"

# 2. NFS 必須掛載到所有節點同一路徑（含 master / data / 將參與 snapshot 的節點）

# 重啟節點

PUT _snapshot/local-fs
{
  "type": "fs",
  "settings": {
    "location": "/mnt/nfs-snapshot/cluster1",
    "compress": true
  }
}

建立 Snapshot

手動

# 全 cluster snapshot
PUT _snapshot/primary/snap-2026-04-15
{
  "indices":         "*",
  "ignore_unavailable": true,
  "include_global_state": true,
  "metadata": {
    "taken_by": "ops",
    "reason":   "before-version-upgrade"
  }
}

# 等待完成
PUT _snapshot/primary/snap-2026-04-15?wait_for_completion=true
{ ... }

# 部分 indices
PUT _snapshot/primary/snap-2026-04-15
{
  "indices":         "logs-*,metrics-*",
  "feature_states":  [ "kibana", "security" ]
}

看 snapshot 狀態

GET /_snapshot/primary/snap-2026-04-15
GET /_snapshot/primary/snap-2026-04-15/_status

# 列所有 snapshot
GET /_snapshot/primary/_all

# 用 _cat
GET /_cat/snapshots/primary?v

Snapshot Lifecycle Management (SLM)

自動化 snapshot 與保留期管理。

PUT _slm/policy/daily-snapshots
{
  "schedule":      "0 30 1 * * ?",          // 每日 01:30
  "name":          "<daily-snap-{now/d}>",   // 名稱範本
  "repository":    "primary",
  "config": {
    "indices":              [ "*" ],
    "ignore_unavailable":   true,
    "include_global_state": true,
    "feature_states":       [ "kibana", "security", "logstash_management" ]
  },
  "retention": {
    "expire_after":  "30d",
    "min_count":     5,
    "max_count":     50
  }
}

retention 邏輯

  • expire_after：超過此時間才考慮刪
  • min_count：永遠保留至少 N 個（防全砍）
  • max_count：超過 N 個就刪最舊

三者組合：例如 expire_after=30d + min=5 + max=50，意思「保留 5-50 個 snapshot，超過 30 天的會刪但至少留 5 個」。

SLM 管理

# 立即執行一次
POST /_slm/policy/daily-snapshots/_execute

# 看 policy 狀態
GET /_slm/policy/daily-snapshots

# 看 SLM 服務狀態
GET /_slm/status

# 暫停 / 啟動 SLM 服務
POST /_slm/stop
POST /_slm/start

# 看執行紀錄
GET /_slm/stats

多 policy（短/長期備份分流）

# 短期：每小時一次，保留 24 小時
PUT _slm/policy/hourly
{
  "schedule":   "0 0 * * * ?",
  "name":       "",
  "repository": "primary",
  "config": { "indices": [ "*" ] },
  "retention":  { "expire_after": "24h", "min_count": 3, "max_count": 24 }
}

# 中期：每日，保留 30 天
PUT _slm/policy/daily
{
  "schedule":   "0 30 1 * * ?",
  "name":       "",
  "repository": "primary",
  "retention":  { "expire_after": "30d", "min_count": 7, "max_count": 30 }
}

# 長期：每月，保留 1 年
PUT _slm/policy/monthly
{
  "schedule":   "0 0 2 1 * ?",
  "name":       "",
  "repository": "primary-archive",     # 用較便宜的 archive bucket
  "retention":  { "expire_after": "365d", "min_count": 6, "max_count": 12 }
}

Restore：還原

基本

POST _snapshot/primary/snap-2026-04-15/_restore
{
  "indices":              "logs-2026.04.10",
  "ignore_unavailable":   true,
  "include_global_state": false,
  "rename_pattern":       "logs-(.+)",
  "rename_replacement":   "restored-logs-$1",
  "index_settings": {
    "index.number_of_replicas": 0
  }
}

關鍵選項

  • indices：哪些 index（支援 wildcard）
  • rename_pattern / rename_replacement：避免覆蓋既有 index
  • include_global_state：是否還原 cluster 設定（templates、ILM 等）。還原到既有 cluster 時通常設 false，避免覆蓋現有設定
  • index_settings：還原時調整 settings
  • partial：true 時即使部分 shard 缺失也還原

看 restore 進度

GET /_cat/recovery?v&active_only=true&h=index,shard,type,stage,bytes_percent,files_percent

recovery type 為 snapshot 即是 restore 中。

跨版本還原

⚠️ 不能還原到更舊版本 9.0 的 snapshot 不能還原到 8.x cluster。但反過來可以（8.x snapshot → 9.x cluster）。

跨多版本還原

例：要還原 7.17 snapshot 到 9.x cluster：

  1. 啟動一個 8.18 cluster
  2. 在 8.18 還原 snapshot（會自動升級 index format）
  3. 在 8.18 take 新 snapshot
  4. 在 9.x 還原該 8.18 snapshot

或用 archive functionality（9.x，把舊 index 以 read-only archive 形式 mount）。

Searchable Snapshot（CH 30/32）

Searchable snapshot 不是備份，而是把 snapshot 直接當 index 用。Cold/Frozen tier 用此實現。

POST _snapshot/primary/snap-2026-04-15/_mount?storage=full_copy
{
  "index":           "logs-2026-01",
  "renamed_index":   "searchable-logs-2026-01"
}

storage 兩種：

  • full_copy：cold tier 用，完整 mount 到本機
  • shared_cache：frozen tier 用，partially mount

Snapshot 加密

伺服器端加密（推薦）

S3 / GCS / Azure 都支援 server-side encryption（SSE）：

{
  "type": "s3",
  "settings": {
    "bucket":                "...",
    "server_side_encryption": true        // SSE-S3 / SSE-KMS
  }
}

客戶端加密（合規嚴格）

ES 沒有內建 repository 加密，但可：

  • 用 encryption repository（Platinum）：在 ES 端用自訂 key 加密
  • 或在 S3 bucket 啟用 SSE-KMS，用自管 key

效能調校

節點併發數

# elasticsearch.yml
snapshot.max_concurrent_operations: 1000     # 預設 1000，幾乎無上限

頻寬限制

{
  "type": "s3",
  "settings": {
    "max_snapshot_bytes_per_sec":  "200mb",   // 寫入限速
    "max_restore_bytes_per_sec":   "400mb"
  }
}

並行 shards

PUT _cluster/settings
{
  "persistent": {
    "indices.recovery.max_concurrent_snapshot_file_downloads": 5
  }
}

疑難排解

Repository corrupted

# 修復 metadata
POST _snapshot/primary/_cleanup

# 從某 snapshot 重建（罕用）
POST _snapshot/primary/_cleanup?master_timeout=60m

Restore 卡住

# 看哪些 shard 卡住
GET /_cat/recovery?v&h=index,shard,stage,type | grep -v done

# allocation explain
POST /_cluster/allocation/explain
{
  "index": "restored-foo",
  "shard": 0,
  "primary": true
}

最佳實踐

  • SLM 多 policy 分層：hourly + daily + monthly
  • 跨 region replication：S3 cross-region replication 到不同地理區
  • SLM retention 設 min_count，防止全砍
  • 用雲端原生身份（IRSA / Workload Identity）取代 access key
  • ILM delete phase 加 wait_for_snapshot，確保有備份再刪
  • 定期測試 restore：每季模擬還原到 staging
  • 監控 SLM 失敗：用 watcher 監控 .slm-history-*
  • 9.x 用 _health_report：含 slm、repository_integrity 健康檢查
  • snapshot 期間限速，避免影響線上
  • 長期 archive 用 source repository + 便宜 storage class

本章重點回顧

  • Snapshot 增量備份；包含 indices + cluster state
  • Repository 支援 fs / s3 / gcs / azure / hdfs / url / source
  • SLM 自動化：schedule + retention + min/max count 三段保留
  • Restore 可 rename、調整 settings；include_global_state=false 不覆蓋現有
  • 跨版本還原：只能新版接舊版；超兩個 major 需中間 cluster 跳板
  • Searchable snapshot（CH 30）把 snapshot 當 index 用，不是備份
  • Snapshot 加密：用 server-side（SSE）即足夠，合規再加 KMS / encryption repo
  • 雲端用 IRSA / Workload Identity 取代 access key
  • 定期 restore 演練是 DR 的真正測試
