圖 19-1：HTTP TLS（外部）與 Transport TLS（節點間）兩個獨立的加密層
層Port用途必要性
Transport TLS9300節點之間互傳 cluster state、shard data、search/index 廣播多節點叢集必須（mutual TLS）
HTTP TLS9200客戶端 REST API（瀏覽器、curl、SDK）強烈建議啟用
⚠️ Production mode 必須啟用 transport TLS Elasticsearch 9.x 在 production mode（network.host 非 loopback）下，沒設定 transport TLS 會拒絕啟動。

兩種設定方式

  1. 自動安全設定（Auto-config）—— 首次啟動自動產生（CH 07 提過）。最簡單，適合單機測試與一般環境
  2. 手動設定—— 自己產 CA、自己控制憑證有效期、可整合企業 PKI。生產環境推薦

自動安全設定（Auto-config）回顧

tar.gz / 套件首次啟動時：

  1. 產生 CA（私鑰存於 keystore）
  2. 產生 transport 與 HTTP 憑證
  3. 更新 elasticsearch.yml 啟用 TLS
  4. 產生 elastic 使用者密碼與 enrollment tokens

檔案位置：$ES_PATH_CONF/certs/

適合場景：開發、測試、單機 PoC。

不適合：你需要全控憑證有效期、與企業 PKI 整合、跨叢集共享 CA、要把 CA 部署給多個 Beat / Logstash 節點。

圖 19-2：手動 TLS 設定的完整流程

Step 1：產生 CA

用 ES bundled 的 elasticsearch-certutil 工具：

cd /usr/share/elasticsearch

# 互動式（提示密碼）
sudo bin/elasticsearch-certutil ca

# 批次（給密碼）
sudo bin/elasticsearch-certutil ca \
  --out /etc/elasticsearch/certs/elastic-stack-ca.p12 \
  --pass "ChangeMeCApassword" \
  --days 3650

輸出：

  • elastic-stack-ca.p12：PKCS#12 格式，內含 CA 私鑰 + 公鑰
  • 放到安全的地方並備份。CA 私鑰外洩 = 整個 ES PKI 失效

取出 PEM 格式 CA cert

給 Kibana / Beats / 外部 client 用：

openssl pkcs12 -in /etc/elasticsearch/certs/elastic-stack-ca.p12 \
  -clcerts -nokeys \
  -out /etc/elasticsearch/certs/ca.crt \
  -passin pass:ChangeMeCApassword

Step 2：產生 Transport 憑證

Transport 用 mutual TLS，所有節點通常用同一份憑證（包含 wildcard 或所有 hostname）：

sudo bin/elasticsearch-certutil cert \
  --ca /etc/elasticsearch/certs/elastic-stack-ca.p12 \
  --ca-pass "ChangeMeCApassword" \
  --out /etc/elasticsearch/certs/transport.p12 \
  --pass "ChangeMeTransportPassword" \
  --days 730

Per-node Transport 憑證（更嚴格）

建立 instances.yml：

instances:
  - name: "es-node-1"
    dns: ["es-node-1.internal", "es-node-1"]
    ip:  ["10.0.1.10"]
  - name: "es-node-2"
    dns: ["es-node-2.internal", "es-node-2"]
    ip:  ["10.0.1.11"]
  - name: "es-node-3"
    dns: ["es-node-3.internal", "es-node-3"]
    ip:  ["10.0.1.12"]

sudo bin/elasticsearch-certutil cert \
  --ca /etc/elasticsearch/certs/elastic-stack-ca.p12 \
  --ca-pass "ChangeMeCApassword" \
  --in /etc/elasticsearch/certs/instances.yml \
  --out /etc/elasticsearch/certs/transport-bundle.zip \
  --pem

解壓後得到每節點的 .crt + .key + ca.crt。

Step 3：產生 HTTP 憑證

HTTP 用單向 TLS（client 驗證 server），通常需要 hostname / SAN 對應：

sudo bin/elasticsearch-certutil http

互動式問答（重點回答）：

  • Generate a CSR? → n
  • Use an existing CA? → y
  • CA Path: → /etc/elasticsearch/certs/elastic-stack-ca.p12
  • CA password: → 你的 CA 密碼
  • Validity: → 730d（2 年）
  • Generate one certificate per node? → y
  • 輸入每節點 hostname/IP（為 SAN 用）
  • Key passphrase → 設一個密碼或留空

輸出：elasticsearch-ssl-http.zip，內含 elasticsearch/ 與 kibana/ 兩個目錄。

Step 4：部署到各節點

# 在 master 上產完所有憑證後，scp 到其他節點
for host in es-node-2 es-node-3; do
  sudo scp /etc/elasticsearch/certs/transport.p12 $host:/etc/elasticsearch/certs/
  sudo scp /etc/elasticsearch/certs/ca.crt        $host:/etc/elasticsearch/certs/
done

# 在每個節點上：
sudo chown -R root:elasticsearch /etc/elasticsearch/certs
sudo chmod 750 /etc/elasticsearch/certs
sudo chmod 640 /etc/elasticsearch/certs/*

Step 5：用 keystore 存憑證密碼

不要把密碼明文寫在 yml：

sudo /usr/share/elasticsearch/bin/elasticsearch-keystore add \
  xpack.security.transport.ssl.keystore.secure_password
# 提示輸入：ChangeMeTransportPassword

sudo /usr/share/elasticsearch/bin/elasticsearch-keystore add \
  xpack.security.transport.ssl.truststore.secure_password
# 提示輸入：ChangeMeTransportPassword

sudo /usr/share/elasticsearch/bin/elasticsearch-keystore add \
  xpack.security.http.ssl.keystore.secure_password
# 提示輸入：HTTP keystore 密碼

Step 6：更新 elasticsearch.yml

每個節點的 elasticsearch.yml 加入：

# ─── Security 啟用 ──────────────────────────
xpack.security.enabled: true

# ─── Transport TLS（節點間 mutual TLS）─────
xpack.security.transport.ssl.enabled: true
xpack.security.transport.ssl.verification_mode: certificate
xpack.security.transport.ssl.client_authentication: required
xpack.security.transport.ssl.keystore.path: certs/transport.p12
xpack.security.transport.ssl.truststore.path: certs/transport.p12
# 密碼存於 keystore（上一步）

# ─── HTTP TLS（外部連線）──────────────────
xpack.security.http.ssl.enabled: true
xpack.security.http.ssl.keystore.path: certs/http.p12
# 密碼存於 keystore

verification_mode 三種模式

模式檢查內容適用
fullCA 有效 + hostname 比對per-node 憑證、嚴格環境
certificate只檢查 CA 有效共用 cluster 憑證（最常用）
none不檢查（不建議）—

Step 7：Rolling Restart

# 一個一個節點重啟
sudo systemctl restart elasticsearch

# 觀察
sudo journalctl -u elasticsearch -f
# 找到：
# [INFO ][o.e.x.s.SecurityFeatureSet] [es-node-1] cluster security has been enabled
# [INFO ][o.e.t.TransportService    ] [es-node-1] publish_address {10.0.1.10:9300}, ssl=true

Step 8：驗證

# 用 CA 驗證
curl --cacert /etc/elasticsearch/certs/ca.crt \
  -u elastic:xxx \
  https://es-node-1.internal:9200/

# 看 cluster 中所有節點 TLS 狀態
curl --cacert ca.crt -u elastic:xxx \
  https://es-node-1.internal:9200/_ssl/certificates?pretty

輸出範例：

[
  {
    "path": "certs/transport.p12",
    "format": "PKCS12",
    "alias": "instance",
    "subject_dn": "CN=instance",
    "expiry": "2028-04-15T...",
    "has_private_key": true
  }
]

使用自有企業 CA

大型企業通常有 enterprise PKI（如 Microsoft AD CS、HashiCorp Vault PKI、Smallstep）。流程：

Step 1：產生 CSR

sudo bin/elasticsearch-certutil http
# 互動式，這次回答 'y' to "Generate a CSR"

輸出 http.csr 與 http.key。

Step 2：交給企業 CA 簽

把 .csr 交給 CA team 簽署，得到 http.crt 與 ca chain。

Step 3：用 PEM 直接設定

xpack.security.http.ssl.enabled: true
xpack.security.http.ssl.key:                certs/http.key
xpack.security.http.ssl.certificate:        certs/http.crt
xpack.security.http.ssl.certificate_authorities: ["certs/enterprise-ca-chain.crt"]

xpack.security.transport.ssl.enabled: true
xpack.security.transport.ssl.verification_mode: certificate
xpack.security.transport.ssl.key:                certs/transport.key
xpack.security.transport.ssl.certificate:        certs/transport.crt
xpack.security.transport.ssl.certificate_authorities: ["certs/enterprise-ca-chain.crt"]

HTTP key 密碼也存 keystore：

sudo /usr/share/elasticsearch/bin/elasticsearch-keystore add \
  xpack.security.http.ssl.secure_key_passphrase

憑證輪替（不停機）

✅ ES 自動偵測憑證更新 Elasticsearch 監控所有 TLS 相關的檔案。當你更新檔案時，ES 會自動 reload，無需重啟。但要注意：
  • 新憑證必須由同一個 CA簽發
  • 檔名與路徑要保持一致（直接 overwrite，不要改名再 symlink）
  • 建議先 reload 一個節點驗證再推全部

操作步驟

# 1. 檢查目前憑證有效期
curl --cacert ca.crt -u elastic:xxx \
  https://localhost:9200/_ssl/certificates?pretty | jq '.[]|{path,expiry}'

# 2. 用同一 CA 產新憑證（如將 730 天）
sudo bin/elasticsearch-certutil cert \
  --ca /etc/elasticsearch/certs/elastic-stack-ca.p12 \
  --ca-pass "xxx" \
  --out /tmp/transport-new.p12 \
  --pass "ChangeMeTransportPassword" \
  --days 730

# 3. 替換舊憑證
sudo cp /tmp/transport-new.p12 /etc/elasticsearch/certs/transport.p12
sudo chown root:elasticsearch /etc/elasticsearch/certs/transport.p12
sudo chmod 640 /etc/elasticsearch/certs/transport.p12

# 4. 在 log 觀察 reload（無需 restart！）
sudo tail -f /var/log/elasticsearch/*.log | grep -i 'reload\|tls\|ssl'
# 應看到 "reloaded SSL configuration"

# 5. 驗證新憑證有效期
curl --cacert ca.crt -u elastic:xxx \
  https://localhost:9200/_ssl/certificates?pretty

CA 輪替（更高難度）

當需要換 CA（例如舊 CA 過期、CA 私鑰外洩）：

  1. 產生新 CA
  2. 用新 CA 產生新節點憑證
  3. 把新舊兩個 CA 都加入 truststore
  4. Rolling restart 讓所有節點同時信任兩個 CA
  5. 把節點憑證逐步替換為新 CA 簽的
  6. 確認所有節點與 client 都用新憑證後，從 truststore 移除舊 CA
# 過渡期 yml
xpack.security.transport.ssl.certificate_authorities:
  - "certs/old-ca.crt"
  - "certs/new-ca.crt"

Kibana 連 ES 的 TLS

Kibana 必須信任 ES 的 CA。kibana.yml：

# ── Kibana → ES（client 端，信任 ES 的 CA）──
elasticsearch.hosts: ["https://es-node-1.internal:9200"]
elasticsearch.ssl.certificateAuthorities: [ "/etc/kibana/certs/ca.crt" ]
elasticsearch.ssl.verificationMode: full

# ── Kibana 對外 HTTPS（server 端，自己也是 HTTPS）──
server.ssl.enabled: true
server.ssl.certificate: /etc/kibana/certs/kibana.crt
server.ssl.key:         /etc/kibana/certs/kibana.key

Beats / Logstash 連 ES 的 TLS

Filebeat

output.elasticsearch:
  hosts: ["https://es-node-1.internal:9200"]
  username: "filebeat_writer"
  password: "${ES_PWD}"
  ssl.certificate_authorities: ["/etc/filebeat/ca.crt"]
  ssl.verification_mode: full

Logstash

output {
  elasticsearch {
    hosts => ["https://es-node-1.internal:9200"]
    user => "logstash_writer"
    password => "${ES_PWD}"
    ssl_enabled => true
    ssl_certificate_authorities => "/etc/logstash/ca.crt"
    ssl_verification_mode => "full"
  }
}

疑難排解

錯誤 1：PKIX path validation failed

javax.net.ssl.SSLHandshakeException: PKIX path validation failed: 
java.security.cert.CertPathValidatorException: validity check failed

原因：憑證過期，或 client 不信任 CA。

# 檢查憑證有效期
openssl x509 -in /etc/elasticsearch/certs/http.crt -noout -dates

# 從 .p12 取出
keytool -list -v -keystore /etc/elasticsearch/certs/transport.p12 -storepass xxx

錯誤 2：hostname verification failed

No subject alternative DNS name matching node-1.example.com found.

原因：憑證 SAN 沒包含這個 hostname。

# 看憑證 SAN
openssl x509 -in /etc/elasticsearch/certs/http.crt -noout -text | grep -A1 "Subject Alternative Name"
# 應有 DNS:node-1.example.com

處理：用正確 hostname 連線；或調整 verification mode 為 certificate；或重新簽發含正確 SAN 的憑證。

錯誤 3：no cipher suites in common

client / server 支援的 cipher 不交集。檢查：

openssl s_client -connect node1:9200 -showcerts -tls1_2

加密套件（cipher suites）與 TLS 版本

9.x 預設啟用 TLSv1.3 與 TLSv1.2。如要限制：

xpack.security.transport.ssl.supported_protocols: [ "TLSv1.3", "TLSv1.2" ]
xpack.security.http.ssl.supported_protocols:      [ "TLSv1.3", "TLSv1.2" ]

xpack.security.http.ssl.cipher_suites:
  - "TLS_AES_256_GCM_SHA384"
  - "TLS_CHACHA20_POLY1305_SHA256"
  - "TLS_AES_128_GCM_SHA256"
  - "TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384"
  - "TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384"

本章重點回顧

  • ES 有兩個 TLS 層：Transport（9300，mutual TLS）與 HTTP（9200，server TLS）
  • Production mode 必須啟用 transport TLS 才能啟動
  • 用 elasticsearch-certutil 三步驟：產 CA → 產 transport cert → 產 HTTP cert
  • 所有憑證密碼放 elasticsearch-keystore，不要明文於 yml
  • verification_mode 通常用 certificate（共用 cluster 憑證）；嚴格環境用 full
  • ES 自動 reload TLS 檔案，憑證輪替不需重啟
  • CA 輪替策略：truststore 暫時同時信任新舊 CA，再逐步替換 cert
  • Kibana / Beats / Logstash 必須信任同一個 CA才能連線
