為何需要 Index Templates？

當你產生大量同類 index（例如 logs-2026-01、logs-2026-02），手動每個都設 mapping/settings 不可行。Index Template 讓你事先定義規則，符合 pattern 的新 index 自動套用。

建立 Component Templates

settings 模組

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_component_template/logs-settings" -d '
{
  "template": {
    "settings": {
      "number_of_shards":   3,
      "number_of_replicas": 1,
      "index.codec":        "best_compression",
      "index.refresh_interval": "5s"
    }
  },
  "_meta": {
    "description": "Common settings for logs",
    "version":     "1.0.0"
  }
}'

{

    "mappings": {
      "properties": {

建立 Index Template

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_index_template/logs-template" -d '
{
  "index_patterns": [ "logs-*" ],
  "data_stream": { },
  "priority": 200,
  "composed_of": [
    "logs-settings",
    "logs-mappings",
    "logs-policy"
  ],
  "template": {
    "settings": {
      "index.lifecycle.rollover_alias": "logs"
    }
  },
  "_meta": {
    "description": "Template for application logs",
    "version":     "1.0.0",
    "managed":     false
  }
}'

關鍵欄位

欄位說明
index_patternsmatch 規則（可多個 pattern）
composed_of引用的 component templates，順序由前到後合併，後者覆蓋前者
template本 template 直接定義的 settings/mappings/aliases，合併最後，覆蓋 composed_of
data_stream有此 key 即代表此 template 用於 data stream（CH 29）
priority多個 template 同時 match 時的優先序，數字大者勝
version純記錄用，不影響行為
_meta自訂 metadata

@custom Component Template 模式

9.x Elastic 的 managed template（如 logs@template）會自動 reference logs@custom component template。不要直接改 managed template，而是建一個 logs@custom 來客製化：

# 不要動 logs@template！改建 logs@custom
curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_component_template/logs@custom" -d '
{
  "template": {
    "settings": {
      "number_of_shards": 5,           # 覆蓋預設的 1
      "index.lifecycle.name": "my-custom-logs-policy"
    },
    "mappings": {
      "properties": {
        "company.tenant_id":  { "type": "keyword" },
        "internal.team_name": { "type": "keyword" }
      }
    }
  }
}'

更新 ES 版本時，logs@template 會跟著升級，但 logs@custom 保留。這是非破壞性客製化的標準做法。

9.3+ 命名規範

Elastic Stack 9.3+ 開始將內建 ILM policy 與 template 統一改為 {type}@lifecycle、{type}@template、{type}@custom 格式：

  • logs@template、logs@custom、logs@lifecycle
  • metrics@template、metrics@custom、metrics@lifecycle
  • synthetics@template、synthetics@custom、synthetics@lifecycle
  • traces@template、traces@custom

升級時舊命名（logs、metrics）的 ILM policy 會被替換為 logs@lifecycle。

多 template 衝突解決

當一個 index 名同時 match 多個 template，priority 大者勝，且不是合併是擇優：

# Template A
PUT _index_template/logs-general
{
  "index_patterns": ["logs-*"],
  "priority": 100,
  "composed_of": ["logs-settings"]
}

# Template B
PUT _index_template/logs-app-special
{
  "index_patterns": ["logs-app-*"],
  "priority": 200,    # 高，會勝出
  "composed_of": ["logs-settings", "app-mappings"]
}

# 建立 logs-app-2026-04 時 → 套用 logs-app-special（不會合併 logs-general）
⚠️ Priority 不是合併 與 component template 內部合併不同，多個 index template 之間是「擇一」。同一 index 只會套用一個 index template（最高 priority 者）。

Dynamic Templates vs Component Templates

項目Dynamic TemplateComponent Template
位置mapping 內cluster 層級
觸發時機新欄位首次出現時新 index 建立時
影響範圍單一 index 內的欄位多個 index（透過 index template）
用途處理未預期的欄位標準化 index 結構

兩者可組合使用：在 component template 的 mapping 裡加 dynamic templates。

Legacy Template 遷移

如果你還在用舊的 _template，9.x 仍可用但已棄用。建議盡快遷移：

# 找出所有 legacy template
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_template?pretty"

# 對每個拆成 component + index template 組合
# 然後刪除 legacy
curl -X DELETE -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_template/old-logs-template"
⚠️ Legacy template 與 composable template 共存時的優先序 Composable index template 永遠優先於 legacy template。所以遷移過程中 legacy 仍可保留，逐步替換。

完整生產級範例

多租戶 SaaS 的 logs：

# 1. settings component
PUT _component_template/saas-logs-settings
{
  "template": {
    "settings": {
      "number_of_shards":   3,
      "number_of_replicas": 1,
      "index.codec":        "best_compression",
      "index.lifecycle.name": "saas-logs-90d",
      "index.routing.allocation.total_shards_per_node": 2
    }
  }
}

# 2. core mappings component
PUT _component_template/saas-logs-mappings-core
{
  "template": {
    "mappings": {
      "dynamic": "strict",
      "_meta": { "version": "1.0" },
      "properties": {
        "@timestamp":        { "type": "date" },
        "message":           { "type": "match_only_text" },
        "log.level":         { "type": "keyword" },
        "service.name":      { "type": "keyword" },
        "service.version":   { "type": "keyword" },
        "host.name":         { "type": "keyword" },
        "host.ip":           { "type": "ip" },
        "tenant_id":         { "type": "keyword" },
        "trace.id":          { "type": "keyword" }
      }
    }
  }
}

# 3. dynamic templates component（給 metrics-style fields）
PUT _component_template/saas-logs-dynamic
{
  "template": {
    "mappings": {
      "dynamic_templates": [
        {
          "metric_long": {
            "match_pattern": "regex",
            "match": "metric\\..*\\.count$",
            "mapping": { "type": "long" }
          }
        },
        {
          "ip_fields": {
            "match": "*_ip",
            "mapping": { "type": "ip" }
          }
        },
        {
          "default_string": {
            "match_mapping_type": "string",
            "mapping": { "type": "keyword", "ignore_above": 256 }
          }
        }
      ]
    }
  }
}

# 4. 組合成 index template
PUT _index_template/saas-logs-template
{
  "index_patterns": [ "saas-logs-*" ],
  "data_stream": { },
  "priority": 500,
  "composed_of": [
    "saas-logs-settings",
    "saas-logs-mappings-core",
    "saas-logs-dynamic"
  ],
  "_meta": {
    "description": "SaaS multi-tenant logs",
    "version":     "1.0.0",
    "owner":       "platform-team"
  }
}

最佳實踐

  • 用 component template 模組化：settings、mappings、dynamic_templates 各自一個
  • 命名一致：{purpose}-{role} 例如 logs-settings、logs-mappings
  • 用 _meta 紀錄 version、owner，便於追蹤
  • Priority 用 100/200/500 等大跳躍值，未來插入新 template 較有彈性
  • Managed template 不直接改，建立對應 {type}@custom
  • 遷移 legacy template，9.x 是過渡期，未來版本可能完全移除
  • 用 _simulate API 先驗證，避免新 template 上線後才發現問題
  • Templates 進入 GitOps 管理：用 Terraform / ECK / kustomize 版本控管

本章重點回顧

  • Index Template：定義 pattern + 引用 components；新 index 自動套用
  • Component Template：可重用模組（settings、mappings、aliases）
  • 合併規則：components 由前到後合併（後者勝），最後再被 index template 自身的 template 覆蓋
  • Priority 是多個 index template 衝突時的「擇優」（不是合併）
  • {type}@custom 是客製內建 template 的標準模式（9.3+ 統一命名）
  • Dynamic templates（mapping 內）vs Component templates（cluster 層級）作用範圍不同
  • 用 _simulate_index 模擬套用結果；用 _simulate 測試 template 定義
  • Legacy _template 已棄用，盡快遷移到 composable
