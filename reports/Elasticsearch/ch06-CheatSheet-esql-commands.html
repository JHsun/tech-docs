📐 A. ES|QL 基本架構

ES|QL 是 9.x 主推的 pipe-based 查詢語言。SQL-like 語法 + Unix pipe 風格，比 Query DSL 簡潔太多。

情境
命令 / Code
說明
透過 _query API
POST /_query?format=txt
{
  "query": """
    FROM logs-*
    | WHERE @timestamp > NOW() - 1 hour
    | STATS count = COUNT(*) BY level
    | SORT count DESC
  """
}
正式 API。format=txt / json / csv / yaml
用 query parameter
POST /_query
{
  "query": "FROM logs-* | WHERE id == ? | LIMIT ?",
  "params": [ "abc-123", 100 ]
}
參數化（防注入）
Kibana Discover
Kibana → Discover → 切換 ES|QL mode
互動式開發 ES|QL 查詢
基本語法結構
FROM <source>
  | <processing-command1>
  | <processing-command2>
  | ...
Pipe-based：源資料 → 處理 → 處理 → 結果
備註與多行

| WHERE level == "ERROR"
/* 統計每服務 */

🔍 C. WHERE / SORT / LIMIT

Command
範例
說明
WHERE 比較
FROM logs-*
| WHERE status >= 400 AND service == "api"
== != > < >= <= AND OR NOT
WHERE IN / NOT IN
FROM logs-*
| WHERE level IN ("ERROR", "FATAL")
| WHERE country NOT IN ("TW", "JP")
清單比對
WHERE LIKE / RLIKE
FROM logs-*
| WHERE message LIKE "*timeout*"
| WHERE path RLIKE "/api/v\\d+/.*"
LIKE 用 * ?；RLIKE 用 regex
WHERE IS NULL
FROM logs-*
| WHERE error_msg IS NOT NULL
檢查欄位是否有值
WHERE 全文 MATCH
FROM articles
| WHERE MATCH(content, "elasticsearch tutorial")
在 ES|QL 內做全文搜尋
WHERE 時間運算
FROM logs-*
| WHERE @timestamp > NOW() - 24 hours
| WHERE @timestamp >= "2026-05-01T00:00:00Z"
NOW() + 時間 literal（hours / days / weeks 等）
SORT
FROM logs-*
| SORT @timestamp DESC, severity ASC
單 / 多欄位排序
SORT NULLS
FROM logs-*
| SORT priority DESC NULLS LAST
NULLS FIRST 或 NULLS LAST
LIMIT
FROM logs-*
| SORT @timestamp DESC
| LIMIT 100
預設 1000，最大 10000

📝 D. Column 操作（KEEP / DROP / RENAME / EVAL）

Command
範例
說明
KEEP
FROM logs-*
| KEEP @timestamp, service, message, level
只留指定欄位（也決定輸出順序）
KEEP wildcard
FROM logs-*
| KEEP @timestamp, host.*, service
host.* 留所有 host. 開頭欄位
DROP
FROM logs-*
| DROP @version, agent.*
刪除指定欄位
RENAME
FROM logs-*
| RENAME service.name AS service, response.status AS status
改欄位名（不影響值）
EVAL
FROM logs-*
| EVAL response_sec = duration_ms / 1000.0
| EVAL is_slow = response_sec > 1.0
計算新欄位（可多個）
EVAL with type cast
FROM logs-*
| EVAL year = TO_INTEGER(year_str)
| EVAL date = TO_DATETIME(date_str)
// 或用 ::
| EVAL year = year_str::INTEGER
:: cast 操作子
EVAL with conditional
FROM events
| EVAL severity = CASE(
    status >= 500, "critical",
    status >= 400, "warning",
    "ok"
  )
CASE 多分支條件
💡 EVAL 與型別轉換
  • 函式：TO_INTEGER()、TO_DOUBLE()、TO_DATETIME()、TO_IP()、TO_STRING()、TO_BOOLEAN()
  • shorthand：field::INTEGER、field::DATETIME
  • 條件：CASE(cond1, val1, cond2, val2, default)、COALESCE()、GREATEST()、LEAST()
完整函式列表見 CheatSheet 07。

📊 E. STATS（聚合 / GROUP BY）

Command
範例
說明
STATS 基本
FROM logs-*
| STATS count = COUNT(*),
        avg_duration = AVG(duration_ms),
        p95 = PERCENTILE(duration_ms, 95)
不分組的統計
STATS ... BY
FROM logs-*
| STATS error_count = COUNT(*) BY service, level
| SORT error_count DESC
依欄位分組（GROUP BY）
BUCKET() 時間分桶
FROM logs-*
| WHERE @timestamp > NOW() - 24 hours
| STATS count = COUNT(*) BY hour = BUCKET(@timestamp, 1 hour)
| SORT hour
取代 date_histogram。BUCKET(field, interval)
BUCKET() 數值
FROM products
| STATS count = COUNT(*) BY price_band = BUCKET(price, 100)
| SORT price_band
數值直方圖
BUCKET() auto
FROM logs-*
| STATS count = COUNT(*) BY hour = BUCKET(@timestamp, 24, "now-7d/d", "now/d")
| SORT hour
指定 bucket 數 + 範圍 → 自動算 interval
STATS 多重分組
FROM events
| STATS 
    visits   = COUNT(*),
    users    = COUNT_DISTINCT(user_id),
    revenue  = SUM(amount)
  BY country, day = BUCKET(@timestamp, 1 day)
| SORT day, revenue DESC
多 metric × 多維度
WHERE 在 STATS 中（HAVING）
FROM logs-*
| STATS errors = COUNT(*) BY service
| WHERE errors > 100
| SORT errors DESC
STATS 之後再 WHERE 等同 SQL HAVING
INLINE STATS🆕 9.2+
FROM logs-*
| INLINE STATS avg_dur = AVG(duration) BY service
| EVAL is_slow = duration > avg_dur * 2
| WHERE is_slow == true
保留原 row + 加 metric column（與每 row 對應）
⚠️ STATS 之後欄位變了：STATS 後只剩 metric 欄位 + BY 欄位，原始欄位都消失。 若需保留 row 細節 + agg，用 INLINE STATS（9.2+）或 top_hits 概念。

✂️ F. DISSECT vs GROK（解析字串）

Command
範例
說明
DISSECT（delimiter）
ROW msg = "2026-05-01 ERROR app-1 Connection refused"
| DISSECT msg "%{date} %{level} %{service} %{message}"
| KEEP date, level, service, message
依固定 delimiter 切割。比 GROK 快、易讀
DISSECT with separator
ROW path = "/api/v2/users/123/profile"
| DISSECT path "/api/%{ver}/%{resource}/%{id}/%{action}"
路徑 / 結構化日誌首選
DISSECT append
ROW name = "John Smith"
| DISSECT name "%{full_name} %{+full_name}"
+ 把多 token 接到同一欄
GROK（regex）
ROW msg = "127.0.0.1 [2026-05-01T12:00:00Z] OK"
| GROK msg """%{IP:ip} \[%{TIMESTAMP_ISO8601:ts}\] %{WORD:status}"""
| KEEP ip, ts, status
用 regex pattern。三引號 """ 避免跳脫地獄
GROK with type
ROW data = "Connections: 42"
| GROK data "Connections: %{NUMBER:count:int}"
%{PATTERN:name:type} 直接 cast

F2. DISSECT vs GROK 對照

方案
原理
建議
DISSECT
固定 delimiter pattern
🟢 快、結構化日誌首選
GROK
Regex pattern（如 %{IP}）
🟡 慢但彈性強，結構不固定時用
混用
先 DISSECT 大段，再 GROK 細節
兩者可在同一 query
💡 何時選哪個？
  • 結構固定的 log（HTTP access log、syslog）→ DISSECT
  • 結構變動 / 需要 regex 特性（如 IP 範圍、複雜字符串）→ GROK
  • 常見模式：DISSECT 切大段 → GROK 解析 message 細節

🔗 G. ENRICH（lookup 加欄位）

ENRICH 在 query 時從 enrich 索引加欄位，比 LOOKUP JOIN 更早期，但仍有效。

情境
命令 / Code
說明
1. 建 enrich policy
PUT /_enrich/policy/lang_policy
{
  "match": {
    "indices":      "languages",
    "match_field":  "lang_id",
    "enrich_fields": [ "name", "is_official" ]
  }
}
從 source index 建立 lookup 索引
2. 執行 policy
POST /_enrich/policy/lang_policy/_execute
產生 .enrich-* 索引
3. ENRICH in ES|QL
FROM developers
| ENRICH lang_policy ON language_id
| KEEP developer, name, is_official
用欄位查 enrich 索引、加欄位回 row
ENRICH with rename
FROM developers
| ENRICH lang_policy ON language_id
  WITH lang_name = name
WITH 重命名 enrich 欄位
💡 ENRICH vs LOOKUP JOIN
  • ENRICH：需先建 policy 與 .enrich-* 索引；資料變更需重 execute
  • LOOKUP JOIN：直接讀 lookup index；即時資料；不需 policy
  • 9.x 新場景 → LOOKUP JOIN；既有 ENRICH policy → 繼續用

🆕 H. LOOKUP JOIN（9.1+）

SQL-like LEFT JOIN：保留 source 全部 row，加上 lookup 索引欄位（match 不到時為 null）。

情境
命令 / Code
說明
建 lookup index
PUT /threat_list
{
  "settings": {
    "index.mode": "lookup"
  },
  "mappings": {
    "properties": {
      "source.ip": { "type": "ip" },
      "threat_level": { "type": "keyword" },
      "threat_type":  { "type": "keyword" }
    }
  }
}
"index.mode": "lookup" 必設！自動 single-shard
基本 LOOKUP JOIN🆕 9.1+
FROM firewall_logs
| LOOKUP JOIN threat_list ON source.ip
| WHERE threat_level IS NOT NULL
| SORT @timestamp DESC
| KEEP @timestamp, source.ip, action, threat_level, threat_type
Left join：保留所有 firewall_logs，加上 threat_list 的欄位
不同欄名 join
FROM employees
| RENAME department_id AS dept_id
| LOOKUP JOIN departments ON dept_id
| KEEP name, dept_id, dept_name, manager
欄名不同時先 RENAME 對齊
多欄位 join🆕 9.2+
FROM events
| LOOKUP JOIN routing_table ON region == src_region AND tier == src_tier
| KEEP event_id, target_cluster
9.2+ 支援多 predicate（tech preview）
SORT/LIMIT 後再 JOIN
FROM logs-*
| WHERE level == "ERROR"
| SORT @timestamp DESC
| LIMIT 100
| LOOKUP JOIN service_metadata ON service.name
先縮減再 join 大幅降成本

⚠️ LOOKUP JOIN 限制
  • Lookup index 必須 "index.mode": "lookup"（自動 single shard）

🧪 K. 經典 Recipes

📈 Recipe 1：過去 24h 各 service Top 10 errors

FROM logs-*
| WHERE @timestamp > NOW() - 24 hours
| WHERE level == "ERROR"
| STATS count = COUNT(*),
        sample = VALUES(message)
  BY service.name
| SORT count DESC
| LIMIT 10

⚡ Recipe 2：API 服務 P50/P95/P99 latency

FROM apm-traces-*
| WHERE @timestamp > NOW() - 1 hour
| STATS p50 = PERCENTILE(duration_ms, 50),
        p95 = PERCENTILE(duration_ms, 95),
        p99 = PERCENTILE(duration_ms, 99),
        count = COUNT(*)
  BY service.name
| WHERE count > 100
| SORT p95 DESC
| LIMIT 20

🕐 Recipe 3：每小時每 host CPU 統計

FROM metrics-*
| WHERE @timestamp > NOW() - 7 days
| STATS avg_cpu = AVG(system.cpu.usage),
        max_cpu = MAX(system.cpu.usage)
  BY hour    = BUCKET(@timestamp, 1 hour),
     host.name
| SORT hour DESC, max_cpu DESC

🔗 Recipe 4：Web log 多 lookup join 富化

FROM web-access-logs
| WHERE @timestamp > NOW() - 1 hour
| LOOKUP JOIN ip_geolocation ON client.ip
| LOOKUP JOIN user_agents ON user_agent.original
| WHERE country NOT IN ("TW", "US")
| STATS requests = COUNT(*) BY country, browser_family
| SORT requests DESC
| LIMIT 50

🤖 Recipe 5：Log pattern 自動分群（CATEGORIZE）

FROM logs-app-*
| WHERE @timestamp > NOW() - 6 hours
| WHERE level IN ("ERROR", "WARN")
| STATS count = COUNT(*),
        first_seen = MIN(@timestamp),
        last_seen  = MAX(@timestamp)
  BY pattern = CATEGORIZE(message)
| WHERE count > 5
| SORT count DESC
| LIMIT 20

🔄 L. SQL vs ES|QL 對照

SQL
ES|QL
說明
SELECT col1, col2 FROM tbl
FROM tbl | KEEP col1, col2
投影

ORDER BY ts DESC LIMIT 10
| SORT ts DESC | LIMIT 10

SELECT COUNT(*) GROUP BY x

HAVING c > 10

(function () {
  function init() {
    document.querySelectorAll('main pre').forEach(function (pre) {
      if (pre.parentElement.classList.contains('code-wrap')) return;
      const wrap = document.createElement('div');
      wrap.className = 'code-wrap';
      pre.parentNode.insertBefore(wrap, pre);
      wrap.appendChild(pre);
      const lang = pre.querySelector('code')?.className.match(/language-(\w+)/)?.[1];
      if (lang) {
        const tag = document.createElement('span');
        tag.className = 'code-lang';
        tag.textContent = lang;
        wrap.appendChild(tag);
      }
      const btn = document.createElement('button');
      btn.className = 'copy-btn';
      btn.type = 'button';
      btn.textContent = '📋 複製';
      btn.addEventListener('click', function () {
        const text = pre.innerText;
        if (navigator.clipboard?.writeText) {
          navigator.clipboard.writeText(text).then(success).catch(fallback);
        } else {
          fallback();
        }
        function success() {
          btn.textContent = '✓ 已複製';
          btn.classList.add('copied');
          setTimeout(function () { btn.textContent = '📋 複製'; btn.classList.remove('copied'); }, 1800);
        }
        function fallback() {
          const ta = document.createElement('textarea');
          ta.value = text;
          document.body.appendChild(ta);
          ta.select();
          try { document.execCommand('copy'); success(); } catch (_) {}
          document.body.removeChild(ta);
        }
      });
      wrap.appendChild(btn);
    });
  }
  if (document.readyState === 'loading') document.addEventListener('DOMContentLoaded', init);
  else init();
})();

// Auto-build TOC from h2/h3 in 
    
(function () {
  function init() {
    const toc = document.querySelector('.toc ul');
    if (!toc) return;
    const headings = document.querySelectorAll('main h2, main h3');
    headings.forEach(function (h) {
      if (!h.id) {
        h.id = h.textContent.trim().toLowerCase()
          .replace(/[^\w\u4e00-\u9fff\s-]/g, '')
          .replace(/\s+/g, '-');
      }
      const li = document.createElement('li');
      const a = document.createElement('a');
      a.href = '#' + h.id;
      a.textContent = h.textContent;
      a.className = 'toc-' + h.tagName.toLowerCase();
      li.appendChild(a);
      toc.appendChild(li);
    });
  }
  if (document.readyState === 'loading') document.addEventListener('DOMContentLoaded', init);
  else init();
})();

    if (myCounter !== renderCounter) return;
    try {
      await mermaid.run({ querySelector: '.mermaid' });
    } catch (e) { console.warn('Mermaid render error', e); }
  }

  function init() {
    renderAll();
    window.addEventListener('jhsun:theme-changed', renderAll);
  }

  if (document.readyState === 'loading') document.addEventListener('DOMContentLoaded', init);
  else init();
})();
