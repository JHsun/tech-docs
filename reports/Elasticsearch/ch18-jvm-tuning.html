圖 18-1：Elasticsearch 在一台機器上的記憶體分配
💡 為什麼 OS file cache 這麼重要？ Lucene 透過 mmap 把 segment 檔案映射到記憶體，讓 OS 自己用 file cache 加速讀取。file cache 越大、搜尋越快。這就是為什麼即使 ES heap 用得不多，整台機器仍應幾乎跑滿 RAM 的原因——剩下的 RAM 全給 OS cache。

Heap 黃金法則

  1. Heap ≤ 50% RAM：另一半留給 OS cache
  2. Heap ≤ 31 GB（compressed oops 限制）：超過後 JVM 改用 64-bit 指標，反而效能下降 20%+
  3. Xms = Xmx：避免動態 resize 造成 GC pause
  4. 不要超過單機 64 GB RAM：超過部分浪費。改用更多節點而非單機更大 heap

實例試算

機器 RAM建議 HeapOS cache 預期
16 GB8 GB~7 GB
32 GB16 GB~15 GB
64 GB30-31 GB（卡 compressed oops）~32 GB
128 GB仍 31 GB；OS cache 變 ~96 GB~96 GB
需要更多 heap跑兩個節點 each 31 GB，或拆機—

9.x 自動 Heap Sizing

從 Elastic 7.11+ 起，不寫 -Xms / -Xmx 即啟用自動計算。ES 會依：

  • 節點記憶體（從 cgroup limit 或 OS 偵測）
  • 節點 roles（純 master ≠ data hot ≠ ML 節點）

計算合適的 heap。範例（在 64 GB 機器上）：

節點角色自動 heap
純 master~ 1.6 GB（小，因為不需大 buffer）
data + ingest~ 31 GB（吃滿 50% / 31 GB 上限）
ML node較少 heap，因為 ML 用 off-heap native processes
✅ 大多數場景：不要手動設定 heap 自動計算已經對絕大多數工作負載最佳化。手動設定的時機是：
  • 容器 cgroup limit 設定不正確時，ES 偵測錯誤
  • 故意限制 heap 給其他 process（不建議在生產 ES 上做）
  • 需要刻意較小 heap 進行測試

手動設定 Heap

建立 jvm.options.d/heap.options：

-Xms16g
-Xmx16g
🚨 永遠 Xms == Xmx
  • 避免 JVM 在 resize 時暫停
  • 啟用 bootstrap.memory_lock 後若 Xms ≠ Xmx，無法 lock 全部 heap
  • 避免 OS 在 boot 時就配滿了，後續才 resize 失敗

G1 Garbage Collector（9.x 預設）

Elastic 9.x 預設使用 G1GC。G1 比舊的 CMS 在大 heap 上有以下優勢：

  • 低 pause 時間（目標 200ms 內）
  • 對 31 GB heap 仍能維持低延遲
  • 更好的記憶體 fragmentation 處理

9.x 預設 G1 GC 參數

位於 $ES_PATH_CONF/jvm.options（不要改原檔，要覆寫請放 jvm.options.d/）：

## GC configuration
14-:-XX:+UseG1GC

## Optimizations enabled by ES on JDK 14+
14-:-XX:G1ReservePercent=25
14-:-XX:InitiatingHeapOccupancyPercent=30

解釋：

  • G1ReservePercent=25：保留 25% heap 給 evacuation 失敗備用，預防 to-space exhausted 錯誤
  • InitiatingHeapOccupancyPercent=30：heap 佔用達 30% 即觸發 concurrent marking，比預設 45% 更積極
⚠️ 不建議調整 G1 參數 Elastic 已經調好了。除非你看到具體 GC 問題（pause time、to-space exhausted、concurrent mode failure），不要動 G1 參數。

ZGC / Shenandoah（實驗性，特殊場景）

對於 heap > 32 GB 的特殊需求（如 ML 節點需要大記憶體做 inference），可考慮：

  • ZGC（JDK 21+ 已 GA）：sub-millisecond pause，但有 CPU 開銷
  • Shenandoah：類似 ZGC，OpenJDK 也支援
# jvm.options.d/zgc.options（非預設，自行測試）
-XX:+UseZGC
-XX:+ZGenerational
⚠️ 不要在生產隨意切換 GC G1 是 Elastic 全力支援與測試的。ZGC 可能在 ES 上有未發現的相容性問題。先在 staging 充分驗證，並開啟 GC log 觀察。

生產級 jvm.options.d/ 範例

sudo mkdir -p /etc/elasticsearch/jvm.options.d/

# 1. Heap 設定
sudo tee /etc/elasticsearch/jvm.options.d/01-heap.options << 'EOF'
# 通常省略，讓 ES 自動計算
# -Xms16g
# -Xmx16g
EOF

# 2. OOM dump 設定
sudo tee /etc/elasticsearch/jvm.options.d/02-oom.options << 'EOF'
# 寫入 heap dump（OOM 時）
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/lib/elasticsearch/heapdumps
-XX:ErrorFile=/var/log/elasticsearch/hs_err_pid%p.log
-XX:OnOutOfMemoryError=systemctl restart elasticsearch.service
EOF

# 3. GC log（生產建議開啟）
sudo tee /etc/elasticsearch/jvm.options.d/03-gc-log.options << 'EOF'
-Xlog:gc*,gc+age=trace,safepoint:file=/var/log/elasticsearch/gc.log:utctime,pid,tags:filecount=32,filesize=64m
EOF

# 確保目錄存在且可寫
sudo mkdir -p /var/lib/elasticsearch/heapdumps
sudo chown elasticsearch:elasticsearch /var/lib/elasticsearch/heapdumps

Hot Threads（CPU 高時診斷）

curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_nodes/hot_threads?threads=10&interval=1s"

會輸出當前最忙的 threads 與其 stack trace，極適合找出哪個 query / thread pool 過載。

Thread Pool 狀態

curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_cat/thread_pool?v&h=node_name,name,active,queue,rejected"

看到 rejected 持續上升即為超載：

  • write rejected → 寫入過快，增加 buffer / 加節點
  • search rejected → 查詢過多，增 replica / 加節點

GC log 解讀

啟用 GC log 後，/var/log/elasticsearch/gc.log 會記錄每次 GC：

[2026-04-15T10:23:45.123+0000][12345][info ][gc          ] GC(123) Pause Young (Normal) (G1 Evacuation Pause) 8192M->2048M(16384M) 45.123ms
[2026-04-15T10:25:12.456+0000][12345][info ][gc          ] GC(124) Pause Young (Concurrent Start) (G1 Humongous Allocation) 12288M->3072M(16384M) 78.456ms
[2026-04-15T10:25:13.000+0000][12345][info ][gc          ] GC(125) Concurrent Cycle
[2026-04-15T10:25:14.222+0000][12345][info ][gc          ] GC(126) Pause Remark 3500M->3500M(16384M) 12.345ms

判讀重點：

類型正常警訊
Young GC pause< 100 ms> 200 ms
Old / Mixed GC pause< 500 ms> 1 s
Young GC 頻率每 30 秒一次每秒多次（heap 不夠）
Old 區佔用< 50%> 75% 持續上升 = memory leak
"to-space exhausted"無出現 = heap 嚴重不足

GC log 視覺化工具

  • GCeasy.io — 免費上傳分析（注意：勿上傳含敏感字串的 log）
  • GCViewer（開源）— 本地分析
  • 送 GC log 到 ES 自身用 Filebeat 監控（自我監控的玩法）

Heap Dump 分析

觸發 heap dump

# OOM 時自動（已在 jvm.options.d/02-oom.options 開啟）

# 手動觸發（找出 ES PID）
ES_PID=$(pgrep -f 'org.elasticsearch.bootstrap.Elasticsearch')

# 或透過 systemd
ES_PID=$(systemctl show elasticsearch -p MainPID --value)

# 用 jcmd（推薦）
sudo -u elasticsearch /usr/share/elasticsearch/jdk/bin/jcmd $ES_PID GC.heap_dump /tmp/heap.hprof

# 或 jmap（傳統）
sudo -u elasticsearch /usr/share/elasticsearch/jdk/bin/jmap -dump:format=b,file=/tmp/heap.hprof $ES_PID

分析工具

  • Eclipse Memory Analyzer (MAT)：圖形化、強大、免費
  • VisualVM：JDK bundled，輕量
  • jhat（已棄用）：命令列工具
  • YourKit / JProfiler：商業，互動式

常見 JVM 問題排查

問題 1：OutOfMemoryError

常見原因：

  • 單一 query 聚合資料量過大（aggregation cardinality 太高）
  • Field data cache / request cache 過大
  • 過多 indices 或 shards 導致 cluster state 過大

處理：

# 1. 查看 OOM 的 stack trace
sudo grep -A 50 "OutOfMemoryError" /var/log/elasticsearch/*.log

# 2. 分析 heap dump（位於 /var/lib/elasticsearch/heapdumps）
ls /var/lib/elasticsearch/heapdumps

# 3. 透過 cat API 看 shard 數
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_cat/allocation?v"
# disk_indices 與 shards 都應在合理範圍

# 4. 檢查 cluster state 大小
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_cluster/state?filter_path=metadata.indices&pretty" | wc -c
# 應 < 100 MB；過大代表 indices 太多

問題 2：長 GC pause（節點不穩）

# GC log 中找有問題的 entry
grep -E "Pause.*[5-9][0-9][0-9]ms|Pause.*[1-9][0-9]{3}ms" /var/log/elasticsearch/gc.log

常見原因：

  • Heap 設太大（> 31 GB），用了 non-compressed oops
  • 機器其他 process 搶 CPU，GC thread 跑不滿
  • 啟用 swap 或 swappiness 高（GC 時 page in/out 慢）
  • Memory leak（old gen 持續上升）

問題 3：Circuit breaker tripped

典型錯誤訊息：

CircuitBreakingException[[parent] Data too large, data for [] would be 
[123456789/117.7mb], which is larger than the limit of [12345/12kb]]

說明 ES 主動拒絕了一個會吃太多記憶體的請求，這是好事（保護節點不 OOM）。處理方式：

  • 找出問題請求並優化（減少聚合、使用 composite aggregation）
  • 用 search-template 或 query DSL 加 size、aggs 限制
  • 升級節點規格
  • 絕對不要透過調高 limit 來「解決」

容器環境的 JVM 注意事項

在 Docker / Kubernetes 中跑 ES：

  1. JVM 會偵測 cgroup memory limit，依此計算 heap（JEP 282 起，所有 JDK 都支援）
  2. 但必須給容器設定 memory limit，否則 JVM 看到的是 host 全部 RAM
  3. 建議：container memory = 2 × heap，給 OS cache 留空間
  4. K8s 中還要設 requests.memory == limits.memory 才能進入 Guaranteed QoS
# Kubernetes pod 範例
resources:
  requests:
    memory: 32Gi
    cpu: 4
  limits:
    memory: 32Gi    # 與 requests 一致
    cpu: 8
env:
  - name: ES_JAVA_OPTS
    value: "-Xms16g -Xmx16g"   # heap = 容器 50%

JVM 調校 Checklist

  • ☐ Heap ≤ 50% RAM 且 ≤ 31 GB
  • ☐ Xms == Xmx（手動時）
  • ☐ 使用 bundled JDK（不要自己安裝）
  • ☐ bootstrap.memory_lock: true 已啟用
  • ☐ HeapDumpOnOutOfMemoryError 已啟用，dump 路徑可寫
  • ☐ GC log 已啟用且有輪替
  • ☐ 監控 heap_used_percent（超過 75% 持續就要警報）
  • ☐ 監控 thread pool rejected 數
  • ☐ 不要安裝任何 javaagent
  • ☐ 容器環境：memory limit 已設，且為 heap × 2

本章重點回顧

  • ES 記憶體 = JVM heap + Off-heap mmap + OS file cache，三者缺一不可
  • Heap 黃金法則：≤ 50% RAM，≤ 31 GB，Xms == Xmx
  • 9.x 預設自動 heap sizing，絕大多數情況不要手動設
  • 9.x 預設 G1 GC，已調好。不要動 G1 參數
  • 所有 JVM 覆寫一律放 jvm.options.d/
  • 生產建議開啟 OOM heap dump、GC log（含輪替）
  • 診斷三大利器：_nodes/stats/jvm、_nodes/hot_threads、_cat/thread_pool
  • Circuit breaker 觸發是保護機制，不是問題本身—去查問題請求
