Logs：事件記錄

Kibana → Observability → Logs（已併入新 Streams 介面）：

  • 即時 tail（類 tail -f）
  • 結構化欄位過濾
  • Log rate analysis：自動偵測 rate 偏移與貢獻欄位
  • Log categorization（ML）：自動把 log 訊息分類，找出 rare patterns
  • Log threshold alert：rate 超閾值告警
  • Log correlation：與 trace.id、service.name 串連

Log Rate Analysis（ML 驅動）

選定基準時間（baseline）vs 偵測時間（deviation），自動找出：

  • 哪些欄位值在偵測期顯著增加
  • 排序貢獻度
  • 下鑽到原始 log 看實例

例：error rate 突然升高 → 看是哪個 service.name + endpoint 貢獻最多。

Log Categorization

用 ML 把類似 log 訊息歸類，把幾百萬 log 行壓縮成幾百個 category，找出：

  • 新出現的 log type
  • 頻率異常的 category

9.x Streams：統一 data stream 管理

9.2+ 推出新 Streams 介面（取代舊 Logs UI），統一管理 data stream：

  • 視覺化所有 data stream（含 logs、metrics、traces）
  • 每 stream 看 ingest rate、storage、retention
  • 直接編輯 ILM、ingest pipeline、index template
  • 整合 Discover、Lens 視覺化
  • 取代分散在 Stack Management 各處的設定

Metrics

Hosts View

所有 host 的 CPU / RAM / disk / network 即時視圖。可下鑽到單一 host 看 process、Docker container、K8s pod。

Inventory：基礎設施視角

9.x 提供 Inventory 統一視圖：

  • Hosts：實體 / VM
  • Containers：Docker / containerd
  • Kubernetes：clusters / namespaces / pods / nodes
  • Services：APM 服務
  • AWS / GCP / Azure：雲端 resources

每實體可一鍵跳到：相關 alert、相關 logs、相關 trace、相關 dashboard。

雲端 Integration

內建 integrations 收集：

  • AWS：CloudWatch metrics、VPC flow logs、Lambda、ECS、EKS
  • GCP：Cloud Monitoring、GKE、Cloud Functions
  • Azure：Monitor、AKS、Functions

9.x 也支援透過 Amazon Data Firehose 整合，與 EDOT collector 統一接收。

Traces（複習 CH 56）

  • Service Map、Service Overview
  • Transaction、Span 詳情
  • Distributed tracing 跨服務
  • Tail-based sampling

Synthetics：主動探測

不像 APM 是被動接 SDK，Synthetics 主動模擬使用者：

兩種 monitor

類型說明適用
Lightweight monitorHTTP / TCP / ICMP pingendpoint 可用性、SSL 過期、快速 health check
Browser monitorPlaywright 腳本，模擬 real user 行為登入流程、結帳流程、UI 互動驗證

Lightweight Monitor 範例

# 在 Fleet → Add Synthetics integration → HTTP monitor
schedule: "@every 30s"
url: "https://example.com/api/health"
check.response.status: [ 200 ]
check.response.body: 'ok'
ssl.supported_protocols: [ "TLSv1.2", "TLSv1.3" ]
ssl.certificate_authorities: [ "/path/to/ca.crt" ]

Browser Monitor 範例

// 用 @elastic/synthetics 寫 Playwright-like script
import { journey, step, expect } from '@elastic/synthetics';

journey('Login flow', ({ page }) => {
  step('go to login page', async () => {
    await page.goto('https://example.com/login');
  });

  step('fill credentials', async () => {
    await page.fill('input[name=email]',    'test@example.com');
    await page.fill('input[name=password]', process.env.TEST_PWD);
    await page.click('button[type=submit]');
  });

  step('see dashboard', async () => {
    await expect(page.locator('h1.dashboard')).toBeVisible({ timeout: 5000 });
  });
});

每執行一次會記錄：

  • 每個 step 成功 / 失敗 / 耗時
  • 截圖（含 step 之間）
  • console error
  • 網路請求 timeline

執行位置（Locations）

  • Elastic-managed：Elastic 雲端多地理區（北美、歐、亞）
  • Private locations：自架 Elastic Agent + Synthetics integration，跑在內網（測 internal services）

SLO（Service Level Objectives）

9.x 提供原生 SLO 管理。建立 SLO 後自動：

  • 計算 error budget
  • burn rate 計算
  • 多 window 告警（fast burn、slow burn）

SLO 範例

Observability → SLOs → Create SLO：

  • Indicator：用 KQL 或 ES|QL 定義「good event」與「total event」
  • Objective：99.9%（目標達成率）
  • Time window：rolling 30 day / calendar aligned
  • Budgeting method：occurrences 或 timeslices

如：

  • Good：http.response.status_code < 500 AND transaction.duration.us < 1000000
  • Total：service.name : "checkout" AND transaction.type : "request"
  • Objective：99.5% in 30 days

Burn Rate Alert

SLO 內建 burn rate alert：當「按目前速度燒，error budget 多久會燒完」超過閾值時告警。

典型 multi-window burn rate（Google SRE book）：

  • Fast burn：5min 與 1hr 同時 burn rate > 14.4× → critical
  • Slow burn：6hr 與 3day 同時 burn rate > 1× → warning

Universal Profiling（9.x 強化）

用 eBPF kernel-level profiler，always-on，不需 instrumentation：

  • Java、Go、Python、.NET、Ruby、Native（C/C++/Rust）
  • Flame graph 視覺化（CPU 熱點）
  • 跨服務、跨語言對比
  • 整合進 APM service overview
  • 9.3+ 整合 service-level cost analysis（用 CPU 估換算成本）

部署：在 host 安裝 universal profiling agent（含在 Elastic Agent integration 中）。

AI Assistant for Observability

9.x 整合 LLM 進 Observability：

  • 自然語言查詢：「過去一小時哪個 service error 最多？」自動轉 ES|QL 查詢並執行
  • 根因分析建議：對 anomaly 提供可能原因 + 證據
  • Runbook 整合：定義 runbook，AI 可參考並執行步驟
  • Custom knowledge sources：可附加 RAG 知識庫（公司 wiki、過往 incident）
  • Connector：OpenAI、Anthropic、Azure OpenAI、Google AI Studio、Bedrock

需在 Stack Management → Connectors 設 LLM connector。

Alerts 與 Cases

Alert

Observability 中的 alert（CH 45）：

  • Log threshold（log rate）
  • Metric threshold（CPU、RAM、custom）
  • Inventory（依實體）
  • Anomaly（ML 驅動）
  • SLO burn rate
  • Uptime monitor 失敗
  • APM transaction 錯誤率 / latency 退化

Cases

把 alert / 觀察轉為 case，做事件追蹤：

  • 多 alert 合併到一個 case
  • 指派人員、status、tags
  • 整合 Slack、Jira、ServiceNow（雙向同步）
  • 事後做 post-mortem

OpenTelemetry 整合（預告 CH 64）

9.x 力推 OTel 為 vendor-neutral 標準：

  • OTLP endpoint 內建在 ES（接收 OTLP gRPC/HTTP）
  • EDOT Collector：Elastic 發行版的 OTel Collector，含 Elastic-specific processors
  • EDOT SDK：各語言的 Elastic 預配置 OTel SDK
  • Managed OTLP：Elastic Cloud 提供 OTLP 端點，OTel 直送

新部署優先 OTel，CH 64 完整介紹。

9.x Observability 路線圖

  • Streams 介面：取代分散的 Logs / Metrics / Traces UI
  • AI Assistant 強化：Agent Builder（9.x 新）做 agentic workflow
  • OTLP 全面 first-class：APM Server 漸退場
  • Universal Profiling 與 service map 整合
  • Cost analysis：profiling 數據對應實際 cloud spend

最佳實踐

  • 三大支柱齊備：logs、metrics、traces 缺一不可
  • service.name / environment 命名一致 跨支柱串連
  • 定義 SLO 取代靜態閾值：alert 更精準
  • Burn rate multi-window：fast + slow 都要設
  • Synthetics 從關鍵 user journey 開始：登入、結帳、付款
  • Browser monitor 用 page object pattern，可維護
  • Universal Profiling 全部開，找隱藏熱點
  • AI Assistant 用 RAG 加自家 runbook，比通用 LLM 更準
  • 9.x 新部署優先 OTel，逐步取代 Beats / APM agent
  • Cases 整合 Jira，post-mortem 流程閉環

本章重點回顧

  • Observability 五大支柱：Logs、Metrics、Traces、Synthetics、Profiling

  • 9.2+ Streams 介面統一 data stream 管理
  • Synthetics：lightweight ping + browser-based Playwright
  • SLO + burn rate alert 取代靜態閾值
  • Universal Profiling：eBPF always-on，跨語言 flame graph
  • AI Assistant for Observability：自然語言查詢 + 根因建議
  • 9.x 全力擁抱 OTel，APM Server 漸退場
