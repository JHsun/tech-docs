CH 05 · 讀寫流程：Refresh / Flush / Merge / Translog

適合對象 🟡 開發者 🔴 維運者
閱讀時間 約 30 分鐘
前置知識 CH 02 · 核心概念（Shard、Segment）、CH 03-04 · 分散式架構
本章目標 完整理解 ES 寫入路徑（buffer → translog → segment）、refresh/flush/merge 三大背景任務、讀取流程的 query 與 fetch 兩階段、NRT 行為與調校方向

為什麼要懂讀寫流程？

很多 Elasticsearch 的「奇怪行為」其實來自其讀寫機制：

  • 「為什麼剛 index 完的資料用 GET 找得到，用 search 卻找不到？」
  • 「為什麼批次寫入後磁碟 I/O 持續高？」
  • 「為什麼明明只刪了一筆，磁碟空間沒減少？」
  • 「為什麼 refresh_interval 調到 30 秒寫入快這麼多？」

本章將完整剖析這些問題的根源。

步驟詳解

① 路由（Routing）

Coordinating node 用以下公式決定 primary shard 編號：

shard_num = hash(_routing) % number_of_primary_shards

_routing 預設等於 _id。也可在 API 中指定自訂 routing key（例如所有同一 user_id 的資料路由到同一 shard，加速查詢）。

② 寫入 In-memory Indexing Buffer

新文件先寫入 JVM heap 內的 indexing buffer（每個 shard 各有一個），這個階段資料還無法被搜尋。

③ 同步寫入 Translog

同時，文件被 append 到 translog（transaction log）檔案，這是磁碟上的「寫入前日誌」（WAL，Write-Ahead Log）。Translog 的作用：在節點突然 crash 時，重啟後可從 translog replay 尚未 flush 的資料，避免遺失。

Translog 預設行為：

# 每筆 request 都 fsync 到磁碟（預設）— 安全但寫入較慢
index.translog.durability: request

# 改為非同步，每 5 秒 fsync 一次 — 快但 crash 時可能丟最多 5 秒資料
index.translog.durability: async
index.translog.sync_interval: 5s

# Translog 達到此大小會觸發 flush（預設 512mb）
index.translog.flush_threshold_size: 512mb

④ 同步到 Replicas

Primary 寫入成功後，並行 forward 給所有 in-sync replicas，replicas 同樣寫入自己的 buffer 與 translog。

⑤ 回應 Client

當 primary 與所需數量的 replicas 全部寫入成功，才回應 client。wait_for_active_shards 控制這個門檻：

  • 1（預設）：只要 primary OK 就回應
  • all：等所有 replica 都 OK
  • 數字 N：等 N 個 active shards

Refresh：讓資料可被搜尋

注意上面的流程結束後，資料還在 buffer，無法被 search 看到。要讓資料可搜尋，必須執行 refresh：

圖 5-2：Refresh 動作 — 讓 buffer 中的資料變得可被搜尋

關鍵點：

  • Refresh 的 segment 還在 OS file system cache，尚未真正 fsync 到磁碟
  • 但只要在 cache 中就可以被 Lucene 開啟並搜尋
  • 這就是 ES 「Near Real-Time」的來源 — 預設 1 秒延遲

refresh_interval

預設每個 shard 每 1 秒自動 refresh：

# 查詢當前設定
curl -X GET "https://localhost:9200/my-index/_settings?pretty" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt

# 改為 30 秒 — 大幅提升寫入吞吐
curl -X PUT "https://localhost:9200/my-index/_settings" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/json' \
  -d '{ "index": { "refresh_interval": "30s" } }'

# 完全關閉自動 refresh（用於 bulk import）
curl -X PUT "https://localhost:9200/my-index/_settings" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/json' \
  -d '{ "index": { "refresh_interval": "-1" } }'
💡 智能 refresh：搜尋前才 refresh 從 7.x 起，若一個 shard 在 index.search.idle.after（預設 30s）內沒有任何 search 請求，會進入 idle 狀態並暫停自動 refresh。下次有 search 請求時才會 refresh。這對於「寫多、搜少」的索引（例如 logs）能省下大量 I/O。

手動觸發 refresh

# refresh 整個 index
curl -X POST "https://localhost:9200/my-index/_refresh" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt

# 在 index 請求中強制 refresh（會大幅拖慢寫入，僅測試用）
curl -X POST "https://localhost:9200/my-index/_doc?refresh=wait_for" \
  ...

refresh=true：立即 refresh（最慢、最即時）；refresh=wait_for：等下次 refresh 後才返回（推薦的「我需要立刻能查到」做法）；refresh=false（預設）：不影響 refresh 排程。

Flush：持久化到磁碟

Refresh 只是讓 segment 可搜尋，並未把資料安全地寫到磁碟。Flush 才是真正的持久化：

觸發時機（任一條件達成即觸發）：

  • Translog 大小超過 index.translog.flush_threshold_size（預設 512MB）
  • Translog 中最舊的操作距今超過 index.translog.flush_threshold_age（預設 30 分鐘，已棄用，改由 size 控制）
  • shard 關閉時（節點正常 shutdown）
  • 手動呼叫 POST /index/_flush
# 手動 flush
curl -X POST "https://localhost:9200/my-index/_flush" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt
⚠️ 一般情況不需手動 flush Flush 是 I/O 密集操作。讓 ES 自動依 translog 大小觸發即可。手動 flush 主要用於：升級前、重要資料寫完後、做 snapshot 前。

Merge：段合併

每秒一次 refresh 會產生大量小 segments。Segments 過多會造成：

  • 搜尋變慢（要查詢所有 segments）
  • file descriptor 耗盡
  • 記憶體用量增加
  • 已刪除文件無法清除（每個 segment 維護自己的 deletion bitmap）

因此 Lucene 在背景持續執行 merge：把多個小 segments 合併成大 segment，順便丟掉被標記為 deleted 的文件。

Merge Policy 設定

# 同時最多執行的 merge 數量（預設 max(1, min(4, processors / 2))）
index.merge.scheduler.max_thread_count: 1   # SSD 可設較高，HDD 設 1

# 每個 segment 的最大目標大小（預設 5gb）
index.merge.policy.max_merged_segment: 5gb

# 同一階層中允許的 segments 數（預設 10）
index.merge.policy.segments_per_tier: 10

# 最大允許 merged segments 包含已刪除文件的比例（預設 0.33）
index.merge.policy.deletes_pct_allowed: 33

Force Merge（強制合併）

對於不再寫入的索引（例如 ILM 已 rollover 過的舊 index、唯讀的歷史資料），可手動 force merge 把它合併成單一大 segment：

curl -X POST "https://localhost:9200/logs-2025.01/_forcemerge?max_num_segments=1" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt
🚨 永遠不要 force merge 還在寫入的 index！ 在還會寫入的 index 上 force merge 會建立巨大 segment（可能 100GB+），未來新寫入造成的 deletes 永遠卡在這個大 segment 裡無法清除（因為超過 max_merged_segment 不會被自動合併），最終導致大量浪費磁碟空間。只在 read-only 的 index 上 force merge。

Real-time GET vs NRT Search

細心的讀者會發現一個差異：

  • 用 GET /index/_doc/{id}：立即可取得（連 buffer 中的最新版本都看得到）
  • 用 POST /index/_search：需等待下一次 refresh

原因：GET 直接走 translog 與 buffer 查詢，而 search 必須走已開啟為可搜尋的 segment。

這個特性讓 ES 在「寫完立刻取」的場景表現很好（例如 user profile update + 立刻顯示），但「寫完立刻搜尋」就有 1 秒延遲。

刪除與 Update 的真相

Delete

當你刪除一筆文件，ES 並沒有真的從 segment 中抹掉，而是：

  1. 在 segment 的 deletion bitmap 標記該文件 ID 為 deleted
  2. 後續搜尋會用 bitmap 過濾掉
  3. 磁碟空間要等到該 segment 被 merge 時才真正釋放

這也解釋了「為什麼大量刪除後磁碟空間沒立即減少」。可用以下方式加速：

# 觀察某 index 的 deleted docs 數量
curl -X GET "https://localhost:9200/_cat/indices/my-index?v&h=index,docs.count,docs.deleted,store.size" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt

輸出範例：

index     docs.count  docs.deleted  store.size
my-index  1000000     320000        12.5gb

若 deleted 比例過高（如上 32%），可考慮 reindex 或對 read-only index 執行 force merge with only_expunge_deletes=true。

讀取流程：Search 的 Query / Fetch 兩階段

Search 是分散式操作，因為資料分散在多個 shards。流程如下：

    C->>Co: search request

為什麼要分兩階段？

假設你要 from=0, size=10。如果合併階段就傳 _source：

  • 10 個 shards × 10 個 docs = 100 個完整文件透過網路
  • 但最後只用 10 個 — 90% 的傳輸是浪費

分兩階段，第一階段只傳 doc ID + score，第二階段才取真正需要的 10 個文件，網路傳輸量降到最低。

深分頁問題（Deep Pagination）

注意 top-K = from + size。當你查 from=10000, size=10 時：

  • 每個 shard 必須排序前 10010 筆並回傳 ID + score
  • 10 個 shards × 10010 = 100,100 筆排序資料在 coordinating 合併
  • 記憶體與 CPU 消耗暴增

因此 ES 預設限制 from + size ≤ 10000。深分頁的正確解法：

  • search_after：用前一頁最後一筆的 sort 值往下查（推薦）
  • scroll API（已棄用，僅向後相容）
  • Point in Time (PIT) + search_after：跨多次查詢保持一致快照
# PIT 範例
curl -X POST "https://localhost:9200/my-index/_pit?keep_alive=1m" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt
# 取得 pit_id 後，後續查詢帶上 pit + search_after

Bulk 寫入優化

用 _bulk API 一次送多筆操作可以大幅提升效能（每個 HTTP request 的固定開銷被多筆攤平）：

curl -X POST "https://localhost:9200/_bulk" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/x-ndjson' \
  --data-binary @bulk_data.ndjson

bulk_data.ndjson 格式（每筆兩行：action + document）：

{"index": {"_index": "my-index", "_id": "1"}}
{"title": "Doc 1", "content": "..."}
{"index": {"_index": "my-index", "_id": "2"}}
{"title": "Doc 2", "content": "..."}
{"update": {"_index": "my-index", "_id": "3"}}
{"doc": {"views": 100}}
{"delete": {"_index": "my-index", "_id": "4"}}

Bulk 大小調校

  • 每個 bulk request 建議 5-15 MB（不是筆數，是位元組大小）
  • 太小：HTTP overhead 占比高
  • 太大：佔用 coordinating node 的記憶體；遇到 http.max_content_length 限制（預設 100MB）
  • 並行 bulk threads 數：通常等於資料節點數的 2-3 倍

高吞吐寫入的完整配方

當你需要進行大批量資料匯入（reindex、初始載入），遵循以下步驟可以最大化寫入速度：

# 1. 暫時關閉 refresh（停止產生小 segments）
curl -X PUT "https://localhost:9200/my-index/_settings" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/json' \
  -d '{ "index": { "refresh_interval": "-1", "number_of_replicas": 0 } }'

# 2. 用 bulk 並行寫入（多 thread / 多 client）

# 3. 寫完後恢復設定
curl -X PUT "https://localhost:9200/my-index/_settings" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/json' \
  -d '{ "index": { "refresh_interval": "1s", "number_of_replicas": 1 } }'

# 4. force merge（如果這份 index 之後不會大量寫入）
curl -X POST "https://localhost:9200/my-index/_forcemerge?max_num_segments=5" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt

實測能比預設設定快 3-10 倍。完整的調校細節見 CH 48。

本章重點回顧

  • 寫入路徑：buffer → translog（同步）→ replicas → 回應 → refresh 後可搜尋
  • Refresh（預設 1s）：buffer → in-memory segment，讓資料可被搜尋
  • Flush（依 translog 大小觸發）：fsync segments、清空 translog，真正持久化
  • Merge：背景合併小 segments、清除 deleted documents
  • Real-time GET 立即可見；Search 有 ~1 秒 NRT 延遲
  • Delete 與 Update 都是「邏輯刪 + 重新索引」，磁碟空間靠 merge 才真正回收
  • Search 採 query/fetch 兩階段；深分頁應改用 search_after 或 PIT
  • Bulk 寫入是高吞吐關鍵；批次匯入時可暫時關閉 refresh 與 replicas
  • 觀察 thread pool 的 rejected 來定位寫入瓶頸
