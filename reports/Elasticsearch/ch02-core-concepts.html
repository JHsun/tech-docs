Document（文件）

Document 是 Elasticsearch 的最基本資料單位，它是一個 JSON 物件。可以把它想成關聯式資料庫的「一列（row）」，但比 row 更靈活：

{
  "@timestamp": "2026-05-01T08:30:00Z",
  "user": {
    "name": "Alice",
    "email": "alice@example.com",
    "age": 30
  },
  "event": {
    "action": "login",
    "ip": "203.0.113.42",
    "success": true
  },
  "tags": ["auth", "production"]
}

每個 document 都有：

  • _id：唯一識別碼，可由你指定或由 ES 自動產生（22 字元的 Base64 字串）
  • _index：所屬的 index 名稱
  • _source：原始的 JSON 內容（預設保留，可關閉但不建議）
  • _version：版本號，每次更新會自增（用於樂觀並行控制）

Document vs RDBMS Row 的關鍵差異

特性RDBMS RowES Document
結構固定 schemaJSON，可有巢狀（nested）結構
欄位變化需 ALTER TABLE動態新增（dynamic mapping）
多值欄位需另開 table 或用陣列型別原生支援（"tags": ["a","b","c"]）
更新方式UPDATE 部分欄位實際是「重新索引整個 document」
JOIN原生支援不建議使用，常以反正規化（denormalization）替代
⚠️ 重要觀念：Document 是不可變的（immutable） Elasticsearch 的「更新」實際上是：標記舊文件為 deleted → 索引一份新文件 → 後續 segment merge 時才真正清掉舊資料。所以高頻 update 對效能影響很大，這也是為什麼 ES 不適合當作 OLTP 主資料庫。

Index（索引）

Index 是具有相似結構的 documents 的集合，類似 RDBMS 的 table，但概念更廣。例如：

  • users — 使用者資料
  • logs-app-2026.05 — 2026 年 5 月的應用日誌
  • products — 商品目錄

Index 名稱有以下限制：

  • 必須全部小寫
  • 不可包含 \ / * ? " < > | （空白）, #
  • 不可以 - _ + 開頭
  • 不可為 . 或 ..
  • 位元組長度需 ≤ 255

建立一個 index 的最簡命令：

curl -X PUT "https://localhost:9200/my-index" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/json'

也可以在建立時指定 settings 與 mappings：

curl -X PUT "https://localhost:9200/my-index" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/json' \
  -d '{
    "settings": {
      "number_of_shards": 3,
      "number_of_replicas": 1
    },
    "mappings": {
      "properties": {
        "title":     { "type": "text" },
        "tags":      { "type": "keyword" },
        "published": { "type": "date" },
        "views":     { "type": "long" }
      }
    }
  }'

Index vs Data Stream

在 8.x/9.x 中，對於時序資料（time-series data）（例如 logs、metrics），建議使用 Data Stream 而非直接操作 index。Data Stream 在底層會自動建立、輪替（rollover）多個隱藏的支援索引（backing indices），詳見 CH 30。

Mapping（欄位映射）

Mapping 定義了 index 中每個欄位的資料型別與分析方式，相當於 schema。常見型別：

類別型別用途
核心字串text / keywordtext 用於全文搜尋；keyword 用於精確比對、聚合、排序
數值long, integer, short, byte, double, float, half_float, scaled_float, unsigned_long不同精度與儲存大小
日期date, date_nanos支援多種格式
布林booleantrue/false
陣列 / 物件object, nested, flattened巢狀結構
地理geo_point, geo_shape位置與多邊形
向量dense_vector, sparse_vectorkNN 與語義搜尋
特殊ip, completion, percolator, joinIP、自動完成、反向查詢、父子關係
💡 text vs keyword 的關鍵差異
  • text：會經過 analyzer 拆分為 tokens（例如 "Hello World" → ["hello", "world"]），用於全文搜尋。不可用於排序或聚合。
  • keyword：保留原始字串完整不切分，用於精確比對、aggregations、sort。例如 status code、tag、enum 值。
  • 大多數字串欄位會同時建立兩種：name（text）+ name.keyword（keyword 多重欄位）— 這是動態 mapping 的預設行為。

倒排索引（Inverted Index）

Elasticsearch 之所以能做到極快的全文搜尋，靠的是 Lucene 的倒排索引（inverted index）結構。原理如下：

假設有 3 個 documents：

Doc 1: "The quick brown fox"
Doc 2: "Quick brown rabbit"
Doc 3: "The lazy dog"

經過分析（小寫化、去停用字、tokenize）後，產生倒排索引：

Term         Documents (postings)
--------    --------------------------
brown        [1, 2]
dog          [3]
fox          [1]
lazy         [3]
quick        [1, 2]
rabbit       [2]
the          [1, 3]

當查詢 "quick brown" 時，ES 只需在 term 字典中找出 quick → [1,2] 和 brown → [1,2]，取交集即得 [1, 2]。完全不需要掃描所有 document，這就是「索引」二字的精髓。

除了 postings，倒排索引還儲存：

  • Term frequency (TF)：詞在某 document 中出現的次數
  • Document frequency (DF)：詞在整個 index 中出現於多少個 documents
  • Position：詞在 document 中的位置（用於 phrase query）
  • Offsets：詞在原始文字中的字元起訖（用於 highlight）

這些資訊讓 Elasticsearch 能用 BM25（預設）或 TF-IDF 等演算法計算相關性分數（_score）。

Shard（分片）

當 index 變大，單一機器無法容納或效能不足時，必須水平切割。Shard 就是 index 的水平切片。

每個 shard 本質上是一個獨立、完整的 Lucene index。一個 ES index 由 1 到多個 primary shards 組成：

Document 如何被分配到某個 Shard？

ES 使用以下公式決定一筆 document 屬於哪個 shard：

shard_num = hash(_routing) % number_of_primary_shards

預設 _routing 就是 _id，因此一旦 index 建立，primary shard 數量無法直接變更（因為改了會讓 hash 結果失效，所有資料都得重新路由）。如要變更，必須透過 _reindex API 把資料搬到新 index。

如何決定 Shard 數量？

這是 ES 設計的關鍵抉擇。Elastic 官方建議的「黃金法則」：

📐 Shard sizing 黃金法則（Elastic 9.x）
  • 單一 shard 大小目標：10 GB ~ 50 GB
  • 每 GB JVM heap 不要超過 20 個 shards（總 shards 數 ÷ 全叢集 heap GB ≤ 20）
  • 避免過小的 shard（< 1 GB），會浪費資源並拖累 metadata 操作
  • 避免過大的 shard（> 100 GB），會延長恢復時間、增加 GC 壓力

例如：預估 1 年內 logs index 會成長到 600 GB，若目標單 shard 30 GB，則 primary shards = 600 / 30 = 20。詳細規劃見 CH 50。

Replica（副本）

Replica 是 primary shard 的完整副本，提供兩個關鍵能力：

  1. 高可用（High Availability）：若某 primary shard 所在的 node 掛了，replica 會被提升為新的 primary，叢集不停擺
  2. 分散讀取負載：搜尋請求可以路由到 primary 或任一 replica，提升查詢吞吐量

💡 Replica 數量可隨時調整 與 primary 不同，replica 數量可以動態變更（無需 reindex）。例如： 
curl -X PUT "https://localhost:9200/my-index/_settings" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/json' \
  -d '{ "index": { "number_of_replicas": 2 } }'

Replica 的代價

  • 儲存空間：replicas=N 表示總儲存 = 原始 × (N+1)
  • 寫入效能：每筆寫入需同步到所有 replicas，replicas 越多寫越慢
  • 記憶體：每個 shard（含 replica）都會占用 file descriptor、heap、檔案快取

生產環境通常設 number_of_replicas: 1，重要資料可以設 2。Dev/test 環境若只有單節點，必須設為 0（否則 replica 永遠分配不到，叢集為 yellow 狀態）。

Segment（段）

每個 shard（即一個 Lucene index）內部由多個 segments 組成。Segment 是磁碟上實際儲存倒排索引的不可變（immutable）檔案。

當有新 document 寫入時：

  1. 先寫入 in-memory buffer + translog
  2. 每隔 refresh_interval（預設 1 秒）將 buffer 寫成新的 segment
  3. 新 segment 立刻可被搜尋（這就是 NRT — near real-time 的來源）
  4. 背景定期執行 merge，把多個小 segments 合併為大 segment，順便清除被標記為 deleted 的 documents

這個機制的詳細流程在 CH 05 會深入講解。

Cluster（叢集）與 Cluster State

多個 ES 節點（nodes）組成一個 cluster。Cluster 內部維護一份「cluster state」資料結構，記錄：

  • 所有 indices 的 settings 與 mappings
  • 每個 shard 在哪個 node 上
  • 每個 node 的角色與健康狀態
  • 所有 templates、ILM policies、snapshots metadata 等

Cluster state 由 master node 統一管理並廣播給所有其他節點，這部分詳見 CH 03 與 CH 04。

叢集健康狀態（Cluster Health）

用一個簡單命令查看：

curl -X GET "https://localhost:9200/_cluster/health?pretty" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt

三種狀態的意義：

狀態意義處理建議
GREEN所有 primary 與 replica 都正常分配正常運行
YELLOW所有 primary 正常，但有 replica 未分配（可能是節點不夠或剛啟動中）檢查節點數、磁碟空間、replica 設定
RED有 primary shard 未分配 — 該 shard 的資料無法搜尋與寫入立即排查，可能需從 snapshot 還原

🚫 不要把 ES 當 RDBMS 用 雖然概念有對照，但 ES 不是 RDBMS。它不支援 transaction、不適合高頻 update、不適合用 JOIN。最佳實踐是把 ES 當成「派生（derived）資料儲存」：用其他系統（PostgreSQL、Kafka 等）作為 source of truth，再把資料同步進 ES 做搜尋與分析。

一個完整範例：建立 index 並索引文件

以下示範完整流程，假設 ES 已在 https://localhost:9200 運行（CH 07 會說明如何安裝）：

Step 1：建立 index 與 mapping

curl -X PUT "https://localhost:9200/blog" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/json' \
  -d '{
    "settings": {
      "number_of_shards": 1,
      "number_of_replicas": 0
    },
    "mappings": {
      "properties": {
        "title":      { "type": "text" },
        "author":     { "type": "keyword" },
        "tags":       { "type": "keyword" },
        "published":  { "type": "date" },
        "views":      { "type": "long" },
        "content":    { "type": "text" }
      }
    }
  }'

Step 2：索引（寫入）一筆文件

curl -X POST "https://localhost:9200/blog/_doc/1" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/json' \
  -d '{
    "title": "Elasticsearch 入門",
    "author": "Alice",
    "tags": ["elasticsearch", "search", "tutorial"],
    "published": "2026-05-01",
    "views": 0,
    "content": "Elasticsearch 是分散式搜尋引擎..."
  }'

Step 3：搜尋

curl -X GET "https://localhost:9200/blog/_search" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/json' \
  -d '{
    "query": {
      "match": { "title": "Elasticsearch" }
    }
  }'

Step 4：直接取單一文件

curl -X GET "https://localhost:9200/blog/_doc/1?pretty" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt

Step 5：檢視 shard 分布

curl -X GET "https://localhost:9200/_cat/shards/blog?v" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt

輸出範例：

index shard prirep state   docs store ip          node
blog  0     p      STARTED   1  4.5kb 172.17.0.2 node-1

本章重點回顧

  • Document 是基本資料單位（JSON），帶有 _id、_index、_source、_version
  • Index 是相似 documents 的集合，透過 mapping 定義欄位結構
  • Mapping 中最常見的字串型別有 text（全文搜尋）與 keyword（精確比對 / 聚合）
  • 倒排索引是 ES 高速搜尋的關鍵，由 term → postings 構成
  • Shard 是 index 的水平切片，每個 shard 是獨立的 Lucene index；primary shard 數一旦設定不可變
  • Replica 提供高可用與讀取負載分散，數量可動態調整
  • Segment 是 shard 內部的不可變檔案，每秒 refresh 產生新 segment（NRT 來源）
  • 叢集健康有 GREEN / YELLOW / RED 三狀態，RED 表示有 primary 未分配，必須立即處理
  • 不要把 ES 當 RDBMS — 它沒有 transaction、不適合高頻 update、不應做 JOIN
