Ingest Pipeline 是什麼？

Ingest pipeline 是 Elasticsearch 內建的 ETL 功能。文件寫入前先過 pipeline，可解析、轉換、補欄位、丟棄。比 Logstash 輕量、不需額外 process。

Ingest Pipeline vs Logstash

項目Ingest PipelineLogstash
位置ES 內獨立 process
記憶體共用 ES heap獨立 JVM
協議只支援 ES API多種 input/output（Kafka、JDBC、files...）
處理能力processor 數十種plugin 上百種
緩衝與背壓無persistent queue + DLQ
資料豐富化enrich processorfilter plugin（如 jdbc_static）
適合輕量轉換、簡單解析複雜 ETL、多源整合
💡 推薦組合
  • Beats / Elastic Agent → ES（用 ingest pipeline 做最後處理）
  • 複雜資料清洗：Logstash → ES（搭配 ingest pipeline 做最終調整）
  • OpenTelemetry：直接送 OTel Collector → ES，用 ingest pipeline 補欄位

Ingest Node 角色

9.x 預設所有 data 節點都有 ingest role。對高流量場景可分離 ingest node，避免影響 search/index：

# 專用 ingest 節點
node.roles: [ "ingest" ]

# 一般節點關閉 ingest
node.roles: [ "data_hot", "data_content" ]   # 不含 ingest

但分離 ingest 會多一跳網路；輕度負載不需要分。

第一個 Pipeline

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_ingest/pipeline/my-first-pipeline?pretty" -d '
{
  "description": "Lowercase service name and add ingest_time",
  "processors": [
    {
      "lowercase": {
        "field": "service.name"
      }
    },
    {
      "set": {
        "field":  "ingest_time",
        "value":  "{{ _ingest.timestamp }}"
      }
    }
  ]
}'

三種使用方式

# 方式 1：單筆 index 時指定
POST my-index/_doc?pipeline=my-first-pipeline
{ "service.name": "ORDER-SVC", "msg": "..." }

# 方式 2：bulk 中指定
POST _bulk?pipeline=my-first-pipeline
{ "index": { "_index": "my-index" } }
{ "service.name": "..." }

# 方式 3：index template 設預設 pipeline
PUT _index_template/logs-tpl
{
  "index_patterns": [ "logs-*" ],
  "template": {
    "settings": {
      "index.default_pipeline":  "my-first-pipeline",   # 預設執行
      "index.final_pipeline":    "logs-final-cleanup"   # 最後執行
    }
  }
}

常用 Processors

Processor用途
set設值 / 加新欄位
remove移除欄位
rename改欄位名
copy複製欄位
lowercase / uppercase大小寫轉換
trim移前後空白
convert型別轉換（string ↔ long ↔ double）
date解析日期字串
grok正則模式解析（log line 用）
dissect輕量字串解析（比 grok 快）
kvkey=value 解析
json解析 JSON 字串
csv解析 CSV 列
split字串切陣列
join陣列合字串
geoipIP → 地理位置
user_agent解析 User-Agent
fingerprint多欄位雜湊（去重 ID 用）
scriptPainless 自訂邏輯
pipeline呼叫另一個 pipeline
enrich從 enrich policy 補欄位
inference呼叫 ML / inference endpoint
reroute路由到別的 index / data stream
drop丟棄文件（不寫入）
fail主動失敗
cef9.x 新：解析 Common Event Format

實戰範例

範例 1：解析 nginx access log

PUT _ingest/pipeline/nginx-access
{
  "description": "Parse nginx access log",
  "processors": [
    {
      "grok": {
        "field": "message",
        "patterns": [
          "%{IPORHOST:client.ip} - %{DATA:user.name} \\[%{HTTPDATE:_temp_timestamp}\\] \"%{WORD:http.method} %{DATA:url.path} HTTP/%{NUMBER:http.version}\" %{NUMBER:http.status_code:int} %{NUMBER:http.body_bytes:int} \"%{DATA:http.referer}\" \"%{DATA:user_agent.original}\""
        ],
        "ignore_failure": false
      }
    },
    {
      "date": {
        "field":   "_temp_timestamp",
        "formats": [ "dd/MMM/yyyy:HH:mm:ss Z" ],
        "target_field": "@timestamp",
        "timezone": "+08:00"
      }
    },
    {
      "remove": { "field": [ "_temp_timestamp", "message" ] }
    },
    {
      "geoip": {
        "field":        "client.ip",
        "target_field": "client.geo"
      }
    },
    {
      "user_agent": {
        "field":        "user_agent.original",
        "target_field": "user_agent"
      }
    },
    {
      "convert": {
        "field": "http.body_bytes",
        "type":  "long"
      }
    },
    {
      "set": {
        "field": "is_error",
        "value": true,
        "if":    "ctx.http?.status_code >= 500"
      }
    }
  ],
  "on_failure": [
    {
      "set": {
        "field": "ingest.error",
        "value": "{{{ _ingest.on_failure_message }}}"
      }
    }
  ]
}

範例 2：fingerprint 去重

PUT _ingest/pipeline/dedupe-by-fingerprint
{
  "processors": [
    {
      "fingerprint": {
        "fields":        [ "user.email", "event.action", "@timestamp" ],
        "target_field":  "_id",
        "method":        "SHA-256"
      }
    }
  ]
}

把多欄位雜湊作為 doc ID，重複事件會被覆寫而非新增。

範例 3：reroute 動態路由

PUT _ingest/pipeline/route-by-team
{
  "processors": [
    {
      "reroute": {
        "dataset":    "{{service.team}}",
        "namespace":  "{{environment}}"
      }
    }
  ]
}

原本要進 logs-app-default 的文件，依 team 與 environment 路由到 logs-{team}-{environment}。

範例 4：條件處理

PUT _ingest/pipeline/conditional-example
{
  "processors": [
    {
      "set": {
        "field": "log.tier",
        "value": "high",
        "if":    "ctx.log?.level == 'ERROR' || ctx.log?.level == 'CRITICAL'"
      }
    },
    {
      "drop": {
        "if": "ctx.log?.level == 'DEBUG' && !ctx.environment?.equals('dev')"
      }
    }
  ]
}

if 用 Painless 表達式（CH 42）。?. 是安全 navigation，欄位不存在時不會 NPE。

範例 5：自訂 script

PUT _ingest/pipeline/script-example
{
  "processors": [
    {
      "script": {
        "lang":   "painless",
        "source": "
          if (ctx.duration_ns != null) {
            ctx.duration_ms = ctx.duration_ns / 1000000.0;
            ctx.remove('duration_ns');
          }

          if (ctx.tags == null) {
            ctx.tags = new ArrayList();
          }
          if (ctx.error_count > 0 && !ctx.tags.contains('has_errors')) {
            ctx.tags.add('has_errors');
          }
        "
      }
    }
  ]
}

Enrich Processor：補靜態資料

用獨立 enrich index 為文件補欄位（IP-to-geo、user-info、product-catalog）。

Step 1：建立 source index

PUT users-source/_doc/u1
{
  "user_id":   "u1",
  "full_name": "Alice Lin",
  "team":      "engineering",
  "country":   "TW"
}

Step 2：建立 enrich policy

PUT _enrich/policy/user-lookup
{
  "match": {
    "indices":      "users-source",
    "match_field":  "user_id",
    "enrich_fields": [ "full_name", "team", "country" ]
  }
}

# 執行（建 lookup index）
POST _enrich/policy/user-lookup/_execute

Step 3：在 pipeline 中用

PUT _ingest/pipeline/enrich-with-user
{
  "processors": [
    {
      "enrich": {
        "policy_name":   "user-lookup",
        "field":         "user_id",
        "target_field":  "user",
        "max_matches":   1
      }
    }
  ]
}

輸入 { "user_id": "u1", "action": "login" } 會變：

{
  "user_id": "u1",
  "action": "login",
  "user": {
    "user_id": "u1",
    "full_name": "Alice Lin",
    "team": "engineering",
    "country": "TW"
  }
}
⚠️ Enrich source index 修改後要重新 execute policy Source 變更後 POST _enrich/policy/{name}/_execute 才會更新 lookup data。

Simulate API：在線測試

POST _ingest/pipeline/_simulate
{
  "pipeline": {
    "processors": [
      {
        "grok": {
          "field": "message",
          "patterns": [ "%{TIMESTAMP_ISO8601:ts} \\[%{LOGLEVEL:level}\\] %{GREEDYDATA:msg}" ]
        }
      },
      {
        "date": {
          "field":   "ts",
          "formats": [ "ISO8601" ]
        }
      }
    ]
  },
  "docs": [
    {
      "_source": {
        "message": "2026-04-15T10:23:45.123Z [ERROR] Database connection failed"
      }
    },
    {
      "_source": {
        "message": "INVALID FORMAT"
      }
    }
  ]
}

輸出每筆文件的處理結果與 error（若有）。

測試已存在的 pipeline

POST _ingest/pipeline/nginx-access/_simulate?verbose=true
{
  "docs": [
    { "_source": { "message": "..." } }
  ]
}

verbose=true 會顯示每個 processor 的中間結果。

錯誤處理（複習 + 進階）

Processor 層級 on_failure

{
  "grok": {
    "field": "message",
    "patterns": [ ... ],
    "on_failure": [
      { "set": { "field": "log.parse_failed", "value": true } }
    ]
  }
}

ignore_failure

{
  "grok": {
    "field": "message",
    "patterns": [ ... ],
    "ignore_failure": true   // 失敗就跳過此 processor
  }
}

ignore_missing

{
  "rename": {
    "field": "old_name",
    "target_field": "new_name",
    "ignore_missing": true   // 欄位不存在就跳過（不視為錯誤）
  }
}

Pipeline 層級 on_failure

{
  "processors": [ ... ],
  "on_failure": [
    {
      "set": {
        "field": "_index",
        "value": "failed-{{ _index }}"
      }
    },
    {
      "set": {
        "field": "ingest.failure_reason",
        "value": "{{ _ingest.on_failure_message }}"
      }
    }
  ]
}

Pipeline Versioning

PUT _ingest/pipeline/my-pipeline
{
  "version":      3,
  "description":  "v3: added geoip processor",
  "_meta": {
    "owner":      "platform-team",
    "managed_by": "github-repo:platform/es-pipelines",
    "last_change": "2026-04-15"
  },
  "processors": [ ... ]
}

更新時若指定 ?if_version=3，version 不對會拒絕更新（樂觀鎖）：

PUT _ingest/pipeline/my-pipeline?if_version=3

效能調校

  • Processor 越少越好：每個都耗 CPU
  • 用 dissect 取代 grok（如果格式固定）：dissect 不用 regex backtrack，速度快數倍
  • geoip 與 user_agent 用了快取：第二次同樣值很快
  • 避免在大 doc 上用 script：每筆都跑 Painless 慢
  • 分離 ingest node（高流量）：保護 data node 的 IO
  • 監控：
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_nodes/stats/ingest?human&pretty"

輸出每個 pipeline 的：

  • count：執行總次數
  • time_in_millis：累計耗時
  • current：當前處理中
  • failed：失敗次數

9.x 內建 OTel Pipelines

9.x 為 OpenTelemetry 提供大量內建 pipelines（無需自己寫）：

  • logs-otel@custom、metrics-otel@custom、traces-otel@custom：使用者客製
  • elastic-distribution-otel@template：自動 ECS 對應
  • OTLP endpoint 直送 ES，pipeline 自動套用

OTel 整合 → CH 64

最佳實踐

  • 用 simulate API 測試新 pipeline，再 deploy
  • 名稱 + version + _meta：可追蹤、避免誤改
  • 每個 processor 評估 on_failure / ignore_failure
  • 失敗的 doc 路由到 DLQ（CH 35 failure store / failed-*）
  • 能用 dissect 不用 grok（簡單格式）
  • 用 enrich processor 做 IP/user/product lookup
  • 大量資料：用 Logstash 處理，ingest pipeline 做最終調整
  • 定期 review 慢的 pipeline（_nodes/stats/ingest）
  • 用 reroute 取代 multi-pipeline，提升可讀性
  • Pipeline 進入 GitOps（Terraform / Ansible / Fleet）

本章重點回顧

  • Ingest pipeline 在 ES 內就地轉換資料，比 Logstash 輕量
  • Pipeline = 一連串 processor；30+ 內建 processor 涵蓋常見需求
  • Processor 三大類：解析（grok/dissect/json）、轉換（set/rename/convert）、豐富（geoip/enrich/inference）
  • Enrich processor 從 source index 補資料，需執行 policy 才生效
  • Reroute processor 動態路由 doc 到不同 data stream
  • Pipeline 三層 on_failure：processor 層、pipeline 層、failure store（CH 35）
  • 用 simulate API 上線前測試，verbose=true 看每步結果
  • Index template 設 default_pipeline / final_pipeline 自動套用
  • 監控 _nodes/stats/ingest，找慢的 processor
  • 9.x OTel 整合提供大量內建 pipelines
