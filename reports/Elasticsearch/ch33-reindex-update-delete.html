_reindex：複製 / 遷移 index

用途：

  • 修改 mapping（型別、analyzer 不可改 → 必須建新 index）
  • 從舊 index 重新整理到 data stream
  • 跨叢集遷移
  • 從 backup 還原後 reindex 修正資料

最基本

curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_reindex?pretty" -d '
{
  "source": { "index": "old-products" },
  "dest":   { "index": "new-products" }
}'

非同步執行（大量資料必備）

curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_reindex?wait_for_completion=false&pretty" -d '
{
  "source": { "index": "old-products" },
  "dest":   { "index": "new-products" }
}'

會回傳 task ID：

{ "task": "abc123:42" }

監控進度：

curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_tasks/abc123:42?pretty"

只 reindex 部分文件

POST _reindex
{
  "source": {
    "index": "old-products",
    "query": {
      "range": { "created_at": { "gte": "2026-01-01" } }
    },
    "_source": [ "name", "price", "category" ]   # 只帶這些欄位
  },
  "dest": { "index": "new-products" }
}

Reindex with Script（資料轉換）

POST _reindex
{
  "source": { "index": "old-orders" },
  "dest":   { "index": "new-orders" },
  "script": {
    "source": "
      // 重新命名欄位
      ctx._source.order_id = ctx._source.id;
      ctx._source.remove('id');

      // 計算新欄位
      ctx._source.total_with_tax = ctx._source.total * 1.05;

      // 條件過濾
      if (ctx._source.status == 'cancelled') { ctx.op = 'noop'; }

      // 改 _id
      // ctx._id = ctx._source.order_id;
    ",
    "lang": "painless"
  }
}

script 中的特殊變數：

變數用途
ctx._source文件內容（可改）
ctx._id文件 ID（可改）
ctx._index目的 index 名稱（可改，動態路由）
ctx.op'index'(預設)、'noop'(跳過)、'delete'

Cross-cluster Reindex

從遠端 cluster 拉資料：

Step 1：在 dest cluster 啟用 remote whitelist

# elasticsearch.yml
reindex.remote.whitelist: [ "remote.example.com:9200", "*.internal:9200" ]

重啟 dest cluster 各節點。

Step 2：執行

POST _reindex
{
  "source": {
    "remote": {
      "host":     "https://source-cluster.example.com:9200",
      "username": "reindex_user",
      "password": "...",
      "socket_timeout":  "1m",
      "connect_timeout": "10s"
    },
    "index": "old-logs",
    "query": {
      "range": { "@timestamp": { "gte": "2026-01-01" } }
    }
  },
  "dest": { "index": "logs-from-old-cluster" }
}
⚠️ Remote reindex 限制
  • 透過 HTTP（非 transport）連線，速度較慢
  • 需要 source cluster 開放 HTTPS 給 dest cluster
  • 大量資料建議用 snapshot/restore 替代（CH 47）
  • 9.x 對 8.x source 大致相容；跨 major version 跳級可能失敗

Reindex 效能調校

Slicing：平行化

POST _reindex?slices=auto&refresh
{
  "source": { "index": "huge-source" },
  "dest":   { "index": "huge-dest" }
}

slices=auto 會依 source shard 數自動切片。每個 slice 平行跑，大幅加速。

batch size

{
  "source": {
    "index": "huge-source",
    "size":  5000      # 每批拉 5000 筆（預設 1000）
  },
  "dest": { "index": "huge-dest" }
}

Dest index 預先建好 + 調快

# 先建 dest，先關 replica + 調慢 refresh
PUT new-products
{
  "settings": {
    "number_of_shards":   3,
    "number_of_replicas": 0,           # reindex 期間關 replica
    "refresh_interval":   "60s"        # 或 "-1" 關 refresh
  },
  "mappings": { ... }
}

# 開始 reindex
POST _reindex?slices=auto
{ ... }

# Reindex 完成後恢復
PUT new-products/_settings
{
  "number_of_replicas": 1,
  "refresh_interval":   "1s"
}

# 立即觸發 refresh
POST new-products/_refresh

Reindex 最佳實踐 Pattern

當你需要改 mapping、把舊 index 換掉但不想停機：

sequenceDiagram
    participant App as 應用程式
    participant Alias as products (alias)
    participant Old as products-v1
    participant New as products-v2

    Note over Alias,Old: 初始：alias 指向 v1
    App->>Alias: search/index 都用 alias 名

    Note over New: 1. 建立 v2（含新 mapping）
    App->>New: PUT products-v2 (new mapping)

    Note over New,Old: 2. Reindex
    App->>Old: POST _reindex (slices=auto)
    Old-->>New: 全部資料複製過去

    Note over Alias: 3. 切換 alias
    App->>Alias: 一個原子操作:
移除 v1, 加 v2

    Note over Old: 4. 觀察一段時間後刪除
    App->>Old: DELETE products-v1
圖 33-2：用 alias 做無縫 reindex 切換
# 1. 建新 index
PUT products-v2 { ... new mapping ... }

# 2. Reindex
POST _reindex?slices=auto&refresh
{
  "source": { "index": "products-v1" },
  "dest":   { "index": "products-v2" }
}

# 3. 原子切換 alias
POST _aliases
{
  "actions": [
    { "remove": { "index": "products-v1", "alias": "products" } },
    { "add":    { "index": "products-v2", "alias": "products" } }
  ]
}

# 4. 觀察一週後
DELETE products-v1

_update_by_query：原地更新

用途：

  • 批次修正資料（normalize 欄位值）
  • 新增 mapping 後，觸發既有文件 re-index（讓新欄位填值）
  • 套用 ingest pipeline 到既有資料

基本範例

POST products/_update_by_query?refresh
{
  "query": {
    "term": { "category": "ELECTRONICS" }
  },
  "script": {
    "source": "ctx._source.category = ctx._source.category.toLowerCase()",
    "lang":   "painless"
  }
}

新增欄位後重新索引

# 加新欄位
PUT products/_mapping
{
  "properties": {
    "discounted_price": { "type": "scaled_float", "scaling_factor": 100 }
  }
}

# 套用到既有文件
POST products/_update_by_query?refresh&conflicts=proceed
{
  "script": {
    "source": "ctx._source.discounted_price = ctx._source.price * 0.9"
  }
}

conflicts=proceed

預設 update_by_query 遇到 version conflict 會中止。conflicts=proceed 讓它跳過繼續。

套用 ingest pipeline

POST products/_update_by_query?pipeline=add-tax-pipeline
{
  "query": { "match_all": {} }
}

等同於把每筆文件再過一次 ingest pipeline。

_delete_by_query：批次刪除

POST products/_delete_by_query?conflicts=proceed&refresh
{
  "query": {
    "bool": {
      "must": [
        { "term":  { "status": "deleted" } },
        { "range": { "deleted_at": { "lt": "2026-01-01" } } }
      ]
    }
  }
}
⚠️ 注意：標記刪除而非實體刪除 ES 的刪除是「標記」（tombstone），不會立即釋放空間。實體釋放在 segment merge 時才發生。
  • 大量刪除後，磁碟空間可能不會立即下降
  • 需要 force merge 才能真正回收（但 forcemerge 很耗 IO）
  • 對 data stream 推薦用 ILM delete phase 整個 index 砍掉，比 delete by query 更高效

Task Management

長時間操作會註冊為 task，可監控、取消：

列出 task

curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_tasks?actions=*reindex*&detailed&pretty"

看 task 細節

curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_tasks/abc123:42?pretty"

輸出：

{
  "completed": false,
  "task": {
    "node":   "abc123",
    "id":     42,
    "type":   "transport",
    "action": "indices:data/write/reindex",
    "status": {
      "total":     1000000,
      "updated":   0,
      "created":   234567,
      "deleted":   0,
      "batches":   235,
      "version_conflicts": 0,
      "noops":     0,
      "retries":   { "bulk": 0, "search": 0 },
      "throttled_millis": 0,
      "requests_per_second": -1.0
    }
  }
}

取消 task

curl -X POST -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_tasks/abc123:42/_cancel"

已寫入的 doc 不會 rollback，但後續會停止。

Throttling（流量控制）

避免 reindex 影響線上查詢：

POST _reindex?requests_per_second=500
{
  "source": { "index": "huge" },
  "dest":   { "index": "huge-new" }
}

執行中也可動態調整：

curl -X POST -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_reindex/abc123:42/_rethrottle?requests_per_second=200"

requests_per_second=-1 即無限制（預設）。

錯誤處理

Version conflict

當文件在 reindex 過程中被外部修改：

  • 預設 reindex 中止
  • 用 conflicts=proceed 跳過
  • 或用 "version_type": "external" 配合 source _version

中斷後復原

Reindex/update_by_query 不能直接 resume，但可加上 query filter 跳過已處理的：

{
  "source": {
    "index": "huge",
    "query": {
      "range": { "@timestamp": { "gte": "2026-04-15T08:00:00Z" } }
    }
  },
  "dest": { "index": "huge-new" }
}

最佳實踐

  • 大量 reindex 一律 async + slices=auto
  • Dest 預先建好 + 關 replica + 調慢 refresh，完成後再恢復
  • Reindex 期間用 throttling 控制速度，保護線上查詢
  • 用 alias 切換達到無停機 reindex
  • 大量刪除：data stream 直接刪整個 backing index 比 _delete_by_query 高效
  • 新增欄位後，用 _update_by_query 觸發 re-index 填值
  • 跨 cluster 大量遷移：用 snapshot + restore 比 remote reindex 快
  • 監控 task 狀態與 version_conflicts、noops 等指標
  • Script reindex 時用 "size": 1 先單筆測試 script，確認後再正式跑

本章重點回顧

  • 三大批次操作：_reindex（轉到新 index）、_update_by_query（原地改）、_delete_by_query（原地刪）
  • Reindex 是修改 mapping 的標準做法（mapping 大多不可變）
  • 大量資料用 slices=auto + wait_for_completion=false async 執行
  • Reindex 配合 alias 達到無停機切換
  • Cross-cluster reindex 需在 dest 設 reindex.remote.whitelist
  • Update by query 常用於新增欄位後填值，或套用 ingest pipeline 到既有資料
  • Delete by query 是標記刪除，空間在 segment merge 才釋放
  • Task API 監控進度、取消、調整 throttle
  • 跨 cluster 大量遷移優先用 snapshot/restore
