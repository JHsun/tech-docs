APM 是什麼？

Application Performance Monitoring (APM) 監控應用程式內部行為：

  • Transaction：請求/操作的完整流程（一個 HTTP request、一個 background job）
  • Span：transaction 內部的子操作（DB query、HTTP 呼叫、function call）
  • Error：例外與 stack trace
  • Metric：JVM heap、GC、execution time histogram
  • Service map：服務間相依視覺化

部署 APM Server

9.x 推薦透過 Elastic Agent + APM integration，不再單獨裝 APM Server。

Step 1：在 Fleet 加 APM integration

Fleet → Integrations → 搜尋 "APM" → Add → 選 policy。

設定：

  • APM Server URL：對外 listen 位置（預設 0.0.0.0:8200）
  • Secret token：給 APM agent 認證用
  • API Key：另一種認證
  • Output：通常與 Fleet 共用（送 ES）

Step 2：把 policy 應用到 host

該 host 上的 Elastic Agent 會啟動 APM Server sub-process。

Step 3：驗證

curl https://apm-server.example.com:8200/
# 回傳 JSON 含 build_date, version, ...

# 9.x 版 status
curl https://apm-server.example.com:8200/api/v1/agents/config

自動 Instrumentation 範圍

Java agent 自動 trace 的（部分）：

  • HTTP frameworks: Spring MVC、Spring Boot、JAX-RS、Servlet、Apache HttpClient、OkHttp
  • DB: JDBC、JPA、Hibernate、MongoDB、Redis、Elasticsearch、Cassandra
  • Messaging: Kafka、RabbitMQ、JMS
  • Async: ExecutorService、CompletableFuture
  • Logging: Logback、Log4j2（log correlation）
  • AWS SDK：S3、SQS、SNS、DynamoDB、Lambda
  • gRPC、Webflux、reactive frameworks

大部分業務代碼無需改動即可獲得 trace。

手動加 Span

// Java
Transaction tx = ElasticApm.currentTransaction();
Span span = tx.startSpan("custom", "process", "items");
try {
    span.setName("Process order items");
    span.setLabel("order_id", orderId);
    // ... 業務邏輯
} catch (Exception e) {
    span.captureException(e);
    throw e;
} finally {
    span.end();
}

# Python
import elasticapm

@elasticapm.capture_span()
def expensive_function():
    # 自動建一個 span
    pass

# 或手動
with elasticapm.capture_span("custom_op", "custom"):
    do_work()

APM 功能模組

Services Overview

所有 instrument 過的服務，含：throughput、latency、error rate（黃金三角指標）。

Service Map

視覺化服務間呼叫關係，自動繪製拓撲圖。包含外部相依（DB、Redis、外部 API）。

Transactions

HTTP request 等的 latency、throughput；按 endpoint 分組。下鑽看 transaction sample（含 span waterfall）。

Dependencies

外部相依（DB、cache、API）的 latency 與 throughput。找出哪個 dependency 是 bottleneck。

Errors

例外清單，依 error message 與 stack trace 分組。可看：

  • occurrence count
  • users affected
  • first / last seen
  • 關聯的 trace

Metrics

Runtime metrics：JVM heap、GC、CPU、threads、Node.js event loop 等。

Logs（log correlation）

透過 trace.id correlation，APM transaction 可一鍵跳到當時的 log。需要 log 中有 trace.id 欄位（agent 自動注入）。

Universal Profiling（9.x）

Continuous profiling：

  • kernel-level eBPF profiler（不需 instrumentation）
  • 整合進 APM 介面
  • flame graph 視覺化
  • 跨服務、跨語言（Go、Java、Python、Native）

需要安裝 elastic-universal-profiling agent 在 host。

RUM (Real User Monitoring)

瀏覽器端的監控：

  • Page load time、Time to First Byte、First Contentful Paint
  • JS 例外、Resource load 失敗
  • SPA 路由變更
  • 使用者體驗（Web Vitals）
<script src="https://cdn.jsdelivr.net/npm/@elastic/apm-rum@5/dist/bundles/elastic-apm-rum.umd.min.js"></script>
<script>
  elasticApm.init({
    serviceName:    'frontend-app',
    serverUrl:      'https://apm-server:8200',
    serviceVersion: '1.0.0',
    environment:    'production',
    distributedTracingOrigins: [ 'https://api.example.com' ]
  })
</script>

APM Data Storage

APM 資料存於 data stream：

  • traces-apm-{namespace}：transactions、spans
  • metrics-apm.app-{namespace}：app metrics
  • metrics-apm.internal-{namespace}：APM agent 內部 metrics
  • logs-apm.error-{namespace}：errors

有對應 ILM policy（traces@lifecycle 等），9.3+ 命名規範。

Sampling：取樣策略

高流量服務 trace 量大，需取樣：

Head-based Sampling（agent 端）

elastic.apm.transaction_sample_rate=0.1   # 10% 取樣

缺點：取樣是隨機的，可能漏掉慢的 / 出錯的。

Tail-based Sampling（APM Server 端，9.x）

所有 span 先 buffer，再決定保留條件（如保留所有錯誤、慢的、抽 10% 正常）：

apm-server.sampling.tail.enabled: true
apm-server.sampling.tail.policies:
  - sample_rate: 1.0
    error: true                       # 所有錯誤 100% 保留
  - sample_rate: 1.0
    trace.duration: ">1s"             # 慢的 100% 保留
  - sample_rate: 0.1                  # 其他抽 10%

APM 與 OpenTelemetry 的關係

9.x Elastic 全力擁抱 OTel：

項目Elastic APM AgentOpenTelemetry SDK
Vendor neutral❌✅
新功能投資維護主推
自動 instrumentation 範圍廣（已成熟）越來越廣
送 ES透過 APM Server透過 OTLP（9.x ES 內建）或 EDOT collector
Elastic 推薦既有部署可用新部署建議

9.x 後不需要 APM Server——直接用 EDOT Collector 接 OTLP，送 ES。CH 64 詳述。

最佳實踐

  • 新部署用 OpenTelemetry SDK + EDOT Collector（vendor neutral）
  • 既有 Elastic APM Agent 部署可繼續用
  • 所有服務都要 instrument，service map 才完整
  • service.name + environment 命名一致
  • 用 secret token 或 API Key，不要裸 endpoint
  • RUM 開啟 distributedTracingOrigins，前後端 trace 串聯
  • Tail-based sampling 是高流量場景必備
  • 啟用 log correlation，trace ↔ log 互通
  • Universal Profiling 是 deep dive performance 的利器
  • 用 ILM 管理 trace data：hot 7d、cold 30d、刪 90d 為起點
  • Service overview / errors 設 alert：error rate 突增、p95 退化

本章重點回顧

  • APM 監控應用內部：transaction、span、error、metric
  • 9.x APM Server 透過 Elastic Agent + APM integration 部署
  • 多語言 agent：Java、Node.js、Python、Go、Ruby、.NET、PHP、RUM
  • 大部分 framework 自動 instrument，無需改業務代碼
  • Distributed tracing 用 W3C traceparent 自動跨服務
  • Service Map、Service Overview、Errors、Logs correlation 是核心介面
  • Sampling：head-based（agent）vs tail-based（server，更聰明）
  • Universal Profiling 9.x 提供 always-on profiling
  • Elastic 主推 OpenTelemetry，新部署優先用 OTel SDK
  • 9.x 後不需要 APM Server，EDOT Collector 直送 ES OTLP
