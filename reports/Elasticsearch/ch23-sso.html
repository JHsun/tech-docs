LDAP / Active Directory Realm

用於企業內部 user directory（OpenLDAP、AD、FreeIPA、389 Directory Server）。AD 在 ES 中其實就是 LDAP 的特化（用 active_directory 類型可省略 user search 配置）。

兩種驗證模式

模式說明適用
user_search先用 bind DN 連入 LDAP，搜尋 user，再用其密碼 bind 一次驗證多種 user OU、需動態查詢
user DN templates直接以 user 提供的密碼 bind 已知模式 DN所有 user 在同一 OU 下

基本 LDAP 設定（user_search 模式）

# elasticsearch.yml
xpack.security.authc.realms.ldap.ldap1:
  order: 2
  url: "ldaps://ldap.example.com:636"
  bind_dn: "cn=ldap-reader,ou=service-accounts,dc=example,dc=com"
  user_search:
    base_dn: "ou=users,dc=example,dc=com"
    filter: "(uid={0})"
    scope: sub_tree
  group_search:
    base_dn: "ou=groups,dc=example,dc=com"
    filter: "(member={0})"
    scope: sub_tree
  unmapped_groups_as_roles: false
  ssl:
    certificate_authorities: [ "config/certs/ldap-ca.crt" ]
    verification_mode: full
  cache:
    ttl: 1h
    max_users: 10000

把 bind DN 密碼存 keystore：

sudo /usr/share/elasticsearch/bin/elasticsearch-keystore add \
  xpack.security.authc.realms.ldap.ldap1.secure_bind_password

Active Directory 簡化設定

xpack.security.authc.realms.active_directory.ad1:
  order: 2
  domain_name: example.com
  url: "ldaps://dc1.example.com:636"
  load_balance:
    type: round_robin
  ssl:
    certificate_authorities: [ "config/certs/ad-ca.crt" ]
    verification_mode: full
  unmapped_groups_as_roles: false

AD 的 user 用 username@domain 或 DOMAIN\username 格式登入，ES 會自動處理。

User DN templates 模式

xpack.security.authc.realms.ldap.ldap1:
  order: 2
  url: "ldaps://ldap.example.com:636"
  user_dn_templates:
    - "uid={0},ou=engineering,dc=example,dc=com"
    - "uid={0},ou=sales,dc=example,dc=com"
  group_search:
    base_dn: "ou=groups,dc=example,dc=com"
  ssl:
    certificate_authorities: [ "config/certs/ldap-ca.crt" ]

驗證 LDAP 連線

# 測試 LDAP user 登入
curl -u "alice:LDAP-Password" --cacert ca.crt \
  "https://localhost:9200/_security/_authenticate?pretty"

# 應看到 authentication_realm.name = "ldap1"

LDAP Role Mapping

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role_mapping/ldap-admins" -d '
{
  "roles": [ "superuser" ],
  "enabled": true,
  "rules": {
    "all": [
      { "field": { "realm.name": "ldap1" } },
      { "field": { "groups": "cn=es-admins,ou=groups,dc=example,dc=com" } }
    ]
  }
}'

SAML 2.0 Realm

SAML 是企業 SSO 最常見的標準，搭配 Okta、Azure AD、Auth0、ADFS、PingFederate、OneLogin 等 IdP。

SAML 認證流程

sequenceDiagram
    participant User as 👤 User
    participant Browser as 🌐 Browser
    participant Kibana as Kibana
    participant ES as Elasticsearch
    participant IdP as 🔐 IdP (Okta)

    User->>Browser: 開啟 Kibana
    Browser->>Kibana: GET /
    Kibana->>ES: prepare SAML auth
    ES-->>Kibana: SAML AuthN request
    Kibana-->>Browser: redirect to IdP
    Browser->>IdP: GET /sso?SAMLRequest=...
    IdP->>User: 顯示登入畫面
    User->>IdP: 輸入帳密 + MFA
    IdP-->>Browser: HTML form auto-POST
SAMLResponse → Kibana ACS
    Browser->>Kibana: POST /api/security/saml/callback
    Kibana->>ES: authenticate SAML response
    ES-->>Kibana: token + user info
    Kibana-->>Browser: Set session cookie
圖 23-2：SAML SP-initiated 流程

IdP 端設定（在 Okta/Azure 設）

  • SP Entity ID：通常用 https://kibana.example.com
  • ACS URL：https://kibana.example.com/api/security/v1/saml（callback）
  • NameID format：urn:oasis:names:tc:SAML:1.1:nameid-format:emailAddress 或 persistent
  • 下載 IdP metadata XML（含 IdP 簽章公鑰、SSO URL）

elasticsearch.yml 設定

xpack.security.authc.realms.saml.saml1:
  order: 3
  attributes.principal: "nameid:persistent"
  attributes.groups:    "groups"
  attributes.name:      "displayName"
  attributes.mail:      "email"
  idp.metadata.path:    "saml/okta-metadata.xml"
  idp.entity_id:        "http://www.okta.com/exk1abc..."
  sp.entity_id:         "https://kibana.example.com"
  sp.acs:               "https://kibana.example.com/api/security/v1/saml"
  sp.logout:            "https://kibana.example.com/logout"
  ssl:
    certificate_authorities: [ "config/certs/ca.crt" ]

Metadata XML 放在 $ES_PATH_CONF/saml/okta-metadata.xml。

kibana.yml 設定

xpack.security.authc.providers:
  saml.saml1:
    order: 0
    realm: saml1
    description: "Login with SSO"
    icon: "logoSecurity"
  basic.basic1:
    order: 1
    description: "Local login"

SAML Role Mapping

根據 SAML attributes 中 groups 對應 ES role：

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role_mapping/saml-engineers" -d '
{
  "roles": [ "developer", "kibana_user" ],
  "enabled": true,
  "rules": {
    "all": [
      { "field": { "realm.name": "saml1" } },
      { "field": { "groups": "es-engineers" } }
    ]
  }
}'

Azure AD / Entra ID 額外注意

  • Application 類型選 Enterprise application，啟用 SAML SSO
  • Group claim 來自 Azure AD groups，可用 GUID 或 display name；通常用 GUID 並在 ES role_mapping 寫對應
  • 若 user 屬於 100+ groups，Azure 會回 groups overage，需改用 Graph API 查詢

OpenID Connect (OIDC) Realm

OIDC 是基於 OAuth 2.0 的現代 SSO 標準，比 SAML 簡單。Google、Microsoft、Auth0、Okta 都支援。

Authorization Code Flow

1. User 開 Kibana → 重導到 IdP
2. User 在 IdP 登入 → IdP 回傳 authorization code
3. Kibana 用 code 向 IdP 交換 ID token + access token
4. ES 驗證 ID token 簽章與 claims，建立 ES session

elasticsearch.yml 設定

xpack.security.authc.realms.oidc.oidc1:
  order: 4
  rp.client_id:    "kibana-client-id"
  rp.response_type: "code"
  rp.redirect_uri: "https://kibana.example.com/api/security/oidc/callback"
  op.issuer:       "https://login.microsoftonline.com/{tenant-id}/v2.0"
  op.authorization_endpoint: "https://login.microsoftonline.com/{tenant-id}/oauth2/v2.0/authorize"
  op.token_endpoint:         "https://login.microsoftonline.com/{tenant-id}/oauth2/v2.0/token"
  op.userinfo_endpoint:      "https://graph.microsoft.com/oidc/userinfo"
  op.jwkset_path:            "https://login.microsoftonline.com/{tenant-id}/discovery/v2.0/keys"
  rp.requested_scopes: [ "openid", "email", "profile", "groups" ]
  claims.principal: "preferred_username"
  claims.groups:    "groups"
  claims.name:      "name"
  claims.mail:      "email"
  ssl:
    certificate_authorities: [ "config/certs/ca.crt" ]

把 client secret 存 keystore：

sudo /usr/share/elasticsearch/bin/elasticsearch-keystore add \
  xpack.security.authc.realms.oidc.oidc1.rp.client_secret

kibana.yml

xpack.security.authc.providers:
  oidc.oidc1:
    order: 0
    realm: oidc1
    description: "Login with Microsoft"

Google Workspace OIDC 範例

xpack.security.authc.realms.oidc.google:
  order: 4
  rp.client_id: "xxx.apps.googleusercontent.com"
  rp.response_type: "code"
  rp.redirect_uri: "https://kibana.example.com/api/security/oidc/callback"
  op.issuer: "https://accounts.google.com"
  op.authorization_endpoint: "https://accounts.google.com/o/oauth2/v2/auth"
  op.token_endpoint:         "https://oauth2.googleapis.com/token"
  op.userinfo_endpoint:      "https://openidconnect.googleapis.com/v1/userinfo"
  op.jwkset_path:            "https://www.googleapis.com/oauth2/v3/certs"
  rp.requested_scopes: [ "openid", "email", "profile" ]
  claims.principal: "email"
  claims.name:      "name"

PKI Realm — 客戶端憑證認證

適合 device-to-ES 場景（IoT、機台 Beats agent），不需密碼。

前置：HTTP TLS 啟用 client auth

# elasticsearch.yml
xpack.security.http.ssl.enabled: true
xpack.security.http.ssl.client_authentication: optional   # 或 required
xpack.security.http.ssl.certificate_authorities:
  - "config/certs/client-ca.crt"

啟用 PKI realm

xpack.security.authc.realms.pki.pki1:
  order: 5
  certificate_authorities:
    - "config/certs/client-ca.crt"
  username_pattern: "CN=(.*?)(?:,|$)"   # 從 cert subject 抽 username

Role Mapping（依 cert DN）

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role_mapping/pki-iot" -d '
{
  "roles": [ "iot_writer" ],
  "enabled": true,
  "rules": {
    "all": [
      { "field": { "realm.name": "pki1" } },
      { "field": { "dn": "*,OU=IoT,O=Example*" } }
    ]
  }
}'

使用 client cert 連線

curl --cert client.crt --key client.key --cacert ca.crt \
  "https://localhost:9200/_security/_authenticate?pretty"
# 應看到 authentication_realm.name = "pki1"

Kerberos / SPNEGO Realm

讓 Windows 域內的 user 自動 SSO（無需輸入密碼），使用 SPNEGO HTTP 認證。

設定流程概述

  1. 在 KDC（通常是 AD DC）為 ES 服務建立 service principal：HTTP/elasticsearch.example.com@EXAMPLE.COM
  2. 產生 keytab 檔，部署到 ES 節點（如 /etc/elasticsearch/krb5.keytab）
  3. elasticsearch.yml 啟用 kerberos realm
xpack.security.authc.realms.kerberos.krb1:
  order: 3
  keytab.path: krb5.keytab
  remove_realm_name: false

系統需要 /etc/krb5.conf

[libdefaults]
default_realm = EXAMPLE.COM

[realms]
EXAMPLE.COM = {
  kdc = dc1.example.com
  admin_server = dc1.example.com
}

[domain_realm]
.example.com = EXAMPLE.COM
example.com = EXAMPLE.COM

使用 kinit 取得 ticket

kinit alice@EXAMPLE.COM

# 用 SPNEGO 認證
curl --negotiate -u : --cacert ca.crt \
  "https://elasticsearch.example.com:9200/_security/_authenticate?pretty"

JWT Realm

適合 service-to-service 場景：上游服務簽 JWT，ES 驗證。比起每次都用 ES API key，可由統一身份提供方產生 JWT。

兩種模式

  • id_token：類似 OIDC ID token，需驗 issuer、audience、簽名
  • access_token：bearer token，可用對方公鑰或共享金鑰驗證
xpack.security.authc.realms.jwt.jwt1:
  order: 6
  token_type: id_token
  allowed_issuer: "https://my-issuer.example.com"
  allowed_audiences: [ "elasticsearch" ]
  allowed_signature_algorithms: [ "RS256" ]
  pkc_jwkset_path: "https://my-issuer.example.com/.well-known/jwks.json"
  claims.principal: "sub"
  claims.groups:    "groups"
  client_authentication.type: shared_secret

Shared secret 存 keystore：

sudo /usr/share/elasticsearch/bin/elasticsearch-keystore add \
  xpack.security.authc.realms.jwt.jwt1.client_authentication.shared_secret

使用方式

curl --cacert ca.crt \
  -H "Authorization: Bearer eyJhbGc..." \
  -H "ES-Client-Authentication: SharedSecret <client_secret>" \
  "https://localhost:9200/_security/_authenticate"

多 Realm 共存組合範例

實務上常同時啟用多種 realm。範例：file（emergency）+ native（內部 user）+ ldap（員工）+ saml（外部夥伴）：

xpack.security.authc:
  realms:
    file:
      file1:
        order: 0    # emergency admin 永遠優先

    native:
      native1:
        order: 1    # 內部 user / API key

    ldap:
      ldap1:
        order: 2
        url: "ldaps://ad.example.com:636"
        user_search:
          base_dn: "ou=users,dc=example,dc=com"
          filter: "(sAMAccountName={0})"
        # ...

    saml:
      saml1:
        order: 3
        idp.metadata.path: saml/okta.xml
        # ...

當 user 提供 username/password：file → native → ldap 依序試。SAML 走獨立 web flow，不會被嘗試。

Cache 與 Performance 調校

每個 realm 都有 cache 避免每次認證打 LDAP/IdP：

xpack.security.authc.realms.ldap.ldap1:
  cache:
    ttl: 1h           # 認證結果快取多久
    max_users: 10000  # 最多快取多少 user

密碼變更後 cache 清除：

curl -X POST -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/realm/ldap1/_clear_cache"

# 清除特定 user
curl -X POST -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/realm/ldap1/_clear_cache?usernames=alice,bob"

疑難排解

LDAP 連不上

# 用 ldapsearch 從 ES 節點測
ldapsearch -x -H ldaps://ldap.example.com -D "cn=ldap-reader,..." -W \
  -b "ou=users,dc=example,dc=com" "(uid=alice)"

# 開 debug log
PUT _cluster/settings
{
  "persistent": {
    "logger.org.elasticsearch.xpack.security.authc.ldap": "DEBUG"
  }
}

SAML 失敗

  • 檢查 IdP 簽章證書是否正確匯入 ES（通常在 metadata XML 裡）
  • 確認 SP Entity ID、ACS URL 在 IdP 與 ES 完全一致
  • 時鐘飄移：SAML 對 NotBefore / NotAfter 嚴格，所有節點 NTP 必須準
  • 開 trace log：
PUT _cluster/settings
{
  "persistent": {
    "logger.org.elasticsearch.xpack.security.authc.saml": "TRACE"
  }
}

OIDC 失敗

  • op.issuer 必須與 token 內 iss claim 完全一致（含 https / 結尾 slash）
  • Client ID 與 Redirect URI 必須在 IdP 端註冊
  • JWKS endpoint 可達且回傳有效 JWK set

本章重點回顧

  • ES 支援多種企業身份來源：LDAP/AD、SAML、OIDC、PKI、Kerberos、JWT
  • 所有 realm 必須有 order，由小到大依序嘗試（SAML/OIDC 走 web flow 例外）
  • LDAP 兩種模式：user_search（彈性）vs user_dn_templates（簡單）
  • SAML 必設：SP Entity ID、ACS URL、IdP metadata；常見 IdP：Okta、Azure AD、ADFS
  • OIDC 適合現代 IdP（Google、Microsoft、Auth0），比 SAML 簡單
  • PKI 用 client certificate 認證，搭配 HTTP TLS client_authentication
  • Role mapping 把外部 group/attribute 對應到 ES roles，是 SSO 整合靈魂
  • 所有密碼/secret 一律 keystore；TLS 必須先就緒
  • 每個 realm 都有 cache，密碼變更後可清除
