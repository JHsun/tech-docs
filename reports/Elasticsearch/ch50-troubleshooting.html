黃金 5 命令

遇到問題先跑這 5 個指令：

# 1. 整體健康
GET /_cluster/health?level=indices&pretty

# 2. 9.x 統一健康報告
GET /_health_report?pretty

# 3. 異常 shard
GET /_cat/shards?v&h=index,shard,prirep,state,unassigned.reason,node | grep -v STARTED

# 4. 節點狀態
GET /_cat/nodes?v&h=name,role,heap.percent,ram.percent,cpu,disk.used_percent,master&s=heap.percent:desc

# 5. Pending tasks
GET /_cluster/pending_tasks?pretty

場景 1：Cluster RED

RED = 至少一個 primary shard unassigned。資料正在寫的 index 可能無法被寫入。

診斷

# 1. 找出 unassigned 的 shard
GET /_cat/shards?v&h=index,shard,prirep,state,unassigned.reason | grep UNASSIGNED

# 2. 對每個 unassigned shard 查原因
POST /_cluster/allocation/explain?pretty
{
  "index":   "logs-2026.04.15",
  "shard":   0,
  "primary": true
}

常見原因 + 修復

原因 (allocation explain)修復
NODE_LEFT + node 不會回等 timeout 自動重分配，或 reroute
NO_VALID_SHARD_COPY所有 copy 都壞 → 從 snapshot 還原；無 snapshot 則資料丟失
ALLOCATION_FAILED max retriesPOST /_cluster/reroute?retry_failed=true
CLUSTER_RECOVERED + 重啟後等 dangling indices 處理 / 等 gateway recovery
DISK_THRESHOLD exceeded清空間或加節點，調 watermark
FILTER 不匹配檢查 index.routing.allocation.include/exclude/require
AWARENESS 不滿足檢查 zone 設定，或不滿足的 shard 暫關 awareness

最後手段：放棄缺失的 primary

當 primary 全壞且無 snapshot：

POST /_cluster/reroute
{
  "commands": [
    {
      "allocate_empty_primary": {
        "index":  "broken-index",
        "shard":  0,
        "node":   "node-3",
        "accept_data_loss": true
      }
    }
  ]
}
⚠️ accept_data_loss=true 會丟失該 shard 全部資料 只在絕對沒辦法時用。先確認真的沒有 snapshot 可還原。

場景 2：Cluster YELLOW

YELLOW = primary 都好，但部分 replica unassigned。Primary 還能寫，但容錯降低。

常見原因

  1. 單節點 cluster：replica 一定 unassigned（自己不能 host 自己的 replica） → 設 "number_of_replicas": 0
  2. 節點數不夠：N replica 需 N+1 節點
  3. 磁碟達 watermark：
# Default watermark
# low: 85%, high: 90%, flood_stage: 95%
GET /_cat/allocation?v

Watermark 行為

Watermark觸發行為
low (85%)不分配新 shardYELLOW 出現
high (90%)嘗試移走 shardshard 重新分配
flood_stage (95%)對該節點 index 設 read-only寫入失敗

flood_stage 後修復

# 清資料後，解除 read-only block
PUT _all/_settings
{
  "index.blocks.read_only_allow_delete": null
}

調 watermark（暫時）

PUT /_cluster/settings
{
  "transient": {
    "cluster.routing.allocation.disk.watermark.low":         "92%",
    "cluster.routing.allocation.disk.watermark.high":        "95%",
    "cluster.routing.allocation.disk.watermark.flood_stage": "97%"
  }
}

但根本解：加磁碟 / 加節點 / ILM 刪舊資料。

場景 3：Heap 用量高 / GC 頻繁

診斷

# 看 heap 與 GC
GET /_nodes/stats/jvm?human&pretty

# 看老世代 GC 次數與時間
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_nodes/stats/jvm?human" | jq '.nodes[] | {
    name,
    heap_used_percent: .jvm.mem.heap_used_percent,
    old_gc_count: .jvm.gc.collectors.old.collection_count,
    old_gc_time:  .jvm.gc.collectors.old.collection_time
  }'

# Hot threads
GET /_nodes/hot_threads

常見原因

  • Heap 設太小：< 8 GB 容易撐不住
  • Mapping explosion：欄位太多，cluster state 過大
  • 大量小 shard：每 shard 各佔 heap
  • field data 載入過多：對 text 做 sort/agg（用 fielddata）
  • 大 bulk + 大 query：Lucene 短暫吃記憶體
  • 大量並發 search：query / fetch heap 不夠

修復

  1. 升 heap（CH 18）：到 RAM/2，最多 31 GB
  2. 清 fielddata cache：
POST /_cache/clear?fielddata=true

# 看 fielddata 用量
GET /_cat/fielddata?v
  1. 找熱欄位：
GET /_cat/fielddata?v&h=node,host,field,size&s=size:desc | head
  1. 大量小 shard 改善：用 ILM shrink + 整併
  2. 取 heap dump 分析：
jcmd $(pgrep -f elasticsearch) GC.heap_dump /var/dumps/heap.hprof
# 用 Eclipse MAT 分析

場景 4：查詢慢

診斷三劍客

# 1. Profile API
POST my-index/_search
{
  "profile": true,
  "query": { ... }
}

# 2. 啟用 slow log
PUT my-index/_settings
{
  "index.search.slowlog.threshold.query.warn": "5s",
  "index.search.slowlog.threshold.fetch.warn": "1s"
}

# 3. 看慢任務
GET /_tasks?actions=*search*&detailed=true

常見原因 + 修復

原因修復
Deep pagination（> 10000）改用 PIT + search_after（CH 37）
track_total_hits=true 但不需要設 false
Filter 用了 now（每秒不同）改用 now/d 提升 cache
OS file cache 不夠給 heap 少一點，留多 RAM 給 OS
Shard 太大（> 50 GB）rollover 切細
Cross-cluster query 跨高 latency 網路地理 region 內用 Data Locality CCR
script_score / function_score 過重改用 runtime field 或 pre-compute
漸增的小 segmentwarm phase 後 force merge
aggregation cardinality 太高用 composite + precision

場景 5：寫入慢 / 拒絕

診斷

# 看 thread pool rejected
GET /_cat/thread_pool/write?v&h=node_name,name,active,queue,rejected

# 看 indexing rate
GET /_cat/nodes?v&h=name,indexing.index_total,indexing.index_time,indexing.index_current

# 看 ingest pipeline 是否拖慢
GET /_nodes/stats/ingest?human&pretty

修復（複習 CH 49）

  • Bulk size 5-15 MB；不要單 doc 高頻
  • Replica 暫關，refresh_interval 拉長
  • 磁碟 IO 是主瓶頸 → 升 NVMe
  • Ingest pipeline 分離到專用 ingest node
  • JVM heap 不夠（write 也吃 heap）
  • Client 端用多 connection 並行

場景 6：Master 不穩 / 選舉頻繁

症狀

  • master_not_discovered_exception
  • cluster_block_exception 短暫
  • Cluster state update 慢（pending_tasks 堆積）
  • 查詢無法存取 cluster 設定

診斷

# 看當前 master
GET /_cat/master?v

# 看 pending tasks
GET /_cluster/pending_tasks
GET /_cluster/state?filter_path=cluster_uuid,master_node

# 看每節點 master log
sudo journalctl -u elasticsearch -f | grep -i master
# 或
tail -f /var/log/elasticsearch/${cluster}.log | grep -E "master|elected|stepping"

常見原因

  • master 與 data 同節點：data heavy load 拖慢 master
  • 網路不穩：master 與 follower 心跳超時
  • cluster state 過大：mapping 過多、template 過多
  • master JVM heap 不足
  • 磁碟慢：master 寫 cluster state 到 disk 慢

修復

  1. 用專用 master 節點（CH 03）：3 個 dedicated master
  2. cluster state 瘦身：合併 mapping、刪舊 templates
  3. 網路檢查：跨 AZ 延遲、防火牆 packet loss
  4. 調 election timeout（最後手段）：
cluster.election.duration: "500ms"   # 預設
cluster.election.initial_timeout: "100ms"
cluster.election.back_off_time: "100ms"
cluster.election.max_timeout: "10s"
cluster.election.strategy: "supports_voting_only"

場景 7：ILM 卡住

# 看出錯的 index
GET /_ilm/explain?only_errors=true&pretty

# 查單一
GET /my-index/_ilm/explain?pretty

# 重試
POST /my-index/_ilm/retry

# 強制推進到下個 step
POST /_ilm/move/my-index
{
  "current_step": { "phase": "warm", "action": "shrink", "name": "shrink" },
  "next_step":    { "phase": "warm", "action": "complete", "name": "complete" }
}

常見錯誤（CH 30 補充）：

  • shrink 找不到目標 node：tier_preference 與實際 node role 不符
  • snapshot repository 不存在：先建 repo
  • force_merge 失敗：磁碟空間不足或 IO 問題
  • rollover alias 缺：data stream 取代

場景 8：CCR 失敗

# 看 follower 狀態
GET /_ccr/stats?pretty

# 看單一 index
GET /follower-index/_ccr/info
GET /follower-index/_ccr/stats

常見錯誤：

  • Lag 持續增長：follower 寫入太慢，調大 max_outstanding_read_requests
  • operations not found：leader history 已過期，調 retention_lease.period
  • auto-follow 沒觸發：API key 權限不夠或 pattern 沒匹配
  • remote cluster disconnect：檢查 9443 port 與 TLS 設定

場景 9：Dangling Indices

節點上有 index 資料但 cluster state 沒登記（通常因為節點離線時 index 被刪）：

# 列出 dangling
GET /_dangling

# 引入到 cluster
POST /_dangling/{uuid}?accept_data_loss=true

# 或刪除
DELETE /_dangling/{uuid}?accept_data_loss=true

場景 10：Circuit Breaker 觸發

ES 用 circuit breaker 防止 OOM。觸發時會收到錯誤：

circuit_breaking_exception: [parent] Data too large, ...

# 看 circuit breaker 狀態
GET /_nodes/stats/breaker?human&pretty

# 各 breaker 預設
# - parent:    95% heap
# - request:   60% heap
# - fielddata: 40% heap
# - in_flight_requests: 100% heap

修復

  • Heap 升級
  • 調小 query / agg：別一次拿太多資料
  • fielddata：避免對 text 排序/聚合，改用 keyword
  • 升 breaker 上限（謹慎）：
PUT /_cluster/settings
{
  "persistent": {
    "indices.breaker.request.limit":   "70%",
    "indices.breaker.fielddata.limit": "50%"
  }
}

官方診斷工具

遇到複雜問題時，elasticsearch-support-diagnostics 可一鍵收集所有需要的資訊：

git clone https://github.com/elastic/support-diagnostics
cd support-diagnostics
./diagnostics.sh \
  --type local \
  --host localhost \
  --port 9200 \
  --user elastic \
  --password '...' \
  --ssl \
  --noVerify \
  --output /tmp/diag

產生 zip 檔含 cluster_state.json、nodes_stats.json、各 thread dump 等。Elastic Support 會請你提供。

故障排除最佳實踐

  • Pre-incident：建立 runbook，把上面常見場景寫成 SOP
  • 監控優先於救火：設好 watcher / Alerting，問題發生時自動通知
  • 留下 timeline：發生什麼、做了什麼、結果，事後做 RCA
  • 保留 cluster state 快照：發生問題時先備份 cluster state，避免修復時失去 evidence
  • 不要 panic 動 cluster setting：transient setting 重啟會消失，persistent 才永久
  • 大改前先快照：reroute、accept_data_loss、forcemerge 前都先 snapshot
  • 跑 support diagnostics：複雜問題交給 Elastic Support 時必備

本章重點回顧

  • 診斷流程：_cluster/health → _health_report → 按症狀分流
  • RED：unassigned primary，用 allocation/explain 找原因
  • YELLOW：常見是 disk watermark；flood_stage 後要解除 read-only block
  • Heap 高：先看 fielddata，再看 mapping size，最後 heap dump
  • 查詢慢三劍客：Profile API、Slow Log、Hot Threads
  • 寫入慢：先看 thread_pool rejected 與 disk IO
  • Master 不穩：用專用 master 節點 + cluster state 瘦身
  • ILM/CCR 卡：用 explain API 找原因，retry 或 move
  • Circuit breaker：避免 OOM 的最後防線
  • elasticsearch-support-diagnostics 是診斷王
