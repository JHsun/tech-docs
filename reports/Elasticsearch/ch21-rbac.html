圖 21-1：User → Role → Privilege 多對多關係

三層核心結構：

  • User（使用者）：人或服務，擁有一個或多個 roles
  • Role（角色）：權限的集合，命名通常代表業務職能
  • Privilege（權限）：具體可做的動作（讀/寫/建立 index、cluster 操作等）

權限三大類別

類別範圍例子
Cluster privileges整個叢集monitor、manage、all、manage_security
Index privileges單一 index 或 patternread、write、create、delete、view_index_metadata
Application privileges外部 app（如 Kibana）kibana 的 space + feature privileges

Cluster Privileges 速查

Privilege說明
all所有 cluster 操作
monitor讀 cluster/node 狀態（_cat、stats、health）
managecluster 管理（除 security 外）
manage_securityusers、roles、API keys 管理
manage_index_templates建立/刪 index template
manage_pipelineingest pipelines
manage_ilmILM policies
manage_mlML jobs
manage_watcherWatcher
read_ccr / manage_ccrCross-cluster replication
create_snapshot建立 snapshot（不可刪）
manage_logstash_pipelinesLogstash 中央管理

monitor_ml讀 ML jobs 但不能管

Index Privileges 速查

Privilege說明

readsearch、get、mget、count
read_cross_cluster從其他 cluster 透過 CCS 來讀

Role 定義範例

例 1：logs_writer

Beats / Logstash 寫入日誌用：

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role/logs_writer" -d '
{
  "cluster": [
    "monitor",
    "manage_index_templates",
    "manage_ilm",
    "read_pipeline"
  ],
  "indices": [
    {
      "names": [ "logs-*", ".ds-logs-*" ],
      "privileges": [
        "create",
        "create_index",
        "view_index_metadata",
        "auto_configure"
      ]
    }
  ]
}'

例 2：分析師唯讀某些 indices

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role/sales_analyst" -d '
{
  "cluster": [ "monitor" ],
  "indices": [
    {
      "names": [ "sales-*", "customer-*" ],
      "privileges": [ "read", "view_index_metadata" ]
    }
  ],
  "applications": [
    {
      "application": "kibana-.kibana",
      "privileges": [ "feature_discover.read", "feature_dashboard.read" ],
      "resources": [ "space:sales" ]
    }
  ]
}'

Document Level Security（DLS）

限制某 role 只能看到 index 中符合條件的部分文件。

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role/region_apac" -d '
{
  "indices": [
    {
      "names": [ "sales-*" ],
      "privileges": [ "read" ],
      "query": "{\"term\":{\"region\":\"APAC\"}}"
    }
  ]
}'

賦予這個 role 的使用者，搜尋 sales-* 時，只看得到 region: APAC 的文件。

用 user metadata 動態過濾

{
  "indices": [{
    "names": ["sales-*"],
    "privileges": ["read"],
    "query": {
      "template": {
        "source": "{\"term\":{\"region\":\"{{_user.metadata.region}}\"}}"
      }
    }
  }]
}

每個 user 看到的範圍不同，依據其 metadata.region。

DLS 授權注意

⚠️ DLS / FLS 需 Platinum 訂閱 Document Level Security 與 Field Level Security 是 Platinum 級功能。Basic / Gold 訂閱不可用。

Field Level Security（FLS）

限制可看到的欄位。

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role/sales_basic" -d '
{
  "indices": [
    {
      "names": [ "sales-*" ],
      "privileges": [ "read" ],
      "field_security": {
        "grant": [ "customer_id", "amount", "region", "@timestamp" ],
        "except": [ "credit_card_*" ]
      }
    }
  ]
}'
  • grant：白名單，列出可看的欄位（支援 wildcard）
  • except：黑名單，從 grant 中排除的欄位

DLS + FLS 組合

{
  "indices": [{
    "names": ["sales-*"],
    "privileges": ["read"],
    "query": "{\"term\":{\"region\":\"APAC\"}}",
    "field_security": {
      "grant": ["customer_id", "amount", "@timestamp"]
    }
  }]
}

使用者只能看到 region=APAC 的文件，且每筆只看得到指定欄位。

Role Mapping — SSO 用戶 → ES Roles

當你用 LDAP / SAML / OIDC，外部 user 沒有直接綁定 ES role。Role mapping 規則決定外部 user 取得哪些 role。

範例：依 LDAP group 對應

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role_mapping/admins" -d '
{
  "roles": [ "superuser" ],
  "enabled": true,
  "rules": {
    "all": [
      { "field": { "realm.name": "ldap1" } },
      { "field": { "groups": "cn=es-admins,ou=groups,dc=example,dc=com" } }
    ]
  }
}'

規則運算子

  • any — OR（任一 sub-rule 為真）
  • all — AND（所有 sub-rule 為真）
  • except — NOT
  • field — 比對欄位（exact、wildcard、regex）

可比對欄位

欄位說明
username使用者名稱
dnLDAP/AD distinguished name
groupsLDAP/AD group memberships（陣列）
realm.name使用的 realm name
realm.typerealm 類型（ldap、saml、oidc...）
metadata.<key>SAML attributes、OIDC claims

範例：SAML attribute 對應

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role_mapping/saml_engineers" -d '
{
  "roles": [ "developer", "kibana_user" ],
  "enabled": true,
  "rules": {
    "all": [
      { "field": { "realm.name": "saml1" } },
      { "field": { "metadata.saml(http://schemas.xmlsoap.org/claims/Group)": "Engineers" } }
    ]
  }
}'

File-based Role Mapping

另一種方式是用檔案 $ES_PATH_CONF/role_mapping.yml（每節點同步）：

superuser:
  - "cn=admins,dc=example,dc=com"
  - "cn=ops,dc=example,dc=com"

monitoring:
  - "cn=monitoring,dc=example,dc=com"

Kibana Application Privileges

Kibana 用 application privileges 細控存取：

{
  "applications": [
    {
      "application": "kibana-.kibana",
      "privileges": [
        "feature_discover.all",
        "feature_dashboard.read",
        "feature_visualize.read",
        "feature_canvas.read"
      ],
      "resources": [ "space:default", "space:marketing" ]
    }
  ]
}

常見 Kibana feature privileges

Feature說明
feature_discoverDiscover
feature_dashboardDashboards
feature_visualizeVisualize Library
feature_canvasCanvas
feature_mapsMaps
feature_mlMachine Learning
feature_dev_toolsDev Tools (Console)
feature_advancedSettingsAdvanced settings
feature_indexPatternsIndex patterns / Data views
feature_securitySecurity app
feature_observabilityObservability

每個都有 .read 與 .all 兩種 level。

Kibana Spaces（多租戶）

Kibana spaces 把 saved objects 分區（dashboards、visualizations、index patterns）。透過 role 限制可存取的 space：

{
  "applications": [
    {
      "application": "kibana-.kibana",
      "privileges": [ "all" ],
      "resources": [ "space:marketing" ]   // 只能進 marketing space
    }
  ]
}

檢查權限 API

讓 user 知道自己有哪些權限（Kibana 內部會用）：

curl -X POST -u john_doe:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/user/_has_privileges" -d '
{
  "cluster": [ "monitor" ],
  "index": [
    {
      "names": [ "logs-*" ],
      "privileges": [ "read", "write" ]
    }
  ]
}'

回應：

{
  "username": "john_doe",
  "has_all_requested": false,
  "cluster": { "monitor": true },
  "index": {
    "logs-*": { "read": true, "write": false }
  }
}

完整 Role 組合範例

案例：開發團隊每個 service 一個 role

# Backend service: 寫 logs，讀自己的 metrics
curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role/svc_orders_writer" -d '
{
  "cluster": [ "monitor" ],
  "indices": [
    {
      "names": [ "logs-orders-*" ],
      "privileges": [ "create", "create_index", "auto_configure" ]
    },
    {
      "names": [ "metrics-orders-*" ],
      "privileges": [ "read" ]
    }
  ]
}'

# 開發者: 讀所有 logs/metrics，Kibana 開發者 space
curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role/developer" -d '
{
  "cluster": [ "monitor" ],
  "indices": [
    {
      "names": [ "logs-*", "metrics-*", "traces-*" ],
      "privileges": [ "read", "view_index_metadata" ]
    }
  ],
  "applications": [
    {
      "application": "kibana-.kibana",
      "privileges": [
        "feature_discover.all",
        "feature_dashboard.all",
        "feature_visualize.all",
        "feature_dev_tools.all"
      ],
      "resources": [ "space:dev" ]
    }
  ]
}'

最佳實踐

  • Least privilege：每個 role 只給需要的權限，不貪方便給 superuser
  • 角色命名規範：用 {scope}_{action}_{resource} 格式，例如 app_read_logs
  • Separation of duties：admin / operator / developer 分開 role
  • 用 role mapping 而非直接綁 user：搭配 SSO，依 group 自動授權
  • 定期 audit：列 roles 與 mapping，移除過時的
  • 避免 role 巢狀過深：role 中互相 include 會難以追蹤
  • 系統服務帳號（如 Beats）：用 dedicated role，不要共用 superuser
  • 稽核日誌啟用，追蹤 privilege 用法（CH 24）

本章重點回顧

  • RBAC 三層：User → Role → Privilege
  • 權限三大類別：cluster、index、application（Kibana）
  • Index privileges：read、write、create、create_index、view_index_metadata 等
  • DLS / FLS 為 Platinum 功能，提供文件級與欄位級過濾
  • Role mapping 把外部認證（LDAP groups、SAML attributes）對應到 ES roles
  • Kibana 整合用 application privileges + space resources
  • 用 _has_privileges API 動態檢查當前 user 權限
  • 遵循 least privilege 原則設計 role
