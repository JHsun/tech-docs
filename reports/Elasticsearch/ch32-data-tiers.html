Data Tiers 是什麼？

Data tier 把節點依儲存 / 計算特性分組，讓 ILM 能把資料按「熱度」放到對應硬體，平衡效能與成本：

Node Roles 速覽

Role說明
data_hotHot tier，最新資料 + 寫入
data_warmWarm tier，唯讀但常查
data_coldCold tier，較少查；可掛 searchable snapshot
data_frozenFrozen tier；專為 partially-mounted snapshot
data_content非時序的一般 index（產品、user...）
data萬用 data role（小叢集用）
⚠️ data 與其他 data_* 衝突 data 是所有 data tier 的合併。如果一個節點設了 data，再加 data_hot 會啟動失敗。生產環境用 specific role（data_hot、data_warm...），不用萬用 data。

節點配置範例

Hot 節點

# elasticsearch.yml on hot-node-1
cluster.name:  production
node.name:     hot-node-1
node.roles:    [ "data_hot", "data_content", "ingest" ]
network.host:  0.0.0.0
discovery.seed_hosts: [ ... ]

# Hot 也常兼任 ingest（高 CPU 環境適合）

Warm 節點

cluster.name:  production
node.name:     warm-node-1
node.roles:    [ "data_warm" ]

Cold 節點

cluster.name:  production
node.name:     cold-node-1
node.roles:    [ "data_cold" ]

Frozen 節點（特別重要）

cluster.name:  production
node.name:     frozen-node-1
node.roles:    [ "data_frozen" ]

# Frozen 節點的 cache disk 配置
xpack.searchable.snapshot.shared_cache.size: "90%"   # 預留給 partial mount cache

Frozen 節點的本機磁碟主要當 cache，實際資料在 S3。建議 cache 大小：

cache 大小 ≈ 預期常被查的 frozen 資料量 × 0.1
例：100 TB 資料每天有 ~10% 被查 → cache ≈ 10 TB × 0.1 = 1 TB

ILM Migrate Action 自動分配 tier

ILM 進入 warm/cold/frozen phase 時，自動加入 migrate action（除非你關閉），把 shard 從上一 tier 移到下一 tier：

{
  "policy": {
    "phases": {
      "hot": { ... },
      "warm": {
        "min_age": "7d",
        "actions": {
          "migrate": { "enabled": true },   # 預設就是 true
          "shrink": { "number_of_shards": 1 }
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "searchable_snapshot": { "snapshot_repository": "primary" }
        }
      }
    }
  }
}

migrate 行為等同：

PUT my-index/_settings
{
  "index.routing.allocation.include._tier_preference": "data_warm,data_hot"
}

tier_preference 是 fallback list — 如果沒 warm 節點，會 fallback 到 hot。

_tier_preference 設定

每個 index/shard 有 _tier_preference 屬性決定它願意去哪些 tier：

Phase_tier_preference
Hotdata_hot
Warmdata_warm,data_hot
Colddata_cold,data_warm,data_hot
Frozendata_frozen（無 fallback）
Contentdata_content

手動設定（極少需要）

PUT my-index/_settings
{
  "index.routing.allocation.include._tier_preference": "data_warm,data_hot"
}

不限制（不推薦）

PUT my-index/_settings
{
  "index.routing.allocation.include._tier_preference": null
}

Searchable Snapshot 深入

Searchable snapshot 是 cold/frozen 的核心：把資料放 S3，但仍可查詢。有兩種 mount 模式：

項目Fully Mounted (Cold)Partially Mounted (Frozen)
本機儲存完整一份幾乎沒有，僅 metadata
查詢延遲類似一般 index第一次慢（從 S3 拉），cache 後快
需要 replica不需要（snapshot 即備份）不需要
儲存成本本機 + S3 ≈ 2xS3 only
可寫入❌ read-only❌ read-only
專用節點data_cold 即可data_frozen + cache disk

建立 snapshot repository（前置）

PUT _snapshot/primary
{
  "type": "s3",
  "settings": {
    "bucket":      "es-snapshots-production",
    "region":      "ap-northeast-1",
    "base_path":   "cluster1/",
    "compress":    true,
    "max_snapshot_bytes_per_sec":  "200mb",
    "max_restore_bytes_per_sec":   "200mb"
  }
}

Frozen tier shared cache 設定

Frozen 節點必設：

# elasticsearch.yml
xpack.searchable.snapshot.shared_cache.size: "90%"
# 或具體大小
# xpack.searchable.snapshot.shared_cache.size: "1tb"

# Cache 寫入路徑（建議獨立 fast disk）
path.data:
  - /var/lib/elasticsearch/data       # 一般 data
  - /mnt/cache/elasticsearch          # frozen cache（NVMe）

# 列出 snapshot 中的 indices
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_snapshot/primary/snap-2026-01/_status?pretty"

# Mount 某 index 為 partially-mounted
curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_snapshot/primary/snap-2026-01/_mount?storage=shared_cache" -d '
{
  "index":           "logs-2026-01",
  "renamed_index":   "partial-logs-2026-01",
  "index_settings": {
    "index.number_of_replicas": 0
  }
}'

完整架構設計範例

場景：每天 1 TB log，保留 90 天

計算：

  • Hot 7 天：7 TB（單 replica = 14 TB）
  • Warm 23 天：23 TB（shrunk + 壓縮 ~ 12 TB）
  • Cold 60 天：60 TB（searchable snapshot，本機 60 TB + S3 60 TB）
  • 或 Frozen 60 天：S3 60 TB only，本機 cache ~ 5 TB

節點配置

Tier節點數規格 / 節點總儲存月成本估算 (AWS)
Master (3)3m6i.xlarge, 50GB SSD—~$450
Hot4r6id.2xlarge (8 vCPU, 64 GB, 1.9 TB NVMe)~7 TB usable~$2,000
Warm2r6i.xlarge (4 vCPU, 32 GB, 4 TB gp3)~7 TB~$700
Frozen2r6i.xlarge (4 vCPU, 32 GB, 2 TB NVMe cache)cache only~$700
S3 storage——60 TB~$1,400
總計11+~$5,250/月

對比：全 hot tier 90 TB 需要 ~12 個 r6id.2xlarge，月成本 ~$6,000，且查詢遠期資料效能差。

非時序資料：data_content tier

產品目錄、用戶資料等非時序資料應該:

  • 放在 data_content 角色節點
  • 不走 ILM 的 hot/warm/cold 流程
  • 用 replicas 確保 HA
# content-node-1
node.roles: [ "data_content" ]

Index template 範例：

PUT _index_template/products
{
  "index_patterns": [ "products-*" ],
  "template": {
    "settings": {
      "number_of_shards":   2,
      "number_of_replicas": 2,
      "index.routing.allocation.include._tier_preference": "data_content"
    }
  }
}

小叢集：用 data 萬用角色

< 5 節點的小叢集不適合分 tier（每 tier 至少 2 節點才有 HA）：

node.roles: [ "data", "ingest", "master" ]

並設 ILM policy 不分 tier：

{
  "policy": {
    "phases": {
      "hot": {
        "actions": { "rollover": { "max_age": "7d" } }
      },
      "delete": {
        "min_age": "30d",
        "actions": { "delete": { } }
      }
    }
  }
}

Allocation Awareness（跨 AZ）

除了 tier，還可加上 zone awareness 確保 replica 不在同 AZ：

# 每個節點設自己的 zone
node.attr.zone: us-east-1a   # 或 1b、1c

# Cluster 設定（透過 yml 或 cluster settings）
cluster.routing.allocation.awareness.attributes: zone

結合：node 同時有 data_hot + zone=us-east-1a，allocator 會選 hot 節點之中跨 zone 分配 replica。

9.x Desired Balance Allocator

9.x 預設用 desired_balance allocator，更聰明地：

  • 計算「理想分配狀態」
  • 逐步推進到該狀態（避免大規模 shard 移動）
  • 考慮 disk usage、shard 數、tier 等多維度

無需特別配置。可看狀態：

curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_internal/desired_balance?pretty"

檢查與監控

查看 shard 分配

curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_cat/allocation?v&h=node,shards,disk.indices,disk.used,disk.avail,disk.percent,node.role"

列出某 tier 的 shard

curl -u elastic:xxx --cacert ca.crt \

Data Tier 最佳實踐

  • 先確認資料量再決定要不要分 tier：小叢集（< 5 TB）無分層必要
  • 每 tier 至少 2 節點，確保節點掛掉資料還在
  • Hot tier 用 NVMe SSD，IOPS 是寫入瓶頸
  • Warm tier 可用 gp3 / 大 HDD，順序讀寫為主
  • Cold tier 必須有 snapshot repo，否則無法用 searchable snapshot
  • Frozen tier 用 NVMe 當 cache，shared_cache.size 設 80-90%
  • 非時序資料用 data_content，獨立節點群
  • 跨 AZ 用 zone awareness + replica
  • Searchable snapshot 不需要 replica（snapshot 本身是備份）
  • 監控 _cat/allocation 看 disk 用量；warm/cold 滿了會卡 ILM

本章重點回顧

  • Data tiers 把節點按熱度分組：hot / warm / cold / frozen / content
  • 每 tier 對應 node role：data_hot、data_warm、data_cold、data_frozen、data_content
  • ILM 透過 migrate action 自動把 shard 移到對應 tier
  • Cold 用 fully-mounted snapshot（本機 + S3）；Frozen 用 partially-mounted（S3 + cache）
  • Frozen 節點需特別設 xpack.searchable.snapshot.shared_cache.size
  • 非時序資料用 data_content tier，不走 hot/warm/cold
  • 小叢集用萬用 data role；不要 mix data 與 data_hot（會啟動失敗）
  • 跨 AZ HA 用 node.attr.zone + cluster awareness 設定
  • 9.x 使用 desired_balance allocator 自動最佳化分配
