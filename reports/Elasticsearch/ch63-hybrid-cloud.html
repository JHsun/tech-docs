CH 63 · Hybrid Cloud / Air-gapped 架構

適合對象 🔴 維運者 ⚡ 進階
閱讀時間 約 25 分鐘
前置知識 CH 16 air-gapped、CH 37 CCS、CH 48 CCR
本章目標 Hybrid 架構模式、Cross-Cluster Search 跨 cloud、災備拓撲、air-gapped 完整方案、私有化部署考量

什麼情況需要 Hybrid / Air-gapped？

  • 合規：資料 sovereignty、政府機密、金融、醫療
  • 安全：核心系統不對外連網
  • 多地理區：不同 region 提供低 latency
  • 多雲容災：避免單一 cloud provider 失效
  • 過渡期：on-prem → cloud 漸進遷移
  • 法規限制：特定行業必須 on-prem

模式 1：Centralized Search（用 CCS）

多個 cluster（不同 cloud / 地理區），用 Cross-Cluster Search 統一查詢：

PUT _cluster/settings
{
  "persistent": {
    "cluster": {
      "remote": {
        "us-east": {
          "mode":          "proxy",
          "proxy_address": "us-east.example.com:9443",
          "skip_unavailable": true
        },
        "eu-west": {
          "mode":          "proxy",
          "proxy_address": "eu-west.example.com:9443"
        },
        "asia": {
          "mode":          "proxy",
          "proxy_address": "asia.example.com:9443"
        }
      }
    }
  }
}

# 查詢
GET us-east:logs-*,eu-west:logs-*,asia:logs-*/_search

單一 Kibana 看所有 cluster。每 cluster 各自 ILM、規模、tier。

模式 2：Data Locality with CCR

主寫 cluster + 多地理區 follower：

  • 使用者就近查詢 follower（低 latency）
  • 單一 leader 維護資料一致性
  • 典型場景：電商 catalog、user profile 全球分發
# Asia region follower follow leader
PUT /products-catalog/_ccr/follow
{
  "remote_cluster": "main-leader",
  "leader_index":   "products-catalog"
}

# Auto-follow 讓 backing indices 自動跟上
PUT /_ccr/auto_follow/products-pattern
{
  "remote_cluster":         "main-leader",
  "leader_index_patterns":  [ "products-*" ]
}

圖 63-2：跨雲災備拓撲

建議組合：

  • CCR：秒級 RPO，DR cluster 隨時可切
  • S3 cross-region replication：snapshot 跨地理區自動複製
  • 定期 fire drill：每季演練 failover

Failover Runbook

  1. 偵測 primary 失效（external monitoring）
  2. 把 follower index 變獨立可寫（CH 48）： 
    POST /products-catalog/_ccr/pause_follow
POST /products-catalog/_close
POST /products-catalog/_ccr/unfollow
POST /products-catalog/_open
  3. 切應用 endpoint 到 DR cluster
  4. 監控確認流量正常
  5. Primary 恢復後，反向 follow（用 reindex 或重建）

模式 4：Edge Aggregation

多個小 edge cluster + 1 個 central cluster，CCR 把資料匯集中央做分析：

  • 製造業：每工廠 1 個 cluster
  • 零售：每 region / 每店 cluster
  • IoT：edge 處理 + central 分析
edge-factory-A → CCR → central-analytics
edge-factory-B → CCR → central-analytics
edge-factory-C → CCR → central-analytics

Air-gapped 完整方案

CH 16 已介紹安裝。本節補充持續維護。

Package Mirror

內網建立鏡像服務器：

  • Linux package：reposync / pulp
  • Docker images：Harbor / Nexus / 自建 registry
  • Helm chart：Helm repo（OCI 也行）
  • Elastic 套件：從 Elastic 官方下載 .deb / .rpm / .tar.gz / Docker image

Self-hosted Elastic Package Registry (EPR)

Fleet 預設從 epr.elastic.co 下載 integrations。Air-gapped 需自架：

docker run -d \
  -p 8080:8080 \
  -e EPR_PACKAGE_PATH=/packages \
  -v /local/packages:/packages \
  docker.elastic.co/package-registry/distribution:latest

從 elastic/package-storage 下載最新 integrations。

設定 Kibana 連到自架 EPR：

xpack.fleet.registryUrl: "https://epr.internal.example.com:8080"

ML Models（air-gapped）

Hugging Face 模型需先下載到 internal 環境，用 eland 從本地匯入：

# 在有網路的機器
pip install eland
eland_import_hub_model \
  --hub-model-id "dslim/bert-base-NER" \
  --task-type ner \
  --output-dir /local/model

# 把整個資料夾搬到 air-gapped 環境

# 在 air-gapped 環境匯入
eland_import_hub_model \
  --url https://es:9200 \
  --hub-model-id "dslim/bert-base-NER" \
  --task-type ner \
  --hub-cache-dir /local/model

Self-hosted Elastic Maps Service (EMS)

Kibana Maps 預設用 tiles.maps.elastic.co。Air-gapped 自架：

docker run -d \
  -p 8080:8080 \
  -e ELASTIC_LICENSE_TYPE=basic \
  docker.elastic.co/elastic-maps-service/elastic-maps-server:9.3.4

Kibana 設定：

map.emsUrl: "https://ems.internal.example.com:8080"

定期更新流程

  1. 準備 staging 環境（可連網）
  2. 下載最新 ES、Kibana、Beats、Agent、integrations、ML models
  3. 同步到內網 mirror
  4. 內網 cluster 用 mirror 升級（CH 51 滾動升級）
  5. 記錄版本與變更

網路與安全考量

跨 region 加密

  • 所有跨 cluster traffic 必須 TLS（CH 19）
  • 用對等網路（VPC Peering / Transit Gateway / Cloud Interconnect）取代 internet
  • 跨 cloud 用 VPN 或 SD-WAN
  • 9.x remote cluster interface（9443）已支援 mTLS

API Key 策略

  • 跨 cluster 用 cross-cluster API key（CH 22, CH 48）
  • 每 region 各有獨立 root user / superuser
  • 不要共用 API key 於多 cluster

Audit Trail

  • 每 cluster 各自啟用 audit log
  • Audit 集中到專屬 monitoring cluster（避免循環）
  • 用 SIEM（CH 58）做跨 cluster 分析

跨雲成本優化

Egress 費用考量

  • AWS：跨 region 約 $0.02/GB
  • GCP：跨 region 約 $0.08-0.12/GB
  • Azure：類似
  • Cross-cloud 通常 $0.08/GB+

策略：

  • CCR 比 CCS 省：CCR 持續複製、CCS 每 query 拉資料
  • 把分析 cluster 與資料 cluster 同 region
  • 用 cold tier + searchable snapshot（CH 30）：S3 比 EBS 便宜
  • 跨 region replication 走 S3（無 egress 費用，前提：用 S3 CRR）

Terraform / IaC for Hybrid

用 Terraform 管理跨雲 cluster：

# Provider 多家
provider "aws"   { region = "us-east-1" }
provider "google" { project = "my-prj" }
provider "kubernetes" { ... }

# AWS EKS cluster + ECK
module "ecs_aws" {
  source       = "./modules/eck"
  cluster_name = "production-aws"
  region       = "us-east-1"
  ...
}

# GCP GKE + ECK
module "ecs_gcp" {
  source       = "./modules/eck"
  cluster_name = "production-gcp"
  region       = "us-central1"
  ...
}

# Cross-cluster setup
resource "elasticsearch_cluster_settings" "remote_aws" {
  persistent = {
    "cluster.remote.aws.proxy_address" = "..."
  }
}

Hybrid 監控

  • 專用 monitoring cluster（CH 46）：建議放在中立位置（如管理 region）
  • 每 cluster 用 Elastic Agent 送 metrics 到 monitoring cluster
  • 外部 uptime monitor（Synthetics CH 57）：從每地理區 ping 每 cluster
  • SLO + burn rate alert（CH 57）：跨 cluster 整合

最佳實踐

  • 從業務驅動設計：合規、地理、災備需求先列清楚
  • 用 IaC：Terraform / Ansible / Helm，可跨環境重複
  • 跨 cluster 用 API key + remote_cluster_client role
  • Mapping / templates / ILM 用 GitOps 同步到所有 cluster
  • Egress 流量是隱形成本，CCR 通常比 CCS 經濟
  • Air-gapped 必備：self-hosted EPR、EMS、package mirror、ML model cache
  • 專用 monitoring cluster：避免循環依賴
  • 定期 fire drill：跨 region failover、air-gapped 升級
  • SLO 定義跨 cluster：region availability、cross-region 一致性
  • Backup 多層：CCR（即時）+ snapshot（長期）+ cross-region replication

本章重點回顧

  • Hybrid 架構五大模式：centralized search、data locality、DR、edge aggregation、air-gapped
  • CCS 跨 cluster 即時查詢；CCR 跨 cluster 即時複製
  • 跨雲 DR 用 CCR + cross-region snapshot 雙保險
  • Failover runbook 必須演練：unfollow → 切流量
  • Air-gapped 補充：self-hosted EPR、EMS、ML model 本地匯入
  • 跨 region traffic 用 VPC peering / Cloud Interconnect
  • 9.x remote cluster interface（9443）+ cross-cluster API key 是新標準
  • Egress 費用是隱形成本，CCR 比 CCS 經濟
  • Terraform / IaC 是 hybrid 管理的必備工具
