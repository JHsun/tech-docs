OpenTelemetry 是什麼？

OpenTelemetry (OTel) 是 CNCF 觀測性標準，提供：

  • Specification：semantic conventions（欄位命名規範）
  • API：應用內呼叫
  • SDK：各語言實作
  • Collector：vendor-neutral data pipeline
  • OTLP：統一傳輸協議（gRPC / HTTP）

核心訴求：vendor neutral——一次 instrument，可送任何 backend（Elastic、Datadog、Honeycomb、Jaeger、Prometheus、Grafana...）。

EDOT：Elastic Distribution of OpenTelemetry

Elastic 提供 OTel 的定制發行版，包含：

  • EDOT Collector：含 Elastic-specific receivers / processors / exporters
  • EDOT SDKs：各語言（Java、Node.js、Python、Go、.NET、PHP、iOS、Android）的 OTel SDK 預配置版
  • 與 OTel 完全相容，可與其他 OTel 元件混用
✅ EDOT 與 contrib OTel 的關係 EDOT 是 OTel contrib distribution 的超集。已有 contrib OTel 部署可直接送 ES，使用 EDOT 只是體驗更順、設定更簡。Vendor lock-in 是0。

Edge Collector 的角色

Edge collector（host 上跑、K8s 上 DaemonSet）做：

  • 收 host log（filelog receiver）
  • 收 host metrics（hostmetrics receiver）
  • 接 application 的 OTLP（從 SDK / Beat / Prometheus）
  • 用 resource attribute 補欄位（host.name、k8s.pod.name）
  • 送到 OTLP backend

Hostmetrics Receiver 範例

receivers:
  hostmetrics:
    collection_interval: 10s
    scrapers:
      cpu: {}
      memory: {}
      load: {}
      filesystem: {}
      network: {}
      process: {}

  filelog:
    include: [ "/var/log/*.log" ]
    operators:
      - type: regex_parser
        regex: '^(?P\S+) (?P\S+) (?P.*)$'

  otlp:
    protocols:
      grpc: { endpoint: "0.0.0.0:4317" }
      http: { endpoint: "0.0.0.0:4318" }

processors:
  resourcedetection:
    detectors: [ env, system, ec2, gcp, azure, k8snode ]

  batch:
    send_batch_size: 1024
    timeout:         10s

exporters:
  otlp/elastic:
    endpoint: "https://otlp.elastic-cloud.com:443"
    headers:
      Authorization: "ApiKey ${ES_API_KEY}"

service:
  pipelines:
    metrics:
      receivers:  [ hostmetrics, otlp ]
      processors: [ resourcedetection, batch ]
      exporters:  [ otlp/elastic ]
    logs:
      receivers:  [ filelog, otlp ]
      processors: [ resourcedetection, batch ]
      exporters:  [ otlp/elastic ]
    traces:
      receivers:  [ otlp ]
      processors: [ resourcedetection, batch ]
      exporters:  [ otlp/elastic ]

EDOT SDK Auto-instrumentation

Java（最簡）

java \
  -javaagent:/path/to/elastic-otel-javaagent.jar \
  -Dotel.service.name=order-service \
  -Dotel.exporter.otlp.endpoint=http://edge-collector:4318 \
  -Dotel.resource.attributes=deployment.environment=production \
  -jar app.jar

EDOT Java agent 是 OTel Java agent + Elastic optimizations（如 inferred span detection）。

Node.js

npm install @elastic/opentelemetry-node

# 用 --import 啟動（Node 20+）
node --import @elastic/opentelemetry-node/setup app.js

# 或環境變數
export OTEL_SERVICE_NAME=my-service
export OTEL_EXPORTER_OTLP_ENDPOINT=http://edge-collector:4318
node --import @elastic/opentelemetry-node/setup app.js

Python

pip install elastic-opentelemetry

# Bootstrap auto-instrumentation
edot-bootstrap

# 啟動應用
OTEL_SERVICE_NAME=my-service \
OTEL_EXPORTER_OTLP_ENDPOINT=http://edge-collector:4318 \
opentelemetry-instrument python app.py

Go

// Go 沒有 magic auto-instrumentation，需 contrib instrumentations
import (
  "go.opentelemetry.io/contrib/instrumentation/net/http/otelhttp"
  "go.opentelemetry.io/otel/sdk/trace"
)

func main() {
  tp := initTracer()
  defer tp.Shutdown(ctx)

  http.Handle("/", otelhttp.NewHandler(http.HandlerFunc(handler), "root"))
  http.ListenAndServe(":8080", nil)
}

Managed OTLP Endpoint（Cloud / Serverless）

Elastic Cloud / Serverless 提供 fully-managed OTLP 接收端：

export OTEL_EXPORTER_OTLP_ENDPOINT="https://<motlp-endpoint>.apm.elastic.cloud:443"
export OTEL_EXPORTER_OTLP_HEADERS="Authorization=ApiKey <your-api-key>"

特性：

  • 無需自架 collector gateway
  • 自動 scaling
  • 原生 OTel 格式儲存（保留 semantic conventions）
  • Cloud Hosted 9.0+ 與 Serverless 都支援

Gateway Collector 設定

receivers:
  otlp:
    protocols:
      grpc: { endpoint: "0.0.0.0:4317" }
      http: { endpoint: "0.0.0.0:4318" }

processors:
  batch:
    send_batch_size: 8192
    timeout:         5s

  # Elastic-specific 處理
  elastictrace:
    {}

  elasticinframetrics:
    {}

exporters:
  elasticsearch:
    endpoints: [ "https://es:9200" ]
    api_key:   "${ES_API_KEY}"
    tls:
      ca_file: "/certs/ca.crt"
    mapping:
      mode: "otel"      # 9.x 推薦：保留 OTel semantic 欄位

service:
  pipelines:
    traces:
      receivers:  [ otlp ]
      processors: [ elastictrace, batch ]
      exporters:  [ elasticsearch ]
    metrics:
      receivers:  [ otlp ]
      processors: [ elasticinframetrics, batch ]
      exporters:  [ elasticsearch ]
    logs:
      receivers:  [ otlp ]
      processors: [ batch ]
      exporters:  [ elasticsearch ]

Mode說明

Kubernetes EDOT 部署

用官方 Helm chart：

helm repo add open-telemetry https://open-telemetry.github.io/opentelemetry-helm-charts

# values.yaml
cat > values.yaml << 'EOF'
opentelemetry-operator:
  manager:
    collectorImage:
      repository: docker.elastic.co/observability/elastic-agent

defaultCRConfig:
  enabled: true
  image:
    repository: docker.elastic.co/elastic-agent/elastic-agent
    tag: 9.3.4

  # DaemonSet collector for node-level metrics + logs
  presets:
    hostMetrics:        { enabled: true }
    kubeletMetrics:     { enabled: true }
    logsCollection:     { enabled: true }
    kubernetesAttributes: { enabled: true }

  config:
    exporters:
      otlp/elastic:
        endpoint: "${ES_OTLP_ENDPOINT}"
        headers:
          Authorization: "ApiKey ${ES_API_KEY}"

# Cluster-level Deployment for kube-state-metrics
clusterCollector:
  enabled: true
  presets:
    clusterMetrics:     { enabled: true }
EOF

helm install edot-stack \
  -f values.yaml \
  open-telemetry/opentelemetry-kube-stack \
  --namespace observability \
  --create-namespace

會部署：

  • OpenTelemetry Operator：管理 Collector CRD
  • DaemonSet Collector：每 node 一個（host metrics + container logs）
  • Deployment Collector：1 個（cluster-level metrics from kube-state-metrics）
  • Instrumentation CRD：給 application 自動 inject SDK

應用自動 instrumentation

對 pod 加 annotation，operator 自動注入 SDK：

apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    metadata:
      annotations:
        instrumentation.opentelemetry.io/inject-java: "observability/elastic-instrumentation"
    spec:
      containers:
        - name: app
          image: my-app:latest

Operator 會 inject Java agent 到 init container 並設環境變數。應用無需改 code 即獲得 trace。

OTel Collector vs Elastic Agent 的關係

9.x 兩條路線並行：

路線優勢適合
Elastic Agent + FleetUI 中央管理、prebuilt integrations、Elastic Defend、policy 推送Elastic-centric 部署、需要 Fleet 管理 + 安全功能
EDOT Collectorvendor-neutral、CNCF 生態、可同時送多 backend需要標準化 OTel、多 backend、混合 vendor

9.x Elastic Agent 內建 EDOT collector，是同一個 binary 兩種運作模式。Fleet UI 也可管理 OTel Collector（technical preview）。

OTel 資料儲存（9.x）

9.x 用 OTel-native data streams：

  • traces-otel.{dataset}-{namespace}
  • metrics-otel.{dataset}-{namespace}
  • logs-otel.{dataset}-{namespace}

欄位保留 OTel semantic conventions（如 service.name、net.peer.ip），與 ECS 並存。

OTel Content Packs（9.x）

Kibana → Integrations → 搜尋 "otel" 找專為 OTel 資料設計的 dashboard pack：

  • System OpenTelemetry Assets
  • Kubernetes OpenTelemetry Assets
  • Nginx OTel
  • JVM OTel

每週 Elastic 持續加入新 pack。

APM Server 與 OTLP 的關係

9.x 兩種接收 trace 路徑：

  • APM Server（Elastic Agent + APM integration）：接 Elastic APM agent + OTLP（向後相容）
  • OTLP 直送 ES：用 EDOT Collector → managed OTLP / gateway collector

方向：APM Server 漸退場，OTLP 成主流。新部署優先 OTLP。

9.x EDOT 獨家功能

  • elastictrace processor：產生 inferred spans、自動 derive transactions
  • elasticinframetrics processor：把 OTel host metrics 轉成 Elastic-specific 格式
  • apmconfig extension：支援中央 APM 配置（rate limit、sampling）
  • OpAMP extension：（9.x tech preview）用 OpAMP 協議遠端管理 OTel Collector
  • LLM Observability：OTel GenAI semantic conventions 支援，追蹤 LLM call

OTel 最佳實踐

  • 新部署優先 OTel，避免 vendor lock-in
  • Cloud / Serverless 用 Managed OTLP Endpoint（最簡）
  • Self-managed 必須 Gateway Collector（不要 edge 直送 ES）
  • Edge collector 用 EDOT（含 Elastic 優化），或 contrib + elasticsearch exporter
  • Mapping mode = otel（9.x 推薦），保留語義
  • 用 Helm kube-stack chart 部署 K8s
  • auto-instrumentation 優先，必要時補 manual span
  • service.name + deployment.environment + service.version 三件套必設
  • resource attribute 補（host、k8s、cloud），讓 trace 可關聯到 infra
  • Sampling 用 tail-based（在 gateway 處理）
  • 批次處理：edge collector 用 batch processor 減少 backend 壓力
  • OTel SDK 與 Elastic APM agent 不要混用（同 application）

本章重點回顧

  • OpenTelemetry 是 CNCF vendor-neutral 觀測標準（API/SDK/Collector/OTLP）
  • EDOT 是 Elastic 定制發行版，與 contrib OTel 完全相容
  • 架構：Edge Collector + Backend（Managed OTLP / Gateway）
  • Cloud / Serverless 用 Managed OTLP Endpoint；Self-managed 必須 Gateway Collector
  • EDOT SDK 提供 Java/Node/Python/Go/.NET/PHP/iOS/Android auto-instrumentation
  • K8s 用 opentelemetry-kube-stack Helm chart：DaemonSet + Deployment + Operator
  • Mapping mode 選 otel（9.x 推薦）保留 semantic conventions
  • 9.x 兩路並行：Elastic Agent（UI 管理）vs EDOT Collector（vendor-neutral）
  • OTel 資料存 traces-otel.*、metrics-otel.*、logs-otel.* data streams
  • 新部署優先 OTLP；APM Server 漸退場
  • EDOT 獨家：elastictrace、elasticinframetrics、OpAMP、LLM Observability
