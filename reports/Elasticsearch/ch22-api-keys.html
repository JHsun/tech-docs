CH 22 · API Keys 與 Service Accounts

適合對象 🟡 開發者 🔴 維運者
閱讀時間 約 22 分鐘
前置知識 CH 20、CH 21
本章目標 用 API Keys 與 Service Accounts 取代密碼、設計適當權限、輪替策略、跨叢集 API key

為何服務間連線不該用 username/password？

  • 密碼通常很長壽，外洩後影響大；API key 短期且可隨時 revoke
  • 密碼必須在 client 安全儲存；API key 可給最小權限與 scope
  • 密碼難以批次管理；API key 可依用途/服務獨立追蹤
  • 密碼登入需 BCRYPT 驗證，較慢；API key 用 hash lookup 快得多

API Key 概念

API key 是由 user 建立的「子身份」，繼承或受限於建立者的權限：

  • 建立 API key 的 user 必須有對應權限
  • API key 的權限不能超過建立者本身（必要時可進一步收窄）
  • 每個 API key 有唯一 ID + secret，client 用 base64(id:secret) 認證
  • 可設定 expiration、metadata
  • 建立者被 revoke 時，其產生的 key 也失效

建立 API Key

前置：啟用 API key service

9.x 預設已啟用。確認 yml：

xpack.security.authc.api_key.enabled: true

建立基本 API key

curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/api_key" -d '
{
  "name": "filebeat-prod-2026",
  "expiration": "30d",
  "role_descriptors": {
    "filebeat_writer": {
      "cluster": [
        "monitor",
        "read_ilm",
        "read_pipeline"
      ],
      "index": [
        {
          "names": [ "filebeat-*", ".ds-filebeat-*" ],
          "privileges": [ "create", "create_index", "auto_configure" ]
        }
      ]
    }
  },
  "metadata": {
    "application": "filebeat",
    "environment": "production",
    "owner": "ops-team"
  }
}'

回應（secret 只會顯示這一次，務必記下）：

{
  "id": "VuaCfGcBCdbkQm-e5aOx",
  "name": "filebeat-prod-2026",
  "expiration": 1735689600000,
  "api_key": "ui2lp2axTNmsyakw9tvNnw",
  "encoded": "VnVhQ2ZHY0JDZGJrUW0tZTVhT3g6dWkybHAyYXhUTm1zeWFrdzl0dk5udw=="
}
🚨 secret 只顯示一次！ api_key 與 encoded 欄位無法事後查詢。記不到只能刪掉重建。建議：建立後立即存入 secrets manager（HashiCorp Vault、AWS Secrets Manager...）。

使用 API Key 認證

方式 1：encoded（base64）

# 用 encoded 直接放 Authorization header
curl -H "Authorization: ApiKey VnVhQ2ZHY0JDZGJrUW0tZTVhT3g6dWkybHAyYXhUTm1zeWFrdzl0dk5udw==" \
  --cacert ca.crt \
  "https://localhost:9200/_security/_authenticate?pretty"

方式 2：自行 base64 編碼

API_ID="VuaCfGcBCdbkQm-e5aOx"
API_KEY="ui2lp2axTNmsyakw9tvNnw"
ENCODED=$(echo -n "$API_ID:$API_KEY" | base64 -w0)

curl -H "Authorization: ApiKey $ENCODED" --cacert ca.crt \
  "https://localhost:9200/"

Filebeat 範例

# filebeat.yml
output.elasticsearch:
  hosts: ["https://es-node-1.internal:9200"]
  api_key: "VuaCfGcBCdbkQm-e5aOx:ui2lp2axTNmsyakw9tvNnw"
  ssl.certificate_authorities: ["/etc/filebeat/ca.crt"]

Logstash 範例

output {
  elasticsearch {
    hosts => ["https://es-node-1.internal:9200"]
    api_key => "VuaCfGcBCdbkQm-e5aOx:ui2lp2axTNmsyakw9tvNnw"
    ssl_enabled => true
    ssl_certificate_authorities => "/etc/logstash/ca.crt"
  }
}

Java SDK 範例

import co.elastic.clients.transport.rest5_client.low_level.Rest5Client;

String apiKey = "VnVhQ2ZHY0JDZGJrUW0tZTVhT3g6dWkybHAyYXhUTm1zeWFrdzl0dk5udw==";

Rest5Client restClient = Rest5Client
    .builder(new HttpHost("https", "es-node-1", 9200))
    .setDefaultHeaders(new Header[]{
        new BasicHeader("Authorization", "ApiKey " + apiKey)
    })
    .build();

Python SDK 範例

from elasticsearch import Elasticsearch

client = Elasticsearch(
    "https://es-node-1.internal:9200",
    api_key=("VuaCfGcBCdbkQm-e5aOx", "ui2lp2axTNmsyakw9tvNnw"),
    ca_certs="/path/to/ca.crt"
)

print(client.info())

管理 API Keys

列出 keys

# 自己擁有的 keys
curl -X GET -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/api_key?owner=true"

# 所有 keys（需要 manage_api_key cluster privilege）
curl -X GET -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/api_key"

# 篩選 active keys
curl -X GET -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/api_key?active_only=true"

用 query 找 key

curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/_query/api_key" -d '
{
  "query": {
    "bool": {
      "must": [
        { "term": { "metadata.environment": "production" } }
      ]
    }
  },
  "sort": [ { "creation": "desc" } ],
  "size": 100
}'

作廢（revoke）keys

# 用 ID
curl -X DELETE -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/api_key" -d '
{
  "ids": [ "VuaCfGcBCdbkQm-e5aOx" ]
}'

# 用 name
curl -X DELETE -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/api_key" -d '
{
  "name": "filebeat-prod-2026"
}'

# revoke 某個 user 所有的 keys
curl -X DELETE -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/api_key" -d '
{
  "username": "filebeat_user",
  "owner": false
}'

更新 key（變權限或 metadata）

curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/api_key/VuaCfGcBCdbkQm-e5aOx" -d '
{
  "role_descriptors": {
    "filebeat_writer": {
      "cluster": [ "monitor", "read_ilm" ],
      "index": [
        { "names": [ "filebeat-*" ], "privileges": [ "create" ] }
      ]
    }
  },
  "metadata": { "rotated_at": "2026-04-15" }
}'

優雅輪替（grace overlap）

避免「key 換了 client 還沒拿到」的空檔：

  1. 建立新 key
  2. 把新 key 部署到 client（重啟 / hot reload）
  3. 等所有 client 都用新 key（看 audit log 中的 key id）
  4. 再 invalidate 舊 key

Cross-Cluster API Key（跨叢集）

9.x 推出的特性。用於 Cross-Cluster Search (CCS) 與 Cross-Cluster Replication (CCR)，比舊的 cert-based 認證更彈性。

在 leader cluster 建立

curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://leader.example.com:9200/_security/cross_cluster/api_key" -d '
{
  "name": "ccr-from-dr-cluster",
  "access": {
    "search": [
      { "names": ["logs-*", "metrics-*"] }
    ],
    "replication": [
      { "names": ["logs-*"] }
    ]
  },
  "metadata": { "purpose": "DR replication" }
}'

在 follower 設定 remote cluster

# 把 encoded api key 存入 keystore
sudo /usr/share/elasticsearch/bin/elasticsearch-keystore add \
  cluster.remote.leader-cluster.credentials
# 貼上 encoded value

# 設定遠端
curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://follower.example.com:9200/_cluster/settings" -d '
{
  "persistent": {
    "cluster": {
      "remote": {
        "leader-cluster": {
          "mode": "proxy",
          "proxy_address": "leader.example.com:9443"
        }
      }
    }
  }
}'

Service Accounts

9.x 推出 service accounts（內建特殊用途的身份），用於 Elastic Stack 自家組件。每個 service account 必須由 service tokens 認證。

內建 service accounts

Service Account用途
elastic/fleet-serverFleet Server
elastic/kibanaKibana 連線（替代 kibana_system user）
elastic/enterprise-search-serverEnterprise Search
elastic/auto-opsAutoOps Agent

建立 service token

# 用 API
curl -X POST -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/service/elastic/kibana/credential/token/my-kibana-1?pretty"

回應：

{
  "created": true,
  "token": {
    "name": "my-kibana-1",
    "value": "AAEAAWVsYXN0aWMva2liYW5hL215LWtpYmFuYS0xOnYzOXBu..."
  }
}

Kibana 使用 service token

# kibana.yml
elasticsearch.serviceAccountToken: "AAEAAWVsYXN0aWMva2liYW5hL215LWtpYmFuYS0xOnYzOXBu..."

# 不要再寫 elasticsearch.username / .password

或用 CLI 建立

sudo /usr/share/elasticsearch/bin/elasticsearch-service-tokens create \
  elastic/kibana my-kibana-1
# Token 會輸出到畫面，存入 $ES_PATH_CONF/service_tokens 檔案

管理 service tokens

# 列出
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/service/elastic/kibana/credential?pretty"

# 刪除
curl -X DELETE -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/service/elastic/kibana/credential/token/my-kibana-1"

API Key vs Service Token：何時用哪個？

項目API KeyService Token
用途一般 client / appElastic Stack 自家組件
權限自訂 role descriptors固定（service account 已定義）
Expiration可設定永久（除非 revoke）
建立者任何 user需 manage_service_account
適合Beats、Logstash、自家 appKibana、Fleet Server

Cache 與 TTL 設定

# elasticsearch.yml
xpack.security.authc.api_key.cache.ttl: 24h    # 預設 24 小時
xpack.security.authc.api_key.cache.max_keys: 10000
xpack.security.authc.api_key.hashing.algorithm: pbkdf2  # 預設

Revoke 後 cache 還在的窗口為 ttl。要立即生效可手動 clear：

curl -X POST -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/api_key/_clear_cache"

最佳實踐

  • 每個服務獨立 key：方便細分流量歸屬與單獨 revoke
  • 給最小權限：只給該服務需要的 cluster + index privilege
  • 設定 expiration：30-90 天，配合自動輪替
  • 命名規範：{service}-{env}-{date} 例如 filebeat-prod-2026q1
  • metadata 記錄擁有者：app、team、purpose
  • 機密存放：用 secrets manager，不入 git、不寫 docker image
  • 監控使用：稽核日誌（CH 24）追蹤 key 使用狀況
  • 設定到期警告：用 watcher 在 key 7 天內到期時通知
  • 離職員工：立即 invalidate 該 user 創建的所有 key

本章重點回顧

  • API Key 取代密碼是現代 ES 推薦做法：可獨立 revoke、設 expiration、限權限
  • 建立 API key 時 secret 只會顯示一次，務必立刻存入 secrets manager
  • 使用方式：HTTP header Authorization: ApiKey <encoded>
  • 輪替策略採 grace overlap：先建新、確認 client 切換、再 revoke 舊
  • Cross-Cluster API Key 是 9.x 跨叢集認證的標準做法（替代舊的 cert-based）
  • Service Account 給 Kibana、Fleet Server 等 Elastic 自家組件用
  • Service Account 用 service token 認證，比 kibana_system user + 密碼更安全
  • API key cache TTL 預設 24h，revoke 後 cache 內仍可用，必要時 clear cache
