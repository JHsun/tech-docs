CH 03 · 分散式架構與 Node Roles

適合對象 🔴 維運者 ⚡ 進階
閱讀時間 約 25-30 分鐘
前置知識 CH 02 · 核心概念
本章目標 理解 Elasticsearch 叢集的分散式架構、所有節點角色（Node Roles）的職責與規劃方式，掌握生產環境的拓撲設計

叢集（Cluster）的本質

一個 Elasticsearch cluster 是由一個或多個 nodes（節點） 組成的群體。它們透過內部協定（transport protocol，預設 port 9300）互相通訊，共同對外提供服務。

關鍵性質：

  • 每個 cluster 由 cluster.name 識別。同名的 nodes 才會自動加入
  • 每個 node 由 node.name 識別（預設為機器 hostname）
  • 所有節點都儲存一份完整的 cluster state，但只有 master 可以變更它
  • 外部 client 可以連接到任何節點發送請求，內部會自動路由到正確的位置

Cluster State 是什麼？

Cluster state 是叢集的「記憶」，包含整個叢集的後設資料（metadata），由 master node 維護：

  • 所有 indices 的 settings、mappings、aliases
  • 每個 shard 的位置（routing table）
  • 所有 nodes 的角色與健康狀態
  • Templates、ILM policies、Snapshot repos、Pipelines、Roles 等

Cluster state 變更時（例如新建 index、新增節點、shard 重新分配），master 會把新版本廣播給所有 nodes。每個 node 會把 state 持久化到磁碟（路徑由 path.data 控制）。

⚠️ Cluster state 不應太大 Cluster state 會被廣播到每個節點且全部載入記憶體。當 indices 數量過多（例如數萬個 daily indices）或 mappings 過於龐大（單一 index 上千個欄位）時，cluster state 會膨脹，導致 master 變更速度劇減、記憶體壓力增加，進而拖垮整個叢集。這也是為什麼要使用 Data Streams + ILM 來控制 indices 數量。

Node Roles（節點角色）

Elasticsearch 的每個節點可以同時擔任多種角色（roles），由 node.roles 設定。這是 ES 生產部署最重要的觀念之一。

Role 代號角色名稱職責
masterMaster-eligible有資格被選為 master，管理 cluster state
dataData儲存資料、執行索引與搜尋
data_contentContent data儲存「內容類」資料（products、users 等）
data_hotHot tier儲存最新的時序資料，I/O 最密集
data_warmWarm tier儲存較舊資料，唯讀為主
data_coldCold tier儲存更舊資料，可用 searchable snapshot
data_frozenFrozen tier長期保留，從 snapshot 直接搜尋（搜尋慢但成本最低）
ingestIngest執行 ingest pipelines（資料寫入前的轉換）
mlMachine Learning執行 ML 工作（anomaly detection、NLP inference）
remote_cluster_clientRemote cluster client能連接到其他 ES cluster（CCS / CCR）
transformTransform執行 Transform 工作（資料聚合與重塑）
voting_onlyVoting-only master只投票、不能被選為 master（用於奇數節點維持）

預設行為

如果你沒有在 elasticsearch.yml 設定 node.roles，預設值為：

node.roles: [ master, data, data_content, data_hot, data_warm, data_cold, data_frozen, ingest, ml, remote_cluster_client, transform ]

意思是每個節點都是「全能型」。這在開發或小型叢集很方便，但生產環境通常需要分離角色以提升可靠性與效能。

深入各角色

Master-eligible Node

負責管理 cluster state 的變更操作：建立/刪除 index、追蹤節點加入/離開、shard allocation 決策等。

Master-eligible 節點群中，會透過選舉（election）產生一個active master。其他 master-eligible 節點處於 standby，隨時準備接手。

node.roles: [ master ]
node.name: master-1
💡 Master 節點配置建議
  • 生產叢集需要 3 個專用 master-eligible 節點（奇數，避免 split-brain）
  • 專用 master 節點不需要強大硬體，但需要：低延遲網路、足夠記憶體（通常 4-8 GB heap）、SSD 儲存 cluster state
  • 不要把 master 與 data 混在同一節點上 — 大型叢集 data node 的高 I/O 會影響 master 的 cluster state 廣播效能

Data Node 與 Data Tiers

Data node 負責儲存 shards 與執行索引、搜尋、聚合。從 7.10 起，Elastic 引入 Data Tiers 概念（CH 33 詳述）：

圖 3-2：Data Tiers 由熱到冷的儲存階層

典型設定：

# Hot tier 節點
node.roles: [ data_hot, data_content, ingest ]

# Warm tier 節點
node.roles: [ data_warm ]

# Cold tier 節點
node.roles: [ data_cold ]

# Frozen tier 節點（記憶體與快取為主，磁碟需求小）
node.roles: [ data_frozen ]

ILM policy 會自動把資料從 hot 移轉到 warm/cold/frozen。詳見 CH 32 / CH 33。

Ingest Node

專門執行 ingest pipelines：在 document 真正寫入 shard 之前，套用一系列 processors（grok parsing、欄位轉換、enrich、geoip 等）。詳見 CH 59。

node.roles: [ ingest ]

讓專用 ingest node 處理重 CPU 的解析工作，可以避免拖累 data node 的索引/搜尋效能。但若 pipeline 簡單、資料量不大，data node 兼任 ingest 亦可。

Coordinating-only Node（協調節點）

沒有設定任何 role 的節點，只負責接收外部請求並協調，不儲存資料、不參與 master 選舉、不執行 ingest。

node.roles: [ ]    # 空陣列 = coordinating-only

它在執行搜尋時的角色：

  1. 接收 client 的 search 請求
  2. 把 request 分發到所有相關 data nodes（query phase）
  3. 合併各 data node 回傳的 hits、計算最終排序（fetch phase）
  4. 回傳結果給 client

大型叢集（特別是聚合密集場景）通常會獨立 1-3 個 coordinating-only 節點作為 client 的負載入口，配合 load balancer 使用。

ML Node

執行所有 Machine Learning 工作，包含：

  • Anomaly Detection jobs
  • Data Frame Analytics（regression、classification、outlier detection）
  • NLP 模型部署（ELSER、E5、自訂 BERT 模型）
  • Inference API 呼叫
node.roles: [ ml, remote_cluster_client ]

ML 工作對記憶體需求高（特別是 NLP 模型載入），通常需要獨立節點配置足夠 RAM。詳見 CH 62。

💡 注意：ML 是付費功能 Machine Learning 屬於 Platinum / Enterprise 訂閱。Basic 免費版本無法使用 anomaly detection 與 DFA，但Inference API（含 ELSER）在 Basic 中可用。

Transform Node

執行 Transform：把一個 source index 透過 pivot 或 latest 聚合，產生另一個摘要 index。例如：把每分鐘的 raw logs 轉成「每小時、每 host 的 error count」摘要 index，加速 dashboard 查詢。詳見 CH 65。

node.roles: [ transform ]

Voting-only Node

這是一個特殊角色：節點可以投票參與 master 選舉，但本身不會被選為 master。用於三節點以上、奇數規模的叢集，作為廉價的 tie-breaker。

node.roles: [ master, voting_only ]

典型情境：兩個強大的 master 節點 + 一個小規格的 voting-only 節點，達成法定人數（quorum）但成本最低。

生產拓撲範例

小型叢集（3 nodes，全能型）

適合 PoC、開發環境、小流量正式服務（每日 < 10 GB 資料、查詢 < 100 qps）：

Node 1, 2, 3：master + data + ingest + ml + transform + remote_cluster_client
（即使用預設 node.roles）

規格建議：每節點 8-16 vCPU、32-64 GB RAM（heap 16-31 GB）、500 GB-1 TB SSD。

中型叢集（7 nodes，分離角色）

適合每日 100 GB 級別、生產服務：

數量角色建議規格
3master-only4 vCPU / 8 GB RAM / 100 GB SSD
3data + ingest16 vCPU / 64 GB RAM / 2 TB NVMe
1coordinating-only8 vCPU / 32 GB RAM

大型叢集（Tiered，10+ nodes）

適合 PB 級時序資料（observability / SIEM）：

數量角色建議規格
3master-only8 vCPU / 16 GB RAM / SSD 200 GB
6+data_hot + ingest32 vCPU / 128 GB RAM / NVMe 4 TB
4+data_warm16 vCPU / 64 GB RAM / SSD 8 TB

節點發現與叢集啟動（Discovery & Bootstrap）

新節點如何找到既有叢集？這是discovery 機制。最常見的設定：

# 列出可用於 master 選舉的節點 hostname/IP
discovery.seed_hosts:
  - master-1.example.com
  - master-2.example.com
  - master-3.example.com

# 首次啟動叢集時，明確列出最初的 master-eligible 節點（避免 split-brain）
cluster.initial_master_nodes:
  - master-1
  - master-2
  - master-3
🚨 cluster.initial_master_nodes 只在「首次 bootstrap」時使用 叢集形成後就要把這個設定移除。如果不移除，重啟單一節點時可能誤觸 bootstrap 流程造成 data loss。詳見 CH 04。

Shard Allocation 基本概念

Master 節點透過 allocation deciders 決定每個 shard 該放在哪個 data node。考量因素：

  • 磁碟空間水位：低於 85%（low）允許新 shard，超過 90%（high）禁止新 shard，超過 95%（flood_stage）強制 read-only
  • Allocation awareness：可設定機架/區域感知，避免同一 shard 的 primary 與 replica 在同一機架
  • Filter 規則：透過 index settings 強制 shard 分配到特定節點（例如 data tier）
  • Same-shard rule：primary 與其 replicas 永遠不在同一節點
  • Disk threshold：定期再平衡（rebalance）負載

檢視當前 allocation 狀態：

curl -X GET "https://localhost:9200/_cluster/allocation/explain?pretty" \
  -u elastic:CHANGEME \
  --cacert /path/to/http_ca.crt \
  -H 'Content-Type: application/json' \
  -d '{
    "index": "blog",
    "shard": 0,
    "primary": true
  }'

這個 API 在排查「為什麼某 shard 一直 unassigned」時非常有用。

Ports 與通訊協定

Port用途說明
9200-9300HTTP REST API外部 client 連接、Kibana、curl、SDK，預設 9200
9300-9400Transport（內部）節點間通訊，預設 9300，使用 binary protocol
5601KibanaKibana Web UI

對應的設定：

# elasticsearch.yml
network.host: 0.0.0.0           # 監聽所有介面
http.port: 9200                 # REST API
transport.port: 9300            # 節點間通訊

# 進階：分離 binding 與 publish address
network.bind_host: 0.0.0.0      # 監聽
network.publish_host: _eth0_    # 對外宣告的位址

本章重點回顧

  • Cluster 由多個 nodes 組成，透過 cluster.name 識別，使用 transport port（9300）內部通訊
  • Cluster state 是叢集的後設資料，由 master 維護並廣播；過大會導致效能問題
  • Node 可以同時擔任多種 roles，由 node.roles 設定
  • 生產環境應分離 master / data / ingest / coordinating / ml 節點
  • Master-eligible 節點建議 3 個（奇數，避免 split-brain）
  • Data nodes 可進一步劃分為 hot / warm / cold / frozen tiers，配合 ILM 自動移轉
  • Coordinating-only 節點作為 client 的負載入口，分擔搜尋協調工作
  • 用 _cat/nodes、_nodes/stats、_cluster/allocation/explain 等 API 觀察與排查
