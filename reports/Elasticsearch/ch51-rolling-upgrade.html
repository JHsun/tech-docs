CH 51 · 滾動升級與版本遷移

適合對象 🔴 維運者 ⚡ 進階
閱讀時間 約 25 分鐘
前置知識 CH 03 架構、CH 47 snapshot
本章目標 升級路徑、相容矩陣、滾動升級流程、deprecation API、回退策略、跨大版本遷移

三種升級類型

類型例方式
Patch9.3.3 → 9.3.4滾動升級
Minor9.2.x → 9.3.x滾動升級
Major8.x → 9.x須先升到最後 minor (8.18+)，再滾動升 major
跨多 Major7.x → 9.x不支援直跳；7.17 → 8.18 → 9.x

版本相容矩陣

  • Index 跨版本相容：N → N+1 主版本 OK；不能跨兩個 major（7 的 index 不能直接在 9 用）
  • Cluster：節點版本差距：滾動升級時可暫時混版（patch / minor），但 master 必須先升
  • Beats / Logstash / Kibana：版本應與 ES 一致或更新；ES 升完再升 stack 元件
  • API client 相容性：ES 8 → 9，client 通常向前 / 向後 1 個 major 相容

升級前 Checklist

1. 用 Deprecation API 找警告

# Cluster 級
GET /_migration/deprecations?pretty

# Node 級
GET /_migration/deprecations?pretty
# 看每個 index 有哪些 deprecated 設定 / 用法

修正後再升級。常見：

  • 使用 deprecated API（如舊版 _template）
  • 使用 deprecated mapping 屬性
  • 使用 _type（早就沒了）
  • 跨 major 不相容的 settings

2. 全 cluster snapshot

PUT /_snapshot/primary/pre-upgrade-9.3.4
{
  "indices":              "*",
  "include_global_state": true,
  "metadata": {
    "reason":  "before 9.3.4 upgrade",
    "version": "9.3.3"
  }
}

# 等完成
GET /_snapshot/primary/pre-upgrade-9.3.4/_status

3. 磁碟空間

升級期間 shard 重新分配可能短暫使用更多空間，至少留 30% 空間。

4. 在 staging 跑相同升級流程

用 production snapshot 還原到 staging cluster，模擬升級。

5. 暫停 ML jobs / Watcher / SLM

# ML jobs
POST /_ml/anomaly_detectors/_all/_stop
POST /_ml/datafeeds/_all/_stop

# Watcher
POST /_watcher/_stop

# SLM 與 ILM
POST /_slm/stop
POST /_ilm/stop

# CCR auto-follow
GET /_ccr/auto_follow
# 對每個 pause
POST /_ccr/auto_follow/{name}/pause

6. 用「optimized rolling restart」script

社群有最佳化滾動腳本，比手動快。也可在 staging 多演練。

滾動升級流程（Patch / Minor）

sequenceDiagram
    participant Op as 維運
    participant ES as Cluster
    participant N1 as Node 1
    participant N2 as Node 2
    participant N3 as Node 3

    Op->>ES: 1. Disable shard allocation (primaries)
    Op->>ES: 2. Stop indexing & sync flush
    Op->>N1: 3. Stop ES, install new version, start
    N1->>ES: 4. Re-join cluster
    Op->>ES: 5. Re-enable allocation, wait GREEN
    Note over N1,ES: 重複 N2、N3...
    Op->>ES: 6. 全部升完，重啟 ML/Watcher/SLM
圖 51-2：滾動升級時序

Step 1：Disable shard allocation

PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": "primaries"
  }
}

Step 2：Stop indexing & flush

# 對所有寫入 client 暫停寫入

# Flush translog
POST /_flush

Step 3：選一個非 master 節點先升級

建議順序：data 節點 → master 節點（master 最後）。

# 在被升級節點上
sudo systemctl stop elasticsearch

# 升級套件
sudo apt-get update && sudo apt-get install --only-upgrade elasticsearch
# 或 RPM
sudo yum update elasticsearch
# 或 Docker
docker pull docker.elastic.co/elasticsearch/elasticsearch:9.3.4

# 啟動
sudo systemctl start elasticsearch

# 等服務就緒
curl -u elastic:xxx --cacert ca.crt \
  https://localhost:9200/_cluster/health

Step 4：Re-enable allocation

PUT /_cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.enable": null
  }
}

# 等到 GREEN 才升下一節點
GET /_cluster/health?wait_for_status=green&timeout=30m

Step 5：對每節點重複

對其他 data 節點重複 Step 1-4。最後升 master 節點。

Step 6：升完恢復服務

# 啟動 ML / Watcher / SLM / ILM / CCR
POST /_ml/anomaly_detectors/_all/_open
POST /_ml/datafeeds/_all/_start

POST /_watcher/_start
POST /_slm/start
POST /_ilm/start

# CCR auto-follow
POST /_ccr/auto_follow/{name}/resume

用 Ansible / Script 自動化

實務上不會手動。用 Ansible playbook 或 K8s operator（ECK）做：

# 簡化的 Ansible playbook
- name: Rolling upgrade ES
  hosts: es_data
  serial: 1                  # 一次一個
  tasks:
    - name: Disable allocation (primaries only)
      uri:
        url: "https://{{ es_host }}/_cluster/settings"
        method: PUT
        body_format: json
        body:
          persistent:
            cluster.routing.allocation.enable: "primaries"
      run_once: true

    - name: Stop ES service
      systemd: name=elasticsearch state=stopped

    - name: Upgrade package
      apt: name=elasticsearch={{ es_version }} state=present
      when: ansible_os_family == "Debian"

    - name: Start ES
      systemd: name=elasticsearch state=started

    - name: Wait for node to rejoin
      uri:
        url: "https://{{ es_host }}/_cluster/health?wait_for_nodes={{ groups['all'] | length }}"
      register: result
      until: result.json.status != "red"
      retries: 60
      delay: 10

    - name: Re-enable allocation
      uri:
        url: "https://{{ es_host }}/_cluster/settings"
        method: PUT
        body_format: json
        body:
          persistent:
            cluster.routing.allocation.enable: null
      run_once: true

    - name: Wait green
      uri:
        url: "https://{{ es_host }}/_cluster/health?wait_for_status=green&timeout=30m"

ECK 升級（K8s）

用 ECK 時，升級就改 CRD 中的 version：

apiVersion: elasticsearch.k8s.elastic.co/v1
kind: Elasticsearch
metadata:
  name: production
spec:
  version: 9.3.4         # 改這個就好
  nodeSets:
    - name: master
      count: 3
    - name: data
      count: 6

ECK operator 會自動：disable allocation → stop pod → 換 image → start → wait green → 下一 pod。

Major 版本升級（8.x → 9.x）

9.0 仍可讀 8.x 建立的 index（向後相容一個 major）。但要先：

  1. 升到 8.x 最後一個 minor（8.18 LTS）
  2. 跑 Deprecation API，修所有警告
  3. 所有 7.x 建立的 index 必須 reindex 或 archive
  4. 全 cluster snapshot
  5. 滾動升 9.x

用 Kibana Upgrade Assistant

Kibana → Stack Management → Upgrade Assistant：

  • 列出所有 deprecation issues
  • 提供 reindex 工具修復不相容的 index
  • checklist 引導完整流程

回退策略

⚠️ ES 不支援 downgrade 一旦 9.0 寫入過 index，不能回退到 8.x。回退方式：
  1. 從升級前的 snapshot 還原到全新 8.x cluster
  2. 把 9.0 期間新進的資料 reindex 到 8.x cluster
所以升級前 snapshot 是最後一道防線。

常見陷阱

  • master 與 data 同節點時順序錯：先升 data，最後升 master
  • 升級期間 ILM 觸發 rollover：可能在新舊節點混版時造成不一致 → 升級前 stop ILM
  • 磁碟空間不夠：shard 重分配導致暫時佔更多
  • Plugin 不相容：第三方 plugin 必須也有對應新版
  • JVM 升級：9.x 推薦 JDK 21，舊 JVM 可能不被支援
  • Index settings 過時：某些 7.x setting 在 9.x 已移除

跨版本還原（複習 CH 47）

  • 新版可以還原舊版 snapshot，但只支援 1 個 major 跨度
  • 9.x 還原 7.x snapshot 需先用 8.x 跳板
  • 9.x 提供 archive 功能：把不相容 index 以 read-only archive 形式 mount
# Archive mode（9.x 新）
POST /_snapshot/old-snap/snap-2025-01/_mount?storage=full_copy
{
  "index":         "old-7.x-index",
  "renamed_index": "archive-old-7.x-index",
  "index_settings": {
    "index.number_of_replicas": 0
  }
}

版本策略建議

  • 跟 LTS 走：8.18 LTS、9.0 LTS 是長期支援
  • 每季升一次 patch，跟上 bugfix 與 security
  • 每年升 minor，獲取新功能
  • major 跟在 GA 後 1-2 個 minor 才升，給社群與第三方 plugin 適應時間
  • 所有環境同版本：staging / prod 一致
  • Beats / Logstash 跟 ES 同版本，至少 patch 一致
  • API client SDK 緊跟：9.x 用 9.x SDK

升級最佳實踐

  • 每次升級都做 snapshot（pre-upgrade，用日期命名）
  • 用 Kibana Upgrade Assistant 找 deprecation
  • 在 staging 跑相同升級腳本，記錄遇到的問題
  • 升級期間暫停 ML / Watcher / SLM / ILM
  • data → master 順序，master 最後升
  • 等 GREEN 才升下一節點，不要趕
  • ECK 用戶：直接改 CRD version，ECK 自動處理
  • 跨 major 走 LTS 跳板（7 → 8.18 → 9）
  • Snapshot 是最後一道防線，downgrade 失敗時靠它
  • 升級紀錄：記錄版本、時間、遇到的 issue 與解法

本章重點回顧

  • 升級類型：patch / minor 滾動升即可；major 必須走 LTS 跳板
  • 不能跨兩個 major 直跳（如 7.x → 9.x，需經 8.x）
  • 升級前必跑 _migration/deprecations 與 Kibana Upgrade Assistant
  • 滾動流程：disable allocation → stop → upgrade → start → enable allocation → wait green
  • 順序：data 節點先升，master 最後
  • 升級期間暫停 ML / Watcher / SLM / ILM / CCR
  • ECK 用 CRD version 自動化升級
  • ES 不支援 downgrade；snapshot 是唯一回退方式
  • 9.x archive 功能可 mount 舊版本 snapshot 為 read-only
  • 用 Ansible / ECK 自動化是 production 必備
