📦 A. Repository（S3 / GCS / Azure / FS / HDFS）

情境
命令 / Code
說明
S3 repository
PUT /_snapshot/s3_repo
{
  "type": "s3",
  "settings": {
    "bucket":          "my-es-snapshots",
    "base_path":       "prod-cluster",
    "region":          "us-east-1",
    "compress":        true,
    "chunk_size":      "1gb",
    "max_restore_bytes_per_sec":  "200mb",
    "max_snapshot_bytes_per_sec": "100mb"
  }
}
需先在 keystore 設 access_key + secret_key
S3 IAM Role 認證（建議）
PUT /_snapshot/s3_repo
{
  "type": "s3",
  "settings": {
    "bucket":   "my-es-snapshots",
    "role_arn": "arn:aws:iam::123:role/es-snapshot-role"
  }
}
IRSA / IMDSv2 自動取憑證；不需 access key
GCS repository
PUT /_snapshot/gcs_repo
{
  "type": "gcs",
  "settings": {
    "bucket":     "my-es-snapshots",
    "client":     "default",
    "base_path":  "prod"
  }
}
gcs.client.default.credentials_file 在 keystore
Azure Blob
PUT /_snapshot/azure_repo
{
  "type": "azure",
  "settings": {
    "container":  "es-snapshots",
    "client":     "default"
  }
}
azure.client.default.account + key 在 keystore
Local FS（測試）
# elasticsearch.yml
path.repo: [ "/mnt/snapshots" ]

# API
PUT /_snapshot/fs_repo
{
  "type": "fs",
  "settings": {
    "location":  "/mnt/snapshots/cluster1",
    "compress":  true
  }
}
path.repo 必須在所有 master + data node 設定且可寫
HDFS repository
PUT /_snapshot/hdfs_repo
{
  "type": "hdfs",
  "settings": {
    "uri":   "hdfs://namenode:8020",
    "path":  "/elasticsearch/snapshots"
  }
}
需安裝 repository-hdfs plugin
看 repository
GET /_snapshot
GET /_snapshot/s3_repo
列所有 / 單一 repo
驗證 repository（重要！）
POST /_snapshot/s3_repo/_verify
檢查所有 node 是否能讀寫 repo
檢查 repository 完整性
POST /_snapshot/s3_repo/_analyze?blob_count=10&max_blob_size=1mb&timeout=30s
深度檢查 repo 一致性（超大 repo 可能跑很久）
清空 repository（謹慎）
POST /_snapshot/s3_repo/_cleanup
刪除孤兒 file（之前有 snapshot 被中斷時會留垃圾）
刪除 repository
DELETE /_snapshot/s3_repo
只刪除 ES 對 repo 的引用；不刪 S3 上的資料
💡 速率限制建議
  • max_snapshot_bytes_per_sec：100mb（避免影響 indexing）
  • max_restore_bytes_per_sec：200mb（restore 通常不影響線上）
  • chunk_size：1gb（小於就分片，平衡 retry 成本與並行）

📸 B. Snapshot 操作

情境
命令 / Code
說明
建 snapshot（同步）
PUT /_snapshot/s3_repo/snap-2026-05-01
{
  "indices":             "logs-*,metrics-*",
  "ignore_unavailable":  true,
  "include_global_state": false,
  "feature_states":      [ "kibana" ],
  "metadata": {
    "taken_by":   "alice",
    "for":        "weekly_backup"
  }
}
預設會等完成。include_global_state: false（不要還原 cluster setting / role）
建 snapshot（非同步）
PUT /_snapshot/s3_repo/snap-2026-05-01?wait_for_completion=false
{ "indices": "*" }
wait_for_completion=false 立即回傳，背景跑
看 snapshot 列表
GET /_snapshot/s3_repo/_all
GET /_snapshot/s3_repo/snap-*?sort=start_time&order=desc
列出 / 排序
看 snapshot 詳情
GET /_snapshot/s3_repo/snap-2026-05-01
看 indices、shard 數、size、duration、state
看 snapshot 狀態（in-progress）
GET /_snapshot/s3_repo/snap-2026-05-01/_status
看每 shard 進度
用 SLM filter 找 snapshots
GET /_snapshot/s3_repo/*?slm_policy_filter=daily-snap
找特定 SLM policy 建的 snapshot
Clone snapshot
PUT /_snapshot/s3_repo/snap-2026-05-01/_clone/snap-2026-05-01-archive
{
  "indices": "logs-2026-04-*"
}
從現有 snapshot 複製出含特定 index 的新 snapshot（dedup）
刪 snapshot
DELETE /_snapshot/s3_repo/snap-2026-05-01
不可逆。SLM 會自動依 retention 刪
刪多筆 snapshot
DELETE /_snapshot/s3_repo/snap-2026-04-*
wildcard 批刪
⚠️ include_global_state 解讀
  • true：snapshot 含 cluster setting + role / user + ILM policy + index template（重建 cluster 需要）
  • false：只含 indices 資料（一般日常備份用這個就好）
  • 建議：每月做一次 true 的全狀態備份；日常備份用 false

♻️ C. Restore 操作

情境
命令 / Code
說明
還原全部
POST /_snapshot/s3_repo/snap-2026-05-01/_restore
{
  "indices":              "*",
  "ignore_unavailable":   true,
  "include_global_state": false,
  "include_aliases":      true
}
目標 index 不能存在！否則先 close / delete

{
  "indices":           "logs-2026.04.30",
  "rename_pattern":    "logs-(.+)",

常用於還原 + 不影響現有 index

  },

還原時調整 setting（如先設 0 replica 加快還原）

⏰ D. SLM Policy（自動排程）

情境
命令 / Code
說明
建 SLM policy（每日）
PUT /_slm/policy/daily-snap
{
  "schedule":   "0 30 1 * * ?",
  "name":       "<daily-snap-{now/d}>",
  "repository": "s3_repo",
  "config": {
    "indices":              "*",
    "ignore_unavailable":   true,
    "include_global_state": false,
    "expand_wildcards":     "open,hidden"
  },
  "retention": {
    "expire_after": "30d",
    "min_count":    7,
    "max_count":    50
  }
}
cron 0:30 AM 每日；保留 30 天但至少 7 個、最多 50 個
每小時 + 每天 + 每月組合
# Hourly
PUT /_slm/policy/hourly
{
  "schedule":   "0 0 * * * ?",
  "name":       "<hourly-{now{yyyy.MM.dd.HH}}>",
  "repository": "s3_repo",
  "config":     { "indices": "logs-*" },
  "retention":  { "expire_after": "2d" }
}

# Daily
PUT /_slm/policy/daily
{
  "schedule":   "0 0 1 * * ?",
  "name":       "<daily-{now/d}>",
  "repository": "s3_repo",
  "config":     { "indices": "*" },
  "retention":  { "expire_after": "30d" }
}

# Monthly
PUT /_slm/policy/monthly
{
  "schedule":   "0 0 2 1 * ?",
  "name":       "<monthly-{now/M}>",
  "repository": "s3_repo",
  "config":     { "indices": "*" },
  "retention":  { "expire_after": "365d" }
}
經典 GFS（Grandfather-Father-Son）備份策略
立即執行 policy（測試）
POST /_slm/policy/daily-snap/_execute
不等排程，馬上跑一次
立即執行 retention
POST /_slm/_execute_retention
不等排程，立即清過期 snapshot
看 SLM 狀態
GET /_slm/status
GET /_slm/stats
RUNNING / STOPPED；統計
看 policies
GET /_slm/policy
GET /_slm/policy/daily-snap
看設定 + 上次執行 + 下次執行
停 / 啟 SLM
POST /_slm/stop
POST /_slm/start
維護期暫停。注意：停 ILM 會連動停 SLM
刪 policy
DELETE /_slm/policy/daily-snap
不會刪已經建好的 snapshots

❄️ F. Searchable Snapshot（cold / frozen tier）

把 snapshot 直接 mount 為可查 index。cold tier 用 fully cached（資料完整下載）；frozen tier 用 partial（只在 S3，按需抓）。

情境
命令 / Code
說明
Mount snapshot（fully cached）
POST /_snapshot/s3_repo/snap-2026-04-01/_mount?wait_for_completion=true
{
  "index":               "logs-2026.04.01",
  "renamed_index":       "restored-logs-2026.04.01",
  "index_settings": {
    "index.number_of_replicas": 0
  }
}
fully cached：cold tier。資料完整下載到 local
Mount snapshot（partial）
POST /_snapshot/s3_repo/snap-2026-04-01/_mount?wait_for_completion=true
{
  "index":  "logs-2026.04.01",
  "storage": "shared_cache"
}
partial：frozen tier。資料只在 S3，需要時才抓
ILM 自動 mount（在 policy 中）
"cold": {
  "actions": {
    "searchable_snapshot": {
      "snapshot_repository": "s3_repo"
    }
  }
},
"frozen": {
  "actions": {
    "searchable_snapshot": {
      "snapshot_repository": "s3_repo",
      "force_merge_index":   true
    }
  }
}
建議做法：ILM 自動處理（見 CS 03）
看 searchable snapshot
GET /_searchable_snapshots/stats
GET /_searchable_snapshots/cache/stats
看快取使用率
清 cache
POST /restored-logs-2026.04.01/_searchable_snapshots/cache/clear
釋放 frozen partial cache 空間

🌐 G. CCR Setup（跨叢集複製）

情境
命令 / Code
說明
1. Local cluster 加遠端
# 在 follower cluster 上設
PUT /_cluster/settings
{
  "persistent": {
    "cluster.remote.leader_cluster.mode":           "proxy",
    "cluster.remote.leader_cluster.proxy_address": "leader.example.com:9443"
  }
}

# 把 cross-cluster API key 放 keystore
bin/elasticsearch-keystore add \
  cluster.remote.leader_cluster.credentials
9.x 推薦用 API key 模型（見 CS 09）
確認連線
GET /_remote/info
看 connected: true
2. 在 leader 建 cross-cluster API key
# 在 leader cluster 執行
POST /_security/cross_cluster/api_key
{
  "name": "follower-key",
  "access": {
    "search":      [ { "names": [ "logs-*", "metrics-*" ] } ],
    "replication": [ { "names": [ "logs-app-*" ] } ]
  },
  "expiration": "365d"
}
注意 search vs replication 是兩種不同 access
💡 CCR 用途速查
  • DR / HA：主備機房異地備援
  • Read scaling：地理上 follow 到讀者較近的 cluster
  • Data locality：將資料 follow 到合規地區
  • Centralized reporting：多 cluster 資料 follow 到一個分析 cluster

🚨 I. Failover SOP（DR 切主）

當 leader cluster 發生災難，需把 follower 切成可寫主：

  1. 確認 leader 真的不可用（避免 split-brain）
  2. 暫停 follow：POST /follower-index/_ccr/pause_follow
  3. 關閉 follower index：POST /follower-index/_close
  4. 解除 follow：POST /follower-index/_ccr/unfollow
  5. 重新打開：POST /follower-index/_open
  6. 切流量：DNS / LB / app 配置切到 follower cluster
  7. 監控寫入正常
  8. 當 leader 恢復後，反向建 CCR（原 follower 變新 leader）

📊 J. DR 策略對照

策略
RPO / RTO
適用
SLM 每日 snapshot
RPO 24h / RTO 數小時（restore 時間）
基本備份；任何規模都該做
SLM 每小時 + 每日
RPO 1h / RTO 數小時
中等需求；GFS 策略
CCR (auto-follow)
RPO 秒 / RTO 分鐘（切流量）
高可用要求；雙 cluster 成本 2×
CCR + SLM
RPO 秒 / RTO 分鐘 + 歷史備份
企業級。CCR 防硬體故障；SLM 防誤刪
Searchable Snapshot
N/A（非 DR，是冷儲存）
省 50-90% 儲存成本，仍可查詢
⚠️ 永遠記得測試還原！
  • 每季做一次 restore drill（還原到 staging）
  • 驗證 snapshot 是否真的可用，沒人想在災難當天才發現備份壞了
  • 記錄 RTO 實測時間
  • 測試 cross-region restore（snapshot 從 S3 us-east 還到 ES eu-west）

          ta.value = text;
          document.body.appendChild(ta);
          ta.select();
          try { document.execCommand('copy'); success(); } catch (_) {}
          document.body.removeChild(ta);
        }
      });
      wrap.appendChild(btn);
    });
  }
  if (document.readyState === 'loading') document.addEventListener('DOMContentLoaded', init);
  else init();
})();

// Auto-build TOC from h2/h3 in 
    
(function () {
  function init() {
    const toc = document.querySelector('.toc ul');
    if (!toc) return;
    const headings = document.querySelectorAll('main h2, main h3');
    headings.forEach(function (h) {
      if (!h.id) {
        h.id = h.textContent.trim().toLowerCase()
          .replace(/[^\w\u4e00-\u9fff\s-]/g, '')
          .replace(/\s+/g, '-');
      }
      const li = document.createElement('li');
      const a = document.createElement('a');
      a.href = '#' + h.id;
      a.textContent = h.textContent;
      a.className = 'toc-' + h.tagName.toLowerCase();
      li.appendChild(a);
      toc.appendChild(li);
    });
  }
  if (document.readyState === 'loading') document.addEventListener('DOMContentLoaded', init);
  else init();
})();

        labelBoxBkgColor: isDark ? '#1a2530' : '#fef3c7',
        labelBoxBorderColor: isDark ? '#a89060' : '#d97706',
        labelTextColor: isDark ? '#e6f2f8' : '#0f172a',
        loopTextColor: isDark ? '#e6f2f8' : '#0f172a',
        noteBkgColor: isDark ? '#3a4f3a' : '#fef3c7',
        noteBorderColor: isDark ? '#86c598' : '#d97706',

      },

      sequence: { useMaxWidth: true, showSequenceNumbers: false, mirrorActors: true, bottomMarginAdj: 10 },
    };
  }

  let renderCounter = 0;
  async function renderAll() {
    if (typeof mermaid === 'undefined') return;
    renderCounter++;
    const myCounter = renderCounter;
    mermaid.initialize(getMermaidConfig());
    const blocks = document.querySelectorAll('.mermaid');
    for (const el of blocks) {
      // restore source if already rendered
      if (el.dataset.source) el.innerHTML = el.dataset.source;
      else el.dataset.source = el.innerHTML;
      el.removeAttribute('data-processed');
    }
    if (myCounter !== renderCounter) return;
    try {
      await mermaid.run({ querySelector: '.mermaid' });
    } catch (e) { console.warn('Mermaid render error', e); }
  }

  function init() {
    renderAll();
    window.addEventListener('jhsun:theme-changed', renderAll);
  }

  if (document.readyState === 'loading') document.addEventListener('DOMContentLoaded', init);
  else init();
})();
