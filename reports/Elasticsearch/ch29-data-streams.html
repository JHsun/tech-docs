什麼是 Data Stream？

Data stream 是 ES 對時序、append-only 資料的抽象層。它在底層管理一組 backing indices，外部只看見一個邏輯名稱。

Data Stream 特性

  • Append-only：只能 index 新文件，不能直接 update / delete by ID
  • 必須有 @timestamp 欄位
  • 自動 rollover：到達某條件（時間、大小）自動建新 backing index
  • 所有 backing indices 共享 mapping/settings（從 index template 來）
  • Backing index 命名固定：.ds-{stream-name}-{date}-{generation}
  • 查詢無感：搜尋 logs-app 自動跨所有 backing indices

Data Stream vs 一般 Index

項目Data Stream一般 Index
用途時序、log、metric、trace業務 entity、需 update 的資料
Update by ID❌✅
Delete by ID❌✅
Update/Delete by query✅✅
Rollover自動 / DSL / ILM需手動或用 alias 配合
@timestamp 必填✅❌
Index template 必須✅❌（可有可無）

建立 Data Stream

Step 1：建立 component templates

# Settings
PUT _component_template/logs-app-settings
{
  "template": {
    "settings": {
      "number_of_shards":   2,
      "number_of_replicas": 1,
      "index.lifecycle.name": "logs-app-30d"
    }
  }
}

# Mappings（必含 @timestamp）
PUT _component_template/logs-app-mappings
{
  "template": {
    "mappings": {
      "properties": {
        "@timestamp":   { "type": "date" },
        "message":      { "type": "match_only_text" },
        "log.level":    { "type": "keyword" },
        "service.name": { "type": "keyword" }
      }
    }
  }
}

Step 2：建立 data stream index template

PUT _index_template/logs-app-template
{
  "index_patterns": [ "logs-app*" ],
  "data_stream": { },                       # 關鍵！宣告為 data stream template
  "priority": 200,
  "composed_of": [
    "logs-app-mappings",
    "logs-app-settings"
  ]
}

Step 3：建立 data stream（顯式或隱式）

方式 A：自動建立（首次寫入時）：

POST logs-app/_doc
{
  "@timestamp": "2026-04-15T10:00:00Z",
  "message":    "Application started",
  "log.level":  "INFO",
  "service.name": "order-svc"
}

方式 B：顯式建立：

PUT _data_stream/logs-app

管理 Data Stream

檢視狀態

# 列出所有 data stream
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_data_stream?pretty"

# 看單一
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_data_stream/logs-app?pretty"

# 看 stats
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_data_stream/logs-app/_stats?pretty"

輸出範例：

{
  "data_streams": [{
    "name": "logs-app",
    "timestamp_field": { "name": "@timestamp" },
    "indices": [
      { "index_name": ".ds-logs-app-2026.01.01-000001", "index_uuid": "..." },
      { "index_name": ".ds-logs-app-2026.01.15-000002", "index_uuid": "..." },
      { "index_name": ".ds-logs-app-2026.02.01-000003", "index_uuid": "..." }
    ],
    "generation":      3,
    "status":          "GREEN",
    "template":        "logs-app-template",
    "ilm_policy":      "logs-app-30d",
    "hidden":          false,
    "system":          false,
    "allow_custom_routing": false,
    "replicated":      false
  }]
}

手動 Rollover

curl -X POST -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/logs-app/_rollover?pretty"

會建立新的 backing index，並把 write 切到它。

把既有 index 加入 data stream

POST _data_stream/_modify
{
  "actions": [
    { "add_backing_index":   { "data_stream": "logs-app", "index": ".ds-logs-app-old-001" } },
    { "remove_backing_index": { "data_stream": "logs-app", "index": ".ds-logs-app-broken-002" } }
  ]
}

刪除 data stream

curl -X DELETE -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_data_stream/logs-app"
# 注意：此舉會刪掉所有 backing indices！

Naming Scheme（data_stream namespace）

Elastic 自家 stack（Beats、Elastic Agent、Fleet）採用三段式命名：

{type}-{dataset}-{namespace}

例：
  logs-nginx.access-default
  logs-nginx.error-default
  metrics-system.cpu-default
  traces-apm.app-default
  • type：logs / metrics / traces / synthetics
  • dataset：資料來源（nginx.access）
  • namespace：環境/租戶（default、production、tenant-a）

自訂 stream 建議遵循此規範，便於與 Elastic ecosystem 整合。

TSDS（Time Series Data Stream）

TSDS 是專為 metrics 優化的 data stream，提供：

  • 強制 @timestamp 排序儲存，提升壓縮率
  • 強制 dimension fields（用於識別「這條 metric 是哪個 instance 的」）
  • 更好的儲存效率（相同 dimension 的 docs 聚集）
  • Downsampling：自動降取樣舊資料

建立 TSDS template

PUT _index_template/metrics-system-tsds-template
{
  "index_patterns": [ "metrics-system*" ],
  "data_stream": { },
  "priority": 500,
  "template": {
    "settings": {
      "index.mode": "time_series",                  # 啟用 TSDS！
      "index.routing_path": [                        # 指定 dimensions 用作路由
        "host.name",
        "metricset.name"
      ]
    },
    "mappings": {
      "properties": {
        "@timestamp":      { "type": "date" },
        "host.name":       { "type": "keyword", "time_series_dimension": true },
        "host.ip":         { "type": "ip",      "time_series_dimension": true },
        "metricset.name":  { "type": "keyword", "time_series_dimension": true },

        "system.cpu.user":      { "type": "double", "time_series_metric": "gauge"   },
        "system.cpu.system":    { "type": "double", "time_series_metric": "gauge"   },
        "system.disk.io_bytes": { "type": "long",   "time_series_metric": "counter" },
        "system.network.in":    { "type": "long",   "time_series_metric": "counter" }
      }
    }
  }
}

TSDS 欄位特殊屬性

屬性說明
time_series_dimension: true標記為 dimension（識別 series）。多個 dimension 組合決定一條 series。
time_series_metric: "gauge"當下值（如 CPU%、temperature）
time_series_metric: "counter"累積計數（如 bytes_sent，只增不減）
time_series_metric: "summary"已聚合的 summary（含 sum、count、min、max）
time_series_metric: "histogram"histogram 預聚合

TSDS Time Bounds（重要！）

TSDS 的 backing index 有嚴格的時間範圍。文件 timestamp 必須落在某個 backing index 的範圍內，否則會被拒絕：

.ds-metrics-system-2026.04.15-000003
  Time bounds: [2026-04-15T00:00:00, 2026-04-15T11:00:00)  ← 寫入中

.ds-metrics-system-2026.04.15-000002
  Time bounds: [2026-04-14T00:00:00, 2026-04-15T00:00:00)  ← 已 rollover

sequenceDiagram
    participant Doc as 📨 Late doc
@timestamp: 2026-04-13
    participant ES as Elasticsearch
    participant Live as Live backing index
(2026-04-15 ~)
    participant Old as Old backing index
(已 rollover, read-only)

    Doc->>ES: index this metric
    ES->>Live: write index?
    Live-->>ES: ❌ timestamp out of bounds
    ES->>Old: backing index for 2026-04-13?
    Old-->>ES: ❌ already read-only / cold
    ES-->>Doc: ❌ rejected: failure store!

遲到資料處理

實務上 metrics 可能因網路、agent buffer 等原因遲到。9.1+ 提供 Failure Store（CH 35）自動捕獲被拒絕的文件：

PUT _index_template/metrics-system-tsds-template
{
  "index_patterns": [ "metrics-system*" ],
  "data_stream": {
    "failure_store": true     # 9.1+ 啟用 failure store
  },
  ...
}

另外可調整 backing index 寫入窗口讓遲到 6 小時內的資料還能寫入：

{
  "settings": {
    "index.mode": "time_series",
    "index.look_ahead_time":   "1h",        # write 窗口往未來延伸 1 小時
    "index.look_back_time":    "6h"         # write 窗口往過去延伸 6 小時
  }
}

Downsampling（降取樣）

TSDS 支援自動把舊 backing index 降取樣（合併成統計值），大幅減少儲存：

# 對單一 backing index 手動降取樣
POST .ds-metrics-system-2026.04.01-000001/_downsample/.ds-metrics-system-2026.04.01-000001-1h
{
  "fixed_interval": "1h"   # 把每秒的點壓成每小時統計
}

透過 ILM 自動化（CH 30）：

{
  "policy": {
    "phases": {
      "warm": {
        "min_age": "7d",
        "actions": {
          "downsample": { "fixed_interval": "1h" }
        }
      }
    }
  }
}

Logs Mode（9.x 新）

9.x 推出 index.mode: logs，類似 TSDS 但專為 log 場景優化：

PUT _index_template/logs-mode-template
{
  "index_patterns": [ "logs-mode-*" ],
  "data_stream": { },
  "template": {
    "settings": {
      "index.mode": "logs"           # ← 啟用 logs mode
    },
    "mappings": {
      "properties": {
        "@timestamp":  { "type": "date" },
        "host.name":   { "type": "keyword" },
        "message":     { "type": "match_only_text" }
      }
    }
  }
}

優勢：

  • 同樣自動依 timestamp 排序，提升壓縮率
  • 不需要 dimensions 設定，比 TSDS 更簡單
  • 對 log 工作負載儲存可省 ~30-50%

Data Stream vs 傳統 write alias 模式

過去 7.x 之前用 alias + rollover 達到類似效果：

傳統：
  logs-write   (alias, is_write_index=true)  → logs-2026-01
  logs-search  (alias, read-only)            → logs-2026-01, logs-2026-02

Data Stream（推薦）：
  logs-app     ← 自動管理 backing indices

Data stream 取代後者，更簡潔且 official 維護。除非有特殊需求，新建一律用 data stream。

自訂 Routing（少用）

預設 data stream 不允許 custom routing。若要啟用：

POST _data_stream/logs-app/_modify
{
  "actions": [
    { "set_data_stream_options": {
        "data_stream": "logs-app",
        "options": { "allow_custom_routing": true }
    } }
  ]
}

啟用後 index 時可帶 ?routing=<value>。但這會與 ILM、shrink 等行為衝突，僅在必要時才用。

Data Stream 最佳實踐

  • 所有 log/metric/trace 用 data stream（取代傳統 daily index）
  • 遵循 {type}-{dataset}-{namespace} 命名規範
  • 必設 ILM 或 DSL（CH 30、CH 31），否則資料會無限增長
  • Metrics workload 用 TSDS（index.mode: time_series）+ downsampling
  • Logs workload 用 9.x 新的 logs mode（index.mode: logs）
  • 啟用 failure store，捕獲被拒絕的文件
  • 遲到資料：估算最大延遲，配合 look_back_time 或 failure store + 重送
  • 初始 shard 數別開太大，rollover 後可重新 size（用 size-based rollover 而非 time-based）
  • Backing index 命名是 .ds- 開頭，預設 hidden，_cat/indices 加 ?expand_wildcards=all 才看得到

本章重點回顧

  • Data stream = append-only 時序資料的抽象，背後管理多個 backing indices
  • 必須用 index template + "data_stream": {} 宣告
  • 必含 @timestamp 欄位
  • 不能用 ID 直接 update/delete（要用 update/delete by query）
  • 命名遵循 {type}-{dataset}-{namespace} 規範
  • TSDS：metrics 專用，強制 dimensions、time bounds、可 downsample
  • Logs mode：9.x 新；log 專用，省空間且不需 dimensions 設定
  • 遲到資料是 TSDS 主要痛點：用 look_back_time 或 failure store 處理
  • Backing index .ds- 開頭、預設 hidden
