CH 20 · 使用者認證與內建使用者

適合對象 🟡 開發者 🔴 維運者

前置知識 CH 19 · TLS
本章目標 認識認證 (authentication) 與授權 (authorization) 的差別、內建使用者用途、native realm、file realm、密碼管理

Authentication（認證） vs Authorization（授權）

術語問題本章涵蓋
Authentication（authn）「你是誰？」驗證身份✅ 本章
Authorization（authz）「你能做什麼？」決定權限下章 CH 21（RBAC）

Reserved（內建）使用者

ES 安裝後即存在，不能刪除，但可重設密碼：

使用者用途權限
elastic超級管理員，初次登入用superuser（所有權限）
kibana_systemKibana 連線到 ES 用kibana_system role
logstash_systemLogstash 監控自身用logstash_system role
beats_systemBeats 內部監控用beats_system role
apm_systemAPM Server 監控用apm_system role
remote_monitoring_user跨叢集監控資料寫入remote_monitoring_collector + remote_monitoring_agent
_anonymous（虛擬）匿名請求由 anonymous role 設定
🚨 永遠改 elastic 預設密碼 Auto-config 會產生隨機密碼，但若你手動部署或從舊版升級，第一件事就是重設 elastic 密碼，且只在初始 setup 用，平時都用其他 user。

重設內建使用者密碼

三種方式，按建議度排序：

方式 1：elasticsearch-reset-password（推薦）

# 產生隨機新密碼（會輸出在畫面）
sudo /usr/share/elasticsearch/bin/elasticsearch-reset-password \
  -u elastic

# 互動式輸入指定密碼
sudo /usr/share/elasticsearch/bin/elasticsearch-reset-password \
  -u elastic -i

# Batch mode（不問確認）
sudo /usr/share/elasticsearch/bin/elasticsearch-reset-password \
  -u kibana_system --batch \
  --password "ChangeMeKibanaPassword"

# 對遠端 cluster 重設
sudo /usr/share/elasticsearch/bin/elasticsearch-reset-password \
  -u elastic \
  --url "https://es-node-1.internal:9200"

需求：local cluster 必須處於可用狀態（至少能與一個節點溝通）。

方式 2：透過 Security API

curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/user/kibana_system/_password" -d '
{
  "password": "NewKibanaPassword"
}'

方式 3：Kibana Stack Management UI

Stack Management → Users → 點選用戶 → Edit → Change password

Native Realm — 在 ES 中建立使用者

Native realm 把使用者存在 .security-7 index 中，是最常用的方式。

建立使用者

# 用 API
curl -X POST -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/user/john_doe" -d '
{
  "password": "InitialP@ssw0rd",
  "roles": [ "superuser" ],
  "full_name": "John Doe",
  "email": "john@example.com",
  "metadata": {
    "department": "engineering"
  },
  "enabled": true
}'

命令列工具：elasticsearch-users（File realm only）

注意：elasticsearch-users 命令操作的是 file realm，不是 native。Native realm 必須用 API 或 Kibana。

列出使用者

curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/user?pretty"

# 看單一使用者
curl -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/user/john_doe?pretty"

啟用 / 停用使用者

curl -X PUT -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/user/john_doe/_disable"

curl -X PUT -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/user/john_doe/_enable"

刪除使用者

curl -X DELETE -u elastic:xxx --cacert ca.crt \
  "https://localhost:9200/_security/user/john_doe"

Kibana UI 建立

Stack Management → Users → Create user，填表單即可。

File Realm — 給超級管理員緊急救援用

File realm 把使用者寫在每個節點本地的檔案中：

  • $ES_PATH_CONF/users（username + bcrypt password）
  • $ES_PATH_CONF/users_roles（username → roles 對應）

用途：

  • Native / SSO realm 全部失靈時，仍能登入修復
  • Bootstrap 流程（首次設定）
  • 不希望某 admin user 受 SSO 影響
⚠️ File realm 缺點
  • 每個節點獨立維護一份檔案——增刪 user 要 sync 到全節點
  • 無法用 Kibana UI 管理
  • 適合少量、不常變動的 admin user

使用 elasticsearch-users 命令

# 建立 user
sudo /usr/share/elasticsearch/bin/elasticsearch-users useradd backup_admin \
  -p ChangeMe -r superuser

# 列出
sudo /usr/share/elasticsearch/bin/elasticsearch-users list
# backup_admin : superuser

# 變更密碼
sudo /usr/share/elasticsearch/bin/elasticsearch-users passwd backup_admin

# 加 role
sudo /usr/share/elasticsearch/bin/elasticsearch-users roles -a kibana_admin backup_admin

# 移除 role
sudo /usr/share/elasticsearch/bin/elasticsearch-users roles -r superuser backup_admin

# 刪 user
sudo /usr/share/elasticsearch/bin/elasticsearch-users userdel backup_admin

同步檔案到其他節點

for host in es-node-2 es-node-3; do
  sudo scp /etc/elasticsearch/users $host:/etc/elasticsearch/
  sudo scp /etc/elasticsearch/users_roles $host:/etc/elasticsearch/
done

檔案會被自動 reload（無需重啟）。

配置 Realm Order

當啟用多種 realm（例如同時有 native + LDAP），需要明確指定優先序：

xpack.security.authc:
  anonymous:
    username: anonymous_user
    roles: ["read_only_role"]
    authz_exception: true

  realms:
    file:
      file1:
        order: 0
    native:
      native1:
        order: 1
    ldap:
      ldap1:
        order: 2
        url: "ldaps://ad.example.com:636"
        # ...

Order 越小越優先。注意：file 通常設 0，確保 emergency admin 永遠能登入。

匿名存取

讓沒有提供 credentials 的請求也能取得部分權限：

xpack.security.authc:
  anonymous:
    username: anonymous_user
    roles: [ "read_only_status" ]
    authz_exception: true

注意：

  • 應只給非常受限的 role（例如僅讀 cluster health）
  • authz_exception: true 表示認證失敗會回 401（讓 client 嘗試帶 credentials）；false 則靜默用 anonymous role

內建 roles 速覽

常用內建 role（CH 21 詳述）：

Role用途
superuser全部權限（小心給）
kibana_adminKibana 全功能管理
kibana_user使用 Kibana 但不能管
monitoring_user僅讀 monitoring data
watcher_admin / watcher_userWatcher 管理 / 使用
ml_admin / ml_userML 功能
data_frame_transforms_admin / ...userTransform 管理
ingest_admin管理 ingest pipelines
logstash_admin管理 Logstash pipeline 設定

密碼雜湊算法

9.x 預設使用 BCRYPT（cost 10）。可調整：

# 提高 hash cost（更安全但 login 變慢）
xpack.security.authc.password_hashing.algorithm: bcrypt12   # cost 12

# 或用 PBKDF2
xpack.security.authc.password_hashing.algorithm: pbkdf2_stretch_1000000

FIPS 環境必須用 PBKDF2（CH 25）。

Run-as：模擬其他使用者

讓某個 admin 暫時用其他 user 身分執行請求：

# 給 admin role 加上 run_as 權限
curl -X PUT -u elastic:xxx --cacert ca.crt \
  -H 'Content-Type: application/json' \
  "https://localhost:9200/_security/role/superuser_runas" -d '
{
  "cluster": ["all"],
  "indices": [{
    "names": ["*"],
    "privileges": ["all"]
  }],
  "run_as": [ "john_doe" ]
}'

# 用 admin 認證，但以 john_doe 身分執行
curl -u admin:xxx --cacert ca.crt \
  -H "es-security-runas-user: john_doe" \
  "https://localhost:9200/_security/_authenticate"

用途：除錯、稽核、模擬支援。

登入失敗保護

9.x 預設已啟用：

# elasticsearch.yml 預設值
# 連續 5 次失敗鎖 1 分鐘
xpack.security.authc.failure_cache.size: 1000
xpack.security.authc.failure_cache.ttl: 60s

# 整體 token cache（用於 SAML、OIDC）
xpack.security.authc.token.timeout: 20m
xpack.security.authc.api_key.cache.ttl: 24h

_authenticate API：檢查當前 user

curl -u john_doe:xxx --cacert ca.crt \
  "https://localhost:9200/_security/_authenticate?pretty"

輸出：

{
  "username": "john_doe",
  "roles": [ "superuser" ],
  "full_name": "John Doe",
  "email": "john@example.com",
  "metadata": { "department": "engineering" },
  "enabled": true,
  "authentication_realm": {
    "name": "native1",
    "type": "native"
  },
  "lookup_realm": {
    "name": "native1",
    "type": "native"
  },
  "authentication_type": "realm"
}

常用於：debug 認證問題、看到底是用哪個 realm 認的。

最佳實踐

  • elastic 用戶只在 emergency 用，平時建立 dedicated admin user
  • 每個服務（Kibana、Logstash、Beats、Filebeat）用獨立 user，給最小權限
  • 建立應用層 user，不要讓所有 app 共用同一個高權限 user
  • 密碼長度 ≥ 16 字元，含大小寫字母、數字、符號
  • 用密碼管理器或 secrets manager（Vault、AWS Secrets Manager、Azure Key Vault）
  • 非人類存取改用 API Keys 或 Service Accounts（CH 22）
  • 啟用稽核日誌追蹤認證事件（CH 24）
  • 定期 rotate 密碼，特別是 service accounts

本章 Checklist

  • ☐ elastic 預設密碼已重設
  • ☐ 為每個服務建立獨立的 user，最小權限原則
  • ☐ File realm 至少有一個 emergency admin user，且檔案已 sync 到所有節點
  • ☐ Kibana 的 elasticsearch.username 改為 kibana_system（不要用 elastic）
  • ☐ 密碼存於 keystore 或外部 secrets manager，不寫在 yml/code
  • ☐ 監控登入失敗事件（CH 50 stack monitoring）

本章重點回顧

  • authn = 你是誰；authz = 你能做什麼
  • Authentication chain：Reserved → Native → File → AD/LDAP → SAML/OIDC → PKI → API Key
  • Reserved users：elastic、kibana_system、logstash_system 等，不能刪
  • Native realm 是最常用方式，使用者存在 .security-7 index
  • File realm 適合 emergency admin（每節點獨立檔案）
  • 三種重設密碼方式：CLI tool、API、Kibana UI
  • 密碼預設 BCRYPT 雜湊；FIPS 用 PBKDF2
  • 每個服務用獨立 user 並給最小權限是黃金原則
  • 使用 _authenticate API 隨時檢查當前 user 與 realm
