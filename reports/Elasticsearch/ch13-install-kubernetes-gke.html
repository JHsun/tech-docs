GKE Standard vs Autopilot

特性StandardAutopilot
節點管理使用者管理 node poolsGoogle 完全託管
計費按 VM 規格按 Pod 資源用量
節點調校可自由 sysctl、kernel 參數受限
可裝 DaemonSet可受限（少數 system 用）
適合 ES？✅ 推薦⚠️ 可用但有限制
⚠️ Autopilot 上的 ES 限制 Autopilot 不允許 privileged init container 修改 sysctl，必須改用 node.store.allow_mmap: false（會犧牲效能），或設 spec.podTemplate.spec.securityContext.sysctls 但只支援 safe sysctls，vm.max_map_count 不在其中。
結論：生產用 GKE Standard。

Step 1：建立 GKE Standard 叢集

PROJECT_ID=my-project
CLUSTER_NAME=elastic-prod
REGION=asia-east1

# 建立 regional 叢集（master 跨 3 zones，比 zonal 高可用）
gcloud container clusters create $CLUSTER_NAME \
  --project=$PROJECT_ID \
  --region=$REGION \
  --release-channel=stable \
  --cluster-version=1.31 \
  --num-nodes=1 \
  --machine-type=e2-medium \
  --workload-pool=$PROJECT_ID.svc.id.goog \
  --enable-ip-alias \
  --network=default \
  --subnetwork=default \
  --enable-shielded-nodes \
  --shielded-secure-boot

# 取得 kubeconfig
gcloud container clusters get-credentials $CLUSTER_NAME --region $REGION

加入 Elasticsearch 專屬 node pools

# Master node pool（小型、跨 3 zones）
gcloud container node-pools create master-pool \
  --cluster=$CLUSTER_NAME \
  --region=$REGION \
  --node-locations=$REGION-a,$REGION-b,$REGION-c \
  --num-nodes=1 \
  --machine-type=e2-standard-4 \
  --disk-type=pd-ssd \
  --disk-size=50 \
  --node-labels=role=elasticsearch-master \
  --node-taints=dedicated=elasticsearch-master:NoSchedule \
  --workload-metadata=GKE_METADATA \
  --enable-autoupgrade \
  --enable-autorepair

# Hot data node pool
gcloud container node-pools create data-hot-pool \
  --cluster=$CLUSTER_NAME \
  --region=$REGION \
  --node-locations=$REGION-a,$REGION-b,$REGION-c \
  --num-nodes=1 \
  --machine-type=n2-highmem-16 \
  --disk-type=pd-ssd \
  --disk-size=200 \
  --node-labels=role=elasticsearch-data-hot \
  --node-taints=dedicated=elasticsearch-data-hot:NoSchedule \
  --workload-metadata=GKE_METADATA \
  --enable-autoscaling \
  --min-nodes=1 \
  --max-nodes=3 \
  --enable-autoupgrade \
  --enable-autorepair

GCP machine type 選擇

類型適用備註
n2-highmem-16Hot data（推薦）16 vCPU, 128 GB RAM
n2d-highmem-16Hot data（AMD，省 10-20%）同等規格更便宜
t2a-standard-16ARM64（最便宜）Ampere Altra，需 ES image arm64
c3-highmem-22新世代 Intel更高 IPC，價錢較高
e2-standard-4Master nodes共享 CPU，便宜

Step 2：Persistent Disk 與 StorageClass

GCP Persistent Disk 類型

類型IOPS（最高）適用
pd-standard（HDD）≈ 0.75 read / 1.5 write per GB不適合 ES
pd-balanced15K read / 15K writeWarm tier
pd-ssd30K read / 30K writeHot tier 預設選
pd-extreme120K（依容量）超高 IO 需求
hyperdisk-extreme350K新世代極致 IO；僅部分 machine series 支援
hyperdisk-balanced160K新世代平衡型

StorageClass 設定

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: pd-ssd-elastic
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: pd.csi.storage.gke.io
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
reclaimPolicy: Retain
parameters:
  type: pd-ssd
  replication-type: none   # 單 zone；ES 已透過 replicas 提供冗餘
  disk-encryption-kms-key: projects/PROJECT/locations/REGION/keyRings/RING/cryptoKeys/KEY  # 可選 CMEK

Hyperdisk 範例（更新世代）：

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: hyperdisk-balanced-elastic
provisioner: pd.csi.storage.gke.io
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
parameters:
  type: hyperdisk-balanced
  provisioned-iops-on-create: "10000"
  provisioned-throughput-on-create: "180Mi"

Step 4：部署 Elasticsearch

apiVersion: elasticsearch.k8s.elastic.co/v1
kind: Elasticsearch
metadata:
  name: production
  namespace: elastic
spec:
  version: 9.3.4
  nodeSets:

    - name: master
      count: 3
      config:
        node.roles: ["master"]
        xpack.ml.enabled: false
      zoneAwareness:
        topologyKey: topology.kubernetes.io/zone
      podTemplate:
        spec:
          tolerations:
            - key: dedicated
              value: elasticsearch-master
              effect: NoSchedule
          nodeSelector:
            role: elasticsearch-master
          initContainers:
            - name: sysctl
              securityContext: { privileged: true, runAsUser: 0 }
              command: ['sh', '-c', 'sysctl -w vm.max_map_count=262144']
          containers:
            - name: elasticsearch
              env:
                - name: ES_JAVA_OPTS
                  value: "-Xms2g -Xmx2g"
              resources:
                requests: { memory: 4Gi, cpu: 1 }
                limits:   { memory: 4Gi, cpu: 2 }
      volumeClaimTemplates:
        - metadata: { name: elasticsearch-data }
          spec:
            accessModes: ["ReadWriteOnce"]
            resources: { requests: { storage: 50Gi } }
            storageClassName: pd-ssd-elastic

    - name: data-hot
      count: 3
      config:
        node.roles: ["data_hot", "data_content", "ingest"]
      zoneAwareness:
        topologyKey: topology.kubernetes.io/zone
      podTemplate:
        spec:
          serviceAccountName: elasticsearch-snapshot   # 後文 Workload Identity 設定
          tolerations:
            - key: dedicated
              value: elasticsearch-data-hot
              effect: NoSchedule
          nodeSelector:
            role: elasticsearch-data-hot
          initContainers:
            - name: sysctl
              securityContext: { privileged: true, runAsUser: 0 }
              command: ['sh', '-c', 'sysctl -w vm.max_map_count=262144']
          containers:
            - name: elasticsearch
              env:
                - name: ES_JAVA_OPTS
                  value: "-Xms16g -Xmx16g"
              resources:
                requests: { memory: 32Gi, cpu: 4 }
                limits:   { memory: 32Gi, cpu: 8 }
      volumeClaimTemplates:
        - metadata: { name: elasticsearch-data }
          spec:
            accessModes: ["ReadWriteOnce"]
            resources: { requests: { storage: 1Ti } }
            storageClassName: pd-ssd-elastic

Step 5：Workload Identity（替代 SA Key）

Workload Identity 讓 K8s ServiceAccount 對應到 GCP IAM Service Account，無需把 GCP SA key 嵌進 Secret。

建立 GCP Service Account

PROJECT_ID=my-project
GSA_NAME=elasticsearch-snapshot
KSA_NAME=elasticsearch-snapshot
KSA_NAMESPACE=elastic

# 建立 GCP Service Account
gcloud iam service-accounts create $GSA_NAME \
  --project=$PROJECT_ID \
  --display-name="Elasticsearch Snapshot SA"

# 給 GCS 權限
gsutil iam ch \
  serviceAccount:${GSA_NAME}@${PROJECT_ID}.iam.gserviceaccount.com:roles/storage.objectAdmin \
  gs://elastic-prod-snapshots

建立 K8s ServiceAccount 並綁定

kubectl create namespace $KSA_NAMESPACE
kubectl create serviceaccount $KSA_NAME -n $KSA_NAMESPACE

# 允許 K8s SA 模擬（impersonate）GCP SA
gcloud iam service-accounts add-iam-policy-binding \
  ${GSA_NAME}@${PROJECT_ID}.iam.gserviceaccount.com \
  --role=roles/iam.workloadIdentityUser \
  --member="serviceAccount:${PROJECT_ID}.svc.id.goog[${KSA_NAMESPACE}/${KSA_NAME}]"

# 註解 K8s SA 指向 GCP SA
kubectl annotate serviceaccount $KSA_NAME \
  -n $KSA_NAMESPACE \
  iam.gke.io/gcp-service-account=${GSA_NAME}@${PROJECT_ID}.iam.gserviceaccount.com

驗證 Workload Identity

kubectl run -it \
  --image=google/cloud-sdk:slim \
  --serviceaccount=$KSA_NAME \
  -n $KSA_NAMESPACE \
  --rm wi-test -- \
  gcloud auth list

# 應該顯示 GSA：elasticsearch-snapshot@PROJECT.iam.gserviceaccount.com

Step 6：GCS Snapshot Repository

建立 GCS bucket

gsutil mb -p $PROJECT_ID -l $REGION -b on gs://elastic-prod-snapshots

# 啟用 versioning 與 lifecycle
gsutil versioning set on gs://elastic-prod-snapshots

cat > lifecycle.json << EOF
{
  "lifecycle": {
    "rule": [
      {
        "action": { "type": "SetStorageClass", "storageClass": "NEARLINE" },
        "condition": { "age": 30 }
      },
      {
        "action": { "type": "SetStorageClass", "storageClass": "COLDLINE" },
        "condition": { "age": 90 }
      }
    ]
  }
}
EOF
gsutil lifecycle set lifecycle.json gs://elastic-prod-snapshots

註冊 GCS repository

PASSWORD=$(kubectl get secret production-es-elastic-user -n elastic \
  -o go-template='{{.data.elastic | base64decode}}')

kubectl exec -it production-es-data-hot-0 -n elastic -- \
  curl -k -u elastic:$PASSWORD -X PUT "https://localhost:9200/_snapshot/gcs_repo" \
  -H 'Content-Type: application/json' \
  -d '{
    "type": "gcs",
    "settings": {
      "bucket": "elastic-prod-snapshots",
      "base_path": "production-cluster"
    }
  }'

# 驗證
kubectl exec -it production-es-data-hot-0 -n elastic -- \
  curl -k -u elastic:$PASSWORD -X POST "https://localhost:9200/_snapshot/gcs_repo/_verify"
✅ 不需要 keystore 傳統 GCS 整合需要把 service account key JSON 放進 elasticsearch-keystore。透過 Workload Identity，完全免去 key 管理，安全性大幅提升。

Step 7：用 GCP Load Balancer 暴露 Kibana

GKE 的內建 Ingress controller 會自動建立 Cloud Load Balancer（HTTP(S) LB）。

部署 Kibana

---
apiVersion: kibana.k8s.elastic.co/v1
kind: Kibana
metadata:
  name: production
  namespace: elastic
spec:
  version: 9.3.4
  count: 2
  elasticsearchRef:
    name: production
  http:
    tls:
      selfSignedCertificate:
        disabled: true
    service:
      spec:
        type: NodePort
---
# Managed Certificate（GCP 自動申請的免費 SSL）
apiVersion: networking.gke.io/v1
kind: ManagedCertificate
metadata:
  name: kibana-cert
  namespace: elastic
spec:
  domains:
    - kibana.example.com
---
# BackendConfig（health check 設定）
apiVersion: cloud.google.com/v1
kind: BackendConfig
metadata:
  name: kibana-backend
  namespace: elastic
spec:
  healthCheck:
    requestPath: /
    port: 5601
    type: HTTP
  timeoutSec: 30
  connectionDraining:
    drainingTimeoutSec: 60
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: kibana
  namespace: elastic
  annotations:
    kubernetes.io/ingress.class: gce
    networking.gke.io/managed-certificates: kibana-cert
    kubernetes.io/ingress.allow-http: "false"
    kubernetes.io/ingress.global-static-ip-name: kibana-ip
spec:
  rules:
    - host: kibana.example.com
      http:
        paths:
          - path: /*
            pathType: ImplementationSpecific
            backend:
              service:
                name: production-kb-http
                port:
                  number: 5601

事先準備靜態 IP

gcloud compute addresses create kibana-ip --global
gcloud compute addresses describe kibana-ip --global
# 把回傳的 IP 加入 Cloud DNS A record 對應 kibana.example.com

內部 Load Balancer（VPC 內存取）

若不需公網暴露，改用 Internal LB：

apiVersion: v1
kind: Service
metadata:
  name: production-es-internal
  namespace: elastic
  annotations:
    networking.gke.io/load-balancer-type: "Internal"
    networking.gke.io/internal-load-balancer-allow-global-access: "true"
spec:
  type: LoadBalancer
  selector:
    elasticsearch.k8s.elastic.co/cluster-name: production
    elasticsearch.k8s.elastic.co/node-data: "true"
  ports:
    - name: https
      port: 9200
      targetPort: 9200

Zonal vs Regional 叢集選擇

類型Master HA費用建議
Zonal單 zone master低非生產 / DR site
Multi-zonal單 zone master，跨 zone workers中過渡階段
Regional3 zones master高（master 收費）✅ 生產建議

費用優化

  • Committed Use Discount：1 年 CUD 省約 25%；3 年省約 50%
  • Custom Machine Type：精準匹配 ES 需求（如 16 vCPU + 64 GB RAM 的客製規格）
  • Tau (T2A/T2D) instances：較 N2 便宜 30-40%；T2A 是 ARM
  • Spot VMs：⚠️ ES 資料節點不建議用，但可用於 ML inference 節點
  • GCS lifecycle：snapshot 30 天後轉 Nearline、90 天 Coldline、365 天 Archive
  • 跨 zone 流量：同 region 不同 zone $0.01/GB；考慮 single-zone deployment for dev

Cloud Monitoring 整合

# GKE 預設啟用 Cloud Monitoring
# 部署 Metricbeat 把 ES metrics 推到 Cloud Monitoring
helm install metricbeat elastic/metricbeat \
  --set daemonset.enabled=true \
  --set elasticsearch.host=https://production-es-http:9200 \
  --set extraEnvs[0].name=GOOGLE_APPLICATION_CREDENTIALS \
  --set extraEnvs[0].value=/var/run/secrets/google/key.json

更佳做法：用 Elastic Agent + Fleet（CH 60），所有遙測流向 Elasticsearch 自身。

GKE 上常見問題

症狀處理
PD 跨 zone 無法 mountStorageClass 設 WaitForFirstConsumer + 不要設 replication-type: regional-pd 在 ES（已有 replicas）
Workload Identity 無效檢查 node pool --workload-metadata=GKE_METADATA
GCS snapshot 認證失敗確認 GSA 有 storage.objectAdmin，IAM binding 對應到正確的 K8s namespace/SA
Ingress 顯示 502BackendConfig 的 healthCheck path 與實際路徑不一致
節點升級導致 pods evict設定 PodDisruptionBudget；ECK 3.x 預設會建

本章重點回顧

  • 生產用 GKE Standard + Regional cluster；Autopilot 對 ES 限制太多
  • Master pool 用 e2-standard-4，data hot 用 n2-highmem-16 或 t2a/t2d ARM 替代
  • 儲存：pd-ssd 為主流，極致效能用 hyperdisk-extreme；不要設 regional-pd
  • node pool 必須加 --workload-metadata=GKE_METADATA 才能用 Workload Identity
  • 用 Workload Identity 取代 SA key，安全且管理簡單
  • GCS snapshot 配上 lifecycle 自動轉 Nearline/Coldline，極大降本
  • Kibana 用 GCP HTTP(S) LB + ManagedCertificate 自動處理 TLS
  • 跨 zone 流量收費，但 multi-zone 對高可用必要
