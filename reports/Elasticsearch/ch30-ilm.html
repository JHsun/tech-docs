什麼是 ILM？

ILM（Index Lifecycle Management）自動化管理 index 的生命週期：從建立、優化、降溫、最終刪除。它由 5 個 phase 組成：

Rollover Action 詳解

Rollover 是 hot phase 的核心：當條件達成時把 write 切到新 backing index：

條件說明
max_age從建立到現在的時間（如 1d、30d）
max_size所有 primary shard 總大小（已棄用，改用 max_primary_shard_size）
max_primary_shard_size單一 primary shard 大小（推薦！）
max_docs整個 index 文件數
max_primary_shard_docs單一 primary shard 文件數
min_age / min_size 等必須都達成才 rollover（and 邏輯）
💡 Rollover 條件是「OR」 多個 max 條件中任一達成就觸發 rollover。建議同時設定 max_age + max_primary_shard_size：
  • 低流量：依 max_age（避免 shard 太老但太小）
  • 高流量：依 max_primary_shard_size（避免 shard 過大）

建議的 shard 大小

  • 10 GB - 50 GB：Elastic 官方建議的 sweet spot
  • 過小（< 1 GB）：cluster state 與 overhead 浪費
  • 過大（> 50 GB）：recovery 慢、shrink 慢、search latency 高

Warm Phase 操作

Shrink：減少 shard 數

"shrink": {
  "number_of_shards": 1   # 從 N 變 1，省 cluster state 與 overhead
}

限制：原 shard 數必須是新 shard 數的整數倍（如 6 → 1, 2, 3, 6）。會建立新的 shrunk-{name} index 取代原來。

Force Merge：合併 segment

"forcemerge": {
  "max_num_segments": 1
}

合併 segment 為單一個，提升查詢速度且省空間。絕對不要在 hot phase 用（耗 IO）。

Set Priority：recovery 優先序

"set_priority": { "priority": 50 }

cluster 重啟後，priority 高的 index 先 recovery。Hot=100、Warm=50、Cold=10 是常見配比。

前置條件

# 1. 必須先建立 snapshot repository
PUT _snapshot/primary
{
  "type": "s3",
  "settings": {
    "bucket": "es-snapshots",
    "region": "ap-northeast-1"
  }
}

# 2. policy 中引用
"cold": {
  "min_age": "30d",
  "actions": {
    "searchable_snapshot": {
      "snapshot_repository": "primary",
      "force_merge_index":   true     # 先 force merge 再 snapshot
    }
  }
}

Delete Phase

"delete": {
  "min_age": "90d",
  "actions": {
    "wait_for_snapshot": { "policy": "daily-snapshots" },
    "delete":            { "delete_searchable_snapshot": true }
  }
}

關鍵點：

  • wait_for_snapshot：刪除前確認最新 snapshot 已包含此 index（防意外資料丟失）
  • delete_searchable_snapshot：刪 index 時連 searchable snapshot 也刪（預設 true）

套用 Policy

透過 data stream（推薦）

PUT _index_template/logs-app-template
{
  "index_patterns": [ "logs-app*" ],
  "data_stream": { },
  "template": {
    "settings": {
      "index.lifecycle.name": "logs-90d-policy"
    }
  }
}

新建的 data stream 與其 backing indices 自動套用 policy。

套用到既有 index

PUT my-index-2026.04.01/_settings
{
  "index.lifecycle.name": "logs-90d-policy"
}
⚠️ 既有 index 套 policy 不要含 rollover 套用到既有 index 的 policy 不該有 rollover action（rollover 由 alias 或 data stream 管）。

疑難排解

問題：index 卡在 ERROR step

# 看錯誤詳情
curl -u elastic:xxx --cacert ca.crt \
  ".../my-index/_ilm/explain?pretty"

# 修復後重試
curl -X POST -u elastic:xxx --cacert ca.crt \
  ".../my-index/_ilm/retry"

常見錯誤

錯誤原因修復
illegal_state: index has no rollover aliasILM policy 含 rollover 但 index 沒 aliasdata stream 取代，或建立 alias 並設 is_write_index
shrink failed: not enough nodes with attributeshrink action 找不到目標 node檢查 data tier 配置與 allocate filter
snapshot repository not foundpolicy 引用 repo 但 repo 沒建先建 repo
index 在 hot 不肯 rollovermin_age 還沒到，或 max_* 條件未達檢視 explain，確認條件

curl -X POST -u elastic:xxx --cacert ca.crt \

{
  "current_step": {
    "phase":  "hot",
    "action": "rollover",
    "name":   "check-rollover-ready"
  },
  "next_step": {
    "phase":  "warm",
    "action": "shrink",
    "name":   "shrink"

調整 polling interval

ILM 預設每 10 分鐘檢查一次 index 狀態，可加快測試：

PUT _cluster/settings
{
  "persistent": {
    "indices.lifecycle.poll_interval": "1m"
  }
}

ILM vs Data Stream Lifecycle (DSL)

9.x 推出更簡化的 DSL（CH 31）。比較：

項目ILMDSL（Data Stream Lifecycle）
適用對象data streams + 一般 index僅 data streams
功能5 個 phase + 多種 action僅 retention + rollover + downsample
Searchable snapshot✅❌
Shrink / Forcemerge✅❌（自動 tail merge）
複雜度高低
適合有 hot/warm/cold 分層需求單純保留期管理

兩者可共存：在 index template 設 index.lifecycle.prefer_ilm 決定優先序。

ILM Best Practices

  • 先用 Kibana ILM UI 建立、再匯出 JSON：UI 友善又有解釋
  • Rollover 用 max_primary_shard_size（不要用 max_size）
  • 大流量配 max_primary_shard_size:50gb + max_age:30d（OR）
  • Hot phase 不要 forcemerge / shrink，太耗 IO
  • Warm phase 才 shrink + forcemerge
  • Cold/Frozen 用 searchable snapshot，必先建 repo
  • Delete phase 加 wait_for_snapshot，避免資料丟失
  • 所有 data stream 都要設 lifecycle，否則資料無限增長
  • 新建 cluster 逐步測試 phase 轉換（用短 min_age 模擬）
  • 監控 ILM 錯誤：用 watcher 或 audit explain?only_errors=true

本章重點回顧

  • ILM 自動化 5 個 phase：hot → warm → cold → frozen → delete
  • 每 phase 有不同可用 actions：rollover、shrink、forcemerge、searchable_snapshot、delete...
  • Rollover 條件用 OR：任一條件達成即觸發；推薦 max_primary_shard_size + max_age 組合
  • Shard 大小甜蜜點 10-50 GB
  • Searchable snapshot 把資料放 S3 但仍可查；cold = full mount，frozen = partial
  • Delete 前用 wait_for_snapshot 防意外資料丟失
  • 所有 data stream 都必須設 ILM 或 DSL（否則資料無限增長）
  • 診斷三大 API：_ilm/explain、_ilm/status、_ilm/retry
  • 9.x 起 DSL 是 ILM 的輕量替代，純為 data stream 而生
