1. Request Flow — 9 階段處理流程

1
Transport Layer
FastHTTP 解析 HTTP 請求（~2µs）。Go SDK 模式直接進入核心，跳過 HTTP 層。
2
Request Routing & Load Balancing
Provider 選擇邏輯：加權隨機 Key 選擇（~10ns）。KeySelector 使用累積權重演算法。
3
Plugin Pipeline — PreHooks
HTTPTransportPreHook（可短路）→ PreLLMHook（auth, rate-limit, cache check）。按設定順序執行。
4
MCP Tool Discovery & Injection
如果請求含 tool_choice，注入已連接 MCP Server 的工具定義。Code Mode 下只注入 4 個 meta-tools。
5
Memory Pool Management
從 sync.Pool 取得預分配物件（channel msg, response, error channel 等），避免 GC 壓力。每個物件有唯一 ID 和時間戳追蹤。
6
Worker Pool Processing
請求進入 Provider 專屬的 buffered channel（佇列），Worker goroutine 取出處理。每個 Provider 獨立隔離。
7
Provider API Communication
Worker 準備 HTTP 請求、加入認證、使用 fasthttp client（連線池）發送。含 context timeout 控制。
8
Tool Execution & Response
如果回應含 tool_calls：Agent Mode 自動執行（平行 goroutine）或回傳給應用審核。
9
PostHooks & Serialization
PostLLMHook（反向執行）→ HTTPTransportPostHook → 回應序列化 → HTTP Response。
Key 選擇演算法（偽碼）
// 加權隨機選擇（~10ns）
func (ks *KeySelector) SelectKey() *APIKey {
    r := rand.Float64() * ks.total  // total = 所有 weight 加總
    cumulative := 0.0
    for i, weight := range ks.weights {
        cumulative += weight
        if r <= cumulative {
            return &ks.keys[i]
        }
    }
    return &ks.keys[len(ks.keys)-1]
}

2. Concurrency 架構

Worker Pool（每 Provider 獨立）
元件說明
Worker Goroutines預生成，啟動時建立（減少延遲）
Job QueueBuffered channel，平滑負載
HTTP Client Poolfasthttp 連線池，每 Provider 獨立
Health MonitorCircuit breaker 偵測並隔離失敗 Provider
Graceful ShutdownWorker 完成當前工作後才終止
Memory Pool（sync.Pool）
Pool 類型內容
Channel Messages請求包裝器
Response Channels回應通道
Error Channels錯誤通道
Stream Channels串流通道
Plugin PipelinesPlugin 執行管線
Provider Response Objects如 AnthropicMessageResponse, OpenAIResponse

每個物件有唯一 ID、時間戳追蹤、可重用性旗標。Pool 使用 sync.Pool + sync.RWMutex 確保執行緒安全。

Goroutine 管理
機制說明
Goroutine 上限防止資源耗盡
Atomic 計數器追蹤當前活躍數
Idle Timeout閒置 goroutine 自動清理
Spawn Channel新建透過 channel 追蹤
Completion Monitor完成時觸發清理信號
Streaming 架構

framework/streaming 套件：Accumulator 管理串流生命週期，使用 sync.Map 追蹤多個並行請求的 chunk，sync.Pool 回收 StreamChunk 物件。包含定期清理 worker 處理孤立的 accumulator。

3. Performance Tuning — 公式與實戰

三大參數
參數範圍預設說明
concurrencyPer Provider1000Worker goroutine 數量
buffer_sizePer Provider5000請求佇列大小（必須 ≥ concurrency）
initial_pool_sizeGlobal5000sync.Pool 預分配物件數
核心公式
concurrency = expected_RPS
buffer_size = 1.5 × expected_RPS
initial_pool_size = 1.5 × total_RPS (所有 Provider 加總)
initial_pool_size ≥ max(所有 Provider 的 buffer_size)

多節點部署
Per-Node = Total ÷ Node 數量 
# 範例：10,000 RPS / 4 節點
{
  "config": {
    "initial_pool_size": 3750,       // 15000 ÷ 4
    "drop_excess_requests": false
  },
  "providers": {
    "openai": {
      "concurrency_and_buffer_size": {
        "concurrency": 2500,          // 10000 ÷ 4
        "buffer_size": 3750           // 15000 ÷ 4
      }
    }
  }
}
Queue Overflow 處理
模式drop_excess_requests行為
Blocking（預設）false佇列滿時阻塞，等待空間
Droptrue佇列滿時立即丟棄並回傳錯誤
💡 生產建議：使用 drop_excess_requests: true + buffer_size = 1.5× concurrency。防止記憶體耗盡，同時處理合理流量尖峰。
Provider 限速注意

不要將 concurrency 設得比 Provider 的實際 rate limit 高，否則只會被限速。例如 Anthropic rate limit 較低時：

"anthropic": {
  "concurrency_and_buffer_size": {
    "concurrency": 25,    // 尊重 Anthropic rate limit
    "buffer_size": 100
  }
}

4. Observability 完整版

Logging Plugin 運作機制
階段動作
PreLLMHook捕獲請求 metadata（provider, model, input, params）
Async Processing背景 goroutine 寫入日誌（sync.Pool 優化）

WebSocket即時廣播至 UI
