2. Routing Rules（動態 CEL 表達式路由）

在 Governance 選擇 Provider 之前執行，可覆寫 Provider/Model。使用 CEL（Common Expression Language） 在請求時動態評估條件。

📐 Scope 優先順序（第一個匹配即中止）
VirtualKey（最高）→ Team→ Customer→ Global（最低）

同一 Scope 內按 priority 升序排列（0 先於 10）。

📦 CEL 可用變數
類別變數說明
請求model請求的模型名稱
provider目前的 Provider
request_typechat_completion / embedding / batch / image_generation …
Headerheaders["x-tier"]HTTP Header（不分大小寫）
params["region"]Query 參數
組織virtual_key_id / _nameVK 資訊
team_id / _name團隊
customer_id / _name客戶
容量budget_used預算使用率 0-100%
tokens_usedToken 限制使用率
request請求限制使用率

3. Provider Routing 完整流程

Bifrost 提供三層路由機制，依優先順序執行：

① Routing Rules (CEL)→ ② Governance (VK 加權)→ ③ Adaptive LB (Enterprise)
🏛️ Governance-based Routing

透過 Virtual Key 的 provider_configs 明確定義路由。

// VK 的 provider_configs 範例
{
  "provider_configs": [
    {"provider":"openai",  "allowed_models":["gpt-4o","gpt-4o-mini"], "weight":0.3,
     "budget":{"max_limit":100.0,"current_usage":45.0}},
    {"provider":"azure",   "allowed_models":["gpt-4o"],               "weight":0.7,
     "rate_limit":{"token_max_limit":100000,"token_reset_duration":"1m"}}
  ]
}
// → 70% Azure, 30% OpenAI，自動依預算/限速排除不可用者
功能說明
加權分配依 weight 機率選擇 Provider
預算執行自動排除超預算的 Provider
限速保護跳過已達限速的 Provider
自動 Fallback失敗後按 weight 降序嘗試其他 Provider
allowed_models[] = 用 Model Catalog；["gpt-4o"] = 僅允許指定模型
⚡ Adaptive Load Balancing（Enterprise）

雙層架構：Level 1（Provider 選擇）+ Level 2（Key 選擇）

Level執行時機演算法
L1: Providermodel 無 / 前綴時效能評分：Error(50%) + Latency(20%) + Util(5%) - Momentum
L2: Key永遠執行依錯誤率、延遲、TPM hits、健康狀態評分
💡 關鍵：即使 Provider 已由 Governance 或 Routing Rules 決定，L2 Key 選擇仍會自動優化選用效能最佳的 API Key。
📊 三層路由互動總結
情境Provider 選擇Key 選擇
VK 有 provider_configsGovernance（加權隨機）Standard 或 Adaptive
VK 無 provider_configs + LBLB Level 1（效能）LB Level 2（效能）
Model 含 provider/ 前綴跳過（已指定）LB Level 2 ✅
無 LBGovernance / 使用者指定靜態 weight
📂 Model Catalog

Bifrost 的模型註冊表，結合兩個資料來源：

來源同步頻率用途
Pricing Data（遠端 URL）啟動時 + 每 24 小時模型定價、Provider 映射
Provider List Models API啟動時 + Provider 變更時實際可用模型、新發布模型

支援跨 Provider 解析：claude-3-5-sonnet 可自動找到 Anthropic、Vertex、Bedrock、OpenRouter。

      "embedding": true

  }
}'

5. MCP Gateway 總覽

MCP（Model Context Protocol）讓 AI 模型在執行時動態發現和執行外部工具。Bifrost 同時作為 MCP Client 和 MCP Server。

🔒 安全第一設計
⚠️ 預設行為：Bifrost 不會自動執行 LLM 回傳的 tool calls。所有工具執行都需要透過明確的 API 呼叫。
原則說明
明確執行LLM 的 tool calls 只是建議，需另行呼叫 /v1/mcp/tool/execute
精細控制可按請求、客戶端、Virtual Key 過濾工具
選擇性自動執行Agent Mode 需明確設定 tools_to_auto_execute
無狀態每次 API 呼叫獨立，應用控制對話狀態
🔗 連接類型
類型說明認證適用場景
STDIO啟動子程序，透過 stdin/stdout 通訊無本機工具、CLI
HTTPHTTP 端點Headers / OAuth 2.0遠端 API、微服務
SSEServer-Sent Events 持久連線Headers / OAuth 2.0即時資料、串流
新增 MCP Client 範例
STDIO（本機）
POST /api/mcp/client
{
  "name": "filesystem",
  "connection_type": "stdio",
  "stdio_config": {
    "command": "npx",
    "args": ["-y",
      "@anthropic/mcp-filesystem"],
    "envs": ["HOME","PATH"]
  },
  "tools_to_execute": ["*"]
}
HTTP + OAuth
POST /api/mcp/client
{
  "name": "web_search",
  "connection_type": "http",
  "connection_string":
    "https://mcp.example.com/mcp",
  "auth_type": "oauth",
  "oauth_config": {
    "client_id": "your-id",
    "client_secret": "your-secret",
    "authorize_url": "https://...",
    "token_url": "https://...",
    "scopes": ["read","write"]
  },
  "tools_to_execute": ["*"]
}
📡 MCP API 端點
POST   /api/mcp/client            # 新增 MCP Client
GET    /api/mcp/clients            # 列出所有 Client（含工具 & 狀態）
PUT    /api/mcp/client/{id}        # 更新 Client
DELETE /api/mcp/client/{id}        # 刪除 Client
POST   /v1/mcp/tool/execute        # 執行工具（chat 格式）
POST   /v1/mcp/tool/execute?format=responses  # Responses 格式

6. MCP Tool Execution 流程

三步驟標準流程
① POST /v1/chat/completions→ LLM 回傳 tool_calls→ ② 審核 + 核准→ ③ POST /v1/mcp/tool/execute→ ④ 帶結果繼續對話 
# Step 1: 送出請求 → LLM 回傳 tool_calls
# Response: finish_reason: "tool_calls"
#   tool_calls: [{id:"call_xyz", function:{name:"filesystem_list_directory", arguments:"{...}"}}]

# Step 2: 執行工具（直接把 tool_call 物件貼過去）
curl -X POST http://localhost:8080/v1/mcp/tool/execute \
  -H "Content-Type: application/json" \
  -d '{"id":"call_xyz","type":"function","function":{"name":"filesystem_list_directory","arguments":"{\"path\":\".\"}"}}'

# Response: {role:"tool", content:"[...]", tool_call_id:"call_xyz"}

# Step 3: 組裝完整對話歷史，繼續呼叫 /v1/chat/completions
#   messages: [user, assistant(tool_calls), tool(result)]
💡 工具命名規則：工具名稱自動加上 MCP Client 名稱前綴（如 filesystem_list_directory），確保跨 Client 唯一性。

7. Agent Mode（自動執行）

讓 Bifrost 自動執行 LLM 回傳的工具呼叫，循環直到完成或達到最大深度。v1.4.0+

🚨 不支援串流：Agent Mode 不相容 streaming（chat_stream / responses_stream），僅可用於非串流端點。
兩個關鍵設定
欄位語意說明
tools_to_execute白名單LLM 可見的工具。["*"]=全部, []=無
tools_to_auto_execute自動執行子集必須是 tools_to_execute 的子集
# 設定：讀取操作自動執行，寫入需人工審核
POST /api/mcp/client
{
  "name": "filesystem",
  "connection_type": "stdio",
  "stdio_config": {"command":"npx","args":["-y","@anthropic/mcp-filesystem"]},
  "tools_to_execute": ["*"],
  "tools_to_auto_execute": ["read_file", "list_directory"]
}

# 全域 Agent 設定
PUT /api/settings/mcp/tool-manager-config
{
  "max_agent_depth": 15,           // 最大迴圈次數（預設 10）
  "tool_execution_timeout": "45s"  // 單一工具逾時（預設 30s）
}
Agent Loop 行為

① LLM 回傳 tool_calls → ② 分類 auto/non-auto → ③ 自動執行 auto 工具（支援平行）→ ④ 結果回饋 LLM → ⑤ 重複直到無 tool_calls 或達 max_depth → ⑥ non-auto 工具回傳給應用審核

8. Code Mode（AI 寫程式編排工具）

當連接 3+ 個 MCP Server 時，強烈建議使用 Code Mode。v1.4.0+

💡 效果：Token 用量減少 50%+，延遲降低 40-50%。
問題 & 解法
經典 MCP（5 Server, 100 工具）Code Mode
每次請求注入 100 個工具定義 → 巨量 Token只注入 4 個 meta-tools
多工具需多次 LLM 來回AI 寫 TypeScript 一次編排多工具
四個 Meta-Tools
工具功能
listToolFiles列出可用的工具檔案（按 Server 或 Tool 分組）
readToolFile讀取特定工具的文件
getToolDocs取得工具的詳細文件（參數 schema 等）
executeToolCode執行 AI 撰寫的 TypeScript 程式碼
Binding Level
LevellistToolFiles 回傳適用場景
server（預設）Server 名稱列表少量 Server，工具分組清晰
tool個別工具名稱列表大量工具，需精確選擇
# 啟用 Code Mode
PUT /api/settings/mcp/tool-manager-config
{
  "code_mode_binding_level": "tool"  // "server" 或 "tool"
}

9. MCP Gateway URL / Tool Hosting / Filtering

MCP Gateway URL（Bifrost 作為 MCP Server）

將 Bifrost 暴露為 MCP Server，讓 Claude Desktop 等 MCP Client 直接連接。

# Bifrost MCP Server endpoint:
http://localhost:8080/mcp/sse

# Claude Desktop 設定（claude_desktop_config.json）
{
  "mcpServers": {
    "bifrost": {
      "url": "http://localhost:8080/mcp/sse"
    }
  }
}
Tool Hosting（自訂工具）

在 Go 應用中直接註冊自訂工具並透過 MCP 暴露。使用 Go SDK 的 MCPConfig.HostedTools 或透過 API 動態註冊。

🔍 Tool Filtering

三層過濾機制：

層級方式說明
Client 層tools_to_executeMCP Client 設定時指定可用工具
請求層Header x-bf-mcp-include-clients
x-bf-mcp-include-tools		每次請求動態過濾
VK 層Virtual Key MCP 工具白名單Per-VK 嚴格 allow-list
# 請求層過濾：只使用 filesystem client 的 read_file 工具
curl http://localhost:8080/v1/chat/completions \
  -H "x-bf-mcp-include-clients: filesystem" \
  -H "x-bf-mcp-include-tools: filesystem_read_file" \
  -d '{"model":"openai/gpt-4o","messages":[...]}'
