CHAPTER 9 應用層隧道穿越

本章導讀

前面八章都在處理「NAT 無心阻擋」這類問題。本章轉到光譜的另一端： 當你面對的是有意阻擋的防火牆——企業 NGFW、學校網管、國家審查—— 傳統穿越技術通通失效。此時的工程哲學從「請設備配合」變成「不讓設備發現」。 我們會看 HTTP CONNECT、SOCKS、TLS 偽裝、WebSocket、域前置（domain fronting）、 以及流量混淆（traffic obfuscation）。本章也會明確界定技術倫理界線—— 這些技術是雙面刃。

📚 學習目標

  • 能說明「協作型穿越」與「對抗型穿越」的根本哲學差異。
  • 能描述 HTTP CONNECT、SOCKS5、SSH 動態 forward 三種傳統隧道的差異。
  • 能解釋為什麼 TLS over 443 是現代企業環境的最後保險。
  • 能說明域前置（domain fronting）的原理與其終結（CDN 政策變化）。
  • 能列舉至少三種流量混淆技術並說明其對抗的 DPI 層級。
  • 能對技術倫理風險做出明確判斷。

9.1 兩種「穿越」的哲學差異

第 1 章我們區分過兩類阻擋。本章對應的是其中比較棘手的一類：

協作型穿越（Ch.4-8）

對 NAT 說：「請開個洞讓我用」
技術重點：探測、協調、重試
成本：低（普通工程）
倫理：通常無爭議

對抗型穿越（本章）

對防火牆說：「我不是你想攔的那種流量」
技術重點：偽裝、混淆、規避
成本：高（要對抗持續演進的偵測）
倫理：高度依場景

技術倫理前提 本章描述的技術，部分在某些司法管轄區或組織政策下使用是違規甚至違法的。 作者提供這些知識是為了讓你理解它們如何運作——以便：
  • 在合法場景（如：在自家網路除錯、合規的安全研究、保護記者來源）中正確使用
  • 在企業/政府防禦角色中能識別與防護
  • 對它們的技術極限與倫理代價有清醒認知
你需對自己的行為負責。

9.2 HTTP CONNECT：最古老的隧道

HTTPS 普及之前，企業 proxy 早就需要「轉送加密流量」這個能力。 RFC 7231 標準化了 CONNECT 方法—— 客戶端透過 HTTP 請求 proxy 開一條到目的地的「透明 TCP 通道」。

9.2.1 CONNECT 的基本流程

────── 客戶端 → Proxy ──────
CONNECT example.com:443 HTTP/1.1
Host: example.com:443
User-Agent: Mozilla/5.0 ...
Proxy-Authorization: Basic dXNlcjpwYXNz   (可選)

────── Proxy → 客戶端 ──────
HTTP/1.1 200 Connection Established
Proxy-Agent: corporate-proxy/1.0

────── 此後雙向透明轉送 raw TCP bytes ──────
[客戶端開始 TLS 握手]
[example.com:443 收到 TLS ClientHello]
...

Proxy 看到 200 回應後，停止解析 HTTP，純粹當 TCP 中繼。 這讓 HTTPS 流量可以穿越企業 proxy 而 proxy 無法（不解密的話）看到內容。

9.2.2 CONNECT 的合法用途與限制

合法用途

  • 正規的企業 HTTPS 透明轉送
  • SSH over HTTPS（透過 proxy 連 SSH 跳板機）
  • RDP、VNC、TURN over TLS 等 TCP-based 服務的代理

常見限制

  • Proxy 通常白名單目的 port（只允 443）
  • 需要 Proxy-Authorization（Basic / NTLM / Kerberos）
  • TLS Inspector 可能解密 CONNECT 後的內容
  • 無法承載 UDP（CONNECT 是 TCP-only）

9.3 SOCKS：通用代理協議

SOCKS（最初是 SOCKetS 的縮寫）是另一種流行的代理協議，比 HTTP CONNECT 更通用。 SOCKS5（RFC 1928）是當前主流版本，支援 TCP 與 UDP。

9.3.1 SOCKS5 握手流程

sequenceDiagram participant C as Client participant S as SOCKS5 Server participant T as Target Note over C,S: 階段 1：認證協商 C->>S: VER=5 NMETHODS=2 [no-auth, user/pwd] S->>C: VER=5 METHOD=user/pwd Note over C,S: 階段 2：使用者認證 (RFC 1929) C->>S: USER alice PASS secret S->>C: STATUS=success Note over C,S: 階段 3：請求建立連線 C->>S: CMD=CONNECT ATYP=domain
ADDR=example.com PORT=443 S->>T: TCP connect to example.com:443 S->>C: REP=success BND.ADDR=... BND.PORT=... Note over C,T: 階段 4：透明轉送 C->>S: TCP payload S->>T: 轉送 T->>S: TCP payload S->>C: 轉送

9.3.2 SOCKS vs HTTP CONNECT

面向 HTTP CONNECT SOCKS5
支援協議TCPTCP + UDP（透過 UDP ASSOCIATE）
位址類型DNS 名稱 + portIPv4 / IPv6 / domain
認證HTTP Basic / NTLM / KerberosSOCKS5 user/pwd / GSSAPI

協議開銷HTTP 訊息頭較小（二進位）

9.3.3 SOCKS 在 P2P 中的角色

Tor 客戶端對應用提供 SOCKS5 介面（預設 9050），所以任何支援 SOCKS5的應用， 幾乎都能透過 Tor 跑。這也是為什麼許多 P2P 軟體（包括 BitTorrent client）支援 「把流量丟給 SOCKS5」這個選項——便於串接到 Tor、Shadowsocks 等隱私工具。

9.4 SSH 動態端口轉送：個人版隧道

如果你有一台公網的 SSH 主機，OpenSSH 內建一個強大功能： -D 選項把 SSH 連線變成本機 SOCKS5 代理。 

# 在你的筆電上執行
$ ssh -D 1080 -N user@my-server.example.com

# 此時本機 localhost:1080 = SOCKS5 proxy
# 任何丟到此 proxy 的流量都從 my-server.example.com 出去

# 設定瀏覽器 proxy = SOCKS5 localhost:1080
# 或讓特定指令走它：
$ curl --socks5 localhost:1080 https://example.com
$ git -c http.proxy=socks5://localhost:1080 clone ...

這個技巧在出差、咖啡廳、機場 Wi-Fi、或限制較多的網路環境中極為實用—— 只要 SSH（port 22 或 443）能出，所有流量都能間接出去。 本書也用這個技巧做過 P2P 應用的內網除錯。

9.5 TLS over 443：終極偽裝

企業防火牆幾乎一定放行對外的 HTTPS（port 443）流量——這是現代網際網路的命脈。 把任何協議封裝成「看起來像 HTTPS」的流量丟到 443，是最廣泛的隧道策略。

9.5.1 TLS 隧道的層次

┌────────────────────────────┐
│  [應用流量：BT / SSH / VPN] │
├────────────────────────────┤
│  [偽裝層：HTTP/2 / WebSocket]│  ← 讓流量「看起來像」HTTPS
├────────────────────────────┤
│  [TLS 1.3]                  │  ← 加密
├────────────────────────────┤
│  [TCP, port 443]            │  ← 偽裝成 HTTPS
└────────────────────────────┘

9.5.2 為什麼 TLS-only 不夠？

「我把 SSH 包進 TLS 不就好了？」這個直覺對抗不了現代 DPI。 TLS 提供了加密，但 TLS 握手本身有 fingerprint：

  • JA3/JA4 fingerprint：根據 ClientHello 的 cipher suites 順序、extensions、ALPN，產生指紋——能區分 OpenSSL、BoringSSL、各瀏覽器、Tor
  • SNI（Server Name Indication）：明文中帶有目的網域，讓 DPI 直接看到你連哪
  • 封包大小與時序：HTTPS 瀏覽是 burst-and-idle，影音串流是穩定流，BT 是雙向對等——機器學習分類器一眼識破

9.5.3 ECH（Encrypted Client Hello）

IETF 正在標準化 ECH（Encrypted ClientHello），把 SNI 也加密。 Cloudflare 與部分瀏覽器已開始實驗性部署。一旦 ECH 普及，DPI 將難以根據 SNI 過濾—— 這是隱私保護方面的重大進展，但也使得防火牆運維更難。

9.6 WebSocket：對 P2P 友善的隧道

WebSocket（RFC 6455）原本是為瀏覽器即時通訊設計的協議—— 從 HTTP/HTTPS 升級為持久雙向連線。它的特性對 P2P 隧道極友善：

跑在 80/443 之上

對防火牆而言，WebSocket 連線跟一個普通 HTTP 升級請求沒兩樣， 幾乎所有環境都能穿越。

瀏覽器原生支援

純 JavaScript 即可建立連線，不需要插件或特殊權限。 libp2p 的 browser node、WebRTC signaling 都常用。

雙向訊息

不像普通 HTTP 是 request-response，WebSocket 是 full-duplex。 可承載任何 P2P 訊號協議。

可承載二進位

支援 binary frames，可以裝任何協議的 raw 封包。 libp2p 的 ws/wss transport 直接這樣用。

9.6.1 WebSocket 升級握手

────── Client → Server ──────
GET /ws HTTP/1.1
Host: signal.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

────── Server → Client ──────
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

9.7 域前置（Domain Fronting）：曾經的銀彈

域前置是一個極具爭議的技巧——利用 CDN 的特性， 讓你連線 看似連到一個無爭議的網站（如 google.com）， 但 HTTPS 內部的 Host header 指向真正的目標。

9.7.1 原理

sequenceDiagram participant C as Client (受審查網路) participant FW as 防火牆 participant CDN as CDN (例：Google) participant T as 目標伺服器 Note over C,T: 設定：CDN 同時為 google.com 與 secret.org 服務 C->>FW: TLS ClientHello SNI=google.com Note over FW: 看到 SNI 是 google.com
放行（無人敢擋 Google） FW->>CDN: 轉送 Note over C,CDN: TLS 握手完成，建立加密通道 C->>CDN: HTTP/2 加密請求
:authority = secret.org Note over CDN: 看內部 Host header
路由到 secret.org CDN->>T: 轉送請求 T->>CDN: 回應 CDN->>C: 回應 (走原連線)
圖 9-2 ｜ 域前置：外面看是 google.com，裡面實是 secret.org

Tor 的 meek pluggable transport、Telegram、Signal 等都曾使用過域前置， 在各國防火牆面前極為有效——攔截 google.com 等於攔截整個搜尋引擎，沒有國家敢這樣做。

9.7.2 域前置的終結

2018 年的轉折 2018 年 4 月，Google 與 AWS 幾乎同時宣布禁止域前置。 CDN 開始要求 SNI 與 Host header 一致，違反就拒絕服務。 Signal 因此失去了在伊朗、阿拉伯聯合大公國等地穿透審查的能力。 域前置的時代基本結束，新一代技術轉向 ECH 與其他混淆方案。

9.8 流量混淆（Traffic Obfuscation）

即使你成功用 TLS 包裝流量，DPI 仍可能根據流量行為識破—— 封包大小分布、時序、雙向比例。對抗這類偵測需要主動偽造流量特徵， 讓你的流量看起來像目標流量類型。

9.8.1 主流混淆技術譜系

技術 偽裝目標 對抗的偵測層級 狀態
obfs2 / obfs3 隨機看似熵 關鍵字 / signature 已棄用
obfs4 偽隨機 + IAT 模糊化 主動探測 + 流量分析 仍主流（Tor）
meek HTTPS over CDN SNI/Host 過濾 因域前置被禁而衰退
Snowflake 偽裝成 WebRTC 協議分類器 Tor 重要工具
Shadowsocks 無狀態加密代理 主動探測（已被部分破解） Shadowsocks-2022 改良中
v2ray / Xray 多層偽裝協議 機器學習分類 持續演化
WireGuard over TLS 正常 HTTPS 流量 UDP 阻擋 新興

9.8.2 obfs4 的設計亮點

obfs4 是 Tor Pluggable Transport 中最成熟的混淆協議，幾個關鍵設計：

  • Bridge identifier：客戶端必須先有伺服器的 public key（防主動探測）
  • 偽隨機加密：所有 byte 看起來像均勻隨機，沒有可識別 fingerprint
  • IAT (inter-arrival time) 模糊化：隨機 padding 與延遲，破壞時序分析
  • 抗主動探測：無 key 的探測者收到「看似隨機」的回應，無法確認是不是 obfs4 server

9.8.3 Snowflake：用 WebRTC 對抗審查

Snowflake 是個聰明的設計——把 Tor 流量偽裝成普通的 WebRTC視訊通話。 防火牆若擋 WebRTC，會誤傷大量視訊會議使用者，所以幾乎不能擋。 Snowflake 的「proxy」是普通使用者透過瀏覽器擴充當志願者參與，動態提供入口—— 防火牆難以維護黑名單。

Snowflake 對 P2P 開發者的啟示 Snowflake 證明了：P2P 技術本身能成為審查規避工具。 這不是因為 P2P 有特殊隱私，而是因為 P2P 流量已經夠普遍—— 視訊通話、雲端遊戲、Tailscale-like VPN——擋它的代價已經很高。 「大眾化」本身就是一種反審查武器。

9.9 透過非典型協議的隧道

Wikipedia 的 Tunneling protocol 條目清楚指出： 能繞過防火牆的隧道方法包括 DNS、MQTT、SMS 等不同協議。 這些「不像通訊管道的通訊管道」在某些極端封鎖環境下是最後手段。

9.9.1 DNS 隧道

DNS 永遠不會被擋——即使最嚴格的網路也得讓 DNS 出去解析網域。 DNS 隧道把資料編碼進 DNS 查詢與回應： 

客戶端要送資料 "Hello"：
  DNS 查詢: aGVsbG8.tunnel.attacker.com  ← Base32 編碼的 "Hello"
  
DNS 伺服器（攻擊者控制）回應：
  TXT record = "dGFhYXM"  ← 編碼回送的資料

優點：幾乎所有網路都放行 DNS
缺點：頻寬極低（每次查詢只能編幾十 byte）
      偵測：行為異常的 DNS 查詢（過長、過頻、TXT 為主）容易被 IDS 發現

工具：iodine（IPv4-over-DNS）、dnscat2

DNS 隧道頻寬低（典型 1-100 Kbps），不適合一般使用， 但作為緊急回連通道（exfiltration、命令通道）在攻擊與防禦領域都有實際應用。

9.9.2 ICMP 隧道

類似 DNS，把資料藏在 ICMP echo request/reply 的 payload 中。 比 DNS 更冷門但實作更簡單。工具：ptunnel。

9.10 防禦視角：怎麼識別這些隧道

本章不只給攻擊面知識，也應提供防禦面的對應思路：

對抗 HTTPS 隧道

  • TLS Inspector 解密檢查（破壞 E2E）
  • SNI 白名單
  • JA3/JA4 fingerprint 比對
  • 流量行為分析

對抗 DNS 隧道

  • DNS 流量基線分析（query 長度、頻率）
  • 限制 TXT/NULL 記錄查詢
  • 強制使用受控 DNS resolver
  • 查詢量異常告警

對抗混淆代理

  • 機器學習流量分類
  • IP reputation 黑名單（雲端 VPS 公開 range）
  • 主動探測（probe + fingerprint）

策略層

  • 零信任架構（ZTNA）取代邊界
  • EDR 在端點側監控
  • 網路流量加密 + 集中認證

9.11 倫理界線：什麼是合理使用？

明確的倫理界線

本章描述的技術，使用合理性從無爭議到明顯違法是一個光譜：

  • ✅ 明確合理：在自家網路測試、合規的安全研究、保護記者來源、突破國家審查保命
  • ⚠️ 需評估：繞過學校封鎖、家長控制、隱私保護用途
  • ❌ 明確不合理：未授權繞過企業政策、繞過反詐騙系統、攻擊他人網路、商業間諜

技術知識本身中立，但使用上下文不是。讀完本書你需要養成的判斷力是： 在按下 enter 之前，先問自己「我若被別人這麼對待，會怎麼想？」

9.12 章末提要

本章核心觀念
  • 對抗型穿越的核心哲學是偽裝，不再是請求配合。
  • HTTP CONNECT、SOCKS5、SSH -D 是傳統三大隧道工具。
  • TLS over 443 是現代企業環境的最後保險，但 TLS fingerprint（JA3/JA4）使其不再隱形。
  • WebSocket 是對 P2P 友善的隧道載體，廣泛用於 signaling 與 libp2p。
  • 域前置曾是反審查銀彈，2018 年因 CDN 政策變化基本失效。
  • 流量混淆是與 DPI 的軍備競賽：obfs4、Snowflake、Shadowsocks 各有戰場。
  • 非典型協議隧道（DNS、ICMP）頻寬低但極難擋，用於最後手段。
  • 所有對抗型穿越都是持續演化的——沒有一次部署一勞永逸的方案。

  1. 你在咖啡廳發現 Wi-Fi 限制只能訪問特定網站。請設計一個合法的方案， 讓你能用自己家裡的網路安全工作。需要哪些前置條件？
  2. JA3/JA4 fingerprint 是怎麼計算的？為什麼即使不解密 TLS 也能識別客戶端類型？
  3. 研究 Snowflake 的工作原理。為什麼防火牆難以區分「真的 WebRTC 視訊」vs「Snowflake 包裝的 Tor 流量」？
