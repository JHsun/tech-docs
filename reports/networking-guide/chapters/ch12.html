CHAPTER 12 WebRTC：瀏覽器中的 P2P

本章導讀

WebRTC 是把 P2P 帶進每個瀏覽器的歷史性協議——它讓「一支 JavaScript」就能跑視訊通話、檔案傳輸、 即時資料同步，而不需要中央伺服器中繼。本章把你前面九章學到的所有東西整合起來看： STUN/TURN/ICE 在 RTCPeerConnection 中的角色、SDP offer/answer 模型、DataChannel 與媒體軌道的差異、 DTLS-SRTP 加密、以及生產環境的實務（健康的 TURN 使用率應在 20-30%）。 讀完本章，你能理解每一通 Google Meet、Discord 語音、線上教育課程背後的協議架構。

📚 學習目標

  • 能描述 WebRTC 的協議堆疊（API、ICE、DTLS、SRTP/SCTP、UDP/TCP）。
  • 能解釋 SDP offer/answer 模型與信令通道（signaling）的角色分離。
  • 能寫出最小的 RTCPeerConnection / RTCDataChannel JS 範例。
  • 能說明 DTLS-SRTP 為什麼比單純 TLS 更適合即時媒體。
  • 能解釋多方通話的三種架構（Mesh / SFU / MCU）與其取捨。
  • 能評估生產環境的 TURN 使用率指標。

12.1 WebRTC 的歷史定位

WebRTC 由 Google 在 2011 年發起，後由 W3C 與 IETF 共同標準化， 在 2021 年達到 W3C Recommendation 狀態，並持續演進中。 它的核心成就是把原本只有桌面應用能做的 P2P 即時通訊， 變成網頁開箱即用的能力。

WebRTC 改變的世界 WebRTC 出現之前，要在瀏覽器做視訊通話必須裝 Flash 或 Java applet。 WebRTC 出現之後：Google Meet、Jitsi、Discord、Zoom（部分功能）、Houseparty、Clubhouse、 甚至雲端遊戲串流（GeForce Now、Stadia 早期）都建立在它之上。它是 2010 年代最被低估的 web 技術變革。

幾個關鍵觀察：

  • 所有應用都走加密：WebRTC 不允許明文，DTLS 是必要層
  • 媒體與資料分流：媒體用 SRTP（為即時優化）、可靠資料用 SCTP（提供 stream 抽象）
  • 一條 5-tuple 多工：同一條 UDP 連線上跑 ICE、DTLS、SRTP、SCTP——靠首 byte 區分（demux）

12.3 三大 API：getUserMedia、RTCPeerConnection、RTCDataChannel

12.3.1 getUserMedia：取得本機媒體

// 取得本機麥克風與相機
const stream = await navigator.mediaDevices.getUserMedia({
  video: { width: 1280, height: 720 },
  audio: true
});

// 顯示自己的影像
document.getElementById('localVideo').srcObject = stream;

這個 API 觸發瀏覽器權限對話框。瀏覽器負責所有底層硬體存取與編碼器選擇，應用拿到的是抽象的 MediaStream。

12.3.2 RTCPeerConnection：核心物件

這是 WebRTC 的中樞——前面學的 ICE、TURN、SDP 都是它的內部運作。 最小可用例子： 

const pc = new RTCPeerConnection({
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },
    {
      urls: 'turn:turn.example.com:3478',
      username: 'user',
      credential: 'pass'
    }
  ]
});

// 加入本地媒體
stream.getTracks().forEach(track => pc.addTrack(track, stream));

// 收到對方媒體時
pc.ontrack = event => {
  document.getElementById('remoteVideo').srcObject = event.streams[0];
};

// ICE 候選產生時通知對方（透過自訂 signaling）
pc.onicecandidate = event => {
  if (event.candidate) {
    signaling.send({ type: 'ice', candidate: event.candidate });
  }
};

// 連線狀態變化
pc.oniceconnectionstatechange = () => {
  console.log('ICE state:', pc.iceConnectionState);
};

12.3.3 RTCDataChannel：純資料的 P2P

WebRTC 不只能傳媒體——RTCDataChannel 提供雙向、可靠（或不可靠）的資料通道， 用法類似 WebSocket 但是 P2P： 

// 建立資料通道（必須在 setLocalDescription 之前，否則需 renegotiation）
const channel = pc.createDataChannel('chat', {
  ordered: true,        // 是否保證順序
  maxRetransmits: 0     // 最多重傳次數（0 = 不重傳，類似 UDP）
});

channel.onopen = () => console.log('Channel open');
channel.onmessage = e => console.log('Got:', e.data);

// 對方端
pc.ondatachannel = event => {
  const ch = event.channel;
  ch.onmessage = e => console.log('Peer says:', e.data);
};

// 送訊息
channel.send('Hello P2P');
channel.send(arrayBuffer);  // 也可送 binary

DataChannel 可以設定為「可靠+有序」（同 TCP）、「可靠但不保證順序」、 或「不可靠」（同 UDP）。WebTorrent 用 DataChannel 傳檔案分塊； 許多多人遊戲用不可靠模式傳遊戲狀態。

12.4 SDP Offer/Answer 模型

WebRTC 的協商過程遵循 SDP 的 offer/answer 模型（RFC 3264）。 發起方產生 offer（描述自己的媒體能力與 ICE 候選），接收方回應 answer（描述自己的對應選擇）。

關鍵元素：

  • m=audio / m=video：媒體軌道描述
  • a=ice-ufrag / a=ice-pwd：ICE 認證
  • a=fingerprint：DTLS 憑證指紋（對應第 12.6 節）
  • a=rtpmap：可用的編碼器（codec）
  • a=rtcp-mux：RTP 與 RTCP 共用同一條 UDP 連線（節省 NAT 表項）
  • a=group:BUNDLE 0 1：所有 m= 區段共用同一條 ICE/DTLS 連線（節省穿越成本）

12.4.2 Signaling 的角色

WebRTC 不規定 signaling 協議 WebRTC 規範刻意不指定如何傳遞 SDP——這是設計上的智慧。signaling 跟 P2P 連線本身是兩件事：
  • WebSocket（最常見）
  • HTTP polling / Server-Sent Events
  • Socket.IO / 框架自有
  • 甚至：QR code、email、IM、紙條（理論上都可以）
你可以把 signaling 想成「怎麼把名片交給對方」——名片內容（SDP）是 WebRTC 規定的，怎麼遞並不限制。

12.5 Trickle ICE 在 WebRTC 中的實作

第 8 章介紹的 Trickle ICE 在 WebRTC 中是預設行為。pc.onicecandidate 事件在每個候選產生時觸發， 應用立即透過 signaling 送給對方： 

pc.onicecandidate = event => {
  if (event.candidate) {
    // 持續送：host → srflx → relay
    signaling.send({
      type: 'candidate',
      data: event.candidate.toJSON()
    });
  } else {
    // candidate === null 表示 ICE gathering 完成
    console.log('ICE gathering complete');
  }
};

// 收到對方候選
signaling.onmessage = msg => {
  if (msg.type === 'candidate') {
    pc.addIceCandidate(new RTCIceCandidate(msg.data));
  }
};

12.6 DTLS-SRTP：為什麼不直接用 TLS？

WebRTC 的安全層使用 DTLS（Datagram TLS）配合 SRTP（Secure RTP）， 而不是直接 TLS。為什麼？

12.6.1 即時媒體的特殊需求

不要 head-of-line blocking

TLS 跑在 TCP 上，一個封包丟了後面全卡住。即時音視訊寧願丟幾個 frame，也不能等—— DTLS 跑在 UDP 上，無此問題。

不要 stream cipher 的 overhead

TLS 設計成連續 stream，每個訊息都帶 sequence number 與重傳邏輯。 SRTP 用 RTP 的 sequence number 直接做加密 nonce，省下很多 byte。

分離握手與資料

DTLS 只用在「初始握手」階段交換 master key； 後續的媒體封包用 SRTP 加密（更輕量）、純資料用 SCTP-over-DTLS。

身份驗證走 fingerprint

DTLS 用 self-signed 憑證——身份是透過 SDP 中的 a=fingerprint 預先約定的。 沒有 CA 機制，因為 P2P 無 CA 概念。

12.6.2 從信令到加密的完整鏈

1. Alice 與 Bob 透過 signaling 交換 SDP
   SDP 中含對方 DTLS 憑證的 SHA-256 fingerprint
   信令本身應該也走 TLS（防 MITM）

2. ICE 找到可用的 5-tuple
3. DTLS handshake 在該 5-tuple 上進行
4. Alice 看到 Bob 的憑證，計算其 SHA-256 比對 SDP 中的 fingerprint
   比對成功 → 確認對方真的是 Bob 不是 MITM
5. DTLS 產生 master key，從中匯出 SRTP keys
6. 媒體流以 SRTP 加密、資料以 SCTP-over-DTLS 加密
7. 雙向 E2E 加密通訊
這是真正的 E2E 加密嗎？ 在純兩方WebRTC 連線中，是的——只有 Alice 跟 Bob 知道 master key， signaling server 只看到 fingerprint 不能解密。但若有 SFU/MCU（下節）介入， E2E 就被打破——SFU 必須能解密才能轉發。WebRTC 的 Insertable Streams API 讓應用層 能在 SFU 之上再加一層 E2E 加密（例如 Google Meet 的端對端加密選項就用這個）。

12.7.1 SFU 主導現代會議

Google Meet、Zoom、Jitsi、Discord 都採用 SFU 架構（有些會在低端裝置 fallback 到 MCU）。 著名開源 SFU 包括 Jitsi Videobridge、mediasoup、Janus、LiveKit。 Selective Forwarding 的精明之處：SFU 可以根據接收方頻寬，選擇性轉發不同畫質的同一視訊 （配合 Simulcast 或 SVC 編碼），而不需要重新編碼——大幅節省 CPU。

12.8 生產環境指標

12.8.1 健康的 TURN 使用率

業界經驗值 健康的 P2P 應用 TURN 使用率應在 20-30%。如果超過 40%，需檢查：
  • STUN 伺服器配置是否錯誤
  • ICE 候選優先序是否被異常排序（如 host 候選被擋）
  • 客戶端網路環境是否異常嚴格
  • WebRTC 函式庫版本是否過舊
生產環境必須監控這個指標，當作網路健康度的核心 KPI。

12.8.2 連線建立時間

使用 Trickle ICE 的健康基準：

  • 同 LAN：< 100ms（host candidate 立即連通）
  • 跨 NAT 直連：100~300ms（srflx hole punching）
  • TURN 中繼：200~500ms（多一跳 + Allocate 開銷）
  • 整體 P95：< 1s 是合格目標

12.8.3 除錯工具：webrtc-internals

Chrome 內建的 chrome://webrtc-internals 是 WebRTC 工程師的必備工具。 能看到：

  • 所有 RTCPeerConnection 的即時狀態
  • ICE candidates 的收集與配對結果
  • 每對 candidate pair 的 STUN check 統計
  • SRTP 加解密狀態、bitrate、丟包率
  • 下載 dump，可以離線重建問題現場

12.9 完整的最小 P2P 範例

把所有東西串起來——一個 100 行內的 P2P 視訊通話骨架（省略 signaling 細節）：

// ====== HTML ======
// <video id="local" autoplay muted></video>
// <video id="remote" autoplay></video>
// <button id="call">Call</button>

// ====== JS ======
const config = {
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },
    {
      urls: 'turn:turn.example.com:3478',
      username: 'demo',
      credential: 'demo'
    }
  ]
};

const pc = new RTCPeerConnection(config);
const ws = new WebSocket('wss://signal.example.com');

// 取得本機媒體
const localStream = await navigator.mediaDevices.getUserMedia({
  video: true, audio: true
});
document.getElementById('local').srcObject = localStream;
localStream.getTracks().forEach(t => pc.addTrack(t, localStream));

// 收到對方媒體
pc.ontrack = e => {
  document.getElementById('remote').srcObject = e.streams[0];
};

// ICE candidate 即時 trickle
pc.onicecandidate = e => {
  if (e.candidate) {
    ws.send(JSON.stringify({ type: 'ice', data: e.candidate }));
  }
};

// 監控連線狀態
pc.oniceconnectionstatechange = () => {
  console.log('iceState:', pc.iceConnectionState);
  if (pc.iceConnectionState === 'failed') {
    pc.restartIce();  // 第 8 章的 ICE restart
  }
};

// 收到 signaling 訊息
ws.onmessage = async ev => {
  const msg = JSON.parse(ev.data);
  if (msg.type === 'offer') {
    await pc.setRemoteDescription(msg.data);
    const ans = await pc.createAnswer();
    await pc.setLocalDescription(ans);
    ws.send(JSON.stringify({ type: 'answer', data: ans }));
  } else if (msg.type === 'answer') {
    await pc.setRemoteDescription(msg.data);
  } else if (msg.type === 'ice') {
    await pc.addIceCandidate(msg.data);
  }
};

// 點按鈕發起通話
document.getElementById('call').onclick = async () => {
  const offer = await pc.createOffer();
  await pc.setLocalDescription(offer);
  ws.send(JSON.stringify({ type: 'offer', data: offer }));
};

這 60 多行 JavaScript 做到的事，在 WebRTC 之前需要桌面應用 + Flash plugin + 上千行 C++ 才能達成。 這就是標準化的力量。

12.10 章末提要

本章核心觀念
  • WebRTC 把 P2P 即時通訊變成瀏覽器原生能力。
  • 三大 API：getUserMedia（取得媒體）、RTCPeerConnection（核心）、RTCDataChannel（純資料）。
  • 協議堆疊：應用 → API → ICE/DTLS → SRTP/SCTP → UDP（TCP fallback）。
  • SDP offer/answer 模型 + signaling（協議不規定，自己選）。
  • Trickle ICE 預設啟用，連線建立可在 200ms 內完成。
  • DTLS-SRTP 取代 TLS，符合即時媒體對丟包友善與 nonce 高效的需求。
  • 多方通話三種架構：Mesh（小規模）、SFU（主流）、MCU（高混合）。
  • 生產指標：TURN 使用率 20-30% 為健康；P95 連線建立 < 1s。
  • chrome://webrtc-internals 是除錯利器。
