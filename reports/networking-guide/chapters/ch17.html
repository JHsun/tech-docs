CHAPTER 17 安全考量

本章導讀

P2P 系統的安全模型與傳統 client-server 不同——沒有「受信中心」可以保護你。 每一個 peer 都可能是攻擊者、每一條連線都需要驗證、每一份資料都需要簽章。 本章把分散在前面各章的安全議題集中梳理：端對端加密、身份驗證、流量分析、 P2P 特有攻擊面（Sybil、Eclipse、放大攻擊、流量推斷），並討論防禦端如何識別 P2P 的合法與惡意使用。最後我們會直面一個關鍵問題：為什麼即使技術上完美， P2P 系統的「誰跟誰通訊」仍可能洩漏。

📚 學習目標

  • 能描述 P2P 系統的威脅模型，與 client-server 的本質差異。
  • 能比較 DTLS-SRTP、Noise、TLS 1.3 三種端對端加密方案。
  • 能列舉 P2P 特有的至少五種攻擊面與其緩解。
  • 能說明流量元資料（metadata）洩漏為何即使加密也是問題。
  • 能從防禦端角度識別惡意 P2P 流量的特徵。

17.1 P2P 的威脅模型

傳統 client-server 安全模型有「信任的中央」這個錨點——伺服器是受信的， 客戶端用 TLS 連到它即可。P2P 沒有這個錨點。

傳統 C/S 威脅模型

  • 攻擊者：external（網路上的陌生人）
  • 信任：CA-signed 憑證的伺服器
  • 主要威脅：MITM、伺服器入侵
  • 緩解：TLS、cert pinning、WAF

P2P 威脅模型

  • 攻擊者：網路中的任何 peer
  • 信任：無，每個 peer 都得自證
  • 主要威脅：Sybil、Eclipse、流量分析、惡意 peer
  • 緩解：密碼學身份 + 多路徑驗證 + 信任分散
P2P 安全的核心原則 「永不信任、總是驗證」（Never Trust, Always Verify）——但這次「總是」更徹底。 在 P2P 中，連你的路由表內鄰居都可能是攻擊者。設計時必須假設最壞情況。

17.2 端對端加密：三種主流方案

P2P 系統的加密層通常從這三個方案中選一個：

面向 DTLS-SRTP Noise TLS 1.3
典型用例WebRTClibp2p, WireGuard, WireHTTPS, libp2p (alt)
身份驗證SDP fingerprint原生 PSK / static keyX.509 憑證或 raw key
握手 RTT2 (DTLS) + 0 (SRTP keying)1-2 RTT，依 pattern1 RTT (or 0-RTT resumption)
程式碼複雜度中（基於 TLS 改）低（< 1000 行）高（OpenSSL 等大型函式庫）
硬體加速普通限制（ChaCha20 vs AES-GCM）佳
QUIC 整合無關QUIC-Noise 可QUIC 內建

17.2.1 Noise Protocol Framework：簡潔的智慧

Noise（noiseprotocol.org）由 Trevor Perrin 設計， 是 Signal Protocol 的密碼學基礎。它不是單一協議，而是一個「協議框架」—— 你選 handshake pattern（XX、IK、NK 等）與 cipher（ChaCha20-Poly1305、AES-GCM 等）， 組合出符合自己需求的協議。 

XX pattern（最常見，雙方互相驗證）：
  -> e
  <- e, ee, s, es
  -> s, se

每行代表一條訊息：
  e = ephemeral key
  s = static key (身份)
  ee/es/se = key 之間做 DH
  
總共 1.5 RTT，雙方互傳長期身份且驗證對方。

Noise 的優雅在於——簡單規範產生強密碼學保證。WireGuard 使用了 Noise IK pattern， 整個密碼學部分不到 200 行 C 程式碼。

17.2.2 為什麼 P2P 不適合 X.509？

X.509 的 P2P 困境 X.509 憑證的設計核心是 CA 信任鏈——你信 CA，CA 簽發你信的伺服器憑證。 在 P2P 中：
  • 沒有自然的 CA——誰來簽發 peer 的憑證？
  • 數百萬 peer 都申請憑證、定期續期，系統爆炸
  • peer 加入退出頻繁，CRL/OCSP 機制跟不上
因此 P2P 系統幾乎都用自簽憑證 + 帶外驗證（fingerprint 透過信令交換，或公鑰本身就是身份）。

17.3 P2P 特有的攻擊面

17.3.1 Sybil 攻擊

圖 17-1 ｜ Sybil 攻擊：用大量假身份操縱 P2P 系統

緩解方案：

  • 身份成本：要求新身份做 PoW（S/Kademlia）、消耗代幣（Filecoin）或人工驗證（社群網路）
  • IP 多樣性檢查：來自同一 /24 子網的多個 peer 算成一個
  • 長期觀察：偏好「線上久」的節點（Kademlia LRS）
  • 多路徑查詢（disjoint path lookup）：不依賴單一鄰居

17.3.2 Eclipse 攻擊

比 Sybil 更精準——攻擊者目標包圍單一受害者：佔據它路由表所有位置、 控制它能看到的整個網路。在區塊鏈領域可能讓受害者被誤導到假鏈上、雙花得逞。

緩解：

  • 多 bootstrap 來源（不同地理、不同營運者）
  • 定期重新探索網路、不黏在固定鄰居
  • 連接到已驗證身份的種子節點
  • 監測網路統計異常（如所有鄰居都從某 ASN 來）

17.3.3 放大攻擊（Amplification）

攻擊者偽造 source IP 為受害者，向某個會「大回應小請求」的服務送包， 把回應的洪水導向受害者。STUN、DNS、NTP、memcached 都曾被利用： 

STUN amplification:
  攻擊者送 20 byte STUN Binding Request
  伺服器回 ~80 byte XOR-MAPPED-ADDRESS response
  → 4x 放大

但實際攻擊倍率：
  STUN: 4-5x（小，但仍可用）
  DNS:  ~50x（中）
  NTP:  ~500x（大）
  memcached: 50,000x（已修復）

RFC 8489 加強了 STUN 防放大設計（強制 source 驗證）；TURN（RFC 8656）禁止對 private IP 中繼避免內網掃描。

17.3.4 流量分析（Traffic Analysis）

即使內容加密了，「誰跟誰通訊」仍是元資料。攻擊者透過觀察 P2P 流量，可推得：

  • 使用者身份：peer A 跟 peer B 經常通訊 → A 與 B 是朋友
  • 應用類型：時序與大小模式 → 視訊？音訊？檔案？
  • 內容大致：BitTorrent 種子大小是公開的 → 對應檔案能猜
  • 地理位置：peer 主要 IP 來源 → 用戶國家
  • 離線時間：peer 心跳消失 → 推得作息
給敏感場景使用者的提醒 記者、社運人士、面臨家暴威脅者——若你需要保護「誰跟誰在通訊」這個事實本身， 純 P2P 系統是不夠的。你需要 onion routing（Tor）這類額外保護元資料的設計。 WebRTC、libp2p、Tailscale 都不能保護元資料。

17.3.5 中繼濫用

TURN、Circuit Relay 等中繼可能被惡意使用：

  • 當作 anonymous proxy 隱藏攻擊源
  • 放大攻擊（前述）
  • 從 relay 內部位址掃描網段（已被 RFC 8656 禁止）
  • 消耗 relay 頻寬資源（DoS）

緩解：強制認證、速率限制、禁止對 private IP 中繼、relay 行為審計。

17.4 加密通訊不等於安全通訊

17.4.1 元資料的詛咒

下表展示「即使內容完美加密」攻擊者仍能得知什麼：

觀察點能推得的資訊
5-tuple (src, dst, sport, dport, proto)誰跟誰、什麼協議
封包大小直方圖應用類型（視訊 vs 文字 vs 檔案）
封包時序間隔互動式 vs 串流，使用者活躍時段
連線總時長會議長度、檔案大小
TLS ClientHello fingerprint (JA3/JA4)客戶端軟體（甚至版本）
SNI 欄位（除非 ECH）目的網域
STUN/TURN 流量模式P2P 應用類型

17.4.2 哪些設計能保護元資料？

Onion Routing (Tor)

多層加密、多跳轉送，每一跳只知道前一跳與下一跳，沒人知道 source-destination 全鏈。 代價：高延遲（300ms-2s）。

Mix Network

節點蒐集多個訊息批次重排再轉送，破壞時序關聯。 代價：吞吐量極低，多用於 email-like 應用。

Padding + Cover Traffic

所有封包補齊到固定大小、間隔固定時間發送（即使無真實資料也送 dummy）。 代價：頻寬浪費大。

Decentralized Mixnets

Nym、Loopix 等較新的去中心化 mix 網路。 嘗試在元資料保護與性能間取得平衡。

17.5 P2P 系統的攻擊面 vs 防禦端

17.5.1 從 P2P 開發者角度

給開發者的安全 checklist：

P2P 開發者安全 checklist
  • ✓ 所有 peer-to-peer 通訊強制端對端加密（無 plaintext fallback）
  • ✓ 身份用密碼學金鑰，不是 IP/email/username
  • ✓ 對來自未知 peer 的訊息嚴格速率限制
  • ✓ 拒絕對 private IP（10.0.0.0/8、192.168.0.0/16）做 relay
  • ✓ 拒絕對 link-local（169.254、fe80::）做任何處理
  • ✓ 連線數上限（防 conntrack 表撐爆）
  • ✓ Bootstrap node 多源驗證（防 Eclipse）
  • ✓ DHT 查詢用 disjoint path（防中間人污染）
  • ✓ 監控異常 peer 行為（過大流量、異常 RPC 模式）
  • ✓ 公開揭露安全模型——告訴使用者「本系統保護什麼、不保護什麼」

17.5.2 從防禦者角度

網路管理者要識別「企業內未授權的 P2P 流量」可看以下訊號：

  • 大量 UDP/443 流量但無對應 HTTPS 瀏覽行為——可能是 QUIC/WebRTC P2P
  • STUN/TURN public servers 連線（stun.l.google.com:19302 等）
  • 連到 Cloudflare/Twilio TURN 但非 WebRTC 視訊應用
  • 多對對外連線同時建立——hole punching 同步特徵
  • 對等流量（上下行幾乎對稱）——typical P2P
  • 連到多個地理分散 IP同時——DHT crawling 特徵

注意：上述「特徵」也適用於合法應用（Tailscale、視訊會議、雲端儲存同步）—— 防禦端必須結合政策（什麼是允許的）才能下判斷，純技術 fingerprint 不足。

17.6 案例研究：BitTorrent 在企業環境的識別

BitTorrent 是企業最頻繁阻擋的 P2P 應用之一。如何識別它？

L2 · Signature

BT handshake 的 19-byte protocol identifier "BitTorrent protocol" 是 fingerprint。 但加密的 BT (BEP 8 obfuscation) 看不到。

L3 · Behavior

同時連數十個 peer、上下行對等、長時間穩定流量——典型 BT 特徵。 機器學習分類器極準。

L4 · DHT 特徵

對 Mainline DHT bootstrap nodes（router.bittorrent.com 等）的 UDP 流量。 或對隨機高位 port 的 DHT 查詢。

反偵測

BT client 用 µTP（隨機 port、加密）規避偵測。 這是攻擊與防禦的長期軍備競賽。

17.7 安全的 P2P 應用設計原則

本書建議的設計原則
  1. 密碼學身份優先：PeerID 必須是 hash(public key)，不能是任何外部識別碼
  2. 加密非可選：所有 P2P 流量強制 E2E 加密，連除錯模式都不放行 plaintext
  3. 最小知識：每個 peer 只應知道完成任務必要的元資料
  4. 抗 Sybil 內建：身份成本（PoW、staking、social proof）必須在協議層
  5. 多路徑驗證：關鍵決策（DHT 查詢、bootstrap）走多個獨立路徑
  6. 速率限制無處不在：每個 peer-to-peer RPC 都應有 per-source 限制
  7. 透明的安全模型：文件中明確說明「本系統威脅模型 / 不保護的攻擊類型」
  8. 定期安全審計：第三方審計、bug bounty、開源透明

17.8 章末提要

本章核心觀念
  • P2P 沒有「受信中心」——每個 peer 都可能是攻擊者。
  • 密碼學身份（PeerID）取代 IP/username 作為長期識別。
  • 三種主流端對端加密：DTLS-SRTP（WebRTC）、Noise（libp2p/WireGuard）、TLS 1.3。
  • X.509 不適合 P2P，因為 CA 信任模型與 P2P 哲學衝突。
  • P2P 特有攻擊：Sybil、Eclipse、放大、流量分析、中繼濫用。
  • 加密保護內容但不保護元資料；保護元資料需 onion routing / mix network。
  • 即使技術完美，「誰跟誰通訊」這個事實仍可能洩漏——敏感場景需額外保護。
  • 防禦端可從多個 layer（signature / behavior / fingerprint）識別 P2P 流量。
