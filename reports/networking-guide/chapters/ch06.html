CHAPTER 6 打洞技術（Hole Punching）

本章導讀

Hole punching 是 NAT 穿越的「皇冠技術」——它看似魔法，實則只是巧妙利用 NAT 表的副作用。 本章是全書最重要的技術章節，從 Bryan Ford 等人 2005 年的奠基論文出發， 詳述 UDP hole punching 的三步驟、TCP 同時開啟的詭譎、symmetric NAT 的 port prediction， 一直到 2025 年最新的大規模測量結果——QUIC 與 TCP 的穿越成功率在精準同步下統計上無顯著差異。 讀完本章，你會明白為何 hole punching 至今仍是現代 P2P（WebRTC、libp2p）的核心技術。

📚 學習目標

  • 能完整描述 UDP hole punching 三步驟流程，並能畫出對應的 NAT 表變化。
  • 能解釋為什麼 UDP hole punching 對 cone 類 NAT 有效但對 symmetric NAT 失效。
  • 能說明 TCP hole punching 的兩種策略（同時 SYN、連線反向）與其失敗模式。
  • 能描述 port prediction 對 symmetric NAT 的補救與其前提條件。
  • 能引用最新測量數據評估各技術的真實成功率。
  • 能寫出一個最小可行的 UDP hole punching 程式。

6.1 為什麼這稱作「打洞」？

「Hole punching」一詞來自一個很視覺化的比喻：NAT 像一道牆，平常擋住所有外人。 當你主動對外送一個封包時，NAT 表會新增一條條目——這條目就是牆上「暫時開的洞」， 允許特定來源的封包進入。

Hole punching 的核心觀察是：兩個 NAT 後的主機如果同時對外送包給對方， 兩邊的 NAT 各自開了一個洞，這兩個洞剛好能讓對方的封包進來。

這個技巧的精妙之處在於——它不需要 NAT 做任何特殊配合，只要 NAT 有狀態且映射穩定 （即第 2 章說的 EIM）就能成功。

6.2 三步驟：rendezvous → exchange → punch

完整的 hole punching 需要一個眾所周知的第三方伺服器協助協調， 因為兩個 NAT 後的主機本身無法直接通信來「商量同時動手的時間」。 這個第三方稱為 rendezvous server（會合伺服器）或 signaling server。

6.2.1 階段 1：註冊（Registration）

Alice 與 Bob 各自連到 rendezvous server（這是主動發出連線，所以 NAT 不會擋）。 Rendezvous 透過觀察封包源位址，得知雙方的對外 endpoint （這個動作本身就是 STUN 風格的反射，第 5 章）。

Rendezvous 同時記錄雙方的本地 endpoint（透過應用層告知）， 因為若兩個 peer 在同一個 NAT 後，可以走 hairpin 或直接內網連線。

6.2.2 階段 2：位址交換（Exchange）

當 Alice 想找 Bob 時，rendezvous 把 Bob 的 endpoints 全部告訴 Alice： 

{
  "peer": "bob",
  "endpoints": [
    { "type": "local",  "ip": "192.168.1.42", "port": 6881 },
    { "type": "public", "ip": "203.0.113.99", "port": 62000 },
    { "type": "ipv6",   "ip": "2001:db8::1",  "port": 6881 }
  ]
}

同時通知 Bob：「Alice 來找你，這是她的 endpoints」。

6.2.3 階段 3：同時打洞（Punch）

雙方收到對方的位址後，立即同時對所有候選 endpoint 送出探測封包。 「同時」這個詞至關重要——時間差過大會導致：

  • 若 Alice 比 Bob 早太多，Alice 的包到 Bob_NAT 時 Bob_NAT 還沒開洞 → 丟棄； 且若 Bob_NAT 是 Address-Dependent Filtering，這個丟棄不會影響後續，但 RFC 4787 EIF 以外的 NAT 會記錄「Alice 是未邀請者」可能更嚴格阻擋後續
  • 若雙方 NAT 都是 Port-Restricted，必須都主動送過包才會開洞

協調「同時」需要 rendezvous 量測 RTT，計算什麼時候各自送， 讓兩端的封包剛好在中間相遇。這個同步機制在 libp2p 的 DCUtR 中被優雅地實現 （第 13 章詳述）。

6.4 TCP Hole Punching：難得多的兄弟

UDP 是無連線的，雙方各自送包就行。TCP 不一樣——它有連線狀態（SYN、SYN-ACK、ACK 三次握手）， 且作業系統 socket 預設只能扮演 client 或 server 之一，不能同時。 TCP hole punching 因此複雜得多。

6.4.1 策略一：TCP 同時開啟（Simultaneous Open）

RFC 793 定義 TCP 時就考慮過一個罕見場景：兩端同時主動 SYN 給對方。 這時不會有單方扮演 server，而是雙方都進入 SYN-SENT 狀態，互相收到對方的 SYN， 變成「同時開啟」狀態，建立一條連線。

(在 SYN-SENT 狀態) NB->>B: 收到對方的 SYN A->>NA: SYN-ACK ack=Y+1 B->>NB: SYN-ACK ack=X+1 NA->>NB: SYN-ACK NB->>NA: SYN-ACK NB->>B: SYN-ACK 收到，連線建立 NA->>A: SYN-ACK 收到，連線建立 Note over A,B: TCP 連線建立完成

問題是：作業系統與 NAT 對這個罕見路徑支援不一致。Linux 從 2.4 之後支援得不錯， Windows XP SP1 之前根本不支援，許多老舊 NAT 也會把「未經完整三次握手就有對方 SYN」視為異常封包丟棄。 Ford 的測試顯示這條路成功率約 64%，但實作敏感度高。

6.4.2 策略二：連線反向（Connection Reversal）

若雙方有一方在公網（例如 Bob 是公網主機），但連線方向反過來—— Alice 想連 Bob，但 Alice 在 NAT 後、Bob 在公網——這場景其實 Bob 主動連 Alice 才合理， 但 Alice 是發起方。

解法：透過 rendezvous 通知 Bob「請反向連 Alice」， Alice 此時對 Bob 的 IP 開個洞（送一個 SYN，會被擋下來但 NAT 開了洞）， Bob 接著主動連 Alice 的對外位址，順利進入。

6.4.3 策略三：Predictable Port + Repeated Attempts

對 symmetric NAT，用 port prediction（下一節）配合大量重試。 TCP 重試比 UDP 慢得多（每次 RST 都要等系統的 backoff），所以這條路不太實用。

6.4.4 為什麼現代 P2P 偏好 UDP / QUIC

TCP hole punching 的不可靠，是現代 P2P 系統偏好 UDP/QUIC的主因之一。 Tailscale 的部落格直白地說：

「你可以用 TCP 做 NAT 穿越，但這會給已經夠複雜的問題再加一層複雜度， 甚至可能需要核心客製化。我們專注在 UDP；若你想要串流連線，請考慮基於 UDP 的 QUIC， 它能讓你既享受 UDP 穿越的容易，又有 stream 協議的便利」

這個建議在 2025 年成為主流——WebRTC（DataChannel over UDP/SCTP）、libp2p（QUIC 優先）、 Tailscale（WireGuard over UDP）都遵循這個哲學。

6.5 對抗 Symmetric NAT：Port Prediction

Symmetric NAT 為每個遠端分配不同外部 port——這毀了 STUN 探得位址的有效性。 但若 NAT 的 port 分配是確定性的（例如 sequential：62000、62001、62002...）， 就能預測下一個會分到的 port。

6.5.1 演算法骨架

階段 1：Alice 在 symmetric NAT 後，先對多個 STUN 伺服器各送一次：
  STUN1 (1.1.1.1) → 觀察到 Alice 對外是 198.51.100.1:62000
  STUN2 (2.2.2.2) → 觀察到 Alice 對外是 198.51.100.1:62001
  STUN3 (3.3.3.3) → 觀察到 Alice 對外是 198.51.100.1:62002

→ 推論：Alice 的 NAT 採 sequential 分配，下一個會是 62003

階段 2：Alice 對外送一個包給 Bob 的位址，
       NAT 分配新 port，預期是 62003

階段 3：告訴 Bob「猜我是 62003」（透過 rendezvous），
       Bob 同時對 198.51.100.1:62003 送多個 punch 包

階段 4：若 NAT 真的給了 62003（沒有別的併發連線打亂順序），
       Bob 的包就能進入 → 成功

6.5.2 預測技巧的進階：開多個 socket

為了提高命中率，Alice 可以從多個本地 port同時對 Bob 送包，每個都產生一個外部 port， 讓 Bob 對連續一段範圍（例如 62000~62050）做並行 punch。一個成功就行。 這個技巧最早由 Yutaka Takeda（2003 年）提出，早稻田大學的研究團隊（2008 年）做了系統化分析。

6.5.3 預測技巧的失敗條件

什麼情況下 port prediction 失效？
  • NAT 採用隨機分配（多數現代 NAT 為了安全已隨機化）
  • 競爭條件：你預測時別的應用剛好併發開連線，把預期 port 搶走
  • NAT 有 port 範圍限制：例如 50000-60000，需要更精準
  • 多層 NAT（CGNAT）：每一層都隨機，連乘下去命中率歸零

因此 port prediction 在 2025 年仍是「偶爾管用的偏方」，主流 P2P 系統不依賴它， symmetric NAT 場景的標準解法仍是fall back to TURN 中繼（第 7 章）。

6.6 兩個 peer 在同一個 NAT 後的特殊情境

一個容易被忽略但實務上常見的場景：Alice 與 Bob 都在同一個家用網路內。

6.6.1 Hairpinning 的需求

並非所有 NAT 都支援 hairpinning。早期家用路由器很多不支援，會把這種「目的 IP 是自己對外 IP」的封包丟掉。 RFC 4787 強烈建議 NAT 支援 hairpinning，但市場執行率不一。

6.6.2 解法：交換本地位址

正確做法是 rendezvous 同時記錄 peer 的本地 endpoint，並一併告訴對方。 Alice 與 Bob 同時嘗試本地位址 + 對外位址，本地位址成功率較高且延遲更低。 這就是 ICE 收集 host candidate 的動機之一（第 8 章）。

6.7 同步機制的精細：DCUtR 的啟示

我們前面說「同時打洞」很關鍵，但「同時」要多精準？ libp2p 的 DCUtR（Direct Connection Upgrade through Relay）給了一個漂亮的答案： 透過 RTT 量測，讓雙方互相預測對方何時動手。

6.7.1 DCUtR 的同步邏輯

[假設 Alice 與 Bob 已透過 relay 建立暫時連線，知道彼此 RTT = 200ms]

Alice 對 Bob 送 Connect 訊息（透過 relay）
       └─ 內含 Alice 的所有候選位址
Alice 開始計時

Bob 收到 Connect 後立即回 Connect
       └─ 內含 Bob 的所有候選位址

Alice 收到 Bob 的 Connect → 計算實際 RTT (假設 200ms)
Alice 立即送 Sync 訊息（透過 relay）
Alice 等待 100ms (= RTT/2)，然後直接 dial Bob

Bob 收到 Sync 訊息 → 立即直接 dial Alice

預期結果：兩邊的封包剛好在中間相遇
        → 同時為彼此開洞 → 直接連線建立

6.7.2 為什麼這個機制重要？

IPFS 網路在 2025 年發表了一份大規模測量研究——440 萬次穿越嘗試、 85,000+ 個獨立網路、167 個國家。研究結論：

大規模測量的關鍵發現
  • 整體 hole punching 成功率約 70%（±7.1%）——這是 2025 年的權威基準
  • 多數成功在第一次嘗試就達成，得益於 RTT 同步
  • 關鍵發現：QUIC 與 TCP 在精準同步下成功率統計上無顯著差異—— 這推翻了「UDP 一定優於 TCP」的長期部落知識
  • 連線反向（connection reversal）作為輔助機制顯著提升成功率
  • 穿越成功後，延遲顯著下降（不再經 relay）

這個結果改寫了教科書——好的同步機制比好的傳輸層更重要。 TCP 不是不能 punch，而是少了 DCUtR 這樣的精準同步工具就不好 punch。

6.8 一個最小可行的 UDP Hole Punching 實作

用 Python 寫一個極簡 hole punching：rendezvous + 兩個客戶端。

6.8.1 Rendezvous Server

# rendezvous.py - 極簡會合伺服器
import socket, json

sock = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
sock.bind(('0.0.0.0', 9999))
peers = {}  # name -> (ip, port)

print("Rendezvous on UDP 9999")
while True:
    data, addr = sock.recvfrom(4096)
    msg = json.loads(data.decode())
    name = msg['name']
    target = msg.get('connect_to')

    # 記錄此 peer 的對外位址（即 addr）
    peers[name] = addr
    print(f"[register] {name} = {addr}")

    if target and target in peers:
        # 通知雙方對方的位址
        info_to_a = json.dumps({'peer': target, 'addr': peers[target]})
        info_to_b = json.dumps({'peer': name,   'addr': peers[name]})
        sock.sendto(info_to_a.encode(), peers[name])
        sock.sendto(info_to_b.encode(), peers[target])
        print(f"[match]   {name} ↔ {target}")

# 階段 1+2：註冊並查詢對方
sock.sendto(json.dumps({
    'name': NAME, 'connect_to': TARGET
}).encode(), RDV_ADDR)

data, _ = sock.recvfrom(4096)
peer_info = json.loads(data.decode())
peer_addr = tuple(peer_info['addr'])
print(f"[info] 對方 {peer_info['peer']} 在 {peer_addr}")

# 階段 3：同時打洞
def receiver():
    while True:
        d, a = sock.recvfrom(4096)
        if a == peer_addr:
            print(f"<< 收到: {d.decode()}")

threading.Thread(target=receiver, daemon=True).start()

print("[punch] 開始打洞...")
for i in range(20):
    msg = f"{NAME} #{i}".encode()
    sock.sendto(msg, peer_addr)
    print(f">> 送出 #{i} → {peer_addr}")
    time.sleep(0.5)

print("[done] 連線建立後可改用實際應用協議")

6.8.3 執行

# 機器 X (公網): rendezvous server
$ python rendezvous.py

# 機器 Y (NAT 後 #1): peer Alice
$ python peer.py alice bob 公網IP:9999

# 機器 Z (NAT 後 #2): peer Bob
$ python peer.py bob alice 公網IP:9999

# 預期看到雙方互相 print "收到 alice #N" / "收到 bob #N"
# 若 5 秒內看不到，至少一方是 symmetric NAT，本範例無法穿越

這個範例不到 70 行，已經完整示範 hole punching 的全部精髓。實際生產環境還需考慮： ICE 的多候選、TURN fallback、加密、認證、keep-alive 等。後續章節會逐一加上。

6.9 NAT keep-alive：讓洞不要關上

Hole punching 成功不是結束——NAT 表項有 timeout（第 2 章），閒置一段時間就會失效。 必須定期送 keep-alive 包讓條目活著。

6.9.1 Keep-alive 的時機

頻率

建議每 15~25 秒一次。比 RFC 4787 的最低 UDP timeout（2 分鐘）留有足夠餘裕， 也避免太頻繁浪費頻寬。

內容

最小：1 byte 的「ping」即可。或用 STUN Binding Indication（不需回應的 STUN 訊息）。 WebRTC 用 consent freshness 作為 keep-alive。

每連線獨立

第 2 章提過：許多 NAT 把 timeout 綁在 (本地, 遠端) 對上。 若 Alice 同時與 Bob、Charlie 通訊，必須對兩者各自送 keep-alive。

移動裝置考量

手機 sleep 時 keep-alive 會被作業系統暫停。WebRTC 連線在手機鎖屏 30 秒後常常斷掉， 應用必須有重連邏輯。

6.10 章末提要

本章核心觀念
  • Hole punching 利用 NAT 表的副作用——「主動送包→開洞」這個機制——讓雙方能直連。
  • 三步驟：rendezvous 註冊、位址交換、同時打洞。
  • UDP punching 對 cone NAT 高度有效（> 80%），對 symmetric NAT 失效。
  • TCP punching 透過 simultaneous open 或 connection reversal，但實作敏感、成功率較低。
  • Port prediction 對 symmetric NAT 有條件有效，但現代 NAT 多採隨機分配。
  • 同步精準度比傳輸層選擇更重要——DCUtR 證明 QUIC 與 TCP 成功率可相當。
  • 2025 年 IPFS 大規模測量基準：去中心化 hole punching 整體成功率 70%。
  • Punch 成功後必須以 keep-alive 維持 NAT 表項。

動腦時間

  1. 用本章程式碼，在你家網路與雲端 VM 之間做一次 hole punching 測試。 若失敗，從 NAT 行為角度分析原因。
  2. 為什麼 hole punching 需要 rendezvous server？能否設計一個「無 rendezvous」的方案？ （提示：DHT 可以扮演這個角色，第 10、13 章會看到）
  3. Symmetric vs Symmetric 的 punch 為何幾乎不可能？除了 port prediction，還有什麼可能的補救？
  4. 研究 WebRTC 的 ICE consent freshness（RFC 7675），它跟本章說的 keep-alive 有什麼異同？
