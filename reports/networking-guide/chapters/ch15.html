CHAPTER 15 傳輸層的選擇：UDP、TCP、QUIC

本章導讀

Part I-III 我們把 NAT 穿越與 P2P 系統各個側面都看過了。Part IV 開始把分散在各章的跨章節主題集中討論。 本章是這個收尾篇的開場——傳輸層選擇。對 P2P 系統而言這個問題尤為關鍵： UDP 易於穿越但需自己重建可靠性；TCP 提供 stream 抽象但穿越成本高； QUIC 兼有兩者優點，且自帶 TLS 1.3 與連線遷移。本章從 P2P 視角重新審視三者的取捨， 並援引第 6 章與第 13 章的測量結果——精準同步下 TCP ≈ QUIC 的最新發現。

📚 學習目標

  • 能列舉 UDP、TCP、QUIC 對 P2P 應用的優劣對比表。
  • 能解釋為什麼 NAT 穿越歷史上偏好 UDP，並說明這個偏好在 2025+ 是否仍成立。
  • 能描述 QUIC 的連線遷移（connection migration）對行動 P2P 的關鍵性。
  • 能評估 QUIC 在企業與行動電信網路的可達性。
  • 能說明 SCTP-over-DTLS 與 QUIC 兩者在 WebRTC DataChannel 上的繼承關係。

15.2 為什麼歷史上 NAT 穿越偏好 UDP？

Tailscale 的部落格直白地寫過這段經典建議：

「你可以用 TCP 做 NAT 穿越，但這會給已經夠複雜的問題再加一層複雜度， 甚至可能需要核心客製化。我們專注在 UDP；若你想要串流連線，請考慮基於 UDP 的 QUIC。」

UDP 之所以在穿越上勝出，有四個結構性原因：

① 無連線狀態

UDP 沒有「連線」概念——任何兩個 endpoint 之間 send/receive 即可。 NAT 不需要追蹤 SYN/ACK 狀態機。

② 同時雙向自然

UDP hole punching 雙方都「主動 send」即可。TCP 的「同時 SYN」 需要作業系統與 NAT 都正確處理罕見路徑。

③ NAT 行為較一致

多數 NAT 對 UDP 的處理是「查映射 + 轉送」——簡單、可預測。 對 TCP 還會看 flag、做 sequence 驗證。

④ Kernel 操控自由

UDP socket 不像 TCP 受限於連線狀態機，應用層能自由操控 （產生重複 SYN、操控源 port 等技巧）。

15.3 但是——2025 年的修正

第 6 章我們已援引過 IPFS 大規模測量的結論：TCP 與 QUIC 在精準同步下成功率統計上無顯著差異。 本章正式擺出來分析。

Ford 2005 年論文的「TCP 64% / UDP 82%」差距，在現代精準同步協議下幾乎消失。 這不是因為 NAT 變得對 TCP 更友善，而是因為同步機制的進步把原本的劣勢補了回來。

15.3.2 對工程決策的意義

2025+ 的 P2P 傳輸層選擇
  • 若你能寫精準同步協議（DCUtR-like）→ TCP 與 QUIC 都可，依其他需求選
  • 若你寫不出來、用簡單 hole punching → 仍應選 UDP/QUIC
  • 需要 stream 多工、低延遲、自帶加密 → QUIC 是最佳選擇
  • 需要在嚴格企業環境 fallback → TCP over TLS:443（QUIC 的 UDP 仍可能被擋）

15.4 QUIC 深入：為什麼是現代 P2P 的首選

15.4.1 QUIC 的設計哲學

QUIC（Quick UDP Internet Connections）由 Google 於 2012 年發起，IETF 於 2021 年標準化（RFC 9000）。 設計目標：把 TCP+TLS 的所有好處保留，把所有缺點修掉。

修 1 · Head-of-Line Blocking

TCP 一個 segment 丟了後面所有資料卡住。QUIC 多 stream 各自獨立—— 一條 stream 卡住不影響其他 stream。

修 2 · 握手成本

TCP+TLS 1.2 = 3 RTT。QUIC = 1 RTT（首次）/ 0 RTT（resumption）。 對短連線（API 呼叫、P2P signaling）省下大量時間。

修 3 · 連線遷移

TCP 連線綁定 5-tuple，換 IP 就斷。QUIC 用 Connection ID 識別—— 手機從 Wi-Fi 切到 4G 時，連線可無縫遷移。

修 4 · 擁塞控制可演進

TCP 的擁塞控制與 kernel 綁死，更新緩慢。QUIC 跑在 userspace， 可實驗性使用 BBR、CUBIC、自訂演算法。

對 P2P 系統，連線遷移有兩個額外好處：

  • NAT rebinding 後不需重建連線——NAT 表項過期重新分配 port 時，QUIC 也能優雅處理
  • 多路徑同時用（Multipath QUIC，正在標準化）——同時用 Wi-Fi 與 4G 提升可靠性

15.4.3 QUIC 在 P2P 中的整合

libp2p

QUIC 是首選 transport。go-libp2p 預設啟用 quic-v1。 DCUtR over QUIC 有最佳成功率（連線從 client hello 就能 punch）。

WebTransport

瀏覽器原生 QUIC API。瀏覽器可直接與 QUIC 伺服器或 libp2p 節點通訊， 彌補 WebRTC DataChannel 的不足。

HTTP/3

HTTP over QUIC。雖然不直接是 P2P，但讓 signaling、coordination、bootstrap 等流量 更快、更可靠。

WebRTC over QUIC

正在發展中——把 WebRTC 的 SCTP 換成 QUIC stream。 Chrome 已實驗性支援。簡化堆疊、提升性能。

15.5 SCTP-over-DTLS 與 QUIC 的歷史

WebRTC 的 RTCDataChannel 用 SCTP-over-DTLS 提供可靠串流（第 12 章）。 這個選擇在 2012 年是合理的——當時 QUIC 還是 Google 內部實驗。

SCTP（RFC 4960）原本是電信業設計的協議，提供類似 TCP 的可靠 + 多 stream + 多歸屬（multi-homing）。但：

  • SCTP 在公網被中間盒廣泛阻擋（router 不認識協議號 132）
  • WebRTC 把它跑在 DTLS 上繞過了這個問題
  • 但仍有許多歷史包袱（INIT/COOKIE 機制、DCEP 子協議）

QUIC 正是「SCTP 設計理念 + UDP 中介盒友善度 + TLS 1.3 整合」的合成。 WebRTC over QUIC 的提案就是用 QUIC 取代 SCTP-over-DTLS—— 得到等價功能，且大幅降低協議堆疊複雜度。

15.6 真實世界的可達性：QUIC 能穿過企業嗎？

15.6.1 QUIC 的部署現狀（2025）

家用網路 ✓

幾乎全部支援。家用 NAT 對 UDP 友善，QUIC 在 port 443 與普通網路無異。

行動電信 ✓

Verizon、T-Mobile、Vodafone 等大型業者全面支援 QUIC。 甚至有業者觀察到 QUIC 比 TCP 在弱網路下表現更好。

消費級 CDN ✓

Cloudflare、Fastly、Google 全面 HTTP/3。 意味著公網的 QUIC 流量已是常態，被中間盒誤殺機率低。

企業網路 ⚠️

仍有相當比例企業 firewall 阻擋 UDP/443。原因： DPI 工具對 QUIC 加密流量無法檢查，企業傾向「不認識的就擋」。 這是 QUIC 在企業滲透率仍低的主因。

15.6.2 解決方案：fallback 鏈

生產環境的 P2P 系統典型做法：

優先序：
  1. 嘗試 QUIC（UDP/443）            ← 最快、最現代
  2. 嘗試 WebSocket over TLS（TCP/443）← 企業最易穿透
  3. fall back to TURN over TLS:443    ← 最後保險

每一層都應該同時嘗試而非序列，配合 happy eyeballs 策略：
  - 開始 QUIC 嘗試
  - 250ms 後若無回應，並行開始 TCP 嘗試
  - 哪個先連通就用哪個

這個策略給 QUIC 一個短窗口優先發揮，但不讓 QUIC 失敗導致使用者等待。

15.8 章末提要

本章核心觀念
  • 三種傳輸：UDP（最易穿越但需自建可靠性）、TCP（內建可靠但穿越難）、QUIC（兩者結合）。
  • 歷史上 P2P 偏好 UDP 的四個結構性原因：無連線、雙向自然、NAT 一致、kernel 自由。
  • 2025 年測量推翻了「TCP 必輸 UDP」的部落知識——關鍵是同步機制。
  • QUIC 修補了 TCP 的四大問題：HoL blocking、握手成本、連線遷移、擁塞控制可演進。
  • 連線遷移對行動 P2P 是革命性——換網不斷線。
  • WebRTC 的 SCTP-over-DTLS 正在被 QUIC 取代（WebRTC over QUIC）。
  • 企業環境是 QUIC 滲透率最低的場景，必須有 TCP/443 fallback。
  • Happy eyeballs 策略：同時嘗試多種傳輸，哪個快用哪個。
