CHAPTER 13 libp2p 與去中心化打洞

本章導讀

libp2p 是「P2P 的 lego 積木」——它把網路堆疊每一層都做成可替換模組，讓你能用同一套 API 跑 TCP、QUIC、WebRTC、WebTransport，套用 Noise 或 TLS 加密，掛上 Kademlia DHT 或 Gossipsub。 它是 IPFS、Filecoin、Polkadot、Ethereum 共識層、Lotus 的共同網路基礎。本章的高潮是 DCUtR（Direct Connection Upgrade through Relay）——這是 ICE 在去中心化網路的對應方案， 用更簡單的精準同步機制達到甚至超越 ICE 的成功率。我們會援引 2025 年 IPFS 大規模測量結果， 看看 DCUtR 在真實世界的成績單。

📚 學習目標

  • 能說明 libp2p 的模組化設計哲學與五大核心抽象。
  • 能描述 multiaddr 的階層式編址與其相對 IP:port 的優勢。
  • 能解釋 PeerID 如何作為密碼學身份。
  • 能描述 AutoNAT、Identify、Circuit Relay v2、DCUtR 的協作流程。
  • 能說明 DCUtR 為何在精準同步下讓 TCP 與 QUIC 表現相當。
  • 能引用 2025 年 IPFS 測量數據評估真實世界成功率。

13.1 libp2p 的設計哲學

libp2p 由 Protocol Labs 在 2014 年從 IPFS 中分拆出來，目標是「解耦」P2P 網路堆疊的每一層， 讓開發者能像組樂高一樣組合自己的 P2P 系統。核心設計原則：

① Transport Agnostic

不假設特定傳輸——TCP、QUIC、WebRTC、WebTransport、藍牙、TOR 都是 transport 模組。 應用層程式碼不需改寫就能切換 transport。

② Address Agnostic

不假設 IP+port——multiaddr 描述任意層級的位址（IPv4 / IPv6 / DNS / Tor / I2P / Bluetooth）， 且能巢狀表達（如 relay 路徑）。

③ Identity-First

身份不是 IP，是密碼學金鑰。PeerID = hash(public_key)， 全網唯一、不可偽造、與位址無關。

④ Streams over Connections

一條 transport 連線可以多工 N 條獨立 stream（類似 HTTP/2 / QUIC 概念）。 每條 stream 跑不同協議。

和 WebRTC 的對照 WebRTC 也是 P2P，但它融合了所有層次——你不能輕易抽掉 ICE 換成別的、不能不用 SDP。 libp2p 反過來——每一層都是可替換模組。這個彈性的代價是複雜度， 但它讓 libp2p 能跑在任何環境（資料中心、瀏覽器、嵌入式裝置、Tor 暗網）。

13.2.1 PeerID：密碼學身份

PeerID 產生流程：
  1. 節點啟動時產生 keypair (Ed25519 / RSA / Secp256k1)
  2. PeerID = multihash(serialize(public_key))
  3. 文字表示：12D3KooWGRS... 或 QmYwAPJ...

特性：
  ・ 全網唯一（碰撞機率近於零）
  ・ 自我驗證（持有對應私鑰才能宣稱身份）
  ・ 與網路位址解耦（節點換 IP，PeerID 不變）
  ・ 用於簽章、加密、DHT key、檔案 provider record

13.2.2 Multiaddr：階層式位址

multiaddr 是個漂亮的設計——把網路位址表達為逐層協議堆疊： 

傳統 IP:port:
  192.168.1.42:6881

multiaddr：
  /ip4/192.168.1.42/tcp/6881
  /ip6/2001:db8::1/udp/443/quic-v1
  /dns4/example.com/tcp/443/wss
  /ip4/1.2.3.4/udp/4001/quic-v1/p2p/12D3KooWGRS...

巢狀（透過 relay）：
  /ip4/1.2.3.4/tcp/4001/p2p/12D3KooWRELAY/p2p-circuit/p2p/12D3KooWTARGET

意思：「先連 Relay 節點 12D3KooWRELAY，

       透過它的 circuit 找到 12D3KooWTARGET」

這個編址比傳統 URL 強大得多——它能精確描述「怎麼到達某節點」的完整路徑， 包含中繼、加密層、傳輸協議。每個 /protocol/value 段都是可解析的二進位元素。

13.2.3 Stream Multiplexer：一連多用

libp2p 在一條底層連線（TCP/QUIC）之上跑 stream multiplexer—— 讓多個邏輯 stream 並存，每條跑不同上層協議： 

┌──────── 一條 TCP 連線 ────────┐
│                                │
│  Stream 1: /ipfs/kad/1.0.0     │  ← Kademlia 查詢
│  Stream 2: /ipfs/bitswap/1.2.0 │  ← 檔案資料交換
│  Stream 3: /ipfs/id/1.0.0      │  ← Identify 協議
│  Stream 4: /libp2p/dcutr       │  ← DCUtR 同步
│                                │
└────────────────────────────────┘

Multiplexer 負責封包 framing 與 stream 識別。
常見實作：yamux（基於 SPDY）、mplex（libp2p 原生）

13.2.4 Connection Security：Noise vs TLS

libp2p 兩種主流安全層：

面向NoiseTLS 1.3
來源WhatsApp / Signal Protocol 系出同門HTTPS 標準
握手 RTT1 RTT (XX pattern)1 RTT (TLS 1.3)

libp2p 預設Noise 是首選TLS 1.3 是相容選項

13.4 AutoNAT：libp2p 的「STUN 替代」

在 ICE 世界，節點透過 STUN 探得自己對外位址。在 libp2p 世界，這個工作由 AutoNAT 完成—— 且做得更嚴謹。

13.4.1 為什麼不用 STUN？

AutoNAT vs STUN 的關鍵差異 STUN 只告訴你「STUN 伺服器看到你長什麼樣」——但這不一定等於「其他人能不能連到你」。 考慮這個情境：你的 NAT 是 Endpoint-Independent Mapping（友善），但 Endpoint-Dependent Filtering（過濾嚴）。STUN 看到的位址完美，但其他人主動連仍然連不到。

AutoNAT 不是觀察，而是實測：節點 A 請隨機幾個其他節點 B、C、D 主動 dial 回來， 若它們都成功 → A 是公網可達；若全部失敗 → A 在 NAT 後。實測比觀察可靠。

13.4.2 AutoNAT 的工作流程

節點 A 想知道自己是否公網可達：

1. A 從 PeerStore 中選 K 個隨機 peer (例如 K=4)
2. 對每個 peer P，發送 AutoNAT DialRequest:
   "請在這些位址 [/ip4/.../tcp/.../, /ip4/.../udp/.../quic] dial 我"
3. 每個 P 嘗試 dial 那些位址
4. P 回傳 DialResponse:
   ・ DIAL_SUCCESS（連通了），附上實際成功的位址
   ・ DIAL_ERROR（全部失敗）
5. A 統計結果：
   ・ 多數成功 → 確定是公網
   ・ 多數失敗 → 確定在 NAT 後
   ・ 結果不一 → 重試或標為「不確定」

A 根據結論調整行為：
  ・ 公網 → 接受入站連線
  ・ NAT 後 → 啟動 Circuit Relay reservation

AutoNAT 有個漂亮特性：它順便提供了位址驗證——若 A 宣稱自己有某位址但 dial 失敗， 該位址會被 PeerStore 標記為 unreliable，避免誤導其他節點。

13.4.3 為什麼 libp2p spec 中還是提到 STUN？

瀏覽器是個例外——libp2p 在瀏覽器中跑時，必須使用瀏覽器內建的 WebRTC stack， 那個 stack 只支援 STUN 不支援 AutoNAT。所以在 browser-to-browser 場景下， libp2p 仍依賴 STUN。在原生節點之間（go-libp2p、rust-libp2p），AutoNAT 是首選。

13.5 Circuit Relay v2：分散式 TURN

第 7 章已介紹過 Circuit Relay v2 的核心機制。本章補充它在 libp2p 整體中的位置：

13.5.1 Reservation 流程

Listener (NAT 後) ──► Relay (公網):
  /libp2p/circuit/relay/0.2.0/hop  HOP RESERVE

Relay 回應：
  STATUS = OK
  Reservation:
    expire: 在 2024-04-28T11:30:00Z 過期
    voucher: <簽章 voucher>  ← 證明 reservation 真實
  Limit:
    duration: 60 seconds
    data: 128 KiB

Listener 收到後，把以下 multiaddr 加入自己的廣告位址：
  /ip4/1.2.3.4/tcp/4001/p2p/RelayPeerID/p2p-circuit/p2p/MyPeerID

13.5.2 Dial 流程

Dialer 發現 multiaddr（透過 DHT 或 PEX）：
  /ip4/RELAY_IP/tcp/4001/p2p/RelayID/p2p-circuit/p2p/TargetID

1. Dialer 連到 RelayID
2. 在連線上開新 stream:
   /libp2p/circuit/relay/0.2.0/hop  HOP CONNECT  TargetID
3. Relay 收到後對 TargetID 開新 stream:
   /libp2p/circuit/relay/0.2.0/stop  STOP CONNECT  DialerID
4. Target 接受，雙方各自確認
5. Relay 在兩個 stream 之間 byte-pump（純 raw 轉送）
6. Dialer 與 Target 在這條 relay 連線上做 Noise/TLS 握手 + multiplex
7. ★ 立即啟動 DCUtR 嘗試升級為直連

13.6 DCUtR：本書理論的最高峰

Direct Connection Upgrade through Relay 是 libp2p 對 ICE 的去中心化重新詮釋。 我們在第 6 章已介紹過它的同步機制；本章看它在 libp2p 整體中的位置與細節。

13.6.2 設計亮點

RTT-based 同步

不像 ICE 的 freezing/throttling 啟發式機制，DCUtR 直接量 RTT、等半 RTT。 數學上的精準勝過工程啟發法。

無集中 signaling 伺服器

relay 本身就是 signaling channel——不需要額外的 WebSocket 伺服器、不需要規劃 signaling 協議。 對去中心化網路至關重要。

對 TCP 與 QUIC 都有效

QUIC 上 DCUtR 比 TCP 簡單（QUIC 連線從 client hello 包開始就能用 path migration）。 TCP 需要 simultaneous open + 多次重試。但兩者都行。

連線反向作為輔助

當 Sync 訊息沒被回應、但對方有公網位址時，DCUtR 會 fall back to「請對方反向 dial 我」。 這對「一邊公網、一邊 NAT」的非對稱場景極友好。

13.6.3 為什麼 RTT/2 等待？

假設 RTT = 200ms（A 到 B 的 round-trip 透過 relay）

Alice 的視角時序：
  T = 0   發 Connect
  T = 200 收到 Bob 的 Connect (RTT 量得)
  T = 200 發 Sync
  T = 300 「按公式應該動手」(等了 RTT/2 = 100ms)

Bob 的視角時序：
  T = 100 收到 Alice 的 Connect (假設網路對稱，relay 在中間)
  T = 100 發回 Connect
  T = 300 收到 Alice 的 Sync
  T = 300 立刻動手

→ Alice 與 Bob 在 T = 300 (絕對時間) 同時 dial
→ Dial 包在中間相遇（每包單程 ~RTT/2）
→ 雙方 NAT 同時為對方開洞
→ 成功！

直觀理解：「等半 RTT」是因為 Sync 訊息走了 RTT/2 到達 Bob，
       Alice 此時直接 dial 也走 RTT/2 在中間與 Bob 的 dial 相遇。

13.7 真實世界的成績單：2025 年 IPFS 大規模測量

2025 年 ArXiv 上發表的論文〈Challenging Tribal Knowledge: Large Scale Measurement Campaign on Decentralized NAT Traversal〉提供了 DCUtR 在實際生產網路的權威數據。

13.7.1 研究規模

  • 440 萬次穿越嘗試
  • 橫跨 85,000+ 個獨立網路
  • 覆蓋 167 個國家
  • 觀察期間 1 年以上
  • 透過 libp2p 的 punchr 工具收集

13.7.2 五大發現

發現 1 · 整體成功率 整體 hole punching 成功率約 70% (±7.1%)。這是迄今為止去中心化NAT 穿越的最權威基準。 注意這是「DCUtR 階段」的成功率，不含因網路故障無法到達 relay 的情況。
發現 2 · 同步效率極高 成功的穿越中，絕大多數在第一次嘗試就成功。 這證明 RTT-based 同步機制非常精準——不需要 ICE 那樣多輪嘗試。
發現 3 · TCP ≈ QUIC（推翻部落知識） 最重要的發現：TCP 與 QUIC 的穿越成功率統計上沒有顯著差異。 這推翻了「UDP 一定優於 TCP」的長期信念。關鍵在於有沒有精準同步， 而非傳輸層本身。Bryan Ford 2005 年論文中 TCP 64% / UDP 82% 的差距， 在現代精準同步協議下幾乎消失。
發現 4 · Relay 特性影響不大 穿越成功率與所選 relay 的特性（頻寬、延遲、地理位置）幾乎無關。 這證明 DCUtR 的設計成功與 relay 解耦——relay 只是 signaling channel。
發現 5 · Connection Reversal 顯著加分 當其中一方有公網位址但對方在 NAT 後時，「連線反向」機制顯著提升成功率。 這在「公網 server 與 NAT 後 client」場景特別重要。

13.7.3 給 P2P 設計者的啟示

這份研究的意義不只在數字。它告訴我們：

  • 不要再用 2005 年的 TCP/UDP 偏見來做架構決定——現在 QUIC 與 TCP 都是合理選項
  • 同步精準度才是關鍵——任何新協議都應該學 DCUtR 量 RTT 的智慧
  • 70% 是基準線——你的系統若低於這個數字，需要檢討；高於它需要量化證明
  • 剩下 30% 主要是 symmetric × symmetric 雙重 NAT 場景——這部分仍需 relay

13.8 一個 libp2p Hello World

用 Go 寫一個極簡 libp2p 節點：

package main

import (
    "context"
    "fmt"
    libp2p "github.com/libp2p/go-libp2p"
    "github.com/libp2p/go-libp2p/core/host"
    "github.com/libp2p/go-libp2p/core/peer"
)

func main() {
    // 建立 host - 自動選擇 transport, security, multiplexer
    h, err := libp2p.New(
        libp2p.ListenAddrStrings(
            "/ip4/0.0.0.0/tcp/0",
            "/ip4/0.0.0.0/udp/0/quic-v1",
        ),
        libp2p.NATPortMap(),       // 自動嘗試 UPnP/NAT-PMP
        libp2p.EnableHolePunching(), // ← DCUtR 自動啟用
        libp2p.EnableAutoRelay(),  // ← Circuit Relay 自動申請 reservation
    )
    if err != nil { panic(err) }

    fmt.Println("PeerID:", h.ID())
    for _, addr := range h.Addrs() {
        fmt.Printf("  %s/p2p/%s\n", addr, h.ID())
    }

    // 連到一個 bootstrap node
    bootstrap, _ := peer.AddrInfoFromString(
        "/dnsaddr/bootstrap.libp2p.io/p2p/QmNnooDu7bfjPFoTZYxMNLWUQJyrVwtbZg5gBMjTezGAJN",
    )
    h.Connect(context.Background(), *bootstrap)

    select {} // 永久運行
}

這 30 行程式碼啟動了一個完整 libp2p 節點： 支援 TCP + QUIC、Noise/TLS 加密、yamux multiplex、自動 NAT 穿越（UPnP + AutoNAT + Relay + DCUtR）、 並透過 bootstrap 進入 IPFS 全球網路。在 ICE/STUN/TURN 的世界寫等價程式至少要 500 行。

13.9 libp2p 之上的協議

libp2p 只是 transport 層；真正的 P2P 應用建立在它之上的應用協議：

Kademlia DHT

第 10 章詳述。libp2p 的版本叫 kad-dht，用於 peer routing 與 content routing。 IPFS 的 provider records 都存這。

Bitswap

IPFS 的「BitTorrent 的繼承者」。內容定址、單一全球 swarm、需要哪個 block 就向 wantlist 廣告。

Gossipsub

抗 Sybil 的 pub/sub 協議——把訊息廣播到「有興趣的」節點。Ethereum 的 consensus layer 用它做區塊與 attestation 廣播。

Noise / TLS Identity

身份驗證協議——透過密碼學金鑰證明 PeerID。所有 libp2p 連線都先做這個握手。

13.10 章末提要

本章核心觀念
  • libp2p 是模組化 P2P 框架，每一層（transport / security / multiplex）都可替換。
  • PeerID 是密碼學身份，與位址解耦；multiaddr 是階層式編址，能巢狀表達 relay 路徑。
  • AutoNAT 用實測（不是觀察）判斷可達性，比 STUN 嚴謹；瀏覽器場景 fallback 到 STUN。
  • Circuit Relay v2 用 reservation 機制與 voucher，讓任何公網節點都能無痛當 relay。
  • DCUtR 用 RTT 量測 + 半 RTT 等待，達到精準同步打洞，無需集中 signaling。
  • 2025 年 IPFS 測量：70% 整體成功率、TCP ≈ QUIC、第一次嘗試多半成功。
  • libp2p hello world 30 行 Go 即可，等價 WebRTC 至少需 500 行。
