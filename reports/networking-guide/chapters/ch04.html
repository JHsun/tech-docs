CHAPTER 4 控制面協作型穿越

本章導讀

這是 Part II 的開場章節，也是最「樂觀」的一章——它假設 NAT 設備願意配合。 我們會看三套相關協議：UPnP IGD（最古老、最普及）、NAT-PMP（Apple 的反擊）、 以及 PCP（IETF 的整合方案）。它們都解決同一個問題：「請 NAT 幫我開一個 port」。 這條路線在家用環境威力強大，但在企業與行動網路幾乎無用——理解它的適用邊界， 跟理解協議本身一樣重要。

📚 學習目標

  • 能說明「協作型穿越」與「協作 NAT」這個概念光譜的位置。
  • 能描述 UPnP IGD 的探索（SSDP）與控制（SOAP）兩階段流程。
  • 能說明 NAT-PMP 為何在 UPnP 之後仍被設計出來。
  • 能說明 PCP 相對前兩者的核心改進，以及為何它仍未取代它們。
  • 能評估這類協議在現代部署中的安全性與可用性風險。

4.1 協作型穿越的哲學

回顧第 1 章的三條穿越路線——本章屬於路線 A：請 NAT 開洞。 它的核心假設是：NAT 設備上跑著一個「管理介面」，端點裝置可以透過這個介面， 正式地請求 NAT 幫自己預留一個外部 port。

跟下一章要介紹的 STUN 不同——STUN 只是觀察 NAT 的行為， 協作型協議則主動指揮 NAT 改變行為。 這也是它的優點與致命缺陷：能力強大，但要求 NAT 願意聽命。

4.2 UPnP IGD：最普及但最尷尬

UPnP（Universal Plug and Play）是微軟在 1999 年主導的家庭網路設備互通框架。 它的IGD（Internet Gateway Device）子協議專門處理 NAT 穿越。 到了 2026 年，UPnP IGD 仍然是最廣泛部署於家用路由器的穿越協議，但它有一個複雜的歷史包袱。

4.2.1 UPnP 的兩階段流程

sequenceDiagram participant C as Client (192.168.1.42) participant R as Router (UPnP IGD) participant N as Multicast
239.255.255.250:1900 Note over C,R: 階段一：探索 (SSDP) C->>N: M-SEARCH
(UDP, ssdp:discover) N->>R: 廣播 R->>C: HTTP 200
LOCATION: http://192.168.1.1:5000/desc.xml Note over C,R: 階段二：取得設備描述 C->>R: GET /desc.xml R->>C: XML（內含 controlURL） Note over C,R: 階段三：操作（SOAP） C->>R: POST controlURL
SOAP: AddPortMapping Note over R: 新增映射規則：
External 9999 → 192.168.1.42:9999 R->>C: SOAP 200 OK
圖 4-2 ｜ UPnP IGD 的探索 + 控制流程

UPnP 的設計很「90 年代」——基於 HTTP + XML + SOAP，over UDP multicast。 它的探索協議叫 SSDP（Simple Service Discovery Protocol）， 端點向 239.255.255.250:1900 廣播 M-SEARCH， 路由器若支援 IGD 就會回應自己的設備描述 URL。

4.2.2 一個 AddPortMapping 的真實 SOAP 請求

POST /upnp/control/WANIPConn1 HTTP/1.1
Host: 192.168.1.1:5000
Content-Type: text/xml; charset="utf-8"
SOAPAction: "urn:schemas-upnp-org:service:WANIPConnection:1#AddPortMapping"

<?xml version="1.0"?>
<s:Envelope xmlns:s="http://schemas.xmlsoap.org/soap/envelope/">
  <s:Body>
    <u:AddPortMapping xmlns:u="urn:schemas-upnp-org:service:WANIPConnection:1">
      <NewRemoteHost></NewRemoteHost>
      <NewExternalPort>9999</NewExternalPort>
      <NewProtocol>UDP</NewProtocol>
      <NewInternalPort>9999</NewInternalPort>
      <NewInternalClient>192.168.1.42</NewInternalClient>
      <NewEnabled>1</NewEnabled>
      <NewPortMappingDescription>MyApp</NewPortMappingDescription>
      <NewLeaseDuration>3600</NewLeaseDuration>
    </u:AddPortMapping>
  </s:Body>
</s:Envelope>

這個請求要求路由器把「外部 UDP 9999」映射到「192.168.1.42:9999」，租期 3600 秒。 租期到了會自動失效，所以應用必須定期續租。

4.2.3 UPnP 在實作上的指令集

SOAP Action 用途
GetExternalIPAddress取得 NAT 的對外公網 IP
AddPortMapping新增一條 port forwarding 規則
DeletePortMapping刪除規則
GetGenericPortMappingEntry列出所有現有映射（用於診斷）
GetSpecificPortMappingEntry查特定 port 的映射
AddAnyPortMapping (IGD v2)「給我任何可用的 port」

4.2.4 UPnP 的安全爭議

UPnP 的設計在當年是「方便至上」，但這份方便埋下深遠的安全債：

UPnP 三大安全問題
  • 無認證：任何能訪問內網的程式（包括惡意軟體）都能新增 port mapping，把自己暴露到公網
  • WAN 側暴露：曾經有為數驚人的家用路由器（2013 年估計超過 5000 萬台）把 UPnP 介面暴露在 WAN 側，攻擊者可遠端利用
  • UPnProxy 攻擊：攻擊者用 UPnP 把路由器當代理，把自己的惡意流量轉送到內網主機

正因如此，許多企業 IT 政策強制關閉 UPnP，部分新版作業系統也預設不啟用客戶端。 在開發 P2P 應用時，把 UPnP 當作「有就用，沒有也別意外」是健康的心態。

4.2.5 程式範例（Python miniupnpc）

import miniupnpc

upnp = miniupnpc.UPnP()
upnp.discoverdelay = 200
n = upnp.discover()
print(f"找到 {n} 台 IGD 裝置")

upnp.selectigd()
print(f"外部 IP: {upnp.externalipaddress()}")

# 新增一條 UDP 9999 → 192.168.1.42:9999 的映射
ok = upnp.addportmapping(
    9999,            # external port
    'UDP',
    upnp.lanaddr,    # internal client (此機 IP)
    9999,            # internal port
    'MyP2PApp',      # description
    ''               # remote host (空字串 = 任意)
)
print("AddPortMapping:", ok)

4.3 NAT-PMP：Apple 的極簡反擊

2005 年，Apple 對 UPnP 那套又重又難用的設計受夠了，推出了 NAT-PMP （NAT Port Mapping Protocol，RFC 6886）。它的設計哲學是：越簡單越好。

4.3.1 NAT-PMP 的封包格式

NAT-PMP 完全摒棄 HTTP/XML/SOAP，直接使用二進位 UDP 封包， 發送到路由器的 port 5351。一個請求只有 12 個位元組： 

┌────────┬────────┬─────────┬──────────┬──────────┬──────────┐
│ 0      │ 1      │ 2-3     │ 4-5      │ 6-7      │ 8-11     │
├────────┼────────┼─────────┼──────────┼──────────┼──────────┤
│ Vers=0 │ Op=1/2 │ Reserved│ Internal │ External │ Lifetime │
│        │ (UDP/  │  (= 0)  │   Port   │   Port   │ (秒)     │
│        │  TCP)  │         │          │ (建議值) │          │
└────────┴────────┴─────────┴──────────┴──────────┴──────────┘

對比 UPnP 的數百行 XML/SOAP，NAT-PMP 的精簡令人耳目一新。

4.3.2 NAT-PMP 的特色操作

Op 0 · 取得外部 IP

送一個 2 byte 的請求，路由器回應自己的 WAN IP。 比 UPnP 的整套 SOAP 流程簡單十倍。

Op 1/2 · 建立 UDP/TCP 映射

送內部 port + 建議外部 port + 租期， 路由器回實際分配的外部 port（可能與建議值不同）。

Multicast 通知

當 WAN IP 改變時，路由器主動廣播給所有客戶端， 這是 UPnP 沒有的優雅特性。

無認證 (同 UPnP)

內網任何人都能呼叫，這個問題 NAT-PMP 也沒解決。 要解決得等下一代 PCP。

4.3.3 部署現狀

NAT-PMP 在 Apple 的 AirPort 系列路由器中是一等公民， 在 OpenWrt、pfSense 等開源系統中也廣泛支援。 但 Windows 與多數家用路由器仍然以 UPnP 為主，所以實務上 P2P 應用通常會同時嘗試兩種協議。

4.4 PCP：IETF 的整合答案

PCP（Port Control Protocol，RFC 6887，2013）是 NAT-PMP 的繼承者， 由 IETF 主導標準化。它把 NAT-PMP 的優點（簡單、二進位、UDP）保留下來， 並加上幾個關鍵新功能：

① 適用 IPv6 防火牆

NAT-PMP 假設一定有 NAT。但 IPv6 通常無 NAT，但仍有防火牆。 PCP 設計成可以對 IPv6 防火牆說「請允許這個 port 的入站連線」。

② 支援 CGNAT

RFC 6887 明確設計了在多層 NAT（包含電信業者 CGNAT）下的協作流程， 讓終端能繞過 ISP 那層 NAT 申請外部映射。

③ Anycast Server

PCP 標準化了 anycast 位址 192.0.0.9 (IPv4) 與 2001::1 (IPv6)， 端點不必先發現路由器，直接送到 anycast 位址即可。

④ MAP 與 PEER opcode

除了傳統的 port mapping (MAP)，新增 PEER opcode： 告訴 NAT「我要連到這個遠端，請給我一條穩定的對外映射」。 對 P2P 應用更精準。

4.4.1 PCP 對 P2P 的特殊意義：PEER opcode

傳統 UPnP/NAT-PMP 的思路是「我要監聽，請開外部 port」——這是伺服器視角。 但 P2P 應用更常見的場景是「我要連這個 peer，請保持映射穩定」——這是對等視角。 PCP 的 PEER opcode 直接表達後者： 

PEER request:
  internal_port:  54321
  remote_peer_ip: 203.0.113.99
  remote_peer_port: 6881
  lifetime: 7200

意思：「我要連 203.0.113.99:6881，
       請給我一個穩定的對外 endpoint，並保持這個映射 2 小時」

回應：
  external_address: 198.51.100.1
  external_port: 62000
  
端點現在可以告訴對方：「請連我 198.51.100.1:62000」

這個機制本質上是 hole punching 的「正式版」——你不再需要透過 STUN 推測 NAT 行為， 而是直接命令 NAT 給你一個穩定的映射。

4.4.2 為什麼 PCP 沒有取代 UPnP？

從技術角度，PCP 全方面優於 UPnP IGD。但十幾年過去，UPnP 仍是部署主流。原因不在技術：

部署慣性 (deployment inertia)
  • 路由器廠商已經在每台設備中放入 UPnP，重新整合 PCP 需要 firmware 大改
  • 應用開發者看到 UPnP 「能用」就不想多寫一套 PCP fallback
  • IPv6 普及不如預期，PCP 對 IPv6 的優勢未能變現
  • 整個產業的「都還行」心態，讓更好的技術無法取代「夠用」的舊技術

這是網路標準史上反覆上演的劇本：better is the enemy of good enough。 認識這一點，能讓你在做技術選型時更務實——不要押注於「未來會普及」的協議， 要設計成同時支援主流與最新方案。

4.6 實務策略：fallback 鏈

既然三套協議都不完美，且部署環境差異很大，正確的做法是依序嘗試：

flowchart TD Start([開始嘗試協作型穿越]) Start --> P1[嘗試 PCP
送至 anycast 192.0.0.9] P1 -->|成功| Done1[✓ 取得映射] P1 -->|超時| P2[嘗試 NAT-PMP
送至 default gateway:5351] P2 -->|成功| Done2[✓ 取得映射] P2 -->|超時或拒絕| P3[嘗試 UPnP IGD
SSDP M-SEARCH] P3 -->|找到 IGD| P3b[呼叫 AddPortMapping] P3b -->|成功| Done3[✓ 取得映射] P3 -->|找不到 / 拒絕| Fall[Fall back to STUN
或 hole punching] P3b -->|拒絕| Fall classDef ok fill:#2d5a3d,stroke:#2d5a3d,color:#fff classDef fb fill:#b8761d,stroke:#b8761d,color:#fff class Done1,Done2,Done3 ok class Fall fb
圖 4-3 ｜ 協作型穿越的 fallback 鏈，最後仍可能落到後續章節的方法

這個 fallback 設計有個重要細節：每個嘗試應該設超時（建議 250ms ~ 1s）， 不要等太久。整個協作型穿越流程應在 3 秒內結束，否則會嚴重拖慢應用啟動體驗。

4.6.1 程式骨架（Python 偽代碼）

async def try_collaborative_traversal(internal_port: int):
    # PCP: 250ms timeout, 試 anycast 與 default gateway
    for target in ['192.0.0.9', get_default_gateway()]:
        try:
            result = await pcp_request(target, internal_port, timeout=0.25)
            if result.success:
                return result.external_addr
        except (TimeoutError, ConnectionRefusedError):
            continue

    # NAT-PMP: 500ms timeout
    try:
        result = await natpmp_request(get_default_gateway(),
                                      internal_port, timeout=0.5)
        if result.success:
            return result.external_addr
    except TimeoutError:
        pass

    # UPnP: 較慢，給 2s
    try:
        igd = await upnp_discover(timeout=2.0)
        if igd:
            ext_port = await igd.add_port_mapping(internal_port, 'UDP')
            return (await igd.get_external_ip(), ext_port)
    except Exception:
        pass

    # 全部失敗，呼叫者應 fall back
    return None

4.7 安全提示

作為 P2P 應用開發者你應該注意
  1. 不要長期持有大量映射：用完即釋放，避免讓使用者的 NAT 表佔滿條目，影響其他應用
  2. 明確設置租期：通常 1~2 小時，由應用主動續租，比要求永久映射更負責
  3. 使用易識別的描述：在 PortMappingDescription 中寫明應用名，方便使用者除錯
  4. 釋出時呼叫 DeletePortMapping：應用退出前清乾淨，避免「殭屍映射」
  5. 不要重試到天荒地老：失敗就 fallback，不要瘋狂重試騷擾路由器
作為終端使用者你應該注意
  • 確認你家路由器的 UPnP 介面沒有暴露在 WAN 側（用 nmap -sU -p 1900 從外部測）
  • 定期檢查路由器後台的「目前 UPnP 映射」清單，看是否有不認識的條目
  • 若不需要 P2P 應用穿越，直接關閉 UPnP是最安全的選擇
  • 企業環境應一律關閉並改用 VPN / ZTNA

4.8 章末提要

本章核心觀念
  • 協作型穿越是「請 NAT 開洞」，前提是 NAT 願意配合。
  • UPnP IGD 普及但設計沉重，且有嚴重歷史安全問題。
  • NAT-PMP 簡潔有力，但生態以 Apple 為主。
  • PCP 是 IETF 的繼承方案，技術全方位更優，但部署緩慢。
  • 實務做法是依序 fallback：PCP → NAT-PMP → UPnP → STUN/hole punching。
  • 所有協作型協議在企業與行動電信網路幾乎都無效，必須有後備方案。
