CHAPTER 14 現代 Mesh VPN：Tailscale、ZeroTier、Nebula

本章導讀

傳統 VPN 是「Hub-and-Spoke」星型——所有流量繞回中央伺服器。 現代 Mesh VPN 反其道——讓每兩台裝置之間都能直接連線，中央只負責協調與認證。 本章介紹三套主流 Mesh VPN：Tailscale（基於 WireGuard，business-friendly）、 ZeroTier（自建協議，全球 anycast 根伺服器）、Nebula（Slack 開源）。 它們是本書所有概念的商業應用結晶——把 STUN、hole punching、DERP relay、加密、身份系統， 全部包裝成「裝個客戶端、登入、就連通」的開發者體驗。

📚 學習目標

  • 能說明 Mesh VPN 與傳統 VPN 在拓樸上的差異。
  • 能描述 Tailscale 的「coordination plane / data plane」分離設計。
  • 能解釋 WireGuard 為何是現代 VPN 的密碼學基礎。
  • 能比較 Tailscale、ZeroTier、Nebula 三者的設計取捨。
  • 能描述 ACL 與身份識別在 mesh VPN 中的角色。
  • 能評估 Mesh VPN 在企業遠距辦公與 IoT 的應用。

14.1 從 Hub-Spoke 到 Mesh：拓樸演進

圖 14-1 ｜ 傳統 VPN 的繞路問題：同地辦公的同事流量繞地球

傳統企業 VPN（OpenVPN、IPSec、SSL VPN）採 hub-and-spoke：所有遠端使用者連到一個或數個 VPN 集中器， 彼此通訊都要繞回 hub。問題：

  • 延遲：兩個東京使用者透過美國 hub 對話，RTT 200ms+
  • 頻寬瓶頸：所有流量過 hub，hub 頻寬就是上限
  • 單點失效：hub 掛了所有人斷線
  • 成本：高頻寬 hub 維護費用昂貴

關鍵設計：把控制平面（control plane）與資料平面（data plane）分離—— 中央伺服器負責認證、ACL、key 分發；實際流量走 P2P。 這個架構需要本書前面幾乎所有技術——身份、加密、NAT 穿越、relay fallback。

14.2 WireGuard：現代 VPN 的密碼學基礎

Mesh VPN 之所以能在 2018 年後爆發，WireGuard 是關鍵推手。 它由 Jason Donenfeld 設計，2020 年正式進入 Linux kernel。設計哲學「少即是多」：

極簡程式碼

核心約 4000 行 C（OpenVPN 約 10 萬行）。攻擊面小、易審計、可整合進 kernel。

一個密碼學套件

固定使用 ChaCha20-Poly1305、Curve25519、BLAKE2、HKDF—— 不像 OpenVPN/IPSec 有 cipher 協商。沒有 cipher downgrade 攻擊。

UDP-only

只跑 UDP，沒 TCP 模式。在企業環境靠 wireguard-go 或 udp2tcp 等轉換。

無連線狀態

每個 peer 用 public key 識別，雙方有對方 pub key 就能直接通訊。 沒有 handshake 重新協商的複雜狀態機。

14.2.1 WireGuard 的限制——為什麼還需要外殼？

純 WireGuard 是極簡的：

  • 需要手動分發每個 peer 的 public key
  • 需要手動配置每個 peer 的 endpoint（IP:port）
  • 沒有身份驗證（key 就是身份，誰偷到 key 就能用）
  • 沒有 NAT 穿越——預設只支援「有公網 endpoint」的場景
  • 沒有 ACL、沒有使用者管理

Mesh VPN 廠商（Tailscale、Netbird、Headscale）做的事， 本質上就是把 WireGuard 包裝成可運維的產品—— 加上 SSO 認證、自動 key 分發、自動 NAT 穿越、ACL、稽核日誌。

14.3 Tailscale 深度剖析

Tailscale 是 Mesh VPN 領域最廣為人知的商業產品，創辦人之一 Avery Pennarun 也是 sshuttle 與 dvcs-autosync 的作者。它的設計直接影響了整個產業。

14.3.2 NetworkMap：協調的核心

Tailscale coordination server 對每個節點維護一個「NetworkMap」—— 包含網路中所有 peer 的 PubKey、可能的 endpoint 候選、ACL 策略。 當有新節點加入或變動，server 推送更新給所有相關節點。 

{
  "self": {
    "id": "alice@example.com",
    "key": "AAAA...",
    "addresses": ["100.64.0.1/32"]
  },
  "peers": [
    {
      "id": "bob@example.com",
      "key": "BBBB...",
      "endpoints": [
        "192.168.1.55:41641",        ← LAN
        "203.0.113.99:41641",        ← public (STUN 探得)
        "derp-tokyo.tailscale.com"   ← relay fallback
      ],
      "allowed_ips": ["100.64.0.2/32"]
    }
  ]
}

14.3.4 DERP：Tailscale 的特色 relay

第 7 章已介紹過 DERP 的「always-on relay」範式。回顧重點：

  • 所有連線「技術上」從 DERP 開始，立刻可通
  • 背景嘗試 NAT 穿越；成功則無痛升級
  • 內部數據顯示 > 90% 的連線最終升級為直連
  • DERP 跑在 HTTPS 443 上，幾乎能穿越所有企業防火牆
  • 所有流量在端對端 WireGuard 加密下，DERP 看不到內容

14.3.5 MagicDNS 與 100.x.x.x 位址空間

Tailscale 給每個節點分配一個 100.64.0.0/10 範圍的 IP（CGNAT 保留範圍，不會與真實 LAN 衝突）。

$ tailscale status
100.64.0.1   alice-laptop    macOS    -
100.64.0.2   bob-laptop      Windows  active; direct
100.64.0.3   home-server     Linux    active; relay "derp-tokyo"

# 此後 SSH/HTTP/任何協議都能直接用：
$ ssh alice-laptop                # MagicDNS 自動解析
$ curl http://home-server:8080
$ ping 100.64.0.2

這個體驗是 Mesh VPN 的核心賣點——「裝起來就像把所有裝置放進同一個區網」。 使用者完全不需要關心 NAT、防火牆、port forwarding——這些複雜性被 Tailscale 用本書前 12 章的技術抽象掉了。

14.4 ZeroTier 對比：另一條路

ZeroTier 比 Tailscale 早 5 年（2014 年），採用了不同的設計選擇：

面向TailscaleZeroTier
底層協議WireGuard自有協議（基於 ECC + libsodium）
網路層L3 (IP)L2 (Ethernet) — 可廣播 ARP、DHCP
關鍵路徑DERP (商業擁有)全球 root servers (anycast)
身份驗證SSO / SAML / OAuth自有帳號或自架
價格定位商業優先免費仁慈、付費企業
自架選項Headscale (社群)原生支援自建 root
NAT 穿越成功率高（~90%）中（依賴 root server 中繼）

ZeroTier 走 L2 是有趣的選擇——能模擬「同一個交換機」的環境， 舊應用（依賴 broadcast、依賴 DHCP）能無痛遷移。代價是更高的協議複雜度與廣播流量。

14.5 Nebula：Slack 的開源選擇

Nebula 由 Slack 在 2019 年開源，是上述兩者的輕量自託管替代方案。 特色：

純自託管

沒有商業中央伺服器。使用者自己建 lighthouse（類 coordination server）， 全部基礎設施可控。

PKI 證書

用 X.509-like 證書簽發，CA 私鑰本地保管。 每個 host 證書包含 IP、群組標籤、過期時間。

Lighthouse 機制

少數 lighthouse 節點維護「誰在哪裡」的資訊。 新節點啟動時對 lighthouse 註冊，查詢時也問 lighthouse。 類似 Tailscale coordination server 但更輕量。

無 GUI 工具

Slack 把它設計給有運維能力的團隊用—— 純 CLI、純 config 檔。對 SRE 友善，對普通使用者門檻較高。

選擇 Nebula 的場景：

  • 不想把網路 metadata 交給商業廠商
  • 有運維能力，想自託管所有元件
  • 純伺服器之間的網路（不是員工筆電）
  • 有合規要求（如 air-gapped）

14.6 ACL：Mesh VPN 的精髓

傳統 VPN「進來了就什麼都看得到」是現代零信任反對的。Mesh VPN 都內建細粒度 ACL： 

// Tailscale ACL 範例
{
  "groups": {
    "group:engineers":  ["alice@", "bob@", "carol@"],
    "group:contractors": ["external@"]
  },
  "tagOwners": {
    "tag:server":     ["group:engineers"],
    "tag:dev-only":   ["group:engineers"]
  },
  "acls": [
    // 工程師可訪問所有伺服器
    {
      "action": "accept",
      "src": ["group:engineers"],
      "dst": ["tag:server:*"]
    },
    // 外部承包商僅可訪問 dev-only 機器的 80/443
    {
      "action": "accept",
      "src": ["group:contractors"],
      "dst": ["tag:dev-only:80,443"]
    },
    // 預設 deny（隱含）
  ],
  "ssh": [
    // 工程師可 SSH 進伺服器，但要 reauth 每天
    {
      "action": "check",
      "src": ["group:engineers"],
      "dst": ["tag:server"],
      "users": ["root", "ubuntu"],
      "checkPeriod": "24h"
    }
  ]
}

這份 ACL 體現了「身份-資源」級別的政策——不再是「網段-網段」級別。 搬離了傳統 firewall 的局限，是零信任架構（ZTNA）的具體展現。

14.7 應用案例

14.7.1 遠距辦公

這是 Mesh VPN 最大的市場。COVID-19 後，企業急需讓遠端員工安全地連回公司資源， 傳統 VPN 集中器頻寬撐不住、體驗也差。Tailscale 的「安裝 → 登入 → 連通」三步驟， 比 IT 部門配置 OpenVPN profile 給每個員工順暢一個數量級。

14.7.2 家用 NAS / 自架服務

熱門 self-hosted 場景。把家裡的 Synology、Plex、Home Assistant 加入 mesh， 外面手機就能直接連——不需要 port forwarding、不需要動態 DNS、不需要繳網域費。 Tailscale 與 ZeroTier 在這個社群極受歡迎。

14.7.3 IoT 與邊緣裝置

邊緣裝置（智慧電表、攝影機、感測器）通常在 NAT 後且無 inbound 能力。 傳統做法是輪詢中央 API；Mesh VPN 讓它們看起來像在公司內網， 維運可以直接 SSH 進去除錯。Tailscale 的 subnet router 模式特別適合這類場景。

14.7.4 多雲互連（Multi-cloud）

AWS、GCP、Azure 之間的網路互通向來複雜（要設定 VPN gateway、BGP、peering）。 把每個 VM 都加入 mesh 後，跨雲通訊就跟本地一樣——大幅簡化雲架構。

14.8 評估與選型指引

何時選哪個？
  • 個人 / 小團隊 / 自家 NAS → Tailscale 免費版（< 100 裝置）
  • 企業遠距辦公 + SSO 整合 → Tailscale 商業版
  • 需要 L2 廣播 / 模擬區網 → ZeroTier
  • 純伺服器互連、不想商業依賴 → Nebula 或 Headscale + WireGuard
  • 有 IT 維運能力，要完全自託管 Tailscale 體驗 → Headscale（開源 coordinator）
  • 軍工、高合規環境 → Nebula 或自建 IPSec

14.9 章末提要

本章核心觀念
  • Mesh VPN 把流量從 hub-and-spoke 解放成 N×N 直連，控制平面與資料平面分離。
  • WireGuard 是現代 mesh VPN 的密碼學基礎——極簡、固定 cipher、kernel-level。
  • Tailscale 用 coordination server + DERP relay + WireGuard，達到「裝即用」體驗。
  • 「先用 relay 連通、後升級為直連」是 Tailscale DERP 的標誌性設計。
  • ZeroTier 走 L2 + 自有協議，適合需要廣播語意的舊應用。
  • Nebula 純自託管，適合有運維能力且不想依賴商業廠商的團隊。
  • ACL 從「網段-網段」進化到「身份-資源」級，是零信任架構的具體表現。
  • Mesh VPN 是本書前 13 章技術的商業終極形態——把所有複雜性打包成「裝個 client、登入」的體驗。

動腦時間

  1. Tailscale 的 DERP 數據顯示 90%+ 連線會升級為直連。剩下 10% 主要在什麼情境？回想第 6 章測量數據與本章內容。
  2. WireGuard 沒有 cipher 協商機制。這個設計如何避免 downgrade 攻擊？又有什麼代價？
  3. 從零信任的角度：傳統「進到 VPN 就什麼都能訪問」與 Tailscale 的細粒度 ACL， 哪些攻擊面被消除了？
  4. 動手任務：在你家裡建 Tailscale tailnet，把筆電與一台 VPS 加入，觀察 tailscale status 顯示「direct」或「relay」。若是 relay，研究為什麼直連失敗。
  5. 研究 Headscale（社群開源的 Tailscale coordinator 替代品），思考「self-hosted Tailscale」 與「商業 Tailscale」的取捨。
