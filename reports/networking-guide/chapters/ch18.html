CHAPTER 18 部署、量測與除錯

本章導讀

P2P 系統的協議學完了，但運維才是真實世界的考驗。本章是「實務手冊」—— 架設 STUN/TURN/signaling 伺服器、規劃容量、選擇監控指標、用工具除錯典型問題。 我們會走過 coturn 完整生產配置、libp2p punchr 量測工具、Wireshark 抓包要點， 並列出 P2P 系統最常見的十個故障模式與修復路徑。讀完本章你能把前面 17 章的知識， 真正投入運行。

📚 學習目標

  • 能規劃 STUN/TURN 伺服器的容量、地理分布與冗餘。
  • 能說出 P2P 系統至少八個必監控的核心指標。
  • 能用 Wireshark 解析 STUN、ICE、DTLS 流量。
  • 能識別 P2P 系統最常見的十個故障模式。
  • 能說明 punchr 量測工具與 webrtc-internals 的使用方法。

注意：「P2P 不需要伺服器」是常見誤解。真實 P2P 系統幾乎都有輔助基礎設施， 只是它不負責資料流量。設計目標是「infra 故障時退化但不全死」。

18.1.2 容量規劃

STUN（極輕量）

每請求 ~36 byte 入、~88 byte 出。一台 1 Gbps VPS 理論能處理 ~1M QPS。 瓶頸在 PPS（每秒封包數）而非頻寬。

TURN（最重）

承擔實際媒體流量。單台 1 Gbps VPS 可服務： 視訊 ~150 通並發、語音 ~3000 通並發、純資料依需求。 必須水平擴展 + 地理就近。

Signaling（中等）

每連線一條 WebSocket。一台 4 vCPU + 8 GB VPS ~10萬並發連線。 瓶頸通常是 file descriptor 與 memory，不是 CPU。

Bootstrap（中等）

每分鐘約數百 QPS（新節點加入頻率）。一台小 VPS 足夠， 但需要高可用——bootstrap 掛了新節點進不來。

18.1.3 地理分布

基礎設施應放在哪？
  • STUN：用 anycast 全球部署最佳；至少分 3 區域（美/歐/亞）
  • TURN：必須就近使用者——RTT 直接影響通話品質。 Tailscale DERP 在全球 30+ 區域有節點
  • Signaling：地理影響較小（建立連線一次性開銷），可集中部署
  • Bootstrap：地理多樣性比效能重要，5-10 個區域分散

18.2 自架 STUN/TURN：coturn 生產配置

第 7 章給過 coturn 的最小設定；本節提供生產級完整配置：

# /etc/turnserver.conf · 生產級

# === 監聽 ===
listening-port=3478
tls-listening-port=5349
listening-ip=0.0.0.0
external-ip=YOUR.PUBLIC.IP/YOUR.PRIVATE.IP

# === Realm 與認證 ===
realm=turn.example.com
fingerprint
lt-cred-mech

# 短期憑證（推薦給 WebRTC 應用）
use-auth-secret
static-auth-secret=YOUR_LONG_RANDOM_SECRET_HERE
# 配合後端產生 username = "expiry:userid", password = HMAC-SHA1(secret, username)

# === TLS ===
cert=/etc/letsencrypt/live/turn.example.com/fullchain.pem
pkey=/etc/letsencrypt/live/turn.example.com/privkey.pem
cipher-list="ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256"

# === 安全限制 ===
no-tcp-relay              # 禁用 TCP 中繼（防 amplification）
no-multicast-peers
no-loopback-peers
denied-peer-ip=10.0.0.0-10.255.255.255
denied-peer-ip=172.16.0.0-172.31.255.255
denied-peer-ip=192.168.0.0-192.168.255.255
denied-peer-ip=127.0.0.0-127.255.255.255
denied-peer-ip=fc00::-fdff:ffff:ffff:ffff:ffff:ffff:ffff:ffff
allowed-peer-ip=YOUR_OWN_PRIVATE_RANGE  # 只開放必要的私網

# === 速率限制 ===
total-quota=1000          # 同時 allocations 總數
user-quota=10             # 每使用者最多 10 個 allocations
max-bps=1000000           # 每 allocation 1 Mbps
bps-capacity=100000000    # 整台機器 100 Mbps 上限

# === 日誌與監控 ===
log-file=/var/log/turn/turnserver.log
verbose
syslog
prometheus-port=9641      # 暴露 Prometheus 指標
prometheus-username-labels=true

coturn 內建 Prometheus exporter。關鍵指標：

18.3 P2P 系統的核心監控指標

對 P2P 系統，從使用者角度必須監控的指標：

① 連線建立成功率

從 ICE 開始到 connected 的成功比例。目標 > 95%。 分桶：直連成功率、TURN 成功率、整體成功率。

② TURN 使用率

有多少連線最終靠 TURN 中繼？健康 20-30%；> 40% 表示異常。 依網路類型細分（家用、行動、企業）。

③ 連線建立時間（TTC）

P50 / P95 / P99。P95 < 1 秒是合格基準。 若 P99 突增，多半是某地區 STUN/TURN 故障。

④ 媒體品質指標

RTT、jitter、packet loss、framerate。WebRTC 透過 RTCStats API 暴露給應用。 建立基準線後追蹤異常。

⑤ NAT 類型分布

使用者群中 cone vs symmetric 比例。象徵著你的市場面對的網路環境。 可指導基礎設施投資。

⑥ 重連率

連線建立後在 N 分鐘內斷線重連的比例。> 5% 表示穩定性問題。 多由 NAT 表 timeout 或網路切換引起。

⑦ ICE Candidate 收集分布

平均一個 client 收集到幾個 host / srflx / relay？分布異常表示 STUN 故障或網路問題。

⑧ 客戶端版本分布

WebRTC stack（瀏覽器版本）顯著影響成功率。 監控版本分布有助於識別「新版本上線後成功率下降」的回歸。

18.4 量測工具：punchr 與 webrtc-internals

18.4.1 libp2p 的 punchr

punchr 是 libp2p 官方的 hole punching 量測工具，第 13 章引用的 2025 大規模測量就是用它收集的。 它包含：

  • Honeypot——故意架設的「NAT 後 listener」，吸引 DCUtR 嘗試以收集資料
  • gRPC Server——後端，接收測量結果並儲存到 PostgreSQL
  • Punching Client——主動 dial honeypot 並上報結果
# 跑自己的 punchr 量測（簡化版）
$ git clone https://github.com/dennis-tra/punchr
$ cd punchr/cmd/client
$ go run main.go \
    --bootstrap /dnsaddr/bootstrap.libp2p.io/p2p/QmNnoo... \
    --duration 1h

# 結果：每次嘗試的成功/失敗、所用 transport、延遲、relay 路徑等

對運行 P2P 服務的工程團隊，內部部署 punchr-like 工具持續量測自家網路是必要的。

頁籤能看到典型用途

18.4.3 Wireshark 抓包要點

P2P 流量除錯，Wireshark 是底層真相之源。常用 filter：

# STUN 流量
stun

# 篩出 binding request/response
stun.type.method == 0x001 && stun.type.class == 0x000   # request
stun.type.method == 0x001 && stun.type.class == 0x002   # success response

# DTLS 握手
dtls.handshake

# QUIC 流量
quic

# 特定 IP 對的所有流量
ip.addr == 1.2.3.4 && ip.addr == 5.6.7.8

# 找出 ICE 候選對的成功配對
stun.attribute.type == 0x0020   # XOR-MAPPED-ADDRESS

# UDP traffic on TURN ports
udp.port == 3478 || udp.port == 5349

18.5 十個常見故障模式

P2P 系統 Top 10 故障
  1. 連線建立後 30~60 秒斷線 → NAT 表 timeout，缺 keep-alive
  2. 對稱式 NAT 場景全部失敗 → 沒有 TURN fallback 或 TURN 故障
  3. 連線建立極慢（> 5s） → 沒用 Trickle ICE，等所有 candidate 才送 offer
  4. 企業環境完全連不上 → UDP 被擋，需要 TCP-only fallback 或 TURN/TLS:443
  5. 手機切換 Wi-Fi/4G 後斷線 → 沒有 ICE Restart 邏輯
  6. 同 LAN 內兩台連不上 → 沒有 host candidate 或 hairpinning 不支援
  7. TURN 流量突增 → STUN 故障導致大量連線 fall back to relay
  8. 音視訊有延遲漸增 → 緩衝區設定錯誤或丟包率高
  9. 少數使用者完全沒法用 → 雙重 CGNAT / 行動電信特殊環境
  10. 新版本上線後成功率下降 → WebRTC stack 行為變化或 codec 不相容

18.7 容量規劃實例

假設你正在做一個視訊會議產品，預期同時 10,000 通通話（每通 2-4 人）。需要哪些後端？ 

假設：
  - 平均每通 3 人，10,000 通 = 30,000 並發 peer
  - 每對 peer 需要 ICE，假設並行 candidate 5 對 = 50,000 STUN check / 啟動
  - 連線生命週期平均 30 分鐘

STUN：
  - 啟動時瞬時尖峰：50,000 STUN req
  - 穩態 keep-alive 每 25s 一次：30,000/25 = 1,200 RPS
  - 1 台 STUN 伺服器（甚至 1 個 vCPU）就足夠
  - 為冗餘部署 2-3 個區域，每區一台

TURN：
  - 假設健康狀態 20% 通話需要 TURN = 2,000 通
  - 每通 720p 視訊雙向 ~3 Mbps
  - 總 TURN 頻寬：2,000 × 3 Mbps = 6 Gbps
  - 部署 6-8 台 1 Gbps 機器，地理分布

Signaling：
  - 30,000 並發 WebSocket 連線
  - 每連線 ~10 KB 記憶體 → 300 MB
  - 1 台 4 vCPU 伺服器足夠，加 1-2 台冗餘

總計：
  - 約 10 台中型 VPS
  - 月成本 $500-1500（依雲端供應商）

對比集中式視訊會議（無 P2P）：
  - 每通走完整 SFU，10,000 × 12 Mbps = 120 Gbps
  - 月成本可能 $50,000+
  
P2P 為公司省了 30 倍以上的伺服器成本。

18.8 章末提要

本章核心觀念
  • P2P 系統的後端：STUN（輕）、TURN（重）、Signaling（中）、Bootstrap（中）。
  • 容量規劃：STUN 極輕，TURN 是頻寬瓶頸，地理分布對 TURN 至關重要。
  • 核心監控指標八項：成功率、TURN 比例、TTC、媒體品質、NAT 分布、重連率、candidate 分布、版本分布。
  • punchr 是 libp2p 量測工具；webrtc-internals 是 WebRTC 除錯利器。
  • P2P 系統 Top 10 故障半數與 NAT/keep-alive 相關。
  • P2P 為公司省下大量伺服器成本，是其商業價值所在。

動腦時間

  1. 規劃任務：你公司的 P2P 視訊會議產品要進入印度市場（IPv6 普及高、多重 CGNAT 常見）。 後端架構需要做哪些調整？
  2. 觀察 webrtc-internals 一次真實通話。記下所選 candidate pair 的類型、總 STUN check 數、成功的 check 比例。
  3. 監控設計：給「連線建立成功率」設計告警。閾值多少？告警觸發後查什麼？
  4. 架設一台 coturn 並用 turnutils_uclient 跑壓力測試。 記錄你能達到多少並發 allocations。
  5. 論述題：你發現 TURN 使用率突然從 25% 升到 60%。可能原因是什麼？怎麼診斷？
