CHAPTER 1 緒論：為什麼連線這麼困難？

本章導讀

本章從一個最樸素的提問出發：「為什麼我家裡的電腦不能直接被朋友連到？」我們會走過一段網路考古，從 IPv4 位址的供需失衡、 端對端原則（end-to-end principle）的破壞、到中介設備（middleboxes）的崛起，理解「連線困難」這個問題的本質。 讀完本章，你會明白接下來十八章想要解決的，並不是某個單一技術問題，而是一場長達三十年的工程妥協。

📚 學習目標

  • 能解釋 IPv4 位址枯竭如何驅動 NAT 的普及，並影響今日的網路架構。
  • 能複述「端對端原則」並說明它為何被破壞、又為何至今仍是 P2P 設計的北極星。
  • 能列舉至少四種因 NAT/防火牆而設計困難的應用場景。
  • 能分辨「NAT 阻擋」與「防火牆阻擋」兩類本質不同的問題。

1.1 一個樸素的提問

想像一個情境：你和朋友想透過一支自己寫的小程式，在兩台筆電之間傳遞檔案。你以為，網路就像郵差——只要知道對方的地址， 資料封包就能寄到家門口。於是你查了 ipconfig，看到自己 IP 是 192.168.1.42，告訴朋友：「往這個地址寄就好」。

結果，朋友怎麼寄都收不到。為什麼？

這個問題的答案，是這本書要花十八章來回答的。但讓我們先從最簡短的版本開始： 你的 192.168.1.42 不是一個真正能在網路上路由的地址，它只在你家的網域裡有意義。 從外面看你家，你和家裡的電視、手機、印表機共用同一個對外 IP；而這個對外 IP 後面藏了一道 叫做「NAT」的轉換層，它只放行你主動發起的連線回應，對所有未經邀請的封包一律丟棄。

「網路是平等的」這句話，在 1990 年之前還大致成立； 在那之後，它就成了一個技術人需要花生涯前段時間去逐步幻滅的浪漫。

1.2 端對端原則的興與衰

1.2.1 原始設計的優雅

1981 年，Saltzer、Reed 與 Clark 三位學者發表了一篇名為〈End-to-End Arguments in System Design〉的論文。 這篇文章雖然不是 RFC，卻深深影響了網際網路的根本哲學：網路核心應該保持簡單，複雜性放在端點。

在這個願景下，每一台連上網路的電腦都是對等的（peer）： 它擁有一個全球唯一的 IP 位址，可以自由地與任何其他 IP 對話。網路中段的路由器只負責把封包送到目的地， 不關心這封包是 email、檔案、還是視訊。在這樣的世界裡，「server」與「client」只是角色差異，不是身份差異—— 任何一台主機都能同時扮演兩者。

這就是為什麼早期的網際網路應用，無論是 FTP、Telnet、SMTP、還是 IRC，都假設雙方可以直接連線。 那個時代的程式設計師，根本不需要「打洞」這個概念。

1.2.2 IPv4 位址的供需崩塌

1983 年，IPv4 規範完成，總共能容納約 43 億個位址。 在當時的工程師看來，這是一個用不完的天文數字。但他們沒料到的是兩件事： 個人電腦的爆炸性普及、與行動裝置的崛起。

到了 1990 年代中期，網路工程界已意識到 IPv4 位址會枯竭。最徹底的解法是設計新的 IPv6 協議 （把位址擴張到 2128），但 IPv6 的部署遠比想像中困難——這個故事我們留到第 16 章。 與此同時，一個原本只是權宜之計的技術，迅速成為了網路的常態：網路位址轉換，Network Address Translation，NAT。

1.2.3 NAT 的妥協邏輯

NAT 的概念極為樸素：

既然全世界沒有那麼多 IP 可分，那就讓很多裝置共用一個 IP。 路由器在中間做地址翻譯：對內，每台裝置有一個私有 IP；對外，所有裝置都假裝是同一個公網 IP。

這個技巧解決了位址枯竭的燃眉之急，但它徹底破壞了端對端原則。 從此以後，「主動發起連線」與「被動接收連線」變成兩件本質不同的事—— NAT 後方的裝置只能扮演 client，不再能輕易扮演 server。

歷史插曲 RFC 1631（1994）正式定義了 NAT，它的標題清楚寫著：The IP Network Address Translator (NAT)， 並坦言這是一個短期過渡方案。然而 IETF 從未預料到，三十年後我們仍在依賴它——而且還變本加厲， 電信業者把 NAT 疊了好幾層（CGNAT，第 19 章詳述）。

1.3 「不能被連到」造成了哪些問題？

讓我們具體看看，當一台裝置「只能主動、不能被動」時，哪些應用會出問題。

應用場景 核心需求 NAT 帶來的困難
VoIP / 視訊會議 雙方互傳即時音視訊封包，越直接越好 雙方都在 NAT 後，沒有任何一方能主動 listen 來接受對方的 RTP 串流
線上遊戲（FPS / MOBA） 玩家互相傳遞遊戲狀態，避免中央伺服器延遲 P2P 模式無法建立，被迫使用集中式中繼，提高 ping 值
BitTorrent 等檔案分享 每個 peer 同時上傳/下載，越多 peer 互連越快 NAT 後的 peer 無法被連入，只能單向下載，破壞分享經濟
IoT 裝置遠端存取 使用者從外網連到家中攝影機、NAS 必須額外架設「跳板」或仰賴廠商雲端，增加成本與隱私風險
區塊鏈 / 去中心化應用 節點之間自由互連，避免單點審查 大量節點淪為「只接收不貢獻」的拖累者（leecher）
遠端開發 / SSH 進入家用機 從外面回到家中環境 需要動態 DNS + Port Forwarding 或 VPN，技術門檻高

這張表想傳達的是：幾乎所有「兩個一般使用者之間的應用」都會撞上同一個牆。 伺服器/客戶端模型之所以變成主流，並不全是因為它「天生比較好」，而是因為它是 NAT 後唯一還能用的模型。 它是一個結構性後果，不是設計選擇。

1.4 兩種不同的「阻擋」

在繼續往下讀之前，我們必須先建立一個關鍵的心智區分：

NAT 的阻擋

NAT 設備本身沒有阻擋意圖。它的設計目的是位址轉換，不是安全。 但因為它必須維護一張「進來的封包要送給誰」的表， 對於沒在表裡的封包，它不知道該送哪裡，只好丟棄。

→ 因此可以「協作穿越」（hole punching、UPnP）

防火牆的阻擋

防火牆明確意圖阻擋未授權流量。 它有政策、有規則、有審計。當你「穿越」防火牆時， 某種意義上你是在違反該網路擁有者的意願。

→ 必須「偽裝穿越」（隧道、應用層封裝）

這兩種阻擋的應對哲學完全不同。前者是合作問題——我們想讓 NAT 知道該怎麼幫忙； 後者是對抗問題——我們在試圖讓流量看起來像是允許的東西。 本書第二部會以前者為主，第 9 章與第 17 章會處理後者。

技術倫理提醒 繞過企業防火牆（即使技術上做得到）通常違反公司資訊安全政策； 繞過國家級防火牆在某些司法管轄區可能涉及法律風險。 本書討論這些技術，是為了讓你理解它們如何運作， 以便防禦或合法使用。請對自己的行為負責。

1.6 「現代」的修正：你不該死記的舊知識

網路技術的書籍最大的風險，是把「歷史正確、但今日已過時」的觀念當成永恆真理。 以下是幾個本書會反覆強調、且與許多舊教材說法不同的當代觀點：

修正 1 · 對稱式 NAT 沒那麼罕見 許多老教材把對稱式 NAT（Symmetric NAT）描述為「邊緣案例」。 事實上，大量行動電信網路與公共 Wi-Fi 採用的就是對稱式 NAT。 在 2025 年，任何認真的 P2P 系統都必須假設「至少一方是對稱式 NAT」是常態。
修正 2 · UDP 不一定優於 TCP 傳統觀點認為 UDP hole punching 成功率遠高於 TCP。最新的大規模測量挑戰了這個說法—— 在 IPFS 網路 440 萬次穿越嘗試的研究中，QUIC 與 TCP 的穿越成功率在統計上沒有顯著差異， 關鍵在於精準的 RTT 同步（DCUtR），而非傳輸層本身。
修正 3 · IPv6 無法獨力解決問題 「等 IPv6 普及，這些都不是問題」是十年前的常見說法。 現實是：IPv6 的部署是部分的——一邊 IPv6、另一邊只有 IPv4 的場景仍是常態。 即使在 2026 年，IPv4/UDP hole punching 仍是大多數 P2P 系統的主路徑。
修正 4 · 中繼伺服器並非「失敗」 舊觀點視 TURN/relay 為「直連失敗時的不得已」。 現代系統（如 Tailscale 的 DERP）主動使用中繼來加速初始連線建立—— 先用 relay 立刻通訊，再背景升級為直連。中繼從「failure mode」變成「always-on infrastructure」。

1.8 章末提要

本章核心觀念
  • 「連線困難」的根源是 IPv4 位址枯竭迫使網路引入 NAT，破壞了端對端原則。
  • NAT 並非為了安全而存在，但它實際上變成了一道安全屏障——這是個歷史巧合。
  • 「NAT 阻擋」與「防火牆阻擋」是兩種不同問題，需要不同解法。
  • 所有穿越技術都是三條路線的組合：請 NAT 開洞、利用 NAT 規律、透過第三方中繼。
  • 許多舊教材的結論在 2025+ 已不成立——本書會反覆校正這些「部落知識」。

動腦時間

  1. 請描述至少三種「即使對方願意，你還是無法主動連到他電腦」的真實情境。
  2. 為什麼「使用 server-client 架構」雖然能繞過 NAT 問題，但對隱私與抗審查不利？
  3. 查詢你現在所處網路的對外 IP（可用 curl ifconfig.me），與你電腦的本地 IP 比較。 它們相同嗎？如果不同，差距告訴你什麼？
