CHAPTER 3 防火牆與中介設備

本章導讀

第 2 章告訴你 NAT 是無心阻擋連線；本章告訴你還有一群設備是有意阻擋連線—— 它們是防火牆、應用層閘道（ALG）、深度封包檢測（DPI）、企業 Proxy。理解這些設備的設計意圖 與行為，是規劃應用層隧道（第 9 章）與安全防護（第 17 章）的前提。 這章也會點出一個常被學生混淆的觀念： NAT 跟防火牆雖然常在同一個盒子裡，但它們解決的是兩個不同問題。

📚 學習目標

  • 能說明有狀態防火牆（stateful firewall）與封包過濾器（stateless filter）的本質差異。
  • 能解釋 ALG 為何曾經是必要設計，以及為何今日多數人想關掉它。
  • 能說明深度封包檢測（DPI）如何偵測加密流量背後的協議。
  • 能列舉企業環境中常見的中介設備，並理解它們對 P2P 的影響。
  • 能用「NAT 阻擋 vs 防火牆阻擋」二分法對任意連線失敗場景做初步診斷。

3.1 防火牆的本質：政策的執行者

防火牆與 NAT 表面上做同一件事——不放某些封包通過。但兩者的設計動機天差地別：

NAT 的視角

「我想讓內部裝置通訊，但因為位址不夠，必須做翻譯。翻譯不出來的就只好丟掉。」

→ 阻擋是副作用，可協作

防火牆的視角

「網路擁有者制定了一份政策，我的工作是執行它。 符合政策的放行，不符合的攔截、記錄、警報。」

→ 阻擋是主動意圖，需偽裝

這個哲學差異，決定了你寫穿越程式碼的策略——對 NAT 你是「請它幫忙」，對防火牆你是「不讓它發現」。

3.1.2 第一代：封包過濾器（Packet Filter）

最古老、最簡單的防火牆。它逐個檢查封包，根據五元組 (源IP, 源port, 目的IP, 目的port, 協議)對照規則表，決定放行或丟棄。 典型實作如早期 Cisco ACL、Linux ipchains、BSD ipfw 的 stateless 模式。

它的弱點是無狀態——不知道某個進來的封包是不是某個出去封包的「回應」。 為了讓 TCP 連線的回程封包能進來，工程師必須開放所有常見高位 port（> 1024）， 這幾乎等於沒設防。 

# 一個樸素的 packet filter 例子
iptables -A INPUT -p tcp --dport 22 -j ACCEPT       # 允許 SSH
iptables -A INPUT -p tcp --dport 80 -j ACCEPT       # 允許 HTTP
iptables -A INPUT -j DROP                           # 其餘全擋

# 問題：客戶端從 1024+ 隨機 port 主動發起 HTTP，
# 回程封包的目的 port 是 1024+，這條規則擋不住也允許不對。
# 必須加：
iptables -A INPUT -p tcp --sport 80 -j ACCEPT
# 但這意味著「任何源 port 80 的封包都進得來」——攻擊者可以偽裝。

3.1.3 第二代：有狀態防火牆（Stateful Firewall）

這是今天「防火牆」一詞的預設意義。它維護一張類似 NAT 的連線追蹤表（conntrack）， 記錄每一條合法連線的狀態（NEW、ESTABLISHED、RELATED）。

(如 FTP data) ESTABLISHED --> CLOSING: 收到 FIN RELATED --> ESTABLISHED: 子連線建立 CLOSING --> [*]: 連線關閉
或 timeout

# 有狀態防火牆的標準寫法
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -p tcp --dport 22 -j ACCEPT       # 允許新進 SSH
iptables -A INPUT -j DROP

# 解讀：
# - 如果是現有連線的回程封包 (ESTABLISHED) → 放行
# - 如果是相關連線 (RELATED，如 FTP 資料通道) → 放行  
# - 如果是<新連線>但目的是 SSH → 放行
# - 其餘新連線 → 丟棄

這個機制的副作用跟 NAT 高度相似——只放行內部主動發起連線的回應。 從 P2P 應用的角度看，它跟一個 Port-Restricted Cone NAT 沒什麼兩樣。 這就是為什麼第 6 章的 hole punching 技術同時對 NAT 與有狀態防火牆都管用。

NAT ≈ 防火牆？ 在家用場景，這個近似很成立——家用 NAT 路由器就是一個簡化版的 stateful firewall。 但在企業場景，差距就大了：企業防火牆會看到 Layer 7（HTTP method、SNI、URL）， 甚至會解密 TLS 流量（透過受信 CA 注入），做深度政策執行。家用 NAT 完全不會做這些事。

3.2 應用層閘道（ALG）：好心做壞事

ALG（Application-Level Gateway）是 NAT/防火牆中的一個小程式，專門理解某些應用層協議 並做必要的協助處理。最經典的例子是 FTP ALG。

3.2.1 為什麼有 ALG？以 FTP 為例

傳統 FTP 使用兩條 TCP 連線：控制連線（port 21）與資料連線。在 active mode 下， 客戶端在控制連線上送一個 PORT 192,168,1,42,7,196 命令，告訴伺服器 「請主動連到 192.168.1.42:2020」（port 計算是 7×256+196=2020）。

問題顯而易見：192.168.1.42 是私有位址，伺服器連不到！

FTP ALG 的工作就是偷看控制連線的內容，當看到 PORT 命令時：

  1. 把命令裡的私有 IP/port 改寫成 NAT 的對外 IP/port
  2. 在 NAT 表中預先建立一條「等待中」的條目，讓伺服器主動發起的資料連線能正確轉送回客戶端

改寫為對外 IP:Port
並預開 NAT 表項 N->>S: PORT 198,51,100,1,242,16 Note over S: 知道要連到 198.51.100.1:62000 S->>N: 主動連到 198.51.100.1:62000 Note over N: 查到預先建立的條目

3.2.2 ALG 的副作用

ALG 的設計初衷是好的，但它有兩個嚴重副作用：

副作用 1：對加密流量無效 ALG 必須讀應用層內容才能改寫。若協議走 TLS（如 FTPS），ALG 看到的是密文，毫無作用。 因此加密化的協議普及反而讓 ALG 失能。
副作用 2：對 SIP / VoIP 常常做錯 SIP 協議在訊息體中內嵌了 SDP（Session Description Protocol），裡面寫著「請對方送 RTP 到我這個 IP:port」。 SIP ALG 試圖改寫這些位址，但因為 SIP 訊息結構複雜（多次 INVITE/200 OK/ACK、不同 dialog）， 錯誤改寫的情況極為普遍。資深 VoIP 工程師有句口號： 「碰到 SIP 不通，先把路由器的 SIP ALG 關了再說」。

現代 P2P 系統的設計哲學，已經不再仰賴ALG。WebRTC、libp2p 都假設中間設備 完全不理解自己的協議，所有位址協商都透過 STUN/ICE 在應用內部處理。 從這個角度看，ALG 是個歷史包袱，正在逐步淡出。

3.3 深度封包檢測（DPI）：看穿你的流量

Stateful firewall 看到的是 IP/port，最多 TCP flags。DPI（Deep Packet Inspection）則更進一步—— 它檢查封包內容，識別應用層協議，甚至從加密流量中提取特徵。

3.3.1 DPI 的識別技術光譜

識別等級 技術 能看到什麼 對抗難度
L1 · Port 看 destination port 僅標準 port（443=HTTPS） 容易（換 port 即可）
L2 · Signature 找協議關鍵字（"GET /"、"BitTorrent protocol"） 明文協議的 fingerprint 中（用加密就破解）
L3 · Behavior 看流量模式（封包大小分布、時序、雙向比例） BT 種子、視訊串流、SSH 難（需要 obfuscation）
L4 · TLS Fingerprint 分析 TLS ClientHello（JA3/JA4） 用了哪個 TLS 函式庫、哪個應用 很難（需要 fingerprint 偽造）
L5 · ML 分類 機器學習分類器 幾乎所有流量類型 極難（需 adversarial 對抗）

3.3.2 為什麼 P2P 開發者需要關心 DPI？

即使你成功穿越了 NAT，封包仍可能被 DPI 認出是「未經授權的 P2P 流量」而被丟棄。 常見場景：

  • 學校 / 企業網路禁止 BitTorrent 等 P2P 應用，DPI 識別 BT handshake 後阻斷
  • 某些 ISP會限速 P2P 流量（traffic shaping），即使法律上不應該
  • 國家防火牆會封鎖 VPN/Tor，這些通常基於 DPI fingerprint

對抗 DPI 的技術通常稱為流量混淆（traffic obfuscation），第 9 章會專門處理。 常見方案如 obfs4、Shadowsocks、Cloak、meek（透過 CDN 域前置）等。

DPI 的軍備競賽 Tor 與某些國家防火牆之間有一場長達十年的軍備競賽。Tor 從明文 → 用 TLS → 用 obfs2/3/4 → 用 meek（偽裝成正常 HTTPS 到 CDN）→ 用 Snowflake（透過 WebRTC 借用普通使用者）。 每一次升級都是對 DPI 進步的回應。這個歷史告訴我們： 沒有永久的混淆，只有持續的演化。

3.4 中介設備生態

「Middleboxes」是 IETF 對所有位於兩個端點之間、會干預流量的設備的統稱。 RFC 3234（2002）列出了 22 種 middleboxes，今天這個清單只會更長。

3.4.1 你會在企業網路中遇到的中介設備

🛡️ NGFW (Next-Gen Firewall)

整合了傳統 stateful FW + DPI + IPS + AV + URL 過濾。 典型廠商：Palo Alto、Fortinet、Check Point。會看 TLS SNI、HTTP method 做政策。

🧱 IDS / IPS

入侵偵測 / 預防系統。基於規則 (Suricata、Snort) 或行為分析。 對 P2P 開發者影響：奇怪的 hole punching 流量可能被誤判為攻擊。

🌐 Forward Proxy

例如 Squid、Zscaler。所有 HTTP/HTTPS 出站流量被強制經過代理， 可能解密 TLS（用內建 CA）、可能拒絕非標準 CONNECT 目的地。

📡 Web Gateway / SWG

Secure Web Gateway，雲端版的 forward proxy。 透過 PAC 檔或 explicit proxy 強制端點走它，做 SaaS 流量管控。

🔐 TLS Inspector / Break-and-Inspect

解密所有出站 TLS 流量做檢查再重新加密。 要求端點安裝企業根憑證（這也是 cert pinning 在這類網路會壞掉的原因）。

📊 Network Tap / DLP

Data Loss Prevention。被動監聽流量，偵測機敏資料外洩。 通常不阻擋，但會記錄與警報。

⚖️ Load Balancer

L4 (TCP/UDP) 或 L7 (HTTP) LB。對 P2P 是透明的，但若 LB 改寫源 IP（SNAT mode）， 會造成你看到的對方 IP 跟實際不同。

🗜️ WAN Optimizer

跨資料中心的流量壓縮 / 去重設備（Riverbed 等）。 對加密流量幾乎失效，逐步淡出。

3.5 連線追蹤的限制

我們在 2.3 節介紹了 NAT 的 conntrack 表。防火牆也有自己的 conntrack 表， 且兩者通常共用實作（Linux netfilter 的 conntrack 模組）。這張表是有限的。

3.5.1 conntrack 表的容量問題

# 查看 Linux conntrack 容量
$ sysctl net.netfilter.nf_conntrack_max
net.netfilter.nf_conntrack_max = 262144

# 查看當前使用量
$ sysctl net.netfilter.nf_conntrack_count
net.netfilter.nf_conntrack_count = 14238

# 接近上限會發生什麼？
$ dmesg | grep conntrack
[1234567.89] nf_conntrack: table full, dropping packet

當 conntrack 表滿時，新連線會被丟棄——這在大型 NAT、爆量 DDoS、或者 某個程式產生大量短連線時都可能發生。對 P2P 系統而言，這意味著：

  • 節點不應同時開啟過多持久連線，否則會撐爆中間設備的 conntrack 表
  • BitTorrent 等 P2P 軟體曾因這個原因觸發家用路由器當機，廠商不得不加上連線數上限
  • 若連線突然全斷又無法重連，第一個要懷疑的就是中間某台設備的 conntrack 滿了

3.5.2 不對稱路由的詛咒

Stateful firewall 假設一條連線的進與出走同一台防火牆。在簡單拓樸下這成立， 但在多 ISP 連線、複雜雲端 VPC、SD-WAN 等場景，可能進出走不同路徑。 此時防火牆 A 看到 SYN，防火牆 B 看到 SYN-ACK，雙方都認為對方那條是異常的， 把 ACK 之後的封包全擋掉。

這在 P2P 場景特別容易出問題：兩個 peer 之間的 hole punching 涉及同時從兩端發包， 若中間設備是 stateful + 不對稱路由，幾乎必然失敗。

3.7 案例研究：三種真實網路環境

3.7.1 家用網路

[拓樸]
  Laptop ──→ 家用路由器 (NAT + Stateful FW) ──→ ISP ──→ Internet

[特徵]
  ・ 通常是 Port-Restricted Cone NAT
  ・ 出站幾乎不限
  ・ 入站無 port forward 即全擋
  ・ UPnP 通常開啟（第 4 章）
  ・ UDP timeout 約 60-180 秒

[P2P 難度]  ★★ 中等
[推薦策略]  STUN + UDP hole punching

3.7.2 行動網路（4G/5G）

[拓樸]
  Phone ──→ 基地台 ──→ ISP CGNAT ──→ Internet

[特徵]
  ・ 多半是 Symmetric NAT（且常常多層 CGNAT）
  ・ 公網 IPv4 可能 100 戶共用
  ・ UDP 出站可能被速率限制
  ・ 大部分電信業者已部署 IPv6（值得優先嘗試）
  ・ 切換基地台時 NAT 表可能被清

[P2P 難度]  ★★★★ 困難
[推薦策略]  IPv6 直連 → 失敗則 fall back to TURN

3.7.3 企業網路

[拓樸]
  Laptop → 802.1X 認證 → NGFW → SWG/Proxy → TLS Inspector → Internet

[特徵]
  ・ UDP 通常被嚴格限制（只放 DNS:53、QUIC:443）
  ・ HTTPS 可能被解密檢查
  ・ 非白名單域名可能被擋
  ・ STUN/TURN public servers 可能被歸類為「未授權通訊」
  ・ 部分企業強制 split tunnel 流量回 corporate

[P2P 難度]  ★★★★★ 極困難
[推薦策略]  TURN over TLS:443 + 自建中繼伺服器於企業核可域名下

3.8 章末提要

本章核心觀念
  • NAT 阻擋與防火牆阻擋是兩種本質不同的問題：前者可協作，後者需偽裝。
  • 有狀態防火牆（stateful firewall）對 P2P 的影響跟 Port-Restricted Cone NAT 類似——hole punching 對兩者都有效。
  • ALG 是好心做壞事的歷史包袱，現代 P2P 系統都假設它不存在或已關閉。
  • DPI 能識別加密流量背後的協議特徵（JA3/JA4、流量行為模式），對抗它需要混淆技術。
  • 企業網路的中介設備鏈遠比家用複雜，P2P 在此環境的成功率顯著下降。
  • Conntrack 表的容量與 timeout 是所有有狀態中介設備的軟肋；P2P 系統必須做 keep-alive 與連線數控制。
