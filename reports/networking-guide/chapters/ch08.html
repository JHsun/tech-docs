CHAPTER 8 ICE：把所有方法整合起來

本章導讀

前面四章我們各自處理了 UPnP、STUN、Hole Punching、TURN——這些是工具。 但「給工具」跟「會用」是兩件事。ICE（Interactive Connectivity Establishment） 是把所有工具整合成一套自動化決策框架的協議。它的核心問題是： 當有 N 條可能的路徑時，如何並行嘗試、量化評估、選最佳？ 本章從 RFC 8445 出發，詳述候選收集、優先序計算、連通性檢查、提名機制， 並介紹 Trickle ICE 等現代優化。讀完本章你會明白為何 WebRTC 不是「採用了」STUN/TURN——它是把這些編排起來。

📚 學習目標

  • 能定義 ICE 的「候選位址」並說出四種類型（host、srflx、prflx、relay）。
  • 能解釋 ICE 候選優先序（priority）的計算公式與設計目的。
  • 能描述完整的連通性檢查流程，包括 ordinary check 與 triggered check。
  • 能說明 Trickle ICE 相對 vanilla ICE 的時間優勢。
  • 能對 WebRTC 連線的 ICE 過程做基本除錯。

8.1 為什麼需要 ICE？

想像你正要建立一條 P2P 連線，你的端點可能有：

  • 本地 IPv4 地址 192.168.1.42（如果對方在同 LAN，這條最快）
  • 本地 IPv6 地址 2001:db8::1（如果雙方都有 IPv6，可能繞過 NAT）
  • 另一張網卡（雙網卡電腦、VPN tunnel）
  • 透過 STUN 探得的對外位址 198.51.100.1:62000
  • 透過 TURN 配的中繼位址 turn.example.com:50001

對方可能也有同樣多種候選。理論上是 N × M 條可能路徑，且不同路徑的延遲、可達性差很多。 你不會想序列嘗試（太慢），也不能全部同時開（資源浪費）。

ICE 的設計就是回答：怎麼並行嘗試這 N×M 條路徑、儘速找到能用的、選最佳那條。

8.2 候選位址的四種類型

① Host Candidate

端點本機網卡上實際綁定的位址（包含 IPv4、IPv6、VPN tunnel 介面等）。 透過 OS API 查詢取得。
延遲最低，若可用則優先。

② Server-Reflexive (srflx)

透過 STUN 伺服器探得的對外位址。
對 cone 類 NAT 通常與 host candidate 不同。
用於跨 NAT 的直連嘗試。

③ Peer-Reflexive (prflx)

連通性檢查過程中對方觀察到我的位址，之前沒列入候選—— 通常是因為 NAT 對不同遠端分配不同 port（symmetric NAT）。
動態發現，補充候選清單。

④ Relay Candidate

透過 TURN 配給我的中繼位址。
所有流量過 TURN 伺服器，延遲最高、最可靠。
最後保險。

8.3 候選優先序：怎麼決定先試哪條？

RFC 8445 明確定義了候選的 32-bit priority 計算公式：

priority = (2^24) × type_preference
         + (2^8)  × local_preference
         + (2^0)  × (256 - component_id)

其中：
  type_preference   :  host=126, prflx=110, srflx=100, relay=0
  local_preference  :  介面偏好（IPv6 通常高於 IPv4）
  component_id      :  RTP=1, RTCP=2

幾個觀察：

  • Type 是主要決定因素——host 永遠優先於 srflx，srflx 優先於 relay
  • Local preference 是次要差異，可區分多介面（如有線 vs Wi-Fi）
  • Component ID 用於 RTP/RTCP 配對，現代 WebRTC 用 mux 通常只有一個 component

8.3.1 Pair Priority

ICE 真正比較的不是單一候選，而是候選對（local, remote 配對）的優先級： 

pair_priority = 2^32 × min(G, D)
              + 2 × max(G, D)
              + (G > D ? 1 : 0)

G = 控制方（controlling agent）的優先級
D = 受控方（controlled agent）的優先級

這個公式設計成雙方計算結果一致——確保兩端對「該先試哪對」的判斷不會分歧。

8.4 連通性檢查（Connectivity Check）

收集完候選並交換給對方後，ICE 進入核心階段：連通性檢查。 雙方對每對候選做 STUN Binding Request，看哪些對能成功往返。

8.4.1 STUN 在 ICE 中的特殊用法

注意：這裡的 STUN 不是發給STUN 伺服器，而是發給 peer！ ICE 把 STUN 的 Binding Request 當作「連通性探測信號」， 並擴充了一些 attribute：

Attribute 用途
USERNAME身份標示，避免別人偽造 ICE 流量混入
MESSAGE-INTEGRITYHMAC 校驗，配合每對 ICE session 的密碼
PRIORITY本端對該候選的計算優先級
USE-CANDIDATE提名訊號：「就用這對！」（只 controlling agent 會送）
ICE-CONTROLLING / ICE-CONTROLLED角色宣告，避免雙方都想當 controlling

8.4.3 Triggered Check

ICE 連通性檢查不是只「我發、等對方回」這麼簡單。當你收到對方的 STUN Binding Request （也就是對方在檢查某對），你應該立即用同一對候選反向發一個 check—— 這就是「triggered check」。

Triggered check 為什麼重要？ Hole punching 的成敗繫於「同步」。當對方主動 punch 過來時， 我立即回擊，能大幅提升雙方 NAT 表同時就緒的機率。 triggered check 把 ICE 的成功率推高到接近理論上限。

8.5 ICE 的兩種角色：Controlling vs Controlled

ICE 規定每個 session 必有一方是 controlling agent（決策者）， 另一方是 controlled agent。職責分工：

Controlling Agent 的職責

  • 選定要用哪對候選
  • 送出 USE-CANDIDATE 提名訊號
  • 處理候選衝突（雙方都聲稱 controlling 時的 tie-break）

Controlled Agent 的職責

  • 對所有 check 都依規回應
  • 等待 controlling 的提名
  • 看到 USE-CANDIDATE 後啟用該對

在 WebRTC 中，發起 offer 的那方通常是 controlling。 在 libp2p 中，由 PeerID 大小決定。

8.6 Vanilla ICE vs Trickle ICE

8.6.1 Vanilla ICE 的問題

RFC 5245（ICE 初版，2010）定義的「正規」流程要求：candidate 必須全部收集完才能 offer。 這意味著： 

[t = 0ms]    開始收集 host candidates (~10ms)
[t = 10ms]   開始查 STUN (~30ms-200ms 視 RTT)
[t = 200ms]  開始 Allocate TURN (~50ms)
[t = 250ms]  ✓ 全部收集完，現在才能送出 offer
[t = 500ms]  signaling 來回後對方收到
[t = 750ms]  對方做完同樣流程
[t = 1000ms] 連通性檢查開始
[t = 1500ms] 連線建立

整個過程要 1.5 秒。對視訊通話而言，這是很糟的體驗。

8.6.2 Trickle ICE 的優化

Trickle ICE（RFC 8838，2021）允許邊收集邊送出—— 發現一個 candidate 就立刻通知對方，不必等全部收集完。

sequenceDiagram participant A as Alice participant S as Signaling participant B as Bob Note over A,B: T=0：開始 A->>S: SDP offer (僅 host candidates) S->>B: forward B->>S: SDP answer (僅 host candidates) S->>A: forward Note over A,B: T=50ms：可以開始用 host 對打了！ par 背景持續探得新候選 A->>S: trickle: srflx candidate S->>B: forward and B->>S: trickle: srflx candidate S->>A: forward end Note over A,B: 邊收集邊測試
找到任何一條可用就用 A->>S: trickle: relay candidate (最後) S->>B: forward Note over A,B: 連線可能早就建立
relay 變成備援
圖 8-3 ｜ Trickle ICE：邊收集邊送，大幅縮短建立時間

Trickle ICE 在實務上把連線建立時間從 1~2 秒降到~200ms， WebRTC 從 2017 年起預設使用，現在已是標準做法。

8.7 候選對狀態機

每對候選在連通性檢查中走過以下狀態：

stateDiagram-v2 [*] --> Frozen: 初始（等其他更高優先級先試） Frozen --> Waiting: 解凍（輪到此對） Waiting --> InProgress: 開始 STUN check InProgress --> Succeeded: ✓ 收到對方 success InProgress --> Failed: ✗ 超時或錯誤 Succeeded --> Nominated: controlling 提名（USE-CANDIDATE） Failed --> [*] Nominated --> [*]: 連線建立
圖 8-4 ｜ ICE 候選對的狀態轉換

Frozen 機制是為了避免一次發太多 STUN check 撐爆網路—— ICE 規範把候選對分到不同 foundation group，先解凍每組最高優先級的， 其他凍著等。這在頻寬受限或大量候選時特別重要。

8.8 ICE Restart：當網路變了

使用者從 Wi-Fi 切到 4G、VPN 連上、IP 換了——這些事件都會破壞既有的 ICE pair。 ICE 規範定義了 ICE Restart——丟棄現有候選、重新跑一次完整流程， 但保持應用層連線不重連（從應用視角是無縫切換）。 

// WebRTC 中觸發 ICE Restart
const offer = await pc.createOffer({ iceRestart: true });
await pc.setLocalDescription(offer);
// 重新走 signaling，產生新的 username/password
// 連通性檢查重跑，找到新路徑後切換

在現代行動應用中，ICE restart 是必要機制，否則使用者切換網路就會斷線。 Tailscale 的 NetworkMap 機制更進一步——主動偵測網路變化，預先觸發 restart。

8.9 ICE 在 WebRTC 中的整合

8.9.1 與 SDP 的關聯

WebRTC 用 SDP（Session Description Protocol）描述媒體會話，候選位址也包含在 SDP 中： 

v=0
o=- 4611732855696449382 2 IN IP4 127.0.0.1
s=-
t=0 0
a=group:BUNDLE 0
m=video 9 UDP/TLS/RTP/SAVPF 96
c=IN IP4 0.0.0.0
a=ice-ufrag:F7gI
a=ice-pwd:x9cml/YzichV2+XlhiMu8g
a=fingerprint:sha-256 5C:5C:...
a=setup:actpass
a=mid:0
a=candidate:1 1 udp 2122260223 192.168.1.42 54321 typ host
a=candidate:2 1 udp 1686052607 198.51.100.1 62000 typ srflx \
              raddr 192.168.1.42 rport 54321
a=candidate:3 1 udp 41819902 198.51.100.7 50001 typ relay \
              raddr 198.51.100.1 rport 62000

每行 a=candidate 對應一個 ICE candidate。其中：

  • foundation = 1, 2, 3（用於 freezing 分組）
  • component = 1（單一 RTP/RTCP mux）
  • 協議 udp
  • priority、IP、port、type
  • raddr / rport = 對應的「related address」（host for srflx, srflx for relay）

8.9.2 ICE-LITE 模式

伺服器（如 SFU、媒體閘道）通常有公網 IP，不在 NAT 後。 RFC 8445 定義了 ICE-LITE 模式—— 伺服器只列 host candidate、不主動發 connectivity check、不擔任 controlling， 只被動回應對方的 check。這簡化了伺服器實作。

8.10 一個現實 ICE 連線的時序剖析

從 Chrome 的 chrome://webrtc-internals 抓一條真實連線的時序：

[T+0ms]      onicecandidate: host  192.168.1.42:54321
[T+5ms]      send offer (with host only, trickle)
[T+150ms]    receive answer (with host only)
[T+155ms]    onicecandidate: srflx 198.51.100.1:62000
[T+155ms]    trickle send: srflx
[T+200ms]    onicecandidate (remote): srflx 203.0.113.99:55001
[T+205ms]    pair: host_A-host_B   STUN check fail (10.x vs 192.x 不同網段)
[T+220ms]    pair: srflx_A-srflx_B STUN check sent
[T+250ms]    pair: srflx_A-srflx_B STUN response received ← 連通!
[T+255ms]    USE-CANDIDATE sent (controlling agent)
[T+265ms]    iceConnectionState: connected
[T+450ms]    onicecandidate: relay turn:50001 (晚到，不影響)
[T+800ms]    iceConnectionState: completed (consent freshness 開始)

從觸發到 connected 不到 270ms——這就是 Trickle ICE 的威力。

8.11 ICE 的限制與替代方案

8.11.1 ICE 的設計局限

ICE 不是萬靈丹
  • 需要外部 signaling——ICE 規範不管「怎麼把候選交換給對方」，這是另外的問題
  • 對 symmetric NAT 仍然只有靠 TURN——直連那條注定失敗
  • 協議複雜——RFC 8445 約 100 頁，許多陷阱（freezing 順序、tie-break、redundant pruning）
  • 瀏覽器實作差異——Chrome / Firefox / Safari 各有怪癖，跨瀏覽器測試是必要的

8.11.2 替代方案：libp2p 的 DCUtR

第 13 章會詳述：libp2p 走了不同的路。它沒有完全拒絕 ICE，但它沒用完整的 ICE 框架。 在 IPFS/libp2p 網路裡，候選交換透過 Circuit Relay v2（不需要外部 signaling）， 打洞同步用 DCUtR（直接量 RTT、半 RTT 等待，比 ICE freezing 簡單）， 也不分 controlling/controlled。它的設計目標是「去中心化網路內的 ICE 替代」。

在 libp2p 規範中為什麼仍提到 STUN？ libp2p 的位址發現用 AutoNAT（不是 STUN），但瀏覽器平台是例外—— libp2p 在瀏覽器上跑時沒有控制權，必須用瀏覽器內建的 WebRTC stack， 而那個 stack 只支援 STUN 不支援 AutoNAT。所以在 browser-to-browser 場景，libp2p 仍依賴 STUN。

8.12 章末提要

本章核心觀念
  • ICE 是把所有穿越工具編排成自動化決策框架的協議。
  • 四種候選：host（本機）、srflx（STUN 探得）、prflx（檢查中發現）、relay（TURN 配的）。
  • 優先序公式：type 主導 + local preference + component；雙方計算結果必須一致。
  • 連通性檢查 = peer-to-peer 的 STUN Binding 配 USERNAME/INTEGRITY。
  • Triggered check 提升 hole punching 同步性，是成功率的關鍵。
  • Trickle ICE（RFC 8838）允許邊收集邊送，把建立時間從秒級壓到 200ms。
  • ICE Restart 機制支援網路切換時無縫重連。
  • ICE-LITE 簡化純伺服器端實作。
  • libp2p DCUtR 是 ICE 在去中心化場景的替代方案。

  5. 研究 ICE 的「freezing」機制，思考為什麼要這個複雜性，能否簡化掉。
