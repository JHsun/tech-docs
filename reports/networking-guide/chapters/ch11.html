CHAPTER 11 BitTorrent 與檔案分享

本章導讀

BitTorrent 是迄今最成功的 P2P 應用，也是許多現代設計（包括 IPFS）的精神祖父。 它由 Bram Cohen 在 2001 年發表，核心創新在於： 把單一檔案切成許多 piece，讓 peer 們以「tit-for-tat」（一報還一報）的方式互相交換， 既解決了單一伺服器的頻寬瓶頸，又透過經濟激勵讓參與者誠實貢獻。 本章從 .torrent 檔案結構出發，講到 piece selection 演算法、tracker 與 DHT 的職責分工， 最後介紹 µTP 這個為 P2P 設計的傳輸協議。

📚 學習目標

  • 能解析 .torrent 檔案的 bencode 結構與 info_hash 的角色。
  • 能描述 tracker、DHT、PEX 三種 peer discovery 機制及其差異。
  • 能解釋 tit-for-tat 與 optimistic unchoking 的經濟設計。
  • 能描述 rarest-first piece selection 為何是核心策略。
  • 能說明 magnet link 如何徹底去中心化 BitTorrent。
  • 能解釋 µTP 為何不直接用 TCP。

11.1 BitTorrent 的核心洞見

2001 年 Bram Cohen 觀察到傳統檔案分享有兩個問題：

  1. HTTP/FTP 下載：所有人都從伺服器下載，伺服器頻寬就是瓶頸；越熱門越塞。
  2. Napster / Gnutella 模式：peer 一邊下載一邊上傳，但缺乏激勵——大量「leecher」只下載不分享。

BitTorrent 的解決方案綜合三個關鍵設計：

① 切 Piece

檔案切成等大小的 piece（典型 256 KB ~ 4 MB），每個 piece 有 SHA-1 hash。 peer 一拿到任何完整 piece 就能立刻分享，不必等整個檔案下完。

② Tit-for-Tat

「你給我，我才給你」——peer 優先把上傳頻寬給正在給自己上傳的對方。 純 leecher 自然吃不到頻寬。

③ Rarest-First

peer 優先下載「所有對等節點中持有率最低」的 piece， 避免熱門 piece 過度集中、稀有 piece 消失。

這三個設計組合起來，讓 BitTorrent 成為越多人下載越快的網路—— 與傳統 client-server 完全相反的擴展曲線。

11.2 .torrent 檔案結構

11.2.1 Bencode 編碼

BitTorrent 使用一種叫 bencode 的簡單序列化格式。四種類型： 

字串：<長度>:<內容>       例：4:spam = "spam"
整數：i<數字>e            例：i42e   = 42
list：l<item1>...e        例：l4:spam4:eggse = ["spam","eggs"]
dict：d<key1><val1>...e   例：d3:cow3:moo4:spam4:eggse
                              = {cow:"moo", spam:"eggs"}
                              （key 必須是字串且按字典序排列）

bencode 設計目標：簡單、可預測、保證唯一序列化（同樣資料只有一種編碼）。 這個唯一性保證對 info_hash 的計算極為關鍵。

11.2.2 .torrent 檔的結構

{
  "announce": "http://tracker.example.com:6969/announce",
  "announce-list": [["http://tracker1..."], ["http://tracker2..."]],
  "creation date": 1714000000,
  "comment": "Linux ISO",
  "created by": "Transmission/3.0",

  "info": {
    "name": "ubuntu-24.04.iso",
    "piece length": 262144,            ← 256 KB
    "pieces": "<每 20 byte 是一個 SHA-1, 串連起來>",
    "length": 4500000000               ← 單檔模式
  }
}

# 多檔案模式時，info 裡用 "files" 陣列代替 "length"

11.2.3 info_hash：torrent 的唯一身份

info_hash = SHA-1(bencode(info_dict))。 這個 hash 是該 torrent 在整個 BitTorrent 世界的唯一身份—— tracker 用它識別 swarm、DHT 用它當 key、peer 之間握手要互報它。

為什麼 bencode 的「唯一性」很重要 若兩個程式對同一個 dict 序列化結果不同，產生的 info_hash 就不同—— 會被當成不同的 torrent。bencode 規定 dict key 必須按字典序， 數字不能有前導 0 等規則，正是為了確保確定性編碼。 這是「用簡單的協議達成複雜的一致性」的優美案例。

11.3.1 Tracker（中心式）

最古老的方式。Tracker 是一台公網伺服器，每個 swarm（同一 torrent 的所有 peer）的所有成員定期向它報到。 Client 透過 HTTP GET 請求拿到 peer 清單： 

GET /announce?info_hash=...&peer_id=...&port=6881
       &uploaded=0&downloaded=0&left=4500000000
       &event=started

Response (bencoded):
{
  "interval": 1800,              ← 多久後再 announce
  "peers": <binary peer list>     ← 6 byte/peer (4 IP + 2 port)
}

優點：簡單、low overhead。
缺點：單點失效——tracker 掛了整個 swarm 就斷；也容易被法律攻擊。

11.3.2 DHT（去中心式）

BEP 5 定義了 BitTorrent 的 DHT 機制（基於 Kademlia，第 10 章詳述）。 在 DHT 中：

  • 每個 peer 節點有 160-bit Node ID（隨機產生）
  • info_hash 直接是 DHT 中的 key
  • 對該 key 「儲存」的不是檔案內容，而是一個 peer 清單（IP, port）
  • 新 peer 對 info_hash 做 get_peers 查詢，拿到 swarm 成員清單
  • 同時做 announce_peer 把自己加入該 swarm

DHT 讓 BitTorrent 從「需要 tracker」變成「不需要 tracker」—— 只要有 info_hash 與一個 bootstrap node，就能找到所有 peer。 這是 magnet link 的基礎。

11.3.3 PEX（Peer Exchange）

最簡單的機制——peer 之間互相交換各自認識的 peer 清單。 我認識 A、B、C；你認識 D、E、F；我們交換後我多認識了 D、E、F。 PEX 是一種 epidemic 傳播，讓網路不必依賴 tracker/DHT 也能擴散 peer 資訊。

11.4 Magnet Link：徹底去中心化

傳統做法是先下載 .torrent 檔再開始。Magnet link 進一步省略這步—— 只要一個 URI 就能開始： 

magnet:?xt=urn:btih:c12fe1c06bba254a9dc9f519b335aa7c1367a88a
       &dn=ubuntu-24.04.iso
       &tr=http://tracker.example.com:6969/announce

關鍵欄位：

  • xt（exact topic）= urn:btih:<info_hash>——torrent 身份
  • dn（display name）= 檔名提示
  • tr（tracker）= 可選的 tracker 提示

11.4.1 從 Magnet 開始的完整流程

這個流程展示了 BitTorrent 的優雅：整個系統不需要 .torrent 檔的儲存伺服器。 info_hash 既是身份、也是驗證——只要 info_hash 正確、其他都能從網路重建。

11.5 Peer Wire Protocol

11.5.1 Handshake

兩個 peer 建立 TCP 連線後，先交換 68 byte 的 handshake：

[1 byte ] pstrlen = 19
[19 byte] pstr    = "BitTorrent protocol"
[8 byte ] reserved (主要是 extension flags)
[20 byte] info_hash
[20 byte] peer_id

雙方比較 info_hash 一致才繼續。

11.5.2 訊息類型

ID名稱意義
0choke「我暫時不上傳給你」
1unchoke「我願意上傳給你」
2interested「我對你的 piece 有興趣」
3not interested「對你那邊沒新東西要的」
4have「我剛拿到 piece N」
5bitfield「我目前持有哪些 piece 的位元圖」
6request「請給我 piece I 的 [offset, len] 區段」
7piece「這是你要的資料」
8cancel「取消之前的 request」
20extendedBEP 10 擴展訊息（PEX、metadata 等）

11.5.3 Choking 演算法（Tit-for-Tat 的實作）

每個 peer 對其他 peer 維護兩個狀態：

  • am_choking：我是否在 choke 對方
  • peer_interested：對方是否表示有興趣

典型 client（如 mainline、libtorrent）的 unchoke 策略：

每 10 秒（regular unchoke round）：
  從所有 peer 中，挑出「過去 20 秒給我上傳速率最高」的前 4 個
  → 對它們 unchoke
  → 對其他全部 choke

每 30 秒（optimistic unchoke）：
  從 choked 但 interested 的 peer 中，隨機挑一個
  → 額外 unchoke 它
  → 給新 peer 一個機會證明自己

純 leecher 拿不到 unchoke → 速度 0
慷慨上傳的 peer 互相發現 → 速度爆發
Optimistic Unchoke 的賽局意涵 Tit-for-Tat 嚴格執行會造成「新 peer 永遠進不來」—— 沒人給你頻寬你也沒得回報，鎖死。Optimistic unchoke 是主動的善意： 「我隨機選一個沒互相給的 peer 試試看」。這個小小的隨機性讓網路充滿活力， 也讓誠實的新 peer 能逐步建立關係。Cohen 的這個設計是賽局理論在 P2P 的經典範例。

11.6 Piece Selection：Rarest-First

11.6.1 為什麼不能隨機？

若 peer 隨機選 piece 下，會發生：

  • 熱門 piece 被反覆下載——浪費
  • 稀有 piece 沒人想要——可能因 seeder 離線而永遠丟失
  • 後加入的 peer 很難找到能反過來上傳的 piece

11.6.2 Rarest-First 演算法

每次需要選下一個 piece 下載時：
  1. 看所有 peer 的 bitfield，統計每個 piece 在 swarm 中的副本數
  2. 排除自己已有的
  3. 從持有率最低的開始挑——副本越少越優先

例外：First Phase（剛開始下載）
  ・ 此時自己沒任何 piece，無法跟人交換
  ・ 改用「random first piece」策略：隨機選 1 個下，趕快有東西能交換
  ・ 等下到 4 個 piece，切換到 rarest-first

例外：Endgame Mode（接近完成）
  ・ 剩最後幾個 piece 時，同時對多個 peer 發 request
  ・ 收到任何一個就立即 cancel 其他
  ・ 避免被慢 peer 拖住

11.7 µTP：為 P2P 訂做的傳輸層

11.7.1 為什麼 TCP 不夠

BitTorrent 早期跑在 TCP 上，但發現幾個問題：

TCP 對 P2P 的不友善
  • 佔滿家用上行頻寬：TCP 的 AIMD 會吃滿頻寬，導致使用者其他應用（瀏覽、視訊通話）變慢
  • 對多連線不友善：BitTorrent 同時開十幾個連線，每個 TCP 都搶頻寬
  • NAT 穿越困難：TCP hole punching 比 UDP 難（第 6 章）
  • 對 ISP 不友善：ISP 的 CGNAT 對大量 TCP 連線消耗 conntrack 表

11.7.2 µTP 的核心創新

µTP（Micro Transport Protocol，BEP 29）由 BitTorrent Inc. 在 2009 年推出， 跑在 UDP 上但提供類 TCP 的可靠串流。它的特殊之處是LEDBAT 擁塞控制： 

傳統 TCP：佔滿可用頻寬，遇到丟包才退讓
LEDBAT：偵測網路排隊延遲的增加，在達到頻寬上限前就退讓

直觀解釋：
  TCP = 「我用全力，撞到牆才退」
  LEDBAT = 「我感覺路上開始塞，提前讓位」

效果：
  ・ 同一連線內 BT 流量自動讓位給瀏覽、視訊通話
  ・ 使用者體感「BT 還在跑但網路不卡」
  ・ ISP 對 BT 的不滿大幅下降

µTP 也順便享受了 UDP 的所有好處：易於 NAT 穿越、無 head-of-line blocking、可疊在 TLS 上。 現代 BT client（µTorrent、qBittorrent、Transmission）預設啟用 µTP，TCP 只當 fallback。

11.8 BitTorrent 與 NAT 穿越的關係

BitTorrent 是真正壓力測試 NAT 穿越技術的實戰場。對 BT 而言：

  • NAT 後的 peer 能下載但無法上傳到外面的 NAT 後 peer——形成 leecher 階級
  • µTorrent 等主流 client 早就支援 UPnP / NAT-PMP（第 4 章）
  • µTP 的 UDP 化讓 hole punching（第 6 章）對它更可行
  • BEP 55 定義了 BitTorrent 的 hole punching 擴展，讓 peer 之間互助打洞

事實上，本書的許多現代 P2P 設計都站在 BitTorrent 巨人的肩膀上—— IPFS 的 bitswap、libp2p 的 transport 抽象，都受到 BT 經驗深遠影響。

11.9 衍生與後續

WebTorrent

把 BT 帶到瀏覽器：用 WebRTC DataChannel 取代 TCP/µTP，info_hash 保持相容。 讓網頁能直接成為 BT peer。

IPFS

「做對的 BitTorrent」：內容定址（CID 取代 info_hash）、單一全域 swarm 而非每 torrent 獨立、 bitswap 取代 peer wire protocol。第 13 章呼應。

BitTorrent v2

2020 年發表的協議升級：用 SHA-256 取代 SHA-1（後者已不安全）、 Merkle tree 取代 piece hash list、更小的 metadata。普及緩慢。

Filecoin / Storj

把 P2P 儲存加上經濟激勵的區塊鏈衍生物。 與 BT 不同的是它們提供「持久儲存」而非「暫存交換」。

11.10 章末提要

本章核心觀念
  • BitTorrent 三大設計：piece 切片、tit-for-tat 激勵、rarest-first 選擇。
  • .torrent 用 bencode 編碼；info_hash 是 swarm 唯一身份。
  • 三種 peer discovery：Tracker（中心）、DHT（去中心）、PEX（互傳）。
  • Magnet link + DHT + metadata extension = 完全去中心化的 BT。
  • Peer wire protocol 透過 choke/unchoke 實現 tit-for-tat；optimistic unchoke 給新 peer 機會。
  • Rarest-first 兼顧效率與資料保存，配合 first-phase / endgame 例外。
  • µTP 用 LEDBAT 擁塞控制，讓 BT 流量自動讓位給互動應用。
  • BT 是 NAT 穿越技術的實戰場，許多技術經驗被現代 P2P 系統繼承。
