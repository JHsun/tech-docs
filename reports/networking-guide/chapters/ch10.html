CHAPTER 10 P2P 拓樸與 Kademlia DHT

本章導讀

Part II 教你怎麼讓兩個 peer 直連，本章開始問一個更大的問題： 當網路中有上百萬個 peer 時，怎麼找到你想找的那一個？ 這就是「peer discovery」與「內容路由」問題。我們會比較結構化（DHT）與非結構化（flooding）兩種覆蓋網路設計， 然後深入Kademlia DHT——這是 BitTorrent、IPFS、Ethereum、libp2p 共同的基礎設施， 也是工程上最優雅的分散式雜湊表設計之一。

10.1 P2P 拓樸的兩條道路

P2P 系統的核心設計選擇之一：節點該怎麼組織？節點與節點間的「鄰居關係」決定了：

  • 查詢的路徑長度（要跳幾次才能找到資料）
  • 查詢的頻寬成本（每次查詢產生多少訊息）
  • 網路的抗變動性（節點頻繁進出時是否仍可用）
  • 網路的抗攻擊性（惡意節點能造成多少破壞）

🌪 非結構化（Unstructured）

節點隨機建立連線，查詢透過洪泛（flooding）或隨機漫步傳播。
例：Gnutella、Freenet 早期版本

優點：實作簡單、極抗變動
缺點：查詢成本高、不保證找得到

🏛 結構化（Structured）

節點與資料都用同一種 ID 空間編址，路由演算法保證 O(log N) 步抵達目標。
例：Chord、Kademlia、Pastry、Tapestry

優點：可保證找到、效率高
缺點：節點頻繁進出時維護成本高

10.2 從 Gnutella 到 Chord：歷史脈絡

10.2.1 Gnutella：洪泛的代價

2000 年代初的 Gnutella 是純粹的非結構化 P2P：每個節點維護幾個鄰居， 查詢時用 TTL（Time-To-Live，預設 7）洪泛廣播給所有鄰居。理論上， TTL=7、平均度 5 的網路，一次查詢產生 5⁷ ≈ 78,125 個訊息。網路一旦規模化就崩潰。

10.2.2 Chord：第一個學術 DHT

2001 年 MIT 的 Chord 論文開啟了結構化 P2P 時代。它把節點與 key 都映射到同一個圓環上， 每個節點維護一張「finger table」指向 2^i 距離外的節點， 查詢透過 finger 跳躍能在 O(log N) 步抵達。

Chord 雖優雅，但實務上並不流行——它的 finger table 維護要求太嚴格， 節點 churn（頻繁進出）會造成大量更新。Kademlia 後來居上的原因之一， 是它的設計天生對 churn 友善。

10.3 Kademlia：工程之美

Kademlia 由 Petar Maymounkov 與 David Mazières 在 2002 年發表。它的設計之美在於： 用一個簡單的數學概念（XOR 距離），優雅地解決了路由、儲存、容錯三個問題。 今天 BitTorrent、IPFS、Ethereum、libp2p 的 DHT 全部基於 Kademlia 變體。

10.3.1 ID 空間與 XOR 距離

Kademlia 使用 160 bit（SHA-1 大小）或 256 bit（現代版本） 的 ID 空間。每個節點隨機產生一個 ID；每筆資料的 key 也是同一空間中的點。

「節點 A 與節點 B 的距離」定義為 distance(A, B) = A XOR B。 這個簡單定義有幾個漂亮性質：

對稱性

d(A,B) = d(B,A)
A 看 B 與 B 看 A 是同樣距離——這讓 Chord 的「順時針」非對稱問題消失。

三角不等式

d(A,C) ≤ d(A,B) ⊕ d(B,C)
保證透過中間節點不會繞遠路。

單位元素

d(A,A) = 0
自己跟自己距離為零。

唯一性

對任一 ID X 與距離 d，恰好存在一個 ID Y 滿足 d(X,Y)=d。
這對 routing 演算法極為關鍵。

10.3.2 一個直觀理解

把 ID 想成 binary tree 的葉子，XOR 距離就是「樹上兩葉子的最近共同祖先到根的距離」：

假設 ID 是 4 bit，畫成 binary tree：

              root
            /      \
          0         1
        / \        / \
       00  01    10  11
      /|  |\    /|  |\
   0000 ... 1111

節點 ID = 0011, key = 0110
  XOR = 0101 = 5

節點 0011 ↔ 0110 在 tree 上的關係：
  共同祖先 = root (因為最高位就不同)
  距離 5 反映了「從 root 走到任一邊」的概念

節點 ID = 0011 vs 0010
  XOR = 0001 = 1
  非常近！只差最後一個 bit
  共同祖先非常深

10.3.3 K-Bucket：路由表

Kademlia 不為每個節點都記，也不只記少數幾個——它的路由表是分層的「k-bucket」結構。

對於一個 160-bit ID 的節點 N，它維護 160 個 buckets。Bucket i 存放距離 N 在 [2^i, 2^(i+1)) 範圍內的節點清單， 每個 bucket 最多存 k 個節點（k 通常為 20）。

幾個關鍵觀察：

  • 近的節點 bucket 多但容量小（高 i 的 bucket 對應遠距離，但範圍大）
  • **整個網路只看到 O(log N) 個 buckets 是非空的**（其他 bucket 的節點根本不存在或極少）
  • 網路規模 N=10⁶ 時，每個節點只需追蹤約 400 個聯絡人就能路由到任何其他節點

10.3.4 K-Bucket 更新策略：Least-Recently-Seen

當節點 N 收到任何訊息（請求、回應、heartbeat）時，會更新對應 bucket： 

收到來自節點 X 的訊息時：

1. 計算 X 與我的距離 d，找出對應 bucket B_i (i = floor(log2(d)))
2. 若 X 已在 B_i 中：
     將 X 移到 bucket 尾端（標記為「最近聯絡」）
3. 若 X 不在 B_i 中：
     若 B_i 有空位 (size < k)：
        把 X 加到尾端
     否則：
        Ping bucket 頂端那個節點 Y（最久未聯絡的）
        若 Y 回應：
           把 Y 移到尾端，丟棄 X（保守策略）
        若 Y 不回應：
           移除 Y，把 X 放尾端
為什麼偏好「老節點」？ Kademlia 的設計者觀察到：線上越久的節點，繼續線上的機率越高 （這是經典的「長尾分布」現象）。新節點可能是攻擊者、可能是短暫連線者， 老節點則證明了自己「能存活」。所以 LRS（Least-Recently-Seen）策略 自然抗 Sybil 攻擊——攻擊者要佔據 bucket 必須持續在線很久才能擠掉老節點。

10.4 Kademlia 的四個 RPC

整個 Kademlia 協議只需要四個 RPC（remote procedure call）：

RPC 作用 回應
PING 確認節點是否在線 echo
STORE(key, value) 請求接收方儲存一筆資料 ack
FIND_NODE(target) 請接收方回報它知道離 target 最近的 k 個節點 k 個 (ID, IP, port)
FIND_VALUE(key) 類似 FIND_NODE，但若接收方有 key 對應的值，直接回傳 value 或 k 個聯絡人

10.4.1 Iterative Lookup：核心演算法

整個 Kademlia 的搜尋邏輯就是一個漂亮的「迭代逼近」過程： 

查詢 target_id（FIND_NODE 或 FIND_VALUE 的核心）：

shortlist = 我的 k-bucket 中離 target 最近的 α 個節點 (α 通常 = 3)
queried = {}
closest_so_far = max distance

迴圈：
  從 shortlist 中選 α 個尚未 query 過的節點，並行送 FIND_NODE
  收集所有回應的 (ID, IP, port)
  把這些新節點加入 shortlist（並排序：最近的在前）

  current_closest = shortlist 中最近的節點
  若 current_closest >= closest_so_far（沒有更近了）：
    終止迴圈
  否則：
    closest_so_far = current_closest

最終：對 shortlist 前 k 個節點再做一次同步 FIND_NODE，得到「最近的 k 個」

10.4.2 為什麼要 α 個並行？

並行度 α 是 Kademlia 的精明設計：

  • α=1 時是純序列查詢——若某個被查的節點不回應或慢，整體會卡住
  • α=3 是論文建議值——三個並行請求只要任何一個回，就能繼續推進
  • α 太大會浪費頻寬與目標節點資源

這個機制讓 Kademlia 對 churn 與不可靠網路天生友善—— 個別節點掛掉不影響整體查詢進度。

10.5 STORE：怎麼放資料？

要儲存 (key, value)：

  1. 對 key 做 FIND_NODE（找出離 key 最近的 k 個節點）
  2. 對這 k 個節點全部送 STORE(key, value)

資料因此被冗餘儲存在 k 份副本上——只要不超過 k-1 個節點同時掛掉，資料不會丟失。 典型 k=20 提供了極高的容錯。

10.5.1 Republishing：對抗自然衰減

節點會進出，原本儲存資料的 k 個節點可能慢慢變少。Kademlia 規定：

  • 原始 publisher 每 24 小時 republish 一次（即重複整個 STORE 流程）
  • 儲存節點每 1 小時 republish 給其他應該存的節點（抓到新加入網路的近鄰）
  • 資料的過期時間預設 24 小時，沒有 republish 就消失

這是一個典型的「持續維護而非一次性放置」的設計——非常 P2P 思維。

10.6 Bootstrap：怎麼進入網路？

新節點如何加入 Kademlia 網路？這是個「雞生蛋」問題——你需要先認識某些節點才能查詢。

新節點 N 加入網路：

1. 從硬編碼的 bootstrap nodes 清單中拿到至少一個聯絡人 B
2. 將 B 放入自己的 k-bucket
3. 對 N 自己的 ID 做一次 FIND_NODE（這個查詢會觸發網路中許多節點認識 N）
4. 對隨機分布的 ID（每個 bucket 對應的範圍）做 FIND_NODE，
   填滿自己的 k-buckets

Bootstrap 是 P2P 系統的關鍵安全議題——bootstrap nodes 是中心化的弱點。 IPFS 預設有 8 個 bootstrap nodes，BitTorrent DHT 有 6 個。若這些節點被攻擊或封鎖，新節點無法加入。

10.7 Kademlia 的安全性

10.7.1 Sybil 攻擊

什麼是 Sybil 攻擊？ 攻擊者建立大量假節點（每個有自己的 ID），透過數量優勢操縱網路。 在 Kademlia 中可以：
  • 佔據受害者 k-bucket 的多數位置 → 接管路由
  • 圍繞特定 key 部署假節點 → 攔截或污染儲存
  • 送錯誤的 FIND_NODE 回應 → 誤導查詢路徑

純 Kademlia 對 Sybil 攻擊抵抗有限。常見緩解：

  • S/Kademlia：要求 Node ID 必須是 hash(public_key)，且 hash 必須有 PoW (proof-of-work)，提高生成假身份的成本
  • 多路徑查詢：透過多條 disjoint paths 查同一個 key，比較結果判斷異常
  • 身份權重：基於聲譽或長期觀察給節點權重

10.7.2 Eclipse 攻擊

Eclipse 攻擊更精準——攻擊者包圍一個特定受害者，讓它的所有 k-bucket 都被假節點填滿， 受害者看到的「網路」全是攻擊者控制的。這在區塊鏈領域是嚴重風險（受害者可能被「分裂」到假鏈上）。

防禦：bootstrap 時連到多個獨立來源、IP 多樣性、定期重新探索網路。

10.8 不同系統的 Kademlia 變體

系統 ID 大小 k 值 特殊修改
BitTorrent (Mainline) 160 bit (info_hash) 8 儲存的是「peer list」而非任意 value；BEP 5
IPFS / libp2p 256 bit (multihash) 20 內容定址；provider records 而非實際資料
Ethereum (devp2p) 512 bit (secp256k1 pub key) 16 節點發現，無資料儲存功能

10.9 Kademlia + NAT：兩個世界相遇

本書的核心議題終於與 DHT 相遇——NAT 後的節點可不可以加入 DHT？

純 Kademlia 假設所有節點都能直接被連線——但這在現代網路是錯誤前提。約 60-70% 的家用節點在 NAT 後。 解決方案：

方案 A · 只當客戶端

NAT 後的節點查詢但不存資料、不路由。 它消費網路服務但不貢獻——這是 BitTorrent client 預設行為。

→ 公平性問題：太多 leecher

方案 B · Hole Punching

透過 STUN + UDP hole punching（第 6 章）讓 NAT 後節點能被連到。 BitTorrent µTorrent 的 NAT-PMP/UPnP 也屬此類。

→ 對 cone NAT 有效

方案 C · Circuit Relay

libp2p 路線：NAT 後節點透過 relay 提供地址，DHT 中的位址記錄即 relay 路徑。 其他節點透過 relay 連到後再升級為直連（DCUtR）。

→ 通用解，第 13 章詳述

方案 D · 雙網模型

區分 full-DHT 節點（公網）與 light-DHT 節點（NAT 後）。 Light 節點只使用 DHT 不維護它。

→ Ethereum、Bitcoin 採用

10.10 一個迷你 Kademlia 實作骨架

用 Python 簡化版骨架展示核心邏輯：

import hashlib, asyncio, random
from collections import OrderedDict

ID_BITS = 160
K = 20

def node_id(material: bytes) -> int:
    return int.from_bytes(hashlib.sha1(material).digest(), 'big')

def distance(a: int, b: int) -> int:
    return a ^ b

def bucket_index(self_id: int, other_id: int) -> int:
    d = distance(self_id, other_id)
    return d.bit_length() - 1 if d > 0 else 0

class KBucket:
    def __init__(self, k=K):
        self.contacts = OrderedDict()  # id -> (ip, port)
        self.k = k

    def update(self, contact_id, addr):
        if contact_id in self.contacts:
            self.contacts.move_to_end(contact_id)
        elif len(self.contacts) < self.k:
            self.contacts[contact_id] = addr
        else:
            # bucket 滿了：應 ping 最舊的，這裡簡化省略
            pass

    def closest_to(self, target_id, n):
        sorted_contacts = sorted(self.contacts.items(),
                                 key=lambda kv: distance(kv[0], target_id))
        return sorted_contacts[:n]

class KademliaNode:
    def __init__(self, my_id):
        self.id = my_id
        self.buckets = [KBucket() for _ in range(ID_BITS)]
        self.storage = {}  # key -> value (本機儲存)

    def update(self, contact_id, addr):
        if contact_id == self.id:
            return
        idx = bucket_index(self.id, contact_id)
        self.buckets[idx].update(contact_id, addr)

    def closest(self, target_id, n=K):
        all_contacts = []
        for b in self.buckets:
            all_contacts.extend(b.contacts.items())
        all_contacts.sort(key=lambda kv: distance(kv[0], target_id))
        return all_contacts[:n]

    async def find_node(self, target_id, alpha=3):
        """迭代查詢核心邏輯"""
        shortlist = self.closest(target_id, K)
        queried = set()
        closest_dist = float('inf')

        while True:
            to_query = [c for c in shortlist[:alpha]
                        if c[0] not in queried][:alpha]
            if not to_query:
                break

            for cid, _ in to_query:
                queried.add(cid)

            # 並行送 FIND_NODE（此處省略網路 I/O）
            results = await asyncio.gather(*[
                self.send_find_node(cid, target_id) for cid, _ in to_query
            ])

            for result_list in results:
                for cid, addr in result_list:
                    self.update(cid, addr)
                    if (cid, addr) not in shortlist:
                        shortlist.append((cid, addr))

            shortlist.sort(key=lambda kv: distance(kv[0], target_id))
            shortlist = shortlist[:K]

            new_closest = distance(shortlist[0][0], target_id)
            if new_closest >= closest_dist:
                break
            closest_dist = new_closest

        return shortlist

    async def send_find_node(self, contact_id, target_id):
        """實際的 RPC 呼叫——此處留空"""
        return []  # 應傳回該節點知道的最近 k 個

完整實作還需加上：UDP 訊息序列化（KRPC / protobuf）、bucket split 邏輯、 republishing 排程、Sybil 緩解。但這個骨架已能展示核心結構。

10.11 章末提要

本章核心觀念
  • P2P 拓樸有結構化（DHT）與非結構化（flooding）兩條路；前者主導現代系統。
  • Kademlia 用 XOR 距離度量，把節點與資料統一在同一 ID 空間。
  • K-bucket 是分層結構：每節點僅需 ~3000 個聯絡人即可路由百萬節點網路。
  • Least-Recently-Seen 更新策略天生抗 Sybil。
  • 四個 RPC（PING / STORE / FIND_NODE / FIND_VALUE）涵蓋所有功能。
  • Iterative lookup + α 並行 = O(log N) 步收斂、容錯佳。
  • STORE 冗餘到 k 個節點，配合 republishing 對抗自然衰減。
  • Bootstrap 是中心化的弱點，需多源備份。
  • NAT 後節點融入 DHT 需配合本書前面的穿越技術。

動腦時間

  1. 為什麼 Kademlia 用 XOR 而不用「環形距離」（如 Chord）？XOR 的對稱性帶來什麼工程好處？
  2. 計算：在 N=10⁶ 節點的 Kademlia 網路中，一次 FIND_NODE 的期望訊息數（含並行 α=3）是多少？ 與 Chord 比較。
  3. K-bucket 的 LRS 策略偏好老節點。請設計一個攻擊：若你能讓某個老節點下線，你能做什麼？
  4. BitTorrent 的 DHT 用 k=8，IPFS 用 k=20。為什麼差距這麼大？分析這個權衡。
  5. 研究 S/Kademlia 的 disjoint path lookup 機制，理解它如何抵禦 Eclipse 攻擊。
