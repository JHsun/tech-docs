CHAPTER 7 中繼穿越：TURN 與 Circuit Relay

本章導讀

當 hole punching 失敗（最常見於 symmetric NAT 對 symmetric NAT）， 我們需要一條「退路」——讓所有流量透過第三方公網主機中繼。 本章介紹兩種設計哲學的中繼方案：IETF 的 TURN（集中式、規範化）與 libp2p 的 Circuit Relay v2（去中心化、資源約束）。 讀完本章，你會理解為什麼現代系統把中繼從「失敗時的最後手段」轉為「總是在線的基礎設施」—— Tailscale 的 DERP 是這個轉變的標竿案例。

📚 學習目標

  • 能說明 TURN 與 STUN 在訊息結構上的繼承關係。
  • 能描述 TURN 的 Allocate / CreatePermission / Send / Channel 四個關鍵操作。
  • 能評估 TURN 的成本（頻寬、延遲）與部署考量。
  • 能說明 Circuit Relay v2 的 reservation 機制與資源約束哲學。
  • 能對比集中式（TURN）與去中心化（Circuit Relay）兩種設計的取捨。
  • 能理解 Tailscale DERP 的「always-on relay」設計範式。

7.1 為什麼需要中繼？

Hole punching 對 cone 類 NAT 高效，但對 symmetric NAT vs symmetric NAT 場景 幾乎一定失敗。考慮到行動電信網路、CGNAT、企業安全閘道大量採用 symmetric NAT， 在現實 P2P 系統中，有相當比例的連線無法靠 punching 建立。 此時必須有一條「退路」——所有流量透過一台能被雙方主動連到的公網主機轉送。

中繼的代價是：頻寬（所有流量都過一台主機）、延遲（多了一跳）、 單點（中繼掛了就斷線）。但它的普世性讓它成為 P2P 系統的最後保險。

7.2 TURN：STUN 的延伸

TURN（Traversal Using Relays around NAT，RFC 5766/8656）是 IETF 為 WebRTC、 SIP/VoIP 等場景設計的標準中繼協議。它的訊息格式直接繼承自 STUN——共享 magic cookie、 TLV attribute 結構、20 byte header——這是工程上的優雅選擇， 讓 STUN 客戶端能輕鬆擴展為 TURN 客戶端。

7.2.1 TURN 的核心概念

Relayed Transport Address

TURN 伺服器在自己的公網位址上，幫客戶端配一個專屬 port， 這個位址叫 relayed transport address，是客戶端對外的「替身位址」。

Server-Reflexive Address

跟 STUN 一樣，TURN 也順便告訴客戶端「從我這看你是這個位址」， 這是 server-reflexive address，可給 ICE 嘗試直連。

Permission

客戶端必須先告訴 TURN「允許這個 peer 透過我中繼來找我」， 才能接收該 peer 的封包。這是 TURN 防濫用的關鍵機制。

Allocation Lifetime

每個 TURN 分配（一個 relayed address）有租期，預設 10 分鐘。 客戶端必須定期 refresh 否則自動釋放。

7.2.3 Send Indication vs Channel

TURN 提供兩種轉送資料的機制：

面向 Send / Data Indication Channel Mechanism
每個訊息開銷 ~36 byte STUN header + attributes 4 byte ChannelData header
需要先建立 無（每包獨立） 需先 ChannelBind
適用場景 低頻控制訊息 音視訊串流、高頻資料
peer 識別 每包都帶 XOR-PEER-ADDRESS 用 channel number 對應

Channel 機制是 TURN 為了高頻資料設計的優化——把 peer 位址壓縮成一個 16 bit 的 channel number， 封包額外開銷從幾十 byte 降到 4 byte。WebRTC 視訊串流幾乎都用 channel。

7.2.4 認證：Long-Term Credential

TURN 不像 STUN 可以匿名使用——中繼意味著消耗伺服器頻寬，必須嚴格認證， 否則任何人都可拿你的 TURN 當 amplifier 攻擊或匿名代理。 

Long-term credential 認證流程：

1. Client 首次發 Allocate 不帶認證
2. Server 回 401 Unauthorized，帶 NONCE 與 REALM
3. Client 計算 HMAC-SHA1(message, key)
   其中 key = MD5(username + ":" + realm + ":" + password)
4. Client 重發，附 USERNAME、REALM、NONCE、MESSAGE-INTEGRITY
5. Server 用同樣方式計算驗證
6. Server 同意分配，後續訊息都帶 MESSAGE-INTEGRITY
短期憑證的好實踐 生產環境不建議使用靜態 username/password。WebRTC 廠商常用「短期憑證」模式： 後端服務器用一個共享密鑰（HMAC key），動態產生限時的 TURN 憑證 （username = "expiry:userid"、password = HMAC(key, username)）。 這樣即使憑證外洩也只在短時間內可用。

7.2.5 TURN over TLS / DTLS

在企業網路中，UDP 常被嚴格限制。TURN 規範因此支援多種傳輸：

  • TURN over UDP（預設，port 3478）— 效率最高
  • TURN over TCP（port 3478）— 當 UDP 被擋時的備援
  • TURN over TLS（port 5349）— 加密傳輸，可偽裝成 HTTPS 流量
  • TURN over DTLS（port 5349）— UDP 上的 TLS

企業環境最有效的選擇是 TURN over TLS on port 443—— 讓中繼流量看起來跟 HTTPS 沒兩樣，幾乎能穿越所有防火牆。 這是許多 WebRTC 廠商生產環境的預設配置。

7.3 部署 TURN：以 coturn 為例

coturn 是業界事實標準的開源 STUN/TURN 伺服器， Google Hangouts、Discord 早期、Jitsi 都使用過它。

7.3.1 最小可用設定

# /etc/turnserver.conf

listening-port=3478
tls-listening-port=5349

# 公網 IP（NAT 後的雲機要寫對外 IP）
external-ip=YOUR_PUBLIC_IP/YOUR_PRIVATE_IP

# Realm（任意，建議用域名）
realm=turn.example.com

# 認證模式
lt-cred-mech                # long-term credentials
user=alice:my_secret_pass   # 簡單測試用
# 生產環境建議用 use-auth-secret 配合動態 token

# 強烈建議
no-tcp-relay                # 禁用 TCP 中繼（防 amplification）
no-multicast-peers          # 禁止對 multicast 中繼
no-loopback-peers           # 禁止對 loopback 中繼
denied-peer-ip=10.0.0.0-10.255.255.255   # 禁止對私網中繼
denied-peer-ip=192.168.0.0-192.168.255.255

# TLS 憑證（建議用 Let's Encrypt）
cert=/etc/letsencrypt/live/turn.example.com/fullchain.pem
pkey=/etc/letsencrypt/live/turn.example.com/privkey.pem

# 日誌
syslog
log-file=/var/log/turnserver.log
verbose

7.3.2 客戶端使用（WebRTC iceServers）

const pc = new RTCPeerConnection({
  iceServers: [
    {
      urls: [
        'stun:turn.example.com:3478',
        'turn:turn.example.com:3478?transport=udp',
        'turn:turn.example.com:3478?transport=tcp',
        'turns:turn.example.com:5349?transport=tcp'
      ],
      username: 'alice',
      credential: 'my_secret_pass'
    }
  ]
});

7.3.3 容量規劃

TURN 頻寬估算
  • 一通 720p 視訊通話：上下行各約 1.5 Mbps，雙向中繼 = 3 Mbps
  • 一台 1 Gbps 頻寬的 VPS：理論上撐 ~330 通同時 720p 視訊
  • 實際因 overhead 與 burst 應折半：~150 通是穩健估計
  • 若目標是「直連 80%、TURN 20%」（合理的健康比例），1 Gbps VPS 約能服務 750 通整體連線

Medium 上一篇 2026 年的 WebRTC 實務文章建議：監控 TURN 使用率， 健康的 P2P 應用應控制在 20~30%；超過 40% 表示網路異常或需檢討策略。 這個指標值得寫進你的監控儀表板。

7.4 libp2p Circuit Relay v2：去中心化的中繼

libp2p 採取了與 TURN 截然不同的設計哲學。它的 Circuit Relay v2 協議 讓網路中任何公網節點都能擔任 relay， 不需要架專屬基礎設施。為了避免被濫用，它引入了精細的資源約束機制。

7.4.1 設計哲學的對比

面向 TURN（IETF） Circuit Relay v2（libp2p）
誰當中繼專門部署的 TURN 伺服器網路中任何公網節點
認證username/password 或短期 token密碼學身份（PeerID）
資源控制由業者選擇授權誰用協議層內建 reservation 機制
計費業者承擔頻寬成本分散到全網，無中心成本
對 hole punching 的關係fallback only必經之路（DCUtR 用 relay 協調）
適合場景商業 WebRTC、VoIP 服務IPFS、Filecoin、去中心化網路

7.4.2 Reservation 機制

Circuit Relay v2 的關鍵創新是 reservation—— 想被當作「可中繼到達」的節點，必須先主動向某個 relay 申請預留：

7.4.3 資源限制：每個 reservation 的約束

Relay v2 規範允許 relay 對每個 reservation 強制執行以下限制（由 relay 決定具體值）：

  • 時間限制：reservation 過期時間（典型 1 小時）
  • 位元組限制：透過此 relay 中繼的總流量（典型 100 KB ~ 1 MB）
  • 連線數限制：同時透過此 relay 的活躍連線數

這些限制設計的目的——讓每個公網節點都能無痛當 relay 而不被資源耗盡—— 正是 IPFS 上每個節點都能被找到的關鍵。一旦透過 relay 短暫通訊建立， 就會立刻嘗試 DCUtR 升級為直連（第 6 章），釋放 relay 資源。

設計範式的轉變 TURN 的世界觀：「中繼是稀缺資源，由專人營運」。
Circuit Relay v2 的世界觀：「中繼是社會公共財，每個人貢獻一點點」。
這個差異折射了兩個社群的根本立場：商業 WebRTC vs 去中心化網路。

7.5 Tailscale DERP：「Always-On Relay」範式

Tailscale 的 DERP（Designated Encrypted Relay for Packets）採取了第三條路： 把中繼從「失敗時的後備」變成「總是在線的基礎設施」。

7.5.1 DERP 的角色定位

Tailscale 部落格直白地寫：「每一條連線在技術上都從 DERP 開始， 之後若可能升級為直連」。這個設計改變了 P2P 的啟動曲線：

sequenceDiagram participant A as Node A participant D as DERP Relay participant B as Node B Note over A,B: T = 0ms：使用者觸發連線 A->>D: 透過 DERP 送出資料 (立即) D->>B: 轉送 B->>D: 回應 D->>A: 轉送 Note over A,B: 使用者已經能用了！ par 背景並行 A-->>B: NAT traversal 嘗試 (UDP) B-->>A: NAT traversal 嘗試 Note over A,B: 失敗或 latency 高時繼續走 DERP end Note over A,B: T = 200ms：直連成功 A->>B: 直接通訊（旁路 DERP） B->>A: 直接通訊 Note over A,B: 流量無痛切換到直連
體感無延遲
圖 7-4 ｜ Tailscale 的 always-on relay 範式：先連通，後升級

這個設計的好處：使用者體感是「瞬間連通」，不必等 ICE 嘗試所有候選。 Tailscale 的內部數據顯示——超過 90% 的連線最終會升級為直連， DERP 只在初始時被使用，或在無法直連的網路條件下作為長期通道。

7.5.2 DERP 的特殊設計

HTTPS-friendly

DERP 跑在 port 443 上的 HTTPS，封裝協議讓它幾乎能穿越所有企業防火牆。 對防火牆而言，DERP 連線跟瀏覽 GitHub 沒兩樣。

End-to-End 加密

DERP 看不到內容——所有流量在端點之間用 WireGuard 加密， DERP 只是傳遞密文。這讓「使用第三方 relay」的隱私風險可以接受。

全球節點與 anycast

Tailscale 在全球部署多個 DERP 節點，客戶端透過延遲測試自動選最近的。 這壓低了即使透過 relay 的 RTT。

升級為直連是常態

連線建立後立即背景嘗試 NAT 穿越，成功則無痛切換。 使用者感受不到「relay → direct」的切換時刻。

7.7 何時用何種：選型指引

如果你正在做……
  • WebRTC 應用（視訊會議、線上教育）→ 用 TURN（成熟、整合度高）
  • 去中心化應用（檔案分享、區塊鏈節點通訊）→ 用 Circuit Relay v2（無需自架）
  • Mesh VPN / 內部辦公（公司內網跨地點）→ 用 always-on relay 範式（如自架 DERP）
  • 嚴企業環境（必經 proxy、TLS inspection）→ TURN over TLS:443 是最後保險

7.8 中繼的安全考量

7.8.1 防止 Amplification 攻擊

TURN 曾被研究發現可被當 amplifier——攻擊者偽造受害者 IP 為 source， 讓 TURN 把放大的 reply 流量打向受害者。RFC 8656 強化了這方面：

  • 強制要求認證（不接受匿名 Allocate）
  • 禁用對 private IP 的 relay（防止內網掃描）
  • 速率限制 per-user / per-IP
  • 禁止 relayed-address 是組播或廣播

7.8.2 中繼隱私

中繼伺服器看得到什麼？
  • 看得到：兩端的對外 IP、流量大小、時序
  • 看得到：訊息密文長度（可能洩漏應用類型——視訊？文字？）
  • 看不到：訊息內容（前提是雙方有 E2E 加密，如 WebRTC 的 DTLS-SRTP、libp2p 的 Noise、Tailscale 的 WireGuard）
在敏感場景（記者、社運人士），即使中繼看不到內容，知道誰跟誰通訊就已是相當的洩漏。 自架中繼或使用支援匿名的中繼（如 Tor）是進一步選擇。

7.9 章末提要

本章核心觀念
  • 中繼是 P2P 系統面對 symmetric NAT、嚴格防火牆時的最終保險。
  • TURN 繼承 STUN 的訊息格式，提供 Allocate / Permission / Send / Channel 機制。
  • Channel 機制把開銷從 36 byte 壓到 4 byte，是視訊串流必備優化。
  • TURN 必須認證；生產環境用短期動態憑證更安全。
  • TURN over TLS:443 是穿越企業防火牆的最後手段。
  • libp2p Circuit Relay v2 把中繼能力分散到所有公網節點，配合 reservation 限額避免濫用。
  • Tailscale DERP 開創「always-on relay」範式：先連通、後升級為直連。
  • 中繼掌握流量元資料；E2E 加密保護內容但不保護「誰在跟誰通訊」這個事實。
