簡介與特性 Introduction

Kafka Streams 是純 Java Client 函式庫，用於建構串流處理應用。不需外部叢集（非 Flink/Storm），以普通 Java 應用部署。

輕量嵌入

嵌入任何 Java 應用，無外部依賴（僅 Kafka）

Exactly-Once

Input Commit + State 更新 + Output 寫入，原子完成

逐筆處理

One-record-at-a-time，毫秒級延遲

Event-Time Windowing

支援亂序處理與 Grace Period

容錯 State Store

本地 RocksDB 狀態 + Changelog Topic 自動備份

雙層 API

高階 Streams DSL + 低階 Processor API

Stream-Table 二元性 Stream-Table Duality

概念KStreamKTableGlobalKTable
語義INSERT（每筆是獨立事件）UPSERT（同 Key 覆蓋）UPSERT（全副本）
資料範圍所有記錄每個 Key 的最新值全部 Partition 的最新值
底層Record StreamChangelog StreamChangelog（全副本）
Join 需求需 Co-partitioning需 Co-partitioning不需
💡 二元性Stream 聚合後成為 Table（groupByKey().count()）。Table 轉回 Stream（table.toStream()）。Kafka 以 Log Compaction 實現此二元性。
03

Processor Topology 處理拓撲

Topology 是一個 DAG（有向無環圖）：節點 = Stream Processor，邊 = Stream。

Source Processor

從 Kafka Topic 消費，無上游。進入點。

Stream Processor

接收輸入 Record → 轉換 → 產出零或多筆 Record 到下游

Sink Processor

將 Record 寫入 Kafka Topic，無下游。輸出點。

Topology 可自動分裂為多個 Sub-Topology（無連接的子圖）— 各自獨立平行化。

04

五種時間概念 Time Concepts

概念說明
Event Time事件在來源端發生的時間（嵌入 Record Timestamp）
Processing TimeRecord 被應用處理的當下時間
Ingestion TimeRecord 被存入 Kafka Broker Partition 的時間
Stream Time資料驅動的時間，僅在新 Record 到來時推進
Wall-Clock Time實際掛鐘時間，用於 WALL_CLOCK_TIME Punctuator

透過 TimestampExtractor 介面控制。預設使用嵌入的 Timestamp（Event Time）。

05

四種 Window 類型 Windowing

類型大小重疊建立方式
Tumbling（翻滾）固定❌TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5))
Hopping（跳躍）固定✅TimeWindows.ofSizeWithNoGrace(5m).advanceBy(1m)
Sliding（滑動）固定✅ 對齊資料SlidingWindows.ofTimeDifferenceAndGrace(10m, 30m)
Session（會話）可變❌ 合併SessionWindows.ofInactivityGapAndGrace(30m, 5m)
💡 Grace Period控制等待亂序 Record 的時間。Grace 過後的遲到 Record 會被丟棄。設 WithNoGrace 表示不容忍任何亂序。

Join 組合矩陣 Joins

左右InnerLeftOuter需 Window需 Co-partition
KStreamKStream✅✅✅✅✅
KStreamKTable✅✅❌❌✅
KStreamGlobalKTable✅✅❌❌❌
KTableKTable（Equi）✅✅✅❌✅

處理保證 Processing Guarantees

保證設定說明
At-least-once（預設）processing.guarantee=at_least_once可能重複處理，效能最佳
Exactly-once v2processing.guarantee=exactly_once_v2Input Offset + State + Output 原子提交（Broker ≥ 2.5）

4.0 起移除舊版 exactly_once，僅保留 exactly_once_v2。

08

架構：Task、Thread、State Architecture

Stream Task

每個 Input Partition 對應一個 Stream Task。Task 是固定的平行化單位，Partition → Task 映射不變。最大平行度 = 最多 Input Partition 數量。

Threading 模型

num.stream.threads 控制每個實例的處理執行緒數。每個 Thread 獨立執行一或多個 Task — 無共享狀態、無協調開銷。

2.8+ 支援動態增減：streams.addStreamThread() / streams.removeStreamThread()

Local State Store

Persistent（RocksDB）

預設。資料存磁碟，重啟快速恢復。適合大狀態。

In-Memory

速度最快但重啟需完整重建。適合小狀態。

每個 State Store 自動建立 Changelog Topic（Log-Compacted）。故障重啟時透過重播 Changelog 還原狀態。

Standby Replicas

num.standby.replicas 在其他實例維護 State Store 的影子副本。Task 遷移時優先分配到已有最新 Standby 的實例，最小化恢復時間。搭配 rack.aware.assignment.tags 跨 Rack 分佈。

應用程式狀態

CREATED → REBALANCING → RUNNING → PENDING_SHUTDOWN → NOT_RUNNING。另有 PENDING_ERROR、ERROR。
