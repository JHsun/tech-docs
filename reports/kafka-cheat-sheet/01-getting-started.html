Kafka 是什麼 Introduction

Apache Kafka 是一個分散式事件串流平台（Distributed Event Streaming Platform），結合三大能力：

發佈 / 訂閱 Publish & Subscribe

向 Topic 寫入或讀取事件串流，支援與外部系統持續匯入匯出

持久儲存 Durable Storage

以可靠、容錯方式將事件串流儲存任意時間長度

串流處理 Stream Processing

即時或回溯式處理事件串流（轉換、聚合、Join、Windowing）

架構：Server 與 Client

Kafka 由 Server（以 Cluster 形式運行，跨多個資料中心 / 雲端區域）與 Client（分散式應用程式與微服務）組成，透過高效能 TCP 協定通訊。

Broker

組成儲存層的 Server 節點。任何 Server 故障時，其他節點自動接管，不遺失資料。

Controller（KRaft 模式）

使用 Raft 協定管理叢集 Metadata，取代 ZooKeeper。4.0 起為唯一模式。

Kafka Connect

內建的資料整合框架，提供可重複使用的 Source / Sink Connector。

核心概念與術語 Core Concepts

術語說明
Event
Record / Message		 最基本資料單位，代表「某件事發生了」。包含 key、value、timestamp 及可選 headers。
範例：Key="Alice", Value="付款 $200 給 Bob"
Producer 發佈（寫入）Event 到 Topic 的 Client 應用程式
Consumer 訂閱（讀取並處理）Topic 中 Event 的 Client 應用程式。
Producer 與 Consumer 完全解耦，互不感知。
Topic Event 的組織與持久儲存單位（類比檔案系統中的資料夾）。
① 支援多 Producer / 多 Consumer；
② Event 消費後不刪除，保留期可設定；
③ 效能與資料量無關（O(1) 查找）。
Partition Topic 被切分為多個 Partition，分散在不同 Broker 上。
① 相同 Key 的 Event 寫入同一 Partition；
② 同 Partition 內保證嚴格順序。
Replication 每個 Topic-Partition 可跨 Broker 複製。生產環境常設 replication.factor=3。
Consumer Group 傳統機制：Partition 獨佔分配給 Group 中的個別 Consumer，並行處理。
Share Group 4.2 新功能 新的 Group 類型（KIP-932）：Consumer 不獨佔 Partition，支援逐筆確認（per-record ACK）與投遞計數。適合佇列式、無序處理場景。
💡 Partition 排序保證
Kafka 只保證 Partition 內的順序，不保證跨 Partition 的全域順序。若需要全域順序，可將 Topic 設為單一 Partition（犧牲平行度）。

七大使用案例 Use Cases

① 訊息佇列 Messaging

取代 ActiveMQ / RabbitMQ。更好的吞吐量、內建 Partitioning、Replication 與容錯。

② 網站活動追蹤 Activity Tracking

Kafka 最初的用途。使用者行為（瀏覽、搜尋、操作）發佈到 Topic，支援即時分析與離線倉儲。流量極高。

③ 指標收集 Metrics

從分散式應用聚合統計資料，產生集中式的營運資料饋送，用於監控。

④ 日誌聚合 Log Aggregation

取代 Scribe / Flume。將日誌資料抽象為訊息串流，更低延遲、更強持久性（Replication）。

⑤ 串流處理 Stream Processing

多階段處理管道：原始資料 → 聚合 / 豐化 / 轉換 → 新 Topic。可搭配 Kafka Streams、Flink、Storm。

⑥ 事件溯源 Event Sourcing

將狀態變更記錄為時間排序的 Event 序列。Kafka 支援超大規模 Log 儲存，是極佳的 Event Store。

⑦ 提交日誌 Commit Log

作為分散式系統的外部 Commit Log，協助節點間資料複製與故障重新同步。搭配 Log Compaction 使用。

Quick Start 指令集 Quick Start

前置需求
需安裝 Java 17+。Kafka 4.2 預設使用 KRaft 模式（無需 ZooKeeper）。

Step 1：下載與解壓

bash
$ tar -xzf kafka_2.13-4.2.0.tgz
$ cd kafka_2.13-4.2.0

Step 2：啟動 Kafka（三種方式）

方式 A — 本機 KRaft 模式（推薦）

bash
# 產生 Cluster UUID
$ KAFKA_CLUSTER_ID="$(bin/kafka-storage.sh random-uuid)"

# 格式化 Log 目錄
$ bin/kafka-storage.sh format --standalone -t $KAFKA_CLUSTER_ID -c config/server.properties

# 啟動 Kafka Server
$ bin/kafka-server-start.sh config/server.properties

方式 B — Docker（JVM）

bash
$ docker run -p 9092:9092 apache/kafka:4.2.0

方式 C — Docker（GraalVM Native，實驗性）

bash
$ docker run -p 9092:9092 apache/kafka-native:4.2.0

Step 3：建立 Topic

bash
$ bin/kafka-topics.sh --create --topic quickstart-events --bootstrap-server localhost:9092

# 查看 Topic 資訊
$ bin/kafka-topics.sh --describe --topic quickstart-events --bootstrap-server localhost:9092

Step 4：發送訊息（Producer）

bash
$ bin/kafka-console-producer.sh --topic quickstart-events --bootstrap-server localhost:9092
> 這是第一筆 Event
> 這是第二筆 Event
# Ctrl-C 結束

Step 5：消費訊息（Consumer）

bash
$ bin/kafka-console-consumer.sh --topic quickstart-events --from-beginning --bootstrap-server localhost:9092
✅ 關鍵觀念
Event 被持久儲存，可被任意數量的 Consumer 重複讀取任意次數。

Kafka Connect 快速範例 Connect Quick Start

設定 Plugin Path

bash
$ echo "plugin.path=libs/connect-file-4.2.0.jar" >> config/connect-standalone.properties

建立測試資料 & 啟動 Connector

bash
# 建立來源檔
$ echo -e "foo\nbar" > test.txt

# 啟動 Standalone 模式（Source + Sink）
$ bin/connect-standalone.sh config/connect-standalone.properties \
    config/connect-file-source.properties \
    config/connect-file-sink.properties

關鍵設定檔

connect-file-source.properties

connector.class=FileStreamSource
file=test.txt
topic=connect-test

connect-file-sink.properties

connector.class=FileStreamSink
file=test.sink.txt
topics=connect-test

驗證 Pipeline

bash
# 查看 Sink 輸出
$ more test.sink.txt

# 追加資料，自動同步到 Sink 與 Consumer
$ echo "Another line" >> test.txt

Kafka Streams 快速範例 WordCount Demo

建立 Input / Output Topic

bash
$ bin/kafka-topics.sh --create --topic streams-plaintext-input --bootstrap-server localhost:9092

$ bin/kafka-topics.sh --create --topic streams-wordcount-output \
    --config cleanup.policy=compact --bootstrap-server localhost:9092

執行 WordCount 應用

bash
$ bin/kafka-run-class.sh org.apache.kafka.streams.examples.wordcount.WordCountDemo

核心程式碼邏輯

Java（概念）
textLines                                    // KStream<String, String>
  .flatMapValues(v -> split(v))              // 拆分為單字
  .groupBy((key, word) -> word)              // 按單字分組
  .count()                                   // 計數 → KTable
  .toStream()                                // 轉回 Stream
  .to("streams-wordcount-output")            // 寫入輸出 Topic

驗證輸出

bash
$ bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 \
    --topic streams-wordcount-output --from-beginning \
    --property print.key=true \
    --property value.deserializer=org.apache.kafka.common.serialization.LongDeserializer

KRaft vs ZooKeeper 架構轉型

KRaft（Kafka Raft） 是 Kafka 內建的共識協定，將 Control Plane 完全整合進 Kafka 本身，消除對 ZooKeeper 的依賴。

演進時間軸

3.3：KRaft 正式可用
→
3.4：ZooKeeper 標記棄用
→
3.5–3.9：Bridge Release（遷移期）
→
4.0：ZooKeeper 完全移除

Process Roles 設定

server.properties
# 僅作為 Broker
process.roles=broker

# 僅作為 Controller
process.roles=controller

# 合併模式（僅供開發）
process.roles=broker,controller
⚠️ 生產環境
生產環境務必分離 Broker 與 Controller 角色。合併模式僅適用於開發測試。Controller 建議部署 3 或 5 個（公式：2N+1 可容忍 N 個故障）。

Controller 設定範例

server.properties（Controller 節點）
process.roles=controller
node.id=1
listeners=CONTROLLER://controller1.example.com:9093
controller.quorum.bootstrap.servers=controller1:9093,controller2:9093,controller3:9093
controller.listener.names=CONTROLLER

靜態 vs 動態 Quorum

特性靜態 Quorum動態 Quorum
設定方式 controller.quorum.voters controller.quorum.bootstrap.servers
最低版本 KRaft 3.3+ KRaft 3.9+（kraft.version=1）
動態增減 Controller ❌ 不支援 ✅ 支援

動態 Quorum 管理指令

bash
# 升級為動態 Quorum
$ bin/kafka-features.sh --bootstrap-server localhost:9092 upgrade --feature kraft.version=1

# 新增 Controller
$ bin/kafka-metadata-quorum.sh --bootstrap-server localhost:9092 add-controller

# 移除 Controller
$ bin/kafka-metadata-quorum.sh --bootstrap-server localhost:9092 remove-controller \
    --controller-id <id> --controller-directory-id <dir-id>

ZooKeeper → KRaft 遷移路徑

① 使用 3.9 Bridge Release
→
② 部署 KRaft Controller Quorum
→
③ 啟用遷移模式
→
④ 滾動重啟 Broker
→
⑤ 退役 ZooKeeper
→
⑥ 升級至 4.0+
遷移模式設定（3.9 Controller）
zookeeper.metadata.migration.enable=true

除錯工具

bash
# 查看 Quorum 狀態
$ bin/kafka-metadata-quorum.sh --bootstrap-server localhost:9092 describe --status

# 傾印 Metadata Log
$ bin/kafka-dump-log.sh --cluster-metadata-decoder \
    --files metadata_log_dir/__cluster_metadata-0/00000000000000000000.log

# Metadata Shell（互動式瀏覽）
$ bin/kafka-metadata-shell.sh --snapshot <snapshot-file>
>> ls /
brokers  local  metadataQuorum  topicIds  topics

升級指南 Upgrading

升級至 4.2 的路徑

支援從 3.3.x ~ 4.1.x 直接滾動升級。步驟：逐一停止 Broker → 更新程式碼 → 重啟，全部完成後執行：

bash
$ bin/kafka-features.sh --bootstrap-server localhost:9092 upgrade --release-version 4.2
⚠️ 4.0 分界線：ZooKeeper 叢集必須先遷移
Kafka 4.0 僅支援 KRaft 模式。仍在 ZooKeeper 上的叢集必須先透過 Bridge Release（3.5–3.9）遷移到 KRaft，然後才能升級到 4.0+。

4.0 重大變更摘要

項目變更內容
元資料管理inter.broker.protocol.version 移除 → 改用 metadata.version
Java 版本Server / Connect 最低 Java 17；Client / Streams 最低 Java 11
ScalaScala 2.12 支援移除
日誌框架Log4j → Log4j2
MirrorMaker原版 MM1 移除（使用 MM2）
CLI所有工具的 --zookeeper 參數移除
linger.ms預設值 0 → 5
segment.bytes最小值 14B → 1 MB

4.2 新功能亮點

功能說明KIP
NEW Share Groups正式可用。佇列式消費，逐筆 ACK，投遞計數KIP-932
NEW Streams Rebalance ProtocolGA。需設定 group.coordinator.rebalance.protocolsKIP-1071
NEW Streams Dead Letter Queue串流處理的失敗訊息自動路由KIP-1034
NEW Java 25 支援Server、Client、Streams、Connect 均支援—
NEW Controller 動態加入controller.quorum.auto.join.enableKIP-853
⚠️ 降級限制
包含 Metadata 變更的版本不支援叢集 Metadata 降級。一旦使用新的 Consumer Rebalance Protocol（KIP-848），叢集只能降級至 3.4.1 或更新。

環境變數設定規則

💡 命名規則
所有 Broker Config 均可透過 KAFKA_ 前綴的環境變數設定。
點號（.）替換為底線（_），例如 num.partitions → KAFKA_NUM_PARTITIONS。
注意：只要透過環境變數覆蓋任何設定，所有預設值都不會套用 — 必須明確指定所有必要設定。

Docker Compose — 單節點（合併模式）

docker-compose.yml
services:
  broker:
    image: apache/kafka:latest
    container_name: broker
    ports:
      - "9092:9092"
    environment:
      KAFKA_NODE_ID: 1
      KAFKA_PROCESS_ROLES: broker,controller
      KAFKA_LISTENERS: PLAINTEXT://localhost:9092,CONTROLLER://localhost:9093
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092
      KAFKA_CONTROLLER_LISTENER_NAMES: CONTROLLER
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT
      KAFKA_CONTROLLER_QUORUM_VOTERS: 1@localhost:9093
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1
      KAFKA_TRANSACTION_STATE_LOG_REPLICATION_FACTOR: 1
      KAFKA_TRANSACTION_STATE_LOG_MIN_ISR: 1
      KAFKA_GROUP_INITIAL_REBALANCE_DELAY_MS: 0
      KAFKA_NUM_PARTITIONS: 3

多節點部署要點

內部 Listener

使用容器名稱作為 hostname（Docker 網路可解析），例如 broker-1:19092

外部 Listener

使用 localhost 搭配唯一映射 Port，例如 localhost:29092、localhost:39092

ADVERTISED_LISTENERS

必須正確反映內部與外部 hostname，否則 Client 無法連線

SSL 部署

SSL 憑證檔案放置於容器內 /etc/kafka/secrets。可參考官方 docker/examples/

Volume 掛載

bash
# 掛載自訂設定檔
$ docker run --volume /path/to/config:/mnt/shared/config -p 9092:9092 apache/kafka:latest

# 容器內預設資料目錄：/var/lib/kafka/data
