01

持久化設計 Persistence

為何依賴檔案系統？

Kafka 將所有資料寫入檔案系統的持久化日誌（Persistent Log），依賴 OS 的 Pagecache 而非自行管理記憶體快取。

循序 I/O 遠勝隨機 I/O

6 顆 7200rpm SATA RAID-5：循序寫入 ~600 MB/s，隨機寫入 ~100 KB/s — 差距 6000 倍

Pagecache 優勢

32GB 機器可快取 28-30GB，無 GC 開銷。服務重啟後快取仍然溫熱（in-process cache 需重建）

O(1) 操作

讀寫皆為簡單的檔案追加與讀取，效能與資料量無關。不使用 BTree（避免磁碟 seek ~10ms）

💡 JVM 特性考量 Java 物件的記憶體開銷通常是資料大小的 2 倍以上；堆內資料越多 GC 越慢。使用 Pagecache 完全避開這些問題。

效率：Zero-Copy 與壓縮 Efficiency

Zero-Copy（sendfile）

Producer、Broker、Consumer 使用相同的二進位訊息格式，資料傳輸時無需修改。

傳統路徑（4 次複製）sendfile 路徑（1 次複製）
磁碟 → Pagecache → 使用者空間 → Socket Buffer → NIC 磁碟 → Pagecache → 直接到 NIC
⚠️ SSL/TLS 限制 啟用 SSL 時不使用 sendfile（TLS 在使用者空間運作）。對 TLS 場景效能有影響。

端到端批次壓縮

Producer 批次壓縮
→
Broker 以壓縮形式儲存
→
Consumer 以壓縮形式接收
→
Consumer 端解壓
壓縮碼演算法特性
0None無壓縮
1GZIP高壓縮比，CPU 較高
2Snappy平衡壓縮比與速度
3LZ4高速壓縮/解壓
4ZStandard最佳壓縮比，可調級別
03

Producer 設計 The Producer

負載平衡

Producer 直接發送到 Partition Leader Broker，無中間路由。所有 Kafka 節點都能回應 Metadata 請求。

分區策略說明
Random隨機負載平衡
Sticky（預設）沾黏到同一 Partition 直到 Batch 填滿，再切換
Key-based以 Key Hash 決定 Partition — 相同 Key 保證進入同一 Partition
Custom實作 Partitioner 介面自訂邏輯

非同步批次發送

Producer 在記憶體中累積資料，以更大批次發送。透過 batch.size（預設 16KB）和 linger.ms（預設 5ms）控制，在延遲與吞吐量之間取捨。

04

Consumer 設計 The Consumer

Pull vs Push 模型

Kafka 採用 Pull-based 模型：Consumer 主動從 Broker 拉取資料。

比較PushPull（Kafka）
速率控制由 Broker 決定 → Consumer 可能被淹沒Consumer 自行控制 → 可自行追趕
批次效率難以最佳化批次大小拉取所有可用訊息，天然支援批次
空閒處理無此問題Long Poll 機制避免忙碌等待

Consumer Position（Offset）

Kafka 的做法極為簡化：每個 Partition 在每個 Consumer Group 中的位置 = 一個整數（Offset）。不需要訊息鎖、確認追蹤。位置定期 Checkpoint 到 __consumer_offsets Topic。Consumer 可隨時倒回（Rewind）重新消費。

Static Membership

設定 group.instance.id 提供持久身份 → 短暫重啟不觸發 Rebalance。需 Broker 與 Client 版本 ≥ 2.3。

05

訊息傳遞語意 Message Delivery Semantics

語意說明實現方式
At-most-once 訊息可能丟失，不重複 Producer 停用重試；Consumer 處理前先 Commit Offset
At-least-once（預設） 訊息不丟失，可能重複 Producer 啟用重試；Consumer 處理後再 Commit Offset
Exactly-once 每條訊息恰好處理一次 Idempotent Producer + Transactional Producer + read_committed

Exactly-Once 實現要件

Idempotent Producer

Broker 為每個 Producer 分配 ID，每條訊息附帶 Sequence Number → 自動去重

Transactional Producer

將訊息原子性發送到多個 Topic-Partition。設定 transactional.id

read_committed Consumer

isolation.level=read_committed → 僅可見已 Commit 的事務訊息

事務錯誤處理分類

例外類型處理方式
RetriableException暫時性，內部自動重試
AbortableException需中止事務 + 重新處理
ApplicationRecoverableException需應用層復原，含 Producer 重啟
InvalidConfigurationException設定問題，無法自動復原

Replication 副本機制 Replication

架構

每個 Partition：1 Leader + N Follower（共 = Replication Factor）。所有寫入發到 Leader；Follower 像 Consumer 一樣從 Leader 拉取複製。

ISR（In-Sync Replicas）

Leader 維護同步副本集（ISR）。Follower 被移出 ISR 的條件：

Session 遺失

Controller 偵測 Heartbeat 逾時 → Follower 被認定為死亡

落後太多

無法在 replica.lag.time.max.ms（預設 30s）內追上 Leader Log 末端

Committed Message 規則

acks 值行為持久性
0不等待任何確認最低
1Leader 寫入即確認中等
all (-1)等待所有 ISR 副本確認最高

搭配 min.insync.replicas：ISR 數量 < 此值時，acks=all 的寫入會被拒絕。

ISR 模型 vs 多數投票

特性多數投票（Raft/Zab）ISR（Kafka）
容忍 f 次故障所需副本2f+1f+1
Commit 條件f+1 副本確認所有 ISR 確認
磁碟用量較多（5 副本容 2 故障）較少（3 副本容 2 故障）
Leader 選舉任何多數成員僅 ISR 成員

Unclean Leader Election

當某 Partition 的所有副本都死亡時：

選項策略風險
unclean.leader.election.enable=false（預設）等待 ISR 副本恢復可能永久不可用
unclean.leader.election.enable=true接受非 ISR 副本為 Leader可能丟失資料

Log Compaction 日誌壓縮

保留每個 Key 的最後已知 Value。設定 cleanup.policy=compact。

Log 結構

Tail（已壓縮）
每個 Key 僅保留最新值
→
Head（未壓縮）
所有新寫入訊息

刪除機制

Key + null payload = Tombstone（墓碑標記）。先前該 Key 的訊息會在壓縮時被移除。Tombstone 本身在 delete.retention.ms（預設 24 小時）後清除。

壓縮保證

#保證
1跟上進度的 Consumer 會看到每條寫入的訊息（受 min.compaction.lag.ms 控制）
2順序永遠保持 — 壓縮不重新排序
3Offset 永遠不變 — 是位置的永久識別碼
4從頭消費可看到每個 Key 的至少最終狀態

壓縮流程

背景 Log Cleaner 執行緒：選擇最髒的 Log → 建立 Key→Last Offset 的雜湊表（每項 24 bytes）→ 重新複製段檔案並跳過已有更新值的 Key。使用 8GB Cleaner Buffer 可清理約 366GB Log Head（假設 1KB 訊息）。

使用案例

資料庫變更訂閱 CDC

跨系統複製資料庫狀態（搭配 Debezium 等）

事件溯源 Event Sourcing

變更日誌作為主要儲存

高可用日記帳 Journaling

串流計算的容錯本地狀態（如 Kafka Streams State Store）

Quotas 配額 Quotas

配額類型引入版本單位說明
Network Bandwidth0.9bytes/sec per broker限制每個 Client Group 每個 Broker 的讀寫速率
Request Rate0.11% of I/O + Network threads限制 Client 可使用的請求處理時間百分比

Client 群組識別

以 (user, client-id) 元組識別。同群組所有連線共享配額。設定優先順序共 8 層（從最具體到最通用）。

節流機制

超出配額 → Broker 計算延遲時間 → 回應中帶入延遲 → 靜音頻道直到延遲結束。Fetch 回應不包含資料。測量視窗：多個小視窗（如 30 個 1 秒視窗）快速偵測。

09

網路層 Network Layer

NIO 架構

1 個 Acceptor 執行緒 + N 個 Processor 執行緒。每個 Processor 處理固定數量的連線。

TCP 協定

二進位、Size-delimited。連線不需握手。建議使用持久連線。每個連線保證按序處理。

批次設計

Send / Fetch API 始終處理訊息集合而非單一訊息。支援跨多個 Topic-Partition 的批次。

雙向相容：新 Client 可與舊 Server 溝通，舊 Client 可與新 Server 溝通（透過 API 版本協商）。

訊息格式 Record Batch v2

目前格式為 magic=2（自 0.11.0）。4.0 起已移除 v0/v1 的寫入支援。

Record Batch 結構

binary layout
baseOffset:           int64    // 8 bytes
batchLength:          int32    // 4 bytes
partitionLeaderEpoch: int32    // 4 bytes — 不含在 CRC 中
magic:                int8     // 1 byte — 目前為 2
crc:                  uint32   // 4 bytes — CRC-32C (Castagnoli)
attributes:           int16    // 2 bytes — 見下表
lastOffsetDelta:      int32    // 4 bytes
baseTimestamp:        int64    // 8 bytes
maxTimestamp:         int64    // 8 bytes
producerId:           int64    // 8 bytes
producerEpoch:        int16    // 2 bytes
baseSequence:         int32    // 4 bytes
recordsCount:         int32    // 4 bytes
records:              [Record] // 壓縮時整體壓縮

磁碟上總大小 = batchLength + 12 bytes。CRC 涵蓋 attributes 到批次末尾。

Attributes 位元遮罩

Bit用途
0–2壓縮類型（0=none, 1=gzip, 2=snappy, 3=lz4, 4=zstd）
3Timestamp 類型（0=CreateTime, 1=LogAppendTime）
4是否為 Transactional（0=否, 1=是）
5是否為 Control Batch（0=否, 1=是）
6是否設定 Delete Horizon
7–15未使用

Record 結構（每筆記錄）

binary layout
length:         varint
attributes:     int8      // 目前未使用
timestampDelta: varlong   // = timestamp - baseTimestamp
offsetDelta:    varint    // = offset - baseOffset
keyLength:      varint
key:            byte[]
valueLength:    varint
value:          byte[]
headersCount:   varint
headers:        [Header]  // headerKeyLength(varint) + headerKey(String) + headerValueLength(varint) + headerValue(byte[])

使用與 Protobuf 相同的 Varint（Zig-Zag）編碼。Delta 編碼大幅減少批次內的重複資訊。

特性v0v1v2（目前）

Varint❌❌✅

磁碟 Log 結構 Log on Disk

目錄與檔案

disk layout
log.dirs/
├── my-topic-0/
│   ├── 00000000000000000000.log       // 日誌段
│   ├── 00000000000000000000.index     // Offset 索引
│   ├── 00000000000000000000.timeindex // 時間索引
│   ├── 00000000000000123456.log
│   ├── 00000000000000123456.index
│   ├── 00000000000000123456.timeindex
│   ├── leader-epoch-checkpoint
│   └── ...(.snapshot, .txnindex)
└── my-topic-1/

每個 .log 檔案以其第一條訊息的 Offset 命名。

索引檔案

索引內容項目大小密度控制
.indexOffset → 檔案中的實體位置（bytes）8 bytes（4B 相對 Offset + 4B Position）log.index.interval.bytes=4096
.timeindexTimestamp → Offset—僅在 Timestamp 大於上次索引時插入

兩者皆為稀疏索引，使用二元搜尋查找。

讀寫流程

寫入

始終追加到最後一個段檔案。達到 log.segment.bytes（預設 1GB）或 log.segment.ms（預設 7 天）時滾動新檔案。

讀取

① 二元搜尋定位段檔案 → ② 在 .index 中找到最近位置 → ③ 循序讀取至目標 Offset

刪除

以整個段為單位。時間策略看段中最大 Timestamp；大小策略持續刪除最舊段。

啟動恢復

迭代最新段的所有訊息，驗證 Size + Offset + CRC32。偵測到損壞時截斷到最後有效 Offset。

版本協商機制

Client 透過 ApiVersionsRequest 查詢 Server 支援的 API 版本範圍，雙方使用共同支援的最高版本。KIP-482 引入的 Tagged Fields 允許 Schema 演進而不需版本提升。
