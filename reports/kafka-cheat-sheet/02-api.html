02

Producer API 發佈 Event

核心類別位於 org.apache.kafka.clients.producer，以 KafkaProducer<K,V> 為中心。

關鍵類別

KafkaProducer<K,V>

實作 Producer<K,V> 介面。執行緒安全，建議整個應用共用一個實例。

ProducerRecord<K,V>

包含 topic、可選 partition、可選 key、value、可選 timestamp、可選 headers。

RecordMetadata

發送成功後回傳：topic、partition、offset、timestamp、序列化後大小。

核心方法

方法說明
send(record)非同步發送，回傳 Future<RecordMetadata>
send(record, callback)非同步發送 + 完成回呼
flush()阻塞直到所有緩衝記錄送出
close() / close(Duration)關閉 Producer，先 flush 剩餘記錄
partitionsFor(topic)取得 Topic 的 Partition 資訊
initTransactions()初始化 Transactional Producer（僅呼叫一次）
beginTransaction()開始新 Transaction
commitTransaction()提交 Transaction
abortTransaction()中止 Transaction

基本發送範例

Java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("key.serializer",   "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");

try (KafkaProducer<String,String> producer = new KafkaProducer<>(props)) {
    producer.send(new ProducerRecord<>("my-topic", "key", "value"),
        (metadata, ex) -> {
            if (ex != null) log.error("失敗", ex);
            else log.info("partition={}, offset={}", metadata.partition(), metadata.offset());
        });
}

Idempotent Producer 與 Transactional Producer

Idempotent Producer

自 3.0 起預設啟用（enable.idempotence=true）。保證 Partition 內 Exactly-Once 語義。

  • 要求 acks=all
  • retries > 0
  • max.in.flight.requests.per.connection ≤ 5
Transactional Producer

在 Idempotent 基礎上，額外設定 transactional.id，啟用跨 Partition / 跨 Topic 的原子寫入。

  • 呼叫順序：initTransactions() → beginTransaction() → send → commitTransaction()
  • Consumer 端需搭配 isolation.level=read_committed
💡 4.0 重要變更
linger.ms 預設值從 0 改為 5，提升 Batching 效率。所有新叢集自動受益。

Consumer API 訂閱與消費 Event

核心類別：KafkaConsumer<K,V>（實作 Consumer<K,V>）。非執行緒安全，每個執行緒需獨立實例。

subscribe() vs assign()

方法模式特性
subscribe(Collection<String>)
subscribe(Pattern)		Consumer Group 自動分配支援 Rebalance、Offset 管理；與 assign() 互斥
assign(Collection<TopicPartition>)手動指定 Partition無 Group 協調、無 Rebalance；適合精確控制場景

標準 Poll Loop

Java
try (KafkaConsumer<String,String> consumer = new KafkaConsumer<>(props)) {
    consumer.subscribe(List.of("my-topic"));
    while (running) {
        ConsumerRecords<String,String> records = consumer.poll(Duration.ofSeconds(1));
        for (ConsumerRecord<String,String> record : records) {
            process(record);
        }
    }
}

Offset 管理

模式設定方法適用場景
自動提交 enable.auto.commit=true（預設）
auto.commit.interval.ms=5000		 由背景執行緒定期提交 簡單場景，可容忍少量重複消費
同步手動提交 enable.auto.commit=false commitSync()
commitSync(Map<TP,OffsetAndMetadata>)		 需要精確控制提交時機
非同步手動提交 enable.auto.commit=false commitAsync()
commitAsync(callback)		 高吞吐場景，不阻塞處理

ConsumerRebalanceListener

回呼方法觸發時機典型用途
onPartitionsRevoked()Rebalance 之前Flush 快取、手動 Commit Offset
onPartitionsAssigned()Rebalance 之後Seek 到特定位置、初始化狀態
onPartitionsLost()Partition 被強制移除時Cooperative Rebalance 場景的清理

其他重要方法

方法說明
seek(tp, offset)重設 Consumer 在特定 Partition 的讀取位置
seekToBeginning() / seekToEnd()跳到最早 / 最新 Offset
position(tp)取得目前讀取位置
committed(tp)取得最近提交的 Offset
wakeup()從另一個執行緒安全中斷 poll()
💡 新版 Consumer Group Protocol（KIP-848）
設定 group.protocol=consumer 可啟用伺服器端 Session / Heartbeat 管理（4.0 GA）。此時 Client 端的 session.timeout.ms 和 heartbeat.interval.ms 會被忽略，改由 Broker 端 group.consumer.* 設定控制。

Share Consumer API 佇列式消費 4.2 正式可用

KIP-932 引入的 Share Group，在 4.2 達到正式可用（Production Ready）。核心類別為 KafkaShareConsumer<K,V>，同樣在 kafka-clients 模組中。

與傳統 Consumer API 的差異

特性Consumer GroupShare Group
Partition 分配每個 Partition 獨佔分配給一個 Consumer多個 Consumer 共享同一 Partition
語義Log 串流（有序）佇列式（類似 RabbitMQ / JMS）
確認機制基於 Offset 提交（批次）逐筆確認（per-record ACK）+ 投遞計數
未確認處理重新 RebalanceAcquisition Lock 逾時後自動重新投遞
assign()✅ 支援❌ 僅支援 subscribe()

確認模式

模式設定值行為
隱式（預設）share.acknowledgement.mode=implicit下次 poll() 或 commitSync() 時自動確認
顯式share.acknowledgement.mode=explicit必須逐筆呼叫 acknowledge(record, type)

AcknowledgeType 列舉

值說明
ACCEPT標記為已處理
REJECT標記為失敗，計入投遞次數
RENEW 4.2延長 Acquisition Lock（KIP-1222），適合長時間處理

程式碼範例

Java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("group.id", "my-share-group");
props.put("group.type", "share");
props.put("key.deserializer",   "...StringDeserializer");
props.put("value.deserializer", "...StringDeserializer");

try (KafkaShareConsumer<String,String> consumer = new KafkaShareConsumer<>(props)) {
    consumer.subscribe(List.of("my-topic"));
    while (running) {
        ConsumerRecords<String,String> records = consumer.poll(Duration.ofMillis(1000));
        for (ConsumerRecord<String,String> record : records) {
            try {
                process(record);
                consumer.acknowledge(record, AcknowledgeType.ACCEPT);
            } catch (Exception e) {
                consumer.acknowledge(record, AcknowledgeType.REJECT);
            }
        }
        consumer.commitSync();
    }
}

設定預設值範圍說明

Kafka Streams API 串流處理

提供兩層抽象：DSL（高階，宣告式）和 Processor API（低階，自訂邏輯）。

DSL 核心類別

類別語義說明
StreamsBuilder進入點stream() → KStream，table() → KTable，globalTable() → GlobalKTable，build() → Topology
KStream<K,V>INSERT（Record Stream）無界的 Key-Value 序列，每筆是獨立事件
KTable<K,V>UPSERT（Changelog）Key-Value 表格，透過 Changelog 更新
GlobalKTable<K,V>全副本表格從所有 Partition 填充，支援 Star Join（不需 Co-partitioning）
KafkaStreams執行期start()、close()、State Query

常用 Stateless 操作

filter、map、mapValues、flatMap、flatMapValues、selectKey、merge、split、peek、foreach、repartition、groupByKey、groupBy

常用 Stateful 操作

count、reduce、aggregate（需先 groupByKey / groupBy）

Join 組合

Join 類型左右是否需要 Window
Inner / Left / Outer JoinKStreamKStream✅ 必須（Tumbling / Hopping / Sliding / Session）
Inner / Left JoinKStreamKTable❌
Inner / Left JoinKStreamGlobalKTable❌（不需 Co-partitioning）
Inner / Left / Outer JoinKTableKTable❌（含 Foreign-Key Join）

DSL 範例

Java
StreamsBuilder builder = new StreamsBuilder();

KStream<String, String> source = builder.stream("input-topic");
source.filter((k, v) -> v.length() > 5)
      .mapValues(v -> v.toUpperCase())
      .to("output-topic");

KafkaStreams streams = new KafkaStreams(builder.build(), config);
streams.start();

Processor API 核心介面

介面說明
Processor<KIn,VIn,KOut,VOut>自訂處理邏輯，init() / process(Record) / close()
ProcessorContext存取 State Store、Forward Record、Schedule Punctuator
KeyValueStoreKey-Value 狀態存儲
WindowStore / SessionStore視窗 / Session 狀態存儲

4.2 新功能

NEW Dead Letter Queue（KIP-1034）

透過 errors.deadletterqueue.topic.name 設定，處理失敗的記錄自動路由到指定 Topic。

GA Streams Rebalance Protocol（KIP-1071）

Task 分配移至 Broker 端，提升可觀測性。需在 Broker 啟用 group.coordinator.rebalance.protocols 包含 streams。

NEW Anchored Punctuation（KIP-1146）

schedule() 新增可選 startTime 參數，可指定 Punctuator 的起始錨點時間。

Kafka Connect API 資料整合

Connect API（connect-api 模組）用於建構可重複使用的 Source / Sink Connector。

核心抽象

介面角色關鍵方法
SourceConnector定義 Source 設定與 TasktaskClass()、start()、stop()、taskConfigs(maxTasks)、config()
SinkConnector定義 Sink 設定與 Task同上
SourceTask從外部系統讀取poll() → List<SourceRecord>、commit()
SinkTask寫入外部系統put(Collection<SinkRecord>)、flush()
Converter序列化格式JSON、Avro、Protobuf
Transformation<R>Single Message Transform (SMT)在 Pipeline 中逐筆轉換

資料模型

SourceRecord

sourcePartition、sourceOffset、topic、schema、key、value、headers

SinkRecord

topic、partition、offset、key、value、headers

REST API 端點

方法路徑說明
GET/connectors列出所有 Connector
POST/connectors建立新 Connector
GET/connectors/{name}/status查看狀態
PUT/connectors/{name}/config更新設定
POST/connectors/{name}/restart重啟
PUT/connectors/{name}/pause暫停
PUT/connectors/{name}/stop停止（3.5+）
DELETE/connectors/{name}刪除

Connector 狀態機

UNASSIGNED → RUNNING → PAUSED / STOPPED / FAILED / RESTARTING

Exactly-Once 支援

Sink Connector

自 0.11.0 起支援（搭配 Transactional Producer）

Source Connector

自 3.3.0 起支援（KIP-618）。可透過 TransactionContext 自訂 Transaction 邊界。

Admin API 叢集管理

核心介面 org.apache.kafka.clients.admin.Admin，透過 Admin.create(config) 建立實例。4.0 起 --zookeeper 已完全移除，僅支援 --bootstrap-server。

常用操作

操作方法
建立 TopiccreateTopics(Collection<NewTopic>)
刪除 TopicdeleteTopics(Collection<String>)
描述 TopicdescribeTopics(Collection<String>)
列出 TopiclistTopics()
描述 ClusterdescribeCluster()
管理 ACLcreateAcls() / deleteAcls() / describeAcls()
Consumer GrouplistConsumerGroups() / describeConsumerGroups()
修改設定incrementalAlterConfigs()
Offset 管理listOffsets() / deleteRecords()

程式碼範例

Java
Properties config = new Properties();
config.put("bootstrap.servers", "localhost:9092");

try (Admin admin = Admin.create(config)) {
    // 建立 3 Partition、RF=3 的 Topic
    NewTopic topic = new NewTopic("my-topic", 3, (short) 3);
    admin.createTopics(List.of(topic)).all().get();

    // 列出所有 Topic
    Set<String> names = admin.listTopics().names().get();

    // 描述 Cluster
    DescribeClusterResult cluster = admin.describeCluster();
    System.out.println("Cluster ID: " + cluster.clusterId().get());
}
