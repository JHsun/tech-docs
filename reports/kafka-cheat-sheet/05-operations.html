Topic 與叢集基本操作 Basic Operations

bash — Topic 管理
# 建立 Topic
$ bin/kafka-topics.sh --bootstrap-server localhost:9092 --create \
    --topic my_topic --partitions 20 --replication-factor 3

# 刪除 Topic
$ bin/kafka-topics.sh --bootstrap-server localhost:9092 --delete --topic my_topic

# 增加 Partition（不可減少）
$ bin/kafka-topics.sh --bootstrap-server localhost:9092 --alter \
    --topic my_topic --partitions 40

# 修改 / 刪除 Topic 設定
$ bin/kafka-configs.sh --bootstrap-server localhost:9092 --entity-type topics \
    --entity-name my_topic --alter --add-config retention.ms=86400000
$ bin/kafka-configs.sh ... --delete-config retention.ms
⚠️ 增加 Partition 的副作用不會重新分配現有資料。使用 Key-based Partitioning 的場景會被打亂（hash(key) % N 改變）。

優雅關機（Graceful Shutdown）

需 controlled.shutdown.enable=true。效果：① 所有 Log Sync 到磁碟（避免重啟時 Log Recovery）② Leadership 先遷移到其他副本 → Partition 不可用時間降至毫秒級。

平衡 Leadership

bash
# 手動觸發 Preferred Leader Election
$ bin/kafka-leader-election.sh --bootstrap-server localhost:9092 \
    --election-type preferred --all-topic-partitions

自動平衡：auto.leader.rebalance.enable=true（預設）。Rack-aware：設定 broker.rack=my-rack-id。

02

叢集擴縮與 Partition 搬移 Cluster Expansion

新加入的 Broker 不會自動接收 Partition，需手動使用 Reassignment 工具。

bash — 三步驟搬移流程
# Step 1：產生搬移計畫
$ bin/kafka-reassign-partitions.sh --bootstrap-server localhost:9092 \
    --topics-to-move-json-file topics.json --broker-list "5,6" --generate

# Step 2：執行搬移（可加限速）
$ bin/kafka-reassign-partitions.sh --bootstrap-server localhost:9092 \
    --reassignment-json-file plan.json --execute \
    --throttle 50000000  # 50 MB/s

# Step 3：驗證完成（同時移除限速！）
$ bin/kafka-reassign-partitions.sh --bootstrap-server localhost:9092 \
    --reassignment-json-file plan.json --verify
⚠️ 重要搬移完成後必須執行 --verify，否則限速（Throttle）不會被移除，影響正常 Replication。

JSON 格式範例

topics-to-move.json

{"topics":[{"topic":"foo1"},{"topic":"foo2"}],"version":1}

自訂 reassignment.json

{"version":1,"partitions":[{"topic":"foo1","partition":0,"replicas":[5,6,7]}]}

增加 Replication Factor

同樣使用 Reassignment 工具，JSON 中指定更多 Replica 即可。

Consumer Group 管理 Consumer Groups

bash
# 查看 Group 詳情（Offset、Lag）
$ bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
    --describe --group my-group

# 查看成員 / 狀態
$ bin/kafka-consumer-groups.sh ... --describe --group my-group --members --verbose
$ bin/kafka-consumer-groups.sh ... --describe --group my-group --state

# 列出 / 刪除 Group
$ bin/kafka-consumer-groups.sh ... --list
$ bin/kafka-consumer-groups.sh ... --delete --group my-group

# 重設 Offset（Consumer 必須先停止）
$ bin/kafka-consumer-groups.sh ... --reset-offsets \
    --group my-group --topic t1 --to-latest --execute

Reset Offset 選項

選項說明
--to-earliest最早 Offset
--to-latest最新 Offset
--to-datetime <ISO>指定時間點
--shift-by <±N>相對位移
--to-offset <N>絕對 Offset
--by-duration <PnDTnH>向前推指定時間
--to-current目前 Offset

Share Group（4.x 新增）使用 bin/kafka-share-groups.sh，子指令類似。

04

資料中心策略 Datacenters

💡 推薦架構每個資料中心部署獨立 Kafka 叢集，應用只與本地叢集互動，跨 DC 使用 MirrorMaker 2 複製。
策略說明適用
Hub-and-Spoke各 DC 本地叢集 → 聚合到中央叢集全域資料檢視
Active-Active雙向複製（A↔B）雙活資料中心
Active-Passive單向複製（A→B）災難復原
⚠️ 不建議在高延遲 WAN 上運行單一跨 DC 叢集：Replication 延遲高，網路中斷可能導致 Split-Brain。
05

MirrorMaker 2 Geo-Replication

基於 Kafka Connect 框架，複製 Topic 資料、設定、Consumer Group Offset 與 ACL。

複製拓撲

Active/Active、Active/Passive、Aggregation（多→一）、Fan-out（一→多）、Forwarding（鏈式）

最小設定

connect-mirror-maker.properties
clusters = primary, secondary
primary.bootstrap.servers = broker1-primary:9092
secondary.bootstrap.servers = broker1-secondary:9092

primary->secondary.enabled = true
primary->secondary.topics = .*

tasks.max = 5

Topic 命名

預設：{source_alias}.{source_topic}（例：us-west.orders）。可透過 replication.policy.separator 或自訂 replication.policy.class 修改。

Exactly-Once（3.5+）

properties
us-east.exactly.once.source.support = enabled
dedicated.mode.enable.internal.rest = true

監控指標

MBean kafka.connect.mirror:type=MirrorSourceConnector：replication-latency-ms、record-rate、byte-rate。

07

Java 版本與 GC 調校 JVM

推薦 Java 17 / 21 / 25（Client / Streams 最低 Java 11）。

推薦 JVM 參數

bash
-Xmx6g -Xms6g -XX:MetaspaceSize=96m -XX:+UseG1GC
-XX:MaxGCPauseMillis=20 -XX:InitiatingHeapOccupancyPercent=35
-XX:G1HeapRegionSize=16M -XX:MinMetaspaceFreeRatio=50
-XX:MaxMetaspaceFreeRatio=80 -XX:+ExplicitGCInvokesConcurrent

參考基準（LinkedIn 最繁忙叢集）：60 Brokers、50k Partitions（RF=2）、800k msg/sec 寫入、1 GB/sec+ 讀出、90% GC Pause ≈ 21ms。

08

硬體與 OS 調校 Hardware & OS

硬體建議

磁碟

主要瓶頸。推薦 8×7200RPM SATA。多磁碟 = 更高吞吐。Kafka Replication 提供應用層冗餘，RAID 非必要。

記憶體

用於 Pagecache。估算：寫入吞吐 × 30 秒。參考：24GB RAM。

網路

高 Replication Factor + Consumer 數量 → 需足夠頻寬。跨 DC 增大 TCP Socket Buffer。

關鍵 OS 調校

項目建議值說明
File Descriptors≥ 100,000每個 Log Segment + 連線都消耗 FD
vm.max_map_count遠高於預設 6553550k Partition = 100k+ Map Areas。預設值會導致 OOM
vm.swappiness1盡量避免 Swap
Socket Buffer依頻寬延遲積調整跨 DC 場景必調

檔案系統

✅ 推薦 XFS請求延遲 ~160ms（vs EXT4 ~250ms+），變異度更低。掛載時加 noatime。

Application-level fsync：建議使用預設（停用），依賴 OS 背景 Flush + Kafka Replication 確保持久性。

KRaft 運維 KRaft Operations

Provisioning 節點

bash
# 產生 Cluster ID
$ bin/kafka-storage.sh random-uuid

# 格式化 Standalone Controller
$ bin/kafka-storage.sh format --cluster-id $ID --standalone --config controller.properties

# 格式化 Broker 加入既有叢集
$ bin/kafka-storage.sh format --cluster-id $ID --config server.properties --no-initial-controllers

動態 Quorum（KIP-853）

bash
# 升級為動態 Quorum
$ bin/kafka-features.sh --bootstrap-server localhost:9092 upgrade --feature kraft.version=1

# 新增 / 移除 Controller
$ bin/kafka-metadata-quorum.sh ... add-controller
$ bin/kafka-metadata-quorum.sh ... remove-controller --controller-id <id> --controller-directory-id <dir-id>

除錯工具

bash
$ bin/kafka-metadata-quorum.sh ... describe --status        # Quorum 狀態
$ bin/kafka-metadata-quorum.sh ... describe --replication   # 複製狀態
$ bin/kafka-dump-log.sh --cluster-metadata-decoder --files <log>   # 傾印 Metadata Log
$ bin/kafka-metadata-shell.sh --snapshot <file>                    # 互動式瀏覽

Controller Quorum 建議：3 個容 1 故障，5 個容 2 故障。記憶體 5GB + 磁碟 5GB。

Tiered Storage 運維 冷熱分層

啟用流程

bash
# Broker 端啟用
remote.log.storage.system.enable=true
remote.log.storage.manager.class.name=<RSM 實作類別>

# 建立 Tiered Topic
$ bin/kafka-topics.sh --create --topic tiered --bootstrap-server localhost:9092 \
  --config remote.storage.enable=true \
  --config local.retention.ms=1000 \
  --config retention.ms=3600000

停用流程

操作設定
停止複製到 Remote（保留讀取）remote.log.copy.disable=true
完全停用 + 刪除 Remote 資料remote.storage.enable=false, remote.log.delete.on.disable=true
⚠️ 限制不支援 Compacted Topic。必須先停用所有 Topic 的 Tiered Storage，才能停用 Broker 層級。

Consumer Rebalance Protocol KIP-848

新協定（4.0 GA）使用完全增量設計，無全域同步屏障，Rebalance 更快。

面向Classic ProtocolConsumer Protocol（新）
Client 設定group.protocol=classic（預設）group.protocol=consumer
Session/HeartbeatClient 端控制Server 端控制
Partition 分配Client Leader 計算Server 端 Assignor
Regex 訂閱Client 端評估Server 端評估（RE2J 格式）
Rebalance全域同步屏障增量式

升級路徑

離線：停止所有 Consumer → 重啟設 group.protocol=consumer。
線上（Rolling）：逐一替換 Consumer → 第一個新協定 Consumer 觸發自動轉換。

13

Transaction Protocol KIP-890

新 Transaction Protocol（4.0 GA，transaction.version=2）在每次 Transaction 時 Bump Producer Epoch，確保無重複寫入。4.0+ Producer Client 自動使用。

改善：Server 端單次 AddPartitions 呼叫取代原本的 Client+Server 多次呼叫。CONCURRENT_TRANSACTIONS 錯誤由 Server 內部重試。

14

ELR 合格 Leader 副本 KIP-966

解決「最後一個副本」問題：當所有 ISR 成員同時離線時，Controller 可從 ELR 中安全選出 Leader。

Leader 選舉順序

① ISR（若非空）→ ② ELR（若非空）→ ③ 最後已知 Leader（Fallback）

ISR vs ELR 關係

集合已知方用途
ISRLeaderReplication Quorum — 控制 HWM 推進
ISR + ELRControllerElection Quorum — Leader 候選池

啟用：eligible.leader.replicas.version=1（4.1+ 新叢集預設啟用）。ELR 提供 min.insync.replicas - 1 次不潔淨關閉的容忍度。

監控：kafka.controller:type=ControllerStats,name=ElectionFromEligibleLeaderReplicasPerSec（預期 0）。
