安全總覽 Security Overview

Kafka 安全為可選功能，支援混合模式（安全 + 非安全 Client 共存）。

認證 Authentication

SSL/TLS 雙向認證 或 SASL（Kerberos、PLAIN、SCRAM、OAUTHBEARER）

加密 Encryption

SSL/TLS 傳輸加密（Client↔Broker、Broker↔Broker、Broker↔Controller）

授權 Authorization

可插拔 ACL。KRaft 使用 StandardAuthorizer（ACL 存於 Metadata Log）

預設行為

無 ACL 時資源受限（僅 Super User 可存取）。設 allow.everyone.if.no.acl.found=true 可改為開放。
Super User：super.users=User:Bob;User:Alice（分號分隔）。

02

Listener 設定 Listener Configuration

格式與安全協定

Listener 格式：{LISTENER_NAME}://{hostname}:{port}

Security Protocol認證加密
PLAINTEXT❌❌
SSL✅（可選 mTLS）✅
SASL_PLAINTEXT✅ SASL❌
SASL_SSL✅ SASL + TLS✅

設定範例（KRaft Broker）

server.properties
process.roles=broker
listeners=BROKER://localhost:9092
inter.broker.listener.name=BROKER
controller.listener.names=CONTROLLER

# Listener 名稱 → 安全協定映射
listener.security.protocol.map=BROKER:SASL_SSL,CONTROLLER:SASL_SSL
⚠️ KRaft 規則controller.listener.names 不能與 inter.broker.listener.name 相同。合併模式時 Controller Listener 必須包含在 listeners 中。
03

SSL/TLS 加密與認證 Encryption & Authentication using SSL

支援 JKS（已棄用）、PKCS12（預設）、PEM（2.7+）格式。Hostname 驗證自 2.0 起預設啟用。

SSL 設定流程

① 產生 Broker 金鑰對
→
② 建立 CA
→
③ 簽署憑證
→
④ 設定 Broker
→
⑤ 設定 Client

Step 1：產生 Broker 金鑰對

bash
$ keytool -keystore server.keystore.jks -alias localhost -validity 365 \
    -genkey -keyalg RSA -storetype pkcs12 \
    -ext SAN=DNS:{FQDN},IP:{IP}

Step 2：建立 CA 並加入 Truststore

bash
# 產生 CA
$ openssl req -x509 -config openssl-ca.cnf -newkey rsa:4096 -sha256 -nodes \
    -out cacert.pem -outform PEM

# 加入 Truststore
$ keytool -keystore server.truststore.jks -alias CARoot -import -file cacert.pem
$ keytool -keystore client.truststore.jks -alias CARoot -import -file cacert.pem

Step 3：簽署並匯入

bash
# 產生 CSR
$ keytool -keystore server.keystore.jks -alias localhost -certreq -file cert-req

# CA 簽署
$ openssl ca -config openssl-ca.cnf -policy signing_policy \
    -extensions signing_req -out cert-signed -infiles cert-req

# 匯入 CA 憑證 + 已簽署憑證
$ keytool -keystore server.keystore.jks -alias CARoot -import -file cacert.pem
$ keytool -keystore server.keystore.jks -alias localhost -import -file cert-signed

Broker SSL 設定

server.properties
listeners=SSL://host:9093
ssl.keystore.location=/var/ssl/server.keystore.jks
ssl.keystore.password=changeit
ssl.key.password=changeit
ssl.truststore.location=/var/ssl/server.truststore.jks
ssl.truststore.password=changeit
ssl.client.auth=required        # none | requested | required
ssl.endpoint.identification.algorithm=https
security.inter.broker.protocol=SSL

Client SSL 設定

client.properties
security.protocol=SSL
ssl.truststore.location=/var/ssl/client.truststore.jks
ssl.truststore.password=changeit
# Mutual TLS（mTLS）時需加：
ssl.keystore.location=/var/ssl/client.keystore.jks
ssl.keystore.password=changeit
ssl.key.password=changeit

PEM 格式（2.7+，免 Keystore 檔案）

properties
ssl.keystore.key=-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----
ssl.keystore.certificate.chain=-----BEGIN CERTIFICATE-----\n...\n-----END CERTIFICATE-----
ssl.truststore.certificates=-----BEGIN CERTIFICATE-----\n...\n-----END CERTIFICATE-----
💡 效能提示預設 Linux NativePRNG 使用全域鎖。設定 ssl.secure.random.implementation=SHA1PRNG 可獲得非阻塞效能。

SASL 認證機制 Authentication using SASL

機制特性建議搭配
GSSAPI（Kerberos）企業級 SSO，基於 KeytabSASL_PLAINTEXT 或 SASL_SSL
PLAIN帳號密碼，簡單但明文傳輸必須搭配 SSL
SCRAM-SHA-256/512Challenge-Response，密碼雜湊儲存建議搭配 SSL
OAUTHBEAREROAuth 2.0 JWT 驗證，現代化SASL_SSL

SASL/SCRAM 設定範例

bash — 建立憑證
# 首次（Broker 啟動前）
$ bin/kafka-storage.sh format -t $(bin/kafka-storage.sh random-uuid) -c config/server.properties \
    --add-scram 'SCRAM-SHA-256=[name="admin",password="secret"]'

# 運行中新增使用者
$ bin/kafka-configs.sh --bootstrap-server localhost:9092 --alter \
    --add-config 'SCRAM-SHA-256=[iterations=8192,password=alice-pw]' \
    --entity-type users --entity-name alice

SASL/OAUTHBEARER 設定

broker — server.properties
sasl.enabled.mechanisms=OAUTHBEARER
listener.name.<name>.oauthbearer.sasl.server.callback.handler.class=\
    org.apache.kafka.common.security.oauthbearer.OAuthBearerValidatorCallbackHandler
listener.name.<name>.oauthbearer.sasl.oauthbearer.jwks.endpoint.url=\
    https://example.com/oauth2/v1/keys
client.properties
security.protocol=SASL_SSL
sasl.mechanism=OAUTHBEARER
sasl.oauthbearer.token.endpoint.url=https://example.com/oauth2/v1/token
sasl.oauthbearer.client.credentials.client.id=my-app
sasl.oauthbearer.client.credentials.client.secret=$3cr3t

多重 SASL 機制

server.properties
sasl.enabled.mechanisms=GSSAPI,PLAIN,SCRAM-SHA-256,SCRAM-SHA-512,OAUTHBEARER
sasl.mechanism.inter.broker.protocol=SCRAM-SHA-256
05

ACL 授權 Authorization and ACLs

Authorizer 設定（KRaft）

server.properties
authorizer.class.name=org.apache.kafka.metadata.authorizer.StandardAuthorizer
super.users=User:admin

ACL 格式

Principal {P} 從 Host {H} 被 [Allow|Deny] 對 Resource {R} 執行 Operation {O}

Operations 與 Resource Types

OperationsResource Types
Read, Write, Create, Delete, Alter, Describe, ClusterAction, DescribeConfigs, AlterConfigs, IdempotentWrite, All Topic, Group, Cluster, TransactionalId, DelegationToken, User

kafka-acls.sh 常用指令

bash
# 授予讀寫權限
$ bin/kafka-acls.sh --bootstrap-server localhost:9092 --add \
    --allow-principal User:Bob --operation Read --operation Write --topic my-topic

# 便捷 Producer ACL（含 Write, Describe, Create）
$ bin/kafka-acls.sh ... --add --allow-principal User:Bob --producer --topic my-topic

# 便捷 Consumer ACL（含 Read on Topic + Read on Group）
$ bin/kafka-acls.sh ... --add --allow-principal User:Bob --consumer \
    --topic my-topic --group my-group

# Prefix ACL（多租戶場景）
$ bin/kafka-acls.sh ... --add --allow-principal User:Jane --producer \
    --topic acme- --resource-pattern-type prefixed

# Deny 特定使用者
$ bin/kafka-acls.sh ... --add --allow-principal User:'*' \
    --deny-principal User:BadBob --operation Read --topic my-topic

# 列出 / 移除 ACL
$ bin/kafka-acls.sh ... --list --topic my-topic
$ bin/kafka-acls.sh ... --remove --allow-principal User:Bob --operation Read --topic my-topic

常見 API → Operation → Resource 映射

APIOperationResource
ProduceWriteTopic + TransactionalId + IdempotentWrite(Cluster)
Fetch（Consumer）ReadTopic
Fetch（Follower）ClusterActionCluster
JoinGroup / HeartbeatReadGroup
OffsetCommitReadGroup + Topic
CreateTopicsCreateCluster 或 Topic
DeleteTopicsDeleteTopic
DescribeConfigsDescribeConfigsCluster 或 Topic

滾動升級加入安全功能 Rolling Security Upgrade

從非安全叢集遷移到安全叢集，需四階段滾動重啟：

Phase 1
開啟安全 Port
（保留 PLAINTEXT）
→
Phase 2
遷移 Client
到安全 Port
→
Phase 3
啟用 Inter-Broker
安全協定
→
Phase 4
關閉 PLAINTEXT
Port

各階段設定

階段Broker listenerssecurity.inter.broker.protocol
Phase 1PLAINTEXT://:9091, SSL://:9092PLAINTEXT
Phase 2同上（Client 端改用 9092）PLAINTEXT
Phase 3PLAINTEXT://:9091, SSL://:9092SSL
Phase 4SSL://:9092SSL
⚠️ 關鍵規則 ① PLAINTEXT Port 在最後一次重啟前保持開放。
② Broker-Client 與 Broker-Broker 安全協定在不同次重啟中啟用。
③ 每次重啟後，等待所有副本重新加入 ISR 再進行下一階段。

Delegation Token（委派 Token）

輕量級認證機制，共享密鑰基於 SASL/SCRAM。適合短期任務授權。

bash
# 建立 Token
$ bin/kafka-delegation-tokens.sh --bootstrap-server localhost:9092 --create \
    --max-life-time-period -1 --command-config client.properties \
    --renewer-principal User:user1

# 續期 / 過期 / 查看
$ bin/kafka-delegation-tokens.sh ... --renew --hmac ABCDE...
$ bin/kafka-delegation-tokens.sh ... --expire --hmac ABCDE...
$ bin/kafka-delegation-tokens.sh ... --describe --owner-principal User:user1

Client 端使用 Token 認證：SCRAM Login Module + tokenauth="true"。

Broker 設定預設說明
delegation.token.secret.key—所有 Broker / Controller 必須相同
delegation.token.expiry.time.ms24 小時Token 過期時間
delegation.token.max.lifetime.ms7 天Token 最大生命週期
