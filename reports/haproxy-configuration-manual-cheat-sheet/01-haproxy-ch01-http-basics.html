1.1

HTTP 交易模型 (Transaction Model)

HTTP 是交易驅動式協定：每一個 request 對應唯一一個 response。隨著 HTTP 版本演進，連線的使用方式經歷四個階段：

① HTTP Close（HTTP/1.0）

一條連線只能處理一筆交易；response 結束後連線關閉。客戶端以連線關閉作為 response 結束訊號，因此無法偵測截斷的回應（早期「斷圖」問題即由此而來）。

▼ Close 模式時序
[CON1] [REQ1] … [RESP1] [CLO1]   [CON2] [REQ2] … [RESP2] [CLO2]

② Keep-Alive（HTTP/1.1 預設）

連線可重複使用。Server 必須標示 Content-Length 或使用 Chunked Transfer Encoding（資料分段傳送，每段前綴長度，以零長度段結尾），讓客戶端判斷回應邊界。優點：減少建連延遲、降低伺服器負載、可偵測截斷。

▼ Keep-Alive 模式時序
[CON] [REQ1] … [RESP1] [REQ2] … [RESP2] … [CLO]

③ Pipelining

客戶端連續送出多個 request，不等第一個 response。伺服器必須按照原始順序回覆。實務上由於伺服器支援度極差，此模式已完全被棄用。

▼ Pipelining 模式時序
[CON] [REQ1] [REQ2] … [RESP1] [RESP2] … [CLO]

④ 多工模式 — HTTP/2 & HTTP/3

單一連線上可同時進行多筆獨立識別的 stream，伺服器可以任意順序回應。客戶端通常同時開啟 100 條以上的 stream。Header 透過 HPACK (H2) / QPACK (H3) 壓縮，無法用肉眼閱讀。

⚠ Head-of-Line Blocking (HoL) HTTP/2 所有 stream 共用一條 TCP 連線 — 任一封包遺失會阻塞所有 stream。
HTTP/3 建構於 QUIC (UDP)，每條 stream 獨立處理遺失封包，從傳輸層根本解決 HoL。

HAProxy 三種連線模式設定

模式關鍵字行為適用場景
keep alive option http-keep-alive 前端 & 後端連線均保持（預設） 現代 Web、HTTP/2、HTTP/3
server close option http-server-close 每筆 response 結束後關閉後端連線 後端不支援 keep-alive 時
close option httpclose response 結束後前後端均關閉 極舊後端、除錯
💡 重要觀念 Server 端連線預設可被任意客戶端重複使用（符合 HTTP 規範），因此不應在後端連線上保留客戶端專屬狀態。
例外：當後端使用 HTTP/2 時，HAProxy 會將該連線綁定至同一客戶端，避免跨客戶端的 HoL blocking。
1.2

核心術語定義 (Terminology)

術語定義備註
Connection
連線		 HAProxy 與遠端（client/server）之間的雙向通訊通道，通常對應一個 TCP socket（IP:Port 對）。 客戶端連線是最早建立的實體，connection 層級規則最先套用。
Session
工作階段		 為 connection 附加的上下文：TLS 金鑰、變數等傳輸層資訊。 歷史用語，CLI 與統計中仍可見。勿混淆與應用層 session（cookie/server affinity）。
Stream
串流		 端到端的雙向應用層通訊；分析與轉換規則在此層套用。一個 stream 包含一組 request + response。 HTTP/2 的 stream ID 與 HAProxy stream 為 1:1。TCP 模式下 stream = connection。
Transaction
交易		 一組 request + response 的配對。 現代 HAProxy 中與 stream 1:1 對應。變數作用域 txn 即涵蓋整個 stream 生命週期。
Request
請求		 從 client 流向 server 方向的流量。 作為流量計量單位。TCP 模式中 request 數 = connection 數。Request 不一定有 response（如錯誤中斷）。
Response
回應		 從 server（或 HAProxy 自身）流向 client 方向的流量。 HAProxy 可自行產生 response（redirect、deny、stats page 等）。
Service
服務		 HAProxy 內部處理流程，不需真正的 server：stats page、cache、Lua app。 讀取 request → 內部處理 → 產生 response。

層級關係圖

Connection → Session → Stream(s) → Transaction (1:1)
                          ↳ Request + Response
1.3

HTTP 請求結構 (HTTP Request)

Line 1 (Request Line):  GET /serv/login.php?lang=en&profile=2 HTTP/1.1
Line 2 (Headers):       Host: www.mydomain.com
Line 3:                  User-Agent: my small browser
Line 4:                  Accept: image/jpeg, image/gif
Line 5:                  Accept: image/png
       (空行 = header 結束)

1.3.1 Request Line 結構

由三個欄位組成，以空白字元（空格、Tab、CR/LF+空白）分隔：

欄位範例規則
METHOD GET 只能包含英文字母，不可含冒號 :
URI /serv/login.php?lang=en 四種形式（見下方）
Version HTTP/1.1 HTTP/2、HTTP/3 不傳送此欄位

URI 的四種形式

形式範例使用情境
相對 URI /serv/login.php?lang=en 反向代理、透明代理、一般伺服器（最常見）
絕對 URI http://host:8080/path?q=1 正向代理（Forward Proxy）接收的格式
星號 * OPTIONS * HTTP/1.1 僅限 OPTIONS 方法，查詢下一跳能力
Address:Port 192.168.0.12:443 CONNECT 方法，建立 TCP 隧道（常見於 HTTPS 代理）

URI 子結構

/serv/login.php ? lang=en&profile=2
├── Path（路徑）    └── Query String（查詢字串）

Path：? 之前的部分，對應伺服器上的資源路徑。
Query String：? 之後，常用於 GET 請求傳遞動態參數。

💡 HTTP/2 & HTTP/3 差異 這兩個協定不傳送完整的 request line，而是拆分成 pseudo-headers（以 : 開頭）：
:method :path :scheme :authority
HAProxy 會自動重組成等效的 request line 供分析與日誌使用，但重組後的格式可能與 HTTP/1 的略有差異。

1.3.2 Request Headers 處理規則

規則說明
格式 名稱緊接冒號 :，後面可選空白，再接值。例：Host: www.example.com
大小寫 Header 名稱不區分大小寫。HAProxy 內部統一轉為小寫處理（不論 HTTP/1 或 HTTP/2）。
多值 同名 header 可出現多次，或合併為逗號分隔。例：Accept: image/jpeg, image/gif
多行折疊 後續行以空白開頭可延續上一行（已過時）。HAProxy 會將折行合併為單行，以空白取代換行。
前後空白 值的前導與尾隨空白會被忽略。
結束標記 第一個空行表示 header 區塊結束。
⚠ 注意 — Header 名稱小寫化 即使 HTTP/1.1 client 發送 Content-Type（CamelCase），經過 HAProxy 後會變成 content-type（全小寫）。撰寫 ACL 比對時請使用小寫。
1.4

HTTP 回應結構 (HTTP Response)

Line 1 (Response Line):  HTTP/1.1 200 OK
Line 2 (Headers):        Content-Length: 350
Line 3:                   Content-Type: text/html

1.4.1 Response Line 結構

欄位範例說明
Version HTTP/1.1 HTTP/2+ 使用 :status pseudo-header 取代
Status Code 200 固定 3 位數字；大於 599 的不可送出
Reason OK 僅供參考，客戶端不解析。HTTP/2+ 無此欄位（HAProxy 代理時會自動補上）

Status Code 分類

區段意義說明
1xx資訊性如 100 Continue、101 Switching Protocols（WebSocket 升級）。HAProxy 轉發但不記錄。
2xx成功後接內容 body
3xx重導向通常不含 body
4xxClient 錯誤請求有誤或被拒
5xxServer 錯誤伺服器端問題
💡 101 Switching Protocols 收到 101 後 HAProxy 切換為隧道模式（如同 CONNECT），常見於 WebSocket 升級。透過 Upgrade header 識別目標協定。

1.4.2 Response Headers

Response header 的格式與解析規則完全等同 Request header（§1.3.2）。HAProxy 使用相同的解析函式處理，所有大小寫正規化、多行折疊、空白處理規則一致適用。
