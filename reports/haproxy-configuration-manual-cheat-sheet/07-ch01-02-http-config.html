1HTTP 快速回顧

1.1 HTTP 交易模型 (Transaction Model)

HTTP 是交易驅動的協議：每個請求(request)對應恰好一個回應(response)。連線模式的演進如下：

模式版本說明特點
close HTTP/1.0 每次交易建立新 TCP 連線，回應後關閉 延遲高、開銷大
keep-alive HTTP/1.1 多個交易共用同一 TCP 連線 減少握手、須標示 Content-Length
pipelining HTTP/1.1 客戶端不等回應即送下一筆請求 順序回應、實務少用
multiplexed HTTP/2, HTTP/3 多個 stream 透過同一連線平行傳輸 stream ID 標識、亂序回應
Client HAProxy Server │──── TCP/TLS ──────────────▶│ │ │ stream 1: GET /index.html │──── connection pool ────────▶│ │ stream 2: GET /style.css │ (keep-alive / multiplex) │ │◀──── response 2 ──────────│◀──────────────────────────────│ │◀──── response 1 ──────────│◀──────────────────────────────│
觀念重點
HAProxy 預設使用 keep-alive 模式。HTTP/2 連線預設會與同一客戶端綁定，避免 head-of-line blocking。HTTP/3 基於 QUIC (UDP) 實現，stream 之間互不影響。

1.2 HAProxy 術語對照

術語定義對應範圍
connection L4 層 TCP/QUIC 雙向連線 一條 socket
session 端對端傳輸層連線的生命週期 在 TCP 中 = connection；勿混淆 application session (cookie)
stream 應用層雙向通訊單元；可做分析和轉換 HTTP 中 1 stream = 1 request + 1 response
transaction (txn) request + response 的配對 與 stream 為 1:1，變數作用域 txn.
request 客戶端→伺服器的流量 計量單位、規則掛載點
response 伺服器→客戶端的流量（含 HAProxy 自產） 如 errorfile、deny 頁面等

1.3 HTTP Request 結構

GET /path/page?query=value HTTP/1.1    # Request line: Method SP URI SP Version
Host: www.example.com                     # 必要 Header
User-Agent: Mozilla/5.0 ...
Accept: text/html, application/json
Cookie: sessionid=abc123
                                               # 空行 = Header 結束
(optional body)                                # POST/PUT 才有 body

🔑 Request Line 三要素

  • Method：GET POST PUT DELETE HEAD OPTIONS PATCH 等
  • URI：路徑 + 查詢字串，HAProxy 可用 path / query / url 等 fetch 取得
  • Version：HTTP/1.0、HTTP/1.1（HTTP/2+ 內部以偽 header 表示）

1.4 HTTP Response 結構

HTTP/1.1 200 OK                  # Status line: Version SP Code SP Reason
Content-Type: text/html
Content-Length: 5842
Set-Cookie: sid=xyz; Path=/
Cache-Control: max-age=3600
                                          # 空行
<html>...</html>                          # Response body
HAProxy 注意事項
HAProxy 會將多行 Header 合併為單行（依 RFC 7231）。HTTP/2+ 的 Header 名稱一律轉為小寫。在設定 req.hdr() 等 fetch 時，Header 名稱不區分大小寫。

相關 RFC 參考：

  • RFC 9110 — HTTP Semantics
  • RFC 9112 — HTTP/1.1
  • RFC 9113 — HTTP/2
  • RFC 9114 — HTTP/3

2設定 HAProxy

2.1 設定檔格式

HAProxy 的設定來源（優先級由高到低）：

  1. 命令列參數（最高優先）
  2. 設定檔（/etc/haproxy/haproxy.cfg）
  3. 環境變數（以 "$ENV(VAR)" 引用）

📝 設定檔基本規則

  • 設定檔是有序陳述的序列
  • 每一行（到 # 之前）為一條陳述
  • 以空白或 Tab 分隔 token（詞）
  • 第一個詞 = 關鍵字，後續詞 = 參數
  • 支援以 \ 在行尾續行
  • # 開頭的為註解

設定檔分為以下主要區塊（section）：

區塊用途可出現次數
global全域參數：行程管理、效能調校、安全性0..1
defaults後續 proxy 的預設值，可多段覆蓋0..N
frontend前端：定義監聽位址和請求接收規則0..N
backend後端：定義伺服器群和負載平衡策略0..N
listen同時扮演 frontend + backend 的簡寫0..N
global
    daemon
    maxconn  4096

defaults
    mode    http
    timeout connect  5s
    timeout client  30s
    timeout server  30s

frontend web_front
    bind  *:80
    default_backend  web_servers

backend web_servers
    balance  roundrobin
    server   srv1  10.0.0.1:8080  check
    server   srv2  10.0.0.2:8080  check

2.2 引號與跳脫

語法效果範例
'single quotes' 保留空白、特殊字元原樣（不做任何跳脫） reqadd 'X-Hdr: some value'
"double quotes" 保留空白，支援 \n \t \\ \" 跳脫 log-format "%ci:%cp [%t]"
\ (行尾) 續行 長規則拆成多行
\x41 十六進位 byte 傳送特殊字元

2.3 環境變數

# 設定檔中引用環境變數
global
    log  "$ENV(SYSLOG_SERVER)"  local0

frontend fe_main
    bind  "$ENV(BIND_ADDR)":"$ENV(BIND_PORT)"
相關指令
setenv、presetenv（不覆蓋已有）、resetenv（清空後只留指定變數）皆在 global 區塊使用。

2.4 條件式區塊

HAProxy 支援編譯期條件判斷，適合多環境共用一份設定檔：

# .if / .elif / .else / .endif
.if streq("$ENV(ENV_NAME)","production")
    maxconn  50000
.elif streq("$ENV(ENV_NAME)","staging")
    maxconn  10000
.else
    maxconn  2000
.endif

# 可用判斷函式
# defined(ENV_VAR)   — 變數是否存在
# feature(OPENSSL)   — 是否編譯了某功能
# streq(A,B)         — 字串相等
# version_atleast(X) — 版本 >= X

2.7 Map 與 ACL 命名規則

Map 和 ACL 檔案的命名可包含路徑，但需注意：

  • 名稱用於在 runtime CLI 中引用：show acl @0 或 show map /etc/haproxy/map/hosts.map
  • 若同一檔案在多處引用，會共享同一份記憶體資料
  • 可透過 -f 指定多個設定檔時注意相對路徑解析

2.8 完整設定範例

global
    log          /dev/log  local0
    chroot       /var/lib/haproxy
    user         haproxy
    group        haproxy
    daemon
    maxconn      50000
    stats socket /run/haproxy/admin.sock mode 660 level admin

defaults
    log      global
    mode     http
    option   httplog
    option   dontlognull
    timeout  connect  5s
    timeout  client  30s
    timeout  server  30s
    retries  3

frontend http_in
    bind             *:80
    bind             *:443 ssl crt /etc/ssl/certs/site.pem
    redirect         scheme https if !{ ssl_fc }
    default_backend  app_servers

backend app_servers
    balance  roundrobin
    option   httpchk GET /health
    server   app1  10.0.1.1:8080  check inter 3s fall 3 rise 2
    server   app2  10.0.1.2:8080  check inter 3s fall 3 rise 2

listen stats
    bind          *:8404
    stats         enable
    stats uri     /stats
    stats refresh 10s
安全提醒
生產環境務必設定 chroot、降權執行（user/group），並限制 stats socket 的存取權限。stats 頁面建議加上 stats auth 帳密或 ACL 限制。
