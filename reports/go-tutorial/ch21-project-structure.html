專案結構 Layout

Go 官方沒有規定專案結構。但社群有幾個共識 — cmd/、internal/、扁平優先。本章給你實務上能直接用的決策樹，而不是教條式的層級模板。

核心概念 · package-per-concept · internal/ 隱私保護

Package 的哲學

Go 的 package ≠ Python 的 module。每個目錄是一個 package，package 名稱有語意 — user、auth、store，而不是 utils、helpers、common。

package name ≠ dir name（但慣例相同）

理論上可以在 foo/ 目錄寫 package bar，但 不要這麼做。一致性勝於小聰明。

package 名用小寫、單字、短

http、json、user — 不要 userService、user_service。

避免「垃圾桶」package

utils、helpers、common、misc — 這些名字是 code smell。該型別屬於哪個概念，就放哪個 package。

避免循環 import

Go 不允許 A import B 又 B import A。撞牆代表設計有問題 — 通常是抽一個更基礎的 C 出來。

Package 命名測試 好的 package 命名，使用端讀起來像自然英語：bytes.Buffer、http.Client、user.Service。壞的命名會 stutter：user.UserService、http.HTTPClient — 名字重複就是訊號。

internal/ · 編譯器層級的隱私

Go 唯一的「檔案夾層級 access control」— 放在 internal/ 下的 package 只能被它的父樹的其他 package import。這是 Go 1.4+ 就有的特性，但很多新手不知道。

myapp/
├── go.mod                       // module github.com/myorg/myapp
├── cmd/
│   └── server/main.go           // ✓ 可以 import internal/auth
├── internal/
│   ├── auth/auth.go             // 只有 myapp/ 下的程式碼可見
│   └── store/store.go
└── pkg/
    └── client/client.go         // ✓ 外部也能 import

// 從 myapp/ 內部：
import "github.com/myorg/myapp/internal/auth"    // ✓ OK

// 從其他 repo：
import "github.com/myorg/myapp/internal/auth"    // ❌ 編譯失敗
import "github.com/myorg/myapp/pkg/client"      // ✓ OK
實務建議 把所有東西預設放 internal/。等到真的有「我要公開這個 API 給外部使用」的需求，才搬出來。這讓你保留「自由重構內部」的權利 — 只要還沒有外部使用者就能改 API。

三種專案類型的標準 layout

類型 A · 單一可執行檔（小工具、腳本）

mytool/
├── go.mod
├── go.sum
├── main.go                // package main，全部邏輯
├── parse.go               // 同一個 package main
├── format.go
└── main_test.go

// 建置：go build -o mytool ./
扁平是美德 很多新手拿到專案立刻建 cmd/、internal/ 一堆資料夾 — 但幾百行的小工具根本不需要。先寫在同一個 package，等程式真的變複雜再切分。過早的抽象是技術債。

類型 B · 函式庫（純 library）

mylib/
├── go.mod
├── go.sum
├── mylib.go              // 主要 API，package mylib
├── mylib_test.go
├── internal/             // 實作細節藏這
│   └── parser/
└── examples/             // 獨立的範例程式（各自 package main）
    └── basic/main.go

// 使用端：import "github.com/myorg/mylib"
//         然後直接用 mylib.Xxx()

類型 C · 標準 Web 服務（最常見）

myapp/
├── go.mod
├── go.sum
├── Makefile / Dockerfile
│
├── cmd/                          // 所有可執行檔
│   ├── server/main.go            // HTTP server
│   ├── worker/main.go            // 背景工作 worker
│   └── migrate/main.go           // DB migration 工具
│
├── internal/                     // 所有業務程式
│   ├── config/                   // 設定載入
│   ├── user/                     // 使用者領域
│   │   ├── user.go               // type User, 業務邏輯
│   │   ├── service.go
│   │   ├── store.go              // interface
│   │   ├── postgres.go           // interface 的 PG 實作
│   │   └── user_test.go
│   ├── auth/
│   ├── httpapi/                  // HTTP handlers 與 middleware
│   │   ├── handlers.go
│   │   ├── middleware.go
│   │   └── routes.go
│   └── db/                       // DB 連線、migration 載入
│
├── pkg/                          // 可選：公開 API（客戶端 SDK 等）
│   └── client/
│
├── api/                          // 可選：OpenAPI/Proto 定義
│   └── openapi.yaml
│
├── migrations/                   // SQL migration 檔
│   ├── 0001_init.up.sql
│   └── 0001_init.down.sql
│
├── web/                          // 可選：前端靜態資源（embed）
│   └── static/
│
└── testdata/                     // 測試用固定資料
    └── fixtures/

cmd / pkg / internal 的經典爭議

cmd/ 放 main 套件

每個子目錄是一個 package main。這讓你用同一個 module 產出多個 binary（server、worker、migrate…），共用 internal/ 程式碼。

internal/ 放業務邏輯

預設的「放東西的地方」。編譯器保證外部無法 import。

pkg/ 是爭議

Go 官方不推薦。Go std lib 不用它。但有些大專案（k8s）用它區隔「公開給生態」的 API。對 95% 的專案：不要 pkg/。

「golang-standards/project-layout」警告

GitHub 上星最多的那個「Go 標準 layout」repo 並非官方，維護者自己也說過。它適合大型專案參考，對小專案是過度工程。

Russ Cox (Go tech lead) 的建議 「When in doubt, don't use pkg/」。標準庫、多數大型 Go 專案（Kubernetes、Docker 是例外）都不用 pkg/。把 pkg/ 的層級多出來的那個「pkg」念出聲，你就知道它沒意義。

依賴方向 · Clean Architecture 的 Go 實作

單一方向的依賴關係是長期可維護的關鍵。核心業務邏輯不該 import 具體的資料庫或 HTTP 框架 — 要倒過來。

// internal/user/user.go — 領域層，純業務
package user

type User struct {
    ID    string
    Email string
    Name  string
}

// Store 介面由領域層定義 — 我需要什麼能力
type Store interface {
    Get(ctx context.Context, id string) (*User, error)
    Create(ctx context.Context, u *User) error
}

type Service struct {
    store Store                  // 依賴介面，不依賴實作
}

func NewService(s Store) *Service { return &Service{store: s} }

func (s *Service) Register(ctx context.Context, email, name string) (*User, error) {
    // 業務邏輯 — 不知道 DB 是 Postgres 還是 Mongo
    u := &User{ID: newID(), Email: email, Name: name}
    return u, s.store.Create(ctx, u)
}


// internal/user/postgres.go — 基礎設施層，實作介面
package user

type PostgresStore struct{ db *sql.DB }

func (p *PostgresStore) Get(ctx context.Context, id string) (*User, error) {
    // SQL 查詢...
}
func (p *PostgresStore) Create(ctx context.Context, u *User) error {
    // INSERT...
}


// cmd/server/main.go — 組裝根（wiring root）
func main() {
    db, _ := sql.Open("postgres", dsn)
    store := &user.PostgresStore{DB: db}
    svc := user.NewService(store)
    // ... 啟動 server，注入 svc ...
}
三個好處 ① 測試簡單：塞個 fake Store 就能測 Service，不用真 DB。
② 換實作不動業務：從 Postgres 換 MongoDB，Service 一行不改。
③ 依賴倒置：基礎設施依賴業務領域，不是反過來 — 正確的依賴方向。

設定與命令列參數

flag 套件 · 內建 CLI parser

import "flag"

var (
    addr    = flag.String("addr", ":8080", "server address")
    dbURL   = flag.String("db", "", "database URL")
    debug   = flag.Bool("debug", false, "enable debug logging")
    timeout = flag.Duration("timeout", 5*time.Second, "request timeout")
)

func main() {
    flag.Parse()
    fmt.Printf("listening on %s\n", *addr)
}

// 使用：./server -addr=:9000 -debug -timeout=10s
// 自動產生 -h / -help 說明

環境變數慣例（12-factor）

// 生產環境通常用環境變數（flag 適合 CLI）
func loadConfig() Config {
    return Config{
        Addr:    envOr("ADDR", ":8080"),
        DBURL:   envRequired("DATABASE_URL"),
        Debug:   os.Getenv("DEBUG") == "1",
    }
}

func envOr(k, def string) string {
    if v := os.Getenv(k); v != "" {
        return v
    }
    return def
}

func envRequired(k string) string {
    v := os.Getenv(k)
    if v == "" {
        log.Fatalf("missing env var %s", k)
    }
    return v
}
對照 Python Python 常用 pydantic.BaseSettings 或 python-decouple。Go 生態有 spf13/viper（功能強，也因此較重）、kelseyhightower/envconfig（struct tag 風格，簡潔）。小專案內建 flag + os.Getenv 就足夠。

日誌 · log/slog（Go 1.21+）

Go 1.21 起有官方結構化日誌套件 log/slog。第三方的 zerolog、zap 仍有效能優勢，但新專案先考慮 slog — 少一個依賴。

import "log/slog"

// 基本使用
slog.Info("server started", "addr", ":8080", "env", "prod")
slog.Error("failed to connect", "err", err, "retry", 3)

// JSON 輸出（生產環境慣用）
func main() {
    logger := slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
        Level: slog.LevelInfo,
    }))
    slog.SetDefault(logger)
    // ... 後面用 slog.Info(...) 都會走 JSON
}

// 帶 context（把 request ID 等附上）
logger := slog.Default().With("request_id", reqID, "user_id", userID)
logger.Info("processing")
// 輸出包含 request_id 和 user_id 欄位

Makefile · 統一開發者指令

Go 不需要 package.json 的 scripts — 但有個 Makefile 能讓團隊成員不用記 10 組不同的 go 指令。

.PHONY: build test lint run clean

BIN_DIR := ./bin
APP := $(BIN_DIR)/server

build:
	@mkdir -p $(BIN_DIR)
	go build -o $(APP) ./cmd/server

test:
	go test -race -cover ./...

lint:
	golangci-lint run ./...

fmt:
	go fmt ./...
	gofmt -s -w .

tidy:
	go mod tidy
	go mod verify

vuln:
	govulncheck ./...

run: build
	$(APP)

ci: fmt lint vuln test build

clean:
	rm -rf $(BIN_DIR)

實戰模板：完整的新專案起手式

TEMPLATE · 新 Web 服務的 Day 1 結構 
# 1. 建立 module
mkdir myapp && cd myapp
go mod init github.com/myorg/myapp

# 2. 最小結構
mkdir -p cmd/server internal/httpapi

# 3. cmd/server/main.go
cat > cmd/server/main.go << 'EOF'
package main

import (
    "context"
    "log/slog"
    "net/http"
    "os"
    "os/signal"
    "syscall"
    "time"

    "github.com/myorg/myapp/internal/httpapi"
)

func main() {
    slog.SetDefault(slog.New(slog.NewJSONHandler(os.Stdout, nil)))

    addr := envOr("ADDR", ":8080")
    mux := http.NewServeMux()
    httpapi.Register(mux)

    srv := &http.Server{
        Addr:         addr,
        Handler:      mux,
        ReadTimeout:  5 * time.Second,
        WriteTimeout: 10 * time.Second,
        IdleTimeout:  60 * time.Second,
    }

    go func() {
        slog.Info("listening", "addr", addr)
        if err := srv.ListenAndServe(); err != http.ErrServerClosed {
            slog.Error("server error", "err", err)
            os.Exit(1)
        }
    }()

    ctx, stop := signal.NotifyContext(context.Background(),
        syscall.SIGINT, syscall.SIGTERM)
    defer stop()
    <-ctx.Done()

    slog.Info("shutting down")
    sctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()
    srv.Shutdown(sctx)
}

func envOr(k, def string) string {
    if v := os.Getenv(k); v != "" {
        return v
    }
    return def
}
EOF

# 4. internal/httpapi/routes.go
cat > internal/httpapi/routes.go << 'EOF'
package httpapi

import "net/http"

func Register(mux *http.ServeMux) {
    mux.HandleFunc("GET /healthz", func(w http.ResponseWriter, r *http.Request) {
        w.Write([]byte("ok"))
    })
}
EOF

# 5. 跑起來
go run ./cmd/server
# 另一個 terminal: curl localhost:8080/healthz

本章速記

  1. package 名要有語意：不要 utils / helpers / common。
  2. package 命名不重複：避免 user.UserService 這種 stutter。
  3. 扁平優先：小專案不要急著分資料夾。
  4. 預設放 internal/：要公開再搬出來。
  5. cmd/xxx 放可執行檔：每個子目錄一個 package main。
  6. pkg/ 通常不需要：除非明確要發 client SDK。
  7. 依賴單向流動：基礎設施 → 領域，不是反過來。
  8. 介面由消費端定義：Store 介面在 user 內定義，不在 db/ 內定義。
  9. main.go 是 wiring root：組裝所有具體依賴。
  10. flag 給 CLI，env 給服務：12-factor 慣例。
  11. log/slog 是官方結構化日誌：新專案優先用。
  12. Makefile 統一團隊指令：build / test / lint / run / ci。
  13. golang-standards/project-layout 是參考，不是聖經：別照抄大專案結構到你的小工具。
