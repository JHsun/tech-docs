效能 & 診斷

Go 內建一整套生產級觀察工具：pprof、trace、runtime metrics、race detector。Go 1.26 全面啟用 Green Tea GC，把 GC 成本降低 10-40%，不需任何 code 改動。

效能優化的三條原則

Go 的哲學：correctness first, then measure, then optimize。沒有測量就優化的東西多半只是加了技術債。

① 先量測再優化

Donald Knuth 那句「過早優化是萬惡之源」在 Go 社群是真的信條。一切從 pprof 開始。

② 別相信微基準

benchmark 裡 faster 的寫法，在真實系統可能因為 cache、branch prediction 表現完全不同。拿真實 workload 驗證。

③ 記憶體是一切

Go 效能八成的敵人是分配次數（allocs/op）。能在 stack 上的就別上 heap，能重用的就別新建。

④ 用 go test -bench -benchmem

數字騙不了人。優化前後各跑一次，看 ns/op、B/op、allocs/op 三個指標同時下降才算成功。

pprof · Go 的效能顯微鏡

pprof 可以產生六種 profile：CPU、heap、goroutine、mutex、block、thread。通常從 CPU 和 heap 開始看。

從 benchmark 取 profile

# 跑 benchmark 同時取 CPU profile
$ go test -bench=. -cpuprofile=cpu.out

# 順便取 memory profile
$ go test -bench=. -cpuprofile=cpu.out -memprofile=mem.out

# 互動式分析
$ go tool pprof cpu.out
(pprof) top10                  # 看最耗 CPU 的 10 個函式
(pprof) list myFunc            # 看 myFunc 原始碼逐行耗時
(pprof) web                    # 產生 SVG 火焰圖在瀏覽器開

# 或直接在瀏覽器分析（更好用）
$ go tool pprof -http=:6060 cpu.out

從活著的服務取 profile（net/http/pprof）

// 在服務中加入（通常放 internal/ 的 debug 套件）
import _ "net/http/pprof"   // 匿名 import — 註冊 /debug/pprof/* routes

// 生產服務通常另開一個 admin port
go func() {
    log.Println(http.ListenAndServe("localhost:6060", nil))
}()

// 存取端點：
// http://localhost:6060/debug/pprof/         // 列表
// http://localhost:6060/debug/pprof/heap     // heap snapshot
// http://localhost:6060/debug/pprof/goroutine // 所有 goroutine stack
// http://localhost:6060/debug/pprof/profile?seconds=30 // 30 秒 CPU profile

# 直接從服務拉 profile 分析
$ go tool pprof -http=:9999 http://localhost:6060/debug/pprof/heap
$ go tool pprof -http=:9999 http://localhost:6060/debug/pprof/profile?seconds=30
生產環境暴露 pprof 要小心 /debug/pprof/* 會洩漏程式結構，也會占 CPU（profile 本身有成本）。絕對不要暴露到公網 — 通常做法：另開一個 admin port 只綁 localhost 或內網，用 bastion / port-forward 連入。

記憶體分析 · 抓出不必要的配置

# 查看 heap profile（當下記憶體分佈）
$ go tool pprof -http=:9999 http://localhost:6060/debug/pprof/heap

# 切換四種視圖：
# inuse_space   — 當下使用的記憶體（預設）
# inuse_objects — 當下物件數量
# alloc_space   — 累計分配的總量（即使已釋放）
# alloc_objects — 累計分配次數（這個最有用）

# alloc_objects 能找出「分配很多、但很快釋放」的熱點 — GC 壓力的主因

常見的分配熱點

// 1. 字串拼接用 += 而非 Builder
// ❌ 每次 += 新增一個字串
s := ""
for _, w := range words { s += w }

// ✅ 一次分配
var b strings.Builder
for _, w := range words { b.WriteString(w) }

// 2. slice 擴容 — 預先 make 好容量
// ❌ 不知道容量
var out []Result
for _, v := range input { out = append(out, process(v)) }

// ✅ 預先分配
out := make([]Result, 0, len(input))
for _, v := range input { out = append(out, process(v)) }

// 3. interface 裝箱 — small int/string 裝進 any 會上 heap
func Println(args ...any) { ... }   // 每個非指標 arg 都可能分配

// 4. 逃逸 — 某些寫法會讓變數從 stack 「逃」到 heap
$ go build -gcflags="-m" ./...    # 顯示逃逸分析結果

sync.Pool · 重用物件

var bufPool = sync.Pool{
    New: func() any {
        return new(bytes.Buffer)
    },
}

func process(data []byte) string {
    buf := bufPool.Get().(*bytes.Buffer)
    buf.Reset()              // ⚠️ 從 pool 取出要 Reset！
    defer bufPool.Put(buf)

    // ... 使用 buf 組合字串 ...
    return buf.String()
}
Pool 的使用時機 只在真的是熱點且物件建立成本高時用（大 buffer、編解碼器）。對小物件，GC 本身就很快，加 Pool 反而讓程式更複雜。先 profile 再決定。

Goroutine / Block / Mutex Profile

Goroutine profile · 抓 leak

# 看當下所有 goroutine 的 stack trace
$ curl http://localhost:6060/debug/pprof/goroutine?debug=2

# 或用 pprof 工具看聚合視圖
$ go tool pprof -http=:9999 http://localhost:6060/debug/pprof/goroutine

# 診斷 leak 的經典做法：
# 1. 啟動後等服務 warm up
# 2. 記錄當下 goroutine 數
# 3. 跑一陣子工作
# 4. 再看 goroutine 數 — 若持續上漲就是 leak
# 5. 用 pprof 看 "stuck" 的 goroutine 卡在哪個函式

Block & Mutex profile · 抓鎖競爭

// 這兩種 profile 預設關閉（有效能成本），要手動開
import "runtime"

runtime.SetBlockProfileRate(1)     // 取樣率 1 = 全記錄
runtime.SetMutexProfileFraction(1) // 每 1 次 mutex 事件記 1 次

# 之後就能取
$ go tool pprof http://localhost:6060/debug/pprof/block
$ go tool pprof http://localhost:6060/debug/pprof/mutex

# block  — 包含 channel 阻塞、select、cond.Wait
# mutex  — 純 mutex 競爭（Lock 等待時間）

runtime/trace · 時間線視角

pprof 看「哪個函式花多久」，trace 看「什麼時候發生什麼事」。想知道為什麼服務偶爾有 200ms spike，或者為什麼 GC 看起來正常但 tail latency 很差 — 要看 trace。

// 程式中採集 trace
import "runtime/trace"

f, _ := os.Create("trace.out")
trace.Start(f)
defer trace.Stop()

// ... 跑一些工作 ...

# 開啟視覺化（會啟動瀏覽器）
$ go tool trace trace.out

# 或從 net/http/pprof 端點取
$ curl http://localhost:6060/debug/pprof/trace?seconds=5 -o trace.out

trace 看得到什麼

Goroutine 調度時間線

每個 goroutine 在哪個 P (processor) 上跑、什麼時候被 preempt。

GC 事件

GC 週期、STW 停頓時長、sweep 時間。

Syscall 阻塞

哪些 goroutine 在等系統呼叫、等多久。

自訂 region / task

用 trace.WithRegion(ctx, "name", fn) 標記程式邏輯區段。

runtime/metrics · 可程式化的 runtime 指標

Go 1.16+ 的官方指標 API，取代了老舊的 runtime.ReadMemStats。適合接到 Prometheus / Datadog。

import "runtime/metrics"

// 列出所有可用指標
descs := metrics.All()
for _, d := range descs {
    fmt.Println(d.Name, d.Description)
}

// 讀特定幾個
samples := []metrics.Sample{
    {Name: "/memory/classes/heap/objects:bytes"},
    {Name: "/gc/cycles/total:gc-cycles"},
    {Name: "/sched/goroutines:goroutines"},
    {Name: "/cpu/classes/gc/total:cpu-seconds"},
}
metrics.Read(samples)
for _, s := range samples {
    switch s.Value.Kind() {
    case metrics.KindUint64:
        fmt.Printf("%s = %d\n", s.Name, s.Value.Uint64())
    case metrics.KindFloat64:
        fmt.Printf("%s = %f\n", s.Name, s.Value.Float64())
    }
}

Go 1.26：Green Tea GC · 新世代垃圾回收器

Green Tea 是 Go 1.26 預設啟用的新 GC 設計。重點改進：scanning 局部性（掃描連續的記憶體區塊而非跳躍）、減少 cache miss、更好的 SIMD 支援。官方 benchmark 顯示 GC 開銷減少 10-40%。

你需要做什麼？

什麼都不用。升級到 Go 1.26 重新編譯，就自動享受新 GC。API 完全相容。

哪些 workload 改進最明顯？

大量短生命週期物件的服務（API server、RPC handler）、record-heavy 的資料處理，都能明顯看到 P99 latency 改善。

GOGC 怎麼調？

預設 GOGC=100（heap 翻倍就 GC）。記憶體充裕環境可以調大（150 / 200）減少 GC 頻率；資源吃緊環境調小。

GOMEMLIMIT（Go 1.19+）

設定「軟性記憶體上限」：GOMEMLIMIT=4GiB。接近上限時 GC 會更積極，避免 OOM。容器化部署很有用。

# 生產環境建議
$ export GOGC=100                 # 預設值
$ export GOMEMLIMIT=3750MiB        # 留 25% 給 non-Go 記憶體（system、container 開銷）

# Go 1.25+ 還會自動偵測 container CPU quota 設定 GOMAXPROCS
# 以前要手動 GOMAXPROCS=$(nproc)，現在可以直接移除

常見優化模式速查

症狀常見解法
字串拼接熱點strings.Builder 或 bytes.Buffer
slice append 頻繁擴容預先 make(..., 0, N)
map 頻繁擴容make(map[K]V, N)
json 序列化慢先確認，真慢才考慮 sonic / easyjson
regex 熱點MustCompile 移到 package 層級變數
fmt.Sprintf 頻繁呼叫改 strconv.*；真不行用 bytes.Buffer + Fprintf
GC 壓力大sync.Pool 重用大物件；減少 interface 裝箱
goroutine 爆炸Worker Pool；semaphore（buffered chan）限制並發
tail latency 高看 trace 找 GC / syscall 阻塞
DB 連線慢調整 SetMaxOpenConns / SetMaxIdleConns / SetConnMaxLifetime
HTTP client 慢重用 http.Client；調整 Transport 的 MaxIdleConns*
別為了效能犧牲可讀性 Go 的標語之一：clear is better than clever。只在 pprof 顯示某段是熱點（占比 >= 5% CPU 或 allocs）時才優化。如果整個熱路徑已經清晰明瞭，硬優化通常得不償失。

最後一哩的微優化（極少用到）

// 1. 避免 interface 裝箱
// 熱路徑傳具體型別，不要 any，除非真的必要

// 2. 避免 defer 在熱路徑（Go 1.14 之後差距已縮到幾 ns，通常無所謂）
// 但超熱的 inner loop 還是會差：
func hot() {
    mu.Lock()
    // ... 只是簡單計算 ...
    mu.Unlock()     // 直接 Unlock 比 defer 快一點點
}

// 3. inline hint（Go 沒有 #[inline]）
// 編譯器自己決定。想看有沒有被 inline：
$ go build -gcflags="-m=2" ./...

// 4. 切片邊界檢查消除（BCE）
for i := 0; i < len(s); i++ {
    _ = s[i]           // 編譯器能證明安全，不插 bounds check
}

// 5. 對 []byte 避免 string() 轉換（分配）
// Go 1.20+ 有 unsafe.String 做零拷貝 — 但只在證實為瓶頸時用

實戰模板：完整效能調優工作流

TEMPLATE · 診斷慢 endpoint 的 SOP 
# Step 1: 量測基線（生產或 staging）
curl -w "@curl-format.txt" -o /dev/null -s http://localhost:8080/slow
# 假設 P50 = 50ms, P99 = 500ms

# Step 2: 壓測下取 CPU profile（30 秒）
go tool pprof -http=:9999 http://localhost:6060/debug/pprof/profile?seconds=30

# Step 3: 看 top 10 — 找最耗 CPU 的函式
(pprof) top10
# 常見發現：JSON 編碼、regex match、DB driver encode...

# Step 4: 看具體行（list）或火焰圖（web/flamegraph）
(pprof) list mySlowFunc

# Step 5: 取 heap profile 看記憶體分配
go tool pprof -http=:9999 http://localhost:6060/debug/pprof/heap
# 切到 alloc_objects 視圖 — 分配最多次數的函式

# Step 6: 優化 + 寫 benchmark 驗證
cat > bench_test.go << 'EOF'
func BenchmarkMyFunc(b *testing.B) {
    for b.Loop() {
        MyFunc(testData)
    }
}
EOF

# 優化前跑 10 次
go test -bench=BenchmarkMyFunc -benchmem -count=10 > before.txt

# 優化...

# 優化後跑 10 次
go test -bench=BenchmarkMyFunc -benchmem -count=10 > after.txt

# 統計比較
benchstat before.txt after.txt
# 看 ns/op、B/op、allocs/op 三項 delta 是否為負且有顯著性

# Step 7: deploy 後再量測一次真實 P50 / P99

本章速記

  1. 先量測再優化：沒看過 pprof 的優化都是亂猜。
  2. benchmark 加 -benchmem：看 ns/op、B/op、allocs/op 三指標。
  3. benchstat 看顯著性：別靠單次數字做決定。
  4. net/http/pprof 一行開啟：import _ "net/http/pprof"。
  5. pprof 別暴露公網：只綁 localhost 或 admin port。
  6. heap 看 alloc_objects：找分配次數的大戶。
  7. goroutine 持續增加 = leak：用 goroutine profile 找卡點。
  8. block & mutex 預設關：要調用 SetRate 打開。
  9. trace 看時間線：pprof 看不到的 GC 停頓、syscall 阻塞。
  10. runtime/metrics 取代 ReadMemStats：接監控系統用。
  11. Go 1.26 Green Tea GC：升級即享受 10-40% GC 改善。
  12. GOMEMLIMIT 設軟性上限：容器部署必設。
  13. 常見優化就那幾個：Builder、預分配、Pool、重用 Client。
  14. Clear > Clever：沒 profile 數字支撐就別優化。
