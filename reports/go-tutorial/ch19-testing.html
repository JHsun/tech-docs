測試 Testing

Go 的測試框架是內建的 — 不用 pytest、不用 jest、不用 JUnit。go test 一行指令搞定單元測試、效能測試、Fuzzing，還有 Go 1.25 新增的 testing/synctest。

套件 · testing · testing/synctest · net/http/httptest 指令 · go test · -race · -cover · -bench

測試的三條規則

Go 測試架構極簡：檔名、函式名、參數都照慣例寫，其他就是普通 Go 程式碼。

  1. 檔名以 _test.go 結尾：math.go 的測試叫 math_test.go。
  2. 函式以 Test 開頭：func TestAdd(t *testing.T) { ... }。
  3. 測試放在相同 package，或同目錄下的 package xxx_test（黑箱測試）。
// math.go
package math
func Add(a, b int) int { return a + b }

// math_test.go
package math

import "testing"

func TestAdd(t *testing.T) {
    got := Add(2, 3)
    if got != 5 {
        t.Errorf("Add(2,3) = %d; want 5", got)
    }
}

t.* 常用方法

方法行為
t.Error(...) / Errorf(...)記錯誤，繼續跑
t.Fatal(...) / Fatalf(...)記錯誤，立即結束本測試
t.Log(...) / Logf(...)記錄訊息（只在失敗或 -v 時顯示）
t.Skip(...) / SkipNow()跳過
t.Helper()呼叫者才是錯誤位置（自訂斷言用）
t.Cleanup(fn)測試結束後執行（LIFO）
t.TempDir()自動清理的臨時目錄
t.Setenv(k, v)設環境變數，結束自動還原
t.Parallel()標記可並行

go test 指令

# 跑當前目錄的測試
go test

# 跑整個專案的測試
go test ./...

# 顯示詳細輸出（不失敗也印 t.Log）
go test -v

# 只跑符合名稱的測試（regex）
go test -run TestAdd
go test -run "^TestUser_"

# race detector（CI 必開）
go test -race ./...

# 覆蓋率
go test -cover ./...
go test -coverprofile=c.out ./...
go tool cover -html=c.out          # 開瀏覽器看覆蓋率熱圖
go tool cover -func=c.out          # 文字報告

# 超時（預設 10 分鐘）
go test -timeout 30s

# 次數與 shuffle
go test -count=10                  # 跑 10 次（測 flaky test）
go test -shuffle=on                # 隨機順序（揪出測試間依賴）

# 關閉測試快取
go test -count=1 ./...

Table-Driven Test · Go 的招牌測試模式

把每個測試案例寫成一筆資料，跑同一段邏輯 — 加案例變成加一行，不是複製貼上一整個函式。

func TestAdd(t *testing.T) {
    tests := []struct {
        name    string
        a, b    int
        want    int
    }{
        {"positive", 2, 3, 5},
        {"zero", 0, 0, 0},
        {"negative", -1, -2, -3},
        {"mixed", 5, -3, 2},
    }

    for _, tc := range tests {
        t.Run(tc.name, func(t *testing.T) {
            got := Add(tc.a, tc.b)
            if got != tc.want {
                t.Errorf("Add(%d, %d) = %d; want %d",
                    tc.a, tc.b, got, tc.want)
            }
        })
    }
}
subtest 的三個好處 ① 單獨執行：go test -run TestAdd/positive 只跑一個 case。
② 失敗報告清楚：看到「TestAdd/negative FAIL」比看到 "case 3 failed" 有用多了。
③ 可獨立平行：每個 subtest 可以 t.Parallel()。

並行測試

func TestSomething(t *testing.T) {
    t.Parallel()            // 標記此測試可與其他並行
    // ... 測試內容 ...
}

// Table-driven 中每個 subtest 並行
func TestParallel(t *testing.T) {
    tests := []struct{...}{...}

    for _, tc := range tests {
        t.Run(tc.name, func(t *testing.T) {
            t.Parallel()    // 每個 subtest 都並行
            // Go 1.22+ 的 loopvar 改動讓 tc 不再需要手動複製
            // ...
        })
    }
}
並行測試注意事項 ① 測試不能共享可變狀態（全域變數、檔案、DB）。
② t.Setenv 的測試不能 Parallel（會改全域 env）— Go 會直接報錯。
③ 如果每個測試需要獨立的目錄、port、DB schema，用 t.TempDir()、動態 port（:0）、每測試一個 schema。

testing/synctest · Go 1.25 測試時間相關的程式

涉及時間、timeout、goroutine 同步的測試以前要嘛跑很慢（真的 sleep），要嘛 flaky（時序不穩）。Go 1.25 正式發布的 testing/synctest 提供虛擬時鐘和 goroutine 同步等待，測試既快又穩定。

import (
    "testing"
    "testing/synctest"
    "time"
)

func TestTimeout(t *testing.T) {
    synctest.Run(func() {
        var result int
        done := make(chan struct{})

        go func() {
            time.Sleep(5 * time.Second)  // 虛擬時間，瞬間完成
            result = 42
            close(done)
        }()

        synctest.Wait()      // 等候氣泡內所有 goroutine 進入阻塞

        <-done
        if result != 42 {
            t.Errorf("got %d", result)
        }
    })
}
核心概念：fake clock + bubble synctest.Run 建立一個「氣泡」，裡面的時間不是真的 — 只有氣泡內所有 goroutine 都阻塞時，時間才推進到「下一個最早會喚醒某 goroutine 的時刻」。寫過 Jest 的 jest.useFakeTimers() 會覺得很熟。

net/http/httptest · 測 HTTP handler

import (
    "net/http"
    "net/http/httptest"
    "testing"
)

// 測 handler — 用 ResponseRecorder，不用啟動真 server
func TestHelloHandler(t *testing.T) {
    req := httptest.NewRequest("GET", "/hello?name=Alice", nil)
    rec := httptest.NewRecorder()

    helloHandler(rec, req)

    if rec.Code != http.StatusOK {
        t.Errorf("status = %d", rec.Code)
    }
    if body := rec.Body.String(); body != "Hello, Alice!" {
        t.Errorf("body = %q", body)
    }
}

// 測 client — 啟動真 server 但在 localhost
func TestClient(t *testing.T) {
    // 建立一個假的 server
    srv := httptest.NewServer(http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        w.Write([]byte(`{"ok": true}`))
    }))
    defer srv.Close()

    // srv.URL 是隨機 port 的 localhost URL
    client := NewAPIClient(srv.URL)
    result, err := client.Fetch()
    // ...
}

Fuzzing · 模糊測試（Go 1.18+）

Go 內建 fuzzing — 自動產生輸入找 bug，特別適合 parser、decoder、URL 處理這類吃「不可信輸入」的程式。

// 函式名以 Fuzz 開頭，參數 *testing.F
func FuzzReverse(f *testing.F) {
    // 初始 corpus（「種子」案例）
    f.Add("hello")
    f.Add("你好")
    f.Add("")

    // Fuzz function — Go 會生成大量變形的輸入
    f.Fuzz(func(t *testing.T, orig string) {
        rev := Reverse(orig)
        doubleRev := Reverse(rev)
        if orig != doubleRev {
            t.Errorf("Reverse(Reverse(%q)) = %q", orig, doubleRev)
        }
        if utf8.ValidString(orig) && !utf8.ValidString(rev) {
            t.Errorf("Reverse produced invalid UTF-8: %q", rev)
        }
    })
}
# 執行 fuzzing（-fuzz 才會進入 fuzz mode）
go test -fuzz=FuzzReverse -fuzztime=30s

# 找到 bug 會自動存到 testdata/fuzz/FuzzReverse/ — 之後的 go test 會把這些當 regression 測試
Fuzzing 的三種經典屬性（property） ① Round-trip：Encode→Decode 應該等於原值。
② 不 panic：任何輸入都不該 crash。
③ 不變量：某些性質必然成立（例如排序後是遞增）。

Example · 可執行的文件

以 Example 開頭的函式同時是「測試」與「文件」 — 顯示在 pkg.go.dev 的 Examples 區塊，也會真的執行驗證輸出。

// 範例會出現在套件的 godoc 裡
func ExampleAdd() {
    fmt.Println(Add(2, 3))
    // Output: 5   ← 這行註解是真的會被驗證的
}

// 針對特定型別/函式的範例
func ExampleUser_FullName() {
    u := User{First: "Alice", Last: "Chen"}
    fmt.Println(u.FullName())
    // Output: Alice Chen
}

// 不關心順序時用 Unordered output
func ExamplePrintMap() {
    m := map[string]int{"a": 1, "b": 2}
    PrintMap(m)
    // Unordered output:
    // a: 1
    // b: 2
}

實用測試輔助：t.Helper() 與 go-cmp

自訂斷言 — 記得 t.Helper()

// 沒 Helper：失敗訊息指向 assertEqual 內部
// 有 Helper：失敗訊息指向呼叫者 — 你想要這個
func assertEqual[T comparable](t *testing.T, got, want T) {
    t.Helper()
    if got != want {
        t.Errorf("got %v, want %v", got, want)
    }
}

func TestFoo(t *testing.T) {
    assertEqual(t, foo(), 42)  // 失敗顯示這一行
}

go-cmp · 比 reflect.DeepEqual 好用

import "github.com/google/go-cmp/cmp"

func TestUser(t *testing.T) {

Mock · 不用 mocking framework

Go 的介面是隱式實作 — 測試時直接寫「假的實作」，完全不需要 mock framework（像 Python 的 unittest.mock）。

// 產品代碼依賴介面
type Clock interface {
    Now() time.Time
}

type Service struct {
    clock Clock
}

// 測試用：手寫假 clock
type fakeClock struct{ now time.Time }
func (f *fakeClock) Now() time.Time { return f.now }

func TestService(t *testing.T) {
    svc := &Service{clock: &fakeClock{now: time.Date(2026, 1, 1, 0, 0, 0, 0, time.UTC)}}
    // ... 用 svc 做事，時間是你控制的
}

// 對於單方法介面，用 funcType 的技巧
type RoundTripFunc func(*http.Request) (*http.Response, error)

func (f RoundTripFunc) RoundTrip(r *http.Request) (*http.Response, error) {
    return f(r)
}

// 使用
client := &http.Client{
    Transport: RoundTripFunc(func(r *http.Request) (*http.Response, error) {
        return &http.Response{StatusCode: 200, Body: io.NopCloser(strings.NewReader(`{"ok": true}`))}, nil
    }),
}

TEMPLATE · Golden File 測試 
// 大量複雜輸出的測試（HTML render、JSON 序列化、Protobuf 編碼）
// 寫一次期望值存檔，之後比對
var update = flag.Bool("update", false, "update golden files")

func TestRender(t *testing.T) {
    got := Render(input)
    golden := filepath.Join("testdata", "render.golden")

    if *update {
        os.WriteFile(golden, []byte(got), 0644)
        return
    }

    want, err := os.ReadFile(golden)
    if err != nil {
        t.Fatal(err)
    }
    if got != string(want) {
        t.Errorf("output mismatch, run `go test -update` if intended")
    }
}

// 第一次：go test -update（產生期望檔）
// 之後：  go test（正常比對）
// 改輸出格式後：go test -update 重新產生，再 diff 看是否如預期

本章速記

  1. _test.go、TestXxx、*testing.T：三條規則，其他都是 Go。
  2. Error 繼續、Fatal 中斷：斷言失敗用哪個想清楚。
  3. Table-driven + t.Run 是標配：失敗報告好看、可單獨跑。
  4. t.Parallel() 讓測試並行；注意共享狀態。
  5. t.Cleanup、t.TempDir、t.Setenv：現代 Go 測試的三個好朋友。
  6. t.Helper()：寫自訂斷言必加，失敗訊息才指對地方。
  7. -race 必跑：CI 每次都開。
  8. -count=1 強制重跑：避開測試快取。
  9. -shuffle=on 找隱藏依賴：測試順序不該影響結果。
  10. testing/synctest（Go 1.25）：時間相關測試的救星。
  11. httptest.NewRecorder / NewServer：HTTP 測試標配。
  12. b.Loop()（Go 1.24+）：取代 for i := 0; i < b.N; i++。
  13. benchstat：比較效能用統計顯著性，不看單次數字。
  14. Fuzzing 有三個屬性：round-trip、no-panic、不變量。
  15. Example 既是測試也是文件：Output 註解會被驗證。
  16. Go 不用 mock framework：手寫 fake，或用 funcType 的 adapter。
  17. go-cmp.Diff 是 DeepEqual 的升級版：測試可讀性差別巨大。
