Module & 依賴管理

Go module 系統是 Go 1.11+ 的標配 — go.mod、go.sum、語意化版本、最小版本選擇。沒有 requirements.txt、沒有 package.json，只有兩個檔案。

核心檔案 · go.mod · go.sum 關鍵指令 · go mod · go get · go work

go.mod · 專案的身份證

每個 Go 專案的根目錄有一個 go.mod。它定義三件事：模組路徑、Go 版本、依賴清單。

// 建立新專案
$ go mod init github.com/myorg/myapp

// 產生的 go.mod
module github.com/myorg/myapp

go 1.26                                  // 最低 Go 版本

toolchain go1.26.2                       // 開發使用的 toolchain（可選）

require (
    github.com/google/go-cmp v0.6.0
    golang.org/x/time v0.5.0
)

require (
    github.com/kr/pretty v0.3.1 // indirect  // 間接依賴
)

go.mod 每個指令的意義

指令作用
module宣告這個 module 的完整路徑（import 時會用）
go 1.26語言版本（會啟用對應版本的語法與 std 行為）
toolchain go1.26.2實際用哪個 toolchain 編譯（Go 1.21+）
require直接或間接依賴
replace把某依賴替換成本地路徑或 fork（debug 神器）
exclude排除特定版本（避開有 bug 的 release）
retract(套件作者專用) 撤回已發布的版本
ignore (Go 1.25+)排除特定目錄不被視為 module 一部分

go.sum · 加密鎖檔

go.sum 記錄每個依賴的 SHA-256 雜湊值。任何依賴的 bytes 被改過（即使是「同版本」重新發布），編譯會立刻失敗。這是 Go 供應鏈安全的核心。

// go.sum 片段（自動產生、自動維護）
github.com/google/go-cmp v0.6.0 h1:ofyhxvXcZhMsU5ulbFiLKl/XBFqE1GSq7atu8tAmTRI=
github.com/google/go-cmp v0.6.0/go.mod h1:17dUlkBOakJ0+DkrSSNjCkIjxS6bF9zb3elmeNeuzSs=

// 每個依賴有兩行：
// 第一行 h1:   整個 module 的 hash
// 第二行 /go.mod h1:  該 module 的 go.mod 檔案 hash
go.sum 必須 commit 進版控 和 Python 的 requirements.txt 或 JS 的 package-lock.json 一樣 — 不 commit 就失去重現性與安全性。CI 跑 go mod verify 確認沒被篡改。

Go checksum database · 全球驗證服務

// Go 預設啟用 sum.golang.org — 全球公開的 checksum 資料庫
// go get 時會把本地 hash 和 sum.golang.org 的紀錄比對
// 確保整個生態拿到的是「同一份 bytes」

// 企業內網環境可關閉：
$ go env -w GOSUMDB=off
// 或走自家 proxy：
$ go env -w GOPROXY=https://goproxy.internal.company.com

依賴管理常用指令

# 新增／升級依賴
go get github.com/foo/bar              # 抓 latest
go get github.com/foo/bar@v1.2.3       # 指定版本
go get github.com/foo/bar@latest       # 顯式 latest
go get github.com/foo/bar@main         # 某 branch 最新 commit
go get github.com/foo/bar@abcd1234     # 特定 commit hash

# 移除不再使用的依賴（從 import 中消失後呼叫）
go mod tidy

# 降級／移除
go get github.com/foo/bar@none         # 完全移除

# 升級整個專案所有依賴到最新次版本
go get -u ./...

# 升級到最新主版本（慎用，可能有 breaking change）
go get -u=patch ./...                  # 只升補丁版本

# 檢視
go list -m all                         # 列出所有依賴
go list -m -u all                      # 順便檢查有沒有更新
go list -m -versions github.com/foo/bar  # 列出該套件所有版本

# 驗證
go mod verify                          # 確認 go.sum 中的 hash 全部正確
go mod download                        # 把依賴抓到 module cache（不編譯）

# 檢視為什麼某個套件在依賴樹裡
go mod why github.com/foo/bar
go mod graph                           # 完整依賴圖

版本規則：SemVer + Major Version Suffix

Go 要求套件遵守語意化版本，且對主要版本 ≥ 2 有特殊規則 — 路徑要加 /v2、/v3。

// 版本號格式：vMAJOR.MINOR.PATCH
// v1.x.x — 穩定版
// v0.x.x — 開發中，任何 minor bump 都可能 breaking

// v2+ 的 import 路徑要加後綴
import "github.com/foo/bar/v2"    // 取 v2.x.x
import "github.com/foo/bar/v3"    // 取 v3.x.x

// 這讓 v1 和 v2 可以在同一個 binary 中共存（不同 package）

最小版本選擇（MVS）

// Go 的 dependency resolution 算法獨特：
// 「使用所有 require 中的最小版本（但不小於 require 指定的）」

// 範例：
//   我的 app 寫 require foo v1.2.0
//   依賴 X 寫 require foo v1.3.0
//   依賴 Y 寫 require foo v1.4.0
// → 最終選擇 v1.4.0（所有 require 中的最大值）

// 與 npm/pip 的差別：npm 會選 latest 符合 semver 的版本
//                   Go 永遠選最保守的 — 你寫的版本就是你拿到的版本
MVS 的好處 可重現性 — 今天跑 go build 和三年後跑，只要 go.mod 不變、版本都還在 proxy，結果完全一樣。不會有「昨天好好的，今天套件作者發新版就壞了」的情況。

replace 與 vendor · 工程必殺技

replace · 把依賴指向別的來源

// 指向本地路徑（debug 最常用）
replace github.com/foo/bar => ../bar

// 指向自己的 fork（等 PR 合併前的 workaround）
replace github.com/foo/bar => github.com/myorg/bar v1.2.4-custom

// 指向特定 commit
replace github.com/foo/bar => github.com/myorg/bar v0.0.0-20260101000000-abc123def456
replace 不會傳遞 你的 app 的 replace 只對自己有效 — 若 app 被別人當依賴，對方看不到你的 replace。這設計避免「供應鏈被依賴綁架」，但也代表 fork 最終還是要自己發版本。

vendor · 把依賴全部拷到本地

$ go mod vendor           # 建立 vendor/ 目錄，所有依賴原始碼拷進去
$ go build -mod=vendor    # 從 vendor 編譯（不走 module cache）

# 常見用途：
# 1. 離線 / airgap 環境
# 2. 資安稽核：要看所有第三方 code
# 3. 對依賴做 patch（直接改 vendor/ 下的檔案）

# 代價：repo 會變肥，大型專案 vendor 可能占數百 MB

Go Workspaces · 多 module 開發（Go 1.18+）

開發「互相呼叫的多個 module」時，以前只能靠 replace — 現在有 go.work。monorepo 或想同時 debug 套件 + 用它的 app 時超有用。

# 假設你在 monorepo 同時維護多個 module
# /projects/
#   libfoo/         ← github.com/myorg/libfoo
#   appbar/         ← github.com/myorg/appbar（import libfoo）

$ cd /projects
$ go work init ./libfoo ./appbar   # 產生 go.work

# go.work 內容
go 1.26
use (
    ./libfoo
    ./appbar
)

# 現在 appbar 中 import github.com/myorg/libfoo 會直接用本地版本
# 不用寫 replace，也不會污染 appbar/go.mod
go.work 不要 commit 它是「開發者自己的工作區設定」，不是專案本身的一部分。加到 .gitignore。另一個好處：每個人可以組合自己想開發的 module 子集。

Go 1.25：go.mod 的 ignore 指令

過去 ./...、go mod tidy、go vet 都會掃 module 下所有 Go 檔 — 包含 testdata/、範例程式碼、第三方產生的假 Go 檔等等。Go 1.25 新增 ignore 讓你明確排除。

module github.com/myorg/myapp

go 1.26

// Go 1.25+：排除這些目錄不被 Go 工具視為 module 一部分
ignore (
    ./testdata
    ./examples/broken
    ./third_party/generated
)

require (
    ...
)

什麼時候會用到？

測試資料

testdata/ 裡如果有 fuzzing 產生的假 Go 檔，go vet 會報錯。加入 ignore 清單。

教材或範例

專案 repo 中示範錯誤用法的 Go 檔 — 你故意要它編不過，但 ./... 會掃到。

第三方產物

vendored 產生器的輸出、語言綁定工具產生的暫存 Go 檔。

Monorepo 其他 module

同 repo 下有多個獨立 module 時，讓各自的 ./... 不互相打擾。

發布一個 Go 套件

Go 沒有 npmjs 或 PyPI — 你的 GitHub repo 就是「套件倉庫」。

  1. 寫程式碼：確保 module 路徑對應你的 repo URL（例如 github.com/myorg/mylib）。
  2. 推上 GitHub/GitLab/etc：Go 會透過 HTTPS + ?go-get=1 協議找原始碼。
  3. 打 tag：git tag v1.0.0 && git push --tags。Go 看 git tag 來識別版本。
  4. 完成：任何人都可以 go get github.com/myorg/mylib@v1.0.0。首次下載時會自動送到 sum.golang.org 建立 checksum。

pkg.go.dev 的文件

// pkg.go.dev 會自動為公開 repo 產生文件頁面
// 想加速可以手動訪問：
// https://pkg.go.dev/github.com/myorg/mylib
// 第一次訪問觸發索引，之後幾分鐘內全文可搜尋
破壞相容怎麼辦？ 想改 API？必須 bump major version：v2.0.0 並且在 repo 內建立 v2/ 子目錄（或 v2 branch），go.mod 的 module 路徑變成 github.com/myorg/mylib/v2。老使用者繼續用 v1，新使用者自己決定何時遷移。

依賴安全性

# 漏洞掃描 — Go 官方工具
go install golang.org/x/vuln/cmd/govulncheck@latest

# 對目前專案掃
govulncheck ./...

# 特色：只報「你實際有呼叫到的」漏洞
# — 不是「依賴樹裡有」就警告（避免 false positive）
# 資料來源：https://vuln.go.dev

# CI 整合範例（GitHub Actions）：
- name: Vulnerability check
  run: |
    go install golang.org/x/vuln/cmd/govulncheck@latest
    govulncheck ./...
定期維護三件事 ① 每月跑 go get -u ./... && go mod tidy 升級小版本。
② CI 每次跑 govulncheck。
③ 主要版本升級（/v2 → /v3）排進 sprint，看 CHANGELOG。

重要的 Go 環境變數

變數作用
GOPATH舊 GOPATH 模式留下的（module cache 存這）
GOPROXY去哪下載 module（預設 proxy.golang.org）
GOSUMDB去哪驗證 checksum（預設 sum.golang.org）
GOPRIVATE私有套件路徑，不走 proxy 和 checksum db
GONOSUMCHECK不驗 checksum 的 module 前綴
GOFLAGS所有 go 指令的預設 flag
 
# 公司內部套件通常需要設定：
$ go env -w GOPRIVATE=github.com/mycompany/*

# 這讓 github.com/mycompany/* 的套件：
# ① 直接從 repo 抓，不經過 proxy.golang.org
# ② 不送 checksum 到 sum.golang.org
# ③ 依賴 git 的認證（通常 SSH key 或 personal access token）

# 查看所有環境變數
$ go env

# 永久設定
$ go env -w GOPROXY=https://proxy.golang.org,direct

本章速記

  1. go mod init 開新專案：module 路徑建議 github.com/org/name。
  2. go.mod 與 go.sum 都要 commit。
  3. go mod tidy 是日常指令：新增/刪 import 後執行。
  4. go get @version 明確升降版本；@none 移除。
  5. v2+ 要在路徑加 /v2：這是 Go 的主要版本規則。
  6. Go 用 MVS：選「所有 require 中最大的最小版本」— 保守且可重現。
  7. replace 指本地：debug 與同步開發的神器。
  8. go work 管 monorepo：不污染個別 go.mod。
  9. go.mod ignore（Go 1.25+）：排除 testdata、範例目錄。
  10. 發布套件 = 打 git tag：沒有 registry，repo 就是 registry。
  11. Breaking 必 bump major：別在 v1 偷偷改 API。
  12. govulncheck 定期跑：只報真的呼叫到的漏洞，false positive 少。
  13. GOPRIVATE 設公司 org：私有套件不外送 proxy。
