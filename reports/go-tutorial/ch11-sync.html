Sync 原語 sync · atomic

Channel 不是萬能 — 保護共享狀態、等候任務完成、讀多寫少的快取，還是 mutex/atomic 更合適。Go 1.25 的 sync.WaitGroup.Go 讓常見流程再縮短 3 行。

主要套件 · sync · sync/atomic 核心工具 · Mutex · WaitGroup · Once · atomic

何時用 channel、何時用 mutex？

Go 社群常引述 Sameer Ajmani 的建議 — 不是 channel 萬用，兩者各有戰場。

情境工具
任務分派、資料流動、串接處理channel
保護共享狀態（map、struct、快取）sync.Mutex
讀多寫少的共享狀態sync.RWMutex
等候一組 goroutine 完成sync.WaitGroup
只執行一次的初始化sync.Once
計數器、旗標sync/atomic
廣播「停止」信號close(channel) 或 context

sync.Mutex · 互斥鎖

保護共享資源最直接的工具。Mutex 的零值即可用 — 不需要 Init()。

type Counter struct {
    mu sync.Mutex       // 零值可用
    n  int
}

func (c *Counter) Inc() {
    c.mu.Lock()
    defer c.mu.Unlock()    // defer 確保 panic 也能釋放
    c.n++
}

func (c *Counter) Value() int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.n
}

RWMutex · 讀寫鎖

type Cache struct {
    mu sync.RWMutex
    m  map[string]string
}

func (c *Cache) Get(k string) string {
    c.mu.RLock()         // 讀鎖：多個讀者可以同時
    defer c.mu.RUnlock()
    return c.m[k]
}

func (c *Cache) Set(k, v string) {
    c.mu.Lock()          // 寫鎖：獨占
    defer c.mu.Unlock()
    c.m[k] = v
}
五條 mutex 金律 ① 別複製 Mutex：複製後兩個物件同時鎖沒意義。把帶 Mutex 的 struct 用指標傳遞。
② defer Unlock：除非有效能熱點，一律用 defer 避免忘記。
③ 鎖的範圍最小化：只鎖「真的需要同步」的程式段。
④ 不要在鎖內執行長時間操作（網路、檔案 I/O）。
⑤ RWMutex 不是免費午餐：少量讀者時它比 Mutex 慢；讀者明顯多於寫者才有益。

sync.WaitGroup · 等候一群 goroutine

這是取代「建一堆 channel 做 counter」的最簡方式。Go 1.25 加入的 Go 方法讓常見模式更俐落。

// 傳統寫法（現在仍能用）
var wg sync.WaitGroup
for _, url := range urls {
    wg.Add(1)
    go func(u string) {
        defer wg.Done()
        fetch(u)
    }(url)
}
wg.Wait()

// Go 1.25+ 新語法：WaitGroup.Go 一次完成 Add + go + Done
var wg sync.WaitGroup
for _, url := range urls {
    wg.Go(func() {
        fetch(url)      // Go 1.22+ 迴圈變數不再需要手動複製
    })
}
wg.Wait()
為什麼要 Add(1) 而不是 Add(len(urls))？ 兩種都對，但 Add 應該在 goroutine 啟動之前呼叫，否則可能 Wait 太早返回。最保險：進 for 之前一次 Add(len(urls))，或每次迭代都 Add(1)。千萬不要寫在 goroutine 內部！
WaitGroup 也別複製 和 Mutex 一樣，WaitGroup 內含計數器，複製後就是兩個獨立的計數器。函式傳遞一律用 *sync.WaitGroup。

sync.Once · 只執行一次

用於 lazy init、singleton 模式。Do 內的函式只會被執行一次，即使多個 goroutine 同時呼叫。

var (
    once sync.Once
    conn *DB
)

func GetDB() *DB {
    once.Do(func() {
        conn = connect("dsn://...")
    })
    return conn
}
// 無論被呼叫幾次、幾個 goroutine 同時來，connect 只會跑一次

Go 1.21+ 的進化版

// sync.OnceFunc: 包裝任意 func()
init := sync.OnceFunc(func() { fmt.Println("initialized") })
init()  // initialized
init()  // (沒輸出)

// sync.OnceValue: 延遲計算單一值
expensive := sync.OnceValue(func() int {
    return computeExpensive()
})
fmt.Println(expensive())  // 第一次計算
fmt.Println(expensive())  // 直接回傳快取結果

// sync.OnceValues: 支援回傳 (T, error)
config := sync.OnceValues(func() (*Config, error) {
    return loadConfig("config.json")
})
cfg, err := config()

sync/atomic · 無鎖原子運算

當你只是在數東西或設旗標，mutex 其實是大砲打蚊子。atomic 的原子操作利用 CPU 指令實現，比 mutex 快一個數量級。

舊 API（仍能用，但比較囉嗦）

import "sync/atomic"

var counter int64
atomic.AddInt64(&counter, 1)
v := atomic.LoadInt64(&counter)
atomic.StoreInt64(&counter, 100)
swapped := atomic.CompareAndSwapInt64(&counter, 100, 200)

新 API（Go 1.19+，強烈推薦）

// atomic.Int64 / Uint64 / Int32 / Uint32 / Bool / Pointer[T] / Value
var counter atomic.Int64

counter.Add(1)            // 等同 AddInt64
v := counter.Load()        // 等同 LoadInt64
counter.Store(100)
counter.CompareAndSwap(100, 200)

// atomic.Bool（原子旗標）
var ready atomic.Bool
if !ready.Load() { ready.Store(true) }

// atomic.Pointer[T] （泛型指標交換，無鎖資料結構的基礎）
var config atomic.Pointer[Config]
config.Store(&Config{Port: 8080})
cfg := config.Load()
什麼時候真的該用 atomic？ ① 計數器（連線數、事件次數）、② 簡單旗標（isRunning、isReady）、③ 需要無鎖熱換的資料（例如熱更新 config）。其他情境通常 Mutex 更清楚易讀。

sync.Map · 並發安全的 map

Go 內建的 map 不是並發安全的。最常見的做法是 sync.RWMutex + map；在某些特殊場景用 sync.Map 會更快。

var m sync.Map

m.Store("key", "value")              // 寫入
v, ok := m.Load("key")               // 讀取
actual, loaded := m.LoadOrStore("k", "v")  // 不存在則寫入
m.Delete("key")                      // 刪除

// 走訪（type assertion 不可避免 — sync.Map 沒用泛型）
m.Range(func(k, v any) bool {
    fmt.Println(k, v)
    return true        // return false 就停止
})
什麼時候用 sync.Map？ Go 官方文件建議的兩種場景：
① 快取：key 一旦寫入後多讀少寫。
② 多 goroutine 各自讀寫不相交的 key 子集。
其他情境 sync.RWMutex + map[K]V 通常更快（且可享受泛型）。

testing/synctest · Go 1.25 的測試利器

測試有涉及時間、timeout、goroutine 的程式一直是 Go 的痛點。Go 1.25 正式發布的 testing/synctest 套件提供「虛擬時間」與 goroutine 同步控制，讓這類測試不再需要 time.Sleep。

import (
    "testing"
    "testing/synctest"
    "time"
)

func TestTimeout(t *testing.T) {
    synctest.Run(func() {
        ch := make(chan int)

        go func() {
            time.Sleep(10 * time.Second)   // 虛擬時間，實際瞬間完成
            ch <- 42
        }()

        select {
        case v := <-ch:
            if v != 42 {
                t.Errorf("got %d", v)
            }
        case <-time.After(20 * time.Second):
            t.Fatal("timeout")
        }
    })
}
原理 synctest.Run 建立一個「氣泡（bubble）」，裡面的時間是假的；只有所有 goroutine 都阻塞時，時間才會推進到「下一個會喚醒某 goroutine 的時刻」。原本跑 30 秒的測試現在瞬間完成，且行為完全可重現。

實戰模板

TEMPLATE · 並發安全快取（sync.RWMutex + generics） 
type Cache[K comparable, V any] struct {
    mu sync.RWMutex
    m  map[K]V
}

func NewCache[K comparable, V any]() *Cache[K, V] {
    return &Cache[K, V]{m: make(map[K]V)}
}

func (c *Cache[K, V]) Get(k K) (V, bool) {
    c.mu.RLock()
    defer c.mu.RUnlock()
    v, ok := c.m[k]
    return v, ok
}

func (c *Cache[K, V]) Set(k K, v V) {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.m[k] = v
}

func (c *Cache[K, V]) Delete(k K) {
    c.mu.Lock()
    defer c.mu.Unlock()
    delete(c.m, k)
}
TEMPLATE · singleflight · 防止重複請求 
import "golang.org/x/sync/singleflight"

var group singleflight.Group

// 10 個 goroutine 同時呼叫只會觸發 1 次 fetch
func GetUser(id string) (*User, error) {
    v, err, _ := group.Do("user:"+id, func() (any, error) {
        return fetchFromDB(id)
    })
    if err != nil {
        return nil, err
    }
    return v.(*User), nil
}

本章速記

  1. channel 做資料流動，mutex 做狀態保護。
  2. Mutex / WaitGroup 的零值可用：不需要初始化。
  3. 不要複製帶鎖的 struct：用指標傳遞。
  4. defer Unlock：習慣成自然。
  5. 鎖的範圍最小化：別在鎖內執行網路、檔案 I/O。
  6. RWMutex 需確實多讀少寫：否則反而慢。
  7. WaitGroup.Add 在 goroutine 啟動前：不要寫在 goroutine 裡。
  8. Go 1.25 的 wg.Go()：省掉 Add/Done 的樣板。
  9. Once / OnceFunc / OnceValue / OnceValues：處理各種「只一次」。
  10. atomic 新 API（Go 1.19+）：用 atomic.Int64 取代 AddInt64。
  11. sync.Map 僅限兩種場景：快取 + 分離 key 集合；其他用 RWMutex+map。
  12. testing/synctest（Go 1.25）：測試含時間邏輯的並發程式神器。
  13. go test -race 必跑：它會抓出你沒想到的 race。
