CHAPTER 07 · 核心語法

指標 Pointers

Go 的指標比 C 簡單得多：沒有指標運算、沒有野指標、不用手動釋放記憶體。它唯一的任務就是回答一個問題 — 「我這個函式要不要改到呼叫端的值？」

核心運算子 · & 取址 · * 解址 關鍵決策 · 何時用指標接收器

指標的基本語法

&x 取得 x 的記憶體位址；*p 取出 p 指向的值。Python 沒有指標概念，但你每天都在用引用 — Go 只是把它顯式化了。

x := 42
p := &x               // p 的型別是 *int，指向 x 的位址

fmt.Println(p)       // 0xc00001a0a0（或類似的位址）
fmt.Println(*p)      // 42（取出 p 指向的值）

*p = 100              // 透過 p 修改 x
fmt.Println(x)       // 100

// 指標的零值是 nil
var q *int
if q == nil {
    fmt.Println("q 還沒指向任何東西")
}
// *q  ← panic: nil pointer dereference

三種建立指標的方式

// 1) 取址運算子
x := 10
p1 := &x

// 2) new() — 回傳指向零值的指標
p2 := new(int)        // p2 是 *int，*p2 == 0

// 3) Go 1.26+：new(expr)
p3 := new(42)         // *p3 == 42

// 4) struct literal + &
type User struct{ Name string }
p4 := &User{Name: "Alice"}  // *User，實務上最常用
對照 Python Python 的一般變數其實都是「引用」，但語言隱藏了這個細節。Go 讓你明確選擇：傳值（複製）或傳指標（共享）。這種顯式控制是 Go 效能可預期性的基礎。

Go 指標沒有的東西

如果你有 C/C++ 背景，先把這些從腦中刪掉；如果你是 Python 背景，本來就沒在擔心。

× 指標運算

不能 p++、p + 1。想存取陣列下一個元素請用 index，不要玩指標跳。

× 手動釋放

Go 有 GC（Green Tea GC，Go 1.26 預設啟用）。不用 free、不用 delete。

× 野指標

GC 追蹤所有引用；只要還有指標指向某物件，它就不會被回收。

× 雙重指標用於陣列

Go 只有「指向值的指標」。要動態陣列請直接用 slice，不要模仿 C 的 **char。

傳值 vs 傳指標

這是指標最實用的場景。Go 一律是傳值 — 差別只在於傳的是「值本身」還是「值的位址」。

// 傳值：函式內修改不影響原本
func incValue(x int) {
    x++
}

// 傳指標：函式內可以修改到原本
func incPointer(p *int) {
    *p++
}

n := 10
incValue(n)      // n 仍為 10
incPointer(&n)   // n 變為 11

大型 struct 幾乎都傳指標

type HugeData struct {
    Rows [10000]float64
    Meta string
}

// ❌ 每次呼叫都複製 80KB
func process(d HugeData) { ... }

// ✅ 只傳 8 bytes（指標大小）
func process(d *HugeData) { ... }

注意：這些型別本身已經是「引用」

// slice、map、channel、func 內部已經帶指標語義
// 傳它們就等同傳引用，不需要額外加 *

func modifySlice(s []int) {
    s[0] = 99        // 修改到原 slice 的底層陣列！
}

func modifyMap(m map[string]int) {
    m["x"] = 99       // 修改到原 map！
}

// 但 append 可能換底層陣列，所以:
func appendFail(s []int) {
    s = append(s, 99)  // 如果擴容，原 slice 不會看到新元素
}
// 要修改長度必須傳 *[]int 或回傳新 slice：
func appendOK(s []int) []int {
    return append(s, 99)
}
slice 的雙重性格 slice 的元素可透過傳入的 slice 修改（共享底層陣列）；但 slice 的長度只對呼叫端有效（slice header 是複製的）。要改長度請回傳新 slice。

方法接收器：值 vs 指標

Go 的方法（下一章深談）也有同樣選擇。什麼時候用指標接收器是 Go 新手最常問的問題之一。

type Counter struct{ n int }

// 值接收器：方法內的 c 是副本
func (c Counter) IncValue() {
    c.n++              // 改不到原本的 Counter！
}

// 指標接收器：方法內的 c 指向原本
func (c *Counter) Inc() {
    c.n++              // Go 自動把 c.n++ 解釋為 (*c).n++
}

ctr := &Counter{}
ctr.Inc()           // 現在 ctr.n == 1
ctr.Inc()           // 2
ctr.IncValue()      // 仍然是 2 — 值接收器沒效果

三條判斷準則

  1. 要修改欄位 → 指標接收器（否則改的是副本）。
  2. struct 很大 → 指標接收器（避免每次呼叫複製）。
  3. 一致性原則：同一型別的所有方法用同一種接收器，不要混用。指標接收器比值接收器通用，所以多數專案全用指標接收器。
// 例外：小型、不可變、類似數值語意的型別用值接收器比較好
type Point struct{ X, Y float64 }

func (p Point) Distance(q Point) float64 {
    dx, dy := p.X-q.X, p.Y-q.Y
    return math.Sqrt(dx*dx + dy*dy)
}
實務共識 初學者若不確定，一律用指標接收器就對了。真的要寫「不可變值語意」的型別（像 time.Time、Point）再切回值接收器。Go 標準庫大部分方法都是指標接收器。

自動解引用的糖衣

Go 編譯器在許多情境會自動幫你解引用，讓指標用起來像值一樣 — 但要知道哪些是糖、哪些不是。

type User struct{ Name string }

u := &User{Name: "Alice"}

// 這三種寫法等價（Go 自動幫你加 *）：
u.Name                // ✓ 最常見
(*u).Name             // ✓ 顯式解引用（通常沒必要）
(&(*u)).Name          // ✓ 但寫這樣的人請去看醫生

// 但方法呼叫對「可取址性」有要求
func (u *User) SetName(n string) { u.Name = n }

users := map[int]User{1: {Name: "A"}}
// users[1].SetName("B")  ❌ 編譯錯：map 元素不可取址

// 必須先取出修改再放回
u := users[1]
u.SetName("B")
users[1] = u

// 或從一開始就用 *User 作為 value type
users := map[int]*User{1: {Name: "A"}}
users[1].SetName("B")   // ✓ 這樣可以

指標的常見使用模式

① Optional 欄位（nil 表示「沒值」）

type User struct {
    Name     string
    Age      *int       // nil 代表「年齡未知」
    Deleted  *time.Time // nil 代表「未刪除」
}

// 設定值（Go 1.26+ 新語法最優雅）
u := User{
    Name:    "Alice",
    Age:     new(30),             // Go 1.26
    Deleted: nil,
}

// 讀取前要檢查 nil
if u.Age != nil {
    fmt.Println(*u.Age)
}

② 建構子（constructor）模式

type Server struct {
    host string
    port int
}

// Go 沒有關鍵字叫 constructor — 慣例是 NewX 函式回傳 *X
func NewServer(host string, port int) *Server {
    return &Server{
        host: host,
        port: port,
    }
}

srv := NewServer("localhost", 8080)

③ 避免大 struct 複製

// Config 可能有上百個欄位 — 全程用指標傳遞
type Config struct { ... }

func LoadConfig(path string) (*Config, error) { ... }
func ApplyConfig(cfg *Config) error { ... }

逃逸分析（Escape Analysis）

Go 編譯器決定變數放在 stack 還是 heap — 你不需要操心，但了解機制對寫出高效程式有幫助。

// 這個函式在 C 裡是大罪：回傳 stack 變數的位址
// 但 Go 完全 OK — 編譯器偵測到 x「逃逸」到 heap，自動放上去
func makeUser() *User {
    u := User{Name: "Alice"}  // 看似區域變數
    return &u                  // 但被回傳 → 編譯器把它放到 heap
}

// 查看逃逸分析結果
// $ go build -gcflags="-m" main.go
// ./main.go:10:6: moved to heap: u
效能訣竅 Heap 分配比 stack 慢，且增加 GC 壓力。熱路徑（hot path）上若看到逃逸分析報告顯示變數上了 heap，考慮：① 讓函式接受指標而非回傳指標，② 重用 sync.Pool，③ 別把介面值塞進 any。

本章速記

  1. & 取址，* 解址。指標型別寫作 *T。
  2. nil 指標解引用會 panic：使用前先檢查。
  3. 沒有指標運算：沒有 p++、沒有 p + 1。
  4. GC 自動管理記憶體：不用 free，回傳區域變數位址完全合法。
  5. slice/map/chan 本身是引用：傳它們等同傳引用，不用加 *。
  6. 改長度要回傳新 slice：s = append(s, x)，不要指望函式能幫你改。
  7. 方法接收器的選擇：要改欄位或型別很大就用指標；小型不可變值用值接收器。
  8. 一致性原則：同一型別的所有方法用同種接收器。
  9. u.Name 自動解引用：不需要寫 (*u).Name。
  10. map 元素不可取址：&m[k] 編譯錯；改存 map[K]*V。
  11. Go 1.26 的 new(expr)：寫 optional 指標欄位的神兵利器。
