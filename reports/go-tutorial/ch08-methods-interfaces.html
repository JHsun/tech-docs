CHAPTER 08 · 核心語法

方法 & 介面

Go 沒有 class、沒有繼承、沒有 implements 關鍵字 — 卻擁有比 Java 更靈活的多型。秘訣是「隱式介面實作」：只要方法簽名對得上，型別自動就實作了介面。

核心哲學 · 組合 > 繼承 關鍵字 · type / interface / any

方法 · 附加在型別上的函式

方法就是「帶接收器的函式」。任何你能 type T ... 定義的型別都可以加方法，不限 struct。

type Rectangle struct {
    Width, Height float64
}

// 值接收器：方法 Area 附加在 Rectangle 上
func (r Rectangle) Area() float64 {
    return r.Width * r.Height
}

// 指標接收器：可修改原值
func (r *Rectangle) Scale(factor float64) {
    r.Width *= factor
    r.Height *= factor
}

// 使用
r := Rectangle{Width: 3, Height: 4}
fmt.Println(r.Area())       // 12
r.Scale(2)
fmt.Println(r.Area())       // 48（r 已被修改）

對任何型別都可以加方法

// 自訂型別（underlying type 是 string）
type UserID string

func (u UserID) Valid() bool {
    return len(u) >= 3
}

// slice 也能加方法
type IntSlice []int

func (s IntSlice) Sum() int {
    total := 0
    for _, n := range s {
        total += n
    }
    return total
}

nums := IntSlice{1, 2, 3}
nums.Sum()  // 6
限制：必須定義在同一 package 不能幫「別人家的型別」加方法（例如無法直接對標準庫的 time.Time 加方法）。變通做法：自訂一個新型別把它包起來。type MyTime time.Time。

介面 · 最小共識

介面定義「一組方法」。任何型別只要實作了這些方法，就「是」這個介面 — 不需要 implements 宣告。

// 定義介面
type Shape interface {
    Area() float64
    Perimeter() float64
}

// Rectangle 實作了 Shape — 但完全不用寫 "implements Shape"
type Rectangle struct{ W, H float64 }
func (r Rectangle) Area() float64      { return r.W * r.H }
func (r Rectangle) Perimeter() float64 { return 2 * (r.W + r.H) }

// Circle 也實作了 Shape
type Circle struct{ R float64 }
func (c Circle) Area() float64      { return math.Pi * c.R * c.R }
func (c Circle) Perimeter() float64 { return 2 * math.Pi * c.R }

// 函式接受 Shape — 可以傳任何實作它的型別
func describe(s Shape) {
    fmt.Printf("面積 %.2f, 周長 %.2f\n", s.Area(), s.Perimeter())
}

describe(Rectangle{W: 3, H: 4})
describe(Circle{R: 5})
Duck Typing 強化版 Python 靠 duck typing：「走得像鴨子、叫得像鴨子，它就是鴨子」— 但只在執行時驗證。
Go 是編譯時驗證的 duck typing：簽名對就是對，對不上編譯不過。兩全其美。

Go 最常見的介面

這些介面在標準庫與第三方套件中無處不在，認得它們等於懂一半 Go。

介面方法用途
errorError() string錯誤型別
fmt.StringerString() string自訂 Println 輸出
io.ReaderRead(p []byte) (int, error)任何可讀的來源
io.WriterWrite(p []byte) (int, error)任何可寫的目標
io.CloserClose() error可關閉的資源
sort.InterfaceLen/Less/Swap自訂排序
http.HandlerServeHTTP(w, r)HTTP 處理器
json.MarshalerMarshalJSON() ([]byte, error)自訂 JSON 輸出
 
// error 只是一個單方法介面
type error interface {
    Error() string
}

// 實作自訂錯誤
type NotFoundError struct{ Key string }
func (e *NotFoundError) Error() string {
    return fmt.Sprintf("key not found: %s", e.Key)
}

// 實作 Stringer — fmt.Println 會自動呼叫
type Color int
func (c Color) String() string {
    return [...]string{"Red", "Green", "Blue"}[c]
}
fmt.Println(Color(0))  // "Red"（不是 0）

空介面 · any

不帶任何方法的介面可以放進「任何型別」。Go 1.18 起有 any 別名，比 interface{} 好看太多。

// any 就是 interface{} 的別名，建議一律用 any
var x any
x = 42
x = "hello"
x = []int{1, 2, 3}   // 什麼都能裝

// 取出需要斷言（type assertion）
var v any = "hello"
s := v.(string)          // 若不是 string 會 panic
s, ok := v.(string)      // 安全版：ok 為 false 代表失敗

// type switch — 處理多型別
func describe(v any) {
    switch x := v.(type) {
    case int:
        fmt.Println("int:", x*2)
    case string:
        fmt.Println("string:", strings.ToUpper(x))
    case []int:
        fmt.Println("slice of ints, len =", len(x))
    case nil:
        fmt.Println("nil")
    default:
        fmt.Printf("未知型別 %T\n", x)
    }
}
少用 any any 類似 Python 的 Any — 繞過型別系統。Go 社群共識：能用具體型別或泛型就不要用 any。真的要用，通常是因為要吃任意輸入（例如 fmt.Println）或解析 JSON 成未知結構。

介面組合 · 小介面拼大介面

Go 鼓勵「小介面原則」：一個介面一個方法。需要多方法時用組合，不用設計龐大的介面階層。

// io 套件的經典設計
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

type Closer interface {
    Close() error
}

// 組合小介面
type ReadWriter interface {
    Reader
    Writer
}

type ReadWriteCloser interface {
    Reader
    Writer
    Closer
}
Go 介面設計原則 Rob Pike 的名言：「介面越大，抽象越弱」。一個方法的介面經常比 10 個方法的介面更有用。定義介面時先問：消費者真的需要所有方法嗎？

介面值的內部結構

介面變數在執行時是「(型別, 值) 的配對」。理解這點能避免 nil 比較的經典陷阱。

// 介面值的兩個欄位：(type, value)
var r io.Reader
// r 現在是 (nil, nil) — 真的 nil

if r == nil {
    fmt.Println("nil")  // ✓ 會印出
}

// 但下面這個經典陷阱：
var f *os.File           // f 是 *os.File 的 nil
var r io.Reader = f       // r 現在是 (*os.File, nil)

if r == nil {
    fmt.Println("nil")  // ❌ 不會印！r 的「型別」欄位不是 nil
}
錯誤處理的經典地雷 永遠不要寫 return *MyError，請寫 return nil。因為即使 MyError 指標是 nil，包成 error 介面後就「不是 nil」了。這是每個 Go 開發者至少會被咬一次的陷阱。 
// ❌ 地雷範例
func broken() error {
    var e *MyError = nil
    return e   // 看起來回傳 nil，但呼叫端 err != nil 為 true!
}

// ✅ 正確寫法
func correct() error {
    return nil  // 直接回傳 nil，不要先包成指標
}

嵌入 · 用組合模擬繼承

把一個型別嵌進另一個 struct，它的方法會被提升到外層 — 外層就像繼承了那些方法。

type Logger struct{ Prefix string }

func (l *Logger) Log(msg string) {
    fmt.Printf("[%s] %s\n", l.Prefix, msg)
}

type Server struct {
    *Logger              // 嵌入指標型別
    Addr string
}

s := &Server{
    Logger: &Logger{Prefix: "SRV"},
    Addr:   ":8080",
}

s.Log("starting")         // ✓ Log 方法被提升
// 相當於 s.Logger.Log("starting")

// 嵌入介面也可以 — 相當於 "我保證會有這些方法"
type DebugServer struct {
    io.Reader          // 任何實作 Reader 的東西
    io.Writer
}

實戰模板

TEMPLATE · Strategy 模式（用介面實現） 
// 定義行為契約
type Storage interface {
    Save(key string, value []byte) error
    Load(key string) ([]byte, error)
}

// 三種實作：Go 不用 implements 關鍵字，只要有對應方法即可
type MemoryStorage struct {
    data map[string][]byte
}

func (m *MemoryStorage) Save(k string, v []byte) error {
    m.data[k] = v
    return nil
}

func (m *MemoryStorage) Load(k string) ([]byte, error) {
    v, ok := m.data[k]
    if !ok {
        return nil, fmt.Errorf("key %q not found", k)
    }
    return v, nil
}

type FileStorage struct{ Dir string }

func (f *FileStorage) Save(k string, v []byte) error {
    return os.WriteFile(filepath.Join(f.Dir, k), v, 0644)
}

func (f *FileStorage) Load(k string) ([]byte, error) {
    return os.ReadFile(filepath.Join(f.Dir, k))
}

// 業務層只依賴介面，不依賴實作
type Service struct {
    storage Storage
}

func NewService(s Storage) *Service {
    return &Service{storage: s}
}

// 測試時可以塞 MemoryStorage，上線時塞 FileStorage — 無縫切換
TEMPLATE · 接受介面，回傳具體型別 
// Go 格言：Accept interfaces, return structs.

// ✓ 函式參數用介面（最大彈性）
func Process(r io.Reader) (*Result, error) {
    data, err := io.ReadAll(r)
    // ...
    return &Result{...}, nil   // ← 回傳具體型別（明確的文件）
}

// 呼叫端能傳任何 Reader：檔案、網路、字串、壓縮流...
Process(os.Stdin)
Process(strings.NewReader("test"))
Process(resp.Body)

本章速記

  1. 方法 = 帶接收器的函式：附加在型別上。
  2. 任何自訂型別都能加方法：不限 struct，但必須在同 package。
  3. 介面是「方法的集合」：不列欄位、不列資料。
  4. 隱式實作：方法對得上就是實作，不用 implements。
  5. 小介面原則：io.Reader 一個方法打天下。
  6. any = interface{}：Go 1.18+ 別名，寫起來清爽，但少用。
  7. type assertion：v, ok := x.(T) 安全取出具體型別。
  8. type switch：處理多型別輸入的工具。
  9. 介面值是 (type, value) 配對：nil 陷阱的根源。
  10. 別回傳 *MyError 當 error：包進介面後就不是 nil 了。
  11. 嵌入 ≠ 繼承：方法會被提升，但沒有 override、沒有 super。
  12. Accept interfaces, return structs：寫 API 時的黃金法則。
