CHAPTER 05 · 設定與安全

Configuration & Security

同一個映像要能跑在 dev、staging、prod，設定不該烤進映像。本章帶你掌握 ConfigMap / Secret 如何做設定注入、ServiceAccount + RBAC 控制誰能做什麼、Security Context 與 Pod Security Standards 限制容器權限——這些是從「能跑」走到「能上生產」的關鍵。

設定與映像解耦：12-Factor 的核心

容器化應用的黃金原則：一份映像，多環境執行。怎麼做到？ 環境相關的參數（資料庫位址、API 金鑰、功能開關）都不要寫死在映像裡，而是從外部注入。 K8s 提供兩個官方機制：

ConfigMap

一般設定：環境變數、設定檔內容、功能開關。明文儲存在 etcd，適合非機敏資料。

Secret

機敏資料：密碼、金鑰、憑證、Token。預設 base64 編碼儲存，應搭配加密與存取限制。

ConfigMap：一般設定

ConfigMap 是一個 key-value 物件，總大小限制 1 MiB。大於這個量應該改用 Volume 或外部儲存。

建立 ConfigMap 的三種方法

Bash · 指令列建立 
# 方法 1：從字面值建立
kubectl create configmap app-config \
  --from-literal=LOG_LEVEL=info \
  --from-literal=MAX_CONNECTIONS=100

# 方法 2：從檔案建立（key 是檔名，value 是內容）
kubectl create configmap nginx-conf --from-file=nginx.conf

# 方法 3：從整個目錄
kubectl create configmap app-configs --from-file=./configs/
configmap.yaml · 聲明式 
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:                         # key-value 字串
  LOG_LEVEL: "info"
  MAX_CONNECTIONS: "100"
  application.yaml: |      # 整個檔案內容
    server:
      port: 8080
    database:
      host: db.production.svc
binaryData:                   # base64 編碼的二進位資料
  favicon.ico: "AAAB..."

ConfigMap 的三種使用方式

方式 1：作為環境變數

env-from-configmap.yaml 
spec:
  containers:
  - name: app
    image: myapp
    env:                         # 逐個引用
    - name: LOG_LEVEL
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: LOG_LEVEL
    envFrom:                     # 一次匯入全部 key 為環境變數
    - configMapRef:
        name: app-config

方式 2：掛成檔案（Volume）

當設定是一整個檔案（如 nginx.conf）時使用。檔案會隨 ConfigMap 更新自動同步（但應用需自己重新載入）。

configmap-volume.yaml 
spec:
  containers:
  - name: nginx
    image: nginx
    volumeMounts:
    - name: config
      mountPath: /etc/nginx/conf.d
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: nginx-conf
      items:                # 可選：只掛特定 key
      - key: nginx.conf
        path: default.conf

方式 3：command line 參數

args-from-configmap.yaml 
env:
- name: MAX_CONN
  valueFrom:
    configMapKeyRef:
      name: app-config
      key: MAX_CONNECTIONS
command: ["/app"]
args: ["--max-conn=$(MAX_CONN)"]
⚠️ 更新行為差異
環境變數形式的 ConfigMap 更新後，Pod 不會重新讀取，需要重啟 Pod。Volume 形式則會自動同步（檔案內容變動），但應用自己要監聽變化或被通知重新載入。用 immutable: true 可讓 ConfigMap 不可變，避免誤改。

Secret：機敏資料

Secret 與 ConfigMap 結構幾乎一樣，差別在於：

  • 儲存時 base64 編碼（注意：base64 不是加密，只是編碼）
  • etcd 可啟用靜態加密（Encryption at Rest）
  • 不會寫入 Node 的磁碟——以 tmpfs 形式掛到 Pod 內
  • 大小上限同樣 1 MiB
Bash · 建立 Secret 
# 從字面值
kubectl create secret generic db-creds \
  --from-literal=username=admin \
  --from-literal=password='S3cure!'

# 從檔案（適合整張憑證）
kubectl create secret generic tls-cert \
  --from-file=tls.crt --from-file=tls.key
secret.yaml · 聲明式 
apiVersion: v1
kind: Secret
metadata:
  name: db-creds
type: Opaque
stringData:                    # K8s 會自動 base64 編碼（寫 YAML 時友善）
  username: admin
  password: "S3cure!"
# 或用 data 欄位直接給 base64：
# data:
#   username: YWRtaW4=
#   password: UzNjdXJlIQ==

在 Pod 內引用

use-secret.yaml 
spec:
  containers:
  - name: app
    image: myapp
    env:
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-creds
          key: password
    volumeMounts:
    - name: tls
      mountPath: /etc/tls
      readOnly: true
  volumes:
  - name: tls
    secret:
      secretName: tls-cert
      defaultMode: 0400    # 檔案權限

內建 Secret 類型

type用途
Opaque（預設）任意 key-value
kubernetes.io/service-account-tokenServiceAccount Token（v1.24+ 不自動產生長期 token）
kubernetes.io/dockerconfigjson拉私有映像的 registry 憑證
kubernetes.io/tlsTLS 憑證與私鑰（Ingress、Service Mesh 常用）
kubernetes.io/basic-auth基本認證 username/password
kubernetes.io/ssh-authSSH 金鑰

建立拉私有映像用的 Secret

Bash · imagePullSecret 
kubectl create secret docker-registry regcred \
  --docker-server=registry.example.com \
  --docker-username=myuser \
  --docker-password=mypass \
  --docker-email=me@example.com

# 在 Pod spec 引用
spec:
  imagePullSecrets:
  - name: regcred
  containers:
  - name: app
    image: registry.example.com/app:1.0

外部 Secret 管理（生產建議）

原生 Secret 放在 etcd 裡，即使加密，仍有「誰有 Cluster 權限就能看到」的問題。 生產環境常見做法是把真正的機密放外部機密管理，由 Operator 同步到 K8s：

方案說明
External Secrets Operator支援 AWS Secrets Manager、GCP Secret Manager、Vault、Azure Key Vault 等
Sealed Secrets (Bitnami)加密後可安全 commit 到 Git
HashiCorp Vault + Agent InjectorPod 啟動時自動注入 Vault 秘密
Secrets Store CSI Driver把外部 Secret 以 Volume 形式掛到 Pod
💡 最佳實踐
原生 Secret 適合開發、測試、非關鍵憑證。生產環境的資料庫密碼、第三方 API 金鑰應走外部機密管理——配合 RBAC 最小權限、audit log、自動輪替。

ServiceAccount：Pod 的身份

當 Pod 內的應用需要呼叫 K8s API（查詢、建立資源），它用什麼身份？答案是 ServiceAccount（SA）。 每個 namespace 有預設的 default SA，但生產環境建議明確指定。

serviceaccount.yaml 
apiVersion: v1
kind: ServiceAccount
metadata:
  name: prometheus-sa
  namespace: monitoring
automountServiceAccountToken: true   # 預設 true；不需要可設 false
---
apiVersion: v1
kind: Pod
metadata:
  name: prometheus
spec:
  serviceAccountName: prometheus-sa  # 指定使用哪個 SA
  containers:
  - name: prometheus
    image: prom/prometheus
⚠️ 預設 SA 陷阱
不指定 serviceAccountName 時用的是 default。預設每個 Pod 會自動掛載 token——若應用不需要呼叫 K8s API，應該設 automountServiceAccountToken: false，降低被利用的風險。

RBAC：授權模型

RBAC（Role-Based Access Control）控制「誰能對哪些資源做什麼操作」。 核心是四個物件：

Subject
User
Group
ServiceAccount
→
RoleBinding
把 Subject 綁到 Role
→
Role

(verbs × resources)

兩組範圍

名稱範圍典型用途
Role單一 namespace只讓某人管理 dev namespace
ClusterRole整個 Cluster管理 Node、CRD、所有 namespace
RoleBinding綁 Role 或 ClusterRole 到某 namespace把 Cluster 級能力限制在某個 ns
ClusterRoleBinding綁 ClusterRole 到整個 Cluster全 Cluster 管理員

Role + RoleBinding 實例

role-read-pods.yaml · 只能讀 default ns 的 Pod 
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]            # core API group
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: default
subjects:
- kind: ServiceAccount
  name: debug-sa
  namespace: default
- kind: User
  name: alice@example.com
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io

常見 verb 列表

verb對應 HTTP說明
getGET（單一物件）讀取特定資源
listGET（集合）列清單
watchGET (streaming)持續監聽變化
createPOST建立
updatePUT完整替換
patchPATCH部分更新
deleteDELETE刪除
deletecollectionDELETE (集合)批次刪除
*全部危險，審慎使用

RBAC 常見組合

檢查有效權限

Bash · kubectl auth can-i 
# 我能不能建立 pod？
kubectl auth can-i create pods

# alice 能不能刪除 prod namespace 的 deployment？
kubectl auth can-i delete deployment -n prod --as=alice@example.com

# prometheus-sa 能不能列所有 namespace 的 pod？
kubectl auth can-i list pods --all-namespaces \
  --as=system:serviceaccount:monitoring:prometheus-sa

# 列出某 SA 在某 ns 所有能做的事
kubectl auth can-i --list -n default \
  --as=system:serviceaccount:default:debug-sa

K8s 內建 ClusterRole（可直接引用）

ClusterRole權限
cluster-admin超級管理員，什麼都能做
adminnamespace 內完全控制（不含 namespace 本身管理）
edit可讀寫大部分資源（不含 Role/RoleBinding）
view唯讀（不含 Secret）
🚫 避免的反模式
不要給應用綁 cluster-admin——即使開發階段方便。被入侵的 Pod 等於被入侵整個 Cluster。遵循最小權限原則（Principle of Least Privilege）：需要什麼就開什麼。

Security Context：容器執行權限

securityContext 定義 Pod 或 Container 的安全屬性，決定容器以誰的身份跑、能不能 root、能不能修改檔案系統等。

secure-pod.yaml · 生產級設定 
apiVersion: v1
kind: Pod
metadata:
  name: secure-app
spec:
  securityContext:              # Pod 層級
    runAsNonRoot: true         # 不允許 root
    runAsUser: 10001
    runAsGroup: 10001
    fsGroup: 10001              # 掛載 Volume 的群組
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: myapp:1.0
    securityContext:            # 容器層級（覆寫 Pod 層）
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]          # 丟掉所有 Linux capabilities
        add: ["NET_BIND_SERVICE"]  # 只加需要的

關鍵欄位說明

欄位功能
runAsNonRoot: true強制非 root，映像是 root 用戶會拒絕啟動
runAsUser / runAsGroup指定 UID / GID
readOnlyRootFilesystem: true容器根檔案系統唯讀（需要寫檔用 emptyDir）
allowPrivilegeEscalation: false禁止 setuid 類提權
capabilities.drop: [ALL]丟掉所有 Linux capabilities（最安全）
privileged: true容器特權模式——幾乎等同 Node 管理員，只有 CNI agent 等少數例外用
seccompProfile限制 syscall（建議 RuntimeDefault）

Pod Security Standards（PSS）

手寫 securityContext 繁瑣易錯。K8s 提供 Pod Security Standards——三個預先定義的安全等級， 搭配 Pod Security Admission（PSA）在 namespace 層級強制執行。

Privileged
完全開放

無限制。只適合 system-level workload（CNI、CSI）。應用不該用。

Baseline
防止已知提權

禁止 hostPath、privileged、hostNetwork 等危險設定，相容大部分應用。

Restricted
強化安全

強制 runAsNonRoot、capabilities 全丟、seccomp RuntimeDefault。生產建議。

在 namespace 啟用 PSA

Bash · 對 namespace 加 label 
# enforce: 違反直接拒絕；audit: 記錄到 log；warn: kubectl 警告
kubectl label namespace production \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/audit=restricted \
  pod-security.kubernetes.io/warn=restricted

# 或宣告式
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/warn: restricted
💡 升級路徑
既有 namespace 升級到 restricted 不可能一次到位。可先用 audit=restricted（只記錄）蒐集違規清單，修完再 warn，最後 enforce。

供應鏈安全

K8s 本身只能管住「Cluster 內」，但被入侵最常見的路徑是映像來源。幾個關鍵做法：

項目做法
映像掃描Trivy、Grype、Snyk：掃 CVE 與設定錯誤
鎖 tag別用 latest，用明確版本或 SHA digest
映像簽章Cosign（Sigstore 生態）驗證映像來源
Admission 驗證Kyverno、OPA Gatekeeper：阻擋不符規範的 Pod
SBOM產生 software bill of materials，追蹤相依
Registry 限制只允許從企業內部 registry 拉映像
Bash · 鎖 digest 確保不可變 
# 不好：tag 可以被 overwrite
image: nginx:1.27

# 好：用 SHA256 digest
image: nginx@sha256:fa32cac1356fe3b58e6a93d0daa52ea24b4ecb4c0fd1d48a1ac70...

本章速查

🎯 必記重點

  1. ConfigMap 放一般設定、Secret 放機敏；兩者都 1 MiB 上限。
  2. ConfigMap 三用法：env、envFrom、Volume；Volume 會自動同步但環境變數不會。
  3. Secret base64 ≠ 加密；生產用 External Secrets Operator、Sealed Secrets、Vault。
  4. ServiceAccount 是 Pod 的身份；不需 K8s API 時設 automountServiceAccountToken: false。
  5. RBAC 四元素：Subject → RoleBinding → Role → Resources/Verbs。
  6. Role 綁單一 namespace、ClusterRole 綁整 Cluster；RoleBinding 可混搭引用 ClusterRole。
  7. 遵循最小權限：不要綁 cluster-admin 給應用。
  8. Security Context：runAsNonRoot + readOnlyRootFilesystem + drop ALL capabilities 是生產三寶。
  9. PSS 三等級：Privileged / Baseline / Restricted；建議 namespace 強制 Restricted。
  10. 映像用 digest、啟用掃描、簽章、Admission 驗證守住供應鏈。
