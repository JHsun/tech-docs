§ 01 / INTRODUCTION

Helm 簡介

為何需要 Kubernetes 的 Package Manager

Helm 是 Kubernetes 的 Package Manager（套件管理器），概念上就像 Linux 世界的 apt、yum、brew，但管理的不是單一檔案，而是一整組 Kubernetes 資源。透過 Helm，你可以把原本散落在數十個 YAML 中的 Deployment、Service、ConfigMap、Ingress 等物件集中成一個稱為 Chart 的套件，一次安裝、一次升級，也可以一次回滾。

Why Helm

模板化部署

同一份 Chart 可以依環境差異（dev / staging / prod）以不同的 values 檔案渲染，避免重複撰寫 YAML。

Why Helm

版本化 Release

每次安裝或升級都會產生新的 revision，隨時可用 helm rollback 回到前一版本。

Why Helm

社群生態

Artifact Hub 上有數千個開源 Chart（Nginx、PostgreSQL、Grafana、Prometheus…）可直接使用。

Helm 從 v3 開始移除 Server 端元件 Tiller，改為純 Client-side 架構，直接以 kubeconfig 對 Kubernetes API Server 操作。權限模型完全對齊 RBAC，更為安全。Helm v4 則進一步強化模組化與 SDK 設計。
§ 02 / CORE CONCEPTS

三個必懂的名詞

Chart、Repository、Release
Chart

套件本體

一個 Chart 是一個打包好的 Kubernetes 應用程式定義，內含 templates、values、metadata。相當於 Debian 的 .deb、macOS 的 Homebrew formula。

Repository

套件倉庫

集中存放 Chart 的 HTTP(S) 伺服器或 OCI Registry。公開的例子有 Bitnami、Artifact Hub；企業內部也可自建 ChartMuseum 或使用 Harbor、GitHub Pages。

Release

執行中的實例

同一個 Chart 可以在同一個 Cluster 被安裝多次，每一次都是一個獨立的 Release，各有各的名稱、values 與歷史版本。

心智圖示意

REPO bitnami/nginx 倉庫中的 Chart
PULL nginx-15.2.tgz 下載到本機
INSTALL helm install 套用到 Cluster
RELEASE my-web-1 執行中的實例
§ 03 / INSTALLATION

安裝 Helm CLI

macOS · Linux · Windows · 腳本
macOS（Homebrew）
$ brew install helm
Windows（Chocolatey）
> choco install kubernetes-helm
Linux（官方安裝腳本）
$ curl -fsSL https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash

$ helm version

§ 04 / QUICKSTART

5 分鐘快速上手

從加入 Repo 到看到服務運行
# 1. 加入 Bitnami 倉庫並更新索引
$ helm repo add bitnami https://charts.bitnami.com/bitnami
$ helm repo update

# 2. 搜尋你要的 chart
$ helm search repo wordpress

# 3. 安裝（release 名稱 = happy-panda）
$ helm install happy-panda bitnami/wordpress

# 4. 查看狀態與資源
$ helm status happy-panda
$ helm list

# 5. 升級 / 回滾 / 卸載
$ helm upgrade happy-panda bitnami/wordpress --set service.type=LoadBalancer
$ helm rollback happy-panda 1
$ helm uninstall happy-panda
Helm 在 install 完成後就會退出，但 Pod 拉 image 可能還要時間。使用 helm status <release> 或 --wait 參數確認實際就緒狀態。
§ 05 / USING HELM

日常使用情境

搜尋、安裝、升級、回滾、卸載

搜尋 Charts

Helm 提供兩種搜尋方式：helm search repo 搜尋「本機已加入」的倉庫；helm search hub 則向 Artifact Hub 發送 API 查詢全球公開的 Chart。 

$ helm search hub wordpress       # 全網搜尋
$ helm search repo database       # 本機 repo 搜尋

安裝時覆寫 Values

有三種方式可在安裝時覆寫 values.yaml 的預設值，優先順序由高到低：--set > --values 檔案 > Chart 預設。

# 以 --values (-f) 指定設定檔（可多次）
$ helm install my-app ./mychart -f prod.yaml -f override.yaml

# 以 --set 直接於命令列覆寫
$ helm install my-app ./mychart --set replicaCount=3,image.tag=1.24

# 陣列需以大括號或反斜線
$ helm install my-app ./mychart --set tags={dev,canary}

升級與回滾

# 升級 release（可同時更改 chart 或 values）
$ helm upgrade my-app ./mychart --atomic --timeout 5m

# 升級但若尚未安裝則自動安裝
$ helm upgrade my-app ./mychart --install

# 查看歷史版本
$ helm history my-app

# 回滾到指定 revision（不加數字則回到上一版）
$ helm rollback my-app 2
--atomic：升級失敗時自動回滾；搭配 --cleanup-on-fail 可清掉升級過程中新建的資源。生產環境強烈建議配對使用。

取得 Release 資訊

  • helm get values <release> — 查看該 release 目前使用的 values。
  • helm get manifest <release> — 列出該 release 產生的所有 Kubernetes YAML。
  • helm get hooks <release> — 查看該 release 的 Hooks。
  • helm get notes <release> — 再次顯示安裝時的 NOTES.txt。
  • helm get all <release> — 一次匯出以上全部資訊。
§ 06 / CHART DEVELOPMENT

開發你自己的 Chart

結構、Chart.yaml、values.yaml

建立新 Chart 的最快方式是 helm create，它會產出一個包含所有必要檔案與範例模板的骨架。 

$ helm create mychart
$ tree mychart
mychart/
├── Chart.yaml          # 必要：定義 chart 元資料
├── LICENSE             # 選用：授權
├── README.md           # 選用：說明文件
├── values.yaml         # 必要：預設設定值
├── values.schema.json  # 選用：以 JSON Schema 驗證 values
├── .helmignore         # 打包時忽略的檔案
├── charts/             # 依賴的子 chart（可由 dependency update 自動下載）
├── crds/               # CustomResourceDefinition（裝前會先建立）
└── templates/
    ├── deployment.yaml
    ├── service.yaml
    ├── ingress.yaml
    ├── serviceaccount.yaml
    ├── hpa.yaml
    ├── _helpers.tpl    # 命名模板 / 可重用片段
    ├── NOTES.txt       # 安裝後顯示的訊息
    └── tests/           # helm test 會執行的 Pod
        └── test-connection.yaml

Chart.yaml 必備欄位

apiVersion: v2                  # Helm 3+ 使用 v2
name: mychart
description: A Helm chart for my app
type: application             # application 或 library
version: 0.1.0                 # Chart 本身的 SemVer
appVersion: "1.16.0"           # 應用程式版本（字串）
keywords:
  - nginx
  - webserver
home: https://example.com
sources:
  - https://github.com/example/mychart
maintainers:
  - name: Dev Team
    email: dev@example.com
dependencies:
  - name: postgresql
    version: "12.x.x"
    repository: https://charts.bitnami.com/bitnami
    condition: postgresql.enabled

打包、檢查、發佈

# 語法檢查 / 最佳實踐檢測
$ helm lint ./mychart

# 將 chart 打包為 tgz
$ helm package ./mychart
→ mychart-0.1.0.tgz

# 本地安裝測試
$ helm install demo ./mychart-0.1.0.tgz --dry-run --debug
§ 07 / TEMPLATE GUIDE

Template 基礎語法

Go Template + Sprig + Helm 擴充

Helm template 採用 Go text/template 語法，以 {{ }} 包圍 Action（指令），在渲染時會被計算替換。熟悉下面這幾類用法，就能看懂 95% 的 Chart。

基本替換與引用

apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ .Release.Name }}-config
data:
  app: {{ .Values.appName | quote }}
  level: {{ .Values.logLevel | default "info" }}

Built-in Objects（內建物件）

以下物件可在任何 template 直接使用，名稱以大寫開頭（Go 命名慣例）。

物件說明
.Release.NameRelease 名稱
.Release.Namespace安裝目標 Namespace
.Release.Service渲染者，永遠為 Helm
.Release.IsInstall當前操作是否為 install
.Release.IsUpgrade當前操作是否為 upgrade 或 rollback
.Release.RevisionRevision 編號（install 為 1，每次 upgrade/rollback +1）
.Values來自 values.yaml 與 --set、-f 的合併結果
.ChartChart.yaml 的內容，如 .Chart.Name、.Chart.Version
.Files存取 chart 內非 templates 的檔案
.Files.Get取得單一檔案內容為字串
.Files.Glob以 glob pattern 找檔
.Files.AsConfig把檔案們轉成 YAML map（for ConfigMap）
.Files.AsSecrets轉成 Base64（for Secret）
.CapabilitiesKubernetes 能力資訊
.Capabilities.KubeVersionKubernetes 版本
.Capabilities.APIVersionsCluster 支援的 API 版本清單
.Template.Name當前 template 的完整路徑名
.Template.BasePathtemplates 目錄路徑

空白與縮排控制

Action 前後加上 - 可消除前導或後隨空白，這對產生乾淨的 YAML 非常關鍵。

# {{- something }}   消除前面空白
# {{ something -}}   消除後面空白
# {{- something -}}  兩邊都消除

labels:
  {{- range .Values.extraLabels }}
  {{ .key }}: {{ .value | quote }}
  {{- end }}
indent 與 nindent 是產出多行 YAML 的救星：{{ .Values.labels | toYaml | nindent 4 }} 會在每行前面補 4 個空白並換行。

Values 的來源與優先序

  • Chart 的 values.yaml — 預設，被任何上層覆寫。
  • 父 Chart 的 values.yaml — 子 chart 預設可被父 chart 覆寫。
  • helm install/upgrade -f file.yaml — 可多次，後者蓋前者。
  • helm install/upgrade --set key=value — 最高優先，直接覆寫。
§ 08 / FUNCTIONS & PIPELINES

函式與管線

Sprig Library 精選

Helm 內建超過 180 個函式，絕大多數來自 Sprig 函式庫。呼叫語法為 functionName arg1 arg2，並可以用 | 串成管線（Pipeline）。 

# 呼叫形式 1：前綴
name: {{ quote .Values.app }}

# 呼叫形式 2：管線（推薦）
name: {{ .Values.app | quote }}

# 多層管線
name: {{ .Chart.Name | trunc 63 | trimSuffix "-" | lower }}

§ 09 / FLOW CONTROL

流程控制

if / with / range

if / else

{{- if .Values.ingress.enabled }}
apiVersion: networking.k8s.io/v1
kind: Ingress
{{- else if .Values.service.external }}
# 使用 LoadBalancer Service 而非 Ingress
{{- else }}
# 預設不暴露
{{- end }}
判斷為 false 的情況包含：空字串、數值 0、nil、空集合。其他皆為 true。

with（切換 scope）

進入 with block 後，. 會指向該物件。要存取根 scope 得用 $。

{{- with .Values.favorite }}
drink: {{ .drink | default "tea" | quote }}
food:  {{ .food  | upper | quote }}
release: {{ $.Release.Name }}   # $ 指向根
{{- end }}

range（迴圈）

# 迴圈 list
toppings: |
  {{- range .Values.pizzaToppings }}
  - {{ . | title | quote }}
  {{- end }}

# 迴圈 dict（取得 key, value）
{{- range $key, $val := .Values.env }}
  - name: {{ $key }}
    value: {{ $val | quote }}
{{- end }}

Variables（變數）

變數以 $ 開頭、用 := 指派，特別常用來在 range 中保留外層 scope。

{{- $fullName := .Release.Name }}
{{- range .Values.services }}
  - name: {{ $fullName }}-{{ .name }}
    port: {{ .port }}
{{- end }}
§ 10 / NAMED TEMPLATES

可重用 Named Templates

define、template、include

把重複出現的片段（如標籤、名稱計算）抽成命名模板放在 _helpers.tpl，底線開頭表示不會被直接輸出。

templates/_helpers.tpl
{{- define "mychart.fullname" -}}
{{- printf "%s-%s" .Release.Name .Chart.Name | trunc 63 | trimSuffix "-" -}}
{{- end -}}

{{- define "mychart.labels" -}}
app.kubernetes.io/name: {{ .Chart.Name }}
app.kubernetes.io/instance: {{ .Release.Name }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
helm.sh/chart: {{ printf "%s-%s" .Chart.Name .Chart.Version }}
{{- end -}}
templates/deployment.yaml 使用
metadata:
  name: {{ include "mychart.fullname" . }}
  labels:
    {{- include "mychart.labels" . | nindent 4 }}
優先使用 include 而非 template。原因：template 是語句（無回傳值），不能接管線；include 則回傳字串，可被 nindent、toYaml 等後續處理。
§ 11 / DEPENDENCIES

Chart 依賴管理

Subcharts 與 Global Values

在 Chart.yaml 的 dependencies 宣告後，用下列指令自動下載子 chart 到 charts/ 目錄。 

$ helm dependency update ./mychart       # 下載依賴
$ helm dependency list ./mychart         # 列出依賴
$ helm dependency build ./mychart         # 依 Chart.lock 重建

條件式啟用與別名

dependencies:
  - name: postgresql
    version: "12.x.x"
    repository: https://charts.bitnami.com/bitnami
    condition: postgresql.enabled      # values 開關
    alias: pg-primary                  # 重新命名，可多次引用同一子 chart
    tags:
      - database
    import-values:                      # 將子 chart 的 values 導到父 chart
      - child: service
        parent: pgService

Global Values

父 chart 的 values.yaml 中定義 global: 段落，所有子 chart 都能以 .Values.global.xxx 讀取，適合共享 image registry、domain 等設定。

# 父 chart values.yaml
global:
  imageRegistry: registry.corp.com
  storageClass: fast-ssd

# 子 chart 引用
image:
  repository: {{ .Values.global.imageRegistry }}/myapp
§ 12 / HOOKS

Chart Hooks

在 Release 生命週期特定時點執行

Hooks 讓你在安裝或升級的特定時間點插入任意 Kubernetes 資源（通常是 Job、Pod、ConfigMap），常見用途：資料庫 migration、憑證初始化、pre-upgrade 備份。

Hook觸發時機
pre-installrender 完成後、任何資源建立前
post-install所有資源都載入 Cluster 後
pre-delete卸載前
post-delete卸載完成後
pre-upgrade升級 render 完成後、套用前
post-upgrade升級完成後
pre-rollback回滾 render 完成後、套用前
post-rollback回滾完成後
test執行 helm test 時
範例：pre-install 執行 DB migration
apiVersion: batch/v1
kind: Job
metadata:
  name: {{ .Release.Name }}-migrate
  annotations:
    "helm.sh/hook": pre-install,pre-upgrade
    "helm.sh/hook-weight": "0"
    "helm.sh/hook-delete-policy": before-hook-creation,hook-succeeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: myapp:{{ .Chart.AppVersion }}
          command: ["./migrate.sh"]
hook-weight 控制執行順序（小的先執行），hook-delete-policy 可選 before-hook-creation、hook-succeeded、hook-failed，組合決定 hook 資源何時被清掉。
§ 13 / TESTS

Chart Tests

helm test

在 templates/tests/ 放置帶有 helm.sh/hook: test annotation 的 Pod，helm test <release> 就會執行；Pod 正常結束視為測試通過。

apiVersion: v1
kind: Pod
metadata:
  name: {{ .Release.Name }}-test-conn
  annotations:
    "helm.sh/hook": test
spec:
  restartPolicy: Never
  containers:
    - name: curl
      image: curlimages/curl:latest
      command: ["curl"]
      args: ["{{ .Release.Name }}:80"]
$ helm test my-release
$ helm test my-release --logs        # 同時顯示 Pod log
§ 14 / REPOSITORIES

Chart Repository

Traditional HTTP 與 OCI Registry

傳統 HTTP Repository

一個 Helm repo 其實就是放了 index.yaml 與多個 *.tgz 檔的 HTTP 伺服器。可以用 GitHub Pages、S3、GCS、ChartMuseum 或 Harbor 提供。 

# 管理本機已加入的 repo
$ helm repo add bitnami https://charts.bitnami.com/bitnami
$ helm repo list
$ helm repo update
$ helm repo remove bitnami

# 製作自有 repo
$ mkdir myrepo && cp mychart-0.1.0.tgz myrepo/
$ helm repo index myrepo/ --url https://charts.example.com
# → 產生 myrepo/index.yaml，把整個目錄放上 HTTP 伺服器即可

OCI Registry（Helm 3.8+）

Helm 支援把 chart 推到 OCI-compliant registry（Docker Hub、Harbor、ECR、GHCR、Artifact Registry…），與 container image 共用一套儲存與權限。 

# 登入 OCI registry
$ helm registry login registry.example.com -u user

# 推送 chart
$ helm push mychart-0.1.0.tgz oci://registry.example.com/charts

# 從 OCI 拉取與安裝
$ helm pull oci://registry.example.com/charts/mychart --version 0.1.0
$ helm install my-app oci://registry.example.com/charts/mychart --version 0.1.0
§ 15 / PLUGINS

Helm Plugins

擴充 Helm 的能力

Plugin 是額外的子指令（例如 helm diff、helm secrets、helm s3），安裝後可像內建指令一樣使用。 

$ helm plugin install https://github.com/databus23/helm-diff
$ helm plugin list
$ helm plugin update diff
$ helm plugin uninstall diff

# 熱門 Plugin 範例
$ helm diff upgrade my-app ./mychart       # 升級前 diff 預覽
$ helm secrets enc values-prod.yaml       # 加密 values
§ 16 / BEST PRACTICES

最佳實踐

來自官方 Best Practices 章節
Values

Values 命名以 camelCase 為主

除非對應到 K8s 資源原生欄位，否則 values key 採 camelCase（maxConnections），並盡量扁平、易於用 --set 覆寫。

Labels

套用標準 Recommended Labels

一律加上 app.kubernetes.io/name、instance、version、managed-by、part-of，並以 helm.sh/chart 標示出處。

Naming

使用 fullname helper 限制長度

K8s 物件名稱上限為 63 字元，用 trunc 63 | trimSuffix "-" 組合，避免超長或以破折號結尾。

Required

關鍵值用 required 函式

{{ required "image.repository is required" .Values.image.repository }} 可在缺值時直接中止並顯示訊息。

Schema

提供 values.schema.json

以 JSON Schema 驗證 values 輸入型別與必填，降低使用者誤設機率。

Secret

Secret 不寫死

避免把敏感值放在 chart 或 values，改由外部注入（External Secrets、Sealed Secrets、helm-secrets plugin）。

Checksum

ConfigMap 變更觸發 Pod 重啟

在 Deployment 的 spec.template.metadata.annotations 加入 checksum/config: {{ include "mychart.configmap" . | sha256sum }}，配置變更時 Pod 會自動 rolling update。

CI/CD

同時跑 lint、template、test

Pipeline 中依序執行 helm lint、helm template --debug、helm install --dry-run、helm test，問題越早發現越便宜。

§ 17 / DEBUGGING

除錯技巧

從 template 到 release
# 1. 只本機渲染，不碰 Cluster
$ helm template my-app ./mychart
$ helm template my-app ./mychart -f prod.yaml --show-only templates/deployment.yaml

# 2. 結合 Kubernetes API dry-run
$ helm install my-app ./mychart --dry-run --debug

# 3. 查詢現有 release 產生的 manifest
$ helm get manifest my-app

# 4. 比較本機 chart 與實際 release 的差異（需安裝 helm-diff plugin）
$ helm diff upgrade my-app ./mychart

# 5. lint 檢查
$ helm lint ./mychart --strict
處理空值陷阱：當 .Values.something 為 nil，用 default 提供安全值；當需要強制存在，用 required；當只在存在時輸出，用 with。這三個組合能消滅八成的 template 錯誤。

Template 語法速查

Action 與空白控制

語法說明
{{ expr }}一般 action
{{- expr }}消除前方空白與換行
{{ expr -}}消除後方空白與換行
{{/* comment */}}模板註解（不輸出）
{{ $var := expr }}宣告變數
{{ $var = expr }}重新指派（同 scope）

內建物件

物件說明
.Release.Name / Namespace / RevisionRelease 基本資訊
.Release.IsInstall / IsUpgrade當前操作判斷
.Values合併後的設定值
.Chart.Name / Version / AppVersionChart 元資料
.Files.Get / Glob / Lines讀取 chart 內檔案
.Files.AsConfig / AsSecrets轉 ConfigMap / Secret 用
.Capabilities.KubeVersionK8s 版本
.Capabilities.APIVersions.Has "apps/v1"判斷 API 是否存在
.Template.Name / BasePath目前模板資訊
$根 scope（不隨 with/range 變化）
.當前 scope

控制結構

語法說明
{{ if PIPE }}...{{ else if PIPE }}...{{ else }}...{{ end }}條件
{{ with PIPE }}...{{ end }}切換 scope
{{ range PIPE }}...{{ end }}迴圈
{{ range $i, $v := .list }}...{{ end }}帶索引的迴圈
{{ range $k, $v := .dict }}...{{ end }}走訪 dict
{{ break }} / {{ continue }}迴圈跳出 / 跳到下一輪
{{ define "name" }}...{{ end }}定義命名模板
{{ template "name" . }}呼叫命名模板（無回傳值）
{{ include "name" . }}呼叫並回傳字串（可接管線）
{{ required "msg" .v }}值必填否則錯誤
{{ fail "msg" }}主動拋錯中止渲染
{{ tpl "{{ .Values.x }}" . }}把字串當模板再渲染
