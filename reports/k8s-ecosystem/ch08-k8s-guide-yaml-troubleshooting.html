T01 · Pod（生產級最小範本）

pod.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: app
  namespace: default
  labels:
    app: app
spec:
  serviceAccountName: default
  automountServiceAccountToken: false
  securityContext:
    runAsNonRoot: true
    runAsUser: 10001
    fsGroup: 10001
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: myapp:1.0
    imagePullPolicy: IfNotPresent
    ports:
    - containerPort: 8080
      name: http
    env:
    - name: LOG_LEVEL
      value: "info"
    resources:
      requests:
        cpu: "100m"
        memory: "128Mi"
      limits:
        memory: "256Mi"
    livenessProbe:
      httpGet: { path: /healthz, port: 8080 }
      periodSeconds: 10
    readinessProbe:
      httpGet: { path: /ready, port: 8080 }
      periodSeconds: 5
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]

T02 · Deployment

deployment.yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  labels: { app: web }
spec:
  replicas: 3
  revisionHistoryLimit: 5
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  selector:
    matchLabels: { app: web }
  template:
    metadata:
      labels: { app: web }
    spec:
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: topology.kubernetes.io/zone
        whenUnsatisfiable: ScheduleAnyway
        labelSelector:
          matchLabels: { app: web }
      containers:
      - name: web
        image: nginx:1.27
        ports: [{ containerPort: 80 }]
        envFrom:
        - configMapRef: { name: app-config }
        - secretRef: { name: app-secret }
        resources:
          requests: { cpu: "100m", memory: "128Mi" }
          limits: { memory: "256Mi" }
        readinessProbe:
          httpGet: { path: /, port: 80 }
        livenessProbe:
          httpGet: { path: /, port: 80 }
          initialDelaySeconds: 15

T03 · Service（四種類型）

services.yaml · 全種類 
# ── ClusterIP：內部通訊預設 ──
apiVersion: v1
kind: Service
metadata: { name: web-svc }
spec:
  type: ClusterIP
  selector: { app: web }
  ports:
  - { name: http, port: 80, targetPort: 8080 }
---
# ── NodePort：透過 Node 高 port 對外 ──
apiVersion: v1
kind: Service
metadata: { name: web-np }
spec:
  type: NodePort
  selector: { app: web }
  ports:
  - { port: 80, targetPort: 8080, nodePort: 30080 }
---
# ── LoadBalancer：雲端 LB ──
apiVersion: v1
kind: Service
metadata:
  name: web-lb
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
spec:
  type: LoadBalancer
  selector: { app: web }
  ports:
  - { port: 80, targetPort: 8080 }
---
# ── Headless：StatefulSet 專用 ──
apiVersion: v1
kind: Service
metadata: { name: db-headless }
spec:
  clusterIP: None
  selector: { app: db }
  ports:
  - { port: 5432 }

T04 · Ingress（多域名 + TLS）

ingress.yaml 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  ingressClassName: nginx
  tls:
  - hosts: [api.example.com, www.example.com]
    secretName: app-tls
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /v1
        pathType: Prefix
        backend:
          service: { name: api-v1, port: { number: 80 } }
      - path: /v2
        pathType: Prefix
        backend:
          service: { name: api-v2, port: { number: 80 } }
  - host: www.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service: { name: web, port: { number: 80 } }

T06 · PVC（含 snapshot restore）

pvc.yaml 
# 動態配置
apiVersion: v1
kind: PersistentVolumeClaim
metadata: { name: data }
spec:
  storageClassName: gp3
  accessModes: [ReadWriteOnce]
  resources:
    requests: { storage: 20Gi }
---

apiVersion: v1
kind: PersistentVolumeClaim
metadata: { name: data-restored }

  storageClassName: gp3
  dataSource:
    apiGroup: snapshot.storage.k8s.io
    kind: VolumeSnapshot
    name: my-snapshot-2026-04-24
  accessModes: [ReadWriteOnce]
  resources:
    requests: { storage: 20Gi }

T07 · StatefulSet

statefulset.yaml 
apiVersion: v1
kind: Service
metadata: { name: db-headless }
spec:
  clusterIP: None
  selector: { app: db }
  ports: [{ port: 5432 }]
---
apiVersion: apps/v1
kind: StatefulSet
metadata: { name: db }
spec:
  serviceName: db-headless
  replicas: 3
  podManagementPolicy: OrderedReady
  selector:
    matchLabels: { app: db }
  template:
    metadata:
      labels: { app: db }
    spec:
      terminationGracePeriodSeconds: 60
      containers:
      - name: db
        image: postgres:16
        ports: [{ containerPort: 5432 }]
        env:
        - name: POSTGRES_PASSWORD
          valueFrom:
            secretKeyRef: { name: db-secret, key: password }
        volumeMounts:
        - { name: data, mountPath: /var/lib/postgresql/data }
  volumeClaimTemplates:
  - metadata: { name: data }
    spec:
      accessModes: [ReadWriteOnce]
      storageClassName: gp3
      resources:
        requests: { storage: 20Gi }

T08 · DaemonSet（可跑在 control-plane）

daemonset.yaml 
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: log-agent
  namespace: logging
spec:
  selector:
    matchLabels: { app: log-agent }
  template:
    metadata:
      labels: { app: log-agent }
    spec:
      tolerations:
      - key: node-role.kubernetes.io/control-plane
        operator: Exists
        effect: NoSchedule
      - operator: Exists      # 容忍所有 taint（日誌要無差別收）
        effect: NoExecute
      containers:
      - name: agent
        image: fluent/fluent-bit:3.0
        resources:
          requests: { cpu: "50m", memory: "64Mi" }
          limits: { memory: "200Mi" }
        volumeMounts:
        - { name: varlog, mountPath: /var/log, readOnly: true }
      volumes:
      - name: varlog
        hostPath: { path: /var/log }

T09 · Job / CronJob

job-cronjob.yaml 
# ── 一次性 Job（平行 3 個，共完成 10 次）──
apiVersion: batch/v1
kind: Job
metadata: { name: data-migration }
spec:
  completions: 10
  parallelism: 3
  backoffLimit: 3
  activeDeadlineSeconds: 3600
  ttlSecondsAfterFinished: 86400   # 完成 1 天後自動清理
  template:
    spec:
      restartPolicy: OnFailure
      containers:
      - name: migrate
        image: migrator:1.0
---
# ── 每日凌晨備份 ──
apiVersion: batch/v1
kind: CronJob
metadata: { name: db-backup }
spec:
  schedule: "0 3 * * *"
  timeZone: "Asia/Taipei"
  concurrencyPolicy: Forbid
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 1
  jobTemplate:
    spec:
      backoffLimit: 2
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: backup
            image: backup:1.0
            command: ["./backup.sh"]

T10 · HPA（CPU + Memory + 細調 behavior）

hpa.yaml 
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: { name: web-hpa }
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 3
  maxReplicas: 30
  metrics:
  - type: Resource
    resource:
      name: cpu
      target: { type: Utilization, averageUtilization: 70 }
  - type: Resource
    resource:
      name: memory
      target: { type: AverageValue, averageValue: "400Mi" }
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0
      policies:
      - { type: Percent, value: 100, periodSeconds: 30 }
      - { type: Pods,    value: 4,   periodSeconds: 30 }
      selectPolicy: Max
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - { type: Percent, value: 50, periodSeconds: 60 }

T11 · RBAC（SA + Role + Binding 三件組）

rbac.yaml 
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
  namespace: default
---
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: app-role
rules:
- apiGroups: [""]
  resources: ["configmaps", "secrets"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list"]
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "patch"]
  resourceNames: ["web"]     # 只對特定名稱
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-binding
  namespace: default
subjects:
- kind: ServiceAccount
  name: app-sa
  namespace: default
roleRef:
  kind: Role
  name: app-role
  apiGroup: rbac.authorization.k8s.io

T12 · NetworkPolicy（分層放行）

netpol.yaml 
# Step 1: default deny 全部
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: default-deny-all }
spec:
  podSelector: {}
  policyTypes: [Ingress, Egress]
---
# Step 2: 放行 DNS（必做！）
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: allow-dns }
spec:
  podSelector: {}
  policyTypes: [Egress]
  egress:
  - to:
    - namespaceSelector:
        matchLabels: { kubernetes.io/metadata.name: kube-system }
    ports:
    - { protocol: UDP, port: 53 }
    - { protocol: TCP, port: 53 }
---
# Step 3: 只允許 frontend → backend
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: backend-allow-frontend }
spec:
  podSelector:
    matchLabels: { tier: backend }
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector:
        matchLabels: { tier: frontend }
    ports:
    - { protocol: TCP, port: 8080 }

Part 02
Troubleshooting · 問題排查
11 個最常見錯誤 × 對應決策樹 × 對應指令。遇到問題翻到這裡直接比對。

除錯流程總覽（遇到任何問題先做這三步）

① 這個資源當前狀態？
kubectl get pod/deploy/svc <name>
② 有什麼 events？
kubectl describe <kind> <name> ← 底部 Events 段是金礦
kubectl get events -n <ns> --sort-by='.lastTimestamp'
③ 容器日誌說什麼？
kubectl logs <pod> [--previous] [-c container]

這三個指令可解 80% 問題。下面針對每種具體症狀展開決策樹：

P1 · Pod 卡 Pending

PROBLEM
Pod 狀態持續 Pending，沒排到 Node
常見原因
  • Cluster 所有 Node 資源都不夠（CPU / Memory）
  • Pod 要求的 Taint Toleration 沒對應 Node
  • NodeSelector / Affinity 條件沒 Node 符合
  • PVC 未綁定（卡在 WaitForFirstConsumer）
  • Image Pull Secret 缺失使得拉映像卡住
排查步驟
kubectl describe pod <pod>
↓ 看 Events 底下訊息
FailedScheduling: Insufficient cpu/memory?
→ kubectl top nodes 看哪些 Node 滿了
→ 擴 Node（CA / Karpenter）或降 Pod requests
didn't match node selector / affinity?
→ kubectl get nodes --show-labels
→ 確認有 Node 符合 affinity；必要時 label Node
node(s) had untolerated taint?
→ kubectl describe node <n> | grep Taint
→ 加對應 toleration 或移除 taint
pod has unbound immediate PersistentVolumeClaims?
→ 見 P7（PVC 卡 Pending）

P2 · ImagePullBackOff / ErrImagePull

PROBLEM
拉映像失敗，容器無法啟動
排查步驟
kubectl describe pod <pod> | grep -A 5 Events
manifest for xxx:tag not found?
→ 映像名稱或 tag 拼錯。docker pull 本機驗證
authentication required / 401 Unauthorized?
→ 私有 registry 需 imagePullSecret
→ kubectl create secret docker-registry regcred ...
→ Pod spec 加 imagePullSecrets: [{ name: regcred }]
dial tcp: i/o timeout?
→ Node 無法連到 registry（防火牆、NAT、DNS）
→ 進 Node 測試 curl -v registry.example.com
pull rate limit exceeded (Docker Hub)?
→ 認證 Docker Hub 帳號或改用鏡像站/私有 registry
快速修正指令
# 建立拉映像憑證
kubectl create secret docker-registry regcred \
  --docker-server=registry.example.com \
  --docker-username=user --docker-password=pass

# 套到 Deployment
kubectl patch deployment web -p '{"spec":{"template":{"spec":{"imagePullSecrets":[{"name":"regcred"}]}}}}'

P3 · CrashLoopBackOff

PROBLEM
容器一直啟動又崩潰
排查步驟
第 1 步：看上次 crash 前的日誌（最關鍵）
→ kubectl logs <pod> --previous
→ 看 stack trace、fatal error 訊息
第 2 步：看 exit code
→ kubectl describe pod <pod> 找 Last State
→ 137 = SIGKILL（通常 OOM，見 P4）
→ 139 = segfault（程式 bug）
→ 1 / 2（應用自己拋錯）
第 3 步：是 Liveness probe 殺掉的嗎？
→ Events 若有 "Liveness probe failed" → 放寬 probe 或查應用為何無法回應
第 4 步：要更深入？換掉 command 進去看
→ kubectl debug <pod> -it --copy-to=shell --image=... -- sh
→ 手動試跑原 command 觀察錯誤
💡 核心原則
99% CrashLoopBackOff 的答案在 kubectl logs --previous 裡。沒看過上次日誌就亂改配置是本末倒置。

P4 · OOMKilled（exit code 137）

PROBLEM
容器超過 memory limit 被 kernel OOM killer 殺掉
確認
kubectl describe pod <pod> | grep -A 3 "Last State"
# 看到：
#   Last State: Terminated
#   Reason: OOMKilled
#   Exit Code: 137
處理
  • 暫時緩解：調高 resources.limits.memory
  • 找根因：容器記憶體洩漏？請看應用 profile（heap dump、pprof）
  • 語言特性：Java / .NET 要調 JVM / GC 參數，讓它「知道」容器 limit。
    Java 17+ 會自動讀 cgroup limit；舊版需 -XX:+UseContainerSupport
  • 觀測：Prometheus 裝 container_memory_working_set_bytes 趨勢圖，持續觀察
⚠️ Node 層級 OOM
如果 Node 整體記憶體不足（不只單一容器），kubelet 會依 QoS 驅逐 Pod——先 BestEffort、再 Burstable。避免方法：確保所有 Pod 都有合理的 requests，升級到 Guaranteed QoS。

P5 · Service 連不通 Pod

PROBLEM
curl Service ClusterIP 或 DNS 名稱連不上後端
七步排查法
① Pod 真的 Ready 嗎？
→ kubectl get pods -l <selector> 看 READY 欄（0/1 就沒通過 Readiness）
② Service selector 對得上 Pod label 嗎？
→ kubectl get svc <svc> -o yaml 看 selector
→ kubectl get pods --show-labels 對比
③ EndpointSlice 有列到 Pod IP 嗎？
→ kubectl get endpointslices -l kubernetes.io/service-name=<svc>
→ 空的 = selector 沒匹配或 Pod 沒 Ready
④ targetPort 與 containerPort 對得上嗎？
→ Service port 是外部看到的，targetPort 才是 Pod 內的 port
⑤ 從另一個 Pod 能直連 Pod IP 嗎？
→ kubectl run test -it --rm --image=nicolaka/netshoot -- curl <pod-ip>:<port>
→ 能 = Service 層問題；不能 = Pod 或網路層問題
⑥ 從另一個 Pod 能連 Service ClusterIP 嗎？
→ curl <cluster-ip>:<port>（用 IP 不用名稱，排除 DNS）
⑦ DNS 解析對不對？
→ 見 P6
⑧ NetworkPolicy 擋住了嗎？
→ kubectl get netpol -A
→ 有 default-deny 卻沒放行對應流量？

P6 · DNS 解析失敗

PROBLEM
Pod 內無法解析 Service 名稱或外部域名
排查
# 開一個 debug pod
kubectl run dns-test -it --rm --image=busybox:1.36 --restart=Never -- sh

# 測試
nslookup kubernetes.default           # 應回 10.96.0.1 類 IP
nslookup my-service.default.svc.cluster.local
nslookup google.com                   # 測試對外 DNS
常見原因
  • CoreDNS 沒跑：kubectl -n kube-system get pods -l k8s-app=kube-dns
  • NetworkPolicy 擋 UDP/TCP 53 到 kube-system（最常見！）
  • CoreDNS ConfigMap 設錯（上游 DNS、forward 規則）
  • Node 本機 /etc/resolv.conf 問題（影響 external DNS）
  • ndots 問題：Pod 預設 ndots: 5，短名查詢會先試多個 search domain
💡 效能優化
高頻外部域名（API 調用）可在 Pod spec 設 dnsConfig.options: [{name: ndots, value: "1"}] 減少查詢次數。

P7 · PVC 卡 Pending

PROBLEM
PersistentVolumeClaim 狀態持續 Pending
排查
kubectl describe pvc <pvc>
# 看 Events 訊息
"waiting for first consumer to be created"
→ 正常行為！StorageClass 是 WaitForFirstConsumer，要等 Pod 用它才配
→ 確認有 Pod 引用這個 PVC
"no StorageClass set"
→ PVC 沒指定 storageClassName，且 Cluster 沒預設 SC
→ kubectl get sc 看有哪些 SC、哪個是 default
"failed to provision volume"
→ 底層問題：雲端配額滿、權限不足、AZ 不對
→ kubectl -n kube-system logs <csi-controller> 看 CSI 驅動日誌
"volume node affinity conflict"
→ PV 在 AZ-A、Pod 被排到 AZ-B
→ 改用 WaitForFirstConsumer 避免此問題

P8 · Rollout 卡住不前進

PROBLEM
kubectl rollout status 一直卡著，新版 Pod 起不來
排查
# 當前兩版 ReplicaSet 的狀態
kubectl get rs -l app=web

# 新 ReplicaSet 的 Pod 為何沒 Ready？
kubectl get pods -l app=web
kubectl describe pod <new-pod>
kubectl logs <new-pod>
典型原因
  • 新映像有問題：回去看 CrashLoopBackOff / ImagePullBackOff 流程
  • Readiness Probe 一直失敗：新 Pod 起來但過不了檢查，舊 Pod 不會縮
  • 資源不夠擴出新 Pod：maxSurge 衝突、Node 滿了
  • PDB 綁死：PDB 要求 minAvailable 較高，加上 maxUnavailable=0，無法進行
緊急回滾
kubectl rollout undo deployment/web
kubectl rollout status deployment/web

P9 · Ingress 404 / 502 / 503

PROBLEM
從瀏覽器打不到應用，出現 404 / 502 / 503
依 HTTP 錯誤碼分流
錯誤碼層次常見原因
404Ingress 找不到規則Host / Path 不匹配、ingressClassName 不對、pathType 設錯
502Ingress → Service 有問題後端 Service 沒 endpoint、上游連線中斷、SSL 設定錯
503暫時無可用後端所有 Pod 都 not ready、Service 無 endpoint
504Timeout應用太慢、Ingress timeout 設太短
排查順序
  1. 檢查 Ingress：kubectl describe ingress <ing>——規則正確？
  2. 檢查 Service：kubectl get svc <svc>——存在？
  3. 檢查 Endpoints：kubectl get endpointslices——有 IP？
  4. 用 port-forward 繞過 Ingress 直測 Service：kubectl port-forward svc/<svc> 8080:80
  5. 看 Ingress Controller 日誌：kubectl -n ingress-nginx logs -l app.kubernetes.io/name=ingress-nginx

P10 · Forbidden / permission denied

PROBLEM
應用或使用者呼叫 K8s API 被拒，錯誤訊息含 forbidden
排查
# 1. 看完整錯誤訊息
# "User \"system:serviceaccount:default:app\" cannot list pods in namespace \"production\""

# 2. 用 can-i 重現並確認
kubectl auth can-i list pods -n production \
  --as=system:serviceaccount:default:app

# 3. 列出這個 SA 所有能做的事
kubectl auth can-i --list -n production \
  --as=system:serviceaccount:default:app

# 4. 看它被綁到哪些 Role
kubectl get rolebinding,clusterrolebinding -A -o wide | grep app

# 5. 反查「誰能做 X」：安裝 who-can 外掛
kubectl who-can list pods -n production
修正
  • 增加 Role 的 rules（最小權限原則）
  • 確認 RoleBinding 的 subjects 寫對（namespace + SA 名稱）
  • 跨 namespace 訪問需 ClusterRole + RoleBinding（綁目標 ns）

P11 · Node 狀態 NotReady

PROBLEM
kubectl get nodes 顯示某 Node STATUS 為 NotReady
排查
kubectl describe node <node>
# 看 Conditions 段：
#   Ready / DiskPressure / MemoryPressure / PIDPressure / NetworkUnavailable
Ready=False / kubelet stopped posting status?
→ kubelet 服務掛了，SSH 進 Node 看 systemctl status kubelet
→ 日誌：journalctl -u kubelet -n 100
DiskPressure=True?
→ Node 磁碟快滿，kubelet 會驅逐 Pod 並拒新 Pod
→ 清理 /var/lib/containerd（舊 image、container 層）
NetworkUnavailable=True?
→ CNI 有問題，看 CNI DaemonSet 日誌（calico / cilium）
所有 Node 都 NotReady?
→ apiserver 或 etcd 問題，看 control-plane 狀態
緊急處理：把 Node 的工作遷走
kubectl cordon <node>        # 禁新 Pod 上來
kubectl drain <node> --ignore-daemonsets --delete-emptydir-data
# 檢修 Node，好了再
kubectl uncordon <node>
