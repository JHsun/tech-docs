CHAPTER 03 · 網路與入口

Services、Networking、Ingress

Pod 起來了，但 IP 會變、還無法對外。這章處理三個問題：Pod 間怎麼互通、 外部怎麼訪問 Cluster 內的服務、怎麼用策略限制流量。 看完你會理解 ClusterIP 到 Ingress、到新世代 Gateway API 的全貌。

Kubernetes 網路模型

K8s 定義了四條基本規則，由 CNI 外掛負責實作：

  1. 每個 Pod 有獨立 IP：Pod 內所有容器共用此 IP 與 port 空間
  2. Pod 之間可直接互通：不需 NAT，任何 Pod 都能用對方 IP 直連
  3. Node 與 Pod 可互通：Node 上的 agent（kubelet、kube-proxy）能連到其上的 Pod
  4. Service 提供穩定入口：抽象一組 Pod，屏蔽底下 Pod 的 IP 變動
🧭 分層思考
把 K8s 網路想成四層：Pod 網路（CNI 負責）→ Service 網路（kube-proxy 負責）→ Ingress / Gateway（L7 路由）→ NetworkPolicy（流量策略）。每層都是上一層的補強，不是取代。

流量由外到內的層級

🌐 External Client
瀏覽器、手機、外部 API 使用者
↓
☁️ LoadBalancer / NodePort
雲端 LB 或 Node 上暴露的高 port，把流量丟入 Cluster
↓
🚪 Ingress / Gateway
L7 路由：依 Host、Path 轉發到不同 Service
↓
🔗 Service (ClusterIP)
穩定的虛擬 IP，背後是一組動態的 Pod
↓
📦 Pod × N
實際處理請求的容器

Service 是什麼

Pod 的 IP 會隨重啟、擴縮變動，前端不可能直接連。Service 是一層穩定的抽象： 給一組 Pod 一個虛擬 IP（ClusterIP）與 DNS 名稱，無論背後 Pod 如何變動，前端永遠連同一個地址。

Service 如何找到 Pod：Label Selector

service.yaml · 最小例子 
apiVersion: v1
kind: Service
metadata:
  name: web-svc
spec:
  selector:              # 把所有 label app=web 的 Pod 納入
    app: web
  ports:
  - name: http
    port: 80            # Service 對外 port
    targetPort: 8080    # Pod 內實際 port
    protocol: TCP

K8s 會持續 watch 所有符合 selector 的 Pod，把它們的 IP 寫入一個叫 EndpointSlice 的物件。 kube-proxy 讀取 EndpointSlice，在每個 Node 上安裝 iptables / IPVS 規則，完成「ClusterIP → Pod IP」的轉發。

四種 Service 類型

Type可訪問範圍典型用途
ClusterIP（預設）僅 Cluster 內內部微服務互相呼叫
NodePortCluster 內 + Node IP 上的高 port本機開發、無 LB 環境
LoadBalancer透過雲端 LB 對外生產環境對外暴露
ExternalNameDNS CNAME 別名把外部服務包成 Cluster 內名稱

ClusterIP：內部通訊基石

預設類型。產生一個只在 Cluster 內可達的虛擬 IP，搭配 DNS 讓其他 Pod 透過名稱訪問。

Bash · 在 Pod 內訪問 Service 
# 同 namespace 內
curl http://web-svc/

# 跨 namespace（全限定域名 FQDN）
curl http://web-svc.production.svc.cluster.local/

# 格式：<service>.<namespace>.svc.<cluster-domain>

NodePort：借 Node 的 port 對外

在每個 Node 上開一個固定 port（預設範圍 30000-32767），讓外部可透過 <Node-IP>:<NodePort> 訪問。 它包含了 ClusterIP，所以 Cluster 內仍可正常使用。

nodeport-service.yaml 
apiVersion: v1
kind: Service
metadata:
  name: web-np
spec:
  type: NodePort
  selector:
    app: web
  ports:
  - port: 80            # ClusterIP 層 port
    targetPort: 8080    # Pod 層 port
    nodePort: 30080     # Node 層 port（不指定則 K8s 隨機選）
⚠️ 非生產環境選擇
NodePort 不適合正式對外：port 範圍受限、需要每個 Node 的 IP（隨 Node 增減會變）、沒有 L7 功能。它主要用於開發測試或搭配上層 LB。

LoadBalancer：雲端標配

在支援的雲環境（AWS ELB、GCP LB、Azure LB）會自動配置一台外部負載平衡器指向所有 Node 的 NodePort。 它包含了 NodePort 與 ClusterIP 的功能。

lb-service.yaml 
apiVersion: v1
kind: Service
metadata:
  name: web-lb
  annotations:
    # 雲端專屬設定透過 annotation 傳遞
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 8080
💰 成本考量
每個 LoadBalancer 都會建立一台雲端 LB（AWS NLB 每月約 $16+ 流量費）。有多個 Service 要對外時，用一個 LoadBalancer + Ingress做 L7 路由更划算。

ExternalName：外部服務別名

不建立 ClusterIP，也不做 proxy，純粹是DNS CNAME 記錄。用於把外部服務包裝成 Cluster 內的名稱，遷移時好替換。

external-service.yaml 
apiVersion: v1
kind: Service
metadata:
  name: legacy-db
  namespace: default
spec:
  type: ExternalName
  externalName: db.legacy.example.com    # Pod 內訪問 legacy-db 會被 DNS 導到這個名稱

Headless Service：要 Pod IP 不要代理

設定 clusterIP: None 就成為 Headless Service。不配虛擬 IP，DNS 查詢直接回傳所有 Pod IP。

典型用途：

  • StatefulSet 的網路身份：每個 Pod 有 pod-0.svc.ns.svc.cluster.local
  • 客戶端自行做 LB：如 gRPC client-side load balancing
  • Service Mesh 場景：Istio 需要知道每個 endpoint
headless-service.yaml 
apiVersion: v1
kind: Service
metadata:
  name: db-headless
spec:
  clusterIP: None           # 關鍵！
  selector:
    app: db
  ports:
  - port: 5432

EndpointSlice：Service 的真正後端

當你建立 Service，K8s 並不直接存「Service 對應哪些 Pod」。它建立 EndpointSlice—— 一組 Pod IP + port 的清單。kube-proxy 讀這個來更新轉發規則。

EndpointSlice 是舊 Endpoints 物件的替代方案，從 K8s v1.21 穩定。切分成多個 slice 後擴展性更好（上萬 Pod 也撐得住）。

Bash · 查看 EndpointSlice 
# 查看某個 Service 對應的 endpoints
kubectl get endpointslices -l kubernetes.io/service-name=web-svc

# 更詳細的資訊
kubectl describe endpointslice <name>

Cluster DNS：服務發現魔法

每個 Cluster 都有 CoreDNS（以 Deployment 形式跑在 kube-system namespace）。 當 Pod 建立時，kubelet 會把 /etc/resolv.conf 指向 CoreDNS。

DNS 命名規則

物件DNS 名稱
普通 Service<svc>.<ns>.svc.cluster.local → ClusterIP
Headless Service<svc>.<ns>.svc.cluster.local → 所有 Pod IP
StatefulSet Pod<pod>.<headless-svc>.<ns>.svc.cluster.local → 單一 Pod IP
ExternalName Service回傳 CNAME
Bash · DNS 解析實驗 
# 開一個 debug pod
kubectl run -it --rm debug --image=busybox:1.36 --restart=Never -- sh

# 在 pod 內測試
nslookup web-svc                               # 當前 namespace
nslookup web-svc.production                    # 其他 namespace（短名）
nslookup web-svc.production.svc.cluster.local  # FQDN
💡 效能小技巧
Pod 內訪問同 namespace 服務時，直接用短名（如 web-svc）比 FQDN 效能稍差（會經過 search domain 補齊），但差異通常可忽略。高流量場景可設 dnsConfig.options: ndots: 1 減少查詢次數。

Ingress：HTTP 入口路由

Service 到 LoadBalancer 做的是 L4（TCP / UDP）轉發。如果你要根據 Host、Path、TLS 做 L7 路由， 就需要 Ingress。

Ingress 本身只是一份路由規則描述，實際執行的是 Ingress Controller（下一節）。 一份 Ingress 範例：

ingress.yaml · 多域名 + TLS 
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app-ingress
  annotations:
    cert-manager.io/cluster-issuer: "letsencrypt"
spec:
  ingressClassName: nginx   # 指定用哪個 Controller
  tls:
  - hosts:
    - api.example.com
    - www.example.com
    secretName: app-tls     # 由 cert-manager 自動填入憑證
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /v1
        pathType: Prefix
        backend:
          service:
            name: api-v1-svc
            port:
              number: 80
      - path: /v2
        pathType: Prefix
        backend:
          service:
            name: api-v2-svc
            port:
              number: 80
  - host: www.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-svc
            port:
              number: 80

pathType 三種值

值規則範例
Exact精確匹配（含結尾斜線與否）/foo 只匹配 /foo
Prefix前綴匹配（以 / 分段）/foo 匹配 /foo/bar 但不匹配 /foobar
ImplementationSpecific由 Controller 自訂不建議用，行為不可攜

Ingress Controller：真正執行的人

K8s 本身不內建 Ingress Controller。你必須先部署一個，Ingress 資源才會生效。

Controller特點
ingress-nginx官方維護，最多人用，功能完整
Traefik自動服務發現，內建儀表板
HAProxy Ingress高效能 L7
AWS Load Balancer Controller直接創建 ALB，AWS 環境首選
CiliumeBPF 實作，同時做 CNI + Ingress

如何安裝 ingress-nginx（範例）

Bash · Helm 安裝 
helm repo add ingress-nginx https://kubernetes.github.io/ingress-nginx
helm repo update

helm install ingress-nginx ingress-nginx/ingress-nginx \
  --namespace ingress-nginx --create-namespace \
  --set controller.service.type=LoadBalancer

# 查看 LB 的外部 IP / hostname
kubectl get svc -n ingress-nginx ingress-nginx-controller
🔑 關鍵認知
通常整個 Cluster 只開一個 LoadBalancer 給 Ingress Controller，所有 HTTP 流量都從這裡進、由 Ingress 規則決定轉發到哪個 Service。大幅降低雲端 LB 成本。

Gateway API：下一代 Ingress

Ingress 設計已有多年，許多進階需求（TCP / UDP 路由、流量分割、可重用設定）只能塞 annotation，各家 Controller 實作還不相容。 Gateway API 是新世代替代方案，從 K8s v1.31 起核心資源穩定（GA）。

三層角色分離

GatewayClass

由基礎設施團隊提供，定義某類 Gateway 實作（如 Istio、Contour）。

Gateway

由Cluster 管理員建立，代表一個實際的流量入口（LB、監聽器、憑證）。

HTTPRoute / TCPRoute

由應用團隊建立，附掛到某個 Gateway，描述路由規則。

gateway.yaml · HTTP 入口 
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: public-gw
spec:
  gatewayClassName: istio
  listeners:
  - name: http
    protocol: HTTP
    port: 80
    allowedRoutes:
      namespaces:
        from: All
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: api-route
spec:
  parentRefs:
  - name: public-gw
  hostnames:
  - api.example.com
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /v1
    backendRefs:
    - name: api-v1-svc
      port: 80

Ingress vs Gateway API

比較項IngressGateway API
協議支援僅 HTTP/HTTPSHTTP、TCP、UDP、TLS
進階功能靠 annotation（各家不一）原生欄位（跨實作一致）
角色分離一份 YAML 混雜所有設定GatewayClass / Gateway / Route 三層
成熟度最多人用、生態完整核心 GA、持續演進
建議既有專案繼續用新專案評估採用

NetworkPolicy：Pod 防火牆

預設情況下，Cluster 內所有 Pod 可以互通。生產環境這是危險的——一個被入侵的 Pod 就能橫向擴散。 NetworkPolicy 讓你用 selector 寫白名單：誰能連誰。

⚠️ 前提
NetworkPolicy 需要 CNI 支援（Calico、Cilium、Weave 都可；某些簡化版 CNI 不支援）。沒有相容 CNI 時，NetworkPolicy 寫了也無效。

三要素：podSelector + policyTypes + 規則

default-deny.yaml · 預設拒絕全部 
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}           # 空 = 套用到所有 Pod
  policyTypes:
  - Ingress
  - Egress
  # 沒寫 ingress / egress 規則 = 全拒絕
allow-web-to-db.yaml · 只允許 web 連 db 
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: db-allow-web
  namespace: production
spec:
  podSelector:              # 保護誰
    matchLabels:
      app: db
  policyTypes:
  - Ingress
  ingress:                    # 誰可以進來
  - from:
    - podSelector:
        matchLabels:
          app: web
    ports:
    - protocol: TCP
      port: 5432

常見組合思路

  1. 先對每個 namespace 套一條 default-deny-all（Ingress + Egress）
  2. 允許對 kube-system 的 DNS 查詢（UDP/TCP 53）——否則應用無法解析名稱
  3. 依應用分層設定：frontend → backend → db 只允許單向
  4. 允許對外 HTTPS（egress TCP 443 to 0.0.0.0/0）讓應用能呼叫外部 API
allow-dns.yaml · 允許 DNS 是必做項 
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
    ports:
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53

本章速查

🎯 必記重點

  1. K8s 網路四規則：每 Pod 一個 IP、Pod 間可直連、Node 與 Pod 互通、Service 提供穩定入口。
  2. Service 四類：ClusterIP（內部）、NodePort（開 Node port）、LoadBalancer（雲端 LB）、ExternalName（DNS 別名）。
  3. Headless Service（clusterIP: None）DNS 直接回傳 Pod IP，StatefulSet 必備。
  4. Service → EndpointSlice → Pod IP，kube-proxy 讀 EndpointSlice 寫 iptables / IPVS。
  5. CoreDNS 負責 Cluster DNS，格式 <svc>.<ns>.svc.cluster.local。
  6. Ingress 是 L7 路由規則；Ingress Controller 才是實際執行者，必須先部署。
  7. 只為 Ingress Controller 開一個 LoadBalancer，其他服務透過 Ingress 暴露，最省錢。
  8. Gateway API 是新一代替代方案，支援更多協議、有角色分離。
  9. NetworkPolicy 預設「白名單」邏輯，沒規則 = 全拒絕；記得放行 DNS。
