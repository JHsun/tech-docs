CHAPTER 02 · Workloads

工作負載：Pod、Deployment 與朋友們

Workload 是你真正在跑的東西。本章帶你掌握 Pod 的生命週期、Deployment 如何做零停機更新、 StatefulSet 為何與眾不同、Job 與 CronJob 怎麼跑一次性與週期性任務。看完這章， 你就能回答「什麼場景用哪個 workload」這個經典面試題。

Workload 家族概觀

Workload（工作負載）泛指「在 Kubernetes 上運行的應用」。直接管理 Pod 非常麻煩—— Pod 本身不會自動重啟、不會跨 Node 遷移、也不會自動擴縮。所以 K8s 提供一系列工作負載控制器（Workload Controller）， 幫你管理一組 Pod 的生命週期。

Workload一句話說明典型場景
Pod最小部署單位幾乎不直接用，都由控制器管理
ReplicaSet維持 N 個同構 Pod幾乎不直接用，由 Deployment 管理
Deployment管理 ReplicaSet 做滾動更新無狀態應用 90% 場景（web、API）
StatefulSet有順序、有穩定身份的 Pod 集合資料庫、Kafka、ZooKeeper
DaemonSet每個 Node 跑一份日誌收集、監控、網路代理
Job跑完就結束批次運算、資料遷移、備份
CronJob定時跑 Job每日報表、定期清理
💡 選擇順序
90% 情境問自己三個問題：(1) 應用需要穩定身份嗎？→ StatefulSet；(2) 要跑完結束？→ Job / CronJob；(3) 都不是？→ Deployment。

Pod 是什麼

Pod 是 Kubernetes 的最小調度單位，不是「一個容器」而是「一組共享資源的容器」。 同一個 Pod 內的容器共享：

  • Network Namespace：共用 IP、port 空間，容器間可透過 localhost 互通
  • Storage Volumes：掛載同一個 Volume 做資料交換
  • IPC Namespace（可選）：共用訊號、共享記憶體
  • 生命週期：一起被排程到同一個 Node、一起被建立與銷毀
🎯 設計原則
一個 Pod 內應該只放一個主容器（加上必要的輔助 sidecar）。不要把「Web + DB」塞在同一個 Pod 裡——它們有各自的生命週期與擴縮需求。

最小 Pod YAML

pod.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: nginx-demo
  labels:
    app: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.27
    ports:
    - containerPort: 80
    resources:
      requests:
        cpu: "100m"
        memory: "128Mi"
      limits:
        cpu: "500m"
        memory: "256Mi"

Pod 生命週期

Pod 的 status.phase 欄位永遠處於以下五個狀態之一：

Pending
已建立，等待排程或拉映像
→
Running
至少一個容器在執行
→
Succeeded
所有容器成功結束
Failed
至少一個容器以非 0 結束
Unknown
無法取得狀態（通常 Node 失聯）

容器狀態（Container State）

Pod phase 是粗粒度，每個容器還有更細的狀態：

Container State意義
Waiting尚未執行（拉映像中、等待 ConfigMap/Secret 等）
Running正在執行中
Terminated已結束（不論成功或失敗），會記錄 exitCode

常見異常 Reason

Reason原因怎麼排查
ImagePullBackOff拉映像失敗檢查映像名稱、registry 憑證
CrashLoopBackOff容器一直重啟失敗kubectl logs --previous 看上次輸出
OOMKilled超過 memory limit 被殺調高 limit 或找記憶體洩漏
ContainerCreating卡在建立中kubectl describe pod 看 Events
CreateContainerConfigErrorConfigMap/Secret 不存在檢查 volume 與 envFrom 引用
⚠️ 重要澄清
restartPolicy 只影響「同一個 Pod 內的容器」是否重啟，不影響 Pod 本身。一個 Pod 死了就是死了，要靠上層控制器（Deployment / ReplicaSet）重建新 Pod。

多容器 Pod 模式

雖然建議一個 Pod 一個主容器，但有幾個經典多容器模式：

Sidecar

輔助主容器做額外工作：日誌收集、設定注入、流量代理（如 Istio proxy）。

Ambassador

作為主容器的「外交官」，代理對外部服務的連線（如連 DB 前的 proxy）。

Adapter

統一主容器的輸出格式（如把 app 日誌轉成 Prometheus 認識的 metrics）。

Init Containers

在主容器啟動之前依序執行的容器，全部成功結束後才會啟動 containers。 常用於：等待相依服務（如 DB 就緒）、預下載資料、初始化設定檔。

init-container.yaml · 等 DB 就緒再啟動 
apiVersion: v1
kind: Pod
metadata:
  name: app-with-init
spec:
  initContainers:
  - name: wait-for-db
    image: busybox:1.36
    command: ['sh', '-c',
      'until nslookup mysql.default.svc.cluster.local; do echo waiting; sleep 2; done']
  containers:
  - name: app
    image: myapp:1.0

ReplicaSet：維持副本數量

ReplicaSet 的唯一職責是「確保 Cluster 內永遠有 N 個符合 selector 的 Pod 在跑」。 少了就建，多了就殺。但實務上你幾乎不會直接寫 ReplicaSet——都透過 Deployment 間接使用。

📌 關係圖
Deployment 管理 → ReplicaSet 管理 → Pod
每次更新 Deployment，會產生新的 ReplicaSet，舊的 ReplicaSet 會被縮到 0（但保留以便回滾）。

Deployment：無狀態應用首選

Deployment 是你最常用的 workload。適合任何無狀態（Stateless）應用： 每個 Pod 都一樣、可互相替代、重啟沒差。

完整 Deployment 範例

deployment.yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  labels:
    app: web
spec:
  replicas: 3                  # 期望 Pod 數
  revisionHistoryLimit: 10      # 保留幾個歷史版本供回滾
  selector:                    # 找哪些 Pod 歸我管
    matchLabels:
      app: web
  strategy:                    # 更新策略
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1              # 最多可多出 1 個 Pod
      maxUnavailable: 0        # 更新過程中可容忍少幾個
  template:                    # Pod 模板
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: nginx:1.27
        ports:
        - containerPort: 80
        readinessProbe:
          httpGet:
            path: /
            port: 80

Rolling Update 運作機制

預設更新策略。K8s 會逐步替換舊 Pod 為新 Pod，過程中服務不中斷。 透過 maxSurge（最多可超出幾個）與 maxUnavailable（最多可少幾個）控制速度。

  1. 你用 kubectl set image 或編輯 spec.template 觸發更新。
  2. Deployment Controller 建立新 ReplicaSet（v2），副本數先為 1。
  3. 新 Pod 通過 Readiness Probe → 舊 ReplicaSet（v1）縮 1 個 Pod。
  4. 重複 3，直到新 ReplicaSet 全滿、舊的縮到 0。
  5. 舊 ReplicaSet 保留但副本為 0，供回滾用。

兩種策略比較

strategy.type行為適合
RollingUpdate（預設）逐步替換，零停機99% 情境
Recreate全部殺掉再全部新建，有停機不能多版本並存（如 DB schema 不相容）
💡 零停機關鍵
一定要設定 Readiness Probe！沒有的話，K8s 會以為 Pod 一啟動就 ready，導致流量打到還沒準備好的新 Pod。

更新與回滾指令

Bash · Deployment 操作 
# 更新映像版本
kubectl set image deployment/web web=nginx:1.28

# 查看滾動更新進度
kubectl rollout status deployment/web

# 查看歷史版本
kubectl rollout history deployment/web

# 查看某個版本的詳細內容
kubectl rollout history deployment/web --revision=2

# 回滾到上一個版本
kubectl rollout undo deployment/web

# 回滾到指定版本
kubectl rollout undo deployment/web --to-revision=2

# 暫停（批次改設定再一次 resume）
kubectl rollout pause deployment/web
kubectl rollout resume deployment/web

# 手動擴縮
kubectl scale deployment/web --replicas=5

StatefulSet：有狀態應用

Deployment 的 Pod 名稱是隨機的（如 web-7d9f8c-abc12），彼此可完全替換。 但資料庫、訊息佇列這種應用需要：

  • 穩定的網路身份：每個 Pod 有固定名稱與 DNS
  • 穩定的儲存：Pod 重建後還能連回原本的 Volume
  • 有序的部署 / 擴縮 / 刪除：Pod-0 起好才起 Pod-1

這就是 StatefulSet 的工作。

核心特徵

特徵DeploymentStatefulSet
Pod 名稱web-<hash>-<rand>db-0, db-1, db-2（固定順序）
DNS無個別 DNSdb-0.mysvc.ns.svc.cluster.local
儲存可共用或無每個 Pod 一個獨立 PVC
啟動順序同時建立0 → 1 → 2 依序
刪除順序同時反向 N → 0
statefulset.yaml · 簡化版 
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: db
spec:
  serviceName: db-headless    # 必須搭配 Headless Service
  replicas: 3
  selector:
    matchLabels:
      app: db
  template:
    metadata:
      labels:
        app: db
    spec:
      containers:
      - name: mysql
        image: mysql:8
        volumeMounts:
        - name: data
          mountPath: /var/lib/mysql
  volumeClaimTemplates:           # 每個 Pod 自動產生一個 PVC
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi
⚠️ 實務建議
如果你只是想跑一個「有狀態」的應用，優先考慮 雲端託管服務（RDS、Cloud SQL）或 Operator（Strimzi、Prometheus Operator）。自己寫 StatefulSet 管理資料庫的 HA、備份、升級——坑非常多。

DaemonSet：每個 Node 一份

DaemonSet 確保每個（符合條件的）Node 上都跑一個 Pod。新增 Node 會自動補一個、移除 Node 則跟著消失。

典型用途

  • 日誌收集：Fluentd、Fluent Bit、Promtail
  • 監控代理：node-exporter、Datadog Agent
  • 網路外掛：Calico、Cilium 的 per-node agent
  • 儲存外掛：CSI node plugin
daemonset.yaml · 每個 Node 一個 fluent-bit 
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluent-bit
  namespace: logging
spec:
  selector:
    matchLabels:
      app: fluent-bit
  template:
    metadata:
      labels:
        app: fluent-bit
    spec:
      tolerations:                # 讓它能跑在 control-plane node
      - key: node-role.kubernetes.io/control-plane
        operator: Exists
        effect: NoSchedule
      containers:
      - name: fluent-bit
        image: fluent/fluent-bit:3.0
        volumeMounts:
        - name: varlog
          mountPath: /var/log
      volumes:
      - name: varlog
        hostPath:
          path: /var/log

Job：一次性任務

Job 跑完就結束，不像 Deployment 永久維持。適合資料遷移、批次運算、備份等。

三種模式

模式設定行為
Non-parallel預設一個 Pod 跑完就結束
Fixed completion countcompletions: 5總共要成功 5 次
Work queueparallelism: 3同時 3 個 Pod 從佇列拿任務，自行決定何時停
job.yaml · 算圓周率 
apiVersion: batch/v1
kind: Job
metadata:
  name: pi
spec:
  completions: 1             # 要成功幾次
  parallelism: 1             # 同時幾個 Pod
  backoffLimit: 3            # 失敗最多重試幾次
  activeDeadlineSeconds: 300  # 最多跑多久（超過直接失敗）
  template:
    spec:
      restartPolicy: OnFailure   # Job 只能是 OnFailure 或 Never
      containers:
      - name: pi
        image: perl:5.38
        command: ["perl", "-Mbignum=bpi", "-wle", "print bpi(2000)"]

CronJob：定時任務

CronJob 依 cron 表達式週期性建立 Job。就是 K8s 版的 crontab。

cronjob.yaml · 每分鐘印 hello 
apiVersion: batch/v1
kind: CronJob
metadata:
  name: hello
spec:
  schedule: "*/1 * * * *"            # 分 時 日 月 週
  timeZone: "Asia/Taipei"            # K8s v1.27+ 支援
  concurrencyPolicy: Forbid         # 上一輪還沒跑完就跳過
  successfulJobsHistoryLimit: 3    # 保留幾個成功歷史
  failedJobsHistoryLimit: 1
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: hello
            image: busybox:1.36
            command: ["echo", "Hello from K8s CronJob"]

concurrencyPolicy 三種值

值行為適合
Allow（預設）允許同時多個 Job 跑互不影響的任務
Forbid上一個沒跑完就跳過這次不能重入的任務（如備份）
Replace殺掉上一個，啟動新的只關心最新結果
⚠️ Cron 坑
K8s 的 CronJob 不保證 100% 準時執行（可能晚幾秒）。對精準度敏感的任務請用外部排程器（Airflow、Argo Workflows）。

決策樹：到底該用哪個

Workload 選擇決策樹 
你的應用是什麼？
├── 要跑完結束？
│   ├── 定期重複 ───────── CronJob
│   └── 只跑一次 ───────── Job
├── 需要每個 Node 一份？ ── DaemonSet
├── 有狀態、需固定身份？
│   └── 資料庫、佇列 ─── StatefulSet
└── 無狀態、可隨意擴縮？
    └── Web、API ─────── Deployment  ← 90% 情境

本章速查

🎯 必記重點

  1. Pod 是最小調度單位，不是容器；同 Pod 容器共享網路與 Volume。
  2. Pod 本身不會自動重建，必須靠控制器（Deployment / StatefulSet 等）。
  3. Deployment = ReplicaSet 包裝 + 滾動更新邏輯，幾乎不會直接寫 ReplicaSet。
  4. RollingUpdate 的核心是 maxSurge 與 maxUnavailable；Recreate 會有停機。
  5. StatefulSet 三特徵：固定名稱、獨立儲存、有序操作。
  6. DaemonSet 適合 per-node agent（日誌、監控、網路）。
  7. Job 要跑完、CronJob 要定時；restartPolicy 只能是 OnFailure 或 Never。
  8. 任何 Pod 都應設 resources.requests/limits 與 readinessProbe。

🔧 常用指令

指令用途
kubectl get pods -o wide列 Pod 含 Node 與 IP
kubectl describe pod <name>看 Events 與詳細狀態
kubectl logs <pod> [-c container]看容器日誌
kubectl logs <pod> --previous看上次重啟前的日誌（除 CrashLoopBackOff 神器）
kubectl exec -it <pod> -- sh進入容器
kubectl rollout status deployment/x看更新進度
kubectl rollout undo deployment/x回滾
kubectl scale deployment/x --replicas=5手動擴縮
kubectl create job --from=cronjob/x manual-run手動觸發 CronJob 一次
