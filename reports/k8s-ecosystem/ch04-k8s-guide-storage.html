CHAPTER 04 · 儲存

Storage：讓資料活得比容器久

容器天生短命，Pod 一重啟資料就沒了。本章從 Volume 基本概念出發， 拆解 PV/PVC/StorageClass 三層抽象如何協作、CSI 怎麼讓各廠商儲存方案統一接入、 以及在實務上如何給 StatefulSet 配獨立儲存、做備份與擴容。

為何需要儲存抽象

容器內部檔案系統是臨時的——容器砍掉重建（K8s 常態），裡面寫的東西全部消失。 而資料庫、上傳檔案、快取這些場景顯然不能這樣。K8s 的儲存體系要解決三個痛點：

🔄 生命週期解耦

資料應該活得比 Pod 久。Pod 重建、遷移到其他 Node，資料還在。

🔌 儲存後端可抽換

應用不該綁定 AWS EBS 或 Ceph。換雲、換儲存方案時 YAML 不用改。

👥 角色分離

管理員負責供給儲存、開發者只負責「我要 10GB」，不用懂底層。

🧭 三層抽象
Volume（Pod 內掛載點）→ PersistentVolume（Cluster 層級儲存資源）→ StorageClass（動態配置模板）。由下往上看是「實體→資源→模板」，由上往下看是「我要什麼→系統配一個→掛進 Pod」。

Volume 基本概念

Volume 定義在 Pod.spec.volumes，然後在容器內 volumeMounts 指定掛載路徑。 Volume 的生命週期綁定 Pod——Pod 消失，某些 Volume（如 emptyDir）就消失； 某些（如 PVC 掛載的）則獨立存在。

volume-basics.yaml · 最小掛載範例 
apiVersion: v1
kind: Pod
metadata:
  name: web
spec:
  containers:
  - name: nginx
    image: nginx:1.27
    volumeMounts:
    - name: html          # 掛哪一個 volume
      mountPath: /usr/share/nginx/html  # 掛到哪裡
      readOnly: true
  volumes:
  - name: html             # Pod 內的 volume 宣告
    emptyDir: {}          # 使用哪種後端

常見 Volume 類型

類型說明生命週期
emptyDirPod 排程時建立的空目錄，容器間共享與 Pod 同生共死
hostPath掛 Node 本機路徑進 Pod資料留在 Node 上
configMap / secret把 ConfigMap/Secret 內容掛成檔案隨 Pod
downwardAPI把 Pod 自身資訊（label、resource）掛成檔案隨 Pod
persistentVolumeClaim引用一個 PVC（主角）獨立於 Pod
projected把多個來源合併到同一個掛載點隨 Pod
csi透過 CSI 驅動對接外部儲存依 PV 設定
⚠️ 已棄用或少用
awsElasticBlockStore、gcePersistentDisk、azureDisk 這些樹內（in-tree）驅動已逐步被 CSI（樹外）取代。新專案一律用 CSI，不要在 Pod 直接寫雲端特定的 Volume 類型。

emptyDir 與 hostPath

emptyDir：Pod 內多容器共享的臨時空間

用途：sidecar 寫日誌主容器讀、多階段處理中間檔、大量小檔用記憶體當 tmpfs 加速。

emptydir.yaml · tmpfs 模式 
volumes:
- name: cache
  emptyDir:
    medium: Memory          # 用 tmpfs（佔 Pod memory limit）
    sizeLimit: 500Mi       # 最大容量

hostPath：連接 Node 檔案系統

通常只在 DaemonSet（日誌收集、節點監控）使用。一般應用應避免——會綁定 Pod 到特定 Node、破壞可攜性、且是安全風險。

🚫 安全警告
hostPath 讓 Pod 能讀寫 Node 上的任意檔案（含 /etc、docker.sock）。PSS 的 Baseline / Restricted 等級禁止使用。生產環境應改用 PVC 或 CSI ephemeral volume。

PersistentVolume 與 PersistentVolumeClaim

PersistentVolume（PV）是 Cluster 層級的儲存資源，由管理員預先建立（靜態）或 StorageClass 動態產生。 PersistentVolumeClaim（PVC）是使用者的「要求單」——「我要 10Gi、可寫一次」。 K8s 負責配對：找一個符合需求的 PV 綁定給這個 PVC。

PV（供給側）

管理員視角：「Cluster 有哪些可用的儲存？」。描述容量、access mode、後端類型（NFS、EBS、Ceph...）。

PVC（需求側）

開發者視角：「我要多大、什麼 access mode」。不關心底層是 NFS 還是 EBS。

靜態配置：預先建立 PV

static-pv.yaml · NFS 範例 
apiVersion: v1
kind: PersistentVolume
metadata:
  name: nfs-pv-10g
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteMany
  persistentVolumeReclaimPolicy: Retain
  nfs:
    server: nfs.internal.lan
    path: /exports/data
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data-claim
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 10Gi
  storageClassName: ""        # 空字串 = 不用動態配置

在 Pod 內引用 PVC

pod-with-pvc.yaml 
apiVersion: v1
kind: Pod
metadata:
  name: web
spec:
  containers:
  - name: nginx
    image: nginx
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: data-claim

PV 生命週期：五個狀態

Avail
able
Available

PV 已建立，但尚未被任何 PVC 綁定。等待配對。

Bound
Bound

已與某個 PVC 綁定，獨佔關係（不能再綁給其他 PVC）。

Rele
ased
Released

PVC 被刪除，但 PV 仍保留資料（依 ReclaimPolicy 決定後續）。

Failed
Failed

自動回收過程失敗，需要人工處理。

Termin
ating
Terminating

PV 已被刪除但仍在清理中（資源釋放、雲端磁碟銷毀）。

AccessModes：誰能掛、怎麼掛

模式縮寫含義典型後端
ReadWriteOnceRWO單一 Node 可讀寫區塊儲存（EBS、GCE PD）
ReadOnlyManyROX多 Node 可唯讀NFS、CephFS
ReadWriteManyRWX多 Node 可讀寫NFS、CephFS、EFS
ReadWriteOncePodRWOP單一 Pod 可讀寫（v1.22+）需 CSI 支援
💡 常見誤會
RWO 不是「只能被一個 Pod 掛」，是「只能被一個 Node 掛」。同 Node 上的多個 Pod 都可以同時掛同一個 RWO PV。要限制到單一 Pod 才用 RWOP。

ReclaimPolicy：PVC 刪了之後呢

策略PVC 刪除後適合
RetainPV 保留、資料保留、狀態變 Released，需人工處理重要資料，不能誤刪
DeletePV 與底層儲存（EBS/磁碟）一併銷毀測試環境、臨時資料
Recycle（已棄用）舊版的 rm -rf 再重用不要用
⚠️ 預設行為
動態配置的 PV 預設為 Delete——刪 PVC 就等於刪資料！生產環境建議改 StorageClass 的 reclaimPolicy: Retain，或至少對關鍵 PV 手動改為 Retain。

StorageClass：動態配置的模板

靜態配置要管理員先建 PV 非常麻煩。StorageClass 讓 PVC 可以自動觸發 PV 的建立—— 你說要 20Gi，StorageClass 背後的 Provisioner 就自動去雲端開一顆磁碟、建一個 PV、綁給 PVC。

storageclass.yaml · AWS gp3 
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: gp3
  annotations:
    storageclass.kubernetes.io/is-default-class: "true"
provisioner: ebs.csi.aws.com      # 誰來建磁碟
parameters:                      # 傳給 provisioner 的參數
  type: gp3
  iops: "3000"
  throughput: "125"
  encrypted: "true"
reclaimPolicy: Retain            # 關鍵：PVC 刪了也保留
allowVolumeExpansion: true      # 允許擴容
volumeBindingMode: WaitForFirstConsumer  # 等 Pod 排程好再配磁碟

用 StorageClass 的 PVC

dynamic-pvc.yaml 
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: app-data
spec:
  storageClassName: gp3     # 指定用哪個 SC；省略則用預設
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 20Gi

volumeBindingMode 兩種值

模式時機問題 / 優勢
Immediate（預設）PVC 一建立就配磁碟可能配到 Pod 跑不到的 AZ（多可用區環境有坑）
WaitForFirstConsumer等 Pod 排程到某 Node 再配避免 AZ 不匹配，多 AZ 環境建議值
💡 一份完整的配置
多 AZ Cluster + 雲端 EBS 建議組合：WaitForFirstConsumer + Retain + allowVolumeExpansion: true。避免 AZ 錯配、保住資料、允許擴容。

CSI：儲存界的標準插座

CSI（Container Storage Interface）是 K8s 與儲存方案的標準介面。儲存廠商只要實作 CSI driver， K8s 就能透過統一方式呼叫它建立、掛載、快照、擴容磁碟。

CSI 架構兩部分

Controller Plugin

以 Deployment 形式跑，負責「建立 / 刪除 / 快照 / 擴容」等 cluster 層級操作。

Node Plugin

以 DaemonSet 形式跑在每個 Node，負責「掛載 / 卸載」到實際容器路徑。

常見 CSI Driver

Driver用途
ebs.csi.aws.comAWS EBS（單 AZ、RWO）
efs.csi.aws.comAWS EFS（多 AZ、RWX）
pd.csi.storage.gke.ioGCP Persistent Disk
disk.csi.azure.comAzure Disk
rook-ceph.rbd.csi.ceph.comCeph RBD（地端首選）
nfs.csi.k8s.ioNFS
local.csi.openebs.io本機磁碟（Local PV）

StatefulSet + PV 實戰：每個 Pod 獨立儲存

StatefulSet 的殺手級欄位是 volumeClaimTemplates。每個 Pod 會自動生成自己的 PVC， 即使 Pod 重建也會掛回原本那個 PV。

statefulset-mysql.yaml · 三副本資料庫 
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: mysql-headless
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8
        env:
        - name: MYSQL_ROOT_PASSWORD
          valueFrom:
            secretKeyRef:
              name: mysql-secret
              key: root-password
        volumeMounts:
        - name: data
          mountPath: /var/lib/mysql
  volumeClaimTemplates:       # 每個 Pod 自動對應一個 PVC
  - metadata:
      name: data
    spec:
      accessModes: [ReadWriteOnce]
      storageClassName: gp3
      resources:
        requests:
          storage: 20Gi

部署後你會看到：

Bash · 自動產生的 PVC 
$ kubectl get pvc
NAME            STATUS   VOLUME       CAPACITY   ACCESS MODES   STORAGECLASS
data-mysql-0    Bound    pvc-abc...   20Gi       RWO            gp3
data-mysql-1    Bound    pvc-def...   20Gi       RWO            gp3
data-mysql-2    Bound    pvc-ghi...   20Gi       RWO            gp3
⚠️ 刪 StatefulSet 的坑
kubectl delete statefulset mysql 只會刪 Pod，不會刪 PVC（保護資料）。要完全清理，必須另外 kubectl delete pvc -l app=mysql。K8s v1.27+ 可設 persistentVolumeClaimRetentionPolicy 自動管理。

VolumeSnapshot：原生備份與還原

K8s 定義了 VolumeSnapshot 機制，前提是 CSI driver 支援。典型流程：建 snapshot → 從 snapshot 建新 PVC。

snapshot.yaml · 建立快照 
# 1. 先要有一個 VolumeSnapshotClass（類似 StorageClass）
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshotClass
metadata:
  name: ebs-snapshot
driver: ebs.csi.aws.com
deletionPolicy: Retain
---
# 2. 建一個快照
apiVersion: snapshot.storage.k8s.io/v1
kind: VolumeSnapshot
metadata:
  name: mysql-snap-2026-04-24
spec:
  volumeSnapshotClassName: ebs-snapshot
  source:
    persistentVolumeClaimName: data-mysql-0
---
# 3. 從快照還原成新 PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data-mysql-restored
spec:
  storageClassName: gp3
  dataSource:
    name: mysql-snap-2026-04-24
    kind: VolumeSnapshot
    apiGroup: snapshot.storage.k8s.io
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 20Gi
💡 生產級備份
VolumeSnapshot 適合快速還原場景，但不是完整備份方案（仍在同一地區）。生產環境建議搭配 Velero（跨 Cluster 備份、跨雲遷移）或雲端原生備份服務。

擴容與遷移

線上擴容 PVC

前提：StorageClass 設定 allowVolumeExpansion: true，且底層 driver 支援。

  1. 編輯 PVC 的 spec.resources.requests.storage，改大（不能改小）。
  2. 等 driver 完成底層磁碟擴容（可看 kubectl describe pvc 的 events）。
  3. 某些 filesystem 擴容需要 Pod 重啟，有些可線上完成（ext4、XFS 都支援線上擴容）。
Bash · 擴容指令 
# 10Gi → 20Gi
kubectl patch pvc data-mysql-0 -p '{"spec":{"resources":{"requests":{"storage":"20Gi"}}}}'

# 查看狀態
kubectl describe pvc data-mysql-0 | grep -A 5 Conditions

跨 Cluster 遷移

  • Velero：備份 PV 與相關資源，到目標 Cluster 還原
  • CSI cloning：dataSource 指向另一個 PVC，快速複製同一 Cluster 內的資料
  • Pipe tools：開 Pod tar 打包、傳到新環境解壓（適合小量資料）

本章速查

🎯 必記重點

  1. 三層抽象：Volume（Pod 內掛載）→ PV（Cluster 資源）→ StorageClass（動態配置模板）。
  2. emptyDir 與 Pod 同生命週期；hostPath 有安全風險只在特定場景用。
  3. 生產環境一律走 PVC，別在 Pod 直接寫雲端 Volume 類型。
  4. AccessModes：RWO 是「單 Node」不是「單 Pod」；RWX 需特殊後端（NFS、CephFS、EFS）。
  5. 動態配置預設 reclaimPolicy: Delete，生產資料請改 Retain。
  6. 多 AZ 環境用 volumeBindingMode: WaitForFirstConsumer。
  7. CSI 取代舊 in-tree 驅動，Controller + Node 兩部分。
  8. StatefulSet 的 volumeClaimTemplates 每個 Pod 一個獨立 PVC。
  9. 刪 StatefulSet 不會刪 PVC（保護資料）——要清乾淨需手動。
  10. 擴容需 allowVolumeExpansion: true，且只能變大不能變小。
