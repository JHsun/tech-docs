CHAPTER 01 · 入門與架構

從零理解 Kubernetes

與其急著背 YAML，先建立正確心智模型。本章從 Kubernetes 的定位出發， 拆解 Cluster 內每一個組件的職責，並用一次完整的請求流程， 讓你看懂「kubectl apply」背後發生了什麼事。

Kubernetes 是什麼

Kubernetes（常縮寫為 K8s，K 與 s 中間省略 8 個字母）是一個 開源的容器編排平台（Container Orchestration Platform）， 由 Google 於 2014 年基於內部 Borg 系統經驗釋出，現交由 CNCF（Cloud Native Computing Foundation）管理。

官方一句話定義可拆成三個關鍵詞：

Portable 可攜

一份 YAML 可在筆電、地端機房、AWS、GCP、Azure 執行，不綁定特定雲平台。

Extensible 可擴充

透過 CRD 與 Operator 可自訂資源類型，K8s 本身就是一個「管理控制器的平台」。

Self-healing 自癒

Pod 崩潰自動重啟、Node 故障自動遷移、透過控制迴圈持續把實際狀態收斂到你宣告的期望狀態。

💡 一句話記憶
Kubernetes = 容器的作業系統。就像 Linux 幫你管理行程（Process），K8s 幫你管理容器（Container）——排程、重啟、連網、擴縮、升級全包。

為何需要 Kubernetes

單機跑 Docker 很簡單，為什麼還需要 K8s？答案在「規模」與「可用性」：

問題純 Docker 的做法Kubernetes 的做法
容器掛了怎麼辦？ 人工 docker restart 自動重啟（restartPolicy）、不健康則重建 Pod
流量變大要擴增？ 人工起 N 個 container + 改 Nginx kubectl scale 或 HPA 自動擴縮
新版本上線 停舊起新，可能斷線 Rolling Update 零停機 / 可回滾
多台機器間分派 自己 ssh 進去決定 Scheduler 依資源與策略自動選 Node
服務發現 硬編碼 IP 或維護 DNS Service + 內建 DNS（CoreDNS）
設定與機敏資料 混在映像或環境變數 ConfigMap / Secret 與映像解耦

核心概念速覽

閱讀後續章節前，先對這幾個名詞有個印象。每個都會在後續章節展開。

名詞一句話定義
Cluster一組由 K8s 管理的機器，包含 Control Plane 與 Node。
Node實際跑容器的機器（虛擬機或實體機）。
PodK8s 最小部署單位，內含 1 個或多個共享網路的 Container。
Deployment管理一組相同 Pod 的副本與版本更新的控制器。
Service把一組 Pod 包裝成一個穩定的網路端點（ClusterIP、NodePort、LoadBalancer）。
IngressHTTP/HTTPS 層的入口路由，把外部網址導向不同 Service。
NamespaceCluster 內的「虛擬資料夾」，用於隔離資源（如 dev / prod）。
ConfigMap / Secret把設定與機敏資料從映像檔抽離。
Volume / PV / PVC容器的持久化儲存機制。
Label / SelectorK8s 資源互相關聯的膠水：用標籤選出目標。

Cluster 整體架構

一個 Kubernetes Cluster 分成兩個平面（Plane）：Control Plane（控制平面，俗稱 Master） 與 Data Plane（資料平面，由一或多個 Worker Node 組成）。 這兩個平面各司其職——一個負責「思考」，一個負責「執行」。

Kubernetes Cluster 架構總覽
🧠 Control Plane（思考層）
kube-apiserver
etcd
kube-scheduler
controller-manager
cloud-controller-manager
⚙️ Worker Node ×N（執行層）
kubelet
kube-proxy
Container Runtime

Control Plane 組件

kube-apiserver

整個 Cluster 的唯一入口與中央匯流排。所有組件——不論是 kubectl、kubelet、controller——都只能透過 apiserver 互相溝通， 不能跳過它直接改 etcd。這也是為何 kubectl 本質上就是一個 apiserver 的 HTTP client。

  • 提供 RESTful API（/api/v1/pods、/apis/apps/v1/deployments ...）
  • 做 Authentication（你是誰）→ Authorization（你能做什麼）→ Admission Control（這個請求合法嗎）
  • 可水平擴充多副本以提升可用性

etcd

Cluster 的唯一真實來源（Single Source of Truth）。一套分散式 key-value 儲存， 保存所有 K8s 資源的期望狀態與當前狀態。

⚠️ 運維重點
etcd 掛了 = 整個 Cluster 停擺。備份 etcd 是 K8s 管理員的首要任務，建議每日自動快照。

kube-scheduler

負責回答一個問題：「這個新 Pod 應該放到哪個 Node 上跑？」 過程分兩階段：

  1. Filtering：依資源需求（CPU / Memory）、Taint、Node Selector、Affinity 等條件過濾掉不合格的 Node。
  2. Scoring：對剩下的 Node 打分，選最高分的那一個。

kube-controller-manager

內含多個「控制器」（Controller），每個都是一個無限迴圈，不斷比對期望狀態與實際狀態，有差異就動作修正。 著名成員：

Controller職責
Deployment Controller確保 Deployment 宣告的 Pod 副本數維持正確
ReplicaSet Controller維持 Pod 副本數量（Deployment 底層呼叫它）
Node Controller監控 Node 健康，失聯時驅逐 Pod
Job Controller執行一次性任務至完成
Endpoints Controller維護 Service 對應的 Pod IP 清單

cloud-controller-manager

僅在雲端部署時出現。把與雲供應商（AWS、GCP、Azure）綁定的邏輯（如建立 LoadBalancer、管理 Node 生命週期）從 core controller 中抽離。

Worker Node 組件

kubelet

每個 Node 上的代理人。向 apiserver 取得「該跑哪些 Pod」，然後呼叫容器執行時把它們跑起來，並定期回報狀態。 可以把它想成 Node 版的「雙手」，負責執行 Control Plane 的命令。

🔑 關鍵認知
kubelet 只管 Pod，不認識 Deployment、Service 這些「高層概念」。它只知道「apiserver 叫我跑某個 Pod」。

kube-proxy

實作 Service 的網路魔法。當你建立一個 Service，kube-proxy 會在每個 Node 上安裝 iptables / IPVS 規則， 讓任何指向 ClusterIP 的流量自動被導向該 Service 的某個後端 Pod。

Container Runtime

實際負責啟停容器的引擎，必須符合 CRI（Container Runtime Interface）規範。

Runtime說明
containerd目前最主流，從 Docker 中抽離出來的核心，輕量
CRI-O專為 K8s 設計，Red Hat / OpenShift 使用
Docker EngineK8s v1.24 起已移除 dockershim，不再原生支援

CNI Plugin（Container Network Interface）

K8s 本身不負責 Pod 間的網路，交給 CNI 外掛實作。常見選擇：Calico、Cilium、Flannel、Weave。

CSI Plugin（Container Storage Interface）

儲存插件規範。讓 K8s 以統一介面接各種儲存方案（AWS EBS、Ceph、NFS、本地硬碟等）。

宣告式 API：K8s 的核心哲學

傳統 shell script 是命令式（Imperative）：「做 A，再做 B，再做 C」。 K8s 則是宣告式（Declarative）：「我要最終長這個樣子，你想辦法達成」。

❌ 命令式

docker run -d --name web -p 80:80 nginx
→ 你下一次還得自己記得下這條指令

✅ 宣告式

kubectl apply -f deployment.yaml
→ YAML 描述「我要 3 個 nginx」，K8s 會自動維持

K8s 內部由大量控制迴圈（Control Loop / Reconcile Loop）驅動。 每個 Controller 都在重複做一件事：

控制迴圈 · 偽代碼 
for {
  desired := getDesiredState()    // 從 apiserver 讀你宣告的樣子
  current := getCurrentState()    // 觀察現在實際的樣子
  if desired != current {
    reconcile(desired, current)   // 採取行動讓兩者一致
  }
  sleep(interval)
}
💡 心智模型
K8s 像一個「有強迫症的管家」：你貼一張字條「廚房要永遠有 3 條麵包」，牠就會不斷檢查、發現少了就去買。你不需要下指令，只需要更新字條。

實例：一個 kubectl apply 的完整旅程

當你執行 kubectl apply -f deploy.yaml（建立一個 3 副本的 Nginx Deployment），背後發生了什麼？

  1. kubectl：解析 YAML，驗證格式，送 HTTPS 請求到 kube-apiserver。
  2. apiserver：AuthN 確認你是誰 → AuthZ 確認你能建立 Deployment → Admission 檢查合法性 → 寫入 etcd。
  3. Deployment Controller 透過 watch 發現新 Deployment，建立對應的 ReplicaSet。
  4. ReplicaSet Controller 發現需要 3 個 Pod，建立 3 個 Pod 物件寫入 etcd（此時 Pod 尚未綁定 Node，狀態為 Pending）。
  5. Scheduler 透過 watch 看到未排程的 Pod，依策略選出合適 Node，把 spec.nodeName 回寫 apiserver。
  6. kubelet（被選中那台 Node 上的）看到自己 Node 被指派新 Pod，呼叫 Container Runtime 拉映像並啟動容器。
  7. kubelet 持續回報 Pod 狀態給 apiserver，更新到 etcd。
  8. 若有 Service 指向這些 Pod，kube-proxy 更新 iptables，讓流量可以導入。
📌 核心觀察
組件之間不直接互相呼叫，全部透過 apiserver watch 機制協調。這就是 K8s 能做到鬆耦合、可擴充的關鍵設計。

K8s Object 結構：四大欄位

不論哪一種資源（Pod、Service、ConfigMap...），YAML 幾乎都長這個骨架：

YAML · 通用骨架 
apiVersion: apps/v1           # 哪一個 API 群組與版本
kind: Deployment              # 資源類型
metadata:                     # 識別資訊
  name: nginx
  namespace: default
  labels:
    app: nginx
spec:                         # 期望狀態（由使用者定義）
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.27
status:                       # 實際狀態（由 K8s 維護，你不用寫）
  readyReplicas: 3
欄位功能誰寫
apiVersion指定 API 群組與版本，例如 v1、apps/v1使用者
kind資源類型（Pod、Deployment、Service...）使用者
metadata名稱、namespace、標籤、註解使用者
spec期望狀態（你想要的樣子）使用者
status實際狀態（現在真正的樣子）K8s 自動維護
💡 查詢 API 文件小技巧
忘記某個欄位怎麼寫？用 kubectl explain 直接查：
kubectl explain deployment.spec.template.spec.containers

建立本地實驗環境

學習階段不需要搞真正的多機 Cluster，三個主流選擇：

工具特點適合
minikube 單節點，支援多種驅動（Docker、VM、Podman） 初學、教學
kind Kubernetes IN Docker，啟動極快，可多節點 CI 測試、快速實驗
Docker Desktop 勾選啟用即可，開箱即用 Mac / Windows 開發機
k3s / k3d 輕量版 K8s，資源佔用少 Edge、IoT、樹莓派

快速上手：kind 範例

Bash · 建立本機 Cluster 
# 1. 安裝 kubectl 與 kind（macOS）
brew install kubectl kind

# 2. 建立一個名為 dev 的 cluster
kind create cluster --name dev

# 3. 確認連線
kubectl cluster-info
kubectl get nodes

# 4. 跑一個測試 Pod
kubectl run hello --image=nginx --port=80
kubectl get pods

# 5. 刪除 cluster
kind delete cluster --name dev

本章速查

🎯 必須記住的 10 件事

  1. K8s 是容器的作業系統，解決規模與可用性問題。
  2. Cluster = Control Plane + 多個 Worker Node。
  3. apiserver 是唯一入口，etcd 是唯一真相來源。
  4. kubelet 在每台 Node 上跑 Pod，kube-proxy 負責 Service 網路。
  5. K8s 是宣告式：你說「要什麼」，它負責「達成」。
  6. 控制迴圈不斷比對 desired vs current 狀態。
  7. Pod 是最小單位，不是 Container。
  8. 組件透過 apiserver watch 協作，彼此不直連。
  9. YAML 四大欄位：apiVersion、kind、metadata、spec。
  10. 學習環境首選 kind 或 minikube，別一開始就搞生產叢集。
