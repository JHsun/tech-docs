CHAPTER 06 · 排程、擴縮、觀測

走向生產級叢集

前五章讓 Pod「跑起來」；這章讓 Pod「跑得好」。包含四大主題：怎麼讓 Pod 排到合適的 Node、 流量變大時如何自動擴縮、如何確保容器真的可服務、以及怎麼看見 Cluster 內部發生了什麼。 這是從「會用」到「能營運」的關鍵跳躍。

Part 01
Scheduling · 排程控制
決定 Pod 跑在哪個 Node 上，才能做到資源隔離、高可用、合規

Scheduler 運作流程

CH.01 講過 kube-scheduler 分兩階段工作，這裡展開細節。當一個 Pod 處於 Pending 且 spec.nodeName 為空時：

  1. Filtering（過濾）：套用一連串 predicate，淘汰不合格的 Node。例如資源不足、Node 壞了、Taint 不匹配、Pod 指定 nodeSelector 不符。
  2. Scoring（評分）：對剩下的 Node 用多個 plugin 打分（0-100），通常加權平均。
  3. Binding（綁定）：選擇最高分 Node，把 nodeName 寫回 apiserver。kubelet 發現後開始拉映像、啟動容器。
🧭 排程控制三個層次
由寬到嚴：資源請求（最基本）→ NodeSelector / Affinity（表明偏好）→ Taint / Toleration（Node 拒絕不合格的 Pod）。一個正確設計的生產 Cluster 會三者並用。

Resource Requests 與 Limits

requests：Pod 保證分到的資源，也是 scheduler 找 Node 的依據。
limits：Pod 最多能用到的上限，超過會被限流（CPU）或 OOMKilled（Memory）。

resources.yaml 
containers:
- name: app
  image: myapp:1.0
  resources:
    requests:            # Scheduler 用這個找 Node
      cpu: "250m"      # 0.25 核
      memory: "256Mi"
    limits:              # Runtime 強制上限
      cpu: "1000m"     # 1 核
      memory: "512Mi"

三種 QoS 等級（自動判定）

QoS Class條件優先級OOM 時誰先死
Guaranteed每個容器 requests == limits（CPU 與 memory 都有設）最高最後
Burstable至少有一個 request 或 limit，但不滿足 Guaranteed中等中間
BestEffort完全沒設 requests/limits最低最先
⚠️ CPU limit 的爭議
CPU 用 CFS throttling 限流，實務上高延遲敏感應用常因此遇到 tail latency 飆升。社群普遍建議：設 request 但不設 CPU limit（或設很寬鬆）。Memory limit 仍必設，避免失控的記憶體洩漏拖垮 Node。

NodeSelector：最簡單的限定

只跑在有特定 label 的 Node 上。語法簡單，但功能陽春（只支援 AND、無法模糊匹配）。

node-selector.yaml 
# 先給 Node 加 label
kubectl label nodes worker-1 disktype=ssd gpu=nvidia-t4

# Pod 指定要跑在 SSD + GPU 機器
spec:
  nodeSelector:
    disktype: ssd
    gpu: nvidia-t4

Affinity / Anti-Affinity：進階排程

Affinity 分兩類：nodeAffinity（對 Node 的偏好）與 podAffinity / podAntiAffinity（相對於其他 Pod）。 每種都有兩個嚴格度：

required...DuringScheduling

硬性規則。找不到符合的 Node，Pod 卡在 Pending。

preferred...DuringScheduling

偏好。找不到理想 Node 會退而求其次，仍能排程。

nodeAffinity：替代 nodeSelector 的加強版

node-affinity.yaml 
spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: topology.kubernetes.io/zone
            operator: In
            values: [ap-northeast-1a, ap-northeast-1c]
          - key: node.kubernetes.io/instance-type
            operator: NotIn
            values: [t2.micro, t3.nano]
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        preference:
          matchExpressions:
          - key: disktype
            operator: In
            values: [nvme]

podAntiAffinity：散開部署避免單點故障

典型用途：同一個 Deployment 的 Pod 分散在不同 Node。

pod-antiaffinity.yaml 
spec:
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchLabels:
            app: web       # 避開也帶 app=web 標籤的 Pod
        topologyKey: kubernetes.io/hostname   # 以 Node 為粒度散開
💡 topologyKey 是關鍵
常用值：kubernetes.io/hostname（散到不同 Node）、topology.kubernetes.io/zone（散到不同 AZ）、topology.kubernetes.io/region（散到不同 region）。生產 Cluster 通常要求副本至少散布於不同 AZ。

Taint 與 Toleration：Node 反向拒絕

Affinity 是 Pod 挑 Node；Taint 則相反——Node 宣告「我有某個標記，只有能『忍受』的 Pod 才能來」。

三種 effect

effect對新 Pod對已跑的 Pod
NoSchedule沒 toleration 則不排程不影響
PreferNoSchedule盡量不排，但可以不影響
NoExecute不排程驅逐出去！
Bash · 常見 Taint 操作 
# 專用 GPU Node：只有請求 GPU 的 Pod 才能進
kubectl taint nodes gpu-1 dedicated=gpu:NoSchedule

# Control plane Node：預設已有 Taint
kubectl describe node master | grep Taints
# Taints: node-role.kubernetes.io/control-plane:NoSchedule

# 移除 Taint（末尾加 -）
kubectl taint nodes gpu-1 dedicated=gpu:NoSchedule-

對應的 Toleration

toleration.yaml 
spec:
  tolerations:
  - key: dedicated
    operator: Equal
    value: gpu
    effect: NoSchedule
  # 容忍 Node 臨時不健康最多 60 秒
  - key: node.kubernetes.io/not-ready
    operator: Exists
    effect: NoExecute
    tolerationSeconds: 60
🧠 記憶法
Affinity = Pod 說「我要去這種 Node」；Taint = Node 說「只有特定 Pod 能來」。兩者結合：GPU Node 打 Taint（排斥一般 Pod），GPU Pod 同時有 Toleration + nodeAffinity（鎖定 GPU Node）。

Topology Spread Constraints：更均衡的散布

podAntiAffinity 只能說「不要跟誰同 Node」，但無法精細控制「每個 AZ 最多幾個」。 Topology Spread Constraints（v1.19+）解決這個。

spread.yaml · 跨 AZ 均衡部署 
spec:
  topologySpreadConstraints:
  - maxSkew: 1                     # AZ 間 Pod 數差距最多 1
    topologyKey: topology.kubernetes.io/zone
    whenUnsatisfiable: DoNotSchedule  # 或 ScheduleAnyway
    labelSelector:
      matchLabels:
        app: web

若有 3 個 AZ、replicas: 6，上面設定會確保各 AZ 各 2 個 Pod。若只有 2 個 AZ、replicas: 7， 會分成 4/3 或 3/4（skew=1 內）。

PriorityClass 與搶佔（Preemption）

資源不夠時誰先跑？PriorityClass 給 Pod 排序，高優先級 Pod 可以搶佔（驅逐）低優先級 Pod 來騰出空間。

priorityclass.yaml 
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 1000000                # 數字越大越優先
globalDefault: false
description: "關鍵業務"
preemptionPolicy: PreemptLowerPriority  # 或 Never（不搶佔）
---
spec:
  priorityClassName: high-priority
⚠️ 內建保留名稱
K8s 預設兩個 PriorityClass：system-cluster-critical（20 億）與 system-node-critical（20 億+1）。僅用於系統組件（kube-dns 等），應用不能用。

PodDisruptionBudget：有計畫中斷的底線

有計畫中斷（Voluntary Disruption）包括：Node 排水升級、Cluster Autoscaler 收縮、kubectl drain。 PDB 保證這類操作不會讓可用 Pod 數掉到安全線以下。

pdb.yaml 
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-pdb
spec:
  minAvailable: 2             # 至少 2 個 Pod 隨時可用
  # maxUnavailable: 25%       # 或用最多多少比例不可用
  selector:
    matchLabels:
      app: web
💡 與 Deployment replicas 搭配
replicas: 3、minAvailable: 2 → 同時最多只能動 1 個。replicas: 10、maxUnavailable: 20% → 同時最多動 2 個。PDB 並不影響應用崩潰這類非計畫中斷。
Part 02
Autoscaling · 自動擴縮
依負載自動加減副本、調整資源、甚至加減 Node，做到成本與效能的平衡

HPA：Horizontal Pod Autoscaler

水平擴縮：依指標（CPU / Memory / 自訂）動態調整 Pod 副本數。 前提是 Cluster 有安裝 metrics-server（提供基本 CPU/Memory）或 Prometheus Adapter（自訂指標）。

hpa.yaml · v2 範本 
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 3
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70    # 平均 CPU 使用率超過 70% 擴容
  - type: Resource
    resource:
      name: memory
      target:
        type: AverageValue
        averageValue: "512Mi"
  behavior:                      # v2 可細調擴縮速率
    scaleDown:
      stabilizationWindowSeconds: 300   # 5 分鐘穩定期才縮
      policies:
      - type: Percent
        value: 50
        periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0     # 立刻擴
      policies:
      - type: Percent
        value: 100
        periodSeconds: 30
⚠️ HPA 能運作的前提
Pod 必須設 resources.requests，否則 HPA 沒有計算基準。沒 requests，HPA 根本不會動作。

VPA：Vertical Pod Autoscaler

不加副本，而是調整單個 Pod 的 CPU/Memory requests。適合批次任務、單體應用、無法水平擴縮的工作負載。 VPA 不是 K8s 內建，需另外安裝。

vpa.yaml 
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: worker-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: worker
  updatePolicy:
    updateMode: Auto            # Off / Initial / Auto / Recreate
  resourcePolicy:
    containerPolicies:
    - containerName: worker
      minAllowed:
        cpu: "100m"
        memory: "128Mi"
      maxAllowed:
        cpu: "4"
        memory: "8Gi"
🚫 HPA + VPA 同時用要小心
同時啟用 HPA（基於 CPU）+ VPA（調整 CPU requests）會相互干擾——VPA 把 requests 調大，HPA 算出來的 utilization 變小，可能造成反覆震盪。建議：HPA 只用自訂指標（如 QPS）；或 VPA 設 updateMode: Off 只做建議。

KEDA：事件驅動擴縮

HPA 限於資源指標與 Prometheus 自訂指標。要依外部事件源擴縮（Kafka lag、SQS 訊息數、PostgreSQL 查詢、cron 時間段）就需要 KEDA（Kubernetes Event-Driven Autoscaler，CNCF graduated project）。

keda-kafka.yaml · 依 Kafka consumer lag 擴縮 
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: kafka-consumer
spec:
  scaleTargetRef:
    name: consumer-deployment
  minReplicaCount: 0                 # 可縮到 0！無訊息時完全不花錢
  maxReplicaCount: 50
  triggers:
  - type: kafka
    metadata:
      bootstrapServers: kafka:9092
      consumerGroup: my-group
      topic: orders
      lagThreshold: "10"           # 每有 10 個積壓訊息多起 1 個 Pod
💡 Scale to Zero
KEDA 最大賣點：無事件時可縮到 0。這讓批次工作、非 24/7 應用成本大幅下降——幾乎是 serverless 體驗但仍跑在自己 Cluster。

Cluster Autoscaler 與 Karpenter

HPA/VPA/KEDA 都是在現有 Node 上調整 Pod。當 Cluster 根本沒空間了怎麼辦？需要加 Node。

Cluster Autoscaler (CA)

檢查是否有 Pending Pod 因資源不足卡住。會觸發雲端 Node Group（Auto Scaling Group）擴容。綁定預先定義的 Node 群組。

Karpenter（AWS 首推、已跨雲）

不用預設 Node 群組——看 Pending Pod 的需求直接挑最便宜的 EC2 類型開出來。排程更快、成本更省。

完整擴縮層級圖

擴縮責任分層 
外部事件源 (Kafka/SQS) ──→ KEDA     ──→ 改 HPA target
CPU / 自訂指標        ──→ HPA      ──→ 改 replica count
容器資源需求          ──→ VPA      ──→ 改 requests/limits
Pod 卡 Pending        ──→ CA / Karpenter ──→ 加 Node
Part 03
Probes · 健康檢查
讓 K8s 知道你的應用何時準備好、何時該重啟、何時需要更多時間初始化

三種 Probe

⏱
Startup Probe

容器是否完成啟動？通過前 Liveness/Readiness 不會執行。

失敗 → kill 容器
💓
Liveness Probe

容器還活著嗎？死鎖、當掉這類情況用。

失敗 → kill 容器重啟
🚦
Readiness Probe

可以接流量了嗎？啟動緩慢、暖機、相依服務未就緒用。

失敗 → 從 Service 摘掉

檢查方式三種

方式描述適合
httpGetHTTP GET 指定 path，2xx/3xx 視為成功Web / API 服務（最常用）
tcpSocket能建立 TCP 連線就成功資料庫、訊息佇列等非 HTTP 服務
exec在容器內執行指令，exit 0 成功自訂邏輯（如檢查檔案）
grpc（v1.24+）gRPC Health Checking 協議gRPC 服務

Probe 實戰設定

probes.yaml · 完整三種配合 
containers:
- name: app
  image: myapp:1.0
  ports:
  - containerPort: 8080

  startupProbe:           # 慢啟動應用必備，給它最多 5 分鐘
    httpGet:
      path: /healthz
      port: 8080
    periodSeconds: 10
    failureThreshold: 30    # 10s × 30 = 300s

  livenessProbe:
    httpGet:
      path: /healthz
      port: 8080
    periodSeconds: 10
    timeoutSeconds: 3
    failureThreshold: 3     # 連續 3 次失敗才重啟

  readinessProbe:
    httpGet:
      path: /ready       # 通常與 liveness 不同 endpoint
      port: 8080
    periodSeconds: 5
    timeoutSeconds: 2
    failureThreshold: 2

關鍵參數

參數預設用途
initialDelaySeconds0容器啟動後等多久才開始檢查（有 startupProbe 時可設 0）
periodSeconds10檢查間隔
timeoutSeconds1單次檢查 timeout
successThreshold1連續幾次成功算通過（liveness/startup 只能 1）
failureThreshold3連續幾次失敗才判失敗
⚠️ Liveness vs Readiness 的致命差異
Liveness 失敗會殺掉容器重啟；Readiness 失敗只是暫時從 Service 摘掉。如果只該「暫時不接流量」的情境用了 Liveness（比如等待相依服務），應用會陷入重啟迴圈 + 串聯故障。記住：能自己修好的用 readiness，自己修不好的才用 liveness。
💡 慢啟動的正確姿勢
Java/.NET 應用啟動可能要數十秒到數分鐘。以前用 initialDelaySeconds: 180；現在應該用 startupProbe——失敗重啟時不用再浪費 3 分鐘空轉，且 liveness 可以設比較緊（快速偵測死鎖）。
Part 04
Observability · 觀測性
看不見就沒辦法修。從日誌、指標、追蹤、事件四個角度看穿 Cluster 內部

三大支柱

📊
Metrics

數值型時序資料。
CPU、記憶體、QPS、延遲。
Prometheus 生態為主。

📜
Logs

文字型事件記錄。
應用 log、錯誤堆疊。
ELK / Loki / Splunk。

🔗
Traces

跨服務的請求軌跡。
分散式微服務除錯。
Jaeger / Tempo / OTel。

K8s 還有第四種資訊：Events——Cluster 自身發生的事件（Pod 被排程、映像拉失敗、Liveness 失敗...）。這是除錯的金礦。

Logging：容器日誌

預設行為

應用把日誌寫到 stdout / stderr，容器運行時轉寫到 Node 本機檔案（/var/log/containers/）。 kubectl logs 實際是透過 kubelet 讀這些檔案。

Bash · kubectl logs 常用 
# 看當前日誌
kubectl logs <pod>

# 多容器 Pod 指定容器
kubectl logs <pod> -c <container>

# 持續追蹤（像 tail -f）
kubectl logs -f <pod>

# 看上次重啟前的日誌（CrashLoopBackOff 必用）
kubectl logs <pod> --previous

# 最近 5 分鐘
kubectl logs <pod> --since=5m

# Deployment 下所有 Pod 的日誌
kubectl logs -l app=web --tail=100

# 合併時間戳
kubectl logs <pod> --timestamps

集中式日誌架構

單一 Pod 用 kubectl logs 夠用，大規模生產環境需要中央化：

組件角色常見選擇
Agent（每 Node 一份）收集 Node 上所有容器日誌Fluent Bit、Fluentd、Promtail、Vector
儲存 / 索引長期保存與快速查詢Elasticsearch、Loki、OpenSearch、Splunk
視覺化查詢 UIKibana、Grafana
💡 結構化日誌的威力
應用輸出 JSON 格式日誌（level、timestamp、trace_id、message），agent 可直接解析欄位，查詢時能做 level=error AND service=payment 這類精確過濾。比純文字好用 10 倍。

Metrics：指標與警報

兩種指標管線

Resource Metrics (metrics-server)

K8s 內建基礎，由 metrics-server 提供 kubectl top、HPA 用的 CPU/Memory。保留 60 秒，不做長期。

Custom Metrics (Prometheus)

事實標準。Prometheus Operator + kube-state-metrics + node-exporter 幾乎是 K8s 標配。

Bash · metrics-server 基本用法 
# 各 Node 資源使用
kubectl top nodes

# 各 Pod 資源使用
kubectl top pods -A --sort-by=cpu
kubectl top pods -n production --sort-by=memory

# 特定 Pod 內每個容器
kubectl top pod <pod> --containers

Prometheus 生態核心組件

  • Prometheus Server：抓取、儲存 metrics
  • kube-state-metrics：把 K8s 物件狀態（Deployment 副本數、Pod 狀態）轉成 metrics
  • node-exporter：每 Node 一個 DaemonSet，暴露 Node 層 metrics
  • Alertmanager：處理警報路由、去重、靜音
  • Grafana：視覺化儀表板
  • kube-prometheus-stack（Helm）：一鍵部署以上全部

Events：Cluster 的日記

Events 是 K8s 紀錄「發生了什麼」的原生機制。Pod 排程失敗、映像拉不到、Liveness 失敗都會產生 event。

Bash · 查看 Events 
# 整個 namespace 所有事件（按時間排序）
kubectl get events -n default --sort-by='.lastTimestamp'

# 只看 Warning
kubectl get events --field-selector type=Warning -A

# 針對特定 Pod 的事件
kubectl describe pod <pod>  # 底部有 Events 段

# 持續追蹤（像 tail -f）
kubectl get events -w
⚠️ Events 有保存時間
預設 Events 只保留 1 小時（event-ttl）！要長期追蹤需發到外部系統。建議把 Events 也納入日誌管線（Fluent Bit 有專屬 plugin）。

kubectl debug：神器

生產容器為了安全，通常沒有 shell、沒有 curl、沒有 ps。怎麼除錯？
kubectl debug（v1.25 穩定）可以附加一個臨時除錯容器到現有 Pod，共享網路與 process namespace。

Bash · kubectl debug 四大招 
# 1. 附加 ephemeral container 到現有 Pod（最常用）
kubectl debug -it <pod> --image=nicolaka/netshoot --target=app

# 2. 複製 Pod 並注入新容器（原 Pod 有 readOnlyRootFs 等限制時用）
kubectl debug <pod> -it --image=busybox --copy-to=debug-pod --share-processes

# 3. 換掉 command 重啟一份複製 Pod（用於 CrashLoopBackOff 時探索環境）
kubectl debug <pod> -it --copy-to=shell-copy --image=busybox \
  --container=app -- sh

本章速查

🎯 必記重點（按主題）

排程

  1. Scheduler 兩階段：Filter 過濾不合格 Node，Score 打分選最高。
  2. requests 決定排程、limits 決定上限；memory 超 limit 會 OOMKilled。
  3. 三級 QoS：Guaranteed（requests=limits）> Burstable > BestEffort，OOM 先殺 BestEffort。
  4. Affinity 是 Pod 挑 Node；Taint 是 Node 拒絕 Pod。
  5. topologyKey 常用三個：hostname、zone、region。
  6. 生產環境一定要設 PodDisruptionBudget 保護可用副本數。

擴縮

  1. HPA 改副本數、VPA 改 requests、KEDA 依外部事件（可到 0）、CA/Karpenter 改 Node 數。
  2. HPA 前提：Pod 有設 resources.requests。
  3. HPA + VPA 同時用會震盪，謹慎搭配。
  4. Karpenter 比傳統 CA 更彈性（不綁 Node Group），AWS 環境首選。

Probes

  1. 三 probe：startup（等啟動）、liveness（活著嗎）、readiness（能接流量嗎）。
  2. Liveness 失敗殺容器、Readiness 失敗只摘 Service。
  3. 慢啟動應用用 startupProbe，別再用 initialDelaySeconds: 180。
  4. 不要濫用 Liveness——能自己恢復的用 Readiness。

觀測性

  1. 三大支柱 + Events 是除錯四大資源。
  2. Events 預設只存 1 小時，重要的要轉到中央日誌。
  3. metrics-server 提供 kubectl top 與 HPA；Prometheus 做生產級 metrics。
  4. 結構化日誌（JSON）比純文字好查詢 10 倍。
  5. kubectl debug 是現代 distroless 映像除錯必備。
