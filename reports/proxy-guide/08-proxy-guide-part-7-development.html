Chapter 53

OpenAPI 全解構
從 Swagger 到 OpenAPI 3.2 — API 描述標準的 15 年演進

OpenAPI 是 reverse proxy 工程師每天面對的議題。本章解構 Swagger → OpenAPI 演進、3.0 / 3.1 / 3.2 的具體差異、JSON Schema 2020-12 對齊、API-first 開發流程，以及 reverse proxy 怎麼從 OpenAPI spec 自動生成 routing、rate limit、validation 配置。

Part VI 看了 reverse proxy 的前沿議題。Part VII 回到日常。OpenAPI 是 reverse proxy 工程師每天會碰到的東西 — 不只是「給開發者看的文件」，而是「API 行為的 single source of truth」。從這個 spec 能自動生成 SDK、產生測試、配置 reverse proxy routing、推導 rate limit、driver mock server、驗證 contract。本章是 Part VII 的起點，幫你建立 OpenAPI 的完整理解 — 從 Swagger 起源到 2025/09 OpenAPI 3.2 的最新功能。

53.1 — 從 Swagger 到 OpenAPI：15 年演進

OpenAPI 不是新東西 — 但歷經 15 年才成熟到今天的樣子。完整時間軸：

年份事件
2011Tony Tam（SmartBear）建立 Swagger 規格，最初用於 Wordnik API
2014Swagger 2.0 釋出，引入結構化 paths、host、basePath、安全定義
2015/11SmartBear 把 Swagger 規格捐給 Linux Foundation OpenAPI Initiative，「Swagger」從規格名變成工具集名（Swagger UI / Editor / Codegen）
2017/07OpenAPI 3.0 釋出：reusable components、OAuth 2.0 / OpenID Connect、requestBody（取代舊的 body / formData）
2021/02OpenAPI 3.1 釋出：完整對齊 JSON Schema 2020-12（核心里程碑）
20233.0.4 / 3.1.1 維護版本
2025/09OpenAPI 3.2 釋出：streaming media types（SSE / JSON Lines / JSON Sequences）、結構化 tag nesting、native QUERY HTTP method、OAuth 2.0 Device Authorization Flow
2026+OpenAPI 4.0「Moonwalk」早期開發中（OpenAPI Initiative 同時持續 3.x 增量發布）

三大命名澄清

  • Swagger（規格）：2011-2015 的舊名，已不存在。所有「Swagger 規格」實際上指 OpenAPI 2.0
  • Swagger（工具集）：SmartBear 維護的 Swagger UI / Swagger Editor / Swagger Codegen，仍存在且廣泛使用
  • OpenAPI Specification（OAS）：Linux Foundation 治理的開放規格，2017+ 主流

當有人說「我們用 Swagger」，往往要追問「是 Swagger 2.0 規格還是 OpenAPI 3.x + Swagger UI 工具？」這兩個答案差了 8 年技術。

53.2 — OpenAPI 3.0 vs 3.1：JSON Schema 對齊的革命

OpenAPI 3.1 看起來只是小版本升級，實際上是規格史最重大變化。原因：完整對齊 JSON Schema Draft 2020-12。

3.0 的問題

OpenAPI 3.0 用「JSON Schema 子集」描述 schema — 大部分相容但不完全。具體差異：

  • nullable：3.0 用 nullable: true，JSON Schema 用 type: ["string", "null"]
  • exclusiveMinimum / exclusiveMaximum：3.0 是 boolean，JSON Schema 是 number
  • 不支援 if / then / else 條件 schema
  • 不支援 $dynamicRef
  • 不支援 const

結果：用 JSON Schema 工具驗證 OpenAPI 3.0 schema 會出錯，反之亦然。整個生態要維護兩套工具。

3.1 的對齊

OpenAPI 3.1 直接引用 JSON Schema 2020-12 而非自己定義。這意味著：

  • 任何合法的 JSON Schema 2020-12 文件都是合法的 OpenAPI 3.1 schema
  • 能用任何 JSON Schema validator 驗證 OpenAPI 3.1 payload
  • type: ["string", "null"] 取代 nullable
  • 支援 if / then / else、dynamicRef、const、tuple validation
  • 更豐富的 examples（examples array 取代單一 example）

遷移現實

2026 年初 OpenAPI 3.1 採用率已過tipping point（Total Shift Left 2026/04 報告）。但遷移工作量被低估：

  • 很多 SDK generator、test tool、API gateway 仍只支援 3.0
  • nullable: true → type: ["string", "null"] 是每個 nullable 欄位都要改
  • Validator 行為微妙差異（如未指定 required 的處理）

對 reverse proxy 工程師：2026 年新建系統用 3.1+，舊系統視 tooling 支援度漸進升級。

53.3 — OpenAPI 3.2（2025/09）：streaming 與 query method

2025/09 釋出的 OpenAPI 3.2 加入 reverse proxy 工程師切身相關的功能：

Streaming Media Types

3.2 第一次原生支援 streaming response：

  • text/event-stream（SSE，Server-Sent Events）— LLM 回應 / 即時通知
  • application/jsonl（JSON Lines）— LLM streaming completion
  • application/json-seq（JSON Sequences，RFC 7464）

意義：OpenAPI 3.2 是第一個能描述 LLM API 的 OpenAPI 版本。OpenAI、Anthropic、Google 的 streaming API 在 3.0/3.1 只能用 text/event-stream + 描述「各自為政」的 hack 方案。

OpenAPI 3.2 streaming response 範例paths:
  /v1/chat/completions:
    post:
      requestBody:
        content:
          application/json:
            schema:
              type: object
              properties:
                stream: { type: boolean }
                messages: { type: array }
      responses:
        '200':
          description: Chat completion
          content:
            # 標準 JSON 回應
            application/json:
              schema:
                $ref: '#/components/schemas/ChatResponse'
            # 串流回應（stream=true 時）
            text/event-stream:
              schema:
                $ref: '#/components/schemas/ChatStreamChunk'
              # 3.2 的新 keyword
              x-streaming: true
              x-event-name: data

Native QUERY HTTP Method

2024 IETF 標準化 QUERY method（draft-ietf-httpbis-safe-method-w-body）— 介於 GET 與 POST 之間的「有 body 的 read-only」操作。場景：複雜搜尋查詢（GET 的 URL 太長）+ 不該改變狀態（POST 的語意太強）。OpenAPI 3.2 原生支援：

paths:
  /search:
    query:                    # 3.2 新增！
      requestBody:
        content:
          application/json:
            schema: { ... }
      responses:
        '200':
          ...

結構化 Tag Nesting

3.2 前 tags 是 flat array（沒層級）。3.2 加入 parent 屬性允許階層化：

tags:
  - name: api
    description: All API endpoints
  - name: chat
    parent: api
    description: Chat completion endpoints
  - name: streaming
    parent: chat
    description: SSE streaming endpoints

對大型 API（如 AWS、Stripe 有數百個 endpoint）的文件導航顯著改善。

OAuth 2.0 Device Authorization Flow

3.2 加入 deviceAuthorization flow type — 給無瀏覽器設備（CLI、IoT、smart TV）的 OAuth flow（RFC 8628）。Anthropic Claude Code、GitHub CLI 等都用此 flow。

53.4 — OpenAPI 對 reverse proxy 工程的意義

reverse proxy 工程師對 OpenAPI 應該有四個用法：

用法 1：自動生成 routing 配置

許多 API gateway 能直接讀 OpenAPI spec 生成 routing：

  • Apache APISIX：直接 import OpenAPI 自動建立 routes + auth + rate limit
  • Kong：Insomnia / Inso CLI 從 OpenAPI 生成 declarative config
  • AWS API Gateway：原生支援 OpenAPI import
  • Azure API Management：OpenAPI 是首選描述方式
  • Higress（Alibaba 開源 CNCF Sandbox）：以 OpenAPI 為配置入口

用法 2：自動生成 SDK

下章 Ch.55 會深入 — Speakeasy、Fern、Stainless 等工具從 OpenAPI 生成多語言 client SDK。OpenAPI 是 SDK 的 source。

用法 3：Contract Testing

Schemathesis、Dredd、Postman 等能從 OpenAPI 自動生成測試：

  • 對每個 endpoint 跑 fuzz testing
  • 驗證 response 符合 schema
  • 偵測 schema drift（implementation 偏離 spec）

用法 4：Mock Server

Prism（Stoplight）、msw（前端）、@mockoon/cli 等能從 OpenAPI 啟動 mock server，讓 frontend 與 backend 平行開發。

Listing 53.1 · OpenAPI 3.1 完整骨架openapi: "3.1.0"

# 元資料
info:
  title: User Management API
  version: "2.0.0"
  description: |
    User management API for ACME Corp.

  contact:

  - url: https://api-staging.acme.com/v2
    description: Staging

security:
  - bearerAuth: []

# 路徑與操作
paths:
  /users:
    get:
      summary: List users
      tags: [users]
      parameters:
        - name: page_size
          in: query
          schema:
            type: integer
            minimum: 1
            maximum: 100
            default: 20
        - name: cursor
          in: query
          schema:
            type: string
      responses:
        '200':
          description: User list
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/UserList'
        '401':
          $ref: '#/components/responses/Unauthorized'

    post:
      summary: Create user
      tags: [users]
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/UserCreate'
      responses:
        '201':
          description: User created
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
        '400':
          $ref: '#/components/responses/BadRequest'

# Reusable components
components:
  securitySchemes:
    bearerAuth:
      type: http
      scheme: bearer
      bearerFormat: JWT

  schemas:
    User:
      type: object
      required: [id, email, name]
      properties:
        id: { type: string, format: uuid }
        email: { type: string, format: email }
        name: { type: string, minLength: 1, maxLength: 100 }
        role:
          type: [string, "null"]    # 3.1 nullable 寫法
          enum: [admin, user, null]

    UserCreate:
      type: object
      required: [email, name]
      properties:
        email: { type: string, format: email }
        name: { type: string }
        password: { type: string, minLength: 12 }

    UserList:
      type: object
      properties:
        users:
          type: array
          items:
            $ref: '#/components/schemas/User'
        next_cursor:
          type: [string, "null"]

  responses:
    Unauthorized:
      description: Missing or invalid token
    BadRequest:
      description: Invalid request body

關鍵設計原則：

  1. $ref 重用：避免 schema 重複，方便維護
  2. 每個 endpoint tag：方便文件分組（3.2 後可 nested）
  3. 4xx 也要描述：不只 200 OK，常見錯誤要列出
  4. 明確 required：不要靠隱式 default
  5. format 提供額外語意：uuid、email、date-time
  6. cursor pagination 而非 page number：scale-friendly

53.6 — API Versioning：reverse proxy 角度

API versioning 是每個 reverse proxy 工程師的痛點。常見策略：

策略例子優點缺點
URL path/v1/users、/v2/users明顯、易 cache、易 routing多版本 URL 蔓延
Custom headerX-API-Version: 2URL 乾淨不易 cache、不易測試
Date-based headerX-GitHub-Api-Version: 2022-11-28精確、長期支援需 version registry
Accept headerapplication/vnd.api+json;v=2HTTP 標準（content negotiation）少數工具支援不佳
Query parameter/users?version=2易測試不該影響 cache key 但容易誤用

2026 年的最佳實踐

  • 外部公開 API：URL path（最直觀，cache 友好，CDN 處理良好）
  • 內部 API：date-based header（GitHub 模式，能精確 pin 到日期）
  • 避免 Accept header 版本化：CDN cache 處理麻煩、debug 困難
  • 提供 deprecation 期：至少 6-12 個月，並用 Deprecation 與 Sunset header（RFC 8594）通知

Chapter 54

AsyncAPI 事件驅動
Kafka、MQTT、WebSocket、SSE — 同步 API 之外的另一個世界

OpenAPI 描述「請求-回應」同步 API；AsyncAPI 描述「訊息-事件」非同步系統。本章解構 AsyncAPI 3.0（2024 釋出）、與 OpenAPI 的整合、Kafka / MQTT / WebSocket / SSE 等 channel binding，以及 reverse proxy 在事件驅動架構的角色。

Ch.53 完整覆蓋了 OpenAPI — 但 OpenAPI 只能描述同步、請求-回應的 API。現代系統有大量非同步互動：Kafka 訊息、MQTT IoT 通訊、WebSocket 即時推送、SSE 串流、AMQP 訊息佇列。AsyncAPI 是這些場景的標準描述語言 — 由同一個概念家族延伸，但事件中心而非request 中心。本章是 reverse proxy 工程師常被忽略但 2026 年越來越重要的議題。

54.1 — 為何需要 AsyncAPI

OpenAPI 的結構性限制：它的 mental model 是 client → server → response。但事件驅動架構不是這樣運作：

  • Kafka：producer 寫入 topic、多個 consumer 訂閱
  • MQTT：device publish 到 broker、其他 device subscribe
  • WebSocket：雙向、長連線、訊息可雙方發起
  • SSE：server 持續推送、client 被動接收
  • AMQP / RabbitMQ：queue + exchange + routing key
  • Webhook：server 對 client 主動 POST 通知

用 OpenAPI 強行描述這些不可能得到合理結果。AsyncAPI 給這些場景原生抽象。

54.2 — AsyncAPI 演進

AsyncAPI 由 Fran Méndez 於 2017 年創立，設計理念明確借鑑 OpenAPI：

年份事件
2017AsyncAPI 1.0 釋出（早期實驗）
20192.0 釋出，社群快速成長
2021加入 Linux Foundation 旗下，與 OpenAPI Initiative 平行治理
2023/12AsyncAPI 3.0 釋出：major rewrite，重構 channel/operation 抽象、改善 OpenAPI 整合、複雜 schema 支援
2024-2026Kafka / MQTT / WebSocket binding 標準化、tooling 成熟

2.x vs 3.0 的核心差異

AsyncAPI 3.0 是不向後相容的 major rewrite。關鍵變化：

  • Operation 從 channel 分離：2.x 的 channel 含 publish/subscribe，3.0 把 operation 提升為 first-class
  • 方向更明確：send / receive 取代 publish / subscribe（避免 server 視角混淆）
  • Channel 加入 address：明確的 topic / queue 名稱
  • Reply pattern：原生支援 request-reply over async
  • 更好的 OpenAPI 互通：能 import OpenAPI schema

2026 年新建系統應該直接用 3.0。舊系統視 tooling 支援度漸進升級。

54.3 — AsyncAPI 文件結構

AsyncAPI 與 OpenAPI 結構接近，但概念對應不同：

OpenAPI 概念AsyncAPI 對應

paths（URL 路徑）channels（topic / queue / exchange）
operations（GET / POST）operations（send / receive）

關鍵點

  1. channels 對應 Kafka topic：address 是真實 topic 名
  2. operations 描述方向：send 是 producer、receive 是 consumer
  3. message contentType 用 CloudEvents：業界事件規格標準（cloudevents.io）

54.5 — CloudEvents 與 AsyncAPI 整合

CloudEvents（CNCF graduated 2022）是事件 metadata 的標準規格 — 跨 cloud / 跨工具的 event 統一格式。AsyncAPI 與 CloudEvents 高度互補。

CloudEvents 核心 attributes

  • specversion：CloudEvents 規格版本（1.0）
  • type：事件類型（如 com.acme.order.created.v1）
  • source：事件來源 URI
  • id：事件 unique ID
  • time：發生時間（ISO 8601）
  • datacontenttype：payload 格式
  • subject（選用）：事件主體

為何 CloudEvents + AsyncAPI

  • AsyncAPI 描述schema 與 channel
  • CloudEvents 標準化事件 metadata
  • 兩者組合：reverse proxy 能 route / filter / observe 任何 CloudEvents

實務：所有事件用 CloudEvents 包裝、AsyncAPI 描述每個 channel 的 payload schema。

54.6 — Reverse Proxy 在事件驅動架構的角色

傳統 reverse proxy（HTTP-only）在事件驅動世界角色受限。但 2026 年的 reverse proxy 越來越多在事件層工作：

場景 1：Webhook Gateway

Webhook 是HTTP-based 事件 — server 對 client 主動 POST。Reverse proxy 角色：

  • Validation：驗證 webhook signature（如 Stripe、GitHub 用 HMAC）
  • Retry：對失敗的 webhook delivery 自動退避重試
  • Rate limit：避免 webhook flood 打死 client
  • Transform：webhook payload 轉換為內部 event 格式
  • Fan-out：一個 webhook 觸發多個 internal subscriber

Cloudflare Worker、Kong webhook plugin、Hookdeck 等專注此場景。

場景 2：MQTT Broker Frontend

MQTT broker（如 EMQX、HiveMQ、Mosquitto）通常需要 reverse proxy 在前面：

  • TLS termination：MQTT over TLS（port 8883）
  • WebSocket bridging：MQTT over WebSocket（port 443）給 web client
  • Geographic routing：客戶 routing 到最近 broker
  • Connection limits：避免 IoT device 爆連

HAProxy、Envoy 都有 MQTT support。Cilium 透過 eBPF 也能處理 MQTT。

場景 3：Kafka Frontend Proxy

Kafka client 通常直連 broker，但企業環境常需要 proxy：

  • 跨網路 / 多 cluster routing
  • 認證 / 授權（OAuth → Kafka SASL）
  • Schema validation（從 schema registry 拉取 + 驗證）
  • Multi-tenancy（virtual cluster 抽象）

專門工具：Kroxylicious（Red Hat 開源）、Strimzi Kafka Bridge（HTTP ↔ Kafka）、WarpStream（zero-disk Kafka）。

場景 4：SSE / WebSocket Gateway

2024-2026 LLM streaming 大量採用 SSE。Reverse proxy 角色：

  • SSE-aware buffering：避免 buffer 太大造成串流卡頓
  • 長連線管理：keepalive、connection limit
  • Authentication 持續驗證：JWT 過期時優雅斷線
  • Reconnect 友好：保留 last-event-id 給 client 重連

Nginx、HAProxy、Envoy 都有專門 SSE / WebSocket buffer 設定。

54.8 — 給 reverse proxy 工程師的建議

Reverse proxy 工程師為何要關心 SDK？因為API 品質 = SDK 品質。再好的 API 設計，如果使用者要花一小時讀文件、寫一個 retry-with-exponential-backoff、處理 pagination cursor 才能用，就是失敗的 DX。SDK 把這些複雜性隱藏，讓 API 變得「一行 import 就用」。本章是 reverse proxy 工程師跨議題的章節 — SDK 不是「後端工程師的事」，是整個 API 平台的事。

55.1 — SDK 為何重要：5 分鐘 vs 5 小時

Fern 2026/03 報告的關鍵洞察：「5 分鐘到 first API call vs 5 小時的差別，往往是幾個月前的設計決定」。糟糕的 API 體驗會：

  • 開發者放棄你的 API（轉投競爭者）
  • 同一問題反覆出現在 support tickets
  • 整合脆弱、易壞
  • 新版本推不動（client 不願升級）

優質 SDK 處理的事情

  • Auth flow：Bearer token / OAuth 2.0 / API key 自動處理
  • Pagination：cursor / offset 自動 iterate（for x in client.list()）
  • Retry：transient error（5xx、429、timeout）自動 exponential backoff
  • Rate limit awareness：解析 X-RateLimit-* header，避免不必要重試
  • Streaming：SSE / chunked transfer 用 idiomatic iterator
  • Error handling：HTTP error 轉成 typed exception
  • Webhook signature verification：HMAC 驗證寫成 helper
  • Discriminated unions：用語言原生 pattern（Rust enum、TS discriminated union）
  • Type safety：完整 type stub / generic

沒 SDK 的代價

每個 API 客戶都要重寫上述邏輯。10,000 個 client × 8 hours 寫 retry logic = 80,000 工時。SDK 把這個成本分攤到 API provider 一次處理。

55.3 — Idiomatic SDK 設計：每語言不同

「同樣的 API，不同的語言該長不同樣子」。糟糕的 SDK 在每個語言看起來都像「Java 翻譯版」；好的 SDK 利用每個語言的原生 idiom。

Python：async / context manager

Pythonic SDKfrom anthropic import Anthropic, AsyncAnthropic

# 同步用 with statement
client = Anthropic(api_key="sk-...")
message = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Hello"}],
)

# 串流用 with + iterator
with client.messages.stream(...) as stream:
    for text in stream.text_stream:
        print(text, end="")

# 非同步版本
async client = AsyncAnthropic()
async with client.messages.stream(...) as stream:
    async for text in stream.text_stream:
        print(text, end="")

# Pagination 用 generator
for message in client.messages.list():
    print(message.id)

TypeScript：discriminated union + Promise

TypeScript-idiomatic SDKimport Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

// Discriminated union：content block 可能是 text 或 tool_use
const message = await client.messages.create({
  model: "claude-sonnet-4-6",
  max_tokens: 1024,
  messages: [{ role: "user", content: "Hello" }],
});

for (const block of message.content) {
  if (block.type === "text") {
    // TS 自動知道這是 TextBlock，有 .text 屬性
    console.log(block.text);
  } else if (block.type === "tool_use") {
    // TS 自動知道這是 ToolUseBlock，有 .input 屬性
    console.log(block.input);
  }
}

// Streaming 用 async iterator
const stream = await client.messages.stream({...});
for await (const chunk of stream) {
  process.stdout.write(chunk.delta?.text ?? "");
}

Go：context、error 顯式

Go-idiomatic SDKimport (
    "context"
    "github.com/anthropics/anthropic-sdk-go"
)

client := anthropic.NewClient()

// 第一參數永遠是 context
message, err := client.Messages.New(ctx, anthropic.MessageNewParams{
    Model:     anthropic.F(anthropic.ModelClaudeSonnet),
    MaxTokens: anthropic.F(1024),
    Messages:  anthropic.F([]anthropic.MessageParam{...}),
})
if err != nil {
    return err
}

// Streaming 用 channel
stream := client.Messages.NewStreaming(ctx, params)
defer stream.Close()
for stream.Next() {
    event := stream.Current()
    switch evt := event.AsAny().(type) {
    case anthropic.ContentBlockDeltaEvent:
        fmt.Print(evt.Delta.Text)
    }
}
if err := stream.Err(); err != nil {
    return err
}

三段程式碼解決同樣問題（streaming chat completion），但用各語言原生 idiom — Python with-statement、TypeScript async iterator、Go channel + switch type assertion。這是 idiomatic SDK 的精髓。

55.4 — Pagination：SDK 的隱形複雜性

Pagination 看似簡單，實際上是 SDK 設計最易犯錯的地方。三種主流模式：

Offset-based（不建議大資料集）

GET /users?page=2&page_size=20

缺點：頁中插入 / 刪除導致重複 / 漏資料；DB 性能差（OFFSET 會 scan 前面所有資料）。

Cursor-based（推薦）

GET /users?cursor=eyJpZCI6IjEyMyJ9&page_size=20

回應：
{
  "users": [...],
  "next_cursor": "eyJpZCI6IjE0MyJ9"
}

Cursor 通常是 base64-encoded（包含 last id + sort key）。穩定對抗插入 / 刪除。GitHub、Stripe、Notion 都用 cursor。

Link-based（HATEOAS）

GET /users?cursor=...

Link: <https://api.example.com/users?cursor=NEXT>; rel="next",
      <https://api.example.com/users?cursor=PREV>; rel="prev"

RFC 5988 標準，最 RESTful 但工具支援較弱。

SDK 怎麼處理

SDK 自動 pagination 範例（Python）# 直接 iterate，SDK 自動 fetch 下一頁
for user in client.users.list(page_size=20):
    print(user.email)

# 或拿到 page object，控制更細
page = client.users.list(page_size=20)
while page.has_more:
    for user in page.data:
        print(user.email)
    page = page.get_next()

關鍵：SDK 把 pagination 從「每次都要寫的樣板」變成「就用就對了」。Stainless、Speakeasy、Fern 全部把 auto-pagination 當 first-class。

55.5 — llms.txt：給 AI agent 看的 API 文件

  • token 消耗極高（HTML markup、CSS、JS bundle）

55.6 — SDK 與 reverse proxy 工程師的關係

讀到這你可能想：「這跟 reverse proxy 有什麼關係？我不寫 SDK」。其實非常相關：

  1. SDK 暴露 API 設計缺陷：當 Speakeasy / Fern 試圖生成 SDK 時失敗，往往因為 OpenAPI spec 不完整或 API 設計差。SDK 是 API 設計的反饋訊號
  2. Reverse proxy 影響 SDK 行為：rate limit header、retry 行為、超時設定都直接影響 SDK 怎麼處理 transient error
  3. 觀察 SDK 流量模式：SDK 的 retry 策略、polling 間隔、batch 行為都是 reverse proxy 上看到的流量模式
  4. API 版本化策略影響 SDK：Ch.53 §53.6 提到的 versioning 策略直接決定 SDK 怎麼處理多版本
  5. SDK breaking change 是重大事件：影響數千 client，與 API breaking change 同等嚴重

55.7 — 實戰：選 SDK 工具的決策框架

情境 1：對外 SaaS、開發者體驗第一

建議：Stainless 或 Speakeasy。投資 SDK 品質直接影響 conversion。Stainless 適合 LLM / 複雜 API，Speakeasy 適合多語言廣度。

情境 2：B2B SaaS、9 語言廣度

建議：Fern。9+ 語言 + AsyncAPI + gRPC + GraphQL 統一平台。

情境 3：內部 API、跨團隊整合

建議：OpenAPI Generator + 自訂模板。免費、靈活，但要維護模板。

情境 4：Microsoft 生態 / Graph API 風格

建議：Kiota（Microsoft 開源）。設計借鑑 Graph SDK，與 .NET / Microsoft 工具整合最深。

情境 5：Go-only ecosystem

建議：oapi-codegen（純 Go SDK），生成的 Go 程式碼非常 idiomatic。

55.8 — Reverse proxy 工程師的 SDK 6 條建議

  1. 有對外 API 一定要有官方 SDK：別讓客戶自己寫 retry / pagination
  2. SDK 從 OpenAPI / AsyncAPI 自動生成：避免 spec 與 SDK drift
  3. 每個語言原生 idiom：別讓 Python SDK 看起來像 Java
  4. 提供 llms.txt：2026 年 AI agent 整合是重要的使用情境

Time-to-first-call、quickstart、error message 與 deprecation 管理

DX 是 reverse proxy 工程師不能外包給 PM 的議題 — 從 onboarding flow、API key 取得、第一個成功 request、到 production 整合，每一步都涉及 reverse proxy 設計決策。本章解構 time-to-first-call 的優化、quickstart 設計、有用的 error message、changelog 與 deprecation 管理。

Reverse proxy 工程師常被誤解為「只管 routing 與 TLS」。實際上，每一個 reverse proxy 決策都直接影響開發者體驗 — rate limit header 怎麼回、error response 是 JSON 還是 HTML、deprecation 怎麼通知、authentication 失敗給什麼線索。本章給 reverse proxy 工程師產品視角的 DX checklist — 不是 PM 的工作，是整個 API 平台的工作。

56.1 — Time-to-First-Call：核心 KPI

DX 領域有一個被 Stripe、Twilio、Postman 等業界共識的核心 metric：Time-to-First-Call（TTFC）。從「看到 API 文件」到「第一個成功 200 OK」需要多久。

優秀 vs 糟糕的 TTFC

場景優秀糟糕
取得 API keysignup 後 1 分鐘內「請聯繫銷售」+ 3 天回覆
第一個 curl 請求doc 上有複製即用 snippet「請參考 SDK 文件」+ 自己拼湊
看懂 error「missing field 'email'」「Bad Request」(沒細節)

5 分鐘 TTFC 的設計

Stripe 是業界標桿。一個未付費的開發者能在 5 分鐘內：

  1. 0:00 — Signup（email + password）
  2. 0:30 — 進 dashboard，看到 test mode API key
  3. 1:00 — 複製 dashboard 上的 curl snippet（已含真實 API key）
  4. 1:30 — paste 到 terminal，第一個 200 OK
  5. 2:00-5:00 — 改 SDK 範例做點客製化

TTFC 5 分鐘不是運氣，是無數 UX 設計決策的結果。對 reverse proxy 工程師：許多 TTFC 摩擦點在 reverse proxy 層 — auth header 格式、rate limit 觸發訊息、400 vs 401 vs 403 的區分、CORS 配置等。

56.2 — Quickstart 設計：5 條原則

原則 1：Copy-Paste-Run（不要要求修改）

糟糕：

curl https://api.example.com/v2/widgets \
  -H "Authorization: Bearer YOUR_API_KEY"  # 要使用者改

優秀（dashboard 上即時顯示）：

curl https://api.example.com/v2/widgets \
  -H "Authorization: Bearer sk_test_51HxYz...UD2"  # 真實 test key

原則 2：Multi-language Tabs

同一個 example，6 個 tab：curl、Python、TypeScript、Go、Ruby、Java。每個都真的能跑。Stripe、Twilio、Anthropic 都這樣做。

原則 3：失敗也要清楚

如果使用者複製範例但失敗，error message 必須明確指向問題。常見糟糕例子：

  • Internal Server Error — 完全沒線索
  • Bad Request — 哪裡 bad？
  • Authentication failed — token 過期？格式錯？權限不夠？

反例（好的 error message，§56.4 詳細）：

{
  "error": {
    "type": "invalid_request_error",
    "message": "The 'email' field is required.",
    "param": "email",
    "doc_url": "https://docs.example.com/errors/missing-field"
  }
}

原則 4：Test Mode / Sandbox

與 production 完全分離的 sandbox。Stripe test mode、Anthropic API 的免費 trial、Twilio test credentials — 都讓開發者在沒承擔商業風險的情況下完整探索。

原則 5：明確的「What's next」

Quickstart 結尾不該是「歡迎加入！」，而是「下一步：(1) 加入 webhook、(2) 切換到 production key、(3) 設定 rate limit alert」。讓開發者有清楚的進階路徑。

56.3 — Authentication 體驗：reverse proxy 的責任

Authentication 是 TTFC 最大殺手。reverse proxy 工程師直接負責這層體驗。

Bearer token：基本卻常被搞砸

常見配置錯誤# 1. 不接受 Authorization header 大小寫變化
authorization: Bearer xxx   # 失敗 ← 規格上 HTTP header 應 case-insensitive
Authorization: bearer xxx   # 失敗 ← scheme 應 case-insensitive

# 2. CORS preflight 沒有放行 Authorization header
Access-Control-Allow-Headers: Content-Type   # 漏 Authorization

# 3. 401 沒有 WWW-Authenticate header
HTTP/1.1 401 Unauthorized
# 缺 WWW-Authenticate: Bearer realm="api"

401 vs 403：區分要清楚

  • 401 Unauthorized：「我不知道你是誰」（沒 token、token 過期、token 無效）
  • 403 Forbidden：「我知道你是誰，但你不能存取這個資源」（權限不足、quota 超出、地理限制）

Reverse proxy 應該區分這兩種，給 client 清楚訊號。誤用 403（應該 401）會讓 client SDK 不會重試 refresh token。

Token 過期的優雅處理

對過期 token（vs 完全無效），考慮回應：

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="api", error="invalid_token", error_description="The access token expired"
Content-Type: application/json

{
  "error": {
    "code": "token_expired",
    "message": "The provided token has expired. Please refresh.",
    "doc_url": "https://docs.example.com/auth/refresh"
  }
}

Client SDK 看到 error="invalid_token" 自動觸發 refresh flow — 這是 OAuth 2.0 標準（RFC 6750）。

56.4 — Error Message 設計

Error message 是 reverse proxy 工程師最常被忽視的 DX 點。優質 error 應有 5 個元素：

5 元素 error response

Listing 56.1 · 完整 error response 範例HTTP/1.1 400 Bad Request
Content-Type: application/problem+json   # RFC 7807 standard
X-Request-Id: req_abc123xyz                # 給 support 用

{
  "type": "https://docs.example.com/errors/missing-field",
  "title": "Missing Required Field",
  "status": 400,
  "detail": "The field 'email' is required for user creation.",
  "instance": "/api/v2/users",

  // 額外 metadata
  "code": "missing_field",
  "param": "email",
  "request_id": "req_abc123xyz",
  "errors": [                        // 多個 field 錯誤
    {
      "path": "$.email",
      "code": "required",
      "message": "Field is required"
    },
    {
      "path": "$.password",
      "code": "too_short",
      "message": "Must be at least 12 characters"
    }
  ]
}

5 元素

  1. HTTP status：精準（400 vs 401 vs 403 vs 404 vs 409 vs 422 vs 429）
  2. Machine-readable code：missing_field、token_expired、quota_exceeded
  3. Human-readable message：清楚說明問題
  4. Doc URL：指向詳細解釋與解法
  5. Request ID：給 support team 對照 log

RFC 7807 / 9457 Problem Details

RFC 7807（2016，2023 更新為 RFC 9457）標準化 HTTP error response 格式。content-type 是 application/problem+json。Stripe、GitHub、Spotify 等都採用。

使用 RFC 7807 的優勢：

  • 標準格式，client SDK 能通用處理
  • 每個 error type 是個 URL（type 欄位），方便導向文件
  • 支援 extension fields（code、errors[]、request_id）

Content negotiation 陷阱

當 reverse proxy 的後端 down 或自己出錯，不能回 HTML（即使預設 server error page 是 HTML）。要根據 Accept header 決定：

Nginx 配置：API endpoint 回 JSON 錯誤location /api/ {
    error_page 500 502 503 504 /api-error.json;
    proxy_pass http://backend;
}

location = /api-error.json {
    internal;
    default_type application/json;
    return 500 '{"type":"https://docs.example.com/errors/server","title":"Internal Server Error","status":500,"request_id":"$request_id"}';
}

避免 client SDK 因為「response is HTML」而 crash 或 silent fail。

56.5 — Rate Limit 訊息：給 client 的合作訊號

Ch.35 §35.9 提過 rate limit header。這裡從 DX 視角再次強調 — 因為這是「糟糕 DX 的常見來源」。

必要 header（事實標準）

HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 100           # 配額總量
X-RateLimit-Remaining: 0         # 剩餘
X-RateLimit-Reset: 1714752000   # Unix timestamp，何時重置
Retry-After: 60                 # 多少秒後可重試
Content-Type: application/problem+json

{
  "type": "https://docs.example.com/errors/rate-limit",
  "title": "Rate Limit Exceeded",
  "status": 429,
  "detail": "You have exceeded the rate limit of 100 requests per minute.",
  "reset_at": "2026-04-26T08:00:00Z"
}

RFC 9331（IETF 2025 提案）

2025 年 IETF 開始標準化 rate limit header（RateLimit + RateLimit-Policy）。雖然還未成為 RFC，但已被 GitHub、Stripe 採用。2026-2027 年將取代非標準的 X-RateLimit-*。

Burst vs Sustained 雙層限流

給 client 兩個訊號：

  • Burst limit：短時間（如 1 秒）允許多少（如 10）
  • Sustained limit：長時間（如 1 小時）允許多少（如 5,000）

Client SDK 看到兩者能聰明 throttle — 不要 burst 後立刻 sustained 也用光。

56.6 — Changelog 與 Deprecation：管理 API 演進

API 演進是持續的。如何讓 client 跟上而不破壞既有整合？

Changelog 設計

每個 API 應該有公開的 changelog。要素：

  • 每次變更日期 + version
  • 分類：Added / Changed / Deprecated / Removed / Fixed / Security
  • Breaking change 明確標記
  • Migration guide：對 deprecation / breaking 提供升級指引

Deprecation 機制（RFC 8594 + RFC 9745）

RFC 8594（2019）定義 Sunset header：

Sunset: Sat, 31 Dec 2026 23:59:59 GMT
Link: <https://docs.example.com/api/v3-migration>; rel="sunset"

RFC 9745（2025/03）更新 Deprecation header（取代之前的 draft）：

Deprecation: @1735603199          # Unix timestamp（@-prefixed）
Link: <https://docs.example.com/api/v3>; rel="successor-version"
Sunset: Sat, 31 Dec 2026 23:59:59 GMT

Deprecation 時程

業界共識：

  • ≥ 6 個月給主動整合的 client（如付費 SaaS 客戶）
  • ≥ 12 個月給長尾 client（個人開發者、非活躍 client）
  • 大型 breaking change ≥ 24 個月（如 OAuth flow 變更）

Stripe 的 deprecation 通常 18-24 個月，且所有變更會同時用 email + dashboard alert + Sunset header 通知 — 三重保險。

Reverse proxy 的角色

  • 對 deprecated endpoint 自動加 Deprecation + Sunset header
  • 記錄 deprecated endpoint 的 caller（識別還在用的 client）
  • EOL 日期 hard-code 在 reverse proxy 配置（到期自動回 410 Gone）

Chapter 57

OpenTelemetry 完整解構
CNCF 第 2 活躍專案、4,000+ contributors、三大 signal 統一

OpenTelemetry 在 2024-2026 年成為事實標準。CNCF 排名第 2 活躍專案（僅次 Kubernetes）、4,000+ contributors、所有主流可觀測性廠商支援。本章解構 OTel 三大 signal、Collector pipeline、OTLP protocol、auto-instrumentation、Beyla → OBI eBPF instrumentation、tail-based sampling，與 reverse proxy 工程師的具體實踐。

可觀測性（Observability）是 reverse proxy 工程師每天必修的議題 — 沒它就是「瞎子開飛機」。但 2020 年前的可觀測性是廠商特定 SDK 的天下：Datadog 一套、New Relic 一套、Splunk 又一套。每換一家廠商要重寫所有 instrumentation。OpenTelemetry（OTel）解決這個問題 — 一套標準、寫一次、跑任何後端。本章是 reverse proxy 工程師必修的觀察性基礎。

57.1 — OpenTelemetry 為何重要：CNCF 第 2 活躍專案

OpenTelemetry 由OpenTracing（2016）+ OpenCensus（2017）合併而成（2019 年宣布、2021 年完成）。2026 年現況：

  • CNCF 第 2 活躍專案，僅次 Kubernetes
  • 2025 年 4,000+ contributors
  • 所有主流可觀測性廠商原生支援：Datadog、New Relic、Grafana、Honeycomb、Dynatrace、Splunk、Elastic
  • Grafana Labs 2026 觀察性調查：大多數組織用 OTel 或正在遷移
  • 2025 年 Grafana 把 Beyla（eBPF zero-code instrumentation）捐給 OTel，改名 OBI（OpenTelemetry eBPF Instrumentation）

OTel 解決的問題

傳統可觀測性 4 大痛點：

  1. Vendor lock-in：換 Datadog → New Relic 要重寫所有 SDK 整合
  2. 三大 signal 各自為政：metrics 用 Prometheus 格式、log 用 syslog、trace 用 Jaeger，無法 correlate
  3. Semantic 不一致：「http_status」vs「status_code」vs「response_status」— 同樣概念名字不同，dashboard 寫死
  4. Manual instrumentation 工作量：每個 service 都要寫一遍

OTel 用三招解：(1) OTLP protocol 統一傳輸、(2) 三大 signal 整合 SDK、(3) Semantic Conventions 統一命名。

57.2 — 三大 Signal：Traces / Metrics / Logs

OTel 統一三類遙測資料的 model：

Traces（追蹤）

對 reverse proxy 工程師：distributed tracing 就是知道 request 卡在哪個 service 的關鍵。

Metrics（指標）

聚合的數值資料 — 計數、分佈、gauge。OTel 的 metric type：

  • Counter：單調遞增（如 request count）
  • UpDownCounter：可增可減（如 active connections）
  • Gauge：當前值（如 queue depth）
  • Histogram：分佈（如 request latency）
  • Exponential Histogram（2024+）：自動 bucket，省記憶體

Logs（日誌）

結構化文字記錄。OTel logs 的關鍵是與 trace 自動關聯 — 每筆 log 自動帶上 trace_id 與 span_id，能直接從 trace 跳到對應 log（反之亦然）。

三者的關係

Metric: 「整體 99p latency 漲了 10×」
   ↓
Trace: 找到一個慢 trace 範例，看哪個 service 慢
   ↓
Log: 看那個 service 在那個 trace_id 下印了什麼

→ 三者必須共享 trace_id才能 correlate

傳統各自獨立的工具難以做到。OTel 的統一讓 correlate 變成原生功能。

57.3 — OTLP Protocol

OTLP（OpenTelemetry Protocol）是 OTel 的傳輸協定。三個版本：

  • OTLP/gRPC：高性能、長連線（生產推薦）
  • OTLP/HTTP（Protobuf）：相容性好、容易過 firewall
  • OTLP/HTTP（JSON）：debug 友善、性能稍差

標準 port

  • 4317：OTLP/gRPC
  • 4318：OTLP/HTTP

所有支援 OTel 的後端（Datadog、New Relic、Grafana Tempo、Honeycomb 等）都接受 OTLP — 寫一次，跑哪都行。

57.4 — OTel Collector：可觀測性的 reverse proxy

OTel Collector是 OTel 生態的核心元件 — 概念上就是「遙測資料的 reverse proxy」。功能：

  • Receive：接收多種格式（OTLP、Prometheus、Jaeger、Zipkin、Statsd 等）
  • Process：批次、過濾、採樣、enrich
  • Export：送到任何後端（OTLP、Datadog、Grafana、Splunk 等）

Collector 部署模式

模式說明適合
Sidecar每個 pod 旁邊一個 CollectorK8s + low-cardinality 場景
Daemonset每個 node 一個 CollectorK8s 主流
Gateway集中 cluster，所有應用送過來大型 cluster + 集中 policy
StandaloneVM / 獨立 process非 K8s 環境

    send_batch_size: 512

  resource:
    attributes:
      - key: environment
        value: production
        action: insert
      - key: cluster
        value: us-east-1
        action: insert

  tail_sampling:
    decision_wait: 10s
    policies:
      - name: errors-always
        type: status_code
        status_code: { status_codes: [ERROR] }
      - name: slow-traces
        type: latency
        latency: { threshold_ms: 1000 }
      - name: probabilistic-sample
        type: probabilistic
        probabilistic: { sampling_percentage: 10 }

        match_type: strict

    verbosity: detailed

      exporters: [prometheusremotewrite]

      receivers: [otlp]
      processors: [batch, resource]

關鍵設計：

  1. Pipeline 分 signal：traces / metrics / logs 獨立配置
  2. Tail-based sampling：100% 錯誤 + 慢 trace + 10% 隨機（典型 production 模式）
  3. Filter 剔除 /health 噪音：別讓 healthcheck 占滿 quota
  4. Resource attributes：環境 / cluster 等 metadata 統一加
  5. 批次處理：減少 export 次數降低成本

57.6 — Semantic Conventions：跨工具統一

OTel 最被低估的貢獻是 Semantic Conventions（語意約定）— 跨工具統一命名。

範例

HTTP request 的 attribute 規範：

  • http.request.method — GET / POST / etc.
  • http.response.status_code — 200、404、500
  • http.route — /users/:id（不是 /users/123）
  • url.full — 完整 URL
  • server.address + server.port
  • client.address — 客戶端 IP
  • user_agent.original

所有 OTel 工具遵守同一套命名。當 Grafana 換到 Honeycomb 時，dashboard query 不需重寫。

2025 年穩定的 conventions

  • HTTP Server / Client：v1.0 stable
  • Database：v1.0 stable
  • Messaging（Kafka / RabbitMQ）：stable
  • RPC（gRPC）：stable
  • Resource（service.name、deployment.environment）：stable
  • GenAI / LLM：experimental（2025-2026 快速演進）
  • FaaS / Serverless：experimental

對 reverse proxy 工程師：遵守 conventions = 跨工具可移植。隨意命名 = vendor lock-in。

58.7 — 2026 年的成本控制策略

觀察性成本失控是2024-2026 年最常見的工程災難之一。預防措施：

原則 1：Cardinality 是最大成本驅動

Metric 的label cardinality（不同 label 組合數）直接影響成本。常見爆炸：

  • http_requests_total{user_id="...", url="..."} — user_id 100K + URL 1K = 100M series
  • 把 trace_id 當 label — 直接爆 cardinality
  • 沒 normalize 的 URL（/users/123 vs /users/{id}）

務必normalize：用 http.route（template）而非 http.target（具體 path）。

原則 2：Sampling Aggressively

  • Tail-based sampling 在 Collector：100% errors + 100% 慢 + 1-10% 隨機
  • 對 healthcheck / metrics endpoint 完全 drop
  • 對 high-volume noisy span 提高 sampling threshold

原則 3：Retention 分層

  • Hot（< 7 天）：完整 trace + metric + log，high-resolution
  • Warm（7-30 天）：aggregated metric、sampled trace
  • Cold（30+ 天）：archived log + metric summary（S3 / GCS 便宜儲存）

原則 4：監控觀察性成本

對「觀察性自己的成本」設 alert：

  • 每月支出 vs budget
  • 每 service / team 的觀察性 cost attribution
  • 異常用量增長（突然 +50% 通常是 bug 或濫用）

tcpdump、Wireshark、mitmproxy、bpftrace — 經典工具與 eBPF 加成

Reverse proxy 工程師不可或缺的能力是「事情壞了能查」。本章解構經典 debug 工具（tcpdump / tshark / Wireshark / mitmproxy / dig / httpie / curl / strace / dtrace / bpftrace）、現代加成（OTel trace、service mesh debug、Pyroscope eBPF profile），與一個系統化的 troubleshoot 方法論。

Reverse proxy 工程師面對的真實工作往往不是「新功能設計」，而是「客戶 A 在 5 分鐘前突然 502，剩下 99% 流量正常」這類 debug。本章把多年來累積的 debug 工具與技術整理成一個戰備系統。從 packet 層的 tcpdump 到 syscall 層的 strace、kernel 層的 bpftrace，到應用層的 OTel trace — 每一層都有它的位置。Debug 能力是 reverse proxy 工程師的真正 試金石。

59.1 — Troubleshoot 方法論：先想再 debug

新手工程師看到問題立刻 tcpdump。資深工程師先想 5 分鐘。系統化方法：

USE 方法（Brendan Gregg 提出）

對每個資源（CPU、memory、disk、network）檢查三件事：

  • Utilization — 使用率（CPU 多高？頻寬多滿？）
  • Saturation — 飽和度（queue 多深？wait time 多長？）
  • Errors — 錯誤（drop packet？OOM？disk error？）

RED 方法（給服務）

對每個 service 監控：

  • Rate — 請求速率（RPS）
  • Errors — 錯誤率（4xx / 5xx）
  • Duration — 延遲分布（p50 / p95 / p99）

7 步 Troubleshoot 流程

  1. 定義症狀：哪個 endpoint？什麼 status code？多少比例？什麼時候開始？
  2. 看 Dashboard：RED metrics 與 USE metrics 哪邊異常？
  3. 看 Trace 範例：找一個壞 trace，看哪個 hop 出問題
  4. 看 Log：那個 trace_id 在 log 裡印了什麼
  5. 形成假設：「可能是 X」
  6. 驗證假設：用 tcpdump / strace / bpftrace 等專業工具
  7. 修復 + 寫 postmortem

跳過前 5 步直接跑 tcpdump 是常見錯誤 — 你會 capture 大量無用 packet，然後在 Wireshark 裡迷路。先有假設再 debug。

59.2 — Packet 層：tcpdump 與 tshark

當你需要看「實際在線上飛的是什麼」，packet capture 是最終仲裁者。

tcpdump 常用 incantation

tcpdump 必備配方# 1. 抓特定 port 流量，存到檔案（給 Wireshark 用）
sudo tcpdump -i any -nn -s 0 -w /tmp/cap.pcap port 443

# 2. 只看特定 host + port
sudo tcpdump -i any -nn host 10.0.1.5 and port 443

# 3. 看 SYN packet（debug TCP 連線建立）
sudo tcpdump -i any -nn 'tcp[tcpflags] & tcp-syn != 0'

# 4. 看 RST packet（debug 連線異常斷）
sudo tcpdump -i any -nn 'tcp[tcpflags] & tcp-rst != 0'

# 5. ASCII dump 看 plain text（HTTP 不加密時）
sudo tcpdump -i any -nn -A port 80

# 6. Ring buffer：滾動 5 個 100MB 檔（避免吃光磁碟）
sudo tcpdump -i any -W 5 -C 100 -w /tmp/cap.pcap port 443

# 7. UDP（QUIC / HTTP/3）
sudo tcpdump -i any -nn -s 0 -w /tmp/quic.pcap udp port 443

tcpdump flag 速查

  • -i any：所有介面（K8s 場景特別有用）
  • -nn：不解析 IP / port name（更快、更可讀）
  • -s 0：完整 packet（不截斷）
  • -w file：存 pcap，給 Wireshark 分析
  • -A / -X：ASCII / hex dump（明文協定 debug）
  • -c N：抓 N 個 packet 後停

tshark：tcpdump 的進階版

tshark（Wireshark 的 CLI 版）能解析協定，在 production 直接跑出可讀輸出：

# 看 TLS handshake，列出 SNI
sudo tshark -i any -Y 'tls.handshake.type == 1' \
  -T fields -e tls.handshake.extensions_server_name

# 看 HTTP/2 frame
sudo tshark -i any -Y http2 -T fields \
  -e http2.streamid -e http2.type -e http2.headers.method

# 計算 TCP retransmission 比例
sudo tshark -i any -Y 'tcp.analysis.retransmission' -q -z io,stat,10

Wireshark：終極分析工具

Wireshark 是 GUI 工具，把 pcap 檔載進去後：

  • Follow TCP/HTTP/QUIC stream：跨多 packet 重組對話
  • Statistics → Conversations：哪些 IP 對話最多
  • Statistics → Flow Graph：視覺化 TCP / HTTP 序列
  • Decode TLS：有 SSLKEYLOGFILE 可解密自家流量
  • Decode QUIC：Wireshark 4.0+ 支援 QUIC v1（HTTP/3）

59.3 — TLS 流量：mitmproxy 與 Wireshark + SSLKEYLOGFILE

2026 年 95%+ 流量是 TLS 加密 — tcpdump 看到只是密文。三條路：

路 1：SSLKEYLOGFILE（Chrome / Firefox / OpenSSL）

瀏覽器或 OpenSSL-based 應用設環境變數：

# 啟動 Chrome 時 export
SSLKEYLOGFILE=/tmp/sslkeys.log google-chrome

# 或對 OpenSSL-based program
SSLKEYLOGFILE=/tmp/sslkeys.log curl https://api.example.com

# 在 Wireshark：Edit → Preferences → Protocols → TLS
# → (Pre)-Master-Secret log filename → /tmp/sslkeys.log
# Wireshark 自動解密這個 capture

限制：只能解自己機器的流量。對 production server 沒幫助。

路 2：mitmproxy（攔截 + 重簽）

mitmproxy（cf. Ch.21）是 forward proxy 設計，client 信任 mitmproxy CA 後就能看到所有 TLS 流量解密後的內容：

# 啟動 mitmproxy（預設 port 8080）
mitmproxy

# 或 web UI
mitmweb

# 或腳本化（自動修改流量）
mitmdump -s script.py

# Client 設 HTTPS_PROXY 並信任 mitmproxy CA
export HTTPS_PROXY=http://localhost:8080
# mitmproxy CA: ~/.mitmproxy/mitmproxy-ca-cert.pem

路 3：eBPF + uprobe（kTLS / 應用層）

2024-2026 年新興工具如 Pixie（New Relic / CNCF Sandbox）、Cilium Tetragon用 eBPF uprobe 攔截 OpenSSL / GnuTLS 函式呼叫，不需 client 信任 CA：

  • Pixie 自動 decode HTTP/2、gRPC、PostgreSQL、MongoDB、MySQL、Redis、DNS、Kafka
  • Cilium Hubble：K8s 流量觀察（含 L7）
  • 無侵入：不需改 application、不需重啟、不需 CA 信任

59.4 — DNS Debug：dig 與 dog

DNS 是 reverse proxy 工程師每週至少一次會碰到的 debug 點。

dig 必備配方

# 1. 基本查詢
dig api.example.com

# 2. 跳過 local cache，直接問權威 DNS
dig +trace api.example.com

# 3. 指定 DNS server（測 ISP DNS / 公共 DNS 差異）
dig @8.8.8.8 api.example.com
dig @1.1.1.1 api.example.com

# 4. 看完整答案 (含 NS / glue records)
dig api.example.com +noall +answer +authority +additional

# 5. 看 IPv6 (AAAA)
dig api.example.com AAAA

# 6. DNSSEC 驗證
dig api.example.com +dnssec +multi

# 7. 查 reverse DNS（PTR）
dig -x 8.8.8.8

dog：dig 的現代版

dog（Rust 寫的 DNS client）有彩色輸出 + 友好 UI，2024+ 越來越多人改用：

dog api.example.com A AAAA MX                  # 一次多 type
dog @1.1.1.1 api.example.com --short        # 簡短輸出
dog api.example.com --json                     # JSON for scripting

常見 DNS debug 場景

  • 「我這邊看正常，他們看不到」：用 dig +trace + 不同 DNS server 比對
  • DNS propagation 延遲：TTL 多久？看 Cache-Control / 用 dnschecker.org
  • CNAME 鏈太長：用 dig +trace 看完整鏈
  • SRV / TXT records：dig _service._proto.example.com SRV
  • DNSSEC fail：dig +dnssec 看是否 AD flag set

59.5 — HTTP 層：curl 與 httpie

Curl 是「HTTP debug 的瑞士刀」。Reverse proxy 工程師應該精通 curl。

curl 必備 flag

# 1. 看完整 request + response（最常用）
curl -v https://api.example.com/users

# 2. 看 headers only
curl -I https://api.example.com/

# 3. 看 HTTP/2 frame（debug 多工問題）
curl --http2 -v https://api.example.com/

# 4. 用 HTTP/3
curl --http3 -v https://api.example.com/

# 5. 跳過 TLS 驗證（debug self-signed cert）
curl -k https://internal.local/

# 6. 強制特定 IP（繞過 DNS）
curl --resolve api.example.com:443:10.0.1.5 https://api.example.com/

# 7. 計時（看 phases）
curl -w "@curl-format.txt" -o /dev/null -s https://api.example.com/
# curl-format.txt:
#   time_namelookup:    %{time_namelookup}\n
#   time_connect:       %{time_connect}\n
#   time_appconnect:    %{time_appconnect}\n  ← TLS 完成
#   time_starttransfer: %{time_starttransfer}\n  ← TTFB
#   time_total:         %{time_total}\n

# 8. 強制 cipher（debug TLS interop）
curl --tlsv1.3 --tls13-ciphers TLS_AES_128_GCM_SHA256 https://...

# 9. POST JSON
curl -X POST https://api.example.com/users \
  -H "Content-Type: application/json" \
  -d '{"email":"a@b.com"}'

httpie：curl 的友好版

httpie 對互動式 debug更友好：

http GET api.example.com/users Authorization:"Bearer xxx"
http POST api.example.com/users email=a@b.com name=Alice
http --json POST api.example.com/users "emails:=[\"a@b.com\"]"
http --verify=no https://internal.local/    # 跳過 cert
http --download https://example.com/file.zip

59.6 — System 層：strace、bpftrace

當網路層看不出問題，要進入 syscall 與 kernel 層。

strace：應用層 syscall 追蹤

# 看一個 process 在做什麼 syscall
sudo strace -p $(pgrep nginx) -f -e trace=network

# 統計：哪個 syscall 最常呼叫
sudo strace -c -p $(pgrep nginx)

# 看檔案操作（debug config 是否載入）
sudo strace -e trace=openat,read,close -p $(pgrep nginx)

# 限制：strace 會顯著拖慢 process
# Production 慎用，開發環境 / staging 為主

bpftrace：eBPF 的 awk

bpftrace 是2026 年 Linux debug 的最強利器 — 用簡單腳本語言寫 eBPF program，極低開銷（適合 production）。

bpftrace 必備配方# 1. 統計 syscall 出現次數
sudo bpftrace -e 'tracepoint:syscalls:sys_enter_* { @[probe] = count(); }'

# 2. 看 TCP retransmit（哪個 IP retransmit 多）
sudo bpftrace -e 'kprobe:tcp_retransmit_skb { @[ntop(args->saddr)] = count(); }'

# 3. HTTP request 延遲分布（須 trace nginx 的 ngx_http_handler）
sudo bpftrace -e '
uprobe:/usr/sbin/nginx:ngx_http_request_handler { @start[tid] = nsecs; }
uretprobe:/usr/sbin/nginx:ngx_http_request_handler /@start[tid]/ {
    @ms = hist((nsecs - @start[tid]) / 1000000);
    delete(@start[tid]);
}'

# 4. 哪個 file descriptor 在 leak
sudo bpftrace /usr/share/bpftrace/tools/opensnoop.bt

Brendan Gregg 的 bpftrace tools

Brendan Gregg（Netflix）維護 bcc-tools + bpftrace tools，幾十個現成腳本：

  • tcptop — 哪個 TCP connection 流量最多
  • tcpretrans — 即時看 retransmit
  • biolatency — disk I/O 延遲分布
  • execsnoop — 哪個 process 在被 fork
  • opensnoop — 哪個檔案在被開

59.8 — eBPF Profiling：Pyroscope 與 Parca

當「service 慢」但找不到具體 span，可能是CPU 在做某些事。Continuous profiling 解這個問題。

Pyroscope（Grafana 收購 2023）

  • eBPF-based，無侵入
  • 持續 sample 每個 process 的 stack trace
  • 產出 flame graph — 「CPU time 都花在哪」
  • 整合 Grafana：trace ↔ profile 跳轉

Parca（Polar Signals）

類似 Pyroscope，eBPF-based、Prometheus 整合。Parca 與 Pyroscope 在 2023-2024 是直接競爭，Grafana 收購 Pyroscope 後 Parca 重點轉向 vendor-neutral profile format（pprof 演進）。

為何 Continuous Profiling 重要

  • Trace 看「哪個 service 慢」
  • Log 看「什麼錯誤訊息」
  • Profile 看「CPU 真的在做什麼」

三者組合是 2026 年完整 observability。eBPF 讓 profile 從「開發環境工具」變成「production always-on」。

59.9 — 給 reverse proxy 工程師的 Debug 8 條原則

  1. 先想再 debug：USE / RED 方法、形成假設、再選工具
  2. 每層有對應工具：應用 OTel → syscall strace → kernel bpftrace → packet tcpdump
  3. tcpdump 用 ring buffer：-W 5 -C 100，避免吃光磁碟
  4. TLS 流量用 mitmproxy 或 Pixie eBPF：別寄望 SSLKEYLOGFILE 在 production 有用
  5. 學 bpftrace：2026 年 production-safe debug 利器
  6. service mesh 工具：Linkerd tap / Istio kiali / Cilium hubble 是 K8s 必備
  7. 持續 profile：Pyroscope / Parca 在 production always-on
  8. 每次 incident 寫 postmortem：debug 知識比工具重要
本章要點
  • 方法論優先：USE（CPU/mem/disk/network 看 Utilization/Saturation/Errors）+ RED（service 看 Rate/Errors/Duration），先想再 debug。
  • Packet 層：tcpdump（ring buffer + flag 速查）+ tshark（協定解析）+ Wireshark（GUI 終極分析、4.0+ 支援 QUIC）。
  • TLS 流量 3 條路：SSLKEYLOGFILE（自家機器）/ mitmproxy（forward proxy 攔截）/ eBPF + uprobe（Pixie / Cilium Tetragon 無侵入）。
  • DNS 層：dig 必備配方（+trace、@server、AAAA、+dnssec）+ dog（Rust 現代版）。
  • HTTP 層：curl --http2/--http3、--resolve、-w 計時、httpie 友好版。
  • System 層：strace（會拖慢，dev 為主）、bpftrace 是 2026 production-safe 利器，Brendan Gregg bpftrace tools（tcptop、tcpretrans、biolatency 等）。
  • 應用層 OTel debug 標準流程 6 步：alert → RED dashboard → 慢 trace → flame graph 找慢 span → log 關聯 → 形成假設 → bpftrace 驗證。
  • Service mesh debug：Linkerd tap / Istio kiali / Cilium hubble / Envoy admin。
  • eBPF Profiling：Pyroscope（Grafana 2023 收購）+ Parca，trace + log + profile 三位一體 observability。
  • 8 條工程師原則：先想再 debug、每層對應工具、tcpdump ring buffer、TLS 用 mitmproxy 或 Pixie、學 bpftrace、service mesh 工具、持續 profile、寫 postmortem。

Chapter 60

性能測試
wrk、vegeta、k6 — load test 方法論與真實 production 場景

「能擋多少 QPS？」是 reverse proxy 工程師最常被問的問題，也是最容易答錯的問題。本章解構性能測試方法論、wrk / vegeta / k6 三大工具對比、capacity planning、與真實 production scenario 的測試設計。

性能測試是 reverse proxy 工程師常做但經常做錯的事。「跑 wrk 半小時、看 RPS」這樣的測試幾乎沒有意義。真實 production 流量有 long-tail latency、突發 burst、multi-tenant 干擾、cold cache 等複雜性。本章建立性能測試的方法論，介紹三大主流工具，並提供真實 production scenario 的測試設計。

60.1 — 為何「跑 wrk 看 RPS」不夠

典型「糟糕的性能測試」長這樣：

wrk -t10 -c100 -d30s http://api.example.com/users
→ Requests/sec: 12,345
→ Latency avg: 8ms
→ 「我們的 API 能擋 12K QPS」

這個測試的 5 個問題

  1. 單一 endpoint：真實流量是混合 endpoint 的（list、create、update、delete 比例不同）
  2. 無 think time：wrk 永遠 keep-pressing；真實使用者有 idle time
  3. Single payload：所有 request 一樣大小；真實有 small / medium / large 分布
  4. Cache 全 hit：第二次跑 cache 全暖；cold-start 場景沒測
  5. 沒看 long tail：avg 8ms 但 p99.9 可能是 5s（真實使用者體驗）

應該回答的問題

性能測試應該回答的真正問題：

  • 當每秒 X 請求時，p99 是多少？
  • 什麼時候開始 5xx error rate 上升？
  • 到極限時，哪個資源先飽和（CPU / memory / network / file descriptor）？
  • 突發 burst（如 5 倍流量 30 秒）後，多久能恢復正常 latency？
  • 單個慢 endpoint 會不會拖累整體？

這些問題無法用「跑 wrk 30 秒」回答。

Open-loop vs Closed-loop

關鍵差異：

  • Closed-loop（wrk 預設）：N 個 thread，每個發一個 request → 等回應 → 立刻發下一個。流量速率由 server 速度決定
  • Open-loop / Constant Arrival Rate（vegeta、k6 可選）：每秒固定 N 個 request 不論 server 是否回應完。流量速率固定

真實流量是 open-loop（使用者不會等你回完才下一個）。用 closed-loop 測試會嚴重低估真實負載。當 server 變慢，closed-loop 自動降低 RPS（看似還 OK），但 open-loop 會 queue 起來、暴露問題。

60.3 — wrk：micro-benchmark 之王

wrk 是最快的單機 load tester（C 寫的）。適合純粹比較場景：「Nginx 1.25 vs 1.27 在這個 endpoint 哪個快」。

wrk 進階用法# 基本：10 threads, 100 連線, 30 秒
wrk -t10 -c100 -d30s --latency http://localhost:8080/

# 加 latency 分布輸出
wrk -t10 -c100 -d30s --latency http://localhost:8080/
# 輸出：
# Latency Distribution
#   50%   2.50ms
#   75%   3.20ms
#   90%   5.10ms
#   99%   12.30ms

# Lua scripting：自訂 request
wrk -t4 -c50 -d30s -s post.lua http://localhost/
# post.lua:
#   wrk.method = "POST"
#   wrk.headers["Content-Type"] = "application/json"
#   wrk.body = '{"email":"test@example.com"}'

wrk2：constant arrival rate fork

原生 wrk 是 closed-loop。wrk2（Gil Tene fork）加入 -R N 固定 rate，是更好的選擇。

60.4 — vegeta：API contract 測試

vegeta（Go 寫）的 unique value：純 open-loop + 容易在 CI 跑。

vegeta 範例# 1. 簡單：100 RPS 持續 30 秒

60.5 — k6：scenario-based 測試之王

k6（Grafana Labs 維護）是2026 年複雜場景測試的首選。用 JavaScript 寫 scenario，能描述真實使用者行為。

Listing 60.1 · k6 真實 scenario// k6-test.js
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Rate, Trend } from 'k6/metrics';

const errorRate = new Rate('errors');
const apiLatency = new Trend('api_latency');

export const options = {
  scenarios: {
    // 流量逐步爬升 + 持平 + 下降
    rampup: {
      executor: 'ramping-arrival-rate',
      startRate: 10,
      timeUnit: '1s',
      preAllocatedVUs: 100,
      maxVUs: 1000,
      stages: [
        { duration: '2m', target: 100 },    // 爬到 100 RPS
        { duration: '5m', target: 100 },    // 持平 5 分鐘
        { duration: '2m', target: 500 },    // burst 到 500 RPS
        { duration: '5m', target: 100 },    // 回到 100 觀察恢復
        { duration: '2m', target: 0 },      // 收
      ],
    },
  },
  thresholds: {
    // SLO：p95 < 200ms、error rate < 1%
    http_req_duration: ['p(95)<200'],
    errors: ['rate<0.01'],
  },
};

export default function () {
  // 模擬使用者行為：list → detail → create
  const listRes = http.get('https://api.example.com/products');
  check(listRes, { 'list 200': (r) => r.status === 200 });
  errorRate.add(listRes.status !== 200);
  apiLatency.add(listRes.timings.duration);
  sleep(1);   // 思考時間 1 秒

  const products = listRes.json('data');
  if (products && products.length > 0) {
    const detailRes = http.get(
      `https://api.example.com/products/${products[0].id}`
    );
    check(detailRes, { 'detail 200': (r) => r.status === 200 });
    sleep(2);
  }

  const createRes = http.post(
    'https://api.example.com/orders',
    JSON.stringify({ product_id: products[0].id, qty: 1 }),
    { headers: { 'Content-Type': 'application/json' } }
  );
  check(createRes, { 'order 201': (r) => r.status === 201 });
}

關鍵設計：

  1. Stages：模擬流量爬升 / 持平 / burst / 恢復
  2. Constant arrival rate（ramping-arrival-rate）：open-loop
  3. Multi-step scenario：list → detail → create 模擬真實使用者
  4. Sleep：模擬 think time
  5. Thresholds（SLO）：p95 < 200ms、error < 1%；不達標 CI fail
  6. Custom metrics：errorRate、apiLatency，dashboard 可視化

k6 整合生態

  • k6 Cloud（Grafana Cloud k6）：分散式跑、視覺化 dashboard
  • k6 Operator（K8s）：Kubernetes 上分散執行
  • xk6 extensions：gRPC、Kafka、Redis、SQL 測試
  • Browser API（k6 Browser）：Chromium-based real user simulation

60.6 — Capacity Planning：從測試到容量決策

性能測試的產出應該是 capacity plan。流程：

Step 1：定義 SLO

  • p95 latency < X ms
  • error rate < Y%
  • availability > Z%

Step 2：找 saturation point

逐步增加流量，找到「最後還能達 SLO」的最大 RPS。這個是單個 instance 的 max RPS。

Step 3：應用 headroom factor

業界共識：production 應該跑在 saturation 的 50-70%，給：

  • 突發流量緩衝
  • 單 instance 故障時其他 instance 接手的餘裕
  • 背景任務（GC、log rotation、健康檢查）

Step 4：算需要多少 instance

需要的 instance = 預期 peak RPS / (max RPS × headroom factor)

範例：
  預期 peak: 50,000 RPS
  Single instance saturation: 5,000 RPS at p95 = 200ms
  Headroom factor: 0.6
  → 需要 50,000 / (5,000 × 0.6) = 17 個 instance
  → 加 N+2 redundancy = 19 個 instance

Step 5：Re-run 與真實流量比對

Production 跑後比對 actual RPS、p95、error rate 與測試預測。差距大代表測試 scenario 不夠真實。

60.7 — 真實 production 場景

除了基礎 load test，真實 production需要驗證的場景：

Scenario 1：Cold Start

新 instance 上線後 cache 全冷。測試：rolling restart 時整體 latency 影響。

Scenario 2：Cache Stampede

熱門 key TTL 過期、千萬使用者同時打 origin。測試：用 single-flight pattern 是否生效。

Scenario 3：Slow Backend

注入 backend 慢（如 5s 回應），看 reverse proxy 是否：

  • 正確 timeout
  • 不會耗盡 connection pool
  • 不影響其他 backend 的請求

Scenario 4：Connection Surge

瞬間 10,000 個新連線（如雙 11 開賣）。測試 file descriptor、TCP backlog、TLS handshake 速率是否成為瓶頸。

Scenario 5：Long-running Soak

跑 24 小時，看記憶體 leak、file descriptor leak、GC pause 累積。

Scenario 6：Chaos Test

邊跑 load 邊隨機 kill instance、注入 network partition、限制 disk I/O。Netflix Chaos Monkey、LitmusChaos、ChaosMesh 工具。

60.8 — 給 reverse proxy 工程師的性能測試 8 條原則

  1. 用 open-loop：vegeta、k6、wrk2 — 真實流量是 constant arrival rate
  2. k6 寫 scenario，不是 endpoint：模擬真實使用者多步驟行為
  3. 看 percentile，不看 average：p99、p99.9 才是真實使用者體驗
  4. SLO 嵌入測試：k6 thresholds、vegeta 報告，不達 CI fail
  5. 多場景：cold start、cache stampede、slow backend、surge、soak、chaos 都要
  6. 50-70% headroom：production 別跑滿 saturation
  7. Continuous performance testing：每次 PR 跑迷你版，主要 release 跑完整版
  8. 產出 capacity plan：測試的目的是決策（要幾台、何時 scale）
本章要點
  • 「跑 wrk 看 RPS」5 個問題：單一 endpoint、無 think time、single payload、cache 全 hit、沒看 long tail。
  • 真正該答的問題：p99 / p99.9 / saturation point / 哪個資源先飽和 / burst 後恢復時間 / 慢 endpoint 是否拖累整體。
  • Open-loop vs Closed-loop：closed-loop（wrk 預設）流量速率由 server 決定 → 嚴重低估負載；真實流量是 open-loop（constant arrival rate）。
  • 三大工具對比：wrk（micro-benchmark / 最快 / closed-loop）、vegeta（純 open-loop / 簡單 CI）、k6（複雜 scenario 之王 / SLO threshold / 分散式）。

  • 真實 production 6 場景：cold start / cache stampede / slow backend / connection surge / long-running soak / chaos test。

Backstage、ArgoCD、Flux — Reverse proxy 在 Internal Developer Platform 的角色

2024-2026 年「Platform Engineering」從 buzzword 變主流。Backstage（Spotify 2020 開源、CNCF graduated 2025）、Port、Cortex、Humanitec — IDP 改變了 reverse proxy 工程師與 application developer 的協作方式。本章是 Part VII 最後一章，解構 IDP、GitOps、與 Part VII 結語。

61.1 — Platform Engineering 的崛起

「Platform Engineering」一詞 2022 年開始流行，2024-2026 年成為事實主流：

  • State of Platform Engineering Vol 4（2025）：32.8% 從業者把 observability 列為主要焦點
  • Gartner 預測：2026 年 80% 大型組織有正式的 Platform Engineering team
  • CNCF 2025 年新成立 Platform Engineering Working Group
  • Spotify 的 Backstage 2025 年 graduate from CNCF Incubation

為何崛起

2010-2020 是 DevOps 黃金期 — 「You build it, you run it」。但隨組織成長，這個模式遇到問題：

  • 每個 application developer 要懂太多（K8s、Terraform、observability、security、networking）
  • 標準化困難 — 100 個 team 100 種做法
  • cognitive load 過載 — application 工程師花 50%+ 時間做 ops

Platform Engineering 是回應：抽出 platform team，提供 self-service abstraction，application developer 專注業務邏輯。

IDP 的核心價值

Internal Developer Platform 是 platform team 的產品。對 application developer 提供：

  • 「給我一個 K8s namespace + DB + reverse proxy + observability + CI/CD」一鍵
  • 「新增一個 API endpoint」自動產生 Gateway API config + monitoring + alert
  • 「看我的 service 健康狀況」單一 dashboard
  • 「升級到 PQC TLS」platform 自動處理

61.4 — GitOps：reverse proxy 配置的演進

傳統 reverse proxy 配置變更：SSH 進 server → 改 nginx.conf → reload。問題：

  • 沒有 audit trail（誰改的？什麼時候？）
  • 沒有 rollback 機制（壞了怎麼還原）
  • 多 server 同步困難
  • 沒有 review process

GitOps是 2018+ 流行的範式：所有 infrastructure 配置都在 git，git 是 single source of truth、controller 自動 reconcile。

GitOps 4 原則（Weaveworks 提出）

  1. Declarative：配置是聲明式（K8s manifest、Terraform），不是命令式 script
  2. Versioned + Immutable：所有變更在 git，永遠可追溯
  3. Pulled Automatically：controller 從 git pull（不是 push 到叢集）
  4. Continuously Reconciled：controller 持續比對「git 狀態」與「cluster 狀態」，drift 自動修
