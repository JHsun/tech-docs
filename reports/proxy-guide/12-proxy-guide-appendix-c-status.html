APPENDIX C · HTTP STATUS

HTTP Status Code 速查
完整 RFC 9110 列表 · 含 reverse proxy 工程師最常碰到

本附錄是 HTTP Status Code 完整速查。按 5 大類別（1xx / 2xx / 3xx / 4xx / 5xx）+ reverse proxy 工程師重點章節。每個 status 含數字、英文名、簡明含意、何時用、常見誤用。基於 RFC 9110（HTTP Semantics, 2022）+ 業界常見非標準 status（如 Nginx 499）。

Status code 是 reverse proxy 工程師每天會看的東西，但很多誤用。最常見錯誤：401 vs 403 不分（給 client SDK 錯誤訊號）、502 vs 503 vs 504 混淆（debug 方向錯）、用 200 包 error response（破壞中介工具如 reverse proxy / monitoring）。本附錄目的：讓你準確用對 status code，並能看一眼就懂意思。

1xx · Informational（資訊性）

表示 server 收到 request、正在處理。Reverse proxy 工程師很少主動產生，但要懂 100 與 101 的處理。

CodeName含意
100ContinueClient 用 Expect: 100-continue header 詢問 server 是否願接受大 body；server 同意後 client 才送 body。Reverse proxy 必須正確 forward。
101Switching ProtocolsWebSocket / HTTP/2 升級用。Reverse proxy 看到此 status 必須切換為 raw TCP forwarding。
102Processing（WebDAV）長時間處理的進度通知。極少用。
103Early HintsServer 先送 Link header 讓 browser preload 資源，主回應仍隨後到達。Cloudflare、Fastly 大量使用優化 LCP。

2xx · Success（成功）

CodeName何時用
200OK萬用成功。GET / PUT / PATCH / DELETE 成功皆可。注意：別把 error response 包進 200 — 破壞中介工具與 monitoring。
201CreatedPOST 建立新資源成功。回應應含 Location header 指向新資源 URL。
202Accepted非同步處理：請求已接受、但尚未完成。Long-running job 場景。回應通常含 polling URL。
203Non-Authoritative Information回應來自 cache 或 transformation proxy，不是 origin。罕用。
204No Content成功但無 body。DELETE 成功、PUT 更新成功通常用。
205Reset Content表示 client 應 reset form / view。極少用。
206Partial ContentRange request 成功（影片串流、resumable download）。配合 Content-Range header。
207Multi-Status（WebDAV）批次操作多個 status。罕用。

3xx · Redirection（重新導向）

CodeName何時用
301Moved Permanently永久遷移。Browser / search engine 應更新書籤與索引。Method 可被改（POST → GET）— 有風險。
302Found暫時性。Method 可被改（同 301 風險）。歷史包袱：被 303 與 307 取代。
303See OtherPOST 後重導去 GET。「POST/Redirect/GET」pattern。明確 method 變更為 GET。
304Not ModifiedCache validation 成功（client 帶 If-None-Match / If-Modified-Since，server 確認沒變）。無 body。
307Temporary Redirect暫時性，method 不變（POST 仍是 POST）。取代 302 的安全選擇。
308Permanent Redirect永久性，method 不變。取代 301 的安全選擇。

2026 年最佳實踐：避免 301 / 302，用 308 / 307（method 不被改）。HTTPS 升級用 308。

4xx · Client Error（客戶端錯誤）

CodeName何時用
400Bad Request請求格式錯誤、無法解析。太籠統 — 應該配合 Problem Details（RFC 7807/9457）詳細說明。
401Unauthorized「我不知道你是誰」— 沒 token、token 過期、token 無效。應附 WWW-Authenticate header。不是「沒授權」（那是 403）。
402Payment RequiredHTTP 規格保留給「需要付費」場景。Stripe 等用於 quota 超出。
403Forbidden「我知道你是誰，但你不能存取這個資源」— 權限不足、quota 超出、地理限制。不是「身份問題」（那是 401）。
404Not Found資源不存在。也常用於「不想透露存在但無權限」（替代 403）。
405Method Not Allowedmethod 對該 URL 不支援（如 POST 到 read-only endpoint）。應附 Allow header 列出支援 method。
406Not AcceptableServer 無法滿足 Accept header（content negotiation 失敗）。
407Proxy Authentication RequiredForward proxy 要求認證。對應 Proxy-Authenticate header。
408Request TimeoutServer 等 client 完成 request 太久。
409Conflict狀態衝突（如 optimistic locking 失敗、資源已存在）。應說明衝突細節。
410Gone資源永久刪除（不會回來）。比 404 強的訊號。Deprecated API endpoint EOL 後用。
411Length Required需要 Content-Length header。少見。
412Precondition FailedIf-Match / If-Unmodified-Since 等 precondition 失敗。Optimistic concurrency 用。
413Content Too LargeRequest body 超過 server 上限。應附 Retry-After（如果是暫時的）。RFC 9110 改名（前為 Payload Too Large）。
414URI Too LongURL 太長。考慮改用 POST + body。
415Unsupported Media TypeContent-Type 不支援（如 server 期待 JSON 但 client 送 XML）。
416Range Not SatisfiableRange request 範圍超出資源大小。
417Expectation FailedExpect header 無法滿足。罕用。
418I'm a Teapot玩笑 — 1998 年愚人節 RFC 2324。Google、Cloudflare 真的有實作。
421Misdirected RequestRequest 送錯 server（HTTP/2 connection coalescing 場景）。
422Unprocessable Content語法 OK 但語意錯（validation 失敗）。RESTful API 常用於 schema validation 失敗。RFC 9110 改名（前為 Unprocessable Entity）。
423Locked（WebDAV）資源被鎖。罕用。
424Failed Dependency（WebDAV）依賴另一個失敗的 request。
425Too Early0-RTT 早期資料風險（replay）— server 拒接受。RFC 8470。
426Upgrade Required需升級 protocol（如要求 HTTP/2 或 TLS）。
428Precondition RequiredServer 強制要求 precondition header（避免 lost update）。
429Too Many RequestsRate limit 命中。應附 Retry-After 與 X-RateLimit-* headers。Reverse proxy 工程師每天會看到。
431Request Header Fields Too LargeHeader size 超過 server 限制（攻擊或 misconfig）。
451Unavailable For Legal Reasons因法律原因（GDPR 資料刪除、政府審查）拒絕。RFC 7725，名稱致敬 Ray Bradbury《華氏 451》。
499Client Closed Request (非標準)Nginx 特有：client 在 server 回應前就斷線。GitHub、Netflix 也用。不在 RFC 中但業界廣泛採用。

5xx · Server Error（服務器錯誤）

CodeName何時用
500Internal Server Error萬用 server 錯誤。太籠統 — 應記錄詳細 log 但對 client 只說 500。
501Not ImplementedServer 不支援該 method（如不支援 PATCH）。
502Bad GatewayReverse proxy 工程師關鍵：reverse proxy 收到 backend 的無效回應（connection reset、protocol error、空回應）。Backend 「有回應但壞」。
503Service UnavailableReverse proxy 工程師關鍵：服務暫時無法處理（rate limit、maintenance、overload）。應附 Retry-After。可重試。
504Gateway TimeoutReverse proxy 工程師關鍵：reverse proxy 等 backend 回應超時。Backend 「沒回應」（vs 502 是「回應壞」）。
505HTTP Version Not SupportedServer 不支援 client 用的 HTTP 版本。罕見。
506Variant Also NegotiatesContent negotiation 設定錯。罕用。
507Insufficient Storage（WebDAV）Server 沒空間存資源。
508Loop Detected（WebDAV）Infinite loop。
510Not Extended需要更多 extension。罕用。
511Network Authentication Required需要先通過 captive portal（如機場 WiFi 登入頁）。
520-527Cloudflare 特有 (非標準)Cloudflare 自訂的 origin 錯誤細分（520 unknown / 521 web server down / 522 timeout / 523 unreachable / 524 a timeout occurred / 525 SSL handshake failed / 526 invalid SSL cert / 527 Railgun error）。

Reverse Proxy 工程師重點：常見混淆與決策

401 vs 403：身份 vs 權限

最常被誤用的兩個 status：

  • 401 Unauthorized：「我不知道你是誰」— token 缺失 / 過期 / 無效。client SDK 看到 401 應觸發 refresh token flow。
  • 403 Forbidden：「我知道你是誰，但你不能存取這個」— 權限不足 / quota 超 / 地理限制。SDK 看到 403 不該重試。

誤用 403（應該 401）會讓 SDK 不會自動 refresh token。誤用 401（應該 403）會讓 SDK 無限重試 refresh。

502 vs 503 vs 504：debug 方向不同

三個都是 reverse proxy 場景常見，但意義完全不同：

CodeBackend 狀態Debug 方向
502有回應但壞（connection reset、protocol error）看 backend log 為何 crash / panic / protocol mismatch
503主動拒絕（過載、maintenance）看 backend 是否在 throttle / capacity 滿了 / 健康檢查失敗
504沒回應（timeout）看 backend 是否 hang / 慢 query / lock 卡住、reverse proxy timeout 設定

499（Nginx 特有）：debug 利器

Nginx 的非標準 499（Client Closed Request）非常有用：表示 client 在 server 回應前就斷線。常見原因：

  • Client 自行 timeout（如 SDK 5s、但 server 8s）
  • 使用者離開頁面 / 關 tab
  • Mobile 連線中斷
  • 上游 reverse proxy timeout

看到大量 499 通常代表：backend 不夠快或 client / proxy timeout 設太短。不是真正的 server error，但要看為什麼 client 沒等到 response。

Retry 規則速查

Code該 retry?怎麼 retry
2xx不需要—
3xx跟著 redirect注意 method 變化（301/302 vs 307/308）
4xx通常不要修 request 再送（401 例外：refresh token 後 retry）
429要，但 backoff遵守 Retry-After + exponential backoff + jitter
500不可重試（idempotent 才行）GET / PUT / DELETE 可，POST 不行（risk of duplicate）
502 / 503 / 504可以 retryexponential backoff + jitter，限制 max retries（如 3 次）
499—（client 早已 close）Server 角度無 retry 概念

非標準但業界廣泛使用

  • 499 Nginx Client Closed Request
  • 520-527 Cloudflare origin errors
  • 444 Nginx Connection Closed Without Response（用於 drop 攻擊流量、不回應）
  • 460 AWS ELB client 在 idle timeout 前 close connection
  • 463 AWS ELB X-Forwarded-For 頭超過 30 個 IP
  • 561 AWS API Gateway Unauthorized（custom）

這些不在 RFC但 reverse proxy 工程師會在 log 看到。知道它們意義就好，自己 API 不要用。

給 reverse proxy 工程師的 5 條 Status Code 原則
  1. 用對 status，不用 200 包 error：破壞中介工具與 monitoring
  2. 401 vs 403 區分清楚：給 SDK 正確訊號
  3. 502 / 503 / 504 各自意義不同：debug 方向也不同
  4. 429 必附 Retry-After 與 RateLimit-* headers：cf. Ch.56 §56.5
  5. 4xx error 用 RFC 7807/9457 Problem Details：含 type、title、status、detail、instance + extension fields

權威來源：IANA HTTP Status Code Registry（最完整、會更新）+ RFC 9110 §15 Status Codes。httpstatuses.io 是友好的速查網站。
