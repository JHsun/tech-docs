寫作之後
過程、致謝、進階閱讀路線

這篇後記給已經讀完 70 章主書的讀者。談寫作緣起、過程的取捨、致謝那些直接或間接影響本書的人，並提供你離開本書之後的進階閱讀路線。

寫作緣起

2025 年初我發現一個現象：reverse proxy 是每個 backend / SRE / platform 工程師每天會碰到的東西，但沒有一本完整、最新、繁體中文的教材。市面上的書要嘛是英文（《Nginx HTTP Server》、Cookbook 系列）、要嘛是針對特定產品（《Envoy in Action》）、要嘛是過時（HTTP/2 時代寫的、沒有 HTTP/3、沒有 service mesh、沒有 LLM API）。沒有一本能幫繁體中文工程師從入門到 2026 年前沿議題建立完整 reverse proxy 知識體系。

本書是這個缺口的回答。2026 年 reverse proxy 工程師該知道的所有議題，用繁體中文 + 英文 technical terms 寫成。70 章 ~156,000 字 — 不是百科全書（那會無法讀完），是能跟著工程師 5-10 年的工具箱。

寫作過程

本書分八個 Part 撰寫，每個 Part 開始前先做 web search 研究 2026 年最新狀態。寫作時間軸：

  • 2025 Q4 — 立大綱：70 章拆分、視覺設計系統、字型選擇
  • 2026/01-02 — Part I-II：基礎理論與協定深度
  • 2026/02-03 — Part III-IV：Forward / Reverse Proxy 實戰
  • 2026/03 — Part V-VI：AI Era 與進階架構
  • 2026/04 — Part VII-VIII：開發整合與維運可靠性
  • 2026/04 — 後記與附錄：你讀的這裡

幾個刻意的取捨

1. 為何用繁體中文 + 英文 technical terms 混寫？

純繁體中文翻譯 technical term（如把「load balancer」翻成「負載均衡器」、「postmortem」翻成「事後分析」）反而難讀。工程師日常用英文術語，混寫貼近真實工作語言。

2. 為何 2026 年焦點而非「永恆原理」？

本書有大量永恆原理（HTTP、TCP、queueing theory、SLO 等），但同時大量反映 2026 年現況（HTTP/3、MASQUE、PQC、LLM API、AI gateway、Backstage、incident.io 等）。工程師需要當下能用的知識，不是 2010 年寫的書。

3. 為何選 HTML 而非 PDF / EPUB / 紙本？

HTML 能：可搜尋、可程式碼高亮、可深色模式、可手機讀、可即時更新。70 章主書 + 6 個檔案 = 一個 portable 的可搜尋知識庫。

4. 為何附 ASCII 視覺化而非真圖？

ASCII 在任何 viewer 都能正確顯示、可複製貼上、不需 CDN、永遠不會壞。對技術書是合理的權衡。

5. 為何不做「code repo」？

本書中所有 Listing（Listing 53.1 到 70.1，總計 30+ 個）都是可獨立複製貼上的 production-ready 範例。沒設計成 repo 是因為配置會隨環境改、讀者該看懂後改寫，不是 copy-paste-and-pray。

進階閱讀路線

讀完本書你已經有 reverse proxy 工程師的完整當代知識體系。如果想再深入特定方向，這裡是 5 條進階路線：

路線 A：協定底層

想懂得「網路怎麼運作」到 packet 層級：

  • RFC 9110-9114（HTTP semantics、HTTP/1.1、HTTP/2、HTTP/3 — 直接讀標準）
  • 《TCP/IP Illustrated Volume 1》Stevens（經典）
  • 《HTTP/2 in Action》Barry Pollard
  • 《QUIC: A Multiplexed Stream Transport over UDP》Lucas Pardue 演講系列
  • Cloudflare blog — protocol 議題深度

路線 B：分散式系統

想懂得 multi-region / consistency / consensus 的根本：

  • 《Designing Data-Intensive Applications》Martin Kleppmann（必讀）
  • Raft paper（Ongaro & Ousterhout 2014）
  • Paxos Made Simple（Lamport）
  • Jepsen 測試報告 — distributed system 真實 bug
  • 《Database Internals》Alex Petrov

路線 C：性能與 Observability

想懂得「系統真的在做什麼」：

  • 《Systems Performance》Brendan Gregg（2 版 2020）
  • 《BPF Performance Tools》Brendan Gregg
  • 《Observability Engineering》Charity Majors, Liz Fong-Jones, George Miranda
  • OpenTelemetry 規格直接讀（opentelemetry.io）
  • SRECon 演講 archive（usenix.org/srecon）

路線 D：Security

想做 reverse proxy / API gateway 的安全：

  • OWASP Top 10（每年更新）+ OWASP API Security Top 10
  • 《The Tangled Web》Michał Zalewski
  • 《Real-World Cryptography》David Wong
  • 《Building Secure and Reliable Systems》Google（2020）
  • Cloudflare Security blog

路線 E：軟體工程文化

想成為更好的工程師（不只技術）：

  • 《The Phoenix Project》+《The Unicorn Project》Gene Kim（DevOps 故事）
  • 《Accelerate》Forsgren, Humble, Kim（DORA metrics 起源）
  • 《Staff Engineer》Will Larson
  • 《An Elegant Puzzle》Will Larson
  • 《The Manager's Path》Camille Fournier

最後的話

2026 年的 reverse proxy 工程師面對前所未有的複雜：HTTP/3、QUIC、MASQUE、後量子密碼、AI agent 流量、Kubernetes Gateway API、eBPF、Service Mesh、Multi-Region Active-Active...

但也面對前所未有的工具：OpenTelemetry 標準化 observability、Backstage 把self-service變現實、incident.io 把 incident response 變 Slack-native、AI SRE 開始輔助 debug、Prophet / Chronosphere 讓 capacity planning 可量化、FinOps 讓成本可治理。

每一代工程師都覺得自己在歷史轉折點。但 reverse proxy 工程師真的是 — 從 1990 年代連線管理開始，到 2010 年代 分散式系統，到 2020 年代 service mesh + AI。每 10 年遊戲規則改一次，但連接 client 與 server 之間的對話這個本質沒變。

祝你的 reverse proxy 永遠快、SLO 永遠達標、postmortem 永遠 blameless、page 永遠在工作時段。

願你在 2030 年某個深夜被 page 起床 debug 時，本書某一章某一節能幫你找到方向。

— 2026 年 4 月 26 日
於某個 reverse proxy 配置檔旁
