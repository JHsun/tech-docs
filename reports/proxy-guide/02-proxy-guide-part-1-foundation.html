Chapter 01

網路分層與封包旅程

要談 Proxy，必先有「層」的概念。一個 Proxy 工作在哪一層、能看見什麼、能改寫什麼，全由它在分層中的位置決定。

網路分層是電腦科學裡少數同時被「教科書奉為圭臬」與「實務工程師私下吐槽」的概念。OSI 七層模型誕生於 1980 年代的 ISO 標準化運動，原本是要與 TCP/IP 競爭的對手，最後卻在 IETF 協定標準的競賽中敗北 — TCP/IP 才是真正建構了今日網際網路的協定堆疊。然而 OSI 的七層命名（Physical、Data Link、Network、Transport、Session、Presentation、Application）卻反過來在後世成為「教學與排錯的通用語言」1，連 Cloudflare、Wireshark、tcpdump 這些當代主流工具的文件，仍然用「Layer 4 攻擊」「L7 規則」的方式溝通。

這個現象並不矛盾。OSI 從未在實作層面成功，但它提供了一張共享心智地圖：當有人說「這是 L7 問題」，所有工程師立刻知道意指應用層、與 HTTP header 或 SQL query 等語意有關，而不是 IP 路由。對 Proxy 技術來說，這張地圖尤其重要，因為Proxy 的能力完全由它運作的層次決定。

1.1 — 兩個模型，一張地圖

實務上同時存在兩個分層模型：理論的 OSI 七層，以及工程的 TCP/IP 四層（或 RFC 1122 風格的五層）。本書採取的立場是 — 用 OSI 名詞做溝通，用 TCP/IP 想實作。原因很簡單：OSI 的「Layer 7」是個業界共識，但真實世界沒有一個元件叫做「Session Layer」；而 TCP/IP 的 Application 層直接對應你 strace 觀察到的 socket 系統呼叫。

歷史札記 · "Layering considered harmful"

IETF 的 RFC 3439 有一節標題就叫「Layering Considered Harmful」2，明確批評嚴格分層在實作上的副作用 — 過度封裝會犧牲效能、阻礙跨層優化。這也解釋了為何 QUIC（HTTP/3 的傳輸層）會把 TLS 整合進去：跨層設計才能達到 0-RTT 連線建立。

本書尊重 OSI 作為溝通框架，但實作討論時會誠實面對「協定本身就是跨層的」這個事實。當第 12 章討論 HTTP/3 時，你會看到嚴格分層完全瓦解的例子。

「能看見什麼」決定「能做什麼」 — 這是本書最重要的單一公理之一。SOCKS5 是 L4 Proxy，它根本不知道流量內容是 HTTP 還是 SSH，所以它無法做 URL 路由、無法注入 OAuth token；Squid 是 L7 Proxy，它能讀 HTTP header，所以能做 URL 過濾與快取，但加密的 HTTPS 對它來說又退回了 L4（除非做 SSL Bump，詳見 Ch.11 與 Ch.19）。

這六個階段每一個都是 Proxy 的潛在介入點，而每種 Proxy 介入的階段不同：

  • DoH / DoT proxy 介入 ①（DNS 解析），改變誰來回答你的域名查詢；
  • VPN 介入 ② 之後，把所有 IP 封包包進新的隧道；
  • SOCKS5 proxy 介入 ③（TCP 連線建立），代客戶端與 Server 握手；
  • SSL Bump 介入 ④（TLS 握手），用偽造憑證代理整個 TLS 連線；
  • Reverse Proxy（如 Nginx）介入 ⑤（HTTP 請求），代表 Server 回應 Client；
  • API Gateway 也介入 ⑤，但更進一步在請求語意層做認證與限流；
  • MCP Server 是 ⑤ 的「再上一層」抽象 — 它代表使用者身分呼叫第三方 API。

1.4 — 常見的分層誤解

有經驗的工程師有時也會踩到這些觀念陷阱。以下是本書反覆會回頭澄清的三個典型誤解：

VPN 不是 L2，通常情況下是 L3

WireGuard、OpenVPN（in TUN mode）、IPsec 都是 Layer 3 隧道。它們處理 IP 封包，把整個 IP datagram 封包到加密通道中。L2 隧道（如 OpenVPN TAP mode、L2TP）是少數例外，主要用於需要橋接 MAC 廣播的場景。把 VPN 一概說成「Layer 2」是常見的口頭錯誤。

SOCKS5 不是純粹的 L4

SOCKS5 通常被歸類為 L5（Session Layer），這是 RFC 1928 自己的定位。它確實在 TCP 連線之上（所以高於 L4），但它只負責「協商目標位址，然後純轉送 byte stream」，不解析應用層協定。所以實務上把它當「L4-ish」也合理，但精確說它是 L5。

HTTP Proxy 不總是 L7

最有趣的誤解。當 HTTP Proxy 處理 CONNECT 方法（用於 HTTPS）時，它退化為 L4 純轉送 — 它看不見加密的 TLS 流量內容，只負責把 byte 雙向搬運。所以「同一個 HTTP Proxy 軟體」在處理 plain HTTP 時是 L7，處理 HTTPS 時實際上是 L4。這個事實對 SSL Bump、企業 DLP 部署都有深遠影響，將在 Ch.10、Ch.11、Ch.19 反覆出現。

Proxy 工作在哪一層 → 看得到什麼資訊 → 能做什麼決策。這個三段論貫穿全書 70 章。當你不確定某個 Proxy 能不能做某件事，先問「它在第幾層」，答案通常自然浮現。

Chapter 02

Proxy 的本質哲學
中介、轉發、終止、改寫的四種行為

跳出工具層次，理解 Proxy 在做的「行為類型」。這個觀念貫穿後面所有章節，比 Forward / Reverse 的二分法更實用。

傳統教科書介紹 Proxy 時，多半從「Forward Proxy / Reverse Proxy / Transparent Proxy」三種方向開始講起。這個分法沒錯，但它告訴你的是「Proxy 朝哪個方向工作」，並沒有回答「Proxy 在做什麼」。當你看到一個現代部署 — 例如 Cloudflare 同時是 Reverse Proxy、CDN、WAF、DDoS 防護 — 用「方向」分類已經完全不夠用。

本書提出另一套更實用的視角：Proxy 的四種核心行為。每個 Proxy 產品實際上是這四種行為的不同組合。理解了這個分類，你看任何 Proxy 設定檔（無論 Nginx、HAProxy、Envoy、Squid、Kong、Istio）時，都能立刻拆解它到底在做什麼。

Forward / Reverse 講的是「Proxy 替誰說話」；
四種行為講的是「Proxy 對流量做了什麼」。

2.2 — Forwarding：純轉送的純粹性

Forwarding 是最古老、最純粹的 Proxy 行為。它的核心承諾是：我不看你的內容，我只負責把 byte 從 A 搬到 B。SOCKS5 是這個行為的代表 — 它在握手階段協商目的地，之後就完全不解析後續傳輸的 byte 是 HTTP、SSH 還是 BitTorrent。

Forwarding 的優勢是協定無關性與效能。HAProxy 的 TCP mode、AWS Network Load Balancer 都是 Forwarding 的代表，它們能處理任何 TCP 流量，且因為不解析內容，CPU 負載很低，可以推到極高的 throughput。劣勢則是「能做的決策有限」 — 你只能根據 IP、port、TLS SNI（這是極少數可以從加密流量中看到的明文資訊）做路由。

2.3 — Terminating：連線的「中繼點」

Terminating 行為的本質是「打斷一個連線，建立另一個連線」。最常見的例子是 SSL/TLS Termination — Reverse Proxy（如 Nginx、AWS ALB）在邊界終止 HTTPS，與後端 server 用明文 HTTP 通訊。

Terminating 與 Forwarding 看起來都「中繼了流量」，但本質完全不同。Forwarding 中 Proxy 是透明的，client 與 server 在邏輯上仍是一條連線；Terminating 中 Proxy 是連線的端點，client → Proxy 是一條 TCP 連線，Proxy → server 是另一條（甚至可能是 HTTP/2 multiplex 出來的某條 stream）。

Terminating 帶來巨大彈性 — Proxy 可以重新協商協定（例如 client 用 HTTP/2，server 用 HTTP/1.1），可以做 connection pooling（一千個 client 連線複用十條到 server 的連線），可以做完整的 TLS 處理（TLS 1.3、OCSP stapling、ALPN）。代價是 Proxy 必須真的理解協定（無論 HTTP、gRPC 還是 WebSocket），複雜度遠高於 Forwarding。

2.4 — Inspecting：讀懂內容才能決策

當 Proxy 有能力解析應用層內容（無論透過 Terminating 解密，或在明文流量中讀取），下一步就是基於內容做決策。這就是 Inspecting 行為。

Inspecting 的典型應用包括：WAF（檢查 SQL injection 模式）、Rate Limiting（按 API path 與 user 限速）、Squid ACL（依 URL 域名決定是否放行）、API Gateway 認證（檢查 JWT 有效性）、AI Gateway 的 prompt injection 偵測（這是 Part V Ch.45 的主題）。

值得注意的是 Inspecting 本身不會改變流量 — 它做的是「放行 / 拒絕 / 記錄」三選一。當 Proxy 開始改寫流量時，就進入了下一個行為：Mutating。

2.5 — Mutating：流量的改寫

Mutating 是最有威力、也最危險的 Proxy 行為。它讓 Proxy 從「中介者」升級為「翻譯者」 — 它收到一個請求，把它改寫後再送出去。

實務上常見的 Mutating 模式：

  • Header injection：加入 X-Forwarded-For、X-Real-IP、Via、X-Request-ID
  • Header removal：移除 Server、X-Powered-By 等揭露後端實作的 header
  • URL rewriting：把 /api/v1/foo 改寫為內部的 /services/foo-svc/handler
  • Compression：在邊界做 gzip/brotli，後端不需處理
  • Authorization injection：API Gateway 收到 user JWT，去後端時改注入 service account token（這就是 OAuth 2.0 Token Exchange，詳見 Ch.39）
  • OAuth token swap：MCP Server 代表使用者身分呼叫第三方 API，自動填入正確的 OAuth token（這是 AI 時代代理的核心職責，詳見 Ch.40）

Mutating 的危險之處在於「它隱藏了真相」。應用層收到的流量已經不是原本的請求，這在 debug 時會非常困擾。Part VIII Ch.70（壓軸的反模式章）會展開這個討論：「過度的 Mutating 讓系統的行為無法被任何單點完整解釋」。

2.6 — 一個 Proxy 通常混合多種行為

理解了四種行為後，再看真實 Proxy 部署，你會發現幾乎沒有任何產品只做單一行為。以一個典型的 Nginx Reverse Proxy 配置為例：

# Listing 2.1 · 拆解一個 Nginx 設定中的四種行為
server {
    listen 443 ssl;                                  # ① Terminating: 結束 TLS
    ssl_certificate /etc/ssl/example.com.pem;
    ssl_certificate_key /etc/ssl/example.com.key;

    location / {
        # ② Inspecting: 檢查 User-Agent
        if ($http_user_agent ~* "BadBot") {
            return 403;
        }

        # ③ Mutating: 加上 X-Real-IP, 移除 Server
        proxy_set_header X-Real-IP $remote_addr;
        proxy_hide_header Server;

        # ④ Forwarding: 把處理過的請求轉送給上游
        proxy_pass http://backend_pool;
    }
}

listen 443 ssl 觸發 Terminating；if 條件做 Inspecting；proxy_set_header 與 proxy_hide_header 是 Mutating；最終 proxy_pass 把流量 Forward 到後端。一行設定混合四種行為。

訓練自己用「四種行為」的視角讀任何 Proxy 設定，能讓你在第一眼就看清楚這個設定的意圖，而不只是機制。當你接手別人寫的 50 行 HAProxy 配置，先掃過去問「哪行做 Forwarding、哪行做 Inspecting、哪行做 Mutating？」 — 整個設定的目的瞬間清晰。

2.7 — 為什麼這個分類比 Forward/Reverse 更實用

Forward / Reverse 只能告訴你「Proxy 站在哪一邊」 — 是站在 client 那側替 client 說話（Forward），還是站在 server 那側替 server 說話（Reverse）。但這個分法在現代部署中有兩個問題：

  1. 同一個產品可以兩用。Nginx 既能當 Forward Proxy，也能當 Reverse Proxy；HAProxy 主要做 Reverse 但 ACL 機制讓它也能當 Forward。Forward / Reverse 不是產品屬性，是部署用途。
  2. 它不告訴你功能。當別人說「我們有一台 Reverse Proxy」，你還是不知道它到底會 Inspecting 還是 Mutating；只知道它替 server 說話。

四種行為的視角直接告訴你「能做什麼」與「會做什麼」。這是工程語言，不是政治語言。本書以下所有討論優先使用四種行為，必要時才補上 Forward / Reverse 標籤。

本章要點

Forwarding（轉送）、Terminating（終止）、Inspecting（檢查）、Mutating（改寫） — 四種行為是本書的核心概念。任何 Proxy 都是這四種行為的組合。下次看設定檔時，試著用這四個動詞拆解每一行。

Chapter 03

三大分類維度全解

把先前討論過的多種視角整合成一個三維座標系。Layer × Direction × Purpose — 任何 Proxy 都能在這個立體空間中找到自己的位置。

讀者讀到這裡，腦中可能已經累積了多種 Proxy 分類方式：依層次（L3/L4/L7/App Logic）、依方向（Forward/Reverse/Transparent）、依行為（四種）、依用途（cache/security/API/mesh/CDN）。哪個才是「正確」的分類？答案是 — 沒有任何單一維度足以分類所有 Proxy。

本書採用三維分類框架：Layer × Direction × Purpose。這三個維度彼此獨立、互相補充，組合起來能精準定位任何 Proxy 產品。先在序言中匆匆掃過的這個分類體系，本章正式給出完整剖析。

3.1 — 為什麼用三維，而不是「幾大類」

傳統教科書喜歡列出「Proxy 的 N 大類型」。問題是這種列法把不同維度的概念混在同一張表上，例如：

  • HTTP Proxy vs SOCKS5 Proxy vs Reverse Proxy
    ← 前兩個是依協定（L7 vs L4-ish），第三個是依方向。混了。
  • Forward Proxy vs Transparent Proxy vs Cache Proxy
    ← 前兩個依方向，第三個依用途。又混了。

這種混亂的結果是：當你問「Cloudflare 是哪一類 Proxy」，沒有正確答案 — 它同時是 Reverse Proxy、CDN、WAF、L7 LB，且能在不同情境下扮演不同角色。三維分類則能精準描述：「Cloudflare 是 L7、Reverse 方向、用途包含 CDN+Security+API Gateway 的混合型代理」。

3.3 — 維度二：Direction（代理方向）

第二個維度告訴我們Proxy 替誰說話。

Forward Proxy（正向代理）

代理 Client 端。Client 主動設定使用 Proxy，目標 Server 看到的是 Proxy 的 IP，不是 Client 的真實 IP。常見用途：翻牆、隱藏 IP、企業上網管控、出站快取。代表產品：SOCKS5、Squid、企業 web proxy、Shadowsocks、V2Ray。

Reverse Proxy（反向代理）

代理 Server 端。Client 不知道背後實際有多少台 server。對 Client 而言，整個服務是一個 IP；對 Server 而言，所有流量都來自 Proxy。常見用途：負載平衡、SSL termination、WAF、快取加速。代表產品：Nginx、HAProxy、Envoy、AWS ALB、Cloudflare。

Transparent Proxy（透明代理）

Client 不知道自己在用 Proxy — 流量在 gateway 被強制劫持轉送。常見於企業 ISP 級別的內容過濾、強制快取。技術上通常透過 iptables REDIRECT 或 WCCP 實現。

Bidirectional / Mesh Proxy（雙向 / 網狀代理）

Service Mesh 的 Sidecar Proxy 同時扮演 Forward（出站時）和 Reverse（入站時）兩個角色，且每個 service 旁都有一個。這在 2010 年代後期才成為主流的新方向。

  • L4 LB（Cloudflare Spectrum）
  • L7 Reverse Proxy（基本服務）
  • CDN（核心業務）
  • WAF（Cloudflare Firewall）
  • API Gateway（API Shield）
  • ZTNA（Cloudflare Access）
  • Edge Compute Platform（Cloudflare Workers）
  • AI Gateway（Cloudflare AI Gateway，2024 年推出）

把 Cloudflare 歸為「CDN」是 2010 年的視角；把它歸為「Edge Cloud Platform」更貼近 2026 年現實。這個現象不只 Cloudflare 有 — Nginx 也從純 web server 演進到 Reverse Proxy、API Gateway、Service Mesh ingress（透過 Nginx Plus / NGINX Gateway Fabric）。

這種邊界模糊不是 bug 而是 feature。產品在解決真實問題時，會跨越人為設定的分類邊界。本書介紹任何產品時，會明確指出它主要位於哪些座標，以及它延伸到哪些座標。

3.7 — 如何用這套分類選擇技術方案

回到讀者最關心的實務問題：給定一個需求，如何選擇正確的 Proxy 類型？三維分類提供一個結構化的選型流程：

  1. 先問 Purpose：你要解決什麼問題？通用代理、API 認證、抗審查、AI tool call？這決定產品大類。
  2. 再問 Direction：是替 client 說話、替 server 說話、還是 mesh 雙向？這決定部署模式。
  3. 最後問 Layer：需要看到 HTTP header 嗎？需要看到 OAuth token 嗎？這決定技術選擇與效能 budget。

舉例：「我要做一個保護內網 API 的閘道，需要驗證 JWT、限流、記錄稽核。」

  • Purpose → API Gateway（認證 + 限流 + 稽核）
  • Direction → Reverse（保護內部服務）
  • Layer → App Logic（要懂 JWT 語意）
  • 結論 → Kong / APISIX / AWS API Gateway 三選一，視部署需求

再舉例：「我想讓所有員工的瀏覽器流量在進入網際網路之前先經過內部稽核。」

  • Purpose → Security + Cache（DLP、URL 過濾）
  • Direction → Forward（Transparent）
  • Layer → L7（要看 URL 與內容）
  • 結論 → Squid（自建）或 Zscaler / Netskope（SaaS）

Part III Ch.21 會展開企業級 Forward Proxy 的完整選型討論；Part V Ch.38 會展開 API Gateway 的選型矩陣。本章只示範方法。

本章要點

三個維度 — Layer × Direction × Purpose — 是讀懂任何 Proxy 的座標系。下次有人說「我們要做一個 reverse proxy」，請追問：「哪一層？什麼用途？」這個追問本身就是工程成熟度的展現。

Chapter 04

Proxy vs VPN vs NAT vs Tunnel vs Gateway

五個常被混用的詞 — 它們有些重疊、有些互斥，沒有一個工程團隊不曾為這些詞起爭執。本章一次釐清。

程式設計師之間有句俏皮話：「我們專業的兩大難題是 cache invalidation、命名（naming）—— 還有 off-by-one error」。網路工程也有自己的命名問題，最典型的就是 Proxy、VPN、NAT、Tunnel、Gateway 這五個詞。它們在實作上經常重疊，但概念上各有獨立內涵；含混使用會導致設計討論變成雞同鴨講。

本章採取的策略是先給每個詞一個單句精準定義，再展示它們在功能維度上如何重疊，最後用幾個現代產品的例子（WireGuard、Tailscale、Cloudflare、CGNAT）展示它們混搭的實況。

4.1 — 五個術語的精準定義

術語單句定義關鍵詞
Proxy代表某一方與另一方互動的中介者「代表」「中介」
VPN把私有網路延伸到公網之上的加密虛擬通道「網路延伸」「加密」
NAT改寫封包的 IP 與 port 以共享有限位址資源「位址改寫」「資源共享」
Tunnel把一個協定封包包進另一個協定的承載機制「封包進另一協定」
Gateway連接兩個不同網路（協定族 / 信任域）的轉換點「跨域轉換」

請仔細讀這五個定義 — 每個關鍵詞抓的是本質，不是實作。例如 VPN 的本質是「網路延伸」（讓你像在公司內網一樣存取資源），加密只是達成這目的的技術手段；Tunnel 的本質是「封包進另一個協定」，與是否加密無關（GRE 就是不加密的 Tunnel）；Gateway 的本質是「跨域轉換」，可以是協定轉換（如 IPv4-to-IPv6 Gateway）也可以是信任域轉換（如 API Gateway 從外部網路進入企業內部）。

4.3 — Proxy vs VPN：經典混淆

最常見的混淆是把 SOCKS5 翻牆工具叫做「VPN」，或者把 WireGuard 叫做「Proxy」。從 4.1 的定義看：

  • SOCKS5 是 Proxy（代表 client 與外部互動），不是 VPN（沒有「延伸網路」的語意 — 你的電腦不會多出一個虛擬網路介面與虛擬 IP）。
  • WireGuard 是 VPN（建立 wg0 介面，分配 10.0.0.x 私有 IP，整個網路延伸到對端），也是 Tunnel（封包包進 UDP 51820 端口），但不是 Proxy（沒有「代表某一方說話」的語意，它只是在傳輸層創造另一個網路平面）。

2026 年的 VPN 主流選擇是 WireGuard。它在 Linux kernel 5.6+ 原生整合，4,000 行程式碼相對於 OpenVPN 的 7 萬行有壓倒性的可審計性優勢。實測 WireGuard 比 OpenVPN/UDP 快 15–30%，比 IKEv2 快 5–15%。OpenVPN 仍保留價值，但只在「需要走 TCP/443 繞過 UDP 封鎖」或「需要細粒度 PKI 與 RADIUS 整合」的特殊場景。本書 Ch.52（ZTNA）會深入討論 Tailscale、Netbird 等基於 WireGuard 的現代 mesh VPN 方案。

4.4 — NAT：被誤解最深的「中介者」

NAT 是 1994 年 IETF 為了暫時緩解 IPv4 位址枯竭而設計的「臨時方案」（RFC 1631），結果這個臨時方案延續了三十年至今仍是主流。NAT 與 Proxy 在外觀上有相似之處 — 兩者都是「中介」，從目標 server 看來都是「中介者的 IP」 — 但內在運作完全不同。

NAT 不解析應用層

NAT 純粹改寫 IP header 的源 / 目的位址與 port，不碰更上層的內容。你的家用路由器在做 NAT 時，並不知道流量是 HTTP 還是 BitTorrent。Proxy 則普遍涉入應用層或至少會話層。

NAT 雙向自動，Proxy 通常單向

NAT 維護一張連線追蹤表（conntrack table）— 內網 client 發起連線時自動記錄映射，回應封包來時依映射轉送回去。這是雙向自動的。Proxy 則通常需要 client 主動使用（除非是 Transparent Proxy）。

CGNAT：當 NAT 變成基礎設施

Carrier-Grade NAT（CGNAT）是 NAT 的進階形態 — 不在你家路由器，而在 ISP 的核心網路。台灣、日本、以及多數開發中國家的行動網路都廣泛使用 CGNAT。架構上呈現「NAT444」：你的私有 IP（192.168.x.x）→ 家用路由器 NAT → 你 ISP 配給的私有 IP（100.64.x.x，CGNAT 專用區段）→ ISP 核心 NAT → 公網 IP。

CGNAT 帶來幾個實務後果：

  • 多個用戶共享同一公網 IP，導致 IP 黑名單誤殺（一人作惡，整個 IP 段被 ban）
  • 無法做 inbound NAT（家用 NAS、自架伺服器、P2P 變得困難）
  • 每個連線都消耗 ISP 一個 (公網 IP, port) 的資源，企業級 NAT 設備可能要支援百萬級的同時連線翻譯（F5 BIG-IP CGN 號稱可達 15 億 concurrent sessions）
  • 合規負擔：ISP 必須記錄每個 NAT 翻譯供執法調查（cf. Ch.7）

過渡到 IPv6 是長期解方，但 2026 年 IPv6 普及率仍未達理想。多種過渡機制在實際部署中並存：NAT64（IPv6-only client 訪問 IPv4-only server）、DS-Lite（IPv6 隧道封裝 IPv4）、464XLAT（行動網路常用），這些都是 Tunnel + NAT + Gateway 的混合形態。

4.5 — Tunnel：協定的套娃

Tunnel 的核心定義是「把一個協定的封包當作酬載，包進另一個協定中傳輸」。這個概念在現代網路中無處不在：

Tunnel 與 Proxy 不互斥。SSH local forward（ssh -L）同時是 Tunnel（TCP-over-SSH）和 Proxy（代表 client 連 server）；HTTP CONNECT 同時是 Tunnel（任意 TCP-over-HTTP）和 Proxy 機制（這是 HTTPS 在 Forward Proxy 下的標準做法，詳見 Ch.10）。

Tunnel 的代價是封裝開銷。每多一層封裝就多一層 header。WireGuard 的封裝 overhead 約 60 bytes（IP 20 + UDP 8 + WG header 32），所以 1500 byte 的乙太網路 MTU 下，內層 IP 封包最多只能用 1440 bytes — 這就是為何 VPN 連線必須處理 MTU 問題（不然會出現「TCP 三次握手 OK 但傳大檔案就掛掉」的詭異現象）。

4.6 — Gateway：信任邊界與協定邊界

Gateway 是這五個術語中最廣泛、也最模糊的一個。本書採取的精準定義是：Gateway 是位於兩個「域」邊界、執行轉換的元件。「域」可以是：

  • 協定族：IPv4 ↔ IPv6（NAT64 是 Gateway）、SS7 ↔ SIP（電信協定 Gateway）
  • 網路域：企業內網 ↔ 網際網路（邊界路由器是 Gateway）
  • 信任域：公開 API ↔ 內部 service（API Gateway 跨越信任邊界）
  • 抽象層：HTTP ↔ AMQP（Message Gateway）、HTTP ↔ Database（Database Gateway）

重要的是：「Gateway」這個詞在不同情境下意思非常不同 — 你電腦的「default gateway」是路由器，「API Gateway」是 Kong / Apigee，「IoT Gateway」是邊緣設備，「LLM Gateway」是 LiteLLM 之類的工具。它們唯一的共通點就是處於某種邊界、執行轉換。

API Gateway 與 Reverse Proxy 的關係是初學者最常混淆之處。本書的立場：API Gateway = Reverse Proxy + 應用語意（API 認證、限流、API 版本路由、API 編組）。所有 API Gateway 都是 Reverse Proxy，但不是所有 Reverse Proxy 都是 API Gateway。詳見 Ch.37。

4.7 — 一個產品多重身分：以 Tailscale 為例

以 2026 年最受歡迎的網狀 VPN 產品 Tailscale 為例，分析它的多重身分：

  • VPN 維度：建立私有網路覆蓋層（100.x.x.x Tailscale IP），所有 device 互相可達
  • Tunnel 維度：使用 WireGuard 加密通道，over UDP
  • NAT 維度：內建 NAT 穿透（STUN / DERP relay），跨越 CGNAT — 這個能力是傳統 WireGuard 沒有的
  • Gateway 維度：Subnet Router 與 Exit Node 模式讓 Tailscale 成為連接 Tailscale 網路 與 傳統網路 的 Gateway
  • Proxy 維度：嚴格意義不是 — Tailscale 沒有「代表 client 與外部說話」的語意

4.8 — 觀念地圖整合

Chapter 05

連線的生命週期

一條連線從建立到關閉的完整故事。Proxy 必須處理這個生命週期的每個階段，包括最容易出錯的「半關閉」與「異常終止」。

一條 TCP 連線從建立到關閉的故事，乍看簡單 — 三次握手開始、四次揮手結束 — 但 Proxy 在中介這條連線時，每個階段都有微妙的決策要做。本章用「連線生命週期」的視角，串起 Ch.1–4 的零散概念，並引入下一章效能模型所需的詞彙。

5.1 — 完整生命週期的七階段

一條 TCP 連線的完整生命週期可以分為七個階段：

  1. ① DNS 解析：client 解析目標域名為 IP（cache hit 為 0 RTT，cache miss 為 1+ RTT）
  2. ② TCP 三次握手：SYN → SYN-ACK → ACK（1 RTT）
  3. ③ TLS 握手（HTTPS only）：TLS 1.3 為 1-RTT；TLS 1.2 為 2-RTT
  4. ④ 應用層握手：HTTP 請求 / SOCKS 協商 / SSH banner
  5. ⑤ 資料傳輸：雙向 byte stream，可能含 keepalive 探測
  6. ⑥ 連線關閉：FIN → ACK → FIN → ACK（雙向獨立關閉）
  7. ⑦ TIME_WAIT：主動關閉端等待 2*MSL（Linux 預設 60s）

Proxy 介入時，每階段都會被「拆成兩段」 — client → Proxy 一段、Proxy → server 另一段（這是 Ch.2 提到的 Terminating 行為）。這意味著 Proxy 自己要做兩次 ① ② ③ ④，且必須協調兩邊的時序。

5.2 — 連線建立的成本

建立一條連線的成本，是理解 Proxy 效能與連線池策略的起點。以一條 client → server（直連，無 Proxy）的 HTTPS 連線為例：

階段RTT 數同地域 RTT≈10ms跨洋 RTT≈150ms
① DNS（cache miss）0–1 RTT0–10ms0–150ms
② TCP 三次握手1 RTT10ms150ms
③ TLS 1.3 握手1 RTT10ms150ms
④ HTTP 請求 + 回應1 RTT10ms150ms
總計3–4 RTT30–40ms450–600ms

這個表格揭示了一個殘酷現實：跨洋 HTTPS 直連，第一個請求至少 0.5 秒，且這個時間幾乎全花在握手而非傳資料。這就是為什麼 connection reuse（keep-alive）、TLS 1.3 的 0-RTT、HTTP/3 的 0-RTT QUIC 在現代 web 是命懸一線的優化。

5.3 — Keep-Alive 的演進

Keep-Alive 是 1990 年代為了解決 HTTP/1.0「每個請求都要新連線」的災難而引入的機制。它的演進史是一部「對抗握手成本」的史詩：

  • HTTP/1.0 + Keep-Alive header（1996，非標準擴充）：client 加上 Connection: keep-alive，server 同意後保持 TCP 連線給下一個請求重用。同一時間只能有一個 in-flight 請求。
  • HTTP/1.1（1997, RFC 2068）：persistent connection 成為預設行為，不需 Keep-Alive header。引入 pipelining（送多個請求不等回應）但實際很少啟用，因為中間 proxy 普遍實作有 bug。
  • HTTP/2（2015, RFC 7540）：徹底重新設計。一條 TCP 連線承載多個 stream，所有 stream 並行 multiplex，連線預期長期保持。Connection header 在 HTTP/2 是禁用的。
  • HTTP/3（2022, RFC 9114）：基於 QUIC。連線從 TCP 轉移到 UDP，TLS 整合進傳輸層，0-RTT resumption 成為常態。

在 Proxy 的世界裡，這些演進帶來複雜度：Reverse Proxy 經常需要協調不同版本的協定。例如 Nginx 上游接的是 HTTP/1.1 後端，但下游服務瀏覽器用 HTTP/2；中間就要做協定翻譯。Cloudflare 邊緣就更極端 — 接到 client 端 HTTP/3，到 origin 端可能還是 HTTP/1.1，整個請求要在邊界跨越三代協定。

5.4 — Connection Pool：Proxy 效能的關鍵

在 Reverse Proxy 場景，不重用後端連線是效能災難。每個 client 請求都建立新的 Proxy → backend 連線，意味著每個請求都付 ②③ 的成本。連線池（connection pool）解決這個問題。

連線池的設計參數通常包含：

  • Pool size：每個 backend 維持多少條長連線（HAProxy 的 http-reuse always、Nginx 的 upstream { keepalive N; }）
  • Idle timeout：閒置多久關閉（一般 60–900 秒；Cloudflare 預設 900s）
  • Max requests per connection：單條連線送多少請求後強制關閉重建（避免單一連線狀態無限累積）
  • Max concurrent streams（HTTP/2 only）：一條連線最多並行多少 stream（Cloudflare 預設 100，可調至 200+）

HTTP/2 的連線池與 HTTP/1.1 不同。HTTP/1.1 的連線池要看「並行度」 — 一千個 client 可能要 100 條 backend 連線並行；HTTP/2 的連線池看「multiplex 容量」 — 同樣一千個 client，可能只需要 10 條 backend 連線就夠（每條跑 100 個並行 stream）。這個差異對 backend 的負載觀察有巨大影響。

5.5 — Idle 與 Half-Open 的隱形殺手

連線生命週期最痛的部分不是建立與正常關閉，而是異常狀態。以下三種情境是 Proxy 必須面對的：

Idle Timeout 不一致

Client、Proxy、backend 三者的 idle timeout 通常不同。如果 backend 設 60s，而 Proxy 認為連線還活著（設了 300s），那當 Proxy 收到下一個請求要重用連線時，會送進已關閉的 socket，得到 ECONNRESET。所有 Proxy 都需要實作「連線健康檢查」或「樂觀重試」來掩蓋這個問題。

Half-Open Connection

最隱晦的失敗模式：client 或 server 一方崩潰、斷網、或經過 NAT 設備而 NAT 表項被清除，但另一方完全不知道。TCP 沒有強制 keepalive，所以 socket 可以無限期保持「ESTABLISHED」狀態。當 Proxy 試著用這條死連線時，要等到送資料失敗才會發現。解決方案有三類：

  • OS 級 TCP keepalive（Linux 預設 7200s 太長，通常要調到 60s）
  • 應用層 ping（HTTP/2 的 PING frame、gRPC keepalive、WebSocket ping）
  • 定期主動關閉重建（max_age、max_requests）

TIME_WAIT 累積

主動關閉連線的一方會進入 TIME_WAIT 狀態，預設保持 2*MSL（Maximum Segment Lifetime，Linux 為 60s）。Reverse Proxy 處理大量短連線時，每秒可能累積數萬個 TIME_WAIT，吃光本機的 ephemeral port 範圍（通常 28k–60k 範圍），導致無法建立新出站連線。Proxy 工程師熟悉的 TIME_WAIT bug 就是這個。解法：盡量讓後端主動關閉、調整 net.ipv4.ip_local_port_range、開 tcp_tw_reuse（注意不是 tcp_tw_recycle，後者在 Linux 4.12+ 已被移除因為破壞 NAT 環境）。

5.6 — 跨層中斷：Proxy 場景特有的麻煩

Proxy 把一條 client–server 連線拆成兩條，這帶來一個 client–server 直連時不存在的問題：當一邊出問題時，要怎麼通知另一邊？

考慮以下情境：

  • Client 突然斷線（手機切換 WiFi 到 4G） — Proxy 應該關閉到 backend 的連線嗎？
  • Backend 回 HTTP 500 但仍開著連線 — Proxy 該不該關閉到 client 的連線？
  • Backend 處理到一半崩潰、TCP RST 過來 — Proxy 該怎麼告訴 client？
  • Client 已收到部分回應但中途斷掉 — Proxy 應該幫忙重試嗎？

這些問題沒有「正確」答案 — 不同 Proxy 軟體有不同策略，且通常可以調整。Nginx 預設 proxy_next_upstream 在 backend 失敗時自動切換另一個 upstream（且只在請求還沒送出 body的情況下重試，避免重複扣款這類副作用）。Envoy 提供完整的 retry policy DSL，包括重試條件、最大次數、超時、abort 條件。HAProxy 的 retry 機制相對保守。本書 Ch.27–32 會分別討論。

5.7 — 連線數學：估算容量

理解連線生命週期後，可以做一個簡單但實用的估算：給定每秒請求數（RPS）與平均連線停留時間，需要多少條同時開啟的連線？

Little's Law：L = λ × W。L 是同時連線數、λ 是 RPS、W 是平均響應時間。

  • 若 RPS = 1,000，平均響應 50ms → 需要 50 條同時連線
  • 若 RPS = 1,000，平均響應 500ms（含 LLM 推論等慢請求）→ 需要 500 條
  • 若 RPS = 100,000（CDN 邊緣），平均 100ms → 需要 10,000 條同時連線

這個簡單公式在容量規劃時非常實用 — 它告訴你「一個 Reverse Proxy 處理 100k RPS 的 LLM 流量（每個請求 5 秒）」需要 50 萬條同時連線。沒有 connection pooling 與 HTTP/2 multiplexing，後端伺服器會被連線數本身壓垮，與請求 CPU 消耗無關。Ch.6 會展開效能模型。

本章要點

連線生命週期的七階段（DNS、TCP、TLS、應用握手、傳輸、關閉、TIME_WAIT），每個階段在 Proxy 場景都會被「拆成兩段」。掌握連線池、idle 處理、half-open 偵測，是 Proxy 工程師日常工作的核心。Little's Law 是你的隨身估算工具。

Chapter 06

Proxy 的效能模型
從 C10K 到 io_uring 與 eBPF

Proxy 是 I/O 密集型應用的典範。本章建立「Proxy 為何快」與「Proxy 為何慢」的理論模型，並追蹤從 1999 年 C10K 到 2026 年 io_uring 的演進軌跡。

要設計或選型一個 Proxy，必須先理解 Proxy 作為一個系統的「效能體質」。所有 Proxy（無論 Nginx、HAProxy、Envoy 還是 SOCKS5 server）本質上都在做同一件事：把資料從一個 socket 搬到另一個 socket。這聽起來很簡單，但要做到「同時搬一百萬條」就是過去三十年系統工程的核心戰場。

本章將追蹤這個戰場的關鍵戰役 — 從 1999 年的 C10K 問題，到 2010 年代的 epoll 與 kqueue，到 2020 年代的 io_uring 與 eBPF — 並建立評估 Proxy 效能的理論工具：吞吐量、延遲、連線數、CPU/Memory 使用、context switch 成本。

6.1 — Proxy 的效能維度

評估 Proxy 效能不是單一數字，而是多個維度的權衡：

維度單位典型優化方向
Throughput（吞吐量）requests/sec、Gbps批次化、零拷貝、SIMD
Latency（延遲）ms（p50/p99/p99.9）少 hop、少 context switch、減握手
Concurrent Connections條數event-driven 取代 thread-per-conn
Connection Establish Rateconns/secSYN cookie、accept queue tuning
CPU Efficiencycycles per byte避免拷貝、AES-NI 加速
Memory per ConnectionKB / conn小 buffer、共享狀態

這六個維度經常互相衝突 — 例如為了極致 latency 可能需要犧牲 throughput（不批次處理）；為了極致同時連線數可能需要犧牲單請求 latency（換用 event loop 而非 thread）。沒有「最快的 Proxy」，只有「最適合特定 workload 的 Proxy」。

6.2 — C10K 問題：1999 年的範式轉移

1999 年，Dan Kegel 寫了一篇至今仍被引用的文章「The C10K Problem」 — 一台 server 如何同時處理一萬個 client？當時主流的「每個 client 一個 thread」（thread-per-connection）模型在這個量級徹底失效：

  • 10,000 threads × 1 MB stack = 10 GB 記憶體
  • 10,000 個 thread 之間 context switch 的 CPU 成本壓垮系統
  • kernel 的 select() / poll() 對每個 fd 都要線性掃描，O(N) 成本

解法是「event-driven I/O multiplexing」 — 一個 thread 用 epoll（Linux 2.6, 2002）或 kqueue（FreeBSD, 2000）監視所有 socket，只有真正有事件的 socket 會被通知。這把 fd 監視從 O(N) 降到 O(1)，把記憶體成本從每連線 MB 級降到 KB 級。

這個範式轉移直接催生了現代 Proxy 的架構：

  • Nginx（2004）：master process + worker processes，每個 worker 用 epoll 處理數萬連線
  • HAProxy（2001 起持續演進）：單 process event loop，極致追求 connection-per-second
  • Squid（更早，但長期演進）：event-driven + worker model
  • Node.js（2009）：把這個模型帶到 JavaScript 世界

C10K 在 2010 年代就被宣告解決，但 Dan Kegel 也預見了下一個挑戰：C10M — 一千萬連線。這個問題的解法不在 user space，而要回到 kernel 與硬體。

6.3 — C10M 與 io_uring：2020 年代的革命

要處理 1000 萬連線，epoll 也不夠用了。瓶頸從「fd 監視」轉移到「系統呼叫的次數」 — 每個 read/write 都是一次 syscall，每次 syscall 都有 user/kernel context switch 的成本（約 1–2 μs）。處理百萬 RPS 時，光 syscall 開銷就吃掉 50% 的 CPU。

Linux 5.1（2019）引入 io_uring — 一個全新的 async I/O 介面，核心理念是「批次提交、批次完成、共享 ring buffer」。它與 epoll 的對比：

epoll（傳統）io_uring（新）
API 性質readiness 通知completion 通知
每 I/O 的 syscall1+ 次（epoll_wait + read）0–1 次（可批次提交）
支援操作read/write/accept幾乎所有 I/O，包括 fsync, statx
2025 新特性—multishot receive（一次 submit, 多次 completion）
當前採用Nginx, HAProxy, Envoy（主流）新興；ScyllaDB、實驗性 web servers

2025–2026 是 io_uring 從實驗走向生產的關鍵時間點。multishot receive（Linux 6.0+）讓一次 submit 對應多次 completion，徹底消除「accept-then-recv」的 syscall 來回。雖然主流 Proxy（Nginx、HAProxy、Envoy）目前還是基於 epoll，但 io_uring 的優勢在 multi-million RPS workload 下已開始顯現。預期 Nginx / Envoy 在 2026–2027 會逐步加入 io_uring backend。

6.4 — eBPF：把代理邏輯放進 kernel

io_uring 解決的是「user space 與 kernel space 來回的成本」。eBPF（extended Berkeley Packet Filter）走得更遠：把代理邏輯本身搬進 kernel，避開「進入 user space」這件事。

傳統 L4 LB 的封包流程，要 4 次 user/kernel 邊界穿越（封包進入網卡 → kernel network stack → socket → user space → Proxy 程式 → 寫回 socket → kernel → 網卡傳出）；eBPF 流程則是封包進入網卡後，eBPF program 在 kernel 直接決策並 forward，0 次 user space 介入。

這是 Cilium（Linux Foundation, 2017→主流 2024+）的核心架構。Cilium 用 eBPF 取代傳統的 kube-proxy（iptables based），效能提升 5–10x。在 2026 年，Cilium 已是 GKE / EKS / AKS 的預設 CNI 選項之一，成為雲原生 Proxy 的重要新範式。詳見 Ch.49。

eBPF 的代價是程式只能用 verifier 認可的子集 C 語言寫；無法做複雜應用層處理（仍需 user space proxy 處理 L7 / TLS）；需要較新 kernel（5.10+）。所以實務上現代雲原生架構是「eBPF + user space Proxy 混合」 — eBPF 處理 L3/L4 快速路徑，遇到需要 L7 處理的流量再轉給 Envoy / sidecar。Cilium 的 Ambient Mesh 與 Istio 的 Ambient Mesh 都採類似策略。

6.5 — Proxy 的延遲累積模型

Proxy 鏈中每加一層都會引入 latency。一個簡單但實用的模型：

Total Latency = Σ (Proxy_i 的處理延遲) + Σ (網路 hop 延遲)

典型 Reverse Proxy（如 Nginx）的處理延遲組成（穩定狀態下單請求約 50–100 μs Proxy 自身延遲）：

  • Accept：~10 μs
  • TLS handshake：~1–5 ms（首次連線；重用為 0）
  • HTTP parsing：~10–50 μs
  • Routing decision：~5–20 μs
  • Backend connection：~10 μs（pool 中）/ ~10 ms（建新）
  • Header rewriting：~5 μs
  • Response forwarding：雙向各加 ~10 μs

這個 50–100 μs 數量級是設計分散式系統的重要 budget。對於需要嚴格延遲（如 high-frequency trading、即時遊戲）的場景，每多一層 Proxy 就是一份成本；對於 web 流量（user 感知 100ms 以下都不會抱怨），這成本可以忽略。

p99 與 p99.9 的尾延遲更為關鍵。一條請求若經過 10 層 Proxy（API Gateway → mesh ingress → sidecar → service → sidecar → DB proxy），每層 p99 = 10ms，整條鏈 p99 不是簡單疊加，而是會更糟（因為每層都有獨立的尾延遲事件）。「p99 in tail amplification」是 Service Mesh 設計時的核心關注點，詳見 Ch.62。

6.6 — 吞吐量的瓶頸定位

當 Proxy 達不到預期 throughput，常見的瓶頸位置依優先順序：

  1. 單一 CPU core：很多 Proxy（如 HAProxy 早期、SOCKS server）是單 thread，瓶在一顆 core 的 100%
  2. 網卡中斷分散：高 PPS（packet per second）下需要 RSS（Receive Side Scaling）把流量分散到多 core
  3. TLS 加解密：未啟用 AES-NI 時 TLS 是 CPU 大戶
  4. 上游連線池不足：backend 已飽和但 Proxy 顯示空閒
  5. Ephemeral port 用盡：TIME_WAIT 累積（cf. Ch.5 §5.5）
  6. file descriptor 上限：Linux 預設 1024，生產環境需調至百萬級
  7. Memory bandwidth：在百 Gbps 級流量下成為瓶頸

偵測這些瓶頸需要的工具：perf、bpftrace、ss -s、ethtool -S、vmstat。Ch.62（效能調校）會展開完整工具箱。

6.7 — Proxy 是 CPU bound 還是 I/O bound？

新手最常問的問題之一。答案是：視 Proxy 在做什麼。

  • 純 L4 forward（如 HAProxy TCP mode）：I/O bound。CPU 主要消耗在 socket 操作。
  • L7 with parsing（如 Nginx 處理一般 HTTP）：混合，但 I/O 通常仍是瓶頸。
  • TLS Termination：可能 CPU bound（特別是無 AES-NI 時）。
  • WAF / 內容檢查：CPU bound，regex 與內容掃描是最大消耗。
  • API Gateway 含 JWT 驗證、限流計算：CPU bound。
  • AI Gateway 含 prompt injection 偵測：嚴重 CPU bound（甚至可能呼叫 ML model）。

理解這一點才能正確規劃硬體 — 純 L4 LB 用便宜 VM、給足網卡頻寬即可；AI Gateway 則需要 CPU 大量核心，甚至 GPU。一個常見錯誤：把所有 Proxy 都當作 I/O bound，結果 AI Gateway 在大流量下成為整個系統的瓶頸。

6.8 — 「快」的相對性

最後一個觀念：別追求絕對的「快」，要看相對的成本效益。

  • HAProxy 處理單請求約 20 μs，Nginx 約 50 μs，Envoy 約 100 μs — 但這個差異對 99% 的應用場景完全無感。
  • Envoy 慢一些是因為它做了更多事：xDS dynamic config、強型別 filter chain、完整 observability。如果你需要這些功能，Envoy 的成本是值得的。
  • Cilium 的 eBPF 比傳統 iptables kube-proxy 快 5–10x — 但這個差異要在大規模 Service Mesh 下才顯著，小集群差別不大。

當有人推銷「比 X 快 N 倍的 Proxy」時，先問三個問題：

  1. 什麼 workload 下測的？（純 forward 還是含 TLS？）
  2. 另一邊的設定是合理的嗎？（很多 benchmark 偷偷對比家用 Nginx vs 調優過的自家工具）
  3. 多出的速度，對我的場景有意義嗎？（從 50k RPS 跳到 200k RPS，但你只需要 10k RPS 怎麼辦？）

Part IV 與 Part VI 在介紹各種 Proxy 產品時，會持續引用本章建立的詞彙與模型。

本章要點

Proxy 效能是六維權衡（throughput、latency、concurrent connections、conn-per-sec、CPU efficiency、memory per conn），沒有單一答案。30 年技術演進的主軸是「減少 user/kernel 來回」 — 從 thread-per-conn → epoll → io_uring → eBPF。理解你的 Proxy 是 CPU bound 還是 I/O bound，比追求絕對速度更重要。

Chapter 07

Proxy 的法律、合規與倫理

技術中立的 Proxy 在法律世界裡並不中立。本章用工程師看得懂的方式，介紹 Proxy 部署不可不知的法律框架與倫理紅線。

Proxy 技術本身是中性的 — 它只是把 byte 從 A 搬到 B。但「在哪裡部署、誰在用、看到什麼內容、留多久 log」這些問題，會把同一個 SOCKS5 server 從工程工具變成法律問題。本章不是法律建議（請務必諮詢真正的律師），而是給工程師一張合規地圖，讓你在設計部署時知道哪些紅線不能踩、哪些設計決策有法律後果。

2026 年是 Proxy 法律環境發生重大變化的年份 — EU AI Act 將於 2026 年 8 月 2 日進入「高風險 AI 系統」執法階段，罰款上限 €35M 或全球營業額 7%，連 Google、Microsoft 這個量級都會痛。GDPR 累計裁罰已達 €5.88 billion（截至 2025 年底，2,245 起裁罰）5。資安監控、員工隱私、AI Agent 自動操作的責任歸屬，這些議題在 2026 年都從「未來問題」變成了「當下問題」。

7.1 — 法律框架的多層結構

要理解 Proxy 的法律問題，先要看清楚它涉及的多層法律框架：

層次 關鍵法規（典型） 對 Proxy 的影響
個人資料保護 GDPR（EU）、CCPA（加州）、PDPA（台灣 / 新加坡）、PIPL（中國） Proxy log 即「個人資料處理」
AI 監管 EU AI Act（2026 年高風險執法） AI Gateway、MCP Server 列入監管
電信 / 通訊 各國電信法、ePrivacy Directive（EU） SSL Bump、流量解密的合法性
產業合規 HIPAA（美醫療）、PCI DSS（信用卡）、SOX（上市財報） Proxy 必須滿足特定加密、log 保留要求
抗審查 / 出口管制 各國網路法、Wassenaar Arrangement 翻牆協定、密碼學產品的出口
勞動法 / 隱私期待 各國勞基法、GDPR Recital 47 企業是否能監控員工流量

同一個 Proxy 部署可能同時觸發多層 — 一個跨國企業在歐洲員工的筆電上部署 SSL Bump，會同時涉及 GDPR（員工個資）、ePrivacy Directive（通訊保密）、勞動法（隱私期待）、可能還有 AI Act（如果該 Proxy 含 ML 模型做內容分類）。合規不是「裝個 SSL termination 就好」，是一個系統工程。

7.2 — GDPR 對 Proxy 的具體要求

GDPR 把所有處理「歐盟居民個人資料」的活動都納管，且具有域外效力 — 你公司不在歐洲，但你的服務有歐洲用戶，GDPR 仍適用。對 Proxy 而言，幾個關鍵點：

IP 位址即個人資料

GDPR Recital 30 明確認定 IP 位址（即使是動態 IP）為個人資料。這意味著任何 Proxy 的 access log 都是「個人資料處理」，需要：

  • 合法基礎（Article 6）：合約必要、合法利益、明確同意三選一
  • 目的限制（Article 5）：只能用於記錄當時宣告的用途
  • 儲存期限（Article 5）：不能無限期保留，要設定明確 retention policy
  • 當事人權利（Articles 15–22）：使用者有權查詢、刪除、攜出資料

實務影響：Nginx 的 access log 預設一個月（甚至更長）才 rotate，這在 GDPR 下其實很危險。多數歐洲合規部署會把 Proxy log 縮短至 7–30 天，且要明確記錄「為什麼留這麼久」。

Cross-border data transfer

當 Proxy 把流量轉送到歐盟外的服務時，等於做了「跨境資料傳輸」。GDPR 對此有嚴格要求（Standard Contractual Clauses、Adequacy Decisions、Binding Corporate Rules 等）。Cloudflare 的 Geo Steering、AWS 的 ALB region routing 都是 GDPR 合規的一部分 — 不是純粹的效能優化。

Right to be Forgotten 的技術挑戰

GDPR Article 17（被遺忘權）要求業者在使用者要求時刪除其資料。但 Proxy 與 CDN 環境下，資料可能散佈在數百個邊界節點 — 每個都要刪除嗎？這是 Cloudflare、AWS 等 CDN 業者長期面對的挑戰，目前主流做法是「中央化使用者資料管理 + 邊界節點僅保留匿名 metric」。

7.3 — EU AI Act：2026 年的新戰場

2024 年 8 月生效、2026 年 8 月 2 日進入「高風險系統執法」階段的 EU AI Act，是全球第一個全面 AI 監管法律。對 Proxy 領域影響最大的是 Part V 涵蓋的 AI Gateway、MCP Server、LLM Gateway。

風險分級制度

  • Unacceptable Risk（禁用）：社會評分、執法用即時生物辨識、情緒辨識用於職場 / 學校
  • High Risk（高度監管，2026/8 執法）：Annex III 列舉的就業、信貸、教育、執法等情境
  • Limited Risk（透明度要求）：聊天機器人需告知使用者「這是 AI」
  • Minimal Risk（無特別規範）

AI Gateway 必須面對的義務

當你部署一個 AI Gateway 路由到 LLM，且該 LLM 的輸出影響「就業決策、信貸決策、教育評估」等高風險場景，整個 Pipeline（包括你的 Gateway）會被視為高風險 AI 系統的一部分，需要滿足：

  • Article 9 風險管理系統 — 持續監控與緩解
  • Article 10 資料治理 — 訓練資料品質、偏見控制
  • Article 11 技術文件 — 完整可審計
  • Article 12 紀錄保留 — 自動化日誌、可追溯
  • Article 13 透明度與資訊提供
  • Article 14 人為監督
  • Article 15 準確性、強健性、資安

罰款結構（Article 99）

  • 禁用 AI 違規：上限 €35M 或全球營業額 7%（取較高者）
  • 高風險不合規：上限 €15M 或 3%
  • 提供誤導資訊給監管者：上限 €7.5M 或 1%

對比 GDPR 的 €20M / 4%，AI Act 的處罰更嚴。對 Meta 而言 7% 的全球營收約是 $8.5B；Google 約 $14B；Microsoft 約 $16B6。這不是「罰款」，是「年度業績風險」。

AI Act × GDPR：雙重義務

高風險 AI 系統如果處理個資，需同時做兩種影響評估：

  • DPIA（Data Protection Impact Assessment，GDPR Article 35）
  • FRIA（Fundamental Rights Impact Assessment，AI Act Article 27）

兩者方法論重疊但範圍不同 — 不能用 DPIA 取代 FRIA，反之亦然。EDPB（歐洲資料保護委員會）與 AI Office 在 2025–2026 年陸續發布指引釐清兩者交互，但這個雙重義務在實務上仍是合規部門的頭痛源。

DPA 的新權力

AI Act 賦予資料保護機關（DPA）超越 GDPR 的調查權 — 在某些情況下，DPA 可以要求業者揭露 AI 系統的訓練資料、技術文件、甚至原始碼（AI Act Article 77(2–3)）。這對「閉源 LLM」的部署有深遠影響：你可能無法引用 OpenAI 的訓練資料來抗辯。

2026 年的時鐘

本書出版時，距離 2026 年 8 月 2 日 EU AI Act 高風險執法只剩 4 個月。如果你的組織部署了任何 AI Gateway / LLM Gateway / MCP Server 路由到歐盟，現在就應該開始評估你的系統是否落入高風險範疇，而不是等到 8 月才動手。等到收到 DPA 的調查信時，技術文件、log 系統、人為監督機制都已經來不及補。

7.4 — SSL Bump 的法律灰區

SSL Bump（HTTPS 解密代理）是企業 DLP（資料外洩防護）與內容過濾的標準技術，但它在法律上極為敏感。Squid 官方文件直接寫了一段警告：

Some countries do not limit what can be done within the home environment.
Some countries permit employment or contract law to overrule privacy.
Some countries require government registration for all decryption services.
Some countries it is an outright capital offence with severe penalties.
DO Seek legal advice before using this configuration, even at home. Squid Wiki, "Intercept HTTPS CONNECT messages with SSL-Bump"

注意最後一句的強度 — 即使在家中使用 SSL Bump，部分國家也可能違法。為什麼這麼極端？因為 SSL Bump 在技術上等同於對受保護的通訊進行 Man-in-the-Middle 解密，這在多數司法管轄區屬於「攔截通訊」的範疇，需要法律授權。

Independent Security Evaluators 的三條件

業界廣泛接受的合法 SSL Bump 三條件：

  1. 使用內部 CA（不接到公開 root CA）— 維持公開 PKI 系統的完整性
  2. 僅限公司擁有 / 控制（含 BYOD）的設備 — 不對員工個人裝置動手
  3. 使用者明確同意（不是 EULA 一行字埋深處）並真正理解其意涵

三條件之外的部署 — 例如在公開 WiFi 強制 MITM、對訪客流量解密、向員工偷偷部署而未告知 — 都在法律上極為危險。多數歐洲國家（特別是德國、法國）對「員工監控」有更嚴格保護，連「公司設備上的個人通訊」都受 ePrivacy Directive 保障。

2025 年的 Trustwave 事件

歷史教訓：Trustwave（公開 root CA）曾於 2012 年發出可被用於 SSL Bump 的子 CA 給某客戶部署 DLP。這導致 Mozilla、Microsoft 幾乎要把 Trustwave 從信任列表中除名（業界稱「死刑威脅」）。最後 Trustwave 撤回該子 CA、同意永不再簽發此類憑證才解圍。這個案例確立了一個鐵則：用於 SSL Bump 的 CA 絕不可接到公開信任鏈。

敏感網站例外

連同意做 SSL Bump 的合規部署，通常也會把以下類型網站列入 bypass list（不解密）：

  • 網路銀行 / 支付服務
  • 醫療、心理健康相關網站
  • 政府服務網站
  • 勞工權益、工會相關網站（重要！監視這類流量可能本身違法）
  • 使用者個人銀行、保險、律師通訊

這個 bypass list 不只是禮貌 — 在許多司法管轄區，攔截律師-客戶通訊或勞工-工會通訊本身就是獨立的犯罪。

7.5 — 抗審查協定的法律處境

本書 Part III Ch.22–24 會詳細介紹 Shadowsocks、V2Ray、Trojan、Reality 等抗審查協定。從本章的法律視角看，需先建立幾個重要區分：

「使用」與「提供」的法律差異

  • 使用者：在多數國家，個人用 VPN / Proxy 翻牆訪問外部網站不違反該國法律；但被翻牆的目的地國（如中國、伊朗）通常認為違法。
  • 提供者：在中國等審查國家，架設並提供翻牆服務有刑事責任。在自由國家，提供 Tor exit node 可能因為他人犯罪行為被誤指控（雖然法律上多次被 EFF 等組織辯護成功）。
  • 商業營運：在中國，未持牌經營「VPN 業務」屬於《刑法》第 285 條範疇；在俄羅斯，2017 年起未在政府名單上的 VPN 視為違法。

出口管制與密碼學

密碼學在歷史上是出口管制商品（美國 1996 年前甚至列為「軍火」）。雖然今日多數民用密碼學已自由出口，但Wassenaar Arrangement 仍把「侵入軟體」列管。這對「滲透測試代理」、「rootkit-style 內網代理」有直接影響。一般 Proxy 軟體不受影響，但你開發的特定產品如果被歸類為「intrusion software」，可能需要出口許可。

技術中立 vs 結果責任

本書討論翻牆協定純粹從技術角度出發 — 同一個 Reality 協定可以用於合法的隱私保護、也可以用於規避法律。本書的立場是技術中立，但讀者應理解：在你的司法管轄區下，使用、開發、營運這些技術可能有不同法律後果。

7.6 — 員工監控與隱私期待

「公司設備上的員工是否還有隱私期待？」這個問題沒有單一答案，各國法律差異極大：

地區 主流立場
美國 雇主權力較大，可監控公司設備上幾乎所有活動（前提是事先告知）
德國 嚴格保護員工隱私；連公司設備上的私人通訊也受 ePrivacy 保障；通常需要 works council 同意
法國 類似德國，強保護；CNIL 對員工監控有詳細指引
英國 須符合 Information Commissioner's Office (ICO) 指引、需「合理期待」分析
加拿大 「Reasonable expectation of privacy」原則；需明確告知並縮小範圍
台灣 個資法保護員工；監控需有法律基礎並明確告知
日本 個人情報保護法保障員工資料；雇主監控要有合理目的
中國 雇主權力較大；《個人信息保護法》對員工監控仍要求明確同意

一個跨國企業在不同國家分公司部署同一套 DLP Proxy，合規設定可能完全不同。這就是為何大型企業的 IT 部門需要 privacy counsel 一起設計 — 不是技術人員自己決定。

7.7 — AI Agent 自動操作的責任歸屬

2025–2026 年新興的法律議題：當 MCP Server 讓 AI Agent 代表使用者呼叫外部 API（轉帳、訂購、發訊息），誰負責結果？這是傳統 Proxy 沒有的問題，因為傳統 Proxy 只是被動轉發；但 AI Gateway / MCP Server 是主動代表使用者做決策。

幾個未解的法律問題：

  • AI Agent 用 MCP 自動發送 email 詐騙了第三方 — 使用者、AI 提供者、MCP Server 開發者、API 提供者，誰負責？
  • AI Agent 透過 MCP 自動執行交易，金額判斷錯誤導致用戶巨額虧損 — 誰賠償？
  • AI Agent 的工具調用紀錄是否屬於「個人資料」？儲存在哪裡？
  • 當 AI Agent 跨境呼叫 API 時，跨境資料傳輸的責任在誰？

2026 年的法律共識正在形成中，但EU AI Act Article 14（人為監督）為這類問題提供了起點 — 高風險 AI 系統必須有「有意義的人為監督」，意味著完全自動化的 AI Agent 在高風險場景需要重新設計（例如，財務交易必須有人類確認步驟）。Anthropic 與 Linux Foundation AAIF 在 2025 年底發布的 MCP 治理白皮書中，將「使用者授權層級」列為核心設計原則之一。Part V Ch.45 會深入討論這個議題的技術實作。

7.8 — 工程師的合規 checklist

本章不能取代法律專業諮詢，但可以提供一個工程設計時的初步 checklist。當你設計或部署一個 Proxy 系統時，至少問這些問題：

  • ☐ 我的 access log 處理個人資料嗎？保留多久？合法基礎是什麼？
  • ☐ 我的 Proxy 跨境傳輸資料嗎？符合適用的 cross-border transfer 機制嗎？
  • ☐ 我有 SSL Bump 嗎？是否符合三條件（內部 CA、公司設備、明確同意）？
  • ☐ 我的 bypass list 是否排除敏感網站（金融、醫療、勞工權益）？
  • ☐ 我的部署涵蓋多國時，是否有為各國法律差異做不同設定？
  • ☐ 如果我的 Proxy 涉及 AI 路由或處理，是否評估過 AI Act 高風險範疇？
  • ☐ 我有 retention policy 文件嗎？刪除流程是否實際可執行？
  • ☐ 我有 incident response plan 嗎？資料外洩 72 小時內如何通報？
  • ☐ 我的供應商（CDN、雲端）是否有 DPA、SCC 等合規文件？
  • ☐ 我是否與法務、隱私團隊一起設計，而非只在 launch 前找他們蓋章？

這個 checklist 的每一項，背後都是真實案例 — 都是某個公司因為沒做而吃了重罰、或被監管機關調查。

Proxy 技術是中性的，但部署是有法律後果的。GDPR 把 IP 視為個資；EU AI Act 將於 2026/8/2 啟動高風險執法（罰款上限 €35M / 7%）；SSL Bump 在多國法律敏感；員工監控的合法性各國差異巨大；AI Agent 的責任歸屬尚未定論。合規不是 launch 前的核對，是設計階段的核心需求。

Proxy 的歷史演進
從 1991 年 CERN httpd 到 2026 年 MCP 時代

Proxy 的歷史就是網際網路的歷史。從第一台 web server 加上 cache 功能開始，三十五年的演進路上每個轉折都映射著時代的技術需求。

Tim Berners-Lee 於 1991 年 6 月在 NeXT 電腦上釋出 CERN httpd 0.1 版時，沒人預見三十五年後會有一本書叫《Proxy 完全教科書》。當時的 CERN httpd 只是一個簡單的 web server，連 cache 功能都還沒有。但短短三年後，CERN 的工程師 Ari Luotonen 與 Intel 的 Kevin Altis 共同發表了「World-Wide Web Proxies」論文（1994 年 5 月，第一屆 WWW 會議於日內瓦），這篇論文奠定了現代 Proxy 的所有核心概念 — gateway 模式、caching、protocol gateway、本機 cache 工作模式（不連網也能 demo）7。

本章用十個關鍵時刻串起這段歷史。每個時刻都不只是技術事件，而是「某個時代的網際網路需要這個 Proxy 形態才能繼續演化」的證據。

8.1 — 1991：CERN httpd 與 Proxy 的史前時代

1991 年 6 月，Tim Berners-Lee 在 NeXT 電腦上釋出 CERN httpd v0.1，世界第一個 web server 誕生。當時還沒有「Proxy」的概念 — server 直接服務 client，HTTP 1.0 都還沒出現。

1993 年 4 月 30 日，CERN 把 httpd 與整個 Web 技術置入公領域（public domain），這是網際網路歷史上極關鍵的一步：放棄專利讓 Web 自由發展。如果 CERN 選擇授權收費，今天很可能不會有開放的網際網路。

8.2 — 1994：第一篇 Proxy 論文

1994 年 5 月，Luotonen 與 Altis 在第一屆 WWW 會議發表「World-Wide Web Proxies」。這篇論文的目錄今日讀來仍然驚人地現代：

  • What is a Proxy？（中介概念）
  • Why use a Proxy？（防火牆穿越、頻寬節省）
  • Caching at the Proxy（cache hit rate、LM factor）
  • Application-level vs Transport-level（已預示 L4 vs L7）
  • Authentication issues

論文裡有一段話放到 2026 年仍然成立："The proxy can be configured to use only the local cache, making it possible to give demos without an internet connection."— 這是 1994 年的「offline-first」設計哲學，遠早於 PWA、Service Worker。

同年，CERN httpd 加入完整的 caching 功能（v3.0，1994 年 11 月）。LM factor（Last-Modified factor）是當時的快取啟發式演算法 — 「文件上次修改在 X 天前，預期它在 X × 0.1 天內不會變」。這個演算法在今日 Cloudflare、Varnish、Squid 仍能找到痕跡。

8.3 — 1995–1996：協定家族的奠基

1995 年是 Proxy 協定家族奠基的一年：

  • SOCKS v4（1992 年由 David Koblas 設計，1996 年 RFC 1928 標準化為 v5）— 第一個獨立於 HTTP 的通用代理協定
  • Squid（1996 年從 Harvest Cache 計畫分支出來）— 取代 CERN httpd 成為 Forward Proxy 的事實標準
  • HTTP CONNECT 方法（1996 年 Ari Luotonen 草案）— 讓 HTTPS 能透過 HTTP Proxy 隧道

同年 7 月 15 日，CERN 釋出 httpd v3.0A，這是最終版本。W3C 把後續開發轉移到 Java 寫的 Jigsaw 計畫（這個轉換失敗了 — Apache、Squid 才是真正繼承者）。CERN httpd 服役了大約 5 年，足以見證 Web 從學術實驗到主流的轉變。

8.4 — 2000–2004：高效能架構的轉折

2000 年代初，網際網路爆發 — Yahoo!、Google 開始處理百萬級請求量。前一個世代的 thread-per-connection 模型不足以應付（cf. Ch.6 的 C10K 討論）。這時兩個關鍵 Proxy 出現：

  • HAProxy（2001，由 Willy Tarreau 開發）— 全 event-driven、極致追求 connection-per-second 的 Reverse Proxy
  • Nginx（2002 年 Igor Sysoev 開始開發，2004 年首次公開釋出）— 為俄羅斯 Rambler 入口網站設計的高效能 web server + Reverse Proxy

這兩個產品至今仍是市佔第一第二的 open-source Reverse Proxy。它們的成功來自一個觀念轉移：Proxy 不只是「中介」，而是「網路架構的核心元件」。沒有它們，現代雲端不可能存在。

8.5 — 2008–2012：CDN 與 Edge 的崛起

2008 年是雲端 Proxy 的轉折年：

  • Cloudflare（2009 年成立，2010 年公開）— CDN + DDoS 防護 + 後續無數產品的起點
  • Amazon CloudFront（2008 年 11 月推出）— AWS 開始進入 CDN
  • Fastly（2011 年成立）— 強調可程式化的邊界 Proxy

這個時期的關鍵概念是 Edge — Proxy 不再是企業內部的工具，而是部署在全球邊界，靠近終端使用者。CDN 公司開始用「我們有多少 PoP（Point of Presence）」當競爭優勢。地理是一個維度。

8.6 — 2014–2016：Microservices 與 Service Mesh

2014 年前後，業界開始大規模採用 microservices 架構。一個應用拆成數十甚至上百個 service，意味著「服務間通訊」成為新的問題。這催生了：

  • Linkerd（2016，Buoyant 開發）— 第一個被稱為「Service Mesh」的產品
  • Envoy（2016 年 9 月，Lyft 開源）— 為 microservices 設計的全功能 Proxy
  • Istio（2017 年 5 月，Google + IBM + Lyft）— 把 Envoy 包裝成完整 Service Mesh 平台

Service Mesh 是 Proxy 概念的範式級創新：每個 service 旁邊都有一個 sidecar Proxy，所有 service-to-service 流量都經過這層代理。這帶來統一的 observability、加密、限流，但也帶來「每個請求多兩跳代理」的成本（這就是後來 Ambient Mesh 試圖解決的問題）。

8.7 — 2017–2022：eBPF 與 kernel 的回歸

2017 年 Cilium 計畫（Linux Foundation）開始把 eBPF 應用到 Kubernetes 網路。從歷史長期看，這是一次有趣的鐘擺 — 從 1990 年代「Proxy 在 user space」到 2010 年代「Proxy 在 user space 但極致優化」到 2020 年代「Proxy 邏輯回到 kernel」。

2022 年 QUIC（HTTP/3 的傳輸層）成為 IETF RFC 9000，徹底重新洗牌了傳輸層的設計。HTTP/3 把 TLS 整合進傳輸層、用 UDP 取代 TCP — 這對 Proxy 的影響極大（Ch.12 會詳述），意味著 Reverse Proxy 必須學會處理一個全新的協定堆疊。

8.8 — 2023–2024：API Gateway 的成熟與 LLM 興起

2023 年是 LLM 驅動 Proxy 演進的開始。隨著 ChatGPT 普及，API Gateway 開始有了新需求 — 不只是 REST API，還有 LLM streaming、token 計費、prompt injection 防禦：

  • Kong AI Gateway（2024）— 在傳統 API Gateway 基礎上加入 LLM 路由
  • LiteLLM（2023）— 開源 LLM Proxy，統一不同廠商 API
  • Cloudflare AI Gateway（2024）— 邊界級的 AI Proxy
  • Portkey、Helicone等新興 AI Gateway

同時，Service Mesh 也在演進。Istio Ambient Mesh 在 2024 年達到 production ready（2025 年正式 GA），把 sidecar 模式改為「ztunnel + waypoint proxy」兩層架構，大幅降低資源消耗。Cilium 1.16+（2024 年）開始支援 Service Mesh 的 sidecar-less 模式。

8.9 — 2024–2025：MCP 改寫 AI Agent 的代理範式

2024 年 11 月，Anthropic 推出 Model Context Protocol（MCP）。它不是「另一個 API Gateway」，而是讓 AI Agent 能代表使用者呼叫外部工具的新型代理協定。MCP 的核心觀念是「Server」— 但這個 Server 不服務人類，而是服務 AI。

MCP 的採用速度堪比早年 Docker、Kubernetes：

2025 年 12 月的治理轉移是關鍵時刻 — Anthropic、Block、OpenAI 共同發起 AAIF（Agentic AI Foundation），把 MCP 從「Anthropic 的協定」變成「跨廠商標準」。這個治理結構的設計刻意參考了 CNCF（Cloud Native Computing Foundation）對 Kubernetes 的處理方式 — 創始公司放棄獨佔，換取生態系廣泛採納。

8.10 — 2026：AI 時代的 Proxy 全景

2026 年的 Proxy 生態，從 1991 年回望，可說是多層共存：

  • L3/L4 層：傳統 NAT、CGNAT、L4 LB 仍服務全球流量；Cilium eBPF 漸成主流
  • L7 層：Nginx、HAProxy、Envoy 三足鼎立；HTTP/3 全面普及
  • 邊界層：Cloudflare、Fastly、Akamai 處理全球 80% 以上流量
  • Service Mesh：Istio Ambient、Cilium、Linkerd 並存；sidecar 與 ambient 模式各有市場
  • API Gateway：Kong、Apigee、AWS API Gateway 主導
  • AI 代理層：MCP Server、AI Gateway、LLM Gateway — 2024 年才出現的新層級

每一層都沒有取代前一層 — 它們疊加共存。一個現代企業的網路堆疊，可能同時包含 1990 年代的 NAT、2000 年代的 Nginx、2010 年代的 Envoy、2020 年代的 Cilium、2024 年才出現的 MCP。三十五年的代理史，全部活在同一個資料中心裡。

8.11 — 觀察：Proxy 演化的三條軸線

從這段歷史可以看出 Proxy 演進的三條軸線：

軸線一：層次的下沉與上升

1990s 的 Proxy 在 user space；2010s 一些回到 kernel（eBPF）；2024+ 又回到應用層之上的「業務語意層」（MCP、AI Gateway）。這不是簡單的「上下移動」，而是每個年代都新增一個更高的抽象層，舊層仍在運作。

軸線二：「為誰代理」的延伸

最初代理 user 連線到 web → 代理 web server 給 user → 代理 service 對 service → 代理 AI Agent 對 API。「主體」從人變成程式變成 AI。「代理」這個概念的範圍持續擴大。

軸線三：自由與控制的拉鋸

Proxy 既能解放（Tor、翻牆、隱私保護），也能控制（CGNAT、企業 DLP、國家審查）。同樣的技術在不同手中、不同司法管轄區下，意義完全不同。這個張力從未消失，而是隨著技術演進不斷以新形式出現。2026 年的 EU AI Act、MCP 的權限模型、AI Agent 的責任歸屬，都是這個張力在 AI 時代的延續。

本章要點

Proxy 的歷史是從 1991 年 CERN httpd 到 2026 年 MCP 的三十五年連續演進，每個時代都在前一代基礎上新增層次。三條軸線貫穿這段歷史：層次的疊加、代理主體的延伸、自由與控制的拉鋸。理解歷史不是懷舊，是看清未來方向 — 下一個 Proxy 範式很可能在你閱讀這段話時正在某個 Anthropic、OpenAI 或新創公司的代碼庫中孕育。
