Chapter 09

SOCKS 協定家族
從 1992 年的 byte stream 到現代 byte-level 完整解析

SOCKS 是最古老、最純粹的代理協定之一。讀完本章你能徒手解碼 Wireshark 抓到的每一個 SOCKS5 握手 byte，並用 60 行 Python 寫出可運行的 SOCKS5 server。

在所有現代代理協定中，SOCKS 有一個獨特的位置 — 它是第一個將「代理」標準化為獨立協定的設計。1992 年，MIPS Computer Systems 的系統管理員 David Koblas 在 Usenix Security Symposium III 發表 SOCKS 論文，把「協助 client 穿越防火牆連到外部 server」這個功能從各家防火牆廠商的私有實作中抽出來，變成一個通用協定。這篇論文的存在本身就是 Proxy 史的轉折點：在那之前，「代理」是各種應用層協定（HTTP、FTP、SMTP）裡各自實作的功能；在那之後，「代理」成為一個獨立的概念，與承載協定無關。

四年後，1996 年 3 月，IETF 將 SOCKS5 標準化為 RFC 1928（作者群包括 Bell-Northern Research、IBM、NEC、Unify、Hewlett-Packard，與當時已成獨立顧問的 Koblas）。RFC 1929（用戶名/密碼認證）同月發布。這兩份規格在過去三十年幾乎沒有更動 — 不是因為它們完美，而是因為簡單的東西不需要修。SOCKS5 至今仍是所有抗審查工具（Tor、Shadowsocks、V2Ray、Trojan、Reality）的基礎協定，每天承載全球數十億的代理流量。

9.1 — 為何 SOCKS 至今仍重要

SOCKS5 在 2026 年仍佔據關鍵位置，有幾個原因：

  • 協定無關：SOCKS5 工作在 L5（會話層），不解析應用層內容。所以任何 TCP 或 UDP 流量都能透過 SOCKS5 代理 — HTTP、SSH、SMTP、BitTorrent、自訂協定都行。
  • 不改變指紋：因為 SOCKS5 不解密、不解析應用層，client 的 TLS ClientHello、HTTP/2 SETTINGS frame、JA3/JA4 指紋會原封不動傳到目標 server。這在需要保持 client 真實指紋的場景（爬蟲、API 整合、隱私保護）至關重要。
  • 實作簡單：完整的 SOCKS5 server 實作只需要約 200 行 C 或 60 行 Python。低門檻使它成為各種代理工具的事實標準。
  • 支援 UDP：UDP ASSOCIATE 命令（CMD=0x03）讓 SOCKS5 能代理 UDP 流量，這是 HTTP Proxy 做不到的。雖然實際使用率低，但對 DNS over UDP、QUIC 等場景有意義。

反過來說，SOCKS5 的侷限也明顯：它不能做 URL 級的決策（看不到 HTTP path）、不能做 cache、不能做認證後的細粒度授權、用戶名密碼明文傳輸（RFC 1929 §3 明確警告）。所以 SOCKS5 通常是「底層通道」 — 上層往往再疊加 HTTP Proxy、TLS、或專屬協定（如 Shadowsocks 在 SOCKS5 之上加密混淆）。

9.2 — 完整握手序列

一個典型的 SOCKS5 連線從 client 與 SOCKS server 建立 TCP 連線（預設 port 1080）開始，然後進行三階段握手：

  1. 方法協商（Method Negotiation）— client 提出支援的認證方法，server 選一個
  2. 方法子協商（Sub-negotiation）— 依選定方法執行（如用戶名/密碼驗證）
  3. 請求（Request）— client 告訴 server 要連到哪裡，server 嘗試連線並回報結果

之後就進入純資料中繼模式 — SOCKS5 server 在 client 與目標 server 之間雙向轉送 byte，直到一方關閉連線。中繼階段沒有任何 SOCKS5 自己的訊息，這就是為什麼 SOCKS5 是「協定無關」的純粹形態。

9.3 — 階段一：方法協商的 byte-level 解構

Client 的第一個訊息（version identifier / method selection message）格式如下：

VER 1 byte 0x05
NMETHODS 1 byte 1–255
METHODS NMETHODS bytes 每方法 1 byte

圖 9.2　Greeting 訊息格式。

這是個典型的 TLV 結構（Tag-Length-Value）：版本號是 tag，方法數量是 length，方法清單是 value。RFC 1928 定義的方法值：

METHOD意義規格
0x00NO AUTHENTICATION REQUIRED（無認證）RFC 1928
0x01GSSAPI（Generic Security Service API）RFC 1961，少用
0x02USERNAME/PASSWORDRFC 1929
0x03–0x7FIANA 分配—
0x80–0xFERESERVED FOR PRIVATE METHODS各家私有擴充
0xFFNO ACCEPTABLE METHODSserver 拒絕用，僅出現於回應

幾個常見的 client 訊息：

05 01 00 # client 提議 1 個方法: 無認證 05 01 02 # client 提議 1 個方法: 用戶名/密碼 05 02 00 02 # client 提議 2 個方法: 無認證 或 用戶名/密碼，由 server 選 05 03 00 02 80 # 提議 3 個方法: 無認證 / 用戶名密碼 / 0x80 自訂方法

Server 的回應永遠是 2 bytes：

05 00 # server 選擇: 無認證 (0x00) 05 02 # server 選擇: 用戶名/密碼 (0x02) 05 FF # server 拒絕: 沒有可接受的方法

如果 server 回傳 0x05 0xFF，client 必須立即關閉連線 — 不能再繼續送任何 byte。這個簡單規則在實作時容易忽略，但對防止錯誤狀態下的資料洩漏很重要。

9.4 — 階段二：用戶名/密碼子協商

如果 server 選擇 0x02，client 必須依 RFC 1929 送出認證訊息。這個子協定有自己的版本號 0x01（注意：不是 SOCKS5 的 0x05）：

VER 1 byte 0x01
ULEN 1 byte 1–255
UNAME ULEN bytes 用戶名
PLEN 1 byte 1–255
PASSWD PLEN bytes 密碼

例如送出用戶名 "alice" 與密碼 "s3cret"：

01 05 61 6c 69 63 65 06 73 33 63 72 65 74 │ │ └────"alice"────┘ │ └─────"s3cret"─────┘ │ └─ ULEN=5 └─ PLEN=6 └─ VER=0x01（RFC 1929 子協定版本）

Server 回應 2 bytes：0x01 STATUS，其中 STATUS=0x00 為成功，非 0 為失敗。

安全警告 · 來自 RFC 1929 §3

「In the username/password method since the request carries the password in cleartext, this subnegotiation is not recommended for environments where 'sniffing' is possible and practical.」

用戶名密碼以明文傳輸 — 這是 SOCKS5 認證的根本弱點。實務上有兩種解法：(1) 在不可信網路只用 SOCKS5 over TLS（如把 SOCKS5 包在 SSH tunnel 或 stunnel 內）、(2) 改用更強的認證機制（GSSAPI 0x01 或私有方法 0x80–0xFE）。Shadowsocks 的設計動機之一就是補上這個洞 — 它在 SOCKS5 之上加了對稱加密與混淆。

9.5 — 階段三：請求與回應

子協商完成後，client 送出真正的請求 — 「我想連到哪裡，做什麼」：

VER 1 byte 0x05
CMD 1 byte 命令
RSV 1 byte 0x00
ATYP 1 byte 位址類型
DST.ADDR 變動 目的位址
DST.PORT 2 bytes big-endian

三個 CMD 值定義 SOCKS5 的所有操作模式：

CMD意義使用情境
0x01CONNECT標準的「我想連到 X」 — 99% 的流量
0x02BIND讓 server 為 client 開一個 listening port，等 inbound 連線。設計初衷是支援 FTP active mode，現代極少使用
0x03UDP ASSOCIATE建立 UDP 中繼通道。control 連線仍用 TCP，但會額外開 UDP relay port

四個 ATYP 值決定 DST.ADDR 欄位的格式：

ATYP意義DST.ADDR 格式
0x01IPv4 address4 bytes（網路位元組順序）
0x03DOMAINNAME1 byte 長度 + 域名（最長 255）
0x04IPv6 address16 bytes

實例 — CONNECT 到 example.com:443（域名）：

05 01 00 03 0b 65 78 61 6d 70 6c 65 2e 63 6f 6d 01 bb │ │ │ │ │ └────────"example.com"────────┘ └─port─┘ │ │ │ │ └─ DST.ADDR 長度=11 │ │ │ └─ ATYP=0x03 (域名) │ │ └─ RSV=0x00（保留欄位，必為 0） │ └─ CMD=0x01 (CONNECT) └─ VER=0x05 port: 0x01bb = 443 (big-endian / network byte order)

CONNECT 到 192.0.2.1:80（IPv4）：

Server 收到請求後嘗試連到目標。無論成功或失敗，回應格式與請求相同（只是 CMD 欄位改名為 REP）：

REP意義
0x00succeeded
0x01general SOCKS server failure
0x02connection not allowed by ruleset
0x03Network unreachable
0x04Host unreachable
0x05Connection refused
0x06TTL expired
0x07Command not supported
0x08Address type not supported

BND.ADDR 與 BND.PORT 是 server 用來連到目標的本地端點。對 CONNECT 而言這個欄位通常被 client 忽略；但對 BIND 與 UDP ASSOCIATE 至關重要 — client 需要這個位址才能建立後續連線（BIND 的 inbound 連線、UDP 的 relay port）。

9.6 — UDP ASSOCIATE：被遺忘的 SOCKS5 特性

SOCKS5 相對 SOCKS4 的最大新增功能就是 UDP 支援。但因為實作複雜（需要 control TCP + relay UDP 兩條通道）、瀏覽器不支援、且實際 UDP 應用場景相對少，UDP ASSOCIATE 在 2026 年仍然是「規格存在但少用」。

運作模式如下：

  1. Client 用 TCP 連到 SOCKS5 server，做完方法協商，送 CMD=0x03（UDP ASSOCIATE）
  2. Server 回應 BND.ADDR 與 BND.PORT — 這是 UDP relay 端點
  3. Client 把要送的 UDP datagram 包進 SOCKS5 UDP request header（含 RSV、FRAG、ATYP、DST.ADDR、DST.PORT 與 DATA），送到 BND.ADDR:BND.PORT
  4. Server 解封裝，把 DATA 部分送到真正的目標
  5. 反向類似
  6. 關鍵：原本的 TCP control 連線必須一直保持。如果 TCP 斷了，UDP relay 立即終止

UDP relay 的最大實用場景是 DNS — 在 SOCKS5 環境下，tun2socks 等工具用 UDP ASSOCIATE 把整台機器的 UDP 流量（特別是 DNS）導入代理。但這條路徑長期有 bug 與相容性問題；許多代理工具乾脆只代理 TCP，DNS 用其他機制（DoT/DoH/或直接代理到 53/tcp）解決。

9.7 — socks5 vs socks5h：DNS 解析的細微差異

在 client 端設定 SOCKS5 時，URL scheme 有兩個變體：

  • socks5://：DNS 解析在 client 本機進行 — client 先解析域名為 IP，再用 ATYP=0x01（IPv4）送請求
  • socks5h://：DNS 解析委由 SOCKS5 server — client 用 ATYP=0x03（域名）送請求，server 自己解析

這個差異看似微小，但在抗審查場景下至關重要。如果你的本地 DNS 被劫持（被 ISP 或國家審查），用 socks5:// 會洩漏「我想去 example.com」給本地 DNS（且可能解析到錯誤 IP）；用 socks5h:// 則整個 DNS 解析都在牆外的 SOCKS5 server 進行，本地完全看不見目標域名。

常見工具的支援：

  • curl：curl --socks5 host:1080 是本地解析；curl --socks5-hostname host:1080 是遠端解析
  • Firefox：about:config 的 network.proxy.socks_remote_dns 設為 true 啟用遠端解析
  • Chrome：用 --proxy-server=socks5://... 啟動參數預設遠端解析
  • Python requests + PySocks：URL 用 socks5h:// 啟用遠端解析

本書 Ch.14（DNS 與 Proxy）會深入展開 DNS 在代理場景的所有細節。

9.8 — Python 實作：60 行的 SOCKS5 server

下面這個極簡 SOCKS5 server 實作 RFC 1928 的核心功能：無認證、CONNECT 命令、IPv4 與域名 ATYP。雖然簡陋（沒有 IPv6、沒有認證、沒有 BIND/UDP、錯誤處理粗糙），但完整展示了協定的每個 byte。

Listing 9.1 · 極簡 SOCKS5 server (Python 3.10+)import socket
import struct
import threading

def handle(client):
    # Phase 1: Method negotiation
    # Read VER + NMETHODS
    header = client.recv(2)
    if len(header) != 2 or header[0] != 0x05:
        client.close(); return
    n_methods = header[1]
    methods = client.recv(n_methods)
    # Pick "no auth" (0x00) if offered, else reject
    if 0x00 in methods:
        client.sendall(bytes([0x05, 0x00]))
    else:
        client.sendall(bytes([0x05, 0xFF])); client.close(); return

    # Phase 2 (skipped: no auth chosen)

    # Phase 3: Request
    # Read VER + CMD + RSV + ATYP
    req = client.recv(4)
    ver, cmd, _, atyp = req[0], req[1], req[2], req[3]
    if ver != 0x05 or cmd != 0x01:  # 0x01 = CONNECT
        client.sendall(bytes([0x05, 0x07, 0, 0x01, 0, 0, 0, 0, 0, 0])); client.close(); return

    # Read DST.ADDR by ATYP
    if atyp == 0x01:  # IPv4
        host = socket.inet_ntoa(client.recv(4))
    elif atyp == 0x03:  # Domain
        dlen = client.recv(1)[0]
        host = client.recv(dlen).decode("ascii")
    else:
        client.close(); return  # IPv6 not supported in this demo

    port = struct.unpack("!H", client.recv(2))[0]

    # Connect to target
    try:
        target = socket.create_connection((host, port), timeout=10)
        # Reply: success
        bnd_addr, bnd_port = target.getsockname()[:2]
        reply = (bytes([0x05, 0x00, 0, 0x01])
                 + socket.inet_aton(bnd_addr)
                 + struct.pack("!H", bnd_port))
        client.sendall(reply)
    except Exception:
        client.sendall(bytes([0x05, 0x05, 0, 0x01, 0, 0, 0, 0, 0, 0])); client.close(); return

    # Phase 4: Pure relay
    def pipe(src, dst):
        try:
            while data := src.recv(8192):
                dst.sendall(data)
        finally:
            src.close(); dst.close()

    threading.Thread(target=pipe, args=(client, target), daemon=True).start()
    threading.Thread(target=pipe, args=(target, client), daemon=True).start()

server = socket.socket()
server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server.bind(("127.0.0.1", 1080))
server.listen(128)
while True:
    c, _ = server.accept()
    threading.Thread(target=handle, args=(c,), daemon=True).start()

清單 9.1　60 行的 SOCKS5 server 完整實作。可以用 curl --socks5 127.0.0.1:1080 https://example.com 測試。

這個實作展示了 SOCKS5 協定的全部精華 — 三階段握手後純粹中繼。Production-ready 的版本還需要：認證、錯誤處理、idle timeout、access control、connection limit、observability、IPv6、UDP ASSOCIATE。Part III Ch.17 會展開生產級實作。

9.9 — SOCKS5 與 TLS 指紋的關鍵性質

本章開頭提到 SOCKS5 「不改變指紋」。這個性質在 2026 年的隱私與爬蟲場景中極為重要，值得單獨展開。

當 client 透過 SOCKS5 連到一個 HTTPS server：

  1. Client 與 SOCKS5 server 建立 TCP 連線並完成 SOCKS5 握手
  2. SOCKS5 server 與目標 HTTPS server 建立 TCP 連線
  3. 從這一刻起，SOCKS5 server 只是一條啞管道 — client 的 TLS ClientHello 一個 byte 不改地透到目標 server

這意味著目標 server 看到的 TLS 指紋（JA3、JA4）完全是 client 的 — 加密套件順序、TLS 擴充順序、版本支援等。同樣地：

  • HTTP/2 SETTINGS frame 的順序與值
  • HTTP/2 frame priority 的特徵
  • 瀏覽器特有的 header 順序（Chrome 的 sec-ch-ua、Firefox 的 Accept 等）
  • HTTP/3 QUIC 的 transport parameters 順序

...全部原封不動透過 SOCKS5。對需要假冒成真實瀏覽器的爬蟲、需要保持指紋一致的 API 整合、想避免被識別為機器流量的隱私使用者，這是 SOCKS5 相對 HTTP Proxy 的關鍵優勢。

反過來說，HTTP Proxy（特別是 Reverse Proxy 與 SSL Termination Proxy）會重組 TLS 與 HTTP/2 — 它在 client 端終止連線，自己重新發起一條到後端，過程中 JA3/JA4 都被換成 Proxy 自己的指紋。這就是為什麼 Cloudflare 看到的所有流量 JA3 指紋都長得像 Cloudflare 的 — 因為 Cloudflare 替你重新產生了 ClientHello。Ch.11 會深入這個議題；Ch.67（紅藍隊章節中的 JA3/JA4 完整探討）會展開所有指紋向量。

9.10 — SOCKS5 在 2026 年的生態位

三十年後，SOCKS5 仍然是抗審查工具的基石。Tor、Shadowsocks、V2Ray、Trojan、Reality、Hysteria 等在 SOCKS5 之上加了不同層的加密與混淆，但底層仍是 SOCKS5 的請求/回應結構。

SOCKS5 的不變本身就是它的價值 — RFC 1928 從 1996 年至今沒改過一個 byte，而它仍然能完美承載 HTTP/3 / QUIC 流量（因為它根本不解析應用層）。這種「向前相容性」是設計簡單協定的天然紅利。

下一章我們會看到 HTTP Proxy 的世界 — 一個更複雜、更有彈性、但也更會洩漏指紋的代理形態。SOCKS5 與 HTTP Proxy 的對比，是理解所有現代代理選型的基礎。

本章要點

SOCKS5（RFC 1928, 1996）是 L5 協定，三階段握手後進入純中繼。Byte-level 你必須記住的訊息：05 01 00（client greeting 提議無認證）、05 00（server 接受）、05 01 00 01/03 [addr] [port]（CONNECT 請求）、05 00 00 01 [bnd_addr] [bnd_port]（成功回應）。SOCKS5 的關鍵性質是「不改變 client 指紋」 — 這在抗審查、爬蟲、隱私場景下至關重要。

Chapter 10

HTTP Proxy 機制完整解構

HTTP Proxy 是最古老也最複雜的代理形態。它在 plain HTTP 與 HTTPS（透過 CONNECT 隧道）下行為截然不同；理解這個雙面性，是看懂 Squid、Nginx、Cloudflare 設定的前提。

如果說 SOCKS5 是「不解析任何東西的純粹中介」，HTTP Proxy 就是「看得懂協定所以能做更多事」的另一個極端。從 1994 年 CERN httpd 加入 caching 起，HTTP Proxy 走過了 cache、URL 過濾、header 改寫、SSL termination、API gateway、AI gateway 等所有形態。它今日依然是 Web 流量的主要代理形態 — 你發出的每個 HTTP 請求，平均會經過 2–4 個 HTTP Proxy。

HTTP Proxy 的複雜性來自一個歷史包袱：HTTP 1990 年代設計時並沒有預想到「加密通訊」。當 HTTPS 在 1995 年出現後，HTTP Proxy 必須學會處理「我看不懂內容」的情況 — 於是 HTTP CONNECT 方法誕生，把原本「應用層代理」退化為「TCP 隧道」。這個雙面性貫穿全章。

10.1 — Plain HTTP Proxy：看得見的代理

當 client 透過 Forward HTTP Proxy 訪問 http://example.com/page（注意：明文 HTTP），請求格式有一個重要差異：

直連 Server 時的請求GET /page HTTP/1.1
Host: example.com
User-Agent: curl/8.5.0
Accept: */*
透過 Forward HTTP Proxy 時的請求GET http://example.com/page HTTP/1.1
Host: example.com
User-Agent: curl/8.5.0
Accept: */*
Proxy-Connection: keep-alive

看出差異了嗎？Request line 從 path-only 變成了 absolute-URI（http://example.com/page 而不是 /page）。這是 HTTP/1.1 規格中專為 proxy 場景設計的 absolute URI form（RFC 7230 §5.3.2）。Proxy 收到後，從 absolute URI 中解出目標 host，連到目標 server，然後送過去的請求才會回到 path-only form。

RFC 7230 §5.3.2 明確要求：

A client MUST send a Host header field in an HTTP/1.1 request, even when the request-target is in absolute-form, since this allows the Host information to be forwarded through ancient HTTP/1.0 proxies that might not have implemented Host. RFC 7230 §5.4

這個「冗餘 Host header」是歷史包袱 — 1996 年的舊 proxy 不認識 absolute-form，只看 path 與 Host。為了向前相容，現代 client 仍然兩個都送。

10.2 — HTTPS 的處理：CONNECT 方法

當目標是 https://example.com 時，問題來了 — proxy 看不懂加密的 TLS 流量，不能像 plain HTTP 那樣解析請求。1996 年 Ari Luotonen 寫了 HTTP CONNECT 方法的 IETF 草案，這個方案至今仍是 HTTPS over Forward Proxy 的標準。

關鍵觀察：CONNECT 方法把 HTTP Proxy 暫時退化為 L4 純轉送。這是 Ch.1 §1.4 提到的「同一個 HTTP Proxy 軟體」處理 plain HTTP 時是 L7、處理 HTTPS 時實際上是 L4 的真實機制。

CONNECT 請求的精確格式：

CONNECT 請求（從 RFC 7231 §4.3.6）CONNECT example.com:443 HTTP/1.1
Host: example.com:443
Proxy-Authorization: Basic dXNlcjpwYXNz

注意幾個特性：

  • Request target 是 authority form（host:port），不是 URI 也不是 path
  • CONNECT 沒有 body，沒有 Content-Length
  • port 必須明確指定（不能省略 :443）
  • 大多數 proxy 預設只允許 CONNECT 到 443（HTTPS）和 80（HTTP），其他 port 需要白名單

Proxy 的成功回應極簡：

CONNECT 成功回應HTTP/1.1 200 Connection Established
Proxy-Agent: Squid/5.7

注意 status text 不是「OK」 — RFC 7231 規定為「Connection Established」。但實務上多數 proxy 用「OK」也能正常運作（瀏覽器只看 status code 200，文字部分忽略）。

如果 proxy 拒絕（例如目標不在白名單），會回 4xx 或 5xx：

HTTP/1.1 403 Forbidden
Proxy-Agent: Squid/5.7
Content-Type: text/html

<html><body>Access denied by corporate policy.</body></html>

Client 收到非 2xx 狀態碼時，必須中斷流程 — 不能繼續送 TLS ClientHello（因為隧道根本沒建立）。

10.3 — Reverse Proxy 的請求形式

Reverse Proxy 與 Forward Proxy 在 HTTP 協定層級有一個重要差異：Reverse Proxy 收到的是「正常的 HTTP 請求」，不是 absolute-form。

原因是 client 不知道 Reverse Proxy 存在 — 對 client 而言，Reverse Proxy 就是 server 本身。所以 client 送的是：

GET /api/users HTTP/1.1
Host: api.example.com
User-Agent: ...

Reverse Proxy 透過 Host header 與本地路由表決定要轉到哪個後端。這就是為什麼 Nginx 設定中 server_name 與 upstream 的關係 — 前者匹配 Host header，後者決定後端目標。

同樣的伺服器軟體（如 Nginx）能同時當 Forward 與 Reverse Proxy，但用法完全不同：

  • Forward：client 設定 proxy_host，HTTP 請求用 absolute-form，Nginx 用 proxy_pass 動態解析
  • Reverse：client 不知道 proxy 存在，HTTP 請求用 origin-form，Nginx 透過 server_name 路由

Squid 主要做 Forward（雖然能設成 reverse mode）；Nginx、HAProxy、Envoy 主要做 Reverse。這個分工符合各自的設計哲學。

10.4 — Via Header：Proxy chain 的足跡

當請求經過多層 Proxy 時，每層都應該在 Via header 加上自己的標記。RFC 7230 §5.7.1 定義：

Via: 1.1 proxy1.example.com, 1.1 cdn.cloudflare.net, 2.0 backend-lb

每個項目格式：HTTP-version received-by-name。你能從 Via header 看出：

  • 請求經過了 3 層 proxy
  • 第一層用 HTTP/1.1 接到 client，第三層用 HTTP/2 接到下一層（HTTP/2 的 ALPN ID 是 "h2" 但 Via 仍寫 "2.0"）
  • 請求路徑：client → proxy1 → cloudflare → backend-lb → origin

實務上多數 proxy 預設不送 Via header 或會把它移除（出於安全考量 — 不想讓客戶知道後面架構）。Squid 預設會送，Nginx 不會（除非 proxy_set_header Via $http_via;），Cloudflare 預設不會。

RFC 7230 §5.7.1 有一個有趣的設計：

A sender MAY replace any received-by host of any recipient server [...] with a pseudonym thereof. If it does so, the sender MUST include the "received-by" in the resulting Via field as a comment. [...] This is intended for security and privacy. RFC 7230 §5.7.1

意思是 proxy 可以把自己的真實 hostname 替換成 pseudonym。Cloudflare 等 CDN 通常採用這策略 — 你看到的是 Via: 1.1 cloudflare，而不是真實邊界節點 hostname。

10.5 — X-Forwarded-* 系列：事實標準

當 Reverse Proxy 把請求轉給後端時，後端只看到 proxy 的 IP，不知道真實 client 是誰。為了解決這個問題，業界在 2000 年代初發明了 X-Forwarded-* 系列 header：

Header意義典型值
X-Forwarded-For原始 client IP（與後續 proxy IP）203.0.113.5, 198.51.100.1
X-Forwarded-Proto原始協定（http 或 https）https
X-Forwarded-Host原始 Host headerwww.example.com
X-Forwarded-Port原始 port443
X-Real-IPNginx 慣用的 client IP（單一值）203.0.113.5

X-Forwarded-For 是逗號分隔的列表 — 每經過一層 proxy 就追加一個 IP。例如收到 X-Forwarded-For: 1.2.3.4, 5.6.7.8 意味著：

  • 真實 client 是 1.2.3.4
  • 第一層 proxy 是 5.6.7.8
  • 當前 proxy 是 TCP 直接的對端（不在 header 裡）

這個 header 帶來幾個常見陷阱：

  • 容易被偽造：client 可以自己送 X-Forwarded-For: 8.8.8.8 偽造 IP。所以 application 必須只信任已知 proxy 後追加的部分，不能直接信任 header 全部。
  • spoofing 攻擊：「我從 127.0.0.1 來」是經典繞過 IP 白名單的手法（如果應用直接 trust X-Forwarded-For）
  • Privacy 與 GDPR：cf. Ch.7，IP 即個資，X-Forwarded-For 在 log 中要納入 retention policy

10.6 — RFC 7239 Forwarded：標準化但採用率低

2014 年 IETF 終於把 X-Forwarded-* 標準化為單一 header — Forwarded（RFC 7239）。設計目標是把多個事實標準的 header 整合：

Forwarded: for=203.0.113.5;proto=https;by=198.51.100.1,
           for=198.51.100.1;proto=https;by=203.0.113.43

每個元素是 ; 分隔的 key=value 對；多個元素用 , 分隔（一層一個）。比 X-Forwarded-* 更明確、更標準。

但採用率至今仍低。原因：

  • Nginx 預設不送 Forwarded（仍預設用 X-Real-IP / X-Forwarded-For）
  • Cloudflare、AWS ALB、多數 CDN 預設用 X-Forwarded-*
  • 多數 application framework（Express、Spring、Django）的 middleware 預設讀 X-Forwarded-For，不讀 Forwarded
  • 已有的 X-Forwarded-* 部署成本太低，沒人有動力遷移

2026 年的實務建議：除非你的環境完全自控且追求標準合規，否則繼續用 X-Forwarded-*。RFC 7239 是個更好的設計，但生態系沒跟上。

10.7 — Hop-by-hop vs End-to-end Headers

HTTP 有一類特殊 header — hop-by-hop headers。它們只在 client 與當前 proxy 之間有意義，必須在 forward 給下一跳前移除。RFC 7230 §6.1 列出標準的 hop-by-hop headers：

  • Connection
  • Keep-Alive
  • Proxy-Authenticate
  • Proxy-Authorization
  • TE
  • Trailers
  • Transfer-Encoding
  • Upgrade

另外 Connection header 自己可以列出更多要視為 hop-by-hop 的 header：

Connection: close, X-My-Custom-Hop-By-Hop

意思是「X-My-Custom-Hop-By-Hop 也視為 hop-by-hop，請 proxy 移除」。

所有未列為 hop-by-hop的 header 都是 end-to-end，必須原封不動傳遞給下一跳。例如 Authorization、User-Agent、Content-Type 都是 end-to-end。

Proxy 沒正確處理 hop-by-hop 是常見 bug 來源 — 例如 Upgrade: websocket 是 hop-by-hop，proxy 必須在 連接 backend 時自己重新發起 upgrade 握手，不能透傳。Nginx 的 WebSocket 設定要求 proxy_set_header Connection "upgrade"; 與 proxy_set_header Upgrade $http_upgrade; 就是這原因。

10.8 — HTTP/2 與 HTTP/3 在 Proxy 場景的特殊性

HTTP/2 與 HTTP/3 給 Proxy 帶來新的複雜度。本節做高層次概覽，Ch.12 會深入。

HTTP/2：禁用的 Connection header

HTTP/2（RFC 7540 §8.1.2.2 及更新版 RFC 9113）禁止使用 Connection、Keep-Alive、Proxy-Connection、Transfer-Encoding、Upgrade 等 hop-by-hop header。HTTP/2 用自己的機制處理連線控制（SETTINGS frame、PING frame、GOAWAY frame）。所以 proxy 在 HTTP/1.1 ↔ HTTP/2 翻譯時，必須把這些 header 刪掉或轉換。

HTTP/2：CONNECT 方法的擴充

HTTP/2 的 CONNECT 與 HTTP/1.1 略有不同（RFC 7540 §8.3）— :authority pseudo-header 取代 request target。RFC 8441 進一步擴充 CONNECT 支援 WebSocket（在 HTTP/2 上跑 WebSocket）。

HTTP/3：UDP-based 帶來的全新挑戰

HTTP/3 over QUIC over UDP 給 proxy 帶來全新挑戰：

  • 傳統 L4 LB（基於 TCP）不能直接處理 QUIC，需要支援 UDP（且要懂 QUIC connection ID）
  • 沒有 5-tuple（src/dst IP+port + proto）能穩定識別連線 — QUIC 的 connection migration 讓 IP 變化也能保持連線
  • 很多防火牆與 DPI 設備預設限制或丟棄 UDP/443 流量
  • 部署成本：Nginx 需要 1.25.0+ + ssl_quic 設定；HAProxy 需要 2.6+；Envoy 從 2022 開始支援；Cloudflare 用 quiche 庫已全網部署

2025 年 10 月 HTTP/3 全球採用率達 35%（Cloudflare 數據）。如果你的 reverse proxy 還沒打開 HTTP/3，是時候考慮了。Caddy 預設啟用，Nginx 加兩行就行：

Listing 10.1 · Nginx 啟用 HTTP/3server {
    listen 443 ssl;
    listen 443 quic reuseport;     # HTTP/3 over QUIC (UDP)
    http2 on;
    http3 on;                       # 啟用 HTTP/3
    server_name example.com;
    ssl_protocols TLSv1.3;            # HTTP/3 強制 TLS 1.3
    ssl_certificate     ...;
    ssl_certificate_key ...;
    ssl_early_data on;                # 啟用 0-RTT
    quic_retry on;                    # DDoS 防護

    # 告訴瀏覽器這個服務有 HTTP/3 — 沒這行瀏覽器不會切過去！
    add_header Alt-Svc 'h3=":443"; ma=86400' always;
}

Alt-Svc header 是關鍵 — 瀏覽器透過這個 header 才會發現「這個服務支援 HTTP/3」，下次連線會優先嘗試 QUIC。沒有 Alt-Svc，瀏覽器會永遠用 HTTP/2。

10.9 — Proxy-Authorization 與 Proxy-Authenticate

Forward Proxy 場景中，proxy 自己可以要求認證 — 這是企業環境最常見的設置：「員工要先登入企業 proxy 才能上網」。

流程：

  1. Client 送請求（沒帶認證）
  2. Proxy 回 HTTP/1.1 407 Proxy Authentication Required 並附 Proxy-Authenticate header 描述認證方式
  3. Client 重送請求，這次帶 Proxy-Authorization: Basic dXNlcjpwYXNz（或其他 scheme）
  4. Proxy 驗證後放行
Proxy 要求認證HTTP/1.1 407 Proxy Authentication Required
Proxy-Authenticate: Basic realm="corporate-proxy"
Proxy-Authenticate: NTLM
Proxy-Authenticate: Negotiate

三種主流認證 scheme：

  • Basic（RFC 7617）：明文 base64 編碼，不安全，只在 HTTPS 環境可用
  • NTLM：Windows 環境企業 proxy 慣用，三步握手
  • Negotiate（GSSAPI / Kerberos / SPNEGO）：AD 環境的 single sign-on，最安全也最複雜

注意 Proxy-Authorization 是 hop-by-hop — proxy 不應該往下游傳。但應用層認證的 Authorization header 是 end-to-end，必須透傳到 origin server。新手常犯的錯誤：把這兩個搞混。

10.10 — Squid 的 SSL Bump：當 HTTP Proxy 想看 HTTPS

本書 Ch.7 §7.4 已從法律角度討論 SSL Bump。從技術角度看，SSL Bump 是 HTTP Proxy 突破「CONNECT 退化為 L4 純轉送」限制的解法 — 它偽裝成目標 server，與 client 做 TLS 握手；同時自己以 client 身分與真目標做另一次 TLS 握手，中間解密看內容。

Squid 的 SSL Bump 模式：

  • Splice：純轉送（如同沒裝 SSL Bump 的 CONNECT），不解密
  • Peek：偷看 ClientHello 但不解密 — 從 SNI 看到目標域名後決定後續動作
  • Stare：偷看 ServerHello（看憑證但不解密內容）
  • Bump：完整解密 — 用 Squid 的 CA 動態簽發目標域名的偽造憑證

實務部署常用「peek-and-splice」策略：peek ClientHello，看 SNI；對銀行、醫療等敏感網站 splice（不解密），對其他網站 bump（解密看內容）。Ch.19 會展開完整 Squid SSL Bump 設定。

SSL Bump 的存在說明一個重要事實：HTTP Proxy 的能力邊界是「能不能解密」。能解密就是 L7（甚至 App Logic）；不能解密就退化為 L4。Bump 讓你選擇 — 但代價是法律風險、效能成本、與信任邊界的擾動。

Chapter 11

TLS 與 HTTPS 的代理
從 SNI 檢查到 Encrypted Client Hello 的範式轉移

2025–2026 是 TLS Proxy 領域的轉折年 — Encrypted Client Hello（ECH）正在普及，這是 SNI 檢查時代的終結。本章帶你看懂 TLS 1.3 ClientHello 的每個 byte，並理解 ECH 對既有 Proxy 設計的衝擊。

TLS 是 Web 加密的基石。從 1995 年 Netscape 推出 SSL 2.0 開始，這個協定經過了三十年的演進 — SSL 2.0/3.0、TLS 1.0/1.1/1.2，到 2018 年的 TLS 1.3（RFC 8446）。每一次改版都是「更安全、更隱私、握手更少」的權衡。但對 Proxy 工程師而言，這段歷史也是「能看到的東西越來越少」的歷史 — TLS 1.2 時代我們還能讀 ClientHello、看憑證鏈；2026 年的 TLS 1.3 + ECH 環境裡，連 SNI 都將從明文變成密文。

這個演進直接衝擊到所有基於 SNI 做路由的 Proxy — HAProxy 的 SNI-based routing、Nginx stream module、AWS NLB SNI listener、Cloudflare 的多租戶邊界、企業 SSL Bump 設備 — 全部需要重新設計。本章從 TLS 1.3 ClientHello 的 byte-level 結構開始，逐步展開 SNI 檢查的當前實踐、ECH 的衝擊、以及 SSL Bump 的技術細節。

對 Proxy 而言，1.3 帶來的影響：

  • 更多東西被加密：1.2 時 Cert、Cert Status 是明文；1.3 起整個 EncryptedExtensions、Cert、Finished 都加密。Proxy（如果它沒做 termination）什麼都看不到，除了 ClientHello。
  • 0-RTT 重連：第二次連線可直接把 HTTP 請求塞進 ClientHello。對 Proxy 的安全考量：早期資料容易被重放（POST、購買、計費等非冪等請求要拒絕在 0-RTT 中執行）。Cloudflare、Nginx 都提供 ssl_early_data on 但建議搭配 Early-Data: 1 header 讓 backend 自行判斷是否接受。
  • SNI 仍是明文：1.3 仍未解決 SNI 隱私問題，這是 ECH 出現的動機。

11.2 — TLS 1.3 ClientHello 的結構

ClientHello 是 TLS 握手的第一個訊息。對 Proxy 而言，這是唯一可檢查的明文部分。理解它的結構是所有 SNI-based 代理技術的前提。

從 wire 上看，ClientHello 的前幾個 byte 是 TLS record layer header：

16 03 01 02 00 # TLS record header │ │ │ └──┘ │ │ │ └─ Length (16 bits): 0x0200 = 512 bytes
│ │ └─ Legacy version: 0x0301 (TLS 1.0，1.3 仍偽裝成 1.0 以相容老設備) │ └─ ContentType: 0x16 = Handshake └─ (record 開始)

之後是 Handshake header 與 ClientHello 主體：

01 00 01 fc # Handshake header │ └────────┘ │ └─ Length (24 bits): 0x0001fc = 508 bytes └─ HandshakeType: 0x01 = ClientHello 03 03 # Legacy version: TLS 1.2 (1.3 仍偽裝) [32 bytes random] # Client random 20 [32 bytes session_id] # Legacy session ID (1.3 已棄用但保留) 00 1e [cipher suites...] # 30 bytes 的 cipher suite 列表 01 00 # Compression methods: null (1.3 強制不壓縮) 01 a3 [extensions...] # 0x01a3 = 419 bytes 的 extensions

關鍵在 extensions 段 — 這裡放著 SNI、ALPN、supported_versions（真正的 1.3 標識）、key_share、signature_algorithms 等。每個 extension 格式統一：

Type 2 bytes extension 類型
Length 2 bytes 資料長度
Data 變動 extension 內容

對 Proxy 來說，最關鍵的幾個 extension：

TypeExtension 名稱Proxy 用途

0x0010application_layer_protocol_negotiation (ALPN)協定協商：h2 / h3 / http/1.1

11.3 — SNI Extension 的精準解構

SNI（Server Name Indication，RFC 6066）的設計動機是讓一個 IP 上能託管多個域名 — server 從 ClientHello 看到 SNI 後，才知道要回傳哪個憑證。整個現代 Web 共享 IP 的基礎都建立在 SNI 之上。

SNI extension data 的二進位格式：

00 00 00 12 # Extension type=0x0000 (SNI), Length=18 00 10 # ServerNameList length=16 00 # NameType=0 (host_name) 00 0d # HostName length=13 65 78 61 6d 70 6c 65 2e 63 6f 6d # "example.com"

關鍵性質：

  • SNI 在 TLS 1.2 與 1.3 都是明文傳輸
  • 因為它在 ClientHello 開頭附近，Proxy 不需要解密就能讀
  • 2019 年以前是純技術設計；2019 年韓國 KCSC 開始用 SNI 檢查取代 DNS 封鎖做網路審查，這個事件加速了 ECH 的開發
  • HAProxy 的 req_ssl_sni、Nginx stream module 的 $ssl_preread_server_name、AWS NLB 的 SNI listener 都依賴這個明文 SNI 做路由

11.4 — ALPN：協定協商的 byte-level 細節

ALPN（Application-Layer Protocol Negotiation，RFC 7301）告訴 server「我支援哪些應用層協定」。對 Reverse Proxy 而言，ALPN 是HTTP/1.1 vs HTTP/2 vs HTTP/3 的關鍵協商機制。

00 10 00 0e # Extension=0x0010 (ALPN), Length=14 00 0c # ProtocolNameList length=12 02 68 32 # Length=2, "h2" (HTTP/2) 08 68 74 74 70 2f 31 2e 31 # Length=8, "http/1.1"

標準的 ALPN protocol IDs：

ID 字串意義
http/1.1HTTP/1.1
h2HTTP/2 over TLS
h2cHTTP/2 cleartext（罕見）
h3HTTP/3 over QUIC
acme-tls/1ACME TLS 驗證（Let's Encrypt 等）
dotDNS over TLS
doqDNS over QUIC

Server 從 client 提供的 ALPN 清單中選一個（通常選效能最好的，如 h2 > http/1.1），透過 ServerHello 的 ALPN extension 回應。順序就是優先級：client 先列的優先。

11.5 — JA3 與 JA4 指紋：ClientHello 的特徵化

2017 年 Salesforce 工程師 John Althouse、Jeff Atkinson、Josh Atkins 發明了 JA3 指紋 — 把 ClientHello 的特定欄位拼成字串，再 MD5 hash 一下，產生一個「TLS client 指紋」。這個指紋揭示一個事實：不同 TLS client 庫（Chrome、Firefox、curl、Go net/http、Python requests）的 ClientHello 各有特徵。

JA3 計算公式：

JA3 計算JA3 = MD5("TLSVersion,Ciphers,Extensions,EllipticCurves,EllipticCurvePointFormats")

範例（Chrome 89 on Windows）:
"771,4865-4866-4867-49195-49199-...,0-23-65281-10-11-35-16-5-13-18-51-45-43-27-17513,29-23-24,0"

MD5 → cd08e31494f9531f560d64c695473da9

2023 年 John Althouse（已轉職至 FoxIO）發表 JA4 — JA3 的繼承者。JA4 解決了 JA3 的幾個問題：

  • JA4 用 SHA256 取代 MD5
  • JA4 的格式是人類可讀的（如 t13d1517h2_8daaf6152771_b0da82dd1658），便於分類與檢索
  • JA4 能涵蓋 HTTP/2、HTTP/3，不只 TLS
  • JA4 系列包含 JA4S（server）、JA4H（HTTP）、JA4X（X.509 憑證）等變體

對 Proxy 工程師的實務意義：

  • SOCKS5 透傳保留 JA3/JA4：cf. Ch.9 §9.9。爬蟲、隱私場景用 SOCKS5
  • HTTP Proxy + TLS Termination 改變 JA3/JA4：你看到的指紋變成 Proxy 自己的（Cloudflare、AWS ALB 都會顯著改變指紋）
  • WAF 用 JA3/JA4 偵測機器人：Cloudflare Bot Management、Akamai Bot Manager、AWS WAF 都把 JA3/JA4 當成機器人偵測的訊號之一
  • TLS 指紋偽造工具（如 utls、curl-impersonate）能讓 Go / curl 的 ClientHello 偽裝成 Chrome — 這在合法的測試與不合法的繞過都有應用

本書 Ch.67（資安洩漏全攻略）會展開 JA3/JA4 的完整紅藍隊應用。

11.6 — Encrypted Client Hello（ECH）：2025–2026 的範式轉移

2018 年 IETF TLS WG 開始討論「Encrypted SNI」（ESNI），最終演化為更全面的 Encrypted Client Hello（ECH）。截至 2026 年初，ECH 仍是 IETF Internet-Draft（draft-ietf-tls-esni），但已有大規模實際部署。

ECH 的核心設計

ECH 採用「外層信封 + 內層信封」設計：

  1. Client 寫一份「真實的 ClientHello」 — 含真實 SNI（如 secret.example.com）、真實 ALPN 等
  2. 用 server 預先公告的公鑰（透過 DNS HTTPS record 取得）加密這個內層 ClientHello
  3. 產生「外層 ClientHello」 — 用一個通用 SNI（如 cloudflare-ech.com），把加密過的內層放在 ECH extension（type 0xfe0d）裡
  4. 把外層 ClientHello 送給 server

網路觀察者只看得到外層 SNI（無法分辨多個域名）；server 收到後解密內層，取得真實 SNI 與 ALPN。

2026 年 ECH 部署現況

  • Cloudflare：2023 年 9 月啟動，2024–2025 年逐步擴大；2026 年所有 Cloudflare 客戶域名預設可用 ECH
  • Firefox：113 版（2023/05）起預設啟用
  • Chrome：117 版（2023/09）起逐步推出
  • iCloud Private Relay：Apple 與 Cloudflare 合作的隱私服務深度整合 ECH

ECH 對 Proxy 的衝擊

2026 年 ECH 部署檢查清單

如果你的基礎設施有以下任何一項，必須立即評估 ECH 衝擊：

• HAProxy / Nginx stream / AWS NLB / GCP TCP LB 用 SNI 做路由
• Squid SSL Bump（peek-and-splice 模式依賴 SNI 判斷敏感網站）
• 企業 SSL inspection 設備（Zscaler、Palo Alto、Forcepoint 等）
• IDS/IPS 用 SNI 做網站分類
• 國家 / ISP 級別的 SNI-based 過濾

AWS Network Firewall 文件已明確聲明：「Traffic encrypted using TLS v1.3 Encrypted SNI and Encrypted Client Hello extensions aren't supported.」企業安全產品仍在追趕中。

解法的方向（沒有單一銀彈）：

  • 對外服務：負載平衡器需要支援 ECH decryption（Cloudflare、Fastly 已內建；自建環境需要新版 HAProxy / Nginx）
  • 企業稽核：完全依賴 SNI 的舊有方案會逐步失效；必須轉向endpoint agent（在使用者裝置上做 inspection）或full TLS termination（裝 CA、做 SSL Bump，但這放大 Ch.7 §7.4 的法律風險）
  • 網路審查：SNI 檢查時代結束。2025 年起多國觀察到的轉向是「整段 IP 封鎖」而非「域名封鎖」 — 這對 CDN 客戶有附帶傷害（一人作惡，整個 CDN IP 段被封）

11.7 — TLS Termination：Reverse Proxy 的核心職責

Reverse Proxy 最常見的 TLS 工作是 termination — 在邊界終止 TLS，與後端 server 用明文 HTTP 通訊（或重新加密成另一條 TLS）。Nginx、HAProxy、AWS ALB 都這樣做。

典型部署：

Termination 帶來的能力：

  • 邊界做完整 HTTP 解析（method / URL / header / body 全部可見）
  • WAF、限流、cache 都能運作
  • 後端不需處理 TLS（CPU 省、設定簡單、憑證集中管理）
  • 可以做 HTTP/2 → HTTP/1.1、HTTP/3 → HTTP/1.1 的協定降級
  • 可以重組 connection pool（一千個 client TLS 連線複用十條 backend 連線）

代價：

  • Proxy 必須處理 TLS（CPU、憑證更新、加密庫安全更新）
  • 後端到 Proxy 之間明文，需要可信內網或自建 mTLS
  • JA3/JA4 指紋變成 Proxy 的，不再是 client 的
  • 0-RTT 早期資料的重放攻擊風險需謹慎處理

許多需要 end-to-end 加密的場景（金融、醫療）會用 TLS passthrough 替代 — Proxy 只看 SNI 做路由，不做解密。HAProxy 的 mode tcp + req_ssl_sni、Nginx 的 stream module、AWS NLB 都是這個模式。

11.8 — SSL Bump：突破 CONNECT 限制

本書 Ch.7 §7.4 從法律角度討論 SSL Bump，Ch.10 §10.10 提到它的存在。本節從技術角度展開。

SSL Bump（Squid 術語，Cisco/Zscaler 等廠商稱「TLS Inspection」「SSL Inspection」）的核心機制：

  1. Client 透過 CONNECT 請求建立到 example.com:443 的隧道
  2. Proxy 接受 CONNECT 但不把流量純轉送，而是自己扮演 example.com與 client 做 TLS 握手
  3. Proxy 用內部 CA 動態簽發 example.com 的偽造憑證 — client 必須事先安裝這個 CA 為信任根（這就是企業 IT 在員工裝置上推送企業 CA 的原因）
  4. 同時 Proxy 以 client 身分與真實 example.com 做另一次 TLS 握手，取得真憑證
  5. Client → Proxy 一條 TLS、Proxy → server 另一條 TLS。Proxy 在中間解密、檢查、重新加密

Squid 的四種模式

模式動作看得到什麼
Splice純轉送（不解密）什麼都看不到（除了五元組）
Peek偷看 ClientHello 但不解密SNI、ALPN、JA3 指紋
Stare偷看 ServerHello（憑證）但不解密應用層真實憑證鏈、CN/SAN
Bump完整解密所有 HTTP 內容

典型企業部署：peek-and-splice 策略。先 peek SNI，對銀行（*.bankofamerica.com）、醫療（*.kaiserpermanente.org）splice 不解密；對其他網站 bump 完整解密。Ch.19 會展開完整 Squid SSL Bump 配置。

Certificate Pinning：SSL Bump 的剋星

Mobile app（特別是銀行、社群、Gmail / Google 系列）廣泛實作 certificate pinning — app 內部硬編碼預期的憑證指紋，連線時驗證憑證鏈中至少有一個符合。SSL Bump 簽發的偽造憑證不在 pin 列表中，連線會失敗。

對企業 SSL Bump 部署而言，這意味著：

  • 大量 app 在 SSL Bump 環境下完全無法使用
  • 必須維護 bypass 列表（mobile.banking.com、 *.googleapis.com 等）
  • 使用者體驗明顯降低（員工抱怨企業 WiFi 上 app 不能用）
  • HSTS、HPKP（已棄用）也是相關防護

11.9 — mTLS：雙向 TLS 認證

Mutual TLS（mTLS）讓 server 也驗證 client 的憑證，在 Service Mesh、Zero Trust 場景至關重要。Reverse Proxy 在 mTLS 中扮演重要角色：

  • Service Mesh（Istio、Linkerd、Consul）：sidecar 之間自動建立 mTLS，憑證由 mesh 自動簽發與輪替
  • Zero Trust：使用者裝置裝設備憑證，Proxy（如 Cloudflare Access、Pomerium）驗證 client cert + 使用者身分
  • API Gateway：合作廠商的 API 整合常用 mTLS 取代 OAuth

Nginx 啟用 mTLS：

    listen 443 ssl http2;

11.10 — TLS 1.3 與後量子密碼學

2024 年 NIST 正式發布後量子密碼學標準（PQC）。Cloudflare、Google、Apple 在 2025 年陸續部署混合 PQC 密鑰交換（X25519 + Kyber768）。對 Proxy 而言，這意味著 ClientHello 的 key_share 變得更大，TLS 握手 byte 量增加。

2026 年的 PQC 部署實況：

  • Chrome 124+ 預設啟用 X25519Kyber768
  • Cloudflare 邊界全網支援
  • OpenSSL 3.5（2025/04）開始整合 PQC
  • 大多數 Reverse Proxy（Nginx、HAProxy、Envoy）依賴 OpenSSL，所以版本升級即可獲得 PQC

PQC 不改變 Proxy 的設計範式，但會讓 ClientHello 變大（多幾百 byte），對 SNI-based routing 沒有直接影響。

Chapter 12

HTTP/2 與 HTTP/3
從文字協定到二進位框架，從 TCP 到 QUIC

HTTP/2 在 2015 年把 HTTP 從文字協定變成二進位框架；HTTP/3 在 2022 年又把整個傳輸層從 TCP 換成 QUIC。本章解構這兩次革命的 byte-level 細節，並理解 Proxy 在跨版本場景下的協定翻譯職責。

HTTP/1.1 從 1997 年（RFC 2068）服役至今將近三十年。它的設計哲學是「用 ASCII 文字寫網路協定」 — 你能用 telnet 手打請求、用 cat 直接讀回應。這個易讀性是它能存活這麼久的關鍵之一，但也是它效能上限的根源 — 解析文字、處理空白、辨識邊界，全部都是 CPU 成本。HTTP/2（2015）與 HTTP/3（2022）兩次革命，目標都是用更精密的 wire format 換更高的效能。

對 Proxy 工程師而言，這兩次演進帶來的是「同時處理三代協定」的現實。Cloudflare 邊界節點接到 client 端是 HTTP/3，到 origin 端可能還是 HTTP/1.1，整個請求要在邊界跨越三代協定。理解每代的 wire format 與互通機制，是設計 Reverse Proxy 的前提。

12.1 — HTTP/2：從 SPDY 到 RFC 9113

HTTP/2 起源於 Google 2009 年發布的 SPDY 協定。SPDY 的核心觀念在 2012 年被 IETF 採納為 HTTP/2 草案；2015 年 5 月發布為 RFC 7540；2022 年 6 月以 RFC 9113 取代（解決一些安全與互操作問題）。RFC 9113 在 2026 年仍是現行標準。

HTTP/2 相對 HTTP/1.1 的核心改變：

  • 二進位 framing：所有訊息分割成 frame，每個 frame 9-byte 標頭 + 變動 payload
  • Multiplexing：一條 TCP 連線承載多個 stream，stream 之間並行（解決 HTTP/1.1 的 head-of-line blocking）
  • HPACK 標頭壓縮：用索引表壓縮重複的 header（Content-Type、Cookie 等通常一樣）
  • Server Push（已棄用）：server 主動推資源
  • 強制加密的事實：規格說「明文 HTTP/2 也合法」，但所有瀏覽器只用 HTTPS

12.2 — HTTP/2 連線建立與 Magic preface

HTTP/2 連線的開頭有一個有趣的「magic」 — 一段 24-byte 的固定文字，用來確認雙方確實要講 HTTP/2：

50 52 49 20 2a 20 48 54 54 50 2f 32 2e 30 0d 0a 0d 0a 53 4d 0d 0a 0d 0a 解碼成 ASCII: PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n

注意 "PRI" 與 "SM" — 連起來是 "PRISM"。這是 RFC 7540 的作者對 2013 年 Snowden 揭露的 NSA 監控計畫（PRISM）的無聲抗議。維基百科明確記錄這個彩蛋。

連線建立流程：

  1. TLS 握手完成，ALPN 協商出 "h2"
  2. Client 送 magic preface（24 bytes）
  3. Client 送 SETTINGS frame（聲明自己的設定）
  4. Server 收到後回 SETTINGS frame
  5. 雙方 ACK 對方的 SETTINGS
  6. 開始送 HEADERS / DATA frame

0x05PUSH_PROMISEServer Push（已棄用，prefer 103 Early Hints）
0x06PING連線層 ping/pong
0x07GOAWAY優雅關閉連線
0x08WINDOW_UPDATEflow control 配額更新
0x09CONTINUATIONHEADERS 太長分塊

Stream ID 是 31-bit 整數，0x00 保留給連線層 frame（如 SETTINGS、PING）。Client 發起的 stream 用奇數 ID，server 發起的用偶數。Stream ID 不重複使用 — 一條 HTTP/2 連線最多用 2³¹ 個 stream，理論上不會耗盡。

12.4 — SETTINGS Frame：連線層設定協商

SETTINGS frame 的 payload 是任意數量的「16-bit 設定 ID + 32-bit 值」對。RFC 9113 定義 6 個標準設定：

ID名稱預設值意義
0x01SETTINGS_HEADER_TABLE_SIZE4096HPACK 動態表大小（bytes）
0x02SETTINGS_ENABLE_PUSH1是否允許 server push（client 設 0 禁用）
0x03SETTINGS_MAX_CONCURRENT_STREAMS無限最多同時 stream 數（Cloudflare 預設 100）
0x04SETTINGS_INITIAL_WINDOW_SIZE65,535flow control 初始視窗
0x05SETTINGS_MAX_FRAME_SIZE16,384單個 frame 最大 size
0x06SETTINGS_MAX_HEADER_LIST_SIZE無限headers 總大小上限

SETTINGS 的重要性質：兩端可以有完全不同的設定。Client 可能說「我接受 200 個並行 stream」，Server 可能說「我只接受 100」。SETTINGS 是 declarative 而非 negotiation — 各說各話，雙方記住對方的設定即可。

SETTINGS 也是 JA4 指紋的一部分 — Chrome 與 Firefox 的 SETTINGS 順序與值不同，這成為 client 識別的訊號之一。

12.5 — HEADERS 與 HPACK 壓縮

HTTP/2 的 HEADERS frame 用 HPACK（RFC 7541）壓縮 header。HPACK 的核心觀念：

  • 靜態表：61 個常見 header 預先編號（如 :method=GET 是 #2、:path=/ 是 #4）
  • 動態表：連線層維護一個 header → index 對照表，重複出現的 header 用短 index 取代
  • Huffman 編碼：剩下無法索引的字串用 Huffman 壓縮

實際效果驚人：一個典型 web 請求的 header（含 cookie、user-agent、accept 等）從 800 bytes 壓到 80 bytes 是常見的。但 HPACK 的動態表是 connection-scoped 狀態 — 每條 HTTP/2 連線都要維護自己的表。對 Proxy 而言，這意味著：

  • HTTP/2 連線重用是非常有價值的（每次新建連線要重新累積 HPACK 狀態）
  • Proxy 在中間翻譯 HTTP/2 ↔ HTTP/1.1 時，必須完整解碼 HPACK 再重新編碼
  • HPACK 的記憶體使用 = SETTINGS_HEADER_TABLE_SIZE × concurrent streams，邊緣 Proxy 處理百萬連線時會佔大量 RAM

12.6 — Pseudo-headers：HTTP/2 取代 request line

HTTP/1.1 的 request line（GET /path HTTP/1.1）在 HTTP/2 變成「pseudo-headers」 — 以冒號開頭的特殊 header：

Pseudo-header意義HTTP/1.1 對應
:methodGET / POST / ...request line 第一個 token
:schemehttp / httpsURL scheme
:authorityhost:portHost header
:path/api/users?id=1URL path + query
:status200 / 404 / ...response status code

規則：

  • Pseudo-headers 必須在所有 regular headers 之前
  • Request 用 :method、:scheme、:authority、:path
  • Response 用 :status
  • Connection、Keep-Alive 等 hop-by-hop header 在 HTTP/2 完全禁用

對 Proxy 翻譯：HTTP/1.1 → HTTP/2 時，把 GET /api HTTP/1.1\r\nHost: example.com 拆成 :method=GET, :path=/api, :scheme=https, :authority=example.com；反向類似。

12.7 — HTTP/3：從 TCP 到 QUIC 的革命

HTTP/3 解決了 HTTP/2 的最後一個瓶頸 — TCP 層的 head-of-line blocking。

HTTP/2 在應用層 multiplex 多個 stream，但所有 stream 仍共享一條 TCP 連線。TCP 是有序 byte stream，任何一個封包遺失，所有後續資料都要等重傳，不管那些後續資料屬於哪個 stream。

HTTP/2 over TCP（仍然有 head-of-line blocking） Stream 1: ███▓░░░ ←── packet 4 lost Stream 2: ███████ ← 也被卡住（屬於同一 TCP） Stream 3: ███████ ← 也被卡住 HTTP/3 over QUIC（每個 stream 獨立） Stream 1: ███▓░░░ ←── packet 4 lost Stream 2: ███████ ← 不受影響 Stream 3: ███████ ← 不受影響
圖 12.1　HTTP/2 vs HTTP/3 的 head-of-line blocking 差異。HTTP/3 的 QUIC 把 stream 獨立性做進傳輸層。

QUIC（RFC 9000，2021）的核心特性：

  • 基於 UDP：避免 OS kernel 的 TCP 處理
  • 每個 stream 獨立可靠：packet loss 只影響該 stream
  • 整合 TLS 1.3：握手與密鑰交換放進傳輸層，1-RTT 連線
  • Connection ID：以 8 byte ID 識別連線，IP/port 變化也不斷線（行動裝置 WiFi ↔ 4G 切換）
  • 0-RTT 重連：第二次連線可直接送 HTTP 請求

HTTP/3 = HTTP 語意 over QUIC（RFC 9114，2022）。它使用：

  • QPACK（RFC 9204）取代 HPACK：避免 HPACK 在亂序 stream 下的同步問題
  • Stream 編號規則類似 QUIC：client-initiated bidirectional 用 0, 4, 8, ...；server-initiated 用 3, 7, 11, ...
  • 不再有 PUSH_PROMISE — Server Push 在 HTTP/3 完全移除（用 103 Early Hints 取代）

12.9 — Alt-Svc：HTTP/3 的「協定發現」機制

HTTP/3 部署的關鍵 puzzle 是「瀏覽器怎麼知道這個 server 支援 HTTP/3」？因為 client 第一次連線通常是 HTTP/1.1 或 HTTP/2 over TCP，不會主動嘗試 UDP/443。

解法是 Alt-Svc header（RFC 7838）：

HTTP/2 200 OK
Content-Type: text/html
Alt-Svc: h3=":443"; ma=86400

<html>...</html>

這個 header 告訴瀏覽器「我這個服務有 HTTP/3 endpoint，在 :443，cache 86400 秒（24 小時）」。瀏覽器下次連線會優先嘗試 HTTP/3，失敗再 fallback HTTP/2。

沒有 Alt-Svc，就沒有 HTTP/3 流量 — Caddy 自動加，Nginx 要手動加（cf. Ch.10 §10.8 的範例設定）。

新興替代方案：DNS 的 HTTPS RR record（RFC 9460）也能聲明 HTTP/3 支援，繞過第一次連線。Apple、Cloudflare 已部署。

12.10 — Proxy 在 HTTP/3 場景的挑戰

HTTP/3 給 Proxy 帶來的挑戰是架構級的：

L4 LB 的根本性挑戰

傳統 L4 LB（HAProxy TCP mode、AWS NLB、F5 LTM）基於 5-tuple（src/dst IP+port + proto）做 connection tracking。QUIC 的 connection migration 讓 IP 變化也保持連線 — 這完全打破了 5-tuple 模型。新的 L4 LB 必須懂 QUIC connection ID 才能正確路由。

解法：

  • L4 LB 解析 QUIC 封包，看 connection ID 做 routing（HAProxy 2.7+ 支援）
  • Server 端用「server-generated connection ID」嵌入 backend 識別資訊（一致性 hash 不可逆地映射到後端）

UDP 在企業網路的處境

許多企業防火牆預設限制或丟棄 UDP/443 — UDP 流量在企業環境長期被視為「DNS 與 NTP 之外都可疑」。對 HTTP/3 部署這是個障礙。

解法是備援機制 — Alt-Svc 沒生效時自動 fallback HTTP/2（瀏覽器都會做），所以企業內網仍可正常使用，只是失去 HTTP/3 性能優勢。

Wireshark 與除錯

Wireshark 從 3.4 起支援 QUIC 解析，但需要 client 的 TLS keylog（環境變數 SSLKEYLOGFILE）才能解密 payload。否則 Wireshark 只能看到加密的 QUIC frame。除錯 HTTP/3 比 HTTP/2 困難得多。

DPI 與 IDS 的失能

傳統 DPI（深度封包檢查）依賴明文流量。QUIC 從第一個封包起就加密，連 packet number 都加密。網路上 IDS/IPS 只能看到「有 QUIC 流量在進行」，看不到內容。對企業安全產品而言這是個重大挑戰，多數轉向「endpoint agent」做 inspection。

Chapter 13

WebSocket、SSE、gRPC 的代理處理

三種「長連線、雙向、串流」的協定 — 每一種都有讓 Proxy 工程師頭痛的特殊需求。本章釐清這三者的本質差異，並展開 Proxy 設定中的關鍵細節。

傳統 HTTP 是「請求-回應」的短交易模型：client 送一個請求，server 回一個回應，連線關閉。但現代 Web 充滿了「持續通訊」的需求 — 即時聊天、live 通知、線上協作、影音串流、物聯網狀態同步。三個協定為這些需求設計：WebSocket（雙向二進位 / 文字 frame）、SSE（伺服器推送純文字事件流）、gRPC（基於 HTTP/2 的 RPC + 雙向串流）。

對 Proxy 而言，這三個協定都帶來挑戰：buffering 策略要不一樣、idle timeout 要拉長、HTTP/2 multiplexing 行為要懂、雙向訊息要能透傳。本章逐一展開。

13.1 — WebSocket：HTTP Upgrade 的雙向 frame 通道

WebSocket（RFC 6455，2011）的設計目標是「讓瀏覽器有一個全雙工持久連線」。在它之前，瀏覽器要做即時通訊只能用 long-polling 或 hidden iframe streaming，極為脆弱。

WebSocket 連線建立的聰明設計：複用 HTTP/1.1 的 Upgrade 機制。

Client 發起 WebSocket 連線GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
Origin: https://example.com
Server 接受切換協定HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

之後這條 TCP 連線不再是 HTTP — 它變成 WebSocket frame 的雙向通道。WebSocket frame 格式：

81 85 37 fa 21 3d 7f 9f 4d 51 58 │ │ │ │ │ │ │ └─ Payload (XOR mask 過): "Hello" │ │ └─ Mask key: 4 bytes (client→server 必須 mask) │ └─ MASK=1 (0x80) | PayloadLen=5 (0x05) └─ FIN=1 (0x80) | RSV=0 | Opcode=0x1 (text frame)

關鍵 opcode：

  • 0x0: continuation frame
  • 0x1: text frame（UTF-8）
  • 0x2: binary frame
  • 0x8: close
  • 0x9: ping
  • 0xA: pong

Proxy 處理 WebSocket 的關鍵細節

Reverse Proxy 處理 WebSocket 必須注意：

  1. Connection 與 Upgrade 都是 hop-by-hop — Proxy 不能透傳，必須自己重新發起 upgrade 握手。Nginx 標準寫法：
Nginx WebSocket Reverse Proxylocation /ws {
    proxy_pass http://backend;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    # WebSocket 連線可能很長，把 timeout 拉大
    proxy_read_timeout 86400s;
    proxy_send_timeout 86400s;
}
  1. idle timeout 必須拉長：WebSocket 連線常常閒置幾分鐘到幾小時。Proxy 預設的 60s 會把連線斷掉。
  2. 不要做 buffering：WebSocket 訊息要立即送達，buffering 會導致延遲與順序問題。Nginx 的 proxy_buffering off;。
  3. 留意 sticky session：WebSocket 連線一直保持，必須路由到同一個 backend，不能 round-robin 換。Sticky session（cookie-based 或 IP hash）是必要設定。
  4. 注意 frame 大小：HTTP/2 / HTTP/3 的 frame 大小限制（預設 16KB）對 WebSocket 大訊息有影響，需要拆分。

WebSocket over HTTP/2 與 HTTP/3

原始 WebSocket 只支援 HTTP/1.1。RFC 8441（2018）擴充 HTTP/2 的 CONNECT 方法以支援 WebSocket；RFC 9220（2022）做了 HTTP/3 版本。當你看到 :protocol=websocket 的 pseudo-header，那就是 WebSocket over HTTP/2/3。

支援狀況：

  • Chrome 87+（2020）、Firefox 91+（2021）支援 WebSocket over HTTP/2
  • Nginx 1.25+ 支援；HAProxy 2.6+ 支援
  • 實際採用率仍低 — 多數應用仍用 HTTP/1.1 WebSocket

13.2 — Server-Sent Events（SSE）：簡單但被低估

SSE（Server-Sent Events，HTML5 標準的一部分）比 WebSocket 早 — 它就是純 HTTP 的單向串流：server 持續送資料，client 只接收。設計極簡：

SSE 回應格式HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive

data: First message

data: Second message

event: userJoined
data: {"name": "Alice", "id": 42}

retry: 10000
data: This message will retry after 10s if disconnected

每個事件是一個區塊，欄位用 field: value 格式，區塊間用空行分隔。client 用 EventSource API 訂閱：

JavaScript clientconst es = new EventSource("/events");
es.onmessage = (e) => console.log("got:", e.data);
es.addEventListener("userJoined", (e) => {
  const user = JSON.parse(e.data);
  console.log("user joined:", user.name);
});

SSE 的優勢：

  • 純 HTTP — 與 HTTP/2、HTTP/3 完全相容
  • 瀏覽器自動重連（retry: 欄位控制）
  • Last-Event-ID 機制讓 server 知道 client 重連時錯過了哪些
  • 沒有 WebSocket 的 frame 格式複雜性

對 Proxy 的關鍵設定：禁用 buffering。如果 Proxy 把回應 buffer 起來，事件會卡住，違反 SSE「即時推送」的本質。Nginx 慣用設定：

location /events {
    proxy_pass http://backend;
    proxy_buffering off;           # 關鍵！
    proxy_cache off;
    proxy_http_version 1.1;
    proxy_set_header Connection "";
    chunked_transfer_encoding on;
    proxy_read_timeout 86400s;     # 長連線
}

SSE 在 LLM streaming（OpenAI / Anthropic API）大量使用 — 「token by token」的回應就是 SSE。Part V Ch.43（LLM Gateway）會展開。

13.3 — gRPC：HTTP/2 上的 RPC 框架

gRPC 由 Google 在 2015 年開源，本質是「HTTP/2 + Protobuf + 嚴格的訊息語義」。它不是新的 wire protocol — 你抓 gRPC 流量到 Wireshark，看到的就是 HTTP/2 frame。但它的訊息格式約定有一套自己的規則。

gRPC 請求的 wire 表現：

gRPC over HTTP/2HEADERS frame:
  :method = POST
  :scheme = https
  :path = /helloworld.Greeter/SayHello
  :authority = grpc.example.com
  content-type = application/grpc
  te = trailers
  grpc-encoding = gzip
  grpc-timeout = 30S

DATA frame:
  [1 byte: compressed flag]
  [4 bytes: message length, big-endian]
  [N bytes: protobuf encoded message]

HEADERS frame (trailers):
  grpc-status = 0
  grpc-message = OK

幾個關鍵特性：

  • Path 是 /Service/Method：服務名 + 方法名直接編在 URL
  • Body 是 length-prefixed protobuf：1 byte flag + 4 byte length + protobuf
  • 狀態用 trailer：HTTP status code 一律 200，真正的 RPC 結果用 trailer 的 grpc-status（gRPC status codes 0-16，類似 errno）
  • 四種 streaming 模式：unary、server streaming、client streaming、bidirectional streaming

Proxy 處理 gRPC 的關鍵

因為 gRPC 是 HTTP/2，絕大多數 Reverse Proxy 都能直接處理。但有幾個陷阱：

  1. 必須是 HTTP/2 → HTTP/2 全程：gRPC 不支援 HTTP/1.1 退化（因為依賴 HTTP/2 的 trailer 機制）。Nginx 用 grpc_pass 代替 proxy_pass。
  2. 不要 termination 後降級：如果 Proxy 用 HTTP/1.1 連 backend，gRPC 完全壞掉。
  3. Streaming 要關 buffering：與 SSE 同樣的問題。
  4. gRPC keepalive：gRPC 有自己的 keepalive 機制（HTTP/2 PING frame），Proxy 必須透傳，不能擅自響應。
  5. Trailers 必須支援：HTTP/1.1 trailer 在多數 Proxy 中支援不好；HTTP/2 trailer 支援良好。
Nginx 反向代理 gRPCserver {
    listen 443 ssl http2;
    location / {
        grpc_pass grpc://backend:50051;     # 注意是 grpc://，不是 http://
        grpc_set_header X-Real-IP $remote_addr;
        # 與 WebSocket / SSE 同樣：拉長 timeout
        grpc_read_timeout 3600s;
    }
}

gRPC-Web：在瀏覽器跑 gRPC 的妥協

瀏覽器原生不支援 gRPC（因為瀏覽器無法控制 HTTP/2 trailer）。gRPC-Web 是個 wrapper protocol — client 寫 gRPC，但 wire format 是「HTTP/1.1 + protobuf」，trailer 改用 base64 編碼放進 body 結尾。

需要一個 gRPC-Web proxy（如 Envoy 的 grpc_web filter、grpcwebproxy 工具）在邊界做翻譯。這是 grpc / grpc-web 場景必須有 Reverse Proxy 的原因。

13.4 — gRPC over HTTP/3：仍在實驗

截至 2026 年初，gRPC over HTTP/3 仍是實驗性質。grpc-go 1.60+ 與 grpc-java 1.66+ 有實驗支援，但 production 部署仍少見。原因：

  • QUIC 的 stream 模型與 gRPC 的 stream 語義對應仍在標準化
  • 多數 gRPC 流量在資料中心內部，UDP 帶來的優勢有限
  • 除錯工具鏈仍不成熟

HTTP/3 對外部 client（mobile app）有意義，但這些場景多用 REST 而非 gRPC。所以 gRPC over HTTP/3 在 2026 年仍是「值得關注但不急著部署」的技術。

Chapter 14

DNS 與 Proxy

DNS 是網際網路上少數仍可被中間設備明文觀察的協定 — 直到 2018 年的 DoH。本章拆解 DoH / DoT / DoQ / ODoH 四種加密 DNS 協定，並釐清 DNS 在代理架構中的位置與洩漏向量。

DNS 是網際網路最古老的協定之一（RFC 1035，1987）。它的設計假設了「網路是可信的」 — 查詢與回應都是明文 UDP，沒有加密、沒有完整性保護、沒有來源驗證。這個設計在學術網路時代沒問題，但在 2010 年代後的監控與審查環境變成嚴重的隱私與安全漏洞 — DNS 查詢直接暴露你想連的域名，比 SNI 還早洩漏。

過去十年 IETF 推出了一系列加密 DNS 協定 — DNS over TLS（DoT, 2016）、DNS over HTTPS（DoH, 2018）、DNS over QUIC（DoQ, 2022）、Oblivious DoH（ODoH, 2022）— 把 DNS 從「最容易洩漏的協定」變成「研究最深入的隱私協定」。對 Proxy 工程師而言，這意味著 DNS 不再是基礎設施的「黑盒子」，而是需要主動設計的環節。

14.1 — 加密 DNS 四協定對比

四個協定的設計取捨：

DoT：簡單，但容易被封鎖

DoT 在獨立 port 853 跑 DNS-over-TLS，協定簡單，效能好。但它容易被偵測與封鎖 — 防火牆只要丟掉 853/tcp 就能阻斷。Android 9+（2018）的 Private DNS 功能用 DoT；多數家用路由器（OpenWRT、AdGuard Home、pi-hole）也支援。

DoH：藏在 HTTPS 流量裡

DoH 用 HTTPS port 443 — 與普通 web 流量無法區分（除了 ALPN 為 "dns-over-https"，但這個只在 ECH 環境之外明文）。Mozilla Firefox 從 2018 年起預設啟用 DoH 給 Cloudflare 1.1.1.1。Chrome 從 83 版（2020）起逐步推出。Apple iOS 14+ 支援 DoH 與 DoT。

DoH 的實際請求格式（RFC 8484）：

DoH GET 請求（base64url 編碼的 DNS query）GET /dns-query?dns=AAABAAABAAAAAAAAA3d3dwdleGFtcGxlA2NvbQAAAQAB HTTP/2
Host: dns.cloudflare.com

DoH POST 請求POST /dns-query HTTP/2

[33 bytes of binary DNS query]

DoH 的政治性是它的核心特性 — 它讓網路審查者「分不出哪個 HTTPS 連線是 DNS 查詢」。但同時也讓企業 IT 與家長控制工具失去 DNS 過濾能力，許多企業環境用 endpoint policy 強制關閉瀏覽器的 DoH。

DoQ：QUIC 的低延遲優勢

DoQ 跑在 QUIC（UDP）上，享有 0-RTT 重連與 connection migration。延遲比 DoT 低 30–50%。但生態系仍小，主要部署在 AdGuard、NextDNS 等隱私 DNS 服務。Cloudflare 在 2022 年實驗性支援後仍主推 DoH。

ODoH：把「誰在查」與「查什麼」分開

ODoH（Oblivious DoH）是最有趣的設計。它解決一個 DoH 仍未解決的問題 — DNS server 知道是誰在查什麼。ODoH 引入 relay 與 target 兩個獨立角色：

  1. Client 用 target 的公鑰加密真正的 DNS query
  2. 送給 relay；relay 看到「使用者 IP + 加密的 query」
  3. Relay 轉送給 target；target 看到「relay IP + 解密後的 query」
  4. Target 回應；relay 加密後回送 client

結果：relay 知道誰但不知道查什麼；target 知道查什麼但不知道誰。Cloudflare 與 Apple iCloud Private Relay 在 2021 年合作部署 ODoH（Apple 用作隱私服務的核心元件之一）。2024 年 ODoH 在 Cloudflare 1.1.1.1 上預設可用。

  • Browser 啟用 DoH 把本地 DNS 跳過
  • 用 SOCKS5h 把 DNS 解析委由 Proxy 端
  • 用 VPN 加密整個網路堆疊（包括 DNS）

14.3 — DNS Hijacking 與抗審查

多年來，全球多國的網路審查機制大量依賴 DNS 操作：

  • NXDOMAIN injection：對封鎖網站回 「不存在」
  • Bogus IP injection：回一個錯誤 IP（黑洞或警告頁）
  • DNS hijacking via MITM：ISP DNS server 攔截特定查詢
  • DNS poisoning：利用 UDP 無認證注入偽造回應

抗審查工具（V2Ray、Xray、Shadowsocks、Trojan）標準做法：

  1. 本地用 fake-ip 或 hosts 假裝有結果（避免本地 DNS 被劫持）
  2. 真正的 DNS 解析在遠端 Proxy server進行（透過 SOCKS5h / 內建協定）
  3. 遠端 server 用 DoH/DoT 連到可信 DNS（1.1.1.1、9.9.9.9）

這個分層讓本地網路完全看不到真實的域名查詢；Part III Ch.22–23 會深入展開。

14.4 — DoH 在企業環境的衝突

DoH 對網路審查者是好消息，但對合法的企業 IT是壞消息。企業常需要：

  • DNS-based 內容過濾（封鎖賭博、惡意網站）
  • DNS log 用於合規與威脅偵測
  • Internal DNS 給 intranet 域名（jira.corp.example.com 不能上公開 DNS）

當員工瀏覽器啟用 DoH，這些 DNS 控制全部失效。2026 年企業的標準對策：

  • Group Policy 強制關閉瀏覽器 DoH（Windows AD / macOS MDM）
  • Endpoint agent 取代 DNS 過濾（在 OS 層級攔截）
  • Canary domain（如 use-application-dns.net）— Mozilla 自願承諾如果企業 DNS server 對這個域名回 NXDOMAIN，Firefox 會自動關閉 DoH
  • Endpoint DoH proxy（如 cloudflared）— 員工裝置仍用本地 DoH，但指向公司控制的 DoH endpoint

14.5 — DNS 在 Service Mesh 與 K8s 環境

Service Mesh 與 Kubernetes 的 DNS 是另一個世界：

  • K8s 內部用 CoreDNS 處理 service discovery（my-service.my-ns.svc.cluster.local）
  • Service Mesh 的 sidecar（Envoy）通常不透過 DNS 找後端，而是從 control plane（Istiod）獲取 endpoint 列表
  • Cilium 的 eBPF DNS Lookup 直接在 kernel 攔截 DNS 並做 policy enforcement

Part VI Ch.47–49 會展開這些主題。本章只強調：容器化環境的 DNS 解析路徑與傳統 OS 完全不同，debug 時不能假設 client 直接用 /etc/resolv.conf。

本章要點

DNS 從明文 UDP（1987）演進到 DoT (2016) → DoH (2018) → DoQ / ODoH (2022)。DoH 是最重要的 — 與 HTTPS 流量無法區分，徹底改變網路審查與企業 IT 的力量平衡。Proxy 工程師必須關注：HTTPS via CONNECT 仍洩漏 DNS、SOCKS5 vs SOCKS5h 的關鍵差異、Service Mesh 的 DNS 路徑與傳統 OS 不同。

其他協定的代理

FTP、SSH、SMTP、SIP、資料庫協定 — 這些「非 web」協定在代理時各有獨特的歷史包袱與現代解法。本章快速掃過幾個重要案例。

前六章（Ch.9–14）涵蓋了現代網路流量的 95%。但仍有許多重要協定在代理場景下有獨特的需求 — FTP 的 active vs passive 模式為 NAT 而生、SSH 的 ProxyJump 是運維日常、SMTP 的 relay 鏈是電子郵件的本質、SIP 的 RTP 媒體流挑戰所有 stateless 中介設備、資料庫協定有專屬代理產品。本章不深入細節（每個都能單獨成書），重點是讓你看到「代理」這個概念的廣度。

15.1 — FTP：active vs passive 與 NAT 友好性

FTP（RFC 959，1985）是早期的雙連線協定 — control 連線在 21/tcp，data 連線另開。data 連線方向有兩種模式：

  • Active mode：client 用 PORT 命令告訴 server「我在 IP X port Y 等你」，server 主動連回 client
  • Passive mode：client 送 PASV，server 回「請來連我 IP X port Y」，client 主動連到 server

NAT 出現後，Active mode 在 NAT 後完全壞掉 — server 連不到 NAT 後的 client。Passive mode 解決了這個問題（client 主動），所以 1990 年代起 passive 成為主流。

SOCKS5 的 BIND 命令（CMD=0x02）原本就是為支援 FTP active mode 設計（cf. Ch.9 §9.5）— 讓 SOCKS server 替 client 開一個 listening port，等 server 連回。但隨著 passive mode 主導，BIND 命令幾乎沒人用了。

2026 年 FTP 已大幅式微（取代品：HTTPS、SFTP、S3、rsync）。但 ICS（工業控制系統）、傳統 EDI（電子資料交換）等領域仍在使用 — 部署這些舊系統時，Proxy 設定要特別處理 FTP 的雙連線特性（HAProxy 的 ftpsess、Squid 的 FTP gateway）。

15.2 — SSH：ProxyJump 與動態 SOCKS

SSH（OpenSSH 等實作）內建三種代理模式：

Local forward（-L）

在本地開 port，把流量轉發到 SSH server 端能到達的某個目標：

# 本地 8080 port → 透過 ssh server bastion → 到達內網 db.internal:5432
ssh -L 8080:db.internal:5432 user@bastion
# 然後本地連 localhost:8080 就等於連 db.internal:5432

Dynamic SOCKS（-D）

本地起一個 SOCKS5 server，把所有流量透過 SSH 通道送到 SSH server 端：

ssh -D 1080 user@bastion
# 任何 SOCKS5 client 連 localhost:1080 都會經過 SSH 通道

這是工程師最愛的「臨時 VPN」 — 不需設定，一行命令就有翻牆隧道。

ProxyJump（-J，OpenSSH 7.3+, 2016）

取代舊的 ProxyCommand，更乾淨：

# 透過 bastion 連到內網 server
ssh -J user@bastion user@internal-server
# 多層跳板
ssh -J u@hop1,u@hop2,u@hop3 u@target

SSH 的代理生態極為豐富，是運維工程師日常工作的核心。Part III Ch.26 會展開 SSH 的所有代理模式。

15.3 — SMTP：Mail Relay 與信任鏈

電子郵件的本質就是 relay 鏈 — 從你的 client → 你公司 SMTP → 中繼 → 收件方 SMTP → 收件人 inbox。每一跳都是 SMTP relay（也就是某種代理）。

關鍵設計：

  • Open relay 是大忌：1990 年代允許任何人透過你的 SMTP 送信，導致垃圾郵件災難。今日 SMTP server 必須做 SPF / DKIM / DMARC 驗證
  • Submission port 587：用戶送信用 587（須 STARTTLS + auth），server 之間 relay 用 25
  • 反垃圾代理：SpamAssassin、rspamd、Cloudflare Email Routing 都是 SMTP 層的 inspecting proxy

2026 年 email 仍是主要通訊渠道，但「代理 SMTP」這個工作多由托管服務（Google Workspace、Microsoft 365、SendGrid、Postmark）處理 — 自建 mail server 已罕見。SMTP 在工程師日常中變成「黑盒子」。

15.4 — SIP / RTP：VoIP 的特殊挑戰

SIP（Session Initiation Protocol，RFC 3261）是 VoIP 的信令協定。它與 RTP（Real-time Transport Protocol，UDP 媒體流）配合使用 — SIP 協商「在 IP X port Y 用 codec Z 開 RTP 流」，雙方接著用該 IP/port 直接傳語音。

對 NAT、防火牆、Proxy 而言這是噩夢：

  • SIP 訊息內含 IP/port（在 SDP 體內），NAT 不能只改 IP header，要解析 SDP 改寫
  • RTP 是 UDP，沒有「連線」概念，難以被傳統 stateful firewall 追蹤
  • 媒體流可能跑在隨機 UDP port range（10000-20000），防火牆要全開

解法：

  • SBC（Session Border Controller）：VoIP 領域的特殊 proxy，同時處理 SIP 信令改寫與 RTP 媒體中繼
  • STUN / TURN / ICE：WebRTC 的 NAT 穿透三件套，TURN 本質就是 RTP 媒體 relay
  • WebRTC SFU（Selective Forwarding Unit）：群組通話的 RTP 媒體 proxy

WebRTC 技術棧大幅取代了傳統 SIP — Zoom、Google Meet、Teams 都基於 WebRTC + 自家 SFU。但 SIP 在電信骨幹仍是主流。

15.5 — 資料庫協定：專屬代理生態

15.6 — IoT 協定：MQTT 與專用代理

MQTT（Message Queuing Telemetry Transport）是 IoT 的標準協定。它是 publish-subscribe 模型，不是 request-response，所以「代理」的概念與 HTTP 完全不同：

  • Broker（如 Mosquitto、HiveMQ、EMQ X）本身就是代理 — 接收 publisher 的訊息、轉發給 subscriber
  • Bridge：Broker 之間的橋接（如把工廠 broker 橋到雲端 broker）
  • IoT Gateway（Greengrass、Azure IoT Edge）：邊緣 broker + 雲端 broker 之間的雙向代理

MQTT 5.0（2019）加入了 enhanced authentication、shared subscriptions、reason codes 等企業級特性。2026 年的 IoT 部署中，MQTT broker 是核心元件。

15.7 — 觀察：「代理」概念的擴張

讀完本章，你會發現「代理」這個概念在不同領域有完全不同的形態：

  • HTTP Proxy 是「請求-回應的中介」
  • SOCKS5 是「TCP / UDP 的純通道」
  • VPN 是「網路平面的延伸」
  • SBC 是「SIP 信令改寫 + RTP 媒體中繼」
  • PgBouncer 是「資料庫連線池與多工」
  • MQTT broker 是「pub/sub 訊息分發中心」

它們唯一的共通點 — 還記得 Ch.4 §4.1 的定義嗎？「代表某一方與另一方互動的中介者」。從這個抽象定義出發，所有上述產品都是「代理」。但實作機制天差地別。

Part II 接下來最後一章（Ch.16）會把 SOCKS5、HTTP、TLS 三個我們已深入討論的協定整合到一個產品 — 一個能自動偵測協定的多協定 Proxy，讓你看到實作整合的工程細節。

本章要點

「代理」是個高度抽象的概念 — FTP 雙連線、SSH ProxyJump/動態 SOCKS、SMTP relay 鏈、SIP/RTP 的 SBC、資料庫的 PgBouncer/ProxySQL、IoT 的 MQTT broker，每個領域都有自己的代理形態。理解 Ch.4 §4.1 的抽象定義（「代表某一方與另一方互動的中介者」）後，能看穿這些表面差異。

綜合實作：從零打造多協定 Proxy

Part II 的壓軸實作。把前八章的協定知識整合成一個真實工具 — 用 Go 寫一個能在同一個 port 上自動偵測 SOCKS、HTTP、TLS 流量並分流處理的多協定 Proxy。

讀完 Ch.9–15，你已經知道每個協定的 byte-level 細節。但在真實 Proxy 部署中，一個有趣的需求經常出現：能不能讓同一個 port 同時接受多種協定？例如一台跳板機開 :443，能同時當 HTTPS Reverse Proxy 又當 SOCKS5 server。這個需求看似違反協定設計（每個協定都有「指定 port」），但實際上完全可行 — 只要看第一個 byte 就能分辨。

0x47 (G)HTTP GET"GET /..." 開頭
0x50 (P)HTTP/2 preface 或 HTTP POST/PUT/PATCH"PRI * HTTP/2.0..." 或 "POST /..."
0x43 (C)HTTP CONNECT"CONNECT host:port..."
0x44 (D)HTTP DELETE"DELETE /..."
0x4F (O)HTTP OPTIONS"OPTIONS /..."
0x48 (H)HTTP HEAD"HEAD /..."

HTTP/2 的 0x50 與 HTTP POST 衝突 — 解法是讀更多 byte確認。HTTP/2 完整 preface 是 24 bytes（PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n），與任何 HTTP/1.1 method 都不同。

這個策略的關鍵問題：讀過的 byte 不能丟掉，因為後續的協定處理器需要從頭開始解析。Go 的 bufio.Reader 提供 Peek(n) 完美解決這個問題 — 看 byte 但不消耗。

16.2 — 完整 Go 實作

我們的目標是一個單一 port 上同時支援 SOCKS5、HTTP（plain）、TLS（passthrough）的 Proxy。為了篇幅簡化，TLS 我們做 SNI-based passthrough（不解密），HTTP 做 plain HTTP forwarding，SOCKS5 用我們在 Ch.9 §9.8 寫的邏輯。

架構分三層：

  1. Listener：accept 連線
  2. Sniffer：peek 第一個 byte 判斷協定
  3. Handler：依協定 dispatch 到對應處理函式

主程式：

main.go — Listener 與 dispatchpackage main

import (
    "bufio"
    "io"
    "log"
    "net"
)

func main() {
    ln, err := net.Listen("tcp", ":8443")
    if err != nil {
        log.Fatal(err)
    }
    log.Println("multi-protocol proxy on :8443")
    for {
        c, err := ln.Accept()
        if err != nil {
            log.Println("accept:", err)
            continue
        }
        go dispatch(c)
    }
}

func dispatch(c net.Conn) {
    defer c.Close()
    br := bufio.NewReader(c)
    // Peek 第一個 byte 不消耗
    head, err := br.Peek(1)
    if err != nil {
        return
    }
    switch head[0] {
    case 0x05:
        log.Println("→ SOCKS5")
        handleSOCKS5(c, br)
    case 0x16:
        log.Println("→ TLS (SNI passthrough)")
        handleTLS(c, br)
    case 'G', 'P', 'H', 'D', 'O', 'C', 'T':
        log.Println("→ HTTP")
        handleHTTP(c, br)
    default:
        log.Printf("unknown protocol, first byte=0x%02x", head[0])
    }
}

// pipe 雙向中繼 byte stream
func pipe(a, b net.Conn) {
    done := make(chan struct{}, 2)
    go func() { io.Copy(a, b); done <- struct{}{} }()
    go func() { io.Copy(b, a); done <- struct{}{} }()
    <-done
}

SOCKS5 處理器 — 套用 Ch.9 §9.8 的邏輯，加上 bufio 處理：

socks5.gopackage main

import (
    "bufio"
    "encoding/binary"
    "net"
    "strconv"
)

func handleSOCKS5(c net.Conn, br *bufio.Reader) {
    // Phase 1: 讀 VER + NMETHODS + METHODS
    hdr := make([]byte, 2)
    if _, err := io.ReadFull(br, hdr); err != nil {
        return
    }
    nmethods := int(hdr[1])
    methods := make([]byte, nmethods)
    if _, err := io.ReadFull(br, methods); err != nil {
        return
    }
    // 接受 no-auth (0x00)
    if !contains(methods, 0x00) {
        c.Write([]byte{0x05, 0xFF})
        return
    }
    c.Write([]byte{0x05, 0x00})

    // Phase 3: VER + CMD + RSV + ATYP
    req := make([]byte, 4)
    if _, err := io.ReadFull(br, req); err != nil {
        return
    }
    if req[1] != 0x01 { // 只支援 CONNECT
        c.Write([]byte{0x05, 0x07, 0, 0x01, 0, 0, 0, 0, 0, 0})
        return
    }
    var host string
    switch req[3] {
    case 0x01: // IPv4
        ip := make([]byte, 4)
        io.ReadFull(br, ip)
        host = net.IP(ip).String()
    case 0x03: // 域名
        ln, _ := br.ReadByte()
        name := make([]byte, ln)
        io.ReadFull(br, name)
        host = string(name)
    default:
        return
    }
    pb := make([]byte, 2)
    io.ReadFull(br, pb)
    port := binary.BigEndian.Uint16(pb)

    // 連到目標
    target, err := net.Dial("tcp", host+":"+strconv.Itoa(int(port)))
    if err != nil {
        c.Write([]byte{0x05, 0x05, 0, 0x01, 0, 0, 0, 0, 0, 0})
        return
    }
    defer target.Close()
    // 回成功（BND.ADDR 用 0.0.0.0）
    c.Write([]byte{0x05, 0x00, 0, 0x01, 0, 0, 0, 0, 0, 0})
    // 純中繼
    pipe(c, target)
}

func contains(b []byte, v byte) bool {
    for _, x := range b {
        if x == v { return true }
    }
    return false
}

HTTP 處理器 — 解析 absolute-URI 與 CONNECT：

http.gopackage main

import (
    "bufio"
    "net"
    "net/http"
)

func handleHTTP(c net.Conn, br *bufio.Reader) {
    req, err := http.ReadRequest(br)
    if err != nil {
        return
    }
    if req.Method == "CONNECT" {
        // HTTPS via Forward Proxy
        target, err := net.Dial("tcp", req.Host)
        if err != nil {
            c.Write([]byte("HTTP/1.1 502 Bad Gateway\r\n\r\n"))
            return
        }
        defer target.Close()
        c.Write([]byte("HTTP/1.1 200 Connection Established\r\n\r\n"))
        pipe(c, target)
        return
    }
    // 一般 HTTP — req.URL 是 absolute URI（因為 client 用 forward proxy 模式送）
    target, err := net.Dial("tcp", req.URL.Host+":"+portOf(req.URL))
    if err != nil {
        c.Write([]byte("HTTP/1.1 502 Bad Gateway\r\n\r\n"))
        return
    }
    defer target.Close()
    // 改寫 request 為 origin-form 後送出
    req.RequestURI = ""
    req.URL.Scheme = ""
    req.URL.Host = ""
    req.Write(target)
    // 把 backend 回應原樣送回
    io.Copy(c, target)
}

func portOf(u *url.URL) string {
    if u.Port() != "" { return u.Port() }
    if u.Scheme == "https" { return "443" }
    return "80"
}

TLS 處理器 — SNI passthrough（看 SNI 但不解密，路由到對應後端）：

tls.go — SNI 解析與路由package main

import (
    "bufio"
    "encoding/binary"
    "errors"
    "io"
    "net"
)

func handleTLS(c net.Conn, br *bufio.Reader) {
    // Peek 整個 TLS record（最多 5+ClientHello bytes）
    hdr, err := br.Peek(5)
    if err != nil { return }
    // hdr[0]=0x16, hdr[1..2]=TLS version, hdr[3..4]=length
    recordLen := int(binary.BigEndian.Uint16(hdr[3:5]))
    full, err := br.Peek(5 + recordLen)
    if err != nil { return }

    sni, err := extractSNI(full[5:])
    if err != nil {
        log.Println("no SNI:", err)
        return
    }
    log.Println("SNI:", sni)

    // 連到目標（直接用 SNI 名稱）
    target, err := net.Dial("tcp", sni+":443")
    if err != nil { return }
    defer target.Close()
    // 純轉送（包括 ClientHello —— 這是關鍵：保留 client 的 TLS 指紋）
    pipe(c, target)
}

// extractSNI 解析 TLS ClientHello 取出 SNI hostname
func extractSNI(ch []byte) (string, error) {
    // 跳過 handshake header (4) + version (2) + random (32)
    p := 4 + 2 + 32
    if p+1 > len(ch) { return "", errors.New("short") }
    p += 1 + int(ch[p])             // session_id
    if p+2 > len(ch) { return "", errors.New("short") }
    p += 2 + int(binary.BigEndian.Uint16(ch[p:p+2])) // cipher_suites
    if p+1 > len(ch) { return "", errors.New("short") }
    p += 1 + int(ch[p])             // compression_methods
    if p+2 > len(ch) { return "", errors.New("short") }
    extLen := int(binary.BigEndian.Uint16(ch[p:p+2]))
    p += 2
    if p+extLen > len(ch) { return "", errors.New("short") }
    end := p + extLen
    for p+4 <= end {
        extType := binary.BigEndian.Uint16(ch[p:p+2])
        thisLen := int(binary.BigEndian.Uint16(ch[p+2:p+4]))
        p += 4
        if extType == 0x0000 { // SNI
            // SNI 結構：list_len(2) + name_type(1) + name_len(2) + name
            if p+5 > len(ch) { return "", errors.New("short SNI") }
            nameLen := int(binary.BigEndian.Uint16(ch[p+3:p+5]))
            if p+5+nameLen > len(ch) { return "", errors.New("short SNI") }
            return string(ch[p+5:p+5+nameLen]), nil
        }
        p += thisLen
    }
    return "", errors.New("no SNI extension") }

注意 TLS 處理器的特殊性：它不解密、不終止 TLS。我們只解析 ClientHello 取出 SNI 做路由決策，然後把原始 ClientHello byte（透過 br 的 buffered 內容）連同後續 TLS 流量一起轉送給後端。這保留了 client 的真實 JA3/JA4 指紋（cf. Ch.9 §9.9 與 Ch.11 §11.5）。

16.4 — 真實世界的延伸

本章的 200 行 Go 程式只是骨架。Production-grade 的多協定 Proxy（如 Xray、shoes、turnout）還會處理：

  • 更多協定：HTTP/2 cleartext、WebSocket、QUIC、Shadowsocks、VMess
  • 認證與權限：每個協定有自己的 auth 機制
  • 路由規則：基於 SNI、域名、地理位置、用戶身分做路由
  • 連線池：對 backend 的連線重用
  • 觀測：metric、log、tracing
  • 限流與防護：rate limit、connection limit、DDoS 防護
  • 協定混淆：抗審查環境下偽裝成其他流量
  • TLS termination 模式：不只 passthrough，還能解密做更深 inspecting

但所有這些都建立在本章展示的核心機制上 — peek 第一個 byte，dispatch 到對應 handler，純通道做中繼。一個概念，多種應用。
