Preface · 序

為什麼還需要一本代理技術書

在 ChatGPT 寫得出「能跑的 Nginx 配置」的時代，寫一本系統性的代理技術書，意義何在？

代理（Proxy）是電腦網路最古老的概念之一。1990 年代 CERN httpd 加入 caching 功能時，沒人會想到三十多年後，這個概念會繁衍成數十種形態 — SOCKS5、HTTP CONNECT、Reverse Proxy、Service Mesh Sidecar、API Gateway、CDN Edge、Zero Trust、MCP Server、A2A Protocol — 並貫穿從 Linux kernel eBPF 到雲端 Worker、從個人翻牆工具到億級流量企業架構的每一個層次。每一種代理形態都是針對某個具體問題的回應，但它們共享的本質卻在資訊碎片化的時代被淡忘了。

本書的寫作動機很單純：繁體中文界沒有一本系統性、面向 2026 年現況的代理技術書。網路上有零散的 Nginx 設定教學、Stack Overflow 上有上千個 SOCKS5 連不上的問題、英文社群有 Cilium 的 deep dive；但很少有資源把整個代理技術光譜串起來，讓讀者既看得到細節（SOCKS5 封包的每個 byte），也理解大局（為什麼 2026 年的 AI Agent 需要 MCP 而不是傳統 API Gateway）。

這本書試圖填補這個空缺。從 OSI 分層的觀念建構開始，到 70 章後以「反模式與設計陷阱」收尾，本書涵蓋：協定握手的 byte-level 細節、Nginx／HAProxy／Envoy 的生產級配置、Squid SSL Bump 的法律邊界、Tor 洋蔥路由的密碼學設計、Reality 與 Hysteria 等 2024 年後的抗審查協定、MCP（2024 年 11 月由 Anthropic 推出，2025 年 12 月轉移至 Linux Foundation 旗下 AAIF）、Istio Ambient Mesh 與 Cilium eBPF mesh 的「Sidecar 之爭」、AI Gateway 與 LLM Token 路由、紅隊穿透 Proxy 的真實 CVE 與 APT 案例。

本書採學術 × 工程混合風格 — 嚴謹引用 RFC 與學術論文，同時提供大量可直接複製使用的生產級配置與程式碼。每個重點章節（Part II–V）都有 hands-on 實作；附錄四篇速查表彙編、工具大全、RFC 索引、術語對照，是日常工作的查詢手冊。

讀者群預設四種角色：網路工程師 / SRE、後端 / 全端開發者、資安 / 紅藍隊、AI Agent 開發者。每種角色都有專屬的閱讀路徑（見下方），但作者真心建議：如果時間允許，從序言一路讀到結語，因為許多看似不相關的章節之間，其實有著驚人的呼應。

最後一句話送給讀者：代理的本質是「中介」，而中介的本質是「翻譯」。當你能在 SOCKS5 server 看見 Nginx，在 MCP Server 看見 1996 年的 CERN httpd，在 Cilium 的 eBPF program 看見 1992 年的 SOCKS 協定 — 你就真的懂代理了。
