Chapter 47

MASQUE 全解構
CONNECT-UDP、CONNECT-IP、巢狀隧道與 Apple iCloud Private Relay

2024-2026 年最重要的 proxy 技術躍進是 MASQUE — 把 HTTP 從「能 proxy TCP」擴展到「能 proxy UDP / IP / Ethernet」。本章解構 IETF MASQUE WG、RFC 9298 (CONNECT-UDP)、RFC 9484 (CONNECT-IP)、RFC 9221 (HTTP Datagrams)，以及 Apple iCloud Private Relay 與 Cloudflare WARP 的真實部署。

Part I-V 大量討論了 reverse proxy、forward proxy、AI Gateway。但有個結構性限制一直存在：HTTP 的 CONNECT 方法只能 proxy TCP。當 QUIC（UDP-based）成為主流、WebRTC 普及、IoT 與 VPN 場景需要 IP 級 tunnel 時，傳統 CONNECT 不夠用。MASQUE（Multiplexed Application Substrate over QUIC Encryption）是 IETF 對這個問題的回應 — 一套讓 HTTP 能 proxy UDP、IP、甚至 Ethernet frame 的協定家族。本章是 Part VI 的起點，幫你理解 MASQUE 為何重要、怎麼運作、以及 Apple、Cloudflare 等已經在用它做什麼。

47.1 — 為何需要 MASQUE：CONNECT 的限制

HTTP CONNECT 方法（RFC 2817, 2000）原本是為 TLS 而設計 — 讓 forward proxy 能盲目轉發 TLS 流量到目標伺服器。一旦 CONNECT tunnel 建立，proxy 只能看到加密的 byte stream，無法解析 TLS 內容。CONNECT 也成為 HTTP 唯一能proxy 任意 TCP 協定的機制。

CONNECT 的三大限制

  • 只能 proxy TCP：UDP、IP、Ethernet frame 都不行
  • 每個目標一個 TCP 連線：無法在單一 HTTP 連線上多工多個 tunnel
  • 對 QUIC 不友好：QUIC 跑在 UDP 上，CONNECT 無法直接 proxy

QUIC / HTTP/3 改變一切

2022 年 HTTP/3（RFC 9114）標準化。HTTP/3 跑在 QUIC 上，QUIC 跑在 UDP 上。當 web 流量從 TCP 轉向 UDP，proxy 工程師面對尷尬問題：

  • 用 CONNECT 把 QUIC 當「UDP datagram」proxy？CONNECT 不支援 UDP
  • 把 QUIC tunneled in TCP？破壞 QUIC 的設計（congestion control 干擾）
  • 用 SOCKS5？舊協定、不在 HTTP 生態

解法：讓 HTTP 本身能 proxy UDP。MASQUE 應運而生。

47.3 — HTTP Datagrams 與 Capsule Protocol（RFC 9297）

MASQUE 的基礎是HTTP Datagrams — 在 HTTP 層提供「不可靠訊息」抽象。傳統 HTTP 訊息是 reliable byte stream（為 TCP 設計）；HTTP Datagrams 加入不可靠 message-oriented 替代。

為何需要不可靠

UDP 流量（如 QUIC、DNS、WebRTC、即時音視訊）本來就不需要可靠。把它們 tunnel 在 reliable HTTP stream 中會：

  • 引入不必要的重傳延遲
  • congestion control 雙層干擾（外層 + 內層 QUIC）
  • head-of-line blocking

HTTP Datagrams 透過HTTP/3 + QUIC DATAGRAM frame解決：DATAGRAM 不會被 retransmit，丟了就丟了，與 UDP 行為一致。對 HTTP/1.1 與 HTTP/2，HTTP Datagrams 透過 Capsule Protocol退回到 reliable stream（次優但能工作）。

Capsule Protocol

Capsule 是 HTTP body 中的「結構化訊息」。每個 capsule 有 type + length + value（TLV 編碼）。CONNECT-UDP / CONNECT-IP 用 capsule 傳遞控制訊息（如建立 UDP association、調整 route）。

Capsule 結構（簡化）┌──────────────┬──────────────┬─────────────────┐
│  Type        │  Length      │  Value          │
│  (varint)    │  (varint)    │  (variable)     │
└──────────────┴──────────────┴─────────────────┘

範例 capsule type:
  • DATAGRAM           — 包裝 UDP/IP datagram
  • COMPRESSION_ASSIGN — 註冊 context ID 用於壓縮
  • COMPRESSION_ACK    — 確認 context ID
  • ROUTE_ADVERTISEMENT — 通告 IP route
  • ADDRESS_ASSIGN     — 分配 IP address

47.4 — CONNECT-UDP（RFC 9298）

CONNECT-UDP 讓 client 透過 HTTP 請求 proxy 開啟一個 UDP socket 到目標伺服器，後續 UDP datagram 透過 HTTP Datagram 雙向傳遞。

建立 UDP tunnel 的 HTTP/3 範例

CONNECT-UDP 請求（HTTP/3）# Client 發出 Extended CONNECT
HEADERS frame:
  :method = CONNECT
  :protocol = connect-udp
  :scheme = https
  :path = /.well-known/masque/udp/192.0.2.1/443/
  :authority = proxy.example.org
  capsule-protocol = ?1

# Proxy 回應
HEADERS frame:
  :status = 200
  capsule-protocol = ?1

# 從此 Client 與 Proxy 之間用 HTTP DATAGRAM frame 雙向傳 UDP payload
# Proxy 把 DATAGRAM payload 用真實 UDP socket 送到 192.0.2.1:443
# 反向亦然：UDP 回應包成 DATAGRAM frame 送回 client

關鍵設計

  • URI Template：/.well-known/masque/udp/{host}/{port}/明示目標
  • Extended CONNECT（RFC 8441 / 9220）：HTTP/2 與 HTTP/3 的擴展
  • HTTP Datagram：UDP payload 用 DATAGRAM frame 傳，不是 reliable stream
  • 每個 stream 一個 UDP association：HTTP/2/3 多工讓單一連線能開多個 UDP tunnel
  • HTTP/1.1 也支援（用 HTTP Upgrade + Capsule Protocol，效能較差）

47.5 — CONNECT-IP（RFC 9484）

CONNECT-IP 把 proxy 範圍從 UDP 擴展到整個 IP 層。Client 不指定目標 host:port，而是請求 proxy 處理任意 IP 封包 — 這基本上是「HTTP-based VPN」。

使用場景

  • Remote access VPN：取代 OpenVPN、IPsec、WireGuard
  • Site-to-site VPN
  • Secure point-to-point communication
  • 通用封包 tunnel

CONNECT-IP 的核心 capsule

  • ADDRESS_ASSIGN：proxy 分配 IP 給 client
  • ROUTE_ADVERTISEMENT：proxy 通告哪些路由可達
  • DATAGRAM：包裝 IPv4 / IPv6 封包

Client 收到 ADDRESS_ASSIGN 後設定 virtual interface（如 TUN device），把整個系統流量導向這個介面。Proxy 端則把收到的 IP 封包實際發送出去，回應再包回 DATAGRAM。這就是 VPN。

47.6 — 巢狀 MASQUE 隧道：partition 信任

MASQUE 最強大的設計之一：巢狀 tunnel — 流量穿過多個 MASQUE proxy，每個 proxy 只看到部分資訊。

關鍵性質：

  • Proxy A 看見：Alice 的真實 IP，但只看到「她在連 Proxy B」 — 不知她最終要去 web.com
  • Proxy B 看見：目標是 web.com，但只看到「請求來自 Proxy A」 — 不知 Alice 是誰
  • web.com 看見：來源 IP 是 Proxy B，與 Alice 完全無關

如果 Proxy A 與 Proxy B 是非合謀（non-colluding）的不同營運者，則沒有任何單一實體能同時看到「誰」與「去哪」。這是 MASQUE 給隱私的根本貢獻。

47.7 — Apple iCloud Private Relay：MASQUE-like 雙代理

2021 年 WWDC，Apple 公布 iCloud Private Relay。架構與 MASQUE 巢狀 tunnel 高度相似：

  • Ingress proxy（Apple）：看 Alice IP，把流量轉給 egress proxy
  • Egress proxy（合作 CDN：Akamai、Cloudflare、Fastly）：看目標網址，但收到的來源是 Apple proxy
  • iOS / macOS Safari 流量自動透過此架構

Apple 部署時 MASQUE 規格還未 RFC 化，所以實際協定不完全是 MASQUE，但設計理念與架構與 MASQUE 一致。Apple 的工程師（包括 Tommy Pauly、Eric Kinnear）也是 MASQUE WG 主要貢獻者。

Private Relay 的實際影響

  • 2022 年起大量 iOS 使用者預設啟用 Private Relay
  • 網站看到的「Apple 來源 IP」其實是匿名化結果
  • 影響 IP-based rate limit、fraud detection、geolocation
  • 對網站營運者是新挑戰：怎麼處理已知是合法但 IP 不是真實的流量

47.8 — Cloudflare WARP 與其他部署

Cloudflare WARP（2025-2026 過渡到 MASQUE）

Cloudflare WARP 是 Cloudflare 1.1.1.1 DNS resolver 內建的 VPN 功能，原本用 WireGuard 變種協定。2025-2026 年 Cloudflare 把 WARP 過渡到 MASQUE — 用 CONNECT-IP over HTTP/3。

  • WARP MASQUE 整合預設啟用 hybrid X25519MLKEM768（cf. Ch.49）
  • 2025/12 加入 PQC，2026 年中進入 Phase 2（拒絕 downgrade）
  • Cloudflare 也是 Apple Private Relay 的 egress proxy 合作方

Google：MASQUE 主要推手

David Schinazi（Google）是 RFC 9298、9484、9729 的主作者。Google 的動機明確：

  • Chrome 是全球最大瀏覽器，控制 client side
  • Google Cloud / YouTube / Search 控制大量 server side
  • QUIC 是 Google 推動的標準（2013 開始 QUIC，2021 RFC 9000）
  • MASQUE 是 QUIC 生態的自然延伸

其他實作

  • quiche（Rust，Cloudflare）：production-grade QUIC + MASQUE

  • OpenSSL 3.x：QUIC 支援，MASQUE 仍待整合

47.9 — Concealed Authentication（RFC 9729, 2025/02）

傳統 proxy 的致命弱點：能被網路偵測到。即使 traffic 被加密，「有 proxy 流量」這件事本身就暴露 — 國家級審查、企業 DLP、ISP 限速都能利用這個 metadata。

Concealed Authentication 的目標

讓 MASQUE proxy 與普通 web server 「無法區分」。如果你能連到 proxy，你就能用它；如果你不知道 proxy 存在，你看到的是普通網站。

  • Client 不能用「標準 Authorization header」要求 proxy（會暴露 proxy 性質）
  • 改用HMAC-based 認證：client 在 ClientHello 的 ALPS 或某個 header 中嵌入 keyed MAC
  • Server 計算同樣 MAC，匹配就提供 proxy 服務，不匹配就當普通網站
  • 沒有 keyed MAC 的探測者看到的是「普通網站行為」

這個設計讓任何 web server 都能悄悄同時是 MASQUE proxy。對審查者，找出 proxy 變得困難 — 你必須事先知道密鑰才能驗證某個域名是不是 proxy。

Chapter 48

HTTP/3-only Proxy
當 reverse proxy 完全跳過 TCP — 0-RTT、connection migration 與 UDP 防火牆現實

Ch.10 §10.7-10.10 介紹 HTTP/3 與 QUIC。本章探討一個更激進的問題：能不能完全不要 TCP？「HTTP/3-only proxy」的設計權衡 — 0-RTT、connection migration 的實戰、UDP 在企業防火牆的現實困境，以及為何 2026 年仍要保留 HTTP/2 fallback。

Ch.10 介紹了 HTTP/3 的技術。本章是個架構思考實驗：能不能架設一個完全不支援 TCP / HTTP/2 的 proxy？ 這叫「HTTP/3-only proxy」。理論上可行 — Cloudflare、Google、Apple 內部都有部署。但現實有諸多挑戰。本章解構這些挑戰，幫你判斷自己場景是否該做 HTTP/3-only。

48.1 — HTTP/3 的核心優勢回顧

Ch.10 已詳述。簡短回顧 reverse proxy 視角下 HTTP/3 比 HTTP/2 的優勢：

  • 真正的 stream 多工：QUIC 在 transport 層提供獨立 stream，徹底消除 HTTP/2 head-of-line blocking
  • 0-RTT：再次連到同一 server 時資料能跟著第一個 packet 走，省一個 RTT
  • Connection migration：用 connection ID 而非 4-tuple 識別連線，IP 換了不斷線（手機切換 WiFi / 4G）
  • 內建 TLS 1.3：handshake 與加密合一，省一次 round trip
  • 更好的 congestion control：QUIC 在 user space，可用 BBR、CUBIC 自由切換
  • Forward error correction（潛力）：QUIC 設計允許未來加入 FEC

48.2 — 「HTTP/3-only」是什麼意思

HTTP/3-only proxy 的嚴格定義：不接受 TCP 連線，client 必須用 QUIC / HTTP/3 才能存取。

三種「HTTP/3 部署」程度

程度說明2026 普及度
Opt-in HTTP/3同時提供 H1/H2/H3，client 用 Alt-Svc 升級到 H3主流（Cloudflare、Google、Facebook、CDN 通用）
HTTP/3 preferred主推 H3，但保留 H2 fallback（無 Alt-Svc 也能用 H2）常見
HTTP/3-only完全不接 TCP，沒 H3 client 進不來罕見，僅內部 / 實驗 / 特殊場景

2026 年大多數網站處於「Opt-in HTTP/3」— Alt-Svc 廣告 H3，但 H1/H2 仍然可用。HTTP/3-only 是未來方向，但仍有重大障礙。

48.3 — 0-RTT：機會與陷阱

0-RTT 是 HTTP/3 最被討論的優勢。它讓 reconnecting client 能在第一個 packet中送資料，省掉一次 RTT。

0-RTT 的工作原理

  1. 第一次連線：正常 TLS 1.3 handshake，server 給 client 一個 session ticket
  2. 後續連線：client 用 session ticket 加密第一個 packet（含應用層資料）
  3. Server 驗證 ticket，立即處理應用層資料 — handshake 同時完成

0-RTT 的安全陷阱：Replay Attack

0-RTT 資料不在 Forward Secrecy 保護中，且攻擊者能重放。RFC 9001 (QUIC) 與 RFC 8446 (TLS 1.3) 明確規定：

  • 0-RTT 只能用於 idempotent 請求（GET、HEAD）
  • POST、PUT、DELETE 絕對不能用 0-RTT
  • Server 必須有 anti-replay 機制（time window + nonce）
  • 實作要 explicit allow-list 哪些 endpoint 可 0-RTT

真實案例：Cloudflare 的 0-RTT 限制

Cloudflare 對 0-RTT 的處理是業界謹慎標桿：

  • 預設關閉 0-RTT
  • 客戶手動 opt-in
  • 只對 GET / HEAD 啟用
  • 對 reorigin 連線一律 disable 0-RTT

這個保守態度反映了 0-RTT 的根本性質：性能優勢真實但有限，安全風險真實且嚴重。對 reverse proxy 工程師：啟用 0-RTT 前要確保 application layer 對 replay 安全。

48.4 — Connection Migration：手機時代的關鍵

HTTP/3 / QUIC 的另一大優勢是 connection migration — IP 換了不斷線。對行動使用者意義重大。

場景

使用者在咖啡店連 WiFi 看 YouTube，離開時切換 4G。

  • HTTP/2 over TCP：4-tuple (src IP, src port, dst IP, dst port) 改變 → TCP 連線斷 → app 重連 → 影片 buffer → 體驗差
  • HTTP/3 over QUIC：QUIC 用 64-bit connection ID 識別 connection。IP 換了，connection ID 不變 → server 收到新 IP 的 packet 認得是同一連線 → 連線繼續

實作挑戰

  • Path validation：server 必須驗證新 IP 不是攻擊者偽造（PATH_CHALLENGE / PATH_RESPONSE frame）
  • Address spoofing 防禦：避免 server 被誘導打 amplification 流量到無辜目標
  • Stateful firewall / NAT：許多企業 firewall 用 4-tuple 維護 state，IP 換了 firewall 不認得
  • Load balancer connection ID 路由：LB 必須能根據 connection ID（而非 4-tuple）把 packet 送到正確 backend

QUIC LB（draft-ietf-quic-load-balancers）

解 LB 路由問題的 IETF draft：在 connection ID 中編碼 server identifier，LB 能解析後路由。Cloudflare、Google 都有自家實作。對自架 reverse proxy + HTTP/3，必須處理這個問題（HAProxy、Nginx 的 QUIC 支援都包含此邏輯）。

48.5 — UDP 在企業防火牆的現實

HTTP/3-only 部署的最大障礙：UDP 在許多企業網路被封鎖或限速。

為何 UDP 不受歡迎

  • 歷史包袱：UDP 過去被 game / DNS amplification 攻擊濫用，許多企業預設封鎖非必要 UDP
  • Stateful firewall 不友好：UDP 沒 connection 概念，stateful firewall 用 timeout 模擬，常 break QUIC connection migration
  • DPI 困難：QUIC 加密 transport metadata，企業 DLP / IDS 看不到內容
  • 反映限速規則：許多 ISP / 企業對 UDP 套用更嚴格 rate limit

真實數據

  • 2024 年 Google 報告：~5-10% 全球連線 UDP 443 被封鎖（依地區差異大）
  • 中國 GFW 對 QUIC（UDP 443）主動干擾，許多場景需 fallback 到 TCP
  • 企業 SASE / proxy 預設常封 UDP，要 IT 團隊 explicit allow
  • 飯店 / 機場 WiFi 經常封鎖 UDP 443

結果：HTTP/3 必須有 H2 fallback，否則 5-10% 使用者進不來。對面向消費者的 reverse proxy，HTTP/3-only 是商業自殺。

48.6 — 何時可以 HTTP/3-only

儘管限制，有些場景 HTTP/3-only 完全合理：

場景 1：內部服務 + 控制 client

如果你同時控制 client 與 server（內部 microservice、行動 app、自家 client），且確認所有 client 網路允許 UDP，可以 HTTP/3-only。例：

  • 內部 microservice 通訊（service mesh 場景）
  • 自家行動 app + 自家後端
  • 容器內 sidecar proxy 通訊

場景 2：MASQUE proxy（cf. Ch.47）

MASQUE proxy 本質就是 HTTP/3-only — 整個概念基於 HTTP/3 + QUIC DATAGRAM。MASQUE proxy 不做 TCP fallback。

場景 3：實驗 / 研究

學術測試、研究新 congestion control 演算法、QUIC 性能研究。

場景 4：私有網路 / VPN 內部

Tailscale、ZeroTier、WireGuard 之類私有 mesh 網路內部能保證 UDP 可達，HTTP/3-only 可行。

48.8 — 給 reverse proxy 工程師的 5 條建議

  1. 面向公網服務：opt-in HTTP/3 + H2 fallback。Alt-Svc 廣告 H3，client 自動升級。保留 H2 給 5-10% UDP 不通的使用者
  2. 內部服務 + 控制 client：考慮 HTTP/3-only。性能、connection migration、0-RTT 是真實價值
  3. 0-RTT 預設關閉。需要時 explicit opt-in，限 GET/HEAD，application layer 要 anti-replay
  4. 監控 H3 / H2 流量比例。當 H3 占比超過 95%，可考慮逐步 deprecate H2
  5. QUIC LB 必須處理 connection ID routing。不能假設 4-tuple 穩定，要看 connection ID
本章要點
  • HTTP/3-only proxy是「完全不接 TCP」的激進部署，2026 年仍罕見。
  • 三種程度：opt-in HTTP/3（主流）/ HTTP/3 preferred / HTTP/3-only（特殊場景）。
  • 0-RTT能省一個 RTT 但有 replay 風險，只能用於 idempotent 請求，Cloudflare 預設關閉。
  • Connection migration用 connection ID 識別連線，IP 換了不斷線（行動裝置關鍵），但 LB / firewall 必須處理 connection ID routing。
  • UDP 防火牆現實：5-10% 全球使用者 UDP 443 被封，企業 / 飯店 / 中國 GFW 常見，面向公網場景必須有 H2 fallback。
  • HTTP/3-only 合理場景：內部服務 + 控制 client、MASQUE proxy（cf. Ch.47）、實驗、私有網路內部。
  • 主流 reverse proxy 全部支援 HTTP/3（Nginx 1.25+、HAProxy 2.6+、Envoy、Caddy、Traefik；Apache 例外）。
  • 5 條建議：面向公網用 opt-in、內部用 only、0-RTT 預設關閉、監控比例、處理 connection ID routing。
4 RFC 9001（QUIC）+ RFC 8446（TLS 1.3）詳細規範 0-RTT 安全要求。實作參考 Cloudflare quiche 與 Google quiche 兩個 open source library。

後量子密碼學
ML-KEM、X25519MLKEM768、harvest-now-decrypt-later 與 reverse proxy 升級之路

2024/08 NIST 標準化 ML-KEM（即原 Kyber）、2025-2026 年 X25519MLKEM768 hybrid 模式在主流瀏覽器預設啟用、Cloudflare 已對 1/3+ 流量啟用 PQC、Cloudflare WARP 2025/12 加入。本章解構 PQC 為何重要、混合模式設計、reverse proxy 工程師的升級路徑，以及為何「不升級就晚了」。

量子電腦能在數小時內破解現在所有的公鑰加密 — RSA、ECDSA、ECDH 全部。還沒有大型量子電腦能做到，但專家預估 2040-2050 年可能達成。對 reverse proxy 工程師而言，威脅不是未來的量子電腦 — 而是「harvest now, decrypt later」攻擊：對手現在截獲並儲存你的加密流量，未來用量子電腦解密。今天的機密 → 20 年後曝光。本章是 reverse proxy 工程師必修的 2026 年議題：你在 production 用的 TLS 是否已經 quantum-safe？

49.1 — 量子電腦與密碼學的基本威脅

三類密碼學在量子時代的命運

對稱加密AES-128、AES-256、ChaCha20輕微（Grover 演算法把 brute force 從 2ⁿ 降到 2^(n/2)）— AES-256 仍安全

對稱加密的安全性輕微下降但仍可用（用 AES-256 取代 AES-128 即可）。真正崩潰的是公鑰系統 — 而 TLS 的 key agreement 與 certificate 認證全部依賴公鑰系統。

Shor 演算法為何致命

Peter Shor 在 1994 年發現：量子電腦能用多項式時間解 integer factorization 與 discrete logarithm — 這正是 RSA 與 ECDH 的數學基礎。一旦有足夠 qubit且低 error rate的量子電腦：

  • RSA-2048 可在數小時內破解
  • ECDH P-256 / X25519 同樣破解
  • ECDSA 簽章可被偽造

當前最強量子電腦（IBM Condor 1,121 qubits, 2023；Google Willow 105 qubits with error correction, 2024/12）遠不夠破解 RSA-2048。但專家估計 2030-2040 年可能達成。

真實的 harvest-now 攻擊者：

49.3 — NIST 後量子密碼學標準化

2016 年 NIST 啟動全球 PQC 競賽，2024/08 完成第一輪標準化：

NIST 標準原名用途核心數學
FIPS 203（ML-KEM）CRYSTALS-KyberKey Encapsulation（取代 ECDH）Module Learning With Errors
FIPS 204（ML-DSA）CRYSTALS-DilithiumDigital Signature（取代 ECDSA）Module Learning With Errors
FIPS 205（SLH-DSA）SPHINCS+Stateless Hash-based Signature（備援）Hash function
未來HQC、BIKE、Classic McEliececode-based KEM 候選（NIST round 4）Error-correcting codes

為何 ML-KEM（前 Kyber）勝出

  • 性能優秀（比 RSA / ECDH 慢一些但合理）
  • 金鑰大小可接受（公鑰 ~800-1600 bytes，比 ECDH 的 32 bytes 大但能用）
  • 數學基礎成熟（lattice-based，研究多年）
  • 實作簡單（沒有 pairing 等複雜運算）

為何同時要備援

NIST 也標準化 SLH-DSA 與保留 HQC、BIKE 為 round 4 候選 — 「不要把所有蛋放同一個籃子」。Lattice-based 密碼學雖然成熟，但若未來發現對 lattice 的古典或量子攻擊，需要備援。

49.4 — Hybrid 模式：X25519MLKEM768

2025-2026 年的主流 PQC 部署不是「純 ML-KEM」，而是 hybrid 模式：把傳統 + 後量子兩種演算法同時用，最終 session key 來自兩者組合。

為何要 hybrid

  • 新演算法的未知風險：ML-KEM 雖經 NIST 多年審查，但仍有可能被未發現的攻擊破解
  • 實作 bug：新加密實作可能有 bug，hybrid 確保即使 ML-KEM 完全失敗，仍有 X25519 保護
  • 過渡期信心：給整個生態時間驗證 PQC 實作

X25519MLKEM768 是什麼

X25519MLKEM768（IANA codepoint 0x11ec）是 hybrid key agreement：

  • X25519：傳統 ECDH（Curve25519，32-byte key）
  • ML-KEM-768：NIST FIPS 203，「中等」security level
  • Client 與 server 同時跑兩個 key exchange
  • Final session key = HKDF(X25519 shared secret ‖ ML-KEM shared secret)
  • 攻擊者必須同時破解 X25519 與 ML-KEM 才能拿到 session key

部署現況（2026 年初）

  • 瀏覽器：Chrome 124+（2024/04 起預設）、Firefox 132+、Edge、Safari 18+ 全部預設啟用 X25519MLKEM768
  • TLS library：OpenSSL 3.4+、BoringSSL、Go crypto/tls（2024+）、Apple Network Stack 全部支援
  • CDN / Edge：Cloudflare、Google Cloud、AWS CloudFront 部分支援
  • Cloudflare 特殊地位：對所有客戶 enabled by default，含到 origin 的 outbound（client → CF → origin 整條鏈 quantum-safe）
  • 2026/03 數據：Cloudflare 處理流量約 35-40% 已是 X25519MLKEM768

49.5 — Hybrid 的代價：handshake 變大

X25519MLKEM768 不是 free lunch。性能影響：

資料大小

  • X25519 公鑰：32 bytes
  • ML-KEM-768 公鑰：1184 bytes
  • X25519MLKEM768 hybrid：~1216 bytes
  • ClientHello 從 ~500 bytes 增加到 ~1700 bytes

為何資料大小重要

  • 單個 packet 限制：MTU 約 1500 bytes，ClientHello 1700 bytes 一定要分兩個 packet
  • HelloRetryRequest 增加：server 不知 client 偏好哪個 group，可能要 HelloRetryRequest 加一個 round trip
  • Server 計算量：ML-KEM-768 比 X25519 慢約 5-10x，但仍 ms 級

Cloudflare 的 HelloRetryRequest 策略

Cloudflare 對 origin 連線採用 conservative 策略：預設只「廣告」支援 X25519MLKEM768，不主動發 keyshare。如果 origin 也支援，origin 用 HelloRetryRequest 要求；不支援就退回 X25519。代價：多一個 RTT（HRR）。但避免破壞舊 origin。

進階配置：客戶可透過 API set tls_pq_origin_settings = preferred 讓 Cloudflare 直接送 keyshare，省 RTT。

49.7 — PQC 證書：尚未到來的挑戰

49.4-49.6 講的都是 key agreement — 已有實際部署。但 TLS 還有另一半：數位簽章 / 證書。這部分2026 年仍未準備好。

為何證書比 key agreement 難

  • ML-DSA 簽章很大：~3-4KB（vs ECDSA 64 bytes），單個證書鏈 30KB+
  • 每次 handshake 都要傳：影響每個連線而非 session 重用
  • CA 生態要全升級：Let's Encrypt、DigiCert、Sectigo 等都要支援，CT log 也要
  • 瀏覽器要全部信任：要新 root CA、新驗證邏輯
  • 硬體 token / HSM：很多企業 HSM 不支援 ML-DSA

時間表（2026 預測）

  • 2026 年：第一張 PQC 證書出現（試驗、非主流瀏覽器信任）
  • 2027 年：主流 CA 開始發放，瀏覽器漸 enable
  • 2028-2030 年：廣泛部署
  • 2030+：成為新基準

因此 2026 年的 reverse proxy 工程師：升級 key agreement 是務實（防 harvest-now），等 PQC 證書再說（沒到時間，別自找麻煩）。

49.8 — 為何「現在升級」是正確答案

讀到這你可能想：「2030 才有量子電腦威脅，現在升級太早？」

三個理由說明這想法錯誤：

  1. Harvest-now 已經在發生。攻擊者今天截獲的流量，未來 10-20 年才解密。今天不升，今天的祕密已經在風險中
  2. 密碼學遷移是 5-10 年的事。看 SHA-1 → SHA-256（2005 提議，2017 才完全棄用）、TLS 1.0 → 1.2（2008 提議，2021 才主流）。PQC 的時程更急，但仍要多年
  3. 升級幾乎免費。OpenSSL 升級、改一行 cipher 設定、X25519MLKEM768 性能影響極小（多 1KB handshake、5-10ms server compute）。代價遠小於風險

給 reverse proxy 工程師的最後告誡

2026 年仍未啟用 PQC key agreement 的 reverse proxy，2030 年回頭看會像今天看「仍用 SSLv3」一樣 — 不是會不會升級，而是什麼時候升級。早升級的優勢：

  • 找出實作 bug 與 interop 問題（在關係不大的時候）
  • 累積運維經驗
  • 對外溝通正面：客戶、合規、稽核都加分
  • 成本低（瀏覽器都支援，server-side 升級成本可控）

晚升級的代價：

  • 當監管要求時手忙腳亂
  • 與其他系統 interop 問題比早升級時難解
  • 可能真的洩漏歷史流量

50.3 — Gateway API 5 個 GA 資源

2026 年 Gateway API standard channel 的 5 個 GA 資源：

GatewayClass（cluster-scoped）

定義「用哪個實作」。一個 cluster 可有多個 GatewayClass（如同時用 Cilium 與 Istio）。

GatewayClass 範例apiVersion: gateway.networking.k8s.io/v1
kind: GatewayClass
metadata:
  name: cilium
spec:
  controllerName: io.cilium/gateway-controller

Gateway（namespaced）

實際的「監聽 endpoint」 — port、protocol、TLS、IP 等。由 cluster operator 管理。

Gateway 範例apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: prod-gateway
  namespace: networking
spec:
  gatewayClassName: cilium
  addresses:
    - type: IPAddress
      value: 192.168.10.95      # 從 CiliumLoadBalancerIPPool 拿
  listeners:
    - name: https
      protocol: HTTPS
      port: 443
      hostname: "*.example.com"
      tls:
        mode: Terminate
        certificateRefs:
          - kind: Secret
            name: wildcard-cert
      allowedRoutes:
        namespaces:
          from: All

HTTPRoute（namespaced）

實際路由規則 — host、path、method、header 匹配，後端 service 與 weight。由 application developer 管理。

HTTPRoute 範例（含 canary）apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: api-route
  namespace: api-team
spec:
  parentRefs:
    - name: prod-gateway
      namespace: networking
  hostnames:
    - api.example.com
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /v2/
      backendRefs:
        - name: api-stable
          port: 8080
          weight: 90
        - name: api-canary
          port: 8080
          weight: 10
      filters:
        - type: RequestHeaderModifier
          requestHeaderModifier:
            add:
              - name: X-Forwarded-Proto
                value: https

GRPCRoute

給 gRPC 服務的 first-class 支援，匹配 service / method 而非 path。2024 加入 GA。

ReferenceGrant

跨 namespace 引用的安全控制 — 例如 HTTPRoute 在 namespace A 引用 namespace B 的 Service，B 必須明確 grant。

仍 experimental 的資源

  • TLSRoute：TLS passthrough 路由（如 ArgoCD、Keycloak 等需要原始 TLS 的場景）
  • TCPRoute / UDPRoute：non-HTTP 協定路由
  • BackendTLSPolicy：上游 TLS 配置

50.5 — Cilium eBPF：重新定義 K8s 網路

Cilium 是2026 年 K8s 網路最有野心的專案。2024/10 Cisco 收購 Isovalent（Cilium 主開發公司），2025-2026 加速整合到 Cisco 整個網路產品線。

eBPF 為何強

eBPF（Extended Berkeley Packet Filter）讓使用者程式碼直接在 Linux kernel 執行，跳過 user space ↔ kernel 的 context switch。對 K8s 網路：

  • kube-proxy 完全替代：傳統 iptables / IPVS 規則被 eBPF map 取代，O(1) 查找
  • L4 throughput 限於硬體 NIC：XDP（eXpress Data Path）讓 packet 在 NIC driver 層級就被處理，user space 完全不參與
  • 容器密度：eBPF 規則直接在 kernel，每個 container 不需要 sidecar
  • Direct Server Return（DSR）：load balancer 不在回應路徑上，性能翻倍
  • Maglev consistent hashing：Google 風格的 LB 演算法

Cilium 的不可取代功能

  • L2 Announcements：取代 MetalLB（裸金屬 K8s LB）
  • LB IPAM：CiliumLoadBalancerIPPool 管理 LB IP
  • BGP 整合：與 ToR / spine switch 直接 peering
  • Cluster Mesh：跨多 cluster 的 service discovery + mTLS
  • Hubble：eBPF-based observability
  • Tetragon：runtime security（行程級監控）

Cilium 的限制

  • L7 流量仍走 per-node Envoy：純 eBPF 處理不了複雜 HTTP（header、JSON parsing）
  • 控制平面 churn 限制：HTTPRoute 大量更新時 Envoy reconcile 仍是瓶頸
  • 除錯難度：eBPF 程式 crash 直接影響 kernel，traceback 在 kernel space
  • Linux 6.x 需求：許多 feature 要新 kernel

50.6 — Istio Ambient：殲滅 sidecar

傳統 Istio 用 sidecar 模式（每個 pod 旁邊一個 Envoy）。2024-2025 年 Istio 力推 Ambient mode，把 sidecar 改成兩層：

  • ztunnel（per-node L4 proxy）：用 Rust 寫，輕量，處理 L4 mTLS
  • waypoint proxy（per-namespace L7 Envoy）：選擇性加入，只在需要 L7 policy 時啟用

每 pod overhead~50-100MB（Envoy）~5MB（ztunnel 共享）

2026 年 Ambient 狀態

  • 2024/05 Ambient v1.22 GA
  • 2026 年成為 Istio 推薦預設模式
  • Solo、Tetrate、Red Hat 等商業支援
  • 性能與 Envoy Gateway 接近（消除 sidecar double-hop）

50.7 — GAMMA：mesh 與 gateway 統一

GAMMA（Gateway API Mesh Management）是 SIG-Network 的子專案，目標：讓 Gateway API 同時管理 east-west（service mesh）與 north-south（ingress）。

GAMMA 核心概念

HTTPRoute 不只能 attach 到 Gateway（north-south），也能 attach 到 Service（east-west）。把 service mesh 的 traffic split / canary 用同一個 API 管理。

GAMMA HTTPRoute 範例apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: internal-canary
spec:
  parentRefs:
    - kind: Service      # 不是 Gateway！
      name: payment-service
  rules:
    - backendRefs:
        - name: payment-v1
          weight: 90
        - name: payment-v2
          weight: 10

2026 年 GAMMA 實作

  • Cilium：v1.18+ 完整支援（含 Service-as-parentRef）
  • Istio Ambient：原生支援
  • Linkerd：experimental 支援

對工程師：GAMMA 把「ingress vs mesh」的二元對立融合為「一致的 traffic policy」。長期看是 K8s 網路的方向。

51.3 — Edge 取代 reverse proxy 的場景

Edge computing 在哪些場景真的取代傳統 reverse proxy？

場景 1：認證與授權邊緣化

傳統：origin 收到請求 → 驗 JWT → 處理或拒絕。問題：未授權請求已經跨太平洋到 origin，浪費頻寬與計算。

Edge：在邊緣 PoP 驗 JWT，未授權請求從未離開使用者區域。Cloudflare Access、Vercel Authentication 都這樣做。

場景 2：A/B Test 與 Feature Flag

傳統：在 origin 或 client 做 A/B 邏輯。問題：全球流量都打到 origin、或前端代碼複雜化。

Edge：根據 cookie / geo / 隨機分配 routing 到不同 backend，origin 看到的就是「已分組的流量」。Cloudflare Workers + Origin Rules 是經典模式。

場景 3：Cache 個性化（Personalized Cache）

傳統 CDN cache 對登入使用者無效（每個 user 不同回應）。Edge：根據 cookie 動態組合 cached fragments + personalized data，讓登入頁面也能 cache。Vercel ISR、Cloudflare Cache API 提供原語。

場景 4：Geo-routing 與 Compliance

EU 客戶 routing 到 EU origin（GDPR）、中國客戶到香港、其他到美國。Edge 做這件事比 DNS-based geo-routing 精準得多。

場景 5：Image / Video 動態變換

邊緣根據 device、network 動態 resize / format conversion（avif、webp）/ compression。Cloudflare Images、Vercel Image Optimization 都基於此。

場景 6：AI Inference 邊緣化

Cloudflare Workers AI、Vercel AI SDK + Edge — 把 LLM inference 跑在邊緣 GPU，避免每次請求跨太平洋。對 chat 應用體驗顯著提升。

51.4 — Edge 不是銀彈：限制與陷阱

Edge computing 不是所有場景的最佳選擇。誠實面對限制：

限制 1：CPU time 嚴格

  • Cloudflare Workers：50ms CPU time（Free 10ms、Paid 50ms、Unbound 30s）
  • Vercel Edge：30s wall-clock
  • Fastly Compute@Edge：60s

對長時間 compute（如 video transcoding、大量 ML 推論）不適合。

限制 2：Memory 上限

  • Workers：128 MB
  • Vercel Edge：4 GB（更寬鬆）

處理大檔案、大資料集會撞壁。

限制 3：Stateful 場景困難

Edge 設計是無狀態的（每個請求可能去不同 PoP）。Stateful 解法：

  • Cloudflare KV：eventually consistent（讀快但寫有 60s 延遲）
  • Cloudflare Durable Objects：strongly consistent（單一 instance per object）
  • Cloudflare D1：分散式 SQLite，讀寫差異
  • Vercel Edge Config：read-heavy 配置儲存

真正的 stateful 應用（資料庫、長 session）仍要 origin。

限制 4：WebSocket 與長連線

Edge 對短請求優化。WebSocket（長連線）支援有限：

  • Cloudflare Workers + Durable Objects：完整 WebSocket 支援
  • Vercel Edge：有限
  • Fastly：不支援

限制 5：Node.js API 不完整

V8 isolate 不是 Node.js — 沒 fs、child_process、net（raw socket）。許多 npm 套件（依賴 Node.js API）無法運作。必須用 Web Standards API（Fetch、Web Crypto、Streams）。

限制 6：Vendor lock-in

Workers 的 KV / Durable Objects、Vercel 的 Edge Config — 都是 vendor-specific。遷移到其他平台困難。

Zero Trust✗✓（Access、Tunnel）

結論：2026 年的 CDN 不是 CDN，是「全球分散的 reverse proxy 平台 + 安全 + 計算」。Cloudflare 自我定位為「Connectivity Cloud」 — 從 reverse proxy 工程師視角，這是最精準的描述。

51.6 — 給 reverse proxy 工程師的 5 條建議

  1. 邊緣化「容易移到邊緣的」：認證、A/B test、geo-routing、cache 個性化
  2. 保留 origin 給「需要 stateful」：資料庫操作、長計算、WebSocket（除非用 Cloudflare DO）
  3. 學 V8 isolate 模型：即使你不寫 Workers，理解這個範式為何重要
  4. Cold start < 5ms 是 game-changer：傳統 serverless 性能瓶頸消失了
  5. 把 CDN 當 reverse proxy 平台：2026 年的 Cloudflare / Akamai / Fastly 不是 cache，是完整 platform
本章要點
  • V8 isolate 革命：冷啟動 < 5ms（vs 容器 100-1000ms）、~5MB 記憶體（vs 容器 50-500MB）、千倍密度。
  • 用「受限」換「速度與密度」：無 filesystem、CPU time 嚴格、有限 Node.js API。
  • Cloudflare Shard and Conquer達 99.99% warm rate。
  • 6 大平台：Cloudflare Workers（330+ PoP，最廣）、Vercel（Next.js 首選）、Fastly Compute@Edge（純 WASM）、Deno Deploy（TS-native）、AWS Lambda@Edge、Akamai EdgeWorkers。
  • Edge 取代 reverse proxy 的 6 場景：邊緣 auth、A/B test、cache 個性化、geo-routing、image transform、AI inference。
  • Edge 不是銀彈：CPU time 嚴格、memory 上限、stateful 困難、WebSocket 限制、Node.js API 不完整、vendor lock-in。

Zero Trust 與 IPv6
BeyondCorp、ZTNA、IPv6-only mobile network 與 Part VI 收尾

Part VI 進入最後一章。前面 5 章涵蓋了 reverse proxy 工程師的前沿協定（MASQUE、HTTP/3、PQC）與前沿平台（Gateway API、eBPF、Edge）。本章是前沿架構：Zero Trust 重新定義「信任邊界」、IPv6-only 終於從研究議題變成生產現實。兩者表面無關，實際都反映同一個趨勢：傳統「網路邊界」已死，identity 與 protocol 才是新邊界。本章是 Part VI 結語，也是 reverse proxy 工程師面對 2030 年的最後準備。

52.1 — Zero Trust 的起源：BeyondCorp

2009 年 Google 遭 Operation Aurora 攻擊（中國駭客滲透 Google 內部網路）。Google 安全團隊得出一個激進結論：「內部網路不該被信任」。傳統「內網安全 / 外網不安全」二元假設失敗 — VPN 內的攻擊者與外部攻擊者一樣危險。

BeyondCorp 核心原則

2014 年 Google 公開 BeyondCorp 架構，三個核心：

  1. 不信任網路位置：在 office 與在家咖啡店沒本質差別
  2. 每個請求都驗證：每次存取都要 identity + device 驗證，不是「連上 VPN 就放行」
  3. 加密所有流量：即使在「內網」，仍用 TLS

為何叫「Zero Trust」

術語由 Forrester 分析師 John Kindervag 2010 年提出。核心：「Never trust, always verify」 — 不論流量來源、身分初步驗證、設備狀態，每次都要 re-verify。

2020/08 NIST 釋出 Special Publication 800-207「Zero Trust Architecture」，給 ZTA 一個官方定義。三大組件：

52.3 — OMB M-22-09：美國聯邦強制過渡

2022/01 美國 OMB（Office of Management and Budget）發布 M-22-09：所有聯邦機構 2024 年底完成 Zero Trust 過渡。具體要求：

  • Identity：phishing-resistant MFA（FIDO2 / WebAuthn）
  • Devices：每個設備有完整 inventory + 持續監控
  • Networks：所有 DNS / HTTP 流量加密；網路分段 + micro-segmentation
  • Applications：所有 app 內部 + 外部都當「公開網際網路」對待
  • Data：分類、加密、DLP

影響：美國聯邦市場是 ZTNA 廠商最大客戶，間接推動全球生態。EU NIS2（2024 年強制）也類似要求。

52.5 — IPv6-only：從研究議題到生產現實

IPv6 自 1998 年標準化（RFC 2460），但30 年仍未全面取代 IPv4。2020-2026 年情況變了：IPv6-only 部署已是生產現實。

2026 年的 IPv6-only 部署

  • Apple iOS / macOS（2016+）：Apple App Store 強制要求 app 在 IPv6-only 網路中能運作。事實上所有 iOS app 都已 IPv6-ready
  • T-Mobile US：mobile network 已是 IPv6-only，IPv4 透過 NAT64 / 464XLAT 提供
  • Facebook / Meta 內部：data center 內部完全 IPv6-only（自 2017 年）
  • LinkedIn 內部：data center 自 2014 年 IPv6-only
  • Microsoft Azure：2025 起完整 IPv6-only support，可建純 IPv6 VNet
  • AWS：2024+ 多個 service 提供 IPv6-only endpoint，VPC 支援 dual-stack 與 IPv6-only
  • Google Cloud：IPv6 在主要 region 全面可用
  • 印度、越南：政府與電信主導的 IPv6 全面部署

為何現在

  • IPv4 地址枯竭：RIR 分配的 IPv4 / 24 block 一個賣 USD $30,000+（2026）
  • NAT 痛苦：CGNAT（Carrier-Grade NAT）造成 P2P、WebRTC、VoIP 問題
  • 雲廠商成本：AWS 從 2024/02 開始對 public IPv4 收費（每小時 $0.005）
  • Mobile network：行動網路的 4G LTE / 5G 設計就是 IPv6-first

52.6 — IPv4-IPv6 過渡：NAT64 與 464XLAT

純 IPv6 部署的現實挑戰：仍有大量 IPv4-only 資源（舊網站、第三方 API、IoT 設備）。解法：

NAT64 + DNS64

網路提供「假裝 IPv6 可達 IPv4」的閘道：

  • DNS64：DNS server 對 IPv4-only 域名回傳合成的 IPv6 地址（前綴 64:ff9b::/96 + IPv4 地址編碼）
  • NAT64：邊界 router 把 IPv6 → 真實 IPv4 連線

缺點：DNS64 對 DNSSEC 不友好；某些 app 在 raw IP 中嵌入 IPv4 文字，不會被翻譯。

464XLAT（client-side translation）

進階方案：client 端有 NAT46 + NAT64 的兩段翻譯：

  • App 用 IPv4 socket → client OS 翻譯成 IPv6（NAT46）→ 邊界 NAT64 → IPv4 server
  • 對 app 完全透明，連 IP literal 都能用
  • Apple iOS、Android 內建 464XLAT
  • T-Mobile US 用此方案

Reverse Proxy 在 IPv6-only 場景

  • Nginx 1.28+：完整 IPv6-only 支援，含 health check 與 upstream
  • HAProxy 3.0+：完整支援
  • Envoy：完整支援
  • Cloudflare：對 origin 同時試 IPv6 與 IPv4，自動 prefer IPv6

Part VI 走完了 6 個前沿議題。整理 reverse proxy 工程師的 2026-2035 年技能地圖：

2026 年「必修」技能（已從 nice-to-have 變必要）

  • HTTP/3 / QUIC：全球流量已超 30%（Cloudflare 數據）
  • Kubernetes Gateway API：Ingress NGINX 退役後沒選擇
  • TLS 1.3 + X25519MLKEM768：harvest-now 攻擊已在發生
  • OAuth / OIDC + Zero Trust 基礎：合規與企業整合

2026-2030「將會必修」

  • MASQUE：當 VPN / privacy proxy 成主流
  • eBPF / Cilium：當 K8s 性能要求超過 Envoy 能達
  • Edge computing（Workers / Edge Functions）：當邊緣化成為部署預設
  • IPv6-only：當雲廠商繼續對 IPv4 加價
  • PQC certificates：當 NIST FIPS 204 ML-DSA 證書 2027-2028 主流化
  • Service Mesh + GAMMA：當 ingress 與 mesh 統一管理

2030+「下一波」（推測）

  • 純 PQC（不要 hybrid，直接用 ML-KEM）
  • WebMCP / agent-aware proxy（cf. Ch.42）
  • Encrypted ClientHello（ECH）廣泛部署
  • HTTP/4 ?（誰知道，但 IETF 已在討論）

三個跨議題趨勢

  1. 「邊界」消失：傳統網路邊界 → identity + protocol 為新邊界（Zero Trust + MASQUE + PQC 都反映這點）
  2. 「分散化」加深：單機 reverse proxy → 全球邊緣 platform（Edge computing + Workers + 330+ PoP）
  3. 「協定升級」加速：HTTP/1.1 → HTTP/2 用了 18 年；HTTP/2 → HTTP/3 用 7 年；MASQUE 從 RFC 到主流會更快
