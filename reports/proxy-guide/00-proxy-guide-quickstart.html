Proxy 代理技術
完整指南

從 SOCKS5 到 Reverse Proxy，從 HTTP 代理到 API Gateway — 涵蓋所有主流 Proxy 類型的運作原理、應用場景、實務限制與速查表。本文件適合網路工程師、後端開發者以及對網路架構有興趣的技術人員參考。

CH.01

Proxy 基本概念

Proxy（代理伺服器）是網路通訊中的「中介者」，位於客戶端與目標伺服器之間，接收並轉發請求。不同的 Proxy 依照運作層次、方向、協定支援等維度有不同的分類。

核心運作原理

Figure 1.1 · 核心運作原理
flowchart LR
    C["<b>Client</b>"]
    P["<b>Proxy</b>"]
    T["<b>Target</b><br/><i>看到來源 IP 是 Proxy<br/>不是 Client 真實 IP</i>"]
    C --> P --> T

三大分類維度

維度一 / LAYER

運作層次

Layer 3（VPN）、Layer 4（SOCKS5、TCP/UDP）、Layer 7（HTTP、HTTPS、API Gateway）— 層次越高理解協定越深，功能越豐富。

維度二 / DIRECTION

代理方向

Forward Proxy 代理客戶端、Reverse Proxy 代理伺服器端、Transparent Proxy 對客戶端完全透明。

維度三 / PURPOSE

特化用途

通用代理、API Gateway、Service Mesh Sidecar、CDN Edge Proxy、翻牆專用（Shadowsocks、V2Ray、Trojan）等。

Key Insight

同一個軟體常可扮演多種角色。例如 Nginx 既可作 Forward Proxy，也能作 Reverse Proxy；HAProxy 同時支援 L4 和 L7 操作；Envoy 則可作為 API Gateway 或 Service Mesh Sidecar。

CH.02

SOCKS5 深度解析

SOCKS5 是最通用的 Proxy 協定之一，運作在 OSI 第 5 層（Session Layer），不理解上層應用協定，純粹負責 TCP/UDP 封包轉送。這讓它具有極高的通用性。

SOCKS5 的優勢

  • 支援 TCP 和 UDP（SOCKS4 僅支援 TCP）
  • 支援 IPv6 位址
  • 支援多種認證機制（username/password、GSSAPI 等）
  • 與應用層協定無關，幾乎任何 TCP 應用都能透過它轉送
  • overhead 小，轉送效能接近直連

SOCKS5 的限制

限制說明解決方案
ICMP 無法代理 ping、traceroute 等非 TCP/UDP 協定 使用 VPN 或 TUN/TAP
UDP 規格支援但實作常不完整，防火牆常擋 UDP 確認 Server 實作、使用 TCP fallback
BIND 主動接收連線的 P2P 場景支援度差 使用 UPnP 或直連
透明度 應用程式必須「知道」自己在用 Proxy proxychains、Proxifier、tun2socks
加密 協定本身不加密，ISP 仍可見目標位址 套 TLS、SSH Tunnel、Shadowsocks

DNS 洩漏問題（SOCKS5 vs SOCKS5h）

Critical

預設的 SOCKS5 會先在本機做 DNS 解析，再把 IP 丟給 Proxy，導致 DNS 查詢暴露真實 IP。使用 SOCKS5h（hostname resolution by proxy）可讓 Proxy 端做 DNS 解析。

# 錯誤：DNS 走本機，會洩漏
curl --socks5 127.0.0.1:1080 https://example.com

# 正確：DNS 走 Proxy，無洩漏
curl --socks5-hostname 127.0.0.1:1080 https://example.com

SOCKS5 的應用場景

  • 換出站 IP：爬蟲、隱藏真實位址、繞過地區限制
  • 跳板機：搭配 ssh -D 建立 SOCKS5，登入內網資源
  • 全系統代理：搭配 tun2socks、Clash 等工具讓所有流量走 SOCKS5
  • 翻牆基礎：Shadowsocks、V2Ray、Trojan 等翻牆工具多基於 SOCKS5 擴充
CH.03

HTTP / HTTPS Proxy

HTTP Proxy 運作於 OSI 第 7 層，是最早、也最普及的 Proxy 類型。它理解 HTTP 協定，可讀寫 header、URL、內容等資訊。

HTTP Proxy 運作模式

Figure 3.1 · HTTP Proxy 運作模式
flowchart LR
    C["<b>Client</b>"]
    P["<b>Proxy</b>"]
    T["<b>Target</b>"]
    C -- "GET http://example.com/" --> P
    P --> T

HTTPS Proxy（CONNECT 方法）

HTTPS 流量是加密的，HTTP Proxy 無法讀取內容。處理方式是透過 CONNECT 方法建立 TLS 通道，Proxy 只負責盲轉資料，看不到加密內容（除非做 SSL Bump / MITM）。

CONNECT example.com:443 HTTP/1.1
Host: example.com:443

HTTP/1.1 200 Connection Established

// 之後 Proxy 就只負責 byte-level 轉送 TLS 資料

環境變數標準

HTTP Proxy 的設定方式在各作業系統、程式語言、CLI 工具中都有統一標準：

export HTTP_PROXY=http://proxy.example.com:8080
export HTTPS_PROXY=http://proxy.example.com:8080
export NO_PROXY=localhost,127.0.0.1,.internal.com
Tip

幾乎所有命令列工具、SDK、套件管理器（npm、pip、apt、git 等）都遵循這套環境變數。設定一次就能讓多數程式走 Proxy。

CH.04

Squid 與 SOCKS5 對比

Squid 是最經典的 HTTP Application Layer Proxy，專門處理 HTTP/HTTPS，主打「快取加速」與「存取控制」。理解 Squid 與 SOCKS5 的差異，能幫助選擇合適的工具。

面向SOCKS5Squid (HTTP Proxy)
運作層次L5 (Session)L7 (Application)
理解協定內容否是
支援協定TCP / UDP 通用僅 HTTP / HTTPS
內容快取無核心功能
URL 過濾 / ACL無強大
修改 HTTP Header無可以
內容檢查（防毒、過濾）無搭配 ICAP
主要用途通用代理、翻牆企業上網管控、快取
典型使用者個人、開發者企業、學校、ISP

何時用 SOCKS5、何時用 Squid

CHOOSE SOCKS5

通用轉送、換 IP

代理 SSH、資料庫、遊戲、IM、BitTorrent 等非 HTTP 應用；需要快速換出站 IP；建立跳板機連線。

CHOOSE SQUID

HTTP 快取、內容管控

企業內網頻寬優化、網頁快取加速、URL 黑白名單、使用者稽核、廣告過濾、上網時間管理。

CH.05

Reverse Proxy（反向代理）

Reverse Proxy 與前面討論的 Forward Proxy 方向相反：它代理的是「伺服器端」，客戶端不知道背後實際有多台伺服器。

典型架構

Figure 5.1 · Reverse Proxy 典型架構
flowchart LR
    C["<b>Client</b><br/><i>只看到 RP 的 IP<br/>不知後端是哪台</i>"]
    RP["<b>Reverse Proxy</b>"]
    S1["Server 1"]
    S2["Server 2"]
    S3["Server 3"]
    SN["Server N"]
    C --> RP
    RP --> S1
    RP --> S2
    RP --> S3
    RP --> SN

Reverse Proxy 的核心功能

  • 負載平衡：將請求分散到多台後端（Round Robin、Least Connections 等演算法）
  • SSL/TLS Termination：在 Proxy 層處理 HTTPS，後端可用明文 HTTP
  • 快取：加速靜態資源、減輕後端負擔
  • 安全防護：WAF、DDoS 防禦、隱藏後端架構
  • 路由：依 URL、Header、Host 轉發到不同服務
  • 壓縮與協定轉換：HTTP/2、HTTP/3、gRPC 適配

常見 Reverse Proxy 產品

產品特色典型場景
Nginx輕量、高效能、生態豐富Web 服務前端、靜態資源
HAProxyL4/L7 雙模式、企業級穩定高可用負載平衡
Traefik雲原生、自動服務發現Kubernetes、Docker Swarm
EnvoyL7 高階功能、可觀測性Service Mesh、API Gateway
Cloudflare全球 CDN、DDoS 防護Edge 層反向代理
CH.06

API Gateway 與 AI OAuth 轉發

現代 AI 服務（如 ChatGPT Connectors、Claude MCP、各類 AI Agent）在呼叫第三方 API 時，需要處理 OAuth token 的儲存、注入、刷新，這類「身分代理」並非傳統 Proxy 的範疇。

架構圖

Figure 6.1 · API Gateway + Token Manager
flowchart LR
    U["<b>User</b>"]
    A["<b>AI App</b>"]
    G["<b>API Gateway</b><br/><i>inject Bearer</i>"]
    T["<b>Token Manager</b><br/>Store tokens<br/>Refresh logic<br/>Scope mgmt"]
    P["<b>3rd Party API</b><br/>Google · GitHub"]
    U --> A --> G --> P
    T -. "tokens" .-> G

為何不是 SOCKS5 也不是 Squid

Key Distinction

網路代理（SOCKS5、Squid）處理「封包轉送」；身分代理（API Gateway）處理「代表使用者身分呼叫 API」。兩者解決的問題層次完全不同。

面向SOCKS5SquidAI OAuth 轉發
運作層次L5L7 (HTTP)L7 + 應用邏輯
看得到 HTTP 內容否是是
修改 Authorization無法可以但非主業核心功能
Token 生命週期——完整管理
主要目的換 IP快取 / 過濾身分代理

典型技術實作

  • API Gateway 產品：Kong、Apigee、AWS API Gateway、Azure APIM
  • Reverse Proxy + 中介層：Nginx / Envoy + Lua / WASM filter
  • Service Mesh Sidecar：Istio、Linkerd 在 mesh 層處理認證
  • MCP Server（Model Context Protocol）：Anthropic 推動的 AI 工具呼叫協定
  • OAuth 2.0 Token Exchange（RFC 8693）：token 互換的標準協定
CH.07

Java / JVM / Tomcat 的 Proxy 支援

JVM 內建 SOCKS 與 HTTP Proxy 支援，透過系統屬性（System Properties）即可啟用。Tomcat 作為 Java 應用，完全繼承這套機制。

JVM System Properties 設定

# SOCKS Proxy
-DsocksProxyHost=127.0.0.1
-DsocksProxyPort=1080
-DsocksProxyVersion=5
-Djava.net.socks.username=user
-Djava.net.socks.password=pass

# HTTP / HTTPS Proxy
-Dhttp.proxyHost=proxy.example.com
-Dhttp.proxyPort=8080
-Dhttps.proxyHost=proxy.example.com
-Dhttps.proxyPort=8080
-Dhttp.nonProxyHosts="localhost|*.internal.com"

Tomcat 設定方式

在 setenv.sh 或 catalina.sh 加入 CATALINA_OPTS：

export CATALINA_OPTS="-DsocksProxyHost=proxy.example.com \
                      -DsocksProxyPort=1080 \
                      -Dhttps.proxyHost=proxy.example.com \
                      -Dhttps.proxyPort=8080"

關鍵注意事項

NIO / NETTY

NIO 框架不吃系統屬性

Netty、Spring WebFlux、gRPC Java 等基於 NIO 的框架不會自動使用 socksProxyHost。需在程式碼顯式加入 Socks5ProxyHandler。

DNS LEAK

預設有 DNS 洩漏風險

Java 預設在本機解析 DNS，要避免洩漏需用 InetSocketAddress.createUnresolved() 建立未解析位址。

UDP

僅支援 TCP

JVM 的 SOCKS 實作基本只處理 TCP，UDP 流量不會透過 Proxy 轉送。

HTTPCLIENT

函式庫層級精細控制

OkHttp、Apache HttpClient、Java 11+ HttpClient 都可用 Proxy 物件做個別連線的 Proxy 設定。

除錯技巧

# 啟動時加入 net debug，確認連線是否走 Proxy
-Djava.net.debug=net

# 更詳細的除錯訊息
-Djava.net.debug=all

02 — 常用指令速查

SSH 建立 SOCKS5

# 建立本地 SOCKS5 proxy 在 port 1080
ssh -D 1080 -N -f user@jumphost.example.com

# Local Port Forward（L:遠端服務到本地）
ssh -L 3306:db.internal:3306 user@jumphost

# Remote Port Forward（R:本地服務到遠端）
ssh -R 8080:localhost:80 user@remote

curl 使用 Proxy

# HTTP Proxy
curl -x http://proxy:8080 https://example.com

# SOCKS5
curl --socks5 127.0.0.1:1080 https://example.com

# SOCKS5h（避免 DNS 洩漏）
curl --socks5-hostname 127.0.0.1:1080 https://example.com

proxychains 強制程式走 SOCKS5

# /etc/proxychains.conf 設定 socks5 127.0.0.1 1080
proxychains4 telnet example.com 23
proxychains4 nmap -sT example.com

git 設定 Proxy

git config --global http.proxy http://proxy:8080
git config --global https.proxy http://proxy:8080

# SOCKS5
git config --global http.proxy socks5://127.0.0.1:1080

06 — 洩漏風險與防護檢查
  • DNS Leak 使用 SOCKS5h、DoH、或 Proxy 端解析
  • WebRTC Leak 瀏覽器關閉 WebRTC 或使用擴充套件
  • IPv6 Leak 停用 IPv6 或確認 Proxy 支援
  • UDP Fallback 某些 UDP 流量繞過 SOCKS5
  • Plaintext SOCKS5 / HTTP Proxy 不加密，需搭配 TLS
  • App Fingerprint User-Agent、Cookie、Canvas 等應用層指紋
