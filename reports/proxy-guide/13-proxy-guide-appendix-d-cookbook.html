APPENDIX D · COOKBOOK

Reverse Proxy Cookbook
Side-by-Side 配置對照 · Nginx / HAProxy / Envoy / Caddy / Cloudflare

本附錄是 reverse proxy 工程師的查找表。8 個常見場景、5 大工具並列對照。當你從一個工具遷移到另一個（如「從 Nginx 遷到 Envoy」），這裡有最小可用配置。不是完整教程 — 是「那個東西怎麼寫」的速查。

Part IV 已詳細介紹各 reverse proxy 工具。本附錄純實作取向 — 給已經懂概念的工程師當查找表。每個場景並列顯示 5 工具的最小可用配置：Nginx（最廣泛）、HAProxy（最高性能 L4/L7）、Envoy（service mesh 主流）、Caddy（自動 HTTPS）、Cloudflare Workers（edge serverless）。每個場景配置都跑得起來，但 production 使用前要依環境調整。

5 大工具速覽

工具配置語言主要 reload 方式Sweet spot
Nginxnginx.conf（自訂 DSL）nginx -s reload gracefulWeb server + reverse proxy 通用、最廣支援
HAProxyhaproxy.cfg（自訂 DSL）SIGUSR2 + master-workerL4/L7 高性能 LB、TCP-heavy 場景
EnvoyYAML / JSON（xDS API）Hot restart / xDS dynamicService mesh、API gateway、動態 reconfig
CaddyCaddyfile / JSONcaddy reload 或 admin API自動 HTTPS、開發者友善、Modern
Cloudflare WorkersJavaScript / TypeScriptwrangler deployEdge logic、無伺服器 reverse proxy

下方所有範例假設：backend 在 backend.example.com:8080、reverse proxy 對外 api.example.com。

場景 1 · 基礎 Reverse Proxy + TLS Termination

最常見場景：reverse proxy 監聽 443，終止 TLS、forward HTTP 到 backend。

Nginxserver {
    listen 443 ssl http2;
    server_name api.example.com;

    ssl_certificate     /etc/ssl/api.crt;
    ssl_certificate_key /etc/ssl/api.key;
    ssl_protocols       TLSv1.2 TLSv1.3;

    location / {
        proxy_pass         http://10.0.1.5:8080;
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto https;
    }
}

# HTTPS redirect
server {
    listen 80;
    server_name api.example.com;
    return 308 https://$host$request_uri;
}
HAProxyfrontend https_in
    bind *:443 ssl crt /etc/ssl/api.pem alpn h2,http/1.1
    bind *:80
    redirect scheme https code 308 if !{ ssl_fc }

    http-request set-header X-Forwarded-Proto https if { ssl_fc }
    default_backend api_backend

backend api_backend
    server srv1 10.0.1.5:8080 check
Envoystatic_resources:
  listeners:
    - address:
        socket_address: { address: 0.0.0.0, port_value: 443 }
      filter_chains:
        - transport_socket:
            name: envoy.transport_sockets.tls
            typed_config:
              "@type": type.googleapis.com/envoy.extensions.transport_sockets.tls.v3.DownstreamTlsContext
              common_tls_context:
                tls_certificates:
                  - certificate_chain: { filename: /etc/ssl/api.crt }
                    private_key: { filename: /etc/ssl/api.key }
          filters:
            - name: envoy.filters.network.http_connection_manager
              typed_config:
                "@type": type.googleapis.com/envoy.extensions.filters.network.http_connection_manager.v3.HttpConnectionManager
                stat_prefix: api_https
                route_config:
                  virtual_hosts:
                    - name: api
                      domains: ["api.example.com"]
                      routes:
                        - match: { prefix: "/" }
                          route: { cluster: api_backend }
                http_filters:
                  - name: envoy.filters.http.router

  clusters:
    - name: api_backend
      type: STRICT_DNS
      load_assignment:
        cluster_name: api_backend
        endpoints:
          - lb_endpoints:
              - endpoint:
                  address: { socket_address: { address: backend.example.com, port_value: 8080 } }
Caddy（自動 HTTPS via Let's Encrypt）api.example.com {
    reverse_proxy 10.0.1.5:8080
}

# 整個檔案就這樣 — Caddy 自動取 cert、自動 redirect 80→443
Cloudflare Workers（無 origin server）export default {
  async fetch(request, env) {
    const url = new URL(request.url);
    url.hostname = "backend.example.com";

    // Cloudflare 自動處理 TLS termination
    return fetch(url, {
      method: request.method,
      headers: request.headers,
      body: request.body,
    });
  },
};

差異重點

  • Caddy 是最簡（4 行），自動 HTTPS
  • Envoy YAML 最冗長，但能 dynamic reconfig（xDS）
  • HAProxy 對 TLS handling 性能最高
  • Cloudflare Workers 是 edge serverless，無 origin 概念

場景 2 · Rate Limit

限制每個 client IP 100 requests/min，附帶 RFC 9331 風格的 RateLimit headers。

Nginxhttp {
    limit_req_zone $binary_remote_addr zone=api:10m rate=100r/m;

    server {
        location /api/ {
            limit_req zone=api burst=20 nodelay;
            limit_req_status 429;

            # Manual RateLimit headers（Nginx 沒原生）
            add_header Retry-After 60;

            proxy_pass http://backend;
        }
    }
}
HAProxyfrontend https_in
    bind *:443 ssl crt /etc/ssl/api.pem
    stick-table type ip size 100k expire 60s store http_req_rate(60s)

    http-request track-sc0 src
    http-request deny deny_status 429 if { sc_http_req_rate(0) gt 100 }

    default_backend api_backend
Envoy（local rate limit）http_filters:
  - name: envoy.filters.http.local_ratelimit
    typed_config:
      "@type": type.googleapis.com/envoy.extensions.filters.http.local_ratelimit.v3.LocalRateLimit
      stat_prefix: rl
      token_bucket:
        max_tokens: 100
        tokens_per_fill: 100
        fill_interval: 60s
      filter_enabled:
        runtime_key: local_rate_limit_enabled
        default_value: { numerator: 100, denominator: HUNDRED }
      response_headers_to_add:
        - append: false
          header: { key: x-local-rate-limit, value: 'true' }
Caddy（rate-limit module）api.example.com {
    rate_limit {
        zone api {
            key {remote_host}
            events 100
            window 1m
        }
    }
    reverse_proxy 10.0.1.5:8080
}
Cloudflare Workers（用 Durable Object 或 KV）export default {
  async fetch(request, env) {
    const ip = request.headers.get("CF-Connecting-IP");
    const key = `rl:${ip}`;
    const count = parseInt(await env.RL.get(key) || "0");

    if (count >= 100) {
      return new Response("Rate limit", {
        status: 429,
        headers: {
          "Retry-After": "60",
          "RateLimit-Limit": "100",
          "RateLimit-Remaining": "0",
        },
      });
    }

    await env.RL.put(key, count + 1, { expirationTtl: 60 });
    return fetch(request);
  },
};

場景 3 · Health Check + Load Balancing

3 個 backend 之間 round-robin，每 5 秒 health check /health，3 次失敗踢除。

Nginx（OSS — passive only；Plus 才有 active）upstream backend {
    server 10.0.1.5:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.6:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.7:8080 max_fails=3 fail_timeout=30s;
}

server {
    location / { proxy_pass http://backend; }
}

# Active health check 需要 Nginx Plus 或 nginx-mod-stream-healthcheck
HAProxy（原生 active health check）backend api_backend
    balance roundrobin
    option httpchk GET /health
    http-check expect status 200

    server srv1 10.0.1.5:8080 check inter 5s fall 3 rise 2
    server srv2 10.0.1.6:8080 check inter 5s fall 3 rise 2
    server srv3 10.0.1.7:8080 check inter 5s fall 3 rise 2
Envoyclusters:
  - name: api_backend
    type: STRICT_DNS
    lb_policy: ROUND_ROBIN

    health_checks:
      - timeout: 2s
        interval: 5s
        unhealthy_threshold: 3
        healthy_threshold: 2
        http_health_check:
          path: "/health"
          expected_statuses:
            - { start: 200, end: 201 }

    load_assignment:
      cluster_name: api_backend
      endpoints:
        - lb_endpoints:
            - endpoint: { address: { socket_address: { address: 10.0.1.5, port_value: 8080 } } }
            - endpoint: { address: { socket_address: { address: 10.0.1.6, port_value: 8080 } } }
            - endpoint: { address: { socket_address: { address: 10.0.1.7, port_value: 8080 } } }
Caddyapi.example.com {
    reverse_proxy 10.0.1.5:8080 10.0.1.6:8080 10.0.1.7:8080 {
        lb_policy round_robin
        health_uri /health
        health_interval 5s
        health_timeout 2s
        fail_duration 30s
        max_fails 3
    }
}
Cloudflare Workers（用 Load Balancer 服務 + Health Check）# 在 Cloudflare Dashboard / API 建立：
# 1. Health Check resource：probe /health every 5s
# 2. Origin Pool：3 個 backend
# 3. Load Balancer：用 origin pool + health check

# Worker 直接 fetch 到 LB hostname，CF 自動分配
export default {
  async fetch(request) {
    const url = new URL(request.url);
    url.hostname = "api-lb.cloudflare.example.com";
    return fetch(url, request);
  },
};

差異重點

  • Nginx OSS 沒原生 active health check（需要 Plus 或第三方 module）— 是常被忽略的限制
  • HAProxy / Envoy / Caddy 都有原生 active health check
  • Cloudflare 用 Dashboard 配置，Workers 只負責 routing logic

場景 4 · Cache（含 stale-while-revalidate）

對 GET request cache 60 秒，stale-while-revalidate 30 秒（過期但未過久時，先回 stale 同時非同步刷新）。

Nginxhttp {
    proxy_cache_path /var/cache/nginx levels=1:2
                     keys_zone=api_cache:10m
                     max_size=1g inactive=60m;

    server {
        location / {
            proxy_pass http://backend;

            proxy_cache api_cache;
            proxy_cache_valid 200 60s;
            proxy_cache_use_stale updating error timeout;
            proxy_cache_background_update on;
            proxy_cache_lock on;

            add_header X-Cache-Status $upstream_cache_status;
        }
    }
}
HAProxy（小型 cache，Varnish 是更專門選擇）cache api_cache
    total-max-size 1024      # MB
    max-object-size 100000    # bytes
    max-age 60                # seconds

frontend https_in
    bind *:443 ssl crt /etc/ssl/api.pem
    http-request cache-use api_cache
    http-response cache-store api_cache
    default_backend api_backend

# 大型 cache 場景建議用 Varnish + HAProxy 組合
Envoy（cache filter — 2024+ 穩定）http_filters:
  - name: envoy.filters.http.cache
    typed_config:
      "@type": type.googleapis.com/envoy.extensions.filters.http.cache.v3.CacheConfig
      typed_config:
        "@type": type.googleapis.com/envoy.extensions.http.cache.simple_http_cache.v3.SimpleHttpCacheConfig
  - name: envoy.filters.http.router

# 注意：Envoy cache 仍相對年輕，complex 場景考慮 Nginx 或 Varnish
Caddy（cache module — 第三方 caddy-cache）api.example.com {
    cache {
        ttl 60s
        stale 30s
    }
    reverse_proxy 10.0.1.5:8080
}
Cloudflare Workers + Cache APIexport default {
  async fetch(request, env, ctx) {
    const cache = caches.default;
    let response = await cache.match(request);

    if (!response) {
      response = await fetch(request);
      response = new Response(response.body, response);
      response.headers.set("Cache-Control", "public, max-age=60, stale-while-revalidate=30");
      ctx.waitUntil(cache.put(request, response.clone()));
    }

    return response;
  },
};

場景 5 · WebSocket / SSE 處理

WebSocket 與 SSE 都需要 long-lived connection、避免 buffer。

Nginx — WebSocketlocation /ws/ {
    proxy_pass http://backend;

    # WebSocket upgrade 必要 headers
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";

    # 長連線 timeout（預設 60s 太短）
    proxy_read_timeout  3600s;
    proxy_send_timeout  3600s;
}

# SSE：禁用 buffer 是關鍵
location /events {
    proxy_pass http://backend;
    proxy_http_version 1.1;
    proxy_set_header Connection "";

    proxy_buffering off;       # 必須關
    proxy_cache off;
    proxy_read_timeout 3600s;

    add_header X-Accel-Buffering no;
}
HAProxy — WebSocket / SSEfrontend https_in
    bind *:443 ssl crt /etc/ssl/api.pem alpn h2,http/1.1

    # WebSocket 路由
    use_backend ws_backend if { path_beg /ws/ }
    # SSE 路由
    use_backend sse_backend if { path_beg /events }
    default_backend api_backend

backend ws_backend
    timeout tunnel 1h      # 長連線
    server srv1 10.0.1.5:8080

backend sse_backend
    option http-no-delay     # 不 buffer
    timeout server 1h
    server srv1 10.0.1.5:8080
Envoy — WebSocket / SSEroute_config:
  virtual_hosts:
    - routes:
        - match: { prefix: "/ws/" }
          route:
            cluster: ws_backend
            upgrade_configs:
              - upgrade_type: websocket
                enabled: true
            timeout: 3600s    # 長連線

        # SSE：disable response timeout、不 buffer
        - match: { prefix: "/events" }
          route:
            cluster: sse_backend
            timeout: 0s        # 0 = 無限
            idle_timeout: 3600s
Caddy — WebSocket / SSEapi.example.com {
    # Caddy 自動處理 WebSocket upgrade
    reverse_proxy /ws/* 10.0.1.5:8080 {
        transport http {
            response_header_timeout 1h
            read_timeout 1h
        }
    }

    # SSE：flush_interval -1 啟用 streaming
    reverse_proxy /events* 10.0.1.5:8080 {
        flush_interval -1
        transport http { read_timeout 1h }
    }
}
Cloudflare Workers — SSE / WebSocket// SSE：直接 stream response
export default {
  async fetch(request) {
    if (request.url.endsWith("/events")) {
      const backend = await fetch("https://backend.example.com/events");
      return new Response(backend.body, {
        headers: {
          "Content-Type": "text/event-stream",
          "Cache-Control": "no-cache",
          "Connection": "keep-alive",
        },
      });
    }

    // WebSocket：使用 Cloudflare 內建 WebSocketPair
    if (request.headers.get("Upgrade") === "websocket") {
      // Forward to Durable Object 或 backend
      return fetch("https://backend.example.com/ws", request);
    }

    return fetch(request);
  },
};

關鍵注意

  • SSE 必須關 buffering（Nginx proxy_buffering off、Caddy flush_interval -1、Envoy idle_timeout）— 否則訊息卡住
  • 長連線 timeout 預設不夠：所有工具預設 60s，要改成至少 1 小時
  • WebSocket 需要 upgrade headers：Nginx 要手動設 Connection / Upgrade，其他工具自動

場景 6 · OAuth / JWT 驗證

檢查 Authorization: Bearer <jwt> header，驗證 JWT 簽章後 forward；驗證失敗回 401。

Nginx — auth_request 子請求模式http {
    server {
        location /api/ {
            auth_request /auth-validate;
            auth_request_set $user $upstream_http_x_user;
            proxy_set_header X-User $user;
            proxy_pass http://backend;
        }

        location = /auth-validate {
            internal;
            proxy_pass http://auth-service/validate;
            proxy_pass_request_body off;
            proxy_set_header Content-Length "";
            proxy_set_header X-Original-URI $request_uri;
            proxy_set_header Authorization $http_authorization;
        }
    }
}

# 進階：用 ngx_http_auth_jwt_module（Plus）或 lua-resty-jwt
HAProxy（用 lua 或外部 SPOE）frontend https_in
    bind *:443 ssl crt /etc/ssl/api.pem

    # 簡單檢查 Authorization header 存在
    http-request deny deny_status 401 if !{ req.hdr(authorization) -m beg "Bearer " }

    # JWT 簽章驗證需要 lua script 或 SPOE：
    # lua-load /etc/haproxy/jwt-validator.lua
    # http-request lua.validate_jwt

    default_backend api_backend
Envoy — JWT filter（原生）http_filters:
  - name: envoy.filters.http.jwt_authn
    typed_config:
      "@type": type.googleapis.com/envoy.extensions.filters.http.jwt_authn.v3.JwtAuthentication
      providers:
        my_provider:
          issuer: https://auth.example.com
          audiences: [api.example.com]
          remote_jwks:
            http_uri:
              uri: https://auth.example.com/.well-known/jwks.json
              cluster: jwks_cluster
              timeout: 5s
            cache_duration: 300s
          forward: true
          payload_in_metadata: jwt_payload

      rules:
        - match: { prefix: /api/ }
          requires: { provider_name: my_provider }
  - name: envoy.filters.http.router
Caddy — JWT module（第三方 caddy-jwt）api.example.com {
    route /api/* {
        jwt {
            primary yes
            trusted_tokens static_secret {
                token_signing_method HS256
                token_secret "your-secret"
            }
            allow user_id "*"
        }
        reverse_proxy 10.0.1.5:8080
    }
}
Cloudflare Workers — JWT 驗證import { jwtVerify, createRemoteJWKSet } from "jose";

const JWKS = createRemoteJWKSet(new URL("https://auth.example.com/.well-known/jwks.json"));

export default {
  async fetch(request, env) {
    const auth = request.headers.get("Authorization");
    if (!auth?.startsWith("Bearer ")) {
      return new Response("Unauthorized", { status: 401 });
    }

    const token = auth.substring(7);
    try {
      const { payload } = await jwtVerify(token, JWKS, {
        issuer: "https://auth.example.com",
        audience: "api.example.com",
      });

      const headers = new Headers(request.headers);
      headers.set("X-User-Id", payload.sub);
      return fetch(request, { headers });
    } catch (e) {
      return new Response("Invalid token", { status: 401 });
    }
  },
};

場景 7 · mTLS（Client Certificate）

要求 client 出示 cert（service mesh、partner API）。

Nginxserver {
    listen 443 ssl;
    server_name api.example.com;

    ssl_certificate     /etc/ssl/server.crt;
    ssl_certificate_key /etc/ssl/server.key;

    # mTLS：要求 client cert
    ssl_client_certificate /etc/ssl/ca.crt;
    ssl_verify_client      on;
    ssl_verify_depth       2;

    location / {
        # 把 client cert info 傳給 backend
        proxy_set_header X-SSL-Client-DN $ssl_client_s_dn;
        proxy_set_header X-SSL-Client-Verify $ssl_client_verify;
        proxy_pass http://backend;
    }
}
HAProxyfrontend mtls_in
    bind *:443 ssl crt /etc/ssl/server.pem ca-file /etc/ssl/ca.crt verify required

    http-request set-header X-SSL-Client-CN %{+Q}[ssl_c_s_dn(CN)]
    http-request set-header X-SSL-Client-DN %{+Q}[ssl_c_s_dn]
    http-request set-header X-SSL-Verify %[ssl_c_verify]

    default_backend api_backend
Envoyfilter_chains:
  - transport_socket:
      name: envoy.transport_sockets.tls
      typed_config:
        "@type": type.googleapis.com/envoy.extensions.transport_sockets.tls.v3.DownstreamTlsContext
        require_client_certificate: true
        common_tls_context:
          tls_certificates:
            - certificate_chain: { filename: /etc/ssl/server.crt }
              private_key: { filename: /etc/ssl/server.key }
          validation_context:
            trusted_ca: { filename: /etc/ssl/ca.crt }
            verify_certificate_hash: []
Caddyapi.example.com {
    tls {
        client_auth {
            mode require_and_verify
            trusted_ca_cert_file /etc/ssl/ca.crt
        }
    }

    reverse_proxy 10.0.1.5:8080 {
        header_up X-SSL-Client-Subject {http.request.tls.client.subject}
    }
}
Cloudflare（API Shield mTLS）# Cloudflare 的 mTLS 在 Dashboard 配置：
# SSL/TLS → Client Certificates → Create Client Certificate
# 然後 hostname 設定 require client cert

# Worker 中讀取 client cert info
export default {
  async fetch(request, env) {
    const clientCert = request.cf?.tlsClientAuth;

    if (clientCert?.certVerified !== "SUCCESS") {
      return new Response("Client cert required", { status: 403 });
    }

    const headers = new Headers(request.headers);
    headers.set("X-Client-Cert-Subject", clientCert.certSubjectDN);
    return fetch(request, { headers });
  },
};

結語

本附錄涵蓋 8 個場景 × 5 工具 = 40 個配置範例。每個都跑得起來，但 production 使用前要：

  1. 調整 timeout / buffer / connection limit給你的流量
  2. 加 monitoring（Prometheus exporter、access log shipper）
  3. 測試 reload / failover（不要假設「config change 不會掉連線」）
  4. review security defaults（如 TLS cipher suite、HSTS、CSP headers）

給工程師的 5 條 Cookbook 原則
  1. 抄之前先讀本書對應章節：知道為何這樣寫才能改
  2. 從最小配置開始：先跑起來、再加複雜度

  4. mTLS / JWT 在 reverse proxy 層處理：應用層只做業務邏輯
