Chapter One · Fundamentals
認識 nvm 與核心概念

什麼是 nvm？

nvm（Node Version Manager）是一套用 POSIX 相容 shell 腳本寫成的 Node.js 版本管理器。它允許你在同一台機器上安裝多個 Node.js 版本，並能以 per-shell（每個 shell 獨立） 的方式切換使用中的版本，完全不需要 sudo。

核心設計哲學

nvm 不是一個可執行的 binary，而是一組被 source 到 shell 裡的函式集合。這意味著：它直接操作你當前 shell 的環境變數（尤其是 PATH），切換版本只是改寫 PATH 的優先順序，而非真的替換系統 node。

為什麼需要版本管理器？

Node.js 演進快速，不同專案常需要不同版本：

  • 舊專案可能卡在 Node 14，新專案需要 Node 22 的新 API
  • LTS（Long Term Support）與 Current 版本的 API 差異
  • CI/CD 需要在多版本上做相容性測試
  • 全域安裝的 npm 套件（如 typescript、eslint）會因 Node 版本不同而行為不一致

重要提醒

在 Windows 上，如果你使用 WSL（Windows Subsystem for Linux）或 Git Bash，請用 nvm-sh/nvm。只有純 Windows 原生環境才用 nvm-windows。兩者不可同時安裝，否則 PATH 會錯亂。

運作原理

nvm 的整個機制可以拆成四步驟：

Shell 啟動 → source nvm.sh → 載入函式 → nvm use X → 改寫 $PATH
  1. Shell 啟動時：~/.bashrc（或 ~/.zshrc）中的片段會把 $NVM_DIR/nvm.sh source 進來。
  2. 函式註冊：nvm 本身變成一個 shell 函式（不是可執行檔），這就是為什麼 which nvm 找不到它，必須用 command -v nvm。
  3. 版本安裝：每個版本的 node 會解壓到 $NVM_DIR/versions/node/vX.Y.Z/，每版有各自的 bin/、lib/node_modules/（含 npm）。
  4. 版本切換：nvm use 會把選定版本的 bin/ 目錄插到 $PATH 最前面，舊版本 PATH 被剝離。

Chapter Two · Installation
安裝、更新與移除

本章示範在各平台安裝 nvm v0.40.4，驗證安裝、更新與完整移除。

一行安裝（curl 或 wget）

ALL官方提供兩種一行安裝方式，擇一即可：

# 方式 A：使用 curl（macOS / 多數 Linux 發行版）
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash

# 方式 B：使用 wget（部分 Linux 伺服器環境）
wget -qO- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash

安裝腳本會：

  1. 把 nvm 以 git clone 到 $NVM_DIR（預設 ~/.nvm）
  2. 偵測你的 shell profile 檔（依序找 ~/.bashrc、~/.bash_profile、~/.zshrc、~/.zprofile、~/.profile）
  3. 在 profile 末端附加兩行載入語句

附加的語句長這樣：

export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"  # 載入 nvm
[ -s "$NVM_DIR/bash_completion" ] && \. "$NVM_DIR/bash_completion"  # 載入 tab 補全

平台專屬注意事項

🍎 macOS

若系統新裝且 ~/.zshrc 不存在：

touch ~/.zshrc
# 再執行一次安裝腳本

若使用 bash，改建立 ~/.bash_profile 或 ~/.bashrc。

🐧 Linux

安裝完後若提示 nvm: command not found，關閉當前終端重開一個，或手動 source：

. ~/.nvm/nvm.sh

若要從原始碼編譯 node，先裝：

sudo apt install -y build-essential libssl-dev
🪟 WSL

等同 Linux 流程。建議使用 WSL 2 以獲得較佳的檔案系統效能。在 WSL 裡安裝的 nvm 與 Windows 原生 Node.js 完全隔離，互不影響。

💻 Git Bash

安裝流程相同，但建議只用預編譯 binary（nvm install 預設行為），避免碰觸 MinGW 編譯問題。

驗證安裝

# ✅ 正確驗證方式
command -v nvm
# 預期輸出：nvm

nvm --version
# 預期輸出：0.40.4

# ❌ 不要這樣驗證
which nvm
# 會失敗！因為 nvm 是 shell 函式，不是執行檔
為什麼 which 不行？

which 只在 $PATH 找可執行檔；command -v 則會檢查 shell 內定義的函式、別名與可執行檔，是 POSIX 標準的做法。

進階安裝：自訂 profile 檔

如果你已經透過 zsh 插件（例如 zsh-nvm）取得 nvm 補全，不希望安裝腳本動你的 profile，可以這樣：

PROFILE=/dev/null bash -c 'curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash'

非互動 shell（Docker 等）特殊處理

在 Docker 容器中，bash 以非互動模式啟動，預設不會載入 ~/.bashrc。nvm 官方建議的做法是使用 BASH_ENV 特殊變數：

# 使用 bash 作為 shell
SHELL ["/bin/bash", "-o", "pipefail", "-c"]

# 建立 bash 互動與非互動皆會 source 的檔案
ENV BASH_ENV ~/.bash_env
RUN touch "${BASH_ENV}"
RUN echo '. "${BASH_ENV}"' >> ~/.bashrc

# 下載並安裝 nvm，指定 PROFILE 為 BASH_ENV
RUN curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | \
    PROFILE="${BASH_ENV}" bash

更新 nvm 本身

最簡單的方法是再跑一次安裝腳本（指定新版本號）。它會自動升級：

# 升級到 v0.40.4
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash

# 之後重新載入 shell 或：
. ~/.nvm/nvm.sh
檢查新版本

訂閱 GitHub Releases：https://github.com/nvm-sh/nvm/releases 或用 nvm --version 與 GitHub 上的最新 tag 比對。

完整移除 nvm

nvm 沒有提供「uninstall self」命令，但移除非常乾淨：

# 1. 從當前 shell 卸載（不影響檔案）
nvm unload

# 2. 移除目錄
rm -rf "$NVM_DIR"
# 或明確寫路徑：
rm -rf ~/.nvm

# 3. 移除 shell profile 裡的載入語句
#    手動編輯 ~/.bashrc / ~/.zshrc，刪除 NVM_DIR 那三行

# 4. 重開終端

安裝後的初始化

剛裝完 nvm，尚未安裝任何 node。先從安裝一個 LTS 開始：

$ nvm install --lts

Downloading and installing node v22.11.0...
Now using node v22.11.0 (npm v10.9.0)

Node.js 每個 LTS 都有一個元素代號，且按字母 A-Z 循環。截至 2026 年初常見的有：

nvm install — 完整語法

nvm install [-s] [<version>] [--reinstall-packages-from=<version>] \
               [--lts] [--lts=<LTS name>] \
               [--skip-default-packages] [--latest-npm] \
               [--no-progress] [--alias=<name>] [--default]

旗標詳解

旗標作用
-s強制從原始碼編譯（即使有預編譯 binary）
--lts安裝最新 LTS
--lts=<codename>安裝指定 LTS 系列最新版（如 --lts=iron）
--reinstall-packages-from=<ver>安裝新版後，從舊版遷移全域 npm 套件
--skip-default-packages忽略 $NVM_DIR/default-packages 中的預設套件清單
--latest-npm安裝完 node 後，把 npm 升到該 node 支援的最新版
--no-progress不顯示下載進度條（CI/CD 友善）
--alias=<name>安裝同時建立別名
--default安裝同時設為 default 別名

實戰範例

# ── 基本：指定版本 ──
$ nvm install 22.11.0            # 精確版本
$ nvm install 22                  # 22 系列最新
$ nvm install 20.18                # 20.18 系列最新 patch

# ── LTS 相關 ──
$ nvm install --lts             # 最新 LTS
$ nvm install --lts=iron        # Node 20 系列最新
$ nvm install --lts=jod         # Node 22 系列最新

# ── 最新 Current ──
$ nvm install node                # 最新（可能不穩定）

# ── 安裝並同步升 npm ──
$ nvm install --lts --latest-npm

# ── 安裝並從舊版遷移全域套件 ──
$ nvm install 22 --reinstall-packages-from=20

# ── 安裝同時設為預設 ──
$ nvm install --lts --default

# ── 強制從源碼編譯（需要編譯工具）──
$ nvm install -s 22.11.0

預編譯 binary vs. 從源碼編譯

nvm 的安裝策略：

  1. 預設行為：先檢查 nodejs.org 是否有你系統對應的預編譯 binary（darwin-arm64、linux-x64 等）。有就直接下載解壓 → 快、穩定。
  2. 無 binary 時：自動 fallback 到下載原始碼並編譯。這時需要 gcc/g++、make、Python、OpenSSL headers。
  3. 強制編譯：加 -s 旗標。適合：架構特殊、想套特定 patch、除錯。
編譯很慢也很吃資源

從源碼編譯 Node.js 在一般筆電上可能耗時 15-40 分鐘，並需要 2GB+ RAM。除非真有需要，請讓 nvm 自己下載 binary。

Chapter Four · Switching & Execution · Deep Dive
版本切換與即時使用

安裝完多個版本後，真正的樂趣是自由切換。本章剖析 use / run / exec / which / current / deactivate 的微妙差異。

nvm use — 切換當前 shell 的版本

nvm use [--silent] [<version> | --lts | --lts=<name>]

實戰：

$ nvm use 22
Now using node v22.11.0 (npm v10.9.0)

$ nvm use --lts
Now using node v22.11.0 (npm v10.9.0)

$ nvm use --lts=iron
Now using node v20.18.0 (npm v10.8.2)

$ nvm use system                    # 切回系統原生 node
Now using system version of node: v18.0.0 (npm v8.6.0)

$ nvm use --silent               # 讀 .nvmrc 但不輸出
per-shell 是關鍵

nvm use 只影響當前 shell session。另開終端分頁會回到 default 別名指向的版本。想讓所有新分頁預設某版本，用 nvm alias default（見第 5 章）。

背後機制

執行 nvm use 22 時發生的事：

  1. 把 22 解析成 v22.11.0
  2. 呼叫內部函式 nvm_change_path()
  3. 從 $PATH 剝離所有舊版 node 的 bin 目錄

專案級配置：.nvmrc 與團隊協作

.nvmrc 是 nvm 的專案契約。當團隊成員 clone 下專案時，能立即知道該用哪個 Node 版本。本章深入 .nvmrc 格式、自動切換 hook、以及團隊協作的最佳實踐。

.nvmrc 檔案規則

.nvmrc 放在專案根目錄（或任何父層目錄），nvm 會向上搜尋。格式規則：

  • 必須有恰好一個版本指定（多了會錯）
  • 支援 # 開頭的註解行
  • 支援 key=value 形式（v0.40.x 新增），但仍須正好一行版本
  • 接受任何 nvm install 接受的版本格式

合法範例

# 最簡單：寫死版本
22.11.0

# 主版號（跟上新 patch）
22

# LTS 名稱
lts/iron

# v 前綴可選
v20.18.0

# 帶註解 + key-value（v0.40.x）
# Node version for this project
# Updated: 2026-01
22.11.0

不合法範例

# 錯：多個版本
22.11.0
20.18.0

自動切換：cd 時自動 nvm use

手動打 nvm use 太煩？下面這些 hook 讓你 cd 進專案時自動切版。

ALLBash 版（加入 ~/.bashrc）

cdnvm() {
  command cd "$@" || return $?
  nvm_path=$(nvm_find_up .nvmrc | tr -d '\n')

  # 如果沒找到 .nvmrc，切回 default
  if [[ ! $nvm_path = *[^[:space:]]* ]]; then
    declare default_version
    default_version=$(nvm version default)
    if [[ $default_version == "N/A" ]]; then
      nvm alias default node
      default_version=$(nvm version default)
    fi
    if [[ $(nvm current) != "$default_version" ]]; then
      nvm use default
    fi
  elif [[ -s "${nvm_path}/.nvmrc" && -r "${nvm_path}/.nvmrc" ]]; then
    declare nvm_version
    nvm_version=$(<"${nvm_path}"/.nvmrc)
    declare locally_resolved_nvm_version
    locally_resolved_nvm_version=$(nvm ls --no-colors "$nvm_version" | tail -1 | tr -d '\-> *' | tr -d '[:space:]')
    if [[ "$locally_resolved_nvm_version" == "N/A" ]]; then
      nvm install "$nvm_version"
    elif [[ $(nvm current) != "$locally_resolved_nvm_version" ]]; then
      nvm use "$nvm_version"
    fi
  fi
}
alias cd='cdnvm'
cd "$PWD"

ALLZsh 版（加入 ~/.zshrc）

autoload -U add-zsh-hook

load-nvmrc() {
  local nvmrc_path
  nvmrc_path="$(nvm_find_nvmrc)"

  if [ -n "$nvmrc_path" ]; then
    local nvmrc_node_version
    nvmrc_node_version=$(nvm version "$(cat "${nvmrc_path}")")

    if [ "$nvmrc_node_version" = "N/A" ]; then
      nvm install
    elif [ "$nvmrc_node_version" != "$(nvm version)" ]; then
      nvm use
    fi
  elif [ "$(nvm version)" != "$(nvm version default)" ]; then
    echo "Reverting to nvm default version"
    nvm use default
  fi
}

Chapter Seven · npm & Global Packages · Deep Dive
npm 與全域套件管理 / 遷移

nvm 的每個 Node 版本都有自己獨立的 npm 與 node_modules。這帶來環境隔離的好處，但也意味著：換版後全域套件（typescript、eslint、yarn 等）都要重裝。本章專治這個痛點。

全域套件的儲存位置

與系統級 node 不同，nvm 管理的全域套件各自獨立：

~/.nvm/versions/node/
├── v20.18.0/
│   └── lib/node_modules/
│       ├── npm/               # 這版的 npm
│       ├── typescript/        # 20 版全域裝的 ts
│       └── eslint/
└── v22.11.0/
    └── lib/node_modules/
        ├── npm/               # 更新的 npm
        └── (空的！必須重裝)

這是為什麼剛 nvm install 新版後 tsc 會 "command not found"。

nvm install-latest-npm — 升級當前版的 npm

Node.js 發布時 bundle 一個 npm，但 npm 獨立發版，通常會落後。用 install-latest-npm 補上：

$ nvm use 20
$ npm -v

$ nvm install-latest-npm

* Installing latest `npm` to Node v20.18.0
...
* npm upgraded to: v11.0.0

$ npm -v
11.0.0
v0.40.x 的改進

v0.40.2 起，nvm_install_latest_npm 改善了版本偵測邏輯（#3564），能更準確找到當前 Node 可支援的最新 npm — 避免把 npm v11 裝到不支援它的舊 Node 上。

也可在 install 時順便升：

nvm install --lts --latest-npm

nvm reinstall-packages — 套件遷移

這是 nvm 最實用的功能之一。從舊版複製全域套件到當前版：

# 新裝 Node 時自動裝清單中的套件
nvm install --lts
# 安裝完 Node 後，nvm 會跑 npm install -g 這些

# 想跳過時
nvm install --lts --skip-default-packages

套件遷移 vs. default-packages 的選擇

策略優點缺點
--reinstall-packages-from 保留舊版實際裝的版本號 只遷一次，且會帶走不需要的套件
default-packages 每次新裝都生效，維護成本低 需手動維護清單

實務上兩者可並用：default-packages 作為長期清單，新版時偶爾用 reinstall-packages 補遷。

npm 版本問題排除

情境：升 Node 後 npm 出現古怪錯誤

# 有時 node 的 bundled npm 對新專案不相容，手動升：
nvm install-latest-npm

# 或指定版本
npm install -g npm@11

情境：EACCES 權限錯誤

好消息

nvm 管理的 node 與 npm 全在 ~/.nvm 底下，不需要 sudo。如果你看到 EACCES 錯誤，八成是你之前用 sudo 裝過系統 node 或 npm，導致 ~/.npm cache 有 root 擁有的檔案：

sudo chown -R "$USER":"$USER" ~/.npm

與 pnpm / yarn 的關係

pnpm 與 yarn 都是建立在 Node 之上的套件管理器，不能替代 nvm：

  • nvm：管理 Node.js 執行環境版本
  • npm / pnpm / yarn：管理專案的 JavaScript 套件

實務上：

# 把 pnpm / yarn 加到 default-packages
echo "pnpm@9" >> ~/.nvm/default-packages
echo "yarn@1" >> ~/.nvm/default-packages

# 或使用 corepack（Node 16.9+ 自帶）
corepack enable
corepack prepare pnpm@latest --activate

疑難排解與常見陷阱

本章是所有用 nvm 的人遲早會需要的章節。針對 10 個高頻問題，給出症狀、根因、解法。

問題 1：nvm: command not found

症狀

$ nvm
zsh: command not found: nvm

根因

nvm 是 shell 函式，必須被 source 進來。當前 shell 沒有載入 nvm.sh。可能情況：

  • 剛裝完沒重開終端
  • shell profile 沒被執行（非互動模式）
  • macOS 新系統沒有 .zshrc
  • 你用 bash 但安裝時 profile 被加到 .zshrc（或反之）

解法

# 解法 1：手動載入（立即生效但不持久）
. ~/.nvm/nvm.sh

# 解法 2：檢查 profile 是否有載入語句
grep -l NVM_DIR ~/.bashrc ~/.bash_profile ~/.zshrc ~/.profile 2>/dev/null

# 解法 3：如果缺，手動加
cat >> ~/.zshrc << 'EOF'
export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"
[ -s "$NVM_DIR/bash_completion" ] && \. "$NVM_DIR/bash_completion"
EOF

# 解法 4：重開終端或 source
. ~/.zshrc

問題 2：which nvm 沒輸出

這不是錯誤

which 只查可執行檔，nvm 是 shell 函式，所以找不到。用：

command -v nvm          # 正確
type nvm                # 也可（顯示函式內容）

問題 3：PATH 衝突（Homebrew / 系統 node）

症狀

$ nvm use 22
Now using node v22.11.0
$ which node
/usr/local/bin/node              # ← 不是 nvm 的！
$ node -v
v18.0.0                           # ← 還是 Homebrew 裝的

根因

Homebrew 或系統 node 的路徑在 $PATH 裡比 nvm 的還前面。

診斷

echo $PATH | tr ':' '\n' | grep -i node
# 看 nvm 的路徑 vs 其他 node 路徑誰在前

解法

# 解法 1（推薦）：移除系統/Homebrew 的 node
brew uninstall node
# 或 Linux：
sudo apt remove nodejs

# 解法 2：確保 nvm 載入在 profile 最後
# 使 nvm 的 PATH 改寫發生在所有其他之後

# 解法 3：重新 use 版本（重新改寫 PATH）
nvm use default
hash -r                          # 刷新 shell 命令快取

問題 4：編譯 node 失敗

症狀

configure: error: C++ compiler cannot create executables
# 或
make: g++: command not found

根因

你的架構/系統沒有預編譯 binary，nvm 嘗試從源碼編譯，但系統缺編譯工具。

解法

🍎 macOS
xcode-select --install
🐧 Debian / Ubuntu
sudo apt install -y \
  build-essential libssl-dev
🐧 RHEL / CentOS / Fedora
sudo dnf groupinstall \
  "Development Tools"
sudo dnf install openssl-devel
🐧 Alpine
apk add build-base \
  openssl-dev python3

問題 5：macOS 新電腦 zsh 預設沒 .zshrc

症狀

裝完 nvm 後，重開終端還是 command not found。

根因

macOS 10.15+ 預設 shell 為 zsh，但不會自動建立 ~/.zshrc，安裝腳本找不到檔案就跳過。

解法

touch ~/.zshrc
# 重跑安裝腳本
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash
# 重開終端

問題 6：bash 用戶但裝到 .zshrc

症狀

用 bash 當預設 shell，但 nvm 寫進了 .zshrc。

解法

# 1. 建立 bash profile
touch ~/.bash_profile # 或 ~/.bashrc

# 2. 複製 nvm 載入片段過去（從 .zshrc 撈）
grep -A1 "NVM_DIR" ~/.zshrc >> ~/.bash_profile

# 3. 重開

問題 7：cd 進專案後 node 版本沒變

原因

沒裝 cd hook。nvm不會自動讀 .nvmrc 切版 — 這需要你自己加 hook（見第 6 章）。

臨時解法

nvm use
# 手動切到 .nvmrc 版

長期解法：加 hook（第 6 章的 bash/zsh/fish 片段）

問題 8：sudo 權限錯亂

症狀

npm ERR! Error: EACCES: permission denied, mkdir '/home/user/.npm/_cacache/...'

根因

你之前用 sudo npm install，導致 ~/.npm 裡有 root 擁有的檔。

sudo chown -R $(whoami) ~/.npm ~/.nvm

# 以後不要用 sudo 跑 npm / nvm！

# 例如，若留 nvm，註解 asdf / volta / fnm 的 init 行

Dockerfile：基於 Ubuntu + nvm（官方範例）

FROM ubuntu:latest

# 安裝 nvm

ENV NVM_DIR=/root/.nvm

# 安裝 node（需要 source nvm.sh）
RUN bash -c "source $NVM_DIR/nvm.sh && nvm install $NODE_VERSION"

# ENTRYPOINT 確保後續命令能找到 node
ENTRYPOINT ["bash", "-c", "source $NVM_DIR/nvm.sh && exec \"$@\"", "--"]

當你需要在 RUN 命令中直接使用 nvm / node（而非每次都 source），用 BASH_ENV：

FROM ubuntu:latest

SHELL ["/bin/bash", "-o", "pipefail", "-c"]

# BASH_ENV 是 bash 在非互動時會 source 的檔案
ENV BASH_ENV=/root/.bash_env
RUN touch "${BASH_ENV}"
RUN echo '. "${BASH_ENV}"' >> ~/.bashrc

RUN apt-get update && \
    apt-get install -y curl && \
    rm -rf /var/lib/apt/lists/*

# 安裝 nvm，並告訴它把 init 寫入 BASH_ENV
RUN curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | \
    PROFILE="${BASH_ENV}" bash

# 從此，每個 RUN 都能直接用 nvm / node
RUN nvm install --lts --no-progress
RUN npm install -g pnpm typescript
RUN node -v && npm -v && pnpm -v

WORKDIR /app
COPY . .
RUN npm ci && npm run build

CMD ["node", "dist/index.js"]

set -euo pipefail

export NVM_DIR="${NVM_DIR:-$HOME/.nvm}"

if [ ! -s "$NVM_DIR/nvm.sh" ]; then
  echo "nvm not found. Installing..."
  curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.4/install.sh | bash
fi

. "$NVM_DIR/nvm.sh"

if [ -f .nvmrc ]; then
  required=$(cat .nvmrc)
  if ! nvm use "$required" 2>/dev/null; then
    echo "Installing Node $required..."
    nvm install "$required" --no-progress
    nvm use "$required"
  fi
fi

echo "Using $(node -v), npm $(npm -v)"

在 package.json 呼叫：

{
  "scripts": {
    "preinstall": "bash scripts/ensure-node.sh"
  }
}

v0.40.x 版本亮點

本附錄整理 v0.40.0 到 v0.40.4 的關鍵變更，幫你判斷是否該升級，以及升級後該注意什麼。

v0.40.4 · 2026-01-29（目前最新）

🔒 安全性修復

  • NVM_AUTH_HEADER 注入防護：修復 wget 路徑下 header 值可能被解釋為參數的問題。若你用私有鏡像 + 認證，強烈建議升級。

🐛 Bug 修復

  • mawk 相容性：處理部分 Linux 系統（Debian 基底的容器）用 mawk 而非 gawk 導致 nvm ls-remote 失敗的問題。
  • portable awk patterns：改寫內部 awk 表達式，避免 GNU-only 語法。
  • nvm_has_colors：改進終端機顏色偵測，在非 TTY 環境（如 pipe）不再誤判。
  • bare lts 拒絕：nvm install lts（無斜線無 codename）現在會明確報錯，而非默默失敗。
升級建議

v0.40.4 應該升級。尤其是：

  • 有用 NVM_AUTH_HEADER 的人（安全性）
  • 在 Alpine / 某些容器環境用 nvm 的人（mawk 相容）
  • 一般用戶也受惠於更明確的錯誤訊息

v0.40.3 · 2025-09-25

✨ 功能

  • zsh ZDOTDIR 支援：當 zsh 使用非預設 ZDOTDIR（把 .zshrc 放在 ~/.config/zsh/ 等自訂路徑），安裝腳本現在能正確找到。

🐛 Bug 修復

  • unbound variable 修復：在啟用 set -u（嚴格模式）的 shell 中，nvm 載入時不再因未設定變數報錯。
  • improve nvmrc search：.nvmrc 向上搜尋邏輯在特殊檔案系統（如 Docker bind mount）下更穩定。

v0.40.2 · 2025-05-11

✨ 功能

  • NPM v11 支援：nvm install-latest-npm 能正確安裝 npm v11（原本會因 Node 版本比對邏輯不識別 v11）。
  • LTS 錯誤訊息改進（#3441）：拼錯 LTS 名稱時會明確指出 Unknown LTS argument，而非默默使用最新版。
  • corepack 跳過（#3544）：reinstall-packages 時自動跳過 corepack，避免與 Node 內建的衝突。
  • 部分 test 改進：CI 測試套件更完整，減少 false positive。

🐛 Bug 修復

  • nvm_install_latest_npm 函式的版本偵測邏輯（#3564）：更準確找到當前 Node 支援的 npm 上限。
  • Git Bash 下的路徑處理。
  • 某些情況下 nvm ls 不顯示 system node 的問題（#1287）。

v0.40.1 · 2024-10-27

主要是小型修復與文件改進

  • shellcheck 相關 lint 改進。
  • 更新 README 範例與 CI/Docker 說明。
  • 內部腳本穩定性修復。

v0.40.0 · 2024-07-23

v0.40.x 系列的起點，相對於 v0.39.x 的主要變化：

✨ 功能

  • Node.js 新版支援：對齊 Node 20、22 的 LTS 代號（iron、jod）。
  • install.sh 改進：更好的 shell 偵測與 profile 自動設定。
  • bash_completion 更新：新版本字串解析支援。

你的 Node 版本不受影響

升級 nvm 本身不會動到已裝的 Node 版本或全域套件。所有 ~/.nvm/versions/node/ 裡的內容原封不動。

未來展望（v0.41 / v1.0）

依 GitHub 倉庫的 roadmap discussion，未來可能的方向：

  • 更完整的 .nvmrc 格式（允許 engines 同步讀取）
  • 更快的 shell 啟動（內建 lazy-load）
  • Windows 原生支援（目前還是依賴 WSL/Git Bash，社群呼聲很高）
  • 可能的 v1.0（API 穩定後）

但 nvm 維護者一向保守、以穩定優先，這些變化不會來得太快。
