CHAPTER 18 · PART 5

多直譯器並行

Python 3.14 的另一項重磅新特性：concurrent.interpreters（PEP 734）。它讓你能在同一進程內建立多個相互隔離的 Python 直譯器，每個有獨立的 GIL，可以真正並行執行。這是繼 threading、multiprocessing、asyncio 之後的第四種並行模型 —— 介於「執行緒」與「進程」的折衷方案。

Python 3.14 · PEP 734 全新標準庫 子直譯器並行 預估閱讀 22 分鐘

§0章節導論

子直譯器（sub-interpreter）這個功能其實已在 CPython 存在超過 20 年，但一直只能透過 C API 使用。PEP 684（3.12）讓每個子直譯器擁有獨立 GIL，實現真正並行；PEP 734（3.14）把高階 API 引入標準庫，讓純 Python 開發者也能使用。

🆕 3.14 新標準庫 concurrent.interpreters

這個模組讓你能在 Python 程式內建立多個彼此隔離的直譯器，支援類似 Erlang Actor Model / Go CSP 的並發風格。是 Python 並行工具箱的重大擴充。

§18.1什麼是子直譯器？

「直譯器」（interpreter）= Python 執行上下文的完整副本，包含：

  • 獨立的 import 狀態（每個直譯器自己 import 模組）
  • 獨立的 builtins、全域變數
  • 獨立的 GIL（3.12+）
  • 獨立的異常狀態、bytecode 執行迴圈

想像它是「比執行緒更隔離、比進程更輕量」的東西：

特性執行緒子直譯器進程
隔離程度共享全部記憶體預設隔離完全隔離
GIL共用一個各自獨立各自獨立
建立成本低中（~10-50ms）高
記憶體使用最省中最高
通訊共享變數Queue（有限型別）pickle / pipe
並行 CPU bound❌（GIL）✅✅

§18.2三種 CPU 並行方式比較

python 
# 1. multiprocessing — 最成熟，跨平台，繞過 GIL
#    ✅ 完全隔離  ❌ 啟動慢  ❌ pickle 開銷大

# 2. Free-threaded Python (3.14) — 無 GIL 的 threading
#    ✅ 最輕量  ❌ 單執行緒慢 5-10%  ❌ C 擴充需相容

# 3. concurrent.interpreters (3.14) — 子直譯器並行
#    ✅ 同進程 → 輕量、啟動快  ✅ 各自 GIL → 可並行
#    ✅ 強隔離適合外掛  ❌ 共享資料有限 ❌ 生態系剛起步

§18.3基本 API

python 
from concurrent import interpreters

# 建立新的子直譯器
interp = interpreters.create()

# 在子直譯器中執行字串形式的程式碼
interp.exec("""
import sys
print(f"Hello from interpreter {id(sys)}")
x = 42
""")

# 在子直譯器中執行函式
def worker(n):
    return sum(i * i for i in range(n))

result = interp.call(worker, 1_000_000)
print(result)

# 關閉
interp.close()

# 取得目前直譯器資訊
current = interpreters.get_current()
main = interpreters.get_main()
print(interpreters.list_all())
⚠️ 每個子直譯器要重新 import

子直譯器有自己的 import 狀態。主直譯器 import 的模組，子直譯器看不到。必須在子直譯器內重新執行 import，或在 exec 傳入的程式碼裡 import。

§18.4配合執行緒才是真正並行

關鍵觀念：建立子直譯器不會自動啟動新執行緒。要真正並行執行，必須把 interp.call(fn) 放進 threading.Thread 中 —— 每個執行緒用自己的直譯器（各自獨立 GIL）。

python 
from concurrent import interpreters
import threading
import time

def cpu_heavy(n):
    return sum(i * i for i in range(n))

def run_in_subinterp(n, results, idx):
    interp = interpreters.create()
    try:
        results[idx] = interp.call(cpu_heavy, n)
    finally:
        interp.close()

N = 4
results = [None] * N
threads = []

t0 = time.perf_counter()
for i in range(N):
    t = threading.Thread(target=run_in_subinterp,
                         args=(5_000_000, results, i))
    t.start()
    threads.append(t)

for t in threads:
    t.join()

print(f"總耗時：{time.perf_counter()-t0:.2f}s")
print(results)

# 因為每個子直譯器有獨立 GIL，4 個執行緒能真正並行
# 4 核 CPU 上可接近 4x 加速（不像標準 threading 被 GIL 限制）

§18.5Queue 跨直譯器通訊

子直譯器隔離記憶體，不能直接共享變數。通訊要用 interpreters.create_queue()，這是個 thread-safe + 跨直譯器安全 的佇列。

python 
from concurrent import interpreters

# 在主直譯器建立 queue
q = interpreters.create_queue()

# 建立子直譯器
interp = interpreters.create()

# 把 queue 物件「放入」子直譯器的命名空間
interp.prepare_main(q=q)

# 在子直譯器內使用 q
interp.exec("""
for i in range(5):
    q.put(i * 10)
q.put(None)  # 結束哨兵
""")

# 從主直譯器讀取
while True:
    item = q.get()
    if item is None: break
    print(item)
# 0, 10, 20, 30, 40

§18.6可共享型別（Shareable Types）

能透過 Queue 或 prepare_main 跨直譯器傳遞的型別有限（截至 3.14）：

可共享說明
None—
bool—
int—
float—
str—
bytes—
memoryview共享二進位緩衝區（零拷貝）
tuple元素也必須都是可共享的
interpreters.Queue—
⚠️ 無法直接傳遞的東西
  • list、dict、set、自訂類別實例 — 不能直接傳
  • 解法：序列化為 JSON 字串、pickle bytes、或拆成基本型別
  • 注意：在子直譯器內無法直接 unpickle 含自訂類別的資料（因為 import 狀態獨立）
python 
# 傳複雜物件：先 JSON 序列化
import json
from concurrent import interpreters

q = interpreters.create_queue()
interp = interpreters.create()
interp.prepare_main(q=q)

# 主直譯器 → 子直譯器
data = {"users": ["alice", "bob"], "count": 2}
q.put(json.dumps(data))     # 轉為 str 才能跨直譯器

interp.exec("""
import json
raw = q.get()
data = json.loads(raw)
print(data["users"])
""")

§18.7InterpreterPoolExecutor

在 concurrent.futures 中有新的 InterpreterPoolExecutor，把熟悉的 Executor API 套用到子直譯器上。這是最好上手的方式。

python 
from concurrent.futures import InterpreterPoolExecutor

def compute(n):
    return sum(i * i for i in range(n))

# API 與 ThreadPoolExecutor / ProcessPoolExecutor 一致！
with InterpreterPoolExecutor(max_workers=4) as pool:
    results = list(pool.map(compute, [10_000_000] * 4))
    print(results)

# submit 方式
with InterpreterPoolExecutor() as pool:
    futures = [pool.submit(compute, 10_000_000) for _ in range(4)]
    for f in futures:
        print(f.result())

# 相比 ThreadPoolExecutor
# - InterpreterPool：4 個核心真正並行（各自 GIL）
# - ThreadPool：被 GIL 限制，CPU 密集任務無法並行
# 相比 ProcessPoolExecutor
# - InterpreterPool：同進程，啟動快，記憶體少
# - ProcessPool：完全隔離，相容性最好
💡 傳給子直譯器的 callable 限制

傳給 InterpreterPoolExecutor 的函式必須是「可序列化」的（類似 multiprocessing）：

  • ✅ 模組級定義的函式
  • ✅ 純函式（不依賴外部狀態）
  • ❌ lambda
  • ❌ 巢狀定義的函式
  • ❌ 引用閉包變數的函式

§18.8適用情境

✅ 很適合

  • 外掛沙箱：執行使用者提供的程式碼，避免汙染主程式狀態
  • CPU 並行運算：想繞過 GIL 但不想用 multiprocessing 的 pickle 開銷
  • 多租戶服務：每個租戶獨立直譯器，強制狀態隔離
  • Actor / CSP 模型：類似 Erlang、Go 的並發模型
  • 嵌入 Python 的 C 應用：在同進程中跑多個腳本

❌ 不適合

  • I/O bound 任務：應用 asyncio 或 threading 更輕量
  • 大量共享資料：跨直譯器傳遞資料有成本與限制
  • 相依複雜 C 擴充：很多 C 擴充還不支援多直譯器（numpy、pandas 等仍在進行中）
  • 需要相容 3.13 或以前：這是 3.14 的新功能

§18.9限制與注意事項

⚠️ 生態系尚在起步
  • API 仍可能微調：PEP 734 強調現在是「基礎 API」，未來會新增高階工具
  • C 擴充相容性：很多 PyPI 套件還不支援多直譯器（需遵循 Isolating Extension Modules 指南）
  • 建立成本：每個直譯器約 10-50ms，不適合非常短任務
  • 記憶體：雖比進程省，但比執行緒多 —— 不要無限制建立
  • 除錯工具：跨直譯器的堆疊追蹤、profiler 支援還在發展
  • 安全性：不是完整沙箱 —— 官方明確提醒「不要用於安全敏感隔離」

與 Free-threaded 混用？

可以！Free-threaded build + concurrent.interpreters 是 2026 年 Python 並行的最強組合 —— 每個子直譯器內又可以多執行緒並行。不過目前還是前沿領域，生態系支援有限，正式專案導入前請充分測試。
