CHAPTER 20 · PART 6

測試與除錯

沒有測試的程式碼是「過不了明天」的程式碼。本章帶你建立現代 Python 測試流程：用 pytest 寫測試、用 mock 隔離外部相依、用 coverage 量測完整度，遇到 bug 時用 pdb 與 3.14 新增的遠端除錯功能快速定位。

Python 3.14 pytest · mock · pdb 預估閱讀 25 分鐘

§0章節導論

本章採用 pytest 為主（業界標準，比內建 unittest 簡潔許多）。你會學到：

  • 寫測試的基本習慣與命名慣例
  • pytest fixtures 管理測試資源與前置條件
  • parametrize 一次跑多個測試資料
  • unittest.mock 隔離外部服務（資料庫、API、時間）
  • pdb 與 breakpoint() 互動除錯
  • Python 3.14 的遠端除錯新功能（sys.remote_exec）

§20.1assert 基礎

python 
# 基本斷言
assert len(lst) > 0
assert age >= 0, f"年齡不能為負：{age}"   # 帶訊息

# 在測試中使用
def test_add():
    assert add(1, 2) == 3
    assert add(-1, 1) == 0

# pytest 會「改寫 assert」讓錯誤訊息清楚
# 不用寫 self.assertEqual 這種冗長語法
⚠️ assert 不是錯誤處理

用 python -O 執行時 assert 會全部被移除！所以：

  • ✅ 測試、開發期不變條件（invariant）檢查
  • ❌ 使用者輸入驗證 → 改用 raise ValueError
  • ❌ 安全相關檢查 → 不能靠 assert

§20.2pytest 入門

bash 
# 安裝（推薦用 uv）
uv add --dev pytest pytest-cov

# 專案結構
my-project/
├── src/myapp/
│   ├── __init__.py
│   └── core.py
├── tests/
│   ├── __init__.py
│   ├── test_core.py      # 檔名必須 test_ 開頭
│   └── conftest.py       # 共用 fixtures
└── pyproject.toml
python · tests/test_core.py 
from myapp.core import add, divide
import pytest

# 測試函式必須 test_ 開頭
def test_add_positive():
    assert add(1, 2) == 3

def test_add_negative():
    assert add(-5, 3) == -2

# 測試例外
def test_divide_by_zero():
    with pytest.raises(ZeroDivisionError):
        divide(10, 0)

    # 檢查錯誤訊息
    with pytest.raises(ValueError, match=r"must be positive"):
        func(-1)

# 測試類別（可分組相關測試）
class TestCalculator:
    def test_add(self):
        assert add(1, 1) == 2

    def test_sub(self):
        assert sub(5, 3) == 2

# 跳過 / 預期失敗
@pytest.mark.skip(reason="尚未實作")
def test_new_feature(): ...

@pytest.mark.skipif(sys.platform == "win32", reason="僅 Unix")
def test_symlink(): ...

@pytest.mark.xfail
def test_known_bug(): ...
bash · 執行 
pytest                          # 跑全部
pytest tests/test_core.py       # 特定檔案
pytest -k "add"                 # 只跑名稱含 "add"
pytest tests/test_core.py::test_add   # 單一測試
pytest -v                       # 詳細
pytest -x                       # 遇錯即停
pytest --lf                     # 只跑上次失敗的
pytest -s                       # 顯示 print 輸出
pytest -n 4                     # 平行執行（需 pytest-xdist）

§20.3Fixtures —— 測試資源管理

Fixtures 是 pytest 的核心概念 —— 測試前的準備 + 測試後的清理 都在一個地方定義。

python 
import pytest

@pytest.fixture
def sample_data():
    return {"users": ["alice", "bob"], "count": 2}

# 測試函式「要求」fixture — 透過參數名
def test_user_count(sample_data):
    assert sample_data["count"] == 2

# 帶 setup + teardown 的 fixture
@pytest.fixture
def db_connection():
    conn = create_connection("test.db")
    yield conn                         # ← 這裡是測試執行的地方
    conn.close()                       # 測試後清理

def test_query(db_connection):
    result = db_connection.query("...")
    assert result is not None

Fixture scope（生命週期）

python 
@pytest.fixture(scope="function")    # 預設：每個測試函式重建
@pytest.fixture(scope="class")       # 每個測試類別
@pytest.fixture(scope="module")      # 每個檔案一次
@pytest.fixture(scope="session")     # 整個測試跑一次

@pytest.fixture(scope="session")
def shared_db():
    # 昂貴的資源建一次就好
    db = create_test_db()
    yield db
    db.drop()

python · tests/conftest.py

§20.4parametrize —— 一次跑多筆測試

python 
@pytest.mark.parametrize("a, b, expected", [
    (1, 2, 3),
    (0, 0, 0),
    (-1, 1, 0),
    (100, 200, 300),
])
def test_add(a, b, expected):
    assert add(a, b) == expected

# pytest 會為每筆產生獨立測試：
# test_add[1-2-3] ✅
# test_add[0-0-0] ✅
# test_add[-1-1-0] ✅
# test_add[100-200-300] ✅

# 多維度組合
@pytest.mark.parametrize("x", [1, 2])
@pytest.mark.parametrize("y", [10, 20])
def test_combo(x, y):         # 4 組組合
    ...

# 命名測試案例（便於辨識）
@pytest.mark.parametrize("input, expected", [
    pytest.param("", "", id="empty"),
    pytest.param("abc", "ABC", id="lowercase"),
    pytest.param("123", "123", id="digits"),
])
def test_upper(input, expected):
    assert input.upper() == expected

§20.5mock 與 patch —— 隔離外部相依

測試不應該真的打 API、寫檔、等時間。用 unittest.mock 替換掉這些「重」或「不確定」的部分。

python 
from unittest.mock import Mock, MagicMock, patch

# 基本 Mock 物件
m = Mock()
m.method(1, 2)              # 自動接受任何呼叫
m.method.return_value = 42   # 設定回傳值
m.method(1, 2)              # → 42

# 檢查呼叫情況
m.method.assert_called()
m.method.assert_called_once()
m.method.assert_called_with(1, 2)
m.method.call_count       # 2
m.method.call_args_list    # 所有呼叫的參數

# 設定拋例外
m.fetch.side_effect = ConnectionError("timeout")
m.fetch()                    # 🚫 ConnectionError

# side_effect 也可放序列（每次呼叫依序回傳）
m.get.side_effect = [1, 2, 3]
m.get(); m.get(); m.get()  # 1, 2, 3

patch —— 替換真實物件

python 
# 假設我們要測的程式：
# myapp/weather.py
import requests
def get_temp(city):
    r = requests.get(f"https://api/weather/{city}")
    return r.json()["temp"]

# tests/test_weather.py
from unittest.mock import patch
from myapp.weather import get_temp

@patch("myapp.weather.requests.get")    # ← 注意要 patch 使用者那邊
def test_get_temp(mock_get):
    mock_get.return_value.json.return_value = {"temp": 25}
    assert get_temp("Taipei") == 25
    mock_get.assert_called_with("https://api/weather/Taipei")

# 或用 context manager
def test_get_temp_cm():
    with patch("myapp.weather.requests.get") as mock_get:
        mock_get.return_value.json.return_value = {"temp": 25}
        assert get_temp("Taipei") == 25

# Patch 屬性
@patch.object(Database, "query", return_value=[])
def test_empty(mock_query):
    ...

# Patch 環境變數 / dict
@patch.dict(os.environ, {"API_KEY": "test"})
def test_with_env(): ...
⚠️ patch 的常見錯誤

「在對的地方 patch」—— 要 patch 的是「使用端的匯入路徑」，不是「定義路徑」。
若 myapp/x.py 有 from requests import get，就要 patch "myapp.x.get" 而非 "requests.get"。

§20.6coverage 覆蓋率

bash 
uv add --dev pytest-cov

# 執行測試並產生覆蓋率報告
pytest --cov=src

# HTML 詳細報告
pytest --cov=src --cov-report=html
# 打開 htmlcov/index.html

# 要求最低覆蓋率（CI 用）
pytest --cov=src --cov-fail-under=80

# 顯示未覆蓋的行號
pytest --cov=src --cov-report=term-missing
toml · pyproject.toml 
[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = "-v --tb=short"

[tool.coverage.run]
source = ["src"]
omit = ["*/tests/*"]

[tool.coverage.report]
exclude_lines = [
    "pragma: no cover",
    "raise NotImplementedError",
    "if TYPE_CHECKING:",
]
💡 覆蓋率迷思
  • 90%+ 覆蓋率不等於「測試品質好」
  • 要求測試「有意義的行為」而非「每行都跑過」
  • 商業邏輯、錯誤分支的覆蓋最重要
  • getter/setter 等瑣碎程式碼不必強求

§20.7pdb 互動除錯

python 
# 在程式中設中斷點（3.7+）
def process(data):
    result = []
    for item in data:
        breakpoint()       # ← 程式執行到這裡會停下來
        result.append(transform(item))
    return result

# 或在命令列後掛 pdb
# python -m pdb myscript.py

# pytest 失敗時進入 pdb
# pytest --pdb

指令意義
nnext — 執行下一行

💡 更好用的替代

試試 ipdb（IPython 版本，有顏色與自動完成）或在 VS Code / PyCharm 中用圖形化 debugger —— 設中斷點、變數監看都比 pdb 舒服得多。

§20.8🆕 遠端除錯（Python 3.14）

🆕 Python 3.14 新功能

Python 3.14 新增了 sys.remote_exec()：可以對執行中的 Python 進程「注入」程式碼執行。這為跨進程除錯、生產環境診斷打開了新可能。

python 
import sys

# 對另一個正在執行的 Python 進程注入程式碼
# （需要權限，且需要兩個進程都是 3.14+）
sys.remote_exec(target_pid, "path/to/script.py")

# 這在生產除錯非常有用：
# - 服務卡住時注入診斷腳本，dump 堆疊
# - 取得即時狀態快照
# - 配合 -X context_aware_warnings 等 3.14 新 flag 使用

# 配合 pdb 使用：向遠端進程注入一段讓它進入 pdb 的程式碼

pdb 本身在 3.14 也新增了 遠端附加（remote attach）功能，可用 python -m pdb -p <pid>（類似 C 世界的 gdb attach），這是 Python 生態圈邁向「生產級可觀測性」的一大步。

§20.9最佳實踐

測試哲學：FIRST 原則

  • Fast — 測試要快，幾秒鐘內跑完
  • Isolated — 測試間不互相依賴
  • Repeatable — 任何環境都產生相同結果
  • Self-validating — 自動判斷 pass/fail，不需人工檢查 log
  • Timely — 開發同時就寫測試

AAA 模式：寫測試的三段結構

python 
def test_user_registration():
    # Arrange — 準備資料
    email = "alice@example.com"
    password = "secure123"

    # Act — 執行被測試的動作
    user = register(email, password)

    # Assert — 驗證結果
    assert user.email == email
    assert user.check_password(password)
    assert user.id is not None

測試金字塔

  • 單元測試（70%）：快、獨立、多，測單一函式/類別
  • 整合測試（20%）：測模組間協作（含真 DB / 外部 API）
  • E2E 測試（10%）：測完整流程，慢但覆蓋全

整合測試可用 testcontainers（用 Docker 啟動真實 DB）；E2E 測試可用 Playwright / Selenium。
