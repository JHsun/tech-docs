CHAPTER 16 · PART 5

多執行緒與多處理

Python 並行程式設計向來受 GIL 限制 —— 直到 Python 3.14 正式支援 Free-threaded（無 GIL）建置。本章帶你理解 threading、multiprocessing 的差異、共通的 concurrent.futures 抽象，以及 2026 年最熱門的話題：真正的多核心並行 Python。

Python 3.14 · PEP 779 新特性 並行程式設計 預估閱讀 25 分鐘

§0章節導論

Python 的並行工具三大類：

  • threading — 執行緒，適合 I/O bound（網路、磁碟）
  • multiprocessing — 多進程，適合 CPU bound（繞過 GIL）
  • asyncio（下一章）— 單執行緒協程，高並發 I/O 之王

加上 3.14 的兩個新武器：

  • 🆕 Free-threaded Python（PEP 779）—— 無 GIL，讓 threading 真正並行
  • 🆕 concurrent.interpreters（PEP 734）—— 下章介紹

§16.1GIL 解密

GIL（Global Interpreter Lock）是 CPython 用來保護記憶體存取的全域鎖，確保同時只有一個執行緒執行 Python bytecode。

📘 GIL 的意義
  • 優點：簡化 C 擴充開發、避免 race condition、垃圾回收容易
  • 代價：多執行緒 CPU 密集型任務無法真正並行（只能輪流跑）
  • I/O 操作會釋放 GIL，所以 threading 對 I/O bound 任務仍有效

GIL 對效能的影響

python 
import threading
import time

def cpu_bound(n):
    sum(i*i for i in range(n))

# 單執行緒：10M 迴圈 2 次
t0 = time.perf_counter()
cpu_bound(10_000_000)
cpu_bound(10_000_000)
print(f"單執行緒：{time.perf_counter()-t0:.2f}s")

# 多執行緒：兩個執行緒並行
t0 = time.perf_counter()
t1 = threading.Thread(target=cpu_bound, args=(10_000_000,))
t2 = threading.Thread(target=cpu_bound, args=(10_000_000,))
t1.start(); t2.start()
t1.join(); t2.join()
print(f"多執行緒：{time.perf_counter()-t0:.2f}s")

# 在有 GIL 的 Python 3.14 建置上：
# 單執行緒：~1.0s
# 多執行緒：~1.0s   ← 完全沒快！GIL 造成序列執行

# 在 Free-threaded 3.14 上：
# 單執行緒：~1.1s（5-10% 單執行緒懲罰）
# 多執行緒：~0.55s  ← 接近真正 2 倍加速！

§16.2threading 基礎

python 
import threading
import time

def worker(name, n):
    for i in range(n):
        print(f"{name}: {i}")
        time.sleep(0.1)

# 建立並啟動執行緒
t1 = threading.Thread(target=worker, args=("A", 3))
t2 = threading.Thread(target=worker, args=("B", 3))
t1.start()
t2.start()

# 等待完成
t1.join()
t2.join()
print("全部完成")

# Daemon 執行緒：主程式結束時自動終止
t = threading.Thread(target=worker, daemon=True)
t.start()

繼承 Thread 類別

python 
class Worker(threading.Thread):
    def __init__(self, name, n):
        super().__init__()
        self.name = name
        self.n = n

    def run(self):           # start() 會呼叫 run()
        for i in range(self.n):
            print(f"{self.name}: {i}")

w = Worker("X", 5)
w.start()
w.join()

§16.3同步原語：Lock、RLock、Semaphore

多執行緒共享資料時，必須用鎖避免 race condition。

Lock —— 基本互斥鎖

python 
# ❌ 沒鎖的 race condition
counter = 0
def increment():
    global counter
    for _ in range(100_000):
        counter += 1         # 讀+改+寫 非原子！

threads = [threading.Thread(target=increment) for _ in range(10)]
for t in threads: t.start()
for t in threads: t.join()
print(counter)   # 可能 < 1_000_000（資料競爭）

# ✅ 用 Lock 保護
counter = 0
lock = threading.Lock()

def safe_increment():
    global counter
    for _ in range(100_000):
        with lock:                 # 自動 acquire/release
            counter += 1

RLock / Semaphore / Event / Condition

python 
# RLock — 可重入鎖（同一個執行緒可重複 acquire）
rlock = threading.RLock()
def foo():
    with rlock:
        bar()
def bar():
    with rlock:     # 同一執行緒再次 acquire — OK（Lock 會死鎖）
        pass

# Semaphore — 限制同時執行數量（如限制連線池）
sem = threading.Semaphore(3)      # 最多 3 個並發
def access_api():
    with sem:
        request("...")

# Event — 旗標型事件通知
event = threading.Event()
def waiter():
    event.wait()          # 阻塞直到被 set
    print("收到訊號")
# 在另一處：
event.set()               # 通知所有 waiter
⚠️ 死鎖（Deadlock）

兩個執行緒各自持有一把鎖、同時等待對方的鎖 → 永遠卡住。預防：(1) 一律用 with lock:、(2) 多鎖時一律按固定順序取得、(3) 用 lock.acquire(timeout=n) 加超時。

§16.4queue — 執行緒安全的佇列

與其手動管理鎖，不如用 queue.Queue。它是 thread-safe 的，內建同步邏輯，實作 Producer-Consumer 模式最輕鬆。

python 
import queue
import threading
import time

q = queue.Queue(maxsize=10)     # 最多放 10 個

def producer():
    for i in range(20):
        q.put(i)              # 滿了會阻塞
        print(f"produced {i}")
    q.put(None)              # 哨兵（sentinel），告知結束

def consumer():
    while True:
        item = q.get()         # 空時會阻塞
        if item is None: break
        print(f"consumed {item}")
        q.task_done()

threading.Thread(target=producer).start()
threading.Thread(target=consumer).start()

# 其他佇列型別
# queue.LifoQueue   — 後進先出（stack）
# queue.PriorityQueue — 優先佇列

§16.5concurrent.futures — 高階並行 API

這是最推薦的日常工具：統一介面，可以用 Thread 或 Process 執行。

python 
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

def fetch(url):
    import urllib.request
    with urllib.request.urlopen(url) as r:
        return len(r.read())

urls = ["https://python.org", "https://pypi.org", "https://docs.python.org"]

# 方式 1：map（最簡單，按順序回傳結果）
with ThreadPoolExecutor(max_workers=5) as pool:
    for size in pool.map(fetch, urls):
        print(size)

# 方式 2：submit + as_completed（任一完成就立刻處理）
from concurrent.futures import as_completed
with ThreadPoolExecutor(max_workers=5) as pool:
    futures = {pool.submit(fetch, url): url for url in urls}
    for future in as_completed(futures):
        url = futures[future]
        try:
            size = future.result()
            print(f"{url}: {size} bytes")
        except Exception as e:
            print(f"{url} 失敗: {e}")

# CPU bound 任務 → 換成 ProcessPoolExecutor（相同 API）
with ProcessPoolExecutor() as pool:
    results = list(pool.map(heavy_compute, data))
🆕 Python 3.14 新增
  • ProcessPoolExecutor 在 Linux 預設從 fork 改為 forkserver（更安全）
  • 新增 terminate_workers() 與 kill_workers() 方法
  • 新增 InterpreterPoolExecutor（結合 concurrent.interpreters，下一章介紹）

§16.6multiprocessing

透過建立子進程繞過 GIL，真正並行。但每個進程有獨立記憶體，通訊需要序列化。

python 
import multiprocessing as mp

def heavy_task(n):
    return sum(i*i for i in range(n))

if __name__ == "__main__":     # ⚠️ 必須！否則 Windows/macOS 會無限遞迴
    # 用 Pool 處理
    with mp.Pool(processes=4) as pool:
        results = pool.map(heavy_task, [10_000_000] * 4)
        print(results)

    # 手動 Process
    p = mp.Process(target=heavy_task, args=(1_000_000,))
    p.start()
    p.join()

    # 進程間通訊：Queue / Pipe
    q = mp.Queue()
    q.put({"data": 42})

    # 共享記憶體（避免 pickle 開銷）
    value = mp.Value('i', 0)            # 共享 int
    array = mp.Array('i', [0, 1, 2])     # 共享陣列
⚠️ multiprocessing 注意事項
  • 一律用 if __name__ == "__main__": 包住（否則 spawn 時會無限遞迴）
  • 傳遞的參數必須可 pickle（lambda、巢狀函式不行）
  • 進程建立成本高，適合粗粒度長時間任務
  • 3.14 起 Linux 預設 forkserver，原 fork 需要明確指定

§16.7🆕 Free-threaded Python（3.14 · PEP 779）

🆕 Python 3.14 最重要的並行變革

從 Python 3.14 開始，Free-threaded 建置（無 GIL 版本）從實驗階段進入「正式支援」（PEP 779 Phase II）。它是與標準 CPython 並存的一個獨立建置，可讓 threading 在多核心上真正並行執行 CPU bound 任務。

Free-threaded 三階段路線圖

階段版本狀態
Phase I3.13實驗性建置
Phase II3.14（目前）正式支援，但仍為選用建置
Phase III預計 2027-2028Free-threaded 成為預設建置

安裝 Free-threaded 建置

效能比較

任務類型普通 3.14Free-threaded 3.14

多執行緒 CPU bound（4 核）~1x（受 GIL 限制）✅ ~3-4x 加速

Free-threaded 的注意事項

python 
# 1. Python 語言層面 thread-safe（不會 crash），但邏輯 race condition 仍需自己處理
# 例如這段在 Free-threaded 下仍可能 race：
counter += 1       # 不是原子操作

# ✅ 仍需要 Lock / 原子計數器
import threading
lock = threading.Lock()
with lock:
    counter += 1

# 2. 檢查 GIL 是否停用
import sys
if not sys._is_gil_enabled():
    print("Free-threaded 模式！")
⚠️ 生態系相容性
  • C 擴充（numpy、pandas、pytorch 等）需要針對 no-GIL 重新編譯
  • numpy 2.3+、pytorch 2.5+ 已支援；pandas 3.0 規劃中
  • 純 Python 套件基本無需修改
  • 正式專案導入前務必實測，並查閱 py-free-threading.github.io 的套件相容矩陣

§16.8選擇決策樹

情境推薦方案原因
高並發網路 I/Oasyncio（下章）單執行緒，高吞吐
少量並行 I/O（檔案、DB）ThreadPoolExecutor簡單、GIL 在 I/O 時釋放
CPU bound（傳統 Python）ProcessPoolExecutor繞過 GIL
CPU bound（3.14+ 實驗）Free-threaded + ThreadPoolExecutor真正並行、無需進程開銷
需要完全隔離（外掛、沙箱）concurrent.interpreters（下章）同進程但隔離狀態
共享大量資料threading（而非 multiprocessing）避免 pickle 開銷
長時間執行的背景服務Thread daemon 或 asyncio.Task方便管理生命週期

🎯 選擇決策
  • I/O bound → asyncio / Thread
  • CPU bound（舊） → Process
  • CPU bound（3.14）→ Free-threaded + Thread
  • 外掛隔離 → Interpreters
  • 共享大資料 → Thread
⚠️ 常見陷阱
  • GIL 讓 CPU thread 無法加速
  • mp 沒加 if __name__
  • mp 傳 lambda → pickle 錯
  • 多鎖取得順序不一 → deadlock
  • Free-threaded 仍需鎖
  • C 擴充需 no-GIL 版本
