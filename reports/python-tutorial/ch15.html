模組與套件

寫超過 100 行就要開始考慮模組化，寫超過 1000 行就要分成套件。本章解析 Python 的 import 系統、__init__.py 的角色、模組與套件的差異，以及現代專案的標準目錄結構。

本章重點：

  • 模組（module）= .py 檔；套件（package）= 含 __init__.py 的資料夾

§15.1模組基礎

Python 中，任何 .py 檔都是一個模組。

python · math_utils.py 
"""數學工具模組"""

PI = 3.14159

def circle_area(r):
    return PI * r ** 2

def circle_perimeter(r):
    return 2 * PI * r

class Circle:
    def __init__(self, r):
        self.r = r
python · main.py 
import math_utils

math_utils.circle_area(5)
math_utils.PI

§15.2import 的各種寫法

python 
# 1. 直接匯入整個模組
import math
math.sqrt(16)
math.pi

# 2. 取別名
import numpy as np
np.array([1, 2, 3])

# 3. 從模組匯入特定名稱（推薦）
from math import sqrt, pi
sqrt(16)

# 4. from 匯入時取別名
from math import sqrt as s

# 5. 匯入全部（⚠️ 不推薦，會汙染命名空間）
from math import *     # 💀 會把 math 的所有公開名稱灌入目前命名空間

# 6. 匯入套件中的模組
from mypackage.submodule import function
from mypackage.submodule.deep import Something
💡 import 風格建議
  • import module + module.func() → 來源最清楚，不易衝突
  • from module import func → 簡潔，適合常用名稱
  • from module import * → 一律避免，除非在互動式 REPL 探索
  • 匯入順序（PEP 8）：(1) 標準庫 (2) 第三方 (3) 本專案，每組間空一行

§15.3__name__ 與 if __name__ == "__main__":

每個 Python 檔執行時都有內建變數 __name__：直接執行時等於 "__main__"，被其他檔案匯入時等於模組名。

python · greeter.py 
def greet(name):
    return f"Hello, {name}"

def main():
    print(greet("World"))

# 只有「直接執行這個檔案」時才會執行
if __name__ == "__main__":
    main()
bash 
# 直接執行
python greeter.py
# 輸出：Hello, World

# 從其他檔案匯入
import greeter
greeter.greet("Alice")    # 可以用 greet()
                          # 但不會印出 "Hello, World"（main() 不會被呼叫）

這個模式讓檔案既能當「可執行腳本」，也能當「可匯入模組」，是 Python 的標準慣例。

可執行套件：__main__.py

bash 
# 套件結構
mypackage/
├── __init__.py
├── __main__.py          # 整個套件作為腳本執行的入口
└── core.py

# 執行
python -m mypackage       # 會執行 mypackage/__main__.py

§15.4套件結構

Package（套件）= 一個包含 __init__.py 的目錄，用來組織多個相關模組。

bash 
myapp/
├── __init__.py                 # 套件標記
├── models.py
├── views.py
├── utils/
│   ├── __init__.py             # 子套件標記
│   ├── string_tools.py
│   └── date_tools.py
└── services/
    ├── __init__.py
    ├── auth.py
    └── billing.py
python 
# 絕對匯入
from myapp.models import User
from myapp.utils.string_tools import slugify
from myapp.services.auth import login

# 相對匯入（僅限套件內部，不能獨立執行檔案時用）
# 在 myapp/services/auth.py 中：
from ..models import User            # 上一層（myapp）
from ..utils.string_tools import slugify
from .billing import charge           # 同目錄
from . import auth                    # 同層的 auth 模組

§15.5__init__.py 的角色

__init__.py 有三種常見用法：

用法 1：空檔案（只當標記）

python · __init__.py 
# 什麼都沒有，純粹把這個目錄標記為套件

用法 2：重新匯出公開 API（推薦）

python · myapp/__init__.py 
"""MyApp - 一個範例套件"""

from .models import User, Order
from .services.auth import login, logout

__version__ = "1.0.0"
__all__ = ["User", "Order", "login", "logout"]

使用端可以寫：from myapp import User，不用關心它實際在 myapp/models.py。這讓套件的「內部結構」與「公開介面」分離，重構時內部可以自由搬動。

用法 3：套件級初始化邏輯

python · __init__.py 
import logging

logging.getLogger(__name__).addHandler(logging.NullHandler())

# 檢查相依
try:
    import numpy
except ImportError:
    raise ImportError("需要 numpy，請 pip install numpy")

§15.6import 搜尋機制

Python 執行 import X 時會依以下順序搜尋：

  1. sys.modules 已匯入的模組快取（避免重複載入）
  2. 內建模組（如 sys、builtins）
  3. sys.path 中的目錄，依序查找
python 
import sys

# 查看搜尋路徑
print(sys.path)
# 依序包含：
# - 當前目錄 ('')
# - PYTHONPATH 環境變數路徑
# - 安裝時的 site-packages
# - Python 標準庫

# 查看已載入的模組
print(sys.modules.keys())

# 動態新增搜尋路徑（不建議，應透過 PYTHONPATH 或 uv）
sys.path.insert(0, "/my/custom/path")

# 重新載入模組（開發時除錯用）
import importlib
importlib.reload(my_module)

§15.7__all__ —— 定義公開 API

__all__ 是一個 list，明確宣告「這個模組的公開 API」。當別人用 from mymodule import * 時，只會匯入 __all__ 列出的名稱。

python · mymodule.py 
# 明確宣告公開 API
__all__ = ["User", "login"]

class User: ...        # 公開
def login(): ...       # 公開
def _hash_password(): ...   # 私有（慣例 _ 開頭）
class InternalCache: ...     # 公開，但不在 __all__，import * 不會帶出
📘 __all__ 的真實用途

即使你不推薦 import *，__all__ 仍然重要：

  • 作為 API 文件 —— 其他開發者知道哪些是公開介面
  • 靜態檢查器（pyright、mypy）用它判斷哪些符號是公開的
  • 套件文件生成工具（Sphinx）用它產生 API 清單

§15.8循環匯入（Circular Import）

python · ❌ 循環 
# a.py
from b import B
class A:
    def work(self):
        return B()

# b.py
from a import A     # 💀 ImportError! a.py 還沒執行完
class B:
    def work(self):
        return A()

四種解法

python 
# 解法 1：改為模組級 import，延後存取
# a.py
import b
class A:
    def work(self):
        return b.B()      # 用到時才透過模組存取

# 解法 2：在函式內 import（lazy import）
class A:
    def work(self):
        from b import B     # 執行時才 import
        return B()

# 解法 3：只在型別註解用 → TYPE_CHECKING
from typing import TYPE_CHECKING
if TYPE_CHECKING:
    from b import B

class A:
    def work(self) -> "B":    # 僅型別檢查時匯入，不影響執行
        ...

# 解法 4：重構 —— 把共同依賴抽到第三個檔案
💡 根本之道

循環匯入通常是設計不良的警訊：兩個模組耦合太緊。思考是否應該把共同的部分抽出來成為第三個模組（如 interfaces.py、types.py）。

§15.9命名空間套件（3.3+）

如果一個資料夾沒有 __init__.py 但仍能被當作套件匯入，它就是「命名空間套件」（PEP 420）。用於讓不同專案的模組共享同一個頂級命名空間。

bash 
# 專案 A：
company_tools/analytics/module_a.py     # 沒有 company_tools/__init__.py

# 專案 B：
company_tools/reporting/module_b.py     # 也沒有 __init__.py

# 兩個分開發布的套件都能用 company_tools.* 匯入

一般應用專案仍應該加 __init__.py（一般套件），命名空間套件主要給企業多團隊協作、大型可擴充生態系使用。

§15.10現代專案結構模板（src-layout）

bash · 推薦結構 
my-project/
├── pyproject.toml          # 專案配置（元資料 + 相依）
├── uv.lock                 # 鎖定檔
├── README.md
├── .gitignore
├── .python-version         # Python 版本（uv 自動維護）
│
├── src/
│   └── myapp/              # ← 你的套件
│       ├── __init__.py
│       ├── __main__.py     # python -m myapp 入口
│       ├── models.py
│       ├── services/
│       │   ├── __init__.py
│       │   └── auth.py
│       └── utils.py
│
├── tests/
│   ├── __init__.py
│   ├── test_models.py
│   └── test_services.py
│
└── docs/
    └── ...
📘 為什麼要 src-layout？
  • 明確區隔「程式碼」與「設定檔、測試、文件」
  • 避免匯入錯誤：開發時不會意外從專案根目錄 import 到未安裝版本
  • 測試必須先 pip install -e .（editable install），與正式環境行為一致
  • 大多數 modern Python 工具（uv、poetry、hatch）的預設結構

pyproject.toml 基本結構

toml · pyproject.toml 
[project]
name = "myapp"
version = "0.1.0"
description = "A sample app"
requires-python = ">=3.14"
dependencies = [
    "requests>=2.32",
    "pydantic>=2.5",
]

[project.scripts]
myapp = "myapp.__main__:main"     # 安裝後可直接呼叫 myapp 指令

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"
