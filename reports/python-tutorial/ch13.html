CHAPTER 13 · PART 4

例外處理與錯誤恢復

錯誤處理是「隱形的程式碼品質」—— 寫得好，程式堅固可靠、日誌清楚；寫得差，bug 沉默地吞噬資料。本章從基本語法走到 Python 3.11+ 新增的 ExceptionGroup 與 except*，以及 3.11 的 add_note，完整掌握現代 Python 的例外機制。

Python 3.14 try / except / ExceptionGroup 預估閱讀 22 分鐘

§0章節導論

Python 採 EAFP（Easier to Ask Forgiveness than Permission）風格：先做，出錯再抓，而不是事前一堆 if 檢查。本章涵蓋：

  • 完整的 try/except/else/finally 流程
  • Python 的例外階層與正確捕捉層級
  • 例外鏈（raise ... from ...）與 context
  • Python 3.11 引入的 ExceptionGroup 與 except* —— 處理並行任務中的多重錯誤
  • 自訂例外類別設計

§13.1try / except 基礎

python 
try:
    x = int(input("輸入數字："))
    result = 10 / x
except ValueError:
    print("不是有效數字")
except ZeroDivisionError:
    print("不可除以零")

# 一次抓多種
try:
    risky_op()
except (ValueError, TypeError) as e:
    print(f"錯誤：{e}")
    print(type(e).__name__)

# 捕捉所有例外（慎用！）
try:
    risky_op()
except Exception as e:           # ✅ 推薦
    logger.exception("未預期錯誤")  # 含完整 traceback
except:                          # ❌ 絕對避免！會連 KeyboardInterrupt 都抓
    pass
⚠️ 絕對避免 bare except

except:（後面什麼都不寫）會連 KeyboardInterrupt、SystemExit 這類繼承自 BaseException 的「系統級訊號」都抓，讓 Ctrl+C 也關不掉程式。一律寫 except Exception:，保留系統退出的能力。

§13.2例外階層

Python 所有例外繼承自 BaseException。多數情境你只需要繼承 Exception。

python 
BaseException
 ├── SystemExit              # sys.exit()
 ├── KeyboardInterrupt       # Ctrl+C
 ├── GeneratorExit
 └── Exception               # ← 你的自訂例外都從這裡繼承
      ├── ArithmeticError
      │    ├── ZeroDivisionError
      │    ├── OverflowError
      │    └── FloatingPointError
      ├── LookupError
      │    ├── IndexError      # lst[999]
      │    └── KeyError        # d["missing"]
      ├── AttributeError       # obj.missing
      ├── NameError            # 未定義變數
      ├── TypeError            # 型別錯誤
      ├── ValueError           # 型別對但值錯
      ├── RuntimeError
      │    └── RecursionError
      ├── StopIteration
      ├── OSError              # 系統 I/O 錯誤
      │    ├── FileNotFoundError
      │    ├── PermissionError
      │    ├── ConnectionError
      │    │    ├── ConnectionRefusedError
      │    │    └── ConnectionResetError
      │    └── TimeoutError
      ├── ImportError
      │    └── ModuleNotFoundError
      └── ...

捕捉順序由具體到抽象

python 
try:
    operation()
except FileNotFoundError:     # ✅ 具體先
    print("檔案不存在")
except OSError:               # ✅ 一般再來
    print("其他 IO 錯誤")
except Exception:             # ✅ 最後兜底
    print("其他錯誤")

# ❌ 順序反了：Exception 在前會吃掉所有後面的
try: ...
except Exception: ...
except FileNotFoundError: ...   # 永遠不會執行

§13.3else 與 finally 子句

python 
try:
    f = open("data.txt", encoding="utf-8")
except FileNotFoundError:
    print("找不到檔案")
else:
    # ✅ 只有 try 沒拋例外才執行
    data = f.read()
    process(data)
finally:
    # ✅ 不論成功或失敗都執行（通常用來釋放資源）
    if 'f' in dir():
        f.close()
📘 else 的意義

為什麼需要 else？因為它能清楚區分「可能出錯的程式碼」與「成功後才要跑的程式碼」。把後續處理放在 else 裡，意外的例外不會被 except 誤抓。try 塊應盡量精簡，只放真正可能拋例外的那幾行。

finally 的回傳陷阱

python 
def f():
    try:
        return 1
    finally:
        return 2      # ❌ 會覆蓋 try 的 return！

f()    # 回傳 2，不是 1！finally 中盡量不要 return

§13.4raise 主動拋出例外

python 
# 簡單拋出
def sqrt(x):
    if x < 0:
        raise ValueError(f"不可對負數取平方根：{x}")
    return x ** 0.5

# 重新拋出（re-raise）—— 記錄後繼續往上丟
try:
    do_work()
except ValueError:
    logger.error("記錄錯誤")
    raise   # 不加引數，原樣往外拋

# 轉換例外類型
try:
    parse(data)
except ValueError as e:
    raise InvalidInputError(f"解析失敗：{e}")   # 隱式鏈（__context__）

# 明確例外鏈
except ValueError as e:
    raise InvalidInputError("解析失敗") from e   # 顯式鏈（__cause__）

# 隱藏底層例外（遮蔽鏈）
except ValueError:
    raise InvalidInputError("解析失敗") from None  # 不顯示底層

§13.5例外鏈：__cause__ 與 __context__

python 
try:
    1 / 0
except ZeroDivisionError as e:
    raise ValueError("bad input") from e

# traceback 顯示：
# ZeroDivisionError: division by zero
#
# The above exception was the direct cause of the following exception:
#
# ValueError: bad input
機制屬性觸發方式顯示訊息
顯式鏈__cause__raise X from Y"was the direct cause of"
隱式鏈__context__在 except 中再 raise（無 from）"During handling of..."
抑制鏈—raise X from None不顯示鏈

§13.6自訂例外類別

python 
# 最基本：繼承 Exception
class InvalidOrderError(Exception):
    """訂單驗證失敗"""

# 攜帶結構化資訊
class ValidationError(Exception):
    def __init__(self, field: str, reason: str):
        self.field = field
        self.reason = reason
        super().__init__(f"{field}: {reason}")

try:
    raise ValidationError("email", "格式錯誤")
except ValidationError as e:
    print(e.field)     # 'email'
    print(e.reason)    # '格式錯誤'

# 模組專屬例外階層（大型專案推薦）
class AppError(Exception):
    """所有應用層例外的根"""

class NotFoundError(AppError): ...
class AuthError(AppError): ...
class RateLimitError(AppError): ...

# 上層可以一次抓全部
try:
    work()
except AppError as e:
    log(e)

§13.7ExceptionGroup 與 except*（3.11+）

處理並行任務時，可能同時發生多個錯誤（例如 asyncio.TaskGroup 多個 task 都失敗）。ExceptionGroup 允許你把多個例外「打包」拋出，用 except* 處理特定子類。

python 
# 建立 ExceptionGroup
raise ExceptionGroup("多個錯誤", [
    ValueError("bad value"),
    TypeError("bad type"),
    KeyError("missing key"),
])

# 用 except* 分別處理各類型（注意星號）
try:
    run_tasks()
except* ValueError as eg:
    print(f"值錯誤: {eg.exceptions}")    # 注意：eg 仍是 ExceptionGroup
except* (TypeError, KeyError) as eg:
    print(f"其他: {eg.exceptions}")
except* Exception as eg:
    print(f"未分類: {eg.exceptions}")

典型應用：asyncio.TaskGroup

python 
import asyncio

async def fetch_all():
    async with asyncio.TaskGroup() as tg:
        tg.create_task(fetch("url1"))
        tg.create_task(fetch("url2"))
        tg.create_task(fetch("url3"))
    # 如果兩個 task 拋 ValueError、一個拋 ConnectionError
    # TaskGroup 會把它們打包成 ExceptionGroup 一起拋出

try:
    asyncio.run(fetch_all())
except* ValueError as eg:
    print(f"ValueError 數量：{len(eg.exceptions)}")
except* ConnectionError as eg:
    print(f"連線錯誤：{eg.exceptions}")

subgroup / split

python 
eg = ExceptionGroup("batch", [
    ValueError("a"), ValueError("b"), TypeError("c")
])

# subgroup：只留下符合的
value_errors = eg.subgroup(ValueError)
# ExceptionGroup("batch", [ValueError("a"), ValueError("b")])

# split：(符合的, 剩餘的)
matched, rest = eg.split(ValueError)

# 用 callable 當條件
eg.subgroup(lambda e: "a" in str(e))
⚠️ except* 不能捕捉裸 ExceptionGroup
try:
    raise ExceptionGroup("x", [ValueError()])
except* ExceptionGroup:           # ❌ TypeError
    ...
except* ValueError:               # ✅ 正確
    ...

§13.8add_note — 為例外加上備註（3.11+）

python 
try:
    process(record)
except Exception as e:
    e.add_note(f"處理記錄 ID={record.id} 時發生")
    e.add_note(f"source: {record.source}")
    raise

# traceback 最後會顯示所有 notes，不需自訂例外類別就能加上下文資訊

§13.9contextlib 相關工具

python 
from contextlib import suppress, contextmanager

# suppress — 忽略特定例外（替代「空 except pass」）
with suppress(FileNotFoundError):
    os.remove("cache.tmp")
# 如果檔案不存在，靜靜忽略；其他例外會照常傳出

# 與自訂 context manager 結合
@contextmanager
def atomic_write(path):
    tmp = path + ".tmp"
    f = open(tmp, "w", encoding="utf-8")
    try:
        yield f
        f.close()
        os.replace(tmp, path)     # 成功才覆蓋
    except:
        f.close()
        os.remove(tmp)            # 失敗就清乾淨
        raise

with atomic_write("config.json") as f:
    f.write(data)

§13.10最佳實踐

  • 只抓你知道如何處理的例外。不知道怎麼處理就讓它往上拋
  • 避免 bare except:，一律用 except Exception: 或更具體類型
  • try 塊保持精簡，只放真正可能拋例外的那幾行。其他放 else
  • 記錄完整堆疊：logger.exception(msg) 會自動包含 traceback
  • 用 raise ... from ... 保留因果鏈，讓除錯時能追溯根因
  • 為套件建立例外根類別（如 AppError），讓使用者能一次抓所有應用錯誤
  • 例外訊息要具體：ValueError("age 必須 >= 0，實際為 -5") 比 ValueError("bad") 有用
  • 不要把例外當流程控制（例外成本比 if 高），除非確實是例外情況
  • 用 add_note 補充上下文，不必每種情境都建自訂例外類別
  • 在並行程式中考慮用 ExceptionGroup 一次報告多個失敗
