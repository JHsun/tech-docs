CHAPTER 21 · PART 6

效能優化與JIT

Python 一向被嘲笑「慢」—— 但 Faster CPython 專案從 3.11 起連續為 CPython 注入活力，3.14 更帶來 Tail-call Interpreter（尾呼叫直譯器）。本章講優化心法、測量工具、Python 技巧，以及 3.14 與 3.15 的效能新引擎。

Python 3.14 Profiling · JIT · 優化 預估閱讀 22 分鐘

§0章節導論

本章的核心信念：

💡 Donald Knuth 的名言

「過早優化是萬惡之源」—— 先寫清楚、正確的程式；測量熱點後才優化；改完再測量是否真的變快。

  • 95% 的程式碼不需要「優化」，只需要「選對工具、選對資料結構」
  • 剩下 5% 的熱點，才值得花時間調校
  • 最大的效能倍率通常來自：演算法選擇 > 資料結構 > Python 技巧 > 編譯優化

§21.1測量先於優化

python 
# 1. timeit — 簡易計時（小段程式碼）
import timeit

# 比較兩種寫法
timeit.timeit("'-'.join([str(i) for i in range(100)])", number=10000)
timeit.timeit("'-'.join(str(i) for i in range(100))", number=10000)
timeit.timeit("'-'.join(map(str, range(100)))", number=10000)

# 命令列快速比較
# python -m timeit "'-'.join(str(i) for i in range(100))"

# 2. time.perf_counter — 精確計時（一般測量）
import time

t0 = time.perf_counter()
do_work()
elapsed = time.perf_counter() - t0
print(f"耗時 {elapsed:.3f}s")

# 3. context manager 封裝
from contextlib import contextmanager

@contextmanager
def timer(label=""):
    t0 = time.perf_counter()
    try:
        yield
    finally:
        print(f"{label}: {time.perf_counter()-t0:.3f}s")

with timer("讀檔"):
    load_data()
with timer("處理"):
    process()

§21.3Python 層級優化技巧

python 
# 1. 內建函式 / 內建型別快於手刻
# ❌
total = 0
for x in nums: total += x
# ✅
total = sum(nums)          # 快很多

# 2. 推導式 / 生成器運算式 快於 map + lambda
[x*2 for x in nums if x>0]    # ✅ 最快
list(map(lambda x: x*2, filter(lambda x: x>0, nums)))

# 3. 字串連接用 join，不是 += 
# ❌
s = ""
for word in words: s += word    # O(n²)！字串不可變，每次都複製
# ✅
s = "".join(words)             # O(n)

# 4. 避免重複屬性查找（放區域變數）
# ❌
for i in range(10000):
    result.append(obj.method(i))
# ✅（提升 ~10-20%）
append = result.append
method = obj.method
for i in range(10000):
    append(method(i))

# 5. 用 set / dict 做成員檢查（O(1) vs list 的 O(n)）
valid_ids = {1, 2, 3, 4, 5}      # set
if user_id in valid_ids: ...      # O(1)

# 6. 盡早退出（short-circuit）
if cheap_check() and expensive_check():
    ...                             # expensive 只在 cheap 通過後執行

# 7. 使用內建快取
from functools import cache, lru_cache

@cache
def fib(n):
    return n if n < 2 else fib(n-1) + fib(n-2)

fib(100)                           # 瞬間，沒快取會當機

python 
# 常見「選錯結構」陷阱

# ❌ 在大 list 裡檢查成員
if item in big_list:     # O(n)，1 萬筆資料超慢
    ...

# ✅ 轉成 set 一次
big_set = set(big_list)
if item in big_set:      # O(1)

# ❌ list 頭部插入
lst.insert(0, x)          # O(n)

# ✅ 雙端佇列
from collections import deque
dq.appendleft(x)          # O(1)

# ❌ 用 list 當 queue
q = []
q.append(x); q.pop(0)      # pop(0) O(n)

# ✅ deque
q = deque()
q.append(x); q.popleft()    # O(1)

# ❌ 手動維護排序 list
for x in stream:
    sorted_list.append(x)
    sorted_list.sort()        # O(n log n) 每次！

# ✅ heapq（優先佇列）
import heapq
heapq.heappush(heap, x)    # O(log n)
heapq.heappop(heap)        # 取最小 O(log n)

# ❌ 用 dict 找「最近 N 筆」
# ✅ 用 deque(maxlen=N)
recent = deque(maxlen=100)
recent.append(event)        # 超過 100 自動丟棄最舊的

§21.5記憶體優化

python 
# 1. 生成器代替 list（處理大資料）
sum([x**2 for x in range(10**8)])    # ❌ 建一大 list 才 sum
sum(x**2 for x in range(10**8))      # ✅ 生成器，省記憶體

# 2. __slots__ 大量實例時省 40-50%
class Point:
    __slots__ = ("x", "y")
    def __init__(self, x, y):
        self.x, self.y = x, y

# 3. array 代替 list（純數字資料）
from array import array
a = array("i", range(1_000_000))   # 比 list 省 4-8 倍記憶體

# 4. numpy 對數值運算是殺手級
import numpy as np
arr = np.arange(1_000_000)
arr ** 2                   # 比 Python list 快 50-100 倍

# 5. 檢查物件大小
import sys
sys.getsizeof(obj)

# 6. 大物件用完後主動釋放
del large_dict
import gc
gc.collect()                # 手動觸發垃圾回收

§21.6🆕 Python 3.14 Tail-call Interpreter

🆕 Faster CPython 專案的最新成果

Python 3.14 導入了新的尾呼叫直譯器（Tail-call Interpreter），利用 Clang 19+ 的 __attribute__((musttail)) 特性改寫 CPython 的 bytecode 分派機制。整體 pyperformance 提升約 3-5%，某些工作負載可達 30%。

技術原理（簡化版）

過去 CPython 的直譯器迴圈是個巨大的 switch-case / computed goto 函式。新版把每個 bytecode 指令變成獨立的小函式，透過尾呼叫（tail call）跳到下一個指令的處理函式。這讓現代編譯器能：

  • 更好地使用暫存器、避免 stack frame 開銷
  • 分別優化每個指令（過去 12k 行單一函式難以優化）
  • 配合 Linux perf 工具分別 profile 每個指令

如何啟用

bash 
# 目前仍需要從原始碼建置時明確開啟
./configure --enable-optimizations --with-tail-call-interp
make

# 好消息：uv 在 macOS 上預設就使用啟用 tail-call 的 3.14 建置
uv python install 3.14

# 需求：
# - Clang 19+ 或 MSVC 18+（Windows 3.15 起）
# - x86-64 或 AArch64
# - 建議搭配 PGO（profile-guided optimization）
📘 效能現實

Tail-call interpreter 在 3.14 剛推出時曾宣稱 10-15% 加速，後來 Ken Jin 澄清部分速度差是因為舊版 computed goto 被編譯器 bug 拖累。真正純粹的改善是 ~2-5% 幾何平均。但這仍是重要基礎 —— 因為它為 JIT 鋪路，且 Python 3.15 的 Windows 版預計藉此帶來 15-20% 加速。

§21.7🆕 實驗性 JIT 編譯器

除了 tail-call interpreter，CPython 也在推進一個真正的 JIT 編譯器。3.13/3.14 的早期 JIT 幾乎沒有加速，但 3.15 alpha 開始有實質改善：

平台vs tail-call / 一般直譯器
macOS AArch64（M 系列晶片）+11-12% 幾何平均
Linux x86-64+5-6% 幾何平均
個別微基準-20% 到 +100% 不等
bash 
# 啟用實驗 JIT（目前需重新編譯 Python）
./configure --enable-experimental-jit
make

# 環境變數控制
PYTHON_JIT=1 python myscript.py    # 啟用
PYTHON_JIT=0 python myscript.py    # 停用
⚠️ 目前狀態
  • 3.14 的 JIT 仍屬實驗性，生產環境勿用
  • 不支援 Free-threaded build（3.15/3.16 規劃中）
  • 單一基準可能出現 slowdown
  • 3.16+ 有望達到成熟階段

§21.8當 CPython 還不夠快 —— 外部工具

替代實作

工具用途典型加速

用 Rust / C 寫熱點

  • PyO3：從 Rust 呼叫 Python / 讓 Python 用 Rust —— 目前最火的 Python 擴充技術（uv、ruff、polars 都是）
  • pybind11：C++ 對應
  • cffi / ctypes：呼叫 C 函式庫
💡 優化的優先順序
  1. 換演算法（O(n²) → O(n log n)）
  2. 換資料結構（list → set/dict）
  3. 用內建/標準庫函式（sum、map、itertools）
  4. 向量化（numpy、pandas、polars）
  5. 快取（@cache）
  6. 並行（threading / asyncio / multiprocessing）
  7. 升級 Python（3.14 幾乎無痛加速）
  8. 編譯熱點（Cython、numba、Rust extension）
  9. 換 PyPy
