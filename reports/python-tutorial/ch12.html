CHAPTER 12 · PART 3

型別註解與靜態檢查

Python 從 3.0 開始就能為函式與變數加「型別註解」，但一直到 3.14 才迎來里程碑級變革：PEP 649 延遲求值 —— 註解不再立刻計算，徹底解決前向參照、循環引用、效能三大痛點。搭配 mypy/pyright，動態語言也能獲得 TypeScript 級的靜態保護。

Python 3.14 · PEP 649 新特性 Typing 全覽 預估閱讀 30 分鐘

§0章節導論

本章涵蓋：

  • 基本與進階的型別註解語法（含 3.10+ 的 |、list[int]）
  • 泛型（Generic）與 3.12 引入的新 type parameter 語法
  • PEP 649：Python 3.14 把所有註解改為延遲求值 —— 重大變革
  • 搭配 mypy、pyright、ty 等靜態檢查器
  • 常見陷阱與 VS Code 整合
📘 型別註解不會影響執行

Python 本身在執行時不會檢查型別註解（除非你用 Pydantic 等工具）。它們是給靜態檢查器、編輯器、文件使用的「提示」。這讓 Python 既保有動態彈性，又能在開發時享受型別安全。

§12.1基本語法

python 
# 變數註解
name: str = "Alice"
age: int = 30
price: float = 9.99
active: bool = True
data: bytes = b"hello"

# 函式參數與回傳值
def greet(name: str, times: int = 1) -> str:
    return f"Hi {name}" * times

# None 回傳
def log(msg: str) -> None:
    print(msg)

# 類別屬性
class User:
    name: str
    age: int
    email: str = ""   # 有預設值

    def __init__(self, name: str, age: int):
        self.name = name
        self.age = age

§12.2容器型別（泛型）

python 
# 3.9+ 直接用內建型別（不用再 from typing import List）
names: list[str] = ["Alice", "Bob"]
scores: dict[str, int] = {"Alice": 90}
tags: set[str] = {"a", "b"}
coord: tuple[int, int] = (3, 4)         # 固定長度
numbers: tuple[int, ...] = (1, 2, 3)      # 可變長度（... = Ellipsis）
frozen: frozenset[int] = frozenset([1, 2])

# 巢狀
matrix: list[list[float]] = [[1.0, 2.0], [3.0, 4.0]]
mapping: dict[str, list[int]] = {"x": [1, 2]}

# 3.8 及以前需從 typing 匯入（仍可用，相容）
from typing import List, Dict, Tuple, Set
names: List[str]

抽象容器（推薦用於參數）

python 
from collections.abc import Iterable, Sequence, Mapping, Callable

# ✅ 接受任何可迭代物件（list、tuple、generator 都行）
def total(nums: Iterable[int]) -> int:
    return sum(nums)

# ❌ 太限制，只收 list
def total(nums: list[int]) -> int:
    return sum(nums)

# 原則：參數用抽象型別（寬鬆接受），回傳用具體型別（明確告知）

§12.3Optional 與 Union

python 
# 3.10+ 用 | 運算子（最簡潔）
def find_user(id: int) -> User | None:
    ...

# 等同於（3.8+ 語法）
from typing import Optional, Union
def find_user(id: int) -> Optional[User]:    # Optional[X] = Union[X, None]
    ...

# 多個型別
def process(value: int | str | float) -> str:
    return str(value)

# 在函式內用 isinstance 或 match 做型別收窄（type narrowing）
def describe(x: int | str) -> str:
    if isinstance(x, int):
        return f"number: {x + 1}"     # 此處 x 被收窄為 int
    else:
        return f"string: {x.upper()}"   # 此處 x 被收窄為 str

Literal、Final、TypeAlias

python 
from typing import Literal, Final

# Literal — 限定字面值
def set_mode(mode: Literal["light", "dark", "auto"]) -> None:
    ...

set_mode("dark")      # ✅
set_mode("yellow")   # ❌ mypy 會報錯

# Final — 常數（不可重新賦值）
MAX_SIZE: Final = 100
MAX_SIZE = 200     # ❌ mypy 會報錯

# 型別別名（3.12+ 新語法）
type UserId = int
type JSON = dict[str, "JSON"] | list["JSON"] | str | int | bool | None

# 3.11 以前
from typing import TypeAlias
UserId: TypeAlias = int

§12.4Callable — 函式型別

python 
from collections.abc import Callable

# Callable[[參數型別...], 回傳型別]
def apply(fn: Callable[[int, int], int], x: int, y: int) -> int:
    return fn(x, y)

apply(lambda a, b: a + b, 3, 4)   # ✅

# 無參數 / 回傳 None
callback: Callable[[], None]

# 可變參數用 ...
handler: Callable[..., str]       # 接受任何參數，回傳 str

§12.5泛型 Generic

python · Python 3.12+ 新語法 
# 3.12+ 的新泛型語法（最推薦）
def first[T](lst: list[T]) -> T | None:
    return lst[0] if lst else None

first([1, 2, 3])      # 回傳 int
first(["a", "b"])      # 回傳 str

# 泛型類別
class Stack[T]:
    def __init__(self) -> None:
        self._items: list[T] = []
    def push(self, item: T) -> None:
        self._items.append(item)
    def pop(self) -> T:
        return self._items.pop()

int_stack: Stack[int] = Stack()
int_stack.push(42)

# 多個型別參數
class Pair[K, V]:
    def __init__(self, k: K, v: V):
        self.k, self.v = k, v

# 3.11 及以前的舊語法
from typing import TypeVar, Generic
T = TypeVar("T")
class Stack(Generic[T]):
    ...

受限的泛型

python 
# bound — T 必須是某型別或其子類
class Animal: ...
class Dog(Animal): ...

def feed[T: Animal](a: T) -> T:
    print(f"feeding {a}")
    return a

# constrained — T 必須是列出型別之一
def concat[T: (str, bytes)](a: T, b: T) -> T:
    return a + b

§12.6🆕 PEP 649：延遲註解求值（3.14 重大變革）

🆕 Python 3.14 最重要的型別相關變革

從 3.14 開始，所有註解都改為「延遲求值」（lazy evaluation）。註解不再於定義當下計算，而是在「真正被存取時」才算。這一個改變解決了型別註解多年來最痛苦的三個問題。

問題 1：前向參照（Forward Reference）

python · Python 3.13 及以前 
class Node:
    # ❌ 3.13 及以前：NameError，因為 Node 還沒定義完
    def next(self) -> Node:
        ...

# 3.13 以前的解法 1：加引號
class Node:
    def next(self) -> "Node":       # 字串形式
        ...

# 3.13 以前的解法 2：from __future__ 開關
from __future__ import annotations   # 整個檔案所有註解都變字串
python · Python 3.14 ✨ 
# ✅ 3.14 起直接這麼寫就好！註解延遲求值，不會 NameError
class Node:
    value: int
    def next(self) -> Node:       # 無需引號
        ...

# 循環引用也沒問題
class Parent:
    children: list[Child]         # Child 在下方定義，OK！

class Child:
    parent: Parent

問題 2：效能

在 3.13 及以前，模組載入時所有註解都立刻計算，讓大型專案的 import 變慢。3.14 改為「需要才算」，大幅改善 import 速度。

問題 3：內省 API 的靈活度

3.14 提供 annotationlib 模組，讓你能用不同「格式」取得註解（見 §12.7）。

⚠️ 需要注意的相容性
  • 大多數程式碼無需修改，但使用 __annotations__ 直接存取的程式碼可能要調整
  • from __future__ import annotations 在 3.14 仍可用，但行為略有不同（註解變字串），會逐步廢棄
  • 型別檢查器（mypy / pyright / ty）需要升級到支援 PEP 649 的版本
  • 舊版 Python（3.13-）執行的程式碼若用了 3.14 的前向參照寫法，會 NameError —— 所以寫函式庫時仍要小心

§12.7annotationlib — 註解內省新 API（3.14）

python 
from annotationlib import get_annotations, Format

def func(x: int, y: Undefined) -> str:
    pass

# Format.VALUE — 傳統行為：直接求值（遇未定義會 NameError）
get_annotations(func, format=Format.VALUE)
# 若 Undefined 未定義 → 🚫 NameError

# Format.FORWARDREF — 未定義名稱轉 ForwardRef（不爆炸）
get_annotations(func, format=Format.FORWARDREF)
# {'x': int, 'y': ForwardRef('Undefined'), 'return': str}

# Format.STRING — 全部以字串形式回傳（像 future annotations）
get_annotations(func, format=Format.STRING)
# {'x': 'int', 'y': 'Undefined', 'return': 'str'}

這組 API 讓像 Pydantic、dataclasses、SQLAlchemy 這類工具能以最適合的方式處理註解，不再需要手動 eval 字串。

§12.9陷阱與最佳實踐

Any vs object vs Unknown

python 
from typing import Any

# Any — 「關閉型別檢查」（盡量避免）
def process(data: Any) -> Any:
    return data.anything.you.want   # 不會被檢查

# object — 「所有物件的共同父類」（什麼都接受，但用時要 narrow）
def safe_print(x: object) -> None:
    print(x)        # OK，但不能用 x.any_method
    x.upper()        # ❌ 需要先 isinstance(x, str)

最佳實踐

  • 新專案一開始就加型別，舊專案可逐步添加
  • 公開 API 一定要加註解（函式簽名、類別屬性）
  • 參數用抽象型別（Iterable、Mapping），回傳用具體型別
  • 避免 Any，它會讓型別檢查失去意義
  • 設定 mypy/pyright 加入 CI pipeline，強制保持型別正確
  • 3.14 之後大膽使用前向參照，不需要再加引號
  • 使用 3.10+ 的 X | Y 取代 Union[X, Y]
  • 使用 3.12+ 的 def f[T](...) 泛型語法
