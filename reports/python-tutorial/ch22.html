CHAPTER 22 · PART 6

現代打包工具

Python 的打包生態系在 2024-2026 大洗牌 —— Astral 推出的 Rust 工具 uv 取代了 pip / poetry / pyenv / virtualenv；ruff 一次取代 flake8 / black / isort；PyPI 從 PGP 轉向 Sigstore。這章給你最新、最精簡、最快的 Python 工作流。

Python 3.14 · 2026 Q2 uv · ruff · Sigstore 預估閱讀 30 分鐘

§0章節導論

🆕 2026 年的現代 Python 工作流

一年前還是 pip + virtualenv + poetry + pyenv + flake8 + black + isort + twine 的複雜組合，現在幾乎都可以用 uv + ruff 兩個工具取代。兩者都是 Astral 公司用 Rust 寫的，速度通常是舊工具的 10-100 倍，且整合緊密。

舊工具新工具（2026）速度
pip + virtualenv + pyenv + poetryuv~10-100x
flake8 + pylintruff check~100x
black + isortruff format~30x
twineuv publish—

§22.1uv 完全指南

安裝

bash 
# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh

# Windows PowerShell
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

# 或用其他套件管理器
brew install uv                 # macOS
scoop install uv                # Windows
uv self update                  # 自我更新

Python 版本管理（取代 pyenv）

bash 
# 列出所有可裝的 Python
uv python list

# 安裝特定版本
uv python install 3.14
uv python install 3.14t               # free-threaded
uv python install 3.13 3.12 3.11        # 多個

# 設當前 shell 的 Python 版本
uv python pin 3.14                  # 寫入 .python-version 檔

# 查看目前版本
uv python find

快速 pip 替代（單檔腳本）

bash 
# uv pip — 與 pip 相同的 CLI（但快 10-100 倍）
uv pip install requests
uv pip install -r requirements.txt
uv pip install -e .
uv pip freeze
uv pip list

# 建立 venv
uv venv                         # 建立 .venv/
uv venv --python 3.14           # 指定版本
source .venv/bin/activate       # 如常啟用（Unix）
.venv\Scripts\activate          # Windows

§22.2專案管理（取代 poetry）

建立新專案

bash 
# 建立新專案
uv init myapp                   # 簡單腳本專案
uv init --lib myapp             # 函式庫（src-layout）
uv init --python 3.14 myapp

# 產生：
# myapp/
# ├── .python-version   (3.14)
# ├── README.md
# ├── pyproject.toml
# ├── src/myapp/__init__.py
# └── ...

相依管理

bash 
# 新增相依（自動更新 pyproject.toml + uv.lock）
uv add requests
uv add "fastapi>=0.110"
uv add numpy pandas polars       # 多個一次

# 新增開發相依（只在 dev group）
uv add --dev pytest ruff mypy
uv add --group docs mkdocs

# 從 Git 或本地
uv add "git+https://github.com/user/repo"
uv add "./local-package"

# 移除
uv remove requests

# 同步環境（確保 venv 與 uv.lock 一致）
uv sync                          # 全部
uv sync --dev                    # 含 dev
uv sync --no-dev                 # 只裝必要（CI 部署）
uv sync --frozen                 # 用 lock，不重新解析

# 鎖定檔（uv.lock 要提交到 git）
uv lock                          # 重新產生鎖定檔
uv lock --upgrade                # 升級所有相依到最新
uv lock --upgrade-package requests

執行程式

bash 
# 在專案 venv 中執行（自動 sync）
uv run myscript.py
uv run python -c "print('hi')"
uv run pytest
uv run python -m myapp

# 不需要 activate venv！

# 指令別名（在 pyproject.toml [project.scripts] 定義後）
uv run myapp                    # 自動找 entry point

# 執行一次性腳本（可指定相依）
uv run --with pandas --with matplotlib script.py

行內腳本相依（PEP 723）

python · script.py 
# /// script
# requires-python = ">=3.14"
# dependencies = [
#     "requests",
#     "rich",
# ]
# ///
import requests
from rich import print

print(requests.get("https://httpbin.org/get").json())
bash 
# 執行時 uv 會自動建立臨時環境、裝相依、執行
uv run script.py

# 一行 pip 腳本搞定，無需 requirements.txt 或虛擬環境管理

§22.5ruff —— Linter + Formatter 二合一

bash 
uv add --dev ruff

# 檢查程式碼（linter）
uv run ruff check .
uv run ruff check --fix .        # 自動修

# 格式化（formatter，取代 black）
uv run ruff format .
uv run ruff format --check .      # 僅檢查不改

# 整合在 git pre-commit
uv run ruff check --fix
uv run ruff format

ruff 配置

toml · pyproject.toml 
[tool.ruff]
line-length = 100
target-version = "py314"

[tool.ruff.lint]
select = [
    "E",      # pycodestyle 錯誤
    "W",      # pycodestyle 警告
    "F",      # pyflakes
    "I",      # isort（import 排序）
    "N",      # pep8-naming
    "UP",     # pyupgrade（用新語法）
    "B",      # flake8-bugbear
    "SIM",    # flake8-simplify
    "C4",     # 推導式規則
    "PTH",    # 強制用 pathlib
    "RUF",    # ruff 專屬
]
ignore = ["E501"]  # 行太長交給 formatter

[tool.ruff.format]
quote-style = "double"
docstring-code-format = true
💡 ruff 的速度

ruff 是用 Rust 寫的。100 萬行的 Django codebase 用 flake8 要跑幾分鐘，ruff 只要 0.5 秒。官方已實作 800+ 規則，幾乎涵蓋所有主流 linter/formatter 功能。

§22.6建置與發布

建置 wheel

bash 
# 建置 sdist + wheel
uv build

# 產出：
# dist/
# ├── myapp-0.1.0.tar.gz      (sdist)
# └── myapp-0.1.0-py3-none-any.whl  (wheel)

# 僅建 wheel

本機測試安裝

bash 
# 安裝自己建的 wheel 測試
uv pip install dist/myapp-0.1.0-py3-none-any.whl

# Editable install（開發中）
uv pip install -e .

# 建後立刻測試
uv run --isolated --with dist/myapp-0.1.0-py3-none-any.whl python -c "import myapp"

發布到 PyPI

bash 
# 發布（取代 twine upload）
uv publish

# 用 API token（推薦用 Trusted Publishing 取代）
UV_PUBLISH_TOKEN=pypi-XXXXXX uv publish

# 或在 ~/.config/uv/uv.toml 設定
# [publish]
# token = "..."

# 發到 TestPyPI 練習

§22.7Trusted Publishing

Trusted Publishing 讓你不必管理 PyPI API token —— 直接用 GitHub Actions / GitLab CI 的身份發布，安全且自動。

yaml · .github/workflows/publish.yml 
name: Publish

on:
  release:
    types: [published]

jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      id-token: write          # OIDC
      attestations: write      # Sigstore 簽章
    steps:
      - uses: actions/checkout@v4

      - uses: astral-sh/setup-uv@v3

      - run: uv build

      - name: Publish to PyPI
        uses: pypa/gh-action-pypi-publish@release/v1
        # 不需 password！也不需要 secret
        # 會自動使用 GitHub OIDC + Sigstore 產生 attestation

在 PyPI 專案設定頁面「Publishing」加入你的 GitHub repo 與 workflow 檔名，就完成 Trusted Publishing 設定。下次 push release 就會自動發布並帶有 Sigstore attestation。

§22.8🆕 Sigstore 簽章（取代 PGP）

🆕 Python 套件簽章的新時代
  • 2023 年 5 月：PyPI 停用 PGP 簽章上傳（因為大多數簽章都無法驗證）
  • 2024 年 11 月：PyPI 正式支援 PEP 740 Digital Attestations（基於 Sigstore）
  • Python 3.14 起：CPython 官方 release 棄用 PGP，改用 Sigstore（PEP 761）

Sigstore 的核心概念

  • 無金鑰簽章：不需維護長效私鑰，改用短效憑證 + OpenID Connect 身份
  • 身份綁定：簽章與「GitHub repo + workflow」等機器身份掛鉤，不是個人金鑰
  • 透明日誌：所有簽章都記錄在公開的 Rekor 日誌中，可追溯
  • 自動化：搭配 Trusted Publishing，發布者不需做任何額外操作

為什麼 PGP 被棄用？

  • PyPI 調查發現：歷史 PGP 簽章中只有 36% 能真正驗證，而且全部簽章只覆蓋 0.3% 的檔案
  • 開發者必須自行維護 7+ 年的長期私鑰（release 生命週期）
  • 金鑰分發機制（keyserver）不可靠，驗證困難
  • 個人金鑰遺失/洩漏很常見

驗證套件的 Sigstore 簽章

bash 
# 安裝驗證工具
uv tool install pypi-attestations

# 驗證某個套件的 attestation
pypi-attestations verify pypi \
    --repository https://github.com/sigstore/sigstore-python \
    pypi:sigstore-3.6.1-py3-none-any.whl

# 這會下載 whl 與 attestation，驗證：
# 1. 簽章有效
# 2. 簽署者確實是指定的 GitHub repo
# 3. 檔案與簽章對應

# 也可透過 PyPI 的 Integrity API 取得 attestation
curl https://pypi.org/integrity/sigstore/3.6.1/sigstore-3.6.1-py3-none-any.whl/provenance
💡 發布者的好處
  • 如果你用 Trusted Publishing + pypa/gh-action-pypi-publish，Sigstore attestation 會自動產生，你什麼都不用做
  • 不需要 GPG key、不需要 passphrase 記錄、不怕金鑰遺失
  • 驗證者能回溯到確切的 source code commit
  • 超過 20,000 個 PyPI 套件已採用

§22.9完整發布流程範例

bash 
# 1. 初始化新專案
uv init --lib mypackage
cd mypackage

# 2. 新增相依
uv add requests pydantic
uv add --dev pytest ruff mypy

# 3. 開發
# ... 寫程式碼 ...
uv run ruff check --fix .
uv run ruff format .
uv run pytest --cov

# 4. 更新版本
# 在 pyproject.toml 改 version

# 5. 建置
uv build

# 6. 提交與 tag
git add .
git commit -m "release 0.2.0"
git tag v0.2.0
git push --tags

# 7. 在 GitHub 建立 Release
# 8. GitHub Actions 自動：
#    - 執行 uv build
#    - 透過 Trusted Publishing 發布到 PyPI
#    - 自動產生 Sigstore attestation

# 任何人都可驗證：
pypi-attestations verify pypi \
    --repository https://github.com/user/mypackage \
    pypi:mypackage-0.2.0-py3-none-any.whl
