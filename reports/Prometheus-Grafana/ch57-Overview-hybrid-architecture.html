Chapter 57 · 混合架構實戰

混合架構實戰：VM + K8s + Cloud + Edge 統一觀測性

真實世界很少是「純 K8s」或「純雲」。多數企業有混合環境：legacy VM、bare metal、 K8s cluster、多雲、邊緣裝置（IoT / 零售門市）。本章涵蓋如何用一套觀測性架構統一所有環境， 收尾 Part 7 實戰案例。

§ 1. 混合環境的挑戰

1.1 真實世界的環境組合

典型企業可能同時有：

  • Legacy VM：on-prem VMware 上的 Java/MySQL 應用（10+ 年歷史）
  • Bare metal：高效能運算 / GPU 機房
  • K8s on-prem：自架 EKS Anywhere / Rancher / OpenShift
  • K8s in Cloud：EKS / GKE / AKS（可能多家）
  • Serverless：Lambda / Cloud Functions / Azure Functions
  • SaaS dependencies：Stripe / SendGrid / Twilio
  • Edge：零售門市 POS、IoT 感測器、CDN edge

1.2 痛點

痛點說明
工具碎片化 每環境用不同 tool，看 5+ 個 dashboard
網路隔離 on-prem 連雲要 VPN / Direct Connect
身份不統一 每環境不同 IAM
Tag / Label 不統一 同 service 在不同地方 label 不同，無法 join
Trace 斷裂 環境邊界 trace 中斷
Cost 黑洞 Telemetry traffic 跨環境費用高

1.3 統一原則

✅ 5 個統一原則
  1. 統一 telemetry pipeline：Alloy / OTel Collector 在每環境部署
  2. 統一 backend：一套 LGTM stack（自架或 Grafana Cloud）
  3. 統一 label schema：所有訊號加 environment / service / version label
  4. 統一 visualization：一個 Grafana 實例查所有 datasource
  5. 統一 SLO：從 user-facing 視角定義，不分環境

§ 2. 統一觀測性架構

2.1 中央化架構

所有 telemetry 集中到一個 LGTM stack：

graph TB VM[Legacy VM
on-prem] BM[Bare Metal
HPC] K8SP[K8s on-prem
OpenShift] EKS[EKS
AWS] GKE[GKE
GCP] AKS[AKS
Azure] LAMBDA[Lambda
Serverless] EDGE[Edge POS
IoT] EDGE_PROXY[Edge Gateway

2.2 Hub-and-Spoke 模型

每個環境是「spoke」，跑本地 collector；中央 Telemetry Hub 統一接收：

  • Spoke（每環境）：Alloy DaemonSet / VM agent 收集本地 telemetry
  • Hub（中央）：Alloy Gateway 做 sampling / enrichment / routing
  • Backend：自架 LGTM 或 Grafana Cloud

2.3 Hub 部署位置

位置適合缺點
主雲 K8s 多數工作負載在主雲 單雲依賴
多雲冗餘 HA、避免單雲 outage cost 高、複雜
Grafana Cloud 不想運維 SaaS 成本
On-prem 機房 data sovereignty 嚴 scale 難

§ 3. VM / Bare Metal 整合

3.1 Linux VM 部署 Alloy

# 安裝
sudo apt-get install alloy
# 或下載 binary
curl -O https://github.com/grafana/alloy/releases/.../alloy-linux-amd64.zip

# /etc/alloy/config.alloy
prometheus.exporter.unix "host" {
  // 收集 OS-level metric（取代 node_exporter）
}

prometheus.scrape "host_metrics" {
  targets    = prometheus.exporter.unix.host.targets
  forward_to = [prometheus.remote_write.hub.receiver]
}

prometheus.scrape "app" {
  // scrape 應用 metric
  targets = [
    { __address__ = "localhost:8080" },
  ]
  forward_to = [prometheus.remote_write.hub.receiver]
}

prometheus.remote_write "hub" {
  endpoint {
    url = "https://hub.example.com/api/v1/push"
    basic_auth {
      username = env("HUB_USER")
      password = env("HUB_PASSWORD")
    }
    headers = {
      "X-Cluster" = "vm-prod-east",
      "X-Environment" = "production",
    }
  }
}

// Log 收集
local.file_match "logs" {
  path_targets = [
    { __path__ = "/var/log/myapp/*.log" },
    { __path__ = "/var/log/syslog" },
  ]
}

loki.source.file "logs" {
  targets    = local.file_match.logs.targets
  forward_to = [loki.write.hub.receiver]
}

loki.write "hub" {
  endpoint {
    url = "https://hub.example.com/loki/api/v1/push"
  }
}

# systemd
sudo systemctl enable alloy
sudo systemctl start alloy

3.2 Windows VM

# Windows 用 windows_exporter 取代 node_exporter
prometheus.exporter.windows "host" {
  enabled_collectors = ["cpu", "memory", "logical_disk", "net", "system"]
}

# Log 收集 Event Log
loki.source.windowsevent "events" {
  eventlog_name = ["Application", "System", "Security"]
  forward_to = [loki.write.hub.receiver]
}

3.3 Bare Metal / HPC

除標準 metric 外，HPC 機常需特別 metric：

  • NVIDIA DCGM Exporter：GPU 監控
  • IPMI Exporter：硬體感測（溫度、風扇、電源）
  • SNMP Exporter：交換器 / 儲存設備
  • Smart Exporter：磁碟健康度（SMART data）
# 加入 GPU 監控
prometheus.scrape "dcgm" {
  targets = [
    { __address__ = "localhost:9400" },  // dcgm-exporter
  ]
  forward_to = [prometheus.remote_write.hub.receiver]
}

3.4 Legacy 應用 Instrumentation

沒辦法改 code 的 legacy 應用：

  • Beyla / OBI eBPF：自動 instrument（Linux 5.8+）
  • Application log parsing：用 loki.process 從 log 萃取 metric（log-based metric）
  • Process Exporter：監控 process resource
  • Blackbox Exporter：HTTP / TCP probe（外部行為）

§ 4. K8s 整合（多 cluster）

4.1 多 cluster 識別

每 cluster 必須有唯一識別：

// Alloy DaemonSet 在每 cluster 設不同 external label
prometheus.remote_write "hub" {
  external_labels = {
    cluster = "production-us-east-1",
    cloud   = "aws",
    region  = "us-east-1",
    env     = "production",
  }
}

4.2 多 cluster 的 cardinality

⚠️ Cardinality 警告

加 cluster label = 每 metric 乘 N（cluster 數）。10 個 cluster × 1M series = 10M。
Mitigation：

  • 用 multi-tenancy（Mimir / Cortex）每 cluster 一個 tenant
  • 不重要 metric 在 cluster 內 aggregate 後再送
  • Federation 模式：每 cluster local Prometheus 算好 recording rule，只送結果

4.3 Cluster-level vs Multi-cluster Dashboard

  • 每 cluster 自己的 dashboard：詳細查看單 cluster 健康
  • Multi-cluster overview dashboard：所有 cluster 健康 + 比較
# Multi-cluster overview 範例
sum by (cluster) (
  rate(http_requests_total{job="api"}[5m])
)

# 「哪個 cluster 最 unhealthy」
topk(5,
  sum by (cluster) (
    rate(http_requests_total{status=~"5.."}[5m])
  )
  /
  sum by (cluster) (rate(http_requests_total[5m]))
)

4.4 Cluster Federation 模式

每 cluster 跑本地 Prometheus + recording rules，只把聚合後的 metric送中央： 

# Local Prometheus rules
- record: cluster:http_requests:rate5m
  expr: sum by (cluster, service) (rate(http_requests_total[5m]))

- record: cluster:cpu:usage
  expr: avg by (cluster) (rate(node_cpu_seconds_total{mode!="idle"}[5m]))

# 只 remote_write 這些 cluster: 開頭的 metric
remote_write:
  - url: https://hub.example.com/...
    write_relabel_configs:
      - source_labels: [__name__]
        regex: 'cluster:.*'
        action: keep

降低 90%+ 中央 storage / cost。缺點：失去細粒度（不能 drilldown 到 pod）。

  MyFunction:
    Type: AWS::Serverless::Function
    Properties:

        Variables:

# Logs → CloudWatch Logs（自動）+ subscription filter 送 Loki

5.3 SaaS dependencies 監控

第三方 SaaS（Stripe / SendGrid）通常無法 push metric。做法：

  • Synthetic check：k6 定時 probe 它們的 health endpoint
  • API webhook：訂閱事件，接收後 emit 為 metric
  • Status page scraping：定時抓 status.stripe.com
  • 從應用視角觀測：自家 app 呼叫第三方的 latency / error rate（最重要）

§ 6. Edge / IoT 監控

6.1 Edge 環境特殊性

  • 裝置數量大（1000+ 門市、10K+ IoT）
  • 網路不穩（4G / 衛星 / 偶爾離線）
  • 計算資源受限（小型 box）
  • 偶爾連網即可（不能 push 即時 telemetry）

6.2 兩層架構

Edge Device (POS / Sensor)
  ↓ local buffer
Edge Gateway (店內 / 區域)
  ↓ batch upload (有網路時)
Central Hub
  ↓
LGTM Stack

6.3 Edge Gateway 設計

# Edge Gateway: Alloy with local buffer
prometheus.remote_write "central" {
  endpoint {
    url = "https://hub.example.com/api/v1/push"
  }

  // 重要：buffer 設定
  queue_config {
    capacity          = 100000   // 大 buffer
    max_samples_per_send = 1000
    batch_send_deadline = "30s"
    max_backoff       = "10m"
    retry_on_http_429 = true
  }

  // WAL on disk（網路斷時不丟資料）
  wal {
    truncate_frequency = "2h"
    min_keepalive_time = "5m"
    max_keepalive_time = "8h"   // 最長保留 8 小時
  }
}

6.4 取捨：採樣 vs 完整

Edge 帶寬有限，必須取捨：

  • Critical metric：100% 送（POS 失敗、卡機）
  • Trace：tail-based sampling（只送 error / slow）
  • Log：只送 ERROR / WARN，INFO 留本地
  • 低頻 metric：5 分鐘 aggregate 一次後送

6.5 Edge 設備自身監控

不要忘了監控 edge 設備本身：

  • 裝置 online / offline 狀態（heartbeat metric）
  • 本地 disk / RAM 容量
  • 網路品質（packet loss / latency to gateway）
  • OS 版本 / 軟體版本（給 fleet management 用）
  • 地理位置（給 incident 報告用）

§ 7. 跨環境網路設計

7.1 連接選項

選項使用
Public Internet + TLS簡單、適合小規模
VPN (IPsec)on-prem ↔ cloud，一般敏感度
AWS Direct Connect / Azure ExpressRoute / GCP Interconnect高敏感、大流量
Private endpoint / PrivateLinkcloud-internal、不過 internet
Mesh VPN（Tailscale / WireGuard）edge / 多 site，現代化

7.2 認證方案

  • mTLS：雙向憑證（嚴格）
  • API Token / Bearer：簡單但要管 rotation
  • Cloud IAM（IRSA / Workload Identity）：cloud 內最佳
  • OIDC：跨環境統一

7.3 Telemetry 流量優化

  • Compression：snappy / gzip 壓縮 OTLP / Prometheus remote_write
  • Batching：合併小 batch 送（max_samples_per_send 調大）
  • Sampling：trace 不全送（tail-based 在 edge 端）
  • Aggregate before send：log-based metric / recording rule
  • Schedule：非 critical telemetry 集中在離峰送

§ 8. Multi-tenancy 設計

8.1 為什麼要 multi-tenancy

  • 不同環境 / 部門 / 客戶 isolated
  • Quota 控制（一個 team 不能用爆 storage）
  • Access control（dev 看不到 prod）
  • Cost allocation（按 tenant 計費）

8.2 Mimir / Loki / Tempo Tenant

三家原生支援 tenant，用 X-Scope-OrgID header 區分：

# Mimir push
curl -X POST https://mimir/api/v1/push \
  -H "X-Scope-OrgID: team-payments" \
  --data-binary @data

# Mimir query
curl -X POST https://mimir/api/v1/query \
  -H "X-Scope-OrgID: team-payments" \
  -d "query=up"

# Limits per tenant
overrides:
  team-payments:
    ingestion_rate: 100000
    max_global_series_per_user: 1000000
    max_query_length: 720h

  team-frontend:
    ingestion_rate: 50000
    max_global_series_per_user: 500000

8.3 Tenant 切分策略

策略適合
Per environment（prod / staging）環境隔離
Per team組織隔離
Per clustercluster 隔離
Per cloud多雲隔離
Per customer（B2B SaaS）客戶隔離 + 計費

實務：常組合使用，例如 tenant = "{env}-{team}" 如 prod-payments。

8.4 Grafana Multi-tenant

Grafana 提供兩層 tenant：

  • Organizations：完全隔離（不同 org 資料源、user 完全分開）
  • Teams + Folder permissions：細粒度（同 org 內權限控制）

§ 9. 跨環境 Trace 串接

9.1 痛點

使用者 request 跨多環境時，trace 容易斷：

User → CDN → Frontend (EKS) → API Gateway → Lambda → on-prem MySQL → SaaS API
       ↑      ↑                ↑            ↑       ↑               ↑
       多個邊界，每個都可能讓 trace 斷

9.2 W3C Trace Context

業界標準。所有環境都用相同 traceparent header：

traceparent: 00-abc123def456...-789xyz...-01
              ^   ^                      ^         ^
              v   trace_id (32 hex)      span_id   flags

tracestate: vendor1=value,vendor2=value

9.3 各環境 propagator 設定

  • OTel SDK：OTEL_PROPAGATORS=tracecontext,baggage
  • API Gateway：要設定 forward header（CloudFront / API GW 預設不 forward）
  • Service Mesh (Istio)：自動傳遞
  • Lambda：ADOT 自動處理
  • Frontend (Faro)：自動加 header

9.4 環境邊界 instrumentation

關鍵：邊界 component（API GW、CDN、Load Balancer）必須 forward header：

# AWS API Gateway
ForwardedHeaders:
  - traceparent
  - tracestate

# nginx
proxy_set_header traceparent $http_traceparent;
proxy_set_header tracestate $http_tracestate;

# CloudFront / Cloudflare
# 在 Origin Request 設定 whitelist headers
- traceparent
- tracestate

9.5 第三方 SaaS 的處理

你發送到 Stripe/Twilio 的 trace context 它不會回傳。但：

  • 呼叫前的 span 是你的
  • 呼叫的 latency / error 在你的 trace 裡
  • 這已足夠看出 SaaS 是不是瓶頸

§ 10. 災難備援與 HA

10.1 Telemetry 系統的 SPOF

⚠️ Observability 自己也要 observable

Telemetry 系統不能是 SPOF。如果 incident 時 monitoring 也壞了，盲飛。

10.2 多層 HA 設計

層HA 方式
Edge collector WAL + retry
Hub Gateway 3+ replicas + load balancer
Backend Mimir / Loki HA mode（multi-replica + replication factor）
Storage S3 / GCS multi-AZ
Grafana multi-replica + shared DB
Alertmanager 3 replicas（gossip）

10.3 跨 region DR

Primary Region (us-east-1)
├── Hub Gateway × 3
├── Mimir / Loki / Tempo（active）
└── Storage in S3 (auto cross-region replication)

DR Region (us-west-2)
├── Mimir / Loki / Tempo（standby/cold）
└── Storage 從 primary 同步

Failover 流程:
1. DNS 切換 hub.example.com → DR region
2. Edge collector retry 自動連到新 endpoint
3. DR 變 active，啟動 Mimir / Loki / Tempo
4. 從 S3 讀歷史資料

10.4 Black Box 監控（observability of observability）

# 用獨立的 monitoring 監控主 monitoring
external-monitoring:
  - synthetic check Hub endpoint（k6 / Pingdom）
  - synthetic query Mimir（確認查得到資料）
  - 比對 metric ingest rate（是否有資料進來）

# 警報通道也要冗餘
alertmanager:
  - Primary: 自家 Alertmanager → PagerDuty
  - Backup: 外部 service（StatusGator / Better Stack）→ SMS
  - 兩家都壞 → 老派的 cron + email

§ 11. 成本優化

11.1 Telemetry cost 三大來源

  1. Network：跨環境 / 跨 region traffic
  2. Storage：metric / log / trace 儲存
  3. Query：Grafana dashboard 每分鐘跑大量 query

11.2 Cardinality 是最大成本驅動

# 找 high-cardinality metric
topk(10, count by (__name__)({__name__=~".+"}))

# Drop 不需要的 label
metric_relabel_configs:
  - source_labels: [request_id]
    action: labeldrop
  - source_labels: [user_id]
    action: labeldrop
  - source_labels: [trace_id]
    action: labeldrop

11.3 Log volume 控制

  • 結構化 log（不要 print stack trace 5 次）
  • 不要 log 整個 request body（敏感 + 大）
  • DEBUG log production 不開
  • 用 Loki LogQL 算「多少 log 從 哪 來」找冗餘 log
// Loki: top 10 noisy services
topk(10,
  sum by (service) (rate({}[5m]))
)

11.4 Trace sampling 調節

  • 邊緣 100% 送（讓 collector 決策）
  • Tail-based 在 collector：error 100% / slow 100% / normal 1-5%
  • 儲存 retention 短（trace 通常 7-15 天就夠）

11.5 監控自身 cost

# Telemetry pipeline cost dashboard
- Network traffic（GB/day）
- Storage growth rate（GB/day）
- Cardinality trend
- Top 10 高 cardinality metric
- Log ingestion rate per service
- Query frequency / duration

# 月度 review
1. 砍掉沒人看的 dashboard
2. 砍掉沒觸發過的 alert
3. 砍掉沒 query 過的 metric label
4. retention 是否合理

  • Multi-tenancy：Mimir/Loki/Tempo 用 X-Scope-OrgID，per-tenant quota；常用 {env}-{team}
  • 跨環境 Trace 用 W3C Trace Context（traceparent / tracestate），邊界 component 必須 forward
  • HA：Edge WAL + Hub 多 replicas + Mimir replication + Storage 多 AZ + 跨 region DR
  • Observability of observability：用獨立工具監控主 monitoring（Black Box）
  • Cost 優化：cardinality / log volume / trace sampling / retention，定期 review 砍冗餘
