Chapter 22 · Alertmanager 概念

Alertmanager 概念、架構與生命週期

Prometheus 告訴您「哪裡出問題」，Alertmanager 決定「該怎麼通知」。 它處理去重、分組、抑制、靜默、路由——五個核心能力把雜亂的告警轉成有用的通知。 本章是進入 Part 2 的起點，從架構到 alert 完整生命週期一次講透。

§ 1. 為什麼需要 Alertmanager

在 Ch 15 我們看過：Prometheus 的 alerting rule 只負責「判斷現在發生了什麼」。 它不處理：

  • 同一個 alert 重複觸發 → 該不該重複通知？
  • 500 個 pod 同時掛 → 要送 500 條訊息還是 1 條？
  • 「Cluster 整個掛」+「每個 pod 都掛」 → 只報根因不報症狀？
  • 維護中時段 → 怎麼暫停通知？
  • 不同 severity / team → 通知不同 channel？
  • HA Prometheus → 如何避免雙倍通知？

這些都是 Alertmanager 的職責。

✅ 關鍵設計原則

Prometheus = 「判斷層」（What is broken?）
Alertmanager = 「通知層」（Who to tell, how often, through which channel?）

§ 3. 五大核心能力

3.1 Deduplication（去重）

當兩個 HA Prometheus 都送同一條 alert 過來，Alertmanager 視為一個。 判斷依據：除了 replica 外的所有 label 一樣 = 同一個 alert。

這也是為什麼 Ch 15 強調 HA 部署要設 alert_relabel_configs 移除 replica。

3.2 Grouping（分組）

把多個相關的 alert 「打包」成一個通知。例：

  • Cluster 網路分區 → 100 個服務 instance 全部告警 → 分組成一個通知
  • 同 service 不同 endpoint 都 5xx → 一個通知列出所有 endpoint

分組依據：group_by 設定的 label。同 group 的 alert 共享一個通知訊息。

3.3 Inhibition（抑制）

當高優先級 alert 已 firing 時，抑制低優先級的相關 alert。例：

  • ClusterDown 已 firing → 抑制該 cluster 內所有的 PodDown warning（症狀型）
  • DiskFull 已 firing → 抑制該機器的 HighDiskLatency

用「根因 alert」抑制「症狀 alert」，on-call 一條告警就夠了。

3.4 Silences（靜默）

手動暫停通知。透過 UI 或 API 建立，不寫在設定檔。例：

  • 維護視窗：「service=web 從 03:00 到 05:00 靜默 2 小時」
  • 已知問題：「正在處理，先靜默 1 小時」
  • 測試環境：「environment=staging 永久靜默部分 alert」

3.5 Routing（路由）

根據 alert 的 label，決定送到哪個 receiver。例： 

severity=critical → PagerDuty
severity=warning + team=platform → Platform Slack
severity=warning + team=database → DBA Email
其他 → Default Slack

3.6 五大能力的差異對比

能力啟動方式是否需設定檔典型用途
去重自動否（自動處理）HA 場景
分組自動是（route.group_by）降低通知量
抑制自動是（inhibit_rules）根因 → 症狀
靜默手動否（UI/API）維護、已知問題
路由自動是（route）分流到不同 receiver

§ 5. Alert 的資料結構

Prometheus 送往 Alertmanager 的 alert 就是這樣的 JSON：

[
  {
    "labels": {
      "alertname": "InstanceDown",
      "severity": "critical",
      "instance": "api-1:9100",
      "job": "node-exporter",
      "team": "platform",
      "cluster": "production-tw"
    },
    "annotations": {
      "summary": "Instance api-1:9100 is down",
      "description": "Has been down for 5 minutes",
      "runbook_url": "https://wiki/runbook/instance-down"
    },
    "startsAt": "2026-05-02T10:23:45Z",
    "endsAt": "0001-01-01T00:00:00Z",
    "generatorURL": "http://prometheus:9090/graph?...",
    "fingerprint": "abc123def456"
  }
]

5.1 重要欄位解讀

欄位意義
labels所有用於 routing / grouping / inhibition 的 key-value
annotations給人看的描述、URL（不參與 routing）
startsAtAlert 開始 firing 的時間
endsAtAlert 結束時間（firing 中為「未來時間」，resolved 後填實際結束）
generatorURLPrometheus 端的查詢連結（給人看現場）
fingerprintlabel set 的 hash，用於識別「同一個 alert」

5.2 Resolved alert 是怎麼產生的

Prometheus 在 alert 不再 firing 時，會送一個帶有「過去時間 endsAt」的版本： 

{
  ...
  "endsAt": "2026-05-02T10:35:00Z"   // 不是 0001-01-01，代表已結束
}

Alertmanager 收到後，若 receiver 設了 send_resolved: true，會發送 resolved 通知。

§ 6. Routing Tree 概念

Alertmanager 的 routing 是一棵樹，根節點是「default route」，下面可以掛多個 child route。 Alert 從根開始，逐層匹配，找到最後一個匹配的 leaf 為止（除非 continue: true）。

6.1 樹形結構範例

route:
  receiver: 'default-slack'      ← 根 (catch-all)
  group_by: ['alertname']

  routes:
    - matchers:                   ← 第一層 child
        - severity = "critical"
      receiver: 'pagerduty'

      routes:                     ← 第二層 child
        - matchers:
            - team = "database"
          receiver: 'db-pagerduty'

    - matchers:
        - team = "frontend"
      receiver: 'frontend-slack'

    - matchers:
        - environment = "staging"
      receiver: 'staging-email'

6.2 路由匹配演算法

  1. 從根節點開始
  2. 對每個 child route，檢查 alert 是否匹配 matchers
  3. 找到第一個匹配的 child → 進入該 child
  4. 遞迴檢查該 child 的 child route
  5. 到不能再下時，該 leaf 的 receiver 就是目標
  6. 若該 child 設了 continue: true，會繼續尋找後面的 sibling 也匹配的 route

default-slack"] A["matchers · severity=critical

§ 7. 時間參數三劍客

這三個參數決定 alert 從觸發到收到通知的延遲，是影響 on-call 體驗最重要的設定。

參數意義典型值
group_wait 新 group 第一次通知前等多久（讓相關 alert 累積） 30s
group_interval 同一 group 已通知過後，再次通知（若有新 alert 加入）的最短間隔 5m
repeat_interval 同樣的 group 已通知過後，重複通知（沒新變化）的間隔 4h

7.1 Timeline 範例

t=0:00  Alert A 觸發 (group X 第一個成員)
t=0:30  group_wait 結束, 通知發送 (含 alert A)
t=0:45  Alert B 觸發 (同 group X)
t=5:30  group_interval 到, 看到 group 有變動, 通知 (含 A+B)
t=4:30:00  repeat_interval 到, 即使 group 沒變化, 重複通知

7.2 為什麼這三個都需要存在？

沒有它會發生什麼哪個參數防止
第一個 alert 一進來就送通知，導致同 group 後續 alert 各自通知group_wait
持續加入新 alert 會把通知頻率拉到很高group_interval（最低間隔）
持續 firing 但沒新東西，幾天都沒 reminderrepeat_interval
⚠️ 三個值的關係

group_wait < group_interval << repeat_interval
典型值：30s < 5m << 4h
若 group_wait 設太大（5m+），critical alert 會晚 5 分鐘才收到，誤事。

§ 8. UTF-8 Matcher 與 Classic Matcher

8.1 兩種語法並存

Alertmanager 自 v0.27 引入 UTF-8 matcher（為了支援 Prometheus 3.x 的 UTF-8 metric/label 名稱）。 現在兩種 matcher 語法並存：

類型語法狀態
Classic Matcher用 match + match_reLegacy，仍可用
UTF-8 Matcher用 matchers（PromQL-like）推薦新寫法

8.2 對照範例

# Classic（v0.27 之前）
- match:
    severity: critical
  match_re:
    service: api|web
  receiver: 'pager'

# UTF-8（v0.27+ 推薦）
- matchers:
    - severity = "critical"
    - service =~ "api|web"
  receiver: 'pager'

8.3 UTF-8 Matcher 四種運算子

運算子意義
=相等
!=不相等
=~regex 匹配（整體錨定）
!~regex 不匹配

跟 PromQL selector 完全一致，不用學新東西。

8.4 三種運作模式

v0.27+ 啟動 Alertmanager 時可指定：

模式意義
Classic Mode只用 classic parser（向後相容）
Fallback Mode（預設）先試 UTF-8 parser，失敗時退回 classic + log warning
UTF-8 Strict Mode只用 UTF-8 parser，遇舊語法直接報錯
 
# 啟用 UTF-8 strict
alertmanager --config.file=alertmanager.yml \
  --enable-feature=utf8-strict-mode

§ 9. 與 Prometheus 的對話協議

9.1 Prometheus 端設定

alerting:
  alertmanagers:
    - api_version: v2          # 必填，固定 v2（v1 已移除）
      static_configs:
        - targets:
            - 'alertmanager-1:9093'
            - 'alertmanager-2:9093'
            - 'alertmanager-3:9093'

9.2 通訊細節

  • 協議：HTTP POST /api/v2/alerts
  • 頻率：每 60 秒（可調 --rules.alert.resend-delay）
  • 內容：所有當前 active alert 的完整 list（不是 delta）
  • HA：Prometheus 對每一個 AM 都送，由 AM gossip 去重

9.3 為什麼要送給「每一個」AM？

🚨 永遠不要在 Prometheus → AM 之間放 Load Balancer

若用 LB，Prometheus 只送給其中一個 AM。如果該 AM 掛了，alert 就遺失。
正確做法：Prometheus 同時送給所有 AM，由 AM cluster 用 gossip protocol 去重。 這樣任何一個 AM 掛掉，其他 AM 仍能接力處理。

§ 10. 設計哲學與限制

10.1 設計哲學

  1. 無狀態優先：alert 在記憶體（有 nflog 持久化通知歷史），重啟後幾分鐘就重建
  2. 跟 Prometheus 解耦：能接 Prometheus 之外的告警源（Cortex、Mimir、自製系統）
  3. Receiver pluggable：webhook 機制讓任何系統都能整合
  4. Operator-friendly UI：Web UI 讓 on-call 直接操作 silence

10.2 已知限制

⚠️ Alertmanager 沒做的事
  1. Escalation policy（5 分鐘沒人 ack 就升級到下一個 on-call）→ 用 PagerDuty / OpsGenie
  2. On-call schedule（誰這週值班）→ 用 PagerDuty / Grafana OnCall
  3. Incident response 工作流（自動建 Jira ticket、Slack channel）→ 用 incident.io / FireHydrant
  4. Alert 歷史儲存（過去半年的 alert 統計）→ 用其他系統存 webhook 紀錄
  5. 真正的 ack / acknowledge（按一個按鈕表示「我看到了」）→ 用整合的 incident tool

Alertmanager 是「告警分發層」，不是 incident management。 生產環境通常: Alertmanager + PagerDuty / Grafana OnCall 搭配使用。

✅ 本章重點摘要

30 秒回顧
  • Alertmanager 處理「該怎麼通知」，跟 Prometheus 「判斷問題」分工
  • 五大能力：Deduplication / Grouping / Inhibition / Silence / Routing
  • 處理順序：接收 → 去重 → 靜默 → 抑制 → 路由 → 分組 → 模板 → 發送
  • Alert 的 fingerprint = label set hash，用於識別「同一個 alert」
  • Routing tree 是樹形結構，由根 catch-all 與 child route 組成
  • 三個時間參數：group_wait（30s）→ group_interval（5m）→ repeat_interval（4h）
  • v0.27+ 推薦用 UTF-8 matchers 語法（PromQL-like），舊語法仍向後相容
  • Prometheus → AM 通訊：固定 api_version: v2，HA 場景絕對不能用 LB，要送給每一個 AM
  • Alertmanager 不做 escalation、on-call schedule、incident workflow，那些用 PagerDuty 等 incident tool
