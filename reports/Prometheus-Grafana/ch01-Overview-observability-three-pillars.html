Chapter 01 · 觀念基礎

可觀測性三本柱與 CNCF 生態

在深入 Prometheus 與 Grafana 之前，我們必須先建立可觀測性（observability）的整體心智模型。 本章說明 Metrics / Logs / Traces 三本柱、它們各自解決什麼問題、以及在 CNCF 生態中的工具地圖。

§ 1. 從監控到可觀測性的演進

在傳統單體（monolith）時代，「監控（monitoring）」的需求很單純： 幾台主機、一個資料庫、一個應用程式，工程師只要透過 Nagios、Zabbix 之類的工具 定期檢查 CPU、記憶體、磁碟使用率即可。當系統出問題， 透過 SSH 進主機看 log 通常就能找到原因。

但雲原生時代徹底改變了這個遊戲規則：

  • 微服務（microservices）：一個業務操作可能跨越 10 個以上的服務
  • 容器化：Pod 隨時可能被重新排程，IP 不固定
  • Auto scaling：實例數量隨負載動態變化
  • 多雲與混合架構：服務跨越多個雲端與資料中心
  • 持續部署：一天可能部署數十次

在這種環境下，問題的形態完全不同。一個 API 變慢的根因，可能是： 某個下游服務的某個版本，在某個 K8s node 上，因為某個 pod 的 GC 暫停所致。 這已經不是「監控」能處理的範疇，我們需要的是「可觀測性」。

ℹ️ Monitoring vs Observability

監控（monitoring）是「我預先定義要看的指標，當指標越界時告警」——這是回答已知未知（known unknowns）的問題。
可觀測性（observability）是「我可以從外部訊號推論系統內部任意狀態」——這是回答未知未知（unknown unknowns）的問題。
可觀測性是系統的一個屬性，監控是達到可觀測性的一種手段。

2.1 Metrics（指標）

Metrics 是隨時間採樣的數值（numeric measurements over time）， 例如「每秒請求數」、「記憶體使用率」、「目前線上使用者數」等。 它的特性是：

  • 儲存效率高：相同 metric 在不同時間點只是數字差異，可高度壓縮
  • 聚合容易：求平均、百分位數、總和都很快
  • 適合告警：閾值告警最直觀的素材
  • 缺乏細節：只知道「QPS 突然下降 50%」，不知道「為什麼」

Prometheus 是這個領域的代表性工具。一個典型的 Prometheus metric 長這樣： 

http_requests_total{method="POST", path="/api/login", status="200"} 12453

這條 metric 包含：metric 名稱（http_requests_total）、 一組 label（method、path、status）， 以及在某個時間點的值（12453）。

2.2 Logs（日誌）

Logs 是離散的事件記錄，例如： 

2026-05-01T10:23:45Z [ERROR] order-service: payment failed for order_id=12345
  reason=insufficient_funds, user_id=u_789, amount=99.99

它的特性是：

  • 資訊密度高：可包含完整的上下文與堆疊追蹤
  • 儲存成本高：相比 metrics，儲存量可能是百倍以上
  • 查詢慢：除非有索引，全文搜尋大量 log 很耗資源
  • 適合事後分析：找出特定錯誤的詳細原因

在 Grafana 生態中，Loki 是 logs 的代表性工具， 它採用「只索引 label，不索引內容」的策略，大幅降低儲存成本。 其他常見的 log 系統包括 Elasticsearch（ELK Stack）、Splunk、Fluentd。

2.3 Traces（追蹤）

Traces 記錄一個請求在分散式系統中的完整路徑。 一個 trace 由多個 span 組成，每個 span 代表一個操作。

20ms] --> B[Auth Service

它的特性是：

  • 因果關係明確：清楚看到「服務 A 呼叫服務 B」的鏈條
  • 定位瓶頸快速：直接看出哪個 span 最慢
  • 需要程式碼埋點：通常需要 OpenTelemetry SDK 或 auto-instrumentation
  • 取樣（sampling）是必要的：100% 取樣對效能與儲存都是負擔

在 Grafana 生態中，Tempo 是 traces 的代表性工具。 其他常見的有 Jaeger、Zipkin、AWS X-Ray、Google Cloud Trace。

2.4 三者協同（Correlation）

三本柱的真正威力在於協同關聯。一個典型的故障排除流程是：

  1. Metrics 告警：「下單成功率 SLO 跌破 99%」
  2. Dashboards 切片：發現只有 EU 區域受影響
  3. Logs 找錯誤：搜尋該區域的 ERROR log，看到 connection timeout
  4. Traces 找根因：點開一個失敗請求的 trace，看到是 inventory-service 的 DB 查詢卡住
  5. 修復：擴展 DB connection pool 或調整慢查詢

Grafana 推出 Exemplars 功能，讓 metrics 直接連結到對應的 trace； Loki 的 LogQL 與 PromQL 語法相近，可從 Grafana 同時查 logs 與 metrics。 這就是「Grafana LGTM Stack」（Loki / Grafana / Tempo / Mimir）的核心價值。

§ 3. 第四本柱（候選）：Profiles 與 Events

近年來業界開始討論「第四本柱」的候選者，主要有兩個：

3.1 Profiles（效能分析）

Profiles 是程式執行時 CPU、記憶體、goroutine 等資源使用的快照（snapshot）， 通常以火焰圖（flame graph）視覺化。傳統上 profiling 是開發階段才用的工具， 但持續性 profiling（continuous profiling）讓我們可以在生產環境 隨時拿到「為什麼這台機器 CPU 100%」的完整呼叫堆疊答案。

Grafana 的 Pyroscope 與 Polar Signals 的 Parca 是這個領域的代表。

3.2 Events（事件）

Events 記錄系統狀態變化（部署、scaling、設定變更等）。 雖然事件本質上也是一種 log，但因其「結構化、業務語義強、頻率不高」的特性， 很適合單獨處理，並與其他訊號交叉關聯。 例如：「告警時間點是否剛好對應到某次部署？」

§ 4. Pull-based vs Push-based 模型

收集 metrics 有兩種主要模式，兩者各有利弊。Prometheus 採用 pull 模型， 這是它與其他監控系統最大的差異之一。

graph TB subgraph Pull["Pull Model · Prometheus "] P[Prometheus] -.HTTP GET /metrics.-> A1[App 1] P -.HTTP GET /metrics.-> A2[App 2] P -.HTTP GET /metrics.-> A3[App 3] end subgraph Push["Push Model · StatsD / InfluxDB "] B1[App 1] -- push --> S[StatsD/InfluxDB] B2[App 2] -- push --> S B3[App 3] -- push --> S end

Pull 模型的優點

  • 監控伺服器健康狀態：抓不到 metrics 自然就知道 target 掛了
  • 應用端不需要知道 Prometheus 的位置：解耦合
  • 易於除錯：可以用 curl 手動拉一次看 metrics
  • 多個 Prometheus 可同時抓同一個 target：HA 與分區簡單

Pull 模型的缺點

  • 需要 service discovery：Prometheus 必須知道 targets 在哪
  • 不適合短任務：批次任務跑完就消失，可能還沒被抓到（解法：Pushgateway）
  • 網路穿透問題：若 target 在防火牆後，Prometheus 抓不到

§ 6. 監控方法論：USE、RED、Four Golden Signals

工具是手段，方法論才能決定我們該監控什麼。 業界有三套經典方法論，分別適用於不同場景：

6.1 USE Method（資源導向）

由 Brendan Gregg 提出，適合監控資源（resources），例如 CPU、記憶體、磁碟、網路。

  • Utilization（使用率）：資源忙碌的百分比
  • Saturation（飽和度）：資源無法處理的工作堆積量
  • Errors（錯誤）：錯誤事件的計數

例如監控一台主機的 CPU，要看 utilization（CPU 使用率）、saturation（runnable 佇列長度）、errors（CPU 錯誤事件）。

6.2 RED Method（服務導向）

由 Tom Wilkie（Grafana 創辦人之一）提出，適合監控服務（services）， 特別是請求/回應型的微服務。

  • Rate（速率）：每秒請求數（QPS / RPS）
  • Errors（錯誤率）：失敗的請求數或比例
  • Duration（延遲）：請求處理時間（通常看 P50、P95、P99）

6.3 Four Golden Signals（Google SRE）

Google SRE 書中提出的方法論，是 RED 的擴充版本：

  • Latency（延遲）：成功與失敗的請求延遲都要看
  • Traffic（流量）：QPS、頻寬、併發連線數
  • Errors（錯誤率）：HTTP 5xx、業務邏輯錯誤、SLO 違反
  • Saturation（飽和度）：系統還能撐多久才達極限

6.4 三套方法論的選擇

✅ Best Practice
  • 監控主機 / 資源層用 USE（CPU、Memory、Disk、Network）
  • 監控應用 / 服務層用 RED（每個 endpoint 都要有）
  • 監控整體系統健康用 Four Golden Signals
  • 三者並非互斥，實務上是分層疊加使用

§ 7. 常見陷阱與最佳實踐

⚠️ 陷阱 1：把 Prometheus 當成所有事情的解答

Prometheus 設計目標是可靠的數值時序資料，不適合： 事件記錄（用 logs）、按請求計費（精度不夠）、事務追蹤（用 traces）。 硬要把 logs 灌進 Prometheus 的 label 中是經典反模式。

⚠️ 陷阱 2：高基數（high cardinality）爆炸

Prometheus 的 label 不能放無界的值，例如：user_id、request_id、email。 每個唯一的 label 組合會產生一條獨立的 time series， 上百萬條 series 會讓 Prometheus 記憶體爆掉。後面章節會詳細討論。

⚠️ 陷阱 3：告警疲勞（alert fatigue）

不是每個 metric 都需要告警。告警太多會讓真正重要的告警被忽略。 原則：告警必須是「有人需要立即行動」的事。Symptom-based alerting 優於 cause-based。

✅ 最佳實踐
  1. 先有方法論，再選工具：知道要監控什麼，再決定怎麼存
  2. 三本柱協同：metrics 觸發告警 → logs 查細節 → traces 找根因
  3. 從外部訊號出發：監控使用者真正體驗到的指標（例如成功率），而非單純的內部狀態（例如 CPU）
  4. 一切皆可程式碼化（as code）：dashboards、alerting rules 都納入 Git 版本控制
