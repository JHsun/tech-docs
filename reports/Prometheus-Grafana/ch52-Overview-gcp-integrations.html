Chapter 52 · GCP 整合

GCP 觀測性整合：Cloud Monitoring、Logging、Trace、Profiler

Google Cloud 提供完整的觀測性服務集（前身為 Stackdriver）：Cloud Monitoring（metric）、 Cloud Logging（log）、Cloud Trace（trace）、Cloud Profiler（profile）。 GCP 還提供 Google Managed Prometheus (GMP) 與 OpenTelemetry 第一公民支援。 本章涵蓋這些服務、與 Grafana 整合、GKE 觀測性、跨雲策略。

§ 1. GCP 觀測性服務全景

1.1 Stackdriver 改名背景

GCP 觀測性原名 Stackdriver（2014 收購）。2020 年起改名為「Google Cloud Operations」， 四個子服務：Cloud Monitoring / Cloud Logging / Cloud Trace / Cloud Profiler。

1.2 GCP 服務矩陣

訊號GCP 原生GCP Managed OSS
MetricsCloud MonitoringGoogle Managed Prometheus (GMP)
LogsCloud Logging—（用 Loki on GKE）
TracesCloud Trace—（用 Tempo on GKE）
ProfilesCloud Profiler—（用 Pyroscope on GKE）
VisualizationCloud Monitoring DashboardsGrafana（自架 / Cloud）
AlertingCloud Monitoring AlertsPrometheus AlertManager / Grafana

1.3 GCP 與 AWS 的觀測性差異

  • GCP 沒有「Managed Grafana」（不像 AWS AMG）→ 需要自架或 Grafana Cloud
  • GMP 是 OSS Prometheus 的原生上游（不是 Cortex fork）
  • OpenTelemetry 第一公民（GCP 是 OTel 主要貢獻者）
  • Cloud Logging 比 CloudWatch Logs 強（query 能力）

1.4 推薦路線

✅ GCP 推薦組合
  • Metrics：GMP + 自架 Grafana on GKE（或 Grafana Cloud）
  • Logs：Cloud Logging（內建好用）
  • Traces：OpenTelemetry → Cloud Trace 或 Tempo
  • Profiles：Cloud Profiler（GCP 限定）或 Pyroscope

§ 2. Cloud Monitoring

2.1 概念

GCP 內建 metric 系統，自動收集 GCP 資源 metric（GCE / Cloud SQL / GKE / Pub/Sub 等）。 支援自訂 metric。

2.2 Metric 結構

Metric Type: compute.googleapis.com/instance/cpu/utilization
  └── Resource Type: gce_instance
      └── Resource Labels: project_id, instance_id, zone
          └── Metric Labels: instance_name
              └── Value Type: DOUBLE
                  └── Metric Kind: GAUGE

2.3 自訂 metric

# Python SDK
from google.cloud import monitoring_v3

client = monitoring_v3.MetricServiceClient()
project_name = f"projects/{project_id}"

series = monitoring_v3.TimeSeries()
series.metric.type = "custom.googleapis.com/my_app/orders_placed"
series.resource.type = "global"
series.resource.labels["project_id"] = project_id

point = monitoring_v3.Point()
point.value.double_value = 1.0
point.interval.end_time.seconds = int(time.time())
series.points = [point]

client.create_time_series(name=project_name, time_series=[series])

2.4 MQL（Monitoring Query Language）

類似 SQL 的語法（取代舊的 GUI builder）：

fetch gce_instance
| metric 'compute.googleapis.com/instance/cpu/utilization'
| filter resource.zone = 'us-central1-a'
| group_by 1m, [value_utilization_mean: mean(value.utilization)]

2.5 PromQL（v2024+）

Cloud Monitoring 同時支援 PromQL 查詢（透過 GMP 的整合）：

rate(compute_googleapis_com:instance_cpu_utilization[5m])

2.6 限制

  • 自訂 metric 也按樣本收費（類似 CloudWatch）
  • retention：6 週（標準）/ 24 個月（高解析）
  • cardinality 限制：每 metric 最多 30K series

§ 3. Cloud Logging

3.1 概念

GCP 內建 log 服務，自動收集所有 GCP 服務 log + 應用 log。 強大的 query language（基於 LogQL 風格）。

3.2 Log 來源

  • GCP services：自動（GCE serial console、Cloud Functions、Cloud Run、GKE）
  • 應用：via Logging Agent（VM）/ Fluent Bit（GKE）/ SDK 直接寫
  • Audit Logs：所有 API 呼叫自動記錄

3.3 從 GKE 收 log

GKE 預設啟用 Cloud Logging（Logging Agent 自動部署）：

# 應用只要 stdout 寫，自動到 Cloud Logging
logger.info("Order placed", { order_id: 12345 })

# 結構化 JSON 自動解析為 jsonPayload
{"severity":"INFO","message":"Order placed","order_id":12345}

3.4 Logs Explorer Query

resource.type="k8s_container"
resource.labels.namespace_name="production"
severity>=ERROR
jsonPayload.service="api"
timestamp>="2026-05-02T00:00:00Z"

3.5 Log Sinks（路由）

把 log 路由到別處：

  • BigQuery：分析、長期儲存
  • Cloud Storage：歸檔
  • Pub/Sub：streaming 給其他系統
  • Logging Bucket：custom retention
gcloud logging sinks create my-bq-sink \
  bigquery.googleapis.com/projects/PROJECT/datasets/logs \
  --log-filter='severity>=ERROR'

3.6 Log-based Metrics

從 log 萃取出 metric（類似 Loki 的功能）：

gcloud logging metrics create error_count \
  --log-filter='severity=ERROR AND resource.labels.service_name="api"' \
  --description="Count of API errors"

# 變成 Cloud Monitoring 中的 metric
custom.googleapis.com/log/user/error_count

§ 4. Cloud Trace 與 OpenTelemetry

4.1 Cloud Trace 概念

GCP 原生 distributed tracing。歷史可追溯到 Google 內部的 Dapper（業界 trace 概念的起源）。

4.2 OpenTelemetry First-class

GCP 是 OpenTelemetry 的主要推手之一。原生的 trace 接收方式：

  • OTLP gRPC：Cloud Trace 直接接收
  • OpenTelemetry SDK：跨語言一致
  • OpenTelemetry Collector：可中間轉送

4.3 SDK 整合範例（Python）

from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.exporter.cloud_trace import CloudTraceSpanExporter
from opentelemetry.sdk.trace.export import BatchSpanProcessor

provider = TracerProvider()
provider.add_span_processor(
    BatchSpanProcessor(CloudTraceSpanExporter())
)
trace.set_tracer_provider(provider)

tracer = trace.get_tracer(__name__)

with tracer.start_as_current_span("my_operation"):
    # ...

4.4 GKE 全自動 Trace

GKE Autopilot / Standard 啟用 Managed Service for Prometheus 後，trace 也自動啟用：

gcloud container clusters create my-cluster \
  --enable-managed-prometheus \
  --workload-pool=PROJECT.svc.id.goog

4.5 Cloud Trace 限制

  • Sampling 預設低（避免成本）
  • UI 不如 Tempo 強
  • 無內建 service graph（需配 Cloud Monitoring）

4.6 Cloud Trace vs Tempo

典型選擇：

  • 輕量、GCP-only：Cloud Trace
  • 進階查詢 / 大規模 / 跨雲：自架 Tempo on GKE

§ 5. Cloud Profiler

5.1 Cloud Profiler 概念

GCP 原生 continuous profiler。歷史比 Pyroscope 早（2018），是業界最早的 production profiling 服務之一。 支援：

  • CPU profile
  • Heap profile
  • Allocation profile
  • Wall time（含 IO 等待）
  • Goroutine（Go）

5.2 語言支援

  • Go（最完整）
  • Java
  • Python
  • Node.js

5.3 Go 整合

import "cloud.google.com/go/profiler"

func main() {
    profiler.Start(profiler.Config{
        Service:        "my-service",
        ServiceVersion: "1.0.0",
    })
    // 跑您的應用
}

5.4 Cloud Profiler vs Pyroscope

面向Cloud ProfilerPyroscope
運維GCP 全管自架
整合 LGTMvia Grafana plugin原生
UIGCP console（簡單）Pyroscope UI（強）
Diff view有有（更直覺）
Trace 整合via 共用 trace_idtracesToProfiles 直接跳
Costper-profile-hourstorage cost

§ 6. Google Managed Prometheus (GMP)

6.1 GMP 簡介

GCP 的 Managed Prometheus 服務，底層其實是 Monarch（Google 內部 metric 系統， 支撐 YouTube / Search 等規模）。對外提供 Prometheus 相容的 API。

6.2 GMP 與 AWS AMP 對比

面向GMPAMP
底層Monarch（Google 內部）Cortex（OSS fork）
規模無實際限制需申請提高 limits
Retention24 個月150 天
Cardinality無人為限制每 workspace 1M（可申請）
Costper-sampleper-sample
Scrape mode支援！不支援（必須 push）
Self-deployed內建選項無

6.3 GMP 三種使用模式

GMP 提供獨特的「漸進採用」路徑：

Mode 1: Managed collection（最簡單）

# 啟用 GKE GMP
gcloud container clusters update my-cluster \
  --enable-managed-prometheus

# 自動部署 collector，自動 scrape PodMonitoring CR

Mode 2: Self-deployed collection（最有彈性）

用自己的 Prometheus 或 Alloy，remote_write 到 GMP 端點。

Mode 3: 混合模式

部分用 managed scrape，部分用 self-deployed。

6.4 PodMonitoring CR（GKE 專屬）

apiVersion: monitoring.googleapis.com/v1
kind: PodMonitoring
metadata:
  name: my-app
  namespace: production
spec:
  selector:
    matchLabels:
      app: my-app
  endpoints:
    - port: metrics
      interval: 30s

類似 Prometheus Operator 的 ServiceMonitor，但是 GMP 原生支援。

6.5 GMP 認證

用 GCP Service Account / Workload Identity（後面詳述）。

§ 7. GMP 部署模式

7.1 Managed Collection on GKE

最簡路徑：

# 1. Create GKE cluster with GMP
gcloud container clusters create my-cluster \
  --enable-managed-prometheus \
  --workload-pool=PROJECT.svc.id.goog \
  --zone us-central1-a

# 2. Deploy app with metrics endpoint
kubectl apply -f my-app.yaml

# 3. Create PodMonitoring
kubectl apply -f podmonitoring.yaml

# 4. Done. metric 自動到 GMP

7.2 Self-deployed with Alloy

# Alloy config.river
prometheus.scrape "k8s_pods" {
  // ... discovery 設定
  forward_to = [prometheus.remote_write.gmp.receiver]
}

prometheus.remote_write "gmp" {
  endpoint {
    url = "https://monitoring.googleapis.com/v1/projects/PROJECT/location/global/prometheus/api/v1/write"

    // OAuth2 認證
    oauth2 {
      client_id     = sys.env("GCP_CLIENT_ID")
      client_secret = sys.env("GCP_CLIENT_SECRET")
      token_url     = "https://oauth2.googleapis.com/token"
      scopes        = ["https://www.googleapis.com/auth/monitoring.write"]
    }
  }
}

7.3 GMP 對非 GKE 環境

GMP 不限 GKE！其他環境也能 push：

  • EC2 上的 Prometheus 可 remote_write 到 GMP（cross-cloud）
  • On-prem K8s 可 push
  • 本機開發 push 到 GMP test 也行

§ 8. GKE 觀測性

8.1 GKE 觀測性 stack

GKE Cluster
├── Application pods
├── Managed Service for Prometheus（GMP collector）
│   └── 自動 scrape PodMonitoring + 寫入 GMP
├── Cloud Logging Agent（自動部署）
│   └── 收 stdout 到 Cloud Logging
└── 可選：自架 Grafana / Loki / Tempo

8.2 GKE 啟用全套觀測性

gcloud container clusters create my-cluster \
  --enable-managed-prometheus \
  --logging=SYSTEM,WORKLOAD \
  --monitoring=SYSTEM,WORKLOAD,POD,DEPLOYMENT,STATEFULSET,DAEMONSET,HPA,STORAGE,CADVISOR,KUBELET \
  --workload-pool=PROJECT.svc.id.goog

8.3 GKE Autopilot

Autopilot 模式更簡單：所有觀測性預設啟用，不用設定。

gcloud container clusters create-auto my-cluster \
  --region us-central1

8.4 GKE Application Performance（v2025+）

新功能：自動 instrument app（無需 SDK）：

  • 用 eBPF 自動產生 trace（類似 Beyla）
  • Latency / error rate 自動 dashboard
  • Service map 自動畫

§ 9. Grafana 整合 GCP

9.1 沒有 Managed Grafana

GCP 不像 AWS 有 AMG。三個選擇：

  • 自架 Grafana on GKE（最常見）
  • Grafana Cloud（SaaS，方便）
  • Cloud Monitoring Dashboards（GCP 原生，能力較弱）

9.2 自架 Grafana 連 GMP

# Datasource (Grafana on GKE)
- name: GMP
  type: prometheus
  url: https://monitoring.googleapis.com/v1/projects/PROJECT/location/global/prometheus
  jsonData:
    httpMethod: POST
    timeInterval: 30s
    prometheusType: Mimir   # GMP 兼容 Mimir API
  secureJsonData:
    httpHeaderName1: 'Authorization'
    httpHeaderValue1: 'Bearer xxxxxxxxxxxxx'  # 用 Service Account token

9.3 連 Cloud Monitoring

# 內建 datasource 類型
- name: Stackdriver
  type: stackdriver
  jsonData:
    authenticationType: gce       # 用 GCE metadata server
    defaultProject: my-gcp-project

9.4 連 Cloud Logging

Grafana 也支援 Cloud Logging datasource（同 Stackdriver type）：

- name: GCP-Logging
  type: googlecloud-logging-datasource
  jsonData:
    authenticationType: gce
    defaultProject: my-gcp-project

9.5 完整 GCP Grafana Stack

Grafana on GKE
├── Datasource: GMP (Prometheus, metric)
├── Datasource: Cloud Monitoring (Stackdriver, GCP service metric)
├── Datasource: Cloud Logging (log)
├── Datasource: Cloud Trace (via plugin)
├── Datasource: Cloud Profiler (via plugin)
└── Dashboard 用全部

§ 10. 認證：Workload Identity

10.1 Workload Identity 概念

GCP 給 GKE pod 賦權的最佳做法。等同 AWS 的 IRSA / Pod Identity。 讓 K8s ServiceAccount 對應 GCP Service Account，不用 secret key。

10.2 設定流程

# 1. 啟用 Workload Identity on GKE
gcloud container clusters create my-cluster \
  --workload-pool=PROJECT.svc.id.goog

# 2. 建 GCP Service Account
gcloud iam service-accounts create alloy-gmp \
  --project=PROJECT

# 3. 賦予 GCP IAM 權限
gcloud projects add-iam-policy-binding PROJECT \
  --member="serviceAccount:alloy-gmp@PROJECT.iam.gserviceaccount.com" \
  --role="roles/monitoring.metricWriter"

# 4. 綁定 K8s SA 到 GCP SA
gcloud iam service-accounts add-iam-policy-binding \
  alloy-gmp@PROJECT.iam.gserviceaccount.com \
  --role roles/iam.workloadIdentityUser \
  --member "serviceAccount:PROJECT.svc.id.goog[observability/alloy]"

# 5. K8s ServiceAccount 加 annotation
kubectl annotate serviceaccount alloy \
  --namespace observability \
  iam.gke.io/gcp-service-account=alloy-gmp@PROJECT.iam.gserviceaccount.com

10.3 應用內使用

應用使用 Google Cloud SDK 時，自動取得 credentials：

// Go
client, err := monitoring.NewMetricClient(ctx)
// 自動用 Workload Identity 認證

// Python
from google.cloud import monitoring_v3
client = monitoring_v3.MetricServiceClient()
// 自動認證

§ 11. AWS vs GCP 觀測性對比

11.1 完整對比

面向AWSGCP
原生 metricCloudWatchCloud Monitoring
Managed PrometheusAMP（Cortex-based）GMP（Monarch-based）
Managed GrafanaAMG（GA）無（用 Grafana Cloud / 自架）
原生 traceX-RayCloud Trace
Profiling無原生（X-Ray Profiler 有限）Cloud Profiler
Log queryLogs Insights（基礎）Cloud Logging（強）
OpenTelemetryADOT原生 first-class
K8s 觀測性EKS + addonGKE 預設啟用
Cardinality 限制嚴寬
Pod identityIRSA / Pod IdentityWorkload Identity

11.2 各家強項

AWS 強的地方：

  • Managed Grafana（AMG）
  • X-Ray 與 Lambda 整合
  • 細緻的 IAM 控制

GCP 強的地方：

  • GMP（基於 Monarch，幾乎無限規模）
  • OpenTelemetry first-class
  • Cloud Profiler（業界最早的 production profiler）
  • GKE 預設觀測性啟用
  • Cloud Logging query 能力

11.3 多雲策略

跨 AWS + GCP 公司的觀測性策略：

  • 路線 1：每雲用 native（CloudWatch + Cloud Monitoring），用 Grafana 統一查
  • 路線 2：自架 LGTM 在主要雲，跨雲 push
  • 路線 3：用 Grafana Cloud（SaaS）統一收
