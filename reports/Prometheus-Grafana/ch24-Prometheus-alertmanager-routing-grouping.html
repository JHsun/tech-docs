Chapter 24 · Routing / Grouping / Inhibition

Routing、Grouping、Inhibition 深度

這三個機制是 Alertmanager 的「降噪三劍客」。寫對了，500 個 alert storm 變成 1 條精準通知； 寫錯了，要嘛漏報，要嘛把 on-call 淹死。本章把每一個語法、每一個雷點、每一個實戰範例講清楚。

§ 1. Routing Tree 語法完整解析

1.1 完整語法欄位

route:
  # ===== 路由相關 =====
  matchers:                          # 匹配條件（root 通常省略 = 全部接住）
    - severity = "critical"

  receiver:           # 必填：哪個 receiver 處理（child 可繼承父）

  continue: false                    # 預設 false：匹配後就停。true 則繼續匹配 sibling

  routes:                            # 子路由（遞迴）
    - matchers:
        - team = "platform"
      receiver: 'platform-slack'

  # ===== 分組相關 =====
  group_by:                          # 用哪些 label 分組
    - alertname
    - cluster

  group_wait: 30s                    # 新 group 的等待時間
  group_interval: 5m                 # 同 group 後續通知最小間隔
  repeat_interval: 4h                # 重複通知間隔

  # ===== 時間靜默 =====
  mute_time_intervals:               # 在這些區間不通知（仍記錄）
    - business_hours

  active_time_intervals:             # 只在這些區間通知（v0.24+）
    - on_call_window

1.2 預設值繼承

子 route 沒設的欄位會繼承父 route。實際運作很微妙：

欄位是否繼承
receiver✅ 繼承
group_by✅ 繼承
group_wait / group_interval / repeat_interval✅ 繼承
mute_time_intervals / active_time_intervals❌ 不繼承（需各自指定）
matchers❌ 各 child 獨立
continue❌ 各 child 獨立

1.3 簡單 vs 完整範例

最簡：

route:
  receiver: 'default'
receivers:
  - name: 'default'
    slack_configs:
      - channel: '#alerts'

實戰：

route:
  receiver: 'default-slack'
  group_by: ['alertname', 'cluster']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 12h

  routes:
    # 1. Critical 全部走 PagerDuty
    - matchers:
        - severity = "critical"
      receiver: 'pagerduty'
      group_wait: 10s            # critical 等少一點
      repeat_interval: 30m       # critical 提醒頻繁一點

      routes:
        # 1.1 Critical + Database team 多送 DB 專屬 channel
        - matchers:
            - team = "database"
          receiver: 'db-pagerduty'
          continue: true         # ← 同時也走父 route 的 pagerduty

    # 2. Warning + Frontend team
    - matchers:
        - severity = "warning"
        - team = "frontend"
      receiver: 'frontend-slack'

    # 3. Staging 環境統一處理
    - matchers:
        - environment = "staging"
      receiver: 'staging-slack'
      repeat_interval: 24h       # staging 別常吵

    # 4. 業務 alert 走專屬 channel
    - matchers:
        - category = "business"
      receiver: 'business-team-email'

§ 2. Matcher 語法（UTF-8 與 Classic）

2.1 UTF-8 Matchers（推薦，v0.27+）

matchers:
  - severity = "critical"             # 相等
  - team != "staging"                 # 不相等
  - service =~ "api|web|db"           # regex 匹配
  - alertname !~ ".*Test.*"           # regex 不匹配

多個 matcher 是 AND 關係（全部符合才算匹配）。

2.2 Classic Matchers（仍支援，但建議遷移）

# 等價的 classic 語法
match:                                # 等於匹配
  severity: critical

match_re:                             # regex 匹配
  service: api|web|db
⚠️ 不能混用

同一 route 不能同時用 matchers 與 match/match_re。 建議全部改為 matchers 統一風格。

2.3 Regex 整體錨定

跟 PromQL 一樣，=~"foo" 等同 =~"^foo$"。要部分匹配寫 =~".*foo.*"。

2.4 含特殊字元的值

# 含空白、特殊字元 → 用引號
matchers:
  - service = "my service"
  - description =~ ".*timeout.*"
  - team = "platform/sre"

§ 3. continue 機制深入

這是 routing 中最被誤解的功能。預設 continue: false：找到第一個匹配的 route 就停。 設 continue: true：繼續嘗試後續 sibling，可能匹配多個。

3.1 經典用例：「重要 alert 同時通知多 channel」

route:
  receiver: 'default'

  routes:
    # 第一條：critical alert 也同時送 PagerDuty
    - matchers:
        - severity = "critical"
      receiver: 'pagerduty'
      continue: true             # ← 重點：繼續匹配下一個

    # 第二條：所有 alert 都記錄到 Slack
    - matchers:
        - severity =~ "critical|warning"
      receiver: 'general-slack'
      continue: true

    # 第三條：critical 同時開 Jira ticket
    - matchers:
        - severity = "critical"
      receiver: 'jira-webhook'

一個 critical alert 進來：先匹配第一條 → 送 PagerDuty + continue → 匹配第二條 → 送 Slack + continue → 匹配第三條 → 開 Jira ticket。三個 receiver 都會收到。

3.2 Routing 流程圖

3.3 Catch-all 機制

若 alert 沒匹配任何 child route，會用 root 的 receiver（catch-all）。 所以一定要設 root receiver，否則沒匹配的 alert 會「悄悄消失」。

§ 4. Grouping 三個層次

4.1 為什麼分組這麼重要

沒分組時的災難情境：

  • Cluster 網路掛了 → 200 個 service × 10 個 instance = 2000 條 alert
  • 沒分組 → on-call 收到 2000 條 Slack 訊息
  • 有分組（按 alertname）→ 收到 1 條訊息列出 2000 個受影響 instance

4.2 分組的三個維度

維度意義典型 group_by
按 alert 類型 同類型 alert 合併 ['alertname']
按範圍 + 類型 同類型且同 cluster 合併 ['alertname', 'cluster']
按服務 同 service 的所有 alert 合併 ['service']

4.3 分組的時序

Group 第一次有 alert 進來時：

  1. group_wait（預設 30s）：等待相關 alert 累積
  2. 時間到 → 發送通知（含目前 group 中所有 alert）
  3. 後續若有新 alert加入 group：
    • 下個 group_interval（預設 5m）到 → 發送更新通知
  4. 若 group 中沒新變化但持續 firing：
    • 每 repeat_interval（預設 4h）→ 發送 reminder 通知
  5. 所有 alert 都 resolved → 發送 resolved 通知（若 receiver 啟用）

§ 5. group_by: '...' 特殊值

除了一般 label 列表，group_by 有兩個特殊值：

5.1 group_by: ['...'] — 不分組

# 每個 alert 都獨立通知（不分組）
route:
  group_by: ['...']
  receiver: 'pagerduty'

適合 critical alert：每個都該獨立 page，不要被 group_wait 延遲。

5.2 沒設 group_by — 全部當一個 group

# 所有 alert 合成「一個」group
route:
  receiver: 'one-slack-message-per-poll'

很少用，會把所有東西塞進一條訊息，難以閱讀。

5.3 對照表

設定分組行為
group_by: ['alertname']同 alertname 合併（典型）
group_by: ['...']每個 alert 獨立（不分組）
未設定所有 alert 當一組（不建議）

§ 6. Inhibition 完整語法

6.1 Inhibition 基本概念

當 source alert（高優先級）正在 firing，抑制 target alert（低優先級或症狀）。

6.2 完整語法

inhibit_rules:
  - source_matchers:
      - severity = "critical"
    target_matchers:
      - severity = "warning"
    equal:                          # ← 關鍵欄位
      - alertname
      - cluster

6.3 解讀規則

  1. 當有 alert 符合所有 source_matchers 且正在 firing → 它是「source」
  2. 對於符合 target_matchers 的 alert：
  3. 檢查 equal 列出的 label，source 與 target 必須在這些 label 上有相同值
  4. 若條件全部成立 → 不發送 target 的通知

6.4 視覺化範例

規則：「ClusterDown critical 抑制同 cluster 的所有 warning」

inhibit_rules:
  - source_matchers:
      - alertname = "ClusterDown"
      - severity = "critical"
    target_matchers:
      - severity = "warning"
    equal:
      - cluster

運作：

正在 firing:
  Source: ClusterDown{cluster="tw", severity="critical"}
  Target1: PodDown{cluster="tw", severity="warning", pod="web-1"}    ← 抑制
  Target2: HighLatency{cluster="tw", severity="warning"}             ← 抑制
  Target3: PodDown{cluster="jp", severity="warning"}                 ← 不抑制 (cluster 不同)
  Target4: PodDown{cluster="tw", severity="critical"}                ← 不抑制 (severity 不符)

§ 7. equal 欄位的關鍵作用

🚨 最常見的災難性誤用

若 equal 列出的 label 在 source 與 target 都不存在， Alertmanager 會視為「相等」（兩邊都 empty == empty），抑制規則會無條件套用。
意思：你以為「只抑制同 cluster」，但設錯讓所有 warning 都被 critical 抑制了。

7.1 錯誤範例

# ❌ 錯：cluster label 在某些 alert 上沒有
inhibit_rules:
  - source_matchers:
      - severity = "critical"
    target_matchers:
      - severity = "warning"
    equal: ['cluster']

# 假設有兩個 alert 都缺 cluster label：
# Source: HighCPU{instance="api-1", severity="critical"}    ← 沒 cluster
# Target: HighMemory{instance="db-1", severity="warning"}   ← 沒 cluster
# 兩者 cluster 都「空」，視為相等 → HighMemory 被抑制
# 但這兩個 alert 完全無關！

7.2 正確做法：強制要 label

# ✅ 對：在 alert label 中強制有 cluster
# 在 Prometheus alerting rule 中：
- alert: HighCPU
  expr: ...
  labels:
    cluster: "{{ $externalLabels.cluster }}"
    severity: critical

# 確保所有 alert 都有 cluster label，equal 才安全

7.3 用 amtool 測試 inhibition

amtool 沒有直接的 inhibition 模擬，但可以用 API 與時序測試：

# 1. POST 一個 source alert
curl -XPOST http://localhost:9093/api/v2/alerts -d '[{
  "labels": {"alertname":"ClusterDown","severity":"critical","cluster":"tw"},
  "startsAt": "2026-05-02T10:00:00Z"
}]'

# 2. POST 一個 target alert
curl -XPOST http://localhost:9093/api/v2/alerts -d '[{
  "labels": {"alertname":"PodDown","severity":"warning","cluster":"tw","pod":"web-1"},
  "startsAt": "2026-05-02T10:00:00Z"
}]'

# 3. 檢查通知（target 應該不被通知）
# 4. 用 /api/v2/alerts 看 alert 狀態
curl http://localhost:9093/api/v2/alerts

§ 8. Time Intervals（時間區間靜默）

v0.22+ 引入 time_intervals，讓您定義「業務時間」、「on-call 時段」等， 用於 mute_time_intervals（這時段內不通知）或 active_time_intervals（只在這時段通知）。

8.1 完整語法

time_intervals:
  - name: business_hours
    time_intervals:
      - times:
          - start_time: 09:00
            end_time: 18:00
        weekdays: ['monday:friday']     # 週一到週五
        location: 'Asia/Taipei'

  - name: weekends
    time_intervals:
      - weekdays: ['saturday', 'sunday']
        location: 'Asia/Taipei'

  - name: maintenance_window
    time_intervals:
      - times:
          - start_time: 02:00
            end_time: 04:00
        weekdays: ['sunday']
        location: 'UTC'

  - name: holiday_2026
    time_intervals:
      - days_of_month: ['1']
        months: ['1']                    # 1 月 1 日
        location: 'Asia/Taipei'

8.2 在 route 中使用

route:
  routes:
    # 非業務時間靜默 info 通知
    - matchers:
        - severity = "info"
      receiver: 'info-slack'
      mute_time_intervals:
        - weekends
        - holiday_2026

    # 維護視窗靜默所有 disk 相關 alert
    - matchers:
        - alertname =~ ".*Disk.*"
      receiver: 'platform-slack'
      mute_time_intervals:
        - maintenance_window

    # 只在業務時間通知 cost-related alert
    - matchers:
        - category = "cost"
      receiver: 'finance-email'
      active_time_intervals:
        - business_hours

8.3 完整可用欄位

欄位說明
times時間段（HH:MM - HH:MM）
weekdays週幾（monday/tuesday.../sunday，可用範圍）
days_of_month每月幾號（1-31，可用範圍）
months月份（january/.../december 或 1-12）
years年份（如 2026）
location時區（IANA tz database name）

§ 9. 實戰範例集

9.1 完整生產級設定

route:
  receiver: 'default'
  group_by: ['alertname', 'cluster', 'service']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 12h

  routes:
    # === 1. Critical 全部走 PagerDuty + 同時送 #incidents ===
    - matchers:
        - severity = "critical"
      receiver: 'pagerduty'
      group_wait: 10s
      repeat_interval: 1h
      continue: true

    - matchers:
        - severity = "critical"
      receiver: 'incidents-channel'
      continue: true

    # === 2. 各團隊的 warning ===
    - matchers:
        - severity = "warning"
        - team = "platform"
      receiver: 'platform-slack'

    - matchers:
        - severity = "warning"
        - team = "database"
      receiver: 'dba-slack'

    - matchers:
        - severity = "warning"
        - team = "frontend"
      receiver: 'frontend-slack'

    # === 3. Staging / dev 環境統一 ===
    - matchers:
        - environment =~ "staging|dev"
      receiver: 'noisy-slack'
      repeat_interval: 24h
      mute_time_intervals:
        - weekends                  # 假日 staging 環境不通知

    # === 4. 業務 alert 給 PM/Business ===
    - matchers:
        - category = "business"
      receiver: 'business-email'
      active_time_intervals:
        - business_hours

inhibit_rules:
  # === 抑制 1：critical 同名 alert 抑制 warning ===
  - source_matchers:
      - severity = "critical"
    target_matchers:
      - severity = "warning"
    equal: ['alertname', 'cluster', 'service']

  # === 抑制 2：cluster 整個掛時，抑制該 cluster 內所有 alert ===
  - source_matchers:
      - alertname = "ClusterDown"
      - severity = "critical"
    target_matchers:
      - severity =~ "warning|info"
    equal: ['cluster']

  # === 抑制 3：node 掛時，抑制該 node 上的 pod 級 alert ===
  - source_matchers:
      - alertname = "NodeDown"
    target_matchers:
      - alertname =~ "Pod.*"
    equal: ['node']

  # === 抑制 4：DB 主庫掛時，抑制 replica lag 等下游 alert ===
  - source_matchers:
      - alertname = "PostgreSQLPrimaryDown"
    target_matchers:
      - alertname =~ "PostgreSQLReplica.*"
    equal: ['cluster']

time_intervals:
  - name: business_hours
    time_intervals:
      - times:
          - start_time: 09:00
            end_time: 18:00
        weekdays: ['monday:friday']
        location: 'Asia/Taipei'

  - name: weekends
    time_intervals:
      - weekdays: ['saturday', 'sunday']
        location: 'Asia/Taipei'

9.2 K8s 場景的標準抑制集

inhibit_rules:
  # 整個 cluster 掛 → 抑制所有 cluster 內 alert
  - source_matchers:
      - alertname = "KubernetesAPIDown"
    target_matchers:
      - severity =~ "warning|info"
    equal: ['cluster']

  # Node 掛 → 抑制該 node 上的 pod
  - source_matchers:
      - alertname = "KubeNodeNotReady"
    target_matchers:
      - alertname =~ "KubePod.*"
    equal: ['cluster', 'node']

  # Namespace 整體有問題 → 抑制單個 pod
  - source_matchers:
      - alertname = "KubeNamespaceQuotaExceeded"
    target_matchers:
      - alertname =~ "KubePodNotReady"
    equal: ['cluster', 'namespace']

  # PV 滿 → 抑制使用該 PV 的 pod 的 disk-related alert
  - source_matchers:
      - alertname = "KubePersistentVolumeFillingUp"
    target_matchers:
      - alertname =~ ".*DiskSpace.*"
    equal: ['cluster', 'persistentvolumeclaim']

§ 10. 常見陷阱

⚠️ 陷阱 1：忘記 root receiver

沒匹配任何 child route 的 alert 會用 root receiver。
若 root receiver 設成 'null' 或忘記設 → alert 悄悄消失。

⚠️ 陷阱 2：continue 用太多

每個 route 都 continue: true 會讓同個 alert 被處理多次， 每個 receiver 都收到 → 可能違反「不重複通知」的初衷。
只在「同時通知多 channel」這類明確需求才用 continue。

⚠️ 陷阱 3：group_by 太細

若 group_by 包含太多 label（例如 ['alertname', 'instance', 'pod']）， 每個 instance 都各自一組 = 跟沒分組差不多。
建議：['alertname', 'cluster'] 或 ['alertname', 'service'] 是好平衡。

⚠️ 陷阱 4：Inhibition 的 equal 用了不存在的 label

如前述，會無條件套用抑制 → 災難性結果。
建議：每次寫 inhibition 都先用 amtool 或測試環境驗證。

⚠️ 陷阱 5：mute_time_intervals 不會繼承

您在父 route 設了 mute_time_intervals: [weekends]，child 不會繼承。 若您要全 routes 在週末靜默，每個 route 都要自己設一次。

⚠️ 陷阱 6：時區忘記設 location

預設是 UTC。台灣 09:00 業務時段 = UTC 01:00，沒設 location 會出大錯。
永遠加 location: 'Asia/Taipei'。

✅ 本章重點摘要

30 秒回顧
  • Routing 樹形結構：root → child → grandchild，預設值大部分繼承（但 mute_time_intervals 不繼承）
  • UTF-8 matchers 推薦：= / != / =~ / !~，跟 PromQL 一致
  • continue: true 讓 alert 被多 receiver 處理，用於「同時送多 channel」
  • Group: group_by: ['alertname', 'cluster'] 是好平衡；['...'] 不分組（每 alert 獨立）
  • 三個時間：group_wait(30s) < group_interval(5m) << repeat_interval(4h)
  • Inhibition：equal 列出的 label 必須兩邊都存在，否則災難
  • 典型 inhibition：critical 抑制 warning、cluster down 抑制內部 alert、node down 抑制 pod alert
  • Time intervals：v0.22+ 支援 mute_time_intervals（這時段不通知）/ active_time_intervals（只在這時段通知）
  • 必設 root catch-all receiver，否則沒匹配的 alert 消失

📝 練習題

  1. 請寫一個 routing tree：critical 走 PagerDuty + Slack #incidents（同時兩個），warning 按 team 分流到對應 Slack channel，staging 環境週末靜默。
  2. 下面的 inhibition 為什麼有問題？怎麼修？ 
    inhibit_rules:
  - source_matchers:
      - severity = "critical"
    target_matchers:
      - severity = "warning"
    equal: []
  3. 解釋 group_by: ['alertname'] 與 group_by: ['...'] 的差別。各自的使用情境？
  4. 用 amtool 測試：給定下面設定，severity=critical, team=database, alertname=PostgreSQLDown 會匹配哪些 receiver？ 
    route:
  receiver: 'default'
  routes:
    - matchers: [severity = "critical"]
      receiver: 'pagerduty'
      continue: true
    - matchers: [team = "database"]
      receiver: 'dba-slack'
  5. 請寫一個 time interval：「台灣時間每週一到週五 09:00-18:00 之外，包含週末，都靜默 cost-related alert」。
