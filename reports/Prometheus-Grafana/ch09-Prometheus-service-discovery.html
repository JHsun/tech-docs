Chapter 09 · Service Discovery

Service Discovery 完整指南

在動態環境（K8s、雲端、auto-scaling）中，手動列舉 targets 是不可能的任務。 Prometheus 內建超過 20 種 Service Discovery（SD）機制，自動維護要被抓取的目標清單。 本章涵蓋所有主流 SD 類型、設定範例、以及最重要的 meta labels 系統。

§ 1. 為什麼需要 Service Discovery

在傳統機房裡，伺服器數量穩定、IP 固定，手動維護 static_configs 還算可行。 但現代環境完全不同：

  • K8s：Pod 隨時被重新排程，IP 不固定
  • Auto Scaling：實例數量隨負載動態變化
  • 容器編排：Docker、Nomad、ECS 等容器隨時誕生與消失
  • 多地域部署：服務分散在多個 AZ、region、cloud provider

Service Discovery 解決的核心問題：「我該抓誰？」這份清單必須能即時、自動更新。

§ 3. Static SD：最簡單的起點

適合：少量固定 host、本地開發、SD 結果再 fan-out。

scrape_configs:
  - job_name: 'fixed-services'
    static_configs:
      - targets:
          - 'web1.example.com:9100'
          - 'web2.example.com:9100'
        labels:
          datacenter: 'tw-1'
          role: 'frontend'

      # 多個 group 可有不同 labels
      - targets:
          - 'db1.example.com:9100'
          - 'db2.example.com:9100'
        labels:
          datacenter: 'tw-1'
          role: 'database'

§ 4. File SD：強大的通用方案

適合：把 target 維護外包給其他工具（Ansible、CMDB、自訂 script）。

4.1 設定方式

scrape_configs:
  - job_name: 'file-sd'
    file_sd_configs:
      - files:
          - '/etc/prometheus/targets/*.json'
          - '/etc/prometheus/targets/*.yml'
        refresh_interval: 30s

4.2 JSON 檔案格式

[
  {
    "targets": ["10.0.0.1:9100", "10.0.0.2:9100"],
    "labels": {
      "env": "production",
      "datacenter": "dc1",
      "service": "node-exporter"
    }
  },
  {
    "targets": ["10.0.1.1:8080"],
    "labels": {
      "env": "production",
      "service": "api-server"
    }
  }
]

4.3 YAML 格式（同樣支援）

- targets:
    - 10.0.0.1:9100
    - 10.0.0.2:9100
  labels:
    env: production
    datacenter: dc1
    service: node-exporter
✅ File SD 是 inotify-based 的

Prometheus 透過 Linux 的 inotify 即時偵測檔案變更， 所以更新可以是近即時的（不必等 refresh_interval）。 refresh_interval 只是「保險」，避免 inotify 漏掉訊號。

§ 5. HTTP SD：自訂的網路化方案

當您的 target 來源不是檔案，而是某個 API 或服務時，HTTP SD 是替代 file SD 的網路化版本。 Prometheus 會定期 GET 您的 endpoint，期望得到相同格式的 JSON 回應。

5.1 Prometheus 端設定

scrape_configs:
  - job_name: 'http-sd'
    http_sd_configs:
      - url: 'http://my-cmdb.internal:8080/prometheus/targets'
        refresh_interval: 60s
        # 認證
        authorization:
          type: Bearer
          credentials_file: /etc/prometheus/secrets/sd-token
        # 或 basic_auth / oauth2 / tls_config

5.2 您要實作的端點規格

  • HTTP GET
  • 回應 200
  • Content-Type: application/json
  • UTF-8 編碼
  • 內容是 JSON array，格式同 file_sd
  • 無 targets 時回 200 加空陣列 []

5.3 實作範例（Python Flask）

from flask import Flask, jsonify

app = Flask(__name__)

@app.route('/prometheus/targets')
def targets():
    # 在實際情境中，這裡可能查 CMDB、雲端 API、或內部資料庫
    services = fetch_from_cmdb()
    response = []
    for svc in services:
        response.append({
            "targets": [f"{svc['ip']}:{svc['port']}"],
            "labels": {
                "service": svc['name'],
                "env": svc['env'],
                "team": svc['team']
            }
        })
    return jsonify(response)

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=8080)

§ 6. DNS SD：A、AAAA、SRV 記錄

6.1 SRV 記錄（最常用）

SRV 記錄包含「服務名稱 + port + 主機」，最適合 Prometheus 的需求：

scrape_configs:
  - job_name: 'dns-srv'
    dns_sd_configs:
      - names:
          - '_prometheus._tcp.api.example.com'
          - '_prometheus._tcp.web.example.com'
        type: SRV
        refresh_interval: 30s

對應的 DNS 記錄：

_prometheus._tcp.api.example.com. 60 IN SRV 0 0 9100 api-1.example.com.
_prometheus._tcp.api.example.com. 60 IN SRV 0 0 9100 api-2.example.com.

6.2 A / AAAA 記錄

A / AAAA 不含 port，要在 SD config 裡指定：

scrape_configs:
  - job_name: 'dns-a'
    dns_sd_configs:
      - names:
          - 'api.example.com'   # 會解析所有 A 記錄
        type: A
        port: 9090
        refresh_interval: 30s

6.3 DNS SD 的優缺點

優點缺點
幾乎所有環境都有 DNS不能傳遞 metadata（無法附加 label）
標準化、跨平台受 TTL 影響更新延遲
可整合 Consul DNS、K8s DNSSRV 記錄的維護需要額外工具

§ 7. Kubernetes SD：6 個 role 詳解

K8s SD 是雲原生環境最重要的 SD。Prometheus 透過 K8s API watch 各種資源：

Role抓取對象典型用途
nodeK8s nodes（kubelet）Node 級別指標、kubelet 自身
serviceK8s Services每個 service 抓一次（與 endpoints 不同）
podK8s Pods 的每個容器 port抓取所有應用 pod 的 /metrics
endpointsService 對應的 endpoints（後端 pod IP）抓 service-managed 服務（最常用）
endpointsliceEndpointSlice（K8s 1.21+ 推薦）endpoints 的更高效替代
ingressK8s Ingresses監控 ingress 健康（搭配 blackbox）

7.1 抓取所有有 annotation 的 Pod

業界最常見的模式 — 「給 pod 加上 prometheus.io/scrape: "true" 就會被抓」：

scrape_configs:
  - job_name: 'kubernetes-pods'
    kubernetes_sd_configs:
      - role: pod

    relabel_configs:
      # 只保留有 annotation 的
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: true

      # 自訂 metrics_path（預設 /metrics）
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_path]
        action: replace
        target_label: __metrics_path__
        regex: (.+)

      # 自訂 port（覆蓋 pod 預設）
      - source_labels: [__address__, __meta_kubernetes_pod_annotation_prometheus_io_port]
        action: replace
        regex: ([^:]+)(?::\d+)?;(\d+)
        replacement: $1:$2
        target_label: __address__

      # 把 K8s namespace 變成 Prometheus label
      - source_labels: [__meta_kubernetes_namespace]
        action: replace
        target_label: kubernetes_namespace

      # 把 pod 名稱變成 label
      - source_labels: [__meta_kubernetes_pod_name]
        action: replace
        target_label: kubernetes_pod_name

      # 從 pod label "app" 提取
      - source_labels: [__meta_kubernetes_pod_label_app]
        action: replace
        target_label: app

7.2 抓取特定 namespace 的服務 endpoint

scrape_configs:
  - job_name: 'monitoring-services'
    kubernetes_sd_configs:
      - role: endpoints
        namespaces:
          names:
            - 'monitoring'
            - 'platform'

    relabel_configs:
      # 只保留 service annotation 標記過的
      - source_labels: [__meta_kubernetes_service_annotation_prometheus_io_scrape]
        action: keep
        regex: true

      # 提取 service 名稱
      - source_labels: [__meta_kubernetes_service_name]
        target_label: service
      - source_labels: [__meta_kubernetes_namespace]
        target_label: namespace

7.4 RBAC 權限要求

Prometheus 的 ServiceAccount 至少需要這些權限：

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: prometheus
rules:
  - apiGroups: [""]
    resources: ["nodes", "nodes/proxy", "nodes/metrics", "services",
                "endpoints", "pods"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["discovery.k8s.io"]
    resources: ["endpointslices"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["networking.k8s.io"]
    resources: ["ingresses"]
    verbs: ["get", "list", "watch"]
  - nonResourceURLs: ["/metrics"]
    verbs: ["get"]
ℹ️ Prometheus Operator 簡化做法

若您用 Prometheus Operator 或 kube-prometheus-stack， 不需要手寫這麼複雜的 relabel_configs。改用 ServiceMonitor / PodMonitor CRD， Operator 會自動為您產生對應的 scrape config。Ch 54 會詳細展示。

§ 8. Consul SD

HashiCorp Consul 是業界知名的 service registry。整合方式：

scrape_configs:
  - job_name: 'consul-services'
    consul_sd_configs:
      - server: 'consul.service.consul:8500'
        # 過濾要哪些 service（空 = 全部）
        services: ['web', 'api', 'db']
        # 或用 tags 過濾
        tags: ['monitoring']
        # 從哪個 datacenter
        datacenter: 'tw-east-1'
        refresh_interval: 30s

    relabel_configs:
      - source_labels: [__meta_consul_service]
        target_label: service
      - source_labels: [__meta_consul_dc]
        target_label: datacenter
      - source_labels: [__meta_consul_tags]
        regex: '.*,monitoring,.*'
        action: keep

§ 9. 雲廠商 SD：AWS、GCP、Azure

9.1 AWS（v3.x 統一格式）

Prometheus 3.x 把多種 AWS SD 統一到 aws_sd_configs，用 role 區分：

scrape_configs:
  - job_name: 'aws-ec2'
    aws_sd_configs:
      - role: ec2          # 或 ecs / elasticache / lightsail / msk / rds
        region: 'ap-northeast-1'
        refresh_interval: 60s

    relabel_configs:
      - source_labels: [__meta_ec2_tag_Environment]
        regex: 'production'
        action: keep
      - source_labels: [__meta_ec2_tag_Service]
        target_label: service
      - source_labels: [__meta_ec2_availability_zone]
        target_label: az
      - source_labels: [__meta_ec2_private_ip]
        target_label: __address__
        replacement: '$1:9100'

需要的 IAM 權限：

  • ec2:DescribeInstances（EC2 SD）
  • ecs:ListClusters + ecs:DescribeTasks 等（ECS SD）
  • kafka:ListClustersV2 + kafka:GetBootstrapBrokers（MSK SD）

9.2 GCP（GCE）

scrape_configs:
  - job_name: 'gce-instances'
    gce_sd_configs:
      - project: 'my-gcp-project'
        zone: 'asia-east1-a'
        refresh_interval: 60s
        port: 9100

    relabel_configs:
      - source_labels: [__meta_gce_label_environment]
        regex: 'production'
        action: keep
      - source_labels: [__meta_gce_zone]
        regex: '.+/zones/(.+)'
        target_label: zone

9.3 Azure

scrape_configs:
  - job_name: 'azure-vms'
    azure_sd_configs:
      - subscription_id: 'xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'
        # 三種認證方式之一
        # OAuth：
        tenant_id: 'xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'
        client_id: 'xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'
        client_secret_file: /etc/prometheus/secrets/azure-secret
        # 或 Managed Identity（推薦在 Azure 上跑時）
        # authentication_method: ManagedIdentity
        # 或 Workload Identity（v3.11+）
        port: 9100
        refresh_interval: 60s

§ 11. Meta Labels 與 Relabeling 銜接

所有 SD 機制有個共同特徵：他們都會產生以 __meta_ 開頭的 meta labels。 這些 label 是 SD 給 Prometheus 的「附加資料」，例如：「這個 target 來自哪個 namespace」、「pod 上有什麼 label」、 「EC2 instance 的 tag 是什麼」。

11.1 Meta Labels 的關鍵特性

  • 所有 __ 開頭的 label 都是內部 label，scrape 完成後會自動丟掉
  • 你必須透過 relabel_configs 把要保留的 meta label 「升級」為正式 label
  • 不升級的 meta label 在最終 series 上看不到

11.2 SD 階段流程

graph TB SD[Service Discovery] M[Meta Labels
__meta_kubernetes_pod_name
__meta_kubernetes_namespace
__address__
...] RL[relabel_configs
關鍵階段!] AT[Approved Target
+ Promoted Labels] SC[Scrape /metrics] MR[metric_relabel_configs
進一步過濾 metric 層級] TS[Final Time Series] SD --> M M --> RL RL --> AT AT --> SC SC --> MR MR --> TS

注意：

  • relabel_configs 在 scrape 之前，操作的是「target 層級」的 label（含 meta）
  • metric_relabel_configs 在 scrape 之後，操作的是「每個 sample 的 label」

§ 12. 多 SD 混用與最佳實踐

12.1 一個 job 可以混用多種 SD

scrape_configs:
  - job_name: 'mixed-targets'
    static_configs:
      - targets: ['legacy-server:9100']
    file_sd_configs:
      - files: ['/etc/prometheus/targets/*.json']
    dns_sd_configs:
      - names: ['_prometheus._tcp.api.svc.cluster.local']
        type: SRV

12.2 為不同環境用不同 SD

環境建議 SD
本地開發 / Docker ComposeStatic、Docker SD
VM 機房（少量機器）Static、File SD
VM 機房（中大規模）Consul、File SD（CMDB 同步）
K8sKubernetes SD（搭配 Operator）
AWSaws_sd_configs（EC2/ECS）+ Kubernetes SD（EKS）
多雲混合每個雲一個 SD + 用 external_labels 區分
無法整合的舊系統HTTP SD（自寫 endpoint）

12.3 最佳實踐

✅ Best Practices
  1. 一個 job 一個邏輯目的：不要把 frontend、backend、database 全塞同一個 job_name
  2. Meta label 該留就留：Pod 名稱、namespace、environment、AZ 都應該升格成 label
  3. Refresh interval 不要太短：K8s SD 預設值即可，太短增加 API server 負擔
  4. Cloud SD 用 instance metadata role：避免硬編 access key
  5. 監控 SD 自身：Prometheus 有 prometheus_sd_* metrics 可看 SD 健康狀態
  6. Static + 其他 SD 並用沒關係：legacy 系統用 static，現代用 dynamic
  7. K8s 環境優先用 Operator：手寫 K8s SD 容易出錯

12.4 監控 SD 自身的 metrics

# SD 提供的 targets 數量
prometheus_sd_discovered_targets

# 每個 SD 機制的失敗次數
prometheus_sd_consul_rpc_failures_total
prometheus_sd_kubernetes_events_total{role="pod"}

# SD 重新整理延遲
prometheus_sd_dns_lookups_total
prometheus_sd_file_read_errors_total

✅ 本章重點摘要

30 秒回顧
  • Prometheus 內建 22+ 種 SD 機制，分通用、容器、雲廠商、Service Registry 等類別
  • Static：固定列表；File：inotify 即時更新；HTTP：自訂網路化
  • DNS SD 支援 A / AAAA / SRV，但無法傳 metadata
  • Kubernetes SD 6 個 role：node / service / pod / endpoints / endpointslice / ingress；最常用是 endpoints 和 pod
  • Cloud SD：AWS 在 v3 統一為 aws_sd_configs + role；GCP / Azure / DigitalOcean / Hetzner 等都有
  • 所有 SD 產生 __meta_* labels，需用 relabel_configs 升格才會出現在最終 series
  • relabel_configs 在 scrape 前處理 target；metric_relabel_configs 在 scrape 後處理 sample
  • K8s 環境推薦用 Prometheus Operator，避免手寫複雜 relabel_configs

📝 練習題

  1. 請寫一段 file_sd 設定，搭配 JSON 檔案，定義 3 組 targets，分別是 web、db、cache，每組各 2 台 server。
  2. 實作一個最小的 HTTP SD endpoint（任何語言），固定回傳 2 個 targets。讓 Prometheus 連上並抓取。
  3. 在 K8s 環境中，您要抓取所有 monitoring namespace 下、有 label app=blackbox 的 pod。請寫出對應的 scrape_config（含 relabel）。
  4. 解釋為什麼下面這條 K8s relabel 會失敗（找出 typo）： 
    - source_labels: [__meta_kubernetes_pod_label_app]
  target_lable: app
  action: replace
  5. 查看您環境的 prometheus_sd_discovered_targets metric，列出每個 SD job 目前發現多少 targets。
