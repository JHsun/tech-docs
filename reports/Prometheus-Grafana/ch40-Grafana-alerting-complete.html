Grafana Alerting：Unified Alerting 完整教學

Grafana 8+ 的 Unified Alerting 是一套獨立於 Prometheus + Alertmanager 之外的 alerting 系統。 優勢是跨多 datasource（metrics + logs + SQL 都能 alert）。本章涵蓋四大概念： Alert Rules / Contact Points / Notification Policies / Templates，並對比與 Prometheus alerting 的取捨。

§ 1. Grafana Alerting 架構

1.1 整體流程

sequenceDiagram participant SCH as Grafana Scheduler participant DS as Data Source participant POL as Notification Policy participant CP as Contact Point participant DEST as Slack · PagerDuty · Email loop 每個 evaluation interval SCH->>DS: 執行 alert query DS-->>SCH: 回傳結果 SCH->>SCH: 評估 condition alt 條件成立 SCH->>SCH: 進入 Pending state end alt for duration 達到 SCH->>SCH: 轉為 Firing SCH->>POL: 用 label 路由 POL->>CP: 送到對應 contact point CP->>DEST: 送通知 end alt 條件不再成立 SCH->>SCH: 轉為 Resolved SCH->>POL: 路由 resolved 通知 POL->>CP: 送 resolved end end

1.2 四大核心概念

概念對應 Prometheus + AM
Alert Rule= Prometheus alerting rule
Contact Point= AM receiver
Notification Policy= AM route tree
Mute Timing= AM mute_time_intervals
Silence= AM silence
Template= AM template

架構非常像（Grafana 的 alerting 內部其實 fork 了 Alertmanager）。但有一個關鍵差別：Grafana 的 alert evaluation 在 Grafana 自己跑，不在 Prometheus。

1.3 Alert State

State說明
Normal條件未成立
Pending條件成立但 for duration 未達到
Alerting (Firing)for 達到，正式送通知
Resolved條件不再成立
ErrorQuery 出錯（datasource 掛 / 語法錯）
NoDataQuery 沒回傳資料

§ 2. Grafana-managed vs Datasource-managed

2.1 兩種 alert rule

類型儲存位置評估位置
Grafana-managed Grafana DB Grafana 自己跑
Datasource-managed（Prometheus / Mimir） Prometheus rule file / Mimir Ruler Prometheus 或 Mimir 跑

2.2 Grafana-managed 優勢

  • 多 datasource：condition 可以是 metric + log + SQL 組合
  • UI 友善：所見即所得
  • Recording rules（v12+）：在 Grafana 寫 recording rule

2.3 Datasource-managed 優勢

  • 不依賴 Grafana：Grafana 掛了 alert 仍 fire
  • 標準化：rule file 可 Git 版本控管
  • 規模化：Mimir Ruler 可分散式評估

2.4 推薦選擇

✅ 選擇原則
  • Pure metric-based 告警：用 Prometheus / Mimir rule（不依賴 Grafana）
  • 跨 datasource 告警（metric + log）：只能 Grafana-managed
  • 視覺化 + alert 一站式建立：Grafana-managed 方便
  • 大規模多租戶：用 Mimir Ruler

§ 3. Alert Rules 完整建立

3.1 從 UI 建立

左側 sidebar → Alerting → Alert rules → + New alert rule

3.2 Step 1: Define query and alert condition

Rule type: Grafana-managed alert rule

Query A: (datasource = Prometheus)
  expr: rate(http_requests_total{status=~"5.."}[5m])
        / rate(http_requests_total[5m])

Expression B: (Reduce / Threshold / Math)
  Type: Reduce
  Function: Last
  Input: A

Expression C: (Threshold)
  Input: B
  Condition: IS ABOVE 0.05    ← 5% error rate
  Set as alert condition: ✅

3.3 Expression Types

Type用途
Math四則運算（A * 100 / B）
Reduce把 series 縮為單值（Last / Mean / Max / Min / Sum / Count）
Resample對齊不同 datasource 的時間點
Threshold條件判斷（is above / below / within / outside）
Classic condition舊版混合 reduce + threshold（v11- legacy）

3.4 Step 2: Set alert evaluation

Folder: Production-Alerts
Evaluation group: api-alerts
Evaluation interval: 1m              ← 多久評估一次
Pending period: 5m                   ← 條件成立後等多久才算 firing

3.5 Pending period 的意義

類似 Prometheus 的 for：

  • 0s = 條件一成立就 firing（容易誤報）
  • 5m = 條件持續 5 分鐘才 firing（避免閃爆）
  • 建議：critical alert 設 1-5m，warning 設 5-15m

3.6 Step 3: Configure labels and notifications

Labels:
  severity: critical
  team: platform
  service: api

Annotations:
  summary: "API error rate is {{ $values.B | printf "%.2f%%" }}"
  description: "P0 issue, page on-call immediately"
  runbook_url: "https://wiki/api-errors"
  dashboard_url: "https://grafana/d/abc"

3.7 Step 4: Notifications

兩種選擇：

  • Use notification policies（推薦）：依 label 自動路由
  • Choose contact point：直接指定 contact point（簡單但不彈性）

§ 4. Multi-Dimensional Alerts

4.1 一個 rule 多個 alert instance

Grafana Alerting 沿襲 Prometheus 的多維度設計： 

Query: rate(http_5xx[5m]) by (service)
       → 回傳 [api: 0.08, web: 0.02, payment: 0.06]

Threshold: > 0.05

→ 產生 3 個 alert instance：
  - {service="api"}     → FIRING (0.08)
  - {service="web"}     → NORMAL (0.02)
  - {service="payment"} → FIRING (0.06)

單一 rule，多 alert instance，每個 instance 有自己的 state。

4.2 跨 datasource 告警範例

「當 metric error rate > 5% 且 log 中含 'panic' 字眼」：

Query A (Prometheus):
  expr: rate(http_5xx[5m]) / rate(http_total[5m])

Query B (Loki):
  expr: count_over_time({app="api"} |= "panic" [5m])

Reduce A → A_value
Reduce B → B_value

Math expression C:
  $A_value > 0.05 AND $B_value > 0
  → boolean

Set C as alert condition

這是 Grafana-managed 才做得到的，Prometheus alerting 不能跨 datasource。

4.3 NoData / Error 處理

每個 rule 可以設定 NoData 與 Error 的行為：

狀況選項
NoData（query 沒回傳資料） Alerting / NoData / OK / KeepLastState
Error（datasource 出錯） Alerting / Error / OK / KeepLastState

建議：

  • NoData → Alerting：嚴重 alert（service down 也是 NoData）
  • NoData → OK：定期 metric（如 nightly batch 沒跑沒事）
  • Error → KeepLastState：避免 datasource 短暫 hiccup 觸發誤報

§ 5. Contact Points

5.1 Contact Point 是什麼

「通知送到哪」的設定。一個 contact point 可包含多個 integration（同時送多處）。

5.2 內建 Integration 類型（30+ 種）

類別整合
聊天 / 協作Slack / Microsoft Teams / Discord / Google Chat / Mattermost / Telegram / Cisco Webex
PagingPagerDuty / OpsGenie / VictorOps / Grafana OnCall
EmailEmail
WebhookWebhook（萬用） / Alertmanager / Pushover / Threema
OtherSMS（via providers） / Telegram / WeCom / Kafka REST

Name: slack-platform-alerts
Integration: Slack

Settings:

  Recipient: #platform-alerts
  Username: Grafana
  Icon emoji: :grafana:
  Title: '{{ template "default.title" . }}'
  Text: '{{ template "default.message" . }}'

Optional settings:
  Mention users: U12345
  Mention groups: subteam12345
  Color: 自訂或 template

Name: pagerduty-critical
Integration: PagerDuty

Settings:
  Integration Key: 
  Severity: '{{ .CommonLabels.severity }}'
  Class: '{{ .CommonLabels.alertname }}'
  Component: '{{ .CommonLabels.service }}'

5.6 多 Integration 在同一 Contact Point

一個 contact point 可加多個 integration（並聯）：

Contact Point: critical-alerts
  Integration 1: Slack #incidents
  Integration 2: PagerDuty
  Integration 3: Email oncall@example.com

→ 一個 alert match 到這個 contact point，
  Slack + PD + Email 三家同時送

5.7 測試 Contact Point

每個 contact point 設定頁有 "Test" 按鈕，發送測試通知，production 上線前必測。

§ 6. Notification Policies（路由樹）

6.1 Tree 結構

Notification Policies 是樹狀結構，根是 Default Policy，下面有任意層 child policy：

Default Policy
  Contact: fallback-email
  Group by: alertname

├── Child: severity = critical
│   Contact: pagerduty-critical
│   Continue: true
│
│   ├── Grandchild: team = database
│   │   Contact: dba-pagerduty
│   │
│   └── Grandchild: team = platform
│       Contact: platform-pagerduty
│
├── Child: team = backend
│   Contact: backend-slack
│
└── Child: severity = warning
    Contact: warning-slack

6.2 Routing 規則

  1. 從 root 開始
  2. Match 第一個符合的 child policy
  3. 進入該 policy，繼續 match 它的 child
  4. 找不到 match → 留在當前 policy 用其 contact
  5. 若 child 設 continue: true，繼續 match 其他 sibling

6.3 Match 的條件

用 label matchers：

severity = critical          # 等於
severity =~ "critical|major" # regex 匹配
team != database             # 不等於
service =~ "api|web"
namespace =~ "production-.+"

6.4 Default Policy 設定

Notification Policies 頁面，最上面的就是 default：

Default contact point: default-email
Group by: ['alertname']
Group wait: 30s
Group interval: 5m
Repeat interval: 4h

6.5 Timing Options 詳解

選項意義預設
Group by 把符合條件的 alert 合併成一條通知 ['alertname']
Group wait 第一個 alert 進來等多久才發（讓同 group 其他 alert 一起發） 30s
Group interval 同一 group 兩次通知之間的最短間隔 5m
Repeat interval 對「沒變化的 firing alert」重複通知間隔 4h

6.6 Override 設定

Child policy 可以「override」父 policy 的設定，否則繼承：

Default:
  group_by: [alertname]
  group_wait: 30s
  repeat_interval: 4h

Child (severity=critical):
  Override grouping: group_by: [alertname, instance]
  Override timings: group_wait: 10s, repeat_interval: 1h
  → critical alert 更快發、更頻繁重複

6.7 Continue 的用法

continue: true 讓 alert 同時 match 多個 policy（不停在第一個 match）。

Policy A (severity=critical, continue=true):
  Contact: pagerduty-critical

Policy B (team=platform):
  Contact: platform-slack

Alert with severity=critical, team=platform:
  → PD-critical 會收到（A）
  → platform-slack 也會收到（B）

若 A 沒設 continue，只 PD 收到

§ 7. Mute Timings

7.1 Mute Timing 是什麼

「定期不發通知的時段」。例：「每天 02:00-04:00 是 maintenance window，不發 warning」。

7.2 設定範例

Name: maintenance-window
Time intervals:
  - Times: 02:00 - 04:00
    Weekdays: saturday
    Months: any
    Days of month: any
    Years: any
    Location: Asia/Taipei

7.3 套用到 Policy

Notification Policy 設定中：

Policy: warning-alerts
  Contact: slack-warning
  Mute timings: [maintenance-window]

→ 這個 policy 的 alert 在 maintenance window 期間不發通知
  （但 alert state 仍是 firing，只是 muted）

7.4 Mute Timing vs Silence

面向Mute TimingSilence
觸發定期重複（每週六、每天某時段）一次性的時間範圍
適用規律 maintenance / 非工作時段臨時停機 / 部署
建立policy 設定UI / API 即時建

§ 8. Silences

8.1 Silence 是什麼

類似 Alertmanager 的 silence：暫時抑制特定 alert 通知。

8.2 建立 Silence

UI: Alerting → Silences → New silence

Matchers:
  - alertname = "HighErrorRate"
  - service = "api"

Start: 2026-05-02 08:00
End: 2026-05-02 14:00 (6 hour duration)

Comment: "Deploying new version, expected errors during rollout"
Created by: alice@example.com

8.3 透過 API 建立

curl -X POST http://localhost:3000/api/alertmanager/grafana/api/v2/silences \
  -H "Authorization: Bearer " \
  -H "Content-Type: application/json" \
  -d '{
    "matchers": [
      {"name": "alertname", "value": "HighErrorRate", "isRegex": false}
    ],
    "startsAt": "2026-05-02T08:00:00Z",
    "endsAt": "2026-05-02T14:00:00Z",
    "createdBy": "ci-bot",
    "comment": "Deploying v2.3.0"
  }'

§ 9. Templates 與通知客製化

9.1 Default Template

Grafana 預設 template 已涵蓋大部分情境。但您可以客製化。

9.2 Template 寫法

Alerting → Contact points → Templates → New template：

Name: custom-slack-title

Content:
{{ define "custom.slack.title" }}
{{- if eq .Status "firing" -}}
🔥 [FIRING:{{ .Alerts.Firing | len }}] {{ .CommonLabels.alertname }}
{{- else -}}
✅ [RESOLVED] {{ .CommonLabels.alertname }}
{{- end -}}
{{ end }}

9.3 在 Contact Point 引用

Slack Contact Point:
  Title: '{{ template "custom.slack.title" . }}'
  Text:  '{{ template "custom.slack.message" . }}'

9.4 Template Data 結構

跟 Alertmanager 完全一樣（Ch 26 詳述）：

  • .Status: firing / resolved
  • .Alerts: alert 集合
  • .CommonLabels / .CommonAnnotations
  • .GroupLabels
  • .ExternalURL

所有 Alertmanager 的 Go template 函式都可用（toUpper / SortedPairs 等）。

§ 10. Provisioning（推薦 Production）

10.1 為什麼用 Provisioning

  • UI 點點建可以但不能 Git 版本控管
  • 多環境（prod / staging）一致
  • CI/CD pipeline 自動 deploy

10.2 完整檔案結構

/etc/grafana/provisioning/alerting/
├── rules.yaml
├── contact-points.yaml
├── policies.yaml
├── mute-timings.yaml
└── templates.yaml

10.3 rules.yaml

apiVersion: 1

groups:
  - orgId: 1
    name: api-alerts
    folder: Production
    interval: 1m
    rules:
      - uid: high-error-rate
        title: HighErrorRate
        condition: C
        data:
          - refId: A
            datasourceUid: prometheus-prod
            model:
              expr: 'rate(http_5xx[5m]) / rate(http_total[5m])'
              refId: A
          - refId: B
            datasourceUid: __expr__
            model:
              type: reduce
              expression: A
              reducer: last
              refId: B
          - refId: C
            datasourceUid: __expr__
            model:
              type: threshold
              expression: B
              conditions:
                - evaluator:
                    type: gt
                    params: [0.05]
              refId: C
        noDataState: NoData
        execErrState: Error
        for: 5m
        annotations:
          summary: 'API error rate is {{ $values.B | printf "%.2f%%" }}'
          runbook_url: 'https://wiki/api-errors'
        labels:
          severity: critical
          team: platform

10.4 contact-points.yaml

apiVersion: 1

contactPoints:
  - orgId: 1
    name: slack-platform
    receivers:
      - uid: slack-platform-receiver
        type: slack
        settings:
          url: https://hooks.slack.com/services/XXX/YYY/ZZZ
          recipient: '#platform-alerts'

  - orgId: 1
    name: pagerduty-critical
    receivers:
      - uid: pd-critical-receiver
        type: pagerduty
        settings:
          integrationKey: $__file{/etc/grafana/secrets/pd-key}

10.5 policies.yaml

apiVersion: 1

policies:
  - orgId: 1
    receiver: default-email
    group_by: [alertname, cluster]
    group_wait: 30s
    group_interval: 5m
    repeat_interval: 12h

    routes:
      - receiver: pagerduty-critical
        matchers:
          - severity = critical
        group_wait: 10s
        repeat_interval: 1h
        continue: true

      - receiver: slack-platform
        matchers:
          - team = platform
        group_by: [alertname, namespace]
        repeat_interval: 4h

      - receiver: slack-backend
        matchers:
          - team = backend
        mute_time_intervals:
          - maintenance-window

10.6 mute-timings.yaml

apiVersion: 1

muteTimes:
  - orgId: 1
    name: maintenance-window
    time_intervals:
      - times:
          - start_time: '02:00'
            end_time: '04:00'
        weekdays: ['saturday']
        location: 'Asia/Taipei'

§ 11. State History 與 Audit

11.1 看 alert history

UI: Alerting → Alert rules → 點選 rule → State history

顯示 state 變化時間軸（Normal → Pending → Alerting → Resolved）。

11.2 Annotation 紀錄（預設）

State 變化會自動寫入 dashboard annotation，看 graph 時會顯示垂直線。

11.3 Loki State History（推薦 production）

大量 alert 時 annotation 表會爆。改用 Loki 儲存 state history：

[unified_alerting.state_history]
enabled = true
backend = loki
loki_remote_url = http://loki:3100

11.4 用 API 查詢

curl -H "Authorization: Bearer " \
  "http://grafana/api/v1/rules/history?ruleUID=high-error-rate&from=2026-05-01T00:00:00Z&to=2026-05-02T00:00:00Z"

§ 12. 與 Prometheus Alerting 的選擇

12.1 完整對比

面向Prometheus + AMGrafana Alerting
評估位置Prometheus / MimirGrafana
跨 datasource否✅
UIAM Web UI（陽春）Grafana 主介面
規模化Mimir Ruler 強有限（依賴 Grafana 自身規模）
HAAM HA 成熟需要 external AM 才能 HA
不依賴 Grafana✅❌ Grafana 掛了 alert 也掛
Best forProm-only 環境多 datasource、想要單一 UI

12.2 兩種共存模式

模式 1：Grafana Alerting 為主

  • 所有 alert 都在 Grafana 寫
  • 適合：小團隊、只用 metric + log + 業務 SQL

模式 2：Prometheus / Mimir 為主

  • Pure metric alert 在 Prometheus rule
  • 少數跨 datasource 在 Grafana
  • Grafana datasource 設「Manage alerts via Alerting UI」可以從 Grafana UI 寫 Prom rule
  • 適合：大規模、有 Mimir 部署

12.3 v12 的進化

Grafana 12 引入 Recording Rules in Grafana UI。原本 recording rule 只能在 Prometheus rule file 寫， 現在可以在 Grafana UI 直接建立並寫回 Mimir。配 alerting 一站式的體驗。
