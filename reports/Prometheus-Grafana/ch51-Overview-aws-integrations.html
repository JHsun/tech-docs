Chapter 51 · AWS 整合

AWS 觀測性整合：CloudWatch、AMP、AMG、X-Ray

§ 1. AWS 觀測性服務全景

1.1 服務矩陣

訊號AWS 原生AWS Managed OSS
MetricsCloudWatch MetricsAmazon Managed Prometheus (AMP)
LogsCloudWatch LogsOpenSearch Service / 自架 Loki on EKS
TracesAWS X-RayOpenTelemetry → AMP / 自架 Tempo
VisualizationCloudWatch DashboardsAmazon Managed Grafana (AMG)
AlertingCloudWatch AlarmsAMP Ruler / AMG Alerting
SyntheticsCloudWatch Syntheticsk6（自架 / Grafana Cloud）
RUMCloudWatch RUMFaro

1.2 兩大選擇路線

路線 A：純 AWS 原生

  • CloudWatch + X-Ray + CloudWatch Logs
  • 整合 AWS service 最深
  • vendor lock-in

路線 B：Managed OSS

  • AMP + AMG + 自架 Loki/Tempo
  • Prometheus / Grafana 標準工具
  • 跨雲遷移容易

1.3 推薦路線

✅ 多數情境推薦

用 AMP + AMG 為主，CloudWatch 為輔：

  • 應用 metric / 業務 metric → AMP
  • AWS service metric（RDS / ELB / Lambda 等）→ CloudWatch（無法移）
  • 用 AMG 整合兩家，Grafana 是統一 UI
  • X-Ray 給 AWS-native（Lambda、Step Functions），其他用 Tempo

§ 2. CloudWatch Metrics

2.1 CloudWatch 是什麼

AWS 原生的監控服務，自動收集大多數 AWS 服務的 metric（EC2 / RDS / ELB / Lambda / ECS / EKS 等）。 1 分鐘解析度（基本）/ 1 秒解析度（detailed monitoring 收費）。

2.2 CloudWatch 結構

Namespace: AWS/EC2
  └── Metric: CPUUtilization
      └── Dimensions: InstanceId=i-1234567890
          └── Statistics: Average / Sum / Min / Max / SampleCount
              └── Period: 60s

2.3 自訂 Metric

# 用 AWS CLI / SDK 寫
aws cloudwatch put-metric-data \
  --namespace MyApp \
  --metric-name OrdersPlaced \
  --value 1 \
  --dimensions Service=checkout,Region=us-east-1

# Python SDK
import boto3
cw = boto3.client('cloudwatch')
cw.put_metric_data(
    Namespace='MyApp',
    MetricData=[{
        'MetricName': 'OrdersPlaced',
        'Value': 1,
        'Dimensions': [
            {'Name': 'Service', 'Value': 'checkout'},
        ],
    }],
)

2.4 CloudWatch 限制

  • 解析度：基本 1 分鐘，1 秒收費高
  • 查詢能力弱：沒有 PromQL 等強大語法
  • 標籤限制：每個 metric 最多 10 dimensions
  • 成本：自訂 metric $0.30/metric/月，cardinality 高很貴

2.5 從 Grafana 查 CloudWatch

// Datasource
- name: CloudWatch
  type: cloudwatch
  jsonData:
    authType: default       // 用 Pod IAM Role
    defaultRegion: us-east-1

// Panel query
Namespace: AWS/EC2
Metric: CPUUtilization
Dimension: InstanceId
Statistic: Average
Period: 60

2.6 CloudWatch Insights / Metrics Insights

類似 SQL 的查詢語法：

SELECT AVG(CPUUtilization)
FROM SCHEMA("AWS/EC2", InstanceId)
WHERE InstanceId = 'i-1234567890'

§ 3. CloudWatch Logs

3.1 結構

Log Group: /aws/lambda/my-function
  └── Log Stream: 2026/05/02/[$LATEST]abc123...
      └── Log Events: { timestamp, message }

3.2 從應用 push log

多種方式：

  • Lambda：自動寫 log group
  • ECS / EKS：用 awslogs / fluent-bit driver
  • EC2：CloudWatch Agent
  • Application：直接用 SDK PutLogEvents

3.3 CloudWatch Logs Insights

類似 SQL 的查詢：

fields @timestamp, @message
| filter @message like /ERROR/
| sort @timestamp desc
| limit 100

# Aggregation
fields @timestamp, @message
| filter @message like /5\d\d/
| stats count() by bin(5m)

3.4 從 Grafana 查 CloudWatch Logs

同 CloudWatch datasource 即可，只是 query 模式選 Logs。

3.5 Cost Optimization

  • 不用的 log group 設 retention（預設 forever）
  • 少 log 用 Lambda Powertools / 結構化 log
  • 大量 log 考慮 export 到 S3
  • 非常大規模考慮 OpenSearch 或自架 Loki

§ 4. AWS X-Ray（Tracing）

4.1 X-Ray 定位

AWS 原生 distributed tracing 服務。整合 Lambda、API Gateway、ECS、EKS、AppSync。 跟 OpenTelemetry 並存（X-Ray 早於 OTel）。

4.2 與 Lambda 整合

# SAM template
Resources:
  MyFunction:
    Type: AWS::Serverless::Function
    Properties:
      Tracing: Active     # 啟用 X-Ray

4.3 SDK 整合

# Python 用 aws-xray-sdk
from aws_xray_sdk.core import xray_recorder, patch_all
patch_all()  # 自動 instrument boto3 / requests / mysql 等

@xray_recorder.capture('my_function')
def my_function():
    # ...

4.4 X-Ray vs OpenTelemetry

面向X-RayOpenTelemetry → Tempo
標準AWS-only業界標準
Lambda 整合★★★ 一鍵需 ADOT layer
跨雲不可可
SamplingX-Ray sampling rulesOTel collector tail-based
UIX-Ray console（陽春）Grafana Tempo（強）
整合 LGTMvia plugin原生

4.5 OpenTelemetry on AWS

AWS 推 ADOT (AWS Distro for OpenTelemetry)，是 OTel Collector 的 AWS 發行版：

# Lambda Layer
Resources:
  MyFunction:
    Type: AWS::Serverless::Function
    Properties:
      Layers:
        - !Sub arn:aws:lambda:${AWS::Region}:901920570463:layer:aws-otel-python-amd64-ver-1-21-0:1
      Tracing: Active

ADOT 可同時送 X-Ray + AMP + 第三方 backend。

§ 5. Amazon Managed Prometheus (AMP)

5.1 AMP 是什麼

AWS Managed 的 Prometheus 服務，內部其實是 Cortex（不是 Mimir）。 您不管 server，只送 metric（remote_write）+ 查詢（PromQL）。

5.2 AMP 與自架 Prometheus 對比

面向AMP自架 Prometheus
運維AWS 全管自己
Scrape不能直接 scrape（需 agent）原生
Long-term storage內建需 Thanos / Mimir
HA內建自己設定
Recording rule支援（rule group）原生
Costper-sample 收費infrastructure 收費
限制9-month retention無

5.3 建立 AMP Workspace

aws amp create-workspace \
  --alias my-workspace \
  --region us-east-1

# 取得 endpoint
aws amp describe-workspace --workspace-id ws-abc123
# Output:
# prometheusEndpoint: https://aps-workspaces.us-east-1.amazonaws.com/workspaces/ws-abc123/

5.4 從 EKS push metric

用 ADOT Collector / Alloy + Pod IAM Role：

# prometheus.yaml on EKS
remote_write:
  - url: https://aps-workspaces.us-east-1.amazonaws.com/workspaces/ws-abc123/api/v1/remote_write
    sigv4:
      region: us-east-1
    queue_config:
      max_samples_per_send: 1000
      max_shards: 200

5.5 IAM Role 設定

# Pod IAM Role with policy
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": [
      "aps:RemoteWrite",
      "aps:GetSeries",
      "aps:GetLabels",
      "aps:GetMetricMetadata"
    ],
    "Resource": "arn:aws:aps:us-east-1:123456789012:workspace/ws-abc123"
  }]
}

5.6 AMP 限制

  • Active series 上限：每 workspace 預設 1M（可申請提高）
  • Retention：最多 150 days（v2025+）
  • Query 限制：max samples 50M
  • 不支援 Federation（用 remote_write）
  • AlertManager：需另設 AMP managed AlertManager

5.7 AMP Ruler 與 AlertManager

# 上傳 rule
aws amp create-rule-groups-namespace \
  --workspace-id ws-abc123 \
  --name api-alerts \
  --data file://rules.yaml

# AMP 內建 AlertManager
aws amp create-alert-manager-definition \
  --workspace-id ws-abc123 \
  --data file://alertmanager.yaml

§ 6. Amazon Managed Grafana (AMG)

6.1 AMG 是什麼

AWS Managed 的 Grafana 服務。AWS 跑 Grafana server、處理升級、HA、認證。 您只用 Grafana UI / API。Grafana 12+ 為基礎。

6.2 與自架 Grafana 對比

面向AMG自架 Grafana
運維AWS 全管自己
升級自動手動
SSOIAM Identity Center / SAML 內建需自己設
AWS 整合原生（discovery datasource）需手設 IAM
Plugin有限制（Grafana 認證的）自由
Costper-user 收費infra cost
Customization有限完整

6.3 建立 AMG Workspace

aws grafana create-workspace \
  --workspace-name production \
  --account-access-type CURRENT_ACCOUNT \
  --authentication-providers AWS_SSO \
  --permission-type SERVICE_MANAGED \
  --workspace-data-sources AMAZON_MANAGED_PROMETHEUS CLOUDWATCH

6.4 認證選項

  • IAM Identity Center (formerly AWS SSO)：推薦，與 AWS 帳號整合
  • SAML：用既有 IdP（Okta / Azure AD）

6.5 Permission Model

  • Service-managed：AMG 自動建 IAM role 連接 AWS service（推薦）
  • Customer-managed：手動管 IAM role（細粒度控制）

6.6 AMG v12 重要變化

📌 AMG 12 SigV4 變動

AMG v12 起，core Prometheus plugin 移除 SigV4 支援。
原本用 core plugin + SigV4 的 AMP datasource 自動遷移到專用 AMP plugin。
若您手動建 datasource，必須用 AMP plugin（type: grafana-amazonprometheus-datasource）。

§ 7. AMP + AMG 完整整合

7.1 整合流程

  1. 建立 AMP workspace
  2. EKS / EC2 上 Alloy / ADOT 設定 remote_write 到 AMP
  3. 建立 AMG workspace（含 AMP service-managed permission）
  4. AMG 自動 discover AMP workspace，加成 datasource
  5. 建立 dashboard

7.2 EKS Setup（含 OTLP）

# values-alloy.yaml
alloy:
  configMap:
    create: true
    content: |
      // K8s pod discovery
      discovery.kubernetes "pods" {
        role = "pod"
      }

      prometheus.scrape "k8s_pods" {
        targets    = discovery.kubernetes.pods.targets
        forward_to = [prometheus.remote_write.amp.receiver]
      }

      prometheus.remote_write "amp" {
        endpoint {
          url = "https://aps-workspaces.us-east-1.amazonaws.com/workspaces/ws-abc123/api/v1/remote_write"
          sigv4 {
            region = "us-east-1"
          }
        }
      }

  serviceAccount:
    create: true
    annotations:
      eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/AlloyAMPRole

7.3 AMG 加 AMP Datasource（Service-managed）

# AMG console 啟用 service-managed permission
aws grafana update-permissions \

    "action": "ADD",

§ 8. EKS 觀測性策略

8.1 EKS 標準配置

典型 EKS 觀測性 stack：

EKS Cluster
├── Application pods（含 OTel SDK）
├── Alloy DaemonSet（每 node 一個）
│   ├── 收 K8s metric（kube-state-metrics、cAdvisor）
│   ├── 收 application metric（pod scrape）
│   ├── 收 application log
│   └── 接 OTLP traces
├── ADOT Collector（OTel collector aws variant）
│   └── X-Ray + AMP + AMG
└── kube-prometheus-stack（可選，含 Prom + Grafana 自架）

8.2 EKS Add-ons

AWS 提供 1-click EKS add-on 安裝：

# CloudWatch Container Insights
aws eks create-addon \
  --cluster-name my-cluster \
  --addon-name amazon-cloudwatch-observability

# ADOT
aws eks create-addon \
  --cluster-name my-cluster \
  --addon-name adot

8.3 IRSA（IAM Roles for Service Accounts）

EKS 給 pod 賦 AWS 權限的標準做法：

aws iam create-role \
  --role-name AlloyAMPRole \
  --assume-role-policy-document file://trust-policy.json

# trust-policy.json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Federated": "arn:aws:iam::123456789012:oidc-provider/oidc.eks.us-east-1.amazonaws.com/id/EXAMPLE"
    },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringEquals": {
        "oidc.eks.us-east-1.amazonaws.com/id/EXAMPLE:sub":
          "system:serviceaccount:observability:alloy"
      }
    }
  }]
}

8.4 Pod Identity (新方法)

2024 年起，AWS 推 EKS Pod Identity 取代 IRSA（更簡單）：

aws eks create-pod-identity-association \
  --cluster-name my-cluster \
  --namespace observability \
  --service-account alloy \
  --role-arn arn:aws:iam::123456789012:role/AlloyAMPRole

§ 9. 自架 vs Managed 取捨

9.1 三種選項

選項優點缺點
1. CloudWatch only 整合最深、設定最少 查詢弱、cardinality 貴、vendor lock-in
2. AMP + AMG 標準 PromQL、AWS 管基礎設施 有 AWS 限制、跨雲難
3. 自架 LGTM on EKS 完全控制、跨雲可 運維成本高

9.2 Cost 對比（粗估，10M series）

  • CloudWatch：自訂 metric $0.30/metric/月 × 10M = ~$3M/月（不可行）
  • AMP：~$0.30/M sample → 10M series × 6/min × 30 days = $5K-15K/月
  • 自架 Mimir on EKS：~$2K-5K/月（infra）+ 運維成本

9.3 場景對應

場景推薦
純 AWS、團隊小CloudWatch + 簡單 AMG
EKS 為主、要 PromQLAMP + AMG
多雲自架 LGTM
已用 Datadog繼續，不用換
大規模、高 cardinality自架 Mimir

§ 10. 從 CloudWatch 遷移到 AMP/AMG

10.1 為什麼遷移

  • CloudWatch 自訂 metric 太貴
  • 需要 PromQL 強查詢能力
  • 需要 cross-account / cross-region 統一視圖
  • 跨雲未來需求

10.2 漸進遷移計畫

  1. Stage 1：建 AMG，加 CloudWatch datasource（不影響現況）
  2. Stage 2：建 AMP workspace
  3. Stage 3：應用加 OTel / Prometheus client，雙寫 CloudWatch + AMP
  4. Stage 4：dashboard 改用 AMP datasource
  5. Stage 5：Alert 改用 AMP Ruler
  6. Stage 6：停止寫 CloudWatch 自訂 metric

10.3 CloudWatch 仍要保留

有些 metric 只能從 CloudWatch 來：

  • RDS / Aurora performance metric
  • ELB / ALB request metric
  • Lambda invocations / errors
  • VPC Flow Logs metric
  • S3 bucket metric

對這些，AMG 的 CloudWatch datasource 直接查，dashboard 中混用兩家。

大企業典型：每個業務一個 AWS 帳號，但要統一 dashboard。
