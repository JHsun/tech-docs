Chapter 05 · Prometheus 核心

Prometheus 概念、架構與資料模型

本章是 Prometheus 教學的核心起點。我們會從零建立完整的心智模型： Prometheus 究竟是什麼、它如何運作、time series 是怎麼被組織與儲存的、 以及為什麼這樣設計。理解這些基礎，後續所有章節都會輕鬆許多。

§ 1. Prometheus 簡史與設計哲學

Prometheus 由 SoundCloud 於 2012 年內部創立，2015 年公開釋出， 2016 年加入 CNCF（Cloud Native Computing Foundation）， 成為繼 Kubernetes 之後第二個 CNCF 託管專案。2018 年達到 Graduated 等級。

它的設計受 Google 內部的 Borgmon（Kubernetes 的前身 Borg 的監控系統）啟發，核心哲學包括：

  • 多維度資料模型：每筆 time series 由 metric 名稱加上一組 key-value labels 識別
  • Pull-based：Prometheus 主動從 targets 抓取 metrics，而非被動接收 push
  • 強大的查詢語言 PromQL：專為時序資料設計
  • 單機獨立運作：不依賴分散式儲存，每個 node 自包含
  • 每個維度可獨立切片：透過 labels 自由聚合與分組
  • 適合「白盒監控（white-box monitoring）」：暴露應用內部狀態
  • 不追求 100% 精確：監控資料容忍少量遺失，可靠度優於完整性
ℹ️ Prometheus 3.x 新時代

Prometheus 在 2024 年 11 月發布 3.0，是七年來首個主版本。 重大變更包括：UTF-8 metric 與 label 名稱預設啟用、全新的 React UI、 le 標籤值的正規化（OpenMetrics canonical numbers）、移除過時的 feature flags。 本教學以 v3.11 為基準，內容對 v2.x 仍大致適用。

§ 3. 資料模型：Time Series 的構成

這是整個 Prometheus 的核心抽象。理解 time series，等於理解 Prometheus 一半的世界。

3.1 一條 Time Series 是什麼

一條 time series 由三個部分組成：

  1. Metric Name（指標名稱）：例如 http_requests_total
  2. Labels（標籤組）：例如 {method="POST", path="/api/login", status="200"}
  3. Samples（樣本序列）：依時間順序排列的（timestamp, value）配對

3.2 Series Identifier（系列識別）

每條 time series 由「metric name + 完整 label set」唯一識別。 換句話說，下面這兩條是不同的 series： 

http_requests_total{method="POST", path="/api/login", status="200"}
http_requests_total{method="POST", path="/api/login", status="500"}

這兩條只差在 status label 不同，因此被視為兩條獨立 series，各自儲存自己的樣本。 這正是 Prometheus 「多維度模型」的基石——你可以用任意 label 組合切片分析。

3.3 Series 的內部表示

在 TSDB 內部，每條 series 會被分配一個 64-bit 整數 ID， 並建立兩種索引：

  • Postings index：從 (label_name, label_value) 找到所有相關 series ID 的倒排索引
  • Series index：從 series ID 反向查到完整的 label set

這個設計讓查詢 http_requests_total{status="500"} 這類條件可以非常快速。

3.4 Cardinality（基數）的概念

⚠️ 高基數是 Prometheus 的頭號殺手

Cardinality 指的是某個 metric 名稱下「不同 label 組合」的數量。 每一個獨特組合 = 一條 series = 佔用記憶體與磁碟。
若你把 user_id 放進 label，每個使用者就產生一條獨立 series， 百萬使用者 = 百萬條 series，Prometheus 記憶體會直接爆炸。
這部分將在 Ch 19、Ch 60 反覆強調，是業界踩雷最多的點。

3.5 簡單算式：Series 數量估算

假設你有一個 metric：

http_requests_total{method, path, status, instance}

若 method 有 5 種、path 有 50 個 endpoint、 status 有 10 種、instance 有 20 個 pod， 最多會產生 5 × 50 × 10 × 20 = 50,000 條 series。 實務上不是所有組合都會出現，但這給你一個量級概念。

§ 4. Sample 與 Timestamp

4.1 Sample 的內部結構

每個 sample 由「時間戳記」加「數值」組成：

欄位類型說明
Timestampint64（毫秒）UNIX epoch 時間，毫秒精度
Valuefloat64IEEE 754 雙精度浮點數

從 Prometheus 2.40 開始，新增了 native histogram sample（複合型樣本）， 一個 sample 可以包含完整的 histogram 結構（buckets、sum、count）。 Prometheus 3.x 持續強化此模型，這部分將在 Ch 06 詳述。

4.2 Timestamp 的兩個來源

來源說明
Scrape 時間（預設） Prometheus 在抓取的當下記錄時間戳記。這是最常見的情況。
Target 自帶 Target 在 /metrics 輸出中明確帶 timestamp（極少用），會被視為「歷史資料」

4.3 Float 與特殊值

因為 value 是 float64，Prometheus 支援 IEEE 754 的特殊值：

  • NaN（Not a Number）：通常表示「無資料」或「不適用」
  • +Inf / -Inf：用於 histogram 的 +Inf bucket（無上界）
  • 整數運算注意：2^53 之後 float64 開始失去整數精度

§ 5. Metric 命名與 Label 命名規範

5.1 Metric Name 規則

合法字元（Prometheus 2.x）：[a-zA-Z_:][a-zA-Z0-9_:]*。

從 Prometheus 3.0 開始，UTF-8 名稱預設啟用， 所以 OpenTelemetry 風格的 http.server.duration 也合法。 但為了向後相容與生態工具支援，建議仍遵守傳統規範。

5.2 命名最佳實踐

✅ Best Practice
  1. 使用單一單字 namespace 前綴：例如 http_、node_、process_
  2. 使用底線分隔字詞：snake_case，不是 camelCase
  3. Counter 必須以 _total 結尾：例如 http_requests_total
  4. 單位作後綴：_seconds、_bytes、_meters，且使用基本單位（不要 milliseconds、KB）
  5. 名稱要表達什麼，不是怎麼採：用 node_filesystem_avail_bytes，不是 node_disk_check_bytes

5.3 Label 命名規則

合法字元：[a-zA-Z_][a-zA-Z0-9_]*（注意：label 名稱沒有冒號）。

保留的特殊 label：

  • __name__：metric 名稱本身就是一個 label，PromQL 中可以用 {__name__="http_requests_total"} 查詢
  • __address__、__metrics_path__、__scheme__ 等：以 __ 開頭的 label 是內部使用，scrape 時會被移除（但 relabeling 階段可用）
  • job、instance：每筆 scrape 的資料會自動加上這兩個 label

5.4 Label 命名最佳實踐

✅ Best Practice
  1. Label 是「維度」，不是「資料」：應該能列舉，不能有無界值
  2. 避免高基數的 label：禁忌名單 → user_id、email、request_id、IP（除非是少量機器）、UUID
  3. 命名一致：同一概念在所有 metric 用相同 label 名（例如統一用 method，不要混用 http_method）
  4. 不要在 label 重複 metric 名：http_requests_total{type="http"} 是冗餘
  5. Label 可以為空字串：foo{x=""} 等同於完全沒有 x label

§ 6. 暴露格式（Exposition Format）

Target 透過 HTTP 在 /metrics（預設路徑）以特定格式暴露 metrics 給 Prometheus 抓。 目前有三種主要格式：

格式用途Content-Type
Prometheus text最常見、人類可讀text/plain; version=0.0.4
OpenMetrics text標準化版本，向 CNCF 提交application/openmetrics-text
Prometheus protobuf支援 native histogram，效能佳application/vnd.google.protobuf; ...

6.1 文字格式範例

# HELP http_requests_total Total HTTP requests received.
# TYPE http_requests_total counter
http_requests_total{method="GET",path="/health",status="200"} 12453
http_requests_total{method="POST",path="/api/login",status="200"} 89
http_requests_total{method="POST",path="/api/login",status="401"} 7

# HELP node_memory_MemAvailable_bytes Memory available in bytes.
# TYPE node_memory_MemAvailable_bytes gauge
node_memory_MemAvailable_bytes 4.294967296e+09

幾個關鍵語法元素：

  • # HELP：metric 的人類可讀說明
  • # TYPE：metric 類型（counter / gauge / histogram / summary / untyped）
  • 每行一個 sample：metric_name + 可選 labels + 數值 + 可選 timestamp
  • 空白行分隔可讀，但非必須
  • 科學記號 4.294967296e+09 也合法

6.2 Histogram 與 Summary 的展開

一個 classic histogram 在文字格式中會被「攤平」成多條 series：

# HELP http_request_duration_seconds Request duration histogram.
# TYPE http_request_duration_seconds histogram
http_request_duration_seconds_bucket{le="0.1"} 24054
http_request_duration_seconds_bucket{le="0.5"} 33444
http_request_duration_seconds_bucket{le="1.0"} 100392
http_request_duration_seconds_bucket{le="+Inf"} 144320
http_request_duration_seconds_sum 53423
http_request_duration_seconds_count 144320

注意：

  • _bucket：每個 bucket 是一條獨立 series，le（less or equal）標籤標示上界
  • _sum：所有觀測值的總和
  • _count：總觀測次數，等同於 {le="+Inf"}
  • Prometheus 3.0+ 的新規則：le 值會被正規化（例如 1.0 → 1），符合 OpenMetrics canonical numbers

§ 7. Scrape 模型運作流程

了解 scrape 的完整生命週期，能幫助您診斷大多數問題。

7.2 自動加上的 labels

每筆 scrape 進來的 sample，Prometheus 會自動附加：

  • job：來自 scrape_configs.job_name
  • instance：通常是 host:port

這兩個 label 構成最基本的「資料來源 ID」。在後續 PromQL 章節會大量用到。

§ 8. 內部 metric 與 staleness

8.1 自動產生的 up metric

每個 target 每次 scrape，Prometheus 會自動寫入：

up{job="node-exporter", instance="server-1:9100"} 1   # scrape 成功
up{job="node-exporter", instance="server-1:9100"} 0   # scrape 失敗

up 是監控 Prometheus 自身運作的最基本工具。 任何告警規則的第一條，幾乎都是「up == 0 持續 N 分鐘」。

8.2 其他自動 metrics

Metric意義
up1 = 抓取成功，0 = 失敗
scrape_duration_seconds本次 scrape 花費的時間
scrape_samples_scraped本次抓到多少 samples
scrape_samples_post_metric_relabelingrelabel 後保留多少 samples
scrape_series_added本次新增的 series 數

8.3 Staleness（陳舊性）

一個 series 若在連續兩次 scrape 中消失（例如 pod 被刪、target 下線）， Prometheus 會在下一次 scrape 時主動寫入一個特殊的「staleness marker」， 讓查詢知道這個 series 已停止更新。預設「lookback 5 分鐘」內若沒新樣本就視為消失。

§ 9. 限制與不適合的場景

Prometheus 強大但不是萬能。理解它的邊界，才能避免錯用。

🚨 Prometheus 不適合的場景
  1. 需要 100% 準確的計費系統：監控資料容忍少量遺失，計費業務不能
  2. 事件記錄（event logging）：用 logs 系統（Loki / ELK）
  3. 原始請求資料儲存：每筆請求的完整資料用 traces 或 logs
  4. 長期歷史資料的精確存查：Prometheus 本地 TSDB 通常只保留 15-30 天，要長期看用 Thanos / Mimir / VictoriaMetrics
  5. 處理高基數的個體識別：每個使用者的指標應該用 logs 或 OLAP，不是 Prometheus
⚠️ Prometheus 的內建限制
  • 單機儲存上限：取決於磁碟，通常實務上 1-2 億 active series 是上限
  • 無原生 HA：跑兩個獨立實例同時抓相同 targets，由 Alertmanager 去重
  • 無原生 multi-tenancy：要多租戶請用 Mimir / Cortex
  • Pull 模型不支援動態：短任務需 Pushgateway
