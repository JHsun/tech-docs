§ 1. Grafana HTTP API

1.1 認證

API 呼叫的三種認證方式：

方式Header適用
Service Account TokenAuthorization: Bearer <token>推薦（v9+）
API Key（legacy）Authorization: Bearer <key>避免
Basic AuthAuthorization: Basic <b64>個人 debug

1.2 常用 API endpoint

用途Endpoint
Health checkGET /api/health
Build infoGET /api/frontend/settings
所有 dashboardGET /api/search?type=dash-db
取 dashboardGET /api/dashboards/uid/{uid}
建/改 dashboardPOST /api/dashboards/db
所有 datasourceGET /api/datasources
建 datasourcePOST /api/datasources
使用者列表GET /api/users
Service AccountsGET /api/serviceaccounts
Alert rulesGET /api/v1/provisioning/alert-rules
Plugin listGET /api/plugins
Settings（admin）GET /api/admin/settings

1.3 範例：批次匯出所有 dashboard

#!/bin/bash
# export-dashboards.sh
GRAFANA="https://grafana.example.com"
TOKEN="glsa_xxxxxxxxxxxxx"
OUT_DIR="./dashboards-backup"
mkdir -p "$OUT_DIR"

# 列出所有 dashboard
curl -s -H "Authorization: Bearer $TOKEN" \
  "$GRAFANA/api/search?type=dash-db" | \
  jq -r '.[] | .uid' | \
  while read uid; do
    echo "Backing up $uid..."
    curl -s -H "Authorization: Bearer $TOKEN" \
      "$GRAFANA/api/dashboards/uid/$uid" \
      | jq '.dashboard' \
      > "$OUT_DIR/${uid}.json"
  done

echo "Done. $(ls $OUT_DIR | wc -l) dashboards backed up."

1.4 範例：批次建立 dashboard

# 從 JSON 檔還原
curl -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d @dashboard.json \
  "$GRAFANA/api/dashboards/db"

1.5 注意：API rate limit

OSS 沒原生 rate limit，但 reverse proxy 通常有。批次操作時加 sleep：

for uid in $UIDs; do
  curl ...
  sleep 0.1   # 100ms 間隔
done

§ 2. grafana-cli 工具

2.1 grafana-cli 是什麼

Grafana 自帶的 CLI，主要用於 plugin 管理與 admin 操作。Binary 安裝後在 /usr/sbin/grafana-cli。

2.2 Plugin 管理

# 列出已安裝
grafana-cli plugins ls

# 搜尋
grafana-cli plugins list-remote | grep panel

# 安裝
grafana-cli plugins install grafana-piechart-panel

# 安裝特定版本
grafana-cli plugins install grafana-clock-panel 2.1.0

# 升級
grafana-cli plugins update grafana-piechart-panel

# 升級全部
grafana-cli plugins update-all

# 移除
grafana-cli plugins remove grafana-piechart-panel

2.3 Admin 操作

# 重設 admin 密碼
grafana-cli admin reset-admin-password new-password

# 加密 / 解密 secret
grafana-cli encryption reencrypt-secrets
grafana-cli encryption rotate-data-encryption-keys

# 從 SQLite 遷到外部 DB
grafana-cli admin data-migration encrypt-datasource-passwords

2.4 Docker / K8s 環境

# Docker
docker exec -it grafana grafana-cli plugins ls

# K8s
kubectl exec -it grafana-pod -n monitoring -- grafana cli plugins ls

2.5 注意

⚠️ Plugin 變動需要重啟

安裝、升級、移除 plugin 後必須重啟 Grafana，否則沒生效： 

# systemd
sudo systemctl restart grafana-server

# Docker
docker restart grafana

# K8s（用 Helm 重新 deploy 或 rollout）
kubectl rollout restart deployment/grafana

§ 3. grafanactl（v12+）

3.1 grafanactl 是什麼

Grafana 12 引入的新一代 CLI，類似 kubectl。針對「Observability as Code」設計， 跟 Foundation SDK 與新 v1beta1 API 配合。

3.2 與 grafana-cli 的差別

面向grafana-cligrafanactl
定位Server admin（plugin / password）Resource CRUD（dashboard / alert）
跑在Grafana server 上本地工作站
連線local filesremote API
輸出plain textYAML / JSON

3.3 安裝

# macOS
brew install grafana/grafana/grafanactl

# Linux
curl -sSL https://github.com/grafana/grafanactl/releases/latest/download/grafanactl_linux_amd64.tar.gz | tar xz

3.4 設定

# 設定 context（類似 kubectl）
grafanactl config set-context production \
  --server=https://grafana.example.com \
  --token=glsa_xxxxxxxxx

grafanactl config use-context production

3.5 常用指令

# List dashboards
grafanactl get dashboards

# Get specific
grafanactl get dashboard api-overview -o yaml

# Apply (create or update)
grafanactl apply -f dashboard.yaml

# Delete
grafanactl delete dashboard api-overview

# 同 kubectl 風格
grafanactl get alertrules
grafanactl get datasources
grafanactl get folders

§ 4. 備份與還原

4.1 備份什麼

項目位置頻率
Internal DBSQLite / PG / MySQL每天（小型）/ 每小時（大型）
Plugin data/var/lib/grafana/pluginsplugin 變動時
grafana.ini/etc/grafana/放 Git
Provisioning/etc/grafana/provisioning/放 Git
Encryption keysecret_key 在 grafana.ini專屬秘密管理

4.2 SQLite 備份

# 簡單複製（須注意 Grafana 寫入時可能不一致）
sudo systemctl stop grafana-server
cp /var/lib/grafana/grafana.db /backup/grafana-$(date +%F).db
sudo systemctl start grafana-server

# 推薦：用 sqlite3 .backup（不需停機）
sqlite3 /var/lib/grafana/grafana.db ".backup /backup/grafana-$(date +%F).db"

4.3 PostgreSQL 備份

# 全表 dump（包含 schema + data）
pg_dump -h postgres-host -U grafana -d grafana \
  -F custom \
  -f /backup/grafana-$(date +%F).dump

# 用 cron 每天備份
0 3 * * * pg_dump -h postgres -U grafana grafana -F custom -f /backup/grafana-$(date +\%F).dump

# 還原
pg_restore -h postgres-host -U grafana -d grafana \
  --clean \
  /backup/grafana-2026-05-02.dump

4.4 透過 API 備份 dashboard（補充）

除了 DB 備份，dashboard JSON 也應該放 Git：

# 每天匯出所有 dashboard 到 Git repo
0 4 * * * /usr/local/bin/export-dashboards.sh && \
  cd /git/grafana-backup && \
  git add . && \
  git commit -m "Daily backup $(date +%F)" && \
  git push

4.5 災難還原 SOP

  1. 用 IaC（Helm / Terraform）重建 Grafana instance
  2. 還原 grafana.ini（從 Git）
  3. 還原 internal DB（從 PG dump）
  4. 還原 secret_key（從 Vault）
  5. 啟動 Grafana
  6. 檢查 dashboard / datasource / alert 是否正常
  7. RTO（Recovery Time Objective）：30 分鐘

§ 5. 版本升級

5.1 升級策略

版本類型建議
Patch（12.4.0 → 12.4.1）立即升，bug fix
Minor（12.4 → 12.5）1-2 週後升
Major（12.x → 13.0）等 13.1+ 再升

5.2 升級前檢查

  • 看 release notes 找 breaking changes
  • 備份 internal DB
  • 檢查 plugin 相容性（v12 移除 Angular）
  • Test 環境先試

5.3 K8s Helm 升級

# 看當前版本
helm get values grafana -n monitoring | grep image

# 看 chart 版本
helm search repo grafana/grafana -l

# 升級
helm upgrade grafana grafana/grafana \
  --namespace monitoring \
  --reuse-values \
  --set image.tag=12.5.0

# 看 pod 升級狀態
kubectl rollout status deployment/grafana -n monitoring

# 若有問題，rollback
helm rollback grafana 1 -n monitoring

5.4 滾動升級（HA）

多副本時 K8s 自動滾動：

spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # 升級時可多開 1 個
      maxUnavailable: 0    # 不允許 unavailable
  replicas: 3

5.5 Schema Migration

Grafana 升級時自動執行 schema migration：

  • 查 log 看 migration 進度
  • 大資料庫（million dashboard）可能需要 minutes
  • 不要中斷 migration（會 corrupt DB）

§ 6. 監控 Grafana 自身

6.1 為什麼要 meta-monitoring

Grafana 是您的監控 hub。它掛了 → 您看不到任何 dashboard、alert 也可能掛。

6.2 Grafana 暴露的 metrics

Endpoint: /metrics（Prometheus 格式）

# 在 prometheus.yml
scrape_configs:
  - job_name: 'grafana'
    static_configs:
      - targets: ['grafana:3000']
    metrics_path: /metrics

6.3 必看 Metrics

Metric說明
up{job="grafana"}Grafana 是否活著
grafana_http_request_duration_secondsHTTP 請求 latency
grafana_database_conn_openDB 連線數
grafana_alerting_rule_evaluation_duration_secondsAlert 評估時間
grafana_alerting_rule_evaluations_totalAlert 評估次數
grafana_alerting_active_alerts當前 firing alert 數
grafana_api_response_status_totalAPI response 狀態（依 status code）
grafana_stat_active_usersactive user 數
grafana_stat_total_dashboardsdashboard 總數
process_resident_memory_bytes記憶體使用
go_goroutinesGo routine 數

6.4 必設告警

groups:
  - name: grafana_self
    rules:
      - alert: GrafanaDown
        expr: up{job="grafana"} == 0
        for: 5m
        labels:
          severity: critical

      - alert: GrafanaSlowRequests
        expr: |
          histogram_quantile(0.99,
            rate(grafana_http_request_duration_seconds_bucket[5m])
          ) > 5
        for: 10m
        labels:
          severity: warning

      - alert: GrafanaAlertingFailures
        expr: rate(grafana_alerting_rule_evaluation_failures_total[5m]) > 0
        for: 5m
        labels:
          severity: warning

      - alert: GrafanaDatabaseHighConnections
        expr: grafana_database_conn_open / grafana_database_conn_max_open > 0.8
        for: 10m
        labels:
          severity: warning

      - alert: GrafanaHighErrorRate
        expr: |
          rate(grafana_api_response_status_total{code=~"5.."}[5m])
          / rate(grafana_api_response_status_total[5m])
          > 0.05
        for: 10m
        labels:
          severity: warning

§ 7. 效能調校

7.1 慢的常見原因

症狀常見原因
Dashboard 慢開Datasource 慢 / 太多 panel
Alert evaluation 慢Rule 太複雜 / Datasource 慢
UI 卡太多 dashboard 在 cache
API 慢DB 連線飽和

7.2 Datasource 端調校

  • 用 recording rule 預先聚合複雜 metric
  • Datasource 設 cache_level: High
  • 啟用 incremental querying（v9+）
  • Mimir / Thanos 等 backend 需要自己優化（query frontend、caching）

7.3 Dashboard 調校

  • Panel 不超過 30 個 / dashboard
  • 用 $__rate_interval 而非固定 [5m]
  • 避免「全 cluster scan」query（用 variable filter）
  • Heavy panel 設 relative time 不被全域影響

7.4 DB 調校

# PostgreSQL config
[database]
max_open_conn = 100              # 大規模設高
max_idle_conn = 25
conn_max_lifetime = 14400

7.5 Resource Sizing

規模CPURAM
< 50 user, 100 dashboard1 vCPU1 GB
50-500 user, 1000 dashboard2-4 vCPU2-4 GB
500-5000 user4-8 vCPU8-16 GB
5000+ user多副本（HA）每副本 8-16 GB

7.6 Image Rendering 調校

Grafana 內建 image rendering（給 Slack alert 預覽圖、PDF 報告）。預設用內嵌方式，CPU 重：

# 用獨立 image renderer service（推薦 production）
[rendering]
server_url = http://grafana-image-renderer:8081/render
callback_url = http://grafana:3000/

[rendering.headed]
mode = clustered
clustering_max_concurrency = 5

§ 8. HA 架構

8.1 HA 設計要點

  1. 多副本：3 個 Grafana instance（K8s replica）
  2. 外部 DB：PostgreSQL / MySQL（SQLite 不行）
  3. Shared Plugin Volume：所有副本看到一樣的 plugin（PVC ReadWriteMany 或預載）
  4. Sticky session（可選）：使用者 session 黏在同副本（Live Tail 等需要）
  5. Load Balancer：分流 + health check
  6. External Alertmanager（可選）：避免 Grafana alerting HA 複雜性

8.2 K8s HA Helm Values

replicas: 3

# 外部 DB
grafana.ini:
  database:
    type: postgres
    host: postgres-grafana.internal:5432

# Plugin 預載到 image
plugins:
  - grafana-piechart-panel

# 或 PVC 共享
persistence:
  enabled: true
  type: pvc
  accessModes: [ReadWriteMany]
  storageClassName: efs-sc       # AWS EFS
  size: 10Gi

# Anti-affinity（不同 node）
affinity:
  podAntiAffinity:
    preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:
          topologyKey: kubernetes.io/hostname

# Pod Disruption Budget
podDisruptionBudget:
  minAvailable: 2

8.3 Alerting HA 注意

Grafana Alerting 在 HA 場景需要設定 cluster mode：

[unified_alerting]
ha_listen_address = "${POD_IP}:9094"
ha_advertise_address = "${POD_IP}:9094"
ha_peers = "grafana-0:9094,grafana-1:9094,grafana-2:9094"
ha_peer_timeout = 15s
ha_gossip_interval = 200ms

§ 9. Plugin 管理

9.1 Plugin 的三類

  • Built-in：預裝（Prometheus、Loki、Tempo 等）
  • External - Signed：Grafana Marketplace 下載
  • External - Unsigned：自寫或第三方未簽署

9.2 Plugin Signing

⚠️ Grafana 12+ 預設只允許 signed plugin

未簽署 plugin 在 v12 預設不會載入。要載入需明確設定： 

[plugins]
allow_loading_unsigned_plugins = my-internal-plugin

# 或 image 環境變數
GF_PLUGINS_ALLOW_LOADING_UNSIGNED_PLUGINS=my-internal-plugin

9.3 Angular Plugin 已死

v12 移除 Angular runtime。升級前檢查：

# 查看當前 Angular plugin
curl -H "Authorization: Bearer $TOKEN" \
  $GRAFANA/api/plugins | \
  jq '.[] | select(.angular.detected == true) | .id'

# v12+ 這些 plugin 將不能載入

9.4 Plugin 自動更新

不建議自動更新（可能 breaking change）。手動審慎升級：

# 看可升級
grafana-cli plugins list-versions grafana-piechart-panel

# 確認 changelog
# 升級
grafana-cli plugins update grafana-piechart-panel
sudo systemctl restart grafana-server

§ 10. Troubleshooting Cheatsheet

10.1 常見問題排查

症狀排查步驟
Dashboard 載入很久 1. Query Inspector 看哪個 query 慢
2. Datasource 直接 query 確認
3. 加 recording rule
Datasource Save & Test 失敗 1. 從 Grafana pod ping datasource
2. URL 用 service FQDN
3. 檢查 auth
Alert 沒送出 1. Alert state history
2. Notification policy 路由是否 match
3. Contact point Test
OAuth 登入失敗 1. /login?disableAutoLogin=true 跳到 Basic auth
2. 開 oauth debug log
3. 檢查 callback URL
升級後 panel 顯示錯 1. 檢查是否為 Angular plugin
2. 看 deprecated panel 提示
3. 換新 panel 類型
Provisioning 沒生效 1. 看 Grafana log（路徑、syntax）
2. 確認 file permission
3. /api/admin/provisioning/datasources/reload
K8s 多副本看不同 dashboard 必用外部 DB（不是 SQLite）
