Query Editor、Transformations 與 Explore 模式

Panel 是「視覺化的容器」，但 panel 顯示什麼資料，由 Query Editor + Transformations 決定。 本章涵蓋 Prometheus / Loki / SQL 三種 query editor 介面、24 種 transformations、 以及 ad-hoc debug 利器 Explore 模式。

§ 1. Query Editor 通用結構

1.1 每個 Panel 可有多個 Query

Query A、B、C... 一個 panel 最多支援 26 個 query。Datasource 可以不同（Mixed datasource）。

Query A: rate(http_requests_total[5m])    [Prometheus]
Query B: rate(http_errors_total[5m])      [Prometheus]
Query C: SELECT count(*) FROM orders      [PostgreSQL]

→ Transform: Merge by time
→ Visualization: Time series

1.2 共通操作

操作用途
Hide / ShowQuery 圖示眼睛圖示，臨時不顯示但保留
Duplicate複製為下一個 query 編輯
Toggle help展開官方範例 cheatsheet
Inspector看 raw response、執行時間（debug 必用）

1.3 Query Inspector（debug 神器）

Panel → Edit → 右上角 menu → Query inspector。看到：

  • Stats：query 執行時間、回傳資料量
  • Query：實際送出的 query（含 variable 已展開）
  • JSON / Data：raw 結果
  • Error：詳細錯誤訊息

Panel 顯示 No data？先打開 inspector 看 query 變成什麼樣子。

§ 2. Prometheus Query Editor

2.1 Two Modes：Builder vs Code

Mode適用
Builder新手、UI 拼出 PromQL
Code熟手、直接寫 PromQL

Code mode 是 production 推薦：複雜 query 用 builder 拖不出來。

2.2 重要選項

Metric: 點擊「Metrics browser」可以瀏覽所有 metric
Legend: '{{ method }} {{ status }}'   # 用 label 自訂 legend 名稱
Min step: 留空 = 自動 / 或設 15s（強制 step）
Format: Time series / Heatmap / Table
Type: Range / Instant
Exemplars: ✅（顯示 trace 連結點）

2.3 Min Step 的影響

# Time range = 24h，dashboard width = 1000 pixels
# 預設 step = 24h / 1000 ≈ 86s

# 若 min step = 15s（match Prometheus scrape interval）
# 但實際 step 仍可能更大（保持資料點數量合理）

# 設定原則：
# - 短 time range（< 1h）: 留空，讓 Grafana 計算
# - 大 time range（> 7d）: 設 5m+ 避免過度密集
# - Recording rule 已聚合：min step = 30s+ OK

2.4 Variables 在 PromQL 的用法

# 單值
up{instance="$instance"}

# Multi-value（必用 =~）
up{instance=~"$instance"}

# All option 配合
# variable.allValue = .+
up{instance=~"$instance"}

# 巢狀（cluster → namespace）
container_cpu_usage{cluster="$cluster", namespace=~"$namespace"}

# Variable in PromQL function
rate(http_requests_total[$__rate_interval])

# Group by variable
sum by ($groupby) (rate(...))

2.5 推薦的 PromQL 模式

# ✅ 用 $__rate_interval（v8+）
rate(http_requests_total[$__rate_interval])

# 為什麼比 [5m] 好：
# - 自動配合 dashboard time range
# - 短 range 用小 interval（5x scrape interval 最少）
# - 長 range 用大 interval（避免精度過高）

# ✅ 用 sum by 而非 sum without
sum by (service) (rate(http_requests_total[$__rate_interval]))

# ✅ histogram_quantile + by (le, ...)
histogram_quantile(0.99,
  sum by (le, service) (
    rate(http_request_duration_seconds_bucket[$__rate_interval])
  )
)

§ 3. Loki Query Editor

3.1 LogQL 語法分層

# Layer 1: Log Stream Selector（必有）
{namespace="prod", app="api"}

# Layer 2: Line Filter
{app="api"} |= "error"           # 包含
{app="api"} != "debug"           # 不包含
{app="api"} |~ "5\\d\\d"         # regex
{app="api"} !~ "200|201"         # regex 否定

# Layer 3: Parser
{app="api"} | json
{app="api"} | logfmt
{app="api"} | regexp `(?P\\w+) (?P.+)`

# Layer 4: Label Filter（用解析後 label）
{app="api"} | json | level="error"
{app="api"} | logfmt | duration > 1s

# Layer 5: 變成 metric（aggregation）
sum by (level) (
  rate({app="api"}[5m])
)

3.2 兩種 Loki Query 類型

Type用途結果
Logs看實際 log 行Logs panel
Metric用 LogQL 算出 metricTime series

同一 query「{app="api"} |= "error"」是 Logs；「rate({app="api"} |= "error"[5m])」是 Metric。

3.3 高效 Loki query 原則

⚠️ Loki query 性能關鍵

Loki 是 label-based index，不索引 log 內容。所以：

  1. Stream selector 要盡量精確：{app="api", namespace="prod"} 比 {app=~".*"} 快 10000 倍
  2. Line filter 越早越好：|= 比 |~ regex 快
  3. 解析（json / logfmt）只在必要時用
  4. Label filter 配合 parser 後使用

3.4 範例

# ❌ 慢
{cluster=~".+"} |~ "error"

# ✅ 快
{cluster="prod-tw", app="api", namespace="orders"} |= "error"

# 用 metric query：每 5m 各 service 的 error log 數
sum by (service) (
  count_over_time({namespace="prod"} |= "error" [5m])
)

§ 4. SQL Query Editor

4.1 兩種模式

  • Visual editor：用 dropdown 拖出 SELECT / WHERE / GROUP BY
  • Code：手寫 SQL（推薦）

4.2 Format 選擇

Format適用SQL 要求
Time seriesTime series panel必有 time column 在第一個
TableTable panel、各種彈性任意
LogsLogs panel需 time + message column

4.3 Time Series SQL 範例

-- 必須符合：第一 column = time, 後續 = numeric values
SELECT
  $__timeGroup(created_at, '5m') AS time,
  COUNT(*) FILTER (WHERE status = 'completed') AS completed,
  COUNT(*) FILTER (WHERE status = 'failed') AS failed
FROM orders
WHERE $__timeFilter(created_at)
GROUP BY 1
ORDER BY 1;

4.4 Multi-series（一 column = 一 series）

SELECT
  $__timeGroup(created_at, $__interval) AS time,
  region,                                  ← 變成 series 名稱
  COUNT(*) AS orders
FROM orders
WHERE $__timeFilter(created_at)
GROUP BY 1, region
ORDER BY 1;

-- 結果在 panel 中：
-- 多個 series：region=us, region=eu, region=ap

4.5 SQL Macros 完整參考

Macro展開為
$__timeFilter(col)col BETWEEN '...' AND '...'
$__timeFrom()當前 time range 起時間
$__timeTo()當前 time range 訖時間
$__timeGroup(col, '1h')分組到 1h bucket
$__timeGroupAlias(col, $__interval)分組 + alias 為 time
$__intervalGrafana 算的最佳 interval
$__interval_ms同上 但毫秒
$__unixEpochFilter(col)UNIX 時間戳的 BETWEEN
$__unixEpochGroup(col, '1m')UNIX 時間戳分組

§ 5. Transformations 概念

5.1 Transformations 是什麼

Query 結果到 panel 顯示之間，可以套用 transformations 做後處理。 Transformations 在 Grafana frontend 執行，不影響原 query。 

Query → 結果（DataFrame）→ Transform 1 → Transform 2 → ... → Visualization

5.2 何時用 Transformations

  • Datasource 不支援所需運算（PromQL 沒有 join）
  • 跨 datasource 合併（Mixed datasource）
  • 需要 frontend 動態運算（受 dashboard variable 影響）
  • 過濾、重命名、改格式

5.3 何時不該用 Transformations

  • Datasource 有原生功能可做（PromQL 能做就用 PromQL）
  • 大量資料（>10k rows）— frontend 處理慢
  • 複雜邏輯，難維護

§ 6. 常用 Transformations

Grafana 內建 24+ transformations。本節聚焦最常用 10 個。

6.1 Reduce（聚合到單值）

Reduce 把每個 series 縮為單一值（給 Stat panel 用）

Calculations: Last / Mean / Max / Min / Total / Count / First
Mode: Series to rows / Reduce fields

範例：
原 series:
  api-1: [10, 20, 30, 40, 50]
  api-2: [5, 15, 25, 35, 45]

Reduce by Last → 
  api-1: 50
  api-2: 45

6.2 Filter Data by Name

過濾 fields（columns）：

Include / Exclude
Match: by name / by regex

範例：只保留 "P99" 開頭的 fields
Match: by regex
Regex: ^P99
Action: Include

6.3 Filter Data by Values

過濾 rows：

Filter type: Include / Exclude
Match: All / Any
Conditions:
  - field: error_rate
    operator: greater than
    value: 0.05

→ 只顯示 error_rate > 5% 的 row

6.4 Organize Fields

最常用的 transformation：

  • 重新排序 column
  • 重命名 column（顯示用）
  • 隱藏 column

6.5 Group by

類似 SQL GROUP BY，把 row 按 field 分組並聚合

Group by: service
Aggregate fields:
  qps: sum
  error_rate: mean
  latency: max

6.6 Merge

合併多個 query 結果（適合 Mixed datasource）：

Query A: SELECT service, qps FROM ...
Query B: SELECT service, errors FROM ...

Merge by: service

結果：
service | qps | errors
api     | 1.2K | 50
web     | 800  | 10

6.7 Outer Join

類似 SQL JOIN，連結兩個 dataframe：

Query A: Prometheus  → service, qps
Query B: PostgreSQL → service, customer_count

Join on: service

結果：service | qps | customer_count

6.8 Add Field From Calculation

計算新 field：

Mode: Reduce row / Binary operation / Cumulative function

# 範例：算錯誤率
Mode: Binary operation
Operation: errors / total
Alias: error_rate

6.9 Sort by

Field: latency
Reverse: ✅ (descending)

→ Table 按 latency 倒序顯示

6.10 Time series to table

把 time series 轉成 table 形式（每 series 一 row）：

原:
time | api-1 qps | api-2 qps
10:00| 100       | 50
10:01| 110       | 55

→ Time series to table（取 last value）

結果:
service | qps
api-1   | 110
api-2   | 55

§ 7. Transformation 實戰範例

7.1 範例：Service Health Table（合併 metrics + business 資料）

Query A (Prometheus):
  expr: rate(http_requests_total[$__rate_interval])
  legend: '{{ service }}'
  format: Table
  instant: true

Query B (Prometheus):
  expr: rate(http_requests_total{status=~"5.."}[$__rate_interval])
    / rate(http_requests_total[$__rate_interval])
  format: Table
  instant: true

Query C (PostgreSQL):
  SELECT service, customer_count FROM service_metadata
  format: Table

Transformations:
1. Outer Join by 'service'
2. Organize fields: rename + reorder
3. Filter by Values: error_rate > 0
4. Sort by error_rate desc
5. Limit: 10

結果：Top 10 服務按錯誤率排

7.2 範例：Top 5 + Others 分組

Query: rate(metric[$__rate_interval]) by (service)
       → 100 個 series

# Transform 1: Sort by Last value desc
# Transform 2: Limit to 5
# Transform 3: 留下「其他全部 sum 起來」？
#   Grafana 沒有原生支援，需要 PromQL：

# 替代做法 - 直接在 PromQL 處理
topk(5, sum by (service) (rate(metric[$__rate_interval])))
+
sum(rate(metric[$__rate_interval])) - sum(topk(5, ...))

7.3 範例：用 Calculations 算出 SLO 達成率

Query A: success_total
Query B: total_requests

Transform: Add field from calculation
  Mode: Binary operation
  A / B
  Alias: slo_success_rate

Transform: Format
  Unit: percent (0-1)

Result: panel 顯示 99.95% 之類

§ 8. SQL Expressions（v12+）

8.1 SQL Expressions 是什麼

Grafana 12 引入的新一代 transformations。讓您用真正的 SQL 對 query 結果做處理（在 frontend 跑 SQLite engine）。

8.2 與傳統 Transformations 的差別

面向傳統 TransformationsSQL Expressions
語法UI 拖拉SQL（MySQL-like）
JOIN有限（Outer join 1 對 1）完整 SQL JOIN
複雜運算多 transformation 串接單一 SQL 表達
學習曲線低需懂 SQL
狀態GAv12 experimental

8.3 範例

Query A: Prometheus → service, qps
Query B: PostgreSQL → service, slo_target

SQL Expression（C）:
SELECT
  A.service,
  A.qps,
  B.slo_target,
  CASE WHEN A.qps < B.slo_target THEN 'OK' ELSE 'BREACH' END AS status
FROM A
LEFT JOIN B ON A.service = B.service
WHERE A.qps > 0
ORDER BY A.qps DESC;

強大但仍 experimental，production 用前等 GA。

§ 9. Explore 模式

9.1 Explore 是什麼

左側 sidebar 的 Explore。它是 Grafana 的「互動式查詢」介面， 跟 dashboard 對立——dashboard 是「儲存好的固定 view」，Explore 是「臨時 ad-hoc 查」。

9.2 何時用 Explore

  • Debug 中：寫一個 query 看結果，不想建 dashboard
  • 學 PromQL / LogQL：即時看 query 變化
  • Trace 探索：搜尋特定 trace
  • Log 搜尋：找特定 error 訊息
  • 查看 metric 是否存在

9.3 介面元素

區塊用途
Datasource selector切換 datasource
Time range右上角，類似 dashboard
Query area跟 panel query editor 一樣
Result panel自動選擇 visualization（time series / logs / table）
Query history過去 query 紀錄（可重用）
Star把 query 加入「favorites」

9.4 Explore 的 visualization

不像 dashboard panel 要選 visualization，Explore 自動：

  • Time series query → 顯示折線
  • Logs query → 顯示 Logs panel
  • Trace query → 顯示 Trace 樹
  • Table format → 顯示 Table

9.5 從 Explore 變成 Dashboard

Explore 中的 query 滿意了，可一鍵：

  • Add to dashboard：直接加進 dashboard 變成 panel
  • Share：產生短網址讓別人重現

§ 10. Explore 進階：Split View 與 Correlations

10.1 Split View

把 Explore 切成左右兩欄，同時看不同 datasource：

左：Prometheus  - rate(http_requests_total{...}[5m])
右：Loki        - {namespace="prod", app="api"} |= "error"

可以 sync time range，看「metric spike 時對應的 log」

Split View 是 Explore 最強功能，ad-hoc debug 利器。

10.2 跨 datasource 跳轉

Ch 34 介紹的 Correlations / Derived Fields / Trace-to-Logs 在 Explore 中發揮最大價值：

  • 看 metric → 點 exemplar → 跳到 trace
  • 看 trace → 點 「Logs for span」→ 跳到 log
  • 看 log → 點 trace_id 連結 → 跳到 trace

10.3 Inspector 在 Explore 中

右上角 menu → Inspector，看 raw response（debug API 問題時必用）。

10.4 Live Tail（Loki）

Explore + Loki 支援 Live tail：

Query: {namespace="prod", app="api"}
Click: "Live" 按鈕

→ 即時 stream 新 log 到畫面，類似 tail -f
（適合即時 debug 中的 deploy）

10.5 Explore 的 Query History

Explore → Query history 顯示過去所有 query：

  • 標星（Star）：之後快速取用
  • Comment：記錄為什麼寫這個 query
  • Search：搜過去 query
  • Show only starred：只看 starred

用得好，Query history 是您的「個人 query knowledge base」。
