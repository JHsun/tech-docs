§ 1. 部署方式選擇

方式適用HA 支援
Binary + Systemd傳統機房、PoC需手動設 cluster flag

§ 2. Binary 安裝（Linux + Systemd）

2.1 下載與解壓縮

# 設定變數
AM_VERSION=0.30.1
ARCH=linux-amd64

# 下載
cd /opt
wget https://github.com/prometheus/alertmanager/releases/download/v${AM_VERSION}/alertmanager-${AM_VERSION}.${ARCH}.tar.gz

# 校驗
wget https://github.com/prometheus/alertmanager/releases/download/v${AM_VERSION}/sha256sums.txt
sha256sum -c sha256sums.txt --ignore-missing

# 解壓
tar xzf alertmanager-*.tar.gz
mv alertmanager-${AM_VERSION}.${ARCH} alertmanager

解壓後您會看到：

alertmanager/
├── alertmanager              # 主執行檔
├── amtool                    # CLI 工具（很重要）
├── alertmanager.yml          # 範例設定
├── LICENSE
└── NOTICE

2.2 建立使用者與目錄

sudo useradd --no-create-home --shell /bin/false alertmanager

sudo mkdir -p /etc/alertmanager /var/lib/alertmanager

sudo cp alertmanager amtool /usr/local/bin/
sudo cp alertmanager.yml /etc/alertmanager/

sudo chown -R alertmanager:alertmanager /etc/alertmanager /var/lib/alertmanager
sudo chown alertmanager:alertmanager /usr/local/bin/alertmanager /usr/local/bin/amtool

2.3 Systemd 服務

建立 /etc/systemd/system/alertmanager.service：

[Unit]
Description=Alertmanager
Documentation=https://prometheus.io/docs/alerting/
Wants=network-online.target
After=network-online.target

[Service]
Type=simple
User=alertmanager
Group=alertmanager
ExecStart=/usr/local/bin/alertmanager \
    --config.file=/etc/alertmanager/alertmanager.yml \
    --storage.path=/var/lib/alertmanager \
    --web.listen-address=0.0.0.0:9093 \
    --web.external-url=https://alertmanager.example.com \
    --cluster.listen-address=0.0.0.0:9094 \
    --cluster.peer=alertmanager-2.internal:9094 \
    --cluster.peer=alertmanager-3.internal:9094

ExecReload=/bin/kill -HUP $MAINPID

NoNewPrivileges=true
ProtectSystem=full
ProtectHome=true
PrivateTmp=true
LimitNOFILE=65536

Restart=on-failure
RestartSec=5s

[Install]
WantedBy=multi-user.target
sudo systemctl daemon-reload
sudo systemctl enable alertmanager
sudo systemctl start alertmanager
sudo systemctl status alertmanager

# 重新載入設定
curl -X POST http://localhost:9093/-/reload

# 看 logs
sudo journalctl -u alertmanager -f

§ 4. K8s 部署（kube-prometheus-stack）

4.1 用 Helm 安裝（最簡單）

helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update

helm install kps prometheus-community/kube-prometheus-stack \
  --namespace monitoring \
  --create-namespace

預設會啟動 3 副本 Alertmanager（StatefulSet）+ 自動 HA gossip。

4.2 自訂 values.yaml

alertmanager:
  alertmanagerSpec:
    replicas: 3                            # HA 副本數
    retention: 120h                        # nflog 保留時間

    resources:
      requests:
        cpu: 100m
        memory: 128Mi
      limits:
        memory: 512Mi

    storage:
      volumeClaimTemplate:
        spec:
          accessModes: ["ReadWriteOnce"]
          resources:
            requests:
              storage: 10Gi

  config:
    global:
      slack_api_url: 'https://hooks.slack.com/services/XXX/YYY/ZZZ'

    route:
      group_by: ['alertname', 'cluster']
      group_wait: 30s
      group_interval: 5m
      repeat_interval: 12h
      receiver: 'default'

      routes:
        - matchers:
            - severity = "critical"
          receiver: 'pagerduty'
        - matchers:
            - severity = "warning"
          receiver: 'slack'

    receivers:
      - name: 'default'
        slack_configs:
          - channel: '#alerts'

      - name: 'slack'
        slack_configs:
          - channel: '#alerts-warning'
            send_resolved: true

      - name: 'pagerduty'
        pagerduty_configs:
          - routing_key: 'YOUR_PD_KEY'
helm upgrade kps prometheus-community/kube-prometheus-stack \
  -n monitoring -f values.yaml

4.3 Prometheus Operator AlertmanagerConfig CRD

更模組化的方式：用 AlertmanagerConfig 把每個團隊的設定獨立成 CRD：

apiVersion: monitoring.coreos.com/v1alpha1
kind: AlertmanagerConfig
metadata:
  name: team-platform
  namespace: monitoring
  labels:
    alertmanagerConfig: team-platform
spec:
  route:
    receiver: 'platform-slack'
    matchers:
      - name: team
        value: platform

  receivers:
    - name: 'platform-slack'
      slackConfigs:
        - channel: '#platform-alerts'
          apiURL:
            name: slack-secret
            key: api-url

§ 5. 設定檔結構總覽

完整的 alertmanager.yml 骨架：

global:
  # 全域 receiver 預設值（後續 receiver 可覆蓋）
  smtp_smarthost: 'smtp.example.com:587'
  smtp_from: 'alertmanager@example.com'
  smtp_auth_username: 'alertmanager'
  smtp_auth_password_file: '/etc/alertmanager/secrets/smtp-pass'

  slack_api_url_file: '/etc/alertmanager/secrets/slack-url'
  pagerduty_url: 'https://events.pagerduty.com/v2/enqueue'

  # alert 多久沒更新就視為 resolved
  resolve_timeout: 5m

# 模板檔案路徑
templates:
  - '/etc/alertmanager/templates/*.tmpl'

# 路由樹（Ch 24 詳述）
route:
  group_by: ['alertname', 'cluster']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h
  receiver: 'default'

  routes:
    # ...

# 抑制規則（Ch 24 詳述）
inhibit_rules:
  - source_matchers:
      - severity = "critical"
    target_matchers:
      - severity = "warning"
    equal: ['alertname', 'cluster']

# 時間區間（用於 mute_time_intervals）
time_intervals:
  - name: business_hours
    time_intervals:
      - times:
          - start_time: 09:00
            end_time: 18:00
        weekdays: ['monday:friday']
        location: 'Asia/Taipei'

# Receivers（Ch 25 詳述）
receivers:
  - name: 'default'
    slack_configs:
      - channel: '#alerts'

§ 6. HA 部署：Cluster 與 Gossip

6.1 Cluster 模式運作

多個 Alertmanager 透過 HashiCorp Memberlist（gossip protocol）形成 cluster：

  • 每個 AM 都接收全量 alert（Prometheus 對所有 AM 都送）
  • AM 之間 gossip 同步：silence、notification log（誰已發過通知）
  • 同一個 alert 只有「第一個處理」的 AM 會發通知，其他自動 dedup
  • 單一 AM 掛掉，其他自動接手

6.2 Cluster 設定 flag

# 自身 listen 地址
--cluster.listen-address=0.0.0.0:9094

# 對外宣告地址（NAT 環境很重要）
--cluster.advertise-address=10.0.0.1:9094

# 其他 peer 列表
--cluster.peer=alertmanager-2:9094
--cluster.peer=alertmanager-3:9094

# Gossip 間隔（預設 200ms，多數情況不需改）
--cluster.gossip-interval=200ms

# Push-pull 間隔（預設 1m）
--cluster.pushpull-interval=1m

6.3 Cluster 健康監控

# cluster 中健康節點數
alertmanager_cluster_members

# Gossip 訊息收/送
rate(alertmanager_cluster_messages_received_total[5m])
rate(alertmanager_cluster_messages_sent_total[5m])

# 節點 fail 次數
alertmanager_cluster_members_failed_total

6.4 必設的 cluster 告警

- alert: AlertmanagerClusterDegraded
  expr: alertmanager_cluster_members != 3       # 預期 3 副本
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "Alertmanager cluster member count is {{ $value }}, expected 3"

- alert: AlertmanagerNotificationFailed
  expr: rate(alertmanager_notifications_failed_total[5m]) > 0
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "Alertmanager failed to send notifications"

§ 7. amtool：CLI 完整使用

amtool 是 Alertmanager 內建的命令列工具，跟 alertmanager 一起發布。 它能做幾乎所有 Web UI 能做的事，加上「測試」與「批次」功能。

7.1 設定 amtool 連線

# 方法 1：每次 command 都帶
amtool --alertmanager.url=http://localhost:9093 alert

# 方法 2：寫設定檔（推薦）
mkdir -p ~/.config/amtool
cat > ~/.config/amtool/config.yml << EOF
alertmanager.url: http://localhost:9093
author: $(whoami)
EOF

# 之後直接：
amtool alert

7.2 列出當前 alert

# 所有 active alert
amtool alert

# 過濾
amtool alert --filter='severity=critical'
amtool alert --filter='severity=critical' --filter='team=platform'

# 用 regex
amtool alert --filter='alertname=~".*Down"'

# JSON 輸出
amtool alert -o json

7.3 管理 silence（最常用）

# 建立 silence（2 小時）
amtool silence add alertname=NodeDown instance=api-1 \
  --duration=2h \
  --comment="Investigating disk issue"

# 列出當前 silence
amtool silence query

# 過濾
amtool silence query alertname=NodeDown
amtool silence query --expired   # 看已過期

# 過期某 silence
amtool silence expire 

# 過期所有 NodeDown silence
amtool silence expire $(amtool silence query -q alertname=NodeDown)

# 匯入大量 silence（從檔案）
amtool silence import silences.json

7.4 測試路由配對（極實用）

這是 amtool 最強大的功能 — 不用送真的 alert 就能測試 routing：

# 給定 label，看哪個 receiver 會收到
amtool config routes test \
  --config.file=/etc/alertmanager/alertmanager.yml \
  severity=critical alertname=NodeDown team=platform

# 輸出例如：
# pagerduty-critical

# 看完整 routing tree
amtool config routes \
  --config.file=/etc/alertmanager/alertmanager.yml

# 顯示樹形結構
amtool config routes show \
  --config.file=/etc/alertmanager/alertmanager.yml

7.5 驗證設定檔語法

# 檢查語法錯誤
amtool check-config /etc/alertmanager/alertmanager.yml

# 範例輸出（成功）
# Checking '/etc/alertmanager/alertmanager.yml'  SUCCESS
# Found:
#  - global config
#  - route
#  - 5 inhibit rules
#  - 8 receivers
#  - 2 templates

7.6 模擬通知（測試 template）

# 渲染指定 template
amtool template render \
  --template.glob='/etc/alertmanager/templates/*.tmpl' \
  --template.text='{{ template "slack.title" . }}' \
  --template.data='{
    "Alerts": [{
      "Labels": {"alertname": "NodeDown", "instance": "api-1"},
      "Annotations": {"summary": "Test alert"}
    }]
  }'

§ 8. CLI Flags 重要參數

Flag意義
--config.file主設定檔路徑
--storage.path持久化目錄（nflog、silence）
--data.retention資料保留時間（預設 120h）
--web.listen-addressHTTP 監聽（預設 :9093）
--web.external-url外部訪問的 URL（給通知裡的連結用）
--web.route-prefixURL 前綴（reverse proxy 後面用）
--cluster.listen-addressCluster gossip 端口（預設 :9094）
--cluster.advertise-address對外宣告位址（NAT 必設）
--cluster.peer其他 peer 位址（重複指定）
--alerts.gc-interval清理過期 alert 的頻率
--enable-feature啟用實驗功能（如 utf8-strict-mode）

8.1 --web.external-url 為什麼重要

⚠️ 通知連結到 wrong URL

當您 AM 跑在內網但通知要發到 Slack 給人點，預設會用 http://localhost:9093 作為連結基底—— Slack 收到的人點不開。
設 --web.external-url=https://alertmanager.example.com 讓通知裡的連結正確指向外部存取的 URL。

9.2 API v2 速覽

# GET 所有 active alert

# POST 一個測試 alert（給非 Prometheus 客戶端用）

  {
    "labels": {
      "alertname": "TestAlert",
      "severity": "warning",
      "team": "platform"
    },
    "annotations": {"summary": "Manual test"},

curl http://localhost:9093/api/v2/silences

# POST 建 silence
curl -XPOST http://localhost:9093/api/v2/silences -H 'Content-Type: application/json' -d '{
  "matchers": [

§ 10. 常見啟動問題

⚠️ 問題 1：Permission denied on /var/lib/alertmanager

解法：sudo chown -R alertmanager:alertmanager /var/lib/alertmanager

⚠️ 問題 2：Cluster member 數量是 1（HA 沒生效）

檢查：
• --cluster.peer 設對了嗎？要寫 peer 的位址，不是自己
• --cluster.advertise-address 設了嗎？（NAT 環境必設）
• Port 9094 防火牆通了嗎？
• 看 log 有沒有 cluster member 訊息

⚠️ 問題 3：通知連結點不開

解法：設 --web.external-url=https://alertmanager.example.com

⚠️ 問題 4：Reload 沒生效

檢查：
• 是否真的 POST 到 /-/reload？
• Log 有沒有 reload error？
• 設定檔語法錯誤會阻止 reload — 先用 amtool check-config 驗證

⚠️ 問題 5：每個 AM 都送通知（dedup 沒生效）

原因：cluster 沒形成。檢查上面「問題 2」。
確認：在 /api/v2/status 看到所有 peer。

✅ 本章重點摘要

30 秒回顧
  • K8s 環境用 kube-prometheus-stack 自動 HA；傳統部署用 systemd + cluster flag
  • HA 必設 --cluster.listen-address + --cluster.peer；NAT 環境多加 --cluster.advertise-address
  • HA 用 HashiCorp Memberlist gossip 同步 silence 與 notification log
  • 設定檔結構：global / templates / route / inhibit_rules / time_intervals / receivers
  • --web.external-url 必設，否則通知連結無效
  • amtool CLI 是 ops 必學工具：alert 查詢、silence 管理、路由模擬、設定驗證、template 渲染
  • 關鍵指令：amtool config routes test（不送真 alert 測 routing）
  • 監控 AM 自身：alertmanager_cluster_members、alertmanager_notifications_failed_total
  • 內建 endpoint：/-/healthy、/-/ready、/-/reload、/metrics、/api/v2/...
