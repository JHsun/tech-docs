Chapter 07 · 安裝與啟動

安裝與快速啟動：Binary、Docker、Kubernetes

本章帶您從零跑起 Prometheus。我們會涵蓋三種主要部署方式： Linux 上的 binary 直接安裝、Docker / Compose、Kubernetes（含 Helm 與 Operator）。 每一種都附上可立即執行的指令與最小可運作範例。

§ 1. 部署方式選擇

方式適用情境難度升級維運
Binary + Systemd 傳統機房、單機 / 少量機器、需要完整控制 ★★ 手動腳本
Docker 本機開發、單機 PoC、簡單測試 ★ 容易
Docker Compose 整套 stack（Prom + AM + Grafana）一起跑 ★ 容易
K8s 手動 Manifest 學習用 / 客製化深度需求 ★★★ 自己維護 YAML
kube-prometheus-stack K8s 環境的業界主流選擇 ★★ Helm 升級簡單
Prometheus Operator 多個 Prometheus 實例 / 大規模 K8s ★★★ CRD 宣告式
✅ 選擇建議
  • 個人學習 → Binary 或 Docker，最快上手
  • 小型公司 / 傳統機房 → Binary + Systemd
  • K8s 已經是基礎建設 → kube-prometheus-stack（Helm）
  • 多 cluster / 多 tenant → Prometheus Operator + 自訂 CRD

§ 2. Binary 安裝（Linux）

2.1 下載與解壓縮

# 設定變數（請改為最新版本）
PROM_VERSION=3.11.2
ARCH=linux-amd64   # 或 linux-arm64 / darwin-amd64

# 下載
cd /opt
wget https://github.com/prometheus/prometheus/releases/download/v${PROM_VERSION}/prometheus-${PROM_VERSION}.${ARCH}.tar.gz

# 校驗 sha256（強烈建議）
wget https://github.com/prometheus/prometheus/releases/download/v${PROM_VERSION}/sha256sums.txt
sha256sum -c sha256sums.txt --ignore-missing

# 解壓縮
tar -xzf prometheus-${PROM_VERSION}.${ARCH}.tar.gz
mv prometheus-${PROM_VERSION}.${ARCH} prometheus
cd prometheus

解壓後您會看到：

prometheus/
├── prometheus              # 主執行檔
├── promtool                # 設定驗證 / 查詢工具
├── prometheus.yml          # 預設設定檔
├── consoles/               # 內建 dashboards (legacy)
├── console_libraries/      # console 模板函式庫
├── LICENSE
├── NOTICE
└── tsdb/                   # （資料目錄會在此建立）

2.2 建立專屬使用者與目錄

生產環境千萬不要用 root 跑：

# 建立系統使用者
sudo useradd --no-create-home --shell /bin/false prometheus

# 建立必要目錄
sudo mkdir -p /etc/prometheus /var/lib/prometheus

# 移動檔案
sudo mv prometheus promtool /usr/local/bin/
sudo mv prometheus.yml /etc/prometheus/
sudo mv consoles console_libraries /etc/prometheus/

# 設定擁有者
sudo chown -R prometheus:prometheus /etc/prometheus /var/lib/prometheus
sudo chown prometheus:prometheus /usr/local/bin/prometheus /usr/local/bin/promtool

2.3 第一次啟動

sudo -u prometheus /usr/local/bin/prometheus \
    --config.file=/etc/prometheus/prometheus.yml \
    --storage.tsdb.path=/var/lib/prometheus \
    --web.console.templates=/etc/prometheus/consoles \
    --web.console.libraries=/etc/prometheus/console_libraries \
    --web.listen-address=0.0.0.0:9090 \
    --web.enable-lifecycle

啟動成功後，瀏覽器開 http://localhost:9090 即可看到 Prometheus Web UI。

ℹ️ 重要 CLI flags
  • --config.file：主設定檔位置
  • --storage.tsdb.path：TSDB 資料目錄
  • --storage.tsdb.retention.time：保留時間（預設 15d）
  • --storage.tsdb.retention.size：保留大小上限（例 50GB）
  • --web.listen-address：HTTP 監聽地址
  • --web.enable-lifecycle：啟用 POST /-/reload 與 POST /-/quit
  • --web.enable-admin-api：啟用管理 API（snapshot、tsdb delete 等）
  • --web.enable-otlp-receiver：啟用 OTLP 接收（v2.47+）

§ 3. Systemd 服務化

建立 /etc/systemd/system/prometheus.service：

[Unit]
Description=Prometheus Monitoring System
Documentation=https://prometheus.io/docs/
Wants=network-online.target
After=network-online.target

[Service]
Type=simple
User=prometheus
Group=prometheus
ExecStart=/usr/local/bin/prometheus \
    --config.file=/etc/prometheus/prometheus.yml \
    --storage.tsdb.path=/var/lib/prometheus \
    --storage.tsdb.retention.time=30d \
    --web.console.templates=/etc/prometheus/consoles \
    --web.console.libraries=/etc/prometheus/console_libraries \
    --web.listen-address=0.0.0.0:9090 \
    --web.enable-lifecycle

ExecReload=/bin/kill -HUP $MAINPID

# 安全強化
NoNewPrivileges=true
ProtectSystem=full
ProtectHome=true
PrivateTmp=true

# 資源限制
LimitNOFILE=65536

Restart=on-failure
RestartSec=5s

[Install]
WantedBy=multi-user.target

啟用並啟動服務：

sudo systemctl daemon-reload
sudo systemctl enable prometheus
sudo systemctl start prometheus
sudo systemctl status prometheus

# 重新載入設定（不重啟服務）
curl -X POST http://localhost:9090/-/reload

# 查看 logs
sudo journalctl -u prometheus -f

§ 4. Docker 與 Docker Compose

4.1 純 Docker 啟動

# 最簡單：用預設設定啟動
docker run -d \
  --name prometheus \
  -p 9090:9090 \
  prom/prometheus:v3.11.2

# 帶上自訂設定 + 持久化資料
docker run -d \
  --name prometheus \
  -p 9090:9090 \
  -v $(pwd)/prometheus.yml:/etc/prometheus/prometheus.yml \
  -v prometheus-data:/prometheus \
  prom/prometheus:v3.11.2 \
  --config.file=/etc/prometheus/prometheus.yml \
  --storage.tsdb.path=/prometheus \
  --storage.tsdb.retention.time=30d
ℹ️ Docker image 兩種變體（v3.11+）

Prometheus 3.11 起提供兩種 Docker image：
prom/prometheus:latest-busybox（預設，與 latest 同義）— 使用 busybox base，可進入容器除錯
prom/prometheus:latest-distroless — distroless base，最小、最安全，UID 65532（nonroot）
若從 busybox 遷移到 distroless 需調整資料 volume 的擁有者： docker run --rm -v prom-data:/prometheus alpine chown -R 65532:65532 /prometheus

4.2 Docker Compose（整套 stack）

下面是一個常用的 Compose 範例，把 Prometheus + Node Exporter + Alertmanager + Grafana 全部跑起來：

# docker-compose.yml
version: '3.8'

services:
  prometheus:
    image: prom/prometheus:v3.11.2
    container_name: prometheus
    restart: unless-stopped
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
      - ./alerts:/etc/prometheus/alerts
      - prometheus-data:/prometheus
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.path=/prometheus'
      - '--storage.tsdb.retention.time=30d'
      - '--web.enable-lifecycle'
      - '--web.console.libraries=/usr/share/prometheus/console_libraries'
      - '--web.console.templates=/usr/share/prometheus/consoles'

  node-exporter:
    image: prom/node-exporter:latest
    container_name: node-exporter
    restart: unless-stopped
    ports:
      - "9100:9100"
    volumes:
      - /proc:/host/proc:ro
      - /sys:/host/sys:ro
      - /:/rootfs:ro
    command:
      - '--path.procfs=/host/proc'
      - '--path.sysfs=/host/sys'
      - '--path.rootfs=/rootfs'

  alertmanager:
    image: prom/alertmanager:v0.32.0
    container_name: alertmanager
    restart: unless-stopped
    ports:
      - "9093:9093"
    volumes:
      - ./alertmanager.yml:/etc/alertmanager/alertmanager.yml
      - alertmanager-data:/alertmanager

  grafana:
    image: grafana/grafana-oss:12.4.0
    container_name: grafana
    restart: unless-stopped
    ports:
      - "3000:3000"
    environment:
      - GF_SECURITY_ADMIN_USER=admin
      - GF_SECURITY_ADMIN_PASSWORD=admin
    volumes:
      - grafana-data:/var/lib/grafana

volumes:
  prometheus-data:
  alertmanager-data:
  grafana-data:

對應的最小 prometheus.yml：

global:
  scrape_interval: 15s
  evaluation_interval: 15s

alerting:
  alertmanagers:
    - static_configs:
        - targets: ['alertmanager:9093']

scrape_configs:
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  - job_name: 'node-exporter'
    static_configs:
      - targets: ['node-exporter:9100']

啟動：

docker compose up -d
docker compose ps
docker compose logs -f prometheus

§ 5. Kubernetes：手動 Manifest 部署

雖然實務上會用 Helm 或 Operator，但理解手動 manifest 對學習很有幫助。 下面是最小可運作的 K8s 部署：

5.1 ConfigMap：放 prometheus.yml

apiVersion: v1
kind: ConfigMap
metadata:
  name: prometheus-config
  namespace: monitoring
data:
  prometheus.yml: |
    global:
      scrape_interval: 15s
    scrape_configs:
      - job_name: 'kubernetes-pods'
        kubernetes_sd_configs:
          - role: pod
        relabel_configs:
          - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
            action: keep
            regex: true
          - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_path]
            action: replace
            target_label: __metrics_path__
            regex: (.+)

5.2 ServiceAccount + RBAC（必要）

apiVersion: v1
kind: ServiceAccount
metadata:
  name: prometheus
  namespace: monitoring
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: prometheus
rules:
  - apiGroups: [""]
    resources: ["nodes", "nodes/proxy", "services", "endpoints", "pods"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["networking.k8s.io"]
    resources: ["ingresses"]
    verbs: ["get", "list", "watch"]
  - nonResourceURLs: ["/metrics"]
    verbs: ["get"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: prometheus
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: prometheus
subjects:
  - kind: ServiceAccount
    name: prometheus
    namespace: monitoring

5.3 PersistentVolumeClaim（資料持久化）

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: prometheus-data
  namespace: monitoring
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 50Gi

5.4 Deployment

apiVersion: apps/v1
kind: Deployment
metadata:
  name: prometheus
  namespace: monitoring
spec:
  replicas: 1
  selector:
    matchLabels:
      app: prometheus
  template:
    metadata:
      labels:
        app: prometheus
    spec:
      serviceAccountName: prometheus
      containers:
        - name: prometheus
          image: prom/prometheus:v3.11.2
          args:
            - '--config.file=/etc/prometheus/prometheus.yml'
            - '--storage.tsdb.path=/prometheus'
            - '--storage.tsdb.retention.time=30d'
            - '--web.enable-lifecycle'
          ports:
            - containerPort: 9090
          resources:
            requests:
              cpu: 500m
              memory: 2Gi
            limits:
              cpu: 2000m
              memory: 8Gi
          volumeMounts:
            - name: config
              mountPath: /etc/prometheus
            - name: data
              mountPath: /prometheus
      volumes:
        - name: config
          configMap:
            name: prometheus-config
        - name: data
          persistentVolumeClaim:
            claimName: prometheus-data

5.5 Service

apiVersion: v1
kind: Service
metadata:
  name: prometheus
  namespace: monitoring
spec:
  selector:
    app: prometheus
  ports:
    - port: 9090
      targetPort: 9090
  type: ClusterIP

套用：kubectl apply -f .，然後 kubectl port-forward -n monitoring svc/prometheus 9090 即可從本機存取。

§ 6. Kubernetes：kube-prometheus-stack（Helm）

在 K8s 環境中，kube-prometheus-stack 是業界事實標準。它提供：

  • Prometheus Operator（管理 CRDs）
  • Prometheus（含 K8s 標準告警規則）
  • Alertmanager
  • Grafana（含預設 dashboards）
  • node-exporter（DaemonSet）
  • kube-state-metrics（K8s 物件狀態）
  • 所有必要的 ServiceMonitor

6.1 安裝步驟

# 加入 Helm repo
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update

# 建立 namespace
kubectl create namespace monitoring

# 安裝（最小設定）
helm install kps prometheus-community/kube-prometheus-stack \
  --namespace monitoring \
  --create-namespace

6.2 自訂 values.yaml

# values.yaml 常用客製化
prometheus:
  prometheusSpec:
    retention: 30d
    retentionSize: "100GB"
    resources:
      requests:
        cpu: 500m
        memory: 4Gi
      limits:
        memory: 8Gi
    storageSpec:
      volumeClaimTemplate:
        spec:
          accessModes: ["ReadWriteOnce"]
          resources:
            requests:
              storage: 200Gi
    # 抓取所有 namespace 的 ServiceMonitor
    serviceMonitorSelectorNilUsesHelmValues: false
    podMonitorSelectorNilUsesHelmValues: false

grafana:
  adminPassword: "ChangeMe123!"
  persistence:
    enabled: true
    size: 10Gi

alertmanager:
  alertmanagerSpec:
    storage:
      volumeClaimTemplate:
        spec:
          accessModes: ["ReadWriteOnce"]
          resources:
            requests:
              storage: 10Gi
# 套用客製化
helm install kps prometheus-community/kube-prometheus-stack \
  -n monitoring -f values.yaml

# 升級
helm upgrade kps prometheus-community/kube-prometheus-stack \
  -n monitoring -f values.yaml

# 查看狀態
kubectl get pods -n monitoring
kubectl get prometheus,alertmanager,servicemonitors -n monitoring

6.3 存取各服務

# Prometheus
kubectl port-forward -n monitoring svc/kps-kube-prometheus-stack-prometheus 9090

# Grafana（admin / 上面設的密碼）
kubectl port-forward -n monitoring svc/kps-grafana 3000:80

# Alertmanager
kubectl port-forward -n monitoring svc/kps-kube-prometheus-stack-alertmanager 9093

§ 7. 系統需求與資源規劃

7.1 基本需求

規模Active SeriesCPURAMDisk（30 天）
實驗 / PoC~10k0.5 核2 GB10 GB
小型生產~100k1-2 核4-8 GB50 GB
中型~1M2-4 核16 GB200 GB
大型~10M8-16 核64 GB1-2 TB
更大10M+建議改用長期儲存方案（Mimir/VictoriaMetrics/Thanos）

7.2 估算 series 數的方式

啟動後查詢：

# 當前 active series 數
prometheus_tsdb_head_series

# 每秒接收 sample 數
rate(prometheus_tsdb_head_samples_appended_total[5m])

# 記憶體用量
process_resident_memory_bytes{job="prometheus"}

# 磁碟用量
prometheus_tsdb_storage_blocks_bytes

7.3 RAM 經驗法則

  • 每 100k active series 大約佔 1-2 GB RAM（不含查詢與快取）
  • 查詢大範圍時間（例 30 天）可能瞬間用 8-16 GB
  • 建議 RAM = 估算 ingestion 用量 × 2-3 倍

7.4 磁碟經驗法則

  • 每 sample 約 1.5-3 bytes（壓縮後）
  • 每 series 每秒 1 個 sample × 86400 秒 × 30 天 × 2 bytes ≈ 5 MB / series / 月
  • 1M series × 30 天 ≈ 150-200 GB
  • 務必用 SSD，HDD 在 compaction 階段會非常吃力

§ 8. 健康檢查與基本驗證

8.1 內建健康端點

端點用途
/-/healthyK8s liveness probe（200 = 活著）
/-/readyK8s readiness probe（200 = 可接受查詢）
/-/reloadPOST 觸發設定重新載入（要 --web.enable-lifecycle）
/-/quitPOST 優雅關閉（要 --web.enable-lifecycle）
/metricsPrometheus 自身的 metrics（自我監控）

8.2 第一個必跑查詢

在 Web UI 的 Graph 頁面執行：

# 確認 Prometheus 自己活著
up

# 看所有 targets 狀態
sum by (job) (up)

# 看 ingestion 速率
rate(prometheus_tsdb_head_samples_appended_total[5m])

# 看當前 series 數量
prometheus_tsdb_head_series

8.3 用 promtool 驗證設定

# 驗證主設定檔
promtool check config /etc/prometheus/prometheus.yml

# 驗證 alerting rules
promtool check rules /etc/prometheus/alerts/*.yml

# 測試 alerting rule（單元測試）
promtool test rules tests.yml

§ 9. 常見啟動問題排除

⚠️ 問題 1：Permission denied on /var/lib/prometheus

原因：使用者沒有 TSDB 目錄的寫入權限。
解法：sudo chown -R prometheus:prometheus /var/lib/prometheus

⚠️ 問題 2：Address already in use (port 9090)

原因：9090 被佔用（可能是另一個 Prometheus、或某個 K8s service）。
解法：--web.listen-address=0.0.0.0:9091 換 port，或 lsof -i :9090 找出占用程式。

⚠️ 問題 3：Out of memory（OOM）

原因：series 數量超出記憶體預期。
解法：
• 立刻：增加 RAM 或加上 K8s memory limit
• 短期：用 topk(20, count by (__name__)(...)) 找出兇手 metrics
• 長期：找出高基數 label，調整 relabeling 移除

⚠️ 問題 4：K8s SD 抓不到 targets

原因：通常是 RBAC 不足。
解法：確認 ServiceAccount 有 nodes, pods, endpoints, services 的 list/watch 權限。

⚠️ 問題 5：Docker 持久化資料消失

原因：忘記 mount volume，容器重啟後 TSDB 全沒了。
解法：-v prometheus-data:/prometheus（named volume）或 -v $(pwd)/data:/prometheus（host bind）。

📝 練習題

  1. 請依本章步驟，在您的本機（Linux / macOS / WSL）用 binary 或 Docker 跑起一個 Prometheus，並能在瀏覽器看到 Web UI。
  2. 修改 prometheus.yml 加入抓取自己 metrics 的 scrape config，重新載入後在 UI 確認看得到自身的 up metric。
  3. 計算：若您的環境預期 active series 約 50 萬，scrape interval 15 秒，計算 30 天大約需要多少磁碟空間？需要多少 RAM？
  4. 對 systemd 啟動的 Prometheus，請寫出三條指令：(a) 重新載入設定但不重啟；(b) 查看最近 100 行 log；(c) 確認服務狀態。
