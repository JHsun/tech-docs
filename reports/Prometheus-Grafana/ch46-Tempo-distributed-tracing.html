§ 1. 分散式 Tracing 入門

1.1 為什麼要 tracing

在微服務環境，單一 user 請求可能經過 10+ 個 service。當「結帳很慢」時：

  • Metric 告訴您「平均 1.5s」
  • Log 告訴您「order-service 在處理 12345」
  • 但哪個 service / 哪個 DB query / 哪個外部 API 慢？

Distributed tracing 解決：把請求路徑完整串起來，每個跨 service 操作叫 span， 所有 span 合起來叫 trace。

1.2 核心概念

概念說明
Trace一個完整 request 的完整路徑（一棵樹）
Span樹中一個節點，代表一個操作（HTTP call、DB query、function）
Trace ID整個 trace 的唯一 ID（128-bit hex）
Span ID單一 span 的唯一 ID（64-bit hex）
Parent Span ID形成樹結構
Attributesspan 的 key-value（http.method, db.statement）
Eventsspan 內的時間點事件（log style）
StatusOK / Error
Durationspan 執行時間

1.3 一個 trace 範例

Trace: checkout-12345
└── frontend.handleCheckout (200ms total)
    ├── auth-service.verify (10ms)
    ├── cart-service.getCart (50ms)
    │   └── database.query (40ms)
    ├── payment-service.charge (120ms)  ← 瓶頸
    │   ├── external-api.stripe (110ms)
    │   └── database.recordPayment (5ms)
    └── notification-service.send (15ms)

看 trace 立刻知道：「payment-service 的 stripe 調用是瓶頸」。

1.4 OpenTelemetry：標準化 Tracing

Tracing 的標準是 OpenTelemetry (OTel)。它提供：

  • SDK（自動 / 手動 instrument 應用）
  • OTLP 協議（傳送格式）
  • Collector（接收、處理、匯出）
  • Vendor-neutral（可換 backend，code 不變）

Tempo 原生支援 OTLP（Ch 04 介紹過 OTel）。

§ 2. 為什麼需要 Tempo

2.1 既有方案的痛點

方案痛點
Jaeger（all-in-one） 需要 Cassandra / ES 當後端，運維複雜
Zipkin 類似 Jaeger 痛點，社群活躍度低
商業 SaaS（Datadog APM 等） per-host / per-trace 收費，scale 後極貴

2.2 Tempo 的設計

💡 核心洞察

Jaeger / Zipkin 都把 trace 完整 indexed（service name、operation、tags），index 巨大。
Tempo 觀察：「實務上，trace ID 通常從 log / metric exemplar 拿到」（不是搜出來）。
所以：
• 不索引，只用 trace ID 取
• Trace 直接壓縮存 object store
• 想搜 trace？從 metric exemplar 跳，或用 TraceQL（v2+ 補上）

2.3 成本對比

同樣 1B span/day：

  • Jaeger + Cassandra：$10000+/月
  • Tempo + S3：$1000-2000/月

2.4 Tempo 的取捨

v1 時 Tempo 完全沒有「搜尋」（只能用 trace ID 取）。v2 加上 TraceQL 補上搜尋功能， 但仍比 Jaeger 慢（必要時可加 Tempo metrics generator 補足常見 query pattern）。

§ 3. 架構與元件

3.1 元件架構圖

graph LR APP[Application] OTEL[OTel Collector · Alloy] DIST[Distributor] INGEST[Ingester] OBJ[(Object Storage
S3 · GCS)] QF[Query Frontend] QUERIER[Querier] METGEN[Metrics Generator] PROM[Prometheus · Mimir] USER[User · Grafana] APP --> OTEL OTEL -- OTLP --> DIST DIST -- shards --> INGEST DIST -- copy --> METGEN INGEST -- write blocks --> OBJ METGEN -- write metrics --> PROM USER -- TraceQL · Trace ID --> QF QF -- splits --> QUERIER QUERIER -- read blocks --> OBJ QUERIER -- read recent --> INGEST

3.2 主要元件

元件職責
Distributor 接收 OTLP / Jaeger / Zipkin push、validation、shard 到 ingester
Ingester 記憶體中累積 trace blocks、定期 flush 到 object store
Query Frontend 切分 TraceQL 查詢、cache、平行化
Querier 從 ingester（最近）+ object store（歷史）讀資料
Compactor 合併小 blocks，提升 query 效能
Metrics Generator 從 trace 自動算 RED metrics + service graph，寫到 Prometheus

架構與 Loki 高度一致（同一團隊設計），運維知識可重用。

3.3 與 Loki 的相似度

  • 同樣 Distributor / Ingester / Querier / Frontend / Compactor 五大元件
  • 同樣 object store + ingester memory 兩段儲存
  • 同樣 multi-tenant via header
  • 同樣三種部署模式（monolithic / simple scalable / microservices）

§ 4. Trace 接收協議

4.1 支援的協議

協議狀態建議
OTLP gRPC（4317） OpenTelemetry 標準 ✅ 推薦
OTLP HTTP（4318） OpenTelemetry HTTP ✅ 防火牆友善
Jaeger gRPC（14250）/ HTTP（14268） 傳統 Jaeger 遷移期使用
Zipkin（9411） 傳統 Zipkin 遷移期使用

4.3 OTel Collector / Alloy 在中間

實務常見架構：

App (instrumented)
  ↓ OTLP
OTel Collector / Alloy
  ↓ OTLP
Tempo

中間層好處：

  • Buffering（Tempo 暫掛時不丟資料）
  • Sampling（先在 collector 過濾）
  • Enrichment（加 cluster / region label）
  • Multi-backend（同 trace 送 Tempo + 商業 APM）

§ 5. TraceQL 完整語法

5.1 基本概念

TraceQL 是 Tempo v2+ 引入的查詢語言，類似 PromQL 風格但給 trace。 用 {} 篩選 span，回傳整個 trace（含所有 span）。

5.2 基本 selector

# 找特定 service 的 trace
{ resource.service.name = "api" }

# Duration filter
{ duration > 500ms }
{ duration < 100ms }

# 多條件 AND
{ resource.service.name = "api" && duration > 500ms }

# Error filter
{ status = error }

# HTTP status
{ span.http.status_code >= 500 }

# Resource 與 span attribute 區分
# resource = service-level（pod / region / version）
# span = operation-level（http.method / db.statement）

5.3 屬性命名空間

Prefix意義
resource.Service-level（service.name, k8s.cluster.name）
span.Span-level（http.method, db.system）
nameSpan 名稱（operation）
durationSpan 持續時間
statusOK / error
kindserver / client / internal / producer / consumer

5.4 Span 內條件

# 任一 span 符合條件
{ span.http.status_code = 500 }

# 多個 span 同時符合（AND in same span）
{ span.http.method = "POST" && span.http.status_code = 500 }

# 不同 span 符合（structural query）
{ resource.service.name = "frontend" } >> { span.http.status_code = 500 }
# >> 表「祖先 → 後代」關係

5.5 結構性查詢

# 直接父子關係
{ ... } > { ... }

# 任意祖先後代
{ ... } >> { ... }

# 兄弟（同 parent）
{ ... } ~ { ... }

# 範例：「frontend 服務的請求中，payment-service 出錯」
{ resource.service.name = "frontend" } >> { resource.service.name = "payment" && status = error }

5.6 Aggregation（v2.3+）

# Trace 數量
count_over_time({ resource.service.name = "api" }[5m])

# Average duration
avg_over_time({ resource.service.name = "api" } | avg(duration))

# Quantile
quantile_over_time({ resource.service.name = "api" } | quantile(duration, 0.99))

5.7 與 Drilldown 整合

Ch 39 介紹的 Traces Drilldown 自動產生 RED 卡片， 讓您不用寫 TraceQL 也能探索。Drilldown 用底層 metrics generator 寫的 RED metric 來做。

§ 6. Metrics Generator

6.1 為什麼需要

沒有 metrics generator 時，要算「service 的 RED 指標」必須跑 TraceQL aggregation，慢。
有了 metrics generator，Tempo 自動從每個 span 算 RED metric，寫到 Prometheus。
Dashboard 用便宜的 metric 而非每次重算。

6.2 啟用

# tempo.yaml
metrics_generator:
  registry:
    external_labels:
      source: tempo
      cluster: production

  storage:
    path: /tmp/tempo/generator/wal
    remote_write:
      - url: http://prometheus:9090/api/v1/write
        send_exemplars: true

  processor:
    span_metrics:
      dimensions:
        - http.method
        - http.status_code
        - http.route
      filter_policies:
        - include:
            match_type: strict
            attributes:
              - key: resource.service.name
                value: ".*"

    service_graphs:
      enabled: true
      dimensions:
        - cluster

# Per-tenant override
overrides:
  defaults:
    metrics_generator:
      processors:
        - service-graphs
        - span-metrics
      collection_interval: 15s

6.3 自動產生的 metric

啟用 span-metrics 後，自動寫到 Prometheus：

traces_spanmetrics_calls_total{
  service="api",
  span_name="GET /api/orders",
  status_code="STATUS_CODE_OK",
  http_method="GET"
}

traces_spanmetrics_latency_bucket{
  service="api",
  span_name="GET /api/orders",
  le="0.1"
}

traces_spanmetrics_size_total{
  service="api"
}

6.4 Service Graph

啟用 service_graphs 後，產生「服務拓樸」metric：

traces_service_graph_request_total{
  client="frontend",
  server="api",
  connection_type="virtual_node"
}

traces_service_graph_request_failed_total{...}

traces_service_graph_request_server_seconds_bucket{...}

Grafana 自動畫出服務拓樸（Tempo datasource → Service Graph view）。

6.5 Cardinality 控制

⚠️ Span Metrics Cardinality 警告

dimensions 加越多 → metric series 越多。
範例：service × span_name × http.method × http.status_code × http.route
若 http.route 有 1000 種、http.status_code 6 種，single service 就 6000 series。
解法：
• 用 http.route（template path）不是 http.target（含 query string）
• 限制 dimensions 數量
• 用 filter_policies 排除高 cardinality 服務

§ 7. Sampling 策略

7.1 為什麼要 sampling

1B span/day 全存太貴。sampling 在 collector 端做，只送一部分 trace。

7.2 Head-based Sampling

在 trace 開始時就決定送不送（用 trace ID hash 機率採樣）：

# OTel Collector / Alloy
processors:
  probabilistic_sampler:
    sampling_percentage: 10    # 10%

# 或用 trace ID 一致性
processors:
  probabilistic_sampler:
    hash_seed: 22
    sampling_percentage: 10

優點：簡單、低 overhead
缺點：可能漏掉重要的 error trace

7.3 Tail-based Sampling

等 trace 完成才決定（基於 status / duration / attribute）：

processors:
  tail_sampling:
    decision_wait: 10s
    num_traces: 100000
    expected_new_traces_per_sec: 100

    policies:
      # 永遠保留 error
      - name: errors
        type: status_code
        status_code: { status_codes: [ERROR] }

      # 永遠保留慢的
      - name: slow
        type: latency
        latency: { threshold_ms: 1000 }

      # 其他 10%
      - name: random
        type: probabilistic
        probabilistic: { sampling_percentage: 10 }

優點：保留有價值的 trace
缺點：collector 需要 buffer 等 trace 完成，記憶體大

7.4 推薦策略

  • 小規模：100% 送（< 1M span/day）
  • 中規模：head-based 50%（1M-100M）
  • 大規模：tail-based + 智能策略

§ 9. 與三本柱整合

9.1 完整觀測性 stack

App
  ↓ instrumented (OTel SDK)
OTel Collector / Alloy
  ├── Metrics → Prometheus / Mimir
  ├── Logs → Loki
  └── Traces → Tempo
                ↓ metrics generator
                Prometheus（RED metric + service graph）

Grafana 連所有三家：
  - Metrics → Prometheus
  - Logs → Loki
  - Traces → Tempo

9.2 串接設定（回顧 Ch 34）

# Tempo datasource
jsonData:
  # Trace → Logs
  tracesToLogsV2:
    datasourceUid: loki-prod
    spanStartTimeShift: '-1m'
    spanEndTimeShift: '1m'
    tags:
      - { key: 'service.name', value: 'service' }
    customQuery: true
    query: '{service="$${__span.tags.service}"} |= "$${__trace.traceId}"'

  # Trace → Metrics
  tracesToMetrics:
    datasourceUid: prometheus-prod
    queries:
      - name: 'Request rate'
        query: 'sum(rate(traces_spanmetrics_calls_total{$$__tags}[5m]))'

  # Trace → Profiles
  tracesToProfiles:
    datasourceUid: pyroscope-prod
    profileTypeId: 'process_cpu:cpu:nanoseconds:cpu:nanoseconds'

  # Service Graph
  serviceMap:
    datasourceUid: prometheus-prod

  # Node Graph
  nodeGraph:
    enabled: true

9.3 完整 debug 流程（回顧 Ch 39 場景）

  1. Metric 飆 → 點 exemplar 進 trace
  2. Trace 看到 payment-service 慢 → 點 "Logs for span" 進 Loki 看 log
  3. Log 看到 "DB timeout" → 點 trace_id link 回 trace 看其他 span
  4. Trace 點 "Profile for span" → 進 Pyroscope 看 CPU

  • Production：tail-based sampling，error 與慢 trace 100% 留
  • 用 OTel Collector / Alloy 做 sampling，不在 app 端
  • 定期檢視 sample rate 與 storage 用量

§ 11. 從 Jaeger / Zipkin 遷移

11.1 為什麼遷移

  • 成本（Tempo 1/10）
  • 運維（Tempo 不需要 Cassandra / ES）
  • 整合（與 Loki / Mimir 同 ecosystem）
  • OpenTelemetry 標準化趨勢

11.2 遷移流程

  1. 部署 Tempo（與 Jaeger 並行）
  2. App 端用 OTel SDK / collector 同時送 Tempo + Jaeger
  3. Grafana 加 Tempo datasource，驗證可用
  4. 切換 Grafana 預設 trace datasource 到 Tempo
  5. 觀察一週確認穩定
  6. 停用 Jaeger Cassandra / ES
  7. 遷移舊 trace（可選；通常不必）

11.3 OTel Collector 雙寫範例

exporters:
  otlp/tempo:
    endpoint: tempo:4317
    tls:
      insecure: true

  jaeger:
    endpoint: jaeger-collector:14250
    tls:
      insecure: true

service:
  pipelines:
    traces:
      receivers: [otlp]
      exporters: [otlp/tempo, jaeger]   # 雙寫

11.4 Code 不用改

若 app 用 OTel SDK 送 OTLP 到 collector，切後端 0 code 修改。 這是 OTel 的核心價值：vendor-neutral instrumentation。
