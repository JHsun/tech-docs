Chapter 17 · Federation / Remote Storage

Federation 與 Remote Storage 全攻略

單一 Prometheus 在 ~10M active series 後就會吃力。 解決方案有兩條路：Federation（讓多個 Prometheus 形成階層） 與 Remote Storage（把資料推到 Mimir/Thanos/VictoriaMetrics 等長期儲存）。 本章教您兩者的時機、架構、設定與選型。

§ 1. 為什麼單機 Prometheus 不夠

單一 Prometheus 達到以下情境就需要擴展：

瓶頸典型門檻
Active series單機 ~10M（取決於 RAM）
Ingestion rate~1M samples/sec（單核）
磁碟 retention本地 30-60 天就吃掉 TB 級磁碟
查詢效能跨 30+ 天查詢開始痛苦
HA 需求單實例 = 單點故障
多租戶原生不支援

1.1 兩條擴展路線

方案適用複雜度
Federation 跨 Prometheus 集中查 / 跨 cluster 看總覽 ★★（純 Prometheus 即可）
Remote Storage 長期儲存、HA、多租戶、超大規模 ★★★（要部署 Mimir/Thanos）

§ 2. Federation 概念與兩種模式

Federation 的核心：讓一個 Prometheus 從另一個 Prometheus 抓 metric。 它就像「Prometheus 之間的 scraping」，使用特殊的 /federate endpoint。

2.1 兩種典型模式

模式 A：Hierarchical（階層式）

多個 cluster-level Prometheus → 一個 global Prometheus。

模式 B：Cross-Service（跨服務）

一個 Prometheus 從另一個拿特定 metric（同等級）。少見，多被 Mimir/Thanos 取代。

§ 3. Hierarchical Federation 設定

3.1 Source Prometheus（被抓的）

不需要特別設定，/federate endpoint 預設啟用。但強烈建議用 recording rules 預先聚合：

# cluster-level Prometheus 的 recording rules
groups:
  - name: federation_aggregations
    interval: 30s
    rules:
      # 聚合掉高基數 label，只保留 cluster/job 級別
      - record: cluster_job:http_requests:rate5m
        expr: sum by (cluster, job, status)(rate(http_requests_total[5m]))

      - record: cluster_job:cpu_usage:avg
        expr: avg by (cluster, job)(rate(node_cpu_seconds_total{mode!="idle"}[5m]))

3.2 Global Prometheus（拉取者）

# global prometheus.yml
scrape_configs:
  - job_name: 'federate'
    scrape_interval: 30s
    honor_labels: true             # ← 必設！保留 source 的 job/instance label
    metrics_path: '/federate'

    params:
      'match[]':                   # 要拉哪些 series
        - '{__name__=~"cluster_job:.+"}'      # 只拉 recording rule 結果
        - '{__name__="up", job="prometheus"}' # 拉 up metric
        - '{job="kubernetes-cadvisor"}'       # 整個 job

    static_configs:
      - targets:
          - 'prometheus-tw.example.com:9090'
        labels:
          source_cluster: tw
      - targets:
          - 'prometheus-jp.example.com:9090'
        labels:
          source_cluster: jp

3.3 重要設定解析

設定意義
honor_labels: true 關鍵！保留 source Prometheus 已經設好的 job/instance label，否則會被改成 exported_job
match[] 用 PromQL selector 語法挑選要拉的 series；務必精確，不要 {__name__=~".+"} 全拉
scrape_interval 建議大於 source 的 interval（例：source 15s、federation 30s）

3.4 測試 federation endpoint

# 直接用 curl 看 source Prometheus 會吐什麼
curl -G http://prometheus-tw.example.com:9090/federate \
  --data-urlencode 'match[]={__name__=~"cluster_job:.+"}' \
  --data-urlencode 'match[]={job="prometheus"}'

§ 4. Cross-Service Federation

某個服務的 Prometheus 想看「另一個服務的某個 metric」（少見，但偶爾用於跨團隊整合）： 

# Service A 的 Prometheus 拉 Service B 的特定 metric
scrape_configs:
  - job_name: 'service-b-shared-metrics'
    honor_labels: true
    metrics_path: '/federate'
    params:
      'match[]':
        - '{__name__="b_service_health_score"}'
        - '{__name__="b_service_cost_per_request"}'
    static_configs:
      - targets: ['prometheus-service-b:9090']
⚠️ Cross-Service Federation 通常不是好主意

它建立服務間耦合（B Prometheus 掛了 A 也少了資料）。
更好的做法是共用一個 remote storage，所有服務寫入同個後端，所有 Grafana 從那查。

§ 5. Federation 的限制

Federation 看似簡單，但有以下重要限制：

🚨 Federation 的 5 個天生限制
  1. 不能傳遞所有資料：每個 federation request 預設限制 --web.max-connections， 且整個 response 是一個 HTTP body，太大會失敗。實務上單次能拉 5-10 萬條 series 已是上限。
  2. 會有採樣對齊問題：global Prometheus 每 30s 抓一次，若 source 是 15s 採樣，會「跳格」遺失資料點。
  3. 不適合「拉所有 raw data」：應只拉 recording rule 結果，不要拉原始 metric。
  4. HA 不友善：兩個 global Prometheus 各自做一份 federation，重複計算。
  5. 查詢仍受限於 global Prometheus 容量：如果 global 也撐不住，問題沒解。

這些限制讓 federation 適合「跨 cluster 看 summary dashboard」， 不適合「全公司統一存儲查詢平台」（後者該用 remote storage）。

§ 6. Remote Write：架構與運作

6.1 概念圖

retention 短期 ~7d] QUEUE[Remote Write Queue

VictoriaMetrics /

查歷史]

6.2 運作流程

  1. Prometheus 把 sample 寫入本地 TSDB（保留短期，例 7 天）
  2. 同時，每個 sample 進入記憶體中的 remote write queue
  3. Queue 按 batch 累積 sample（預設 500 個或 5 秒）
  4. 批次 protobuf 編碼 + Snappy 壓縮，HTTP POST 到後端
  5. 後端回 200 = 成功；429 = 限流（Prometheus 退避重試）；其他錯誤 = 重試或丟棄

6.3 與 Federation 的差別

面向FederationRemote Write
方向Pull（global 主動拉）Push（Prometheus 主動推）
頻率定期（30s 一次）串流（持續推）
資料量受限於 HTTP response無上限（按佇列調配）
後端必須是另一個 Prometheus任何相容 Remote Write API 的服務
長期儲存不適合主要用途

§ 7. Remote Write 設定深度

7.1 完整設定範例

remote_write:
  - url: 'https://mimir.example.com/api/v1/push'
    name: 'mimir-prod'
    remote_timeout: 30s

    # 多租戶（Mimir/Cortex）
    headers:
      X-Scope-OrgID: 'tenant-platform'

    # 認證
    basic_auth:
      username: 'prometheus'
      password_file: '/etc/prometheus/secrets/mimir-pass'

    # TLS
    tls_config:
      ca_file: '/etc/prometheus/ca.pem'
      insecure_skip_verify: false

    # 過濾要送的 series
    write_relabel_configs:
      - source_labels: [__name__]
        regex: 'go_(gc|memstats)_.*'
        action: drop

    # 佇列調校（很重要！）
    queue_config:
      capacity: 10000              # 記憶體中佇列大小
      max_shards: 30               # 並發送出的 shard 數
      min_shards: 1
      max_samples_per_send: 2000   # 一次 batch 多少 sample
      batch_send_deadline: 5s      # 多久強制 flush
      min_backoff: 30ms
      max_backoff: 5s
      retry_on_http_429: true      # 429 時重試（建議啟用）

    # 啟用 RW 2.0（v2.50+ experimental）
    protobuf_message: 'io.prometheus.write.v2.Request'

7.2 Queue 調校關鍵參數

參數預設調整時機
capacity 10000 後端慢時加大避免 sample 丟失
max_shards 50 送不過去時自動加 shard，吞吐瓶頸時上調
max_samples_per_send 2000 後端能吃多大 batch（Mimir 預設 5000 OK）
batch_send_deadline 5s 低延遲場景縮短，高吞吐場景拉長
retry_on_http_429 false 建議開（後端限流時自動退避）

7.3 監控 Remote Write

# 佇列中等待 sample 數
prometheus_remote_storage_samples_pending

# 已成功送出
rate(prometheus_remote_storage_samples_sent_total[5m])

# 失敗
rate(prometheus_remote_storage_samples_failed_total[5m])

# 重試
rate(prometheus_remote_storage_samples_retried_total[5m])

# 丟棄（佇列爆滿時）
rate(prometheus_remote_storage_samples_dropped_total[5m])

# 當前 shard 數
prometheus_remote_storage_shards

# 與後端連線健康
prometheus_remote_storage_string_interner_zero_reference_releases_total

# 端到端延遲（送出時間 vs sample 產生時間）
prometheus_remote_storage_queue_highest_sent_timestamp_seconds
- prometheus_remote_storage_queue_oldest_unsent_timestamp_seconds

7.4 Remote Write 1.0 vs 2.0

面向RW 1.0RW 2.0（v2.50+）
狀態穩定Experimental（仍演進）
Native Histogram需 client 啟用原生支援
Metadata（Help / Type）單獨送（延遲幾秒）內嵌每筆 series（即時）
Created Timestamp不支援支援（counter reset 偵測更準）
Exemplars支援（v2.30+）原生支援

§ 8. Remote Read 與其替代

8.1 Remote Read 是什麼

讓 Prometheus 在執行 PromQL 時，除了讀本地 TSDB，也讀遠端後端， 把結果合併回給使用者： 

remote_read:
  - url: 'https://mimir.example.com/prometheus/api/v1/read'
    name: 'mimir-historical'

    # true = 連本地有的時段也查（讓兩邊資料校驗）
    # false = 只查本地沒有的舊資料（推薦）
    read_recent: false

    headers:
      X-Scope-OrgID: 'tenant-platform'

8.2 為什麼很少人用 Remote Read

  • 每次查詢都要送網路請求，延遲大
  • 後端要支援 remote read API（不是所有都支援）
  • 資料量大時會把 Prometheus 拖垮

8.3 更好的替代：Grafana 多 datasource

實務上 99% 採用：

  • Prometheus 本地保留 7 天（快速 dashboard）
  • Mimir 保留 1 年（歷史查詢）
  • Grafana 設兩個 datasource，用 mixed datasource 或變數切換

8.4 直接全查 Mimir/Thanos

或者更激進：所有 dashboard 直接查 Mimir，本地 Prometheus 只負責 ingestion + remote_write。 Grafana 看的查詢都從 Mimir querier 來，不再用 Prometheus 查。

§ 9. 長期儲存方案選型

主流方案對比（2026 年現狀）：

方案架構強項弱項
Thanos Sidecar / Receive，物件儲存 沿用 Prometheus 原生 TSDB；輕量起步；社群成熟 查詢延遲較高；多 component 部署複雜
Grafana Mimir Push-only，物件儲存 + cache 真正 horizontal scale；多租戶；查詢快 需要更多元件（11+ services）；運維成本高
VictoriaMetrics 單 binary 或 cluster 單 binary 部署最簡；高效；高壓縮 非 100% Prometheus 相容（PromQL extensions）
Cortex 類似 Mimir（Mimir 從這 fork） 原始多租戶 Prometheus solution 開發已停滯，建議遷至 Mimir
Prometheus 原生 Federation 無新元件；簡單 規模上限低

9.1 選型決策矩陣

需求推薦
單組織單 cluster，< 10M seriesFederation 或單機 Prometheus
單組織多 cluster，1-100M seriesThanos（最易落地）
多租戶 / 多事業群 / 100M+ seriesMimir
追求極簡部署 / 成本敏感VictoriaMetrics
已有 Cortex 部署規劃遷移到 Mimir
用 Grafana Cloud使用其 hosted Mimir

10.2 兩個 HA Prometheus 設定

# Prometheus 1
global:
  external_labels:
    cluster: 'production-tw'
    replica: 'prom-1'

remote_write:
  - url: 'http://mimir-distributor:8080/api/v1/push'
    name: 'mimir'
    headers:
      X-Scope-OrgID: 'team-platform'
    queue_config:
      max_shards: 50
      max_samples_per_send: 2000

# Prometheus 2（一樣，只改 replica）
global:
  external_labels:
    cluster: 'production-tw'
    replica: 'prom-2'

10.3 Mimir 端的 dedup

Mimir 透過 -distributor.ha-tracker.enabled=true 啟用 HA tracker， 它會根據 cluster + replica label 自動 dedup： 每個 cluster 的某時段只接受一個 replica 的資料。 

# mimir.yaml
distributor:
  ha_tracker:
    enable_ha_tracker: true
    kvstore:
      store: memberlist

limits:
  ha_cluster_label: cluster
  ha_replica_label: replica
  accept_ha_samples: true

§ 11. 常見問題與最佳實踐

⚠️ 問題 1：Remote Write 跟不上 ingestion

症狀：prometheus_remote_storage_samples_pending 一直成長，samples_dropped_total 開始增加。
解法：
• 加 max_shards 與 max_samples_per_send
• 用 write_relabel_configs drop 掉不重要的 metric
• 後端擴容（Mimir distributor / VictoriaMetrics）

⚠️ 問題 2：Federation 拉不到完整資料

症狀：global Prometheus 有些 series 缺資料點。
原因：response 太大被截斷、scrape_timeout 不夠。
解法：
• 縮小 match[] 範圍（只拉 recording rule 結果）
• 加大 scrape_timeout
• 拆成多個 federation job 並行

⚠️ 問題 3：HA dedup 沒生效

症狀：Mimir 中看到雙倍 sample。
檢查：
• 兩個 Prometheus 是否設了不同 replica label？
• Mimir HA tracker 是否啟用？
• cluster label 是否一致？

⚠️ 問題 4：Tenant 互相看到對方資料

症狀：A 團隊查到 B 團隊的 metric。
原因：X-Scope-OrgID header 沒設或設錯。
解法：每個 Prometheus 的 remote_write 都要設專屬 X-Scope-OrgID， Grafana datasource 同樣要帶該 header。

✅ Best Practices 總結
  1. 本地 Prometheus 短期 retention（7-15 天），長期靠遠端
  2. Federation 只拉 recording rule 結果，不拉原始 metric
  3. Federation 必設 honor_labels: true
  4. Remote Write 必設 retry_on_http_429: true
  5. Remote Write 用 write_relabel_configs 過濾節省成本
  6. HA pair 用 cluster + replica external_labels
  7. 監控 remote write 健康：pending、failed、shards、end-to-end latency
  8. 選 Mimir 或 Thanos 看規模：Mimir 高規模、Thanos 易起步
  9. VictoriaMetrics 是部署最簡的選擇
  10. Grafana 設多 datasource，本地查近期、遠端查歷史

✅ 本章重點摘要

30 秒回顧
  • 單 Prometheus 上限約 10M active series，超過要擴展
  • 兩條路：Federation（簡單但有限）或 Remote Storage（強大）
  • Federation 必須：honor_labels: true + 用 recording rule 預先聚合 + 精確的 match[]
  • Federation 限制：HTTP body 上限、跳格採樣、不適合 raw data
  • Remote Write 是 push 模型，串流批次傳送，後端可選 Mimir/Thanos/VM
  • Queue 調校：max_shards / max_samples_per_send / retry_on_http_429
  • RW 2.0（v2.50+ experimental）：原生 native histogram、metadata 內嵌、Created Timestamp
  • Remote Read 很少用，Grafana 多 datasource 是主流方案
  • 選型：Thanos 易起步、Mimir 高規模多租戶、VictoriaMetrics 最簡
  • HA：用 cluster + replica external_labels，Mimir HA tracker dedup
