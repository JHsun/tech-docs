§ 1. 聚合運算子完整解析

聚合運算子把多條 series 合成一條（或多條）。完整列表：

運算子意義輸入型別
sum加總instant vector
min最小值instant vector
max最大值instant vector
avg平均instant vector
group所有值設為 1（用於存在性檢查）instant vector
stddev母體標準差instant vector
stdvar母體變異數instant vector
count非空 series 個數instant vector
count_values按不同值分組計數instant vector
bottomk(N, ...)最小的 N 條instant vector
topk(N, ...)最大的 N 條instant vector
quantile(φ, ...)分位數（單一值估算，不是 histogram）instant vector
limitk(N, ...)取前 N 條（v2.55+）instant vector
limit_ratio(R, ...)取 R 比例的 series（v2.55+）instant vector

1.1 基本語法

aggregator(expression)                       # 全部聚合成一個 series
aggregator by (label_list) (expression)      # 按 label 分組
aggregator without (label_list) (expression) # 排除這些 label 後分組
aggregator(parameter, expression)            # 帶參數（topk/quantile）

1.2 範例對比

# 1. 全部 sum 成一條 series（沒有任何 label）
sum(rate(http_requests_total[5m]))
# => {} 1234.5

# 2. 按 service 分組
sum by (service) (rate(http_requests_total[5m]))
# => {service="api"} 800
# => {service="web"} 434.5

# 3. 排除 instance、status 後分組（與 1 等價，但保留其他 label）
sum without (instance, status) (rate(http_requests_total[5m]))
# => {service="api", method="POST"} 500
# => ...

§ 2. by 與 without 對比

這是 PromQL 最容易搞混的地方之一。看清楚這兩個的差別。

2.1 by：「只保留這些 label」

# 原始 series：
# http_requests_total{instance="a", method="GET", path="/", status="200"} 100
# http_requests_total{instance="b", method="GET", path="/", status="200"} 200
# http_requests_total{instance="a", method="POST", path="/api", status="200"} 50

sum by (method) (http_requests_total)
# => {method="GET"}  300
# => {method="POST"} 50

2.2 without：「砍掉這些 label」

sum without (instance) (http_requests_total)
# => {method="GET", path="/", status="200"}     300
# => {method="POST", path="/api", status="200"} 50
✅ 該選哪個？
  • 知道要保留什麼 → 用 by
  • 知道要砍什麼（其他都留） → 用 without
  • 實務上很多人習慣用 without，因為新增 label 時不用每次都改 query

§ 3. topk / bottomk 與 quantile

3.1 topk：找出最大的 N 條

# 找出 CPU 用量最高的 5 個 instance
topk(5, rate(node_cpu_seconds_total{mode!="idle"}[5m]))

# 找出每個 service 中流量最高的 3 個 endpoint
topk(3, sum by (service, path) (rate(http_requests_total[5m])))

3.2 bottomk：找出最小的 N 條

# 找出最少請求的 5 個 service（找冷門 endpoint）
bottomk(5, sum by (service) (rate(http_requests_total[5m])))

3.3 quantile（聚合）vs histogram_quantile（histogram）

很容易搞混！

函式輸入用途
quantile(0.95, expr) 多條 series 的單一值 「跨機器的 95 分位」（從多個值估算）
histogram_quantile(0.95, expr) Histogram（buckets） 「分布的 95 分位」（精確估算）
quantile_over_time(0.95, v[5m]) 單條 series 的時間區段 「過去 5 分鐘的 95 分位」

3.4 count_values：罕用但很強

# 統計每個 status code 各有幾條 series
count_values("status_value", up)
# => {status_value="0"} 3
# => {status_value="1"} 47

§ 4. 函式分類與最常用函式

Prometheus 內建約 75 個函式（含 native histogram 專用）。我們依用途分類：

4.1 數學函式（Scalar 級）

函式用途
abs(v)絕對值
ceil(v)向上取整
floor(v)向下取整
round(v, to_nearest)四捨五入到指定倍數
sqrt(v)平方根
exp(v)e 的 v 次方
ln(v) / log2(v) / log10(v)對數
clamp(v, min, max)限制在範圍內
clamp_max(v, max) / clamp_min(v, min)單邊限制
sgn(v)正負號（-1/0/1）

4.2 Counter/Gauge 函式（Range Vector）

函式用途
rate(v[5m])Counter 的每秒平均增量（最常用）
irate(v[5m])Counter 的瞬時 rate（看最後兩點）
increase(v[5m])Counter 的區間總增量
delta(v[5m])Gauge 的區間差值
idelta(v[5m])Gauge 的瞬時差值
resets(v[5m])Counter 重置次數
changes(v[5m])值變化的次數（Gauge）
predict_linear(v[1h], 4*3600)線性預測 4 小時後的值
deriv(v[5m])每秒導數（Gauge）
holt_winters / double_exponential_smoothing指數平滑（v3 改名）

4.3 rate vs irate 的差別

⚠️ 常見誤用：用 irate 配長 range

irate 只看 range vector 的最後兩個 sample，所以 irate(...[5m]) 與 irate(...[1h]) 結果幾乎一樣，但消耗資源不同。
使用建議：
• 大部分情境用 rate，圖表平滑
• 需要看「立即變化」（除錯時）用 irate
• Alerting rule 一律用 rate（避免誤觸發）

4.4 _over_time 系列（Gauge 聚合）

avg_over_time(v[5m])         # 5 分鐘內的平均
sum_over_time(v[5m])         # 5 分鐘內的總和（很少用）
min_over_time(v[5m])         # 5 分鐘最小值
max_over_time(v[5m])         # 5 分鐘最大值
quantile_over_time(0.95, v[5m])  # 5 分鐘的 95 分位
stddev_over_time(v[5m])      # 5 分鐘標準差
count_over_time(v[5m])       # 5 分鐘 sample 數
last_over_time(v[5m])        # 5 分鐘最後一個 sample
present_over_time(v[5m])     # 有 sample 就回 1
absent_over_time(v[5m])      # 完全沒 sample 才回 1
mad_over_time(v[5m])         # 中位數絕對偏差（experimental）

§ 5. 時間相關函式

函式意義
time()當前 query 時間（Unix epoch 秒）
timestamp(v)每個 sample 的 timestamp
day_of_week()0=星期天 ... 6=星期六
day_of_month()1-31
day_of_year()1-366
days_in_month()當月天數
month()1-12
year()四位年
hour()0-23
minute()0-59

5.1 經典：判斷 metric 多久沒更新

# 該 series 上次更新到現在過了多久（秒）
time() - timestamp(my_metric)

# 超過 5 分鐘沒更新
(time() - timestamp(my_metric)) > 300

5.2 經典：在週末降低告警閾值

# 平日（週一至週五）error rate > 1% 才告警，週末 > 0.5% 就告警
(
  rate(http_errors_total[5m]) / rate(http_requests_total[5m]) > 0.01
  and on() day_of_week() != 0 and on() day_of_week() != 6
)
or
(
  rate(http_errors_total[5m]) / rate(http_requests_total[5m]) > 0.005
  and on() (day_of_week() == 0 or day_of_week() == 6)
)

§ 6. 標籤操作函式：label_replace、label_join

6.1 label_replace

從一個 label 用 regex 抽出新 label：

label_replace(v instant-vector, dst_label, replacement, src_label, regex)
# 從 instance label "api-1:9090" 抽出 hostname "api-1" 寫入 host label
label_replace(up, "host", "$1", "instance", "([^:]+):.*")

6.2 label_join

把多個 label 用分隔符串接成新 label：

label_join(v instant-vector, dst_label, separator, src_label1, src_label2, ...)
# 把 namespace 與 pod 串成 "default/web-1"
label_join(up, "ns_pod", "/", "namespace", "pod")

6.3 實戰：跨服務 join

常見需求：把兩個有不同 label 命名的 metric 配對。

# A 用 instance="a:9090"，B 用 host="a"
# 用 label_replace 把 A 的 instance 轉成 host
label_replace(metric_a, "host", "$1", "instance", "([^:]+):.*")
  / on (host) metric_b

§ 7. Histogram 函式（含 Native Histogram）

7.1 經典：classic histogram 的分位數

# P99 延遲（必須保留 le 標籤！）
histogram_quantile(0.99,
  sum by (le, service) (rate(http_request_duration_seconds_bucket[5m]))
)
⚠️ histogram_quantile 三大鐵律
  1. 必須保留 le label：聚合時 by (le, ...) 不能漏
  2. 必須先 rate 再聚合再 quantile：「rate → sum → quantile」順序
  3. 輸入要是 _bucket 系列，不是 _count 或 _sum

7.2 Native Histogram 專用函式

函式用途
histogram_count(v)取出 native histogram 的 count
histogram_sum(v)取出 sum
histogram_avg(v)平均（v2.50+）
histogram_stddev(v)標準差（v2.50+）
histogram_stdvar(v)變異數
histogram_fraction(lo, hi, v)觀測落在 [lo, hi] 區間的比例
histogram_quantile(φ, v)分位數（同名，自動偵測 classic / native）
histogram_quantiles(v, "label", φ1, φ2, ...)一次算多個分位數（experimental）

7.3 Native histogram 的查詢長相

# Native histogram 不需要 _bucket 後綴！
histogram_quantile(0.99, sum(rate(http_request_duration_seconds[5m])))

# 平均延遲
histogram_avg(sum(rate(http_request_duration_seconds[5m])))

# 算 200ms-500ms 區間的請求佔比
histogram_fraction(0.2, 0.5,
  sum(rate(http_request_duration_seconds[5m]))
)

§ 8. Subquery

8.1 概念

Subquery 是 PromQL 強大但少用的功能： 讓您把一個 instant vector 表達式「展開」成 range vector。 語法：expr[range:resolution]。 

# 過去 1 小時，每分鐘計算一次的 5 分鐘 rate
rate(http_requests_total[5m])[1h:1m]
# 結果是 range vector，可繼續用 _over_time 函式

8.2 為什麼需要 subquery？

因為某些「想做」的事不能直接做。例如：

# ❌ 不能：rate 結果（instant vector）不能直接 _over_time
max_over_time(rate(http_requests_total[5m])[1h])

# ✅ 對：用 subquery 把 rate 結果展開成 range
max_over_time(rate(http_requests_total[5m])[1h:1m])

這條意思是：「過去 1 小時內，每 1 分鐘計算一次 5 分鐘 rate，找出最大值」— 也就是「過去 1 小時內最高的 5 分鐘 rate」。

8.3 Subquery 的 resolution 解析度

expr[1h:1m]   # 每 1 分鐘解析一次
expr[1h:30s]  # 每 30 秒
expr[1h:]     # 用全域 evaluation_interval（預設 15s）
⚠️ Subquery 很耗 CPU

Subquery 對每個 step 都要重新計算內部表達式。 [1h:1m] 等於 60 次計算，[1d:1m] 等於 1440 次。 盡量用大的 resolution，或在 recording rule 中預先計算。

§ 9. info() 函式（v3.x experimental）

Info metric（如 kube_pod_info、node_meta）的常見問題是： 要把它的 label 加到主 metric 很麻煩，需要寫複雜的 group_left。 Prometheus 3.x 新增 info() 函式簡化這件事： 

# 傳統做法（很囉嗦）
rate(http_server_request_duration_seconds_count[5m])
  * on (job, instance) group_left (k8s_cluster_name)
    target_info

# 用 info()
info(rate(http_server_request_duration_seconds_count[5m]))
# 自動偵測並把相關 info metric 的 label 加進來

這個函式仍是 experimental，需要 --enable-feature=promql-experimental-functions。 語意還在演進，正式 API 可能改變。

§ 10. 進階範例彙整

10.1 找出 series 數量爆炸的 metric

topk(20, count by (__name__)({__name__=~".+"}))

10.2 計算 SLO 錯誤預算消耗

# 30 天內錯誤率（目標 99.9% = 預算 0.1%）
1 - (
  sum(rate(http_requests_total{status=~"5.."}[30d]))
  / sum(rate(http_requests_total[30d]))
)
# 結果若 < 0.999，代表已耗光預算

10.3 用 predict_linear 預測磁碟滿

# 預測 4 小時後磁碟剩餘 < 0 (即會在 4 小時內滿)
predict_linear(node_filesystem_avail_bytes{mountpoint="/"}[1h], 4 * 3600) < 0

10.4 計算當週新增的 unique 服務數

count(count by (service) (
  count_over_time(up{job="services"}[7d])
))

10.5 找出近期啟動的 instance（<= 1 小時前）

(time() - process_start_time_seconds) < 3600

10.6 跨服務 join：把 deployment 版本帶到 metric 裡

sum by (deployment, version) (
  rate(http_requests_total[5m])
  * on (deployment) group_left (version)
    deployment_info
)

10.7 計算「整點」的 sample 值

# 抓每個整點時刻的當前值
my_metric @ start()

10.8 用 absent 偵測 metric 失蹤

# 如果完全沒有 up{job="critical-service"} 就回 1
absent(up{job="critical-service"})

# 通常用於告警：
- alert: ServiceMissing
  expr: absent(up{job="critical-service"}) == 1
  for: 5m
