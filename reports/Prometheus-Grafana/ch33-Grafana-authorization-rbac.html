Chapter 33 · 授權與 RBAC

Grafana Authorization：Org、Team、Folder、RBAC、Service Account

Ch 32 解決「誰能登入」，本章解決「誰能看 / 改什麼」。Grafana 的權限模型有四個維度： Organization、User Role、Team、Folder。再加上 Enterprise 的 RBAC 細粒度權限。 本章把每個機制如何疊加講清楚。

§ 1. 權限模型總覽

1.1 四層權限結構

graph TB SERVER[Grafana Server
GrafanaAdmin · super] subgraph Org["Organization · 租戶層"] ORG[Org · Main Org · 預設] ORG_ADMIN[Org Admin] EDITOR[Editor] VIEWER[Viewer] end subgraph Resources["Org 內資源"] FOLDER[Folder
含 dashboard · alert] DS[Datasource] TEAM[Team
使用者群組] end SERVER --> ORG ORG --> ORG_ADMIN ORG --> EDITOR ORG --> VIEWER ORG_ADMIN --> FOLDER ORG_ADMIN --> DS ORG_ADMIN --> TEAM TEAM -.授權給.-> FOLDER TEAM -.授權給.-> DS

1.2 一個 user 的「最終權限」是什麼決定的？

權限疊加規則（取最寬鬆者）：

  1. User 的 Server-level role（GrafanaAdmin?）
  2. User 在這個 Org 的 role（Admin / Editor / Viewer）
  3. User 加入的 Team 對特定 folder/datasource 的權限
  4. User 個別被授予 特定 folder/dashboard 的權限
  5. Folder 繼承 給其下 dashboard 的權限

舉例：alice 是 Org Viewer，但同時是 platform team 成員，platform team 對「Production」folder 有 Editor 權限 → alice 在 Production 內可以編輯。

§ 2. Organization（多租戶基礎）

2.1 Org 是什麼

Organization（簡稱 Org） 是 Grafana 的最頂層資源容器。每個 Org 有獨立的：

  • Datasources
  • Dashboards / Folders
  • Alerting rules
  • API keys / Service accounts
  • Users（user 可以同時在多 org，但 role 各自獨立）

2.2 Org 的典型用法

場景用法
單一公司、簡單環境只用一個 Main Org（最常見）
多事業部每個事業部一個 Org
SaaS 服務多客戶每個客戶一個 Org（早期做法）
強隔離團隊每個 team 一個 Org

2.3 Org 的限制

⚠️ Org 不再被推薦做多租戶

早期 Grafana 用 Org 實現「不同團隊看不同東西」。但現代做法是：
• 多 team / 同 org：用 Folder + Team permission 隔離
• 多客戶 SaaS：用 Mimir/Loki 等的 tenant 機制 + Folder
• 真的需要強隔離：跑多個 Grafana instance
Grafana 12+ 的方向是逐漸淡化 Org 概念，許多新功能（如 RBAC、SCIM）只在主 Org 完整支援。

2.4 設定 Org

# grafana.ini
[users]
auto_assign_org = true             # 新使用者自動加入主 org
auto_assign_org_id = 1             # 主 org id（預設 1）
auto_assign_org_role = Viewer

allow_org_create = false           # 一般 user 不能建新 org

§ 3. User Roles（內建四種）

3.1 四種 role 的權限詳細

能力ViewerEditorOrg AdminServer Admin
看 dashboard✅✅✅✅
用 Explore❌（除非設定）✅✅✅
建/改 dashboard❌✅✅✅
建/改 alert rule❌✅✅✅
建/改 datasource❌❌✅✅
管理使用者（在當前 Org）❌❌✅✅
管理 Team❌❌✅✅

修改 server settings❌❌❌✅

3.2 Role 變動的常見途徑

  • Org Admin 在 UI 改：Administration → Users
  • OAuth role mapping 同步（Ch 32）
  • LDAP group mapping 同步
  • API：PATCH /api/orgs/<id>/users/<userId>

3.3 Viewers Can Edit / Editors Can Admin

[users]
viewers_can_edit = false           # 開啟可讓 viewer 用 explore（debug 友善）
editors_can_admin = false          # 開啟讓 editor 能管理 folder permission

§ 4. Team（群組管理）

4.1 Team 是什麼

Team 是一群 user 的集合。Team 自身不直接給 role（user 仍各自有 Org role）， 但可以給 team 對特定資源（folder / datasource / dashboard）的權限。

4.2 Team 的價值

  • 大規模 user 管理：給 team 授權，不用一個一個 user 設
  • 對應 OAuth/LDAP 的 group：sync 後權限自動更新
  • Dashboard 個人化（每個 team 有自己的「starred dashboard」）

4.3 建立 Team

UI: Administration → Users and access → Teams → New team

API:

curl -X POST http://admin:admin@localhost:3000/api/teams \
  -H "Content-Type: application/json" \
  -d '{"name": "platform-team", "email": "platform@example.com"}'

# 加 user 到 team
curl -X POST http://admin:admin@localhost:3000/api/teams//members \
  -d '{"userId": 5}'

4.4 Team Sync（同步 OAuth/LDAP group）

Team Sync 是 Enterprise 功能。OSS 用戶可以用 OAuth 的 role_attribute_path + 手動加 team。

§ 5. Folder 與權限繼承

5.1 Folder 結構

Dashboard 與 alert rule 都放在 folder 內。Folder 有自己的 permission set， folder 內的 dashboard 繼承 folder 權限（除非個別 dashboard 覆蓋）。 

📁 Production
  ├─ 📊 API Service Health
  ├─ 📊 Database Overview
  └─ 📁 Frontend                  # 子 folder（v9+）
      └─ 📊 Web Vitals

📁 Staging
  └─ 📊 ...

📁 Personal · alice
  └─ 📊 alice's experiments

5.2 Folder 權限設定

每個 folder 可以針對：Role、Team、User 設定 4 種權限：

權限意義
View看 folder 內的 dashboard
Edit+ 建/改 dashboard
Admin+ 管理 folder 自身權限

5.3 預設權限

Role預設對 Folder 的權限
ViewerView 所有 folder
EditorEdit 所有 folder
Org AdminAdmin 所有 folder

5.4 限制 folder 給特定 team

場景：「Production folder 只給 platform team 看，DBA team 不能看」。

  1. 到 Production folder → Permissions
  2. 把 Editor / Viewer 預設權限移除
  3. 新增：platform-team → Edit；database-team → 不加（無權限）
  4. 記得保留 Admin role（否則 admin 也看不到）

5.5 Provisioning Folder Permission

OSS 沒有原生 provisioning folder permission 的方式（dashboards-provisioning 只能放 dashboard）。 解法：

  • 用 Terraform Provider 管理（推薦）
  • 用 API 在 deploy 時 set
  • Enterprise / Cloud 可用 Provisioning RBAC

§ 6. Datasource Permissions

6.1 Datasource 權限的特殊性

Datasource 預設「所有 user 都能 query」。
這對多 team 共用 Grafana 是大問題：所有 user 都能對所有 Prometheus 寫任意 PromQL。

6.2 Datasource Permissions（OSS 部分支援，v8+）

UI: Connections → Data sources → 選一個 → Permissions tab

給 platform-team query「Prometheus-Production」
給 dba-team query「PostgreSQL-Production」
其他 user → 無權限

6.3 在 OSS 中啟用

[datasources]
allow_uid_settings = true

沒設 permission 時，預設「全 org 可 query」。

6.4 Datasource 權限的限制

  • OSS 只能「能/不能 query 整個 datasource」，沒有「query 哪些 metric」這種細粒度
  • 細粒度 query 限制需要在 Prometheus 端設多個獨立 instance（Mimir 多租戶）
  • Enterprise 有 Data Source Permissions（更細粒度）+ Recorded Queries

§ 7. Dashboard 細粒度權限

7.1 Dashboard 通常繼承 folder 權限

除非您要對單一 dashboard 設特殊權限（罕見），否則不要個別設 dashboard permission，難維護。

7.2 個別 dashboard 設權限

Dashboard → Settings → Permissions：

限定 alice 是這個 dashboard 的 Edit
限定 bob 是 View

效果：dashboard 在 folder 顯示，但其他人打開會被擋

7.3 v11+ 的「Public Dashboards」

對外公開特定 dashboard，無需登入：

  1. 啟用 public dashboards：[public_dashboards] enabled = true
  2. 到 dashboard → Share → Public dashboard → enable
  3. 取得帶 secret token 的 URL
  4. 外人用該 URL 可看（不能改）

§ 8. RBAC（Enterprise，部分 OSS）

RBAC（Role-Based Access Control） 是 Grafana 的細粒度權限系統， 取代「只有 Viewer/Editor/Admin 三選一」的限制。OSS 部分支援，Enterprise 完整。

8.1 RBAC 的核心概念

  • Action：原子操作（如 dashboards:read、datasources:create）
  • Permission：Action + Scope（dashboards:read on folders:uid:abc）
  • Role：一組 permission 的集合
  • Role Assignment：把 role 給 user / team / 內建 role

8.2 內建 Role

除了 Viewer / Editor / Admin / GrafanaAdmin 之外，Enterprise 有更多 fixed role：

  • fixed:datasources:reader
  • fixed:datasources:writer
  • fixed:dashboards:reader
  • fixed:annotations:writer
  • fixed:reports:writer（Enterprise）
  • fixed:alerting:writer
  • ...上百個

8.3 RBAC 用例：建立 "Read-Only Dashboard Editor"

需求：可看所有 dashboard 但只能改 alerting rule。

POST /api/access-control/roles
{
  "name": "alerting-only-editor",
  "displayName": "Alerting Editor",
  "permissions": [
    {"action": "dashboards:read", "scope": "*"},
    {"action": "alert.rules:read", "scope": "*"},
    {"action": "alert.rules:write", "scope": "*"},
    {"action": "alert.rules:create", "scope": "*"},
    {"action": "alert.rules:delete", "scope": "*"}
  ]
}

# 把這個 role 指派給 user
POST /api/access-control/users//roles
{
  "roleUid": ""
}

8.4 OSS vs Enterprise RBAC

能力OSSEnterprise
Service Account RBAC✅✅
Folder fine-grained permission✅✅
Datasource permission有限完整
Custom roles❌✅
Team Sync (LDAP/OAuth)❌✅
SCIM provisioning❌✅（v12+）
Provisioning RBAC YAML❌✅

§ 9. Service Account 完整使用

Ch 32 已介紹 Service Account 的概念。本節深入操作。

9.1 為什麼用 Service Account 而非 API key

面向API Key（舊）Service Account（v9+）
身份識別沒有有專屬 user identity（audit log 看得到）
多 token否（一個 key 一個身份）是（一個 SA 可有多個 token）
Role 變動需重建 keySA 上改 role 就好
RBAC有限支援完整 RBAC
狀態v9+ deprecated推薦

9.2 完整工作流

# 1. 建立 service account
curl -X POST http://admin:admin@localhost:3000/api/serviceaccounts \
  -H "Content-Type: application/json" \
  -d '{"name": "ci-pipeline", "role": "Editor"}'

# Response: {"id": 5, "name": "ci-pipeline", "role": "Editor", ...}

# 2. 為 SA 建立 token（限 90 天）
curl -X POST http://admin:admin@localhost:3000/api/serviceaccounts/5/tokens \
  -H "Content-Type: application/json" \
  -d '{
    "name": "deploy-token-2026q2",
    "secondsToLive": 7776000
  }'

# Response: {"id": 1, "name": "deploy-token-2026q2", "key": "glsa_xxxxxxxx"}
# 這個 key 只顯示一次！立刻存起來

# 3. 用 token 呼叫 API
curl -H "Authorization: Bearer glsa_xxxxxxxx" \
  http://localhost:3000/api/dashboards/uid/abc

# 4. 列出所有 token
curl -u admin:admin http://localhost:3000/api/serviceaccounts/5/tokens

# 5. 撤銷 token
curl -X DELETE -u admin:admin \
  http://localhost:3000/api/serviceaccounts/5/tokens/

9.3 Token 命名規範

建議命名：<purpose>-<environment>-<rotation-period>

terraform-prod-2026q2          # CI 用
backup-cron-permanent          # 定期備份用
debug-alice-2026-05            # 暫時 debug 用

9.4 Token Rotation

建議每季 rotate 一次：建新 token → 更新使用方 → 撤銷舊 token。 可寫成 Terraform / Helm 自動化。

§ 10. 權限設計實戰

10.1 場景：3 個 team、共用 Grafana

需求：

  • Platform team：管所有 production dashboard、所有 datasource
  • Backend team：管 backend service dashboard、只 query Prometheus
  • Frontend team：管 frontend dashboard、只 query 特定 datasource
  • 所有人能看 status 公開 dashboard

10.2 設計

結構：
├── 📁 Status (公開, 所有人可看)
├── 📁 Production
│   ├── 📁 Platform     (platform-team Edit, others 看不到)
│   ├── 📁 Backend      (backend-team Edit, frontend-team 看不到)
│   └── 📁 Frontend     (frontend-team Edit)
└── 📁 Personal
    └── (個別 user 自己的)

Datasources:
- Prometheus-Production:
  - platform-team: Query
  - backend-team: Query
  - frontend-team: 不可 query
- Loki-Production:
  - platform-team: Query
  - backend-team: Query
  - frontend-team: 不可 query
- Prometheus-Frontend (專屬):
  - frontend-team: Query

Teams:
- platform-team
- backend-team
- frontend-team

Roles:
- All users: 預設 Viewer (Org-level)
- Team-based: 在各 folder / datasource 細設

10.3 設定步驟

  1. 建立 Teams（3 個）
  2. 把 user 加入對應 team
  3. 建立 Folder 結構
  4. 每個 folder 設定 permissions：先「移除預設 Editor/Viewer」，再「加 team 對應權限」
  5. Datasource permissions：同樣方式
  6. 「Status」folder 維持預設（Viewer 可看）

10.4 用 Terraform 自動化

# 範例：Terraform 管理 folder + permission
resource "grafana_folder" "production" {
  title = "Production"
}

resource "grafana_team" "platform" {
  name = "platform-team"
  email = "platform@example.com"
}

resource "grafana_folder_permission" "production" {
  folder_uid = grafana_folder.production.uid

  permissions {
    team_id    = grafana_team.platform.id
    permission = "Edit"
  }

  # 移除預設 Editor/Viewer
  permissions {
    role       = "Viewer"
    permission = "View"
  }
}

§ 11. Audit Log 與監控

11.1 OSS 的選擇

OSS 沒有內建 audit log。但可以：

  • 啟用 access log：router_logging = true 在 [server]
  • 從 reverse proxy 日誌：NGINX access log 可看誰打了什麼 API
  • 自寫 webhook：用 Grafana audit-style HTTP middleware（社群實作）

11.2 Enterprise Audit Log

記錄詳盡：誰登入、誰建/改/刪 dashboard、誰改 permission、誰查 datasource。可導到 Loki / Splunk。

11.3 認證/授權相關 metrics

# 登入次數（成功 / 失敗）
rate(grafana_api_login_post_total[5m])

# 當前 active user
grafana_stat_active_users

# Service account / API key 使用
grafana_api_response_status_total{handler=~".*/serviceaccounts/.*"}

11.4 監控建議

  • 失敗登入率高 → 可能 brute force
  • 新增 admin user → 可疑時告警
  • 大量 datasource permission 改動 → 記錄 + 告警
  • Service account token 即將過期 → 提醒 rotate

✅ 本章重點摘要

30 秒回顧
  • 權限四層：Server / Org / User Role / Team / Folder + Resource permission
  • 最終權限取「最寬鬆者」（疊加而非取交集）
  • Org 不再被推薦做多租戶（v12 方向是淡化 Org）
  • 四種內建 role：Viewer / Editor / Org Admin / Server Admin（最後一個是跨 Org）
  • Team 是 user 群組，不直接給 role，給特定資源權限
  • Folder permission 是 dashboard 隔離的核心；child dashboard 預設繼承
  • Datasource permission 預設「全 org 可 query」，多 team 環境必設
  • Public Dashboards（v10+）取代 Anonymous Auth 做公開 dashboard
  • RBAC 提供細粒度 permission；自訂 role 是 Enterprise feature，OSS 用內建 fixed role
  • Service Account（v9+）取代 API key，有 user identity、可 rotate token
  • Token 命名規範：<purpose>-<env>-<rotation>，建議季度 rotate
  • 大規模權限管理用 Terraform Provider 自動化
  • OSS 沒內建 audit log；用 reverse proxy log 替代或升 Enterprise
