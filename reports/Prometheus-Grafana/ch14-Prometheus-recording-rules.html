Chapter 14 · Recording Rules

Recording Rules：把昂貴查詢預先算好

Recording rules 把複雜或常用的 PromQL 查詢「預先計算」並存成新的 time series。 一個好的 recording rule 體系能讓 dashboard 載入快 10-100 倍、降低 Prometheus CPU 壓力、 並讓 alerting 更穩定。本章從語法到命名規範到實戰範例，一次教完。

§ 1. 為什麼需要 Recording Rules

Recording rule 解決三個核心問題：

問題沒有 recording rule有 recording rule
Dashboard 慢 每次刷新都重算複雜 query（30 秒以上） 查預算好的 series，毫秒級回應
Alerting 不穩 Alert 依賴複雜計算，evaluation 超時 Alert 引用簡單的 recording 結果
多 dashboard 重複算 同一個查詢在 10 個 panel 算 10 次 算 1 次，10 個 panel 共用

1.1 性能影響的數字感

ℹ️ 真實案例的差距

一個跨 100 個 service、按 path / region / le 聚合的 P99 latency 查詢：
• 直接查：每次掃 5 億 sample，CPU 30s+，記憶體 8GB+
• 用 recording rule（每 30s 算一次）：dashboard 查 50ms，記憶體幾 MB
性能差距可達數百倍。

1.2 Recording rules 是「物化 view」

類似 SQL 資料庫的 materialized view 概念：把昂貴的 join/aggregation 預先算好存起來。 每隔固定時間（預設 1 分鐘）重新計算一次，結果作為新的 time series 存入 TSDB。

graph LR SRC[原始 metrics
http_requests_total] RR[Recording Rule
每 30s 評估一次] NEW[新 time series
service:requests:rate5m] DASH[Dashboard / Alert] SRC --> RR RR --> NEW NEW --> DASH

§ 2. 基本語法與 group 結構

2.1 最簡單的 rule file

# /etc/prometheus/rules/recording.yml
groups:
  - name: example
    interval: 30s
    rules:
      - record: code:prometheus_http_requests_total:sum
        expr: sum by (code)(prometheus_http_requests_total)

並在 prometheus.yml 裡引用：

rule_files:
  - "rules/recording.yml"
  - "rules/alerts/*.yml"

2.2 完整的 group 結構

groups:
  - name:                 # 必填，檔案內唯一
    interval:               # 此 group 的執行間隔，預設 = global.evaluation_interval
    limit:                       # 此 group 每條 rule 最多產生多少 series（0 = 無上限）
    query_offset:           # 評估時間往後推（用於 HA 場景）
    labels:                           # 加在所有此 group 結果上的 label
      team: platform

    rules:
      - record:          # 新 metric 名稱
        expr:                 # 要計算的表達式
        labels:                       # 加在此 rule 結果上的 label
          severity: info

2.3 多個 group 範例

groups:
  # Group 1：每 30 秒算一次的常用聚合
  - name: common_aggregations
    interval: 30s
    rules:
      - record: service:requests:rate5m
        expr: sum by (service)(rate(http_requests_total[5m]))

      - record: service:errors:rate5m
        expr: sum by (service)(rate(http_requests_total{status=~"5.."}[5m]))

  # Group 2：每 1 分鐘算一次的 SLO 計算
  - name: slo_calculations
    interval: 1m
    rules:
      - record: service:availability:ratio5m
        expr: |
          sum by (service)(rate(http_requests_total{status!~"5.."}[5m]))
          /
          sum by (service)(rate(http_requests_total[5m]))

  # Group 3：每 5 分鐘算一次的長窗口
  - name: long_window_aggregations
    interval: 5m
    rules:
      - record: service:requests:rate30d
        expr: sum by (service)(rate(http_requests_total[30d]))

§ 3. 命名規範：level:metric:operations

Prometheus 官方推薦的命名規範（必背）：level:metric:operations， 用兩個冒號分成三段。

3.1 三段語意

段意義範例
level聚合層級（保留了哪些 label）instance / job / service / cluster
metric原始 metric 名稱（去除後綴）node_cpu / http_requests
operations對 metric 做的操作rate5m / avg_rate5m / p99

3.2 命名範例對照

Recording rule 名稱對應 expr 的意思
instance:node_cpu:avg_rate5m每 instance 的 CPU 5 分鐘平均 rate
job:http_requests:rate5m每 job 的請求 5 分鐘 rate
service:request_duration_seconds:p99_5m每 service 的 P99 延遲
cluster:cpu_usage:avg_rate5m整個 cluster 的 CPU 平均
code:prometheus_http_requests_total:sum按 code 加總的 HTTP 請求
⚠️ 為什麼這個慣例重要？

看 metric 名稱就能猜到「它是怎麼算的」、「保留了哪些 label」、「適合用在什麼場景」。 這對跨團隊協作非常重要。當有人看到 service:request_duration:p99_5m， 能立刻理解：「按 service 分組、5 分鐘窗口、99 分位的延遲」。

3.3 Counter 的後綴慣例

對 counter 做了 rate 後，原本的 _total 後綴必須去掉：

# ✅ 對：去掉 _total（因為已經不是累積值了）
- record: service:http_requests:rate5m
  expr: sum by (service)(rate(http_requests_total[5m]))

# ❌ 錯：誤導後續使用者以為這是 counter
- record: service:http_requests_total:rate5m

§ 4. 評估頻率（interval）的選擇

4.1 預設行為

若 group 沒有自訂 interval，會使用 global.evaluation_interval（預設 1 分鐘）。 

global:
  evaluation_interval: 15s   # 全域預設：每 15s 評估一次所有 rule

4.2 不同 interval 的取捨

Interval適用權衡
15s關鍵服務的核心指標CPU 高，但反應快
30s大多數 SLI 與 dashboard業界主流
1m非關鍵的長期趨勢節省 CPU
5m+跨日/週的長窗口聚合大幅節省，但偵測延遲大

4.3 規則：interval 應 <= 1/4 的 range

若 expr 用 [5m]，interval 不應超過 5m / 4 = 75s。 否則同一個 sample 會被多次納入相鄰 evaluation 的計算（不影響結果，但浪費 CPU）。

更嚴重的反例：

# interval 太大，跟 range 一樣！
- name: bad
  interval: 5m
  rules:
    - record: x:rate5m
      expr: rate(x_total[5m])
# 每 5 分鐘算一次，每次只算「過去 5 分鐘」 — sample 邊界對齊不上，數值會抖

§ 5. Group 內的順序與依賴

5.1 同 group 內：依序執行

同一個 group 內的 rules 是循序執行的。 這代表後面的 rule 可以引用前面 rule 的輸出。 

groups:
  - name: with_dependencies
    interval: 30s
    rules:
      # Step 1：先算分母
      - record: service:requests:rate5m
        expr: sum by (service)(rate(http_requests_total[5m]))

      # Step 2：再算分子
      - record: service:errors:rate5m
        expr: sum by (service)(rate(http_requests_total{status=~"5.."}[5m]))

      # Step 3：用前面兩個結果算錯誤率
      - record: service:error_ratio:rate5m
        expr: service:errors:rate5m / service:requests:rate5m

5.2 不同 group：並行執行（v2.50+）

Prometheus 從 v2.50 起，不同 group 之間可以並行執行，前提是它們之間沒有依賴。 啟用方式：--rules.alert.resend-delay 加上 concurrent-rule-eval-limit。 

prometheus \
  --enable-feature=concurrent-rule-eval \
  --rules.max-concurrent-rule-evals=4

5.3 跨 group 引用要小心

# Group A 算結果
- name: groupA
  interval: 30s
  rules:
    - record: A:value
      expr: ...

# Group B 引用 Group A 的結果
- name: groupB
  interval: 30s
  rules:
    - record: B:value
      expr: A:value * 2

這沒問題，但要注意：Group B 看到的 A:value 可能是「上一輪」的結果， 最多落後 1 個 interval。對精確要求高的計算，把它們放在同一個 group。

§ 6. 實戰範例：分層 SLI/SLO

下面是一個生產級的「分層 SLO」recording rule 體系：

# /etc/prometheus/rules/slo_recording.yml
groups:
  # ===== 層 1：基礎聚合 =====
  - name: slo_base
    interval: 30s
    rules:
      # 各時窗的請求 rate
      - record: service:http_requests:rate5m
        expr: sum by (service)(rate(http_requests_total[5m]))
      - record: service:http_requests:rate30m
        expr: sum by (service)(rate(http_requests_total[30m]))
      - record: service:http_requests:rate1h
        expr: sum by (service)(rate(http_requests_total[1h]))
      - record: service:http_requests:rate6h
        expr: sum by (service)(rate(http_requests_total[6h]))
      - record: service:http_requests:rate1d
        expr: sum by (service)(rate(http_requests_total[1d]))

      # 各時窗的錯誤 rate
      - record: service:http_errors:rate5m
        expr: sum by (service)(rate(http_requests_total{status=~"5.."}[5m]))
      - record: service:http_errors:rate30m
        expr: sum by (service)(rate(http_requests_total{status=~"5.."}[30m]))
      - record: service:http_errors:rate1h
        expr: sum by (service)(rate(http_requests_total{status=~"5.."}[1h]))
      - record: service:http_errors:rate6h
        expr: sum by (service)(rate(http_requests_total{status=~"5.."}[6h]))
      - record: service:http_errors:rate1d
        expr: sum by (service)(rate(http_requests_total{status=~"5.."}[1d]))

  # ===== 層 2：錯誤率 =====
  - name: slo_error_ratio
    interval: 30s
    rules:
      - record: service:error_ratio:5m
        expr: service:http_errors:rate5m / service:http_requests:rate5m
      - record: service:error_ratio:30m
        expr: service:http_errors:rate30m / service:http_requests:rate30m
      - record: service:error_ratio:1h
        expr: service:http_errors:rate1h / service:http_requests:rate1h
      - record: service:error_ratio:6h
        expr: service:http_errors:rate6h / service:http_requests:rate6h
      - record: service:error_ratio:1d
        expr: service:http_errors:rate1d / service:http_requests:rate1d

  # ===== 層 3：燒率（burn rate）=====
  # 假設 SLO=99.9% → 預算=0.001
  - name: slo_burn_rate
    interval: 30s
    labels:
      slo_target: "0.999"
    rules:
      - record: service:burn_rate:5m
        expr: service:error_ratio:5m / 0.001
      - record: service:burn_rate:30m
        expr: service:error_ratio:30m / 0.001
      - record: service:burn_rate:1h
        expr: service:error_ratio:1h / 0.001
      - record: service:burn_rate:6h
        expr: service:error_ratio:6h / 0.001

  # ===== 層 4：延遲分位數 =====
  - name: slo_latency
    interval: 30s
    rules:
      - record: service:request_duration:p50_5m
        expr: |
          histogram_quantile(0.50,
            sum by (service, le)(rate(http_request_duration_seconds_bucket[5m])))
      - record: service:request_duration:p90_5m
        expr: |
          histogram_quantile(0.90,
            sum by (service, le)(rate(http_request_duration_seconds_bucket[5m])))
      - record: service:request_duration:p99_5m
        expr: |
          histogram_quantile(0.99,
            sum by (service, le)(rate(http_request_duration_seconds_bucket[5m])))

有了這套 recording rules，後續的 dashboard 與 alert 就可以直接寫 service:burn_rate:5m、service:request_duration:p99_5m， 完全不用重複那一大段 PromQL。

§ 7. 監控 Recording Rules 自身

Recording rule 自己也會出問題（評估失敗、超時、產生太多 series）。Prometheus 提供以下 metric：

Metric意義
prometheus_rule_evaluation_duration_seconds每條 rule 評估耗時
prometheus_rule_evaluations_total累積評估次數
prometheus_rule_evaluation_failures_total累積評估失敗次數
prometheus_rule_group_iterations_missed_total跳過的評估次數（rule 太慢來不及）
prometheus_rule_group_iterations_total應執行的總次數
prometheus_rule_group_duration_seconds每個 group 的總耗時
prometheus_rule_group_last_duration_seconds最近一次評估耗時

7.1 必設的告警

- alert: PrometheusRuleEvaluationFailures
  expr: rate(prometheus_rule_evaluation_failures_total[5m]) > 0
  for: 5m
  annotations:
    summary: "Prometheus rule evaluation failing"
    description: "{{ $value }} failures/sec"

- alert: PrometheusRuleEvaluationSlow
  expr: |
    prometheus_rule_group_last_duration_seconds
    > prometheus_rule_group_interval_seconds
  for: 10m
  annotations:
    summary: "Rule group {{ $labels.rule_group }} taking longer than its interval"

§ 8. 用 promtool 驗證與測試

8.1 語法檢查

# 驗證單個檔案
promtool check rules /etc/prometheus/rules/recording.yml

# 批次驗證
promtool check rules /etc/prometheus/rules/*.yml

# 驗證主設定 + 引用的 rules
promtool check config /etc/prometheus/prometheus.yml

8.2 規則測試（unit test）

Prometheus 提供 promtool test rules 做單元測試：

# tests/recording_test.yml
rule_files:
  - /etc/prometheus/rules/recording.yml

evaluation_interval: 30s

tests:
  - interval: 30s
    input_series:
      - series: 'http_requests_total{service="api", status="500"}'
        values: '0+10x10'                    # 0, 10, 20, 30, ... 100（每 30s +10）
      - series: 'http_requests_total{service="api", status="200"}'
        values: '0+90x10'                    # 0, 90, 180, ...

    promql_expr_test:
      - expr: service:error_ratio:5m{service="api"}
        eval_time: 5m
        exp_samples:
          - labels: '{service="api"}'
            value: 0.1                        # 10/(10+90) = 0.1

執行：

promtool test rules tests/recording_test.yml

§ 9. 常見陷阱與最佳實踐

⚠️ 陷阱 1：用 by 而非 without

官方建議使用 without，這樣加了新 label 不用改 query。但若您明確要過濾掉除某些 label 外的所有東西，用 by 更明確。
看情境抉擇，但團隊應統一風格。

⚠️ 陷阱 2：Counter 後綴沒去掉 _total

service:http_requests_total:rate5m 會誤導使用者以為這還是 counter。 Recording rule 的結果是 gauge（已不再累積），名稱不該帶 _total。

⚠️ 陷阱 3：在 alerting rule 直接用昂貴 expr

Alert evaluation 會被 rule_group_iterations_missed 卡住。 把昂貴 expr 拆成 recording rule，alert 只引用結果。

⚠️ 陷阱 4：高基數爆炸

Recording rule 的結果是新 series。若您的 expr 保留太多 label（例如 by (service, instance, path)）， 可能新增百萬條 series。用 limit 設上限： 

groups:
  - name: high_cardinality_warning
    limit: 10000     # 此 group 每條 rule 最多 10k series
    rules: ...
⚠️ 陷阱 5：在 group 中引用「自己」

service:request_rate:rate5m 引用 service:request_rate（同 group 中早於它的 rule）OK。 但循環引用會在重新載入時被 Prometheus reject。

✅ Best Practices 總結
  1. 命名遵守 level:metric:operations（業界共通）
  2. Counter 計算結果去掉 _total 後綴
  3. 分層設計：基礎聚合 → 比率 → 燒率 → SLO
  4. Interval 配合 range：interval ≤ range/4
  5. Long-window rules 用更長 interval：[1d] 用 interval: 5m
  6. 同一邏輯放同 group：依賴 rule 寫一起
  7. 用 limit 防 cardinality 爆炸
  8. 用 promtool 驗證 + 單元測試
  9. 監控 rule 自身的健康：prometheus_rule_evaluation_failures_total
  10. 版本控管 + Code Review：rules 檔案放 Git
