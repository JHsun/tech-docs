Chapter 49 · Alloy + Beyla

Grafana Alloy + Beyla：統一 Telemetry Collection

LGTM Stack 各家都有 backend，但怎麼把 telemetry 送進去？傳統做法是每家一個 agent （Promtail for Loki、Prometheus for metrics、OTel Collector for traces）。 Grafana Alloy 統一這四種訊號的收集，Grafana Beyla（已捐給 OpenTelemetry 改名 OBI） 用 eBPF 自動 instrument 應用。本章涵蓋 Alloy 元件模型、Beyla/OBI、與 OTel Collector 的關係、Promtail/Grafana Agent 遷移。

§ 1. Agent 演進史

1.1 早期：每家一個 agent

App → Prometheus（拉 metric）
App → Promtail → Loki（推 log）
App → OTel Collector → Tempo（trace）
App → Pyroscope agent → Pyroscope（profile）

→ 4 個 agent 在每台 host / pod！

1.2 第一代統一：Grafana Agent

2020 推出 Grafana Agent，一個 binary 整合 Prometheus + Promtail + OTel：

  • Static mode：類似 Prometheus 的 scrape config
  • Flow mode：用 River 設定語言（後來變成 Alloy 基礎）

1.3 第二代：Grafana Alloy（取代 Agent）

📌 名稱演進

2024 年起 Grafana Agent 正式 deprecated，改名 Grafana Alloy（基於原 Flow mode）。
原因：強調「OpenTelemetry Collector distribution」定位，並擴展 profiling 等新訊號。

1.4 完整時間軸

2018  Prometheus + Promtail
2019  Loki release
2020  Grafana Agent (Static mode)
2022  Grafana Agent Flow mode（River）
2023  Pyroscope acquisition
2024  Grafana Alloy 取代 Grafana Agent
      Beyla released
2025  Promtail deprecated（用 Alloy）
      Beyla 捐給 OpenTelemetry → OBI
      Grafana Agent end-of-life

1.5 現在的標準

訊號建議 agent
Metrics（scrape）Alloy（取代 Prometheus agent mode）
LogsAlloy（取代 Promtail）
TracesAlloy（取代 OTel Collector）
ProfilesAlloy（內建 Pyroscope eBPF）
Auto-instrumentAlloy + Beyla / OBI

§ 2. Grafana Alloy 概念

2.1 元件模型（Component Model）

Alloy 的核心設計：每個功能是一個元件（component），元件之間用「線」連起來形成 pipeline。

graph LR DISC[discovery.kubernetes] SCRAPE[prometheus.scrape] RELAB[prometheus.relabel] WRITE[prometheus.remote_write] DISC -- targets --> SCRAPE SCRAPE -- metrics --> RELAB RELAB -- metrics --> WRITE

每個元件有輸入（arguments）與輸出（exports），輸出可以餵給其他元件的輸入。

2.2 元件分類

類別前綴範例
Discoverydiscovery.kubernetes / consul / dns / aws
Prometheusprometheus.scrape / relabel / remote_write
Lokiloki.source.kubernetes / process / write
OTelotelcol.receiver / processor / exporter
Pyroscopepyroscope.scrape / ebpf / write
Beylabeyla.ebpf
Local IOlocal.file / file_match

2.3 Alloy 的價值

  • 單一 binary：一個 agent 處理 4 種訊號
  • 統一設定：River 語言一致
  • Pipeline 表達力：複雜 routing / filtering 容易表達
  • OTel 原生：完全相容 OpenTelemetry Collector
  • UI debug：內建 Web UI 看 pipeline 與資料流

§ 3. River 設定語言

3.1 基本語法

River 是 Alloy 的設定語言，類似 HCL（Terraform）：

// 元件定義: TYPE.NAME "LABEL" { ... }
prometheus.scrape "my_app" {
  targets = [
    { __address__ = "localhost:8080" },
  ]
  forward_to = [prometheus.remote_write.default.receiver]
}

prometheus.remote_write "default" {
  endpoint {
    url = "http://prometheus:9090/api/v1/write"
  }
}

3.2 表達式與函式

// 環境變數
url = env("PROMETHEUS_URL")

// 字串拼接
name = "service-" + env("ENV")

// 列表 / map
labels = ["env", "service", "version"]
config = {
  region = "us-east-1",
  zone   = "a",
}

// 條件
endpoint = env("ENV") == "prod"
  ? "https://prom-prod"
  : "https://prom-staging"

3.3 元件引用（dataflow）

// 元件間用 . 引用
discovery.kubernetes "pods" {
  role = "pod"
}

prometheus.scrape "kubernetes" {
  targets = discovery.kubernetes.pods.targets   // ← 引用 pods 元件的 targets export

prometheus.relabel "add_cluster" {
  forward_to = [prometheus.remote_write.default.receiver]
  rule {
    target_label = "cluster"
    replacement  = "production"
  }
}

§ 4. 完整 Alloy 範例

4.1 K8s Pod 全訊號收集

// ===== Discovery =====
discovery.kubernetes "pods" {
  role = "pod"
}

discovery.relabel "filter_pods" {
  targets = discovery.kubernetes.pods.targets

  // 只收 annotation prometheus.io/scrape=true 的 pod
  rule {
    source_labels = ["__meta_kubernetes_pod_annotation_prometheus_io_scrape"]
    regex         = "true"
    action        = "keep"
  }

  // 加 namespace label
  rule {
    source_labels = ["__meta_kubernetes_namespace"]
    target_label  = "namespace"
  }

  // 加 pod label
  rule {
    source_labels = ["__meta_kubernetes_pod_name"]
    target_label  = "pod"
  }
}

// ===== Metrics =====
prometheus.scrape "k8s_pods" {
  targets    = discovery.relabel.filter_pods.output
  forward_to = [prometheus.remote_write.mimir.receiver]

  scrape_interval = "15s"
  scrape_timeout  = "10s"
}

prometheus.remote_write "mimir" {
  endpoint {
    url = "http://mimir-distributor:9009/api/v1/push"
    headers = {
      "X-Scope-OrgID" = "tenant-prod",
    }
  }
}

// ===== Logs =====
loki.source.kubernetes "pods" {
  targets    = discovery.kubernetes.pods.targets
  forward_to = [loki.process.parse.receiver]
}

loki.process "parse" {
  forward_to = [loki.write.loki.receiver]

  // JSON 解析
  stage.json {
    expressions = {
      level = "level",
      msg   = "message",
    }
  }

  // 把 level 變 label
  stage.labels {
    values = {
      level = "level",
    }
  }
}

loki.write "loki" {
  endpoint {
    url = "http://loki:3100/loki/api/v1/push"
  }
}

// ===== Traces =====
otelcol.receiver.otlp "default" {
  grpc {
    endpoint = "0.0.0.0:4317"
  }
  http {
    endpoint = "0.0.0.0:4318"
  }

  output {
    traces = [otelcol.processor.batch.default.input]
  }
}

otelcol.processor.batch "default" {
  output {
    traces = [otelcol.exporter.otlp.tempo.input]
  }
}

otelcol.exporter.otlp "tempo" {
  client {
    endpoint = "tempo:4317"
    tls {
      insecure = true
    }
  }
}

§ 5. Alloy vs OTel Collector

5.1 Alloy 的本質

💡 Alloy = OTel Collector 的「distribution」

Alloy 不是「跟 OTel Collector 競爭」，是基於 OTel Collector 加上 Grafana 生態整合。
所有 OTel Collector 的 receiver / processor / exporter 都能用（前綴 otelcol.）。

5.2 三家 OTel distribution 對比

面向OTel Collector（vanilla）Grafana AlloyAWS Distro for OTel (ADOT)
誰OpenTelemetry 社群Grafana LabsAWS
設定YAMLRiver（更強）YAML
Prometheus 整合via receiver原生（discovery + scrape）via receiver
Loki 整合via exporter原生（loki.*）via exporter
Pyroscope 整合無原生（pyroscope.*）無
Beyla / eBPFupstream（OBI）原生（beyla.ebpf）無
K8s integration有限強AWS-focused
UI debug無內建 Web UI無

5.3 何時選哪個

  • Grafana Stack 為主：Alloy（最佳整合）
  • Vendor-neutral / 多 backend：OTel Collector
  • AWS 環境：ADOT 或 Alloy 都可
  • 需要 eBPF 自動 instrument：Alloy（內建 Beyla）

5.4 互通性

三者都用 OTLP 協議，可級聯：

App → Alloy（每 pod / node）→ OTel Collector Gateway（中央）→ Backend

實務上常見大型 setup：edge 用 Alloy（K8s DaemonSet），中央用 OTel Collector Gateway 做集中處理（sampling、enrichment）。

§ 6. Beyla 與 OBI

6.1 Beyla 的歷史

2023-07  Grafana 發布 Beyla
2024-05  整合進 Grafana Alloy
2025     Beyla 捐給 OpenTelemetry → OBI
         （OpenTelemetry eBPF Instrumentation）

6.2 OBI 是什麼

📌 名稱與授權變化

OBI（OpenTelemetry eBPF Instrumentation）是 Beyla 捐給 OpenTelemetry 後的 upstream 名稱。
Grafana Beyla 仍存在，是 Grafana Labs 的 OBI distribution（類似 Alloy 是 OTel Collector distribution）。
Beyla 大多數 code 從 OBI 上游 vendor 進來。新貢獻者送 PR 到 OBI 上游。

6.3 為什麼需要 eBPF auto-instrumentation

傳統 instrumentation：

  • Go：手動加 OTel SDK + tracepoint，重編譯
  • Java：加 -javaagent
  • Python：手動 import opentelemetry SDK

痛點：

  • 需改 code（廠商提供的 binary 不可能）
  • 多語言混合 → 每個都要做
  • 升級 SDK 要重 deploy
  • 沒 SDK 支援的語言（C / Erlang）束手無策

6.4 Beyla / OBI 的解

用 eBPF 在 kernel level 看 HTTP / gRPC traffic：

  • 不用改 code
  • 支援所有語言（Go / C / Rust / Python / Ruby / Java / Node / .NET 等）
  • 產生 RED metric（rate / errors / duration）
  • 產生 trace span（含 W3C traceparent header 自動傳播）
  • 輸出標準 OTLP，與 Tempo / Mimir 整合

6.5 Beyla 的限制

  • 只支援 HTTP/HTTPS/gRPC（不支援 message queue / DB protocol）
  • 只看 transaction-level（無 application-level context）
  • 需要 Linux kernel 5.8+ with BTF（或 RHEL/CentOS 4.18+ with backports）
  • 需要 elevated privileges（CAP_BPF 等）

6.6 Beyla vs SDK

面向OTel SDKBeyla / OBI
需改 code是否
語言支援各語言獨立 SDK所有 Linux 應用
細節function levelHTTP/gRPC transaction
自訂 attribute強有限
業務 span可不可
Overhead各語言不同低（kernel level）

Best Practice：兩者組合用。關鍵 service 用 SDK 拿細節；fleet 廣度用 Beyla。

§ 7. Beyla 在 Alloy 中

7.1 beyla.ebpf 元件

beyla.ebpf "default" {
  // K8s 整合
  attributes {
    kubernetes {
      enable       = "true"
      cluster_name = "production"
    }
  }

  // Discovery: 哪些服務要 instrument
  discovery {
    services {
      // 用 K8s pod label 篩選
      kubernetes {
        pod_labels = {
          instrument = "beyla",
        }
      }
    }
  }

  // 開啟 trace context propagation
  ebpf {
    track_request_headers       = true
    enable_context_propagation  = true
  }

  // 開啟哪些 metrics
  metrics {
    features = ["application"]   // application / network
  }

  // 輸出 traces 給 OTel pipeline
  output {
    traces = [otelcol.processor.batch.tempo.input]
  }
}

// scrape Beyla 自身產生的 Prometheus metric
prometheus.scrape "beyla" {
  targets    = beyla.ebpf.default.targets
  forward_to = [prometheus.remote_write.mimir.receiver]
}

7.2 K8s Pod 標記

# app-deployment.yaml
spec:
  template:
    metadata:
      labels:
        app: my-api
        instrument: beyla    # ← 讓 Beyla 自動 instrument
    spec:
      containers:
        - name: api
          image: my-api:1.0

            - containerPort: 8080

7.3 Discovery 條件

discovery {
  services {
    // 1. 用 port 篩選

§ 8. eBPF 原理速覽

8.1 什麼是 eBPF

eBPF (extended Berkeley Packet Filter) 讓使用者在 Linux kernel 「掛載」自訂程式（不重啟 kernel）。 程式跑在 kernel 的 sandbox VM 裡，安全 + 高效。

8.2 工作流程

  1. 寫 eBPF 程式（C / Rust）
  2. 編譯為 eBPF bytecode
  3. Loader 把 bytecode 推進 kernel
  4. Kernel verifier 驗證安全性（不 panic / 不洩漏記憶體）
  5. JIT 編譯為原生機器碼
  6. 掛載到 hook point（syscall / kprobe / uprobe / tracepoint）
  7. 事件觸發時執行
  8. 把資料送回 user-space（via maps / ring buffer）

8.3 Beyla 用的 hook point

  • uprobe：在 user-space binary 內函式入口（HTTP server handler）
  • kprobe：在 kernel function 入口（tcp_sendmsg）
  • tracepoint：kernel 提供的 stable hook（system call enter/exit）

8.4 Verifier 的安全保證

  • 不能無限 loop
  • 不能存取無效記憶體
  • 所有 register 必須初始化
  • 檢查每條指令的安全性

所以 eBPF 出 bug 不會 crash kernel，只會被 reject 或在 sandbox 內失敗。

8.5 eBPF 在 observability 的應用

用途工具
Auto-instrumentationBeyla / OBI / Pixie
ProfilingPyroscope / Parca
Network observabilityCilium / Hubble
SecurityTetragon / Falco
PerformanceBCC / bpftrace

§ 10. 從 Promtail / Grafana Agent 遷移

10.1 從 Promtail 遷移

原 Promtail config：

# promtail.yaml
clients:
  - url: http://loki:3100/loki/api/v1/push

scrape_configs:
  - job_name: kubernetes-pods
    kubernetes_sd_configs:
      - role: pod
    pipeline_stages:
      - json:
          expressions:
            level: level

Alloy 等價：

discovery.kubernetes "pods" {
  role = "pod"
}

loki.source.kubernetes "pods" {
  targets    = discovery.kubernetes.pods.targets
  forward_to = [loki.process.parse.receiver]
}

loki.process "parse" {
  forward_to = [loki.write.default.receiver]
  stage.json {
    expressions = { level = "level" }
  }
}

loki.write "default" {
  endpoint {
    url = "http://loki:3100/loki/api/v1/push"
  }
}

10.2 自動轉換工具

# 安裝 alloy CLI
brew install grafana/grafana/alloy

# 從 Promtail 轉
alloy convert --source-format=promtail \
  --output=alloy-config.river \
  promtail-config.yaml

# 從 Static Mode Grafana Agent 轉
alloy convert --source-format=static \
  --output=alloy-config.river \
  agent-config.yaml

# 從 Prometheus 轉
alloy convert --source-format=prometheus \
  --output=alloy-config.river \
  prometheus.yaml

10.3 Grafana Agent Flow Mode 遷移

Flow mode 的設定幾乎完全相容 Alloy（Alloy 就是 Flow mode 的延續）。 主要差別是 binary 名稱：

# 安裝 Alloy
sudo systemctl stop grafana-agent-flow
sudo apt install alloy
sudo systemctl enable alloy
sudo systemctl start alloy

10.4 Cutover 策略

  1. 新 cluster：直接用 Alloy
  2. 既有 Promtail：edge 上仍跑 Promtail，Alloy 部署到一邊驗證 OK 後切
  3. Grafana Agent：直接 in-place upgrade 到 Alloy

§ 11. Production Best Practices

11.1 設定組織

# 建議檔案結構
config/
├── main.river           # 主入口
├── discovery.river      # discovery 元件
├── metrics.river        # metric pipeline
├── logs.river           # log pipeline
├── traces.river         # trace pipeline
└── profiles.river       # profile pipeline

# Alloy 支援多檔載入
alloy run --config.path=config/

11.2 監控 Alloy 自身

// Alloy 內建 metric endpoint /metrics
prometheus.scrape "alloy_self" {
  targets    = [{ __address__ = "localhost:12345" }]
  forward_to = [prometheus.remote_write.mimir.receiver]
}

必看 metrics：
- alloy_component_evaluation_seconds  // 元件評估時間
- alloy_component_dependencies_wait   // 依賴等待
- prometheus_remote_storage_samples_in_total
- loki_write_sent_entries_total

必設 alert：
- AlloyComponentEvaluationFailing
- AlloyRemoteWriteBacklog

11.3 Pipeline 設計原則

  • Filter 越早越好：在 source 層 filter，不要送到後面再丟
  • Batch 在 export 前：減少 backend 寫入頻率
  • Sampling 集中在 gateway：edge agent 全送，gateway 統一決策
  • 避免複雜 transformation：能在 backend 做就在 backend 做

11.4 Beyla Cardinality 控制

// 限制 high-cardinality attribute
beyla.ebpf "default" {
  metrics {
    features = ["application"]

    // 排除 path 中 ID（避免 /api/users/12345 變 unique series）
    instrumentations = ["http"]

    // HTTP route 用 template，不是真實 path
    http_route_pattern = ["/api/users/:id", "/api/orders/:id"]
  }
}

11.5 安全性（K8s）

alloy:
  securityContext:
    privileged: true   // eBPF 需要

# 較嚴格做法：只給最小需要的 capabilities
securityContext:
  capabilities:
    add:
      - BPF
      - SYS_PTRACE
      - NET_RAW
      - DAC_READ_SEARCH
      - PERFMON
      - CHECKPOINT_RESTORE

11.6 Debug Workflow

Alloy 內建 Web UI（預設 port 12345）：

  • 看每個元件的當前狀態 / 健康度
  • 看資料流（哪個元件送多少 metric / log / trace）
  • 看設定錯誤（loop 中的元件）
  • 線上 reload 設定

  • Beyla 用 eBPF 做 zero-code auto-instrumentation：HTTP/gRPC RED + traces
  • 支援所有 Linux 應用（Go/Java/Python/Ruby/.NET/Node 等），不用改 code
  • 需要 kernel 5.8+ with BTF，elevated privileges
  • Beyla 限制：只看 transaction-level、無 application context、無 business span
  • Best Practice：SDK + Beyla 組合用（關鍵 service SDK，fleet 廣度 Beyla）
  • Beyla v2.5+ 支援 GenAI（OpenAI / Anthropic）API trace
  • 大規模架構：edge DaemonSet + central Alloy Gateway
  • Migration：alloy convert 從 Promtail / Grafana Agent / Prometheus 自動轉
