Chapter 45 · Loki Log Aggregation

Loki：Prometheus-style 的 Log Aggregation

歡迎來到 Part 5 — Grafana 生態系。本章開始 LGTM Stack 的「L」：Loki， Grafana Labs 推出的 log 系統。設計哲學「不索引內容、只索引 label」讓它比 Elasticsearch 便宜 10 倍。 本章涵蓋架構、LogQL、部署模式（單 binary 到 microservices）、agent 選擇（Promtail vs Alloy）。

§ 1. 為什麼需要 Loki

1.1 既有 log 系統的痛點

方案痛點
Elasticsearch + Kibana (ELK) Full-text indexing 成本高、query 慢、JVM 吃記憶體、運維複雜
Splunk 商業授權、按 GB 收費、scale 大時超貴
檔案 + grep 分散式環境不可行、無 retention 管理
CloudWatch / Stackdriver vendor lock-in、跨 cloud 困難、價格高

1.2 Loki 的位置

Loki 由 Grafana Labs 在 2018 創立，靈感來自 Prometheus：「就像 Prometheus 但是給 log」。 成本是 Elasticsearch 的 1/10，與 Grafana 完美整合。

1.3 Loki 不適合的場景

  • Full-text search 為主（電商搜商品 / 知識庫）→ 用 Elasticsearch
  • Compliance 需要不可變 log（SOX / HIPAA）→ Splunk + WORM storage
  • 需要複雜 NLP 分析 → ELK 或專業工具

但對 SRE 場景的 log（debug、incident 排查、metric 萃取），Loki 是首選。

§ 2. 核心設計哲學

2.1 Label-only Indexing

💡 核心洞察

多數 log 系統都把 log 全文 indexed，造成 index 膨脹。
Loki 觀察：「查 log 時，通常先用 label 縮小範圍（service=api, namespace=prod）， 然後 grep 內容」。所以：
• 只索引 label（namespace / pod / app / level / ...）
• log 內容存壓縮 chunk，按 label 分組
• Query 時：先用 label 找到 chunks → 再 grep 內容

2.2 為什麼這樣便宜

  • Index 小（10:1 vs full-text）→ RAM 與 SSD 用量低
  • Storage 用便宜的 object store（S3 / GCS）— 不用昂貴 SSD
  • 無 schema、無 mapping，運維簡單
  • 跟 Prometheus 用一樣的 label model，學習成本低

2.3 取捨

沒有免費午餐：

  • Full-text query 比 ES 慢
  • 跨 stream 的 join 不容易
  • Cardinality 爆炸時也會壞（跟 Prometheus 一樣）

§ 3. 架構與元件

3.1 邏輯路徑

graph LR APP[Application] AGENT[Agent · Promtail · Alloy] DIST[Distributor] INGEST[Ingester] OBJ[(Object Storage
S3 · GCS · Azure)] QUERIER[Querier] QF[Query Frontend] USER[User · Grafana] APP -- log lines --> AGENT AGENT -- HTTP push --> DIST DIST -- shards --> INGEST INGEST -- write chunks --> OBJ USER -- LogQL --> QF QF -- splits query --> QUERIER QUERIER -- read chunks --> OBJ QUERIER -- read recent --> INGEST

3.2 主要元件職責

元件職責
Distributor 接收 push、validation、按 hash 路由到 ingester
Ingester 記憶體中累積 chunks（壓縮 log lines）、定期 flush 到 object store
Query Frontend 切分大查詢為小查詢、cache、平行化
Querier 從 ingester（最近）+ object store（歷史）讀資料、執行 LogQL
Compactor 合併 ingester flush 的小 chunks，提升 query 效能
Ruler 評估 alerting / recording rule（從 log 算 metric）

3.3 Stream 與 Chunk

Stream：一組 label 的 log 集合（例如 {namespace="prod", app="api"}）。
Chunk：Stream 內按時間切分的壓縮資料塊（10MB 或 2 小時）。 

Stream: {namespace="prod", app="api", pod="api-1"}
  ├── Chunk 1: 10:00-12:00, 10MB compressed
  ├── Chunk 2: 12:00-14:00, 10MB compressed
  └── Chunk 3: 14:00-now,   in memory

3.4 Schemaconfig 與 Storage

v3+ 推薦使用 tsdb schema（取代舊 boltdb-shipper）：

schema_config:
  configs:
    - from: 2024-01-01
      store: tsdb
      object_store: s3
      schema: v13
      index:
        prefix: index_
        period: 24h

§ 4. 三種部署模式

4.1 三種模式對比

模式適合每天 log 量
Monolithic（single binary） PoC、小團隊 < 100 GB
Simple Scalable 中型企業 100 GB - 1 TB
Microservices 大規模、多租戶 > 1 TB

4.2 Monolithic 範例（Helm）

helm repo add grafana https://grafana.github.io/helm-charts

helm install loki grafana/loki \
  --namespace logging --create-namespace \
  --set deploymentMode=SingleBinary \
  --set loki.commonConfig.replication_factor=1 \
  --set loki.storage.type=filesystem

4.3 Simple Scalable（推薦中型）

# values.yaml
deploymentMode: SimpleScalable

loki:
  commonConfig:
    replication_factor: 3
  storage:
    type: s3
    bucketNames:
      chunks: loki-chunks
      ruler:  loki-ruler
    s3:
      region: us-east-1
      endpoint: s3.us-east-1.amazonaws.com

read:
  replicas: 3
backend:
  replicas: 3
write:
  replicas: 3

三組 stateful pods：write（含 distributor + ingester）、read（querier + frontend）、backend（compactor + ruler）。

4.4 Microservices 模式

每個元件獨立 deploy，可獨立 scale。適合 1TB+/day 的大規模。詳見 官方文件。

4.5 部署模式漸進路徑

  1. POC → Monolithic + filesystem
  2. 進入 production → Simple Scalable + S3
  3. 規模 > 1TB/day → Microservices
  4. 多租戶 SaaS → Microservices + multi-tenant config

§ 5. LogQL 完整語法

5.1 基本結構：5 層

Layer 1: Stream Selector（必有）
  {namespace="prod", app="api"}

Layer 2: Line Filter（可選，可多層）
  {app="api"} |= "error"

Layer 3: Parser（可選）
  {app="api"} |= "error" | json
  {app="api"} | logfmt
  {app="api"} | regexp `(?P\w+)`

Layer 4: Label Filter（解析後）
  {app="api"} | json | level="error"

Layer 5: Aggregation（變 metric）
  sum by (level) (rate({app="api"}[5m]))

5.2 Stream Selector

跟 PromQL 一樣的 matcher 語法：

{app="api"}                       # 等於
{app=~"api|web"}                  # regex 匹配
{app="api", env!="dev"}           # 不等於
{cluster=~"prod-.+"}              # regex
{__name__="some_metric"}          # 不適用 Loki（這是 PromQL）

5.3 Line Filter

運算子意義性能
|=包含最快
!=不包含快
|~regex 匹配慢
!~regex 不匹配慢

多 filter 串接（從左到右）：

{app="api"} |= "error" != "test" |~ "user_\d+"

5.4 Parser

# JSON parser
{app="api"} | json
# 自動把 JSON 欄位變成 label

# 指定特定欄位
{app="api"} | json level, message

# Logfmt parser（key=value）
{app="api"} | logfmt
# log line: level=info msg="hello" duration=1s
# 變 label: level, msg, duration

# Regex parser
{app="api"} | regexp `(?P\w+) (?P/\S+) (?P\d+)`

# Pattern parser（v2.7+，比 regex 快）
{app="api"} | pattern `  `

# Unpack（用於 Promtail 預先打包的 log）
{app="api"} | unpack

5.5 Label Filter（解析後使用）

{app="api"} | json | level="error"
{app="api"} | logfmt | duration > 1s
{app="api"} | json | status_code >= 400 and method="POST"
{app="api"} | json | __error__=""    # 過濾解析錯誤的

5.6 Line Format（修改顯示）

{app="api"}
  | json
  | line_format `[{{.level}}] {{.user_id}} - {{.message}}`

5.7 Metric Queries（從 log 算 metric）

# Rate（每秒 log 行數）
rate({app="api"}[5m])

# Count over time
count_over_time({app="api"} |= "error" [5m])

# Bytes over time（log size）
bytes_over_time({app="api"}[5m])

# 從解析後欄位算數值
quantile_over_time(0.99,
  {app="api"} | json | unwrap duration_ms [5m]
)

# Aggregation
sum by (level) (rate({app="api"} | json [5m]))
topk(10, sum by (path) (count_over_time({app="api"}[5m])))

5.8 LogQL 性能優化

⚠️ Loki 性能黃金規則
  1. Stream selector 越精確越好：{app="api"} 比 {app=~".+"} 快 10000 倍
  2. Line filter 越早越好：|= 在 parser 之前才有效
  3. 用 |= 不用 |~：regex 慢，能用字串就用
  4. 避開 high-cardinality label：log 行 ID、UUID 不該當 label
  5. 限制 time range：1h 比 24h 快 24 倍

limits_config:
  retention_period: 720h    # 30 天

# 不同 tenant 不同 retention

  retention_period: 720h
  retention_stream:
    - selector: '{namespace="audit"}'
      priority: 1
      period: 8760h         # audit log 保留 1 年

6.4 Compactor（必設）

compactor:
  working_directory: /var/loki/compactor
  shared_store: s3
  compaction_interval: 10m
  retention_enabled: true
  retention_delete_delay: 2h

6.5 成本估算範例

1 TB log/月，30 天 retention：

  • S3 storage（30TB total，with 30 天 lifecycle）：~$700/月
  • Loki compute（3 read + 3 write + 3 backend）：~$300/月
  • 合計：~$1000/月
  • 同等規模 ES 通常需要 $5000-10000/月

§ 7. Promtail vs Alloy（Agent）

7.1 Loki 的 agent 演進

Agent狀態
Promtail傳統 Loki agent，2025 年起 deprecated
Grafana Agent合併 Promtail + Prometheus + OTel collector，2024 deprecated
Grafana Alloy2024 年起的主推 agent，整合所有功能
⚠️ Promtail Deprecated

Promtail 在 2025 年正式 deprecated，新部署用 Alloy。
既有 Promtail 部署仍能用，但長期需遷移。Alloy 完全相容 Promtail config，遷移簡單。

7.2 Alloy 的 Loki 設定範例

// alloy config.river
loki.source.kubernetes_pods "default" {
  forward_to = [loki.write.default.receiver]
}

loki.process "default" {
  forward_to = [loki.write.default.receiver]

  // JSON 解析
  stage.json {
    expressions = {
      level   = "level",
      message = "msg",
    }
  }

  // 加 label
  stage.labels {
    values = {
      level = "level",
    }
  }
}

loki.write "default" {
  endpoint {
    url = "http://loki:3100/loki/api/v1/push"
  }
}

7.3 K8s 部署

helm repo add grafana https://grafana.github.io/helm-charts
helm install alloy grafana/alloy \
  --namespace logging \
  --set alloy.configMap.create=true

7.4 為什麼 Alloy 優於 Promtail

  • 支援所有 Grafana stack（metrics + logs + traces + profiles）
  • OpenTelemetry 原生支援
  • 更強的 pipeline 表達力（River 語法）
  • 持續開發新 feature（Promtail 已凍結）

§ 8. 多租戶

8.1 Tenant 概念

Loki 支援多租戶：每個 tenant 有獨立的 log streams、quota、retention。 Tenant 由 HTTP header X-Scope-OrgID 識別。

8.2 啟用多租戶

auth_enabled: true

8.3 Push log 帶 tenant

# 用 Alloy
loki.write "tenant_a" {
  endpoint {
    url = "http://loki:3100/loki/api/v1/push"
    tenant_id = "tenant-a"
  }
}

# 直接 curl
curl -H "X-Scope-OrgID: tenant-a" \
  -H "Content-Type: application/json" \
  -X POST \
  -d '{...}' \
  http://loki:3100/loki/api/v1/push

8.4 Query 帶 tenant

# Grafana datasource
- name: Loki-TenantA
  type: loki
  url: http://loki:3100
  jsonData:
    httpHeaderName1: 'X-Scope-OrgID'
  secureJsonData:
    httpHeaderValue1: 'tenant-a'

overrides:
  tenant-a:
    ingestion_rate_mb: 10
    max_streams_per_user: 10000
    retention_period: 168h     # 7 天

  tenant-b:
    ingestion_rate_mb: 50

§ 9. 與 Prometheus + Tempo 整合

9.1 三本柱串接（回顧 Ch 34）

  • Log → Metric：用 LogQL aggregation 從 log 算 metric
  • Log → Trace：Loki datasource derivedFields 從 log 抓 trace_id
  • Metric → Log：在 Grafana exemplar 跳，或 Drilldown 跨

9.2 Loki Ruler（從 log 寫 alert）

ruler:
  alertmanager_url: http://alertmanager:9093
  enable_alertmanager_v2: true
  storage:
    type: s3
    s3:
      bucketnames: loki-rules

# rules.yaml
groups:
  - name: api-errors
    rules:
      - alert: HighErrorLogRate
        expr: |
          sum by (app) (
            rate({namespace="prod"} |~ "ERROR|FATAL" [5m])
          ) > 10
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "{{ $labels.app }} error log rate is {{ $value }}/s"

讓 log pattern 觸發 alert，跟 Prometheus alerting 整合。

9.3 Recording Rules

groups:
  - name: log-recording
    rules:
      # 從 log 算 metric，pre-aggregate
      - record: api:log_errors:rate5m
        expr: |
          sum by (app) (rate({namespace="prod"} |~ "ERROR" [5m]))

產生的 metric 寫到 Mimir / Prometheus，dashboard 用便宜的 metric 而非每次 query Loki。

§ 10. Production Best Practices

10.1 Label 設計

✅ Label 設計原則
  • 低 cardinality：每個 label 值不超過 1000 個
  • 有限數量：每個 stream 不超過 10-15 個 label
  • 常用過濾條件：namespace / app / level / cluster
  • 不要用：user_id、session_id、request_id（cardinality 爆炸）
  • 動態欄位用 parser：log 內容用 JSON / logfmt，需要時 parse 出來

10.2 Cardinality 控制

limits_config:
  # 限制 stream 數量
  max_streams_per_user: 10000
  max_global_streams_per_user: 5000

  # 限制 label 數
  max_label_name_length: 1024
  max_label_value_length: 2048
  max_label_names_per_series: 30

  # Ingestion rate
  ingestion_rate_mb: 10
  ingestion_burst_size_mb: 20

10.3 Query Limits

limits_config:
  max_query_series: 500          # 單 query 最多 500 series
  max_query_parallelism: 32
  max_query_length: 720h          # 不能查超過 30 天
  query_timeout: 1m
  max_entries_limit_per_query: 5000

§ 11. 常見陷阱

⚠️ 陷阱 1：High Cardinality 拖垮 Loki

把 request_id 設成 label → 每個 request 一個 stream → ingester 爆。
解法：移到 log 內容（JSON），需要時用 parser 取出。

⚠️ 陷阱 2：Stream selector 太廣

{} 或 {cluster=~".+"} 會 scan 所有 stream，超慢。
解法：永遠至少有一個 = matcher。

⚠️ 陷阱 3：忘了 retention

Object storage 不會自己清理過期資料，必設 retention_enabled: true，
也要設 S3 lifecycle policy（避免 compactor 來不及刪）。

⚠️ 陷阱 4：Replication factor 設不對

replication_factor: 1 = 沒 HA，3 = 寫 3 份。
Production 必設 3，否則 ingester 重啟會丟 in-memory chunks。

⚠️ 陷阱 5：用 line filter |~ 而非 |=

{app="api"} |~ "error" 用 regex 引擎，比 {app="api"} |= "error" 慢數倍。
能用字串就用字串。
