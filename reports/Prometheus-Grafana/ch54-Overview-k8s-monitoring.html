§ 1. K8s 監控的層次

1.1 五個監控層次

K8s 監控不只是「應用 metric」。完整 stack 需考慮五層：

層次監控內容來源
L1: 基礎設施 Node CPU / Memory / Disk / Network Node Exporter
L2: K8s Cluster Pod / Deployment / Service 狀態 kube-state-metrics
L3: Container Container CPU / Memory / IO cAdvisor (kubelet)
L4: K8s Control Plane API server / etcd / scheduler K8s 自己（kubelet metrics）
L5: Application 業務 metric / RED metric App SDK / Beyla / OTel

1.2 為什麼分層重要

排查問題時要從最低層往上：

  • 「服務慢」→ 看 L5 應用 metric
  • 「應用看起來正常」→ 看 L3 container（被 limit 了？）
  • 「container 健康」→ 看 L1 node（host 資源耗盡？）
  • 「node 健康」→ 看 L4 control plane（scheduler 出錯？）
  • 「scheduler 正常」→ 看 L2 cluster state（pod 沒被排上？）

1.3 不要忽略的層

⚠️ 常見漏監控
  • Control plane 沒監控：API server 慢時所有 deployment 都會卡，但無人知
  • etcd 沒監控：etcd disk 滿 → cluster 崩
  • 節點 disk 沒監控：disk pressure 觸發 pod eviction
  • Pod limit 沒監控：CPU throttling 看不到，誤判應用慢

§ 2. 三大關鍵 exporter

2.1 Node Exporter

負責L1（Node）。每個 node 跑一個 pod（DaemonSet）。提供 OS-level metric：

  • CPU 使用率（user / system / idle / iowait / steal）
  • Memory（available / used / cached / swap）
  • Disk（usage / IOPS / latency）
  • Network（rx / tx / errors / drops）
  • Load average
  • File descriptors
  • Uptime

常見 metric：

# CPU
node_cpu_seconds_total{mode="idle"}
node_load1 / node_load5 / node_load15

# Memory
node_memory_MemTotal_bytes
node_memory_MemAvailable_bytes
node_memory_Cached_bytes

# Disk
node_filesystem_avail_bytes
node_filesystem_size_bytes
node_disk_io_time_seconds_total

# Network
node_network_receive_bytes_total
node_network_transmit_bytes_total
node_network_receive_errs_total

2.2 kube-state-metrics (KSM)

負責L2（Cluster）。把 K8s API 的 object state 變成 Prometheus metric。不是監控 K8s 本身效能，是監控 K8s 內 object 狀態。

常見 metric：

# Pod
kube_pod_status_phase{phase="Running"}
kube_pod_container_status_restarts_total
kube_pod_container_resource_requests
kube_pod_container_resource_limits

# Deployment
kube_deployment_status_replicas
kube_deployment_status_replicas_available
kube_deployment_spec_replicas
kube_deployment_status_condition

# Node
kube_node_status_condition{condition="Ready",status="true"}
kube_node_status_allocatable
kube_node_spec_taint

# Namespace
kube_namespace_status_phase

# Service / Ingress / PVC / Job / CronJob 等
# 幾乎所有 K8s object 都有對應 metric

2.3 cAdvisor

負責L3（Container）。內建在 kubelet，不用單獨部署。 收集每個 container 的資源使用： 

# 每個 container 的 metric
container_cpu_usage_seconds_total
container_memory_usage_bytes
container_memory_working_set_bytes
container_cpu_cfs_throttled_seconds_total   # CPU throttle（重要！）
container_network_receive_bytes_total
container_fs_usage_bytes

從 kubelet 的 /metrics/cadvisor endpoint scrape。

2.4 三家組合的 metric power

實務上，查詢常結合三家。例：

# 「找出 CPU 用量超過 limit 80% 的 container」
# 需要 cAdvisor（usage）+ KSM（limit）

(
  rate(container_cpu_usage_seconds_total[5m])
)
/
on (namespace, pod, container)
group_left
(
  kube_pod_container_resource_limits{resource="cpu"}
)
> 0.8

§ 3. kube-prometheus-stack 概覽

3.1 是什麼

kube-prometheus-stack（前身 prometheus-operator helm chart）是業界標準的 K8s 監控解決方案。它整合：

  • Prometheus（with Prometheus Operator）
  • Alertmanager
  • Grafana
  • kube-state-metrics
  • Node Exporter（DaemonSet）
  • 預配置 ServiceMonitor / PodMonitor
  • 預載 dashboard（Node / K8s Cluster / Workload）
  • 預設 alert rule（K8s 標準）

3.2 元件關係

graph LR OP[Prometheus Operator] PROM[Prometheus
StatefulSet] AM[Alertmanager
StatefulSet] GRAF[Grafana] KSM[kube-state-metrics] NE[Node Exporter
DaemonSet] APP[Application Pods] OP -- 管理 --> PROM OP -- 管理 --> AM OP -- watch ServiceMonitor --> APP PROM -- scrape --> KSM PROM -- scrape --> NE PROM -- scrape · cAdvisor --> APP PROM -- scrape via SM/PM --> APP PROM -- alerts --> AM GRAF -- query --> PROM

3.3 Prometheus Operator

Operator 概念（Ch 44 介紹過）：用 K8s CRD 管 Prometheus instance、ServiceMonitor、PodMonitor、PrometheusRule、AlertManagerConfig。 

kind: Prometheus           # Prometheus instance（可多個）
kind: ServiceMonitor       # 透過 Service 找 scrape target
kind: PodMonitor           # 直接 scrape pod
kind: Probe                # blackbox-style 監控
kind: PrometheusRule       # alerting / recording rule
kind: Alertmanager         # AM instance
kind: AlertmanagerConfig   # AM 設定
kind: ScrapeConfig         # 自訂 scrape config（K8s 外目標）

§ 5. 重要 metrics 詳解

5.1 Pod 健康度

# Pod 是否 Running
kube_pod_status_phase{phase="Running"} == 1

# Pod restart 次數（high 表示可能有問題）
rate(kube_pod_container_status_restarts_total[15m]) > 0

# Pod 是否 Ready
kube_pod_status_ready{condition="true"} == 1

# 找 CrashLoopBackOff 的 pod
kube_pod_container_status_waiting_reason{reason="CrashLoopBackOff"} == 1

5.2 Resource 使用

# CPU 使用 vs request
sum by (namespace, pod) (rate(container_cpu_usage_seconds_total{container!=""}[5m]))
/
sum by (namespace, pod) (kube_pod_container_resource_requests{resource="cpu"})

# Memory 使用 vs limit
sum by (namespace, pod) (container_memory_working_set_bytes{container!=""})
/
sum by (namespace, pod) (kube_pod_container_resource_limits{resource="memory"})

# CPU throttling（容易被忽略！）
sum by (namespace, pod) (rate(container_cpu_cfs_throttled_seconds_total[5m]))

5.3 Node 健康

# Node Ready
kube_node_status_condition{condition="Ready",status="true"} == 1

# Node 上 pod 數 vs 容量
sum by (node) (kube_pod_info)
/
kube_node_status_capacity{resource="pods"}

# Node disk pressure / memory pressure
kube_node_status_condition{condition="MemoryPressure",status="true"} > 0
kube_node_status_condition{condition="DiskPressure",status="true"} > 0
kube_node_status_condition{condition="PIDPressure",status="true"} > 0

5.4 Deployment / StatefulSet

# Available replicas vs desired
kube_deployment_status_replicas_available
/
kube_deployment_spec_replicas

# Stuck rollout
kube_deployment_status_observed_generation
!=
kube_deployment_metadata_generation

5.5 PVC / Storage

# PVC 用量
kubelet_volume_stats_used_bytes
/
kubelet_volume_stats_capacity_bytes

# 預測 4 天內滿
predict_linear(kubelet_volume_stats_available_bytes[6h], 4 * 86400) < 0

5.6 Control Plane

# API server latency
histogram_quantile(0.99,
  rate(apiserver_request_duration_seconds_bucket[5m])
)

# API server error rate
sum(rate(apiserver_request_total{code=~"5.."}[5m]))
/
sum(rate(apiserver_request_total[5m]))

# etcd disk sync 延遲
histogram_quantile(0.99,
  rate(etcd_disk_wal_fsync_duration_seconds_bucket[5m])
)

§ 6. ServiceMonitor / PodMonitor

6.1 ServiceMonitor

對應 K8s Service，找到背後的 endpoints scrape。

apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: my-app
  namespace: production
  labels:
    release: monitoring   # 重要：matchLabels Prometheus selector
spec:
  selector:
    matchLabels:
      app: my-app
  namespaceSelector:
    matchNames:
      - production
  endpoints:
    - port: metrics       # 必須是 Service 中定義的 port name
      interval: 30s
      path: /metrics
      relabelings:
        - sourceLabels: [__meta_kubernetes_namespace]
          targetLabel: namespace

6.2 PodMonitor

直接從 Pod scrape，不透過 Service。適合沒有 Service的場景（如 Jobs / CronJobs）。

apiVersion: monitoring.coreos.com/v1
kind: PodMonitor
metadata:
  name: my-job
  namespace: production
spec:
  selector:
    matchLabels:
      job-name: my-job
  podMetricsEndpoints:
    - port: metrics
      interval: 30s

6.3 ServiceMonitor 沒被 Prometheus 抓到？

常見問題排查：

  1. Label 不對：Prometheus operator 預設只抓有 release: monitoring label 的 SM
  2. Namespace 不對：預設可能限制 namespace
  3. Port name 不對：要與 Service 中的 port name 完全一致
  4. Selector 沒 match：service 沒有對應 label

解法：在 values.yaml 設 serviceMonitorSelectorNilUsesHelmValues: false 讓 Prometheus 抓全 cluster 的 ServiceMonitor。

8.2 自加應用層 alert

apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: app-alerts
  namespace: production
spec:
  groups:
    - name: app
      rules:
        - alert: HighErrorRate
          expr: |
            sum by (service) (rate(http_requests_total{status=~"5.."}[5m]))
            /
            sum by (service) (rate(http_requests_total[5m]))
            > 0.05
          for: 5m
          labels:
            severity: critical
          annotations:
            summary: "{{ $labels.service }} error rate > 5%"

        - alert: HighLatency
          expr: |
            histogram_quantile(0.99,
              sum by (service, le) (rate(http_request_duration_seconds_bucket[5m]))
            ) > 1
          for: 10m
          labels:
            severity: warning

        - alert: ContainerCPUThrottling
          expr: |
            sum by (namespace, pod, container) (rate(container_cpu_cfs_throttled_seconds_total[5m]))
            > 0.5
          for: 15m
          labels:
            severity: warning
          annotations:
            summary: "{{ $labels.pod }} CPU 被 throttle > 50%"
            description: "考慮提高 CPU limit"

        - alert: PodMemoryUsageHigh
          expr: |
            sum by (namespace, pod) (container_memory_working_set_bytes{container!=""})
            /
            sum by (namespace, pod) (kube_pod_container_resource_limits{resource="memory"})
            > 0.9
          for: 10m
          labels:
            severity: warning

9.1 各元件 resource 估算

元件小型（< 50 nodes）中型（50-200）大型（> 200）

9.2 Storage 估算

# Prometheus storage 公式
storage_per_day = active_series × samples_per_day × bytes_per_sample × compression_factor

# 範例：100 nodes, 200 pods/node = 20K pods
# 每 pod 100 metrics = 2M active series
# 每 series 240 samples/h × 24h = 5760/day
# bytes_per_sample ≈ 1.7 (壓縮後)

storage_per_day = 2M × 5760 × 1.7 = ~20 GB/day
30 days storage = ~600 GB

# 加上 30% buffer + WAL
建議 PV 大小: 800 GB

9.3 何時 shard Prometheus

單一 Prometheus 撐不住時：

  • > 10M active series → 考慮 shard
  • > 30 GB memory 需求 → 考慮 shard
  • 需要更長 retention → 用 Mimir / Thanos
prometheusSpec:
  replicas: 2     # HA pair
  shards: 4       # 4 shards × 2 replicas = 8 instances

Shards 用 hash 分配 target，不能 cross-shard query（用 Mimir 才能聚合）。

§ 10. Production Best Practices

10.1 Cardinality 控制

prometheusSpec:
  # 限制單一 scrape 的 sample 數
  enforcedSampleLimit: 10000
  enforcedTargetLimit: 1000

  # Drop high-cardinality label
  additionalScrapeConfigs:
    - job_name: my-app
      metric_relabel_configs:
        - source_labels: [request_id]
          action: labeldrop

10.2 Recording Rules 加速

常用 query 預先算成 metric：

groups:
  - name: app:recording
    interval: 30s
    rules:
      - record: namespace:http_requests:rate5m
        expr: sum by (namespace) (rate(http_requests_total[5m]))

      - record: cluster:memory_usage:percent
        expr: |
          sum(node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes)
          / sum(node_memory_MemTotal_bytes) * 100

10.3 Long-term Storage

Production 必設 long-term storage（不能只靠本地 PV）：

prometheusSpec:
  remoteWrite:
    - url: http://mimir-distributor:9009/api/v1/push
      writeRelabelConfigs:
        # 過濾不重要的 metric 不送
        - sourceLabels: [__name__]
          regex: 'go_.*|process_.*'
          action: drop

10.4 監控 Prometheus 自身

必看 metrics：
- prometheus_tsdb_head_series          # active series
- prometheus_target_scrape_pool_targets
- prometheus_rule_evaluation_duration_seconds
- prometheus_remote_storage_samples_pending
- up{job="prometheus"}

必設 alert：
- PrometheusTSDBCorruption
- PrometheusOutOfMemory
- PrometheusHighRuleEvaluationDuration
- PrometheusRemoteWriteBacklog
- PrometheusTargetScrapingSlow

10.5 安全性

  • Prometheus / Grafana 不對外：用 Ingress + auth proxy
  • NetworkPolicy：限制 monitoring namespace 對外連線
  • Pod Security：non-root、read-only filesystem
  • Secret 管理：用 ExternalSecrets 從 Vault / SecretsManager 拉

§ 11. Troubleshooting

11.1 Prometheus OOM

常見原因：

  • active series 太多（cardinality 爆）
  • 太大 query 拖垮（> 50M samples）
  • WAL replay 時 memory 暫時飆高（重啟後）

解法：

  • 找 high-cardinality metric：topk(10, count by (__name__)({__name__=~".+"}))
  • 用 metric_relabel_configs drop 不需要的 metric
  • 增加 memory limit 或 shard
  • 用 Mimir 取代

11.2 Target 沒被 scrape

  1. UI: Status → Targets，看 target 狀態
  2. UI: Status → Configuration，確認 scrape config 對
  3. UI: Status → Service Discovery，看 SD 找到的 target
  4. 檢查 ServiceMonitor label 是否與 Prometheus selector 一致
  5. 檢查 NetworkPolicy 是否 block

11.3 Alert 沒觸發

  1. UI: Alerts，看 alert state
  2. UI: Status → Rules，看 rule 是否載入
  3. 用 PromQL 直接跑 alert expression，看結果
  4. 檢查 Alertmanager 是否收到（AM UI）
  5. 檢查 routing tree 是否走到 receiver

11.4 Grafana 看不到資料

  1. Datasource Test：Save & Test 看是否通
  2. Explore 直接查 metric 是否存在
  3. 檢查 datasource URL：應指向 Service 名稱（cluster 內 DNS）
  4. 檢查 time range：可能查的時間沒資料
  5. 檢查 variable：dashboard variable 可能沒選對

  • cAdvisor metric：特別注意 container_cpu_cfs_throttled_seconds_total（CPU throttle 容易被忽略）
  • kube-prometheus-stack：業界標準 Helm chart，整合 Prometheus + AM + Grafana + KSM + Node Exporter + 預設 dashboard / alert
  • Prometheus Operator 用 CRD 管：ServiceMonitor / PodMonitor / PrometheusRule / Alertmanager / Probe
  • 必設 selector: 用 serviceMonitorSelectorNilUsesHelmValues: false 抓全 cluster SM
  • Production 必做：HA（2 replicas）+ Persistent storage + remote_write 到 Mimir 做 long-term
  • Resource 估算：> 10M series 考慮 shard，> 200 nodes 用 4 shards × 2 replicas
  • 必備 dashboard：K8s/Cluster + Compute Resources + Node Exporter Full + 應用 RED dashboard
  • 必設 alert：CrashLooping / NotReady / Replicas mismatch / Disk filling / API errors / etcd / OOM
  • Cardinality 控制：enforcedSampleLimit + metric_relabel_configs labeldrop
  • Recording Rules 加速常用 query
  • 排查思路：Prometheus OOM 找 high-cardinality / target 沒抓檢查 SM label / alert 沒觸發看 rule loaded
