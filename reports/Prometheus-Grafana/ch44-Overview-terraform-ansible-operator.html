§ 1. 編排層 IaC 的角色

1.1 什麼是「編排層」

前面兩章的 Provisioning 與 SDK 解決的問題是：「Grafana 內的 resources（dashboard / alert / datasource）怎麼用 IaC 管理」。

但實際 production 還有更高層問題：

  • 多個 Grafana instance（dev / staging / prod / 多 region）
  • Grafana 自己的部署（K8s / VM / 雲服務）
  • Backend 依賴（PostgreSQL DB、Loki、Tempo、Mimir）
  • 跨團隊權限管理（多 org、多 team）
  • 跟其他基礎設施的依賴（VPC、IAM、Secrets Manager）

編排層工具解決這些。

1.2 三種編排工具

工具適用
Terraform多 cloud / 多 Grafana instance / 跟其他 infra 一起管
Ansible傳統 VM 環境、agent-less 配置管理
Grafana Operator純 K8s 環境，跟 ArgoCD 配 GitOps

1.3 與 Provisioning / SDK 的關係

編排層（Terraform / Operator）
  └── 部署 Grafana 自己 + DB
  └── 管理 datasource、folder、user、team
  └── 引用 dashboard JSON（可能 SDK 產生）

中間層（Foundation SDK）
  └── 產生 dashboard JSON

底層（Grafana 本身）
  └── 載入 Provisioning files

三層互補。

§ 2. Terraform Provider

2.1 Grafana Terraform Provider

官方 provider，名為 grafana/grafana（registry 上）。 支援大多數 Grafana resources，包含 dashboard、datasource、folder、team、user、alert、organization、library panel 等。

2.2 安裝

# versions.tf
terraform {
  required_version = ">= 1.5"
  required_providers {
    grafana = {
      source  = "grafana/grafana"
      version = "~> 3.0"
    }
  }
}

# main.tf
provider "grafana" {
  url  = "https://grafana.example.com"
  auth = var.grafana_token   # 用 Service Account token
}

2.3 多 instance 設定

provider "grafana" {
  alias = "production"
  url   = "https://grafana-prod.example.com"
  auth  = var.grafana_prod_token
}

provider "grafana" {
  alias = "staging"
  url   = "https://grafana-staging.example.com"
  auth  = var.grafana_staging_token
}

# 用 alias 指定 provider
resource "grafana_folder" "prod_folder" {
  provider = grafana.production
  title    = "Production"
}

resource "grafana_folder" "staging_folder" {
  provider = grafana.staging
  title    = "Staging"
}

2.4 Cloud Stack Provider（Grafana Cloud）

若用 Grafana Cloud，還有第二個 provider 管 stack 自身（建立 / 刪 cloud instance）：

provider "grafana" {
  alias        = "cloud"
  cloud_access_policy_token = var.gc_token
}

# 建一個 Cloud Stack
resource "grafana_cloud_stack" "team_a" {
  provider    = grafana.cloud
  name        = "team-a"
  slug        = "team-a"
  region_slug = "prod-us-east-0"
}

  type = string

resource "grafana_data_source" "prometheus" {
  type        = "prometheus"
  name        = "Prometheus"
  url         = "http://prometheus.monitoring.svc.cluster.local:9090"
  is_default  = true
  uid         = "prometheus-prod"

  json_data_encoded = jsonencode({
    httpMethod         = "POST"
    manageAlerts       = true
    prometheusType     = "Prometheus"
    prometheusVersion  = "3.11.2"
    timeInterval       = "15s"
    cacheLevel         = "High"
    incrementalQuerying = true
  })
}

resource "grafana_data_source" "loki" {
  type        = "loki"
  name        = "Loki"
  url         = "http://loki.monitoring.svc.cluster.local:3100"
  uid         = "loki-prod"

  json_data_encoded = jsonencode({
    maxLines = 1000
    derivedFields = [{
      datasourceUid  = grafana_data_source.tempo.uid
      matcherRegex   = "trace_id=(\\w+)"
      name           = "TraceID"

  })
}

3.3 Alert Rules

resource "grafana_rule_group" "api_alerts" {
  name             = "api-alerts"
  folder_uid       = grafana_folder.production.uid
  interval_seconds = 60

  rule {
    name      = "APIErrorRate"
    for       = "5m"
    condition = "C"

    data {
      ref_id         = "A"
      datasource_uid = grafana_data_source.prometheus.uid

      relative_time_range {
        from = 600
        to   = 0
      }

      model = jsonencode({
        expr    = "rate(http_requests_total{job=\"api\",status=~\"5..\"}[5m]) / rate(http_requests_total{job=\"api\"}[5m])"
        refId   = "A"
      })
    }

    data {
      ref_id         = "B"
      datasource_uid = "__expr__"
      model = jsonencode({
        type       = "reduce"
        expression = "A"
        reducer    = "last"
        refId      = "B"
      })
    }

    data {
      ref_id         = "C"
      datasource_uid = "__expr__"
      model = jsonencode({
        type       = "threshold"
        expression = "B"
        conditions = [{
          evaluator = {
            type   = "gt"
            params = [0.05]
          }
        }]
        refId = "C"
      })
    }

    labels = {
      severity = "critical"
      team     = "platform"
    }

    annotations = {
      summary     = "API error rate exceeded 5%"
      runbook_url = "https://wiki/api-errors"
    }

    no_data_state  = "NoData"
    exec_err_state = "Error"
  }
}

      label = "severity"

4.1 為什麼要模組化

跨環境（prod / staging / dev）共用設定，但允許 override。

4.2 Module 結構

├── modules/
│   └── grafana/
│       ├── main.tf
│       ├── variables.tf
│       ├── outputs.tf
│       └── versions.tf
├── environments/
│   ├── production/
│   │   └── main.tf
│   ├── staging/
│   │   └── main.tf
│   └── dev/
│       └── main.tf

4.3 Module 定義

# modules/grafana/variables.tf
variable "environment" {
  type = string
}

variable "prometheus_url" {
  type = string
}

variable "loki_url" {
  type = string
}

variable "teams" {
  type = list(object({
    name  = string
    email = string
  }))
  default = []
}

# modules/grafana/main.tf
resource "grafana_data_source" "prometheus" {
  type = "prometheus"
  name = "Prometheus"
  url  = var.prometheus_url
  uid  = "prometheus-${var.environment}"
}

resource "grafana_team" "teams" {
  for_each = { for t in var.teams : t.name => t }

  name  = each.value.name
  email = each.value.email
}

§ 5. Ansible 整合

5.1 何時用 Ansible

  • 傳統機房 / VM 環境（沒有 K8s）
  • 已有 Ansible 為主的 ops
  • 同時管 OS-level config（systemd、firewall）+ Grafana

5.2 Grafana Ansible Collection

官方 collection：grafana.grafana

# 安裝
ansible-galaxy collection install grafana.grafana

5.3 安裝 Grafana 的 playbook

# install-grafana.yml
- hosts: grafana_servers
  become: true

  tasks:
    - name: Install Grafana
      ansible.builtin.apt:
        name: grafana
        state: present
        update_cache: yes

    - name: Configure grafana.ini
      ansible.builtin.template:
        src: grafana.ini.j2
        dest: /etc/grafana/grafana.ini
      notify: restart grafana

    - name: Provision datasources
      ansible.builtin.copy:
        src: provisioning/datasources/
        dest: /etc/grafana/provisioning/datasources/
      notify: restart grafana

    - name: Enable and start Grafana
      ansible.builtin.systemd:
        name: grafana-server
        enabled: yes
        state: started

  handlers:
    - name: restart grafana
      ansible.builtin.systemd:
        name: grafana-server
        state: restarted

5.4 用 Ansible 模組管 Grafana resources

# manage-resources.yml
- hosts: localhost
  vars:
    grafana_url: https://grafana.example.com
    grafana_api_key: "{{ vault_grafana_token }}"

  tasks:
    - name: Create folder
      grafana.grafana.folder:
        url: "{{ grafana_url }}"
        url_api_key: "{{ grafana_api_key }}"
        title: Production

    - name: Create datasource
      grafana.grafana.datasource:
        url: "{{ grafana_url }}"
        url_api_key: "{{ grafana_api_key }}"
        name: Prometheus
        ds_type: prometheus
        ds_url: http://prometheus:9090

    - name: Create dashboard
      grafana.grafana.dashboard:
        url: "{{ grafana_url }}"
        url_api_key: "{{ grafana_api_key }}"
        path: ./dashboards/api-overview.json
        commit_message: "Updated by Ansible"

5.5 Ansible vs Terraform

面向AnsibleTerraform
State無 state（imperative）有 state（declarative）
Drift detection每次跑就 reconcileplan 顯示 diff
適合OS-level + 應用部署infra 資源
學習曲線低（YAML）中（HCL）

實務常組合用：Terraform 管 infrastructure，Ansible 管 OS-level config。

§ 6. Grafana Operator（K8s）

6.1 Operator 是什麼

K8s Operator 把「Grafana 部署 + 管理」變成 native K8s pattern。安裝 Operator 後， 您寫 Grafana、GrafanaDashboard、GrafanaDatasource 等 CRD（Custom Resource）。

6.2 安裝 Grafana Operator

# 用 Helm
helm repo add grafana-operator https://grafana.github.io/grafana-operator
helm install grafana-operator grafana-operator/grafana-operator \
  --namespace grafana-operator-system \
  --create-namespace

6.3 部署 Grafana instance

apiVersion: grafana.integreatly.org/v1beta1
kind: Grafana
metadata:
  name: grafana-prod
  namespace: monitoring
  labels:
    dashboards: "grafana-prod"     # 之後 dashboard 用這個 label 連接
spec:
  config:
    log:
      mode: "console"
    auth:
      disable_login_form: "false"
    security:
      admin_user: admin
      admin_password: $__file{/etc/grafana/admin/password}
  deployment:
    spec:
      template:
        spec:
          containers:
            - name: grafana
              image: grafana/grafana:12.4.0
              resources:
                requests:
                  cpu: 100m
                  memory: 256Mi
                limits:
                  memory: 512Mi
  service:
    spec:
      type: ClusterIP
  ingress:
    spec:
      rules:
        - host: grafana.example.com
          http:
            paths:
              - path: /
                pathType: Prefix
                backend:
                  service:
                    name: grafana-service
                    port:
                      number: 3000

6.4 跨 instance 管理

多個 Grafana instance 可以同時被 Operator 管理，每個 dashboard 用 selector 對應到 instance。

§ 8. GitOps with ArgoCD / Flux

8.1 GitOps 模型

Operator + ArgoCD/Flux 是 K8s 環境的標準 GitOps：

  1. 程式碼在 Git repo
  2. ArgoCD/Flux watch repo
  3. K8s 中的 state 自動 reconcile 到 Git 定義
  4. UI 修改會被 ArgoCD 覆蓋回去（除非設定 ignore）

8.2 ArgoCD Application

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: grafana-resources
  namespace: argocd
spec:
  project: default

  source:
    repoURL: https://github.com/myorg/grafana-config
    targetRevision: main
    path: kubernetes/

  destination:
    server: https://kubernetes.default.svc
    namespace: monitoring

  syncPolicy:
    automated:
      prune: true       # 刪除 git 中不存在的資源
      selfHeal: true    # UI 改的會被覆蓋回去
    syncOptions:
      - CreateNamespace=true

9.3 Promotion Pipeline

# GitHub Actions
on:
  pull_request:
    branches: [main]

jobs:
  test:
    # 跑在 dev
    steps:
      - terraform plan -chdir=envs/dev
      - apply

  staging:
    needs: test
    if: github.event.pull_request.merged == true
    steps:
      - terraform plan -chdir=envs/staging
      - apply

  production:
    needs: staging
    environment: production         # 需要 manual approve
    steps:
      - terraform plan -chdir=envs/production
      - apply

§ 10. 三方法選型決策

10.1 完整對比表

面向TerraformAnsibleOperator
State有無K8s etcd
Drift detectionplan每次 run 都 checkReconciler 持續
學習曲線中（HCL）低（YAML）中（K8s 概念）
跨 cloud★★★ 最佳普通不適用
K8s 整合Provider 完整K8s module★★★ 原生
非 K8s 環境★★★★★★不能
GitOpsvia Atlantisvia Tower★★★ 完美

✅ 本章重點摘要

30 秒回顧
  • 編排層 IaC：管 Grafana 自己 + resources + 多 instance + 跟其他 infra 一起
  • 三大工具：Terraform（多 cloud）/ Ansible（VM）/ Grafana Operator（K8s）
  • Terraform Provider grafana/grafana：完整支援 datasource / folder / team / dashboard / alert
  • Cloud Stack Provider 額外管 Grafana Cloud instance lifecycle
  • 多 instance：用 provider "grafana" { alias = "..." }
  • Module per env 是推薦的多環境模式（共用 module + 不同 var）
  • Ansible 適合 OS-level + 應用部署，official collection: grafana.grafana
  • Ansible vs Terraform：Ansible 無 state（imperative），Terraform 有 state（declarative）
  • Grafana Operator K8s 原生 GitOps 選擇，主要 9 個 CRD（Grafana / Dashboard / Datasource / Folder / AlertRuleGroup / ...）
  • Operator + ArgoCD/Flux = K8s 標準 GitOps，UI 改動會被覆蓋
  • Dashboard 來源 3 種：inline JSON / URL / ConfigMap
  • 選型：純 K8s 用 Operator、多 cloud 用 Terraform、VM 用 Ansible
  • 實務常組合用：Terraform infra + Operator app + SDK dashboards + ArgoCD GitOps
  • Part 4 收尾：檔案層 + 程式碼層 + 編排層 = 完整 Observability as Code
