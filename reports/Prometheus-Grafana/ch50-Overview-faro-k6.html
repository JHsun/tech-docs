Faro + k6：前端 RUM 與 Load Testing

Part 5 收尾。前面章節都聚焦 backend（metric / log / trace / profile）。但 user 真正體驗的是前端： 瀏覽器載入慢、JS 錯誤、API 互動延遲。Grafana Faro（Real User Monitoring）抓真實使用者瀏覽器資料， Grafana k6 主動模擬流量做壓力測試。本章涵蓋兩家工具，組合起來給觀測性畫面收尾。

§ 1. 前端 Observability 的缺口

1.1 Backend 監控完整還不夠

典型場景：

  • Backend dashboard 顯示「API p99 = 100ms」很健康
  • 但用戶反映「網站慢」
  • 原因可能：JS bundle 5MB、CDN 慢、第三方 script 阻塞、客戶端 react render 慢
  • Backend 完全看不到

1.2 Real User Monitoring (RUM)

從真實使用者瀏覽器收集 telemetry：

  • 頁面載入時間
  • Core Web Vitals（LCP / CLS / INP）
  • JS 錯誤
  • 使用者點擊路徑
  • API 互動延遲

1.3 RUM 工具光譜

工具定位
Datadog RUM商業 SaaS、APM 整合
Sentryerror tracking 為主
New Relic Browser商業 APM
Google Analyticsmarketing 為主
Grafana FaroOSS、與 LGTM 整合

1.4 Synthetic Monitoring vs RUM

面向Synthetic（k6）RUM（Faro）
誰機器模擬真實使用者
頻率固定（每分鐘 1 次）每次造訪
環境固定（同 IP / browser）真實（各國 / 各裝置）
用途監控可用性、SLI 量測找真實 user 體驗問題
適合Critical path uptime性能 / 體驗最佳化

兩者互補：Synthetic 確保「能用」，RUM 知道「用得好不好」。

§ 3. Faro 收集的訊號

3.1 五大訊號類型

類型內容
Errors JS exception、unhandled promise rejection、console.error
Logs console.log / .info / .warn（自動 capture）
Measurements 自訂的數值（page render time、business KPI）
Web Vitals LCP、CLS、INP、FCP、TTFB 自動收集
Traces 頁面內 fetch / XHR 的 OTel span，含 trace context propagation

3.2 Web Vitals 詳解

Metric意義Good / Needs / Poor
LCP（Largest Contentful Paint） 最大內容繪製時間 ≤ 2.5s / ≤ 4s / > 4s
INP（Interaction to Next Paint） 互動延遲（取代了 FID） ≤ 200ms / ≤ 500ms / > 500ms
CLS（Cumulative Layout Shift） 視覺穩定性 ≤ 0.1 / ≤ 0.25 / > 0.25
FCP 首次內容繪製 ≤ 1.8s / ≤ 3s / > 3s
TTFB 首位元組時間 ≤ 800ms / ≤ 1.8s / > 1.8s

LCP / INP / CLS 是 Google 的 Core Web Vitals，影響 SEO 排名。

3.3 Trace Context Propagation

Faro 自動把瀏覽器內 fetch / XHR 加上 W3C traceparent header，讓 frontend trace 與 backend trace 連起來： 

// 自動加 header
fetch('/api/orders')

// 變成
fetch('/api/orders', {
  headers: {
    'traceparent': '00-{trace-id}-{span-id}-01'
  }
})

// Backend service 接收後，繼續同個 trace

結果：在 Tempo 看到「使用者點擊按鈕 → fetch /api/orders → backend handler → DB query」的完整 trace。

§ 4. Faro Web SDK 整合

4.1 安裝

npm install @grafana/faro-web-sdk @grafana/faro-web-tracing

4.2 基本初始化

// app.tsx 或 main.js
import { initializeFaro } from '@grafana/faro-web-sdk';
import { TracingInstrumentation } from '@grafana/faro-web-tracing';

initializeFaro({
  url: 'https://faro-collector.example.com/collect',
  apiKey: 'xxxxxxxxxx',     // 可選
  app: {
    name: 'my-frontend',
    version: '1.2.3',
    environment: 'production',
  },

  // 啟用 OpenTelemetry tracing
  instrumentations: [
    new TracingInstrumentation({
      // 哪些 URL pattern 要 propagate trace context
      propagateTraceHeaderCorsUrls: [
        /^https:\/\/api\.example\.com/,
      ],
    }),
  ],
});

4.6 隱私與 PII

⚠️ PII 處理

Faro 預設會收集 URL、user agent、stack trace。注意 PII：

  • URL query string 可能含 token
  • Form data 可能含密碼
  • Console log 可能洩漏 user data

解法：用 beforeSend hook 過濾敏感資料。

initializeFaro({
  // ...
  beforeSend: (event) => {
    // 移除 query string
    if (event.meta?.page?.url) {
      event.meta.page.url = event.meta.page.url.split('?')[0];
    }
    // 移除特定 attribute
    delete event.attributes?.email;
    return event;
  },
});

§ 5. Faro 後端架構

5.1 Faro Collector

瀏覽器 push 到 Faro Collector（HTTP endpoint），collector 路由到 Loki / Tempo / Mimir。

5.2 用 Alloy 當 Collector

// alloy config.river
faro.receiver "default" {
  server {
    listen_address = "0.0.0.0"
    listen_port    = 12347
  }

  // 路由各訊號
  output {
    logs   = [loki.write.default.receiver]
    traces = [otelcol.exporter.otlp.tempo.input]
  }
}

loki.write "default" {
  endpoint {
    url = "http://loki:3100/loki/api/v1/push"
  }
}

otelcol.exporter.otlp "tempo" {
  client {
    endpoint = "tempo:4317"
    tls {
      insecure = true
    }
  }
}

5.3 Standalone Faro Collector

也可用獨立的 Faro Collector binary（github.com/grafana/faro）：

docker run -p 12347:12347 \
  grafana/faro-collector:latest

5.4 CORS 設定

瀏覽器 push 跨 domain 必須設 CORS：

faro.receiver "default" {
  server {
    cors_allowed_origins = [
      "https://www.example.com",
      "https://app.example.com",
    ]
  }
}

§ 6. k6 概念與定位

6.1 k6 簡史

2017  k6 開源（Load Impact 公司）
2021  Grafana Labs 收購 k6
2022+ k6 整合進 Grafana Cloud
2024+ k6 Browser、k6 Studio 等新功能

6.2 k6 是什麼

Load Testing Tool：寫 JavaScript / TypeScript script，執行模擬大量 virtual users（VUs） 對 API / website 發送請求，量測延遲、錯誤率、吞吐量。

6.3 與其他工具對比

工具定位
k6JavaScript script、CLI-first、CI 友善
JMeterGUI-first、複雜功能多但難維護
GatlingScala script、強大但學習曲線高
Apache Bench (ab)單純 ad-hoc test
LocustPython script、社群活躍

6.4 k6 的優勢

  • JavaScript 開發者熟悉的語言
  • CLI / config 為主，易於 CI/CD 整合
  • 用 Go 寫的 runtime，單機能跑 10000+ VU
  • 內建 metrics 結構化輸出
  • 與 Grafana Cloud / 自架 Prometheus 直接整合

6.5 k6 的訊號

k6 不只是發請求，也會輸出 metric：

Metric意義
http_reqs總請求數
http_req_duration請求耗時（含分位數）
http_req_failed失敗請求
http_req_blocked連線等待時間
http_req_connecting建立 TCP 連線
http_req_tls_handshakingTLS 握手
http_req_sending送 request body
http_req_receiving收 response body
http_req_waiting等 server response (TTFB)
vus當前 VU 數
iterations完成的 iteration 數

§ 7. k6 Script 入門

7.1 第一個 script

// load-test.js
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  vus: 10,           // 10 個 virtual users
  duration: '30s',   // 跑 30 秒
};

export default function () {
  const res = http.get('https://api.example.com/users');

  check(res, {
    'status is 200': (r) => r.status === 200,
    'duration < 200ms': (r) => r.timings.duration < 200,
  });

  sleep(1);   // 1 秒 think time
}

7.3 進階 script

import http from 'k6/http';
import { check, sleep, group } from 'k6';

export const options = {
  scenarios: {
    main_flow: {
      executor: 'ramping-vus',
      stages: [
        { duration: '30s', target: 50 },   // 30s 升到 50 VU
        { duration: '1m',  target: 100 },  // 1m 升到 100
        { duration: '30s', target: 0 },    // 30s 降回 0
      ],
    },
  },
  thresholds: {
    http_req_duration: ['p(95)<500'],     // p95 必須 < 500ms
    http_req_failed:   ['rate<0.01'],     // 錯誤率 < 1%
  },
};

export default function () {
  group('Login flow', () => {
    const loginRes = http.post('https://api.example.com/login', {
      email: 'test@example.com',
      password: 'pass',
    });
    check(loginRes, { 'login OK': (r) => r.status === 200 });

    const token = loginRes.json('token');

    group('Browse products', () => {
      const res = http.get('https://api.example.com/products', {
        headers: { Authorization: `Bearer ${token}` },
      });
      check(res, { 'products OK': (r) => r.status === 200 });
    });
  });

  sleep(1);
}

§ 8. k6 進階：Stages、Thresholds、Modules

8.1 Test Type 對照

Type目的k6 設定
Smoke test 確認服務 OK 1 VU, 1m
Load test 正常負載 50-100 VUs, 10-30m
Stress test 找崩潰點 Ramping VUs 從 100 到 1000+
Spike test 突發流量 從 0 → 500 → 0 in 1m
Soak test 長時間穩定性 200 VUs, 8h+
Breakpoint test 找確切容量上限 慢慢 ramping 到失敗

8.2 Executors 詳解

k6 提供 7 種 executor：

  • shared-iterations：固定 iteration 數，分給所有 VU
  • per-vu-iterations：每 VU 跑固定 iteration 數
  • constant-vus：固定 VU 數，跑指定時間
  • ramping-vus：階段式調整 VU 數
  • constant-arrival-rate：固定 RPS（與 VU 解耦）
  • ramping-arrival-rate：階段式調整 RPS
  • externally-controlled：外部 API 控制

8.3 Thresholds 與 Pass/Fail

export const options = {
  thresholds: {
    // 基本 threshold
    http_req_duration: ['p(95)<500', 'p(99)<1000'],
    http_req_failed:   ['rate<0.01'],

    // 用 abortOnFail 提早結束
    'http_req_duration{name:critical}': [
      { threshold: 'p(95)<200', abortOnFail: true },
    ],

    // 自訂 metric
    'errors{type:db}': ['count<10'],
  },
};

Threshold 失敗時，k6 結束 exit code 非 0，CI/CD 抓得到。

8.4 自訂 Metric

import { Counter, Trend, Rate } from 'k6/metrics';

const failureRate = new Rate('login_failures');
const checkoutDuration = new Trend('checkout_duration_ms');
const ordersCreated = new Counter('orders_created');

export default function () {
  const start = Date.now();
  const res = http.post('/checkout', payload);

  failureRate.add(res.status !== 200);
  checkoutDuration.add(Date.now() - start);

  if (res.status === 201) {
    ordersCreated.add(1);
  }
}

8.5 k6 Browser（v0.43+）

k6 也支援真實 Chromium 跑前端 test（取代 Puppeteer / Playwright 用於 load test）：

import { browser } from 'k6/browser';
import { check } from 'k6';

export const options = {
  scenarios: {
    ui: {
      executor: 'shared-iterations',
      options: {
        browser: { type: 'chromium' },
      },
    },
  },
};

export default async function () {
  const page = await browser.newPage();
  await page.goto('https://example.com');

  await page.waitForSelector('h1');
  const title = await page.title();
  check(page, { 'title is correct': () => title === 'Example' });

  await page.click('#login');
  await page.locator('#email').fill('test@example.com');
  await page.locator('#password').fill('pass');
  await page.click('#submit');

  await page.close();
}

§ 9. k6 Cloud 與分散式

9.1 為什麼需要分散式

單機 k6 能跑 10K+ VU，但有時不夠：

  • 需要 100K+ VU（電商雙 11 場景）
  • 需要從多 region 發起（模擬全球 user）
  • 本機網路頻寬限制

9.2 三種分散式選項

選項適合
Grafana Cloud k6 商業 SaaS、最方便
k6 Operator (K8s) 自架，K8s 環境
k6 Distribution OSS 多機自架（手動編排）

9.5 與 Tempo 整合（trace）

k6 v0.50+ 內建 OpenTelemetry tracing，request 自動帶 trace context：

import { Trace } from 'k6/x/tracing';

export const options = {
  // ...
};

export default function () {
  const trace = new Trace();
  trace.startSpan('checkout-flow');

  const res = http.get('https://api.example.com/checkout', {
    headers: { traceparent: trace.getTraceContext() },
  });

  trace.endSpan();
}

結果：load test 中產生的 trace 在 Tempo 看得到，跟 backend 的 trace 串連。

10.2 完整 user journey 監控

  1. 使用者點擊「Checkout」（Faro 抓 user click event）
  2. Frontend 呼叫 /api/checkout（Faro 啟動 trace）
  3. Backend 接收 trace，繼續 span（Tempo）
  4. Backend 呼叫 payment-service / DB（trace 延伸）
  5. Response 回 frontend（trace 完成）
  6. Frontend render（Faro 抓 LCP / INP）
  7. 使用者看到結果（Faro 抓 page navigation）

整段在同一個 trace，從 user click 到 DB query 完整可見。

10.4 跨工具串聯範例

典型 incident 場景：

  1. k6 synthetic 跑 checkout flow → 失敗率 5%
  2. Grafana 跳到 Tempo 看具體失敗 trace
  3. Trace 顯示 payment-service 慢
  4. 跳 Pyroscope 看 payment-service profile
  5. 找出 jsonEncode 占 50% CPU（Ch 48 案例）
  6. 同時看 Faro：真實 user 體驗也降，confirms impact
  7. 修復 → 部署 → k6 自動 re-test → 確認恢復

✅ 本章重點摘要

30 秒回顧
  • Backend 監控完整不夠，需要 RUM 看真實 user 體驗
  • Synthetic（k6） 確保「能用」，RUM（Faro） 知道「用得好不好」
  • Grafana Faro：OSS RUM，與 LGTM 整合，OTel-aligned
  • Faro 收 5 種訊號：Errors / Logs / Measurements / Web Vitals / Traces
  • Web Vitals：LCP / INP / CLS 是 Google Core，影響 SEO
  • Faro 自動 trace context propagation：browser fetch 與 backend trace 串起來
  • Faro Web SDK + 後端 Faro Collector（可用 Alloy faro.receiver）
  • 注意 PII：URL、form、console log 可能洩漏，用 beforeSend 過濾
  • k6 = JavaScript script load testing，Grafana 收購（2021）
  • 單機跑 10K+ VU，CLI-first，CI 友善
  • Test types：Smoke / Load / Stress / Spike / Soak / Breakpoint
  • 7 種 executors，constant-arrival-rate 與 VU 解耦
  • Thresholds 失敗讓 CI 抓到
  • k6 Browser 用真實 Chromium 跑 UI test
  • 分散式：Grafana Cloud k6 / k6 Operator (K8s) / 自架
  • Faro + k6 + LGTM = 完整觀測性，從 user click 到 DB query 完整 trace
