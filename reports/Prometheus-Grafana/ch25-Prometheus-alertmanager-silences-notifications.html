§ 1. Silence vs Inhibition vs Mute Interval

三者都「讓 alert 不通知」，但機制完全不同：

機制觸發方式持續時間典型用途
Silence 手動（UI / API / amtool） 建立時指定 startsAt + endsAt 維護視窗、已知問題、debug 中
Inhibition 自動（其他 alert 觸發） source alert firing 期間 根因 → 症狀的自動降噪
Mute Interval 時間到自動觸發 固定時間區間 非業務時間、週末、假日

1.1 處理順序

從 Ch 22 我們知道處理順序是：去重 → 靜默 → 抑制 → 路由。 Mute interval 在路由階段檢查（route 上的 mute_time_intervals）。

§ 2. Silence 的建立與管理

2.1 透過 Web UI

到 Alertmanager 的 Silences 頁面 → New Silence：

  • Start / End：時間範圍（UTC）
  • Matchers：哪些 alert 要靜默（label 鍵值對）
  • Creator / Comment：建立者 + 原因（不能空）

2.2 透過 amtool（推薦）

# 簡單建立
amtool silence add alertname=NodeDown \
  --duration=2h \
  --comment="Investigating disk issue (ticket-1234)"

# 更精確：多個 matchers
amtool silence add alertname=DiskFull instance=db-1 \
  --duration=4h \
  --comment="DBA on it"

# 用 regex
amtool silence add 'alertname=~Pod.*' namespace=staging \
  --duration=24h \
  --comment="Staging maintenance"

# 設定建立者（覆蓋 amtool config）
amtool silence add alertname=Test --duration=1h \
  --author="alice" \
  --comment="Testing"

2.3 列出與過期

# 列出所有 active silence
amtool silence query

# 過濾
amtool silence query alertname=NodeDown
amtool silence query --within=1h     # 還剩不到 1 小時的
amtool silence query --expired       # 已過期的

# 過期某個（給 ID）
amtool silence expire 

# 批次過期：把所有 staging 的 silence 提前結束
amtool silence expire $(amtool silence query -q namespace=staging)

# 過期所有 silence（謹慎使用！）
amtool silence expire $(amtool silence query -q)

2.4 透過 API

# 建立
curl -XPOST http://alertmanager:9093/api/v2/silences \
  -H 'Content-Type: application/json' \
  -d '{
    "matchers": [
      {"name": "alertname", "value": "NodeDown", "isRegex": false, "isEqual": true},
      {"name": "instance",  "value": "api-.*",   "isRegex": true,  "isEqual": true}
    ],
    "startsAt": "2026-05-02T10:00:00Z",
    "endsAt":   "2026-05-02T12:00:00Z",
    "createdBy": "alice@example.com",
    "comment": "Investigating timeout (incident-456)"
  }'

# 查 silence
curl http://alertmanager:9093/api/v2/silences

# 刪除（過期）
curl -XDELETE http://alertmanager:9093/api/v2/silence/

§ 3. Silence 最佳實踐

✅ Silence 操作守則
  1. 所有 silence 都要有 comment：寫 ticket 號 / 原因 / 預期解除時間
  2. 盡量設短的 duration：寧可建 4h 後續續，不要一次建 7d
  3. matchers 越精確越好：不要 service=.*，靜默太多
  4. 定期檢查長期 silence：避免遺忘的 silence 隱藏新問題
  5. 用 amtool 而非 UI：可寫進 runbook、版本控管、自動化
  6. 區分 Silence 與 Inhibition：可重複觸發的 → 寫 inhibition；一次性的 → silence

3.1 監控 silence 狀況

有幾個 metric 幫您追蹤 silence 健康：

# 當前 active silence 數量
alertmanager_silences{state="active"}

# 過去 5 分鐘建立 silence 的數量（突然暴增表示警急狀態）
rate(alertmanager_silences_query_duration_seconds_count[5m])

# 監控「永久 silence」(超過 30 天還 active)
# 需自寫 dashboard query alertmanager_silences API

3.2 防止「忘記的 silence」

業界常用做法：

  • 限制最大 duration（amtool 沒原生支援，需 wrapper script）
  • 用 incident management 工具（PagerDuty、Grafana OnCall）建立 silence，工具自動到期
  • 每週 review：列出所有 active silence，confirm 還需要保留的

§ 4. Receiver 共通結構

所有 receiver 設定的共同模式：

receivers:
  - name: 'my-receiver'                 # 在 routing 中引用的名稱

    # 任意數量的 N 個 channel（一個 receiver 可同時送多個地方）
    email_configs:
      - to: 'team@example.com'
        # ...

    slack_configs:
      - channel: '#alerts'
        # ...

    webhook_configs:
      - url: 'https://my-system.example.com/webhook'
        # ...

4.1 Receiver 內可同時設多個

例：critical alert 同時送 Email + Slack + 自家 webhook：

receivers:
  - name: 'critical-multi'
    email_configs:
      - to: 'oncall@example.com'

    slack_configs:
      - channel: '#alerts-critical'

    webhook_configs:
      - url: 'https://incident.io/api/v1/alerts'

4.2 全部支援的 receiver type

§ 5. Email 整合

5.1 Global SMTP 設定

global:
  # SMTP 伺服器
  smtp_smarthost: 'smtp.gmail.com:587'
  smtp_from: 'alertmanager@example.com'

  # 認證
  smtp_auth_username: 'alertmanager@example.com'
  smtp_auth_password_file: '/etc/alertmanager/secrets/smtp-password'
  # 或：smtp_auth_password: 'plaintext-not-recommended'

  # TLS
  smtp_require_tls: true
  smtp_hello: 'mailer.example.com'

5.2 Receiver 設定

receivers:
  - name: 'team-email'
    email_configs:
      - to: 'team@example.com, oncall@example.com'  # 多收件者用逗號

        # 是否在 alert resolved 時也通知
        send_resolved: true

        # 自訂主旨（用 Go template）
        headers:
          Subject: '{{ template "email.subject" . }}'
          X-Priority: '{{ if eq .CommonLabels.severity "critical" }}1{{ else }}3{{ end }}'

        # HTML 內容（template）
        html: '{{ template "email.html" . }}'

        # 純文字 fallback
        text: '{{ template "email.text" . }}'

        # 覆蓋 global SMTP（可選）
        from: 'critical-alerts@example.com'
        smarthost: 'smtp-internal.example.com:25'
        require_tls: false

5.3 常見問題

⚠️ Email 整合常見雷點
  • Gmail SMTP：要用「App password」，不是帳號密碼。需先在 Google 帳號開啟 2FA + 產生 app password。
  • SES / SendGrid：sender 必須先在 SES/SendGrid 驗證
  • 內部 SMTP：可能需 require_tls: false 或 tls_config.insecure_skip_verify: true
  • 大量收件者：別在 to 寫 50 個 email — 用 mailing list

§ 6. Slack 整合

6.1 取得 Webhook URL

到 https://api.slack.com/apps → Create App → Incoming Webhooks → Add Webhook to Workspace。

得到形如：https://hooks.slack.com/services/T0XXX/B0YYY/abc123

6.2 Receiver 設定

# 全域 URL（推薦）
global:
  slack_api_url_file: '/etc/alertmanager/secrets/slack-url'

receivers:
  - name: 'slack-alerts'
    slack_configs:
      - channel: '#alerts'
        send_resolved: true

        # Standard fields
        title: '{{ template "slack.title" . }}'
        text: '{{ template "slack.text" . }}'

        # Slack-specific
        username: 'Alertmanager'
        icon_emoji: ':rotating_light:'
        # 或 icon_url: 'https://example.com/logo.png'

        # 動態顏色（v0.20+）
        color: '{{ if eq .Status "firing" }}danger{{ else }}good{{ end }}'

        # 短連結
        link_names: true

6.3 多 channel 多 webhook

不同 channel 用同一個 webhook URL（webhook 屬於整個 workspace）：

receivers:
  - name: 'slack-multi-channel'
    slack_configs:
      - channel: '#alerts-platform'
        title: 'Platform: {{ .CommonLabels.alertname }}'

      - channel: '#alerts-incidents'
        title: '🔥 INCIDENT: {{ .CommonLabels.alertname }}'

6.4 進階：Block Kit Message

用 Slack Block Kit 做更豐富的格式：

slack_configs:
  - channel: '#alerts'
    title: '{{ template "slack.title" . }}'
    text: '{{ template "slack.text" . }}'

    # 用 attachments 讓訊息分塊（早期格式）
    actions:
      - type: button
        text: 'View in Grafana'
        url: 'https://grafana.example.com/d/abc'
      - type: button
        text: 'Runbook'
        url: '{{ (index .Alerts 0).Annotations.runbook_url }}'

§ 7. PagerDuty 整合

7.1 取得 Routing Key（Integration Key）

PagerDuty UI → Services → 您的 service → Integrations → Add → Events API V2 → 取得 Integration Key。

7.2 Receiver 設定

receivers:
  - name: 'pagerduty-critical'
    pagerduty_configs:
      - routing_key: ''
        # 或 routing_key_file: '/etc/alertmanager/secrets/pd-key'

        # 嚴重程度（PD 自己的等級）
        severity: '{{ .CommonLabels.severity }}'   # critical / error / warning / info

        # 客製化欄位
        details:
          firing: '{{ .Alerts.Firing | len }}'
          resolved: '{{ .Alerts.Resolved | len }}'
          alertname: '{{ .CommonLabels.alertname }}'

        # 連結
        client: 'Alertmanager'
        client_url: 'https://alertmanager.example.com'

        # 連結到 runbook / dashboard（可從 annotation 拉）
        links:
          - href: '{{ (index .Alerts 0).Annotations.runbook_url }}'
            text: 'Runbook'
          - href: '{{ (index .Alerts 0).GeneratorURL }}'
            text: 'View in Prometheus'

        # 圖片連結（PD UI 顯示）
        images:
          - src: 'https://grafana.example.com/render/d/abc/...'
            alt: 'Service Overview'

        send_resolved: true

7.3 PagerDuty Severity 對應

PagerDuty 的 severity 與 Prometheus alert label 不一定一致。建議在 receiver 中映射：

severity: |
  {{- if eq .CommonLabels.severity "critical" -}}
  critical
  {{- else if eq .CommonLabels.severity "warning" -}}
  warning
  {{- else -}}
  info
  {{- end -}}

7.4 PagerDuty 用法注意事項

⚠️ PagerDuty 注意事項
  • PD 自己會做 grouping (Auto-Resolve、De-duplication)，跟 Alertmanager grouping 並存。建議讓 Alertmanager 做大部分聚合。
  • PD 的「Auto-Resolve」會在沒收到 keep-alive 時自動解除（預設 24h），請設好 repeat_interval 比 PD 設定短。
  • 每個 routing_key 對應一個 PD Service，不同 team / severity 用不同 service 才能對應 escalation policy。

§ 8. Microsoft Teams

v0.26+ 引入原生 msteams_configs（experimental，使用 v1 webhook 格式）。但 Microsoft 已棄用 Office 365 Connector，新用 Workflow，建議用通用 webhook + bridge 工具。

8.1 用 prom2teams（社群方案，最穩）

# 部署 prom2teams（容器化即可）
docker run -d \
  -p 8089:8089 \
  -e APP_CONFIG_FILE=/config.ini \
  -v $(pwd)/config.ini:/config.ini \
  idealista/prom2teams
# Alertmanager 設定
receivers:
  - name: 'teams-via-bridge'
    webhook_configs:
      - url: 'http://prom2teams:8089/v2/dev'
        send_resolved: true

8.2 直接用原生 msteams_configs

# Power Automate / Workflow 取得 webhook URL
receivers:
  - name: 'teams-native'
    msteams_configs:
      - webhook_url: 'https://prod-xxx.westus.logic.azure.com/...'
        title: '{{ template "msteams.title" . }}'
        text: '{{ template "msteams.text" . }}'
        send_resolved: true

§ 9. Discord 與 OpsGenie

9.1 Discord（v0.25+）

# 在 Discord channel 設定 Webhook (Server Settings → Integrations → Webhooks)
receivers:
  - name: 'discord-alerts'
    discord_configs:
      - webhook_url: 'https://discord.com/api/webhooks/XXX/YYY'
        title: '{{ template "discord.title" . }}'
        message: '{{ template "discord.message" . }}'
        send_resolved: true

9.2 OpsGenie

# 取得 API key (OpsGenie → Teams → Integrations → API)
global:
  opsgenie_api_url: 'https://api.opsgenie.com/'
  opsgenie_api_key_file: '/etc/alertmanager/secrets/opsgenie-key'

receivers:
  - name: 'opsgenie-critical'
    opsgenie_configs:
      - message: '{{ .CommonLabels.alertname }}'
        description: '{{ template "opsgenie.description" . }}'

        # 嚴重程度（P1-P5）
        priority: |
          {{- if eq .CommonLabels.severity "critical" }}P1
          {{- else if eq .CommonLabels.severity "warning" }}P3
          {{- else }}P5
          {{- end }}

        tags: '{{ .CommonLabels.team }},{{ .CommonLabels.environment }}'

        # 路由到特定 team
        responders:
          - name: 'platform-team'
            type: 'team'

        send_resolved: true

§ 10. Webhook（萬用通道）

當您要整合的系統沒有原生 receiver，用 webhook 連自家系統 / 第三方。 這也是建立自訂工作流（如自動建 ticket）的方式。

10.1 Webhook receiver 設定

receivers:
  - name: 'incident-system'
    webhook_configs:
      - url: 'https://incidents.example.com/api/v1/alerts'
        send_resolved: true

        # HTTP method (預設 POST)
        # 注意：v0.x 仍只支援 POST

        # HTTP 認證
        http_config:
          basic_auth:
            username: 'prometheus'
            password_file: '/etc/alertmanager/secrets/incident-pass'
          # 或 bearer_token / oauth2 / TLS
          tls_config:
            ca_file: '/etc/alertmanager/ca.pem'

        # Max alerts per webhook call (v0.21+)
        max_alerts: 100

10.2 Webhook 收到的 payload 格式

{
  "version": "4",
  "groupKey": "{}/{alertname=\"NodeDown\"}:...",
  "truncatedAlerts": 0,
  "status": "firing",
  "receiver": "incident-system",
  "groupLabels": {"alertname": "NodeDown"},
  "commonLabels": {"alertname": "NodeDown", "severity": "critical"},
  "commonAnnotations": {"summary": "Node down"},
  "externalURL": "http://alertmanager.example.com",
  "alerts": [
    {
      "status": "firing",
      "labels": {"alertname": "NodeDown", "instance": "api-1", "severity": "critical"},
      "annotations": {"summary": "...", "runbook_url": "..."},
      "startsAt": "2026-05-02T10:00:00Z",
      "endsAt": "0001-01-01T00:00:00Z",
      "generatorURL": "http://prometheus:9090/graph?...",
      "fingerprint": "abc123"
    }
  ]
}

10.3 Webhook 自家服務範例（Python Flask）

from flask import Flask, request, jsonify
import requests

app = Flask(__name__)

@app.route('/alertmanager-webhook', methods=['POST'])
def receive_alert():
    payload = request.json

    for alert in payload['alerts']:
        if alert['status'] == 'firing':
            severity = alert['labels'].get('severity')
            alertname = alert['labels'].get('alertname')

            # 範例：建 Jira ticket
            if severity == 'critical':
                create_jira_ticket(alert)

            # 範例：自動執行修復腳本
            if alertname == 'DiskSpaceLow':
                trigger_cleanup_job(alert['labels']['instance'])

    return jsonify({"status": "ok"}), 200

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)

§ 11. send_resolved 與通知去重

11.1 send_resolved 控制

每個 receiver 可設 send_resolved: true/false，決定是否在 alert resolved 時也送通知。

receiver建議值原因
Slack（高頻）true看到 alert 結束很重要
Emailtrue歸檔有用
PagerDutytruePD 才會 Auto-Resolve incident
SMS（短信）false避免擾人
Webhook 自家系統看用途需要 resolved 邏輯就 true

11.2 通知去重原理

Alertmanager 用 nflog（notification log）追蹤「哪些 alert 已通知過誰」。 HA cluster 中的 nflog 用 gossip 同步，避免每個 AM 都送一次。 

# 觀察 notification 健康
alertmanager_notifications_total
alertmanager_notifications_failed_total

# 各 receiver 的延遲
alertmanager_notification_latency_seconds_bucket

11.3 必設的告警

- alert: AlertmanagerNotificationFailed
  expr: rate(alertmanager_notifications_failed_total[5m]) > 0
  for: 5m
  labels:
    severity: critical
  annotations:
    summary: "Alertmanager failed to send to {{ $labels.integration }}"

- alert: AlertmanagerHighNotificationLatency
  expr: |
    histogram_quantile(0.99,
      rate(alertmanager_notification_latency_seconds_bucket[5m])) > 30
  for: 10m
  annotations:
    summary: "Alertmanager P99 notification latency > 30s"

✅ 本章重點摘要

30 秒回顧
  • 三個降噪機制：Silence（手動）、Inhibition（自動規則）、Mute Interval（時段）
  • Silence 操作優先用 amtool 而非 UI（可寫進 runbook）
  • 所有 silence 都要有 comment（ticket / 原因 / 預期解除時間）
  • 一個 receiver 可同時設多個 channel（email + slack + webhook ...）
  • Email 用 Gmail 要 App Password 不是帳號密碼
  • Slack 整合用 Incoming Webhook URL，注意 webhook 屬於 workspace
  • PagerDuty 用 routing_key（=Integration Key），不同 team / severity 用不同 service
  • MS Teams Office 365 Connector 已棄用，建議用 prom2teams bridge 或 Workflow webhook
  • Webhook 是萬用通道，連自家系統做客製工作流
  • send_resolved: true 對 Slack/PD/Email 推薦；對 SMS 等高干擾通道避免
  • 監控 AM 自身：alertmanager_notifications_failed_total 必設告警
