Alertmanager 進階：HA、規模化、Mimir Ruler、多租戶

§ 1. HA 部署深度

Ch 23 介紹了 HA 的基本設定。本章深入 cluster 內部的細節：訊息類型、推送間隔、failure modes。

1.1 Cluster 同步什麼？

HA cluster 中的 Alertmanager 透過 gossip 同步三類資料：

資料類型用途同步機制
Notification Log（nflog） 「誰已經發過這個通知」 持續 gossip，避免重複通知
Silences 當前 active 的所有 silence 持續 gossip + 定期 push-pull 修復
Cluster Membership 哪些 peer 還活著 SWIM 協議（Memberlist）

注意：alert 本身不同步——每個 AM 都從 Prometheus 各自接收，所以 Prometheus 必須對所有 AM 都送。

1.2 通知去重的具體流程

sequenceDiagram participant P as Prometheus participant A1 as AM-1 participant A2 as AM-2 participant A3 as AM-3 participant S as Slack P->>A1: POST alerts P->>A2: POST alerts P->>A3: POST alerts Note over A1,A3: 三個 AM 都收到一樣的 alert A1->>A1: 計算 hash · 自己排在第一 A2->>A2: 計算 hash · 自己排第二
等 1× pushpull-interval A3->>A3: 計算 hash · 自己排第三
等 2× pushpull-interval A1->>S: POST 通知 S-->>A1: 200 OK A1->>A2: gossip · 已通知 A1->>A3: gossip · 已通知 Note over A2,A3: 收到 nflog 更新 · 跳過通知

機制：每個 AM 根據自己在 cluster 中的排名（hash 計算），決定等多久才嘗試發通知。 第一個（排名 0）立刻發送；第二個等 --cluster.pushpull-interval（預設 1m）；以此類推。
在等待期間若收到 nflog 同步，知道別的 AM 已發送，就不發。

1.3 關鍵 cluster flag 詳解

Flag預設調整時機
--cluster.gossip-interval 200ms cluster 大時加大到 500ms-1s 減 CPU
--cluster.pushpull-interval 1m 影響 dedup 等待時間，通常不改
--cluster.tcp-timeout 10s 慢網路加大
--cluster.probe-timeout 500ms FailureDetector 的 ping 超時
--cluster.probe-interval 1s 檢查 peer 健康的間隔
--cluster.reconnect-interval 10s 嘗試重連 dead peer
--cluster.peer-timeout 15s Peer 多久沒回應視為 dead

1.4 Brain Split 處理

⚠️ Network Partition 場景

若 3 台 AM 因網路分裂為 [AM-1] 與 [AM-2, AM-3]：
每個分裂部分都會獨立發通知（無法 dedup）。
這是「at-least-once」設計：寧可重複通知，不要漏報。
網路恢復後，nflog 會同步，後續通知重新 dedup。

2.1 必看 metrics

# 1. Cluster member 數（最重要）

# 預期 = 副本數。< 副本數 = HA 退化

# 2. Gossip 訊息流量
rate(alertmanager_cluster_messages_received_total[5m])
rate(alertmanager_cluster_messages_sent_total[5m])

# 3. 失敗的 gossip
rate(alertmanager_cluster_messages_received_failure_total[5m])
rate(alertmanager_cluster_messages_pushed_total[5m])

# 4. Peer 失敗檢測
alertmanager_cluster_failed_peers
alertmanager_cluster_health_score    # 0=healthy, 越大越糟

# 5. 排名（在 dedup 中的位置）
alertmanager_peer_position

# 6. Notification log 大小
alertmanager_nflog_gc_duration_seconds
alertmanager_nflog_snapshot_size_bytes

2.2 經典問題：member 數量隨機跳動

症狀：alertmanager_cluster_members 在 2 與 3 之間跳。

常見原因：

  • K8s pod 之間網路延遲大（probe-timeout 太緊）
  • NAT / firewall 偶爾 drop UDP 封包（gossip 用 UDP）
  • 節點 CPU 飽和導致 gossip 延遲

解法：加大 --cluster.probe-timeout=2s、--cluster.probe-interval=2s。

§ 3. 大規模部署考量

3.1 何謂「大規模」

規模典型指標建議架構
小（小於 100 alert / 小時）1-5 個服務、單一 team3 副本 HA 已綽綽有餘
中（100-10k / 小時）幾十個服務、3-5 teamsHA + 分團隊 routing tree
大（> 10k / 小時）幾百服務、多 cluster、多 team多套 AM（按 team 或 region）
巨型SaaS / 多 tenantMimir Alertmanager 多租戶

3.2 規模化策略

策略 1：分流（Sharding by Team / Region）

不同團隊用不同的 AM cluster：

# Prometheus alerting config
alerting:
  alertmanagers:
    - api_version: v2
      static_configs:
        - targets: ['platform-am:9093']
      relabel_configs:
        - source_labels: [team]
          regex: 'platform'
          action: keep
    - api_version: v2
      static_configs:
        - targets: ['data-am:9093']
      relabel_configs:
        - source_labels: [team]
          regex: 'data'
          action: keep

每組 AM 處理自己 team 的 alert，互不影響。

策略 2：按 cluster 分

每個 K8s cluster 各自一組 AM，避免 cross-region latency：

# cluster TW
alerting:
  alertmanagers:
    - static_configs:
        - targets: ['am-tw-0:9093', 'am-tw-1:9093', 'am-tw-2:9093']

# cluster JP（不同 prom 設定）
alerting:
  alertmanagers:
    - static_configs:
        - targets: ['am-jp-0:9093', 'am-jp-1:9093', 'am-jp-2:9093']

3.3 Cluster 大小建議

副本數適用注意
1開發 / 測試無 HA
2不建議奇數副本才有 quorum 概念（雖然 AM 沒嚴格需要）
3絕大多數情境業界標準
5+超大規模gossip overhead 變大，管理複雜

3.4 通知吞吐量瓶頸

當您的 alert 數量爆炸時，瓶頸通常是下游 receiver，不是 AM 自己：

  • Slack：每 channel 每秒 1 訊息，超過會限流
  • PagerDuty：API rate limit (每 routing_key 約 60 events/min)
  • Email：SMTP server 容量

解決：用 grouping 把多 alert 合併成一條訊息（前面章節已詳述）。

3.5 持久化考量

Alertmanager 的 --storage.path 儲存 nflog 與 silences。在 K8s 環境用 PVC：

# StatefulSet (kube-prometheus-stack 自動處理)
volumeClaimTemplates:
  - metadata:
      name: alertmanager-data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi      # 通常 1-10Gi 足夠

沒持久化的後果：重啟後 silences 短暫失效（從 cluster gossip 重建，但有時間差）。

§ 4. 多租戶設計

4.1 Vanilla Alertmanager 的多租戶限制

原生 Alertmanager 不支援多租戶：

  • 單一設定檔（每個租戶不能有自己的 routing / receivers）
  • 單一 silence 命名空間（任何人都能看 / 操作所有 silence）
  • 無認證 / 授權（誰都能看 alert）

4.2 三種多租戶實現方案

方案方法複雜度
多 AM Instance 每個 tenant 部署獨立 AM cluster 低，但浪費資源
AlertmanagerConfig CRD K8s 用 Operator 切分 中
Mimir Alertmanager 原生支援 tenant ID 高，但最完整

4.3 AlertmanagerConfig CRD 範例

# 主 Alertmanager（kube-prometheus-stack 安裝）
apiVersion: monitoring.coreos.com/v1
kind: Alertmanager
metadata:
  name: main
  namespace: monitoring
spec:
  replicas: 3
  alertmanagerConfigSelector:
    matchLabels:
      alertmanager: main      # ← 選擇符合此 label 的 AlertmanagerConfig
  alertmanagerConfigNamespaceSelector: {}    # ← 跨所有 namespace

---
# Team A 的 routing 與 receiver
apiVersion: monitoring.coreos.com/v1alpha1
kind: AlertmanagerConfig
metadata:
  name: team-a-config
  namespace: team-a
  labels:
    alertmanager: main
spec:
  route:
    receiver: 'team-a-slack'
    matchers:
      - name: namespace
        value: team-a-.*
        matchType: '=~'
  receivers:
    - name: 'team-a-slack'
      slackConfigs:
        - apiURL:
            name: team-a-slack-secret
            key: url
          channel: '#team-a-alerts'

---
# Team B 在自己的 namespace 建自己的 config
apiVersion: monitoring.coreos.com/v1alpha1
kind: AlertmanagerConfig
metadata:
  name: team-b-config
  namespace: team-b
  labels:
    alertmanager: main
spec:
  # ...

Operator 把所有 AlertmanagerConfig 合併到主 Alertmanager 的 routing tree。

§ 5. Mimir Ruler 整合

5.1 什麼是 Mimir Ruler

Grafana Mimir（Ch 17 介紹過）有一個 ruler 元件， 它取代了 Prometheus 的 alerting/recording rule 評估，並能直接送 alert 到 Alertmanager。

graph LR subgraph Traditional["傳統架構"] T1[Prometheus
scrape + 評估 rule] T2[Alertmanager] T1 -- POST alerts --> T2 end subgraph WithMimir["有 Mimir + Ruler"] M1[Prometheus Agent
純 scrape · remote_write] M2[Mimir
儲存] M3[Mimir Ruler
評估 rule] M4[Mimir Alertmanager
多租戶] M1 --> M2 M2 --> M3 M3 -- POST alerts --> M4 end

5.2 Mimir Ruler 的優勢

  • 跟 ingestion 解耦：scrape Prometheus 可以重啟，rule 評估持續
  • 多租戶：每個 tenant 自己的 rules，互不干擾
  • Horizontal scale：rules 可以分片到多個 ruler instance
  • API 管理 rules：用 cortextool 或 Mimir API CRUD rules，不用改檔案 reload
  • 內建 Alertmanager：Mimir 內建多租戶 Alertmanager（也可外掛標準 AM）

5.3 Mimir Ruler 設定範例

# mimir.yaml (簡化)
ruler:
  enable_api: true
  evaluation_interval: 1m

  # 把 alert 送到 Alertmanager
  alertmanager_url: http://alertmanager:9093/alertmanager

  # 用 Mimir 內建 AM 的話：
  # alertmanager_url: http://mimir:8080/alertmanager

ruler_storage:
  backend: s3
  s3:
    endpoint: s3.amazonaws.com
    bucket_name: mimir-rules
    region: us-east-1

5.4 用 cortextool 管理 rules

# 安裝
go install github.com/grafana/cortex-tools/cmd/cortextool@latest

# 列出所有 rule group（特定 tenant）
cortextool rules list \
  --address=http://mimir:8080 \
  --id=tenant-1

# Push 一組 rules
cortextool rules load \
  --address=http://mimir:8080 \
  --id=tenant-1 \
  rules.yaml

# 比較本地 rules 與 Mimir 上的差異
cortextool rules diff \
  --address=http://mimir:8080 \
  --id=tenant-1 \
  rules.yaml

# 同步
cortextool rules sync \
  --address=http://mimir:8080 \
  --id=tenant-1 \
  rules.yaml

§ 6. 從 Cortex 遷移到 Mimir

Mimir 是 Grafana Labs 從 Cortex fork 出來的版本（2022 年）。 Cortex 開發已停滯，建議遷至 Mimir。

6.1 遷移步驟

  1. 檢查當前 Cortex 版本（v1.10+ 直接相容）
  2. 用 Mimir image 部署一個新的 Mimir instance（讀同個 object storage）
  3. 導引 Prometheus remote_write 到 Mimir（保留 Cortex 一段時間並行）
  4. 遷移 rules：用 cortextool dump → load 到 Mimir
  5. 遷移 Alertmanager 設定（如果用了 Cortex AM）
  6. 切換 Grafana datasource 到 Mimir querier
  7. 驗證一週後關閉 Cortex

6.2 主要差異

面向CortexMimir
開發狀態停滯活躍
Native Histogram有限完整支援
查詢效能—大幅優化（特別是高基數）
Authentication—強化的多租戶
LicenseApache 2.0AGPL v3

§ 7. Grafana Alerting 與雙系統共存

Grafana 8+ 引入 Grafana Alerting（內建在 Grafana 裡的 alerting 系統，獨立於 Prometheus）。 它跟 Prometheus + Alertmanager 是兩套不同系統。

7.1 兩者的差別

面向Prometheus + AlertmanagerGrafana Alerting
評估位置PrometheusGrafana 內建 ruler
資料來源只能 Prometheus任何 datasource（Prom、Loki、SQL...）
UIAlertmanager Web UIGrafana 主介面
多 datasource否是（multi-condition）
Best forProm-only 環境多 datasource、想要單一 UI

7.2 共存模式

實務常見：

  • Prometheus alerting rules → Alertmanager → 通知
  • Grafana 寫一些跨 datasource 的 alert（例：metric + log 條件）→ Grafana Alerting → 同一個 AM

Grafana Alerting 可以設定「外部 Alertmanager」作為 receiver：

# Grafana datasource 設定
- name: Alertmanager
  type: alertmanager
  url: http://alertmanager:9093
  jsonData:
    implementation: prometheus      # 或 cortex / mimir
    handleGrafanaManagedAlerts: true

7.3 Grafana 12 的進化

Grafana 12 為 alerting 帶來重要更新：原生支援 recording rule、從 Prometheus alert 一鍵匯入到 Grafana-managed alert。 對於想統一 alerting UI 的團隊，這是個遷移路徑。

§ 8. Performance Tuning

8.1 監控 Alertmanager 自身負載

# 處理 alert 的速率
rate(alertmanager_alerts_received_total[5m])

# 處理通知的速率與延遲
rate(alertmanager_notifications_total[5m])
histogram_quantile(0.99, rate(alertmanager_notification_latency_seconds_bucket[5m]))

# 失敗率
rate(alertmanager_notifications_failed_total[5m]) /
  rate(alertmanager_notifications_total[5m])

# 記憶體用量
process_resident_memory_bytes{job="alertmanager"}

# Goroutine
go_goroutines{job="alertmanager"}

8.2 調校建議

症狀調整
記憶體用太多降低 --data.retention（預設 120h）
nflog 太大同上 + 縮短 silence 保留
Slack/PD 限流調 grouping 把 alert 合併
Goroutine 暴增檢查 webhook receiver 是否 timeout
CPU 高（gossip）加大 --cluster.gossip-interval

8.3 Resource Sizing

典型 production AM（3 副本 cluster）：

resources:
  requests:
    cpu: 100m
    memory: 256Mi
  limits:
    cpu: 1000m            # AM 大部分時間 idle，限制可寬鬆
    memory: 512Mi          # 預期 200-300MB

# 容納 alert 量級
# 每秒 100 alert + 1000 active alerts → 上述配置充裕
# 每秒 1000+ alert → 加 memory 到 1Gi，加多 sharding

§ 9. Disaster Recovery

9.1 備份什麼？

  • alertmanager.yml：放 Git 版本控管
  • Templates：放 Git
  • Silences：API 定期匯出
  • Secrets（SMTP password 等）：用 Sealed Secrets / Vault

9.2 Silence 備份腳本

#!/bin/bash
# backup-silences.sh
DATE=$(date +%Y%m%d-%H%M%S)
curl -s http://alertmanager:9093/api/v2/silences \
  | jq '.[] | select(.status.state == "active")' \
  > /backup/silences-$DATE.json

# 還原時：
# cat silences-DATE.json | jq -c '.[]' | while read s; do
#   curl -XPOST http://alertmanager:9093/api/v2/silences \
#     -H 'Content-Type: application/json' -d "$s"
# done

9.3 重建 cluster

若整個 cluster 同時掛了：

  1. 用 GitOps 重新部署（kube-prometheus-stack / Operator）
  2. Cluster 重新 gossip 形成（無需手動）
  3. 從備份還原 silences（可選，過期的 silence 會丟掉）
  4. Prometheus 重新送 alert，AM 重新建立 nflog

RTO（Recovery Time Objective）：5-10 分鐘。

§ 10. 進階故障排查

10.1 「通知沒送出」debug 流程

  1. 檢查 alert 是否到達 AM：curl AM:9093/api/v2/alerts
  2. 檢查路由匹配：amtool config routes test ...
  3. 檢查 silence：amtool silence query alertname=...
  4. 檢查 inhibition：看是否有 source alert active
  5. 檢查 receiver 是否 fail：rate(alertmanager_notifications_failed_total[5m])
  6. 看 AM logs：journalctl -u alertmanager -f 找 error

10.2 「同一個 alert 收到多次」

可能原因：

  • HA cluster 沒形成（每個 AM 各自送）
  • Prometheus 沒設 alert_relabel_configs 移除 replica
  • 多個 receiver 都匹配到 + continue: true（這是預期行為）
  • repeat_interval 太短

10.3 「Alert 看似 firing 但 AM 沒收到」

檢查：

  • Prometheus 端 prometheus_notifications_dropped_total
  • Prometheus 與 AM 之間網路
  • AM --web.listen-address 是否正確
  • Prometheus alerting config api_version: v2 設了嗎
