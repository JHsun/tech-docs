Chapter 28 · 監控自身 + Incident Tools

Alertmanager 自監控與 Incident Tools 整合

Alertmanager 是您的「告警基礎設施」——若它自己掛了，您不會收到任何告警。本章涵蓋 self-monitoring 必設的 alert、與業界主流的 incident management 工具整合：PagerDuty、Grafana OnCall、incident.io、FireHydrant。 這是 Part 2 Alertmanager 的收官。

§ 1. 為什麼需要監控告警系統自己

🚨 你的告警系統壞了，誰告訴你？

想像：Alertmanager 進入死循環無法發通知，但您看不到任何 alert。 第二天您接到客戶電話：「為什麼我們的服務掛了 10 小時你們不知道？」
告警系統的可靠性 = 整體可靠性的上限。

1.1 三層監控

完整的告警基礎設施需要三層觀察：

層觀察什麼機制
內部 metrics Prometheus / AM 自己暴露的 metric 另一個 Prometheus 抓 + alert
Watchdog Alert 定期 firing 的測試 alert，驗證整個 pipeline 內部
Dead Man's Switch 外部系統定期 receive AM 通知，沒收到就警告 第三方（PagerDuty Heartbeat 等）

§ 2. Prometheus 端的自監控

Prometheus 自己暴露大量 metric，最關鍵的告警相關：

2.1 Prometheus 必設告警

groups:
  - name: prometheus_self
    rules:
      # 1. Prometheus 自己掛了（最高優先）
      - alert: PrometheusDown
        expr: up{job="prometheus"} == 0
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Prometheus instance {{ $labels.instance }} is down"

      # 2. 重新載入設定失敗
      - alert: PrometheusConfigReloadFailed
        expr: prometheus_config_last_reload_successful == 0
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Prometheus config reload failed"
          description: "Last reload was unsuccessful. Check syntax."

      # 3. 評估 rule 失敗
      - alert: PrometheusRuleEvaluationFailures
        expr: rate(prometheus_rule_evaluation_failures_total[5m]) > 0
        for: 5m
        labels:
          severity: warning

      # 4. Rule 評估超時（rule 太重 / Prometheus 太忙）
      - alert: PrometheusRuleGroupSlow
        expr: |
          prometheus_rule_group_last_duration_seconds
          > prometheus_rule_group_interval_seconds
        for: 15m
        labels:
          severity: warning

      # 5. 通知 AM 失敗（連不上 / queue 滿）
      - alert: PrometheusNotificationsToAlertmanagerFailing
        expr: rate(prometheus_notifications_errors_total[5m]) > 0
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Prometheus failing to send alerts to Alertmanager"

      # 6. 找不到 Alertmanager（discovery 失敗）
      - alert: PrometheusAlertmanagerDiscoveryFailing
        expr: rate(prometheus_notifications_dropped_total[5m]) > 0
        for: 5m
        labels:
          severity: critical

      # 7. TSDB compaction 失敗（會導致磁碟爆）
      - alert: PrometheusTSDBCompactionFailing
        expr: rate(prometheus_tsdb_compactions_failed_total[2h]) > 0
        for: 1h
        labels:
          severity: warning

      # 8. Scrape 失敗率高
      - alert: PrometheusTargetScrapeFailing
        expr: |
          sum by (job) (rate(prometheus_target_scrapes_sample_out_of_order_total[5m]))
          > 0
        for: 15m

§ 3. Alertmanager 端的自監控

3.1 Alertmanager 必設告警

groups:
  - name: alertmanager_self
    rules:
      # 1. AM 自己掛了
      - alert: AlertmanagerDown
        expr: up{job="alertmanager"} == 0
        for: 5m
        labels:
          severity: critical

      # 2. Cluster 退化（副本數不對）
      - alert: AlertmanagerClusterDegraded
        expr: |
          alertmanager_cluster_members
          != on() count(up{job="alertmanager"} == 1)
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "AM cluster has {{ $value }} members, expected 3"

      # 3. 設定載入失敗
      - alert: AlertmanagerConfigInconsistent
        expr: count by (service) (alertmanager_config_hash) != 1
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "AM cluster has inconsistent config"

      # 4. 通知失敗
      - alert: AlertmanagerNotificationsFailing
        expr: |
          rate(alertmanager_notifications_failed_total[5m])
          / rate(alertmanager_notifications_total[5m])
          > 0.1
        for: 10m
        labels:
          severity: critical
        annotations:
          summary: "{{ $value | humanizePercentage }} notifications failing on {{ $labels.integration }}"

      # 5. 通知延遲高
      - alert: AlertmanagerNotificationsHighLatency
        expr: |
          histogram_quantile(0.99,
            rate(alertmanager_notification_latency_seconds_bucket[5m])
          ) > 30
        for: 15m
        labels:
          severity: warning

      # 6. Cluster gossip 不健康
      - alert: AlertmanagerClusterPeersFailed
        expr: alertmanager_cluster_failed_peers > 0
        for: 10m
        labels:
          severity: warning

      # 7. 接收太多 alert（可能有問題）
      - alert: AlertmanagerAlertFloodingDetected
        expr: rate(alertmanager_alerts_received_total[5m]) > 100
        for: 15m
        labels:
          severity: warning

3.2 觀察的關鍵 metrics

# 整體流量
rate(alertmanager_alerts_received_total[5m])
rate(alertmanager_notifications_total[5m])

# 每個 receiver / integration 的延遲
histogram_quantile(0.99,
  rate(alertmanager_notification_latency_seconds_bucket[5m])
)

# Cluster 狀態
alertmanager_cluster_members
alertmanager_cluster_health_score      # 0=healthy, >0=有問題

# Silence 狀況
alertmanager_silences{state="active"}

# 設定 hash（cluster 所有 AM 應該一樣）
alertmanager_config_hash

§ 4. 「Watchdog Alert」機制

4.1 概念

一條永遠 firing 的測試 alert，定期透過整個 pipeline。 如果某天您沒收到 watchdog 通知，代表 pipeline 某環節壞了。

4.2 設定 Watchdog

# Prometheus alerting rule
- alert: Watchdog
  annotations:
    summary: |
      This is a watchdog meant to ensure the alerting pipeline is functional.
      You should always have this alert firing.
  expr: vector(1)            # 永遠 = 1（永遠 firing）
  labels:
    severity: none

用 vector(1) 這個 trick 讓 expr 永遠回傳值。

4.3 Alertmanager 路由

route:
  routes:
    # Watchdog 走特殊 receiver
    - matchers:
        - alertname = "Watchdog"
      receiver: 'watchdog'
      group_wait: 0s
      group_interval: 15s     # 每 15 秒就重發
      repeat_interval: 15s    # 持續通知

receivers:
  - name: 'watchdog'
    webhook_configs:
      - url: 'https://heartbeat.example.com/alertmanager-watchdog'
        send_resolved: false

4.4 Watchdog 接收端

您需要一個服務接收 watchdog webhook，並追蹤「最後收到的時間」。 若超過 N 分鐘沒收到，觸發外部告警（例如 PagerDuty heartbeat）。 

# 簡易實作（Python）
from flask import Flask, request
import time

app = Flask(__name__)
last_received = {"timestamp": 0}

@app.route('/alertmanager-watchdog', methods=['POST'])
def watchdog():
    last_received["timestamp"] = time.time()
    return 'OK', 200

@app.route('/health')
def health():
    age = time.time() - last_received["timestamp"]
    if age > 300:    # 超過 5 分鐘沒收到
        return f"Watchdog stale: {age}s ago", 503
    return f"OK: {age}s ago", 200

用另一個監控（如 BetterUptime / Pingdom）打 /health，503 時通知。

§ 5. Dead Man's Switch（外部監控）

更高層次的保險：用第三方服務驗證 AM pipeline。 如果整個內部監控掛了（IaaS 出問題、DC 斷電），第三方仍能告訴您。

5.1 PagerDuty Heartbeat

# Watchdog receiver 改成發到 PagerDuty Heartbeat URL
receivers:
  - name: 'pd-heartbeat'
    webhook_configs:
      - url: 'https://events.pagerduty.com/v2/enqueue'
        send_resolved: false
        http_config:
          # 用 PagerDuty 的 Heartbeat integration key
        # 但通常更簡單的是直接 HTTP POST：
      - url: 'https://heartbeat.example.com/'

PD 設定：建立一個「Heartbeat」服務，預期每分鐘收到一次 ping。沒收到就 page on-call。

5.2 第三方 Dead Man's Switch 服務

  • Dead Man's Snitch
  • Healthchecks.io（開源 + SaaS）
  • PagerDuty 內建的 Heartbeat
  • Grafana Cloud 的 Synthetic Monitoring

5.3 完整自監控架構

§ 6. Incident Tools 全景

Alertmanager 把通知送出去，但接下來呢？ Incident Management 工具處理「告警變成事件後的工作流」：誰值班、escalation、post-mortem。

6.1 三類工具

類別核心能力代表
傳統 Paging On-call schedule、escalation、SMS/電話 PagerDuty、OpsGenie、VictorOps
Modern Incident Response 整合 Slack 工作流、自動 status page、post-mortem incident.io、FireHydrant、Rootly
OSS / Self-Hosted 類似上述但開源 Grafana OnCall、Cachet

6.2 對照表

工具類型適用

§ 7. PagerDuty 完整整合

7.1 PD 的核心概念

  • Service：一個被監控的對象（一個 AM routing_key 對應一個 service）
  • Escalation Policy：第 1 個 on-call 沒 ack？升級到下一個
  • Schedule：誰這週值班
  • Incident：alert firing 後產生的事件
  • Notification Rule：on-call 自己設定如何被通知（SMS / email / call）

7.2 Escalation Policy 範例

Step 1 (0 分):  通知 primary on-call (SMS)
Step 2 (5 分):  若沒 ack，加上 secondary on-call (電話)
Step 3 (15 分): 加上 manager (電話 + email)
Step 4 (30 分): 加上整個 team (Slack channel)

7.3 Alert → PD → Incident 流程

sequenceDiagram participant P as Prometheus participant A as Alertmanager participant PD as PagerDuty participant ON as On-call P->>A: alert firing A->>PD: POST events API
(routing_key) PD->>PD: 建立 Incident PD->>ON: SMS · App push ON->>PD: Acknowledge PD->>PD: 標記 Acknowledged Note over PD,ON: 處理問題中... P->>A: alert resolved A->>PD: POST events API
(event_action=resolve) PD->>PD: Auto-Resolve Incident

7.4 PD-Slack 雙向整合

PD 在 Slack 中能：

  • 自動發訊息到 channel：「@alice has been paged for HighErrorRate」
  • 讓人在 Slack 中 ack / resolve（按按鈕）
  • 建立 Slack channel 給 incident 討論用

設定：在 PagerDuty 安裝 Slack integration、為 service 啟用 Slack updates。

7.5 PagerDuty Routing 進階

同一個 service 可以有多個 integration（routing key）：

# Critical 走 PD-Critical 服務
receivers:
  - name: 'pd-critical'
    pagerduty_configs:
      - routing_key: ''

  - name: 'pd-warning'
    pagerduty_configs:
      - routing_key: ''

  - name: 'pd-database-team'
    pagerduty_configs:
      - routing_key: ''

每個 routing_key 對應不同 PD service，可有各自的 escalation policy（critical 立即電話、warning 隔 30 分才升級）。

§ 8. Grafana OnCall

Grafana OnCall 是 Grafana Labs 在 2022 年開源的 on-call management（基於 Amixr 收購）。 特色：免費、自架、跟 Grafana 深度整合。

8.1 主要能力

  • On-call schedule（rotation、override、shadow）
  • Escalation policy
  • 多通知 channel（Slack、Telegram、Mobile App、Phone Call、SMS）
  • 跟 Grafana datasource 整合（直接從 panel 觸發）
  • 整合 Alertmanager（webhook receiver）

8.2 從 Alertmanager 接 Grafana OnCall

# Alertmanager
receivers:
  - name: 'grafana-oncall'
    webhook_configs:
      - url: 'https://oncall.grafana.com/integrations/v1/alertmanager//'
        send_resolved: true

Grafana OnCall 收到 webhook 後，按其內建 escalation policy 通知 on-call。

8.3 何時選 Grafana OnCall vs PagerDuty

成熟度2022 起，仍快速演進10+ 年成熟
Mobile App有但較陽春業界最佳
Phone Call需付費供應商（Twilio）內建
已用 Grafana★★★ 最好仍可整合

§ 9. incident.io / FireHydrant

這兩個是「modern incident response」工具。它們不取代 PagerDuty， 而是處理「page 之後」的工作流——尤其是 Slack-centric 的事件管理。

9.1 incident.io 工作流範例

  1. PD page 主 on-call
  2. 主 on-call 在 Slack 打 /incident declare
  3. incident.io 自動建一個 Slack channel #inc-2026-05-02-api-down
  4. Channel 中有 incident.io bot：/inc severity major、/inc role assign commander @alice
  5. 自動更新 status page（顯示給 customer）
  6. 事件解決後，bot 自動：通知 stakeholder、產生 timeline、開啟 retrospective doc

9.2 與 Alertmanager 整合

# 用 webhook 把 alert 送進 incident.io（產生 incident）
receivers:
  - name: 'incident-io'
    webhook_configs:
      - url: 'https://api.incident.io/v2/integrations/alertmanager/your-alert-source-id'
        http_config:
          authorization:
            type: Bearer
            credentials_file: /etc/alertmanager/secrets/incident-io-token
        send_resolved: true

9.3 為什麼這類工具流行？

傳統 PD 解決「叫醒 on-call」，但 incident response 涉及更多——溝通、status update、post-mortem。 Modern tools 把這些標準化到 Slack 工作流，減少 incident 處理的人工成本。

10.1 完整端到端流程

10.2 Severity 與工具對應

Severity動作
SEV-1（嚴重） PD page + Slack #incidents + incident.io 自動建 channel + 自動更新 status page
SEV-2（重要） PD page + Slack channel
SEV-3（警告） Slack 通知（不 page）
SEV-4（資訊） 只記錄到 dashboard，不通知

10.3 從 Alert 觸發整套工作流的範例設定

route:
  receiver: 'default'

  routes:
    # SEV-1：page + Slack + incident.io
    - matchers:
        - severity = "critical"
      group_wait: 10s
      repeat_interval: 1h

      routes:
        # 同時送多個系統，全部 continue
        - matchers: ['severity = "critical"']
          receiver: 'pd-critical'
          continue: true
        - matchers: ['severity = "critical"']
          receiver: 'slack-incidents'
          continue: true
        - matchers: ['severity = "critical"']
          receiver: 'incident-io'

    # SEV-2/3：只通知，不 page
    - matchers:
        - severity =~ "warning|info"
      receiver: 'slack-team-{{ .CommonLabels.team }}'

10.4 Post-Mortem 文化

每個 SEV-1/2 事件都該寫 post-mortem：

  • What happened：時間軸、影響範圍
  • Root cause：5 Whys 分析
  • Detection：怎麼發現的？花了多久？
  • Resolution：怎麼修的？
  • Action items：避免再次發生
  • Lessons learned

incident.io / FireHydrant 都會自動產生 post-mortem template + timeline。

✅ 本章重點摘要

30 秒回顧
  • 三層自監控：內部 metrics + Watchdog Alert + Dead Man's Switch（外部）
  • Prometheus 必設告警：down / config reload / rule eval / notification 失敗
  • Alertmanager 必設告警：down / cluster degraded / config inconsistent / notification failed
  • Watchdog：永遠 firing 的 alert（用 vector(1)）+ webhook receiver 確認 pipeline 通
  • Dead Man's Switch：第三方服務（PD Heartbeat / healthchecks.io）監控 watchdog
  • Incident tools 三類：傳統 paging（PD/OpsGenie）、modern（incident.io）、OSS（Grafana OnCall）
  • PagerDuty 核心：Service + Escalation Policy + Schedule
  • Grafana OnCall：免費 OSS 替代 PD，Grafana 環境的好選擇
  • incident.io / FireHydrant：管「page 之後」的 Slack 工作流 + status page + post-mortem
  • SEV-1 工作流：PD page + Slack + 自動 incident channel + status page
