Prometheus 生態系全景

Prometheus 並不是孤立的工具，而是一個由核心、輔助元件、資料來源、長期儲存方案構成的完整生態系。 本章帶您鳥瞰整個生態地圖，了解何時該選用哪個元件。

§ 3. Exporters：把任何系統變成 Prometheus 來源

§ 4. Client Libraries：為應用程式埋點

當您自己寫的應用程式需要暴露 metrics 時，使用對應語言的 client library 是最佳做法。 這些函式庫負責：

  • 定義 Counter / Gauge / Histogram / Summary 等 metric 類型
  • 處理併發安全（thread/goroutine safe）
  • 提供 HTTP handler 暴露 /metrics endpoint
  • 支援 push 模式（給 Pushgateway 用）

§ 5. Pushgateway：短任務的橋樑

Prometheus 的 pull 模型對短任務（short-lived jobs）不友善。 例如：每 5 分鐘執行一次的批次清理 cron job，跑 30 秒就結束， 可能根本還沒被 Prometheus 抓到就消失了。

Pushgateway 解決這個問題：批次任務把 metrics push 到 Pushgateway， Prometheus 再從 Pushgateway 拉取。Pushgateway 像個「快取暫存」。

sequenceDiagram participant J as Cron Job participant PG as Pushgateway participant P as Prometheus Note over J: 每 5 分鐘啟動 J->>J: 執行清理任務 J->>PG: POST metrics (job_duration, items_cleaned) Note over J: 任務結束 Note over PG: 保存最新值 P->>PG: GET /metrics (定期拉取) PG-->>P: 回傳 metrics
⚠️ Pushgateway 不是 push 模型的萬能解

官方明確指出：Pushgateway 只應用於「服務級別批次任務（service-level batch jobs）」。 它不適合：機器級別的指標（用 node_exporter）、 取代 pull 模型（會失去自動健康檢查）、儲存大量短期資料（沒有時序資料庫的能力）。 Ch 21 會深入討論。

§ 6. 長期儲存方案：Thanos vs Mimir vs VictoriaMetrics

Prometheus 的本地 TSDB（time-series database）有幾個天生的限制：

  • 單機儲存：磁碟空間是上限，無法水平擴展
  • 無高可用：單點故障
  • 資料保留有限：通常設定 15-30 天
  • 無全域查詢：多個 Prometheus 互相不知道對方的資料

當資料量、保留期、多租戶需求超出單機能力時，就需要長期儲存方案。 目前業界三大主流選擇：

6.1 Thanos

由 Improbable 開源、現為 CNCF Incubating 專案。採sidecar 模式： 在每個 Prometheus 旁邊跑一個 Thanos sidecar，把資料區塊（block）上傳到物件儲存（S3/GCS/Azure Blob）。

graph TB subgraph Cluster1["叢集 1"] P1[Prometheus] --> S1[Thanos Sidecar] end subgraph Cluster2["叢集 2"] P2[Prometheus] --> S2[Thanos Sidecar] end S1 -->|upload blocks| OS[(物件儲存
S3/GCS/Azure)] S2 -->|upload blocks| OS SG[Thanos Store Gateway] --> OS CMP[Thanos Compactor] --> OS Q[Thanos Querier] Q -.gRPC.-> S1 Q -.gRPC.-> S2 Q -.gRPC.-> SG G[Grafana] --> Q

優勢：

  • 對現有 Prometheus 改動小，遷移友善
  • 使用物件儲存，成本低
  • CNCF Incubating，社群成熟
  • 支援 downsampling（降採樣，5 分鐘、1 小時粒度）

劣勢：

  • 元件多（Sidecar / Querier / Store / Compactor / Receiver / Ruler），維運複雜
  • 跨多 sidecar 查詢的延遲較高
  • 多租戶支援較弱

6.2 Grafana Mimir

由 Grafana Labs 開源（從 Cortex fork 而來）。採microservices 架構， 所有 Prometheus 透過 remote_write 寫入 Mimir 集中處理。

graph TB subgraph Sources["資料來源"] P1[Prometheus 1] P2[Prometheus 2] P3[Prometheus N] end subgraph Mimir["Grafana Mimir"] D[Distributor] I[Ingester] QF[Query Frontend] Q[Querier] SG[Store Gateway] CMP[Compactor] R[Ruler] end OS[(物件儲存)] P1 -.remote_write.-> D P2 -.remote_write.-> D P3 -.remote_write.-> D D --> I I --> OS SG --> OS CMP --> OS QF --> Q Q --> I Q --> SG G[Grafana] --> QF

優勢：

  • 原生強大的多租戶（multi-tenancy）支援
  • 水平擴展能力強，可達億級 active series
  • Grafana Labs 商業支援
  • 查詢效能優於 Thanos sidecar 模式

劣勢：

  • microservices 架構複雜，通常需要 Kubernetes 與專業 SRE 團隊
  • 從現有 Prometheus 遷移需設定 remote_write

6.3 VictoriaMetrics

由 VictoriaMetrics Inc. 開源，從零打造、不基於 Prometheus 程式碼， 主打高效能與低資源消耗。

graph TB subgraph Sources["資料來源"] P[Prometheus] VMA[vmagent
· 替代 Prometheus ] OT[OpenTelemetry] end subgraph VM["VictoriaMetrics Cluster"] VINS[vminsert] VSEL[vmselect] VSTOR[vmstorage] end VMR[vmalert] DISK[(本地 SSD
或網路硬碟)] P -.remote_write.-> VINS VMA --> VINS OT --> VINS VINS --> VSTOR VSTOR --> DISK VSEL --> VSTOR G[Grafana] --> VSEL VMR --> VSEL

優勢：

  • 單檔執行（single binary）部署簡單
  • 記憶體用量比 Prometheus 少 4-5 倍，CPU 也低
  • 儲存壓縮率最高（10x vs Prometheus 原生）
  • 支援 PromQL 加上自家擴充的 MetricsQL
  • 支援多種寫入協議（Prometheus remote_write、InfluxDB、Graphite、OpenTSDB、CSV、JSON）

劣勢：

  • 不在 CNCF 託管
  • 使用區塊儲存（block storage）而非物件儲存，災難恢復策略不同
  • 商業擴充功能（如進階 anomaly detection）需付費

Cortex 是 Mimir 的前身，原為 Weaveworks 開源，後 Grafana Labs fork 成 Mimir。 Cortex 維護者大多遷移到 Mimir，目前 Cortex 處於維護模式， 不建議新部署選用。如果您現在還在用 Cortex，建議規劃遷移到 Mimir。

§ 7. Prometheus Operator 與 kube-prometheus-stack

在 Kubernetes 環境中，手動管理 Prometheus 設定檔非常繁瑣。 Prometheus Operator 透過 Kubernetes Custom Resource Definitions（CRDs） 讓您用宣告式（declarative）方式定義監控設定。

7.1 主要 CRDs

CRD作用
Prometheus定義一個 Prometheus 實例
Alertmanager定義一個 Alertmanager 實例
ServiceMonitor定義要抓取哪些 K8s Service 的 metrics
PodMonitor定義要抓取哪些 Pod 的 metrics（不需 Service）
Probe定義 Blackbox 探測目標
PrometheusRule定義 Recording Rules 和 Alerting Rules
AlertmanagerConfig定義 Alertmanager 路由與接收器
ScrapeConfig定義原生 scrape config（v0.65+）
ThanosRuler定義 Thanos Ruler 實例

7.2 kube-prometheus-stack

這是最常用的 Helm Chart，「一鍵」部署整套監控方案，包含：

  • Prometheus Operator
  • Prometheus（含 K8s 標準 alerting rules）
  • Alertmanager
  • Grafana（含 K8s 標準 dashboards）
  • node_exporter
  • kube-state-metrics
  • 所有必要的 ServiceMonitor 設定

Ch 54 將以實戰案例完整演示 kube-prometheus-stack 的部署與客製化。

✅ 本章重點摘要

30 秒回顧
  • Prometheus 生態系分五大類：採集 / 核心 / 告警 / 長期儲存 / 視覺化
  • Exporters（200+ 個）讓任何系統都能變成 Prometheus 的資料來源
  • Client libraries 支援 Go / Java / Python / Ruby / Rust 官方，加上 .NET / Node.js / PHP / C++ 等社群版本
  • Pushgateway 只用於服務級短任務，不要濫用
  • 長期儲存三大選擇：Thanos（已有 Prom）、Mimir（多租戶）、VictoriaMetrics（簡單高效）
  • K8s 環境用 Prometheus Operator + kube-prometheus-stack 是業界主流
  • Cortex 已停滯，新部署不要選
