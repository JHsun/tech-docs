Chapter 18 · Exporters 全景

Exporters 全景：官方、主流、選型

當您要監控的應用本身不直接暴露 Prometheus metrics（資料庫、訊息佇列、舊系統等）， 您需要 exporter — 一個翻譯層，把目標系統的指標轉換成 Prometheus 格式。 本章帶您認識官方提供的核心 exporter、主流社群版本，以及該怎麼選。

§ 1. Exporter 是什麼？分類有哪些

Exporter 的本質是一個 HTTP server，它從目標系統取資料， 重新格式化為 Prometheus exposition format，暴露在 /metrics endpoint 供 Prometheus 抓。

graph LR TARGET[目標系統
MySQL / Redis / Server] EXPORTER[Exporter
HTTP server :PORT] PROM[Prometheus] TARGET -- 原生 API ·
SQL · /proc 等 --> EXPORTER EXPORTER -- /metrics
Prometheus 格式 --> PROM

1.1 兩種主要部署模式

模式說明適用
Sidecar / Co-located Exporter 跑在被監控系統旁邊（同 pod / VM） node_exporter、postgres_exporter
Centralized / Multi-Target 單一 exporter 服務多個 target，target 透過 URL 參數傳入 blackbox_exporter、snmp_exporter

1.2 Exporter 來源分類

類型說明範例
官方（Official） 由 Prometheus organization 維護，最高品質 node、blackbox、mysqld、snmp、memcached、graphite、consul、statsd、pushgateway
第三方（Community） 社群維護，品質參差，但很多很成熟 postgres_exporter、redis_exporter、nginx-prometheus-exporter
內建直接暴露（Direct） 軟體本身就吐 Prometheus 格式，無需 exporter Kubernetes、etcd、Envoy、Istio、Loki、Tempo、CockroachDB...
ℹ️ 越來越多軟體「內建」

過去十年趨勢：新軟體幾乎都直接吐 Prometheus 格式，不再需要 exporter。 OpenTelemetry 也讓「應用層直接吐 metric」更容易。
Exporter 主要服務於：傳統軟體（MySQL、Redis 等）、無法改的舊系統、外部探測。

§ 2. node_exporter（機器指標）

node_exporter 是最被廣泛部署的 exporter，提供 Linux/Unix 機器的硬體與作業系統指標。 監聽預設 port 9100。

2.1 提供什麼指標

類別典型 metric
CPUnode_cpu_seconds_total（按 mode 分）、node_load1/5/15
Memorynode_memory_MemTotal_bytes、node_memory_MemAvailable_bytes
Disknode_filesystem_avail_bytes、node_disk_io_time_seconds_total
Networknode_network_receive_bytes_total、node_network_transmit_bytes_total
Systemnode_uname_info（OS info）、node_boot_time_seconds
Processnode_processes_state（zombie / running...）

2.2 安裝與啟動

# Binary 啟動
wget https://github.com/prometheus/node_exporter/releases/download/v1.9.1/node_exporter-1.9.1.linux-amd64.tar.gz
tar xvf node_exporter-*.tar.gz
cd node_exporter-*
./node_exporter

# Docker 啟動（注意要 mount 宿主目錄）
docker run -d \
  --name node-exporter \
  --net="host" \
  --pid="host" \
  -v "/:/host:ro,rslave" \
  prom/node-exporter:latest \
  --path.rootfs=/host

# K8s（DaemonSet 部署，每節點一個）— kube-prometheus-stack 已包含

2.3 Collector 啟用 / 停用

node_exporter 由眾多「collector」模組組成，可獨立開關：

# 預設啟用：cpu, diskstats, filesystem, loadavg, meminfo, netdev, ...

# 停用 hwmon collector（某些主機 sensor 太多會爆 series）
./node_exporter --no-collector.hwmon

# 啟用 textfile collector（自訂 metric 從檔案讀）
./node_exporter --collector.textfile.directory=/var/lib/node_exporter/textfile_collector

# 啟用 systemd（看 service 狀態）
./node_exporter --collector.systemd

2.4 textfile collector（重要技巧）

讓您透過寫檔案暴露自訂 metric，無需寫程式： 

# /var/lib/node_exporter/textfile_collector/backup.prom
backup_last_success_timestamp_seconds 1714521600
backup_size_bytes 5368709120

# 您的 cron job 結束後寫入這個檔
# node_exporter 會自動讀取並暴露這些 metric

適合 batch job、cron task、無法改程式碼的腳本。

§ 3. Blackbox Exporter（外部探測）

blackbox_exporter 從外部探測 endpoint 是否可達， 支援 HTTP、HTTPS、TCP、ICMP、DNS、gRPC。預設 port 9115。

3.1 為什麼叫 "blackbox"？

相對於「白盒監控」（從應用內部看），blackbox 是「黑盒監控」 — 不知道內部，只看外部行為。 例：「我能不能 HTTP GET 這個 URL 並拿到 200？」、「我能 ping 通這個 IP 嗎？」

3.2 設定 module

# /etc/blackbox/blackbox.yml
modules:
  http_2xx:
    prober: http
    timeout: 5s
    http:
      valid_status_codes: [200, 201, 204]
      method: GET
      preferred_ip_protocol: ip4
      fail_if_ssl: false
      fail_if_not_ssl: false

  http_post_login:
    prober: http
    timeout: 10s
    http:
      method: POST
      headers:
        Content-Type: application/json
      body: '{"username":"check","password":"check"}'
      valid_status_codes: [401]      # 401 也算 "endpoint 活著"

  tcp_connect:
    prober: tcp
    timeout: 5s

  icmp_ping:
    prober: icmp
    timeout: 3s

  dns_check:
    prober: dns
    timeout: 5s
    dns:
      query_name: example.com
      query_type: A
      preferred_ip_protocol: ip4

3.3 Prometheus 端設定（關鍵的 relabeling）

Blackbox 用 multi-target pattern，需要透過 relabeling 把 target 傳進 URL：

scrape_configs:
  - job_name: 'blackbox'
    metrics_path: /probe
    params:
      module: [http_2xx]
    static_configs:
      - targets:
          - 'https://api.example.com/health'
          - 'https://www.example.com'
          - 'https://login.example.com'
    relabel_configs:
      # 把 target 變成 ?target=... query param
      - source_labels: [__address__]
        target_label: __param_target
      # 把 target 設為 instance label（給人看）
      - source_labels: [__param_target]
        target_label: instance
      # 把 __address__ 改成 blackbox exporter 的位址
      - target_label: __address__
        replacement: blackbox-exporter:9115

3.4 主要 metrics

Metric意義
probe_success1 = 成功，0 = 失敗
probe_duration_seconds探測總耗時
probe_http_status_code實際 HTTP 狀態碼
probe_http_ssl是否使用 SSL
probe_ssl_earliest_cert_expirySSL 憑證最早到期時間（重要！）
probe_http_duration_seconds各階段耗時（resolve/connect/tls/processing）
probe_dns_lookup_time_secondsDNS 解析耗時

3.5 經典告警：SSL 憑證即將過期

- alert: SSLCertificateExpiringSoon
  expr: probe_ssl_earliest_cert_expiry - time() < 86400 * 14
  for: 1h
  labels:
    severity: warning
  annotations:
    summary: "SSL cert for {{ $labels.instance }} expires in less than 14 days"

§ 4. SNMP Exporter（網路設備）

監控交換器、路由器、防火牆、UPS 等網路設備的標準方案。 SNMP 是工業界主流協定，幾乎所有網路硬體都支援。

4.1 三步驟流程

  1. 用 snmp_exporter generator 從 MIB 檔產生 snmp.yml
  2. 啟動 snmp_exporter 讀取此設定
  3. Prometheus 用 multi-target 模式（類似 blackbox）抓不同設備

4.2 Prometheus 設定

scrape_configs:
  - job_name: 'snmp_switches'
    static_configs:
      - targets:
          - 192.168.1.1   # core switch
          - 192.168.1.2   # access switch
    metrics_path: /snmp
    params:
      module: [if_mib]    # 在 snmp.yml 中定義的 module
    relabel_configs:
      - source_labels: [__address__]
        target_label: __param_target
      - source_labels: [__param_target]
        target_label: instance
      - target_label: __address__
        replacement: snmp-exporter:9116

§ 5. mysqld_exporter / postgres_exporter

5.1 mysqld_exporter（官方）

監聽 port 9104。需要建立專屬 MySQL user：

-- 在 MySQL 中建立 metrics 用 user
CREATE USER 'exporter'@'localhost' IDENTIFIED BY 'StrongPassword' WITH MAX_USER_CONNECTIONS 3;
GRANT PROCESS, REPLICATION CLIENT, SELECT ON *.* TO 'exporter'@'localhost';
# DSN 設定
export DATA_SOURCE_NAME='exporter:StrongPassword@(localhost:3306)/'

./mysqld_exporter \
  --collect.global_status \
  --collect.global_variables \
  --collect.info_schema.processlist \
  --collect.info_schema.innodb_metrics \
  --collect.slave_status

5.2 postgres_exporter（社群，prometheus-community）

# 環境變數
export DATA_SOURCE_NAME='postgresql://exporter:password@localhost:5432/postgres?sslmode=disable'

# 啟動（port 9187）
./postgres_exporter

# 啟用自訂 query（很常用）
./postgres_exporter \
  --extend.query-path=/etc/postgres_exporter/queries.yaml

自訂 query 範例：

# queries.yaml
pg_replication_lag:
  query: |
    SELECT EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp()))::FLOAT AS lag
  metrics:
    - lag:
        usage: "GAUGE"
        description: "Replication lag in seconds"

5.3 主要 metrics

類別典型 metric
連線mysql_global_status_threads_connected、pg_stat_activity_count
QPSmysql_global_status_queries、pg_stat_database_xact_commit
InnoDBmysql_innodb_buffer_pool_pages_*
Replicationmysql_slave_lag_seconds、自訂 pg_replication_lag

§ 6. JMX Exporter（JVM 應用）

從任何 JVM 應用（Java、Scala、Kotlin）暴露 metric，包括 Kafka、Cassandra、Spark、Tomcat。 有兩種運行模式：

模式說明
Java Agent（推薦） JVM 啟動時用 -javaagent 載入，metric 內嵌在 application port
Standalone HTTP 透過 JMX 連到應用，獨立 process

6.1 Java Agent 模式

# 啟動 Java 應用時加上 agent
java \
  -javaagent:./jmx_prometheus_javaagent-0.20.0.jar=8080:config.yaml \
  -jar your-app.jar

# config.yaml 範例
rules:
  - pattern: 'java.langused'
    name: jvm_memory_heap_used_bytes
    type: GAUGE
  - pattern: 'kafka.serverOneMinuteRate'
    name: kafka_messages_in_per_sec
    type: GAUGE

6.2 常用 JMX rules 集合

業界已有現成 config 可直接用：

  • 官方 example_configs — Kafka、Cassandra、Tomcat、Hadoop 等
  • criteo 的 config generator

§ 7. kube-state-metrics（K8s 物件）

在 K8s 環境裡，kube-state-metrics（KSM）暴露 K8s API 物件的狀態 — Deployment、ReplicaSet、Pod、Node、PV 等等。 它不是監控 K8s control plane（那是 K8s 內建 metric），而是看物件當前狀態。

7.1 與 cAdvisor、metrics-server 的區別

元件看什麼
kube-state-metricsK8s 物件狀態（pod 數、deployment replicas、PV 容量...）
cAdvisor容器資源實際使用（CPU、memory、network）
metrics-server用於 HPA / kubectl top，不是給 Prometheus 用
K8s control plane metricskube-apiserver、kube-controller-manager 等內建 metric

7.2 主要 metrics

# Deployment 期望與實際 replica 數
kube_deployment_status_replicas
kube_deployment_status_replicas_available
kube_deployment_spec_replicas

# Pod 狀態
kube_pod_status_phase{phase="Pending|Running|Succeeded|Failed"}
kube_pod_container_status_restarts_total
kube_pod_status_ready

# Node 資源
kube_node_status_allocatable{resource="cpu|memory"}
kube_node_status_capacity

# PVC
kube_persistentvolumeclaim_status_phase
kube_persistentvolumeclaim_resource_requests_storage_bytes

7.3 部署

kube-prometheus-stack（Helm chart）已自動包含 KSM，無需另外設定。獨立部署：

helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm install kube-state-metrics prometheus-community/kube-state-metrics \
  --namespace monitoring

§ 8. cAdvisor（容器資源）

cAdvisor（Container Advisor）由 Google 開發，內建在 kubelet， 暴露每個容器的實際 CPU、memory、network、filesystem 使用情況。

8.1 不需要單獨部署

K8s 環境下 kubelet 自帶 cAdvisor，metrics 在每個 node 的 /metrics/cadvisor 暴露：

scrape_configs:
  - job_name: 'kubelet-cadvisor'
    kubernetes_sd_configs:
      - role: node
    scheme: https
    tls_config:
      ca_file: /var/run/secrets/kubernetes.io/serviceaccount/ca.crt
    bearer_token_file: /var/run/secrets/kubernetes.io/serviceaccount/token
    relabel_configs:
      - target_label: __address__
        replacement: kubernetes.default.svc:443
      - source_labels: [__meta_kubernetes_node_name]
        regex: (.+)
        target_label: __metrics_path__
        replacement: /api/v1/nodes/${1}/proxy/metrics/cadvisor

8.2 主要 metrics

# 容器 CPU
container_cpu_usage_seconds_total
container_cpu_cfs_throttled_seconds_total       # 被 CPU throttle 的時間（重要！）

# 容器記憶體
container_memory_working_set_bytes              # 工作集（建議用這個，不是 RSS）
container_memory_max_usage_bytes
container_memory_failures_total

# 容器網路
container_network_receive_bytes_total
container_network_transmit_bytes_total

# 容器檔案系統
container_fs_usage_bytes

8.3 cAdvisor 的高基數警告

⚠️ 預設 cAdvisor metric 可能爆炸

cAdvisor 預設帶大量 label（image、container_name、id、pod、namespace、node...）， 在 1000+ pod 環境下可能產生數百萬 series。
用 metric_relabel_configs drop 不必要的 metric： 

metric_relabel_configs:
  - source_labels: [__name__]
    regex: 'container_(blkio_|memory_failures_total|tasks_state).*'
    action: drop
  - source_labels: [container_label_io_kubernetes_pod_name]
    regex: '(.+)'
    target_label: pod
    action: replace

§ 10. Multi-Target Pattern

這是 Prometheus 監控架構的重要 pattern。 blackbox、snmp、與某些自訂 exporter 都採用此模式 — 單一 exporter，目標透過 URL 參數傳入。

10.1 為什麼這個 pattern 存在

考慮 SSL 憑證監控：您要監控 100 個網域。如果每個 exporter 只能服務一個 target， 您需要部署 100 個 exporter — 不切實際。
Multi-target：部署一個 blackbox_exporter，Prometheus 對每個 target 發起 ?target=https://xxx 的 HTTP 請求，exporter 動態探測。

10.2 流程圖

sequenceDiagram participant P as Prometheus participant R as Relabeling participant E as Blackbox Exporter participant T as Target URL P->>R: 取得 target list
例: api.example.com R->>R: 改寫 __address__
變成 blackbox-exporter:9115
__param_target=api.example.com R->>P: 改寫後的 scrape job P->>E: GET /probe?target=api.example.com&module=http_2xx E->>T: 發起 HTTP probe T-->>E: HTTP response E-->>P: probe_success=1, probe_duration_seconds=0.234, ...

10.3 通用 relabeling 模板

scrape_configs:
  - job_name: 'multi-target'
    metrics_path: /probe              # 或 /snmp
    params:
      module: [your_module]
    static_configs:
      - targets:
          - target1.example.com
          - target2.example.com
    relabel_configs:
      # 1. 把要探測的 target 變成 query param
      - source_labels: [__address__]
        target_label: __param_target
      # 2. 用 target 當 instance label
      - source_labels: [__param_target]
        target_label: instance
      # 3. 把 __address__ 改成 exporter 自身的位址
      - target_label: __address__
        replacement: exporter-host:9115
