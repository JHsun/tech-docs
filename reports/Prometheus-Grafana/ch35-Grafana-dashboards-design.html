Chapter 35 · Dashboards 設計

Dashboards 設計：Layout、Variables、Annotations、版本控管

Dashboard 不只是「把 panel 拼一起」。好的 dashboard 能讓 on-call 5 秒看出問題， 讓 PM 一眼看到業務指標，讓新人 30 秒上手 service。本章涵蓋 layout、variables、annotations、 links、版本控管，以及讓 dashboard 從「炫技」變成「有用」的設計原則。

§ 1. 設計原則：Dashboard 為誰而設計

1.1 三類 dashboard 對應三類使用者

類型讀者目的長度建議
Service Overview On-call、SRE 5 秒看出 service 健康狀況 1 螢幕（不需 scroll）
Deep Dive Engineer debugging 細節數據、各種維度切片 多螢幕、多 row
Business / KPI PM、Manager、CEO 單一可量化的 KPI 與趨勢 1-2 螢幕，少 panel

1.2 黃金原則

✅ 設計核心
  1. 每個 dashboard 一個目的：能用一句話說清楚這個 dashboard 解決什麼問題
  2. 最重要資訊放最上面：on-call 看一眼就知道 service 健康嗎
  3. Stat / Threshold 配色：紅 = 壞、綠 = 好，不要為了「色彩豐富」打破慣例
  4. Panel 最少化：刪掉「也許有用」的 panel，只留「一定有用」的
  5. 連結要明確：每個 panel 要能跳到 deeper view（log、trace、exporter raw metric）

§ 2. Layout 與 Grid System

2.1 24 column grid

Grafana 用 24 column grid。Panel 寬度是 1-24，typical：

寬度用途
24滿寬，row title 或關鍵 metric chart
12左右並列
8三欄
6四欄（典型 stat 列）
4六欄（小 stat）
3八欄（icon-like）

2.2 經典 Layout 模板

Service Overview Layout（最常見）：

第一列（高度 4）：4 個 stat panel（寬 6）
  [QPS] [Errors] [P99 Latency] [Saturation]

第二列（高度 8）：2 個 time series（寬 12）
  [Request Rate trend]  [Error Rate trend]

第三列（高度 8）：2 個 time series（寬 12）
  [Latency Percentiles] [Resources]

第四列（高度 6）：1 個 logs panel（寬 24）
  [Recent Errors from Loki]

2.3 Row（垂直分區）

Row 把相關 panel 群組起來，可以折疊：

📊 Overview (collapsed/expanded)
  └─ stat panels...

📊 HTTP Layer
  └─ request rate, errors, latency...

📊 Database Layer
  └─ query rate, slow queries...

📊 Infrastructure
  └─ CPU, memory, disk...

Row 加 collapse + variable 控制：

Row title: "Database - $instance"  ← 用 variable
Repeat: "instance" variable        ← 每個 instance 一行

§ 3. Dynamic Dashboards（v12+）

3.1 三大新能力

Grafana 12 引入 Dynamic Dashboards（experimental），帶來：

  • Tabs：水平分頁（之前只有 row）
  • Conditional Rendering：根據 variable 顯示/隱藏 panel
  • Auto-grid Layout：panel 自動調整大小填滿空間

3.2 Tabs 範例

Service Overview
├── Tab: "Production" (visible by default)
├── Tab: "Staging"
└── Tab: "Development"

每個 tab 有自己的 panel 集合
切 tab 不重新載 dashboard，只切 view

Tabs 適合：環境切換、地區切換、產品線切換。

3.3 Conditional Rendering

# 設定 panel 顯示條件
Panel: "Loki Logs"
  Show when: $datasource_type == "loki"

Panel: "Database queries"
  Show when: $service =~ ".*db.*"

避免「給所有 service 用同個 dashboard」時 panel 過多 / 不適用的問題。

3.4 Auto-grid

傳統 grid 是手動拖拉設大小。Auto-grid 讓 Grafana 自動：

  • 每行最多 N 個 panel（您指定）
  • 螢幕窄時自動換行
  • Panel 高度跟內容自適應

3.5 還在 experimental

⚠️ Dynamic Dashboards 是 experimental

v12 引入但仍在 experimental。啟用後 dashboard 會自動轉為 v2 schema， 無法 rollback 到 v1。
Production 等待 GA（預計 v13/v14）再用。Dev 環境可以試。

§ 4. Variables 完整使用

4.1 Variable 是什麼

Dashboard 頂部的下拉選單，讓您切換不同 instance/service/cluster 時，整個 dashboard 都跟著變。 Panel 用 $variable 引用其值。

4.2 Variable 類型

類型用途範例
Query從 datasource 動態取值label_values(up, instance)
Custom固定列表production,staging,dev
Constant固定常數（隱藏）API key、租戶 ID
Datasource選 datasource切換多 Prometheus instance
Interval時間 interval1m,5m,1h
Text Box使用者輸入自訂搜尋字串
Ad hoc filters動態加 label filter不用改 query

4.3 Query Variable（最常用）

Variable name: instance
Type: Query
Datasource: Prometheus

Query: label_values(node_uname_info, instance)
Sort: Alphabetical (asc)

Multi-value: ✅ (允許多選)
Include All option: ✅
Refresh: On Time Range Change

4.4 串聯 Variable（重要技巧）

讓「選 cluster 後，instance 下拉只顯示該 cluster 的」：

Variable 1: cluster
  Query: label_values(node_uname_info, cluster)

Variable 2: instance
  Query: label_values(node_uname_info{cluster="$cluster"}, instance)
                                              ^^^^^^^^^^^ 用上一個 variable

這個機制讓 dashboard scale 到「上百個 cluster + 上萬個 instance」也仍然好用。

4.5 Multi-value 與 All

多選時，PromQL 自動展開為 regex：

選擇了 alice, bob, carol 三個 instance：
$instance => "alice|bob|carol"

PromQL 中：
up{instance="$instance"}     ❌ 只支援單值
up{instance=~"$instance"}    ✅ regex 匹配

Multi-value variable 必須用 =~。

4.6 All option 處理

當使用者選 "All"，您可能要：

# 選項 1：All 自動展開為所有具體值（預設）
Custom all value: (留空)
=> 變成 "alice|bob|carol"

# 選項 2：用 .+ 匹配所有
Custom all value: .+
=> 變成 ".+"

# Panel：up{instance=~"$instance"}

4.7 隱藏 Variable

有些 variable 是「內部使用」，不想顯示給使用者：

  • Hide: Label：只隱藏 label，下拉仍可見
  • Hide: Variable：完全隱藏

§ 5. Variables 的進階技巧

5.1 全域變數（內建）

Grafana 提供一些預設變數，無需建立：

變數意義
$__intervalGrafana 算出的最佳 interval（隨 time range 變）
$__rate_intervalPrometheus 推薦的 rate interval（v8+）
$__range當前 time range 大小
$__from / $__to當前 time range 起 / 訖（UNIX MS）
$__dashboardDashboard 名稱（v8+）
$__user當前使用者名
$__org當前 org

5.2 Variable Format

同一個 variable 可以在不同地方有不同格式：

$instance              # 預設（multi-value 為 a|b|c）
${instance}            # 同上，明確分界
${instance:csv}        # a,b,c
${instance:json}       # ["a","b","c"]
${instance:raw}        # 原始值（不 escape）
${instance:doublequote} # "a","b","c"
${instance:singlequote} # 'a','b','c'
${instance:regex}      # 已 regex escape

SQL query 中用 :singlequote：

SELECT * FROM logs
WHERE service IN (${instance:singlequote})

5.3 Variable 用於 Repeat Panel

強大功能：一個 panel 設定 repeat: instance，會自動複製成 N 個 panel（每個 instance 一個）。 

Panel: "CPU Usage - $instance"
  Repeat: instance variable
  Direction: horizontal / vertical
  Max per row: 4

選 5 個 instance → 自動產生 5 個 panel。比手動建 5 個方便太多。

5.4 Repeat Row

Row 也能 repeat：「每個 instance 一個 row，row 內有 CPU、Memory、Disk」。

Row: "Instance: $instance"
  Repeat: instance variable

  Panel 1: CPU Usage
    expr: rate(node_cpu_seconds_total{instance="$instance"}[5m])
  Panel 2: Memory
    expr: node_memory_MemAvailable_bytes{instance="$instance"}

§ 6. Annotations

6.1 Annotations 是什麼

在 time series chart 上疊加「事件標記」（垂直線 / 區段），標示部署、incident、營運事件等。 幫助 correlate「出問題」與「發生了什麼」。

6.2 內建 Annotations

每個 dashboard 預設有 "Annotations & Alerts"，顯示 Grafana alert 觸發點。

6.3 自訂 Annotations

Dashboard settings → Annotations → New：

Name: Deployments
Datasource: Prometheus
Query: deployment_total{cluster="$cluster"}
Title: '{{ service }} v{{ version }}'
Tags: deployment

# 顯示為垂直線，hover 顯示 service / version

6.4 從 SQL 抓 Annotations

Datasource: PostgreSQL
Query:
SELECT
  $__time(deployed_at),
  service AS title,
  CONCAT(service, ' deployed: ', version) AS text,
  ARRAY['deployment', service] AS tags
FROM deployments
WHERE $__timeFilter(deployed_at);

6.5 從 Loki 抓事件

Datasource: Loki
Query: {app="ci"} |= "Deployment success"
Title: '{{ service }}'
Tags: ci-deploy

6.6 手動加 annotation

在 graph panel 上 Ctrl + Click（或拖一段時間區間），可以即時加 annotation。 用於 incident 時記錄「這時候開始處理」、「這時候 rollback」等。

§ 7. Dashboard Links 與 Panel Links

7.1 Dashboard Links

顯示在 dashboard 標題附近，跳到其他 dashboard 或外部 URL。

Dashboard settings → Links

類型 1: Dashboards (連到其他 dashboard)
  Tags: ['production', 'service']  ← 自動列出帶這些 tag 的 dashboard

類型 2: Link (外部 URL)
  URL: https://wiki.example.com/runbooks
  Title: Runbook

7.2 Panel Links

每個 panel 可加自己的 links（hover panel title 時顯示）：

Panel → Edit → Links → Add link

Title: View raw metric in Prometheus
URL: https://prometheus/graph?expr={query}&time={time}

支援變數：
${__series.name}          # series 名稱
${__field.name}           # field 名稱
${__value.numeric}        # 數值
${__from}                 # time range 起
${__to}                   # time range 訖
$variable                 # 任何 dashboard variable

7.3 Data Links（最進階）

Panel 內每個 data point 都能有自己的 link。例：點擊 graph 上某個 series → 跳到該 service 的詳細 dashboard。 

Panel → Edit → Data links → Add

Title: View ${__series.labels.service} dashboard
URL: /d/abc?var-service=${__series.labels.service}&from=${__from}&to=${__to}

§ 8. Time Range 與 Refresh

8.1 Time Range 控制

右上角時間選擇器：

  • Quick ranges：Last 5m / 1h / 24h / 7d / 30d
  • Absolute：精確時間範圍
  • Refresh interval：自動 reload（off / 5s / 10s / 30s / 1m / 5m...）
  • Time zone：dashboard / browser / UTC / 自訂

8.2 預設 Time Range

Dashboard settings → Time options：

Default time range: now-6h to now
Default refresh: 30s
Auto refresh intervals: 5s,10s,30s,1m,5m,15m,30s,1h,2h,1d
Now delay: 1m       # 避免 query 「未來」（資料還沒進 Prometheus）

8.3 Now Delay 的重要性

⚠️ Prometheus 有 ingestion delay

Prometheus scrape 後資料到查詢可用有 1-2 分鐘 delay。
若 dashboard 查 now()，最新 1 分鐘的 panel 可能空白。
設 Now delay = 1m 讓所有 query 都「往前 1 分鐘」，避免空白。

8.4 Panel 個別 Time Range

Panel 可以覆蓋 dashboard time range（例：「永遠顯示過去 24h」）：

Panel → Edit → Panel options → Override relative time
Relative time: 24h

適合：「Service Health History」類 panel 不被 dashboard zoom 影響。

§ 9. 版本控管與 JSON 結構

9.1 Dashboard 是 JSON

每個 dashboard 在 backend 都是一個 JSON 結構。Settings → JSON Model 可看 / 編輯：

{
  "uid": "abc123",
  "title": "Service Overview",
  "tags": ["production", "service"],
  "timezone": "browser",
  "schemaVersion": 39,
  "version": 12,
  "refresh": "30s",
  "time": {
    "from": "now-6h",
    "to": "now"
  },
  "templating": {
    "list": [...]      // variables
  },
  "annotations": {
    "list": [...]
  },
  "panels": [
    {
      "id": 1,
      "title": "Request Rate",
      "type": "timeseries",
      "datasource": {...},
      "targets": [...],   // queries
      "fieldConfig": {...},
      "gridPos": {"h": 8, "w": 12, "x": 0, "y": 0}
    },
    ...
  ]
}

9.2 內建版本歷史

Dashboard → Settings → Versions：看每次改動的歷史，可一鍵 revert。

限制：只在 Grafana 內部，跨 Grafana instance 不能用。

9.3 Git 版本控管（推薦）

Production 應該把 dashboard JSON 放 Git：

# Provisioning 從目錄載入
# /etc/grafana/provisioning/dashboards/default.yml
apiVersion: 1
providers:
  - name: 'default'
    folder: ''
    type: file
    options:
      path: /var/lib/grafana/dashboards

# Git repo 結構：
dashboards/
├── production/
│   ├── api-service.json
│   ├── db-service.json
│   └── ...
├── staging/
│   └── ...
└── README.md

9.4 Grafana 12 的 Git Sync（GA）

Grafana 12 把 Git Sync GA。UI 直接連 GitHub repo，dashboard 變動走 PR：

  1. 到 Administration → Git Sync
  2. 連 GitHub（personal access token 或 GitHub App）
  3. 選 repo + branch + path
  4. 之後 dashboard 在 UI 改 → 提 PR → review/merge → 同步回 Grafana

這結合了「UI 編輯方便」與「Git 版本控管」兩個優點。

9.5 Terraform 管理 Dashboard

resource "grafana_dashboard" "api_service" {
  config_json = file("${path.module}/dashboards/api-service.json")
  folder      = grafana_folder.production.id
}

§ 10. RED / USE / Golden Signals 模板

10.1 RED Method（適合 service / API）

Rate / Errors / Duration

# Rate
sum by (service) (rate(http_requests_total[5m]))

# Errors
sum by (service) (rate(http_requests_total{status=~"5.."}[5m]))
  / sum by (service) (rate(http_requests_total[5m]))

# Duration p99
histogram_quantile(0.99,
  sum by (service, le) (rate(http_request_duration_seconds_bucket[5m]))
)

10.2 USE Method（適合 resource / 機器）

Utilization / Saturation / Errors

# CPU Utilization
1 - avg by (instance) (
  rate(node_cpu_seconds_total{mode="idle"}[5m])
)

# CPU Saturation（runqueue / load）
node_load1 / count by (instance) (node_cpu_seconds_total{mode="idle"})

# Errors（disk / network）
rate(node_disk_io_errors_total[5m])
rate(node_network_transmit_errs_total[5m])

10.3 Four Golden Signals（Google SRE）

  • Latency：請求響應時間
  • Traffic：請求量
  • Errors：錯誤率
  • Saturation：資源飽和度

10.4 Service Overview 範例 Dashboard

📊 服務名: $service Overview

[Stat] QPS               [Stat] Error %         [Stat] P99 (ms)        [Stat] Saturation
   1.2K                     0.05%                  234                    47%

[Time series 12wide]            [Time series 12wide]

                                 (latency percentiles overlay)

§ 11. 設計反模式

⚠️ 反模式 1：「萬用 dashboard」

把每個 metric 都放上去 → 有 50+ panel → 沒人看完。
解：分多個 dashboard，每個一個 focus（service / resource / business）。

⚠️ 反模式 2：色彩濫用

每個 panel 都用不同色彩、漸層、3D 效果 → 看起來炫，但難 scan。
解：色彩要有意義（紅=壞、綠=好），其他用中性色。

⚠️ 反模式 3：沒有 threshold 的 stat panel

Stat panel 只顯示數字 234，看不出「這算高還算低」。
解：設 threshold（< 200 綠 / 200-500 黃 / > 500 紅）。

⚠️ 反模式 4：每個 panel 都有自己的 time range

Panel A 看 1h、Panel B 看 24h、Panel C 看 5m → 對齊困難，無法 cross-correlate。
解：除非有特殊原因，所有 panel 用 dashboard time range。

⚠️ 反模式 5：忘記 unit / format

request_duration_seconds 顯示 0.000234，看了無感。
解：設 panel unit = "Time/seconds (s)" 自動轉成 234µs。

⚠️ 反模式 6：「死掉」的 dashboard

建好沒人改、沒人看，metric 名稱已過時、threshold 不對。
解：每季 review、建 owner、淘汰沒在用的 dashboard。
