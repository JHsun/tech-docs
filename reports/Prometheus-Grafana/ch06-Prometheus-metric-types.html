Chapter 06 · Metric Types

Metric Types 完全解析

Counter、Gauge、Histogram、Summary 這四個概念是 Prometheus 應用埋點的核心。 再加上 v2.40 開始的 Native Histogram、v3.x 的 NHCB（Native Histogram with Custom Buckets）， 讓 metric 類型體系變得更豐富也更需要釐清。本章幫您一次搞懂所有類型該用在哪、怎麼用、雷在哪。

§ 1. 為什麼需要區分 metric 類型？

從 Ch 05 我們知道，Prometheus 內部所有 sample 在儲存層其實都是 (timestamp, float64)（native histogram 是後來新增的複合型例外）。 那為什麼還要區分類型？

核心理由有三個：

  1. 不同類型有不同的 PromQL 操作邏輯
    例如：rate() 函式只對 counter 有意義（會自動處理 reset），對 gauge 用會出錯。
  2. 讓使用者知道資料的語意
    看到 http_requests_total 是 counter，您就知道要套 rate()，不能直接看絕對值。
  3. 暴露格式中保留了 # TYPE 行
    讓 Grafana、Alertmanager UI 等工具能正確顯示與處理。
ℹ️ Prometheus 還不是真正的「強型別」

在 v3.x 之前，Prometheus server 內部其實不儲存 type 資訊， 所有的 type 判斷都依賴 metric name 的命名慣例與暴露格式裡的 # TYPE。 Prometheus 3.x 已開始把 type 持久化，但這仍是進行中的工程。 實務上，正確的命名與正確的類型一致很重要，否則 PromQL 會給出誤導的結果。

§ 2. Counter（單調遞增計數器）

2.1 概念

Counter 是一個只會增加（或重啟時歸零）的計數器。 它累積記錄某件事發生的次數或量，例如：

  • HTTP 請求總數 http_requests_total
  • 處理過的 byte 數 process_network_receive_bytes_total
  • 錯誤發生次數 auth_failures_total
  • 處理過的訊息數量 messages_processed_total

2.2 Counter 的兩個鐵律

🚨 Counter 鐵律
  1. 命名必須以 _total 結尾（OpenMetrics 規範）
  2. 絕對值沒有意義，只看「速率（rate）」或「增量（increase）」

為什麼絕對值沒意義？因為 counter 在進程重啟時會歸零。 http_requests_total = 12345 可能是「跑了 5 分鐘，每秒 41 個請求」， 也可能是「跑了 3 小時，每秒 1 個請求」。所以您要看的是變化速率： 

# 過去 5 分鐘的每秒平均請求數
rate(http_requests_total[5m])

# 過去 5 分鐘總共增加了多少（rate × 秒數）
increase(http_requests_total[5m])

2.3 客戶端使用範例（Go）

import "github.com/prometheus/client_golang/prometheus"

var requestsTotal = prometheus.NewCounterVec(
    prometheus.CounterOpts{
        Name: "http_requests_total",
        Help: "Total HTTP requests received.",
    },
    []string{"method", "path", "status"},
)

func init() {
    prometheus.MustRegister(requestsTotal)
}

// 在處理請求時：
requestsTotal.WithLabelValues("POST", "/api/login", "200").Inc()

2.4 Counter 的常見錯誤

⚠️ Anti-pattern：拿 counter 的絕對值畫圖

在 Grafana 看到 http_requests_total 會是個一直往上爬的線——這沒有意義。 請永遠用 rate(http_requests_total[5m])。

§ 3. Gauge（瞬時值）

3.1 概念

Gauge 表示一個可上可下的瞬時值。它代表「現在當下」的狀態， 例如：

  • 當前記憶體用量 node_memory_MemAvailable_bytes
  • 佇列中等待處理的任務數 queue_pending_tasks
  • 當前線上連線數 websocket_active_connections
  • 溫度感測器讀數 device_temperature_celsius

3.2 Gauge 與 Counter 的差別

面向CounterGauge
可否減少不能（除非重啟歸零）可以
名稱後綴_total無強制後綴（常見：_bytes、_count、_size）
常用 PromQLrate()、increase()直接使用值；可用 delta()、deriv()
適合的問題「過去 N 分鐘總共發生了多少」「現在當下是多少」

3.3 客戶端範例（Python）

from prometheus_client import Gauge

queue_size = Gauge('queue_pending_tasks', 'Number of pending tasks')

# 設值
queue_size.set(42)
# 加減
queue_size.inc(5)
queue_size.dec(2)
# 設為當前時間
queue_size.set_to_current_time()

§ 4. Classic Histogram（傳統直方圖）

4.1 概念與用途

Histogram 用來觀測「數值的分布」，最常見的用途是： 請求延遲、回應大小、處理時間。它記錄「有多少觀測值落在每個 bucket 裡」，加上總和與總次數。

4.2 Histogram 在 wire 上的展開

一個名為 http_request_duration_seconds 的 histogram 會在暴露格式中變成多條 series：

# TYPE http_request_duration_seconds histogram
http_request_duration_seconds_bucket{le="0.005"} 24054
http_request_duration_seconds_bucket{le="0.01"}  33444
http_request_duration_seconds_bucket{le="0.025"} 100392
http_request_duration_seconds_bucket{le="0.05"}  129389
http_request_duration_seconds_bucket{le="0.1"}   133988
http_request_duration_seconds_bucket{le="0.25"}  142005
http_request_duration_seconds_bucket{le="0.5"}   143442
http_request_duration_seconds_bucket{le="1"}     144206
http_request_duration_seconds_bucket{le="2.5"}   144309
http_request_duration_seconds_bucket{le="5"}     144319
http_request_duration_seconds_bucket{le="10"}    144320
http_request_duration_seconds_bucket{le="+Inf"}  144320
http_request_duration_seconds_sum                53423
http_request_duration_seconds_count              144320

三大重點：

  • Bucket 是累積（cumulative）：le="0.05" 的值已經包含了 le="0.005" 等所有更小的 bucket
  • 必有 +Inf bucket：等於 _count
  • Bucket 邊界在客戶端建立時就決定，之後改不了（影響後續的 query 精度）

4.3 用 histogram_quantile 算分位數

Histogram 強大之處在於可以聚合再算分位數。例如要算「過去 5 分鐘所有 instance 的 P99 延遲」： 

histogram_quantile(0.99,
  sum by (le) (rate(http_request_duration_seconds_bucket[5m]))
)

步驟拆解：

  1. rate(..._bucket[5m])：每個 bucket 的每秒增量
  2. sum by (le) (...)：把所有 instance 的 bucket 聚合成單一 histogram
  3. histogram_quantile(0.99, ...)：在聚合後的 histogram 上算 P99

4.4 Bucket 設計的取捨

⚠️ Bucket 設計影響整個未來

Bucket 邊界一旦決定就難以修改（會破壞歷史資料的可比性）。 Bucket 太少 → 分位數估算誤差大；bucket 太多 → 高基數問題（每個 bucket 都是一條 series）。 預設值（10 個 bucket）對大多數情境合適。

4.5 Histogram 高基數警告

假設您有 10 個 bucket、4 種 method、20 個 endpoint、50 個 pod，那麼：

10 buckets × 4 methods × 20 endpoints × 50 pods
= 40,000 series（光一個 metric！）

加上 _sum 和 _count，每個 metric 都是 12 倍展開。 所以 histogram 一定要謹慎使用，不要把高基數 label 與 histogram 結合。

§ 5. Summary（摘要 / 客戶端分位數）

5.1 概念

Summary 同樣用來觀測分布，但與 histogram 最大差別是： 分位數（quantile）在客戶端就計算好。 

# TYPE rpc_duration_seconds summary
rpc_duration_seconds{quantile="0.5"}  0.0095
rpc_duration_seconds{quantile="0.9"}  0.012
rpc_duration_seconds{quantile="0.99"} 0.018
rpc_duration_seconds_sum   1234.5
rpc_duration_seconds_count 9876

5.2 Summary vs Histogram 的關鍵差別

面向HistogramSummary
分位數計算地點查詢時，server 端埋點時，client 端
是否可聚合★★★ 可以（across instances）❌ 不可（不同 instance 的 P99 不能直接平均）
分位數可調★★★ 可以（用 PromQL 任意算 P50/P95/P99）❌ 寫死（要改要改程式碼）
客戶端 CPU 成本低高（streaming quantile estimation）
分位數準確度取決於 bucket 設計取決於演算法（誤差由 client 控制）
儲存量每個 bucket 一條 series每個 quantile 一條 series + sum + count
✅ Best Practice：絕大多數情境用 Histogram

官方明確建議：能用 histogram 就用 histogram。Summary 唯一不可取代的優勢 是「對單一 instance 的精確分位數」，但這在分散式系統的需求很少。

§ 6. Native Histogram（原生直方圖）

6.1 為什麼要 Native Histogram？

Classic histogram 有幾個天生痛點：

  • Bucket 邊界寫死，需求改變時很痛苦
  • 每個 bucket 一條 series，高基數爆炸
  • 不同 service 的 bucket 邊界不同就無法跨服務聚合
  • Bucket 太少準度不夠，bucket 太多儲存爆炸

Native Histogram（在 Prometheus v2.40 引入，v3.x 趨於穩定）解決了這些問題： 它是一個複合型 sample，整個 histogram 是一條 series 中的單一複合值。

6.2 核心特性

  • 指數分桶（exponential buckets）：bucket 邊界自動依公比 2^(2^-schema) 產生
  • 動態 schema：bucket 太多時自動降低解析度合併相鄰桶
  • 單一 time series：整個 histogram 用一條 series 表示，徹底解決高基數
  • 自動處理 reset：跨進程重啟仍能正確 rate
  • 跨服務可聚合：所有 native histogram 共享同一套 schema

6.3 啟用 Native Histogram

Native histogram 需要客戶端與伺服器端都支援。

Server 端（Prometheus v2.40 ~ v2.x）：

prometheus --enable-feature=native-histograms

Prometheus 3.x 已預設啟用，無需 feature flag。

Client 端（以 Go 為例）：

import "github.com/prometheus/client_golang/prometheus"

duration := prometheus.NewHistogramVec(
    prometheus.HistogramOpts{
        Name: "http_request_duration_seconds",
        Help: "Request duration in seconds.",
        // 啟用 native histogram
        NativeHistogramBucketFactor:     1.1,
        NativeHistogramMaxBucketNumber:  100,
        NativeHistogramMinResetDuration: time.Hour,
    },
    []string{"method", "path"},
)

目前各語言支援程度：

語言狀態
Go★★★ 完整支援
Java★★★ 完整支援
Python實驗性
Ruby / Rust / .NET / Node.js規劃中或社群實驗

6.4 在 PromQL 中使用

查詢語法相同，histogram_quantile 與 rate 仍可用：

# 與 classic histogram 完全相同的語法
histogram_quantile(0.99, sum(rate(http_request_duration_seconds[5m])))

注意：對 native histogram 不要對 _bucket 做事，因為根本沒有 _bucket 的展開。

§ 7. NHCB：Native Histogram with Custom Buckets

7.1 是什麼？解決什麼問題？

NHCB（schema -53）是 native histogram 的一個變種，它保留 classic histogram 的自訂 bucket 邊界， 但用 native histogram 的儲存結構儲存。

它解決的痛點是：

  1. 您已經有 classic histogram，遷移到 native histogram 會破壞 SLO 的 bucket 邊界
  2. 您需要 OpenTelemetry 的 explicit bucket histogram 在 Prometheus 中的高效儲存
  3. 想吃到 native histogram 的儲存效率（30%+ 壓縮），但保留 bucket 自訂能力

7.2 啟用方式

在 scrape_config 中啟用：

scrape_configs:
  - job_name: 'my-service'
    convert_classic_histograms_to_nhcb: true
    static_configs:
      - targets: ['localhost:8080']

客戶端不需要改變，仍然吐 classic histogram；Prometheus 在抓進來時自動轉成 NHCB。

ℹ️ 三種 histogram 的關係

Classic Histogram：原始的、bucket 為各自 series 展開
Native Histogram (Standard Exponential Buckets, schema 0~+8)：自動指數分桶，最高效
NHCB (schema -53)：保留 classic 的自訂邊界，但用 native 結構儲存，過渡期最佳選擇

§ 10. 常見陷阱與最佳實踐

⚠️ 陷阱 1：Counter 沒加 _total

雖然 Prometheus 不會強制檢查，但 OpenMetrics 規範要求 counter 必須以 _total 結尾。 未來若您把 metrics 送往 OpenMetrics-strict 的後端會被拒絕。

⚠️ 陷阱 2：對 counter 用 Gauge 的方式畫圖

http_requests_total 直接畫圖只會看到一條一直往上爬的曲線。 永遠記得用 rate() 或 increase()。

⚠️ 陷阱 3：Histogram 的 bucket 設計太細

看到別人用 50 個 bucket 別覺得「越細越精確」。每多一個 bucket 就是一條新 series， 乘上其他 label 的維度可能爆炸。預設 10 個 bucket 對 99% 情境足夠。 若需要 fine-grained 分位數，請用 native histogram。

⚠️ 陷阱 4：把 Summary 跨 instance 平均

avg(rpc_duration_seconds{quantile="0.99"}) 數學上是錯的！ P99 不能用平均算（這是統計學的一個經典謬誤）。要嘛改用 histogram，要嘛只看單一 instance。

⚠️ 陷阱 5：負值觀測在 classic histogram / summary 中

classic histogram 與 summary 的 _sum 在「有負值觀測」時可能下降， 被 rate() 誤判為 counter reset，造成數值錯誤。 Native histogram 沒有這個問題（因為它是整體被 rate）。

✅ Best Practices 總結
  1. Counter 加 _total，數值用 rate，不看絕對值
  2. Gauge 直接用值，跨 instance 用 sum / avg
  3. 新專案的延遲分布優先用 native histogram
  4. 從 classic 遷移時用 NHCB 過渡
  5. 單位後綴永遠用基本單位（seconds 不是 milliseconds）
  6. Histogram 的 bucket 數預設即可，需要 fine-grained 用 native
  7. Summary 用很少，遇到都先想能不能改 histogram
