Grafana LGTM+ Stack 生態系全景

Grafana 早已不只是「畫 dashboard 的工具」。Grafana Labs 打造的 LGTM+ Stack （Loki / Grafana / Tempo / Mimir 加上其他成員）已成為一套完整的開源可觀測性平台。 本章帶您鳥瞰整個生態圖，了解每個成員的定位。

§ 1. 從 Grafana 到 LGTM+ Stack 的演進

Grafana 最初（2014 年）是從 Kibana fork 出來的視覺化工具， 主打對 Graphite 的支援。多年來它演進為「能連接任何資料來源」的視覺化平台。 但 Grafana Labs 並未止步於此——他們發現整個可觀測性領域缺乏一套 「設計理念一致、彼此整合良好、且都採用 Prometheus 模型」的後端套件。

於是 Grafana Labs 陸續開源了一系列產品：

年份產品解決的問題
2014Grafana視覺化、儀表板
2018LokiLogs 聚合（與 Prometheus 模型一致）
2020Tempo分散式追蹤後端
2021Cortex（後分裂為 Mimir）Prometheus 長期儲存
2022Grafana Mimir水平擴展的 Prometheus 後端
2022Grafana Pyroscope持續性效能分析（continuous profiling）
2023Grafana Faro前端 RUM（real user monitoring）
2023Grafana BeylaeBPF 自動埋點
2024Grafana AlloyOpenTelemetry Collector + Prometheus pipeline
2021+Grafana k6負載測試（從 Load Impact 收購）

這套組合常被稱為 LGTM Stack（Loki / Grafana / Tempo / Mimir）。 隨著新成員加入，「+」代表 Pyroscope、Faro、Alloy 等周邊產品，故稱 LGTM+ Stack。

2.1 Loki（Logs）

Loki 是「為 Kubernetes 設計的 Prometheus-style log 系統」。 它最大的特色是：只索引 label，不索引 log 內容。

  • 儲存便宜：log 內容直接存在物件儲存（S3/GCS）中
  • label 與 Prometheus 一致：可從 Prometheus alert 直接跳到對應的 logs
  • 查詢語言 LogQL：語法借鑑 PromQL，學習成本低
  • 無需全文索引：適合「我已經知道是哪個服務、哪個 pod 的 log，給我看內容」的場景
# LogQL 範例：找 production 環境 order-service 的 ERROR
{namespace="production", app="order-service"} |= "ERROR"

# 計算 5 分鐘內 ERROR 數量
sum(rate({namespace="production"} |= "ERROR" [5m])) by (app)

2.2 Grafana（視覺化中樞）

Grafana 是整個 Stack 的「介面」。它的核心能力是：

  • Data sources：原生支援 50+ 種資料來源（不只 Prom/Loki/Tempo，還有 SQL、雲端、APM 系統）
  • Dashboards：透過 panels、variables、templating 建立互動式儀表板
  • Explore：對未知資料的探索式查詢介面
  • Alerting：統一管理跨資料來源的告警
  • Drilldown：v11+ 推出，無需建立 dashboard 即可下鑽 Logs / Traces / Metrics / Profiles
  • Plugins：豐富的官方與社群插件生態

2.3 Tempo（Traces）

Tempo 是高吞吐、成本低的分散式追蹤後端。設計理念與 Loki 類似： 「不索引 trace 細節，只透過 trace ID 查詢」。

  • 儲存物件儲存：可保留遠超過 Jaeger 的長期 trace 資料
  • 支援多種 trace 協議：OpenTelemetry（OTLP）、Jaeger、Zipkin
  • TraceQL 查詢語言：強大的 trace 過濾與分析能力
  • Service Graph：自動從 trace 推導服務拓撲圖
  • Metrics generator：從 trace 自動生成 RED metrics
# TraceQL 範例：找延遲超過 1 秒、且狀態為 error 的 trace
{ duration > 1s && status = error }

# 找 checkout service 中的慢 SQL
{ resource.service.name = "checkout" && span.db.statement =~ ".*SELECT.*" && duration > 500ms }

2.4 Mimir（Metrics）

Mimir 是 Prometheus 的水平擴展後端，支援多租戶、PB 級資料量。 完整內容已在 Ch 02 介紹過，這裡僅補充 LGTM 角色定位：

  • 角色：Prometheus 的長期儲存與全域查詢層
  • 原生整合 Grafana：作為 datasource 與 Prometheus 完全相容
  • 支援 alerting rules、recording rules（透過內建 Ruler 元件）
  • 適用情境：跨多個 K8s 叢集、跨地區的統一監控視角

§ 3. 資料採集成員：Alloy、Beyla

3.1 Grafana Alloy

Alloy 是 Grafana 的「全能採集器」，前身是 Grafana Agent。 它把 OpenTelemetry Collector 與 Prometheus pipeline 融合在一起， 一個 binary 可同時處理 metrics、logs、traces 的採集與轉發。

Alloy 的優勢：

  • 取代 Prometheus Agent、Promtail、OpenTelemetry Collector 三個工具
  • 使用 River 設定語言（類似 HCL）
  • 支援動態 reload
  • OpenTelemetry Collector 的 100% 上游相容性

3.2 Grafana Beyla（eBPF 自動埋點）

Beyla 利用 Linux 核心的 eBPF（extended Berkeley Packet Filter） 技術，在不改任何程式碼的情況下自動為 HTTP/gRPC 服務產生 RED metrics 與 traces。

ℹ️ 什麼是 eBPF？

eBPF 是 Linux 核心提供的一個沙盒環境，可以安全地執行使用者空間定義的程式來監控核心事件、 網路封包、系統呼叫等，無需修改核心程式碼或載入核心模組。 Beyla 利用 eBPF 觀察 socket 與 HTTP 流量，自動產生監控資料。

支援的語言：Go、C/C++、Rust、Python、Node.js、Java、Ruby、.NET。

§ 4. 進階觀測成員：Pyroscope、Faro、k6

4.1 Grafana Pyroscope（Continuous Profiling）

Pyroscope 是「持續性 profiling」的後端。它收集應用程式的 CPU、記憶體、goroutine 等 profile 資料，並以火焰圖（flame graph）視覺化，幫您回答：

  • 「為什麼這台機器 CPU 用量這麼高？」
  • 「這次部署後記憶體洩漏的元兇是哪段程式碼？」
  • 「過去 7 天最熱門的函式是哪一個？」

支援 Go、Java、Python、Ruby、.NET、PHP、Rust、Node.js 等語言， 並可整合 eBPF 做系統級 profiling。

4.2 Grafana Faro（Frontend Observability）

Faro 是前端可觀測性 SDK，把瀏覽器中發生的事件 （JavaScript 錯誤、頁面載入時間、使用者操作、Web Vitals）發送到後端。 它與 Loki / Tempo / Mimir 整合，讓您可以看到：

  • 使用者實際體驗到的 LCP、FID、CLS 等 Web Vitals
  • JavaScript 錯誤的完整堆疊追蹤
  • 從前端使用者操作 → 後端服務的完整 trace 鏈

4.3 Grafana k6（Load Testing）

k6 是高效能負載測試工具，使用 JavaScript 撰寫測試腳本。雖然它本身是測試工具， 但與 Grafana 整合後可以：

  • 把測試結果直接存入 Mimir，與正式環境 metrics 同 dashboard 比對
  • 測試期間自動拉取被測系統的真實 metrics 與 traces
  • 用於 chaos engineering 與 SLO 驗證

✅ 本章重點摘要

30 秒回顧
  • LGTM Stack = Loki（logs）/ Grafana（視覺化）/ Tempo（traces）/ Mimir（metrics），加上 Pyroscope/Alloy/Beyla/Faro/k6 構成 LGTM+
  • Loki 與 Tempo 共同特色：只索引 label，不索引內容，儲存便宜
  • Alloy = Prometheus Agent + Promtail + OpenTelemetry Collector 的合併體
  • Beyla 使用 eBPF 無侵入產生 RED metrics 與 traces
  • 三大版本：OSS（自部署免費）/ Enterprise（自部署付費）/ Cloud（SaaS）
  • LGTM+ 的優勢：低儲存成本、原生 Prometheus 整合、開源授權友善
