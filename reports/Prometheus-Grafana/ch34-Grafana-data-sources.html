Chapter 34 · Data Sources 完整使用

Data Sources：Prometheus、Loki、Tempo、SQL、Cloud、Mixed

Data Source 是 Grafana 的「資料入口」。Grafana 內建支援 60+ 種 datasource。 本章涵蓋觀測性三本柱（Prometheus / Loki / Tempo）的完整設定與互連、SQL 與 Cloud datasource、 Mixed datasource 的進階用法、以及讓 metrics ↔ logs ↔ traces 串聯起來的 derived fields。

§ 1. Data Source 全景

1.1 內建 Datasource 分類

類別主要選項
Time SeriesPrometheus、InfluxDB、Graphite、Mimir
LogsLoki、Elasticsearch、Splunk（plugin）
TracesTempo、Jaeger、Zipkin
ProfilingPyroscope
SQLPostgreSQL、MySQL、Microsoft SQL Server
CloudCloudWatch、Azure Monitor、Google Cloud Monitoring
NoSQL / OtherMongoDB（plugin）、Snowflake、Redis、Cassandra
SpecialTestData、JSON API、Infinity（萬用 HTTP）

1.2 Access Mode：Server vs Browser

Mode說明建議
Server (proxy) Grafana backend 代為查詢 ✅ 預設、推薦
Browser 使用者瀏覽器直接連 datasource ❌ 已棄用，CORS / 安全問題多

1.3 兩種設定方式

  • UI：Connections → Data sources → Add data source（適合探索）
  • Provisioning：YAML 檔放在 /etc/grafana/provisioning/datasources/（適合 production）

§ 2. Prometheus Datasource

2.1 完整設定

apiVersion: 1

datasources:
  - name: Prometheus
    type: prometheus
    uid: prometheus-prod                # 重要：給其他 datasource 引用
    access: proxy
    url: http://prometheus:9090
    isDefault: true
    editable: false                     # 鎖住，不讓 UI 改

    jsonData:
      # === HTTP ===
      httpMethod: POST                  # POST 避免大查詢 URL 太長
      timeout: 60                       # 秒

      # === Prometheus 版本（v9+）===
      prometheusType: Prometheus        # Prometheus / Cortex / Mimir / Thanos
      prometheusVersion: 3.11.2

      # === 查詢優化 ===
      timeInterval: '15s'               # 對應 Prom scrape interval
      queryTimeout: '60s'

      # === 快取 ===
      cacheLevel: 'High'                # Low / Medium / High
      incrementalQuerying: true         # v9+ 增量查詢
      incrementalQueryOverlapWindow: '10m'

      # === Alerting integration ===
      manageAlerts: true                # 從 UI 寫 Prometheus rule

      # === Custom HTTP Headers（多租戶用）===
      httpHeaderName1: 'X-Scope-OrgID'
    secureJsonData:
      httpHeaderValue1: 'tenant-1'

2.2 為什麼設 prometheusType

v9+ 引入。讓 Grafana 知道 backend 真實類型，影響：

  • 哪些 PromQL 函式可用
  • 多租戶 header 行為
  • Alerting 寫入端點不同

2.3 多 Prometheus 場景

Production 常見有多個 Prometheus instance：

datasources:
  - name: Prometheus-Production
    type: prometheus
    uid: prom-prod
    url: http://prom-prod:9090
    isDefault: true

  - name: Prometheus-Staging
    type: prometheus
    uid: prom-staging
    url: http://prom-staging:9090

  - name: Prometheus-Edge-TW
    type: prometheus
    uid: prom-edge-tw
    url: http://prom-edge-tw:9090

2.5 Exemplar 設定

讓 Prometheus metric 點出 trace ID：

jsonData:
  exemplarTraceIdDestinations:
    - name: traceID
      datasourceUid: tempo-prod         # 連到 Tempo datasource
      url: 'https://grafana.example.com/explore?...'

Panel 中啟用 exemplar 後，histogram metric 會顯示小圓點，點選跳到對應 trace。

§ 3. Loki Datasource

3.1 完整設定

- name: Loki
  type: loki
  uid: loki-prod
  access: proxy
  url: http://loki:3100

  jsonData:
    timeout: 60
    maxLines: 1000                      # 單次最多回傳行數
    maxDataPoints: 10000

    # 多租戶
    httpHeaderName1: 'X-Scope-OrgID'

    # === Derived Fields（重要！）===
    derivedFields:
      - datasourceUid: tempo-prod       # 跳到的 datasource UID
        matcherRegex: 'traceID=(\w+)'   # 從 log 內容抓 trace ID
        name: TraceID
        url: '$${__value.raw}'
        urlDisplayLabel: 'View Trace'

      - matcherRegex: 'order_id=(\w+)'
        name: OrderID
        url: 'https://internal.example.com/orders/$${__value.raw}'
        urlDisplayLabel: 'View Order'

  secureJsonData:
    httpHeaderValue1: 'tenant-1'

3.2 LogQL 簡介

Loki 用 LogQL，跟 PromQL 像但是查 log：

# 按 label 篩選
{namespace="production", app="api"}

# Stream 內 grep
{namespace="production"} |= "error"

# JSON 解析
{app="api"} | json | level="error"

# Logfmt 解析
{app="api"} | logfmt | duration > 1s

# 計算 rate（變成 metric）
sum by (level) (
  rate({namespace="production"}[5m])
)

# 從 log 中萃取 metric
sum by (status) (
  rate({app="api"} |= "GET" | logfmt | __error__="" [5m])
)

3.3 Derived Fields 進階

Derived fields 能從 log 行內容自動提取出可點擊的連結。常見模式：

derivedFields:
  # OpenTelemetry trace ID 格式
  - matcherRegex: 'trace_id=([a-f0-9]+)'
    name: TraceID
    datasourceUid: tempo-prod
    url: '$${__value.raw}'

  # JSON log 中的 trace_id
  - matcherRegex: '"trace_id":"([a-f0-9]+)"'
    name: TraceID
    datasourceUid: tempo-prod
    url: '$${__value.raw}'

  # 連到外部系統
  - matcherRegex: 'order_id=(\w+)'
    name: Order
    url: 'https://admin.example.com/orders/$${__value.raw}'
    urlDisplayLabel: 'Order detail'

§ 4. Tempo Datasource

Tempo 是 Grafana Labs 的分散式 trace backend，跟 Jaeger / Zipkin 同類但更便宜（只需 object storage）。

4.1 完整設定

- name: Tempo
  type: tempo
  uid: tempo-prod
  access: proxy
  url: http://tempo:3200

  jsonData:
    httpMethod: GET

    # === Trace to Logs（從 trace 跳到 logs）===
    tracesToLogsV2:
      datasourceUid: loki-prod
      tags:
        - { key: 'service.name', value: 'service' }
        - { key: 'job' }
        - { key: 'instance' }
      filterByTraceID: true
      filterBySpanID: false
      spanStartTimeShift: '-1m'
      spanEndTimeShift: '1m'
      customQuery: true
      query: '{service="$${__span.tags.service}"} |= "$${__trace.traceId}"'

    # === Trace to Metrics（從 trace 跳到 metrics）===
    tracesToMetrics:
      datasourceUid: prometheus-prod
      tags:
        - { key: 'service.name', value: 'service' }
      spanStartTimeShift: '-1m'
      spanEndTimeShift: '1m'
      queries:
        - name: 'Request rate'
          query: 'sum(rate(traces_spanmetrics_calls_total{$$__tags}[5m]))'
        - name: 'Error rate'
          query: 'sum(rate(traces_spanmetrics_calls_total{$$__tags, status_code="STATUS_CODE_ERROR"}[5m]))'

    # === Trace to Profiles（v10+, 連到 Pyroscope）===
    tracesToProfiles:
      datasourceUid: pyroscope-prod
      tags:
        - { key: 'service.name', value: 'service_name' }
      profileTypeId: 'process_cpu:cpu:nanoseconds:cpu:nanoseconds'

    # === Service Graph（用 Prom metric 畫服務拓樸）===
    serviceMap:
      datasourceUid: prometheus-prod

    # === Node Graph（顯示 trace 的 service 節點圖）===
    nodeGraph:
      enabled: true

    # === Search 設定 ===
    search:
      hide: false

    # === Span bar 顯示 ===
    spanBar:
      type: 'Duration'

4.2 TraceQL 查詢

Tempo 用 TraceQL（v2+）查 trace，類似 PromQL/LogQL 風格：

# 找特定 service 慢的 span
{ resource.service.name = "api" && duration > 500ms }

# 錯誤 trace
{ status = error }

# 多條件
{ resource.service.name = "checkout" && duration > 1s && status = error }

# 用 attribute 篩選
{ span.http.status_code = 500 && resource.cluster = "production" }

4.3 Service Graph 視覺化

Tempo 自動從 trace 計算 RED metric（Rate / Error / Duration），寫回 Prometheus， Grafana 可畫出服務拓樸圖。不需任何手動設定，只要 Tempo 開了 metrics generator。

§ 5. 三本柱串聯：Correlations

5.1 為什麼串聯重要

故障排查的標準流程：

graph LR METRIC[Prometheus metric
error rate 飆升] EXEMPLAR[點選 exemplar] TRACE[Tempo trace
看哪個 service 慢] SPANLINK[從 span 跳] LOG[Loki log
看實際 error 訊息] PROFILE[Pyroscope profile
看 CPU 哪邊燒] METRIC --> EXEMPLAR EXEMPLAR --> TRACE TRACE --> SPANLINK SPANLINK --> LOG TRACE --> PROFILE

5.2 完整串聯設定矩陣

From → To機制設定位置
Metric → TraceExemplarPrometheus datasource exemplarTraceIdDestinations
Log → TraceDerived FieldsLoki datasource derivedFields
Trace → LogTrace to Logs V2Tempo datasource tracesToLogsV2
Trace → MetricTrace to MetricsTempo datasource tracesToMetrics
Trace → ProfileTrace to ProfilesTempo datasource tracesToProfiles
任意 → 任意Correlations（v10+）Connections → Correlations

5.3 Correlations（v10+ 萬用機制）

Correlations 是新一代「跨 datasource 跳轉」機制，不限定 trace ↔ log。 可定義「點選 metric 跳 dashboard」、「點選 log 開外部 ticket 系統」等任意連結。 

correlations:
  - sourceUid: prometheus-prod
    targetUid: loki-prod
    label: "View related logs"
    description: "Jump to logs for this service"
    config:
      type: query
      target:
        expr: '{service="$${service}"}'
      field: 'service'

§ 6. SQL Datasources（PG、MySQL）

SQL datasource 讓您查業務資料庫、orders、users 等，跟 metric 並列。

6.1 PostgreSQL 範例

- name: PostgreSQL-Analytics
  type: postgres
  uid: pg-analytics
  url: postgres-readonly.internal:5432
  user: grafana_reader
  jsonData:
    database: analytics
    sslmode: require
    maxOpenConns: 10
    maxIdleConns: 2
    connMaxLifetime: 14400
    postgresVersion: 1500              # 15.0+
    timescaledb: true                  # 若用 TimescaleDB
  secureJsonData:
    password: $__file{/etc/grafana/secrets/pg-pass}

6.2 在 Panel 裡寫 SQL

-- 查每小時訂單數（用 $__timeFilter macro）
SELECT
  $__timeGroup(created_at, '1h') AS time,
  COUNT(*) AS order_count
FROM orders
WHERE $__timeFilter(created_at)
GROUP BY 1
ORDER BY 1;

-- 用 $__interval（自動配合 dashboard time range）
SELECT
  $__timeGroupAlias(created_at, $__interval),
  AVG(amount) AS avg_amount
FROM orders
WHERE $__timeFilter(created_at)
  AND status = 'completed'
GROUP BY 1;

6.3 SQL 重要 macros

Macro展開為
$__timeFilter(col)col BETWEEN '...' AND '...'
$__timeGroup(col, '1h')把時間分組到 1 小時 bucket
$__timeGroupAlias(col, $__interval)同上 + alias 為 time
$__intervalGrafana 計算的最佳 interval
$__timeFrom() / $__timeTo()當前 time range 的起 / 訖

6.4 MySQL

- name: MySQL-Orders
  type: mysql
  url: mysql-readonly.internal:3306
  user: grafana_reader
  jsonData:
    database: orders_db
    maxOpenConns: 10
    timeInterval: '1m'
  secureJsonData:
    password: $__file{/etc/grafana/secrets/mysql-pass}

6.5 SQL Datasource 安全建議

⚠️ 給 Grafana 唯讀帳號

Grafana 的 SQL datasource 預設讓 user 寫任意 query。絕對用唯讀 DB user： 

-- PostgreSQL
CREATE USER grafana_reader WITH PASSWORD '...';
GRANT CONNECT ON DATABASE analytics TO grafana_reader;
GRANT USAGE ON SCHEMA public TO grafana_reader;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO grafana_reader;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT ON TABLES TO grafana_reader;

§ 7. Cloud Datasources

7.1 AWS CloudWatch

- name: CloudWatch
  type: cloudwatch
  jsonData:
    authType: default                  # default / keys / credentials / arn
    defaultRegion: us-east-1
    customMetricsNamespaces: 'AWS/RDS,AWS/ELB,AWS/Lambda'
    assumeRoleArn: 'arn:aws:iam::123456789:role/grafana-cloudwatch'

    # === CloudWatch Logs ===
    logGroups:
      - 'production/api'
      - 'production/db'

    # === Trace to logs ===
    tracingDatasourceUid: 'xray-uid'   # 連 X-Ray trace

三種驗證方式：

  • default：用 Grafana pod 的 IAM Role（K8s 推薦）
  • keys：固定 AccessKey + SecretKey（避免）
  • arn：assume role（跨帳號 / 細粒度）

7.2 GCP Cloud Monitoring

- name: Stackdriver
  type: stackdriver
  jsonData:
    authenticationType: gce            # gce / jwt
    defaultProject: my-gcp-project
  secureJsonData:
    privateKeyPath: /etc/grafana/gcp-sa.json

7.3 Azure Monitor

- name: AzureMonitor
  type: grafana-azure-monitor-datasource
  jsonData:
    azureAuthType: msi                 # msi / clientsecret / workloadidentity
    cloudName: azuremonitor
    subscriptionId: '...'
  secureJsonData:
    clientSecret: '...'

7.4 多雲場景

大公司常同時用 3 雲。Grafana 可以同個 dashboard 並列三雲 metric：

Panel 1: AWS RDS CPU      → CloudWatch datasource
Panel 2: GCP CloudSQL CPU → Stackdriver datasource
Panel 3: Azure DB CPU     → Azure Monitor datasource

或用 Mixed datasource 一個 panel 顯示三家

§ 8. TestData / JSON / Infinity

8.1 TestData（內建）

Grafana 內建的「假資料」datasource，學習與展示必用：

- name: TestData
  type: testdata
  isDefault: false

提供：random walks、CSV content、CSV file、Live、predictable pulse 等資料生成器。 寫範例 dashboard 時用 TestData 不用真實 backend。

8.2 JSON API datasource（plugin）

查任何回傳 JSON 的 HTTP API：

# 安裝
grafana-cli plugins install marcusolsson-json-datasource

# Datasource
- name: JSON-API
  type: marcusolsson-json-datasource
  url: https://api.example.com

Panel 寫 JSONPath：

// API 回傳：
// { "metrics": [{"time": 1234567890, "value": 100}, ...] }

// JSONPath: $.metrics[*]
// Time field: $.time (UNIX MS)
// Value field: $.value (number)

8.3 Infinity datasource（萬用）

更強大的 API datasource，支援 JSON、CSV、XML、HTML、GraphQL：

grafana-cli plugins install yesoreyeram-infinity-datasource

適合：

  • 查 GitHub API（PR、issue）
  • 查內部 SaaS API（無 Prometheus exporter 的）
  • 從靜態 CSV / Google Sheets 取資料
  • GraphQL endpoint

§ 9. Mixed Datasource

9.1 Mixed 是什麼

內建特殊 datasource，讓單一 panel 同時查多個不同 datasource。 每個 query 可以指定不同 datasource。

9.2 範例：在同 panel 比較生產 / 預發

Panel 設 datasource = Mixed，然後加兩個 query：

Query A: datasource = Prometheus-Production
  expr: rate(http_requests_total[5m])
  legend: production

Query B: datasource = Prometheus-Staging
  expr: rate(http_requests_total[5m])
  legend: staging

9.3 用例

  • 多 region / cluster 對比（一張圖看所有 region）
  • 多雲對比（AWS + GCP + Azure 同 metric）
  • Metric + Annotation（從 PG 讀 deployment 事件 + 從 Prom 讀 metric）
  • 新舊系統對比（遷移中）

9.4 限制

  • Alerting 不能用 Mixed datasource（每個 alert 限定一個 datasource）
  • 有些 transformation 在 Mixed 下行為不同
  • 性能：Grafana 並行查所有 datasource，最慢者決定總時間

§ 10. Provisioning 完整範例

10.1 一份完整的 datasources.yaml

# /etc/grafana/provisioning/datasources/all.yaml
apiVersion: 1

datasources:
  # === Metrics ===
  - name: Prometheus
    type: prometheus
    uid: prometheus-prod
    access: proxy
    url: http://prometheus.monitoring.svc.cluster.local:9090
    isDefault: true
    editable: false
    jsonData:
      httpMethod: POST
      manageAlerts: true
      prometheusType: Prometheus
      prometheusVersion: 3.11.2
      cacheLevel: High
      incrementalQuerying: true
      timeInterval: 15s
      exemplarTraceIdDestinations:
        - name: traceID
          datasourceUid: tempo-prod

  # === Logs ===
  - name: Loki
    type: loki
    uid: loki-prod
    access: proxy
    url: http://loki.monitoring.svc.cluster.local:3100
    editable: false
    jsonData:
      maxLines: 1000
      derivedFields:
        - datasourceUid: tempo-prod
          matcherRegex: 'trace_id=(\w+)'
          name: TraceID
          url: '$${__value.raw}'

  # === Traces ===
  - name: Tempo
    type: tempo
    uid: tempo-prod
    access: proxy
    url: http://tempo.monitoring.svc.cluster.local:3200
    editable: false
    jsonData:
      tracesToLogsV2:
        datasourceUid: loki-prod
        tags:
          - { key: 'service.name', value: 'service' }
        spanStartTimeShift: '-1m'
        spanEndTimeShift: '1m'
      tracesToMetrics:
        datasourceUid: prometheus-prod
        tags:
          - { key: 'service.name', value: 'service' }
      serviceMap:
        datasourceUid: prometheus-prod
      nodeGraph:
        enabled: true

  # === SQL ===
  - name: PostgreSQL-Analytics
    type: postgres
    uid: pg-analytics
    url: postgres-readonly:5432
    user: grafana_reader
    editable: false
    jsonData:
      database: analytics
      sslmode: require
      postgresVersion: 1500
      maxOpenConns: 10
    secureJsonData:
      password: $__file{/etc/grafana/secrets/pg-password}

  # === Cloud ===
  - name: CloudWatch
    type: cloudwatch
    uid: cw-prod
    editable: false
    jsonData:
      authType: default
      defaultRegion: us-east-1

  # === Special ===
  - name: TestData
    type: testdata
    uid: testdata
    editable: false

§ 11. 常見問題

⚠️ 問題 1：Datasource 在 UI 出現但 query 失敗

症狀：Save & Test 通過，但 panel 顯示 No data 或 timeout。
檢查：
• 網路：Grafana 容器/pod 能 ping 到 datasource 嗎
• URL：用 service FQDN，不是 localhost
• Auth：API key / basic auth 對嗎
• Time range：dashboard 時間範圍內有資料嗎

⚠️ 問題 2：UID 衝突

Provisioning 時 uid 重複，後者覆蓋前者。所有 datasource UID 必須全 org 唯一。

⚠️ 問題 3：Secret 在 UI 顯示為「configured」但不對

Provisioning 時 secureJsonData 改了，但 UI 看到的還是舊的。
解法：刪掉 datasource → Provisioning 重建。或用 API force update。

⚠️ 問題 4：Multi-tenant header 沒生效

Mimir/Cortex 多租戶 setup，X-Scope-OrgID 設了但 query 失敗。
檢查：header 設在 jsonData.httpHeaderName1 + secureJsonData.httpHeaderValue1， 編號從 1 開始，不是 0。

⚠️ 問題 5：Trace to Logs 沒跳轉

Tempo trace span 上沒有「Logs for this span」按鈕。
檢查：
• Tempo datasource 的 tracesToLogsV2 設了 datasourceUid 嗎
• 該 UID 的 datasource 存在嗎（拼錯字常見）
• tags 中的 key 在 trace span 裡有嗎

✅ 本章重點摘要

30 秒回顧
  • Grafana 內建 60+ datasource，分 8 類（time series / logs / traces / SQL / cloud...）
  • Access mode：Server (proxy) 是預設與推薦，Browser 已棄用
  • Provisioning 是 production 最佳做法（YAML 在 /etc/grafana/provisioning/datasources/）
  • Prometheus 必設 uid（給其他 datasource 引用）、prometheusType、POST httpMethod
  • Loki 用 derived fields 從 log 抓 trace_id 變連結
  • Tempo 三大整合：tracesToLogsV2 / tracesToMetrics / tracesToProfiles + serviceMap + nodeGraph
  • 三本柱串聯矩陣：metric → trace 用 exemplar；log → trace 用 derived fields；trace → log/metric/profile 用 Tempo jsonData
  • Correlations（v10+）是萬用跨 datasource 跳轉機制
  • SQL 必用唯讀 DB user，用 $__timeFilter / $__timeGroup macros
  • Cloud datasource 推薦用 IAM Role（AWS）/ MSI（Azure），避免 hard-code key
  • Mixed datasource 在單 panel 查多 datasource，但 alerting 不支援
  • TestData 是學習與 demo 必用
