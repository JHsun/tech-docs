Chapter 10 · Relabeling

Relabeling 深度解析

Relabeling 是 Prometheus 最強大也最讓新手困惑的功能。它能在 scrape 前後改寫 label、過濾 target、 甚至完全改變 metric 樣貌。學會 relabeling 等於拿到 Prometheus 的瑞士刀。 本章把所有 actions 一次講清楚，附上實戰範例與 debug 流程。

§ 1. Relabeling 在哪裡發生？

Relabeling 在 Prometheus 的資料流中有四個不同的位置。理解這四個位置的差異， 是學好 relabeling 的關鍵。

graph TB SD[Service Discovery
__meta_* labels] R1[relabel_configs
1. 過濾 / 改寫 target] SCRAPE[HTTP GET /metrics] R2[metric_relabel_configs
2. 過濾 / 改寫 sample 的 label] TSDB[(TSDB
本地儲存)] R3[write_relabel_configs
3. 過濾要送往 remote_write 的資料] RW[Remote Write 後端] RULES[Alerting Rules] R4[alert_relabel_configs
4. 改寫送往 Alertmanager 的 alert] AM[Alertmanager] SD --> R1 R1 --> SCRAPE SCRAPE --> R2 R2 --> TSDB TSDB --> R3 R3 --> RW TSDB --> RULES RULES --> R4 R4 --> AM
位置處理對象典型用途
relabel_configs Target（含 meta labels） 過濾 target、修改 __address__、把 meta 升格成正式 label
metric_relabel_configs Scrape 進來的每個 sample 丟掉特定 metric、清理 high-cardinality label
write_relabel_configs 送往 remote_write 的 sample 選擇性外送（本地保留、後端只收子集）
alert_relabel_configs 送往 Alertmanager 的 alert 移除 replica label、加注 cluster 等

§ 2. 一條 relabel rule 的解剖

每條 rule 是一個 YAML 物件，最完整的形式：

- source_labels: [, ...]   # 要讀的 label（可多個，會用分隔符串接）
  separator: ;                          # 串接分隔符，預設 ";"
  regex: ''                       # 正則表達式，預設 ".*"
  modulus:                         # hash modulus（用於 hashmod action）
  target_label:              # 要寫入的 label 名（部分 action 用）
  replacement: ''                 # 要寫入的值（可參考 capture group $1）
  action:                  # 預設 "replace"

2.1 流程：每條 rule 怎麼跑？

  1. 讀取 source_labels 列出的 label 值
  2. 用 separator（預設 ;）串接成一個字串
  3. 把這個字串去比對 regex
  4. 根據 action 決定接下來怎麼處理
  5. 若觸發了 keep/drop，target/sample 就被丟掉，後面的 rule 不會跑

2.2 為什麼要 source_labels 是陣列？

因為很多時候你需要組合多個 label 才能做出判斷。例如：

# 取出 __address__ 的 host 和 annotation 中的 port，組合成新 __address__
- source_labels: [__address__, __meta_kubernetes_pod_annotation_prometheus_io_port]
  regex: '([^:]+)(?::\d+)?;(\d+)'
  replacement: '$1:$2'
  target_label: __address__

這條規則的作用是：把 __address__（例如 10.0.0.1:9090） 和 annotation 上的自訂 port（例如 8080）組合， 最終產生 10.0.0.1:8080 寫入 __address__。

§ 3. 所有 8 種 actions 詳解

3.1 replace（預設）

用 regex 比對 source_labels，把符合的部分用 replacement 替換後寫入 target_label。

# 從 __address__ 抽出 hostname，寫入 instance label
- source_labels: [__address__]
  regex: '([^:]+)(?::\d+)?'
  replacement: '$1'
  target_label: instance
  action: replace      # 可省略

關鍵行為：

  • 若 regex 不匹配，這條 rule 不做任何事（不是錯誤）
  • 若 regex 匹配，target_label 會被設成 replacement 的值（含 capture group $1, $2...）
  • replacement 預設是 $1
  • regex 預設是 .*（永遠匹配整個字串）

3.2 keep

只保留 regex 匹配的 target / sample，其他丟掉。

# 只保留 annotation prometheus.io/scrape="true" 的 pod
- source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
  regex: 'true'
  action: keep

3.3 drop

跟 keep 相反，丟掉匹配的 target / sample。

# 丟掉所有 go_* 與 prometheus_* metrics
- source_labels: [__name__]
  regex: '(go_|prometheus_).*'
  action: drop

3.4 keepequal 與 dropequal（v2.41+）

不用 regex，而是直接比較兩個 label 是否相等：

# 只保留 instance == node 的 target（K8s endpoint role 常見）
- source_labels: [__meta_kubernetes_pod_node_name]
  target_label: __tmp_node
- source_labels: [instance, __tmp_node]
  action: keepequal     # 兩值相等才保留
  target_label: instance

注意：keepequal / dropequal 比 regex 高效，因為不用編譯正則。 用法：比較 source_labels 串接後的值 vs target_label 的值。

3.5 labelmap

批次重新命名 label。常用於把所有 K8s pod label 一次升格：

# __meta_kubernetes_pod_label_app  -->  app
# __meta_kubernetes_pod_label_team -->  team
# __meta_kubernetes_pod_label_xxx  -->  xxx
- regex: '__meta_kubernetes_pod_label_(.+)'
  action: labelmap

用 capture group 取得新 label 名稱。要小心：所有匹配的 label 都會被批次處理。

3.6 labeldrop

移除符合 regex 的 label：

# 移除所有 __ 開頭的「假 meta」label（這些其實在 scrape 後就會被移除，舉例用）
- regex: 'pod_template_hash|controller-revision-hash'
  action: labeldrop

3.7 labelkeep

只保留符合 regex 的 label，其他全砍：

# 只保留 __ 開頭與 instance / job 三類，其他都砍
- regex: '__.*|instance|job'
  action: labelkeep
⚠️ labelkeep 的危險性

labelkeep 用錯會把必要的內部 label 也砍掉，例如 __address__， 導致 target 無法被抓。建議優先用 labeldrop 點名移除特定 label。

3.8 hashmod

對 source_labels 的值做 hash，取餘數寫入 target_label。常用於水平分片：

# 把 1000 個 target 分到 4 個 Prometheus 各抓 250 個
- source_labels: [__address__]
  modulus: 4
  target_label: __tmp_hash
  action: hashmod
- source_labels: [__tmp_hash]
  regex: '0'              # 這個 Prometheus 只抓 hash%4==0 的 target
  action: keep

用法：先用 hashmod 算出 0-3 的數字寫入暫存 label，再用 keep 過濾。

3.9 uppercase / lowercase（v2.36+）

# 把 environment label 全轉小寫
- source_labels: [__meta_consul_tag_env]
  target_label: environment
  action: lowercase

3.10 完整 action 對照表

Action用途需要欄位
replace寫入 / 改寫 label（最常用）source_labels, target_label, replacement, regex
keep只保留匹配的source_labels, regex
drop丟掉匹配的source_labels, regex
keepequal兩 label 相等才保留source_labels, target_label
dropequal兩 label 相等就丟source_labels, target_label
labelmap批次改名regex
labeldrop移除符合的 labelregex
labelkeep只保留符合的 labelregex
hashmod分片用source_labels, modulus, target_label
uppercase / lowercase大小寫轉換source_labels, target_label

§ 4. relabel_configs vs metric_relabel_configs

兩者語法相同，但作用對象與時機不同：

面向relabel_configsmetric_relabel_configs
時機Scrape 之前Scrape 之後
作用對象每個 target（一次）scrape 進來的每個 sample（多次）
可改寫__address__、__metrics_path__、__scheme__ 等內部 label無法改寫上述（已 scrape 完）
能用的 label所有 SD meta label（__meta_*）scrape 進來的 metric label（無 meta）
典型用途選擇要抓誰、改寫抓取地址過濾 metric、清理高基數 label

4.1 一個錯誤示範

🚨 常見誤用

很多新手把這條規則放錯位置：

- source_labels: [__meta_kubernetes_pod_label_app]
  target_label: app
  action: replace

這條規則用了 __meta_kubernetes_*，必須放在 relabel_configs。 若放在 metric_relabel_configs 會完全沒效果（因為 scrape 後 meta label 已被清除）。

§ 5. 實戰範例：K8s 環境

5.1 抓取所有有 annotation 的 Pod 的完整範例

scrape_configs:
  - job_name: 'kubernetes-pods'
    kubernetes_sd_configs:
      - role: pod

    relabel_configs:
      # 1. 只保留 annotation prometheus.io/scrape=true
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: true

      # 2. 自訂 metrics_path（從 annotation）
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_path]
        action: replace
        target_label: __metrics_path__
        regex: (.+)

      # 3. 自訂 port（從 annotation 覆寫）
      - source_labels: [__address__, __meta_kubernetes_pod_annotation_prometheus_io_port]
        action: replace
        regex: ([^:]+)(?::\d+)?;(\d+)
        replacement: $1:$2
        target_label: __address__

      # 4. 把所有 K8s pod label 一次升格
      - regex: '__meta_kubernetes_pod_label_(.+)'
        action: labelmap

      # 5. 加入 namespace label
      - source_labels: [__meta_kubernetes_namespace]
        action: replace
        target_label: kubernetes_namespace

      # 6. 加入 pod 名稱
      - source_labels: [__meta_kubernetes_pod_name]
        action: replace
        target_label: kubernetes_pod_name

      # 7. 加入 node 名稱
      - source_labels: [__meta_kubernetes_pod_node_name]
        action: replace
        target_label: kubernetes_node

    metric_relabel_configs:
      # 8. 丟掉 cAdvisor 的高基數 metric
      - source_labels: [__name__]
        regex: 'container_tasks_state|container_memory_failures_total'
        action: drop

      # 9. 移除特定 high-cardinality label
      - regex: 'pod_template_hash|controller_revision_hash'
        action: labeldrop

5.2 對應的 Pod annotation 範例

apiVersion: v1
kind: Pod
metadata:
  name: my-app
  annotations:
    prometheus.io/scrape: "true"
    prometheus.io/port: "8080"
    prometheus.io/path: "/custom-metrics"
  labels:
    app: my-app
    team: platform

§ 6. 實戰範例：高基數消除

當您發現某個 metric 拖垮 Prometheus 時，metric_relabel_configs 是急救工具：

6.1 找出兇手

# 在 Prometheus UI 跑這段，找出 series 數最多的 metric
topk(20, count by (__name__)({__name__=~".+"}))

# 找出某 metric 的高基數 label
topk(20, count by (path) (http_requests_total))

6.2 緊急止血策略

scrape_configs:
  - job_name: 'misbehaving-app'
    static_configs:
      - targets: ['app:9100']

    metric_relabel_configs:
      # 策略 1：丟掉整個高基數 metric
      - source_labels: [__name__]
        regex: 'http_request_duration_microseconds_(bucket|sum|count)'
        action: drop

      # 策略 2：保留 metric，但砍掉問題 label
      - regex: 'user_id|request_id|session_id'
        action: labeldrop

      # 策略 3：把 high-cardinality URL 正規化（/users/123 -> /users/:id）
      - source_labels: [path]
        regex: '/users/\d+'
        replacement: '/users/:id'
        target_label: path
      - source_labels: [path]
        regex: '/orders/[a-f0-9-]+'
        replacement: '/orders/:uuid'
        target_label: path

§ 7. write_relabel_configs 與 alert_relabel_configs

7.1 write_relabel_configs：選擇性外送

典型情境：本地保留所有資料，但只把關鍵 metric 送到遠端長期儲存（節省成本）：

remote_write:
  - url: 'https://mimir.example.com/api/v1/push'
    write_relabel_configs:
      # 不送 Go runtime metrics（本地能查就好）
      - source_labels: [__name__]
        regex: 'go_(gc|memstats|goroutines).*'
        action: drop

      # 不送 process metrics
      - source_labels: [__name__]
        regex: 'process_.*'
        action: drop

      # 移除高基數 label 後再送
      - regex: 'pod_template_hash'
        action: labeldrop

7.2 alert_relabel_configs：改寫 alert

HA 部署時最常見的需求 — 移除 replica label 讓 Alertmanager 能去重：

alerting:
  alert_relabel_configs:
    - regex: 'replica'
      action: labeldrop

  alertmanagers:
    - static_configs:
        - targets: ['alertmanager-1:9093', 'alertmanager-2:9093']

另一個常見用法 — 強制加上 cluster label：

alerting:
  alert_relabel_configs:
    - target_label: cluster
      replacement: 'production-tw'
      action: replace

§ 8. Debug 與測試方法

8.1 用 Web UI 看 target 的 labels

瀏覽 http://prometheus:9090/targets，點任一 target 旁的「Show more」，可看到：

  • Discovered Labels：SD 給的所有 meta labels
  • Target Labels：經過 relabel_configs 後保留下來的 label

這是最直接的 debug 方法。看不到預期的 label？對照兩欄就知道哪一條 rule 出問題。

8.2 用 promtool 驗證 config

promtool check config /etc/prometheus/prometheus.yml

8.3 用 Service Discovery API 看原始 meta

curl http://localhost:9090/api/v1/targets/metadata
curl http://localhost:9090/api/v1/targets | jq

8.4 增加暫存 label 觀察中間狀態

當 relabel 出錯找不到原因時，把中間結果寫到一個暫存 label 觀察：

relabel_configs:
  # 把組合後的字串保存到 __tmp_debug 看看
  - source_labels: [__address__, __meta_kubernetes_pod_annotation_prometheus_io_port]
    separator: ';'
    target_label: __tmp_debug
    action: replace

  # 之後實際的 rule
  - source_labels: [__address__, __meta_kubernetes_pod_annotation_prometheus_io_port]
    regex: ([^:]+)(?::\d+)?;(\d+)
    replacement: $1:$2
    target_label: __address__

在 /targets 頁面就能看到 __tmp_debug 的值，確認 separator 和 source_labels 的組合是否符合預期。 Debug 完記得移除暫存 label（或用 labeldrop）。

8.5 Relabel testing 工具

社群有些線上工具可以測試 relabel 規則：

  • promtool tsdb dump：把已存的 series 倒出來看 labels
  • relabeler（社群工具）：給定 input labels 與 rules，模擬輸出
  • Grafana Cloud Relabeler：圖形化的 relabel 試算

§ 9. 常見陷阱與最佳實踐

⚠️ 陷阱 1：忘記 regex 是「整體匹配」

Prometheus 的 regex 是整體錨定的（implicitly anchored）， 也就是 regex: 'foo' 等同於 ^foo$。
要部分匹配，請寫 .*foo.*。

⚠️ 陷阱 2：rule 的順序很重要

rules 是按順序執行的。一個 keep / drop 把 target 過濾掉後，後面的 rule 都不會跑。
建議順序：過濾 → 改寫 address/path → 升格 meta label → 微調 → metric 層級過濾。

⚠️ 陷阱 3：target_label 名稱有特殊字元

target_label 必須符合 [a-zA-Z_][a-zA-Z0-9_]*。 若您從 K8s annotation 動態提取 label 名（用 capture group），結果可能含 . 或 -，會被 Prometheus reject。
解法：先用 replace 把特殊字元換成 _。

⚠️ 陷阱 4：把 metric_relabel_configs 當 relabel_configs 用

您不能在 metric_relabel_configs 裡引用 __meta_* 或改寫 __address__。 Scrape 完成後這些都已不存在 / 已凍結。

⚠️ 陷阱 5：忘記內部 label 必須以 __ 開頭

像 __address__、__metrics_path__、__scheme__ 等是內部 label， 它們在 scrape 時被使用，scrape 完成後自動移除。
若您寫 target_label: address（沒有底線），它會變成正式 label，且不會影響 scrape 行為。

✅ Best Practices 總結
  1. 先過濾，再改寫：keep / drop 放在前面減少後續處理量
  2. 用 labelmap 批次升格 K8s label：比一條一條 replace 簡潔
  3. 暫存 label 用 __tmp_ 前綴：方便辨識，最後 labeldrop
  4. regex 加上 anchors 思考：部分匹配記得加 .*
  5. 每個 scrape_config 都該設 sample_limit：搭配 metric_relabel_configs 防 OOM
  6. K8s 用 ServiceMonitor / PodMonitor：別自己手寫 K8s SD relabel
  7. Debug 時看 /targets 頁面對照 Discovered/Target Labels：90% 的問題能直接看出
  8. HA 部署一定要用 alert_relabel_configs 移除 replica：否則 AM 無法去重

  1. 請寫 relabel 規則：把 K8s pod 的 label environment 升格為 Prometheus label，但只升格 production / staging 的 pod，development 不抓。
  2. 下面的規則為什麼不會過濾任何 target？請修正： 
    - source_labels: [__meta_consul_tag]
  regex: 'monitoring'
  action: keep
