SLO 與錯誤預算（Error Budget）實戰

SRE 的核心：不是「達成 100% 可用」，而是「達成承諾的可靠度」。 本章涵蓋 SLI / SLO / SLA 概念、錯誤預算、burn rate alerting、SLO dashboard 設計、 以及怎麼用 SLO 在「新功能 vs 穩定性」之間做平衡。

§ 1. SLI / SLO / SLA 基本概念

1.1 三個術語

術語意義範例
SLI（Service Level Indicator） 可量測的指標 「成功 request 比例」
SLO（Service Level Objective） 對 SLI 的內部目標 「99.9% 成功率（per 30 days）」
SLA（Service Level Agreement） 對客戶的合約承諾 「99.5% 達不到退費」

1.2 三者關係

SLI 量測:                成功率 99.95% （實際數字）
       │
SLO 目標:                99.9% （我們希望達成）
       │       ← 比 SLO 嚴比 SLA 寬，當作緩衝
SLA 承諾:                99.5% （給客戶的合約）

原則：SLI > SLO > SLA。SLO 比 SLA 嚴是為了在客戶看到 SLA 違反前先警覺。

1.3 為什麼不追求 100%

  • 100% 不可能（網路、硬體、第三方依賴都會壞）
  • 100% 極貴（每多一個 9，成本 10x）
  • 使用者感受不到 99.9% 與 99.99% 差別（多數情境）
  • 追求 100% 會讓團隊不敢部署新功能

SRE 的洞察：「不可靠的成本」與「可靠性的成本」中間有最佳平衡點。

1.4 9 對應的時間

SLO每月可 down每年可 down
99%7.3 小時3.65 天
99.5%3.65 小時1.83 天
99.9%（3 nines）43 分鐘8.76 小時
99.95%21 分鐘4.38 小時
99.99%（4 nines）4.3 分鐘52 分鐘
99.999%（5 nines）26 秒5 分鐘

多數 web 應用：3 nines 已是優秀。

§ 2. 錯誤預算（Error Budget）

2.1 概念

若 SLO 是 99.9%，0.1% 是可以失敗的「預算」。 這個預算可以「花」在部署新功能、實驗、計畫性 maintenance。

2.2 計算

# 30 天 SLO 99.9% 的 error budget
total_minutes = 30 × 24 × 60 = 43200
allowed_failure_minutes = 43200 × (1 - 0.999) = 43.2 分鐘

# 換成 request 觀點
total_requests = 30M（假設）
allowed_failures = 30M × 0.001 = 30K errors

2.3 Error Budget Policy

明確規範「budget 用完時做什麼」。Google SRE 的範例：

Budget 狀態動作
Budget 充足（> 50% remaining） 正常開發、可實驗、可改變架構
Budget 緊（10-50% remaining） 謹慎部署、降低風險
Budget 用完（0%） 凍結 feature deployment，只修 reliability
Budget 嚴重赤字 緊急會議、重新評估 SLO 是否合理

2.4 Error Budget 的精神

💡 SRE 核心哲學

錯誤預算讓 dev 與 ops 站在同一邊：
• Dev 想快速 ship feature → 預算讓他們能實驗
• Ops 想穩定 → 預算用完時 ops 有理由 push back
不是吵架，是用數字決定。

§ 3. 設計好的 SLI

3.1 SLI 公式

SLI = 好的事件 / 總事件 × 100%

範例:
- Availability SLI = 成功 request / 總 request
- Latency SLI = 快於閾值的 request / 總 request
- Quality SLI = 高品質回應 / 總回應

3.2 「好的事件」怎麼定義

關鍵是：從使用者視角定義「好」。

不好的 SLI 定義：

  • 「Server CPU 不超 80%」（user 不在乎）
  • 「沒有 panic」（panic 但沒 user 影響也沒事）
  • 「response time 平均 < 200ms」（用平均隱藏 outlier）

好的 SLI 定義：

  • 「成功 request 比例 ≥ 99.9%」
  • 「P95 response time < 500ms 的比例 ≥ 99%」
  • 「結帳完成率 ≥ 99.5%」

3.3 SLI 的四大類別

類別定義範例
Availability 系統能否服務 成功 request 比例
Latency 系統多快 P95 < 500ms 的比例
Throughput 系統處理量 QPS、訂單/秒
Quality 結果正確性 搜尋有結果比例、推薦相關性

3.4 SLI 量測位置

同一個指標可在不同層測，意義不同：

位置優點缺點
Server side（應用 metric） 容易實作、細粒度 看不到 client 真實體驗（CDN / network）
Load balancer / CDN edge 更接近 user 仍非真 user
Synthetic（k6 probe） 主動量、穩定 不是真 user 流量
RUM（Faro real user） 最貼近 user noisy（user 網路差也算）

實務：結合 server-side（穩定）+ synthetic（主動）+ RUM（真實）。

§ 4. SLO 目標設定

4.1 設定原則

  1. 從現況反推：如果 service 過去 3 個月平均 99.93%，設 99.9% 合理
  2. 看 user 痛點：99.5% 就崩潰嗎？99.9% 已被 reach 嗎？
  3. 看 cost：每多一個 9 成本翻倍，值不值得
  4. 留 buffer：SLA 99.5% → SLO 99.9% 給內部緩衝

4.2 不同類型 service 的 SLO 範例

Service 類型建議 SLO
金融 / 醫療 critical path99.99%+
電商主流程（結帳）99.9% - 99.95%
內部 dashboard99.5% - 99.9%
非 critical 的 batch job99% 或不設
實驗性新功能不設或 99%

4.3 多窗時段

SLO 通常用 30 天 rolling window：

  • 1 天：太短，雜訊大
  • 7 天：可看週末/平日趨勢
  • 28-30 天：標準
  • 90 天：給長期 trend 看

§ 5. PromQL 計算 SLO

5.1 Availability SLO

# 過去 30 天 availability
1 - (
  sum(increase(http_requests_total{service="api",status=~"5.."}[30d]))
  /
  sum(increase(http_requests_total{service="api"}[30d]))
)

# 範例結果: 0.9993 = 99.93%

5.2 Latency SLO

用 histogram 算「快的 request 比例」：

# 過去 30 天 P95 < 500ms 的比例
sum(rate(http_request_duration_seconds_bucket{service="api",le="0.5"}[30d]))
/
sum(rate(http_request_duration_seconds_count{service="api"}[30d]))

# 結果: 0.992 = 99.2% 的 request 在 500ms 內

5.3 Error Budget Burn

# Error budget 已用比例
(
  sum(increase(http_requests_total{status=~"5.."}[30d]))
  /
  sum(increase(http_requests_total[30d]))
)
/
0.001   # 1 - 0.999

# 結果 = 1.0 表示 budget 剛好用完
# 結果 < 1.0 表示還有 budget

5.4 Recording Rules

SLO query 通常複雜 + 多窗時段，用 recording rules 預先算：

groups:
  - name: slo:api
    interval: 30s
    rules:
      # 每分鐘的 error rate（多窗時段）
      - record: slo:api:errors:rate5m
        expr: |
          sum(rate(http_requests_total{service="api",status=~"5.."}[5m]))
          /
          sum(rate(http_requests_total{service="api"}[5m]))

      - record: slo:api:errors:rate1h
        expr: |
          sum(rate(http_requests_total{service="api",status=~"5.."}[1h]))
          /
          sum(rate(http_requests_total{service="api"}[1h]))

      - record: slo:api:errors:rate6h
        expr: |
          sum(rate(http_requests_total{service="api",status=~"5.."}[6h]))
          /
          sum(rate(http_requests_total{service="api"}[6h]))

      - record: slo:api:errors:rate3d
        expr: |
          sum(rate(http_requests_total{service="api",status=~"5.."}[3d]))
          /
          sum(rate(http_requests_total{service="api"}[3d]))

§ 6. Burn Rate Alerting

6.1 為什麼不能直接 alert error rate

直覺做法：「error rate > 0.1% alert」。問題：

  • 瞬間小尖峰 → false positive
  • 慢慢漏氣（每分鐘 0.05% error）→ 永遠不 alert，但 budget 在燒

6.2 Burn Rate 概念

Burn rate = 「當前 budget 消耗速度 / 正常速度」。 SLO 99.9% (budget 0.1%) 在 30 天均勻消耗，正常 burn rate = 1。 burn rate = 10 表示 10 倍速消耗 → 3 天就會用完整月 budget。

6.3 Multi-window Multi-burn-rate Alert

Google SRE Workbook 推薦：用多個窗 + 多個 burn rate。

嚴重度長窗短窗Burn rate意義
Critical 1h 燒 > 14.4× 5m 燒 > 14.4× 14.4 2% budget in 1h（很快壞）
Critical 6h 燒 > 6× 30m 燒 > 6× 6 5% budget in 6h
Warning 24h 燒 > 3× 2h 燒 > 3× 3 10% budget in 24h
Warning 3d 燒 > 1× 6h 燒 > 1× 1 10% budget in 3d

6.4 為什麼長+短窗

  • 長窗：避免 false positive（瞬間 spike 不算）
  • 短窗：avoid alert 一直 firing 即使問題已解
  • 兩者必須同時超才 alert

6.5 完整 PrometheusRule

apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: slo-api-burn-rate
spec:
  groups:
    - name: slo-burn-rate
      rules:
        - alert: APISLOBudgetBurnFast
          expr: |
            (
              slo:api:errors:rate1h > (14.4 * 0.001)
              and
              slo:api:errors:rate5m > (14.4 * 0.001)
            )
          labels:
            severity: critical
            slo: api-availability
            burn_rate: "14.4x"
          annotations:
            summary: "API SLO budget burning at 14.4× (2% in 1h)"
            description: "At this rate, budget exhausted in < 2 days"
            runbook_url: "https://wiki/runbook/slo-budget"

        - alert: APISLOBudgetBurnMedium
          expr: |
            (
              slo:api:errors:rate6h > (6 * 0.001)
              and
              slo:api:errors:rate30m > (6 * 0.001)
            )
          labels:
            severity: critical
            burn_rate: "6x"

        - alert: APISLOBudgetBurnSlow
          expr: |
            (
              slo:api:errors:rate24h > (3 * 0.001)
              and
              slo:api:errors:rate2h > (3 * 0.001)
            )
          labels:
            severity: warning
            burn_rate: "3x"

        - alert: APISLOBudgetBurnVerySlow
          expr: |
            (
              slo:api:errors:rate3d > (1 * 0.001)
              and
              slo:api:errors:rate6h > (1 * 0.001)
            )
          labels:
            severity: warning
            burn_rate: "1x"

§ 7. SLO Dashboard 設計

7.1 SLO Dashboard 必備 panel

  1. SLO 達成狀態（Stat）：當前 30 天 SLO 達成率
  2. Budget Remaining（Gauge）：剩多少 budget
  3. Burn Rate Heatmap：當前 burn rate（多窗時段）
  4. Error Rate Trend（Time series）：30 天 error rate 趨勢
  5. SLO Compliance Trend（Time series）：30 天 SLO 達成趨勢
  6. Top Error Sources（Bar chart）：哪個 endpoint / status 貢獻錯誤

7.2 範例 PromQL

# Panel 1: 當前 SLO 狀態
1 - slo:api:errors:rate30d
# Display: 99.93%

# Panel 2: Budget Remaining
1 - (slo:api:errors:rate30d / 0.001)
# 0.7 = 70% budget remaining

# Panel 3: Current burn rate (multi-window)
slo:api:errors:rate1h / 0.001
# 5.2x burn rate

# Panel 4: Error rate trend
slo:api:errors:rate5m   # 過去 30 天

# Panel 5: SLO compliance
1 - slo:api:errors:rate30d   # 30 天滾動

# Panel 6: Top errors by endpoint
topk(5,
  sum by (route) (rate(http_requests_total{service="api",status=~"5.."}[1h]))
)

7.3 視覺化技巧

  • 用顏色閾值明確（綠 / 黃 / 紅）
  • SLO 達成率閾值：> SLO 綠 / 等於 SLO 黃 / 低於 SLO 紅
  • Budget 用 Gauge 直觀顯示
  • Annotation 標 deployment / incident

§ 8. Sloth：SLO Generator

8.1 Sloth 是什麼

手寫 SLO PrometheusRule 很繁瑣（4 個 burn rate alert × 多 service）。 Sloth 是 OSS 工具，從簡單 YAML 自動產生完整 SLO rules + alerts。

8.2 Sloth SLO 定義

# sloth-slos.yaml
version: "prometheus/v1"
service: "api"
slos:
  - name: "availability"
    objective: 99.9
    description: "API availability SLO"
    sli:
      events:
        error_query: |
          sum(rate(http_requests_total{service="api",status=~"5.."}[{{.window}}]))
        total_query: |
          sum(rate(http_requests_total{service="api"}[{{.window}}]))
    alerting:
      name: APIAvailabilitySLO
      page_alert:
        labels:
          severity: critical
      ticket_alert:
        labels:
          severity: warning

8.3 產生 PrometheusRule

sloth generate -i sloth-slos.yaml -o prometheus-rules.yaml

# 自動產生：
# - 30 個 recording rules（多窗時段）
# - 4 個 burn rate alert
# - SLO metadata metric

8.4 為什麼用 Sloth

  • 標準化 SLO 定義（避免每團隊各寫各的）
  • 自動產生 multi-window multi-burn-rate alert
  • 內建 Grafana dashboard 模板
  • 整合 K8s Operator（可選）

8.5 替代方案

  • Pyrra：類似 Sloth，K8s-native，有 UI
  • OpenSLO：標準格式（Sloth / Pyrra 都支援）
  • Grafana Cloud SLO：商業 SaaS，UI 直接編輯
  • 純手寫：對少量 SLO 也行

§ 9. SLO Decision Framework

9.1 用 SLO 做產品決策

SLO 不只是技術指標，是產品決策工具。範例情境：

情境 1: 「要不要 ship 這個新功能」

  • 看 SLO budget remaining
  • > 50% → ship，可實驗
  • < 10% → 暫停 feature work，做 reliability

情境 2: 「要不要做大重構」

  • 近 90 天 SLO 都遠超目標 → budget 多，可重構
  • 近 90 天 SLO 緊 → 重構增風險，先穩定

情境 3: 「資源該投到哪」

  • SLO 連續 miss → 投資 reliability（infra / observability / oncall）
  • SLO 持續超標 → 投資 feature（user 不在乎更可靠）

9.2 SLO 是 vs 不是

SLO 是SLO 不是
產品決策依據個人 KPI
溝通工具處罰工具
長期目標瞬間達成
會調整定後不變
少數關鍵服務每個 service 都要

9.3 SLO Review 週期

  • Quarterly：團隊 review SLO 是否合理
  • 每月：檢視 budget 使用
  • 每週：oncall handoff 看 burn rate
  • 每天：dashboard 看狀態

§ 10. 實戰：電商 4 個 SLO

用 Ch 55 的電商系統，定 4 個關鍵 SLO：

10.1 SLO 1: Checkout Availability

service: checkout-flow
slos:
  - name: availability
    objective: 99.95%
    sli:
      good_query: |
        sum(rate(http_requests_total{
          service=~"frontend|cart|payment|order|inventory",
          route=~"/checkout/.*",
          status=~"2..|3.."
        }[{{.window}}]))
      total_query: |
        sum(rate(http_requests_total{
          service=~"frontend|cart|payment|order|inventory",
          route=~"/checkout/.*"
        }[{{.window}}]))

意義：結帳流程的端到端成功率（涉及 5 個 service）。

10.2 SLO 2: Search Latency

service: catalog-search
slos:
  - name: latency
    objective: 99%
    sli:
      good_query: |
        sum(rate(http_request_duration_seconds_bucket{
          service="catalog-service",
          route="/api/search",
          le="0.5"
        }[{{.window}}]))
      total_query: |
        sum(rate(http_request_duration_seconds_count{
          service="catalog-service",
          route="/api/search"
        }[{{.window}}]))

意義：99% 的搜尋在 500ms 內完成。

10.3 SLO 3: Order Processing Quality

service: order-processing
slos:
  - name: quality
    objective: 99.9%
    sli:
      good_query: |
        sum(rate(orders_processed_successfully_total[{{.window}}]))
      total_query: |
        sum(rate(orders_received_total[{{.window}}]))

意義：99.9% 的訂單成功被處理（包含 inventory 扣 / payment / 通知）。業務視角而非 HTTP。

10.4 SLO 4: Frontend LCP

service: frontend
slos:
  - name: lcp
    objective: 90%
    sli:
      good_query: |
        sum(rate(faro_web_vitals_lcp_bucket{le="2.5"}[{{.window}}]))
      total_query: |
        sum(rate(faro_web_vitals_lcp_count[{{.window}}]))

意義：90% 的頁面 LCP < 2.5s（Core Web Vitals「Good」標準）。

10.5 4 SLO 的層次

SLO類型觀點
Checkout AvailabilityAvailability跨 service 的關鍵 user journey
Search LatencyLatency單 service 的速度
Order Processing QualityQuality業務正確性
Frontend LCPLatency真實 user 體驗

4 個 SLO 從不同層次涵蓋系統，不會太多（5-10 個是黃金數）。

§ 11. 常見陷阱

⚠️ 陷阱 1: SLO 設太嚴

設 99.99% 但實際只能做到 99.5% → 永遠 alert、團隊麻木。
解法：從現況 +5% 開始，逐步提升。

⚠️ 陷阱 2: 用平均值當 SLI

「平均 latency < 200ms」隱藏 outlier。
解法：用「比例（percentile-bucket）」，例如「95% 的 request < 200ms」。

⚠️ 陷阱 3: 太多 SLO

每個 service 都設 5 個 SLO → 30 個 SLO 沒人看。
解法：聚焦在 user-facing critical journey（5-10 個 SLO 是好的數字）。

⚠️ 陷阱 4: SLO 不影響決策

SLO 達不到但繼續 ship feature → SLO 是裝飾品。
解法：明確 Error Budget Policy + leadership 支持。

⚠️ 陷阱 5: 追求 100% SLO

「我們要 100%」 → 不可能，反而不能投資新功能。
解法：教育 leadership「不可靠的成本 vs 可靠的成本」。
