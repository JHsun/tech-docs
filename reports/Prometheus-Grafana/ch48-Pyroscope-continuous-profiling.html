Chapter 48 · Pyroscope Continuous Profiling

Pyroscope：持續性能 Profiling

Grafana Stack 的「第四本柱」：profiling。傳統 metric / log / trace 告訴您「服務慢」、 「哪個 service 慢」、「哪個 request 慢」，但不告訴您「哪行 code 慢」。 Pyroscope 持續從 production 應用收集 CPU、記憶體 profile，讓您看到 function-level 的瓶頸。 本章涵蓋 profiling 概念、5 種 profile types、語言整合、Flame Graph、與 trace 整合。

§ 1. 什麼是 Continuous Profiling

1.1 從「event-based」到「continuous」

傳統 profiling：

  • 本地 dev：用 perf、pprof、Py-Spy profile 一段時間
  • Production 慢時：手動 attach profiler，收 30 秒，下載分析

痛點：需要重現問題、production 中緊急 attach 不方便。

Continuous Profiling：持續低 overhead 收集 profile（每 10s 取 sample），24/7 production 都有。

1.2 Profile 是什麼

Profile = 「採樣統計」。例：CPU profile 每 10ms 中斷一次，記下當前的 call stack。 跑 30 秒收 3000 個 sample，統計後就知道「哪些函式占了大部分時間」。

1.3 跟其他三本柱的關係

訊號回答的問題
Metric「服務有多慢？」（latency p99 = 800ms）
Log「在處理什麼時慢？」（message: handling order 12345 slow）
Trace「哪個 service 慢？哪個 span 占時間？」（payment-service 占 700ms）
Profile「哪行 code 慢？」（jsonEncode 占 400ms，字串拼接占 200ms）

Profile 是「把 trace 從 service 級下鑽到 function 級」。

1.4 Continuous Profiling 帶來的改變

  • 不需重現 → 直接看歷史 profile
  • 「peak hour」自動有資料（用戶最多時的 profile）
  • 對比 deploy 前後（regression detection）
  • 跨 instance 統計（找全 fleet 的 hot path）

§ 3. 架構與元件

3.1 元件架構

graph LR APP[Application
+ Profiler SDK] ALLOY[Alloy · eBPF] DIST[Distributor] INGEST[Ingester] OBJ[(Object Storage)] QUERIER[Querier] STORE[Store Gateway] COMP[Compactor] GRAFANA[Grafana
Pyroscope datasource] APP -- pprof push --> DIST ALLOY -- scrape · push --> DIST DIST -- shards --> INGEST INGEST -- flush --> OBJ COMP --> OBJ GRAFANA -- query --> QUERIER QUERIER -- recent --> INGEST QUERIER -- old --> STORE STORE -- read --> OBJ

3.2 元件職責

跟 Mimir / Loki / Tempo 結構幾乎一致：

  • Distributor：接收 profile push（pprof 格式）、shard
  • Ingester：記憶體 buffer + 定期 flush block
  • Querier：執行查詢，從 ingester + store gateway 讀
  • Store Gateway：從 object store 讀歷史資料
  • Compactor：合併 block

3.3 Pprof 格式

Pyroscope 內部用 Google 的 pprof 格式（業界 de facto）。 所有支援的 SDK / Agent 都送 pprof： 

# pprof binary 格式（protobuf）
samples: [
  { stack: [main, runQuery, jsonEncode], value: 1234 },
  { stack: [main, runQuery, dbQuery], value: 5678 },
  ...
]
# value 單位依 profile type：CPU 是 nanoseconds、memory 是 bytes

§ 4. 五種 Profile Types

4.1 完整對照

Type看出Overhead
CPU 哪些函式吃 CPU 低（< 1%）
Memory: alloc_objects 分配最多物件的函式 中
Memory: alloc_space 分配最多 bytes 的函式 中
Memory: inuse_objects / inuse_space 當下記憶體佔用最多的函式（Go GC 視角） 中
Goroutine（Go only） goroutine 數量、leak 低
Block（Go only） blocking call（mutex、channel） 中
Mutex（Go only） 競爭 lock 的函式 中

4.2 CPU Profile 最常用

多數場景從 CPU 開始：「哪個函式 CPU 用最多」是最直覺的瓶頸發現方式。 其他 profile types 等懷疑特定問題（記憶體 leak、goroutine leak）時再看。

4.3 何時看其他類型

症狀看哪種 profile
CPU 高CPU profile
OOMMemory: inuse_space
GC pause 長Memory: alloc_objects
Goroutine leak（Go）Goroutine profile
Lock contention（Go）Mutex / Block profile
網路 IO 等待Block profile

§ 5. Flame Graph 解讀

5.1 Flame Graph 結構

          ┌──────────────────────────────┐
高度       │ main (100%)                  │  ← 最底，全部時間
（堆疊深度） │ ├──────────┬─────────┬──────│
          │ │ runQuery │  parse  │ logs │
          │ │  (60%)   │  (30%)  │ (10%)│
          │ ├──┬──┬─┐  │ ┌──┬──┐│      │
          │ │db│js│..│  │..│..││      │
          │ └──┴──┴──┘  │  │  ││      │
寬度 ∝ 時間佔比
  • X 軸：佔比（不是時間順序！）
  • Y 軸：堆疊深度（最底是 main，往上是被呼叫者）
  • 寬度 ∝ 該函式佔用的時間百分比
  • 顏色：通常無語意，只是區分

5.2 解讀技巧

💡 找瓶頸的方法
  1. 看 flame graph 中最寬的 leaf（樹頂葉子）
  2. 該函式直接占用 CPU 最多
  3. 追溯它的 caller 路徑（往下看 stack）
  4. 判斷是否合理（jsonEncode 占 50%？合理嗎？）

5.3 Flame Graph vs Icicle Graph

方向不同：

  • Flame：根在底，火焰向上燒（傳統）
  • Icicle：根在頂，向下倒掛（Pyroscope 預設，新一代）

內容相同，視覺方向反。Pyroscope UI 都支援切換。

5.4 Pyroscope UI 功能

  • Search：在 flame graph 中找特定函式名
  • Diff：兩個時間段的 flame graph 差異（找 regression）
  • Single view：單一 flame graph
  • Comparison：左右並列比
  • Sandwich view：選一個函式，看誰呼叫它（caller）+ 它呼叫誰（callee）

6.2 Go 整合範例

// main.go
package main

import (
    "github.com/grafana/pyroscope-go"
)

func main() {
    pyroscope.Start(pyroscope.Config{
        ApplicationName: "myapp",

        // 連到 Pyroscope server
        ServerAddress: "http://pyroscope:4040",

        // 啟用哪些 profile
        ProfileTypes: []pyroscope.ProfileType{
            pyroscope.ProfileCPU,
            pyroscope.ProfileAllocObjects,
            pyroscope.ProfileAllocSpace,
            pyroscope.ProfileInuseObjects,
            pyroscope.ProfileInuseSpace,
        },

        // Tags（會變成 label）
        Tags: map[string]string{
            "region":  "us-east-1",
            "version": "v1.2.3",
        },
    })

    // 跑您的應用
    runApp()
}

6.3 Java 整合範例

# 啟動時加 Java agent

import pyroscope

pyroscope.configure(

    },
)

# 跑您的應用

6.5 Tags 與 Labels

Tags 在 Pyroscope 中變 label，用來篩選與分組（類似 Prometheus）。 常用 tag：service、region、environment、version。

不要用 high-cardinality tag（user_id、request_id），跟 Loki 一樣會炸 cardinality。

§ 7. Auto-instrumentation（Alloy + eBPF）

7.1 為什麼要 auto-instrumentation

SDK 整合需要改 code，有些情境不方便：

  • 程式碼是別人的（廠商產品）
  • 多語言混合（每個都改太麻煩）
  • 沒有 SDK 支援的語言（Erlang、Elixir 等）

7.2 eBPF Profiling

用 Linux eBPF（kernel-level）取 profile，不用改 code、不用改執行檔， 只要把 profiler 放在同個 host / pod 就行。

7.3 Grafana Alloy 內建 eBPF Profiler

// alloy config.river
pyroscope.ebpf "default" {
    forward_to = [pyroscope.write.default.receiver]

    // 自動發現 K8s pod
    targets = discovery.kubernetes.pods.targets
}

pyroscope.write "default" {
    endpoint {
        url = "http://pyroscope:4040"
    }
}

7.4 K8s DaemonSet 部署

helm install alloy grafana/alloy \
  --namespace observability \
  --set 'controller.type=daemonset' \
  -f alloy-values.yaml

# alloy-values.yaml
alloy:
  # 給 eBPF 需要的權限
  securityContext:
    privileged: true
  hostNetwork: true
  hostPID: true
  configMap:
    create: true
    content: |
      # 上面 pyroscope.ebpf 設定...

7.5 eBPF 限制

  • Linux only（Windows 不行）
  • 需要 kernel 4.18+（推薦 5.x+）
  • 需要 privileged 權限（K8s 安全性考量）
  • 跟 SDK 比，沒有 application-level context（trace_id、user 自訂 tag）

7.6 SDK vs eBPF 選擇

選 SDK 當選 eBPF 當
能改 codecode 不能改
需要 trace 整合整 fleet 統一收
需要 tag 自訂不需 application context
單一語言多語言混合

實務常組合：關鍵 service 用 SDK（精細），其他 fleet 用 eBPF（廣度）。

§ 8. 與 Trace 整合

8.1 Trace ↔ Profile

Pyroscope 與 Tempo 緊密整合：

  • Trace 的某個 slow span → 點 "Profile for span" → 開 Pyroscope，filter 到該時間段 + service
  • 看到 flame graph 中 hot function
  • 確認瓶頸到 function level

8.2 Span Profiling（v2.10+）

更新功能：每個 span 直接嵌入 profile data。Tempo trace 顯示 flame graph icon， 點選看該 span 的 profile（不只整個 service 的）。

需要 SDK 啟用：

// Go
import "github.com/grafana/pyroscope-go/godeltaprof/http/pprof"

// 把 trace 和 profile 連結
profiler.Start(pyroscope.Config{
    // ...
    HTTPHeaders: map[string]string{
        "X-Trace-ID": traceID,    // 自動關聯
    },
})

8.3 Tempo Datasource 設定

# Ch 34 介紹過
- name: Tempo
  type: tempo
  jsonData:
    tracesToProfiles:
      datasourceUid: pyroscope-prod
      tags:
        - { key: 'service.name', value: 'service_name' }
      profileTypeId: 'process_cpu:cpu:nanoseconds:cpu:nanoseconds'
      customQuery: false

8.4 Profile Type ID 格式

{name}:{sample_type}:{unit}:{period_type}:{period_unit}

範例:
process_cpu:cpu:nanoseconds:cpu:nanoseconds       # CPU
memory:alloc_objects:count:space:bytes            # alloc objects
memory:alloc_space:bytes:space:bytes              # alloc space
memory:inuse_objects:count:space:bytes
memory:inuse_space:bytes:space:bytes
goroutine:goroutine:count:goroutine:count
mutex:contentions:count:mutex:count
mutex:delay:nanoseconds:mutex:count
block:contentions:count:block:count
block:delay:nanoseconds:block:count

8.5 整合 Profiles Drilldown

Ch 39 介紹的 Profiles Drilldown 自動列出所有 service，點選看 flame graph， 不用寫 query。最低門檻使用 Pyroscope。

§ 9. 部署模式

9.1 三種部署

跟 Loki / Tempo / Mimir 一樣的 monolithic / read-write-backend / microservices。

9.2 Helm 安裝（單一 binary）

helm repo add grafana https://grafana.github.io/helm-charts

helm install pyroscope grafana/pyroscope \
  --namespace pyroscope --create-namespace

9.3 Helm 安裝 Microservices

helm install pyroscope grafana/pyroscope-distributed \
  --namespace pyroscope --create-namespace \
  -f pyroscope-values.yaml

# pyroscope-values.yaml
pyroscope:
  structuredConfig:
    multitenancy_enabled: true
    storage:
      backend: s3
      s3:
        bucket_name: pyroscope-data
        region: us-east-1

distributor:
  replicas: 3
ingester:
  replicas: 3
  persistence:
    size: 100Gi
querier:
  replicas: 2
queryFrontend:
  replicas: 2
storeGateway:
  replicas: 2
compactor:
  replicas: 1

9.4 Storage Sizing

Profile 資料比 metric 大很多。粗估：

  • 每個 service 每天 ~1-5 GB（含 5 種 profile types）
  • 100 services + 30 天 = ~300-500 GB
  • S3 成本 ~$10-30/月

§ 10. Production Best Practices

10.1 Profile Types 選擇

  • 必開：CPU profile（最有用、overhead 最低）
  • 看狀況：Memory profile（懷疑 OOM 或 GC 問題時）
  • 選擇性：Goroutine / Mutex / Block（深度 debug 用）

10.2 Sampling Rate

// 預設 100Hz（每 10ms 取 sample）
// CPU 重的應用可降到 50Hz
ProfileSampleRate: 50,

10.3 Cardinality 控制

  • 不要把 user_id / request_id 當 tag
  • service 數 × tag values × profile types 不要爆
  • 用 limits.max_global_series_per_tenant 限制

10.4 Retention

compactor:
  block_retention: 720h    # 30 天

# 短 retention 是 OK 的：
# - profile 主要用於 debug（即時/近期）
# - 7-14 天通常夠 regression detection

10.5 監控 Pyroscope 自身

必看 metrics：
- pyroscope_distributor_received_bytes_total
- pyroscope_ingester_blocks_total
- pyroscope_ingester_memory_segments
- pyroscope_querier_query_duration_seconds

必設告警：
- PyroscopeIngestionFailing
- PyroscopeQuerierSlow
- PyroscopeStorageGrowthHigh

10.6 Cost 控制

  • 用 eBPF for the masses, SDK for critical
  • Sample rate 預設 100Hz 通常 ok，不用更高
  • S3 Intelligent Tiering（profile 老資料很少看）
  • 不要每個 service 都開所有 profile types

§ 11. 實戰：找一個 CPU bottleneck

11.1 場景

「api-service CPU 使用率 80%，看 dashboard 沒明顯 spike，要找瓶頸。」

11.2 用 Pyroscope

  1. 進 Grafana → Drilldown → Profiles
  2. 選 Pyroscope datasource
  3. 選 service: api-service
  4. 選 profile type: process_cpu:cpu:nanoseconds:cpu:nanoseconds
  5. 看 flame graph：
main (100%)
└── http.handler (95%)
    ├── parseRequest (15%)
    ├── runBusinessLogic (30%)
    │   └── jsonEncode (20%)         ← 占大量時間！
    ├── databaseQuery (40%)
    └── writeResponse (10%)

11.3 發現問題

jsonEncode 占 20% CPU，異常多。深入：點選 jsonEncode → 看它的 caller / callee。 發現是 nested struct 每次 marshal 重複 reflection。

11.4 對比 Diff

修復後 deploy。用 Pyroscope Diff View：

  • 左：deploy 前一天的 profile
  • 右：deploy 後當天的 profile
  • 看到 jsonEncode 從 20% 降到 3%
  • 整體 CPU 從 80% 降到 50%

11.5 與 Trace 配合

還可從特定 slow trace 跳到 profile：

  1. Tempo 看到一個 trace P99 800ms
  2. 點該 span 的 "Profile for span"
  3. Pyroscope 開啟，filter 到該時間段
  4. 看到 jsonEncode 占 40%（比平均更糟）
  5. 確認是 jsonEncode 的鍋
