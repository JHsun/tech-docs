§ 1. Prometheus 常見故障

1.1 Prometheus OOM / 記憶體飆高

症狀：Prometheus pod 反覆被 OOM killed。

原因排查：

  1. Active series 太多：prometheus_tsdb_head_series
  2. WAL replay：重啟時暫時 memory 飆（正常）
  3. 大 query 拖垮：prometheus_engine_queries
  4. Cardinality 爆炸：見 §6

解法：

# 找 high-cardinality metric
topk(10, count by (__name__)({__name__=~".+"}))

# 找 high-cardinality label
count by (label_name) (group by (label_name, label_value)({...}))

# Drop 不必要的 metric
metric_relabel_configs:
  - source_labels: [__name__]
    regex: 'go_.*|process_collector_.*'
    action: drop

# 限制 query 大小
--query.max-samples=50000000   # 50M samples per query

# Shard
prometheusSpec.shards: 4   # 分散 series 到 4 shards

1.2 Target 顯示 DOWN

排查順序：

  1. UI: Status → Targets，看 lastError
  2. 從 Prometheus pod 直接 curl target：kubectl exec prom-0 -- wget -qO- http://target:port/metrics
  3. 檢查 target 是否真的有 /metrics endpoint
  4. 檢查 NetworkPolicy 是否 block
  5. 檢查 ServiceMonitor port name 對不對
  6. 檢查 TLS / auth 設定

1.3 ServiceMonitor 沒被抓

常見 5 個原因（見 Ch 54），最常見是 label 不對：

# 確認 Prometheus selector
kubectl get prometheus -o yaml | grep -A 10 serviceMonitorSelector

# 解法：抓全 cluster
prometheusSpec.serviceMonitorSelectorNilUsesHelmValues: false

1.4 Disk full / TSDB corruption

# 檢查
df -h /var/lib/prometheus

# 如果是 TSDB corruption
# 1. Stop Prometheus
# 2. 移走損壞的 block：mv /var/lib/prometheus/data/01ABC... /tmp/
# 3. 重啟，會跳過該 block

# 預防
# 設 retention size + time
--storage.tsdb.retention.time=30d
--storage.tsdb.retention.size=180GB

1.5 Recording Rule 沒執行

# 檢查
curl http://prometheus:9090/api/v1/rules | jq '.data.groups[].rules[] | select(.health!="ok")'

# 看 evaluation duration
prometheus_rule_evaluation_duration_seconds

# 太慢 → 拆 rule group / 加 interval

1.6 Remote write backlog

# 檢查
prometheus_remote_storage_samples_pending

# 持續飆高 → backend 寫不進
# 看 backend：Mimir distributor 是否 OK
# 看 network：跨 region 是否慢

# 調整
remote_write:
  - url: ...
    queue_config:
      max_shards: 200       # 提高並行
      capacity: 10000
      max_samples_per_send: 5000

§ 2. Alertmanager 常見故障

2.1 Alert 沒到通知 channel

排查順序：

  1. Prometheus 是否真的觸發？看 Alerts UI
  2. Alert 有沒到 AM？看 AM UI 是否顯示
  3. Routing tree 是否走到正確 receiver？用 amtool config routes test
  4. Receiver 設定是否對？webhook URL / token
  5. Silence / inhibit 是否誤抓？看 AM Silences UI
  6. Alertmanager log：kubectl logs alertmanager-0

2.2 太多通知（noise）

# 解法 1: Group properly
route:
  group_by: [alertname, namespace, service]
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h    # 不要太頻繁重複

# 解法 2: Inhibit rule
inhibit_rules:
  - source_match: { severity: 'critical' }
    target_match: { severity: 'warning' }
    equal: [alertname, namespace]

# 解法 3: 提高閾值
expr: rate(...) > 0.05   # 從 0.01 提高
for: 10m                 # 從 5m 拉長

# 解法 4: 分 channel 避免 noisy 影響重要

2.3 HA cluster 不同步

# 三個 AM replica 應該 gossip

# 檢查
curl http://alertmanager:9093/api/v2/status | jq '.cluster'

# 應該看到 peers > 0
# peers = 0 表示 cluster 沒形成

# 解法
# 1. 確認 --cluster.peer 參數對
--cluster.peer=alertmanager-0.alertmanager:9094
--cluster.peer=alertmanager-1.alertmanager:9094

# 2. 開 9094/UDP+TCP
# 3. K8s headless service 設對

2.4 Silence 沒 silence 住

常見：matcher 寫錯。

# Matcher 必須 exact match alert label
# 看 alert label 用：
amtool alert query alertname=MyAlert -o json | jq '.[0].labels'

# Silence matcher 完全對應
amtool silence add \
  alertname=MyAlert \
  severity=critical \
  cluster=prod-east-1 \
  --duration=1h

§ 3. Grafana 常見故障

3.1 Dashboard 看不到資料

順序排查：

  1. Datasource 健康？Test connection
  2. 用 Explore 直接查 metric，是否有資料？
  3. Dashboard time range 對嗎？
  4. Variable 選擇對嗎？（常見：選了不存在的 namespace）
  5. Panel query 語法錯？
  6. Datasource 權限：service account 能不能查？

3.2 「No data」vs「Null」差異

  • No data：query 完全沒結果（label 不存在 / time range 沒資料）
  • Null：series 存在但某點無值（連線中斷）

Panel options 可調 Null value: connect / null / zero。

3.3 Dashboard 慢

排查：

  1. 面板數量：> 30 個 panel 通常太多
  2. Query 複雜度：用 Inspector → Stats 看每 query 耗時
  3. Time range 太長：30 天 high-resolution 很慢
  4. Datasource 慢：Prometheus 自身吃掉

解法：

  • 用 Recording Rules 預先算 panel query
  • 降低 step（讓 datasource 自動 downsample）
  • 用 Query Caching（Grafana Enterprise）
  • 分拆 dashboard（避免一個 dashboard 30+ panel）

3.4 SSO 登入失敗

# 檢查 grafana.ini
[auth.generic_oauth]
enabled = true
client_id = ...
client_secret = ...
auth_url = https://idp/oauth/authorize
token_url = https://idp/oauth/token
api_url = https://idp/oauth/userinfo

# Common 錯
# - redirect_uri 沒在 IdP 註冊
# - scopes 沒包含 user info
# - role mapping 寫錯
# - email_attribute_path 不對

# Debug log
[log]
level = debug
filters = oauth:debug

3.5 Plugin 安裝後不顯示

# 確認重啟 Grafana
kubectl rollout restart deployment grafana

# 確認 plugin signed
GF_PLUGINS_ALLOW_LOADING_UNSIGNED_PLUGINS=my-plugin

# 看 log
kubectl logs grafana-pod | grep -i plugin

§ 4. Mimir / Loki / Tempo 常見故障

4.1 Mimir Distributor 拒收

# Symptom: 429 Too Many Requests
# Cause: ingestion rate 超 limit

# 看 metric
cortex_distributor_received_samples_total
cortex_ingester_max_samples_per_query_in_use

# 解法
overrides:
  tenant-prod:
    ingestion_rate: 200000          # 提高
    ingestion_burst_size: 400000
    max_global_series_per_user: 5000000

4.2 Loki 查詢「too many outstanding requests」

# Cause: querier 處理不過來

# 解法
querier:
  max_concurrent: 16   # 提高並行

# Scale querier
kubectl scale deployment loki-querier --replicas=10

# Query timeout 提高
limits_config:
  query_timeout: 5m

4.3 Tempo trace 找不到

# 1. Sampling 把它丟了？
# Tail-based 設定看 keep policies

# 2. trace_id 拼錯？
# Tempo 用 32-char hex（不是 UUID）

# 3. ingestion 沒寫進？
# 看 tempo_distributor_spans_received_total

# 4. retention 過了？
# 預設 14 天

4.4 LGTM 訊號間關聯不上

例：log 裡有 trace_id 但 Loki derivedFields 沒跳到 Tempo。

檢查：

  1. Loki datasource 的 derivedFields regex 對嗎？
  2. Tempo datasource UID 寫對嗎？
  3. 用 Inspector 看 transformed value
  4. 確認 trace 在 Tempo 真的存在

§ 5. 效能調優總集

5.1 Prometheus

議題調整
Memory 4GB per 1M active series（粗估）
Disk SSD 必要；NVMe 更佳
WAL 分開 SSD / RAM disk（高 IOPS）
Scrape 分散 scrape interval（避免同步 spike）
Recording rules 常用 query 都 record
Query log 啟用 --query.log-file 找慢 query

5.2 Loki

  • Index：用 boltdb-shipper / TSDB（新）
  • Chunk Store：S3 + cache
  • Querier：scale out（每個 query 大量並行 chunk read）
  • Compactor：定期跑（避免 small chunk 太多）
  • Cardinality：log label < 10 個（user_id 等放 log line）

5.3 Tempo

  • Object Storage：S3 必須（不要用 local disk）
  • Compaction：開啟（block 太多會慢）
  • Cache：開 Redis / memcached
  • Sampling：在 collector tail-based（避免 push 100% 進 Tempo）

5.4 Mimir

  • Ingester：StatefulSet + 大 memory + SSD（chunk in-memory）
  • Compactor：必須跑（merge blocks）
  • Store-gateway：cache + memory（lazy load chunks from S3）
  • Querier：split query（小 sub-query）
  • Cardinality limit：per-tenant override

5.5 Grafana

  • Backend DB（Postgres）：scale 到 db pool size = 30+
  • Session storage：Redis
  • Reverse proxy：nginx 加 cache static asset
  • Alertmanager: 不要在 Grafana 跑（用 standalone）

§ 6. Cardinality 完全控制

6.1 Cardinality 公式

active_series = ∏(unique values per label)

範例：
http_requests_total{
  service: 10,
  endpoint: 100,
  status: 6,
  method: 5
}
= 10 × 100 × 6 × 5 = 30,000 series 給單一 metric！

加 user_id（1M users）：
= 30,000 × 1,000,000 = 30 BILLION series  💥

6.2 高風險 label

絕對不放 metric label：

  • user_id / customer_id
  • order_id / transaction_id / request_id
  • email / username
  • IP address
  • URL path with ID（用 route template /users/:id）
  • timestamp
  • UUID
  • session_id

6.3 找出兇手

# Top 10 高 cardinality metric
topk(10, count by (__name__)({__name__=~".+"}))

# 看單一 metric 的 cardinality 分布
count by (label_name)(
  count by (label_name, label_value) (your_metric)
)

# 用 mimirtool（Mimir）
curl http://mimir/api/v1/cardinality/label_names \
  -H "X-Scope-OrgID: tenant" | jq

# 用 cardinality dashboard（Mimir 內建）

6.4 緩解策略

策略適用
Drop label（labeldrop） 該 label 不重要
Drop metric（metric_relabel drop） 整個 metric 不重要
Aggregate before send（recording rule） 原始太細，聚合後夠用
Bucket reduce histogram bucket 太多
Move to log 那個維度的細節放 log（如 user_id）
Move to trace 放 span attribute（trace 是 sampled，可承受）

6.5 預防

# 在 Prometheus 設 enforced limit
prometheusSpec:
  enforcedSampleLimit: 10000      # 每 scrape 上限
  enforcedTargetLimit: 1000        # 每 job target 上限
  enforcedLabelLimit: 30           # 每 series label 上限
  enforcedLabelNameLengthLimit: 200
  enforcedLabelValueLengthLimit: 200

# Mimir 也設 per-tenant
overrides:
  tenant-prod:
    max_global_series_per_user: 1000000
    max_global_series_per_metric: 50000

§ 7. 安全強化 Checklist

7.1 認證 / 授權

  • ✅ Prometheus 不對外（Reverse Proxy + auth）
  • ✅ Grafana 用 SSO（IAM Identity Center / Entra / Okta）
  • ✅ Service Account Token 有 expiry，定期 rotate
  • ✅ API Key（舊式）禁用
  • ✅ Alertmanager API 用 mTLS
  • ✅ Loki / Mimir / Tempo 用 multi-tenant header（X-Scope-OrgID）

7.2 網路

  • ✅ NetworkPolicy 限制 monitoring namespace
  • ✅ TLS everywhere（內部也要）
  • ✅ Internal traffic：用 Service Mesh mTLS（Istio / Linkerd）
  • ✅ External：Ingress + cert-manager 自動 cert rotation
  • ✅ Egress：限制能對外的 service（避免 telemetry exfiltration）

7.3 Secret 管理

  • ✅ Grafana datasource credentials 用 ExternalSecret（Vault / SecretsManager）
  • ✅ Webhook URL（Slack / PagerDuty）放 Secret
  • ✅ Grafana secureJsonData 加密儲存
  • ✅ DB password 不在 Helm values 中

7.4 RBAC

  • ✅ Grafana 內 Folder permission 細粒度
  • ✅ Datasource permission 限制（dev 不能查 prod datasource）
  • ✅ Service Account 給最小權限（Viewer 不要 Admin）
  • ✅ K8s RBAC：Prometheus SA 只能讀 metric endpoint

7.5 PII / 敏感資料

  • ✅ Faro beforeSend 過濾 URL query / form
  • ✅ Log 不打整個 request body
  • ✅ Trace span attribute 過濾（user.email 改 hash）
  • ✅ Metric label 不放 PII（user_id 改 user_tier）
  • ✅ DLP 工具掃 log 確認沒洩漏（如 Cloud DLP）

7.6 容器安全

  • ✅ Pod 不跑 root（除了 Beyla / eBPF）
  • ✅ Read-only filesystem
  • ✅ Drop ALL capabilities，只加需要的
  • ✅ Image scan（Trivy / Snyk）
  • ✅ 鎖 image tag（不用 :latest）

7.7 Audit

  • ✅ Grafana audit log 開啟（Enterprise feature 或自架）
  • ✅ K8s audit log 收進 Loki
  • ✅ AlertManager silence 記錄誰建
  • ✅ Dashboard 修改紀錄（Grafana 內建 version history）

§ 8. 容災與備份

8.1 Backup 對象

對象方法
Prometheus TSDB Snapshot API（短期）/ remote_write 到 Mimir（長期）
Mimir blocks S3 versioning + cross-region replication
Loki chunks S3 versioning + cross-region replication
Tempo blocks S3 versioning + cross-region replication
Grafana DB PostgreSQL backup（每日 + WAL）
Grafana dashboards Git repo（GitOps）
Prometheus / AM rules Git repo

8.2 RTO / RPO 目標

RTO（Recovery Time Objective）= 多久要恢復
RPO（Recovery Point Objective）= 可接受多少資料遺失

範例：
- Critical（金融）   : RTO 1h, RPO 5min
- 一般 production    : RTO 4h, RPO 1h
- 內部 dashboard     : RTO 1day, RPO 4h

8.3 DR Drill

必須定期演練，不然不確定真的能恢復：

  • 每季：恢復一個 Grafana 從 backup
  • 每年：完整 region failover drill
  • 每月：驗證 backup 真的能 restore（不只是有檔）

8.4 GitOps 完整 backup

把所有 config 放 Git，極端情境用 Git 重建一切：

git-repo/
├── prometheus/
│   ├── prometheus.yml
│   └── rules/*.yaml
├── alertmanager/
│   └── alertmanager.yml
├── grafana/
│   ├── dashboards/
│   ├── datasources/
│   └── alert-rules/
├── mimir/
│   └── mimir.yaml
└── helm/
    ├── kube-prometheus-stack/values.yaml
    ├── loki/values.yaml
    └── tempo/values.yaml

§ 9. Production Readiness Checklist

準備上線時逐項勾選：

9.1 部署

  • ☐ HA：Prometheus 2+ replicas、Alertmanager 3+ replicas、Grafana 2+ replicas
  • ☐ Persistent storage 設定（不用 emptyDir）
  • ☐ Resource requests / limits 設好
  • ☐ HPA 設定（如果適用）
  • ☐ PodDisruptionBudget 設好
  • ☐ 跨 AZ 部署（podAntiAffinity）

9.2 監控自身

  • ☐ Meta-monitoring：另一套 Prometheus 監控主 Prometheus
  • ☐ Black box probe 確認服務 endpoint OK
  • ☐ Self alerts: PrometheusOOM / TSDBCorruption / RemoteWriteBacklog
  • ☐ Dashboard for monitoring 系統自身

9.3 觀測性

  • ☐ 每個 service 有 RED metric
  • ☐ 每個 service 有 dashboard
  • ☐ Critical service 有 SLO
  • ☐ Alert 從 user 視角（symptom-based）
  • ☐ 完整 user journey 能 trace
  • ☐ Log structured + trace_id 自動注入

9.4 自動化

  • ☐ Dashboard / Alert 都在 Git
  • ☐ CI 跑 promtool / mimirtool lint + test
  • ☐ CD 自動 deploy 到 Grafana / Mimir
  • ☐ Maintenance window 用 amtool 自動建 silence

9.5 安全

  • ☐ 所有 endpoint 有認證
  • ☐ TLS everywhere
  • ☐ NetworkPolicy 限制
  • ☐ Secret 用 ExternalSecret
  • ☐ 沒 PII 進 metric label
  • ☐ Audit log 開啟

9.6 容災

  • ☐ Backup 自動化（每日）
  • ☐ Restore 流程文件化 + 演練過
  • ☐ Cross-region replication（適用時）
  • ☐ RTO / RPO 明確定義
  • ☐ Runbook 完整（每個 alert 有對應 runbook URL）

9.7 流程

  • ☐ Oncall rotation 設好
  • ☐ Incident response 流程
  • ☐ Postmortem template
  • ☐ SLO review 每季
  • ☐ Cost review 每月

§ 10. 常見反模式

⚠️ 反模式 1：「監控所有東西」

初期想監控所有 metric → cardinality 爆 / cost 失控。
正解：先有 RED + USE，業務 metric 按需求加，定期砍 unused。

⚠️ 反模式 2：「Cause-based alerting」

對 CPU / Memory / Disk 設 alert → 半夜被叫但 user 完全沒受影響。
正解：Symptom-based（對 user 影響的 alert）。Cause-based 留 dashboard。

⚠️ 反模式 3：「100% trace + 100% retention」

100% 送 trace + 30 天 retention → 儲存費用爆。
正解：tail-based sampling（error/slow 100%、其他 1-5%）+ 7-15 天 retention。

⚠️ 反模式 4：「Dashboard 過多」

300+ dashboard、80% 沒人看。
正解：每月 review dashboard usage stats，砍掉沒人看的。標記 owner。

⚠️ 反模式 5：「Alert 太多」

每天 100+ alert、團隊 alert fatigue → 真正重要的被忽略。
正解：定期 review，砍從未觸發的、噪音的。Severity 嚴格分級。

⚠️ 反模式 6：「沒有 runbook」

Alert 響了但不知道怎麼辦 → MTTR 飆高。
正解：每 critical alert 必有 runbook_url annotation，連到清楚步驟。

⚠️ 反模式 7：「SLO 設了不看」

設了 SLO 但 budget 用完還繼續 ship feature → SLO 變裝飾。
正解：明確 Error Budget Policy + leadership 支持。

⚠️ 反模式 8：「Monitoring 自己沒監控」

Prometheus 壞了沒人知道，incident 時盲飛。
正解：Meta-monitoring + black box probe + 雙通道告警。

🎓 觀測性的核心精神
  • Observability ≠ Monitoring。Monitoring 知道什麼壞了，Observability 知道為什麼。
  • 三本柱（metric / log / trace）互補，加上 profile 是現代完整 stack。
  • 從 user 視角定義「健康」，不是 server 視角。
  • SLO 是工具不是目的，幫助做產品決策，不是 KPI。
  • RED / USE / 4 Golden：給不同層次用，每個 service 必有。
  • Cardinality 是最大成本驅動，從 day 1 就要管。
  • Alert 鐵則：對 user 影響的 symptom alert，內部 cause 留 dashboard。
  • Trace_id 自動注入是 LGTM stack 的關鍵：log ↔ trace ↔ metric 串聯。
  • OpenTelemetry 是業界統一標準，新專案優先用。
  • GitOps 一切：dashboard / alert / rule / config 都進 Git。
  • 監控自身：observability 系統不能是 SPOF。
  • Cost 與 reliability 平衡：每月 review，砍冗餘。
  • Continuous improvement：postmortem → action items → 改進系統。

📝 最終練習：建立您自己的 Production Stack

把 60 章知識整合成一個完整實作計畫：

  1. 需求分析：列出您系統的服務、語言、規模、雲環境、user journey
  2. 架構設計：選擇路線（純 Cloud-native / Managed / 自架 / 混合），畫架構圖
  3. SLO 定義：5 個關鍵 SLO（含 SLI query、目標、burn rate alert）
  4. Instrumentation 計畫：每 service 的 RED + 業務 metric + trace + log
  5. Cost 估算：每月 storage / network / cardinality 預估
  6. Roadmap：3 個月 / 6 個月 / 12 個月分階段實作
  7. Review 流程：weekly / monthly / quarterly review 定義
