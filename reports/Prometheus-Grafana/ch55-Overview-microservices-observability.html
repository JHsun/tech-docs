§ 3. RED + USE 設計

3.1 RED for 每個 service

每個 service 必有的 3 個 metric：

RED metricPrometheus name
Ratehttp_requests_total（OTel: http_server_request_duration_seconds_count）
Errorshttp_requests_total{status=~"5.."}
Durationhttp_request_duration_seconds_bucket（histogram）

3.2 USE for 資源

USE method 給資源用：

USE metric意義
Utilization資源使用率（CPU / memory / disk）
Saturation飽和（queue 深度、wait time）
Errors錯誤事件（OOM、disk full）

3.3 對應 metric 範例

# DB 連線池 USE
db_connections_active / db_connections_max         # Utilization
db_connections_wait_total                          # Saturation
db_connection_errors_total                         # Errors

# Redis cache USE
redis_memory_used_bytes / redis_memory_max_bytes   # Utilization
redis_blocked_clients                              # Saturation
redis_rejected_connections_total                   # Errors

# Kafka consumer USE
kafka_consumergroup_lag                           # Saturation
kafka_consumer_errors_total                        # Errors

3.4 Four Golden Signals（Google SRE）

Google SRE Book 提出 4 個必備 signal，跟 RED 高度重疊：

  • Latency（=Duration）
  • Traffic（=Rate）
  • Errors（=Errors）
  • Saturation（多了一個資源視角）

實務上：對 service 用 RED，對資源用 USE，對整體 SLO 用 4 Golden。

§ 4. Service-level instrumentation

4.1 Go service（catalog-service）

// main.go
import (
    "go.opentelemetry.io/otel"
    "go.opentelemetry.io/otel/exporters/otlp/otlptrace/otlptracegrpc"
    "go.opentelemetry.io/otel/sdk/trace"
    "go.opentelemetry.io/contrib/instrumentation/net/http/otelhttp"
    "github.com/grafana/pyroscope-go"
)

func main() {
    // OTel tracing
    exporter, _ := otlptracegrpc.New(ctx)
    tp := trace.NewTracerProvider(
        trace.WithBatcher(exporter),
        trace.WithResource(resource.NewWithAttributes(
            semconv.SchemaURL,
            semconv.ServiceNameKey.String("catalog-service"),
            semconv.ServiceVersionKey.String("1.5.0"),
        )),
    )
    otel.SetTracerProvider(tp)

    // Pyroscope
    pyroscope.Start(pyroscope.Config{
        ApplicationName: "catalog-service",
        ServerAddress:   "http://pyroscope:4040",
        ProfileTypes: []pyroscope.ProfileType{
            pyroscope.ProfileCPU,
            pyroscope.ProfileAllocSpace,
        },
    })

    // HTTP handler 自動 trace
    handler := otelhttp.NewHandler(myHandler, "catalog-service")
    http.ListenAndServe(":8080", handler)
}

5.2 業務 metric 範例（payment-service）

// Java OTel Metrics API
import io.opentelemetry.api.metrics.Meter;

Meter meter = openTelemetry.getMeter("payment-service");

// Counter: 成功支付
LongCounter paymentSuccess = meter
    .counterBuilder("payment_success_total")
    .setDescription("Total successful payments")
    .build();

// Counter with attributes
DoubleHistogram paymentAmount = meter
    .histogramBuilder("payment_amount_dollars")
    .setUnit("USD")
    .build();

// 記錄
paymentSuccess.add(1, Attributes.of(
    AttributeKey.stringKey("currency"), "USD",
    AttributeKey.stringKey("method"), "credit_card",
    AttributeKey.stringKey("country"), "TW"
));

paymentAmount.record(amount, Attributes.of(
    AttributeKey.stringKey("currency"), "USD"
));

5.3 Cardinality 警告

⚠️ 業務 metric 也會 cardinality 爆
  • 不要把 user_id 當 attribute（每 user 一條 series）
  • 不要把 order_id 當 attribute
  • 不要把 email 當 attribute（PII + cardinality）
  • 用類別（country / payment_method / user_tier）即可

5.4 避免 cardinality 的技巧

# 不好：
http_requests_total{path="/api/users/12345"}    # 12345 是 user ID

# 好：
http_requests_total{route="/api/users/{id}"}    # template path

# 不好：
order_value{order_id="abc-123"}

# 好：
order_value{country="US", method="credit_card"}
# 用 histogram 看分布

§ 6. Distributed Tracing 實作

6.1 Trace 完整 user journey

當 user 結帳時，產生的 trace 應該長這樣：

checkout-trace [trace_id: abc123]
└── frontend.handleCheckout (200ms total) [span_id: 001]
    ├── auth.verify (10ms) [span_id: 002, parent: 001]
    │   └── postgres.query (8ms) [span_id: 003, parent: 002]
    ├── cart.getCart (50ms) [span_id: 004, parent: 001]
    │   └── redis.get (45ms) [span_id: 005, parent: 004]
    ├── inventory.check (30ms) [span_id: 006, parent: 001]
    │   └── postgres.query (28ms) [span_id: 007, parent: 006]
    ├── payment.charge (90ms) [span_id: 008, parent: 001]
    │   ├── stripe.api.call (80ms) [span_id: 009, parent: 008]
    │   └── postgres.query (8ms) [span_id: 010, parent: 008]
    ├── order.create (40ms) [span_id: 011, parent: 001]
    │   ├── postgres.insert (35ms) [span_id: 012, parent: 011]
    │   └── kafka.produce (3ms) [span_id: 013, parent: 011]
    └── notification.send_async (5ms) [span_id: 014, parent: 001]
        └── kafka.consume + email_send (handled by notification-service)

6.2 Span Attributes 設計

每個 span 要有有用的 attribute：

// Standard semantic conventions
http.method="POST"
http.route="/api/checkout"
http.status_code=200
http.client.ip="1.2.3.4"
db.system="postgresql"
db.statement="SELECT ..." (sanitized!)
db.operation="SELECT"

// 業務 attribute
order.id="abc-123"          // 用於 search trace（OK in trace, NOT OK in metric label）
user.tier="premium"
payment.method="credit_card"
payment.amount=99.99
payment.currency="USD"

6.3 Trace 與業務 ID 連結

關鍵設計：把業務 ID（order_id / user_id）放在 span attribute，不放在 metric label。

  • Metric 用 cardinality-safe label（country / status）
  • Trace attribute 可放 high-cardinality（trace 是 sample 的）
  • 用業務 ID 從 log → trace → 找完整 flow

6.4 Tail-based Sampling

# Alloy Gateway sampling
otelcol.processor.tail_sampling "default" {
  decision_wait = "10s"
  num_traces = 50000

  policy {
    name = "errors"
    type = "status_code"
    status_code {
      status_codes = ["ERROR"]
    }
  }

  policy {
    name = "slow"
    type = "latency"
    latency {
      threshold_ms = 1000
    }
  }

  policy {
    name = "high_value_orders"
    type = "string_attribute"
    string_attribute {
      key = "user.tier"
      values = ["premium", "vip"]
    }
  }

  policy {
    name = "random"
    type = "probabilistic"
    probabilistic {
      sampling_percentage = 5
    }
  }

  output {
    traces = [otelcol.exporter.otlp.tempo.input]
  }
}

§ 9. Alert 設計

9.1 Alert 階層

Severity反應範例
Critical立即 PagerDuty / 半夜叫人結帳完全壞掉、DB down
WarningSlack 上班時間處理Latency 超標但服務在
Info記錄不通知Deployment 完成

9.2 Symptom-based vs Cause-based

規則：對使用者影響的事件 alert，內部原因不 alert。

  • ✅ 「結帳失敗率 > 5%」（symptom，user 受影響）
  • ❌ 「DB CPU 80%」（cause，可能根本沒人受影響）

Cause-based 留給 dashboard，不上 alert。

9.3 完整 PrometheusRule 範例

apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: ecommerce-alerts
spec:
  groups:
    - name: critical-user-facing
      rules:
        - alert: CheckoutFailureRate
          expr: |
            sum(rate(http_requests_total{service="payment-service",status=~"5.."}[5m]))
            /
            sum(rate(http_requests_total{service="payment-service"}[5m]))
            > 0.05
          for: 5m
          labels:
            severity: critical
            team: platform
            page: "true"
          annotations:
            summary: "Checkout failure rate > 5%"
            runbook_url: "https://wiki/runbook/checkout-failure"
            dashboard_url: "https://grafana/d/checkout"

        - alert: OrdersStopped
          expr: |
            absent(rate(orders_created_total[5m]))
            or rate(orders_created_total[5m]) == 0
          for: 5m
          labels:
            severity: critical
            page: "true"

    - name: warning-degraded
      rules:
        - alert: HighLatency
          expr: |
            histogram_quantile(0.99,
              sum by (service, le) (rate(http_request_duration_seconds_bucket[5m]))
            ) > 1
          for: 10m
          labels:
            severity: warning

        - alert: ErrorRateElevated
          expr: |
            sum by (service) (rate(http_requests_total{status=~"5.."}[5m]))
            /
            sum by (service) (rate(http_requests_total[5m]))
            > 0.01
          for: 10m
          labels:
            severity: warning

route:

§ 10. 完整 incident playbook

10.1 場景：「結帳失敗率飆高」

Step 1: PagerDuty 響

Alert: CheckoutFailureRate > 5% for 5 minutes

Step 2: 開 Grafana dashboard

從 alert 的 dashboard_url 跳到 checkout dashboard：

  • RED panel 看到 5xx error 飆 12%
  • Latency P99 從 200ms 飆到 1500ms
  • QPS 正常（不是流量問題）

Step 3: 用 Drilldown 探索

進 Metrics Drilldown：

  • 看 payment-service：error rate 飆
  • 看 stripe.api.call latency：飆 4x
  • 初步診斷：Stripe 慢

Step 4: 看 Trace 確認

進 Traces Drilldown，filter service=payment-service status=error：

  • 看到很多 trace 在 stripe.api.call 卡住 20s timeout
  • span 顯示 Stripe API 回 503
  • 確認：Stripe outage

Step 5: 看 Log 找 root cause

進 Logs Explorer，filter service=payment-service level=ERROR：

{"level":"ERROR","msg":"Stripe API error","status":503,"error":"service_unavailable","trace_id":"abc..."}
  • error 訊息明確指 Stripe 503
  • 每分鐘 50+ 條同樣 error

Step 6: 確認外部狀態

查 Stripe status page：確認 Stripe ongoing incident。

Step 7: 緩解

  • 啟動 fallback：把 payment 切到第二個 PSP
  • UI 顯示「目前支付暫時不便，可改用 PayPal」
  • 降低 Stripe API timeout 從 20s 到 5s（避免 thread block）

Step 8: Postmortem

事後檢討：

  • 為什麼 fallback 沒自動觸發？
  • SLO budget burn 多少？
  • Action items：實作 circuit breaker、加 fallback 自動化

10.2 整段時間用了

  • Alert：通知
  • Dashboard：總覽
  • Metrics Drilldown：找症狀（哪個 service / endpoint）
  • Traces Drilldown：找 root cause（哪個外部 dependency）
  • Logs：確認 error 細節
  • 外部 status：最終確認

整個過程約 5-10 分鐘從 alert 到緩解，這是 observability 的價值。

  • Trace 要含完整 user journey，business ID 放 span attribute（trace OK，metric 不行）
  • Tail-based sampling：error 與 slow 100% 留 + premium user 100% 留 + 其他 5%
  • Log 用結構化 JSON，trace_id 自動注入（OTel logging instrumentation）
  • Loki derivedFields 讓 trace_id 變 link 跳 Tempo
  • Dashboard 4 階層：Executive Summary / Service-level / Resource / Drilldown
  • Alert 鐵則：對 user 影響的 symptom alert，內部 cause 不 alert
  • Alert 分 critical / warning，routing 用 severity + team
  • 完整 incident playbook：Alert → Dashboard → Drilldown → Trace → Log → Mitigate
  • 5-10 分鐘從 alert 到 root cause 是好觀測性的指標
  • Postmortem 找 action items 改進系統

📝 練習題

  1. 解釋 RED / USE / 4 Golden Signals 的差異與適用場景。
  2. 設計 cart-service 的 metric：5 個 RED + 業務 metric，含 cardinality 控制。
  3. 實作 Alloy Gateway tail-based sampling：error 100% + slow (> 1s) 100% + premium user 100% + 其他 3%。
  4. 寫 incident playbook：「inventory-service P99 latency 飆」，從 alert 到 root cause 完整流程，列要看的 dashboard / drilldown / log query。
  5. 設計完整 alert：3 個 critical + 5 個 warning，含 routing rules 與 mute timing。
