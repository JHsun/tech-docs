Chapter 19 · 寫自己的 Exporter

寫自己的 Exporter：模式、實作、踩雷

當沒有現成 exporter 時，您需要自己寫一個。本章涵蓋官方推薦的兩種設計模式 （Custom Collector vs Direct Instrumentation）、Multi-Target Pattern 實作、 Cardinality 控制、以及 Prometheus 維護者親自整理的 best practices。

§ 1. 何時該寫自己的 Exporter

寫 exporter 是有成本的（持續維護、跟上 client library 更新、處理 edge cases）。先確認以下情況：

✅ 該寫的情況
  1. 系統沒有現成 exporter（搜過 awesome-prometheus、官方清單、GitHub）
  2. 有現成但不維護了（commits 半年沒更新）
  3. 系統有 API 但格式特殊，需要解析
  4. 內部自有系統，外部不會有 exporter
⚠️ 不該寫的情況
  1. 應用是您自己寫的 → 直接用 client library 在應用裡埋 metric（Ch 20），不要寫獨立 exporter
  2. 已有現成 exporter，只是某些 metric 不滿意 → 提 PR，不要重造輪子
  3. 需求是一次性的 → 寫個 cron 用 textfile collector 就好

§ 2. 兩種實作模式：Direct vs Custom Collector

2.1 Direct Instrumentation

Exporter 啟動時建立 metric 物件，主動更新它們的值（如 counter.inc()）。 當 Prometheus 抓 /metrics 時，就把當前值序列化吐出。

適用：事件驅動、自己的應用、當下發生時就更新。

# Pseudocode
counter = Counter('http_requests_total', ...)
gauge = Gauge('queue_size', ...)

def on_request(req):
    counter.inc()           # 事件發生時增加

def on_queue_change(size):
    gauge.set(size)         # 狀態改變時更新

# Prometheus scrape /metrics 時，直接讀目前值

2.2 Custom Collector

Exporter 不持有 metric 物件，而是在每次 scrape 時： 呼叫目標系統 API、把回傳資料即時轉成 Prometheus metric、回應給 Prometheus。

適用：查詢驅動、第三方系統、scrape 時才即時抓。

# Pseudocode
class MyCollector:
    def collect(self):
        # 每次 Prometheus scrape 都會呼叫這個
        data = call_third_party_api()
        for item in data:
            yield Metric('foo_value', item.value, labels={...})

2.3 兩者比較

面向Direct InstrumentationCustom Collector
適用場景自己的應用、事件驅動第三方系統、查詢驅動
資料更新時機事件發生時Scrape 觸發時
內部狀態持有 counter/gauge 物件無持久狀態
Counter 累積自動處理需自己確保 monotonic
實作難度低較高
典型例應用程式埋點blackbox、mysqld、snmp
🚨 寫第三方 exporter 一定要用 Custom Collector

如果您寫的是「監控別人的系統」的 exporter（像 mysqld、redis），必須用 custom collector。
若用 direct instrumentation，會有致命問題：被監控系統重啟時，您的 exporter 還持有舊的 counter 累積值， 導致 rate() 看不到 reset。

§ 3. 範例：Direct Instrumentation（Python）

最簡單的場景：給自己寫的 Flask 應用加上 metric。

# app.py
from flask import Flask, request
from prometheus_client import Counter, Histogram, generate_latest, CONTENT_TYPE_LATEST
import time

app = Flask(__name__)

# 1. 定義 metric
REQUESTS_TOTAL = Counter(
    'app_http_requests_total',
    'Total HTTP requests',
    ['method', 'endpoint', 'status']
)

REQUEST_DURATION = Histogram(
    'app_http_request_duration_seconds',
    'Request duration',
    ['method', 'endpoint'],
    buckets=[0.005, 0.01, 0.05, 0.1, 0.5, 1, 5]
)

# 2. 中介層：每個請求結束後更新 metric
@app.before_request
def start_timer():
    request._start_time = time.time()

@app.after_request
def record_request(response):
    duration = time.time() - request._start_time
    REQUESTS_TOTAL.labels(
        method=request.method,
        endpoint=request.path,
        status=response.status_code
    ).inc()
    REQUEST_DURATION.labels(
        method=request.method,
        endpoint=request.path
    ).observe(duration)
    return response

# 3. 暴露 /metrics endpoint
@app.route('/metrics')
def metrics():
    return generate_latest(), 200, {'Content-Type': CONTENT_TYPE_LATEST}

@app.route('/api/users')
def users():
    return {"users": []}

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=8080)

啟動後 curl http://localhost:8080/metrics 就能看到 metric。

§ 4. 範例：Custom Collector（Go）

場景：監控一個內部系統的 REST API（提供 /api/stats 回傳 JSON 統計）。

package main

import (
	"encoding/json"
	"net/http"
	"time"

	"github.com/prometheus/client_golang/prometheus"
	"github.com/prometheus/client_golang/prometheus/promhttp"
)

// 1. 定義 metric descriptor（不是 metric 物件）
var (
	queueSize = prometheus.NewDesc(
		"myapp_queue_size",
		"Current queue size by queue name",
		[]string{"queue"},   // 變數 label
		nil,                 // 常數 label
	)

	processedTotal = prometheus.NewDesc(
		"myapp_processed_total",
		"Total processed items by queue",
		[]string{"queue"},
		nil,
	)
)

// 2. 實作 Collector interface
type myCollector struct {
	apiURL string
}

func (c *myCollector) Describe(ch chan<- *prometheus.Desc) {
	ch <- queueSize
	ch <- processedTotal
}

// Collect 在每次 Prometheus scrape 時被呼叫
func (c *myCollector) Collect(ch chan<- prometheus.Metric) {
	// 呼叫第三方 API 取得即時資料
	resp, err := http.Get(c.apiURL + "/api/stats")
	if err != nil {
		// 若呼叫失敗，不要 panic，靜默回傳沒有 metric
		// up 會自動由 Prometheus 寫成 0
		return
	}
	defer resp.Body.Close()

	var stats struct {
		Queues []struct {
			Name      string `json:"name"`
			Size      int    `json:"size"`
			Processed int    `json:"processed_total"`
		} `json:"queues"`
	}
	if err := json.NewDecoder(resp.Body).Decode(&stats); err != nil {
		return
	}

	// 把資料轉成 metric
	for _, q := range stats.Queues {
		ch <- prometheus.MustNewConstMetric(
			queueSize,
			prometheus.GaugeValue,
			float64(q.Size),
			q.Name,
		)
		ch <- prometheus.MustNewConstMetric(
			processedTotal,
			prometheus.CounterValue,    // 注意：這個是 counter
			float64(q.Processed),
			q.Name,
		)
	}
}

func main() {
	// 註冊 collector
	collector := &myCollector{apiURL: "http://internal-system:8080"}
	prometheus.MustRegister(collector)

	// 暴露 /metrics
	http.Handle("/metrics", promhttp.Handler())
	http.ListenAndServe(":9200", nil)
}

4.1 Custom Collector 三大要點

  • 用 NewDesc 而不是 NewCounter/NewGauge：因為您不在自己持有狀態
  • 每次 Collect 都即時去抓：不要 cache 太久（資料會過期）
  • 失敗時靜默回傳：不要因為單次 API 失敗讓整個 exporter crash

§ 5. 範例：Multi-Target Pattern（Go）

更進階的場景：類似 blackbox_exporter，target 透過 URL 參數傳入。

package main

import (
	"net/http"

	"github.com/prometheus/client_golang/prometheus"
	"github.com/prometheus/client_golang/prometheus/promhttp"
)

// 對「特定 target」做監控的 collector
func probeHandler(w http.ResponseWriter, r *http.Request) {
	target := r.URL.Query().Get("target")
	if target == "" {
		http.Error(w, "Missing target parameter", http.StatusBadRequest)
		return
	}

	// 為這次 scrape 建立 registry
	registry := prometheus.NewRegistry()

	probeSuccess := prometheus.NewGauge(prometheus.GaugeOpts{
		Name: "probe_success",
		Help: "1 if probe succeeded, 0 otherwise",
	})
	probeDuration := prometheus.NewGauge(prometheus.GaugeOpts{
		Name: "probe_duration_seconds",
	})

	registry.MustRegister(probeSuccess)
	registry.MustRegister(probeDuration)

	// 對 target 做檢查
	success, duration := doMyCheck(target)
	if success {
		probeSuccess.Set(1)
	} else {
		probeSuccess.Set(0)
	}
	probeDuration.Set(duration)

	// 用這次的 registry 回應
	h := promhttp.HandlerFor(registry, promhttp.HandlerOpts{})
	h.ServeHTTP(w, r)
}

func main() {
	// 兩個 endpoint
	http.HandleFunc("/probe", probeHandler)              // 對特定 target 探測
	http.Handle("/metrics", promhttp.Handler())          // exporter 自身 metric
	http.ListenAndServe(":9300", nil)
}

5.1 Multi-Target 的兩個 endpoint

Endpoint用途
/probe?target=...對特定 target 做檢查，每次新建 registry
/metricsexporter 自身運作 metric（probe 次數、錯誤率等）

這個架構讓 Prometheus 透過 relabeling 動態傳入 target，一個 exporter 服務上千個目標。

§ 6. 命名與 Label 規範

官方 best practices（Prometheus 維護者 Brian Brazil 親自整理）：

6.1 Metric 命名

✅ Best Practices
  1. 用 namespace 前綴：myapp_ 而不是 requests_
  2. 用基本單位後綴：_seconds 不是 _milliseconds；_bytes 不是 _kb
  3. Counter 加 _total
  4. Snake_case：http_request_duration_seconds
  5. Info metric 用 _info 後綴：node_uname_info

6.2 Label 命名

  • 用 snake_case
  • 不要用「重複 metric 名」的 label（http_requests{type="http"} 是冗餘）
  • 避免高基數 label
  • 同一概念在不同 metric 用相同 label 名（例：所有 metric 都用 method，不要混 http_method）

6.3 慣例：用 ConstLabels

對於「整個 exporter 共用」的常數 label（例：版本號），用 ConstLabels 一次設好：

var requestsTotal = prometheus.NewCounterVec(
    prometheus.CounterOpts{
        Name: "myapp_requests_total",
        ConstLabels: prometheus.Labels{
            "version": "1.2.3",
            "build":   "abc123",
        },
    },
    []string{"method"},
)

§ 7. Exporter 自身的 Metric

Exporter 應該暴露自身運作狀態，方便監控 exporter 自己。常見做法：

// 整體 scrape 計數
myExporterScrapesTotal = prometheus.NewCounter(
    prometheus.CounterOpts{
        Name: "myexporter_scrapes_total",
        Help: "Total scrapes against the exporter.",
    },
)

// 失敗計數
myExporterScrapeErrorsTotal = prometheus.NewCounterVec(
    prometheus.CounterOpts{
        Name: "myexporter_scrape_errors_total",
    },
    []string{"collector"},
)

// 每個 scrape 耗時（gauge，不是 histogram）
myExporterScrapeDuration = prometheus.NewGauge(
    prometheus.GaugeOpts{
        Name: "myexporter_scrape_duration_seconds",
        Help: "Time taken to scrape target.",
    },
)

7.1 重要：scrape 元 metric 不要當 histogram

⚠️ Brian Brazil 的建議

像「scrape 耗時」這類 meta metric，用 gauge 暴露最後一次值就好， 不要做成 histogram 或 counter。Prometheus 自身已經有 scrape_duration_seconds， 重複會造成冗餘。

§ 8. Cardinality 控制

這是寫 exporter 最常踩的雷。

🚨 永遠記住：每個 unique label 組合 = 一條 series

如果您把 user_id（百萬可能值）放進 label，您的 exporter 會在幾分鐘內讓 Prometheus 爆炸。

8.1 高基數陷阱

  • 不要把 user_id、session_id、request_id、UUID 放進 label
  • 不要把任意的 query string、URL path 放進 label
  • 不要把 timestamp、numeric ID 當 label 值
  • 動態增長的 enum（例：DB 表名、queue 名）要設上限或用 allowlist

8.2 路徑正規化範例

HTTP path 是常見高基數來源。要正規化：

# Bad（每個 ID 一條 series）
/users/123
/users/456
/users/789

# Good（正規化成 :id）
/users/:id

# 在 instrument 時就要做：
def normalize_path(path):
    # /users/123 -> /users/:id
    return re.sub(r'/\d+', '/:id', path)

REQUESTS.labels(path=normalize_path(request.path)).inc()

8.3 設上限機制

# Pseudocode：超過 N 個不同值就用 "other" 取代
class CardinalityLimiter:
    MAX_LABELS = 100

    def __init__(self):
        self.seen = set()

    def get(self, value):
        if value in self.seen:
            return value
        if len(self.seen) >= self.MAX_LABELS:
            return "other"
        self.seen.add(value)
        return value

§ 9. 安全性考量

9.1 不要在 metric 中洩漏敏感資料

  • API token、密碼、PII 個資絕對不能當 label 值
  • 客戶 email、IP 等若必要，先 hash 處理
  • /metrics endpoint 應該被視為「半敏感」資料來源（誰能看就知道您系統的內部狀態）

9.2 認證與 TLS

v2.24+ 起 Prometheus 與所有官方 exporter 支援 TLS + basic auth。設定檔：

# web-config.yml（給您的 exporter 啟動用）
tls_server_config:
  cert_file: /etc/exporter/cert.pem
  key_file: /etc/exporter/key.pem

basic_auth_users:
  prom: $2y$05$...   # bcrypt hash

./your_exporter --web.config.file=/etc/exporter/web-config.yml

9.3 Resource 限制

  • 設定 timeout：每次抓第三方 API 要有上限
  • 限制並發：一次 scrape 的最大查詢數
  • 記憶體上限：使用語言層級的 limit / K8s resource limit
  • Rate limit：避免突然湧入太多 scrape 請求

§ 10. 上 production 前 checklist

✅ Production-ready Checklist
  1. ☐ Metric 命名遵守 best practices（namespace、單位、_total）
  2. ☐ 沒有高基數 label（user_id 等）
  3. ☐ 失敗時不會 crash，靜默回傳
  4. ☐ 有 timeout 機制（避免 hang）
  5. ☐ 暴露自身運作 metric（scrape 次數、錯誤）
  6. ☐ 用 custom collector（如果是第三方系統）
  7. ☐ 有 /metrics 也有 /-/healthy endpoint
  8. ☐ Resource limit 設好（timeout、並發）
  9. ☐ 不洩漏 secrets（不在 label / log / metric）
  10. ☐ 啟用 TLS + basic auth（公開或多租戶環境）
  11. ☐ 寫了 README 與範例 prometheus.yml
  12. ☐ 寫了單元測試
  13. ☐ 用 promtool check 驗證輸出格式
  14. ☐ 在 staging 環境跑過 24 小時無問題

10.1 用 promtool 驗證 exporter 輸出

# 把 exporter 的 /metrics 輸出存檔
curl -s http://localhost:9200/metrics > metrics.txt

# 驗證格式
promtool check metrics < metrics.txt

# 應該看到：
# (no errors found)

10.2 觀察生產上的 cardinality

# 您 exporter 產生多少 series（給定一個 instance）
count by (__name__)({job="my-exporter", instance="exporter-1:9200"})

# Top 10 series 數最多的 metric
topk(10, count by (__name__)({job="my-exporter"}))
