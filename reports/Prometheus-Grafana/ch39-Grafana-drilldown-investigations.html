Chapter 39 · Drilldown Apps + Investigations

Drilldown：Queryless 探索 Metrics、Logs、Traces、Profiles

傳統 Explore 需要寫 PromQL / LogQL / TraceQL。Drilldown apps 是新一代「queryless」探索介面—— 點擊式找到想要的資料。本章涵蓋 Metrics / Logs / Traces / Profiles 四個 Drilldown app 的完整使用， 以及 v12+ 引入的 Investigations 跨 signal 串接工具。

§ 1. Drilldown 概念與演進

1.1 為什麼要 Drilldown

傳統 Grafana 探索需要：

  • 知道 metric / log / trace 的確切名稱
  • 會寫 PromQL / LogQL / TraceQL 語法
  • 知道有哪些 label 可以 filter

對 SRE / 資深工程師沒問題，但對 PM、新人、業務團隊太難。 Drilldown apps 提供「不用寫 query」的探索介面：點擊 + filter + 自動生成圖。

1.2 從 Explore Apps 改名為 Drilldown

📌 名稱變化

原名 "Explore Apps"（v10-v11 早期），2025 年起正式改名 Drilldown：
• Explore Metrics → Metrics Drilldown
• Explore Logs → Logs Drilldown
• Explore Traces → Traces Drilldown
• Explore Profiles → Profiles Drilldown
文件、blog、選單都會看到兩種名稱混用，但是同一個工具。

1.3 GA 狀態

AppGA 時間v12 內建
Metrics Drilldown2024-10（ObsCon）✅
Logs Drilldown2024-10（ObsCon）✅
Traces Drilldown2025-04（GrafanaCON）✅
Profiles Drilldown2025-Q3✅

v12+ 預設安裝，無需額外操作。v11 需手動裝 plugin。

§ 2. Metrics Drilldown

2.1 入口

左側 sidebar → Drilldown → Metrics（或舊名 "Explore metrics"）。

2.2 主介面

三大區塊：

  1. Datasource selector：選 Prometheus / Mimir 等
  2. Filter sidebar：左邊欄，按 label 過濾
  3. Metric grid：自動產生所有 metric 的小 panel

2.3 進階 Filter（v12.4+）

左側 filter 不只是 label values，還有：

  • Prefix filter：metric name 前綴（node_* / http_*）
  • Suffix filter：後綴（*_total / *_seconds_bucket）
  • Group by labels：把相關 metric 群組顯示

2.4 點選一個 metric → Breakdown

點擊 grid 中任一 metric，進入 detailed view：

Tab內容
Breakdown 每個 label-value pair 一張小圖（by status、by method 各一張）
Related metrics 名稱相關的其他 metric（http_* 系列）
Logs 對應這個 metric 的 logs（要設 Loki datasource correlation）

2.5 Sort 排序

v12+ 支援按以下排序：

  • Recent history（最近活躍）
  • Associated dashboards（有多少 dashboard 用到）
  • Related alerts（有多少 alert）
  • Cardinality（series 數量）

「找出最熱的 metric」/「找出 dashboard 最常用的 metric」很方便。

2.6 從 Breakdown 加到 Filter

在 Breakdown 看到「status=500 那條飆高」，點 "Add to filter"：

原本：rate(http_requests_total[5m]) by (status)

加上 status=500 filter →

新查詢：rate(http_requests_total{status="500"}[5m])
       並回到 metric grid 重新顯示

這個漸進式的 filter 機制是 Drilldown 的核心。

2.7 從 Drilldown 跳到 Explore / Dashboard

找到想要的 metric 後，右上角：

  • Open in Explore：把當前 query 帶到 Explore，可進一步改 PromQL
  • Add to dashboard：直接加成 panel
  • Add to investigation（v12+）：加到調查紀錄

§ 3. Logs Drilldown

3.1 適用情境

  • 「我的 service 最近有 error log 嗎」
  • 不知道 LogQL，但想看 log volume 異常
  • 找出錯誤 pattern（自動偵測）

3.2 主介面

左側 sidebar → Drilldown → Logs。

頁面分區：

  1. Service selection：列出所有 service（從 service label / app label）
  2. Volume chart：每個 service 的 log volume 趨勢
  3. Pattern detection：自動找出常見的 log pattern

3.3 Workflow

  1. 選 Loki datasource
  2. 選一個 service（或不選看全體）
  3. 看 volume 是否異常飆升
  4. 點 service 進入 detailed view
  5. 看 Breakdown by label（level / instance / container）
  6. 看 Patterns（自動分群常見 log）
  7. 看 Fields（從 log 解析的欄位分布）
  8. 找到問題 → Open in Explore 寫精細 LogQL

3.4 Pattern Detection（強大功能）

Logs Drilldown 自動偵測 log pattern。例： 

原始 log（萬筆）：
"GET /api/users/12345 200 45ms"
"GET /api/users/67890 200 67ms"
"GET /api/users/abc123 500 234ms"
...

Pattern detection 自動歸類：
Pattern 1: "GET /api/users/<*> 200 ms"   占 95% (firing 9500 logs)
Pattern 2: "GET /api/users/<*> 500 ms"   占 3% (firing 300 logs)
Pattern 3: "Error: ..."                       占 2%

不用寫 regex 就找出 log 形態，找錯誤模式神器。

3.5 Multi-include Filter（v12.4+）

v12.4 支援多個 include / exclude filter：

Include: namespace = production
Include: app =~ "api|web"
Exclude: pod =~ ".*-test-.*"

3.6 JSON Visualization

JSON 結構化 log 在 panel 中以可展開樹狀顯示，比 LogQL `| json` 後再看還清楚。

§ 4. Traces Drilldown

4.1 與 Tempo 的關係

⚠️ 必要條件

Traces Drilldown 需要 Tempo 2.6+，並且 Tempo 啟用 TraceQL metrics generator。
否則沒有 RED metrics 來自動生成圖表。

4.2 RED 自動展示

Traces Drilldown 進入後，自動顯示每個 service 的：

  • Rate：每秒 trace 數
  • Errors：錯誤 trace 比率
  • Duration：latency 分布

不需要寫 TraceQL，這些 RED 指標由 Tempo metrics generator 自動算，寫回 Prometheus。

4.3 主介面流程

  1. 左側選 Tempo datasource
  2. 看每個 service 的 RED 卡片
  3. 找到問題 service（紅色錯誤率）
  4. 點選 → 進入 service detail
  5. 看 spans breakdown（按 operation、status_code、http_route 分群）
  6. 找出問題 endpoint 或操作
  7. 點選查看具體 trace
  8. 看 trace 樹狀圖（spans + duration breakdown）

4.4 Trace Drawer（v2025-04+）

GA 版改進：trace 不再是「全頁開啟」，而是「抽屜（drawer）」滑出， 保留原來的列表 context。一邊看 trace 一邊看其他 trace 比較方便。

4.5 Exemplar 整合

Traces Drilldown 顯示的 RED metric 上有 exemplar 點：

  • P99 latency line 上的小圓點 = 代表 trace
  • 點選 exemplar → 直接開該 trace（不用先 query）

4.6 Streaming Results

TraceQL 查詢時，partial results 即時顯示，不用等查完。 對於跨 1000s of trace 的搜尋體驗大幅改善。

§ 5. Profiles Drilldown

5.1 與 Pyroscope 的關係

Profiles Drilldown 需要 Pyroscope datasource。 Pyroscope 是 Grafana Labs 的 continuous profiling backend（取自 2023 收購）。

5.2 適用情境

  • 「為什麼 service 突然吃 80% CPU」
  • 「Memory leak 在哪個函式」
  • 沒寫過 flame graph 也能用

5.3 主介面

  1. 左側 sidebar → Drilldown → Profiles
  2. 選 Pyroscope datasource
  3. 選 service / profile type（CPU / memory / goroutine 等）
  4. 自動產生 flame graph
  5. 點選函式進一步 drilldown

5.4 Profile Types

Type看出
cpu哪個函式吃 CPU
memory:alloc_objects分配最多物件的函式
memory:inuse_space當前記憶體佔用最多的函式
goroutinegoroutine 數量（Go 專屬）
blockblocking call 的函式（IO / mutex）

5.5 Flame Graph 解讀

Flame graph 是金字塔狀的視覺化：

  • X 軸：CPU 時間佔比（不是時間順序）
  • Y 軸：呼叫堆疊深度
  • 寬度：該函式在 CPU 上花的時間 ∝ 寬度
  • 看 wide 的 leaf node = 是優化目標

§ 6. 四個 Drilldown 的 workflow 整合

6.1 跨 signal 跳轉

四個 Drilldown 互通：

從跳到用途
MetricsLogs看這個 service 的 log
MetricsTraces (via exemplar)看這次飆高的具體 trace
LogsTraces (via derived field)看 log 對應的 trace
TracesLogs看 span 期間的 log
TracesProfiles看 span 對應的 CPU / memory profile

6.2 整合需要的設定

底層仍是 Ch 34 介紹的：

  • Prometheus exemplars → trace
  • Loki derived fields → trace
  • Tempo tracesToLogs → Loki
  • Tempo tracesToProfiles → Pyroscope

設定好這些，Drilldown 自動利用。

§ 7. Investigations（v12+）

7.1 Investigation 是什麼

Grafana 12 引入的新功能：在 debug 過程中，把看到的 metric / log / trace 截圖都加進一個 "investigation"， 形成調查紀錄。可分享、評論、最後變成 post-mortem。

7.2 Workflow

  1. 遇到事件，按頂部「Start investigation」
  2. 命名 investigation（例：2026-05-02 API 5xx spike）
  3. 瀏覽 dashboard / Explore / Drilldown，看到關鍵畫面就 "Add to investigation"
  4. 每個 add 自動截 query + time range，可加 comment
  5. 事件解決後，investigation 變成完整 timeline
  6. Share URL 給團隊 review，或匯出為 post-mortem 草稿

7.3 Investigation 的元素

  • Items：每個 add 來的 metric panel / log query / trace
  • Timeline：按時間排
  • Comments：每個 item 可加文字
  • Actions：share / export / archive

7.4 與其他工具的對比

工具定位
Grafana Investigation在 Grafana 內紀錄探索過程
incident.io / FireHydrant跨工具的 incident 流程
Notion / Confluence手動寫 post-mortem

三者互補：用 Investigation 收集證據 → 過程中 incident.io 跑 Slack 工作流 → 最後匯到 Notion 寫正式 post-mortem。

§ 8. Drilldown vs Explore vs Dashboard

三個工具看似重疊，定位其實清楚：

工具適用需要
Dashboard 「我已經知道要監看什麼」
固定的常規檢視		 建好的 dashboard
Drilldown 「我不知道從哪開始查」
新人 / 不寫 query 的角色		 不會 query 也能用
Explore 「我知道大概要查什麼，但要寫精細 query」
SRE / 進階 debug		 會 PromQL / LogQL / TraceQL

8.1 漸進使用流程

1. Dashboard 看：定期看 service health 看板

2. 發現異常 → Drilldown：
   點點看是哪個 service / endpoint / pattern 有問題

3. Drilldown 找到大方向 → Open in Explore：
   寫具體的 PromQL / LogQL 進一步切片

4. 想保留結果 → Add to dashboard / Investigation

§ 9. 實戰：用 Drilldown debug 一個 production 事件

9.1 場景

凌晨 3 點，PagerDuty 響：「API ErrorRate > 5%」。您打開 Grafana。

9.2 用 Drilldown 一連串行動

  1. Metrics Drilldown：選 Prometheus → Filter by label service=api → 看到 http_requests_total{status="500"} 飆 → 點選 → Breakdown by endpoint
  2. Breakdown 顯示 endpoint=/api/orders 占 90% errors → Add to filter endpoint=/api/orders
  3. 右上角 Open in Logs Drilldown → 自動帶 service=api filter → 看到 log volume 沒爆，但 error pattern 中有 90%「DB connection timeout」
  4. Traces Drilldown： 看 service=api 的 RED → 點 P99 上的 exemplar → 開啟一個 5xx trace → 看到 span tree 中 db.query 占了 95% 時間
  5. Trace span 上點 "Profile for this span" → Profiles Drilldown 開啟 → Flame graph 顯示 connectDB() 函式異常 wide
  6. 到此確認：DB connection pool 飽和。檢查 DB metric 確認連線數爆表。
  7. Start investigation → 把上述 metric / log / trace / profile 截圖都加入 → 命名「2026-05-02 API DB pool exhaustion」 → share 給 oncall 同事

整個過程沒寫一行 PromQL / LogQL / TraceQL。對深夜 oncall 是巨大解放。

§ 10. 安裝與設定

10.1 Grafana v12+

所有 Drilldown apps 內建，無需安裝。在 Apps → Drilldown 直接用。

10.2 Grafana v11 自架

# 安裝 Metrics Drilldown
grafana-cli plugins install grafana-metricsdrilldown-app

# Logs Drilldown
grafana-cli plugins install grafana-lokiexplore-app

# Traces Drilldown
grafana-cli plugins install grafana-exploretraces-app

# Profiles Drilldown
grafana-cli plugins install grafana-pyroscope-app

# 重啟 Grafana
sudo systemctl restart grafana-server

10.3 K8s Helm

# grafana values.yaml
plugins:
  - grafana-metricsdrilldown-app
  - grafana-lokiexplore-app
  - grafana-exploretraces-app
  - grafana-pyroscope-app

10.4 啟用 Tempo metrics generator（給 Traces Drilldown）

# tempo.yaml
metrics_generator:
  registry:
    external_labels:
      source: tempo
  storage:
    path: /tmp/tempo/generator/wal
    remote_write:
      - url: http://prometheus:9090/api/v1/write
        send_exemplars: true
  processor:
    span_metrics:
      dimensions:
        - http.method
        - http.status_code
    service_graphs:
      enabled: true

overrides:
  defaults:
    metrics_generator:
      processors:
        - service-graphs
        - span-metrics

✅ 本章重點摘要

30 秒回顧
  • Drilldown = queryless 探索（不用寫 PromQL/LogQL/TraceQL）
  • 原名 "Explore Apps"，2025 年改名 Drilldown
  • 四個 app 都已 GA：Metrics / Logs / Traces / Profiles，v12+ 內建
  • Metrics Drilldown：metric grid + Breakdown by label + Related metrics + 多種 sort（recent / dashboards / alerts）
  • Logs Drilldown：service selection + volume chart + 自動 pattern detection（不用 regex）
  • Traces Drilldown：自動 RED 卡片 + trace drawer + exemplar 整合 + streaming results。需 Tempo 2.6+ 啟用 metrics generator
  • Profiles Drilldown：選 service + profile type → 自動 flame graph，需 Pyroscope
  • 四個 Drilldown 互通：metric ↔ log ↔ trace ↔ profile（底層仍是 Ch 34 的 correlations）
  • Investigations（v12+）：邊 debug 邊收集證據，最後變成 post-mortem 草稿
  • 三工具定位：Dashboard 已知檢視 / Drilldown 不會 query / Explore 進階 debug
  • v11 自架要手動裝 4 個 plugin；v12+ 內建
  • Knowledge Graph + AI 助手是 Cloud 限定
