Chapter 08 · 配置深度

配置檔深度解析（prometheus.yml）

prometheus.yml 是 Prometheus 唯一的設定檔。它看似簡單， 實則包含全域設定、scrape 設定、規則檔引用、告警對接、遠端讀寫、 以及 v3 新增的 OTLP 設定等多個區塊。本章逐區塊深入解析每個欄位的意義與最佳實踐。

§ 1. 設定檔整體結構

prometheus.yml 的頂層 keys：

global:                 # 全域設定（scrape_interval、external_labels 等）
  ...

alerting:               # 告警對接（Alertmanager 連線）
  ...

rule_files:             # Recording / Alerting rules 的檔案路徑
  - "rules/*.yml"

scrape_configs:         # 抓取設定（最重要、最複雜）
  - job_name: ...
    ...

remote_write:           # 把資料推到遠端後端
  - url: ...

remote_read:            # 從遠端後端拉資料補查詢
  - url: ...

storage:                # TSDB 進階設定（v2.39+）
  ...

otlp:                   # OTLP receiver 的處理規則（v3.x）
  ...

tracing:                # Prometheus 自身的 OTLP tracing 輸出
  ...

scrape_config_files:    # 把 scrape_configs 拆到外部檔案（v2.43+）
  - "scrapes/*.yml"
ℹ️ YAML 注意事項
  • 所有時間單位接受 1ms、5s、1m、1h、1d、1w、1y
  • 所有大小單位接受 B、KB、MB、GB、TB、PB（1024 進位）
  • 支援環境變數展開：用 $(VAR) 語法（自 v2.36+，需 --enable-feature=expand-external-labels，但僅限 external_labels）
  • 布林值：true / false（不要寫 yes/no）

§ 2. global 區塊：全域預設值

global:
  # 預設抓取間隔（被個別 scrape_config 覆寫）
  scrape_interval: 15s

  # 預設 scrape 逾時
  scrape_timeout: 10s

  # 規則計算間隔（recording / alerting rules 的執行頻率）
  evaluation_interval: 15s

  # 抓取失敗的容忍時間（v2.49+）
  scrape_failure_log_file: /var/log/prometheus/scrape_failures.log

  # 全域 external_labels：本 Prometheus 對外送資料時自動加上
  # 這些 label 會出現在：alerts 送往 Alertmanager、remote_write 送出時
  external_labels:
    cluster: "production-tw"
    region: "ap-northeast-1"
    replica: "0"

  # 預設 query 逾時
  query_log_file: /var/log/prometheus/query.log

2.1 external_labels 的關鍵作用

external_labels 是「跨 Prometheus 識別」的關鍵。例如：

  • 當您的告警送往 Alertmanager，Alertmanager 看到的標籤裡會包含 cluster=production-tw，可用於 routing
  • 當您 remote_write 到 Mimir / Thanos，這些 label 會自動附加到所有 series，方便跨叢集查詢
  • HA 部署時，每個 replica 用不同的 replica label，後端可以做 deduplication

2.2 scrape_interval 的選擇

值適用場景權衡
5s關鍵服務、低延遲告警儲存量×3，CPU 高
15s大多數情境（業界預設）平衡點
30s低變化的基礎設施儲存省，但告警延遲變高
1m+非關鍵的容量規劃儲存最省

§ 3. scrape_configs 區塊

這是最重要也最複雜的區塊。一個典型的 scrape_config：

scrape_configs:
  - job_name: 'node-exporter'           # 必填，作為 job 標籤值

    # 抓取頻率（覆寫 global）
    scrape_interval: 30s
    scrape_timeout: 15s

    # HTTP 路徑與協定
    metrics_path: '/metrics'             # 預設值
    scheme: 'https'                      # 預設 http

    # 認證
    basic_auth:
      username: 'metrics'
      password_file: '/etc/prometheus/secrets/metrics-pass'

    # 或 bearer token
    # authorization:
    #   type: Bearer
    #   credentials_file: /var/run/secrets/token

    # TLS
    tls_config:
      ca_file: '/etc/prometheus/ca.pem'
      cert_file: '/etc/prometheus/client.pem'
      key_file: '/etc/prometheus/client-key.pem'
      insecure_skip_verify: false        # 嚴格驗證憑證

    # Targets 來源（多種 SD 方式擇一或混用）
    static_configs:
      - targets:
          - 'host1:9100'
          - 'host2:9100'
        labels:
          datacenter: 'tw-1'

    # Relabeling 規則（Ch 10 詳述）
    relabel_configs:
      - source_labels: [__address__]
        target_label: __metrics_path__
        # ...

    metric_relabel_configs:
      - source_labels: [__name__]
        regex: 'go_gc_.*'
        action: drop                      # 丟掉不要的指標

    # 樣本層級限制
    sample_limit: 0                       # 0 = 無上限。建議設為 10000-50000
    label_limit: 30                       # 單個 sample 最多 label 數
    label_name_length_limit: 200
    label_value_length_limit: 200
    target_limit: 0                       # 此 job 最多 target 數

    # 是否使用 target 自帶 timestamp
    honor_timestamps: true

    # 是否保留 target 自帶的 job/instance label
    honor_labels: false

    # Body 大小上限（v2.41+）
    body_size_limit: 10MB

3.1 sample_limit 的重要性

✅ 強烈建議設定 sample_limit

預設 sample_limit: 0 表示無上限，這是非常危險的。 若有應用因 bug 突然吐出百萬條 series，您的 Prometheus 會在幾分鐘內 OOM。
建議起始值：sample_limit: 10000（一般 web app）或 50000（資料庫類 exporter）。
被拒絕的 scrape 會以 up=0 與 scrape_samples_post_metric_relabeling 顯示，可立即發現問題。

3.2 honor_labels 的兩種情境

honor_labels行為適用
false（預設） 若 target 暴露的 metrics 自帶 job/instance，會被改名為 exported_job/exported_instance 大多數情境（避免衝突）
true 保留 target 的 job/instance，覆蓋 Prometheus 自動加上的 Pushgateway、聯邦（federation）

3.3 v2.43+ 的 scrape_config_files

當 scrape_configs 過於龐大時，可以拆到外部檔案：

# prometheus.yml
scrape_config_files:
  - "scrapes/applications/*.yml"
  - "scrapes/infrastructure/*.yml"

# scrapes/applications/web-services.yml
- job_name: 'web-frontend'
  static_configs:
    - targets: ['frontend-1:9100', 'frontend-2:9100']
- job_name: 'web-backend'
  static_configs:
    - targets: ['backend-1:9100']

§ 4. alerting 區塊：告警對接

alerting:
  # 對 alerts 套用 relabeling（在送往 Alertmanager 之前）
  alert_relabel_configs:
    # 移除某些不該外洩的 label
    - regex: 'replica'
      action: labeldrop

  alertmanagers:
    # 第一組（也可以多組做容錯）
    - scheme: http
      timeout: 10s
      api_version: v2                 # 必填，固定 v2

      # 多種 SD 方式
      static_configs:
        - targets:
            - 'alertmanager-1:9093'
            - 'alertmanager-2:9093'
            - 'alertmanager-3:9093'

      # 路徑前綴（若 AM 跑在 reverse proxy 後面）
      path_prefix: '/'

      # TLS 與認證（同 scrape_configs）
      tls_config:
        ca_file: '/etc/prometheus/am-ca.pem'

      basic_auth:
        username: 'prom'
        password_file: '/etc/prometheus/secrets/am-pass'
⚠️ Alertmanager HA 連線的關鍵

如果您部署多個 Alertmanager（HA），不要用 load balancer！ 應該把所有 AM 的位址都列在 targets 裡， Prometheus 會把同一個 alert 送給每一個 AM，由 AM 之間用 gossip 去重。 這是 Alertmanager HA 的標準做法。詳見 Ch 28。

§ 5. rule_files 區塊：規則檔

rule_files:
  - "rules/recording.yml"
  - "rules/alerts/*.yml"           # 支援 glob

規則檔本身的格式（範例）：

# rules/alerts/node.yml
groups:
  - name: node_health
    interval: 30s                  # 此 group 的執行間隔（覆寫 global）
    rules:
      - alert: NodeDown
        expr: up{job="node-exporter"} == 0
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Node {{ $labels.instance }} is down"
          description: "Node has been unreachable for 5 minutes."

      - record: instance:node_cpu_utilization:rate5m
        expr: |
          1 - avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m]))

規則的詳細寫法在 Ch 14（recording rules）與 Ch 15（alerting rules）。

§ 6. remote_write 與 remote_read

6.1 remote_write 基本用法

remote_write:
  - url: "https://mimir.example.com/api/v1/push"
    name: "mimir-prod"
    remote_timeout: 30s

    # Mimir 的 multi-tenancy
    headers:
      X-Scope-OrgID: "tenant-1"

    # 認證
    basic_auth:
      username: 'prom'
      password_file: '/etc/prometheus/secrets/mimir-pass'

    # 過濾要送的 metric（不送的留在本地）
    write_relabel_configs:
      - source_labels: [__name__]
        regex: 'go_(gc|memstats)_.*'
        action: drop

    # 佇列調校（重要！）
    queue_config:
      capacity: 10000              # 記憶體佇列大小
      max_shards: 30               # 最大分片數
      min_shards: 1
      max_samples_per_send: 2000   # 一次最多送多少 samples
      batch_send_deadline: 5s
      min_backoff: 30ms
      max_backoff: 5s
      retry_on_http_429: true      # 收到限流時重試

6.2 Remote Write 1.0 vs 2.0

面向RW 1.0RW 2.0（v2.50+ 實驗）
序列化Snappy + ProtobufSnappy + Protobuf（新 schema）
Native Histogram需 enable原生支援
Metadata（Help/Type）單獨送內嵌每筆 series
Created Timestamp不支援支援
Exemplars支援（v2.30+）原生支援
 
remote_write:
  - url: "..."
    protobuf_message: "io.prometheus.write.v2.Request"   # 啟用 RW 2.0

6.3 remote_read 用法

讓 Prometheus 在查詢時也能讀取遠端後端的歷史資料：

remote_read:
  - url: "https://mimir.example.com/prometheus/api/v1/read"
    name: "mimir-prod"
    read_recent: false             # true = 重複查（含本地有的資料），false = 只查本地沒有的時段
    headers:
      X-Scope-OrgID: "tenant-1"
ℹ️ remote_read 的實務替代

實務上很多人不用 remote_read，而是改在 Grafana 設定多個 datasource： 一個指向本地 Prometheus（最近資料）、一個指向 Mimir（歷史資料）， 由 dashboard 切換。或者直接全部查 Mimir，本地 Prometheus 只負責 ingestion。

§ 7. storage 區塊：TSDB 與 OOO 設定

storage:
  tsdb:
    # Out-of-order 樣本容忍時間（OTel 多副本必備）
    out_of_order_time_window: 30m

大多數 TSDB 設定都是透過 CLI flag 設的，storage 區塊主要用於 OOO。

常用 CLI flag（在啟動指令中設定）：

Flag說明
--storage.tsdb.pathTSDB 資料目錄
--storage.tsdb.retention.time保留時間（預設 15d）
--storage.tsdb.retention.size保留大小上限（達到先到的條件就裁切）
--storage.tsdb.min-block-duration最小 block 區間（預設 2h，遠端儲存可調更短）
--storage.tsdb.wal-compressionWAL 壓縮（v2.20+ 預設啟用）

§ 8. otlp 區塊（v3.x）

當您啟用 --web.enable-otlp-receiver，可在 yml 中調整轉換策略：

otlp:
  # 把 OTel resource attribute 升格為 Prometheus label
  promote_resource_attributes:
    - service.instance.id
    - service.name
    - service.namespace
    - service.version
    - cloud.availability_zone
    - cloud.region
    - deployment.environment.name
    - k8s.cluster.name
    - k8s.namespace.name
    - k8s.pod.name
    - k8s.deployment.name
    - k8s.replicaset.name
    - k8s.container.name

  # OTel metric 名稱翻譯策略
  translation_strategy: NoUTF8EscapingWithSuffixes
  # 可選值：
  # - UnderscoreEscapingWithSuffixes（預設，最相容）
  # - NoUTF8EscapingWithSuffixes（保留 UTF-8，但加單位後綴）
  # - NoTranslation（完全保留 OTel 名稱，需 UTF-8）

翻譯策略的差別（範例 http.server.duration）：

策略結果
UnderscoreEscapingWithSuffixeshttp_server_duration_seconds
NoUTF8EscapingWithSuffixeshttp.server.duration_seconds
NoTranslationhttp.server.duration

§ 9. tracing 與其他進階區塊

Prometheus 自身可以送 trace 給 OTLP backend（用於 self-observability）：

tracing:
  endpoint: "tempo:4317"
  client_type: grpc
  insecure: true
  sampling_fraction: 1.0
  headers:
    "x-tenant-id": "ops"

§ 10. 設定模板化與 secrets 管理

10.1 用 file 引用 secrets

避免把密碼直接寫在 yml：

basic_auth:
  username: 'prom'
  password_file: '/etc/prometheus/secrets/scrape-pass'   # 不要 password:

10.2 K8s 環境用 Secret 掛載

volumes:
  - name: secrets
    secret:
      secretName: prometheus-secrets
volumeMounts:
  - name: secrets
    mountPath: /etc/prometheus/secrets
    readOnly: true

10.3 Helm template 化

kube-prometheus-stack 的 values.yaml 把 prometheus.yml 的所有欄位都暴露為 Helm 參數，例如：

prometheus:
  prometheusSpec:
    externalLabels:
      cluster: "{{ .Values.global.clusterName }}"
    additionalScrapeConfigs:
      - job_name: 'custom-job'
        static_configs:
          - targets: ['app:9100']

10.4 環境變數展開

限制：原生只支援 external_labels 內的環境變數展開（且需 feature flag）：

prometheus --enable-feature=expand-external-labels
global:
  external_labels:
    replica: $(POD_NAME)
    cluster: $(CLUSTER_NAME)

對於更廣泛的模板化需求，建議用外部工具（envsubst、Jsonnet、Helm）預先處理。

§ 11. 完整生產級範例

global:
  scrape_interval: 15s
  scrape_timeout: 10s
  evaluation_interval: 15s
  external_labels:
    cluster: "production-tw"
    region: "ap-northeast-1"
    replica: "prom-1"
  query_log_file: /var/log/prometheus/query.log

alerting:
  alert_relabel_configs:
    - regex: 'replica'
      action: labeldrop
  alertmanagers:
    - api_version: v2
      static_configs:
        - targets:
            - 'alertmanager-1.monitoring.svc:9093'
            - 'alertmanager-2.monitoring.svc:9093'
            - 'alertmanager-3.monitoring.svc:9093'

rule_files:
  - "rules/recording/*.yml"
  - "rules/alerts/*.yml"

scrape_configs:
  # Prometheus 自監控
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  # K8s pods（透過 annotation 啟用）
  - job_name: 'kubernetes-pods'
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: true
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_path]
        target_label: __metrics_path__
        regex: (.+)
      - source_labels: [__address__, __meta_kubernetes_pod_annotation_prometheus_io_port]
        target_label: __address__
        regex: (.+?)(?::\d+)?;(\d+)
        replacement: $1:$2
    sample_limit: 50000
    label_limit: 30

  # Node exporter
  - job_name: 'node-exporter'
    kubernetes_sd_configs:
      - role: endpoints
        namespaces:
          names: [monitoring]
    relabel_configs:
      - source_labels: [__meta_kubernetes_service_name]
        regex: 'node-exporter'
        action: keep

remote_write:
  - url: "https://mimir.svc.cluster.local/api/v1/push"
    name: "mimir-tw"
    headers:
      X-Scope-OrgID: "team-platform"
    queue_config:
      capacity: 10000
      max_shards: 30
      max_samples_per_send: 2000
    write_relabel_configs:
      - source_labels: [__name__]
        regex: 'go_(gc|memstats)_.*'
        action: drop

storage:
  tsdb:
    out_of_order_time_window: 30m

otlp:
  promote_resource_attributes:
    - service.name
    - service.namespace
    - service.version
    - k8s.namespace.name
    - k8s.pod.name
    - k8s.deployment.name
  translation_strategy: NoUTF8EscapingWithSuffixes

📝 練習題

  1. 請列出您的環境中應該設定的 external_labels（至少 3 個）。
  2. 請寫一個 scrape_config，抓取 app1.local:8080/custom-metrics，使用 HTTPS 與 basic auth，且設 sample_limit 為 5000。
  3. 解釋下面這段 alerting 設定的問題在哪： 
    alerting:
  alertmanagers:
    - static_configs:
        - targets: ['alertmanager-lb.example.com:9093']
  4. 您要把 metrics 推到 Grafana Mimir 的 tenant-a，並丟掉所有 go_* metrics（不送出，但本地保留）。請寫出對應的 remote_write 區塊。
