Chapter 16 · Storage 深度

Storage 深度：TSDB 內部、Blocks、WAL、Compaction、OOO

Prometheus 的 TSDB 是它最被低估的部分。理解 head block / WAL / persistent blocks / compaction 流程能幫您診斷 90% 的儲存問題：磁碟用量爆炸、查詢慢、OOM、compaction 失敗。 本章解剖 TSDB 從寫入到查詢的完整路徑。

§ 1. TSDB 高層架構

Prometheus TSDB 的設計受 Facebook Gorilla 論文與 LSM-tree 啟發， 把寫入路徑與查詢路徑分開最佳化：

graph TB SCRAPE[Scrape Manager] subgraph WritePath["寫入路徑 · Write Path"] WAL[WAL
磁碟順序寫] HEAD[Head Block
記憶體] MMAP[mmap chunks
已壓縮但仍 head] BLOCK[Persistent Block
2h 區段] end subgraph QueryPath["查詢路徑 · Query Path"] QE[PromQL Engine] IDX[Index
label → series] CHUNK[Chunks
壓縮樣本] end SCRAPE -->|寫| WAL SCRAPE -->|append| HEAD HEAD -->|chunk 寫滿| MMAP MMAP -->|每 2h compact| BLOCK QE --> IDX QE --> HEAD QE --> BLOCK BLOCK --> CHUNK

1.1 三層儲存

層位置用途
Head Block 記憶體 + WAL（磁碟） 最近 2-3 小時的資料，仍可寫入
Persistent Block 磁碟（不可變） 2 小時為單位的歷史區段
Compacted Block 磁碟（不可變） 多個 block 合併後的更大區段

§ 2. 目錄結構解讀

看一個跑了一陣子的 Prometheus TSDB 目錄：

/var/lib/prometheus/
├── 01HQ2X3K7Z9R5M8N4P6T0Y1V/    ← Persistent block（ULID 命名）
│   ├── chunks/
│   │   ├── 000001              ← Chunk segment（最大 512MB）
│   │   └── 000002
│   ├── index                   ← series → chunk 的倒排索引
│   ├── meta.json               ← block 中介資料（時間範圍、stats）
│   └── tombstones              ← 軟刪除標記
├── 01HQ3X4L8A0S6N9P7Q2U1Y3W/   ← 另一個 block
│   ├── chunks/
│   ├── index
│   └── meta.json
├── chunks_head/                ← Head 已 mmap 的 chunks
│   ├── 000001
│   └── 000002
├── wal/                        ← Write-Ahead Log
│   ├── checkpoint.000123/      ← 檢查點
│   │   └── 00000000
│   ├── 00000124                ← 各 WAL segment（最大 128MB）
│   ├── 00000125
│   └── 00000126
├── wbl/                        ← Write-Behind Log（OOO 用，可選）
│   └── 00000001
├── lock                        ← 防多進程同時寫的鎖
└── queries.active              ← 當前執行中的 query 紀錄

2.1 ULID 是什麼？

每個 persistent block 用 ULID（Universally Unique Lexicographically Sortable Identifier）命名。 ULID = 時間戳 + 隨機，特性：

  • 排序即時間順序：ls 看到的順序就是時間順序
  • 26 字元，比 UUID 短
  • 不容易碰撞

2.2 看 block 的中介資料

cat /var/lib/prometheus/01HQ.../meta.json
{
  "ulid": "01HQ2X3K7Z9R5M8N4P6T0Y1V",
  "minTime": 1714521600000,
  "maxTime": 1714528800000,
  "stats": {
    "numSamples": 12345678,
    "numSeries": 543210,
    "numChunks": 654321
  },
  "compaction": {
    "level": 1,
    "sources": ["..."]
  },
  "version": 1
}

level 代表 compaction 層級：1 = 還沒被合併、2 = 二次合併、3 = 三次合併...。

§ 3. Head Block：記憶體中的最近資料

Head block 是最近資料的容器，同時存在於記憶體（active）與磁碟（WAL）。 這個雙重儲存讓 Prometheus 既能高速寫入，又有崩潰恢復能力。

3.1 Head 的內部結構

  • memSeries map：series ID → 該 series 的最近 chunks
  • Postings index：(label_name, label_value) → 所有相關 series ID
  • Head chunks：每個 series 在記憶體中持有最新 chunk
  • mmap chunks：寫滿的 chunk 從記憶體 flush 到 chunks_head/ 並 mmap

3.2 一個 Sample 的寫入流程

  1. Scrape 拿到 sample，找到（或建立）對應 memSeries
  2. 把 sample 寫入該 series 的 active chunk（記憶體）
  3. 同步寫一條 record 到 WAL segment（磁碟順序寫，極快）
  4. 當 active chunk 寫滿（120 個 sample 或時間達到上限）：
    • 把它 mmap 到 chunks_head/
    • 建立新的 active chunk

3.3 為什麼 WAL 用順序寫？

順序寫在 SSD/HDD 上都遠快於隨機寫。Prometheus 的 ingestion 能達到每秒百萬 sample 級別， 關鍵就是把所有寫入包裝成 WAL 順序追加，記憶體裡的更新是幾乎免費的。

§ 4. WAL（Write-Ahead Log）與 checkpoint

4.1 WAL 的目的：崩潰恢復

記憶體中的 head chunks 還沒 flush 成 persistent block 時，進程崩潰會遺失資料。 WAL 解決這個問題：每次 sample 寫入記憶體前先寫 WAL，崩潰後重啟時 replay WAL 重建 head 狀態。

4.2 WAL 內的記錄類型

類型內容
Series新建 series 的 label set 與 ID 對應
Samples批次的（series_ref, timestamp, value）
HistogramsNative histogram samples
Tombstones刪除請求
ExemplarsExemplar samples（trace ID 對應）
MetadataHelp / Type 資訊（v2.50+）

4.3 Segment 與 truncation

WAL 由多個 segment file 組成（預設每個 128MB）。 當 head 完成 compaction 後，舊的 WAL segment 不再需要，會被刪除： 

wal/
├── 00000124   ← 已 truncate（要刪）
├── 00000125   ← 已 truncate
├── 00000126   ← 仍需要（最近資料）
└── 00000127   ← 當前在寫

刪除策略：每次 head compaction 後，刪除「最舊的 2/3」 segment， 剩下的 1/3 留作冗餘（崩潰時的安全邊際）。

4.4 Checkpoint：加速 replay

崩潰恢復時要 replay 整個 WAL，segment 多時很慢。 Checkpoint 是把當前 active 的 series list 快照到一個壓縮的 checkpoint segment， 讓 replay 時可以跳過更早的 segment。 

wal/
├── checkpoint.000125/
│   └── 00000000           ← 包含到 segment 125 為止的 active series
├── 00000126               ← Replay 從這裡開始
└── 00000127

4.5 WAL Compression

自 v2.20 起預設啟用 Snappy 壓縮，WAL 大小減半，CPU 影響極小： 

--storage.tsdb.wal-compression                    # snappy（v2.20+ 預設）
--storage.tsdb.wal-compression-type=zstd          # 更高壓縮率，CPU 較高（v2.50+）

§ 5. Persistent Blocks 與 chunks

5.1 Block 的生命週期

  1. Head block 累積到 2 小時（預設 min-block-duration）
  2. Compactor 把 head 中該時段的資料寫成 persistent block
  3. Block 一旦寫好就不可變（只能整體刪除）
  4. 後續 compaction 會合併多個 block 成更大的
  5. 達到 retention 後整個 block 刪除

5.2 Chunk 內部

每條 series 在 block 內被切成多個 chunk，每個 chunk：

  • 包含連續的 sample（時間相鄰）
  • 使用 XOR 壓縮（Gorilla 演算法）
  • 典型大小：120 個 sample 或 30-60 分鐘
  • 平均每 sample 只佔 1-3 bytes（壓縮後）

5.3 Index 結構

每個 block 的 index 檔案是查詢的關鍵，包含：

  • Postings list：(label_name, label_value) → series ID 列表
  • Series：series ID → 完整 label set + chunk references
  • Symbol table：所有 unique label values 的字典編碼
  • Label index：每個 label 的所有可能值

5.4 mmap 加速查詢

Prometheus 把所有 block 的 index 與 chunks mmap 到記憶體。 OS page cache 會自動管理「常用部分留記憶體、冷資料退磁碟」，達成：

  • 啟動快（不必把所有資料讀入 RAM）
  • 查詢快（熱資料在 page cache）
  • 記憶體用量受 OS 控制（不會 OOM）

§ 6. Compaction 機制

6.1 兩種 compaction

類型觸發動作
Head Compaction Head 達到 2 小時 把 head 中的資料寫成第一個 persistent block
Block Compaction 背景定時 合併相鄰的 block 成更大的

6.2 Block Compaction 階梯

合併規則：

  • 每 3 個相鄰的同層級 block 合併為一個下一層級的 block
  • 2h → 6h → 18h → 54h（達到上限後不再合併）
  • 上限：--storage.tsdb.max-block-duration（預設 = retention/10）
合併前：[2h] [2h] [2h] [2h] [2h] [2h] [2h] [2h] [2h]
合併後：[6h]          [6h]          [6h]
最終：  [18h]

6.3 為什麼要 compaction？

  • 查詢更快：較大的 block，索引、字典更有效
  • 儲存更省：相同 series 跨 block 的 metadata 去重
  • 清理 tombstones：軟刪除真正消失
  • 處理 retention：超過保留期的 block 整個刪除

6.4 Compaction 監控

# Compaction 進度
prometheus_tsdb_compactions_total

# 失敗次數（>0 立刻警報）
prometheus_tsdb_compactions_failed_total

# 觸發次數
prometheus_tsdb_compactions_triggered_total

# 跳過次數（之前的還沒做完）
prometheus_tsdb_compactions_skipped_total

# 上次 compaction 耗時
prometheus_tsdb_compaction_duration_seconds

§ 7. Retention：保留時間與大小

7.1 兩個 retention 條件

--storage.tsdb.retention.time=30d     # 時間上限
--storage.tsdb.retention.size=100GB   # 磁碟上限

若同時設兩個，哪個先到就用哪個。例：

  • 設 30d / 50GB，磁碟用到 50GB 時就刪舊 block 即使還沒 30 天
  • 設 30d / 50GB，30 天到了就刪即使遠不到 50GB

7.2 Retention 不是即時生效

  • Block 是不可變的，只能整個刪除
  • 每次 compaction 才檢查 retention
  • 實際保留可能比設定多 10-20%（差一個 block 沒被刪）

7.3 WAL 與 chunks_head 不算 retention size

⚠️ 磁碟可能比 retention.size 多很多

retention.size 只計算 persistent block 的大小，不包括 WAL（最多 ~5GB）與 chunks_head/（最多幾 GB）。 規劃磁碟時要額外保留 10-20GB 給這兩者。

§ 8. Out-of-Order Ingestion 與 WBL

8.1 OOO 是什麼問題？

Prometheus 預設拒絕「比 head 中已有 sample 還早」的資料。 這個假設在某些情境會失效：

  • OpenTelemetry 多副本：多個 collector 寫入時間不同步
  • Remote Write 重試：網路恢復後重送舊資料
  • Federation：上游 Prometheus 慢於下游

8.2 啟用 OOO（v2.39+）

# prometheus.yml
storage:
  tsdb:
    out_of_order_time_window: 30m   # 接受過去 30 分鐘內的資料

8.3 OOO 的內部運作

  1. OOO sample 不寫到 WAL，而是寫到 WBL（Write-Behind Log）
  2. OOO chunk 與 in-order chunk 在 head 中分開儲存
  3. 查詢時兩者合併（Prometheus 自動處理）
  4. Compaction 時也把兩者合進同一個 persistent block

8.4 OOO 的代價

  • 記憶體用量增加（WBL + 額外 chunks）
  • 查詢稍慢（要合併 in-order 與 ooo）
  • Window 越大代價越高，建議 30m-2h 之間

§ 9. 容量規劃公式

9.1 磁碟用量公式

官方推薦公式：

needed_disk_space = retention_time_seconds × ingested_samples_per_second × bytes_per_sample

典型值：bytes_per_sample = 1.5-3（壓縮後）。

9.2 計算範例

假設：

  • Retention：30 天 = 2,592,000 秒
  • Active series：500,000
  • Scrape interval：15s（每秒 ingestion = 500,000 / 15 ≈ 33,333）
  • Bytes per sample：2
= 2,592,000 × 33,333 × 2 ≈ 173 GB

加上 WAL + chunks_head + 安全邊際 ≈ 200 GB。

9.3 RAM 估算

更難精算，經驗法則：

  • 每 100k active series ≈ 1-2 GB（基本 ingestion）
  • 查詢峰值可瞬間吃 8-16 GB
  • 建議 RAM = ingestion_estimate × 2-3 倍

9.4 用 metric 即時估

# 當前 series
prometheus_tsdb_head_series

# 每秒 ingestion
rate(prometheus_tsdb_head_samples_appended_total[5m])

# 每個 sample 平均 byte 數
prometheus_tsdb_storage_blocks_bytes / prometheus_tsdb_head_chunks_storage_size_bytes

# 用 predict_linear 估 N 天後
predict_linear(prometheus_tsdb_storage_blocks_bytes[7d], 30 * 86400)

§ 10. 常見儲存問題排除

⚠️ 問題 1：磁碟用量遠超 retention.size

檢查：
• WAL 是否爆炸（compaction 失敗會讓 WAL 不斷成長）：du -sh /var/lib/prometheus/wal
• Compaction 失敗：rate(prometheus_tsdb_compactions_failed_total[5m]) > 0
• OOO window 太大累積大量 WBL
解法：修復 compaction（看 logs）或縮短 retention

⚠️ 問題 2：Compaction 失敗

常見原因：
• 磁碟空間不足（compaction 需要 2x 暫存空間）
• WAL segment 損壞（看 log corruption in segment）
• 進程被 OOM killed 中斷 compaction
解法：先擴磁碟，再用 promtool tsdb analyze 找壞 block 手動移除

⚠️ 問題 3：啟動極慢（10+ 分鐘）

WAL 太大需要 replay。檢查：
• ls /var/lib/prometheus/wal | wc -l 看 segment 數量
• 預期 50-100 個，若 1000+ 表示 checkpoint 失效
解法：確認磁碟健康、考慮 --storage.tsdb.no-lockfile（測試用）

⚠️ 問題 4：Out-of-order samples 被拒

日誌見 out of order sample。
原因：應用 timestamp 跳動 / 多副本 OTel collector / federation 延遲
解法：啟用 out_of_order_time_window: 30m

⚠️ 問題 5：High Cardinality 拖垮 TSDB

Series 數爆炸 → index 變大、查詢慢、記憶體爆。
診斷：promtool tsdb analyze /var/lib/prometheus
解法：找出兇手 metric/label，用 metric_relabel_configs drop（Ch 10）

⚠️ 問題 6：大查詢直接 OOM

查詢區間太大 / series 太多。
解法：
• 設 --query.max-samples=50000000（防單一 query 失控）
• 設 --query.timeout=2m（強制終止慢 query）
• 把長窗口算進 recording rule

✅ 本章重點摘要

30 秒回顧
  • 三層儲存：Head（記憶體+WAL）→ Persistent Block（2h）→ Compacted Block
  • WAL 用磁碟順序寫達成高速 ingestion，崩潰後 replay 重建狀態
  • Block 用 ULID 命名，內含 chunks（XOR 壓縮）+ index（postings）+ meta.json
  • 每 sample 壓縮後僅 1-3 bytes（Gorilla 演算法）
  • Compaction 階梯：2h → 6h → 18h → 54h，定期合併相鄰 block
  • Retention 兩條件（time / size）取先到，WAL 與 chunks_head 不算 size
  • OOO Ingestion（v2.39+）：用 WBL 接收過去資料，OTel/RW 場景必設
  • 容量公式：retention × samples_per_sec × ~2 bytes
  • RAM 公式：每 100k series ≈ 1-2 GB，總 RAM = ingestion × 2-3
  • 用 mmap 讓 OS page cache 自動管理熱資料，避免 OOM
