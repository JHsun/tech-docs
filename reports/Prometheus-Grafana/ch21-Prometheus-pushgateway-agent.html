Chapter 21 · Pushgateway / Agent

Pushgateway 與 Agent 模式：何時 Push、何時別 Push

Prometheus 是 pull 模型，但有兩個官方 push 變體：Pushgateway（給短任務用） 與 Prometheus Agent Mode（純收集器）。本章講清楚這兩個工具的正確用途、 設定方法、以及最常被誤用的場景。

§ 1. Pull vs Push 之爭

Prometheus 預設是 pull 模型。Pull 的優勢：

  • 自動健康檢查：scrape 失敗就知道 target 掛了
  • 集中設定：哪些 target 要抓在 Prometheus 統一管
  • 容易橫向擴展：多個 Prometheus 各自抓不同子集
  • 可重複：testing/debug 時可以手動 curl /metrics

但有些情境 push 是必要的：

  • 短任務（短到根本來不及被 scrape）：cron job、CI build
  • 無法被 scrape 的環境：NAT 後面、序列功能、無法暴露 HTTP port
  • 純資料收集 agent：edge node 數量極大，pull 不可行

Prometheus 給了兩個 push 解決方案：

工具用途
Pushgateway緩衝短任務的 metric，被 Prometheus 抓
Prometheus Agent Mode把 Prometheus 變成純 push 的收集器

§ 2. Pushgateway：什麼時候用

Pushgateway 是一個獨立服務，扮演「中介緩衝區」：短任務 push metric 進去， Prometheus 從 pushgateway pull。它不是替代 pull 模型的萬能工具。

graph LR JOB1[Cron Job
批次任務] JOB2[CI Build
結束後 push] JOB3[資料 ETL Pipeline] PG[Pushgateway :9091
緩衝區] PROM[Prometheus
pull from PG] JOB1 -- POST /metrics/job/... --> PG JOB2 -- POST --> PG JOB3 -- POST --> PG PG -- /metrics --> PROM

2.1 Pushgateway 唯一適合的場景

✅ Pushgateway 是為這個情境設計的

「服務級的短任務」：跑完就消失、跑很短、不該被 scrape。例：

  • 每晚 backup job 結束時 push「成功 / 失敗 / 耗時」
  • CI/CD pipeline 結束時 push「build 成功率、duration」
  • 每小時跑的資料 ETL 結束時 push 處理筆數

2.2 不該用 Pushgateway 的情況

🚨 這些情境用 Pushgateway 是「antipattern」
  1. 長期運行的應用：用 client library + scrape，不要 push
  2. 每個進程的個別 metric：pushgateway 只應記「服務級彙總」，不該每個 worker process 各自 push
  3. 當作「中介佇列」處理高吞吐量：pushgateway 是單機緩衝區，不擴展
  4. 取代 service discovery：服務本身可以被 pull，就用 pull
  5. HA 場景：pushgateway 是單點，掛了就沒資料

§ 3. Pushgateway 安裝與基本用法

3.1 啟動

# Binary
wget https://github.com/prometheus/pushgateway/releases/download/v1.10.0/pushgateway-1.10.0.linux-amd64.tar.gz
tar xvf pushgateway-*.tar.gz
./pushgateway

# Docker
docker run -d -p 9091:9091 prom/pushgateway

# 預設 port: 9091
# 持久化：--persistence.file=/tmp/pushgateway.data --persistence.interval=5m

3.2 Push metric（基本）

# 用 curl push 一個 metric
echo "backup_last_success_timestamp_seconds $(date +%s)" | \
  curl --data-binary @- http://pushgateway:9091/metrics/job/db_backup

# 帶 grouping label（很重要）
echo "backup_last_success_timestamp_seconds $(date +%s)" | \
  curl --data-binary @- http://pushgateway:9091/metrics/job/db_backup/instance/db-1

3.3 用 client library push

Python：

from prometheus_client import CollectorRegistry, Gauge, push_to_gateway

registry = CollectorRegistry()
g = Gauge('backup_last_success_timestamp_seconds',
          'Last successful backup timestamp', registry=registry)
g.set_to_current_time()

push_to_gateway('pushgateway:9091',
                job='db_backup',
                grouping_key={'instance': 'db-1'},
                registry=registry)

Go：

import "github.com/prometheus/client_golang/prometheus/push"

err := push.New("http://pushgateway:9091", "db_backup").
    Grouping("instance", "db-1").
    Collector(myGauge).
    Push()

3.4 刪除舊資料

# 刪除某 job + grouping 的所有 metric
curl -X DELETE http://pushgateway:9091/metrics/job/db_backup/instance/db-1

# 刪除整個 job
curl -X DELETE http://pushgateway:9091/metrics/job/db_backup

§ 4. Pushgateway 重要陷阱

🚨 陷阱 1：Pushgateway 不會自動清舊資料

您 push 的 metric 會永遠留在 pushgateway，直到您主動 DELETE。
意思：您的 backup job 從 instance db-1 改名為 db-prod-1， pushgateway 會同時保留兩份，查詢時看到雙倍資料。
解法：每次 push 前先 DELETE 舊的（或用同樣 grouping key 覆蓋）。

🚨 陷阱 2：時間戳「是被 scrape 的時間，不是 push 的時間」

Pushgateway 不保留 push 進來時的時間戳，當 Prometheus 抓 pushgateway 時， 用抓的時間當作 sample 時間戳。
意思：您的 cron job 凌晨 3 點 push 了 metric，但 Prometheus 早上 9 點才抓到， sample timestamp 是早上 9 點，不是凌晨 3 點。
解法：把實際時間放進 metric value 自己（用 backup_last_success_timestamp_seconds 這種 metric）。

🚨 陷阱 3：用 Counter 累加但 push 是「整體覆蓋」

Pushgateway 的 PUT 是「整組覆蓋」 — 你 push jobs_processed_total 100， 下次 push jobs_processed_total 50，pushgateway 變 50（不是累加成 150）。
解法：要嘛在 client 自己累加再 push，要嘛改用 gauge（記「目前狀態」而非「累計次數」）。

🚨 陷阱 4：Pushgateway 是 SPOF（single point of failure）

它不支援 HA。pushgateway 掛了 → 您的 cron job push 失敗 → metric 遺失。
若可以，讓 cron job 自己重試（push_to_gateway 會 retry）， 或部署兩個 pushgateway 用 LB 前置。

§ 5. Pushgateway 與 Prometheus 設定

關鍵：抓 pushgateway 時要設 honor_labels: true。

scrape_configs:
  - job_name: 'pushgateway'
    honor_labels: true                    # ← 關鍵！
    static_configs:
      - targets: ['pushgateway:9091']

5.1 為什麼必須 honor_labels: true？

Pushgateway 已經為每個 push 標記了 job 與 grouping label。 若沒有 honor_labels: true，Prometheus 會把「job=pushgateway」覆蓋上去， 導致所有 metric 都被歸屬到 pushgateway 這個 job，喪失原本的 job 區分。

5.2 從 push job 自身告警

典型告警：「backup job 超過 24 小時沒成功」

- alert: BackupJobNotRunning
  expr: time() - backup_last_success_timestamp_seconds > 86400
  for: 5m
  labels:
    severity: critical
  annotations:
    summary: "{{ $labels.instance }} backup hasn't succeeded in 24+ hours"

§ 6. Prometheus Agent Mode 是什麼

Prometheus Agent Mode 是 v2.32 引入的第二種運行模式。 在 agent mode 下，Prometheus 變成純收集器：抓 target → remote_write 到後端， 不存本地 TSDB、不能查詢、不能告警。

6.1 為什麼設計 agent mode？

傳統 Prometheus 服務做太多事 — 抓取 + 儲存 + 查詢 + 告警，在某些場景太重：

  • 邊緣 node 數量大（千節點），每個都跑完整 Prometheus 浪費資源
  • 用 Mimir/Thanos/VictoriaMetrics 集中儲存，本地不需要 TSDB
  • 記憶體受限環境（IoT、small VM）
  • 純收集任務，沒有本地查詢需求

6.2 Agent mode 的特性

功能Server Mode（預設）Agent Mode
Scraping✅✅
Service Discovery✅✅
Remote Write✅✅（必要）
本地 TSDB✅ 完整❌（只有 WAL，無持久 block）
查詢 PromQL✅❌
Recording / Alerting Rules✅❌
記憶體用量取決於 series大幅降低

§ 7. Agent Mode vs Server Mode

7.1 啟用 Agent Mode

prometheus --agent --config.file=/etc/prometheus/prometheus-agent.yml

7.2 Agent 的設定檔

# prometheus-agent.yml
global:
  scrape_interval: 15s
  external_labels:
    cluster: edge-tw-1
    region: asia-east

scrape_configs:
  - job_name: 'apps'
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      # ...

remote_write:
  - url: 'https://mimir-central.example.com/api/v1/push'
    headers:
      X-Scope-OrgID: 'team-edge'
    queue_config:
      max_shards: 50
      max_samples_per_send: 5000

# 注意：agent mode 下這些區塊會被忽略：
# - rule_files
# - alerting
# - remote_read

7.3 何時用 Agent Mode

✅ 適合 Agent Mode 的情境
  1. 已有集中式長期儲存（Mimir / Thanos / VictoriaMetrics）
  2. 邊緣 / 多 cluster 部署，本地不需查詢
  3. 記憶體 / 磁碟受限環境
  4. 純粹的「metric pipeline collector」角色

7.4 不適合 Agent Mode 的情境

⚠️ 這些情境別用 Agent Mode
  1. 本地需要查 dashboard（沒 TSDB 可查）
  2. 本地需要 alerting（agent 不評估 rule）
  3. 沒有遠端後端可寫
  4. 單機小型部署（一台 server 用完整 Prometheus 就好）

§ 8. Agent Mode 部署架構

典型多 cluster 架構：

graph TB subgraph Edge1["邊緣 Cluster TW"] A1[Prometheus Agent
scrape + remote_write] T1[Targets] T1 --> A1 end subgraph Edge2["邊緣 Cluster JP"] A2[Prometheus Agent] T2[Targets] T2 --> A2 end subgraph Edge3["邊緣 Cluster US"] A3[Prometheus Agent] T3[Targets] T3 --> A3 end subgraph Central["中央資料中心"] MIMIR[Grafana Mimir
儲存 + 查詢] ALERTING[獨立 Prometheus
跑 alerting rules] GRAFANA[Grafana] MIMIR --> ALERTING MIMIR --> GRAFANA end A1 -- remote_write --> MIMIR A2 -- remote_write --> MIMIR A3 -- remote_write --> MIMIR

這個架構讓邊緣節點輕量、中央集中查詢與告警。 Mimir 也支援 ruler 元件做 alerting rule 評估，可不需獨立 alerting Prometheus。

§ 9. 替代方案：Grafana Alloy

Grafana Alloy（前 Grafana Agent）是另一個熱門選擇，不只取代 Prometheus Agent， 還支援 logs、traces、profiling 三本柱：

能力Prometheus AgentGrafana Alloy
Metrics（Prometheus 格式）✅✅
OpenTelemetry receiver有限✅
Logs（push to Loki）❌✅
Traces（push to Tempo）❌✅
Profiling（push to Pyroscope）❌✅
eBPF auto-instrumentation（Beyla）❌✅
設定語法YAMLHCL（River）
記憶體較輕量較重

9.1 Alloy 取代 Prometheus Agent 範例

# config.alloy
prometheus.scrape "default" {
  targets = [
    {"__address__" = "localhost:9100"},  // node-exporter
  ]
  forward_to = [prometheus.remote_write.mimir.receiver]
}

prometheus.remote_write "mimir" {
  endpoint {
    url = "https://mimir.example.com/api/v1/push"

    headers = {
      "X-Scope-OrgID" = "team-platform",
    }
  }
}

Alloy 是新一代 Grafana 全家桶 agent。Part 5 會更詳細介紹。

✅ 本章重點摘要

30 秒回顧
  • Pull 是預設、Push 是例外。要先想能不能用 pull
  • Pushgateway 唯一適合：服務級的短任務（cron / batch）
  • Pushgateway 4 大陷阱：不自動清舊資料、用 scrape 時間不是 push 時間、PUT 是覆蓋不是累加、SPOF
  • 抓 Pushgateway 必設 honor_labels: true
  • Prometheus Agent Mode（v2.32+）：純收集，無 TSDB / 無查詢 / 無 alert
  • Agent 適用：邊緣 node 多、集中式儲存、資源受限
  • Agent 啟動：prometheus --agent
  • Grafana Alloy：升級版 agent，支援 metrics + logs + traces + profiling
  • 長期應用 不應該用 Pushgateway，用 client library 直接被 scrape
