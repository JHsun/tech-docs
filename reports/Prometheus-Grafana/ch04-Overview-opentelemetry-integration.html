Chapter 04 · 標準化與整合

OpenTelemetry 與 Prometheus／Grafana 的整合定位

OpenTelemetry（OTel）是當前可觀測性領域最重要的標準化專案。它與 Prometheus 之間 既是合作關係，也存在某些角色重疊。本章釐清兩者的定位、整合方式， 以及在實務上「該用哪個」的決策思路。

§ 1. 什麼是 OpenTelemetry？為什麼出現？

在 2019 年之前，分散式追蹤領域有兩套互不相容的開源標準：

  • OpenTracing（CNCF）：定義 trace API 規範
  • OpenCensus（Google）：定義 trace + metric SDK

這兩套標準各有支持者，但對使用者來說是噩夢——選了其中一個就被綁死。 2019 年，兩個專案決定合併，並命名為 OpenTelemetry， 目標是成為「vendor-neutral 的可觀測性標準」。

OpenTelemetry 想解決的核心痛點：

痛點OTel 的解法
每個 APM/監控廠商都有自己的 SDK，換一家就要重寫埋點 定義統一的 instrumentation API 與 SDK，廠商提供對應的 exporter 即可
Metrics、Logs、Traces 各自為政，難以關聯 三種訊號共用 trace context、resource attribute 等基礎概念
切換後端需要改 application 程式碼 應用只送 OTLP 給 Collector，Collector 負責路由到任何後端
每種語言的埋點品質不一 OTel 規範要求所有官方 SDK 一致性

OpenTelemetry 已成為 CNCF Graduated 專案，是 CNCF 上活躍度第二高的專案 （僅次於 Kubernetes 本身）。當前已支援 11+ 種主流語言。

§ 2. OTel 核心概念：Signals、SDK、Collector、OTLP

2.1 Signals（訊號類型）

OTel 把可觀測性訊號分類為以下 signals：

2.3 OTLP 協議（OpenTelemetry Protocol）

OTLP 是 OpenTelemetry 定義的傳輸協議，採用 Protocol Buffers 序列化。 支援兩種傳輸方式：

  • OTLP/gRPC：預設 port 4317，效能最佳
  • OTLP/HTTP：預設 port 4318，使用 protobuf 或 JSON

OTLP 是應用程式與 Collector / 後端之間傳輸 metrics、traces、logs 的統一協議。 它取代了過去每個訊號類型各自有不同協議的混亂狀況（Jaeger 用 Thrift、Zipkin 用自家 JSON、 Prometheus 用 remote_write、log 系統各自為政）。

2.4 OTel Collector 的三個元件

  • Receivers：接收訊號的入口，可以是 push（OTLP receiver）或 pull（Prometheus receiver）
  • Processors：訊號的處理層——批次、取樣、過濾、加工屬性、k8s metadata 注入等
  • Exporters：把訊號送往後端的出口

2.5 兩種 Collector 發行版

版本內容適用
otelcol（Core）核心元件，最小集單純的 OTLP 路由
otelcol-contrib含社群第三方元件（雲廠商、廠商整合等）大多數實務情境

§ 3. OTel 與 Prometheus：競爭還是互補？

這是新手最常困惑的問題。簡短答案：互補大於競爭。 Prometheus 與 OTel 兩個社群都明確表態，雙方致力於互相相容。

3.1 角色定位差異

面向PrometheusOpenTelemetry
主要角色Metrics 後端 + 收集標準 + 多訊號採集
訊號範圍主要 MetricsMetrics + Traces + Logs + Profiles
傳輸模型主要 Pull主要 Push（OTLP）
Metric 模型累積式（cumulative）為主支援 cumulative 與 delta
SDK 數量5 種官方語言11+ 種官方語言
儲存能力有（TSDB）無（要外接後端）
查詢語言PromQL無（依後端）

3.2 Prometheus 的官方表態

ℹ️ Prometheus 對 OpenTelemetry 的承諾

Prometheus 官方在「Our commitment to OpenTelemetry」聲明中表示： Prometheus 的目標是成為「OpenTelemetry metrics 的原生後端」。 具體做法包括：原生支援 OTLP 接收、UTF-8 metric 名稱、 原生 histogram、delta temporality 處理、以及 exemplar 支援。

3.3 兩者的相容性里程碑

§ 4. 整合方式 1：OTel → Prometheus（多種路徑）

當您的應用使用 OTel SDK 埋點，但想用 Prometheus 作為儲存後端，有四種路徑：

4.1 路徑 A：直接送 OTLP 到 Prometheus（最簡單）

從 Prometheus 2.47 起內建 OTLP receiver，3.x 已穩定。

graph LR APP["App + OTel SDK"] -->|OTLP/HTTP| PRO[Prometheus] PRO --> G[Grafana]

啟用方式：

# 啟動 Prometheus 加上 flag
prometheus --web.enable-otlp-receiver --config.file=prometheus.yml

prometheus.yml 中 OTLP 相關設定：

otlp:
  # 將 OTel resource attribute 升格為 metric label
  promote_resource_attributes:
    - service.instance.id
    - service.name
    - service.namespace
    - service.version
    - cloud.availability_zone
    - cloud.region
    - deployment.environment.name
    - k8s.cluster.name
    - k8s.container.name
    - k8s.deployment.name
    - k8s.namespace.name
    - k8s.pod.name
    - k8s.replicaset.name

  # OTel metric 名稱翻譯策略
  translation_strategy: NoUTF8EscapingWithSuffixes

# 啟用 out-of-order 容忍（建議 30 分鐘）
storage:
  tsdb:
    out_of_order_time_window: 30m

應用端設定環境變數：

export OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
export OTEL_EXPORTER_OTLP_METRICS_ENDPOINT=http://prometheus:9090/api/v1/otlp/v1/metrics

4.2 路徑 B：經過 OTel Collector → Prometheus OTLP receiver

適合需要在中間做處理（取樣、加工、k8s metadata 注入）的場景。

graph LR APP["App + OTel SDK"] -->|OTLP| C[OTel Collector] C -->|OTLP HTTP| PRO[Prometheus] PRO --> G[Grafana]

4.3 路徑 C：Collector 用 Prometheus exporter（拉模式）

Collector 把 OTel 訊號轉換成 Prometheus 格式，由 Prometheus scrape 來抓。

# OTel Collector 配置
exporters:
  prometheus:
    endpoint: "0.0.0.0:8889"
    namespace: "myapp"
    resource_to_telemetry_conversion:
      enabled: true

service:
  pipelines:
    metrics:
      receivers: [otlp]
      exporters: [prometheus]

Prometheus 端：

scrape_configs:
  - job_name: 'otel-collector'
    static_configs:
      - targets: ['otel-collector:8889']

4.4 路徑 D：Collector 用 Prometheus Remote Write exporter

適合直接送到長期儲存（Mimir / Thanos / VictoriaMetrics）。

# OTel Collector 配置
exporters:
  prometheusremotewrite:
    endpoint: "https://mimir.example.com/api/v1/push"
    headers:
      X-Scope-OrgID: "my-tenant"
    resource_to_telemetry_conversion:
      enabled: true
    external_labels:
      cluster: "production"

service:
  pipelines:
    metrics:
      receivers: [otlp]
      exporters: [prometheusremotewrite]

4.5 四種路徑比較

路徑傳輸優點缺點
A. OTLP → Prom 直連 OTLP HTTP 最簡單、無中間元件 無法做進階處理、Prometheus 必須能直接被應用觸達
B. OTel Collector → Prom OTLP OTLP 可做處理、可緩衝 多一層元件
C. Collector + Prometheus exporter（拉） HTTP scrape 沿用 Prom 拉模型、無 push 認證問題 Collector stateful 多副本要 sharding
D. Collector + Remote Write exporter Remote Write 直送長期儲存、生態成熟 Remote Write 不傳遞 metadata（Help/Type 資訊遺失）

§ 5. 整合方式 2：Prometheus → OTel

反過來，當您有 Prometheus 既有環境，但想把資料送到 OTel-native 後端：

5.1 路徑 E：OTel Collector 的 Prometheus receiver（拉）

讓 Collector 扮演「Prometheus server」去抓 /metrics 端點：

receivers:
  prometheus:
    config:
      scrape_configs:
        - job_name: 'node-exporter'
          static_configs:
            - targets: ['node-exporter:9100']
          scrape_interval: 30s

exporters:
  otlp:
    endpoint: backend.example.com:4317

service:
  pipelines:
    metrics:
      receivers: [prometheus]
      exporters: [otlp]

5.2 路徑 F：Collector 的 prometheusremotewrite receiver

讓 Prometheus 用 remote_write 把資料推給 Collector：

# OTel Collector
receivers:
  prometheusremotewrite:
    endpoint: 0.0.0.0:19291

# Prometheus.yml
remote_write:
  - url: http://otel-collector:19291/api/v1/write

§ 6. 整合方式 3：Grafana Alloy 作為融合方案

如同 Ch 03 介紹過，Grafana Alloy 同時是 OTel Collector 的發行版， 也內建 Prometheus 風格的 scrape 引擎。一個 Alloy 就能扮演：

  • OTLP receiver（接收應用的 OTel 訊號）
  • Prometheus scraper（抓 exporters 與 /metrics）
  • Promtail（抓 logs 送 Loki）
  • Tail-based sampling 處理 traces
  • 同時把資料 fan-out 到多個後端

§ 7. 選型決策：Prometheus Client Library 還是 OTel SDK？

對新專案來說，最常見的問題是：「我的應用要用 prometheus-client 還是 OTel SDK？」 這個決定影響重大，因為它決定了未來的可移植性與擴展性。

7.1 選 Prometheus Client Library 的時機

  • 只關心 metrics，不打算做 traces 或 logs
  • 確定後端永遠是 Prometheus / VictoriaMetrics / Mimir
  • 應用是 Go/Python 等 Prometheus 原生支援良好的語言
  • 需要 Prometheus 特有功能（exemplars、native histogram、precise relabeling）
  • 團隊已熟悉 Prometheus 生態，沒有跨後端遷移需求

7.2 選 OTel SDK 的時機

  • 需要同時做 metrics + traces（甚至 logs）
  • 未來可能切換到不同 APM 廠商或後端
  • 多語言混合環境，希望有一致的 API
  • 需要 trace context propagation（W3C TraceContext）
  • 使用 auto-instrumentation 簡化埋點

7.3 折衷方案

✅ 業界較主流的折衷做法
  • Metrics：用 Prometheus Client Library（成熟、效能好、特性完整）
  • Traces：用 OTel SDK（業界標準、auto-instrumentation 強）
  • Logs：用語言原生 logger，加上結構化欄位（trace_id、span_id）
  • 三者都用 OTel Collector / Alloy 統一收集與路由

7.4 OTel SDK 提供 Prometheus exporter

最後值得一提：OTel 各語言 SDK 都提供 Prometheus exporter 模組， 讓您即使選 OTel SDK 也能直接暴露 /metrics endpoint， 讓 Prometheus 拉取。這是「拿 OTel 的 API、用 Prometheus 的拉模型」的折衷做法。

§ 8. 常見陷阱與最佳實踐

⚠️ 陷阱 1：Resource attribute 沒升格成 label

OTel 的 service.name、k8s.pod.name 等 resource attribute 預設 不會自動成為 Prometheus label，而是被存進一條特別的 target_info metric。 查詢時若沒做 join 會看不到這些屬性。解法：在 Prometheus 設定 otlp.promote_resource_attributes 或在 Collector 啟用 resource_to_telemetry_conversion。

⚠️ 陷阱 2：Out-of-order 樣本被拒絕

多副本 OTel Collector 同時發送可能造成樣本順序錯亂。Prometheus 預設會拒絕亂序樣本。 解法：在 storage.tsdb 設定 out_of_order_time_window: 30m。

⚠️ 陷阱 3：Delta temporality 直接送 Prometheus

OTel 支援 cumulative 與 delta 兩種時間性，但 Prometheus 原生只認 cumulative。 若應用送 delta 給 Prometheus，數值會不正確。解法：應用設 OTEL_EXPORTER_OTLP_METRICS_TEMPORALITY_PREFERENCE=cumulative， 或在 Collector 用 deltatocumulative processor 轉換。

⚠️ 陷阱 4：Metric 名稱翻譯造成混淆

OTel 的 http.server.duration 在 Prometheus 預設會被翻譯成 http_server_duration_seconds（多了單位後綴與底線替換）。 若團隊不知道這個規則，會找不到 metric。解法：明確設定 translation_strategy，並在文件中說明。

✅ 最佳實踐
  1. 新專案優先選 OTel SDK 做 traces，metrics 視團隊熟悉度而定
  2. 用 OTel Collector / Alloy 作為統一管線，避免應用直連後端
  3. 應用只送 OTLP，後端可隨時更換（Prometheus → Mimir → 商業 SaaS）
  4. 明確設定 promote_resource_attributes，避免查詢時找不到 service name
  5. 遵循 OTel Semantic Conventions，metric/attribute 命名一致
  6. 啟用 Prometheus 的 UTF-8 + out-of-order（v3+ 預設啟用）
