§ 1. IaC 三大層次

1.1 為什麼需要 IaC

  • 跨環境一致（prod / staging / dev 的 dashboard 一樣）
  • 版本控管（Git history、PR review）
  • 災難復原（Grafana 全掛了能 30 分鐘重建）
  • 規模化（管理 100+ dashboard 不可能手動）
  • Audit（誰改了什麼）

1.2 三層 IaC 工具

層次工具適合
檔案 Provisioning YAML、grafanactl 初級、單一 Grafana
程式碼 Foundation SDK（Go/TS/Python/PHP/Java） 進階、複雜邏輯、模板化
編排 Terraform / Ansible / Grafana Operator 多環境、多 instance、跨團隊

1.3 Grafana 12 的轉折

Grafana 12（2025-04 發表）標誌「Observability as Code」進入新時代：

  • 新 v1beta1 API（K8s-style resource）
  • grafanactl CLI 取代部分 grafana-cli 功能
  • Foundation SDK 公開 preview
  • Git Sync GA（UI 直連 Git repo）
  • Terraform Provider 完整支援

§ 2. Provisioning 完整路徑

2.1 Provisioning 是什麼

Grafana 啟動時讀取 /etc/grafana/provisioning/ 下的 YAML 檔，自動建立 / 更新對應資源。 這是最簡單的 IaC，不需要外部工具。

2.2 完整目錄結構

/etc/grafana/provisioning/
├── datasources/        # 資料源
│   ├── prometheus.yml
│   └── loki.yml
├── dashboards/         # Dashboard 提供者（指向 JSON 檔目錄）
│   └── default.yml
├── alerting/           # Unified Alerting（v9+）
│   ├── rules.yml
│   ├── contact-points.yml
│   ├── policies.yml
│   ├── mute-timings.yml
│   └── templates.yml
├── notifiers/          # Legacy alerting（v9+ 不用）
└── plugins/            # App plugin
    └── apps.yml

2.3 Provisioning 何時觸發

  • Grafana 啟動時
  • API 觸發 reload：POST /api/admin/provisioning/{datasources,dashboards,alerting}/reload
  • Dashboard 還支援檔案變動自動偵測（updateIntervalSeconds）

2.4 reload API 範例

# 改了 datasources YAML 後 reload
curl -X POST \
  -H "Authorization: Bearer $TOKEN" \
  http://grafana:3000/api/admin/provisioning/datasources/reload

# 同樣 endpoint 給 dashboards / alerting / plugins / access-control

不用重啟 Grafana 就能套用新 config，CI/CD 友善。

§ 3. Datasource Provisioning 進階

3.1 完整欄位範例

# /etc/grafana/provisioning/datasources/all.yaml
apiVersion: 1

# 刪除某些 datasource（在啟動時）
deleteDatasources:
  - name: OldPrometheus
    orgId: 1

datasources:
  - name: Prometheus
    type: prometheus
    uid: prometheus-prod        # 重要：穩定 UID 給其他 ds 引用
    orgId: 1
    access: proxy
    url: http://prometheus:9090
    isDefault: true
    editable: false             # UI 鎖住

    # 一般設定（明文）
    jsonData:
      httpMethod: POST
      manageAlerts: true
      prometheusType: Prometheus
      prometheusVersion: 3.11.2
      timeInterval: 15s
      cacheLevel: High
      incrementalQuerying: true
      exemplarTraceIdDestinations:
        - name: traceID
          datasourceUid: tempo-prod
      httpHeaderName1: 'X-Scope-OrgID'

    # 敏感設定（會加密儲存）
    secureJsonData:
      httpHeaderValue1: 'tenant-1'
      basicAuthPassword: '$__file{/run/secrets/prom-pass}'

3.2 editable 的取捨

設定效果建議
editable: falseUI 完全鎖住，只能改 YAMLProduction 推薦
editable: trueUI 可改但下次 reload 會被覆蓋不建議（造成困惑）

3.3 多環境 datasource 範例

datasources:
  # 三個 Prometheus instance（多 region）
  - name: Prometheus-TW
    uid: prom-tw
    type: prometheus
    url: http://prom-tw.internal:9090

  - name: Prometheus-JP
    uid: prom-jp
    type: prometheus
    url: http://prom-jp.internal:9090

  - name: Prometheus-US
    uid: prom-us
    type: prometheus
    url: http://prom-us.internal:9090

  # Mixed datasource 引用上述三家
  # 在 dashboard variable 中用 ${prom-tw} / ${prom-jp} 切換

3.4 Secret 從外部取的三種方式

# 方式 1：$__file{} 從檔案讀
secureJsonData:
  password: $__file{/run/secrets/db-pass}

# 方式 2：$__env{} 從環境變數讀（v9+）
secureJsonData:
  password: $__env{DB_PASSWORD}

# 方式 3：Vault（Enterprise）
secureJsonData:
  password: $__vault{secret/data/grafana#db-pass}

§ 4. Dashboard Provisioning 進階

4.1 Provider 設定

# /etc/grafana/provisioning/dashboards/default.yaml
apiVersion: 1

providers:
  - name: 'production'
    orgId: 1
    folder: 'Production'         # Folder 名稱（不存在會自動建）
    folderUid: 'production-uid'
    type: file
    disableDeletion: false       # UI 刪除是否同步刪除底層 JSON
    updateIntervalSeconds: 30    # 多久 rescan 檔案
    allowUiUpdates: false        # UI 修改是否可寫回檔案
    options:
      path: /var/lib/grafana/dashboards/production
      foldersFromFilesStructure: true   # ✨ 從子目錄自動建 folder

  - name: 'staging'
    orgId: 1
    folder: 'Staging'
    type: file
    options:
      path: /var/lib/grafana/dashboards/staging

4.2 foldersFromFilesStructure 的妙用

# 啟用後，目錄結構自動對應 folder：
/var/lib/grafana/dashboards/production/
├── Backend/
│   ├── api-service.json     → folder=Production/Backend
│   └── auth-service.json
├── Frontend/
│   └── web.json             → folder=Production/Frontend
├── Infrastructure/
│   ├── nodes.json
│   └── network.json
└── overview.json            → folder=Production（root）

4.3 allowUiUpdates 與 disableDeletion 的搭配

場景allowUiUpdatesdisableDeletion
純 IaC（Git only） false false
UI 編輯後同步回 Git true false
「禁止破壞」 false true

4.4 Dashboard JSON 的「動態」部分

Dashboard JSON 有些欄位會自動由 Grafana 寫入：id, version, uid。 Provisioning 時要小心：

  • uid：必設（穩定，給 link 用）
  • id：留 null（Grafana 自動分配）
  • version：留 0 或省略

4.5 用 datasource UID 變數

Production 多環境，dashboard JSON 中 datasource 寫死會出問題：

// dashboard.json - 錯誤做法（寫死 UID）
"datasource": {
  "type": "prometheus",
  "uid": "prom-tw"          // 在其他環境會失敗
}

// 正確做法 1：用 dashboard variable
{
  "templating": {
    "list": [{
      "name": "DS_PROMETHEUS",
      "type": "datasource",
      "query": "prometheus"
    }]
  },
  "panels": [{
    "datasource": {
      "type": "prometheus",
      "uid": "${DS_PROMETHEUS}"
    }
  }]
}

// 正確做法 2：用 default datasource
"datasource": {
  "type": "prometheus",
  "uid": "default"          // 用 org 的 default
}

§ 5. Alerting Provisioning 完整

5.1 五個 YAML 檔的職責

檔案內容
rules.yamlAlert rule groups
contact-points.yamlSlack / Email / PD 設定
policies.yamlNotification routing tree
mute-timings.yaml定期靜音時段
templates.yaml通知 message templates

5.2 完整 rules.yaml 範例

# /etc/grafana/provisioning/alerting/rules.yaml
apiVersion: 1

groups:
  - orgId: 1
    name: api-alerts
    folder: Production-Alerts
    interval: 1m
    rules:
      # Rule 1: error rate
      - uid: api-error-rate
        title: APIErrorRate
        condition: C
        data:
          - refId: A
            relativeTimeRange: { from: 600, to: 0 }
            datasourceUid: prometheus-prod
            model:
              refId: A
              expr: 'rate(http_requests_total{job="api",status=~"5.."}[5m]) / rate(http_requests_total{job="api"}[5m])'
              instant: false
              range: true
              intervalMs: 1000
              maxDataPoints: 43200

          - refId: B
            datasourceUid: __expr__
            model:
              refId: B
              type: reduce
              expression: A
              reducer: last
              settings: { mode: dropNN }

          - refId: C
            datasourceUid: __expr__
            model:
              refId: C
              type: threshold
              expression: B
              conditions:
                - evaluator: { type: gt, params: [0.05] }

        noDataState: NoData
        execErrState: Error
        for: 5m
        annotations:
          summary: 'API error rate is {{ $values.B | printf "%.2f%%" }}'
          runbook_url: 'https://wiki/api-errors'
          dashboard_url: 'https://grafana/d/api-overview'
        labels:
          severity: critical
          team: platform
          service: api

5.3 完整 contact-points.yaml

# /etc/grafana/provisioning/alerting/contact-points.yaml
apiVersion: 1

contactPoints:
  - orgId: 1
    name: slack-platform
    receivers:
      - uid: slack-platform-alerts
        type: slack
        settings:
          url: $__file{/etc/grafana/secrets/slack-webhook}
          recipient: '#platform-alerts'
          title: '{{ template "slack.title" . }}'
          text: '{{ template "slack.message" . }}'
        disableResolveMessage: false

  - orgId: 1
    name: pagerduty-critical
    receivers:
      - uid: pd-critical
        type: pagerduty
        settings:
          integrationKey: $__file{/etc/grafana/secrets/pd-key}
          severity: '{{ .CommonLabels.severity }}'
          class: '{{ .CommonLabels.alertname }}'
          component: '{{ .CommonLabels.service }}'
          group: '{{ .CommonLabels.team }}'

  - orgId: 1
    name: email-fallback
    receivers:
      - uid: email-fallback
        type: email
        settings:
          addresses: 'oncall@example.com;backup@example.com'
          singleEmail: false

5.4 完整 policies.yaml

# /etc/grafana/provisioning/alerting/policies.yaml
apiVersion: 1

policies:
  - orgId: 1
    receiver: email-fallback
    group_by: [alertname, cluster]
    group_wait: 30s
    group_interval: 5m
    repeat_interval: 12h

    routes:
      # Critical → PD + Slack（同時發）
      - receiver: pagerduty-critical
        matchers:
          - severity = critical
        group_wait: 10s
        repeat_interval: 1h
        continue: true              # ← 繼續 match sibling

      - receiver: slack-platform
        matchers:
          - severity = critical
        continue: true

      # Warning → 只 Slack
      - receiver: slack-platform
        matchers:
          - severity =~ "warning|info"
        group_by: [alertname, namespace]
        repeat_interval: 4h
        mute_time_intervals:
          - maintenance-window

5.5 完整 mute-timings.yaml

# /etc/grafana/provisioning/alerting/mute-timings.yaml
apiVersion: 1

muteTimes:
  - orgId: 1
    name: maintenance-window
    time_intervals:
      - times:
          - start_time: '02:00'
            end_time: '04:00'
        weekdays: ['saturday']
        location: 'Asia/Taipei'

  - orgId: 1
    name: weekends
    time_intervals:
      - weekdays: ['saturday', 'sunday']
        location: 'Asia/Taipei'

5.6 完整 templates.yaml

# /etc/grafana/provisioning/alerting/templates.yaml
apiVersion: 1

templates:
  - orgId: 1
    name: slack.title
    template: |
      {{ define "slack.title" -}}
      {{ if eq .Status "firing" }}🔥{{ else }}✅{{ end }}
      [{{ .Status | toUpper }}{{ if eq .Status "firing" }}:{{ .Alerts.Firing | len }}{{ end }}]
      {{ .CommonLabels.alertname }}
      {{- end }}

  - orgId: 1
    name: slack.message
    template: |
      {{ define "slack.message" -}}
      {{ range .Alerts -}}
      *Severity:* {{ .Labels.severity }}
      *Service:* {{ .Labels.service }}
      *Summary:* {{ .Annotations.summary }}
      {{ if .Annotations.runbook_url -}}
      📖 <{{ .Annotations.runbook_url }}|Runbook>
      {{ end -}}
      {{ end -}}
      {{- end }}

§ 6. grafanactl（v12+）

6.1 grafanactl 是什麼

Grafana 12 新增的 CLI，類 kubectl 風格。給 IaC 工作流設計：

  • 從 remote Grafana CRUD resources
  • 支援 YAML / JSON 格式
  • 多 context（管多 Grafana instance）
  • 跟 Foundation SDK / CI/CD 整合

6.2 安裝

# macOS
brew install grafana/grafana/grafanactl

# Linux
curl -sSL \
  https://github.com/grafana/grafanactl/releases/latest/download/grafanactl_linux_amd64.tar.gz \
  | tar xz
sudo mv grafanactl /usr/local/bin/

# 驗證
grafanactl version

6.3 設定 Context

# 建 production context
grafanactl config set-context production \
  --server=https://grafana.example.com \
  --token=glsa_xxxxxxxx \
  --org-id=1

# 建 staging context
grafanactl config set-context staging \
  --server=https://grafana-staging.example.com \
  --token=glsa_yyyyyyyy

# 切換
grafanactl config use-context production

# 看當前
grafanactl config current-context

# 列出
grafanactl config get-contexts

6.4 常用指令

# Get
grafanactl get dashboards                    # 列出所有
grafanactl get dashboard api-overview        # 特定一個
grafanactl get dashboard api-overview -o yaml > backup.yaml

# Apply（建或更新，IaC 主要用法）
grafanactl apply -f dashboard.yaml
grafanactl apply -f ./dashboards/             # 整個目錄

# Delete
grafanactl delete dashboard api-overview

# Diff（看本地與 remote 差異）
grafanactl diff -f dashboard.yaml

# Resources
grafanactl get datasources
grafanactl get folders
grafanactl get alertrules
grafanactl get contactpoints

6.5 跨 context 同步

# Backup production
grafanactl --context=production get dashboards -o yaml > production-dashboards.yaml

# 套用到 staging
grafanactl --context=staging apply -f production-dashboards.yaml

6.6 與 grafana-cli 的職責劃分

工具主要用途
grafana-cliServer admin（plugin / password / encryption）
grafanactlResource CRUD（dashboard / alert / datasource）

§ 7. v1beta1 Resource API

7.1 v1beta1 是什麼

Grafana 12 引入新的Kubernetes-style Resource API。原來 dashboard JSON 是 spec-only， 新版加上 apiVersion + kind + metadata，跟 K8s 完全同樣 pattern。

7.2 K8s-style YAML 範例

# dashboard.yaml（v1beta1 格式）
apiVersion: dashboard.grafana.app/v1beta1
kind: Dashboard
metadata:
  name: api-overview            # = uid
  namespace: default            # 對應 org
  labels:
    team: platform
    service: api
  annotations:
    grafana.app/folder: production
spec:
  title: API Service Overview
  tags: [production, service]
  refresh: 30s
  time:
    from: now-6h
    to: now
  panels:
    - id: 1
      type: timeseries
      title: Request Rate
      gridPos: { h: 8, w: 12, x: 0, y: 0 }
      targets:
        - expr: 'rate(http_requests_total[$__rate_interval])'
          datasource: { uid: prometheus-prod }

7.3 用 grafanactl 套用

grafanactl apply -f dashboard.yaml

# Output:
# dashboard/api-overview created
# 或
# dashboard/api-overview unchanged
# 或
# dashboard/api-overview updated

7.4 跟 K8s ecosystem 整合

K8s-style API 讓 Grafana resources 可以：

  • 用 kubectl apply -f 直接套用（透過 Operator）
  • 用 ArgoCD / Flux 做 GitOps
  • 用 Kustomize 做環境覆蓋
  • 用 Helm 做模板化

Ch 44 詳述 Operator。

§ 8. Git Sync（v12 GA）

8.1 Git Sync 是什麼

Grafana 12 GA 的功能：Grafana 直接連 GitHub repo， 結合「UI 編輯方便」與「Git 版本控管」。

8.2 設定流程

  1. 啟用 Git Sync（feature flag provisioning 預設啟用）
  2. 到 Administration → Repositories
  3. 連 GitHub（Personal Access Token 或 GitHub App）
  4. 選 repo + branch + path
  5. 選 sync 方向：
    • Read-only：只從 Git 讀進 Grafana
    • Bidirectional：UI 改寫回 PR

8.3 工作流

graph LR USER[Grafana User] GUI[Grafana UI] REPO[GitHub Repo] PR[Pull Request] REVIEW[Code Review] MERGE[Merge to main] USER --> GUI GUI -- 編輯 dashboard --> PR PR --> REVIEW REVIEW --> MERGE MERGE --> REPO REPO -- sync --> GUI

8.4 Git Sync vs Provisioning vs grafanactl

面向ProvisioninggrafanactlGit Sync
檔案位置Grafana server 本地遠端 pushGit repo
需要Grafana accessAPI tokenGitHub access
UI 修改選擇性 sync 回不影響產生 PR
適合傳統 opsCI/CDUI-first 團隊
v12 推薦仍可用✅✅

§ 9. CI/CD Pipeline 範例

9.1 GitHub Actions 部署 Dashboard

# .github/workflows/deploy-grafana.yml
name: Deploy Grafana Dashboards

on:
  push:
    branches: [main]
    paths:
      - 'grafana/**'

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install grafanactl
        run: |
          curl -sSL https://github.com/grafana/grafanactl/releases/latest/download/grafanactl_linux_amd64.tar.gz | tar xz
          sudo mv grafanactl /usr/local/bin/

      - name: Configure context
        env:
          GRAFANA_TOKEN: ${{ secrets.GRAFANA_TOKEN }}
        run: |
          grafanactl config set-context production \
            --server=https://grafana.example.com \
            --token=$GRAFANA_TOKEN
          grafanactl config use-context production

      - name: Diff (preview)
        run: grafanactl diff -f grafana/dashboards/

      - name: Apply
        run: grafanactl apply -f grafana/dashboards/

      - name: Notify Slack
        if: always()
        uses: 8398a7/action-slack@v3
        with:
          status: ${{ job.status }}

9.2 多環境 deploy

# .github/workflows/multi-env.yml
on:
  push:
    branches: [main, staging]

jobs:
  deploy:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        env: [staging, production]
    steps:
      - name: Set context
        run: |
          if [ "${{ matrix.env }}" = "production" ]; then
            TOKEN=${{ secrets.GRAFANA_PROD_TOKEN }}
            URL=https://grafana.example.com
          else
            TOKEN=${{ secrets.GRAFANA_STAGING_TOKEN }}
            URL=https://grafana-staging.example.com
          fi
          grafanactl config set-context ${{ matrix.env }} \
            --server=$URL --token=$TOKEN

      - name: Apply
        run: |
          grafanactl --context=${{ matrix.env }} \
            apply -f grafana/dashboards/${{ matrix.env }}/

9.3 PR Preview

# 在 PR 建立時跑 diff，留 comment
on:
  pull_request:
    paths: ['grafana/**']

jobs:
  diff:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install grafanactl
        run: ...
      - name: Diff
        id: diff
        run: |
          OUTPUT=$(grafanactl diff -f grafana/dashboards/)
          echo "diff<> $GITHUB_OUTPUT
          echo "$OUTPUT" >> $GITHUB_OUTPUT
          echo "EOF" >> $GITHUB_OUTPUT

      - name: PR Comment
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: '## Grafana Diff\n\n```\n${{ steps.diff.outputs.diff }}\n```'
            })

§ 10. 三方法的選擇

10.1 Provisioning vs grafanactl vs Git Sync

場景推薦
單一 Grafana、ops team 主導Provisioning（簡單）
多環境、CI/CD pipelinegrafanactl + GitHub Actions
UI-first 團隊、PM 也參與Git Sync（v12+）
K8s 環境Operator（Ch 44）
Multi-Grafana 大企業Terraform（Ch 44）

10.2 漸進採用

  1. 第 1 階段：Provisioning datasources（最容易）
  2. 第 2 階段：dashboard JSON 從 Git 載入（Provisioning 或 Git Sync）
  3. 第 3 階段：Alerting Provisioning（rules + contact points + policies）
  4. 第 4 階段：CI/CD pipeline（grafanactl）
  5. 第 5 階段：複雜邏輯改用 Foundation SDK（Ch 43）
  6. 第 6 階段：多 Grafana 用 Terraform（Ch 44）

10.3 反模式

⚠️ 常見反模式
  • UI 與 Provisioning 混用：UI 改的不在 Git，下次 reload 全沒
  • 沒設 uid：每次 Provisioning 創新 dashboard
  • secret 寫在 YAML：用 $__file{} 從外部讀
  • 太多 datasource UID 寫死：dashboard 不能跨環境
  • Provisioning 但不版本控管：失去 IaC 意義

  • Grafana 12 是「Observability as Code」分水嶺：v1beta1 API + grafanactl + Foundation SDK + Git Sync GA
  • Provisioning 5 大目錄：datasources / dashboards / alerting / notifiers / plugins
  • Reload API：POST /api/admin/provisioning/{type}/reload 不用重啟
  • Datasource Provisioning：uid 必設、editable=false、secret 用 $__file{}
  • Dashboard Provisioning：foldersFromFilesStructure 從子目錄自動建 folder
  • Alerting Provisioning 5 個 YAML：rules / contact-points / policies / mute-timings / templates
  • grafanactl（v12+）類 kubectl，多 context、apply / get / diff / delete
  • v1beta1 Resource API：K8s-style YAML，跟 ArgoCD / Operator 整合
  • Git Sync（v12 GA）：UI 編輯產生 PR，結合 UI 方便與 Git 控管
  • CI/CD：GitHub Actions + grafanactl，多環境矩陣，PR diff preview
  • 反模式：UI/Provisioning 混用、沒 uid、secret 寫在 YAML、UID 寫死、不放 Git
