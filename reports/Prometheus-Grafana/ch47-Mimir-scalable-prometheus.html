Chapter 47 · Mimir Scale-out Prometheus

Mimir：橫向擴展的 Prometheus 後端

LGTM Stack 的「M」：Mimir。Grafana Labs 從 Cortex fork 出來的多租戶、橫向擴展的 metric 後端， 可規模化到 10 億 active series。本章涵蓋為什麼 Prometheus 規模上會撞牆、Mimir 架構、多租戶、 Ruler、與 Thanos / VictoriaMetrics 對比、從 Cortex 遷移。

§ 1. 為什麼需要 Mimir

1.1 Prometheus 規模上的痛點

單一 Prometheus 在規模成長時撞到三道牆：

痛點規模門檻
記憶體限制 10M active series（30-50 GB RAM）
長期儲存 本地 SSD 不適合超過 30 天
HA 跑兩台是 HA，但 query 不去重
多租戶 每團隊各 deploy 一台 Prom 變運維噩夢
Federation 慢 查 100 台 Prom 不切實際

1.2 Federation vs 集中式

Federation（Ch 17 介紹過）讓上層 Prometheus 從下層拉聚合 metric。問題：

  • 上層 Prom 仍是單機限制
  • 聚合後失去細節（高 cardinality 不可保留）
  • 長期儲存仍受限

1.3 Mimir 解決的問題

  • 1B+ active series：橫向擴展到任意規模
  • 長期儲存：使用 object store（S3 / GCS）
  • 多租戶：單一 cluster 服務多個團隊
  • HA + 去重：自動 dedup HA Prometheus pair
  • 全相容 PromQL：app / dashboard 不用改

§ 2. Mimir 的歷史與位置

2.1 演進路線

2017: Cortex（Weaveworks 創立，CNCF 收）
2022: Grafana Labs fork → Mimir（AGPLv3）
2023+: 兩個 project 並存，Mimir 主導
2025+: Mimir GA，1B series 是新標竿

2.2 為什麼 Grafana Labs fork

  • License 改 AGPLv3（鎖住 SaaS 競爭者）
  • Performance optimizations（聲稱 40x 比 Cortex 快）
  • 新功能：optimized PromQL engine、cardinality analysis
  • 整合 Grafana Stack（Tempo / Loki 同設計理念）

2.3 與 Prometheus 的關係

Mimir 不是取代 Prometheus，是 Prometheus 的「後端」：

應用 → Prometheus (scrape)
       ↓ remote write
       Mimir（長期 + 集中）
       ↑ PromQL query
       Grafana

Prometheus 仍負責 scrape，Mimir 負責長期儲存 + 集中查詢。

2.4 不是所有人都需要

⚠️ 何時不該用 Mimir
  • 單一團隊、< 1M series：純 Prometheus 完全夠用
  • 30 天 retention 內：本地 SSD 比 Mimir 便宜
  • 不需多租戶：複雜度不值得

Mimir 是「規模化解決方案」，小規模反而是負擔。

§ 3. 架構與元件

3.1 元件架構

graph LR PROM[Prometheus · Alloy] DIST[Distributor] INGEST[Ingester] OBJ[(Object Storage
S3 · GCS · Azure)] QF[Query Frontend] QUERIER[Querier] SC[Store Gateway] COMP[Compactor] RULER[Ruler] AM[Alertmanager] USER[User · Grafana] PROM -- remote_write --> DIST DIST -- shards --> INGEST INGEST -- 2h blocks --> OBJ COMP -- compact --> OBJ USER -- PromQL --> QF QF -- splits --> QUERIER QUERIER -- recent --> INGEST QUERIER -- old --> SC SC -- read --> OBJ RULER -- evaluate rules --> QUERIER RULER -- fire alerts --> AM

3.2 主要元件職責

元件職責
Distributor接收 remote_write、validation、HA dedup、shard 到 ingester
Ingester記憶體中 chunks、每 2 小時 flush block 到 object store
Query Frontend切分大查詢、cache、平行化（同 Loki / Tempo 概念）
Querier從 ingester（最近）+ store gateway（歷史）讀資料
Store Gateway從 object store 讀歷史 block（cache index 在記憶體）
Compactor合併小 block、down-sample、retention
Ruler評估 recording / alerting rule
Alertmanager（內建）跟外部 Alertmanager 行為相同

3.3 與 Loki / Tempo 對照

三家 backend 架構高度一致（Grafana Labs 共用設計）：

§ 4. 寫入路徑詳解

4.1 Prometheus remote_write 設定

# prometheus.yml
remote_write:
  - url: http://mimir-distributor:9009/api/v1/push
    headers:
      X-Scope-OrgID: tenant-a       # 指定 tenant
    queue_config:
      max_shards: 200
      capacity: 10000
      max_samples_per_send: 2000
    metadata_config:
      send: true
      send_interval: 1m

4.2 HA Pair Deduplication

常見 setup：兩台 Prometheus 跑相同 scrape config（HA），都送到 Mimir。 Mimir 自動 dedup：

# Prometheus 啟動時加 external labels
external_labels:
  cluster: prod
  __replica__: replica-1     # 第一台
# 另一台
external_labels:
  cluster: prod
  __replica__: replica-2

# Mimir distributor 設定
distributor:
  ha_tracker:
    enable_ha_tracker: true
    kvstore:
      store: consul    # 或 etcd / memberlist
    cluster_label: cluster
    replica_label: __replica__

Mimir 用 consul 記錄「目前哪個 replica 是 leader」，只接受 leader 的資料，其他被 drop。Leader 失聯時自動 failover。

4.3 Ingester 與 Replication

ingester:
  ring:
    replication_factor: 3      # 每 series 寫 3 份

# 寫入：1 個 series → hash → 3 個 ingester
# 任 1 個 ingester 掛仍可寫
# 重啟 ingester 不丟資料（其他 2 個有）

4.4 Block Format

Mimir 使用 Prometheus TSDB block 格式（直接相容）：

blocks/
└── 01HX.../          # ULID block ID
    ├── meta.json
    ├── index
    ├── chunks/
    │   ├── 000001
    │   └── 000002
    └── tombstones

每 block 預設 2 小時，compactor 後合併成 12h、24h、7d 大 block。

§ 5. 查詢路徑詳解

5.1 Query Frontend 切分

大查詢進來，QF 切成小查詢平行跑：

原查詢：
sum(rate(http_requests_total[5m])) by (service)
range: 24h

QF 切成：
- 0-2h, 2-4h, 4-6h, ..., 22-24h（12 個小查詢）
- 平行送給 12 個 querier
- 結果合併

5.2 Cache 機制

QF 有兩層 cache：

  • Results cache：相同查詢直接給快取結果
  • Splits cache：同個小區段不重複算
frontend:
  results_cache:
    backend: memcached
    memcached:
      addresses: dnssrv+_memcached._tcp.memcached.svc:11211

5.3 Querier 路由

Querier 收到請求，決定資料來源：

  • 過去 12 小時：從 ingester 讀（記憶體 + WAL）
  • 12 小時 - 24 小時：從 ingester + store gateway 同時讀（重疊期間）
  • 超過 24 小時：純 store gateway 讀

5.4 Store Gateway 優化

Store Gateway 把 block index 載到記憶體（不是 chunks）：

1. Querier 問：「找 service=api, 過去 7 天」
2. Store Gateway 從記憶體 index 找出符合的 block list
3. 從 S3 download 需要的 chunks（可 cache）
4. 解析 chunk 回給 querier

5.5 Limits 控制

limits:
  max_query_length: 720h           # 30 天
  max_query_lookback: 8760h        # 1 年
  max_query_parallelism: 14
  max_series_per_query: 100000
  max_samples_per_query: 50000000

  # Per-tenant 可 override
overrides:
  tenant-power-user:
    max_query_length: 8760h        # 1 年

§ 6. Multi-Tenancy

6.1 啟用 multi-tenancy

multitenancy_enabled: true

6.2 Tenant identifier

所有 API 都需要 X-Scope-OrgID header：

# 寫入
curl -X POST \
  -H "X-Scope-OrgID: tenant-a" \
  -H "Content-Type: application/x-protobuf" \
  --data-binary @data \
  http://mimir:9009/api/v1/push

# 查詢
curl -H "X-Scope-OrgID: tenant-a" \
  "http://mimir:9009/prometheus/api/v1/query?query=up"

6.3 Per-Tenant Overrides

overrides:
  tenant-a:
    ingestion_rate: 100000
    max_global_series_per_user: 1000000
    compactor_blocks_retention_period: 720h    # 30 天

  tenant-b:
    ingestion_rate: 500000
    max_global_series_per_user: 5000000
    compactor_blocks_retention_period: 8760h   # 1 年（高級客戶）

6.4 Tenant Federation

v2.4+ 支援查多 tenant：

# 多 tenant query header
X-Scope-OrgID: tenant-a|tenant-b|tenant-c

# 結果聚合多 tenant 資料
# 用於 admin 看全域 metric

6.5 Grafana 整合

# Datasource per tenant
- name: Mimir-TenantA
  type: prometheus
  url: http://mimir:9009/prometheus
  jsonData:
    prometheusType: Mimir
    httpHeaderName1: 'X-Scope-OrgID'
  secureJsonData:
    httpHeaderValue1: 'tenant-a'

§ 7. Mimir Ruler

7.1 為什麼用 Ruler

Prometheus 自己也有 rule（Ch 14）。為什麼需要 Mimir Ruler？

  • 規模化：rule 太多時 Prometheus 跑不動
  • 多租戶：每 tenant 自己的 rule
  • HA：多副本評估，去重
  • 分散式：rule 自動分到多 ruler instance

7.2 Ruler 設定

ruler:
  alertmanager_url: http://alertmanager:9093

  rule_path: /tmp/mimir-rules

  storage:
    backend: s3
    s3:
      bucket_name: mimir-ruler
      region: us-east-1

  ring:
    kvstore:
      store: memberlist

  # 每個 ruler 評估自己負責的 rule
  enable_sharding: true

7.3 上傳 Rule

Ruler 從 object store 載 rule，可用 mimirtool 管理：

# 安裝 mimirtool
curl -L -O https://github.com/grafana/mimir/releases/latest/download/mimirtool-linux-amd64
chmod +x mimirtool-linux-amd64
sudo mv mimirtool-linux-amd64 /usr/local/bin/mimirtool

# 設定
export MIMIR_ADDRESS=https://mimir.example.com
export MIMIR_TENANT_ID=tenant-a
export MIMIR_API_KEY=...

# Rule 內容（標準 Prometheus rule 格式）
cat > rules.yaml <<EOF
groups:
  - name: api-alerts
    interval: 30s
    rules:
      - alert: HighErrorRate
        expr: rate(http_5xx[5m]) / rate(http_total[5m]) > 0.05
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "High error rate"
EOF

# 上傳
mimirtool rules load --rule-files=rules.yaml

# 列出
mimirtool rules list

# 刪除
mimirtool rules delete --rule-group=api-alerts

7.4 從 Grafana 寫 Rule（v12+）

Grafana 12 引入「Recording Rules in UI」，可從 Grafana 直接寫 rule 給 Mimir。 Datasource 設 manageAlerts: true 啟用。

7.5 Rule Sharding

Ruler 多副本時，每個 ruler 負責一部分 rule（hash by group name）。 一個 rule group 完整在一個 ruler 跑（避免分裂）。

ingester:
  zoneAwareReplication:
    enabled: true
    topologyKey: topology.kubernetes.io/zone
    zones:
      - name: zone-a
        nodeSelector:
          topology.kubernetes.io/zone: us-east-1a
      - name: zone-b
        nodeSelector:
          topology.kubernetes.io/zone: us-east-1b
      - name: zone-c
        nodeSelector:
          topology.kubernetes.io/zone: us-east-1c

# 3 replication × 3 zone → 寫 3 份分散在 3 zones
# 1 zone 整個掛仍能寫

§ 9. 從 Prometheus / Cortex 遷移

9.1 從 Prometheus 遷移

  1. 部署 Mimir cluster
  2. Prometheus 加 remote_write 送到 Mimir
  3. 觀察一週確認資料正確
  4. Grafana 加 Mimir datasource
  5. 切換 dashboard 用 Mimir datasource（或設 default）
  6. 視需求縮減本地 Prometheus retention（短期 buffer 即可）

9.2 從 Cortex 遷移

Mimir 從 Cortex fork，config 大致相容。Grafana Labs 提供 mimirtool 自動轉： 

mimirtool config convert \
  --yaml-file=cortex-config.yaml \
  --output-file=mimir-config.yaml

9.3 Schema 不變

Mimir 用 Prometheus TSDB block，Cortex 的 block 直接讀得到。 遷移時 storage 不必動，只需切換 binary。

10.4 Thanos 的不同模型

Thanos 設計很不一樣：每個 Prometheus 旁邊跑 sidecar，sidecar 把 block 上傳到 object store， Thanos Query 從多個 sidecar + store gateway 拉資料聚合。

  • 優點：不用改 Prometheus，只加 sidecar
  • 缺點：query 時要連多個 source，慢

11.4 Cardinality Limits

limits:
  max_global_series_per_user: 1000000      # 全 cluster series 上限
  max_global_series_per_metric: 100000     # 單 metric series 上限
  max_label_names_per_series: 30
  max_label_name_length: 1024
  max_label_value_length: 2048

11.5 Mimir 看自己（meta-monitoring）

用另一個 Prometheus（或 Mimir 不同 tenant）看 Mimir 自身 metric，避免「Mimir 掛了沒人知道」。

11.6 容量規劃

# 估算
ingester memory ≈ active_series × 8KB
ingester WAL    ≈ active_series × samples × bytes_per_sample
storage         ≈ samples_per_day × bytes_per_sample × retention_days × 0.3 (壓縮)

# 範例：10M series, 15s scrape, 30 天
ingester memory = 10M × 8KB = 80GB（across all ingester）
samples/day     = 10M × 5760 (6/min) = 57.6B
storage/day     = 57.6B × 1.7 bytes × 0.3 = 30 GB compressed
30 day storage  = 900 GB
