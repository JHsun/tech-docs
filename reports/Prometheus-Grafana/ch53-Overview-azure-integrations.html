Chapter 53 · Azure 整合

Azure 觀測性整合：Azure Monitor、Managed Prometheus、Managed Grafana

§ 1. Azure 觀測性全景

1.1 Azure Monitor 是統稱

跟 AWS / GCP 不同，Azure 用「Azure Monitor」當整體觀測性平台的名字， 底下分多個子服務：

子服務用途
Azure Monitor Metrics原生 metric（類 CloudWatch / Cloud Monitoring）
Log Analyticslog + 進階 query，用 KQL（Kusto Query Language）
Application InsightsAPM、front-end RUM、分散式 tracing
Azure Monitor Managed PrometheusManaged Prometheus 服務
Azure Managed GrafanaManaged Grafana（v12 GA 2026-03）
Azure Monitor WorkspacePrometheus metric 存放（類 AMP workspace）

1.2 與 AWS / GCP 的差異

Azure 強的地方：

  • Application Insights 是業界第一批 APM 服務（2015）
  • KQL（Kusto）是 Azure 獨家強大查詢語言，跨多種資料源
  • Azure Monitor Managed Grafana 與 AKS 整合最深
  • 所有三大雲中，免費 Grafana dashboard 體驗最好（Azure Monitor dashboards with Grafana，內嵌 portal）

Azure 弱的地方：

  • Managed Prometheus 比 GMP / AMP 晚（2024 GA）
  • 原生 trace 透過 Application Insights，與 OTel 整合需多一層
  • Profiling 功能弱（沒有專屬服務）

1.3 推薦組合

✅ Azure 推薦組合
  • Metrics：Azure Monitor Managed Prometheus + Azure Managed Grafana
  • Logs：Log Analytics（KQL 強大）
  • Application APM / Trace：Application Insights + OpenTelemetry
  • Profile：自架 Pyroscope
  • 視覺化：Azure Managed Grafana 連所有家

§ 2. Azure Monitor 核心

2.1 Azure Monitor Metrics

原生 metric 系統，自動收集 Azure 資源 metric（VM / Storage / SQL / AKS 等）。

2.2 Metric 結構

Resource: /subscriptions/.../providers/Microsoft.Compute/virtualMachines/my-vm
  └── Namespace: Microsoft.Compute/virtualMachines
      └── Metric: Percentage CPU
          └── Dimensions: instanceName
              └── Aggregation: Average / Total / Min / Max
                  └── Granularity: PT1M (1 min)

2.3 自訂 metric

# 用 Azure SDK 寫
from azure.monitor.ingestion import LogsIngestionClient

# 或用 Application Insights Custom Metrics（推薦）
from opencensus.ext.azure import metrics_exporter
exporter = metrics_exporter.new_metrics_exporter(
    connection_string='InstrumentationKey=...'
)

2.4 Diagnostic Settings

把 Azure 服務的 metric / log 路由到不同地方：

az monitor diagnostic-settings create \
  --name my-diag \
  --resource /subscriptions/.../my-vm \
  --logs '[{"category":"Administrative","enabled":true}]' \
  --metrics '[{"category":"AllMetrics","enabled":true}]' \
  --workspace /subscriptions/.../my-log-analytics \
  --storage-account /subscriptions/.../my-storage

2.5 限制

  • 原生 metric retention：93 天
  • 解析度：1 分鐘（部分 service 1 秒）
  • 自訂 metric dimensions 限制：每 metric 最多 10

§ 3. Log Analytics 與 KQL

3.1 Log Analytics 是什麼

Azure 的 log + 進階 telemetry 系統。資料存在 Log Analytics Workspace。 用 KQL（Kusto Query Language）查詢，是 Azure 觀測性的核心。

3.2 KQL 簡介

KQL 是 Microsoft 的查詢語言，類似 SQL 但更強大。同時用於 Log Analytics、Azure Data Explorer、 Microsoft Defender、Azure Sentinel 等。

3.3 KQL 基本語法

// 查最近 1 小時的 ERROR log
AppExceptions
| where TimeGenerated > ago(1h)
| where SeverityLevel >= 3
| project TimeGenerated, OperationName, OuterMessage
| order by TimeGenerated desc
| take 100

// 統計每 5 分鐘的 request 數
AppRequests
| where TimeGenerated > ago(24h)
| summarize Count = count() by bin(TimeGenerated, 5m), Name
| render timechart

// Join 多個 table
AppRequests
| join kind=leftouter (
    AppExceptions
    | project OperationId, ExceptionType
) on OperationId

// 解析 JSON
AppTraces
| extend payload = parse_json(Properties.payload)
| where payload.user_id == "12345"

3.4 KQL 與 PromQL / LogQL 對比

面向KQLPromQLLogQL
類型Pipeline-basedFunctionalPipeline-based
適合log + metricmetric onlylog only
SQL-like有點像不像類似 LogQL/Loki
學習曲線中陡（functional 風格）低
跨 table join強有限無

3.5 從 Grafana 查 Log Analytics

# Datasource: Azure Monitor
# Service: Logs
# Query 用 KQL：

AppRequests
| where TimeGenerated > $__timeFrom() and TimeGenerated < $__timeTo()
| summarize Count = count() by bin(TimeGenerated, $__timeInterval), Name
| render timechart

§ 4. Application Insights

4.1 Application Insights 是什麼

Azure 的 APM（Application Performance Monitoring）服務。歷史可追溯到 2015， 是業界最早的雲原生 APM 之一。提供：

  • Application metric（QPS / latency / errors）
  • Distributed tracing
  • Browser RUM（前端 telemetry）
  • Live Metrics Stream（即時監控）
  • Smart Detection（AI-based anomaly detection）

4.2 整合方式

方式說明
Auto-instrumentation不改 code（.NET、Java、Node、Python on App Service / Functions / AKS）
SDK傳統 ApplicationInsights SDK（深度自訂）
OpenTelemetry SDK用 OTel + Azure Monitor exporter（推薦）

4.3 OpenTelemetry 整合範例（Python）

from azure.monitor.opentelemetry import configure_azure_monitor

configure_azure_monitor(
    connection_string="InstrumentationKey=...",
)

# 之後一切自動：metric / trace / log 都送到 Application Insights

4.4 跟 Tempo / Jaeger 對比

面向Application InsightsTempo + Grafana
運維Azure 全管自架
UIAzure Portal（強大但封閉）Grafana（開放、可自訂）
Smart Detection有（AI 異常偵測）無
Live Metrics Stream有（即時 1s 解析度）無
跨雲否可
OTel 相容透過 exporter原生

4.5 從 Grafana 查 Application Insights

# Datasource: Azure Monitor
# Service: Application Insights
# Resource: 選擇 AppInsights 資源
# 用 KQL 或 metric query

# 範例 KQL
AppRequests
| where AppRoleName == "my-api"
| summarize Avg=avg(DurationMs), P95=percentile(DurationMs, 95)
  by bin(TimeGenerated, 1m)

§ 5. Azure Monitor Managed Prometheus

5.1 服務定位

全名 Azure Monitor Managed Service for Prometheus（簡稱 Azure Managed Prometheus）。 2024 年 GA，是三大雲中最晚推出的 Managed Prometheus，但功能完整。

5.2 與 AWS AMP / GCP GMP 對比

Azure 的儲存免費 18 個月是獨特優勢。但 query 收費。

5.3 建立 Workspace

# 建 Azure Monitor Workspace（給 Prometheus metric）
az monitor account create \
  --name my-prometheus-workspace \
  --resource-group my-rg \
  --location eastus

# 取得 query endpoint
az monitor account show \
  --name my-prometheus-workspace \
  --resource-group my-rg \

5.4 從 AKS 收 metric（Managed mode）

# 啟用 Managed Prometheus on AKS
az aks update \
  --name my-cluster \
  --resource-group my-rg \
  --enable-azure-monitor-metrics \
  --azure-monitor-workspace-resource-id /subscriptions/.../my-prometheus-workspace

# 自動 scrape PodMonitoring / ServiceMonitor

§ 6. Azure Managed Grafana (v12)

6.1 Azure Managed Grafana 是什麼

Azure 的 Managed Grafana 服務，2022 GA。v12 於 2026-03 發布。 與 Azure 整合非常深（Managed Identity、Entra ID auth、Azure Monitor data source 自動配置）。

6.2 v12 重要新功能（2026-03）

  • Current-user Entra authentication：用登入使用者的權限 query data source（個人權限隔離）
  • Azure Monitor logs query builder：免寫 KQL 也能查（GUI）
  • Logs Explorer 性能：可載 30K log records，查詢加速
  • OpenTelemetry mode：Prometheus datasource 加 OTel mode，自動處理 OTLP metric 的 label-join 複雜性
  • Native histogram support：Prometheus native histogram
  • Pre-built database dashboards：PostgreSQL / SQL Database / SQL MI

6.3 與 AWS AMG 對比

面向Azure Managed GrafanaAWS AMG
Grafana 版本v12（2026-03 GA）v12
認證Entra ID（深整合）IAM Identity Center / SAML
原生 datasourceAzure Monitor 自動配置CloudWatch / AMP 自動
PricingPer workspace（不限 user）Per active user
VNet 整合Private endpointVPC endpoint
Plugin限制（Marketplace 認證）限制（Grafana 認證）

Azure Managed Grafana 的 per-workspace pricing（不限 user 數）對大團隊比 AWS AMG 划算。

6.4 建立 Workspace

az grafana create \
  --name my-grafana \
  --resource-group my-rg \
  --location eastus \
  --grafana-version 12   # v12

# 賦權給使用者
az grafana user create \
  --name my-grafana \
  --user user@contoso.com \
  --role Admin

6.5 自動 Datasource 配置

建立 Azure Managed Grafana 時，自動配置：

  • Azure Monitor datasource（含 Managed Identity）
  • Application Insights datasource
  • Log Analytics datasource

Managed Identity 預設有 Monitoring Data Reader role 給 subscription 內所有 monitoring data。

6.6 Azure Monitor dashboards with Grafana（免費版）

Azure 還提供「免費的 Grafana 體驗」內嵌在 Azure Portal：

  • 無需建 workspace
  • 免費使用
  • 支援 Azure Monitor / Managed Prometheus / Logs / Resource Graph
  • 可 import 社群 dashboard
  • 適合輕量使用、不需要 user management

選擇：純 Azure 環境輕量需求 → 用免費版；複雜需求或要管 user → Azure Managed Grafana。

§ 7. AKS 觀測性

7.1 AKS 觀測性 stack

AKS Cluster
├── Application pods（含 OTel SDK）
├── Container Insights（Azure Monitor agent for Logs）
│   └── 把 stdout / kube-state log 收進 Log Analytics
├── Managed Prometheus addon（ama-metrics DaemonSet）
│   └── 自動 scrape pod / ServiceMonitor / PodMonitor
└── Application Insights（auto-instrumentation 可選）

7.2 一鍵啟用觀測性

# 建 AKS 時就啟用全部
az aks create \
  --name my-cluster \
  --resource-group my-rg \
  --enable-managed-identity \
  --enable-azure-monitor-metrics \
  --azure-monitor-workspace-resource-id /subscriptions/.../my-prometheus-workspace \
  --workspace-resource-id /subscriptions/.../my-log-analytics \
  --enable-app-routing \
  --enable-cluster-autoscaler

7.3 Container Insights

Azure 的 K8s log 收集服務。把 container log + node metric + K8s 事件送到 Log Analytics。 對應 GCP 的 Cloud Logging Agent、AWS 的 CloudWatch Container Insights。

7.4 ama-metrics 設定 ConfigMap

自訂哪些 pod / service 要 scrape：

apiVersion: v1
kind: ConfigMap
metadata:
  name: ama-metrics-prometheus-config
  namespace: kube-system
data:
  prometheus-config: |
    scrape_configs:
      - job_name: my-app
        kubernetes_sd_configs:
          - role: pod
        relabel_configs:
          - source_labels: [__meta_kubernetes_pod_label_app]
            regex: my-app
            action: keep

7.5 OpenTelemetry on AKS

新趨勢：用 OpenTelemetry Collector 統一收集，再送各家 backend：

# Helm 安裝 OTel Collector
helm install otel-collector open-telemetry/opentelemetry-collector \
  --namespace observability \
  -f otel-values.yaml

# otel-values.yaml
config:
  exporters:
    azuremonitor:
      connection_string: "${env:APPLICATIONINSIGHTS_CONNECTION_STRING}"
    prometheusremotewrite:
      endpoint: "${env:AZURE_PROMETHEUS_ENDPOINT}"

  service:
    pipelines:
      traces:
        exporters: [azuremonitor]
      metrics:
        exporters: [prometheusremotewrite]
      logs:
        exporters: [azuremonitor]

§ 8. 認證：Managed Identity 與 Workload Identity

8.1 Azure 賦權的選項

從舊到新：

  1. Connection String / Key（最舊，不推薦）
  2. Service Principal（傳統，仍常見）
  3. System-assigned Managed Identity（VM / App Service）
  4. User-assigned Managed Identity（多資源共用）
  5. Workload Identity（AKS 推薦，2024 GA）

8.2 Managed Identity 簡介

Azure 給資源「免管理憑證」的方法。Azure 自動處理 secret rotation。 應用透過 IMDS（Instance Metadata Service）取 token。 

# 啟用 system-assigned MI for VM
az vm identity assign --name my-vm --resource-group my-rg

# 賦權給 MI
az role assignment create \
  --assignee  \
  --role "Monitoring Data Reader" \
  --scope /subscriptions/.../my-prometheus-workspace

8.3 Workload Identity for AKS

對應 AWS IRSA / GCP Workload Identity，AKS 推薦做法：

# 1. 啟用 OIDC issuer + Workload Identity
az aks update \
  --name my-cluster \
  --enable-oidc-issuer \
  --enable-workload-identity

# 2. 取得 OIDC issuer URL
OIDC_URL=$(az aks show --name my-cluster --query "oidcIssuerProfile.issuerUrl" -o tsv)

# 3. 建 User-assigned Managed Identity
az identity create --name alloy-mi --resource-group my-rg
MI_CLIENT_ID=$(az identity show --name alloy-mi --query clientId -o tsv)
MI_PRINCIPAL_ID=$(az identity show --name alloy-mi --query principalId -o tsv)

# 4. 賦權
az role assignment create \
  --assignee $MI_PRINCIPAL_ID \
  --role "Monitoring Metrics Publisher" \
  --scope /subscriptions/.../my-prometheus-workspace

# 5. 建 federated credential（綁 K8s SA 到 MI）
az identity federated-credential create \
  --name alloy-fc \
  --identity-name alloy-mi \
  --resource-group my-rg \
  --issuer $OIDC_URL \
  --subject "system:serviceaccount:observability:alloy"

# 6. K8s SA 加 annotation
kubectl annotate serviceaccount alloy \
  --namespace observability \
  azure.workload.identity/client-id=$MI_CLIENT_ID

§ 9. 自架 Grafana 連 Azure

9.1 為什麼要自架

  • 跨雲架構（Azure + 其他雲）
  • 需要 Grafana plugin Azure Managed Grafana 不支援
  • 已有 Grafana on K8s
  • Cost 考量（per-workspace 雖好但仍有成本）

9.2 在 AKS 上自架 Grafana 連 Azure Managed Prometheus

關鍵：開啟 Azure Authentication for Prometheus datasource。

Step 1: 設 grafana.ini

[auth]
azure_auth_enabled = true

[azure]
managed_identity_enabled = true   # 啟用 MI 認證
workload_identity_enabled = true  # 啟用 Workload Identity

Step 2: 啟用 Workload Identity（如 §8.3）

Step 3: 設定 Datasource

apiVersion: 1
datasources:
  - name: AzurePrometheus
    type: grafana-azureprometheus-datasource   # 注意：要用 Azure plugin（非 core Prometheus）
    url: https://my-workspace.eastus.prometheus.monitor.azure.com
    jsonData:
      azureCredentials:
        authType: workloadidentity   # 或 msi（managed identity）
      httpMethod: POST

9.3 重要：Plugin 變動

📌 Azure Prometheus Plugin 遷移

Grafana 11.5+ 起，Azure Monitor Managed Prometheus 必須用專屬 plugin grafana-azureprometheus-datasource，不再用 core Prometheus plugin。
Grafana Cloud 自動遷移；自架需啟用 prometheusTypeMigration feature toggle。
Grafana 13 起這個 toggle 移除，遷移自動執行。

9.4 OSS Grafana 限制

Grafana OSS（自架，無 Enterprise）無法直接用 Azure Managed Identity / Workload Identity。 解法：

  • Reverse proxy 處理 Azure AD auth（如 nginx + oauth2-proxy）
  • 用 Grafana Enterprise / Cloud
  • 用 Service Principal + secret（不推薦，需管 secret rotation）
