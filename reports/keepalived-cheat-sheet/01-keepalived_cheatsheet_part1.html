1 Introduction — 簡介

一句話理解：Keepalived = LVS 健康檢查 + VRRP 故障轉移，兩者可獨立或搭配使用。

Keepalived 提供兩大核心功能框架：

⚖️ 負載均衡（Load Balancing）

基於 Linux 核心模組 IPVS（IP Virtual Server），在 Layer 4 進行傳輸層負載分配，並透過健康檢查動態維護後端伺服器池。

IPVS L4 Healthcheck
🔄 高可用（High Availability）

實作 VRRP（Virtual Router Redundancy Protocol）協議，提供路由器等級的故障轉移機制，確保 Director 本身的可用性。

VRRP Failover RFC 2338
💡 在 Keepalived 的語境中，Load Balancer 也常被稱為 Director 或 LVS Router。

2 Software Design — 軟體設計

2.1 行程架構（三行程模型）

Keepalived 以純 ANSI/ISO C 撰寫，拆分為 3 個獨立行程，以 WatchDog 機制互相監控：

PID 111  Keepalived  ← 父行程（監控子行程）
            ├─ 112  Keepalived  ← VRRP 子行程
            └─ 113  Keepalived  ← Healthcheck 子行程
為何分離？VRRP 排程對時序非常敏感，獨立行程可避免健康檢查的 I/O 阻塞影響 VRRP 封包的即時性。WatchDog 每 5 秒透過 Unix Domain Socket 發送心跳，偵測子行程死鎖或異常時自動重啟。

2.2 核心使用的 Linux 核心元件

LVS Framework 透過 getsockopt/setsockopt 操作 IPVS 規則
Netfilter Framework 支援 NAT 與 Masquerading 的 IPVS 程式碼
Netlink Interface 在網路介面上設定/移除 VRRP 的虛擬 IP
Multicast VRRP 通告發送到保留群播位址 224.0.0.18

2.3 內部模組速覽

模組功能說明
Control Plane解析 keepalived.conf，使用關鍵字樹層級對應設定處理器
Scheduler (I/O Multiplexer)核心事件排程器，使用 select() 而非 POSIX Thread，為網路用途最佳化
Memory Management提供 debug_mode 以追蹤記憶體洩漏，所有緩衝區固定長度防止溢位
WatchDog父行程透過 Unix Domain Socket 監控子行程的心跳，偵測 deadloop
Checkers以 FSM（有限狀態機）設計，即時檢查後端伺服器存活，決定加入/移除 LVS 拓撲
VRRP Stack完整實作 RFC 2338，支援 sync_group 與 IPSEC-AH (MD5-96bit) 加密
Netlink Reflector反映核心網路事件到 Keepalived 內部資料結構（RTMGRP_LINK、RTMGRP_IPV4_IFADDR）
SMTP管理通知郵件發送（RFC 821），可對接 GSM-SMS、Pager 等
IPVS Wrapper內部資料表示 ⇄ IPVS rule_user 的轉譯層，使用 libipvs

2.4 Healthcheck 框架 — 四種檢查器

Checker層級原理
TCP_CHECK L4 非阻塞 TCP 連線嘗試，超時即判定失敗
HTTP_GET L5/L7 對指定 URL 發送 HTTP GET，回傳的內容做 MD5 校驗（使用 genhash 工具產生）
SSL_GET L5/L7 同 HTTP_GET，但走 SSL 加密連線
MISC_CHECK 自訂 執行使用者定義的腳本，回傳 0（成功）或 1（失敗）

2.5 VRRP 框架 — 核心功能

  • ✅ Failover — 基於浮動 VRRP VIP 的原生故障轉移
  • ✅ Sync Group — 兩個 VRRP Instance 同步狀態，確保路由路徑一致
  • ✅ IPSEC-AH ICV — 封包完整性驗證，防止偽造通告
  • ✅ Notify Script — 狀態轉換時觸發外部腳本
  • ✅ VMAC Support — 使用 macvlan 驅動建立虛擬 MAC，降低接管影響

2.6 VMAC 使用注意事項

啟用 use_vmac 後，Keepalived 會建立 macvlan 介面（預設名稱 vrrp.{virtual_router_id}），需搭配核心 sysctl 調整：

# 全域設定
net.ipv4.conf.all.arp_ignore = 1
net.ipv4.conf.all.arp_announce = 1
net.ipv4.conf.all.arp_filter = 0

# 實體介面
net.ipv4.conf.eth0.arp_filter = 1

# VMAC 介面
net.ipv4.conf.vrrp.250.arp_filter = 0
net.ipv4.conf.vrrp.250.accept_local = 1
net.ipv4.conf.vrrp.250.rp_filter = 0

3 Load Balancing Techniques — 負載均衡技術

LVS/Keepalived 支援三種轉發模式，各有不同的擴展性與架構需求：

NAT 模式

Director 擁有雙網卡（外網+內網），所有請求與回應都經過 Director 做位址轉譯。

  • ✅ 後端隱藏於內網
  • ✅ 可用私有 IP
  • ❌ Director 成為瓶頸
  • ❌ 請求+回應皆經過
Tunneling 模式

Director 透過 IP 隧道將請求封裝轉發，後端直接回應客戶端。

  • ✅ 高擴展性
  • ✅ 節點可跨網段
  • ❌ 架構複雜度高
  • ❌ 依賴網路拓撲
Direct Routing 模式

Director 修改 MAC 後直接轉發到同網段的後端，後端直接回應客戶端。

  • ✅ 最高擴展性
  • ✅ Director 負擔最輕
  • ❌ ARP 問題需處理
  • ❌ 後端需同網段
💡 效能排序：Direct Routing > Tunneling > NAT。
💡 使用建議：同網段首選 DR 模式；跨機房選 Tunneling；需要隱藏後端選 NAT。

4 Installing Keepalived — 安裝

4.1 套件庫安裝（快速）

🎩 RHEL / CentOS
yum install keepalived

RHEL 6.4+ 已收錄於 base repo

🐧 Debian / Ubuntu
apt-get install keepalived
⚠️ 套件庫版本通常落後數個版本，若需最新功能建議從原始碼編譯。

4.2 原始碼編譯

步驟 ① 安裝相依套件：

RHEL / CentOS

yum install curl gcc openssl-devel \
    libnl3-devel net-snmp-devel

Debian / Ubuntu

apt-get install curl gcc libssl-dev \
    libnl-3-dev libnl-genl-3-dev libsnmp-dev

步驟 ② 下載、編譯、安裝：

curl --progress http://keepalived.org/software/keepalived-X.Y.Z.tar.gz | tar xz
cd keepalived-X.Y.Z
./configure --prefix=/usr/local/keepalived-X.Y.Z
make
sudo make install
版本管理技巧：使用 --prefix 安裝到獨立目錄，再用 symlink 指向想使用的版本：
ln -s keepalived-X.Y.Z /usr/local/keepalived
如此可同時保留多個版本，方便回滾。

步驟 ③ 設定 Init Script：

RHEL

ln -s /etc/rc.d/init.d/keepalived.init \
      /etc/rc.d/rc3.d/S99keepalived

Debian

ln -s /etc/init.d/keepalived.init \
      /etc/rc2.d/S99keepalived
