10 Case Study: Healthcheck — 健康檢查案例

10.1 架構元件

此案例為典型 DMZ 架構中的單 Director LVS 拓撲：

元件角色
LVS Router負載均衡器，擁有 VIP 網段 192.168.100.0/24
Network Router內網預設閘道
DNS Server內網名稱解析
SMTP Server接收告警郵件
Server Pool 1HTTP + SSL 服務（雙應用伺服器：WebSphere + WebLogic）
Server Pool 2SMTP 服務

10.2 設定摘要與解讀

HTTP VIP 192.168.200.15:80
排程：wrr（加權輪詢） / 轉發：NAT / 持續連線：50s / 檢查間隔：30s
Sorry Server：192.168.100.100:80（全部 RS 下線時啟用）
RS 192.168.100.2 weight=2 → HTTP_GET 檢查 2 個 URL（MD5 校驗），timeout=3s，retry=3，delay=2s
RS 192.168.100.3 weight=1 → HTTP_GET 檢查 1 個 URL，同上超時設定
SSL VIP 192.168.200.15:443
排程：rr（輪詢） / 轉發：NAT / 持續連線：360s / 檢查間隔：20s
RS 192.168.100.2 & .3 weight=1 → TCP_CHECK，timeout=3s
💡 可升級為 SSL_GET 檢查，加入 URL + digest 進行應用層驗證
SMTP VIP 192.168.200.15:25
排程：wlc（加權最少連線） / 轉發：NAT / 持續連線：50s / 檢查間隔：15s
RS 192.168.100.4 weight=1，RS 192.168.100.5 weight=2 → TCP_CHECK，timeout=3s

10.3 完整設定檔

global_defs {
    notification_email { admin@domain.com; 0633225522@domain.com }
    notification_email_from keepalived@domain.com
    smtp_server            192.168.200.20
    smtp_connect_timeout   30
    lvs_id                 LVS_MAIN
}

virtual_server 192.168.200.15 80 {
    delay_loop 30
    lb_algo    wrr
    lb_kind    NAT
    persistence_timeout 50
    protocol   TCP
    sorry_server 192.168.100.100 80

    real_server 192.168.100.2 80 {
        weight 2
        HTTP_GET {
            url { path /testurl/test.jsp
                  digest ec90a42b99ea9a2f5ecbe213ac9eba03 }
            url { path /testurl2/test.jsp
                  digest 640205b7b0fc66c1ea91c463fac6334c }
            connect_timeout 3
            retry 3
            delay_before_retry 2
        }
    }
    real_server 192.168.100.3 80 {
        weight 1
        HTTP_GET {
            url { path /testurl/test.jsp
                  digest 640205b7b0fc66c1ea91c463fac6334c }
            connect_timeout 3
            retry 3
            delay_before_retry 2
        }
    }
}

# SSL 與 SMTP virtual_server 結構類似，略

10.4 產生 MD5 Digest

genhash -s 192.168.100.2 -p 80 -u /testurl/test.jsp
# 輸出最後一行即為 MD5 Digest，貼到設定檔的 digest 欄位

11 Case Study: Failover — VRRP 故障轉移案例

11.1 架構規格

目標：建立 Active-Active 的虛擬 LVS Director，使用 VRRPv2 + IPSEC-AH 認證。

兩台 LVS Director 各有 4 個 VRRP Instance，彼此對稱配置：

VRRP InstanceDirector 1 狀態Director 2 狀態擁有的 IP同步夥伴
VI_1 (VRID 51)MASTER P:150BACKUP P:100VIP1(200.10), VIP2(200.11)VI_2
VI_2 (VRID 52)MASTER P:150BACKUP P:100DIP1(100.10)VI_1
VI_3 (VRID 53)BACKUP P:100MASTER P:150VIP3(200.12), VIP4(200.13)VI_4
VI_4 (VRID 54)BACKUP P:100MASTER P:150DIP2(100.11)VI_3

11.2 Sync Group 的行為

關鍵機制：VG1 = {VI_1, VI_2}、VG2 = {VI_3, VI_4}。

若 Director 1 的 eth0 故障 → VI_1 在 Director 2 變為 MASTER → 因為 sync_group，VI_2 也被迫跟著切換到 BACKUP → 最終 Director 2 擁有所有 4 個 Instance。

當 eth0 恢復 → 自動回到初始對稱分佈。

11.3 Director 1 設定檔

vrrp_sync_group VG1 {
    group { VI_1; VI_2 }
}
vrrp_sync_group VG2 {
    group { VI_3; VI_4 }
}

vrrp_instance VI_1 {
    state             MASTER
    interface         eth0
    virtual_router_id 51
    priority          150
    advert_int        1
    authentication { auth_type AH; auth_pass k@l!ve1 }
    virtual_ipaddress { 192.168.200.10; 192.168.200.11 }
}

vrrp_instance VI_2 {
    state             MASTER
    interface         eth1
    virtual_router_id 52
    priority          150
    advert_int        1
    authentication { auth_type AH; auth_pass k@l!ve2 }
    virtual_ipaddress { 192.168.100.10 }
}

# VI_3, VI_4 在 Director 1 上為 BACKUP / priority 100（對稱配置）

11.4 驗證指令

# 查看 VIP 綁定狀態
ip address list

# 預期結果（Director 1 啟動、Director 2 未啟動時）：
# eth0 → 200.10, 200.11, 200.12, 200.13
# eth1 → 100.10, 100.11

# 兩台皆啟動後：
# Director 1: eth0 → 200.10, 200.11  |  eth1 → 100.10
# Director 2: eth0 → 200.12, 200.13  |  eth1 → 100.11
💡 此 VRRP 設定不限於 LVS，可套用於任何需要 Active-Active 高可用的 Linux 路由器場景。

12 Case Study: Mixing — 混合部署案例

核心概念：將 VRRP VIP 直接作為 LVS 的 VIP，實現「高可用 + 負載均衡」的完整方案。

12.1 架構特點

沿用 Failover 案例的對稱 VRRP 配置，在此基礎上加入 virtual_server 區塊。兩台 Director 各自負責不同服務：

🖥 Director 1 (MASTER for VG1)

VIP 192.168.200.10:80 → HTTP 服務
排程 wrr / NAT / 持續連線 50s
RS: .2(w=2, HTTP_GET), .3(w=1, HTTP_GET)

🖥 Director 2 (MASTER for VG2)

VIP 192.168.200.12:443 → SSL 服務
排程 rr / NAT / 持續連線 360s
RS: .2(w=1, TCP_CHECK), .3(w=1, TCP_CHECK)

12.2 設定結構

# ──── VRRP 區塊（與 Failover 案例相同）────
vrrp_sync_group VG1 { group { VI_1; VI_2 } }
vrrp_sync_group VG2 { group { VI_3; VI_4 } }
vrrp_instance VI_1 { ... virtual_ipaddress { 192.168.200.10; 192.168.200.11 } }
... 略 ...

# ──── Virtual Server 區塊（新增）────
virtual_server 192.168.200.10 80 {
    delay_loop 30
    lb_algo wrr
    lb_kind NAT
    sorry_server 192.168.100.100 80
    real_server 192.168.100.2 80 { weight 2; HTTP_GET { ... } }
    real_server 192.168.100.3 80 { weight 1; HTTP_GET { ... } }
}

virtual_server 192.168.200.12 443 {
    delay_loop 20
    lb_algo rr
    lb_kind NAT
    real_server 192.168.100.2 443 { weight 1; TCP_CHECK { connect_timeout 3 } }
    real_server 192.168.100.3 443 { weight 1; TCP_CHECK { connect_timeout 3 } }
}
運作原理：VRRP 確保 VIP 始終存在於某一台 Director 上，而 virtual_server 區塊確保擁有該 VIP 的 Director 會對後端 RS 做健康檢查與流量分發。當 VRRP 故障轉移時，VIP 與其對應的 LVS 規則一起飄移。

13 Terminology — 術語速查

13.1 LVS 元件術語

VIP（Virtual IP）
客戶端存取的虛擬 IP 位址，是服務的唯一入口。
Real Server
實際提供服務的後端伺服器（例如 Web Server）。
Server Pool
一組 Real Server 的集合，又稱伺服器農場（Farm）。
Virtual Server
Server Pool 的存取端點，將 VIP + Port 對應到一組 Real Server。
Virtual Service
與 VIP 關聯的特定 TCP/UDP 服務。
Director / LVS Router
負載均衡器本身，負責接收與分發流量。

13.2 VRRP 元件術語

VRRP
Virtual Router Redundancy Protocol，用於 Director 故障轉移的協議。
MASTER 狀態
正在負責轉發封包的 VRRP Instance，圖示中通常以紅線表示。
BACKUP 狀態
待命中的 VRRP Instance，MASTER 失效時接手。
IP Address Owner
以 VIP 作為真實介面位址的 VRRP Instance，會回應 ICMP/TCP 連線。
Advertisement
MASTER 定期發送的 VRRPv2 封包（通告），告知 BACKUP 自己仍存活。
Synchronized Instance
透過 sync_group 綁定的 VRRP Instance，確保成組切換。
Real Load Balancer
執行一或多個 VRRP Instance 的實體 LVS Director。
Virtual Load Balancer
一組 Real Load Balancer 組成的虛擬負載均衡器（HA 集合）。
