§ 3.3.1

Proxying — 代理轉發

🔗 核心概念

Proxying = 在 client 與 server 之間建立兩條獨立連線進行資料傳輸。

📋 14 項關鍵能力

能力說明
乾淨連線保護隔離 client 端缺陷/攻擊，server 只收到正常連線
多重監聽多 IP、多 port、甚至 port 範圍
Transparent Accept攔截目的地為任意 IP 的流量（不需本機擁有該 IP）
Server port 轉換後端 port 可與監聽 port 不同，支援固定偏移
Transparent Connect連線後端時偽裝客戶端 IP
多站點回傳 IP確保 server 看到穩定來源 IP
連線卸載緩衝 + 短連線 → 降低 server 併發數與記憶體占用
TCP 最佳化SACK、壅塞控制、降低 RTT 影響
跨協定族兩側可 IPv4 / IPv6 / Unix Socket 任意組合
多層 Timeout依連線階段設不同 timeout — 防止死客戶端/攻擊者占用資源
協定驗證HTTP / SSL / payload 檢查，無效元素拒絕
策略強制只轉發允許的內容
Network NamespaceLinux 限定 — 跨容器、多租戶負載均衡
PROXY Protocol非 HTTP 也能傳遞 client IP。支援者：nginx, varnish, postfix, ELB, squid, node.js…
§ 3.3.2

SSL/TLS 功能

🔒 三種部署模型

SSL Termination（最常見） Client ═══TLS═══▶ HAProxy ───plain───▶ Server SSL Passthrough（TCP mode，無法檢視內容） Client ═══TLS═══▶ HAProxy ═══TLS═══▶ Server （HAProxy 不解密） SSL Re-encryption（最安全，最耗 CPU） Client ═══TLS═══▶ HAProxy ═══TLS═══▶ Server （HAProxy 解密再加密）

⭐ 11 項關鍵 SSL 能力

功能細節
SNI 多站台無站台數限制。已知部署含 50,000+ 域名各自憑證
萬用字元憑證減少憑證管理負擔
Client 憑證驗證可設定失敗策略（如導向不同 farm 重新產生憑證）
Backend 伺服器驗證雙向：HAProxy 驗證 server，server 驗證 HAProxy
ALPN / NPN可靠卸載 HTTP/2、SPDY 連線並以明文轉給 backend
OCSP Stapling內聯 OCSP 回應 → 減少首頁載入時間
動態 TLS Record Sizing兼顧高效能與低延遲，瀏覽器可在封包在途中就開始抓取新物件
SSL/TLS 資訊存取所有層資訊可用於日誌、ACL、注入 header、PROXY Protocol 擴展
攻擊偵測即使 SSL 函式庫有漏洞（如 Heartbleed），仍可偵測、記錄並阻擋
Stateless Session ResumeRFC 5077 TLS Ticket，可從 CLI 輪替 → 實現 Perfect Forward Secrecy
Runtime 憑證更新不需重啟即可更新憑證
🚨 SSL 陷阱
  • OpenSSL 3.0 多執行緒效能大幅退化 — 升級後務必測試 TLS 吞吐
  • TLS 1.2 用 ssl-default-bind-ciphers，TLS 1.3 用 ssl-default-bind-ciphersuites — 名稱格式不同，混用會導致啟動失敗
  • 建議使用 Mozilla SSL Configuration Generator 產生密碼套件設定
§ 3.3.3

Monitoring — 健康檢查 & Agent 檢查

💓 健康檢查參數速查

參數預設說明
inter2s正常檢查間隔
fastinter= inter狀態轉換期間的檢查間隔（更快偵測）
downinter= interDOWN 狀態時的檢查間隔
fall3連續失敗次數 → 判定 DOWN
rise2連續成功次數 → 判定 UP

雙遲滯（Hysteresis）：fall + rise 組合防止狀態抖動（flapping）。

🔍 檢查類型

TCP connect（L4） HTTP request（L7） SMTP hello SSL hello LDAP SQL Redis send/expect script 以上全部 ± SSL

檢查可送到不同地址/port/協定（如只檢查 HTTPS port 代表 HTTP+HTTPS 兩服務）。

🤖 Agent Check — 伺服器自主回報

在 server 上跑一個簡單 TCP agent，回傳 ASCII 文字：

ready up down drain maint 50%（權重） maxconn:30

Agent check 與 health check 獨立運作，提供 server 自身觀點的健康/負載狀態。

🔗 Server Tracking

多個 server 可互相追蹤：一個掛掉 → 關聯的全部同時下線。適合同一主機提供多服務的場景，確保原子性故障。

§ 3.3.4

High Availability — 高可用

🛡️ 五大 HA 機制

機制說明
自動剔除僅使用有效 server；故障 server 自動從 farm 移除
Graceful Shutdown可從 farm 移除 server 而不影響任何現有連線（drain 模式）
Backup Serveractive server 全掛時自動啟用 backup。也可用於同一 server 的多路徑
Farm 全域失敗狀態太多 server 掛掉 → 回報整個 farm 失敗，上游可切換到其他 LB 節點
VRRP / keepalivedHAProxy 主動向 keepalived 回報狀態；完美支援浮動 VIP
⚠️ 官方強烈建議
只用 IP 冗餘協定（VRRP / CARP），不要用叢集方案（Heartbeat 等）。前者提供最快、最無縫、最可靠的切換。
💡 無狀態設計
HAProxy 天生無狀態 → 切換時無資訊遺失 → 接管最平順。
§ 3.3.5

Load Balancing — 十大演算法

⚖️ 演算法完整比較表

演算法類型動態權重最佳使用場景
roundrobin輪替✅（≤256 servers）預設；均質 server、短連線
static-rr輪替❌確定性分配；無 server 數量限制
leastconn動態✅處理時間不均、長連線、資料庫
sourceHash❌無 cookie 時的基本持久性（L4）
uriHash❌快取代理（cache locality）
url_paramHash❌依 URL 參數做 session 親和
hdr(name)Hash❌依 header 值路由（多租戶）
rdp-cookieHash❌Remote Desktop / Terminal Services
first填充N/A成本最佳化：填滿第一台 → 關閉空閒機器
random(N)隨機✅大規模/動態池；N=2 為 power-of-two-choices

🎚️ 權重、Slow-start 與 Consistent Hashing

權重系統

  • 所有演算法支援 per-server 權重（0–256）
  • 動態權重變更（CLI / agent）：roundrobin, leastconn, consistent hashing

Slow-start

  • server 恢復後漸進接收流量
  • 適用：JIT 編譯的 app server、冷快取需要預熱
  • ⚠️ 僅在從 DOWN 恢復時生效，HAProxy 啟動/reload 時不生效
⚠️ Consistent Hashing 至關重要

沒有 hash-type consistent，使用 balance source 時任何 server 增減 → 全部用戶重新分配。大型 cache farm 必啟。

Stickiness — 會話黏著

📌 Stickiness 架構概覽

Stick-table（LB 端記憶） Cookie（Client 端記憶） ┌──────────────────────┐ ┌──────────────────────┐ │ 儲存在 HAProxy 記憶體 │ │ insert: 插入新 cookie │ │ 最多 8 個平行來源 │ │ rewrite: 改寫現有值 │ │ 透過 peers 多主複寫 │ │ prefix: 加前綴 │ │ ACL 可動態控制規則 │ │ indirect: server 端 │ │ 支援 reload 保留 │ │ 不看到 cookie │ └──────────────────────┘ └──────────────────────┘ │ │ └─────── 可組合使用 ──────────────────┘

🔑 Stickiness 來源

Source Address TCP payload (offset+length) HTTP Query String Header Value Cookie SSL-ID RDP Cookie

🍪 Cookie 操作模式

模式行為適用場景
insertHAProxy 在回應中插入新 cookie最常用 — server 完全不知 stickiness cookie 存在
rewrite改寫 server 設定的 cookie 值server 已設 cookie 但值不適用
prefix在 server cookie 值前加前綴不能新增 cookie 但需要黏著

常用選項：insert indirect nocache httponly secure

🔧 進階 Stickiness 技巧

  • 排除 backup server：nominal server 恢復時自動奪回流量
  • Cookie 最大閒置時間 + 持續時間：處理永不關閉的裝置（手機、智慧電視、家電）
  • Soft-stop：僅有已黏著的用戶繼續到該 server，新用戶不再被送往
  • 多路徑 stickiness key 共用：一條路徑斷掉不會失去黏著
  • Server 可在 logout 時清除 cookie：離開的用戶自動解除綁定
§ 3.3.7

Logging — 日誌系統

📝 日誌內容 — 預設即非常詳細

一筆 HTTP 請求日誌包含： 來源：Source IP:Port 路由：Frontend / Backend / Server 五大計時： Tq = 請求接收耗時 （client → HAProxy 完整請求） Tw = 佇列等待耗時 （等待可用 server） Tc = 連線建立耗時 （HAProxy → server TCP handshake） Tr = 回應 header 耗時 （server 思考 + 產生回應 header） Tt = 總傳輸耗時 （整個事務從頭到尾） 狀態：行程狀態、連線計數、佇列狀態、重試次數 動作：Stickiness 動作、斷線原因 擷取：Header captures（安全編碼輸出）

⚡ 日誌關鍵特性

  • 毫秒精度 + 精確連線接受時間（可對照防火牆 NAT 日誌）
  • Log level 可透過 ACL 逐請求調整 — 靜默雜訊、對異常行為提高警告等級
  • 輸出方式：UDP（首選）、stdout/stderr、file descriptor、Ring buffer（客戶端訂閱）
  • 支援 JSON 編碼
  • 可用 ~50 種快捷標籤 + 任意 sample fetch 表達式擴展格式
  • 每個 frontend/backend 可用多個獨立日誌輸出（利於多租戶）
§ 3.3.8

Statistics — 統計與監控介面

📊 三種存取方式

🌐 Web Stats Page

  • 內建認證、安全等級、範圍控制
  • 可藏在常規網站的隱藏 URL（不需開新 port）
  • 自動重整 → 適合大螢幕監控牆
  • Administration mode：直接變更 server 狀態

📈 Prometheus + CSV

  • 內建 Prometheus exporter
  • CSV 匯出供外部工具繪圖
  • CLI 即時文字查詢
  • 可報告其他 HAProxy 節點可用性
# 典型 stats 設定範例 listen stats bind :8404 stats enable stats uri /stats stats refresh 10s stats admin if LOCALHOST # Prometheus endpoint http-request use-service prometheus-exporter if { path /metrics }
