OCSP Stapling

📌 作用

HAProxy 預先取得憑證撤銷狀態，在 TLS 握手中附帶 → 免去用戶端另連 CA 的 100–300ms 延遲。

✅ 啟用方式（HAProxy 2.8+）

# 方法一：crt-list # /etc/haproxy/crt-list.txt /etc/haproxy/certs/site.pem [ocsp-update on] # 方法二：crt-store（HAProxy 3.2） crt-store web load crt "site.crt" key "site.key" ocsp-update on alias "site" # 全域調校更新間隔 global ocsp-update.mindelay 300 # 最小間隔（秒） ocsp-update.maxdelay 3600 # 最大間隔（秒）

🔧 Runtime API 管理

echo "show ssl ocsp-response" | socat stdio $SOCK # 查看所有 OCSP 回應 echo "show ssl ocsp-updates" | socat stdio $SOCK # 查看排程中的更新 echo "update ssl ocsp-response /path/cert.pem" | socat stdio $SOCK # 手動觸發 # 驗證是否生效 openssl s_client -connect example.com:443 -status -servername example.com 2>&1 | grep "OCSP Response" # 應看到：OCSP Response Status: successful (0x0)

憑證熱更新（零停機）

🔄 完整工作流程

set ssl cert ─▶ 建立交易（記憶體中） ─▶ commit ssl cert ─▶ 立即生效 │ abort ssl cert ─▶ 撤銷（若需取消） ⚠️ Runtime API 變更僅存在記憶體 → 務必同步更新磁碟檔案

📋 指令步驟

SOCK="/var/run/haproxy/admin.sock" CERT="/etc/haproxy/certs/site.pem" # 1) 查看目前憑證 echo "show ssl cert $CERT" | socat stdio $SOCK # 2) 上傳新憑證（建立交易） echo -e "set ssl cert $CERT <<\n$(cat ./new_cert.pem)\n" | socat stdio $SOCK # 3) 驗證待定交易（帶 * 前綴） echo "show ssl cert *$CERT" | socat stdio $SOCK # 4) 提交（立即生效！） echo "commit ssl cert $CERT" | socat stdio $SOCK # 5) 同步磁碟 cp ./new_cert.pem $CERT

➕ 動態新增 / 移除 crt-list 條目

# 新增 echo "new ssl cert /etc/haproxy/certs/newsite.pem" | socat stdio $SOCK echo -e "set ssl cert /etc/haproxy/certs/newsite.pem <<\n$(cat ./newsite.pem)\n" | socat stdio $SOCK echo "commit ssl cert /etc/haproxy/certs/newsite.pem" | socat stdio $SOCK echo -e "add ssl crt-list /etc/haproxy/crt-list.txt <<\n/etc/haproxy/certs/newsite.pem newsite.example.com\n" | socat stdio $SOCK # 移除 echo "del ssl crt-list /etc/haproxy/crt-list.txt /etc/haproxy/certs/old.pem" | socat stdio $SOCK

Let's Encrypt / ACME 整合

🏆 方法 A：acme.sh Stateless 模式（推薦）

HAProxy 直接回應 HTTP-01 challenge，不需額外後端。

# HAProxy 組態 frontend ft_web bind :80 bind :443 ssl crt /etc/haproxy/certs/ strict-sni # ACME challenge（HAProxy 直接計算 token） http-request return status 200 content-type text/plain \ lf-string "%[path,field(-1,/)].${ACCOUNT_THUMBPRINT}\n" \ if { path_beg '/.well-known/acme-challenge/' } http-request redirect scheme https unless { ssl_fc } default_backend bk_web
# 簽發 + 部署 acme.sh --register-account # 記下 ACCOUNT_THUMBPRINT acme.sh --issue -d example.com --stateless # 自動部署到 HAProxy（含熱更新） DEPLOY_HAPROXY_HOT_UPDATE=yes \ DEPLOY_HAPROXY_STATS_SOCKET=UNIX:/var/run/haproxy/admin.sock \ DEPLOY_HAPROXY_PEM_PATH=/etc/haproxy/certs \ acme.sh --deploy -d example.com --deploy-hook haproxy

🆕 方法 B：HAProxy 3.2 原生 ACME（實驗性）

global master-worker expose-experimental-directives # 必須！ acme letsencrypt directory https://acme-v02.api.letsencrypt.org/directory account-key /etc/haproxy/account.key contact admin@example.com challenge HTTP-01 crt-store my_certs load crt "example.com.pem" acme letsencrypt domains "example.com" alias "example" frontend ft_web bind :80 bind :443 ssl http-request return status 200 content-type text/plain \ lf-string "%[path,field(-1,/)].%[path,field(-1,/),map(virt@acme)]\n" \ if { path_beg '/.well-known/acme-challenge/' } ssl-f-use crt "@my_certs/example" default_backend bk_web
# Runtime 管理 echo "acme renew @my_certs/example" | socat stdio $SOCK # 手動續約 echo "acme status" | socat stdio $SOCK # 查看狀態

到期前 7 天自動續約。HAProxy 3.3 新增 DNS-01 challenge 支援。

TLS 1.3 & 0-RTT 早期資料

🔒 TLS 1.3 Only 模式

global ssl-default-bind-ciphersuites TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256 ssl-default-bind-curves X25519:prime256v1:secp384r1 ssl-default-bind-options prefer-client-ciphers ssl-min-ver TLSv1.3 no-tls-tickets # 不需要 ssl-default-bind-ciphers（TLS 1.2 指令）

僅支援 Chrome 63+、Firefox 63+、Safari 12.1+、Edge 79+。

⚡ 0-RTT（allow-0rtt）

恢復連線時夾帶加密應用資料 → 零往返建立連線。但有重播攻擊風險。

frontend ft_tls13 bind :443 ssl crt /path/cert.pem alpn h2,http/1.1 allow-0rtt # 安全策略：只允許冪等方法使用 0-RTT http-request deny deny_status 425 if { ssl_fc_has_early } !{ method GET HEAD OPTIONS } # 或：等待握手完成（犧牲效能換安全） # http-request wait-for-handshake if { ssl_fc_has_early }
⚠️ 已知問題
allow-0rtt 在 HAProxy 3.0–3.3 可能無法正常運作（GitHub #3270）。2.8 正常。部署前請充分測試。

🌐 HTTP/2 & HTTP/3

# 前端 HTTP/2 bind :443 ssl crt /cert.pem alpn h2,http/1.1 # 後端 HTTP/2 server web1 10.0.1.10:443 ssl alpn h2,http/1.1 verify required ca-file /ca.pem # HTTP/3 over QUIC（HAProxy 3.2+） bind quic4@:443 ssl crt /cert.pem alpn h3

SSL 效能調校

🚨 OpenSSL 3.0 效能衰退

OpenSSL 3.0 的模組化重構引入嚴重多執行緒鎖競爭，TLS 握手效能衰退 32%+。

替代方案效能建議
AWS-LC🟢 最佳✅ HAProxy Enterprise 3.2 預設。社群版用 USE_OPENSSL_AWSLC=1
OpenSSL 3.5 LTS🟡 改善仍不及 1.1.1
BoringSSL / QuicTLS🟢 良好HAProxy 原生支援
OpenSSL 1.1.1🟢 最快⚠️ 已 EOL

📊 ECDSA vs RSA

ECDSA P-256 簽章速度是 RSA 2048 的 15 倍以上，CPU 用量約 1/8，憑證鏈更小。

# 雙憑證自動載入（ECDSA 優先，RSA 後備） # 檔案命名： # /etc/haproxy/certs/example.pem.ecdsa # /etc/haproxy/certs/example.pem.rsa # 引用時不帶副檔名 → HAProxy 自動載入兩者 bind :443 ssl crt /etc/haproxy/certs/example.pem alpn h2,http/1.1

SSL 除錯速查

🔍 常見錯誤對照表

錯誤訊息原因修正
unable to load SSL certificate檔案不存在、權限不足、CRLF 換行確認路徑、chmod 600、dos2unix
certificate verify failedCA 鏈不完整、憑證過期openssl verify -CAfile ca.pem cert.pem
no shared cipher加密套件不匹配放寬 cipher list 或加入雙憑證
SSL handshake failure協定版本不匹配、缺少用戶端憑證檢查 ssl_fc_err 日誌
unknown ca用戶端 CA 不在 ca-file 中加入缺少的 CA

🧪 openssl 測試指令集

# 基本連線 openssl s_client -connect host:443 -servername www.example.com # 指定 TLS 版本 openssl s_client -connect host:443 -tls1_3 # mTLS 測試 openssl s_client -connect host:443 -cert client.crt -key client.key -CAfile ca.crt # OCSP 檢查 openssl s_client -connect host:443 -status # Session Resumption 測試 openssl s_client -connect host:443 -sess_out /tmp/sess.pem openssl s_client -connect host:443 -sess_in /tmp/sess.pem # 看到 "Reused, TLSv1.3" = 成功

📊 SSL 日誌欄位

# 推薦 HTTPS log-format log-format "%ci:%cp [%tr] %ft %b/%s %TR/%Tw/%Tc/%Tr/%Ta %ST %B %tsc %{+Q}r %sslv/%sslc %[ssl_fc_sni]/%[ssl_c_verify]/%[ssl_fc_is_resumed]"

🖥️ Runtime API 除錯

echo "show ssl cert" | socat stdio $SOCK # 列出所有憑證 echo "show ssl crt-list" | socat stdio $SOCK # 列出 crt-list echo "show ssl ca-file" | socat stdio $SOCK # 列出 CA 檔案 echo "show ssl ocsp-response" | socat stdio $SOCK # OCSP 回應 echo "show info" | socat stdio $SOCK | grep -i ssl # SSL 統計計數器 echo "show errors" | socat stdio $SOCK # 含 SSL 握手錯誤
💡 關鍵 SSL 計數器（show info）
SslFrontendKeyRate = 目前 SSL 金鑰協商速率。SslFrontendSessionReuse_pct = Session 重用率。快取命中率 = (CacheLookups - CacheMisses) / CacheLookups。
