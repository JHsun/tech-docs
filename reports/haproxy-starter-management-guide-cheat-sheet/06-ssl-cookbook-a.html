01

PEM 檔案準備與驗證

🔑 正確的串接順序

# ✅ 正確：伺服器憑證 → 中繼 CA → 私鑰 cat server.crt intermediate.crt > /etc/haproxy/certs/site.pem cat server.key >> /etc/haproxy/certs/site.pem chmod 600 /etc/haproxy/certs/site.pem

第一個 CERTIFICATE 區塊 = 伺服器憑證，後續 = 中繼 CA。私鑰頭或尾皆可，但憑證鏈順序必須 leaf → root。

❌ 常見錯誤

錯誤症狀修正
中繼 CA 排在伺服器憑證前面瀏覽器信任但 openssl s_client 報 chain error重排：leaf cert 在前
遺漏中繼 CA部分用戶端 certificate verify failed加入完整中繼鏈
Windows CRLF 換行unable to load SSL certificatesed -i 's/\r$//' site.pem
私鑰與憑證不匹配unable to load SSL private key比對 modulus（見下方）

🔍 驗證指令

# 確認私鑰與憑證匹配（MD5 必須相同） openssl x509 -noout -modulus -in site.pem | openssl md5 openssl rsa -noout -modulus -in site.key | openssl md5 # 驗證憑證鏈完整性 openssl verify -CAfile ca-bundle.crt server.crt # 測試線上連線 openssl s_client -connect example.com:443 -servername example.com # 檢查到期日 openssl x509 -in site.pem -noout -enddate

三大 SSL 部署模式架構圖

① SSL Termination（最常見 — 最靈活） Client ══TLS══▶ HAProxy 解密 ──HTTP 明文──▶ Server ✅ 可用所有 L7 功能 ② SSL Passthrough（最安全 — 功能最少） Client ══TLS══▶ HAProxy 不解密 ══TLS══▶ Server ❌ 僅 TCP 層功能（SNI 路由可用） ③ SSL Re-encryption / Bridging（平衡 — 最耗 CPU） Client ══TLS══▶ HAProxy 解密→再加密 ══TLS══▶ Server ✅ L7 功能全開 + 端到端加密

03

SSL Termination 完整設定

global ssl-default-bind-ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384 ssl-default-bind-ciphersuites TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256 ssl-default-bind-options prefer-client-ciphers ssl-min-ver TLSv1.2 no-tls-tickets tune.ssl.default-dh-param 2048 defaults mode http option httplog option forwardfor except 127.0.0.0/8 timeout connect 5s timeout client 30s timeout server 30s frontend ft_web bind *:80 bind *:443 ssl crt /etc/haproxy/certs/site.pem alpn h2,http/1.1 # HTTP → HTTPS 強制重導 http-request redirect scheme https unless { ssl_fc } # HSTS（2 年） http-response set-header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" # 傳遞原始協定給後端 http-request set-header X-Forwarded-Proto https if { ssl_fc } default_backend bk_web backend bk_web balance roundrobin option httpchk GET /healthz http-check expect status 200 server web1 10.0.1.10:80 check server web2 10.0.1.11:80 check

alpn h2,http/1.1 啟用 HTTP/2 ALPN 協商。ssl_fc = 連線使用 SSL 時為 true。

04

SSL Passthrough + SNI 路由

defaults mode tcp option tcplog timeout connect 5s timeout client 30s timeout server 30s frontend ft_ssl_pass bind *:443 mode tcp # 等待 ClientHello 到達（最多 5 秒） tcp-request inspect-delay 5s tcp-request content accept if { req_ssl_hello_type 1 } # 基於 SNI 路由 use_backend bk_app1 if { req.ssl_sni -i app1.example.com } use_backend bk_app2 if { req.ssl_sni -i app2.example.com } default_backend bk_default backend bk_app1 mode tcp server srv1 10.0.1.10:443 check backend bk_app2 mode tcp server srv2 10.0.1.20:443 check
⚠️ 必要設定
tcp-request inspect-delay 讓 HAProxy 緩衝足夠資料以讀取 SNI。沒有它，封包可能在 SNI 到達前就被轉發。
05

SSL Re-encryption (Bridging)

frontend ft_bridge bind *:443 ssl crt /etc/haproxy/certs/public.pem alpn h2,http/1.1 http-request redirect scheme https unless { ssl_fc } default_backend bk_reencrypt backend bk_reencrypt mode http balance roundrobin option httpchk GET /healthz http-check expect status 200 # ✅ 生產環境：驗證後端憑證 default-server ssl verify required ca-file /etc/haproxy/ca/internal-ca.pem check server web1 10.0.1.10:443 server web2 10.0.1.11:443

verify none vs verify required

verify noneverify required
使用場景開發環境、自簽憑證生產環境、合規要求
需要 ca-file？否是
風險內網中間人攻擊後端憑證管理開銷
🚨 不要在生產環境用 verify none
這是最常見的「快速修復」變成永久安全漏洞的情況。
06

mTLS 雙向 TLS 驗證

🔐 核心設定

frontend ft_mtls bind :443 ssl \ crt /etc/haproxy/certs/server.pem \ ca-file /etc/haproxy/certs/client-ca.crt \ verify optional \ crl-file /etc/haproxy/certs/revoked.crl # ⚠️ 安全性：清除用戶端可能注入的 Header http-request del-header X-SSL-Client-Verify http-request del-header X-SSL-Client-DN http-request del-header X-SSL-Client-CN # 注入經驗證的用戶端憑證資訊 http-request set-header X-SSL-Client-Verify %[ssl_c_verify] http-request set-header X-SSL-Client-DN %{+Q}[ssl_c_s_dn] http-request set-header X-SSL-Client-CN %{+Q}[ssl_c_s_dn(cn)] # 公開路徑不需要憑證 acl is_public path_beg /health /public /login # ⚠️ 用 ssl_c_used（非 ssl_fc_has_crt） # ssl_fc_has_crt 在 TLS Session Resume 時會誤報 false http-request deny deny_status 403 if !is_public !{ ssl_c_used } http-request deny deny_status 403 if !is_public !{ ssl_c_verify 0 } default_backend bk_app

🔑 verify required vs verify optional

verify requiredverify optional
TLS 握手時無有效憑證 → 直接拒絕連線允許無憑證連線建立
按路徑控制❌ 全部路徑都需要✅ 可在 HTTP 層按路徑判斷
適用場景全站 mTLS（API 對 API）混合站台（部分公開、部分需憑證）
💡 按路徑 mTLS 必須用 verify optional
TLS 握手發生在 HTTP 路徑解析之前，所以 verify required 無法按路徑區分。

📋 Client 憑證 Fetch 速查

Fetch回傳值用途
ssl_c_verify整數（0=成功）驗證結果
ssl_c_s_dn/CN=user/O=corp完整主體 DN
ssl_c_s_dn(cn)user特定欄位
ssl_c_i_dn發行者 DN識別 CA
ssl_c_serial十六進位序號搭配 ,hex
ssl_c_used布林✅ 推薦替代 ssl_fc_has_crt
07

SNI 多站台 & crt-list

📄 crt-list 檔案格式

# /etc/haproxy/crt-list.txt # 預設憑證 /etc/haproxy/certs/default.pem * # 精確 SNI /etc/haproxy/certs/api.example.com.pem api.example.com # 萬用字元 /etc/haproxy/certs/wildcard.example.com.pem *.example.com # 含 SSL 選項 /etc/haproxy/certs/secure.pem [ssl-min-ver TLSv1.3] secure.example.com # 多 SNI 共用一張憑證 /etc/haproxy/certs/multi.pem site1.com site2.com site3.com # 個別啟用 mTLS /etc/haproxy/certs/internal.pem [verify required ca-file /etc/haproxy/ca/client-ca.crt] internal.example.com
# 引用方式 frontend ft_multi bind :443 ssl crt-list /etc/haproxy/crt-list.txt

選擇優先順序：精確 SNI → 萬用字元 → 預設。加 strict-sni 則無匹配時拒絕連線。

🆕 HAProxy 3.2：crt-store + ssl-f-use

# 不需要外部 crt-list 檔案 crt-store mycerts crt-base /etc/haproxy/ssl/certs/ key-base /etc/haproxy/ssl/private/ load crt "site1.pem" alias "site1" load crt "site2.crt" key "site2.key" alias "site2" frontend ft_multi bind :443 ssl bind quic4@:443 ssl # HTTP/3 ssl-f-use crt "@mycerts/site1" ssl-min-ver TLSv1.2 alpn h2,http/1.1 ssl-f-use crt "@mycerts/site2" ssl-min-ver TLSv1.3 alpn h2,http/1.1 default_backend bk_web

優勢：同一份設定同時套用 TCP 和 QUIC bind，且 ECDSA + RSA 雙憑證用 .ecdsa/.rsa 副檔名自動載入。

Cipher 設定 — 兩套命名慣例的陷阱

🚨 這是 HAProxy SSL 最常出錯的地方

TLS 1.2
ssl-default-bind-ciphers		TLS 1.3
ssl-default-bind-ciphersuites
命名格式OpenSSL 簡寫名IANA/RFC 標準名
分隔符連字號 -底線 _
前綴無TLS_
範例ECDHE-RSA-AES256-GCM-SHA384TLS_AES_256_GCM_SHA384
可用數量數十種僅 3 種標準套件
🚨 混用格式 = 啟動失敗
把 TLS 1.3 的 TLS_AES_128_GCM_SHA256 放進 ssl-default-bind-ciphers（TLS 1.2 指令）→ 無效。反之亦然。

✅ Mozilla Intermediate 設定（推薦）

global # TLS 1.2（OpenSSL 格式，連字號） ssl-default-bind-ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384 # TLS 1.3（IANA 格式，底線 + TLS_ 前綴！） ssl-default-bind-ciphersuites TLS_AES_128_GCM_SHA256:TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256 ssl-default-bind-options prefer-client-ciphers ssl-min-ver TLSv1.2 no-tls-tickets

🔍 確認指令

# 確認 HAProxy TLS 支援 haproxy -vv | grep -E "OpenSSL|TLSv" # 列出 TLS 1.2 可用套件 openssl ciphers -v 'ECDHE+AESGCM:ECDHE+CHACHA20:DHE+AESGCM' # 列出 TLS 1.3 可用套件 openssl ciphers -v -s -tls1_3
