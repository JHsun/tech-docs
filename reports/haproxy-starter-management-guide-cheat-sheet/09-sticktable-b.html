Tarpit 分層策略

🪤 Tarpit 運作方式

接受連線 → 接收完整請求 → 佔住連線不處理 → timeout tarpit 到期 → 送回錯誤。迫使攻擊者每個連線被「黏住」。

🏗️ 分層回應：deny → tarpit → silent-drop

frontend ft_web bind *:80 mode http timeout tarpit 15s stick-table type ip size 200k expire 60s store http_req_rate(10s),conn_cur http-request track-sc0 src # 從最嚴重到最輕微（先匹配先執行） acl extreme sc_http_req_rate(0) gt 100 # DDoS acl severe sc_http_req_rate(0) gt 50 # 嚴重 acl moderate sc_http_req_rate(0) gt 20 # 中度 http-request silent-drop if extreme # 零資源消耗 http-request deny deny_status 429 if severe # 快速回應 http-request tarpit deny_status 429 if moderate # 佔住連線 default_backend bk_servers
🚨 Tarpit 是雙面刃
每個被 tarpit 的連線佔用一個 maxconn 插槽。若 maxconn=50000 且 timeout tarpit=30s → 僅需 ~1,667 req/s 的攻擊即可耗盡所有插槽。大規模 DDoS 務必用 silent-drop。

📋 策略選擇指南

策略最佳場景優點風險
denyAPI 限速、標準防護快速釋放資源攻擊者瞬間得知被封鎖
tarpit暴力破解、少量 bot消耗攻擊者連線佔用 maxconn 插槽
silent-drop大規模 DDoS零資源消耗中間防火牆可能殘留狀態

多時間窗口分層限速

⏱️ 短期爆發 → 中期持續 → 長期累計

# 三層時間窗口 backend rate_1s stick-table type ip size 1m expire 10s store http_req_rate(1s) backend rate_15m stick-table type ip size 1m expire 30m store http_req_rate(15m) backend rate_1h stick-table type ip size 1m expire 2h store http_req_rate(1h) frontend ft_api bind *:443 ssl crt /etc/haproxy/certs/api.pem acl is_api path_beg /api http-request track-sc0 src table rate_1s if is_api # 短期 http-request track-sc1 src table rate_15m if is_api # 中期 http-request track-sc2 src table rate_1h if is_api # 長期 acl burst sc_http_req_rate(0) gt 5 # > 5/秒 acl medium sc_http_req_rate(1) gt 500 # > 500/15分 acl long sc_http_req_rate(2) gt 1000 # > 1000/時 http-request deny deny_status 429 if burst is_api http-request deny deny_status 429 if medium is_api http-request deny deny_status 429 if long is_api default_backend bk_api
💡 expire 設定原則
expire 應為最長速率週期的 1.5–2 倍。例如 http_req_rate(15m) → expire 30m。

三維防禦矩陣（生產推薦模板）

🛡️ 連線 × 請求 × 錯誤

backend tbl_conn stick-table type ip size 200k expire 30s store conn_rate(10s),conn_cur backend tbl_req stick-table type ip size 200k expire 60s store http_req_rate(60s) backend tbl_err stick-table type ip size 200k expire 5m store http_err_rate(30s),gpc0 frontend ft_main bind *:80 bind *:443 ssl crt /etc/haproxy/certs/site.pem acl whitelist src -f /etc/haproxy/whitelist.lst # 三維追蹤 tcp-request connection track-sc0 src table tbl_conn if !whitelist http-request track-sc1 src table tbl_req if !whitelist http-request track-sc2 src table tbl_err if !whitelist # 偵測 acl conn_abuse sc_conn_rate(0) gt 20 acl conn_full sc_conn_cur(0) gt 50 acl req_abuse sc_http_req_rate(1) gt 300 acl err_abuse sc_http_err_rate(2) gt 30 acl is_banned sc_get_gpc0(2) gt 0 acl mark_ban sc_inc_gpc0(2) gt 0 # 執行（從嚴到輕） tcp-request connection reject if conn_abuse !whitelist tcp-request connection reject if conn_full !whitelist http-request silent-drop if is_banned http-request tarpit deny_status 429 if err_abuse mark_ban http-request deny deny_status 429 hdr Retry-After "60" if req_abuse default_backend bk_servers
💡 突破 3 個 sc 限制
把多個計數器存入同一表共用一個 sc 插槽：store conn_rate(10s),http_req_rate(60s),http_err_rate(30s),conn_cur。HAProxy Enterprise 提供 sc0–sc11 共 12 個。

Peers 同步 — 重載持久化 & 多節點

🔴 最重要的設定：單節點 peers（防重載資料遺失）

global localpeer haproxy1 # 必須與 peer 名稱一致 peers mypeers peer haproxy1 127.0.0.1:10000 frontend ft_main bind *:80 stick-table type ip size 100k expire 30s \ store http_req_rate(10s) peers mypeers http-request track-sc0 src http-request deny deny_status 429 if { sc_http_req_rate(0) gt 20 } default_backend be_srv
🚨 沒有 peers = 每次 reload 全部歸零
速率限制歸零 → 攻擊者在重載瞬間獲得完全豁免。即使單節點也必須設定 peers。

🌐 多節點叢集

# 所有節點用相同設定，以 -L 區分身份 # haproxy -f haproxy.cfg -L haproxy1 （節點 1） # haproxy -f haproxy.cfg -L haproxy2 （節點 2） peers haproxy_peers peer haproxy1 10.0.1.1:1024 peer haproxy2 10.0.1.2:1024 peer haproxy3 10.0.1.3:1024 backend st_global stick-table type ip size 200k expire 60s \ store http_req_rate(10s),conn_rate(10s),gpc0 \ peers haproxy_peers
🚨 致命限制：Peers 只覆寫，不聚合
節點 A 看到 IP X 的 50 個請求、節點 B 也看到 50 個 → 計數器不會顯示 100，而是最後一次推送的值（50）。攻擊者分散流量到多節點即可繞過。對策：閾值減半。或用 HAProxy Enterprise 的 Stick Table Aggregator。

CLI 管理指令速查

# ===== 查看 ===== echo "show table" | socat stdio $SOCK # 列出所有表 echo "show table st_req_rate" | socat stdio $SOCK # 所有條目 echo "show table st_req_rate key 192.168.1.100" | socat stdio $SOCK # 特定 IP echo "show table st_req_rate data.http_req_rate gt 100" | socat stdio $SOCK # 過濾 # 支援的比較運算子：eq, ne, gt, lt, ge, le # ===== 手動設定（封鎖 IP）===== echo "set table st_err key 10.0.0.5 data.gpc0 1" | socat stdio $SOCK # ===== 清除 ===== echo "clear table st_req_rate" | socat stdio $SOCK # 清空整表 echo "clear table st_err key 1.2.3.4" | socat stdio $SOCK # 解鎖特定 IP echo "clear table st_err data.gpc0 ge 1" | socat stdio $SOCK # 批量解鎖 # ===== 即時監控 ===== watch -n 1 'echo "show table st_req_rate" | socat stdio /var/run/haproxy.sock' # ===== Peers 狀態 ===== echo "show peers" | socat stdio $SOCK
⚠️ use > 0 的條目無法清除
有活躍 session 引用的條目必須等 session 結束。替代方案：set table ... data.gpc0 0 解除封鎖效果（條目仍在但不再觸發規則）。
