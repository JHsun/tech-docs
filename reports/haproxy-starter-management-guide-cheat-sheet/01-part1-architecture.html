CH 2

負載均衡基礎概念

⚖️ 核心定義

負載均衡 = 聚合多個元件，使總處理量超過單一元件。

單一操作仍然在單一元件上執行 — 負載均衡提升的是吞吐量，不是單一請求速度。

🏗️ 負載均衡的四大類別

CPU 排程（多處理器） 鏈路聚合（EtherChannel / Bonding） IP 層（ECMP / DNS Round-robin） 伺服器層（負載均衡器）← HAProxy

🔀 L4（封包級） vs L7（代理級）— 最關鍵的架構分野

L4 封包級（Packet-based）

  • 處理單位：個別封包
  • 輸入/輸出封包 1:1 對應
  • 可達線速（ASIC 硬體）
  • 通常無狀態
  • 幾乎無內容感知
  • 支援 DSR（封包未修改時）
  • 兩側必須相同協定
  • Stickiness 有限（hash 易碎）
  • 適合：網路層 LB、高速基本伺服器 LB
vs

L7 代理級（Proxy-based）

  • 處理單位：重組後的完整 session
  • 兩側封包無對應關係
  • 有處理成本，不一定達線速
  • 永遠有狀態
  • 完整內容感知
  • 不支援 DSR（回程必經 LB）
  • 兩側可不同協定（IPv4↔IPv6, 明文↔SSL）
  • 豐富 Stickiness（cookie, 內容, 記憶）
  • 適合：伺服器負載均衡 ← HAProxy

🏢 大規模參考架構（分層設計）

業界標準的大規模部署模式： 前端路由器 ──ECMP──▶ 多台 L4 LB ──────▶ 更多台 L7 LB ──────▶ 伺服器群 │ │ 有狀態封包級 │ 代理級 │ │ 多鏈路聚合流量 │ 快速分配連線 │ 解析內容做精細決策 │ 實際處理請求

🏥 健康檢查的四種方法

方法原理優缺點
週期性探針（最常見）定期發送探測請求✅ 不影響生產流量 ⚠️ 需要足夠頻率
生產流量取樣觀察真實流量的回應⚠️ 犧牲部分流量 ✅ 最真實
混合式兩者搭配：兩者偵測故障，僅探針確認恢復✅ 最佳方案
集中式回報中央監控代理定期更新所有 LB✅ 全局視角 ⚠️ 精度/反應稍差
⚠️ 檢查粒度至關重要
Ping → 只確認主機在線；Port 連線 → 只確認 port 開啟；HTTP 請求 → 確認應用正常；含資料庫查詢 → 確認整個堆疊正常。

📌 Session Stickiness 核心觀念

Hash 演算法（如 Source IP）的致命弱點：任何伺服器數量變動 → 所有用戶重新分配。

解法是記憶化（Memorization）：

LB 端記憶

Stick-table 儲存在 HAProxy 記憶體，透過 peers 多主複寫到其他節點

Client 端記憶

Cookie 插入、子域名重導 — 客戶端每次請求帶回 server 資訊

💡 官方原文強調
記憶化是「選擇 L7 而非 L4 負載均衡器最有力的理由」。
§ 3.1

HAProxy 是什麼 — 14 種功能角色

#角色關鍵細節
1TCP Proxy接受 TCP → 連線後端 → 雙向轉發。支援 IPv4 / IPv6 / Unix Socket 任意組合
2HTTP 反向代理HTTP 術語中的「Gateway」。兩側可任意混用 HTTP/1.x 或 HTTP/2，TLS + ALPN 自動偵測協定
3SSL 終止/發起/卸載Per-SNI 設定、執行期更新憑證無需重啟。已報告部署含數十萬張憑證
4TCP 正規化器本地終止連線 → 無效封包、SYN Flood 永遠不到達後端。可獨立最佳化 TCP 參數
5HTTP 正規化器僅轉發完整有效請求。修復規範容忍的偏差（如多行 header）
6HTTP 修改工具修改/新增/移除/改寫 URL 及任何 request/response header
7內容交換器依請求任意元素決定路由。同一 port 可處理多種協定（HTTP, HTTPS, SSH）
8伺服器負載均衡器TCP mode: 每連線決策；HTTP mode: 每請求決策
9流量調節器速率限制、優先順序調整、封包標記（影響外部網路元件）
10DDoS / 濫用防護Per-IP/URL/Cookie 統計、偵測異常模式、自動反制（限速/封鎖/導向舊內容）
11網路除錯觀測點日誌精度極高，常用於縮小網路問題範圍
12HTTP 壓縮卸載壓縮後端未壓縮的回應，改善低頻寬/行動用戶體驗
13快取代理RAM 快取、免維護。設計目標是節省 HAProxy 資源，非節省後端資源
14FastCGI 閘道直接負載均衡 FastCGI 應用，無需額外閘道層
§ 3.1

HAProxy 不是什麼 — 4 項明確排除

✗ 不是正向代理（Forward Proxy）

不是瀏覽器用來上網的 proxy。用 Squid；HAProxy 可放在其前面做 LB

✗ 不是資料清洗器

不會修改 request/response body（只改 header/URL）

✗ 不是靜態 Web Server

啟動後 chroot + 降權 → 零檔案系統存取。用 Apache/Nginx；HAProxy 可放在前面

✗ 不是封包級負載均衡器

看不到 IP 封包或 UDP datagram、不做 NAT/DSR。用 IPVS（Linux Virtual Server）— 與 HAProxy 完美互補

§ 3.2

事件驅動架構 & 執行緒模型

⚙️ 核心架構特性

事件驅動 非阻塞 優先順序排程 多執行緒 CPU 快取最佳化 分層旁路機制

Kernel / Userspace 處理比例：

TCP / HTTP close mode：~15% HAProxy / ~85% Kernel

HTTP keep-alive mode：~30% HAProxy / ~70% Kernel

→ OS 調校對效能影響極大

規模實績：

單一行程可承載 300,000 個獨立 proxy 設定

單核心足以應付 >99% 使用場景

🧵 執行緒模型

使用 nbthread（每 CPU 核心一個執行緒），取代已棄用的 nbproc（多程序模式）。

多執行緒主要有利於：SSL 運算或轉發速率 > 40 Gbps 的場景。

🚨 關鍵限制
  • 主機絕對不能 swap
  • CPU 不能被虛擬化限速（sub-CPU allocation）
  • CPU 不能與 CPU 密集型行程共用
  • 遇到效能問題？反直覺第一步：減少 CPU 數量

🚀 啟動 & 執行行為

所需檔案：僅可執行檔 + 設定檔。

啟動流程：解析所有設定 → 綁定所有監聽 socket → 任何綁定失敗即拒絕啟動。

💡 零執行期失敗保證
通過啟動階段後，HAProxy 不可能再失敗（直到被停止）。這是架構層面的設計保證。

執行中只做三件事：

① 處理進入的連線 ② 週期性伺服器健康檢查 ③ 與其他 HAProxy 節點交換資訊

🧩 Frontend / Backend / Listen 的關係

實體定義適用場景
frontend半代理：只關注客戶端（bind、ACL、use_backend）HTTP — 一個 frontend 可轉發到多個 backend
backend半代理：只關注伺服器（server 清單、balance、check）HTTP — 不同 backend 可用不同演算法
listen完整代理 = frontend + backend 合一TCP-only — 分開寫通常無額外好處

⚠️

關鍵警告與常見陷阱

陷阱說明
nbproc 已棄用改用 nbthread。多程序模式導致 stick-table / stats / 限制都各自隔離
mode tcp 停用 HTTP在 TCP mode 下所有 HTTP 功能失效 — 無 header ACL、無 cookie 操作
未設 maxconn全域和 per-frontend/server 都需設定，否則資源耗盡
僅用 TCP checkTCP check 只確認 port 開啟。HTTP 後端應用 option httpchk
Sticky 未配 option redispatch目標伺服器掛掉時，客戶端會持續收到錯誤而不會被轉派
使用 Heartbeat 而非 VRRP官方建議 VRRP/CARP，提供最快、最無縫的切換
Stick-table reload 遺失若未設定 peers（即使單節點），reload 時 stick-table 資料丟失
快取定位錯誤HAProxy 快取節省的是 HAProxy 自身資源。要節省後端資源 → 用 Varnish
IP-based stickiness + NATNAT/共享 IP 會破壞 IP hash → 改用 cookie-based persistence
Reload 前未驗證設定永遠先跑 haproxy -c -f config.cfg
