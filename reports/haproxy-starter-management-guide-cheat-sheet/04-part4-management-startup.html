CH 3

啟動命令列選項（精選分類）

📂 設定檔載入

旗標參數說明
-f<file|dir>加入設定檔/目錄（可重複）。目錄只載 .cfg，按字典序
--<files>*後續所有引數都視為設定檔。適合 shell wildcard
-C<dir>載入前先 cd 到指定目錄（方便相對路徑）

⚙️ 運作模式

旗標說明注意
-DDaemon 模式（背景執行）init script 必加
-WMaster-Worker 模式（master 監控 workers）建議搭配 systemd
-WsMaster-Worker + systemd notify
-dDebug 模式（前景，顯示所有 I/O）絕不用於 init script
-db前景模式，無背景化開發/小測試用

✅ 驗證與檢查

旗標說明
-c檢查設定檔後退出。Exit 0 = 有效。搭配 -V 顯示成功訊息
-v顯示版本與建置日期
-vv詳細版本：建置選項、函式庫版本、可用 poller。回報 bug 時必附
-dD診斷模式：輸出可疑設定的額外警告（不阻止啟動）
-dW零警告：任何警告都拒絕啟動。等同 zero-warning
-dC[key]dump 解析後設定檔。加 key 可匿名化敏感欄位

🔧 效能 / Debug 旗標

旗標說明
-dM[byte]記憶體毒化（預設填 0x50）。偵測未初始化 bug。-dM0 = calloc
-dS停用 splice() → strace 可看轉發資料
-dZ停用 zero-copy 轉發（也停用 kernel splicing）
-dR停用 SO_REUSEPORT
-dV停用 SSL server verify。絕不用於生產
-de/-dk/-dp/-dv停用 epoll/kqueue/poll/evports poller
-dt [desc]啟用 stderr 追蹤。無參數 = 所有來源 error 等級

📦 資源 / 行程管理

旗標說明
-n <limit>覆寫全域 maxconn
-m <MB>全域記憶體上限（MB）。Master-Worker 下各自獨立計算
-p <file>PID 檔（chroot 前開啟）
-q靜默模式。搭配 -c 做無聲設定驗證
-sf <pid>*啟動後送 SIGUSR1（graceful stop）給舊行程
-st <pid>*啟動後送 SIGTERM（hard stop）給舊行程
-x <socket>從舊行程取回監聽 FD。Hitless reload 關鍵
-S <bind>綁定 Master CLI socket（建議用 UNIX socket）
CH 3

安全啟動 & Reload 範例

# 標準安全啟動（init script） haproxy -f /etc/haproxy/haproxy.cfg \ -D -p /var/run/haproxy.pid \ -sf $(cat /var/run/haproxy.pid) # 驗證設定（必須在每次 reload 前執行） haproxy -c -f /etc/haproxy/haproxy.cfg # 靜默驗證（腳本用） haproxy -c -q -f /etc/haproxy/haproxy.cfg && echo OK || echo FAIL # 多檔案載入 haproxy -f /etc/haproxy/global.cfg \ -f /etc/haproxy/defaults.cfg \ -f /etc/haproxy/frontends/ \ -f /etc/haproxy/backends/ # Master-Worker 模式啟動 haproxy -W -f /etc/haproxy/haproxy.cfg \ -S /var/run/haproxy-master.sock \ -p /var/run/haproxy.pid
⚠️ 每個檔案必須以 section boundary 開頭
多檔案載入時，每個檔案第一個關鍵字必須是 global、defaults、frontend、backend、listen、peers 之一。不能只有 server 清單。
CH 4

七個操作信號完整表

信號名稱效果使用時機
SIGUSR1Graceful Stop解除所有 listener 綁定；等所有現有連線自然結束後才退出-sf 觸發；日常 reload
SIGTERMHard Stop立即終止，所有連線直接斷開-st 觸發；緊急停止
SIGINTHard Stop同 SIGTERMCtrl+C（前景模式）
SIGUSR2Master ReloadMaster-Worker 模式下：master 重新執行自身，fork 新 worker設定變更後 reload
SIGTTOUPause暫停監聽（解除 port 綁定）。Reload 過程中自動使用自動 — 不需手動
SIGTTINResume恢復監聽。新行程綁定失敗時回滾用自動 — 不需手動
SIGQUITFlush Pools釋放所有未使用記憶體。前景模式下會輸出 pool 狀態記憶體回收
# 實用指令 kill -USR1 $(cat /var/run/haproxy.pid) # Graceful stop kill -USR2 $(cat /var/run/haproxy.pid) # Master-Worker reload kill -QUIT $(cat /var/run/haproxy.pid) # 釋放記憶體
CH 4

Seamless Reload 完整流程

🔄 傳統模式 Reload（-sf / -st）

① 驗證設定 haproxy -c -f config.cfg │ ② 啟動新行程（帶 -sf + 舊 PID） haproxy -f config.cfg -sf $(cat haproxy.pid) │ ③ 新行程嘗試綁定所有 port │ ├── 成功 ──▶ 送 SIGUSR1 給舊行程（graceful stop） │ 舊行程停止接收新連線 │ 舊行程等現有連線結束後退出 │ （hard-stop-after 設定最長等待時間） │ └── 失敗（port 被占用） │ ▼ 送 SIGTTOU 給舊行程 → 舊行程暫停監聽 新行程重試綁定 │ ├── 成功 ──▶ 送 SIGUSR1（如上） └── 失敗 ──▶ 送 SIGTTIN 給舊行程 → 恢復監聽 Reload 取消
💡 SO_REUSEPORT 的角色
Linux kernel ≥ 3.9 支援 SO_REUSEPORT，允許新行程直接綁定已被舊行程占用的 port → 跳過 SIGTTOU/SIGTTIN 階段 → 失敗率更低。
⚠️ Reload 期間連線失敗率
約 每 10,000 新連線/秒 reload 一次會丟失 1 個連線。30,000 conn/s 的站台每次 reload 約掉 3 個連線。
CH 4

Master-Worker 模式 Reload

👑 Master-Worker 模式（推薦）

Master 行程（PID 不變，持續監控） │ ├── Worker 1（處理流量） ├── Worker 2（舊 worker，draining） │ │ 收到 SIGUSR2 │ ▼ │ Master 重新解析設定 │ Fork 新 Worker（自動透過 sockpair@ 傳遞監聽 FD） │ 送 SIGUSR1 給舊 Worker │ 舊 Worker drain 後退出 │ └── 結果：零丟失 reload（監聽 socket 從未關閉）
# Master-Worker 模式 reload 指令 kill -USR2 $(cat /var/run/haproxy.pid) # 或透過 Master CLI echo "reload" | socat stdio /var/run/haproxy-master.sock # 強制 reload（hard） echo "hard-reload" | socat stdio /var/run/haproxy-master.sock

關鍵優勢：Master 自動透過 sockpair@ 傳遞監聽 FD，無需在 stats socket 設定 expose-fd listeners。

🚨 QUIC 限制
QUIC 連線不支援 soft-stop，reload 時 QUIC 連線不會被保留。
CH 5

File Descriptor 限制

📐 FD 需求計算公式

FD 需求 ≈ 2 × maxconn + listeners + health_checks + agent_checks + peers + loggers + misc 簡易估算：maxconn 的兩倍再加數十個 預設 UNIX 限制 1024 FD ≈ 支撐約 500 個併發連線

🖥️ Poller 與 FD 上限

PollerFD 限制建議
epoll無實際限制✅ Linux 首選。常數級喚醒時間
kqueue無實際限制✅ BSD 首選
poll無硬限制⚠️ >1000 連線後變慢
select1024 FD❌ 最後手段，無法停用

🔧 系統限制調整

限制錯誤修正
行程 FD（ulimit -n）EMFILE讓 HAProxy 自動計算（移除舊的 ulimit-n 設定）
系統全域（fs.file-max）ENFILEsysctl fs.file-max=<value>
每行程硬上限（fs.nr_open）setrlimit 失敗sysctl fs.nr_open=<value>（預設 1,048,576）
⚠️ 移除舊的 ulimit-n
舊設定檔中的 ulimit-n 經常計算錯誤導致連線被拒。HAProxy 現在會自動計算正確值。
CH 6

記憶體管理

🧠 Pool-based 記憶體架構

HAProxy 使用 LIFO pool 機制：最近釋放的物件（仍在 CPU cache 中 = hot）優先重用。相似大小的 pool 自動合併以減少碎片。物件正常不會歸還 OS，僅在分配失敗時自動 flush + retry。

📊 關鍵記憶體參數

參數預設說明
tune.bufsize16384每個 buffer 大小（bytes）
tune.buffers.limit無限可分配 buffer 上限。設為 maxconn/10 可大幅降低記憶體用量
tune.buffers.reserve2保留 buffer 數量
maxconn2000最大併發連線
-m <MB>無限全域記憶體上限
💡 每連線記憶體估算
約 2 × tune.bufsize（buffer）+ ~960B（stream）+ ~160B（session）+ ~800B（2×connection）+ ~144B（task）+ header/variable 開銷。以預設值計 ≈ ~35KB/連線。

🔍 記憶體檢查指令

echo "show pools" | socat stdio /var/run/haproxy.sock

顯示：pool 名稱、物件大小、已分配/使用數、失敗次數。

CPU 使用與執行緒

📊 CPU 時間分佈（飽和負載下）

工作負載System %User %
長 TCP / 大物件 HTTP95%5%
短 TCP / 小物件 HTTP（close）85%15%
小物件 HTTP（keep-alive）70%30%

單核 3.5 GHz CPU ≈ 80,000 端到端 TCP 連線建立/關閉/秒。

🧵 執行緒與 CPU 設定（HAProxy 3.2）

指令說明
nbthreadWorker 執行緒數（預設：每允許 CPU 一個，最大 1024）
cpu-policy🆕 3.2：簡化 CPU 綁定指令，取代手動 cpu-map
cpu-set🆕 3.2：指定 HAProxy 使用的 CPU 集合
-dcDebug CPU 親和性：顯示執行緒→CPU 配置

🚀 效能調校要點

  • Idle_pct 是關鍵指標：接近 100% = 低負載；接近 0% = 飽和
  • 檢查方式：echo "show info" | socat - /var/run/haproxy.sock | grep ^Idle
  • 網路密集負載下停止 irqbalance
  • HAProxy 與 NIC 中斷 pin 到共享同一 L3 cache 的不同核心
  • 卸載 nf_conntrack（狀態防火牆）可大幅降低 system CPU
CH 8

Logging 設定與除錯

📡 日誌目標設定

# 推薦：UDP 到 syslog（預設 port 514） global log 127.0.0.1:514 local0 # 流量日誌 log 127.0.0.1:514 local1 notice # 管理日誌（選用）
🚨 不建議 UNIX socket 日誌
syslog 重啟會替換 socket → chroot 中的 HAProxy 無法重連 → 日誌中斷。且 UNIX socket buffer 極小。

🔧 Syslog 守護程式設定速查

守護程式關鍵設定
rsyslogd$ModLoad imudp + $UDPServerRun 514
syslog-ngsource s_udp { udp(ip(127.0.0.1) port(514)); };
sysklogd啟動時加 -r 參數

🐛 日誌除錯三步驟

① 重啟 HAProxy → 每個 frontend/backend 會產生一行啟動日誌 │ 沒看到？ ▼ ② 用 strace 確認送出 strace -tt -s100 -etrace=sendmsg -p <pid> │ 有送出但 syslog 沒收到？ ▼ ③ 用 tcpdump 在網路層確認 tcpdump -As0 -ni lo port 514

📊 halog 工具

HAProxy 附帶的日誌分析工具，處理速度 1–2 GB/秒。可依 HTTP status / 終止狀態 / 回應時間範圍過濾，依 URL 按時間/次數排序，依 client 按存取次數排序。偵測 bot 和異常的利器。
