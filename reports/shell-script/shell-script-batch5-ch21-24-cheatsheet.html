第 21 章 · Bash 內部機制

本章打開 Bash 的「黑盒子」── 讓你看見一條命令從你按下 Enter 到輸出結果之間發生了什麼：詞法分析、parser、fork/exec、檔案描述符的繼承、訊號傳播。理解這些不是炫技，而是讓你看懂後面章節討論「為什麼 subshell 變數消失」「為什麼 PID 1 不收訊號」這類問題的根本。

21.1 一條命令的生命週期 ¶

當你輸入 ls -la /tmp 並按 Enter，shell 經歷七個階段：

階段動作產生的問題範例
1. 詞法（tokenize）把字串切成 token：命令名、選項、參數引號未閉合 → syntax error
2. 別名展開（alias）把 ll 換成 ls -laalias 在腳本中通常不生效
3. 解析（parse）產生抽象語法樹（AST）if/for/while 語法在此檢查
4. 大括號展開{1..3} → 1 2 3—
5. 變數 / 命令展開$var、$(...)、$((..)) 替換為值未引號變數會 word splitting
6. 路徑展開（globbing）*.txt 展成檔名清單沒匹配時行為依 nullglob 設定
7. 引號移除剝掉用於保護的引號—
8. 執行fork() + exec() 實際跑命令PATH 找不到 → 127
💡 為什麼順序很重要

記住這個順序能解釋很多怪事。例如：echo $var 中的 $var 在「變數展開」階段已被替換，所以若 var="*.txt"，後面的「路徑展開」會把 *.txt 展成檔名 ── 這就是為什麼 echo "$var"（雙引號跳過 globbing）能保護你。

21.2 fork() 與 exec() ¶

Unix 上「執行一個程式」的標準動作是這對組合：

  • fork()：複製當前行程 ── 父行程拿到子行程的 PID，子行程拿到 0。父子兩邊執行同一份程式碼，但能透過 PID 區分。
  • exec()：把當前行程的記憶體完全替換成另一支程式 ── 子行程「變身」成 ls、grep 等。

shell 執行外部命令時，這對組合的標準流程是：

fork-exec.pseudo 概念 
# 父 shell 要執行 `ls -la`

pid = fork()              # 複製出子行程

if pid == 0:             # 我是子行程
    # （此時還是 shell，但即將變身）
    exec("/bin/ls", ["ls", "-la"])
    # 變身後再也回不來，下一行不會執行

else:                    # 我是父 shell
    waitpid(pid)         # 等子行程結束
    exit_code = $?

用 strace -ff -e clone,execve 觀察一條 pipe 命令，會看到 shell 真的呼叫 clone()（Linux 對應 fork）兩次、execve() 兩次：

strace-pipe.sh Bash 
$ strace -ff -e clone,execve,pipe,dup2 -- bash -c '/bin/echo Hi | grep H' 2>&1
execve("/usr/bin/bash", ...)              # 父 shell 啟動
pipe([3, 4])                              # 建立 pipe (read fd=3, write fd=4)
clone(...) = 19493                        # fork 出 echo 子行程
clone(...) = 19494                        # fork 出 grep 子行程
[pid 19493] dup2(4, 1)                    # echo 把 stdout 接到 pipe write 端
[pid 19494] dup2(3, 0)                    # grep 把 stdin  接到 pipe read 端
[pid 19493] execve("/bin/echo", ...)
[pid 19494] execve("/usr/bin/grep", ...)

21.3 為什麼 subshell 變數會消失 ¶

有了 fork 概念，subshell 行為就清楚了。$(cmd)、(cmd)、pipe 右側、背景任務 cmd & 都是 fork 出獨立的子行程。子行程修改變數，是改自己那份記憶體拷貝──父行程看不到。

subshell-truth.sh Bash 
count=0

# pipe 右側是 subshell（fork 出新行程）
seq 5 | while read n; do
    count=$((count + 1))      # 改的是 subshell 裡的 count
done
echo "$count"                  # 0 ── 父 shell 的 count 沒變

# 解法 1：用 process substitution，迴圈在父 shell 跑
count=0
while read n; do
    count=$((count + 1))
done < <(seq 5)
echo "$count"                  # 5 ── 正確

# 解法 2：Bash 4.2+ 的 lastpipe 選項
shopt -s lastpipe
seq 5 | while read n; do
    count=$((count + 1))
done
# 注意：lastpipe 需要在非互動模式（腳本內）才生效

21.4 檔案描述符的繼承 ¶

fork 時，子行程繼承父行程所有開啟的檔案描述符（除非標記 FD_CLOEXEC）。這是 shell 重導向能運作的根本：shell 在 fork 後、exec 前，先用 dup2() 把 fd 替換掉，子行程 exec 後仍指著正確的「檔案」。

fd-inheritance.sh Bash 
# shell 處理 `ls -la > out.txt` 的內部步驟：
#  1. fork() 出子行程
#  2. 子行程 open("out.txt") -> 拿到 fd=3
#  3. 子行程 dup2(3, 1) -> stdout(fd=1) 變成指向 out.txt
#  4. 子行程 close(3) -> 關掉多餘的 fd
#  5. 子行程 execve("/bin/ls") -> ls 寫 stdout 就寫進 out.txt

# shell 中你可以直接操作 fd
exec 3> debug.log               # 開 fd 3 寫到 debug.log
echo "diagnostics" >&3          # 寫到 fd 3
exec 3>&-                       # 關掉 fd 3

# 同時讀寫 fd 4 → 雙向 pipe
exec 4<> /dev/tcp/example.com/80
echo -e "GET / HTTP/1.0\r\n\r\n" >&4
cat <&4
exec 4>&-

# 列出當前行程的所有 fd
ls -l /proc/$$/fd/

21.5 內建命令 vs 外部命令 ¶

Bash 命令分三種，效能差異巨大：

類別是否 fork是否 exec速度範例
內建（builtin）❌❌⚡⚡⚡ 微秒echo、read、cd、printf、[、[[
函式（function）❌（除非用 pipe）❌⚡⚡⚡ 微秒使用者自己定義的
外部命令✅✅⚡ 毫秒grep、awk、sed、ls
builtin-vs-external.sh Bash 
# 用 type 查命令是內建還外部
type echo                # echo is a shell builtin
type ls                  # ls is /usr/bin/ls
type cd                  # cd is a shell builtin（必須是，否則 shell 自己的 PWD 不會變）
type [                   # [ is a shell builtin
type [[                  # [[ is a shell keyword

# 強制使用外部 / 強制使用內建
\ls                      # 跳過 alias

/bin/echo "hi"           # 強制外部

# 列出所有內建命令
compgen -b               # . : alias bg bind break builtin caller cd...

21.6 訊號的傳播 ¶

訊號的內部模型對排錯極為關鍵：

  • 核心把訊號送給單一行程，不會自動轉發給子行程
  • shell 收到 SIGINT（Ctrl+C）時，預設行為會把該訊號轉發給其前景行程組（foreground process group）
  • 背景任務（cmd &）不在前景組，所以 Ctrl+C 不會殺它
  • PID 1 是核心特例：不會自動處理任何訊號，必須明確註冊 handler
  • kill -TERM <pid> 只送一個行程；kill -TERM -<pid>（負號 PGID）送整個行程組
⚙️ 進階：FD_CLOEXEC 與安全

fd 預設在 exec 後保留。當你開了一個敏感檔案（如含密碼的設定檔），又 exec 別的程式，那程式會繼承這個 fd ── 這是個訊息洩漏面。Bash 用 {fd}<file 開啟的 fd 預設帶 close-on-exec 旗標；舊式 exec 3<file 沒有，必要時手動 close。

第 22 章 · 效能調校

當腳本要處理 10 萬個檔案、要在 CI 限時內跑完、要在容器啟動時快速完成初始化，效能就是議題。Shell 的效能瓶頸幾乎全在「fork 太多次」。本章教你：先量、再優化。

22.1 量測 ── 沒測過就別優化 ¶

measuring.sh Bash 
# 1. time：最基本的計時
time ./script.sh
#   real    0m3.421s   ← 牆上時間
#   user    0m1.823s   ← CPU 時間（user mode）
#   sys     0m0.987s   ← CPU 時間（kernel mode，多代表 fork/IO）

# 2. /usr/bin/time -v：詳細資源使用
/usr/bin/time -v ./script.sh 2>&1 | grep -E 'Elapsed|Maximum|System|User'

# 3. Bash 5.0+ 的 EPOCHREALTIME（微秒精度）
start=$EPOCHREALTIME
do_work
end=$EPOCHREALTIME
echo "耗時: $(awk "BEGIN { print $end - $start }") 秒"

# 4. 計算 fork 次數（Linux）
strace -c -e clone,fork,vfork ./script.sh 2>&1 | tail -5

# 5. 找出最慢的部分（粗略 profiling）
PS4='+ $EPOCHREALTIME '
set -x
do_step_1
do_step_2
set +x
# trace 中的時戳能看出哪一步最慢

22.2 七個最有效的優化 ¶

① 用內建命令取代外部命令

use-builtins.sh Bash 
# ❌ 慢：每次呼叫都 fork+exec
file=$(basename "$path")
dir=$(dirname "$path")
ext=$(echo "$file" | sed 's/.*\.//' )
upper=$(echo "hello" | tr '[:lower:]' '[:upper:]')

# ✅ 快：純 Bash 參數展開（零 fork）
file="${path##*/}"
dir="${path%/*}"
ext="${file##*.}"
upper="${greeting^^}"

# 1000 次測試結果：
#   外部命令版：~5.2 秒
#   內建展開版：~0.05 秒
#   差 100 倍以上

② 減少迴圈內的命令替換

avoid-cmdsubst-in-loop.sh Bash 
# ❌ 慢：1000 次 fork 來取時間
for i in {1..1000}; do
    ts=$(date '+%s')
    log "$ts: processing $i"
done

# ✅ 快：用 Bash 5.0+ 內建的 EPOCHSECONDS（不 fork）
for i in {1..1000}; do
    log "$EPOCHSECONDS: processing $i"
done

# ✅ Bash 5.3+ 的不 fork 命令替換
for i in {1..1000}; do
    val=${ read -r line < "$file"; printf '%s' "$line"; }
done

③ 批次處理而非逐項

batch-processing.sh Bash 
# ❌ 慢：1000 個檔案 → 1000 次 fork rm
for f in *.bak; do
    rm "$f"
done

# ✅ 快：一次 rm 多個（單次 fork）
rm -f -- *.bak

# ✅ 大量檔案（避免 argv 過長）：用 xargs 批次
find . -name '*.bak' -print0 | xargs -0 rm -f

# ✅ 或 find -delete（沒有 fork rm）
find . -name '*.bak' -delete

# 同樣道理：grep 多個 pattern 用 -e，不要呼叫多次
# ❌ grep "a" file; grep "b" file
# ✅ grep -e "a" -e "b" file

④ 用 awk 取代 grep+sed+cut 鏈

awk-instead.sh Bash 
# ❌ 慢：4 個程序串接
cat /etc/passwd | grep '^root' | cut -d: -f3 | head -1

# ✅ 快：awk 一次完成（一個程序）
awk -F: '/^root/ { print $3; exit }' /etc/passwd

# 順帶提：UUOC（Useless Use of Cat）── ShellCheck SC2002
# ❌ cat file | grep x
# ✅ grep x file
# ✅ < file grep x   （另一種風格）

⑤ 平行化 ── xargs -P / GNU parallel

parallel-tools.sh Bash 
# xargs 的 -P 旗標：平行執行
find . -name '*.jpg' -print0 | xargs -0 -P 4 -I {} convert {} -resize 800 {}.thumb.jpg

# -P 數量決定並行度：
#   -P 1   ── 順序執行
#   -P 4   ── 4 個同時跑（CPU 核數常用）
#   -P 0   ── 全部同時（小心 fork bomb）

# 自動使用全部核數
find . -name '*.log' -print0 | xargs -0 -P "$(nproc)" gzip

# GNU parallel：更強大的平行工具
parallel -j 4 gzip ::: *.log
parallel --bar --jobs 4 process_file ::: file1 file2 file3
parallel --keep-order 'echo Job {}' ::: 1 2 3 4   # 保持輸出順序

# 跨機器執行（GNU parallel 殺手鐧）
parallel -S server1,server2 'process_file {}' ::: *.dat

# 純 Bash 背景任務（小規模適用）
for task in "${tasks[@]}"; do
    # 達上限就 wait 任一
    while (( $(jobs -rp | wc -l) >= 4 )); do
        wait -n
    done
    process "$task" &
done
wait

⑥ 用陣列取代重覆 read

arrays-faster.sh Bash 
# ❌ 慢：每次處理都重讀檔
for i in {1..100}; do
    line=$(sed -n "${i}p" big.txt)
    process "$line"
done

# ✅ 快：mapfile 一次讀進陣列
mapfile -t lines < big.txt
for i in {0..99}; do
    process "${lines[$i]}"
done

⑦ 知道何時放棄 shell

當你需要做以下任一件事，應該換成 Python / Go / Rust：

  • JSON / XML / YAML 結構化資料的嚴肅處理（jq 簡單查詢可，複雜變換就難）
  • 數學運算（特別是浮點）
  • 需要正確處理 Unicode 邊緣情況
  • 多執行緒或非同步 I/O（不只是平行 fork）
  • HTTP/API 客戶端含複雜重試、分頁、認證
  • 業務邏輯超過 500 行
💡 經驗法則

shell 適合做「把多個程式組合起來、移動資料、設定環境」的膠水工作。當邏輯本身很複雜時，shell 就不是好選擇。多數 SRE 團隊的內部慣例是：超過 100 行的腳本要嚴肅評估「這該不該重寫成其他語言」。

第 23 章 · 安全進階

第 17 章涵蓋了基本注入防禦。本章進一步：TOCTOU 競態條件、SUID 風險、稽核軌跡、安全沙箱。這些議題在腳本以特權執行（root、SUID、systemd 服務）時尤其重要。

23.1 TOCTOU：時間檢查/時間使用 ¶

TOCTOU（Time of Check, Time of Use）是經典的競態條件漏洞：你「檢查」一個檔案的狀態，然後「使用」它──但兩步之間，攻擊者可能把檔案抽換掉。

toctou-vulnerable.sh Bash脆弱 
# ❌ 經典 TOCTOU 漏洞
if [[ -f "$file" && -O "$file" ]]; then
    # 在這個極短時間窗口內，攻擊者可能：
    #   rm "$file"
    #   ln -s /etc/passwd "$file"
    cat "$file" > /tmp/output      # ← 現在讀到 /etc/passwd 了
fi

# ❌ 另一個常見場景：先檢查、後寫
if [[ ! -e "$temp" ]]; then
    # 攻擊者在這建 symlink: $temp → /etc/important
    echo "data" > "$temp"           # 寫到 /etc/important 了
fi

防禦策略

toctou-defense.sh Bash 
# ✅ 1. 用 mktemp（核心保證唯一、O_EXCL 原子建立）
tmp=$(mktemp)
trap 'rm -f "$tmp"' EXIT
echo "data" > "$tmp"             # 不需要先 -e 檢查

# ✅ 2. 在私有目錄工作（攻擊者沒寫權限就無法 race）
readonly WORK_DIR="$HOME/.myapp/work"
mkdir -p "$WORK_DIR"
chmod 700 "$WORK_DIR"
# 之後在 $WORK_DIR 裡的操作都安全

# ✅ 3. 用 noclobber 防止覆寫
set -C   # noclobber：> 不會覆寫已存在的檔案
echo "data" > "$file" || die "檔案已存在或建立失敗"

# 配合：>| 強制覆寫（明確意圖）
echo "data" >| "$file"

# ✅ 4. 最強保證：直接用 fd（一次 open，後續用 fd 操作）
exec 3> "$tmp"                  # 一次 open
echo "data" >&3                # 寫到那個 fd（同一個 inode）
exec 3>&-

# ✅ 5. 檢查連結：用 -L 排除 symlink
if [[ -f "$file" && ! -L "$file" ]]; then
    cat "$file"
fi
⚠️ /tmp 的內生風險

/tmp 是所有人可寫的共享目錄，最容易發生 TOCTOU。機敏腳本應避免在 /tmp 建檔，改用 $XDG_RUNTIME_DIR（systemd 提供，0700 權限）、~/.cache/myapp/，或腳本自己的私有目錄。若一定要用 /tmp，永遠走 mktemp。

23.2 SUID 腳本的危險 ¶

SUID 是 Unix 給特定執行檔「以 owner 身分（通常是 root）執行」的權限位元。shell 腳本絕對不該設 SUID。原因：

  • 多數 Linux 核心會直接忽略 shell 腳本的 SUID 位元（從 1990s 起的安全防禦）
  • 即使忽略了，shebang 與 exec 之間有時間窗口可被替換
  • 環境變數（IFS、PATH、BASH_ENV、ENV）都能成為攻擊面
suid-alternatives.sh Bash 
# ❌ 絕對不要
sudo chmod +s my-script.sh

# ✅ 替代 1：用 sudoers 細粒度授權
# /etc/sudoers.d/myapp:
#   alice ALL=(root) NOPASSWD: /usr/local/bin/my-script.sh
# 然後 alice 執行: sudo /usr/local/bin/my-script.sh

# ✅ 替代 2：寫一個 C wrapper（很短）
# wrapper.c:
#   #include <unistd.h>
#   int main() {
#       setuid(0);
#       execv("/usr/local/bin/my-script.sh", ...);
#   }
# gcc wrapper.c -o wrapper
# chown root:root wrapper; chmod 4755 wrapper

# ✅ 替代 3：systemd service（最現代）
# 把腳本包成 service，使用者透過 systemctl 觸發
# systemctl start my-script.service

23.3 安全沙箱 ¶

sandboxing.sh Bash 
# 1. systemd 沙箱（一行加好幾層防護）
# /etc/systemd/system/myapp.service:
#   [Service]
#   ExecStart=/usr/local/bin/myapp.sh
#   PrivateTmp=true              # 私有 /tmp
#   PrivateNetwork=true          # 沒網路
#   ProtectSystem=strict         # /usr 與 /boot 唯讀
#   ProtectHome=read-only        # /home 唯讀
#   NoNewPrivileges=true         # 禁用 setuid
#   RestrictRealtime=true        # 禁用即時 schedule
#   SystemCallFilter=@system-service  # syscall 白名單
#   ReadWritePaths=/var/lib/myapp     # 唯一可寫的路徑

# 2. firejail（替代/補強）
firejail --read-only=/etc --net=none ./script.sh

# 3. 在容器中跑（最簡單的沙箱）
docker run --rm --read-only --tmpfs /tmp \
    --cap-drop=ALL --no-new-privileges \
    -v "$PWD:/work:ro" \
    bash:5 /work/script.sh

# 4. unshare：手動使用 namespace
unshare --net --mount --pid --fork bash script.sh

# 5. 在腳本內降權限
if [[ $EUID -eq 0 ]]; then
    # 做完需要 root 的事後 ── 切到非特權使用者繼續

J · Top 20 反模式（要記下來） ¶

  1. 未引號變數 ── rm $file。永遠 rm "$file"，並考慮加 --。
  2. 解析 ls ── for f in $(ls *.txt)。改用 glob：for f in *.txt。
  3. 等號旁邊有空格 ── name = "x"（被當命令）。寫成 name="x"。
  4. 沒 shebang ── 由呼叫者的 shell 決定行為。永遠加 #!/usr/bin/env bash。
  5. 沒嚴格模式 ── 缺 set -euo pipefail，錯誤被吞。
  6. 用 cd 不檢查 ── cd dir; rm -rf *。改 cd dir \|\| exit 1。
  7. 函式內忘加 local ── 變數污染呼叫者命名空間。
  8. local var=$(cmd) ── 隱藏 cmd 的退出碼。分兩行寫。
  9. 用 eval ── 多數情境有更好替代（${!var}、nameref）。
  10. shell form 的 ENTRYPOINT ── CMD myapp 變成 sh -c 包起來，訊號收不到。改 CMD ["myapp"]。
  11. 用可預測的 tmp 檔名 ── /tmp/myapp.$$。永遠用 mktemp。
  12. 用 [ $a = $b ] ── $a 為空時語法錯。改 [ "$a" = "$b" ] 或 [[ $a == $b ]]。
  13. cmd \| while read 想改變數 ── subshell 變數消失。改 while ... done < <(cmd)。
  14. Bash ((var++)) 在 set -e 下死掉 ── 因後綴自增回傳 0。改 ((++var))。
  15. 2>&1 > out 順序錯 ── stderr 仍在原處。正確：> out 2>&1。
  16. shell 腳本設 SUID ── 多數核心忽略，且環境變數可被攻擊。改用 sudoers 或 systemd。
  17. 密碼寫進命令列 ── 會被 ps、history 看到。改用環境變數或檔案。
  18. GitHub Actions 直接 ${{ pr.title }} ── 命令注入面。改 env 傳入。
  19. BSD getopt ── 不能處理空白引數。用 getopts + 手刻長選項。
  20. 路徑寫死絕對路徑或假設 cwd ── 用 SCRIPT_DIR=$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)。
