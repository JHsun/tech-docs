第 16 章 · 品質工具鏈

2026 年寫 shell 腳本的工程師，在編輯器與 CI 上應該都有這三樣工具：ShellCheck（靜態分析）、shfmt（格式化）、Bats（單元測試）。它們三個合起來等同於 Python 界的 ruff + black + pytest，是讓你寫的 shell 腳本可被同事看懂、可被機器驗證、可被長期維護的基礎建設。

16.1 ShellCheck：靜態分析的最強武器 ¶

ShellCheck（GPLv3 授權，Haskell 寫成）是 shell 界最重要的工具，沒有之一。它能在腳本執行前就抓出 90% 的常見錯誤，包含：未引號變數、解析 ls 的危險、cd 失敗未檢查、read 沒加 -r、shebang 與實際語法不符等等。每一個 SC 警告都對應到 wiki 上的詳細解釋。

安裝

install-shellcheck.sh Bash 
# macOS
brew install shellcheck

# Ubuntu / Debian
sudo apt install shellcheck

# RHEL / Fedora
sudo dnf install ShellCheck

# Arch
sudo pacman -S shellcheck

# Docker（不想裝在本機）
docker run --rm -v "$PWD:/mnt" koalaman/shellcheck:stable myscript.sh

# 線上即時檢查（適合分享範例）
# https://www.shellcheck.net

常見的 10 個 SC 警告

編號含義修正
SC2086變數未引號（最常見的錯誤）$var → "$var"
SC2046$(cmd) 未引號，會被 word splittingecho $(date) → echo "$(date)"
SC2155local var=$(cmd) 會掩蓋 cmd 的退出碼分兩行：先 local，後賦值
SC2164cd 沒檢查失敗cd dir \|\| exit 1
SC2148缺 shebang加 #!/usr/bin/env bash
SC2002沒必要的 cat（UUOC）cat f \| grep x → grep x f
SC2068$@ 未引號$@ → "$@"
SC2129連續多次 >> 重導向到同檔用 { ...; } >> file
SC2034變數宣告未使用移除或加 # shellcheck disable
SC1091找不到 source 的目標檔加 # shellcheck source=path

使用範例

shellcheck-usage.sh Bash 
# 基本用法
shellcheck script.sh

# 嚴格度設定
shellcheck --severity=warning script.sh    # error/warning/info/style

# 排除特定警告
shellcheck --exclude=SC1091,SC2034 script.sh

# 多種輸出格式
shellcheck -f tty script.sh        # 預設彩色終端
shellcheck -f json script.sh       # JSON（給 CI 處理）
shellcheck -f checkstyle script.sh # Jenkins/CheckStyle
shellcheck -f gcc script.sh        # 多數 IDE 可解析

# 檢查目錄下所有 sh 腳本
find . -name '*.sh' -exec shellcheck {} +

# 在腳本中暫時關閉某條警告（內嵌指令）
# shellcheck disable=SC2086
result=$some_var     # 故意不引號的合理場景

# 整檔指令（放在 shebang 後）
#!/usr/bin/env bash
# shellcheck shell=bash
# shellcheck disable=SC1091  全檔禁用

16.2 shfmt：自動格式化 ¶

shfmt 之於 shell，等同於 Prettier 之於 JavaScript、black 之於 Python ── 一個有觀點的格式化工具，幫團隊一致縮排、引號風格、case 對齊。它支援 bash、POSIX sh、mksh，並可附帶簡化（-s）。

shfmt-usage.sh Bash 
# 安裝
brew install shfmt                              # macOS
go install mvdan.cc/sh/v3/cmd/shfmt@latest      # Go
sudo apt install shfmt                          # Ubuntu 22.04+

# 看會怎麼改（diff 格式）
shfmt -d script.sh

# 直接改檔（in-place）
shfmt -w script.sh

# 整個專案
shfmt -w -l .   # -l 列出修改了哪些檔

# 推薦選項組合（Google Shell Style Guide）
shfmt -i 2 -ci -s -w script.sh
#  -i 2  ── 縮排 2 空格（不要用 tab）
#  -ci   ── case 內陳述對齊縮排
#  -s    ── simplify（簡化等價的寫法，例如 [ ] -> [[ ]]）
#  -bn   ── 二元運算子放行首（cmd && \\
#                                cmd2）
#  -ln=bash ── 明確指定方言

# 也能格式化 stdin
echo 'if true;then echo hi;fi' | shfmt -i 4

16.3 Bats：單元測試框架 ¶

Bats（Bash Automated Testing System）是 shell 界的 pytest／jest。它把 .bats 檔當成測試套件，每個 @test 區塊獨立執行、自動報告通過/失敗。2026 年的 bats-core（社群維護分支）支援平行測試、tap-13 輸出、JUnit XML、setup/teardown hook 等現代化功能。

install-bats.sh Bash 
# 安裝
brew install bats-core                              # macOS
sudo apt install bats                               # Ubuntu
git clone https://github.com/bats-core/bats-core.git
cd bats-core && ./install.sh /usr/local           # 從原始碼

第一支測試

tests/example.bats Bash 
#!/usr/bin/env bats

# setup() 在每個測試前都會執行
setup() {
    load '../src/utils.sh'        # 載入要測的腳本
    TMP=$(mktemp -d)
}

# teardown() 在每個測試後執行
teardown() {
    rm -rf "$TMP"
}

@test "echo prints hello" {
    result=$(echo "hello")
    [ "$result" = "hello" ]
}

@test "addition function works" {
    run my_add 2 3
    [ "$status" -eq 0 ]
    [ "$output" = "5" ]
}

@test "fails when given no args" {
    run my_add
    [ "$status" -ne 0 ]
    [[ "$output" =~ "missing" ]]
}

@test "creates lock file" {
    run create_lock "$TMP/test.lock"
    [ "$status" -eq 0 ]
    [ -f "$TMP/test.lock" ]
}

# 跳過測試（暫時不跑）
@test "future feature" {
    skip "尚未實作"
    run new_feature
    [ "$status" -eq 0 ]
}

執行

run-bats.sh Bash 
# 跑單一檔
bats tests/example.bats

# 跑整個目錄
bats tests/

# 平行執行（bats-core 1.x+）
bats --jobs 4 tests/

# 輸出 JUnit XML（給 CI 收集結果）
bats --report-formatter junit -o reports tests/

# 過濾測試名稱
bats --filter "addition" tests/

# 詳細 trace
bats --verbose-run tests/
💡 run 與 $status、$output

Bats 提供的 run cmd 會執行命令並把退出碼存入 $status、stdout 存入 $output、按行存入 ${lines[@]}。即使被測命令失敗，run 本身一定成功，這樣你能斷言失敗情境。這是 Bats 最核心的 API。

16.4 把工具串成 CI ¶

有了三大工具，下一步是讓機器自動跑──開發者不會永遠記得跑 lint。常見三種整合方式：

方式 1：pre-commit hook（提交前自動）

.pre-commit-config.yaml YAML 
repos:
  - repo: https://github.com/shellcheck-py/shellcheck-py
    rev: v0.10.0.1
    hooks:
      - id: shellcheck
        args: [--severity=warning]

  - repo: https://github.com/scop/pre-commit-shfmt
    rev: v3.8.0-1
    hooks:
      - id: shfmt
        args: [-i, '2', -ci, -s]

# 安裝後：pre-commit install，提交前自動跑

方式 2：GitHub Actions（每次推送）

.github/workflows/shell-ci.yml YAML 
name: Shell CI

on: [push, pull_request]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: ShellCheck
        uses: ludeeus/action-shellcheck@master
        with:
          severity: warning
          scandir: ./scripts

      - name: shfmt
        uses: luizm/action-sh-checker@master
        env:
          SHFMT_OPTS: '-i 2 -ci -s -d'

      - name: Bats tests
        run: |
          sudo apt-get install -y bats
          bats tests/

方式 3：Makefile 或 task runner

Makefile Make 
SCRIPTS := $(shell find . -name '*.sh' -not -path './node_modules/*')

.PHONY: lint fmt test all

lint:
	shellcheck --severity=warning $(SCRIPTS)

fmt:
	shfmt -i 2 -ci -s -w $(SCRIPTS)

fmt-check:
	shfmt -i 2 -ci -s -d $(SCRIPTS)

test:
	bats tests/

all: lint fmt-check test
	@echo "✓ all checks passed"

第 17 章 · 安全與防禦性編程

Shell 腳本最常見的安全問題是 命令注入 ── 因為 shell 「先展開、再執行」的特性，未經檢查的使用者輸入可能被當成命令。本章把所有腳本注入面攤開來講，並給出每個面的防禦策略。

17.1 注入攻擊面 ¶

injection-examples.sh Bash攻擊面 
# ❌ 危險 1：直接展開使用者輸入到命令中
filename="$1"
rm $filename           # 若 $1 = "* /home/me/important", 就災難了

# ❌ 危險 2：eval（最危險的命令）
eval "echo $user_input"   # $user_input = ';rm -rf /' → boom

# ❌ 危險 3：未引號的命令替換
file=$(get_user_input)
ls $file                # 若值含 & 或 ; 或 |，會做意想不到的事

# ❌ 危險 4：把 user input 拼進 SSH/curl 命令
ssh user@host "ls $dir"  # 引號保護不了，遠端 shell 會再展開一次

# ❌ 危險 5：sudoers / cron 中的 PATH 污染
# 如果腳本只寫 `tar` 而非 `/usr/bin/tar`，
# 攻擊者可在 PATH 前段放偽造的 tar

17.2 防禦清單（必讀） ¶

① 永遠引號變數

always-quote.sh Bash 
# ❌
rm $file
ls $@
echo $(get_value)

# ✅
rm -- "$file"               # -- 防止檔名以 - 開頭被當選項
ls "$@"
echo "$(get_value)"

② 用 -- 終止選項

rm "$file" 在 $file = "-rf" 時仍會被當選項。rm -- "$file" 才是真的安全。cp/mv/rm/grep/touch 等 GNU coreutils 都支援 --。

③ 避免 eval

eval 是 shell 中最危險的單一構造。幾乎所有需要 eval 的場景都有更好的替代：

avoid-eval.sh Bash 
# ❌ eval 的常見錯誤用法
key="username"
eval "value=\$$key"          # 想做間接引用

# ✅ 用 ${!var}（Bash 內建間接引用）
value=${!key}

# ✅ 或用 nameref（Bash 4.3+，更現代）
declare -n ref="$key"
value="$ref"

# ❌ 用 eval 動態建陣列
for i in 1 2 3; do
    eval "arr_$i=value_$i"
done

# ✅ 直接用陣列
declare -a arr
for i in 1 2 3; do
    arr+=("value_$i")
done

④ 嚴格檢查輸入

validate-input.sh Bash 
# 數字：用 regex 強制驗證
if [[ ! "$port" =~ ^[0-9]+$ ]]; then
    die "port 必須是數字: $port"
fi
(( port >= 1 && port <= 65535 )) || die "port 範圍 1-65535"

# 識別字（如使用者名稱、容器名）
if [[ ! "$name" =~ ^[A-Za-z][A-Za-z0-9_-]{0,31}$ ]]; then
    die "無效名稱: $name"
fi

# 路徑：必須在預期的根目錄下（防止 ../../../etc/passwd）
canonical=$(realpath -- "$user_path")
[[ "$canonical" == /allowed/root/* ]] || die "路徑不被允許"

# 白名單比黑名單可靠
case "$action" in
    start|stop|restart|status) ;;
    *) die "未知動作: $action" ;;
esac

⑤ 控制 PATH

safe-path.sh Bash 
# 在腳本開頭明確設定 PATH（防止繼承的 PATH 含惡意路徑）
export PATH="/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"

# 或在敏感命令使用絕對路徑
/usr/bin/tar czf backup.tar.gz /data
/usr/bin/curl -fsS https://api.example.com

⑥ 暫存檔安全

tmp-files.sh Bash 
# ❌ 可預測檔名 → race condition / symlink attack
tmp="/tmp/myapp.$$"

# ✅ 用 mktemp（核心保證唯一性、權限正確）
tmp=$(mktemp -t myapp.XXXXXX)
trap 'rm -f "$tmp"' EXIT

# ✅ 暫存目錄
tmp_dir=$(mktemp -d -t myapp.XXXXXX)
trap 'rm -rf "$tmp_dir"' EXIT

# ✅ 進階：用 0700 權限
umask 077
tmp=$(mktemp)

⑦ 機敏資料處理

secrets.sh Bash 
# ❌ 把密碼寫進命令列 ── 會被 ps、history 看到
mysql -u root -psecret123

# ✅ 用環境變數（仍要小心 export 範圍）
mysql -u root -p"$MYSQL_PASSWORD"

# ✅ 從檔案讀（mode 0600）
mysql --defaults-file=/etc/myapp/.my.cnf

# ✅ 從 stdin / heredoc
mysql -u root -p <<'EOF'
SELECT 1;
EOF

# 不要把機敏內容 echo / 印到 log
log "Connecting with token: $TOKEN"   # ❌
log "Connecting (token: ${TOKEN:0:4}...)"   # ✅ 只印前 4 碼

# 在腳本結束前清空變數
TOKEN=""
unset TOKEN

# 避免在 set -x 模式洩漏密碼
set +x
mysql -p"$PASSWORD"
set -x

17.3 權限最小化 ¶

  • 不必要的 sudo：寫腳本時想清楚「這條指令真的需要 root 嗎？」── 多數時候不需要。
  • 檔案權限：含密碼的腳本應 chmod 700（自己讀寫執行）；含設定的 chmod 600。
  • sudoers 細粒度：若腳本必須 sudo 一個特定命令，在 /etc/sudoers.d/ 只授權那一個命令給特定使用者，而非全 root。
  • 檢查 effective UID：[[ $EUID -eq 0 ]] 用來判斷「現在是 root 嗎？」可阻止意外以 root 執行。

第 18 章 · 可移植性

「我的腳本在 Linux 跑得好好的，怎麼到 macOS 就壞？」──這是 shell 的常態問題。本章讓你的腳本能跨 macOS、各 Linux 發行版、BSD、WSL 都跑得起來。

18.1 三軸差異圖 ¶

18.2 偵測平台與寫條件分支 ¶

detect-os.sh Bash 
# 用 uname 判斷
case "$(uname -s)" in
    Linux*)   PLATFORM="linux" ;;
    Darwin*)  PLATFORM="macos" ;;
    CYGWIN*|MINGW*|MSYS*) PLATFORM="windows" ;;
    *BSD)     PLATFORM="bsd" ;;
    *)        PLATFORM="unknown" ;;
esac

# 區分 Linux 發行版
if [[ -f /etc/os-release ]]; then
    . /etc/os-release
    echo "Distro: $ID $VERSION_ID"   # ubuntu, fedora, alpine...
fi

# 判斷可用命令再選擇
if command -v gsed >/dev/null; then
    SED=gsed                             # macOS 上裝 brew install gnu-sed 後
else
    SED=sed
fi

# 寫跨平台 sed in-place
sed_inplace() {
    if [[ "$PLATFORM" == "macos" ]]; then
        sed -i '' "$@"
    else
        sed -i "$@"
    fi
}
# 但更簡單的解法是 sed -i.bak（兩家都接受），最後 rm *.bak

18.3 POSIX 子集寫法 ¶

當你的腳本必須在「最小相容環境」（Alpine 容器、嵌入式 BusyBox、Solaris）跑時，要降到 POSIX sh 子集。不能用：

非 POSIX 的 Bash 功能POSIX 替代
[[ ]][ ] 配合引號保護
(( ))$((expr)) 算術展開或 expr
陣列 arr=(a b c)用 $@ 或字串切割
關聯陣列無，要改用 awk 或檔案
local無正式對應，但 dash/ash 也支援
echo -e用 printf
function name() {}用 name() {}
${var,,} 大小寫用 tr
=~ 正規用 grep / case glob
<<< here-string用 echo "..." \| 或 here-doc
<(cmd) 行程替換用暫存檔
set -o pipefail無，需自己檢查
💡 checkbashisms 自動偵測 bashism

Debian 提供的 checkbashisms 工具能掃出 POSIX sh 中誤用的 bashism。apt install devscripts 安裝。
ShellCheck 也能做：在 shebang 寫 #!/bin/sh，它會自動切換到 POSIX 模式並警告所有 bashism。

第 19 章 · 系統整合

真實世界的 shell 腳本不只是手動執行──它會被 cron 排程、被 systemd 管理、會寫日誌、需要與其他系統服務協作。本章串起這些。

19.1 cron 排程 ¶

crontab-syntax cron 
# crontab 格式：分 時 日 月 週 命令
#
#  ┌────────── 分鐘 (0 - 59)
#  │ ┌──────── 小時 (0 - 23)
#  │ │ ┌────── 日   (1 - 31)
#  │ │ │ ┌──── 月   (1 - 12)
#  │ │ │ │ ┌── 週   (0 - 7, 0/7=Sun)
#  │ │ │ │ │
#  * * * * *  command

# 常用範例：
0 3 * * *       /usr/local/bin/backup.sh           # 每天 3:00
*/5 * * * *     /usr/local/bin/health-check.sh     # 每 5 分鐘
0 */6 * * *     /usr/local/bin/sync.sh             # 每 6 小時
0 0 1 * *       /usr/local/bin/monthly.sh          # 每月 1 號 0:00
0 9 * * 1-5     /usr/local/bin/weekday-only.sh     # 平日早上 9:00
@reboot         /usr/local/bin/on-startup.sh       # 開機後
@daily          /usr/local/bin/daily.sh            # = 0 0 * * *

cron 的常見坑

坑解法
$PATH 極簡（只有 /usr/bin:/bin）腳本內絕對路徑、或在 crontab 開頭設 PATH=...
沒有 $HOME/環境變數腳本內明確 source ~/.bashrc（小心用）或顯式宣告
沒 TTY，互動命令會卡死避免 read、vim 之類；輸出重導到 log
只 email 給 owner，不易看到失敗把 stdout/stderr 都寫到 log，並另加 alerting
排程時間漂移（負載重時）用 flock 防止重疊，或改用 systemd timer
cron-template.sh Bash 
# crontab 中的「正確」呼叫方式
#
# 0 3 * * * /usr/bin/flock -n /tmp/backup.lock /usr/local/bin/backup.sh \
#           >> /var/log/backup.log 2>&1
#
# 重點：
#  - flock 防止重疊執行
#  - 絕對路徑（不依賴 PATH）
#  - 重導 log 含 stderr

19.2 systemd timer（cron 的現代替代） ¶

cron 仍可用，但在 systemd 系統（多數現代 Linux）上，systemd timer 是更好的選擇：能補跑（Persistent=true）、能依賴其他單元、能設資源限制、有結構化日誌。

/etc/systemd/system/backup.service systemd 
[Unit]
Description=Daily Backup
After=network.target
ConditionPathExists=/data

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh
StandardOutput=journal
StandardError=journal

# 安全強化
User=backup
Group=backup
PrivateTmp=true
ProtectSystem=strict
ProtectHome=read-only
NoNewPrivileges=true
ReadWritePaths=/data /backups

# 資源限制
MemoryMax=512M
CPUQuota=50%

19.3 日誌：syslog 與 journal ¶

logging.sh Bash 
# 用 logger 寫到 syslog（或 systemd-journal）
logger -t myapp "Service started"
logger -t myapp -p user.error "Failed to connect"
logger -t myapp -p local0.info "Custom facility"

# 同時寫到 stdout 和 syslog
echo "Message" | tee >(logger -t myapp)

# 結構化日誌函式（推薦）
log() {
    local level=$1; shift
    local ts; ts=$(date '+%Y-%m-%dT%H:%M:%S%z')
    printf '%s [%s] %s\n' "$ts" "$level" "$*"
    logger -t "${0##*/}" -p "user.$(tr '[:upper:]' '[:lower:]' <<< "$level")" "$*"
}

log INFO  "Starting deployment"
log ERROR "Connection failed: $err"
log WARN  "Retrying in 5s"

# 從 systemd journal 查
journalctl SYSLOG_IDENTIFIER=myapp
journalctl -p err SYSLOG_IDENTIFIER=myapp   # 只看 error
journalctl _PID=$(pgrep myapp)           # 用 PID 過濾

第 20 章 · 容器與 CI/CD

2026 年的 shell 腳本，半數場景是在容器內、CI runner 中執行。Dockerfile 的 RUN、ENTRYPOINT、CMD 都要寫 shell；GitHub Actions 的每個 step 大量用 bash。本章涵蓋這兩個主場景的最佳實踐。

20.1 Dockerfile 中的 shell ¶

Dockerfile-shell-tips Dockerfile 
# 1. RUN 預設用 /bin/sh -c，要明確指定 bash
SHELL ["/bin/bash", "-Eeuo", "pipefail", "-c"]

# 2. RUN 連續命令用 && 串接（單層 layer + 失敗即停）
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        ca-certificates \
        curl \
        jq && \
    rm -rf /var/lib/apt/lists/*

# 3. ENTRYPOINT vs CMD：用 exec 形式（陣列）─ 訊號才能正確傳遞
ENTRYPOINT ["/usr/local/bin/entrypoint.sh"]
CMD ["server"]

# 4. 啟動腳本最後用 exec ── 替換 shell 行程，讓你的 app 成為 PID 1
# entrypoint.sh:
#   #!/usr/bin/env bash
#   set -Eeuo pipefail
#   ... 初始化 ...
#   exec "$@"      # ← 重要！

# 5. STOPSIGNAL 明確宣告（預設 SIGTERM）
STOPSIGNAL SIGINT

20.2 PID 1 與訊號處理 ¶

容器內，第一個行程是 PID 1，Linux 核心對它特殊處理：不會主動傳 SIGTERM 給沒有註冊 handler 的 PID 1。所以如果 PID 1 是個沒有 trap 的 shell，docker stop 就會等 10 秒後 SIGKILL 強殺，造成資料遺失。

⚠️ 經典陷阱：PID 1 是 shell

當 ENTRYPOINT 或 CMD 用「shell 形式」（CMD myapp 而非 CMD ["myapp"]），會變成 /bin/sh -c myapp，PID 1 是 shell。Docker 發 SIGTERM 給 shell，shell 沒處理也沒轉發給子行程 myapp，於是 myapp 永遠收不到優雅關機訊號。

正確做法：三選一

方法 A：用 exec form（最推薦）

Dockerfile-exec-form Dockerfile 
# ❌ shell form ── 變成 sh -c 包起來
ENTRYPOINT myapp --port 8080

# ✅ exec form ── myapp 直接成為 PID 1
ENTRYPOINT ["myapp", "--port", "8080"]

# 驗證：docker exec container ps aux 看 PID 1 是否就是你的 app

方法 B：entrypoint 腳本最後用 exec "$@"

docker-entrypoint.sh Bash 
#!/usr/bin/env bash
set -Eeuo pipefail

# 1. 環境變數驗證
: "${DATABASE_URL:?必須提供 DATABASE_URL}"
: "${PORT:=8080}"

# 2. 等待依賴服務（如 DB）就緒
until nc -z db 5432; do
    echo "等待 db..."
    sleep 1
done

# 3. migrations / 初始化
./manage.sh migrate

# 4. exec 替換 shell 行程 ── myapp 變成 PID 1，能收訊號
exec "$@"

方法 C：用 init 系統（tini / dumb-init）

當你的 app 真的不能改、又無法是 PID 1（比如有子行程、有 zombie 問題），用 tini。它輕量、Docker 1.13+ 內建（docker run --init），多數情境下能解決問題。

Dockerfile-with-tini Dockerfile 
FROM alpine:3.20

RUN apk add --no-cache tini

COPY app /usr/local/bin/app

# tini 接 SIGTERM，正確 forward 給 app；同時收割 zombie
ENTRYPOINT ["/sbin/tini", "--"]
CMD ["app"]

# 或者 docker run 時加 --init 旗標：
# docker run --init my-image

含 trap 的 entrypoint（手動 forward 訊號）

entrypoint-with-trap.sh Bash 
#!/usr/bin/env bash
# 進階：entrypoint 想保留控制權（記錄、健康檢查），需手動 forward 訊號
set -Eeuo pipefail

CHILD_PID=""

forward() {
    local sig=$1
    [[ -n "$CHILD_PID" ]] && kill -"$sig" "$CHILD_PID" 2>/dev/null || true
}

trap 'forward TERM' TERM
trap 'forward INT'  INT
trap 'forward QUIT' QUIT

"$@" &
CHILD_PID=$!

# wait 必須在 trap 後（不然訊號到達時 wait 會被中斷但沒做 forward）
wait "$CHILD_PID"
exit $?

20.3 GitHub Actions 中的 shell ¶

.github/workflows/ci.yml YAML 
name: CI

on: [push, pull_request]

defaults:
  run:
    shell: bash       # 全 workflow 預設 bash（Linux 預設就是，Windows 要明指）

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      # 多行 shell 用 | ── 自動加 set -e（GH Actions 特性）
      - name: Build
        run: |
          # GitHub 預設加 set -eo pipefail
          # 等同 #!/usr/bin/env bash + set -eo pipefail
          npm ci
          npm run build

      # 自定 shell 旗標
      - name: Strict mode
        shell: bash
        run: |
          set -euo pipefail
          IFS=$'\n\t'
          ./scripts/deploy.sh

      # 用環境變數傳遞輸出（避免命令注入）
      - name: Use input safely
        env:
          USER_INPUT: ${{ github.event.pull_request.title }}
        run: |
          # ✅ 正確：透過 env 變數，不會被 ${{ }} 直接展開到 shell
          echo "PR title: $USER_INPUT"

      # ❌ 危險示範：直接 ${{ }} 展開 ── 攻擊者可在 PR 標題注入命令
      # run: echo "PR title: ${{ github.event.pull_request.title }}"
      #   若 PR 標題是 "); rm -rf /; echo (" 就 GG
🚨 GitHub Actions 命令注入

GitHub Security Lab 已報告許多熱門專案被此攻擊。永遠把使用者可控的輸入（PR title、issue body、commit message）透過 env: 傳入，而非直接在 run: ${{ ... }} 中展開。前者把值放進環境變數、後者把值拼進腳本本體。

20.4 容器內的 shell 腳本最佳實踐總結 ¶

  1. 用 SHELL ["/bin/bash", "-Eeuo", "pipefail", "-c"] 讓 RUN 走嚴格模式
  2. ENTRYPOINT/CMD 用陣列形式（exec form）
  3. entrypoint 腳本最後用 exec "$@" 替換 shell
  4. 需要 init 系統時用 tini（或 docker run --init）
  5. STOPSIGNAL 若 app 期待非 SIGTERM 訊號（如某些 nginx）
  6. 單一 RUN 多命令用 && 串接，最後 rm -rf /var/lib/apt/lists/*
  7. 不要在 image 內留下 secrets（每層都會被打包）
  8. 用非 root 使用者跑 app（USER appuser）
  9. HEALTHCHECK 寫個簡單的 shell 腳本檢查存活
