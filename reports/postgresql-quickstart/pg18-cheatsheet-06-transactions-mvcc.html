🔄 交易基本操作 (Ch13.1)
-- 開始交易
BEGIN;
BEGIN TRANSACTION;
BEGIN ISOLATION LEVEL ...;
START TRANSACTION;

-- 提交
COMMIT;
END;

-- 回滾
ROLLBACK;
ABORT;

-- 儲存點
SAVEPOINT sp1;
ROLLBACK TO SAVEPOINT sp1;
RELEASE SAVEPOINT sp1;

-- 設定交易特性
SET TRANSACTION
  ISOLATION LEVEL ...
  READ ONLY | READ WRITE
  DEFERRABLE | NOT DEFERRABLE;
PostgreSQL 的每個 SQL 語句都在交易中執行（autocommit 模式自動 BEGIN + COMMIT）
🔒 交易隔離層級 (Ch13.2)
隔離層級髒讀不可重複讀幻讀序列化異常說明
READ UNCOMMITTED❌*可能可能可能PG 中等同 READ COMMITTED
READ COMMITTED❌可能可能可能預設值。每條語句看到已提交資料
REPEATABLE READ❌❌❌*可能快照在交易開始時固定。*PG 不會有幻讀
SERIALIZABLE❌❌❌❌完全可序列化（SSI 演算法）

READ COMMITTED 行為

-- 每條 SELECT 看到的是該語句
-- 開始時已提交的資料快照
-- UPDATE/DELETE 會等待衝突行的
-- 交易提交後重新評估 WHERE
BEGIN;
SELECT * FROM t; -- 快照A
-- 其他交易提交了變更...
SELECT * FROM t; -- 快照B（可能不同！）
COMMIT;

REPEATABLE READ / SERIALIZABLE

-- 交易開始時取得快照，
-- 整個交易看到一致的資料
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT * FROM t; -- 快照A
-- 其他交易提交了變更...
SELECT * FROM t; -- 仍是快照A
-- 若更新被其他交易已修改的列：
-- ERROR: could not serialize access
COMMIT;
SERIALIZABLE 會偵測序列化衝突並中止交易。應用程式必須準備好重試被中止的交易
🔬 MVCC 機制

核心原理

多版本UPDATE 不覆蓋原列，而是建立新版本
快照每個交易/語句看到特定時間點的資料
不阻塞讀取不阻塞寫入，寫入不阻塞讀取

列版本控制

xmin建立此版本的交易 ID
xmax刪除/更新此版本的交易 ID
cmin/cmax交易內的命令序號

可見性規則

-- 一個列版本對交易 T 可見，當且僅當：
-- 1. xmin 已提交 且 xmin < T 的快照
-- 2. xmax 未設定 或 xmax 未提交 或 xmax >= T 的快照
VACUUM 負責清理已不被任何交易可見的舊版本（死元組）

鎖等待選項

SELECT ... FOR UPDATE NOWAIT;
-- 無法取鎖時立即拋出錯誤

SELECT ... FOR UPDATE SKIP LOCKED;

💀 死鎖處理 (Ch13.3.4)
-- 死鎖場景：
-- TX1: UPDATE t SET ... WHERE id=1; (等 id=2)
-- TX2: UPDATE t SET ... WHERE id=2; (等 id=1)
-- → deadlock detected!

-- PostgreSQL 自動偵測死鎖（每秒檢查）
-- 會中止其中一個交易

預防死鎖

1以一致的順序取得鎖（如總是先鎖 id 小的）
2縮短交易時間
3使用 NOWAIT 或 SKIP LOCKED
4降低隔離層級

相關參數

deadlock_timeout等待多久開始檢查死鎖（預設 1s）
lock_timeout等待鎖的最大時間（預設 0=無限）
statement_timeout語句最大執行時間
idle_in_transaction_session_timeout閒置交易超時
🎯 諮詢鎖 (Advisory Locks) (Ch13.3.5)
-- 會話級（直到釋放或斷線）
SELECT pg_advisory_lock(12345);
SELECT pg_advisory_unlock(12345);

-- 嘗試取鎖（不等待）
SELECT pg_try_advisory_lock(12345);  -- true/false

-- 交易級（COMMIT/ROLLBACK 時自動釋放）
SELECT pg_advisory_xact_lock(12345);

-- 共享鎖（允許多讀但排斥寫）
SELECT pg_advisory_lock_shared(12345);

-- 雙參數（兩個 int4 → 一個 int8 鍵）
SELECT pg_advisory_lock(1, 100);

應用層互斥防止重複處理同一工作
分散式鎖多進程協調
限速控制並發操作數量
工作佇列搭配 SKIP LOCKED
📊 鎖監控與診斷

查看目前鎖

SELECT l.locktype, l.relation::regclass,
  l.mode, l.granted, l.pid,
  a.usename, a.query, a.state
FROM pg_locks l
JOIN pg_stat_activity a ON l.pid = a.pid
WHERE l.relation IS NOT NULL;

查看鎖等待

SELECT
  blocked.pid AS blocked_pid,
  blocked.query AS blocked_query,
  blocking.pid AS blocking_pid,
  blocking.query AS blocking_query
FROM pg_stat_activity blocked
JOIN pg_locks bl ON bl.pid = blocked.pid
  AND NOT bl.granted
JOIN pg_locks kl ON kl.locktype = bl.locktype
  AND kl.database IS NOT DISTINCT FROM bl.database
  AND kl.relation IS NOT DISTINCT FROM bl.relation
  AND kl.granted
JOIN pg_stat_activity blocking
  ON kl.pid = blocking.pid;

🛡️ Serializable Snapshot Isolation (SSI)

運作機制

基礎REPEATABLE READ + 依賴追蹤
讀寫依賴 (rw)T1 讀了 T2 將修改的資料
偵測當出現「危險結構」（rw-antidependency cycle）時中止交易

使用建議

1將 default_transaction_isolation 設為 serializable
2所有交易都用同層級（混用會失去保證）
3應用層必須準備好重試序列化失敗
4盡量使用短交易
5READ ONLY DEFERRABLE 用於長讀取交易（等待安全快照）

相關參數

SET default_transaction_isolation = 'serializable';
-- max_pred_locks_per_transaction = 64 (預設)
-- max_pred_locks_per_relation = -2 (預設)
-- max_pred_locks_per_page = 2 (預設)
READ ONLY DEFERRABLE 交易在安全時刻開始，保證不會被中止，適合長時間報表查詢
📮 工作佇列模式 (SKIP LOCKED)
-- 工作表
CREATE TABLE job_queue (
  id serial PRIMARY KEY,
  payload jsonb,
  status text DEFAULT 'pending',
  created timestamptz DEFAULT now()
);

-- Worker 取得工作
UPDATE job_queue
SET status = 'processing'
WHERE id = (
  SELECT id FROM job_queue
  WHERE status = 'pending'
  ORDER BY created
  LIMIT 1
  FOR UPDATE SKIP LOCKED
)
RETURNING *;
SKIP LOCKED 讓多個 worker 可以同時從佇列取得不同工作，無需等待
