專案概覽

Superpowers 是由 Jesse Vincent 與 Prime Radiant 團隊打造的開源框架，GitHub 星數超過 108,000，是目前最受歡迎的 AI 編程代理技能框架。它的核心理念不是讓 AI 更聰明，而是讓 AI 更有紀律。

未經約束的 AI 編程代理會跳過測試、在理解需求之前就開始寫程式碼、用猜測來修 bug。Superpowers 透過一套可組合的「技能（Skills）」系統，強制代理在每個步驟都遵循工程最佳實踐。

核心哲學

測試驅動開發（TDD）優先。系統性勝過臨時做法。降低複雜度為首要目標。證據勝過聲稱。驗證後才能宣稱完成。

關鍵效益

效益說明
強制規劃代理必須先腦力激盪並獲得設計批准，才能寫任何程式碼
嚴格 TDDRED-GREEN-REFACTOR 循環強制執行；在測試前寫的程式碼必須刪除
自主長時間工作子代理驅動開發讓代理能自主工作數小時而不偏離計畫
兩階段審查每個任務都經過規格合規 + 程式碼品質雙重審查
極輕量核心框架 < 2K tokens；每個技能載入 < 5K tokens

4
14 個技能詳細機制

4.1 brainstorming（腦力激盪）

在任何創造性工作之前自動觸發。遵循 8 個必須依序完成的步驟：探索專案脈絡、逐一提問、提出 2-3 種方案、分段呈現設計、撰寫設計文件（儲存至 docs/plans/）、派遣規格審查子代理（最多 3 輪）、使用者審閱、轉交 writing-plans。

反模式警告

「這太簡單不需要設計」——即使是待辦清單、單函式工具，都必須經過此流程。設計可以很短，但必須呈現並獲得批准。

4.2 writing-plans（撰寫計畫）

假設執行者完全沒有程式碼脈絡、品味可疑、對測試有抵觸。每個任務遵循嚴格的 TDD 格式：寫失敗測試（附完整程式碼）→ 執行確認失敗 → 寫最少實作程式碼 → 執行確認通過 → Commit。任務粒度為 2-5 分鐘。

4.3 subagent-driven-development（子代理驅動開發）

v5 的預設執行方式。每個任務派遣全新的子代理，子代理永遠不繼承主代理的對話歷史。

階段說明
派遣發送實作者子代理，附帶任務描述和相關脈絡
規格審查派遣規格審查子代理，驗證實作是否符合需求
品質審查派遣程式碼品質審查子代理，檢查標準與架構
迭代任何審查未通過 → 實作者修正 → 重新審查
完成雙重審查通過 → 標記任務完成，進入下一個

4.4 test-driven-development（TDD）

鐵律：沒有失敗的測試，就沒有產品程式碼。 在測試之前寫的程式碼必須徹底刪除，不是「留作參考」。

RED
寫最小的失敗測試
驗證 RED
確認測試正確地失敗
GREEN
寫最少的通過程式碼
REFACTOR
重構，保持測試綠色

4.5 systematic-debugging（系統除錯）

四個必須依序完成的階段。沒有根因調查，就不能嘗試修復。

階段名稱關鍵活動
1根因調查閱讀錯誤訊息、一致性重現、檢查 git diff、追蹤資料流
2模式分析找到正常運作的範例、完整閱讀參考實作、列出每個差異
3假設與測試形成單一假設、做最小改變、一次只改一個變數
4實作用 TDD 建立失敗測試、實作單一修復、驗證
3 次規則

如果嘗試 3 次修復仍失敗——停下來質疑架構，與人類夥伴討論。不要在失敗的修復上疊加更多修復。

4.6 程式碼審查（requesting / receiving）

requesting-code-review：每個任務後強制執行。派遣審查子代理，回饋按嚴重性分類——Critical 阻擋進度、Important 必須修復、Minor 記錄稍後處理。

receiving-code-review：遵循 READ → UNDERSTAND → VERIFY → EVALUATE → RESPOND → IMPLEMENT 六步流程。奉承語句被明確禁止。若審查建議有誤，代理必須反推回去並附技術理由。

4.7 其他技能一覽

協作
using-git-worktrees
建立隔離工作區，確保 .gitignore 安全驗證，自動偵測並執行專案設定（npm/cargo/pip/poetry/go），驗證乾淨測試基線。
完成
finishing-a-development-branch
呈現恰好 4 個選項：本地合併 / 推送並建立 PR / 保留分支 / 捨棄工作。捨棄需輸入「discard」確認。
備用
executing-plans
在不支援子代理的平台上使用。以每批 3 個任務依序執行，每批後報告進度並等待人類回饋。
平行
dispatching-parallel-agents
當存在多個獨立問題時觸發。判斷問題是否獨立、能否平行執行。精確範圍界定是關鍵。
元技能
writing-skills
TDD 應用於流程文件。測試案例 = 壓力情境，產品程式碼 = SKILL.md。Description 必須以「Use when...」開頭。
元技能
verification-before-completion
確保代理在宣稱完成之前確實驗證了修復。using-superpowers 強制執行 1% 規則並反駁各種繞過理由。

Token 效率

掃描所有技能 metadata 約需 100 tokens。載入完整技能約 < 5,000 tokens。核心框架總計 < 2,000 tokens。一個完整的規劃到實作流程約消耗 100K tokens。

八大使用情境
1從零開始建立新專案

告訴代理你想做什麼。brainstorming 自動啟動，一次一個問題釐清需求，提出 2-3 種架構方案。確認後 writing-plans 產出分步計畫，subagent-driven-development 逐一實作。全程 TDD 強制。

2為既有專案新增功能

流程與新專案類似，但 brainstorming 會先檢查現有檔案、文件和近期 commit 來理解脈絡。計畫中的任務標注確切的檔案路徑和行號範圍，確保與現有架構相容。

3除錯與修復 Bug

systematic-debugging 自動啟動四階段流程。代理被禁止嘗試修復直到完成根因調查。TDD 先寫重現 bug 的失敗測試，再實作最小修復。3 次失敗後停下來質疑架構。

4程式碼重構

TDD 強制：先確保現有測試全部通過（基準線），每次重構改動都是原子性的，改完立即驗證。代理不會在重構中「順便」加新功能（YAGNI 原則）。

5大型遷移（Migration）

brainstorming 識別多子系統範圍並協助拆分為子專案。每個子專案有自己的設計與計畫。dispatching-parallel-agents 可同時處理獨立子系統。Jesse Vincent 用此重寫 chardet 7.0.0，達成 41 倍速度提升。

6多人協作

專案級技能在 .claude/skills/ 中共享。程式碼審查確保品質一致。Git worktrees 讓多人同時在不同工作區工作。finishing-a-development-branch 標準化 PR 建立流程。

7自定義技能建立

使用 writing-skills 技能，遵循 TDD。先建立壓力情境（沒有技能時的失敗），記錄失敗模式，撰寫 SKILL.md 解決問題，再透過更多壓力測試堵住漏洞。個人技能存放在 ~/.claude/skills/。

8長時間自主開發

subagent-driven-development 專為此場景設計。主代理維持高層脈絡，每個子代理拿到精確的任務描述。兩階段自動審查確保品質不隨時間下降。建議用 Haiku 執行有詳細計畫的實作任務以節省 token。

社群評價與最佳實踐

正面評價

使用者最常讚揚的是強制規劃階段：「最大的改變是它主動問你問題，而不是把負擔放在你身上。」子代理驅動開發被描述為「數量級層次的功能」。有使用者聲稱個人產出超過了之前整個團隊的產量。

批評與限制

問題細節
規劃開銷修正錯字、更新依賴等小任務，brainstorm + plan 過於繁重
嚴格 TDD極度固執己見；偏好不同工作流程的使用者需大量自定義
Token 消耗5 個子代理各 50K+ tokens，在付費方案上成本可觀
平台成熟度Claude Code 整合最佳；Codex 和 OpenCode 仍在成熟中
脈絡管理缺乏內建指引管理 80K+ token 的早期對話脈絡

最佳實踐彙整

1. 簡單任務（錯字修正、依賴更新）不需觸發完整管線

2. 親自審閱設計文件和計畫，不要盲目批准

3. 在 brainstorming 階段主動提供測試策略資訊

4. 超過 48K 字元的計畫考慮拆分，避免脈絡窗口問題

5. 用 CLAUDE.md 覆蓋不喜歡的 Superpowers 預設行為

6. 模型分配：Opus 做 brainstorming/planning，Haiku 做實作任務

7. 推薦流程：/brainstorm → /write-plan → 手動編輯計畫 → /execute-plan

8. 長時間自主任務保持定期檢查，避免代理進入「YOLO 模式」

8
結論

Superpowers 代表了 AI 輔助開發的範式轉移。它的核心洞見是：AI 編程代理的最大問題不是不夠聰明，而是不夠有紀律。 透過強制執行 brainstorm → plan → implement → review 的管線、嚴格的 TDD、系統性除錯和自動程式碼審查，它將 AI 代理從衝動的初級工程師轉變為有紀律的開發團隊。

但這不是萬能藥。對於複雜、多步驟的專案，它的價值毋庸置疑；對於快速小修補，它的開銷過於沉重。最聰明的用法是發展出何時啟用、何時關閉的判斷力。
