OpenSpec 入門・核心概念・安裝

涵蓋 getting-started.md · concepts.md · installation.md

安裝方式（installation.md）

系統需求

Node.js 20.19.0 以上。驗證版本：node --version

npm（推薦）

npm install -g @fission-ai/openspec@latest

pnpm

pnpm add -g @fission-ai/openspec@latest

yarn

yarn global add @fission-ai/openspec@latest

bun

bun add -g @fission-ai/openspec@latest

Nix 安裝方式

# 直接執行（不安裝）
nix run github:Fission-AI/OpenSpec -- init

# 安裝到 profile
nix profile install github:Fission-AI/OpenSpec

# flake.nix 整合
{
  inputs.openspec.url = "github:Fission-AI/OpenSpec";
  outputs = { nixpkgs, openspec, ... }: {
    devShells.x86_64-linux.default =
      nixpkgs.legacyPackages.x86_64-linux.mkShell {
        buildInputs = [ openspec.packages.x86_64-linux.default ];
      };
  };
}

驗證與升級

openspec --version                           # 確認安裝
npm install -g @fission-ai/openspec@latest   # 升級
openspec update                              # 升級後在專案中執行
升級後務必在每個專案中執行 openspec update，重新產生 AI 指令檔。
核心概念（concepts.md）

四大設計哲學

原則說明
流動而非僵化沒有階段閘門，按需求順序工作，隨時可回頭修改任何文件
迭代而非瀑布需求會改變、理解會加深，OpenSpec 擁抱這個現實
簡單而非複雜秒級初始化，最少儀式感，只在需要時才自訂
Brownfield 優先專為既有程式碼庫設計，Delta 機制讓修改成為一等公民

雙目錄架構

openspec/
├── specs/          ← 真實來源（Source of Truth）：系統目前行為
│   └── <domain>/
│       └── spec.md
└── changes/        ← 提議修改：每個變更獨立一個資料夾
    └── <change-name>/
        ├── proposal.md    提案：為什麼要做
        ├── specs/         Delta specs：要改什麼
        ├── design.md      設計：怎麼做
        └── tasks.md       任務：實作清單
specs/ 是系統行為的單一真實來源，按領域組織（如 auth/、payments/）。
changes/ 中的 delta specs 歸檔時會自動合併回 specs/。

Delta Specs — Brownfield 的關鍵機制

操作Markdown 標記歸檔時的行為
新增## ADDED Requirements附加到主規格末尾
修改## MODIFIED Requirements依標題替換對應需求
移除## REMOVED Requirements從主規格中刪除
重新命名## RENAMED Requirements修改章節標題，保留內容

歸檔套用順序：RENAMED → REMOVED → MODIFIED → ADDED

Delta Specs 範例格式

# Delta for Auth

## ADDED Requirements
### Requirement: Two-Factor Authentication
The system MUST support TOTP-based 2FA.
#### Scenario: OTP required
- GIVEN a user with 2FA enabled
- WHEN valid credentials submitted
- THEN an OTP challenge is presented

## MODIFIED Requirements
### Requirement: Session Timeout
The system SHALL expire sessions after 30 min.
(Previously: 60 minutes)

## REMOVED Requirements
### Requirement: Remember Me
(Deprecated in favor of 2FA)

Schema 與 Artifact 依賴圖

proposal (root)           ← 無依賴，最先建立
    │
    ├──► specs             ← 需要 proposal
    │
    ├──► design            ← 需要 proposal（可與 specs 平行）
    │
    └──► tasks             ← 需要 specs + design
              │
              └──► apply   ← 需要 tasks

依賴是啟用器（enabler），不是閘門（gate）。你可以隨時回頭更新任何已完成的文件。

Artifact 三種狀態

狀態符號條件
BLOCKED○必要依賴尚未完成
READY◆依賴已存在，文件尚未建立
DONE✓文件已存在於檔案系統

Spec 層級

類型適用場景
Lite spec（預設）行為優先的簡短需求，明確範圍
Full spec跨團隊/repo、API 變更、安全性
快速測試：如果實作可以改變但外部可見行為不變，就不需要寫進 spec。

變更生命週期

Explore→ Create→ Implement→ Verify→ Archive

每次歸檔都將 delta specs 合併回主規格，specs/ 隨開發自然成長。Archive 目錄保存完整上下文（proposal、design、tasks）作為審計歷史。

快速上手（getting-started.md）

初始化專案

cd your-project
openspec init

初始化精靈會引導你選擇 AI 工具（如 Claude Code、Cursor），並根據選擇產生對應的設定檔。OpenSpec 會自動偵測既有的工具目錄（.claude/、.cursor/ 等）。

初始化後的專案結構

your-project/
├── openspec/
│   ├── specs/              # 真實來源（初始為空）
│   ├── changes/            # 變更提案
│   └── config.yaml         # 專案設定
├── .claude/skills/         # Claude Code 技能（如有選擇）
└── src/                    # 你的既有程式碼（不受影響）

完整工作流程範例：新增暗色模式

步驟 1 — 提案與規劃

你：/opsx:propose add-dark-mode
AI：Created openspec/changes/add-dark-mode/
    ✓ proposal.md — 為什麼要做、改什麼
    ✓ specs/      — 需求與情境
    ✓ design.md   — 技術方案
    ✓ tasks.md    — 實作清單
    Ready for implementation!

步驟 2 — 實作

你：/opsx:apply
AI：Implementing tasks...
    ✓ 1.1 Add theme context provider
    ✓ 1.2 Create toggle component
    ✓ 2.1 Add CSS variables
    ✓ 2.2 Wire up localStorage
    All tasks complete!

步驟 3 — 歸檔

你：/opsx:archive
AI：Archiving add-dark-mode...
    ✓ Merged specs into openspec/specs/ui/spec.md
    ✓ Moved to openspec/changes/archive/2025-01-24-add-dark-mode/
    Done!

Expanded 工作流（逐步模式）

如果啟用了擴展 profile，可以用更細緻的步驟控制：

/opsx:new→ /opsx:continue→ ×N /opsx:apply→ /opsx:verify→ /opsx:archive
Brownfield 專案建議使用 /opsx:continue 逐步產生文件，讓你能在每個環節審查 AI 是否正確理解既有架構。

Profile 配置

Profile包含指令適用場景
core（預設）propose, explore, apply, archive一般使用、快速開發
custom可自選 11 個 workflow 的任意子集需要細粒度控制的團隊
openspec config profile    # 互動式切換
openspec update            # 套用到專案

驗證指令

openspec list                     # 列出進行中的變更
openspec show add-dark-mode       # 查看變更詳情
openspec validate add-dark-mode   # 驗證 spec 格式
openspec view                     # 互動式儀表板
