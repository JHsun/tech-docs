自訂設定（customization.md）

三層自訂機制

層級內容適用場景
專案設定 config.yamlContext、Rules、預設 Schema大多數團隊
自訂 Schema定義自己的 artifact 流程有特殊流程的團隊
全域覆蓋跨專案共用 schema進階使用者

config.yaml 完整結構

# openspec/config.yaml
schema: spec-driven          # 新變更的預設 schema

context: |                   # 注入到所有 artifact 指令中（上限 50KB）
  Tech stack: TypeScript, React, Node.js, PostgreSQL
  API style: RESTful, documented in docs/api.md
  Testing: Jest + React Testing Library
  We value backwards compatibility for all public APIs

rules:                       # 附加到對應 artifact 的指令中
  proposal:
    - Include rollback plan
    - Identify affected teams
  specs:
    - Use Given/When/Then format
    - Reference existing patterns before inventing new ones
  design:
    - Include sequence diagrams for complex flows
  tasks:
    - Each task completable in under 2 hours
    - Always include a task for adding/updating tests

AI 實際接收的指令結構

<context>
Tech stack: TypeScript, React, Node.js, PostgreSQL
API style: RESTful...
</context>
<rules>
- Include rollback plan
- Identify affected teams
</rules>
<template>
[Schema 的內建模板內容]
</template>
context 是被動的 project.md 的進化版——它會被主動注入到每個 OpenSpec 規劃請求中，可靠性更高。

自訂 Schema 目錄結構

openspec/schemas/my-workflow/
├── schema.yaml          # 工作流定義
└── templates/
    ├── proposal.md      # 提案模板
    ├── spec.md          # 規格模板
    ├── design.md        # 設計模板
    └── tasks.md         # 任務模板

schema.yaml 完整定義

name: my-workflow
version: 1
description: 團隊自訂工作流

artifacts:
  - id: proposal
    generates: proposal.md
    description: 初始提案
    template: proposal.md
    instruction: |
      建立提案說明「為什麼」需要此變更。
      聚焦問題，而非解決方案。
    requires: []             # 無依賴

  - id: design
    generates: design.md
    description: 技術設計
    template: design.md
    requires: [proposal]     # 依賴 proposal

  - id: tasks
    generates: tasks.md
    description: 實作清單
    template: tasks.md
    requires: [design]       # 依賴 design

apply:
  requires: [tasks]          # apply 需要 tasks 完成
  tracks: tasks.md           # 追蹤此檔的勾選框

快速建立自訂 Schema

# 從內建 schema 複製修改（推薦）
openspec schema fork spec-driven my-workflow

# 從零建立（互動式）
openspec schema init research-first

# 從零建立（非互動式）
openspec schema init rapid \
  --description "快速迭代工作流" \
  --artifacts "proposal,tasks" \
  --default

Schema 解析優先順序

優先序來源路徑
1（最高）專案本地openspec/schemas/<name>/
2使用者全域~/.local/share/openspec/schemas/<name>/
3（最低）套件內建npm 套件中的 schemas/
openspec schema which my-workflow      # 查看解析來源
openspec schema which --all            # 列出全部 schema 及來源

多語言支援（multi-language.md）

設定方式

在 config.yaml 的 context 欄位加入語言指令即可：

schema: spec-driven
context: |
  語言：繁體中文
  所有產出文件必須使用繁體中文撰寫。
  # 以下繼續寫其他專案設定...
  Tech stack: Python 3.11, FastMCP

各語言設定範例

繁體中文

context: |
  語言：繁體中文
  所有產出文件必須使用繁體中文撰寫。

簡體中文

context: |
  语言：中文（简体）
  所有产出物必须用简体中文撰写。

日文

context: |
  言語：日本語
  すべての成果物は日本語で作成してください。

西班牙文

context: |
  Idioma: Español
  Todos los artefactos deben escribirse en español.

法文

context: |
  Langue : Français
  Tous les artefacts doivent être rédigés en français.

德文

context: |
  Sprache: Deutsch
  Alle Artefakte müssen auf Deutsch verfasst werden.

混合語言模式

context: |
  語言：繁體中文
  使用繁體中文撰寫，但：
  - 技術術語如 "API"、"REST"、"GraphQL" 保持英文
  - 程式碼範例和檔案路徑保持英文
  - Given/When/Then 關鍵字保持英文

驗證語言設定

openspec instructions proposal --change my-change
# 輸出中應包含你的語言設定
OPSX 工作流架構（opsx.md）

Legacy vs OPSX 對比

面向Legacy（舊版）OPSX（新版）
指令/openspec:proposal 等/opsx:propose 等 + 擴展指令
文件產生全部一次建立可逐步或一次建立
回頭修改階段閘門，不便回頭隨時更新任何文件
自訂性固定結構Schema 驅動，完全可自訂
設定方式CLAUDE.md 標記 + project.md統一的 config.yaml
指令模板硬寫在 TypeScript 中外部 YAML schema + Markdown 模板

OPSX 架構：Artifact Graph Engine

Schema 定義 (YAML)
    ↓
Artifact Graph Engine
  • 拓撲排序（依賴排序）
  • 狀態偵測（檔案系統存在性）
  • 豐富的指令產生（template + context + rules）
    ↓
Skill 檔案（.md + YAML frontmatter）
    ↓
AI 助手讀取並執行

動態指令產生

AI 助手在執行時查詢即時狀態，而非硬編碼的提示：

CLI 指令AI 得到的資訊
openspec status --json下一步可以做什麼
openspec instructions --json怎麼做（完整指令）
openspec templates解析後的模板路徑

指令組裝順序：Context → Artifact-specific Rules → Template Content

OPSX 實用提示

場景建議
先想清楚再動手用 /opsx:explore
知道要做什麼用 /opsx:ff 一次產生
不確定方向用 /opsx:continue 逐步審查
實作中發現問題直接修改 artifact 檔案，再繼續 /opsx:apply
隨時查看進度openspec status --change "name"

全域設定結構

存放於 ~/.config/openspec/config.json

欄位值說明
profilecore / custom工作流選擇模式
deliveryskills / commands / both產生的檔案類型
workflows字串陣列custom 時的 workflow 清單

遙測（Telemetry）

僅收集指令名稱和版本號。不收集引數、路徑、內容或個人資訊。CI 中自動停用。

# 停用方式
export OPENSPEC_TELEMETRY=0
# 或
export DO_NOT_TRACK=1
