Part VI · 生態系 · Chapter 15

systemd 整合

在現代 Linux 系統，systemd 是 cgroup v2 的主要管理者——PID 1 直接管 root cgroup，所有 service 都自動進 cgroup tree。 這不只是方便：v2 嚴格的 single-writer rule 讓「誰管理 cgroup」成為架構問題，systemd 提供了標準答案。 本章拆解 slice / scope / service 三角色的職責、resource control directives 的完整參考、 Delegate 機制（為什麼只能用在 service 與 scope，不能用在 slice）、drop-in files 的優雅修改方式， 以及生產環境每天會用到的 systemctl set-property、systemd-run、systemd-cgls 三大工具。

01 · Highlights

本章重點

  1. systemd 是 cgroup v2 的 single writer：PID 1 管 root cgroup，所有 service / scope 自動納入。不要 自己 mkdir 在 root——會違反 single-writer rule、被 systemd 對抗。
  2. 三種 unit 類型 = 三種 cgroup 角色：slice（inner node，組織用）、service（leaf，systemd 啟動的 process）、scope（leaf，外部啟動但 systemd 接管的 process，如 user session、container）。
  3. Delegate= 只能用在 leaf（service / scope），不能用在 slice：因為 slice 是 inner node，systemd 與你會搶著管 child，違反 single-writer。
  4. Resource control directives 完整覆蓋所有 controller：CPUWeight、CPUQuota、MemoryMin/Low/High/Max、IOWeight、IOMax、AllowedCPUs、TasksMax 等。不需 直接寫 cgroup 介面檔案。
  5. 三層修改方式優雅遞進：unit file（永久）→ drop-in file（永久但隔離）→ systemctl set-property（runtime + 持久化）。systemd-run 是 transient unit 的快速路徑，debug 必備。
02 · Latest

最新動態（2025–2026）

2025 systemd v258 移除 force cgroup v1 boot parameter。systemd.unified_cgroup_hierarchy=0 與 SYSTEMD_CGROUP_ENABLE_LEGACY_FORCE=1 都被棄用，systemd 完全綁定 cgroup v2。
2025 systemd 256+ 完整支援 partition。AllowedCPUs= 配合新的 partition 控制 directive，systemd unit 可直接設定 partition 角色（root / isolated）。
2025 systemd-oomd 普及到所有主流 distro。Fedora、Ubuntu、openSUSE 都預設啟用，user.slice 預設 ManagedOOMMemoryPressure=kill。
2024–2025 BPFProgram= directive 成熟（systemd 256+）。可以在 unit 直接 attach BPF program 到 cgroup，BPFProgram=type:program-path 格式。
2025 新 directive：DelegateSubgroup=（systemd 256+）。把 service 的主 process 放到 delegated subtree 內的特定 subcgroup，方便 sd_notify 等 IPC。

2026 年的時點，systemd 與 cgroup v2 的整合已是 Linux distribution 的標配——RHEL 10、Fedora、Ubuntu、Arch、Debian 12+ 都默認 v2 + systemd 全管理。 本章重點不只是「directive 怎麼寫」，更包含「為什麼這樣設計」、「什麼時候用 unit file vs drop-in vs set-property」這類設計決策。

03 · Core Concepts

核心內容

3.1 為什麼 systemd 要管 cgroup

前面 Ch.5 講過 v2 的 single-writer rule：每個 cgroup 只能有一個 manager。 在 systemd 系統，這個 manager 是 PID 1（systemd 本身）。

為什麼必須這樣？因為 cgroup 樹的結構與 service 生命週期、依賴關係緊密相關——service 啟動時要進對的 cgroup，停止時 cgroup 要清理，limit 變更要同步介面檔案。 沒有 統一管理者，會出現多個 process 同時 mkdir / rmdir、寫 controller 設定，造成 race condition、記憶體洩漏、limit 不同步等問題。

v1 容忍多 manager（multi-hierarchy）已經很亂；v2 直接強制 single writer。在 systemd 系統，這個 writer 就是 systemd。

3.2 預設的 systemd cgroup 樹

圖 15.1 · systemd 預設 cgroup tree（簡化版）

三個頂層 slice 對應 Linux 系統的三大 process 來源：

  • system.slice：systemd 啟動的系統 daemon——nginx、postgres、ssh、cron 等。
  • user.slice：使用者登入 session——透過 PAM 自動建立 user-UID.slice，內含 session-N.scope（terminal session）與 user@UID.service（user systemd instance）。
  • machine.slice：VM 與 container——docker-XXX.scope、qemu-XXX.scope、systemd-nspawn 容器等。

3.3 三種 unit 類型的 cgroup 角色

Unit 類型cgroup 角色定義典型例子
.slice inner node（無 process） 邏輯分組，不直接執行任何東西 system.slice、user-1000.slice、kubepods.slice
.service leaf（有 process） 由 systemd 啟動的 process（ExecStart=） nginx.service、postgresql.service
.scope leaf（有 process） 外部啟動但 systemd 接管的 process session-2.scope、docker-XXX.scope

service vs scope 差別在於「誰啟動 process」：

  • service：systemd 自己 fork+exec（透過 ExecStart=）。生命週期由 systemd 完整掌控。
  • scope：外部 process 已經跑起來，systemd 透過 D-Bus API 「接管」它——把 PID 放進新建的 cgroup、套用 resource control。

典型 scope 場景：

  • 使用者 ssh 登入：sshd 建立 session，systemd 接管成 session-N.scope。
  • docker run：dockerd 啟動 container process，告訴 systemd 接管成 docker-XXX.scope。
  • qemu-system 啟動 VM：libvirtd 接管成 machine-vm.scope。

3.5 Delegate=：把 cgroup 子樹的管理權交出去

前面 Ch.5 詳細講過 delegation。systemd 的視角：

  • 沒設 Delegate=：systemd 完全管理這個 unit 的 cgroup。process 在它指定的位置，不能自己 mkdir 子 cgroup。
  • 設 Delegate=yes：systemd 在這個 unit 之下不再插手，service 自己可以 mkdir 子 cgroup、設介面、移 process。
  • 設 Delegate=memory pids：只把指定 controllers 的管理權給 service，其他仍由 systemd 管。

關鍵限制：Delegate 只能用在 leaf（service / scope）

kernel 的 single-writer rule + cgroup v2 的 inner/leaf 規則導致一個重要結論：

Don't attempt to set Delegate= in slice units, and in particular not in -.slice. It's not supported, and will generate an error. — systemd 官方文件

為什麼？slice 是inner node，systemd 會在它之下放 service / scope。如果允許在 slice 上 Delegate，service 啟動時 systemd 要 mkdir 子 cgroup、Delegate 對象也要 mkdir——兩個 writer 衝突。

正確做法： 

# ✗ 錯：在 slice 設 Delegate
[Slice]
Delegate=yes

# ✓ 對：在 service / scope 設 Delegate
[Service]
Delegate=yes
ExecStart=/usr/local/bin/my-container-runtime

# 這個 service 跑起來後，可以在它的 cgroup 之下自由 mkdir 子 cgroup
# /sys/fs/cgroup/system.slice/my-container-runtime.service/...

Delegate 的擴展屬性訊號

systemd 251+ 在被 delegated 的 cgroup 目錄上設定 xattr user.delegate=1，service 可以用 getxattr() 確認自己是否在 delegated subtree： 

# 確認自己是否被 delegated
getfattr -n user.delegate /sys/fs/cgroup/system.slice/my-container.service
# user.delegate="1"  ← 是 delegated

# 從程式內讀
getxattr("/sys/fs/cgroup/system.slice/my-container.service",
         "user.delegate", buf, sizeof(buf));

3.6 三層修改方式

要修改 unit 的 resource control，systemd 提供三種方式，各有優缺：

方式 1：直接編輯 unit file

# 直接編輯 /etc/systemd/system/myapp.service（或 /lib/systemd/system/）
[Service]
ExecStart=/usr/bin/myapp
MemoryMax=2G
CPUQuota=100%

# 套用變更
sudo systemctl daemon-reload
sudo systemctl restart myapp

優：直觀，所有設定在一個檔案。 缺：直接改 distro 提供的 unit file 會被 package update 覆蓋；改 /lib/systemd/system/ 是嚴重錯誤。

方式 2：drop-in file（推薦）

# 建立 drop-in 目錄
sudo mkdir -p /etc/systemd/system/myapp.service.d/

# 加 override
cat <

優：原 unit file 不動，drop-in 之間獨立可分開維護（10-resources.conf、20-network.conf），lexicographic order 套用。distro update 不會碰你的 drop-in。 缺：需要 daemon-reload + restart。

更快的編輯方式：systemctl edit myapp.service 會自動建立 drop-in 目錄並打開編輯器。

方式 3：systemctl set-property（runtime + 永久）

# Runtime + 永久（寫入 /etc/systemd/system.control/）
sudo systemctl set-property myapp.service MemoryMax=2G CPUQuota=100%

# 只 runtime（reboot 後消失）
sudo systemctl set-property --runtime myapp.service MemoryMax=2G

# 看當前值
systemctl show -p MemoryMax myapp.service

優：不需 restart service，runtime 即時生效。比起 drop-in 更快。 缺：寫進 /etc/systemd/system.control/ 而非標準 drop-in 路徑，新 sysadmin 可能找不到——這是個常見「為什麼設定改了但找不到」的坑。

3.7 systemd-run：transient unit

systemd-run 建立 transient unit——臨時 unit，不寫到 disk，process 結束後自動消失。debug 與一次性測試的最快路徑： 

# 跑一個有限制的命令
sudo systemd-run --slice=user.slice --unit=test \
    -p MemoryMax=500M \
    -p CPUQuota=50% \
    --pty \
    bash

# 跑完即清理（unit 自動刪除）
exit

# 啟動 service 風格的 transient unit（不阻塞 terminal）
sudo systemd-run --unit=bg-task \
    -p MemoryMax=1G \
    /usr/local/bin/long-running-task

# 之後可以 systemctl stop bg-task

常用 systemd-run 場景：

  • 測試新 limit 的影響：跑 transient 看 PSI、events，確認後再寫 drop-in。
  • 單次 batch job：一次性的資料處理、backup，不需要寫 unit file。
  • cgroup 隔離臨時 process：對某個 ad-hoc command（如 cron 沒設限的腳本）臨時加 limit。

3.8 systemd-cgls 與 systemd-cgtop

看 cgroup tree 與用量的官方工具： 

# 看 cgroup tree（整個系統）
systemd-cgls

# 看特定 slice
systemd-cgls /system.slice

# 即時資源監控（top-like）
systemd-cgtop

# 按 IO 排序
systemd-cgtop --order=io

# 看單一 service 完整資訊（包含 cgroup path、PID 樹）
systemctl status myapp.service

systemctl show：看 unit 所有屬性的關鍵 工具： 

# 看所有資源相關設定
systemctl show -p MemoryMax,MemoryHigh,CPUQuota,CPUWeight myapp.service

# 看 ControlGroup path
systemctl show -p ControlGroup myapp.service

# 看完整 unit 屬性
systemctl show myapp.service

3.9 BPF 整合：BPFProgram=

systemd 256+ 的 BPFProgram= directive 直接在 unit attach BPF program 到 cgroup： 

[Service]
ExecStart=/usr/local/bin/myapp

# Attach BPF egress filter
BPFProgram=egress:/usr/local/lib/bpf/my-egress.bpf.o

# Attach BPF device filter
BPFProgram=device:/usr/local/lib/bpf/my-device.bpf.o

# 也可以用 IPEgressFilterPath= / IPIngressFilterPath= 等具體 directive
IPEgressFilterPath=/usr/local/lib/bpf/firewall.bpf.o

systemd 自動處理 BPF program load + attach + 生命週期（service 停止時自動 detach）。

3.10 Best practices

  • 用 drop-in 不直接改 unit file：distro update-friendly、可拆解管理。
  • set-property 寫了要記得：留 git log / 寫進 ansible playbook。runtime 改完忘了，下個 sysadmin 看不到。
  • 對 system.slice / user.slice 設保護下限：避免 user 的 batch job 把系統 daemon 餓死。
  • 用 ManagedOOMMemoryPressure 取代手寫 oomd policy：systemd-oomd 整合度好。
  • Delegate 給 container runtime / nested systemd：自己起的 daemon 通常不需要 Delegate。
  • Slice= 用來分組：把性質相近的 service 放同 slice（如 db.slice 集中所有 DB），方便整批設限。

05 · Pitfalls

常見陷阱

陷阱 1 · 直接編輯 /lib/systemd/system/ 的 unit file

編輯 distro 提供的 unit file（在 /lib/systemd/system/ 或 /usr/lib/systemd/system/）——下次 package update 會被覆蓋，所有自訂消失。 修法：永遠用 systemctl edit 建 drop-in（在 /etc/systemd/system/）。/etc 路徑才是 user-controlled 區。

陷阱 2 · 在 .slice 設 Delegate=yes

想對整個 user.slice 啟用 delegation——systemd 會回 error。slice 是 inner node，delegation 違反 single-writer。 修法：對 service / scope 設 Delegate，不對 slice。如果需要對 slice 之下整批 delegated subtree，把它們各自設 Delegate。

陷阱 3 · 自己 mkdir cgroup 在 root 之下

想「繞過 systemd 自己建 cgroup」：mkdir /sys/fs/cgroup/myapp——這違反 single-writer rule。systemd 會在你不注意時清掉你的 cgroup，或是設定衝突。 修法：只在 Delegate=yes 的子樹之下建 cgroup。需要自由 cgroup 管理就用 systemd-run + Delegate=yes。

陷阱 4 · daemon-reload 後忘了 restart service

改了 unit file / drop-in，daemon-reload 後設定看似套用——但 service 還在用舊設定跑。 原因：daemon-reload 只是讓 systemd 重讀 unit 定義，已啟動的 service 不會自動重啟。 修法：設定改完要 systemctl restart。或用 systemctl set-property（runtime 套用不需 restart）。

陷阱 5 · set-property 設定後找不到

新 sysadmin 接手系統，看 /etc/systemd/system/myapp.service.d/ 是空的，但 systemctl show 顯示有 MemoryMax=2G 設定——「從哪來的？」 原因：上一個 admin 用 systemctl set-property 設的，存在 /etc/systemd/system.control/（不是標準 drop-in 路徑）。 修法：find /etc/systemd -name "*.conf" 兩個目錄都看。長期 production 設定建議用 drop-in（明顯位置）。

陷阱 6 · CPUQuota 寫成 cpu mask

錯誤：CPUQuota=200（沒 % 號）——systemd parse 失敗或當 200% (2 cores)。 錯誤：AllowedCPUs=0xff（hex mask）——systemd 不接受 mask，只接受 CPU list。 修法：CPUQuota=200%、AllowedCPUs=0-7（用 dash）或 AllowedCPUs=0,1,2（用逗號）。

陷阱 7 · 期待 Slice= 變更立即生效

對 running service 改 Slice=newslice.slice——daemon-reload 後設定看似 改了，但 process 還在舊 slice。 原因：systemd 不會把 running process 從一個 slice 搬到另一個（避免破壞 cgroup state）。 修法：改 Slice 後一定要 restart service 才能搬家。

陷阱 8 · transient unit ID 每次重啟都變

用 systemd-run 啟動 service 時記住的 cgroup path，下次跑時 path 變了——transient unit 每次都是新的。 修法：debug 時即時取 path（不要寫死）。production 用 persistent unit（unit file），不用 transient。

06 · Exercises

動手練習

練習 1 · 寫完整的 production unit file（15 分鐘）

對假設的 web service 寫完整 unit file，包含：

  1. 放在 web.slice（自訂 slice）
  2. memory: min=512M、high=1.5G、max=2G、swap禁用
  3. cpu: weight=200、quota=300%（3 核）
  4. io: weight=200、寫 bandwidth 上限 100M
  5. tasks max=500
  6. OOM 時整個 unit 一起死（kill policy）
  7. 啟用 systemd-oomd 監控（pressure > 50% 殺）

用 unit file + drop-in 兩種寫法各寫一次。

練習 2 · 探索 systemd-oomd 預設行為（10 分鐘）

for unit in $(systemctl list-units --type=service --state=active --no-legend --plain | awk '{print $1}'); do
    # 取所有重要 limit
    mem_max=$(systemctl show -p MemoryMax --value "$unit")
    mem_high=$(systemctl show -p MemoryHigh --value "$unit")
    cpu_quota=$(systemctl show -p CPUQuotaPerSecUSec --value "$unit")
    tasks_max=$(systemctl show -p TasksMax --value "$unit")
    
    # 判斷是否 unbounded
    issues=""
    [ "$mem_max" = "infinity" ] && issues+=" mem_unbounded"
    [ "$cpu_quota" = "infinity" ] && issues+=" cpu_unbounded"
    [ "$tasks_max" = "infinity" ] && issues+=" tasks_unbounded"
    
    [ -n "$issues" ] && echo "$unit:$issues"
done

分析：你的系統有多少 service 是 unbounded？哪些應該設限？

自我檢查題

Q1.systemd 的三種 unit 類型（slice、service、scope）對應的 cgroup 角色是什麼？什麼時候用哪個？
對應關係：
  • .slice：cgroup inner node。沒有 process，只是邏輯分組——把 service / scope / 子 slice 組織起來。例：system.slice、user-1000.slice。
  • .service：cgroup leaf。由 systemd 啟動的 process，透過 ExecStart=。例：nginx.service、postgresql.service。
  • .scope：cgroup leaf。外部啟動但 systemd 接管的 process。例：session-2.scope（user 登入）、docker-XXX.scope（docker 啟動 container）。
選擇邏輯：
  • 「我寫一個 daemon 要長期跑」→ service（systemd 自己 fork+exec）
  • 「我有一個 process 想讓 systemd 接管」→ scope（透過 D-Bus API）
  • 「我要組織一群相關的 service」→ 自訂 slice
cgroup 規則對應：v2 的 inner/leaf 規則直接對應——slice 是 inner（不放 process），service / scope 是 leaf（放 process）。
Q2.為什麼 Delegate= 不能用在 slice？
核心原因：違反 single-writer rule。
  1. slice 是 inner node，systemd 會在它之下放 service / scope（systemd 是 writer）。
  2. 若 Delegate slice，外部 service 也想在它之下放 cgroup（service 是 writer）。
  3. 兩個 writer 同時 mkdir / rmdir / 寫設定 → race condition、衝突、不可預測行為。
正確做法：在 leaf（service / scope）設 Delegate=yes，把這個 leaf 連同其下整個子樹 交給應用管。應用可以在 leaf 之下自由建 cgroup（包括子 slice / service 等概念，但是它自己管）。 實作層：systemd 245+ 在 Delegate=yes 的 cgroup 設 xattr user.delegate=1，service 可以用 getxattr() 確認自己是否被 delegated。 典型場景：container runtime（runc / crun）需要在它的 cgroup 之下管理多個 container 的 cgroup——所以 docker.service / podman.service 都會 Delegate=yes。
Q3.三種修改 unit resource 的方式（unit file 直接改、drop-in、set-property）各自優缺？什麼時候用哪個？
  • 直接改 unit file（/etc/systemd/system/myapp.service）：
    • 優：簡單，所有設定一個檔案。
    • 缺：原來在 /lib/ 的 distro 提供 unit 不能直接改（會被覆蓋）。在 /etc/ 的可以但結構亂。
    • 場景：你自己寫的 service。
  • drop-in file（/etc/systemd/system/myapp.service.d/10-resources.conf）：
    • 優：原 unit file 不動，distro update friendly。多個 drop-in 可拆解管理。
    • 缺：需要 daemon-reload + restart 套用。
    • 場景：絕大多數情況。永久性 resource control 最佳選擇。
  • systemctl set-property：
    • 優：runtime 即時生效，不需 restart。寫入 system.control/ 持久化。
    • 缺：寫到非標準路徑（system.control/），新 admin 找不到。
    • 場景：緊急調整不能 restart 的 production service。記得寫進 runbook / ansible。
建議：
  • 長期設定 → drop-in（systemctl edit）
  • 緊急調整 → set-property（事後寫進 drop-in 持久化）
  • 測試新限制 → systemd-run（transient unit）
Q4.resource control directives 與 cgroup 介面檔案是 1:1 對應嗎？舉幾個 directive 與其對應的 cgroup file。
大致 1:1 但有細節：
  • 直接對應：
    • CPUWeight= ↔ cpu.weight
    • MemoryMax= ↔ memory.max
    • IOWeight= ↔ io.weight
    • TasksMax= ↔ pids.max
    • AllowedCPUs= ↔ cpuset.cpus
  • 單位轉換：
    • CPUQuota=50% → cpu.max 寫成 50000 100000（quota period 表示）
    • CPUQuotaPeriodSec= 配合 CPUQuota 算出實際 cpu.max 值
  • Composite directive：
    • IODeviceWeight=/dev/sda 200 對應 io.weight 中 device-keyed 的部分（lookup major:minor 自動填）
    • OOMPolicy=kill 對應 memory.oom.group=1（行為而非數值）
  • BPF compose：
    • DeviceAllow=/dev/null rw 不對應 cgroup file，systemd 編譯成 BPF program attach 上去
    • BPFProgram=type:path 直接 attach 既有 BPF object
查對應關係：man systemd.resource-control 每個 directive 都標明對應的 cgroup attribute。
Q5.systemd-run 的核心用途？跟 unit file 比有何優勢與限制？
systemd-run 核心用途：建立 transient unit——臨時 unit，不寫到 disk，process 結束後自動清理。 vs unit file 比較：
  • 優勢：
    • 不需要寫 unit file、daemon-reload
    • 命令列直接設 resource control（-p 開關）
    • process 結束自動清理，不留垃圾
    • debug / 一次性 batch / 測試的最快路徑
  • 限制：
    • 每次跑都是新 unit ID（不適合腳本記住 path）
    • 系統 reboot 後消失（不能用於 boot 時自動啟動）
    • 沒有完整 unit file 的依賴管理（After=、Requires= 等）
    • 大量設定時命令列冗長（多個 -p 不易讀）
典型用例：
  1. 「跑這個程式並限制只能用 500MB RAM」
  2. 「測試 CPUQuota=50% 對 workload 影響」
  3. 「cron 跑 backup，希望它不影響其他服務」
  4. 「debug 時開個 limit shell 測新設定」
反例：production 跑 web server—應該用 unit file（持久化、自動 boot、依賴管理）。
Q6.systemd 怎麼自動啟用 cgroup controller？什麼時候會自動啟用？
自動啟用機制：systemd 偵測 unit 設定，需要哪個 controller 就自動啟用。沒有 directive 就不啟用——避免 overhead。 觸發條件：
  • 設 CPUWeight= 或 CPUQuota= → 啟用 cpu controller
  • 設 MemoryMax= 或 MemoryHigh= 等 → 啟用 memory controller
  • 設 IOWeight= 等 → 啟用 io controller
  • 設 TasksMax= → 啟用 pids controller
  • 設 AllowedCPUs= → 啟用 cpuset controller
顯式啟用 accounting（不設 limit 但開計量）：
  • MemoryAccounting=yes → 啟用 memory（看用量但不限制）
  • CPUAccounting=yes → 啟用 cpu
  • IOAccounting=yes → 啟用 io
  • TasksAccounting=yes → 啟用 pids（多數 distro 預設 yes）
階層傳播：
  • 因為 cgroup hierarchical，controller 在某個 unit 啟用，所有 parent 與該 parent 之下所有 sibling 都自動啟用該 controller。
  • 例：對 myapp.service 設 MemoryMax → memory controller 在 -.slice、system.slice、myapp.service 全鏈啟用。
查當前狀態：cat /sys/fs/cgroup/cgroup.subtree_control 看 root 啟用了哪些。
Q7.OOMPolicy= 的三個值（kill、continue、stop）行為差別？跟 ManagedOOMMemoryPressure= 怎麼配合？
OOMPolicy= 的三個值：
  • continue：kernel OOM killer 殺 process 後 unit 繼續跑（其他 process 還活著）。傳統 v1 行為，不適合 atomic workload。
  • kill：kernel OOM 觸發時殺整個 cgroup（memory.oom.group=1）。Pod-style atomic 行為。
  • stop：kernel OOM 後 systemd 停止整個 unit（不嘗試重啟）。適合需要外部 orchestrator 介入的場景。
ManagedOOMMemoryPressure= 是不同層次：
  • 設定 ManagedOOMMemoryPressure=kill 啟用 systemd-oomd 監控這個 unit/slice 的 PSI。
  • 當 memory.pressure full avg10 超過 ManagedOOMMemoryPressureLimit，systemd-oomd 主動殺（不等 kernel OOM）。
  • 殺的對象通常是子 cgroup 中最佔記憶體的，使用 cgroup.kill atomic 介面。
兩者協作：
  1. systemd-oomd（user-space）在 PSI 高時提早介入——更快、更聰明。
  2. kernel OOM killer 是最後防線——當 systemd-oomd 反應不夠快時兜底。
  3. OOMPolicy 控制 kernel OOM 之後 unit 怎麼處理（是否整個殺、是否停 unit）。
建議組合： 
[Service]
OOMPolicy=kill                          # kernel OOM 時整個殺
ManagedOOMMemoryPressure=kill           # systemd-oomd 提早介入
ManagedOOMMemoryPressureLimit=50%       # 觸發閾值
Q8.實作思考題：你接手一個 systemd-managed Linux 機器，要審計所有 service 的資源限制健康度。請設計完整審計流程。
完整審計流程： 
[Phase 1: 盤點現況]
1. 列出所有 active service：
   systemctl list-units --type=service --state=active

2. 對每個 service 取資源設定：
   systemctl show -p MemoryMax,MemoryHigh,CPUQuota,TasksMax,Slice $unit

3. 找出 unbounded service（無限制）：
   寫腳本（Lab 類似）找 limit=infinity 的 service

[Phase 2: 看實際使用]
4. 收集每個 service 的當前 + peak：
   for cg in /sys/fs/cgroup/system.slice/*.service; do
       echo "$(basename $cg) current=$(cat $cg/memory.current) peak=$(cat $cg/memory.peak)"
   done

5. 看 events（過去事件）：
   cat $cg/memory.events  # OOM、throttle 計數
   cat $cg/cpu.stat       # throttle 累計

6. 看 PSI 歷史：
   cat $cg/cpu.pressure  cat $cg/memory.pressure

[Phase 3: 分類與優先級]
7. 分類 service：
   - critical（必須有保護）：DB、認證、網路 stack
   - production（要設限）：應用 service
   - non-critical（弱保護）：log shipper、metrics
   - system（已由 systemd 管）：systemd-* 等

8. 找 limit 設定問題：
   - production unbounded → 高優先設限
   - critical 沒 MemoryMin → 加保護
   - 有過 OOM 的 → 容量規劃
   - 有過 throttle 的 → cpu.max 設太緊

[Phase 4: 設限與驗證]
9. 對 critical 加 MemoryMin / CPUWeight 高
10. 對 production 用 PSI-driven 設 MemoryMax / CPUQuota
11. 對 non-critical 設較緊 limit + ManagedOOM
12. 對 system.slice 整體保護（MemoryMin=2G）

[Phase 5: 持續監控]
13. 部署 PSI / events 採集到 Prometheus
14. 設 alert：PSI 高、OOM、throttle 發生
15. 月度 review：用過去 30 天資料重算 limit
16. Runbook：每個 alert 對應 playbook
關鍵原則：
  1. 先觀測再設限：沒 baseline 不要拍腦袋設數字。
  2. 分類優先級：不是所有 service 都該設一樣嚴的限。
  3. 系統 daemon 給 MemoryMin 保護：避免應用拖垮整機。
  4. 用 drop-in 不直接改 unit：distro update friendly。
  5. 記錄變更：所有 limit 設定進 git / ansible（不是 set-property 黑盒子）。
