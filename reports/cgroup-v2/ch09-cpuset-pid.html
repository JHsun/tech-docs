cpuset 與 PID controllers

cpuset controller 把 process 綁到特定 CPU 與 NUMA node，是 latency-sensitive、HPC、DPDK、K8s CPU manager static 的核心。 partition 機制（kernel 5.18+）讓 cgroup 動態建立 isolcpus 等價的隔離分區，不需要重開機。 PID controller 簡單但實用——一行 pids.max 就能擋下 fork bomb。本章把兩個 controllers 一次講透。

01 · Highlights

本章重點

  1. cpuset 兩大維度：cpuset.cpus（CPU 親和性）與 cpuset.mems（NUMA node 親和性）。前者決定 process 能在哪些核上跑，後者決定記憶體從哪些 NUMA node 分配。
  2. effective vs requested：cpuset.cpus 是請求值，cpuset.cpus.effective 是實際生效值（受 parent、CPU hotplug、partition 影響）。看 effective 才是真實狀況。
  3. partition 機制（kernel 5.18+）：cpuset.cpus.partition 把 cgroup 升級成 partition root，獨佔指定 CPU、與其他 cgroup 隔離。isolated 模式提供類似 isolcpus= boot parameter 的效果，但動態可調。
  4. cpuset.cpus.exclusive（kernel 6.7+）：明確列出獨佔的 CPU 子集，是 remote partition 的基礎——不需要連續的祖先鏈也能建 partition。
  5. PID controller 兩個重點：pids.max 防 fork bomb、pids.peak + pids.events 監控逼近上限的歷史。systemd TasksMax= 對應介面。
02 · Latest

最新動態（2025–2026）

2025-09 kernel 6.x 修正 isolated partition 與 isolcpus 共存問題（Tejun Heo patch series）。在含有 isolcpus 的系統上，isolated partition 之前無法正確啟用，現已修正。
2024–2025 Kubernetes CPU Manager static policy 全面採用 cpuset v2。Guaranteed Pod 的 integer CPU request 自動透過 cpuset.cpus 綁定，避免 noisy neighbor。
2024-06 Red Hat 發布 cgroup v2 cpuset isolation 官方指引，明確推薦 isolated partition 取代 isolcpus boot parameter，特別是動態 workload（NFV、即時系統）。
2025 cpuset.cpus.exclusive 與 remote partition 普及（kernel 6.7+）。現在可以在非直接祖先鏈上建立 partition，配置更靈活，DPDK / NFV 場景大量採用。
2025 systemd 256+ 完整支援 partition。AllowedCPUs= 與新的 partition 控制 directive 讓 systemd unit 可以直接設定 partition 角色，無需手寫 cgroup 介面。

2026 年的時點，cpuset 已從「NUMA 系統的小眾工具」演進成「容器 CPU 隔離的標準機制」。 partition 與 isolated partition 取代了 isolcpus boot parameter，動態可調且整合 cgroup tree——這是雲原生 latency-sensitive 工作負載的關鍵能力。 本章從基礎到 partition 完整覆蓋。

03 · Core Concepts

核心內容

3.1 cpuset 介面總覽

檔案類型作用
cpuset.cpusRW允許使用的 CPU 列表（如 0-3,8-11）
cpuset.cpus.effectiveR實際生效的 CPU 列表（考慮 parent、hotplug 後）
cpuset.memsRW允許分配的 NUMA node 列表
cpuset.mems.effectiveR實際生效的 NUMA node 列表
cpuset.cpus.exclusiveRW獨佔的 CPU 子集（kernel 6.7+，partition 用）
cpuset.cpus.exclusive.effectiveR實際獨佔生效的 CPU
cpuset.cpus.partitionRWpartition 角色：member(預設) / root / isolated
cpuset.cpus.isolatedR · 唯讀（在 root cgroup）當前所有 isolated partition 的 CPU 集合
cpuset.mems.partitionRWmemory partition（kernel 6.7+，較少用）

3.2 cpuset.cpus 與 cpuset.mems 基礎

v2 的 cpuset 介面使用標準 CPU range 格式： 

# 允許 CPU 0、1、2、3
echo "0-3" > cpuset.cpus

# 允許 CPU 0-3 和 8-11（混合範圍與單一 CPU）
echo "0-3,8-11" > cpuset.cpus

# 允許 NUMA node 0
echo "0" > cpuset.mems

# 允許所有可用 NUMA node（清空 = 繼承 parent）
echo "" > cpuset.mems

kernel 把 cpuset 規則作為 sched_setaffinity() 與 mbind() 的前置 filter—— 即使應用呼叫 sched_setaffinity 想設更廣的 mask，最終也會被 cpuset 限制住。

effective 與 requested 的差異

cpuset.cpus 是你設定的請求值。 cpuset.cpus.effective 是實際生效值。兩者可能不同，原因：

  • parent 約束：parent 的 effective 縮小，child 的 effective 也跟著縮（無法超過）。
  • CPU hotplug：機器上某些 CPU 被 offline，effective 自動排除。
  • partition 邊界：sibling partition 占用了某些 CPU，這個 cgroup 自動排除。

實務上：debug cpuset 一律先看 effective，不要看 cpus。

3.3 NUMA 親和性的重要性

現代 server 多半是多 socket NUMA 架構——CPU 訪問 local NUMA node 的記憶體比 remote 快 1.5–3 倍。 cpuset.mems 配合 cpuset.cpus 把 process 與記憶體都綁在同一 NUMA node，大幅降低 cross-socket latency。 

# 把 service 綁到 NUMA node 0（CPU 0-15、記憶體 node 0）
echo "0-15" > /sys/fs/cgroup/myapp/cpuset.cpus
echo "0" > /sys/fs/cgroup/myapp/cpuset.mems

# 看當前 NUMA topology
numactl --hardware
# 範例輸出：
# node 0 cpus: 0-15 32-47
# node 1 cpus: 16-31 48-63

典型受惠 workload：DB（PostgreSQL、MySQL、Redis）、JVM、HPC、ML 訓練、所有 latency 敏感的 service。

3.4 partition 機制：cgroup 內的 isolcpus

傳統的 CPU 隔離靠 isolcpus= kernel boot parameter——指定的 CPU 從 scheduler 的 load balancing 中移除，必須重開機才能改。 對動態 workload（NFV、即時系統、租戶隔離）這個限制太死。

kernel 5.18+ 引入 partition 機制，把 cgroup 升級成「partition root」，獨佔指定 CPU 並建立獨立 sched domain。cpuset.cpus.partition 接受三個值：

partition 角色意義典型用途
member 普通 cgroup（預設值）。CPU 屬於 parent partition。 絕大多數 cgroup
root partition root。cgroup 獨佔 cpuset.cpus 列出的 CPU，與其他 partition 隔離。仍有 load balancing。 K8s Guaranteed Pod 的 CPU 隔離
isolated 類似 root 但禁用 sched load balancing。等同動態的 isolcpus。 NFV、DPDK、即時系統 worker thread

建立 partition 的步驟

# 1. 把 cgroup 的 cpuset.cpus 設好
sudo mkdir /sys/fs/cgroup/dpdk-app
echo "8-15" | sudo tee /sys/fs/cgroup/dpdk-app/cpuset.cpus
echo "0" | sudo tee /sys/fs/cgroup/dpdk-app/cpuset.mems

# 2. 把 cgroup 升級成 isolated partition
echo "isolated" | sudo tee /sys/fs/cgroup/dpdk-app/cpuset.cpus.partition

# 3. 確認狀態
cat /sys/fs/cgroup/dpdk-app/cpuset.cpus.partition
# 預期: isolated

# 4. 看 root cgroup 的 isolated CPU 集合
cat /sys/fs/cgroup/cpuset.cpus.isolated
# 預期包含: 8-15

效果：CPU 8-15 從這刻起不會被 scheduler 用於其他 cgroup 的 task，且這個分區內沒有 load balancing——對 DPDK 這類 polling-based 應用是必要條件（避免 scheduler 偷打斷 polling thread）。

partition 的限制

建 partition 的條件相當嚴格：

  • parent 必須是 partition root（root cgroup 是 implicit partition root）。
  • cpus 必須是 parent partition CPU 的子集。
  • cpus 不能與 sibling partition 重疊。
  • 建 partition 後，cpuset.cpus 不能改成造成衝突的值（會回 EINVAL 或進入 invalid 狀態）。

失敗時 partition 會進入 invalid 狀態： 

cat cpuset.cpus.partition
# 範例：root invalid (Cpu list in cpuset.cpus mismatches with cpuset.cpus.exclusive in parent)

3.5 cpuset.cpus.exclusive 與 remote partition

kernel 6.7+ 引入 cpuset.cpus.exclusive，解決必須建立連續祖先鏈才能建 partition 的限制。

傳統限制：要在 /A/B/C 建 partition，必須先把 A 與 B 都改成 partition root——這對 systemd-managed 系統很麻煩（system.slice 不能輕易改 partition）。

cpuset.cpus.exclusive 讓你明確列出獨佔的 CPU 子集，從父層往下繼承，建立「遠端 partition」： 

# Step 1: 在祖先 cgroup 列出獨佔 CPU
echo "8-15" | sudo tee /sys/fs/cgroup/system.slice/cpuset.cpus.exclusive

# Step 2: 在後代 cgroup 直接建立 isolated partition（不需祖先鏈都是 partition）
echo "8-15" | sudo tee /sys/fs/cgroup/system.slice/myapp.service/cpuset.cpus
echo "isolated" | sudo tee /sys/fs/cgroup/system.slice/myapp.service/cpuset.cpus.partition

這個機制讓 systemd 等 managed 系統可以無侵入地支援 partition——不需要強迫整條祖先鏈變成 partition。

3.6 partition 與 sched_load_balance

v1 cpuset 有 cpuset.sched_load_balance 開關，v2 把這個能力整合到 partition 機制中：

  • member：跟著 parent partition 的 load balancing 設定。
  • root：建立獨立 sched domain，內部仍有 load balancing，但不跨 partition 邊界。
  • isolated：建立獨立 sched domain 並禁用 load balancing。task 在 partition 內的 CPU 之間不會自動遷移。

3.7 isolated partition vs isolcpus= boot parameter

面向isolcpus= bootisolated partition
動態調整❌ 改要重開機✅ runtime 切換
cgroup 整合❌ 無✅ 完整整合
per-workload 控制❌ 全機共用✅ per-cgroup 設定
workqueue 排除✅ 自動✅ kernel 6.7+ 自動
kthread 排除✅ 完整⚠️ 部分
boot 階段保護✅ 從 boot 就生效❌ 需 cgroup 啟動後才生效

建議：

  • 動態 workload（容器、租戶隔離、可變的 NFV）→ isolated partition
  • 從 boot 階段就要絕對隔離（極端 latency 要求）→ isolcpus= + 配合 isolated partition 雙重保護
  • HPC、即時系統：通常組合使用——isolcpus 給 boot 層保護，partition 給 runtime 動態調整

3.8 PID controller：簡單但實用

PID controller 在 v2 介面非常簡潔，但它的價值在於它是阻擋 fork bomb 的最後一道防線。

檔案類型作用
pids.maxRWcgroup 能持有的最大 task 數量（含 thread）
pids.currentR當前 task 數量
pids.peakR歷史最高 task 數量（kernel 6.0+）
pids.eventsR事件統計：max（觸 limit 次數）

典型用法

# 限制這個 service 最多 100 個 task
echo "100" > /sys/fs/cgroup/myapp/pids.max

# 看當前用量
cat /sys/fs/cgroup/myapp/pids.current
cat /sys/fs/cgroup/myapp/pids.peak

# 看是否曾經 hit limit
cat /sys/fs/cgroup/myapp/pids.events
# max 234   ← 234 次 fork 失敗（已被擋下）

關鍵特性：

  • 計數包含 thread：100 個限制不是 100 個 process，是 100 個 task（kernel 視角）。一個多 thread 的 process 算多個 task。
  • fork 失敗時回 EAGAIN：到 limit 後 fork()、clone() 會直接失敗，不會殺現有 process。
  • 不影響已存在 task：把 limit 設低過 current 不會殺 task，只是擋下未來 fork。

fork bomb 防護案例

一個經典 fork bomb： 

# 危險 - 不要在 production 跑
:(){ :|:& };:

在沒設 pids.max 的 cgroup 跑這個會把 host 撐爆（task table 爆滿）。設 pids.max=200 後，fork 到第 200 個就被擋下，host 安全。

3.9 systemd 介面對應

systemd directivecgroup 介面備註
AllowedCPUs=cpuset.cpusCPU 親和性，systemd 244+
StartupAllowedCPUs=cpuset.cpus啟動階段的 CPU set
AllowedMemoryNodes=cpuset.memsNUMA node 親和性
StartupAllowedMemoryNodes=cpuset.mems啟動階段的 mem set

TasksAccounting=yes啟用 pids controller多數 distro 預設 yes

partition 控制目前 systemd 沒有 direct directive，仍需用 ExecStartPost 寫入 cpuset.cpus.partition。 systemd 256+ 已加入更完整的 partition 支援，但語法仍在演進。

04 · Hands-on

實戰範例

Lab 1 · 把 service 綁到指定 CPU
# 簡單範例：把 service 限制只能在 CPU 0-3 跑
sudo systemd-run --slice=user.slice --unit=cpu-pinned \
    -p AllowedCPUs=0-3 \
    bash -c 'while :; do :; done'

# 確認設定生效
CG=$(systemctl show -p ControlGroup --value cpu-pinned.service)
cat /sys/fs/cgroup$CG/cpuset.cpus
cat /sys/fs/cgroup$CG/cpuset.cpus.effective

# 看 process 實際的 affinity
PID=$(systemctl show -p MainPID --value cpu-pinned.service)
taskset -pc $PID
# 預期：current affinity list: 0-3

# 清理
sudo systemctl stop cpu-pinned
Lab 2 · NUMA 親和性配置（需多 socket 機器）
# 先看機器 NUMA topology
numactl --hardware

# 把 service 完整綁到 NUMA node 0
sudo systemd-run --slice=user.slice --unit=numa-pinned \
    -p AllowedCPUs=0-15 \
    -p AllowedMemoryNodes=0 \
    --pty \
    python3 -c '
import time
data = bytearray(500 * 1024 * 1024)  # 分配 500MB
print("allocated 500MB on NUMA 0")
time.sleep(60)
'

# 在另一 terminal 看記憶體分配的 NUMA 位置
PID=$(systemctl show -p MainPID --value numa-pinned.service)

# 用 numastat 看
sudo numastat -p $PID
# 預期：所有記憶體都在 Node 0，Node 1 = 0

# 對比：未綁 NUMA 的 service 通常記憶體散在多個 node
# 測試結束清理
sudo systemctl stop numa-pinned
Lab 3 · 建立 isolated partition（DPDK 模擬）
# 假設我們要為一個 DPDK 應用獨佔 CPU 4-7

# Step 1: 在 user.slice 預先設好 cpus.exclusive
echo "4-7" | sudo tee /sys/fs/cgroup/user.slice/cpuset.cpus.exclusive

# Step 2: 啟動 service 並設置 cpus
sudo systemd-run --slice=user.slice --unit=dpdk-sim \
    -p AllowedCPUs=4-7 \
    -p AllowedMemoryNodes=0 \
    --pty \
    bash -c '
while :; do :; done &
while :; do :; done &
wait
'

# Step 3: 把 service cgroup 升級成 isolated partition
CG=$(systemctl show -p ControlGroup --value dpdk-sim.service)
echo "isolated" | sudo tee /sys/fs/cgroup$CG/cpuset.cpus.partition

# 確認狀態
cat /sys/fs/cgroup$CG/cpuset.cpus.partition
# 預期：isolated

# 看 root cgroup 知道哪些是 isolated
cat /sys/fs/cgroup/cpuset.cpus.isolated
# 預期包含：4-7

# 觀察：CPU 4-7 上不會有 scheduler load balancing
# 用 perf 看
sudo perf stat -e sched:sched_migrate_task -C 4-7 sleep 5
# 預期：很少甚至沒有 migrate event

# 清理
sudo systemctl stop dpdk-sim
echo "" | sudo tee /sys/fs/cgroup/user.slice/cpuset.cpus.exclusive
Lab 4 · pids.max 防 fork bomb
# 警告：在 VM 或測試機跑，不要在重要機器跑

# 啟動受限 shell
sudo systemd-run --slice=user.slice --unit=fork-test \
    -p TasksMax=50 \
    --pty \
    bash

# 在裡面跑類 fork bomb（已用 pids.max 擋下，安全）
# 先看當前 task 數
cat /proc/$$/cgroup
ls /proc/$$/task | wc -l

# 試圖 fork 100 次
for i in $(seq 1 100); do
    sleep 60 &
done
echo "Got $(jobs | wc -l) jobs"

# 預期：到 50 之後 fork 失敗，看到 -bash: fork: retry: Resource temporarily unavailable

# 看 pids.events
CG=$(systemctl show -p ControlGroup --value fork-test.service)
cat /sys/fs/cgroup$CG/pids.events
# 預期：max > 0

# 退出 shell
exit
sudo systemctl stop fork-test

觀察點：fork 被擋下時系統還活著——這是 pids.max 的核心價值。沒設的話，host 可能已經 hang 了。

Lab 5 · 用 pids.peak 監控容量規劃
# 跑一個會啟很多 thread 的應用
sudo systemd-run --slice=user.slice --unit=thread-app \
    --pty \
    python3 -c '
import threading, time
def worker():
    time.sleep(30)
threads = [threading.Thread(target=worker) for _ in range(80)]
for t in threads: t.start()
print(f"started {len(threads)} threads")
time.sleep(5)
# 多開一波
threads2 = [threading.Thread(target=worker) for _ in range(40)]
for t in threads2: t.start()
print(f"now {len(threads)+len(threads2)} threads total")
time.sleep(30)
'

# 跑完看 pids.peak
CG=$(systemctl show -p ControlGroup --value thread-app.service)
cat /sys/fs/cgroup$CG/pids.peak
# 預期約 122（80 + 40 + main thread + python overhead）

# 啟示：用 pids.peak 規劃 TasksMax 設多少
# 規則：peak * 1.5 是合理 limit

# 清理
sudo systemctl stop thread-app 2>/dev/null
05 · Pitfalls

常見陷阱

陷阱 1 · 看 cpuset.cpus 而不看 effective

debug「為什麼我的 process 沒在 CPU 5 上跑」時，只看 cpuset.cpus 是錯的——應該看 cpuset.cpus.effective。 典型場景：parent 的 effective 是 0-3，你 child 設 0-7，effective 仍是 0-3（被 parent 限制）。 養成習慣：cpuset 任何 debug 都先看 effective。

陷阱 2 · cpuset.cpus 與 cpuset.mems 必須都設

在 v1，cpuset.cpus 設好就能用。v2 開始要求 cpus 與 mems 都必須有效，否則 process 不能進入 cgroup。 典型踩雷：只設 cpuset.cpus，把 process 寫進 cgroup.procs 失敗（EINVAL）。 修法：同時設 cpuset.mems。如果不在意 NUMA，可以設成所有可用 node（如 0-1）。

陷阱 3 · partition root invalid 沒看到原因

建 partition 失敗時，cpuset.cpus.partition 顯示 root invalid (...)，括號裡是失敗原因。 例如：root invalid (Cpu list in cpuset.cpus mismatches with cpuset.cpus.exclusive in parent)。 多數人沒看完整字串就以為是「這版 kernel 不支援」——其實 kernel 已經告訴你怎麼修了。 養成習慣：partition 出錯時 cat cpuset.cpus.partition 完整讀一遍。

陷阱 4 · 在含有 isolcpus 的 host 上建 isolated partition 失敗

kernel 6.x 之前的版本有 bug：機器設了 isolcpus=4-7 boot parameter，再嘗試把 cgroup 改成 isolated partition 包含 4-7 會失敗或進 invalid。 修法（kernel 6.x 之前）：要嘛只用 isolcpus，要嘛拿掉 isolcpus 改用 isolated partition，不要混用。 kernel 6.x 後 Tejun Heo 已修正，可以共存（2025-09 patch series）。檢查 kernel 版本：uname -r，至少要 > 6.10 才安全混用。

陷阱 5 · pids.max 計數包含 thread，limit 容易設太低

把 pids.max=20 給 Java service，發現啟動失敗——JVM 一啟動就有 20+ thread（GC、JIT、main、watcher 等等）。 pids.max 計數所有 task（thread + process），不是只計 process。Java app 通常需要 100+。 正確做法：先觀察 pids.peak，再設 limit 為 peak 的 1.5–2 倍。

陷阱 6 · cpuset.cpus 在 systemd unit 寫 CPU mask 而非 list

錯誤：AllowedCPUs=0xff（hex mask）。systemd 不接受 mask，只接受 CPU list（0-7）。 錯誤：AllowedCPUs=4 5 6 7（空格分隔）。應該用逗號或 dash：AllowedCPUs=4-7 或 AllowedCPUs=4,5,6,7。

陷阱 7 · taskset 與 cpuset 衝突

process 在 cpuset.cpus=0-3 的 cgroup 內，呼叫 taskset -pc 4-7 $PID 想搬到 4-7——失敗。 kernel 把 cpuset 作為 sched_setaffinity() 的前置 filter，taskset 設的範圍會被 cgroup 限制住。 實際 affinity 是「taskset 設的 ∩ cpuset.cpus.effective」。

陷阱 8 · 在 isolated partition 裡的 task 不會自動 spread

建了 isolated partition 包含 CPU 4-7，但 service 內 4 個 thread 全部跑在 CPU 4？ 因為 isolated partition 禁用 load balancing，scheduler 不會自動把 task 分散到其他 CPU。 修法：應用層自己用 pthread_setaffinity_np 把 thread 綁到不同 CPU。或用 root（非 isolated）partition 保留 load balancing。

練習 1 · 探索系統 NUMA 配置（10 分鐘）
# 步驟 1：列出 NUMA topology
numactl --hardware

# 步驟 2：看每個 CPU 對應的 NUMA node

# 步驟 3：看一個 service 是否做了 NUMA 親和
SERVICE=systemd-resolved.service
CG=$(systemctl show -p ControlGroup --value $SERVICE)
echo "cpuset.cpus.effective: $(cat /sys/fs/cgroup$CG/cpuset.cpus.effective 2>/dev/null)"
echo "cpuset.mems.effective: $(cat /sys/fs/cgroup$CG/cpuset.mems.effective 2>/dev/null)"

回答：你的機器有幾個 NUMA node？哪些 CPU 屬於哪個？預設 service 有沒有做 NUMA 綁定？

練習 2 · 量化 NUMA local vs remote 性能差異（15 分鐘）

需要多 socket 機器。寫一個 memory benchmark：

# Local: CPU 與 mem 都在 NUMA 0
sudo systemd-run --slice=user.slice --unit=numa-local \
    -p AllowedCPUs=0 -p AllowedMemoryNodes=0 \
    --pty \
    bash -c 'time dd if=/dev/zero of=/tmp/test bs=1M count=1000 oflag=direct; rm /tmp/test'

# Remote: CPU 在 NUMA 0，mem 在 NUMA 1
sudo systemd-run --slice=user.slice --unit=numa-remote \
    -p AllowedCPUs=0 -p AllowedMemoryNodes=1 \
    --pty \
    bash -c 'time dd if=/dev/zero of=/tmp/test bs=1M count=1000 oflag=direct; rm /tmp/test'

回答：local vs remote 速度差幾倍？對你的 production workload 影響有多大？

練習 3 · 建立 root partition 給 K8s Guaranteed Pod 模擬（15 分鐘）
# 模擬 K8s static CPU manager 的行為：給一個 Pod 獨佔 CPU 8-9

# Step 1: 在 system.slice 設 exclusive
echo "8-9" | sudo tee /sys/fs/cgroup/user.slice/cpuset.cpus.exclusive

# Step 2: 建立 service 模擬 Pod
sudo systemd-run --slice=user.slice --unit=guaranteed-pod \
    -p AllowedCPUs=8-9 \
    --pty \
    bash -c 'while :; do :; done'

# Step 3: 升級成 root partition（保留 load balancing，但 CPU 獨佔）
CG=$(systemctl show -p ControlGroup --value guaranteed-pod.service)
echo "root" | sudo tee /sys/fs/cgroup$CG/cpuset.cpus.partition

# Step 4: 確認其他 service 看不到 CPU 8-9
cat /sys/fs/cgroup/user.slice/cpuset.cpus.effective
# 應該排除了 8-9

# 清理
sudo systemctl stop guaranteed-pod
echo "" | sudo tee /sys/fs/cgroup/user.slice/cpuset.cpus.exclusive

回答：root partition 與 isolated partition 的差別是什麼？這個 Pod 用哪個比較合適？

練習 4 · 寫一個 fork bomb 防護腳本（10 分鐘）

對所有 user.slice 下的 service 自動設 TasksMax：

#!/bin/bash
# auto-task-limit.sh
# 對 user.slice 下所有 active service 設 TasksMax=current_peak * 2

for unit in $(systemctl list-units --type=service --state=active --no-legend --plain | awk '{print $1}'); do
    cg=$(systemctl show -p ControlGroup --value "$unit" 2>/dev/null)
    [ -z "$cg" ] && continue
    [ ! -f "/sys/fs/cgroup$cg/pids.peak" ] && continue
    
    peak=$(cat /sys/fs/cgroup$cg/pids.peak)
    if [ "$peak" -gt 10 ]; then  # 過濾沒任務的
        new_limit=$((peak * 2))
        echo "Setting TasksMax=$new_limit for $unit (peak=$peak)"
        # 真實執行請取消註解：
        # sudo systemctl set-property "$unit" "TasksMax=$new_limit"
    fi
done

練習目的：把 pids.peak 當成容量規劃工具。

07 · Self-check

自我檢查題

Q1.cpuset.cpus 與 cpuset.cpus.effective 有什麼差別？debug 時應該看哪個？
cpuset.cpus：你請求的 CPU 範圍——cgroup 「希望」用這些 CPU。 cpuset.cpus.effective：實際生效的 CPU 範圍——考慮所有限制因素後的真實值。 影響 effective 的因素：
  1. parent 的 effective 縮小（無法超過 parent）
  2. CPU hotplug（被 offline 的 CPU 自動排除）
  3. sibling partition 占用了某些 CPU
  4. cpus.exclusive 的限制
debug 規則：永遠先看 effective。如果 effective 與 cpus 不同，順著樹往上找原因——多半是某個祖先的限制造成。
Q2.cpuset.cpus.partition 的三個值（member、root、isolated）各自什麼意思？什麼場景用哪個？
  • member（預設）：普通 cgroup，CPU 屬於 parent partition。絕大多數 cgroup 都是這個。
  • root：partition root。獨佔 cpuset.cpus 列出的 CPU，與其他 partition 隔離。內部仍有 load balancing。
  • isolated：類似 root 但禁用 load balancing。等同動態的 isolcpus boot parameter。
場景對應：
  • K8s Guaranteed Pod 的 CPU 獨佔 → root（內部多 thread 仍可 load balance）
  • DPDK 應用 polling thread → isolated（不要被 scheduler 打斷）
  • NFV 即時系統 worker → isolated
  • 一般容器隔離 → root（不需要禁 load balance）
Q3.cpuset.cpus.exclusive 解決什麼問題？沒有它的話建 partition 有什麼限制？
解決問題：傳統限制——要在 /A/B/C 建 partition，必須先把 A 與 B 都升級成 partition root。整條祖先鏈都要是 partition，限制太死。 cpuset.cpus.exclusive 機制（kernel 6.7+）：在祖先 cgroup 明確列出獨佔 CPU 的子集，後代 cgroup 可以直接拿這些 CPU 建 partition，不需要祖先都是 partition root。這稱為「remote partition」。 為什麼重要：systemd-managed 系統的 system.slice、user.slice 等本來就不是 partition root（會破壞太多現有行為）。有了 cpus.exclusive，systemd 可以無侵入地支援 partition——只在 system.slice 設 exclusive，service 自己升級 partition 就行。 實務範例：在 system.slice 的 cpus.exclusive 裡放 8-15，然後 myapp.service 直接 echo isolated > cpus.partition 就能成功。
Q4.isolated partition 與 isolcpus= boot parameter 的差別？什麼時候用哪個？
isolcpus= boot：
  • 從 boot 階段就生效（連 init 都不會用這些 CPU）
  • 需要重開機才能改
  • 全機共用（不是 per-cgroup）
  • 對 kthread / workqueue 排除最完整
isolated partition：
  • runtime 動態切換
  • per-cgroup 設定，不同 service 可獨立配置
  • 整合 cgroup tree
  • 需要 cgroup 啟動後才生效
建議：
  • 動態 workload（容器、租戶、可變的 NFV）→ isolated partition
  • 絕對隔離 + 從 boot 就要保護 → isolcpus= + isolated partition 雙重
  • HPC / 即時系統 → 通常組合使用
kernel 6.x+ 後兩者可以共存，之前版本不要混用。
Q5.cpuset.mems 為什麼跟 cpuset.cpus 一樣重要？只設 cpus 會發生什麼？
cpuset.mems 的作用：限制 cgroup 內 process 的記憶體只能從哪些 NUMA node 分配。對 multi-socket NUMA 機器影響巨大——cross-socket memory access 比 local 慢 1.5–3 倍。 只設 cpus 不設 mems 的問題：
  1. v2 強制要求兩者都設：mems 為空時 process 無法進入 cgroup（EINVAL）。
  2. NUMA mismatch：把 process 綁到 NUMA 0 的 CPU（cpus=0-15），但記憶體分散在 NUMA 0 與 NUMA 1，每次 access NUMA 1 的記憶體都跨 socket，性能差。
正確做法： 
echo "0-15" > cpuset.cpus       # CPU 在 NUMA 0
echo "0" > cpuset.mems           # 記憶體也在 NUMA 0
如果不在意 NUMA（單 socket 機器），mems 設成所有可用 node（如 0）即可。
Q6.pids.max 計數包含 thread 還是只算 process？實務上設多少合理？
計數包含 thread。kernel 視角下「task」既是 process 也是 thread，pids.max 限制 task 總數。 實務影響：
  • JVM service 一啟動就 20+ thread（GC、JIT、main）。設 pids.max=20 會啟動失敗。
  • Web server 用 thread pool（如 Tomcat）很容易 200+ thread。
  • fork-based 應用（如 PHP-FPM）每個 worker 一個 process，但每個 process 又可能多 thread。
合理設定方式：
  1. 先觀察 pids.peak 跑一段時間（24 小時包含 traffic peak）
  2. limit 設為 peak 的 1.5–2 倍，給 burst headroom
  3. 對 untrusted workload（multi-tenant）設更嚴格的 limit 防 fork bomb
  4. 對 trusted system service 設較寬，避免影響正常 spawn
典型值：systemd default TasksMax 是 80% × kernel.pid_max（通常幾萬）；個別 service 通常 1000–10000 之間。
Q7.在 isolated partition 內為什麼 task 不會自動 spread 到所有 CPU？怎麼手動分配？
原因：isolated partition 禁用 sched load balancing。scheduler 不會自動把 task 從一個 CPU 搬到另一個來平衡負載。建立 4 個 thread 在 partition 內，可能全部跑在同一個 CPU。 為什麼這樣設計：DPDK / 即時系統的 polling thread 不希望被 scheduler 干擾——load balancing 本身會偷走 CPU 時間。所以 isolated 是明確選擇「讓應用自管 affinity」。 手動分配方法：
  • 應用層：用 pthread_setaffinity_np 或 sched_setaffinity 把 thread 綁到特定 CPU。
  • 外部工具：用 taskset -pc CPU PID 手動設定（但被 cpuset 限制住，只能在 partition 內）。
  • DPDK 自帶：DPDK EAL 啟動參數 --lcores 直接指定每個 lcore 跑哪個 CPU。
替代方案：如果不需要禁 load balancing，用 root partition 而非 isolated——保留 load balancing，task 自動 spread。
Q8.實作思考題：你管理一個 32 核 NUMA 機器（NUMA 0: CPU 0-15、NUMA 1: CPU 16-31），要跑：（A）DPDK 網路應用（要 4 核獨佔、無打斷）、（B）K8s container workload（用 16 核）、（C）系統服務（其餘）。設計完整的 cpuset 配置。
方案： 
[boot args]
# 不用 isolcpus（用 isolated partition 代替）

[user.slice / system.slice cpus.exclusive]
echo "16-19" > /sys/fs/cgroup/.../cpus.exclusive
# DPDK 獨佔的 CPU（從 NUMA 1 選，與 K8s 隔離）

[A: dpdk.service]
AllowedCPUs=16-19
AllowedMemoryNodes=1                       # NUMA 1 local
# 然後 ExecStartPost: echo isolated > cpus.partition
# 結果：CPU 16-19 獨佔，無 load balancing，DPDK polling 不被打斷

[B: kubepods.slice]
AllowedCPUs=0-15                           # NUMA 0 全部
AllowedMemoryNodes=0                       # NUMA 0 local
# 結果：K8s Pod 在 NUMA 0 跑，避免跨 socket
# 內部用 K8s CPU manager static 給 Guaranteed Pod 做 root partition

[C: system.slice]
AllowedCPUs=20-31                          # NUMA 1 剩餘
AllowedMemoryNodes=1                       # NUMA 1 local
# 系統服務在 NUMA 1，與 DPDK 共享 NUMA 但 CPU 不衝突
設計理由：
  1. DPDK 用 isolated partition：保證無 scheduler 打斷、polling 性能可預測。
  2. DPDK 與 K8s 分開 NUMA：避免 K8s 大量 task 在同 NUMA 影響 DPDK。
  3. 每個 cgroup 都做 NUMA 親和：cpus 與 mems 都在同 NUMA，避免跨 socket。
  4. K8s 用 root partition（非 isolated）：內部多 Pod 仍要 load balancing，整體吞吐優先。
  5. system.slice 留 NUMA 1 一半 CPU：給 systemd、journal、network stack 等系統服務空間。
監控配套：對 DPDK 監控 perf stat -e sched:sched_migrate_task 確認沒被搬動；對 K8s Pod 監控 numastat 確認 NUMA locality；整體監控 cpu.pressure。
