Part VII · 遷移與附錄 · Chapter 21

cgroup v1 → v2 遷移完整指南

到 2026 年，主流發行版預設都已經是 cgroup v2——但仍有大量 production 機器跑在 v1。 RHEL 8 / 9 預設 v1 但支援 v2，RHEL 10 only v2；Ubuntu 22.04 之後預設 v2，舊機器升級時碰到的 systemd / Docker / Slurm 三方配合是業界經典痛點。 切換靠單一 kernel boot parameter systemd.unified_cgroup_hierarchy=1——但這背後牽涉 整個系統的 process 都要重新進 cgroup tree、舊 libcgroup 工具失效、 容器 runtime 配置可能不對、HPC scheduler 需要重新設定。 本章把企業 fleet 的完整遷移實戰整理：發行版時間軸、kernel 切換、systemd 角色、Slurm/K8s/Docker 配合、回滾、防護。

01 · Highlights

本章重點

  1. 切換靠 kernel boot parameter：systemd.unified_cgroup_hierarchy=1 啟用 v2，=0 退回 v1。需要 reboot；切換不能 hot swap。同時用 v1 + v2 hybrid 模式可以但生產不建議。
  2. RHEL 8/9 預設仍是 v1，RHEL 10（2025）only v2；Ubuntu 22.04+ 預設 v2；Debian 11+ 預設 v2；Fedora 31+ 早就 v2。Rocky / Alma 跟 RHEL 走。檢查指令：stat -fc %T /sys/fs/cgroup/。
  3. libcgroup-style（cgconfig.conf、cgrulesengd）已 deprecated。Red Hat 強烈建議遷移到 systemd-style（CPUWeight=、MemoryMax= drop-in）。舊 cgrulesengd 在 v2 不工作，沒等價物。
  4. Docker 20.10+、containerd 1.4+、Slurm 22.05+ 都支援 v2。但 cgroup driver 配置要全 stack 一致——kubelet、container runtime、systemd 必須都 用 systemd driver（Ch.16 / Ch.17 重點再次強調）。
  5. 企業 fleet migration 看「一次切還是分階段」：保守策略「新節點上 v2、舊節點不動」最安全。RHEL 9 → 10 跳版時是天然遷移點。HPC 因 Slurm 需停機，常選 maintenance window 一次切。
02 · Latest

最新動態（2025–2026）

2025RHEL 10 發布，only cgroup v2。RHEL 8/9 仍預設 v1（向後相容），但 RHEL 10 完全不再支援 v1。從 9 升 10 的人必須先處理 v2 相容性。
2026-04Ubuntu 26.04 LTS（Resolute Raccoon）完全移除 v1 fallback。systemd 259 不再 mount v1 hierarchy——舊腳本寫 /sys/fs/cgroup/memory/... 直接 fail。
2025Kubernetes kubelet 在 v1 節點預設 fail to start。1.36+ 加入 failCgroupV1: true 預設。要繞過需明確設 false，「operationally dangerous」官方說法。
2025Slurm cgroup/v1 plugin 正式 deprecated。SchedMD 文件「no new features」。新 cluster 建議用 cgroup/v2。HPC 中心多在 25.x 升級時切換。
2025libcgroup 移除提案討論。Red Hat 與 Fedora 多次討論 libcgroup-tools（cgcreate、cgexec、cgconfigparser）的移除——目前還在，但官方建議改用 systemd-run / systemctl set-property。

2026 年的時點，「還沒遷移到 v2 就是 risk」——K8s 1.36 拒啟動、Ubuntu 26.04 移除 v1、RHEL 10 only v2、Slurm v1 plugin EOL。本章講的不只是「怎麼切」，更是「怎麼在生產環境安全地 切」——分階段、有 rollback、考慮所有依賴。

3.2 切換的核心機制：kernel boot parameter

切換 cgroup 版本是boot-time 決定，不能 hot swap： 

# 啟用 v2（unified hierarchy）
systemd.unified_cgroup_hierarchy=1

# 退回 v1
systemd.unified_cgroup_hierarchy=0

# 進階：禁用特定 v1 controller，讓它們 v2-only
cgroup_no_v1=memory,cpu

# 完全禁用 v1
cgroup_no_v1=all

為什麼是 boot-time：

  1. systemd PID 1 啟動非常早——它需要決定走哪個 cgroup 版本來組織所有 process。
  2. cgroup 一旦 mount 並有 process，就不能輕易切換——所有 process 都已掛在 hierarchy 上。
  3. kernel 設計：v1 是多 hierarchy（每個 controller 一個），v2 是單 hierarchy；同一 controller 不能同時掛在 v1 與 v2。

3.3 v1 vs v2 hybrid 模式（不推薦但可能存在）

RHEL 8/9、Ubuntu 22.04/24.04 在預設 v1 時，systemd 會額外 mount v2 在 /sys/fs/cgroup/unified——形成hybrid mode： 

# Hybrid mode 結構
$ mount | grep cgroup
tmpfs on /sys/fs/cgroup type tmpfs
cgroup on /sys/fs/cgroup/systemd type cgroup (name=systemd)
cgroup on /sys/fs/cgroup/cpu,cpuacct type cgroup (cpu,cpuacct)
cgroup on /sys/fs/cgroup/memory type cgroup (memory)
... 共 10+ v1 mounts ...
cgroup2 on /sys/fs/cgroup/unified type cgroup2  ← v2 在這

問題：

  • v2 unified mount 之下沒有 controllers（都被 v1 occupied）
  • 應用同時看到 v1 與 v2，行為不可預測
  • K8s / Slurm 多數忽略 hybrid mode 的 v2 mount

實務建議：不要靠 hybrid。要嘛 v1 要嘛 v2，明確選擇。

3.6 各 component 配合：Docker

Docker 20.10+ 完整支援 cgroup v2（先前版本部分支援）。切換 v2 後檢查： 

# 確認 Docker 看到 v2
$ docker info --format '{{.CgroupDriver}} / {{.CgroupVersion}}'
systemd / 2

# 確保用 systemd cgroup driver（v2 強烈建議）
# /etc/docker/daemon.json
{
    "exec-opts": ["native.cgroupdriver=systemd"]
}
sudo systemctl restart docker

# 測試容器
$ docker run --rm --cpus=1 --memory=500M alpine sh -c 'cat /sys/fs/cgroup/cpu.max /sys/fs/cgroup/memory.max'
100000 100000     ← 1 CPU
524288000         ← 500M

常見問題：cgroup driver 沒設 systemd（用 cgroupfs）→ resource pressure 下節點不穩定（Ch.16 已詳述）。

3.7 各 component 配合：Kubernetes

K8s 1.25+ cgroup v2 GA、1.36+ 預設拒絕 v1： 

# Step 1: 升 kernel + 切 v2（如上）
# Step 2: 確認 containerd 設 systemd driver
# /etc/containerd/config.toml
[plugins.'io.containerd.cri.v1.runtime'.containerd.runtimes.runc.options]
  SystemdCgroup = true

# Step 3: kubelet 設 systemd driver
# /var/lib/kubelet/config.yaml
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
cgroupDriver: systemd

# 1.36+ 自動偵測（KubeletCgroupDriverFromCRI）
# kubelet 從 containerd 取 driver——不需手動設

# Step 4: drain node、重啟 kubelet 與 containerd
kubectl drain <node> --ignore-daemonsets --delete-emptydir-data
sudo systemctl restart containerd kubelet
kubectl uncordon <node>

# 驗證
$ kubectl get nodes <node> -o jsonpath='{.status.nodeInfo}' | jq
{
  "kubeletVersion": "v1.36.x",
  "containerRuntimeVersion": "containerd://2.0.x",
  ...
}

# Pod 啟動測試
kubectl run test --image=alpine --restart=Never -- sleep 3600
kubectl exec test -- cat /sys/fs/cgroup/cpu.max

3.8 各 component 配合：Slurm

Slurm 22.05+ 支援 cgroup/v2 plugin（Ch.18 已詳述）： 

# 升 Slurm 到 22.05+ 之後

# /etc/slurm/cgroup.conf
CgroupPlugin=cgroup/v2          # 明確指定 v2（或 autodetect）
ConstrainCores=yes
ConstrainRAMSpace=yes
ConstrainSwapSpace=yes
ConstrainDevices=yes

# slurmd.service 加 drop-in
sudo mkdir -p /etc/systemd/system/slurmd.service.d/
cat <<EOF | sudo tee /etc/systemd/system/slurmd.service.d/cgroup-v2.conf
[Service]
Slice=system.slice
Delegate=yes
EOF

sudo systemctl daemon-reload

# 重啟流程：
# 1. drain 所有 node
sudo scontrol update NodeName=ALL State=DRAIN Reason="cgroup v2 migration"

# 2. 等所有 job 結束（或 cancel）
squeue

# 3. 重啟 slurmctld + slurmd
sudo systemctl restart slurmctld
sudo pdsh -w compute[01-N] "systemctl restart slurmd"

# 4. resume node
sudo scontrol update NodeName=ALL State=RESUME

# 驗證
sbatch --mem=2G --cpus-per-task=2 my_job.sh
ssh compute01 "ls /sys/fs/cgroup/slurmstepd.scope/"
# 應該看到 job_NNNN 子目錄

3.9 libcgroup 工具的命運

libcgroup-tools（cgcreate、cgexec、cgconfigparser、cgrulesengd 等）是早年 v1 主流工具。不要在 v2 用——多數命令會失敗或行為不正確：

libcgroup 命令v1 行為v2 替代
cgcreate建 cgroup 目錄用 systemd-run --slice=... 或 systemctl set-property
cgexec把 process 跑進 cgroupsystemd-run --scope ...
cgclassify移動 process 到 cgroupecho 寫 cgroup.procs 或 systemctl --user ...
cgconfigparser讀 cgconfig.conf 應用寫 systemd unit drop-in
cgrulesengd規則 daemon 自動分類 process沒等價，用 systemd unit 或自寫
cgget讀 cgroup 屬性systemctl show <unit> 或直接 cat 介面檔案
cgset設 cgroup 屬性systemctl set-property <unit> ...

cgconfig.conf 遷移範例： 

# 舊 v1 cgconfig.conf
group myapp {
    cpu {
        cpu.shares = 512;
    }
    memory {
        memory.limit_in_bytes = 1G;
    }
}

# 對應 v2 systemd unit（推薦）
# /etc/systemd/system/myapp.service
[Unit]
Description=My application

[Service]
Type=simple
ExecStart=/usr/local/bin/myapp
CPUWeight=50          # 對應 cpu.shares=512（v1 default 1024 → v2 weight 100；512 ≈ 50）
MemoryMax=1G          # 對應 memory.limit_in_bytes=1G

[Install]
WantedBy=multi-user.target

# 啟用
sudo systemctl daemon-reload
sudo systemctl enable --now myapp

3.10 Rollback 策略

切換失敗時 rollback 非常簡單——改回 boot parameter 即可： 

# RHEL / Rocky
sudo grubby --update-kernel=ALL --args="systemd.unified_cgroup_hierarchy=0"
sudo reboot

# Ubuntu / Debian
sudo sed -i 's/systemd.unified_cgroup_hierarchy=1//' /etc/default/grub
sudo update-grub
sudo reboot

# 確認回到 v1
$ stat -fc %T /sys/fs/cgroup/
tmpfs    ← v1（hybrid 或純 v1）

但要注意：

  • 應用如果改用 v2-style 設定（drop-in 用 CPUWeight= 而非 CPUShares=），仍會在 v1 work——systemd 會自動轉換。
  • 但 v2-only 功能（Memory QoS 的 memory.high、PSI）在 v1 沒有。
  • K8s 1.36+ 會拒絕 v1 啟動（除非 failCgroupV1: false）——如果你的 K8s 太新但要 rollback v1，要同時調整 kubelet 設定。

3.11 best practices 總結

  • 新節點直接 v2：不要建 v1 新機器。
  • RHEL 9 → 10 跳版時切：天然遷移點。
  • HPC 用 maintenance window 一次切：drain 全部 job 後 reboot。
  • K8s 用 rolling node migration：drain 一個 node、reboot 切 v2、加回 cluster；逐個進行。
  • 遷移前確認所有 component 版本：Docker 20.10+、containerd 1.4+、Slurm 22.05+、kubelet 1.25+。
  • 移除 libcgroup 依賴：寫 systemd unit 取代 cgconfig.conf。
  • 監控 metric baseline：切換前後對比 throttle、OOM、latency；任何 regression 立即查。
  • 有 rollback plan：1 行 GRUB 改即可，但要事先測試。

Lab 4 · 寫 fleet migration audit 腳本
#!/bin/bash
# fleet-cgroup-audit.sh
# 對 100+ 節點檢查當前 cgroup 狀態，輸出 CSV

NODES_FILE="$1"  # 一行一個 hostname

if [ -z "$NODES_FILE" ]; then
    echo "Usage: $0 nodes.txt"
    exit 1
fi

echo "host,cgroup_version,kernel,systemd,docker,kubelet,migration_ready"

while read host; do
    if ! ssh -o ConnectTimeout=5 -o StrictHostKeyChecking=no $host true 2>/dev/null; then
        echo "$host,UNREACHABLE,,,,,no"
        continue
    fi
    
    cgroup_v=$(ssh $host "stat -fc %T /sys/fs/cgroup/" 2>/dev/null)
    kernel=$(ssh $host "uname -r" 2>/dev/null)
    systemd_v=$(ssh $host "systemctl --version 2>/dev/null | head -1 | awk '{print \$2}'" 2>/dev/null)
    docker_v=$(ssh $host "docker version --format '{{.Server.Version}}' 2>/dev/null" 2>/dev/null)
    kubelet_v=$(ssh $host "kubelet --version 2>/dev/null | awk '{print \$2}'" 2>/dev/null)
    
    # 簡單判斷是否 ready：kernel ≥ 5.8、systemd ≥ 245
    kernel_major=$(echo $kernel | cut -d. -f1)
    kernel_minor=$(echo $kernel | cut -d. -f2)
    if [ "$kernel_major" -ge 5 ] && [ "$kernel_minor" -ge 8 ]; then
        ready="yes"
    else
        ready="no_kernel"
    fi
    
    echo "$host,$cgroup_v,$kernel,$systemd_v,$docker_v,$kubelet_v,$ready"
done < "$NODES_FILE"

# 用法
# $ echo -e "node1\nnode2\nnode3" > nodes.txt
# $ ./fleet-cgroup-audit.sh nodes.txt | tee audit.csv
# $ awk -F, '$2=="cgroup2fs"' audit.csv  # 已 v2
# $ awk -F, '$2=="tmpfs"' audit.csv      # 仍 v1
Lab 5 · K8s node 滾動遷移到 v2
# 對 100 個 K8s node 漸進遷移
NODES=$(kubectl get nodes -o jsonpath='{.items[*].metadata.name}')

for node in $NODES; do
    echo "=== Migrating $node ==="
    
    # 1. 看當前 cgroup version
    current=$(kubectl debug node/$node -it --image=alpine -- \
        chroot /host stat -fc %T /sys/fs/cgroup/ 2>/dev/null)
    echo "Current: $current"
    
    if [ "$current" = "cgroup2fs" ]; then
        echo "  Already v2, skipping"
        continue
    fi
    
    # 2. drain
    kubectl drain $node --ignore-daemonsets --delete-emptydir-data --timeout=10m
    
    # 3. SSH 到 node 改 GRUB + reboot
    ssh $node "sudo grubby --update-kernel=ALL --args='systemd.unified_cgroup_hierarchy=1' && \
               sudo systemctl reboot"
    
    # 4. 等 node 回來（大概 5-10 分鐘）
    echo "Waiting for $node to come back..."
    while ! kubectl get node $node 2>/dev/null | grep -q Ready; do
        sleep 30
    done
    
    # 5. 確認 v2
    new_v=$(kubectl debug node/$node -it --image=alpine -- \
        chroot /host stat -fc %T /sys/fs/cgroup/)
    if [ "$new_v" != "cgroup2fs" ]; then
        echo "ERROR: $node migration failed!"
        exit 1
    fi
    
    # 6. uncordon
    kubectl uncordon $node
    
    echo "  $node migrated to v2 successfully"
    
    # 7. 等 5 分鐘觀察新 Pod 跑得 OK
    sleep 300
done

echo "Fleet migration complete!"
05 · Pitfalls

常見陷阱

陷阱 1 · Hybrid mode 誤以為已遷移

RHEL 8/9 預設 v1，但 systemd 在 /sys/fs/cgroup/unified 額外 mount 了 v2——應用看到 cgroup2 mount 誤以為 v2 in use，但實際上 controller 都還在 v1。檢查：stat -fc %T /sys/fs/cgroup/ 必須回 cgroup2fs，不是 看是否有 cgroup2 mount。

陷阱 2 · 切到 v2 後 libcgroup-tools 命令 silent fail

cgcreate -g cpu:test 在 v2 看似成功實際沒效果（v2 controllers 是統一啟用）。修法：移除所有 cgconfig.conf 與 libcgroup 依賴，全部改 systemd unit。長期：考慮 uninstall libcgroup-tools 強制清乾淨。

陷阱 3 · K8s 1.36 升級後 v1 節點 kubelet 拒啟動

升 K8s 到 1.36，v1 節點 kubelet 退出，「cgroup v1 is deprecated and unsupported」。Node 變 NotReady、Pod 全部 evict。修法：(1) 緊急：kubelet config 設 failCgroupV1: false 暫時繞過；(2) 正解：升所有節點到 v2。不要 一次升 K8s，先確保所有節點 v2 才升 K8s。

陷阱 4 · cgroup driver 不一致導致切換後 K8s 節點不穩

切到 v2 後 kubelet 仍設 cgroupDriver: cgroupfs（預設）但 systemd 是 PID 1——two managers 衝突。Pod 啟動偶發失敗、cAdvisor 資料錯亂。修法：kubelet + containerd 都明確設 systemd driver；K8s 1.36+ 用 KubeletCgroupDriverFromCRI 自動偵測。

陷阱 5 · Slurm 切 v2 後 user 看不到自己的 cgroup

使用者 SSH 到 compute node 看 cat /proc/self/cgroup 顯示 0::/user.slice/...——不是 /slurmstepd.scope/job_NNNN。原因：使用者透過 SSH session 進入 user.slice，不是 透過 sbatch 進到 job cgroup。修法：使用者要在 job script 裡看自己的 cgroup（執行 srun 後）才正確；不是登入 node 隨便 cat。

陷阱 6 · 應用直接 hardcode v1 路徑

應用 code 寫死讀 /sys/fs/cgroup/memory/memory.limit_in_bytes——v2 沒這個 path（是 /sys/fs/cgroup/memory.max）。應用拋 FileNotFoundError 或當 unlimited。修法：寫 helper 同時試 v1 與 v2 路徑（fallback 順序）；或檢查 /sys/fs/cgroup/cgroup.controllers 是否存在判斷版本。

陷阱 7 · 切換後忘記 enable controllers

v2 切換成功，但 cat /sys/fs/cgroup/cgroup.controllers 缺少 io 或 cpuset——舊 kernel 或 systemd 版本。Slurm EnableControllers=yes 可手動 propagate；新版 systemd（240+）會自動處理。檢查：echo "+io +cpuset" > /sys/fs/cgroup/cgroup.subtree_control。

陷阱 8 · 一次切整 fleet 沒 rollback plan

「反正測試環境跑過了」一次切 100 節點——某些應用 production 才暴露 v2 incompatibility，無法快速 rollback。修法：(1) 永遠 分階段（5% → 25% → 100%）；(2) 寫好 GRUB 改回的腳本確保 1 命令 rollback；(3) 留一個 v1 canary node 作緊急回退。

07 · Self-check

自我檢查題

Q1.切換 cgroup v1 ↔ v2 為什麼必須 reboot？能不能 hot swap？
必須 reboot 的根本原因：
  1. systemd PID 1 啟動極早：kernel boot 後第一個 user-space process。它必須立刻 決定走哪個 cgroup 版本（建立 cgroup tree、組織所有後續 process）。
  2. cgroup mount 是「單次決策」：v1 是多 hierarchy（每 controller 一個 mount），v2 是單 hierarchy。同一 controller 不能同時 在兩個 hierarchy。
  3. 所有 process 都進了 cgroup tree：切換意味著要把整個 system 的 process 重新組織——只能在啟動時一次做完。
能不能 hot swap：
  • 理論上：可以「移動 controller」——把一個 controller 從 v1 移到 v2 hierarchy。但需要先 disable 該 controller 在 v1（沒 reference），才能 enable 在 v2。
  • 實務上：strongly discouraged for production use（kernel 文件原話）。原因：disable controller 的過程是非同步的，可能要等很久；且 systemd 不支援這種動態切換。
  • 結論：reboot 是唯一實務 路徑。
對 production 影響：
  • K8s node：drain → reboot → uncordon（5-10 分鐘 downtime）
  • HPC：drain job → reboot → resume（依 cluster size 1-N 小時）
  • Standalone server：reboot 期間 service unavailable
Q2.RHEL 8/9 預設 v1，怎麼確認當前版本？怎麼安全切到 v2？
確認當前版本（最權威）： 
$ stat -fc %T /sys/fs/cgroup/
cgroup2fs   ← v2
tmpfs       ← v1（含 hybrid）

# 配合看 mount 確認
$ mount | grep '^cgroup'
# v1 看到多個 cgroup mount（每個 controller 一個）
# v2 只看到一個 cgroup2 mount
不要靠：
  • 看 /sys/fs/cgroup/unified 是否存在——hybrid mode 下兩者並存。
  • 看 RHEL 版本——版本只決定預設，可能已被改過。
安全切到 v2 步驟：
  1. 前置：確認 kernel ≥ 5.14（RHEL 9 標配）；應用 / runtime 支援 v2。
  2. drain workload：K8s drain node、HPC drain job、application stop。
  3. 改 GRUB：sudo grubby --update-kernel=ALL --args="systemd.unified_cgroup_hierarchy=1"。
  4. 確認 grubby 寫入：sudo grubby --info=DEFAULT | grep cgroup。
  5. reboot。
  6. 啟動後驗證：stat -fc %T /sys/fs/cgroup/ 應為 cgroup2fs。
  7. 重啟 service：Docker / containerd / kubelet / slurmd（很多會自動 detect 新 cgroup version 但保險起見）。
  8. 監控 baseline：throttle、OOM、latency 對比切換前。
  9. 有問題 rollback：sudo grubby --update-kernel=ALL --args="systemd.unified_cgroup_hierarchy=0" + reboot。
RHEL 10 沒選擇：only v2，不需要切換。
Q3.libcgroup（cgconfig.conf、cgrulesengd）為什麼 deprecated？怎麼遷移？
為什麼 deprecated：
  1. v2 single-writer rule：只能一個 manager。systemd 是 PID 1 已經是 manager；libcgroup 直接寫 cgroup 介面變成第二個 writer，違反規則。
  2. cgrulesengd（自動分類 daemon） 在 v2 完全不工作——v2 沒有「自動依規則 classify」概念。
  3. cgconfig.conf 設計面向 v1 多 hierarchy，無法表達 v2 unified hierarchy。
  4. Red Hat 官方：「strongly recommends migrate from libcgroup style to systemd style」。
遷移對應表：
libcgroupsystemd 對應
cgconfig.confsystemd unit drop-in
cgcreate -g cpu:foosystemd-run --slice=foo.slice
cgexec -g cpu:foo cmdsystemd-run --scope --slice=foo.slice cmd
cgclassify (移動 PID)echo PID > cgroup.procs（直接）
cgrulesengd（自動分類）沒等價，要應用自己進對的 cgroup
cgset -r cpu.shares=512systemctl set-property foo.slice CPUWeight=50
cpu.shares → CPUWeight 換算：v1 default 1024、v2 default 100。weight = 1 + ((shares - 2) × 9999) / 262142。實務上 1024 → 39，512 → 20。 cgrulesengd 沒等價的處理：應用啟動時自己進對的 cgroup（透過 systemd unit 的 Slice= 指定）。
Q4.K8s 1.36 拒絕 v1 啟動的 failCgroupV1 是什麼？該怎麼處理？
failCgroupV1 是什麼：K8s 1.36 引入的 kubelet config 選項。
  • failCgroupV1: true（預設）：偵測到 v1 → kubelet 直接 exit、Node NotReady。
  • failCgroupV1: false：仍允許 v1 但 log warning。
K8s 官方原話：「cgroup v1 is deprecated and unsupported」。設 false「operationally dangerous」——失去 Memory QoS、PSI 等 v2-only 功能；K8s 後續版本會完全移除。 該怎麼處理：
  1. 正解：升級 v1 節點到 v2。這是 K8s 1.36+ 的事實要求。
  2. 緊急繞過：kubelet config 加： 
    apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
failCgroupV1: false   # 暫時繞過
    重啟 kubelet。但這是暫時 措施。
  3. 分階段：先升所有節點到 v2，再升 K8s 到 1.36+。不要倒過來——倒過來會大批 node NotReady。
升級順序建議： 
[Phase 1] 升 kernel 到 ≥ 5.8（v2 + Memory QoS 支援）
[Phase 2] 升 containerd 到 ≥ 2.0（K8s 1.36 強制）
[Phase 3] 切 cgroup 到 v2（rolling drain + reboot）
[Phase 4] 升 K8s 到 1.36+（這時所有節點都 v2）
反過來操作會在 Phase 4 大批 NotReady。
Q5.什麼是 cgroup hybrid mode？為什麼不該依賴它？
Hybrid mode：v1 與 v2 同時 mount——v1 在傳統位置（/sys/fs/cgroup/<controller>）、v2 在額外位置（/sys/fs/cgroup/unified）。 誰會 hybrid：
  • RHEL 8/9 預設模式
  • Ubuntu 22.04/24.04 預設（保留 v1 fallback）
  • Ubuntu 26.04+ 移除 hybrid，純 v2
看起來的樣子： 
$ mount | grep cgroup
tmpfs on /sys/fs/cgroup type tmpfs
cgroup on /sys/fs/cgroup/systemd type cgroup
cgroup on /sys/fs/cgroup/cpu,cpuacct type cgroup
cgroup on /sys/fs/cgroup/memory type cgroup
... (10+ v1 mounts)
cgroup2 on /sys/fs/cgroup/unified type cgroup2  ← v2 在這
為什麼不該依賴：
  1. v2 unified mount 沒 controllers：所有 controller 都被 v1 occupied。cat /sys/fs/cgroup/unified/cgroup.controllers 是空的。意思是 v2 mount 在但沒功能。
  2. 應用混亂：應用同時看到兩個 mount，可能讀錯或寫錯。
  3. K8s / Slurm 都忽略 hybrid 的 v2：認為是 v1 模式。
  4. misleading 的「好像在用 v2」假象：管理員以為已遷移，實際沒。
怎麼確認：stat -fc %T /sys/fs/cgroup/。回 cgroup2fs 才是真正 v2；回 tmpfs 就是 v1 或 hybrid（無論有沒有 unified mount）。 結論：明確選擇——要嘛 v1 要嘛 v2，不要靠 hybrid。
Q6.遷移完成後怎麼驗證所有 component 都 work？
分層驗證 checklist： 
[Layer 1: kernel / mount]
1. stat -fc %T /sys/fs/cgroup/  → cgroup2fs
2. cat /sys/fs/cgroup/cgroup.controllers
   應該有 cpu, memory, io, pids 等
3. cat /sys/fs/cgroup/cgroup.subtree_control
   確認 controller 已 propagate

[Layer 2: systemd]
4. systemctl --version | head -1  → systemd 245+
5. systemd-cgls / | head -20
   階層清楚（system.slice / user.slice 都在）
6. systemctl status user.slice
   active

[Layer 3: container runtime]

   或 containerd version  → 1.4+
8. docker info --format '{{.CgroupDriver}} / {{.CgroupVersion}}'
   應該是 systemd / 2

[Layer 4: 容器測試]
9. docker run --rm --cpus=1 --memory=500M alpine \
       cat /sys/fs/cgroup/cpu.max /sys/fs/cgroup/memory.max
   100000 100000
   524288000

[Layer 5: K8s（如有）]
10. kubectl get nodes -o jsonpath='{.items[*].status.nodeInfo}' | jq
    確認 kubelet 與 containerd 版本
11. kubectl run test --image=alpine --requests='memory=100Mi' \

    然後看 cgroup 設定

[Layer 6: Slurm（如有）]

Q7.應用 code 直接讀 /sys/fs/cgroup 介面，怎麼寫才能 v1/v2 兼容？
核心策略：先偵測版本，再讀對應介面。 偵測版本： 
def detect_cgroup_version():
    """回傳 'v1', 'v2', 或 None"""
    # v2 有 cgroup.controllers 在 root
    if os.path.exists('/sys/fs/cgroup/cgroup.controllers'):
        return 'v2'
    # v1 有 /sys/fs/cgroup/<controller>/ 目錄
    if os.path.exists('/sys/fs/cgroup/memory'):
        return 'v1'
    return None
取 memory limit： 
def get_memory_limit():
    v = detect_cgroup_version()
    
    if v == 'v2':
        try:
            with open('/sys/fs/cgroup/memory.max') as f:
                val = f.read().strip()
            return None if val == 'max' else int(val)
        except IOError:
            return None
    
    elif v == 'v1':
        try:
            with open('/sys/fs/cgroup/memory/memory.limit_in_bytes') as f:
                val = int(f.read().strip())
            # v1 unlimited 是個極大值
            return None if val > 2**62 else val
        except IOError:
            return None
    
    return None
取 CPU quota： 
def get_cpu_quota():
    v = detect_cgroup_version()
    
    if v == 'v2':
        try:
            with open('/sys/fs/cgroup/cpu.max') as f:
                parts = f.read().split()
            if parts[0] == 'max':
                return None
            quota, period = int(parts[0]), int(parts[1])
            return quota / period
        except IOError:
            return None
    
    elif v == 'v1':
        try:
            with open('/sys/fs/cgroup/cpu/cpu.cfs_quota_us') as f:
                quota = int(f.read())
            if quota <= 0:
                return None
            with open('/sys/fs/cgroup/cpu/cpu.cfs_period_us') as f:
                period = int(f.read())
            return quota / period
        except IOError:
            return None
    
    return None
注意點：
  1. cgroup namespace：容器內看到的可能是 namespaced view。
  2. process 可能在 nested cgroup：讀 root 不一定對，要讀 /proc/self/cgroup 找自己 path 後讀。
  3. v2 介面名變了：memory.max（不是 memory.limit_in_bytes）；cpu.max（不是 cpu.cfs_quota_us / period_us 兩檔）。
