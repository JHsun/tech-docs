容器引擎整合

容器是 cgroup v2 的頭號使用者——但很少人知道一個 docker run 背後有三層工具鏈： OCI runtime（runc / crun）執行容器、container engine（podman / docker）管理生命週期、 CRI runtime（containerd / CRI-O）連結 Kubernetes。每一層都要正確操作 cgroup， single-writer rule 要求它們選一致的 cgroup driver——選錯整個 stack 會出事。 本章拆解三層架構，比較 runc vs crun（C vs Go 的性能差），講解 systemd vs cgroupfs driver、 OCI spec 怎麼對應 cgroup 介面，最後講 rootless container 的特殊挑戰與 CLONE_INTO_CGROUP 機制。

01 · Highlights

本章重點

  1. 三層工具鏈：OCI runtime（執行）→ container engine（管理）→ CRI runtime（K8s 介面）。每層都要操作 cgroup，必須選同一個 cgroup driver 否則 single-writer rule 衝突。
  2. systemd vs cgroupfs driver 是關鍵選擇：systemd-managed 系統必須用 systemd driver。cgroupfs driver 在系統有 systemd 時會造成 two managers 衝突——「resource pressure 下節點不穩定」是經典症狀。
  3. runc vs crun：兩者都是 OCI 標準實作。runc 是 Go 寫的、crun 是 C 寫的（podman 預設）。crun 啟動延遲低 37%、記憶體少、原生 sd-notify 支援，但功能等價。
  4. CLONE_INTO_CGROUP 是現代 runtime 的核心（kernel 5.7+）：fork 時直接指定 cgroup，原子操作——傳統「先 fork 再 echo PID」會有短暫 race window。
  5. OCI spec 的 linux.resources 精確對應 cgroup v2 介面：cpu.shares → cpu.weight、memory.limit → memory.max、pids.limit → pids.max。理解對應關係讓你能 debug 任何 OCI runtime。

03 · Core Concepts

核心內容

3.1 容器三層工具鏈

一個 docker run 或 podman run 背後實際發生了什麼？看似簡單，其實穿過三層工具：

圖 16.1 · 容器三層工具鏈架構

每層的具體職責：

  • Layer 3 - container engine：podman、docker、containerd 處理使用者指令、image pull、storage driver、network plugin。不直接 fork process。
  • Layer 2 - OCI runtime：runc、crun。讀 config.json（OCI spec）、fork+exec 容器 process、建立 namespaces、套用 cgroup limit。直接操作 cgroup 介面。
  • Layer 1 - kernel：cgroup controllers 限制 CPU / memory / IO；namespaces 提供隔離；seccomp 限制 syscall。

3.2 OCI runtime：runc vs crun

runc 與 crun 都實作 OCI Runtime Specification，可互換。但實作語言與架構不同：

選擇建議：

  • 新部署：用 crun。低延遲、低記憶體、原生 sd-notify。
  • 已用 docker / containerd：用 runc（已預設）。
  • 大規模 K8s：crun 在啟動延遲與記憶體上的優勢顯著（每節點數百容器時記憶體節省 GB 級）。
  • 等價性：功能完全互換。OCI spec 嚴格定義行為。

3.4 cgroup driver：systemd vs cgroupfs

cgroup driver 決定 runtime 怎麼創建與管理 cgroup：

  • cgroupfs driver：直接 mkdir /sys/fs/cgroup/...、寫介面檔案。最直接但繞過 systemd。
  • systemd driver：透過 D-Bus 呼叫 systemd API，請 systemd 建 transient unit（scope）。

systemd 系統必須用 systemd driver。Kubernetes 官方原話：「nodes that are configured to use cgroupfs for the kubelet and container runtime, but use systemd for the rest of the processes become unstable under resource pressure.」

為什麼會不穩定？回到 single-writer rule：

  1. systemd 是 PID 1，擁有 root cgroup 管理權。
  2. cgroupfs driver 跳過 systemd 直接 mkdir——成了第二個 writer。
  3. systemd 看到陌生 cgroup 在它的領域，可能清掉、改設定、跟它衝突。
  4. resource pressure 下兩個 manager 對「誰負責 reclaim」沒共識——unpredictable behavior。

3.5 各 runtime 的 cgroup driver 設定

# Docker：/etc/docker/daemon.json
{
  "exec-opts": ["native.cgroupdriver=systemd"]
}

# containerd 2.x：/etc/containerd/config.toml
[plugins.'io.containerd.cri.v1.runtime'.containerd.runtimes.runc.options]
  SystemdCgroup = true

# Podman：/etc/containers/containers.conf
[engine]
cgroup_manager = "systemd"

# CRI-O：/etc/crio/crio.conf
[crio.runtime]
cgroup_manager = "systemd"
conmon_cgroup = "system.slice"

# kubelet：/var/lib/kubelet/config.yaml
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
cgroupDriver: systemd

K8s 1.36+ 自動偵測：kubelet 透過 RuntimeConfig CRI RPC 從 container runtime 取 cgroup driver——不需手動設。但 containerd 必須 ≥ 2.0。1.37 將移除 fallback，舊 containerd 會 break。

3.6 CLONE_INTO_CGROUP：原子 cgroup 加入

kernel 5.7+ 引入的關鍵 syscall flag。容器啟動時的 cgroup 操作從「兩步」變成「一步原子」。

傳統做法（race-prone）：

// Step 1: fork
pid_t pid = fork();
if (pid == 0) {
    // child 在 parent cgroup 跑（暫時！）
    sleep(1);  // 假設 parent 還沒寫 cgroup.procs
    exec(...);
}
// Step 2: parent 把 child 寫進 target cgroup
write_file("/sys/fs/cgroup/myapp/cgroup.procs", pid);

問題：步驟 1 與步驟 2 之間有 window，child 在parent 的 cgroup 跑——可能短暫違反限制、計量錯亂、被錯誤的 cgroup 監控。

CLONE_INTO_CGROUP 解法：fork 時直接指定 cgroup，原子進入。

struct clone_args args = {
    .flags = CLONE_INTO_CGROUP | CLONE_NEWUSER | ...,
    .cgroup = cgroup_fd,  // 已經 open 好的 cgroup fd
};
pid_t pid = clone3(&args, sizeof(args));
// child 從第一個 instruction 就在 target cgroup

採用情況：runc 1.1+、crun 0.18+、podman 4.x+、systemd 245+。現代 container 啟動應該看到 CLONE_INTO_CGROUP 在用。

3.7 Rootless container 的特殊挑戰

Rootless container 讓非 root user 跑容器。在 cgroup v1 時代幾乎不可能——v1 controllers 需 root 權限。v2 透過 delegation 機制讓 rootless 可行。

關鍵機制：systemd 對 user@UID.service 預設 Delegate=memory pids，每個 user session 取得一個 delegated 子樹。

# 看 user systemd 是否被 delegated
getfattr -n user.delegate /sys/fs/cgroup/user.slice/user-$UID.slice/user@$UID.service
# user.delegate="1"  ← 是

# 看可用 controllers
cat /sys/fs/cgroup/user.slice/user-$UID.slice/user@$UID.service/cgroup.controllers
# 可能是 "memory pids"  ← 預設不含 io、cpuset

常見限制：

  • 預設 delegated controllers 不含 io、cpuset。
  • 結果：rootless podman --cpus=1 失敗，「cpu.max not found」。
  • 修法：drop-in 加 Delegate=memory pids cpu io cpuset。

3.8 觀察 container 的 cgroup

怎麼從容器 ID 找 cgroup path：

# Docker（systemd driver）
CONTAINER_ID=$(docker inspect <name> --format '{{.Id}}')
CG="/sys/fs/cgroup/system.slice/docker-${CONTAINER_ID}.scope"

# Podman rootful
CG="/sys/fs/cgroup/machine.slice/libpod-${CONTAINER_ID}.scope"

# Podman rootless
CG="/sys/fs/cgroup/user.slice/user-${UID}.slice/.../libpod-${CONTAINER_ID}.scope"

# K8s（QoS 影響 path）
POD_UID=$(kubectl get pod <name> -o jsonpath='{.metadata.uid}')
# Guaranteed:
CG="/sys/fs/cgroup/kubepods.slice/kubepods-pod${POD_UID//-/_}.slice/"
# Burstable:
CG="/sys/fs/cgroup/kubepods.slice/kubepods-burstable.slice/.../"

# 通用方法（從容器內 process 反查）
CONTAINER_PID=$(docker inspect <name> --format '{{.State.Pid}}')
cat /proc/$CONTAINER_PID/cgroup
# 0::/system.slice/docker-XXX.scope

3.9 Conmon 與容器監控

Podman / CRI-O 有個元件叫 conmon——「container monitor」。每個容器一個 conmon process，負責：

  • 持有容器 stdout/stderr fd 寫到 log
  • PID 1 死掉時通知 engine
  • Detached 模式下做為容器的「守護者」

conmon 的 cgroup 位置很重要：

  • conmon_cgroup = "pod"：放在 pod cgroup 內，計入 pod 資源——透明但會吃容器配額。
  • conmon_cgroup = "system.slice"（systemd 預設）：放系統 slice，不吃容器配額。
  • cgroupfs 模式下強制 "pod"，systemd 模式下兩個都行。

3.10 Best practices 總結

  • 新部署選 crun + systemd driver：性能最好、整合最緊。
  • cgroup driver 全 stack 一致：kubelet、container runtime 必須相同。
  • 檢查 CLONE_INTO_CGROUP 在用：strace 看 clone3。沒在用表示 runtime 太舊。
  • Rootless 加 Delegate：給 user@.service 加 cpu / io / cpuset delegation。
  • conmon_cgroup 看你需求：嚴格 accounting 用 "pod"，避免 noise 用 "system.slice"。
  • 定期升級 runtime：cgroup v2 整合一直在進化（K8s 1.34/35/36）。

05 · Pitfalls

常見陷阱

陷阱 1 · kubelet 與 container runtime 用不同 cgroup driver

kubelet 設 cgroupDriver: systemd、containerd 設 SystemdCgroup = false——兩個 manager 衝突。症狀：Pod 啟動偶發失敗、resource pressure 下節點不穩定、cAdvisor 取資源資料錯亂。修法：兩邊都設 systemd。K8s 1.36+ 用 KubeletCgroupDriverFromCRI 自動偵測。

陷阱 2 · rootless podman 設 --cpus 失敗

錯誤：cpu.max: no such file or directory。原因：systemd 對 user@.service 預設不 delegate cpu。修法：drop-in Delegate=memory pids cpu io cpuset 並重新登入。

陷阱 3 · --cgroup-parent 寫錯格式

cgroupfs driver 期待 raw path、systemd driver 期待 unit 名稱。systemd driver 下用 --cgroup-parent=myapp.slice（unit name with .slice）。

陷阱 4 · 容器內看 host 的 cgroup 路徑

容器內 cat /sys/fs/cgroup/... 看到的多半是容器自己的 cgroup，因為現代 runtime 啟用 cgroup namespace。容器內看 /proc/self/cgroup 通常顯示 "0::/" 而非 host 路徑。

陷阱 5 · K8s container 設 limit 但實際沒生效

Pod yaml 寫 resources.limits.memory: 1Gi 但容器內看 memory.max 是 max？檢查 Pod yaml 的 resources，limits 與 requests 都要設。Memory QoS 看 K8s 版本（1.34+ Beta）。

陷阱 6 · CLONE_INTO_CGROUP 不可用

舊 runc（< 1.1）跑在新 kernel 上——不會用 CLONE_INTO_CGROUP，啟動有 race window。大規模啟動場景（Job batch、autoscaling）累積延遲明顯。修法：升 runtime。strace 確認 clone3。

陷阱 7 · 切換 cgroup driver 不重啟舊 Pod

Node 切換 driver 後，舊 Pod 仍在舊 cgroup tree。症狀：cAdvisor 顯示一半 Pod、kubelet 操作失敗。修法：切換 driver 必須 drain node、重啟 kubelet/runtime、重新 schedule 所有 Pod。

陷阱 8 · conmon_cgroup="pod" 計量混淆

Pod memory.current 比 application 真實使用多 50MB？conmon 在 Pod cgroup 內吃配額。修法：conmon_cgroup = "system.slice"（systemd driver 才能）。或容量規劃 reserve 50MB。

07 · Self-check

自我檢查題

Q1.容器三層工具鏈各自職責？OCI runtime 在 cgroup 整合中扮演什麼角色？
三層工具鏈職責：
  • Layer 3 - container engine：使用者介面層，處理 image、生命週期、network、storage。
  • Layer 2 - OCI runtime：執行層。讀 OCI spec、fork+exec、建 namespaces、設 cgroup。直接操作 cgroup 介面。
  • Layer 1 - kernel：實際 enforce。
OCI runtime 的核心角色：是唯一直接寫 cgroup 介面的層。具體做：
  1. 讀 config.json 的 linux.resources。
  2. 翻譯 OCI 抽象（cpu.shares、memory.limit）成 v2 介面值（cpu.weight、memory.max）。
  3. 用 systemd driver 或 cgroupfs driver 建立 cgroup。
  4. 用 CLONE_INTO_CGROUP 原子地把容器 process 放進 cgroup。
為什麼分層重要：debug 時要知道哪一層出問題。介面值不對是 OCI runtime（layer 2）；計量看不到多半是 driver 不一致。
Q2.systemd vs cgroupfs cgroup driver 差別？為什麼 systemd 系統必須用 systemd driver？
差別：
  • cgroupfs driver：runtime 直接 mkdir + 寫介面檔案。繞過 systemd。
  • systemd driver：透過 D-Bus 呼叫 systemd 建 transient unit。systemd 處理實際 cgroup 操作。
systemd 系統必須用 systemd driver 的原因：
  1. single-writer rule（v2 強制）：每個 cgroup 只能一個 manager。
  2. systemd 是 PID 1，擁有 root cgroup 管理權。
  3. cgroupfs driver 跳過 systemd 直接 mkdir → 變成第二個 writer。
  4. systemd 看到陌生 cgroup 可能清掉、改設定、衝突。
  5. resource pressure 下兩個 manager 對 reclaim 沒共識——不可預測行為。
K8s 官方原話：「nodes that are configured to use cgroupfs ... become unstable under resource pressure.」 例外：嵌入式系統、無 systemd host（Alpine 用 OpenRC）——這時 cgroupfs 是對的。
Q3.runc 與 crun 主要差別？什麼情況該選 crun？
差別：
  • 實作語言：runc Go、crun C
  • 啟動延遲：crun 比 runc 低約 37%
  • 記憶體 footprint：runc ~30 MB、crun ~3 MB
  • sd-notify v3：crun 1.14+ 原生；runc 透過 proxy
  • 預設於：runc 在 docker / containerd 2.x；crun 在 podman / OpenShift
選 crun 的場景：
  1. 大規模 K8s 節點：每節點數百容器時記憶體節省 GB 級。
  2. 高頻啟動：CI/CD batch、auto-scaling。
  3. 低資源環境：edge node、IoT。
  4. 需要 sd-notify v3 原生支援。
關鍵：功能等價。OCI 標準嚴格定義行為，兩者表現一致。差別只在性能與啟動延遲。
Q4.OCI spec 的 cpu.shares 怎麼對應 cgroup v2 cpu.weight？是 1:1 嗎？
不是 1:1。需要轉換：
  • OCI cpu.shares：v1 風格，預設 1024，範圍 2–262144
  • cgroup v2 cpu.weight：預設 100，範圍 1–10000
轉換公式：weight = (1 + ((shares - 2) * 9999) / 262142)
  • shares=1024（OCI default）→ weight=39（比 v2 default 100 低）
  • shares=2048 → weight=78
  • shares=512 → weight=20
實務影響：
  1. 寫 cpu.shares=1024 期望「標準權重」——v2 上其實 weight=39，比 default 低。
  2. v1 → v2 遷移時 cpu.shares 行為會變。
  3. 新部署直接用 v2 native cpu.weight（systemd CPUWeight=）避免歧義。
類似不對等：cpu.quota+period → cpu.max（直接組合）；memory.disableOOMKiller → 無對應；blockIO.weight → io.weight（需 enable blk-iocost）。
Q5.CLONE_INTO_CGROUP 解決什麼問題？沒這個 flag 會發生什麼？
解決問題：原子地把新 fork process 放進指定 cgroup，避免「fork 後在錯 cgroup 跑一段時間」的 race。 傳統做法（race-prone）：fork → child 在 parent cgroup 跑 → parent 寫 cgroup.procs 把 child 搬家。step 1 與 step 2 之間 child 已執行任意 code。 問題：
  • 違反 limit：window 內 child 不被限制。
  • 計量錯亂：window 內資源算到 parent cgroup。
  • 觀測誤導：parent cgroup 出現奇怪的短時 spike。
  • 容器 escape 風險：理論上 attacker 可以在 window 內嘗試 escape。
CLONE_INTO_CGROUP 解法（kernel 5.7+）：fork 時透過 clone3 syscall flag 直接指定 cgroup fd。child 從第一個 instruction 就在 target cgroup。 採用：runc 1.1+、crun 0.18+、podman 4.x+、systemd 245+。 檢查：strace -e clone3 podman run ... 看到 CLONE_INTO_CGROUP flag 就是有用。
Q6.Rootless container 在 v2 為什麼可行？v1 為什麼不行？
v1 為什麼不行：
  1. v1 controllers 大多需 root 權限。
  2. v1 沒完整 delegation 機制——non-root 無法擁有自己的 cgroup 子樹。
  3. 結果：rootless 在 v1 只能沒有 resource limit。
v2 為什麼可行：
  1. delegation 機制：v2 設計時就考慮 sub-tree 委派。
  2. systemd 整合：systemd 246+ 對 user@UID.service 預設 Delegate=memory pids。
  3. nsdelegate mount option：user namespace 內看不到祖先限制（host 仍 enforce）。
  4. 權限模型：non-root 可在 delegated 子樹內自由 mkdir、寫部分介面。
當前限制：預設 delegated controllers 不含 io、cpuset。rootless podman --cpus=1 預設失敗。修法：對 user@.service 設 Delegate=memory pids cpu io cpuset。 建議：rootless 適合 dev / lab。production 大規模仍以 rootful（K8s）為主。
Q7.怎麼從容器 ID 找到對應的 cgroup path？docker / podman / kubernetes 各自格式？
Docker（systemd driver）： 
CONTAINER_ID=$(docker inspect <name> --format '{{.Id}}')
CG="/sys/fs/cgroup/system.slice/docker-${CONTAINER_ID}.scope"
Podman rootful： 
CG="/sys/fs/cgroup/machine.slice/libpod-${CONTAINER_ID}.scope"
Podman rootless： 
CG="/sys/fs/cgroup/user.slice/user-${UID}.slice/.../libpod-${CONTAINER_ID}.scope"
Kubernetes（QoS 影響 path）： 
POD_UID=$(kubectl get pod <name> -o jsonpath='{.metadata.uid}')
# Guaranteed:
CG="/sys/fs/cgroup/kubepods.slice/kubepods-pod${POD_UID//-/_}.slice/"
# Burstable:
CG="/sys/fs/cgroup/kubepods.slice/kubepods-burstable.slice/.../"
# BestEffort:
CG="/sys/fs/cgroup/kubepods.slice/kubepods-besteffort.slice/.../"
通用方法（從容器內 process）： 
CONTAINER_PID=$(docker inspect <name> --format '{{.State.Pid}}')
cat /proc/$CONTAINER_PID/cgroup
# 0::/system.slice/docker-XXX.scope
cgroupfs driver：path 像 /sys/fs/cgroup/docker/<ID>/（沒 .scope 後綴）。
Q8.實作思考題：K8s cluster Pod 啟動偶發失敗，懷疑 cgroup driver 問題。請設計完整診斷與修復流程。
診斷流程： 
[Phase 1: 確認 driver 一致性]
1. kubelet driver: grep cgroupDriver /var/lib/kubelet/config.yaml
2. containerd driver: grep SystemdCgroup /etc/containerd/config.toml
3. init system: ps -p 1 -o comm=  # 應該是 systemd
4. 一致性: kubelet=systemd + runtime=systemd + init=systemd → ✓

[Phase 2: 確認 cgroup version]
5. mount | grep cgroup
   - cgroup2fs → v2 ✓
6. 看 controllers: cat /sys/fs/cgroup/cgroup.controllers

[Phase 3: 看 Pod 啟動失敗訊息]
7. kubectl describe pod failing-pod | tail -50
8. kubelet log: journalctl -u kubelet -n 100 | grep -i error
9. runtime log: journalctl -u containerd -n 100

[Phase 4: 確認 runtime 版本]
10. runc/crun/containerd 版本
11. K8s 1.36+ 需 containerd 2.0+ for KubeletCgroupDriverFromCRI

[Phase 5: 修復]
12. 不一致時改 containerd: SystemdCgroup = true
13. drain node: kubectl drain <node> --ignore-daemonsets
14. restart kubelet + containerd
15. uncordon: kubectl uncordon <node>

[Phase 6: 防護]
16. 寫 monitoring：alert on driver mismatch / kubelet restart
17. Runbook：把流程寫進 SRE 文件
18. CI/CD：部署前 audit driver 一致性
常見根因排序：
  1. kubelet 與 runtime driver 不一致（最常見）
  2. runtime 版本太舊，沒 CLONE_INTO_CGROUP
  3. rootless 模式 controller 沒 delegate
  4. cgroup v1/v2 混亂（hybrid mount）
  5. K8s 與 runtime 版本不匹配
