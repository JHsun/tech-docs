PSI 與事件監控

過去 30 年，Linux 用 load average 衡量系統忙碌——但這個指標分不清「忙」與「卡住」。 PSI（Pressure Stall Information，kernel 4.20+）首次提供「系統因資源不足而 stall 多少時間」的量化指標， 徹底改變了 Linux 的觀測方法論。本章把 PSI 講透——三維度（CPU/Memory/IO）與兩層（some/full）、 poll-based trigger 機制、systemd-oomd 的決策邏輯、生產監控建議。

~6,800 字 閱讀時間 26 分鐘 難度：核心 最後更新 2026-04-28
01 · Highlights

本章重點

  1. PSI 解決 load average 的根本問題：load average 把「busy」與「stall」混為一談——CPU 100% 跑滿 vs CPU 100% 等 IO，load 數字差不多，但意義天差地別。PSI 直接量化「等待時間」。
  2. 三維度 × 兩層次：cpu / memory / io 三個資源，每個有 some（至少一個 task stalled）與 full（所有 task 都 stalled）兩個層次。共 6 個關鍵指標。
  3. per-cgroup PSI 是 v2 獨有：v1 沒有 cgroup-level PSI，只能看全機。v2 每個 cgroup 都有 cpu.pressure / memory.pressure / io.pressure，per-container 觀測得以實現。
  4. poll-based trigger：用 write() 設定閾值 + poll(POLLPRI) 阻塞等待——當壓力超過閾值才喚醒，零 polling overhead。systemd-oomd、oomd 都靠這個。
  5. cgroup.events 補完監控全景：PSI 看「慢」，cgroup.events 看「事件」（freeze、populated、frozen）。memory.events / pids.events / io.stat 各有獨立事件流。組合監控才完整。

03 · Core Concepts

核心內容

3.1 為什麼 load average 不夠用

Unix 的 load average 自 1970s 沿用至今——一個取在 1/5/15 分鐘的「runnable + uninterruptible task 移動平均」。問題：它分不清兩種完全不同的狀況：

  • 系統忙（busy）：CPU 100% 跑滿、所有 core 都在工作——應用 throughput 滿載但正常。
  • 系統卡（stalled）：CPU 沒滿但 task 在等 IO / memory reclaim / 鎖——應用 throughput 嚴重下降，使用者感受到 latency。

兩者 load average 數字可能相同，但意義天差地別。 對 SRE / DevOps 而言，真正關心的是「使用者多慢」而非「機器多忙」。PSI 直接量化前者。

3.2 PSI 的核心定義

kernel 文件原話：「pressure stall information ... measures the time tasks spend stalled on a resource」。 PSI 把「task 因等資源而 stall 的時間」轉成百分比（占 wall-clock time），分三個資源（cpu / memory / io）、兩個層次（some / full）：

層次定義意義
some 至少有一個 task 在 stall 等資源的時間百分比 「有人卡住」——使用者已感受到 latency
full 所有 runnable task 都 stalled 的時間百分比 「整個 cgroup 動不了」——嚴重資源飢荒

full 一定 ≤ some。「所有都 stalled」必然包含「至少一個 stalled」。 full 是 some 的子集，且通常遠小於 some——除非系統真的飢荒，否則 full 多半是 0。

3.3 PSI 介面格式

三個介面檔案：cpu.pressure、memory.pressure、io.pressure，格式統一： 

$ cat /sys/fs/cgroup/myapp/memory.pressure
some avg10=0.12 avg60=0.08 avg300=0.05 total=12345678
full avg10=0.00 avg60=0.00 avg300=0.00 total=0

欄位解讀：

  • avg10 / avg60 / avg300：過去 10 秒、1 分鐘、5 分鐘的 stalled 時間百分比（指數移動平均）。
  • total：累計 stalled 時間，微秒。用來偵測短暫 spike——如果 avg 都是 0 但 total 在快速增長，表示有秒級 stall 但被平均稀釋了。

細節差別：

  • cpu.pressure 沒有 full（早期 kernel）。kernel 6.x+ 加入了 cpu full（雖然 CPU full stall 罕見）。
  • memory.pressure、io.pressure 都有完整 some + full。

3.4 怎麼解讀 PSI 數字

新手常見問題：「some avg10=2.5 算高嗎？」沒有絕對標準，但有實務經驗值：

some avg10解讀建議行動
< 1% 極健康，無壓力 正常運作
1–5% 偶有壓力，仍可接受 觀察趨勢，不需介入
5–10% 壓力明顯，latency 受影響 容量規劃、檢查是否近 limit
10–25% 嚴重壓力，可能影響 SLA 立即介入：擴容、降載、調 limit
> 25% 飢荒，systemd-oomd 可能介入 緊急處理

full 的標準更嚴：

  • full avg10 > 1% — 已經很糟，多半 imminent OOM
  • full avg10 > 5% — 整個 cgroup 卡死，立即行動
  • memory full avg10 > 5% — Netdata 文件指出這是「imminent OOM kill 的強烈訊號」

3.5 poll-based PSI trigger 機制

這是 PSI 的殺手鐧——你不需要 polling，可以「等」。 kernel 提供 trigger 機制：對 pressure file 寫入閾值，用 poll() 阻塞，當壓力超過閾值才喚醒。

trigger 寫入格式：$LEVEL $WINDOW $THRESHOLD

  • $LEVEL：some 或 full
  • $WINDOW：時間窗（微秒），最少 500ms（500000）、最多 10 秒（10000000）
  • $THRESHOLD：在這個 window 內 stall 多久就觸發（微秒）

範例：「memory some 在 1 秒內 stall 超過 150ms 就喚醒我」： 

// 完整範例（C）
#include <fcntl.h>
#include <poll.h>
#include <unistd.h>
#include <string.h>

int main() {
    const char trig[] = "some 150000 1000000";  // 150ms in 1s window
    struct pollfd fds;
    
    fds.fd = open("/sys/fs/cgroup/myapp/memory.pressure",
                  O_RDWR | O_NONBLOCK);
    fds.events = POLLPRI;
    
    write(fds.fd, trig, strlen(trig));
    
    while (1) {
        if (poll(&fds, 1, -1) < 0) break;
        if (fds.revents & POLLPRI) {
            // 觸發！執行 mitigation 邏輯
            handle_memory_pressure();
        }
    }
    return 0;
}

為什麼這個機制重要：

  • 零 polling overhead：傳統做法是每秒 cat 檔案——浪費 CPU，且 1 秒內的 spike 會錯過。
  • 即時反應：壓力一達到閾值立刻 wake——比 polling 早 1 秒能反應。
  • 正是 systemd-oomd 的工作機制：用 PSI trigger 監控所有 cgroup，達到閾值才介入。

3.6 systemd-oomd 內部運作

systemd-oomd 是 systemd 244+ 的 user-space OOM 守護程序，取代傳統 earlyoom，行為完全基於 PSI。

核心邏輯：

  1. 讀取設定（OOMSwapKillPercent、OOMMemoryKillPercent 等）
  2. 對指定 cgroup 設定 PSI trigger
  3. 等待 trigger 觸發
  4. 觸發時掃描子 cgroup，找最該被殺的目標（基於 oom_score_adj、memory.current、swap usage）
  5. 用 cgroup.kill（v2 新介面，atomic kill 整個 subtree）殺掉目標
  6. 記錄 journal log 並繼續監控

系統管理員設定：systemd unit 加 ManagedOOMSwap=kill 或 ManagedOOMMemoryPressure=kill： 

# 對 user.slice 啟用 oomd 監控
sudo systemctl edit user.slice
# 加入：
[Slice]
ManagedOOMMemoryPressure=kill
ManagedOOMMemoryPressureLimit=50%
# 當 user.slice 的 memory.pressure full avg10 > 50% 持續一段時間，
# systemd-oomd 自動殺最佔記憶體的子 cgroup

為什麼比 kernel OOM 好：

  • 更早介入：在 OOM 之前先 throttle 階段就動手，避免 process 真的 OOM 死掉。
  • 選擇更好：可以根據 cgroup 結構（如「殺 user-1.slice 整個 user session」）做整批決策，而非 kernel 的「殺 oom_score 最高的單一 process」。
  • policy 可調：在 unit 層級設定，不像 kernel OOM 全機統一行為。

3.7 cgroup.events 與其他事件介面

PSI 看「慢」，但 cgroup 還有事件系列檔案——記錄具體發生什麼：

檔案欄位意義
cgroup.events populated 0 / 1：cgroup 內是否有 process
frozen 0 / 1：是否被 cgroup.freeze 凍結
memory.events low 從受保護額度被 reclaim 的次數
high用量超過 memory.high 後 throttle 次數
max用量超過 memory.max 次數
oom觸發 OOM 條件次數
oom_kill實際被殺的 process 數
oom_group_kill因 oom.group=1 整批被殺次數
pids.eventsmax達到 pids.max 上限次數
cpu.stat (含事件) nr_throttled cpu.max throttle 次數
throttled_usecthrottle 累計時間
nr_bursts使用 cpu.max.burst 次數

事件監聽：inotify

cgroup events 檔案支援 inotify 監聽——檔案變動時 wake： 

# 監聽 memory.events 變化（每次 OOM 等事件都會喚醒）
inotifywait -m /sys/fs/cgroup/myapp/memory.events |
while read path event filename; do
    echo "Event! Reading new values..."
    cat /sys/fs/cgroup/myapp/memory.events
done

實務上 systemd-oomd、cAdvisor、Prometheus node_exporter 都用 inotify + read 機制收集事件，不用 polling。

3.8 PSI vs 傳統指標的對應

傳統指標限制PSI 對應
load average 分不清 busy 與 stalled cpu.pressure some
%CPU usage 看不到 CPU 等待 cpu.pressure some
memory usage / RSS 數字接近 limit 不一定是壓力 memory.pressure some/full
iostat %util 裝置忙不等於應用慢 io.pressure some/full
vmstat si/so 看 swap activity 但不看 latency 影響 memory.pressure full

建議轉變思維：在 cgroup v2 環境，把 PSI 當第一指標，傳統指標當輔助。 PSI 直接反映「使用者感受到的 latency」，這才是 SRE 真正該關心的。

3.9 cgroup.pressure：開關 PSI 計算

kernel 6.x+ 引入的per-cgroup 開關。寫 0 關閉、寫 1 啟用，預設 1： 

# 對深層中間 cgroup 關掉 PSI 計算（減少 overhead）
echo 0 > /sys/fs/cgroup/system.slice/myservice.service/middle-layer/cgroup.pressure

為什麼需要：PSI 對每個 cgroup 獨立計算並逐層 aggregate。深層 cgroup hierarchy（10+ 層）下，PSI overhead 變得不可忽略。 對中間層 cgroup 通常不關心 PSI（你關心 leaf 與 root），可以關掉中間層計算。

非 hierarchical：在 A 關掉不影響 child B、C。每個 cgroup 獨立決定。

生產環境推薦三層監控架構：

圖 12.1 · 生產環境 PSI + events 三層監控架構

  • Layer 1 處理「急性」問題（OOM、severe stall）——sub-second 反應
  • Layer 2 處理「慢性」趨勢（pressure 開始上升、events 累積）——分鐘級告警

Lab 2 · 寫 PSI trigger Python 監控
#!/usr/bin/env python3
# psi_monitor.py
# 等待 memory pressure 超過閾值才介入

import os
import select

CGROUP = "/sys/fs/cgroup/user.slice"
TRIGGER = "some 150000 1000000"  # 1 秒內 stall 超 150ms

# 開啟 pressure 檔案
fd = os.open(f"{CGROUP}/memory.pressure", os.O_RDWR | os.O_NONBLOCK)

# 註冊 trigger
os.write(fd, TRIGGER.encode())

# 用 poll 等待
poller = select.poll()
poller.register(fd, select.POLLPRI)

print(f"Monitoring {CGROUP}/memory.pressure with trigger: {TRIGGER}")
while True:
    events = poller.poll()
    for f, ev in events:
        if ev & select.POLLPRI:
            # 觸發！讀當前值
            os.lseek(fd, 0, 0)
            content = os.read(fd, 1024).decode()
            print(f"=== TRIGGERED ===\n{content}")
            # 在這裡執行 mitigation 邏輯
            # 例如：通知監控系統、暫停 batch jobs

跑這個 script 然後跑 Lab 1 的記憶體壓力測試，會看到 trigger 觸發。

Lab 4 · 寫 cgroup.events 監聽 daemon
#!/usr/bin/env python3
# events_monitor.py
# 監聽所有 service 的 memory.events，事件變化就 alert

import os
import time
import inotify_simple as inotify  # pip install inotify_simple

WATCH_BASE = "/sys/fs/cgroup/system.slice"

i = inotify.INotify()
watches = {}

# 註冊所有 service 的 memory.events
for service in os.listdir(WATCH_BASE):
    path = os.path.join(WATCH_BASE, service, "memory.events")
    if os.path.exists(path):
        wd = i.add_watch(path, inotify.flags.MODIFY)
        watches[wd] = path

print(f"Monitoring {len(watches)} memory.events files...")

# 記錄初始值
last_values = {p: open(p).read() for p in watches.values()}

while True:
    for event in i.read():
        path = watches[event.wd]
        new = open(path).read()
        old = last_values[path]
        if new != old:
            print(f"=== CHANGE ===")
            print(f"Path: {path}")
            print(f"Old:\n{old}")
            print(f"New:\n{new}")
            last_values[path] = new
Lab 5 · Prometheus PSI metrics 採集設計
# Prometheus node_exporter 預設不 export PSI per-cgroup
# 需要 cAdvisor（Kubernetes 用）或自己寫 exporter

# 用 textfile collector 簡單實作：
cat <<'EOF' > /usr/local/bin/psi_textfile.sh
#!/bin/bash
# 為每個 service 輸出 PSI metrics

OUT=/var/lib/node_exporter/textfile_collector/psi.prom

{
    echo "# HELP psi_pressure_some Pressure some avg10"
    echo "# TYPE psi_pressure_some gauge"
    
    for cg in /sys/fs/cgroup/system.slice/*.service; do
        name=$(basename "$cg" .service)
        for resource in cpu memory io; do
            file="$cg/$resource.pressure"
            [ -f "$file" ] || continue
            avg10=$(grep '^some' "$file" | grep -oP 'avg10=\K[\d.]+')
            echo "psi_pressure_some{service=\"$name\",resource=\"$resource\"} $avg10"
        done
    done
} > "$OUT.tmp"

mv "$OUT.tmp" "$OUT"
EOF

chmod +x /usr/local/bin/psi_textfile.sh

# 設定 cron 每分鐘跑一次
echo "* * * * * /usr/local/bin/psi_textfile.sh" | sudo crontab -

# Prometheus alert rule 範例：
cat <<'EOF'
groups:
- name: psi_alerts
  rules:
  - alert: HighMemoryPressure
    expr: psi_pressure_some{resource="memory"} > 10
    for: 2m
    annotations:
      summary: "{{ $labels.service }} memory pressure > 10%"
EOF
05 · Pitfalls

常見陷阱

陷阱 1 · 把 PSI some 當作 CPU usage 看

看到 cpu.pressure some avg10=2.5，以為「2.5% CPU 利用率」——錯！PSI 不是 utilization 而是 stall time。 2.5% 表示「有人因等 CPU 而卡住的時間占比」。可能 host CPU 才用 30%（沒滿），但有 task 在等 CPU——例如綁了 cpuset 只用 1 核但有多個 task，一直在等。 修法：另外看 cpu.stat 的 usage 欄位才是真實 CPU usage。PSI 與 usage 是不同維度。

陷阱 2 · 只看 avg10、忽略 total 的 spike

avg10、avg60 都顯示低數字（< 1%），以為沒事——但 total 在 1 分鐘內漲了 5 秒（5,000,000 微秒）。 這表示有秒級的嚴重 stall，但被 10/60 秒平均稀釋了。對 latency-critical 應用這已經是大問題。 修法：監控 total 的 delta（每秒增量）。delta 大表示有 spike，要進一步調查。

陷阱 3 · cpu.pressure 沒有 full（舊 kernel）

老 kernel（5.x 之前）的 cpu.pressure 只有 some 沒有 full。寫監控腳本期待 full 欄位會 parse 失敗。 kernel 6.x+ 已加入 cpu full（雖然 CPU full stall 罕見）。 修法：監控腳本要 graceful 處理 missing full 欄位。或檢查 kernel 版本決定要不要採集 cpu full。

陷阱 4 · PSI trigger 的 window 設太小被 reject

嘗試設 some 50000 100000（100ms window，stall 50ms 觸發）——write() 回 EINVAL。 kernel 對 trigger window 有下限 500ms（500000 us）、上限 10 秒（10000000 us）。少於 500ms 會拒絕。 修法：window 至少 500ms。如果需要更快反應，threshold 調低（如 some 50000 500000 = 500ms 內 stall 50ms）。

陷阱 5 · systemd-oomd 沒對應 unit 啟用

啟動了 systemd-oomd，但「怎麼都不殺東西」？ 原因：systemd-oomd 只 監控有 ManagedOOM*=kill directive 的 unit。預設沒任何 unit 啟用，所以 oomd 待著沒事做。 修法：對想被監控的 slice / service 加上 ManagedOOMMemoryPressure=kill 與 ManagedOOMMemoryPressureLimit=。Fedora、Ubuntu 對 user.slice 預設啟用。

陷阱 6 · cgroup.events 用 polling 反應太慢

寫 cron 每分鐘 cat memory.events 算 OOM rate——OOM 已經發生 30 秒才被偵測，太慢。 正確做法：用 inotify 監聽檔案變動，事件發生 ms 級即偵測。實作參考 Lab 4。 systemd-oomd / cAdvisor 等都用 inotify 機制，才能即時反應。

陷阱 7 · 容器內讀 host 的 /proc/pressure 被誤導

在容器內 cat /proc/pressure/memory——數字反映的是 host 整體壓力，不是容器內。 因為 /proc/pressure/ 由 host kernel 提供，沒有 namespace 化。 要看容器自己的 pressure：cat /sys/fs/cgroup/memory.pressure（容器內看到的是容器 cgroup root 的 PSI，已經 namespace 化）。

陷阱 8 · PSI 在深層 hierarchy 帶來 overhead

有些系統（特別是 K8s）cgroup 樹深達 8–10 層。PSI 對每層獨立計算並逐層 aggregate，深層 hierarchy 下 PSI overhead 不可忽略。 某些 benchmark 顯示 8 層 hierarchy 下 PSI 計算占 1-2% CPU。 修法：對中間層（不關心 PSI 的）寫 0 > cgroup.pressure 關掉。kernel 6.x+ 才有這個介面。

練習 4 · 寫 PSI 趨勢分析腳本（15 分鐘）
#!/bin/bash
# psi_trend.sh
# 每 10 秒記錄一次 PSI，每分鐘印出趨勢

CG=${1:-/sys/fs/cgroup}
DURATION=${2:-300}  # 5 分鐘

echo "ts,cpu_some_avg10,memory_some_avg10,io_some_avg10,memory_full_avg10"
for ((i=0; i<$DURATION; i+=10)); do
    ts=$(date +%s)
    cpu=$(grep '^some' "$CG/cpu.pressure" | grep -oP 'avg10=\K[\d.]+')
    mem_s=$(grep '^some' "$CG/memory.pressure" | grep -oP 'avg10=\K[\d.]+')
    mem_f=$(grep '^full' "$CG/memory.pressure" | grep -oP 'avg10=\K[\d.]+')
    io=$(grep '^some' "$CG/io.pressure" | grep -oP 'avg10=\K[\d.]+')
    echo "$ts,$cpu,$mem_s,$io,$mem_f"
    sleep 10
done

跑 5 分鐘收集資料，畫成圖看趨勢。練習從 PSI 數字判斷系統健康。

07 · Self-check

自我檢查題

Q1.PSI 解決 load average 什麼問題？舉一個 load average 誤導但 PSI 正確的場景。
load average 的根本問題：把「busy」（CPU 全滿且都在工作）與「stalled」（task 在等資源）混為一談。兩種狀況的 load 數字相同，但意義天差地別。 誤導場景：機器有 8 核，跑 IO-heavy 應用——4 個 worker 因等 disk read 而 stalled。load average 顯示 4.0，看起來「使用率 50%」沒事；實際上應用 latency 已經很糟。 PSI 怎麼揭露：
  • cpu.pressure some avg10 ≈ 0.5（CPU 很閒）
  • io.pressure some avg10 = 30+（IO 嚴重 stall）
看 PSI 立刻知道是 IO 問題不是 CPU——load average 看不出來。 結論：PSI 告訴你「你的應用有多慢」，load average 告訴你「機器有多忙」。前者是 SRE 真正關心的。
Q2.some 與 full 的差別是什麼？什麼場景該用 full alert 而非 some？
some：至少有一個 task 因等資源而 stall 的時間百分比。代表「有人慢了」——使用者已感受到部分 latency。 full：所有 runnable task 都 stalled 的時間百分比。代表「整個 cgroup 完全卡住」——沒有任何 task 在跑。 full ≤ some：「所有都 stalled」必然包含「至少一個 stalled」。 什麼時候用 full alert：
  • 嚴重程度區分：some 是「warn」、full 是「critical」。例如 some > 10% 是 warn、full > 1% 是 critical page。
  • memory 場景：memory.pressure full 非零是「imminent OOM kill 強烈訊號」（Netdata 文件）。應該觸發立即介入（reduce load、kill non-critical）。
  • IO 場景：io.pressure full > 0 表示整個 cgroup 在等 IO，throughput 為零——比 some 嚴重得多。
cpu 例外：cpu.pressure full 罕見（CPU 很少完全沒人能跑）。多數時候只看 cpu some 即可。
Q3.poll-based PSI trigger 怎麼運作？比 polling 好在哪？
poll-based trigger 機制：
  1. open pressure file（O_RDWR | O_NONBLOCK）
  2. write 閾值字串："some 150000 1000000"（在 1 秒 window 內 stall 超 150ms 就觸發）
  3. poll(POLLPRI) 阻塞等待
  4. kernel 持續監控；當壓力達閾值，wake poll 並回 POLLPRI
  5. userspace handle 該事件、做 mitigation
比 polling 好：
  • 零 polling overhead：傳統「每秒 cat 檔案」浪費 CPU，且無壓力時也在跑。trigger 模式只在實際達閾值才喚醒。
  • 反應速度快：壓力一達閾值立刻 wake，比 polling 早 1 秒能反應。對 OOM 場景這 1 秒可能是「救活還是 OOM」的差別。
  • 單 process 監控多 cgroup：一個 daemon 開多個 fd、用 epoll 同時等多個 trigger，scalable。
限制：window 必須在 [500ms, 10s] 之間。極短窗口（< 500ms）kernel 會 reject。
Q4.systemd-oomd 跟 kernel OOM killer 有什麼不同？哪個應該先生效？
kernel OOM killer：
  • 在 kernel 真的 allocation 失敗（reclaim 都搞不定）才介入
  • 選最高 oom_score 的單一 process 殺
  • 整機統一 policy
systemd-oomd：
  • 在 PSI 顯示嚴重壓力時主動殺（OOM 之前）
  • 用 cgroup 做整批決策（殺整個 user session、整個 service）
  • per-unit policy（systemd directive）
應該先生效的是 systemd-oomd，原因：
  1. 更早介入：在 throttle 階段就動手，不等到 OOM 才反應。
  2. 選擇更聰明：可以「整個 user session 都殺」而非保留半死的 process。
  3. policy 可調：對 user.slice 與 system.slice 用不同策略。
協作關係：systemd-oomd 先試圖優雅處理；如果還是失控（記憶體飆漲速度太快），kernel OOM 作為最後防線。兩者互補，不衝突。
Q5.cgroup.events、memory.events、cpu.stat 三個事件介面分別記錄什麼？
  • cgroup.events：cgroup 整體狀態事件。
    • populated：是否有 process
    • frozen：是否被 cgroup.freeze 凍結
  • memory.events：記憶體相關閾值觸發事件。
    • low：從受保護額度 reclaim
    • high：超過 memory.high 後 throttle
    • max、oom、oom_kill、oom_group_kill
  • cpu.stat：CPU 統計（包含事件）。
    • nr_throttled：cpu.max throttle 次數
    • throttled_usec：累計 throttle 時間
監控建議：
  • 用 inotify 監聽 events 檔案，事件發生即刻反應
  • combine PSI（看「慢」）+ events（看「事件」）= 完整觀測
  • 把這些 metrics 推到 Prometheus，alert rule 要包含 events delta（如 oom_kill rate > 0/min）
Q6.cgroup.pressure 介面是做什麼的？什麼情況該關掉某個 cgroup 的 PSI？
cgroup.pressure 介面（kernel 6.x+）：per-cgroup 的 PSI 計算開關。寫 1 啟用、寫 0 關閉，預設 1。 為什麼需要這個開關：PSI 對每個 cgroup 獨立計算並逐層 aggregate。深層 hierarchy（10+ 層）下，PSI 計算 overhead 變得不可忽略——某些 benchmark 顯示 8 層下占 1–2% CPU。 該關掉 PSI 的場景：
  • 中間層 cgroup：你關心 root 與 leaf 的 PSI，不關心 user-1.slice / user@1.service 這種中間層。可關掉中間層減 overhead。
  • 高密度 K8s 節點：跑幾百個 Pod 時，多層 cgroup 的 PSI 累積 overhead 顯著。
  • 性能敏感的 latency-critical workload：每個 us 都計較，關掉非必要的 PSI 計算。
非 hierarchical：在 A 關掉不影響 child B、C。每個 cgroup 獨立決定。可以「葉子層繼續算、中間層不算」。
Q7.容器內 cat /proc/pressure/memory 看到的數字代表什麼？怎麼看容器自己的 pressure？
/proc/pressure/memory：由 host kernel 提供，沒有 namespace 化。容器內 cat 看到的是 host 整體 memory pressure，不是容器自己。 為什麼這樣：/proc/pressure/* 是 system-wide 介面，從 PSI 引入時就是這樣設計。namespace 化會破壞 host 視角的 monitoring 工具。 看容器自己的 pressure： 
cat /sys/fs/cgroup/memory.pressure
在 cgroup namespace 化的容器內，/sys/fs/cgroup 看到的是容器 cgroup root——這個 root 的 memory.pressure 就是容器自己的 PSI。 實務影響：
  • 容器內監控應用：用 /sys/fs/cgroup/*.pressure
  • K8s sidecar / DaemonSet 採集 host 指標：直接讀 /proc/pressure/*（這時候要的就是 host 視角）
  • cAdvisor、node_exporter 等工具會兩個都讀，分別 export per-container 與 per-node metrics
Q8.實作思考題：你管理一個 K8s cluster，要設計完整 PSI 監控與告警系統，從 Pod 級到 cluster 級。請描述完整架構。
三層監控架構： 
[Layer 1: Node-local 即時防禦]
- 每個 node 跑 systemd-oomd
- 對 user.slice、system.slice 設 ManagedOOMMemoryPressure=kill
  ManagedOOMMemoryPressureLimit=50%
- 反應時間 < 1 秒，避免 kernel OOM

[Layer 2: 監控告警]
- DaemonSet 跑 cAdvisor 採集所有 cgroup PSI + events
- Prometheus 抓 cAdvisor metrics（每 15 秒）
- 關鍵 alert rules:
  * container_pressure_memory_full_avg10 > 1% for 2m → critical
  * container_pressure_memory_some_avg10 > 10% for 5m → warning
  * container_pressure_io_some_avg10 > 20% for 5m → warning
  * rate(container_oom_events[5m]) > 0 → critical

[Layer 3: 容量規劃]
- Long-term Prometheus retention (30 天)
- VPA（Vertical Pod Autoscaler）讀 PSI history 自動調整 limits
- HPA（Horizontal Pod Autoscaler）配合 PSI 而非只看 CPU usage
- 每週 review report：哪些 service 的 PSI 趨勢上升？該擴容嗎？
關鍵設計決策：
  1. Layer 1 用 systemd-oomd 而非 K8s eviction：oomd 反應更快（< 1s vs eviction 30s+）。
  2. Layer 2 alert 用 PSI 而非 memory.current：PSI 直接反映 latency 影響，比 utilization 更可靠。
  3. Layer 2 區分 some / full：full 是 critical（imminent failure）、some 是 warning。
  4. Layer 3 整合 VPA/HPA：用 PSI 自動 tune，比靜態 limit 更靈活。K8s 1.34+ 支援 PSI-based autoscaling。
  5. 多維度監控：cpu / memory / io 三個都看，不要只盯 memory。
Runbook：alert 觸發時，從 PSI + memory.events + memory.stat 三方資料判斷是 leak、流量尖峰、還是 limit 設太緊。
