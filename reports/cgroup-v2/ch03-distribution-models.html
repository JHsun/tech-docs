Part I · 基礎篇 · Chapter 03

資源分配三模型

cgroup v2 把所有 controllers 統一到三個資源分配模型：Weights、Limits、Protections。 這三個模型不是擇一使用，而是互補組合。 理解何時用哪一個，是寫好 controller 配置的核心能力。

~5,500 字 閱讀時間 20 分鐘 難度：核心 最後更新 2026-04-28
01 · Highlights

本章重點

  1. 四種模型，三種主流：Weights（比例分配）、Limits（硬上限）、Protections（保護下限），加上特殊的 Allocations（exclusive 配額，只用於 realtime）。
  2. 命名規範：{controller}.weight（範圍 1–10000，預設 100）、{controller}.max（硬上限）、{controller}.high（軟上限）、{controller}.min（硬保護）、{controller}.low（軟保護）。
  3. 競爭場景才生效：Weights 與 Protections 只在「有資源競爭」時起作用，沒競爭時誰都可以全速跑；Limits 永遠生效。
  4. 三模型互補：實務上常見組合是「Protection（min/low）+ Limit（max/high）」，給保護下限與保護上限，中間用 Weight 做動態分配。
  5. Kubernetes 對應：requests → memory.min、limits → memory.max、Burstable QoS 觸發 memory.high；CPU 部分 requests → cpu.weight、limits → cpu.max。
02 · Latest

最新動態（2025–2026）

2025-09 Kubernetes 1.34 PSI metrics 升 Beta，配合 memory.high 軟上限的觀測能力，讓 throttling 在轉成 OOM 之前就能被監控。
2026-04 Kubernetes Memory QoS 在 v1.36 持續擴展：Burstable Pod 預設用 memory.high 做 throttle，Guaranteed Pod 不設 memory.high 確保不會早期 throttle，Best-effort Pod 完全不做 memory protection。
2025 memory_recursiveprot 在多數新 distro 預設開啟，讓 memory.min / memory.low 自動套用到整個子樹，不再需要手動向下傳遞。Ubuntu 22.04+、RHEL 9+、Fedora 31+ 都已預設啟用。
2025 cpu.weight.nice（kernel 5.16+）在主流 distro 已普遍可用，提供與 nice 值相容的 CPU weight 設定方式（範圍 -20 到 +19）。

2025 年起，Memory Protection（memory.min / memory.low）已從「進階功能」變成「主流預設」。 Kubernetes 在 Burstable QoS class 預設啟用 memory.high，systemd-oomd 用 PSI + memory.high 做 proactive kill， 各家 distro 預設啟用 memory_recursiveprot。本章介紹的三模型已經是現代 Linux 資源管理的事實標準。

03 · Core Concepts

核心內容

3.1 為什麼要有「模型」這個概念

cgroup v1 的 controllers 是各做各的：CPU 用 cpu.shares、memory 用 memory.limit_in_bytes、blkio 用 blkio.weight，每個 controller 自己決定怎麼分配資源、用什麼介面、特殊值是什麼。 這對使用者很不友善——每接一個新 controller 就要重學一遍規則。

v2 一開始就規範：controllers 必須選擇一種或多種「資源分配模型」，並依照模型統一介面命名與行為。 kernel 文件 §3 "Resource Distribution Models" 是這套規範的權威來源。共有四種模型：Weights、Limits、Protections、Allocations。 後三者比較直觀，第一個（Weights）需要花一點力氣理解。

3.2 Model 1: Weights（比例分配）

核心概念：父 cgroup 的資源依照子 cgroup 的 weight 「按比例」分給它們。

kernel 文件原文：「parent's resource is distributed by adding up the weights of all active children and giving each the fraction matching the ratio of its weight against the sum.」

舉例：parent 有 3 個 child，weight 分別是 100、200、700，總和是 1000。 當三個 child 都「想要」資源時（active），它們會分到 10% / 20% / 70% 的 parent 資源。

v2 規範所有 weight 介面命名為 {controller}.weight，範圍 1–10000，預設 100。 這個範圍不是隨便挑的——文件解釋：「chosen to allow enough and symmetric bias in both directions while keeping it intuitive」。 以 100 為中心，向上向下都有兩個數量級的調整空間。

Weights 的關鍵特性

  • 只在競爭時生效：如果只有一個 child 在用 CPU，它會吃滿，不管 weight 是多少。Weight=100 並不意味著「最多用 100 單位」。
  • active children 才參與計算：閒置（idle）的 child 不會分到資源（也不會浪費比例）。
  • 可超額（over-commit）：所有 child weight 加總可以遠遠超過 parent 的 weight——比例分配天然支援這點。
  • 適合 stateless 資源：CPU 時間、IO bandwidth 這類「用過就消失」的資源最適合 weight 模型。

支援 Weights 的 controllers

  • cpu.weight — CPU 時間比例分配（取代 v1 的 cpu.shares）
  • cpu.weight.nice — 用 nice 值（-20 到 +19）的形式設定 weight，方便從傳統 nice 思維過渡
  • io.weight — IO bandwidth 比例分配

3.3 Model 2: Limits（硬性上限）

核心概念：cgroup 最多能用多少資源，超過就會被卡住或殺掉。

Limits 又分兩個子類型：

  • 絕對上限（max）：{controller}.max，超過就觸發強制行為（OOM、被 throttle 到 0）。例如 memory.max=1G 表示一旦用超過 1GB，OOM killer 就會在這個 cgroup 內挑 process 殺掉。
  • 盡力上限（high）：{controller}.high，超過後 kernel 會「盡力」拉回去（throttle、reclaim），但不會直接殺。實務上這是更安全的軟剎車。

Limits 可以over-commit——所有 child 的 limit 加總可以超過 parent 的 limit。這是合理的，因為不一定所有 child 同時跑滿。

支援 Limits 的 controllers

  • memory.max + memory.high — 記憶體硬上限與軟上限
  • memory.swap.max — Swap 上限
  • cpu.max — CPU bandwidth 上限（格式：$MAX $PERIOD，例如 50000 100000 代表每 100ms 最多用 50ms）
  • cpu.max.burst — 允許短時間突破 cpu.max 的 burst 容量
  • io.max — IO bandwidth 與 IOPS 上限（per device）
  • pids.max — Process 數量上限

3.4 Model 3: Protections（保護下限）

核心概念：當系統有資源壓力時，盡量不要從這個 cgroup 拿資源，先去搶別人的。

Protections 是 v2 引入的全新概念，v1 完全沒有。它解決一個現實問題：在 over-commit 的系統上，當記憶體吃緊時，kernel 怎麼決定要從哪裡 reclaim？ v1 的答案是 LRU + memory pressure，沒有 cgroup 層級的保護。v2 補上了這塊。

和 Limits 一樣，Protections 也分兩個子類型：

  • 硬保護（min）：{controller}.min，受保護的部分絕對不會被 reclaim。例如 memory.min=512M 保證這個 cgroup 永遠擁有 512MB 不被回收的記憶體。
  • 軟保護（low）：{controller}.low，受保護部分盡量不被 reclaim，但在系統真的撐不住時還是可以動。

Protections 的關鍵特性

  • 有條件生效：cgroup 受保護到 min 額度，前提是所有祖先 cgroup 也都在它們的 min 額度以下。如果祖先已經吃超 min，保護失效。
  • 可 over-commit：children 的 min 加總可以超過 parent 能給的，這時只有「parent 能提供的部分」會被保護。
  • memory_recursiveprot mount option（kernel 5.7+，現在多數 distro 預設開啟）：讓 min/low 自動沿著子樹遞迴傳播，不需要每個 child 重設。

支援 Protections 的 controllers

  • memory.min + memory.low — Memory 硬保護與軟保護
  • io.cost.qos — 部分 QoS 機制（有限支援，仍在演進中）

3.5 Model 4: Allocations（exclusive 配額）

Allocations 是少數場景用的模型——把固定數量的資源 exclusive 配給某個 cgroup，其他人不能用。 最典型的例子是 realtime CPU bandwidth（cpu.max 在 SCHED_FIFO/SCHED_RR 排程下的行為），給 RT process 預留 CPU。

除了 realtime 場景，一般 server / container workload 幾乎不會用到 Allocations。 本書只在這裡簡略介紹，不深入展開。

3.6 三模型對照速覽

模型 介面命名 競爭時行為 無競爭時 典型應用
Weights weight 按比例分配 active children 誰想用就讓誰用 CPU 時間、IO bandwidth（stateless 資源）
Limits (max) max 絕對上限，超過 OOM/throttle 仍然是上限 記憶體硬上限、CPU quota、PID 上限
Limits (high) high 盡力拉回，throttle 而不殺 仍然是軟剎車 Memory 軟上限、Burstable Pod throttling
Protections (min) min 絕不被 reclaim（硬保證） 沒競爭時與此無關 Guaranteed Pod、關鍵服務
Protections (low) low 盡量不 reclaim（best effort） 沒競爭時與此無關 重要但非關鍵的服務
Allocations 各種 exclusive 配額 已被佔用 Realtime CPU bandwidth

3.8 三模型互補：Kubernetes Memory QoS 案例

Kubernetes 1.27+ 的 Memory QoS 是三模型互補使用的經典範例。它依 Pod QoS class 決定 cgroup 設定：

QoS Class memory.min memory.high memory.max
Guaranteed
request = limit		 = request 未設定（不 throttle） = limit
Burstable
request < limit		 = request = request + 0.9 × (limit − request) = limit
BestEffort
沒設 request 與 limit		 未設定 未設定 未設定（被 node allocatable 限制）

這套設計的妙處：

  • Guaranteed Pod 同時用 Protection 與 Limit：min 保證它至少有 request 的記憶體不被 reclaim；max 限制它不能超過 limit。但故意不設 high，避免在還沒到 limit 之前就先被 throttle，影響 latency。
  • Burstable Pod 三模型全用：min 給 request 的保護，high 在到達 90% (limit−request) 處設軟剎車（防止接近 OOM 才反應），max 是最後防線。
  • BestEffort Pod 完全不保護：當系統有壓力時，這類 Pod 是第一批被 reclaim 與 evict 的對象。

理解這個設計，你就掌握了三模型互補的精髓。後續章節會更深入到每個 controller 的細節，但思考框架不離這三個模型。

04 · Hands-on

實戰範例：三模型實作體驗

下面三個 lab 分別示範三種模型的實際行為。所有 lab 使用 systemd-run，避免直接動 cgroup 介面而與 systemd 衝突。

Lab 1 · Weights 體驗：兩個無限迴圈搶 CPU

啟動兩個 100% CPU 的 process，但給它們不同的 CPUWeight，觀察實際分配比例：

# 啟動 weight=100 的 worker
sudo systemd-run --slice=user.slice --unit=worker-light \
    -p CPUWeight=100 \
    bash -c 'while :; do :; done'

# 啟動 weight=400 的 worker（4 倍權重）
sudo systemd-run --slice=user.slice --unit=worker-heavy \
    -p CPUWeight=400 \
    bash -c 'while :; do :; done'

# 觀察 1 分鐘內的 CPU 使用率（會看到 1:4 的比例）
top -p $(pgrep -f 'while :' | tr '\n' ',' | sed 's/,$//')

# 預期：worker-heavy 約占 80% CPU、worker-light 約占 20%
# 比例 = 400 : 100 = 4 : 1

# 清理
sudo systemctl stop worker-light worker-heavy

觀察點：weight 是比例，不是百分比。weight=100 與 weight=400 在競爭時是 1:4，但如果 worker-light 自己沒事做（idle），worker-heavy 可以吃滿全部 CPU。

Lab 2 · Limits 體驗：memory.max 觸發 OOM

限制一個 process 最多用 50MB 記憶體，跑一個會吃 100MB 的程式：

# 用 systemd-run 限制 memory
sudo systemd-run --slice=user.slice --unit=mem-test \
    -p MemoryMax=50M \
    --pty \
    python3 -c '
data = bytearray(100 * 1024 * 1024)  # 嘗試吃 100MB
print("allocated 100MB")
input("press enter...")
'

# 預期：process 在分配到約 50MB 時被 OOM kill
# 看 journal 確認被 OOM 殺
sudo journalctl -u mem-test --no-pager | tail -20

# 應該看到類似訊息：
# Memory cgroup out of memory: Killed process ...

# 看 cgroup events 統計
cat /sys/fs/cgroup/user.slice/mem-test.service/memory.events
# oom 1
# oom_kill 1

觀察點：memory.events 是觀察 OOM 行為的權威來源。每次觸發都會增加計數，可以用來建告警。

Lab 3 · 軟限 high vs 硬限 max 的差別

用同一個吃記憶體程式，比較 MemoryHigh 與 MemoryMax 的行為：

# 場景 A：只設 high（軟限）
sudo systemd-run --slice=user.slice --unit=mem-soft \
    -p MemoryHigh=50M \
    --pty \
    python3 -c '
import time
data = bytearray(0)
for i in range(20):
    data.extend(bytearray(10 * 1024 * 1024))  # 每次 +10MB
    print(f"allocated {(i+1)*10}MB")
    time.sleep(0.5)
'
# 預期行為：超過 50MB 後 process 被嚴重 throttle（變慢），但不會被殺
# 可在另一個 terminal 觀察 memory.pressure
cat /sys/fs/cgroup/user.slice/mem-soft.service/memory.pressure

# 場景 B：設 max（硬限）
sudo systemd-run --slice=user.slice --unit=mem-hard \
    -p MemoryMax=50M \
    --pty \
    python3 -c '同上'
# 預期行為：超過 50MB 立即 OOM kill

# 清理
sudo systemctl stop mem-soft mem-hard 2>/dev/null

觀察點：軟限會讓程式「痛苦地慢下來」，硬限會直接「送它去死」。軟限適合給有彈性的服務，硬限給絕對不能超的場景（共享主機、多租戶）。

Lab 4 · 觀察 Protection 在系統壓力下的效果

建立兩個 cgroup，A 設 memory.min，B 不設，然後刻意製造記憶體壓力，看 kernel 怎麼選擇 reclaim：

# 啟動 A：吃 100MB，受保護 80MB
sudo systemd-run --slice=user.slice --unit=protected-app \
    -p MemoryMin=80M \
    --pty \
    python3 -c '
data = bytearray(100 * 1024 * 1024)
import time; time.sleep(60)
'

# 啟動 B：吃 100MB，無保護
sudo systemd-run --slice=user.slice --unit=normal-app \
    --pty \
    python3 -c '同上'

# 製造系統 memory pressure（吃 80% 記憶體）
sudo stress-ng --vm 1 --vm-bytes 80% --timeout 30s

# 觀察兩個 cgroup 的 memory.current 變化
watch -n 0.5 '
  echo === protected ===
  cat /sys/fs/cgroup/user.slice/protected-app.service/memory.current
  echo === normal ===
  cat /sys/fs/cgroup/user.slice/normal-app.service/memory.current
'
# 預期：normal-app 的 memory.current 會明顯下降（被 reclaim）
#       protected-app 的 memory.current 維持不變（受 min 保護）

這個 lab 直觀展示 Protection 的價值——在系統壓力下，被保護的 cgroup 是真的「免疫」於 reclaim。

05 · Pitfalls

常見陷阱

陷阱 1 · 把 weight 當成百分比

最常見的誤解：「我設 cpu.weight=50，這個 service 就只能用 50% CPU」。 錯。weight 是比例，且只在競爭時生效。weight=50 的 service 在沒人搶 CPU 時可以吃滿所有核。 要硬上限請用 cpu.max。

陷阱 2 · 只設 max 不設 min，遇到系統壓力被踢爆

很多人配 Kubernetes 時只設 limits 不設 requests（變成 BestEffort），或讓 request << limit。 遇到系統 memory pressure 時，這類 Pod 因為沒有 memory.min 保護，會被優先 reclaim 甚至 evict。 正確做法：關鍵服務一定要設 request（對應 memory.min），給它至少一個保護下限。

陷阱 3 · memory.high 設太低，服務在 OOM 之前就先死於 throttle

memory.high 設定的位置很關鍵。設太低，服務會持續被 reclaim throttle，變得極度緩慢，使用者看到的是「service 還活著但完全沒回應」——比直接 OOM 還難 debug。 Kubernetes Memory QoS 把 high 設在 0.9 × (limit − request)，是經過調校的合理位置，不要隨意縮小。

陷阱 4 · 忘記設 memory_recursiveprot，min/low 在子樹失效

在沒啟用 memory_recursiveprot 的舊系統（kernel < 5.7 或舊 distro），memory.min 在 parent 設定後不會自動傳到 child。 如果你在 kubepods.slice 設 min=8G，但 kubepods.slice/foo-pod 沒設，foo-pod 不會受保護。 檢查方式：mount | grep cgroup2 看是否有 memory_recursiveprot mount option。沒有的話，需要 remount。

陷阱 5 · cpu.max 與 cpu.weight 混用結果出乎意料

cpu.max 與 cpu.weight 可以同時設，但要理解互動：cpu.max 是絕對天花板，cpu.weight 在天花板下決定比例。 例如 cpu.max=50000 100000（50% 一核）、cpu.weight=400，當沒有競爭時 service 也只能用到 50%——weight 大到飛上天也救不了你。 混用時記住：cpu.max 是 Plan B（防爆），cpu.weight 是 Plan A（日常分配）。

動手練習

練習 1 · 用 cpu.weight 模擬 4 個 service 的優先級（10 分鐘）

啟動 4 個 CPU 滿載的 service，weight 分別設 100, 200, 400, 800，觀察 CPU 分配比例：

for i in 1 2 4 8; do
  sudo systemd-run --slice=user.slice --unit=worker-w${i}00 \
    -p CPUWeight=${i}00 \
    bash -c 'while :; do :; done'
done

# 5 秒後觀察
sleep 5
sudo systemd-cgtop --order=cpu --iterations=2 user.slice

回答：

  1. 看到的 CPU 比例是不是接近 1:2:4:8？
  2. 所有 weight 加起來是 1500，但實際 CPU 怎麼分？
  3. 如果你停掉 weight=800 的那個，剩下三個的比例會變嗎？
練習 2 · 設計一個 Burstable Pod 的 cgroup 配置（10 分鐘）

假設一個 Burstable Pod 設定如下：

resources:
  requests:
    memory: 256Mi
  limits:
    memory: 1Gi

請計算它的 cgroup 應該如何設置 memory.min、memory.high、memory.max（依 Kubernetes Memory QoS 公式）：

  1. memory.min = ?
  2. memory.high = ? （提示：request + 0.9 × (limit − request)）
  3. memory.max = ?
  4. 用 systemd-run 模擬一個這樣的 cgroup：sudo systemd-run -p MemoryMin=... -p MemoryHigh=... -p MemoryMax=... ...
練習 3 · 觀察自己系統上的實際配置（5 分鐘）
# 找一個 systemd service（例如 nginx 或 sshd）
SERVICE=sshd.service

# 看它的資源分配設定
systemctl show -p MemoryMin,MemoryHigh,MemoryMax,CPUWeight,IOWeight $SERVICE

# 看實際 cgroup 中的值
CGROUP_PATH=$(systemctl show -p ControlGroup --value $SERVICE)
ls /sys/fs/cgroup$CGROUP_PATH | grep -E '^(memory|cpu|io)\.'
cat /sys/fs/cgroup$CGROUP_PATH/memory.max
cat /sys/fs/cgroup$CGROUP_PATH/cpu.weight

回答：你的系統服務有設 limits 嗎？大多數預設是 infinity 還是 max？這代表什麼？

練習 4 · 用 stress-ng 驗證 Protection 真的有效（15 分鐘）

本章 Lab 4 的延伸版：建立一個有 memory.min 保護的 cgroup，用 stress-ng 製造系統壓力，量化保護效果：

# 啟動 protected
sudo systemd-run --slice=user.slice --unit=p-app \
    -p MemoryMin=200M \
    --pty \
    python3 -c '
data = bytearray(300 * 1024 * 1024)  # 吃 300MB
for i in range(60):
    # 持續 touch 記憶體避免被 swap out
    for j in range(0, len(data), 4096):
        data[j] = 1
    import time; time.sleep(1)
'

# 記錄初始 memory.current
cat /sys/fs/cgroup/user.slice/p-app.service/memory.current

# 在另一 terminal 製造壓力
sudo stress-ng --vm 4 --vm-bytes 90% --timeout 30s

# 30 秒後看 p-app 的 memory.current 變化
watch -n 1 'cat /sys/fs/cgroup/user.slice/p-app.service/memory.current'

# 清理
sudo systemctl stop p-app

回答：

  1. p-app 的 memory.current 在壓力下有沒有下降？
  2. 如果有，下降到約多少就停住了？這對應 memory.min 嗎？
  3. 如果你把 MemoryMin 拿掉重做，行為會怎樣？
07 · Self-check

自我檢查題

Q1.cgroup v2 的四種資源分配模型是哪四種？簡述各自的核心概念。
Weights（比例分配）：父 cgroup 的資源依 child 的 weight 比例分配，只在競爭時生效。介面 {controller}.weight，範圍 1–10000，預設 100。
Limits（硬上限）：cgroup 最多能用多少。max 超過直接 OOM/throttle 到 0；high 超過盡力拉回（軟剎車）。
Protections（保護下限）：當系統壓力大時，盡量不要從這個 cgroup 拿資源。min 是硬保證，low 是 best-effort。
Allocations（exclusive 配額）：固定配給某個 cgroup，其他人不能用。少數場景（如 realtime CPU bandwidth）使用。
Q2.cpu.weight 預設值是多少？範圍是多少？為什麼設計成這個範圍？
預設 100，範圍 1–10000。 kernel 文件解釋：「chosen to allow enough and symmetric bias in both directions while keeping it intuitive」——以 100 為中心向上向下都有 100 倍的調整空間（1 是 100 的 1/100，10000 是 100 倍），對稱且足夠涵蓋各種優先級需求。 100 作為預設讓「未設定」與「設成 100」沒有差異，符合直覺。
Q3.memory.high 與 memory.max 的本質差別是什麼？什麼場景應該用哪一個？
memory.max 是硬上限：超過會觸發 OOM killer 在這個 cgroup 內挑 process 殺掉。
memory.high 是軟剎車：超過會 throttle process，kernel 努力 reclaim 把使用量壓回 high 以下，但不會殺 process。
場景：
  • 需要絕對防爆（多租戶、共享主機、預算控制）→ memory.max
  • 有彈性、優先保活（latency-sensitive 服務）→ memory.high（搭配 memory.max 當最後防線）
Kubernetes Memory QoS 對 Burstable Pod 兩個都設，high 在 0.9×(limit−request)、max 在 limit，是教科書級的搭配。
Q4.Weights 與 Protections 都「只在有競爭時生效」嗎？這句話怎麼正確理解？
是，但意義不同：
  • Weights：沒競爭時，weight 不影響任何事——cgroup 想吃多少就吃多少；有競爭時 weight 決定按比例怎麼分。
  • Protections：沒競爭時，protection 也不影響任何事——所有 cgroup 都拿到自己想要的；有壓力（如 memory pressure 觸發 reclaim）時，min/low 決定誰免於被 reclaim。
關鍵差別：weight 在分配競爭資源，protection 在保護已有資源不被搶走。所以實務上常一起用：weight 處理日常分配、protection 處理意外壓力。
Q5.什麼是 over-commit？哪些模型可以 over-commit？為什麼這是合理設計？
over-commit：children 的設定值加總超過 parent 能提供的資源量。 Limits 與 Protections 都可以 over-commit：
  • Limits over-commit 合理，因為不一定所有 child 同時跑滿。例如 parent 有 4GB 記憶體，10 個 child 各設 max=2GB，總和 20GB，但實際同時頂多有 2-3 個吃滿，整體可控。
  • Protections over-commit 也合理：parent 能提供多少就保護多少。例如 parent 有 4GB，3 個 child 各設 min=2GB（總和 6GB），實際只有 4GB 會被分給三者保護。
Weights 不需要 over-commit 概念，因為 weight 是純比例，加總大小無意義。
Q6.為什麼 Kubernetes 對 Guaranteed Pod 不設 memory.high，但 Burstable Pod 設？
Guaranteed Pod：request = limit，意思是「我說多少就要多少」。設 memory.high 會在到達 limit 之前就 throttle，等於違背 Guaranteed 的承諾——把高優先級服務搞慢。所以只設 max 當最後防線、設 min = request 給保護。 Burstable Pod：request < limit，意思是「我會用 request，但需要時可以 burst 到 limit」。memory.high 設在 0.9×(limit−request) 處，給服務一個「快到 limit 了，慢一點」的軟剎車，避免直接撞到 limit 觸發 OOM。Burstable 設計上接受 throttle，所以這合理。 BestEffort Pod：request 與 limit 都不設，沒有任何承諾，不需要 cgroup 介入；遇到壓力直接被踢。
Q7.memory_recursiveprot mount option 解決什麼問題？怎麼確認是否啟用？
解決什麼：原始設計中，memory.min / memory.low 不會自動傳到子樹。如果你在 parent 設 min=8G，但 child 沒設，child 不受保護。 這在 Kubernetes 之類的多層 cgroup（kubepods.slice → pod → container）中非常麻煩——要在每一層手動傳遞值。 解決方式：memory_recursiveprot（kernel 5.7+）讓 min/low 自動沿子樹遞迴生效。 確認啟用：mount | grep cgroup2，看 mount options 列表是否包含 memory_recursiveprot。多數現代 distro（Ubuntu 22.04+、RHEL 9+、Fedora 31+）預設啟用。 kernel 文件原文：「this should have been the default behavior but is a mount-option to avoid regressing setups relying on the original semantics」——是設計缺憾的補救，但為了相容性沒設成預設。
Q8.實作思考題：你要為一個有狀態的資料庫 service 設計 cgroup 配置，避免它影響同主機的其他服務。請組合三種模型給出完整方案。
典型方案（假設 32GB RAM、8 核機器，要給 DB 預留 16GB / 4 核）：
  • memory.min = 16G（Protection）：DB 至少有 16GB 不被 reclaim，避免 page cache 被踢爆而磁碟 I/O 暴增。
  • memory.high = 24G（軟限）：超過 24G 就 throttle，給其他服務留 8GB 緩衝。
  • memory.max = 28G（硬限）：絕不超過 28G，留 4GB 給 OS。
  • cpu.weight = 800（Weight）：競爭時拿到 80% CPU 比例（其他服務 weight=100）。
  • cpu.max = 600000 100000（Limit）：絕對天花板 6 核（保留 2 核給 OS+其他）。
  • io.weight = 500（Weight）：磁碟 I/O 拿較高優先級，畢竟 DB 是 I/O 密集。
對應的 systemd 設定： 
[Service]
MemoryMin=16G
MemoryHigh=24G
MemoryMax=28G
CPUWeight=800
CPUQuota=600%
IOWeight=500
關鍵思路：三模型互補，protection 保活、limit 防爆、weight 處理日常競爭。
