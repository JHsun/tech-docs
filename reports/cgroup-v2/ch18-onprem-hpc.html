Part VI · 生態系 · Chapter 18

地端與特殊場景

雖然 Kubernetes 主導雲端世界，但 cgroup v2 在非雲端場景 同樣重要。 HPC（high-performance computing）有 Slurm 這個 30 年歷史的 scheduler，整套基礎設施與 K8s 截然不同； 多人共享的開發機 / 跳板機 / 教學機需要 systemd-managed 多租戶隔離；嵌入式 / IoT 系統有資源極限的特殊需求。 本章把這些 Kubernetes 之外的世界一次講完——讓你在任何 Linux 場景都能掌握 cgroup v2 整合。

01 · Highlights

本章重點

  1. HPC 用 Slurm 不用 K8s：Slurm scheduler 在超算、研究機構、ML 訓練 cluster 廣泛使用。設計目標與 K8s 不同——批次作業（jobs）為主、不追求高密度多租戶、追求 MPI / GPU / RDMA 整合。但底層仍用 cgroup v2 做資源限制。
  2. Slurm cgroup/v2 plugin 設計巧妙：透過 systemd D-Bus 建 scope 給 slurmstepd，與 systemd 共存。每個 job / step / task 對應一層 cgroup，slurmd 在 delegated subtree 自由管理。
  3. 共享 workstation 的 systemd 多租戶：開發機、跳板機、研究室共用機。systemd-managed 模式下，每個 user@UID.service 自動取得 delegated cgroup，可設 user-level limit 防止單一 user 拖垮系統。
  4. HPC vs K8s 設計哲學差很大：K8s 服務式、長期跑、scaling；HPC 批次式、有限時間、節點獨佔。理解差異才能選對工具——AI 訓練混合 workload 開始模糊邊界。
  5. 嵌入式 / IoT 用 cgroup v2 的不同方式：openrc、busybox-init 等替代 init 系統不支援 systemd 的 cgroup 整合。需要直接 cgroupfs driver、手動管理 cgroup tree。

03 · Core Concepts

核心內容

3.1 HPC 與 K8s 的設計差異

為什麼 HPC 不用 K8s？因為兩者解決的是不同問題：

面向Kubernetes（雲）Slurm（HPC）
主要 workload長期服務（24/7）批次作業（job 跑完就走）
節點 utilization追求高密度（多 Pod 共享）job 通常獨佔節點
scheduling 維度resource request + label節點數、wall-time、partition、QoS
跨節點通訊K8s service / DNSMPI（PMI / OpenMPI / Intel MPI）
資源類型CPU / memory / extended（GPU）CPU / memory / GPU / RDMA / 特殊網路
檔案系統PV / PVC / CSI共享 NFS / Lustre / GPFS / BeeGFS
使用者模式kubectl / APIsbatch / srun / SSH 登入
容器化必須（Pod = 容器）選用（Singularity / Apptainer）

核心差異：K8s 預設「多租戶共享、彈性 scaling」，Slurm 預設「單一 job 獨佔資源、跑完釋放」。 HPC workload（CFD、量子化學、天氣模擬）通常一個 job 跑數小時到數天、用幾百到幾千個節點——K8s 的高密度 pack 不適合。

但AI / ML 訓練模糊了這條界線——大型訓練任務既需要 HPC 的 RDMA / GPU 整合，又需要 K8s 的彈性管理。2025 年起越來越多研究機構部署「Slurm + K8s 混合」架構。

3.2 Slurm cgroup/v2 plugin 設計

Slurm 從 v22.x 開始支援 cgroup v2，到 v24.x 已成熟穩定。設計上有巨大挑戰——必須與 systemd 的 single-writer rule 共存：

  • 傳統做法（v1 時代）：slurmd 直接 mkdir /sys/fs/cgroup/.../slurm/。
  • v2 不能這樣：systemd 看到陌生目錄可能清掉。違反 single-writer。

Slurm v2 解法：透過 systemd D-Bus 申請 scope——

  1. slurmd 啟動時，systemd 把它放進 slurmd.service 的 cgroup（system.slice 之下）。
  2. job 啟動時，slurmstepd 透過 D-Bus 向 systemd 請求建立新 scope slurmstepd-{job_id}.scope。
  3. systemd 給這個 scope 設定 Delegate=yes——slurmstepd 在 scope 之下自由管理 cgroup。
  4. slurmstepd 在 delegated subtree 建 job/step/task 三層 cgroup。

典型 Slurm cgroup hierarchy

/sys/fs/cgroup/system.slice/slurmstepd.scope/      ← systemd-delegated
└── job_42/                                         ← slurmstepd 建
    ├── step_0/                                     ← slurmstepd 建
    │   ├── task_0/                                 ← per-task
    │   ├── task_1/
    │   └── ...
    └── step_extern/                                ← extern step（IO 等）

為什麼 slurmstepd 在獨立 scope 而非 slurmd 自己的 cgroup：因為 slurmd 自己會被 systemd 重啟（patch、reconfig）。如果 slurmd cgroup 之下有子目錄（job_42 等），slurmd 變 inner node，systemd restart 會失敗（無法把新 slurmd PID 放進 non-leaf）。所以 slurmstepd 用獨立 scope 隔離。

3.3 Slurm cgroup.conf 完整設定

# /etc/slurm/cgroup.conf
CgroupPlugin=cgroup/v2     # 強制 v2，autodetect 也可
CgroupMountpoint=/sys/fs/cgroup
SystemdTimeout=1000        # systemd 啟動 scope 等待時間（毫秒）
SlurmgroupSlice=system.slice  # slurmd 所在 slice

# task/cgroup plugin 限制
ConstrainCores=yes         # 用 cpuset 限定 CPU
ConstrainRAMSpace=yes      # 用 memory.max 限制記憶體
ConstrainSwapSpace=yes
ConstrainDevices=yes       # 用 BPF device controller 限制設備（GPU）
AllowedRAMSpace=100        # 100% 配額
AllowedSwapSpace=0         # 不允許 swap

# v2 額外控制器（給 GPU / RDMA workload）
EnableControllers=io,pids,rdma,hugetlb,misc

關鍵 directive：

  • ConstrainCores=yes：每個 task 透過 cpuset 綁到分配的 CPU。MPI 應用必備。
  • ConstrainRAMSpace=yes：每個 step 設 memory.max。
  • ConstrainDevices=yes：用 BPF device cgroup 限制可見 GPU。配合 --gres=gpu:N 使用。
  • EnableControllers=：v2 額外啟用的 controllers，讓 job 能設 io / rdma / hugetlb 限制。

3.4 Slurm job 啟動的 cgroup 流程

# 用戶提交 job
$ sbatch --nodes=2 --cpus-per-task=4 --mem=8G --gres=gpu:1 my_job.sh

# Slurm 內部流程：
# 1. slurmd 收到 job request
# 2. slurmd 透過 D-Bus 請 systemd 建 slurmstepd-{job_id}.scope
# 3. systemd 建 scope，設 Delegate=yes
# 4. slurmstepd 在 scope 之下建 job_{ID}/ cgroup
# 5. 設 cpuset.cpus（指定 4 個 CPU）
# 6. 設 memory.max=8G
# 7. attach BPF device program 限制 GPU
# 8. 在 job/{ID}/ 之下建 step_0/、task_0/ 等
# 9. fork+exec user 的 my_job.sh，process 進入 task_0/
# 10. job 結束時清理整個 scope 樹

# 看 cgroup 結構
ssh node-001 systemd-cgls /system.slice/slurmstepd.scope
# slurmstepd-42.scope
#   └── job_42/
#       └── step_0/
#           └── task_0/
#               └── 12345 my_job.sh

# 看資源限制
JOB_CG=/sys/fs/cgroup/system.slice/slurmstepd-42.scope/job_42
cat $JOB_CG/memory.max     # 8589934592（8G）
cat $JOB_CG/cpuset.cpus    # 4-7

3.5 共享 workstation 的多租戶隔離

場景：研究室、教室、共用開發機。多個 user SSH 進來工作。痛點：

  • 有人跑 stress test 把整機卡住
  • 有人 fork bomb 把 PID 用光
  • 有人記憶體洩漏觸發 OOM 殺到無辜進程

解法：systemd 的 user.slice 多租戶隔離。 每個 user 登入時，systemd 自動建 user-UID.slice 與 user@UID.service。 對 user.slice 設總體 limit，每個 user 在 slice 下分配。

實戰：限制 user.slice 並啟用 systemd-oomd

# /etc/systemd/system/user-.slice.d/00-resource.conf
# （注意：user-.slice 是 template，匹配所有 user-UID.slice）
[Slice]
# 每個 user 最多 16GB 記憶體
MemoryMax=16G
# 預留 2GB 保證
MemoryMin=2G
# CPU 軟限制（公平分配）
CPUWeight=100
# task 上限
TasksMax=10000
# 啟用 systemd-oomd 監控
ManagedOOMMemoryPressure=kill
ManagedOOMMemoryPressureLimit=50%

# 對整個 user.slice 設總限
sudo systemctl set-property user.slice MemoryMax=128G CPUWeight=500

# 看效果
systemd-cgls /user.slice
# user-1001.slice (alice)
# user-1002.slice (bob)
# user-1003.slice (charlie)
# 每人最多 16G、總共不超過 128G

系統 daemon 的保護：對 system.slice 設 MemoryMin 保證，避免 user 拖垮系統 daemon： 

sudo systemctl set-property system.slice \
    MemoryMin=4G \
    CPUWeight=200

# 即使 user.slice 把記憶體吃光，sshd / dbus 等系統 daemon 仍有 4GB 保證

3.6 跳板機 / bastion host 的特殊考量

跳板機特殊在「許多人短暫 ssh 進來執行少量命令」。設計重點：

  • 每 session 限制嚴格：避免單個 session 影響後續 ssh 登入。
  • 強制 task / fork 限制：防 fork bomb。
  • I/O 限制：防有人 dd 卡住 disk。
  • session timeout：自動清理 idle session。
# /etc/systemd/system/user-.slice.d/00-bastion.conf
[Slice]
# 比一般共享機器更嚴格
MemoryMax=2G
TasksMax=500
CPUQuota=200%       # 最多 2 cores
IOWeight=50         # 較低 I/O 優先級（避免影響其他 session）
ManagedOOMMemoryPressure=kill
ManagedOOMMemoryPressureLimit=70%

# 配合 sshd_config 設 ClientAliveInterval / MaxSessions

3.7 嵌入式 / IoT：non-systemd 環境

嵌入式 Linux（IoT、embedded device）常用替代 init：openrc（Alpine）、busybox init（buildroot）、s6 / runit。 這些 init 不提供 systemd 的 cgroup 整合——必須手動管理 cgroup。 

# busybox init 範例：手動建立 cgroup
# /etc/local.d/cgroup-setup.start
#!/bin/sh
mkdir -p /sys/fs/cgroup/services
mkdir -p /sys/fs/cgroup/services/critical
mkdir -p /sys/fs/cgroup/services/non-critical

# 啟用 controllers
echo "+memory +cpu +pids" > /sys/fs/cgroup/cgroup.subtree_control
echo "+memory +cpu +pids" > /sys/fs/cgroup/services/cgroup.subtree_control

# 設 limits
echo "256M" > /sys/fs/cgroup/services/critical/memory.max
echo "100M" > /sys/fs/cgroup/services/non-critical/memory.max

# 把 process 放進去
echo "$pid_of_critical_app" > /sys/fs/cgroup/services/critical/cgroup.procs
echo "$pid_of_logger" > /sys/fs/cgroup/services/non-critical/cgroup.procs

常見場景：路由器 / NAS / 工業 controller / smart device。資源緊（256MB - 2GB RAM）、process 少（10-30 個）、要求穩定（不能因小應用 leak 拖死整機）。

3.8 HPC + K8s 混合架構（AI / ML 場景）

現代 AI 訓練既要 HPC 的底層性能（RDMA、NVLink、MPI），又要 K8s 的彈性管理（image 部署、自動 scaling）。三種混合模式：

模式架構典型使用場景
K8s + RDMA K8s 跑訓練 Pod，CNI 整合 RDMA 單機多 GPU、中等規模分散式訓練
Slurm + Singularity Slurm 排程，job 跑容器（Singularity / Apptainer） 傳統 HPC 中心轉型用容器
Slurm-on-K8s（新） K8s 提供基礎設施，Slurm 在 Pod 中跑做 batch scheduling 研究機構同時要 web service 與 HPC job

NVIDIA 的 Slurm Nemesis、CSCS 的 Pyxis、HPE 的 Slingshot 整合 等都在嘗試讓 HPC 工作流跑在容器化的雲環境，cgroup v2 是底層共同基礎。

3.9 Best practices

  • HPC node 用 cgroup/v2 plugin：v1 已 deprecated，新 feature 不再加
  • Slurm 啟用 EnableControllers：給 job 完整資源控制（io、rdma、hugetlb）
  • 共享 workstation 設 user.slice limit：每個 user 公平配額 + systemd-oomd 監控
  • 對 system.slice 設 MemoryMin：保護系統 daemon 不被 user 拖垮
  • 跳板機嚴格限制 per-session：MemoryMax、TasksMax、CPUQuota 都要設緊
  • 嵌入式手動管 cgroup：非 systemd 環境就接受手動，但要寫 init script 啟動時建好結構
  • HPC + K8s 混合場景考慮 Slurm-on-K8s：兩者並存而非取代

05 · Pitfalls

常見陷阱

陷阱 1 · Slurm 仍用 cgroup/v1 plugin 在新 distro

升級到 RHEL 9 / Ubuntu 22.04+ 後 Slurm 仍跑 cgroup/v1——但這些 distro 預設 unified v2，混用導致 plugin 找不到 controller。修法：cgroup.conf 改 CgroupPlugin=cgroup/v2，重啟 slurmd。或用 CgroupPlugin=autodetect（v22+ 預設）。

陷阱 2 · 忘了 EnableControllers 啟用 io / rdma

job 想 sbatch --gres=gpu:1 加上 IO 限制——但 ConstrainDevices 有效、IO 限制無效。原因：Slurm v2 預設只啟 cpu / memory / cpuset / pids；io / rdma / hugetlb 要在 cgroup.conf 加 EnableControllers=io,rdma,hugetlb 才會啟用。

陷阱 3 · slurmd 重啟後 job 都消失

修改 cgroup.conf 後 systemctl restart slurmd——所有跑中 job 全消失。原因：slurmd 重啟時 systemd 把它放回 root cgroup，但已經有 slurmstepd 子目錄，違反 leaf rule，slurmd 啟動失敗。修法：升級到新版 Slurm（已修復——slurmstepd 用獨立 scope）。或先 drain node、scancel 所有 job、再重啟 slurmd。

陷阱 4 · user.slice template drop-in 沒生效

建 /etc/systemd/system/user-1001.slice.d/override.conf——但 user 1001 登入後設定沒生效。原因：systemd 對 user.slice 用template instance，drop-in 應放在 /etc/systemd/system/user-.slice.d/（注意 dash 後沒 UID）才會 apply 到所有 user-XXX.slice。修法：用 template 路徑，不寫死 UID。

陷阱 5 · systemd-oomd 在沒 swap 系統行為奇怪

共享機器設了 ManagedOOMSwap=kill 但系統沒 swap——oomd 預期 swap 用量資訊計算 pressure，沒 swap 時可能誤判。修法：沒 swap 系統用 ManagedOOMMemoryPressure 而非 ManagedOOMSwap。或加 small swap 給 oomd 訊號計算。

陷阱 6 · 嵌入式手動 cgroup 沒處理 OOM

嵌入式系統手動建 cgroup 設 memory.max——OOM 時 kernel 殺 process 但整個 service 沒重啟。原因：沒有 systemd 的 OOMPolicy=kill 行為，process 死了 service 就掛了。修法：寫 init script 監控 OOM 事件（inotify memory.events），偵測到 oom_kill 後重啟 service。

陷阱 7 · HPC 容器 mount 共享 filesystem 沒 cgroup namespace

Slurm 配 Singularity 跑容器，容器內 cat /sys/fs/cgroup/... 看到的是 host 路徑而非容器自己的。原因：Singularity 預設不啟用 cgroup namespace（追求性能）。修法：明確 --cgroup-mode 設定，或接受這個 trade-off（HPC 通常是 single-user 場景，namespace 隔離不關鍵）。

陷阱 8 · 教室 / lab 共享機器沒設 PID limit

學生 fork bomb 把整機 PID 用光（kernel 預設 32K）——所有人無法登入。原因：user.slice 沒設 TasksMax。修法：對 user-.slice template 設 TasksMax=5000（或更嚴）。defense in depth 同時設 sysctl kernel.pid_max 與 systemd 限制。

#!/bin/bash
# slurm-cgroup-audit.sh
# 在 HPC node 上跑，看 Slurm cgroup 設定健康度

echo "=== Slurm cgroup plugin ==="
grep -E '^(CgroupPlugin|EnableControllers|Constrain)' /etc/slurm/cgroup.conf

echo "=== current job cgroups ==="
systemd-cgls /system.slice/slurmstepd*.scope 2>/dev/null

echo "=== resource usage per job ==="
for scope in /sys/fs/cgroup/system.slice/slurmstepd-*.scope/; do
    JOB=$(basename "$scope" | sed 's/slurmstepd-//;s/.scope//')
    [ -d "$scope/job_$JOB" ] || continue
    
    MEM=$(cat "$scope/job_$JOB/memory.current" 2>/dev/null)
    MAX=$(cat "$scope/job_$JOB/memory.max" 2>/dev/null)
    CPU=$(cat "$scope/job_$JOB/cpuset.cpus" 2>/dev/null)
    
    echo "Job $JOB: memory=$MEM/$MAX cpuset=$CPU"
done

跑在你的 HPC node，看 Slurm 是否正確設限。

自我檢查題

Q1.HPC 為什麼不用 K8s？Slurm 與 K8s 設計差異核心是什麼？
核心差異：解決不同問題。
  • K8s：服務式、長期運行、多租戶高密度共享、彈性 scaling、有狀態 service。
  • Slurm：批次作業、有限時間、節點獨佔、MPI 跨節點通訊、研究運算。
具體差異：
  1. workload 模式：K8s 跑 24/7 service；Slurm 跑「幾小時跑完就走」的 job。
  2. scheduling：K8s 看 resource request / labels；Slurm 看節點數、wall-time、partition、QoS。
  3. 跨節點通訊：K8s 用 service mesh；Slurm 用 MPI（PMI / OpenMPI）。
  4. 檔案系統：K8s 用 PV / PVC；Slurm 用 NFS / Lustre / GPFS。
  5. 使用者 UX：K8s 用 kubectl；Slurm 用 sbatch / srun + SSH。
cgroup v2 整合差異：兩者都用 cgroup v2，但 hierarchy 結構不同：
  • K8s：kubepods.slice/kubepods-pod{UID}.slice/container-{ID}.scope
  • Slurm：system.slice/slurmstepd-{job}.scope/job_{ID}/step_{N}/task_{M}
AI / ML 模糊邊界：大型訓練既要 HPC 的 RDMA / GPU，又要 K8s 的彈性管理。研究機構開始用混合架構。
Q2.Slurm cgroup/v2 plugin 為什麼透過 D-Bus 建獨立 scope 給 slurmstepd？不直接 mkdir 不行嗎？
不能直接 mkdir 的原因：
  1. 違反 single-writer rule：systemd 是 PID 1，擁有 root cgroup 管理權。slurmd 直接 mkdir 變成第二個 writer，systemd 可能清掉或衝突。
  2. slurmd restart 問題：若 slurmd 在自己 cgroup 之下建 job 子目錄，它的 cgroup 變成 inner node。systemd restart slurmd 時無法把新 PID 放進 non-leaf（違反 leaf rule），啟動失敗。
D-Bus 解法：
  1. 每個 job 啟動時，slurmstepd 透過 D-Bus 向 systemd 請建立新 scope（slurmstepd-{job_id}.scope）。
  2. systemd 建 scope，設 Delegate=yes——slurmstepd 在 scope 之下自由管理 cgroup。
  3. slurmstepd 在 delegated subtree 建 job/step/task 三層 cgroup。
  4. slurmd 自己仍在 system.slice/slurmd.service，不會變 inner node——可以正常 restart。
關鍵設計：分離 slurmd 與 slurmstepd 的 cgroup。slurmd 是 daemon（systemd 完全管），slurmstepd 是 job-specific（自己管 delegated subtree）。 結果：systemd 看不到 slurmstepd 之下的細節（被 delegate 了），但可以隨時 restart slurmd 不影響跑中的 job。
Q3.共享 workstation 的 user.slice template drop-in 怎麼寫？為什麼不能寫 user-1001.slice.d？
正確路徑：/etc/systemd/system/user-.slice.d/override.conf（注意 dash 後沒有 UID）。 為什麼這樣：
  • systemd 對 user-XXX.slice 是 template instance——XXX 是動態的 UID。
  • template drop-in 路徑用 user-.slice.d（dash 後無內容）匹配所有 instance。
  • 寫死 user-1001.slice.d 只對 user 1001 生效，其他 user 不受影響。
範例： 
# /etc/systemd/system/user-.slice.d/00-multi-tenant.conf
[Slice]
MemoryMax=4G
TasksMax=5000
CPUWeight=100
ManagedOOMMemoryPressure=kill
ManagedOOMMemoryPressureLimit=50%
對 user.slice 整體（所有 user 加起來）：用 user.slice（沒 dash）： 
sudo systemctl set-property user.slice MemoryMax=128G CPUWeight=500
per-user 不同設定：對特定 UID 寫死路徑：/etc/systemd/system/user-1001.slice.d/special.conf——優先級高過 template，只對 user 1001 生效。 實務做法：先寫 template 設預設限制，特殊 user 用寫死的 override 加更嚴 / 寬鬆設定。
Q4.跳板機 (bastion) 與一般共享機器在 cgroup 設定上有何不同？
使用模式差異：
  • 跳板機：許多 user 短暫 ssh 登入執行少量命令（看 log、查 ip、跳到內網）。session 短、量大。
  • 共享開發機：少數 user 長期使用、跑訓練 / build / test。session 長、resource 用量大。
cgroup 設定差異：
設定跳板機共享開發機
MemoryMax512M-2G（嚴）16-64G（寬）
TasksMax200-500（嚴）5000-10000（寬）
CPUQuota100-200%（嚴）800%+ 或不設
IOWeight50（低）100（中）
session timeout10-30 分鐘無或數小時
跳板機特殊考量：
  1. 強制 timeout：sshd ClientAliveInterval + IdleAction=kill 雙重保險。
  2. 嚴格 IOWeight：避免有人 dd / rsync 影響其他 session 登入。
  3. per-session 而非 per-user：每個 ssh session 獨立 limit（透過 PAM session）。
  4. tmux / screen 的處理：用 systemd-run --scope 把長期 session 放進獨立 scope 不被 timeout 殺。
Q5.嵌入式 / IoT 系統（無 systemd）怎麼用 cgroup v2？跟 systemd 系統最大差別？
主要做法：手動管理。沒 systemd，必須在 init script 中手動：
  1. 確認 cgroup2 mounted（mount | grep cgroup2）
  2. 啟用 controllers：echo "+memory +cpu +pids" > cgroup.subtree_control
  3. 建子 cgroup：mkdir /sys/fs/cgroup/services/critical
  4. 設 limits：echo "256M" > .../memory.max
  5. 啟動 process 後寫 PID：echo $PID > .../cgroup.procs
跟 systemd 系統最大差別：
  • 無 single-writer 強制：嵌入式 init 簡單，不會像 systemd 那樣積極管 cgroup。
  • 無 OOMPolicy 自動處理：OOM 後 process 死，service 不會自動重啟。需要 init script 自己監控（用 inotify 監 memory.events）。
  • 無 user session 自動隔離：embedded 通常 single-user 或 root，不需要 user.slice。
  • 啟動順序自管：systemd 的 After= / Requires= dependency 變成 init script 順序問題。
  • cgroup driver 強制 cgroupfs：沒 systemd 就沒 systemd driver。
典型場景：路由器、NAS、工業 controller、smart device。資源緊（256MB-2GB RAM）、process 少（10-30）、求穩定（不能因小應用 leak 拖死）。 實作建議：寫 init script 用 inotify 監 memory.events，偵測到 oom_kill 自動重啟對應 service。defense in depth：sysctl pid_max + systemd-style cgroup 限制。
Q6.HPC + K8s 混合架構有哪幾種模式？AI / ML 訓練該選哪個？
三種混合模式：
  1. K8s + RDMA：K8s 跑訓練 Pod，CNI 整合 RDMA（Cilium、Multus + RoCE）。場景：單機多 GPU、中等規模分散式。優點：k8s 生態完整。缺點：MPI 整合不如 Slurm 直接。
  2. Slurm + Singularity / Apptainer：Slurm 排程，job 跑容器。場景：傳統 HPC 中心轉型用容器。優點：保留 HPC 工作流，user 不用學 K8s。缺點：與 K8s 生態（image registry、CI/CD）整合難。
  3. Slurm-on-K8s：K8s 提供基礎設施（運維、監控、GPU operator），Slurm 在 Pod 中跑做 batch scheduling。場景：研究機構同時要 web service 與 HPC job。優點：兼顧兩世界。缺點：複雜度高。
AI / ML 訓練選擇：
  • 大規模分散式（> 100 GPU）：仍以 Slurm 為主——MPI 與 NCCL 整合最直接。Slurm + Pyxis（NVIDIA enroot 整合）是 NVIDIA 推薦。
  • 中規模（< 32 GPU）：K8s + Kubeflow / Volcano 已成熟。MLOps 工具鏈完整（MLflow、Argo Workflows）。
  • 實驗室 / 研究：Slurm + Singularity 適合 user 自由提交 job，K8s 適合需要 web UI / dashboard。
趨勢：2024-2025 大型 AI 公司（OpenAI、Anthropic、Meta）多用 Slurm + 自家 orchestration；雲廠商（AWS、GCP、Azure）推 K8s + GPU operator。融合架構研究持續。 cgroup v2 角度：兩者都用 cgroup v2 做底層資源限制，差別在 scheduler 與工作流——cgroup 整合大致相同。
Q7.實作思考題：你管理一個 50 節點 GPU cluster 給 30 位研究員用，需求是「容易使用、公平分配、防止濫用」。請設計完整方案。
完整方案設計： 
[Phase 1: 選平台]
方案 A: Slurm（傳統 HPC 風）
- 學習曲線低，sbatch / srun 簡單
- MPI / RDMA 整合好
- GPU sharing 用 --gres=gpu:N
- 公平：fair-share + QoS partition

方案 B: K8s + Volcano scheduler
- 學習曲線中，需懂 yaml
- MLOps 工具完整（Kubeflow）
- GPU sharing 用 NVIDIA device plugin
- 公平：ResourceQuota + LimitRange

選擇：Slurm。research user 比較熟、MPI 場景多。

[Phase 2: cgroup v2 設定]
1. 所有節點 cgroup v2（kernel 5.9+）
2. cgroup.conf:
   CgroupPlugin=cgroup/v2
   ConstrainCores=yes
   ConstrainRAMSpace=yes
   ConstrainSwapSpace=yes
   ConstrainDevices=yes
   EnableControllers=io,pids,rdma,hugetlb,misc

3. slurm.conf 設 partition 與 QoS:
   PartitionName=normal Default=YES MaxTime=24:00:00
   PartitionName=long MaxTime=72:00:00 Priority=50
   PartitionName=interactive MaxTime=01:00:00 Priority=200

[Phase 3: 公平分配]
4. 啟用 fair-share scheduling
5. 對每個 user 設 default account 與 GrpTRES limit:
   sacctmgr add user alice DefaultAccount=research_team
   sacctmgr modify user alice set GrpTRESMins=cpu=10000,gres/gpu=2000

6. 對 partition 設 MaxTRESPerUser:
   PartitionName=normal MaxTRESPerUser=gpu=2

[Phase 4: 防濫用]
7. 限制 login node:
   /etc/systemd/system/user-.slice.d/00-login.conf
   [Slice]
   MemoryMax=8G
   TasksMax=2000
   CPUQuota=400%

8. systemd-oomd 對 user.slice 啟用
9. 對 user job 設 wall-time hard limit (--time)

[Phase 5: 監控與審計]
10. Prometheus + slurm-exporter 採集 job metrics
11. Grafana dashboard:
    - per-user GPU hour usage
    - PSI metrics on compute nodes
    - OOM events
12. 月報：哪些 user 用了多少 GPU hour

[Phase 6: 文件與培訓]
13. Wiki 教 sbatch 範例
14. Office hours 解答 user 問題
15. 「fair-share 怎麼運作」說明
16. 寫 troubleshooting playbook（OOM、quota 滿了等）
關鍵設計決策：
  1. 選 Slurm 不選 K8s：30 位 research user 對 yaml 不友善，sbatch 學習成本低。
  2. fair-share 自動公平：不需要 admin 手動分配，根據歷史使用自動 priority。
  3. 分 partition 給不同需求：interactive 短 job 與 long 長 job 不互卡。
  4. login node 嚴格限制：避免有人在 login node 跑 stress / 訓練。
  5. cgroup v2 + EnableControllers：完整資源控制（含 GPU、RDMA）。
常見坑：fair-share 配置複雜（需要 accounting db）；GPU MIG 需要額外設定；user 不熟 sbatch 需要教學。
Q8.實作思考題：你接手一個傳統 HPC cluster（仍用 cgroup v1 plugin），要遷移到 cgroup v2。請設計完整 migration 計劃。
完整 migration 計劃（5 階段）： 
[Phase 1: 前置評估]
1. 盤點現況：
   - kernel 版本（cgroup v2 推薦 5.9+，最低 4.15）
   - distro 版本（RHEL 9+ / Ubuntu 22.04+ 預設 v2）
   - Slurm 版本（v22+ 才有完整 v2 plugin）
   - systemd 版本（246+ 對 user delegation 完整）

2. 識別風險：
   - 跑中的 long job（不能中斷）
   - 自訂 prolog / epilog scripts 是否寫 cgroup 路徑
   - GPU driver / RDMA 整合是否影響

[Phase 2: 升級基礎設施]
3. 升級 distro 到 v2-default（RHEL 9 / Ubuntu 22.04+）
   - 一次升一個節點，drain 後升級
4. 升級 Slurm 到 v22+ 或 v23+
5. 升級 systemd 到 246+
6. 在新節點驗證 cgroup v2 正常 mount:
   mount | grep cgroup2fs

[Phase 3: 試點測試]
7. 選 1-2 個非生產節點當試點
8. 修改 cgroup.conf:
   CgroupPlugin=cgroup/v2  # 或 autodetect
   EnableControllers=io,pids,rdma,hugetlb,misc
   ConstrainCores=yes
   ConstrainRAMSpace=yes
   ConstrainDevices=yes

9. 重啟 slurmd
10. 跑測試 job 驗證:
    - 簡單 sleep job
    - MPI job（多節點）
    - GPU job
    - 大記憶體 job

11. 觀察 cgroup tree:
    systemd-cgls /system.slice/slurmstepd*.scope

[Phase 4: 漸進 rollout]
12. 一次升級一個 partition 的節點
13. 公告 maintenance window
14. drain 節點 → 升級 → 測試 → return to service
15. 重點觀察：
    - PSI metrics 是否正常
    - OOM event 是否上升
    - GPU / RDMA job 性能對比
16. 持續監控 1 週

[Phase 5: 完成與最佳化]
17. 全 cluster 完成 v2 遷移
18. 啟用 v2 新功能：
    - PSI alert 在 controller node
    - cgroup.kill atomic kill
    - memory_recursiveprot mount option
19. 更新文件：
    - user 看到 cgroup 路徑變了（過去寫 v1 path 的腳本要改）
    - admin runbook
20. 訓練 admin team

[防護措施]
- 每階段都有 rollback plan（可改 CgroupPlugin=cgroup/v1）
- 保留 1-2 個 v1 節點當 fallback（直到完全穩定）
- 跑 60 天 burn-in，確認無 regression
關鍵風險點：
  1. Slurm 版本太舊：v21 之前 v2 plugin 不穩，必須升級。
  2. 自訂 prolog / epilog：寫死 v1 cgroup 路徑（如 /sys/fs/cgroup/memory/...）會壞。
  3. GPU driver：NVIDIA driver 對 cgroup v2 device controller（BPF）要求新版本。
  4. 跑中的 long job：可能要 drain 等到 job 結束，long-running 工作流可能超過 maintenance window。
成功指標：所有節點 cgroup v2、Slurm 跑得順、PSI metrics 可採、OOM rate 下降、user 滿意度不變或上升。
