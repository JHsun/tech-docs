Python 整合與容器化開發

Python 是資料科學、ML、自動化的主流語言——但在容器化部署上，比 Go 與 Java 落後一代。 Go 1.25 內建 cgroup-aware GOMAXPROCS、Java 自 8u191/JDK 10 起就有 UseContainerSupport， Python 至今 2026 年仍沒有 runtime-level 的 cgroup awareness—— os.cpu_count()、multiprocessing.cpu_count()、甚至 psutil.cpu_count() 通通看 host CPU。 結果：Dask、autogluon、duckdb 全部踩雷，多 worker 在 limit=2 CPU 容器搶資源被 throttle。 本章拆解 Python 為何特殊、實務 work-around、社群 library 與 Dask 等實際案例、Java JVM 對比，最後指引怎麼正確 部署 Python 到 K8s。

本章重點

03 · Core Concepts

核心內容

3.1 Python 的「三層 CPU count」陷阱

Python 提供至少四個查 CPU 數量的 API，全部都不看 cgroup：

API看的是容器內結果
os.cpu_count()host 邏輯 CPU 數（透過 sysconf）host CPU 數（錯）
multiprocessing.cpu_count()包裝 os.cpu_count()同上（錯）
psutil.cpu_count(logical=True)讀 /proc/cpuinfohost CPU 數（錯）
psutil.cpu_count(logical=False)physical CPU 數host physical CPU（錯）
os.sched_getaffinity(0)process CPU affinity maskcpuset.cpus（部分對）
讀 /sys/fs/cgroup/cpu.maxcgroup CPU quota正確（v2）

關鍵差別：

  • os.sched_getaffinity 看 cpuset.cpus——若用 K8s CPU Manager static policy 把 Pod pin 到特定 CPU，這個會準確。
  • 但不 看 cpu.max（CFS quota）——Pod 設 cpu limit=2 但沒 affinity 時，sched_getaffinity 仍回 host CPU 數量。
  • 大多 K8s Pod 是後一種情境——所以 sched_getaffinity 也不夠。

psutil 文件原話：「cpu_count() may not necessarily be equivalent to the actual number of CPUs the current process can use. That can vary in case process CPU affinity has been changed, Linux cgroups are being used...」——但 psutil 本身仍沒提供「計入 cgroup」的 API。

3.2 完整的 Python helper

實務上需要寫 helper，按優先順序嘗試多個來源。這是事實標準寫法： 

import os
import math


def get_effective_cpu_count() -> int:
    """
    取得當前 process 真正可用的 CPU 數，按優先順序：
    1. 環境變數 CPU_LIMIT 或 OMP_NUM_THREADS（顯式覆寫）
    2. cgroup v2 cpu.max
    3. cgroup v1 cpu.cfs_quota_us / cpu.cfs_period_us
    4. os.sched_getaffinity（cpuset / affinity）
    5. os.cpu_count（最後 fallback）
    """
    # 1. 顯式環境變數
    for var in ("CPU_LIMIT", "OMP_NUM_THREADS", "MKL_NUM_THREADS"):
        if val := os.environ.get(var):
            try:
                n = int(val)
                if n > 0:
                    return n
            except ValueError:
                pass

    # 2. cgroup v2
    try:
        with open("/sys/fs/cgroup/cpu.max") as f:
            content = f.read().strip()
        # "max 100000" = 沒 limit；"200000 100000" = 2 CPU
        parts = content.split()
        if len(parts) == 2 and parts[0] != "max":
            quota, period = int(parts[0]), int(parts[1])
            if quota > 0 and period > 0:
                return max(1, math.ceil(quota / period))
    except (IOError, OSError, ValueError):
        pass

    # 3. cgroup v1
    try:
        with open("/sys/fs/cgroup/cpu/cpu.cfs_quota_us") as f:
            quota = int(f.read().strip())
        with open("/sys/fs/cgroup/cpu/cpu.cfs_period_us") as f:
            period = int(f.read().strip())
        if quota > 0 and period > 0:
            return max(1, math.ceil(quota / period))
    except (IOError, OSError, ValueError):
        pass

    # 4. CPU affinity（cpuset 場景）
    try:
        return len(os.sched_getaffinity(0))
    except (AttributeError, OSError):
        pass

    # 5. 最後 fallback
    return os.cpu_count() or 1


# 在 main 開頭呼叫
if __name__ == "__main__":
    n = get_effective_cpu_count()
    print(f"Effective CPUs: {n}")
    
    # 用這個數字配置 worker pool
    from multiprocessing import Pool
    with Pool(n) as pool:
        results = pool.map(work, data)

3.3 Dask、scikit-learn、autogluon 的踩雷

資料 / ML library 多數預設 n_jobs=cpu_count() 或類似。容器內全部錯：

Library預設行為建議解法
Dask 2024.5+ dask.system.CPU_COUNT 嘗試讀 cgroup（透過 psutil），不可靠 顯式 n_workers=N、threads_per_worker=M
scikit-learn n_jobs=-1 對應 cpu_count() 顯式 n_jobs=N 或設 OMP_NUM_THREADS
NumPy / OpenBLAS 透過 OpenMP 用 host CPU 數 環境變數 OMP_NUM_THREADS、OPENBLAS_NUM_THREADS
autogluon auto-detect，常算錯 明確 num_cpus=N 在 fit() 參數
DuckDB auto 用 host CPU SQL PRAGMA threads=N;
PyTorch torch.get_num_threads() 看 host torch.set_num_threads(N) 顯式設

常見災難案例

K8s Pod limit=2 CPU、Python 應用啟動：

  1. scikit-learn RandomForest n_jobs=-1 → 起 32 個 worker（host CPU 數）
  2. OpenBLAS 看到 32 → 每個 worker 又 spawn 32 thread
  3. 結果：32 × 32 = 1024 thread 搶 2 CPU 配額
  4. CFS throttle 嚴重，整個應用 hang
  5. monitoring：Pod 看似 100% CPU 但實際在 throttle 中睡

3.4 環境變數防線

不能改應用 code 時，用環境變數設 thread / worker 數量。四個關鍵環境變數： 

# Dockerfile / K8s yaml
env:
- name: OMP_NUM_THREADS
  value: "2"      # OpenMP（NumPy、OpenBLAS、scikit-learn）
- name: OPENBLAS_NUM_THREADS
  value: "2"      # OpenBLAS（顯式）
- name: MKL_NUM_THREADS
  value: "2"      # Intel MKL
- name: NUMEXPR_NUM_THREADS
  value: "2"      # numexpr

# K8s downward API：自動 inject pod CPU limit
env:
- name: OMP_NUM_THREADS
  valueFrom:
    resourceFieldRef:
      containerName: app
      resource: limits.cpu

downward API 的限制：只能取整數值，分數 CPU（如 1500m）會 round down 或失敗。

3.5 Python memory 與 cgroup

Python 沒有像 Go GOMEMLIMIT 那樣的記憶體軟限。Python 進程記憶體達到 cgroup memory.max → 直接 OOM。但有幾個工具：

  • resource.setrlimit(resource.RLIMIT_AS)：應用自己設記憶體上限，超過 raise MemoryError。Python-level 但不能取代 cgroup。
  • Python GC 沒 GOMEMLIMIT 那種「接近上限自動 GC」：reference counting 即時釋放，但循環引用要等 GC 跑。可手動 gc.collect()。
  • 第三方：resource_monitor、memory_profiler：可寫 daemon thread 監控、超過閾值主動 GC 或 raise。

建議：對 memory-heavy Python（DataFrame、ML training）用容器外 監控（K8s VPA、Prometheus alert）而非靠 Python 自己。

3.7 Python 3.14 新 default：forkserver

Python 3.14（2025-10）改了 multiprocessing 預設啟動方式：Linux 從 fork → forkserver（macOS 已是 spawn）。

fork 的問題：

  • 主程式有多 thread 時 fork 容易 deadlock（child 繼承 lock 但沒 thread 釋放）
  • fork 完整複製記憶體（COW 緩解但仍有 overhead）
  • 對 cgroup memory 不友善——大型主程式 fork 出多 worker 時，初期 memory 暴增（COW 緩解，但 metrics 看起來嚇人）

forkserver 的優勢：

  • 從乾淨 server process fork，記憶體 footprint 小
  • 避免 thread + fork 的死鎖問題
  • 對 cgroup memory accounting 更可預測
import multiprocessing as mp

if __name__ == '__main__':
    # 顯式選擇（推薦）
    ctx = mp.get_context('forkserver')  # Python 3.14 預設
    # 或 'spawn' / 'fork'

    with ctx.Pool(processes=N) as pool:
        results = pool.map(worker, data)

對 cgroup 的影響：forkserver child 進入同一個 cgroup（fork、forkserver 都是）——所以 limit 仍適用整個 worker pool。但 forkserver 的記憶體 baseline 小，memory.current 看起來較合理。

3.8 Python 3.14 free-threaded build

Python 3.14（PEP 703 stable）支援無 GIL build——thread 真正 parallel。對 cgroup 影響：

  • 可用 thread 取代 process：sklearn n_jobs / Dask 可用 threading 而非 multiprocessing。記憶體共享，cgroup 計量單純。
  • thread 數量仍要看 cgroup：os.cpu_count() 還是錯。沒 GIL 不代表自動知道 limit。
  • OpenBLAS、NumPy 等仍走 OpenMP thread：OMP_NUM_THREADS 仍是關鍵環境變數。

3.9 Web framework 的 worker 數量

Gunicorn / uWSGI / Hypercorn 跑 Python web app，worker 數量直接影響 CPU 使用： 

# Gunicorn 推薦公式
# workers = (2 × CPU) + 1
# 但「CPU」要是 cgroup CPU，不是 host CPU

# 錯誤做法
gunicorn -w $(nproc) app:app
# nproc 看 host，容器內錯

# 正確做法 1：環境變數
gunicorn -w $WEB_CONCURRENCY app:app  # K8s 設 WEB_CONCURRENCY

# 正確做法 2：自己讀 cgroup
gunicorn -w "$(python3 -c '
import math, os
try:
    with open("/sys/fs/cgroup/cpu.max") as f:
        q, p = f.read().split()
    if q != "max":
        cpus = math.ceil(int(q) / int(p))
    else:
        cpus = os.cpu_count() or 1
except:
    cpus = os.cpu_count() or 1
print(2 * cpus + 1)

3.10 best practices 總結

  • 不要假設 Python runtime 知道 cgroup：永遠手動設或用 helper。
  • 關鍵環境變數一律設：OMP_NUM_THREADS、MKL_NUM_THREADS、OPENBLAS_NUM_THREADS 都設成 cgroup CPU 數。
  • 避免 n_jobs=-1：scikit-learn、joblib 等明確設整數。
  • K8s downward API inject CPU limit：Pod yaml 直接從 limit 推算環境變數。
  • 顯式選 forkserver 啟動方式：避免 fork 的 thread 問題。
  • 記憶體保護靠 K8s VPA + alert：Python 沒 GOMEMLIMIT 等價物。
  • 升 Python 3.14 享受 free-threaded build：但仍要正確設 thread 數。

import math


def get_effective_cpu_count() -> int:

    except (AttributeError, OSError):

    return os.cpu_count() or 1

05 · Pitfalls

常見陷阱

陷阱 1 · 假設 psutil 會看 cgroup limit

psutil.cpu_count() 看 host CPU——文件明確說「not necessarily equivalent to the actual number of CPUs the current process can use」。修法：別用 psutil 取「可用 CPU」，自己讀 cgroup 介面或寫 helper。

陷阱 2 · scikit-learn / joblib n_jobs=-1 在容器

K8s Pod limit=2，n_jobs=-1（cpu_count）spawn 32 worker 加 OpenMP nested thread——實際 1024 個 thread 搶 2 CPU。整個應用 hang。修法：明確 n_jobs=2、設 OMP_NUM_THREADS=2、避免任何 -1 auto-detect。

陷阱 3 · OpenMP 與 multiprocessing 重複起 thread

用 multiprocessing 開 N 個 worker，每個 worker 跑 NumPy 又 OpenMP nested 起 N thread——總 thread = N × N。修法：用 multiprocessing 時設 OMP_NUM_THREADS=1（避免 nested）；或用 threadpoolctl 在 worker 內 limit thread。

陷阱 4 · downward API 取 limits.cpu 遇分數失敗

Pod 設 limits.cpu: "1500m"——downward API 取的是「1500m」字串，不是「2」整數。直接當 OMP_NUM_THREADS 會 parse error。修法：用 init container 把 1500m 轉成 2 寫入 shared volume；或設 limit 為整數值。

陷阱 5 · Python 沒設 GOMEMLIMIT 等價物，記憶體達上限直接 OOM

Python 應用接近 cgroup memory.max 沒有「提前 GC」機制——直接被 OOM kill。對比 Go GOMEMLIMIT 自動 aggressive GC。修法：靠 K8s VPA + memory.events.high alert（Memory QoS 提供 throttle 緩衝）；或應用內手動 gc.collect() 在記憶體高位時呼叫。

陷阱 6 · fork-based multiprocessing 在多 thread 主程式 deadlock

主程式有 background thread（如 logger、metrics exporter），fork 出 worker——child 繼承 lock 但沒對應 thread，直接 deadlock。Python 3.14 改 forkserver 預設就是為此。修法：顯式 mp.get_context('forkserver') 或 'spawn'；3.14+ 自動，舊版手動。

陷阱 7 · DuckDB / Polars 內建 thread pool 不知 cgroup

DuckDB SELECT 預設用 host CPU 數開 worker。Pod limit=2 時嚴重 throttle。修法：DuckDB PRAGMA threads=2;、Polars POLARS_MAX_THREADS=2。每個 lib 環境變數不同，部署前查文件。

陷阱 8 · Gunicorn 用 nproc 算 worker 數

gunicorn -w $(nproc) app:app——nproc 看 host，容器內錯。Pod limit=2 但 worker=32，記憶體爆掉。修法：WEB_CONCURRENCY 環境變數明確設、或 inline Python 讀 cgroup 算。

07 · Self-check

自我檢查題

Q1.Python 各個 CPU count API 在容器內各自看到什麼？哪個最接近正確？
各 API 行為：
  • os.cpu_count() → host 邏輯 CPU 數（透過 sysconf）。容器內錯。
  • multiprocessing.cpu_count() → 包裝 os.cpu_count()，同上。
  • psutil.cpu_count(logical=True) → 讀 /proc/cpuinfo，看 host。同上錯。
  • os.sched_getaffinity(0) → process CPU affinity mask。如果用 cpuset 綁 CPU 會準（如 K8s CPU Manager static policy）；但對 cpu.max 配額場景無感。
  • 讀 /sys/fs/cgroup/cpu.max → cgroup CPU 配額。正確。需自己 parse。
哪個最接近正確：os.sched_getaffinity 在「K8s CPU Manager static policy + Guaranteed Pod + 整數 CPU」場景準確（因為實際 cpuset 綁定）。大多數 Pod 是後者場景，所以 sched_getaffinity 也不夠——必須讀 /sys/fs/cgroup/cpu.max。 結論：寫 helper 按優先順序試多個來源（環境變數 → cgroup 介面 → sched_getaffinity → cpu_count fallback）。絕對不要 假設 Python runtime 會自動處理。
Q2.為什麼 Python 至今沒有像 Go 1.25 那樣的 runtime cgroup awareness？
原因（多重）：
  1. 向後相容：Python 用了 30 年，os.cpu_count() 改變行為會 break 大量現有 code。CPython issue 80235 從 2019 卡住至今主因。
  2. Python 沒有等價的「scheduler 預設並發數」概念：Go GOMAXPROCS 是 runtime 內建概念，Python 沒有——cpu_count 只是查詢函式，不是 scheduler 設定。
  3. 應用多樣性：Web server、ML、scientific computing 各自並發模型不同——通用 runtime 難以一刀切。
  4. core dev 認為這是「應用層問題」：CPython 哲學偏向 explicit。
  5. 沒有 Uber-level driver：Go 的 automaxprocs 是 Uber 主導變成事實標準後 Go 1.25 才整合。Python 沒有等價的 driver。
實際影響：Python 應用必須 應用層處理。社群 library（cgroupspy 等）不是事實標準，每家自己寫。 近期變化：Python 3.14（2025-10）改進 multiprocessing 預設啟動方式（fork → forkserver）對 cgroup memory accounting 友善，但不 觸及核心問題。
Q3.OMP_NUM_THREADS、OPENBLAS_NUM_THREADS、MKL_NUM_THREADS 各自影響什麼？為什麼都要設？
三個環境變數的差別：
  • OMP_NUM_THREADS → OpenMP thread 數量。控制 NumPy（部分）、scikit-learn、scipy、xgboost 等。
  • OPENBLAS_NUM_THREADS → OpenBLAS（線性代數 backend）thread 數。NumPy 大部分操作走這。
  • MKL_NUM_THREADS → Intel MKL（如果安裝）thread 數。Anaconda 預裝 MKL；conda-forge 的 NumPy 走 OpenBLAS。
  • 其他類似：NUMEXPR_NUM_THREADS、VECLIB_MAXIMUM_THREADS（macOS）、BLIS_NUM_THREADS。
為什麼都要設：
  1. 應用可能用多個 backend——同時走 OpenBLAS、MKL、自己的 OpenMP code。
  2. 每個 library 看自己的環境變數，沒設就用 host CPU。
  3. 「thread 風暴」最容易發生在多 nested thread——multiprocessing × OpenMP × OpenBLAS 三層 nested，1024 thread 不誇張。
建議實務：在 K8s yaml 一次都設好，值都等於 cgroup CPU limit。container image entrypoint 可加一層保護。 注意：如果在 multiprocessing 內，每個 worker 仍會起 OMP_NUM_THREADS 個 thread。nested 場景 設 OMP_NUM_THREADS=1（worker 之間不再 thread parallel，避免風暴）。
Q4.Java JVM 在容器化整合上比 Python 強多少？哪些設計值得 Python 學？
Java 強在哪：
  1. Runtime 自動偵測：JDK 8u191+ / JDK 10+ UseContainerSupport 預設啟用，自動讀 cgroup CPU / memory。
  2. Heap 大小自動算：-XX:MaxRAMPercentage=75 從 cgroup memory limit 算 heap（沒 -Xmx 時）。
  3. v2 完整支援：8u372+、11.0.16+、17+ 完整支援 cgroup v2，含 swap 計算（修了 JDK-8253727）。
  4. Introspection 工具：java -XshowSettings:system 顯示完整 cgroup 偵測結果——是 Python 完全沒有的能力。
  5. Thread pool 自動匹配：ForkJoinPool common pool 大小自動用 cgroup CPU。
  6. 明確 opt-out：-XX:-UseContainerSupport 可關，不像 Python 沒有開關因為根本沒有。
Python 該學：
  1. 內建 cgroup 偵測：CPython 應該加 os.cgroup_cpu_quota() 之類 API（不破壞 cpu_count）。
  2. introspection：python -X showsettings 之類顯示 cgroup 偵測。
  3. 事實標準 library：類似 Uber 之於 automaxprocs 的角色。
  4. 環境感知預設：multiprocessing.Pool() 不傳 processes 時應該看 cgroup。
但有差別：Java 是 JIT，啟動時偵測就用一輩子；Python 啟動快但 import 多、混合 native lib，每個 lib 自己處理 thread——Python 即使加了 runtime awareness 也只解一部分問題（NumPy/OpenBLAS 仍要環境變數）。
Q5.Python 3.14 改 multiprocessing 預設為 forkserver，對 cgroup 有什麼影響？
fork vs forkserver vs spawn：
  • fork（Linux 舊預設）：直接 os.fork() 複製當前 process。
  • forkserver（Linux 3.14+ 預設）：先 fork 一個乾淨 server process，之後從 server fork worker。
  • spawn（Windows / macOS 預設）：execve 啟動新 Python 直譯器。
對 cgroup 的具體影響：
  1. memory accounting：fork COW 短期記憶體看起來爆，但物理頁共享；forkserver baseline 小、worker 啟動更可預測。對 cgroup memory.current 監控更乾淨。
  2. cgroup membership：三種啟動方式 child 都進入同一 cgroup。limit 仍適用整 worker pool。
  3. thread + fork bug：fork 在多 thread 主程式（如 Flask + background thread）會 deadlock；forkserver 從乾淨 process 出發避免這個。
  4. 啟動速度：forkserver 比 fork 慢、比 spawn 快。對 short-lived Pool 影響可感知。
建議：
  • Python 3.14+：享受 forkserver 預設。
  • Python 3.13-：手動 mp.get_context('forkserver')，特別在容器內 production code。
  • 仍要正確設 worker 數量——啟動方式不解決 cpu_count() 看 host 的問題。
Q6.K8s downward API 怎麼把 Pod CPU limit 自動 inject 到 Python 環境變數？有什麼坑？
基本用法： 
env:
- name: OMP_NUM_THREADS
  valueFrom:
    resourceFieldRef:
      containerName: app
      resource: limits.cpu
坑 1：分數 CPU 表示法：
  • Pod 設 limits.cpu: "2" → downward API 給 "2"（OK）
  • Pod 設 limits.cpu: "1500m" → 給 "1500m"（字串，不是 1.5 也不是 2）
  • Python 直接用會 parse error
解法 1：用整數 CPU limit（推薦）。 解法 2：用 divisor: "1"，"1500m" 變 "2"（K8s 會 round up 到 divisor 倍數）。 
env:
- name: OMP_NUM_THREADS
  valueFrom:
    resourceFieldRef:
      containerName: app
      resource: limits.cpu
      divisor: "1"   # round up 到整數
坑 2：沒設 limits 時 inject 失敗：only-request 的 Pod 沒 limits.cpu，downward API 會 fail——Pod 啟動失敗。修法：要嘛強制有 limits（建議），要嘛 try/except 在 init 時讀。 坑 3：requests 與 limits 差很大時：env 取 limits 但實際排程是 requests，可能不匹配 capacity。應用要意識到「能用最多 limits.cpu，但常態 requests.cpu」。 坑 4：limits 後續改變不會更新環境變數：In-Place Resize 改了 limit，環境變數仍是舊值。Python 應用需要重啟（雖然 CPU cgroup 已更新）。
Q7.Dask、autogluon、DuckDB 各自怎麼正確配置容器 worker 數？
Dask（local cluster）： 
from dask.distributed import Client, LocalCluster
n = get_effective_cpu_count()  # 自寫 helper
cluster = LocalCluster(
    n_workers=n,           # 顯式設
    threads_per_worker=1,  # 每 worker 單 thread 避免 nested
    memory_limit='auto'    # Dask 2024.5+ 試讀 cgroup
)
client = Client(cluster)
scikit-learn / joblib： 
from joblib import parallel_backend
n = get_effective_cpu_count()
with parallel_backend('loky', n_jobs=n):
    clf.fit(X, y)
# 或在 estimator 直接設
RandomForestClassifier(n_estimators=100, n_jobs=n)
# 不要用 n_jobs=-1
autogluon： 
from autogluon.tabular import TabularPredictor
n = get_effective_cpu_count()
predictor.fit(
    train_data,
    num_cpus=n,        # 明確設
    num_gpus=1,
    presets='medium_quality',
)
DuckDB： 
import duckdb
n = get_effective_cpu_count()
conn = duckdb.connect()
conn.execute(f"PRAGMA threads={n};")
# 或環境變數
# DUCKDB_NUM_THREADS=2
PyTorch： 
import torch
n = get_effective_cpu_count()
torch.set_num_threads(n)         # 計算 thread
torch.set_num_interop_threads(1) # interop 避免 nested
# DataLoader
loader = DataLoader(dataset, num_workers=n, ...)
Polars：環境變數 POLARS_MAX_THREADS。 核心原則：每個 lib 都有自己的 thread pool，每個都要明確設。寫一個共用 init module 一次設好。
