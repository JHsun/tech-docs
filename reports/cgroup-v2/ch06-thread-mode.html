Part II · 操作篇 · Chapter 06

Thread-level 控制

v2 預設以 process 為粒度，但對於需要把同一個 process 的不同 thread 分開治理的場景， v2 補回了限制版的 thread mode。本章拆解 threaded subtree 的拓撲規則、四個 supported controllers， 並用 JVM thread pool 隔離當主案例展示真實使用場景。

01 · Highlights

本章重點

  1. 大多數人不需要：threaded mode 是 niche feature，9 成以上的 workload 用 domain 就足夠。學它主要是為了 debug、JVM 調優、即時系統設計。
  2. 啟用條件嚴格：cgroup 沒有 child、沒啟用任何 domain controller、parent 必須是 valid threaded domain 或 threaded cgroup——不滿足就會進 domain invalid 狀態。
  3. 狀態機四態：domain（預設）→ 寫入 threaded → 變 threaded，同時 parent 自動變 domain threaded。整個轉換單向不可逆。
  4. 四個 supported controllers：cpu、cpuset、perf_event、pids。memory、io 等 domain controllers 不支援——它們的計量都歸到 threaded subtree 的 root（thread root）。
  5. 實戰主案例：JVM 把 GC threads 與 worker threads 分到不同 cgroup，給 worker 更高 cpu.weight，讓即時請求不被 GC 拖慢。
02 · Latest

最新動態（2025–2026）

2025 JVM cgroup v2 awareness 已在 JDK 8u372、11.0.16、17 主流版本支援。OpenJDK 直接讀 cgroup limits 來計算 heap size 與 thread pool 大小。但 thread mode 進階用法仍需手動配置。
2025-11 Red Hat 發布 OpenShift 4 上 Java/.NET/Node.js 的 cgroups v2 影響分析，明確指出 CPU 設定錯誤會導致 thread pool 過大、probe 失敗等問題。
2024–2025 realtime workload 仍是 thread mode 的主要使用者。電信、金融、遊戲伺服器把 RT threads 與 best-effort threads 分到不同 cgroup，用 cpuset 綁特定 isolcpus 核。
2025 thread mode 仍未廣泛使用於 Kubernetes。Pod 級別的隔離已足夠 90% 場景，container 內進一步分 thread cgroup 通常需要應用主動配合，自動化工具尚未成熟。

2026 年的時點，thread mode 仍然是「需要時才用」的進階工具。 本章不會誘導讀者去用它，而是讓你看得懂它在哪些場景出現、會用它解決特定問題、會避開常見陷阱。 實作上，我建議先看完本章的觀念部分，動手 lab 可以略讀——多數人實務上很少需要動到 thread mode。

03 · Core Concepts

核心內容

3.1 為什麼需要 thread mode

Ch.1 講過：v1 允許 thread 散佈到不同 cgroup，這個特性被嚴重濫用，後來被當成主要遺憾之一寫進 v2 設計動機。 但「thread 級別控制」本身在某些場景確實有用——v2 完全砍掉它太絕對了，於是 kernel 4.14（2017）以「限制版」形式補回。

kernel 文件 §2.2.2 "Threads" 說明 thread mode 的設計目標：

cgroup v2 supports thread granularity for a subset of controllers to support use cases requiring hierarchical resource distribution across the threads of a group of processes.

典型場景是單一 process 內有不同類型的 thread，需要差別對待：

  • JVM：GC threads、worker threads、JIT compiler threads 屬於同一個 process，但對 latency 敏感度截然不同
  • 遊戲伺服器：tick worker、network IO、physics 計算 thread，希望 tick worker 永遠優先
  • 即時系統：RT threads 跑 SCHED_FIFO，其他 admin / logging threads 跑 best-effort
  • 資料庫：query worker、checkpoint、WAL writer，有時希望 checkpoint 不要影響 query

v1 的解法是無腦把 thread 寫進不同 cgroup（造成各種衝突）；v2 用「限制版 thread mode」精準解決：thread 可以散佈，但只有部分 controllers 支援，且整個 process 的 domain 資源（如 memory）仍然統一計算。

3.2 狀態機與啟用條件

Ch.2 已經介紹過 cgroup.type 的四個狀態。本章把它們的轉換規則講透：

圖 6.1 · cgroup.type 狀態機與轉換規則

啟用 threaded 的三個必要條件

kernel 文件 §2.2.2 列出啟用 thread mode 的條件，三條全部滿足才會成功：

  1. cgroup 沒有 child：要轉的 cgroup 必須是 leaf。有任何 child 都會失敗。
  2. cgroup 沒啟用任何 domain controller：cgroup.subtree_control 不能含 memory、io 等 domain controllers。可以含 cpu、cpuset、pids、perf_event。
  3. parent 必須適配：parent 是 valid threaded domain（root 或自身已 threaded）或 parent 是 threaded cgroup。對 unthreaded domain 來說，parent 也不能有 domain controllers 啟用或 populated 的 child。

條件 3 最容易踩雷——很多 case 都是「parent 已經有別的 child 在跑 process」或「parent 已經啟用了 memory」就不能轉了。

3.3 threaded subtree 的結構

一旦 cgroup 被轉成 threaded，它的 parent 自動變成 domain threaded，從這個 parent 開始往下的整個子樹形成 threaded subtree。 threaded subtree 內：

  • thread 可以散佈在不同 cgroup（透過 cgroup.threads）
  • 不受 no-internal-process constraint 約束（threaded controllers 可以在非 leaf 啟用）
  • 所有 domain controllers 的計量都歸到 thread root（domain threaded 那個 cgroup）

從外面看，整個 threaded subtree 對 domain controllers 而言就是一個 cgroup——這就是「resource domain」這個名字的由來。

3.4 四個 supported controllers

kernel 規定 controller 必須明確宣告自己支援 thread mode 才能用在 threaded subtree。截至 2026 年，只有四個：

Controller thread 粒度語意 典型用法
cpu 每個 thread 獨立 schedule，cpu.weight 影響各 thread 的 CPU 時間比例。 JVM GC 與 worker 分開、即時系統 RT vs best-effort
cpuset 每個 thread 可以綁不同 CPU 核或 NUMA node。 把 RT thread 綁到 isolcpus 核、worker thread 限制在某個 NUMA
perf_event 每個 thread 獨立 perf 計數收集。 專門 profiling 某些 worker thread 而不影響其他
pids 限制這個 cgroup 能持有的 thread 數量。 防 thread leak、限 thread pool 上限

不支援 thread mode 的 controllers

其他 controllers 全部是 domain controllers：

  • memory：同一 process 的 thread 共享 address space，「這塊 page 屬於哪個 thread」沒有意義
  • io：file descriptor 與 page cache 在 process 層級共享，writeback 是非同步的（kworker 觸發），實作上無法分 thread
  • hugetlb、rdma、misc：類似的 process 共享資源

這就是為什麼 thread mode 要求「parent 不能有 domain controllers 啟用」——一旦把 cgroup 變 threaded，這些資源計量「自動往上滾」到 thread root，threaded cgroup 自己看不到、改不了這些介面檔案。

3.5 cgroup.threads 介面

Ch.4 介紹過 cgroup.threads，這裡進一步說明它的使用條件：

  • 讀：在任何 cgroup 都可讀，列出 thread (TID) 列表。對 domain cgroup 而言，等同列出所有 process 的所有 thread。
  • 寫：只在 threaded subtree 內可寫。對 domain cgroup 寫入會被拒絕（EOPNOTSUPP）。
  • 遷移粒度：寫一個 TID 只遷移該 thread，不影響同 process 的其他 thread——這是 cgroup.procs 與 cgroup.threads 的關鍵差異。

3.6 實戰主案例：JVM thread pool 隔離

假設你跑一個 latency-sensitive 的 Java service，遇到「GC pause 偶爾尖峰，把 worker 拖慢」的問題。 想法：把 GC threads 與 worker threads 放到不同 cgroup，給 worker 高 cpu.weight，GC 低 cpu.weight。

步驟概要（完整指令在 Lab 4）：

  1. 建立 thread root：/sys/fs/cgroup/myapp（domain，準備轉 threaded）
  2. 建立兩個 child：myapp/workers、myapp/gc
  3. 把 workers 與 gc 都轉成 threaded：echo threaded > cgroup.type
  4. 此時 myapp 自動變成 domain threaded
  5. 啟用 cpu controller：echo +cpu > myapp/cgroup.subtree_control
  6. 設 weight：workers/cpu.weight=800、gc/cpu.weight=200
  7. JVM 啟動後，把 GC thread 的 TID 寫進 gc/cgroup.threads

識別 GC thread 的方法：JVM 啟動時用 -XX:+PrintGCDetails 看 thread name；或 runtime 用 jstack 看（GC threads 命名包含 GC Thread、Concurrent GC Thread 等）。

效果：當系統有 CPU 競爭時，worker 拿到 80% CPU、GC 拿 20%。GC pause 仍然存在（GC 邏輯沒變），但 GC 跑得慢一點，worker 不會被 GC 搶 CPU 而尖峰。 memory 計量在 myapp（domain threaded）統一算，符合 JVM 共享 heap 的事實。

3.7 哪些場景不該用 thread mode

最後反過來說：以下場景強烈建議用 process cgroup（不是 threaded）：

  • 多容器隔離：每個容器是獨立 process group，用 process cgroup 給 memory.max + cpu.max 已經足夠
  • 限 service 用多少資源：systemd service 每個是獨立 process tree，用標準 cgroup
  • 容器內隔離不同 sub-process：fork 出新 process 不需要 thread mode
  • 「我覺得 thread mode 應該很酷」：真的不要無腦上，後面 debug 會痛苦

thread mode 只有在同一 process 內 thread 有差別需求時才用。判斷標準：你需要 memory.max 分開設定嗎？需要的話別用 thread mode。

# 2. 建兩個 child

# 5. group-b 此時被卡到什麼狀態？

# 6. 救它：轉成 threaded
echo threaded | sudo tee /sys/fs/cgroup/lab6/group-b/cgroup.type

# 7. 嘗試把 group-a 變回 domain（不可逆操作）

Lab 3 · 用 cgroup.threads 移動 thread
# 1. 建立 threaded subtree
sudo mkdir /sys/fs/cgroup/lab6
sudo mkdir /sys/fs/cgroup/lab6/{a,b}
echo threaded | sudo tee /sys/fs/cgroup/lab6/a/cgroup.type
echo threaded | sudo tee /sys/fs/cgroup/lab6/b/cgroup.type
echo "+cpu" | sudo tee /sys/fs/cgroup/lab6/cgroup.subtree_control
echo 100 | sudo tee /sys/fs/cgroup/lab6/a/cpu.weight
echo 800 | sudo tee /sys/fs/cgroup/lab6/b/cpu.weight

# 2. 啟動一個多 thread 的 program
sudo bash -c '
    python3 -c "
import threading, os
def worker(name):
    print(f\"{name} TID={os.gettid()}\")
    while True:
        pass
threads = [threading.Thread(target=worker, args=(f\"t{i}\",)) for i in range(4)]
for t in threads: t.start()
for t in threads: t.join()
" &
'
PYTHON_PID=$!

sleep 1

# 3. 列出該 process 的所有 thread
ls /proc/$PYTHON_PID/task/

# 4. 把 PID 寫進 cgroup.procs（整個 process 都進來）
echo $PYTHON_PID | sudo tee /sys/fs/cgroup/lab6/cgroup.procs
# 注意：thread mode 下，process 仍透過 cgroup.procs 進入 thread root

# 5. 取得各 thread TID（task 目錄名就是 TID）
TIDS=$(ls /proc/$PYTHON_PID/task/)

# 6. 把前兩個 thread 移到 a，後兩個移到 b
TIDS_ARR=($TIDS)
echo ${TIDS_ARR[0]} | sudo tee /sys/fs/cgroup/lab6/a/cgroup.threads
echo ${TIDS_ARR[1]} | sudo tee /sys/fs/cgroup/lab6/a/cgroup.threads
echo ${TIDS_ARR[2]} | sudo tee /sys/fs/cgroup/lab6/b/cgroup.threads
echo ${TIDS_ARR[3]} | sudo tee /sys/fs/cgroup/lab6/b/cgroup.threads

# 7. 確認 thread 分布
cat /sys/fs/cgroup/lab6/a/cgroup.threads
cat /sys/fs/cgroup/lab6/b/cgroup.threads

# 8. 看 CPU 分配（b 的 thread 應該拿到 800/(100+800) ≈ 89% 比例）
top -p $PYTHON_PID -H

# 清理
sudo kill $PYTHON_PID
sudo rmdir /sys/fs/cgroup/lab6/{a,b}
sudo rmdir /sys/fs/cgroup/lab6
Lab 4 · JVM GC vs worker 隔離（完整範例）
# 1. 建立 threaded subtree
sudo mkdir /sys/fs/cgroup/jvmapp
sudo mkdir /sys/fs/cgroup/jvmapp/{workers,gc}
echo threaded | sudo tee /sys/fs/cgroup/jvmapp/workers/cgroup.type
echo threaded | sudo tee /sys/fs/cgroup/jvmapp/gc/cgroup.type

# 2. 啟用 cpu，分配 weight
echo "+cpu" | sudo tee /sys/fs/cgroup/jvmapp/cgroup.subtree_control
echo 800 | sudo tee /sys/fs/cgroup/jvmapp/workers/cpu.weight
echo 200 | sudo tee /sys/fs/cgroup/jvmapp/gc/cpu.weight

# 3. 啟動 JVM（範例用 simple Java program）
java -XX:+UnlockDiagnosticVMOptions \
     -XX:+PrintGCDetails \
     YourApp &
JVM_PID=$!

# 4. 把 JVM main process 移進 thread root
echo $JVM_PID | sudo tee /sys/fs/cgroup/jvmapp/cgroup.procs

# 5. 識別 GC thread (用 jstack)
GC_TIDS=$(jstack $JVM_PID | \
    grep -E "GC Thread|Concurrent GC Thread|G1 GC Thread" | \
    grep -oP 'nid=\K0x[a-f0-9]+' | \
    while read hex; do printf "%d\n" $hex; done)

# 6. 把 GC thread 移到 gc cgroup
for tid in $GC_TIDS; do
    echo $tid | sudo tee /sys/fs/cgroup/jvmapp/gc/cgroup.threads
done

# 7. 其他 thread (worker) 移到 workers cgroup
for tid in $(ls /proc/$JVM_PID/task); do
    # 跳過已在 gc 的 thread
    grep -q "^$tid$" /sys/fs/cgroup/jvmapp/gc/cgroup.threads && continue
    echo $tid | sudo tee /sys/fs/cgroup/jvmapp/workers/cgroup.threads
done

# 8. 觀察分布
echo "GC threads:"
wc -l < /sys/fs/cgroup/jvmapp/gc/cgroup.threads
echo "Worker threads:"
wc -l < /sys/fs/cgroup/jvmapp/workers/cgroup.threads

# 9. 此時跑 workload，CPU 競爭時 worker 拿 80%、GC 拿 20%
# memory 計量仍在 jvmapp 上統一（因為 thread root）
cat /sys/fs/cgroup/jvmapp/memory.current

# 清理
kill $JVM_PID
sudo rmdir /sys/fs/cgroup/jvmapp/{workers,gc}
sudo rmdir /sys/fs/cgroup/jvmapp

注意：這個 lab 在 production 不建議直接照搬——應該包成 systemd service，並讓 JVM 啟動時就帶 -XX:+PerfDisableSharedMem 等選項。實作通常用 JVMTI agent 自動分類 thread。

05 · Pitfalls

常見陷阱

陷阱 1 · 想隔離容器卻用 thread mode

最常見的誤用：聽說 thread mode 可以「更精細地控制」，於是想把 Kubernetes Pod 搞成 threaded subtree。 錯——Pod 內每個 container 是獨立 process group，本來就用 process cgroup 隔離。thread mode 只在「同一個 process 內 thread 有差別」時才有意義。 套用在 Pod 上會失去 memory.max 等核心 limits，整個事情變得更複雜不是更簡單。

陷阱 2 · domain invalid 狀態救不回來時整棵砍掉

當你看到 cat cgroup.type 顯示 domain invalid，第一反應通常是「怎麼讓它變回 domain」。 答案是：不能。domain invalid 只能轉 threaded 或被 rmdir。 如果整個拓撲設計錯了，砍掉重建是唯一解。所以做 thread mode 之前要想清楚拓撲。

陷阱 3 · 啟用 memory 後再想轉 threaded

典型流程錯誤：先 echo +memory > subtree_control 設好 memory limits，再想 echo threaded > cgroup.type 把 child 轉 threaded——失敗。 正確順序：先決定要不要走 thread mode，要的話不要啟用 domain controllers。 memory 等資源的控制要在更上層的 domain cgroup 設，threaded subtree 內只能用 cpu / cpuset / pids / perf_event。

陷阱 4 · cgroup.threads 寫不進去（EOPNOTSUPP）

在 domain cgroup 寫 cgroup.threads，會收到 Operation not supported。 原因：cgroup.threads 只在 threaded subtree 內可寫。 修法：先把對應 cgroup 轉成 threaded（會自動把 parent 轉成 domain threaded），才能寫入 TID。

陷阱 5 · JVM thread 識別後沒辦法持續追蹤新 thread

用 jstack 識別 GC thread、寫進 cgroup.threads 之後，JVM 動態產生的新 GC thread（例如 concurrent GC scaling up）不會自動加入。 這需要 JVMTI agent 在 thread 啟動時 hook，或寫 daemon 定期 reconcile。產線使用必備持續監控與 reconcile，否則隔離效果會逐漸失效。

陷阱 6 · systemd 不直接管 thread mode

systemd 的 Delegate=yes 給的是 process-level cgroup。 如果你想在 service 內用 thread mode，必須用 Delegate=yes 拿到自己的子樹後，再手動建立 threaded structure。 systemd 沒提供類似 ThreadedCgroup=yes 的選項——thread mode 是應用自管的事，systemd 只負責給 process 邊界。

07 · Self-check

自我檢查題

Q1.啟用 thread mode 的三個必要條件是什麼？哪個最容易踩雷？
  1. cgroup 沒有 child：必須是 leaf。
  2. cgroup 沒啟用任何 domain controller：subtree_control 不能含 memory、io 等。
  3. parent 必須適配：parent 是 valid threaded domain（root 或自身已 threaded）或 是 threaded cgroup。對 unthreaded domain，parent 也不能有 domain controllers 啟用或 populated 的 child。
最容易踩雷的是條件 3——多數情況都是「parent 已經有別的 child 在跑 process」或「parent 已經啟用了 memory」就不能轉。實作上要先規劃好 thread root 的位置，避免事後重來。
Q2.cgroup.type 的四個狀態之間的轉換規則是什麼？哪些是自動觸發、哪些是手動觸發？
手動觸發：
  • domain → threaded：寫 threaded 到 cgroup.type
  • domain invalid → threaded：寫 threaded 到 cgroup.type
自動觸發：
  • domain → domain threaded：當任一 child 變成 threaded 時，parent 自動變
  • domain → domain invalid：當 parent 變成 threaded（即所在的子樹變成 threaded subtree），原本是 domain 的 sibling 自動變 invalid
不存在的轉換：threaded → domain。一旦轉成 threaded 就不可逆。要回到 domain 必須刪除整個 threaded subtree 重建。
Q3.哪四個 controllers 支援 thread mode？為什麼這四個可以而 memory 不能？
支援 thread mode 的四個 controllers：
  • cpu：每個 thread 獨立 schedule，CPU 時間天然 thread-level
  • cpuset：affinity 可以 per-thread 設定（pthread_setaffinity_np）
  • perf_event：perf 計數本來就是 per-thread 收集
  • pids：限制 thread 數量是計數行為，per-thread 也合理
memory 不能的原因：同一 process 的所有 thread 共享 address space。一塊 page 的「歸屬權」在 process 層級才有意義——把不同 thread 放到不同 cgroup，「這塊 page 算誰的記憶體」沒有合理答案。 io 不能的原因：file descriptor、page cache、writeback 都在 process 層級共享，writeback 由非同步 kworker 觸發，根本無法歸到原始 thread。 結論：thread mode 適用於「排程行為」類資源（CPU/perf），不適用於「共享狀態」類資源（memory/io）。
Q4.什麼是 thread root（resource domain）？它在 threaded subtree 中扮演什麼角色？
thread root：threaded subtree 中最頂端的 cgroup，cgroup.type 顯示為 domain threaded。 角色與功能：
  • 承載所有 domain 資源計量：threaded subtree 內所有 thread 用的 memory、io 等都歸到這裡。從外面看，整個 threaded subtree 對 domain controllers 而言就是這一個 cgroup。
  • 提供共同的 resource ground：threaded controllers 在 subtree 內做 thread-level 控制，但所有 thread 共用 thread root 的 domain 資源。
  • 對外是 process cgroup：父層的 cgroup 看到 thread root 就是個普通 cgroup，可以對它設 memory.max、io.weight 等。
kernel 文件用語：「The root of a threaded subtree, that is, the nearest ancestor which is not threaded, is called threaded domain or thread root interchangeably and serves as the resource domain for the entire subtree.」
Q5.cgroup.procs 與 cgroup.threads 在 threaded subtree 內的行為差異？
cgroup.procs：在 threaded subtree 內，寫入 PID 仍會把整個 process 的所有 thread 一起遷入該 cgroup。實務上典型用法：寫進 thread root 把整個 process 帶進 subtree。 cgroup.threads：寫入 TID 只遷移該 thread。只在 threaded subtree 內可寫——對 domain cgroup 寫入會收到 EOPNOTSUPP。 典型工作流：
  1. 用 cgroup.procs 把整個 process 寫進 thread root
  2. 用 cgroup.threads 把特定 thread 分散到 child threaded cgroups
記住：process 進入用 procs，thread 分散用 threads。
Q6.JVM thread 隔離為什麼要用 thread mode？用 process cgroup 替 JVM 整個設限不就好了嗎？
用 process cgroup 給 JVM 整體設限（如 memory.max=8G、cpu.max=4 cores）確實能做基本資源隔離，這是 99% 場景的做法。 但 thread mode 解決的是同一 process 內不同 thread 的差別優先級問題：
  • process cgroup 限制了 JVM 整體用多少 CPU，但 GC pause 仍然會跟 worker thread 搶這個整體配額——當 GC 跑時，worker 看起來就是慢了。
  • thread mode 把 GC 與 worker 分開，給 worker 高 cpu.weight。當 GC 跑時，scheduler 仍然優先給 worker CPU 時間，GC 跑慢一點，但 worker 不被波及。
判斷標準：你只在意「JVM 整體不超過多少資源」嗎？用 process cgroup。你還在意「JVM 內哪些 thread 優先」？用 thread mode。 注意：thread mode 不會讓 GC pause 消失，只會讓 GC 不影響 worker 的 schedule 優先權。GC pause 本身要靠 GC tuning（G1、ZGC、Shenandoah）解決。
Q7.什麼情況下會看到 domain invalid？舉一個具體拓撲。
典型觸發場景：拓撲是 A (domain) → B (domain)，後來把 B 轉成 threaded。kernel 自動把 A 變成 domain threaded，但如果 A 還有其他 sibling C（也是 domain），C 就會自動變 domain invalid，因為它的 parent A 已經是 threaded subtree root 了。 具體例： 
[初始] A (domain) ├─ B (domain) └─ C (domain)
[做] echo threaded > B/cgroup.type
[結果] A (domain threaded) ├─ B (threaded) └─ C (domain invalid)
另一觸發：在已是 threaded subtree 內 mkdir 新 child——新 child 預設是 domain，立刻陷入 invalid。 修復方式：把 invalid 的 cgroup 也轉 threaded（echo threaded > cgroup.type）或刪除（rmdir）。無法回到 domain。
Q8.實作判斷題：以下哪些情境應該用 thread mode？哪些應該用 process cgroup？（A）容器隔離；（B）JVM GC vs worker 分配；（C）Kubernetes Pod 級資源限制；（D）即時系統 RT thread 與 best-effort 隔離；（E）資料庫 query thread 限制總數。
process cgroup（domain）：
  • (A) 容器隔離：每個容器是獨立 process group，不需 thread mode。memory.max + cpu.max 標準做法。
  • (C) Kubernetes Pod 級資源限制：Pod 是 process boundary，不需 thread mode。Pod 內 container 也是獨立 process。
  • (E) 資料庫 query thread 限制總數：限制總數是 process 級的整體控制，用 pids.max（domain controller 也支援，更簡單）。
thread mode（threaded）：
  • (B) JVM GC vs worker 分配：同一 process 內不同 thread 的差別優先級，thread mode 的典型場景。
  • (D) RT thread 與 best-effort 隔離：同一 RT 應用的不同 thread 角色，thread mode 配合 cpuset 綁 isolcpus 是經典做法。
判斷規則：「是否需要對同一 process 內不同 thread 做差別資源分配？」是 → thread mode；否 → process cgroup。
