Part V · 可觀測性 · Chapter 13

eBPF 與 cgroup 整合

eBPF 與 cgroup 是 Linux 雲原生的兩大支柱。前者讓你在 kernel 跑安全沙盒程式收集資料， 後者把資源管理組織成樹狀結構——兩者結合解鎖了傳統工具做不到的觀測與控制能力。 本章拆解 BPF cgroup program 9 種類型、用 bpftrace 寫 per-cgroup 的 one-liner、 用 bpftool 操作 cgroup BPF programs，最後實戰找出 noisy neighbor、debug latency 異常。

01 · Highlights

本章重點

  1. eBPF 與 cgroup 是互補的：cgroup 把 process 組織成樹、定義資源邊界；eBPF 提供 kernel 內可程式化的觀測與控制。per-cgroup 過濾是 eBPF 在 multi-tenant 環境最強的能力。
  2. BPF cgroup program 9 種類型：device、skb、sock_create、sock_addr（4/6 個 hook 點）、sock_ops、sysctl、sockopt 等。每種對應不同 hook，附加到 cgroup 後對該 cgroup 內所有 process 生效。
  3. bpftrace 是 cgroup-aware 觀測首選：用 cgroup 內建變數 + cgroupid() helper，可以對特定 cgroup 過濾任何 tracepoint / kprobe，不需要自己寫 BPF C code。
  4. bpftool 是 cgroup BPF 的瑞士刀：bpftool cgroup show 看 attached programs、bpftool prog dump 看 BPF 程式內容、bpftool prog tracelog 看 BPF 印出的 log。
  5. 低 overhead：純 cgroup-filtered eBPF 觀測通常 1–3% CPU，遠低於 sidecar agent（10–40%）。Pixie、Cilium、Parca 等都基於這個架構。
02 · Latest

最新動態（2025–2026）

2025 BCC 加入 BTF（BPF Type Format）支援，提供 CO-RE 風格能力——同一個 BCC script 可在不同 kernel 版本跑，不用重編譯。對 cgroup 觀測工具的可移植性大幅改善。
2024–2025 Cilium、Pixie、Parca 全面採用 cgroup-aware eBPF。kernel-level 觀測的 overhead 普遍降到 1-3%，比 DataDog Agent / OpenTelemetry sidecar（10-40%）有顯著優勢。
2025 sched_ext（kernel 6.12+）成熟。BPF scheduler 可以實作監控-only policy——profile scheduling events 不改變排程行為，是 cgroup 內 task scheduling 觀測的新工具。
2025 per-cgroup struct_ops 與 page cache policy 自訂（SOSP'25 paper）。研究界開始把 BPF hook 用到 cgroup 內部記憶體管理 policy，未來可能成為 production feature。
2025 bpftrace 新增更多 cgroup helper：cgroup 內建變數、cgroupid() 函數、cgroup_path()。寫 cgroup-aware one-liner 變得極簡單。

2026 年的時點，eBPF 已是 cgroup v2 觀測性的事實標準。 從個人 debug（bpftrace one-liner）到大規模生產（Pixie、Cilium）都依賴這個組合。 本章除了介面說明，會把怎麼從 bpftrace 起步、怎麼找出真實的 noisy neighbor、 怎麼用 bpftool debug 講清楚——這些是文檔上找不到的實戰技巧。

03 · Core Concepts

核心內容

3.1 eBPF 與 cgroup 為什麼互補

前面 Ch.11 講過 v2 device controller 完全改用 BPF。但這只是開始——eBPF 在 cgroup 領域有更廣的應用：

  • 觀測（observability）：tracepoint / kprobe / uprobe 配合 cgroup 過濾，只觀察特定容器或 service。
  • 控制（control）：device、network packet、socket 等多種 hook 點上 attach BPF program，做 cgroup 範圍的存取控制。
  • 策略（policy）：K8s NetworkPolicy 用 cgroup BPF 實作 per-Pod 網路策略；Cilium 全面 BPF-based。

關鍵優勢：cgroup BPF program 一旦 attach，cgroup 內所有 process 自動受影響——新 fork 出來的也算。process 移出 cgroup，policy 也自動失效。不需要 per-process attach，也不需要清理。

3.2 BPF cgroup program 完整類型表

BPF program typeHook 時機主要用途
BPF_PROG_TYPE_CGROUP_DEVICE device access（mknod / read / write） v2 device controller（Ch.11）
BPF_PROG_TYPE_CGROUP_SKB network packet（ingress / egress） 網路流量過濾、計量
BPF_PROG_TYPE_CGROUP_SOCK socket create 限制 socket 類型、設 socket 屬性
BPF_PROG_TYPE_CGROUP_SOCK_ADDR connect / bind / sendmsg / recvmsg 透明 redirect、限制連線目的
BPF_PROG_TYPE_SOCK_OPS TCP 狀態變化（connect、accept、retransmit） TCP 調優、congestion control
BPF_PROG_TYPE_CGROUP_SYSCTL sysctl read / write 限制容器修改 sysctl
BPF_PROG_TYPE_CGROUP_SOCKOPT setsockopt / getsockopt 修改 socket option（如 TCP_NOTSENT_LOWAT）

共同特性：每個 program type 接收特定 context（如 bpf_cgroup_dev_ctx、bpf_sock_addr），return 1 = allow、return 0 = deny（操作回 EPERM）。

3.3 attach BPF program 到 cgroup

典型流程： 

# 1. 編譯 BPF program（C → bytecode）
clang -O2 -target bpf -c my_filter.c -o my_filter.o

# 2. attach 到 cgroup
sudo bpftool cgroup attach /sys/fs/cgroup/myapp \
    device pinned /sys/fs/bpf/my_filter

# 3. 看 attached
sudo bpftool cgroup show /sys/fs/cgroup/myapp
# ID       AttachType      AttachFlags     Name
# 234      device                          (none)

# 4. detach
sudo bpftool cgroup detach /sys/fs/cgroup/myapp \
    device id 234

實務上：使用者通常不直接 attach。容器 runtime（runc、crun）、Cilium、systemd 等內部處理。 但理解這個機制對 debug 與深度自訂很重要。

3.4 bpftrace：cgroup 觀測的瑞士刀

bpftrace 是 awk-like 的 BPF 高階語言，一行 就能寫複雜觀測邏輯。對 cgroup 觀測特別有用——內建變數與 helper 直接支援。

關鍵內建：

  • cgroup：當前 task 的 cgroup ID（uint64）
  • cgroupid("/sys/fs/cgroup/myapp")：把路徑轉成 cgroup ID
  • pid、tid、comm、kstack、ustack：標準 bpftrace 變數

10 個 cgroup-aware 實戰 one-liner

# 1. 計算特定 cgroup 內每個 process 的 syscall 次數
bpftrace -e '
tracepoint:raw_syscalls:sys_enter
/cgroup == cgroupid("/sys/fs/cgroup/system.slice/myapp.service")/
{ @[comm] = count(); }
'

# 2. 看特定 cgroup 內哪些檔案被開啟
bpftrace -e '
tracepoint:syscalls:sys_enter_openat
/cgroup == cgroupid("/sys/fs/cgroup/myapp")/
{ printf("%s opened %s\n", comm, str(args->filename)); }
'

# 3. 統計特定 cgroup 內 TCP connect 目的 port
bpftrace -e '
kprobe:tcp_v4_connect
/cgroup == cgroupid("/sys/fs/cgroup/myapp")/
{ @[comm, ntop(((struct sock *)arg0)->__sk_common.skc_daddr)] = count(); }
'

# 4. 量測特定 cgroup 內 read syscall 的 latency 分布
bpftrace -e '
tracepoint:syscalls:sys_enter_read
/cgroup == cgroupid("/sys/fs/cgroup/myapp")/
{ @start[tid] = nsecs; }

tracepoint:syscalls:sys_exit_read
/@start[tid]/
{ @us = hist((nsecs - @start[tid]) / 1000); delete(@start[tid]); }
'

# 5. 找出特定 cgroup 內的 page fault 次數
bpftrace -e '
software:page-faults:1
/cgroup == cgroupid("/sys/fs/cgroup/myapp")/
{ @[comm] = count(); }
'

# 6. 看特定 cgroup 內 disk IO 的 size 分布
bpftrace -e '
kprobe:vfs_read,kprobe:vfs_write
/cgroup == cgroupid("/sys/fs/cgroup/myapp")/
{ @[func] = hist(arg2); }
'

# 7. 找特定 cgroup 內 kill signal 的發送
bpftrace -e '
tracepoint:signal:signal_generate
/cgroup == cgroupid("/sys/fs/cgroup/myapp")/
{ printf("%s killed pid=%d sig=%d\n", comm, args->pid, args->sig); }
'

# 8. 監控特定 cgroup 內 fork 速率
bpftrace -e '
tracepoint:sched:sched_process_fork
/cgroup == cgroupid("/sys/fs/cgroup/myapp")/
{ @forks = count(); }
interval:s:1
{ printf("forks/sec: %d\n", @forks); clear(@forks); }
'

# 9. 看特定 cgroup 內哪些 process 正在 schedule
bpftrace -e '
tracepoint:sched:sched_switch
/cgroup == cgroupid("/sys/fs/cgroup/myapp")/
{ @[comm] = count(); }
interval:s:5
{ exit(); }
'

# 10. 統計特定 cgroup 內 TCP retransmit
bpftrace -e '
kprobe:tcp_retransmit_skb
/cgroup == cgroupid("/sys/fs/cgroup/myapp")/
{ @retransmits = count(); }
interval:s:5
{ printf("retransmits in last 5s: %d\n", @retransmits); clear(@retransmits); }
'

3.5 BCC 工具與 cgroup

BCC（BPF Compiler Collection）提供大量現成 tools。多數支援 --cgroupmap 或類似參數做 cgroup 過濾：

BCC 工具用途cgroup 過濾範例
opensnoop 追蹤檔案開啟 opensnoop --cgroupmap=/sys/fs/cgroup/myapp
execsnoop 追蹤 exec syscall execsnoop --cgroupmap=/sys/fs/cgroup/myapp
tcpconnect 追蹤 TCP connect tcpconnect --cgroupmap=/sys/fs/cgroup/myapp
biolatency block IO 延遲分布 本身不直接支援，用 bpftrace 自寫
profile CPU profiler（flamegraph 用） 支援 cgroup 過濾
memleak 記憶體 leak 偵測 per-pid，搭配 cgroup.procs 過濾 PID

使用建議：先用 bpftrace 探索（一行寫完），確認方向後用 BCC 寫成 production tool（Python wrapper、CO-RE 支援），最後可以用 libbpf C 寫長期跑的 daemon。

3.6 bpftool：cgroup BPF 的 debug 工具

bpftool 是 kernel-shipped 工具，操作 cgroup BPF programs 的 swiss army knife： 

# 看所有 cgroup 與 attached programs
sudo bpftool cgroup tree

# 看特定 cgroup 上的 programs
sudo bpftool cgroup show /sys/fs/cgroup/system.slice/docker-XXX.scope

# 看所有 loaded BPF programs
sudo bpftool prog list

# dump 某個 BPF program 的 bytecode（人類可讀）
sudo bpftool prog dump xlated id 234

# dump JIT 編譯後的 native code
sudo bpftool prog dump jited id 234

# 看 BPF map 內容
sudo bpftool map dump id 567

# 看 BPF program 印的 log（透過 bpf_trace_printk）
sudo bpftool prog tracelog
# 同等於：cat /sys/kernel/debug/tracing/trace_pipe

典型 debug 流程：

  1. bpftool cgroup tree 看整個系統的 cgroup BPF 概況
  2. 聚焦到問題 cgroup：bpftool cgroup show /path/to/cgroup
  3. 查特定 program：bpftool prog show id N
  4. 必要時 dump 看內容：bpftool prog dump xlated id N
  5. 看 BPF program 是否有印 log：bpftool prog tracelog

3.7 性能與 overhead

eBPF 觀測看似該很慢——對每個 syscall 跑一段 BPF code，怎麼會快？實際上 overhead 多半 1-3%，原因：

  • JIT 編譯：BPF bytecode 在 attach 時 JIT 成 native code，執行接近 C 速度。
  • Verifier 強制 bounded loops：BPF program 不能有無限迴圈，最大 instruction 數限制。每次執行時間有上界。
  • Per-CPU maps：不需要 lock，無 cacheline contention。
  • Early filtering：cgroup filter 在 kernel 內就過濾掉不感興趣的事件，不送到 userspace。
  • In-kernel aggregation：用 BPF map 在 kernel 內聚合，userspace 只讀最終結果。

對比真實 production 數字（來自 michaeleakins.com 2025 年文章）：

  • DataDog APM agent：~28% CPU overhead
  • OpenTelemetry SDK（manual instrumentation）：~15% CPU overhead
  • libbpf-based eBPF observer：~1.8% CPU overhead

差距巨大。對 latency-critical workload，eBPF 是唯一能在 production 全量採集 metrics 不影響應用的方案。

3.8 實戰：找出 noisy neighbor

典型場景：multi-tenant 機器上某個 service 變慢，但不知道是被誰拖慢的。 傳統工具（top、iostat、perf）只能看「機器整體誰在做什麼」，不容易精確找出受害者與肇事者的關係。

eBPF 流程： 

# Step 1: 確認受害者的 cgroup
systemctl status myapp.service
# 找出 ControlGroup: /system.slice/myapp.service

# Step 2: 量化 noisy neighbor 的影響——在不同維度比較
# CPU off-CPU time（被搶走的時間）
sudo bpftrace -e '
kprobe:finish_task_switch
{
    $prev = (struct task_struct *)arg0;
    $cgid = $prev->sched_task_group->css.cgroup->kn->id;
    @off_cpu[$cgid] = count();
}
interval:s:5 { print(@off_cpu); clear(@off_cpu); exit(); }
'

# Step 3: 找出搶走 CPU 的肇事者——schedule 次數最多的 cgroup
sudo bpftrace -e '
tracepoint:sched:sched_switch
{
    @switches[cgroup] = count();
}
interval:s:5
{
    print(@switches);
    clear(@switches);
    exit();
}
'

# Step 4: 確認對應 cgroup ID 是哪個 service
# cgroup ID 對應 inode number，可以用：
sudo find /sys/fs/cgroup -inum <ID>

真實案例：某 production 環境 Redis 偶發 spike。用 bpftrace 看到一個 cron job cgroup 在 spike 時做大量 page fault——它跑大型 batch 報表時誘發 page reclaim，連帶讓 Redis 的 page 被 evict，造成 latency。 修法：對 cron cgroup 設 memory.high + io.weight 較低，避免它影響其他 service。

3.9 觀測架構整合

實戰建議的觀測架構：

  1. Layer 1（per-incident）：bpftrace one-liner——快速 debug，現場確認假設。
  2. Layer 2（per-host metrics）：BCC 工具或自寫 libbpf agent，export 到 Prometheus。整合 PSI、cgroup events 等。
  3. Layer 3（cluster-wide）：Pixie、Parca、Cilium Hubble 等成熟工具——自動化 service map、flame graph、network flow。

三層職責清楚：bpftrace 是「探查」工具、libbpf agent 是「採集」工具、Pixie 是「視覺化分析」工具。 理解 BPF cgroup program 機制讓你在三層都能 debug——出問題時不用等廠商工具。

05 · Pitfalls

常見陷阱

陷阱 1 · cgroup 內建變數在某些 hook 點不可用

bpftrace 的 cgroup 變數對 當前 task 有效。但有些 hook（如 BPF_PROG_TYPE_XDP）發生在還沒進入 task context 時，cgroup 變數會回 0 或失敗。 修法：不確定時用 tracepoint 或 kprobe（兩者都有 task context），不要用太底層的 hook。如果一定要用，需要從 socket / sk_buff 自己取 cgroup ID。

陷阱 2 · cgroupid() 找不到路徑

cgroupid("/sys/fs/cgroup/myapp") 在 service 還沒啟動或路徑寫錯時回 0——所有 task 都會被當成 cgroup ID 0，filter 失效（看起來是「所有 task 都 match」）。 修法：先ls 確認路徑存在。或在 bpftrace BEGIN 印出 cgroupid() 結果驗證。

陷阱 3 · BPF program load 失敗看不到原因

bpftool prog load my.o /sys/fs/bpf/my 失敗，只回 "Operation not permitted"——其實是 verifier reject。 修法：加 --debug 參數，或dmesg | tail -50 看 verifier 詳細訊息（哪行 instruction 失敗、為什麼）。也可以用 libbpf_set_print() 把 libbpf log 印出來。

陷阱 4 · bpftrace 跑久了 OOM

@latency[tid] = nsecs 這類 map 沒清理，跑久了 map 會持續變大直到 BPF map size 上限。 修法：用 delete(@latency[tid]) 在 exit handler 清掉、或用 LRU map（BPF_MAP_TYPE_LRU_HASH）。 bpftrace 預設 max_map_keys 是 4096，超過就會 silent drop——也是常見 debug 點。

陷阱 5 · cgroup ID 對 systemd transient unit 變動

systemd-run 啟動的 transient unit 每次 PID 與 cgroup ID 都不同。寫死 cgroupid("/path/to/transient") 在 unit 重啟後失效。 修法：debug 用 transient unit 時即時取 cgroup ID（每次重新跑 bpftrace 命令）。production 用persistent unit（systemctl edit）寫穩定路徑。

陷阱 6 · BCC 與 kernel headers 版本不匹配

BCC tools 報 "Unable to find kernel headers" 或 "kernel struct definition mismatch"——多半是 kernel headers 沒裝、或裝了但版本與 running kernel 不符。 修法：sudo apt install linux-headers-$(uname -r)。BTF + CO-RE 機制成熟後此問題減少（kernel 5.4+ + 新 BCC + libbpf 不需要 headers）。

陷阱 7 · perf event 與 cgroup BPF 互動

想用 perf record -G /sys/fs/cgroup/myapp 配合 BPF 一起用——遇到 perf 採不到 sample 或 sample 不準的問題。 原因：perf 與 BPF 都用 perf event 子系統，某些情況下會搶 fd 或 buffer。 修法：跑 perf 時先停 BPF observer。或改用 BCC 的 profile 工具（也是 perf-based 但內建 cgroup filter）。

陷阱 8 · 容器內跑 bpftrace 失敗

在容器內 bpftrace -e ... 直接 fail——多半是 container 沒給 BPF 相關 capability。 修法：跑 container 時加 --privileged 或--cap-add SYS_ADMIN --cap-add BPF --cap-add PERFMON。 production 不建議在 container 內跑 bpftrace；該在 host 跑（本來就是 system-wide 觀測工具）。

  2. 容器 start latency 通常多少？哪些因素影響？
  3. container runtime（runc/crun）對這個 latency 影響有多大？

07 · Self-check

自我檢查題

Q1.eBPF 與 cgroup 怎麼互補？per-cgroup 過濾為什麼是 multi-tenant 環境的關鍵能力？
互補關係：
  • cgroup：把 process 組織成樹、定義資源邊界——是結構。
  • eBPF：在 kernel 跑可程式化的觀測與控制——是工具。
兩者結合：用 eBPF 對 cgroup 邊界做精確觀測與控制。 per-cgroup 過濾的價值（multi-tenant 環境）：
  1. 隔離：A 容器的 debug 不影響 B 容器的觀測。
  2. 精準：「看 myapp 容器的 syscall」、不是「看整機 syscall 然後 grep」。
  3. 低 overhead：在 kernel 早期就 filter 掉不感興趣的事件——overhead 多半 1-3%（vs sidecar agent 10-40%）。
  4. 自動跟隨 process 生命週期：新 fork 自動納入、process 移出 cgroup 自動移除。
傳統工具做不到：strace 只 attach 單 PID、tcpdump 是 system-wide、perf record 雖然有 -G 但不能對 syscall 過濾。eBPF 是唯一能做到的。
Q2.BPF cgroup program 的 9 種類型分別 attach 在什麼 hook？哪個是 v2 device controller 用的？
主要類型與 hook：
  • CGROUP_DEVICE：device access（mknod / read / write）— v2 device controller 用這個
  • CGROUP_SKB：network packet（ingress / egress）
  • CGROUP_SOCK：socket create
  • CGROUP_SOCK_ADDR：connect / bind / sendmsg / recvmsg（IPv4/IPv6 各有版本）
  • SOCK_OPS：TCP 狀態變化（connect / accept / retransmit）
  • CGROUP_SYSCTL：sysctl read / write
  • CGROUP_SOCKOPT：setsockopt / getsockopt
v2 device controller：用 BPF_PROG_TYPE_CGROUP_DEVICE。Ch.11 講過 v2 完全沒有 device.allow / deny 介面，純 BPF。 共同特性：每個 program type 接收特定 context struct，return 1 = allow、0 = deny（操作回 EPERM）。一旦 attach 到 cgroup，cgroup 內所有 process 自動受影響——包括之後 fork 的。
Q3.bpftrace 怎麼對特定 cgroup 過濾？cgroup 變數與 cgroupid() 函數差別是什麼？
過濾語法： 
tracepoint:xxx:yyy
/cgroup == cgroupid("/sys/fs/cgroup/myapp")/
{ ... }
cgroup 內建變數：當前 task 的 cgroup ID（uint64）。在有 task context 的 hook 內可用（tracepoint、kprobe、uprobe）。 cgroupid() helper 函數：把路徑轉成 cgroup ID。讀的是 cgroup 目錄的 inode number。 用法：兩者結合做 filter——cgroup（task 的 ID）== cgroupid("/path"，路徑的 ID)。如果 task 在這個 cgroup 內，兩個值相等。 注意：
  • cgroup 變數在無 task context 的 hook（如 XDP）會回 0 或失敗
  • cgroupid() 對不存在路徑回 0——所有 task 都會 match（因為 cgroup ID 也可能是 0），filter 失效
  • systemd transient unit 每次 ID 不同，要即時取
Q4.bpftool 是做什麼的？典型 debug cgroup BPF 流程怎麼走？
bpftool：kernel-shipped 工具，操作 BPF objects（programs / maps / cgroup attachments / links）。是 cgroup BPF debug 的瑞士刀。 典型 debug 流程：
  1. bpftool cgroup tree：看整個系統的 cgroup BPF 概況（哪些 cgroup 有 attached programs）
  2. 聚焦問題 cgroup：bpftool cgroup show /sys/fs/cgroup/PATH 看 attach 的 program ID 與 attach type
  3. 查特定 program：bpftool prog show id N 看 type、name、loaded by 等元資料
  4. 必要時 dump bytecode：bpftool prog dump xlated id N（人類可讀的 BPF 指令）
  5. 看 BPF program 印的 log：bpftool prog tracelog（等同 cat /sys/kernel/debug/tracing/trace_pipe）
  6. 查 BPF map：bpftool map dump id M（看程式內部累積的資料）
實戰場景：
  • 「為什麼 docker 容器突然不能 mknod？」→ 看 container cgroup 上的 device program 內容
  • 「哪個 BPF program 在跑？」→ bpftool cgroup tree
  • 「BPF program 有印錯誤訊息嗎？」→ bpftool prog tracelog
Q5.為什麼 eBPF 觀測 overhead 這麼低？跟 sidecar agent 比有何優勢？
為什麼低 overhead（1-3%）：
  1. JIT 編譯：BPF bytecode 在 attach 時 JIT 成 native code，執行接近 C 速度（不是 interpret）。
  2. Verifier 強制 bounded：BPF program 不能有無限迴圈，最大 instruction 數限制。每次執行時間有上界。
  3. Per-CPU maps：不需要 lock，無 cacheline contention。
  4. Early filtering：cgroup filter 在 kernel 內就過濾掉不感興趣的事件——不感興趣的事件只多了一個 if 判斷。
  5. In-kernel aggregation：用 BPF map 聚合（histogram、count），userspace 只 export 結果而非每事件。
跟 sidecar agent 比：
  • sidecar：應用每次系統呼叫都被 trace、所有資料拷到 userspace、序列化成網路訊息送出去。每個 step 都有 overhead，加總可達 10-40%。
  • eBPF：trace 在 kernel 內跑 BPF；早期 filter 大幅減少資料量；只 export 聚合後的指標。production 數字 1-3% 是常態。
具體數字（Michael Eakins 2025）：DataDog APM agent ~28%、OpenTelemetry SDK ~15%、libbpf-based eBPF observer ~1.8%。差距巨大。
Q6.實戰問題：multi-tenant 機器某 service 偶發 latency spike。怎麼用 eBPF 找到 noisy neighbor？
診斷流程：
  1. 確認受害者 cgroup：systemctl status myapp.service 找 ControlGroup path
  2. 確認受害類型：先看 PSI——是 CPU、memory、IO 哪個 pressure 高？
  3. 找肇事者 cgroup：用 bpftrace 統計事件類型最多的 cgroup
    • CPU 競爭：tracepoint:sched:sched_switch + cgroup count
    • memory 競爭：software:page-faults + cgroup count
    • IO 競爭：kprobe:vfs_read,vfs_write + cgroup count
  4. 反查 cgroup ID 對應 path：find /sys/fs/cgroup -inum <ID>
  5. 確認假設：把肇事者 cgroup 暫時降權（io.weight 設低、memory.high 設緊）後看受害者是否恢復
真實案例：某 production Redis 偶發 spike。bpftrace 看到 cron cgroup 在 spike 時做大量 page fault——它跑大型 batch 報表時誘發 page reclaim，連帶讓 Redis 的 page 被 evict。 修法：對 cron cgroup 設 memory.high + io.weight 較低，避免影響其他 service。
Q7.bpftrace one-liner、BCC tool、libbpf agent 三者該怎麼選？
選擇邏輯：
  • bpftrace one-liner：探查、debug 場景。「不知道是不是這個問題、先看一下」。一行寫完，5 秒看到結果。不適合 production 長期跑（無法 export 到 Prometheus、無 alerting）。
  • BCC tool：semi-production。Python wrapper 包 BPF program，有完整 CLI 介面。適合 debug team 共用、寫成可重複使用的 tool。CO-RE 支援後可移植性好。
  • libbpf agent：production 長期跑。C/Go/Rust 寫的 daemon，integrate Prometheus exporter，運維友好。常見於 service mesh、APM 工具。
典型工作流程：
  1. 用 bpftrace 探查問題、確認假設（「真的有這個問題嗎？」）
  2. 確認方向後寫 BCC script，team 共用 tool
  3. 需要長期 monitor 時改寫成 libbpf agent，整合到 Prometheus / Alertmanager
業界產品：Pixie、Cilium Hubble、Parca 都是 libbpf-based。bpftrace 適合個人 debug。BCC 介於兩者之間，適合 team 級工具。
Q8.實作思考題：你要為公司設計一套 cgroup-aware eBPF 觀測架構，從個人 debug 到 cluster-wide。請描述完整三層設計與工具選擇。
三層觀測架構： 
[Layer 1: 個人 debug（per-incident）]
工具：bpftrace、BCC tools
場景：「為什麼這個 service 突然慢？」
- bpftrace one-liner 即時 debug
- BCC tools (opensnoop, execsnoop, biolatency) 標準工具
- 訓練 team 寫 cgroup-aware bpftrace
- 安裝：每台 host 都裝 bpftrace + BCC

[Layer 2: 主機級監控（per-host metrics）]
工具：libbpf-based agent
場景：「節點上每個 cgroup 健康度」
- 自寫 libbpf agent，CO-RE 確保跨 kernel 版本
- export per-cgroup metrics 到 Prometheus
  * syscall rate per service
  * IO latency histogram per service  
  * page fault rate per service
- DaemonSet 部署，每 node 一個

[Layer 3: cluster-wide 視覺化]
工具：Pixie / Parca / Cilium Hubble
場景：「整個 cluster 服務拓撲、性能熱點」
- Pixie：service map、HTTP request tracing
- Parca：continuous profiling，所有 service flame graph
- Cilium Hubble：network flow visibility
- 整合 Grafana 統一 dashboard

[供應商選擇考量]
- 開源 vs 商業：Pixie 開源、DataDog/Dynatrace 商業
- 自建 vs SaaS：自建省成本但要團隊投入
- kernel 要求：多數工具需 kernel 5.4+（CO-RE）

[Runbook 整合]
alert 觸發 (Layer 2 Prometheus) 
  → SRE 用 bpftrace (Layer 1) 確認
  → 看 Pixie service map (Layer 3) 找關聯
  → 修復後 Layer 2 metrics 驗證
關鍵設計決策：
  1. 三層分工清楚：debug / metrics / visualization 各有專責工具。
  2. CO-RE 是基礎：跨 kernel 版本可移植，省維護成本。
  3. cgroup-aware 是核心：multi-tenant 環境必須能 per-service 觀測。
  4. 低 overhead 一致性：三層都基於 eBPF，total overhead < 5% CPU。比傳統 sidecar 大幅優於。
  5. 逐步推行：先 Layer 1 訓練團隊、再 Layer 2 部署 metrics、最後 Layer 3 視覺化。
