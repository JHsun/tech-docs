Go 語言整合與容器化開發

Go 是雲原生時代最主流的語言之一——Kubernetes、Docker、containerd、Prometheus、etcd 都用 Go 寫。 但 Go runtime 長期忽略 cgroup 限制，把 host CPU 數誤認為自己的——導致 Kubernetes Pod 內 GOMAXPROCS 過高、 goroutine 大量爭搶被 CFS throttle、tail latency 飆漲。10 年來解法靠社群 library（uber-go/automaxprocs）。 2025 年 8 月 Go 1.25 發布，container-aware GOMAXPROCS 終於進入官方 runtime——「Just upgrade go.mod 就好」。 本章拆解 Go GMP scheduler 與 cgroup 互動、Go 1.25 內建機制、automaxprocs 仍存在的價值、GOMEMLIMIT 怎麼配合 memory.high。

01 · Highlights

本章重點

  1. Go 1.25 內建 container-aware GOMAXPROCS（2025-08）：runtime 自動讀 cgroup CPU quota，動態調整 GOMAXPROCS。Just bump go.mod to 1.25，不需 import library。
  2. uber-go/automaxprocs 仍有價值：Go 1.24- 仍是必備；Go 1.25+ 在「需要對 cpu.weight（request）而非 cpu.max（limit）反應」、「需要 log 變化」、「需要 round-down 而非 round-up」場景仍勝出。
  3. Go 1.19+ 的 GOMEMLIMIT 與 cgroup memory.max 配合：GC 提前介入避免 OOM。配合 cgroup memory.high 形成「三層保護」（GC trigger → kernel throttle → kernel OOM）。
  4. GMP scheduler 對 CFS throttle 極敏感：Go runtime 大量 goroutine context switch，被 CFS throttle 100ms 整個應用停擺——比一般語言更怕 throttle。
  5. Round up vs round down 是設計選擇：Go 1.25 round up（更積極用 CPU），automaxprocs round down（保守、避免 throttle）。沒有絕對對錯，取決 workload。

最新動態（2025–2026）

2025-08-12Go 1.25 發布，container-aware GOMAXPROCS 內建。Linux 上 runtime 自動讀 cgroup v2 cpu.max，動態調整。「Production-ready out-of-the-box」（Go team 官方說法）。
2025實測效果顯著：開發者報告 K8s deployment 在 Go 1.25 後 CPU 使用率下降 35%、p99 latency 改善（DEV community / Medium 多篇）。

2025Go 1.25 動態追蹤 cgroup 變化：~30 秒重新檢查一次 cpu.max。配合 K8s 1.36 In-Place Pod Resize 形成「K8s 改 limit → 30 秒後 Go 自動跟上」的閉環。

2026 年的時點，Go 是第一個 mainstream 語言把 cgroup-aware 內建到 runtime——是雲原生時代的成熟訊號。但理解原理仍重要：什麼時候 Go 1.25 內建夠用、什麼時候要自己處理（如 only-request 場景），這章把這些講透。

03 · Core Concepts

核心內容

3.1 Go 的 GMP scheduler 與 GOMAXPROCS

Go runtime 的核心是 GMP scheduler：

  • G：goroutine（輕量 user-space coroutine）
  • M：OS thread（kernel-level）
  • P：processor（goroutine 排程的虛擬 CPU）

GOMAXPROCS 決定P 的數量——也就是同時能跑 Go code 的 OS thread 上限。 Go 1.5+ 預設 GOMAXPROCS = runtime.NumCPU()（host 邏輯 CPU 數）。

3.2 Go 1.25 container-aware GOMAXPROCS

2025 年 8 月 Go 1.25 發布，把 container-aware 內建到 runtime：

  • 啟動時：runtime 讀 /sys/fs/cgroup/cpu.max（v2）或 cpu.cfs_quota_us + cpu.cfs_period_us（v1）
  • 計算 effective CPU = quota / period
  • GOMAXPROCS = min(NumCPU, ceil(effective CPU))（注意是 round up）
  • 最低 GOMAXPROCS = 2（保留基本平行度，除非 host 只有 1 CPU）
  • 執行中：每 ~30 秒重新檢查一次，cpu.max 變化時自動更新

啟用方式：升 go.mod 到 1.25.0 或更高即可，不需 import library、不需設環境變數。 

// go.mod
module myapp
go 1.25

// main.go
package main

import (
    "fmt"
    "runtime"
)

func main() {
    fmt.Printf("GOMAXPROCS = %d\n", runtime.GOMAXPROCS(0))
    // 容器 limit = 2 CPU 時，輸出 "GOMAXPROCS = 2"
    // host 64 CPU 但容器無限制時，輸出 "GOMAXPROCS = 64"
    fmt.Printf("NumCPU = %d\n", runtime.NumCPU())
    // 永遠是 host 邏輯 CPU 數（這個沒變）
}

關掉自動行為：手動呼叫 runtime.GOMAXPROCS(n)（n > 0）就會 disable 自動。

3.3 uber-go/automaxprocs 仍有價值

Uber 的 automaxprocs 是 Go 1.25 之前的事實標準，許多 production 仍在用。Go 1.25 之後 automaxprocs 仍有特定優勢：

差異點Go 1.25 內建uber-go/automaxprocs
觸發方式升級 go.mod 即可需 import 並 init
最小 GOMAXPROCS21（預設可調）
分數 CPU 處理round up（積極）round down（保守）
動態追蹤是（30 秒重檢）否（只 init 時）
Log 行為不 log預設 log（可調）
支援 cgroup v1是是
支援 cgroup v2是1.5+（早期版本只支援 v1）
對 cpu.weight 反應否（只看 cpu.max）否（也只看 limit）

什麼時候該用 automaxprocs（Go 1.25 之後）

  • 需要 round down：保守策略避免 throttle。Go 1.25 round up 在分數 CPU + 高並發 場景仍可能 throttle。
  • 需要 log GOMAXPROCS 變化：debug 用。Go 1.25 不 log（cleaner production output 但 debug 不便）。
  • 應用是 Go 1.24 或更早：等不及升級時的最佳折衷。
  • 需要 GOMAXPROCS=1：低資源 sidecar、極限制環境。Go 1.25 最低 2。

什麼時候直接用 Go 1.25

  • 新 service：升 go.mod 即可，少一個依賴。
  • K8s 動態 resize 場景：In-Place Resize 改 limit 時 Go 1.25 自動跟上。
  • 不在意 round up vs down：多數 service 在意的是「不要過度 throttle」，Go 1.25 已避免最糟情況。

3.4 GOMEMLIMIT 與 cgroup memory 配合

Go 1.19 引入 GOMEMLIMIT——讓 Go 主動提前 GC 而非等 OOM。在 cgroup 限制場景特別重要。

傳統 Go GC 行為：

  • GC 觸發看 GOGC（預設 100，意思是「記憶體使用達上次 GC 後 2 倍時 GC」）。
  • 沒考慮絕對記憶體上限——可能 GC 還沒跑就觸到 cgroup memory.max → OOM。

GOMEMLIMIT 行為：

  • 設定一個絕對軟上限。
  • 當記憶體接近這個值時，runtime 大幅提高 GC 頻率與強度。
  • 能避免（不能保證）OOM，犧牲部分 CPU 換記憶體穩定。
// 在 K8s 容器設定（建議）
spec:
  containers:
  - name: app
    env:
    - name: GOMEMLIMIT
      value: "900MiB"   # 比 memory.limit 小 10%
    resources:
      limits:
        memory: "1Gi"

三層保護機制（搭配 K8s Memory QoS）：

  1. GOMEMLIMIT 達 900MB：Go runtime 提高 GC，嘗試降低用量
  2. memory.high 達 950MB（Memory QoS 設）：kernel throttle，給 monitoring 介入時間
  3. memory.max 達 1024MB：kernel OOM kill

三層保護給應用提早警告與緩衝——不是直接從「沒事」跳到「OOM」。

3.5 自動讀 cgroup memory limit 設 GOMEMLIMIT

像 GOMAXPROCS 一樣自動，但 Go runtime 還沒 內建 GOMEMLIMIT 自動偵測。社群方案： 

// KimMachineGun/automemlimit
// go get github.com/KimMachineGun/automemlimit

package main

import (
    _ "github.com/KimMachineGun/automemlimit"
)

func main() {
    // GOMEMLIMIT 自動設為 cgroup memory.max 的 90%
}

或自己讀 cgroup 介面： 

// 自己讀 memory.max 設 GOMEMLIMIT
package main

import (
    "fmt"
    "os"
    "runtime/debug"
    "strconv"
    "strings"
)

func init() {
    // 讀 cgroup v2 memory.max
    data, err := os.ReadFile("/sys/fs/cgroup/memory.max")
    if err != nil {
        return
    }
    s := strings.TrimSpace(string(data))
    if s == "max" {
        return  // 沒設限
    }
    
    limit, err := strconv.ParseInt(s, 10, 64)
    if err != nil {
        return
    }
    
    // 設 GOMEMLIMIT 為 90%
    target := int64(float64(limit) * 0.9)
    debug.SetMemoryLimit(target)
    fmt.Printf("GOMEMLIMIT set to %d MiB\n", target / 1024 / 1024)
}

3.7 throttle 觀察與 alert

判斷 Go app 是否被 CFS throttle： 

# 看 cpu.stat 的 throttle 累計
cat /sys/fs/cgroup/cpu.stat | grep -E 'nr_throttled|throttled_usec'
# nr_throttled 持續增加 = throttle 嚴重

# 計算 throttle 百分比
nr_periods=$(awk '/nr_periods/{print $2}' /sys/fs/cgroup/cpu.stat)
nr_throttled=$(awk '/nr_throttled/{print $2}' /sys/fs/cgroup/cpu.stat)
throttle_pct=$(echo "scale=2; $nr_throttled * 100 / $nr_periods" | bc)
echo "Throttle: $throttle_pct%"

# > 5% 表示嚴重，需要：
# 1. 確認 GOMAXPROCS 設對
# 2. 升 cpu.max
# 3. 啟用 cpu.max.burst（kernel 5.14+）

Prometheus 採集 cAdvisor metrics： 

# 從 cAdvisor 取
container_cpu_cfs_throttled_seconds_total{pod="my-go-app"}
container_cpu_cfs_throttled_periods_total{pod="my-go-app"}

# 計算 throttle ratio
rate(container_cpu_cfs_throttled_periods_total[5m]) /
rate(container_cpu_cfs_periods_total[5m])

# Alert
- alert: GoAppThrottling
  expr: |
    rate(container_cpu_cfs_throttled_periods_total{pod=~".*-go-.*"}[5m])
    / rate(container_cpu_cfs_periods_total{pod=~".*-go-.*"}[5m]) > 0.05
  for: 5m
  annotations:
    summary: "Go app {{ $labels.pod }} CFS throttle > 5%"

3.8 GOGC 與 cgroup 的互動

GOGC（預設 100）控制 GC 觸發頻率。在 cgroup 限制場景的選擇：

  • 記憶體緊（< 1GB）：GOGC=50（更頻繁 GC、CPU 消耗高、記憶體用量低）
  • 記憶體寬（> 4GB）：GOGC=200（少 GC、CPU 省、記憶體用更滿）
  • CPU 緊（cpu.max=1）：GOGC 高一點減少 GC 開銷

但更推薦用 GOMEMLIMIT——不需要手動 tune GOGC，runtime 自動在接近 limit 時調整。

3.9 best practices

  • 升級到 Go 1.25：container-aware GOMAXPROCS 是 production must。
  • 設 GOMEMLIMIT：手動或用 automemlimit。建議 cgroup memory.max 的 90%。
  • 監控 CFS throttle 與 GC pause：兩個常見性能殺手。
  • K8s Pod 設 cpu/memory limit：Go 1.25 才有可讀資料。only-request 場景 Go runtime 仍會誤判。
  • K8s 1.36 In-Place Resize 配合 Go 1.25：閉環自動調節。
  • profiling 在 production 開啟：pprof endpoint 永遠開——cgroup throttle 時看得到 trace。

05 · Pitfalls

常見陷阱

陷阱 1 · 容器內 Go 1.24- 沒設 GOMAXPROCS 直接 production

Go 1.24- 容器內 GOMAXPROCS 預設等於 host CPU 數——Pod limit=2 但 host=64 時 GOMAXPROCS=64，嚴重 CFS throttle。修法：升 Go 1.25、import uber-go/automaxprocs、或手動設 GOMAXPROCS 環境變數。

陷阱 2 · K8s Pod 只設 request 沒設 limit，Go 1.25 仍誤判

Pod 設 cpu request=2 但沒 limit——cgroup 只有 cpu.weight 沒 cpu.max。Go 1.25 看不到絕對限制，仍按 host 設 GOMAXPROCS。修法：production 一律設 cpu limits（不是只 requests）。或用社群 library 讀 cpu.weight 反推。

陷阱 3 · 同時用 Go 1.25 + automaxprocs 衝突

Go 1.25 已內建，又 import automaxprocs——automaxprocs init 時呼叫 runtime.GOMAXPROCS(n) 會關掉 Go 1.25 動態追蹤。結果是 fixed GOMAXPROCS，K8s resize 後不會跟上。修法：升 Go 1.25 後移除 automaxprocs，或保留 automaxprocs（如需 round down 行為）但接受 dynamic 失效。

陷阱 4 · GOMEMLIMIT 設太緊導致 GC 風暴

GOMEMLIMIT 設成 cgroup memory.max 的 50%——應用接近 50% 時 runtime 瘋狂 GC，CPU 直接吃滿，吞吐崩潰。修法：GOMEMLIMIT 建議設 cgroup memory.max 的 85-95%。太緊不如放寬。

陷阱 5 · pprof CPU profile 看不到 throttle 影響

用 go tool pprof /debug/pprof/profile 看 CPU heavy 函式——但 throttle 期間 process 在睡覺 pprof 採不到，不知道 throttle 才是元兇。修法：用 wall-clock profiling（go tool pprof -wall ...）看實際時間流逝；同時看 cpu.stat 的 throttled_usec。

陷阱 6 · 用 NumCPU() 設並發數而非 GOMAXPROCS

worker pool 程式碼 numWorkers := runtime.NumCPU()——NumCPU 永遠是 host CPU 數，不會跟 cgroup limit 走。即使 Go 1.25，這個值不變。修法：用 runtime.GOMAXPROCS(0) 取當前值。或硬編設定值。

陷阱 7 · GOMAXPROCS < 2 導致 GC 瓶頸

容器只給 --cpus=0.5，automaxprocs round down 到 1——Go GC 是 concurrent 但需要至少一個 thread 給 mutator，GOMAXPROCS=1 時 GC 期間應用幾乎停擺。Go 1.25 強制最低 2 就是這原因。修法：給容器至少 1 整 CPU；如不可避，用 Go 1.25 內建（強制 2）。

陷阱 8 · Go runtime metrics 沒對 cgroup 做標籤

Prometheus 採 Go runtime metrics（go_goroutines、go_gc_duration_seconds），但沒 cgroup 上下文——多 Pod 時看不出哪個出問題。修法：metric 加 pod / namespace label（kube-prometheus-stack 自動處理）。並結合 cAdvisor cgroup metrics 看 throttle。

自我檢查題

Q1.Go 的 GMP scheduler 是什麼？GOMAXPROCS 控制什麼？
GMP scheduler 是 Go runtime 的核心：
  • G（goroutine）：Go 的輕量 user-space coroutine。比 OS thread 便宜（KB 級 stack）。
  • M（machine / OS thread）：kernel-level thread，跑 Go code 的實際 worker。
  • P（processor）：虛擬 CPU 概念，掛載 G queue，分配給 M 跑。
GOMAXPROCS 控制 P 的數量——也就是同時能跑 Go code 的 OS thread 上限。
  • Go 1.5+ 預設 = runtime.NumCPU()（host 邏輯 CPU 數）
  • Go 1.25+ 預設 = min(NumCPU, ceil(cgroup CPU 限制))
為什麼這個值關鍵：太小（1）→ 並發吞吐低；太大（> cgroup 限制）→ 大量 thread 爭搶被 CFS throttle、context switch 暴增、tail latency 飛漲。 NumCPU vs GOMAXPROCS：NumCPU 永遠是 host 邏輯 CPU 數（不會變），GOMAXPROCS 是 P 數量（可動態調整）。不要用 NumCPU 設 worker pool 大小！
Q2.Go 1.24- 在容器中為什麼會 throttle？解法是什麼？
throttle 根源：
  1. Go 1.24- 預設 GOMAXPROCS = runtime.NumCPU()。
  2. NumCPU 看 host 邏輯 CPU 數（如 32），不看 cgroup 限制（如 2）。
  3. Go runtime 起 32 個 worker thread 處理 goroutine。
  4. 32 個 thread 同時搶 cgroup 給的 2 CPU 配額。
  5. 每 100ms 配額用完，所有 thread 全停（CFS throttle）。
  6. 後果：context switch 4× 倍增；tail latency 從毫秒爆到 30+ 秒；GC 也被 throttle pause 暴漲。
解法：
  1. 升 Go 1.25（最簡單）：runtime 自動讀 cgroup cpu.max 設 GOMAXPROCS。
  2. uber-go/automaxprocs：Go 1.24- 的標準解法。import _ "go.uber.org/automaxprocs" 即可。
  3. 手動設環境變數：env: [{name: GOMAXPROCS, value: "2"}] 寫死成 cgroup 限制值。
  4. K8s downward API：用 resourceFieldRef 把 cpu limit 自動 inject 為環境變數。
實測效果：DEV community 報告「CPU 使用率下降 35%」、p99 latency 大幅改善（DEV community 文章）。
Q3.Go 1.25 內建 GOMAXPROCS 與 uber-go/automaxprocs 主要差別？什麼時候用哪個？
主要差別：
  • 觸發方式：Go 1.25 升 go.mod 即可；automaxprocs 需 import + init
  • 最小 GOMAXPROCS：Go 1.25 是 2；automaxprocs 是 1
  • 分數 CPU 處理：Go 1.25 round up（積極）；automaxprocs round down（保守）
  • 動態追蹤：Go 1.25 是（30 秒重檢）；automaxprocs 否（只 init 時）
  • Log：Go 1.25 不 log；automaxprocs 預設 log
  • cgroup v2 支援：Go 1.25 是；automaxprocs 1.5+
選 Go 1.25 內建：
  • 新 service，少一個依賴
  • K8s 動態 resize 場景（In-Place Resize）需要 dynamic GOMAXPROCS
  • 不在意 round up vs down
  • 需要 cleaner production output（不想 log）
選 automaxprocs：
  • 仍跑 Go 1.24 或更早
  • 需要 round down（保守策略避免 throttle）
  • 需要 log GOMAXPROCS 變化（debug）
  • 需要 GOMAXPROCS=1（極限制環境）
不建議：Go 1.25 + automaxprocs 同時用。automaxprocs 設 GOMAXPROCS 後 Go 1.25 動態追蹤就 disable 了。
Q4.round up 與 round down 的選擇邏輯？哪個比較好？
背景：cgroup CPU 限制可以是分數（如 2.5 CPU），但 GOMAXPROCS 必須整數。 Round up（Go 1.25）：
  • 2.5 → 3
  • 優：充分用 CPU 配額；高吞吐場景受益。
  • 缺：理論上更容易 throttle（partial second 內 thread 多於可用 CPU）。
  • 論點：throttle 是 200ms-level，多用 CPU 通常更好。
Round down（automaxprocs）：
  • 2.5 → 2
  • 優：保守、絕不 throttle；tail latency 穩定。
  • 缺：浪費部分 CPU 配額。
  • 論點：「分數 CPU 表示用戶刻意保留 burst」（automaxprocs 文件）。
哪個比較好？取決 workload：
  • latency-critical（API、DB）→ round down 更保險
  • throughput-critical（batch、ML）→ round up 更快
  • 多 Pod 同節點→ round down 公平避免抖動
  • 單 Pod 獨佔節點→ round up 用滿資源
實務：多數 service 兩者表現差不多。如果你不確定就用預設（Go 1.25 = round up）。發現嚴重 throttle 再考慮換 automaxprocs。
Q5.GOMEMLIMIT 是什麼？跟 cgroup memory limits 怎麼配合？
GOMEMLIMIT（Go 1.19+）：Go runtime 的軟記憶體上限。當記憶體接近這個值時，runtime 大幅提高 GC 頻率與強度，嘗試降低用量避免 OOM。 傳統 Go GC 行為（沒 GOMEMLIMIT）：
  • GC 觸發看 GOGC（預設 100，「記憶體成上次 GC 後 2 倍時 GC」）。
  • 沒考慮絕對上限——可能 GC 還沒跑就觸到 cgroup memory.max → OOM。
跟 cgroup 配合：三層保護： 
GOMEMLIMIT 達 900MB → Go runtime 提高 GC（嘗試降量）
              ↓（無效時）
memory.high 達 950MB（K8s Memory QoS）→ kernel throttle
              ↓（無效時）
memory.max 達 1024MB → kernel OOM kill
建議設定：
  • 絕對值：cgroup memory.max 的 85-95%
  • K8s 設法：環境變數 GOMEMLIMIT，或自動讀 cgroup（自寫或用 KimMachineGun/automemlimit）
注意：
  • GOMEMLIMIT 設太緊（如 50%）→ GC 風暴 CPU 吃滿
  • 不能保證避免 OOM，只是降低機率
  • 設 GOMEMLIMIT 後 GOGC 仍生效，兩者協作
Q6.怎麼判斷 Go service 在 K8s 內被 CFS throttle？怎麼處理？
判斷方式：
  1. 看 cpu.stat： 
    cat /sys/fs/cgroup/cpu.stat
# nr_throttled 持續增加 → throttle 在發生
# throttled_usec 顯示累計 throttle 時間
  2. 從 Prometheus（cAdvisor）： 
    rate(container_cpu_cfs_throttled_periods_total[5m]) /
rate(container_cpu_cfs_periods_total[5m]) > 0.05
  3. 從 Go pprof：用 wall-clock profile 看 process 是不是在睡覺： 
    go tool pprof -wall http://...:6060/debug/pprof/profile
  4. 看 cpu.pressure： 
    cat /sys/fs/cgroup/cpu.pressure
# some avg10 持續 > 5 = 有 task 等 CPU
處理方式（按優先級）：
  1. 升 Go 1.25：根本解法，runtime 自動匹配 cgroup。
  2. 確認 GOMAXPROCS = cgroup limit：可能應用手動 override 過高。
  3. 升 cpu.max：簡單粗暴但有效。
  4. 啟用 cpu.max.burst（kernel 5.14+）：允許短期 burst 不被 throttle。
  5. K8s 1.36+ In-Place Resize：動態擴容，配合 PSI HPA。
常見誤區：throttle > 0 不代表一定有問題——關鍵是 throttle 百分比（> 5%）與對應的 latency 影響。
Q7.Go 1.25 為什麼最低 GOMAXPROCS 設成 2？為什麼不是 1？
背景：Go runtime 的 GC 是 concurrent——GC 跟 mutator（應用 code）並行跑，但需要至少一個 thread 給 mutator。 GOMAXPROCS=1 的問題：
  1. 只有 1 個 P。
  2. GC 開始時 P 用來跑 GC worker。
  3. 應用 goroutine（mutator）要等 GC 釋放 P 才能跑。
  4. 結果：GC 期間應用幾乎停擺（不再是 concurrent）。
GOMAXPROCS=2 的好處：
  • GC worker 跑在一個 P，mutator 在另一個 P。
  • 真正的 concurrent GC，pause 時間大幅降低。
  • cgroup CPU burst（quota 突發）才能用上。
為什麼 automaxprocs 預設 1：
  • 歷史包袱（早期 GC 設計不同）。
  • 「用戶顯式設小 limit 應該尊重」的哲學。
例外：Go 1.25 在 host 只有 1 邏輯 CPU 時仍允許 GOMAXPROCS=1（如極限制 IoT）。 實務建議：給容器至少 1 整 CPU（cgroup --cpus=1.0），讓 Go 1.25 能用 GOMAXPROCS=1（host 1 CPU 場景）或 2（一般場景）。不要給 0.3、0.5 的分數 CPU——會大幅影響 GC 行為。
Q8.實作思考題：你管理 50 個 Go service 在 K8s 上跑，要全面升級到「cgroup-aware production-ready」。請設計完整方案。
完整方案（5 階段）： 
[Phase 1: 現況盤點（1 週）]
1. 列出所有 Go service 的版本：
   for s in $(list-services); do
       grep '^go ' $s/go.mod
   done

2. 分類：
   - Go < 1.19：沒 GOMEMLIMIT 支援，最舊
   - Go 1.19-1.24：要 automaxprocs
   - Go 1.25+：可不用 automaxprocs

3. 跑 audit：哪些 Pod 有 throttle？
   PromQL：rate(container_cpu_cfs_throttled_periods_total[1h]) /
          rate(container_cpu_cfs_periods_total[1h]) > 0.05

[Phase 2: K8s yaml 標準化（2 週）]
4. 對所有 Go Pod 確保：
   - 設 cpu limits（不只 requests）
   - 設 memory limits
   - GOMAXPROCS / GOMEMLIMIT 環境變數（暫時手動設）
   
5. 範例 yaml：
spec:
  containers:
  - name: app
    env:
    - name: GOMEMLIMIT
      valueFrom:
        resourceFieldRef:
          containerName: app
          resource: limits.memory
          divisor: "1"
      # 注意 K8s 沒辦法直接 inject 90% 值
      # 通常硬寫或 init container 算
    resources:
      limits: { cpu: "2", memory: "2Gi" }
      requests: { cpu: "2", memory: "2Gi" }

[Phase 3: 升級 Go 版本（4-6 週）]
6. 從 dev/staging 開始升 Go 1.25
7. CI 加 lint：禁止 Go < 1.21（GOMEMLIMIT 需要）
8. 對 critical service：phased rollout，1 個 instance 試 → 10% → 100%
9. 觀察指標：
   - GC pause time（go_gc_duration_seconds）
   - throttle rate（cAdvisor）
   - p99 latency
   - CPU 使用率（應該下降）

[Phase 4: 標準化中間件（4 週）]
10. 建立內部 Go template：
    - go.mod 強制 1.25+
    - import KimMachineGun/automemlimit（自動 GOMEMLIMIT）
    - 預設 expose pprof endpoint
    - 預設 expose /metrics 含 GOMAXPROCS、GOMEMLIMIT

11. 對 Go 1.24 service：保留 automaxprocs，標 TODO

[Phase 5: 持續優化]
12. 建立 dashboard：
    - 每 service throttle rate
    - GOMAXPROCS / cgroup CPU 對比
    - GC pause time
    - GOMEMLIMIT vs memory.max 比例

13. Alert：
    - throttle rate > 5%
    - GC pause > 100ms
    - GOMAXPROCS != cgroup CPU（不同步）

14. 定期 review：season change 時 traffic pattern 變化，
    重新算 limit 與 GOMEMLIMIT
關鍵原則：
  1. 升級分階段：critical service 最後，先驗證再 rollout。
  2. K8s yaml 與 Go runtime 雙管：兩邊都要設對才有效。
  3. 監控優先：升級前後 baseline 對比，否則看不到效果。
  4. 內部 template：避免新 service 重蹈覆轍。
陷阱：Go 1.25 升級可能暴露之前隱藏的 bug（dynamic GOMAXPROCS 變化導致 worker pool 行為改變）。CI 加 stress test 與 long-running test 可發現。
