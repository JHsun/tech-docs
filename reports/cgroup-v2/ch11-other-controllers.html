Part IV · Controllers 下 · Chapter 11

其他 controllers

除了 cpu、memory、io、cpuset、pid 五大主流 controllers，cgroup v2 還提供 hugetlb、rdma、misc、device、perf_event 五個 niche controllers， 加上 freezer 介面（cgroup.freeze）。本章把它們一次講完——多數人不會每天用到，但遇到對應場景時不知道有這個工具是大忌。 device controller 在 v2 完全改用 BPF，是最大架構變化；hugetlb 對 DB 與 ML 重要；misc 給特殊資源（GPU、加速卡）用。

01 · Highlights

本章重點

  1. device controller 完全改用 BPF：v2 沒有 device.allow / device.deny 介面檔案，改用 BPF_CGROUP_DEVICE program 控制 device access。架構變化最大。
  2. hugetlb 與 memory 是分開計量：HugeTLB pages 預設不算進 memory.current。kernel 6.6+ 提供 memory_hugetlb_accounting mount option 可選擇加進來。
  3. rdma 控制兩種資源：hca_handle（HCA 句柄）與 hca_object（QP/CQ/MR 等 verb 物件）。v2 介面與 v1 幾乎相同。
  4. misc controller 是 GPU/TPU/加速卡的標準介面：kernel 維護 misc.capacity，每個資源有名稱與 capacity；v2 透過 misc.max 設限。NVIDIA、Intel 等廠商陸續加入支援。
  5. cgroup.freeze 是core 介面不是 controller：寫 1 凍結整個 cgroup，所有 process 進入 D 狀態（uninterruptible sleep）。可作為 checkpoint/restore 與安全 debug 工具。
02 · Latest

最新動態（2025–2026）

2025 misc controller 在 GPU multi-tenancy 普及。NVIDIA 透過 NVIDIA Container Toolkit 利用 misc controller 實現 fine-grained GPU 資源分配；Intel oneAPI 與 AMD ROCm 跟進類似機制。
2024–2025 device controller v2 BPF 模式成熟。runc、crun、podman 都已完整實作 BPF_CGROUP_DEVICE 程式編譯，從 v1 device.allow / device.deny 自動轉換。容器使用者通常感受不到差別。
2024 kernel 6.6+ 引入 memory_hugetlb_accounting mount option，把 HugeTLB pages 算進 memory.current，整合 OOM、reclaim、protection 邏輯。對 DB（PG、MySQL、Oracle）的容量規劃影響較大。
2025 DMEM controller 引入（kernel 6.x），用於 device-attached memory（如 GPU VRAM、CXL memory）。仍在演進，主要場景是 AI/ML 訓練 GPU memory 配額。
2025 cgroup.freeze 整合到 systemd-run。systemd-run --freeze、systemctl freeze / systemctl thaw 已成標準操作，常用於 checkpoint 與安全 debug。

2026 年的時點，本章的「其他」controllers 多數已成熟穩定。 device 改 BPF 是最大架構變化，但對使用者透明（runtime 處理）； misc controller 因 GPU 普及成為新熱點； hugetlb 在 DB 與 HPC 場景仍是必備。本章把它們一次講完，每個都帶實戰場景。

3.2 hugetlb controller

Linux 提供「HugeTLB pages」——大尺寸記憶體頁（2MB 或 1GB），減少 TLB miss、提升 DB 與 HPC 性能。 但 HugeTLB 是預先分配的固定資源，使用前要先在 kernel 配置 pool（echo N > /proc/sys/vm/nr_hugepages）。

hugetlb controller 的角色：限制每個 cgroup 能用多少 HugeTLB pages。對 multi-tenant 共享 HugeTLB pool 的場景必要。

介面：每個頁面大小都有獨立介面（2MB、1GB 等）： 

# 列出可用頁面大小
ls /sys/kernel/mm/hugepages/
# hugepages-2048kB  hugepages-1048576kB

# 配置 pool（root cgroup 操作）
echo "100" > /proc/sys/vm/nr_hugepages       # 100 個 2MB pages = 200MB

# 在 cgroup 限制 HugeTLB 用量
echo "+hugetlb" > /sys/fs/cgroup/cgroup.subtree_control
mkdir /sys/fs/cgroup/db
echo "100M" > /sys/fs/cgroup/db/hugetlb.2MB.max     # 最多 100MB
cat /sys/fs/cgroup/db/hugetlb.2MB.current             # 當前用量
cat /sys/fs/cgroup/db/hugetlb.2MB.events              # max 事件
cat /sys/fs/cgroup/db/hugetlb.2MB.rsvd.max            # 預訂保留

HugeTLB 與 memory controller 是分開計量。kernel 6.6+ 才有 memory_hugetlb_accounting mount option 整合進 memory.current： 

# 啟用 hugetlb 計入 memory（mount 時加 option）
sudo mount -o remount,memory_hugetlb_accounting /sys/fs/cgroup

典型應用場景：

  • PostgreSQL：huge_pages=on 用 HugeTLB 給 shared_buffers，減少 TLB miss
  • MySQL InnoDB：large-pages 啟用，buffer pool 用 HugeTLB
  • Oracle DB：SGA 用 HugeTLB 是傳統做法
  • DPDK：必須用 HugeTLB（為了 zero-copy 與 polling 效能）
  • JVM：-XX:+UseLargePages 啟用

3.3 rdma controller

RDMA（Remote Direct Memory Access）是高效能網路技術——InfiniBand、RoCE、iWARP 等。 應用透過 verbs API 申請 HCA（Host Channel Adapter）資源（QP、CQ、MR 等）。

問題：HCA 資源有限。一個應用可以輕易把所有 QP（Queue Pair）耗盡，讓其他應用完全無法用 RDMA——這就是 rdma controller 要解決的。

介面：rdma.max（per-device-keyed），限制每個裝置的兩種資源： 

# 啟用 rdma controller
echo "+rdma" > /sys/fs/cgroup/cgroup.subtree_control
mkdir /sys/fs/cgroup/myapp

# 設定 mlx5_0 裝置的限制
echo "mlx5_0 hca_handle=2 hca_object=2000" > /sys/fs/cgroup/myapp/rdma.max

# 看當前用量
cat /sys/fs/cgroup/myapp/rdma.current
# mlx5_0 hca_handle=1 hca_object=523

兩種資源：

  • hca_handle：HCA 句柄數量（典型應用每個只開幾個）
  • hca_object：所有 verb 物件總數（QP、CQ、MR、PD、AH 等）

RDMA 資源歸屬：由創建者 cgroup 擁有（不是當前 cgroup）。process 搬 cgroup 不會搬 RDMA 資源——和 Memory Ownership 規則類似。

典型應用場景：HPC（MPI 應用）、分散式儲存（Ceph、GPFS）、AI 訓練（NCCL）、超低延遲交易系統。

3.4 misc controller

misc controller 是 v2 引入的通用資源限制機制（kernel 5.13+）。 kernel 不預設知道這些資源是什麼——由裝置驅動或 module 動態註冊資源類型與容量。

介面： 

# 看系統有哪些 misc 資源
cat /sys/fs/cgroup/misc.capacity
# 範例輸出：
# sev 15
# sev_es 15

# 查看當前 cgroup 的 misc 用量
cat /sys/fs/cgroup/myapp/misc.current

# 設定限制
echo "sev 5" > /sys/fs/cgroup/myapp/misc.max
echo "sev_es max" > /sys/fs/cgroup/myapp/misc.max

# 看事件
cat /sys/fs/cgroup/myapp/misc.events
# sev.max 0

已註冊的 misc 資源範例：

  • SEV / SEV-ES：AMD 加密虛擬機 ASID（kernel 5.13+ 支援）
  • NVIDIA GPU：透過 NVIDIA Container Toolkit 註冊「compute / video / 3D」等 GPU 資源
  • Intel TDX：Intel 機密運算 enclave 數量
  • FPGA / 加速卡：自訂註冊任意資源名稱

misc 的設計優勢：不需要每加一種新資源就改 kernel。drivers 註冊資源名稱與容量，kernel 通用機制處理 accounting 與 limit。

3.5 device controller：v2 的 BPF 革命

device controller 是 v2 最大架構變化——沒有任何介面檔案。kernel 文件原話：「The v2 devices controller provides no interface files; instead, device control is gated by attaching an eBPF (BPF_CGROUP_DEVICE) program to a v2 cgroup.」

v1 的 device controller 用 devices.allow、devices.deny、devices.list 三個檔案靜態白名單／黑名單。v2 砍掉這個，改用 BPF program——更靈活、可編程，但使用者不再直接寫介面檔案。

BPF_CGROUP_DEVICE program 簽章： 

// 來自 kernel: tools/testing/selftests/bpf/dev_cgroup.c
SEC("cgroup/dev")
int bpf_prog1(struct bpf_cgroup_dev_ctx *ctx) {
    short type = ctx->access_type & 0xFFFF;
    short access = ctx->access_type >> 16;
    
    // 例：禁止 mknod 設備
    if (type == BPF_DEVCG_DEV_BLOCK)
        return 0;  // deny
    
    // 例：允許 read 但禁 write 某 device
    if (ctx->major == 1 && ctx->minor == 3) {
        if (access & BPF_DEVCG_ACC_WRITE)
            return 0;
    }
    
    return 1;  // allow
}

實際使用：使用者通常不直接寫 BPF program。容器 runtime（runc、crun）內部把 OCI spec 的 device rules 編譯成 BPF program 自動 attach。 

# 看 cgroup 上 attach 了哪些 BPF program
sudo bpftool cgroup show /sys/fs/cgroup/system.slice/docker-XXX.scope
# 會看到 device cgroup BPF program

# 手動 attach（高階用法）
sudo bpftool cgroup attach /sys/fs/cgroup/myapp/ device pinned /sys/fs/bpf/my_device_filter

v1 → v2 比較：

面向v1 device controllerv2 device controller
介面devices.allow / deny / list無介面檔案，用 BPF
規則靜態白名單/黑名單程式邏輯（任意條件）
使用者直接寫 cgroup 檔案透過 runtime 或 bpftool
表達能力有限完整 BPF（含 helpers）
效能O(規則數)編譯後 JIT，極快

3.6 perf_event controller

最特殊的 controller——不需要明確啟用、自動可用。kernel 文件：「perf_event controller, if not mounted on a legacy hierarchy, is automatically enabled on the v2 hierarchy so that perf events can always be filtered by cgroup v2 path.」

用途：限制 perf record / perf stat 等工具只 profile 特定 cgroup 的 process： 

# profile 特定 cgroup 的 CPU events
sudo perf stat -e cycles,instructions \
    -G /sys/fs/cgroup/myapp \
    -- sleep 10

# 範例輸出：
#   1234567 cycles      myapp
#    987654 instructions myapp

沒有介面檔案。perf_event 不像其他 controller 提供 interface files，它純粹是讓 perf 工具「知道」cgroup 邊界，從 perf syscall (perf_event_open) 用 cgroup fd 過濾。

典型用途：debug 容器內應用性能、生產環境 profile 不影響其他 service、HPC 統計特定 task 的 cache miss。

3.7 cgroup.freeze：core 介面，不是 controller

嚴格說 freezer 不是 controller——它是 cgroup core 提供的核心介面。寫 1 進 cgroup.freeze，整個 cgroup 的所有 process 進入 D 狀態（uninterruptible sleep），全部停止執行。 

# 凍結整個 cgroup
echo 1 > /sys/fs/cgroup/myapp/cgroup.freeze

# 看狀態
cat /sys/fs/cgroup/myapp/cgroup.events
# populated 1
# frozen 1

# 解凍
echo 0 > /sys/fs/cgroup/myapp/cgroup.freeze

應用場景：

  • checkpoint / restore（CRIU）：先 freeze 才能可靠 dump 記憶體狀態
  • 安全 debug：凍結容器後用 gdb 連入分析，不會被 process 自身狀態變化干擾
  • 暫停 batch job：暫時釋放資源給其他 workload，之後解凍繼續
  • 容器 pause：docker pause、podman pause 內部就是用 cgroup.freeze
  • systemd-managed：systemctl freeze unit / systemctl thaw unit（systemd 246+）

注意：freeze 不影響 process 的 RAM / file descriptor / connection 狀態——只是停止執行。連線可能因 timeout 斷開（看對端應用），但 process 的 state 完整保留。解凍後 process 繼續執行，就像時間暫停一樣。

3.8 DMEM controller（演進中）

kernel 6.x 引入的新 controller，用於 device-attached memory——GPU VRAM、CXL memory、加速卡上的 memory。

現代 AI/ML 場景，GPU memory 是稀缺資源。傳統 cgroup memory controller 只管 system RAM，管不到 GPU VRAM。 DMEM controller 補上這塊。

狀態：仍在演進，介面與行為可能變化。使用 NVIDIA / AMD GPU 的人建議追 kernel docs 最新章節。

3.9 systemd 介面對應

systemd directivecgroup 介面備註

04 · Hands-on

實戰範例

Lab 1 · 配置 HugeTLB 給 PostgreSQL
# 1. 配置 host HugeTLB pool（需要 reboot 或 sysctl）
sudo sysctl -w vm.nr_hugepages=512   # 512 * 2MB = 1GB pool

# 2. 確認 pool 配置成功
cat /proc/meminfo | grep -i huge
# HugePages_Total:     512
# HugePages_Free:      512
# Hugepagesize:       2048 kB

# 3. 在 cgroup 啟用 hugetlb
echo "+hugetlb" | sudo tee /sys/fs/cgroup/cgroup.subtree_control

# 4. 啟動 PostgreSQL service，限制 HugeTLB 為 800MB
sudo systemd-run --slice=user.slice --unit=pg-test \
    --pty \
    bash

# 在 service 內：
CG=$(systemctl show -p ControlGroup --value pg-test.service)
echo "800M" | sudo tee /sys/fs/cgroup$CG/hugetlb.2MB.max

# 5. 看當前 HugeTLB 用量
cat /sys/fs/cgroup$CG/hugetlb.2MB.current

# 6. 檢查事件
cat /sys/fs/cgroup$CG/hugetlb.2MB.events

# 7. 清理
exit
sudo systemctl stop pg-test
Lab 2 · 觀察 misc.capacity 與限制 SEV ASID
# 1. 看系統有哪些 misc 資源
cat /sys/fs/cgroup/misc.capacity
# 範例：sev 15  sev_es 15  （AMD SEV）

# 如果系統沒這些資源，輸出可能空白——表示 driver 沒註冊任何 misc 資源

# 2. 啟用 misc controller
echo "+misc" | sudo tee /sys/fs/cgroup/cgroup.subtree_control
sudo mkdir /sys/fs/cgroup/vm-tenant

# 3. 限制這個 cgroup 最多用 5 個 SEV ASID
echo "sev 5" | sudo tee /sys/fs/cgroup/vm-tenant/misc.max

# 4. 看當前用量
cat /sys/fs/cgroup/vm-tenant/misc.current

# 5. 看事件
cat /sys/fs/cgroup/vm-tenant/misc.events
# sev.max 0   ← 還沒撞到上限

# 清理
sudo rmdir /sys/fs/cgroup/vm-tenant
Lab 3 · 體驗 cgroup.freeze
# 1. 啟動一個會打印計數的 service
sudo systemd-run --slice=user.slice --unit=counter \
    --pty \
    bash -c 'i=0; while :; do echo "count=$i"; i=$((i+1)); sleep 1; done'

# 2. 看到它在打印 count=1, count=2, ...

# 3. 在另一 terminal 凍結
CG=$(systemctl show -p ControlGroup --value counter.service)
echo 1 | sudo tee /sys/fs/cgroup$CG/cgroup.freeze

# 4. 看 events
cat /sys/fs/cgroup$CG/cgroup.events
# populated 1
# frozen 1

# 5. 觀察：原 terminal 應該停止打印（process 進入 D 狀態）

# 6. 解凍
echo 0 | sudo tee /sys/fs/cgroup$CG/cgroup.freeze

# 7. 觀察：原 terminal 又開始打印——而且計數從停止的地方繼續

# 等同於 systemctl freeze / thaw（systemd 246+）
# sudo systemctl freeze counter.service
# sudo systemctl thaw counter.service

# 清理
sudo systemctl stop counter

觀察點：freeze 是「時間暫停」——process 完全停止但 state 完整保留。

Lab 4 · 用 perf 配合 cgroup 過濾 profiling

sudo systemd-run --slice=user.slice --unit=cpu-app1 \

05 · Pitfalls

常見陷阱

陷阱 1 · 沒先配 HugeTLB pool 就設 hugetlb 限制

hugetlb.2MB.max=100M 設好了，但應用申請 HugeTLB 失敗（SIGBUS）？ 原因：host 的 HugeTLB pool 還沒配置（cat /proc/sys/vm/nr_hugepages 是 0）。 cgroup 限制只是「我能用多少」，pool 配置才是「系統有多少」。 修法：先 sysctl -w vm.nr_hugepages=N（boot time 配最穩），再設 cgroup 限制。

陷阱 2 · HugeTLB 預設不算進 memory.current

以為 memory.max=2G 已經包含 HugeTLB 用量？錯。 kernel 6.6+ 預設 HugeTLB 與 memory分開計量。應用用了 1G HugeTLB + 1.5G 一般 memory，cgroup 看起來才用 1.5G（不爆 max），但實際吃了 2.5G 系統記憶體。 修法：mount cgroup 時加 memory_hugetlb_accounting option，整合計量。或分別設 memory.max 與 hugetlb.max。

陷阱 3 · v2 device controller 期望 device.allow 介面

從 v1 過來的 sysadmin 想找 devices.allow、devices.deny——v2 沒有！ v2 device controller 純 BPF，沒有任何 cgroup 檔案介面。 修法：用 systemd 的 DeviceAllow= / DevicePolicy= directive，systemd 會自動編譯成 BPF program 並 attach。或直接寫 BPF（runc / crun 內部做的事）。

陷阱 4 · misc.capacity 是空的

cat /sys/fs/cgroup/misc.capacity 完全沒輸出？這是正常的——表示沒有任何 driver 註冊 misc 資源。 misc.capacity 只列出已註冊的資源（SEV、TDX、NVIDIA GPU 等）。沒有對應硬體或 driver，misc controller 不會自動產生 entry。 要用 GPU 配額？需要 NVIDIA Container Toolkit 或對應廠商 driver 註冊。

陷阱 5 · cgroup.freeze 對 RT process 失效

對含有 RT process（SCHED_FIFO / SCHED_RR）的 cgroup 寫 cgroup.freeze=1，RT process 仍在跑。 原因：cgroup freezer 用 D 狀態實現「凍結」，但 RT priority 高過 freezer 機制，無法可靠地暫停 RT task。 修法：對 RT 應用使用 SIGSTOP / SIGCONT 而非 cgroup.freeze。或先把 RT process 降為普通 priority 再 freeze。

陷阱 6 · perf_event 找不到 -G option（舊 kernel）

老 kernel（4.x 以前）的 perf 沒有 -G（cgroup filter）option。 perf stat -G /path 會回 unknown option 錯誤。 修法：升 perf 版本（與 kernel 對齊）。perf 4.11+ 才支援 cgroup v2 path。

陷阱 7 · RDMA controller 在容器中沒效果

在容器內設 rdma.max，發現完全沒擋住 RDMA 申請？ 原因：RDMA 資源歸屬於創建者 cgroup，不是當前 cgroup。如果應用是在 host process（如 daemon）創建 RDMA resource 後給容器用，這些 resource 屬於 host 的 cgroup，container 的 rdma.max 對它無效。 修法：確認 RDMA resource 由 container 內 process 創建（不要從 host 傳遞 fd 進去）。

陷阱 8 · DMEM controller 介面變動

DMEM controller 仍在演進，介面與行為可能在 kernel 版本間變化。 一個版本上的 dmem.max 設定，在另一版本上可能格式不同或介面位置不同。 修法：對 GPU memory 限制需求，目前以 NVIDIA / AMD container toolkit 為主——它們抽象了底層介面。 直接用 DMEM controller 屬於前沿，需要持續追 kernel docs。

練習 1 · 探索系統 misc 資源（5 分鐘）
# 看你的系統有哪些 misc 資源

07 · Self-check

自我檢查題

Q1.v2 device controller 為什麼沒有任何介面檔案？怎麼控制 device access？
沒有介面檔案的原因：v2 把 device 控制完全改用 BPF，這是架構性決定。 v1 用 devices.allow / deny / list 三檔案做靜態白名單／黑名單，但表達能力有限——例如「read 允許但 write 拒絕同一個 device」、「根據檔案屬性動態決定」這類複雜邏輯做不到。 v2 的方式：attach BPF_CGROUP_DEVICE program 到 cgroup。program 收到 bpf_cgroup_dev_ctx struct（含 major、minor、access_type），return 1 = allow、0 = deny。 使用者層：通常不直接寫 BPF，透過 systemd DeviceAllow=、DevicePolicy= 或 OCI runtime（runc、crun）內部編譯。對使用者透明。 檢查方法：bpftool cgroup show /sys/fs/cgroup/PATH 看 attach 了哪些 BPF program。
Q2.HugeTLB 與 memory controller 是什麼關係？memory.max 包含 HugeTLB 嗎？
預設關係：兩者分開計量。HugeTLB pages 不算進 memory.current 與 memory.max。 原因：HugeTLB 是預先分配的固定 pool，與一般 page allocator 走不同路徑。歷史上分開計量，避免雙重計算。 kernel 6.6+ 的選擇：mount cgroup 時加 memory_hugetlb_accounting option，HugeTLB 才會算進 memory.current。 實務影響：
  • 應用用了 1G HugeTLB + 1.5G 一般 RAM，沒啟用 mount option 時，memory.current 顯示 1.5G（不含 HugeTLB）。
  • 容量規劃要分別計算：memory.max = 一般 RAM 上限；hugetlb.NM.max = HugeTLB 上限；總實體記憶體 = memory.max + hugetlb 用量。
  • 啟用 memory_hugetlb_accounting 後，memory.max 變成「所有記憶體類型總和」，更直覺但與舊行為不相容。
Q3.misc controller 是什麼？kernel 怎麼決定有哪些 misc 資源？
misc controller：v2 引入的通用資源限制機制（kernel 5.13+）。kernel 不預設知道有什麼資源——由裝置驅動或 module 動態註冊。 註冊機制：driver 呼叫 kernel API 註冊「資源名稱 + 系統容量」。例如 AMD SEV driver 啟動時註冊「sev: 15」（15 個 ASID 可用）。 cat misc.capacity 看什麼：所有已註冊的資源。如果你看到空白，表示 driver 沒註冊。 已知會註冊 misc 資源的 driver / 場景：
  • AMD SEV / SEV-ES：sev、sev_es
  • Intel TDX：tdx
  • NVIDIA Container Toolkit：compute、video 等 GPU 資源
  • FPGA / 自訂加速卡：driver 自己註冊
設計優勢：kernel 不需要為每種新資源加 controller。driver 自己註冊，misc 通用 accounting 與 limit 機制處理。
Q4.cgroup.freeze 的行為是什麼？為什麼是「core 介面」而不是 controller？什麼場景會用？
行為：寫 1 進 cgroup.freeze，整個 cgroup 的所有 process 進入 D 狀態（uninterruptible sleep）。process 完全停止執行，但 state（記憶體、fd、connection）完整保留。寫 0 解凍。 為什麼是 core 介面：freezer 不分配資源，只控制 process 執行狀態——這是 cgroup core 的能力，不需要獨立 controller。cgroup.freeze 在所有 cgroup（即使沒啟用任何 controller）都可用。 典型應用場景：
  • checkpoint / restore：CRIU 先 freeze 才能可靠 dump 記憶體
  • 安全 debug：凍結容器後用 gdb 分析，process 不會跑掉
  • backup：對 DB freeze 一秒鐘，做 LVM snapshot，再 thaw（避免 partial commit）
  • 容器 pause：docker pause / podman pause 內部用這個
  • 暫時讓位：batch job freeze 給 latency-critical workload 用，後再 thaw
限制：對 RT process 不可靠（RT priority 高過 freezer）。
Q5.perf_event controller 為什麼自動啟用？沒有介面檔案怎麼用？
自動啟用原因：perf_event 不是「分配資源」的 controller，是「讓 perf 工具知道 cgroup 邊界」的機制。 kernel 認為這個能力是 cgroup v2 體系的一部分，所以自動啟用，使用者不需要寫 cgroup.subtree_control。 為什麼沒介面檔案：perf_event 不需要使用者寫設定。它的功能是讓 perf syscall (perf_event_open) 能用 cgroup fd 過濾——使用者透過 perf 工具間接用，不直接寫 cgroup 檔案。 使用方式： 
# profile 特定 cgroup 的 events
perf stat -e cycles,instructions \
    -G /sys/fs/cgroup/PATH \
    -- COMMAND
典型場景：
  • 容器化應用 profiling：只看容器內，不被 host 干擾
  • 多租戶環境：對特定租戶的 service 做性能分析
  • HPC：統計 worker thread 的 cache miss、IPC、branch prediction
Q6.RDMA controller 控制哪兩種資源？資源 ownership 怎麼決定？
兩種資源：
  • hca_handle：HCA（Host Channel Adapter）句柄數量。每個應用打開 RDMA device 時申請一個。一般應用只開幾個。
  • hca_object：所有 verb 物件總數，包含 QP（Queue Pair）、CQ（Completion Queue）、MR（Memory Region）、PD（Protection Domain）、AH（Address Handle）等。一般應用會申請大量（通常上千）。
Ownership 規則：RDMA 資源屬於創建者 cgroup，不是當前 cgroup。 process 搬 cgroup 時，已創建的 RDMA 資源仍歸原 cgroup。 實務影響：
  1. 容器化應用要在容器內創建 RDMA resource，才會歸容器 cgroup。
  2. 如果 host process 創建 RDMA resource 後傳 fd 給容器用，這些 resource 算在 host cgroup，container 的 rdma.max 對它無效。
  3. 規劃 HPC / NCCL / Ceph 等應用時，要確認 RDMA resource 創建路徑。
Q7.本章 6 個 controllers/介面（hugetlb、rdma、misc、device、perf_event、cgroup.freeze）誰是 niche 誰是常用？對一般 Kubernetes 環境哪些值得學？
常用程度排序（一般 K8s 環境）：
  1. device（BPF）：最常用。所有容器都隱含使用——但通常透過 OCI runtime 透明處理，使用者不直接操作。
  2. cgroup.freeze：常用。docker pause、systemctl freeze 等都用。debug 與 backup 常見。
  3. hugetlb：跑 DB（PostgreSQL、MySQL、Oracle）、DPDK、JVM 重度使用者必學。多數 web 服務用不到。
  4. perf_event：性能 debug 必備工具，但不是日常配置。
  5. misc：GPU / FPGA / 加速卡才會用到。對一般 K8s 環境較少，但 AI workload 越來越重要。
  6. rdma：HPC / 高效能網路才用。一般雲環境少見。
K8s 環境學習優先級：
  • 必學：device（理解 BPF 模式）、cgroup.freeze（pause 機制）
  • 看場景：跑 DB / DPDK 學 hugetlb；跑 GPU workload 學 misc
  • 選學：perf_event（debug 才用）、rdma（HPC 才用）
Q8.實作思考題：你管理一台機器跑：（A）3 個 PostgreSQL（要 HugeTLB）、（B）DPDK 網路應用（要 HugeTLB + isolated CPU）、（C）每天用 LVM snapshot 做 backup。設計用到本章的所有相關工具的配置。
方案： 
[host setup]
# 1. 配置 HugeTLB pool
echo "vm.nr_hugepages = 4096" >> /etc/sysctl.conf  # 4096 * 2MB = 8GB
sysctl -p

# 2. 啟用 cgroup hugetlb（在 cgroup.subtree_control）
echo "+hugetlb" > /sys/fs/cgroup/cgroup.subtree_control

[A: postgresql-1/2/3.service]
# HugeTLB 各 1.5GB（總 4.5GB）
ExecStartPost=/bin/sh -c 'echo "1.5G" > $cgroup/hugetlb.2MB.max'
MemoryMax=4G          # 一般 RAM 限制（HugeTLB 不算進來）
MemoryHigh=3.5G

[B: dpdk-app.service]
# HugeTLB 3GB
ExecStartPost=/bin/sh -c 'echo "3G" > $cgroup/hugetlb.2MB.max'
# CPU isolated partition（Ch.9）
AllowedCPUs=8-11
ExecStartPost=/bin/sh -c 'echo isolated > $cgroup/cpuset.cpus.partition'

[C: backup-script]
#!/bin/bash
# Step 1: freeze PG（避免 partial commit）
for i in 1 2 3; do
    systemctl freeze postgresql-$i.service
done

# Step 2: LVM snapshot
lvcreate -L 10G -s -n pg_snap /dev/vg/data

# Step 3: thaw PG（snapshot 已建好，db 可繼續）
for i in 1 2 3; do
    systemctl thaw postgresql-$i.service
done

# Step 4: 從 snapshot 慢慢複製（不影響 PG）
mount /dev/vg/pg_snap /mnt/snap
rsync -a /mnt/snap/data /backup/
umount /mnt/snap
lvremove /dev/vg/pg_snap
用到本章的工具：
  1. hugetlb controller：分配 HugeTLB pool 給 PG 與 DPDK，每個 service 獨立 limit。
  2. cpuset isolated partition（Ch.9）：DPDK 獨佔 CPU，無 scheduler 干擾。
  3. cgroup.freeze：backup 時凍結 PG 一秒鐘，做 LVM snapshot 不會 partial commit。
  4. device controller（BPF）：透明使用——systemd 自動編譯 DeviceAllow rules 成 BPF。
監控配套：對 PG 監控 hugetlb.events.max（hit limit 次數）、對 DPDK 監控 cpu.pressure 確認沒被打斷、對 backup 監控 freeze 時間（不應超過 1 秒）。
