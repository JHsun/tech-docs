為什麼需要 cgroup v2

要理解 cgroup v2 的設計哲學，必須先理解 v1 為什麼需要被取代。 本章從 v1 的歷史包袱出發，回到 Tejun Heo 重新設計 cgroup 時所做的關鍵取捨。

01 · Highlights

本章重點

  1. cgroup 是什麼：Linux kernel 提供的資源管理機制，將 process 組織成樹狀階層，並在階層上施加 CPU、memory、I/O 等資源的限制、計量與隔離。
  2. v1 的核心問題：每個 controller 一棵獨立的 hierarchy tree，導致 controller 之間的協調困難、delegation 不安全、thread 與 process 混雜、interface 不一致。
  3. v2 的設計核心：unified hierarchy（單一階層）、controllers 跟著 cgroup 走（不是 cgroup 跟著 controllers 走）、no internal process constraint、明確的 delegation model。
  4. 產業現況：RHEL 9 / Ubuntu 22.04 / Debian 11 起預設 v2；RHEL 10 已移除 v1；Kubernetes 1.31 起 v1 進入維護模式。
  5. 本書脈絡：本章奠定「為什麼」，後續章節將回答「是什麼」與「怎麼用」。
02 · Latest

最新動態（2025–2026）

2025-08 Kubernetes 1.31 正式將 cgroup v1 支援移入維護模式（maintenance mode），未來版本將逐步停止在 cgroup v1 節點上啟動 kubelet。
2025-10 RHEL 10 正式發行並完全移除 cgroup v1，所有資源控制全面走 unified hierarchy。
2025-09 Kubernetes 1.34 將 PSI（Pressure Stall Information）metrics 升至 Beta，這是 v2 才提供的能力，標誌 v1 與 v2 觀測能力差距持續擴大。
2025-08 Go 1.25 在 runtime 內建 cgroup-aware GOMAXPROCS，自動讀取 cpu.max，讓 Go 服務在容器內可以開箱即用避免 CPU throttling。
2026-01 systemd v258 移除 SYSTEMD_CGROUP_ENABLE_LEGACY_FORCE=1 開機參數，使新系統更難回退到 v1。

2026 年的時間點，cgroup v2 已是事實上的標準。所有主流 Linux distro 預設 v2，主流容器引擎（containerd、CRI-O、Docker、Podman）全面支援，Kubernetes 也宣告 v1 進入維護模式。 如果你在 2026 年才開始學 cgroup，跳過 v1 是合理選擇——除非你維運的是仍在 RHEL 7/8 或舊版 Kubernetes 的 legacy system。

03 · Core Concepts

核心內容

3.1 cgroup 是什麼

cgroup（control group，恆小寫，從不大寫）是 Linux kernel 自 2.6.24（2008 年）就有的功能，由 Google 工程師 Paul Menage 與 Rohit Seth 主導開發，最早叫做 process containers。 它的本質很簡單：把 process 組成階層化的群組，然後在群組這個粒度上做資源管理。

管理什麼？三件事——限制（limit）、計量（accounting）、隔離（isolation）。 限制例如「這個群組最多用 4GB 記憶體」；計量例如「這個群組過去 1 小時消耗了多少 CPU」；隔離例如「這個群組看不到、影響不到其他群組」。 我們今天熟悉的所有容器（Docker、Kubernetes Pod、systemd service unit）骨子裡都是 cgroup 在做事。

3.2 為什麼會有 v2：v1 的歷史包袱

cgroup v1 從 2008 年合入 kernel 後一路演進，歷代加入了 cpu、memory、blkio、devices、freezer、cpuset、pids、rdma 等十多個 controllers。 但因為「邊做邊設計」（用 Tejun Heo 自己的話："design followed implementation"），不同 controllers 由不同團隊在不同時期加入，行為不一致、彼此整合困難。 到了 2014 年，繼任維護者 Tejun Heo 評估後做了一個關鍵判斷：v1 的問題已經無法在不破壞 ABI 的前提下修好，必須開新版本。

v2 的開發從 2014 年開始，2016 年 3 月隨 Linux 4.5 正式釋出。 kernel 文件中的 R. Issues with v1 and Rationales for v2 章節就是 Tejun Heo 親筆寫下來的「為什麼要重做」說明書，可以說是理解 v2 設計動機最權威的文件。 以下是其中最重要的五大痛點：

痛點一：Multiple Hierarchies（多階層之亂）

v1 中，每個 controller 都掛在自己獨立的階層樹。也就是說系統上會有 /sys/fs/cgroup/cpu/、/sys/fs/cgroup/memory/、/sys/fs/cgroup/blkio/ 等多個獨立的 hierarchy， 而同一個 process 可以同時被掛在多個 hierarchy 的不同位置。要對某個 service 同時施加 CPU 與 memory 限制，必須在兩棵樹的對應位置都建立目錄並把 process 寫進去。 這在小規模系統上勉強可以管理，但隨著容器與 systemd 的興起，跨 controllers 的協調成為惡夢——例如要讓 memory 配額參考 I/O 的 throttling 狀態，根本沒辦法做。

痛點二：Thread Granularity（執行緒粒度的濫用）

v1 允許把同一個 process 的不同 thread 放進不同 cgroup。這個設計初衷是給「app 想自己做 thread-level 資源管理」用的，但 Tejun 在文件中直白地說： "thread-granular management caused cgroup to be abused as a way for a process to manage itself"——本來該用 syscall 處理的事，被人塞到 cgroup filesystem 來做，造成介面混亂。 更實際的問題是：同一個 process 的所有 thread 共享同一份地址空間，把它們放進不同 memory cgroup 在語意上根本講不通。

痛點三：Internal Process Competition（內部 process 與子 cgroup 競爭）

v1 允許一個 cgroup 同時擁有自己的 process 和子 cgroup。這聽起來很自由，但對 controller 實作極不友善： 當 parent cgroup 自己跑著 process A，又有子 cgroup 裡跑著 process B，到底要怎麼算 parent 的資源使用量？ 是只算 A 還是包含 B 的子樹？不同 controllers 給出了不同答案，行為不一致。 v2 直接砍掉這個自由度——非 root cgroup 一旦啟用任何 domain controller，就不能直接持有 process，process 必須放在 leaf cgroup 上（這就是 no internal process constraint）。

痛點四：Delegation 不安全

v1 的 delegation model 沒有完整定義——把 cgroup 子樹委派給非 root 使用者後，使用者可能可以做出意料之外的事，例如把 process 從委派的子樹移出去、改動不該改的 controller 介面。 v2 為此重新設計了完整的 delegation 規則（這是後面 Ch.5 的主題），包含 nsdelegate mount option 與 /sys/kernel/cgroup/delegate 檔案清單， 確保委派出去的子樹是「真正封閉的沙箱」。

痛點五：Interface 不一致

memory.limit_in_bytes（v1）、cpu.shares（v1）、blkio.weight（v1）……每個 controller 用不同的命名習慣、不同的單位、不同的特殊值（有的用 -1 代表無限，有的用空字串）。 對 systemd、Kubernetes、Docker 這些「cgroup 的使用者」來說，每接一個新 controller 就要重學一次規則。 v2 統一規範介面命名（{controller}.{property}，例如 memory.max、cpu.weight）、特殊值（max 代表無限）、與三大資源分配模型（這是 Ch.3 的主題）。

3.3 v2 的設計核心：Unified Hierarchy

v2 最根本的改變是統一階層。整個系統只有一棵 cgroup tree，掛在 /sys/fs/cgroup/。 每個 cgroup（也就是樹上的每個目錄）可以選擇啟用哪些 controllers，不再有「memory cgroup」與「cpu cgroup」這種說法——cgroup 就是 cgroup，controllers 是它的屬性。

用 Red Hat 部落格的精準說法：

  1. v1：cgroups associated with controllers（cgroup 從屬於 controllers）
  2. v2：controllers associated with cgroups（controllers 從屬於 cgroup）

這個翻轉看似只是文字遊戲，但帶來三個直接好處：

  • 跨 controller 協調變可能：同一個 cgroup 同時持有 cpu、memory、io 三個 controllers 的設定，kernel 可以做出 controller 之間的整合決策（例如 memory writeback 走對應的 io.weight）。
  • 路徑唯一性：一個 process 的 cgroup membership 由 /proc/$PID/cgroup 一行 0::/path 完整描述，不再有多行對應多個 hierarchy 的混亂。
  • delegation 邊界清楚：要把整個子樹委派出去，只需要一棵樹的權限管理，而不是同步多棵樹。

3.5 cgroup 與 namespace 的關係

很多人把 cgroup 與 namespace 搞混，它們是兩個正交的機制：

  • namespace：負責「看得到什麼」——程序看到的 PID、network interface、mount point、UID 是哪些。它做的是視野隔離（visibility isolation）。
  • cgroup：負責「能用多少」——CPU、memory、I/O 的配額與計量。它做的是資源管理（resource control）。

一個 Linux container（不管是 Docker、Podman 還是 Kubernetes Pod）都是 namespace + cgroup 的組合： namespace 把容器與 host 在視野上隔離開，cgroup 把容器在資源上限制住。 少了任何一個，都不算完整的容器。本書聚焦在 cgroup，namespace 部分讀者可參考 man 7 namespaces。

04 · Hands-on

實戰範例：檢查你的系統

在開始學習 v2 之前，先確認你手上的系統用的是哪一版。下面三個指令一起跑，可以完整判斷現況。

Lab 1 · 確認 cgroup 版本

方法一：透過 stat 看 /sys/fs/cgroup 的檔案系統類型。這是最快、最可靠的判斷方式。

$ stat -fc %T /sys/fs/cgroup/

# 如果輸出 cgroup2fs，你已經在 v2
# 如果輸出 tmpfs，你還在 v1（v1 把 /sys/fs/cgroup 掛成 tmpfs，下面再掛各 controller 子目錄）

方法二：看實際的 mount。可以同時看到 v1 與 v2 的掛載狀況。

$ mount | grep cgroup

# v2-only 系統（典型 RHEL 9+ / Ubuntu 22.04+）只會看到一行：
cgroup2 on /sys/fs/cgroup type cgroup2 (rw,nosuid,nodev,noexec,relatime,nsdelegate,memory_recursiveprot)

# v1 系統會看到很多行（每個 controller 一行）

方法三：看 process 的 cgroup membership。v2 的格式永遠是 0::/path，沒有冒號前的 controller 名稱。

$ cat /proc/self/cgroup

# v2 格式（單行）：
0::/user.slice/user-1000.slice/user@1000.service/app.slice/...

# v1 格式（多行，每行對應一個 hierarchy）：
12:cpu,cpuacct:/user.slice
11:memory:/user.slice
10:blkio:/user.slice
...
Lab 2 · 看一眼系統正在執行的 cgroup tree

使用 systemd-cgls（systemd 提供的工具）看 cgroup 階層全貌：

$ systemd-cgls --no-pager | head -30

Control group /:
-.slice
├─user.slice
│ └─user-1000.slice
│   ├─user@1000.service …
│   │ ├─app.slice
│   │ │ └─dbus.service
│   │ └─session.slice
│   │   └─dbus-daemon
│   └─session-2.scope
└─system.slice
  ├─sshd.service
  ├─cron.service
  └─NetworkManager.service

這裡可以清楚看到 systemd 預設把 cgroup 切成 system.slice（系統服務）、user.slice（使用者 session）兩大區塊， 每個 service unit 對應到一個 cgroup directory。這就是後面 Ch.15 會詳細講的 systemd 整合。

Lab 3 · 看一個 cgroup 啟用了哪些 controllers
$ cat /sys/fs/cgroup/cgroup.controllers
cpuset cpu io memory hugetlb pids rdma misc

$ cat /sys/fs/cgroup/cgroup.subtree_control
cpuset cpu io memory pids

cgroup.controllers 列出當前 cgroup 「可以啟用」的所有 controllers（由 kernel build 與 v1/v2 綁定狀況決定）。
cgroup.subtree_control 列出當前 cgroup 「實際對子 cgroup 啟用」的 controllers。
這兩個檔案是 v2 操作的核心，後續章節會大量出現。

05 · Pitfalls

常見陷阱

陷阱 1 · v1 與 v2 不能完全共存

理論上 v1 與 v2 可以「hybrid mode」共存——一些 controllers 走 v1，另一些走 v2。但 Red Hat 文件明確說明：同時掛載 v1 與 v2 是不被支援（unsupported）的配置。 實務上你會遇到 controller 不知道該綁哪邊的混亂、systemd 行為不一致、容器 runtime 報錯。 如果系統還在 v1，要嘛維持 v1，要嘛完整切到 v2，不要嘗試混合。

陷阱 2 · 看到 /sys/fs/cgroup 有 cpu/、memory/ 子目錄不代表 v1

這是初學者最常踩的雷。在 v2-only 系統上，/sys/fs/cgroup/ 下也會有看起來像 controller 名字的子目錄——但那其實是 cgroup 名稱，不是 controller hierarchy。 要判斷版本永遠用 stat -fc %T /sys/fs/cgroup/，看 filesystem type，不要看目錄結構猜。

陷阱 3 · /proc/$PID/cgroup 在 hybrid 系統會誤導

在某些舊系統上 /proc/$PID/cgroup 會同時印出 v1 多行 + v2 一行（0::/path）。 看到多行不代表你在 v1，要看「真正控制這個 process 的是哪一行」。在 systemd 接管的系統上，v2 的那一行（0::/...）才是實際生效的。

陷阱 4 · 直接讀 cgroup 檔案的應用要小心 v1/v2 路徑不同

例如 v1 的 memory.limit_in_bytes 在 v2 是 memory.max；v1 的 cpu.cfs_quota_us + cpu.cfs_period_us 在 v2 合併為 cpu.max。 如果你的應用（特別是 Java、Python、Go 等需要讀 cgroup 設定來自我調整的 runtime）寫死了 v1 路徑，遷移到 v2 後會直接失效。 Ch.19、Ch.20 會專門處理 Go 與 Python 的這類問題。

06 · Exercises

動手練習

下面四題練習由淺入深，建議全做一次。每題預期 5–15 分鐘。

練習 1 · 判斷你的系統版本（5 分鐘）

用三種不同方式判斷你當前 Linux 系統使用的 cgroup 版本，並回答：

  1. 你的系統是 v1、v2 還是 hybrid？
  2. 如果是 v2，/sys/fs/cgroup/ 掛載時帶了哪些 mount options？
  3. 你的 systemd 版本是多少？（systemctl --version）
練習 2 · 找出當前 shell 所在的 cgroup（5 分鐘）
# Step 1：取得當前 shell 的 PID
echo $$

# Step 2：查它在哪個 cgroup
cat /proc/$$/cgroup

# Step 3：到那個 cgroup directory 看裡面有什麼
ls /sys/fs/cgroup/<path-from-step-2>/

觀察並回答：你看到了哪些 cgroup.* 與 {controller}.* 開頭的檔案？哪些是 read-only、哪些是可寫入？

練習 3 · 啟動一個 transient cgroup 並觀察（10 分鐘）

用 systemd-run 在隔離的 cgroup 中跑一個 sleep，然後找到它對應的 cgroup directory：

# 啟動一個 transient unit，限制最多用 50% CPU
sudo systemd-run --slice=user.slice --unit=test-sleep \
    -p CPUQuota=50% sleep 300

# 找到它的 cgroup
systemctl status test-sleep | grep CGroup
# 應該會看到類似：CGroup: /user.slice/test-sleep.service

# 進去看看檔案
sudo ls /sys/fs/cgroup/user.slice/test-sleep.service/
sudo cat /sys/fs/cgroup/user.slice/test-sleep.service/cpu.max

# 清理
sudo systemctl stop test-sleep

回答：cpu.max 的內容是什麼？它的格式是 $MAX $PERIOD，請計算 50% 是怎麼換算成那兩個數字的（提示：預設 period 是 100000μs）。

練習 4 · 列出系統中 cgroup 數量最多的 service（10 分鐘）
# 找出 system.slice 下每個 service 各自有多少子 cgroup
for d in /sys/fs/cgroup/system.slice/*/; do
    count=$(find "$d" -mindepth 1 -type d 2>/dev/null | wc -l)
    echo "$count $d"
done | sort -rn | head -10

回答：哪個 service 建立了最多 cgroup？想想為什麼（提示：跟 worker pool、container runtime 有關）。

07 · Self-check

自我檢查題

Q1.「cgroup」這個詞應該寫成 cgroup、Cgroup 還是 CGROUP？
cgroup。kernel 文件第一段就明確規定：「cgroup is never capitalized」。複數時寫 cgroups，但要注意這是指「多個 control groups」，講整體機制時應使用單數 cgroup。本書全文遵守這個慣例。
Q2.v1 與 v2 最根本的結構差異是什麼？用一句話描述。
v1 是「每個 controller 一棵獨立階層樹」，v2 是「全系統共用單一階層樹（unified hierarchy），controllers 是 cgroup 的屬性」。 這個差異衍生出後面所有設計決策：跨 controller 協調、delegation、no internal process constraint 都是建立在 unified hierarchy 之上才得以實現。
Q3.為什麼 v2 不允許「同時把同一個 process 的不同 thread 放進不同 cgroup」？
兩個原因： （1）語意問題——同一個 process 的 thread 共享 address space，把它們放進不同 memory cgroup 在語意上講不通。 （2）濫用問題——這個機制原本給「app 自我管理」用，但 thread 自我管理本來該用 syscall 處理，cgroup filesystem 介面不適合。 v2 預設只支援 process-level cgroup，但 v2 後來補回了限制版的 thread mode（threaded controllers，這是 Ch.6 的主題）。
Q4.cgroup 與 namespace 的職責分別是什麼？舉一個必須兩者都用的場景。
namespace 管「看得到什麼」（visibility isolation），包含 PID、network、mount、UID 等視野；cgroup 管「能用多少」（resource control），包含 CPU、memory、I/O 等資源配額。 必須兩者都用的典型場景：容器（Docker、Kubernetes Pod）。沒有 namespace 容器看得到 host 的所有 process；沒有 cgroup 容器可以吃光 host 的記憶體。兩者缺一不可。
Q5.判斷 cgroup 版本最可靠的指令是什麼？為什麼不能用「看 /sys/fs/cgroup/ 下有沒有 cpu/ 子目錄」來判斷？
最可靠的指令是 stat -fc %T /sys/fs/cgroup/。輸出 cgroup2fs 是 v2，輸出 tmpfs 是 v1。 不能看子目錄是因為 v2 系統下 /sys/fs/cgroup/ 也會有看起來像 controller 名字的子目錄——但那是 cgroup 名稱（例如 systemd 建立的 slice），不是 controller hierarchy。判斷版本要看 filesystem type，不能看目錄結構猜。
Q6.2026 年的時點，主流 Linux distro 中還有哪些版本仍預設使用 cgroup v1？
絕大多數現代發行版都已經預設 v2：RHEL 9+ / Rocky Linux 9+ / AlmaLinux 9+、Ubuntu 22.04+、Debian 11+、Fedora 31+、CentOS Stream 9+、Bottlerocket 1.13+ 全都預設 v2。 仍預設 v1 的是已經 EOL 或進入維護的舊版：RHEL 7（已 EOL）、RHEL 8（仍維護中但可手動切 v2）、Rocky Linux 8（同 RHEL 8）、Ubuntu 20.04（EOL 2025-04）。 如果 2026 年新建系統，幾乎不會碰到 v1 預設環境。
Q7.Tejun Heo 為什麼選擇開新版而不是漸進式修改 v1？
因為 v1 的核心問題（多階層、thread 粒度、internal process competition、interface 不一致）都涉及使用者可見的 ABI， 修這些問題會破壞既有應用的相容性。kernel 的 ABI 穩定性是不可妥協的紅線（Linus Torvalds 著名的 "we don't break userspace" 原則）， 所以唯一辦法是開一個全新的、平行的子系統，讓使用者自願遷移。這就是 cgroup v2 的誕生。
Q8.如果你的應用直接讀取 /sys/fs/cgroup/memory/memory.limit_in_bytes 來決定自己的記憶體上限，遷移到 v2 後會發生什麼？該怎麼修？
會直接失效——這個檔案在 v2 不存在。v2 對應的檔案是 /sys/fs/cgroup/memory.max（注意路徑結構也變了，沒有 memory/ 子目錄），且特殊值從 v1 的「一個極大的數字」改為字串 max。 修法是：先用 stat -fc %T /sys/fs/cgroup/ 判斷版本，依版本走不同程式碼路徑；或讀 /proc/self/cgroup 找到自己的 cgroup path，再讀對應的 memory.max。 這也是 Java（JDK 11.0.16+）、Go（1.25+）、Uber automaxprocs 等做的事。
