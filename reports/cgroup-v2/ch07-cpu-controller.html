CPU controller 深度解析

CPU controller 是 cgroup v2 最常用的 controller，也是 Kubernetes、容器、systemd 的資源管理核心。 本章深入每個介面檔案——cpu.weight 的 CFS share 對應、cpu.max 的 throttle 行為、 cpu.weight.nice 的轉換公式、cpu.idle 的 SCHED_IDLE 整合、 cpu.uclamp 對 schedutil 的影響、cpu.pressure 與 cpu.stat 的監控用途。

01 · Highlights

本章重點

  1. 三類介面：分配（cpu.weight、cpu.weight.nice、cpu.idle）、限制（cpu.max、cpu.max.burst、cpu.uclamp.{min,max}）、觀測（cpu.stat、cpu.pressure）。
  2. cpu.weight 與 nice 的對應：weight 1–10000 線性對應到 CFS load weight。cpu.weight.nice（kernel 5.16+）讓你用熟悉的 nice 值（-20 到 +19）設定，內部公式 weight = 1024 / (1.25 ^ nice)。
  3. cpu.max 格式：$MAX $PERIOD，預設 max 100000。每個 PERIOD（預設 100ms）內 cgroup 最多用 MAX 微秒 CPU 時間。cpu.max.burst 允許短期累積額度突破上限。
  4. cpu.idle：寫 1 把整個 cgroup 的 process 改用 SCHED_IDLE 排程——只在系統真的閒置時才跑。對 batch / 背景 workload 是黃金設定。
  5. cpu.uclamp：影響 schedutil governor 的 CPU frequency 決策，主要用於行動裝置與 EAS（Energy Aware Scheduling）系統，server 場景使用較少。

2025 cpu.idle 在資料中心普及。Meta、Google 公開分享用 cpu.idle 跑 background analytics，與 latency-critical 工作負載完美 collocation，顯著提升 CPU 利用率。

2026 年的時點，CPU controller 已是非常成熟穩定的 API。介面層基本沒變，但底層 scheduler 從 CFS 演進到 EEVDF（kernel 6.6+），加上 cpu.idle、cpu.uclamp 的廣泛採用， 讓 CPU 資源管理達到前所未有的精細度。本章把這些介面講透，並用實戰示範告訴你什麼場景配什麼設定。

03 · Core Concepts

核心內容

3.1 CPU controller 介面總覽

現代 kernel（6.6+）的 CPU controller 在 v2 提供以下介面檔案：

檔案 類型 作用
cpu.weight RW · 數值 CPU 比例分配，1–10000，預設 100
cpu.weight.nice RW · 數值 用 nice 值（-20 到 +19）設定 weight，預設 0
cpu.max RW · 雙值 CPU bandwidth 上限，格式 $MAX $PERIOD，預設 max 100000
cpu.max.burst RW · 數值 burst 容量（μs），預設 0（無 burst）
cpu.idle RW · 開關 1 = SCHED_IDLE，整個 cgroup 只在系統閒置時跑
cpu.uclamp.min RW · 百分比 schedutil 視角的最小 CPU 利用率（boost），預設 0
cpu.uclamp.max RW · 百分比 schedutil 視角的最大 CPU 利用率（cap），預設 max
cpu.stat R · 統計 累計 CPU 時間、throttle 次數、burst 統計
cpu.pressure RW · PSI CPU pressure stall information（some/full avg10/60/300/total）

3.2 cpu.weight：比例分配的核心

cpu.weight 是 v1 cpu.shares 的 v2 替代品。範圍 1–10000、預設 100，這是 v2 規範對所有 weight 介面的統一要求（詳見 Ch.4）。

kernel 內部，cpu.weight 線性對應到 CFS load weight（取代 EEVDF 後語意相同）。具體公式（簡化版）： 

// CFS share = cpu.weight * 1024 / 100
// 等價於 nice 值換算（見 cpu.weight.nice 一節）

// cpu.weight=100  → CFS share = 1024  (nice 0)
// cpu.weight=200  → CFS share = 2048  (約 nice -3)
// cpu.weight=400  → CFS share = 4096  (約 nice -7)
// cpu.weight=10000 → CFS share = 102400  (極端優先)

實務上記住 「weight 是相對的、競爭時才生效、可超額」 三條原則就夠了——細節 scheduler 會處理。

典型配置場景

  • 系統服務 vs 使用者 workload：system.slice/cpu.weight=100（預設）+ user.slice/cpu.weight=100，平等競爭。
  • 關鍵服務優先：critical.service/cpu.weight=400，4 倍於其他預設 service。
  • 背景作業降權：backup.service/cpu.weight=10，幾乎只用閒置 CPU（但更好的選擇是 cpu.idle=1）。

3.3 cpu.weight.nice：nice 值風格的 weight

cpu.weight.nice 是 kernel 5.16+ 新增的替代介面，讓你用熟悉的 nice 值（-20 到 +19，預設 0）設定 weight。 kernel 內部用 PeterZ 提議的固定公式換算： 

// 數學公式（從 sched/core.c sched_prio_to_weight[] 表）
// weight = round(1024 / (1.25 ^ nice))

// nice -20 → weight 88761  →  cpu.weight 約 8665
// nice -10 → weight 9548   →  cpu.weight 932
// nice 0   → weight 1024   →  cpu.weight 100   (預設)
// nice +10 → weight 110    →  cpu.weight 10
// nice +19 → weight 15     →  cpu.weight 1

cpu.weight 與 cpu.weight.nice 是同一份內部狀態的兩種視角——讀寫任一個，另一個會跟著變。

選擇建議：

  • 從 nice/renice 思維轉過來 → 用 cpu.weight.nice
  • 從零學 cgroup → 用 cpu.weight（範圍直觀，1–10000 線性）
  • systemd 兩者都支援：CPUWeight= 對應 cpu.weight，Nice= 對應 cpu.weight.nice

3.4 cpu.max：硬性 bandwidth 上限

cpu.max 是 v1 cpu.cfs_quota_us + cpu.cfs_period_us 的 v2 替代品，合併成單一檔案。格式： 

$MAX $PERIOD

$MAX：每個 PERIOD 內這個 cgroup 最多能用多少微秒 CPU 時間。寫 max 表示無上限。
$PERIOD：以微秒為單位的時間窗，預設 100000（100ms）。

計算公式：cgroup 可用的 CPU 比例 = MAX / PERIOD。例如： 

# 限制最多用 1 顆 CPU（100% 一核）
echo "100000 100000" > cpu.max

# 限制最多用 0.5 顆 CPU（50% 一核）
echo "50000 100000" > cpu.max

# 限制最多用 2.5 顆 CPU
echo "250000 100000" > cpu.max

# 解除限制
echo "max 100000" > cpu.max

幾個關鍵特性：

  • 可超過一核：MAX > PERIOD 表示允許跑滿多核（例如 200000 100000 = 2 cores worth）。
  • 可 over-commit：所有 child cgroup 的 cpu.max 加總可超過 parent 的——只要實際同時跑的不超過。
  • throttle 機制：cgroup 用完 quota 後會被 throttle 到 PERIOD 結束才能繼續。throttle 次數記在 cpu.stat。
  • 與 cpu.weight 互動：cpu.max 是天花板，cpu.weight 在天花板下決定比例。詳見 Ch.3 §3.7。

cpu.max.burst：應對短期尖峰

cpu.max 的硬性 quota 對「平均流量低、偶爾尖峰」的 workload 不友善——尖峰時被 throttle，但其他時間明明有閒置 CPU。 kernel 5.14+ 引入 cpu.max.burst 解決這問題： 

# cpu.max=50000 100000 (0.5 cores)
# cpu.max.burst=20000  (允許短期累積 20ms 額度)

# 行為：在不被 throttle 的 PERIOD 中累積 burst 額度（最多 20ms），
# 尖峰時可以「借用」這些額度，最多突破到 50000+20000=70000us。
echo 50000 100000 > cpu.max
echo 20000 > cpu.max.burst

典型場景：批次作業有間歇式 CPU 需求、Web 服務應對突發請求。burst 不會讓平均超過 cpu.max——只是允許更平滑的瞬時表現。

3.5 cpu.idle：SCHED_IDLE 整合

cpu.idle（kernel 5.15+）是 v2 的一個大殺器——寫 1 進去，整個 cgroup 的 process 都改用 SCHED_IDLE 排程策略。 SCHED_IDLE 是 Linux scheduler 的最低優先級——只在所有 SCHED_NORMAL（一般）process 都在 sleep 時才跑。 

# 把 cgroup 設為 idle（所有 process 變 SCHED_IDLE）
echo 1 > /sys/fs/cgroup/batch.slice/cpu.idle

# 反向：恢復為 SCHED_NORMAL
echo 0 > /sys/fs/cgroup/batch.slice/cpu.idle

cpu.idle 的優勢（相對於只設 cpu.weight=1）：

  • 絕對讓位：weight=1 仍會搶到一點 CPU；SCHED_IDLE 真的只在系統空閒時跑。
  • 對 latency 友好：不影響其他 cgroup 的 latency-critical workload，連 cache thrashing 都更少。
  • 適合 collocation：把 batch / analytics 跟 latency-critical workload 放在同一 host，免費利用空閒 CPU。

實際 production 案例：Meta、Google 在 2024-2025 公開分享，把背景 ML training、log processing 標為 cpu.idle， 與 user-facing service co-locate，整體 CPU 利用率提升 30–50%。

3.6 cpu.uclamp：utilization clamping

cpu.uclamp.min 與 cpu.uclamp.max（kernel 5.3+）控制 schedutil CPU frequency governor 對這個 cgroup 的決策。 值是百分比（如 50.00），對應 0–100% 的 CPU 利用率。 

# 確保 cgroup 的 process 跑時 CPU 至少跳到 50% utilization
# 等同於提示 schedutil 把 frequency 拉高
echo 50.00 > cpu.uclamp.min

# 反向：限制 CPU 不超過 50%（節能）
echo 50.00 > cpu.uclamp.max

主要應用場景：

  • 行動裝置（Android）：UI thread 設 uclamp.min=80 強制高頻、背景 service 設 uclamp.max=20 強制低頻省電。
  • EAS（Energy Aware Scheduling）系統：big.LITTLE 架構決定 task 該放 little 還是 big core。
  • Server 用法少見：資料中心 CPU 通常已是固定 P-state 或統一頻率管理，uclamp 影響不大。

3.7 cpu.stat：累計統計

cpu.stat 是監控與容量規劃的關鍵介面。讀出範例： 

$ cat /sys/fs/cgroup/myapp/cpu.stat
usage_usec 1234567890
user_usec 987654321
system_usec 246913569
nr_periods 12345
nr_throttled 234
throttled_usec 12345678
nr_bursts 56
burst_usec 1234567

欄位解讀：

欄位說明
usage_usec累計 CPU 時間（user + system），微秒
user_usec累計 user-mode CPU 時間
system_usec累計 kernel-mode CPU 時間
nr_periods(僅 cpu.max 啟用時) PERIOD 經過的總次數
nr_throttled(僅 cpu.max 啟用時) 被 throttle 的 PERIOD 次數
throttled_usec(僅 cpu.max 啟用時) 被 throttle 的累計時間
nr_bursts(burst 啟用時) 使用 burst 額度的次數
burst_usec(burst 啟用時) 累計 burst 使用時間

關鍵監控指標：nr_throttled / nr_periods 比例就是「throttled rate」。 如果這個值 > 1%，cpu.max 設得太緊；> 5% 通常 latency 已經有感影響。 實務上把 cpu.stat 推到 Prometheus / Datadog 做 graph，能看出 service 的真實 CPU 狀況。

3.8 cpu.pressure：PSI 監控

cpu.pressure 是 PSI（Pressure Stall Information）對 CPU 的監控介面。讀出範例： 

$ cat /sys/fs/cgroup/myapp/cpu.pressure
some avg10=2.04 avg60=0.75 avg300=0.40 total=157656722
full avg10=0.00 avg60=0.00 avg300=0.00 total=0

欄位解讀：

  • some：「至少有一個 task 在等 CPU」的時間百分比。
  • full：「所有 runnable task 都在等 CPU」的時間百分比（system-wide CPU 飽和）。
  • avg10/60/300：過去 10 秒、1 分鐘、5 分鐘的滑動平均（百分比）。
  • total：累計 stall 時間（微秒），用來偵測短暫的 stall 尖峰。

cpu.pressure 是 Ch.12 PSI 章節的主題，這裡只簡介。實務上 some avg10 > 5% 是 CPU 開始緊繃的訊號。 PSI 比 CPU utilization 更能反映「使用者實際感受到的延遲」——CPU 100% 不一定 stall 高（可能就是有事做），但 stall 高一定是 latency 有問題。

3.9 systemd 介面對應

production 上幾乎都透過 systemd 設定 CPU controller，而非直接寫 cgroup 介面。對應關係：

systemd directivecgroup 介面備註
CPUWeight=cpu.weight1–10000，預設 100
StartupCPUWeight=cpu.weight啟動階段的 weight，啟動完成後切回 CPUWeight
CPUQuota=cpu.max百分比形式，50% = 50000 100000
CPUQuotaPeriodSec=cpu.max 的 PERIOD預設 100ms
(無)cpu.max.burstsystemd 244+ 部分版本支援
(無，間接)cpu.idle透過 systemd OOM policy / Slice 設計，未來版本可能加直接 directive
Nice=process nice (非 cgroup)注意：這設的是 process nice，不是 cpu.weight.nice

注意：systemd 的 Nice= 設定的是個別 process 的 nice 值（影響 scheduler 內部 weight），不是 cgroup 的 cpu.weight.nice。 要設 cgroup-level 的 weight 一律用 CPUWeight=。

04 · Hands-on

實戰範例

Lab 1 · cpu.weight 比例驗證
# 跑兩個 100% CPU 的 worker，分別給 weight=100 和 weight=400
sudo systemd-run --slice=user.slice --unit=w100 \
    -p CPUWeight=100 \
    bash -c 'while :; do :; done'

sudo systemd-run --slice=user.slice --unit=w400 \
    -p CPUWeight=400 \
    bash -c 'while :; do :; done'

# 5 秒後觀察 CPU 比例
sleep 5
sudo systemd-cgtop --order=cpu --iterations=1 user.slice

# 預期：w400 約占 80% CPU、w100 約占 20%（比例 1:4）
# 注意：實際測試需要把 host 限制到單核才看得出比例，
# 否則兩者都各跑滿一核，看不出 weight 差別

# 清理
sudo systemctl stop w100 w400

觀察點：weight 在有競爭時 才生效。如果 host 有多核且兩個 worker 都跑得下，比例可能看不出。

Lab 2 · cpu.max 與 throttle 觀察
# 限定 service 只能用 50% 一核
sudo systemd-run --slice=user.slice --unit=throttled \
    -p CPUQuota=50% \
    bash -c 'while :; do :; done' &

sleep 3

# 讀 cpu.stat 看 throttle 統計
CG=$(systemctl show -p ControlGroup --value throttled.service)
cat /sys/fs/cgroup$CG/cpu.stat

# 預期看到（部分欄位）：
# nr_periods 30
# nr_throttled 25         ← 多數 PERIOD 都被 throttle
# throttled_usec 1500000  ← 累計 throttle 時間

# 計算 throttled rate
NR_THROTTLED=$(grep nr_throttled /sys/fs/cgroup$CG/cpu.stat | awk '{print $2}')
NR_PERIODS=$(grep nr_periods /sys/fs/cgroup$CG/cpu.stat | awk '{print $2}')
echo "Throttled rate: $((NR_THROTTLED * 100 / NR_PERIODS))%"

# 清理
sudo systemctl stop throttled
Lab 3 · cpu.idle vs cpu.weight=1 行為比較
# 場景：一個前景 worker（CPU 滿載）+ 一個背景 worker

# 方案 A：背景 worker 用 cpu.weight=1
sudo systemd-run --slice=user.slice --unit=fg-a \
    -p CPUWeight=10000 \
    bash -c 'while :; do :; done'

sudo systemd-run --slice=user.slice --unit=bg-weight \
    -p CPUWeight=1 \
    bash -c 'while :; do :; done'

sleep 5
sudo systemd-cgtop --order=cpu --iterations=1 user.slice
# 觀察：bg-weight 仍會搶到一點 CPU（因為 weight 只是極低，不是 0）

sudo systemctl stop fg-a bg-weight

# 方案 B：背景 worker 用 cpu.idle
sudo systemd-run --slice=user.slice --unit=fg-b \
    bash -c 'while :; do :; done'

sudo systemd-run --slice=user.slice --unit=bg-idle \
    bash -c 'while :; do :; done'

# 把 bg-idle 設為 SCHED_IDLE
CG=$(systemctl show -p ControlGroup --value bg-idle.service)
echo 1 | sudo tee /sys/fs/cgroup$CG/cpu.idle

sleep 5
sudo systemd-cgtop --order=cpu --iterations=1 user.slice
# 觀察：bg-idle 應該 ≈ 0% CPU（因為 fg-b 一直在跑）
# 真正的 SCHED_IDLE 行為：完全讓位

# 清理
sudo systemctl stop fg-b bg-idle

觀察點：cpu.idle 是「絕對讓位」，cpu.weight=1 是「盡量讓位但仍有最小份額」。生產環境用 cpu.idle 更乾淨。

Lab 4 · cpu.weight.nice 與 cpu.weight 互通驗證
# 建立測試 cgroup
sudo mkdir /sys/fs/cgroup/lab7
echo "+cpu" | sudo tee /sys/fs/cgroup/cgroup.subtree_control 2>/dev/null
sudo mkdir /sys/fs/cgroup/lab7/test

# 寫 cpu.weight，看 cpu.weight.nice 怎麼變

cat /sys/fs/cgroup/lab7/test/cpu.weight.nice
# 預期：0

echo 400 | sudo tee /sys/fs/cgroup/lab7/test/cpu.weight
cat /sys/fs/cgroup/lab7/test/cpu.weight.nice
# 預期：約 -6 或 -7（400/100 = 4 = 1.25^約6.2）

echo 25 | sudo tee /sys/fs/cgroup/lab7/test/cpu.weight
cat /sys/fs/cgroup/lab7/test/cpu.weight.nice
# 預期：約 +6 或 +7

# 反向：寫 nice，看 weight 怎麼變

# 預期：約 932（公式 1024 / 1.25^-10 = 1024 * 9.31 ≈ 9531，對應 cpu.weight 932）

# 清理
sudo rmdir /sys/fs/cgroup/lab7/test
sudo rmdir /sys/fs/cgroup/lab7

# 啟動超量 CPU worker（4 個 100% CPU，但 host 只有 4 核）

05 · Pitfalls

常見陷阱

陷阱 1 · cpu.max 比想像中的更早 throttle

設 cpu.max=100000 100000（1 顆 CPU）但發現 process 只用了 60% 就被 throttle？ 原因：cpu.max 是「同一 PERIOD 內的累計」，不是即時上限。 如果一個 burst 在 PERIOD 開頭把 100ms 用光，接下來 PERIOD 結束前都被 throttle，平均看起來只用 60%。 修法：用 cpu.max.burst 平滑、或 調短 PERIOD（例如 50ms）讓 throttle 更頻繁但每次更短，平均行為更平順。

陷阱 2 · cpu.weight 在沒競爭時完全不影響行為

典型誤解：「我設 cpu.weight=1，這個 service 只會用很少 CPU」。 錯。cpu.weight 是比例，沒競爭時 weight=1 的 service 也能跑滿 CPU。 要硬上限請用 cpu.max，要絕對讓位請用 cpu.idle=1。

陷阱 3 · realtime process 在非 root cgroup 會讓 cpu controller 啟用失敗

kernel 文件警告：「The CPU controller can only be enabled when all realtime processes are in the root cgroup.」 如果 systemd 啟動時把某些 RT process（例如 chronyd 設了 CPUSchedulingPolicy=fifo）放到 service cgroup， 嘗試在這個 cgroup 啟用 cpu controller 會失敗。 修法：把 RT process 移回 root cgroup，或避免在含 RT process 的子樹啟用 cpu controller。

陷阱 4 · CPUQuota=200% 在 systemd 上的語意

systemd 的 CPUQuota= 接受百分比，但允許超過 100%——意思是允許用超過一核。 例如 CPUQuota=250% 對應 cpu.max=250000 100000，允許用 2.5 顆 CPU。 踩雷點：很多人以為 CPUQuota=200% 是「使用率上限 200%」（沒意義），實際是「最多用 2 顆 CPU」。

陷阱 5 · cpu.idle 不能用於有 RT process 的 cgroup

cpu.idle 把 process 改用 SCHED_IDLE，但 SCHED_FIFO / SCHED_RR 的 RT process無法降級到 SCHED_IDLE。 如果 cgroup 內有 mixed RT 與普通 process，寫 cpu.idle=1 會失敗或行為不如預期。 修法：確保 cgroup 內全是普通 SCHED_NORMAL process。

陷阱 6 · cpu.weight 的 nested cgroup 計算非直觀

典型問題：A/cpu.weight=100, B/cpu.weight=100（兄弟），它們各分 50%。 但 A 內又有兩個 child A/x, A/y，A/x/cpu.weight=100, A/y/cpu.weight=100——它們在 A 的 50% 內各分 50%，所以實際只各佔 host 的 25%。 每多一層 cgroup，比例會被「稀釋」一次。實作大型 hierarchy 時要逐層算下來。

陷阱 7 · Java 在 cgroup v2 取 CPU 數量的方式

OpenJDK 8u372+、11.0.16+、17+ 已支援 v2，Runtime.getRuntime().availableProcessors() 會回 cgroup limit 對應的核數，不是 host 的。 例如 cpu.max=200000 100000（2 cores）的容器，Java 會看到 2 cores（向上取整）。 但舊版 Java 只認 v1，在 v2 host 上會看到 host 的全部核數，導致 thread pool 過大、context switch 暴增。 升級 Java 是首要解決方法。

練習 2 · 用 cpu.stat 計算實時 throttled rate（15 分鐘）

寫一個 monitoring script，每 10 秒讀一次 cpu.stat，計算最近 10 秒的 throttled rate：

#!/bin/bash
# cpu-throttle-monitor.sh

CG="$1"
[ -z "$CG" ] && { echo "Usage: $0 /sys/fs/cgroup/path"; exit 1; }

prev_periods=0
prev_throttled=0

while true; do
    cur_periods=$(grep nr_periods $CG/cpu.stat | awk '{print $2}')
    cur_throttled=$(grep nr_throttled $CG/cpu.stat | awk '{print $2}')
    
    delta_periods=$((cur_periods - prev_periods))
    delta_throttled=$((cur_throttled - prev_throttled))
    
    if [ $delta_periods -gt 0 ]; then
        rate=$((delta_throttled * 100 / delta_periods))
        echo "$(date +%T) throttled rate (last 10s): ${rate}%"
    fi
    
    prev_periods=$cur_periods
    prev_throttled=$cur_throttled
    sleep 10
done

跑這個 script 監控某個 service，觀察什麼負載下 throttle rate 會暴增。

07 · Self-check

自我檢查題

Q1.cpu.weight、cpu.weight.nice、cpu.max、cpu.idle 各自解決什麼問題？什麼場景該選哪一個？
  • cpu.weight：比例分配。用於正常 workload 之間的相對優先級。範圍 1–10000，預設 100。
  • cpu.weight.nice：nice 值風格的 weight。內部與 cpu.weight 同一份狀態，給習慣 nice/renice 的人用。範圍 -20 到 +19。
  • cpu.max：絕對 bandwidth 上限。需要硬性限制（多租戶、預算控制）時用。格式 $MAX $PERIOD。
  • cpu.idle：SCHED_IDLE。整個 cgroup 只在系統閒置時跑。給 batch / 背景 workload 用，與 latency-critical workload 完美 collocation。
選擇邏輯：要絕對防爆 → cpu.max；要正常優先級 → cpu.weight；要絕對讓位 → cpu.idle。三者可組合（cpu.max 當天花板 + cpu.weight 處理日常）。
Q2.cpu.max 格式是 $MAX $PERIOD，怎麼設「最多用 2.5 顆 CPU」？怎麼設「無上限」？
2.5 顆 CPU：echo "250000 100000" > cpu.max。意思是每 100ms 內可用 250ms CPU 時間（橫跨多核累計），等於 2.5 cores。 無上限：echo "max 100000" > cpu.max。不是 -1，不是 0，是字串 max——v2 規範統一用法。 等價的 systemd：CPUQuota=250%。systemd 接受百分比形式並自動換算。 $PERIOD 怎麼選：預設 100ms 對多數場景合適。短 PERIOD（如 50ms）讓 throttle 更頻繁但每次更短，適合 latency-sensitive 場景；長 PERIOD（500ms）允許更大的瞬時 burst。
Q3.cpu.idle=1 與 cpu.weight=1 的行為差別？production 應該用哪個跑 batch jobs？
cpu.weight=1：把 CPU 分配 weight 設成最低，但仍是 SCHED_NORMAL。在系統有空時可以跑滿 CPU；在系統繁忙時拿到的 CPU 約 1/(1+其他人 weight 總和)，仍會搶走極少量 CPU。 cpu.idle=1：整個 cgroup 改用 SCHED_IDLE，scheduler 視角下這個 cgroup 內的 process絕對讓位於 SCHED_NORMAL——只有當所有 SCHED_NORMAL task 都 sleep 時才被 schedule。 production 建議：跑 batch jobs 用 cpu.idle=1。原因：
  1. 真的不影響 latency-critical workload（cache thrashing 都更少）
  2. 能與 latency-critical workload 完美 collocation，提升整體 CPU 利用率 30–50%（Meta、Google 案例）
  3. 不用花心思調 weight 大小
Q4.cpu.max.burst 解決什麼問題？舉一個適合用它的場景。
解決問題：cpu.max 是「每個 PERIOD 內的累計」，對「平均流量低、偶爾尖峰」的 workload 不友善——尖峰時被 throttle，但其他時間明明有閒置 CPU。 cpu.max.burst 機制：在不被 throttle 的 PERIOD 中累積 burst 額度（最多 burst 值），尖峰時可以「借用」這些額度突破 cpu.max。平均不會超過 cpu.max，只是允許更平滑的瞬時表現。 適用場景：
  • Web 服務應對突發請求：一般時段 CPU 用量低，但 traffic spike 時短暫需要更多 CPU。
  • 批次作業有間歇式 CPU 需求：平均使用低，但每次處理時需要快速完成。
  • JVM startup：JIT compile 階段需要短暫的高 CPU，啟動完成後降下來。
典型配置：cpu.max="50000 100000"（0.5 cores 平均）+ cpu.max.burst="50000"（允許短期突破到 1 core）。
Q5.cpu.stat 的 nr_throttled / nr_periods 比例代表什麼？多少算太高？
意義：在 cpu.max 啟用的情況下，這個 cgroup 在多少比例的 PERIOD 內被 throttle。比例越高表示越常撞 cpu.max 上限。 判讀標準（經驗值，會因 workload 而異）：
  • < 1%：正常，cpu.max 設定合理。
  • 1% – 5%：偶爾撞牆，latency 開始有感受到尖峰。考慮用 cpu.max.burst 平滑或調整 cpu.max。
  • > 5%：cpu.max 設得太緊。Latency 已明顯惡化，p99 會不穩定。
  • > 20%：嚴重不足。或是 workload 變大，或是設定錯了。
監控做法：把 cpu.stat 推到 Prometheus，設 alert 在 throttled rate > 5%。Kubernetes 中 cAdvisor 已自動 export container_cpu_cfs_throttled_periods_total / container_cpu_cfs_periods_total。
Q6.cpu.pressure 的 some 與 full 有什麼差別？哪個對 latency 監控更有用？
some：「至少有一個 task 在等 CPU」的時間百分比。表示這個 cgroup 中有 task 受到 CPU 不足影響——這個 task 的 latency 會增加。 full：「所有 runnable task 都在等 CPU」的時間百分比。表示整個 cgroup 完全卡在 CPU 等待，沒有任何 task 在跑——這時 cgroup 整體吞吐量降為零。 對 latency 監控：
  • some 更有用：反映「使用者實際感受到的延遲」。任何一個 task 在等 CPU，使用者就感受到了。
  • full 更嚴重：full > 0 表示嚴重 CPU 飽和，整個 cgroup 卡住。應該觸發 alarm。
實務建議：日常監控用 some avg10，閾值 5–10%；嚴重 alarm 用 full avg10，閾值 1%。
Q7.systemd 的 Nice= 與 CPUWeight= 有什麼差別？
兩者不是同一件事，雖然名字相似：
  • Nice=：設定 service 啟動時的個別 process nice 值，影響 scheduler 對個別 task 的優先級。對應 syscall setpriority(PRIO_PROCESS)，作用在 task struct 上。
  • CPUWeight=：設定 service cgroup 的 cpu.weight，影響整個 cgroup 在競爭時的比例。作用在 cgroup struct 上。
關鍵差異：
  1. Nice= 影響的是「單個 process 與其他 process 的相對優先」（不論在不在同一 cgroup）。
  2. CPUWeight= 影響的是「這個 cgroup 整體與兄弟 cgroup 的相對比例」。
  3. 在 cgroup 內部，所有 process 的相對優先仍由 nice 決定。
建議：production 用 CPUWeight=（cgroup 層級），Nice= 留給特殊場景。要設 cgroup-level 的 nice 風格用 cpu.weight.nice 介面（systemd 沒有直接 directive，需要 ExecStartPost 寫入）。
Q8.實作思考題：你管理一個 4 核機器，要跑：（A）latency-critical Web service、（B）一般 batch job、（C）夜間 ML training。如何用 CPU controller 配置三者，最大化 CPU 利用率同時保護 A 的 latency？
方案： 
[A: web.service]
CPUWeight=400      # 高權重，競爭時拿大份額
CPUQuota=300%      # 最多 3 cores（保留 1 core 給 OS+其他）
# 不設 cpu.idle（A 是 latency-critical）

[B: batch.service]
CPUWeight=100      # 預設，普通優先級
CPUQuota=200%      # 最多 2 cores（防爆）

[C: ml-training.service]
# 直接把整個 cgroup 設 cpu.idle=1
ExecStartPost=/bin/bash -c 'echo 1 > /sys/fs/cgroup/system.slice/ml-training.service/cpu.idle'
# 不設 CPUQuota（讓它在 idle CPU 上跑滿）
設計理由：
  1. A 用 weight + quota：weight=400 確保競爭時優先；quota=300% 防爆，避免 A 自己出 bug 吃滿 host。
  2. B 用標準配置：與其他普通 service 同等競爭，quota 防止失控。
  3. C 用 cpu.idle：完全讓位給 A 與 B，免費 利用閒置 CPU。夜間 A、B 流量低時 C 自然跑滿；白天 A 流量高時 C 自動讓位，A 的 latency 不受影響。
監控配套：對 A 監控 cpu.pressure some avg10 < 5% 與 cpu.stat nr_throttled / nr_periods < 1%。任一超標就要重新檢視配置。
