效能調校與診斷實戰

前 13 章把 cgroup v2 各個 controller 與觀測介面講透，但知道介面 與 會調出好性能 之間還有距離。 本章是 Part V 的收尾——也是全書的「實戰篇」。 涵蓋 Brendan Gregg 的 USE / RED 方法論在 cgroup 場景的應用、PSI-driven tuning 流程、 benchmark 三劍客（fio / stress-ng / wrk），最後給出 4 個生產 troubleshooting playbook： OOM 調查、CPU throttling 排查、IO 瓶頸定位、PSI 告警處理。

01 · Highlights

本章重點

  1. USE method 是調校的起點：對每個資源（CPU/Memory/IO/Network）檢查 Utilization、Saturation、Errors。cgroup v2 把 USE 從「per-host」變成「per-cgroup」——這是巨大進步。
  2. PSI-driven tuning 流程：以 PSI 為主指標，迭代式 tuning。設 limit → 跑 workload → 看 PSI → 調整。比拍腦袋設數字可靠得多。
  3. Benchmark 三劍客：fio（IO 性能與 IO controller 驗證）、stress-ng（CPU/memory/syscall 壓力測試）、wrk（HTTP 端到端 latency）。三個工具配合 cgroup 限制是調校驗證的標配。
  4. 容量規劃公式：peak × 1.5（hard limit）、p95 × 1.2（soft limit）、p50（min/request）。基於歷史資料而非猜測。
  5. 4 個 production playbook：OOM 調查（從 events → stat → dmesg）、CPU throttling 排查（cpu.stat → quota 設定）、IO 瓶頸（io.pressure → fio 重現）、PSI 告警處理（trigger → cgroup 定位 → 緩解）。

最新動態（2025–2026）

2025 學術界發布 cgroup v2 IO isolation 完整研究（Atlarge IISWC 2025）。系統性評估 io.weight、io.max、io.cost、io.latency 五種 control knob 在 NVMe 上的 isolation 效果，提供生產級 tuning 數據。

2025–2026 stress-ng 持續更新對 cgroup 場景的驗證。新增 cgroup-specific stressors 直接驗證 cgroup behavior，是 v2 限制 testing 的標準工具。
2026 K8s 1.36 In-Place Pod Resize 持續演進。可以動態調整 limit 不需重啟 Pod——配合 PSI-driven tuning 形成完整 closed-loop 自動調校。
2025 benchmark 普及包含 PSI baseline。Phoronix Test Suite、OpenBenchmarking.org 等 benchmark 結果開始包含 PSI 數據——「跑分時 PSI 多少」成為新指標。

2026 年的時點，cgroup v2 性能調校已從「玄學」走向「科學」——有完整方法論、權威 benchmark、學術研究背書。 本章把這些整合成「從不知道到精通」的實戰路徑。

03 · Core Concepts

核心內容

3.1 USE method：cgroup 版本

Brendan Gregg 的 USE method（Utilization、Saturation、Errors）是性能分析的經典框架。 傳統 USE 是 per-host：看整機 CPU、memory、IO、network 的這三個指標。 cgroup v2 讓 USE 變成 per-cgroup，分析顆粒度更精細：

資源Utilization（用了多少）Saturation（多塞）Errors（錯誤）
CPU cpu.stat usage_usec rate cpu.pressure some / cpu.stat throttled_usec cpu.stat nr_throttled、sched_switch failure
Memory memory.current / memory.peak memory.pressure some/full memory.events oom_kill
IO io.stat rbytes/wbytes rate io.pressure some/full EIO syscall（用 bpftrace 抓）
PIDs pids.current / pids.peak fork rate（bpftrace） pids.events max

USE 檢查順序：先 Errors（最緊急）→ 再 Saturation（PSI 高表示壓力大）→ 最後 Utilization（用量分析）。 多數人直覺看 utilization，但使用率高不一定有問題——使用率 90% 但 saturation 為 0，表示用得很滿但沒卡住。反之使用率 30% 但 saturation 高，表示被某個東西卡住（鎖、IO wait、scheduler 不公）。

3.2 RED method：服務視角

USE 是「資源」視角，RED method（Rate、Errors、Duration）是「服務」視角：

  • Rate：每秒 request 數
  • Errors：每秒錯誤數（5xx、failed RPC）
  • Duration：request latency（p50、p95、p99）

cgroup 場景下，RED 跟 USE 互補：RED 告訴你「使用者體驗多差」，USE 告訴你「資源層為什麼差」。 典型 debug 流程：RED 偵測異常（latency p99 上升）→ USE 找原因（哪個資源 saturation 高）→ cgroup 介面定位（哪個 cgroup 是肇事者）→ 修復。

3.3 PSI-driven tuning：迭代式調校

傳統的 cgroup limit 設定靠經驗值與「看起來合理的數字」，後果就是常見的兩種失敗——設太緊（throttle 嚴重）或設太鬆（沒保護效果）。 PSI-driven tuning 用 PSI 作為迭代訊號：

具體 PSI 閾值參考

Workload 類型some avg10 目標full avg10 目標
Latency-critical（DB、cache、API） < 1% 0%
互動服務（web app、microservice） < 5% < 0.5%
Best-effort（batch、async job） < 20% < 5%
Background（log shipping、metrics） 容忍高 容忍高

3.4 容量規劃公式

從歷史資料推算合理 limit 是性能調校的基礎。實務經驗法則：

介面建議公式意義
memory.max / cpu.max peak × 1.5 給 burst 留 50% headroom，hard limit
memory.high p95 × 1.2 或 limit × 0.9 正常運作 95% 時間在 high 之下，超過先 throttle
memory.min / memory.low p50 working set 核心 working set，受保護
cpu.weight 業務優先級比例（4:2:1） 不需要絕對數字，比例最重要
pids.max peak × 2 fork-heavy 應用要寬鬆，防 abnormal spike
io.max 裝置總頻寬 × 80% 不要設裝置滿頻寬，保留給 system

3.5 Benchmark 三劍客

cgroup limit 設好後，必須驗證它真的有效。三個 benchmark 工具是標配：

fio：IO benchmark + cgroup IO controller 驗證

# 基本 IO benchmark
fio --name=random-rw \
    --rw=randrw --rwmixread=70 \
    --bs=4k --size=1G --numjobs=4 \
    --runtime=60 --group_reporting \
    --time_based --ioengine=libaio \
    --direct=1 --iodepth=32

# 在 cgroup 內跑（驗證 io.max 限制）
sudo systemd-run --slice=user.slice --unit=fio-test \
    -p IOWriteBandwidthMax="/dev/nvme0n1 100M" \
    --pty \
    fio --name=write-test --rw=write --bs=1M --size=2G \
        --runtime=30 --time_based --direct=1 \
        --filename=/data/test.bin

# 預期：write throughput 約 100MB/s（被 cgroup 限制）
# 如果跑出 800MB/s，io.max 沒生效（檢查 +io 是否啟用、filesystem 支援）

stress-ng：CPU / memory / syscall 壓力測試

# CPU 壓力（驗證 cpu.max throttle）
sudo systemd-run --slice=user.slice --unit=cpu-test \
    -p CPUQuota=50% \
    --pty \
    stress-ng --cpu 4 --timeout 30s --metrics-brief

# 預期：總 CPU 利用率約 50%（被 cgroup 限制）

# Memory 壓力（驗證 memory.max OOM）
sudo systemd-run --slice=user.slice --unit=mem-test \
    -p MemoryMax=200M \
    --pty \
    stress-ng --vm 1 --vm-bytes 500M --timeout 30s

# 預期：被 OOM kill（cgroup 限制 200M，要 500M）

# 完整 cgroup behavior 驗證
stress-ng --vm 4 --vm-bytes 256M \
          --cpu 4 \
          --io 2 \
          --timeout 60s \
          --metrics-brief

wrk：HTTP 端到端 latency

# 對 service 跑 HTTP 壓測
wrk -t 4 -c 100 -d 60s --latency \
    http://your-service/api/endpoint

# 範例輸出：
# Latency Distribution
#    50%    1.20ms
#    75%    2.40ms
#    90%    5.50ms
#    99%   25.30ms
# Requests/sec:  8534.21

# 在跑 wrk 的同時觀察 PSI（另一 terminal）
watch -n 1 "cat /sys/fs/cgroup/system.slice/your-service.service/cpu.pressure"

# 對比有/無 cgroup limit 的 latency
# 如果 limit 太緊，p99 會大幅上升

三工具搭配場景：

  • fio：驗證 io.max / io.weight、找 IO 瓶頸、測 NVMe / SSD 真實效能
  • stress-ng：驗證 cpu.max / memory.max、模擬 noisy neighbor、測試 OOM 行為
  • wrk：端到端 latency、user-perceived performance、SLA 驗證

3.6 Playbook 1：OOM 調查

Pod / service 莫名重啟，懷疑 OOM。診斷流程： 

# Step 1: 確認是 OOM（不是 crash 或 SIGKILL）
journalctl -u myapp.service | grep -i 'oom\|killed'
dmesg | grep -i 'killed process'

# Step 2: 看 cgroup events（最權威來源）
CG=$(systemctl show -p ControlGroup --value myapp.service)
cat /sys/fs/cgroup$CG/memory.events
# 如果 oom_kill > 0，確認是 cgroup OOM

# Step 3: 看 OOM 時的 memory.stat
cat /sys/fs/cgroup$CG/memory.stat | head -20
# 重點：anon、file、kernel_stack
# anon 大表示真實 leak / 高需求；file 大可能只是 page cache

# Step 4: 看歷史峰值
cat /sys/fs/cgroup$CG/memory.peak
# 跟 memory.max 比，差距多少？

# Step 5: 看 dmesg 完整 OOM 訊息
dmesg | grep -A 30 'Memory cgroup out of memory'
# 會顯示被殺的 process、其他 process 用量、reason

# Step 6: 判斷原因
# - peak ≈ max → 容量不足，調大 limit
# - peak << max 但 OOM → 突發 spike，需要看 anon 增長率（用 bpftrace）
# - oom_group_kill > 0 → Pod 整體 atomic kill（K8s 預期行為）

# Step 7: 修復
# 短期：調大 memory.max
# 中期：設 memory.high 提早 throttle warn
# 長期：找應用 leak、優化 working set

3.7 Playbook 2：CPU throttling 排查

應用響應變慢，懷疑被 cgroup CPU throttle： 

# Step 1: 看 cpu.stat 確認 throttle
CG=$(systemctl show -p ControlGroup --value myapp.service)
cat /sys/fs/cgroup$CG/cpu.stat
# 重點：
#   nr_throttled        ← throttle 次數，> 0 表示被限制
#   throttled_usec      ← throttle 累計微秒
#   nr_periods          ← cpu.max period 數

# Step 2: 計算 throttle 比例
# throttle_pct = throttled_usec / (nr_periods * period)
# 如果 > 5%，throttle 已影響性能

# Step 3: 看 cpu.pressure
cat /sys/fs/cgroup$CG/cpu.pressure
# some avg10 高表示有 task 在等 CPU

# Step 4: 看 cpu.max 設定
cat /sys/fs/cgroup$CG/cpu.max
# "$QUOTA $PERIOD" 格式，例如 "100000 100000" = 1 CPU

# Step 5: 看實際使用量
# 算每秒 CPU usage
sleep 10
USAGE_NOW=$(awk '/usage_usec/{print $2}' /sys/fs/cgroup$CG/cpu.stat)
sleep 10
USAGE_LATER=$(awk '/usage_usec/{print $2}' /sys/fs/cgroup$CG/cpu.stat)
echo "CPU usage rate: $((($USAGE_LATER - $USAGE_NOW) / 10000000))%"

# Step 6: 修復
# - 應用真的需要更多 CPU → 調大 cpu.max
# - 應用 over-fork（每 fork 都 hit max）→ 看 thread 數，可能要 limit 應用
# - 短期 burst → 啟用 cpu.max.burst 允許 burst（kernel 5.14+）
echo "200000 100000 50000" > /sys/fs/cgroup$CG/cpu.max  # quota period burst

3.8 Playbook 3：IO 瓶頸定位

Service latency p99 突然上升，看起來是 IO 問題： 

# Step 1: 確認是 IO 問題
CG=$(systemctl show -p ControlGroup --value myapp.service)
cat /sys/fs/cgroup$CG/io.pressure
# some/full avg10 高 → 確認 IO 是瓶頸

# Step 2: 看 IO 模式
cat /sys/fs/cgroup$CG/io.stat
# rbytes / wbytes / rios / wios per device
# 計算 IOPS、bandwidth、avg io size

# Step 3: 區分讀寫
# 讀 spike：看是不是 cache miss（memory.stat workingset_refault_file）
# 寫 spike：看是不是 buffered write 累積（memory.stat file_dirty）

# Step 4: 找肇事者（如果是 multi-tenant）
# 對所有 cgroup 的 io.pressure 排序
for cg in /sys/fs/cgroup/system.slice/*.service; do
    name=$(basename "$cg")
    some=$(grep '^some' "$cg/io.pressure" 2>/dev/null | grep -oP 'avg10=\K[\d.]+')
    [ -n "$some" ] && echo "$some $name"
done | sort -rn | head -10

# Step 5: 用 fio 重現問題
# 在受害者 cgroup 內跑 fio，量 latency 分布
sudo systemd-run --slice=$(systemctl show -p Slice --value myapp.service) \
    --pty \
    fio --name=test --rw=randread --bs=4k --size=1G \
        --runtime=30 --time_based --ioengine=libaio \
        --direct=1 --iodepth=32 \
        --output-format=json+ | jq '.jobs[].read.lat_ns.percentile'

# Step 6: 修復
# - 設 io.latency 保護（如果是 multi-tenant noisy neighbor）
# - 設 io.weight 調整優先級
# - 設 io.max 給 batch jobs 防爆
# - filesystem 層：確認 writeback 設定（vm.dirty_*）
# - 硬體層：升級 NVMe、加裝 read cache

3.9 Playbook 4：PSI 告警處理

Prometheus alert 觸發 memory.pressure full avg10 > 5%： 

# Step 1: 確認 alert 範圍（哪個 cgroup？）
# alert label 應該包含 service / pod 名稱

# Step 2: 看當前 PSI 與歷史趨勢
CG=$(systemctl show -p ControlGroup --value $SERVICE)
cat /sys/fs/cgroup$CG/memory.pressure
# 看 avg10、avg60、avg300 比較——是 spike 還是持續？

# Step 3: 看 events
cat /sys/fs/cgroup$CG/memory.events
# 有 oom_kill？已經死過了，是 K8s 重啟
# 只有 high count？throttling 中，可能還活著

# Step 4: 看當前 stat
cat /sys/fs/cgroup$CG/memory.stat | head -10
cat /sys/fs/cgroup$CG/memory.current
cat /sys/fs/cgroup$CG/memory.peak

# Step 5: 找根因
# - 是不是流量 spike？看 RED metrics 的 rate
# - 是不是 memory leak？看 anon 增長趨勢（多採幾次 stat）
# - 是不是 noisy neighbor 觸發 reclaim？看其他 cgroup 的 events

# Step 6: 緩解（短期 + 長期）
# 短期：
#   調大 memory.max（如果 hardware 允許）
#   殺長期跑的 background job 釋放記憶體
#   減少 traffic（degrade non-critical features）
# 長期：
#   修 leak
#   重新設計 working set
#   水平擴容（再加 instance）

# Step 7: 寫 post-mortem，更新 runbook

3.10 性能調校的 Don'ts

實戰累積的經驗——這些事不要做：

  • 不要拍腦袋設 limit：先量，再設。沒 baseline 的 limit 都是猜的。
  • 不要只設 hard limit 不設 soft limit：memory.max 沒搭配 memory.high 等於沒緩衝，OOM 直接發生。
  • 不要用 utilization 當 alert 指標：utilization 高不一定有問題，用 PSI。
  • 不要 production 上做 destructive benchmark：fio direct=1 可能毀檔案，stress-ng vm 可能 OOM。先 staging。
  • 不要對所有 service 用同一套 limit：DB、web、batch 的 access pattern 完全不同，公式不同。
  • 不要忽略 PSI 的 total：avg10 看似正常但 total 在快速增長 = 有秒級 spike，使用者已感受到。

    --pty \
    python3 -m http.server 8888 &

# Phase 4: 看 PSI

# Phase 5: 判斷
# some avg10 < 1% → limit 合理
# some > 5% → 太緊，調大
# events oom > 0 → 嚴重不夠

# 清理
sudo systemctl stop baseline

Lab 5 · 寫 OOM 自動 post-mortem 腳本
#!/bin/bash
# oom-postmortem.sh
# 偵測 OOM 後自動收集 cgroup 完整狀態

CG=${1:?usage: $0 /sys/fs/cgroup/PATH}
OUT=/tmp/oom-pm-$(date +%s).txt

{
    echo "=== OOM POST-MORTEM ==="
    echo "Time: $(date)"
    echo "Cgroup: $CG"
    echo ""
    
    echo "=== memory.events ==="
    cat $CG/memory.events
    echo ""
    
    echo "=== memory.stat (key fields) ==="
    grep -E '^(anon|file|kernel_stack|slab|pgfault|pgmajfault)' $CG/memory.stat
    echo ""
    
    echo "=== memory.current / peak / max ==="
    echo "current: $(cat $CG/memory.current)"
    echo "peak:    $(cat $CG/memory.peak)"
    echo "max:     $(cat $CG/memory.max)"
    echo ""
    
    echo "=== memory.pressure ==="
    cat $CG/memory.pressure
    echo ""
    
    echo "=== Recent dmesg OOM ==="
    dmesg | grep -A 20 'Memory cgroup out of memory' | tail -50
    echo ""
    
    echo "=== Top processes in cgroup ==="
    for pid in $(cat $CG/cgroup.procs 2>/dev/null); do
        ps -o pid,rss,comm -p $pid 2>/dev/null
    done | sort -k2 -rn | head -10
    
} > $OUT

echo "Post-mortem saved to: $OUT"
05 · Pitfalls

常見陷阱

陷阱 1 · benchmark 不在 production-equivalent 環境跑

在開發機跑 fio 看到 NVMe 1.5GB/s，把 production limit 設成 1.5GB/s——production 上 NVMe 是不同型號，實際只有 800MB/s，limit 反而太高沒保護效果。 修法：benchmark 在實際 production 機型上跑（staging 用相同 hardware）。OpenBenchmarking.org 有大量實機數據可參考。

陷阱 2 · 只跑 1 次 benchmark 就下結論

fio 一次跑出 950 MB/s，認為這是真實值——但這次可能 cache hit 高、或剛好沒 noisy neighbor。 修法：至少跑 3 次取平均，最好跑 10 次看分布。stress-ng 的 --times 參數讓 benchmark 自動重複。注意 standard deviation——如果 stdev 大，limit 要設保守。

陷阱 3 · fio direct=1 沒注意造成 buffered write 偽訊號

fio 預設 buffered IO，吃 page cache 時 throughput 看似超高（記憶體頻寬），不是真實 disk 速度。 修法：要量 disk 真實效能，--direct=1 必加（bypass page cache）。要量含 cache 效能（real workload）就不加，但要區分清楚。

陷阱 4 · 用 stress-ng 在 production 機器跑

production 機跑 stress-ng --vm 4 --vm-bytes 4G——4GB 記憶體分配可能觸發 OOM 殺到 production service。 修法：stress-ng 永遠在 staging / dedicated test 機跑。production 要 stress test 用 canary deploy + 漸進流量加壓，不直接跑 stressor。

陷阱 5 · CPU throttling 看 cpu.stat 卻忘了 cpu.pressure

cpu.stat 的 nr_throttled = 0，覺得「沒被 throttle」——但 cpu.pressure 顯示 some avg10 = 30%。 原因：throttle 是「被 cgroup quota 限制」、pressure 是「等 CPU 排程」。沒 throttle 不代表沒 CPU 競爭——可能是同 cgroup 內 thread 互相搶。 修法：兩個一起看。pressure 高但 throttle 為 0 表示是內部競爭，調 cpu.max 沒用，要看 thread 數或上下游 dependency。

陷阱 6 · OOM 後沒看 memory.events 就改 limit

看到 OOM 直接「調大 memory.max 從 1G 到 2G」——但其實 events 顯示 oom_kill = 0、只有 high count 飆漲，根本沒 OOM。實際是 memory.high 太緊造成 throttle，被誤認為 OOM。 修法：先看 events 確認類型：oom_kill > 0 才是真 OOM；只有 high > 0 是 throttling，調 memory.high 而非 max。

陷阱 7 · capacity planning 用 average 而非 percentile

看到 cgroup average memory 用 500MB，把 limit 設 750MB（×1.5）——但實際 p99 是 900MB，每天都 OOM 幾次。 修法：永遠用 percentile（p95、p99）規劃，不用 average。Prometheus 的 histogram_quantile 或 quantile_over_time 是必備工具。

陷阱 8 · benchmark 結果套用到不同 workload pattern

fio 跑 randread 4k 看 NVMe 性能，套用到 sequential write 1MB 場景——數字差 10 倍以上。 修法：benchmark 要模擬真實 workload pattern。DB 跑 4k random read+write 70/30；分析 workload 跑 1M sequential read；log shipping 跑 4k+ sequential write。每個 service 要量自己的 IO pattern。

06 · Exercises

動手練習

練習 1 · 完整 USE 分析（30 分鐘）

對你機器上一個 active service 跑完整 USE 分析：

  1. 列出所有 active services（systemctl list-units --state=active）
  2. 選一個有意思的（如 systemd-resolved、postgresql 等）
  3. 對每個資源（CPU、Memory、IO、PIDs）填出 Utilization、Saturation、Errors 三個指標的當前值
  4. 判斷哪個資源「最該擔心」？
練習 2 · 設計完整 capacity planning（20 分鐘）

假設你要規劃一個 8GB 機器跑：

  • PostgreSQL：working set p50=2GB、p95=3GB、peak=4GB
  • Redis cache：working set p50=1GB、p95=1.2GB、peak=1.5GB
  • 應用 service：working set p50=500MB、p95=800MB、peak=1.2GB

用本章公式：

  1. 每個 service 的 memory.min / memory.high / memory.max 各設多少？
  2. 系統還剩多少 RAM 給 OS / page cache？
  3. 如果加一個 batch service（peak 1GB）能塞進來嗎？怎麼塞？
練習 3 · 跑 wrk benchmark + PSI 對照（30 分鐘）

對自己的某個 web service 跑：

# Baseline
wrk -t 4 -c 100 -d 60s --latency $URL > /tmp/baseline.txt

# 跑同時觀察 PSI（背景）
(
    while :; do
        cat /sys/fs/cgroup/system.slice/$SERVICE/memory.pressure
        cat /sys/fs/cgroup/system.slice/$SERVICE/cpu.pressure
        sleep 5
    done
) > /tmp/psi.log &
PSI_PID=$!

wrk -t 4 -c 100 -d 60s --latency $URL > /tmp/with-load.txt

kill $PSI_PID

分析：

  1. load 下 p99 latency 比 baseline 多多少？
  2. load 期間 PSI 數字怎麼變？
  3. PSI 與 latency 的 correlation 強嗎？
練習 4 · 用 4 個 playbook 解一個歷史 incident（思考題，15 分鐘）

回想你過去處理過的一個 production incident（OOM、CPU 飽和、IO 慢、latency spike），用本章 playbook 重做：

  1. 當時你用的調查路徑是什麼？
  2. 跟 playbook 比，少了哪些步驟？
  3. 有沒有可能更快定位？
  4. 修法是治標還是治本？
07 · Self-check

自我檢查題

Q1.USE method 與 RED method 各自視角是什麼？兩者怎麼互補？
USE method（Brendan Gregg）：資源視角。對每個資源（CPU、Memory、IO、Network）檢查：
  • Utilization：用了多少（百分比、用量）
  • Saturation：多塞（queue depth、PSI）
  • Errors：錯誤數
RED method：服務視角：
  • Rate：req/sec
  • Errors：error/sec
  • Duration：latency p50 / p95 / p99
互補關係：
  1. RED 告訴你「使用者體驗多差」
  2. USE 告訴你「哪個資源是瓶頸」
  3. cgroup 介面告訴你「哪個 cgroup / 應用是肇事者」
典型流程：RED 偵測 latency p99 上升 → USE 找到 memory saturation 高 → cgroup events 看到某 cgroup 大量 oom_kill → 修復。 cgroup v2 的價值：讓 USE 從 per-host 變成 per-cgroup，故障定位顆粒度大幅提升。
Q2.PSI-driven tuning 為什麼比拍腦袋設 limit 好？完整流程是什麼？
為什麼 PSI-driven 好：
  • 傳統做法：「看起來合理就設」——常見失敗：太緊（throttle）或太鬆（沒保護）。
  • PSI 直接量化「使用者感受」——客觀指標。
  • 迭代式逼近最佳解，不是一次性猜測。
完整流程：
  1. baseline：無 limit 跑 workload，記錄 peak、p95、p50。
  2. 設第一輪 limit：peak × 1.5（hard）、p95 × 1.2（soft）、p50（min/request）。
  3. 跑 load test：完整 traffic cycle（含 peak 時段）。
  4. 看 PSI：
    • some > 10% → 太緊，調大
    • some < 1% → 可能太鬆（如果是 cost-sensitive）
    • events 有 oom/throttle → 嚴重需調
  5. 迭代：回 step 2，調整 limit 再測。
  6. 合理範圍後部署 production。
K8s 整合：1.36+ In-Place Pod Resize 配合 PSI metrics 可做自動調校（VPA-style）。
Q3.容量規劃公式（peak × 1.5、p95 × 1.2、p50）的意義？為什麼不用 average？
公式意義：
  • peak × 1.5（memory.max / cpu.max）：給 burst 留 50% headroom，hard limit 防爆。
  • p95 × 1.2（memory.high / cpu soft cap）：正常運作 95% 時間在 high 之下，超過時先 throttle 警告——給 monitor 介入時間。
  • p50（memory.min / memory.low）：核心 working set，受保護不被 reclaim。
為什麼不用 average：
  1. 掩蓋 spike：average 500MB 但 p99 900MB——按 average 設 limit 每天 OOM 幾次。
  2. 長尾分布：實際 workload 分布長尾（多數低、偶爾高），average 嚴重低估真實需求。
  3. SLA 視角：使用者體驗看尾延遲（p99），不看平均。
實務工具：
  • Prometheus：histogram_quantile(0.95, rate(memory_usage[5m]))
  • 記錄 7-30 天，覆蓋完整 traffic cycle（週末 vs 平日、月初月底）
  • 定期 review，traffic pattern 變化時 limit 也要調
Q4.fio、stress-ng、wrk 三個 benchmark 工具各自最適合什麼場景？
  • fio（Flexible IO Tester）：
    • 場景：IO performance benchmark、io.max / io.weight 驗證、找 IO 瓶頸
    • 強項：完整 IO pattern（random / sequential、read / write、mixed）、direct IO 支援、多 worker 並行
    • 典型用法：fio --rw=randrw --bs=4k --direct=1 --iodepth=32
  • stress-ng：
    • 場景：CPU、memory、syscall 多維度壓力測試、模擬 noisy neighbor、cpu.max / memory.max 行為驗證
    • 強項：300+ stressors，包含異常情境（fork bomb、page fault storm）；--metrics-brief 自動 report
    • 典型用法：stress-ng --vm 4 --vm-bytes 256M --cpu 4 --timeout 60s
  • wrk：
    • 場景：HTTP service 端到端 latency、user-perceived performance、SLA 驗證
    • 強項：高並發 client（c4 c500+）、Lua script 自訂 request、percentile latency report
    • 典型用法：wrk -t 4 -c 100 -d 60s --latency $URL
搭配使用：典型流程是 stress-ng 模擬資源壓力 + wrk 量 user-side latency + fio 量底層 IO——三者組合複現 production 場景。
Q5.OOM 調查 playbook 的關鍵步驟是什麼？怎麼區分真 OOM、throttling、與 SIGKILL？
關鍵步驟：
  1. journalctl + dmesg：確認是 OOM（"Memory cgroup out of memory"）還是其他
  2. memory.events：看 oom_kill 數，> 0 確認是 cgroup OOM
  3. memory.stat：看 anon / file 比例判斷 leak vs cache
  4. memory.peak vs max：差距多少？
  5. dmesg OOM 訊息：被殺的 process、其他 process 用量
  6. 判斷原因：peak ≈ max（容量不足）、peak << max（突發 spike）、oom_group_kill（K8s atomic）
區分三種 kill：
  • 真 OOM kill：dmesg 有 "Memory cgroup out of memory"、memory.events oom_kill 增加。
  • throttling（誤認 OOM）：應用慢但活著，memory.events 只有 high 增加，oom_kill = 0。修法是調 memory.high。
  • 外部 SIGKILL：systemd 觸發（OOMScoreAdjust）、K8s evict、運維手動 kill。dmesg 沒 OOM 訊息，journal 顯示 "Killed signal received"。
實務技巧：寫 OOM post-mortem 腳本（Lab 5）自動收集所有資料，事後分析比即時 debug 高效。
Q6.CPU throttling 與 CPU pressure 的差別？兩者都看可以發現什麼問題？
CPU throttling（cpu.stat）：「被 cgroup quota 限制」。當 cgroup 在一個 period 內用完 quota，剩下時間 task 被 throttle 不能跑。看 nr_throttled、throttled_usec。 CPU pressure（cpu.pressure）：「有 task 在等 CPU schedule」。可能是被 throttle、也可能是同 cgroup 內 thread 太多互搶。 兩者組合分析：
  • throttle 高 + pressure 高：cpu.max 設太緊，調大就能改善。
  • throttle 為 0 + pressure 高：內部競爭——thread 數太多、互相搶 CPU。調 cpu.max 沒用，要看應用 thread 池大小或 CPU pinning。
  • throttle 高 + pressure 低：奇怪——通常是 thread 數少於 quota，每個 thread 都被 throttle 但其他 thread 沒在等。可能是 application sleep / IO wait 多。
  • 都低：CPU 沒問題，瓶頸在其他資源。
實務 debug 步驟：
  1. 看 cpu.pressure → 確認有 CPU 競爭
  2. 看 cpu.stat → 區分 quota 限制 vs 內部競爭
  3. 看 thread 數（ls /proc/PID/task | wc -l）→ 確認 thread pool
  4. 用 bpftrace sched_switch 看具體 schedule 模式
Q7.Multi-tenant 機器發現某個 service IO latency p99 上升。從本章 playbook 怎麼一步步定位？
步驟：
  1. 確認 IO 是瓶頸：cat io.pressure，some / full avg10 高 → 確認。
  2. 看 IO 模式：cat io.stat 算 IOPS、bandwidth、avg io size。判斷是 read spike 還是 write spike。
  3. 區分原因：
    • 讀 spike：cat memory.stat 看 workingset_refault_file（page cache miss）
    • 寫 spike：cat memory.stat 看 file_dirty（buffered write 累積）
  4. 找肇事者（multi-tenant）：對所有 service 的 io.pressure some 排序，找 top 10。 
    for cg in /sys/fs/cgroup/system.slice/*.service; do
    some=$(grep some $cg/io.pressure | grep -oP 'avg10=\K[\d.]+')
    [ -n "$some" ] && echo "$some $(basename $cg)"
done | sort -rn | head
  5. 用 fio 重現：在受害者 cgroup 內跑 fio，量 latency 分布。 
    systemd-run --slice=victim.slice fio --rw=randread --bs=4k --runtime=30 --time_based
  6. 修復：
    • 對受害者設 io.latency target（multi-tenant 保護首選）
    • 對肇事者設 io.weight 調低或 io.max 防爆
    • filesystem 層調 vm.dirty_ratio（writeback 行為）
    • 硬體層升級 NVMe / 加 read cache
真實案例：DB latency p99 從 5ms 上升到 50ms。pressure some 25% → 確認 IO。io.stat 看 wbytes 飆升 → 寫 spike。對所有 cgroup 排序找到 backup.service 在跑 → 對 backup 設 io.weight=50，對 DB 設 io.latency target=10ms → p99 恢復 8ms。
Q8.實作思考題：你接手一個沒有 cgroup limit 的 production 環境，要建立完整的 cgroup-based 性能管理體系。請設計從 baseline 量測到自動調校的完整流程。
完整流程（5 階段）： 
[Phase 1: 觀測基礎建設（1-2 週）]
- 部署 PSI metrics 採集（Prometheus + node_exporter textfile）
- 部署 cgroup events 監聽（基於 inotify 的 daemon）
- 部署 eBPF 觀測（cAdvisor / Pixie / 自寫 libbpf agent）
- 確保收集：所有 service 的 PSI、events、stat
- Grafana dashboard 視覺化

[Phase 2: Baseline 量測（2-4 週）]
- 對每個 service 收集至少 7 天資料（含週末）
- 計算 p50 / p95 / p99 / peak
- 識別 traffic pattern：daily peak、weekly peak、event-driven spike
- 對 latency-critical service 額外跑 wrk benchmark 建立 SLA baseline
- 對 IO-heavy service 跑 fio 量裝置真實能力

[Phase 3: 第一輪 limit（漸進式 rollout）]
- 用本章公式：peak × 1.5、p95 × 1.2、p50
- 從非 critical service 先設限（log shipper、metrics agent）
- 觀察 1 週，看 PSI、events 反應
- 確認沒誤殺，再對 critical service 設限
- 每次只動 1 個 service，留 24 小時觀察

[Phase 4: 進階 tuning]
- 對 latency-critical 加 io.latency / cpu.idle co-location
- 對 multi-tenant 加 io.weight 比例分配
- 啟用 systemd-oomd（ManagedOOMMemoryPressure）
- 設定 memory_recursiveprot mount option
- 對 K8s 啟用 Memory QoS（v1.34 Beta）

[Phase 5: 自動化 + closed loop]
- Prometheus alert：PSI / events 閾值觸發
- VPA / HPA 整合 PSI metrics 自動調 limit
- K8s 1.36+ In-Place Resize 不重啟調整
- 月度 review：用過去 30 天資料重算 limit
- 寫 runbook：每個 alert 對應 playbook
關鍵原則：
  1. 觀測先行：沒 metrics 不要設 limit，盲目限制比沒限制糟。
  2. 漸進 rollout：一次一個 service，留觀察期。
  3. 非 critical 先試：失敗代價低。
  4. PSI 是主指標，utilization 是輔助。
  5. 定期 review：traffic pattern 會變，limit 要隨之調整。
常見失敗：直接套網路上找的「建議 limit」、跳過 baseline、所有 service 同時設限。本章 playbook 是用來避免這些坑。
