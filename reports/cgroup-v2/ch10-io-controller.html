IO controller 深度解析

v1 IO 控制最大的失敗在於「buffered IO 完全管不到」——絕大多數應用是 buffered write，v1 對它們束手無策。 v2 透過 memory + io controller 協同工作徹底解決這問題。本章拆解 io.weight、io.max、io.cost、io.latency 四種模型、 cgroup writeback 機制、Memory Ownership 與 IO Ownership 的差異，並用 Meta fbtax2 實戰案例把所有概念串起來。

~7,500 字 閱讀時間 28 分鐘 難度：核心 最後更新 2026-04-28
01 · Highlights

本章重點

  1. v2 解決 v1 最大缺陷：v1 只能控制 direct IO（read 與 metadata），buffered write 完全失控。v2 透過 memory + io controller 協同實現完整 IO 控制。
  2. 四種控制模型：io.weight（比例分配，需 blk-iocost）、io.max（BPS/IOPS 硬上限）、io.cost（基於成本模型的 QoS 控制）、io.latency（基於延遲的 workload 保護）。
  3. cgroup writeback 三要件：（1）v2 啟用 memory + io controller、（2）filesystem 支援（ext4 / btrfs / xfs / f2fs / ext2）、（3）正確掛載。三者缺一，buffered write 會歸到 root cgroup。
  4. IO Ownership 與 Memory Ownership 不同：memory 按 page 計（first-touch），io 按 inode 計（且會根據主要 dirtier 動態遷移）。理解這個對 debug 多 cgroup 共用檔案場景至關重要。
  5. device-keyed 介面：io.* 介面多數是 nested-keyed，每個裝置（$MAJ:$MIN）獨立配置。lsblk 看 device 編號，systemd 用 IODeviceWeight= 等 directive。

03 · Core Concepts

核心內容

3.1 為什麼 v1 IO 控制失敗

Ch.1 提過 v1 的 blkio 只能控制 direct IO，無法控制 buffered IO。這裡進一步解釋為什麼。

當應用呼叫 write(2) 寫檔案時，預設是 buffered write——資料先寫到 kernel page cache（記憶體），之後由 kernel 的 writeback worker thread 在背景把 dirty page 寫到磁碟。

關鍵問題：writeback worker 是 kernel thread，不屬於原始應用 cgroup。當 bio 送到 block layer 時，block layer 看到的是 worker thread 的 cgroup（root）——v1 的 blkio 控制只看 bio 的 cgroup，所以所有 buffered write 都歸到 root cgroup。

實際影響：你寫 blkio.throttle.write_bps_device 8:0 10485760 限速 10MB/s，但 dd if=/dev/zero of=/data/file 跑了 800MB/s——因為 dd 寫的是 page cache，根本沒受 blkio 限制。

v2 怎麼解決：memory controller 知道哪個 cgroup 弄髒了 page，writeback worker 把這個資訊帶到 block layer，bio 的 cgroup 變成「真正寫資料的 cgroup」而非 worker thread 的 cgroup。這需要 memory + io controller 在同一 hierarchy 協同——這在 v1 multi-hierarchy 模型下根本做不到，是 v2 unified hierarchy 設計的直接收益。

3.3 io.weight：比例分配（blk-iocost）

io.weight 的預設 backend 是 blk-iocost——一套基於成本模型的 IO QoS 控制器。要使用，需要先在 root cgroup 啟用 blk-iocost： 

# 1. 確認裝置
lsblk
# 假設目標是 /dev/sda（major:minor = 8:0）

# 2. 啟用 blk-iocost（root cgroup）
echo "8:0 enable=1" | sudo tee /sys/fs/cgroup/io.cost.qos

# 3. 自動偵測裝置成本模型
echo "8:0 ctrl=auto" | sudo tee /sys/fs/cgroup/io.cost.model

# 4. 在 cgroup 設 weight
echo "200" | sudo tee /sys/fs/cgroup/myapp/io.weight
# 對特定裝置設 weight
echo "8:0 200" | sudo tee /sys/fs/cgroup/myapp/io.weight

weight 行為：1–10000，預設 100。和其他 weight 一樣只在競爭時生效。沒競爭時，cgroup 可以吃滿 IO bandwidth，不被 weight 限制。

blk-iocost 的成本模型

blk-iocost 用「每個 IO 的成本」（virtual time）來計量，不同裝置成本不同。模型有兩種：

  • linear（手動）：管理員提供 rbps/rseqiops/rrandiops/wbps/wseqiops/wrandiops 六個參數，blk-iocost 用這個 baseline 計算每個 IO 的成本。
  • auto（自動）：kernel 用內建的裝置 profile（如 SSD、NVMe）自動推斷成本。多數情況夠用。

3.4 io.max：硬性 BPS / IOPS 上限

最直觀的 IO 限制——每個方向獨立設 bandwidth 與 IOPS 上限： 

# 對 /dev/sda 設限：
# - 讀 bandwidth 最多 10MB/s
# - 寫 bandwidth 最多 10MB/s
# - 讀 IOPS 最多 1000
echo "8:0 rbps=10485760 wbps=10485760 riops=1000" | \
    sudo tee /sys/fs/cgroup/myapp/io.max

# 移除特定限制
echo "8:0 rbps=max wbps=max riops=max" | sudo tee io.max

# 看當前設定
cat /sys/fs/cgroup/myapp/io.max
# 8:0 rbps=10485760 wbps=10485760 riops=1000 wiops=max

支援的 key：rbps、wbps（bandwidth）、riops、wiops（IOPS）。每個方向獨立。

行為：超過 limit 後 IO 會被 delay（不是直接 fail），允許短期 burst。io.max 是硬限，平均不會超過——適合多租戶硬性 SLA 場景。

3.5 io.cost：blk-iocost QoS 控制

io.cost.qos 是 root-only 介面，啟用 blk-iocost 並設定 latency 為基礎的 QoS 參數： 

# 對 /dev/sda 啟用 blk-iocost，設定 QoS 參數
echo "8:0 enable=1 ctrl=auto rpct=95.00 rlat=5000 wpct=95.00 wlat=5000 min=50.00 max=150.00" \
    | sudo tee /sys/fs/cgroup/io.cost.qos

關鍵參數：

  • rpct=95.00：以讀延遲的 95th percentile 作為衡量點
  • rlat=5000：讀延遲目標 5000μs (5ms)
  • wpct/wlat：寫的對應參數
  • min=50.00：當裝置壓力大時，throttle 到正常 vrate 的 50%
  • max=150.00：當裝置充裕時，允許跑到正常 vrate 的 150%

運作原理：blk-iocost 持續監控裝置 latency，當超過目標就降低整體 throughput（拉低 max-min 範圍上界），裝置壓力下降後再放開。 這個機制讓多 cgroup 在共用裝置上能動態平衡——比 io.max 的固定 limit 更彈性。

3.6 io.latency：基於延遲的 workload 保護

io.latency 是 v2 IO controller 的殺手鐧——對「受保護」cgroup 設 latency target，當 latency 超過目標時，throttle 同層的其他 cgroup。

kernel 文件原話：「You provide a group with a latency target, and if the average latency exceeds that target the controller will throttle any peers that have a lower latency target than the protected workload.」

圖 10.1 · io.latency 的 peer-level 影響範圍

io.latency 配置範例（fbtax2 案例）

Meta 的 fbtax2 專案是 io.latency 的經典案例： 

# 對 /dev/sda 設定保護
# workload.slice 是優先工作負載，latency target 50ms
echo "8:0 target=50" | sudo tee \
    /sys/fs/cgroup/workload.slice/io.latency

# system.slice 是次要，latency target 75ms
echo "8:0 target=75" | sudo tee \
    /sys/fs/cgroup/system.slice/io.latency

# 結果：
# 當 workload.slice 的 IO latency > 50ms，kernel 會 throttle system.slice
# 但只在 system.slice (target 75ms) 比 workload.slice 寬鬆時 throttle

選擇 target 的訣竅（kernel 文件建議）：

  1. 不要設低於裝置物理 latency（HDD ~5–10ms、SSD ~0.1–1ms、NVMe ~0.05–0.5ms）
  2. 先觀察 io.stat 的 avg_lat 看正常運作 latency
  3. target 設成平常 latency 的 1.2 倍左右——夠寬鬆不會誤觸發，夠嚴格能保護 latency

3.7 cgroup writeback：buffered write 的關鍵

前面講過 v2 的 IO 控制需要 memory + io 協同。具體機制是 cgroup writeback。

當 process 在 cgroup A 寫 buffered IO：

  1. 資料寫進 page cache，page 被標記為「屬於 cgroup A」（memory ownership）。
  2. writeback kernel thread 看到 dirty page，用 page 的 cgroup A 來建構 bio。
  3. bio 送到 block layer，io controller 看到 bio 屬於 cgroup A，套用 A 的 io 限制。

要這個機制運作，需要：

  • cgroup v2 + memory + io 都啟用
  • filesystem 支援 cgroup writeback：ext2、ext4、btrfs、xfs、f2fs。其他 fs 上 buffered write 一律歸 root cgroup（fall back 到 v1 行為）。

memory pressure → IO pressure

cgroup writeback 也讓記憶體壓力轉化為自身的 IO 壓力：當 cgroup A 的記憶體 reclaim 觸發 dirty page flush，flush 走的是 A 的 io 限制——不會吵到別的 cgroup 的 IO。 這在 v1 是不可能的：v1 的 reclaim writeback 算到 root，會把所有 cgroup 的 IO 都拖慢。

3.8 IO Ownership 與 Memory Ownership 的差異

kernel 文件 §5.3 「Writeback」明確指出兩者的關鍵差異：

面向Memory OwnershipIO Ownership
追蹤粒度 per-page per-inode
歸屬時機 page 第一次被 touch（first-use） inode 第一次被 dirty
是否會遷移 否（page release 才釋放） 會（kernel 偵測「主要 dirtier」變動會切換 inode 歸屬）
多 cgroup 共用 各 page 獨立歸屬 整個 inode 只能歸一個 cgroup（會有 foreign pages 問題）

foreign pages 問題

當 cgroup A 寫了一個檔案，inode 歸 A；但 cgroup B 也開始寫同一個檔案——B 寫的 page 在 memory 視角下歸 B（per-page），但 writeback 仍按 inode 算（歸 A）。這些「歸屬不一致」的 page 稱為 foreign pages。

kernel 持續監控 foreign pages 數量，當某個 cgroup 變成主要 dirtier，會切換 inode 歸屬。但這個切換不是即時的，且多 cgroup 同時寫一個檔案的場景仍然支援不好——這是 cgroup writeback 的已知限制。

3.9 io.prio.class：IO 優先級類別

kernel 6.0+ 引入的介面，對應 ionice 的概念但作用於 cgroup： 

# 把整個 cgroup 的 IO 設為 idle（最低優先）
echo "idle" > /sys/fs/cgroup/batch.slice/io.prio.class

# 提升為 realtime（最高優先）
echo "promote-to-rt" > /sys/fs/cgroup/critical.slice/io.prio.class

支援的值：

  • no-change：不改變 process 自身的 IO priority（預設）
  • promote-to-rt：所有 IO 提升到 RT class
  • restrict-to-be：限制在 best-effort class（即使 process 設了 RT 也降級）
  • idle：所有 IO 變 idle class（CFQ 排程下只在無人用時跑）

3.10 io.stat 與 io.pressure 監控

io.stat 範例： 

$ cat /sys/fs/cgroup/myapp/io.stat
8:0 rbytes=1459200 wbytes=314773504 rios=192 wios=353 dbytes=0 dios=0

每個裝置一行，欄位：

  • rbytes / wbytes：讀 / 寫總 bytes
  • rios / wios：讀 / 寫總 IO 次數
  • dbytes / dios：discard 操作（TRIM）

io.pressure（PSI 介面）：與 cpu/memory pressure 相同格式。 some 表示「至少一個 task 在等 IO」、full 表示「所有 runnable task 都在等 IO」。 實務監控：some avg10 > 10% 是 IO 緊張的訊號，full avg10 > 1% 是嚴重 IO 飽和。

04 · Hands-on

實戰範例

Lab 1 · 證明 v2 能控制 buffered write
# 用 dd 跑 buffered write，看 v2 能不能限速

# 1. 建立 cgroup，啟用 memory + io
sudo mkdir /sys/fs/cgroup/lab10
echo "+memory +io" | sudo tee /sys/fs/cgroup/cgroup.subtree_control

# 2. 找出 /tmp 所在裝置
df /tmp
# 假設是 /dev/nvme0n1，major:minor 通常是 259:0

# 3. 設 write bandwidth 上限 10MB/s
DEVICE="259:0"  # 用你實際的 major:minor
echo "$DEVICE wbps=10485760" | sudo tee /sys/fs/cgroup/lab10/io.max

# 4. 啟動 shell 進入 cgroup
sudo bash -c "echo \$\$ > /sys/fs/cgroup/lab10/cgroup.procs; bash"

# 5. 跑 buffered write 測試
time dd if=/dev/zero of=/tmp/test bs=1M count=200
# 預期：~200 / 10 = 20 秒（因為 10MB/s 限速）
# v1 時代這個測試會在 1 秒內跑完（buffered write 完全不受限）

# 6. 清理
exit
rm /tmp/test
sudo rmdir /sys/fs/cgroup/lab10

觀察點：v2 真的能限制 buffered write！這是相對 v1 最重要的進步之一。

Lab 3 · io.latency 保護 latency-critical workload
# 場景：保護 db.service 的 IO latency 不被 batch.service 影響

# 啟動 db.service 和 batch.service（在同層 user.slice）

    -p IODeviceLatencyTargetSec="/dev/sda 20ms" \
    --pty \

        sleep 0.1
    done'

    -p IODeviceLatencyTargetSec="/dev/sda 100ms" \
    --pty \
    bash -c 'while :; do
        dd if=/dev/zero of=/tmp/batch.dat bs=1M count=50 oflag=direct 2>/dev/null
    done'

# 看 db-app 的 io.stat 看 avg_lat
DB_CG=$(systemctl show -p ControlGroup --value db-app.service)
cat /sys/fs/cgroup$DB_CG/io.stat

05 · Pitfalls

常見陷阱

陷阱 1 · 沒啟用 memory controller，buffered IO 不受限

在只啟用 io controller、沒啟用 memory controller 的 cgroup，buffered write 會繞過 io 限制——因為沒有 memory ownership 資訊讓 writeback 知道要套哪個 io 限制。 修法：cgroup.subtree_control 同時啟用 +memory +io。systemd 預設會啟用兩者，但手動建 cgroup 時要記得。

陷阱 2 · filesystem 不支援 cgroup writeback

在不支援 cgroup writeback 的 filesystem（NFS、tmpfs、CIFS、overlayfs 等）上，buffered write 全部歸 root cgroup。 即使你正確啟用 memory + io，行為跟 v1 一樣壞。 檢查：mount | grep YOUR_FS，目前支援的有 ext2、ext4、btrfs、xfs、f2fs。 修法：對 NFS / overlayfs 等，用 direct IO 或 sync mount。

陷阱 3 · io.weight 設了沒效果——blk-iocost 沒啟用

寫 io.weight=400 但沒看到 IO 比例變化？多半是 blk-iocost 沒啟用。 檢查：cat /sys/fs/cgroup/io.cost.qos，看裝置欄位是否有 enable=1。 修法：在 root cgroup 啟用 echo "8:0 enable=1 ctrl=auto" > io.cost.qos。

陷阱 4 · io.max 對 NVMe 沒效果

在某些 NVMe 裝置，io.max 似乎完全沒有 throttle 效果。原因：NVMe 走 multi-queue blk-mq，部分早期 kernel 對 blk-mq 的 IO throttle 支援不完整。 現代 kernel（5.x+）已修正，但若仍遇到，先確認 kernel 版本（uname -r）、再看 cat /sys/block/nvme0n1/queue/scheduler 看 scheduler 設定。 替代方案：用 io.cost / io.latency 取代硬性 io.max，更適合高速儲存。

陷阱 5 · io.latency target 設太低造成過度 throttle

把 io.latency target 設成 1ms（NVMe 物理 latency 之下），結果所有 cgroup 都被 throttle——因為「實際 latency > target」永遠成立。 修法：先看 io.stat 的 avg_lat（cgroup 正常運作的 latency），target 設成它的 1.2 倍左右。 不要設低於裝置物理 latency（NVMe ~100μs、SSD ~500μs、HDD ~5ms）。

陷阱 6 · 多 cgroup 寫同一檔案造成 IO 統計失真

cgroup A 與 B 同時寫 /data/shared.log，inode 歸某一個 cgroup（取決於 first dirtier 與遷移狀態）。 IO 限制只會套到 inode 的 owner，另一個 cgroup 的限制完全沒效果。 kernel 文件原話：「use cases where multiple cgroups write to a single inode simultaneously are not supported well」。 修法：避免多 cgroup 寫同一檔案。對 shared log 用 syslog 等 broker 集中寫入。

陷阱 7 · IO Priority 互動衝突

process 自己呼叫 ionice -c 1 cmd 設 RT priority，但 cgroup 設 io.prio.class=restrict-to-be——後者會把 process 強制降回 best-effort。 反過來，promote-to-rt 會把所有 IO 強制升 RT，覆蓋 應用自己設的 ionice。 修法：除非有明確意圖，io.prio.class 用 no-change（預設），讓應用自己控制 IO priority。

陷阱 8 · io.weight 數值不直覺

和 cpu.weight 類似，io.weight 範圍 1–10000、預設 100，但範圍取捨完全不同。 典型誤解：「系統服務設 io.weight=10，application 設 io.weight=100」期待 1:10 的比例——但要看實際裝置 vrate 與兩個 cgroup 的活躍狀態。 在 NVMe 等高速裝置上，差別小到看不出來；在 HDD 上才比較明顯。 建議：weight 比例不要差太多（最多 4–8 倍），裝置差別才會顯著反映在 throughput 上。

練習 1 · 比較 v1 與 v2 對 buffered write 的控制（15 分鐘）

練習 2 · 寫一個 io.pressure 監控腳本（10 分鐘）
#!/bin/bash
# io-pressure-monitor.sh
# 監控所有 service 的 io.pressure，當 some avg10 > 10% 就 alert

while true; do
    for cg in /sys/fs/cgroup/system.slice/*.service; do
        [ -f "$cg/io.pressure" ] || continue
        some=$(grep "^some" "$cg/io.pressure" | grep -oP 'avg10=\K[\d.]+')
        if [ -n "$some" ] && (( $(echo "$some > 10" | bc -l) )); then
            echo "$(date +%T) ALERT $(basename $cg): IO pressure some=$some%"
        fi
    done
    sleep 5
done

跑一段時間，看哪些 service 是 IO heavy。

07 · Self-check

自我檢查題

Q1.v1 為什麼控制不了 buffered write？v2 透過什麼機制解決？
v1 失敗原因：buffered write 透過 page cache 中介——資料先寫到 page cache，由 kernel writeback worker thread 在背景 flush 到磁碟。block layer 看到 bio 時，bio 的 cgroup 是 worker thread 的 cgroup（root），不是原始應用的 cgroup。所以 v1 只能限制 direct IO 與 read。 v2 解決方式：透過 memory + io controller 協同實現 cgroup writeback。
  1. memory controller 知道每個 page 屬於哪個 cgroup（per-page ownership）
  2. writeback worker 看 dirty page 時，把 page 的 cgroup 資訊帶到 bio
  3. block layer 看到 bio 的真實 cgroup，套對應的 io 限制
前提條件：v2 unified hierarchy（memory 與 io 在同 cgroup）+ filesystem 支援 cgroup writeback（ext4 / btrfs / xfs / f2fs / ext2）。
Q2.io.max、io.weight、io.cost、io.latency 各自解決什麼問題？什麼場景該選哪個？
  • io.max：硬性 BPS / IOPS 上限。場景：multi-tenant 硬性 SLA，「絕不超過 X MB/s」。
  • io.weight：比例分配（基於 blk-iocost）。場景：相對優先級控制，「service A 拿到 4 倍 service B 的 IO」。需要先 enable blk-iocost。
  • io.cost：blk-iocost 進階控制——管理員提供 latency target 與 vrate 範圍（min/max%），kernel 動態調整。場景：精細的 IO QoS，雲端 multi-tenant。
  • io.latency：基於 latency target 的 workload 保護。當受保護 cgroup 的 latency 超過 target，throttle 同層其他 cgroup。場景：latency-critical 服務（DB、cache）保護。
選擇邏輯：
  • 需要絕對防爆 → io.max（避免某 cgroup 把磁碟吃光）
  • 正常優先級分配 → io.weight
  • 保護 latency-critical workload → io.latency（首選）
  • 雲端精細 QoS → io.cost
實務上常組合用：weight 處理日常 + latency 保護關鍵 workload + max 當最後防線。
Q3.cgroup writeback 需要哪三個條件？少了任何一個會發生什麼？
三個條件：
  1. v2 啟用 memory + io controller：兩個都在同一 cgroup hierarchy 下
  2. filesystem 支援 cgroup writeback：ext2、ext4、btrfs、xfs、f2fs
  3. 正確掛載：filesystem 以支援 cgroup writeback 的方式 mount（一般預設就是）
缺一的後果：
  • 缺 memory controller → 沒有 page ownership 資訊，buffered write 歸 root
  • 缺 io controller → 沒有 io 限制可套，限制完全失效
  • filesystem 不支援（NFS、tmpfs、CIFS、overlayfs）→ 整個 cgroup writeback 機制 fall back，buffered write 歸 root
檢查方法：寫個 dd buffered write 測試，看是否真的被限速。不要相信介面值，要實測。
Q4.Memory Ownership 與 IO Ownership 的差別是什麼？多 cgroup 共用一個檔案會發生什麼？
關鍵差別：
  • Memory Ownership：per-page 追蹤。每個 page 由「第一個 touch」的 cgroup 擁有，page release 才釋放。多 cgroup 寫同檔案，不同 page 可能歸不同 cgroup。
  • IO Ownership：per-inode 追蹤。整個 inode 只能歸一個 cgroup。kernel 偵測「主要 dirtier」變動會切換 inode 歸屬。
多 cgroup 共用檔案的問題（foreign pages）：cgroup A 寫了大部分內容，inode 歸 A；cgroup B 也開始寫——B 寫的 page 在 memory 視角歸 B（per-page），但 writeback 仍按 inode 歸 A。這些「歸屬不一致」的 page 是 foreign pages。 kernel 的處理：持續追蹤 foreign pages 比例，當 B 變成主要 dirtier，inode 歸屬切換到 B。但這個切換不是即時的，且多 cgroup 同時寫同檔案 kernel 文件明確說「not supported well」——一部分 IO 必然歸錯，無法精準控制。 實務避免方法：架構上不讓多 cgroup 寫同檔案。共享 log 用 syslog/journald 集中寫入。
Q5.io.latency 為什麼只在 peer level 影響？怎麼設計 cgroup hierarchy 才能讓 latency target 真正保護整個 workload？
peer-level 設計：io.latency 只影響「同一 parent 下的 sibling cgroup」之間的 throttle。對 child 的 child（隔層）不直接影響——它們有自己的 peer group。 原因：cgroup hierarchy 的階層性決定資源是「逐層分配」。在 A/B 層級設 latency target 控制 A 與 B 的競爭關係；A 內部的 child 是 A 自己的事。 正確設計：
  1. 頂層按 workload 類型分：workload.slice（latency-critical）、system.slice（system services）、batch.slice（low-priority）。
  2. 在頂層設 io.latency：對 workload.slice 設 50ms、system.slice 設 100ms。它們互相影響（peer），達到主要保護效果。
  3. workload.slice 內可再細分但不需設 io.latency：頂層保護已經夠用，內部用 io.weight 做相對優先即可。
fbtax2 案例：Meta 對 workload.slice 設 50ms、system.slice 設 75ms，效果非常好。設更深層的 io.latency 會讓設定變複雜但收益小。
Q6.blk-iocost 是什麼？跟 io.weight 是什麼關係？
blk-iocost：是 v2 io controller 的底層 backend，全稱 block I/O cost。它用「每個 IO 的成本（virtual time）」計量裝置使用情況——比 v1 的 CFQ-based blkio 精細許多，特別適合 NVMe 等高速裝置。 跟 io.weight 的關係：
  • io.weight 是 user 寫入的數值（1–10000）
  • blk-iocost 是實作機制，把 weight 換算成 vrate 配額並 enforce
  • 沒啟用 blk-iocost 時，io.weight 介面存在但沒有效果——這是常見踩雷點
啟用方式：在 root cgroup 寫入 io.cost.qos：echo "8:0 enable=1 ctrl=auto" > /sys/fs/cgroup/io.cost.qos。 監控工具：kernel source 提供 tools/cgroup/iocost_monitor.py（drgn-based），可看每個 cgroup 的 weight、hweight%、vrate、debt 等內部狀態。
Q7.io.prio.class 的 4 個值各自意思？跟 process-level ionice 怎麼互動？
四個值：
  • no-change（預設）：cgroup 不干預，process 自己用 ionice 設的優先級生效。
  • promote-to-rt：所有 IO 強制提升到 RT（realtime）class，覆蓋 process 自己的設定。
  • restrict-to-be：限制在 best-effort class——即使 process 設了 RT 也降級。
  • idle：所有 IO 變 idle class，CFQ 排程下只在無人用時跑。
跟 ionice 的互動：cgroup 設定覆蓋（override）process 自己的 ionice。
  • process 設 RT，cgroup 設 restrict-to-be → 最終是 best-effort（cgroup 贏）
  • process 設 best-effort，cgroup 設 promote-to-rt → 最終是 RT（cgroup 贏）
  • process 設 RT，cgroup 設 no-change → 最終是 RT（process 自己決定）
建議：除非有明確理由要覆蓋，io.prio.class 用 no-change（預設）讓應用自己決定。
Q8.實作思考題：你管理一台 NVMe 機器跑：（A）關鍵 PostgreSQL（latency-sensitive）、（B）Redis cache（讀多寫少）、（C）每日備份 batch（寫密集）。設計完整 IO 配置策略。
方案： 
[root cgroup setup]
# 啟用 blk-iocost（一次性 setup）
echo "259:0 enable=1 ctrl=auto" > /sys/fs/cgroup/io.cost.qos

[A: postgresql.service] - 關鍵 DB
IOWeight=400              # 較高權重
IODeviceLatencyTargetSec=/dev/nvme0n1 5ms  # NVMe 物理 ~100μs，5ms 寬鬆
# 不設 io.max（不希望被限制）
# 不設 io.prio.class（讓 PG 自己控制）

[B: redis.service] - cache
IOWeight=200              # 中等權重
IODeviceLatencyTargetSec=/dev/nvme0n1 5ms  # 也保護 latency
# Redis 主要走記憶體，IO 不重，這樣足夠

[C: backup.service] - batch
IOWeight=50               # 低權重
IODeviceLatencyTargetSec=/dev/nvme0n1 50ms # 可容忍較高 latency
IOWriteBandwidthMax=/dev/nvme0n1 200M     # 寫入 bandwidth 上限
IOReadIOPSMax=/dev/nvme0n1 5000           # 讀 IOPS 上限
ExecStartPost=/bin/bash -c 'echo idle > $cgroup/io.prio.class'
# idle class 讓 batch 真的讓位
設計理由：
  1. PG 與 Redis 用 io.latency 保護：5ms target 對 NVMe 寬鬆，但能擋住 batch 的 IO 干擾。
  2. 權重分配：400:200:50 比例反映業務優先級。
  3. backup 加雙重限制：io.max 防爆 + io.prio.class=idle 完全讓位。即使 PG 沒競爭，backup 不會把 NVMe 跑飽。
  4. 不設 io.max 給 PG：DB 偶爾的 burst（checkpoint、WAL flush）需要瞬時高 IO，硬限會適得其反。
監控配套：
  • 對 PG 監控 io.pressure some avg10 < 5% 與 io.stat avg_lat
  • 對 backup 容忍 throttle（這是預期行為）
  • 整體機器監控 /proc/pressure/io full avg10 < 1%
