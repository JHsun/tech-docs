Part II · 操作篇 · Chapter 05

Delegation 與權限模型

cgroup v2 把 v1 「含糊不清的委派」徹底重做，讓非 root 使用者也能安全地管理自己的子樹。 本章解開「為什麼 systemd 是 root cgroup 的 single writer」的謎題， 並深入 nsdelegate、cgroup namespace、containment——這是 rootless container 與 Kubernetes Pod 安全模型的根基。

01 · Highlights

本章重點

  1. 兩種委派方式：（1）權限委派——透過 chown 把目錄與部分檔案交給非 root 使用者；（2）namespace 委派——配合 nsdelegate mount option，讓 cgroup namespace 自動成為委派邊界。
  2. delegation containment：委派出去的子樹是「真正的封閉沙箱」——委派者改不了 parent 的設定，也無法把外部 process 拉進來、把內部 process 推出去。
  3. 可委派的檔案清單：/sys/kernel/cgroup/delegate 列出哪些檔案可以交給委派者。typical 包含 cgroup.procs、cgroup.threads、cgroup.subtree_control，但不包含 resource control interface files（如 memory.max）。
  4. systemd Delegate=yes：應用想自己管理子樹，必須在 systemd unit 加 Delegate=yes 申請。systemd 不會 touch 委派出去的子樹，從而符合 single-writer rule。
  5. rootless container 的鑰匙：v1 沒有 rootless cgroup 概念；v2 + nsdelegate + systemd user@.service Delegate 才讓 podman、rootless docker 真正可用。

2026 年的時點，delegation 不再是進階主題，而是日常基礎。 任何在 Linux 上跑容器、開發 systemd service、或想理解 Kubernetes Pod 為什麼這樣設計的人， 都需要懂本章內容。下面從最基本的權限委派開始講起。

03 · Core Concepts

核心內容

3.1 為什麼 cgroup 要委派

想像一個場景：你是系統管理員，配給某個使用者 alice 一塊 cgroup（限定她最多用 8GB 記憶體、4 核 CPU）。 在這塊配額內，alice 想自己決定資源怎麼分配——例如她跑了 3 個專案，希望「專案 A 拿 4GB、B 拿 2GB、C 拿 2GB」。

最簡單的做法：把這塊 cgroup 交給 alice 自己管。 alice 在裡面建子 cgroup、移動 process、設子配額，但她絕對不能：

  • 把 8GB 上限改大（突破你給的配額）
  • 把外面的 process 抓進她的 cgroup（影響別人）
  • 把自己的 process 推到外面（逃逸到 host）

這就是委派（delegation）——授權使用者管理子樹，但用結構性規則保證她跑不出沙箱。 v1 對這個需求的支援很差，導致 rootless container 在 v1 時代基本上做不到。v2 從零設計了完整的 delegation model。

3.2 v1 的委派為什麼失敗

kernel 文件 §R-3 "Issues with v1 and Rationales for v2" 直接點出 v1 委派的根本問題：

cgroup v1 had an ambiguously defined delegation model which got abused in combination with thread granularity. cgroups were delegated to individual applications so that they can create and manage their own sub-hierarchies and control resource distributions along them. This effectively raised cgroup to the status of a syscall-like API exposed to lay programs.

翻譯：v1 的委派把 cgroup 提升到「syscall 等級的 API」，但 cgroup filesystem 的介面根本不適合給應用直接用—— 要修改自己的設定，process 必須讀 /proc/self/cgroup 找到路徑、組合檔案路徑、open + read/write，整個流程笨重且容易出 race condition。 加上 v1 的 thread granularity，process 還會用各種 hack 手法管理自己的 thread，造成介面被嚴重濫用。

v2 的回答是：cgroup 是給外部管理者用的工具，不是給 app 自我管理的 syscall。 要讓非 root 安全地管子樹，需要重新設計一套含結構性保證的委派機制。

3.3 委派方式 1：權限委派

最直接的做法：把目錄與部分檔案的擁有權交給非 root 使用者。kernel 文件 §2.5.1 "Model of Delegation" 規定：

  • 給予寫權限的檔案：cgroup directory 本身、cgroup.procs、cgroup.threads、cgroup.subtree_control。
  • 不給寫權限的檔案：memory.max、cpu.max 等所有 controller interface files。

為什麼這樣切？因為controller interface files 是給 parent 用的，控制 parent 如何分配資源給 children（Ch.2 學過）。 把 memory.max 給 alice，等於讓 alice 可以改動 parent 給她的配額——這當然不行。 

# 系統管理員執行：建立委派的子樹
sudo mkdir /sys/fs/cgroup/alice-workspace

# 設限：alice 最多用 8GB 記憶體（注意：這是在 parent 設的）
echo "8G" | sudo tee /sys/fs/cgroup/alice-workspace/memory.max

# 啟用 controllers 給 alice 的 child 用
echo "+memory +cpu +pids" | sudo tee /sys/fs/cgroup/alice-workspace/cgroup.subtree_control

# 把目錄 chown 給 alice
sudo chown -R alice:alice /sys/fs/cgroup/alice-workspace

# 但要把 controller interface files 改回 root 擁有
# （否則 alice 可以改自己的上限）
sudo chown root:root /sys/fs/cgroup/alice-workspace/{memory.max,cpu.max}

# 從現在起，alice 可以：
# - mkdir 在 alice-workspace/ 下建子 cgroup
# - 寫 cgroup.procs 移動自己的 process
# - 寫 cgroup.subtree_control 對她的 child 啟用 controllers
# - 設子 cgroup 的 memory.max（但加總不能超過她拿到的 8GB）

3.4 委派方式 2：namespace 委派（nsdelegate）

權限委派要管理目錄擁有權與檔案 chmod，比較繁瑣。v2 提供更乾淨的方式：cgroup namespace。

cgroup namespace（Linux 4.6+）讓 process「看不到自己 cgroup 上層」——對它而言，自己所在的 cgroup 就是 root，看不到 host 的全貌。 這天然形成隔離，但單獨 cgroup namespace 還不夠安全——namespace 內的 process 仍可以寫 namespace root 的 controller files、影響到 namespace 外的設定。

kernel 4.13 加入 nsdelegate mount option 補上這個漏洞。啟用後：

  • cgroup namespace 自動成為委派邊界
  • namespace 內的 process 寫 namespace root 的 controller files 會收到 EPERM
  • 跨 namespace 邊界遷移 process 會收到 ENOENT（看不到目的地）

這個設計的妙處：不需要手動 chown，安全屬性由 namespace 結構自動保證。容器運行時建立 cgroup namespace，子 cgroup 自動變成委派子樹。

3.5 Delegation Containment（封閉性保證）

這是本章最重要的概念——委派出去的子樹是真正封閉的沙箱，委派者跑不出去、外部抓不進來。

kernel 文件 §2.5.2 "Delegation Containment" 給出兩種封閉方式：

權限委派的 containment

對非 root 使用者寫 PID 進 cgroup.procs 移動 process，kernel 強制要求兩個條件同時滿足：

  1. writer 對目標 cgroup 的 cgroup.procs 有寫權限
  2. writer 對來源與目標的共同祖先的 cgroup.procs 也有寫權限

舉例：alice 拿到 C0 與 C1 兩個 cgroup 的權限，分別建了 C0/sub 與 C1/sub。 她想把 C1/sub 內的 process 移到 C0/sub。kernel 會檢查：alice 對共同祖先（C0/C1 的 parent）有寫權限嗎？ 沒有——委派只給到 C0、C1 自己，parent 不在 alice 控制範圍，所以遷移失敗（EACCES）。

這條規則保證 alice 不能藉著「移到我管的另一棵樹」這種手法把 process 帶出她原本的 sandbox 邊界。

namespace 委派的 containment

對 namespace 委派：遷移時要求來源與目標都在 namespace 視野內。看不到的 cgroup 路徑會回 ENOENT。 因為 cgroup namespace 內的 process 看不到 namespace 外的 path，這條規則自動阻止跨 namespace 遷移。

3.6 /sys/kernel/cgroup/delegate 檔案

這個唯讀檔案列出哪些 core interface files 可以「合理地」委派給非 root： 

$ cat /sys/kernel/cgroup/delegate
cgroup.procs
cgroup.threads
cgroup.subtree_control
memory.oom.group

這個清單由 kernel 維護——新增 core interface files 時，kernel 開發者會評估是否該加入此清單。 不在清單上的檔案（特別是 controller interface files）絕不該交給委派者，會破壞封閉性。

nsdelegate 模式下，kernel 讀這個檔案來決定哪些寫入要拒絕——對非 listed file 從 namespace root 寫入會直接 EPERM。

3.7 systemd Delegate=yes 的真實意義

Ch.2 提過 systemd 是 root cgroup 的 single writer。但應用怎麼合作？答案是 systemd 提供 Delegate= 屬性。 

# 在 service unit 裡加 Delegate=yes
[Service]
Delegate=yes
ExecStart=/usr/bin/myapp

# 或限定哪些 controllers
[Service]
Delegate=cpu memory pids

這背後發生什麼事？

  • systemd 為這個 service 建立 cgroup 後，把該 cgroup 標記為「delegated」，不再 touch 子樹內部。
  • systemd 啟用指定的 controllers 對這個 cgroup 的 child 生效。
  • service 內的 process（容器 runtime、自管理的應用）可以自由建立子 cgroup、設子配額。

這就是 single-writer rule 的具體實踐——透過明確的「申請與授權」機制讓多個 writer 共存：systemd 寫委派之外、應用寫委派之內，互不干擾。

3.8 rootless container 為什麼必須 v2

v1 沒有 rootless cgroup 的概念——所有 cgroup 操作都要 root 權限。Podman、rootless docker 在 v1 時代根本無法限制資源（只能跑，限不了）。

v2 + nsdelegate + systemd user@.service 的 Delegate 配置，組合成 rootless container 的完整方案：

3.9 委派的限制與資源逃逸

最後一個關鍵概念：委派子樹內的所有設定，都只能在 parent 給的範圍內生效。 kernel 文件原話：「Once delegated, the user can build sub-hierarchy under the directory, organize processes inside it as it sees fit and further distribute the resources it received from the parent. The limits and other settings of all resource controllers are hierarchical and regardless of what happens in the delegated sub-hierarchy, nothing can escape the resource restrictions imposed by the parent.」

翻譯：alice 可以在自己的 8GB 配額內愛怎麼分就怎麼分（child 1 配 4GB，child 2 配 4GB）， 但所有 child 的設定不論怎麼設，加總都不會突破 parent 給的 8GB。這是 hierarchical 設計的天然保證。

04 · Hands-on

實戰範例：完整委派流程

Lab 1 · 啟用 user@.service 的 cgroup 委派

讓非 root 使用者可以管理自己的 cgroup（rootless container 的前置步驟）：

# 1. 確認 cgroup v2
stat -fc %T /sys/fs/cgroup/

# 2. 確認 nsdelegate 啟用
mount | grep cgroup2 | grep -o nsdelegate

# 3. 看當前使用者拿到哪些 controllers
USER_CG=/sys/fs/cgroup/user.slice/user-$(id -u).slice/user@$(id -u).service
cat $USER_CG/cgroup.controllers
# 多半預設只有: memory pids

# 4. 加 drop-in 委派更多 controllers
sudo mkdir -p /etc/systemd/system/[email protected]
cat < $USER_CG/myapp/cgroup.procs
echo "100M" > $USER_CG/myapp/memory.max  # 注意：這需要 user@.service 啟用 memory
Lab 2 · 體驗 delegation containment

驗證委派出去的子樹真的封閉：

# 1. 系統管理員建立兩個獨立委派子樹
sudo mkdir /sys/fs/cgroup/sandbox-A
sudo mkdir /sys/fs/cgroup/sandbox-B

sudo chown -R $(id -u):$(id -u) /sys/fs/cgroup/sandbox-A
sudo chown -R $(id -u):$(id -u) /sys/fs/cgroup/sandbox-B

# 2. 在 sandbox-A 內建立子 cgroup 並放一個 process
mkdir /sys/fs/cgroup/sandbox-A/leaf
sleep 999 &
PID=$!
echo $PID > /sys/fs/cgroup/sandbox-A/leaf/cgroup.procs

# 3. 嘗試把 sandbox-A 的 process 搬到 sandbox-B
echo $PID > /sys/fs/cgroup/sandbox-B/cgroup.procs
# 預期失敗：Permission denied
# 原因：你對共同祖先 / 沒有寫權限

# 4. 試試在 sandbox-A 子樹內移動（被允許）
mkdir /sys/fs/cgroup/sandbox-A/another
echo $PID > /sys/fs/cgroup/sandbox-A/another/cgroup.procs
# 成功：因為共同祖先 sandbox-A 你有權限

# 清理
kill $PID
sudo rmdir /sys/fs/cgroup/sandbox-A/* /sys/fs/cgroup/sandbox-B
sudo rmdir /sys/fs/cgroup/sandbox-A

觀察點：第 3 步失敗證實了「不能跨委派邊界遷移」。沒有這條規則，整個委派模型就破功。

Lab 3 · 用 nsdelegate 體驗 namespace 委派
# 1. 確認系統有啟用 nsdelegate
mount | grep cgroup2 | grep nsdelegate

# 2. 用 unshare 建立一個有自己 cgroup namespace 的 shell
sudo systemd-run --slice=user.slice --unit=ns-test \
    --pty \
    -p Delegate=yes \
    bash -c 'unshare -C bash'

# 3. 在 namespace 內看 cgroup
cat /proc/self/cgroup
# 應該看到 0::/  (namespace 內看到自己是 root)

# 4. 嘗試寫 namespace root 的 controller files
echo "1G" > /sys/fs/cgroup/memory.max
# 預期：Permission denied  (nsdelegate 阻止)

# 5. 但可以建子 cgroup 並設它
mkdir /sys/fs/cgroup/test
echo "100M" > /sys/fs/cgroup/test/memory.max
# 成功

# 6. 退出後清理
exit
sudo systemctl stop ns-test
Lab 4 · rootless Podman 的完整資源限制
# 前提：完成 Lab 1 的 user@.service delegate 設定

# 1. 跑 rootless podman 容器，限 memory + cpu
podman run --rm --memory=256m --cpus=1 alpine \
    sh -c 'cat /sys/fs/cgroup/memory.max; cat /sys/fs/cgroup/cpu.max'

# 預期輸出：
#   268435456
#   100000 100000

# 2. 看容器 cgroup 對應到 host 的位置
podman run -d --name=test --memory=256m alpine sleep 999
podman inspect test --format '{{.State.CgroupPath}}'
# 範例: /user.slice/user-1000.slice/user@1000.service/user.slice/libpod-XXX.scope

# 3. 在 host 端確認資源限制真的有套用
HOST_CG=$(podman inspect test --format '{{.State.CgroupPath}}')
cat /sys/fs/cgroup$HOST_CG/memory.max

# 清理
podman stop test
podman rm test

05 · Pitfalls

常見陷阱

陷阱 1 · 把 controller interface files 也 chown 給使用者

最常見的設定錯誤：sudo chown -R alice /sys/fs/cgroup/alice-workspace，把所有檔案都交出去。 這讓 alice 可以改 memory.max 突破自己的配額，整個沙箱形同虛設。 正確做法：用 -R 後再把 controller interface files chown 回 root（如本章 §3.3 範例）。 或更乾淨：用 systemd Delegate=yes 讓 systemd 自己處理。

陷阱 2 · 沒設 nsdelegate 就用 cgroup namespace

Linux 4.6 加入 cgroup namespace，但 4.13 才加入 nsdelegate。沒有 nsdelegate 的 cgroup namespace 不安全—— namespace 內的 process 仍可以寫 namespace root 的 controller files，破壞 parent 的配額。 現代 distro 預設啟用 nsdelegate，但跑舊 kernel 或自己 mount 時要記得加。

陷阱 3 · user@.service 沒設 Delegate，rootless container 沒有資源限制

在 RHEL/CentOS/Fedora 預設配置上，user@.service 通常只 delegate memory pids。 rootless podman 看似能跑，但 --cpus、--memory（部分情況）、--device-read-bps 全部默默無效——容器跑著，但限制完全沒套用。 檢查方式：跑容器後到 host 看 cpu.max 是否真的有設。 修法：本章 Lab 1 的 drop-in 配置。

陷阱 4 · 委派出去後手動干涉子樹

典型錯誤：管理員委派了 alice-workspace 給 alice，後來想說「幫她調整 alice-workspace/foo 的設定」直接動手改。 這違反 single-writer rule——alice 可能已經有自己的管理邏輯（例如她的容器 runtime），雙方會打架。 正確做法：委派出去後就完全不碰，要調整就修改 parent 的整體配額。

陷阱 5 · cpuset 委派需要新版 systemd

kernel 5.4+ 才支援 cpuset 在 v2 委派、systemd 244+ 才能正確 delegate cpuset。 在舊系統上即使你寫 Delegate=cpu cpuset io memory pids，cpuset 也不會生效，沒有錯誤訊息。 檢查方式：登入後看自己的 cgroup.controllers 是否包含 cpuset。

陷阱 6 · 期望從 namespace 內看到 host cgroup tree

在容器內跑 ls /sys/fs/cgroup，看到的是 namespace 視角下「容器自己的 cgroup 子樹」，不是 host 的全貌。 這是設計，不是 bug——namespace 隔離的目的就是讓容器看不到 host。 要在 host 端看真實 cgroup path，從 host 的 podman inspect、docker inspect 或 /proc/$PID/cgroup 取得。

練習 4 · 寫一個 systemd Delegate audit script（15 分鐘）

掃描所有 service unit，找出有 Delegate=yes 的：

#!/bin/bash
# delegate-audit.sh

echo "Services with Delegate enabled:"
echo "================================"

for unit in $(systemctl list-units --type=service --no-legend --plain | awk '{print $1}'); do
    delegate=$(systemctl show -p Delegate --value "$unit")
    controllers=$(systemctl show -p DelegateControllers --value "$unit")
    
    if [ "$delegate" = "yes" ] || [ -n "$controllers" ]; then
        echo
        echo "Unit: $unit"
        echo "  Delegate: $delegate"
        echo "  Controllers: $controllers"
    fi
done

07 · Self-check

自我檢查題

Q1.cgroup v2 提供哪兩種委派方式？各自的核心機制是什麼？
方式 1：權限委派。透過 chown 把 cgroup directory 與部分 core interface files（cgroup.procs、cgroup.threads、cgroup.subtree_control）的擁有權交給非 root 使用者。controller interface files（memory.max、cpu.max）保留 root 擁有，避免使用者突破配額。 方式 2：namespace 委派。配合 nsdelegate mount option，讓 cgroup namespace 自動成為委派邊界。namespace 內的 process 寫 namespace root 的 controller files 收到 EPERM；跨 namespace 邊界遷移收到 ENOENT。容器運行時建立 namespace，子樹自動委派，不需要手動 chown。 兩者結果等價，可以單獨用也可以組合用。namespace 方式更乾淨，是現代容器的主流。
Q2.為什麼 controller interface files（如 memory.max）絕對不能交給委派者？
因為controller interface files 是給 parent 用的，控制 parent 如何分配資源給 children（Ch.2 學過）。 如果把 alice-workspace/memory.max 給 alice，等於讓她可以改自己的上限——本來限她 8GB，她改成 100GB，整個沙箱破功。 正確的設計：memory.max 屬於 parent (root)，由 parent 決定給 alice 多少；alice 在收到的配額內用 alice-workspace/sub/memory.max 設子配額（而 alice-workspace/sub/memory.max 屬於 alice，因為她是 sub 的 parent）。 這個「controllers 屬於 parent」的設計就是讓 hierarchical 資源配額能可靠運作的關鍵。
Q3.delegation containment 是什麼？對權限委派與 namespace 委派如何分別實現？
delegation containment：委派出去的子樹是真正封閉的沙箱，委派者跑不出去、外部抓不進來。 權限委派的 containment：寫 cgroup.procs 移動 process 時，kernel 要求 writer 對「來源與目標的共同祖先」也有寫權限。委派只給到自己的子樹，共同祖先在外面，所以跨子樹遷移失敗（EACCES）。 namespace 委派的 containment：遷移時要求來源與目標都在 namespace 視野內。namespace 內的 process 看不到外部 path，自動拒絕跨 namespace 遷移（ENOENT）。 共同效果：alice 不能藉著「移到我管的另一個地方」把 process 帶出原本的 sandbox 邊界。封閉性由結構規則自動保證，不依賴使用者的好意。
Q4./sys/kernel/cgroup/delegate 是做什麼的？為什麼 memory.max 不在這個清單上？
用途：唯讀檔案，列出哪些 core interface files 可以「合理地」委派給非 root。 nsdelegate 模式下，kernel 用此清單決定哪些寫入要拒絕——對非 listed file 從 namespace root 寫入會 EPERM。 典型內容：cgroup.procs、cgroup.threads、cgroup.subtree_control、memory.oom.group。 為什麼 memory.max 不在：因為它是 controller interface file，屬於 parent，不該交給委派者。委派者改它就是破壞 containment。 kernel 開發者新增 core interface files 時會評估是否該加入此清單；controller interface files 絕不加入。
Q5.systemd Delegate=yes 在背後做了哪三件事？
  1. 標記委派：systemd 為這個 service 建立 cgroup 後，把它標記為「delegated」，systemd 自己不再 touch 子樹內部。從這一刻起，service 內的 process 是這個子樹的 single writer。
  2. 啟用 controllers：根據 Delegate=cpu memory io 之類的設定，啟用對應的 controllers 對這個 cgroup 的 child 生效。
  3. 授予寫權限：把可委派的 core interface files chown 給 service 執行的使用者（如果 User= 有設）；對 root 跑的 service 不需要 chown。
這三件事讓「單一系統有多個 writer」變成可能：systemd 寫委派之外（包含這個 service 的 cgroup 本身），service 寫委派之內。各管各的，不互相干擾，符合 single-writer rule。
Q6.為什麼 rootless container 必須要 cgroup v2？v1 的問題是什麼？
v1 的根本問題：v1 沒有「rootless cgroup」的概念，所有 cgroup 操作都需要 root 權限。即使非 root 透過 user namespace 在容器內成為「fake root」，host 上他還是 unprivileged user，無法寫 cgroup 介面。 所以 v1 時代的 rootless docker / podman 跑得起來，但完全限不了資源——--memory、--cpus 全部默默無效。 v2 的解決方式：v2 設計了完整的非 root 委派模型。透過 nsdelegate + systemd user@.service Delegate 配置，非 root 使用者可以在自己的子樹內安全地建立子 cgroup、設子配額。 所以 podman 4+、rootless docker 全面要求 v2 才能正確工作。2026 年新建的 container 系統如果還在 v1，請優先升級。
Q7.典型 distro 上 user@.service 預設 delegate 哪些 controllers？要 delegate cpuset 需要什麼條件？
預設委派：多數 distro 預設只 delegate memory pids。這是個保守的選擇，因為 cpu / cpuset / io 在早期 v2 實作不夠成熟。 要 delegate cpuset 的條件：
  • kernel 5.4+ （cpuset 在 v2 委派的支援）
  • systemd 244+ （正確 delegate cpuset 的能力）
在舊系統上即使你寫 Delegate=cpu cpuset io memory pids，cpuset 也不會生效，沒有錯誤訊息。 檢查方式：登入後看 /sys/fs/cgroup/user.slice/user-$UID.slice/user@$UID.service/cgroup.controllers，列出來的就是你實際拿到的。 修法：drop-in /etc/systemd/system/[email protected]/delegate.conf 加上 [Service] Delegate=cpu cpuset io memory pids，重啟 user 服務或重新登入生效。
Q8.實作場景題：你管理一個 multi-tenant 機器，要讓三個團隊（A、B、C）各自管自己的 workload，但彼此不能干涉、且加總不超過 host 容量。請設計 cgroup 結構與委派策略。
結構設計： 
/sys/fs/cgroup/
├── system.slice/        # 系統服務（root 管）
├── tenants.slice/       # 自建 slice，root 管理 limits
│   ├── teamA/           # 委派給 A，內部她們自己管
│   ├── teamB/           # 委派給 B
│   └── teamC/           # 委派給 C
└── user.slice/          # 一般使用者
委派策略：
  1. tenants.slice 由 root 管理，設定全域 limits（如 80% memory、90% CPU），啟用 +cpu +memory +io 對 child。
  2. 三個 team slice 由 root 設 limits：teamA/memory.max=4G、teamA/cpu.max=4 cores 等。
  3. chown 權限：把 teamA 目錄與 cgroup.procs / cgroup.subtree_control chown 給 team A 的 group；保留 controller files 為 root。
  4. 啟用 nsdelegate：確保 namespace-based isolation 也能配合，給 team A 跑容器用。
  5. 用 systemd unit 包裝：建立 teamA.service 帶 Delegate=yes，避免手動 chown 維護成本。
結果：三個團隊各自在 4G/4-core 配額內愛怎麼分就怎麼分，但絕對不能突破自己的配額，也不能干涉其他團隊。host 整體被 tenants.slice 的全域 limits 保護。
