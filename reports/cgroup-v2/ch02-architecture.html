架構與核心概念

cgroup v2 是一棵樹，但這棵樹不是隨便長的——它有嚴格的拓撲規則。 本章拆解 core 與 controllers 的職責切分、domain 與 threaded 的兩種 cgroup 型態，並建立貫穿全書的五個鐵律。

01 · Highlights

本章重點

  1. core vs controllers：cgroup core 負責「樹的結構」與「process 歸屬」；controllers 負責「資源分配」。core 的介面檔案以 cgroup. 開頭，controllers 的以 {name}. 開頭。
  2. domain vs threaded cgroup：domain 是預設型態，以 process 為粒度；threaded 允許同一 process 的 thread 散佈在子樹中，但只有 cpu / cpuset / perf_event / pids 四個 controllers 支援。
  3. 五大鐵律：unified hierarchy、top-down constraint、no internal process constraint、single-writer rule、organize once and control。
  4. 關鍵 interface files：cgroup.controllers、cgroup.subtree_control、cgroup.procs、cgroup.type、cgroup.events，這五個是後續所有操作的基礎。
  5. systemd 的角色：在 systemd-managed 系統上，systemd 是 root cgroup 的唯一 writer，這正是 single-writer rule 的真實體現。

最新動態（2025–2026）

2025 systemd CGROUP_DELEGATION 文件持續更新，明確強調 single-writer rule 是 cgroup v2 設計的兩大支柱之一（另一個是 no-processes-in-inner-nodes），並批評目前許多容器管理工具實作不佳。

從 2025 年起，v2 的設計拓撲規則已經是業界共識。不論你用 systemd、Kubernetes、LXC 還是寫自己的 container runtime，都會反覆遇到本章介紹的五大鐵律。 理解這些規則的「為什麼」遠比記住「是什麼」重要——後續所有章節在遇到看似奇怪的限制時，都可以回頭從本章找到答案。

03 · Core Concepts

核心內容

3.1 cgroup tree 結構

cgroup v2 是一棵單一階層樹，掛在 /sys/fs/cgroup/。 樹上的每個目錄就是一個 cgroup，目錄之間的父子關係就是 cgroup 之間的階層關係。 Root cgroup 永遠存在，路徑是 /sys/fs/cgroup/ 本身；子 cgroup 透過 mkdir 建立、rmdir 刪除。

每個 cgroup 目錄裡有兩類檔案：

  • core 介面檔案：以 cgroup. 開頭，例如 cgroup.procs、cgroup.type、cgroup.controllers。這些檔案在每個 cgroup 都存在，由 cgroup core 提供。
  • controller 介面檔案：以 controller 名字開頭，例如 cpu.weight、memory.max、io.weight。這些檔案只有當對應 controller 在 parent 啟用時才出現。

這個區分非常重要：core 介面是結構性的（樹怎麼長、process 屬於哪個 cgroup）；controller 介面是資源性的（這個 cgroup 能用多少資源）。 本書 Part I–II 主要講 core，Part III–IV 才深入 controllers。

3.3 五大鐵律

cgroup v2 有五個核心規則，是設計哲學的具體體現。後續所有章節都建立在這五條規則之上。

鐵律 1：Unified Hierarchy（單一階層）

全系統只有一棵 cgroup tree。每個 process 在這棵樹上只有一個位置，由 /proc/$PID/cgroup 的單行 0::/path 完整描述。 這條規則我們在 Ch.1 已經詳細討論過——它解決了 v1 多階層的所有歷史問題。

鐵律 2：Top-down Constraint（自頂向下約束）

一個 cgroup只能對它的子 cgroup 啟用 parent 已經啟用的 controllers。 換句話說，子 cgroup 永遠不能擁有 parent 沒給它的能力。kernel 文件原話：

Resources are distributed top-down and a cgroup can further distribute a resource only if the resource has been distributed to it from the parent.

實作上，這透過兩個檔案配合：cgroup.controllers（這個 cgroup 「可用」的 controllers）和 cgroup.subtree_control（這個 cgroup 「對子 cgroup 開放」的 controllers）。 你只能把 cgroup.controllers 列出的東西寫進 cgroup.subtree_control。

鐵律 3：No Internal Process Constraint（非 leaf 不准放 process）

一個非 root cgroup一旦在 cgroup.subtree_control 啟用任何 domain controller，就不能直接持有 process。 process 必須放在這棵樹的葉子節點上。這條規則是為了避免「parent 自己的 process」與「parent 的子 cgroup 的 process」之間的資源競爭歧義（這正是 v1 的痛點之一）。

man-page 對這條規則有更精確的描述：規則並不是「不能同時有 process 和 child cgroup」， 而是「不能同時有 process 和啟用 controllers 的 child cgroup」（即 cgroup.subtree_control 非空）。 這個微妙差異留給 root cgroup 一個豁免權——root 可以同時持有 kernel thread 和 child cgroups。

實務上，建議建立一個 leaf/ 子目錄，把所有原本要放在這個 cgroup 的 process 移到 leaf/，然後在 parent 啟用 controllers。 這是 kernel 文件本身推薦的做法。

鐵律 4：Single-writer Rule（單一寫入者）

每個 cgroup只應該由一個 process 管理。systemd 文件直白地說：

Each cgroup only has a single writer, i.e. a single process managing it. It's OK if different cgroups have different processes managing them.

這條規則 kernel 不會強制（root 想做什麼都可以），但違反它會帶來無止盡的痛苦——多個工具會互相覆蓋對方的設定，行為變得不可預期。 在 systemd-managed 系統上，systemd 是 root cgroup 的唯一 writer， 其他想管 cgroup 的軟體（容器 runtime、應用本身）必須透過 systemd 的 Delegate= 機制申請一塊「可以自己管」的子樹（這是 Ch.5 的主題）。

鐵律 5：Organize Once and Control（一次組織、長期控制）

把一個 process 從一個 cgroup 移到另一個是昂貴的操作——除了 syscall 開銷，已經累積的 stateful 資源（最重要的是 memory）不會跟著遷移。 kernel 文件明確警告：

Migrating processes across cgroups frequently as a means to apply different resource restrictions is discouraged. A workload should be assigned to a cgroup according to the system's logical and resource structure once on start-up.

正確做法是：啟動時把 workload 放進對的 cgroup，然後透過修改 controller interface files 動態調整，而不是頻繁搬 process。 這也是 Kubernetes In-Place Resize（v1.35 GA）試圖解決的問題——讓 Pod 在不重啟的情況下調整資源。

3.4 Domain vs Threaded cgroup

前面說過 v2 預設以 process 為粒度。但 v2 後來加回了限制版的 thread mode，於是 cgroup 多了一個型態（type）的概念。

每個 cgroup 都有一個 cgroup.type 檔案，可以是以下四種值之一：

cgroup.type 意義 典型用途
domain 標準 cgroup，以 process 為粒度。新建 cgroup 預設就是這個型態。 絕大多數場景：systemd service、Kubernetes Pod、容器
domain threaded 「承載 threaded subtree 的 root」。一個 domain cgroup 在 child 變成 threaded 後，自動變成這個型態。 需要做 thread-level 控制的應用入口
threaded thread-granular cgroup。可以放同一 process 的不同 thread。寫入 "threaded" 到 cgroup.type 就會轉成這個型態。 JVM thread pool 隔離、即時系統的 worker thread 分組
domain invalid 無效狀態。在某些拓撲下會自動進入此狀態（見下方說明）。 不該出現，需要修正

把 cgroup 變成 threaded 是單向操作——一旦轉成 threaded，就不能再變回 domain。同時，並非所有 controllers 都支援 threaded mode：

類別 Controllers 說明
Threaded controllers
支援 threaded subtree		 cpu, cpuset, perf_event, pids 這些資源天然可以 thread-level 分配（CPU 時間、affinity、perf 計數），所以 v2 補回了這部分功能。
Domain controllers
只支援 process-level		 memory, io, hugetlb, rdma, misc 這些資源在 thread 層級分配沒有意義（如 memory）或實作上很困難（如 io）。永遠在 process 粒度。

Threaded subtree 的拓撲規則

一旦你把 cgroup B 變成 threaded，整個從 B 開始往下的子樹都必須是 threaded。從根往上找，會遇到一個 domain threaded 型態的祖先，這個祖先就是整個 threaded subtree 的 resource domain——所有 domain controllers（memory、io 等）的計量都歸到這個祖先頭上。

什麼時候才會用 threaded？

老實說，絕大多數應用都不需要 threaded mode。它的設計目的非常具體：那些一個 process 內有「不同類型 thread」需要差別對待的場景。 最常被引用的例子是 JVM——你可能想把 GC threads 限制在某些 CPU 核上，把 worker threads 放開；或是即時系統把 high-priority threads 與 background threads 分開排程。

如果你只是想做容器、限 service 用多少 CPU，停在 domain 就好，不需要碰 threaded。Ch.6 會詳細討論 threaded 的進階用法。

3.5 關鍵 interface files 速覽

本章最後介紹貫穿全書的五個 core interface files。後續每章都會用到它們：

檔案 類型 作用
cgroup.controllers read-only 列出當前 cgroup 「可以啟用」的所有 controllers（由 parent 決定）。
cgroup.subtree_control read-write 啟用對 child cgroup 生效的 controllers。寫入 +cpu +memory 啟用、-io 停用。
cgroup.procs read-write 讀取列出此 cgroup 內所有 PID；寫入 PID 把該 process（連同所有 thread）遷入此 cgroup。
cgroup.type read-write 顯示型態（domain/domain threaded/threaded/domain invalid）；可寫入 threaded 轉換型態。
cgroup.events read-only + poll 包含 populated（子樹是否有 live process）等事件。可用 inotify/poll 監聽變化。

其他 core interface files（cgroup.kill、cgroup.pressure、cgroup.threads、cgroup.max.depth 等）會在後續對應章節介紹。 附錄 A 會提供完整的 interface files 速查表。

04 · Hands-on

實戰範例：建立並觀察一棵樹

下面三個實作從零建立一個簡單的 cgroup tree，逐步驗證本章每個概念。需要 root 權限。

Lab 1 · 建立第一個 cgroup 並啟用 controllers

先看 root cgroup 「可給」哪些 controllers，再啟用對 child 的 controllers，然後建立 child cgroup：

# 1. 看 root 可用的 controllers
sudo cat /sys/fs/cgroup/cgroup.controllers
# 預期輸出: cpuset cpu io memory hugetlb pids rdma misc

# 2. 看 root 已對 child 啟用哪些（多半 systemd 已經啟用 cpu memory pids）
sudo cat /sys/fs/cgroup/cgroup.subtree_control

# 3. 確認 root 沒禁用 io（沒有 io 就需要先啟用）
echo "+io" | sudo tee /sys/fs/cgroup/cgroup.subtree_control 2>/dev/null

# 4. 建立 child cgroup（注意：要建在不會跟 systemd 衝突的位置）
sudo mkdir /sys/fs/cgroup/mydemo

# 5. 觀察 child 收到了哪些 controllers
ls /sys/fs/cgroup/mydemo/ | head -20
cat /sys/fs/cgroup/mydemo/cgroup.controllers

觀察點：當你在 root 啟用 +io 後，/sys/fs/cgroup/mydemo/ 應該會出現 io.weight、io.max 等檔案。 這就是 top-down constraint 的具體表現——child 收到 parent 給的 controllers。

Lab 2 · 違反 no internal process constraint

故意觸發違規，看 kernel 會回什麼錯：

# 1. 把當前 shell 移進 mydemo
echo $$ | sudo tee /sys/fs/cgroup/mydemo/cgroup.procs

# 2. 確認 shell 已經在 mydemo
cat /proc/$$/cgroup
# 預期: 0::/mydemo

# 3. 嘗試在 mydemo 啟用 controllers 給 child
echo "+cpu" | sudo tee /sys/fs/cgroup/mydemo/cgroup.subtree_control
# 預期失敗：tee: write error: Device or resource busy
# 原因：mydemo 內已有 process，違反 no internal process constraint

# 4. 正確做法：先建 leaf 子目錄，把 process 移進去
sudo mkdir /sys/fs/cgroup/mydemo/leaf
echo $$ | sudo tee /sys/fs/cgroup/mydemo/leaf/cgroup.procs

# 5. 再次嘗試啟用 cpu controller
echo "+cpu" | sudo tee /sys/fs/cgroup/mydemo/cgroup.subtree_control
# 預期成功

這個 lab 親身體驗了 kernel 文件推薦的 leaf/ 模式：非 leaf cgroup 只負責「管」controllers，process 都放 leaf。

Lab 3 · 觀察 cgroup.events 的 populated 變化

cgroup.events 會在子樹從「有 process」變成「無 process」（或反過來）時觸發 inotify 事件。可以用一個簡單腳本觀察：

# Terminal 1: 持續監看 cgroup.events
while true; do
  cat /sys/fs/cgroup/mydemo/cgroup.events
  sleep 1
done

# Terminal 2: 把 shell 從 mydemo/leaf 搬走
# (假設 mydemo/leaf 內只有這個 shell，搬走後 mydemo 子樹就空了)
echo $$ | sudo tee /sys/fs/cgroup/user.slice/cgroup.procs

# Terminal 1 應觀察到 populated 從 1 變成 0

這個機制是 systemd-oomd 與 cAdvisor 等工具的基礎——它們監聽事件而非 polling。

Lab 4 · 清理
# 把所有 process 移走後刪除 cgroup
sudo rmdir /sys/fs/cgroup/mydemo/leaf
sudo rmdir /sys/fs/cgroup/mydemo
05 · Pitfalls

常見陷阱

陷阱 1 · 與 systemd 搶 cgroup 控制權

在 systemd-managed 系統上，systemd 是 root cgroup 的 single writer。 如果你直接 mkdir 在 /sys/fs/cgroup/ 下建立 cgroup（像本章 Lab 那樣做測試是 OK 的，但放生產不行）， systemd 不會理會它，但會在某些操作時把你的設定覆寫掉。 正確做法是用 systemd-run 或在自己的 service unit 加上 Delegate=yes 申請一塊委派的子樹。Ch.5 會詳細說明。

陷阱 2 · 誤以為「啟用 controller」是在 child 操作

要讓 cgroup B 受 cpu controller 控制，必須在它的 parent A 寫入 +cpu 到 cgroup.subtree_control，不是在 B 自己。 這是因為 controller interface files（如 cpu.max）「由 parent 持有，控制資源如何分配給 children」。 記住這句話：controller 介面檔案是給 parent 用的，不是給自己用的。

陷阱 3 · 寫入 cgroup.procs 一次只能一個 PID

cgroup.procs 在單次 write(2) syscall 只接受一個 PID。要遷移多個 process，必須每個 PID 一次寫入。 常見錯誤是把多個 PID 用換行串起來一次寫入，會只寫進第一個。 遷移整個 process group 應該寫個 loop，或用 systemctl move 之類的高階工具。

陷阱 4 · domain invalid 狀態自動觸發

在以下拓撲：A (threaded domain) → B (threaded) → C (新建的 domain)， C 因為 parent B 是 threaded，而它自己被預設建為 domain，立即進入 domain invalid 狀態，無法做任何操作。 修法是：把 C 也轉成 threaded（echo threaded > C/cgroup.type），或者直接刪除 C。 kernel 提供 domain invalid 而不是直接幫你轉換，是為了未來可能有更多 domain 子型態保留彈性。

陷阱 5 · 嘗試把 threaded cgroup 變回 domain

cgroup.type 從 domain 轉 threaded 是單向操作，不能反過來。如果寫錯了型態，唯一辦法是刪除整個 threaded subtree 重建。 所以做 threaded mode 之前要想清楚拓撲設計。

06 · Exercises

動手練習

本章四題練習，總時間約 30 分鐘。建議全做完再進入下一章。

練習 1 · 走訪整棵 cgroup tree（5 分鐘）

用 find 列出系統上所有 cgroup directory，並統計總數：

sudo find /sys/fs/cgroup/ -maxdepth 5 -type d | wc -l
sudo find /sys/fs/cgroup/ -maxdepth 5 -type d | head -30

回答：

  1. 你的系統有多少個 cgroup？
  2. 排前面的多半是 systemd 的 slice 還是 service？
  3. 找一個 service 對應的 cgroup directory，列出它有哪些檔案——能找到多少 cgroup.*？多少 {controller}.*？
練習 2 · 驗證 top-down constraint（10 分鐘）

跟著做，每一步觀察結果：

# Step 1: 在 systemd 的 user slice 下建一個測試 cgroup
sudo mkdir /sys/fs/cgroup/user.slice/test_topdown

# Step 2: 看它收到哪些 controllers
cat /sys/fs/cgroup/user.slice/test_topdown/cgroup.controllers

# Step 3: 把 user.slice 對 child 啟用的 controllers 列出來
cat /sys/fs/cgroup/user.slice/cgroup.subtree_control

# Step 4: 觀察兩者的關係（前者應該等於後者，或是後者的子集）

回答：你看到的兩個輸出有什麼關係？這對應到本章哪個鐵律？

練習 3 · 找出系統中持有 process 最多的 cgroup（10 分鐘）
sudo find /sys/fs/cgroup -name cgroup.procs -exec sh -c '
  count=$(wc -l < "$1")
  if [ "$count" -gt 0 ]; then
    echo "$count $(dirname $1)"
  fi
' _ {} \; 2>/dev/null | sort -rn | head -10

回答：你最大的 cgroup 是哪一個？放了多少 process？這跟你跑的 workload 有什麼關係？

練習 4 · 嘗試 threaded mode（15 分鐘）

建立一個小的 threaded subtree 並觀察：

# 1. 建立 domain cgroup A
sudo mkdir /sys/fs/cgroup/test_thread

# 2. 在 A 下建立 child B
sudo mkdir /sys/fs/cgroup/test_thread/B

# 3. 把 B 變成 threaded
echo threaded | sudo tee /sys/fs/cgroup/test_thread/B/cgroup.type

# 4. 看 A 的 type 變成什麼了
cat /sys/fs/cgroup/test_thread/cgroup.type
# 預期：domain threaded

# 5. 試圖在 B 啟用 memory controller（應失敗，因為 memory 是 domain controller）
echo "+memory" | sudo tee /sys/fs/cgroup/test_thread/B/cgroup.subtree_control 2>&1

# 6. 在 A 啟用 cpu 給 B
echo "+cpu" | sudo tee /sys/fs/cgroup/test_thread/cgroup.subtree_control

# 7. 看 B 收到的 controllers
cat /sys/fs/cgroup/test_thread/B/cgroup.controllers

# 8. 清理
sudo rmdir /sys/fs/cgroup/test_thread/B
sudo rmdir /sys/fs/cgroup/test_thread

回答：

  1. 第 5 步為什麼失敗？對應到本章哪個概念？
  2. 第 7 步顯示的 controllers 跟 root 的 cgroup.controllers 有什麼差異？為什麼？
07 · Self-check

自我檢查題

Q1.cgroup core 與 controllers 的職責怎麼劃分？怎麼從檔案名快速判斷是哪一類？
core 負責「樹的結構」——cgroup 怎麼建、process 屬於哪個 cgroup、樹的拓撲約束。介面檔案以 cgroup. 開頭，例如 cgroup.procs、cgroup.type、cgroup.controllers。 controllers 負責「資源分配」——CPU、memory、IO 等資源的計量與限制。介面檔案以 controller 名字開頭，例如 cpu.weight、memory.max、io.weight。 看開頭就能分辨。
Q2.解釋 top-down constraint，並說明它為什麼必要。
規則：一個 cgroup 只能對它的 child 啟用 parent 已經啟用的 controllers。具體實作上：要把某個 controller 寫進 cgroup.subtree_control，必須先在 cgroup.controllers 中可見，而 cgroup.controllers 由 parent 的 cgroup.subtree_control 決定。 為什麼必要：保證資源分配的一致性與可預測性——子 cgroup 永遠不能擁有 parent 沒給它的能力，避免「parent 想限制 cpu，但 child 偷偷不受 cpu 控制」這種矛盾。同時也讓 delegation 的安全模型成為可能。
Q3.no internal process constraint 的精確定義是什麼？root cgroup 為什麼可以豁免？
精確定義：一個非 root cgroup不能同時「直接持有 process」與「在 cgroup.subtree_control 啟用任何 domain controller」。注意這不是「不能同時有 process 和 child cgroup」——可以有，只要 child 沒啟用 controllers 就可以。 root 豁免：root cgroup 必須能持有 kernel thread（idle thread、kthreadd、各種 kworker），這些 thread 不屬於任何使用者 process，無法移到 child。所以 root 是設計上的特例，可以同時有 process 與啟用 controllers 的 child。 推薦寫法：建立 leaf/ 子目錄專門放 process，parent 只負責管 controllers，把 process 與 controller 管理徹底分離。
Q4.single-writer rule 是 kernel 強制的嗎？違反了會發生什麼？
不是 kernel 強制的——root 想做什麼都可以，kernel 不會阻止你同時用多個工具去寫同一個 cgroup。 但違反它會帶來無止盡的痛苦：不同工具會互相覆蓋對方的設定，比如你用 systemctl set-property 設了 MemoryMax=1G，又直接 echo 2G > memory.max，下一次 systemd reload 時你的 2G 會被覆寫回 1G。 實務上：systemd-managed 系統由 systemd 是唯一 writer，其他軟體想管自己的子樹要透過 Delegate=yes 機制申請，這是 cgroup v2 設計的合作模型。
Q5.「organize once and control」這條規則的實務意義是什麼？舉一個違反的例子。
實務意義：把 workload 放進對的 cgroup 是啟動時的一次性決定，後續用 controller interface files 動態調整資源，不要頻繁搬 process。 違反例：寫一個 cron job，每天根據負載把某個 service 從 highprio.slice 搬到 lowprio.slice，希望實現「白天高優、晚上低優」。這會踩兩個雷：（1）累積的 memory 不會跟著遷移，搬完後 cgroup 統計會失真；（2）syscall 開銷在大型 service 上不可忽略。 正確做法：放在固定 cgroup 內，用 systemctl set-property xxx.service CPUWeight=... 動態調整 weight，不搬 process。
Q6.哪些 controllers 支援 threaded mode？為什麼 memory 與 io 不支援？
Threaded controllers：cpu、cpuset、perf_event、pids。這四個資源天然可以 thread-level 分配——CPU 時間每個 thread 獨立排程、cpuset affinity 可以 per-thread 設定、perf 計數本來就是 per-thread、pids 數量是計數而非歸屬。 memory 不支援的原因：同一 process 的所有 thread 共享 address space，把它們放進不同 memory cgroup 在語意上講不通——同一塊 page 屬於誰？ io 不支援的原因：file descriptor 與 page cache 在 process 層級共享，writeback 是非同步的（kworker 觸發），thread 層級的 io 歸屬實作上極為困難。 所以這兩個典型的 domain controllers 永遠在 process 粒度。
Q7.什麼情況下 cgroup 會進入 domain invalid 狀態？怎麼修復？
觸發情境：當 cgroup 的 parent 是 threaded 型態，但 cgroup 自己被建為 domain（所有新建的 cgroup 預設都是 domain），它就會立即進入 domain invalid 狀態。 典型例子：A (threaded domain) → B (threaded) → C (domain, just created)，C 因為 parent B 是 threaded 而陷入 invalid。 修復方式：兩條路。（a）把 C 也轉成 threaded：echo threaded > C/cgroup.type，融入 threaded subtree；（b）刪除 C：rmdir C。 kernel 不直接幫你自動轉換，是為了未來可能有更多 domain 子型態保留 API 彈性。
Q8.解釋下面這個指令的每一步在做什麼：
echo "+cpu +memory" | sudo tee /sys/fs/cgroup/myapp/cgroup.subtree_control
echo $$ | sudo tee /sys/fs/cgroup/myapp/leaf/cgroup.procs
第一行：在 myapp cgroup 啟用 cpu 與 memory controllers 對它的 child cgroups 生效。注意是對 child 生效，不是對 myapp 自己。執行後 myapp/leaf/ 等所有 child 會多出 cpu.* 與 memory.* 介面檔案。 第二行：把當前 shell（$$ 是 shell PID）遷入 myapp/leaf。寫入後 /proc/$$/cgroup 會顯示 0::/myapp/leaf。 整體意義：建立了一個典型的 cgroup 操作模式——在 parent 設好 controllers，把 process 都放在 leaf。這正是 kernel 文件推薦的標準做法，避開了 no internal process constraint。
