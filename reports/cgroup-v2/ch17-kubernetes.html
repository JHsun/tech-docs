Kubernetes 整合（雲原生）

Kubernetes 是 cgroup v2 在生產環境最大規模的使用者。每個 Pod 對應一個 cgroup 子樹、 每個 QoS class 對應不同 cgroup 階層、每個 limit / request 對應特定 cgroup 介面。 但 K8s 也是 cgroup v2 最複雜的整合——kubelet、container runtime、CRI、CPU Manager、Memory Manager 多層協作， 加上 2024–2026 年 K8s 1.34/1.35/1.36 推出的 PSI metrics、Memory QoS、In-Place Resize、Pod-level resources 等新功能讓 cgroup 整合更深入。 本章把 K8s + cgroup v2 完整講透——從 kubepods 階層到 QoS class、從 Memory QoS 到 In-Place Resize、 從 PSI-based autoscaling 到生產級 troubleshooting。

01 · Highlights

本章重點

  1. kubelet 維護的 cgroup 階層：/kubepods.slice/ 三層樹——root → QoS slice → Pod slice → container scope。每層 cgroup 設不同 limit，由 kubelet 與 container runtime 分工管理。
  2. QoS class 決定 cgroup 行為：Guaranteed（限=請求）、Burstable（限>請求）、BestEffort（無限）。三類各自進不同 slice，OOM 順序與 reclaim 優先級由 cgroup hierarchy 自動 enforce。
  3. Memory QoS（K8s 1.34 Beta）：用 cgroup v2 的 memory.min（保證請求記憶體不被 reclaim）+ memory.high（接近 limit 時 throttle 而非 OOM）取代 v1 的純 hard limit 設計。需要 kernel 5.9+。
  4. In-Place Pod Resize（K8s 1.35 GA / 1.36 Pod-level Beta）：不重啟 Pod 就能調整 resources，靠 cgroup v2 的 dynamic update。配合 PSI 可做 closed-loop autoscaling。
  5. PSI metrics（K8s 1.34 Beta）：kubelet 直接 export 節點與 Pod 的 PSI 到 metrics endpoint，是autoscaling 的 killer metric——告訴你 process 是不是真的在等資源，而非「看似忙」。

03 · Core Concepts

核心內容

3.1 kubelet 的 cgroup 階層

K8s 啟動時，kubelet 在 /sys/fs/cgroup 下建立三層樹狀結構：root cgroup → QoS slice → Pod slice → container scope。 所有 Pod 都進這個 tree，cgroup 限制由分層繼承與 enforcement。

圖 17.1 · K8s kubelet 維護的 cgroup tree

關鍵設計：

  • kubelet 管前三層：kubepods.slice、QoS slice、Pod slice。透過 libcontainer 直接寫 cgroup 介面。
  • container runtime 管最內層：containerd / CRI-O 建 container scope，在 Pod slice 之下。
  • Guaranteed 直接掛 root：沒有 中間的 guaranteed.slice——這是設計簡化，因為 Guaranteed Pod limit 確定，不需要中間層匯總。
  • Burstable / BestEffort 走中間 slice：方便整類 Pod 統一限制（如「所有 BestEffort 加起來不能超過 1GB」）。

3.2 QoS class 完整對應 cgroup 設定

QoS class 在 Pod 創建時自動決定（看 resources.requests / limits）：

QoS class判斷規則cgroup 設定
Guaranteed 所有 container 的 cpu / memory limits == requests，且都有設 memory.max = limit
cpu.max = limit
cpu.weight 高（最大優先級）
OOM 最後被殺
Burstable 至少有一個 container 的 limit ≠ request（含部分沒設） memory.max = limit
memory.min = request（Memory QoS）
cpu.weight = request × 比例
OOM 中等優先級
BestEffort 所有 container 都沒有 request 與 limit memory.max = max（無）
cpu.weight 最低（2）
OOM 最先被殺

OOM 順序的 cgroup 機制

K8s「BestEffort 先殺、Guaranteed 最後殺」靠 kernel OOM 計算 score——每個 process 的 oom_score_adj 由 kubelet 設定：

  • Guaranteed：oom_score_adj = -997（極低，幾乎不會被選中）
  • Burstable：oom_score_adj = 計算值（基於 memory.request / node 容量）
  • BestEffort：oom_score_adj = 1000（最高，最先被殺）

3.3 Memory QoS（K8s 1.34 Beta）

傳統 K8s 記憶體限制只用 v1 的 hard limit——應用接近 limit 直接 OOM。Memory QoS 引入 v2 的 memory.high 做軟限制 + throttle，避免直接 OOM：

QoSmemory.minmemory.highmemory.max
Guaranteed = request 不設（豁免 throttle） = limit
Burstable（有 limit） = request request + 0.9 × (limit − request) = limit
Burstable（無 limit） = request 0.9 × node allocatable = max（無）
BestEffort 0 不設 max

memoryThrottlingFactor：預設 0.9（可調）。意思是「當記憶體使用達到 limit 與 request 之間 90% 時開始 throttle」。

啟用方式： 

# /var/lib/kubelet/config.yaml
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
featureGates:
  MemoryQoS: true
memoryQoS:
  memoryThrottlingFactor: 0.9   # 預設

需求：kernel 5.9+（livelock 修復）、cgroup v2、CRI runtime 支援 LinuxContainerResources.Unified（containerd 1.6+、CRI-O 1.22+）。

觀察 Memory QoS 在運作： 

POD_UID=$(kubectl get pod my-pod -o jsonpath='{.metadata.uid}')
CG=/sys/fs/cgroup/kubepods.slice/kubepods-burstable.slice/kubepods-burstable-pod${POD_UID//-/_}.slice

# 看 memory.min（保證）
cat $CG/memory.min     # 應該等於 request

# 看 memory.high（throttle 閾值）
cat $CG/memory.high

# 看 throttle 事件
cat $CG/memory.events
# high counter 增加 = 應用達到 throttle，受 Memory QoS 保護

# 看是否在 throttle 中（PSI）
cat $CG/memory.pressure
# full avg10 > 5 表示嚴重 throttle

3.4 In-Place Pod Resize（K8s 1.35 GA）

傳統 Pod 的 resources 是 immutable——改 limit 必須重啟 Pod。在 production 這意味著：

  • JVM 重啟丟失 JIT，需要 5–15 分鐘 warm-up
  • Redis 重啟丟失 cache，下游服務 cache miss spike
  • PostgreSQL 重啟觸發 WAL replay，可能 5+ 分鐘 unavailable
  • 結果：VPA 只能當「建議工具」，沒人敢自動 resize 生產 Pod

In-Place Resize 利用 cgroup v2 的動態更新能力——kubelet 直接寫新值到 memory.max / cpu.max，不重啟 process。

API：subresource resize（K8s 1.32+ kubectl 支援）： 

# 增加 CPU（不重啟）
kubectl patch pod my-pod --subresource resize --patch \
  '{"spec":{"containers":[{"name":"app","resources":{"requests":{"cpu":"800m"},"limits":{"cpu":"800m"}}}]}}'

# 等同於 kubelet 寫入：
echo "800000 100000" > /sys/fs/cgroup/.../cpu.max
# process 立即取得新 quota，沒有任何中斷

# 看狀態
kubectl get pod my-pod -o yaml | grep -A 10 'status'
# observedResources 顯示實際 cgroup 中的值
# allocatedResources 顯示 kubelet 已套用的值

resizePolicy 控制重啟行為

spec:
  containers:
  - name: app
    resizePolicy:
    - resourceName: cpu
      restartPolicy: NotRequired   # CPU 改變不重啟（推薦）
    - resourceName: memory
      restartPolicy: RestartContainer  # 記憶體減少時重啟（保守）
    resources:
      limits: { cpu: "1", memory: "1Gi" }
      requests: { cpu: "500m", memory: "512Mi" }

1.36 進入 Pod-level Beta：除了 container-level，也能調整 Pod 整體的 spec.resources（feature gate InPlacePodLevelResourcesVerticalScaling）。

限制：

  • 只支援 CPU 與 memory（其他 resource type 不行）
  • QoS class 不能變（Guaranteed → Burstable 不允許）
  • 記憶體減少時 best-effort——若實際使用 > 新 limit 會跳過
  • 需要 cgroup v2 + containerd 2.0+

3.5 PSI metrics（K8s 1.34 Beta）

前面 Ch.12 講過 PSI 是 cgroup v2 的核心觀測能力。K8s 1.34 把它整合進 kubelet metrics endpoint： 

# 看節點 PSI
curl https://node-ip:10250/metrics | grep psi
# node_cpu_pressure_some_avg10
# node_memory_pressure_full_avg10
# node_io_pressure_some_avg10

# 看 Pod PSI（per-pod metrics）
kubectl top pod --use-protocol-buffers my-pod
# 或從 Prometheus
kube_pod_pressure_memory_full_avg10{pod="my-pod"}

為什麼 PSI 是「autoscaling killer metric」：

  • 傳統 HPA 看 CPU usage——但 80% CPU 可能是「真的忙」也可能是「有人卡住」。
  • PSI 直接告訴你「process 因等資源而 stall 多久」——這是 user-perceived latency 的根源。
  • 用 PSI 做 autoscaling target 比 CPU usage 準確得多——「memory.pressure full avg10 > 5%」是「imminent OOM 強烈訊號」（Netdata 文件）。

PSI-based HPA 範例

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: psi-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-app
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Pods
    pods:
      metric:
        name: kube_pod_pressure_memory_some_avg10
      target:
        type: AverageValue
        averageValue: "5"   # PSI > 5% 就 scale out

PSI-based node tainting（K8s 1.36 Alpha）：節點 PSI 過高自動加 taint，scheduler 不再 schedule 新 Pod 到該節點——避免雪崩。

3.6 CPU Manager 與 cpuset

Pod 設 Guaranteed QoS + CPU 為整數時，CPU Manager（kubelet feature）可以把 container 綁到特定 CPU——透過 cgroup v2 的 cpuset。 

# 啟用 CPU Manager（static policy）
# /var/lib/kubelet/config.yaml
cpuManagerPolicy: "static"
reservedSystemCPUs: "0,1"  # CPU 0-1 給系統，不給 Pod 用

# Pod 必須是 Guaranteed + 整數 CPU
spec:
  containers:
  - name: app
    resources:
      limits:    { cpu: "4", memory: "8Gi" }
      requests:  { cpu: "4", memory: "8Gi" }

# CPU Manager 自動把 container 綁到 4 個獨佔 CPU
# 看 cgroup
cat /sys/fs/cgroup/.../cpuset.cpus
# 例：4-7（4 個獨佔 CPU）

適用場景：低延遲應用（DB、game server、media processing）需要避免 CPU 上下文切換。設 static policy 後，這些 Pod 獨佔 CPU，不被其他 Pod 干擾。

3.7 Pod-level resources（K8s 1.34 Beta，1.36 持續完善）

傳統 K8s 只能在 container 層設 resources。Pod-level resources 讓你在 Pod spec 直接設整體限制： 

apiVersion: v1
kind: Pod
metadata:
  name: pod-level-demo
spec:
  resources:
    requests: { cpu: "2", memory: "4Gi" }
    limits:   { cpu: "4", memory: "8Gi" }
  containers:
  - name: app
    image: my-app
    # 不設 resources，從 Pod-level 繼承
  - name: sidecar
    image: my-sidecar
    resources:
      limits: { cpu: "500m", memory: "1Gi" }

意義：Pod-level resources 對應 Pod cgroup（kubepods-pod{UID}.slice）的限制，整個 Pod 共享這個上限。container-level limit 仍可在 Pod-level 之下細分。

啟用：feature gate PodLevelResources（1.34 Beta、1.35 預設啟用）。

3.8 Pod 排查與 cgroup 對應

實戰技巧——從 Pod 找 cgroup path： 

# 取 Pod UID
POD_UID=$(kubectl get pod my-pod -o jsonpath='{.metadata.uid}')

# Guaranteed
CG=/sys/fs/cgroup/kubepods.slice/kubepods-pod${POD_UID//-/_}.slice/

# Burstable
CG=/sys/fs/cgroup/kubepods.slice/kubepods-burstable.slice/kubepods-burstable-pod${POD_UID//-/_}.slice/

# BestEffort
CG=/sys/fs/cgroup/kubepods.slice/kubepods-besteffort.slice/kubepods-besteffort-pod${POD_UID//-/_}.slice/

# 看 Pod 整體限制（Pod-level cgroup）
cat $CG/memory.max
cat $CG/cpu.max
cat $CG/memory.pressure

# 看 container 限制
ls $CG  # 看到 cri-containerd-{ID}.scope/
cat $CG/cri-containerd-{ID}.scope/memory.max

# 看 Pod 即時用量
cat $CG/memory.current
cat $CG/memory.peak
cat $CG/cpu.stat

3.10 best practices

  • 所有節點升級到 cgroup v2 + kernel 5.9+：Memory QoS 與 PSI 都要這個底
  • kubelet 與 runtime 用 systemd driver：1.36+ 用 KubeletCgroupDriverFromCRI 自動偵測
  • 啟用 Memory QoS：Burstable Pod 享受 throttle 而非直接 OOM
  • VPA 配合 In-Place Resize：終於可以讓 VPA 在 production 自動 resize
  • HPA 用 PSI metrics：比 CPU usage 準得多，特別對 memory-bound、IO-bound workload
  • Guaranteed QoS 給 critical service：保證 cgroup 優先級，OOM 最後被殺
  • CPU Manager static policy 給 latency-critical：DB、game server、media 等
  • 監控 PSI + memory.events：是 alert / autoscaling 的最佳訊號

05 · Pitfalls

常見陷阱

陷阱 1 · Memory QoS 在舊 kernel 觸發 livelock

kernel < 5.9 啟用 Memory QoS：memory.high 達到後 kernel 持續嘗試 reclaim，process 進入「looking running but doing nothing」狀態。修法：升 kernel 到 5.9+，或 disable MemoryQoS feature gate。檢測：cat memory.pressure full avg10 持續高、memory.events high counter 暴增。

陷阱 2 · In-Place Resize 改 QoS class 失敗

Pod 是 Burstable（limit ≠ request），想 resize 成 Guaranteed（limit = request）——失敗。原因：QoS class 在 Pod 創建時固定，不能透過 resize 改變。修法：要改 QoS 必須重建 Pod。In-Place Resize 只能在同 QoS 內調整。

陷阱 3 · 記憶體 In-Place 減少觸發 OOM

Pod 當前用 800MB，把 limit 從 1Gi 減到 500Mi——預期：直接 OOM。實際 K8s 1.36 行為：kubelet 偵測到 current > new limit，跳過 resize，狀態 "In Progress"。但這只是 best-effort，race condition 下仍可能 OOM。修法：減記憶體前先確認 current usage < 新 limit，或設 resizePolicy=RestartContainer 接受重啟。

陷阱 4 · kubelet 與 runtime cgroup driver 不一致

kubelet 設 systemd、containerd 設 cgroupfs（或反之）——Pod 啟動偶發失敗、cAdvisor 取資料錯亂。Ch.16 講過。修法：兩邊一致用 systemd。K8s 1.36+ 用 KubeletCgroupDriverFromCRI 自動偵測。

陷阱 5 · CPU Manager static policy 配 Burstable Pod 沒效果

啟用 static policy，期望 CPU 綁定——但 Burstable 與非整數 CPU Pod 完全沒 CPU pinning。原因：static policy 只 對 Guaranteed + 整數 CPU 的 container 生效。修法：要 CPU pinning 必須 Guaranteed + 整數 cpu.limits（cpu: "4" 而非 "3500m"）。

陷阱 6 · 嘗試在 Pod cgroup 自己 mkdir 子 cgroup

應用想在自己的 cgroup 之下建子 cgroup（如 nested container）——失敗。原因：K8s 預設沒對 Pod cgroup 設 Delegate=yes，違反 single-writer。修法：Pod 若需要管自己的 cgroup 子樹，要透過 RuntimeClass 自訂 runtime 加 Delegate，或用 user namespace + rootless container（K8s 1.33+ user namespaces on-by-default）。

陷阱 7 · 想看 BestEffort Pod 用了多少資源

BestEffort Pod 沒設 limit，kubectl top pod 顯示「?」或空白？原因：metrics-server 部分 metric 依賴 limit 計算，沒 limit 就無法顯示「used / limit」百分比。修法：直接看 cgroup memory.current / cpu.stat。或部署 cAdvisor 看絕對值。

陷阱 8 · K8s 1.36 升級後 containerd 1.x 整個炸

升級 K8s 1.35 → 1.36，所有 containerd 1.x 節點 Pod 啟動失敗——KubeletCgroupDriverFromCRI 在 1.36 強制使用，舊 containerd 不支援 RuntimeConfig CRI RPC。修法：升級前 確認所有節點 containerd ≥ 2.0。1.37 將完全移除 fallback。

練習 1 · 設計 Pod resources 對應特定 cgroup 行為（10 分鐘）

對以下需求各寫一個 Pod yaml：

  1. 關鍵 DB Pod：保證 4 CPU + 8GB，OOM 最後被殺
  2. Web service Pod：平常 100m CPU、burst 到 500m，記憶體 200Mi-500Mi 區間，受 Memory QoS throttle 保護
  3. Background batch：BestEffort，可被 evict
  4. Pod-level resources：3 個 container 共享 2 CPU + 4GB 上限
練習 2 · 寫 audit script 找出 unbounded Pods（15 分鐘）
#!/bin/bash
# k8s-resource-audit.sh

echo "=== Unbounded Pods ==="
kubectl get pods --all-namespaces -o json | \
    jq -r '.items[] | select(
        .spec.containers[] |
        (.resources.limits.memory == null or .resources.limits.cpu == null)
    ) | "\(.metadata.namespace)/\(.metadata.name)"'

echo "=== BestEffort Pods（無 request 與 limit）==="
kubectl get pods --all-namespaces -o json | \
    jq -r '.items[] | select(.status.qosClass == "BestEffort") | 
        "\(.metadata.namespace)/\(.metadata.name)"'

echo "=== 高優先級但非 Guaranteed 的 Pod ==="
# priorityClassName=high 但 QoS != Guaranteed 是常見錯配
kubectl get pods --all-namespaces -o json | \
    jq -r '.items[] | select(
        .spec.priorityClassName == "high" and
        .status.qosClass != "Guaranteed"
    ) | "\(.metadata.namespace)/\(.metadata.name)"'

跑在你的 cluster，看有多少 Pod 設定不健康。

07 · Self-check

自我檢查題

Q1.K8s 的 cgroup tree 有幾層？kubelet 與 container runtime 各自管哪些？
三層結構：
  1. root：/kubepods.slice——所有 Pod 容器的根（不含 system / user.slice）
  2. QoS slice：burstable.slice、besteffort.slice（Guaranteed 直接掛 root，沒中間層）
  3. Pod slice：kubepods-pod{UID}.slice 或 kubepods-burstable-pod{UID}.slice
  4. Container scope：cri-containerd-{ID}.scope（在 Pod slice 之下）
分工：
  • kubelet 管前三層（root、QoS slice、Pod slice）：透過 libcontainer 直接寫 cgroup 介面。維護 hierarchy 結構與 Pod-level limit。
  • container runtime 管最內層 container scope：containerd / CRI-O 透過 OCI runtime（runc / crun）建 scope，套用 container-level limit。
為什麼 Guaranteed 沒 QoS slice 中間層：Guaranteed Pod 的 limit 都是確定的，不需要中間層匯總。BestEffort / Burstable 走中間 slice 是為了「整類 Pod 統一限制」（如系統設「BestEffort 加起來不超過 X%」）。
Q2.QoS class 怎麼決定？三類各自對應的 cgroup 設定差別？
判斷規則：
  • Guaranteed：所有 container 的 cpu / memory limits == requests 且都有設
  • Burstable：至少一個 container 的 limit ≠ request
  • BestEffort：所有 container 都沒 request 與 limit
cgroup 對應：
QoSmemory.maxmemory.mincpu.weightoom_score_adj
Guaranteed= limit= request高-997
Burstable= limit= request中計算
BestEffortmax02（最低）1000
OOM 順序：oom_score_adj 高的先被殺。BestEffort（1000）→ Burstable（變動）→ Guaranteed（-997，幾乎不會被殺）。 實務影響：critical service 必設 Guaranteed（DB、auth）；web app 可 Burstable；可丟的 batch job 可 BestEffort（但要注意 K8s 內某些 controller 限制 BestEffort）。
Q3.Memory QoS（K8s 1.34 Beta）解決什麼問題？memory.high 怎麼計算？
解決問題：傳統 K8s 記憶體限制只用 hard limit（memory.limit_in_bytes）——應用接近 limit 直接 OOM kill。沒有「慢慢 throttle」中間態。 Memory QoS 設計：用 cgroup v2 的三介面：
  • memory.min = request：保證請求記憶體不被 reclaim
  • memory.high：軟限制 + throttle threshold
  • memory.max = limit：硬限，超過 OOM
memory.high 計算（throttling factor 預設 0.9）：
  • Guaranteed：不設（豁免 throttle，享受最高優先級）
  • Burstable（有 limit）：request + 0.9 × (limit − request)
  • Burstable（無 limit）：0.9 × node allocatable
  • BestEffort：不設
意義：當應用記憶體用到 90% 區間時，kernel 開始 throttle 而不是直接 OOM，給 monitor / autoscaler 介入時間。 需求：kernel 5.9+（livelock 修復）、cgroup v2、CRI runtime 支援 LinuxContainerResources.Unified（containerd 1.6+、CRI-O 1.22+）。 1.34 Beta 啟用：feature gate MemoryQoS、kubelet config 設 memoryQoS.memoryThrottlingFactor（預設 0.9）。
Q4.In-Place Pod Resize 解決什麼問題？怎麼實作？有什麼限制？
解決問題：傳統 K8s 的 resources 是 immutable——改 limit 必須重啟 Pod。在 production 痛點：
  • JVM 重啟丟失 JIT，5–15 分鐘 warm-up
  • Redis 重啟丟失 cache，下游 cache miss spike
  • PostgreSQL 重啟觸發 WAL replay，5+ 分鐘 unavailable
  • 結果：VPA 只能當「建議工具」，沒人敢自動 resize 生產 Pod
實作機制：利用 cgroup v2 的動態更新能力。kubelet 直接 write 新值到 memory.max / cpu.max——process 立即取得新 quota，不重啟。 API：kubectl patch pod NAME --subresource resize --patch '{...}'（需 kubectl 1.32+）。 resizePolicy：每個 resource 可獨立設 NotRequired（不重啟）或 RestartContainer（重啟）。CPU 通常 NotRequired，memory 視 application 而定。 限制：
  1. 只支援 cpu 與 memory（其他 resource type 不行）
  2. QoS class 不能變
  3. 記憶體減少時 best-effort——若 current usage > 新 limit 會跳過或重啟
  4. 需要 cgroup v2 + containerd 2.0+ + K8s 1.32+
1.35 GA / 1.36 Pod-level Beta：1.35 container-level GA；1.36 Pod-level（spec.resources）進 Beta。VPA 終於可以在 production 自動 resize。
Q5.PSI metrics 在 K8s 1.34 Beta 整合，為什麼是「autoscaling killer metric」？
傳統 HPA 指標的問題：CPU usage 80% 可能是「真的忙」也可能是「有人卡住」。HPA 看不到差別，可能誤判：
  • application 卡在 IO（CPU 30% 但 latency 飛漲）→ CPU-based HPA 不 scale
  • application CPU 真的滿載做有用工作 → CPU-based HPA scale 也合理
  • memory 接近 limit 但還沒 OOM → CPU-based HPA 看不到
PSI 的優勢：直接告訴你「process 因等資源而 stall 多少時間」——這就是 user-perceived latency 的根源。
  • cpu.pressure some：有 task 在等 CPU。> 5% 表示明顯擁擠。
  • memory.pressure full：所有 process 都在 stall 等記憶體。> 5% 是「imminent OOM 強烈訊號」（Netdata 文件）。
  • io.pressure full：整個 cgroup 都在等 IO。throughput 為零。
K8s 1.34 整合：kubelet 直接 export 節點與 Pod 的 PSI 到 metrics endpoint：
  • node_cpu_pressure_some_avg10
  • node_memory_pressure_full_avg10
  • kube_pod_pressure_memory_full_avg10{pod="..."}
HPA 範例： 
metrics:
- type: Pods
  pods:
    metric:
      name: kube_pod_pressure_memory_some_avg10
    target:
      type: AverageValue
      averageValue: "5"   # PSI > 5% 就 scale out
1.36 Alpha 進階：PSI-based node tainting——節點 PSI 過高自動加 taint，scheduler 不再 schedule 新 Pod，避免雪崩。
Q6.CPU Manager 的 static policy 怎麼運作？什麼 Pod 才能受益？
static policy 運作機制：
  1. kubelet 啟動時讀 reservedSystemCPUs（給系統保留的 CPU）
  2. Pod 啟動時，若是 Guaranteed + 整數 cpu limit，CPU Manager 從可用 pool 分配獨佔 CPU
  3. 透過 cgroup v2 的 cpuset.cpus 把 container 綁到這些 CPU
  4. 其他 Pod 不能使用這些 CPU——避免上下文切換干擾
什麼 Pod 受益：
  • 必須是 Guaranteed QoS（limit = request）
  • cpu limit 必須是整數（"4" 而非 "3500m"）
  • 典型 workload：低延遲應用（DB、game server、media processing、HFT）
非 Guaranteed Pod：跑在 shared CPU pool（剩下的 CPU），可隨意排程，沒 pinning。 啟用方式： 
# /var/lib/kubelet/config.yaml
cpuManagerPolicy: static
reservedSystemCPUs: "0,1"  # 給系統用
注意：靜態策略一旦啟用不能直接切回 none——需要 drain node、清理 cpu_manager_state 檔案、再重啟 kubelet。
Q7.Pod-level resources 與 container-level resources 的關係？什麼時候用 Pod-level？
關係：
  • Pod-level（spec.resources，K8s 1.34 Beta）：對應 Pod cgroup（kubepods-pod{UID}.slice）的限制。整個 Pod 的所有 container 加起來不能超過。
  • container-level（spec.containers[].resources）：對應 container scope 的限制。單一 container 的限制。
cgroup 階層： 
/kubepods.slice/kubepods-pod{UID}.slice/    ← Pod-level limit
    cri-containerd-{ID-1}.scope/            ← container 1 limit
    cri-containerd-{ID-2}.scope/            ← container 2 limit
Pod-level 是外部上限，container-level 在它之下細分。container-level 不能超過 Pod-level。 什麼時候用 Pod-level：
  1. 多 container Pod 共享配額：3 個 sidecar，總共給 2 CPU + 4GB，container 之間自動分配 idle 資源。
  2. 難以準確估各 container 用量：沒辦法精確分到每個 container，但 Pod 整體合理。
  3. 避免過度規劃：傳統做法每個 container 設 limit 加起來容易超過實際 Pod 需求；Pod-level 更精確。
啟用：feature gate PodLevelResources（1.34 Beta）。 1.36 進階：Pod-level In-Place Resize 進 Beta（feature gate InPlacePodLevelResourcesVerticalScaling）——Pod 整體配額也可以 in-place 調整。
Q8.實作思考題：你管理 200 節點 K8s 1.32 cluster，跑生產 workload。要升到 1.36 並啟用 Memory QoS / In-Place Resize / PSI HPA。請設計完整 migration 計劃。
完整 migration 計劃（5 階段）： 
[Phase 1: 前置檢查（1-2 週）]
1. 所有節點檢查：
   - kernel 版本 ≥ 5.9（Memory QoS 需要）
   - cgroup v2 in use：mount | grep cgroup2fs
   - containerd 版本 ≥ 2.0（K8s 1.36 強制）
   - cgroup driver = systemd（kubelet + runtime）
2. 寫 audit script 找問題節點
3. 升級 kernel / containerd 到符合需求

[Phase 2: cluster 升級到 1.34 first（不跳版）]
4. 1.32 → 1.33（測試 In-Place Resize Beta）
5. 1.33 → 1.34（PSI metrics、Memory QoS 進 Beta）
6. 觀察 1-2 週，確認沒有 regression
7. 在 1.34 stage 啟用 feature gates（先 staging cluster）：
   - MemoryQoS=true
   - InPlacePodVerticalScaling=true
   - PodLevelResources=true

[Phase 3: 漸進啟用新功能]
8. Memory QoS：先對 dev / staging 啟用
   - 觀察 memory.events high counter
   - 確認 PSI full avg10 不會持續 > 5
   - 1 週後 rollout 到 production
9. PSI metrics 採集：
   - 部署 Prometheus PSI scraping
   - 建 Grafana dashboard
   - 設 alert（PSI > 閾值）
10. In-Place Resize：
    - 從非 critical service 試
    - 改 resizePolicy 為 NotRequired
    - 結合 VPA recommend mode 測試 resize 效果
    - 確認沒有 OOM / 不正確 throttle 後 rollout

[Phase 4: 升到 1.35 / 1.36]
11. 1.34 → 1.35（In-Place Resize GA）
12. 1.35 → 1.36（containerd 2.0 強制、PSI node tainting Alpha）
13. 啟用 1.36 新 feature gate：
    - InPlacePodLevelResourcesVerticalScaling
    - KubeletCgroupDriverFromCRI（自動偵測）

[Phase 5: 進階 closed-loop]
14. PSI-based HPA：
    - 對 latency-critical service 設 HPA target = memory.pressure some
    - 對 batch service 設 cpu.pressure some
15. VPA + In-Place Resize 自動化：
    - VPA 改成 Auto mode（不再只是 recommend）
    - 配合 PSI alerts 做 closed-loop
16. PSI node tainting：
    - 啟用後 node 過載自動避免新 Pod schedule

[防護措施]
17. 每階段都有 rollback plan
18. canary deploy：先 1 個 node，再 10%，再 100%
19. 寫 runbook：每個 alert 對應 playbook
20. 定期 review：traffic pattern 變化時調整 PSI 閾值
關鍵風險點：
  1. kernel 升級：必須 reboot node，drain → upgrade → reboot 流程要熟。
  2. containerd 1.x → 2.0：API 有 breaking change，需重啟 runtime。
  3. Memory QoS 在舊 kernel livelock：必須先確認 kernel 5.9+ 再 enable。
  4. cgroupfs → systemd driver 切換：必須 drain node 並重啟所有 Pod（不能線上切換）。
  5. PSI HPA 抖動：閾值要保守、設 stabilization window 防 thrashing。
成功指標：
  • 所有節點 cgroup v2 + containerd 2.0+ + kernel 5.9+
  • Memory QoS 啟用，OOM rate 下降 50%+
  • VPA + In-Place Resize 可生產用，cluster utilization 從 30% 升到 60%+
  • PSI HPA 啟用，latency p99 改善
