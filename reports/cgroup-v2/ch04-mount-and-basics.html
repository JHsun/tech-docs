01 · Highlights

本章重點

  1. 掛載指令：mount -t cgroup2 none /sys/fs/cgroup。但實務上 systemd 會自動處理，使用者很少需要手動掛載。
  2. 五個 mount options：nsdelegate（namespace 委派邊界）、favordynmods（優化動態修改）、memory_recursiveprot（保護遞迴傳播）、memory_localevents（事件只算自己）、memory_hugetlb_accounting（HugeTLB 計入記憶體）。
  3. Interface files 命名鐵則：cgroup. 開頭是 core；{controller}. 開頭是 controller；weight 範圍 1–10000、min/max/low/high 是固定四種命名、特殊值統一用 max。

  5. v2 獨有功能：cgroup.kill（一次殺光整個子樹）、cgroup.freeze（凍結整個子樹）、cgroup.events（poll/inotify 事件通知）。
02 · Latest

最新動態（2025–2026）

2025 Oracle Linux 在 UEK8 預設啟用 favordynmods，container 啟動時間延遲降低 30%。CONFIG_CGROUP_FAVOR_DYNMODS=y 成為某些 distro 的預設 build option。
2025 memory_recursiveprot 在多數新 distro 預設開啟。Ubuntu 22.04+、RHEL 9+、Debian 11+、Fedora 31+ 都已預設啟用，不再需要手動 remount。
2024–2025 kernel 5.20+ 移除 cgroup v2 mount options 的 no 前綴形式（如 nofavordynmods、nonsdelegate）。新版只保留正向開關，更乾淨的 API。
2024-09 cgroup.kill 進入主流穩定使用，systemd-oomd 與 Kubernetes evict 機制都改用此介面，比逐 PID kill 更可靠（避免 fork bomb 倖存）。
2025 CLONE_INTO_CGROUP（kernel 5.7+）已成主流容器 runtime 預設方式。runc 1.1+、crun、podman 4.x+ 都用此 syscall 取代「先 fork 再 echo PID」的傳統做法。

2025 年起，cgroup v2 操作有了新的「現代化」標準姿勢： mount options 用 favordynmods + memory_recursiveprot 是預設，程序遷移走 CLONE_INTO_CGROUP，殺光子樹用 cgroup.kill。 本章除了教傳統 API（仍要懂，看舊系統會用到），也帶讀者熟悉這些新介面。

03 · Core Concepts

核心內容

3.1 cgroup2 filesystem 掛載

cgroup v2 透過偽檔案系統（pseudo-filesystem）暴露介面。掛載指令很短： 

# 最簡形式
mount -t cgroup2 none /sys/fs/cgroup

# 帶 mount options
mount -t cgroup2 -o nsdelegate,memory_recursiveprot none /sys/fs/cgroup

這個 filesystem 的 magic number 是 0x63677270（ASCII "cgrp"）。 實務上你不會手動掛載——systemd 在開機時自動處理，並依照 distro policy 帶上適當的 mount options。 Ch.1 學的 stat -fc %T /sys/fs/cgroup 是檢查掛載結果的標準方式。

v1 v2 的相容性：v2 系統上仍可用 v1 介面，但這需要把對應的 controller 從 v2 「退掛」才能在 v1 重新使用——實務上強烈不建議混用。

3.2 Mount options 完整解析

v2 的 mount options 是系統全域設定，影響整個 cgroup tree 的行為。每個都有自己的應用場景與取捨：

Mount option Kernel 作用 建議
nsdelegate 4.15+ 把 cgroup namespace 視為委派邊界。namespace 內的 process 不能寫 namespace root 的 controller files、不能跨越 namespace 移動 process。 幾乎所有現代系統應啟用，是 rootless container 必要前提
favordynmods 6.0+ 優化動態修改（task migration、controller on/off）的延遲，代價是 fork/exit 變慢。 容器密集啟動的 host 建議啟用；長執行 process 為主的 host 不需要
memory_recursiveprot 5.7+ 讓 memory.min / memory.low 自動沿子樹遞迴生效，不需手動向下傳遞。 強烈建議啟用，多數新 distro 預設已開
memory_localevents 5.2+ 讓 memory.events 只算自己這個 cgroup，不含子樹。v1 行為。 只在遷移 v1 應用時才用；新系統別碰
memory_hugetlb_accounting 6.4+ 讓 HugeTLB 記憶體使用計入 memory cgroup 的總量。 跑 DB 或大頁面 workload 時需要，但要小心 SIGBUS 風險
pids_localevents 6.6+ 讓 pids.events.max 只算自己這個 cgroup（v1 行為）。 只在遷移時用

各 distro 的 mount options 預設

在現代 distro 上，跑 mount | grep cgroup2 通常會看到： 

# Ubuntu 22.04+ / RHEL 9+ / Fedora 31+
cgroup2 on /sys/fs/cgroup type cgroup2 (rw,nosuid,nodev,noexec,relatime,
    seclabel,nsdelegate,memory_recursiveprot)

# Oracle Linux UEK8 (2025+) 還會加上：
cgroup2 on /sys/fs/cgroup type cgroup2 (rw,nosuid,nodev,noexec,relatime,
    seclabel,nsdelegate,favordynmods,memory_recursiveprot)

要修改 mount options 不需要重啟，remount 即可： 

# 加入 favordynmods
sudo mount -o remount,favordynmods /sys/fs/cgroup

# 注意：mount options 是 host-wide 影響，不能對子目錄單獨設定

但要持久化，需要修改 /etc/fstab 或 systemd 的 cgroup_v2_mount_options 配置。

3.3 Interface files 命名規範

v2 的一大進步是規範化 interface 命名。kernel 文件 §4 "Interface Files" 明確規定：

  • core 檔案：以 cgroup. 開頭，例如 cgroup.procs、cgroup.subtree_control。
  • controller 檔案：以 controller 名字開頭，後接點與屬性名，例如 cpu.weight、memory.max。
  • weight 介面：必為 {controller}.weight，範圍固定 1–10000，預設 100。
  • limit 介面：絕對上限用 max，best-effort 用 high。
  • protection 介面：絕對保證用 min，best-effort 用 low。
  • 特殊值：「無限大」一律寫 max，不是 -1、不是大數字。
  • 單位：時間預設微秒（μs）；其他單位需要明確 suffix。
  • 百分比：使用十進位小數（如 13.40），至少兩位小數。

這套規範讓 systemd、Kubernetes、Docker 這些 cgroup 使用者只要學一次規則就能套用所有 controllers。 相比 v1 的 memory.limit_in_bytes（需要寫巨大數字）、cpu.shares（無上下界）、blkio.throttle.read_bps_device（多級命名混亂），v2 友善太多。

3.4 Core Interface Files 詳解

Ch.2 速覽過五個關鍵檔案，這裡完整列出 v2 提供的 core interface files：

檔案 類型 作用
cgroup.controllers R 列出可用 controllers（受 parent 限制）
cgroup.subtree_control RW 啟用 / 停用對 child 生效的 controllers（+name 啟用，-name 停用）
cgroup.procs RW 讀：列出 PID；寫：遷入 process（一次一個 PID）
cgroup.threads RW 類似 procs，但操作 TID（thread ID）。只在 threaded subtree 內可寫
cgroup.type RW 顯示型態：domain/domain threaded/threaded/domain invalid
cgroup.events R + poll 事件：populated（子樹是否有 process）、frozen（是否凍結）
cgroup.stat R 統計：nr_descendants、nr_dying_descendants 等
cgroup.freeze RW 寫 1 凍結整個子樹的 process（SIGSTOP-like），寫 0 解凍
cgroup.kill W 寫 1 一次殺光整個子樹的所有 process（v2 獨有的強大 API）
cgroup.max.depth RW 限制子樹深度（防止 fork bomb 用無限深度）
cgroup.max.descendants RW 限制子樹中 cgroup 數量上限
cgroup.pressure RW 啟用 / 停用 PSI 監控（在這個 cgroup 是否計算 pressure metrics）

3.5 程序遷移的三種方法

把 process 放進 cgroup 是最基本的操作。v2 提供三種方式，各有適用場景：

方法 1：寫入 cgroup.procs（傳統）

# 把 PID 1234 移到 /sys/fs/cgroup/myapp
echo 1234 | sudo tee /sys/fs/cgroup/myapp/cgroup.procs

# 等同於寫 /proc/$PID/cgroup
echo "0::/myapp" | sudo tee /proc/1234/cgroup

重要規則：單次 write(2) 只能寫一個 PID。要遷移多個必須一個個寫。 kernel 文件原話：「Only one process can be migrated on a single write(2) call.」

遷移單一 thread 時，整個 process 的所有 thread 一起被移走，除非在 threaded subtree 內。

方法 2：CLONE_INTO_CGROUP（現代）

kernel 5.7+（2020 年）加入 CLONE_INTO_CGROUP flag，讓新 process 在 fork 時直接「生在」目標 cgroup，省去「fork → 寫 PID → 變遷」的競態。 

// 程式碼示例（C）
struct clone_args args = {
    .flags = CLONE_INTO_CGROUP,
    .cgroup = target_cgroup_fd,  // 目標 cgroup 的 fd
};
pid_t pid = clone3(&args, sizeof(args));

這是 runc 1.1+、crun、systemd v245+ 採用的方式。對使用者透明，但對 container 啟動性能有顯著提升（避免 race）。 kernel 文件提到：「The static usage pattern of creating a cgroup, enabling controllers, and then seeding it with CLONE_INTO_CGROUP is not affected by [favordynmods] option.」 意即用 CLONE_INTO_CGROUP 的場景不需要 favordynmods 也很快。

方法 3：systemd-run（高階）

# 用 systemd-run 啟動一個 transient unit，自動處理 cgroup
sudo systemd-run --slice=user.slice --unit=myapp \
    -p MemoryMax=1G \
    -p CPUWeight=200 \
    /usr/bin/myapp

# 看實際用了哪個 cgroup
systemctl show -p ControlGroup myapp.service

這是最推薦的「應用層」做法——你不需要自己管 cgroup tree，systemd 會處理路徑、權限、清理。 production 上幾乎都這樣做，下章 Delegation 與 Ch.15 systemd 整合會深入。

3.6 v2 獨有：cgroup.kill 與 cgroup.freeze

這兩個是 v2 引入的無 v1 等價物的強大 API，值得單獨講。

cgroup.kill：原子化殺光整個子樹

v1 殺一個 cgroup 內所有 process 的傳統做法是讀 cgroup.procs 然後 kill -9 每個 PID。 但這有 race condition：讀完 procs 之後，被殺的 process 可能 fork 出新的 child，新的 child 繼承 cgroup 但沒被殺到——典型 fork bomb 的倖存模式。

v2 的 cgroup.kill 解決這問題：寫 1 進去，kernel 原子化地對整個子樹送 SIGKILL，包含當下這刻 fork 出的新 child。 

# 殺光 myapp 子樹中所有 process
echo 1 | sudo tee /sys/fs/cgroup/myapp/cgroup.kill

# 之後 cgroup 應該變空
cat /sys/fs/cgroup/myapp/cgroup.events
# populated 0

systemd-oomd、Kubernetes Pod eviction、container runtime 的 stop 流程都已經改用這個介面。

cgroup.freeze：暫停整個子樹

類似 SIGSTOP，但作用於整個子樹的所有 process，且 kernel 內部用更高效的機制（不會被 SIGCONT 解凍）。 

# 凍結 myapp 子樹
echo 1 | sudo tee /sys/fs/cgroup/myapp/cgroup.freeze

# 看 events 確認
cat /sys/fs/cgroup/myapp/cgroup.events
# frozen 1

# 解凍
echo 0 | sudo tee /sys/fs/cgroup/myapp/cgroup.freeze

典型用途：debug（凍結後 attach gdb）、CRIU process checkpoint/restore、安全暫停可疑 workload。

04 · Hands-on

實戰範例：完整工作流程

下面四個 lab 是一個完整的 cgroup 操作流程：建立、配置、放 process、觀察、殺光、清理。所有指令在 systemd-managed Linux 上可直接執行（需要 root）。

Lab 1 · 確認目前 mount options
# 1. 看 cgroup2 是怎麼掛的
mount | grep cgroup2

# 範例輸出：
# cgroup2 on /sys/fs/cgroup type cgroup2 (rw,nosuid,nodev,noexec,relatime,
#   seclabel,nsdelegate,memory_recursiveprot)

# 2. 確認各個 option 是否啟用
for opt in nsdelegate favordynmods memory_recursiveprot memory_localevents; do
    if mount | grep -q "cgroup2.*$opt"; then
        echo "  $opt: ENABLED"
    else
        echo "  $opt: disabled"
    fi
done

# 3. 如果要試 favordynmods（runtime 加，重啟會失效）
sudo mount -o remount,favordynmods /sys/fs/cgroup
mount | grep cgroup2  # 確認加上去了

觀察點：你的 distro 預設了哪些？對應到本章 §3.2 表格，看看 distro 維護者選擇符合預期嗎？

Lab 2 · 完整建立、配置、用一個 cgroup
# 1. 建立 cgroup
sudo mkdir /sys/fs/cgroup/lab2

# 2. 看 root 已對 child 啟用了哪些 controllers
cat /sys/fs/cgroup/cgroup.subtree_control

# 3. 在 lab2 啟用 +cpu +memory 對 lab2 的 child（注意：這是給 future child 用的）
echo "+cpu +memory" | sudo tee /sys/fs/cgroup/lab2/cgroup.subtree_control

# 4. 在 lab2 下建一個 leaf
sudo mkdir /sys/fs/cgroup/lab2/leaf

# 5. 對 leaf 設限：CPU max 50% 一核、memory max 100M
echo "50000 100000" | sudo tee /sys/fs/cgroup/lab2/leaf/cpu.max
echo "100M" | sudo tee /sys/fs/cgroup/lab2/leaf/memory.max

# 6. 在 leaf 啟動一個 yes process（吃 CPU）
sudo bash -c '(echo $$ > /sys/fs/cgroup/lab2/leaf/cgroup.procs; exec yes >/dev/null)' &
TEST_PID=$!

# 7. 觀察：這個 process 應該被限制在 50% 一核
top -p $TEST_PID
# 預期 CPU 使用率約 50%

# 8. 看 cpu.stat 統計（包含 throttle 次數）
cat /sys/fs/cgroup/lab2/leaf/cpu.stat
# 重點看 nr_throttled 與 throttled_usec

# 9. 清理
sudo kill $TEST_PID
sudo rmdir /sys/fs/cgroup/lab2/leaf
sudo rmdir /sys/fs/cgroup/lab2

觀察點：第 7 步的 50% CPU 是怎麼來的？提示：cpu.max 格式 $MAX $PERIOD，50000μs / 100000μs = 0.5 一核。

Lab 3 · 用 cgroup.kill 處理失控 process
# 1. 建立一個 cgroup
sudo mkdir /sys/fs/cgroup/lab3

# 2. 啟動一個會 fork 子 process 的 shell
sudo bash -c '
    echo $$ > /sys/fs/cgroup/lab3/cgroup.procs
    # 模擬 fork bomb 的友善版本
    for i in 1 2 3 4 5; do
        sleep 999 &
    done
    wait
' &
SHELL_PID=$!

sleep 1

# 3. 看 lab3 內有幾個 process
echo "Process 數量："
wc -l < /sys/fs/cgroup/lab3/cgroup.procs

# 4. 用 cgroup.kill 一次殺光（傳統方式要 loop kill 每個 PID）
echo 1 | sudo tee /sys/fs/cgroup/lab3/cgroup.kill

# 5. 確認都死光
sleep 0.5
cat /sys/fs/cgroup/lab3/cgroup.events
# 預期看到：
#   populated 0

# 6. 清理
sudo rmdir /sys/fs/cgroup/lab3

觀察點：第 4 步只有一行指令就把整棵樹清乾淨。這比 v1 時代的「讀 procs → 逐個 kill → 再讀 procs → 確認」可靠太多。

Lab 4 · 體驗 cgroup.events 的 inotify 通知
# Terminal 1：監聽 events
sudo mkdir /sys/fs/cgroup/lab4

# 用 inotifywait 監聽 modify 事件
inotifywait -m /sys/fs/cgroup/lab4/cgroup.events &
WATCH_PID=$!

# Terminal 2 / 同 terminal 等 5 秒：放一個 process 進去
sleep 5
sudo bash -c '(echo $$ > /sys/fs/cgroup/lab4/cgroup.procs; sleep 3) &'

# 預期觀察到：
#   /sys/fs/cgroup/lab4/cgroup.events MODIFY
# 因為 populated 從 0 變 1

# 等 sleep 結束（process 退出）
sleep 5
# 應該再次觀察到 MODIFY 事件，populated 從 1 變 0

# 清理
kill $WATCH_PID
sudo rmdir /sys/fs/cgroup/lab4

這個機制是 systemd-oomd、cAdvisor 等服務的基礎——它們不 polling，而是 subscribe events，省 CPU 又即時。

05 · Pitfalls

常見陷阱

陷阱 1 · 寫 cgroup.procs 一次只能一個 PID

最常見的錯誤：echo "1234 5678 9012" > cgroup.procs。kernel 只會把第一個 PID（1234）移進去，後面的會被忽略，且沒有錯誤訊息。 要遷移多個必須一個一個 echo： 

# 對的做法
for pid in 1234 5678 9012; do
    echo $pid | sudo tee /sys/fs/cgroup/myapp/cgroup.procs
done
陷阱 2 · favordynmods 不是萬靈丹

favordynmods 對「頻繁建立短命 cgroup」場景（如 Kubernetes 大量 Pod 啟停）有顯著效益，但它讓 fork/exit 變慢。 如果你的 host 是「少量長執行 process」（如資料庫主機），啟用 favordynmods 反而會讓主要 workload 變慢。 不要無腦啟用，先看 workload pattern。

陷阱 3 · memory_localevents 是「v1 行為相容」

如果你看到舊文件建議「啟用 memory_localevents 才能正確監控」，那是錯的——這個 option 是讓行為退回 v1 的相容開關，新系統不該啟用。 v2 的預設行為（含子樹的事件）才是設計意圖。只有在遷移 v1 時為了相容才考慮開啟。

陷阱 4 · cgroup.subtree_control 啟用條件錯誤

想啟用 controller 寫 +cpu 進 cgroup.subtree_control，但拿到錯誤 Invalid argument 或 Device or resource busy。常見三種原因：

  1. parent 沒給：cpu 不在 cgroup.controllers（parent 沒啟用 cpu）
  2. 有 process 在 cgroup 裡：違反 no internal process constraint
  3. language ambiguity：你想啟用 this cgroup 自己受控，但 subtree_control 是「對 child 啟用」，要在 parent 操作
陷阱 5 · cgroup.kill 殺不掉 zombie

cgroup.kill 對 zombie process 沒用——zombie 已經沒在執行，等 parent reap。 如果 cgroup 裡只剩 zombie，cgroup.events 的 populated 已經是 0，rmdir 也能成功（v2 規定 zombie-only 算空）。 但 /proc/$ZOMBIE_PID/cgroup 還是會顯示原 cgroup path，有 (deleted) 標記。

陷阱 6 · CLONE_INTO_CGROUP 需要對應 fd 而非路徑

新 syscall 介面要求傳 cgroup_fd（透過 open() 開啟的 fd），不能傳路徑字串。 要 open("/sys/fs/cgroup/myapp", O_PATH) 取得 fd 再傳。這個介面對 process 在 cgroup namespace 內也有效——它能跨越普通檔案系統權限。

06 · Exercises

動手練習

練習 1 · 寫一個檢查 mount options 的腳本（10 分鐘）

寫 bash script 檢查當前 mount options 並給建議：

#!/bin/bash
# mount-check.sh
# 列出當前 cgroup2 mount options，並依本章建議給評分

OPTS=$(mount | grep -oP 'cgroup2.*\((\K[^)]+')
echo "當前 mount options: $OPTS"

# TODO: 實作以下檢查並輸出建議
# - nsdelegate: 應該啟用 (▲ 加分)
# - memory_recursiveprot: 應該啟用 (▲ 加分)
# - favordynmods: 看 workload，給出建議而非死板規則
# - memory_localevents: 不應該啟用 (▼ 扣分)

練習目的：把本章 §3.2 表格內化成可執行的 audit script。

練習 2 · 比較三種程序遷移方式的延遲（15 分鐘）

用 perf 或 time 量測三種方式遷移 1000 次 process 的總時間：

# 方法 A：寫 cgroup.procs（傳統）
time for i in $(seq 1 1000); do
    sleep 100 &
    PID=$!
    echo $PID | sudo tee /sys/fs/cgroup/test/cgroup.procs > /dev/null
    sudo kill $PID
done

# 方法 B：systemd-run（高階）
time for i in $(seq 1 1000); do
    sudo systemd-run --slice=test.slice --unit=t-$i \
        --quiet sleep 100 > /dev/null 2>&1
    sudo systemctl stop t-$i 2>/dev/null
done

# 方法 C：CLONE_INTO_CGROUP 需要 C 程式或新版 systemd 才能直接體驗

回答：哪個快？為什麼？對 100 萬個 Pod 的 cluster 影響有多大？

練習 3 · 用 cgroup.freeze 凍結並 attach gdb（10 分鐘）
# 1. 啟動一個示範 process
sudo systemd-run --slice=user.slice --unit=frozen-test \
    sleep 999

# 2. 取得它的 cgroup 路徑
CG=$(systemctl show -p ControlGroup --value frozen-test.service)

# 3. 取得 PID
PID=$(systemctl show -p MainPID --value frozen-test.service)

# 4. 凍結整個 cgroup
echo 1 | sudo tee /sys/fs/cgroup$CG/cgroup.freeze

# 5. 確認凍結
cat /sys/fs/cgroup$CG/cgroup.events
# 應看到 frozen 1

# 6. 嘗試 strace（會看到 process 卡住）
sudo strace -p $PID  # Ctrl+C 退出

# 7. 解凍
echo 0 | sudo tee /sys/fs/cgroup$CG/cgroup.freeze

# 清理
sudo systemctl stop frozen-test
練習 4 · 設計一個 cleanup automation（15 分鐘）

很多時候我們想對 idle 的 cgroup 自動清理。寫一個監聽 cgroup.events 的 daemon：

#!/bin/bash
# cleanup-empty-cgroups.sh
# 監聽某個 slice 下的所有 cgroup，當任何 cgroup populated 變 0 就 rmdir

WATCH_PATH="/sys/fs/cgroup/user.slice"

inotifywait -m -r --format '%w%f %e' \
    --include '.*cgroup\.events$' "$WATCH_PATH" |
while read FILE EVENT; do
    DIR=$(dirname "$FILE")
    POPULATED=$(grep -oP 'populated \K\d' "$FILE" 2>/dev/null)
    
    if [ "$POPULATED" = "0" ]; then
        # cgroup 空了，等 1 秒後嘗試 rmdir（避免 race）
        sleep 1
        rmdir "$DIR" 2>/dev/null && echo "Cleaned: $DIR"
    fi
done

07 · Self-check

自我檢查題

Q1.cgroup2 filesystem 的標準掛載指令是什麼？magic number 是多少？
掛載指令：mount -t cgroup2 none /sys/fs/cgroup。none 是 device 名稱（cgroup2 是 pseudo-fs，沒有真實 device）。 Magic number：0x63677270，ASCII 是「cgrp」。可以用 stat -fc %T /sys/fs/cgroup 確認，輸出 cgroup2fs 代表掛對。 實務上不會手動掛——systemd 開機時自動處理，且不能輕易 unmount（會卡所有 cgroup-aware 服務）。
Q2.把以下五個 mount options 對應到正確的功能：nsdelegate、favordynmods、memory_recursiveprot、memory_localevents、memory_hugetlb_accounting。
  • nsdelegate：把 cgroup namespace 視為委派邊界，提供 namespace 內的安全保護。所有現代系統應啟用。
  • favordynmods：優化動態修改延遲（task migration、controller on/off），代價是 fork/exit 變慢。容器密集啟動的 host 適用。
  • memory_recursiveprot：讓 memory.min/memory.low 自動沿子樹遞迴生效。強烈建議啟用，多數新 distro 預設已開。
  • memory_localevents：讓 memory.events 只算自己這個 cgroup 不含子樹（v1 行為），新系統不該開。
  • memory_hugetlb_accounting：把 HugeTLB 計入 memory 總量。有大頁面 workload 才需要，要小心 SIGBUS 風險。
Q3.v2 的 interface files 命名規範規定特殊值「無限大」要寫什麼？這跟 v1 有什麼不同？
v2：一律寫字串 max，例如 echo max > memory.max 表示無記憶體限制。 v1：每個 controller 自己定義特殊值——memory 用 -1、CPU shares 沒有上限只能改大、blkio 用 0 代表 no-limit……完全不一致。 v2 規範化的好處：systemd、Kubernetes 等 cgroup 使用者只要學一次「max 是無限大」就能套用所有 controllers。
Q4.v2 的 weight 介面範圍是 1–10000、預設 100。如果某個 controller 想用 0–100 範圍當 weight 可以嗎？為什麼？
不行。kernel 文件 §4 "Conventions" 明確規定：「If a controller implements weight based resource distribution, its interface file should be named "weight" and have the range [1, 10000] with 100 as the default.」 範圍是規範要求，不是建議。新加入的 controller 要走 weight 模型必須遵守。這個範圍也是經過設計的——以 100 為中心、向上向下都有 100 倍的對稱調整空間。 例外：cpu.weight.nice 是 nice 值（-20 到 +19）的形式，內部仍會轉成 1–10000 的 weight 值。
Q5.解釋 cgroup.kill 與「讀 cgroup.procs 然後 kill -9 每個 PID」的差別。為什麼前者更可靠？
原子性差異：
  • 傳統 loop kill：讀 cgroup.procs（拿到當下的 PID 清單）→ 對每個 PID 送 SIGKILL → 重讀 procs 確認 → 如果還有就重來。race 點：被殺的 process 在被殺前可能 fork 出新 child，新 child 繼承 cgroup 但不在你拿到的清單上。
  • cgroup.kill：寫 1 進去，kernel 內部用一個 lock 把整個子樹的 process 都標記為 PF_EXITING 並送 SIGKILL，包含當下這刻 fork 出的新 child。
實務影響：fork bomb 在 v1 時代很難完整殺乾淨；v2 一行指令解決。systemd-oomd、Kubernetes Pod evict、container runtime stop 都已改用 cgroup.kill。
Q6.程序遷移的三種方法（cgroup.procs、CLONE_INTO_CGROUP、systemd-run）各自的適用場景是什麼？
cgroup.procs（傳統）：手動 debug、寫測試 script、簡單的一次性操作。問題：每次 write(2) 只能一個 PID；race condition（讀 PID → 寫 → 但 process 可能已 exit）。 CLONE_INTO_CGROUP（現代）：container runtime 內部使用——runc、crun、podman。優勢：fork 與 cgroup 設置原子化，無 race；對性能敏感的場景（K8s 大量 Pod 啟動）顯著加速。需要 C 程式或新版 systemd 才能直接呼叫。 systemd-run（高階）：應用層的標準姿勢——啟動 service、作業、ad-hoc 工作。優勢：自動處理 cgroup 路徑、權限、清理；可同時設資源限制；遵守 single-writer rule。 建議：production 用 systemd-run（透過 unit file 或 transient unit），自寫工具用 CLONE_INTO_CGROUP，debug 才用 cgroup.procs。
Q7.cgroup.events 包含哪些欄位？怎麼用 inotify 監聽變化？
包含欄位（在 v2-only 的 cgroup 上）：
  • populated：0 = 子樹無 live process；1 = 有 live process
  • frozen：0 = 未凍結；1 = 已凍結（透過 cgroup.freeze）
監聽方式：對檔案做 inotifywait 監聽 MODIFY 事件，每次值變化都會觸發；也可以用 poll(2)/epoll(2) 等 fd 讀就緒。 實際應用：systemd-oomd 監聽 events 來判斷 cgroup 何時可以回收；container runtime 用來偵測容器何時完全退出（無殘留 process）；cAdvisor 用來節省 polling 成本。 範例：inotifywait -m /sys/fs/cgroup/myapp/cgroup.events，每次變化就印一行。
Q8.為什麼 kernel 要區分 cgroup.procs 與 cgroup.threads 兩個檔案？兩者的關係是什麼？
兩者操作粒度不同：
  • cgroup.procs：操作 process 粒度。寫入 PID（任一 thread 的 TID 都行）會把整個 process 的所有 thread 一起遷移。讀取顯示 process 的 PID 列表。
  • cgroup.threads：操作 thread 粒度。寫入 TID 只遷移該 thread。讀取顯示 thread 的 TID 列表。
使用限制：cgroup.threads 只在 threaded subtree 內可寫——在 domain cgroup 寫入會被拒絕。這對應 Ch.2 的 thread mode 概念。 關係：cgroup.procs 列出的 PID 集合，是 cgroup.threads 列出的 TID 集合中「每個 process 取一個代表 TID」的子集（或說 main thread）。 典型用途：domain cgroup 用 procs，threaded cgroup 用 threads。
