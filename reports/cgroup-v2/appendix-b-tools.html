Appendix B · Tools & References

工具與參考資料速查

cgroup v2 實戰常用工具速查——systemd 工具家族、觀測工具、壓測工具、BPF 工具、 最後加上權威參考資料清單（kernel docs、LWN 文章、書籍、blog）。 搭配本書 21 章正文使用，作為工程師 troubleshoot 時的 cheatsheet。

  • 實戰 tip：少寫的 gotcha 與技巧

最後一節列出權威參考資料——kernel doc、LWN articles、書籍、blog——是延伸學習的入口。

B.1 · systemd 工具家族

systemd 提供的工具

systemd 自帶完整的 cgroup 管理 CLI，是 v2 時代的事實標準——大多數工作不需要直接操作 /sys/fs/cgroup。

systemd-cgls
內建 觀測

視覺化 cgroup tree——按階層顯示所有 cgroup 與其中 process。最常用的 cgroup 探索工具。 

# 看完整系統 cgroup tree
systemd-cgls

# 看特定 slice
systemd-cgls /system.slice
systemd-cgls /user.slice

# 看 K8s Pod cgroup
systemd-cgls /kubepods.slice | head -50

# 限制深度（對大型樹）
systemd-cgls --max-level=3
tip：輸出第一行就是 cgroup 路徑（/system.slice/sshd.service）——這個 path 直接對應 /sys/fs/cgroup 下的目錄。
systemd-cgtop
內建 觀測

類似 top 的 cgroup 即時用量視圖——按 CPU / Memory / Tasks / IO 排序顯示。troubleshoot 哪個 cgroup 吃資源最強的工具。 

# 預設按 CPU% 排序
systemd-cgtop

# 按記憶體
systemd-cgtop -m

# 按 IO
systemd-cgtop --order=io

# 看遞迴的子 cgroup（看 K8s Pod 之下的 container）
systemd-cgtop -r

# 1 秒刷新一次（預設 1 秒）
systemd-cgtop -d 1
tip：要看 IO 與 Memory 數據必須 enable accounting：systemctl set-property -- "*.slice" CPUAccounting=yes MemoryAccounting=yes IOAccounting=yes。或寫入 /etc/systemd/system.conf 的 [Manager] 段預設啟用。
systemd-run
內建 執行

把命令丟進 transient unit 跑，自動建立 cgroup 並套用 limit。完全取代 cgexec。 

# 把 stress-ng 限制 1G memory + 1 CPU
systemd-run --scope -p MemoryMax=1G -p CPUQuota=100% \
    stress-ng --vm 1 --vm-bytes 800M --timeout 30s

# 跑成 service（背景，可以 systemctl status 看）
systemd-run --unit=myapp -p MemoryMax=2G \
    /usr/local/bin/myapp

# rootless（user.slice 之下）
systemd-run --user --scope -p MemoryMax=500M command

# 不少參數對應 cgroup 介面
# 完整列表：man systemd.resource-control
tip：--scope 是同步阻塞（命令結束 cgroup 清掉），--unit 是建 service（背景跑）。pipe / stdin 與 scope 配合最直覺。
systemctl set-property
內建 設定

對既有 unit 動態調整 cgroup 屬性。寫入 /etc/systemd/system/<unit>.d/ drop-in，重啟 systemd 後仍生效。 

# 動態調 service limit（運作中生效）
sudo systemctl set-property nginx.service MemoryMax=2G CPUWeight=200

# 對 user.slice 設預設限制
sudo systemctl set-property user-1001.slice TasksMax=5000 \
    MemoryMax=16G IOWeight=100

# 永久 vs 暫時
sudo systemctl set-property --runtime nginx.service CPUQuota=50%
# --runtime: 只在當前 boot 生效；不寫 = 永久（寫入 drop-in 檔）

# 看當前生效值
systemctl show nginx.service -p MemoryMax,CPUWeight,IOWeight
tip：set-property 對 正在跑 的 unit 立即生效（cgroup 介面直接更新），不需要 restart。但對於 SliceProtect= 等少數屬性需要 restart。
systemctl status & show
內建 觀測

看 unit 當前 cgroup 狀態與資源用量——status 給人類看，show 給機器解析。 

# 看 cgroup path 與當前用量
systemctl status nginx.service
# 輸出含：
#   CGroup: /system.slice/nginx.service
#   Memory: 235.2M
#   Tasks: 25 (limit: 5000)
#   CPU: 1min 23.456s

# 機器友善格式（給 script 用）
systemctl show nginx.service \
    -p MainPID,MemoryCurrent,CPUUsageNSec,TasksCurrent

# 看完整 cgroup 設定
systemctl show nginx.service | grep -E '^(Memory|CPU|IO|Tasks)'
B.2 · 觀測工具

cgroup 觀測與監控

除了 systemd-cgtop，還有更專業的監控工具——採集到 Prometheus / Grafana 是 production 標配。

cAdvisor
Google monitoring

容器與 cgroup 的標準 metrics 採集器，K8s 整合內建（kubelet 包了 cAdvisor）。Prometheus 採 cAdvisor metrics 是業界標準。 

# 獨立部署（非 K8s）
docker run -d --name=cadvisor \
    -v /:/rootfs:ro \
    -v /var/run:/var/run:ro \
    -v /sys:/sys:ro \
    -v /var/lib/docker/:/var/lib/docker:ro \
    -v /dev/disk/:/dev/disk:ro \
    --publish=8080:8080 \
    gcr.io/cadvisor/cadvisor:latest

# K8s 內訪問（每節點 kubelet 自帶）
curl -k https://<node>:10250/metrics/cadvisor | grep container_

# 重要 metrics：
# container_cpu_usage_seconds_total
# container_memory_working_set_bytes
# container_memory_rss
# container_cpu_cfs_throttled_periods_total
# container_cpu_cfs_throttled_seconds_total
# container_fs_reads_total / writes_total
# container_network_receive_bytes_total
tip：throttle 監控的關鍵 PromQL：
rate(container_cpu_cfs_throttled_periods_total[5m]) / rate(container_cpu_cfs_periods_total[5m]) > 0.05 ——即 throttle ratio > 5%。是 Go/Python service 的核心 alert。
node_exporter
Prometheus monitoring

節點層級 metrics，含 cgroup PSI、systemd unit 狀態、kernel counters。配 cAdvisor 形成完整監控。 

# 重要 metrics（cgroup-related）：
# node_pressure_cpu_waiting_seconds_total（PSI cpu）
# node_pressure_memory_waiting_seconds_total（PSI memory）
# node_pressure_io_waiting_seconds_total（PSI io）
# node_systemd_unit_state（systemd unit 狀態）
# node_cgroup_*（部分 cgroup 統計）

# 啟動參數
./node_exporter \
    --collector.systemd \
    --collector.pressure \
    --collector.cgroups
PSI 直讀
kernel 4.20+ 觀測

不需任何外部工具，直接 cat 看 cgroup 壓力——troubleshoot 即時用最快。 

# 整機 PSI
cat /proc/pressure/cpu
cat /proc/pressure/memory
cat /proc/pressure/io
# some avg10=2.34 avg60=1.50 avg300=0.85 total=12345678

# 特定 cgroup PSI
cat /sys/fs/cgroup/system.slice/<unit>/cpu.pressure
cat /sys/fs/cgroup/kubepods.slice/.../memory.pressure

# 寫入觸發器（poll 等待）
echo "some 1000000 100000" > /sys/fs/cgroup/.../memory.pressure
# (1s 窗口內 stall 100ms 觸發 poll())
iotop / pidstat
傳統 觀測

per-process IO 與 CPU 用量——排查單一 process 級別。不是 cgroup-aware 但常配合用。 

# IO usage（按 process）
sudo iotop -o     # 只顯示有 IO 的

# CPU stat per-process（5 秒一次）
pidstat -u 5

# memory（包含 RSS、PSS、Swap）
pidstat -r 5

# 與 cgroup 結合：對特定 cgroup 內 PID
cat /sys/fs/cgroup/myapp/cgroup.procs | xargs -I {} pidstat -p {} 1 1
B.3 · BPF 工具

BPF / bpftrace 工具

v2 把 device cgroup 改成 BPF program——BPF 工具是 v2 troubleshoot 的核心。也用於 PSI、scheduler、network 觀測（Ch.13 詳述）。

bpftool
linux-tools BPF

BPF 程式管理工具——看哪些 BPF 掛在哪個 cgroup、查看 device cgroup 規則（v2 不再有 devices.allow 介面）。 

# 看哪些 BPF 程式 attach 到 cgroup
sudo bpftool cgroup tree

# 看特定 cgroup 的 BPF
sudo bpftool cgroup show /sys/fs/cgroup/myapp/

# 例如看容器的 device cgroup（v2 用 BPF）
sudo bpftool cgroup show /sys/fs/cgroup/system.slice/docker-XXX.scope/

# 列所有 BPF program
sudo bpftool prog list

# 看 program 的 source（如有 BTF）
sudo bpftool prog dump xlated id 123
bcc-tools / bpftrace
iovisor tracing

動態追蹤 kernel 與應用——cgroup throttle、OOM、PSI 等都可寫一行 bpftrace 觀察。 

# 看哪些 process 觸發 OOM
sudo bpftrace -e 'kprobe:oom_kill_process { 
    printf("OOM kill: %s pid=%d\n", comm, pid); 
}'

# 看 cgroup 切換（task migration）
sudo /usr/share/bcc/tools/cgroupless

# 系統級 PSI 觸發次數
sudo bpftrace -e 'kprobe:psi_trigger_create { 
    @[comm] = count(); 
}'

# OOM 詳細：哪個 cgroup 觸發
sudo bpftrace -e 'tracepoint:oom:mark_victim { 
    printf("victim=%d\n", args->pid); 
}'
perf
linux-tools profiling

kernel + 應用 profiling——cgroup-filtered 採樣可看特定 cgroup 內的 hotspot。 

# 對特定 cgroup 採樣 30 秒
sudo perf record -G /system.slice/myapp.service -- sleep 30
sudo perf report

# 看 throttle 事件
sudo perf stat -e 'sched:sched_stat_*' \
    -a sleep 10

# 看 page fault per-cgroup
sudo perf top --cgroup=/system.slice/myapp.service -e page-faults
B.4 · 壓測工具

驗證 cgroup limit 的壓測

設好 limit 不代表生效——壓測驗證 是 production deployment 的 must。

stress-ng
deb/rpm 壓測

萬用壓測工具——CPU、memory、IO、syscall 各種 workload 都有。對 cgroup 驗證最實用。 

# CPU stress（4 worker，10 秒）
stress-ng --cpu 4 --timeout 10s

# Memory stress（吃 800MB，30 秒）
stress-ng --vm 1 --vm-bytes 800M --timeout 30s

# IO stress（純 disk write）
stress-ng --hdd 1 --hdd-bytes 1G --timeout 30s

# 在 cgroup 內驗證 limit
systemd-run --scope -p MemoryMax=500M -p CPUQuota=100% \
    stress-ng --vm 1 --vm-bytes 600M --timeout 10s
# 預期：800MB 的 vm 在 500MB limit 下應該 OOM 或 throttle

# fork bomb 驗證 pids.max
systemd-run --scope -p TasksMax=100 \
    stress-ng --fork 200 --timeout 5s
fio
deb/rpm IO 壓測

IO 壓測黃金標準——驗證 io.max、io.weight、blk-iocost 是否生效。 

# 隨機讀（4K 區塊，64 並發，30 秒）
fio --name=randread --filename=/data/test \
    --rw=randread --bs=4k --iodepth=64 --time_based --runtime=30

# 在 cgroup 內驗證 io.max
systemd-run --scope -p IOReadBandwidthMax="/dev/sda 10M" \
    fio --name=test --filename=/data/test \
    --rw=read --bs=4k --runtime=30

# blk-iocost 對比兩個 cgroup
systemd-run --slice=high.slice -p IOWeight=800 \
    fio ... &
systemd-run --slice=low.slice -p IOWeight=100 \
    fio ... &
# 觀察兩個 cgroup IO 比例
wrk / hey
github HTTP 壓測

HTTP 壓測——配合 K8s service 看 cgroup throttle 對 latency 的影響。 

# wrk：30 秒、100 連線、4 個 thread
wrk -t4 -c100 -d30s --latency http://service.example.com/api

# hey：100 並發、1000 個請求
hey -n 1000 -c 100 http://service.example.com/api

# 同時觀察 cgroup throttle
watch -n 1 'cat /sys/fs/cgroup/.../cpu.stat | grep throttle'
B.5 · OOM 與保護

OOM 工具與保護機制

v2 的 PSI 與 memory.high 帶來新的 user-space OOM management 工具——比 kernel OOM killer 更溫和、可預測。

systemd-oomd
systemd 247+ OOM

systemd 內建的 user-space OOM killer，看 PSI memory.pressure 主動殺最佔記憶體 cgroup。比 kernel OOM killer 更早介入、更精準。 

# 啟用（多數 distro 預設裝）
sudo systemctl enable --now systemd-oomd

# 對 user.slice 設定保護
sudo systemctl set-property user.slice \
    ManagedOOMMemoryPressure=kill \
    ManagedOOMMemoryPressureLimit=70%

# 看當前狀態
oomctl

# 看歷史（哪些 cgroup 被殺）
journalctl -u systemd-oomd --since "1 hour ago"
tip：systemd-oomd 看 PSI memory.pressure full avg 持續超過閾值 → 殺整個 cgroup。比 kernel OOM 殺單一 process 更乾淨——一次清空整個 user session 或 batch job。
Facebook oomd
github OOM

Facebook 開源的進階 user-space OOM killer——可程式化規則（基於 PSI、memory.events、自訂指標）。systemd-oomd 是它的簡化版。 

# 安裝（手動 build 或從 distro 包）
git clone https://github.com/facebookincubator/oomd
cd oomd && meson build && cd build && ninja

# 規則範例 /etc/oomd/oomd.conf
{
    "rulesets": [{
        "name": "user.slice protection",
        "drop-in": {
            "cgroup": "/user.slice"
        },
        "rules": [{
            "detectors": [
                {
                    "name": "memory pressure above 60",
                    "args": {"resource": "memory", "threshold": "60", "duration": "10"}
                }
            ],
            "actions": [
                {"name": "kill_by_memory_size_or_growth", "args": {}}
            ]
        }]
    }]
}

# 啟動
sudo oomd --config /etc/oomd/oomd.conf
earlyoom
github OOM

輕量級 user-space OOM killer——比 systemd-oomd 少依賴，桌面 / 嵌入式 / Alpine 環境常用。 

# 安裝
sudo apt install earlyoom    # Debian/Ubuntu
sudo dnf install earlyoom    # Fedora

# 設定（記憶體低於 10% 開始殺）
sudo systemctl edit earlyoom
[Service]
ExecStart=
ExecStart=/usr/bin/earlyoom -m 10 -s 10 -r 60

# 重啟
sudo systemctl restart earlyoom
B.6 · Legacy 工具

Legacy 工具（避免使用）

這些工具是 v1 時代的標準，現在不再推薦使用——但 legacy 系統可能還有，列出供識別與遷移。

libcgroup-tools
deprecated legacy

cgcreate / cgexec / cgclassify / cgconfigparser / cgrulesengd——v1 時代主流。Red Hat 「strongly recommends migrate to systemd-style」。v2 多數命令 silent fail 或行為不正確。 

# cgcreate → systemd-run --slice=
# OLD:  cgcreate -g cpu:/myapp
# NEW:  systemctl set-property myapp.slice CPUWeight=100

# cgexec → systemd-run --scope --slice=
# OLD:  cgexec -g cpu:/myapp /usr/bin/myapp
# NEW:  systemd-run --scope --slice=myapp.slice /usr/bin/myapp

# cgclassify → echo PID > cgroup.procs
# OLD:  cgclassify -g cpu:/myapp 1234
# NEW:  echo 1234 > /sys/fs/cgroup/myapp.slice/cgroup.procs

# cgconfigparser /etc/cgconfig.conf → systemd unit drop-in
# OLD:  /etc/cgconfig.conf
# NEW:  /etc/systemd/system/<unit>.d/cgroup.conf

# cgrulesengd → 沒等價，要應用自己進對的 cgroup
# 通常透過 systemd unit 的 Slice= 指定
建議：在 v2 系統上 sudo dnf remove libcgroup-tools（RHEL）或 apt remove cgroup-tools（Debian）強制清乾淨——避免有人不小心用到。所有用法都遷移到 systemd 命令。
B.7 · 命令速查

最常用命令一頁速查

日常 troubleshoot 的必備命令——印出來貼牆上的程度。

10 秒系統診斷 
# 1. cgroup version
stat -fc %T /sys/fs/cgroup/

# 2. 哪個 cgroup 吃最多 CPU
systemd-cgtop -n 5

# 3. 哪個 cgroup memory pressure 高
for f in /sys/fs/cgroup/system.slice/*/memory.pressure; do
    awk -v f="$(dirname $f | xargs basename)" '/full/ {print f, $0}' "$f"
done | sort -k4 -t= -nr | head -5

# 4. 看當前系統 PSI
cat /proc/pressure/{cpu,memory,io}

# 5. 查 OOM 歷史
journalctl --since "1 hour ago" | grep -i oom
查找特定容器/Pod 的 cgroup 
# Docker
CID=$(docker inspect <name> --format '{{.Id}}')
ls /sys/fs/cgroup/system.slice/docker-${CID}.scope/

# Podman rootful
ls /sys/fs/cgroup/machine.slice/libpod-${CID}.scope/

# K8s Pod（按 QoS class）
UID=$(kubectl get pod <name> -o jsonpath='{.metadata.uid}')
QOS=$(kubectl get pod <name> -o jsonpath='{.status.qosClass}')
case $QOS in
    Guaranteed)
        ls /sys/fs/cgroup/kubepods.slice/kubepods-pod${UID//-/_}.slice/ ;;
    Burstable)
        ls /sys/fs/cgroup/kubepods.slice/kubepods-burstable.slice/kubepods-burstable-pod${UID//-/_}.slice/ ;;
    BestEffort)
        ls /sys/fs/cgroup/kubepods.slice/kubepods-besteffort.slice/kubepods-besteffort-pod${UID//-/_}.slice/ ;;
esac

# 從 process 反查 cgroup
cat /proc/<PID>/cgroup
# 0::/system.slice/<path>
設定 / 修改 cgroup 屬性 
# 對 service 動態調整（立即生效）
sudo systemctl set-property nginx.service \
    MemoryMax=2G CPUWeight=200 IOWeight=100

# 對 user.slice 設預設限制
sudo systemctl set-property user.slice \
    TasksMax=10000 MemoryMax=64G

# 跑命令並限制
systemd-run --scope -p MemoryMax=1G -p CPUQuota=200% \
    --slice=test.slice /usr/local/bin/cmd

# 寫入 drop-in（永久）
sudo mkdir -p /etc/systemd/system/myapp.service.d/
echo "[Service]
MemoryMax=2G
CPUWeight=200" | sudo tee /etc/systemd/system/myapp.service.d/limits.conf
sudo systemctl daemon-reload
sudo systemctl restart myapp.service
監控與 alert 範本 
# Prometheus alert：CFS throttle
# alert: CgroupThrottling
- alert: CgroupCPUThrottling
  expr: |
    rate(container_cpu_cfs_throttled_periods_total[5m])
    / rate(container_cpu_cfs_periods_total[5m]) > 0.05
  for: 5m
  annotations:
    summary: "Container {{ $labels.pod }} CPU throttle > 5%"

# Memory pressure imminent OOM
- alert: PodMemoryPressureHigh
  expr: kube_pod_pressure_memory_full_avg10 > 5
  for: 2m
  annotations:
    summary: "Pod {{ $labels.pod }} memory pressure > 5%"

# OOM 觸發
- alert: PodOOMKilled
  expr: |
    increase(kube_pod_container_status_restarts_total[10m]) > 0
    and on(pod, namespace)
    kube_pod_container_status_last_terminated_reason{reason="OOMKilled"} == 1
  annotations:
    summary: "Pod {{ $labels.pod }} OOM killed"
