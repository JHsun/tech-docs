Memory controller 深度解析

Memory controller 是 cgroup v2 介面最豐富、設計最精細、實戰最重要的 controller。 本章拆解四階保護限制（min/low/high/max）、events 機制、swap 與 zswap 整合、 OOM group 行為、Memory Ownership 規則、stat 與 numa_stat 監控介面， 並用三模型互補與 Kubernetes Memory QoS 把所有觀念串起來。

~8,500 字 閱讀時間 32 分鐘 難度：核心 最後更新 2026-04-28
01 · Highlights

本章重點

  1. 四階保護限制：memory.min（硬保護）→ memory.low（軟保護）→ memory.high（軟限制 + throttle）→ memory.max（硬限制 + OOM）。每階對應不同的 reclaim / OOM 行為。
  2. memory.high 是 v2 的殺手鐧：v1 沒有對應介面。它讓你在到達 OOM 前就 throttle，給 monitor 機會介入——是 Kubernetes Memory QoS 的核心。
  3. Swap 三層介面：memory.swap.max（硬限）、memory.swap.high（軟限）、memory.swap.peak（峰值）。設 0 等於關 swap。
  4. memory.oom.group：寫 1 把整個 cgroup 視為「不可分割的 workload」，OOM 時要嘛全活、要嘛全死。對 Pod / Container 是黃金設定。
  5. memory.events 是監控 backbone：累計 throttle、OOM、max 事件的觸發次數，可以 inotify 監聽——systemd-oomd、Kubernetes evict 都靠這個。
02 · Latest

最新動態（2025–2026）

2025-09 Netdata 發布診斷指南：詳細說明 cgroup v2 的 tiered memory 系統，並指出 throttling 經常在 OOM 之前發生，需要監控 memory.high 與 PSI。
2025 memory.zswap 介面成熟並普及：memory.zswap.current、memory.zswap.max、memory.zswap.writeback。zswap 比 disk-backed swap 快 100 倍，配合 cgroup-level 限制成為新趨勢。
2025-09 Kubernetes Memory QoS 在 v1.34 進 Beta，預設啟用對 Burstable Pod 自動設 memory.high，搭配 memory.min 給 request 保護。
2024–2025 memory.peak 與 memory.swap.peak（kernel 5.19+）已普及。記錄 cgroup 啟動以來的最高記憶體使用，對容量規劃比 memory.current 即時值更有用。
2025 memory.reclaim 介面（kernel 5.19+）主動觸發 cgroup 內部 reclaim，用於 Pod evict、container shutdown 前的記憶體回收。systemd-oomd 內部使用。

2026 年的時點，Memory controller 已經是 cgroup v2 介面最豐富、最完整的 controller，介面數量遠超 cpu。 對 Kubernetes / 容器 / multi-tenant 機器來說，把 memory 介面用對是性能與穩定的關鍵。 本章篇幅是全書最長之一，但讀完你會掌握生產環境記憶體管理的所有必要知識。

3.2 四階保護限制：memory.{min, low, high, max}

v2 把記憶體分配的所有需求歸納為四個層級，從最寬鬆到最嚴格：

圖 8.1 · memory 四階保護限制與對應行為

memory.max（硬上限）

cgroup 最多能用的記憶體 bytes。超過 → kernel 嘗試 reclaim → reclaim 不夠就觸發 OOM killer 在 cgroup 內挑 process 殺掉。 類似 v1 的 memory.limit_in_bytes，但語意更乾淨。 

# 設 1GB 硬上限
echo "1G" > memory.max

# 解除限制
echo "max" > memory.max

memory.high（軟上限 + throttle）

v2 的殺手鐧，v1 沒有對應介面。超過 high → kernel 進入「強力 reclaim 模式」並 throttle 該 cgroup 的 process，但絕不觸發 OOM killer。 kernel 文件原話：「memory.high is the main mechanism to control memory usage. Over-committing on high limit (sum of high limits > available memory) and letting global memory pressure to distribute memory according to usage is a viable strategy. Because breach of the high limit doesn't trigger the OOM killer but throttles the offending cgroup, a management agent has ample opportunities to monitor and take appropriate actions such as granting more memory or terminating the workload.」

實務意義：memory.high 給你一個「緩衝區」，讓監控系統有機會介入，不會像 max 一樣直接 OOM。

memory.low（軟保護）

cgroup 用量低於 low 時，受到 reclaim 保護——除非整個系統都受到嚴重壓力，否則 kernel 不會從這個 cgroup 回收記憶體。 是 best-effort 保護，不像 min 是絕對。

memory.min（硬保護）

最強保護。cgroup 用量低於 min 時，reclaim 絕對不會碰。即使全系統都缺記憶體，kernel 寧可 OOM 其他 cgroup 也不會碰受保護額度。 kernel 文件警告：「If a memcg's memory.min setting causes the parent to be unable to satisfy its own memory.min, the parent's memory.min becomes effective only up to what's available.」——min 加總不能超過 parent 能給的，否則部分保護會失效。

四階介面互動範例

# 典型的 Burstable Pod（K8s 實際做法）
# Pod request=512Mi, limit=2Gi
echo "512M" > memory.min     # request 量受硬保護
echo "1843M" > memory.high   # request + 0.9*(limit-request) ≈ 90% 軟煞車
echo "2G" > memory.max       # limit 是最後防線

# 行為：
# 用量 < 512M     → 完全不被 reclaim
# 512M-1843M      → 一般情況不被 reclaim
# 1843M-2G        → throttle + 強力 reclaim
# > 2G            → reclaim 失敗就 OOM

3.3 memory.events：事件監控

memory.events 是 v2 的監控核心。每次觸發某個閾值就增加對應計數，可以用 inotify 監聽變化： 

$ cat /sys/fs/cgroup/myapp/memory.events
low 0
high 234
max 12
oom 1
oom_kill 3
oom_group_kill 0

欄位解讀：

欄位觸發時機
lowcgroup 因系統壓力被迫從受保護的 memory.low 額度中 reclaim 的次數
highcgroup 用量超過 memory.high 後 throttle 的次數
maxcgroup 用量超過 memory.max 的次數（緊接著會 reclaim 或 OOM）
oomcgroup 觸發 OOM 的次數（即將殺 process）
oom_kill實際被 OOM killer 殺掉的 process 總數
oom_group_kill因 oom.group=1 而整個 cgroup 被殺的次數（kernel 5.10+）

events 與 events.local 的差別

memory.events 是遞迴累計——包含所有子樹中發生的事件。 memory.events.local 是只算自己這個 cgroup 的事件。 實務上：監控 Pod 用 events（含 container 內所有事件）；監控具體 container 用 events.local。

用 inotify 監聽事件

# 寫一個 daemon 監聽 OOM 事件，發現就 alert
inotifywait -m /sys/fs/cgroup/myapp/memory.events |
while read path event filename; do
    OOM_KILL=$(grep oom_kill /sys/fs/cgroup/myapp/memory.events | awk '{print $2}')
    if [ "$OOM_KILL" -gt "$LAST_OOM_KILL" ]; then
        # 觸發 alert
        send_alert "OOM detected in /sys/fs/cgroup/myapp"
    fi
    LAST_OOM_KILL=$OOM_KILL
done

systemd-oomd、Kubernetes kubelet、cAdvisor 都用類似機制——inotify + read events 比 polling 高效得多。

3.4 Swap 與 zswap：分層 swap 控制

v2 把 swap 從 v1 的 memory.memsw.* 模型徹底重做。v2 中 memory 與 swap 是分開計量的——這是巨大改進。

memory.swap.{current, high, max, peak}

# 限制這個 cgroup 最多使用 1GB swap
echo "1G" > memory.swap.max

# 完全禁用 swap（latency-sensitive 服務常用）
echo "0" > memory.swap.max

# 軟限：超過後 throttle 但繼續 swap
echo "512M" > memory.swap.high

# 看歷史峰值
cat memory.swap.peak

Meta 的生產經驗（fbtax2 案例）：把主 workload 的 swap 設 0，避免 latency 受 disk swap 影響；但其他系統 binary 開放 swap，最大化記憶體利用率。這個策略只在 v2 才能實現——v1 的 memory+swap 合併計量無法分開設定。

zswap：壓縮 swap

zswap 是記憶體中壓縮的 swap pool——把要 swap 的 page 壓縮後放在記憶體裡，比真的寫到 disk 快 100 倍。 kernel 5.19+ 提供 cgroup-level zswap 介面： 

# 限制這個 cgroup 最多用 256M 記憶體做 zswap
echo "256M" > memory.zswap.max

# 看當前 zswap 用量（已壓縮後的大小）
cat memory.zswap.current

# 禁用 zswap 寫回 disk（讓它變成「記憶體 only swap」）
echo "0" > memory.zswap.writeback

實務應用：在 RAM 充足但偶爾尖峰的機器上，啟用 zswap 給 burst 用，比 disk swap 友好得多。

3.5 memory.oom.group：群體 OOM

v2 引入的另一個關鍵創新。寫 1 進去，告訴 kernel：「這個 cgroup 是不可分割的 workload，OOM 時要嘛全活、要嘛全死」。 

# 把整個 cgroup 標記為 atomic
echo 1 > memory.oom.group

為什麼要這樣？典型場景：一個 Pod 有 N 個 container 互相依賴，其中一個被 OOM 殺了，剩下的會半死不活、無法完成工作。 設 oom.group=1 後，kernel 會整個 cgroup 一起殺——觸發 oom_group_kill 事件——讓 orchestrator（K8s）能乾淨地重啟整個 Pod。

Kubernetes 對 Pod cgroup 預設啟用 oom.group=1，這是為什麼 OOMKilled 事件是「整個 Pod 重啟」而非「某個 container 倖存」。

判讀 memory.stat 的小撇步

  • anon > 80% 總用量：應用本身吃記憶體，要看是不是 leak
  • file > 50% 總用量：page cache 大，可能是 IO-heavy workload
  • pgmajfault 非零且增長快：在 swap 或 page cache thrashing，性能會很差
  • pgsteal > 0：cgroup 在被 reclaim，可能 memory.high 設太低

3.7 memory.numa_stat：NUMA 感知

在多 socket / NUMA 機器上，記憶體 locality 對性能影響巨大。memory.numa_stat 顯示 per-NUMA-node 的細分用量： 

$ cat /sys/fs/cgroup/myapp/memory.numa_stat
anon N0=268435456 N1=256000000
file N0=134217728 N1=134217728
unevictable N0=0 N1=0
...

如果某個 cgroup 只在 NUMA 0 跑，但記憶體分散在 N0、N1，每次跨 socket access 都有延遲（remote NUMA）。 搭配 cpuset.mems（Ch.9）綁特定 NUMA node 是常見優化。

3.8 memory.reclaim：主動回收

kernel 5.19+ 加入的主動 reclaim 介面（write-only）： 

# 主動回收 100MB（盡力而為）
echo "100M" > memory.reclaim

# 嘗試成功則寫入 syscall 立即返回；失敗則回傳錯誤

典型用途：

  • 容器 shutdown 前清理：減少 graceful kill 後 kernel reclaim 的開銷
  • 準備 over-commit：在新 workload 進來前主動釋放記憶體
  • systemd-oomd 內部：嘗試 reclaim 在轉成 kill 之前

3.9 Memory Ownership 規則

kernel 文件 §5.2.3 "Memory Ownership" 規定一個重要原則：

A memory area is charged to the cgroup which instantiated it and stays charged to the cgroup until the area is released. Migrating a process to a different cgroup doesn't move the memory usages that it instantiated while in the previous cgroup to the new cgroup.

翻譯成白話：記憶體跟著「創建它的 cgroup」走，process 搬家時記憶體不跟著搬。 這對 Ch.2 「organize once and control」原則是直接呼應——頻繁搬 process 會讓記憶體統計失真。

實務影響：

  • 把 process 從 A 搬到 B，A 的 memory.current 不會降，B 的不會增（除非 process 又分配新記憶體）。
  • shared memory（tmpfs）按「第一個 touch 該頁的 cgroup」計量。
  • process 結束後，它創建的 page 仍計在原 cgroup（直到實際被釋放或 reclaim）。

3.10 systemd 介面對應

systemd directivecgroup 介面備註
MemoryMin=memory.min硬保護下限
MemoryLow=memory.low軟保護下限
MemoryHigh=memory.high軟限 + throttle
MemoryMax=memory.max硬限 + OOM
MemorySwapMax=memory.swap.maxswap 硬限
MemoryZSwapMax=memory.zswap.maxzswap 硬限（systemd 254+）
OOMPolicy=killmemory.oom.group=1群體 OOM
StartupMemoryMax=memory.max啟動階段的限制（高），啟動後切回 MemoryMax

systemd 接受多種單位寫法：1G、1.5G、500M、20%（佔總實體記憶體百分比）。

04 · Hands-on

實戰範例

Lab 1 · 觀察 memory.high throttle 行為
# 用 systemd-run 限制 memory.high=100M，觀察緩慢分配 200MB 的反應
sudo systemd-run --slice=user.slice --unit=mem-throttle \
    -p MemoryHigh=100M \
    -p MemoryMax=300M \
    --pty \
    python3 -c '
import time
data = bytearray(0)
for i in range(20):
    data.extend(bytearray(10 * 1024 * 1024))  # 每次 +10MB
    print(f"allocated {(i+1)*10}MB")
    time.sleep(0.5)
print("done")
'
# 預期觀察：超過 100MB 後變得明顯緩慢（throttle）
# 但不會被殺，最終跑完 200MB

# 在另一 terminal 看 events
CG=$(systemctl show -p ControlGroup --value mem-throttle.service)
watch -n 0.5 "cat /sys/fs/cgroup$CG/memory.events"
# 預期看到 high 計數持續增長，oom_kill 保持 0

# 清理
sudo systemctl stop mem-throttle 2>/dev/null

觀察點：memory.high 是「緩衝」機制——讓 process 變慢但活著，給 monitor 介入的時間。

Lab 2 · 體驗 memory.oom.group 群體殺
# 建立一個 cgroup 啟用 oom.group
sudo mkdir /sys/fs/cgroup/lab8
echo 1 | sudo tee /sys/fs/cgroup/lab8/memory.oom.group
echo "100M" | sudo tee /sys/fs/cgroup/lab8/memory.max

# 啟動三個 process（一個吃記憶體，兩個睡覺）
sudo bash -c '
echo $$ > /sys/fs/cgroup/lab8/cgroup.procs
sleep 999 &
sleep 999 &
PIDS=$!
wait
' &

# 啟動吃記憶體的 process（會觸發 OOM）
sudo bash -c '
echo $$ > /sys/fs/cgroup/lab8/cgroup.procs
python3 -c "data = bytearray(200 * 1024 * 1024)"
'

# 立刻看 memory.events
cat /sys/fs/cgroup/lab8/memory.events

# 預期：oom_group_kill > 0
# 同時所有 sleep process 也都被殺（不只吃記憶體那個）
# 對比：如果 oom.group=0，只有吃記憶體那個會被殺，sleep 還活著

# 清理
sudo rmdir /sys/fs/cgroup/lab8
Lab 3 · 完整 Burstable Pod 配置（K8s Memory QoS 模擬）
# 模擬 Pod: request=256Mi, limit=1Gi
# K8s Memory QoS 公式：
#   memory.min  = request = 256M
#   memory.high = request + 0.9 * (limit - request) = 256 + 691 = 947M
#   memory.max  = limit = 1G

sudo systemd-run --slice=user.slice --unit=burstable \
    -p MemoryMin=256M \
    -p MemoryHigh=947M \
    -p MemoryMax=1G \
    --pty \
    bash

# 進入 shell 後：
CG=$(systemctl show -p ControlGroup --value burstable.service)
echo "Pod cgroup: $CG"

# 看設定值
cat /sys/fs/cgroup$CG/memory.min
cat /sys/fs/cgroup$CG/memory.high
cat /sys/fs/cgroup$CG/memory.max

# 跑分配測試
python3 -c '
import time
data = bytearray(0)
for i in range(120):
    data.extend(bytearray(10 * 1024 * 1024))
    time.sleep(0.2)
'

# 觀察 events 怎麼變
cat /sys/fs/cgroup$CG/memory.events
# 預期：
#   high 多次（用量超過 947M 時觸發）
#   max 偶爾（接近 1G 時）
#   oom 0（只要 reclaim 來得及，就不 OOM）

# 清理
exit
sudo systemctl stop burstable 2>/dev/null
Lab 4 · 用 memory.reclaim 主動回收
# 跑一個吃 page cache 的程式
sudo systemd-run --slice=user.slice --unit=cache-eater \
    --pty \
    bash -c '
        # 讀大檔案讓 page cache 充滿
        dd if=/dev/zero of=/tmp/big.bin bs=1M count=500
        cat /tmp/big.bin > /dev/null
        sleep 60
    '

# 看當前用量（應該 page cache 較高）
CG=$(systemctl show -p ControlGroup --value cache-eater.service)
cat /sys/fs/cgroup$CG/memory.current
grep -E '^(file|anon)' /sys/fs/cgroup$CG/memory.stat

# 主動 reclaim 200MB
echo "200M" | sudo tee /sys/fs/cgroup$CG/memory.reclaim

# 再看用量（應該下降）
cat /sys/fs/cgroup$CG/memory.current

# 清理
sudo systemctl stop cache-eater
rm /tmp/big.bin
Lab 5 · 用 memory.peak 做容量規劃
# 跑一個有尖峰的 service
sudo systemd-run --slice=user.slice --unit=peaky \
    -p MemoryMax=2G \
    --pty \
    python3 -c '
import time
# 階段 1：低用量
data = bytearray(50 * 1024 * 1024)
time.sleep(2)
# 階段 2：尖峰
spike = bytearray(800 * 1024 * 1024)
time.sleep(1)
del spike
# 階段 3：恢復低用量
time.sleep(5)
'

# 跑完後看
CG=$(systemctl show -p ControlGroup --value peaky.service)
cat /sys/fs/cgroup$CG/memory.current  # 當前低（50M+）
cat /sys/fs/cgroup$CG/memory.peak     # 歷史最高（850M+）

# 啟示：memory.peak 比 memory.current 對容量規劃更有用
# 容器 limit 應該設 peak 的 1.5 倍以保留 burst headroom

# 清理
sudo systemctl stop peaky 2>/dev/null
05 · Pitfalls

常見陷阱

陷阱 1 · memory.high 設太低，service 慢死

最常見錯誤：「我設 memory.high=100M 防止超用」，但 workload 正常需要 200M。 結果 process 永遠在 throttle，CPU 使用率 100% 但 throughput 極低——看似活著但完全不工作。 判斷方法：memory.events 的 high 計數每秒增長 100+ 是嚴重訊號。 修法：把 memory.high 設成「正常 working set 的 1.2–1.5 倍」，太緊比太鬆糟糕。

陷阱 2 · 忘記 memory.min 不能加總超過 parent

「我給 5 個 child 各設 min=2G，總和 10G，但 parent 只有 8G」。kernel 會「按比例縮放」每個 child 的 effective min——你以為的 2G 其實只有 1.6G 受保護。 檢查方式：看 memory.events 的 low 計數，非零表示某個 cgroup 被迫從受保護額度回收。 修法：min 加總要 ≤ parent 容量，預留 headroom 給 file cache 與 kernel 開銷。

陷阱 3 · 沒設 memory.swap.max=0 但又不希望被 swap

Latency-sensitive 服務（DB、Redis）跑在開了 swap 的系統上，遇到 memory pressure 時 kernel 把熱資料 swap 出去，第一次 access 變得超慢（major page fault）。 修法：對這類 service 設 MemorySwapMax=0 完全禁用 swap（這個 cgroup）。 注意：還要監控 pgmajfault（memory.stat），非零表示已經被 swap 了。

陷阱 4 · OOM kill 後沒看 memory.events

「service 重啟很多次」但不知道為什麼？看 memory.events：

  • oom > 0：cgroup 觸發 OOM
  • oom_kill > 0：實際殺了 process
  • oom_group_kill > 0：整個 cgroup 一起被殺（K8s Pod 預設行為）
加上 dmesg 看 OOM 細節：dmesg | grep -i "killed process"。

陷阱 5 · memory.stat 的 file 算進「已用記憶體」造成誤判

看到 memory.current=900M（接近 limit 1G）就以為快 OOM？不對。看 memory.stat：如果 file 占 600M，這部分是 page cache，kernel 隨時可以 reclaim，不是真正壓力。 真正壓力來自 anon。判斷方式：anon + kernel_stack > 80% memory.max 才是危險。

陷阱 6 · zswap 不知不覺把記憶體吃光

zswap 預設沒有 cgroup 限制，所有 cgroup 共享系統 zswap pool。如果某個 cgroup 大量產生 swap pages，整個 zswap 占用上升，影響其他 cgroup 的可用記憶體。 修法：對重要 cgroup 設 memory.zswap.max，避免它成為 zswap 大戶。

陷阱 7 · 把 process 搬 cgroup 期待記憶體跟著搬

前面講過 Memory Ownership——記憶體跟創建它的 cgroup 走，搬 process 不會搬記憶體。 典型踩雷：「把 process 從 high-memory cgroup 搬到 low-memory cgroup，期望降低用量」。實際上原 cgroup 的 memory.current 不會降，新 cgroup 也不會增，process 在新 cgroup 中分配新記憶體才開始計入新 cgroup。 對策：要正確限制記憶體，啟動時就放對 cgroup。

陷阱 8 · Java/JVM 在 v2 取 memory limit 的方式不一致

OpenJDK 8u372+、11.0.16+、17+ 才支援 v2 識別。舊版 JVM 在 v2 host 上看到的是 host 的記憶體，不是 cgroup 的——導致 heap 設太大、容器 OOM 反覆發生。 進一步：JDK-8253727 bug——v2 中 memory.swap.max 只是 swap，不含 memory；JVM 早期版本沒分清楚。升 JDK 到至少 17.0.6+ / 21+ 是安全選項。

06 · Exercises

動手練習

練習 1 · 為 Redis service 設計完整記憶體配置（15 分鐘）

假設一個 4GB 機器跑 Redis，working set 約 1.5GB，希望：

  1. Redis 不會被其他 service 搶記憶體
  2. Redis 不會 swap（latency 敏感）
  3. Redis 不會吃光全機（給 OS 留 1GB）
  4. Redis OOM 時整個重啟（不要部分死）

請寫出 systemd unit 的 Memory* 設定。

練習 2 · 寫一個 OOM 監聽 daemon（15 分鐘）
#!/bin/bash
# oom-watcher.sh
# 監聽 systemd services 的 memory.events，每次 oom_kill 增加就 alert

WATCH_BASE="/sys/fs/cgroup/system.slice"

declare -A LAST_OOM

while true; do
    for cg in "$WATCH_BASE"/*.service; do
        [ -f "$cg/memory.events" ] || continue
        oom_kill=$(grep oom_kill "$cg/memory.events" | awk '{print $2}')
        last="${LAST_OOM[$cg]:-0}"
        if [ "$oom_kill" -gt "$last" ]; then
            echo "$(date) OOM in $(basename $cg): $((oom_kill - last)) processes killed"
        fi
        LAST_OOM[$cg]=$oom_kill
    done
    sleep 5
done

改造這個腳本：

  1. 用 inotifywait 取代 polling，更省 CPU
  2. 加上 oom_group_kill 監聽，區分 group kill 與單 process kill
  3. 整合 memory.peak，每次 OOM 時記下當時 peak 用量做 post-mortem
練習 3 · 比較有/無 memory.high 的反應（10 分鐘）

跑同一個逐步分配 800MB 的程式，對比兩種設定：

  1. 方案 A：只設 memory.max=500M
  2. 方案 B：memory.high=400M + memory.max=500M

記錄各自的：執行時間、是否 OOM、events 計數變化。回答：哪個更適合 latency-sensitive workload？

練習 4 · 容量規劃：用 memory.peak 訂 limit（15 分鐘）

跑你的真實 workload 一段時間（建議 24 小時，至少跑滿一個 traffic cycle），定期記錄 memory.peak：

# 每 5 分鐘記錄一次 peak
while true; do
    PEAK=$(cat /sys/fs/cgroup/myapp.service/memory.peak)
    echo "$(date +%s),$PEAK" >> /tmp/mem-peaks.csv
    sleep 300
done

分析資料：

  1. 歷史 peak 是多少？
  2. 建議 limit 設成多少？（提示：peak × 1.5 是常見公式）
  3. memory.high 設多少合理？（提示：working set 的 1.2 倍）
07 · Self-check

自我檢查題

Q1.memory.{min, low, high, max} 各自的行為是什麼？四階限制觸發後 kernel 各自做什麼？
  • memory.min：硬保護下限。用量低於 min 時 kernel絕對不 reclaim。即使全系統缺記憶體寧可 OOM 其他 cgroup 也不碰。
  • memory.low：軟保護下限。用量低於 low 時 best-effort 保護，只在系統極端壓力時被 reclaim。
  • memory.high：軟上限。用量超過 high 時 kernel 進入「強力 reclaim 模式」並 throttle process，絕不觸發 OOM。
  • memory.max：硬上限。用量超過 max 時 kernel 嘗試 reclaim，失敗就觸發 OOM killer。
記憶口訣：min 護、low 軟護、high 慢、max 死。
Q2.memory.high 解決什麼問題？v1 為什麼沒這個介面？
解決問題：v1 只有硬上限（memory.limit_in_bytes），超過就 OOM——沒有「緩衝」機會。 如果你的 workload 偶爾尖峰超過 limit，v1 直接 kill process；管理員無法在 OOM 之前介入。 v2 memory.high：超過 high 後 throttle + 強力 reclaim，絕不觸發 OOM。給管理員介入時間（granting more memory or terminating gracefully）。 v1 為何沒有：v1 設計時主要考慮「絕對限制」場景，沒設想到「動態管理」需求。v2 從 Meta、Google 等大型 production 經驗學到「tiered memory management」更實用。 實務：Kubernetes Memory QoS 把 high 設在 request + 0.9 × (limit − request)，讓 Burstable Pod 在到 limit 之前先 throttle，避免直接 OOM。
Q3.memory.events 的 oom 與 oom_kill 有什麼差別？
oom：cgroup 觸發 OOM 條件（即將 reclaim 失敗）的次數。不一定真的殺了 process——可能 kernel 在最後一刻 reclaim 成功了。 oom_kill：實際被 OOM killer 殺掉的 process總數（注意是 process 數量，不是事件次數）。一次 OOM 事件可能殺多個 process。 oom_group_kill（kernel 5.10+）：整個 cgroup 因 oom.group=1 而被殺的次數。一次 group_kill 殺掉的 process 數會反映在 oom_kill。 實務監控建議：
  • oom > 0 但 oom_kill = 0：cgroup 在邊緣，但 reclaim 救回來了
  • oom_kill 持續增長：service 反覆 OOM，要調 limit 或修 leak
  • oom_group_kill：對 K8s Pod 通常是預期行為（整個 Pod 重啟）
Q4.memory.events 與 memory.events.local 的差別？分別什麼場景用哪個？
memory.events：遞迴累計。包含這個 cgroup 自己以及所有子樹的事件。 memory.events.local：只算自己這個 cgroup 的事件，不含子樹。 場景對比：
  • 監控整個 Pod / Slice：用 events。Pod 內 container 的事件都算進來。
  • 監控具體 container：用 events.local。只看這個 container 自己。
  • 系統管理員看 system.slice 整體狀況：用 events，包含所有子 service 的累計。
  • Application 自己監控：用 events.local，只關心自己直接產生的事件。
注意：events 與 events.local 是累計值，要算 rate 必須讀兩次相減。
Q5.什麼時候應該設 memory.swap.max=0？什麼時候應該開放 swap？
應設 swap.max=0（禁 swap）的場景：
  • Latency-sensitive 服務（Redis、ElasticSearch、PostgreSQL 等 DB）：disk swap 的 latency（10ms+）對使用者體驗破壞性大
  • 實時系統 / 遊戲伺服器：絕不能容忍突然的 100x latency spike
  • 跑在 SSD、NVMe 但仍認為 swap 的 latency 不可接受
應開放 swap 的場景：
  • 非關鍵服務（log processing、batch job）：swap 比 OOM 好
  • Page cache 大但少觸碰的 workload：unused page 可以安全 swap 出
  • 跑 zswap：壓縮後在 RAM 內，速度比 disk swap 快 100 倍
Meta 經驗（fbtax2）：把主 workload swap 設 0，但系統 binary 開放 swap，最大化整體記憶體利用率，主 workload 不受 latency 影響。這個策略只在 v2 才能實現（v1 的 memsw 模型無法分開設定）。
Q6.memory.oom.group=1 的行為與目的是什麼？K8s Pod 為什麼預設啟用？
行為：寫 1 進去，告訴 kernel 把整個 cgroup 視為「不可分割的 atomic workload」。OOM 觸發時，kernel整個 cgroup 一起殺，不會只殺其中一個 process。事件記在 oom_group_kill。 目的：避免 partial kill 造成的「半死不活」狀態。例如一個 Pod 有 web server + sidecar + cache 三個 container 互相依賴，如果只殺了 web server 留下另兩個，這個 Pod 就壞掉但又不會自動重啟，會卡很久。 K8s 預設啟用的原因：
  1. Pod 是 K8s 最小調度單位，整體是 workload。
  2. Container 互相依賴常見（init container、sidecar、shared volume）。
  3. OOMKilled 觸發 K8s controller 自動重啟整個 Pod，比 partial 死了更可恢復。
反例：systemd-managed 系統的 user.slice 不建議啟用 oom.group——使用者 session 不是 atomic workload，殺光所有 user process 太激進。
Q7.memory.stat 中 anon vs file 為什麼意義不同？memory.current 接近 limit 該看哪個欄位判斷風險？
anon：匿名頁——heap、stack、mmap MAP_ANON。應用真實使用的記憶體，無法被 kernel 隨意 reclaim（除非 swap）。reclaim anon 需要寫到 swap，要不然就是殺 process。 file：file-backed 頁——page cache、mmap file。kernel 隨時可以 reclaim（dirty file 寫回後就能釋放），不算嚴格使用。 判斷風險：memory.current 接近 limit 時：
  • anon < 50% limit：file/cache 占大宗，沒問題——kernel 可以 reclaim file，不會 OOM。
  • anon > 80% limit：危險。anon 很難 reclaim（要 swap 或 OOM），接近 limit 隨時可能爆。
  • anon + kernel_stack + slab_unreclaimable > 80% limit：嚴重危險。這些都是不可回收記憶體。
正確判斷壓力的指標：搭配 memory.pressure 看 some avg10 是否大幅上升、pgmajfault 是否在增加。memory.current 數字本身不能判斷。
Q8.實作思考題：你管理一個有 32GB RAM 的機器，跑：（A）關鍵 PostgreSQL（working set 12GB）、（B）Redis cache（working set 4GB）、（C）非關鍵 batch jobs。設計完整的 memory 配置。
方案： 
[postgresql.service] - 關鍵 DB
MemoryMin=12G       # 硬保護 working set
MemoryHigh=16G      # 提早 throttle 警告
MemoryMax=20G       # 絕對天花板
MemorySwapMax=0     # 禁 swap，DB 對 latency 敏感
OOMPolicy=stop      # OOM 時整個 service 停（不要 partial）

[redis.service] - cache
MemoryMin=4G        # 保護 working set
MemoryHigh=5G       # 提早警告
MemoryMax=6G        # 防爆
MemorySwapMax=0     # 禁 swap

[batch.slice] - 非關鍵
MemoryMax=8G        # 留給 batch
MemorySwapMax=4G    # 允許 swap，可以接受變慢
# 不設 MemoryMin（它優先級低）

# 預留：32G - 12G - 4G - 8G - 4G(OS+page cache) = 4G 緩衝
設計理由：
  1. 關鍵 service 設 MemoryMin：保證 working set 不被 reclaim，性能可預測。
  2. 禁 swap 給 latency-critical：DB、cache 都關 swap。
  3. batch 允許 swap：可以接受變慢，最大化整體 utilization。
  4. 留 4GB OS buffer：給 page cache、kernel 結構、突發負載。
監控配套：
  • 對 A、B 監控 memory.pressure some avg10 < 5% 與 memory.events oom = 0
  • 對 batch 容忍 OOM（自動重試），但監控 oom_kill rate
  • 整體機器監控 /proc/pressure/memory，full avg10 > 1% 是嚴重訊號
