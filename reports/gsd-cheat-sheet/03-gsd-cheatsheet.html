🧠 設計原理 — 為什麼 GSD 有效

上下文工程（Context Engineering）

GSD 解決的核心問題是上下文腐爛（context rot）— 隨著對話變長，Claude 的上下文窗口填滿後品質退化。 GSD 的解法：所有狀態持久化為結構化 Markdown 檔案，每個子任務在全新的 200K 上下文中執行。

檔案角色何時載入
PROJECT.md專案願景與核心上下文始終載入
REQUIREMENTS.md帶 ID 的分階段需求規劃 / 驗證時
ROADMAP.md階段分解與完成狀態導航時
STATE.md決策記錄、阻塞項、跨會話記憶恢復會話時
CONTEXT.md每階段的實作偏好規劃 / 執行時
RESEARCH.md領域研究發現規劃時
PLAN.md原子執行計畫（XML 結構）執行時
SUMMARY.md執行結果與決策記錄後續階段參考
seeds/前瞻想法（對的時機自動浮出）新里程碑時
threads/持久上下文線程跨會話工作時
XML Prompt 格式

每個計畫是為 Claude 最佳化的結構化 XML：精確指示、不猜測、內建驗證。

<task type="auto">
  <n>Create login endpoint</n>
  <files>src/app/api/auth/login/route.ts</files>
  <action>
    Use jose for JWT (not jsonwebtoken - CommonJS issues).
    Validate credentials against users table.
    Return httpOnly cookie on success.
  </action>
  <verify>curl -X POST localhost:3000/api/auth/login
    returns 200 + Set-Cookie</verify>
  <done>Valid credentials return cookie,
    invalid return 401</done>
</task>
多代理編排（Multi-Agent Orchestration）

瘦編排器模式：精簡的協調器不做重活，只生成專門代理 → 等待 → 整合結果。 你的主會話上下文保持在 30-40%。

階段編排器動作代理動作
研究 協調、呈現發現 4 個並行研究員調查技術棧、功能、架構、陷阱
規劃 驗證、管理迭代 規劃器建立計畫 → 檢查器驗證 → 循環直到通過
執行 分組為波次、追蹤進度 執行器並行實作，每個有全新 200K 上下文
驗證 呈現結果、路由下一步 驗證器比對目標、除錯器診斷失敗

原子 Git 提交

每個 task 完成後立即產生獨立提交：

abc123f docs(08-02): complete user registration plan
def456g feat(08-02): add email confirmation flow
hij789k feat(08-02): implement password hashing
lmn012o feat(08-02): create registration endpoint

✅ git bisect 精確定位故障 ・ 每個 task 可獨立回滾 ・ 為未來會話提供清晰歷史

🔒 安全加固

GSD v1.27+ 內建縱深防禦（defense-in-depth）安全機制：

防護層機制
路徑穿越防護所有使用者路徑（--text-file、--prd）驗證解析在專案目錄內
Prompt 注入偵測security.cjs 掃描角色覆蓋、指令繞過、系統標籤注入等模式
Runtime Hookgsd-prompt-guard.js 掃描 .planning/ 寫入中的注入向量（諮詢性）
工作流守衛gsd-workflow-guard.js 警告 GSD 工作流外的檔案編輯（opt-in）
CI 掃描器prompt-injection-scan.test.cjs 掃描代理/命令檔中的嵌入注入
工作流名稱淨化防止透過名稱進行路徑穿越
保護敏感檔案

在 .claude/settings.json 中加入 deny 規則，防止 Claude 讀取機密：

{
  "permissions": {
    "deny": [
      "Read(.env)", "Read(.env.*)",
      "Read(**/secrets/*)",
      "Read(**/*credential*)",
      "Read(**/*.pem)", "Read(**/*.key)"
    ]
  }
}
