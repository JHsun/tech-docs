正確名稱是「Get Shit Done」，npm 套件名稱為 get-shit-done-cc，由 TÂCHES 開發。支援 Claude Code、OpenCode、Gemini CLI、Kilo、Codex、Copilot、Cursor、Windsurf、Antigravity、Augment、Trae、Qwen Code、Cline、CodeBuddy 等 15 個以上平台。

為什麼需要這套架構？ AI 語言模型（LLM）沒有跨對話的持久記憶——每次新建對話，AI 對你的專案背景一無所知。更嚴重的是 context rot：隨著 context window 填滿，Claude 的輸出品質會逐漸下降。GSD 用結構化文件系統模擬「外腦」，並以 每個計畫分配獨立的 200k token context window 解決品質降解問題。

核心設計哲學：「需要時載入，不需要時歸檔」。Orchestrator 永遠不做重活，而是派生專門的 subagent，讓主 context window 維持在 30–40%，保持快速和響應性。

製作教程：從零建立你的 .planning/
逐步說明如何初始化、填寫、維護每一層文件

前置條件：你已有一個專案目錄，並且使用 Claude（或其他 LLM）協助開發。 整個設置過程通常只需要 15–30 分鐘，之後 AI 每次對話都能直接接上你的脈絡。 以下步驟以 綠地（greenfield）新專案 為主，棕地接手舊專案見步驟 8。

01

安裝 GSD：一行指令搞定

透過 npx 安裝最新版，互動式選擇目標 runtime 與安裝位置

GSD 以 npm 套件形式發布，無需 git clone，直接用 npx 執行安裝程式：

# 安裝最新版（互動式，推薦）
npx get-shit-done-cc@latest

# 非互動式安裝（CI / Docker / Scripts 使用）
# Claude Code - 全域
npx get-shit-done-cc --claude --global

# Cursor
npx get-shit-done-cc --cursor --local

# Windsurf
npx get-shit-done-cc --windsurf --local

# 一次安裝所有支援的 runtime
npx get-shit-done-cc --all --global

安裝程式會問兩個問題：

① Runtime：Claude Code、OpenCode、Gemini CLI、Kilo、Codex、Copilot、Cursor、Windsurf、Antigravity、Augment、Trae、Qwen Code、CodeBuddy、Cline（可多選）

② 位置：Global（所有專案）或 Local（僅當前專案）

安裝完成後建立 .planning/ 目錄，以下指令一次到位：

mkdir -p .planning/{research,seeds,threads,todos,quick,spikes,sketches,intel,graphs,debug/{active,resolved,knowledge-base},forensics,reports,codebase}

完成後，你的目錄結構應該如下：

.planning/ ├── # ── 核心文件（手動建立 / /gsd-new-project 自動生成） ├── PROJECT.md ├── REQUIREMENTS.md ├── ROADMAP.md ├── STATE.md ├── MILESTONES.md ├── DISCUSSION-LOG.md ├── config.json # GSD 設定檔（/gsd-settings 管理） │ ├── research/ # 領域研究（project 級） ├── seeds/ # 帶 trigger 條件的前瞻想法 ├── todos/ # 待辦事項與臨時想法 ├── threads/ # 跨會話的長期任務線程 ├── quick/ # /gsd-quick 的 ad-hoc 任務 ├── spikes/ # v1.37 /gsd-spike 可行性實驗 ├── sketches/ # v1.37 /gsd-sketch UI 設計變體 ├── intel/ # 程式碼庫智能 JSON（intel.enabled） ├── graphs/ # v1.36 /gsd-graphify 知識圖譜 ├── debug/ │ ├── active/ │ ├── resolved/ │ └── knowledge-base/ ├── forensics/ ├── reports/ └── codebase/ # 棕地分析（/gsd-map-codebase 生成）
💡
Claude Code v2.1.88+ 使用 skills 格式，安裝在 ~/.claude/skills/gsd-*/SKILL.md；舊版本使用 commands/gsd/。安裝程式自動處理，無需手動區分。安裝後執行 /gsd-help 驗證。
⚠️
推薦使用 --dangerously-skip-permissions flag 執行 Claude Code：
claude --dangerously-skip-permissions
每次批准 date、git commit 等操作會嚴重影響效率，這是 GSD 的預設使用方式。
02

撰寫 PROJECT.md：專案的「DNA」

這是 AI 最重要的一份文件，每次對話都應載入

PROJECT.md 是 AI 的「北極星」——它定義了專案的目標、受眾、核心限制，讓每次對話都從正確的視角出發。

以下是一個範本結構：

# PROJECT: [專案名稱]

## 一句話目標
[用一句話說明這個專案要解決什麼問題]

## 背景與動機
- 為什麼要做這個？
- 解決了誰的什麼痛點？

## 核心受眾
- 主要使用者：[描述]
- 次要使用者：[描述]

## 成功定義
- [可量化的成功指標 1]
- [可量化的成功指標 2]

## 核心約束
- 技術：[例：必須使用 TypeScript, Node.js 18+]
- 商業：[例：預算、時程]
- 設計：[例：必須支援行動裝置]

## 明確的 Out of Scope
- [不做的事情 1]
- [不做的事情 2]

## 關鍵術語
- [術語 A]：[定義]
- [術語 B]：[定義]
⚠️
「Out of Scope」區塊非常重要！明確告訴 AI 不要做什麼，能防止 AI 過度發揮、加入你不需要的功能。
03

建立 REQUIREMENTS.md：帶 ID 的需求清單

每個需求都有唯一識別碼，讓驗收、追蹤、引用都有依據

需求 ID 的命名慣例：REQ-[PHASE]-[NUMBER]，例如 REQ-AUTH-001。

# REQUIREMENTS

## Phase 1: 核心認證 (AUTH)

### REQ-AUTH-001
**標題：** 使用者可以用 Email + 密碼註冊
**描述：** 系統允許新使用者建立帳號，需驗證 Email 格式與密碼強度
**驗收標準：**
- [ ] Email 需符合標準格式
- [ ] 密碼至少 8 字元，包含大小寫與數字
- [ ] 重複 Email 應顯示明確錯誤訊息
**優先級：** P0（必做）
**狀態：** 待實作

### REQ-AUTH-002
**標題：** 支援 Google OAuth 登入
**描述：** 提供 Google 第三方登入選項，降低使用者摩擦
**驗收標準：**
- [ ] 點擊「用 Google 登入」可跳轉至 OAuth 流程
- [ ] 成功後自動建立或連結帳號
**優先級：** P1（應做）
**狀態：** 待實作
✅
驗收標準使用 checkbox（- [ ]）格式，讓 VERIFICATION.md 後續可以逐一打勾確認，形成完整的驗收閉環。
04

初始化專案：/gsd-new-project

一行指令，讓 AI 問清楚你的需求，自動生成 ROADMAP 和 REQUIREMENTS

手動建立 PROJECT.md 後，執行以下指令啟動完整初始化流程：

# 標準初始化（互動式問答）
/gsd-new-project

# 自動模式（跳過確認，適合重複跑的場景）
/gsd-new-project --auto

系統執行四個步驟：

① 問題階段：AI 持續提問直到完全理解你的想法——目標、約束、技術偏好、邊界案例

② 研究階段（可選但推薦）：派生 4 個並行 agent 調查技術棧、功能、架構、陷阱

③ 需求提取：明確 v1、v2 與 out of scope 的邊界

④ Roadmap 生成：建立與需求對應的 phase 規劃

你確認 roadmap 後，系統自動產生：

.planning/ ├── PROJECT.md # 你已手動填寫 ├── REQUIREMENTS.md # ✨ 自動生成，帶唯一 ID ├── ROADMAP.md # ✨ 自動生成，phase 對應需求 ├── STATE.md # ✨ 自動生成，初始狀態 └── research/ # ✨ 研究 agent 的輸出
✅
已有程式碼？先執行 /gsd-map-codebase 掃描現有 codebase，再跑 /gsd-new-project。這樣初始化時的問題會集中在你要「新增什麼」，規劃會自動載入你的現有架構模式。

06

建立執行計畫：XX-YY-PLAN.md

把 phase 分解成精確的原子任務，每個任務都有明確的驗收標準

命名規則：02-01-PLAN.md 代表 Phase 02、第 1 個計畫。計畫使用 XML 結構讓 AI 更容易解析。

<!-- .planning/02-01-PLAN.md -->
# PLAN: P02-01 使用者認證基礎

<plan>
  <meta>
    <phase>02</phase>
    <requirements>REQ-AUTH-001, REQ-AUTH-002</requirements>
    <estimated_hours>4</estimated_hours>
  </meta>

  <task id="02-01-01">
    <title>建立 User 資料庫 Schema</title>
    <context>使用 Prisma 定義 User model，包含認證必要欄位</context>
    <instructions>
      1. 在 prisma/schema.prisma 建立 User model
      2. 必要欄位：id, email(unique), passwordHash, createdAt, updatedAt
      3. 可選欄位：name, avatarUrl, googleId（為 OAuth 預留）
      4. 執行 npx prisma migrate dev --name add-user-model
    </instructions>
    <acceptance_criteria>
      - migration 成功執行無錯誤
      - npx prisma studio 可看到 User 表格
      - email 欄位有 unique constraint
    </acceptance_criteria>
    <rollback>npx prisma migrate reset（警告：清除所有資料）</rollback>
  </task>

  <task id="02-01-02">
    <title>實作密碼雜湊工具函式</title>
    <instructions>
      1. 安裝 bcrypt：npm install bcrypt @types/bcrypt
      2. 建立 lib/auth/password.ts
      3. 實作 hashPassword(plain: string): Promise<string>
      4. 實作 verifyPassword(plain: string, hash: string): Promise<boolean>
      5. salt rounds 設為 12（見 STATE.md 技術決策）
    </instructions>
    <acceptance_criteria>
      - 單元測試通過（hashPassword + verifyPassword 往返測試）
      - 錯誤密碼應返回 false 而非拋出例外
    </acceptance_criteria>
  </task>
</plan>
07

執行後更新 SUMMARY 與 VERIFICATION

完成計畫後，讓 AI 記錄實際發生了什麼、需求是否達標

每個 PLAN 執行完成後，應該產生對應的 SUMMARY.md 和 VERIFICATION.md。

# 02-01-SUMMARY.md — 執行結果記錄

## 完成狀態：✅ 完成（2024-01-15）

## 實際執行情況
- Task 02-01-01：按計畫完成，schema 加入了 emailVerified 欄位（原計畫未列，但 OAuth 需要）
- Task 02-01-02：完成，額外加入了密碼強度驗證工具函式

## 遇到的問題與解法
- **問題**：Prisma 與 Next.js 15 的 bundler 有衝突
  - **解法**：在 next.config.js 加入 serverExternalPackages: ['@prisma/client']
  - **學到的**：Next.js 15 預設嘗試 bundle server packages，Prisma 不相容

## 對後續 phase 的影響
- P02 後續任務可以直接 import `lib/auth/password.ts`
- emailVerified 欄位需要在 REQ-AUTH 追加一個需求（Email 驗證流程）

## 建立的檔案清單
- `prisma/schema.prisma` — User model
- `lib/auth/password.ts` — 密碼工具函式
- `lib/auth/password.test.ts` — 單元測試
✅
每次執行結束後，把「遇到的問題與解法」詳細記錄下來。這些知識是最有價值的部分——下次同樣的問題出現，AI 能直接查到解法。
08

棕地專案：接手既有程式碼的特殊程序

先讓 AI 分析現有 codebase，再開始規劃新功能

接手舊專案時，不要跳過分析步驟直接開始加功能。先執行棕地映射，填寫 codebase/ 目錄，讓 AI 對既有架構有完整認識。

🔍

1. 執行 /gsd-map-codebase 指令

讓 AI 掃描整個 codebase，自動生成 STACK.md、ARCHITECTURE.md、CONVENTIONS.md

⚠️

2. 手動補充 CONCERNS.md

加入你知道的技術債、已知 bug、不能動的地雷區。AI 不一定能從程式碼中看出「這個不能改」

📋

3. 審查並補充 AI 的分析

AI 的分析可能有遺漏或誤判，人工確認後再開始規劃新功能

🚀

4. 開始正常 GSD 流程

有了 codebase/ 的基礎，後續的 RESEARCH → PLAN → EXECUTE 流程都能更準確

📥
v1.38 加碼：/gsd-ingest-docs [path] — 若你的舊專案已散落 ADR、PRD、SPEC、設計文件，這個指令會並行分類、檢測 LOCKED-vs-LOCKED 衝突，並產出三桶報告（自動解決 / 競爭變體 / 未解阻擋）一次匯入 .planning/。比手動移植快 10 倍。
09

規格不確定？先 /gsd-spec-phase 鎖 WHAT（v1.37）

Socratic 模糊度評分，讓你寫程式前先確認規格夠不夠清楚

v1.37 新增的 /gsd-spec-phase N 是 GSD 的「事前防呆」：在 discuss-phase 之前進行蘇格拉底式提問，幫每條需求打模糊度分數（ambiguity score），逼你把含糊的「使用者體驗應該很好」變成 falsifiable 的「P95 載入 < 200ms 且首屏可互動 < 500ms」。

# 在 discuss/plan 之前先跑 SPEC 釐清
/gsd-spec-phase 3
# → 系統檢視 phase 3 的需求，標出模糊段落
# → 互動式提問，每答一題模糊度分數下降
# → 達到鎖定門檻後產出 03-SPEC.md（locked）
/gsd-discuss-phase 3        # SPEC.md 自動載入為 prior context
/gsd-plan-phase 3
💡
什麼時候用 spec-phase？當 phase 涉及商業規則、UX 細節、效能指標、邊界條件這些「容易雞同鴨講」的部分。純技術 phase（例如「升級 Node.js 22」）通常可以跳過。
10

寫程式前的探索：spike ＆ sketch（v1.37）

不確定可行性或設計方向時，跑實驗比直接 plan 更省時

v1.37 把可行性實驗與 UI 探索升格為一等公民指令，且不需要先 /gsd-new-project。做完後用 -wrap-up 把結論打包成 project skill，未來會話自動載入。

# 不確定技術可行性
/gsd-spike "can we stream LLM tokens through SSE without proxy buffering"
# → 跑 2–5 個 Given/When/Then 實驗，回傳 VALIDATED/INVALIDATED/PARTIAL
# → 結果存於 .planning/spikes/NNN-name/

# UI 設計多種方向
/gsd-sketch "dashboard layout"
# → 產出 2–3 個互動 HTML mockup 變體（共用主題 CSS）
# → 瀏覽器並排比較 → 選 winner

# 結論定案，打包成 project skill
/gsd-spike-wrap-up
/gsd-sketch-wrap-up
🎯
後續整合：/gsd-discuss-phase、/gsd-plan-phase、/gsd-new-project、/gsd-next 都會自動感知並載入相關的 spike/sketch 結論——不用每次手動帶入。

GSD 指令速查表
常用的 slash commands 及其觸發的文件操作
/gsd-new-project [--auto]
完整初始化：問答 → 研究 → 需求 → roadmap。這是每個新專案的第一個指令。
使用時機：全新專案的第一步
/gsd-discuss-phase [N] [--auto] [--analyze] [--chain]
在規劃前捕捉實作決策偏好，生成 XX-CONTEXT.md。--chain 自動串接 plan+execute。
使用時機：每個 phase 規劃前，確認你的設計意圖
/gsd-plan-phase [N] [--auto]
研究 + 規劃 + 驗證。派生研究 agent，生成帶 XML 結構的原子任務計畫，並驗證計畫是否達成 phase 目標。
使用時機：discuss 完成後，開始規劃
/gsd-execute-phase <N>
以 Wave 並行執行所有計畫。每個 executor 得到獨立的 200k token context。每個 task 獨立 git commit。
使用時機：計畫就緒後，走開讓它跑
/gsd-verify-work [N]
人工驗收測試（UAT）。逐一確認功能是否如預期運作，失敗時自動派生 debug agent 診斷並生成修復計畫。
使用時機：execute 完成後的品質確認
/gsd-ship [N] [--draft]
從已驗收的 phase 建立 PR，自動生成 PR 描述內容。
使用時機：verify 通過後，準備合併
/gsd-next
自動偵測當前工作流程狀態，執行下一個邏輯步驟。不確定該做什麼時的萬用指令。
使用時機：不確定下一步時
/gsd-quick [--full] [--validate] [--discuss] [--research]
Ad-hoc 任務，跳過完整規劃流程但保留 GSD 品質保證（原子 commit、狀態追蹤）。儲存於 .planning/quick/。
使用時機：不需要完整流程的小任務
/gsd-spike [idea] [--quick]
執行 2–5 個聚焦實驗驗證可行性，每個有 Given/When/Then 結論。不需先初始化專案。
使用時機：規劃前驗證技術方向
/gsd-sketch [idea] [--quick]
生成 2–3 個互動式 HTML mockup 變體探索設計方向。不需先初始化專案。
使用時機：UI phase 前的設計探索
/gsd-map-codebase [area]
分析既有 codebase，自動生成 codebase/STACK.md、ARCHITECTURE.md、CONVENTIONS.md、CONCERNS.md。
使用時機：接手舊專案時的第一步
/gsd-complete-milestone
歸檔里程碑，打 release tag。
使用時機：所有 phase 完成後
/gsd-new-milestone [name]
開始下一個版本：問答 → 研究 → 需求 → roadmap。每個里程碑是一個完整的 define → build → ship 循環。
使用時機：milestone 完成後繼續開發
/gsd-pause-work / /gsd-resume-work
暫停時生成 HANDOFF.json 快照完整狀態；恢復時讀取並無縫接續。
使用時機：工作被中斷時
/gsd-debug [desc]
6 階段結構化除錯，結果存入 debug/knowledge-base/，下次遇到同樣問題直接查到解法。
使用時機：遇到難以快速定位的 bug
/gsd-ui-phase [N] / /gsd-ui-review [N]
生成 UI 設計契約（UI-SPEC.md）；執行 6 面向視覺審計並生成評分報告（UI-REVIEW.md）。
使用時機：UI phase 前/後
/gsd-workstreams create/switch/complete
管理並行的里程碑工作流（命名空間隔離）。適合需要同時進行多條開發線的專案。
使用時機：多功能並行開發時
/gsd-settings / /gsd-set-profile
設定 model profile（quality/balanced/budget/inherit）與 workflow agent 開關。設定存在 .planning/config.json。
使用時機：調整品質/速度/成本平衡
/gsd-health [--repair]
驗證 .planning/ 目錄完整性；加 --repair 自動修復損壞的狀態。
使用時機：工作流異常時的診斷
/gsd-forensics [desc]
工作流失敗的事後分析。診斷卡住的迴圈、遺失的 artifact、git 異常。
使用時機：AI 行為不符合預期時
/gsd-plant-seed <idea>
儲存帶 trigger 條件的前瞻想法，在對的里程碑自動浮出，不會因為「現在不是時候」就忘掉。
使用時機：有好想法但現在不該做
/gsd-review / /gsd-secure-phase [N]
跨 AI peer review 當前 phase 或 branch；security enforcement 掛接威脅模型的驗證。
使用時機：程式碼品質與安全審查
/gsd-fast <text>
行內執行瑣碎任務，完全跳過規劃。適合一行改動等微任務。
使用時機：快速的一行修正
/gsd-do <text>
自動把自由輸入的文字路由到對應的 GSD 指令。不確定該用哪個指令時的萬用入口。
使用時機：不知道該用哪個指令時
/gsd-spec-phase [N] v1.37
Socratic SPEC 釐清，含模糊度評分。在 discuss 之前先鎖定 phase 的 WHAT（falsifiable requirements）。
使用時機：規格不夠清楚就不要開始實作
/gsd-ingest-docs [path] v1.38
批次匯入混合 ADR/PRD/SPEC/DOC，並行分類，產出三桶衝突報告（自動解決 / 競爭變體 / 未解阻擋）。LOCKED-vs-LOCKED 硬性阻擋。
使用時機：已有大量規劃文件想匯入 GSD
/gsd-ultraplan-phase <N> v1.38 BETA
[Claude Code only] 把 plan-phase 卸載到 ultraplan 雲端起草，瀏覽器看 inline 評論，定案後用 /gsd-import 拉回 .planning/。
使用時機：不想佔用本地終端跑 plan-phase
/gsd-plan-review-convergence <N> v1.37
跨 AI 計畫收斂迴圈：plan → review → replan → re-review，直到 HIGH 議題歸零（最多 3 輪）。--all 平行跑所有審查者。
使用時機：關鍵 phase 想多輪審查迭代
/gsd-progress --forensic v1.37
標準進度報告 + 6 項完整性審計（STATE 一致性、孤兒 handoff、scope drift、待辦、未提交程式碼）。
使用時機：長時間沒看，怕狀態漂移
