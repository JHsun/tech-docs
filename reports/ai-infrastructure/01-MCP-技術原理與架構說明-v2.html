MCP 協議總覽

Model Context Protocol（MCP）是一個開放式、有狀態的會話協議，旨在標準化 AI 應用程式與外部工具、資料來源之間的連接方式。它由 Anthropic 於 2024 年 11 月首次發布，現由 Linux Foundation 旗下的 Agentic AI Foundation 治理。

MCP 的核心設計哲學可以用一句話概括：AI 領域的「USB-C」。它解決的核心問題是 N×M 整合問題——當 N 個工具需要與 M 個 AI 前端整合時，傳統方式需要 N×M 個獨立整合；MCP 將此降低為 N+M。

N×M 整合問題 vs MCP 統一協議
MCP 的五個設計原則

1. 能力協商——客戶端與伺服器在初始化時明確宣告各自支援的功能。
2. 安全隔離——每個 Client-Server 連線維持獨立的 Session。
3. 傳輸無關——協議邏輯與傳輸機制解耦。
4. 漸進式採用——Server 可以只實作部分功能。
5. 可擴展性——Extensions 框架允許社群試驗新功能。

2

核心架構：Host-Client-Server

MCP 遵循 Host → Client → Server 三層架構。理解這三個角色的邊界是掌握整個協議的關鍵。

MCP Host-Client-Server 三層架構

Host（宿主）

Host 是運行 LLM 的應用程式。它負責建立和管理多個 MCP Client 實例、控制 Client 可以連接的 Server 範圍、在不同 Server 之間強制執行安全邊界。

Client（客戶端）

每個 Client 由 Host 建立，與恰好一個 Server 維持 1:1 的有狀態連線。Client 負責協議協商、訊息路由、訂閱管理，以及斷線偵測。

Server（伺服器）

Server 是一個輕量級程序，透過標準化的 MCP 原語向 Client 暴露特定能力。Server 獨立運作，專注於單一職責。

JSON-RPC 2.0 通訊層

MCP 的所有通訊皆基於 JSON-RPC 2.0。MCP 在此基礎上建立有狀態的 Session，使 Client 與 Server 能在整個互動過程中維持上下文。

6

三大原語：Tools、Resources、Prompts

原語性質控制權用途典型操作
Tools 可執行模型控制的函式LLM 決定執行操作、產生副作用查詢 DB、發送訊息
Resources 可讀取結構化資料應用程式控制提供上下文讀取檔案、API 回應
Prompts 可重用提示範本使用者選擇預定義互動模式程式碼審查範本
Tasks（2025-11-25 新增，實驗性）

任何 Request 都可以回傳一個 Task Handle，讓 Client 稍後查詢狀態。Task 可在以下狀態間轉換：working → input_required → completed / failed / cancelled。

授權與安全模型

MCP 從 2025 年 3 月起強制要求遠端連線使用 OAuth 2.1 + PKCE。

Server 宣告為 OAuth 2.0 Resource Server——提供 /.well-known/oauth-protected-resource 中繼資料（RFC 9728）。
Client 發現授權伺服器——透過中繼資料取得 /.well-known/oauth-authorization-server 配置。
Client 註冊（CIMD）——2025-11-25 使用 Client ID Metadata Documents 取代動態客戶端註冊。
授權碼流程 + PKCE——生成 code_verifier/code_challenge，取得授權碼後交換存取令牌。
帶令牌存取——Authorization: Bearer {token} 存取 MCP Server。

核心技術機制

1. 傳輸轉換

JSON-RPC 訊息內容完全不變，只有傳輸封裝改變。Proxy 接收 Streamable HTTP 請求，轉為 stdio 輸入送入本地 Server 子程序。

2. 工具聚合與命名空間

攔截 tools/list，向所有後端 Server 並行發送，合併結果並加上命名空間前綴（如 github.create_issue），避免名稱衝突。

3. 能力協商代理

對 Client 宣告所有後端能力的聯集；對每個後端 Server 獨立進行能力協商。

Token 代理模式

Gateway 持有存取後端 API 的原始 OAuth Token，並向 Client 發行範圍受限的代理 Token。Client 永遠不會直接接觸後端 API 的真實憑證。

部署拓撲模式

三種部署拓撲對照
💡 架構選型建議

選擇部署拓撲時，關鍵考量因素是 Session 狀態管理。stdio Server 天然綁定到單一程序，適合單機部署。Streamable HTTP Server 支援無狀態請求模型，適合水平擴展。當 Gateway 需要管理跨多個實例的 Session 時，應使用外部狀態儲存（如 Redis）來解耦 Session 與特定 Gateway 實例。
