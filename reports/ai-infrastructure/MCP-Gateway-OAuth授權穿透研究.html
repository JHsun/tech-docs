核心問題：為什麼不能直接轉發 Token？

當 MCP Gateway 位於 Client 與 Server 之間時，最直覺的做法是將使用者的 Bearer Token 原封不動轉發給後端 MCP Server。然而，MCP 規範明確禁止這種做法。規範聲明：MCP Server 不得接受或傳遞來自其他服務的 Token，也不得將從 MCP Client 收到的 Token 傳遞給其他服務。

⚠️ 直接轉發 Token 的三大風險

1. Audience 混淆攻擊 — 如果 Token 的 aud 聲明未綁定特定 Server，攻擊者可以用同一個 Token 跨服務存取。
2. 權限擴大 — 使用者原本取得的 Token Scope 可能超出特定 MCP Server 所需，導致過度授權。
3. Token 洩露面擴大 — 每個 Gateway 後方的 Server 都能看到原始 Token，任何一個被入侵都會完全暴露使用者憑證。

這產生了一個核心設計張力：如何在 Gateway 中介層維持每用戶身份（per-user identity），同時遵守 Audience 隔離要求？答案是：Gateway 必須代理（broker）而非轉發（forward）Token。

2

MCP OAuth 2.1 授權規範機制

MCP 授權模型經歷三次重大修訂。2025 年 3 月規範要求每個 MCP Server 自行實作 Authorization Server；2025 年 6 月將兩者完全解耦 — MCP Server 成為純粹的 OAuth 2.0 Resource Server，只負責驗證 Token 而不發行 Token；2025 年 11 月進一步引入 CIMD 和企業級擴展。

五大 Gateway 授權委派模式

當 Gateway 位於 Client 與 Server 之間，它必須在維持每用戶身份的同時遵守 Audience 隔離。以下五種模式各有不同的安全與複雜度取捨。

1RFC 8693 Token Exchange（最推薦）

原理：Gateway 收到使用者的寬範圍 Access Token 後，呼叫 IdP 的 Token Exchange 端點（grant_type=urn:ietf:params:oauth:grant-type:token-exchange），以特定後端 MCP Server 的主機名稱作為 audience，取得窄範圍的新 Token。每個後端 Server 只收到僅對自己有效的 Token。

代表實作

Red Hat Kuadrant MCP Gateway — 透過 AuthPolicy CRD 從入站請求中提取 Bearer Token，提交到 Keycloak 的 Token Exchange 端點，以目標 MCP Server 主機名為 audience，替換 Authorization Header 後轉發。

安全優勢

✅ 完美 Audience 隔離 ✅ 保留使用者身份（sub 聲明） ✅ 符合 MCP 規範 ✅ 支援 Scope 縮減

2On-Behalf-Of (OBO) 流程

原理：MCP Server（而非 Gateway）使用 grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer 將使用者的 Token 作為 assertion 提交給 IdP，取得新 Token（aud 匹配下游 API），保持相同的 sub 聲明。Gateway 的角色非常輕量 — 只負責轉發或移除 Authorization Header。

代表實作

Microsoft Entra ID + FastMCP — 工具處理器呼叫 confidential_client.acquire_token_on_behalf_of(user_assertion=token, scopes=[...])，取得代表使用者身份的下游 API Token。

安全優勢

✅ 完整的委派語意 ✅ Gateway 無需管理上游 Token ⚠️ 需要 Server 端實作交換邏輯

3Token Brokering（Token 代理工廠）

原理：Gateway 實作自己的 AS 介面（向 Client 發行自己簽名的 JWT），同時在內部維護到上游 OAuth Provider 的靜態 Client 關係。Gateway 加密儲存上游 Token，向 Client 發行 Proxy Token（JTI 作為查找鍵），MCP Client 永遠看不到上游 Provider Token。

代表實作

FastMCP OAuth Proxy — 使用 Fernet 加密（AES-128-CBC + HMAC-SHA256）加密上游 Token，發行 HS256 簽名的 JWT。
Cloudflare Workers — 使用「全假 Token」模式：Proxy 授權碼、Proxy Access Token，透過 KV 儲存映射到真實 Provider Token。

安全優勢

✅ 上游憑證完全不暴露 ✅ 可跨多種非 OAuth 後端 ⚠️ Gateway 成為單一信任點

4Dual-Sided Authentication（雙側認證）

原理：Gateway 管理兩個獨立的認證上下文：入站（驗證使用者身份）和出站（使用服務帳號或特定憑證存取後端）。使用者身份透過自訂 Header（如 X-User-Id、X-User-Email）傳遞到後端，而非透過 Token。

代表實作

Kong AI MCP OAuth2 Plugin — 透過 Token Introspection 驗證入站 Token，提取 Claims 作為自訂 Header 轉發，明確不轉發原始 Token。
Traefik Hub Triple Gate 模式 — Gate 1（AI Gateway）處理 PII 過濾；Gate 2（MCP Gateway）執行 JWT 驗證與 TBAC；Gate 3（API Gateway）管理後端認證。三道閘門使用獨立憑證。

安全優勢

✅ 清晰的關注點分離 ✅ 可疊加多層安全 ⚠️ 後端必須信任 Gateway 注入的 Header

5Credential Vaulting（憑證保險庫）

原理：Gateway 在運行時從 HashiCorp Vault 查詢每用戶、每服務的憑證（PAT、API Key），注入到出站請求中。這解決了不支援 OAuth 的舊式後端問題。當 Vault 中無對應條目時，回退到 RFC 8693 Token Exchange。

代表實作

Red Hat MCP Gateway — 查詢 /v1/secret/data/{username}/{mcp-server-host} 取得使用者在該服務的個人憑證。同一 Gateway 可同時服務 OAuth 和非 OAuth 的 MCP Server。

安全優勢

✅ 支援異質後端 ✅ 集中化憑證管理 ⚠️ 需要額外的 Vault 基礎設施

企業級實作案例

IBM ContextForge

雙 Header 模式：Authorization 認證 Gateway 本身；X-Upstream-Authorization 攜帶每用戶憑證直接傳遞到上游 MCP Server。支援 OAuth 2.0 Authorization Code + PKCE 的每伺服器 OAuth 註冊。安全審查後已從預設 passthrough headers 中移除 Authorization。

Traefik Hub

Task-Based Access Control (TBAC)：JWT Claims 編碼授權的任務、工具和交易限制。策略引擎評估如 Lte(mcp.params.arguments.amount, jwt.tools.expense_mcp.max_amount)。Gateway 同時過濾 tools/list 回應，使用者只能看到被授權的工具。

Red Hat Kuadrant

三合一架構：① OPA 策略從 JWT 提取權限，生成簽名「Wristband」JWT 注入 x-authorized-tools Header；② RFC 8693 Token Exchange 縮減 Scope；③ Vault 查詢非 OAuth 後端的每用戶憑證。Wristband 的加密簽名防止竄改。

Cloudflare MCP Portals

Zero Trust 整合：將 Cloudflare Access 的同一細粒度策略（MFA 要求、裝置狀態檢查、地理限制）應用到 MCP 連線。「Linked Apps」功能讓 MCP Server 代表特定使用者取得 OAuth Token 存取自託管應用，保留個人權限和 Scope。

Kong AI MCP OAuth2

Token Introspection + Header 注入：驗證入站 Token 後提取 Claims 作為自訂 Header 轉發（X-User-Id、X-User-Email），明確不轉發原始 Token。支援從 REST API 自動生成 MCP Server 並附加 OAuth 保護。

Higress + Nacos

Bearer + passthrough 模式：server.passthroughAuthHeader 配置啟用透明轉發客戶端憑證到後端 API。閘道層認證防止未認證請求產生 Serverless 計算成本。中國企業的務實中間方案。

5

SEP 提案與未來演進

SEP-990：Cross-App Access（跨應用存取）

🔑 最具影響力的提案 — 已納入 2025-11-25 規範作為授權擴展

解決的核心問題：當 AI Agent 連接到 MCP Server（如 Asana），企業 IdP 只看到「使用者登入了 Asana」，而看不到「一個 AI 應用正在代表使用者操作 Asana」。

運作機制：使用者透過企業 IdP 登入 MCP Client → 取得 ID Token → 請求 IdP 交換為 ID-JAG（Identity Assertion JWT Authorization Grant，一種短期簽名 JWT，宣告「此 Client 被允許代表此使用者操作該下游應用」）→ IdP 評估企業策略（此 Agent 是否被核准？此使用者是否有權限？）→ Client 將 ID-JAG 提交給 MCP Authorization Server → AS 發行標準 Access Token。

企業價值：集中管理員控制、完整可見性（哪些 AI 應用存取哪些工具）、消除影子 AI 連接。

SEP-1932：DPoP（Proof-of-Possession）

透過 RFC 9449 實現 Sender-Constrained Token — 將 Token 綁定到加密金鑰對。即使 Token 被攔截，沒有 Client 的私鑰也無法使用。目前正在積極審查中。核心爭論在於 Content Digest（將 DPoP 證明綁定到請求正文）的複雜度是否合理。

SEP-1933：Workload Identity Federation

讓非人類工作負載（容器、CI/CD 管線、雲端服務）使用其雲端平台的聯邦 OIDC Token 認證 MCP Server，而非使用靜態密鑰。遵循 GCP Workload Identity Federation 和 Azure Entra Workload ID 建立的模式。

🔐 Phantom Token 模式（Curity 推薦）

MCP Client 收到的是不透明 (Opaque) Token，不包含任何可讀 Claims。Gateway 在內部透過 Token Introspection 將其轉換為包含完整 Claims 的 JWT，供後端 MCP Server 使用。即使 Opaque Token 被截取，攻擊者也無法從中提取任何身份資訊。

推薦的生產部署架構

綜合 MCP 社群、企業供應商和安全研究者的共識，以下分層架構代表當前最佳實踐：

推薦的 MCP Gateway OAuth 分層架構

五項關鍵實作原則

Gateway 集中 OAuth 終結 — Gateway 處理完整的 OAuth 2.1 交握（PKCE、CIMD、Resource Indicators），向 MCP Client 發行自己的 Session Token，永遠不向下游暴露上游憑證。
RFC 8693 Token Exchange 進行 Audience 縮減 — 將寬範圍的使用者 Token 轉換為每伺服器、Audience 特定的 Token 後再轉發。這是最低安全基線。
基於身份的工具過濾 — 使用簽名的 JWT「Wristband」或 TBAC 策略限制 tools/list 回應，使用者只看到被授權的工具，並在參數層級強制約束。
Credential Vault 作為後備 — 為不支援 OAuth 的 MCP Server 提供 HashiCorp Vault 或等效方案，查詢每用戶、每服務的 PAT/API Key。
短期 Token + 輪換 Refresh Token — 減少任何 Token 洩露的影響範圍。Access Token 有效期建議 5-15 分鐘；Refresh Token 必須輪換。
💡 給正在選型的團隊

立即實作：RFC 8693 Token Exchange 作為最低安全基線。
規劃採用：CIMD（目前僅 ~4% 的 AS 支援 DCR，CIMD 是更務實的替代方案）。
密切關注：SEP-990 Cross-App Access（企業 IdP 可見性）、SEP-1932 DPoP（Token 綁定）、SEP-1933 Workload Identity Federation（非人類身份）。
避免：直接 Token 轉發、共用 Service Account 存取多用戶資源、將 Token 記錄到日誌中。
