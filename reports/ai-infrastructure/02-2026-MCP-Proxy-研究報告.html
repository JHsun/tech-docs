01

MCP 協議現況與演進

Model Context Protocol（MCP）由 Anthropic 於 2024 年 11 月發布，採用 JSON-RPC 2.0 作為訊息格式，定義了 AI 應用程式（Host）透過客戶端（Client）連接工具伺服器（Server）的標準化介面。Server 可暴露三種原語：Tools（可執行操作）、Resources（可讀取資料）和 Prompts（可重用提示範本）。

2025 年 12 月，Anthropic 將 MCP 捐贈給 Linux Foundation 旗下的 Agentic AI Foundation，由 Block 與 OpenAI 共同創立。Google、Microsoft、AWS、Cloudflare、GitHub 及 Bloomberg 加入為支持會員。至此，所有主流 AI 平台均已原生支援 MCP。

規範版本演進時間線

2024-11-05 · 初始版本

MCP 首次公開發布。支援 stdio（本地子程序）和 HTTP+SSE（雙端點遠端通訊）兩種傳輸方式。

2025-03-26 · Streamable HTTP

引入 Streamable HTTP 傳輸（單一端點取代雙端點 SSE）、OAuth 2.1 + PKCE 授權、工具註解（Annotations）。SSE 正式標記為棄用。

2025-06-18 · 結構化輸出

新增結構化工具輸出（outputSchema）、Elicitation（伺服器向使用者提問）、JSON-RPC Batching。

2025-11-25 · 當前版本

Tasks（非同步長時間工作流）、簡化授權（Client ID Metadata Documents 取代 DCR）、Extensions 框架、Sampling with Tool Calling、MCP Apps Extension（SEP-1865，Anthropic 與 OpenAI 聯合制定的互動式 UI 渲染標準）。

2026-06（暫定）· 下一版本

預計正式化無狀態傳輸模式、Server Cards 伺服器發現、企業級閘道模式規範。

02

Proxy / Gateway 三層分類

MCP Proxy/Gateway 市場已從早期的實驗性傳輸橋接工具，發展為擁有 30+ 解決方案的成熟企業級軟體類別。根據功能複雜度，可分為三個層級：

第一層：傳輸橋接（Transport Bridge）

解決問題

將僅支援 stdio 的客戶端（如 Claude Desktop、Cursor）連接到遠端 HTTP 伺服器。

代表方案：

sparfenyuk/mcp-proxy（Python，~2,400 GitHub Stars）— 支援雙向 stdio↔SSE/Streamable HTTP 轉換，支援 OAuth2 Client Credentials。
mcp-remote（npm，327,000+ 下載）— 最廣泛推薦的遠端橋接套件。
mcp-remote-go（Anthropic 官方，Go，6MB 靜態二進制）— 完整 OAuth 2.1 支援。

第二層：聚合代理（Aggregating Proxy）

解決問題

將多個 MCP Server 聚合到單一端點後方，提供命名空間分組與統一管理。

代表方案：

tbxark/mcp-proxy（Go）— 多傳輸支援、速率限制。
MetaMCP（TypeScript，Web GUI）— 視覺化管理多伺服器。
VeriTeknik/plugged.in — 附帶 RAG 知識管理與 AI Playground。

第三層：企業級閘道（Enterprise Gateway）

解決問題

提供完整的 RBAC、稽核軌跡、可觀測性、多租戶隔離、安全防護與合規支援。

詳見第五節 企業級 Gateway 比較。

技術架構深度解析

整體架構

MCP Gateway 作為反向代理與編排層，位於客戶端與伺服器之間，將 N×M 的連接複雜度降低為 N+M。閘道檢查傳入的 JSON-RPC 請求，並根據工具定義、Session 親和性或租戶身份進行路由。

傳輸協議

Streamable HTTP 是 2026 年的推薦遠端傳輸方式。它使用單一端點（如 https://example.com/mcp），客戶端透過 POST 發送 JSON-RPC 請求，伺服器可選擇以 JSON 或 SSE 串流回應。透過 Mcp-Session-Id Header 管理會話。此設計相容標準負載均衡器與 CDN。

stdio 保留用於本地開發場景（IDE 插件、CLI 工具），透過 stdin/stdout 進行同步通訊。

認證與授權

認證遵循 2025 年 6 月起強制的 OAuth 2.1 規範。MCP Server 作為 OAuth 2.0 Resource Server，提供 /.well-known/oauth-protected-resource 中繼資料（RFC 9728）。所有客戶端必須使用 PKCE。2025-11-25 規範透過 Client ID Metadata Documents 簡化了註冊流程 — 客戶端只需提供一個描述其 OAuth 屬性的 JSON 文件 URL，無需完整的動態客戶端註冊。

授權在多個粒度上運作：OAuth Scope 對應工具級權限；Step-up Authorization 支援敏感操作的增量範圍擴展。企業閘道還疊加 RBAC（角色）、ABAC（屬性策略，使用 OPA 或 Cedar）及關係型存取控制。虛擬伺服器群組可按團隊分割工具集合 — 業務團隊取得 CRM 工具，工程團隊取得部署工具。

速率限制與負載均衡

遵循成熟的 API Gateway 模式：每用戶/每 Agent 節流、每工具速率限制（用於 API 受限後端）、Session 親和性路由（多步驟工作流）、斷路器（故障隔離）、健康檢查驅動的池管理。Streamable HTTP 的無狀態請求模型使 Kubernetes HPA 水平擴展變得簡單。快取適用於 resources/list、tools/list、prompts/list，但絕不適用於 tools/call（有副作用的操作）。

多租戶隔離

透過伺服器群組或命名空間映射到已認證的租戶身份實現。Docker 的閘道提供容器級隔離（CPU/記憶體限制、無主機檔案系統存取）。企業平台如 TrueFoundry 實現邏輯隔離，含每租戶速率限制、稽核日誌、PII 過濾及成本歸因。

04

2026 官方路線圖

官方 2026 路線圖由首席維護者 David Soria Parra 於 2026 年 3 月 9 日發布，圍繞四大優先領域組織，由工作小組（Working Groups）而非固定發布里程碑來推進。

🔑 四大優先領域

1. 傳輸演進與擴展性 — 核心目標是讓 Streamable HTTP 能在多個伺服器實例間無狀態運行。正在探索類似 Cookie 的機制來解耦 Session 狀態與傳輸層。MCP Server Cards（SEP-2127）將建立標準 .well-known 中繼資料格式，使瀏覽器、爬蟲和注冊中心能索引 MCP Server。本週期不會新增傳輸方式。

2. Agent 通訊 — 迭代 Tasks 原語（2025-11-25 實驗性推出）。生產部署暴露了重試語意（誰決定重試瞬態失敗的任務）和過期策略（完成結果保留多久）的缺口。

3. 企業就緒 — 最未定義的優先領域（有意為之，期待從業者主導）。四個待提案領域：稽核軌跡、企業管理式認證（Cross-App Access）、閘道與代理模式（授權傳播與 Session 語意的明確行為定義）、配置可攜性。預期以 Extension 而非核心規範變更形式落地。

4. 未來展望 — 觸發器/Webhooks（事件驅動的 Server→Client 通知）、串流結果（增量輸出）、更細粒度授權（DPoP SEP-1932、Workload Identity Federation SEP-1933）。

📊 分析師預測

Gartner 預測：到 2026 年底，75% 的 API Gateway 供應商將具備 MCP 功能。
Forrester 預測：30% 的企業應用供應商將推出 MCP Server。

中國生態系統

中國科技公司在 2025 年 4 月的六週內密集採納 MCP。阿里巴巴以最全面的 MCP 策略領先：開源的 Higress 閘道（基於 Envoy，支援 Wasm 插件載入 MCP Server）搭配 Nacos 作為 MCP 注冊中心，形成中國企業的事實標準基礎設施。此架構支援零程式碼 REST/OpenAPI→MCP 轉換、Redis 管理 SSE Session、K8s 原生部署、透過 Nacos 命名空間實現多租戶隔離。

Spring AI Alibaba 提供 Java 優先的 MCP 路徑，專門為運行 Spring Boot/Dubbo 微服務的企業設計 — 這反映了中國企業大規模 Java 微服務的安裝基礎。

中國主要雲廠商 MCP 布局

百度 — 推出中國首個 MCP Store 市場及首個電商交易 MCP Server
騰訊 — Cloud TI 平台託管 MCP 插件，整合微信生態
字節跳動 — Coze Space 整合 MCP，用於抖音/TikTok 電商 API
阿里巴巴 — Qwen3 模型專門訓練以增強 MCP 調度效果

⚠️ 值得注意的差距

儘管平台級支援已經到位，大多數面向消費者的中國網路服務（微信、抖音、美團、攜程、餓了麼）尚未正式發布 MCP Server，限制了實際的 Agent 應用能力。中國企業也特別重視資料本地化模式 — 在無網際網路存取的私有 K8s 叢集上部署 Higress + Nacos。

08

安全風險與防禦

MCP 安全在技術選型中需要特別關注。研究發現 43% 的受測 MCP 實作存在命令注入漏洞（2025 年 3 月）。已記錄的安全事件包括 WhatsApp MCP 資料外洩、GitHub MCP 私有倉庫洩露、Smithery 供應鏈入侵、MCP Inspector RCE（CVE-2025-49596）及 mcp-remote 命令注入（CVE-2025-6514）。

MCPTox 基準測試發現工具中毒攻擊（在 LLM 可見但使用者不可見的工具描述中隱藏惡意指令）在 MCP 客戶端中普遍存在。

🛡️ 生產部署關鍵防禦措施

✅ 工具調用的人機迴圈審批（Human-in-the-Loop）
✅ 輸入驗證 — 使用參數化命令，絕不使用 Shell 串接
✅ DNS Rebinding 防護 — Origin Header 驗證
✅ 本地伺服器綁定 127.0.0.1
✅ Token 基礎而非 Session 基礎的認證
✅ 輸出清理 — 在將工具結果餵入 LLM 上下文前進行消毒
✅ 安全閘道層：Lasso（PII 偵測、Prompt Injection 過濾）、Traefik Hub（PII/越獄控制、NVIDIA/IBM 護欄整合）

💡 選型關鍵原則

必須支援的能力（2026 年穩定架構承諾）：
Streamable HTTP 傳輸、OAuth 2.1 + PKCE、Extensions 框架。

關注即將到來的規範變更：
2026 年 6 月版本預計正式化無狀態傳輸模式與 Server Discovery 中繼資料 — 這兩者對生產級閘道部署至關重要。路線圖中明確納入「閘道與代理模式」，表示中介層行為（授權傳播、Session 語意）的正式規範化即將到來。

避免過早鎖定：
優先選擇支援 Extension 機制的方案，以便隨規範演進而擴展功能，而非依賴特定供應商的自定義實作。
