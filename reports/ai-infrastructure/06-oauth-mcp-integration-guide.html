問題背景與設計動機

當 MCP Server 需要代表使用者存取 Redmine 這類僅支援 API Key 驗證的服務時，OAuth 的標準「委派授權」流程無法直接套用，需要額外的橋接機制。

核心問題在於：如何讓 OAuth Provider 端的身份驗證，安全地映射到目標服務的 API Key，同時保持使用者身份的追蹤與最小權限原則。

本文整理了七種實作方式與兩種應避免的反模式，每種均包含完整步驟與時序圖。

A

雙 API Key 架構

Dual API Key — Server Init Key + Per-User Key
低複雜度 靜態工具清單 職責分離
架構說明

使用兩組不同用途的 API Key：一組由 Server 管理員設定在環境變數中（Server Key），用於 MCP Server 啟動時向 Redmine 探索可用工具；另一組由每位使用者在 OAuth 驗證流程中提供（User Key），用於實際的 Redmine 資料操作。兩組 Key 職責完全分離。

詳細步驟
1
Server 啟動時使用 Server Key 初始化
MCP Server 啟動時，從環境變數讀取 REDMINE_SERVER_API_KEY，向 Redmine 發起 GET /issues.json 等探索請求，確認連線並載入可用的工具 schema（如 create_issue、list_projects 等）。
# 環境變數設定（OAuth Provider 側） REDMINE_SERVER_API_KEY="admin-readonly-xxxx" REDMINE_BASE_URL="https://redmine.example.com" # MCP Server 初始化邏輯 const tools = await loadRedmineTools({ apiKey: process.env.REDMINE_SERVER_API_KEY, baseUrl: process.env.REDMINE_BASE_URL });
2
OAuth Provider 暴露工具清單（靜態）
MCP Server 將載入的工具清單快取，對所有連入的 MCP Client 提供相同的工具 schema，不需要每次重新探索。
3
使用者完成 OAuth 驗證並提交 Redmine User Key
使用者在 OAuth 授權頁面輸入個人 Redmine API Key。OAuth Provider 驗證使用者身份後，將 User Key 加密儲存，並與 OAuth session 關聯。
// OAuth 授權流程中的額外參數 POST /oauth/authorize { response_type: "code", client_id: "mcp-client", redmine_api_key: "user-personal-key-yyyy" // 加密後儲存 }
4
MCP Client 取得 OAuth Token 並連線
OAuth 標準流程完成後，MCP Client 持有 OAuth access token，以此連線至 MCP Server。
5
MCP Server 驗證 OAuth Token 並注入 User Key
每次工具呼叫時，MCP Server 驗證 OAuth access token，解析出對應的使用者身份，從加密儲存取得該使用者的 Redmine User Key，注入實際的 API 請求中。
async function handleToolCall(oauthToken, toolName, params) { const userId = await validateOAuthToken(oauthToken); const redmineKey = await getUserApiKey(userId); // 從加密儲存取得 return callRedmineApi(toolName, params, redmineKey); }

優劣分析

✦ 優點

  • Server Key 與 User Key 職責完全分離，可獨立輪換
  • 工具清單在啟動時快取，響應速度快，無冷啟動延遲
  • Server Key 即使洩漏，攻擊者也無法操作任何使用者資料
  • 架構直覺清晰，適合大多數中小型部署

✦ 缺點

  • 需要維護兩組不同的 credential，運維負擔稍增
  • 工具清單是靜態的，無法依據使用者權限動態過濾
  • 使用者在 OAuth 流程中需額外提交 Redmine Key，UX 多一步驟
  • User Key 需在 OAuth Provider 端加密存放，引入存儲責任
安全性分析
身份隔離
User Key 與 Server Key 完全分開，使用者操作不會影響系統層
Key 洩漏影響
Server Key 洩漏：只影響工具探索；User Key 洩漏：僅影響單一使用者
儲存風險
User Key 必須在 OAuth Provider 端加密儲存，需使用 AES-256-GCM 或等級以上
Token 轉傳
OAuth Token 不直接轉傳給 Redmine，符合安全最佳實踐
B

單一 User Key 延遲初始化

Single User Key with Lazy Initialization
最低複雜度 動態工具清單 首次連線延遲
架構說明

完全移除 Server-level Key，僅使用使用者在 OAuth 驗證時提供的個人 API Key。MCP Server 在每位使用者首次連線時才進行初始化，用該使用者的 Key 探索工具清單，可依據使用者實際的 Redmine 權限動態載入可用工具。

詳細步驟
1
使用者完成 OAuth 驗證並提交 Redmine User Key
與模式 A 相同，使用者在 OAuth 授權時提供個人 Redmine API Key。差異是這是唯一一組 Key，後續所有操作都依賴它。
2
MCP Client 首次連線觸發 Per-User 初始化
當 MCP Client 使用 OAuth Token 第一次連線到 MCP Server 時，Server 檢查是否已有該使用者的工具快取。若無，立即開始初始化流程。
async function onClientConnect(oauthToken) { const userId = await validateOAuthToken(oauthToken); if (!toolCache.has(userId)) { const userKey = await getUserApiKey(userId); const tools = await discoverUserTools(userKey); // 依權限動態載入 toolCache.set(userId, { tools, key: userKey, ttl: Date.now() + 3600000 }); } return toolCache.get(userId).tools; }
3
用 User Key 探索該使用者的可用 Redmine 工具
MCP Server 使用 User Key 呼叫 Redmine API，確認使用者可存取哪些 Projects、哪些操作有權限。依此動態生成工具 schema，無管理員權限的使用者不會看到管理員工具。
4
工具呼叫：直接使用快取的 User Key
後續工具呼叫直接使用快取的 User Key，不需要再次查詢。快取有 TTL，到期後重新初始化。
async function handleToolCall(oauthToken, toolName, params) { const userId = await validateOAuthToken(oauthToken); const { key } = toolCache.get(userId); // 已快取，直接取用 return callRedmineApi(toolName, params, key); }
時序圖
優劣分析

✦ 優點

  • 架構最簡單，無需管理任何 server-level credential
  • 工具清單依使用者權限動態生成，低權限使用者看不到高權限工具
  • 部署環境 env variable 設定最少，易於維護
  • 每位使用者 session 完全獨立，資料隔離最徹底

✦ 缺點

  • 首次連線有初始化延遲（需等待 Redmine API 探索完成）
  • 每位使用者各有獨立 session，memory 消耗隨使用者數線性增長
  • 同一把 Key 同時用於 init 和操作，安全邊界稍模糊
  • User Key 需要安全儲存在 OAuth Provider 端
C

OAuth Proxy（Cloudflare Workers 模式）

OAuth Authorization Proxy / Custom Delegation Grant
高複雜度 完全符合 MCP OAuth 規範 Token 映射
架構說明

在 MCP Client 與真實 OAuth Provider 之間建立一個完整的 OAuth 授權伺服器 Proxy。Proxy 對外暴露完整的 OAuth 2.1 端點（含 Dynamic Client Registration），MCP Client 只與 Proxy 互動；Proxy 對內持有固定的 client_id/secret，並在 KV Store 中維護「Proxy Token ↔ User Key」的映射關係。這是 Cloudflare Workers 最常見的實作方式。

詳細步驟
1
部署 OAuth Proxy（實作完整 OAuth 2.1 端點）
Proxy 需實作以下端點：/.well-known/oauth-authorization-server、/register（DCR）、/authorize、/token。使用 Cloudflare workers-oauth-provider library 可大幅簡化。
// Cloudflare Worker (wrangler.toml) [[kv_namespaces]] binding = "OAUTH_KV" # 儲存 token 映射 id = "xxxxxxxx" // workers-oauth-provider 設定 import { OAuthProvider } from "workers-oauth-provider"; const provider = new OAuthProvider({ apiRoute: "/sse", tokenStore: env.OAUTH_KV, # token 僅存 hash });
2
MCP Client 向 Proxy 進行 Dynamic Client Registration
MCP Client 發現 Proxy 的 OAuth metadata，自動完成 DCR，取得 client_id（每次連線可不同）。Proxy 內部始終使用同一組固定 credentials 對接真實後端。
3
使用者在 Proxy 的自訂授權頁面輸入 Redmine Key
Proxy 提供自訂的授權 UI（非標準 OAuth UI），讓使用者輸入個人 Redmine API Key。Proxy 驗證 Key 是否有效後，生成授權碼（auth code），並在 KV Store 中建立映射。
// Proxy 授權頁 submit 處理 async function handleAuthorize(req, env) { const { redmine_api_key, state } = await req.json(); // 驗證 key 有效性 const valid = await verifyRedmineKey(redmine_api_key); if (!valid) return error("Invalid API Key"); // 生成 proxy token，儲存映射（只存 hash） const proxyToken = generateSecureToken(); await env.OAUTH_KV.put( hash(proxyToken), encrypt(redmine_api_key), { expirationTtl: 3600 } ); return redirect(`/callback?code=${authCode}&state=${state}`); }
4
MCP Client 用 auth code 換取 Proxy Token
標準 OAuth token endpoint 流程。MCP Client 取得的 access token 是 Proxy 自行生成的「假 token」，與真實的 Redmine Key 在 KV Store 中維持映射。
5
工具呼叫：Proxy 解映射並注入真實 Key
每次工具呼叫，Proxy 用 hash(proxy_token) 查 KV Store，解密取得 Redmine Key，注入後端請求。
時序圖
優劣分析

✦ 優點

  • MCP Client 完全感知不到後端的複雜度，體驗最佳
  • 完全符合 MCP OAuth 2.1 規範，支援 DCR
  • Proxy Token 可隨時撤銷，不影響真實 Redmine Key
  • 可同時支援多個 OAuth Provider 統一入口

✦ 缺點

  • 本質上是建立一個完整的分散式授權伺服器，複雜度高
  • 每次 auth check 都有額外的 KV 查詢 hop，延遲增加
  • 需要維護 Proxy 基礎設施（Cloudflare KV 等），成本增加
  • Token 映射安全性完全依賴 KV Store 保護
⚠ 安全警告
Proxy 的 OAuth state 參數必須使用密碼學強亂數，並在 redirect 前綁定到 session。2025 年 7-8 月已有多起生產環境 MCP Server 因 state 未驗證而遭到 session fixation 攻擊。Token 請務必只存 hash，明文絕不持久化。
D

Token Vault（Credential Manager）

Token Vault / Centralized Credential Lifecycle Management
中複雜度 最完整稽核 自動輪換
架構說明

引入專用的 Credential Vault（如 HashiCorp Vault、Auth0 Token Vault、Azure Key Vault）集中管理所有使用者的 Redmine API Key。MCP Server 完全不持有任何 credential，每次工具呼叫前向 Vault 請求有效的 Key。Vault 自動處理加密、輪換、審計、TTL。

詳細步驟
1
使用者首次授權：OAuth + Redmine Key 存入 Vault
使用者完成 OAuth 後，OAuth Provider 將 Redmine API Key 透過 Vault API 安全存入，以 userId 作為索引鍵。Vault 使用 envelope encryption（AES-256-GCM + KMS）存儲。
# 使用 HashiCorp Vault 存入 user secret POST https://vault.example.com/v1/secret/redmine/{userId} Authorization: Bearer {vault-service-token} { "api_key": "user-redmine-key", "ttl": "24h", "created_by": "oauth-provider" }
2
MCP Server 向 Vault 進行服務驗證
MCP Server 以自身的 service token（或 Kubernetes service account）向 Vault 驗證身份，取得存取特定路徑的授權。此 token 與使用者完全無關。
# MCP Server 取得 Vault token（使用 K8s Service Account） const vaultToken = await vault.auth.kubernetes({ role: "mcp-server", jwt: fs.readFileSync("/var/run/secrets/kubernetes.io/serviceaccount/token") });
3
工具呼叫前向 Vault 請求使用者 Key
每次工具呼叫，MCP Server 先解析 OAuth Token 取得 userId，然後向 Vault 請求該使用者的 Redmine Key。Vault 回傳永遠是有效的（自動處理輪換/refresh）。
async function handleToolCall(oauthToken, toolName, params) { const userId = await validateOAuthToken(oauthToken); // 向 Vault 請求，永遠是 fresh credential const { api_key } = await vault.secrets.get( `secret/redmine/${userId}` ); return callRedmineApi(toolName, params, api_key); }
4
Vault 自動管理 Key 輪換（可選）
Vault 可設定定期輪換 Redmine API Key（若 Redmine 支援 key rotation API）。MCP Server 不感知這個過程，始終取得最新有效 Key。
時序圖
優劣分析

✦ 優點

  • 所有 credential 集中管理，最完整的稽核追蹤（誰、何時、存取什麼）
  • MCP Server 完全無狀態，不持有任何 credential，blast radius 最小
  • 支援 Key 自動輪換，到期不影響任何 MCP Server 程式碼
  • 可用於所有類型的 API Key / credential，不限於 Redmine

✦ 缺點

  • 需要額外的 Vault 基礎設施，增加運維依賴與成本
  • Vault 成為 single point of failure，可用性需要額外保障
  • 每次工具呼叫都有一次額外的 Vault API round-trip，延遲增加
  • Redmine API Key 本身通常不支援自動輪換，輪換功能受限
E

Token Exchange / 身份鏈傳遞

OAuth Token Exchange · RFC 8693 · On-Behalf-Of Pattern
最高複雜度 使用者身份全鏈路 需 OIDC/SAML 支援
架構說明

依循 RFC 8693 標準，MCP Server 持有代表使用者的 JWT identity assertion，向內部 IdP 換取 JWT Identity Assertion Grant (id-jag)，再向外部服務的 IdP（若 Redmine 有 OIDC SSO）換取有範圍限制的 access token。這是最符合企業零信任架構的方式，使用者身份在整條鏈路中完整保留。

⚠️ 注意：原生 Redmine 不支援 OIDC，此模式需要 Redmine 有 SSO 外掛或企業版配置。

ℹ 適用場景
企業內部部署、已有 SAML/OIDC SSO 的 Redmine 環境、或需要完整使用者稽核軌跡的合規要求（SOC2、ISO27001）。
詳細步驟
1
使用者完成 OAuth 驗證，MCP Server 取得 JWT
標準 OAuth 流程，MCP Server 取得代表使用者的 JWT access token（含使用者 sub、email 等 claims）。
2
MCP Server 向內部 IdP 執行 Token Exchange（RFC 8693）
MCP Server 持使用者 JWT，向內部 IdP 執行 urn:ietf:params:oauth:grant-type:token-exchange grant type，換取 id-jag（Identity Assertion Grant）token。
POST /token HTTP/1.1 Host: internal-idp.example.com grant_type=urn:ietf:params:oauth:grant-type:token-exchange &subject_token=<user-jwt> &subject_token_type=urn:ietf:params:oauth:token-type:jwt &requested_token_type=urn:ietf:params:oauth:token-type:id-jag &audience=https://redmine-idp.example.com &scope=redmine:issues:write
3
用 id-jag 向 Redmine IdP 換取 Redmine 存取 Token
MCP Server 持 id-jag 向 Redmine 端的 IdP（需事先建立 federation 信任）換取 Redmine 專用的 access token，audience 精確限定為 Redmine 服務。
4
用 Redmine-scoped Token 呼叫 API
MCP Server 持 Redmine-scoped token 呼叫 Redmine API。整條鏈路的每個 token 都攜帶完整的使用者 identity + delegation chain，稽核日誌可追蹤到原始使用者。
時序圖
優劣分析

✦ 優點

  • 使用者身份在整條調用鏈完整保留，稽核日誌最精確
  • 每個 token 都有精確的 audience 限制，防止 token 被誤用
  • 完全符合 OAuth 2.0 Token Exchange 標準（RFC 8693）
  • 最適合企業零信任架構與合規需求

✦ 缺點

  • 實作複雜度最高，需要 Redmine 有 OIDC/SAML SSO
  • 需要事先建立跨 IdP 的 federation 信任關係
  • 原生 Redmine 不支援，需要額外外掛或企業版本
  • 多次 token exchange 引入較高的延遲
F

API Gateway Credential Injection

API Gateway as Credential Manager (Azure APIM / Kong / AWS)
企業級 MCP Server 零 Credential 集中流量控制
架構說明

在 MCP Server 與 Redmine API 之間部署 API Gateway（Azure APIM、Kong、AWS API Gateway 等），並利用 Gateway 的 Credential Manager 功能自動管理所有 API Key 的取得、快取、注入與輪換。MCP Server 呼叫 Gateway 端點時完全不需攜帶任何 credential，Gateway 的 Policy 層自動注入對應的 Redmine API Key。

詳細步驟（以 Azure APIM 為例）
1
在 API Gateway 設定 Credential Provider
在 APIM Credential Manager 中為每位使用者建立 Connection，存入對應的 Redmine API Key。Gateway 使用 envelope encryption 儲存，支援多種驗證流程。
# Azure APIM Bicep 設定 resource credentialProvider 'Microsoft.ApiManagement/service/credentials@2023-05-01' = { name: 'redmine-credential' properties: { authorizationMethods: ['GET'] credentials: { header: { X-Redmine-API-Key: '{{user-api-key}}' # 由 Policy 動態解析 } } } }
2
設定 Gateway Policy：驗證 OAuth Token 並注入 Key
Gateway inbound policy 驗證 Bearer token，從 token claims 取得 userId，從 Credential Manager 取對應 Key，注入至後端請求 header。
<!-- APIM Policy (XML) --> <inbound> <validate-jwt header-name="Authorization" failed-validation-httpcode="401"> <openid-config url="https://oauth.example.com/.well-known"/> </validate-jwt> <get-authorization-context provider-id="redmine-credential" authorization-id="@(context.User.Id)" context-variable-name="redmineAuth"/> <set-header name="X-Redmine-API-Key"> <value>@(((Authorization)context.Variables["redmineAuth"]).Parameters["apiKey"])</value> </set-header> </inbound>
3
MCP Server 直接呼叫 Gateway，不附帶任何 Redmine Credential
MCP Server 只需要 OAuth access token，呼叫 Gateway 暴露的 Redmine 端點。Gateway 全程處理所有 credential 邏輯，MCP Server 程式碼完全不涉及 Key 管理。
// MCP Server 呼叫（無需 Redmine Key） const response = await fetch(`${gatewayUrl}/redmine/issues`, { headers: { 'Authorization': `Bearer ${oauthToken}` // 只需 OAuth token // Redmine Key 由 Gateway 自動注入 } });
時序圖
優劣分析

✦ 優點

  • MCP Server 程式碼完全不接觸任何 credential，最乾淨的分離
  • API Gateway 統一提供限流、日誌、監控、WAF 等安全功能
  • 最適合已有 API Gateway 基礎設施的企業部署
  • 水平擴展能力最強，Gateway 層可獨立擴展

✦ 缺點

  • 需要 API Gateway 基礎設施，成本較高
  • Gateway 成為 critical path，可用性要求高
  • Gateway Policy 配置複雜，學習曲線陡峭
  • per-user credential 管理需要 Gateway 支援此功能
G

Per-User Subprocess 隔離

Per-User Isolated Subprocess (mcp-front Pattern)
隔離最徹底 不修改 MCP Server 擴展性受限
架構說明

為每位使用者啟動一個獨立的 MCP Server 子進程（stdio 模式），透過閘道（mcp-front）管理所有使用者的 OAuth 驗證，並將個人 API Key 注入對應的子進程環境變數。隔離在作業系統進程層，任何一位使用者的操作完全無法影響其他使用者。不需修改任何 MCP Server 程式碼。

詳細步驟
1
mcp-front 部署：設定 OAuth Provider 連接
部署 mcp-front 閘道，設定 OIDC provider（Google、Azure AD、GitHub 等），指定 MCP Server 的執行路徑與允許的 domain/organization。
# mcp-front config.yaml identity_provider: type: google client_id: "xxxx" allowed_domains: ["example.com"] services: redmine: command: "npx redmine-mcp" per_user_subprocess: true env_inject: REDMINE_API_KEY: "{{user.redmine_key}}" # 從 user profile 注入 REDMINE_URL: "https://redmine.example.com"
2
使用者透過 Web UI 設定個人 Redmine API Key
使用者登入 mcp-front 提供的 Web UI，輸入個人 Redmine API Key，mcp-front 以 AES-256-GCM 加密後存入自身的 encrypted store，與使用者 profile 關聯。
3
使用者連線：mcp-front 為其啟動獨立子進程
當使用者的 MCP Client 連線時，mcp-front 驗證 OAuth Token，然後為該使用者 fork 一個新的 MCP Server 子進程，並將使用者的 Redmine Key 作為環境變數注入，子進程間完全隔離。
// mcp-front 內部邏輯（簡化版） async function onUserConnect(oauthToken) { const user = await verifyToken(oauthToken); const redmineKey = await getEncryptedKey(user.id); // 為此使用者啟動獨立子進程 const process = spawn('npx', ['redmine-mcp'], { env: { ...process.env, REDMINE_API_KEY: decrypt(redmineKey), // 僅此進程可見 REDMINE_URL: config.redmineUrl } }); userProcessMap.set(user.id, process); return proxyStdio(user, process); }

優劣分析

✦ 優點

  • 進程層隔離，安全性最徹底，一個使用者無法影響其他使用者
  • 完全不需修改 MCP Server 程式碼（符合任何 stdio MCP Server）
  • 子進程崩潰不影響其他使用者
  • 適合需要隔離的小型內部工具部署

✦ 缺點

  • 每位使用者各一個進程，記憶體與 CPU 消耗高，擴展性差
  • 不適合大量使用者的公開部署
  • 進程管理複雜（崩潰重啟、健康檢查、清理 zombie 進程）
  • 首次連線有子進程啟動的冷啟動時間
⚠ 應避免的反模式
✗

反模式一：Token Passthrough（直接轉傳 Token）

Poor-Man's Delegation · Token Forwarding Anti-Pattern
⚠ 嚴重安全問題 授權模型破壞
問題說明
⛔ 嚴禁使用
MCP Server 取得 OAuth access token 後，直接將此 token 傳遞給 Redmine（或任何下游 API）。雖然程式碼最簡單，但破壞了整個授權模型，是嚴重的安全反模式。

✦ 為何危險

  • OAuth access token 的 audience（aud claim）是 MCP Server，不是 Redmine。轉傳給 Redmine 時，Redmine 收到的 token 的受眾不是自己，正確的 Redmine 實作應拒絕此 token
  • 若 Redmine 不嚴格驗證 audience，攻擊者只需取得一個有效的 OAuth token，就可以直接繞過 MCP Server 存取 Redmine，完全破壞存取控制
  • 這種模式本質是「冒充（Impersonation）」而非「委派（Delegation）」，違反 OAuth 的設計原則
  • 一旦 Redmine 可公開存取，任何持有 MCP token 的人都能直接存取 Redmine
✗

反模式二：Client Credentials 喪失 User Context

Client Credentials Grant for User-Context Operations
⚠ 授權邊界問題 使用者身份遺失
問題說明
⚠ 有限度可用，但有明確限制
MCP Server 收到使用者的工具呼叫後，用自己的 client_id/secret 向 Auth Server 申請新的 token 來呼叫 Redmine。技術上可以運作，但喪失了使用者身份追蹤，並引入過度授權的風險。

✦ 問題點

  • 新 token 的 resource owner 是 MCP Server（client application），不是使用者
  • Redmine 收到的請求看起來像是 MCP Server 本身的操作，無法區分是哪個使用者發起的
  • 為讓 Redmine 允許 MCP Server 存取所有使用者的資料，需要給予 MCP Server 過度廣泛的權限
  • 稽核日誌完全喪失使用者維度，合規需求無法滿足

✦ 有限適用場景

  • 完全信任的內部服務之間的 server-to-server 呼叫
  • 不需要使用者身份追蹤的背景任務（如排程報告）
  • Redmine 不對外開放，只有 MCP Server 能存取的封閉環境
Per-User 隔離層級深度解說

Per-User 隔離的四個層級

比較表中的「隔離類型」欄位涉及四種本質不同的隔離機制，以下逐一深度解說其原理、邊界與適用場景。

Data-layer Isolation
強制者：應用邏輯
模式：A · C · D · F
② Session+資料
Session + Data Isolation
強制者：應用邏輯
模式：B
③ 身份鏈
Identity Chain Isolation
強制者：密碼學 / IdP
模式：E
④ 進程層
Process-layer Isolation
強制者：作業系統
模式：G
①

資料層隔離（Data-layer Isolation）

適用於模式 A · C · D · F ── 最普遍的隔離方式

架構上只有一個共用的 MCP Server 進程持續運行，隔離發生在「每次 API 呼叫時注入哪把 Key」這個應用邏輯上。記憶體空間、工具清單（模式 A）、執行環境均為所有使用者共享，僅資料存取結果因 Key 不同而隔離。

✦ 有效隔離的範圍
  • ✓Redmine 資料存取（Alice 看不到 Bob 的 Issues）
  • ✓API Key 本身不在請求鏈路上外露
  • ✓使用者不需知道其他使用者的存在
✦ 沒有隔離的範圍
  • ✗記憶體空間共享，有 bug 可能交叉讀取
  • ✗一人的記憶體洩漏影響全體效能
  • ✗工具清單靜態共用（模式 A）
  • ✗進程崩潰影響所有使用者
核心安全前提：credential lookup 邏輯的正確性是唯一防線。MCP Server 的 getUserApiKey(userId) 函式絕對不能發生「把 Alice 的 Key 回傳給 Bob 的 session」的 mapping 錯誤。
②

Session + 資料層隔離（Session + Data-layer Isolation）

適用於模式 B ── 在資料層之上，額外為每人建立獨立 in-memory session

與純資料層的差異在於：每位使用者在同一進程內擁有獨立的 in-memory session 物件，包含該使用者的工具清單快取、Key 快取、狀態、TTL 等。工具清單不再靜態共用，而是依每位使用者在 Redmine 的實際權限動態生成後存入 session。進程本身依然共享。

✦ 與純資料層相比，多了哪些隔離
  • ✓工具清單是 per-user 動態生成的：Bob 只有 list_issues 而看不到 manage_users，Carol 的管理員工具不會出現在 Bob 的清單中
  • ✓Key 快取在 session 物件內，減少每次工具呼叫都查加密 store 的延遲
  • ✓Session TTL 個別管理，Alice 的 session 過期不影響 Bob 的 session
  • ✓首次連線時依使用者實際 Redmine 權限初始化，後續呼叫直接從 session 讀取，無冷查詢
✦ 仍然沒有隔離的部分（與純資料層相同）
  • ✗記憶體空間依然共享，session 物件之間無 OS 強制邊界
  • ✗進程崩潰依然影響所有使用者
  • ✗使用者數量增加，session 物件線性佔用記憶體（這是模式 B 擴展性差的根本原因）
③

身份鏈隔離（Identity Chain Isolation）

適用於模式 E（RFC 8693 Token Exchange）── 概念上最不同的隔離方式

前兩種隔離依賴「把不同的 Key 分開存放和查找」，而身份鏈隔離的核心思路完全不同——隔離由密碼學強制執行，而非由應用程式邏輯維持。每個 token 本身攜帶完整的使用者身份與委派鏈路，整個系統任何節點都能驗證「這個請求究竟代表哪位使用者、經過哪些合法委派路徑發出的」。

✦ 身份鏈隔離的核心優勢
  • ✓隔離由 IdP 密碼學強制，而非依賴應用程式邏輯
  • ✓即使 MCP Server 被 compromise，也無法偽造其他使用者的 token
  • ✓稽核日誌不依賴應用層 logging，token claims 本身就是不可篡改的記錄
  • ✓每個 token 的 audience 精確限定，Redmine token 只能用在 Redmine
✦ 限制與代價
  • ✗需要 Redmine 端有 OIDC/SAML SSO（原生 Redmine 不支援）
  • ✗需要事先建立跨 IdP 的 federation 信任關係
  • ✗多次 token exchange 引入較高延遲
  • ✗實作複雜度最高
與其他隔離的本質差異：前兩種隔離的安全性依賴「程式邏輯不出錯」，身份鏈隔離的安全性依賴「密碼學不被破解」。在零信任架構與高合規要求（SOC2 Type II、ISO 27001）的場景下，後者是唯一真正可驗證的方式。
④

進程層隔離（Process-layer Isolation）

適用於模式 G（Per-user Subprocess）── 作業系統強制的最強硬性邊界

為每位使用者分配一個完全獨立的 OS 進程，進程間由作業系統的記憶體保護機制（virtual memory, page table isolation）強制隔離。沒有任何應用層的邏輯可以跨越這個邊界——除非存在作業系統層級的漏洞。Key 作為環境變數存在於進程私有的 env 空間，其他進程根本無法讀取。

✦ 進程層隔離的獨特優勢
  • ✓作業系統強制邊界，不依賴任何應用程式邏輯
  • ✓即使 MCP Server 程式碼有嚴重記憶體漏洞，最壞只影響一個使用者
  • ✓Alice 的進程崩潰完全不影響 Bob 和 Carol
  • ✓不需修改任何 MCP Server 程式碼（stdio 進程直接 spawn）
  • ✓工具清單 per-process 動態初始化，真正依照使用者權限
✦ 代價與限制
  • ✗n 位同時在線使用者 = n 個 OS 進程，無法用於大量使用者
  • ✗進程管理複雜（崩潰重啟、zombie 清理、健康檢查）
  • ✗首次連線有子進程啟動的冷啟動時間
  • ✗水平擴展困難（進程與 session 綁定在同一台主機）
直觀比喻：辦公室場景
① 資料層

大家在同一間辦公室工作，桌上各有鎖著的抽屜。只要不亂開別人的抽屜，資料就是分開的。但若有人失手把文件掉在走道上，別人可能撿到。

② Session + 資料

同一間辦公室，但每個人多了個人的鎖碼文件夾，自己的工作清單也放在裡面。安全性稍強，但辦公室本身還是共用的。

③ 身份鏈

同一間辦公室，但每份文件都蓋上防偽的個人印章並記錄完整傳遞路徑。即使文件出了辦公室，任何人都能驗證「這是 Alice 授權的」，偽造立刻被發現。

④ 進程層

每個人有完全獨立的辦公室，連走廊都不共用。一間辦公室失火不影響隔壁。隔離最強，但租金最貴（每人一間房）。
