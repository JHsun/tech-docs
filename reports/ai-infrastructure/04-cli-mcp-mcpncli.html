進階 AI Agent 架構設計：純 CLI vs. 純 MCP vs. 混合架構

隨著 AI Agent 應用場景越來越複雜，單純依賴「純 CLI」或「純 MCP」往往會面臨極端抉擇。業界目前最強大的實踐方式是「將 CLI 包裝在 MCP 之下 (CLI under MCP)」，讓 MCP 擔任守門員（大腦與盾牌），CLI 擔任執行者（肌肉）。

以下為您全面對比這三種架構在 6 大核心維度上的表現：

核心維度深度解析

1. Token 成本 (Token Cost)

🏆 純 CLI 勝出
純 CLI (極低) 無 Schema 負擔，直接輸出簡短指令（約 200 tokens）。
純 MCP API (極高) 需載入龐大的自定義 API Schema 與欄位描述，吃掉大量 Context Window。
CLI 包裝在 MCP (中等) 仍需載入一個如 execute_bash(command) 的工具 Schema，但遠比載入數十個獨立 API Schema 輕量得多。

2. 原生知識 (Native Knowledge)

🏆 CLI 🏆 混合架構
純 CLI (極佳) 充分利用 LLM 預訓練時對 git, curl 等指令的「肌肉記憶」。
純 MCP API (較弱) 模型必須閱讀並學習您新定義的工具 Schema，容易出現幻覺或猜錯參數。
CLI 包裝在 MCP (極佳) 完美保留優勢。LLM 依然生成它熟悉的 CLI 指令，只是將其作為字串放入 MCP 工具的參數中發送。

3. 可組合性 (Composability)

🏆 CLI 🏆 混合架構
純 CLI (極佳) 依賴作業系統的 Unix Pipes (|, &&)，單次呼叫即可完成串流。
純 MCP API (較弱) 缺乏原生管線，需等待 LLM 多次回覆（Round-trips）來組合不同工具，增加延遲。
CLI 包裝在 MCP (極佳) MCP 工具可以直接接收一整串包含 Pipe 的指令，在 Server 端一次性執行完畢後再回傳最終結果。

4. 多用戶授權 (Multi-user Auth)

🏆 MCP 🏆 混合架構
純 CLI (極度危險) 依賴伺服器全域變數，難以隔離多租戶，極易導致越權存取。
純 MCP API (極佳) 天生支援獨立 OAuth 憑證與 Token 管理。
CLI 包裝在 MCP (極佳) 完美解決 CLI 資安痛點。MCP 攔截請求後，在底層動態注入該使用者的憑證（如特定 AWS Profile）去跑 CLI，LLM 完全碰不到實體密碼。

5. 有狀態的 Session (Stateful Sessions)

🏆 MCP 🏆 混合架構
純 CLI (無狀態) 每次執行皆需新建 Process，無法記憶上一動的環境變數或目錄 (cd 無效)。
純 MCP API (持續連線) 伺服器常駐，維持高頻呼叫的效能。
CLI 包裝在 MCP (持續連線) MCP Server 可以幫忙記住「目前的工作目錄 (CWD)」或「連線狀態」，確保連續幾次 CLI 呼叫具有上下文連貫性。

6. 企業治理 (Enterprise Governance)

🏆 MCP 🏆 混合架構
純 CLI (無法控管) 猶如讓 LLM 在伺服器上裸奔，極可能誤刪檔案或執行惡意程式碼。
純 MCP API (極佳) 擁有完整的權限控制、資料遮蔽與監控機制。
CLI 包裝在 MCP (極佳) 建立安全沙盒 (Sandbox)。MCP Server 內建正則表達式或 Allowlist 白名單，只允許 LLM 透過 CLI 執行特定指令（例如封鎖 rm -rf），兼顧執行力與絕對安全。

架構總結：為什麼「混合架構」是終極解法？

從上述比較可以看出，「CLI 包裝在 MCP 之下」幾乎成為了最佳實踐的「六邊形戰士」。它解決了單一技術的根本缺陷：

  • 給了 CLI 大腦與盾牌： 原本無法受控的 CLI 操作，現在有了 MCP 作為 Middleware 進行鑑權、日誌稽核與指令攔截，達到了企業級的合規要求。
  • 給了 MCP 肌肉與靈活度： 開發者不再需要為了每一個小功能手寫龐大的 MCP API Schema，直接透過受控的 Bash 工具讓 LLM 發揮其強大的預訓練原生編碼能力。

推薦場景： 企業級 CI/CD 自動化 Agent、需要操作雲端基礎設施 (AWS/GCP) 的 AI 運維機器人，以及任何需要接觸敏感資料卻又想保有高度執行彈性的 AI SaaS 產品。
