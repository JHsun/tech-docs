🗺️ 什麼是 Auto DevOps?

Auto DevOps 是 GitLab 提供的預設 pipeline 模板,讓你的專案不用寫 .gitlab-ci.yml 就能有完整 CI/CD。GitLab 會:

  • 自動偵測語言(Node、Python、Ruby、Go、Java、PHP...)
  • 自動建構(Dockerfile 或 Buildpack)
  • 自動測試(依語言慣例)
  • 自動掃描(SAST、Dependency、Container、Secret)
  • 自動部署到 Kubernetes(需前置設定)
  • 自動監控(Prometheus)
💡 概念來自 Heroku 的 PaaS 體驗,但開放、免費、可自訂。

Quick Start一句話總結

「把 .gitlab-ci.yml 省掉,讓 GitLab 給你合理預設」。適合快速原型、小型專案、不想碰 CI 設定的團隊。

架構視角與自訂 pipeline 的關係

Auto DevOps 本身就是一個可載入的 template(Auto-DevOps.gitlab-ci.yml)。你可以:

  • 完全不寫 .gitlab-ci.yml → 全套用 Auto DevOps
  • 寫 .gitlab-ci.yml → Auto DevOps 不會自動套用,以你的設定為準
  • 只 include 部分 Auto Jobs(見第 6 節)

🔄 1. 12 個 Auto Stage

Auto DevOps pipeline 預設包含這 12 個 stages(部分需 Premium/Ultimate):

#Stage用途層級
1Auto Build用 Dockerfile 或 Buildpack 打包 imageFree
2Auto Test依語言跑測試(Heroku buildpack)Free
3Auto Code QualityCode Climate 做可維護性分析Free
4Auto SAST靜態程式碼安全分析Ultimate
5Auto Secret Detection找程式碼中洩漏的 tokenFree(Ultimate 有 MR widget)
6Auto Dependency Scanning套件漏洞掃描Ultimate
7Auto License Compliance授權合規檢查Ultimate
8Auto Container ScanningDocker image 漏洞掃描Ultimate
9Auto Review Apps每個 MR 自動部 preview 環境Free(需 K8s)
10Auto DAST部署後的動態安全掃描(OWASP ZAP)Ultimate
11Auto Deploy部到 staging / productionFree(需 K8s)
12Auto MonitoringPrometheus 監控 app metricFree(需 K8s)

⚡ 2. 啟用方式

Quick StartProject 層級啟用

  1. 確認專案沒有 .gitlab-ci.yml(若有,以你的為準)
  2. Project → Settings → CI/CD → Auto DevOps(展開)
  3. 勾選 Default to Auto DevOps pipeline
  4. (可選)填 Base domain(例:example.com)
  5. (可選)選 Deployment strategy(continuous / manual / timed incremental)
  6. Save

Quick StartGroup 層級啟用

路徑:Group → Settings → CI/CD → Auto DevOps

啟用後,所有子 group 與 project 自動繼承,但個別專案仍可關掉。

SRE 實務三種部署策略

策略行為適用
Continuous(預設)main 合併即 deploy 到 production小型、低風險
Manual / Protectedproduction deploy 需手動點擊中等風險
Timed incremental分階段推送(10% → 50% → 100%)生產環境

SRE 實務觸發首次 pipeline

  1. 啟用 Auto DevOps 後,要手動觸發第一次:Build → Pipelines → Run pipeline
  2. 之後的 push、MR 都會自動跑
  3. 若 pipeline 失敗,Auto DevOps 可能自動關閉(管理員可改設定)

📦 3. 前置需求(K8s 部署)

Quick Start只做 Build + Test 的最低需求

  • 專案有 Dockerfile 或對應 buildpack 支援的語言
  • 至少一個 runner(Docker executor,privileged 模式)

這種情況 Auto DevOps 可以跑 Build、Test、Security Scan,但跳過 Deploy。

Quick Start加上 Deploy 的需求

  • 連接的 Kubernetes cluster(透過 GitLab Agent for Kubernetes)
  • Kubernetes 1.12+(建議 1.25+)
  • Ingress controller(NGINX 最穩,canary 功能必要)
  • Base domain 設定 wildcard DNS(*.example.com → ingress IP)
  • runner 能連到 cluster

SRE 實務GitLab Agent 設定

# 1. 在 K8s 建立 namespace
kubectl create namespace gitlab-agent-my-app

# 2. 在 GitLab repo 建 agent 設定
# .gitlab/agents/my-agent/config.yaml
ci_access:
  projects:
    - id: my-group/my-project
  groups:
    - id: my-group

# 3. GitLab UI 註冊 Agent(Infrastructure → Kubernetes clusters → Connect a cluster)
# 取得 token

# 4. 在 cluster 安裝 agent
helm upgrade --install my-agent gitlab/gitlab-agent \
  --namespace gitlab-agent-my-app \
  --set config.token=$TOKEN \
  --set config.kasAddress=wss://kas.gitlab.com

SRE 實務簡易測試:用 nip.io

沒有真實 domain 時,可以用 nip.io 測試:

# 假設 ingress IP 是 10.0.2.2
# Base domain 填:10.0.2.2.nip.io
# 會自動解析 *.10.0.2.2.nip.io → 10.0.2.2
# 開發/測試非常方便

架構視角Deploy target 可選

Target說明
Kubernetes(主要)透過 Helm 部署,用 auto-deploy-app chart
GKE / EKS官方有專門整合 guide
AWS ECSFargate / EC2 都支援
AWS EC2透過 CloudFormation
Bare metal需自備 Ingress 與 MetalLB 等

🛠️ 4. Build 機制(Dockerfile / Buildpack)

Quick Start優先順序

  1. 若有 Dockerfile → 用它 build
  2. 否則用 Cloud Native Buildpacks(預設 heroku/buildpacks:22)

Quick StartBuildpack 支援語言

語言偵測檔案
Node.jspackage.json
Pythonrequirements.txt / Pipfile / pyproject.toml
RubyGemfile
Gogo.mod
Javapom.xml / build.gradle
PHPcomposer.json
.NET*.csproj

SRE 實務自訂 buildpack

# .gitlab-ci.yml
variables:
  BUILDPACK_URL: "heroku/python"

# 或用 project.toml 指定多個
# project.toml
[[io.buildpacks.group]]
uri = "heroku/python"

[[io.buildpacks.group]]
uri = "heroku/procfile"

SRE 實務自訂 Dockerfile

# 專案根目錄放 Dockerfile,Auto Build 會直接用
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --production
COPY . .
EXPOSE 5000
CMD ["node", "server.js"]

架構視角傳入 build arg

variables:
  # Docker build args(Auto Build 會轉成 --build-arg)
  AUTO_DEVOPS_BUILD_IMAGE_EXTRA_ARGS: "--build-arg=NODE_VERSION=20"

  # 或用 CNB builder 的 env
  AUTO_DEVOPS_BUILD_IMAGE_CNB_BUILDER: "heroku/buildpacks:22"
⚠️ 不要把 secret 當 build arg。會寫進 image layer,任何 pull image 的人都看得到。secret 應透過 runtime env var 傳。

🎛️ 5. 自訂 Auto DevOps

Quick Start三種自訂層級

層級方式
淺層用 CI/CD Variables 調參數(見常用變數)
中層.gitlab/auto-deploy-values.yaml 調 Helm chart
深層.gitlab-ci.yml include Auto DevOps 後覆蓋個別 job

Quick Start常用控制變數

變數用途
TEST_DISABLED關掉 Auto Test
CODE_QUALITY_DISABLED關掉 Code Quality
SAST_DISABLED關掉 SAST
DEPENDENCY_SCANNING_DISABLED關掉 Dependency Scan
CONTAINER_SCANNING_DISABLED關掉 Container Scan
DAST_DISABLED關掉 DAST
REVIEW_DISABLED關掉 Review App
STAGING_ENABLED啟用 staging 環境
CANARY_ENABLED啟用 canary 部署
INCREMENTAL_ROLLOUT_MODE漸進 rollout(manual/timed)
REPLICASPod 數量
POSTGRES_ENABLED是否自動部 PostgreSQL container
DATABASE_URL自家 DB 連線字串
HELM_UPGRADE_EXTRA_ARGS額外 helm upgrade 參數
KUBE_NAMESPACE部署 namespace
KUBE_CONTEXTGitLab Agent context

SRE 實務自訂 Helm values

在 repo 放 .gitlab/auto-deploy-values.yaml:

replicaCount: 3
image:
  repository: $CI_REGISTRY_IMAGE

resources:
  limits:
    cpu: 500m
    memory: 512Mi
  requests:
    cpu: 100m
    memory: 256Mi

ingress:
  enabled: true
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"

# Sidekiq worker
workers:
  sidekiq:
    replicaCount: 2
    command:
      - /cnb/lifecycle/launcher
      - sidekiq
    preStopCommand:
      - /cnb/lifecycle/launcher
      - sidekiqctl
      - quiet
    terminationGracePeriodSeconds: 60

架構視角深度覆蓋單一 job

# .gitlab-ci.yml
include:
  - template: Auto-DevOps.gitlab-ci.yml

# 覆蓋 build job(保留其他 Auto job)
build:
  script:
    - echo "自訂 build 流程"
    - docker build --build-arg CUSTOM=1 -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA

# 加新 job
compliance-check:
  stage: test
  script:
    - ./scripts/compliance.sh

# 關掉特定 job
code_quality:
  rules:
    - when: never

🧩 6. 只用部分 Auto Jobs

不想全套,但想用某些現成 job(如 SAST、Container Scan),可以單獨 include:

# .gitlab-ci.yml
stages: [build, test, deploy]

include:
  # 只要 SAST
  - template: Jobs/SAST.gitlab-ci.yml

  # 只要 Secret Detection
  - template: Jobs/Secret-Detection.gitlab-ci.yml

  # 只要 Dependency Scanning
  - template: Jobs/Dependency-Scanning.gitlab-ci.yml

  # 只要 Container Scanning
  - template: Jobs/Container-Scanning.gitlab-ci.yml

  # 只要 Auto Build
  - template: Jobs/Build.gitlab-ci.yml

build:
  stage: build       # template 裡有 build job 就會自動跑
✅ 這種做法比啟用整個 Auto DevOps 更實用。你可以用 GitLab 精心調過的安全掃描,又不被迫走 Helm 部署流程。

SRE 實務GitLab 提供的 template 清單

  • Auto-DevOps.gitlab-ci.yml — 完整套餐
  • Jobs/Build.gitlab-ci.yml — Auto Build
  • Jobs/Test.gitlab-ci.yml — Auto Test
  • Jobs/Code-Quality.gitlab-ci.yml — Code Quality
  • Jobs/Deploy.gitlab-ci.yml — Auto Deploy
  • Security/SAST.gitlab-ci.yml — SAST
  • Security/DAST.gitlab-ci.yml — DAST
  • Security/Container-Scanning.gitlab-ci.yml — Container Scan
  • Security/Dependency-Scanning.gitlab-ci.yml — Dep Scan
  • Security/Secret-Detection.gitlab-ci.yml — Secret Detection

也可以用 Component 版本(更新更快):

include:
  - component: $CI_SERVER_FQDN/components/sast/sast@main
  - component: $CI_SERVER_FQDN/components/secret-detection/secret-detection@main

⚖️ 7. 何時該/不該用

Quick Start適合 Auto DevOps

  • 快速原型 / POC
  • 新進團隊 onboarding,不想碰複雜 CI
  • 應用是 12-factor,部到 K8s
  • 想快速啟用 GitLab 安全掃描
  • Review App 需求明確

Quick Start不適合

  • 複雜的 monorepo(多服務 / 多語言)
  • 部署目標不是 K8s(VM、bare metal、FaaS)
  • 需要精細的 pipeline 流程控制
  • Enterprise 合規流程要求嚴格
  • 已經有成熟的 pipeline,不想重寫

SRE 實務混合策略(最常見)

許多成熟團隊的做法:

  • 不啟用完整 Auto DevOps
  • 自己寫 .gitlab-ci.yml 控制 build/deploy 邏輯
  • Include GitLab 安全掃描 template(SAST、Secret Detection 等)
  • 自訂 Helm chart 或 Kustomize 替代 auto-deploy-app

架構視角從 Auto DevOps 畢業

若一開始用 Auto DevOps,後來需要更多控制,漸進式畢業:

  1. 先 include Auto-DevOps.gitlab-ci.yml 並加自訂 job
  2. 覆蓋特定 job(如 build)
  3. 逐步把 Auto job 換成自家 job
  4. 最後只保留 GitLab template 的 security scan

🚀 3 分鐘啟用 Auto DevOps

  1. Push 有 Dockerfile 的專案
  2. Settings → CI/CD → Auto DevOps → Default 打勾
  3. Build → Pipelines → Run pipeline(第一次)
  4. 完成

🔧 常見控制變數(寫在 Project Variables 或 .gitlab-ci.yml)

variables:
  # 關掉不需要的
  DAST_DISABLED: "true"
  CODE_QUALITY_DISABLED: "true"

  # Deploy 設定
  REPLICAS: "3"
  KUBE_NAMESPACE: "my-app-prod"
  POSTGRES_ENABLED: "false"
  DATABASE_URL: "postgres://user:pass@rds.example.com/db"

  # Buildpack
  BUILDPACK_URL: "heroku/nodejs"

  # 漸進 rollout
  INCREMENTAL_ROLLOUT_MODE: "manual"

📦 只用 GitLab 的安全掃描(不啟用完整 Auto DevOps)

include:
  - template: Jobs/SAST.gitlab-ci.yml
  - template: Jobs/Secret-Detection.gitlab-ci.yml
  - template: Jobs/Dependency-Scanning.gitlab-ci.yml
  - template: Jobs/Container-Scanning.gitlab-ci.yml

stages: [test]

# 你自己的 build / deploy job
build:
  script: docker build -t $CI_REGISTRY_IMAGE .

✅ Auto DevOps Checklist

  • 專案有 Dockerfile 或 buildpack 支援
  • 若要 Deploy:K8s cluster 已連接 GitLab Agent
  • Base domain 設定 wildcard DNS
  • Ingress controller 已裝
  • 選擇適當 deployment strategy
  • review app 有 auto-stop(省資源)
  • 不需要的 scan 關掉(省時間)
  • production 走 manual 或 incremental

🆘 常見問題

問題解法
Auto DevOps pipeline 沒跑確認專案沒 .gitlab-ci.yml、已啟用、手動觸發過
Build 失敗找不到 buildpack用 BUILDPACK_URL 指定,或加 Dockerfile
Deploy 失敗K8s 未連、base domain 錯、ingress 沒裝
自家 CI/CD 覆蓋被 Auto DevOps 忽略只要有 .gitlab-ci.yml,Auto DevOps 就不自動套,需 include
Pipeline 失敗後 Auto DevOps 自動關閉Admin 設定可改,或手動重啟
