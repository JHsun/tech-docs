🗺️ Secure 總覽與威脅模型

GitLab 儲存一家公司最敏感的資產:原始碼、CI/CD 憑證、部署管道。安全強化的重點分為四層:

👤 身分層

2FA、SSH、Token、Session — 防止帳號被盜

🛡️ 應用層

Rate Limit、Admin Mode、Visibility — 防濫用與橫向擴散

🔒 資料層

Secrets、加密、備份保護 — 防外洩

🌐 基礎建設層

Network、TLS、OS hardening — 防穿透
💡 GitLab 官方提供三份 Hardening 指南,建議同時參考:
  • General:基礎強化
  • Application:應用層建議(本章主要來源)
  • SaaS:GitLab.com 使用者專屬

攻擊對應的防禦

🔑 1. 雙因素驗證(2FA)

Quick Start三種強制 2FA 的範圍

範圍設定位置
全站所有使用者Admin → Settings → General → Sign-in restrictions
僅管理員同上,勾 Enforce 2FA for administrators
特定 Group(含子群組)Group → Settings → General → Permissions and group features

Quick Start基本操作

  1. Admin → Settings → General → Sign-in restrictions
  2. 勾 Enforce two-factor authentication
  3. Grace period 設 8 小時(官方 hardening 建議,預設 48 是太寬)
  4. Save changes

SRE 實務2FA 的重要例外

⚠️ 2FA 只強制在「Web 登入」+「Git 操作」,以下情境不適用:
  • Access Token:本身就是第二因素的替代
  • GitLab Incoming Email:收信建 issue / 回 MR 不走 2FA
  • 2FA 啟用前發出的 Token:仍然有效
  • SSO 使用者(若 IdP 已經做 2FA,GitLab 這邊可能不需要再強制)

SRE 實務群組 2FA 的陷阱

Group A 強制 2FA,Group B 沒強制。若 Project P 屬於 A 且分享給 B,B 的成員可以不 2FA 存取 P。

防範作法:2FA Group 關閉 project sharing。

SRE 實務緊急停用使用者 2FA(救援)

# Rails console 停用單一使用者 2FA(管理員操作)
admin = User.find_by_username('admin_user')
user_to_disable = User.find_by_username('locked_out_user')
TwoFactor::DestroyService.new(admin, user: user_to_disable).execute

# 全站停用 2FA(危險!救援用)
sudo gitlab-rake gitlab:two_factor:disable_for_all_users

# 關閉全站 2FA 強制(保留已啟用使用者)
Gitlab::CurrentSettings.update!(require_two_factor_authentication: false)
⚠️ 以上指令不可逆,使用者需重新註冊 2FA。並且停用使用者 2FA 不會停用「強制 2FA」設定,下次登入又會被要求。

架構視角2FA 策略選擇

場景建議
SSO (SAML/OIDC) + IdP 已 2FAGitLab 不用再強制,避免雙重驗證打擾使用者
本地帳號為主全站強制,8h grace
高敏感環境(金融 / 政府)強制 WebAuthn(FIDO2 硬體金鑰),禁用 OTP
混合(本地 + SSO)強制全站,SSO 使用者會在 IdP 做完就跳過

🔐 2. SSH 金鑰限制

Quick Start金鑰類型建議

路徑:Admin → Settings → General → Visibility and access controls → SSH key types

金鑰類型建議說明
ED25519✅ 至少 256 bits快、安全,最推薦
ED25519-SK✅ 至少 256 bitsFIDO/U2F 硬體金鑰,最安全
ECDSA-SK✅ 至少 256 bits硬體金鑰(FIPS 可用)
RSA⚠️ 至少 2048(3072+ 更佳)若環境不支援 ED25519 才用
ECDSA⚠️ 至少 256 bits部分密碼學家有疑慮
DSA❌ Are forbidden有已知缺陷,務必禁用
💡 FIPS 環境只能用 RSA 和 ECDSA(不能用 ED25519)。一般環境推薦 ED25519 為主。

SRE 實務SSH Key 生命週期

路徑:Admin → Settings → General → Account and limit → Maximum allowable lifetime for SSH keys (days)

建議設定:

  • 一般環境:365 天
  • 高敏感環境:90 天
  • 超高敏感:30 天 + 強制硬體金鑰
⚠️ 設定後,使用者加新 key 時必須設定不晚於上限的到期日。舊 key 不會被自動砍,但管理員可從 UI 批次失效。

架構視角SSH Certificate Authority(進階)

在超大型組織,管理個別 SSH key 會很混亂。GitLab 支援 SSH Certificate:

  • 中央 CA 簽發短期(如 8 小時)certificate
  • 使用者不用把 key 加到 GitLab,而是用 cert 登入
  • 與 HashiCorp Vault SSH Secrets Engine 整合
  • 離職或 key 洩漏直接停 CA 簽發,不需逐個撤銷

路徑:/etc/gitlab/gitlab.rb → gitlab_shell['authorized_keys_file']、gitlab_rails['trusted_ssh_ca_keys']

🎫 3. Token 生命週期治理

Quick Start四種 Token 類型

類型用途範圍
Personal Access Token (PAT)個人程式化存取單一使用者
Project Access Token專案機器人單一專案
Group Access Token群組機器人單一群組
Deploy TokenCI/部署單一專案 (read_repo / read_registry)

SRE 實務Token 生命週期設定

路徑:Admin → Settings → General → Account and limit

  • Maximum allowable lifetime for access tokens (days):建議 90 天
  • Enforce personal access token expiration:✅ 勾選
  • Personal Access Token prefix:預設 glpat-,可改為公司專屬如 mycorp-glpat-(幫助 leaked token 偵測)
⚠️ 設定 token 最長壽命後會發生什麼:
  • 新建 token 必須設到期日,且不超過上限
  • 3 小時後,既有「沒到期日」或「超過上限」的 token 會被自動撤銷
這 3 小時 grace 是讓你反悔的時間。大型站要先做 inventory。

SRE 實務Token Inventory(盤點)

# Rails console:列出所有未撤銷的 PAT
sudo gitlab-rails console
> PersonalAccessToken.active.pluck(:user_id, :name, :expires_at, :scopes)

# 找到即將過期的
> PersonalAccessToken.active
    .where('expires_at < ?', 14.days.from_now)
    .pluck(:user_id, :name, :expires_at)

# 找到沒設到期日的
> PersonalAccessToken.active.where(expires_at: nil).count

# 找特定使用者的所有 token
> User.find_by_username('alice').personal_access_tokens.pluck(:name, :expires_at)

架構視角Token 治理模型

  • 人員 Token(PAT):短壽命 + 強制到期 + 每月 review
  • 服務 Token(Project / Group):放進 Vault 集中管理,定期輪替
  • CI Token(CI_JOB_TOKEN):限制只能存取特定專案,不跨專案
  • 自動化:寫 cron script 掃快到期 token 並發信通知擁有者
  • Leak 偵測:Custom prefix + 公開網路掃描(SecretScan / TruffleHog)

架構視角Service Account(服務帳號)

GitLab 提供「Service Account」(虛擬 bot 使用者),比 PAT 更好:

  • 不計入 license seat
  • Owner/Maintainer 可管理
  • 有到期日
  • 比 PAT 更適合 CI/CD 自動化

👑 4. Admin Mode 與管理員強化

Quick StartAdmin Mode 核心概念

第 4 章已介紹過,這裡補充安全面向。Admin Mode 類似 Linux 的 sudo:

  • 平時管理員以一般使用者身分操作
  • 做管理動作時,重新驗證 → 進入 Admin Mode
  • Admin Mode 6 小時自動逾時(不可延長)

SRE 實務Admin 帳號分離原則

🔴 重要原則
不要用自己的日常帳號當 admin。應該建立專用 admin 帳號,與日常帳號分開。

推薦結構:

帳號權限用途
alice一般使用者寫程式、review MR、日常工作
alice-adminAdmin僅用於管理操作
root / emergencyAdmin鎖在密封信封,僅災難時用

SRE 實務API Token 搭配 Admin Mode

# 建 admin token 時必須勾選 admin_mode scope
# 沒勾的話,即便是 admin 使用者,API 也拿不到 admin 權限

# 檢查:
curl --header "PRIVATE-TOKEN: $TOKEN" \
  "https://gitlab.example.com/api/v4/application/settings"
# 若沒勾 admin_mode,會 403

架構視角Break-glass 程序

緊急救援用的 root 帳號管理:

  • 密碼存在 密封信封 + 保險箱
  • SSH key 存在 HashiCorp Vault 的 emergency path
  • 使用需有人見證 + 記錄理由
  • 使用後 24 小時內重設密碼、輪替 key
  • 在 Audit Events 裡特別監控此帳號活動,發現就告警

⏲️ 5. Session 與登入政策

Quick Start基本設定

路徑:Admin → Settings → General → Account and limit

項目建議值
Session duration (minutes)480(8 小時)
Session duration for Git operations when 2FA enabled15(預設)
Prevent users from changing their profile name✅ 勾選(維持 Audit 一致)
Email notification for unknown sign-ins✅ 勾選(必開)
Gravatar enabled❌ 關閉(內部站不需要)

SRE 實務IP Allowlist(全站 / 群組)

路徑:Admin → Settings → General → Visibility and access controls

# Global-allowed IP ranges(全站)
10.0.0.0/8
192.168.0.0/16
# 公司 VPN 出口 IP
203.0.113.0/24

設定後,只有這些 IP 能存取 GitLab。同樣可以在 Group 層級設定。

架構視角Zero Trust 思路

  • IP allowlist + 2FA + 短 Session + Token 短壽命:多層防護
  • Device binding(WebAuthn + 硬體金鑰):防止密碼+OTP被釣走
  • 未知裝置通知:任何新裝置登入都發信給使用者
  • Admin Mode + Audit Events:高敏感操作都留軌跡

🗝️ 6. Secret Management(Vault / KMS)

🔴 黃金原則
密碼、Token、金鑰、任何需要保護的 Secret,絕對不能以明文存放(包含 gitlab.rb、CI variable、環境變數、程式原始碼)。

Quick StartGitLab 原生 secret 位置

位置適用備註
CI/CD Variablespipeline 使用可 mask / protected
Group / Instance CI Variables跨專案共用管理員控制
Secure FilesKeystore、憑證檔iOS/Android 簽名用
External Secrets從 Vault 動態拉取Premium+

SRE 實務CI Variable 使用原則

  • ✅ Mask variable:log 中不顯示
  • ✅ Protect variable:只有 protected branch / tag 可讀
  • ❌ 避免 File variable 存大量敏感資料
  • ❌ 不要在 Group Variable 放 prod 密碼,走 Vault

SRE 實務HashiCorp Vault 整合(Premium+)

# .gitlab-ci.yml
job:
  id_tokens:
    VAULT_ID_TOKEN:
      aud: https://vault.example.com
  secrets:
    DB_PASSWORD:
      vault: production/db/password@secret
  script:
    - echo $DB_PASSWORD

優點:

  • Pipeline 執行時動態拉取,不落地 GitLab
  • Vault 有 audit log
  • 使用 OIDC,無需存 Vault token 在 GitLab
  • Vault 可用 short-lived credentials(如 DB username 每次都不同)

架構視角Secret Detection 與 Push Protection

路徑:Project → Secure → Security configuration → Secret Detection

  • Secret Detection SAST:掃描 commit 是否含 secret
  • Secret Push Protection(Ultimate):在 git push 當下就擋下,不讓 secret 進 repo
  • Pipeline Secret Detection:每次 MR 檢查

Custom prefix 的影響:若 token 改了 prefix,secret detection 預設抓不到。要自訂偵測規則。

架構視角Secret 洩漏應變

發現 secret 外洩的處理順序(不要只改 commit,token 必須先作廢):

  1. 立即在 GitLab / AWS / DB 撤銷該 token
  2. 檢查 audit log 看是否有異常使用
  3. 重發新 token
  4. 更新所有用到的 pipeline / 系統
  5. 最後再從 Git 歷史清掉(git filter-repo 或 BFG)
  6. 寫事故報告,檢討流程
⚠️ 從 Git 歷史刪 secret 不能保證沒人已經 clone 過。所以撤銷>清理。

🌐 7. 網路與 Outbound 安全

Quick StartOutbound 限制

路徑:Admin → Settings → Network → Outbound requests

⚠️ SSRF 攻擊主戰場。GitLab 預設允許 webhook / integration 存取本地網路,這等於把整個內網暴露給 webhook 設定者。

必做:

  • 取消 Allow requests to the local network from webhooks and integrations
  • 取消 Allow requests to the local network from system hooks
  • 明確列出必要的內部服務到 Local IP addresses and domain names allowlist
  • 取消 Allow requests to the local network from services during import

SRE 實務TLS / HTTPS 配置

# /etc/gitlab/gitlab.rb

# 強制 HTTPS
external_url 'https://gitlab.example.com'
nginx['redirect_http_to_https'] = true

# 現代 TLS(至少 1.2,推薦 1.3)
nginx['ssl_protocols'] = "TLSv1.2 TLSv1.3"
nginx['ssl_ciphers'] = "ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384"
nginx['ssl_prefer_server_ciphers'] = "off"

# HSTS
nginx['hsts_max_age'] = 63072000      # 2 年
nginx['hsts_include_subdomains'] = true

# 自己的憑證
nginx['ssl_certificate'] = "/etc/gitlab/ssl/gitlab.example.com.crt"
nginx['ssl_certificate_key'] = "/etc/gitlab/ssl/gitlab.example.com.key"

架構視角分層防護

不要只靠 GitLab 本身防護:

  • WAF(Cloudflare / AWS WAF):檔 L7 攻擊、OWASP top 10
  • Reverse Proxy:再加一層 header filter、IP allowlist
  • VPN / Bastion:只讓 VPN / 特定網段進得來
  • GitLab Rate Limit:最後一道
  • Network policy(K8s 部署):限制 pod 互相通訊

🛡️ 8. Repository / CI 安全

Quick StartProtected Branches 與 Tags

每個 project 都該設:

  • Protected Branches:main、release/*
  • Allowed to merge:至少 Maintainer
  • Allowed to push:No one(強制走 MR)
  • Require approval from code owners

SRE 實務Delayed Project Deletion

路徑:Admin → Settings → General → Visibility and access controls

  • ✅ Enable delayed project deletion
  • 刪除會進入7 天 grace period,可還原
  • 防止誤刪、惡意刪除
💡 GitLab 沒有「從備份恢復單一 project」的簡單方式。Delayed deletion 是最後的安全網。

SRE 實務CI_JOB_TOKEN 限制

路徑:Project → Settings → CI/CD → Token Access

預設 CI_JOB_TOKEN 能存取其他 public project。惡意使用者可利用這點從外部 project 攻擊你。

  • ✅ 勾 Limit access to this project
  • 只把信任的 project 加到 Allow listed projects

架構視角SAST / DAST / SCA / Secret Detection(Ultimate)

完整的 DevSecOps 應該在 pipeline 裡跑:

掃描作用
SAST靜態程式碼分析(找出程式碼本身的漏洞)
DAST動態應用程式掃描(對跑起來的 app 打)
Dependency Scanning找出相依套件的 CVE
Container Scanning掃描 Docker image 的漏洞
Secret Detection掃描 hard-coded secret
License Scanning找出不合規的授權
IaC ScanningTerraform / K8s manifest 掃描

架構視角Scan Execution Policies(Ultimate)

Ultimate 有「Policy」可在Group 層強制所有專案的 pipeline 都要跑特定掃描,開發者繞不過。類似「合規框架」:

type: scan_execution_policy
name: Enforce SAST
rules:
  - type: schedule
    cadence: "0 0 * * *"
actions:
  - scan: sast
  - scan: dependency_scanning
  - scan: secret_detection

🆘 Secret / Token 外洩應變 5 步驟

  1. 撤銷:立刻在 GitLab revoke,不管歷史提交
  2. 調查:查 Audit Events 看是否已被使用
  3. 通知:若已被用,依規模通知安全團隊 / 客戶
  4. 輪替:發新 token,更新所有消費端
  5. 清理:從 Git 歷史移除(最低優先)
