🗺️ 遷移計畫

⚠️ 遷移不是 1:1 轉換工具。不同 CI 工具的設計理念差異大,硬要 1:1 對映往往得到不好用的結果。把遷移當「重新設計」更好。

Quick Start遷移流程概覽

1. 盤點(Audit)
   ├─ 列出所有現有 pipeline
   ├─ 識別關鍵 / 邊緣 pipeline
   └─ 統計 build 量、執行時間、失敗率

2. 規劃(Plan)
   ├─ 選擇遷移策略(全切 / 漸進 / 混合)
   ├─ 設定時程與里程碑
   └─ 定義成功指標

3. 試行(Pilot)
   ├─ 選 1-3 個 pipeline 試遷
   ├─ 學習 GitLab 慣用模式
   └─ 建立內部 best practice

4. 大規模遷移(Migrate)
   ├─ 依優先序逐個遷移
   ├─ 兩系統並行運行
   └─ 持續優化

5. 收尾(Decommission)
   ├─ 停用舊系統
   └─ 知識傳承

SRE 實務事前盤點問題

  • 有多少 pipeline?哪些每天跑、哪些月跑一次?
  • 哪些是關鍵(production deploy)、哪些是輔助(nightly scan)?
  • 有哪些秘密(API key、cert、SSH key)需要遷移?
  • 有哪些外部整合(Slack、Jira、Artifactory)?
  • 有哪些自製腳本 / plugin?是否有 GitLab 對應功能?
  • Runner / agent 規格?能否重用現有機器?

架構視角抗拒因素管理

技術之外,團隊接受度是最大挑戰:

  • 溝通:為什麼要遷?能拿到什麼好處?
  • Sponsorship:領導層支持很重要
  • 培訓:給時間讓開發者學習
  • quick win:先遷一個有感的 pipeline,展示優勢
  • 支援:遷移期間有專人 / channel 解問題

🎯 1. 三種遷移策略

Quick StartBig Bang(全切)

適合小團隊、pipeline 不多、有完整時間窗
時程2-4 週
風險高(同時遷,出問題影響大)
成本短期高,長期低

Quick Start漸進(Gradual)— 最常用

適合中大型組織、多團隊
時程3-12 個月
風險低(逐 pipeline 遷,可隨時 rollback)
成本長期維護兩系統

典型流程:

  1. 新專案直接用 GitLab CI
  2. 低風險專案先遷
  3. 逐步擴大
  4. 最後遷高風險專案

Quick StartHybrid(永久共存)

適合有特殊 plugin / 工具只 Jenkins 有
時程長期
風險中(需持續維運兩系統)

SRE 實務JenkinsFile Wrapper(過渡)

GitLab 提供 JenkinsFile Wrapper,可以在 GitLab CI job 裡跑完整 Jenkins instance(含 plugin)。適合:

  • 把所有專案都先導到 GitLab,但 pipeline 還是用 Jenkins
  • 之後再逐個重寫成原生 GitLab CI
⚠️ JenkinsFile Wrapper 不在 GitLab 官方支援範圍內。

SRE 實務對照範例

# Jenkinsfile
pipeline {
  agent { docker { image 'maven:3.9' } }
  stages {
    stage('Build') {
      steps { sh 'mvn package' }
    }
    stage('Test') {
      steps { sh 'mvn test' }
    }
    stage('Deploy') {
      when { branch 'main' }
      steps {
        input "Deploy to prod?"
        sh './deploy.sh'
      }
    }
  }
  post {
    always {
      junit 'target/surefire-reports/*.xml'
    }
  }
}

# 對應 .gitlab-ci.yml
default:
  image: maven:3.9

stages: [build, test, deploy]

build:
  stage: build
  script: mvn package

test:
  stage: test
  script: mvn test
  artifacts:
    when: always
    reports:
      junit: target/surefire-reports/*.xml

deploy:
  stage: deploy
  script: ./deploy.sh
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      when: manual
  environment: production

SRE 實務對照範例

# GitHub Actions
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [18, 20]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
      - run: npm ci
      - run: npm test
      - uses: actions/upload-artifact@v4
        with:
          name: coverage
          path: coverage/

# 對應 .gitlab-ci.yml
test:
  parallel:
    matrix:
      - NODE_VERSION: ["18", "20"]
  image: node:$NODE_VERSION
  script:
    - npm ci
    - npm test
  artifacts:
    paths: [coverage/]

架構視角Action 的替代

Marketplace Action 沒有直接對應,要自己改寫成:

  • shell script(最直接)
  • CI/CD Component(可重用)
  • Docker image(若 action 是 docker-based)
  • 查 GitLab CI/CD Catalog 是否有現成元件

SRE 實務對照範例

# .circleci/config.yml
version: 2.1
jobs:
  test:
    docker:
      - image: cimg/node:20.0
    parallelism: 4
    steps:
      - checkout
      - restore_cache:
          keys:
            - deps-{{ checksum "package-lock.json" }}
      - run: npm ci
      - save_cache:
          key: deps-{{ checksum "package-lock.json" }}
          paths: [node_modules]
      - run: npm test
workflows:
  ci:
    jobs:
      - test:
          filters:
            branches:
              only: main

# 對應 .gitlab-ci.yml
test:
  image: cimg/node:20.0
  parallel: 4
  cache:
    key:
      files: [package-lock.json]
    paths: [node_modules/]
  script:
    - npm ci
    - npm test
  rules:
    - if: $CI_COMMIT_BRANCH == "main"

🎋 5. 其他來源

Quick Start常見對應原則

不論從哪個 CI 遷,核心對應觀念類似:

概念GitLab CI 對應
Pipeline / Workflow整個 .gitlab-ci.yml
Stage / Phasestages:
Job / Task / Steptop-level job
Agent / Runner / WorkerGitLab Runner
Variable / ParameterYAML variables: 或 UI
Secret / CredentialMasked + Protected UI Variable
Trigger / Hookrules: 或 schedule
Artifact / Build outputartifacts:
Cache / Workspacecache:
Plugin / Action / OrbComponent / Template
Approval gatewhen: manual + Protected Environment

SRE 實務Bamboo 特色處理

  • Bamboo 的 "Plan" → GitLab pipeline
  • "Stage" 概念對應
  • "Task" → script 步驟
  • Variable Plans / Deployment Plans → 用 trigger + multi-project pipeline

SRE 實務TeamCity 特色處理

  • "Build Configuration" → GitLab job
  • "Build Steps" → script
  • "Snapshot Dependencies" → needs:
  • "Artifact Dependencies" → artifacts: + dependencies:
  • "Build Templates" → extends: 或 include:

架構視角Travis CI / Drone

  • 結構簡單,通常很容易對應
  • .travis.yml 的 stages、jobs、env、cache 都有直接對應
  • 主要工作:把 Travis 的 marketplace 替換成 GitLab template

⚠️ 6. 常見陷阱

SRE 實務1. 想做 1:1 轉換

不同 CI 的設計理念差異大。硬轉常造成「醜陋的對映」。先理解 GitLab 慣用模式,再決定怎麼轉。

SRE 實務2. 沒處理 secret 遷移

  • 不要直接複製貼上 — 趁機輪替
  • 從 Jenkins credentials 一個個搬到 GitLab UI
  • 同時加 masked + protected
  • 考慮直接上 Vault

SRE 實務3. Plugin 直接 1:1 找替代

有些 Jenkins plugin 在 GitLab 沒有對應,但 GitLab 有更好的方式達成相同目的。例如:

  • Jenkins 用 plugin 跑 SonarQube → GitLab 用 Component
  • Jenkins 用 plugin 通知 Slack → GitLab UI Integration
  • Jenkins 用 Shared Library → GitLab Component

SRE 實務4. 不重視變數優先順序

⚠️ 從 Jenkins 來:Jenkins 變數順序與 GitLab 不同。GitLab 的 UI variables 優先於 YAML。第一次遇到變數「明明設了卻沒生效」的問題就是這個。

SRE 實務5. 忽略 GitLab 內建好處

遷完只是「能用」,要用 GitLab 的優勢才划算:

  • 內建 Container Registry(不用 Artifactory)
  • 內建 SAST/DAST(不用 Jenkins SonarQube + ZAP)
  • MR widget 整合(看到 test、coverage、scan 結果)
  • Auto DevOps(快速啟動)
  • Component / Catalog(取代 Shared Library)

架構視角6. after_script 不能讓 pipeline 失敗

Jenkins 的 post { failure { ... } } 可以讓整個 pipeline 失敗。GitLab 的 after_script 不行 — 出錯也算 job 成功。影響 pipeline 結果的邏輯要寫在 script 裡。

架構視角7. 工具假設不同

場景差異
每 job 環境GitLab 預設每 job 全新容器(無狀態);Jenkins 可重用 workspace
跨 job 傳資料用 artifacts(Jenkins 用 stash 或 workspace)
Pipeline 觸發GitLab 走 rules;Jenkins 看 multibranch / job DSL
Manual 步驟GitLab 用 when: manual;Jenkins 用 input

📋 遷移 Checklist

  • 盤點完成(專案數、pipeline 數、複雜度)
  • 選定遷移策略(漸進最常用)
  • 領導層支持
  • 培訓材料準備
  • 選 1-3 個試點 pipeline
  • 建立內部 best practice / template
  • Secret 遷移計畫(輪替 + Vault)
  • Runner 資源準備
  • 並行運行期間的監控
  • 舊系統下線時程

🛠️ 自動轉換工具

  • JenkinsFile Wrapper:在 GitLab job 裡跑 Jenkins(過渡用)
  • butler2fox:CLI 把 Jenkinsfile 轉 GitLab CI YAML(社群)
  • GitHub Actions Importer:GitHub 提供反向工具(從 GitLab 到 GitHub,不是反過來)
  • 手動最可靠:複雜 pipeline 還是建議人工重寫

🆘 常見問題

問題解法
變數沒生效檢查優先順序、UI 是否同名覆蓋
跨 job 拿不到資料用 artifacts + needs 或 dotenv
after_script 出錯但 pipeline 成功把邏輯移到 script
找不到 Plugin 對應Catalog 找;不行就寫 script;再不行用 Component 包裝
Pipeline 變慢用 cache、DAG(needs)、parallel
團隊抗拒小規模試點展示優勢、培訓、在地專家
