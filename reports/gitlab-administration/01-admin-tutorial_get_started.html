🗺️ 總覽:新管理員的 6 步路線圖

GitLab 官方對新管理員建議的上手順序是:先把自己的帳號鎖緊 → 規劃組織 → 把現有專案搬進來 → 設定備份 → 認識支援管道 → 開啟 rate limit 防濫用。以下章節完全對應此流程。

🔰 新手(Quick Start)

先開 Admin → Users 看一圈,再用本章節末的 Checklist 逐項勾完,就能完成 80% 必備設定。

⚙️ SRE/DevOps

重點放在備份策略(Rake vs Snapshot)與 Rate Limit 的參數計算,這兩塊最常出事。

🏛️ 架構師

思考:Group 層級規劃(最多 20 層)、SSO/LDAP 整合時機、備份 RPO/RTO 目標、>200GB repo 的替代備份策略。

🔐 1. 身分驗證與帳號安全

Quick Start管理員帳號自保 5 件事

  • 設強密碼,存進密碼管理器
  • 為自己的帳號啟用 2FA(雙因素驗證)
  • 加一組備用 Email(遺失帳號時 Support 才找得回你)
  • 儲存或列印 Recovery Codes(備援裝置遺失時救命用)
  • 把 SSH Key 加到 Profile

SRE 實務全站強制 2FA

GitLab 強烈建議 Self-Managed 全體使用者強制 2FA。啟用路徑:

Admin Area → Settings → General → Sign-in restrictions
→ 勾選「Require all users to set up two-factor authentication」
→ 設定寬限期 (Grace period,建議 7-14 天)
⚠️ 強制 2FA 後,若使用者在寬限期內沒完成設定,Git 操作(push/pull)會被阻擋。請先公告、再開啟。

架構視角驗證策略選擇

場景建議方案備註
小型團隊 (< 50)原生帳號 + 2FA簡單夠用
中型組織LDAP / Active Directory 整合可同步群組成員
大型企業SAML SSO + SCIM集中管控,自動開關帳號
高敏感環境SAML + 強制 2FA + External Auth疊加防護

🏗️ 2. 組織結構:Group / Subgroup / Project

Quick Start三個核心概念

  • Project(專案):一個 Git repo + 議題 + CI/CD,最小單位
  • Group(群組):多個 Project 的容器,可統一設權限
  • Subgroup(子群組):群組下再分群組,最多 20 層

權限繼承:在 Group 設定的成員角色,會自動繼承到所有 Subgroup 和 Project。

SRE 實務建立標準流程

# 典型新部門上線流程
1. 建立 Group(以部門為單位,如 "platform-team")
2. 建立 Subgroup(以業務線區分,如 "platform-team/backend")
3. Add members to group(角色:Owner/Maintainer/Developer/Reporter/Guest)
4. Create projects under subgroup
5. (選用)Turn on external authorization control
6. (選用)Sync group memberships by using LDAP

架構視角命名與層級設計

💡 常見反模式:一開始沒規劃,結果所有 project 都塞在一層 Group。半年後要套權限政策會非常痛。

建議命名模式:

  • Group 第一層 = 組織單位(部門 / BU / 產品線)
  • Subgroup = 功能域或技術棧
  • Project = 實際 repo
  • 範例:fintech/payment/api-gateway

層級不要超過 3-4 層(雖然支援 20 層)。太深會讓 CI/CD variable、權限繼承變得難以追蹤。

📦 3. 從外部系統遷移進來

Quick Start常見來源

GitLab 支援從 GitHub、Bitbucket、另一個 GitLab instance 等來源匯入。

基本路徑:New project → Import project → 選擇來源

SRE 實務遷移方式比較

方式適用優缺
Direct TransferGitLab → GitLab✅ 保留 Issue/MR/Wiki
❌ 不建議用於備份
Repository Mirroring持續同步✅ 雙向鏡像
❌ 只同步 Git 內容
Import/Export API批次排程✅ 可自動化
❌ 非所有欄位都匯出
Professional Services大型/複雜遷移付費服務,聯絡 GitLab AM

架構視角遷移計畫關鍵點

  • 先試點(Pilot):挑 1-2 個 repo 跑完整流程,驗證資料完整性
  • 凍結期(Freeze Window):遷移期間來源端要鎖寫入
  • 對照表(Mapping):使用者帳號、權限角色需事先對應
  • 回滾計畫:保留來源系統至少 30 天
  • CI/CD 重建:pipeline 設定、secret、runner 通常要重新配
⚠️ 專案匯出檔(project export)不是備份工具。官方明確說明:並非所有項目都會匯出。

💾 4. 備份與還原

Quick Start最快的備份指令

# Linux package(Omnibus)手動備份
sudo gitlab-backup create

# 備份會存在(預設)
/var/opt/gitlab/backups/

# 還原(需停用連線服務)
sudo gitlab-ctl stop puma
sudo gitlab-ctl stop sidekiq
sudo gitlab-backup restore BACKUP=<timestamp>
sudo gitlab-ctl restart
⚠️ 還原只能還原到「相同版本、相同版次(CE/EE)」的 GitLab。

SRE 實務排程與存放

Cron 排程(建議凌晨離峰):

0 3 * * * /opt/gitlab/bin/gitlab-backup create CRON=1

設定保留天數與遠端存放(/etc/gitlab/gitlab.rb):

# 保留 7 天
gitlab_rails['backup_keep_time'] = 604800

# 上傳到 S3
gitlab_rails['backup_upload_connection'] = {
  'provider' => 'AWS',
  'region' => 'ap-northeast-1',
  'aws_access_key_id' => 'AKIA...',
  'aws_secret_access_key' => '...'
}
gitlab_rails['backup_upload_remote_directory'] = 'my-gitlab-backups'
💡 別忘了單獨備份 /etc/gitlab/gitlab-secrets.json 和 /etc/gitlab/gitlab.rb,否則還原後無法解密資料。

架構視角大型實例的備份策略

當 Git repository 資料超過 ~200GB,官方 Rake backup script 會開始變慢,特別是備份到異地時。

建議替代方案:

  • File System Snapshot(AWS EBS Snapshot / LVM / ZFS)
  • 儲存層複寫(S3 Cross-Region Replication)
  • PostgreSQL WAL Archiving + PITR(Point-in-Time Recovery)
  • Gitaly Cluster 複寫(多副本)

RPO/RTO 設計參考:

等級RPORTO方案
基本24h4-8h每日 Rake backup
中等1h2h小時快照 + 每日 full
高可用<5min<30minHA + WAL + 異地複寫

🆘 5. 取得支援

Quick Start求救順序

  1. GitLab Docs — 自助查詢最快
  2. GitLab Forum — 社群互助
  3. Support Ticket — 付費方案用,提交前先備齊 subscription 資訊
  4. Status Page — 訂閱以掌握服務中斷

SRE 實務開票前該帶的資料

  • GitLab 版本號(Admin → 右下 helper 可看)
  • 部署方式(Omnibus / Helm / Docker / Source)
  • 作業系統版本
  • 相關 log 檔片段(/var/log/gitlab/<service>/)
  • 錯誤重現步驟
  • Subscription ID

架構視角企業支援的價值點

Priority 支援方案最常被用到的場景:

  • Live Upgrade Assistance:重大版本升級時,由 GitLab 工程師線上陪同
  • Severity 1 Response SLA:生產斷線時的承諾回應時間
  • Architecture Review:部署前的架構審查

🚦 6. Rate Limit 防濫用

Quick Start為什麼要設?

Rate Limit 是 GitLab 抵擋 DoS、暴力破解、爬蟲的主要機制。可以依單一 IP 限制每分鐘請求數。

設定入口:Admin Area → Settings → Network → 各 rate limit 區塊

SRE 實務常用 Rate Limit 類別

類別作用建議值(一般內部站)
Unauthenticated API未登入 API60 次/分
Authenticated API已登入 API2000 次/分
Unauthenticated Web未登入網頁100 次/分
Protected Paths/users/sign_in 等10 次/分
Issues Creation建立議題300 次/分
Raw Blob Requestraw 檔下載300 次/分/project

架構視角設計原則

  • 先觀察再限制:一開始設寬鬆,看 Prometheus metric 再縮緊
  • 區分 CI Runner IP:CI 來源通常流量大,要加進 allowlist 或提高上限
  • 搭配 WAF / LB 層:多層防護,不要把壓力全押在 GitLab 自身
  • 監控被限制的請求數:gitlab_rate_limiting_events_total metric
💡 誤擋的典型案例:Webhook 回調、CI Runner 大量 clone、Renovate 之類機器人。記得把這些 IP / User 加到 allowlist。

⚡ 速查表:新管理員第一週 Checklist

Day 1 — 帳號自保

  • 啟用個人 2FA
  • 存好 Recovery Codes
  • 設定備用 Email
  • 加入 SSH Key

Day 2 — 全站安全

  • Admin → Settings → General 檢視一輪
  • 設定登入失敗鎖帳
  • 規劃強制 2FA 公告
  • 調整 password 複雜度

Day 3 — 組織結構

  • 規劃 Group 命名規則
  • 建立第一批 Group
  • 設 default project visibility
  • 限制誰能建 top-level group

Day 4 — 備份

  • 跑一次 gitlab-backup create
  • 設定 cron 排程
  • 上傳到異地(S3 等)
  • 備份 gitlab-secrets.json
  • 做一次還原演練(測試機)

Day 5 — 防濫用

  • 啟用 Rate Limit 基礎項目
  • 設定 Protected Paths
  • Runner/Webhook IP 加白名單
  • 確認 Prometheus 有收到 metric

Day 6-7 — 監控與支援

  • 訂閱 GitLab Status Page
  • 檢查 /var/log/gitlab/ 權限
  • 設定告警收件群組
  • 紀錄 Subscription ID 存檔

📌 指令速查

# 檢查服務狀態
sudo gitlab-ctl status

# 重啟所有服務
sudo gitlab-ctl restart

# 重載設定(改完 gitlab.rb 後)
sudo gitlab-ctl reconfigure

# 跑一次備份
sudo gitlab-backup create

# 看系統健康
sudo gitlab-rake gitlab:check SANITIZE=true

# 列出環境資訊(開票用)
sudo gitlab-rake gitlab:env:info

# 進 Rails Console(高權限,謹慎)
sudo gitlab-rails console
