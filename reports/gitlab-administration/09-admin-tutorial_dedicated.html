🗺️ 什麼是 GitLab Dedicated?

GitLab Dedicated 是 GitLab 提供的「全託管、單租戶、部署在 AWS」的 SaaS 服務。它介於 GitLab.com(多租戶 SaaS)與 Self-Managed(自行維運)之間。

🏢 單租戶(Single-tenant)

你有專屬的 AWS 帳號、專屬 VPC、專屬資料庫。不與其他客戶共用任何資源。

☁️ 全託管

升級、備份、監控、補丁、HA — 全部由 GitLab 團隊負責。

📍 資料主權

可選擇部署於任一 AWS 區域(24 個 / 共 30 個支援),資料不出該區域。

🛡️ 合規就緒

SOC II、ISO 27001、FedRAMP Moderate(Government 版)直接符合。
💡 這章對 管理員 的意義:Dedicated 雖是 SaaS,但仍有大量設定由客戶端管理員操作(SSO、網路、使用者等)。只是底層基礎建設不用自己顧。

🎛️ 3. Switchboard 管理入口

Quick StartSwitchboard 能做什麼?

Switchboard 是客戶端管理員的主要入口,用於管理「基礎設施層級」的設定。GitLab 應用本身的設定仍在 Admin Area。

Switchboard 管Admin Area 管
建立 instanceSign-up/Sign-in 政策
版本升級排程使用者管理
維護視窗設定Group / Project
自訂網域CI/CD 預設
SSO 設定(SAML)Rate Limit
IP allowlistAppearance
PrivateLink 設定Broadcast Messages
備份與 DR 設定Admin Mode

SRE 實務版本升級管理

  • Dedicated 版本 = Latest minus one (n-1),但包含最新安全補丁
  • 每月固定維護視窗升級
  • 客戶可在 Switchboard 選擇偏好的維護視窗(例如凌晨)
  • 緊急安全修補可能不照排程(會事前通知)
💡 與 Self-Managed 不同,你不能拒絕升級,但可以選時間。這也是 Dedicated 的賣點之一:永遠不會卡在舊版。

架構視角Pre-production 環境

可以付費加購 Pre-production 環境(非標配),用途:

  • 測試新版本(先在 pre-prod 試,再上 prod)
  • 驗證設定變更
  • 測試 migration 流程

限制:

  • 單區域部署(無 Geo DR)
  • 無 SLA 保證
  • 版本不能比 prod 新
  • 授權不需多付(算入既有訂閱)

🌐 4. 網路連線:Public vs PrivateLink

Quick Start兩種連線模式

🌍 Public + IP Allowlist(預設)

Instance 對網際網路開放,但可限制只有特定 CIDR 能進。

🔒 AWS PrivateLink

Inbound / Outbound 私有連線,不走公網。支援信任憑證。

SRE 實務Inbound PrivateLink(員工存取)

用在「員工從公司 VPN/專線連到 GitLab 不走公網」:

  1. 在 Switchboard 開啟 Inbound PrivateLink
  2. GitLab 建立 VPC Endpoint Service
  3. 你的 AWS 帳號建立對應的 VPC Endpoint
  4. 公司內網透過該 Endpoint 存取

SRE 實務Outbound PrivateLink(Webhook 回打內網)

用在「Webhook、整合要連到公司內部服務」:

  1. 你在 AWS 建立 VPC Endpoint Service 指向內部服務
  2. 提供 Service Name 給 GitLab
  3. GitLab 在 tenant VPC 建立對應 Endpoint
  4. Webhook URL 用私有 DNS 名稱
⚠️ Dedicated 是 SaaS,無法直接連到你網路裡的 private IP(不像 Self-Managed)。所有對內網的連線都要透過 PrivateLink。

架構視角TLS 憑證信任

若內部服務用自簽或私有 CA 憑證:

  • 可在 Switchboard 上傳信任的 CA 憑證
  • GitLab 會把這些憑證加到 tenant 的 trust store
  • 從 GitLab 呼叫內部 HTTPS 服務就不會擋

🔑 5. 身分驗證與 SSO

Quick Start支援的驗證

驗證方式支援備註
本地帳號✅管理員可建
SAML SSO (instance-level)✅企業主力
OIDC✅支援
SCIM✅自動 provisioning
LDAP❌不支援
Kerberos❌不支援
Smart Card❌不支援
Multiple Login Providers❌僅一個 IdP

SRE 實務SAML 設定流程

GitLab Dedicated 作為 Service Provider,在 Switchboard 提供 SP metadata:

  1. Switchboard → SSO 設定,取得 ACS URL、SP EntityID
  2. 在 IdP(Okta / Azure AD / OneLogin)建立 Application
  3. 把 IdP 的 SSO URL、Certificate、Issuer 填回 Switchboard
  4. 設定 attribute mapping(email、name、groups)
  5. 測試登入
  6. 設定 SCIM(若 IdP 支援)

架構視角LDAP 遷移到 SAML

若你從 Self-Managed 遷移到 Dedicated,且原本用 LDAP,需要:

  • 評估 IdP 替代:Azure AD、Okta、JumpCloud 都可讀 AD
  • 使用者身分必須重新對應(LDAP DN → SAML NameID)
  • Group Sync 改用 SAML group 或 SCIM
  • 建議先在 Self-Managed 並行跑 SAML 驗證過,再遷移

🚫 6. 限制與不可用功能

Dedicated 提供 Self-Managed Ultimate 功能集,但有以下例外:

Quick Start主要不可用項目

類別不支援
驗證LDAP、Kerberos、Smart Card、多重 IdP
RunnerGitLab-managed Runner(需自備)
PagesCustom domain(只能用 tenant_name.gitlab-dedicated.site)
Git HooksServer-side hooks(用 push rules 代替)
Service Desk不支援(部分版本)
Feature Flags系統級 Feature Flag 不能改
Bring Your Own KeyKMS 金鑰管理不開放
Object Storageproxy_download = true 不支援,必須直接下載 S3
外掛不能安裝自訂 Ruby gem / 程式碼

SRE 實務Rate Limit 管理

💡 重要差異:Dedicated 的 Rate Limit 由 GitLab 依 instance size 自動設定,維護視窗可能會重置為預設值。也就是:你在 Admin Area 改的 rate limit,GitLab 可能在升級時改回來。

原因:防止單一使用者或自動化拖垮整個 tenant 的效能。

架構視角需要 Runner 怎辦?

因為 Dedicated 不提供 managed runner,你必須:

  1. 在自己的 AWS / 其他雲 / 自家機房部署 Runner
  2. Runner 連到 Dedicated URL 註冊
  3. 若 Dedicated 使用 PrivateLink,Runner 也要能走 PrivateLink 到 Dedicated

Runner 詳細設定看第 10 章。

架構視角從 Self-Managed 遷移的常見障礙

  • Server-side hooks:改寫成 Push Rules(Premium+)
  • LDAP:要換成 SAML/OIDC
  • 自訂 Rails patch:必須拿掉
  • GitLab Pages custom domain:改用其他靜態 host
  • Rack attack 自訂規則:改為 Admin Area 可設的 Rate Limit
  • Raw Git access:Dedicated 不開放 SSH 到底層主機

🏛️ 7. GitLab Dedicated for Government

Quick Start與一般 Dedicated 的差異

一個特殊版本,專為美國政府機關 / 規管產業設計:

特性GitLab DedicatedDedicated for Government
AWS 區域一般 AWS(24 區域)AWS GovCloud (US-West)
合規SOC II、ISO 27001+ FedRAMP Moderate
加密AES-256FIPS 140 compliant
維運團隊一般工程師政府專責團隊
SLA一般99.9% monthly
版本n-1n-1(含安全補丁)

架構視角FedRAMP 的現實意義

  • 持續監控(Continuous Monitoring)符合聯邦要求
  • 資料主權保證透過 GovCloud
  • 帳號完全隔離(比一般 Dedicated 隔離更徹底)
  • 功能 rollout要過合規審查才會開放
  • 不可用功能更多(部分功能不合規)

🎯 8. 何時該選 Dedicated?

架構視角選擇決策樹

需要資料主權、私網、合規?
├─ 是 ─┬─ 能自行運維 GitLab?
│      ├─ 是 ──▶ Self-Managed(成本可能更低)
│      └─ 否 ──▶ GitLab Dedicated ✅
│
└─ 否 ─┬─ 超過 2000 seats 或敏感資料?
       ├─ 是 ──▶ 評估 Dedicated vs Self-Managed Ultimate
       └─ 否 ──▶ GitLab.com SaaS(最經濟)

架構視角典型選擇場景

場景推薦理由
金融業(受 SOC 監管)Dedicated資料主權 + 合規 + 不用自維運
美國政府承包商Dedicated for GovFedRAMP 必要
歐盟企業(GDPR)Dedicated (EU region)資料不出 EU
技術實力強的新創Self-Managed on K8s成本最低、自訂最高
小團隊GitLab.com零維運成本
已有大型 SRE 團隊Self-Managed控制力最強
內部離線環境Self-ManagedDedicated 不支援離線

架構視角成本考量

Dedicated 起訂門檻較高(最少 1000 seat,特定場景需 2000+),單價也高於 Self-Managed Ultimate。總擁有成本(TCO)對比:

項目Self-ManagedDedicated
授權Ultimate 單價Ultimate + Dedicated premium
AWS / 機房費用自付包含
備份儲存自付包含
SRE 人力2-5 FTE0
升級停機機會成本有無
合規稽核準備自費部分包含

大致規模門檻:若 SRE 人力成本 > Dedicated premium,選 Dedicated 划算。

✅ Dedicated 管理員 Checklist

  • Switchboard 管理員帳號已建立 + 2FA
  • 選定偏好維護視窗
  • SAML SSO 已設定並測試
  • SCIM 已設定(若 IdP 支援)
  • IP Allowlist 或 PrivateLink 已規劃
  • 自訂網域(若需要)已指向並驗證
  • Runner 自行部署並註冊
  • Webhook 目標可透過 Outbound PrivateLink 到達
  • 內部 HTTPS 憑證已上傳(若需要)
  • Admin Area Day-1 設定完成(第 4 章)
  • Application logs 存取路徑確認
  • Pre-prod 環境(若有買)已啟用
  • 訂閱 Status Page 接收通知
