🗺️ 除錯流程

CI 問題的診斷依層級展開:

┌────────────────────────────────────────┐
│ 1. 沒跑 pipeline?                       │
│    → YAML 語法錯 / workflow 排除        │
├────────────────────────────────────────┤
│ 2. 有 pipeline 但少 job?                │
│    → rules / only / except 問題         │
├────────────────────────────────────────┤
│ 3. Job pending 很久?                    │
│    → runner tag、protected、忙碌        │
├────────────────────────────────────────┤
│ 4. Job 執行失敗?                        │
│    → script 錯、變數空、依賴失敗        │
├────────────────────────────────────────┤
│ 5. 結果不如預期?                        │
│    → artifact、cache、變數優先順序      │
└────────────────────────────────────────┘

📝 1. YAML 語法錯誤

Quick Start三個工具

工具路徑 / 用途
Pipeline EditorBuild → Pipeline editor,即時語法檢查 + 視覺化
CI LintBuild → Pipeline editor → Lint tab
IDE schemaVS Code 等 IDE 載入 GitLab CI schema,寫 YAML 時即時提示

Quick StartPipeline Editor 功能

  • Edit:編輯 .gitlab-ci.yml
  • Visualize:graph 看 stage / job 依賴
  • Full configuration:展開 include、extends 後的完整 YAML
  • Lint:語法檢查 + 警告

SRE 實務用 API 做 lint(CI of CI)

# 可以在 pre-commit hook 或 meta-pipeline 跑
curl --header "Content-Type: application/json" \
     --header "PRIVATE-TOKEN: $TOKEN" \
     "https://gitlab.example.com/api/v4/projects/42/ci/lint" \
     --data @- <<EOF
{
  "content": "$(cat .gitlab-ci.yml | jq -Rsa .)"
}
EOF

架構視角把 .gitlab-ci.yml 當 code 治理

  • PR review .gitlab-ci.yml 變更
  • 用 meta-pipeline 測試自己(先 lint,再跑 dry run)
  • 把 include 檔案釘 commit SHA
  • 保留 CHANGELOG 記錄大改動

🎯 2. Rules / 觸發問題

Quick StartJob 沒加入 pipeline?

第一步檢查 rules / only / except,九成問題是規則寫錯。

# 典型錯誤:用 $CI_PIPELINE_SOURCE 但值不對
job:
  script: echo test
  rules:
    - if: $CI_PIPELINE_SOURCE == "push"
# MR pipeline 下,$CI_PIPELINE_SOURCE == "merge_request_event"
# 所以這個 job 不會加入 MR pipeline

SRE 實務用 Pipeline Editor Visualize

Visualize tab 顯示哪些 job 會被加進來。若某 job 沒出現,規則不符。

也可以手動觸發 pipeline 時試不同變數組合,看哪些 job 出現。

SRE 實務重複 pipeline

最常見:push + MR 同時產生兩條 pipeline。解法:設 workflow:rules(見第 2 章)。

架構視角從 only/except 遷移到 rules

⚠️ only/except 與 rules 預設行為不同。同一 pipeline 混用會很混亂。
關鍵字預設 when
only/except(舊)on_success
rules(新)依第一個 match 的 rule 決定

建議一口氣改全部 job 到 rules,不混用。

🔧 3. Runtime 失敗

Quick Start看 Job Log

點 failed job → 看完整 log。重點看:

  • 最後 20 行:通常錯誤在這裡
  • section_start / section_end:log 的折疊段,展開看細節
  • Exit code:通常是 1(一般錯誤)、137(OOM)、143(SIGTERM)
  • Runner version:log 最上面

SRE 實務常見 exit code

Code意義
0成功
1一般錯誤(script 失敗)
2Misuse of shell builtins
126指令不可執行(權限)
127找不到指令
128+N被 signal N 終止(如 130 = Ctrl+C、137 = kill -9)
137Out of memory 或 kill -9
143SIGTERM(timeout / cancel)

SRE 實務shell trace debug

job:
  script:
    # 方式 1:整個 script 都開 trace
    - set -x
    - echo "A"
    - echo "B"
    - set +x    # 關掉

    # 方式 2:單行 trace
    - bash -x ./my-script.sh

    # 方式 3:全開(整個 runner job)
  variables:
    CI_DEBUG_TRACE: "true"    # 會曝露所有變數!
⚠️ CI_DEBUG_TRACE=true 是安全風險:
  • 會把所有 CI 變數(包含 masked)印到 log
  • log 可能被專案任何有權限的人看到
  • 用完一定要刪除 log 或轉為私有
  • 不要設為 pipeline 層級變數,只用於單 job 臨時除錯

架構視角Retry failed job

Job 頁面右上「Retry」可以重跑失敗 job。

Run job again 會重新套用當前的 .gitlab-ci.yml(不是當時的快照)。所以若你已修好 YAML,retry 會用新設定跑。

🔍 4. 變數除錯

Quick Start印出所有變數(安全版)

check-vars:
  stage: .pre
  script:
    # 只印 CI_* 系列(通常不含 secret)
    - env | grep ^CI_ | sort

    # 列特定變數(確認值)
    - echo "BRANCH=$CI_COMMIT_BRANCH"
    - echo "SOURCE=$CI_PIPELINE_SOURCE"
    - echo "DEFAULT_BRANCH=$CI_DEFAULT_BRANCH"

  # 避免 env | sort 印 secret
  # 避免用 CI_DEBUG_TRACE

SRE 實務rules 評估時的變數

⚠️ rules:if 在 pipeline 建立時評估,此時只能用:
  • Predefined 變數
  • UI 設定的變數
  • YAML 定義的變數
不能用:dotenv(還沒跑)、script 內 export 的(還沒跑)、部分跨 pipeline 的變數。

SRE 實務設計可插入 debug 的 job

flaky-test:
  variables:
    DEBUG_FLAGS: ""          # 預設空
  script:
    - pytest $DEBUG_FLAGS tests/

# 手動跑 pipeline 時,在表單設 DEBUG_FLAGS="-vv --pdb"
# 就能開 debug 模式,不改 YAML

架構視角變數除錯流程

  1. 先檢查優先順序:UI > YAML > Predefined
  2. 檢查是否 Protected:feature branch 拿不到 protected 變數
  3. 檢查 Environment scope:job 沒 environment 拿不到 scoped 變數
  4. 檢查 mask 規則:值被當空可能是 mask 失敗
  5. 用 echo ${VAR:-UNSET} 區分「空值」vs「未設定」

💻 5. 本地重現

Quick StartDocker 本地重現

# 1. 用相同 image 起容器
docker run -it --rm \
  -v "$PWD:/workspace" \
  -w /workspace \
  node:20-alpine sh

# 2. 進容器後跑 job 的 script 指令
npm ci
npm test

SRE 實務gitlab-runner exec(已 deprecate)

舊版 runner 支援 gitlab-runner exec docker job_name 本地跑 job,但已在新版 deprecated。現在推薦用上面的 Docker 做法。

SRE 實務Interactive web terminal

若管理員啟用,可以在 job 執行期直接 SSH 進 runner除錯:

  • Runner 需啟用 [session_server]
  • Job 頁面會出現 "Debug" 按鈕
  • 開啟瀏覽器終端,看即時環境

架構視角把「卡住」的 job 留下來調查

debug-on-failure:
  script:
    - ./my-command || true
  # 失敗後 keep 5 分鐘不清掉,管理員可 ssh 到 runner 看
  after_script:
    - |
      if [ "$CI_JOB_STATUS" != "success" ]; then
        echo "Job failed, sleeping 300s for investigation"
        sleep 300
      fi

🆘 6. 常見錯誤一覽

🔴 Pipeline 相關

症狀可能原因 / 解法
Pipeline 根本沒建立YAML invalid;workflow:rules 全 false;push 被 hook 擋
重複 pipeline(branch + MR)設定 workflow:rules
"YAML invalid" badgePipeline Editor 看錯誤;縮排、必填欄位缺失
"too many jobs"Pipeline 超過 instance 限制,拆 parent-child
"Pipelines must succeed" 卡住workflow 排除了這個 pipeline,改設定或關掉此功能

🟡 Job pending / 不執行

症狀可能原因 / 解法
永遠 pending沒 runner 匹配 tag
Pending 後 "stuck"runner offline、tag 不對、protected 問題
"This job is stuck because..."訊息已告訴你問題
Job 該跑卻不在 pipelinerules 沒 match;needs 指向不存在的 job
Fork MR 沒 pipeline需要在 parent project 啟用 fork MR pipeline

🔴 Job 執行失敗

症狀可能原因 / 解法
Exit 137 (OOM)記憶體不夠,加大 runner 或減少並發
Exit 143 (timeout)job timeout,看 Project / Runner / Job 三層設定
"command not found"image 沒裝;PATH 沒設
"permission denied"chmod +x;runner user 權限
"no space left"runner 磁碟滿;清 /builds、/cache
Docker-in-Docker 失敗需 privileged + services:docker:dind
Flaky test不要用 retry 掩蓋,修 test

🟠 資料相關

症狀可能原因 / 解法
Artifact 找不到上游 job 失敗;dependencies 設錯;檔名拼錯
Cache 沒命中key 不同;不同 runner 各有 cache;沒啟用分散式 cache
變數空優先順序、protected、scope(見上節)
Dotenv 沒傳過來format 錯、需要 needs: [build]、不是 key=val
Masked 沒作用值不符 mask 規則(< 8 字元、有空格)

🔵 其他

症狀可能原因 / 解法
"Argument list too long"變數總長度超 ARG_MAX,用 File type 或 Secure Files
Downstream pipeline 失敗trigger project 權限;下游 ref 不存在
Schedule 沒跑owner 權限;cron 時區;active toggle
MR "Checking pipeline status" 一直轉workflow 排除了 MR pipeline,卻要求必須成功

⚡ 速查表

🧭 Debug 5 分鐘流程

  1. 看 Pipeline 頁面 — YAML valid?Job 數量對?
  2. 點 failed job — 看 log 最後 20 行
  3. 看 exit code — 知道是 script / infra / timeout
  4. 看 job variables section — 確認變數值
  5. 本地 Docker 起同一個 image 重現

🔧 實用指令

# 在 job 最開始印關鍵資訊
before_script:
  - echo "Ref: $CI_COMMIT_REF_NAME"
  - echo "Source: $CI_PIPELINE_SOURCE"
  - echo "Runner: $CI_RUNNER_DESCRIPTION ($CI_RUNNER_TAGS)"
  - echo "Job: $CI_JOB_NAME stage=$CI_JOB_STAGE"
  - echo "Build dir: $PWD"
  - ls -la

# 安全印環境(只 CI_*)
script:
  - env | grep -E "^(CI_|GITLAB_)" | sort | grep -v TOKEN

# Shell trace(單段)
script:
  - set -x
  - ./something
  - set +x

# Early fail
script:
  - set -euo pipefail      # 未定義變數 / pipe 失敗都停

🔬 診斷變數的指令

# 檢查變數是否存在(未設定 vs 空)
echo "VAR=${VAR:-(UNSET)}"
echo "VAR=${VAR:+(HAS VALUE)}"

# 檢查變數長度(不洩漏值)
echo "VAR length: ${#VAR}"

# 檢查變數開頭 / 結尾字元(確認沒多空白)
echo "First chars: ${VAR:0:4}..."
echo "Last chars: ...${VAR: -4}"

🧪 可插入 debug 的設計

variables:
  DEBUG_MODE: "false"
  EXTRA_FLAGS: ""

job:
  script:
    - |
      if [ "$DEBUG_MODE" = "true" ]; then
        set -x
        echo "Debug mode ON"
      fi
    - pytest $EXTRA_FLAGS tests/

# 手動觸發時輸入 DEBUG_MODE=true、EXTRA_FLAGS=-vv

✅ Debug Checklist

  • Pipeline Editor 顯示 YAML valid
  • Visualize 顯示預期的 job
  • 手動觸發時能指定變數
  • Runner online 且 tag 匹配
  • 變數值符合預期
  • artifact / cache 保留正確
  • 本地能重現
  • 失敗的 log 已妥善保存

🆘 最後手段

  1. Clear runner caches:Project → CI/CD → General pipelines → Clear runner caches
  2. Retry job:會用當前 YAML 重跑
  3. 手動新起 pipeline:Build → Pipelines → Run pipeline(測試變數)
  4. 新 runner:若問題只在特定 runner 發生,停用該 runner
  5. GitLab Support(Premium+):附 log + YAML 開 ticket
  6. Community Forum:forum.gitlab.com
