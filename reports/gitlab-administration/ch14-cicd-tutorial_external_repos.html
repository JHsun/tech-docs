🗺️ 應用場景

有時候你的程式碼在 GitHub / Bitbucket / 自家 Git server,但你想用 GitLab 的 CI/CD(因為 GitLab CI 強大、有完整 security scan、cost 友善...)。

Quick Start典型情境

  • 程式碼留 GitHub:團隊熟悉,但 GitHub Actions 太貴 / 不夠用
  • 過渡期:遷移到 GitLab 中,先讓 CI 跑起來
  • 合規:某些客戶碼必須留 Bitbucket,但 CI 用 GitLab
  • 自家 Git Server:Gitea / Gogs / 自架 Bitbucket Server
💡 這跟「migration」(13 章)不同 — migration 是整個搬家;這章是原 repo 不動,只跑 CI。

⚙️ 運作機制

Quick Start三步驟

┌─────────────────┐                  ┌─────────────────┐
│  External Repo  │                  │     GitLab      │
│ (GitHub/Bitb.)  │                  │   (lightweight  │
│                 │                  │     project)    │
└────────┬────────┘                  └────────┬────────┘
         │                                     │
         │ 1. Pull Mirroring                   │
         │    (定期 pull)                      │
         │ ──────────────────────────────────▶ │
         │                                     │
         │ 2. Webhook 通知有新 commit          │
         │    (即時觸發 mirror)                │
         │ ──────────────────────────────────▶ │
         │                                     │
         │                                     │ 3. GitLab 跑 .gitlab-ci.yml
         │                                     │    (在 mirror 上)
         │                                     │
         │ 4. Status push 回去                 │
         │    (commit status / check)          │
         │ ◀────────────────────────────────── │
         │                                     │

關鍵:GitLab 建立的是 lightweight project,issues/MR/wiki 預設關掉(可開)。.gitlab-ci.yml 跟著外部 repo 一起 mirror 過來。

🐙 1. GitHub 整合

Quick Start設定步驟

  1. GitLab 右上 Create new → New project/repository
  2. 選 Run CI/CD for external repository
  3. 選 GitHub(用 OAuth 或 personal access token 連線)
  4. 選擇要連的 GitHub repo
  5. GitLab 自動:
    • 建立 lightweight project
    • 設定 pull mirroring
    • 註冊 GitHub webhook
    • 設定 status push 回 GitHub

SRE 實務Personal Access Token 權限

GitHub PAT 需要 scope:

  • repo(public 與 private 都用)
  • admin:repo_hook(註冊 webhook)

SRE 實務外部 PR Pipeline

GitHub 的 pull request 在 GitLab 端可以觸發特殊 pipeline,$CI_PIPELINE_SOURCE = external_pull_request_event:

always-run:
  script: echo "branch push 都跑"

on-pull-requests:
  script: echo "只在 GitHub PR 跑"
  rules:
    - if: $CI_PIPELINE_SOURCE == "external_pull_request_event"

except-pull-requests:
  script: echo "PR 不跑,其他都跑"
  rules:
    - if: $CI_PIPELINE_SOURCE == "external_pull_request_event"
      when: never
    - when: on_success

架構視角GitHub Enterprise

  • 支援(需手動連線:Repository by URL)
  • ⚠️ 手動連線時,external pull request 不會觸發 pipeline
  • 需自行設 webhook + status API 回傳

架構視角狀態顯示在 GitHub

整合自動把 GitLab CI 結果 push 回 GitHub:

  • commit 旁邊顯示 ✓ / ✗
  • PR 頁面顯示 check 列表
  • 可設「檢查必須通過才能 merge」

🪣 2. Bitbucket Cloud 整合

Quick Start設定步驟

  1. Bitbucket 建 App password(repo write 權限)
  2. GitLab → New project → Run CI/CD for external repository → Repository by URL
  3. 填:Git URL、username、App password
  4. GitLab 自動 import + 設定 Pull Mirroring
  5. GitLab 建 Personal Access Token(scope: api)
  6. Bitbucket 設 Webhook(指向 GitLab API,觸發 mirror)

SRE 實務Webhook URL 範例

https://gitlab.com/api/v4/projects//mirror/pull?private_token=

SRE 實務把 status push 回 Bitbucket

Bitbucket Cloud 沒有像 GitHub 那麼自動,需要在 GitLab CI 跑一個 script 去 call Bitbucket API:

# 1. 在 GitLab Variables 設定:
#    BITBUCKET_ACCESS_TOKEN(masked,Bitbucket app password)
#    BITBUCKET_USERNAME
#    BITBUCKET_NAMESPACE(若名稱不同)
#    BITBUCKET_REPOSITORY(若名稱不同)

# 2. 在 repo 加 build_status 腳本(會被 mirror 過來):
# build_status (chmod +x)
#!/usr/bin/env bash
BITBUCKET_API="https://api.bitbucket.org/2.0/repositories/$BITBUCKET_NAMESPACE/$BITBUCKET_REPOSITORY/commit/$CI_COMMIT_SHA/statuses/build"
curl --user "$BITBUCKET_USERNAME:$BITBUCKET_ACCESS_TOKEN" \
  --request POST \
  --header "Content-Type: application/json" \
  --data "{\"state\":\"$BITBUCKET_STATE\",\"key\":\"ci/gitlab-ci/$CI_JOB_NAME\",\"name\":\"GitLab CI\",\"url\":\"$CI_JOB_URL\"}" \
  "$BITBUCKET_API"

# 3. .gitlab-ci.yml 加 status job
stages: [test, ci_status]

unit-tests:
  stage: test
  script: npm test

success:
  stage: ci_status
  script:
    - BITBUCKET_STATE=SUCCESSFUL ./build_status
  when: on_success

failure:
  stage: ci_status
  script:
    - BITBUCKET_STATE=FAILED ./build_status
  when: on_failure

🌐 3. 任何 Git Server(URL)

Quick Start通用流程

對於 Gitea、Gogs、自架 Bitbucket Server,或任何 Git URL,可以用通用方式:

  1. GitLab → New project → Run CI/CD for external repository → Repository by URL
  2. 輸入 Git URL + 帳密(若 private)
  3. GitLab 設 Pull Mirroring
  4. (若有 webhook)在外部設定 webhook 通知 GitLab
  5. (若沒 webhook)mirror 走定期輪詢,延遲較大

SRE 實務Pull Mirroring 設定

路徑:Project → Settings → Repository → Mirroring repositories

  • 方向:Pull(從外部拉到 GitLab)
  • Authentication:Password、SSH key
  • 頻率:約 30 分鐘自動 pull(觸發 webhook 可即時)
  • 支援 trigger pipeline on update

架構視角SSH 認證

# 1. GitLab 自動產生 SSH key(在 mirroring 設定頁)
# 2. 把 public key 加到外部 server 的 deploy key
# 3. URL 用 git@host:repo.git 格式

🔀 4. External Pull Request Pipelines

Quick Start運作方式

GitHub 整合會自動訂閱 push 與 pull_request webhook。當 PR 開啟 / 更新:

  • GitLab 建立 external_pull_request_event 來源的 pipeline
  • 同時還有 branch push 的 pipeline(會有兩個,要用 rules 控制)
  • 兩條 pipeline 的狀態都會回報到 GitHub PR

SRE 實務避免重複跑

workflow:
  rules:
    # 開 PR 時只跑 PR pipeline,不跑 branch pipeline
    - if: $CI_COMMIT_BRANCH && $CI_PIPELINE_SOURCE != "external_pull_request_event"
      when: never
    - when: always

SRE 實務PR 專用變數

External PR 時可用變數:

變數意義
CI_EXTERNAL_PULL_REQUEST_IIDPR 編號
CI_EXTERNAL_PULL_REQUEST_SOURCE_BRANCH_NAME來源分支
CI_EXTERNAL_PULL_REQUEST_TARGET_BRANCH_NAME目標分支
CI_EXTERNAL_PULL_REQUEST_SOURCE_REPOSITORY來源 repo
CI_EXTERNAL_PULL_REQUEST_TARGET_REPOSITORY目標 repo

架構視角限制

  • Fork 的 PR 被忽略(安全考量)
  • 手動連線(GitHub Enterprise)不支援 external PR pipeline
  • 每次 push 會產生兩個 pipeline(branch + PR),需 workflow 控制

↩️ 5. 將 status 推回外部

外部平台是否自動?方式
GitHub.com(OAuth 連線)✅ 自動GitLab Status API 直接更新
GitHub Enterprise(手動連線)❌需自寫 script
Bitbucket Cloud❌需自寫 script + 4 個變數
Bitbucket Server❌需自寫 script
Gitea / Gogs❌需自寫 script

SRE 實務通用 status push 模板

# 任何外部 Git server(假設有 commit status API)
.notify_status:
  script:
    - |
      curl -X POST \
        -H "Authorization: Bearer $EXTERNAL_TOKEN" \
        -H "Content-Type: application/json" \
        -d "{
          \"state\": \"$STATE\",
          \"target_url\": \"$CI_JOB_URL\",
          \"description\": \"$DESCRIPTION\",
          \"context\": \"gitlab-ci/$CI_JOB_NAME\"
        }" \
        "$EXTERNAL_API_URL/repos/$REPO/commits/$CI_COMMIT_SHA/statuses"

notify-success:
  extends: .notify_status
  variables:
    STATE: "success"
    DESCRIPTION: "Pipeline 通過"
  when: on_success

notify-failure:
  extends: .notify_status
  variables:
    STATE: "failure"
    DESCRIPTION: "Pipeline 失敗"
  when: on_failure

⚠️ 6. 已知限制

  • 單向同步:只能 pull(GitLab 拉外部),不能在 GitLab 上 push 改外部 repo
  • Fork PR 不跑:安全考量,fork 的 PR 不會被 mirror
  • Mirror 有延遲:沒設 webhook 時,30 分鐘輪詢一次
  • Mirror 失敗會 hard fail:外部 force push 等情境會卡住,需手動修
  • 輕量 project:預設關 issues/MR/wiki(可後啟)
  • 無真正 MR:GitLab 的 MR 用不上(因為要去外部 PR/MR)
  • Two pipelines per push:GitHub PR 會建 branch + PR 兩條 pipeline

架構視角什麼時候該真的搬家?

情境建議
暫時用 GitLab CIExternal Repo 整合(本章)
長期 hybridExternal Repo,但要忍受兩平台維運
想用 GitLab MR review真正搬家(13 章 migration)
想用 GitLab security dashboard 整合真正搬家(會更順)
大團隊長期使用真正搬家

⚡ 速查表

🚀 快速啟用流程

  1. GitLab → Create new → New project → Run CI/CD for external repository
  2. 選 GitHub(OAuth)/ Bitbucket(URL)/ 任意 URL
  3. 填認證資訊
  4. 等 mirror 完成
  5. 在外部 repo 加 .gitlab-ci.yml,push,看 GitLab 跑起來
  6. (若需要)設 status push 回外部

🔧 .gitlab-ci.yml 模板

# 適用 external repo 的 workflow
workflow:
  rules:
    # GitHub PR
    - if: $CI_PIPELINE_SOURCE == "external_pull_request_event"
    # 不重複跑 branch
    - if: $CI_COMMIT_BRANCH && $CI_PIPELINE_SOURCE == "external_pull_request_event"
      when: never
    - if: $CI_COMMIT_BRANCH

stages: [test, ci_status]

test:
  stage: test
  script: ./run-tests.sh

# 若用 Bitbucket,加 status push job
notify-bitbucket:
  stage: ci_status
  when: always
  script:
    - ./build_status

✅ Checklist

  • 外部 repo 已連線到 GitLab
  • Pull mirroring 顯示 healthy
  • 外部 webhook 已設定(若可)
  • 新 commit 能即時觸發 GitLab pipeline
  • GitLab status 有顯示在外部 commit / PR
  • 不重複跑 pipeline(workflow 已設)
  • 機敏 token(GitHub PAT、Bitbucket App password)妥善保護

🆘 常見問題

問題解法
Pull mirroring 失敗token 過期 / 權限不足;Settings → Repository → Mirror 看錯誤
新 commit 沒觸發檢查 webhook 設定;測試手動 trigger mirror
External PR 沒跑確認 OAuth 連線(非手動);workflow 沒 exclude
Status 沒回到外部檢查 PAT 權限;script 是否真的有跑
重複 pipelineworkflow:rules 控制
"Run CI/CD for external repository" 看不到Admin 沒啟用 import sources;mirroring 被關
