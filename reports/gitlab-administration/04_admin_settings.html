🗺️ Admin Area vs gitlab.rb 差別

GitLab 的「設定」分兩層,要分清楚才不會改錯地方:

gitlab.rb(第 3 章)Admin Area Settings(本章)
操作編輯檔案 + reconfigureUI 點選 + Save
層級基礎建設層(服務、埠、儲存)產品行為層(政策、規則)
儲存寫入 gitlab.yml寫入資料庫(application_settings 表)
重啟需要 reconfigure即時生效
典型例external_url、SMTP host是否開放註冊、密碼複雜度、CI 預設
💡 進入路徑:右上角大頭照旁 → Admin → Settings。僅 Admin 帳號看得到。
⚠️ GitLab.com 使用者沒有 Admin Area。只有 Self-Managed / Dedicated 管理員能進入。

👥 1. Sign-up / New user 限制

路徑:Admin → Settings → General → New user account restrictions

Quick Start內部站必做 3 件事

  • 取消勾選 Allow new user accounts(關閉公開註冊)
  • 或勾選 Require admin approval(開放但需審核)
  • 設定 Email confirmation:Hard(強制驗證信箱)
⚠️ 若 GitLab 開放於公網且未限制註冊,會被挖礦腳本大量註冊濫用 CI,是最常見的公網 GitLab 被攻擊方式。

SRE 實務細部項目

選項建議值(內部企業站)
Allow new user accounts❌ 關閉(管理員手動建帳號或走 LDAP/SAML)
Require admin approval✅ 開啟
Email confirmationHard
Minimum password length12 以上
Password complexity全勾(需數字、大寫、小寫、符號)
Allowlist domainscompany.com *.company.com
Denylist domains放拋棄式信箱(mailinator.com、guerrillamail.com…)
User cap依授權座位數設定

SRE 實務Allowlist / Denylist 格式

允許萬用字元 *,以空白、分號、逗號或換行分隔:

# 允許公司所有子網域
*.company.com

# 封鎖整個 .io 頂級域
*.io

# 同時多個
company.com *.company.com partner.com
💡 Allowlist 只在自助註冊時檢查;管理員手動建帳號或使用者改 email 不會被擋。

架構視角帳號生命週期治理

Sign-up 限制只是第一關。完整帳號治理應該包含:

  • 入職:透過 LDAP/SAML Just-In-Time 自動建立,並設 block_auto_created_users=true 先鎖住
  • 審核:管理員批准 + 指派初始 Group
  • 週期檢視:定期查 Admin → Users 排序 Last activity,dormant 超過 90 天封鎖
  • 離職:SCIM 自動 deprovision,或 API 自動化腳本
  • 審計:所有帳號變更記錄於 Audit Events

🔐 2. Sign-in 限制與 Admin Mode

路徑:Admin → Settings → General → Sign-in restrictions

Quick Start常用設定項

項目作用
Allow password authentication for web interface是否允許 Web 介面用密碼登入
Allow password authentication for Git over HTTP(S)是否允許 Git clone/push 用密碼
Disable password and passkey auth for SSO users強制 SSO 使用者只能走 SSO
Enforce two-factor authentication強制全站 2FA
Enforce 2FA for administrators只強制管理員 2FA
Two-factor grace period寬限期(hours),0 = 下次登入立刻要
Enable Admin Mode(重要)啟用管理員模式

SRE 實務Admin Mode(強烈推薦)

Admin Mode 是一個「日常操作時降權,動管理功能時才升權」的機制,類似 sudo:

  • 開啟後,管理員平常的 UI 只看得到自己有權的群組/專案
  • 要進 Admin Area,必須點 Enter Admin Mode 重新驗證
  • Admin Mode 自動 6 小時逾時(不可調整)
✅ 為什麼要開:管理員帳號若被盜用,沒開 Admin Mode 等於直接拿到全站控制權。開啟後至少要多一層驗證才能動管理功能。

SRE 實務Admin Mode 對 API 的影響

# 管理員 API 操作需要 admin_mode scope
# 建立 personal access token 時要勾選 admin_mode

# 使用 token 進行管理 API 呼叫
curl --header "PRIVATE-TOKEN: <token-with-admin_mode>" \
  "https://gitlab.example.com/api/v4/application/settings"
⚠️ Admin Mode 不保護 Git client(SSH/HTTPS)。也就是說,管理員仍可用 SSH key 直接操作 Git,不用再驗證一次。

架構視角緊急救援機制

外部 IdP(LDAP/SAML)掛點時怎麼登入?預先規劃:

  1. 保留至少 1 個本地 root 帳號(強密碼 + 2FA + 密封保存)
  2. 確認 Allow password authentication for web interface 沒關(否則 SSO 掛了沒人進得來)
  3. 透過 Rails console 緊急開啟密碼登入:
sudo gitlab-rails console
> ApplicationSetting.current.update!(password_authentication_enabled_for_web: true)

🔄 3. CI/CD 全站預設

路徑:Admin → Settings → CI/CD

Quick Start最常調整的項目

項目說明
Default to Auto DevOps pipeline預設啟用 Auto DevOps(建議關閉,專案自己決定)
Runner registration tokenShared runner 註冊 token
Continuous Integration and DeploymentCI artifact、cache 設定
Package RegistryPackages 保留政策
Container RegistryImage 清理政策

SRE 實務關鍵 CI 參數

參數建議原因
Maximum artifacts size (MB)500 ~ 2000防單一 job 上傳過大
Default artifacts expiration30 days減少儲存累積
Archive jobs after3 months歸檔舊 pipeline 日誌
Maximum includes150防 include 爆炸攻擊
CI/CD limits — pipeline size500單一 pipeline 最多 job 數

架構視角CI 資源治理

  • Shared Runner 使用量追蹤:設定 Pipeline minutes 配額(Premium+)
  • Runner Token 輪替:定期 reset,避免洩漏
  • Protected Environment:prod deploy 加審批
  • Instance-level CI/CD Variables:集中管理 secret(但仍建議外接 Vault)
  • Compliance Framework:強制特定 pipeline 配置(Ultimate)

📁 4. Repository / Visibility

路徑:Admin → Settings → General → Visibility and access controls

Quick Start內部站必設

項目建議值
Default project visibilityPrivate
Default group visibilityPrivate
Default snippet visibilityPrivate
Restricted visibility levels勾 Public(禁止任何人選 public)
Enabled Git access protocolsBoth SSH & HTTP(或只開 SSH)
Custom Git clone URL for HTTP(S)前面若有反向代理可調整
RSA SSH keysForbidden(SHA-1,不安全)
ECDSA / Ed25519 keys允許

SRE 實務Import sources 控制

同一頁面下方有 Import sources 區塊。除非有明確需求,否則只勾選實際會用到的來源(如 GitLab/GitHub),其他關閉。可減少攻擊面。

SRE 實務Repository 容量限制

路徑:Admin → Settings → General → Account and limit

# 預設 repository 大小上限(MB)
Maximum repository size:  10240  (10 GB)

# 附檔大小(單檔)
Maximum attachment size:  100 MB

# Push 單次大小
Maximum push size:        1024 MB

# Import 匯入檔大小
Maximum import size:      100 MB

架構視角何時需要分開限制?

  • Monorepo 團隊:提高限制,並考慮啟用 Git LFS partial clone
  • data team:限制極嚴(避免 model artifact 存 Git),引導至 Object Storage
  • 行動應用團隊:通常需要較大的 attachment 上限

可以透過 Group-level / Project-level override 針對特定團隊放寬,不必全站拉高。

🌐 5. Network / Rate Limit

路徑:Admin → Settings → Network

Quick Start先開哪些 Rate Limit?

  • Protected paths(/users/sign_in 等敏感路徑)
  • User and IP rate limits
  • Package registry rate limits
  • Issues creation rate limit
  • Files API rate limit

SRE 實務建議預設值

類別UnauthenticatedAuthenticated
API (req/min/IP or user)602000
Web (req/min)100800
Protected paths1010
Raw blob request300 / project300 / project
Search rate limit1030

SRE 實務Outbound Requests

路徑:Admin → Settings → Network → Outbound requests

控制 GitLab 自己可以往外連到哪些地方(Webhook 目的地、Import 來源)。預設允許本地網路(private IP),這在 SSRF 攻擊場景下可能有問題。建議:

  • ❌ 取消勾選 Allow requests to the local network from webhooks and integrations
  • ❌ 取消勾選 Allow requests to the local network from system hooks
  • ✅ 明確列出必要的內部 IP/網段到 Local IP addresses and domain names allowlist

架構視角Rate Limit 的監控與調校

Prometheus 指標:

gitlab_rate_limiting_events_total
gitlab_throttle_events_total

調校流程:

  1. 先設保守值、觀察 2 週
  2. 看 99th percentile 請求率
  3. 把上限設為 p99 × 1.5 左右
  4. 對 Runner / Webhook / Bot 加 allowlist
  5. 持續監控 rate_limited 事件

🎨 6. Appearance / 公告訊息

路徑:Admin → Settings → Appearance 及 Admin → Messages

Quick Start可以改什麼

  • 登入頁:公司 Logo / 登入頁 title / 登入頁描述
  • Favicon:瀏覽器 tab 圖示
  • 頁首 / 頁尾訊息:內部公告、合規聲明
  • Pipeline Emails:從 GitLab 發信的自訂 footer

SRE 實務Broadcast Messages(維運公告)

路徑:Admin → Messages

用於發布全站公告(例如維護時段)。支援:

  • 顯示時段:start / end datetime
  • Banner or Notification:Banner 橫跨上方,Notification 右下角
  • 針對特定 path:只在特定 URL 顯示
  • Dismissible:使用者是否可關閉
✅ 維護前 48 小時就設好公告,排好 start/end 時間,省得臨時手忙腳亂。

架構視角企業客製化選項

  • Enterprise templates:自訂 issue/MR 預設範本
  • Custom project templates:統一的 starter project(專案骨架)
  • Email custom additional text:合規聲明、保密條款
  • Terms of Service:強制新使用者接受條款

🔌 7. API 操作 Settings

Quick Start為什麼要用 API?

UI 適合臨時調整;但要做 IaC、多環境一致、變更可追蹤,就必須用 API:

# 讀取所有設定
curl --header "PRIVATE-TOKEN: <admin-token>" \
  "https://gitlab.example.com/api/v4/application/settings"

# 更新單一項目
curl --request PUT \
  --header "PRIVATE-TOKEN: <admin-token>" \
  --data "signup_enabled=false" \
  "https://gitlab.example.com/api/v4/application/settings"

SRE 實務批次更新範例(多參數)

curl --request PUT \
  --header "PRIVATE-TOKEN: $TOKEN" \
  --data-urlencode "signup_enabled=false" \
  --data-urlencode "require_admin_approval_after_user_signup=true" \
  --data-urlencode "minimum_password_length=14" \
  --data-urlencode "password_number_required=true" \
  --data-urlencode "password_symbol_required=true" \
  --data-urlencode "password_uppercase_required=true" \
  --data-urlencode "password_lowercase_required=true" \
  --data-urlencode "email_confirmation_setting=hard" \
  --data-urlencode "domain_allowlist[]=company.com" \
  --data-urlencode "domain_allowlist[]=*.company.com" \
  "https://gitlab.example.com/api/v4/application/settings"

架構視角把 Admin Settings 當作程式碼

# 範例:把 settings 存成 YAML,用 script 套用
# settings.yaml
signup_enabled: false
require_admin_approval_after_user_signup: true
minimum_password_length: 14
default_project_visibility: private
default_group_visibility: private
max_attachment_size: 100
session_expire_delay: 480   # 8 小時
# apply.sh
#!/bin/bash
yq -r 'to_entries | .[] | "--data-urlencode \"\(.key)=\(.value)\""' settings.yaml \
  | xargs curl --request PUT \
    --header "PRIVATE-TOKEN: $TOKEN" \
    "https://gitlab.example.com/api/v4/application/settings"
💡 把 settings.yaml 放進 Git repo,每次變更走 MR review。這樣 Admin Settings 也被納入 GitOps 流程。

🎯 新站 Day-1 設定 Checklist

  • 關閉 Sign-up(或設 admin approval)
  • Hard email confirmation
  • 密碼長度 ≥ 12,啟用複雜度
  • Default project/group visibility = Private
  • Restricted visibility 禁止 Public
  • 禁用 RSA SSH key
  • 啟用 Admin Mode
  • 啟用必要的 Rate Limits
  • 取消 Outbound → local network 勾選
  • 設定 session 逾時(建議 8 小時內)
  • 設定公司 logo / 合規頁尾
  • 把 Settings 匯出 yaml 存入版控

關閉註冊::Gitlab::CurrentSettings.update!(signup_enabled: false)
