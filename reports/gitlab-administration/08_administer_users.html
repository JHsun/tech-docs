🗺️ 使用者狀態總覽

GitLab 使用者有七種狀態,能準確區分是除錯成功的關鍵。

狀態說明佔用 seat可登入
Active正常使用✅ 是✅ 是
Pending approval已註冊待批准❌ 否❌ 否
Blocked被封鎖(可解)❌ 否❌ 否
Deactivated因長期不活躍停用❌ 否⚠️ 可自助啟用
Banned因濫用被永久封鎖❌ 否❌ 否
LDAP BlockedLDAP 端已移除❌ 否❌ 否
Deleted已刪除❌ 否❌ 否
💡 Block vs Deactivate 的差別:
  • Block:主動封鎖,使用者不能自己解。適用於離職、安全事件。
  • Deactivate:因閒置自動停用,使用者下次登入可自助恢復。適用於清季座位。

SRE 實務建立使用者的三種方式

方式路徑適用

架構視角帳號生命週期治理

  • 入職(Joiner):SSO JIT 自動建帳號 → 預設 block_auto_created_users=true 先鎖 → 管理員或自動化批准 → 指派到 Group
  • 異動(Mover):部門調動 → 透過 LDAP/SCIM group 映射自動調整 Group 成員
  • 離職(Leaver):SCIM 自動 deprovision → GitLab Block → 30 天後 Delete(保留 contribution)
  • 閒置:自動 Deactivate 政策(90 天)

⚡ 2. 常用動作:Block / Deactivate / Ban / Delete

Quick Start四種「停用」方式比較

動作使用者能感知資料可還原seat用途
Block登入被拒保留✅ 管理員可解不佔離職、安全事件
Deactivate收到通知信保留✅ 使用者自助不佔長期閒置清座位
Ban登入被拒保留⚠️ 需管理員解不佔濫用、垃圾訊息
Delete user帳號消失個人 repo 保留(歸 Ghost User)❌ 不可不佔永久移除
Delete user & contributions帳號消失個人 repo / issue / MR 全刪❌ 不可不佔GDPR 刪除

SRE 實務UI 操作路徑

統一入口:Admin → Overview → Users → 選擇使用者 → User administration ⋮

下拉選單會顯示目前狀態可用的動作:

  • Block / Unblock
  • Deactivate / Activate
  • Ban / Unban
  • Trust / Untrust
  • Delete user
  • Delete user and contributions

SRE 實務API 操作

# 找使用者 ID
curl --header "PRIVATE-TOKEN: $TOKEN" \
  "https://gitlab.example.com/api/v4/users?username=alice"

# Block
curl --request POST --header "PRIVATE-TOKEN: $TOKEN" \
  "https://gitlab.example.com/api/v4/users/$USER_ID/block"

# Unblock
curl --request POST --header "PRIVATE-TOKEN: $TOKEN" \
  "https://gitlab.example.com/api/v4/users/$USER_ID/unblock"

# Deactivate
curl --request POST --header "PRIVATE-TOKEN: $TOKEN" \
  "https://gitlab.example.com/api/v4/users/$USER_ID/deactivate"

# Ban
curl --request POST --header "PRIVATE-TOKEN: $TOKEN" \
  "https://gitlab.example.com/api/v4/users/$USER_ID/ban"

# Delete(不含 contribution)
curl --request DELETE --header "PRIVATE-TOKEN: $TOKEN" \
  "https://gitlab.example.com/api/v4/users/$USER_ID"

# Delete 並刪除所有 contribution
curl --request DELETE --header "PRIVATE-TOKEN: $TOKEN" \
  "https://gitlab.example.com/api/v4/users/$USER_ID?hard_delete=true"

架構視角Ghost User 概念

刪除使用者(非 hard delete)後,他的 commit、issue、MR 仍留著,但 author 會指向系統內的 "Ghost User"。這樣可保留歷史脈絡。

hard_delete=true 才會把這些 contribution 都刪掉,通常只在 GDPR / 合規要求時才用。

架構視角自動 Deactivation 政策

路徑:Admin → Settings → General → Account and limit

  • ✅ Dormant users:啟用自動停用
  • Period(天):預設 90(最少),可設更大
  • 對象:建立超過 7 天 + 無活動
💡 搭配 gitlab:dormant_users rake task 檢視,避免一次停掉太多人。

✅ 3. 審批與 Role Promotion

Quick StartPending approval 批准流程

  1. 使用者自助註冊(若已開放)或 SSO JIT 自動建立
  2. 狀態變 Pending approval
  3. 管理員:Admin → Overview → Users → 篩選 State=Pending approval
  4. 對每位使用者選 Approve 或 Reject
動作結果
Approve狀態變 Active、開始佔 seat、使用者可登入
Reject阻止登入 + 刪除帳號

SRE 實務Role Promotion Approval(Premium+)

若開啟 Administrator approval for role promotions,把使用者升到 billable role(Developer+)時需要管理員批准。

路徑:Admin → Overview → Users → Role Promotions

可一次批核多筆待升級請求。

架構視角企業審批流程

  • 全 SSO 環境:IdP 已經是企業 HR 系統,通常 JIT 自動批准即可
  • 公網站:必須手動批准,否則會被挖礦腳本大量濫用
  • Ultimate 等級:可結合 Guest role + Seat control 避免 seat 暴增
  • Role Promotion 審批:結合 ITSM(ServiceNow / Jira)流程

🎭 4. 角色與權限

Quick Start內建角色

角色Access level能做什麼(概述)
Guest10看 Issue、評論(Private 專案看不到程式碼)
Planner15規劃工作、改 Issue、不碰程式碼(Premium+)
Reporter20看程式、下載、建 Issue / label
Developer30Push 到非保護分支、建 MR、跑 CI
Maintainer40Merge 到保護分支、設 project、加成員
Owner50Group 擁有者、刪除 group / project
Minimal Access5占位用(SSO 同步但不給權限)

SRE 實務使用者層級屬性

除了 Group/Project 角色,使用者本身還有幾個關鍵旗標:

  • Admin:全站管理員
  • Auditor(Premium+):只能讀,不能改,用於合規審計
  • External:外部使用者,看不到 Internal 等級專案
  • Can create top-level group:是否能建頂層 Group(建議一般使用者關閉)
  • Can create projects:是否能建專案
  • Private profile:隱藏個人資料

架構視角External User 的重要性

若你的 GitLab 有「承包商 / 合作夥伴」之類的外部人員,一定要把他們設為 External:

  • 他們看不到 Internal visibility 的專案(除非被明確加入)
  • 只能存取被加入的 Group / Project
  • 預設不能建 top-level group

架構視角Custom Roles(Ultimate)

Ultimate 提供 Custom Roles,可以在內建角色上加細部權限(如「只讀程式碼」、「僅可批准 MR」)。適用於:

  • 外部審計人員
  • 合規團隊(只能看 vulnerability 但不能改)
  • 跨團隊的特殊權限組合

🏛️ 5. LDAP / SAML 使用者管理

Quick StartLDAP 使用者特徵

LDAP 使用者會多一個 Identities 頁籤,顯示 LDAP DN。特別注意:

  • LDAP 端被移除 → GitLab 端狀態變 LDAP Blocked
  • Unblock 可能灰掉:需先刪除 LDAP Identity 才能解
  • 密碼永遠以 LDAP 為準,GitLab 側改密碼無效

SRE 實務LDAP 常用維運

# 測試 LDAP 連線
sudo gitlab-rake gitlab:ldap:check

# 手動觸發 LDAP 同步(比排程快)
sudo gitlab-rake gitlab:ldap:sync

# 重新命名使用者(LDAP 來源改 DN 時)
sudo gitlab-rake gitlab:ldap:rename_provider

# 單一使用者強制同步
sudo gitlab-rails console
> user = User.find_by_username('alice')
> user.identities.where(provider: 'ldapmain').first.external_uid
> LdapGroupSyncWorker.new.perform(user.id)

SRE 實務SCIM 使用者管理(SAML)

SCIM 是「Identity Provider → GitLab」的使用者 lifecycle 同步協定。Premium+ 支援。

路徑:Admin → Overview → Users → 選擇使用者 → Identities 看到 SCIM identity

常見問題:

  • 使用者 email 對不上 → 從 IdP 端修正,GitLab 會 re-sync
  • SCIM 停用 → GitLab 端會 block,但要 IdP 端「恢復」才會 unblock

架構視角SSO 治理模型

目標機制
Provisioning(入職)SCIM + JIT
Authentication(登入)SAML / OIDC
Authorization(權限)Group Sync + Group/Project 成員
Deprovisioning(離職)SCIM 自動 block
AuditIdP log + GitLab Audit Events

🎭 6. Impersonate(模擬登入)

Quick Start為什麼要 impersonate?

管理員可以模擬任何使用者,以該使用者身分操作。典型用途:

  • 排查使用者回報的「我看不到 X」問題
  • 協助使用者操作(代為處理)
  • 驗證權限設定是否正確

Quick Start操作步驟

  1. Admin → Overview → Users → 選擇使用者
  2. 右上角 Impersonate
  3. 操作完畢 → 左上角 Stop impersonating
⚠️ 所有 impersonation 動作都會寫入 Audit Events,包含「誰模擬了誰、做了什麼」。

SRE 實務Impersonation Token(程式化)

管理員可為特定使用者建立 Impersonation Token,以 API 方式操作:

# 建立(管理員 token + 指定 user_id)
curl --request POST \
  --header "PRIVATE-TOKEN: $ADMIN_TOKEN" \
  --data "name=support-session" \
  --data "scopes[]=api" \
  --data "expires_at=2026-05-01" \
  "https://gitlab.example.com/api/v4/users/$USER_ID/impersonation_tokens"

# 之後用這個 token 操作 API,看起來就像該使用者操作

架構視角關閉 Impersonation

高敏感環境可以完全禁用 impersonation(不建議,會影響支援能力):

# /etc/gitlab/gitlab.rb
gitlab_rails['impersonation_enabled'] = false

若必須保留,建議:

  • 限定特定管理員才能用(透過 audit 監控)
  • Impersonation 事件觸發 real-time alert
  • 每季審視 impersonation 使用記錄

📚 7. 批次管理與 API

Quick Start從 UI 匯出使用者

路徑:Admin → Overview → Users → Export permissions as CSV

  • 最多匯出 100,000 個 active 使用者
  • 欄位:username、email、groups/projects 直接成員資格、角色、最後活動時間

SRE 實務批次建立使用者

#!/bin/bash
# bulk_create_users.sh
while IFS=',' read -r email username name; do
  curl --request POST \
    --header "PRIVATE-TOKEN: $ADMIN_TOKEN" \
    --data "email=$email" \
    --data "username=$username" \
    --data "name=$name" \
    --data "reset_password=true" \
    --data "skip_confirmation=true" \
    --data "external=true" \
    "https://gitlab.example.com/api/v4/users"
done < new_users.csv

SRE 實務找出不活躍使用者

# API 搜尋最後登入超過 90 天的 active 使用者
curl --header "PRIVATE-TOKEN: $TOKEN" \
  "https://gitlab.example.com/api/v4/users?active=true&last_activity_before=$(date -d '90 days ago' -Iseconds)"

# Rails console
sudo gitlab-rails console
> User.active.where('last_activity_on < ?', 90.days.ago)
>   .find_each { |u| puts "#{u.username},#{u.last_activity_on}" }

# 批次 deactivate(謹慎!)
> User.active.where('last_activity_on < ?', 90.days.ago)
>   .find_each { |u| u.deactivate }

架構視角使用者管理自動化

  • HR → GitLab:SCIM 是主力,不要手動管
  • Group membership → LDAP/SAML groups:改動在 IdP,GitLab 自動 sync
  • 定期健康檢查腳本:
    • 找出 90 天沒登入的,寄通知後 deactivate
    • 找出 2FA 未啟用的,寄提醒
    • 找出擁有高風險角色且長期閒置的,特別提醒 Owner review
  • 報表產出:每月匯出使用者-權限對照,給合規稽核

🚫 8. 濫用處理與 Moderation

Quick StartAbuse Reports

路徑:Admin → Abuse Reports

使用者可以檢舉其他使用者(政治垃圾、挖礦 CI、色情內容等)。管理員在此處理檢舉:

  • 查看檢舉內容
  • 直接從檢舉頁封鎖 / Ban 使用者
  • 或駁回檢舉

SRE 實務Trust / Untrust

預設新使用者是「不受信任」狀態,建立 issue / note / snippet 可能被當成 spam 檔住。管理員可以:

  • Trust user:建立內容不再被 spam 檢查
  • Untrust user:恢復預設狀態

路徑:Admin → Users → 選擇使用者 → User administration → Trust user

SRE 實務Spam Protection 設定

路徑:Admin → Settings → Reporting → Spam and Anti-bot Protection

  • reCAPTCHA:註冊 / 登入 / Issue 建立都可啟用
  • Akismet:若遇到大量 spam issue/comment
  • Disallowed IP:封鎖特定來源 IP

架構視角公網 GitLab 的防濫用

若 GitLab 對公網開放,高風險場景:

  1. 挖礦 CI(最常見):關閉自助註冊 / 限制 free runner / 限制 pipeline 用量
  2. Spam issue / comment:reCAPTCHA + Akismet + Trust list
  3. Fork 後濫用:限制 public fork 的 CI 執行
  4. 大量 API 嘗試:Rate Limit + Protected Paths
⚠️ GitLab.com 自己曾因自助註冊 + free runner 被大量挖礦濫用,後來改成手機驗證。Self-managed 若公開,必須做這些防護。

✅ 使用者治理 Checklist

  • 啟用 Dormant users 自動 deactivate(90+ 天)
  • External user 預設勾選(外部人員)
  • Can create top-level group 只給特定人
  • SCIM / LDAP 同步已啟用
  • 定期檢視 Admin 帳號清單
  • Audit Events 已存到外部系統
  • Impersonation 使用有監控
  • 離職流程有自動化 Block
  • Abuse Reports 有人 on-call 處理
  • 每季匯出 user 權限報表給合規

🆘 救援指令

# 重設使用者密碼(root 亦同)
sudo gitlab-rake "gitlab:password:reset[username]"

# 停用單一使用者 2FA
sudo gitlab-rails console
> admin = User.find_by_username('admin_user')
> user = User.find_by_username('stuck_user')
> TwoFactor::DestroyService.new(admin, user: user).execute

# 把某使用者設為 admin
> User.find_by_username('alice').update!(admin: true)

# 查使用者最後登入
> User.find_by_username('alice').current_sign_in_at

# 讓某使用者變 External
> User.find_by_username('contractor').update!(external: true)

# 列出所有 admin
> User.admins.pluck(:username, :email)

# 解鎖被 login 失敗鎖住的帳號
> User.find_by_username('alice').unlock_access!
