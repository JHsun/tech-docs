🗺️ Runner 總覽

GitLab Runner 是實際執行 CI/CD job 的獨立程式。GitLab Server 只負責排程與存放結果,job 的真正執行在 Runner。

┌──────────────┐       ┌──────────────┐       ┌──────────────┐
│ GitLab Server│──────▶│ Runner       │──────▶│  Executor    │
│ (scheduler)  │ poll  │ (manager)    │ run   │ (actual env) │
└──────────────┘       └──────────────┘       └──────────────┘
     .gitlab-ci.yml       config.toml           Docker / K8s /
     觸發 pipeline        決定怎麼跑            VM / Shell

Quick Start為什麼需要管 Runner?

  • GitLab.com 提供免費 runner,但有分鐘數限制
  • Self-Managed 預設不附 runner,必須自架
  • Dedicated 完全不提供 managed runner
  • 效能、成本、安全性考量常常要用自家 Runner

🧩 1. 核心概念:Runner / Executor / Manager

Quick Start名詞辨別

名詞定義
GitLab Runner整個執行器軟體(gitlab-runner 二進位)
Runner ManagerGitLab Runner 主程序,讀 config.toml 執行 jobs
Runner Configurationconfig.toml 中一個 [[runners]] 區塊
Runner(UI 顯示)UI 中看到的一個註冊紀錄(對應一個 token)
Executor執行 job 的方式(Docker / K8s / Shell / SSH / VM)
Machine實際跑 job 的 VM / Pod / Container

SRE 實務執行流程

1. 使用者 push 程式碼
2. GitLab Server 看到 .gitlab-ci.yml,建立 pipeline 與 jobs
3. Runner Manager 定期 poll GitLab:「有我可以做的 job 嗎?」
4. 有的話,GitLab 傳回 job payload + job token
5. Runner Manager 依 executor 設定建立環境(Docker container / K8s pod…)
6. Executor 在該環境 clone 程式碼、跑 script
7. 結果傳回 GitLab(log、artifact)

🎯 2. Runner 範圍:Instance / Group / Project

Quick Start三種範圍

範圍可用於誰能設定
Instance Runner全站所有 projectAdmin
Group Runner該 group 及子 group 的 projectGroup Owner
Project Runner單一 projectProject Maintainer

SRE 實務Tag 分流

Runner 可以打標籤(Tag),job 指定標籤後只有對應 runner 會接:

# .gitlab-ci.yml
build:
  tags:
    - docker
    - linux-x86
  script:
    - make

security-scan:
  tags:
    - gpu          # 只在 GPU runner 上跑
  script:
    - ./scan

常用 tag 策略:

  • linux / windows / macos — OS
  • docker / shell / k8s — executor
  • gpu / arm64 / amd64 — 硬體
  • prod / staging / secure — 環境/用途
  • fast / heavy — 規格

架構視角Runner 拓撲規劃

  • 共用池(Shared Pool):instance-level runner,給所有人用(預設)
  • 團隊專屬:Group runner,該團隊 SLA 保證
  • 敏感環境:Project-specific runner,執行 prod deploy,鎖 protected branch only
  • GPU / ARM / 特殊硬體:打 tag,job 主動指定

⚙️ 3. 七種 Executor 比較

Executor 隔離性 擴展性 速度 適用
Shell ❌ 無 ❌ 單機 ✅ 最快 簡單專案、內部測試
Docker ✅ 容器 ⚠️ 單機 ✅ 快 一般用途,最常見
Docker Machine(已廢棄) ✅ ✅ 自動擴展 VM 中等 🚫 不建議新用
Docker Autoscaler ✅ ✅ 雲端 VM 中等 Docker Machine 的繼任者
Kubernetes ✅ Pod ✅✅ 最強 中等 企業主力、已有 K8s
SSH ⚠️ 遠端機 ❌ 固定 中等 特殊硬體、legacy
VirtualBox / Parallels ✅ VM ❌ 慢 ❌ 慢 macOS / Windows 測試
Custom 自訂 自訂 自訂 AWS Fargate、自家 VM pool

架構視角決策指南

已有 Kubernetes?
├─ 是 ──▶ Kubernetes executor(Runner Helm chart)
└─ 否 ─┬─ 需要大量擴展?
       ├─ 是 ──▶ Docker Autoscaler(EC2 / GCP)
       └─ 否 ─┬─ 需要 container 隔離?
              ├─ 是 ──▶ Docker executor
              └─ 否 ──▶ Shell executor(不建議,只在特殊情況)

📥 4. 安裝與註冊

Quick StartLinux 安裝(最常見)

# Ubuntu / Debian
curl -L "https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh" | sudo bash
sudo apt-get install gitlab-runner

# RHEL / CentOS
curl -L "https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.rpm.sh" | sudo bash
sudo yum install gitlab-runner

# 確認版本
gitlab-runner --version
sudo gitlab-runner status

Quick Start註冊(新版 Authentication Token)

GitLab 15.10+ 改用在 UI 建立 Runner 取得 token 的流程,不再使用 registration token:

  1. GitLab UI → Admin / Group / Project → CI/CD → Runners → New instance/group/project runner
  2. 填寫描述、tags、是否鎖定專案
  3. 建立後取得 token(格式 glrt-xxx)
  4. 在 Runner 主機:
sudo gitlab-runner register \
  --url "https://gitlab.example.com" \
  --token "glrt-xxxxxxxxxxxxx" \
  --executor "docker" \
  --docker-image "alpine:latest" \
  --description "my-docker-runner"

SRE 實務非互動(自動化)註冊

sudo gitlab-runner register \
  --non-interactive \
  --url "https://gitlab.example.com" \
  --token "$RUNNER_TOKEN" \
  --name "ci-runner-prod-01" \
  --executor "docker" \
  --docker-image "alpine:latest" \
  --docker-privileged \
  --tag-list "docker,linux,amd64" \
  --locked="false" \
  --access-level="not_protected"

SRE 實務Windows / macOS 安裝

# Windows(PowerShell 管理員)
New-Item -Path 'C:\GitLab-Runner' -ItemType Directory
Invoke-WebRequest "https://s3.dualstack.us-east-1.amazonaws.com/gitlab-runner-downloads/latest/binaries/gitlab-runner-windows-amd64.exe" -OutFile "C:\GitLab-Runner\gitlab-runner.exe"
cd C:\GitLab-Runner
.\gitlab-runner.exe install
.\gitlab-runner.exe start

# macOS(需自行 daemonize)
brew install gitlab-runner
brew services start gitlab-runner

架構視角Runner 版本策略

⚠️ Runner 版本必須 >= GitLab Server 版本的前一個 major release。最佳作法:Runner 與 Server 保持同版本。

🛠️ 5. config.toml 設定重點

Quick Start檔案位置

OS / 部署路徑
Linux(Root)/etc/gitlab-runner/config.toml
Linux(User)~/.gitlab-runner/config.toml
WindowsC:\GitLab-Runner\config.toml
macOS/usr/local/etc/gitlab-runner/config.toml
KubernetesConfigMap(Helm chart values)

Quick Start基本結構

concurrent = 10            # 這個 manager 同時能跑幾個 job
check_interval = 0         # poll GitLab 間隔(秒,0=預設)
log_level = "info"

[session_server]
  session_timeout = 1800

[[runners]]
  name = "docker-runner"
  url = "https://gitlab.example.com"
  token = "glrt-xxx"
  executor = "docker"

  [runners.custom_build_dir]
  [runners.cache]
    [runners.cache.s3]
    [runners.cache.gcs]
    [runners.cache.azure]

  [runners.docker]
    tls_verify = false
    image = "alpine:latest"
    privileged = false
    disable_entrypoint_overwrite = false
    oom_kill_disable = false
    disable_cache = false
    volumes = ["/cache"]
    shm_size = 0

SRE 實務Docker Executor 進階設定

[runners.docker]
  image = "alpine:latest"
  privileged = false
  # 可用的 image(正則 / 萬用字元)
  allowed_images = ["my.registry.example.com/*:*", "alpine:*", "node:*"]
  allowed_services = ["my.registry.example.com/*:*"]
  # 拉取策略
  pull_policy = ["always"]           # always / if-not-present / never
  allowed_pull_policies = ["always", "if-not-present"]
  # 限制
  memory = "2g"
  cpus = "1.5"
  # 網路
  network_mode = "bridge"
  # 檔案掛載
  volumes = ["/cache", "/var/run/docker.sock:/var/run/docker.sock"]
  # 安全 capability
  cap_add = ["SYS_PTRACE"]
  cap_drop = ["SYS_ADMIN"]

SRE 實務快取設定(S3 範例)

[runners.cache]
  Type = "s3"
  Shared = true
  [runners.cache.s3]
    ServerAddress = "s3.amazonaws.com"
    BucketName = "gitlab-runner-cache"
    BucketLocation = "ap-northeast-1"
    AccessKey = "AKIA..."
    SecretKey = "..."
    # 或用 IAM Role(推薦)
    # AuthenticationType = "iam"
✅ 用 S3 共享 cache 後,跨 runner 的 job 也能重用 dependencies,明顯加速 pipeline。

架構視角設定管理最佳實踐

  • 版本控制:把 config.toml(mask token)放進 Git repo
  • IaC 推送:Ansible / Terraform 管理
  • Template 功能:用 --template-config 註冊,固定部分設定
  • 敏感值外部化:token、S3 key 從 Vault / env 拿,不寫死
  • config.toml 權限:chmod 600,root only

☸️ 6. Kubernetes Executor 詳解

Quick StartHelm 安裝

# 加入 Helm repo
helm repo add gitlab https://charts.gitlab.io
helm repo update

# 建立 values.yaml
cat > values.yaml <<EOF
gitlabUrl: https://gitlab.example.com
runnerRegistrationToken: "glrt-xxxx"  # 新版用 runnerToken

rbac:
  create: true

concurrent: 10

runners:
  config: |
    [[runners]]
      [runners.kubernetes]
        namespace = "gitlab-runner"
        image = "ubuntu:22.04"
        privileged = false
        cpu_request = "500m"
        memory_request = "1Gi"
        poll_timeout = 600
EOF

# 部署
kubectl create ns gitlab-runner
helm install gitlab-runner -n gitlab-runner -f values.yaml gitlab/gitlab-runner

SRE 實務K8s Pod 結構

Kubernetes executor 為每個 job 開一個新 pod,內含:

  • build container:跑 script 的主體
  • helper container:clone、下載 cache/artifact、上傳結果
  • svc-0 / svc-1 / …:services: 定義的每個服務(DB 等)

這些容器共用同一個 localhost,所以 services 可以直接透過 127.0.0.1 存取。

SRE 實務資源限制

[runners.kubernetes]
  # 請求 / 上限
  cpu_request = "500m"
  cpu_limit = "2"
  memory_request = "1Gi"
  memory_limit = "4Gi"

  # 每個 service 容器的資源
  service_cpu_request = "100m"
  service_memory_request = "256Mi"

  # Helper 容器
  helper_cpu_request = "100m"
  helper_memory_request = "128Mi"

  # Node 選擇
  node_selector = { "workload" = "ci" }

  # Toleration
  [[runners.kubernetes.node_tolerations]]
    key = "ci-only"
    operator = "Equal"
    value = "true"
    effect = "NoSchedule"

SRE 實務Image 白名單

[runners.kubernetes]
  allowed_images = ["my.registry.example.com/*:*"]
  allowed_services = ["my.registry.example.com/*:*"]
  allowed_pull_policies = ["always", "if-not-present"]

結合內部 registry,強制所有 CI 只用公司掃過的 image。

架構視角K8s Runner 最佳實踐

  • 獨立 node pool:CI 流量尖峰大,隔離避免影響 prod workload
  • Spot / Preemptible VM:大幅降低成本
  • Cluster Autoscaler:動態擴節點
  • NetworkPolicy:限制 runner pod 能連去哪
  • Pod Security Standards:嚴格 mode + 禁 privileged
  • DinD:若需要 build image,優先考慮 Kaniko / BuildKit 而非 Docker-in-Docker

📈 7. 自動擴展(Autoscaling)

Quick Start三種擴展模式

模式擴展單位冷啟動
Kubernetes executorPod秒級
Docker AutoscalerVM(EC2 等)1-2 分鐘
Custom(Fargate)Task30 秒

SRE 實務Docker Autoscaler(AWS 範例)

Docker Autoscaler 是 Docker Machine 的繼任者,官方推薦新用這個。

[[runners]]
  name = "autoscaler-aws"
  url = "https://gitlab.example.com"
  token = "glrt-xxx"
  executor = "docker-autoscaler"

  [runners.docker]
    image = "alpine:latest"

  [runners.autoscaler]
    plugin = "fleeting-plugin-aws"
    capacity_per_instance = 2    # 每台 VM 能跑幾個 job
    max_use_count = 100          # 跑完幾個 job 後丟掉
    max_instances = 20

    [runners.autoscaler.plugin_config]
      name             = "gitlab-runner-asg"
      profile          = "default"
      region           = "ap-northeast-1"

    [[runners.autoscaler.policy]]
      periods          = ["* * 9-18 * * mon-fri *"]
      timezone         = "Asia/Taipei"
      idle_count       = 5     # 上班時間預熱 5 台
      idle_time        = "30m"

    [[runners.autoscaler.policy]]
      periods          = ["* * * * * * *"]
      idle_count       = 1     # 其他時間 1 台

架構視角擴展策略考量

  • Off-hours idle count = 0:省錢,但接第一個 job 會慢
  • Business hours 預熱:常用規格先準備好
  • Spot 混用:非 critical job 用 spot,critical 用 on-demand
  • Monitor queue depth:若 job 等太久,提高 idle_count

🔒 8. 安全考量

⚠️ Runner 是 CI/CD 攻擊面最大的一塊。Job 能存取 Runner 主機、拉 image、跑任意程式、讀 secret。設計上要假設 job 是「不可信的」。

Quick Start基礎安全原則

  • 不要用 Shell executor 跑不可信程式碼
  • Docker executor 不要開 privileged=true(除非絕對必要)
  • Runner 主機與 GitLab Server 分離
  • Runner 主機不要放 production 憑證
  • 每 job 用乾淨 container/pod,不重用
  • 限制 allowed_images 只用公司內部 registry

SRE 實務常見高風險設定

設定風險建議
privileged = true容器可逃逸到主機用 Kaniko / BuildKit 代替 DinD
Shell executorJob 直接跑在主機用 Docker / K8s executor
Runner 有 AWS admin IAMJob 可控制 AWS最小權限 IAM,分 role
掛載 /var/run/docker.sock等同 root改用 sibling container 模式
未限制 allowed_images可跑任意 image白名單內部 registry
Network 無限制可攻擊內網NetworkPolicy / firewall

架構視角分層 Runner 設計

依信任等級分開 Runner:

  • Tier 1 — Untrusted(Fork / Public CI):最嚴格,沙箱化,無生產金鑰存取
  • Tier 2 — Internal(主分支以外):標準隔離,可讀非生產資源
  • Tier 3 — Protected(主分支 / tag):可用部署金鑰,限制 Protected branches/tags
  • Tier 4 — Deploy(生產部署):專用 Runner,單一用途,強 audit

用 Protected tag + Runner access_level=ref_protected 來實現分層。

架構視角Runner 主機加固

  • OS 最小化安裝
  • 定期 patch
  • 僅開必要 port(egress only)
  • Log 送到集中系統
  • osquery / auditd 監控異常行為
  • 定期輪替 runner token

📊 9. 監控

Quick Start啟用 Prometheus endpoint

# config.toml 加這行
listen_address = "[::]:9252"

或在註冊時 --listen-address。

SRE 實務關鍵指標

Metric意義
gitlab_runner_jobs當前執行中 job 數
gitlab_runner_concurrentconcurrent 設定值
gitlab_runner_limit單 runner limit
gitlab_runner_request_concurrency同時 poll 的請求數
gitlab_runner_request_concurrency_exceeded_totalpoll 超過 limit 的次數
gitlab_runner_errors_total累計錯誤
gitlab_runner_api_request_statuses_totalAPI 請求狀態分布
gitlab_runner_autoscaling_machine_creation_duration_seconds擴展 VM 耗時
gitlab_runner_autoscaling_machine_statesmachine 各狀態數量

SRE 實務常用告警

# Runner 擠滿了
- alert: RunnerSaturated
  expr: gitlab_runner_jobs / gitlab_runner_concurrent > 0.9
  for: 10m

# Runner 錯誤率攀升
- alert: RunnerErrorRate
  expr: rate(gitlab_runner_errors_total[5m]) > 0.5
  for: 10m

# 擴展機器常失敗
- alert: AutoscaleCreationSlow
  expr: histogram_quantile(0.95,
    rate(gitlab_runner_autoscaling_machine_creation_duration_seconds_bucket[10m])
  ) > 180
  for: 15m

架構視角GitLab 端的 Runner 健康

GitLab Admin Area → CI/CD → Runners 可以看到所有註冊 Runner 的狀態:

  • Online(< 1.5 分鐘內有聯繫)
  • Offline
  • Stale(3+ 月沒聯繫,可考慮清除)
  • Last contact、version、tag

API 查詢:

# 列出所有 runner
curl --header "PRIVATE-TOKEN: $TOKEN" \

✅ Runner 建置 Checklist

  • Runner 主機與 GitLab 分離
  • OS 最小化、定期 patch
  • 用 Docker / K8s executor(非 Shell)
  • 禁用 privileged(若需 build image 用 Kaniko)
  • 設定 allowed_images 白名單
  • Token 最小權限 + 定期輪替
  • Cache 用 S3/GCS 共享
  • Prometheus metrics 已啟用
  • 有告警規則(飽和、錯誤率)
  • config.toml 納入版本控制
  • 有分層 Runner(trusted / untrusted)
  • Production Runner 限定 protected branch
  • Autoscale idle count 依時段調整
