🗺️ 什麼是 CI/CD Component?

CI/CD Component 是 GitLab 16.6 後推出的可重用 pipeline 設定單元。類似 GitHub Actions、Jenkins Shared Library 的概念,但更整合。

Quick StartComponent vs include / extends

includeextendsComponent
重用粒度整個檔案單一 jobPipeline 片段
版本控制依 ref無Semantic Version ✅
輸入參數無原生支援無spec:inputs ✅
Catalog 可見否否✅ 可發布
文件無無README + 標準格式
💡 Component 主要解決「團隊間共享 CI 設定」的問題。若只在單專案內重用,extends 仍是首選。

架構視角CI/CD Catalog

CI/CD Catalog 是 GitLab 提供的元件市集:

  • 瀏覽與發現社群 / GitLab 官方的 Component
  • GitLab 官方元件(如 SAST、Code Quality、Secret Detection)位於 gitlab.com/components
  • Self-Managed 可建立自家 Catalog,讓內部團隊共享

🎯 1. 使用現有 Component

Quick Start最簡 include

include:
  - component: $CI_SERVER_FQDN/components/code-quality/codequality@1.0.0
    inputs:
      stage: test

Quick Start完整範例

stages: [build, test, deploy]

include:
  # GitLab 官方 SAST
  - component: $CI_SERVER_FQDN/components/sast/sast@1
    inputs:
      stage: test

  # 自家團隊元件
  - component: $CI_SERVER_FQDN/my-group/ci-components/docker-build@2.1.0
    inputs:
      image_name: myapp
      stage: build
      dockerfile: Dockerfile.prod

  # 同專案內測試(未發布)
  - component: $CI_SERVER_FQDN/$CI_PROJECT_PATH/templates/test@$CI_COMMIT_SHA
    inputs:
      python_version: "3.12"

# 你自己的 job
test:
  script: npm test

SRE 實務URL 格式

# 格式:
# [FQDN]/[組織]/[專案]/[元件名稱]@[版本]

# 範例(GitLab.com):
gitlab.com/components/sast/sast@4.0.0

# Self-Managed:
gitlab.example.com/devops/ci-components/docker-build@2.1.0

# 動態(用 CI 變數):
$CI_SERVER_FQDN/my-group/ci-components/lint@~1

# 同專案內(開發 / 測試用):
$CI_SERVER_FQDN/$CI_PROJECT_PATH/my-component@$CI_COMMIT_SHA
⚠️ URL 裡的「元件名稱」是 templates/ 下的資料夾或檔案名,不是專案名。見下節 "建立元件"。

🔖 2. 版本策略

Quick Start五種版本指定方式

形式範例建議
完整版本@1.0.0⭐ 最推薦(prod)
部分版本(patch 範圍)@1.0 → 最新 1.0.x✅ 自動拿 patch 修正
部分版本(minor 範圍)@1 → 最新 1.x.x⚠️ 會自動升 minor
latest@~latest❌ 不建議(可能含 breaking change)
分支@main開發測試用,不建議 prod
Commit SHA@abc123...開發期,最精確

SRE 實務版本策略決策

# 方式 1:釘死特定版本(最安全)
include:
  - component: $CI_SERVER_FQDN/components/sast/sast@4.2.1
    # 更新需要手動改版本號

# 方式 2:接受 patch 更新
include:
  - component: $CI_SERVER_FQDN/components/sast/sast@4.2
    # 自動拿到 4.2.0、4.2.1、4.2.2...

# 方式 3:接受 minor 更新(有風險)
include:
  - component: $CI_SERVER_FQDN/components/sast/sast@4
    # 會拿到 4.x.x 最新

# 方式 4:用 commit SHA(開發測試用)
include:
  - component: $CI_SERVER_FQDN/my-group/components/test@abc1234567

架構視角Semantic Versioning 契約

版本改變含意
MAJOR(1.x → 2.x)破壞性變更,使用者必須修改
MINOR(1.2 → 1.3)新增功能,向下相容
PATCH(1.2.3 → 1.2.4)bug fix,向下相容
💡 「釘 major、自動 minor/patch」是最常見的折衷: 
# 用 @1 接受 1.x 自動更新,但不會跳到 2.x
- component: $CI_SERVER_FQDN/components/sast/sast@1

🛠️ 3. 建立自己的 Component

Quick Start最小結構

my-ci-components/               # 專案根
├── README.md                   # 必要:使用說明
├── .gitlab-ci.yml              # 必要:元件自己的 CI(發布用)
├── LICENSE                     # 建議
└── templates/
    ├── build.yml               # 單檔元件
    └── deploy/
        └── template.yml        # 多檔元件

Quick Start第一個 Component

檔案 templates/docker-build.yml:

spec:
  inputs:
    image_name:
      description: "要 build 的 image 名稱"
      # 沒 default 就是必填

    dockerfile:
      default: Dockerfile
      description: "Dockerfile 路徑"

    stage:
      default: build

    push:
      default: false
      type: boolean
      description: "是否推送 registry"
---
# 下面是實際的 job 定義(---- 以上是 spec,以下是內容)
"docker-build-$[[ inputs.image_name ]]":
  stage: $[[ inputs.stage ]]
  image: docker:24
  services:
    - docker:24-dind
  script:
    - docker build -t $[[ inputs.image_name ]]:$CI_COMMIT_SHA -f $[[ inputs.dockerfile ]] .
    - |
      if [ "$[[ inputs.push ]]" = "true" ]; then
        docker push $[[ inputs.image_name ]]:$CI_COMMIT_SHA
      fi

SRE 實務spec:inputs 型別

spec:
  inputs:
    # 1. string(預設)
    name:
      type: string
      default: "hello"

    # 2. number
    retries:
      type: number
      default: 3

    # 3. boolean
    debug:
      type: boolean
      default: false

    # 4. array
    tags:
      type: array
      default: [docker, linux]

    # 5. 限定選項
    env:
      default: staging
      options:
        - staging
        - production

    # 6. 正則驗證
    branch_name:
      regex: '^[a-z][a-z0-9-]*$'
      description: "必須小寫字母開頭"

SRE 實務Inputs Interpolation 語法

在 template 裡用 $[[ inputs.xxx ]] 引用,雙中括號:

# ✅ 正確:產生 pipeline 時插值
script:
  - echo "Building $[[ inputs.image_name ]]"

# ❌ 錯誤:一般變數展開在 script 執行期,此時 inputs 已插值完
# script: - echo "$inputs.image_name"   # 拿不到

# ⚠️ 注意:$[[ inputs.x ]] 在 pipeline 建立期解析,
# $CI_COMMIT_SHA 等變數在 job 執行期解析

架構視角多元件專案結構

一個專案可放多個元件(版本一起 bump):

my-ci-components/
├── README.md
├── .gitlab-ci.yml
└── templates/
    ├── build.yml              # → my-ci-components/build@x.x.x
    ├── test.yml               # → my-ci-components/test@x.x.x
    └── deploy/
        └── template.yml       # → my-ci-components/deploy@x.x.x

規則:

  • 單檔元件:templates/NAME.yml
  • 多檔元件:templates/NAME/template.yml(內附文件、scripts 等)
  • 單一專案最多 100 個元件(18.5 之後)
  • 所有元件版本統一(無法單獨發某個元件)

架構視角元件內測試元件自己

# .gitlab-ci.yml(元件專案自己的 CI)
stages: [test, release]

# 測試自己
test-docker-build:
  stage: test
  trigger:
    include:
      - component: $CI_SERVER_FQDN/$CI_PROJECT_PATH/docker-build@$CI_COMMIT_SHA
        inputs:
          image_name: test-app
          push: false
    strategy: depend

# Release
create-release:
  stage: release
  image: registry.gitlab.com/gitlab-org/cli:latest
  rules:
    - if: $CI_COMMIT_TAG =~ /^\d+\.\d+\.\d+$/
  script:
    - echo "Release $CI_COMMIT_TAG"
  release:
    tag_name: $CI_COMMIT_TAG
    description: "Release $CI_COMMIT_TAG"

📢 4. 發布到 Catalog

Quick Start發布步驟

  1. 專案設為公開(Public)或 Internal
  2. 啟用 CI/CD Catalog project:Settings → General → Visibility → CI/CD Catalog project toggle
  3. 用 release job 建立 release(不能用 Releases API)
  4. 打 semantic version tag(1.0.0),推到 GitLab
  5. tag pipeline 跑 release job,元件就進 Catalog

SRE 實務典型 release 流程

# 元件專案的 .gitlab-ci.yml
stages: [validate, test, release]

# 1. 驗證元件格式
validate:
  stage: validate
  image: alpine:latest
  script:
    - |
      for f in templates/*.yml templates/*/template.yml; do
        [ -f "$f" ] && grep -q "^spec:" "$f" \
          && echo "PASS $f" \
          || echo "FAIL $f"
      done

# 2. 自我測試(用 trigger 跑自己)
test-self:
  stage: test
  trigger:
    include:
      - component: $CI_SERVER_FQDN/$CI_PROJECT_PATH/my-component@$CI_COMMIT_SHA
        inputs: {...}
    strategy: depend

# 3. 只在 tag 時發布
release:
  stage: release
  image: registry.gitlab.com/gitlab-org/cli:latest
  rules:
    - if: $CI_COMMIT_TAG =~ /^\d+\.\d+\.\d+$/
  script: echo "Releasing $CI_COMMIT_TAG"
  release:
    tag_name: $CI_COMMIT_TAG
    description: "See CHANGELOG.md"

SRE 實務打 tag 發布

git tag 1.0.0
git push origin 1.0.0
# tag pipeline 跑完 release job 後,
# 在 CI/CD Catalog 就能搜尋到

🔒 5. 安全與最佳實踐

Quick Start使用第三方元件的原則

⚠️ 任何人都能發布到 Catalog。第三方元件有風險,必須自行審查。GitLab 不保證第三方元件安全。
  • 審查原始碼:看有沒有奇怪的 curl / 敏感資料存取
  • 釘死版本:用 commit SHA 或完整版本,不用 ~latest
  • 最小權限 token:不讓元件拿到你的 production 金鑰
  • 監控 token 使用:定期 audit 元件用了什麼 credential
  • 定期審查更新:元件升級前看 changelog

SRE 實務建立自家 Catalog 的好處

場景好處
大型組織統一 DevOps 標準(SAST、部署、安全)
合規要求內部元件通過安全審查
離線環境不依賴外部 Catalog
多團隊共享一處管理,多處使用

架構視角元件設計原則

  • 單一職責:一個元件做一件事(build、test、deploy)
  • 最少 inputs:只暴露必要參數,其他走合理預設
  • 有 README:含用法、inputs 表格、variables 說明
  • 有 regex 驗證:對敏感 input(如 branch、URL)加驗證
  • 向後相容:minor / patch 版本不破壞既有使用
  • Dogfooding:自家專案先用,再發布

架構視角version + image 配對

元件若依賴特定 Docker image,可用 $[[ component.version ]] 保持一致:

spec:
  component: [version, reference]
  inputs:
    stage:
      default: test
---
run-tool:
  stage: $[[ inputs.stage ]]
  # 自動用與元件同版本的 image
  image: $CI_REGISTRY_IMAGE/my-tool:$[[ component.version ]]
  script:
    - ./run-tool

🎯 版本指定速查

寫法解析
@1.2.3釘死 1.2.3
@1.2最新 1.2.x
@1最新 1.x.x
@~latestCatalog 最新(有風險)
@mainmain 分支
@abc1234commit SHA

✅ 發布 Catalog Checklist

  • 專案 Public 或 Internal
  • Settings → 啟用 CI/CD Catalog project
  • 有 README.md(含 Overview、Components、Usage、Inputs)
  • 每個元件有 spec:inputs 與 description
  • 有 release job(用 release keyword)
  • 用 semantic version 打 tag
  • 打 tag 後檢查 Catalog 是否出現

🆘 常見問題

問題解法
Component not foundURL 格式、版本是否存在、專案是否公開
Spec must be valid json schemaspec:inputs 不可為空,要全拿掉或填內容
~latest 抓不到版本確認已發布到 Catalog(Public + 有 release)
Inputs 插值沒作用用 $[[ inputs.x ]],不是 $inputs.x
版本升級破壞 pipeline改用 @1.2 或釘死,避免 @1
