🗺️ CI/CD 核心觀念

🔄 CI(Continuous Integration)

每次 push 自動編譯、跑測試、靜態檢查,確保主幹永遠是可運作的。

🚀 CD(Continuous Delivery / Deployment)

延續 CI,自動或按需部署到 staging / production。

📄 Pipeline

一條自動化流程,由多個 stage 組成,每個 stage 有多個 job。

⚙️ Runner

實際執行 job 的獨立程式(Docker / K8s / VM)。

在 GitLab,所有 CI/CD 行為都定義在 repo 根目錄的 .gitlab-ci.yml。每次 push 或 MR 都會觸發 pipeline。

✅ 1. 前置需求

Quick Start開始之前你需要

  • 一個 GitLab 專案(GitLab.com 或 Self-Managed)
  • 該專案的 Maintainer 或 Owner 角色
  • 至少一個 runner 可用(GitLab.com 已內建免費 shared runner)

SRE 實務確認 Runner 可用

路徑:Project → Settings → CI/CD → Runners

  • 綠色圓圈 = Runner 在線,會接 job
  • 灰色圓圈 = Offline
  • 若全無 runner:GitLab.com 預設會給;Self-Managed 需管理員註冊

架構視角CI/CD 專案規劃

開始寫 .gitlab-ci.yml 前,先回答:

  • 什麼時候觸發?每次 push?只 MR?還是排程?
  • 要做什麼?build / test / scan / deploy?
  • 產物去哪?artifact、container registry、Pages、外部服務?
  • 誰能改設定?Protected branch / environment?
  • 密鑰怎麼管?CI variable / Vault / File?

🚀 2. 第一條 pipeline

Quick Start最小可運作範例

在 repo 根目錄建立 .gitlab-ci.yml:

test-job:
  script:
    - echo "This is my first job!"
    - date

commit + push → 去 Build → Pipelines 看結果。

Quick Start加上 stages

stages:
  - build
  - test
  - deploy

build-job:
  stage: build
  script:
    - echo "Welcome, $GITLAB_USER_LOGIN!"

test-job1:
  stage: test
  script:
    - echo "This job is a test"

test-job2:
  stage: test
  script:
    - echo "This takes longer"
    - sleep 30

deploy-prod:
  stage: deploy
  script:
    - echo "Deploying from $CI_COMMIT_BRANCH branch."
  environment: production
💡 同一 stage 的 job 並行執行;不同 stage 依序。若 stage 內任一 job 失敗,整個 stage 標記失敗,後續 stage 預設不會跑。

SRE 實務使用 Pipeline Editor

路徑:Project → Build → Pipeline editor

相比直接改檔案的優勢:

  • 即時語法檢查(不用 push 才發現錯)
  • Visualize 預覽 pipeline 流程圖
  • merged YAML 功能 — 展開 include / extends
  • 可看完整 include 之後最終設定

🔬 3. .gitlab-ci.yml 解剖

Quick Start典型結構

# 1. 全域設定(選用)
default:
  image: node:20
  timeout: 1h

# 2. 變數(選用)
variables:
  APP_ENV: staging

# 3. Stages 順序定義
stages:
  - build
  - test
  - deploy

# 4. 匯入其他設定(選用)
include:
  - local: '/ci/shared.yml'

# 5. Job 定義
build-app:
  stage: build
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/

test-app:
  stage: test
  script:
    - npm test
  needs: [build-app]

deploy-prod:
  stage: deploy
  script:
    - ./deploy.sh
  environment: production
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      when: manual

SRE 實務必學的 10 個關鍵字

關鍵字用途
stage指定 job 屬於哪個 stage
scriptjob 要執行的 shell 指令(必填)
image該 job 用什麼 Docker image
before_script / after_scriptscript 前後執行
variablesjob 層級變數
rules決定 job 何時執行
needs跨 stage 依賴(DAG pipeline)
artifacts保留的產物(給下游 job 或下載)
cache加速的快取(如 node_modules)
environment標示部署目標環境

架構視角YAML 組織策略

  • 單檔 vs 拆檔:<200 行放單檔;複雜時用 include 拆
  • hidden jobs(. 開頭):定義可被 extends 繼承的模板
  • anchor & alias(&, *):少用,可讀性差
  • CI/CD components:跨專案重用(看第 6 章)
  • 一致縮排:用 2 空格,不混 tab

📖 4. 關鍵術語

術語定義
Pipeline一次觸發產生的整組執行(由多 stage 組成)
Stagejob 的邏輯分組,依序執行(build → test → deploy)
Jobpipeline 的最小執行單位,跑在 runner 上
Runner執行 job 的代理程式
Executorrunner 執行 job 的方式(Docker/K8s/Shell)
Artifactjob 留下的產物檔案,可供下游 job 用或下載
Cache加速用的快取(例如 dependencies)
Variablekey-value 設定值,預定義的以 CI_ 開頭
Environment邏輯部署目標(production、staging 等)
Trigger啟動 pipeline 的事件(push、MR、schedule 等)

SRE 實務Artifact vs Cache 常搞混

ArtifactCache
用途job 產物(測試報告、build 輸出)加速(dependencies)
作用域pipeline 內傳遞 + UI 下載僅加速下一次同類型 job
是否保留是(會存到 GitLab)否(runner 端 best-effort)
可過期artifacts:expire_in自動 LRU
範例build 出來的 dist/node_modules/、.m2/

👁️ 5. 在 UI 看 pipeline

Quick Start常用入口

  • Build → Pipelines — 列出所有 pipeline
  • Build → Jobs — 跨 pipeline 的 job 清單
  • Build → Pipeline editor — 設定檔編輯
  • MR 頁面下方 — 該 MR 的 pipeline 即時狀態
  • Commit 頁面右上 — 該 commit 觸發的 pipeline

SRE 實務讀懂 Pipeline 圖

點擊 Pipeline ID 可看到圖形化流程。每個 job 的顏色代表狀態:

  • 🟢 成功 (passed)
  • 🔴 失敗 (failed)
  • 🟡 執行中 / 手動待觸發 (manual)
  • ⚪ 等待中 (pending)
  • ⚫ 被 skip (skipped)
  • 🔵 警告(allow_failure 的失敗)

SRE 實務Job 頁面能做什麼

  • 看完整 log(可下載、可展開折疊區段)
  • Retry 重跑該 job
  • Cancel 取消執行中
  • Erase job log(若有敏感資料)
  • 查看 artifacts、取得下載連結
  • debug → Run with trace(CI_DEBUG_TRACE=true)

🏗️ 6. 從簡單到完整(build → test → deploy)

Quick Start典型 Node.js 靜態站範例

stages:
  - build
  - test
  - deploy

default:
  image: node:20

.standard-rules:
  rules:
    - if: $CI_PIPELINE_SOURCE == 'merge_request_event'
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH

build-job:
  extends: .standard-rules
  stage: build
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - build/

lint-markdown:
  extends: .standard-rules
  stage: test
  dependencies: []
  script:
    - npm install markdownlint-cli2 --global
    - markdownlint-cli2 "docs/**/*.md"
  allow_failure: true

test-html:
  extends: .standard-rules
  stage: test
  dependencies: [build-job]
  script:
    - npm install --save-dev htmlhint
    - npx htmlhint build/

pages:
  stage: deploy
  image: busybox
  script:
    - mv build/ public/
  artifacts:
    paths:
      - public/
  rules:
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH

SRE 實務這個範例的重點

  • default:把所有 job 預設 image 設為 node:20
  • .standard-rules:hidden job 定義共用規則,用 extends 繼承
  • dependencies: []:這個 job 不需要上游 artifact
  • allow_failure: true:lint 失敗不會擋 pipeline
  • pages:GitLab 內建的 Pages 部署機制,把 public/ 當作網站根目錄

架構視角進階概念預習

  • DAG Pipelines:用 needs 建立 job 間依賴,不必等整個 stage(見第 4 章)
  • Child Pipelines:動態產生子 pipeline,用於 monorepo
  • Multi-project Pipelines:跨專案串連
  • Merge Request Pipelines:只在 MR 觸發,與一般 branch pipeline 區分
  • Merge Trains:高並發 merge 時的排隊機制

✅ 第一條 Pipeline Checklist

  • Repo 根目錄有 .gitlab-ci.yml
  • 至少一個 runner online
  • 至少一個 job,含 script
  • Push 後去 Build → Pipelines 確認
  • 失敗時:點 job 看 log、檢查 runner tag、看 YAML lint

🆘 新手最常遇到的錯

症狀原因
pending 很久runner tag 不對、runner offline、runner 被占滿
YAML invalid縮排錯、必填欄位(script)缺少
job 失敗但看不懂打開 CI_DEBUG_TRACE=true 看詳細 log
artifact 找不到上游 job 失敗?dependencies 設錯?
變數是空的在 only/rules 評估階段,部分變數尚未可用
改了沒反應忘記 commit,或 pipeline 用到 cached YAML
