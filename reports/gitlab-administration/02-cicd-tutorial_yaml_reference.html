🗂️ 關鍵字分類

GitLab CI YAML 關鍵字依作用範圍分為三類:

分類作用範圍代表關鍵字
Header檔案最上方(CI/CD Component 用)spec
Global整個 pipelinestages、workflow、default、variables、include
Job單一 jobscript、image、rules、needs、artifacts 等 30+ 個
💡 本章採「分類 → 關鍵字 → 範例」格式,讓你快速查到任一關鍵字的用法。若找不到,記得去 官方 YAML reference。

📋 1. Header 關鍵字 Header

SRE 實務spec:inputs(CI/CD Components 專用)

只在寫 CI/CD Component(第 6 章)時用。定義該元件對外暴露的輸入參數:

spec:
  inputs:
    python_version:
      default: "3.11"
      description: "Python 容器 image tag"
      type: string
    deploy_env:
      options:
        - staging
        - production
      description: "部署目標"
    test_count:
      default: 3
      type: number
---
# 以下是 component 的實際內容
test-job:
  image: python:$[[ inputs.python_version ]]
  script:
    - echo "Deploy to $[[ inputs.deploy_env ]]"

重點:header 與內容之間用 ---(三個 dash)分隔,這是 YAML 的 document 分隔語法。

🌐 2. Global 關鍵字 Global

Quick Startstages — 定義 stage 順序

stages:
  - build
  - test
  - deploy

若省略,預設是 .pre → build → test → deploy → .post。

💡 .pre 永遠在最前、.post 永遠在最後。即使沒在 stages 列出也生效。

SRE 實務workflow — 控制整條 pipeline 是否執行

workflow:
  name: "Build for $CI_COMMIT_REF_NAME"
  rules:
    # 草稿 MR 不跑
    - if: $CI_MERGE_REQUEST_TITLE =~ /^Draft:/
      when: never
    # MR 觸發時跑
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
    # push 到 default branch 跑
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
    # tag 觸發跑
    - if: $CI_COMMIT_TAG
  auto_cancel:
    on_new_commit: interruptible   # 新 commit 時取消可中斷的 job
    on_job_failure: all            # 任一 job 失敗時取消所有 job
⚠️ 沒設 workflow 時,push branch + 開 MR 會產生重複 pipeline。這是最常見的新手問題。

Quick Startdefault — 所有 job 的預設值

default:
  image: node:20
  timeout: 1h
  retry: 2
  interruptible: true
  tags:
    - docker
  before_script:
    - echo "Starting job on $CI_COMMIT_REF_NAME"
  after_script:
    - echo "Job finished"

所有 job 沒自訂時都會套用這些值。個別 job 可自行覆蓋。

Quick Startvariables — pipeline 層級變數

variables:
  DEPLOY_ENV: staging
  LOG_LEVEL: info
  # 進階:設 description 讓手動觸發時顯示 UI 輸入框
  DEPLOY_TARGET:
    value: "prod-us"
    description: "要部署的區域"
    options:
      - prod-us
      - prod-eu
      - prod-asia

SRE 實務include — 引入其他設定

include:
  # 本專案內的檔案
  - local: '/ci/shared.yml'

  # 其他專案(需有權限)
  - project: 'my-group/my-project'
    ref: main
    file: 'templates/.gitlab-ci-template.yml'

  # 遠端 URL
  - remote: 'https://gitlab.com/awesome/raw/main/.template.yml'

  # GitLab 內建 template
  - template: Auto-DevOps.gitlab-ci.yml

  # CI/CD Component(第 6 章)
  - component: gitlab.com/my-group/my-component@1.0
    inputs:
      stage: test

  # 條件引入
  - local: builds.yml
    rules:
      - if: $CI_COMMIT_BRANCH == "main"

合併規則:include 依序讀取、深層合併(deep merge);主檔有同 key 會覆蓋 include 的值。

架構視角其他 Global 關鍵字

關鍵字用途
image全域預設 image(已被 default:image 取代)
services全域附加服務(同上)
cache全域快取(建議放 default 內)
before_script / after_script全域前後執行腳本
⚠️ 新版官方推薦把以上這些放進 default: 區塊,而非全域。

🎯 3. Job 關鍵字 Job

30+ 個 job 關鍵字。以下按用途分組,標示⭐的是最常用。

🧾 執行內容

關鍵字說明
script ⭐job 要執行的指令(必填)
before_scriptscript 前執行
after_scriptscript 後執行(即使 script 失敗也會跑)
stage ⭐指定屬於哪個 stage
image ⭐執行環境 Docker image
services附加服務 image(如 MySQL、Redis)
tags ⭐指定 runner tag

⏰ 執行時機

關鍵字說明
rules ⭐決定 job 是否加入 pipeline(見第 4 節)
only / except舊語法,逐漸被 rules 取代
whenon_success(預設) / always / on_failure / manual / delayed / never
needs跨 stage 依賴(見第 5 節)
allow_failure失敗不擋 pipeline
manual_confirmationmanual 時要求二次確認

📦 資料與產物

關鍵字說明
artifacts ⭐保留產物(路徑、報告、到期)
cache ⭐快取目錄(加速用)
dependencies限定要哪些上游 artifact(空陣列 [] = 不要)
release建立 GitLab Release
pagesjob 產物要上傳到 GitLab Pages(0 或 1 個 job 可用)

🔐 安全與變數

關鍵字說明
variablesjob 層級變數
secrets從 Vault / HashiCorp 等拉取機密
id_tokens生成 OIDC JWT token(與雲端整合)
environment ⭐標註部署目標環境

⚙️ 流程控制

關鍵字說明
retry失敗自動重試次數(0-2)
timeout自訂 job 逾時時間
interruptible可被新 commit 取消
resource_group限制同一 resource group 一次只能跑一個 job
parallel平行跑多個 instance(含 matrix)
trigger觸發下游 pipeline(child / multi-project)

📊 報告

關鍵字說明
coverage測試覆蓋率正則擷取
artifacts:reports:*測試報告類型(見第 11 章)

🔄 重用

關鍵字說明
extends ⭐繼承其他 job / hidden job(見第 6 節)
inherit控制是否繼承 default / variables
hooks生命週期 hook(如 pre_get_sources_script)

SRE 實務完整 Job 範例(用到大部分關鍵字)

deploy-production:
  stage: deploy
  image: alpine:3.18
  tags: [docker, production]

  before_script:
    - apk add --no-cache curl

  script:
    - ./deploy.sh

  after_script:
    - ./notify.sh

  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      when: manual

  needs:
    - job: build
      artifacts: true
    - job: test

  variables:
    DEPLOY_TARGET: prod

  environment:
    name: production
    url: https://app.example.com
    on_stop: stop-production

  artifacts:
    paths: [deploy.log]
    expire_in: 1 month
    when: always

  cache:
    key: "$CI_COMMIT_REF_SLUG"
    paths: [.deploy-cache/]

  timeout: 30 minutes
  retry:
    max: 2
    when:
      - runner_system_failure
      - stuck_or_timeout_failure
  interruptible: false
  resource_group: production-deploy
  allow_failure: false

📏 4. rules 深度解析

rules 是最重要也最常被誤用的關鍵字。規則由上而下評估,第一個符合的決定命運。

Quick Start三種規則關鍵字

關鍵字說明
if條件表達式(最常用)
changes指定檔案有變更才加入
exists指定檔案存在才加入
when如何執行:on_success / manual / always / never / delayed
allow_failure失敗是否擋 pipeline
variables匹配此 rule 時套用的變數

Quick Start最常見 if 條件

rules:
  # 只在 default branch 跑
  - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH

  # 只在 MR 跑
  - if: $CI_PIPELINE_SOURCE == "merge_request_event"

  # 只在 tag 跑
  - if: $CI_COMMIT_TAG

  # 排除 draft MR
  - if: $CI_MERGE_REQUEST_TITLE =~ /^Draft:/
    when: never

  # 排程(schedule)才跑
  - if: $CI_PIPELINE_SOURCE == "schedule"

  # 正則匹配分支
  - if: $CI_COMMIT_BRANCH =~ /^(feature|hotfix)\//

  # 多條件
  - if: $CI_COMMIT_BRANCH == "main" && $DEPLOY_ENV == "prod"

  # 否定條件
  - if: $CI_COMMIT_BRANCH != "main"

SRE 實務changes + if 組合

docker-build:
  script: docker build -t myapp .
  rules:
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
      changes:
        - Dockerfile
        - docker/**/*
        - src/**/*.go
      when: on_success
    # 其他情況都不執行(沒有後續 rule)

意義:只在 main branch + 指定檔案有變更時才跑。

SRE 實務exists(檔案存在)

lint-yaml:
  script: yamllint .
  rules:
    - exists:
        - "**/*.yml"
        - "**/*.yaml"

SRE 實務用 rules 切換變數

deploy:
  script: ./deploy.sh $TARGET
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      variables:
        TARGET: production
    - if: $CI_COMMIT_BRANCH == "develop"
      variables:
        TARGET: staging
    - when: never   # 其他分支不跑

架構視角rules 常見陷阱

⚠️ 重複 pipeline:同時有 branch 和 MR rule 時,若沒在 workflow 處理會產生兩條 pipeline。
# ❌ 會產生重複 pipeline
rules:
  - if: $CI_PIPELINE_SOURCE == "push"
  - if: $CI_PIPELINE_SOURCE == "merge_request_event"

# ✅ 在 workflow 層級處理(推薦)
workflow:
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
    - if: $CI_COMMIT_BRANCH && $CI_OPEN_MERGE_REQUESTS
      when: never     # 有開 MR 的 branch 不跑 branch pipeline
    - if: $CI_COMMIT_BRANCH
⚠️ 不要混用 only/except 與 rules。同一 pipeline 內的不同 job 用了不同語法,會造成難以排查的行為差異。

架構視角共用 rules 模式

.default-rules:
  rules:
    - if: $CI_PIPELINE_SOURCE == "schedule"
      when: never
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH

job1:
  rules:
    - !reference [.default-rules, rules]
  script: echo "使用共用規則"

job2:
  rules:
    - !reference [.default-rules, rules]
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
  script: echo "共用規則 + 額外 MR"

🔗 5. needs 與 DAG

Quick Start為什麼要用 needs?

預設 pipeline 必須整個 stage 跑完才能進下一個。用 needs 可建立 DAG(有向無環圖),job 一旦依賴項完成就開始跑,不用等整個 stage。

Quick Start基本用法

stages: [build, test, deploy]

build-frontend:
  stage: build
  script: npm run build:frontend

build-backend:
  stage: build
  script: make build-backend

test-frontend:
  stage: test
  needs: [build-frontend]   # 只等 frontend build,不等 backend
  script: npm test

test-backend:
  stage: test
  needs: [build-backend]
  script: make test

deploy:
  stage: deploy
  needs:
    - test-frontend
    - test-backend
  script: ./deploy.sh

SRE 實務進階 needs

# 指定要不要上游 artifact
test-job:
  stage: test
  needs:
    - job: build-job
      artifacts: true      # 下載 artifact(預設)
    - job: lint-job
      artifacts: false     # 不下載

# 跨 pipeline needs(需要 trigger 設定)
downstream-job:
  needs:
    - project: group/upstream-project
      job: build
      ref: main
      artifacts: true

# 可選 needs(parent 沒跑也 OK)
optional-job:
  needs:
    - job: optional-dep
      optional: true

架構視角DAG vs 傳統 pipeline

傳統(純 stage)DAG(needs)
順序stage 順序依賴關係
速度慢(等整個 stage)快(提早開始)
可讀性高中
適用簡單專案monorepo、微服務
💡 Pipeline editor 的「Job dependencies」視圖能畫出 DAG 圖,方便檢查。

♻️ 6. 重用:extends / include / !reference

Quick Start三種重用機制比較

機制適用特點
extends多個 job 共用設定多層繼承,建議 ≤ 3 層
include跨檔案 / 跨專案重用深層合併
!reference重用特定欄位精確,不用整個 job
Anchor (&)YAML 原生可讀性差,不建議
Component跨團隊共享有版本控制,見第 6 章

Quick Startextends 範例

# hidden job(以 . 開頭)作為模板
.test-template:
  stage: test
  image: node:20
  before_script:
    - npm ci
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"

# 繼承 + 覆蓋
unit-test:
  extends: .test-template
  script: npm run test:unit

e2e-test:
  extends: .test-template
  script: npm run test:e2e
  timeout: 30 minutes  # 覆蓋

# 多重繼承
performance-test:
  extends:
    - .test-template
    - .on-schedule-only
  script: npm run test:perf

SRE 實務include 完整範例

# .gitlab-ci.yml
include:
  # 本地
  - local: 'ci/shared/rules.yml'
  - local: 'ci/**/*.yml'  # glob 模式

  # 同 group 另一專案的檔案
  - project: 'devops/ci-templates'
    ref: v1.2.0
    file:
      - '/templates/build.yml'
      - '/templates/deploy.yml'

  # 遠端
  - remote: 'https://example.com/templates/security.yml'

  # 官方 template
  - template: Security/SAST.gitlab-ci.yml
  - template: Security/Secret-Detection.gitlab-ci.yml

  # 條件 include
  - local: 'ci/staging.yml'
    rules:
      - if: $DEPLOY_ENV == "staging"

stages: [build, test, deploy]

SRE 實務!reference(精準取用特定欄位)

.setup:
  script:
    - apt-get update
    - apt-get install -y curl jq

.cleanup:
  script:
    - rm -rf /tmp/workspace

build:
  script:
    - !reference [.setup, script]    # 插入 setup 的 script
    - make build
    - !reference [.cleanup, script]  # 插入 cleanup

優勢:不用整個 extend,只取需要的欄位。適合「script 內插入共用步驟」的場景。

架構視角重用策略決策樹

要重用的東西?
├─ 整個 job → extends(配 hidden job)
├─ 多個 job 的一組設定檔 → include local
├─ 跨專案模板 → include project
├─ 業界共享的元件 → CI/CD Component
└─ 某 job 內要插入某段腳本 → !reference

🔤 7. 變數進階

(第 7 章會深入,這裡重點講 YAML 語法)

Quick Start三個層級

# 1. Pipeline 層級(全域)
variables:
  APP_ENV: staging

# 2. Workflow rules 內
workflow:
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      variables:
        APP_ENV: production      # 主幹時改為 production

# 3. Job 層級(覆蓋全域)
deploy:
  variables:
    APP_ENV: production
  script: ./deploy.sh $APP_ENV

SRE 實務UI 手動觸發時的輸入框

variables:
  DEPLOY_REGION:
    value: "us-east-1"
    description: "部署區域"
    options:
      - "us-east-1"
      - "eu-west-1"
      - "ap-northeast-1"
    expand: false           # 不展開其他變數

  DEBUG:
    value: "false"
    description: "啟用 debug log?"
    options: ["true", "false"]

在 UI 手動觸發 pipeline 時,這些會顯示為下拉/輸入框,方便使用者選擇。

架構視角變數展開注意事項

  • $VAR 可以巢狀展開其他變數
  • 若要字面 $ 而不展開,用 $$
  • 在 rules:if 的評估時間點,script 產生的變數不可用(因為還沒跑)
  • dotenv artifact 可以跨 job 傳遞變數(下一章詳述)

📦 8. artifacts / cache

Quick Startartifacts 完整語法

build:
  script:
    - npm run build
  artifacts:
    name: "$CI_JOB_NAME-$CI_COMMIT_REF_SLUG"   # 下載檔名
    paths:
      - dist/
      - build/
    exclude:
      - dist/**/*.map
    expire_in: 1 week          # 1 hour, 30 days, never ...
    when: on_success           # on_success(預設) / always / on_failure
    expose_as: "Built website" # 在 MR 顯示的名稱
    public: false              # 是否公開可下載
    # 特殊類型 artifact
    reports:
      junit: reports/junit.xml
      coverage_report:
        coverage_format: cobertura
        path: coverage.xml
      dotenv: build.env        # 變數傳給下游 job
      sast: gl-sast-report.json

SRE 實務dotenv — 變數跨 job 傳遞

build:
  stage: build
  script:
    - echo "APP_VERSION=1.2.3" > build.env
    - echo "BUILD_ID=$CI_JOB_ID" >> build.env
  artifacts:
    reports:
      dotenv: build.env

deploy:
  stage: deploy
  needs: [build]
  script:
    - echo "Deploying version $APP_VERSION (build $BUILD_ID)"

dotenv 是非常實用的功能,可以把 build job 產生的版本號等資訊帶給下游。

Quick Startcache 完整語法

test:
  script:
    - npm ci
    - npm test
  cache:
    key: "$CI_COMMIT_REF_SLUG"  # 每個分支獨立 cache
    # key 可以依檔案內容:
    # key:
    #   files: [package-lock.json]
    #   prefix: "$CI_JOB_NAME"
    paths:
      - node_modules/
      - .npm/
    policy: pull-push   # pull-push(預設) / pull / push
    when: on_success    # 何時存 cache
    untracked: false    # 是否包含 untracked files
    unprotect: false    # protected branch 的 cache 是否給其他用

SRE 實務典型 cache 策略

場景keypathspolicy
Node.js依 package-lock.jsonnode_modules/pull-push
Java/Maven依 pom.xml.m2/repository/pull-push
Python依 requirements.txt.venv/ 或 .pip-cache/pull-push
Go依 go.sum$GOPATH/pkg/mod/pull-push
只讀(test)同上同上pull

架構視角cache vs artifact 使用時機

是 job 的輸出產物、需要下載或給下游 job 用?
└─▶ artifact

只是為了加速下次執行?
└─▶ cache

需要跨 pipeline / 跨 branch 共享?
└─▶ 用 S3/GCS cache + 精心設計的 cache key

✅ 設計 Checklist

  • 有 workflow 避免重複 pipeline
  • 所有 job 都有明確的 rules 或 workflow 控管
  • 高頻 dependency 有 cache
  • 跨 job 資料走 artifact
  • secret 不在 YAML 寫死
  • 重複設定用 extends 或 include 抽出
  • default 設定 image / timeout
  • deploy job 有 environment 與 resource_group
  • 重要 job 可 interruptible: false
  • rules 沒混用 only/except

🆘 常見 YAML 錯誤

錯誤解法
縮排錯用 2 空格,不混 tab。Pipeline editor 會自動檢查
script 沒填每個 job 必須有 script,或改用 trigger/extends
duplicated keyYAML 不允許同層級重複 key
rules 寫反第一個 match 決定,從特殊到通用排序
include 找不到本地路徑相對於 repo 根、project include 需權限
變數不展開單引號 vs 雙引號;某些 scope 不支援展開
