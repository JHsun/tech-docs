🗺️ Maintain 總覽

Maintain 涵蓋 GitLab 從日常保養到災難復原的全部運維工作。本章的核心原則:

📦 備份不是做出來就結束

必須定期演練還原,沒驗證過的備份等於沒備份。

🔑 Secrets 一定要另外備份

GitLab 備份不含 gitlab-secrets.json。沒了它,備份無法還原。

🧹 Git 倉庫會越用越慢

定期 Housekeeping(git gc、repack)維持效能。

⚠️ 還原版本必須一致

只能還原到相同版本、相同版次(CE/EE)的 GitLab。

💾 1. 備份(Backup)

Quick Start最基本的備份指令

# 建立備份(歸檔在 /var/opt/gitlab/backups)
sudo gitlab-backup create

# 執行過程會產生類似檔名
# 1714053314_2024_04_25_17.0.0-pre_gitlab_backup.tar
#    ↑ timestamp     ↑ date       ↑ GitLab 版本

一個 backup 檔案會包含:資料庫 dump、Git repositories、Uploads、Artifacts、LFS、Packages、Container Registry、CI Secure Files、Terraform state、Pages、External merge request diffs 等。

🔴 備份不包含的東西 ❌ /etc/gitlab/gitlab.rb 組態檔
❌ /etc/gitlab/gitlab-secrets.json 密鑰檔
❌ /etc/gitlab/ssl/ TLS 憑證
❌ /etc/gitlab/trusted-certs/ 信任憑證
❌ SSH host keys
❌ Redis 資料(僅快取,但包含執行中的 Sidekiq jobs)

SRE 實務備份核心指令與參數

# 基本備份
sudo gitlab-backup create

# 壓縮方法(預設 tar,可改 tar.gz、lzip)
sudo gitlab-backup create STRATEGY=copy       # 先複製再打包(較安全)
sudo gitlab-backup create GZIP_RSYNCABLE=yes  # 友善 rsync 的 gzip

# 排除特定元件(加速 / 省空間)
sudo gitlab-backup create SKIP=artifacts,lfs,registry

# 指定檔名(不含預設 timestamp prefix)
sudo gitlab-backup create BACKUP=customname

# 並行(加速)
sudo gitlab-backup create GITLAB_BACKUP_MAX_CONCURRENCY=4 \
                          GITLAB_BACKUP_MAX_STORAGE_CONCURRENCY=2

# Cron 模式(錯誤才輸出)
sudo gitlab-backup create CRON=1

SKIP 可排除的項目:

項目說明
dbPostgreSQL dump
repositoriesGit repos(通常最大宗)
uploads圖片、附件
buildsCI job traces(log)
artifactsCI artifacts
lfsGit LFS objects
registryContainer Registry
pagesGitLab Pages 資料
packagesPackage Registry
terraform_stateTerraform state

SRE 實務Cron 排程與遠端存放

/etc/gitlab/gitlab.rb 設定:

# 保留 7 天
gitlab_rails['backup_keep_time'] = 604800

# 壓縮
gitlab_rails['backup_archive_permissions'] = 0644
gitlab_rails['backup_pg_schema'] = 'public'

# 備份路徑
gitlab_rails['backup_path'] = "/var/opt/gitlab/backups"

# 上傳到 S3
gitlab_rails['backup_upload_connection'] = {
  'provider' => 'AWS',
  'region' => 'ap-northeast-1',
  'aws_access_key_id' => 'AKIA...',
  'aws_secret_access_key' => '...'
}
gitlab_rails['backup_upload_remote_directory'] = 'company-gitlab-backups'
gitlab_rails['backup_multipart_chunk_size'] = 104857600  # 100 MB chunks

# 備份加密(SSE-S3 或 SSE-KMS)
gitlab_rails['backup_encryption'] = 'AES256'

Cron 範例:

# /etc/cron.d/gitlab-backup
# 每天凌晨 3 點做全備份
0 3 * * * root /opt/gitlab/bin/gitlab-backup create CRON=1

# 也順便備份 secrets / config(用 tar + 上傳 S3)
30 3 * * * root tar -czf /tmp/gitlab-conf-$(date +\%F).tar.gz \
  /etc/gitlab && aws s3 cp /tmp/gitlab-conf-*.tar.gz \
  s3://company-gitlab-backups/config/

架構視角大型實例的備份挑戰

當 Git repository 資料超過 ~200GB,傳統 Rake backup 會變得很慢,特別是備份到異地。

替代策略:

方案適用注意
File System Snapshot
(EBS / LVM / ZFS)		單機大量資料需要停 Gitaly 或用 Maintenance Mode
Incremental Repository Backup只變動少量 repoGitLab 16+ 支援,需 Gitaly 端設定
Concurrent Repository Backup多 repo透過 GITLAB_BACKUP_MAX_CONCURRENCY
PostgreSQL 獨立備份
(pg_dump / WAL-E / pgBackRest)		DB 在獨立機SKIP=db 跳過 Rake 的 DB
Object Storage 端複寫資料已放 S3S3 CRR、GCS Multi-Region
gitlab-backup-cli(新工具)未來主力目前僅支援 Google Cloud,尚在開發
💡 Gitaly Cluster(Praefect)不支援 File System Snapshot 備份。必須用 Rake backup 或逐 repo 的 bundle 備份。

架構視角備份一致性(防寫入)

Git repository 若在備份期間仍被寫入,會造成毀損。防寫入兩種方式:

  1. Maintenance Mode(推薦):全站進入只讀,使用者可讀取但不能推送
  2. 停掉 Gitaly:sudo gitlab-ctl stop gitaly → 複製 → start gitaly(有 downtime)

♻️ 2. 還原(Restore)

⚠️ 還原是破壞性操作 執行 restore 會抹除現有資料。只能還原到相同版本 + 相同版次(CE/EE)的 GitLab。版本不一致就要先升降級,再 restore。

Quick Start完整還原 5 步驟

# 1. 確認 GitLab 版本與備份來源版本一致
sudo gitlab-rake gitlab:env:info | grep "GitLab information" -A3

# 2. 把備份檔放到指定目錄(預設 /var/opt/gitlab/backups)
sudo cp 1714053314_2024_04_25_17.0.0-pre_gitlab_backup.tar \
  /var/opt/gitlab/backups/
sudo chown git:git /var/opt/gitlab/backups/1714053314_*

# 3. 停掉依賴資料庫的服務
sudo gitlab-ctl stop puma
sudo gitlab-ctl stop sidekiq

# 4. 執行還原(BACKUP= 是去除 _gitlab_backup.tar 後的名稱)
sudo gitlab-backup restore BACKUP=1714053314_2024_04_25_17.0.0-pre

# 5. 重啟並檢查
sudo gitlab-ctl restart
sudo gitlab-rake gitlab:check SANITIZE=true
sudo gitlab-rake gitlab:doctor:secrets

SRE 實務遷移到新伺服器的還原

把 GitLab 從 A 機搬到 B 機的標準流程:

  1. 在 B 機安裝完全相同版本的 GitLab
  2. 在 B 機跑過至少一次 gitlab-ctl reconfigure
  3. 從 A 機複製:
    • 最新備份 tar
    • /etc/gitlab/gitlab-secrets.json(最重要)
    • /etc/gitlab/gitlab.rb(依需求)
    • /etc/gitlab/ssl/(若用自簽憑證)
  4. 在 B 機放置檔案後 gitlab-ctl reconfigure
  5. 執行 restore
  6. 重啟後測試登入、push、pipeline
⚠️ 換了 FQDN?WebAuthn 裝置會全部失效(YubiKey 等)。使用者需重新註冊。2FA TOTP(app)則不受影響。

架構視角Restore 內部流程

理解 restore 做了什麼,才能排錯:

  1. 建立臨時工作目錄
  2. 解壓備份 tar 到工作目錄
  3. 讀 backup_information.yml 驗證版本
  4. Drop DB 的所有資料表 → 還原 SQL dump
  5. 把每個 repo bundle 透過 Gitaly 還原到正確 shard
  6. 解壓 artifacts / lfs / uploads / packages / 等 tarball 到對應儲存
  7. 重建 authorized_keys
  8. 清除 Rails cache

任一步失敗 → 整個 restore 中斷。所以還原前務必確保磁碟空間(至少 2× 備份大小)與記憶體足夠。

架構視角演練(DR Drill)

✅ 建議每季做一次還原演練:
  1. 準備一台與 prod 同版本的演練機
  2. 從 S3 拉最新備份
  3. 執行完整 restore
  4. 測試:登入 / clone / push / pipeline / MR / issue / 搜尋
  5. 記錄 RTO(花多少時間)
  6. 記錄發現的問題,回饋到 DR 文件

🔑 3. Secrets 與 Config 備份

🔴 最常被忽略、最會出事的一環 GitLab 的 gitlab-backup create 不會備份密鑰檔。沒了它,2FA、CI 變數、OAuth token 等所有加密內容都無法解密。

Quick Start必須另外備份的三個檔案

檔案內容
/etc/gitlab/gitlab-secrets.jsonDB 加密金鑰、CI variable 金鑰、OAuth secret
/etc/gitlab/gitlab.rb全部自訂組態
/etc/gitlab/ssl/TLS 憑證與私鑰
# 每日排程
tar -czf /backup/gitlab-config-$(date +%F).tar.gz \
  /etc/gitlab/gitlab-secrets.json \
  /etc/gitlab/gitlab.rb \
  /etc/gitlab/ssl/ \
  /etc/gitlab/trusted-certs/

# 加密(gpg)後上傳 S3
gpg --encrypt --recipient backup@company.com \
  /backup/gitlab-config-*.tar.gz
aws s3 cp /backup/gitlab-config-*.tar.gz.gpg \
  s3://company-gitlab-backups/config/

SRE 實務若 secrets 已遺失怎麼辦?

官方文件「When the secrets file is lost」建議:

  • 所有加密欄位(CI variable、integration tokens、2FA 等)將不可用
  • 需要:清除所有 CI/CD variable、清除所有 integration config、請所有使用者重設 2FA、重建 OAuth app
  • 實務上,若 secrets 遺失 + 備份有價值,選擇從備份 restore 並接受部分加密資料無效,然後清除重建
⚠️ 重點記住:secrets 檔案外流等於全站加密資料被破解。存放必須加密(Vault / KMS / sops)。

架構視角密鑰治理

  • 分離儲存:secrets 不與 backup 放同一 bucket/位置
  • 加密:靜態加密(gpg / KMS)+ 傳輸加密(TLS)
  • 存取控制:只有 DR 團隊能拿到,IAM Role 最小化
  • 輪替:定期輪替 GitLab 的加密金鑰(參考官方 gitlab-rake gitlab:doctor:reset_encryption_key)
  • Vault 整合:機敏的 gitlab.rb 值(SMTP 密碼、S3 key)可改用 Vault 取代明文

🧹 4. Housekeeping(儲存維護)

Quick StartHousekeeping 是什麼?

Git repository 用久了會累積 loose objects、碎片化的 packfile,導致效能下降。Housekeeping 就是跑 git gc、git repack、git prune 等維護指令。

GitLab 預設會自動跑,但大型 instance 需要手動介入。

SRE 實務Manual Housekeeping

UI 操作:Project → Settings → General → Advanced → Housekeeping

API:

curl --request POST \
  --header "PRIVATE-TOKEN: $TOKEN" \
  "https://gitlab.example.com/api/v4/projects/$PROJECT_ID/housekeeping"

Rake:

# 對所有 repo 執行
sudo gitlab-rake gitlab:git:repack

# 或透過 Sidekiq 全站跑
sudo gitlab-rake gitlab:housekeeping:all

SRE 實務自動 Housekeeping 參數調整

在 Admin Area → Settings → Repository 可調:

  • Enable automatic repository housekeeping(建議開啟)
  • Incremental repack period(預設 10)
  • Full repack period(預設 50)
  • Git GC period(預設 200)

數字代表:每 N 次 push 觸發一次對應動作。

架構視角大型 repo 的最佳化

  • commit-graph:GitLab 會自動維護,加速 log / blame
  • bitmap index:加速 clone / fetch
  • Git LFS:大檔案從 Git 搬出,減少 repack 負擔
  • Shallow Clone:CI 端用 GIT_DEPTH 減輕拉取負擔
  • Partial Clone:按需拉取 object(--filter=blob:none)
  • Geo:多地區緩存 clone 流量

🔧 5. 常用 Rake Tasks

GitLab 提供大量 Rake 任務,用於診斷、修復、遷移。最常用的分類如下。

Quick Start日常最常用 10 個

sudo gitlab-rake gitlab:check SANITIZE=true

# 環境資訊(開 support ticket 必備)
sudo gitlab-rake gitlab:env:info

# LDAP 連線測試

# 重設 root 密碼

sudo gitlab-rake cache:clear

sudo gitlab-rake gitlab:shell:setup

# Artifact 遷移至 Object Storage

# LFS 遷移
sudo gitlab-rake gitlab:lfs:migrate

# Uploads 遷移
sudo gitlab-rake gitlab:uploads:migrate:all

# 診斷 secrets 是否與 DB 一致
sudo gitlab-rake gitlab:doctor:secrets

架構視角跑 Rake 的注意事項

  • 不要直接用 rake,要用 gitlab-rake(會自動載入正確環境)
  • 部分任務需要停服務(如 restore)
  • 大量資料遷移要用 tmux/screen(避免 SSH 斷線中斷)
  • 遷移類任務可重跑(idempotent,卡住再跑一次通常沒事)
  • Helm 部署用 kubectl exec 進 toolbox pod 執行

🚧 6. Maintenance Mode(只讀模式)

Quick Start什麼是 Maintenance Mode?

Premium+ 功能。開啟後,整個 GitLab 進入只讀:

  • ✅ 使用者可以瀏覽、clone、下載
  • ❌ 不能 push、建立 issue、改設定、跑 pipeline
  • ✅ 管理員仍可登入與操作

適用場景:備份 / 遷移 / 升級 / Geo failover。

SRE 實務開啟與關閉

# UI:Admin → Settings → General → Maintenance mode → 勾選 Enable

# Rails console
::Gitlab::CurrentSettings.update!(maintenance_mode: true)
::Gitlab::CurrentSettings.update!(maintenance_mode_message: '系統維護中,預計 30 分鐘')

# API
curl --request PUT \
  --header "PRIVATE-TOKEN: $TOKEN" \
  --data "maintenance_mode=true" \
  --data "maintenance_mode_message=維護中" \
  "https://gitlab.example.com/api/v4/application/settings"

架構視角適用情境選擇

情境建議
短時間維護(< 15 分)Broadcast Message 預告 + 直接做
備份(需一致性)Maintenance Mode 15-30 分
升級(1-2 小時)Maintenance Mode + 預告
大型遷移(半天以上)Maintenance Mode + 斷線公告 + 外部通訊
災難復原直接停服務

🚚 7. 搬遷(Move / Migrate)

Quick Start三種搬遷場景

場景推薦作法
整台 GitLab 換機備份 + 還原
搬特定 Project / GroupDirect Transfer(GitLab → GitLab)
換 Gitaly 儲存 shardRepository Move API

SRE 實務Move Repositories(在 storage 間搬移)

# 列出所有 storage
sudo gitlab-rails runner 'puts Gitlab.config.repositories.storages.keys'

# 把所有 project 從 default 搬到 new_storage
curl --request POST \
  --header "PRIVATE-TOKEN: $TOKEN" \
  --data "destination_storage_name=new_storage" \
  "https://gitlab.example.com/api/v4/projects/$PROJECT_ID/repository_storage_moves"

# 批次搬移
sudo gitlab-rake gitlab:storage:migrate_from_legacy_storage

SRE 實務Direct Transfer(GitLab 對 GitLab)

  1. 來源 GitLab:Admin → Settings → General → Import and export settings → Enable migrating GitLab groups and projects by direct transfer
  2. 目標 GitLab:同樣啟用
  3. 目標 GitLab → New group → Import group → 輸入來源 URL + PAT

架構視角遷移 vs 備份還原

  • 備份還原:適合「整台換機」,保留所有內容(包含 CI 變數、secrets)
  • Direct Transfer:適合「部分資料遷入/遷出」,速度較快但不含 secrets / CI variables
  • Project Export:⚠️ 已明確不建議當備份用

⬆️ 8. 版本升級

Quick Start升級基本流程

# 1. 完整備份(含 secrets)
sudo gitlab-backup create
sudo tar -czf /backup/gitlab-etc-$(date +%F).tar.gz /etc/gitlab

# 2. 升級套件
sudo apt-get update
sudo apt-get install gitlab-ee=X.Y.Z-ee.0

# 3. reconfigure
sudo gitlab-ctl reconfigure

# 4. 驗證
sudo gitlab-rake gitlab:check SANITIZE=true
sudo gitlab-ctl status

SRE 實務升級路徑規則

⚠️ 不能跨大版本直接跳升!例如:13.x → 16.x 不行,必須經過 stopover versions。

官方提供「Upgrade Path Tool」:

https://gitlab-com.gitlab.io/support/toolbox/upgrade-path/

範例路徑:14.0.0 → 17.0.0

14.0.12 → 14.3.6 → 14.9.5 → 14.10.5
→ 15.0.5 → 15.4.6 → 15.11.13
→ 16.0.9 → 16.3.7 → 16.7.x → 16.11.x
→ 17.0.x

每個中間版本通常需要:升 → 等 background migration 跑完 → 再升下一版。

架構視角Zero-downtime Upgrade

HA 架構下可以做零停機升級,大致流程:

  1. 先升級 deploy node(Rails 機台)
  2. 從 Load Balancer 下線一半,升級,上線
  3. 對另一半重複
  4. Sidekiq node 逐一升級
  5. Gitaly node 逐一升級
  6. PostgreSQL / Redis 通常最後,或採藍綠切換
💡 大型升級前務必:閱讀該版本的 Release Notes、Upgrade Notes、Background Migration 狀態、Deprecated 項目清單。GitLab 把重要警告都放在這幾個地方。

✅ 備份策略 Checklist

  • Cron 每日備份(凌晨低峰)
  • 備份包含 gitlab-secrets.json(另外 tar)
  • 備份保留天數設定(backup_keep_time)
