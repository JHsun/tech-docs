🗺️ 變數體系總覽

GitLab CI 變數有兩個切分維度:

📝 依來源

  • Predefined(GitLab 自動提供)
  • UI 設定(Project / Group / Instance)
  • YAML 定義(pipeline / job 層級)
  • Trigger 傳入(API、schedule、manual)
  • Dotenv artifact(跨 job)
  • External Secrets(Vault 等)

🏷️ 依型別

  • env_var(一般環境變數,預設)
  • file(值寫成檔案,變數是檔名)

Quick Start最常見的三種用法

# 1. 用 GitLab 預定義變數
deploy:
  script:
    - echo "Deploy $CI_COMMIT_SHORT_SHA to $CI_ENVIRONMENT_NAME"

# 2. YAML 裡自訂
variables:
  APP_ENV: staging
test:
  script:
    - echo "Running in $APP_ENV"

# 3. UI 設定(敏感資料)
deploy:
  script:
    - ./deploy.sh --token $AWS_TOKEN   # AWS_TOKEN 從 UI Variables 來

📝 2. 在 .gitlab-ci.yml 定義

Quick Start三個層級

# 1. Pipeline 層級(全域)
variables:
  APP_VERSION: "1.0.0"
  LOG_LEVEL: info

# 2. Job 層級(覆蓋全域)
deploy:
  variables:
    LOG_LEVEL: debug         # 此 job 用 debug
    DEPLOY_TARGET: prod       # 新增 job 專屬
  script:
    - echo "Version=$APP_VERSION, Log=$LOG_LEVEL, Target=$DEPLOY_TARGET"

# 3. workflow:rules 內(依條件切換)
workflow:
  rules:
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
      variables:
        APP_ENV: production
    - when: always
      variables:
        APP_ENV: staging

SRE 實務手動觸發時的表單變數

variables:
  DEPLOY_REGION:
    value: "us-east-1"                # 預設值
    description: "部署的 AWS 區域"
    options:                           # 下拉選項
      - "us-east-1"
      - "eu-west-1"
      - "ap-northeast-1"
    expand: false                      # 不展開(保留字面)

  DEBUG:
    value: "false"
    description: "啟用 debug log?"
    options: ["true", "false"]

  VERSION:
    value: ""
    description: "要部署的版本號(留空用 latest)"

這樣在 UI「Run pipeline」時會看到下拉 / 輸入框。

🎛️ 3. 在 UI 定義(UI Variables)

Quick Start三個層級

層級路徑適用
ProjectProject → Settings → CI/CD → Variables單一專案敏感資料
GroupGroup → Settings → CI/CD → Variables跨多專案共用
InstanceAdmin → Settings → CI/CD → Variables全站共用(管理員)
✅ 推薦:敏感資料一律放 UI,不要寫進 YAML。YAML 進 Git,會被所有有 repo 權限的人看到。

SRE 實務變數欄位全覽

欄位說明
Key變數名,只能字母數字 _,不能有空格
Value值,最多 10,000 字元
TypeVariable(預設)或 File
Environment scope*(全部)、特定環境、萬用字元(Premium)
Protect variable只能在 protected branch / tag 用
VisibilityVisible / Masked(預設) / Masked and hidden
Expand variable reference是否展開內嵌的其他變數
Description說明(幫助團隊成員理解)

SRE 實務API 管理變數

# 列出專案變數
curl --header "PRIVATE-TOKEN: $TOKEN" \
  "https://gitlab.example.com/api/v4/projects/42/variables"

# 新增變數
curl --request POST \
  --header "PRIVATE-TOKEN: $TOKEN" \
  --url "https://gitlab.example.com/api/v4/projects/42/variables" \
  --form "key=AWS_SECRET" \
  --form "value=xxxx" \
  --form "masked=true" \
  --form "protected=true" \
  --form "environment_scope=production"

# 更新
curl --request PUT \
  --header "PRIVATE-TOKEN: $TOKEN" \
  --url "https://gitlab.example.com/api/v4/projects/42/variables/AWS_SECRET" \
  --form "value=new_value"

# 刪除
curl --request DELETE \
  --header "PRIVATE-TOKEN: $TOKEN" \
  --url "https://gitlab.example.com/api/v4/projects/42/variables/AWS_SECRET"

🔒 4. Mask / Protect / Hidden

Quick Start三種安全屬性

屬性效果注意
Maskedlog 中顯示 [MASKED]需滿足 mask 規則
Masked and hiddenlog 遮蔽 + UI 也看不到GitLab 17.4+ ,一旦設了 UI 再也看不到值
Protected只在 protected branch / tag 的 pipeline 可用feature branch 拿不到

SRE 實務Mask 的限制

值必須符合:

  • 單行,無空格
  • 至少 8 字元
  • 只能用:字母數字、_、:、@、-、+、.、~、=、/
  • 不能是預定義或既有變數名
⚠️ Mask 不是萬能的:
  • 若 script 改寫了值(如 base64、URL encode),改寫後的形式不會被 mask
  • Shell 加 \ escape 後不會 mask
  • Debug trace(CI_DEBUG_TRACE=true)會曝露變數
  • 大於 4KB 的值無法可靠 mask

SRE 實務Protected 變數完整保護鏈

Protected Branch "main"        (Settings → Repository)
    │
    ▼
Protected Variable AWS_PROD_KEY  (Settings → CI/CD → Variables → ☑ Protect)
    │
    ▼
Protected Environment production (Settings → CI/CD → Protected env)
    │
    ▼
Protected Runner deploy-prod     (Runner settings → ☑ Protected)
    │
    ▼
只有 main branch 的 pipeline 能取得 AWS_PROD_KEY
在 deploy-prod runner 上執行,部署到 production 環境

架構視角Masked 仍不是「保證安全」

即便 mask 了,惡意使用者仍可能:

  • 把值 base64 後印到 log(繞過 mask 偵測)
  • 發 HTTP 請求把值傳到外部
  • 寫到 artifact 檔案

真正敏感的 secret 應該用 External Secrets(Vault、AWS Secrets Manager、GCP Secret Manager)。見 第 8 節 與第 8 章(Pipeline Security)。

🎯 5. Environment Scope(Premium+)

Quick Start為什麼要限縮 scope?

預設所有 CI 變數全 pipeline 可用。這表示 test job 若用了被攻破的套件,可能偷到 deploy job 的 production key — 典型的供應鏈攻擊。

Environment scope 讓變數只在特定 environment 的 job 可用。

Quick Start設定方式

UI → 變數編輯 → Environment scope:

  • *(預設):全部環境
  • production:只有 environment: production 的 job
  • production/*:production 及其所有子環境
  • review/*:所有 review app

SRE 實務範例:多環境金鑰隔離

# UI 設定(3 個變數同名,不同 scope)
AWS_KEY(scope: staging)         → staging-key-xxx
AWS_KEY(scope: production)      → prod-key-yyy
AWS_KEY(scope: *)                → default-key

# .gitlab-ci.yml
deploy-staging:
  script: ./deploy.sh
  environment: staging           # 拿到 staging-key

deploy-prod:
  script: ./deploy.sh
  environment: production        # 拿到 prod-key

random-job:
  script: echo "$AWS_KEY"        # 拿到 default-key

架構視角完整保護策略

層次工具效果
最弱一般變數(* scope)任何 job 可用
中Protected只 protected branch 能用
強Protected + Environment scope只特定環境 + protected branch
最強External Secret(Vault)動態拉取、短期、有 audit

📄 6. File Type 變數

Quick Start概念

File type 變數的值會寫成檔案,變數本身是檔案路徑。

適用於需要檔案輸入的工具:SSH key、TLS 憑證、kubeconfig、GCP service account JSON。

Quick Start範例

# UI 建立 File type 變數:
#   Key: KUBECONFIG
#   Value: (貼 kubeconfig YAML 內容)
#   Type: File

deploy:
  script:
    # $KUBECONFIG 是檔案路徑(如 /builds/tmp/KUBECONFIG)
    - kubectl --kubeconfig=$KUBECONFIG apply -f deployment.yml
    # 或直接 export(kubectl 會自動找 $KUBECONFIG 環境變數)
    - kubectl apply -f deployment.yml

SRE 實務常見使用場景

# SSH deploy key
deploy-via-ssh:
  variables:
    SSH_KEY: $SSH_PRIVATE_KEY     # File type
  script:
    - eval $(ssh-agent -s)
    - ssh-add $SSH_KEY
    - ssh user@host "./deploy.sh"

# GCP service account
deploy-gcp:
  script:
    - gcloud auth activate-service-account --key-file=$GCP_SA_KEY
    - gcloud deploy ...

# TLS 憑證
build-with-cert:
  script:
    - cp $TLS_CERT certs/server.crt
    - cp $TLS_KEY certs/server.key
    - make build

架構視角File type 常見誤用

⚠️ 新手常犯:把 $VAR 當成值來用,結果拿到的是檔案路徑。 
# ❌ 錯誤:SSH_KEY 是 File type,$SSH_KEY 是路徑
ssh-add "$SSH_KEY"   # 這樣是對的,傳路徑給 ssh-add
echo "$SSH_KEY"      # 印出路徑(/tmp/xxx),不是 key 內容

# ✅ 如果工具要內容而非路徑
CONTENT=$(cat "$SSH_KEY")
some-tool --key "$CONTENT"

🏆 7. 變數優先順序

當同名變數在多處定義,GitLab 依優先順序決定用哪個(數字小優先):

  1. Trigger 變數、Schedule 變數、Manual pipeline 變數
  2. Project 層 UI 變數
  3. Group 層 UI 變數
  4. Instance 層 UI 變數
  5. Inherited 變數(父 pipeline 的)
  6. YAML variables: 區塊(job 層 > workflow > pipeline 層)
  7. Dotenv(artifact)變數
  8. Predefined 變數(CI_*)

SRE 實務常見陷阱

# YAML 定義
variables:
  APP_ENV: staging

# 但 Project UI 有設 APP_ENV=production
# 結果:實際執行時 APP_ENV=production(UI 覆蓋 YAML)
⚠️ 若 YAML 的變數「看起來沒生效」,第一個要檢查的是:是否有同名 UI 變數覆蓋。

架構視角變數解析時機

變數有三個展開時機,都在不同階段發生:

  1. GitLab 內部展開(pipeline 建立時):用於 rules:if、include 等
  2. Runner 展開(runner 取到 job 時):處理 job config
  3. Shell 展開(script 執行時):在 bash/sh 裡的 $VAR

不同階段可用的變數不同:

  • rules:if 可用 predefined 和 YAML variables,不可用 dotenv 或 script 內 export 的
  • script 內可用所有前面階段產生的變數
  • after_script 與 script 是分離的 shell context,script 裡 export 的變數在 after_script 拿不到

🚀 8. 進階:expand、raw、dotenv、inputs

SRE 實務expand — 變數內引用變數

variables:
  BASE_URL: "https://api.example.com"
  API_V1: "$BASE_URL/v1"       # 展開:https://api.example.com/v1
  API_V2: "$BASE_URL/v2"       # 展開:https://api.example.com/v2

SRE 實務raw — 不展開

有些值字面就含 $,你不想它被當成變數展開:

# UI 變數設定 raw = true
# Value: $SECRET_WITH_DOLLAR_SIGN

# 或 YAML
variables:
  MY_VAR:
    value: "literal-$not-a-variable"
    expand: false

SRE 實務Dotenv — 跨 job 傳變數

build:
  stage: build
  script:
    - VERSION="1.2.$CI_PIPELINE_ID"
    - echo "VERSION=$VERSION" > build.env
    - echo "IMAGE_TAG=myapp:$VERSION" >> build.env
  artifacts:
    reports:
      dotenv: build.env

deploy:
  stage: deploy
  needs: [build]
  script:
    # $VERSION 和 $IMAGE_TAG 自動可用
    - echo "Deploying $IMAGE_TAG"

Dotenv 格式:

  • 每行 KEY=value
  • Key 只能字母數字 + _
  • 最多 20 行(預設)

SRE 實務External Secrets(Vault)

job:
  id_tokens:
    VAULT_ID_TOKEN:
      aud: https://vault.example.com
  secrets:
    DB_PASSWORD:
      vault: production/db/password@secret
      file: false       # 當作 env var(預設 true 是寫成檔案)
  script:
    - echo "使用 $DB_PASSWORD"

優點:動態拉取、不落地 GitLab、Vault 端有 audit、可用 short-lived credentials。(詳見第 8 章)

架構視角Inputs Context(CI/CD Components)

spec:
  inputs:
    deploy_env:
      default: staging
---
deploy:
  script:
    # Inputs 用 $[[ inputs.xxx ]] 雙中括號
    - echo "Deploying to $[[ inputs.deploy_env ]]"
    # 注意和 shell 變數 $VAR 的語法差異

Inputs 在 pipeline 建立時插值,不是 runtime 展開。

架構視角Matrix Context

test:
  parallel:
    matrix:
      - OS: [linux, mac]
        VERSION: [18, 20]
  # 可在 needs 用 matrix context 建立對應
  # (CI/CD 18.x 支援)

🌟 9. 最佳實踐

  • 敏感資料放 UI,不放 YAML
  • Production secret 加 Protected + Environment scope
  • Secret 長度 ≥ 8 且符合 mask 規則
  • File type 給需要檔案的工具用
  • 避免用 CI_DEBUG_TRACE(會曝露變數)
  • 定期輪替 UI variables 裡的 token
  • 使用 description 讓團隊理解變數用途
  • 重要 secret 考慮 External Secrets(Vault)
  • 跨 job 傳值用 dotenv,不要用檔案或環境變數
  • Group 層統一管共用變數,避免重複

SRE 實務常見反模式

❌ 避免✅ 改用
YAML 裡寫密碼UI Variables + Masked
Group 變數給所有子專案共用 prod keyEnvironment scope + Protected
CI_DEBUG_TRACE=true 忘記關用 set -x 單點開啟
script 裡 echo $SECRET不 echo,或用 set +x 關 trace
把 secret 放 artifactVault 動態拉取
所有變數都 masked只 mask 真的敏感的

✅ Variables 設計 Checklist

  • Secret 都在 UI,YAML 裡沒有密碼
  • Production secret 設 Protected
  • Mask 符合規則(≥ 8 字、無空格)
  • Environment scope 隔離多環境 secret
  • File type 用於需要檔案的工具
  • Manual pipeline 用 description + options
  • 跨 job 傳值用 dotenv
  • 大量 secret 走 Vault
  • 每個 UI 變數有 description
  • 定期輪替 token

🆘 變數除錯

問題檢查
變數是空的優先順序、scope、protected 是否適用
Mask 沒生效長度 ≥ 8、字元符合規則
rules 判斷失敗rules 階段某些變數不可用(dotenv、runtime)
after_script 拿不到shell context 分離,要寫到檔或 dotenv
Expand 沒作用Masked 變數不能 expand;檢查設定
