📋 Job 基本要求

一個有效的 job 必須:

  • 定義在 YAML 的最頂層(不是巢狀於其他 key)
  • 有唯一 job 名稱(255 字元內)
  • 有 script 或 trigger 其中一個
  • 名稱不能與保留字衝突(stages, variables, workflow, include, default, types, image, services, cache, before_script, after_script, pages)

Quick StartJob 命名慣例

✅ 推薦❌ 避免
build-backendbuild(太泛用)
test:unittest1(無意義編號)
deploy-prodDeploy to Production(空格可讀性差)
.template(hidden)template(與保留字相近)
💡 Job 名稱以 .(句點)開頭代表 hidden job,不會在 pipeline 執行,但可以被 extends 繼承當模板用。

♻️ 1. Job 生命週期

┌──────────────────────────────────────────┐
│  1. Runner 接到 job                       │
│  2. Prepare:準備容器/環境                 │
│  3. Get sources:clone / fetch 程式碼      │
│  4. Restore cache(若有)                   │
│  5. Download artifacts(從上游 job)       │
│  6. Execute before_script                 │
│  7. Execute script     ←── 核心邏輯       │
│  8. Execute after_script(無論成敗都跑)   │
│  9. Upload artifacts(若成功或設 always)  │
│  10. Upload cache                         │
│  11. Clean up                             │
└──────────────────────────────────────────┘

SRE 實務關鍵理解點

  • cache 在 artifacts 之前還原 — 若兩者 path 重疊,artifacts 會覆蓋 cache
  • after_script 不影響 job 結果 — 就算 script 失敗,after_script 還是會跑
  • after_script 預設 5 分鐘超時(可用 RUNNER_AFTER_SCRIPT_TIMEOUT 調)
  • after_script 產生的檔案也會被納入 artifact / cache 上傳
  • 若 job 被 cancel,after_script 可能不會跑

📜 2. script / before_script / after_script

Quick Start三個階段

test:
  before_script:
    # 準備工作(設環境變數、安裝依賴)
    - echo "Setting up..."
    - apt-get update && apt-get install -y curl
    - export DB_URL=postgresql://localhost

  script:
    # 主要邏輯(失敗就是 job 失敗)
    - npm ci
    - npm test

  after_script:
    # 清理工作(不論 script 成敗都跑)
    - echo "Cleaning up..."
    - rm -rf tmp/

SRE 實務多行 script 的正確寫法

# ✅ 每個指令各自一行,失敗立刻停
script:
  - npm ci
  - npm test
  - npm run lint

# ✅ 需要多行邏輯:用 YAML 的 | 保留換行
script:
  - |
    if [ "$CI_COMMIT_BRANCH" == "main" ]; then
      echo "Deploying..."
      ./deploy.sh
    else
      echo "Skip deploy"
    fi

# ❌ 避免把長邏輯塞一行
script:
  - if [ ... ]; then ...; else ...; fi  # 難 debug

SRE 實務early exit 控制

# 預設:任一指令失敗 job 就失敗

# 允許個別指令失敗
script:
  - npm test || true   # 失敗也繼續(不建議,除非真的無所謂)

# 指令組合
script:
  - npm run lint && npm test   # AND:前者成功才跑後者
  - make check || make debug    # OR:前者失敗才跑後者

# 正確的 early exit
script:
  - set -e                 # 任一指令失敗立刻停(預設就是這樣)
  - set -euo pipefail      # 更嚴格:未定義變數、pipe 失敗都停

架構視角after_script 的限制

⚠️ after_script 不會影響 job exit code。就算 after_script 出錯,job 仍顯示成功。若 after_script 必須成功,設定: 
variables:
  AFTER_SCRIPT_IGNORE_ERRORS: "false"

架構視角script timeout 三層控制

job-long:
  timeout: 2h                        # 1. Job 整體 timeout
  variables:
    RUNNER_SCRIPT_TIMEOUT: 1h40m     # 2. script 階段 timeout
    RUNNER_AFTER_SCRIPT_TIMEOUT: 10m # 3. after_script timeout
  script:
    - ./long-running-task.sh
  after_script:
    - ./cleanup.sh

這三個 timeout 的值:script + after_script ≤ job timeout,讓 after_script 有時間跑完(清理、上傳 log)。

📦 3. Artifacts 完整指南

Quick Startartifacts 概念

Artifacts 是 job 產生的檔案,可以:

  • 傳遞給下游 job(透過 needs 或 stage 順序)
  • 在 GitLab UI 上下載
  • 顯示在 MR widget(若是 report 類型)
  • 給 GitLab Pages 用(若 job 叫 pages)

Quick Start完整語法

build:
  script:
    - npm run build
  artifacts:
    # 1. 基本設定
    name: "$CI_JOB_NAME-$CI_COMMIT_REF_SLUG"   # 下載時的檔名
    paths:
      - dist/
      - build/**/*.js
    exclude:
      - dist/**/*.map    # 排除 source map

    # 2. 控制
    expire_in: 1 week    # 或 30 days / never / 1 hour
    when: on_success     # on_success(預設) / on_failure / always

    # 3. 大小與曝光
    expose_as: "Built artifacts"   # 在 MR 顯示的名稱
    public: false                   # 公開可下載(GitLab.com)

    # 4. 特殊類型(見下)
    reports:
      junit: reports/junit.xml
      dotenv: build.env

SRE 實務artifacts:reports 類型

Report 類型用途層級
junit測試結果(xUnit XML)Free
coverage_report覆蓋率(Cobertura)Free
dotenv變數傳遞給下游Free
codequality程式品質(CodeClimate)Free
sast靜態安全分析Ultimate
dependency_scanning相依套件漏洞Ultimate
container_scanningContainer 掃描Ultimate
dast動態安全分析Ultimate
secret_detection機密偵測Ultimate
license_scanning授權掃描Ultimate
performance效能測試Premium
load_performance負載測試Premium
terraformTerraform 計劃Free

SRE 實務dotenv 跨 job 傳變數

build:
  stage: build
  script:
    - VERSION="1.2.3-$CI_PIPELINE_ID"
    - echo "VERSION=$VERSION" > build.env
    - echo "BUILD_IMAGE=myapp:$VERSION" >> build.env
    - docker build -t "myapp:$VERSION" .
    - docker push "myapp:$VERSION"
  artifacts:
    reports:
      dotenv: build.env

deploy:
  stage: deploy
  needs: [build]
  script:
    # $VERSION 和 $BUILD_IMAGE 自動可用
    - echo "Deploying $BUILD_IMAGE"
    - kubectl set image deployment/app app=$BUILD_IMAGE

SRE 實務只在失敗時上傳 log

e2e-test:
  script:
    - npm run e2e
  artifacts:
    when: on_failure       # 只在失敗時
    paths:
      - screenshots/
      - videos/
      - logs/
    expire_in: 2 days

架構視角控制下游哪些 artifact 要下載

build-frontend:
  artifacts:
    paths: [frontend-dist/]

build-backend:
  artifacts:
    paths: [backend.jar]

test:
  # 預設:下載所有上一 stage 的 artifact
  # 明確指定:
  dependencies:
    - build-backend   # 只下載這個
  # 或完全不要:
  # dependencies: []
  script: ...

test-no-deps:
  dependencies: []   # 不下載任何 artifact(加速)
  script: echo "independent"

💾 4. Cache 完整指南

Quick Start完整語法

test:
  script:
    - npm ci --prefer-offline
    - npm test
  cache:
    # 1. key 控制共享範圍
    key:
      files: [package-lock.json]    # lock 檔改變才失效
      prefix: "$CI_JOB_NAME"

    # 2. 要 cache 的路徑
    paths:
      - node_modules/
      - .npm/

    # 3. policy:pull / push / pull-push
    policy: pull-push       # 預設

    # 4. fallback
    fallback_keys:
      - cache-$CI_DEFAULT_BRANCH
      - cache-default

    # 5. 其他
    when: on_success       # 何時 push cache
    untracked: false        # 是否納入 untracked 檔案
    unprotect: false        # 是否與 protected branch 共享

SRE 實務cache:key 進階

# 方式 1:固定 key(所有人共用)
cache:
  key: node-deps
  paths: [node_modules/]

# 方式 2:依分支(每個分支獨立 cache)
cache:
  key: $CI_COMMIT_REF_SLUG
  paths: [node_modules/]

# 方式 3:依 lock file(改 lock 才失效)⭐ 推薦
cache:
  key:
    files: [package-lock.json]
  paths: [node_modules/]

# 方式 4:組合(job 名 + lock file)
cache:
  key:
    prefix: $CI_JOB_NAME-$CI_COMMIT_REF_SLUG
    files: [package-lock.json, yarn.lock]
  paths: [node_modules/]

SRE 實務pull-push policy 策略

# 方式 1:設定 cache 的 job(只有一個)
prepare-cache:
  script:
    - npm ci
  cache:
    key:
      files: [package-lock.json]
    paths: [node_modules/]
    policy: pull-push     # 會 push

# 方式 2:使用 cache 的 job(多個)
test:
  script:
    - npm test
  cache:
    key:
      files: [package-lock.json]
    paths: [node_modules/]
    policy: pull          # 只 pull,不 push

lint:
  script:
    - npm run lint
  cache:
    key:
      files: [package-lock.json]
    paths: [node_modules/]
    policy: pull          # 只 pull
✅ 這個模式避免多個 job 同時 push 相同 cache,節省 runner 頻寬且避免競爭。

SRE 實務多個 cache 條目

test:
  script: npm test
  # 最多 4 個 cache 條目
  cache:
    - key:
        files: [package-lock.json]
      paths: [node_modules/]
      policy: pull-push

    - key: build-cache-$CI_COMMIT_REF_SLUG
      paths: [.build-cache/]
      policy: pull-push

架構視角各語言 cache 最佳實踐

語言 / 工具cache 路徑key 依據
Node.js (npm)node_modules/package-lock.json
Node.js (yarn).yarn-cache/ + node_modules/yarn.lock
Python (pip).cache/pip/ + venv/requirements.txt
Python (poetry).venv/poetry.lock
Java (Maven).m2/repository/pom.xml
Java (Gradle).gradle/caches/*.gradle*
Go$GOPATH/pkg/mod/go.sum
Rusttarget/ + ~/.cargo/Cargo.lock
Rubyvendor/ruby/Gemfile.lock
.NET~/.nuget/packages/*.csproj

架構視角分散式 cache(S3)

多台 runner 共用 cache 的設定(管理員在 config.toml):

[[runners]]
  [runners.cache]
    Type = "s3"
    Shared = true
    [runners.cache.s3]
      ServerAddress = "s3.amazonaws.com"
      BucketName = "gitlab-runner-cache"
      BucketLocation = "ap-northeast-1"
      AuthenticationType = "iam"   # 推薦用 IAM Role

這樣任何 runner 都能拉同一份 cache,明顯加速 parallel build。

🔀 5. Parallel 與 Matrix

Quick Start簡單平行

# 跑 5 個一模一樣的 job
test:
  parallel: 5
  script: npm run test:parallel

會產生 test 1/5, test 2/5 ... test 5/5,job 內部用 $CI_NODE_INDEX 和 $CI_NODE_TOTAL 變數決定該跑哪部分。

SRE 實務切分測試範例

test:
  parallel: 4
  script:
    # 用 CI_NODE_INDEX 決定自己跑哪些檔案
    - |
      TEST_FILES=$(find test -name "*.spec.js" | awk "NR%$CI_NODE_TOTAL==$CI_NODE_INDEX-1")
      npm test -- $TEST_FILES

# 或用 knapsack(Ruby 專案)
rspec:
  parallel: 4
  script:
    - bundle exec rake knapsack:rspec

Quick StartMatrix 平行

test:
  parallel:
    matrix:
      - NODE_VERSION: ["18", "20", "22"]
        OS: ["linux", "windows"]
  script:
    - echo "Testing on Node $NODE_VERSION / $OS"
    - npm test
# 展開為 3 × 2 = 6 個 job

SRE 實務Matrix 進階

deploy:
  parallel:
    matrix:
      # 單一組合
      - PROVIDER: aws
        REGION: ap-northeast-1

      - PROVIDER: aws
        REGION: eu-west-1

      - PROVIDER: gcp
        REGION: asia-east1
  script:
    - ./deploy.sh $PROVIDER $REGION

架構視角Parallel 考量

  • Runner 容量:parallel: 10 需要 10 個 runner slot
  • Cache 競爭:多 job 同時 push 同 cache key 會失敗,改用 policy: pull
  • External service 負載:跑 10 個 e2e 同時連 DB,要先擴容
  • Report 合併:GitLab 會自動合併 parallel job 的 junit report

🔁 6. Retry / Timeout / Interruptible

Quick Start基本 retry

# 簡單:失敗重試最多 2 次
test:
  script: npm test
  retry: 2

# 進階:指定什麼錯才重試
test:
  script: npm test
  retry:
    max: 2
    when:
      - runner_system_failure    # runner 本身問題
      - stuck_or_timeout_failure  # 卡住
      - api_failure              # GitLab API 出錯
      - scheduler_failure        # 排程問題
      - stale_schedule           # 排程過期
      - data_integrity_failure   # 資料錯誤
      # 注意:不要包含 script_failure,因為 flaky test 應該修

SRE 實務retry when 的值

when 值情境
always任何失敗
script_failurescript 回傳非零(小心!可能掩蓋 flaky test)
runner_system_failurerunner 軟硬體出問題
stuck_or_timeout_failure長時間無回應
api_failureGitLab API 問題
scheduler_failure排程失敗
unknown_failure未分類失敗

Quick Starttimeout

job:
  timeout: 30 minutes    # 或 1h / 2h 30m / 3600(秒)
  script: long-task.sh

有三個層級的 timeout,取最小值:

  1. Job 層:timeout: 30m
  2. Project 層:Settings → CI/CD → General pipelines → Timeout
  3. Runner 層:註冊時或 config.toml 的 maximum_timeout

SRE 實務interruptible

default:
  interruptible: true    # 預設可中斷

build:
  script: make
  # 繼承 default:interruptible: true

deploy-prod:
  script: ./deploy.sh
  interruptible: false    # 部署不可中斷

搭配 workflow auto_cancel:

workflow:
  auto_cancel:
    on_new_commit: interruptible   # 新 commit 取消可中斷 job

架構視角retry 策略

  • 不要對 flaky test 用 retry 掩蓋 — 修 test
  • 要對基礎設施問題用 retry — 如 runner 錯、網路抖動
  • 部署 job慎用 retry — 重試可能重複部署
  • external API 呼叫加 retry 合理 — transient failure 常見

🎛️ 7. when 與 manual job

Quick Startwhen 的六個值

when行為
on_success(預設)前一個 stage 都成功才跑
on_failure前一個 stage 有失敗才跑
always無論前面成敗都跑(清理、通知用)
manual需要手動點擊觸發
delayed延遲一段時間再跑
never不建立這個 job

Quick Start典型組合

notify-success:
  stage: notify
  when: on_success
  script: ./notify.sh "Build succeeded"

notify-failure:
  stage: notify
  when: on_failure
  script: ./notify.sh "Build FAILED"

cleanup:
  stage: cleanup
  when: always
  script: rm -rf /tmp/build/*

deploy-prod:
  stage: deploy
  when: manual           # 手動觸發
  script: ./deploy.sh
  environment: production

# 延遲 deploy(冷卻期)
deploy-canary:
  stage: deploy
  when: delayed
  start_in: 30 minutes   # 前一 stage 成功後延 30 分鐘
  script: ./canary.sh

SRE 實務Manual job 進階

# 1. 允許失敗的 manual(非 blocking)
optional-deploy:
  when: manual
  allow_failure: true     # 沒跑也不影響 pipeline 狀態
  script: ./optional.sh

# 2. blocking manual(預設,不跑就卡住)
required-approval:
  when: manual
  allow_failure: false
  script: ./approve.sh

# 3. 要求二次確認
dangerous-action:
  when: manual
  manual_confirmation: "確定要刪除生產資料庫?"
  script: ./destroy-db.sh

架構視角when 與 rules 的關係

# when 在 rules 裡有優先權
job:
  when: manual            # 這會被 rules 裡的 when 覆蓋
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      when: on_success    # main 時自動跑
    - if: $CI_COMMIT_BRANCH =~ /^feature/
      when: manual        # feature branch 需手動
  script: echo "smart"

🚀 8. Deployment Job

Quick Start什麼是 Deployment Job?

只要 job 有 environment:,就是 Deployment Job。GitLab 會追蹤部署歷史、顯示在 Environments 頁。

Quick Start基本範例

deploy-staging:
  stage: deploy
  script: ./deploy.sh staging
  environment:
    name: staging
    url: https://staging.example.com
  rules:
    - if: $CI_COMMIT_BRANCH == "develop"

deploy-prod:
  stage: deploy
  script: ./deploy.sh production
  environment:
    name: production
    url: https://example.com
    deployment_tier: production
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      when: manual

SRE 實務Dynamic Environment(Review App)

review-app:
  stage: review
  script:
    - ./deploy-preview.sh $CI_COMMIT_REF_SLUG
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    url: https://$CI_COMMIT_REF_SLUG.preview.example.com
    on_stop: stop-review-app       # MR 關閉時怎麼停
    auto_stop_in: 1 week            # 一週沒活動自動停
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"

stop-review-app:
  stage: review
  script:
    - ./destroy-preview.sh $CI_COMMIT_REF_SLUG
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    action: stop
  when: manual
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"

SRE 實務environment 的 action

action意義
start(預設)部署 / 啟動環境
prepare準備但不計入部署歷史(例:審批前)
stop停止環境
verify驗證部署(不改狀態)
access存取環境(如拿 kubeconfig)

架構視角Protected Environment

路徑:Project → Settings → CI/CD → Protected environments

  • 限制誰能部署到 production
  • 要求審批(Ultimate)
  • 要求特定 role / group
  • 跟 protected branch + protected variable 組合,形成完整保護鏈

🔧 9. 進階:resource_group / hooks

SRE 實務resource_group — 確保 serial execution

deploy-prod:
  stage: deploy
  script: ./deploy.sh
  environment: production
  resource_group: production-deploy    # 同 group 一次只能跑一個

用途:

  • 避免並發部署:同 group 的 job 一次只跑一個
  • DB migration:不能多個並發改 schema
  • 共用資源:同一個 Kubernetes namespace 一次只改一個

SRE 實務resource_group process_mode

deploy-prod:
  resource_group:
    key: production
    process_mode: oldest_first   # oldest_first / newest_first / unordered
  • unordered(預設):隨便排
  • oldest_first:先進先出(適合排隊部署)
  • newest_first:後進先出(適合「只部最新」)

架構視角hooks(Runner hooks)

job:
  hooks:
    pre_get_sources_script:
      - echo "在 clone 程式碼之前執行"
      - ./setup-credentials.sh
  script:
    - make build

何時用:需要在 git clone 之前做事,例如設定 SSH agent、配置網路、預先下載工具。

架構視角inherit — 控制繼承

default:
  image: alpine:3.18
  before_script:
    - apk add --no-cache curl

variables:
  GLOBAL_VAR: "set"

# 明確關閉繼承
isolated-job:
  image: ubuntu:22.04     # 覆蓋 default image
  inherit:
    default: false         # 不繼承 default 的任何東西
    variables: false       # 不繼承全域 variables
  script:
    - echo "from scratch"

# 只繼承特定項目
selective-job:
  inherit:
    default: [image]       # 只繼承 default 的 image
    variables: [GLOBAL_VAR] # 只繼承此變數
  script:
    - echo "$GLOBAL_VAR"

💾 Cache 速查

# 基本(per-branch)
cache:
  key: $CI_COMMIT_REF_SLUG
  paths: [node_modules/]

# 推薦(依 lock file)
cache:
  key:
    files: [package-lock.json]
  paths: [node_modules/]
  policy: pull-push

# 只讀(下游 job)
cache:
  key:
    files: [package-lock.json]
  paths: [node_modules/]
  policy: pull

📦 Artifact 速查

# 基本
artifacts:
  paths: [dist/]
  expire_in: 1 week

# 測試報告(JUnit)
artifacts:
  when: always
  reports:
    junit: reports/junit.xml

# 變數傳遞
artifacts:
  reports:
    dotenv: build.env

# 只在失敗時留
artifacts:
  when: on_failure
  paths: [logs/]
  expire_in: 3 days

🎛️ Manual / Deployment 速查

# 手動觸發 + 允許失敗
deploy:
  when: manual
  allow_failure: true

# 部署到 production(含 URL)
deploy-prod:
  environment:
    name: production
    url: https://example.com
    deployment_tier: production

# Review App(每 branch 一個環境)
review:
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    url: https://$CI_COMMIT_REF_SLUG.preview.com
    on_stop: stop-review
    auto_stop_in: 1 week

🔁 Retry 速查

retry:
  max: 2
  when:
    - runner_system_failure
    - stuck_or_timeout_failure
    - api_failure

✅ Job 設計 Checklist

  • job 名稱有意義且唯一
  • 每個 job 有明確 stage 或 needs
  • dependency 有 cache 加速
  • 跨 job 資料用 artifact
  • 部署 job 有 environment
  • 生產部署有 manual + resource_group
  • retry 只對 infra 錯誤
  • timeout 合理(不會無限卡)
  • interruptible 正確設定
  • 測試 job 有 JUnit report

🆘 常見問題

問題解法
Cache 沒命中key 檢查;不同 runner 可能各有 cache
Artifact 太大用 exclude、分拆 job、設 expire_in
Job 卡在 pendingrunner tag、protected、忙碌度
deploy 重複觸發用 resource_group
Flaky test 一直 retry不要用 retry 掩蓋,修 test
Manual job 擋住 pipeline加 allow_failure: true
