⚠️ 先釐清:兩種不同的 Feature Flag

GitLab 裡面「Feature Flag」這個詞會出現在兩個地方,新手很容易搞混:

A. 管理員 Feature Flags(本章主題) B. 專案用 Feature Flags(非本章)
目的 控制 GitLab 自身新功能的開關 給開發者用,控制自家應用程式功能
設定位置 Rails Console / API Project → Deploy → Feature flags
誰能改 GitLab 管理員 專案 Maintainer 以上
底層 Flipper gem Unleash 相容 API
文件路徑 /administration/feature_flags/ /operations/feature_flags/
💡 本章只討論 A。B 是給你的應用程式用的 A/B testing 工具,屬於產品功能,不是管理主題。

🎯 1. 核心概念與生命週期

Quick Start為什麼 GitLab 要用 Feature Flag?

GitLab 每個月都會發佈新版本。一個新功能從開發到穩定,不會一次全開,而是:

  1. 功能上線,但藏在旗標後面,預設關閉
  2. GitLab.com 先開啟測試,收集回饋
  3. 改為預設開啟,但保留旗標可關閉
  4. 完全穩定後,移除旗標,永久啟用

SRE 實務三種旗標類型

類型典型用途是否可在 prod 用
development開發中新功能❌ 不建議
beta進入測試階段⚠️ 謹慎評估
ops運維用開關(如降壓、旁路)✅ 可以

架構視角為什麼不該把 Feature Flag 當作設定 (Settings)?

GitLab 官方強調:Feature Flag 是短命的,不是長期設定。如果你希望長期維持某個功能開關,官方建議改用:

  • Application Settings(透過 Admin Area UI)
  • Cascading Settings(階層式設定)
⚠️ 重要限制:GitLab Dedicated 不支援 Feature Flag 操作。架構選型時要注意。

🛠️ 2. 操作方式:Rails Console / API

Quick Start最基本的三件事

所有操作都要有 Admin 權限 或能登入 Rails Console 的伺服器帳號。

SRE 實務Rails Console 指令大全

進入 console:

sudo gitlab-rails console

查詢單一旗標狀態:

Feature.enabled?(:example_feature)
# => true / false

啟用 / 停用旗標:

# 全站啟用
Feature.enable(:example_feature)

# 全站停用
Feature.disable(:example_feature)

# 只對特定 project 啟用
Feature.enable(:example_feature, Project.find(42))

# 只對特定 group 啟用
Feature.enable(:example_feature, Group.find(1))

# 只對特定 user 啟用
Feature.enable(:example_feature, User.find_by_username('alice'))

# 按百分比啟用(gradual rollout)
Feature.enable_percentage_of_time(:example_feature, 50)    # 50% 請求
Feature.enable_percentage_of_actors(:example_feature, 25)  # 25% 使用者

重設旗標(回到預設值):

Feature.remove(:example_feature)

列出所有已設定過的旗標:

Feature.all.map { |f| [f.name, f.state] }

SRE 實務用 API 操作(自動化推薦)

# 查詢所有 feature flag
curl --header "PRIVATE-TOKEN: <admin-token>" \
  "https://gitlab.example.com/api/v4/features"

# 啟用旗標
curl --data "value=true" \
  --header "PRIVATE-TOKEN: <admin-token>" \
  "https://gitlab.example.com/api/v4/features/my_feature"

# 停用旗標
curl --request DELETE \
  --header "PRIVATE-TOKEN: <admin-token>" \
  "https://gitlab.example.com/api/v4/features/my_feature"

# 按百分比 rollout(40%)
curl --data "value=40" \
  --header "PRIVATE-TOKEN: <admin-token>" \
  "https://gitlab.example.com/api/v4/features/my_feature"

架構視角大規模管理考量

  • 不要手工管,要版本控制:把所有 feature flag 操作寫成 script,進 Git repo
  • 與 IaC 整合:Terraform GitLab provider 可以管部分設定(但 feature flags 目前仍以 API 為主)
  • 審計追蹤:每次改動都要在 change log 記錄,包含 who / when / why
  • 多環境驗證:staging 先開 → 觀察 → prod 再開

🚨 3. 風險管理與升級注意事項

Quick Start三條保命原則

⚠️ 開啟預設停用的旗標,可能導致:資料毀損、穩定性劣化、效能衰退、安全性問題。GitLab Support 不涵蓋預設停用功能造成的問題。
  1. 不要把預設停用的旗標直接開到 prod
  2. 升級 GitLab 前檢查所有自訂旗標狀態
  3. 開啟前先讀該功能的官方文件,確認當前版本的 flag 狀態

SRE 實務升級期間的旗標檢查流程

# 升級前:匯出當前旗標清單
sudo gitlab-rails runner 'puts Feature.all.map { |f| "#{f.name}=#{f.state}" }' \
  > /tmp/flags_before_upgrade.txt

# 升級後:再匯出一次
sudo gitlab-rails runner 'puts Feature.all.map { |f| "#{f.name}=#{f.state}" }' \
  > /tmp/flags_after_upgrade.txt

# 比對差異
diff /tmp/flags_before_upgrade.txt /tmp/flags_after_upgrade.txt
💡 升級時 GitLab 會變更旗標預設值(某些 flag 會從 off 變 on,或直接被移除)。這是影響行為變化的主因之一。

架構視角治理策略

層級實踐
政策公司內部訂定「哪些旗標允許生產啟用」白名單
審核啟用前需經 change review(特別是 experimental 類)
監控對關鍵旗標設 Prometheus alert(若狀態意外改變)
回滾每個旗標變更都要有 rollback plan
清理定期檢查,移除已從 codebase 消失的舊旗標設定
⚠️ 安全性熱點:預設停用功能中發現的安全漏洞,不適用常規維護政策的 backport。也就是說,舊版不會被 patch,必須升級。

💼 4. 實務場景

Quick Start你最可能遇到這幾種情境

📢 官方文件叫你開某個 flag

功能文件頁尾通常有 "Enable or disable" 區塊。直接照著指令做即可。

🐛 版本升級後某功能行為變了

很可能是旗標預設值改了。查 release notes 的 "Feature flag changes" 段落。

🚫 想停用某個你不喜歡的新 UI

部分 UI 改動也是旗標控制。搜尋功能名稱 + "feature flag"。

⚡ Support 請你暫時停用某功能

故障排除時,最常被要求關掉的是效能相關的旗標。

SRE 實務常見的 Ops 類旗標(運維用)

這類旗標通常可以放心在 prod 操作,GitLab 官方文件會明確標記為 "ops":

  • 限流類:在流量尖峰時暫時降級某些非關鍵功能
  • 索引重建:資料庫索引遷移的階段開關
  • Sidekiq queue 啟停:特定背景工作的開關
  • Cache 行為:快取策略切換

架構視角何時 不要 動 Feature Flag?

  • ❌ 不要用 flag 做長期功能差異化 — 用 Settings / License tier 代替
  • ❌ 不要把 flag 當作 kill switch(除非 GitLab 官方標示為 ops)
  • ❌ 不要在沒讀文件的情況下啟用預設停用的 flag
  • ❌ 不要在 Dedicated 上嘗試改 flag(不支援)
  • ❌ 不要讓 flag 成為你架構相依的一環(它會消失)

📋 啟用前 Checklist

  • 讀過官方文件上該功能的說明
  • 確認當前 GitLab 版本支援此 flag
  • 該 flag 不是 development-only
  • 有在 staging 先測過
  • 有記錄 change log(誰、何時、為何啟用)
  • 有回滾指令備妥
  • 已通知可能受影響的使用者

🆘 故障排除速查

症狀可能原因動作
啟用指令報 "unknown feature"當前版本已移除該 flag查該版本 release notes
啟用了但功能沒出現需要 sidekiq / puma 重啟或快取過期gitlab-ctl restart
升級後狀態不一樣預設值變更對比升級前後清單
部分使用者看得到部分看不到flag 用了 percentage rollout用 Feature.remove 重設
