🗺️ 測試體系總覽

GitLab 支援多種測試 report,透過 artifacts:reports 整合到 MR / pipeline 頁:

Report 類型用途層級MR Widget
junit測試結果(成功/失敗/錯誤)Free✅
coverage_report覆蓋率視覺化(行級)Free✅(Diff)
coverage(keyword)覆蓋率百分比(總覽)Free✅
codequality程式碼品質問題Free✅
accessibilityWCAG 無障礙檢查Free✅
browser_performance網頁效能(Lighthouse)Premium✅
load_performance負載測試(k6)Premium✅
metrics自訂 metricPremium✅
terraformTerraform planFree✅
api_fuzzingAPI 模糊測試Ultimate✅
💡 artifacts:reports 的共同特性:成功失敗都上傳、可用 expire_in 調保留期、可配合 artifacts:paths 讓使用者下載原檔。

Quick Start典型測試 job 結構

test:
  stage: test
  image: node:20
  script:
    - npm ci
    - npm test -- --reporters=default --reporters=jest-junit --coverage
  coverage: '/All files\s+\|\s+([0-9.]+)/'    # 抓整體 %
  artifacts:
    when: always                               # 失敗也要上傳
    expire_in: 1 week
    paths:
      - coverage/                              # 可下載的原檔
    reports:
      junit: junit.xml
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml

✅ 1. JUnit Report — 測試結果

Quick Start功能

JUnit XML 格式是業界標準,GitLab 能解析並顯示:

  • MR widget 顯示 Pass / Fail / Error 總數
  • Fail case 可展開看錯誤訊息 + stack trace
  • 顯示新失敗(本 MR 造成的)
  • 顯示執行時間(找慢的 test)

Quick Start基本設定

test:
  script:
    - npm test -- --reporters=jest-junit
  artifacts:
    when: always
    reports:
      junit: junit.xml

SRE 實務多個 JUnit 檔案合併

# 方式 1:多路徑
test:
  artifacts:
    reports:
      junit:
        - junit-unit.xml
        - junit-integration.xml

# 方式 2:萬用字元(多個 parallel job)
test-parallel:
  parallel: 4
  artifacts:
    reports:
      junit: junit-*.xml
⚠️ 不支援指向資料夾(junit: test-results/),只能檔名或萬用字元。

SRE 實務讓 job 失敗仍上傳 report

預設 artifacts 只在 job 成功時上傳。測試 job 失敗(有紅字)時也要看 report,加 when: always:

test:
  script: npm test
  artifacts:
    when: always          # 關鍵!
    reports:
      junit: junit.xml

架構視角Flaky Test Detection(Ultimate)

GitLab Ultimate 會根據 JUnit report 歷史追蹤,自動識別 flaky test(時好時壞):

  • 在 MR widget 顯示 "Flaky" 標籤
  • 提供 flaky test 儀表板
  • 可設定「flaky test 不擋 pipeline」

📊 2. Code Coverage — 覆蓋率

Quick Start兩種層次

方式顯示位置
coverage keyword(regex)MR 上方的百分比、Pipeline badge
artifacts:reports:coverage_reportMR diff 的行級標示

Quick Start整體 % 設定

test:
  script:
    - pytest --cov=src --cov-report=term
  # regex 抓測試輸出最後的百分比
  coverage: '/TOTAL.+ ([0-9]{1,3}%)/'

SRE 實務各語言 coverage regex

語言 / 工具Regex 範例
Python (pytest-cov)/TOTAL.+ ([0-9]{1,3}%)/
Python (coverage.py)/TOTAL\s+\d+\s+\d+\s+(\d+%)/
Node (Jest)/All files\s+\|\s+([\d.]+)/
Node (nyc/istanbul)/Lines\s*:\s*(\d+(?:\.\d+)?)%/
Go/coverage: \d+\.\d+% of statements/
Ruby (SimpleCov)/\(\d+\.\d+\%\) covered/
Java (JaCoCo)/Total.*?([0-9]{1,3})%/
.NET (Coverlet)/Total\s*\|\s*([\d.]+)%/
PHP (PHPUnit)/^\s*Lines:\s*(\d+(?:\.\d+)?%)/

SRE 實務多 job 合併覆蓋率

test-unit:
  script: coverage run unit/
  coverage: '/TOTAL.+ ([0-9]{1,3}%)/'

test-integration:
  script: coverage run integration/
  coverage: '/TOTAL.+ ([0-9]{1,3}%)/'

# GitLab 會自動算兩個 job 的平均並顯示

架構視角Coverage Check Approval Rule

路徑:Project → Settings → Merge requests → Approval rules

  • 啟用 Coverage-Check rule
  • 若 MR 的 coverage 比 target branch 低,需要特定 reviewer 批准
  • 強制開發者重視覆蓋率

🔍 3. Coverage Visualization — 行級視覺化

Quick Start功能

在 MR 的 Changes tab 可看到哪行有被測試涵蓋(綠色)、哪行沒有(紅色)。

Quick Start支援格式

Format原生支援常用於
Cobertura✅多語言通用(Java、JS、Python、Ruby)
JaCoCo✅Java / Kotlin / Scala

SRE 實務設定範例

# Node.js + Jest + jest-junit + cobertura
test:
  script:
    - npm ci
    - npm test -- --coverage --coverageReporters=cobertura --reporters=jest-junit
  coverage: '/All files\s+\|\s+([\d.]+)/'
  artifacts:
    when: always
    paths: [coverage/]
    reports:
      junit: junit.xml
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml

# Java + Gradle + JaCoCo
test:
  script:
    - gradle test jacocoTestReport
  artifacts:
    reports:
      junit: build/test-results/test/TEST-*.xml
      coverage_report:
        coverage_format: jacoco
        path: build/reports/jacoco/test/jacocoTestReport.xml

# Python + pytest-cov
test:
  script:
    - pip install pytest pytest-cov
    - pytest --cov=src --cov-report=xml --junitxml=junit.xml
  coverage: '/TOTAL.+ ([0-9]{1,3}%)/'
  artifacts:
    reports:
      junit: junit.xml
      coverage_report:
        coverage_format: cobertura
        path: coverage.xml

架構視角多個 coverage report 合併

# parallel job 產多個 cobertura,GitLab 自動合併
test:
  parallel: 4
  script:
    - pytest --cov=src --cov-report=xml:coverage-$CI_NODE_INDEX.xml
  artifacts:
    reports:
      coverage_report:
        coverage_format: cobertura
        path: coverage-*.xml    # 萬用字元
💡 Coverage 資料預設 1 週過期,MR 若超過時間,視覺化會消失。

🧹 4. Code Quality — 程式碼品質

Quick Start功能

在 MR 顯示可維護性問題:

  • 重複 code
  • 函式過長
  • 過度複雜度(cyclomatic)
  • Lint 類問題

Quick Start用 GitLab 內建模板

include:
  - template: Jobs/Code-Quality.gitlab-ci.yml

# 或 Component 版
# - component: $CI_SERVER_FQDN/components/code-quality/codequality@1

預設用 CodeClimate engine,支援多語言。

SRE 實務自家 lint 整合成 CodeQuality 格式

lint:
  stage: test
  image: node:20
  script:
    - npm ci
    - npm run lint -- --format json --output-file gl-codequality.json
    # 需要把 eslint 輸出轉成 GitLab CodeQuality schema
  artifacts:
    when: always
    reports:
      codequality: gl-codequality.json

GitLab CodeQuality schema:

[
  {
    "description": "Missing semicolon",
    "check_name": "eslint/semi",
    "fingerprint": "abc123...",
    "severity": "minor",
    "location": {
      "path": "src/app.js",
      "lines": { "begin": 42 }
    }
  }
]

SRE 實務各工具轉換

  • ESLint:eslint-formatter-gitlab 套件直接產 GitLab 格式
  • Rubocop:rubocop --format json | jq ... 轉換
  • Python(ruff, flake8):社群工具如 flake8-gl-codeclimate
  • Go(golangci-lint):--out-format code-climate 原生支援

架構視角MR 顯示邏輯

  • 比較 source branch 的 report 與 target branch 的 report
  • 顯示新增的問題(MR 造成的)
  • 顯示已修復的問題(MR 改好的)
  • diff tab 也會在對應行標示

♿ 5. Accessibility Testing — 無障礙

Quick Start檢查項目

用 pa11y 檢查 WCAG 2.1 規則:

  • alt text 缺失
  • 顏色對比不足
  • 無法鍵盤操作
  • input 無 label
  • heading / landmark 結構錯

Quick Start設定

include:
  - template: Verify/Accessibility.gitlab-ci.yml

a11y:
  stage: test
  variables:
    a11y_urls: "https://example.com https://example.com/about"
  # 會產 gl-accessibility.json report

SRE 實務配合 Review App

a11y:
  stage: test
  extends: .accessibility
  variables:
    # 用 review app 的 URL
    a11y_urls: "https://$CI_COMMIT_REF_SLUG.preview.example.com"
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
  needs: [deploy-review]

架構視角何時該跑

  • 每個 MR:對前端專案強制
  • 排程:定期掃 production
  • 發版前:release pipeline

  • Total Blocking Time

⚡ 7. Load Performance(Premium)

Quick Start用途

用 k6 做負載測試,測量 API 在不同流量下的反應。

Quick Start設定

include:
  - template: Verify/Load-Performance-Testing.gitlab-ci.yml

load_performance:
  stage: performance
  variables:
    K6_TEST_FILE: ./test-script.js
  # 產 load-performance.json

SRE 實務k6 script 範例

// test-script.js
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '30s', target: 10 },   // warm up
    { duration: '1m', target: 50 },    // sustained
    { duration: '30s', target: 0 },    // cool down
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],   // 95% < 500ms
    http_req_failed: ['rate<0.01'],     // < 1% error
  },
};

export default function () {
  const res = http.get('https://api.example.com/users');
  check(res, {
    'status is 200': (r) => r.status === 200,
    'response < 500ms': (r) => r.timings.duration < 500,
  });
  sleep(1);
}

架構視角何時跑

  • 不在每個 MR 跑(太慢、太吃資源)
  • Schedule:每晚跑 staging
  • 發版前:release pipeline 跑 pre-prod
  • Major feature:針對特定 API 驗證

🔗 8. E2E 與跨 job 整合

Quick Start典型 E2E 流程

stages: [build, test, e2e]

build:
  stage: build
  script: npm run build
  artifacts:
    paths: [dist/]

unit-test:
  stage: test
  script: npm test -- --coverage
  artifacts:
    when: always
    reports:
      junit: junit.xml
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml

e2e-test:
  stage: e2e
  needs: [build]           # DAG:build 完就跑,不等 unit-test
  image: mcr.microsoft.com/playwright:v1.50.0
  services:
    - name: nginx:alpine
      alias: webapp
  script:
    - cp -r dist/* /usr/share/nginx/html/
    - npx playwright test --reporter=junit
  artifacts:
    when: always
    paths:
      - test-results/screenshots/    # 失敗時截圖
      - test-results/videos/         # 失敗時錄影
    reports:
      junit: results.xml
    expire_in: 3 days

SRE 實務Service Container(例:DB / Redis)

integration-test:
  image: python:3.12
  services:
    - name: postgres:16
      alias: db
    - name: redis:7
      alias: cache
  variables:
    # Service 可用 env var 設定
    POSTGRES_USER: test
    POSTGRES_PASSWORD: test
    POSTGRES_DB: testdb
    # 應用端連線設定
    DATABASE_URL: "postgresql://test:test@db:5432/testdb"
    REDIS_URL: "redis://cache:6379"
  script:
    - pip install -r requirements.txt
    - pytest tests/integration/

架構視角Parallel 切分加速

e2e:
  parallel: 4
  script:
    # Playwright 原生 shard 支援
    - npx playwright test --shard=$CI_NODE_INDEX/$CI_NODE_TOTAL
  artifacts:
    reports:
      junit: results.xml       # GitLab 自動合併

🎲 9. Flaky Test 處理

Quick Start什麼是 flaky test?

時好時壞的測試,同 commit 重跑有時過、有時失敗。破壞信心、浪費時間、掩蓋真問題。

SRE 實務處理原則

⚠️ 不要用 retry 掩蓋 flaky test。這會讓問題越來越多。

正確流程:

  1. 識別:GitLab Ultimate 自動追蹤,或自己記
  2. 標記:用 tag / skip,不擋 pipeline
  3. 分析:找出原因(race condition、外部依賴、時間相關)
  4. 修復:加 wait、mock 外部、修 bug
  5. 恢復:驗證穩定後重新啟用

SRE 實務Retry 的正確用法

# ❌ 避免:對 script_failure 無腦 retry
test:
  retry: 2   # 任何失敗都重試,可能掩蓋 bug

# ✅ 只對 infrastructure 問題 retry
test:
  retry:
    max: 2
    when:
      - runner_system_failure
      - stuck_or_timeout_failure
      - api_failure
      # 注意!不包含 script_failure

架構視角Flaky Test Dashboard(Ultimate)

GitLab 基於 JUnit 歷史自動偵測:

  • 同 test 近 14 天在 main 通過但 MR 失敗 → flaky
  • 儀表板顯示 top flaky tests
  • MR widget 會標示 "Flaky",可選擇不擋合併

架構視角常見 flaky 原因

原因解法
時間相關(sleep/timeout)用明確的 wait_for condition,不要固定 sleep
Race condition加 lock、正確同步
外部服務Mock 或用固定版本的 test double
Order 相依(test 間互相影響)每個 test 獨立 setup/teardown
環境差異(UTC vs local)固定 TZ、locale、seed
並發資源衝突每 test 獨立 DB/file

⚡ 速查表

🎯 Testing Report 速查

想顯示需要
MR widget 的 test 數量、失敗列表reports:junit
MR 上方的 coverage %coverage: keyword + regex
MR diff 上的行級覆蓋reports:coverage_report(cobertura / jacoco)
MR widget 的 code qualityreports:codequality
MR widget 的無障礙reports:accessibility
MR widget 的效能reports:browser_performance / load_performance

📋 Golden Test Job Template

test:
  stage: test
  image: node:20
  script:
    - npm ci
    - npm test -- --coverage --ci --reporters=default --reporters=jest-junit
  coverage: '/All files\s+\|\s+([\d.]+)/'
  artifacts:
    when: always
    expire_in: 1 week
    paths:
      - coverage/
      - junit.xml
    reports:
      junit: junit.xml
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml
  cache:
    key:
      files: [package-lock.json]
    paths: [node_modules/]
    policy: pull

✅ Testing Checklist

  • 每個 test job 都產 JUnit report
  • artifacts:when: always(失敗也要看 report)
  • Coverage regex 正確抓到總 %
  • Cobertura / JaCoCo 路徑正確
  • Failed screenshot / video 有存(E2E)
  • Flaky test 有識別機制
  • Parallel 適當切分慢測試
  • E2E 跑完 artifact 會清掉(別佔空間)
  • Code Quality + Security scan 都開
  • MR widget 正確顯示所有 report

🆘 常見問題

問題解法
JUnit widget 沒出現XML format 錯誤;檔案路徑不對;artifact 未上傳
Coverage % 顯示 0 或 unknownregex 不符合輸出;測試工具沒印出 %
Coverage 行級視覺化沒出現coverage_report 格式錯;路徑不對;過了 1 週
失敗 job 沒上傳 artifact加 when: always
多 parallel job 的 coverage 沒合併用萬用字元 path,或 GitLab 自動合併
Service container 連不到用 alias 名,不是 localhost
E2E 找不到測試 URL用 service alias + 設好 localhost hosts
