🗺️ 整合架構總覽

GitLab 與 Google Cloud 有官方深度整合,主要包含:

🔐 Workload Identity Federation

免 service account key,用 OIDC 身分驗證

📦 Artifact Registry 整合

在 GitLab 直接看 GCP container registry

🚀 Cloud Run / GKE Components

Google 維護的官方 CI/CD 元件

🏃 GCP 上的 Runner

在 Compute Engine 跑 autoscaling runner
💡 核心亮點:整套整合的安全基石是 WLIF(Workload Identity Federation),完全不用 service account key 也能讓 CI/CD 拿到 GCP 資源。

Quick Start整合啟用路徑

路徑:Project → Settings → Integrations → Google Cloud IAM → Configure → 選 Guided setup(GitLab UI 會給你貼到 Google Cloud Shell 跑的指令)。

🔐 1. Workload Identity Federation 設定

Quick StartWLIF 解決什麼?

傳統做法把 GCP service account JSON key 存 CI Variable,問題:

  • 長期憑證,洩漏難擋
  • JSON 格式無法 mask(一定有空格、換行)
  • 沒有細粒度時間限制

WLIF 用 GitLab 簽發的短期 OIDC JWT 換 GCP 短期 token,完全不用 service account key。

Quick StartGuided Setup(最簡單)

  1. Project → Settings → Integrations → Google Cloud IAM
  2. 選 Configure → Guided setup
  3. 填入 GCP Project ID 和 Project Number
  4. 複製 GitLab 給的 shell script
  5. 在 Google Cloud Shell 貼上執行
  6. script 會自動建立:Workload Identity Pool + Provider + IAM 綁定

SRE 實務手動 CLI 設定

# 1. 建 Workload Identity Pool
gcloud iam workload-identity-pools create gitlab-pool \
  --location=global \
  --project=$GCP_PROJECT_ID

# 2. 建 Provider(指向 GitLab OIDC discovery)
gcloud iam workload-identity-pools providers create-oidc gitlab-provider \
  --location=global \
  --workload-identity-pool=gitlab-pool \
  --issuer-uri=https://gitlab.com \
  --attribute-mapping="google.subject=assertion.sub,attribute.project_id=assertion.project_id,attribute.namespace_id=assertion.namespace_id" \
  --attribute-condition="assertion.namespace_id=='12345'" \
  --project=$GCP_PROJECT_ID

# 3. 授權給該 GitLab 專案的成員
gcloud projects add-iam-policy-binding $GCP_PROJECT_ID \
  --member="principalSet://iam.googleapis.com/projects/$GCP_PROJECT_NUMBER/locations/global/workloadIdentityPools/gitlab-pool/attribute.project_id/12345" \
  --role="roles/artifactregistry.writer"

架構視角JWT claims 限制

用 --attribute-condition 限縮可信來源:

  • assertion.project_id:GitLab 專案 ID(限定特定專案)
  • assertion.namespace_id:Group ID(限定某 group)
  • assertion.ref_protected:必須 protected branch
  • assertion.user_email:特定使用者
⚠️ 務必加 attribute condition!不限制的話任何 GitLab 專案都能 assume 這個 role(史上 CircleCI 事故同樣風險)。

📦 2. Google Artifact Registry 整合

Quick Start啟用步驟

  1. WLIF 已設定(上一節)
  2. GCP 端建 Docker format / Standard mode 的 Artifact Registry repo
  3. Project → Settings → Integrations → Google Artifact Management
  4. 填 Project ID、Repository name、Location、Pool/Provider ID
  5. Save

Quick StartUI 上看 GCP 容器

啟用後,左側欄會多一個 Deploy → Google Artifact Registry,直接看到 GCP 上的 image。

SRE 實務用 identity keyword 認證

push-to-gar:
  image: google/cloud-sdk:slim
  identity: google_cloud   # ⭐ 自動配置 GOOGLE_APPLICATION_CREDENTIALS
  script:
    # gcloud / crane 自動偵測 credential
    - gcloud auth configure-docker $LOCATION-docker.pkg.dev
    - docker tag myapp:$CI_COMMIT_SHA \
        $LOCATION-docker.pkg.dev/$PROJECT_ID/$REPO/myapp:$CI_COMMIT_SHA
    - docker push $LOCATION-docker.pkg.dev/$PROJECT_ID/$REPO/myapp:$CI_COMMIT_SHA

SRE 實務啟用後可用的環境變數

變數意義
GOOGLE_APPLICATION_CREDENTIALScredential 檔案路徑
CLOUDSDK_AUTH_CREDENTIAL_FILE_OVERRIDE同上(gcloud 用)
GOOGLE_ARTIFACT_REGISTRY_PROJECT_IDGCP project ID
GOOGLE_ARTIFACT_REGISTRY_REPOSITORY_NAMErepo 名
GOOGLE_ARTIFACT_REGISTRY_REPOSITORY_LOCATION區域

架構視角IAM 角色對映

整合 script 自動建立:

  • GitLab Reporter 以上 → Artifact Registry Reader(可 pull)
  • GitLab Developer 以上 → Artifact Registry Writer(可 push)

🧩 3. Google Cloud Components

Quick StartGoogle 維護的官方元件

位於 gitlab.com/google-gitlab-components,Google Cloud 團隊維護:

Component用途
artifact-registry/upload-artifact-registry把 image 推到 Artifact Registry
cloud-run/deploy-cloud-run部署到 Cloud Run
cloud-deploy/create-cloud-deploy-release建 Cloud Deploy release
kubernetes-engine/deploy-gke部署到 GKE
✅ 優於用 gcloud CLI:官方標榜這些元件比直接跑 gcloud 更小、更快(預先優化過的 image)。

SRE 實務典型 build → push → deploy

stages: [build, upload, deploy]

variables:
  GITLAB_IMAGE: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
  AR_IMAGE: $LOCATION-docker.pkg.dev/$PROJECT_ID/$REPO/myapp:$CI_COMMIT_SHORT_SHA

build:
  stage: build
  image: docker:24-cli
  services: [docker:24-dind]
  script:
    - echo "$CI_REGISTRY_PASSWORD" | docker login -u "$CI_REGISTRY_USER" --password-stdin "$CI_REGISTRY"
    - docker build -t $GITLAB_IMAGE .
    - docker push $GITLAB_IMAGE

include:
  # 上傳到 Artifact Registry
  - component: gitlab.com/google-gitlab-components/artifact-registry/upload-artifact-registry@main
    inputs:
      stage: upload
      source: $GITLAB_IMAGE
      target: $AR_IMAGE

  # 部署到 Cloud Run
  - component: gitlab.com/google-gitlab-components/cloud-run/deploy-cloud-run@main
    inputs:
      stage: deploy
      project_id: my-gcp-project
      service: my-service
      region: us-central1
      image: $AR_IMAGE

🚀 4. Cloud Run 部署

Quick Start必要 IAM 角色

給 Workload Identity Pool 對應 principal:

  • roles/run.admin — 可建立/更新 service
  • roles/iam.serviceAccountUser — 可以以 service account 身分執行
  • roles/artifactregistry.reader — Cloud Run 讀取 image

SRE 實務授權範例

WORKLOAD_IDENTITY=principalSet://iam.googleapis.com/projects/$PROJECT_NUMBER/locations/global/workloadIdentityPools/$POOL_ID/attribute.developer_access/true

gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="$WORKLOAD_IDENTITY" \
  --role='roles/run.admin'

gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="$WORKLOAD_IDENTITY" \
  --role='roles/iam.serviceAccountUser'

架構視角完整部署 pipeline 範例

stages: [build, push, deploy]

variables:
  IMAGE_TAG: $CI_COMMIT_SHORT_SHA
  AR_IMAGE: us-central1-docker.pkg.dev/$PROJECT_ID/my-repo/python-service

build:
  stage: build
  image: docker:24
  services: [docker:24-dind]
  script:
    - docker build -t $CI_REGISTRY_IMAGE:$IMAGE_TAG .
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker push $CI_REGISTRY_IMAGE:$IMAGE_TAG

include:
  - component: gitlab.com/google-gitlab-components/artifact-registry/upload-artifact-registry@0.1.0
    inputs:
      stage: push
      source: $CI_REGISTRY_IMAGE:$IMAGE_TAG
      target: $AR_IMAGE:$IMAGE_TAG

  - component: gitlab.com/google-gitlab-components/cloud-run/deploy-cloud-run@0.1.0
    inputs:
      stage: deploy
      project_id: $PROJECT_ID
      region: us-central1
      service: python-service
      image: $AR_IMAGE:$IMAGE_TAG

🏃 5. GCP 上的 Runner

Quick StartGitLab on Google Cloud Runner

GitLab 提供官方整合,在 GCP Compute Engine 跑 autoscaling runner fleet:

  • 一個 runner manager + 多個臨時 runner
  • 並發跑多個 job
  • 有需要才開機,沒 job 就關
  • Terraform 模板自動產生

Quick Start建立步驟

  1. Settings → CI/CD → Runners → New project runner
  2. 選 Google Cloud 環境
  3. 填規格(machine type、區域、最大並發)
  4. 下載 Terraform 設定
  5. 在 Cloud Shell 跑 terraform apply
  6. 等 runner 起來,進入待命狀態

架構視角適用場景

  • 需要 Compute Engine 大規格(GPU、高記憶體)
  • 大量平行 job(autoscale 省成本)
  • 合規要求 runner 在自家 GCP 環境
  • GCP-only 架構,網路成本考量

⚡ 速查表

🎯 整合啟用順序

  1. 建 GCP Project + 啟用必要 API(artifactregistry, run, compute)
  2. GitLab UI 啟用 Google Cloud IAM(WLIF)
  3. 啟用 Google Artifact Management
  4. (可選)在 GCP 跑 GitLab Runner
  5. 把 Google Cloud Components 加進 .gitlab-ci.yml

🔐 三種 GCP 認證方式對比

方式安全性難度
Service Account JSON Key 當 CI Variable❌ 差易
手寫 OIDC + WLIF✅ 高難
GitLab Google Cloud Integration✅ 高易(UI 嚮導)

📦 完整 push + deploy 模板

include:
  - component: gitlab.com/google-gitlab-components/artifact-registry/upload-artifact-registry@main
    inputs:
      stage: upload
      source: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
      target: $LOCATION-docker.pkg.dev/$PROJECT_ID/$REPO/$APP:$CI_COMMIT_SHORT_SHA

  - component: gitlab.com/google-gitlab-components/cloud-run/deploy-cloud-run@main
    inputs:
      stage: deploy
      project_id: $PROJECT_ID
      region: $LOCATION
      service: $APP
      image: $LOCATION-docker.pkg.dev/$PROJECT_ID/$REPO/$APP:$CI_COMMIT_SHORT_SHA

✅ Checklist

  • WLIF Pool + Provider 已建
  • Attribute condition 限制 namespace_id 或 project_id
  • 不再用 service account JSON key
  • Artifact Registry 整合啟用
  • 用官方 Components(不自己寫 gcloud)
  • Production 部署用 protected branch + protected variable
  • IAM 最小權限
