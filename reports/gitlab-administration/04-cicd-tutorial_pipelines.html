🗺️ 七種 Pipeline 類型

類型何時觸發特徵
Branchpush commit 到分支預設,最基本
Merge Request開 / 更新 MR在 source branch 上跑
Merged ResultsMR 上(Premium+)模擬合併後的結果跑

💡 一條 push 可能產生多個 pipeline 類型(branch + MR)。務必用 workflow:rules 避免重複。

🌿 1. Branch Pipeline(預設)

Quick Start何時觸發

只要 push 到任何分支(含 default branch)就會觸發。

# 最簡單的 branch pipeline
test:
  script: npm test
# 不加任何 rules → 任何 push 都跑

SRE 實務限制只在特定分支跑

test:
  script: npm test
  rules:
    # 只在 main 和 feature 分支跑
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
    - if: $CI_COMMIT_BRANCH =~ /^feature\//

架構視角Branch Pipeline 限制

  • 不能用 MR 相關變數:$CI_MERGE_REQUEST_* 為空
  • 只看單一 ref:不知道要合進哪
  • 不能用 Code Quality / Security 在 MR widget 顯示:需要 MR pipeline

因此大多數現代 GitLab 專案會改用 MR Pipeline,見下節。

🔀 2. Merge Request Pipeline

Quick Start為什麼要用?

  • MR widget 可以顯示測試結果、覆蓋率、安全掃描
  • 可以用 $CI_MERGE_REQUEST_* 變數
  • MR 上的「Pipeline must succeed」才能正確運作
  • 是 Merged Results 和 Merge Train 的前提

Quick Start最簡設定

workflow:
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
    - if: $CI_COMMIT_TAG

test:
  script: npm test
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"

SRE 實務避免重複 pipeline

⚠️ 最常見問題:push feature 分支時同時產生 branch pipeline + MR pipeline。

解法:workflow:rules 精確控制

workflow:
  rules:
    # MR 時用 MR pipeline
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
    # 有 MR 的 branch push 不跑 branch pipeline
    - if: $CI_COMMIT_BRANCH && $CI_OPEN_MERGE_REQUESTS
      when: never
    # 其他 branch push 才跑 branch pipeline
    - if: $CI_COMMIT_BRANCH

SRE 實務常用 MR 變數

變數意義
$CI_MERGE_REQUEST_ID全站 MR ID
$CI_MERGE_REQUEST_IID專案內 MR 序號(UI 顯示的 !123)
$CI_MERGE_REQUEST_TITLEMR 標題
$CI_MERGE_REQUEST_LABELS標籤(逗號分隔)
$CI_MERGE_REQUEST_SOURCE_BRANCH_NAME來源分支
$CI_MERGE_REQUEST_TARGET_BRANCH_NAME目標分支
$CI_MERGE_REQUEST_APPROVED是否已批准
$CI_MERGE_REQUEST_DRAFT是否為草稿

架構視角Fork MR 的安全考量

外部貢獻者的 fork MR,預設在 fork 專案的 runner + 設定跑,不會拿到 parent project 的 secret。

但若 parent project 成員手動點 "Run pipeline",pipeline 會用 parent project 的 runner 與 variable 跑,此時:

  • ⚠️ Fork 的惡意程式可能偷 parent project 的 secret
  • GitLab 會跳警告,必須勾確認才能跑
  • 建議先 review 程式變更再觸發

🔗 3. Merged Results Pipeline(Premium+)

Quick Start解決什麼問題?

一般 MR pipeline 是在 source branch 跑。但 target branch 可能早就更新了,合併後可能產生衝突或失敗。

Merged Results Pipeline 會先暫時合併 source + target,在「合併後的結果」上跑 pipeline,更精確反映「合併後」的狀態。

SRE 實務啟用方式

路徑:Project → Settings → Merge requests

  • ✅ 勾選 Enable merged results pipelines
  • 可設定「Pipelines must succeed before merging」

架構視角何時啟用?

  • main 分支變動頻繁的大型專案
  • 要求 MR merge 前必須通過 CI 的嚴格專案
  • 搭配 Merge Train 使用(下節)
⚠️ 會消耗較多 CI 資源(多一次 merge simulation)。

🚂 4. Merge Train(Premium+)

Quick Start解決什麼問題?

多個 MR 同時準備合併時:

  • MR A 測試通過 → 合併
  • MR B 測試通過(基於舊 main)→ 合併 → 意外壞掉(A 和 B 互相衝突)

Merge Train 讓 MR 排隊依序合併,每個 MR 都以「前面的 MR 已合併」為前提跑 pipeline。

SRE 實務運作機制

Train: [MR1] → [MR2] → [MR3]

MR1 pipeline: 跑於 main + MR1
MR2 pipeline: 跑於 main + MR1 + MR2
MR3 pipeline: 跑於 main + MR1 + MR2 + MR3

→ 依序合併,每個都保證「合進來不會壞」

架構視角Merge Train 適用

適合不適合
main 合併頻繁的 monorepo小專案、合併量少
團隊嚴格禁止 main 壞掉pipeline 本身不穩定(常 flaky)
CI 能在 < 30 分鐘內完成CI 很慢(長排隊)
💡 Merge Train 需先啟用 Merged Results Pipeline。

👨‍👧 5. Parent-Child Pipeline

Quick Start用途

同一專案內把 pipeline 拆成多個子 pipeline,適用於:

  • Monorepo(一個 repo 多個子服務)
  • 動態生成 pipeline(依程式碼內容決定要跑什麼)
  • 降低單一 pipeline 的複雜度

Quick Start靜態 child pipeline

# .gitlab-ci.yml(parent)
stages: [triggers]

trigger-frontend:
  stage: triggers
  trigger:
    include: frontend/.gitlab-ci.yml
    strategy: depend   # 等 child 完成並依結果決定 parent 狀態

trigger-backend:
  stage: triggers
  trigger:
    include: backend/.gitlab-ci.yml
    strategy: depend

SRE 實務動態 child pipeline(進階)

# parent: 產生 child yml
generate-config:
  stage: build
  script:
    - ./scripts/generate-ci.sh > generated.yml
  artifacts:
    paths: [generated.yml]

trigger-dynamic:
  stage: test
  needs: [generate-config]
  trigger:
    include:
      - artifact: generated.yml
        job: generate-config
    strategy: depend

動態 pipeline非常適合 monorepo「只跑變動部分」的場景。

SRE 實務Monorepo:只跑變動服務

# parent
include: services.yml

services.yml:
---
trigger-service-a:
  trigger:
    include: services/a/.gitlab-ci.yml
    strategy: depend
  rules:
    - changes: ["services/a/**/*"]

trigger-service-b:
  trigger:
    include: services/b/.gitlab-ci.yml
    strategy: depend
  rules:
    - changes: ["services/b/**/*"]

架構視角Parent-Child 限制

  • 最多 3 層(parent → child → grandchild)
  • Child pipeline 共用 parent 的 project、ref、SHA
  • Child 內部 $CI_PIPELINE_SOURCE 永遠是 parent_pipeline
  • 在 MR pipeline 中觸發 child 時,要用 $CI_MERGE_REQUEST_ID 判斷而非 $CI_PIPELINE_SOURCE

🌐 6. Multi-project Pipeline

Quick Start用途

跨 不同專案 串連 pipeline,適用微服務架構。

Quick Start上游觸發下游

# 專案 A 觸發專案 B
trigger-deploy:
  stage: deploy
  trigger:
    project: my-group/deployment
    branch: main
    strategy: depend    # 等下游完成

SRE 實務傳遞變數

trigger-deploy:
  stage: deploy
  variables:
    APP_VERSION: $CI_COMMIT_SHORT_SHA
    DEPLOY_ENV: production
  trigger:
    project: my-group/deployment
    branch: main
    strategy: depend
  # forward 傳遞
  inherit:
    variables: false   # 不自動繼承 parent 的所有變數

SRE 實務下游接收 artifact

下游專案可以拉上游的 artifact:

# 下游專案的 .gitlab-ci.yml
download-upstream:
  needs:
    - project: my-group/upstream
      job: build
      ref: main
      artifacts: true
  script:
    - ls upstream-artifacts/

架構視角Parent-Child vs Multi-project 比較

Parent-ChildMulti-project
同專案✅❌
跨專案❌✅
共用 SHA / ref✅❌(各自)
適用Monorepo、動態微服務、多 repo
UI 顯示嵌套在 parent獨立 pipeline

⏰ 7. Scheduled Pipeline

Quick Start設定排程

路徑:Project → Build → Pipeline schedules → New schedule

  • Description:排程用途描述
  • Interval pattern:Cron 格式(例:0 2 * * * 每天 02:00)
  • Cron timezone
  • Target branch / tag
  • Variables:排程專用變數
  • Active:啟用

SRE 實務常見用途

workflow:
  rules:
    - if: $CI_PIPELINE_SOURCE == "schedule"

nightly-regression:
  script: npm run test:full
  rules:
    - if: $CI_PIPELINE_SOURCE == "schedule" && $SCHEDULE_TYPE == "nightly"

weekly-security-scan:
  script: ./security-scan.sh
  rules:
    - if: $CI_PIPELINE_SOURCE == "schedule" && $SCHEDULE_TYPE == "security"

daily-db-backup:
  script: ./backup.sh
  rules:
    - if: $CI_PIPELINE_SOURCE == "schedule" && $SCHEDULE_TYPE == "backup"

透過 $SCHEDULE_TYPE 變數(在 Schedule 設定)區分不同排程用途。

架構視角排程最佳實踐

  • 時區明確:跨國團隊避免誤解
  • 錯開尖峰:多個排程不要同一時刻
  • Owner:schedule 繫結在一個使用者(該使用者離職會失效,建議用 Bot/Service Account)
  • 失敗通知:確認排程 pipeline 失敗會通知到正確的人
  • 變數隔離:排程專用的 prod key 建議設為 scheduled pipeline 專用

🎛️ 8. Manual / Trigger API Pipeline

Quick Start手動觸發

路徑:Build → Pipelines → Run pipeline(右上)

  • 選擇 ref(branch / tag)
  • 輸入變數(若 YAML 有 description 會顯示輸入框)
  • 按 Run pipeline

此時 $CI_PIPELINE_SOURCE = "web"。

SRE 實務Trigger Token(API 觸發)

適合外部系統(CI 系統、webhook、cron)觸發 GitLab pipeline。

# 1. 在 Project → Settings → CI/CD → Pipeline triggers 建立 token

# 2. 使用 curl 觸發
curl -X POST \
  --fail \
  -F token=$CI_TRIGGER_TOKEN \
  -F ref=main \
  -F "variables[DEPLOY_ENV]=production" \
  -F "variables[VERSION]=1.2.3" \
  https://gitlab.example.com/api/v4/projects/42/trigger/pipeline

觸發的 pipeline $CI_PIPELINE_SOURCE = "trigger"。

⚡ 9. 效能最佳化

Quick Start加速原則

  1. 用 DAG(needs):不必等整個 stage
  2. 用 cache:dependency 重複下載最浪費時間
  3. 用 changes rule:沒改的部分不跑
  4. Shallow clone:GIT_DEPTH: 10
  5. Interruptible:新 commit 時取消舊 job
  6. Parallel 測試:用 parallel: 4 切分

SRE 實務Auto-Cancel 設定

workflow:
  auto_cancel:
    on_new_commit: interruptible    # 新 commit 取消可中斷 job
    on_job_failure: all              # 任一 job 失敗取消全部

default:
  interruptible: true    # 預設 job 可被取消

deploy-prod:
  interruptible: false   # 部署不可中斷

SRE 實務Parallel 測試切分

# 把測試平均分 4 份
unit-test:
  parallel: 4
  script:
    - bundle exec rspec --knapsack

# Matrix 組合
e2e-test:
  parallel:
    matrix:
      - BROWSER: [chrome, firefox]
        OS: [linux, macos]
  # 展開為 4 個 job
  script: npm run e2e

架構視角診斷 Pipeline 慢

路徑:Analytics → CI/CD Analytics

  • 總 duration 趨勢圖
  • 成功率趨勢
  • 各 job 的平均執行時間(找出瓶頸)
  • DORA metrics(Premium+):deployment frequency、lead time 等

診斷流程:

  1. 找出平均最久的 job
  2. 檢查是否能拆分 / 並行
  3. 檢查 cache 是否命中
  4. 檢查是否可以改用 DAG
  5. 檢查 runner 規格是否夠用

🎨 防重複 pipeline 的 workflow

workflow:
  rules:
    # MR 時走 MR pipeline
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
    # 有 MR 的 branch push 不跑
    - if: $CI_COMMIT_BRANCH && $CI_OPEN_MERGE_REQUESTS
      when: never
    # Default branch 和 tag 一定跑
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
    - if: $CI_COMMIT_TAG
    # 其他 branch push
    - if: $CI_COMMIT_BRANCH

🔗 下游 pipeline 設定

# Child pipeline(同專案)
trigger-child:
  trigger:
    include: sub/.gitlab-ci.yml
    strategy: depend

# Multi-project(跨專案)
trigger-downstream:
  variables:
    UPSTREAM_VERSION: $CI_COMMIT_SHA
  trigger:
    project: my-group/other-project
    branch: main
    strategy: depend

📊 Pipeline 效率 Checklist

  • 有用 needs(DAG)
  • dependency 有 cache
  • shallow clone(GIT_DEPTH)
  • 重複觸發可自動取消
  • 不需要的測試用 changes rule
  • 長時間測試用 parallel
  • Analytics 有監控
  • 沒有重複 pipeline

🆘 常見問題

問題解法
重複 pipeline設 workflow:rules
Child pipeline 沒觸發檢查 parent job 有跑 + trigger include 正確
Multi-project downstream 沒結果加 strategy: depend
Schedule 沒跑檢查 schedule 狀態、owner 是否有權、cron 表達式
Merge Train 卡住通常是 pipeline 不穩定,先修 flaky test
