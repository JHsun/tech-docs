🗺️ 本章定位

💡 與 Administration 章節的差異:Administration 第 10 章詳細講「如何安裝、註冊、設定 Runner」(管理員視角);本章從 使用者 / 開發者視角 出發,講「如何在 .gitlab-ci.yml 裡正確選用 Runner、troubleshoot job 為什麼沒跑起來」。

作為 CI 使用者,你需要知道的 Runner 事情不多:

  • 有哪些 runner 可以用(scope)
  • 怎麼指定我這個 job 要用哪個
  • 哪些 runner 是「受保護」的(跟 protected branch 有關)
  • job 卡住不跑時,是不是 runner 的問題

🎯 1. 三種 Runner Scope

類型可服務範圍UI 標籤誰能管
Instance Runner全站所有專案InstanceAdmin
Group Runner該 group + 所有子 group 的專案GroupGroup Owner
Project Runner單一專案(可鎖定不分享)ProjectProject Maintainer

Quick Start查看你專案有哪些 Runner

路徑:Project → Settings → CI/CD → Runners(需 Maintainer 以上)

會看到三個區塊:

  1. Instance runners — 站上共用的
  2. Group runners — 你所屬 group 分享的
  3. Project runners — 你專案專屬的

綠色圓圈 = online;灰色 = offline / stale。

SRE 實務啟用 / 停用

  • Instance runner:可在專案層停用(若不想用共享池)
  • Group runner:子專案可拒絕繼承
  • Project runner:可「lock」讓此專案獨佔,其他專案拿不到

架構視角選擇建議

場景建議
輕量 CI、不敏感Instance runner
團隊有自家 runner 池Group runner
需特殊硬體(GPU、ARM、MacOS)Project runner + lock
生產部署 jobProject runner + protected
合規需求(資料不外流)Self-hosted project runner,禁用 instance

🎯 2. 如何選 Runner

Quick Start三種匹配方式

GitLab 把 job 指派給 runner 的邏輯依序判斷:

  1. Scope 對不對:這個 runner 能不能服務這個 project?
  2. Tag 對不對:job 的 tags 與 runner 的 tags 是否匹配?
  3. Protected 對不對:若 job 來自 protected branch/tag,runner 需是 protected
  4. 狀態對不對:runner 必須 online + not busy

Quick Start最簡單的指定

job-a:
  tags:
    - docker       # 要有 docker tag 的 runner
    - linux
  script: make

Runner 必須同時有這兩個 tag 才會接這個 job(AND 關係,不是 OR)。

⚠️ 找不到匹配 runner 時,job 會永遠 pending。60 分鐘後(預設)可能被 cancel,或持續卡住直到 timeout。

🏷️ 3. Tags 機制

Quick Start命名慣例

Runner tags 沒有強制格式,但建議用一致的命名:

維度範例
執行方式docker、shell、kubernetes
作業系統linux、windows、macos
架構amd64、arm64
規格small、medium、large、gpu
用途build、test、deploy、security-scan
環境prod、staging、dev

SRE 實務Untagged Jobs

若 job 沒寫 tags,只有勾選「Run untagged jobs」的 runner 才會接。

# 建議:實務上給 job 打 tag,避免被意外的 runner 接走
# ❌ 避免
test:
  script: npm test

# ✅ 推薦
test:
  tags: [docker, small]
  script: npm test

SRE 實務複合策略範例

stages: [build, test, deploy]

build:
  stage: build
  tags: [docker, linux, amd64, build-large]
  script: make build

unit-test:
  stage: test
  tags: [docker, linux, small]
  script: npm test

e2e-test:
  stage: test
  tags: [docker, linux, large]   # 需要更大規格
  script: npm run e2e

security-scan:
  stage: test
  tags: [security-scan]          # 專用 runner,有掃描工具
  script: ./scan.sh

deploy-prod:
  stage: deploy
  tags: [deploy, prod]           # 只有 prod 專用 runner 接
  script: ./deploy.sh
  environment: production
  rules:
    - if: $CI_COMMIT_BRANCH == "main"

架構視角Tag 設計原則

  • 正交:每個 tag 代表一個獨立維度,不要混合
  • 文件化:在 project README 或 CI guide 列出可用 tags
  • 最小必要:不要疊太多 tag,否則匹配變難
  • 區分共用與專用:prod 部署用專屬 tag,避免被共享 runner 接走

🔒 4. Protected Runner

Quick Start什麼是 Protected Runner?

勾選「Protected」的 runner 只接 protected branch / protected tag 的 job,不接一般 feature branch 的 job。

這是 GitLab 防止機敏資料外洩的重要機制:

  • Protected branch 上可以有 protected CI variable(如 AWS production key)
  • 你不希望 feature branch 的 job 拿到這些 key
  • 解法:prod key 只給 protected runner 用,protected runner 只跑 protected branch

SRE 實務完整保護鏈

完整 Production Deploy 保護鏈:
┌──────────────────────────────────────────┐
│ 1. Branch "main" 設為 Protected           │
│ 2. Variable AWS_PROD_KEY 設為 Protected   │  
│ 3. Runner "prod-deploy" 設為 Protected    │
│ 4. Job 指定 tags: [prod-deploy]           │
│ 5. Job 使用 $AWS_PROD_KEY                 │
└──────────────────────────────────────────┘

結果:只有 main branch 的 commit 才能拿到 key 並在
prod-deploy runner 上執行。feature branch 就算偷塞
同樣的 tag,因為 branch 不是 protected,runner 不會接。

架構視角Protected 的完整規則矩陣

Runner Job 來自 Protected branch Job 來自一般 branch
Protected ✅ 接 ❌ 不接
Unprotected ✅ 接 ✅ 接

也就是:Protected runner 比較嚴格,Unprotected runner 什麼都接。

🎛️ 5. Runner 相關變數與行為控制

Quick Start.gitlab-ci.yml 中可控制 Runner 行為的變數

job:
  variables:
    # Git 操作行為
    GIT_STRATEGY: fetch           # none / fetch(預設) / clone
    GIT_DEPTH: 50                 # shallow clone 深度(0=無限)
    GIT_SUBMODULE_STRATEGY: recursive  # none / normal / recursive
    GIT_CLEAN_FLAGS: "-ffdx"      # git clean 參數
    GIT_CHECKOUT: "true"          # 是否 checkout

    # 超時(runner 端二次控制)
    RUNNER_SCRIPT_TIMEOUT: 30m
    RUNNER_AFTER_SCRIPT_TIMEOUT: 5m

    # Clone 路徑(建議多 runner 並發時用)
    GIT_CLONE_PATH: $CI_BUILDS_DIR/$CI_CONCURRENT_ID/project-name

    # 除錯
    CI_DEBUG_TRACE: "true"        # 開詳細 log(注意會曝露變數!)
    CI_DEBUG_SERVICES: "true"     # 開 services 容器 log

  script:
    - echo "Runner 行為已自訂"

SRE 實務Git 策略比較

GIT_STRATEGY行為適用
clone每次 job 重新 cloneDocker executor、最乾淨
fetch(預設)重用舊 clone,只 fetch 新 commitShell executor,快但有殘留風險
none完全不取 repo純下載 artifact 的 job

SRE 實務並發衝突避免

同一台 runner 多 job 並發時,若 GIT_STRATEGY=fetch 且共用 builds_dir,可能互相干擾。解法:

variables:
  # 每個 job 獨立目錄
  GIT_CLONE_PATH: $CI_BUILDS_DIR/$CI_CONCURRENT_ID/$CI_PROJECT_NAME

架構視角Runner 資源變數

Runner 端提供的並發資訊變數:

變數意義
CI_CONCURRENT_ID該 runner 上所有 job 的唯一 ID
CI_CONCURRENT_PROJECT_ID該 runner 上該 project 的 job 唯一 ID
CI_BUILDS_DIRbuilds 根目錄
CI_RUNNER_IDrunner 的 ID
CI_RUNNER_DESCRIPTIONrunner 描述(在 UI 顯示的名稱)
CI_RUNNER_TAGSrunner 擁有的 tags(逗號分隔)
CI_RUNNER_EXECUTABLE_ARCHrunner 的架構(如 linux/amd64)

☁️ 6. GitLab-Hosted vs Self-Managed Runners

Quick Start兩大類別

GitLab-HostedSelf-Managed
誰提供GitLab你自己
維運GitLab 負責你負責
費用CI/CD minutes自己的 infra 成本
執行環境Docker,規格固定完全自訂
可用於 Dedicated❌ 不可✅ 可

SRE 實務GitLab.com hosted runner tags

GitLab.com 提供的 SaaS runner(2025 主要 tier):

Tag規格消耗 minutes 倍率
saas-linux-small-amd642 vCPU, 8GB1x
saas-linux-medium-amd644 vCPU, 16GB2x
saas-linux-large-amd648 vCPU, 32GB3x
saas-linux-medium-arm64ARM 4 vCPU, 16GB2x
saas-macos-medium-m1macOS M16x
saas-windows-medium-amd64Windows Server2x
💡 倍率、規格、預設額度都會變動,請以 官方文件 為準。

架構視角選擇考量

  • Self-Managed 划算點:CI 量大、特殊硬體(GPU)、合規要求資料不出組織
  • Hosted 划算點:量不大、不想維運、需要跨區域
  • 混合:一般 job 用 hosted、特殊 job 用 self-managed
  • Dedicated:完全不提供 hosted runner,必須自備(見 Administration 第 9 章)

🆘 7. 常見問題

Quick StartJob 一直 pending?

診斷順序:

  1. 看有沒有 runner:Project → Settings → CI/CD → Runners,至少一個綠圈
  2. 檢查 tag:job 的 tags 是否與 runner 的 tags 匹配?
  3. Untagged:若 job 沒 tag,runner 要勾 "Run untagged jobs"
  4. Protected:protected branch 的 job 需要 protected runner
  5. Runner 是否忙碌:並發數滿了?
  6. Runner concurrency limit:該 runner 同時能跑幾個?

SRE 實務錯誤訊息對照

訊息含意
"This job is stuck because the project doesn't have any runners online"專案沒有可用 runner
"This job does not have a trace"job 還沒開始執行(通常是卡住)
"Job failed (system failure)"runner 本身出問題(非 job script 錯)
"Job exceeded the maximum execution time"timeout。看 project CI timeout 或 runner maximum timeout
"runner '..' is not allowed to pick protected jobs"需要 protected runner

架構視角Runner 狀態分類

狀態意義
Online1.5 分鐘內有聯繫
Offline超過 1.5 分鐘沒聯繫
Stale3 個月沒聯繫(會自動清除)
Active啟用接 job
Paused手動暫停,不接新 job

🎯 決策樹:我該用哪種 Runner?

我的 job 是?
├─ 一般 build/test
│  └─ Instance runner(tag:docker, linux, small)
│
├─ 需要 prod 部署密鑰
│  └─ Protected project runner(tag:deploy, prod)
│
├─ 需要 GPU / 特殊硬體
│  └─ Locked project runner(tag:gpu)
│
└─ 跨團隊共用的工具類
   └─ Group runner(tag:security-scan)

✅ Runner 設計 Checklist

  • 所有 job 都有明確 tags
  • Production 部署用 protected runner
  • Sensitive variable 設為 Protected
  • 長時間 job 不跑在 instance runner
  • 特殊硬體 job 用 project-locked runner
  • pipeline 沒卡住 pending
