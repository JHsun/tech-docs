🗺️ Configure 總覽與三種部署方式

GitLab 的組態檔路徑,取決於你的部署方式:

部署方式組態入口套用指令
Linux package (Omnibus) /etc/gitlab/gitlab.rb gitlab-ctl reconfigure
Helm Chart (Kubernetes) values.yaml helm upgrade
Docker 環境變數 GITLAB_OMNIBUS_CONFIG 重啟容器
Self-compiled (source) config/gitlab.yml 重啟服務
💡 本章以最常見的 Linux package (Omnibus) 為主,這也是 GitLab 官方最推薦的部署方式。

📝 1. gitlab.rb 基本工作流程

Quick Start三步曲

# 1. 編輯組態檔
sudo vi /etc/gitlab/gitlab.rb

# 2. 套用變更(這一步一定要做)
sudo gitlab-ctl reconfigure

# 3. 驗證
sudo gitlab-ctl status
⚠️ 常見新手錯誤:只改 gitlab.rb 但沒跑 gitlab-ctl reconfigure,設定不會生效。
⚠️ 絕對不要編輯:/var/opt/gitlab/gitlab-rails/etc/gitlab.yml(自動產生,下次 reconfigure 會被覆蓋)

SRE 實務工作流程最佳實踐

# 改前先備份
sudo cp /etc/gitlab/gitlab.rb /etc/gitlab/gitlab.rb.bak.$(date +%F)

# 編輯
sudo vi /etc/gitlab/gitlab.rb

# 語法檢查(避免打錯字造成 reconfigure 失敗)
sudo gitlab-ctl check-config

# 套用
sudo gitlab-ctl reconfigure

# 若只需重啟特定服務
sudo gitlab-ctl restart puma
sudo gitlab-ctl restart sidekiq

範本檔位置(完整預設值參考):

/opt/gitlab/etc/gitlab.rb.template

架構視角設定的版本控制與 IaC

  • 把 gitlab.rb 納入 Git 管理(敏感值用 Vault 或 sops 加密)
  • 用 Ansible / Chef / Puppet 推送,避免手工改
  • 變更一定走 MR review,留下 who/why
  • 多實例要統一範本,差異只留在環境專屬變數
💡 GitLab 官方提供 gitlab.rb.template,超過 2000 行,所有選項都在裡面。建議 diff 自己的設定檔與 template 比對差異。

⭐ 2. 必改的基礎設定

Quick StartDay 1 一定要設的三個東西

# 1. External URL(Git clone 時的 URL)
external_url 'https://gitlab.example.com'

# 2. 時區
gitlab_rails['time_zone'] = 'Asia/Taipei'

# 3. Admin 初始密碼(僅首次安裝時有用)
gitlab_rails['initial_root_password'] = 'ChangeMe!SuperSecret'

SRE 實務更改 External URL 的注意事項

⚠️ 改 external_url 的副作用:
  • 所有 Git remote URL 會變(使用者要重設 remote)
  • Webhook URL 需重新確認
  • CI 產生的 artifact 下載 URL 會變
  • Container Registry 、Pages 可能連動受影響

安裝時若用環境變數帶 URL,注意:

# ✅ 推薦:限定只在安裝指令傳入
sudo EXTERNAL_URL="https://gitlab.example.com" apt-get install gitlab-ee

# ❌ 避免:設為 shell 全域變數
# 否則套件更新時會無聲覆蓋 /etc/gitlab/gitlab.rb

SRE 實務Allowed Hosts(防 Host Header Attack)

gitlab_rails['allowed_hosts'] = [
  'gitlab.example.com',
  '127.0.0.1',
  'localhost'
]

若前面有 Apache/Nginx 代理,可能需要加入 127.0.0.1,並在代理層加 Host header 過濾。

架構視角相對路徑 URL 不建議

GitLab 支援把自己裝在 https://example.com/gitlab 這種子路徑,但官方提醒:

  • ❌ Geo 複寫有已知問題
  • ❌ 測試覆蓋度較低
  • ✅ 強烈建議用子網域:gitlab.example.com

📧 3. 郵件發送(SMTP)

Quick Start沒設 SMTP 會怎樣?

預設使用本機 Postfix,在無 SMTP relay 的環境下,大部分 SaaS / 公司信箱都會擋下來。結果:

  • 密碼重設信收不到
  • 2FA 開通信收不到
  • Merge Request / Issue 通知全斷

SRE 實務典型 SMTP Relay 設定

# 關閉內建 Postfix
postfix['enable'] = false

# 使用外部 SMTP
gitlab_rails['smtp_enable'] = true
gitlab_rails['smtp_address'] = "smtp.sendgrid.net"
gitlab_rails['smtp_port'] = 587
gitlab_rails['smtp_user_name'] = "apikey"
gitlab_rails['smtp_password'] = "SG.xxxxxxx"
gitlab_rails['smtp_domain'] = "sendgrid.net"
gitlab_rails['smtp_authentication'] = "login"
gitlab_rails['smtp_enable_starttls_auto'] = true
gitlab_rails['smtp_tls'] = false
gitlab_rails['smtp_openssl_verify_mode'] = 'peer'

# 寄件人
gitlab_rails['gitlab_email_enabled'] = true
gitlab_rails['gitlab_email_from'] = 'gitlab@example.com'
gitlab_rails['gitlab_email_display_name'] = 'GitLab'
gitlab_rails['gitlab_email_reply_to'] = 'noreply@example.com'

測試發信(reconfigure 後):

sudo gitlab-rails console
# 在 console 中:
Notify.test_email('you@example.com', 'Test', 'Hello').deliver_now

架構視角企業常見選項

方案優注意
AWS SES便宜、高送達率需通過 production 申請;有 sandbox 限制
SendGrid / Mailgun快速啟用按量計費
Exchange / O365企業一致性驗證方式限制多(需 app password / OAuth)
自架 Postfix relay完全控制要自己顧 SPF/DKIM/DMARC、IP 信譽
💡 別忘設定 SPF / DKIM / DMARC,否則送到 Gmail / Outlook 很容易進垃圾郵件。

💽 4. 儲存設定:Object Storage / NFS

Quick Start哪些資料可以放 Object Storage?

GitLab 幾乎所有大檔案類資料都能放 S3 相容儲存:

  • Artifacts(CI/CD 產物)
  • LFS objects
  • Uploads(圖片、附件)
  • Container Registry images
  • Packages(npm / Maven / PyPI 等)
  • Backups
  • Terraform state
⚠️ 不能放 Object Storage:Git repository 本身(必須留在 Gitaly 管理的本地/NFS)

SRE 實務Consolidated Object Storage 設定(推薦)

以前每種 object 要單獨設,現在可以用 consolidated 一次設完:

gitlab_rails['object_store']['enabled'] = true
gitlab_rails['object_store']['proxy_download'] = true
gitlab_rails['object_store']['connection'] = {
  'provider' => 'AWS',
  'region' => 'ap-northeast-1',
  'aws_access_key_id' => 'AKIA...',
  'aws_secret_access_key' => '...'
}
gitlab_rails['object_store']['objects']['artifacts']['bucket']  = 'gitlab-artifacts'
gitlab_rails['object_store']['objects']['lfs']['bucket']        = 'gitlab-lfs'
gitlab_rails['object_store']['objects']['uploads']['bucket']    = 'gitlab-uploads'
gitlab_rails['object_store']['objects']['packages']['bucket']   = 'gitlab-packages'
gitlab_rails['object_store']['objects']['dependency_proxy']['bucket'] = 'gitlab-dep-proxy'
gitlab_rails['object_store']['objects']['terraform_state']['bucket'] = 'gitlab-tfstate'

遷移既有資料到 S3:

# 把現有 artifacts 搬到 object storage
sudo gitlab-rake gitlab:artifacts:migrate

# LFS
sudo gitlab-rake gitlab:lfs:migrate

# Uploads
sudo gitlab-rake gitlab:uploads:migrate:all

架構視角NFS 是否還能用?

⚠️ GitLab 官方已不建議使用 NFS 存 Git repository。替代方案:
  • Gitaly Cluster(HA 的 Git 儲存,支援多副本)
  • Praefect(Gitaly 的路由與複寫管理)

大型實例儲存架構:

┌────────────┐   ┌──────────────┐   ┌──────────────┐
│ Rails/Puma │──▶│ Gitaly Cluster│──▶│ Local SSD/NVMe│
└────────────┘   └──────────────┘   └──────────────┘
      │
      │  其他大檔案
      ▼
┌──────────────────────────────────┐
│ Object Storage (S3/GCS/Azure/MinIO)│
└──────────────────────────────────┘

🔑 5. 外部驗證:LDAP / SAML / OIDC

Quick Start選哪種?

場景推薦
公司有 AD / OpenLDAPLDAP
公司用 Okta / Azure AD / OneLoginSAML
要整合 Google / Microsoft / GitHub 帳號OmniAuth OIDC/OAuth
企業要集中開停帳號SAML + SCIM

SRE 實務LDAP 範例(最常見)

gitlab_rails['ldap_enabled'] = true
gitlab_rails['ldap_servers'] = {
  'main' => {
    'label' => 'Company LDAP',
    'host' =>  'ldap.example.com',
    'port' => 636,
    'uid' => 'sAMAccountName',
    'encryption' => 'simple_tls',
    'bind_dn' => 'CN=gitlab,OU=ServiceAccounts,DC=example,DC=com',
    'password' => 'SecretBindPwd',
    'active_directory' => true,
    'base' => 'OU=Users,DC=example,DC=com',
    'user_filter' => '(memberOf=CN=GitLabUsers,OU=Groups,DC=example,DC=com)',
    'verify_certificates' => true
  }
}

驗證 LDAP 連線:

sudo gitlab-rake gitlab:ldap:check

SRE 實務SAML 範例(Okta/Azure AD)

gitlab_rails['omniauth_enabled'] = true
gitlab_rails['omniauth_allow_single_sign_on'] = ['saml']
gitlab_rails['omniauth_block_auto_created_users'] = false
gitlab_rails['omniauth_auto_link_saml_user'] = true
gitlab_rails['omniauth_providers'] = [
  {
    name: 'saml',
    label: 'Corporate SSO',
    args: {
      assertion_consumer_service_url: 'https://gitlab.example.com/users/auth/saml/callback',
      idp_cert_fingerprint: 'AA:BB:CC:...',
      idp_sso_target_url: 'https://idp.example.com/sso/saml',
      issuer: 'https://gitlab.example.com',
      name_identifier_format: 'urn:oasis:names:tc:SAML:1.1:nameid-format:emailAddress'
    }
  }
]

架構視角身分整合設計原則

  • 不要混合多種來源:同一帳號既能 LDAP 又能本地登入,治理會很亂
  • Just-In-Time Provisioning:首次登入自動建帳號(省管理)
  • SCIM 做 deprovisioning:離職自動停權
  • 保留至少 1 個本地 admin:IdP 掛點時的救援通道
  • Group Sync:用 LDAP/SAML group → GitLab group 對應,一次管權限

⚙️ 6. 核心服務調校

Quick StartGitLab 主要服務

服務角色吃什麼資源
PumaWeb/API 請求處理CPU + RAM
Sidekiq背景工作(寄信、CI trigger、通知)CPU + RAM
GitalyGit repo 操作CPU + Disk IO
PostgreSQL主資料庫RAM + Disk IO
Rediscache / queue / sessionRAM
Workhorse大檔上傳/下載代理CPU + 網路
NGINX反向代理網路

SRE 實務Puma 與 Sidekiq 調整

# Puma(web worker)
puma['worker_processes'] = 4          # 通常設為 CPU 核心數
puma['min_threads'] = 4
puma['max_threads'] = 4
puma['worker_memory_limit_mb'] = 1024  # 超過就 restart,防記憶體洩漏

# Sidekiq(背景工作)
sidekiq['max_concurrency'] = 20       # 併發數
sidekiq['min_concurrency'] = 5
sidekiq['queue_groups'] = ['*']        # 所有 queue 都處理

SRE 實務PostgreSQL 參數

# 預設 PostgreSQL 佔總 RAM 的 1/4,若資料庫在獨立機器可調高
postgresql['shared_buffers'] = "4GB"
postgresql['max_connections'] = 500
postgresql['work_mem'] = "16MB"
postgresql['maintenance_work_mem'] = "256MB"

架構視角從單機到分散式的演進路線

  1. 單機 all-in-one(< 1000 users):一台機器跑所有服務
  2. 拆資料庫與 Redis(1000-3000):Rails 仍共機
  3. 拆 Gitaly(3000-5000):獨立 Git 儲存節點
  4. 拆 Sidekiq(5000-10000):獨立背景工作節點
  5. HA 架構(10000+):Puma × N、Gitaly Cluster、Patroni HA、Redis Sentinel
  6. Cells/Geo(超大/多區域):多站點複寫
💡 GitLab 提供「Reference Architectures」文件,從 1k 到 50k users 各有建議規格,別自己亂拆,照官方指引最安全。

✅ Day-1 基礎設定 Checklist

  • 設 external_url 並用 HTTPS
  • 設 time_zone(預設 UTC)
  • 設 SMTP 並測試寄信
  • 設 allowed_hosts
  • 規劃 Object Storage(bucket 先建好)
  • 規劃外部驗證(LDAP/SAML)
  • 備份 gitlab-secrets.json 到安全位置
  • 設定 backup_keep_time
  • 把 gitlab.rb 納入版本控制
