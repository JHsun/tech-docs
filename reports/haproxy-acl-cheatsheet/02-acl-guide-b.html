R1–R3

存取控制食譜

1IP 白名單 + 黑名單（檔案載入）

# /etc/haproxy/whitelist.lst — 每行一個 IP 或 CIDR # 10.0.0.0/8 # 192.168.1.100 # /etc/haproxy/blacklist.lst # 203.0.113.50 frontend ft_web bind *:80 acl is_whitelisted src -f /etc/haproxy/whitelist.lst acl is_blacklisted src -f /etc/haproxy/blacklist.lst http-request deny deny_status 403 if is_blacklisted http-request allow if is_whitelisted # 僅管理頁面需白名單 acl is_admin path_beg /admin /management http-request deny deny_status 403 if is_admin !is_whitelisted default_backend bk_web
💡 Runtime 動態管理
echo "add acl /etc/haproxy/blacklist.lst 1.2.3.4" | socat stdio $SOCK — 即時加入黑名單。但 reload 後會消失，需同步更新檔案。

2HTTP Basic Auth 保護特定路徑

# /etc/haproxy/users.lst userlist admin_users user admin password $6$rounds=5000$salt$hashedpassword user viewer insecure-password viewerpass frontend ft_web bind *:443 ssl crt /etc/haproxy/certs/site.pem acl is_stats path_beg /stats /haproxy acl is_authed http_auth(admin_users) http-request auth realm "Admin Area" if is_stats !is_authed default_backend bk_web

3僅允許特定 HTTP Methods

frontend ft_api bind *:443 ssl crt /etc/haproxy/certs/api.pem # 使用 Pre-defined ACLs http-request deny deny_status 405 if METH_TRACE http-request deny deny_status 405 if METH_CONNECT # 自訂允許清單 acl valid_method method GET HEAD POST PUT PATCH DELETE OPTIONS http-request deny deny_status 405 unless valid_method default_backend bk_api
R4–R7

路由決策食譜

4Host-based 虛擬主機（map 動態路由）

# /etc/haproxy/hosts.map # api.example.com bk_api # www.example.com bk_web # admin.example.com bk_admin frontend ft_web bind *:443 ssl crt /etc/haproxy/certs/ # 動態路由：Host → Backend（免寫大量 ACL） use_backend %[req.hdr(Host),lower,map(/etc/haproxy/hosts.map,bk_default)] default_backend bk_default

5路徑 + 方法組合路由

frontend ft_web bind *:443 ssl crt /etc/haproxy/certs/site.pem # 靜態資源 → CDN 後端 use_backend bk_static if { path_end .css .js .png .jpg .gif .svg .woff2 .ico } # API 讀寫分離 use_backend bk_api_read if { path_beg /api } !{ method POST PUT PATCH DELETE } use_backend bk_api_write if { path_beg /api } { method POST PUT PATCH DELETE } # WebSocket use_backend bk_ws if { hdr(Upgrade) -i websocket } # GraphQL（POST /graphql） use_backend bk_graphql if { path /graphql } { method POST } default_backend bk_web

6API 版本路由

frontend ft_api bind *:443 ssl crt /etc/haproxy/certs/api.pem # 路徑版本 use_backend bk_api_v1 if { path_beg /v1/ } use_backend bk_api_v2 if { path_beg /v2/ } # Header 版本（Accept-Version: v2） use_backend bk_api_v2 if { req.hdr(Accept-Version) -m str v2 } use_backend bk_api_v1 if { req.hdr(Accept-Version) -m str v1 } # 預設最新版 default_backend bk_api_v2

7多協定同 Port（HTTP / SSH / SMTP 偵測）

frontend ft_multi bind *:443 mode tcp tcp-request inspect-delay 5s # SSL ClientHello 開頭為 0x16 (22) acl is_ssl req.payload(0,1) -m bin 16 # SSH 開頭為 "SSH" acl is_ssh req.payload(0,3) -m bin 535348 # SMTP 等待 server 先說話，所以 client 送空 tcp-request content accept if is_ssl tcp-request content accept if is_ssh use_backend bk_ssl if is_ssl use_backend bk_ssh if is_ssh default_backend bk_smtp
R8–R11

安全防護食譜

8User-Agent 黑名單 + Bot 偵測

# /etc/haproxy/bad-ua.lst — 惡意 User-Agent 子字串 # sqlmap # nikto # masscan # python-requests frontend ft_web bind *:80 acl is_bad_ua req.hdr(User-Agent) -m sub -i -f /etc/haproxy/bad-ua.lst acl no_ua req.hdr(User-Agent) -m len 0 acl short_ua req.hdr(User-Agent),length lt 10 acl is_good_bot req.hdr(User-Agent) -m sub -i Googlebot Bingbot http-request deny deny_status 403 if is_bad_ua http-request deny deny_status 403 if no_ua !is_good_bot http-request deny deny_status 403 if short_ua !is_good_bot default_backend bk_web

9阻擋敏感路徑（.git / .env / wp-admin）

frontend ft_web bind *:80 # 版控、設定檔、常見掃描目標 acl is_sensitive path_beg /.git /.svn /.env /.htaccess /.htpasswd acl is_sensitive path_beg /wp-admin /wp-login /phpmyadmin /adminer acl is_sensitive path_end .bak .old .swp .sql .log .conf http-request deny deny_status 404 if is_sensitive # 阻擋目錄遍歷 http-request deny deny_status 400 if { path -m sub ../ } http-request deny deny_status 400 if { path -m sub %2e%2e } default_backend bk_web

10Request Body 大小限制

frontend ft_api bind *:443 ssl crt /etc/haproxy/certs/api.pem # 一般 API：限 1MB acl body_too_large req.body_size gt 1048576 # 上傳端點：限 50MB acl is_upload path_beg /api/upload acl upload_too_large req.body_size gt 52428800 http-request deny deny_status 413 if body_too_large !is_upload http-request deny deny_status 413 if is_upload upload_too_large default_backend bk_api

11Referer / Origin 檢查（CSRF 防護）

frontend ft_web bind *:443 ssl crt /etc/haproxy/certs/site.pem acl is_mutating method POST PUT PATCH DELETE acl valid_origin req.hdr(Origin) -m dom example.com acl valid_referer req.hdr(Referer) -m dom example.com acl has_origin req.hdr(Origin) -m found # 修改操作必須有合法 Origin 或 Referer http-request deny deny_status 403 if is_mutating has_origin !valid_origin http-request deny deny_status 403 if is_mutating !has_origin !valid_referer default_backend bk_web
R12–R15

流量管理食譜

12維護模式（一鍵切換）

# 方法一：檔案觸發（touch /etc/haproxy/maintenance 啟用） frontend ft_web bind *:80 acl maintenance_mode nbsrv(bk_web) eq 0 acl maintenance_file path -f /etc/haproxy/maintenance # 檔案存在即啟用 # 或用環境變數 acl maintenance_env env(MAINTENANCE) -m str true http-request return status 503 \ content-type "text/html" \ file /etc/haproxy/pages/maintenance.html \ if maintenance_mode || maintenance_file default_backend bk_web # 方法二：Runtime API（不需 reload） # echo "disable server bk_web/web1" | socat stdio $SOCK # echo "disable server bk_web/web2" | socat stdio $SOCK # → nbsrv(bk_web) 自動降為 0 → 觸發維護頁

13A/B 測試（百分比分流）

frontend ft_web bind *:80 # 10% 流量到新版本 acl is_canary rand(100) lt 10 # 或用 cookie 固定分組（同用戶始終看到同版本） acl has_ab_cookie req.cook(ab_group) -m found acl is_group_b req.cook(ab_group) -m str B use_backend bk_v2 if is_group_b use_backend bk_v2 if !has_ab_cookie is_canary default_backend bk_v1 backend bk_v2 # 設定 cookie 記住分組 http-response set-header Set-Cookie "ab_group=B; Path=/; Max-Age=86400" unless { res.cook(ab_group) -m found }

14降級模式（後端不健康時回傳快取/靜態頁）

frontend ft_web bind *:80 acl backend_down nbsrv(bk_web) eq 0 acl api_down nbsrv(bk_api) lt 2 # 少於 2 台 = 降級 # 主後端掛了 → 靜態頁 use_backend bk_static_fallback if backend_down # API 降級 → 回傳固定回應 http-request return status 503 \ content-type "application/json" \ string '{"status":"degraded","message":"Service temporarily unavailable"}' \ if api_down { path_beg /api } default_backend bk_web

15依時間條件路由（工作時間 / 離峰）

frontend ft_web bind *:80 # 工作時間（週一到週五 09:00-18:00 UTC） acl is_weekday date,ltime(%u),int le 5 acl is_work_hr date,ltime(%H),int ge 9 acl is_work_hr date,ltime(%H),int lt 18 # 工作時間 → 正式環境；離峰 → 可做維護 use_backend bk_prod if is_weekday is_work_hr default_backend bk_prod # 或切到維護模式
R16–R20

進階場景食譜

16JWT Bearer Token 驗證 & 路由

frontend ft_api bind *:443 ssl crt /etc/haproxy/certs/api.pem # 擷取 Bearer token http-request set-var(txn.bearer) req.hdr(Authorization),field(2,' ') # JWT payload 提取角色 http-request set-var(txn.role) var(txn.bearer),b64dec,json_query('.role') # 驗證 JWT 簽章（需要公鑰） http-request deny deny_status 401 unless { var(txn.bearer) -m found } # http-request deny deny_status 401 unless { var(txn.bearer),jwt_verify(RS256,/etc/haproxy/jwt-pub.pem) -m int 1 } # 依角色路由 use_backend bk_admin if { var(txn.role) -m str admin } default_backend bk_user

17JSON Body 解析路由

frontend ft_api bind *:443 ssl crt /etc/haproxy/certs/api.pem option http-buffer-request # ⚠️ 必須！否則 req.body 為空 # 從 JSON body 提取 action 欄位 http-request set-var(txn.action) req.body,json_query('.action') use_backend bk_create if { var(txn.action) -m str create } use_backend bk_delete if { var(txn.action) -m str delete } default_backend bk_default
⚠️ option http-buffer-request 必須啟用
沒有它，req.body 永遠為空。且注意這會在轉發前等待完整 body，增加延遲。

18動態限速（依用戶等級差異化）

backend st_rates stick-table type ip size 200k expire 60s store http_req_rate(10s) frontend ft_api bind *:443 ssl crt /etc/haproxy/certs/api.pem http-request track-sc0 src table st_rates # 從 API Key 查表取得等級 http-request set-var(txn.tier) req.hdr(X-API-Key),map(/etc/haproxy/apikey-tier.map,free) # 依等級設不同閾值 acl is_free var(txn.tier) -m str free acl is_pro var(txn.tier) -m str pro acl rate_free sc_http_req_rate(0) gt 10 # free: 10/10s acl rate_pro sc_http_req_rate(0) gt 100 # pro: 100/10s http-request deny deny_status 429 if is_free rate_free http-request deny deny_status 429 if is_pro rate_pro default_backend bk_api

19HTTP/2 vs HTTP/1.1 分流

frontend ft_web bind *:443 ssl crt /etc/haproxy/certs/site.pem alpn h2,http/1.1 # 偵測 HTTP 版本 acl is_h2 fc_http_major eq 2 # H2 client → 支援 server push 的後端 use_backend bk_h2_optimized if is_h2 default_backend bk_standard

20多條件 WAF 規則組合

frontend ft_web bind *:443 ssl crt /etc/haproxy/certs/site.pem # SQL Injection 偵測（基礎） acl sqli_attempt url_param -m reg -i '(union|select|insert|update|delete|drop)\s' acl sqli_in_path path -m sub -i "'" -- acl sqli_in_cookie req.cook -m reg -i '(union|select|drop)' # XSS 偵測（基礎） acl xss_attempt url_param -m sub -i <script acl xss_in_path path -m sub -i <script # 異常 header acl too_many_headers req.hdr_cnt(X-Forwarded-For) gt 5 acl oversized_cookie req.cook,length gt 4096 # 綜合阻擋 http-request deny deny_status 403 if sqli_attempt || sqli_in_path || sqli_in_cookie http-request deny deny_status 403 if xss_attempt || xss_in_path http-request deny deny_status 400 if too_many_headers http-request deny deny_status 400 if oversized_cookie default_backend bk_web
⚠️ 這不是真正的 WAF
HAProxy ACL 做基礎防護有效，但誤判率高。生產環境建議搭配 SPOE 整合 ModSecurity 或 Coraza。

ACL 除錯技巧

🔍 四大除錯工具

# 1. 設定檢查 — 偵測無效 ACL / 未使用 ACL haproxy -c -f /etc/haproxy/haproxy.cfg haproxy -c -dD -f /etc/haproxy/haproxy.cfg # 診斷模式：更多警告 # 2. Runtime API — 查看 / 管理 ACL 內容 echo "show acl" | socat stdio $SOCK # 列出所有 ACL echo "show acl #0" | socat stdio $SOCK # 第 0 個 ACL 的內容 echo "get acl #0 1.2.3.4" | socat stdio $SOCK # 測試 1.2.3.4 是否匹配 echo "add acl #0 5.6.7.8" | socat stdio $SOCK # 動態加入 echo "del acl #0 5.6.7.8" | socat stdio $SOCK # 動態移除 # 3. 變數注入 Header 觀察 — 將 ACL 結果寫入 header http-request set-header X-Debug-Is-API %[path,map_beg(/etc/haproxy/debug.map,no)] http-request set-header X-Debug-Src %[src] http-request set-header X-Debug-ReqRate %[sc_http_req_rate(0)] http-request set-header X-Debug-Country %[var(txn.country)] # 4. 日誌 — 在 log-format 中加入 ACL 相關變數 log-format "%ci %[var(txn.country)] %[sc_http_req_rate(0)] %r"

🧪 逐步除錯流程

症狀：ACL 規則似乎沒生效。

Step 1: 確認 ACL 有被評估到 → 加 http-request set-header X-ACL-Hit "yes" if <acl> → curl -v 觀察 header 是否出現 Step 2: 確認 fetch 有回傳值 → 加 http-request set-header X-Fetch-Val %[<fetch>] → curl -v 觀察值是否符合預期 Step 3: 確認比對方法正確 → 常見錯誤：用 -m str 但值含前後空白 → 用 -m sub 測試子字串匹配是否成功 Step 4: 確認規則順序 → HAProxy 從上到下評估，先匹配者先執行 → 把除錯規則移到最前面測試

ACL 常見錯誤速查

#錯誤症狀修正
1path_beg /admin 匹配了 /administrator過度匹配改用 path -m dir admin 或 path_beg /admin/
2大小寫不匹配Host header 是 API.Example.COM加 -i flag 或用 ,lower converter
3同名 ACL = OR 不是 AND預期 AND 但實際是 OR不同條件用不同名稱
4req.body 永遠為空未啟用 buffer加 option http-buffer-request
5TCP mode 使用 HTTP fetchACL 永遠不匹配確認 mode http
6ACL 檔案路徑是 chroot 後的找不到檔案使用 chroot 內的絕對路徑
7hdr(host) 含 portexample.com:8080 ≠ example.com用 hdr(host),field(1,':') 或 host_only
8正則未跳脫特殊字元path -m reg /api/v1.0 的 . 匹配任意字元path -m reg '/api/v1\.0'
9Runtime 加入的 ACL reload 後消失安全規則失效同步更新磁碟檔案
10Pattern 以 - 開頭被解析為 flag語法錯誤在 pattern 前加 --：acl x hdr(X) -- -value
