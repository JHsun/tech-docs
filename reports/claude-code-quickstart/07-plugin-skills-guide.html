什麼是 Skill？ Skill 是 Claude Code 的按需載入知識模組，透過 /plugin 安裝後自動啟用。當你的對話內容匹配 Skill 的觸發描述時，Claude 會自動調用對應 Skill 來增強回應品質。你也可以直接用 /skill-name 手動觸發。

安裝方式： 執行 /plugin 開啟插件管理介面，勾選需要的插件安裝。執行 /reload-plugins 重新載入已安裝的插件。

⚡ Superpowers — 核心開發工作流 (14 Skills)

最重要的插件，涵蓋從規劃到交付的完整開發生命週期。
規劃與設計
brainstorming /brainstorm
創意發想與需求探索。在任何創造性工作之前必須使用 — 建立功能、元件、新行為前，先探索使用者意圖、需求與設計方案，避免直接跳入實作。
建立功能、新增元件、修改行為前自動觸發
writing-plans
撰寫實作計畫。當你有規格或需求要進行多步驟任務時使用。產出結構化的分步計畫，包含檔案清單、依賴關係、風險評估。在動手寫程式碼之前先對齊方向。
有 spec 或多步驟任務需求時
executing-plans
執行實作計畫。在獨立 session 中按計畫逐步執行，設有 review checkpoint。適合將大型任務拆分到隔離環境中完成。
已有完成的實作計畫，準備執行時
開發與測試
test-driven-development
測試驅動開發（TDD）。在實作任何功能或修 bug 前，先撰寫測試案例，再寫實作程式碼。確保程式碼品質從一開始就有測試保障。
實作功能或修 bug 前
systematic-debugging
系統化除錯。遇到任何 bug、測試失敗或非預期行為時使用。不猜測原因，而是有系統地定位根因：重現問題 → 縮小範圍 → 驗證假設 → 修復。
遇到 bug、測試失敗、非預期行為時
subagent-driven-development
子代理驅動開發。在當前 session 中執行有獨立任務的實作計畫，將各任務分派給子代理並行處理。
計畫中有多個可並行的獨立任務時
dispatching-parallel-agents
平行代理調度。面對 2 個以上彼此獨立、不需共享狀態或循序執行的任務時，同時啟動多個代理並行處理。
2+ 獨立任務需要並行處理時
Git 與整合
using-git-worktrees
Git Worktree 隔離開發。需要隔離開發時建立 worktree，包含智慧目錄選擇和安全性驗證。適合執行實作計畫前建立獨立工作環境。
需要與目前工作區隔離的功能開發時
finishing-a-development-branch
完成開發分支。實作完成且測試通過後，引導你選擇整合方式：合併、建立 PR 或清理。提供結構化選項避免遺漏步驟。
實作完成、測試通過，準備整合時
Code Review 與驗證
requesting-code-review
發起 Code Review。完成任務、實作主要功能或合併前使用，驗證工作成果是否符合需求。
receiving-code-review
接收 Code Review 回饋。收到 review 意見時使用，特別是回饋不清楚或技術上有疑問時。要求技術嚴謹性，不盲目接受或敷衍同意。
verification-before-completion
完成前驗證。在宣稱工作完成、修復成功或測試通過之前，必須先執行驗證命令並確認輸出。「證據先於斷言」。
準備提交或建立 PR 前自動觸發
writing-skills
撰寫 Skill。建立新 Skill、編輯現有 Skill 或部署前驗證 Skill 是否正常運作。
using-superpowers
Superpowers 入口。開始任何對話時使用，建立如何找到和使用 Skills 的方式。確保在回應前先查找合適的 Skill。

📖 實戰範例 — 核心 Skills 使用方式

範例 1：從零規劃到實作一個新功能
Step 1 — brainstorming
你：「我想為 API 加入 rate limiting 功能」
→ Claude 自動觸發 brainstorming，探索：
  • 限流演算法選擇（Token Bucket / Sliding Window）
  • 儲存方式（Redis / 記憶體）
  • 需要限流的 endpoint 範圍
  • 錯誤回應格式（429 + Retry-After）

Step 2 — writing-plans
你：「好的，就用 Sliding Window + Redis，請規劃」
→ 產出結構化計畫：
  • 檔案清單：middleware、config、tests
  • 實作順序與依賴關係
  • 每步驟的驗收標準

Step 3 — using-git-worktrees
→ 建立隔離的 worktree 分支：
feature/rate-limiting

Step 4 — test-driven-development
→ 先寫測試，再寫實作：
  • 測試正常請求通過
  • 測試超過限制返回 429
  • 測試 window 重置後恢復

Step 5 — verification-before-completion
→ 執行所有測試 + lint，確認全部通過後才 commit

Step 6 — commit-push-pr
→ 一鍵提交、推送、開 PR
範例 2：systematic-debugging 除錯流程
你：「使用者反映登入後偶爾被導向 404 頁面」

→ Claude 觸發 systematic-debugging：

① 重現問題
  • 檢查錯誤日誌，找出 404 的路由
  • 確認是否特定條件才觸發

② 縮小範圍
  • 追蹤 auth callback → redirect 的程式碼路徑
  • 發現 redirect URL 在 session 過期時為 null

③ 驗證假設
  • 寫測試模擬 session 過期場景
  • 確認 null redirect 觸發 404

④ 修復並驗證
  • 加入 fallback redirect 到首頁
  • 測試通過 → verification-before-completion
範例 3：dispatching-parallel-agents
你：「重構這三個模組：auth、payment、notification」

→ 判斷三個模組彼此獨立，同時派出 3 個 agent：

Agent 1 → 重構 auth 模組
Agent 2 → 重構 payment 模組
Agent 3 → 重構 notification 模組

各自在隔離環境中作業，最後合併結果。
效率：原本 30 分鐘 → 並行約 12 分鐘

❓ 常見問題 (FAQ)

Q：Skill 沒有自動觸發怎麼辦？
確認插件已安裝（/plugin）並重載（/reload-plugins）。若仍未觸發，在 prompt 中加入 Skill 相關關鍵字，例如「請用 TDD 方式」「請先做 brainstorming」。也可直接用斜線命令 /commit 手動觸發。
Q：Skill 和 CLAUDE.md 有什麼差別？
CLAUDE.md 每次對話都會載入（always-on），適合放專案慣例和架構。Skill 按需載入（on-demand），只在觸發時才注入 context，不佔用基礎 context window。大型知識庫建議用 Skill。
Q：多個 Skill 同時觸發會衝突嗎？
不會。Claude 可以在同一對話中使用多個 Skill，它們是互補的知識模組。例如 brainstorming 完成後接 writing-plans 是常見的組合，兩者會自然銜接。
Q：如何建立自己的 Skill？
使用 skill-creator 或 plugin-dev:skill-development。基本結構是在 .claude/skills/my-skill/ 下建立 SKILL.md（含 frontmatter 定義觸發條件）和可選的 scripts/references/assets 目錄。
Q：dispatching-parallel-agents 和 subagent-driven-development 差在哪？
dispatching-parallel-agents 是將完全獨立的任務分給不同 agent 並行處理。subagent-driven-development 是在當前 session 中按實作計畫將步驟分派給子代理。前者更適合無關聯的任務，後者適合有計畫的分步驟執行。
Q：Hooks 和 Skills 的關係？
Hooks 是確定性的生命週期事件回呼（100% 執行），用於自動化驗證、格式化。Skills 是 LLM 按需載入的知識，影響 Claude 的思維方式。兩者互補 — Hooks 做硬性阻擋，Skills 做軟性引導。
Q：Skill 會消耗額外的 token 嗎？
會，但只有在被觸發時。Skill 內容被注入到 context 中，會佔用 context window。所以 Skill 內容應保持簡潔，大量參考資料放在 references/ 目錄，按需讀取。
Q：如何查看 Skill 的原始定義？
插件快取目錄在 ~/.claude/plugins/cache/，每個 Skill 在 skills/<skill-name>/SKILL.md。你可以直接閱讀 SKILL.md 了解觸發條件、指令內容和結構。
