🗺️ 進階模式全景圖

① Brainstorming → ② Writing Plans → ③ Git Worktree → ④ Subagent / TDD → ⑤ Verify → ⑥ Review → ⑦ Finish Branch
本篇深入每個環節的內部運作原理、操作細節和實戰範例。基礎概念請參閱第 1-6 篇，Skills 清單參閱第 7 篇。

🌳 Git Worktree 隔離開發實戰

目錄選擇優先順序
1
專案內 .worktrees/ — 若已存在，直接使用
2
專案內 worktrees/ — 次選（無點前綴）
3
CLAUDE.md 指定路徑 — 若有設定偏好目錄
4
全域 ~/.config/superpowers/worktrees/ — 免 .gitignore 檢查
5
詢問使用者 — 以上皆無時互動選擇
安全驗證：專案內目錄必須在 .gitignore 中，避免 worktree 被提交。全域目錄免驗證。
自動設定偵測
建立 worktree 後自動偵測並執行：
package.json → npm install ｜ Cargo.toml → cargo build ｜ requirements.txt → pip install
pyproject.toml → poetry install ｜ go.mod → go mod download
接著執行基線測試，確認乾淨環境測試通過後才宣告 Ready。
完成分支：4 個選項
Merge
合併回 main + 刪分支 + 清 worktree
PR
push -u + gh pr create + 清 worktree
Keep
保留 worktree 不動
Discard
需輸入「discard」確認 + 強刪分支 + 清 worktree

💡 Brainstorming 設計優先

硬性閘門：設計未被批准前，絕對不寫程式碼、不建腳手架、不實作。
9 步流程
1
探索專案上下文（檔案、文件、近期 commits）
2
提供視覺化伴侶（若涉及 UI）
3
一次一個問題釐清需求（偏好多選題）
4
提出 2-3 個方案及各自的取捨分析
5
逐段呈現設計（每段取得批准）
6
撰寫設計文件 → docs/superpowers/specs/
7
Spec 自我審查（佔位符掃描、內部一致性、範圍檢查、歧義檢查）
8
使用者 Review 書面 Spec
9
→ 調用 writing-plans（唯一出口）
核心原則：YAGNI（不做不需要的）、一次一問、多選偏好、永遠探索替代方案、漸進驗證。

📋 Writing Plans — 計畫驅動開發

Plan 文件模板
# [功能名稱] Implementation Plan > **For agentic workers:** REQUIRED SUB-SKILL: > Use superpowers:subagent-driven-development > to implement this plan task-by-task. Goal: [一句話描述要建構什麼] Architecture: [2-3 句描述方法] Tech Stack: [關鍵技術/函式庫] --- ### Task 1: [元件名稱] Files: - Create: `exact/path/to/file.py` - Modify: `exact/path/to/existing.py:123-145` - Test: `tests/exact/path/to/test.py` - [ ] Step 1: 撰寫失敗測試 [完整測試程式碼] - [ ] Step 2: 執行測試確認失敗 Run: `pytest` → Expected: FAIL - [ ] Step 3: 撰寫最小實作 [完整實作程式碼] - [ ] Step 4: 執行測試確認通過 Run: `pytest` → Expected: PASS - [ ] Step 5: Commit
存放位置：docs/superpowers/plans/YYYY-MM-DD-<feature>.md
咬碎粒度原則
每一步只做一件事：
✓ 「撰寫失敗測試」— 一步
✓ 「執行確認失敗」— 一步
✓ 「撰寫最小實作」— 一步
✓ 「執行確認通過」— 一步
✓ 「Commit」— 一步
禁止佔位符
計畫中絕對不允許出現：
✗ TBD、TODO、implement later
✗ 「加入適當的錯誤處理」
✗ 「撰寫上述的測試」
✗ 「類似 Task N」
✗ 只描述 WHAT 不描述 HOW 的步驟
✗ 引用未定義的型別/函式
Plan 自我審查清單
① Spec 覆蓋 — 每個需求都能指向對應 Task？
② 佔位符掃描 — 有任何紅旗詞彙？
③ 型別一致性 — 跨 Task 的型別定義一致？
執行選項
A. Subagent-Driven（推薦）
每 Task 派新 agent + 兩階段 Review

B. Inline Execution
在當前 session 逐步執行 + checkpoint

⚡ Dispatching Parallel Agents

✓ 適用場景

  • 3+ 測試檔案各有不同根因
  • 多個子系統獨立故障
  • 每個問題不需要其他問題的上下文
  • 沒有共享狀態

✗ 不適用

  • 故障相互關聯（修一個可能修全部）
  • 需要完整系統狀態才能理解
  • 探索性除錯（不知道壞在哪）
  • 有共享狀態（agent 會互相干擾）
Agent Prompt 三要素
Focused — 一個明確的問題領域
Self-contained — 包含所有需要的上下文
Specific output — 明確定義 agent 該回傳什麼
實戰範例
情境：6 個測試失敗，分布在 3 個檔案

Agent 1 → agent-tool-abort.test.ts
結果：將 timeout 替換為 event-based
Agent 2 → batch-completion.test.ts
結果：修復 event 結構 bug
Agent 3 → race-conditions.test.ts
結果：加入 async 等待

修復彼此獨立、無衝突、全套測試通過

🤖 Subagent-Driven Development — 兩階段 Review 系統

核心工作流
1
派出 Implementer — 提供完整 Task 上下文
2
Implementer 實作 + 測試 + commit + 自我 review
3
Spec Compliance Review — 是否完成規格要求？（不多不少）
4
若有問題 → Implementer 修復
5
Code Quality Review — 實作品質是否良好？（乾淨、有測試、可維護）
6
若有問題 → Implementer 修復
7
標記 Task 完成 → 下一個 Task
關鍵順序：Spec Compliance Review 必須在 Code Quality Review 之前。先確認做對了，再確認做好了。
Implementer 狀態處理
DONE
→ 進入 Spec Review
DONE_WITH_CONCERNS
→ 標記疑慮，先讀再決定
NEEDS_CONTEXT
→ 提供缺失上下文，重新派出
BLOCKED
→ 評估阻礙、提供更多資訊或拆分更小
Model 選擇策略
機械任務（1-2 檔案、明確規格）
→ haiku 快速、便宜

整合任務（多檔案協調）
→ sonnet 平衡

架構/設計/Review（需要深度判斷）
→ opus 最強能力
關鍵禁忌
✗ 未經同意就在 main/master 上開始
✗ 跳過 Review
✗ 帶著未修復問題繼續
✗ 同時派多個 Implementer（衝突風險）
✗ 讓 subagent 自己讀 plan 檔（提供全文）
✗ 跳過場景設定上下文
✗ 忽略 subagent 提出的問題
✗ 接受「差不多」的 Spec Compliance
✗ 跳過 Review 迴圈

🔌 Plugin 開發實戰

目錄結構
my-plugin/ ├── .claude-plugin/ │ └── plugin.json # 必要 manifest ├── commands/ # 斜線命令 (.md) ├── agents/ # 代理定義 (.md) ├── skills/ │ └── my-skill/ │ └── SKILL.md # 必要 ├── hooks/ │ └── hooks.json # 事件處理 ├── .mcp.json # MCP 伺服器 └── scripts/
plugin.json 必填
{ "name": "my-plugin", "version": "1.0.0", "description": "...", "author": { "name": "..." } }
路徑：永遠用 ${CLAUDE_PLUGIN_ROOT}，不要硬編碼絕對路徑。Manifest 放 .claude-plugin/，元件放根目錄。
Agent 定義格式
--- name: code-reviewer description: | Use this agent when... <example> Context: ... user: "..." assistant: "..." <commentary>...</commentary> </example> model: inherit color: blue tools: ["Read", "Grep", "Glob"] --- You are [角色描述]... **Core Responsibilities:** 1. ... 2. ... **Output Format:** ...
Color 語意
blue
分析/review
cyan
搜尋/導航
green
成功導向
yellow
驗證/警告
magenta
創意/生成
red
關鍵/安全
Hook 類型對比

Prompt-Based（推薦）

{ "type": "prompt", "prompt": "Evaluate if appropriate: $TOOL_INPUT", "timeout": 30 }
上下文感知、邊界情況處理更好

Command

{ "type": "command", "command": "bash ${CLAUDE_PLUGIN_ROOT}/scripts/check.sh", "timeout": 60 }
確定性執行，最常用 (95%)
Matcher 語法
"Write" — 精確匹配
"Read|Write|Edit" — 多個工具
"*" — 萬用字元
"mcp__.*__delete.*" — 正規表達式
⚠ 區分大小寫
Exit Code
0 成功（stdout 顯示）
2 阻擋錯誤（stderr 餵給 Claude）
其他 非阻擋錯誤

🎯 進階技巧 & 陷阱

Skill 優先順序規則
指令優先順序：
① 使用者明確指令（最高）
② Superpowers Skills
③ 系統預設 prompt（最低）

多 Skill 衝突時：
① 流程 Skill 優先（brainstorming、debugging）
② 實作 Skill 其次
紅旗偵測（合理化陷阱）
「這只是簡單問題」
→ 問題也是任務，檢查 Skills
「我需要先了解更多」
→ Skill 檢查在釐清之前
「讓我先探索 codebase」
→ Skill 會告訴你如何探索
「讓我先收集資訊」
→ Skill 會告訴你如何收集
Context 管理策略
Worktree 隔離：
每個功能在獨立 worktree，context 不互相汙染。

Subagent 隔離：
每個 Task 用新 agent，自帶乾淨 context。不要讓 subagent 自己讀 plan — 直接提供完整文字。

/compact 壓縮：
長對話用 /compact 整理，保留關鍵資訊。

Skill 按需載入：
CLAUDE.md 永遠載入，Skill 按需。大量參考資料放 references/，不要塞進 SKILL.md。
常見錯誤與解法
✗ 跳過 brainstorming 直接寫 code
→ 返工成本 > 規劃成本

✗ Plan 中用佔位符
→ Subagent 無法自行補完

✗ 同時派多個 Implementer
→ 檔案衝突、合併噩夢

✗ Code Quality 在 Spec 之前 Review
→ 打磨了錯誤的實作

✗ Worktree 目錄沒加 .gitignore
→ worktree 被提交到 repo

✗ Hook 硬編碼絕對路徑
→ 用 ${CLAUDE_PLUGIN_ROOT}

✗ 忽略 subagent 的 BLOCKED 狀態
→ 評估阻礙、拆分更小任務

🖥️ Computer Use in CLI（Research Preview）

Claude 可以直接從終端機開啟原生應用程式、點擊 UI 元素、測試變更並修復錯誤。
啟用方式
/mcp → 開啟 computer-use toggle 即可啟用
適用場景
1
測試 UI 變更 — 開啟應用程式驗證視覺呈現
2
自動化桌面工作流 — 操作原生應用完成重複任務
3
除錯視覺問題 — 截圖、點擊、檢視實際渲染結果
注意：Research Preview 功能，行為可能不穩定。適合探索性使用，不建議用於生產關鍵流程。

🔄 Routines（Research Preview · 4月14日）

儲存的配置組合（prompt + repos + connectors），在 Anthropic 雲端基礎設施上執行。
排程化、重複性自動化任務，不需要本機開機。
建立方式
1
claude.ai/code/routines — 網頁介面建立
2
/schedule — CLI 內直接建立
Webhook Routines
綁定 GitHub 事件自動觸發：
PR PR 留言回應
CI CI 失敗自動修復
Event 自訂 GitHub event 觸發
範例用途
1
每日代碼審查 — 定時掃描新 PR 提供建議
2
自動化依賴更新 — 排程檢查過時依賴並建立 PR
3
排程測試執行 — 定時運行完整測試套件並回報
優勢：完全雲端執行，不佔用本機資源。Routine 會在指定時間自動啟動，結果可透過 webhook 或郵件通知。

📡 Remote Control

將本機 Claude Code session 橋接至 claude.ai/code、iOS App 和 Android App。
核心功能
1
REST API — 程式化控制本機 session
2
WebSocket Stream — 即時輸出串流
3
Push 通知 — 手機端推播提醒（需啟用）
設定方式
在 VS Code 中執行 /remote-control 即可啟用
支援的指令
/context
管理上下文
/exit
結束 session
/reload-plugins
重載插件
/extra-usage
額外用量
跨裝置：從手機遠端操作桌面 Claude Code，通勤時也能監控和控制長時間任務。

👥 Agent Teams

協調多個 Claude Code 實例，一個 session 作為 Team Lead，其餘為獨立 Teammate。
架構特點
1
獨立 Context Window — 每個 Teammate 擁有自己的上下文
2
直接通訊 — Agent 之間可直接傳遞訊息
3
Team Lead 協調 — 分配任務、匯整結果
適用場景
✓ 平行功能開發（多個不相關的 feature）
✓ 跨模組複雜重構
⚠ 需要注意共享檔案的衝突風險

✨ Flicker-free 全螢幕渲染

啟用方式
/tui fullscreen — 啟用 alt-screen renderer + 虛擬捲動緩衝區

環境變數：
CLAUDE_CODE_NO_FLICKER=1 — 永久啟用無閃爍模式
相關指令
1
/focus — 將 focus 模式從 transcript toggle 分離
2
autoScrollEnabled — 設定中可停用自動捲動
效果：消除長輸出時的畫面閃爍，特別適合大量程式碼生成和除錯場景。

⚙️ Native Binary CLI 架構（v2.1.113）

CLI 現在會生成一個原生 Claude Code 二進位檔，取代過去捆綁的 JavaScript。
架構變更
1
Per-platform 選配依賴 — 每個平台獨立的 optional dependency
2
免 Node.js — native binary 安裝不需要 Node.js 環境
3
更快啟動 — 原生二進位檔大幅縮短冷啟動時間
影響：對使用者透明升級，CLI 行為不變。主要改善安裝體驗和啟動效能。
