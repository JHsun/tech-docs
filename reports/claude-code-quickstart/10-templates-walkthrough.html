🤖 Implementer Subagent Prompt 模板

派出 Implementer 時使用此模板。直接複製 替換 [方括號] 內容即可。
## 派出方式 Agent tool (general-purpose): description: "Implement Task N: [task name]" prompt: | You are implementing Task N: [task name] ## Task Description [完整 Task 文字 — 直接貼上，不要讓 subagent 自己讀檔案] ## Context [場景設定：這在哪裡、依賴什麼、架構上下文] ## Before You Begin If you have questions about: - The requirements or acceptance criteria - The approach or implementation strategy - Dependencies or assumptions - Anything unclear in the task description **Ask them now.** Raise any concerns before starting work. ## Your Job Once you're clear on requirements: 1. Implement exactly what the task specifies 2. Write tests (following TDD if task says to) 3. Verify implementation works 4. Commit your work 5. Self-review (see below) 6. Report back Work from: [directory] **While you work:** If you encounter something unexpected, ask questions. Don't guess or make assumptions. ## Code Organization - Follow the file structure defined in the plan - Each file: one clear responsibility with a well-defined interface - If a file is growing beyond the plan's intent → report DONE_WITH_CONCERNS - In existing codebases, follow established patterns ## When You're in Over Your Head It is always OK to stop and say "this is too hard for me." Bad work is worse than no work. **STOP and escalate when:** - The task requires architectural decisions with multiple valid approaches - You need to understand code beyond what was provided - You feel uncertain about whether your approach is correct - The task involves restructuring code the plan didn't anticipate - You've been reading file after file without progress How to escalate: Report BLOCKED or NEEDS_CONTEXT ## Before Reporting Back: Self-Review Completeness: Did I implement everything? Missing requirements? Edge cases? Quality: Is this my best work? Names clear? Code clean? Discipline: YAGNI? Only built what was requested? Follow existing patterns? Testing: Tests verify behavior (not mocks)? TDD if required? Comprehensive? If you find issues during self-review, fix them now. ## Report Format - Status: DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT - What you implemented - What you tested and test results - Files changed - Self-review findings - Any issues or concerns

🔍 Spec Compliance Reviewer 模板

驗證 Implementer 是否完成規格要求。在 Code Quality Review 之前執行。
Agent tool (general-purpose): description: "Review spec compliance for Task N" prompt: | You are reviewing whether an implementation matches its specification. ## What Was Requested [完整 Task 需求文字] ## What Implementer Claims They Built [Implementer 的回報內容] ## CRITICAL: Do Not Trust the Report The implementer finished suspiciously quickly. Their report may be incomplete, inaccurate, or optimistic. You MUST verify everything independently. **DO NOT:** - Take their word for what they implemented - Trust their claims about completeness - Accept their interpretation of requirements **DO:** - Read the actual code they wrote - Compare actual implementation to requirements line by line - Check for missing pieces they claimed to implement - Look for extra features they didn't mention ## Your Job Read the implementation code and verify: Missing requirements: - Did they implement everything requested? - Requirements they skipped or missed? - Claimed something works but didn't implement it? Extra/unneeded work: - Built things that weren't requested? - Over-engineered or unnecessary features? Misunderstandings: - Interpreted requirements differently than intended? - Solved the wrong problem? Verify by reading code, not by trusting report. Report: - ✅ Spec compliant (if everything matches after code inspection) - ❌ Issues found: [list with file:line references]

✅ Code Quality Reviewer

Spec 通過後才執行此 Review。
Agent tool (superpowers:code-reviewer): WHAT_WAS_IMPLEMENTED: [Implementer 回報內容] PLAN_OR_REQUIREMENTS: Task N from [plan-file] BASE_SHA: [task 前的 commit] HEAD_SHA: [當前 commit] DESCRIPTION: [task 摘要]
額外檢查項目
✓ 每個檔案有單一職責？
✓ 單元可獨立理解和測試？
✓ 遵循 Plan 的檔案結構？
✓ 新檔案是否已過大？
✓ 是否改善了接觸到的程式碼？
回傳格式
Strengths — 做得好的地方
Issues — Critical / Important / Minor
Assessment — 整體評估

🔄 完整 Subagent Review 迴圈 — 端到端流程圖

派出 Implementer → 實作+測試+commit → Self-Review → Spec Compliance Review → ❌ 有問題？→ 修復 → 重審 → Code Quality Review → ❌ 有問題？→ 修復 → 重審 → ✅ Task 完成
DONE
一切順利 → 直接進入 Spec Review
DONE_WITH_CONCERNS
完成但有疑慮 → 先讀疑慮內容再決定
NEEDS_CONTEXT
缺少資訊 → 提供上下文後重新派出
BLOCKED
無法完成 → 提供資訊 / 換更強模型 / 拆更小

🔌 Plugin 開發端到端 Walkthrough — 建立一個 Code Style Enforcer 插件

完整範例：從零建立一個插件，包含 Skill、Hook、Command 和 Agent。
Step 1：建立腳手架
# 建立目錄結構 mkdir -p code-style-enforcer/{.claude-plugin,commands,skills/style-check,agents,hooks,scripts}
Step 2：plugin.json
// .claude-plugin/plugin.json { "name": "code-style-enforcer", "version": "1.0.0", "description": "Enforce code style conventions with hooks, skills and agents", "author": { "name": "Your Name" } }
Step 3：斜線命令
--- description: Check code style for current changes --- Review all staged and unstaged changes for code style violations based on this project's conventions in CLAUDE.md. Report findings grouped by severity: - 🔴 Critical (blocking) - 🟡 Warning (should fix) - 🟢 Suggestion (nice to have)
Step 4：Skill 定義
--- name: style-check description: | Use this skill when reviewing code for style compliance, when the user asks about coding conventions, or when preparing code for review. <example> Context: User just finished coding user: "Check if my code follows our style guide" assistant: Invokes style-check skill <commentary> User explicitly asked for style check </commentary> </example> --- # Code Style Check ## Conventions Read the project's CLAUDE.md and any .eslintrc / .prettierrc configs. ## Process 1. Identify modified files 2. Check naming conventions 3. Check import ordering 4. Check function complexity 5. Check documentation coverage ## Output Format Group by file, then by severity. Include file:line references.
Step 5：Agent 定義
--- name: style-reviewer description: | Use this agent for deep style review. <example> Context: PR ready for review user: "Do a thorough style review" assistant: Launches style-reviewer </example> model: haiku color: yellow tools: ["Read", "Grep", "Glob"] --- You are a code style reviewer. Read each changed file and compare against the project's conventions. **Report Format:** - File path - Line number - Issue description - Suggested fix
Step 6：Hook 設定
// hooks/hooks.json { "description": "Auto-format on save", "hooks": { "PostToolUse": [{ "matcher": "Write|Edit", "hooks": [{ "type": "command", "command": "bash ${CLAUDE_PLUGIN_ROOT}/scripts/format.sh", "timeout": 30 }] }] } }
Step 7：格式化腳本
#!/bin/bash # scripts/format.sh FILE="$CLAUDE_TOOL_INPUT_FILE_PATH" if [[ "$FILE" == *.ts ]] || \ [[ "$FILE" == *.js ]]; then npx prettier --write "$FILE" fi
完成！將 code-style-enforcer/ 放入 ~/.claude/plugins/local/ 或提交到 GitHub。執行 /plugin 安裝 → /reload-plugins 啟用。輸入 /check-style 測試命令，修改檔案時 Hook 自動觸發格式化。

⚡ 實用 Hook 範例集 — 直接複製使用

阻止危險命令
// PreToolUse: 阻止 rm -rf / 和 force push { "matcher": "Bash", "hooks": [{ "type": "command", "command": "bash -c 'echo \"$CLAUDE_TOOL_INPUT\" | grep -qE \"rm\\s+-rf\\s+/|push.*--force.*main\" && exit 2 || exit 0'" }] }
自動執行測試
// PostToolUse: 寫入檔案後自動跑測試 { "matcher": "Write|Edit", "hooks": [{ "type": "command", "command": "npm test --silent 2>&1 | tail -5", "timeout": 60 }] }
Session 啟動注入上下文
// SessionStart: 注入 git 狀態 { "hooks": [{ "type": "command", "command": "echo \"Branch: $(git branch --show-current)\nUncommitted: $(git status -s | wc -l) files\"" }] }
Prompt-based 語意判斷
// PreToolUse: 用 LLM 判斷是否適當 { "matcher": "Bash", "hooks": [{ "type": "prompt", "prompt": "Is this command safe and relevant to the current task? Command: $TOOL_INPUT. Reply ALLOW or DENY with reason.", "timeout": 30 }] }
Stop 時驗證完整性
// Stop: Claude 結束回覆時驗證 { "hooks": [{ "type": "prompt", "prompt": "Did Claude fully complete the user's request? Check for incomplete implementations, untested code, or missing edge cases. Reply COMPLETE or INCOMPLETE with specifics." }] } // exit 2 → Claude 被迫繼續工作
防止提交 secrets
// PreToolUse: 阻止寫入含 secrets 的檔案 { "matcher": "Write|Edit", "hooks": [{ "type": "command", "command": "bash -c 'echo \"$CLAUDE_TOOL_INPUT\" | grep -qiE \"(api_key|secret|password|token)\\s*[:=]\\s*[\\x27\\\"][^\\x27\\\"]{8,}\" && exit 2 || exit 0'" }] }

📝 SKILL.md 完整模板

--- name: my-skill-name description: | Use this skill when [觸發條件]. Examples: [使用者會說什麼]. <example> Context: [情境描述] user: "[使用者請求]" assistant: "[Claude 回應]" <commentary> [為什麼要觸發此 skill] </commentary> </example> # 提供 2-4 個 example 區塊 # description: 200-1,000 chars --- # Skill 標題 ## 目的 [這個 Skill 做什麼] ## 流程 1. [Step 1] 2. [Step 2] 3. [Step 3] ## 規則 - [規則 1] - [規則 2] ## 輸出格式 [預期的輸出結構]
