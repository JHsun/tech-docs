持久化策略比較：RDB vs AOF

特性RDB（快照）AOF（追加日誌）混合模式（推薦）
原理定期將記憶體快照寫入 .rdb 二進位檔記錄每個寫入操作至日誌檔AOF 檔以 RDB 前導 + 後續 AOF 命令
資料遺失風險最後一次快照後的資料取決於 fsync 策略（最多 1 秒）最小（兼具兩者優勢）
檔案大小較小（壓縮二進位）較大（命令文字日誌）適中
重啟速度快（直接載入快照）慢（重放所有命令）快（載入 RDB 部分 + 少量命令）
效能影響fork() 時短暫阻塞fsync 策略影響寫入效能綜合
適用場景備份、災難復原、可接受分鐘級資料遺失需高資料持久性生產環境推薦
📸

RDB 快照配置

# redis.conf — RDB 設定
save 900 1        # 900秒內≥1次修改觸發
save 300 10       # 300秒內≥10次修改
save 60 10000     # 60秒內≥10000次修改
save ""            # 完全停用 RDB

dbfilename dump.rdb
dir /var/lib/redis
rdbcompression yes          # LZF 壓縮
rdbchecksum yes              # CRC64 校驗
stop-writes-on-bgsave-error yes  # 快照失敗時停止寫入

# 手動觸發
BGSAVE    # 背景快照（不阻塞）
SAVE      # 同步快照（阻塞！生產勿用）
LASTSAVE  # 最後成功快照的時間戳
⚠️
RDB 使用 fork() + Copy-on-Write。大資料集的 fork 可能瞬間阻塞主執行緒，且需預留 2 倍記憶體空間。
📝

AOF 追加日誌配置

# redis.conf — AOF 設定
appendonly yes
appendfilename "appendonly.aof"

# fsync 策略
appendfsync always      # 每次寫入都 fsync（最安全，最慢）
appendfsync everysec     # 每秒 fsync（推薦，平衡效能與安全）
appendfsync no           # 由 OS 決定（最快，風險最高）

# 自動重寫（壓縮 AOF）
auto-aof-rewrite-percentage 100   # 增長 100% 時觸發重寫
auto-aof-rewrite-min-size 64mb    # 最小重寫大小

# 混合持久化（Redis 4.0+，推薦開啟）
aof-use-rdb-preamble yes

# 手動操作
BGREWRITEAOF              # 背景重寫 AOF
CONFIG SET appendonly yes  # 動態啟用（記得 CONFIG REWRITE）
✅
生產建議：開啟 AOF + appendfsync everysec + aof-use-rdb-preamble yes。同時保留 RDB 用於備份。
🔗

主從複製 (Replication)

# Replica 配置（redis.conf）
replicaof 192.168.1.100 6379  # 指向 Primary
masterauth your_password       # Primary 密碼
masteruser repl_user           # ACL 使用者（Redis 6+）
replica-read-only yes           # Replica 唯讀（建議）
replica-serve-stale-data yes    # 同步中仍回應舊資料

# 動態設定
REPLICAOF 192.168.1.100 6379  # 動態設為 Replica
REPLICAOF NO ONE               # 提升為獨立 Primary

# 檢查複製狀態
INFO replication
特性說明
複製方式非同步（asynchronous）。WAIT 命令可實現同步等待
初始同步完整 RDB 傳輸 + 緩衝區命令重放
部分重同步短暫斷線後透過 backlog 增量同步（PSYNC）
鏈式複製支援 Replica of Replica，減輕 Primary 負擔
無磁碟複製repl-diskless-sync yes 直接串流 RDB 至 Replica
🛡

Redis Sentinel（自動故障轉移）

功能說明
監控持續檢查 Primary 和 Replica 是否正常運作
通知故障時透過 API 通知管理員或其他系統
自動故障轉移Primary 失敗時自動提升 Replica 為新 Primary
服務發現客戶端連線 Sentinel 以獲取當前 Primary 地址
 
# sentinel.conf
port 26379
sentinel monitor mymaster 192.168.1.100 6379 2   # quorum=2
sentinel auth-pass mymaster password
sentinel down-after-milliseconds mymaster 5000     # 5秒無回應視為 SDOWN
sentinel failover-timeout mymaster 60000          # 故障轉移逾時
sentinel parallel-syncs mymaster 1               # 並行同步 Replica 數

# 啟動 Sentinel
redis-sentinel /etc/redis/sentinel.conf

# 查詢命令
SENTINEL master mymaster        # 主節點資訊
SENTINEL replicas mymaster      # 從節點列表
SENTINEL sentinels mymaster     # 其他 Sentinel
SENTINEL get-master-addr-by-name mymaster  # 當前 Primary 地址
💡
故障轉移流程：SDOWN（主觀下線）→ 投票確認 → ODOWN（客觀下線）→ 選舉 Leader Sentinel → 提升最佳 Replica → 重新配置其他 Replica。建議至少部署 3 個 Sentinel 實例。
🌐

Redis Cluster（叢集模式）

概念說明
Hash Slotskey 空間分為 16384 個 slot，依 CRC16(key) % 16384 分配
最少節點3 個 Primary（建議各配 1 個 Replica，共 6 節點）
Gossip 協定節點間透過 cluster bus（port+10000）通訊
自動故障轉移Primary 失敗時 Replica 自動接管（無需 Sentinel）
MOVED 重導客戶端收到 MOVED 時連線正確節點
Hash Tag{tag}key 確保相同 tag 的 key 在同一 slot
 
# 建立叢集（redis-cli）
redis-cli --cluster create \
  192.168.1.1:6379 192.168.1.2:6379 192.168.1.3:6379 \
  192.168.1.4:6379 192.168.1.5:6379 192.168.1.6:6379 \
  --cluster-replicas 1
# redis.conf — Cluster 設定
cluster-enabled yes
cluster-config-file nodes.conf
cluster-node-timeout 15000
cluster-require-full-coverage no  # 部分 slot 失敗時仍服務

# 叢集管理命令
CLUSTER INFO               # 叢集狀態
CLUSTER NODES              # 節點列表
CLUSTER SLOTS              # Slot 分配
CLUSTER KEYSLOT mykey      # 查詢 key 的 slot
CLUSTER COUNTKEYSINSLOT 0  # slot 中的 key 數
CLUSTER FAILOVER           # 手動故障轉移
CLUSTER RESET              # 重置節點

# Hash Tag 範例
SET {user:1001}:profile "..."
SET {user:1001}:session "..."
# 兩個 key 保證在同一 slot
⚠️
Cluster 限制：不支援跨 slot 的多 key 操作（除非使用 Hash Tag）。MULTI/EXEC 的所有 key 必須在同一 slot。SELECT 只有 DB 0。Lua 腳本的 key 必須在同一 slot。

架構選擇指南

架構適用場景資料量複雜度自動故障轉移
單機開發/測試、小型應用、快取<數 GB低❌
主從複製讀多寫少、需要讀擴展<數十 GB低❌ 需手動
Sentinel需自動故障轉移但不需分片<100 GB中✅
Cluster大資料集、需水平擴展、高寫入量>100 GB高✅
Cluster + Sentinel企業級大規模部署>1 TB高✅
