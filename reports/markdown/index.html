Markdown 完全教學

Markdown 是一套用純文字標記 (markup) 來描述格式的輕量級語言。它的核心理念很簡單：文件原始碼本身就應該易讀，不需要看到渲染後的結果，也能理解內容結構。這個哲學讓 Markdown 在 2026 年仍然是寫文件最有效率的工具之一。

1 · 認識 Markdown

1.1 Markdown 的誕生

2004 年 3 月 19 日，蘋果社群的部落客 John Gruber 在他的網誌 Daring Fireball 上發表了 Markdown，並描述為一個「給網頁寫作者的 text-to-HTML 轉換工具」。Aaron Swartz 也參與了早期語法的設計。Markdown 1.0 是用 Perl 寫成的腳本，將特定符號 (例如 **bold**) 轉換為 HTML 標籤 (<strong>bold</strong>)。

它的設計目標只有一個：讓寫作者把注意力放在內容，而不是排版。Markdown 並不是要取代 HTML，而是讓那些在純文字環境中也想保有結構感的寫作行為，可以被機械地轉換為網頁。

1.2 為什麼 2026 年仍要學 Markdown？

過去十多年間，Markdown 從一個 perl 小腳本演變成幾乎所有現代寫作工具的底層格式。它在 2026 年的重要性甚至比過去更高，原因有四：

Reason 01

AI 助手的母語

Claude、ChatGPT、Gemini 等 LLM 預設以 Markdown 輸出。當你能讀寫 Markdown，就能跟 AI 高效協作，反過來也能餵給 AI 更乾淨的輸入。

Reason 02

無所不在的格式

GitHub、GitLab、Reddit、Discord、Slack、Notion、Obsidian、Stack Overflow ⋯⋯ 學一次語法，幾乎所有開發與寫作平台都通。

Reason 03

純文字 = 永遠可開啟

1990 年代的 WordPerfect 檔案如今多半打不開。一個 2004 年的 .md 檔今天打開、2044 年也還打得開——它就是純文字。

Reason 04

版本控制友善

Markdown 是純文字，每次修改 git diff 都看得清清楚楚——這是 .docx 永遠做不到的事。

1.3 Markdown vs HTML vs Word

三者目的不同，使用情境也不同。下表幫你快速判斷該用哪一種：

面向 Markdown HTML Word (.docx)
原始碼可讀性 極高，文字即文件 低，標籤干擾閱讀 不適用 (二進位)
排版能力 有限，刻意保持簡單 幾乎無限 豐富但封閉
學習曲線 5 分鐘上手基礎 需數週至數月 UI 點選即可
版本控制 ✅ 純文字 diff ✅ 純文字 diff ❌ 二進位無法 diff
適用情境 README、文件、筆記、部落格 正式網頁、複雜 UI 傳統商業文件

1.4 Markdown 的方言 (Flavors)

因為 Gruber 原始的設計留下了不少模糊地帶，多年來各家工具產生了不同的解讀，這些變體稱為 Markdown flavors。理解它們的差異很重要，否則你寫的文件可能在 GitHub 渲染正常，到了 Pandoc 卻出問題。

📘
CommonMark
由 John MacFarlane 等人主導的標準化規範，解決了原始 Markdown 的歧義問題。絕大多數現代工具都以 CommonMark 為基礎。
🐙
GitHub Flavored Markdown (GFM)
CommonMark 的嚴格超集，加上表格、任務清單、刪除線、自動連結、Alert 警示框等延伸。在 2026 年，當人們說「Markdown」時通常意指 GFM。
🦊
GitLab Flavored Markdown (GLFM)
在 GFM 之上又加了 GitLab 特有功能：Issue 引用、Math (KaTeX)、彩色預覽、Mermaid 圖等。
⚛️
MDX (Markdown + JSX)
允許在 Markdown 中嵌入 React 元件。是現代文件網站 (Docusaurus、Nextra) 的常用格式，但已超出純 Markdown 範疇。

本指南以 CommonMark + GFM 為主，這個組合涵蓋了實務上 95% 的使用情境。

2 · 基礎語法

本章涵蓋 John Gruber 原始設計中所定義的核心元素。幾乎所有 Markdown 工具都支援這些，是最具可攜性的部分。每一節都附上「原始碼 vs 渲染結果」對照範例。

2.1 標題 (Headings)

用 # 字元開頭表示標題，# 的數量代表層級 (1 到 6)。記得 # 與標題文字之間要有一個空格。

Markdown# 第一層 H1 ## 第二層 H2 ### 第三層 H3 #### 第四層 H4 ##### 第五層 H5 ###### 第六層 H6
Render

第一層 H1

第二層 H2

第三層 H3

第四層 H4

第五層 H5

第六層 H6

另外還有 Setext 風格的兩層標題寫法 (用 === 或 --- 在底下畫線)，但 ATX 風格 (#) 更常用、更靈活，建議優先使用。

💡
最佳實踐
  • 每份文件只有一個 H1，作為文件主標題
  • 標題前後加空行，避免某些 parser 誤判
  • 不要跳級 (例如 H2 直接跳 H4)，破壞語意結構

2.2 段落與換行 (Paragraphs & Line Breaks)

段落由空行分隔。這是新手最常踩的雷：在 Markdown 裡，單純按一次 Enter 不會換行，必須空一行才會分段。

Markdown第一段。 這行會跟上一行黏在一起。 這是第二段。
Render

第一段。
這行會跟上一行黏在一起。

這是第二段。

如果想在不換段的情況下換行 (<br>)，可以在行尾加兩個空格，或直接寫 <br>。CommonMark 也支援行尾加反斜線 \，但相容性較差。

2.3 強調 (Emphasis)

三種強調方式：粗體、斜體、粗斜體。可用星號或底線，但建議統一使用星號，因為某些 parser 對單字中間的底線處理不一致 (例如 un_frigging_believable 可能不會被視為斜體)。

Markdown這是 **粗體**。 這是 *斜體*。 這是 ***粗斜體***。 這是 ~~刪除線~~ (GFM)。
Render

這是 粗體。
這是 斜體。
這是 粗斜體。
這是 刪除線 (GFM)。

2.4 區塊引用 (Blockquotes)

用 > 開頭表示引用。可以巢狀使用，也可以在引用內混合其他 Markdown 語法。

Markdown> 這是一段引用。 > 多行引用每行都加 >。 > > > 巢狀引用：再加一層 >。 > > 這層在更裡面。
Render
這是一段引用。
多行引用每行都加 >。
巢狀引用：再加一層 >。
這層在更裡面。

2.5 清單 (Lists)

無序清單用 -、* 或 + 開頭 (建議統一用 -)。有序清單用數字加點 (1.)。

Markdown- 蘋果 - 香蕉 - 葡萄 - 巨峰葡萄 - 麝香葡萄 1. 第一項 2. 第二項 3. 第三項
Render
  • 蘋果
  • 香蕉
  • 葡萄
    • 巨峰葡萄
    • 麝香葡萄
  1. 第一項
  2. 第二項
  3. 第三項
🎯
有序清單的小撇步
實際上你可以全部寫 1.，渲染時會自動編號。這在頻繁增刪項目時特別好用——不用重新編號整份清單。

2.6 程式碼 (Code)

行內程式碼用反引號 (`) 包起來；程式碼區塊用三個反引號 (柵欄式 fenced code block) 並可指定語言以啟用語法高亮。

Markdown使用 `console.log()` 輸出。 ```python def greet(name): return f"Hello, {name}!" ```
Render

使用 console.log() 輸出。

def greet(name):
    return f"Hello, {name}!"

如果程式碼裡本身有反引號，可以用更多反引號來包：例如 ``有 ` 反引號的內容``。

2.7 連結 (Links)

三種寫法：內聯連結、參考式連結、自動連結。

Markdown內聯：[Anthropic](https://anthropic.com) 帶 title：[Claude](https://claude.ai "Claude AI") 參考式：請見 [說明文件][docs]。 自動連結：<https://example.com> [docs]: https://docs.claude.com
Render

內聯：Anthropic

帶 title：Claude

參考式：請見 說明文件。

自動連結：https://example.com

參考式連結很適合長篇文件——把所有 URL 集中在文末管理，正文不會被一堆網址干擾。

2.8 圖片 (Images)

語法跟連結幾乎一樣，只是前面多一個 !。中括號裡是 alt 文字 (替代文字、無障礙與 SEO 關鍵)。

![圖片的替代文字](path/to/image.png)
![帶 title](path/to/image.png "滑鼠停留時顯示")
![參考式][image-id]

[image-id]: path/to/image.png

原始 Markdown 沒有指定圖片尺寸的語法。需要控制大小時，直接寫 HTML <img> 標籤即可。

2.9 水平分隔線 (Horizontal Rule)

用三個或以上的 ---、*** 或 ___ 單獨成行。

Markdown第一段。 --- 第二段。
Render

第一段。

第二段。

2.10 跳脫字元 (Backslash Escapes)

當你想顯示 *、_、# 這些有特殊意義的字元時，前面加反斜線 \：

\*這不是斜體\*
\# 這不是標題

可被跳脫的字元：\ ` * _ {} [] () # + - . !

2.11 在 Markdown 中混用 HTML

Markdown 設計上允許直接寫 HTML——當原生語法不夠用時，這是最後的逃生口。例如要置中文字、加 <sub> 下標、用 <details> 折疊區塊：

<details>
<summary>點我展開更多細節</summary>

這裡可以寫 **Markdown** 內容。

</details>
⚠️
安全考量
出於安全因素，部分 Markdown 平台 (如 Reddit、StackOverflow) 會把 HTML 過濾掉。在不確定的環境，盡量用純 Markdown 語法。

3 · 擴充語法 (GitHub Flavored Markdown)

本章介紹 GFM 在 CommonMark 之上加的功能。這些語法在 GitHub、GitLab、HackMD、Obsidian 等主流平台幾乎都支援，是現代 Markdown 寫作的事實標準。

3.1 表格 (Tables)

用 pipe (|) 與連字號繪製表格。中間那行的冒號控制對齊：:--- 左對齊、:---: 置中、---: 右對齊。

Markdown| 商品 | 單價 | 數量 | | :--- | :---: | ----: | | 蘋果 | 30 | 5 | | 香蕉 | 20 | 12 |
Render
商品單價數量
蘋果305
香蕉2012

表格不能換行內含區塊元素 (例如多段落、清單)，但可以放行內 Markdown (粗體、斜體、行內 code、連結)。儲存格內想換行可用 <br>；儲存格內含 pipe 字元時用 \| 跳脫。

3.2 任務清單 (Task Lists)

用 - [ ] 表示未完成、- [x] 表示已完成。在 GitHub Issues / PR 裡是核心生產力工具。

Markdown- [x] 寫初稿 - [x] 蒐集回饋 - [ ] 修訂 - [ ] 發佈
Render
  • ☑ 寫初稿
  • ☑ 蒐集回饋
  • ☐ 修訂
  • ☐ 發佈

3.3 刪除線 (Strikethrough)

用兩個波浪 ~~ 包起來。GFM 也支援單一波浪 (~)，但請優先使用雙波浪以求最大相容性。

~~這段文字會有刪除線~~

3.4 自動連結 (Autolinks)

GFM 會自動偵測 URL、電子信箱字串並轉成連結，不需要 <...> 包起來：

請參考 https://example.com 或寄信到 hi@example.com

3.5 Alert / Callout 警示框 ⭐

2023 年 GitHub 正式推出的語法，用於凸顯重要資訊。寫法是 blockquote 的第一行加 [!TYPE]，共五種類型：

> [!NOTE]
> 一般資訊
>
> [!TIP]
> 有用的訣竅
>
> [!IMPORTANT]
> 關鍵資訊
>
> [!WARNING]
> 需要立即注意
>
> [!CAUTION]
> 操作可能有負面後果
📘
NOTE
使用者應該知道、即使順著讀也行的補充資訊。
💡
TIP
讓讀者做得更好、更輕鬆的訣竅與建議。
📌
IMPORTANT
讀者必須知道、不能略過的關鍵資訊。
⚠️
WARNING
需要使用者立即注意的緊急訊息。
🚫
CAUTION
警告該操作可能造成嚴重後果，請勿輕忽。

3.6 程式碼語法高亮

柵欄式程式碼區塊在開頭三反引號後加上語言名稱，GFM 支援 100+ 種語言。常用：js、ts、python、bash、sql、html、css、json、yaml、markdown、diff。

```typescript
interface User {
  id: number;
  name: string;
}
const u: User = { id: 1, name: "Ada" };
```

沒寫語言時可用 plaintext 或乾脆留空，避免不必要的高亮。diff 語言對展示變更很實用：

```diff
- const old = "舊版";
+ const next = "新版";
```

3.7 腳註 (Footnotes)

非標準 CommonMark，但 GFM 從 2022 年起原生支援。用 [^id] 標記，文末或文中定義 [^id]: 內容。

這個論點有充分證據[^1]，也有反對意見[^opposing]。

[^1]: 來源：某某研究 2025。
[^opposing]: 反方觀點請見...

3.8 Emoji 短代碼

GitHub、GitLab、Slack、Discord 等支援 :name: 的 emoji 短代碼，例如：

短代碼結果短代碼結果
:rocket:🚀:warning:⚠️
:white_check_mark:✅:x:❌
:thumbsup:👍:fire:🔥
:bug:🐛:sparkles:✨

完整清單可在 emoji-cheat-sheet 或 emojibase 查到。

4 · 進階功能

本章介紹生產力大幅躍升的進階能力。這些功能在主流平台 (GitHub、HackMD、Obsidian、Notion、現代靜態網站產生器) 大多支援，但純 CommonMark 不一定能渲染——使用前先確認你的平台。

4.1 數學公式 (LaTeX)

GitHub、GitLab、HackMD、Obsidian 都支援以 LaTeX 語法寫數學式。行內用單一 dollar 包起來，區塊用兩個 dollar 包起來：

行內：愛因斯坦質能方程式 $E = mc^2$。

區塊置中：

$$
\sum_{i=1}^{n} i = \frac{n(n+1)}{2}
$$

底層引擎通常是 MathJax (功能完整、較慢) 或 KaTeX (速度快、語法子集)。GitHub 用 MathJax；HackMD、Obsidian 預設用 KaTeX。

📐
想完整學 LaTeX 數學公式？
請見 進階參考 → LaTeX 完整章節，內含上下標、分數、根號、矩陣、希臘字母等完整對照與實例。

4.2 圖表 (Mermaid)

Mermaid 是用文字描述圖表的工具。GitHub 自 2022 年原生支援，HackMD、GitLab、Obsidian 也都能渲染。語法是在程式碼柵欄指定 mermaid 語言：

```mermaid
flowchart LR
    A[使用者] --> B{已登入？}
    B -->|是| C[顯示首頁]
    B -->|否| D[導向登入]
```
📊
Mermaid 完整章節
Mermaid 支援 10+ 種圖表類型 (流程圖、序列圖、類別圖、ER 圖、甘特圖、Git graph、心智圖、圓餅圖、時間軸⋯⋯)。 請見 進階參考 → Mermaid 完整章節。

4.3 Front Matter (YAML 中介資料)

在文件最開頭用三條短橫線 --- 包起來的 YAML 區塊，用來提供文件的中介資料。Hugo、Jekyll、Astro、Docusaurus、Obsidian 等都使用：

---
title: 我的第一篇文章
date: 2026-04-29
author: Jane Doe
tags: [markdown, tutorial]
draft: false
---

# 文章內容從這裡開始

4.4 目錄與錨點 (TOC & Anchors)

Markdown 渲染器通常會自動為標題產生 ID (錨點)。規則大致是：轉小寫、空格換成 -、移除符號。例如 ## 4.4 目錄與錨點 會生成 #44-目錄與錨點。

許多平台支援 [[TOC]] 或 [toc] 自動產生目錄；GFM 沒有原生 TOC 語法，但 GitHub README 會在標題旁顯示「Sections」按鈕。

4.5 定義清單 (Definition Lists)

Pandoc、PHP Markdown Extra 支援，GFM 不原生支援但 HackMD 等可以：

Markdown
:   一種輕量級標記語言

CommonMark
:   Markdown 的標準化規範

4.6 縮寫與註腳 (Abbreviations)

非標準功能，但 Pandoc 支援。文件內提及的縮寫會自動加上 tooltip：

HTML 是一種網頁語言。

*[HTML]: HyperText Markup Language

README 是專案的門面。一份好的 README 應該在讀者捲動兩次內回答這些問題：「這是什麼？」「為何我該用？」「怎麼安裝？」「怎麼用？」

5.2 技術部落格寫作流程

建議流程：HackMD 草稿 → GitHub repo 同步 → 靜態網站發佈。HackMD 適合線上協作與快速迭代；GitHub 提供版本控制；Hugo / Astro / Docusaurus 等則把 .md 轉成漂亮網站。

5.3 個人知識管理 (PKM)

2025 年 2 月後 Obsidian 開放商業使用，加上其原生支援 Markdown、雙向連結、外掛生態系，成為 PKM 的首選工具之一。常用方法論：

  • PARA：Project / Area / Resource / Archive 四大資料夾
  • CODE：Capture / Organize / Distill / Express 四步驟
  • Zettelkasten：用唯一 ID 與雙向連結建立卡片盒筆記

5.4 與 AI 助手協作

LLM 在訓練時讀過數十億份 Markdown 文件，所以它們的「母語」就是 Markdown。實務建議：

🤖
餵給 AI 的最佳格式
  • 把 Word / PDF 先轉成 Markdown 再丟給 AI——準確度遠高於直接傳檔
  • 用 H1 / H2 切分長 prompt，AI 對結構化輸入理解更好
  • 需要 AI 輸出表格？直接要求 GFM table——比 CSV 更方便人類閱讀
  • 程式碼一定要用柵欄碼塊，```语言名 讓 AI 知道上下文

6.1 轉換工具

Pandoc 是 Markdown 轉換的瑞士刀——可在 Markdown ↔ Word ↔ PDF ↔ LaTeX ↔ HTML ↔ EPUB 之間自由轉換：

# Markdown 轉 Word
pandoc input.md -o output.docx

# Markdown 轉 PDF (含中文字體)
pandoc input.md -o output.pdf \
  --pdf-engine=xelatex \
  -V mainfont="Noto Sans CJK TC"

# 多個 Markdown 合併成一份 ePub 電子書
pandoc ch1.md ch2.md ch3.md -o book.epub

7 · 最佳實踐與常見陷阱

7.1 風格指南

  • 標題前後空行：避免某些 parser 把標題誤判為段落內容
  • 清單符號統一：整份文件選 - 或 * 其中一種，不混用
  • 程式碼一律指定語言：方便高亮、方便搜尋、方便 AI 理解
  • 圖片永遠寫 alt 文字：無障礙閱讀必備、SEO 加分
  • 長 URL 用參考式連結：保持正文乾淨可讀

7.2 常見陷阱

🪤
陷阱 1：單行不會換行
Markdown 忽略單一換行——必須空行才會分段，或行尾留兩個空格才會 <br>。這是新手最常踩的雷。
🪤
陷阱 2：底線在單字中行為不一致
un_frigging_believable 在某些 parser 不會被視為斜體。單字中強調請改用星號：un*frigging*believable。
🪤
陷阱 3：清單裡放程式碼塊
4 空格縮排的程式碼塊在清單內會破壞編號。改用柵欄式碼塊 (三反引號) 並縮排對齊清單項目。
🪤
陷阱 4：表格無法跨平台一致
原始 Markdown 沒有表格，GFM 才加入。在 Reddit 等平台可能失效；複雜表格 (合併儲存格) 也只能改用 HTML <table>。
🪤
陷阱 5：HTML 在某些平台會被過濾
Reddit、StackOverflow 出於安全考量會清掉大部分 HTML。若文件要跨多個平台，盡量靠純 Markdown。

7.3 跨平台相容寫法 (Compatibility-First)

當文件要在多個平台流通時，把自己限縮在 CommonMark + 最常用的 GFM (表格、任務清單、刪除線)。把進階功能 (Alert、Mermaid、Math) 留給單一平台的內部文件。

7.4 版本控制好習慣

  • 每行一句：方便 git diff 看出實際改動
  • 不要硬斷行：讓編輯器處理 word wrap，不要手動 80 字元換行
  • Front Matter 鎖定欄位順序：減少無意義的 diff
