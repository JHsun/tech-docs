Mermaid 完整章節

為什麼要用 Mermaid

Mermaid 是一個用純文字描述圖表的開源工具。傳統上要在文件中放圖表，你得開 draw.io、Lucidchart 或 Visio 畫好再匯出 PNG，每次修改都要重新匯出再上傳。Mermaid 把這整個流程「文字化」——你寫幾行 Markdown，渲染器就直接生成可縮放的 SVG 圖。

2022 年 GitHub 在 README 與 Issue 中原生支援 Mermaid，這個格式從此成為技術文件的事實標準。今天 GitHub、GitLab、HackMD、Notion (部分支援)、Obsidian (透過外掛)、Docusaurus、MkDocs 等主流平台都能直接渲染。

💡
Mermaid 的三大優勢
  • 版控友善：純文字，git diff 一目瞭然
  • 易於協作：改一行就改完，不用重新匯出圖片
  • SVG 輸出：可無限縮放，無解析度問題

通用語法規則

所有 Mermaid 圖在 Markdown 中都用相同的格式包裹——程式碼柵欄指定語言為 mermaid：

```mermaid
<圖表類型> <方向或選項>
    <節點與關係的描述>
```

方向關鍵字 (用於 flowchart)

方向意義範例情境
TB / TDTop to Bottom (由上而下)樹狀結構、組織圖
BTBottom to Top繼承關係 (子→父)
LRLeft to Right (由左到右)時間流、流程順序
RLRight to Left少用，特殊閱讀方向

註解

所有 Mermaid 圖使用 %% 開頭的行作為註解，渲染時會被忽略：

%% 這是註解，不會被渲染
flowchart LR
  A --> B

1 · 流程圖 Flowchart

最常用的圖表類型——任何「一個步驟接著下一個步驟」的事都能畫成流程圖。Mermaid 流程圖用 flowchart (新語法) 或 graph (舊語法) 起始，建議用前者。

1.1 基本範例

flowchart LR A[開始] --> B{條件判斷} B -->|是| C[執行動作] B -->|否| D[結束] C --> D
flowchart LR A[開始] --> B{條件判斷} B -->|是| C[執行動作] B -->|否| D[結束] C --> D

1.2 節點形狀

不同的括號代表不同的節點形狀，每種形狀有對應的語意：

flowchart TB A[方形：一般步驟] B(圓角矩形：起終點) C([跑道形：起終點 v2]) D[[子程序]] E[(資料庫)] F((圓形：連接點)) G>旗幟形：標註] H{菱形：判斷} I{{六邊形：準備}} J[/平行四邊形：輸入\] K[\\反平行四邊形：輸出/]
flowchart TB A[方形：一般步驟] B(圓角矩形：起終點) C([跑道形：起終點 v2]) D[[子程序]] E[(資料庫)] F((圓形：連接點)) G>旗幟形：標註] H{菱形：判斷} I{{六邊形：準備}} J[/平行四邊形：輸入\] K[\\反平行四邊形：輸出/]
語法形狀典型用途
A[文字]方形 (Rectangle)一般處理步驟
A(文字)圓角矩形起點 / 終點
A([文字])跑道形 (Stadium)新版起終點
A[[文字]]雙線矩形子程序、模組
A[(文字)]圓柱形資料庫
A((文字))圓形連接點 / 起點
A>文字]旗幟形標籤、標註
A{文字}菱形 (Rhombus)判斷 / 條件分歧
A{{文字}}六邊形 (Hexagon)準備 / 預處理
A[/文字\]平行四邊形輸入

1.3 箭頭與連線

flowchart LR A --> B A --- C A -.-> D A ==> E A --文字--> F A -->|含邊標| G
flowchart LR A --> B A --- C A -.-> D A ==> E A --文字--> F A -->|含邊標| G
語法意義
A --> B實線箭頭
A --- B實線無箭頭
A -.-> B虛線箭頭
A -.- B虛線無箭頭
A ==> B粗線箭頭
A === B粗線無箭頭
A --文字--> B線上加文字 (寫法 1)
A -->|文字| B線上加文字 (寫法 2，較常用)
A <--> B雙向箭頭
A o--o B圓點端點
A x--x B叉號端點

1.4 子圖 (Subgraph)

用 subgraph 把節點分組，常用於畫架構圖：

flowchart TB subgraph 前端 A[React App] B[Vue App] end subgraph 後端 C[API Server] D[(Database)] end A --> C B --> C C --> D
flowchart TB subgraph 前端 A[React App] B[Vue App] end subgraph 後端 C[API Server] D[(Database)] end A --> C B --> C C --> D

1.5 樣式 (classDef)

用 classDef 定義樣式，class 套用到節點。也可以直接 style 節點 屬性：

flowchart LR A[正常] --> B[警告] B --> C[錯誤] classDef warning fill:#fef3c7,stroke:#d97706,color:#000 classDef danger fill:#fee2e2,stroke:#dc2626,color:#000 class B warning class C danger

2 · 序列圖 Sequence Diagram

序列圖描述多個系統間的時序互動——適合展示 API 呼叫流程、認證流程、訊息傳遞等。讀法是由上而下的時間流，左右兩邊是參與者 (actor / participant)。

2.1 基本範例

sequenceDiagram participant U as 使用者 participant W as 網頁前端 participant S as 伺服器 participant D as 資料庫 U->>W: 點擊登入 W->>S: POST /login S->>D: 查詢使用者 D-->>S: 回傳資料 S-->>W: 200 OK + token W-->>U: 顯示首頁
sequenceDiagram participant U as 使用者 participant W as 網頁前端 participant S as 伺服器 participant D as 資料庫 U->>W: 點擊登入 W->>S: POST /login S->>D: 查詢使用者 D-->>S: 回傳資料 S-->>W: 200 OK + token W-->>U: 顯示首頁

2.2 訊息類型

語法意義
A->B實線無箭頭 (少用)
A-->B虛線無箭頭
A->>B實線+箭頭：發送請求
A-->>B虛線+箭頭：回傳結果
A-x B實線+叉號：訊息丟失
A--xB虛線+叉號：回應丟失
A-)B實線+開放箭頭：非同步呼叫

2.3 啟用 / 停用 (Activation)

用 activate 與 deactivate 可以顯示參與者「正在處理中」的時間區段：

sequenceDiagram Alice->>Bob: Hello activate Bob Bob->>Bob: 思考中... Bob-->>Alice: Hi! deactivate Bob
sequenceDiagram Alice->>Bob: Hello activate Bob Bob->>Bob: 思考中... Bob-->>Alice: Hi! deactivate Bob

更簡潔的寫法：在箭頭尾加 + / - 簡寫：A->>+B 等於發送+activate；B-->>-A 等於回應+deactivate。

2.4 控制結構：loop / alt / opt / par / break

sequenceDiagram participant 使用者 participant 系統 使用者->>系統: 嘗試登入 alt 帳密正確 系統-->>使用者: 登入成功 else 帳密錯誤 系統-->>使用者: 401 Unauthorized end opt 啟用兩階段驗證 系統->>使用者: 發送 OTP end loop 每 5 分鐘 系統->>系統: 檢查 session end
sequenceDiagram participant 使用者 participant 系統 使用者->>系統: 嘗試登入 alt 帳密正確 系統-->>使用者: 登入成功 else 帳密錯誤 系統-->>使用者: 401 Unauthorized end opt 啟用兩階段驗證 系統->>使用者: 發送 OTP end loop 每 5 分鐘 系統->>系統: 檢查 session end

2.5 註記 (Note)

Note left of Alice: Alice 發送訊息前
Note right of Bob: Bob 收到訊息後
Note over Alice,Bob: 這個註記跨越兩個參與者

3 · 類別圖 Class Diagram

類別圖描述物件導向程式中的類別結構與關係。Mermaid 支援 UML 標準的繼承、組合、聚合、關聯、依賴等關係。

3.1 基本範例

classDiagram class Animal { +String name +int age +eat() +sleep() } class Dog { +String breed +bark() } class Cat { +Boolean indoor +meow() } Animal <|-- Dog Animal <|-- Cat
classDiagram class Animal { +String name +int age +eat() +sleep() } class Dog { +String breed +bark() } class Cat { +Boolean indoor +meow() } Animal <|-- Dog Animal <|-- Cat

3.2 可見性符號

符號意義
+public
-private
#protected
~package / internal
$ (在類別名後)static
* (在類別名後)abstract

3.3 關係類型

語法關係意義
A <|-- B繼承 (Inheritance)B is-a A
A <|.. B實作介面 (Realization)B 實作 A 介面
A *-- B組合 (Composition)B 是 A 的一部分，A 銷毀時 B 也銷毀
A o-- B聚合 (Aggregation)A 包含 B，但 B 可獨立存在
A --> B關聯 (Association)A 知道 B
A ..> B依賴 (Dependency)A 暫時用到 B
A -- B單純連結—

3.4 多重性 (Cardinality)

在關係兩端用引號標註多重性："1"、"0..1"、"*"、"1..*"、"n"

classDiagram Customer "1" --> "*" Order : places Order "*" --> "1..*" Product : contains
classDiagram Customer "1" --> "*" Order : places Order "*" --> "1..*" Product : contains

4 · 狀態圖 State Diagram

狀態圖描述物件在生命週期中的狀態轉換——適合畫狀態機 (FSM)、UI 狀態、訂單流程等。Mermaid 用 stateDiagram-v2 (新版，建議用此)。

4.1 基本範例

stateDiagram-v2 [*] --> Idle Idle --> Loading: 點擊載入 Loading --> Success: 載入成功 Loading --> Error: 載入失敗 Success --> Idle: 重置 Error --> Idle: 重試 Success --> [*]
stateDiagram-v2 [*] --> Idle Idle --> Loading: 點擊載入 Loading --> Success: 載入成功 Loading --> Error: 載入失敗 Success --> Idle: 重置 Error --> Idle: 重試 Success --> [*]

[*] 表示起點 (進入箭頭) 與終點 (離開箭頭)。

4.2 組合狀態 (Composite State)

stateDiagram-v2 [*] --> FirstStep state Active { [*] --> Running Running --> Paused Paused --> Running Running --> [*] } FirstStep --> Active Active --> [*]
stateDiagram-v2 [*] --> FirstStep state Active { [*] --> Running Running --> Paused Paused --> Running Running --> [*] } FirstStep --> Active Active --> [*]

4.3 平行狀態 (Fork & Join)

stateDiagram-v2 state fork_state <<fork>> state join_state <<join>> [*] --> fork_state fork_state --> State1 fork_state --> State2 State1 --> join_state State2 --> join_state join_state --> [*]
stateDiagram-v2 state fork_state <<fork>> state join_state <<join>> [*] --> fork_state fork_state --> State1 fork_state --> State2 State1 --> join_state State2 --> join_state join_state --> [*]

5 · ER 圖 (Entity-Relationship Diagram)

ER 圖描述資料庫的實體與關聯。每個實體 (entity) 有屬性 (attribute)，實體之間透過關係 (relationship) 連結，用 crow's foot 符號表示多重性。

5.1 基本範例

erDiagram CUSTOMER ||--o{ ORDER : places ORDER ||--|{ LINE-ITEM : contains PRODUCT ||--o{ LINE-ITEM : "ordered in" CUSTOMER { string id PK string name string email } ORDER { string id PK date created_at string status } LINE-ITEM { string id PK string order_id FK string product_id FK int quantity decimal price } PRODUCT { string id PK string name decimal price }
erDiagram CUSTOMER ||--o{ ORDER : places ORDER ||--|{ LINE-ITEM : contains PRODUCT ||--o{ LINE-ITEM : "ordered in" CUSTOMER { string id PK string name string email } ORDER { string id PK date created_at string status } LINE-ITEM { string id PK string order_id FK string product_id FK int quantity decimal price } PRODUCT { string id PK string name decimal price }

5.2 關係符號 (Crow's Foot Notation)

左符號 + 中間 (連線) + 右符號：

符號 (左)意義 (左側)符號 (右)意義 (右側)
|o0 或 1 (零或一)o|0 或 1
||1 且只有 1 (恰好一)||恰好一
}o0 或多 (零或多)o{零或多
}|1 或多 (一或多)|{一或多

例：CUSTOMER ||--o{ ORDER 表示「一個客戶可以有零或多個訂單」。

6 · 甘特圖 Gantt Chart

甘特圖用於專案時程規劃，展示任務的起迄時間、進度、依賴關係。

6.1 基本範例

gantt title 專案時程規劃 dateFormat YYYY-MM-DD axisFormat %m/%d section 規劃階段 需求收集 :a1, 2026-05-01, 7d 技術評估 :a2, after a1, 5d section 開發階段 後端 API :b1, after a2, 14d 前端 UI :b2, after a2, 21d 整合測試 :b3, after b1, 7d section 上線 UAT 驗收 :milestone, m1, 2026-06-15, 0d 正式上線 :milestone, m2, 2026-06-22, 0d
gantt title 專案時程規劃 dateFormat YYYY-MM-DD axisFormat %m/%d section 規劃階段 需求收集 :a1, 2026-05-01, 7d 技術評估 :a2, after a1, 5d section 開發階段 後端 API :b1, after a2, 14d 前端 UI :b2, after a2, 21d 整合測試 :b3, after b1, 7d section 上線 UAT 驗收 :milestone, m1, 2026-06-15, 0d 正式上線 :milestone, m2, 2026-06-22, 0d

6.2 任務語法

任務名稱 :狀態, 識別碼, 開始時間, 結束時間或天數
狀態關鍵字說明
(空)正常待辦
active進行中 (突出顯示)
done已完成
crit關鍵任務 (紅色標示)
milestone里程碑 (顯示為菱形)

結束時間有兩種寫法：2026-05-08 直接給日期，或 5d 給持續時間。依賴用 after id1 或 after id1, id2。

13 · C4 架構圖 (Context Diagram)

C4 模型由 Simon Brown 提出，用 4 個層級 (Context、Container、Component、Code) 描述軟體架構。Mermaid 支援前 3 層：

C4Context title 線上書店系統架構 Person(customer, "顧客", "瀏覽與購買書籍") System(bookstore, "線上書店", "提供瀏覽、訂購、付款") System_Ext(payment, "金流系統", "外部第三方") System_Ext(email, "Email 系統", "通知服務") Rel(customer, bookstore, "使用") Rel(bookstore, payment, "付款請求") Rel(bookstore, email, "寄送通知")
C4Context title 線上書店系統架構 Person(customer, "顧客", "瀏覽與購買書籍") System(bookstore, "線上書店", "提供瀏覽、訂購、付款") System_Ext(payment, "金流系統", "外部第三方") System_Ext(email, "Email 系統", "通知服務") Rel(customer, bookstore, "使用") Rel(bookstore, payment, "付款請求") Rel(bookstore, email, "寄送通知")
⚠️
C4 在 GitHub 尚為實驗性
C4 圖在 Mermaid 中仍標記為 experimental。在 GitHub 渲染可能有少數細節差異，但基本可用。

LaTeX 數學公式完整章節

前言：MathJax vs KaTeX

當你在 Markdown 中寫 $E = mc^2$ 時，背後負責把它渲染成漂亮數學式的，是兩大引擎之一：MathJax 或 KaTeX。它們都接受 LaTeX 語法，但實作差異會影響你能用什麼指令。

面向MathJaxKaTeX
語法支援近乎完整 LaTeX子集 (90%+ 常用功能)
渲染速度較慢 (50-200ms)快 (10-30ms)
檔案大小較大較小，更輕量
輸出格式HTML+CSS / SVG / MathMLHTML+CSS
同步渲染非同步同步 (無閃爍)
錯誤處理較寬容較嚴格 (預設拋錯)
代表使用者GitHub、ObsidianHackMD、GitLab、本頁

對 95% 的使用者而言，KaTeX 與 MathJax 的差別你不會感受到——兩者都能渲染常見的數學式。但若你需要某些罕見指令 (例如 \rgroup、\unitlength 等)，可能要切到 MathJax。

顏色支援度

常見錯誤與相容性陷阱

陷阱 1：忘記 {} 大括號

x^10 → $x^10$ (錯誤)；x^{10} → $x^{10}$ (正確)。

陷阱 2：行內公式緊鄰其他字元

有些 parser 對 $ 周圍空白要求嚴格。如果寫 $abc$xyz 渲染失敗，改成 $abc$ xyz。

陷阱 3：dollar 符號被誤解為金額

當文件混合金額與公式時，金額 $5 可能被誤判為公式起點。建議：金額用 \$5 跳脫，或文章專心用 $$...$$ 區塊式。

陷阱 4：KaTeX 未支援的 MathJax 指令

少數 LaTeX 指令 KaTeX 不支援。如果你的文件目標是 GitHub (MathJax) 但用 KaTeX 預覽，可能在以下指令踢到鐵板：\rgroup、\unitlength、\rule 部分用法、舊式 \over。建議優先用 \frac、\left(...\right) 等通用語法。

陷阱 5：跨平台行為差異

部分平台 (Notion、舊版 HackMD) 的 $...$ 必須前後有空白才會啟用，否則會被視為文字。寫成 請看 $E=mc^2$ 公式 比 請看$E=mc^2$公式 更穩。
