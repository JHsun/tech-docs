CHAPTER 01

什麼是 Hermes Agent？初階

設計哲學、核心差異、以及它能為你做什麼

核心設計哲學

Hermes Agent 由 Nous Research（Hermes、Nomos、Psyche 模型家族的開發者）於 2026 年 2 月開源發布。它不是一個聊天機器人的包裝，也不是綁在 IDE 上的程式碼副駕駛（Coding Copilot）。

它的核心命題是：「一個會持續學習、長期居住在你的伺服器上的 AI Agent」。每次對話都會累積記憶，每次解決複雜問題都可能自動建立可重用的 Skill，越用越聰明。

🧠
自我學習迴圈
從對話中建立 Skills、固化記憶、跨 Session 持續累積
🌐
模型無關 (Model-Agnostic)
支援 200+ 模型，切換 Provider 不需改程式碼
📡
多平台入口
Telegram / Discord / Slack 等 18 個平台，一個 Gateway
🔧
47 個內建工具
終端機、瀏覽器、網路搜尋、MCP、子 Agent 委派等
💾
完全自架 (Self-hosted)
資料留在本機，無遙測、無雲端鎖定
⚡
MIT 開源授權
可自由修改、商用、貢獻社群

與其他工具的差異

工具定位記憶自我改善部署方式
Hermes Agent自主 Agent（長期運行）✅ 跨 Session✅ 自動建立 Skills任意伺服器 / VPS
Claude CodeIDE 程式碼助理❌ Session 結束即失憶❌本機 CLI
Cursor / WindsurfIDE 副駕駛△ 有限的 Rules❌IDE 內
ChatGPT通用聊天機器人△ 有限記憶❌OpenAI 雲端
OpenClaw個人 Agent（手動管理）✅❌ 需手動撰寫 Skills本機

適合的使用情境

  • 需要一個「長期了解你工作環境」的個人 AI 助理
  • 部署 Telegram / Discord Bot，讓 AI 在背景執行自動化任務
  • 需要跨工作階段（Session）保持上下文的複雜研究專案
  • 想要訓練自訂工具呼叫模型的 ML 研究者
  • 想要完全掌控資料且不依賴雲端服務的隱私敏感應用
CHAPTER 02

安裝與環境設定初階

支援平台、系統需求、一行安裝指令

系統需求

ℹ️
必要條件：64K Context

Hermes Agent 需要至少 64,000 tokens context 的模型。這是 Multi-step Tool Calling 的最低工作記憶需求。不足的模型會在啟動時被拒絕。大多數主流雲端模型（Claude、GPT、Gemini、Qwen）都符合要求。

平台支援狀態備註
Linux (x86_64)✅ 完整支援推薦生產環境
macOS (Intel / Apple Silicon)✅ 完整支援Homebrew 可用
WSL2 (Windows)✅ 完整支援Windows 原生不支援
Android / Termux△ 部分支援Voice 相依套件有限制
NixOS✅ 支援有專用 Nix flake

一行安裝指令

1
執行安裝腳本

自動偵測平台、安裝依賴、設定 PATH

bash
curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
2
重新載入 Shell 環境
bash
# 根據你的 Shell 選擇
source ~/.bashrc   # Bash
source ~/.zshrc    # Zsh
3
執行互動式設定精靈
bash
hermes setup
4
驗證安裝成功
bash
hermes doctor   # 診斷設定問題
hermes dump     # 顯示完整系統狀態

設定檔結構

安裝完成後，Hermes 在 ~/.hermes/ 建立以下目錄結構：

~/.hermes/
├── config.yaml # 所有非機密設定（模型、終端機後端等）
├── .env # API Keys 與機密資訊
├── auth.json # OAuth 憑證（Nous Portal 等）
├── SOUL.md # Agent 人格定義（System Prompt 第一個）
├── memories/ # MEMORY.md、USER.md
├── skills/ # Agent 自動建立的技能文件
├── cron/ # 排程任務 jobs.json
├── sessions/ # Gateway Session 資料
└── logs/ # 錯誤與 Gateway 日誌（自動遮蔽機密）
💡
機密與設定分離

Hermes 自動判斷設定值該存到哪個檔案。執行 hermes config set OPENROUTER_API_KEY sk-xxx 時，API Key 自動存入 .env；模型名稱等非機密值存入 config.yaml。

CHAPTER 04

常見錯誤排除 Troubleshooting初階

最常見的卡關問題與系統化排除流程

診斷工具優先

遇到問題時，請依以下順序執行診斷指令，從系統狀態到逐步縮小問題範圍：

hermes doctor          # 自動診斷設定問題
hermes model           # 重新確認 Provider / 模型設定
hermes setup           # 重新走一次設定精靈
hermes sessions list   # 確認 Session 是否存在
hermes dump            # 輸出完整系統狀態（貼到 Issue 用）

常見問題對照表

症狀可能原因解法
開啟後回覆空白或亂碼Provider 驗證失敗或模型選錯重新執行 hermes model
自訂 Endpoint 能連但回傳怪異內容URL、模型名稱或格式不相容先用 curl 直接測試 Endpoint
Gateway 啟動但收不到訊息Bot Token、白名單或平台設定不完整重新執行 hermes gateway setup
--continue 找不到舊 Session切換了 Profile 或 Session 未正常儲存執行 hermes sessions list 確認
模型 Context 不足警告本機模型未設定足夠 Context 長度加上 --ctx-size 65536 參數
hermes doctor 顯示設定問題更新後設定欄位不相容執行 hermes config migrate

本機模型（Ollama）常見問題

# 確認模型 context 大小設定
# 在 Modelfile 中加入：
PARAMETER num_ctx 65536

# 或啟動時指定
ollama run qwen3:8b --ctx-size 65536
⚠️
黃金法則：先讓基礎對話正常

如果 Hermes 連普通對話都無法完成，不要先加 Gateway、Cron、Skills 等進階功能。逐層確認：Provider 正常 → 基礎對話正常 → Session 持久化正常 → 再加入下一層功能。

CHAPTER 06

記憶系統 Memory System中階

MEMORY.md、跨 Session 持久化、FTS5 回溯與 Honcho 建模

記憶的三個層次

📄
MEMORY.md
事實記憶：你的環境、偏好、專案位置、Agent 學到關於你的事
👤
USER.md
使用者畫像：你的工作方式、溝通風格、長期偏好
🗄️
Session DB (SQLite + FTS5)
所有對話的全文搜尋索引，讓 Agent 能回溯過去的討論

記憶 vs. 技能：分工原則

✅ 存入 Memory（MEMORY.md）

「我的 Django 專案在 ~/projects/api」
「我偏好使用 TypeScript 而非 JavaScript」
「伺服器 IP 是 192.168.1.100」

📚 建立成 Skill

「如何部署到 staging 環境（5 個步驟）」
「處理 Git conflict 的標準流程」
「跑完整測試套件的指令組合」

法則：Memory 記「是什麼」，Skills 記「怎麼做」。

手動管理記憶

# 查看當前記憶內容
hermes memory show

# 直接編輯記憶檔案
hermes memory edit

# 清除記憶
hermes memory clear

Honcho 進階使用者建模（選配）

Honcho 是 Plastic Labs 開發的對話式使用者建模系統。啟用後，Hermes 會透過辯證式（Dialectic）推論主動更新對你的理解，而不只是被動記錄。

# 在 config.yaml 中啟用
memory:
  provider: honcho

# 管理 Honcho 狀態
hermes honcho status
hermes honcho reset

記憶設定調整

# config.yaml
memory:
  memory_enabled: true
  user_profile_enabled: true
  memory_char_limit: 2200    # ~800 tokens
  user_char_limit: 1375      # ~500 tokens
ℹ️
記憶大小限制的考量

記憶內容會注入到每一個 System Prompt 中，直接消耗 Token Budget。memory_char_limit 限制大小是刻意的設計，建議保持簡潔、聚焦重要事實。

CHAPTER 07

技能系統 Skills System中階

自動建立流程、Progressive Disclosure、Skills Hub

Skill 是什麼？

Skill 是一個 Markdown 格式的「程序性知識文件」，描述如何完成某件事。Agent 在解決複雜問題後可以自動建立 Skill，下次遇到相同問題時直接載入，不需要重新摸索。

所有 Skill 文件存放在 ~/.hermes/skills/，相容 agentskills.io 開放標準，可跨 Agent 分享與複用。

Progressive Disclosure：兩層載入機制

為了節省 Token 消耗，Skill 採用兩層載入機制：

層級內容Token 消耗時機
Level 0所有 Skill 的名稱與一行描述~3,000 tokens每次對話開始時自動載入
Level 1指定 Skill 的完整內容依 Skill 大小而定Agent 判斷需要時才按需載入

自動建立 Skill

# 在對話中直接要求 Agent 建立 Skill
Save what you just did as a skill called deploy-staging.

# 下次直接呼叫
/deploy-staging

安裝社群 Skills

# 搜尋相關 Skills
hermes skills search kubernetes
hermes skills search react --source skills-sh

# 安裝 Skill
hermes skills install official/migration/openclaw-migration
hermes skills install skills-sh/vercel-labs/json-render/json-render-react

# 查看已安裝的 Skills
hermes skills list

# 更新全部 Skills
hermes skills update

手動建立 Skill（進階）

直接在 ~/.hermes/skills/ 建立 .md 文件即可。建議格式：

---
name: deploy-staging
description: 部署到 staging 環境的完整流程
---

## 步驟
1. 確認 git status 乾淨
2. 執行 `npm run build`
3. 透過 SSH 部署到 192.168.1.100
4. 執行 smoke test：`curl https://staging.example.com/health`
💡
5 步驟法則

如果某個任務需要 5 個以上步驟，且你未來會再做一次，就值得建立成 Skill。讓 Agent 幫你記住「怎麼做」，你只需要記得「叫它做」。

Gateway 安全建議

部署為 Bot 時，建議對 Telegram / Discord 等平台不要開放終端機工具，避免未授權使用者執行任意系統指令。使用 terminal.backend: docker 可提供額外的隔離保護。

# config.yaml
agent:
  approval_mode: auto     # 自動批准（本機開發用）
  approval_mode: always   # 永遠詢問（生產環境建議）

Context Files 與 SOUL.md中階

專案情境注入、Agent 人格定義

自動載入的 Context Files

Hermes 在啟動時會從當前工作目錄自動發現以下檔案，注入到 System Prompt 中：

檔案名稱用途相容性
.hermes.mdHermes 專用專案設定Hermes
AGENTS.md專案架構、編碼規範、Agent 指示通用（Hermes / Claude Code / Cursor）
CLAUDE.md同 AGENTS.md（Claude 系工具）Claude Code 相容
.cursorrulesCursor 規則（Hermes 也支援）Cursor 相容
SOUL.mdAgent 全域人格定義Hermes 專用
💡
Context Files 的 Token 成本

Context Files 會注入到每一條訊息的 System Prompt，每個字元都直接消耗 Token Budget。建議保持簡潔，只寫 Agent 真正需要的資訊。

AGENTS.md 撰寫範例

# My Project — Agent Instructions

## 架構
- Backend: Django REST Framework (Python 3.11)
- Frontend: Next.js 14 + TypeScript
- Database: PostgreSQL 15

## 編碼規範
- 使用 `ruff` 格式化 Python 程式碼
- TypeScript 嚴格模式，禁用 `any`
- 所有 PR 需通過 `pytest` 完整測試套件

## 常用路徑
- API: ~/projects/api/
- Frontend: ~/projects/web/
- 部署腳本: ~/scripts/deploy.sh

SOUL.md：定義 Agent 人格

SOUL.md（位於 ~/.hermes/SOUL.md）是 System Prompt 的第一個區塊，定義 Agent 的核心行為模式、語氣、與限制。 

# SOUL.md 範例

你是一個專注於後端開發的技術助理。
- 回答要簡潔直接，避免不必要的解釋
- 程式碼優先，文字說明補充
- 遇到不確定的事情，明說不確定而非猜測
- 繁體中文溝通，技術術語保持英文

切換臨時人格

# 對話中切換（不影響 SOUL.md）
/personality formal     # 正式語氣
/personality pirate     # 有趣的海盜模式
/personality concise    # 極簡回應

CHAPTER 10

訊息平台整合 Gateway Setup中階

Telegram / Discord Bot 設定、授權機制、多使用者管理

Gateway 架構

Gateway 是一個長期運行的背景行程（Daemon），負責連接 18 個訊息平台。所有平台共享同一個 Agent 實例，透過統一的 Session 路由機制分開管理每個使用者的對話。

設定流程（以 Telegram 為例）

1
建立 Telegram Bot

在 Telegram 中搜尋 @BotFather，執行 /newbot 取得 Bot Token。

2
執行互動式設定 
hermes gateway setup
3
設定使用者白名單 
# 取得你的 Telegram User ID（傳訊息給 @userinfobot）
# 設定到 .env
TELEGRAM_ALLOWED_USERS=123456789,987654321

# 或使用跨平台白名單
GATEWAY_ALLOWED_USERS=123456789
4
啟動 Gateway 
hermes gateway start           # 前景執行（測試用）
hermes gateway install         # 安裝為 systemd 使用者服務
sudo hermes gateway install --system  # 開機自動啟動
5
確認運行狀態 
hermes gateway status
hermes logs --follow

MCP Server 整合中階

Model Context Protocol、設定方式、工具過濾與安全

什麼是 MCP？

MCP（Model Context Protocol）是 Anthropic 提出的開放標準，讓 AI Agent 能透過統一介面連接外部工具和資料來源。Hermes 支援所有符合 MCP 標準的 Server，無需修改核心程式碼。

在 config.yaml 設定 MCP Server

# ~/.hermes/config.yaml

mcp_servers:
  # GitHub 整合
  github:
    command: npx
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "ghp_xxxxxxxx"

  # 本機檔案系統（限制路徑）
  filesystem:
    command: npx
    args: ["-y", "@modelcontextprotocol/server-filesystem",
           "/home/user/projects"]

  # PostgreSQL 資料庫
  postgres:
    command: npx
    args: ["-y", "@modelcontextprotocol/server-postgres"]
    env:
      POSTGRES_URL: "postgresql://user:pass@localhost:5432/mydb"

  # 自訂 HTTP MCP Server
  my-internal-api:
    type: http
    url: "https://mcp.internal.company.com/sse"

工具過濾：只開放必要工具

# config.yaml — 只允許特定 MCP 工具
mcp_servers:
  github:
    command: npx
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: "ghp_xxx"
    # 只開放這些工具（白名單）
    allowed_tools:
      - create_issue
      - list_issues
      - get_pull_request
    # 或者排除特定工具（黑名單）
    # denied_tools:
    #   - delete_repository

管理已設定的 MCP Servers

hermes mcp list           # 列出所有 MCP Server

hermes mcp tools github   # 列出 github MCP 提供的所有工具
⚠️
MCP 安全注意事項

MCP Server 工具會直接暴露給 Agent 執行。建議：
(1) 使用 allowed_tools 白名單限制工具範圍
(2) 資料庫 MCP 使用唯讀帳號
(3) 在 Gateway 模式下謹慎開放破壞性操作

排程自動化 Cron Jobs中階

自然語言排程、jobs.json 結構、跨平台輸出

建立排程任務

Cron Jobs 支援三種時間格式：自然語言、標準 Cron 表達式、以及一次性 ISO 8601 時間：

# 自然語言格式（最直覺）
hermes cron create "every 2h" "Check server status and report"
hermes cron create "every 1h" "Summarize HN top posts" --skill blogwatcher
hermes cron create "0 9 * * 1-5" "Send daily standup reminder to Slack"

# 在對話中建立（更自然）
/cron add "每天早上 9 點" "檢查 Hacker News AI 相關新聞並傳到 Telegram"

支援的時間格式

格式範例說明
相對時間（一次性）30m、2h、1d30 分鐘後執行一次
重複自然語言every 30m、every 2h每隔固定時間重複
Cron 表達式0 9 * * *每天 9:00 執行
ISO 86012026-06-01T09:00:00指定日期時間執行一次

Cron Job 管理指令

hermes cron list              # 查看所有任務
hermes cron status            # 顯示排程器狀態
hermes cron pause <job_id>   # 暫停
hermes cron resume <job_id>  # 恢復
hermes cron run <job_id>     # 立即執行一次（測試用）
hermes cron edit <job_id> --schedule "every 4h"  # 修改排程
hermes cron remove <job_id>  # 刪除

附加 Skills 的 Cron Job

在 Cron Job 執行前，可以先注入 Skills 作為 Context，讓每次執行都具備特定知識：

# 附加一個 Skill
hermes cron create "every 1h" "Summarize new items" --skill blogwatcher

# 附加多個 Skills
hermes cron create "every 2h" "Generate report" \
  --skill data-analysis \
  --skill report-template

Cron Job 執行機制

每次 Cron Job 執行時，Gateway 會建立一個全新的 AIAgent Session（沒有任何對話歷史），注入指定的 Skills，然後執行任務。輸出結果自動傳送到設定的目標平台（如 Home Channel）。

🚫
Cron Job 的限制

Cron Job 中禁用了 cronjob 工具本身，防止排程任務遞迴建立更多排程任務，避免失控的排程迴圈。

CLI Session 資料流

使用者輸入
  → HermesCLI.process_input()
    → AIAgent.run_conversation()
      → prompt_builder.build_system_prompt()
          # 組裝：SOUL.md + MEMORY.md + Skills Level 0
          #       + Context Files + Tool 說明 + 歷史對話
      → runtime_provider.resolve_runtime_provider()
          # 解析：Provider → API Mode + API Key + Base URL
      → API Call (chat_completions / codex_responses / anthropic_messages)
      → tool_calls? → model_tools.handle_function_call() → 迴圈
      → 最終回應 → 顯示 → 存入 SessionDB (SQLite)

Prompt Assembly 組裝順序

prompt_builder.py 按以下順序組裝 System Prompt，順序決定了 Token 快取的效率：

槽位內容來源
Slot 1人格定義SOUL.md
Slot 2使用者記憶MEMORY.md
Slot 3使用者畫像USER.md
Slot 4Skills Level 0（名稱清單）~/.hermes/skills/
Slot 5Context FilesAGENTS.md、.hermes.md 等
Slot 6工具說明Tool Registry
Slot 7模型特定指示model_metadata.py

Context Compression

當對話長度接近模型的 Context 上限時，context_compressor.py 會對中間的對話輪次進行有損摘要（Lossy Summarization），保留頭尾的完整對話：

# config.yaml
compression:
  threshold: 0.85    # Context 使用率達 85% 時觸發壓縮
  strategy: middle   # 壓縮中間部分，保留頭尾

Provider Runtime Resolution

# 三種 API Mode
# 1. chat_completions  — OpenAI 相容格式（最通用）
# 2. codex_responses   — OpenAI Responses API
# 3. anthropic_messages — Anthropic Messages API 原生格式
CHAPTER 15

自訂工具開發 Adding Tools進階

Tool 介面規格、Registry 自動註冊、危險指令偵測

工具系統架構

每個工具是一個 Python 函數，透過 tools/registry.py 集中管理。工具文件在 import 時自動完成自我註冊（Self-registration），無需修改核心程式碼。

建立一個最小工具

# tools/my_tool.py

from tools.registry import register_tool
from typing import Any

@register_tool(
    name="my_custom_tool",
    description="說明這個工具做什麼，讓 Agent 知道何時使用它",
    toolset="custom",  # 所屬工具組
    parameters={
        "type": "object",
        "properties": {
            "input_text": {
                "type": "string",
                "description": "要處理的文字"
            },
            "mode": {
                "type": "string",
                "enum": ["fast", "thorough"],
                "description": "處理模式"
            }
        },
        "required": ["input_text"]
    }
)
def my_custom_tool(input_text: str, mode: str = "fast") -> dict[str, Any]:
    """實際執行邏輯"""
    result = process_something(input_text, mode)
    return {
        "success": True,
        "result": result
    }

在 toolsets.py 中加入新工具組

# toolsets.py — 加入自訂工具組
TOOLSET_DEFINITIONS = {
    # ... 現有工具組 ...
    "custom": {
        "description": "My custom tools",
        "tools": ["my_custom_tool"],
        "platforms": ["cli", "telegram"]  # 只在這些平台啟用
    }
}

危險指令偵測

如果你的工具執行可能有破壞性的操作，應在 tools/approval.py 中加入對應的偵測規則：

# tools/approval.py 的偵測邏輯示意
DANGEROUS_PATTERNS = [
    r"rm\s+-rf",          # 刪除指令
    r"DROP\s+TABLE",      # SQL 刪表
    r"kubectl\s+delete",  # K8s 刪除
    # 加入你自訂工具的危險操作...
]

透過 Plugin 系統加入工具（不修改核心）

# ~/.hermes/plugins/my_plugin/plugin.py

from hermes_cli.plugins import PluginContext

def register(ctx: PluginContext):
    """Plugin 入口點"""

    @ctx.tool(
        name="my_plugin_tool",
        description="來自 Plugin 的工具"
    )
    def my_tool(query: str) -> str:
        return f"Plugin processed: {query}"

實作一個 Logging Hook 範例

# ~/.hermes/plugins/audit_logger/plugin.py

from hermes_cli.plugins import PluginContext
from datetime import datetime
import json

def register(ctx: PluginContext):

    @ctx.hook("on_message_received")
    def log_incoming(event):
        entry = {
            "ts": datetime.utcnow().isoformat(),
            "platform": event.platform,
            "user_id": event.user_id,
            "message_length": len(event.text)
        }
        with open("/var/log/hermes_audit.jsonl", "a") as f:
            f.write(json.dumps(entry) + "\n")

    @ctx.hook("on_tool_call")
    def intercept_dangerous_tool(event):
        if event.tool_name in ["terminal", "write_file"]:
            # 可以在這裡加入額外的授權檢查
            pass
