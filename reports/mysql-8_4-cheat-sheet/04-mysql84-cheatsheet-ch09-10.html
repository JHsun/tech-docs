Backup and Recovery — 備份與還原

🟢 備份類型總覽
分類物理備份 (Physical)邏輯備份 (Logical)
內容原始檔案複製（.ibd / ibdata / redo log）SQL 語句（CREATE + INSERT）或分隔文字
速度快（尤其還原）較慢（需重新執行 SQL）
可移植性受限同平台/版本高（跨平台/版本）
工具MEB / cp / rsync / LVM snapshotmysqldump / MySQL Shell dump
備份模式伺服器狀態寫入影響
Hot（線上）執行中InnoDB DML 可繼續
Warm（溫）執行中但鎖定全域唯讀
Cold（離線）已停止無法存取
🚨mysqlpump 已在 MySQL 8.4.0 移除。替代方案為 MySQL Shell dump 工具或 mysqldump。
🟢 mysqldump 備份與還原
# 生產環境推薦備份（InnoDB 一致快照，不鎖表）
mysqldump -u root -p \
  --single-transaction \
  --source-data=2 \
  --routines --events --triggers \
  --hex-blob \
  --all-databases | gzip > full_$(date +%F).sql.gz

# 還原
gunzip < full_2025-01-15.sql.gz | mysql -u root -p
⚠️--source-data 取代已廢棄的 --master-data。--single-transaction 期間禁止 DDL（ALTER/DROP/TRUNCATE TABLE）。
🟢 MySQL Shell 平行 Dump / Load（推薦）
// 全實例匯出（多執行緒 + zstd 壓縮）
util.dumpInstance("/backup/dump")

// 指定 Schema
util.dumpSchemas(["mydb", "sakila"], "/backup/dump")

// 指定表
util.dumpTables("mydb", ["orders", "items"], "/backup/dump")

// 匯入（平行載入）
util.loadDump("/backup/dump")
選項預設說明
threads4平行執行緒數
consistenttrueLOCK INSTANCE FOR BACKUP 確保一致性
compressionzstd壓縮演算法
chunkingtrue大表自動切片（預設 32MB/chunk）
deferTableIndexesfulltext延遲建立索引加速匯入
💡MySQL Shell dump 比 mysqldump 快數倍：多執行緒、內建壓縮、大表自動切片、中斷可恢復、支援雲端（OCI/S3/Azure）。
🔶 Point-in-Time Recovery（PITR）
# 場景：14:06 誤執行 DROP TABLE，需恢復至 14:05

# Step 1：還原最近的全量備份
mysql -u root -p < full_backup.sql

# Step 2：定位誤操作的確切 binlog 位置
mysqlbinlog --start-datetime="2025-01-15 14:05:00" \
            --stop-datetime="2025-01-15 14:08:00" \
            --verbose binlog.000042 | grep -C 15 "DROP TABLE"
# 發現 DROP 在 position 232，end_log_pos 355

# Step 3：重播至誤操作之前
mysqlbinlog --start-position=155 --stop-position=232 binlog.000042 | mysql -u root -p

# Step 4：跳過誤操作，繼續重播後續
mysqlbinlog --start-position=355 binlog.000042 | mysql -u root -p
⚠️優先用 --start-position 而非 datetime（多事件可能共享時間戳）。多個 binlog 檔案必須在單一連線處理。
🔶 LOCK INSTANCE FOR BACKUP vs FLUSH TABLES WITH READ LOCK
特性FLUSH TABLES WITH READ LOCKLOCK INSTANCE FOR BACKUP
權限RELOADBACKUP_ADMIN
InnoDB DML❌ 全部阻塞✅ 允許繼續
DDL❌ 阻塞❌ 阻塞
適用場景檔案系統快照線上備份工具（推薦）
🔶 可攜式表空間（Transportable Tablespaces）
-- 目標實例：建表後丟棄表空間
CREATE TABLE t1 (c1 INT) ENGINE=InnoDB;
ALTER TABLE t1 DISCARD TABLESPACE;

-- 來源實例：凍結並複製檔案
FLUSH TABLES t1 FOR EXPORT;
-- 複製 t1.ibd 和 t1.cfg 到目標（保持連線開啟）
UNLOCK TABLES;

-- 目標實例：匯入
ALTER TABLE t1 IMPORT TABLESPACE;

僅適用 innodb_file_per_table 表空間。來源與目標 innodb_page_size 必須一致。

🔶 InnoDB 崩潰恢復與 innodb_force_recovery

正常崩潰恢復只需重啟 MySQL。流程：表空間探索 → Redo log 重播 → 未完成交易回滾（背景平行）→ Change buffer 合併。

等級名稱行為安全性
0正常正常啟動✅
1IGNORE_CORRUPT忽略損壞頁面✅
2NO_BACKGROUND停止 master/purge thread✅
3NO_TRX_UNDO不執行交易回滾✅
4NO_IBUF_MERGE不合併 insert buffer；唯讀⚠️
5NO_UNDO_LOG_SCAN未完成交易視為已提交⚠️
6NO_LOG_REDO不做 redo log 重播⛔
# 在 my.cnf 設定後重啟
[mysqld]
innodb_force_recovery = 1   # 從 1 開始逐步提升

等級 > 0 時 INSERT/UPDATE/DELETE 被阻塞。等級 ≤ 3 可 SELECT + DROP TABLE 匯出資料。

🟢 備份最佳實務
  • 始終啟用 binary log（8.4 預設啟用）
  • 定期全量 + 增量（FLUSH LOGS 輪替 binlog）
  • InnoDB 用 --single-transaction
  • 優先用 MySQL Shell dump 工具取代 mysqldump
  • 在 Replica 上備份以避免影響主庫
  • Binlog 存放於與資料檔不同的磁碟
  • 定期驗證備份完整性（實際還原測試）

Optimization — 效能優化

🟢 EXPLAIN 查詢計畫
EXPLAIN SELECT ...;                     -- 傳統表格
EXPLAIN FORMAT=JSON SELECT ...;         -- JSON（含成本）
EXPLAIN FORMAT=TREE SELECT ...;         -- 樹狀（顯示 hash join）
EXPLAIN ANALYZE SELECT ...;              -- 實際執行 + 真實統計
🟢 EXPLAIN type 欄位（最優→最差）
type含義
system表僅一列（系統表）
constPK/UNIQUE 全部與常數比較，最多一列
eq_ref前表每列讀取一列（PK/UNIQUE NOT NULL 全使用）
ref索引最左前綴匹配（非唯一）
range索引範圍掃描（BETWEEN / < / > / IN）
index全索引掃描
ALL全表掃描（最差）
🔶 EXPLAIN Extra 欄重要值
值意義性質
Using index覆蓋索引，僅讀索引樹✅ 好
Using index conditionICP 條件下推至儲存引擎✅ 好
Using whereWHERE 在 Server 層過濾中性
Using filesort額外排序（無法從索引解決）⚠️
Using temporary需臨時表⚠️
Using join buffer (hash join)Hash Join中性
Select tables optimized away僅索引/聚合即可⚡ 最佳
🔶 EXPLAIN ANALYZE 解讀
-> Inner hash join (t2.c2 = t1.c1)  (cost=4.70 rows=6)
                                     (actual time=0.032..0.035 rows=6 loops=1)

cost/rows = 優化器估計值
actual time=A..B  A=第一列 B=最後一列（毫秒）
rows = 實際返回列數
loops = 執行次數

估計與實際差距大時，執行 ANALYZE TABLE 更新索引統計。

🟢 索引類型與策略
索引類型支援操作說明
B-Tree（InnoDB 預設）= > < BETWEEN LIKE 'prefix%'最左前綴匹配 + ORDER BY
Hash（僅 MEMORY）僅 = <=>等值極快，不支援範圍/排序
FulltextMATCH ... AGAINST全文搜尋
Spatial空間函式GIS 資料
🟢 複合索引最左前綴規則

索引 (a, b, c) 可用於：WHERE a=1、WHERE a=1 AND b=2、WHERE a=1 AND b=2 AND c=3。不可用於 WHERE b=2 AND c=3（跳過 a）。最多 16 欄。

覆蓋索引：查詢欄位全在索引中，不需回表。EXPLAIN Extra 顯示 Using index。

🔶 不可見索引・降序索引・函式索引
-- 不可見索引（安全測試移除影響）
CREATE INDEX idx ON t1(col) INVISIBLE;
ALTER TABLE t1 ALTER INDEX idx INVISIBLE;  -- 優化器忽略
ALTER TABLE t1 ALTER INDEX idx VISIBLE;    -- 恢復

-- 降序索引（混合排序）
CREATE INDEX idx ON t1 (col1 ASC, col2 DESC);

-- 函式索引
CREATE INDEX idx ON t1 ((UPPER(col1)));
CREATE INDEX idx ON t1 ((col1 + col2));

不可見索引仍被維護但優化器忽略。主鍵不可設不可見。函式索引只優化完全相同表達式的查詢。

🔶 JOIN 優化：Hash Join & BKA

MySQL 8.4 等值 JOIN 無索引時預設使用 Hash Join（Block Nested-Loop 已完全移除）。

-- 控制 Hash Join
SELECT /*+ BNL(t1) */ ... ;     -- 啟用
SELECT /*+ NO_BNL(t2) */ ... ;  -- 停用

-- 啟用 BKA（預設 OFF）
SET optimizer_switch='mrr=on,mrr_cost_based=off,batched_key_access=on';
⚠️HASH_JOIN/NO_HASH_JOIN 提示在 8.4 無效，必須用 BNL/NO_BNL。

🟢 InnoDB Buffer Pool 優化
參數預設建議
innodb_buffer_pool_size128MB專用伺服器設 50-80% RAM
innodb_buffer_pool_instances自動每實例至少 1GB
innodb_buffer_pool_chunk_size128MB僅啟動時可改
-- 線上調整（無需重啟！）
SET GLOBAL innodb_buffer_pool_size = 8589934592;  -- 8GB

-- 命中率監控（目標 > 99%）
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_read_requests';  -- 邏輯讀
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_reads';          -- 物理讀
-- 命中率 = 1 - (物理讀 / 邏輯讀)
🔶 InnoDB 交易與 Redo Log 優化
innodb_flush_log_at_trx_commit行為效能 vs 安全
1（預設）每次提交刷盤完整 ACID，最安全
2寫 OS 快取，每秒刷盤OS 崩潰最多丟 1 秒
0每秒刷盤最快，任何崩潰丟 1 秒

innodb_redo_log_capacity（8.4 新參數）取代舊 innodb_log_file_size + innodb_log_files_in_group。InnoDB 維護 32 個 redo log 檔，可動態設定。

🟢 批次載入最佳化
SET autocommit = 0;
SET unique_checks = 0;         -- 確保無重複才關閉
SET foreign_key_checks = 0;

LOAD DATA INFILE '/path/data.csv' INTO TABLE t1 ...;
-- 或按主鍵順序的批次 INSERT

COMMIT;
SET unique_checks = 1;
SET foreign_key_checks = 1;
💡LOAD DATA INFILE 比 INSERT 快數倍。按主鍵順序插入利用叢集索引特性，避免隨機 I/O。
🔶 鎖定優化與死鎖處理
-- 查看最近死鎖
SHOW ENGINE INNODB STATUS\G

-- 查看目前鎖定（Performance Schema）
SELECT * FROM performance_schema.data_locks;
SELECT * FROM performance_schema.data_lock_waits;

-- sys schema 易讀分析
SELECT * FROM sys.innodb_lock_waits\G
參數預設說明
innodb_deadlock_detectON自動偵測死鎖並回滾較小交易
innodb_lock_wait_timeout50 秒鎖等待超時
innodb_print_all_deadlocksOFF所有死鎖記錄至 error log

死鎖最小化：保持交易小而短、一致順序存取表和列、提前用 SELECT ... FOR UPDATE 取得鎖、WHERE 欄位建索引。

🔶 Performance Schema 查詢優化
-- Top N 最耗時查詢
SELECT DIGEST_TEXT, COUNT_STAR,
  ROUND(SUM_TIMER_WAIT/1000000000000, 4) AS total_sec,
  ROUND(AVG_TIMER_WAIT/1000000000000, 4) AS avg_sec
FROM performance_schema.events_statements_summary_by_digest
ORDER BY SUM_TIMER_WAIT DESC LIMIT 10;

sys schema 實用視圖：sys.statements_with_runtimes_in_95th_percentile（慢查詢）、sys.statements_with_full_table_scans（全表掃描）、sys.schema_unused_indexes（未使用索引）、sys.schema_index_statistics（索引統計）。
